# 8.925 Extended Events — Capturing Slow Queries

## 1. Overview — Extended Events for Performance Tracing

Extended Events is the modern, lightweight tracing infrastructure in SQL Server, designed as the replacement for SQL Profiler and server-side traces. It provides a highly configurable event processing pipeline with minimal overhead, making it suitable for production use — even on busy OLTP systems.

The core value proposition of Extended Events is precision. Unlike Profiler traces that capture every event and force the server to filter after capture, Extended Events applies predicates at the source — events that do not match the predicate are never generated, never buffered, and never written to the target. This predicate push-down dramatically reduces overhead.

For slow query capture, the most common use case is:
- Capture queries that exceed a duration threshold (e.g., 1,000 ms)
- Collect actionable data: SQL text, query hash, plan handle, duration, CPU, reads, writes
- Store results in a file target for historical analysis
- Query captured data using T-SQL functions

The key architectural components are:
- **Event Session** — A named container configured with events, predicates, actions, and targets
- **Events** — Points in the SQL Server code path where data can be collected (e.g., `sqlserver.rpc_completed`, `sqlserver.sql_batch_completed`)
- **Actions** — Additional data collected when an event fires (e.g., `sql_text`, `plan_handle`, `session_id`)
- **Predicates** — Filters applied at the event source to limit what is captured
- **Targets** — Destinations for event data (ring_buffer, event_file, histogram, etc.)

Extended Events captures are fundamentally different from DMV snapshots. DMVs provide cumulative aggregates; Extended Events captures each qualifying event as it happens. This gives you individual query-level detail that DMVs cannot provide.

The tradeoff is that Extended Events has overhead for each captured event. The key to production use is setting predicates that limit capture to only the queries you care about — typically the slowest 1–5% of queries.

## 2. Architecture — Event Session Design

The architecture of an Extended Events session for slow query capture follows a pipeline model:

```
flowchart LR
    A[Query Execution] --> B{rpc_completed / sql_batch_completed}
    B --> C{Predicate: duration > 1000000}
    C -->|No| D[Event discarded - zero overhead]
    C -->|Yes| E[Actions fire: sql_text, plan_handle, etc.]
    E --> F[Event Buffer]
    F --> G[Event Target: event_file]
    G --> H[T-SQL Read: sys.fn_xe_file_target_read_file]
    H --> I[Analysis & Reporting]
```

### 2.1 Event Selection

Two primary events capture slow queries:

- **sqlserver.rpc_completed** — Fires when a stored procedure or Remote Procedure Call (RPC) completes. Captures the procedure name, duration, CPU, reads, writes. This is the primary event for OLTP workloads that use stored procedures.
- **sqlserver.sql_batch_completed** — Fires when an ad-hoc SQL batch or prepared statement completes. Captures the batch text. Use this for ad-hoc query workloads.

For a comprehensive slow query capture, both events are recommended:

```sql
ADD EVENT sqlserver.rpc_completed
    (WHERE duration > 1000000),
ADD EVENT sqlserver.sql_batch_completed
    (WHERE duration > 1000000)
```

### 2.2 Predicate Design

Predicates should be carefully designed to capture only what is necessary:

| Predicate | Purpose | Recommendation |
|-----------|---------|----------------|
| `duration > 1000000` | Capture queries > 1 second | Core threshold |
| `cpu_time > 500000` | Capture high CPU queries | Complementary |
| `logical_reads > 10000` | Capture high IO queries | Complementary |
| `physical_reads > 1000` | Capture physical IO | High overhead on predicate |
| `session_id <> 0` | Exclude system sessions | Always include |
| `database_id = 5` | Filter to specific database | Reduces noise |


### 2.3 Action Selection

Actions capture additional context when an event fires:

```sql
ACTION (
    sqlserver.sql_text,
    sqlserver.plan_handle,
    sqlserver.query_hash,
    sqlserver.session_id,
    sqlserver.database_id,
    sqlserver.client_hostname,
    sqlserver.client_app_name,
    sqlserver.username
)
```

Each action adds some overhead. The essential actions for slow query analysis are `sql_text`, `plan_handle`, and `query_hash`. The others are useful for identifying the source of the query.

### 2.4 Target Selection

The event target determines where captured data is stored:

| Target | Use Case | Pros | Cons |
|--------|----------|------|------|
| `ring_buffer` | Development, debugging | No files, memory only | Lost on restart, limited capacity |
| `event_file` | Production | Persistent, manageable, rollover | File management required |
| `histogram` | Aggregation | Low storage for counts | No individual event details |
| `event_counter` | Counts only | Minimal overhead | No detail data |
| `etw_classic_sync_target` | ETW integration | Advanced analysis with PerfMon | Complex setup |

For production always use `event_file`.

## 3. Implementation — Event Session Creation

The complete script to create a slow query capture session:

```sql
-- Create Extended Events session for slow queries
CREATE EVENT SESSION [SlowQueries]
ON SERVER
ADD EVENT sqlserver.rpc_completed
(
    ACTION (
        sqlserver.sql_text,
        sqlserver.plan_handle,
        sqlserver.query_hash,
        sqlserver.session_id,
        sqlserver.database_id,
        sqlserver.client_hostname,
        sqlserver.client_app_name,
        sqlserver.username
    )
    WHERE (
        duration > 1000000           -- > 1 second
        AND sqlserver.database_id > 4  -- Exclude system databases
    )
),
ADD EVENT sqlserver.sql_batch_completed
(
    ACTION (
        sqlserver.sql_text,
        sqlserver.plan_handle,
        sqlserver.query_hash,
        sqlserver.session_id,
        sqlserver.database_id,
        sqlserver.client_hostname,
        sqlserver.client_app_name,
        sqlserver.username
    )
    WHERE (
        duration > 1000000
        AND sqlserver.database_id > 4
    )
)
ADD TARGET package0.event_file
(
    SET filename = N'D:\XELogs\SlowQueries.xel',
        max_file_size = 256,           -- MB per file
        max_rollover_files = 10         -- Keep 10 files, total ~2.5 GB
)
WITH (
    MAX_MEMORY = 4096 KB,
    EVENT_RETENTION_MODE = ALLOW_SINGLE_EVENT_LOSS,
    MAX_DISPATCH_LATENCY = 30 SECONDS,
    MAX_EVENT_SIZE = 0 KB,
    MEMORY_PARTITION_MODE = NONE,
    TRACK_CAUSALITY = OFF,
    STARTUP_STATE = ON                 -- Auto-start when SQL Server starts
);
GO
```

## 4. Configuration — Event Target Options

### 4.1 Event File Target

The event_file target writes events to a `.xel` file on disk. Key configuration options:

```sql
ADD TARGET package0.event_file
(
    SET filename = N'D:\XELogs\SlowQueries.xel',   -- Base filename
        max_file_size = 256,                          -- MB per file (default 1 GB)
        max_rollover_files = 10,                      -- Number of files to keep
        metadatafile = N'D:\XELogs\SlowQueries.xem'    -- Metadata file (optional)
)
```

File naming convention:
- First file: `SlowQueries.xel`
- After rollover: `SlowQueries_0_129496000000000000.xel`
- Metadata: `SlowQueries.xem`

The metadata file tracks file sequence and should not be deleted manually.

### 4.2 Ring Buffer Target (Development Only)

For development and testing where persistence is not needed:

```sql
ADD TARGET package0.ring_buffer
(
    SET max_events_limit = 5000,        -- Max events in buffer
        max_memory = 10240              -- Max memory in KB (10 MB)
)
```

The ring buffer is kept in memory and lost on server restart. It is suitable for ad-hoc troubleshooting but not for production monitoring.

### 4.3 Histogram Target (Aggregation)

To capture only aggregate counts by a key (e.g., count of slow queries by query_hash):

```sql
ADD TARGET package0.histogram
(
    SET filtering_event_name = N'sqlserver.rpc_completed',
        source_type = 1,            -- 1 = action, 0 = event column
        source = N'query_hash',
        slots = 256
)
```

This target stores counts instead of individual events, drastically reducing storage. However, you lose individual query details like duration and CPU.

### 4.4 Maximum Memory and Dispatch Latency

```sql
WITH (
    MAX_MEMORY = 4096 KB,                    -- Buffer size for events
    MAX_DISPATCH_LATENCY = 30 SECONDS,        -- Flush buffer to target every 30 seconds
    EVENT_RETENTION_MODE = ALLOW_SINGLE_EVENT_LOSS,  -- Drop oldest event if buffer full
    MAX_EVENT_SIZE = 0 KB,                   -- No limit on event size
    MEMORY_PARTITION_MODE = NONE,             -- Single buffer
    TRACK_CAUSALITY = OFF                    -- No causal tracking for performance
)
```

For high-throughput systems, consider:
```sql
MAX_MEMORY = 8192 KB,              -- More buffer capacity
MEMORY_PARTITION_MODE = PER_NODE,  -- Per NUMA node buffer to reduce contention
MAX_DISPATCH_LATENCY = 15 SECONDS, -- More frequent writes
```

## 5. Query Patterns — Reading Captured Data

### 5.1 Reading from Event File

The primary function to read event file data is `sys.fn_xe_file_target_read_file`:

```sql
-- Read all captured events
SELECT 
    event_data.value('(event/@name)[1]', 'VARCHAR(50)') AS EventName,
    event_data.value('(event/@timestamp)[1]', 'DATETIME2') AS EventTime,
    event_data.value('(event/data[@name="duration"]/value)[1]', 'BIGINT') AS DurationMicroseconds,
    event_data.value('(event/data[@name="cpu_time"]/value)[1]', 'BIGINT') AS CPU,
    event_data.value('(event/data[@name="logical_reads"]/value)[1]', 'BIGINT') AS LogicalReads,
    event_data.value('(event/data[@name="physical_reads"]/value)[1]', 'BIGINT') AS PhysicalReads,
    event_data.value('(event/data[@name="writes"]/value)[1]', 'BIGINT') AS Writes,
    event_data.value('(event/data[@name="result"]/text)[1]', 'VARCHAR(20)') AS Result,
    event_data.value('(event/data[@name="statement"]/value)[1]', 'NVARCHAR(MAX)') AS Statement,
    event_data.value('(event/action[@name="sql_text"]/value)[1]', 'NVARCHAR(MAX)') AS SqlText,
    event_data.value('(event/action[@name="plan_handle"]/value)[1]', 'VARCHAR(64)') AS PlanHandle,
    event_data.value('(event/action[@name="query_hash"]/value)[1]', 'VARCHAR(40)') AS QueryHash,
    event_data.value('(event/action[@name="database_id"]/value)[1]', 'INT') AS DatabaseID,
    event_data.value('(event/action[@name="session_id"]/value)[1]', 'INT') AS SessionID,
    event_data.value('(event/action[@name="client_hostname"]/value)[1]', 'VARCHAR(64)') AS ClientHost,
    event_data.value('(event/action[@name="client_app_name"]/value)[1]', 'VARCHAR(128)') AS AppName,
    event_data.value('(event/action[@name="username"]/value)[1]', 'VARCHAR(64)') AS UserName
FROM (
    SELECT CAST(event_data AS XML) AS event_data
    FROM sys.fn_xe_file_target_read_file(
        N'D:\XELogs\SlowQueries*.xel',  -- Pattern matches all rollover files
        N'D:\XELogs\SlowQueries*.xem',  -- Metadata file pattern
        NULL, NULL                      -- No offset (read from beginning)
    )
) AS ed
ORDER BY EventTime DESC;
```

### 5.2 Top N Slow Queries

```sql
WITH XEventData AS (
    SELECT 
        event_data.value('(event/@name)[1]', 'VARCHAR(50)') AS EventName,
        event_data.value('(event/@timestamp)[1]', 'DATETIME2') AS EventTime,
        event_data.value('(event/data[@name="duration"]/value)[1]', 'BIGINT') / 1000.0 AS DurationMs,
        event_data.value('(event/data[@name="cpu_time"]/value)[1]', 'BIGINT') / 1000.0 AS CpuMs,
        event_data.value('(event/data[@name="logical_reads"]/value)[1]', 'BIGINT') AS LogicalReads,
        event_data.value('(event/action[@name="sql_text"]/value)[1]', 'NVARCHAR(MAX)') AS SqlText,
        event_data.value('(event/action[@name="query_hash"]/value)[1]', 'VARCHAR(40)') AS QueryHash
    FROM sys.fn_xe_file_target_read_file(
        N'D:\XELogs\SlowQueries*.xel', NULL, NULL, NULL
    )
    CROSS APPLY (SELECT CAST(event_data AS XML) AS event_data) AS ed
)
SELECT TOP 50
    LEFT(SqlText, 200) AS SqlTextSample,
    COUNT(*) AS Occurrences,
    AVG(DurationMs) AS AvgDurationMs,
    MAX(DurationMs) AS MaxDurationMs,
    AVG(CpuMs) AS AvgCpuMs,
    AVG(LogicalReads) AS AvgReads,
    QueryHash
FROM XEventData
WHERE EventTime > DATEADD(HOUR, -24, GETUTCDATE())
GROUP BY QueryHash, LEFT(SqlText, 200)
ORDER BY AvgDurationMs DESC;
```

### 5.3 Reading from Ring Buffer

```sql
-- Read from ring buffer (session must be running)
SELECT 
    event_data.value('(event/@name)[1]', 'VARCHAR(50)') AS EventName,
    event_data.value('(event/@timestamp)[1]', 'DATETIME2') AS EventTime,
    event_data.value('(event/data[@name="duration"]/value)[1]', 'BIGINT') / 1000.0 AS DurationMs,
    event_data.value('(event/action[@name="sql_text"]/value)[1]', 'NVARCHAR(MAX)') AS SqlText
FROM (
    SELECT CAST(target_data AS XML) AS target_data
    FROM sys.dm_xe_session_targets
    WHERE event_session_id = (
        SELECT session_id FROM sys.dm_xe_sessions WHERE name = 'SlowQueries'
    )
    AND target_name = 'ring_buffer'
) AS rb
CROSS APPLY target_data.nodes('RingBufferTarget/event') AS XEvent(event_data)
ORDER BY EventTime DESC;
```

### 5.4 Delta Analysis Between Two Time Periods

```sql
DECLARE @Period1Start DATETIME2 = '2026-06-26 08:00:00';
DECLARE @Period1End DATETIME2 = '2026-06-26 09:00:00';
DECLARE @Period2Start DATETIME2 = '2026-06-27 08:00:00';
DECLARE @Period2End DATETIME2 = '2026-06-27 09:00:00';

WITH Period1 AS (
    SELECT 
        event_data.value('(event/action[@name="query_hash"]/value)[1]', 'VARCHAR(40)') AS QueryHash,
        COUNT(*) AS ExecCount,
        AVG(event_data.value('(event/data[@name="duration"]/value)[1]', 'BIGINT') / 1000.0) AS AvgDurationMs
    FROM sys.fn_xe_file_target_read_file(N'D:\XELogs\SlowQueries*.xel', NULL, NULL, NULL)
    CROSS APPLY (SELECT CAST(event_data AS XML) AS event_data) AS ed
    WHERE event_data.value('(event/@timestamp)[1]', 'DATETIME2') BETWEEN @Period1Start AND @Period1End
    GROUP BY event_data.value('(event/action[@name="query_hash"]/value)[1]', 'VARCHAR(40)')
),
Period2 AS (
    SELECT 
        event_data.value('(event/action[@name="query_hash"]/value)[1]', 'VARCHAR(40)') AS QueryHash,
        COUNT(*) AS ExecCount,
        AVG(event_data.value('(event/data[@name="duration"]/value)[1]', 'BIGINT') / 1000.0) AS AvgDurationMs
    FROM sys.fn_xe_file_target_read_file(N'D:\XELogs\SlowQueries*.xel', NULL, NULL, NULL)
    CROSS APPLY (SELECT CAST(event_data AS XML) AS event_data) AS ed
    WHERE event_data.value('(event/@timestamp)[1]', 'DATETIME2') BETWEEN @Period2Start AND @Period2End
    GROUP BY event_data.value('(event/action[@name="query_hash"]/value)[1]', 'VARCHAR(40)')
)
SELECT 
    COALESCE(p1.QueryHash, p2.QueryHash) AS QueryHash,
    p1.AvgDurationMs AS Period1AvgDuration,
    p2.AvgDurationMs AS Period2AvgDuration,
    (p2.AvgDurationMs - p1.AvgDurationMs) AS DeltaMs,
    CASE 
        WHEN p1.AvgDurationMs = 0 THEN 100.0
        ELSE ((p2.AvgDurationMs - p1.AvgDurationMs) * 100.0) / p1.AvgDurationMs
    END AS PctChange
FROM Period1 p1
FULL JOIN Period2 p2 ON p1.QueryHash = p2.QueryHash
WHERE p2.AvgDurationMs > p1.AvgDurationMs
    AND p2.AvgDurationMs > 1000
ORDER BY DeltaMs DESC;
```

## 6. Production Deployment — Deploying Event Sessions

### 6.1 Deployment Script

Create a deployment script that can be run in CI/CD or manually:

```sql
-- Drop existing session if present (will lose any captured data)
IF EXISTS (SELECT 1 FROM sys.server_event_sessions WHERE name = 'SlowQueries')
BEGIN
    ALTER EVENT SESSION [SlowQueries] ON SERVER STATE = STOP;
    DROP EVENT SESSION [SlowQueries] ON SERVER;
END
GO

-- Create the session
CREATE EVENT SESSION [SlowQueries]
ON SERVER
ADD EVENT sqlserver.rpc_completed
(
    ACTION (sqlserver.sql_text, sqlserver.plan_handle, sqlserver.query_hash,
            sqlserver.session_id, sqlserver.database_id, sqlserver.client_hostname,
            sqlserver.client_app_name, sqlserver.username)
    WHERE (duration > 1000000 AND sqlserver.database_id > 4)
),
ADD EVENT sqlserver.sql_batch_completed
(
    ACTION (sqlserver.sql_text, sqlserver.plan_handle, sqlserver.query_hash,
            sqlserver.session_id, sqlserver.database_id, sqlserver.client_hostname,
            sqlserver.client_app_name, sqlserver.username)
    WHERE (duration > 1000000 AND sqlserver.database_id > 4)
)
ADD TARGET package0.event_file
(
    SET filename = N'D:\XELogs\SlowQueries.xel',
        max_file_size = 256,
        max_rollover_files = 10
)
WITH (
    MAX_MEMORY = 4096 KB,
    EVENT_RETENTION_MODE = ALLOW_SINGLE_EVENT_LOSS,
    MAX_DISPATCH_LATENCY = 30 SECONDS,
    MAX_EVENT_SIZE = 0 KB,
    MEMORY_PARTITION_MODE = NONE,
    TRACK_CAUSALITY = OFF,
    STARTUP_STATE = ON
);
GO

-- Start the session
ALTER EVENT SESSION [SlowQueries] ON SERVER STATE = START;
GO
```

### 7.2 File Rollover

When `max_rollover_files` is reached, SQL Server deletes the oldest file before creating a new one. This is automatic and requires no maintenance. However, during peak workloads, rollover can happen faster than expected.

Set an alert for file count:
```sql
-- Monitor rollover rate
SELECT 
    COUNT(*) AS CurrentFileCount,
    MIN(CAST(event_data.value('(event/@timestamp)[1]', 'DATETIME2') AS DATETIME2)) AS OldestEvent,
    MAX(CAST(event_data.value('(event/@timestamp)[1]', 'DATETIME2') AS DATETIME2)) AS NewestEvent
FROM sys.fn_xe_file_target_read_file(N'D:\XELogs\SlowQueries*.xel', NULL, NULL, NULL)
CROSS APPLY (SELECT CAST(event_data AS XML) AS event_data) AS ed;
```

### 7.3 Performance Impact Monitoring

Track the overhead of the event session:

```sql
-- Session overhead
SELECT 
    xs.name,
    xs.total_events_generated,
    xs.total_events_lost,
    xs.total_bytes_lost,
    xs.total_events_generated * 100.0 / NULLIF(xe.total_events, 0) AS PctOverhead
FROM sys.dm_xe_sessions xs
CROSS JOIN (
    SELECT SUM(total_events_generated) AS total_events
    FROM sys.dm_xe_sessions
    WHERE name != 'SlowQueries'  -- Exclude self
) xe
WHERE xs.name = 'SlowQueries';

-- Check for event loss (events lost means the session needs larger buffer)
SELECT 
    name,
    total_events_lost,
    total_bytes_lost
FROM sys.dm_xe_sessions
WHERE total_events_lost > 0;
```

### 7.4 Analyzing Captured Queries Over Time

Create a daily summary of slow queries:

```sql
-- Daily slow query summary
SELECT 
    CAST(EventTime AS DATE) AS CaptureDate,
    QueryHash,
    LEFT(SqlText, 200) AS SqlTextSample,
    COUNT(*) AS ExecutionCount,
    AVG(DurationMs) AS AvgDurationMs,
    MAX(DurationMs) AS MaxDurationMs,
    AVG(CpuMs) AS AvgCpuMs,
    AVG(LogicalReads) AS AvgReads,
    AVG(PhysicalReads) AS AvgPhysicalReads
FROM (
    SELECT 
        event_data.value('(event/@timestamp)[1]', 'DATETIME2') AS EventTime,
        event_data.value('(event/action[@name="query_hash"]/value)[1]', 'VARCHAR(40)') AS QueryHash,
        event_data.value('(event/action[@name="sql_text"]/value)[1]', 'NVARCHAR(MAX)') AS SqlText,
        event_data.value('(event/data[@name="duration"]/value)[1]', 'BIGINT') / 1000.0 AS DurationMs,
        event_data.value('(event/data[@name="cpu_time"]/value)[1]', 'BIGINT') / 1000.0 AS CpuMs,
        event_data.value('(event/data[@name="logical_reads"]/value)[1]', 'BIGINT') AS LogicalReads,
        event_data.value('(event/data[@name="physical_reads"]/value)[1]', 'BIGINT') AS PhysicalReads
    FROM sys.fn_xe_file_target_read_file(N'D:\XELogs\SlowQueries*.xel', NULL, NULL, NULL)
    CROSS APPLY (SELECT CAST(event_data AS XML) AS event_data) AS ed
) AS RawData
WHERE EventTime > DATEADD(DAY, -1, GETUTCDATE())
GROUP BY CAST(EventTime AS DATE), QueryHash, LEFT(SqlText, 200)
ORDER BY CaptureDate DESC, AvgDurationMs DESC;
```

## 8. Troubleshooting — Common Issues

### 8.1 Session Does Not Start

Check permissions and configuration:

```sql
-- Check session state
SELECT name, state, startup_state
FROM sys.dm_xe_sessions
WHERE name = 'SlowQueries';

-- Check for errors in session creation
SELECT 
    s.name,
    s.event_session_id,
    s.create_time,
    s.modify_time
FROM sys.server_event_sessions s
LEFT JOIN sys.dm_xe_sessions d ON s.name = d.name
WHERE s.name = 'SlowQueries';
```

If the session shows state=0 (stopped) when it should be running:
- Verify the `event_file` directory exists and is writable
- Check for permission issues (ALTER ANY EVENT SESSION)
- Review the SQL Server error log for XE-related errors

### 8.2 Event Loss

If `total_events_lost > 0`, the buffer is too small:

```sql
-- Check for event loss
SELECT 
    name,
    total_events_generated,
    total_events_lost,
    total_bytes_lost
FROM sys.dm_xe_sessions
WHERE name = 'SlowQueries';

-- Increase buffer size
ALTER EVENT SESSION [SlowQueries] ON SERVER
WITH (MAX_MEMORY = 8192 KB);
GO

-- Or partition memory per NUMA node
ALTER EVENT SESSION [SlowQueries] ON SERVER
WITH (MEMORY_PARTITION_MODE = PER_NODE);
GO
```

### 8.3 Capturing Too Many Events

If predicates are too broad, the session can generate excessive data:

Symptom: Event files grow too fast, filling the disk.

Solutions:
- Tighten duration predicate (e.g., from 1s to 3s)
- Add database_id filter
- Use histogram target instead of event_file
- Increase `max_rollover_files` to accommodate

```sql
-- Check capture rate
SELECT 
    COUNT(*) AS EventsPerMinute,
    AVG(DurationMs) AS AvgDurationMs
FROM (
    SELECT 
        event_data.value('(event/@timestamp)[1]', 'DATETIME2') AS EventTime,
        event_data.value('(event/data[@name="duration"]/value)[1]', 'BIGINT') / 1000.0 AS DurationMs
    FROM sys.fn_xe_file_target_read_file(N'D:\XELogs\SlowQueries*.xel', NULL, NULL, NULL)
    CROSS APPLY (SELECT CAST(event_data AS XML) AS event_data) AS ed
) AS d
WHERE EventTime > DATEADD(MINUTE, -10, GETUTCDATE());
```

### 8.4 Predicate Not Working

If you see events that should have been filtered out, the predicate may be syntactically incorrect or not applied at the correct level:

```sql
-- Verify predicate is applied
SELECT 
    evnt.name,
    evnt.predicate
FROM sys.server_event_sessions ses
JOIN sys.server_event_session_events evnt
    ON ses.event_session_id = evnt.event_session_id
WHERE ses.name = 'SlowQueries';
```

### 8.5 Event File In Use

Reading an event file while the session is writing to it is supported — the function handles active files. However, if you need to read a file that is exclusively locked:

- Stop the session first: `ALTER EVENT SESSION [SlowQueries] ON SERVER STATE = STOP;`
- Read the files
- Restart the session

### 8.6 Permission Errors

Creating and altering event sessions requires:

```sql
-- Grant permission to specific login
GRANT ALTER ANY EVENT SESSION TO [Domain\User];
GRANT ALTER ANY EVENT SESSION TO [NT Service\MSSQLSERVER];

-- View permissions for all sessions
SELECT 
    dp.name AS PrincipalName,
    dp.type_desc AS PrincipalType,
    perm.permission_name,
    perm.state_desc
FROM sys.server_permissions perm
JOIN sys.server_principals dp ON perm.grantee_principal_id = dp.principal_id
WHERE perm.class_desc = 'SERVER'
    AND perm.permission_name LIKE '%EVENT%';
```

## 9. Summary — Key Takeaways

Extended Events is the definitive tool for capturing slow queries in SQL Server. It is lightweight, precise, and designed for production use — a direct replacement for SQL Profiler that provides better performance and more flexible data capture.

| Component | Best Practice |
|-----------|---------------|
| Event Selection | Use `rpc_completed` for procs, `sql_batch_completed` for ad-hoc |
| Duration Threshold | Start at 1,000,000 microseconds (1 second), adjust based on workload |
| Target | `event_file` for production, `ring_buffer` for development |
| File Size | 256 MB per file, 10 files max |
| Startup | Set `STARTUP_STATE = ON` for automatic start |
| Retention | Event file rollover handles cleanup automatically |
| Permission | Requires `ALTER ANY EVENT SESSION` |
| Monitoring | Watch `total_events_lost` and file growth rate |

The key to successful Extended Events deployment is:
1. Start with a reasonable predicate (duration > 1s)
2. Monitor for event loss and disk consumption
3. Tighten predicates over time as you learn the workload pattern
4. Integrate with baseline DMV data for comprehensive observability

Extended Events captures the individual query details that DMV snapshots aggregate over time. Together, they provide both the tree-level view (what is happening) and the forest-level view (what changed).

When combined with Query Store for plan-level tracking and DMV baselines for trend analysis, Extended Events completes the observability triad — giving you the ability to drill from a performance alert down to the exact query text that caused it.
