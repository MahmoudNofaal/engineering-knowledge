# 8.929 SQL Server — Prometheus Exporter

Priority: 2 | Domain: Databases | Group: Database Monitoring & Observability

---

## Overview — Introduction & Purpose

The Prometheus SQL Server Exporter bridges SQL Server's internal performance data with Prometheus's time-series ecosystem. It connects to SQL Server instances, executes queries against Dynamic Management Views (DMVs) and performance counters, transforms the results into Prometheus-format metrics, and exposes them on an HTTP endpoint for Prometheus to scrape.

This approach replaces traditional polling-based monitoring with a pull-based architecture where the monitoring system (Prometheus) requests data at its own pace. The exporter is stateless and lightweight — each scrape request triggers fresh queries against SQL Server DMVs, ensuring the data reflects the current state of the instance.

Two primary implementations exist:

- **sql_exporter** (official, burningalchemist/sql_exporter) — A dedicated Go-based exporter designed specifically for SQL Server. It uses a YAML configuration file to define collectors, each of which maps to one or more DMV queries. Configuration is flexible: you can write custom queries and define metrics with custom labels.

- **Telegraf** (InfluxData) — A general-purpose metric collection agent with a sqlserver input plugin. Telegraf supports both Prometheus output (exposing a /metrics endpoint) and direct pushes to various backends. It is a better choice if you already run Telegraf for other infrastructure monitoring and want a unified agent.

Both implementations serve the same purpose: collect SQL Server performance data and make it available to Prometheus. This note covers both, with emphasis on the dedicated sql_exporter due to its SQL Server-specific optimizations and wider adoption in the Grafana dashboard ecosystem.

The exporter is the foundation of the Prometheus-based SQL Server monitoring stack documented in this group. Combined with [[8.928 — SQL Server Dashboards — Grafana Setup]], it enables comprehensive visualization and alerting. The metrics it exposes are documented in [[8.916 — SQL Server Monitoring — Key Metrics]], and the interpretation of specific metric categories (like wait statistics) is covered in [[8.917 — Wait Statistics — Top Waits Analysis]].

## Architecture — Data Flow & Collector Design

The exporter operates as a translating proxy between SQL Server and Prometheus. On each scrape request, it follows a defined pipeline:

```
┌──────────────────────────────────────────────────────────────────────┐
│                          SQL Server Instance                        │
│                                                                      │
│  sys.dm_os_wait_stats    sys.dm_os_performance_counters              │
│  sys.dm_io_virtual_file_stats   sys.dm_os_schedulers                 │
│  sys.dm_exec_query_stats   sys.dm_os_memory_clerks                   │
│  sys.dm_db_index_usage_stats   sys.dm_os_sys_info                   │
│  sys.master_files   sys.dm_tran_active_transactions                  │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌──────────────────────────────────────────────────────────────────────┐
│                       SQL Server Exporter                            │
│                                                                      │
│  1. Receive HTTP GET /metrics from Prometheus                        │
│  2. For each enabled collector:                                      │
│     a. Open connection to SQL Server                                 │
│     b. Execute collector SQL query against DMVs                      │
│     c. Transform result rows into Prometheus metric samples          │
│     d. Apply metric metadata (help text, type, labels)               │
│  3. Aggregate all metric families into response body                 │
│  4. Return HTTP 200 with text/plain body in Prometheus exposition    │
│     format                                                            │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌──────────────────────────────────────────────────────────────────────┐
│                       Prometheus Server                              │
│                                                                      │
│  1. Scrape job requests GET http://exporter:9399/metrics             │
│  2. Parse exposition format into time series                         │
│  3. Append samples to TSDB with current timestamp                    │
│  4. Apply metric relabeling if configured in scrape config           │
│  5. Store with retention policy                                      │
│  6. Evaluate alerting and recording rules                            │
│                                                                      │
│  TSDB Storage: ~1.3 bytes per sample + index overhead                │
│  Retention: default 15d, configurable                                │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌──────────────────────────────────────────────────────────────────────┐
│                    Grafana Dashboards & Alerts                       │
│                                                                      │
│  PromQL queries → Panels → Visualizations                            │
│  Alert rules → Notifications → PagerDuty / Slack / Email             │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### Collector Execution Model

Each collector in the configuration file defines:

- **Collector name** — unique identifier referenced in Prometheus metric labels
- **Query** — SQL statement executed against the target SQL Server instance
- **Metrics** — mapping of query result columns to Prometheus metric names, types, and labels
- **Collect interval override** (optional) — some collectors can run at different frequencies than the Prometheus scrape interval

When Prometheus scrapes the /metrics endpoint, the exporter iterates through all enabled collectors, executes their queries sequentially (or in parallel, depending on configuration), and aggregates results into a single response.

### Metric Naming Convention

Metrics follow Prometheus best practices:

- **Counter**: cumulative totals (e.g., `mssql_wait_stats_wait_time_ms_total`) — use rate() in PromQL
- **Gauge**: point-in-time values (e.g., `mssql_counter_value`) — use directly in PromQL
- **Histogram**: bucketed values (less common with sql_exporter, more common with custom collectors)

The `mssql_` prefix is conventional but configurable. Some collectors use `sqlserver_` especially when using Telegraf.

## Setup — Installation & Configuration

### Option 1: sql_exporter (Dedicated)

**Download and Install**

```
# Linux (AMD64)
wget https://github.com/burningalchemist/sql_exporter/releases/download/v0.6.0/sql_exporter-0.6.0.linux-amd64.tar.gz
tar xzf sql_exporter-0.6.0.linux-amd64.tar.gz
sudo cp sql_exporter-0.6.0.linux-amd64/sql_exporter /usr/local/bin/

# Windows
# Download sql_exporter-0.6.0.windows-amd64.zip from releases
# Extract and run sql_exporter.exe
```

**Configuration file: sql_exporter.yml**

The main configuration file defines connection strings, collectors, and global settings:

```yaml
global:
  scrape_timeout: 10s
  scrape_timeout_offset: 500ms
  min_interval: 0s
  max_connections: 3
  max_idle_connections: 3

target:
  # Single instance configuration
  data_source_name: "sqlserver://username:password@hostname:1433/instance?encrypt=disable&app+name=sql_exporter"

  # You can also use environment variables for credentials:
  # data_source_name: "sqlserver://${SQL_EXPORTER_USER}:${SQL_EXPORTER_PASS}@${SQL_SERVER_HOST}:1433/master?encrypt=disable&app+name=sql_exporter"

collectors:
  - mssql_wait_stats.yml
  - mssql_performance_counters.yml
  - mssql_io_stats.yml
  - mssql_schedulers.yml
  - mssql_volume_space.yml
```

**Running the exporter:**

```
sql_exporter --config.file sql_exporter.yml --web.listen-address :9399
```

The exporter logs to stdout. Test the endpoint:

```
curl http://localhost:9399/metrics
```

You should see Prometheus-format metrics beginning with `# HELP mssql_` comments, followed by metric lines.

**Installing as a Windows Service:**

```
# Using NSSM
nssm install SQLExporter "C:\Program Files\sql_exporter\sql_exporter.exe" "--config.file C:\Program Files\sql_exporter\sql_exporter.yml"

# Or run as a scheduled task (less ideal but works)
```

**Installing as a Linux Systemd Service:**

Create `/etc/systemd/system/sql_exporter.service`:

```
[Unit]
Description=Prometheus SQL Server Exporter
After=network.target

[Service]
Type=simple
User=prometheus
Group=prometheus
ExecStart=/usr/local/bin/sql_exporter --config.file /etc/sql_exporter/sql_exporter.yml --web.listen-address :9399
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

Enable and start:

```
systemctl daemon-reload
systemctl enable sql_exporter
systemctl start sql_exporter
systemctl status sql_exporter
```

### Option 2: Telegraf

**Install Telegraf** following the official documentation for your OS.

**Telegraf configuration** (in telegraf.conf or a snippet in conf.d/):

```toml
[[inputs.sqlserver]]
  servers = [
    "Server=hostname\\instance;Port=1433;User Id=monitor;Password=password;${app=telegraf};log=1;"
  ]
  
  # Collectors to enable
  query_version = 2
  
  # Enable specific collectors
  include_query = [
    "PerformanceCounters",
    "WaitStats",
    "IOStats",
    "MemoryClerk",
    "Schedulers",
    "VolumeSpace",
    "DatabaseIO",
    "ServerProperties"
  ]
  
  # Optional: exclude specific databases from queries
  exclude_query = ["QueryStats"]  # exclude expensive query-level stats
  
  # Metric export format
  [[outputs.prometheus_client]]
    listen = ":9273"
    metric_version = 2
```

Telegraf's sqlserver plugin supports multiple query versions. Version 2 (recommended) provides comprehensive coverage with optimized queries.

### Connection String Formats

**sql_exporter (sqlserver driver):**

```
sqlserver://username:password@hostname:1433/instance?encrypt=disable&app+name=sql_exporter
```

For named instances: `sqlserver://username:password@hostname:1433/InstanceName?encrypt=disable`

**Telegraf (pure ODBC):**

```
Server=hostname;Port=1433;User Id=username;Password=password;app=telegraf;log=1;
```

For named instances: `Server=hostname\InstanceName;Port=1433;...`

**Always use a dedicated monitoring login** with minimal required permissions (see Gotchas section). Never use sa or sysadmin accounts.

## Collectors — Key Collector Configuration

Each collector is a YAML file that defines the SQL query and metric mapping. Below are the essential collectors.

### mssql_wait_stats.yml

Captures wait statistics from `sys.dm_os_wait_stats`. This is one of the most valuable collectors for performance troubleshooting.

```yaml
collector_name: mssql_wait_stats

queries:
  - query: |
      SELECT
        wait_type,
        waiting_tasks_count,
        wait_time_ms,
        max_wait_time_ms,
        signal_wait_time_ms
      FROM sys.dm_os_wait_stats
      WHERE wait_type NOT LIKE '%SLEEP%'
        AND wait_type NOT LIKE '%QUEUE%'
        AND waiting_tasks_count > 0;
    metrics:
      - column_name: waiting_tasks_count
        metric_name: mssql_wait_stats_wait_count
        type: counter
        help: "Total number of waits by type"
      - column_name: wait_time_ms
        metric_name: mssql_wait_stats_wait_time_ms
        type: counter
        help: "Total wait time in milliseconds by type"
      - column_name: max_wait_time_ms
        metric_name: mssql_wait_stats_max_wait_time_ms
        type: gauge
        help: "Maximum wait time in milliseconds by type"
      - column_name: signal_wait_time_ms
        metric_name: mssql_wait_stats_signal_wait_time_ms
        type: counter
        help: "Total signal wait time in milliseconds by type"
    labels:
      - column_name: wait_type
        label_name: wait_type
```

This collector exposes four metrics with a `wait_type` label. Each distinct wait type generates a time series. There are typically 100-300 wait types, so this collector contributes 400-1,200 time series per instance — well within Prometheus capacity.

For wait categorization, you may add computed columns in the query to bucket wait types:

```sql
CASE
  WHEN wait_type LIKE 'PAGEIOLATCH%' OR wait_type IN ('WRITELOG', 'ASYNC_IO_COMPLETION') THEN 'IO'
  WHEN wait_type LIKE 'LCK%' THEN 'Lock'
  WHEN wait_type LIKE 'LATCH%' THEN 'Latch'
  WHEN wait_type IN ('CXCONSUMER', 'CXPACKET', 'EXCHANGE') THEN 'Parallelism'
  WHEN wait_type IN ('THREADPOOL', 'SOS_SCHEDULER_YIELD') THEN 'CPU'
  WHEN wait_type LIKE 'BROKER%' OR wait_type LIKE 'MIRROR%' THEN 'Service'
  ELSE 'Other'
END AS wait_category
```

Add `wait_category` as a label for simplified alerting and dashboard filtering.

### mssql_performance_counters.yml

Captures SQL Server performance counters from `sys.dm_os_performance_counters`. This collector exposes hundreds of counters covering every aspect of SQL Server behavior.

```yaml
collector_name: mssql_performance_counters

queries:
  - query: |
      SELECT
        RTRIM(object_name) AS object_name,
        RTRIM(counter_name) AS counter_name,
        RTRIM(instance_name) AS instance_name,
        cntr_value,
        cntr_type
      FROM sys.dm_os_performance_counters
      WHERE object_name LIKE '%:SQL Server%'
        AND counter_name IN (
          'Batch Requests/sec',
          'SQL Compilations/sec',
          'SQL Re-Compilations/sec',
          'Page life expectancy',
          'Buffer cache hit ratio',
          'User Connections',
          'Logins/sec',
          'Logouts/sec',
          'Deadlocks/sec',
          'Page reads/sec',
          'Page writes/sec',
          'Checkpoint pages/sec',
          'Transactions/sec',
          'Full Scans/sec',
          'Index Searches/sec',
          'Lock Waits/sec',
          'Avg. Disk Read Queue Length',
          'Avg. Disk Write Queue Length'
        );
    metrics:
      - column_name: cntr_value
        metric_name: mssql_counter_value
        type: gauge
        help: "SQL Server performance counter value"
    labels:
      - column_name: object_name
        label_name: object_name
      - column_name: counter_name
        label_name: counter_name
      - column_name: instance_name
        label_name: instance_name
```

Filter to relevant counters to avoid excessive metric volume. The full query (without the WHERE counter_name IN clause) exposes thousands of counters — many are not useful for monitoring and add noise.

### mssql_io_stats.yml

Collects file-level I/O statistics from `sys.dm_io_virtual_file_stats` joined with `sys.master_files`.

```yaml
collector_name: mssql_io_stats

queries:
  - query: |
      SELECT
        DB_NAME(database_id) AS database_name,
        file_id,
        CASE WHEN file_id = 1 THEN 'data' ELSE 'log' END AS file_type,
        num_of_reads,
        num_of_writes,
        io_stall_read_ms,
        io_stall_write_ms,
        io_stall,
        size_on_disk_bytes,
        num_of_bytes_read,
        num_of_bytes_written
      FROM sys.dm_io_virtual_file_stats(NULL, NULL) fs
      INNER JOIN sys.master_files mf ON fs.database_id = mf.database_id AND fs.file_id = mf.file_id;
    metrics:
      - column_name: num_of_reads
        metric_name: mssql_io_reads_total
        type: counter
        help: "Total number of reads per file"
      - column_name: num_of_writes
        metric_name: mssql_io_writes_total
        type: counter
        help: "Total number of writes per file"
      - column_name: io_stall_read_ms
        metric_name: mssql_io_stall_read_ms
        type: counter
        help: "Total I/O stall time for reads in milliseconds"
      - column_name: io_stall_write_ms
        metric_name: mssql_io_stall_write_ms
        type: counter
        help: "Total I/O stall time for writes in milliseconds"
      - column_name: io_stall
        metric_name: mssql_io_stall_ms
        type: counter
        help: "Total I/O stall time in milliseconds"
      - column_name: size_on_disk_bytes
        metric_name: mssql_io_file_size_bytes
        type: gauge
        help: "File size on disk in bytes"
    labels:
      - column_name: database_name
        label_name: database_name
      - column_name: file_id
        label_name: file_id
      - column_name: file_type
        label_name: file_type
```

I/O latency is calculated in PromQL: `rate(mssql_io_stall_read_ms[5m]) / rate(mssql_io_reads_total[5m])`.

### mssql_schedulers.yml

Exposes CPU scheduling information from `sys.dm_os_schedulers`.

```yaml
collector_name: mssql_schedulers

queries:
  - query: |
      SELECT
        scheduler_id,
        cpu_id,
        status,
        is_online,
        current_tasks_count,
        runnable_tasks_count,
        pending_disk_io_count,
        active_workers_count,
        work_queue_count,
        preemptive_switches_count,
        context_switches_count,
        total_cpu_idle_capped_worker_count,
        total_cpu_usage_ms,
        total_scheduler_delay_ms,
        load_factor
      FROM sys.dm_os_schedulers
      WHERE status = 'VISIBLE ONLINE';
    metrics:
      - column_name: current_tasks_count
        metric_name: mssql_scheduler_current_tasks
        type: gauge
        help: "Current number of tasks on the scheduler"
      - column_name: runnable_tasks_count
        metric_name: mssql_scheduler_runnable_tasks
        type: gauge
        help: "Number of tasks waiting to be scheduled on the runner"
      - column_name: pending_disk_io_count
        metric_name: mssql_scheduler_pending_disk_io
        type: gauge
        help: "Pending disk I/O count"
      - column_name: active_workers_count
        metric_name: mssql_scheduler_active_workers
        type: gauge
        help: "Active worker count"
      - column_name: work_queue_count
        metric_name: mssql_scheduler_work_queue
        type: gauge
        help: "Work queue length"
      - column_name: total_cpu_usage_ms
        metric_name: mssql_scheduler_cpu_usage_ms
        type: counter
        help: "Total CPU usage in milliseconds"
    labels:
      - column_name: scheduler_id
        label_name: scheduler_id
      - column_name: cpu_id
        label_name: cpu_id
```

Runnable task count is a key indicator of CPU pressure. If `runnable_tasks_count` consistently exceeds 10-20 across multiple schedulers, the system is CPU-bound.

### mssql_volume_space.yml

Captures database file space usage from `sys.master_files` and `sys.dm_db_file_space_usage` (for TempDB) or calculates from `FILEPROPERTY`.

```yaml
collector_name: mssql_volume_space

queries:
  - query: |
      SELECT DISTINCT
        DB_NAME(database_id) AS database_name,
        file_id,
        CASE WHEN type = 0 THEN 'data' WHEN type = 1 THEN 'log' ELSE 'filestream' END AS file_type,
        physical_name,
        CAST(size AS BIGINT) * 8192 AS size_bytes,
        CAST(max_size AS BIGINT) * 8192 AS max_size_bytes,
        CAST(growth AS BIGINT) AS growth_units,
        CASE WHEN is_percent_growth = 1 THEN growth ELSE growth * 8192 END AS growth_bytes
      FROM sys.master_files;
    metrics:
      - column_name: size_bytes
        metric_name: mssql_volume_space
        type: gauge
        help: "Current file size in bytes"
      - column_name: max_size_bytes
        metric_name: mssql_volume_space_max_size
        type: gauge
        help: "Maximum file size in bytes (0 = no limit, -1 = unlimited growth)"
    labels:
      - column_name: database_name
        label_name: database_name
      - column_name: file_id
        label_name: file_id
      - column_name: file_type
        label_name: file_type
      - column_name: physical_name
        label_name: file_path
```

Space used percentage is calculated in PromQL: `mssql_volume_space / mssql_volume_space_max_size * 100`.

### Additional Optional Collectors

**mssql_query_stats.yml** — Collects query execution statistics from `sys.dm_exec_query_stats`. This is the most expensive collector because it iterates through the entire plan cache. It exposes top queries by CPU, duration, reads, and writes. **Warning**: high cardinality from `query_hash` and optional `query_text` labels.

**mssql_memory_clerks.yml** — Collects memory usage by clerk type from `sys.dm_os_memory_clerks`. Useful for understanding how SQL Server allocates memory across buffer pool, plan cache, query memory grants, and other clerks.

**mssql_databases.yml** — Collects database-level properties: status, recovery model, compatibility level, last backup time, log reuse wait, and read-only status.

**mssql_availability_groups.yml** — For AlwaysOn Availability Groups: replica health, synchronization state, log send queue, estimated recovery time, and flow control waits.

**mssql_active_sessions.yml** — Captures active session information from `sys.dm_exec_sessions` and `sys.dm_exec_requests`: session count by database, blocking session chain, open transactions, wait resource.

## Collector Selection — Which to Enable

Not all collectors should be enabled on every server. Collector selection depends on workload type, monitoring objectives, and available SQL Server resources.

### Tier 1 — Always Enable (Low Cost, High Value)

| Collector | Query Cost | Time Series | Value |
|-----------|-----------|-------------|-------|
| mssql_wait_stats | Low | ~400-1200 | Critical for performance troubleshooting |
| mssql_performance_counters | Low | ~20-50 (filtered) | Core metrics for every dashboard |
| mssql_io_stats | Low | ~30-200 | I/O latency and throughput |
| mssql_schedulers | Low | ~10-80 | CPU scheduling and runnable tasks |
| mssql_volume_space | Low | ~20-100 | Disk space monitoring |

### Tier 2 — Enable if Relevant (Medium Cost)

| Collector | Query Cost | Time Series | Value |
|-----------|-----------|-------------|-------|
| mssql_memory_clerks | Low | ~50-100 | Memory pressure analysis |
| mssql_databases | Low | ~10-50 | Database health and configuration |
| mssql_active_sessions | Medium | ~20-100 | Real-time concurrency monitoring |

### Tier 3 — Enable Selectively (High Cost)

| Collector | Query Cost | Time Series | Value |
|-----------|-----------|-------------|-------|
| mssql_query_stats | High | 1000+ per db | Query-level performance optimization |
| mssql_availability_groups | Medium | ~50-200 | AG health monitoring |

The mssql_query_stats collector deserves special caution. On a server with a large plan cache (thousands of queries), it can generate 50,000+ time series per scrape. Each `query_hash` becomes a unique label value, and if `query_text` is included as a label, the cardinality problem compounds because query text is often unique per row.

**Cost analysis methodology:**

Estimate the collector cost using these factors:

1. **SQL Query Cost**: Run the collector query in SSMS with SET STATISTICS TIME ON and SET STATISTICS IO ON. If the query takes >100ms or scans millions of rows, consider disabling it or running it less frequently.

2. **Metric Volume**: Each enabled collector contributes to the total metric count. Multiply rows returned × metrics per row = new time series per scrape. A single collector generating 10,000+ metrics significantly increases Prometheus storage and query latency.

3. **Label Cardinality**: Each unique combination of label values creates a new time series. High-cardinality labels (query_hash, query_text, session_id, object_name with instance names) multiply the metric volume.

## Prometheus Configuration — Scrape Target Setup

After the exporter is running, configure Prometheus to scrape it.

### prometheus.yml scrape configuration

```yaml
scrape_configs:
  - job_name: 'sqlserver'
    scrape_interval: 15s
    scrape_timeout: 10s
    metrics_path: /metrics
    scheme: http
    
    # Static target configuration (single instance)
    static_configs:
      - targets:
        - 'sql-server-01:9399'
        - 'sql-server-02:9399'
        labels:
          environment: production
          cluster: primary-db
    
    # Or use service discovery:
    # file_sd_configs:
    #   - files: ['/etc/prometheus/targets/sqlserver/*.yml']
    #     refresh_interval: 5m
    
    # Metric relabeling (optional)
    metric_relabel_configs:
      # Drop high-cardinality labels that aren't needed
      - source_labels: [query_text]
        regex: '.+'
        action: drop
        # This drops the query_text label if present, reducing cardinality
      
      # Rename for consistency
      - source_labels: [__name__]
        regex: 'sqlserver_(.+)'
        replacement: 'mssql_${1}'
        action: replace
```

### Scrape Interval Guidelines

| Environment | Scrape Interval | Rationale |
|-------------|----------------|-----------|
| Production OLTP | 15s | Fine-grained view of rapid changes (deadlocks, blocking) |
| Production DW/BI | 30s | Longer-running queries, less sensitive to second-level changes |
| Staging/QA | 30-60s | Good enough for trend analysis, lower resource usage |
| Development | 60-120s | Minimal overhead, trend-level visibility only |

### Multiple Instance Configuration

For environments with many SQL Server instances, the exporter can be configured in two ways:

**One exporter per instance** (recommended): Install the exporter on or near each SQL Server, configure each with its own connection string. This isolates failures, simplifies updates, and allows instance-specific collector configurations.

**Single exporter, multiple data sources**: Use a connection pool or configure multiple targets in the exporter config. This is simpler to manage but creates a single point of failure and a single metric endpoint. The `instance` label distinguishes different SQL Servers.

```yaml
# Multi-instance exporter (advanced)
targets:
  - name: "SQL-Prod-01"
    data_source_name: "sqlserver://user:pass@prod01:1433/master?encrypt=disable"
  - name: "SQL-Prod-02"
    data_source_name: "sqlserver://user:pass@prod02:1433/master?encrypt=disable"
```

## Performance — Impact of Collectors on SQL Server

### Query Execution Cost

Each collector query runs against SQL Server DMVs on every scrape. Understanding the cost profile is essential to avoid impacting production workloads.

**Low-cost queries** (<10ms CPU, negligible I/O):

- `sys.dm_os_performance_counters` — in-memory view, no blocking
- `sys.dm_os_schedulers` — in-memory scheduler state
- `sys.dm_os_wait_stats` — in-memory running totals since last restart
- `sys.master_files` — metadata view, cached

**Medium-cost queries** (10-100ms CPU, some logical reads):

- `sys.dm_io_virtual_file_stats` — aggregates I/O stats since instance start
- `sys.dm_os_memory_clerks` — memory allocation by component
- `sys.dm_exec_sessions` — active sessions, may filter heavily

**High-cost queries** (100ms+ CPU, significant logical reads):

- `sys.dm_exec_query_stats` — iterates entire plan cache, can be CPU-intensive on servers with large caches
- `sys.dm_db_index_usage_stats` — large views on databases with many indexes
- `sys.dm_tran_active_transactions` — may contend with transaction commit path
- `sys.dm_db_file_space_usage` — requires TempDB database access

### Mitigation Strategies

**Stagger collector execution**: Configure the exporter to run expensive collectors at different intervals than the main scrape interval. Use the `min_interval` setting:

```yaml
# In collector config
collector_name: mssql_query_stats
min_interval: 60s  # Only execute every 60s, not on every scrape
```

**Use filtered queries**: Add WHERE clauses to limit rows returned. For query_stats, filter to top N queries by duration or CPU rather than returning all queries. For performance counters, select only the 20-30 counters that matter instead of returning all 2,000+.

**Use snapshot isolation**: Execute collector queries under read-committed snapshot isolation (RCSI) if enabled. This avoids blocking from monitoring queries:

```sql
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
-- or use WITH (NOLOCK) if the driver supports it
```

Note that DMVs accessed under READ UNCOMMITTED may return inconsistent data (e.g., duplicate rows, missing rows). For most monitoring use cases, this inconsistency is acceptable.

**Query plan analysis for collector queries**: Run each collector query through SSMS with actual execution plan enabled. Key things to check:

- Are there index scans on system tables? (Cannot be indexed, but should be limited)
- Are there spills to TempDB? (Indicates memory pressure)
- Does the query use parallelism? (Can saturate CPU on many-core systems)
- Estimated vs. actual rows — large discrepancies indicate plan quality issues

## Security — Permission Requirements

The SQL Server login used by the exporter requires specific permissions. The principle of least privilege dictates granting only the minimum permissions needed.

### Minimum Required Permissions

```sql
-- Create dedicated monitoring login
CREATE LOGIN [monitor_sql_exporter] WITH PASSWORD = 'StrongPasswordHere';
CREATE USER [monitor_sql_exporter] FOR LOGIN [monitor_sql_exporter];

-- Grant VIEW SERVER STATE (covers most DMVs)
GRANT VIEW SERVER STATE TO [monitor_sql_exporter];

-- Grant VIEW ANY DEFINITION (for schema metadata)
GRANT VIEW ANY DEFINITION TO [monitor_sql_exporter];

-- For sys.master_files (volume space)
GRANT SELECT ON sys.master_files TO [monitor_sql_exporter];

-- For each user database, grant connect and select on specific views
-- Needed for database-level DMVs
USE master;
GRANT VIEW DATABASE STATE TO [monitor_sql_exporter];
```

### Permission Requirements by Collector

| Collector | Required Permissions | Additional Notes |
|-----------|---------------------|------------------|
| mssql_wait_stats | VIEW SERVER STATE | sys.dm_os_wait_stats |
| mssql_performance_counters | VIEW SERVER STATE | sys.dm_os_performance_counters |
| mssql_io_stats | VIEW SERVER STATE | sys.dm_io_virtual_file_stats |
| mssql_schedulers | VIEW SERVER STATE | sys.dm_os_schedulers |
| mssql_volume_space | SELECT on sys.master_files | master_files is in master database |
| mssql_query_stats | VIEW SERVER STATE | sys.dm_exec_query_stats + sys.dm_exec_sql_text |
| mssql_memory_clerks | VIEW SERVER STATE | sys.dm_os_memory_clerks |
| mssql_active_sessions | VIEW SERVER STATE | sys.dm_exec_sessions + sys.dm_exec_requests |
| mssql_databases | VIEW ANY DEFINITION | Also needs database-level VIEW DATABASE STATE |

### Security Best Practices

1. **Never use sa or sysadmin accounts.** Create a dedicated login with only the permissions listed above.
2. **Encrypt connections**: Use `encrypt=true` in the connection string (or `Encrypt=YES` for ODBC) to protect credentials in transit.
3. **Use Windows Authentication** when possible (integrated security). This avoids storing passwords in the exporter configuration file.
4. **Restrict network access**: The exporter only needs outbound connectivity to SQL Server and inbound connectivity from Prometheus. Use firewalls and network policies to isolate.
5. **Audit monitoring activity**: Enable login auditing or use SQL Server Audit to track monitoring query execution.

### Windows Authentication Configuration

For sql_exporter running on Windows with integrated security:

```yaml
data_source_name: "sqlserver://hostname:1433/master?encrypt=disable&integrated+security=true&app+name=sql_exporter"
```

This uses the Windows identity of the process running the exporter. Ensure the service account has the required SQL Server permissions.

For Linux-based exporters connecting to Windows SQL Server, Windows Authentication requires Kerberos configuration or falling back to SQL Authentication with encrypted credentials.

## Troubleshooting — Common Issues & Gotchas

### Installation and Connectivity Issues

**Exporter starts but /metrics returns 500 Internal Server Error**

The exporter cannot connect to SQL Server. Check:

- Connection string validity (username, password, hostname, port, instance)
- SQL Server network configuration: TCP/IP must be enabled, port must match (default 1433)
- Firewall: test with `telnet hostname 1433` or `Test-NetConnection hostname -Port 1433` on Windows
- SQL Server authentication mode: must be "SQL Server and Windows Authentication mode" unless using Windows Auth
- Login status: EXEC sp_locklogin 'monitor_sql_exporter' should show the login is not locked

**No metrics returned for a specific collector**

- Check that the collector YAML file is in the correct directory and referenced in sql_exporter.yml
- Test the collector query in SSMS with the same login credentials
- Check exporter logs (enable debug logging with `--log.level=debug`)
- Verify that the metrics defined in the collector match the column names returned by the query

**Windows — exporter fails to authenticate with SQL Server**

- For SQL Authentication: ensure "SQL Server and Windows Authentication mode" is enabled in SQL Server
- For Windows Authentication: run exporter as a user/account that has been granted SQL Server access
- Check connection string escaping: semicolons, backslashes, and special characters may need escaping
- Use ODBC connection string format for Windows compatibility issues

### Metric Issues

**Metrics show cardinality explosion**

This is the most common issue with sql_exporter at scale. Symptoms:

- Prometheus scrape takes >5 seconds
- Prometheus memory usage grows continuously
- TSDB size grows faster than expected (>2GB per instance per day)
- Dashboard queries time out or return partial data

Causes and solutions:

1. **query_hash label without aggregation**: Each unique query_hash creates a time series. A server with 5,000 cached queries generates 5,000 time series per query metric (each metric type multiplies this). Solution: do not expose query_hash for all queries — pre-aggregate to top N or bucket by query_hash range.

2. **query_text label enabled**: Query text as a label is almost always unique per row, creating one time series per query execution plan. Solution: disable the `query_text` label entirely. If you need query text for debugging, log it separately to a file, not as a Prometheus label.

3. **Unfiltered performance counters**: The full set of sys.dm_os_performance_counters includes ~2,000+ rows covering every object instance (databases, indexes, wait types). Many of these are not useful for monitoring. Solution: filter to explicitly listed counter names.

4. **object_name as label with instance granularity**: When returning all performance counters, the `object_name` label may include database-specific counters (e.g., `SQLServer:Databases(AdventureWorks)`). This creates time series per database per counter. Solution: strip the instance name from object_name labels, or filter counters to instance-independent categories.

**Rate() or increase() shows negative values or flatlines**

- Counter reset: when the exporter restarts, all counters reset to zero. Prometheus rate() handles resets but may show a brief dip. Solution: use `resets()` function in rules to detect counter resets.
- Low scrape frequency: if Prometheus scrapes every 60s but the counter increment rate is very low, rate() may return 0 or NaN. Solution: increase the rate() range to capture at least 2 increments. For very slow counters, use increase() over 10m+ windows.
- SQL Server restart: Counter reset issue applies at a larger scale when SQL Server restarts. All DMV counters reset to zero. Annotations for SQL Server restarts help explain metric gaps.

**Exporter memory leak or growing memory usage**

- sql_exporter maintains a connection pool to SQL Server. If connections are not properly closed, memory grows over time.
- Collectors returning large result sets (especially query_stats without TOP N) consume memory.
- Solution: set `max_idle_connections` to a reasonable value (3-5). Enable metric collection limits. Restart the exporter periodically if needed.

### Configuration Gotchas

**Metric naming conflicts**

If the same metric name is defined in multiple collectors with conflicting types (gauge vs. counter), Prometheus will reject the second definition. Ensure unique metric names across all collectors, or use different names completely.

**Label naming conflicts**

Underscore labels (`__name__`, `__scheme__`, `__address__`) are reserved by Prometheus. Avoid using them as column names in collector queries. Similarly, avoid labels starting with `__`.

**Scrape timeout exceeded**

If a collector query takes longer than the Prometheus scrape_timeout, the scrape will fail. Configure `scrape_timeout_offset` in the exporter config to allow the exporter to fail gracefully:

```yaml
global:
  scrape_timeout: 10s
  scrape_timeout_offset: 500ms
```

The exporter returns what it has collected so far when the timeout minus offset is reached, rather than returning nothing.

**Version compatibility**

sql_exporter v0.5+ uses a different configuration format than v0.4. When using community dashboards (Dashboard ID 9628), verify the metric names match your exporter version. The dashboard may use `mssql_` or `sqlserver_` prefix depending on exporter version.

**Telegraf-sql_exporter parity**

If switching from Telegraf to sql_exporter (or vice versa), note that metric names differ:

- sql_exporter: `mssql_counter_value{counter_name="Page life expectancy"}`
- Telegraf: `sqlserver_performance.page_life_expectancy`

Dashboard panels must be updated for the metric name change.

## Production — Operational Best Practices

### Startup Configurations

**Run as a managed service**: Use systemd (Linux), Windows Service (Windows), or a container orchestration platform (Kubernetes) to ensure the exporter starts automatically and restarts on failure.

**Service monitoring**: Monitor the exporter's /healthz endpoint (if available) or the `up` Prometheus metric. Configure alerting for exporter down:

```yaml
alert: SQLServerExporterDown
expr: up{job="sqlserver"} == 0
for: 1m
labels:
  severity: critical
annotations:
  summary: "SQL Server Exporter down for {{ $labels.instance }}"
```

**Connection resilience**: Configure the exporter to retry SQL Server connections on failure. The exporter should survive brief SQL Server restarts or network interruptions.

### Configuration Management

**Version control collector configs**: Store sql_exporter.yml and all collector YAML files in Git. Tag versions with exporter releases for traceability. Use pull requests for configuration changes.

**Secrets management**: Never hardcode database credentials in configuration files. Use:

- Environment variables with a master config (sql_exporter supports `${VAR}` substitution)
- Secret management tools (Hashicorp Vault, Azure Key Vault, AWS Secrets Manager)
- Docker secrets (if running in containers)
- Windows Authentication (integrated security) to avoid credential storage entirely

**Automated deployment**: Use configuration management (Ansible, Puppet, Chef) or container images to deploy the exporter consistently across all SQL Server instances.

### Capacity Planning for Prometheus

| Factor | Estimate | Notes |
|--------|----------|-------|
| Metrics per instance per scrape | 2,000-5,000 | Depends on enabled collectors |
| Time series per instance | 10,000-50,000 | After all collectors, active series |
| Storage per instance per day | 50-250MB | Depends on label cardinality |
| Storage per instance per year | 18-90GB | At default retention + downsampling |
| Scrape bandwidth per instance | 5-50KB/s | At 15s interval |

For an organization monitoring 50 SQL Server instances:

- Total time series: 500,000-2.5M (within single Prometheus capacity at 5M series)
- Daily storage: 2.5-12.5GB
- Monthly storage: 75-375GB (at 15-day retention, this is 37.5-187.5GB)
- Queries per second on Prometheus: depends on dashboard usage — 10 dashboards × 30 panels each = 300 queries per refresh

Scale Prometheus vertically (more CPU, more memory) up to ~5M active series. Beyond that, consider:

- Horizontal sharding: one Prometheus per group of SQL Server instances
- Thanos or Cortex: multi-Prometheus with global query view and long-term storage
- VictoriaMetrics: Prometheus-compatible with better compression and performance at scale

### Collector Performance Baseline

Establish a performance baseline for the exporter itself:

- Measure exporter CPU usage (should be <1% for most configurations)
- Measure exporter memory usage (should stabilize at 20-50MB)
- Measure scrape duration (should be <2s for a well-configured exporter with 5-8 collectors)
- Track Prometheus scrape failures or timeouts

If the exporter CPU exceeds 5% or memory exceeds 200MB, review collector configuration for overly expensive queries.

### Upgrading the Exporter

1. Review changelog for breaking configuration or metric changes
2. Test in a staging environment with a copy of SQL Server metrics (or point to a dev instance)
3. Compare metric names and labels between old and new versions
4. Use metric relabeling in Prometheus to bridge naming differences during migration
5. Roll out progressively (canary one instance before full rollout)
6. Update dashboards if metric names changed
