## 1. Overview - Categorizing Waits for Root Cause Analysis

Raw wait statistics show hundreds of individual wait types. The diagnostic power comes from grouping them into categories. Once categorized, you can determine at a glance whether the bottleneck is CPU, IO, Lock, Memory, or something else. Each category maps to a specific subsystem and has distinct remediation strategies.

### 1.1 The Four Primary Categories

| Category | Wait Types Included | What It Indicates | Typical Healthy % |
|---|---|---|---|
| CPU | SOS_SCHEDULER_YIELD, THREADPOOL | Processor saturation, inefficient queries | Under 20% |
| IO | PAGEIOLATCH_SH/EX/UP, WRITELOG, ASYNC_IO_COMPLETION | Storage subsystem or memory pressure | Under 40% |
| Lock | LCK_M_S, LCK_M_X, LCK_M_U, LCK_M_SCH_S, LCK_M_SCH_M | Concurrency contention, blocking | Under 5% |
| Memory | RESOURCE_SEMAPHORE, RESOURCE_SEMAPHORE_QUERY_COMPILE, CMEMTHREAD | Memory grant pressure, cache issues | Under 5% |

### 1.2 Threshold Interpretation

```
Wait Category Distribution - Decision Flow

        IO > 50%?    Yes -> Storage problem OR memory pressure
        No
        CPU > 30%?   Yes -> Query tuning OR parallelism issue
        No
        Lock > 20%?  Yes -> Concurrency/blocking problem
        No
        Memory > 10%? Yes -> Memory grant configuration OR pressure
        No
        Mixed < thresholds -> Investigate next-level waits
```

## 2. Core Concepts - Wait Category Grouping

### 2.1 Why Categorize

Individual wait types can be misleading in isolation. For example:
- PAGEIOLATCH_SH high by itself could indicate slow storage OR memory pressure
- SOS_SCHEDULER_YIELD high by itself could indicate CPU pressure OR parallelism skew
- WRITELOG high could indicate log disk latency OR excessive logging

Categorizing tells you which subsystem to focus on. The percentage breakdown tells you the relative severity.

### 2.2 Category Membership Rules

The categorization query uses CASE WHEN with LIKE patterns on wait_type:

| Pattern | Category | Notes |
|---|---|---|
| SOS_SCHEDULER_YIELD | CPU | Query voluntarily yielding scheduler |
| THREADPOOL | CPU | All worker threads busy |
| PAGEIOLATCH% | IO | Page load from disk |
| WRITELOG | IO | Log write completion |
| IO_COMPLETION | IO | Async IO completion |
| ASYNC_IO_COMPLETION | IO | Async IO completion |
| LCK% | Lock | Lock waits (all modes) |
| RESOURCE_SEMAPHORE | Memory | Memory grant waits |
| CMEMTHREAD | Memory | Memory object contention |

### 2.3 Signal Wait Split by Category

Signal waits affect all categories - the CPU queue time is added to whatever resource wait the thread was experiencing. When CPU pressure exists, it inflates other categories' wait times. This is why we analyze signal wait percentage separately and subtract it for "pure" category analysis.

```
Thread Timeline - Signal Wait Impact on Categories

[Resource Wait Start] -- IO, Lock, or Memory wait begins
    |
    |--- 200ms --- Resource IO wait (PAGEIOLATCH_SH)
    |
[Resource Available] -- Thread becomes Runnable
    |
    |--- 50ms --- Signal wait (CPU queue)
    |
[Thread Running] -- Query execution continues

Recorded: wait_time_ms = 250 (200 resource + 50 signal)
          signal_wait_time_ms = 50

The IO category includes the 50ms of CPU queue time.
For pure IO analysis, subtract signal wait from IO waits.
```

## 3. Deep Mechanics - Categorization Query

### 3.1 The Core Categorization Query

```sql
-- Wait categorization with percentage calculation
-- Groups all waits into CPU, IO, Lock, Memory, and Other
WITH WaitCategories AS (
    SELECT
        wait_type,
        wait_time_ms,
        waiting_tasks_count,
        signal_wait_time_ms,
        CASE
            WHEN wait_type = 'SOS_SCHEDULER_YIELD' THEN 'CPU'
            WHEN wait_type = 'THREADPOOL' THEN 'CPU'
            WHEN wait_type LIKE 'PAGEIOLATCH%' THEN 'IO'
            WHEN wait_type = 'WRITELOG' THEN 'IO'
            WHEN wait_type = 'IO_COMPLETION' THEN 'IO'
            WHEN wait_type = 'ASYNC_IO_COMPLETION' THEN 'IO'
            WHEN wait_type LIKE 'LCK%' THEN 'Lock'
            WHEN wait_type = 'RESOURCE_SEMAPHORE' THEN 'Memory'
            WHEN wait_type = 'RESOURCE_SEMAPHORE_QUERY_COMPILE' THEN 'Memory'
            WHEN wait_type = 'CMEMTHREAD' THEN 'Memory'
            WHEN wait_type IN ('CXPACKET', 'CXCONSUMER', 'EXCHANGE') THEN 'Parallel'
            WHEN wait_type = 'PAGELATCH_UP' THEN 'TempDB'
            WHEN wait_type = 'PAGELATCH_EX' THEN 'TempDB'
            WHEN wait_type = 'ASYNC_NETWORK_IO' THEN 'Network'
            WHEN wait_type LIKE 'LATCH%' THEN 'Latch'
            WHEN wait_type = 'OLEDB' THEN 'OLEDB'
            WHEN wait_type IN ('MIRROR_SEND','DBMIRROR_SEND') THEN 'Mirroring'
            ELSE 'Other'
        END AS category
    FROM sys.dm_os_wait_stats
    WHERE wait_type NOT IN (
        'BROKER_EVENT_HANDLER', 'BROKER_INTERNAL', 'BROKER_RECEIVE_WAIT',
        'BROKER_TO_FLUSH', 'BROKER_TRANSMITTER', 'CHECKPOINT_QUEUE',
        'CHKPT', 'CLR_AUTO_EVENT', 'CLR_MANUAL_EVENT', 'CLR_SEMAPHORE',
        'DBMIRROR_DBM_EVENT', 'DBMIRROR_EVENTS_QUEUE', 'DBMIRROR_WORKER_QUEUE',
        'DBMIRRORING_CMD', 'DIRTY_PAGE_POLL', 'DISPATCHER_QUEUE_SEMAPHORE',
        'EXECSYNC', 'FSAGENT', 'FT_IFTS_SCHEDULER_IDLE_WAIT', 'FT_IFTSHC_MUTEX',
        'HADR_CLUSAPI_CALL', 'HADR_FILESTREAM_IOMGR_IOCOMPLETION',
        'HADR_LOGCAPTURE_WAIT', 'HADR_NOTIFICATION_DEQUEUE', 'HADR_TIMER_TASK',
        'HADR_WORK_QUEUE', 'LAZYWRITER_SLEEP', 'LOGMGR_QUEUE', 'MEMORY_ALLOCATION_EXT',
        'ONDEMAND_TASK_QUEUE', 'REQUEST_DISPENSER_PAUSE',
        'SLEEP_BPOOL_FLUSH', 'SLEEP_DBSTARTUP', 'SLEEP_DCOMSTARTUP',
        'SLEEP_MSDBSTARTUP', 'SLEEP_SYSTEMTASK', 'SLEEP_TASK', 'SLEEP_TEMPDBSTARTUP',
        'SNI_HTTP_ACCEPT', 'SP_SERVER_DIAGNOSTICS_SLEEP', 'SQLTRACE_BUFFER_FLUSH',
        'SQLTRACE_INCREMENTAL_FLUSH_SLEEP', 'SQLTRACE_WAIT_ENTRIES', 'WAIT_FOR_RESULTS',
        'WAITFOR', 'WAITFOR_TASKSHUTDOWN', 'XE_BUFFERMGR_ALLPROCESSED_EVENT',
        'XE_DISPATCHER_JOIN', 'XE_DISPATCHER_WAIT', 'XE_LIVE_TARGET_TVF', 'XE_TIMER_EVENT',
        'XE_TIMER_MUTEX', 'PREEMPTIVE_OS_LIBRARYOPS', 'PREEMPTIVE_OS_COMOPS',
        'PREEMPTIVE_OS_CRYPTOPS', 'PREEMPTIVE_OS_PIPEOPS',
        'PREEMPTIVE_OS_AUTHENTICATIONOPS', 'PREEMPTIVE_OS_GENERICOPS',
        'PREEMPTIVE_OS_VERIFYTRUST', 'PREEMPTIVE_OS_FILEOPS',
        'PREEMPTIVE_OS_DEVICEOPS', 'PREEMPTIVE_OS_QUERYREGISTRY',
        'PREEMPTIVE_OS_WRITEFILE', 'PREEMPTIVE_XE_CALLBACKEXECUTE',
        'PREEMPTIVE_XE_DISPATCHER', 'PREEMPTIVE_XE_GETTARGETSTATE',
        'PREEMPTIVE_XE_SESSIONCOMMIT', 'PREEMPTIVE_XE_TARGETINIT',
        'PREEMPTIVE_XE_TARGETFINALIZE', 'PREEMPTIVE_XPS'
    )
)
SELECT
    category,
    COUNT(*) AS wait_type_count,
    SUM(wait_time_ms) AS total_wait_time_ms,
    SUM(wait_time_ms - signal_wait_time_ms) AS total_resource_wait_time_ms,
    SUM(signal_wait_time_ms) AS total_signal_wait_time_ms,
    SUM(waiting_tasks_count) AS total_waiting_tasks_count,
    CAST(SUM(wait_time_ms) * 100.0 / NULLIF(SUM(SUM(wait_time_ms)) OVER (), 0) AS DECIMAL(5,2)) AS pct_of_total_waits,
    CAST(SUM(signal_wait_time_ms) * 100.0 / NULLIF(SUM(wait_time_ms), 0) AS DECIMAL(5,2)) AS signal_wait_pct_in_category,
    CASE
        WHEN SUM(wait_time_ms) * 100.0 / NULLIF(SUM(SUM(wait_time_ms)) OVER (), 0) > 50 AND category = 'IO'
            THEN 'CRITICAL - IO subsystem is primary bottleneck'
        WHEN SUM(wait_time_ms) * 100.0 / NULLIF(SUM(SUM(wait_time_ms)) OVER (), 0) > 30 AND category = 'CPU'
            THEN 'CRITICAL - CPU pressure detected'
        WHEN SUM(wait_time_ms) * 100.0 / NULLIF(SUM(SUM(wait_time_ms)) OVER (), 0) > 20 AND category = 'Lock'
            THEN 'CRITICAL - Lock contention detected'
        WHEN SUM(wait_time_ms) * 100.0 / NULLIF(SUM(SUM(wait_time_ms)) OVER (), 0) > 15 AND category = 'Memory'
            THEN 'CRITICAL - Memory pressure detected'
        ELSE 'Monitor if growing'
    END AS assessment
FROM WaitCategories
WHERE category IN ('CPU', 'IO', 'Lock', 'Memory', 'Parallel', 'TempDB', 'Latch', 'Network')
GROUP BY category
ORDER BY total_wait_time_ms DESC;
```

### 3.4 Pure Resource Wait Analysis (Removing Signal Wait)

To get a clean picture of each category's resource contribution, subtract signal waits from each category:

```sql
-- Pure resource wait analysis - removes CPU queue time from each category
WITH WaitCategories AS (
    SELECT
        wait_type,
        wait_time_ms,
        signal_wait_time_ms,
        wait_time_ms - signal_wait_time_ms AS resource_wait_time_ms,
        CASE
            WHEN wait_type = 'SOS_SCHEDULER_YIELD' THEN 'CPU'
            WHEN wait_type = 'THREADPOOL' THEN 'CPU'
            WHEN wait_type LIKE 'PAGEIOLATCH%' THEN 'IO'
            WHEN wait_type = 'WRITELOG' THEN 'IO'
            WHEN wait_type = 'IO_COMPLETION' THEN 'IO'
            WHEN wait_type = 'ASYNC_IO_COMPLETION' THEN 'IO'
            WHEN wait_type LIKE 'LCK%' THEN 'Lock'
            WHEN wait_type = 'RESOURCE_SEMAPHORE' THEN 'Memory'
            WHEN wait_type = 'RESOURCE_SEMAPHORE_QUERY_COMPILE' THEN 'Memory'
            WHEN wait_type = 'CMEMTHREAD' THEN 'Memory'
            ELSE NULL
        END AS category
    FROM sys.dm_os_wait_stats
    WHERE wait_type NOT IN ('BROKER_EVENT_HANDLER','SLEEP_TASK','WAITFOR','XE_TIMER_EVENT')
)
SELECT
    category,
    SUM(resource_wait_time_ms) AS total_resource_wait_ms,
    CAST(SUM(resource_wait_time_ms) * 100.0
        / NULLIF(SUM(SUM(resource_wait_time_ms)) OVER (), 0) AS DECIMAL(5,2)) AS resource_pct,
    SUM(signal_wait_time_ms) AS total_signal_wait_ms,
    CAST(SUM(signal_wait_time_ms) * 100.0
        / NULLIF(SUM(signal_wait_time_ms) OVER (), 0) AS DECIMAL(5,2)) AS signal_pct_of_all_signals,
    CASE
        WHEN SUM(resource_wait_time_ms) * 100.0 / SUM(SUM(resource_wait_time_ms)) OVER () > 50
            AND category = 'IO' THEN 'PURE IO BOTTLENECK - Focus on storage'
        WHEN SUM(resource_wait_time_ms) * 100.0 / SUM(SUM(resource_wait_time_ms)) OVER () > 30
            AND category = 'CPU' THEN 'PURE CPU BOTTLENECK - Focus on queries'
        ELSE ''
    END AS pure_bottleneck_assessment
FROM WaitCategories
WHERE category IS NOT NULL
GROUP BY category
ORDER BY total_resource_wait_ms DESC;
```

### 3.5 Category Trend Analysis Over Time

```sql
-- Track category percentages across snapshots for trend analysis
-- Requires the monitoring.WaitStatsHistory table from 8.917
WITH SnapshotCategories AS (
    SELECT
        CaptureTime,
        CASE
            WHEN wait_type LIKE 'PAGEIOLATCH%' OR wait_type IN ('WRITELOG','IO_COMPLETION','ASYNC_IO_COMPLETION') THEN 'IO'
            WHEN wait_type IN ('SOS_SCHEDULER_YIELD','THREADPOOL') THEN 'CPU'
            WHEN wait_type LIKE 'LCK%' THEN 'Lock'
            WHEN wait_type IN ('RESOURCE_SEMAPHORE','RESOURCE_SEMAPHORE_QUERY_COMPILE','CMEMTHREAD') THEN 'Memory'
            WHEN wait_type IN ('CXPACKET','CXCONSUMER') THEN 'Parallel'
            ELSE 'Other'
        END AS category,
        wait_time_ms
    FROM monitoring.WaitStatsHistory
    WHERE wait_type NOT IN ('BROKER_EVENT_HANDLER','SLEEP_TASK','WAITFOR','XE_TIMER_EVENT')
),
PerSnapshot AS (
    SELECT
        CaptureTime,
        category,
        SUM(wait_time_ms) AS category_total_ms,
        SUM(SUM(wait_time_ms)) OVER (PARTITION BY CaptureTime) AS snapshot_total_ms
    FROM SnapshotCategories
    GROUP BY CaptureTime, category
)
SELECT
    CaptureTime,
    category,
    category_total_ms,
    CAST(category_total_ms * 100.0 / snapshot_total_ms AS DECIMAL(5,2)) AS pct,
    LAG(CAST(category_total_ms * 100.0 / snapshot_total_ms AS DECIMAL(5,2)))
        OVER (PARTITION BY category ORDER BY CaptureTime) AS previous_pct,
    CAST(category_total_ms * 100.0 / snapshot_total_ms AS DECIMAL(5,2))
        - LAG(CAST(category_total_ms * 100.0 / snapshot_total_ms AS DECIMAL(5,2)))
            OVER (PARTITION BY category ORDER BY CaptureTime) AS pct_change
FROM PerSnapshot
ORDER BY CaptureTime, category;
```

## 4. Production Configuration - Categorization Script

### 4.1 Full Production Categorization Procedure

```sql
CREATE OR ALTER PROCEDURE monitoring.AnalyzeWaitCategories
    @ExcludeHarmless BIT = 1
AS
BEGIN
    SET NOCOUNT ON;

    -- Step 1: Calculate category totals
    ;WITH RawCategorized AS (
        SELECT
            wait_type,
            wait_time_ms,
            signal_wait_time_ms,
            wait_time_ms - signal_wait_time_ms AS resource_wait_ms,
            CASE
                WHEN wait_type = 'SOS_SCHEDULER_YIELD' THEN 'CPU'
                WHEN wait_type = 'THREADPOOL' THEN 'CPU'
                WHEN wait_type LIKE 'PAGEIOLATCH%' THEN 'IO'

                WHEN wait_type = 'IO_COMPLETION' THEN 'IO'

                WHEN wait_type = 'CMEMTHREAD' THEN 'Memory'
                WHEN wait_type IN ('CXPACKET','CXCONSUMER','EXCHANGE') THEN 'Parallel'

                WHEN wait_type = 'ASYNC_NETWORK_IO' THEN 'Network'

            'BROKER_EVENT_HANDLER','BROKER_INTERNAL','BROKER_RECEIVE_WAIT',

            'HADR_WORK_QUEUE','LAZYWRITER_SLEEP','LOGMGR_QUEUE','MEMORY_ALLOCATION_EXT',

            'SLEEP_BPOOL_FLUSH','SLEEP_DBSTARTUP','SLEEP_DCOMSTARTUP',
            'SLEEP_MSDBSTARTUP','SLEEP_SYSTEMTASK','SLEEP_TASK','SLEEP_TEMPDBSTARTUP',
            'SNI_HTTP_ACCEPT','SP_SERVER_DIAGNOSTICS_SLEEP','SQLTRACE_BUFFER_FLUSH',
            'SQLTRACE_INCREMENTAL_FLUSH_SLEEP','SQLTRACE_WAIT_ENTRIES','WAIT_FOR_RESULTS',
            'WAITFOR','WAITFOR_TASKSHUTDOWN','XE_BUFFERMGR_ALLPROCESSED_EVENT',
            'XE_DISPATCHER_JOIN','XE_DISPATCHER_WAIT','XE_LIVE_TARGET_TVF','XE_TIMER_EVENT'
        )
    ),
    CategorySummary AS (
        SELECT
            category,
            SUM(wait_time_ms) AS total_wait_ms,
            SUM(resource_wait_ms) AS total_resource_ms,
            SUM(signal_wait_time_ms) AS total_signal_ms,
            COUNT(*) AS wait_type_count,
            SUM(SUM(wait_time_ms)) OVER () AS grand_total_ms
        FROM RawCategorized
        WHERE category IN ('CPU', 'IO', 'Lock', 'Memory', 'Parallel', 'TempDB', 'Network')
        GROUP BY category
    )
    SELECT
        category,
        total_wait_ms,
        total_resource_ms,
        total_signal_ms,
        wait_type_count,
        CAST(total_wait_ms * 100.0 / NULLIF(grand_total_ms, 0) AS DECIMAL(5,2)) AS pct_of_total,
        CAST(total_signal_ms * 100.0 / NULLIF(total_wait_ms, 0) AS DECIMAL(5,2)) AS signal_wait_pct,
        CASE
            WHEN category = 'IO' AND total_wait_ms * 100.0 / NULLIF(grand_total_ms, 0) > 50
                THEN 'PRIMARY BOTTLENECK - IO subsystem'
            WHEN category = 'IO' AND total_wait_ms * 100.0 / NULLIF(grand_total_ms, 0) > 30
                THEN 'Significant IO - check storage and memory'
            WHEN category = 'CPU' AND total_wait_ms * 100.0 / NULLIF(grand_total_ms, 0) > 30
                THEN 'PRIMARY BOTTLENECK - CPU pressure'
            WHEN category = 'CPU' AND total_wait_ms * 100.0 / NULLIF(grand_total_ms, 0) > 20
                THEN 'Moderate CPU - monitor trending'
            WHEN category = 'Lock' AND total_wait_ms * 100.0 / NULLIF(grand_total_ms, 0) > 20
                THEN 'PRIMARY BOTTLENECK - Lock contention'
            WHEN category = 'Lock' AND total_wait_ms * 100.0 / NULLIF(grand_total_ms, 0) > 10
                THEN 'Elevated lock waits - investigate blocking'
            WHEN category = 'Memory' AND total_wait_ms * 100.0 / NULLIF(grand_total_ms, 0) > 15
                THEN 'PRIMARY BOTTLENECK - Memory pressure'
            WHEN category = 'Memory' AND total_wait_ms * 100.0 / NULLIF(grand_total_ms, 0) > 5
                THEN 'Memory pressure trending - check grants'
            ELSE 'Within normal parameters'
        END AS assessment,
        CASE
            WHEN category = 'IO' THEN 'Check: sys.dm_io_virtual_file_stats, PLE, missing indexes'
            WHEN category = 'CPU' THEN 'Check: top CPU queries, MAXDOP, signal waits'
            WHEN category = 'Lock' THEN 'Check: blocking chains, RCSI, index tuning'
            WHEN category = 'Memory' THEN 'Check: max server memory, plan cache, grants'
            WHEN category = 'Parallel' THEN 'Check: cost threshold, MAXDOP, query plans'
            WHEN category = 'TempDB' THEN 'Check: tempdb file count, sizing, allocation'
            WHEN category = 'Network' THEN 'Check: application data consumption patterns'
            ELSE NULL
        END AS investigation_steps
    FROM CategorySummary
    ORDER BY total_wait_ms DESC;
END
GO
```

## 6. Troubleshooting - Category-Specific Diagnosis

### 6.1 IO Category Dominates (Over 50% of Waits)

When IO exceeds 50% of total waits, the storage subsystem is the primary bottleneck. However, the root cause might be:

**True Storage Problem:**
- High avg read/write latency in sys.dm_io_virtual_file_stats
- PLE is healthy (over 600s)
- IO is consistently high across all databases

**Memory Pressure Masquerading as IO:**
- Low PLE (under 300s)
- High lazy writes/sec
- PAGEIOLATCH_SH dominates other IO waits
- Adding more memory would reduce the IO

**Missing Indexes Causing IO:**
- High logical reads in query stats
- Frequent table scans
- IO tied to specific user queries

```sql
-- Distinguishing IO root cause
SELECT
    (SELECT cntr_value FROM sys.dm_os_performance_counters
     WHERE counter_name = 'Page Life Expectancy') AS ple,
    (SELECT cntr_value FROM sys.dm_os_performance_counters
     WHERE counter_name = 'Lazy writes/sec') AS lazy_writes,
    (SELECT cntr_value FROM sys.dm_os_performance_counters
     WHERE counter_name = 'Page reads/sec') AS page_reads,
    CASE
        WHEN (SELECT cntr_value FROM sys.dm_os_performance_counters
              WHERE counter_name = 'Page Life Expectancy') < 300
            THEN 'IO caused by MEMORY PRESSURE'
        WHEN (SELECT AVG(runnable_tasks_count) FROM sys.dm_os_schedulers
              WHERE status = 'VISIBLE ONLINE') > 10
            THEN 'IO may be caused by CPU pressure (signal waits inflating IO)'
        ELSE 'IO likely storage subsystem issue'
    END AS root_cause_assessment;
```

### 6.2 CPU Category Dominates (Over 30% of Waits)

CPU category includes SOS_SCHEDULER_YIELD and THREADPOOL.

**SOS_SCHEDULER_YIELD** = queries voluntarily yielding the CPU after running for 4ms (the quantum). This is normal for parallel queries and OLTP workloads. Only problematic if consistently over 30% of waits.

**THREADPOOL** = all worker threads are busy. This is ALWAYS a serious condition. Means SQL Server cannot process new work.

```sql
-- CPU category deep dive
SELECT
    wait_type,
    wait_time_ms,
    waiting_tasks_count,
    wait_time_ms / NULLIF(waiting_tasks_count, 0) AS avg_per_wait_ms,
    CASE wait_type
        WHEN 'SOS_SCHEDULER_YIELD' THEN 'High SOS_SCHEDULER_YIELD might indicate: parallelism skew, CPU-bound queries, or many concurrent queries'
        WHEN 'THREADPOOL' THEN 'THREADPOOL indicates worker thread exhaustion - CRITICAL'
    END AS analysis
FROM sys.dm_os_wait_stats
WHERE wait_type IN ('SOS_SCHEDULER_YIELD', 'THREADPOOL');
```

### 6.3 Lock Category Dominates (Over 20% of Waits)

Lock waits indicate concurrency contention. The specific LCK_ wait type tells you the lock mode:

| Wait Type | Lock Mode | Typical Cause |
|---|---|---|
| LCK_M_S | Shared | Reader waiting for writer (or schema modification) |
| LCK_M_X | Exclusive | Writer waiting for another writer |
| LCK_M_U | Update | Update waiting for another update |
| LCK_M_SCH_S | Schema Stability | Schema modification lock contention |
| LCK_M_SCH_M | Schema Modification | DDL operation blocking queries |
| LCK_M_IX | Intent Exclusive | Page/index lock escalation |
| LCK_M_BU | Bulk Update | Bulk insert operations |

```sql
-- Lock category deep dive
SELECT
    wait_type,
    wait_time_ms,
    waiting_tasks_count,
    wait_time_ms / NULLIF(waiting_tasks_count, 0) AS avg_lock_wait_ms,
    CASE
        WHEN wait_type = 'LCK_M_S' THEN 'Shared lock wait - reader blocked by writer'
        WHEN wait_type = 'LCK_M_X' THEN 'Exclusive lock wait - writer blocked by another writer'
        WHEN wait_type = 'LCK_M_U' THEN 'Update lock wait - update blocked by update'
        WHEN wait_type = 'LCK_M_SCH_S' THEN 'Schema stability wait - query blocked by DDL'
        WHEN wait_type = 'LCK_M_SCH_M' THEN 'Schema modification wait - DDL blocked by query'
        WHEN wait_type = 'LCK_M_IX' THEN 'Intent exclusive wait - lock escalation issue'
        ELSE 'Other lock mode - check lock compatibility'
    END AS lock_mode_description
FROM sys.dm_os_wait_stats
WHERE wait_type LIKE 'LCK_M_%'
ORDER BY wait_time_ms DESC;
```

### 6.4 Memory Category Dominates (Over 15% of Waits)

Memory waits indicate that queries cannot get the memory grants they need.

| Wait Type | Meaning | Common Cause |
|---|---|---|
| RESOURCE_SEMAPHORE | Query waiting for memory grant | Large hash joins or sorts |
| RESOURCE_SEMAPHORE_QUERY_COMPILE | Query waiting for compile memory | Many concurrent compilations |
| CMEMTHREAD | Memory object contention | High thread count accessing memory objects |

```sql
-- Memory category deep dive
SELECT
    wait_type,
    wait_time_ms,
    waiting_tasks_count,
    CASE wait_type
        WHEN 'RESOURCE_SEMAPHORE' THEN 'Queries waiting for memory grants - check query plans for large hash joins'
        WHEN 'RESOURCE_SEMAPHORE_QUERY_COMPILE' THEN 'Compilation memory pressure - check for ad-hoc queries'
        WHEN 'CMEMTHREAD' THEN 'Memory object contention - often caused by many concurrent sessions'
    END AS analysis
FROM sys.dm_os_wait_stats
WHERE wait_type IN ('RESOURCE_SEMAPHORE', 'RESOURCE_SEMAPHORE_QUERY_COMPILE', 'CMEMTHREAD');
```

## 9. Gotchas - Common Pitfalls

### 9.1 Harmless Waits Inflate Category Totals

If you don't exclude background waits (BROKER_EVENT, SLEEP_TASK, WAITFOR, XE_TIMER_EVENT, CLR_AUTO_EVENT), the "Other" category will be large and dilute the primary categories. Always use the exclude filter shown in Section 3.1.

### 9.2 Signal Wait Inflates Every Category

Signal wait time (CPU queue time) is attributed to the wait type that the thread was waiting for. This means:
- High CPU pressure inflates ALL category percentages, not just CPU
- When CPU pressure exists, IO and Lock percentages look higher than they really are
- Solution: use `resource_wait_time_ms` (wait_time_ms - signal_wait_time_ms) for pure category analysis

### 9.3 IO Might Be Caused by Memory Pressure

The most common misdiagnosis: seeing high PAGEIOLATCH waits and concluding the storage is slow. Before blaming storage, check:
- PLE (low PLE = memory pressure forcing IO)
- Lazy writes/sec (high = memory pressure evicting pages)
- If PLE is healthy and disk latency is high, THEN blame storage

### 9.4 WRITELOG Belongs in IO Category

WRITELOG is technically a log write wait, but it belongs in the IO category for analysis purposes. A high WRITELOG percentage indicates the transaction log disk is slow or there's excessive logging activity.

### 9.5 CXPACKET and CXCONSUMER Belong in Parallel, Not CPU

CX* waits are often confused with CPU pressure. They represent parallel query coordination overhead, not CPU computation time. A query with high CXPACKET might have efficient parallelism that looks like a wait but is actually productive work being done in parallel.

### 9.6 Distinguishing Root Cause vs Symptom

| Symptom | Could Be Caused By |
|---|---|
| High IO | Slow disk, memory pressure, missing indexes |
| High CPU | Heavy queries, parallelism skew, compilation churn |
| High Lock | Missing indexes, long transactions, high concurrency |
| High Memory | Large hash joins, ad-hoc queries, low max server memory |

### 9.7 Category Analysis Requires Snapshot Comparison

A single point-in-time category analysis shows cumulative data since server start. To understand current behavior, always compare two snapshots (delta analysis). The cumulative data may be dominated by a batch job that ran 3 days ago and has since completed.

### 9.8 "Other" Category Should Be Under 5%

If the "Other" category exceeds 5% of total waits, check what waits are being categorized as "Other". They may be worth their own category (e.g., LATCH waits indicate memory object contention, which might need separate tracking).

### 9.9 Thresholds Depend on Workload Type

An OLTP system with IO at 40% might be healthy if it's a data warehouse. A data warehouse with CPU at 40% might be healthy if it's running heavy aggregations. Always baseline your specific workload before applying generic thresholds.

### 9.10 Use Resource Wait Analysis for Pure Bottleneck Detection

```sql
-- The cleanest way to find the true bottleneck
WITH ResourceOnly AS (
    SELECT
        CASE
            WHEN wait_type LIKE 'PAGEIOLATCH%' THEN 'IO'
            WHEN wait_type IN ('SOS_SCHEDULER_YIELD','THREADPOOL') THEN 'CPU'
            WHEN wait_type LIKE 'LCK%' THEN 'Lock'
            WHEN wait_type IN ('RESOURCE_SEMAPHORE','RESOURCE_SEMAPHORE_QUERY_COMPILE') THEN 'Memory'
            ELSE 'Other'
        END AS category,
        wait_time_ms - signal_wait_time_ms AS resource_wait_ms
    FROM sys.dm_os_wait_stats
    WHERE wait_type NOT IN ('BROKER_EVENT_HANDLER','SLEEP_TASK','WAITFOR')
)
SELECT
    category,
    SUM(resource_wait_ms) AS total_resource_wait_ms,
    CAST(SUM(resource_wait_ms) * 100.0 / NULLIF(SUM(SUM(resource_wait_ms)) OVER (), 0) AS DECIMAL(5,2)) AS pure_pct
FROM ResourceOnly
WHERE category IN ('IO', 'CPU', 'Lock', 'Memory')
GROUP BY category
ORDER BY total_resource_wait_ms DESC;
```

This query subtracts signal waits from every wait type, giving a "pure" resource wait percentage. If IO drops from 55% to 35% after subtracting signal waits, then CPU pressure was inflating the IO number. This is the gold standard for bottleneck detection.
