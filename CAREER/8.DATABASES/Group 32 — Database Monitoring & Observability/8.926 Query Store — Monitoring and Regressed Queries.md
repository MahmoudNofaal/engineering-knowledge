# 8.926 Query Store — Monitoring and Regressed Queries

## 1. Overview — Query Store for Performance Monitoring

Query Store is a built-in SQL Server feature that automatically captures query execution history — including query text, execution plans, and runtime statistics — and persists this data inside the user database. It acts as a "flight data recorder" for queries, enabling DBAs and developers to track performance changes over time without external tools.

The primary use cases for Query Store are:
- **Regression detection** — Identifying queries whose performance degraded after a plan change
- **Plan analysis** — Viewing all execution plans for a query and comparing their metrics
- **Plan forcing** — Pinning a known good plan when the optimizer chooses a worse plan
- **Performance troubleshooting** — Correlating performance degradation with time, plan changes, or query patterns

Query Store is fundamentally different from DMV snapshots and Extended Events:
- DMV snapshots provide cumulative, server-wide aggregates that require external storage
- Extended Events captures individual events with full detail but requires manual setup
- Query Store is per-database, automatic, and persistent — it tracks every query within the database with minimal configuration

The key architectural innovation of Query Store is that it separates query text from plans from runtime stats, enabling efficient storage and analysis. It stores:
- **Query information** (query_id, query_text, object_id) — in `sys.query_store_query`
- **Plan information** (plan_id, plan_handle, query_plan XML) — in `sys.query_store_plan`
- **Runtime statistics** (execution count, avg_duration, avg_cpu, avg_logical_reads) — in `sys.query_store_runtime_stats`

These three stores are linked by IDs, enabling complex analytical queries that would be expensive against raw DMVs.

## 2. Architecture — Query Store Data Pipeline

```
flowchart TD
    A[Query Execution] --> B[Query Optimizer]
    B --> C{Plan exists in QS?}
    C -->|No| D[Insert query_text]
    C -->|Yes| F[Use existing query_id]
    D --> E[Generate new query_id and plan_id]
    E --> G[Insert runtime_stats]
    F --> G
    G --> H[Capture: query_hash, plan_hash, parameters]
    H --> I[Query Store Internal Tables]
    I --> J[sys.query_store_query]
    I --> K[sys.query_store_plan]
    I --> L[sys.query_store_runtime_stats]
    I --> M[sys.query_store_runtime_stats_interval]
    J --> N[Query Store Catalog Views]
    K --> N
    L --> N
    N --> O[Regressed Queries Report]
    N --> P[Top Resource Consuming Queries]
    N --> Q[Tracked Queries]
```

### 2.1 Internal Architecture

Query Store uses four internal tables for storage:
- **sys.query_store_query_text** — Stores the actual SQL text. Each unique text gets one row.
- **sys.query_store_query** — Stores query-level metadata (object_id, query_hash, initial_compile_time).
- **sys.query_store_plan** — Stores execution plans. Each plan variation gets one row. Includes plan_handle, query_plan XML, and compile time.
- **sys.query_store_runtime_stats** — Stores aggregated runtime statistics per plan per time interval. Columns include avg_duration, avg_cpu_time, avg_logical_reads, avg_physical_reads, avg_rowcount, execution_count, and count of compilations.

The runtime stats are aggregated over time intervals (default 60 minutes) controlled by `INTERVAL_LENGTH_MINUTES`. Each interval groups executions of the same plan that occurred within that window.

### 2.2 Data Flow

When a query executes:
1. The optimizer generates a plan and checks Query Store
2. If the query text is new, it inserts into `query_store_query_text`
3. If the query_id does not exist, it inserts into `query_store_query`
4. If the plan is new (different plan_hash), it inserts into `query_store_plan`
5. At the end of each interval, runtime stats for each plan are aggregated and stored in `query_store_runtime_stats`

This design means Query Store writes to the user database. The writes are batched and asynchronous by default, but they still generate IO. This is why `MAX_STORAGE_SIZE_MB` is critical — if Query Store grows unbounded, it can fill the database.

## 3. Implementation — Enabling and Configuring Query Store

### 3.1 Enabling Query Store

```sql
-- Enable Query Store on a database
ALTER DATABASE [AdventureWorks]
SET QUERY_STORE = ON;

-- Enable with specific configuration
ALTER DATABASE [AdventureWorks]
SET QUERY_STORE = ON
(
    OPERATION_MODE = READ_WRITE,
    CLEANUP_POLICY = (STALE_QUERY_THRESHOLD_DAYS = 30),
    DATA_FLUSH_INTERVAL_SECONDS = 900,
    INTERVAL_LENGTH_MINUTES = 60,
    MAX_STORAGE_SIZE_MB = 1024,
    QUERY_CAPTURE_MODE = AUTO,
    SIZE_BASED_CLEANUP_MODE = AUTO,
    MAX_PLANS_PER_QUERY = 200
);
GO
```

### 3.2 Configuration Options in Detail

```sql
-- View current Query Store configuration
SELECT 
    actual_state_desc,
    desired_state_desc,
    readonly_reason,
    current_storage_size_mb,
    max_storage_size_mb,
    flush_interval_seconds,
    interval_length_minutes,
    stale_query_threshold_days,
    max_plans_per_query,
    query_capture_mode_desc,
    size_based_cleanup_mode_desc
FROM sys.database_query_store_options;
```

#### OPERATION_MODE

| Mode | Description |
|------|-------------|
| READ_WRITE | Query Store collects and stores data (normal operation) |
| READ_ONLY | Query Store is read-only. Data is not collected. May auto-transition when storage is full |
| OFF | Query Store is disabled |

When `MAX_STORAGE_SIZE_MB` is reached and `SIZE_BASED_CLEANUP_MODE` is AUTO, Query Store transitions to READ_ONLY. It will purge old data and transition back to READ_WRITE automatically.

#### QUERY_CAPTURE_MODE

| Mode | Description |
|------|-------------|
| ALL | Captures all queries. Highest storage required |
| AUTO | Captures queries based on execution count and resource consumption. Ignores infrequent queries |
| NONE | Stops capturing new queries. Existing data is still queryable |

AUTO is the recommended mode for production. It captures queries that matter while ignoring one-off ad-hoc queries that would bloat storage.

#### MAX_STORAGE_SIZE_MB

The maximum size Query Store can use within the database. Default is 1024 MB. Setting this too high can starve the database of space. Setting it too low causes frequent READ_ONLY transitions.

#### INTERVAL_LENGTH_MINUTES

The aggregation interval for runtime statistics. Default is 60 minutes.

| Interval | Storage | Granularity |
|----------|---------|-------------|
| 15 minutes | Higher | Fine-grained trend analysis |
| 60 minutes | Medium | Standard for most workloads |
| 1440 minutes (daily) | Low | Long-term trends only |

#### CLEANUP_POLICY

Controls how long data is retained. `STALE_QUERY_THRESHOLD_DAYS` defaults to 30. Reduce for busy OLTP systems, increase for development environments.

### 3.3 Enabling on All Databases via Policy

For SQL Server 2017+ with Enterprise Edition:

```sql
-- Enable Query Store on all user databases
DECLARE @sql NVARCHAR(MAX) = '';

SELECT @sql = @sql + '
    ALTER DATABASE [' + name + '] 
    SET QUERY_STORE = ON
    (
        OPERATION_MODE = READ_WRITE,
        CLEANUP_POLICY = (STALE_QUERY_THRESHOLD_DAYS = 30),
        DATA_FLUSH_INTERVAL_SECONDS = 900,
        INTERVAL_LENGTH_MINUTES = 60,
        MAX_STORAGE_SIZE_MB = 1024,
        QUERY_CAPTURE_MODE = AUTO,
        SIZE_BASED_CLEANUP_MODE = AUTO,
        MAX_PLANS_PER_QUERY = 200
    );
'
FROM sys.databases
WHERE database_id > 4
    AND state = 0
    AND is_query_store_on = 0;

EXEC sp_executesql @sql;
```

## 4. Configuration — Query Store Sizing and Management

### 4.1 Storage Sizing Guidance

Storage requirements depend on query volume, plan churn, and retention:

```sql
-- Estimate Query Store size
SELECT 
    CAST(MAX(current_storage_size_mb) AS DECIMAL(12,2)) AS CurrentSizeMB,
    CAST(MAX(max_storage_size_mb) AS DECIMAL(12,2)) AS MaxSizeMB,
    COUNT(DISTINCT qsq.query_id) AS TotalQueries,
    COUNT(DISTINCT qsqp.plan_id) AS TotalPlans,
    COUNT(DISTINCT qsrs.runtime_stats_id) AS TotalStatsRows
FROM sys.database_query_store_options dqso
CROSS JOIN sys.query_store_query qsq
CROSS JOIN sys.query_store_plan qsqp
CROSS JOIN sys.query_store_runtime_stats qsrs;
```

General sizing rules:
- **100 MB base** for the Query Store itself
- **20 KB per query** tracked
- **10 KB per plan** tracked
- **50 bytes per runtime stats interval** per plan

For a database with 10,000 queries and 2 plans each, over 30 days (720 hourly intervals):
- Base: 100 MB
- Queries: 10,000 x 20 KB = 200 MB
- Plans: 20,000 x 10 KB = 200 MB
- Stats: 20,000 x 720 x 50 bytes = 720 MB

Total: approximately 1.2 GB. Set `MAX_STORAGE_SIZE_MB` to at least 2048 for such workloads.

### 4.3 Managing Query Store Space

```sql
-- Check current space usage
SELECT 
    current_storage_size_mb,
    max_storage_size_mb,
    current_storage_size_mb * 100.0 / NULLIF(max_storage_size_mb, 0) AS PctUsed
FROM sys.database_query_store_options;

-- Manually purge old data
ALTER DATABASE [AdventureWorks]

-- Or change retention to force cleanup
ALTER DATABASE [AdventureWorks]

```

### 4.4 Monitoring Query Store Health

```sql
-- Query Store state
SELECT 
    actual_state_desc,
    readonly_reason,
    current_storage_size_mb,
    max_storage_size_mb,
    query_capture_mode_desc,
    size_based_cleanup_mode_desc
FROM sys.database_query_store_options;

-- Reasons for READ_ONLY state
SELECT 
    readonly_reason,
    CASE readonly_reason
        WHEN 0 THEN 'Read-write'
        WHEN 1 THEN 'Database is in read-only mode'
        WHEN 2 THEN 'Database is in single-user mode'
        WHEN 4 THEN 'Database is in emergency mode'
        WHEN 8 THEN 'Database is AG secondary replica'
        WHEN 65536 THEN 'MAX_STORAGE_SIZE_MB reached'
        WHEN 131072 THEN 'Size-based cleanup triggered, transitioning to read-write'
        WHEN 262144 THEN 'In-memory data size limit reached'
        ELSE 'Unknown'
    END AS ReasonDescription
FROM sys.database_query_store_options;

-- Query execution count vs capture count
SELECT 
    COUNT(*) AS TotalCapturedQueries,
    SUM(COUNT(*)) OVER() AS TotalExecutions
FROM sys.query_store_runtime_stats;
```

## 5. Query Patterns — Regression Detection

### 5.1 Comparing Average Duration Between Two Time Periods

This is the core regression detection pattern. Compare runtime statistics from two time windows:

```sql
DECLARE @Period1Start DATETIME2 = '2026-06-26 08:00:00';
DECLARE @Period1End DATETIME2 = '2026-06-26 09:00:00';
DECLARE @Period2Start DATETIME2 = '2026-06-27 08:00:00';
DECLARE @Period2End DATETIME2 = '2026-06-27 09:00:00';

WITH Period1 AS (
    SELECT 
        qsq.query_id,
        qsqp.plan_id,
        COUNT(qsrs.runtime_stats_id) AS IntervalsWithData,
        SUM(qsrs.count_executions) AS TotalExecutions,
        AVG(qsrs.avg_duration) AS AvgDurationPerInterval -- microseconds
    FROM sys.query_store_query qsq
    JOIN sys.query_store_plan qsqp ON qsqp.query_id = qsq.query_id
    JOIN sys.query_store_runtime_stats qsrs ON qsrs.plan_id = qsqp.plan_id
    JOIN sys.query_store_runtime_stats_interval qsrsi 
        ON qsrs.runtime_stats_interval_id = qsrsi.runtime_stats_interval_id
    WHERE qsrsi.start_time >= @Period1Start
        AND qsrsi.end_time <= @Period1End
        AND qsrs.execution_type_desc = 'Regular'
    GROUP BY qsq.query_id, qsqp.plan_id
    HAVING SUM(qsrs.count_executions) > 10  -- Minimum sample size
),
Period2 AS (
    SELECT 
        qsq.query_id,
        qsqp.plan_id,
        COUNT(qsrs.runtime_stats_id) AS IntervalsWithData,
        SUM(qsrs.count_executions) AS TotalExecutions,
        AVG(qsrs.avg_duration) AS AvgDurationPerInterval
    FROM sys.query_store_query qsq
    JOIN sys.query_store_plan qsqp ON qsqp.query_id = qsq.query_id
    JOIN sys.query_store_runtime_stats qsrs ON qsrs.plan_id = qsqp.plan_id
    JOIN sys.query_store_runtime_stats_interval qsrsi 
        ON qsrs.runtime_stats_interval_id = qsrsi.runtime_stats_interval_id
    WHERE qsrsi.start_time >= @Period2Start
        AND qsrsi.end_time <= @Period2End
        AND qsrs.execution_type_desc = 'Regular'
    GROUP BY qsq.query_id, qsqp.plan_id
    HAVING SUM(qsrs.count_executions) > 10
)
SELECT 
    p2.query_id,
    p2.plan_id AS CurrentPlanId,
    p1.plan_id AS PreviousPlanId,
    p1.AvgDurationPerInterval AS PreviousAvgDuration,
    p2.AvgDurationPerInterval AS CurrentAvgDuration,
    (p2.AvgDurationPerInterval - p1.AvgDurationPerInterval) AS Delta,
    CASE 
        WHEN p1.AvgDurationPerInterval = 0 THEN 100.0
        ELSE ((p2.AvgDurationPerInterval - p1.AvgDurationPerInterval) * 100.0) / p1.AvgDurationPerInterval
    END AS PctChange,
    p2.TotalExecutions AS RecentExecutions,
    qt.query_sql_text
FROM Period2 p2
JOIN Period1 p1 ON p2.query_id = p1.query_id
JOIN sys.query_store_query qsq ON qsq.query_id = p2.query_id
JOIN sys.query_store_query_text qt ON qt.query_text_id = qsq.query_text_id
WHERE p2.AvgDurationPerInterval > p1.AvgDurationPerInterval * 1.5  -- 50% regression
    AND p2.AvgDurationPerInterval > 50000  -- > 50ms average
ORDER BY Delta DESC;
```

### 5.2 Top Resource Consuming Queries

```sql
-- Queries with highest total duration in last 24 hours
SELECT TOP 20
    qt.query_sql_text,
    qsq.query_id,
    MAX(qsqp.plan_id) AS CurrentPlanId,
    SUM(qsrs.count_executions) AS TotalExecutions,
    AVG(qsrs.avg_duration) AS AvgDurationMicroseconds,
    AVG(qsrs.avg_cpu_time) AS AvgCpuTime,
    AVG(qsrs.avg_logical_io_reads) AS AvgLogicalReads,
    AVG(qsrs.avg_physical_io_reads) AS AvgPhysicalReads,
    AVG(qsrs.avg_rowcount) AS AvgRowCount,
    SUM(qsrs.count_executions * qsrs.avg_duration) / 1000000.0 AS TotalDurationSeconds
FROM sys.query_store_runtime_stats qsrs
JOIN sys.query_store_runtime_stats_interval qsrsi 
    ON qsrs.runtime_stats_interval_id = qsrsi.runtime_stats_interval_id
    AND qsrsi.start_time > DATEADD(HOUR, -24, GETUTCDATE())
JOIN sys.query_store_plan qsqp ON qsrs.plan_id = qsqp.plan_id
JOIN sys.query_store_query qsq ON qsqp.query_id = qsq.query_id
JOIN sys.query_store_query_text qt ON qt.query_text_id = qsq.query_text_id
WHERE qsrs.execution_type_desc = 'Regular'
GROUP BY qt.query_sql_text, qsq.query_id
ORDER BY TotalDurationSeconds DESC;
```

### 5.3 Queries with Multiple Plans (Plan Churn)

```sql
-- Queries that have changed plans recently
SELECT 
    qt.query_sql_text,
    qsq.query_id,
    COUNT(DISTINCT qsqp.plan_id) AS PlanCount,
    MIN(qsqp.initial_compile_start_time) AS FirstPlanCompile,
    MAX(qsqp.initial_compile_start_time) AS LastPlanCompile,
    STRING_AGG(CAST(qsqp.plan_id AS VARCHAR(10)), ', ') AS PlanIds
FROM sys.query_store_query qsq
JOIN sys.query_store_query_text qt ON qt.query_text_id = qsq.query_text_id
JOIN sys.query_store_plan qsqp ON qsqp.query_id = qsq.query_id
GROUP BY qt.query_sql_text, qsq.query_id
HAVING COUNT(DISTINCT qsqp.plan_id) > 3
ORDER BY PlanCount DESC;
```

### 5.4 Track a Specific Query Over Time

```sql
DECLARE @QueryId INT = 12345;  -- From sys.query_store_query

SELECT 
    qsrsi.start_time,
    qsrsi.end_time,
    qsqp.plan_id,
    qsqp.query_plan_hash,
    qsrs.count_executions,
    qsrs.avg_duration,
    qsrs.avg_cpu_time,
    qsrs.avg_logical_io_reads,
    qsrs.avg_physical_io_reads,
    qsrs.avg_rowcount,
    qsrs.max_duration,
    qsrs.min_duration,
    qsrs.stdev_duration
FROM sys.query_store_runtime_stats qsrs
JOIN sys.query_store_runtime_stats_interval qsrsi 
    ON qsrs.runtime_stats_interval_id = qsrsi.runtime_stats_interval_id
JOIN sys.query_store_plan qsqp ON qsrs.plan_id = qsqp.plan_id
WHERE qsqp.query_id = @QueryId
    AND qsrs.execution_type_desc = 'Regular'
ORDER BY qsrsi.start_time DESC;
```

## 6. Production Deployment — Operational Considerations

### 6.1 High Availability Impact

On Always On Availability Group secondaries, Query Store operates differently:

```sql
-- Check Query Store state on replica
SELECT 
    database_id,
    is_query_store_on,
    actual_state_desc,
    is_primary_replica,
    secondary_role_query_store_mode_desc
FROM sys.dm_database_replica_states
JOIN sys.database_query_store_options 
    ON sys.dm_database_replica_states.database_id = sys.database_query_store_options.database_id;
```

On secondaries, Query Store can be configured to run in one of three modes:
- **NONE** — Query Store is disabled on the replica (default)
- **ALL** — Query Store captures all queries on the replica
- **AUTO** — Query Store captures automatically

Recommendation: leave the default (NONE) on secondaries to avoid unnecessary writes.

### 6.2 Query Store Failover Behavior

When a failover occurs, Query Store data on the primary is available on the new primary. Data on the old primary (now secondary) with `SECONDARY_ROLE = NONE` will be empty. The time window during the failover may have a gap in Query Store data.

### 6.3 Impact on Database Operations

```sql
-- Monitor Query Store write IO
SELECT 
    DB_NAME(database_id) AS DatabaseName,
    SUM(io_stall_write_ms) AS TotalWriteStallMs,
    SUM(num_of_bytes_written) / 1048576 AS TotalWrittenMB
FROM sys.dm_io_virtual_file_stats(NULL, NULL)
WHERE database_id = DB_ID('AdventureWorks')
GROUP BY database_id;

-- Check Query Store waits
SELECT 
    wait_type,
    wait_time_ms,
    waiting_tasks_count
FROM sys.dm_os_wait_stats
WHERE wait_type LIKE '%QDS%'
ORDER BY wait_time_ms DESC;
```

### 6.4 Best Practice Configuration for Production

```sql
-- Recommended production configuration
ALTER DATABASE [ProductionDB] SET QUERY_STORE = ON
(
    OPERATION_MODE = READ_WRITE,
    CLEANUP_POLICY = (STALE_QUERY_THRESHOLD_DAYS = 30),
    DATA_FLUSH_INTERVAL_SECONDS = 900,
    INTERVAL_LENGTH_MINUTES = 60,
    MAX_STORAGE_SIZE_MB = 2048,
    QUERY_CAPTURE_MODE = AUTO,
    SIZE_BASED_CLEANUP_MODE = AUTO,
    MAX_PLANS_PER_QUERY = 200
);
```

After enabling, monitor for the first week:
- Check storage growth daily
- Validate no READ_ONLY transitions
- Review captured queries (ensure critical queries are included)
- Adjust `MAX_STORAGE_SIZE_MB` if needed

## 7. Monitoring & Maintenance — Plan Forcing

### 7.1 When to Force a Plan

Plan forcing should be used when:
- A query regressed after an optimizer plan change
- The previous plan was demonstrably faster (verified with actual execution plans)
- The regression is consistent across multiple executions (not parameter sniffing)

```sql
-- Check if a plan was forced and its status
SELECT 
    qt.query_sql_text,
    qsq.query_id,
    qsqp.plan_id,
    qsqp.is_forced_plan,
    qsqp.force_failure_count,
    qsqp.last_force_failure_reason_desc
FROM sys.query_store_query qsq
JOIN sys.query_store_query_text qt ON qt.query_text_id = qsq.query_text_id
JOIN sys.query_store_plan qsqp ON qsqp.query_id = qsq.query_id
WHERE qsqp.is_forced_plan = 1;
```

### 7.2 Forcing a Plan

```sql
-- Force a specific plan for a query
EXEC sp_query_store_force_plan 
    @query_id = 12345,
    @plan_id = 67890;
GO

-- Verify the plan is forced
SELECT 
    qt.query_sql_text,
    qsq.query_id,
    qsqp.plan_id,
    qsqp.is_forced_plan
FROM sys.query_store_plan qsqp
JOIN sys.query_store_query qsq ON qsqp.query_id = qsq.query_id
JOIN sys.query_store_query_text qt ON qt.query_text_id = qsq.query_text_id
WHERE qsq.query_id = 12345 AND qsqp.plan_id = 67890;
```

### 7.3 Unforcing a Plan

```sql
-- Unforce a plan
EXEC sp_query_store_unforce_plan
    @query_id = 12345,
    @plan_id = 67890;

-- Or unforce all forced plans
DECLARE @sql NVARCHAR(MAX) = '';
SELECT @sql = @sql + 
    'EXEC sp_query_store_unforce_plan @query_id = ' + 
    CAST(qsq.query_id AS VARCHAR(20)) + ', @plan_id = ' + 
    CAST(qsqp.plan_id AS VARCHAR(20)) + ';'
FROM sys.query_store_plan qsqp
JOIN sys.query_store_query qsq ON qsqp.query_id = qsq.query_id
WHERE qsqp.is_forced_plan = 1;
EXEC sp_executesql @sql;
```

### 7.4 Monitoring Forced Plan Failures

```sql
-- Plans that failed to be forced
SELECT 
    qt.query_sql_text,
    qsq.query_id,
    qsqp.plan_id,
    qsqp.force_failure_count,
    qsqp.last_force_failure_reason_desc,
    qsqp.last_execution_time
FROM sys.query_store_plan qsqp
JOIN sys.query_store_query qsq ON qsqp.query_id = qsq.query_id
JOIN sys.query_store_query_text qt ON qt.query_text_id = qsq.query_text_id
WHERE qsqp.force_failure_count > 0;
```

Common force failure reasons:
- **NO_PLAN** — The forced plan's shape is no longer valid (index changed, statistics updated)
- **NOT_FORCEABLE** — The plan depends on features that prevent forcing
- **PARAMETERIZATION** — Forcing failed due to parameterization differences

When a forced plan fails, the query reverts to optimization. The failure counter increments. After too many failures, auto-cleanup may remove the plan from Query Store.

## 8. Troubleshooting — Common Issues

### 8.1 Query Store In Read-Only Mode

The most common issue is Query Store transitioning to READ_ONLY due to storage limits:

```sql
-- Diagnose READ_ONLY state
SELECT 
    actual_state_desc,
    readonly_reason,
    current_storage_size_mb,
    max_storage_size_mb,
    current_storage_size_mb * 100.0 / NULLIF(max_storage_size_mb, 0) AS StoragePct,
    query_capture_mode_desc
FROM sys.database_query_store_options;
```

Solutions:
```sql
-- Solution 1: Increase max storage
ALTER DATABASE [AdventureWorks]
SET QUERY_STORE (MAX_STORAGE_SIZE_MB = 4096);

-- Solution 2: Clear Query Store (loses all data)
ALTER DATABASE [AdventureWorks]
SET QUERY_STORE CLEAR;

-- Solution 3: Reduce retention to force cleanup
ALTER DATABASE [AdventureWorks]
SET QUERY_STORE (CLEANUP_POLICY = (STALE_QUERY_THRESHOLD_DAYS = 7));

-- Solution 4: Set size-based cleanup to auto
ALTER DATABASE [AdventureWorks]
SET QUERY_STORE (SIZE_BASED_CLEANUP_MODE = AUTO);
```

### 8.2 Query Store Impact on Database IO

Query Store writes to the user database. In extreme cases, it can generate significant IO:

```sql
-- Monitor Query Store write activity
SELECT 
    DB_NAME(database_id) AS DatabaseName,
    FILE_ID,
    num_of_writes,
    num_of_bytes_written / 1048576 AS TotalWrittenMB,
    io_stall_write_ms,
    io_stall_write_ms / NULLIF(num_of_writes, 0) AS AvgWriteStallMs
FROM sys.dm_io_virtual_file_stats(DB_ID('AdventureWorks'), NULL);
```

If Query Store IO is too high:
- Increase `DATA_FLUSH_INTERVAL_SECONDS` (less frequent writes)
- Reduce `INTERVAL_LENGTH_MINUTES` (fewer aggregation points)
- Use `QUERY_CAPTURE_MODE = AUTO` to capture fewer queries

### 8.3 Queries Not Appearing in Query Store

If a query is not captured:
- Check `QUERY_CAPTURE_MODE` — AUTO mode ignores infrequent queries
- Verify Query Store is in READ_WRITE mode
- Check if the query is ad-hoc and below the AUTO capture threshold

```sql
-- Force capture of a specific query
ALTER DATABASE [AdventureWorks]
SET QUERY_STORE (QUERY_CAPTURE_MODE = ALL);
-- Run the query
ALTER DATABASE [AdventureWorks]
SET QUERY_STORE (QUERY_CAPTURE_MODE = AUTO);
```

### 8.4 Plan Forcing Not Working

```sql
-- Investigate force failure
SELECT 
    qt.query_sql_text,
    qsq.query_id,
    qsqp.plan_id,
    qsqp.is_forced_plan,
    qsqp.force_failure_count,
    qsqp.last_force_failure_reason_desc,
    qsqp.plan_forcing_type_desc
FROM sys.query_store_plan qsqp
JOIN sys.query_store_query qsq ON qsqp.query_id = qsq.query_id
JOIN sys.query_store_query_text qt ON qt.query_text_id = qsq.query_text_id
WHERE qsqp.is_forced_plan = 1;
```

Common fixes:
- Drop and recreate the index the plan depends on
- Update statistics to make the plan valid again
- Unforce and re-force the plan after index maintenance

### 8.5 Query Store on TempDB

Query Store cannot be enabled on tempdb in SQL Server 2016/2017. SQL Server 2019+ supports it, but it is not recommended due to the transient nature of tempdb objects.

### 8.6 Database Migration and Restore

When restoring a database with Query Store enabled:
```sql
-- Check Query Store state after restore
SELECT 
    actual_state_desc,
    current_storage_size_mb
FROM sys.database_query_store_options;

-- If in READ_ONLY after restore, clear and re-enable
ALTER DATABASE [RestoredDB]
SET QUERY_STORE CLEAR;
ALTER DATABASE [RestoredDB]
SET QUERY_STORE (OPERATION_MODE = READ_WRITE);
```

## 9. Summary — Key Takeaways

Query Store is the most important performance feature added to SQL Server in the last decade. It solves the fundamental problem of tracking query performance over time without external infrastructure.

| Component | Best Practice |
|-----------|---------------|
| Enable | All production databases, OLTP and DW |
| Capture Mode | AUTO (balances detail vs storage) |
| Storage | 2048 MB for most databases, monitor growth |
| Retention | 30 days for production, 90 for dev |
| Interval | 60 minutes for most workloads |
| Plan Forcing | Use sparingly, monitor failures |
| High Availability | Disable on replicas |
| Monitoring | Watch for READ_ONLY transitions and storage growth |

The regression detection pattern is the primary value driver:
1. Enable Query Store with appropriate sizing
2. Monitor `sys.query_store_runtime_stats` for duration spikes
3. Identify queries with plan changes and regression
4. Force the known good plan if needed
5. Investigate why the optimizer chose a worse plan (stats, indexes, parameter sniffing)

Query Store does not replace DMV baselines or Extended Events. Together, these three tools form a complete observability stack:
- **Query Store** — Per-database, always-on, plan-level tracking
- **DMV Baselines** — Server-wide trend analysis, wait statistics
- **Extended Events** — Deep-dive individual query capture on demand

When a query regresses and you need to understand why, the workflow is: Query Store identifies the regressed query → DMV baselines show what else changed (waits, IO) → Extended Events captures the actual execution for detailed analysis.
