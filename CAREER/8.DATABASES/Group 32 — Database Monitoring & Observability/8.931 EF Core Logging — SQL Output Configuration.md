# 8.931 EF Core Logging — SQL Output Configuration

Priority: 2 | Domain: Databases | Group: Database Monitoring & Observability

---

## Overview — Introduction & Purpose

Entity Framework Core generates SQL queries from LINQ expressions, LINQ method chains, and raw SQL commands. Understanding what SQL is actually executed is essential for debugging performance issues, validating query behavior, optimizing database interactions, and ensuring that the LINQ code produces the expected T-SQL.

EF Core provides multiple mechanisms for capturing generated SQL output:

**LogTo** — Simple, built-in logging to a delegate (Console.WriteLine, file, or any TextWriter). Introduced in EF Core 5.0 as a streamlined logging API on the DbContext options builder. Best for quick development debugging and minimal configuration.

**ToQueryString()** — Development-time convenience method on IQueryable<T> that returns the SQL query WITHOUT executing it. Best for debugging and query inspection during development when you want to see SQL without a database round-trip.

**ILoggerFactory integration** — Connects EF Core logging to the .NET logging infrastructure (ILogger), enabling structured logging, log levels, filtering, and integration with logging frameworks (Serilog, NLog, Application Insights). This is the standard approach for production applications.

**IDbCommandInterceptor** — Low-level ADO.NET interception that captures SQL at the command execution boundary — before execution, after execution, on failure. This is the most powerful approach for production monitoring because it provides execution time, parameter values, and error details. It operates at the ADO.NET layer, making it independent of whether EF Core or Dapper is used.

This note covers all four approaches, their use cases, configuration, filtering, and the production considerations and gotchas associated with each. It is part of the Database Monitoring & Observability group and relates to [[8.932 — Dapper Logging — MiniProfiler Integration]] (Dapper-level SQL logging), [[8.933 — MiniProfiler — ASP.NET Core Integration]] (minimal profiling for ASP.NET Core applications), and [[8.930 — Application Insights — SQL Dependency Tracking]] (automatic SQL dependency collection at the ADO.NET layer).

The prerequisite [[3.001 — DbContext and Change Tracking Fundamentals]] covers the foundational DbContext concepts needed to understand where SQL generation fits in the EF Core pipeline. [[3.010 — EF Core — Logging and Interception]] provides additional depth on the interception framework.

## Architecture — SQL Generation Pipeline in EF Core

Understanding where SQL logging and interception fit into the EF Core pipeline is essential for choosing the right approach and diagnosing logging issues.

```
┌───────────────────────────────────────────────────────────────────────────┐
│                    EF Core SQL Generation Pipeline                        │
│                                                                           │
│  LINQ Query (C#)                                                          │
│    │                                                                      │
│    ▼                                                                      │
│  Expression Tree (System.Linq.Expressions)                                │
│    │  ── Built from LINQ operators (Where, Select, Join, GroupBy, etc.)  │
│    ▼                                                                      │
│  IQueryable  ── ToQueryString() captures SQL at this stage (no execution) │
│    │                                                                      │
│    ▼                                                                      │
│  Query Compilation (EF Core Internal)                                     │
│    │  ── Evaluates client-side expressions                                │
│    │  ── Translates expression tree to SQL                                │
│    │  ── Caches compiled query model                                      │
│    ▼                                                                      │
│  SQL Statement (string)                                                   │
│    │                                                                      │
│    ▼                                                                      │
│  Parameterization                                                         │
│    │  ── Converts constants to parameters (@p0, @p1, ...)                 │
│    │  ── EnableSensitiveDataLogging shows actual parameter values         │
│    ▼                                                                      │
│  IDbCommandInterceptor.BeforeCommandExecution                             │
│    │  ── Raw SQL + parameters visible at this stage                       │
│    │  ── Can suppress execution or modify command                         │
│    │  ── LogTo/ILoggerFactory also fire at this stage                     │
│    ▼                                                                      │
│  ADO.NET Command Execution (SqlCommand.ExecuteReader/NonQuery/Scalar)     │
│    │                                                                      │
│    ▼                                                                      │
│  IDbCommandInterceptor.AfterCommandExecution                              │
│    │  ── Duration, result, and errors available after execution           │
│    ▼                                                                      │
│  Materialization (read data reader into objects)                          │
│    │                                                                      │
│    ▼                                                                      │
│  Results returned to LINQ expression                                      │
│                                                                           │
└───────────────────────────────────────────────────────────────────────────┘
```

### Where Each Logging Approach Intercepts

| Approach | Pipeline Stage | Database Execution | Best For |
|----------|---------------|-------------------|----------|
| ToQueryString() | After query compilation | No | Development, unit tests, query inspection |
| LogTo | Before/After command execution | Yes | Quick debugging, simple file logging |
| ILoggerFactory | Before/After command execution | Yes | Production structured logging |
| IDbCommandInterceptor | At ADO.NET boundary | Yes | Production monitoring, custom metrics |

### Log Event Categories and IDs

EF Core categorizes log events using the `EventId` struct and `DbLoggerCategory` constants. The categories relevant to SQL logging are:

| Event ID | Name Constant | Category | Description | Level |
|---------|--------------|----------|-------------|-------|
| 20100 | RelationalEventId.CommandExecuting | Database.Command | Before SQL execution | Information |
| 20101 | RelationalEventId.CommandExecuted | Database.Command | After SQL execution | Information |
| 20102 | RelationalEventId.CommandError | Database.Command | SQL execution failed | Error |
| 20103 | RelationalEventId.ConnectionOpening | Database.Connection | Opening connection | Debug |
| 20104 | RelationalEventId.ConnectionOpened | Database.Connection | Connection opened | Debug |
| 20105 | RelationalEventId.ConnectionClosing | Database.Connection | Closing connection | Debug |
| 20106 | RelationalEventId.ConnectionClosed | Database.Connection | Connection closed | Debug |
| 20107 | RelationalEventId.TransactionStarted | Database.Transaction | Transaction begin | Debug |
| 20108 | RelationalEventId.TransactionCommitted | Database.Transaction | Transaction commit | Debug |
| 20109 | RelationalEventId.TransactionRolledBack | Database.Transaction | Transaction rollback | Debug |
| 20110 | RelationalEventId.TransactionDisposed | Database.Transaction | Transaction disposed | Debug |

The recommended filtering approach uses `DbLoggerCategory.Database.Command.Name` which resolves to the string `"Microsoft.EntityFrameworkCore.Database.Command"`. This category covers events 20100-20102 (command execution) without including connection and transaction noise.

## LogTo — Simple Built-in SQL Logging

`LogTo` was introduced in EF Core 5.0 as a lightweight logging API. It is the simplest way to see generated SQL.

### Basic Usage

```csharp
public class ApplicationDbContext : DbContext
{
    protected override void OnConfiguring(DbContextOptionsBuilder optionsBuilder)
    {
        optionsBuilder
            .UseSqlServer(connectionString)
            .LogTo(Console.WriteLine, LogLevel.Information);
    }
}
```

With this configuration, every SQL command executed by the DbContext is printed to Console. The output includes the log level, timestamp, event ID, category, and the SQL command with duration.

Example output:

```
info: 6/27/2026 14:30:45.123 RelationalEventId.CommandExecuted[20101]
      (Microsoft.EntityFrameworkCore.Database.Command)
      Executed DbCommand (42ms) [Parameters=[@p0='123' (Size = 0)],
      CommandType='Text', CommandTimeout='30']
      SELECT [u].[Id], [u].[Name], [u].[Email]
      FROM [Users] AS [u]
      WHERE [u].[Id] = @p0
```

### Logging to a File

```csharp
using StreamWriter logStream = new StreamWriter("ef-log.txt", append: true);

optionsBuilder.LogTo(
    logStream.WriteLine,
    LogLevel.Information,
    DbContextLoggerCategory.Database.Command.Name
);
```

For production environments, log rotation is critical. Never log to a single file indefinitely. Prefer structured logging with Serilog rolling file sink for production file-based logging.

### Filtering by Event ID or Category

Filter by category string:

```csharp
optionsBuilder.LogTo(
    Console.WriteLine,
    LogLevel.Information,
    DbContextLoggerCategory.Database.Command.Name
);
```

Filter by specific event IDs:

```csharp
optionsBuilder.LogTo(
    Console.WriteLine,
    LogLevel.Information,
    [RelationalEventId.CommandExecuted, RelationalEventId.CommandError]
);
```

This approach explicitly includes only the event IDs you care about. `RelationalEventId.CommandExecuted` (20101) is the standard SQL execution event. Include `RelationalEventId.CommandError` (20102) to capture failures.

Use event IDs from:
- `CoreEventId` — EF Core infrastructure events (model building, context initialization, change tracking)
- `RelationalEventId` — Relational (SQL) events (command execution, transaction, connection)
- `SqlServerEventId` — SQL Server-specific events
- `ProxiesEventId` — Proxy creation events (if using lazy loading proxies)

### Controlling Output Format

```csharp
optionsBuilder.LogTo(
    Console.WriteLine,
    LogLevel.Information,
    DbContextLoggerCategory.Database.Command.Name,
    options: DbContextLoggerOptions.SingleLine | DbContextLoggerOptions.UtcTime
);
```

`DbContextLoggerOptions` flags:

| Flag | Effect |
|------|--------|
| None | Default multi-line format with event ID, time, and category |
| SingleLine | Everything on one line, suitable for log aggregation tools |
| UtcTime | Use UTC timestamps instead of local time |
| LocalTime | Use local timestamps (default) |
| WithDbName | Include the database name |
| WithCategory | Include the log category (default) |
| WithEventId | Include the numeric event ID (default) |

### LogTo Limitations

- **No structured logging**: The output is a formatted string. Structured logging frameworks (Serilog, NLog) work better with ILoggerFactory integration for production scenarios.
- **No execution time filtering**: Cannot easily filter to show only slow queries without custom handling in the callback delegate.
- **No parameter value inspection**: Without EnableSensitiveDataLogging, parameter values show as `@p0='?'`. With it enabled, values are visible but formatting is simplistic and may expose PII.
- **Not suitable for production at scale**: File I/O on every SQL execution adds overhead. Use interceptors or structured logging for production monitoring.
- **Single callback**: Cannot route different event categories to different outputs (e.g., SQL to file, errors to console) without complex callback logic.

## ToQueryString() — Development-Time SQL Inspection

`ToQueryString()` is an extension method on `IQueryable<T>` that returns the SQL representation of the query without executing it against the database.

```csharp
IQueryable<User> query = context.Users
    .Where(u => u.IsActive && u.LastLogin < oneMonthAgo)
    .OrderBy(u => u.Name);

string sql = query.ToQueryString();
Console.WriteLine(sql);
```

Output:

```
SELECT [u].[Id], [u].[Name], [u].[Email], [u].[LastLogin]
FROM [Users] AS [u]
WHERE [u].[IsActive] = CAST(1 AS bit) AND [u].[LastLogin] < @p0
ORDER BY [u].[Name]
```

### Key Characteristics

- **No database round-trip**: The query is never executed. This makes it safe to use in development against any query, including potentially destructive ones.
- **Captures SQL with parameter placeholders**: Parameters are shown as `@p0`, `@p1`, etc. Parameter values are NOT shown unless EnableSensitiveDataLogging is active.
- **Works after query building**: Call ToQueryString() at any point in the query chain before materialization methods (ToList, FirstOrDefault, Count, etc.).
- **Shows only server-side SQL**: If the query includes client-side evaluation (e.g., in-memory filtering after ToList), the SQL shows only what is sent to the database.
- **Requires EF Core 5.0+**: Not available in EF Core 3.x or earlier.
- **Ignores query filters**: Model-level query filters (HasQueryFilter) ARE included in the output.
- **Includes split query behavior**: For split query configurations, only the first query is shown.

### Best Practices for Development

**Use in unit tests to verify LINQ produces correct SQL:**

```csharp
[Fact]
public void ActiveUsersQuery_GeneratesCorrectSql()
{
    using var context = CreateTestContext();
    IQueryable<User> query = context.Users.Where(u => u.IsActive);

    string sql = query.ToQueryString();
    Assert.Contains("WHERE", sql);
    Assert.Contains("[u].[IsActive]", sql);
}
```

**Inspect SQL during debugging:**

Add a watch or use the Immediate Window: `query.ToQueryString()`. This is faster than running the query and checking SQL Server Profiler.

**Compare SQL patterns:**

```csharp
string sql1 = context.Users.Where(u => u.IsActive).ToQueryString();
string sql2 = context.Users.Where(u => u.IsActive == true).ToQueryString();
Console.WriteLine($"Pattern 1: {sql1}");
Console.WriteLine($"Pattern 2: {sql2}");
```

### Limitations of ToQueryString()

- **No execution plan**: Returns the SQL text only. Does not show the actual execution plan or query performance.
- **No actual parameter values**: Shows parameterized SQL with placeholders. Use EnableSensitiveDataLogging with LogTo for inline values.
- **Only single query**: For split query scenarios (multiple result sets), only the first query is displayed.
- **Development only**: Not suitable for production monitoring as it requires code changes to use.
- **Does not work with query execution**: Cannot be called after the query is executed (on materialized results).

## ILoggerFactory — Structured Logging Integration

For production applications, ILoggerFactory integration is the standard approach. It connects EF Core logging to the .NET logging infrastructure, enabling log levels, filtering, structured logging, and integration with logging frameworks.

### Basic Configuration

In Program.cs (or Startup.cs), add the DbContext and configure logging:

```csharp
WebApplicationBuilder builder = WebApplication.CreateBuilder(args);

builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    options.UseSqlServer(builder.Configuration.GetConnectionString("DefaultConnection"));
});

// Logging is configured via the builder.Logging property
builder.Logging.ClearProviders();
builder.Logging.AddConsole();
builder.Logging.AddFilter("Microsoft.EntityFrameworkCore.Database.Command", LogLevel.Information);

WebApplication app = builder.Build();
```

The DbContext receives an ILoggerFactory through DI when AddDbContext is used. EF Core will automatically use this logger factory to emit log events.

### Filtering SQL Logs

To show ONLY SQL commands and filter out EF Core infrastructure events:

```csharp
builder.Logging.AddFilter("Microsoft.EntityFrameworkCore.Database.Command", LogLevel.Information);
builder.Logging.AddFilter("Microsoft.EntityFrameworkCore", LogLevel.Warning);  // suppress non-command EF logs
```

The first filter enables SQL command logging at Information level and above. The second filter suppresses other EF Core logging (model building, change tracking, etc.) unless they are at Warning level or above.

### Integration with Serilog

```csharp
builder.Host.UseSerilog((context, config) =>
{
    config.ReadFrom.Configuration(context.Configuration)
        .WriteTo.Console()
        .WriteTo.File("logs/ef-sql-.log",
            rollingInterval: RollingInterval.Day,
            restrictedToMinimumLevel: LogEventLevel.Information)
        .MinimumLevel.Override("Microsoft.EntityFrameworkCore.Database.Command", LogEventLevel.Information)
        .MinimumLevel.Override("Microsoft.EntityFrameworkCore", LogEventLevel.Warning);
});
```

### Integration with Application Insights

```csharp
builder.Services.AddApplicationInsightsTelemetry();

// EF Core logs are automatically collected if Application Insights logging is configured
builder.Logging.AddApplicationInsights(
    configureTelemetryConfiguration: config => config.ConnectionString = connectionString,
    configureApplicationInsightsLoggerOptions: options => { }
);
```

Application Insights can capture SQL dependency data directly via SqlClient instrumentation (covered in [[8.930 — Application Insights — SQL Dependency Tracking]]), but ILogger integration captures EF Core's log output as trace telemetry.

### ASP.NET Core Logging Best Practices

```csharp
// In appsettings.json or appsettings.Development.json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning",
      "Microsoft.EntityFrameworkCore.Database.Command": "Warning",
      "Microsoft.EntityFrameworkCore": "Warning"
    }
  }
}
```

Use environment-specific configuration:
- **Development**: `LogLevel.Information` for Database.Command — see every SQL statement
- **Staging/QA**: `LogLevel.Warning` — only see costly queries and errors
- **Production**: `LogLevel.Error` — only see failed SQL commands; use interceptors for detailed monitoring

## EnableSensitiveDataLogging — Parameter Value Visibility

By default, EF Core does not include parameter values in log output to protect sensitive data. EnableSensitiveDataLogging exposes these values.

### Enabling Sensitive Data Logging

```csharp
optionsBuilder
    .UseSqlServer(connectionString)
    .EnableSensitiveDataLogging()
    .LogTo(Console.WriteLine, LogLevel.Information,
           DbContextLoggerCategory.Database.Command.Name);
```

With this enabled, SQL log output shows actual parameter values:

```
Executed DbCommand (42ms) [Parameters=[@p0='john.doe@email.com' (Size = 256)],
 CommandType='Text', CommandTimeout='30']
SELECT [u].[Id], [u].[Name], [u].[Email]
FROM [Users] AS [u]
WHERE [u].[Email] = @p0
```

Without it:

```
Executed DbCommand (42ms) [Parameters=[@p0='?' (Size = 256)],
 CommandType='Text', CommandTimeout='30']
SELECT [u].[Id], [u].[Name], [u].[Email]
FROM [Users] AS [u]
WHERE [u].[Email] = @p0
```

### Conditional Enablement

```csharp
public class ApplicationDbContext : DbContext
{
    private readonly IWebHostEnvironment _env;

    public ApplicationDbContext(DbContextOptions<ApplicationDbContext> options,
                                IWebHostEnvironment env)
        : base(options)
    {
        _env = env;
    }

    protected override void OnConfiguring(DbContextOptionsBuilder optionsBuilder)
    {
        if (_env.IsDevelopment())
        {
            optionsBuilder.EnableSensitiveDataLogging();
        }
    }
}
```

### Security Considerations

- **NEVER enable in production**: Parameter values may contain PII (customer names, email addresses, credit card numbers, health records). Enable only in development and staging environments with sanitized data.
- **Use with caution in staging**: Even staging databases may contain representative data. Consider masking sensitive columns.
- **Compliance implications**: GDPR, HIPAA, and PCI-DSS may be violated if sensitive data is logged with actual values. Ensure compliance before enabling.
- **Alternatives for production**: Use IDbCommandInterceptor to selectively log parameter values for specific queries or capture anonymized execution statistics without storing parameter data.

## IDbCommandInterceptor — ADO.NET Level SQL Capture

`IDbCommandInterceptor` is the most powerful and flexible approach for capturing SQL in production. It operates at the ADO.NET layer, intercepting `SqlCommand` execution before and after the database call.

### Creating an Interceptor

```csharp
using System.Data.Common;
using Microsoft.EntityFrameworkCore.Diagnostics;

public class SqlPerformanceInterceptor : IDbCommandInterceptor
{
    private readonly ILogger<SqlPerformanceInterceptor> _logger;

    public SqlPerformanceInterceptor(ILogger<SqlPerformanceInterceptor> logger)
    {
        _logger = logger;
    }

    public InterceptionResult<DbDataReader> ReaderExecuting(
        DbCommand command,
        CommandEventData eventData,
        InterceptionResult<DbDataReader> result)
    {
        // Capture SQL before execution
        string sql = command.CommandText;
        var parameters = command.Parameters.Cast<DbParameter>()
            .ToDictionary(p => p.ParameterName, p => p.Value);

        _logger.LogDebug("Executing SQL: {Sql}", sql);
        return result;  // Allow execution to proceed
    }

    public DbDataReader ReaderExecuted(
        DbCommand command,
        CommandExecutedEventData eventData,
        DbDataReader result)
    {
        // Capture execution details after completion
        long durationMs = eventData.Duration.TotalMilliseconds;
        if (durationMs > 1000)  // Log slow queries (>1s)
        {
            _logger.LogWarning(
                "Slow SQL detected ({DurationMs}ms): {Sql}",
                durationMs, command.CommandText);
        }
        return result;
    }

    public int NonQueryExecuting(
        DbCommand command,
        CommandEventData eventData,
        InterceptionResult<int> result)
    {
        return result;
    }

    public int NonQueryExecuted(
        DbCommand command,
        CommandExecutedEventData eventData,
        int result)
    {
        return result;
    }

    public object? ScalarExecuting(
        DbCommand command,
        CommandEventData eventData,
        InterceptionResult<object?> result)
    {
        return result;
    }

    public object? ScalarExecuted(
        DbCommand command,
        CommandExecutedEventData eventData,
        object? result)
    {
        return result;
    }
}
```

### Registering the Interceptor

In Program.cs or Startup.cs:

```csharp
builder.Services.AddDbContext<ApplicationDbContext>((serviceProvider, options) =>
{
    var logger = serviceProvider.GetRequiredService<ILogger<SqlPerformanceInterceptor>>();

    options
        .UseSqlServer(connectionString)
        .AddInterceptors(new SqlPerformanceInterceptor(logger));
});
```

### Available Interception Points

| Method | Called When | Use Case |
|--------|-----------|----------|
| ReaderExecuting | Before ExecuteReader | Log SQL, check for N+1 patterns, abort dangerous queries |
| ReaderExecuted | After ExecuteReader completes | Log duration, capture result set metrics |
| NonQueryExecuting | Before ExecuteNonQuery | Log INSERT/UPDATE/DELETE SQL |
| NonQueryExecuted | After ExecuteNonQuery | Log duration and rows affected |
| ScalarExecuting | Before ExecuteScalar | Log scalar queries |
| ScalarExecuted | After ExecuteScalar | Log duration and single value result |
| CommandFailed | On exception | Log failure details with error message |
| DataReaderClosing | When DataReader closes | Calculate total data read duration |
| DataReaderDisposing | When DataReader is disposed | Cleanup resources |

### Building a Production Interceptor

A production interceptor typically captures:

```csharp
public class SqlMetricsInterceptor : IDbCommandInterceptor
{
    private readonly ConcurrentDictionary<Guid, DateTime> _commandStarts = new();

    public InterceptionResult<DbDataReader> ReaderExecuting(
        DbCommand command,
        CommandEventData eventData,
        InterceptionResult<DbDataReader> result)
    {
        _commandStarts[eventData.CommandId] = DateTime.UtcNow;
        return result;
    }

    public DbDataReader ReaderExecuted(
        DbCommand command,
        CommandExecutedEventData eventData,
        DbDataReader result)
    {
        if (_commandStarts.TryRemove(eventData.CommandId, out var startTime))
        {
            var duration = DateTime.UtcNow - startTime;

            // Extract command type and table names
            string commandType = command.CommandText.Trim().Split(' ')[0].ToUpper();

            // Record metric (pseudo-code)
            Metrics.RecordSqlExecution(
                commandType: commandType,
                database: command.Connection?.Database ?? "unknown",
                durationMs: duration.TotalMilliseconds,
                isSlow: duration.TotalMilliseconds > 1000,
                commandText: command.CommandText
            );
        }
        return result;
    }

    public void CommandFailed(
        DbCommand command,
        CommandErrorEventData eventData)
    {
        _logger.LogError(eventData.Exception,
            "SQL command failed: {Sql}, Duration: {DurationMs}ms",
            command.CommandText,
            eventData.Duration.TotalMilliseconds);
    }
}
```

### Slow Query Threshold Configuration

```csharp
public class SlowQueryInterceptor : IDbCommandInterceptor
{
    private readonly TimeSpan _threshold;
    private readonly ILogger _logger;

    public SlowQueryInterceptor(TimeSpan threshold, ILogger logger)
    {
        _threshold = threshold;
        _logger = logger;
    }

    // Called for ReaderExecuted, NonQueryExecuted, ScalarExecuted
    private void CheckSlow(DbCommand command, CommandExecutedEventData eventData)
    {
        if (eventData.Duration > _threshold)
        {
            _logger.LogWarning(
                "Slow SQL ({DurationMs}ms, threshold {ThresholdMs}ms): {Sql}",
                eventData.Duration.TotalMilliseconds,
                _threshold.TotalMilliseconds,
                command.CommandText);
        }
    }
}
```

Configure threshold via appsettings.json:

```json
{
  "SlowQueryThresholdMs": 500
}
```

## Interceptors — Advanced Usage Patterns

### Multiple Interceptors and Execution Order

When multiple interceptors are registered, EF Core executes them in registration order. Each interceptor can pass or modify the command to the next interceptor.

```csharp
options.AddInterceptors(
    new SqlMetricsInterceptor(),    // 1st: capture metrics
    new SlowQueryInterceptor(),     // 2nd: log slow queries
    new ReadOnlyGuardInterceptor()  // 3rd: block writes if applicable
);
```

If an interceptor returns a non-default result from an Executing method, subsequent interceptors may not execute (the result is short-circuited). Be careful with interceptor order — place validation/guard interceptors before metric collectors.

### Inhibiting Command Execution

Interceptors can prevent SQL execution by returning `InterceptionResult<>.SuppressWithResult()`:

```csharp
public InterceptionResult<DbDataReader> ReaderExecuting(
    DbCommand command,
    CommandEventData eventData,
    InterceptionResult<DbDataReader> result)
{
    if (command.CommandText.Contains("DROP TABLE"))
    {
        _logger.LogWarning("Blocked dangerous SQL: {Sql}", command.CommandText);
        return InterceptionResult<DbDataReader>.SuppressWithResult(
            new DataTable().CreateDataReader());
    }
    return result;
}
```

### Modifying Command Text

Interceptors can modify commands before execution:

```csharp
public InterceptionResult<DbDataReader> ReaderExecuting(
    DbCommand command,
    CommandEventData eventData,
    InterceptionResult<DbDataReader> result)
{
    // Add query hint for read-only routing (e.g., Azure SQL read replica)
    command.CommandText = command.CommandText.TrimEnd(';') + " OPTION (READONLY);";
    return result;
}
```

### Capturing Complete Query Execution Flow

```csharp
public class FullSqlLogger : IDbCommandInterceptor
{
    private readonly ILogger<FullSqlLogger> _logger;

    public FullSqlLogger(ILogger<FullSqlLogger> logger) => _logger = logger;

    public ValueTask<InterceptionResult<DbDataReader>> ReaderExecutingAsync(
        DbCommand command,
        CommandEventData eventData,
        InterceptionResult<DbDataReader> result,
        CancellationToken ct = default)
    {
        _logger.LogInformation(
            "[{Database}] {Sql}",
            command.Connection?.Database,
            FormatCommand(command));
        return new ValueTask<InterceptionResult<DbDataReader>>(result);
    }

    private static string FormatCommand(DbCommand command)
    {
        var sb = new StringBuilder();
        sb.AppendLine(command.CommandText);
        if (command.Parameters.Count > 0)
        {
            sb.AppendLine("-- Parameters:");
            foreach (DbParameter p in command.Parameters)
            {
                sb.AppendLine($"--   {p.ParameterName} = {p.Value} ({p.DbType})");
            }
        }
        return sb.ToString();
    }
}
```

## Pattern — N+1 Query Detection

The N+1 query problem occurs when a parent query loads data, then lazy-loads related data for each parent row individually. Interceptors can detect this pattern.

### N+1 Detection Interceptor

```csharp
public class NPlusOneDetector : IDbCommandInterceptor
{
    private static readonly ConcurrentDictionary<string, int> _queryCount = new();
    private const int NPlusOneThreshold = 5;

    public DbDataReader ReaderExecuted(
        DbCommand command,
        CommandExecutedEventData eventData,
        DbDataReader result)
    {
        string key = $"{command.Connection?.Database}:{command.CommandText.GetHashCode()}";

        int count = _queryCount.AddOrUpdate(key, 1, (_, c) => c + 1);

        if (count == NPlusOneThreshold)
        {
            var stackTrace = new System.Diagnostics.StackTrace(true);
            _logger.LogWarning(
                "N+1 query pattern detected: Same SQL executed {Count} times.{NewLine}{Sql}{NewLine}Stack trace: {StackTrace}",
                count,
                command.CommandText,
                stackTrace.ToString());
        }

        return result;
    }
}
```

### Detection Strategy

The interceptor tracks repeated executions of the same SQL (identified by command text hash). When the same SQL executes more than N times (typically 5-10), it logs a warning with the call stack, enabling developers to identify the N+1 source code location.

This is more effective for development/staging environments. For production, limit detection to specific endpoints or use sampling to avoid performance impact.

## Production — Operational Best Practices

### What to Log in Production

| Scenario | Log Level | Method | Notes |
|----------|-----------|--------|-------|
| SQL errors | Error | Interceptor/ILogger | Always log SQL failures with error details |
| Slow queries | Warning | Interceptor with threshold | Log queries exceeding 500ms-1s |
| All SQL | Debug | ILoggerFactory | Enable temporarily for debugging specific endpoints |
| SQL parameters | Never in production | N/A | Use sensitive data logging only in development |
| Execution plans | On-demand | Interceptor or Extended Events | Capture plans for slow queries in production |

### Performance Impact of Each Approach

| Approach | Overhead per SQL Call | Notes |
|----------|----------------------|-------|
| LogTo (Console) | ~0.5-2ms | Console I/O adds latency, avoid in high-throughput code paths |
| LogTo (File) | ~1-5ms | File I/O on every call, use buffered logging |
| ILoggerFactory (Console) | ~0.5-2ms | Similar to LogTo, depends on log provider |
| ILoggerFactory (Serilog File) | ~0.3-1ms | Asynchronous sinks mitigate overhead |
| IDbCommandInterceptor (no-op) | ~0.01-0.1ms | Minimal overhead, suitable for production |
| IDbCommandInterceptor (logging) | ~0.1-0.5ms | Depends on logger performance |
| ToQueryString() | 0ms | No DB call, development only |

### Mitigation Strategies for Production

**Use async logging**: Configure your logger (Serilog, NLog) with async sinks to offload I/O from the main thread:

```csharp
// Serilog async sink example
Log.Logger = new LoggerConfiguration()
    .WriteTo.Async(a => a.File("logs/ef-.log", rollingInterval: RollingInterval.Day))
    .CreateLogger();
```

**Batch slow query logging**: Instead of logging every slow query individually, batch them and log periodically:

```csharp
public class BatchedSlowQueryLogger : IDbCommandInterceptor
{
    private readonly ConcurrentBag<string> _slowQueries = new();
    private readonly ILogger _logger;
    private int _batchCounter;

    public DbDataReader ReaderExecuted(
        DbCommand command,
        CommandExecutedEventData eventData,
        DbDataReader result)
    {
        if (eventData.Duration.TotalMilliseconds > 1000)
        {
            _slowQueries.Add($"[{eventData.Duration.TotalMilliseconds}ms] {command.CommandText}");
        }

        // Flush every 100 slow queries or every 60 seconds
        if (Interlocked.Increment(ref _batchCounter) % 100 == 0)
        {
            FlushBatch();
        }

        return result;
    }

    private void FlushBatch()
    {
        if (_slowQueries.IsEmpty) return;

        var queries = Interlocked.Exchange(ref _slowQueries, new ConcurrentBag<string>());
        _logger.LogWarning("Slow SQL batch ({Count} queries):{NewLine}{Queries}",
            queries.Count, string.Join(Environment.NewLine, queries));
    }
}
```

**Use structured logging for analysis**: Log structured data so slow queries can be aggregated and queried:

```csharp
_logger.LogWarning(
    "Slow SQL: {DurationMs}ms, Type: {CommandType}, Database: {Database}",
    durationMs,
    commandType,
    command.Connection?.Database);
```

This allows log aggregation tools (Splunk, Elasticsearch, Azure Log Analytics) to query and visualize slow query patterns without parsing text.

## Diagnostics — Correlating Logs with Application Code

### Adding Correlation IDs to SQL Logging

To trace SQL execution back to specific application operations, add correlation IDs through the interceptor:

```csharp
public class CorrelationInterceptor : IDbCommandInterceptor
{
    private readonly IHttpContextAccessor _httpContextAccessor;
    private readonly ILogger _logger;

    public CorrelationInterceptor(IHttpContextAccessor httpContextAccessor, ILogger<CorrelationInterceptor> logger)
    {
        _httpContextAccessor = httpContextAccessor;
        _logger = logger;
    }

    public InterceptionResult<DbDataReader> ReaderExecuting(
        DbCommand command,
        CommandEventData eventData,
        InterceptionResult<DbDataReader> result)
    {
        var correlationId = _httpContextAccessor.HttpContext?
            .TraceIdentifier ?? "NONE";

        command.Comments = $"CorrelationId={correlationId};";

        _logger.LogInformation(
            "[{CorrelationId}] {Sql}",
            correlationId,
            command.CommandText);

        return result;
    }
}
```

### Integrating with OpenTelemetry

```csharp
public class OpenTelemetryInterceptor : IDbCommandInterceptor
{
    private static readonly ActivitySource Source = new("EFCore.Sql");

    public DbDataReader ReaderExecuted(
        DbCommand command,
        CommandExecutedEventData eventData,
        DbDataReader result)
    {
        using var activity = Source.StartActivity("SQL Query");
        activity?.SetTag("db.system", "mssql");
        activity?.SetTag("db.statement", command.CommandText);
        activity?.SetTag("db.name", command.Connection?.Database);
        activity?.SetTag("db.duration_ms", eventData.Duration.TotalMilliseconds);

        if (eventData.Duration.TotalMilliseconds > 1000)
        {
            activity?.SetTag("slow_query", true);
        }

        return result;
    }
}
```

This integration enables SQL queries to appear as spans in OpenTelemetry traces alongside HTTP requests, service calls, and other operations.

## Troubleshooting — Common Issues & Gotchas

### No SQL Output Visible

**Check logging configuration**: Verify that the log level allows Information-level messages for the EF Core category. If `LogLevel.Warning` or higher is configured, SQL execution logs (Information level) will be suppressed.

**Verify DbContext is resolved through DI**: If you manually instantiate DbContext (with `new ApplicationDbContext()`), it will NOT receive the ILoggerFactory from DI. Either:
- Use `AddDbContext` and resolve through DI
- Pass the logger factory explicitly: `new ApplicationDbContext(new DbContextOptionsBuilder<ApplicationDbContext>().UseLoggerFactory(loggerFactory).Options)`

**Check for multiple LogTo calls**: Calling LogTo multiple times adds multiple loggers. Make sure you're not overriding with a more restrictive configuration later.

**Verify event category filter**: If you use the event IDs filter and specify only `RelationalEventId.CommandExecuted`, events from raw SQL commands (NonQuery, Scalar) will not be logged. Include all three: `CommandExecuted`, `NonQueryExecuted`, `ScalarExecuted`.

### SQL Logging Shows Only '?' for Parameter Values

This is the default behavior (parameter values are hidden for security). Enable `EnableSensitiveDataLogging()` to see actual values — but only in non-production environments.

### Interceptor Not Firing

**Check registration**: Ensure the interceptor is registered via `AddInterceptors()` on the `DbContextOptionsBuilder`, not via DI alone. The DbContext must use the configured options builder.

**Check for short-circuiting**: If another interceptor suppresses the result earlier in the chain, subsequent interceptors won't execute. Verify interceptor registration order.

**Check DbContext pooling**: With `AddDbContextPool`, the DbContext is reused. Interceptors registered per-context will work correctly, but static state in interceptors may cause issues across requests.

### EnableSensitiveDataLogging Exposes PII

**Gotcha**: Enabling sensitive data logging in production — even accidentally through a wrong configuration — can expose customer data (emails, names, financial data) in logs. This has compliance implications for HIPAA, GDPR, PCI-DSS, and SOC 2.

**Prevention**: 
- Use conditional enablement based on environment (IWebHostEnvironment.IsDevelopment())
- Set up code analysis rules to flag EnableSensitiveDataLogging() usage
- Use configuration validation to ensure it's not enabled in production configs
- Consider using a custom interceptor that allows selective parameter logging

### Logging All SQL in Production Adds IO

**Impact**: Logging every SQL statement to a file creates write IO that competes with application throughput. On high-traffic applications (100+ SQL calls/second), file-based logging can:

- Increase disk utilization (potentially affecting database performance if on shared storage)
- Increase application latency due to log file contention
- Fill disk space rapidly (100 SQL/s × 500 bytes per log entry = 4.3GB/day)

**Mitigation**: Use structured logging with async batch writers. Set different log levels per environment. Use interceptors for selective logging (slow queries only, errors only). Implement log rotation and retention policies.

### ToQueryString() Doesn't Execute — Only Shows SQL

**Gotcha**: Developers often call ToQueryString() and assume the query has been executed. It has not. The SQL is generated but never sent to the database. This is useful for debugging but can lead to confusion if you expect side effects (e.g., calling SaveChanges after ToQueryString() on an INSERT query — the INSERT won't happen).

### Interceptor Order with Multiple Interceptors

**Gotcha**: When multiple interceptors are registered, the execution order follows registration order for the Executing phase, but REVERSE order for the Executed phase (like a middleware pipeline). This means:

```
Registration: A → B → C
Executing:    A → B → C → SQL Server
Executed:     C → B → A
```

If interceptor A suppresses the result, B and C don't execute for the Executing phase, but A's Executed handler runs.

### Thread Safety in Interceptors

**Gotcha**: Interceptors are singletons by default. All state accessed in interceptor methods must be thread-safe. Use `ConcurrentDictionary`, `Interlocked` operations, or immutable data structures for shared state.

```csharp
// BAD: Non-thread-safe collection
private readonly List<string> _slowQueries = new();

// GOOD: Thread-safe collection
private readonly ConcurrentBag<string> _slowQueries = new();
```

### Memory Leaks with Interceptors

**Gotcha**: Holding references to DbCommand, DbDataReader, or other disposable objects beyond the interceptor method scope can prevent garbage collection and cause memory leaks.

```csharp
// BAD: Storing command for later use
private DbCommand _lastCommand;
public DbDataReader ReaderExecuted(...)
{
    _lastCommand = command;  // Holds reference, prevents GC
}

// GOOD: Extract only needed data
public DbDataReader ReaderExecuted(...)
{
    string sql = command.CommandText;  // Extract string, discard command reference
    _lastSql = sql;
}
```

## Comparison — Choosing the Right Approach

| Approach | Best For | Production Safety | Setup Complexity | Performance Impact |
|----------|---------|------------------|-----------------|-------------------|
| LogTo | Quick dev debugging, simple apps | Low (can flood logs) | Minimal | Medium |
| ToQueryString() | Development inspection only | N/A (development only) | Minimal | None |
| ILoggerFactory | Structured logging in production | High | Low | Low-Medium |
| IDbCommandInterceptor | Production monitoring, custom logic | High | Medium | Very Low |

### Decision Tree

**Are you debugging a specific query in development?**
→ Use ToQueryString() for the query SQL without executing.
→ Use LogTo for seeing all SQL executed by a code path.

**Are you developing a new feature and want to see the SQL?**
→ Use LogTo with category filter to Database.Command in development.

**Are you setting up production monitoring?**
→ Use IDbCommandInterceptor with a slow query threshold and structured logging.
→ Optionally combine with ILoggerFactory for general EF Core error logging.

**Do you need SQL performance metrics over time?**
→ Use IDbCommandInterceptor with a metrics collector (Prometheus, AppMetrics, etc.).
→ Export as metrics for dashboarding in Grafana.

**Do you already use Application Insights?**
→ Use AI's built-in SQL dependency tracking ([[8.930 — Application Insights — SQL Dependency Tracking]]).
→ This captures SQL at the ADO.NET layer without EF Core-specific configuration.

**Do you need to correlate SQL queries with application request traces?**
→ Use IDbCommandInterceptor with OpenTelemetry or correlation ID integration.
