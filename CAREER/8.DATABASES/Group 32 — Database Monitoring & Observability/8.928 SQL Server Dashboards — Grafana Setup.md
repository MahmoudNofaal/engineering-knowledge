# 8.928 SQL Server Dashboards — Grafana Setup

Priority: 2 | Domain: Databases | Group: Database Monitoring & Observability

---

## Overview — Purpose & Scope

Grafana dashboards provide real-time visualization of SQL Server performance metrics collected via Prometheus. This note covers the end-to-end pipeline from SQL Server to visual dashboards, including data source configuration, dashboard import and customization, alert rule setup, and production considerations for running Grafana-backed SQL Server monitoring at scale.

Monitoring SQL Server through Grafana replaces traditional SSMS-based monitoring with a centralized, historical, and shareable observability platform. By combining Prometheus as a time-series database with Grafana as the visualization layer, teams gain the ability to correlate SQL Server performance with application-level metrics, infrastructure metrics, and other data sources in a single pane of glass.

The scope of this note includes the architecture of the monitoring stack, step-by-step configuration of Grafana to consume Prometheus metrics, recommended community dashboards and their key panels, alerting strategies for common SQL Server conditions, and deployment patterns for production environments. It assumes the Prometheus SQL Server Exporter (see [[8.929 — SQL Server — Prometheus Exporter]]) is already installed and configured to expose metrics.

Key objectives of implementing Grafana dashboards for SQL Server include reducing mean time to detection (MTTD) for performance anomalies, enabling capacity planning through historical trend analysis, providing self-service visibility for developers and DBAs, and establishing a single source of truth for database performance across the organization.

This note is part of a broader observability strategy documented in Group 32 — Database Monitoring & Observability. It relates directly to [[8.916 — SQL Server Monitoring — Key Metrics]] for understanding which metrics matter, [[8.917 — Wait Statistics — Top Waits Analysis]] for interpreting wait-based performance data, and [[8.924 — Baseline Capture — DMV Snapshot Strategy]] for establishing performance baselines.

## Architecture — Data Flow & Components

The monitoring pipeline follows a pull-based architecture where Prometheus scrapes metrics from the SQL Server Exporter, stores them in its time-series database (TSDB), and Grafana queries Prometheus as a data source to render dashboards and evaluate alert rules.

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐     ┌──────────────────────┐
│   SQL Server    │     │  sql_exporter    │     │   Prometheus    │     │       Grafana        │
│   Instances     │ ◄──►│  /metrics        │ ◄───│  Server         │ ◄───│  Dashboards + Alerts │
│   (DMVs, perf   │     │  HTTP endpoint   │     │  Scrape + TSDB  │     │  Visualization       │
│   counters, IO) │     │  port 9399       │     │  port 9090      │     │  port 3000           │
└─────────────────┘     └──────────────────┘     └─────────────────┘     └──────────────────────┘
```

**Component roles in the pipeline:**

SQL Server — The source of all metrics. The exporter queries Dynamic Management Views (DMVs) such as sys.dm_os_wait_stats, sys.dm_os_performance_counters, sys.dm_io_virtual_file_stats, sys.dm_exec_query_stats, and sys.dm_os_schedulers. Performance data is extracted from these views at each scrape interval.

SQL Server Exporter — A lightweight Go service or Telegraf plugin that connects to SQL Server, executes collector queries against DMVs, transforms the results into Prometheus-format metrics, and exposes them on an HTTP endpoint (typically port 9399 for the official sql_exporter, or port 9273 for Telegraf). The exporter is stateless and re-queries all metrics on each scrape request.

Prometheus Server — Pulls metrics from the exporter at a configured interval (typically 15-30 seconds), stores them in its on-disk TSDB with configurable retention, evaluates alerting rules, and serves data to Grafana via its HTTP API on port 9090. Prometheus handles time-series storage with a custom compression algorithm, achieving ~1.3 bytes per sample on average.

Grafana Server — Queries Prometheus as a data source, renders pre-built dashboards using PromQL queries against stored time-series data, evaluates alert rules defined in dashboards or as standalone alerting rules, and sends notifications through configured channels (Slack, PagerDuty, email, webhook, OpsGenie, etc.). Grafana also supports annotations for marking deployments, incidents, or other events on timeline panels.

**Alternative architectures:**

Telegraf-based deployment — Instead of the dedicated sql_exporter, Telegraf can be used with its sqlserver input plugin. Telegraf supports both pull (via /metrics endpoint) and push (via InfluxDB output) models. When using push, Prometheus is replaced by or supplemented with InfluxDB or another backend.

Direct SQL Server data source — Grafana also offers a native SQL Server data source plugin that queries SQL Server directly via ODBC/JDBC. This bypasses Prometheus entirely and is useful for ad-hoc queries, real-time data, and environments where a full Prometheus stack is not justified. However, it lacks the historical TSDB capabilities and alerting sophistication of Prometheus-based monitoring.

Hybrid approach — Many organizations run both: Prometheus-based monitoring for historical trending and alerting, and the direct SQL Server data source in Grafana for operational queries, index analysis, and detailed DMV exploration.

**Data flow sequence:**

1. Prometheus sends an HTTP GET request to the exporter's /metrics endpoint at each scrape interval.
2. The exporter connects to SQL Server, executes all enabled collector queries, and formats results as Prometheus metrics (gauges, counters, histograms).
3. Prometheus receives the metrics response, parses it, and appends samples to its TSDB with the current timestamp.
4. Grafana periodically queries Prometheus (or receives pushed data via the Prometheus remote write protocol) to populate dashboard panels.
5. Dashboard panels render time-series data as graphs, tables, gauges, heatmaps, or stat panels based on PromQL queries.
6. Alert rules evaluate continuously against the TSDB and trigger notifications when conditions are met.

**Network topology considerations:**

The exporter should be installed on or near the SQL Server to minimize network latency and avoid firewall complications. Prometheus can be centralized in a monitoring cluster with network access to all exporter endpoints. Grafana typically runs alongside Prometheus or in a separate highly-available configuration with a shared PostgreSQL database for dashboard and user state.

For multi-region or multi-datacenter deployments, consider running a Prometheus instance per data center with global federation or Thanos/Cortex for long-term storage and cross-cluster querying.

## Setup — Grafana Data Source & Dashboard Import

**Step 1: Add Prometheus as a data source in Grafana**

Navigate to Configuration > Data Sources > Add data source in the Grafana UI. Select Prometheus from the list of supported data sources. Configure the following fields:

- Name: Prometheus (or Prometheus-SQL-Server for clarity)
- HTTP URL: http://prometheus-server:9090 (adjust for your environment)
- Access: Server (default) — Grafana's backend proxies requests to Prometheus
- Scrape interval: 15s (should match or be greater than the Prometheus scrape interval)
- Query timeout: 60s (increase for complex PromQL queries)
- Type: Prometheus
- Version: 2.x (auto-detected)

Enable "Use POST method" for complex queries that exceed URL length limits. Configure custom HTTP headers if using authentication or reverse proxy.

For production environments, set up a dedicated Prometheus service account and configure HTTP basic auth or API token authentication in both Prometheus and Grafana. Consider encrypting the connection with TLS between Grafana and Prometheus.

**Step 2: Import the community SQL Server dashboard**

The most widely used Grafana dashboard for SQL Server monitoring is **Dashboard ID 9628** (created by ilovesoup and maintained by the community). This dashboard provides comprehensive coverage of SQL Server metrics and is actively maintained.

To import:

1. In Grafana, go to Dashboards > Import
2. Enter Dashboard ID 9628 in the "Import via grafana.com" field
3. Click Load
4. Select the Prometheus data source from the dropdown
5. Click Import

The dashboard includes the following rows and panels:

- **SQL Server Instance Overview**: CPU usage, memory pressure, batch requests/sec, logins/sec, user connections
- **Wait Statistics**: Top wait types with real-time and historical views, wait time breakdown by category (IO, locking, latches, network, etc.)
- **Query Performance**: Top queries by duration, CPU, logical reads, and execution count (requires query_stats collector)
- **Database I/O**: Read/write latency by database file, IOPS by file, file size and growth
- **Buffer Pool**: Page life expectancy, buffer pool size in GB, cache hit ratio, checkpoint pages/sec
- **Database Space**: Data file and log file size by database, autogrowth events, space used vs. free
- **Index Usage**: Top indexes by seek/scan operations, unused indexes, index fragmentation
- **Activity**: Active sessions by database, blocking chains, longest-running queries
- **TempDB**: TempDB file size and growth, allocation rates, contention metrics

Alternative dashboards include:

- **Dashboard ID 12684** — SQL Server Metrics by Jitendra Zaa (simpler, fewer panels)
- **Dashboard ID 9452** — SQL Server Overview by Kevin Chant (focused on Azure SQL)
- **Dashboard ID 11454** — SQL Server Monitoring by Monika W (comprehensive with alerts)

**Step 3: Customize the dashboard**

Rarely does an imported dashboard perfectly match your environment. Plan to spend time customizing:

- Remove panels for collectors you haven't enabled (e.g., if query_stats is disabled, remove the Top Queries panels)
- Adjust thresholds (red/yellow/green) to match your performance baselines — e.g., Page Life Expectancy warning at 300 seconds, critical at 150 seconds
- Update variable queries for Instance, Database, or Collector to match your SQL Server instance names and labels
- Add service-specific annotations: map application deployment markers, maintenance windows, or known incident timelines
- Rename panels for team clarity — use naming conventions your team will understand, not exporter internal names
- Adjust time range defaults: set dashboard default to "last 6 hours" for operational view, "last 30 days" for capacity planning

**Step 4: Create additional dashboards for specific workloads**

Beyond the general-purpose SQL Server dashboard, consider creating specialized dashboards:

- **Query Performance Dashboard**: Deep dive into query metrics — top N by duration, CPU, reads, writes; query plan changes; execution count trends; plan cache health
- **Wait Stats Dashboard**: Dedicated to wait statistics by type, historical wait trends, signal wait percentage, and resource wait breakdown
- **Database Growth Dashboard**: File size over time, autogrowth frequency, space forecast columns, file count by auto-growth type
- **TempDB Dashboard**: TempDB file configuration, allocation rates, version store usage, contention diagnostics (PFS, GAM, SGAM pages)
- **Availability Group Dashboard**: Replication lag, synchronization health, log send queue, estimated recovery time, flow control waits
- **Agent Job Dashboard**: Job run duration over time, failure counts, step-level duration breakdown, schedule adherence

Each dashboard should have consistent variables (Instance, Database, interval) to enable quick switching and filtering.

## Usage — Key PromQL Queries & Panels

Understanding PromQL is essential for customizing and troubleshooting dashboards. Below are the key PromQL queries used in SQL Server monitoring dashboards, organized by metric category.

### CPU Usage

```
100 - (avg by (instance) (rate(mssql_os_counter_value{counter_name="Process cpu usage%"}[5m])))
```

This query calculates the CPU utilization percentage of the SQL Server process. The counter_name varies by collector implementation — with Telegraf it may be `sqlserver_cpu_percent`. The 5m rate smoothes out short-term spikes to show sustained CPU load.

For total system CPU (including non-SQL processes):

```
avg by (instance) (rate(windows_cpu_time_total{mode!="idle"}[5m]))
```

### Batch Requests per Second

```
rate(mssql_counter_value{counter_name="Batch Requests/sec"}[1m])
```

Batch Requests/sec is a classic indicator of SQL Server throughput. Sustained high values indicate heavy workload; sudden drops may signal a blocking chain or application failure. Use a 1m rate for real-time view and 5m rate for trend analysis.

### Page Life Expectancy (PLE)

```
mssql_counter_value{counter_name="Page life expectancy"}
```

PLE measures how long (in seconds) a data page stays in the buffer pool. Low PLE (below 300 seconds) indicates memory pressure. PLE is a gauge metric — no rate function needed. For multi-NUMA systems, PLE is per-node and should be averaged:

```
avg by (instance) (mssql_counter_value{counter_name="Page life expectancy"})
```

### Wait Statistics

Top waits by wait type:

```
topk(10, rate(mssql_wait_stats_wait_time_ms[5m]) / rate(mssql_wait_stats_wait_count[5m]) > 0)
```

Average wait time per wait:

```
rate(mssql_wait_stats_wait_time_ms[5m]) / rate(mssql_wait_stats_wait_count[5m])
```

Total wait time by category:

```
sum by (wait_category) (rate(mssql_wait_stats_wait_time_ms[5m]))
```

Wait categories are derived from the wait_type label and can be bucketed into: Resource (IO, network), Queue (locks, latches), External (backup, mirroring), or Unknown.

### I/O Latency

```
rate(mssql_io_stall_read_ms[5m]) / rate(mssql_io_stall_read_count[5m])
```

Average read latency in milliseconds per file. Similar query for writes:

```
rate(mssql_io_stall_write_ms[5m]) / rate(mssql_io_stall_write_count[5m])
```

High latency (>50ms for reads, >100ms for writes) indicates storage subsystem issues. Compare across files and databases to identify problematic volumes.

### Top Queries by Duration

```
topk(10, avg by (query_hash, database_name) (rate(mssql_query_duration_ms[5m])))
```

This requires the mssql_query_stats collector to be enabled. The query_hash label identifies unique queries (parameterized). Note that high-cardinality labels can significantly increase metric volume — see Gotchas section.

### Database File Size and Space Used

```
mssql_volume_space{type="data"}
mssql_volume_space{type="log"}
```

Space used as a percentage:

```
mssql_volume_space{type="data"} / ignoring(type) mssql_volume_space{type="data_max_size"}
```

### Active Sessions by Database

```
count by (database_name) (mssql_active_sessions)
```

Or from schedulers:

```
sum by (database_name) (rate(mssql_scheduler_total[5m]))
```

### Connection Count

```
mssql_counter_value{counter_name="User Connections"}
```

### Buffer Pool Cache Hit Ratio

```
mssql_counter_value{counter_name="Buffer cache hit ratio"}
```

A ratio above 95% is healthy. Values below 90% indicate memory pressure and warrant investigation into PLE and page reads.

### Logins per Second

```
rate(mssql_counter_value{counter_name="Logins/sec"}[5m])
```

Spikes in logins can indicate application recycling, connection pooling issues, or brute-force attacks.

### SQL Compilations vs. Re-Compilations

```
rate(mssql_counter_value{counter_name="SQL Compilations/sec"}[5m])
rate(mssql_counter_value{counter_name="SQL Re-Compilations/sec"}[5m])
```

High re-compilation rates (more than 10% of compilations) indicate plan cache churn and warrant investigation into parameterization, plan freezing, or forced parameterization settings.

### Lock Waits and Deadlocks

```
rate(mssql_counter_value{counter_name="Lock Waits/sec"}[5m])
rate(mssql_counter_value{counter_name="Deadlocks/sec"}[5m])
```

Lock waits per second indicate contention. Deadlocks should always be zero; any positive count requires immediate investigation into transaction isolation levels, index design, and query ordering.

### Index Usage Metrics

```
rate(mssql_counter_value{counter_name="Index Searches/sec"}[5m])
rate(mssql_counter_value{counter_name="Full Scans/sec"}[5m])
```

A high ratio of Full Scans to Index Searches suggests missing or suboptimal indexes. Track this over time per database to prioritize indexing efforts.

### Memory Grants Pending

```
mssql_counter_value{counter_name="Memory Grants Pending"}
```

Any non-zero value indicates queries waiting for memory grants — a sign of memory pressure or poorly tuned query memory requirements. Correlate with RESOURCE_SEMAPHORE wait type.

### Log Growth Events

```
increase(mssql_volume_space_max_size[1h]) - increase(mssql_volume_space[1h])
```

Track autogrowth events. Frequent small autogrowths cause log fragmentation and performance degradation. Compare T-Log growth events against database size to right-size initial allocation.

### TempDB Allocation Rates

```
rate(mssql_counter_value{counter_name="Version generation rate (KB/s)"}[5m])
rate(mssql_counter_value{counter_name="Temp Tables Creation Rate"}[5m])
```

High TempDB allocation rates indicate excessive use of temporary objects, version store pressure (read-committed snapshot isolation), or poorly tuned queries that spill to TempDB.

### Query Store Metrics (When Available)

For Azure SQL Database or SQL Server 2022+ with Query Store enabled, expose Query Store metrics:

```
mssql_query_store_cpu_time
mssql_query_store_logical_reads
mssql_query_store_duration
```

These metrics come from Query Store queries via the exporter. They provide SQL Server-native query performance data alongside Prometheus-gathered metrics.

### Full Scan Detection Per Database

```
topk(10, rate(mssql_counter_value{counter_name="Full Scans/sec", instance_name=~".+"}[5m]))
```

Break down full scans by database instance to identify which databases need index tuning. The instance_name label in performance counters often contains the database name.

### Plan Cache Metrics

```
mssql_counter_value{counter_name="Cache Hit Ratio"}
mssql_counter_value{counter_name="Cache Object Counts"}
mssql_counter_value{counter_name="Cache Pages"}
```

Monitor plan cache health. A low cache hit ratio combined with high recompilation rates indicates plan cache thrashing. Investigate parameterized queries, forced parameterization, or plan guides.

### Active Parallel Threads

```
mssql_counter_value{counter_name="Active parallel threads"}
```

Monitors parallelism usage. Sustained high values suggest many large queries running in parallel, potentially consuming CPU and IO resources. Correlate with CXCONSUMER waits.

### Log Flush Wait Time

```
rate(mssql_counter_value{counter_name="Log Flush Wait Time"}[5m])
```

High log flush wait times indicate the transaction log is a bottleneck. This often manifests as WRITELOG waits and impacts write-heavy workloads. Consider faster storage for the log file or splitting the log to dedicated high-performance storage.

## Alerts — Configuring Alert Rules in Grafana

Grafana provides a unified alerting system that evaluates PromQL queries against the Prometheus data source. Alert rules can be created at the dashboard panel level or as standalone rules in the Alerting UI.

### Core Alert Rules for SQL Server Monitoring

**High CPU Usage Alert**

```
Condition: avg by (instance) (rate(mssql_os_counter_value{counter_name="Process cpu usage%"}[5m])) > 85
Evaluation: Every 1m, for 5m
Severity: Warning
Message: SQL Server CPU usage on {{ $labels.instance }} is at {{ $value | humanizePercentage }} for the last 5 minutes
```

Refine the threshold based on baseline data. Some servers run consistently at 70-80% CPU under normal load — adjust the alert threshold to 5-10% above the P95 baseline.

**Low Page Life Expectancy Alert**

```
Condition: avg by (instance) (mssql_counter_value{counter_name="Page life expectancy"}) < 300
Evaluation: Every 1m, for 3m
Severity: Critical
Message: Page Life Expectancy on {{ $labels.instance }} has dropped below 300 seconds — possible memory pressure
```

Consider different thresholds for different workloads. OLTP systems need higher PLE (>600s), while data warehouse workloads can tolerate lower PLE (>150s).

**High Wait Time Alert**

```
Condition: rate(mssql_wait_stats_wait_time_ms{wait_type=~"PAGEIOLATCH_.*|WRITELOG|ASYNC_IO_COMPLETION"}[5m]) / rate(mssql_wait_stats_wait_count{wait_type=~"PAGEIOLATCH_.*|WRITELOG|ASYNC_IO_COMPLETION"}[5m]) > 100
Evaluation: Every 1m, for 5m
Severity: Warning
Message: Average I/O wait time on {{ $labels.instance }} is above 100ms — storage subsystem may be degraded
```

**Database Space Alert**

```
Condition: (mssql_volume_space{type="data"} / mssql_volume_space{type="data_max_size"}) * 100 > 90
Evaluation: Every 5m, for 10m
Severity: Warning
Message: Database file {{ $labels.database }} on {{ $labels.instance }} is {{ $value | humanizePercentage }} full
```

Two-tier threshold: Warning at 80%, Critical at 95%. Include predicted exhaustion date using linear regression for proactive alerts.

**Deadlock Alert**

```
Condition: rate(mssql_counter_value{counter_name="Deadlocks/sec"}[5m]) > 0
Evaluation: Every 1m, for 2m
Severity: Critical
Message: Deadlocks detected on {{ $labels.instance }} — investigate blocking chains and transaction isolation levels
```

**Batch Requests/Sec Anomaly Alert**

```
Condition: abs(rate(mssql_counter_value{counter_name="Batch Requests/sec"}[5m]) - avg_over_time(rate(mssql_counter_value{counter_name="Batch Requests/sec"}[5m])[1h:])) > (3 * stddev_over_time(rate(mssql_counter_value{counter_name="Batch Requests/sec"}[5m])[1h:]))
Evaluation: Every 1m, for 5m
Severity: Warning
Message: Throughput deviation detected on {{ $labels.instance }} — batch requests differ significantly from hourly baseline
```

### Alert Notification Channels

Configure notification channels in Grafana Alerting > Contact points:

- **Slack**: Send alerts to a dedicated #dba-alerts channel with severity emoji prefixes (🟡 Warning, 🔴 Critical)
- **PagerDuty/OpsGenie**: Route Critical alerts to on-call DBAs with auto-acknowledgment and escalation policies
- **Email**: Primary notification for Warning severity, digest format for non-urgent alerts
- **Webhook**: Trigger automated remediation scripts (e.g., kill blocking session, growth file, restart service)
- **SMS/Phone**: Critical-only for production systems with SLAs

### Alert Silencing and Muting

Use Grafana's mute timing and silence features to suppress alerts during:

- Scheduled maintenance windows (patch Tuesday, index rebuilds, backups)
- Known deployment cycles (application releases that spike CPU)
- Holiday periods with reduced staffing
- Time-of-day patterns (lower threshold tolerance during business hours)

Create notification policies that route alerts by severity, label, or namespace. For example, route all Production SQL Server alerts to PagerDuty with immediate escalation, but Development alerts to a Slack low-priority channel.

## Dashboard — Custom Panel Configuration

Beyond imported dashboards, custom panels address specific monitoring needs. Below are patterns for building effective custom panels.

### Custom Stat Panel: PLE Trend

Query:

```
avg by (instance) (mssql_counter_value{counter_name="Page life expectancy"})
```

Set Stat panel to show current value, sparkline for 1h trend, and color thresholds (green > 600, yellow 300-600, red < 300). Add a secondary query showing 7-day minimum for long-term perspective.

### Custom Time Series: IO Latency by File

Query:

```
label_replace(
  rate(mssql_io_stall_read_ms[5m]) / rate(mssql_io_stall_read_count[5m]),
  "file", "$1", "file_id", "(.*)"
)
```

Add series override to color by database name. Use the legend format `{{database_name}} - {{file_type}}` for readability. Set Y-axis unit to "ms" and logarithmic scale to show both small and large latencies.

### Custom Table: Top 10 Longest Running Queries

Query:

```
topk(10, avg by (query_hash, database_name) (rate(mssql_query_duration_ms[5m])))
```

Configure table panel with columns: query_hash, database_name, avg_duration_ms. Add a link to copy the query_hash for further investigation in SSMS or Azure Data Studio. Tip: query_hash alone is not useful without the query text — consider adding a query_text label (but beware of cardinality).

### Custom Heatmap: Wait Statistics Over Time

Use a heatmap panel with:

```
rate(mssql_wait_stats_wait_time_ms[5m])
```

Set the bucket bound to the wait_type label. X-axis: time. Y-axis: wait type. Color intensity: total wait time in ms. This visualization quickly reveals which wait types dominate at different times of day.

### Custom Gauge: Buffer Pool Health

Combine three metrics into a single gauge:

- Page Life Expectancy: gauge arc with green/yellow/red zones
- Buffer cache hit ratio: additional gauge or stat
- Free pages: supplementary stat below

### Custom Bar Chart: Database Space Usage

Stacked bar chart comparing used vs. free space per database:

```
mssql_volume_space{type="data"}
mssql_volume_space_max_size{type="data"} - mssql_volume_space{type="data"}
```

Group by database_name. Show as percentage with absolute value tooltip.

### Custom Annotations: Deployment Tracking

Configure annotation queries in Grafana dashboard settings:

```
SELECT time, title, text FROM deployment_events WHERE environment = 'production'
```

This requires a separate annotation data source (SQLite, PostgreSQL, or even a CSV file). Annotations render as vertical lines on all panels, helping correlate performance changes with application deployments.

Alternatively, use a Prometheus info metric or the Grafana annotation API via webhook at deployment time.

## Production — Operational Considerations

### Grafana Deployment Architecture

For production deployments, Grafana should be configured for reliability and performance:

- **High Availability**: Run multiple Grafana instances behind a load balancer with a shared PostgreSQL or MySQL database for dashboard, user, and alert state
- **Storage**: Use persistent volumes for Grafana's SQLite database (or the shared RDBMS), plugin directory, and provisioning files
- **Backup**: Regularly backup the Grafana database, dashboard JSON models (export via API), and provisioning YAML files
- **Version Control**: Store dashboard JSON definitions in Git using Grafana's provisioning system. This enables code review, change tracking, and automated deployment of dashboards across environments

### Prometheus Performance Tuning

Prometheus performance directly impacts dashboard responsiveness:

- **Retention**: Default 15-day retention. For longer storage, use Thanos, Cortex, or VictoriaMetrics for downsampling and archival
- **Scrape Interval**: Balance between granularity and storage. 15s for production, 30-60s for development or low-priority instances
- **Rule Evaluation**: Alert rules evaluate every 10-30s. Keep rule count manageable — use recording rules for complex PromQL expressions
- **Storage Volume**: Calculate storage needs: ~1GB per 1,000 time series per month (rough estimate, varies by label cardinality)
- **Memory**: Prometheus holds recent data in memory. Allocate at least 50% of the TSDB size as available memory

### Dashboard Performance Optimization

Slow dashboards frustrate users and should be optimized:

- **Use Recording Rules**: Pre-compute expensive queries in Prometheus recording rules. Example: `record: sqlserver:ple:avg_5m` for a smoothed PLE metric
- **Reduce Panel Count**: Each panel with a unique PromQL query adds load. Combine related metrics into multi-stat panels
- **Query Range**: Set appropriate time ranges. "Last 24 hours" is significantly faster than "Last 30 days" with full resolution
- **Use Template Variables**: Pre-compute variable values with template variable queries instead of embedding them in every panel
- **Avoid Wildcards**: Prefer `{label="value"}` over `{label=~".*"}` in PromQL selectors to limit series lookups
- **Rate Function Placement**: Apply `rate()` at query time, not in recording rules, to preserve flexibility

### Authentication and Authorization

Secure Grafana access:

- **Authentication**: Integrate with LDAP, Active Directory, Azure AD, or OAuth 2.0 (Google, GitHub, GitLab)
- **Role-Based Access Control**: Define Viewer, Editor, and Admin roles. Viewers cannot modify dashboards; Editors can create and modify; Admins manage data sources and users
- **API Keys**: Use service account tokens for automated dashboard provisioning, report generation, or alert channel configuration
- **Audit Logging**: Enable Grafana audit logs to track user actions, dashboard changes, and data source modifications

### Monitoring Grafana Itself

Don't forget to monitor the monitoring system:

- Grafana's own metrics endpoint (port 3000/metrics) exposes request latency, active users, HTTP status codes, and alert evaluation duration
- Set up dashboards for Grafana uptime and performance
- Configure alerts for Grafana service availability
- Monitor Prometheus targets (up metric) to detect exporter failures

### Grafana Provisioning via Code

For teams managing multiple environments, use Grafana's provisioning system to declare dashboards and data sources as code:

```yaml
# /etc/grafana/provisioning/datasources/prometheus.yml
apiVersion: 1
datasources:
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://prometheus:9090
    isDefault: true
    editable: false
    jsonData:
      timeInterval: 15s
      queryTimeout: 60s
```

```yaml
# /etc/grafana/provisioning/dashboards/sqlserver.yml
apiVersion: 1
providers:
  - name: SQL Server
    type: file
    options:
      path: /etc/grafana/dashboards/sqlserver
      foldersFromFilesStructure: false
```

Store dashboard JSON files in version control and sync them to the provisioning directory. This enables GitOps workflows for dashboard changes, peer review through pull requests, and automated rollback of dashboard changes.

### Dashboard Versioning and Change Management

Track dashboard changes with a systematic approach:

- **Git-based dashboard storage**: Export dashboards as JSON files and commit to the monitoring repository
- **Dashboard change review**: Require code review for dashboard changes — PromQL changes can mask problems or generate excessive load
- **Dashboard testing**: Validate dashboard changes against a staging Prometheus data source before deploying to production
- **Dashboard annotations as changelog**: Use Grafana's built-in version history to track who changed what in each dashboard
- **Dashboard naming conventions**: Use consistent naming with environment prefix (e.g., "PRD - SQL Server Overview", "DEV - SQL Server Overview")

### Alerting Rule Provisioning

Alert rules can also be provisioned as code:

```yaml
# /etc/grafana/provisioning/alerting/rules/sqlserver.yml
apiVersion: 1
groups:
  - name: SQL Server Alerts
    interval: 30s
    rules:
      - alert: HighCPUUsage
        expr: avg by (instance) (rate(mssql_os_counter_value{counter_name="Process cpu usage%"}[5m])) > 85
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "SQL Server CPU usage is above 85% on {{ $labels.instance }}"
      - alert: LowPLE
        expr: avg by (instance) (mssql_counter_value{counter_name="Page life expectancy"}) < 300
        for: 3m
        labels:
          severity: critical
        annotations:
          summary: "Page Life Expectancy critically low on {{ $labels.instance }}"
```

This approach ensures alert configuration changes are tracked, reviewed, and auditable alongside application code changes.

### Scaling Monitoring for Large Environments

As the number of monitored SQL Server instances grows, consider these scaling strategies:

**Horizontal Prometheus sharding**:
- Group SQL Server instances by region, data center, or business unit
- Assign each group to a dedicated Prometheus server
- Use Thanos or Cortex for global query view

**Thanos deployment pattern**:
```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ Prometheus  │    │ Prometheus  │    │ Prometheus  │
│ Region 1    │    │ Region 2    │    │ Region 3    │
│ SQL Servers │    │ SQL Servers │    │ SQL Servers │
└──────┬──────┘    └──────┬──────┘    └──────┬──────┘
       │                  │                  │
       └──────────────────┼──────────────────┘
                          │
                    ┌─────▼─────┐
                    │Thanos Sidecar│
                    │(per Prometheus)│
                    └─────┬─────┘
                          │
                    ┌─────▼──────┐
                    │Thanos Query │◄── Grafana
                    │(global view)│
                    └────────────┘
```

**Metric deduplication**: Thanos deduplicates metrics across Prometheus replicas, enabling high availability without data duplication.

**Long-term storage**: Use Thanos Store Gateway with object storage (S3, GCS, Azure Blob) for retention beyond Prometheus's local storage limits.

### Disaster Recovery for Monitoring Stack

Monitoring is critical infrastructure and must be backed up:

- **Grafana**: Backup the Grafana database (PostgreSQL/SQLite) and provisioning files. Export dashboard JSON as a secondary fallback.
- **Prometheus**: Backup TSDB snapshots periodically. Prometheus has a snapshot API: `POST /api/v1/admin/tsdb/snapshot`. Store backups in object storage.
- **Alert rules**: Keep alert rule definitions in version control (provisioning files).
- **Notification channels**: Document channel configuration (webhook URLs, API keys) in a secrets manager or encrypted vault.

### Monitoring Cross-Database Dependencies

For applications that span multiple SQL Server instances (linked servers, distributed queries), create cross-instance dashboards:

- **Linked server query latency**: Measure latency of distributed queries by tracking duration spikes across instances
- **Cross-instance deadlocks**: Correlate deadlock alerts from multiple instances around the same timestamp
- **Application dependency map**: Use Grafana's node graph panel to visualize SQL Server dependency chains for application-tier services

### Security Hardening for Grafana

Production Grafana instances should be secured:

- **TLS termination**: Configure HTTPS with a valid certificate. Use a reverse proxy (Nginx, Caddy, Azure Application Gateway) for TLS termination and additional security headers.
- **Authentication**: Enforce strong authentication — integrate with OAuth 2.0, SAML, or LDAP for single sign-on.
- **Authorization**: Use Grafana teams and folders to control dashboard access. DBAs get Editor access to SQL Server dashboards; developers get Viewer access.
- **API key rotation**: Rotate Grafana service account tokens regularly. Use short-lived tokens for CI/CD pipeline integrations.
- **Audit logging**: Enable Grafana's audit log to track dashboard modifications, data source changes, and user access patterns.
- **Network isolation**: Place Grafana in a private network with access restricted to authorized users through a VPN or identity-aware proxy.

## Annotations — Deployment & Event Correlation

Grafana annotations are a powerful but underutilized feature for correlating SQL Server performance changes with external events.

### Types of Annotations

**Manual Annotations**: Add on-demand by holding Ctrl (or Cmd) and clicking on a graph panel. Use for marking incidents, deployments, or configuration changes during investigation.

**Dashboard Annotations**: Configured in Dashboard Settings > Annotations. Queries an external data source (like a PostgreSQL table) that records:

- `time` — timestamp of the event
- `title` — event title or ID
- `text` — detailed description or link to change request
- `tags` — comma-separated tags (deployment, incident, maintenance)

**API Annotations**: Grafana's HTTP API can create annotations programmatically. This is useful for CI/CD pipelines to mark deployments:

```
POST /api/annotations
{
  "dashboardUID": "sql-server-overview",
  "time": 1719480000000,
  "timeEnd": 1719483600000,
  "text": "Deployment v2.4.1 rolled out",
  "tags": ["deployment", "app-xyz"]
}
```

**Annotation from Prometheus**: Create an annotation query in Grafana that watches a Prometheus metric (like a gauge that spikes during deployments) and automatically creates annotations when the metric crosses a threshold.

### Practical Annotation Use Cases

- **Application Deployments**: Mark the time window for each deployment. If PLE drops or batch requests spike immediately after a deployment, the team can quickly correlate the change with the new release
- **Index Maintenance Windows**: Annotate index rebuild periods to explain temporary I/O spikes or queuing
- **Backup Operations**: Full backups, differentials, and log backups all impact I/O. Annotations help distinguish backup-related performance from application performance
- **Configuration Changes**: MAXDOP change, cost threshold for parallelism adjustment, or memory configuration updates — all should be annotated
- **Scaling Events**: CPU/memory/storage scaling in cloud environments
- **Incident Timelines**: Mark the start and end of incidents with tags for postmortem analysis

### Setting Up an Annotation Pipeline

1. Create a PostgreSQL/MySQL table accessible to Grafana as a data source
2. Configure CI/CD pipelines to INSERT rows into the table on each deployment
3. Configure the annotation query in Grafana: `SELECT time, title, text, tags FROM annotations WHERE $__timeFilter(time)`
4. Bind the annotation query to all relevant SQL Server dashboards

For teams using Kubernetes, the Keptn or ArgoCD event systems can automatically create annotations when deployments succeed or roll back.

## Troubleshooting — Common Issues & Gotchas

### Grafana Data Source Connection Issues

**Dashboard shows "No data" or "Data source not found"**

- Verify Prometheus is running: `curl http://prometheus:9090/api/v1/targets`
- Check the Prometheus data source URL in Grafana configuration
- Ensure the Prometheus data source has the correct access mode (Server vs. Browser)
- Verify firewall rules allow Grafana to reach Prometheus on port 9090

**Dashboard panels show "Bad gateway" or HTTP 502 errors**

- Prometheus is overloaded or unresponsive. Check Prometheus memory usage and query concurrency
- Increase Prometheus query timeout from default 1m to 2m in data source settings
- Simplify PromQL queries — remove topk() with very large k values, reduce rate() range windows
- Consider upgrading Prometheus hardware or splitting scrape targets across multiple Prometheus instances

**Some panels show data while others don't**

- Inconsistent collector configuration across instances. All instances must expose the same metrics for universal dashboard panels
- Missing labels in PromQL queries. Add `or on() vector(0)` to handle missing series (returns 0 instead of no data)
- The specific metric name changed between exporter versions. Verify metric names using the Prometheus expression browser

### PromQL Query Issues

**rate() returns no data or errors**

- rate() requires at least 4 samples. For a 15s scrape interval, the rate range must be at least 1m (60s)
- rate() cannot be applied to monotonic counters that reset. Use irate() for counters that reset frequently
- Ensure the metric is a counter type (monotonically increasing), not a gauge

**Cardinality explosion from query_hash labels**

The mssql_query_stats collector adds labels for `query_hash`, `database_name`, and `query_text`. Each unique query_hash creates a new time series. On servers with thousands of unique queries, this can generate hundreds of thousands of time series, overwhelming Prometheus storage and impacting query performance.

Solutions:
- Disable the query_stats collector entirely if not needed
- Pre-aggregate query metrics: only expose top N queries rather than all queries
- Reduce label cardinality by aggregating on server side before exposing
- Use recording rules to collapse query_hash metrics into aggregate buckets
- Set a lower max_samples per scrape in the exporter configuration

**Missing metrics after exporter restart**

Counter metrics reset to zero on exporter restart, causing rate() to show artificial negative spikes. Handle this by:
- Using `increase()` instead of `rate()` for counter resets (increase handles resets slightly better)
- Accepting the gap — short gaps in dashboard data are better than incorrect spikes
- Setting up alerting on the `up` metric to detect exporter restarts and annotate them

### Alerting Issues

**Alerts firing incorrectly or not firing**

- Check alert rule evaluation interval — rules may not fire if they require more samples than the evaluation interval provides
- Verify the PromQL query returns the expected results by running it in the Explore view
- Ensure alert rule timing conditions (for "5m" duration) are consistent with scrape interval
- Check for alert rule "no data" and "error" handling settings — choose "Alerting" if silence is dangerous, "No Data" if OK

**Notification channel not sending**

- Verify Grafana notification channel configuration (API keys, webhook URLs, authentication tokens)
- Check Grafana server logs for delivery errors
- For Slack: ensure the webhook URL is correct and the channel exists
- For email: SMTP configuration may require authentication or TLS settings

### Dashboard Import Issues

**Dashboard import fails or renders incorrectly**

- Dashboard 9628 expects specific metric names from the sql_exporter. If using Telegraf or a different collector, metric names differ and panels must be remapped
- Renaming panels is safe, but renaming internal metric references requires editing the JSON model
- Version differences: the dashboard may reference metrics from an older exporter version. Check compatibility in the dashboard description
- Some panels expect specific label names (e.g., instance, database, wait_type). Verify label names match your exporter output

### Performance Impact of Monitoring

**Prometheus scrape overhead**

At 15s scrape intervals with 6 collectors enabled, each scrape returns approximately 2,000-5,000 metrics (varies by workload). This imposes:
- ~10-50ms SQL Server CPU per scrape on the exporter queries
- ~0.5-2ms network latency per scrape
- ~50-200KB network bandwidth per scrape

For high-traffic OLTP systems, consider increasing the scrape interval to 30s and reducing the number of queries per scrape.

**Dashboard refresh impact**

Each dashboard panel query adds load on Prometheus. A dashboard with 40 panels and 10 concurrent users generates 400 queries at each refresh interval. Prometheus handles this well, but complex PromQL queries (topk with high k, multiple queries with large ranges) add latency.

Optimize by:
- Reducing dashboard auto-refresh interval (30s minimum for operational dashboards, 5m for capacity dashboards)
- Using dashboard caching (Grafana Enterprise feature)
- Implementing Prometheus query sharding for very large environments
