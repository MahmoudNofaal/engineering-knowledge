### Where This Fits

A non-equi JOIN uses `<`, `>`, `<=`, `>=`, `BETWEEN`, or `<>` in the ON predicate instead of `=`. The canonical use case is date-range overlap: finding orders placed when a promotion was active, prices that were effective at the time of an order (SCD Type 2), or customers whose order dates fall within a campaign window. Every .NET engineer working with temporal data, versioned entities, slowly changing dimensions, or event time-series will need non-equi joins at some point. The critical interview signal is understanding that **non-equi joins can ONLY use Nested Loops** — Hash Match and Merge Join require at least one equality condition in the join predicate. This means performance degrades to O(N × M) worst case unless the outer input is small or the inner table has a supporting index. The most common production mistake is writing a BETWEEN join on unindexed date columns, causing a 5-minute query that scans 50M rows per outer row.

## Core Mental Model

### Classification

Non-equi JOIN is a join predicate classification, not a T-SQL keyword. The ON clause uses inequality operators instead of (or in addition to) equality. The optimiser treats non-equi joins as requiring Nested Loops — it cannot use Hash Match or Merge Join because those operators depend on hash equality or sorted order matching. Non-equi predicates are SARGable when the column is on one side of the operator and the value is a constant or parameter: `OrderDate >= @StartDate` is SARGable. `DATEDIFF(day, OrderDate, @Date) BETWEEN 1 AND 7` is NOT SARGable because the column is wrapped in a function.

```mermaid
flowchart TD
    A[Non-equi JOIN: <, >, <=, >=, BETWEEN, <>] --> B{Physical join operator}
    B --> C[Nested Loops - ONLY option]
    B --> D[Hash Match - NOT supported]
    B --> E[Merge Join - NOT supported]
    C --> F{Outer input small?}
    F -->|Yes| G[Acceptable: O(N × log M) with index]
    F -->|No - large outer| H[Problematic: O(N × M) worst case]
    G --> I[Inner side must have index on range column]
    H --> J[Consider alternative: temp table,<br>window function, or rewrite]
    I --> K[Index Seek per outer row<br>on range predicate]
    K --> L[Index on range column supports<br>seek for >, >=, <, <=, BETWEEN]
```

### Key Properties

|Property|Value|Notes|
|---|---|---|
|Physical join operators|Nested Loops only|Hash Match and Merge Join require at least one equi predicate|
|Nested Loops complexity|O(N × M) worst case|Full scan per outer row if no index; O(N × log M) with index|
|SARGable|Yes (bare column)|`col > @val` SARGable; `FUNC(col) > @val` NOT SARGable|
|Index support|Seek on range column|Index on the range column enables range seek per outer row|
|NULL behavior|Excluded|`NULL > 5` is UNKNOWN — row excluded from result|
|Write Cost|None|JOINs are read-only|

## Deep Mechanics

### How the Engine Executes This

1. **Parsing** — The parser sees the inequality operator in the ON clause. It treats it identically to any other predicate — no special parsing rules for non-equi vs equi.

2. **Binding** — The algebrizer resolves column references and records the predicate type. At this stage, the optimiser notes that the join predicate contains no equality conditions.

3. **Simplification** — The optimiser checks:
   - Can the non-equi predicate be converted? `a.col1 > b.col1` is inherently non-equi — it cannot be converted to an equi-join.
   - If the non-equi predicate is combined with an equi predicate (`ON a.col1 = b.col1 AND a.col2 > b.col2`), the equi part enables Hash Match or Merge Join, with the non-equi part applied as a residual filter.
   - Predicate pushdown pushes the non-equi predicate as close to the data access as possible.

4. **Physical join operator selection — non-equi specific:**

   **Nested Loops is the ONLY option for purely non-equi joins:**
   - The outer input is scanned (or seeked if filtered). For each outer row, the inner table is searched for rows matching the non-equi predicate.
   - **With an index on the inner table's non-equi column:** SQL Server can perform an Index Seek for range predicates (`>`, `>=`, `<`, `<=`, `BETWEEN`). For example, for `ON a.OrderDate < b.ShipDate`, if `ShipDate` is indexed, the inner seek finds all rows with `ShipDate > OuterOrderDate`.
   - **Without an index:** The inner table is fully scanned for each outer row — O(N × M).
   - **For `<>` predicates:** Indexes are not helpful — `<>` cannot use an Index Seek. The inner side is always fully scanned for each outer row.

   **Hash Match:**
   - Hash Match requires at least one equi predicate to build the hash table. Purely non-equi joins cannot use Hash Match.
   - If the join has both equi and non-equi (`ON a.id = b.id AND a.date > b.date`), Hash Match can be used for the equi part, and the non-equi part is applied as a residual filter on the probe side.

   **Merge Join:**
   - Merge Join requires sorted inputs on the join key. Since non-equi predicates do not produce a sorted match (M:N relationships are common in non-equi joins), Merge Join is not supported for purely non-equi joins.

5. **Execution** — Nested Loops executes. For each outer row, the inner side is accessed. With an index on the range column, the inner access is a range seek. Without an index, it is a full scan.

```sql
-- Non-equi JOIN: price at time of order (SCD Type 2 lookup)
SELECT
    o.OrderId,
    o.OrderDate,
    oi.ProductId,
    oi.Quantity,
    pp.UnitPrice AS PriceAtOrderTime,
    (oi.Quantity * pp.UnitPrice) AS LineTotal
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi
    ON o.OrderId = oi.OrderId
INNER JOIN dbo.ProductPricing AS pp
    ON oi.ProductId = pp.ProductId
    AND o.OrderDate >= pp.EffectiveDate      -- non-equi: date range start
    AND o.OrderDate < pp.NextEffectiveDate   -- non-equi: date range end
WHERE o.OrderDate >= '2024-01-01'
ORDER BY o.OrderId;

-- Non-equi JOIN: date range overlap - promotions active during order
SELECT
    o.OrderId,
    o.OrderDate,
    p.PromotionName,
    p.DiscountPercent,
    oi.Quantity * oi.UnitPrice * (p.DiscountPercent / 100.0) AS DiscountAmount
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi
    ON o.OrderId = oi.OrderId
INNER JOIN dbo.Promotions AS p
    ON p.ProductId = oi.ProductId
    AND o.OrderDate >= p.StartDate           -- non-equi: order during promotion
    AND o.OrderDate <= p.EndDate             -- non-equi: order during promotion
WHERE o.OrderDate >= '2024-01-01'
ORDER BY o.OrderId;

-- Non-equi JOIN: BETWEEN for date range
SELECT
    o.OrderId,
    o.OrderDate,
    p.PromotionName,
    p.DiscountPercent
FROM dbo.Orders AS o
INNER JOIN dbo.Promotions AS p
    ON o.OrderDate BETWEEN p.StartDate AND p.EndDate
    AND p.ProductId = oi.ProductId  -- assumes OrderItems joined above
-- Note: BETWEEN is inclusive on both ends

-- Non-equi JOIN: segmentation by value ranges
-- Orders falling into customer tier value ranges
SELECT
    o.OrderId,
    o.TotalAmount,
    cs.TierName,
    cs.DiscountRate
FROM dbo.Orders AS o
INNER JOIN dbo.CustomerSegments AS cs
    ON o.TotalAmount >= cs.MinAmount         -- non-equi: range lower bound
    AND o.TotalAmount < cs.MaxAmount         -- non-equi: range upper bound
WHERE o.OrderDate >= '2024-01-01'
ORDER BY o.OrderId;

-- Non-equi anti-join: products never ordered (using <>)
SELECT p.ProductId, p.ProductName
FROM dbo.Products AS p
LEFT JOIN dbo.OrderItems AS oi
    ON p.ProductId <> oi.ProductId           -- non-equi: <> anti-join
WHERE oi.OrderItemId IS NULL;
-- Note: this is semantically wrong — use NOT EXISTS instead
-- This pattern is shown only for demonstration of <>

-- Non-equi JOIN: consecutive orders comparison
SELECT
    curr.OrderId,
    curr.OrderDate,
    prev.OrderId AS PreviousOrderId,
    prev.OrderDate AS PreviousOrderDate,
    DATEDIFF(day, prev.OrderDate, curr.OrderDate) AS DaysSinceLastOrder
FROM dbo.Orders AS curr
INNER JOIN dbo.Orders AS prev
    ON curr.CustomerId = prev.CustomerId    -- equi: same customer
    AND curr.OrderDate > prev.OrderDate     -- non-equi: previous order
    AND NOT EXISTS (                         -- non-equi: no order in between
        SELECT 1 FROM dbo.Orders AS mid
        WHERE mid.CustomerId = curr.CustomerId
          AND mid.OrderDate > prev.OrderDate
          AND mid.OrderDate < curr.OrderDate
    )
ORDER BY curr.CustomerId, curr.OrderDate;
```

```csharp
public async Task<List<OrderPricingDto>> GetOrderPricingInMemoryAsync(
    DateTime startDate,
    CancellationToken cancellationToken = default)
{
    var orders = await dbContext.Orders
        .Include(o => o.OrderItems)
            .ThenInclude(oi => oi.Product)
        .Where(o => o.OrderDate >= startDate)
        .ToListAsync(cancellationToken);

    var productIds = orders.SelectMany(o => o.OrderItems)
        .Select(oi => oi.ProductId)
        .Distinct()
        .ToList();

    var pricing = await dbContext.Set<ProductPricing>()
        .Where(p => productIds.Contains(p.ProductId))
        .ToListAsync(cancellationToken);

    // In-memory non-equi join
    var results = new List<OrderPricingDto>();
    foreach (var order in orders)
    {
        foreach (var item in order.OrderItems)
        {
            var price = pricing
                .Where(p => p.ProductId == item.ProductId
                            && order.OrderDate >= p.EffectiveDate
                            && order.OrderDate < p.NextEffectiveDate)
                .OrderByDescending(p => p.EffectiveDate)
                .FirstOrDefault();

            results.Add(new OrderPricingDto
            {
                OrderId = order.OrderId,
                OrderDate = order.OrderDate,
                ProductId = item.ProductId,
                Quantity = item.Quantity,
                PriceAtOrderTime = price?.UnitPrice ?? 0
            });
        }
    }
    return results;
}
```

### Execution Plan Analysis

**Non-equi JOIN with index on range column (Nested Loops):**

```
  [Index Seek (IX_Orders_OrderDate)]  -- outer: filtered, 10K rows
  → [Nested Loops (Inner Join)]
      [Index Seek (IX_ProductPricing_ProductId_Date)]  -- inner: range seek
          Seek Predicate: ProductId = OrderItems.ProductId
                       AND EffectiveDate <= Orders.OrderDate
                       AND NextEffectiveDate > Orders.OrderDate
  → [SELECT]
Estimated Cost: ~4.5  |  Logical Reads: ~1,200 (10K outer + 10K seeks × ~2 pages)
```

**Non-equi JOIN WITHOUT index on range column (full scan per outer row):**

```
  [Index Seek (IX_Orders_OrderDate)]  -- outer: 10K rows
  → [Nested Loops (Inner Join)]
      [Table Scan (ProductPricing)]    -- inner: full scan per outer row
          — No Seek Predicate — full scan for each of 10K outer rows
  → [SELECT]
Estimated Cost: ~850  |  Logical Reads: ~85,000,000 (10K outer × 8,500 pages per scan)
```

**Non-equi anti-join (`<>` in ON clause):**

```
  [Table Scan (Products)]  -- outer: 50K rows
  → [Nested Loops (Left Anti Semi Join)]  -- conceptually
      [Table Scan (OrderItems)]  -- inner: full scan per outer row (5M rows)
  Estimated Cost: ~250,000 (50K × 5M rows evaluated)
  -- This is catastrophically expensive
```

### Failure Modes

**Non-equi join without index on range column:** The most expensive failure mode. A non-equi join with 10K outer rows and a 100K inner table without an index on the range column causes 10K full scans of the inner table = 1B row comparisons. This query takes minutes. The fix: create an index on the range column(s) used in the non-equi predicate.

**Using `<>` for anti-join:** `<>` in the ON clause cannot use any index. Each outer row triggers a full scan of the inner table. For 50K products × 5M order items, this is 250B comparisons — the query will never complete. Always use `NOT EXISTS` for anti-joins: `WHERE NOT EXISTS (SELECT 1 FROM OrderItems WHERE ProductId = Products.ProductId)`.

**Cross join + WHERE filter disguise:** EF Core's attempt at non-equi join generates `CROSS JOIN ProductPricing WHERE o.OrderDate >= pp.EffectiveDate`. This is a Cartesian product of both tables before filtering — for 1M orders × 500K pricing rows = 500B logical comparisons. The `CROSS JOIN` produces 500B intermediate rows in the plan. The query will exhaust tempdb and never return.

**Non-SARGable non-equi predicate:** `WHERE DATEDIFF(day, OrderDate, @Date) BETWEEN 1 AND 7` wraps the column in a function, making it non-SARGable. The index on `OrderDate` cannot be used. Rewrite as: `OrderDate >= DATEADD(day, -7, @Date) AND OrderDate < @Date`.

```sql
-- ============================================================
-- Pattern 5: Self non-equi join — compare rows within table
-- (used with equi predicate to avoid O(N²))
-- ============================================================
-- Find 3 most recent price changes for each product
SELECT
    a.ProductId,
    a.EffectiveDate AS FromDate,
    b.EffectiveDate AS ToDate,
    a.UnitPrice AS OldPrice,
    b.UnitPrice AS NewPrice,
    (b.UnitPrice - a.UnitPrice) AS PriceChange
FROM dbo.ProductPricing AS a
INNER JOIN dbo.ProductPricing AS b
    ON a.ProductId = b.ProductId
    AND a.EffectiveDate < b.EffectiveDate       -- non-equi
    AND NOT EXISTS (                              -- no pricing between
        SELECT 1
        FROM dbo.ProductPricing AS mid
        WHERE mid.ProductId = a.ProductId
          AND mid.EffectiveDate > a.EffectiveDate
          AND mid.EffectiveDate < b.EffectiveDate
    )
ORDER BY a.ProductId, a.EffectiveDate;

-- ============================================================
-- Pattern 6: Non-equi join with aggregation — running total
-- ============================================================
WITH DailySales AS (
    SELECT
        o.OrderDate,
        SUM(oi.Quantity * oi.UnitPrice) AS DailyRevenue
    FROM dbo.Orders AS o
    INNER JOIN dbo.OrderItems AS oi
        ON o.OrderId = oi.OrderId
    WHERE o.OrderDate >= '2024-01-01'
    GROUP BY o.OrderDate
)
SELECT
    curr.OrderDate,
    curr.DailyRevenue,
    SUM(prev.DailyRevenue) AS RunningTotal
FROM DailySales AS curr
INNER JOIN DailySales AS prev
    ON prev.OrderDate <= curr.OrderDate          -- non-equi: cumulative
GROUP BY curr.OrderDate, curr.DailyRevenue
ORDER BY curr.OrderDate;

-- Pattern 7: Anti-join correctly implemented with NOT EXISTS
-- (NOT the wrong <> approach)
-- ============================================================
-- Correct: products never ordered
SELECT p.ProductId, p.ProductName
FROM dbo.Products AS p
WHERE NOT EXISTS (
    SELECT 1
    FROM dbo.OrderItems AS oi
    WHERE oi.ProductId = p.ProductId
)
ORDER BY p.ProductName;

-- Correct: employees who never managed anyone
SELECT e.EmployeeId, e.FirstName + ' ' + e.LastName AS EmployeeName
FROM dbo.Employees AS e

    FROM dbo.Employees AS sub
    WHERE sub.ManagerId = e.EmployeeId
```

## Gotchas and Production Pitfalls

### Gotcha 1 — Non-Equi Join Without Index Causes O(N × M) Scan

**Pitfall:** Writing a non-equi join to find the price at the time of order without an index on the ProductPricing date columns.

```sql
-- ❌ No index on (ProductId, EffectiveDate, NextEffectiveDate)
SELECT o.OrderId, oi.ProductId, pp.UnitPrice
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.ProductPricing AS pp
    ON oi.ProductId = pp.ProductId
    AND o.OrderDate >= pp.EffectiveDate
    AND o.OrderDate < pp.NextEffectiveDate;
```

**Symptom:** The execution plan shows a Table Scan of ProductPricing for each outer row. For 10K orders and 500K pricing rows, this is 10K × 500K = 5B row comparisons. The query runs for 5+ minutes.

**Fix:**

```sql
-- ✅ Create composite index covering the equi + non-equi columns
CREATE INDEX IX_ProductPricing_ProductId_Dates
    ON dbo.ProductPricing (ProductId, EffectiveDate, NextEffectiveDate)
    INCLUDE (UnitPrice, CostPrice);
```

**Cost of not fixing:** The query that should take 50ms takes 5 minutes. The application times out. Users refresh and cause queue buildup. The on-call engineer kills the query and tempdb has grown by 10GB.

### Gotcha 2 — EF Core Generates CROSS JOIN for Non-Equi Predicates

**Pitfall:** Attempting a non-equi join in EF Core LINQ using `Join()` with a `Where()` clause:

```csharp
// ❌ EF Core generates CROSS JOIN (Cartesian product)
var results = await dbContext.Orders
    .Join(
        dbContext.ProductPricing,
        o => o.OrderId,         // EF Core translates Join to equi only
        pp => pp.ProductId,
        (o, pp) => new { o, pp })
    .Where(x => x.o.OrderDate >= x.pp.EffectiveDate)
    .Select(x => new { x.o.OrderId, x.pp.UnitPrice })
    .ToListAsync(cancellationToken);
```

**Symptom:** Generated SQL is `FROM Orders CROSS JOIN ProductPricing WHERE OrderDate >= EffectiveDate`. This creates a Cartesian product of both tables before filtering. For 1M orders × 500K pricing rows = 500B intermediate rows. The query exhausts tempdb and never returns.

**Fix:**

```csharp
// ✅ Use raw SQL for non-equi joins
var results = await dbContext.Database
    .SqlQuery<OrderPricingDto>(@"
        SELECT o.OrderId, pp.UnitPrice
        FROM Orders o
        INNER JOIN ProductPricing pp
            ON o.OrderDate >= pp.EffectiveDate
            AND o.OrderDate < pp.NextEffectiveDate")
    .ToListAsync(cancellationToken);
```

**Cost of not fixing:** The endpoint returns HTTP 503 after 30 seconds. The error log shows "Query timeout expired". The DBA sees a query consuming 100% CPU with a Cartesian product estimated at 500B rows.

### Gotcha 3 — `<>` in ON Clause Is Catastrophically Slow

**Pitfall:** Using `<>` (not equal) as the non-equi join predicate for anti-join logic:

```sql
-- ❌ Wrong: catastrophic O(N × M)
SELECT p.ProductId, p.ProductName
FROM dbo.Products AS p
LEFT JOIN dbo.OrderItems AS oi
    ON p.ProductId <> oi.ProductId    -- NOT EQUAL
WHERE oi.OrderItemId IS NULL;
```

**Symptom:** The query never completes. The execution plan shows Nested Loops with a full clustered index scan of OrderItems (5M rows) for each Product (50K rows). This is 250B row comparisons. SQL Server runs out of tempdb and the query is killed.

**Fix:**

```sql
-- ✅ Use NOT EXISTS — can use index and semi-join
SELECT p.ProductId, p.ProductName
FROM dbo.Products AS p
WHERE NOT EXISTS (
    SELECT 1
    FROM dbo.OrderItems AS oi
    WHERE oi.ProductId = p.ProductId  -- equi predicate, index seeks
);
```

**Cost of not fixing:** The query runs indefinitely. The DBA must kill it. The application shows no results. The bug makes it to production because the developer tested on 100 rows where it worked "fine."

### Gotcha 4 — Non-SARGable Non-Equi Predicate

**Pitfall:** Using a function on the column in the non-equi predicate:

```sql
-- ❌ Not SARGable — YEAR() wraps the column
SELECT o.OrderId, cs.TierName
FROM dbo.Orders AS o
INNER JOIN dbo.CustomerSegments AS cs
    ON o.TotalAmount >= cs.MinAmount
    AND o.TotalAmount < cs.MaxAmount
WHERE YEAR(o.OrderDate) = 2024;
```

**Symptom:** The index on `OrderDate` is not used. The plan shows a Clustered Index Scan instead of an Index Seek. The WHERE filter is applied as a residual predicate.

**Fix:**

```sql
-- ✅ SARGable — bare column comparison
SELECT o.OrderId, cs.TierName
FROM dbo.Orders AS o
INNER JOIN dbo.CustomerSegments AS cs
    ON o.TotalAmount >= cs.MinAmount
    AND o.TotalAmount < cs.MaxAmount
WHERE o.OrderDate >= '2024-01-01'
  AND o.OrderDate < '2025-01-01';
```

**Cost of not fixing:** Full table scan on a 10M row Orders table adds 2-3 seconds per query. If this runs 10,000 times/hour, the cumulative extra I/O is 20M logical reads/hour.

### Gotcha 5 — Running Total Non-Equi Self Join vs Window Function

**Pitfall:** Using a non-equi self join to compute running totals:

```sql
-- ❌ Expensive: non-equi self join for running total
SELECT curr.OrderDate, curr.DailyRevenue,
       SUM(prev.DailyRevenue) AS RunningTotal
FROM DailySales AS curr
INNER JOIN DailySales AS prev
    ON prev.OrderDate <= curr.OrderDate
GROUP BY curr.OrderDate, curr.DailyRevenue
ORDER BY curr.OrderDate;
-- For 365 days, this does 365 × 365 / 2 = 66,705 row comparisons
-- For 5 years (1825 days): 1.6M comparisons
```

**Symptom:** The query works on small date ranges but becomes slow as history grows. At 10 years of daily data (3652 days), the non-equi self join processes 6.6M row comparisons.

**Fix:**

```sql
-- ✅ Use window function — single pass
SELECT OrderDate, DailyRevenue,
       SUM(DailyRevenue) OVER (ORDER BY OrderDate
           ROWS UNBOUNDED PRECEDING) AS RunningTotal
FROM DailySales
ORDER BY OrderDate;
```

**Cost of not fixing:** The running total query that should complete in 50ms takes 5 seconds for 5 years of data. The dashboard page that shows running revenue totals times out for the "All Time" range.

## Interview Arsenal

### Question Bank

1. **What is a non-equi JOIN and what operators are used in its execution plan?** — Definition: JOIN with <, >, <=, >=, BETWEEN, <> in the ON predicate. Only Nested Loops is supported for purely non-equi joins.

2. **Why can Hash Match and Merge Join not be used for non-equi joins?** — Hash Match requires equality to compute hash keys. Merge Join requires sorted equality matches. Non-equi predicates operate on ranges and inequalities, which are incompatible with both.

3. **What is the performance implication of a non-equi join without an index on the range column?** — O(N × M) — each outer row triggers a full scan of the inner table. With an index, O(N × log M).

4. **When should you use `<>` in a join predicate — and what should you use instead?** — Never. Use `NOT EXISTS` for anti-joins. `<>` forces full scans on both sides and cannot use indexes.

5. **Non-equi JOIN vs Window Function (LAG/LEAD) for sequential data — which is better?** — Window functions are vastly better: single pass vs O(N²) self join. Use LAG/LEAD for previous/next row comparisons, running totals, and moving averages.

6. **What does the execution plan look like for a non-equi join with a properly indexed date range column?** — Index Seek on outer filter → Nested Loops → Index Seek (range seek) on inner table's composite index (equi + non-equi columns) → SELECT.

7. **How does a non-equi join behave at 100M rows?** — Without index: impossible — 100M × inner_scan = 100M scans. With composite index: feasible only if outer input is filtered to < 10K rows. At larger volumes, restructure the data model (denormalise the effective price onto the OrderItems row).

8. **How do EF Core and Dapper handle non-equi joins?** — EF Core: raw SQL via `SqlQuery<T>()` or `FromSqlRaw`. LINQ does not support non-equi in Join. Dapper: raw SQL with `QueryAsync<T>` — identical to any other query.

### Spoken Answers

**Q: What is a non-equi JOIN and what operators are used in its execution plan?**

> **Average answer:** A JOIN that uses < or > instead of =. It uses Nested Loops.

> **Great answer:** A non-equi JOIN uses inequality operators — `<`, `>`, `<=`, `>=`, `BETWEEN`, or `<>` — in the ON predicate instead of or in addition to `=`. The critical constraint is that only the Nested Loops physical join operator supports purely non-equi predicates. Hash Match cannot be used because it needs an equality condition to compute the hash value for the hash table. Merge Join cannot be used because it relies on sorted equality — the inputs would need to be sorted on the non-equi column, but Merge Join's advance-and-compare algorithm depends on matching keys, which does not work for inequality ranges. This means the execution plan for a purely non-equi join always shows a Nested Loops operator. If the inner table has a useful index on the range column, SQL Server can perform a range seek for each outer row — the execution plan shows an Index Seek with a seek predicate like `EffectiveDate <= outer.OrderDate`. Without the index, the plan shows a full table scan per outer row — O(N × M). The most common production pattern is a date-range join with a composite index: the first column of the index covers the equi part (like `ProductId`), and the remaining columns cover the range part (like `EffectiveDate` and `NextEffectiveDate`).

**Q: Non-equi JOIN vs Window Function (LAG/LEAD) for sequential data — which is better?**

> **Average answer:** Window functions are faster. Use LAG/LEAD.

> **Great answer:** Window functions are always the better choice for sequential data comparisons. A non-equi self join to find the previous order for each customer requires the ON clause `curr.CustomerId = prev.CustomerId AND curr.OrderDate > prev.OrderDate AND NOT EXISTS (no orders in between)`. This produces a Nested Loops plan that scans the OrderHistory table for each row — O(N²) in the worst case. At 100K orders, that is 5B row comparisons. The LAG window function does the same work in a single pass — it reads the table once, orders it by CustomerId and OrderDate, and applies the LAG analytic function to each row. The execution plan shows a single Clustered Index Scan (or Index Scan on the cover index) followed by a Segment + Sequence Project operator for the window function. Logical reads are typically under 1,000 vs 250,000 for the non-equi self join. The only reason to use a non-equi join instead of a window function is when the relationship cannot be expressed as a fixed offset — for example, finding the product price at the time of each order (SCD Type 2) where there is no fixed row offset, only a date range comparison. For any "previous row" or "next row" pattern, LAG and LEAD are the correct tools.

### Interview Trigger

If you describe a date-range join for SCD Type 2 lookup, the interviewer will likely ask: "The query is slow on a 100M row fact table. What do you check?" The deep follow-up is: "What if you cannot create an index on the range column — how else can you solve this?" — testing whether the candidate considers denormalising the effective price onto the fact row at ETL time, using a window function approach, or restructuring the data model to eliminate the non-equi join entirely.

### Comparison Table

| | Non-Equi JOIN (Range) | Non-Equi JOIN (`<>`) | Window Function (LAG/LEAD) |
|---|---|---|---|
| What it does | Range comparison in ON clause | Inequality comparison in ON clause | Analytic function for row offset |
| Performance profile | O(N × log M) with index; O(N × M) without | O(N × M) always — catastrophically bad | O(N log N) — single pass + sort |
| Supported join operators | Nested Loops only | Nested Loops only | No join — analytic function |
| EF Core support | Raw SQL required | Raw SQL required | LINQ with Select + EF.Functions |
| Dapper support | Raw SQL | Raw SQL | Raw SQL with OVER clause |
| When to choose | SCD Type 2, date range overlap, segmentation | NEVER — use NOT EXISTS | Previous/next row, running totals, moving avg |

## Decision Framework

### When to Apply

```mermaid
flowchart TD
    A[JOIN predicate includes <br><, >, <=, >=, BETWEEN, <>] --> B{What type of comparison?}
    B -->|Date range overlap / SCD Type 2| C{Inner table has index<br>on range column?}
    B -->|Anti-join / not-equal| D[Do NOT use <> in ON clause<br>Use NOT EXISTS instead]
    B -->|Running total / sequential| E[Use window function<br>instead of self non-equi join]
    C -->|Yes| F[Use Nested Loops<br>with range seek]
    C -->|No| G[Create composite index<br>(equi col first, range col second)]
    F --> H[Expected logical reads:<br>outer N × ~3 per seek]
    G --> I[After index: same plan as F]
    D --> J[Correct: NOT EXISTS uses<br>semi-join with index seek]
    E --> K[LAG/LEAD/SUM OVER: single pass]
    K --> L[Logical reads: ~table size<br>no O(N²) risk]
```

### Application Checklist

- [ ] The ON predicate uses <, >, <=, >=, BETWEEN (not `<>`)
- [ ] An index exists on the inner table covering the equi columns first, then the range column(s)
- [ ] The outer input is expected to be small (< 10K rows) — verify with actual vs estimated rows
- [ ] The non-equi predicate is SARGable (bare column on one side, value/parameter on the other)
- [ ] EF Core is using raw SQL (`SqlQuery<T>` or `FromSqlRaw`) — not `Join()` with `Where()`
- [ ] Dapper uses raw SQL with proper parameterisation
- [ ] Window functions are used instead of non-equi self joins for running totals and previous/next row
- [ ] `<>` in ON clause is not present — anti-joins use `NOT EXISTS`

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Expresses range-based relationships naturally|Nested Loops only — scale limited by outer input size|
|Enables SCD Type 2 and temporal queries|Requires careful index design for acceptable performance|
|Single query instead of multiple round trips|Cannot use Hash Match or Merge Join|
|ANSI SQL standard (works everywhere)|EF Core LINQ does not support it — raw SQL required|

### Scale Thresholds

- **Relevant when outer input exceeds ~1K rows** — Below 1K, even full scans of the inner table are acceptable
- **Index on range column is critical when outer input exceeds ~10K rows** — Without index, 10K × inner_table_scan becomes minutes instead of milliseconds
- **Non-equi join becomes impractical when outer input exceeds ~100K rows** — Even with indexes, 100K range seeks add up. Consider denormalising the data at ETL time instead
- **`<>` anti-join becomes impractical at ANY scale beyond 1K rows** — The O(N × M) cost makes it unusable for any realistic table size

**Challenge 1 — Write the SQL**

You have `Orders` (OrderId, OrderDate, CustomerId, TotalAmount) and `CustomerSegments` (SegmentId, TierName, MinAmount, MaxAmount, DiscountRate). Write a query that, for each order placed in January 2025, determines which customer tier the order falls into based on its TotalAmount. A tier is defined as `TotalAmount >= MinAmount AND TotalAmount < MaxAmount`. Include OrderId, OrderDate, TotalAmount, TierName, and DiscountRate.

<details>
<summary>Solution</summary>

```sql
SELECT
    o.OrderId,
    o.OrderDate,
    o.TotalAmount,
    cs.TierName,
    cs.DiscountRate
FROM dbo.Orders AS o
INNER JOIN dbo.CustomerSegments AS cs
    ON o.TotalAmount >= cs.MinAmount
    AND o.TotalAmount < cs.MaxAmount
WHERE o.OrderDate >= '2025-01-01'
  AND o.OrderDate < '2025-02-01'
ORDER BY o.OrderId;
```

**Logical reads:** ~550 (index seek on Orders + clustered index scan of CustomerSegments — small table)
**Execution plan:** [Index Seek on IX_Orders_OrderDate] → [Nested Loops] → [Clustered Index Scan on CustomerSegments] → [SELECT]
**EF Core equivalent:**

```csharp

        AND o.TotalAmount < cs.MaxAmount
```

**Challenge 3 — Explain the execution plan**

```sql
-- Query: price lookup with non-equi date range
SELECT o.OrderId, o.OrderDate, pp.UnitPrice
FROM Orders o
INNER JOIN ProductPricing pp
    ON o.OrderDate >= pp.EffectiveDate
    AND o.OrderDate < pp.NextEffectiveDate
WHERE o.OrderId = 1001;

-- Plan A (actual):
-- Clustered Index Seek (PK_Orders, OrderId=1001) → Nested Loops
--   → Index Seek (IX_ProductPricing_Dates)
--       Seek Predicate: EffectiveDate <= '2024-06-15'
--                    AND NextEffectiveDate > '2024-06-15'
-- Estimated rows: 1 | Actual rows: 1 | Cost: 0.045

-- Plan B (hypothetical — without the index):
-- Clustered Index Seek (PK_Orders, OrderId=1001) → Nested Loops
--   → Clustered Index Scan (ProductPricing)
--       No seek predicate — full scan
-- Estimated rows: 1 | Actual rows: 1 | Cost: 8.5
```

Why does Plan A have a cost of 0.045 (almost free) while Plan B costs 8.5?

<details> <summary>Solution</summary>

**Why Plan A is cheap:** The index `IX_ProductPricing_Dates` on `(EffectiveDate, NextEffectiveDate)` enables a range seek. Given the outer value `OrderDate = '2024-06-15'`, SQL Server seeks to the first row where `EffectiveDate <= '2024-06-15'`, then scans forward until `NextEffectiveDate > '2024-06-15'` is false. The number of pages read is typically 3-4 (B-tree traversal + 1 leaf page). The index is highly selective because each date range defines a small number of matching rows.

**Why Plan B is expensive:** Without the index, SQL Server must scan the entire ProductPricing table (clustered index) for each outer row. For 500K pricing rows, this is 8,500+ logical reads per outer row. Even with one outer row (OrderId = 1001), the cost is 8,500 reads vs 4 reads.

**To get Plan B (bad):** Drop the index on `(EffectiveDate, NextEffectiveDate)`.

**Tradeoff:** The index adds write overhead: each INSERT into ProductPricing adds an entry to the non-clustered index (approximately 20 bytes per row). For 500K rows, the index size is ~10MB. Write cost: ~1 additional page write per 800 rows inserted.

</details>

**Challenge 4 — Diagnose the concurrency problem**

An hourly job runs a non-equi join query that finds all orders placed during active promotions. The query joins `Orders` (500K rows filtered to last hour = ~1000 rows) to `Promotions` (200K rows) on `o.OrderDate BETWEEN p.StartDate AND p.EndDate`. The query uses `READ COMMITTED` isolation. After a promotion update that inserts 50K new promotion rows covering broad date ranges, the query starts taking 30 seconds instead of 200ms. `sys.dm_exec_query_stats` shows `wait_type = PAGEIOLATCH_SH` with high `avg_read_stall_ms`. The execution plan shows Nested Loops with a full scan of Promotions per outer row.

<details> <summary>Solution</summary>

**Root cause:** The new promotion rows caused the optimiser to change the plan. The index on Promotions `(StartDate, EndDate)` was previously selective enough for range seeks, but the new 50K "always active" promotions (covering all dates) made the range seek return too many rows. The optimiser's cardinality estimate crossed the threshold where it switched from Index Seek to Table Scan — because seeking 1K times and getting 50K rows each = 50M rows returned, which is more expensive than a single scan of 200K rows. However, the single scan is still per outer row: 1K outer × 200K inner = 200M row comparisons, which is even worse.

**Detection query:**

```sql
SELECT
    OBJECT_NAME(s.object_id) AS TableName,
    s.name AS IndexName,
    sp.rows AS RowCount,
    sp.modification_counter AS Modifications,
    STATS_DATE(s.object_id, s.stats_id) AS StatsUpdated
FROM sys.stats s
CROSS APPLY sys.dm_db_stats_properties(s.object_id, s.stats_id) sp
WHERE s.object_id = OBJECT_ID('Promotions');
```

**Fix:**

```sql
-- Option 1: Update statistics
UPDATE STATISTICS Promotions IX_Promotions_Dates WITH FULLSCAN;

-- Option 2: Add a covering index specific to this query
CREATE INDEX IX_Promotions_ActiveDates ON Promotions (StartDate, EndDate)
    INCLUDE (PromotionName, DiscountPercent, ProductId);

-- Option 3: Restructure query to filter active promotions first
WITH ActivePromotions AS (
    SELECT ProductId, PromotionName, DiscountPercent,
           StartDate, EndDate
    FROM dbo.Promotions
    WHERE StartDate <= GETUTCDATE()
      AND EndDate >= DATEADD(hour, -1, GETUTCDATE())
)
SELECT o.OrderId, o.OrderDate, ap.PromotionName, ap.DiscountPercent
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN ActivePromotions AS ap
    ON oi.ProductId = ap.ProductId
    AND o.OrderDate BETWEEN ap.StartDate AND ap.EndDate
WHERE o.OrderDate >= DATEADD(hour, -1, GETUTCDATE());
```

**Challenge 5 — Design the index**

**Scenario:** An e-commerce platform has an `Orders` table (2M rows) and a `Promotions` table (100K rows). The most expensive query is a non-equi join that finds promotions active when each order was placed:

```sql
SELECT o.OrderId, o.OrderDate, oi.ProductId,
       p.PromotionName, p.DiscountPercent
FROM Orders o
INNER JOIN OrderItems oi ON o.OrderId = oi.OrderId
INNER JOIN Promotions p
    ON oi.ProductId = p.ProductId
    AND o.OrderDate BETWEEN p.StartDate AND p.EndDate
WHERE o.OrderDate >= '2025-01-01';
```

The query runs hourly, processing ~50K new orders each time. 80% of promotions have a specific ProductId (the other 20% are global across all products). The read/write ratio is 90/10 on Promotions (new promotions added daily).

Design the optimal index strategy.

<details> <summary>Solution</summary>

```sql
-- Index 1: Covering composite index for the primary non-equi join
-- ProductId first (equi part), then StartDate/EndDate (range part)
CREATE INDEX IX_Promotions_ProductId_Dates
    ON dbo.Promotions (ProductId, StartDate, EndDate)
    INCLUDE (PromotionName, DiscountPercent);

-- Index 2: Supporting index for global promotions (ProductId IS NULL)
CREATE INDEX IX_Promotions_Global_Dates
    ON dbo.Promotions (StartDate, EndDate)
    INCLUDE (PromotionName, DiscountPercent)
    WHERE ProductId IS NULL;

-- Index 3: Cover the OrderItems side of the join
CREATE INDEX IX_OrderItems_ProductId_OrderId
    ON dbo.OrderItems (ProductId, OrderId)
    INCLUDE (Quantity, UnitPrice);
```

**Explain:** `IX_Promotions_ProductId_Dates` is the primary index. The ProductId column is first for the equality seek (`oi.ProductId = p.ProductId`). The StartDate and EndDate columns enable the range seek (`o.OrderDate BETWEEN p.StartDate AND p.EndDate`). The INCLUDE columns cover the SELECT without key lookups. `IX_Promotions_Global_Dates` is a filtered index for promotions that apply to all products (ProductId IS NULL) — this avoids scanning rows that do not match any specific product. `IX_OrderItems_ProductId_OrderId` covers the OrderItems side of the join.

**Tradeoffs:** The composite index adds ~30 bytes per row × 100K promotions = ~3MB. Write overhead: each INSERT into Promotions adds ~2 index entries (one for each index) at ~1 additional page write. At 10 new promotions/hour, this is negligible.

**What NOT to index:** Do not create a separate single-column index on `StartDate` alone — the composite index with ProductId as the leading column is more selective for the typical query pattern (promotions for a specific product). Do not include `OrderId` in the Promotions index — it is not used in the Promotions join predicate.

</details>
