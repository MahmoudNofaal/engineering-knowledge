### Where This Fits

A composite JOIN uses two or more columns in the ON predicate: `ON a.col1 = b.col1 AND a.col2 = b.col2`. This is required when the relationship between tables uses a compound foreign key — a natural key spanning multiple columns, a composite primary key, or a dimensional model with degenerate dimensions. Every .NET engineer working with legacy databases (which often use composite keys), data warehouse fact-to-dimension joins (which join on multiple key columns), or junction tables with additional columns reaches for composite joins. The interview signal: composite join tests whether a candidate understands that the column ORDER in the ON predicate must match the column ORDER in the index for Merge Join, and that Hash Match can handle composite keys in any order but at higher memory cost. The most common production mistake is creating a composite index with the wrong column order — leading to Index Scan instead of Index Seek.

## Core Mental Model

A composite JOIN pairs rows from two tables using equality on multiple columns. The logical model is: treat the paired columns as a tuple (col1, col2) and compare the tuples for equality. SQL Server evaluates the composite predicate as a conjunction of equalities: row_a matches row_b when `a.col1 = b.col1 AND a.col2 = b.col2`. The optimiser treats composite keys as a single compound key for physical join selection. For Nested Loops, the inner side needs a composite index with the key columns in the same order as the ON predicate. For Merge Join, both inputs must be sorted on ALL join columns in the same order — a composite sorted input from a composite index. Hash Match hashes the concatenation of the join column values; the hash is computed on the combined key, so an index on one column alone does not help. The critical performance insight: composite join efficiency depends on a composite index whose leading column is the most selective, or more precisely, whose column order matches the join order and the ON predicate column order exactly.

### Classification

Composite JOINs are equi-joins with a compound ON predicate. They are fully SARGable when a composite index exists with the join columns in the same order. SQL Server can use a composite index seek for the entire key (seek on col1 AND col2) or for the leading column only (seek on col1, residual predicate on col2). The composite join is the foundation of star-schema fact-to-dimension joins, where a fact table references a dimension table through a surrogate key that is often a single INT column but may also be a natural key across 2-3 columns.

## Deep Mechanics

### How the Engine Executes This

1. **Parsing** — The parser sees the ON clause with two equality predicates joined by AND. It parses this as a conjunction of search conditions: `(a.col1 = b.col1) AND (a.col2 = b.col2)`.

2. **Binding** — The algebrizer resolves all column references and builds the join graph. Each equality pair becomes an edge in the join condition. The algebrizer notes that this is a multi-column equi-join, qualifying it for all three physical join operators.

3. **Simplification** — The optimiser may:
   - **Reorder columns in the ON predicate** (the optimiser can commute the AND-ed equalities — `a.col2 = b.col2 AND a.col1 = b.col1` is treated the same as the written order).
   - **Push ON clause predicates down** — each equality can be pushed to the appropriate table's access path if the join is a Nested Loops join.
   - **Detect implied predicates** — if the composite join includes all columns of a unique constraint, the optimiser knows the join is one-to-one or many-to-one, which influences cardinality estimation.

4. **Physical join operator selection — composite specifics:**

   **Nested Loops with composite index:**
   - The outer input is scanned. For each outer row, the inner side performs a composite index seek on the leading column. If the index is `(col1, col2)` and the join is `ON a.col1 = b.col1 AND a.col2 = b.col2`, the seek uses both columns — O(log N) to find the starting position, then a scan of matching rows.
   - If the index is `(col2, col1)` but the join predicate uses `a.col1 = b.col1 AND a.col2 = b.col2`, the index seek uses only `col2` (the leading column) and `col1` is applied as a residual predicate. This is less efficient.

   **Hash Match with composite key:**
   - The hash function is computed on the concatenation of all join columns. SQL Server converts the composite key to a single hash key by combining the column values. The hash table stores the full composite key for collision detection.
   - Index order does not matter for Hash Match — it works the same regardless of column order.

   **Merge Join with composite key:**
   - Both inputs must be sorted on ALL join columns in the same order. A composite index `(col1, col2)` provides sorted input on (col1, col2). A composite index `(col2, col1)` provides sorted input on (col2, col1) — the Merge Join cannot use it if the join predicate order is (col1, col2) because the sort order does not match.
   - The Merge Join advances through both inputs comparing the composite key tuple lexicographically: compare col1; if equal, compare col2; if both equal, output match and advance both.

5. **Execution** — The chosen operator runs. For Nested Loops with composite seek, each outer row triggers one seek on the composite index. For Hash Match, the build input is scanned, hash table built on the composite key, and probe input scanned. For Merge Join, both sorted inputs are scanned concurrently.

SELECT
    o.OrderId,
    o.OrderDate,
    oi.ProductId,
    oi.Quantity,
    p.ProductName,
    pr.UnitPrice AS HistoricalPrice,

INNER JOIN dbo.Products AS p

        FROM dbo.ProductPricing AS sub
        WHERE sub.ProductId = oi.ProductId
          AND sub.EffectiveDate <= o.OrderDate

-- Composite on (Email, Phone) to find potential duplicates

    a.CustomerId AS OriginalId,
    a.FirstName + ' ' + a.LastName AS OriginalName,
    b.CustomerId AS DuplicateId,
    b.FirstName + ' ' + b.LastName AS DuplicateName,
    a.Email,
    a.Phone

    AND a.Phone = b.Phone

  AND a.Phone IS NOT NULL

### Execution Plan Analysis

**Composite join with composite index (exact match — both columns in seek):**

```
  [Index Scan (IX_Orders_OrderDate)]  -- outer: filtered, 50K rows
  → [Nested Loops (Inner Join)]
      [Index Seek (PK_OrderItems)]    -- seek on (OrderId, ProductId)
          Seek Predicate: OrderId = Orders.OrderId
                       AND ProductId = OrderItems.ProductId
  → [Nested Loops (Inner Join)]
      [Clustered Index Seek (PK_Products)]  -- seek on ProductId
  → [SELECT]
Estimated Cost: ~2.8  |  Logical Reads: ~3,200
```

**Composite join WITHOUT composite index (only single-column index on leading column):**

```
  [Index Scan (IX_Orders_OrderDate)]  -- outer: 50K rows
  → [Nested Loops (Inner Join)]
      [Index Seek (IX_OrderItems_OrderId)]  -- seek on OrderId only
          Seek Predicate: OrderId = Orders.OrderId
          Residual Predicate: ProductId = OrderItems.ProductId  ← applied per row
  → [SELECT]
Estimated Cost: ~4.5  |  Logical Reads: ~5,800 (more residual predicate evaluations)
```

**Composite Hash Match (no useful index):**

```
  [Clustered Index Scan (DimProduct)]  -- build input: 50K rows
  [Clustered Index Scan (FactSales)]   -- probe input: 10M rows
  → [Hash Match (Inner Join)]
      Hash Keys: ProductCategoryId, ProductBrandId
  → [SELECT]
Estimated Cost: ~32  |  Memory Grant: ~150 MB  |  Logical Reads: ~210,000
```

### Failure Modes

**Composite index column order mismatch:** Creating a composite index `(BrandId, CategoryId)` but joining on `CategoryId, BrandId`. The index seek uses only `BrandId` (the leading column), and `CategoryId` is applied as a residual predicate. For a 50K row dimension, this means scanning all rows with matching BrandId instead of seeking directly to the exact (CategoryId, BrandId) pair. Fix: align index column order with the most selective column first in the join predicate.

**OR in composite join predicate:** `ON a.col1 = b.col1 OR a.col2 = b.col2` cannot use a composite index seek. The optimiser must use Nested Loops with full scan on the inner side (O(N²)) or Hash Match with a concatenated hash that includes both OR branches. Always rewrite OR conditions as UNION ALL of two separate queries.

**NULL in composite join columns:** If any column in the composite key is NULL on either side, the row is excluded because `NULL = anything` evaluates to UNKNOWN. For `ON a.col1 = b.col1 AND a.col2 = b.col2`, if `a.col1 IS NULL`, the entire ON predicate is UNKNOWN. This is a silent row disappearance.

**Data type mismatch in composite join:** Mixing INT and VARCHAR, or VARCHAR and NVARCHAR, in different columns of the composite predicate causes implicit CONVERT on the indexed column side, defeating index seeks. Each column pair must have matching data types.

    o.OrderId,
    o.OrderDate,
    oi.ProductId,
    oi.Quantity,
    p.ProductName,
    pr.UnitPrice AS HistoricalUnitPrice,

    ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p
    ON oi.ProductId = p.ProductId

    SELECT TOP 1 pr.UnitPrice
    FROM dbo.ProductPricing AS pr
    WHERE pr.ProductId = oi.ProductId
      AND pr.EffectiveDate <= CAST(o.OrderDate AS DATE)
    ORDER BY pr.EffectiveDate DESC
) AS pr
WHERE o.OrderDate >= '2024-01-01'

    oi.Quantity,
    c.CustomerId,

    ch.ValidTo

    ON o.CustomerId = c.CustomerId

    ON c.CustomerId = ch.CustomerId

    ON o.OrderId = oi.OrderId

    b.EffectiveDate AS NextPeriodStart,
    b.UnitPrice AS PriceB

    AND a.EffectiveDate < b.EffectiveDate
    AND NOT EXISTS (
        SELECT 1
        FROM dbo.ProductPricing AS mid
        WHERE mid.ProductId = a.ProductId
          AND mid.EffectiveDate > a.EffectiveDate
          AND mid.EffectiveDate < b.EffectiveDate
    )

SELECT

    b.ProductId AS ProductB,
    pA.ProductName AS ProductAName,
    pB.ProductName AS ProductBName
FROM dbo.Products AS a
INNER JOIN dbo.Products AS b
    ON a.ProductId < b.ProductId
INNER JOIN dbo.Products AS pA ON a.ProductId = pA.ProductId
INNER JOIN dbo.Products AS pB ON b.ProductId = pB.ProductId
WHERE NOT EXISTS (
    SELECT 1
    FROM dbo.OrderItems AS oiA
    INNER JOIN dbo.OrderItems AS oiB
        ON oiA.OrderId = oiB.OrderId
        AND oiA.ProductId = a.ProductId
        AND oiB.ProductId = b.ProductId
)
ORDER BY a.ProductId, b.ProductId;

## Gotchas and Production Pitfalls

### Gotcha 1 — Composite Index Column Order Mismatch

**Pitfall:** Creating a composite index `(BrandId, CategoryId)` but joining on `CategoryId, BrandId`. The index seek uses only `BrandId` (leading column), and `CategoryId` is applied as a residual predicate — scanning more rows than necessary.

```sql
-- ❌ Index columns mismatch join predicate order
CREATE INDEX IX_FactSales_Composite ON dbo.FactSales (BrandId, CategoryId);

SELECT f.SaleDate, d.CategoryName
FROM dbo.FactSales AS f
INNER JOIN dbo.DimProduct AS d
    ON f.CategoryId = d.CategoryId        -- Second column in index
    AND f.BrandId = d.BrandId             -- First column in index
WHERE f.SaleDate >= '2024-01-01';
-- Seek on BrandId only; CategoryId is residual predicate
-- Plan: Index Seek + Residual Predicate (scans more rows)
```

**Symptom:** The execution plan shows "Seek Predicate" on one column and "Predicate" (residual) on the other. Logical reads are 2-5x higher than expected.

**Fix:**

```sql
-- ✅ Align index column order with join predicate order
CREATE INDEX IX_FactSales_Composite ON dbo.FactSales (CategoryId, BrandId)
    INCLUDE (SaleDate, Quantity, SaleAmount);
```

**Cost of not fixing:** 3x more logical reads than necessary. On a 10M row fact table, this adds 50ms-200ms per query. Aggregated over 100K queries/day, this is 5-20 seconds of extra I/O per day.

### Gotcha 2 — OR in Composite Join Predicate Disables Seeks

**Pitfall:** Using `OR` between join columns in the ON clause:

```sql
-- ❌ Wrong: OR forces scan, cannot seek
SELECT o.OrderId, oi.ProductId, oi.Quantity
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi
    ON o.OrderId = oi.OrderId
    OR o.ShippingAddressId = oi.AddressId
WHERE o.OrderDate >= '2024-01-01';
```

**Symptom:** The execution plan shows a full scan of `OrderItems` regardless of indexes. The Hash Match or Nested Loops scan reads all rows from the inner table.

**Fix:**

```sql
-- ✅ Rewrite as UNION ALL (each branch seeks independently)
SELECT o.OrderId, oi.ProductId, oi.Quantity
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi
    ON o.OrderId = oi.OrderId
WHERE o.OrderDate >= '2024-01-01'
UNION ALL
SELECT o.OrderId, oi.ProductId, oi.Quantity
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi
    ON o.ShippingAddressId = oi.AddressId
WHERE o.OrderDate >= '2024-01-01'
  AND NOT EXISTS (
      SELECT 1 FROM dbo.OrderItems oi2
      WHERE oi2.OrderId = o.OrderId
  );
```

**Cost of not fixing:** Query runs in 25 seconds instead of 200ms. The full scan of OrderItems (5M rows) dominates CPU and I/O.

### Gotcha 3 — NULL in Composite Key Silently Drops Rows

**Pitfall:** Joining on a composite key where any column can be NULL:

```sql
-- ❌ If CustomerHistory.ValidTo IS NULL (meaning "current"),
-- the comparison ValidTo = NULL is UNKNOWN
SELECT o.OrderId, ch.Segment
FROM dbo.Orders AS o
INNER JOIN dbo.CustomerHistory AS ch
    ON o.CustomerId = ch.CustomerId
    AND o.OrderDate >= ch.ValidFrom
    AND o.OrderDate < ch.ValidTo    -- NULL ValidTo → UNKNOWN → row excluded
```

**Symptom:** Current records (those with NULL ValidTo) are missing from historical lookups. The application shows "no history found" for current customers.

**Fix:**

```sql
-- ✅ Handle NULL explicitly
SELECT o.OrderId, ch.Segment
FROM dbo.Orders AS o
INNER JOIN dbo.CustomerHistory AS ch
    ON o.CustomerId = ch.CustomerId
    AND o.OrderDate >= ch.ValidFrom
    AND (ch.ValidTo IS NULL OR o.OrderDate < ch.ValidTo)
```

**Cost of not fixing:** All current customer records show no history. The application displays "N/A" for the customer segment. Business users lose confidence in the data.

### Gotcha 4 — EF Core Does Not Support Composite Foreign Keys in `HasMany().WithMany()`

**Pitfall:** EF Core's `HasMany().WithMany()` for many-to-many relationships does not support composite foreign keys. If your join table has a composite PK that includes additional columns (e.g., `OrderId, ProductId, StoreId`), EF Core cannot map it via `HasMany().WithMany()`.

```csharp
// ❌ EF Core cannot map this — composite FK with 3 columns
modelBuilder.Entity<Order>()
    .HasMany(o => o.Products)
    .WithMany()
    .UsingEntity<OrderItem>(
        j => j.HasOne(oi => oi.Product)
              .WithMany()
              .HasForeignKey(oi => new { oi.ProductId, oi.StoreId }),  // not supported
        j => j.HasOne(oi => oi.Order)
              .WithMany()
              .HasForeignKey(oi => new { oi.OrderId, oi.StoreId }));   // not supported
```

**Symptom:** `InvalidOperationException: The relationship has a composite foreign key` or EF Core generates incorrect SQL with missing join columns.

**Fix:**

```csharp
// ✅ Map the join table as a separate entity with composite PK
public class OrderItem
{
    public int OrderId { get; set; }
    public int ProductId { get; set; }
    public int StoreId { get; set; }
    // ... other columns

    public Order Order { get; set; } = null!;
    public Product Product { get; set; } = null!;
    public Store Store { get; set; } = null!;
}

// Use SelectMany or explicit Join to navigate
var results = await dbContext.Set<OrderItem>()
    .Include(oi => oi.Product)
    .Include(oi => oi.Store)
    .Where(oi => oi.OrderId == orderId)
    .ToListAsync(cancellationToken);
```

**Cost of not fixing:** Development blocked until the mapping workaround is implemented. The engineer wastes a day trying to force EF Core to accept the composite pattern before resorting to raw SQL.

### Gotcha 5 — Data Type Mismatch in One Column of Composite Key

**Pitfall:** One column in the composite join has mismatched data types between the two tables:

```sql
-- ❌ FactSales.ProductCategoryId is INT, DimProduct.CategoryId is VARCHAR
SELECT f.SaleDate, d.CategoryName
FROM dbo.FactSales AS f
INNER JOIN dbo.DimProduct AS d
    ON f.ProductCategoryId = d.CategoryId   -- INT = VARCHAR → CONVERT_IMPLICIT
    AND f.ProductBrandId = d.BrandId;
```

**Symptom:** The execution plan shows `CONVERT_IMPLICIT` on the indexed column side. The composite index seek is downgraded to a scan on the CategoryId column.

**Fix:**

```sql
-- ✅ Fix the data type at the schema level
ALTER TABLE dbo.DimProduct
    ALTER COLUMN CategoryId INT NOT NULL;
-- Or create the table with matching types

-- If schema change is impossible, cast the non-indexed side:
SELECT f.SaleDate, d.CategoryName
FROM dbo.FactSales AS f
INNER JOIN dbo.DimProduct AS d
    ON f.ProductCategoryId = TRY_CAST(d.CategoryId AS INT)  -- cast on non-indexed side
    AND f.ProductBrandId = d.BrandId;
```

**Cost of not fixing:** The composite index on `(CategoryId, BrandId)` cannot be used for a seek on CategoryId. The query falls back to a full scan of the fact table, increasing logical reads from 500 to 12,000.

## Interview Arsenal

### Question Bank

1. **What is a composite JOIN and when is it required?** — Definition: an equi-join using multiple columns in the ON predicate. Required when the relationship spans multiple columns (composite FK, natural key).

2. **How does the optimiser handle a composite JOIN — what determines whether it can seek or scans?** — A composite index with the join columns in the same order enables Index Seek on all columns. Wrong order = partial seek + residual predicate.

3. **What is the performance difference between a composite index with correct vs incorrect column order?** — Correct order: seeks on both columns, ~N logical reads. Wrong order: seeks on leading column only + residual filter, ~3-5x more logical reads.

4. **What happens when you use OR between columns in a composite JOIN ON clause?** — OR disables index seeks. The optimiser must scan the inner table. Fix: rewrite as UNION ALL.

5. **Composite JOIN vs single surrogate key — what are the tradeoffs?** — Composite key: no surrogate column needed, natural meaning, but more complex joins and worse performance. Surrogate key: single INT, faster joins, simpler code, but loses natural meaning.

6. **What does the execution plan look like for a composite JOIN with a correctly-ordered covering index?** — Index Seek on composite index (both seek predicates) → Nested Loops → Clustered Index Seek on dimension table → SELECT.

7. **How does a composite JOIN behave at 100M rows?** — Composite index with correct order keeps seeks efficient (O(log N) per seek). Wrong order causes partial scans that multiply I/O by 3-5x. Merge Join with composite sorted input is ideal for data warehouse ETL.

8. **How do EF Core and Dapper handle composite JOINs?** — EF Core supports composite FKs via `HasForeignKey(new { col1, col2 })`. Dapper supports composite join via multi-mapping with anonymous types or raw SQL with splitOn.

### Spoken Answers

**Q: What is a composite JOIN and when is it required?**

> **Average answer:** A join on multiple columns. Used when tables have composite foreign keys.

> **Great answer:** A composite JOIN uses two or more columns in the ON predicate with AND between the equalities — `ON a.col1 = b.col1 AND a.col2 = b.col2`. The database engine treats the column pair as a compound key. It is required in three main scenarios: first, when tables are related through a natural key that spans multiple columns — for example, a `DimProduct` dimension identified by `(CategoryId, BrandId)` that is referenced by a `FactSales` fact table. Second, when a junction table has a composite primary key, like `OrderItems(OrderId, ProductId)`. Third, in versioned data or SCD Type 2 lookups where the join includes both the entity ID and a date range. The critical performance detail is that a composite JOIN requires a composite index with the columns in the same order as the ON predicate. If the index is `(BrandId, CategoryId)` but the join is `ON CategoryId AND BrandId`, SQL Server can only seek on `BrandId` — `CategoryId` is applied as a residual predicate, scanning all matching BrandId rows. I have seen this cause a 5x increase in logical reads on a 50M row fact table. The fix is always to align the index order: the leftmost column in the index should be the most selective column in the join predicate.

**Q: Composite JOIN vs single surrogate key — what are the tradeoffs?**

> **Average answer:** Surrogate keys are simpler and faster. Composite keys are more natural.

> **Great answer:** I prefer surrogate keys for OLTP systems and composite keys for data warehouse dimensions where the natural key has business meaning. A surrogate key — a single INT IDENTITY column — reduces the join to a single equality, which is the simplest and fastest join possible. A single INT column join uses 4 bytes per comparison, supports a simple index seek, and simplifies EF Core mapping. The tradeoff is that you need to join the source table to the dimension table on the natural keys to look up the surrogate key during ETL, which is an extra step. A composite key eliminates the lookup step — the fact table directly stores the natural key columns — but makes every join more complex because it involves multiple columns. The composite join is slightly slower than a surrogate join: comparing two INTs in a composite index is about 2x the CPU cost of comparing one INT, and the composite index is larger (8 bytes vs 4 bytes per key entry). The real performance difference shows up in Nested Loops joins: a single INT seek is marginally faster than a composite INT seek because the composite B-tree is deeper (more bytes per key = fewer keys per page = more pages to traverse). At 100M rows, the difference is about 10-15% in query time. I choose based on the workload: surrogate keys for high-throughput OLTP, composite keys for dimensional models where the natural key is frequently queried directly.

### Interview Trigger

The question "What is the difference between `ON a.col1 = b.col1 AND a.col2 = b.col2` and `ON a.col1 = b.col1 OR a.col2 = b.col2`?" tests whether the candidate understands that AND enables index seeks while OR forces scans. The deep follow-up on composite joins is: "If I have a composite index (A, B) and the query joins on B and A (in that order), what does the execution plan look like?" — testing whether the candidate knows that the column order in the join predicate must match the index column order for a full seek.

### Comparison Table

| | Composite Join (Multi-column) | Simple Join (Single-column) | OR Join (Disjunction) |
|---|---|---|---|
| What it does | Equality on 2+ columns | Equality on 1 column | Equality on one of multiple columns |
| Index seek support | Full seek if index order matches | Full seek on single-column index | No seek — full scan required |
| Performance profile | O(N × log M) with correct index | O(N × log M) | O(N × M) |
| Row matching semantics | All columns must match | Single column must match | Any column can match |
| EF Core support | Composite FK mapping | Simple FK mapping | Raw SQL required |
| Dapper support | Multi-mapping with splitOn | Simple multi-mapping | Raw SQL with UNION ALL |
| When to choose | Natural keys, SCD, composite PK | Surrogate keys, simple FK | Rare — prefer UNION ALL |

## Decision Framework

### When to Apply

```mermaid
flowchart TD
    A[JOIN requires multiple columns in ON clause] --> B{All predicates are<br>equality = ?}
    B -->|Yes - all equalities| C{Composite index exists?}
    B -->|No - has range / OR| D[Not a composite equi-join]
    D --> E{OR in ON clause?}
    E -->|Yes| F[Rewrite as UNION ALL]
    E -->|Range predicate| G[Non-equi join - see 8.104]
    C -->|Yes - correct column order| H[Use Index Seek - efficient]
    C -->|Yes - wrong order| I[Recreate index with correct order]
    C -->|No index| J[Create composite index]
    H --> K[Monitor logical reads - expect O(log N)]
    I --> L[Current: residual predicate scans too many rows]
    J --> M[Without index: Hash Match or scan]
    K --> N[Scale: 10M rows, <100ms response]
```

### Application Checklist

- [ ] The ON predicate uses AND between all join columns (not OR)
- [ ] A composite index exists with the join columns in the ON predicate order
- [ ] The leading column of the composite index is the most selective column in the join
- [ ] Data types match between corresponding columns on both sides of the join
- [ ] NULL handling is explicit: COALESCE or ISNULL for nullable composite key columns
- [ ] EF Core composite FK mapping uses `HasForeignKey(new { col1, col2 })` with matching principal key
- [ ] Dapper multi-mapping `splitOn` splits on the correct column boundary
- [ ] For Merge Join, both inputs are sorted on all composite key columns in the same order

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Natural key relationship without surrogate|More complex ON predicate|
|Faster than single-column join for composite FK? No — slightly slower|Larger composite index (more bytes per key)|
|Schema self-documents the relationship|EF Core mapping is more complex|
|No extra lookup during ETL|Join column order must match index order exactly|

### Scale Thresholds

- **Relevant when table exceeds ~50K rows** — Below 50K, even a full scan of both sides is acceptable for composite joins
- **Composite index becomes critical at ~500K rows** — Without the correct index, Hash Match or partial seek adds 3-5x logical reads
- **Composite index column order must match join predicate order at every scale** — Mismatch costs 2-5x even at 100K rows
- **Merge Join with composite keys is ideal for data warehouse ETL at 100M+ rows** — Both inputs sorted on the composite key enable a single-pass join with no memory grant

## Self-Check

### Conceptual Questions

1. What is the syntax difference between a composite JOIN ON clause and a simple JOIN ON clause?
2. How does the query optimiser decide whether it can do a composite index seek on both columns?
3. Which DMV or command shows whether a composite join is using a seek or a residual predicate?
4. What is the most common composite join mistake and how many extra logical reads does it cause?
5. How does EF Core map a composite foreign key and what is the one limitation to be aware of?
6. Write a Dapper multi-mapping query that joins two tables on a composite key (two columns).
7. What is the structural difference between a composite equi-join and an OR-based multi-column join?
8. At approximately what table size does a composite index column order mismatch become a problem?
9. What composite index would you create to support the join: `ON f.A = d.A AND f.B = d.B`?
10. Explain in 60 seconds how you would debug a composite join that uses a residual predicate instead of a seek.

<details>
<summary>Answers</summary>

1. Composite JOIN: `ON a.col1 = b.col1 AND a.col2 = b.col2`. Simple JOIN: `ON a.col1 = b.col1`. The composite version chains multiple equality predicates with AND.

2. The optimiser can do a composite index seek on both columns when the composite index has the columns in the same order as the join predicate. If the index is `(col1, col2)` and the join is `ON a.col1 = b.col1 AND a.col2 = b.col2`, SQL Server performs a seek on `(col1, col2)`. If the order is reversed (index `(col2, col1)`), it seeks on `col2` only and applies `col1` as a residual predicate.

3. The execution plan shows "Seek Predicates" for columns used in the seek and "Predicate" for columns applied as residual. In SSMS, hover over the Index Seek operator and read the Seek Predicates vs Predicate sections. `SET STATISTICS PROFILE ON` also reveals this in the plan output.

4. Creating a composite index with the wrong column order (not matching the join predicate). At 5M rows, this adds 2-5x extra logical reads — from ~1,000 (full composite seek) to ~4,000-5,000 (partial seek + residual). The fix is to recreate the index with columns in the join predicate order.

5. EF Core maps composite FKs using `HasForeignKey(e => new { e.Col1, e.Col2 })` with a matching `HasPrincipalKey(p => new { p.Col1, p.Col2 })`. The limitation: `HasMany().WithMany()` does not support composite foreign keys — the join table's composite FK referencing the two principal tables cannot be mapped. The workaround is to map the join table as a separate entity.

6. Dapper multi-mapping for composite join:
```csharp
var results = await connection.QueryAsync<FactSale, DimProduct, FactSale>(
    sql,
    (sale, product) =>
    {
        sale.CategoryName = product.CategoryName;
        return sale;
    },
    splitOn: "CategoryId");
```

7. Composite equi-join: `ON a.x = b.x AND a.y = b.y` — all conditions must be true; supports index seeks on composite index. OR-based: `ON a.x = b.x OR a.y = b.y` — any condition can be true; forces full scan because the optimiser cannot seek on two independent columns linked by OR.

8. Column order mismatch becomes noticeable at ~100K rows and becomes a significant problem at ~1M rows. At 10M rows, the difference is dramatic: ~1,000 logical reads with correct order vs ~5,000 with wrong order.

9. `CREATE INDEX IX_FactSales_A_B ON dbo.FactSales (A, B) INCLUDE (C, D, E);` — The index columns must be `(A, B)` in the same order as the join predicate `ON f.A = d.A AND f.B = d.B`. The leading column `A` should be the most selective one (highest cardinality / most distinct values).

10. "First, I examine the execution plan. I look at the Index Seek operator and check whether both columns appear in Seek Predicates or one appears in Predicate (residual). If I see a residual predicate, the index column order does not match the join predicate order. I check the index definition with `sp_helpindex` or `sys.index_columns`. Then I compare: if the index is `(BrandId, CategoryId)` but the join is `ON CategoryId AND BrandId`, the fix is to recreate the index as `(CategoryId, BrandId)` — matching the join order. If the join predicate order cannot change, I make the index match it. After the change, I verify the plan shows both columns in Seek Predicates and logical reads drop by the expected 60-80%."

</details>

### Query Challenges

**Challenge 1 — Write the SQL**

You have `FactSales` (SaleDate, ProductCategoryId, ProductBrandId, StoreId, Quantity, SaleAmount) and `DimProduct` (CategoryId, BrandId, CategoryName, BrandName, Manufacturer). Write a query that returns total sales amount by CategoryName and BrandName for January 2024. Join on the composite key (ProductCategoryId = CategoryId AND ProductBrandId = BrandId). Ensure the query can use a composite index seek.

<details>
<summary>Solution</summary>

```sql
-- Create the composite index if it does not exist
CREATE INDEX IX_FactSales_Composite ON dbo.FactSales (ProductCategoryId, ProductBrandId)
    INCLUDE (SaleDate, SaleAmount);

SELECT
    d.CategoryName,
    d.BrandName,
    SUM(f.SaleAmount) AS TotalSales,
    COUNT(*) AS SaleCount
FROM dbo.FactSales AS f
INNER JOIN dbo.DimProduct AS d
    ON f.ProductCategoryId = d.CategoryId
    AND f.ProductBrandId = d.BrandId
WHERE f.SaleDate >= '2024-01-01'
  AND f.SaleDate < '2024-02-01'
GROUP BY d.CategoryName, d.BrandName
ORDER BY TotalSales DESC;
```

**Logical reads:** ~2,500 (composite index seek on IX_FactSales_Composite + PK seeks on DimProduct)
**Execution plan:** [Index Seek on IX_FactSales_Composite (both seek predicates)] → [Nested Loops] → [Clustered Index Seek on PK_DimProduct] → [Hash Match (Aggregate)] → [SELECT]
**EF Core equivalent:**

```csharp
var results = await dbContext.Set<FactSale>()
    .Where(f => f.SaleDate >= new DateTime(2024, 1, 1)
                && f.SaleDate < new DateTime(2024, 2, 1))
    .GroupBy(f => new { f.Product.CategoryName, f.Product.BrandName })
    .Select(g => new SaleSummaryDto
    {
        CategoryName = g.Key.CategoryName,
        BrandName = g.Key.BrandName,
        TotalSales = g.Sum(f => f.SaleAmount),
        SaleCount = g.Count()
    })
    .OrderByDescending(s => s.TotalSales)
    .ToListAsync(cancellationToken);

// EF Core generates the composite join automatically from the
// navigation property configuration with composite FK.
```

</details>

**Challenge 2 — Fix the performance problem**

```sql
-- This query aggregates sales by product category. It runs in 45 seconds
-- on a 20M row FactSales table.
-- SET STATISTICS IO: logical reads = 420,000

SELECT
    d.CategoryName,
    SUM(f.SaleAmount) AS TotalSales
FROM dbo.FactSales AS f
INNER JOIN dbo.DimProduct AS d
    ON f.ProductBrandId = d.BrandId
    AND f.ProductCategoryId = d.CategoryId
WHERE f.SaleDate >= '2024-01-01'
  AND f.SaleDate < '2024-02-01'
GROUP BY d.CategoryName
ORDER BY TotalSales DESC;
```

<details> <summary>Solution</summary>

**Root cause:** The composite index on FactSales has the wrong column order. The index is `IX_FactSales_Composite (BrandId, CategoryId)` but the join predicate is `(BrandId, CategoryId)` — wait, the join predicate is `ON f.ProductBrandId = d.BrandId AND f.ProductCategoryId = d.CategoryId`. If the index is `(BrandId, CategoryId)`, that actually matches the join order here. Let me check: the index might not exist, or it might be `(CategoryId, BrandId)` while the join is `(BrandId, CategoryId)`.

Actually, looking more carefully: the join predicate has `BrandId` first, then `CategoryId`. If the index is `(CategoryId, BrandId)` (the natural order), the seek would be on `CategoryId` only with a residual on `BrandId`.

**Root cause:** The index IX_FactSales_Composite has column order `(ProductCategoryId, ProductBrandId)` but the join predicate references `ProductBrandId` first. The column order mismatch causes a partial seek.

```sql
-- ✅ Fix: recreate index matching the join predicate order
CREATE INDEX IX_FactSales_Composite ON dbo.FactSales (ProductBrandId, ProductCategoryId)
    INCLUDE (SaleDate, SaleAmount)
    WITH (DROP_EXISTING = ON);
```

**After fix — logical reads:** ~1,800 (from 420,000)

**Alternatively, if the join predicate order can change, rewrite the ON clause:**
```sql
-- Match the index order (CategoryId first)
SELECT d.CategoryName, SUM(f.SaleAmount) AS TotalSales
FROM dbo.FactSales AS f
INNER JOIN dbo.DimProduct AS d
    ON f.ProductCategoryId = d.CategoryId
    AND f.ProductBrandId = d.BrandId  -- now matches index order
WHERE f.SaleDate >= '2024-01-01' AND f.SaleDate < '2024-02-01'
GROUP BY d.CategoryName;
```

</details>

**Challenge 3 — Explain the execution plan**

```sql
-- Query: FactSales joined to DimProduct on composite key
SELECT f.SaleDate, f.Quantity, d.CategoryName, d.BrandName
FROM FactSales f
INNER JOIN DimProduct d
    ON f.ProductCategoryId = d.CategoryId
    AND f.ProductBrandId = d.BrandId
WHERE f.SaleDate >= '2024-01-01';

-- Plan A (actual):
-- Index Seek (IX_FactSales_Date) → Nested Loops
--   → Index Seek (IX_FactSales_Composite)
--   → Nested Loops
--     → Clustered Index Seek (PK_DimProduct)
-- Estimated rows: 50,000 | Actual rows: 48,200 | Cost: 3.2

-- Plan B (what happens without composite index):
-- Clustered Index Scan (FactSales) → Hash Match
--   → Clustered Index Scan (DimProduct)
-- Estimated rows: 500,000 | Actual rows: 1,200,000 | Cost: 48.5
```

Why does Plan A include two Index Seeks on FactSales? What is the second seek doing?

<details> <summary>Solution</summary>

**Why two seeks on FactSales:** The first seek is on `IX_FactSales_Date` to filter by `SaleDate >= '2024-01-01'`. This returns RID values (or clustered key values depending on the index). The second seek is on `IX_FactSales_Composite` to get the `ProductCategoryId` and `ProductBrandId` values to join with DimProduct. This is a **key lookup** pattern: the Date index does not include the join columns, so SQL Server must look up those columns from another index on the same table. If `IX_FactSales_Composite` were a covering index including `SaleDate` as a leading column, the second seek would be eliminated.

**To optimise further:** Create a single composite index that covers both the WHERE filter and the JOIN:
```sql
CREATE INDEX IX_FactSales_Date_Composite ON dbo.FactSales
    (SaleDate, ProductCategoryId, ProductBrandId)
    INCLUDE (Quantity, SaleAmount);
```
This eliminates the second seek on FactSales entirely — the Date index seek directly provides the join columns.

**Tradeoff:** The covering index is wider (more bytes per row in the B-tree), increasing storage and write overhead. At 20M rows, each additional 8 bytes per row adds ~160MB to the index size.

</details>

**Challenge 4 — Diagnose the concurrency problem**

A data warehouse ETL job runs every night and performs a composite join of the nightly batch of 500K new FactSales rows to the DimProduct table (50K rows) on `(CategoryId, BrandId)`. During the ETL, the DimProduct table is locked for updates (new products added). The ETL job uses `SERIALIZABLE` isolation level for consistency. A concurrent reporting query that also joins FactSales to DimProduct on the composite key blocks on `LCK_M_RANGE_S` (range-shared lock) on the DimProduct table, causing the dashboard to time out.

<details> <summary>Solution</summary>

**Root cause:** The ETL's `SERIALIZABLE` isolation level acquires range locks (`KEY` range locks) on the DimProduct composite PK `(CategoryId, BrandId)` when reading rows to join with the batch. If the ETL queries a range of composite keys that overlaps with the reporting query's composite key range, the reporting query's shared range lock request is blocked by the ETL's range lock.

**Detection query:**

```sql
SELECT
    request_session_id AS blocked_session,
    blocking_session_id AS blocking_session,
    resource_type,
    resource_description,
    request_mode,
    request_status
FROM sys.dm_tran_locks
WHERE resource_database_id = DB_ID()
  AND resource_associated_entity_id = OBJECT_ID('DimProduct')
  AND request_mode LIKE '%RANGE%';
```

**Fix:**

```sql
-- Option 1: Use READ COMMITTED SNAPSHOT ISOLATION
ALTER DATABASE YourWarehouse SET READ_COMMITTED_SNAPSHOT ON;

-- Option 2: Lower the ETL isolation level to READ COMMITTED
-- if phantom reads are acceptable for this batch
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
INSERT INTO FactSales (...)
SELECT ...
FROM StagingTable s
INNER JOIN DimProduct d
    ON s.CategoryId = d.CategoryId
    AND s.BrandId = d.BrandId;

-- Option 3: Use page-level locking hint to reduce lock granularity
SELECT ...
FROM StagingTable s
INNER JOIN DimProduct d WITH (PAGLOCK)
    ON s.CategoryId = d.CategoryId
    AND s.BrandId = d.BrandId;
```

**In .NET:** Set the isolation level on the DbContext for the ETL batch:

```csharp
await dbContext.Database.ExecuteSqlRawAsync(
    "SET TRANSACTION ISOLATION LEVEL READ COMMITTED;");
```

</details>

**Challenge 5 — Design the index**

**Scenario:** A reporting query runs hourly and joins:
- `FactInventory` (100M rows) with columns: `WarehouseId`, `ProductCategoryId`, `ProductBrandId`, `InventoryDate`, `QuantityOnHand`, `ReorderPoint`
- `DimProduct` (50K rows) with composite PK `(CategoryId, BrandId)`

The query filters on `InventoryDate = '2024-06-25'` (returns ~500K rows) and joins on `ProductCategoryId = CategoryId AND ProductBrandId = BrandId`. It currently takes 18 seconds. 95% of the join columns used in queries have `ProductCategoryId` as the first column.

Design the optimal index strategy.

<details> <summary>Solution</summary>

```sql
-- Index 1: Covering index for the most common query pattern
-- Leading column is the filter column (InventoryDate) for seek
-- Followed by the join columns in query order (CategoryId first)
CREATE INDEX IX_FactInventory_Date ON dbo.FactInventory
    (InventoryDate, ProductCategoryId, ProductBrandId)
    INCLUDE (WarehouseId, QuantityOnHand, ReorderPoint);

-- Index 2: Supporting index for queries without date filter
-- ProductCategoryId first (most selective join column)
CREATE INDEX IX_FactInventory_Composite ON dbo.FactInventory
    (ProductCategoryId, ProductBrandId)
    INCLUDE (InventoryDate, QuantityOnHand, ReorderPoint);
```

**Explain:** `IX_FactInventory_Date` supports the most common query pattern: filter by date, then join. The `InventoryDate` seek returns exactly the rows needed, and the join columns are immediately available — no key lookup or second seek required. The index covers all SELECT columns. `IX_FactInventory_Composite` supports queries that join without a date filter or filter on different dates. Write overhead: each INSERT adds entries to both indexes — approximately 2 additional leaf page writes per 800 rows inserted.

**Tradeoffs:** `IX_FactInventory_Date` is a wide index (~40 bytes per row × 100M = ~4GB). Write overhead is significant for high-throughput OLTP inserts but acceptable for nightly batch loads. The index eliminates the 18-second query time, reducing it to ~200ms.

**What NOT to index:** Do not create separate single-column indexes on `ProductCategoryId` and `ProductBrandId` — the query needs both columns in the same index for the composite join. Separate indexes would force SQL Server to choose one or use index intersection (less efficient).

</details>
