### Where This Fits

COUNT(DISTINCT col) is one of the most expensive aggregate operations in SQL Server because the engine must eliminate duplicates before counting — requiring either a Sort + Stream Aggregate or a Hash Aggregate with full hash table build. A .NET backend engineer encounters this in dashboard queries counting unique customers, distinct product views, unique sessions, or distinct error types. The performance trap is that COUNT(DISTINCT) on a high-cardinality column over a large table can consume GB of memory grant, spill to tempdb, and run for minutes. The interview signal is understanding when COUNT(DISTINCT) is appropriate, when to replace it with APPROX_COUNT_DISTINCT, and when to restructure the query entirely.

## Core Mental Model

COUNT(DISTINCT col) is a grouped aggregate that counts the number of unique non-NULL values in column col within each group. The engine first filters out NULLs (DISTINCT ignores NULLs), then removes duplicates via sorting or hashing, then counts the remaining rows. The invariant: for every distinct value of col within the group, COUNT(DISTINCT col) increments by exactly 1, regardless of how many rows contain that value. The cost is driven entirely by the cardinality and distribution of the distinct values — not by the total row count alone. A column with 10M rows but only 100 distinct values is cheap (the hash table fits in memory). A column with 10M rows and 9M distinct values is expensive (the hash table grows to hundreds of MB). The recognition pattern: if you need to count unique values within a group and the cardinality is expected to be a significant fraction of row count, you are in the expensive zone.

### Classification

- **SQL clause/operator family:** Aggregate function with DISTINCT modifier — belongs to the GROUP BY and aggregate family
- **Optimizer behavior:** Forces either Sort + Stream Aggregate (ordered) or Hash Aggregate (unordered) to deduplicate; uses significant memory for hash table
- **SARGable:** No — COUNT(DISTINCT col) is computed post-scan, not a predicate

## Deep Mechanics

### How the Engine Executes This

1. **Scan/Seek phase:** The storage engine reads rows from the table or index, applying any WHERE clause filters. The rows flow through the access operator (Table Scan, Clustered Index Scan, or Non-Clustered Index Seek).

2. **NULL filtering (implicit):** The engine inherently excludes NULL values from the DISTINCT computation because NULL is not a value — DISTINCT excludes NULLs from the result set. If the column has significant NULL density, the effective row count for the distinct operation is reduced.

3. **Deduplication phase:** The optimizer chooses one of two strategies:
   - **Sort + Stream Aggregate:** If the input is already sorted on the DISTINCT key (or the optimizer chooses to sort), the Stream Aggregate reads rows in order and emits only the first occurrence of each distinct value. The Sort operator needs a memory grant proportional to the total row count.
   - **Hash Aggregate:** For larger inputs, the Hash Aggregate builds a hash table in memory, hashing each distinct key. If the hash table exceeds the memory grant (typically 5-10% of total memory per query), it spills to tempdb — one of the most expensive things that can happen in a query.

4. **Counting phase:** After deduplication, the Stream Aggregate increments a counter for each remaining row. For COUNT(DISTINCT col), it counts 1 per distinct value. For SUM(DISTINCT col), it sums the distinct values. For AVG(DISTINCT col), it divides SUM(DISTINCT col) by COUNT(DISTINCT col).

5. **Projection:** The result is output to the SELECT list.

### SQL Visibility

```sql
-- COUNT(DISTINCT) in action: unique customers per product category
SELECT
    p.Category,
    COUNT(DISTINCT o.CustomerId) AS UniqueCustomers,
    COUNT(*) AS TotalOrders,
    COUNT(DISTINCT o.CustomerId) * 1.0 / COUNT(*) AS DistinctivenessRatio
FROM Orders o
INNER JOIN Products p ON o.ProductId = p.ProductId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY p.Category
ORDER BY UniqueCustomers DESC;
```

This is a known EF Core anti-pattern: it generates a correlated subquery for each COUNT(DISTINCT) call, causing N+1-like behavior. Each subquery re-scans the Orders + Products join. With 50 product categories, this runs 50+ separate COUNT(DISTINCT) operations instead of one.

**Better EF Core approach — raw SQL for COUNT(DISTINCT):**

```csharp

    .SqlQuery<CategoryStats>(@"

            p.Category,
            COUNT(DISTINCT o.CustomerId) AS UniqueCustomers,
            COUNT(*) AS TotalOrders,

        FROM Orders o
        INNER JOIN Products p ON o.ProductId = p.ProductId
        WHERE o.OrderDate >= @StartDate
        GROUP BY p.Category
        ORDER BY UniqueCustomers DESC",
        new SqlParameter("@StartDate", startDate))
    .ToListAsync(cancellationToken);
```

### Execution Plan Analysis

For the core COUNT(DISTINCT) query:

1. **Clustered Index Scan** on Orders (filter by OrderDate)
2. **Index Seek** on Products (PK lookup for Category)
3. **Nested Loops** join Orders → Products
4. **Sort by Category, CustomerId** (for Stream Aggregate if ordered input chosen)
5. **Stream Aggregate** (GROUP BY Category)
6. **Stream Aggregate** (COUNT(DISTINCT CustomerId) — dedup within each group)

With Hash Aggregate alternative:
1. Same scans + joins
2. **Hash Aggregate** (build hash table on Category, CustomerId)
3. **Stream Aggregate** (COUNT(DISTINCT) from hash output)

```
Plan A (Sort + Stream Aggregate):
[Clustered Index Scan] → [Nested Loops] → [Index Seek] → [Sort (Category, CustomerId)] → [Stream Aggregate (Group By)] → [SELECT]

Plan B (Hash Aggregate):
[Clustered Index Scan] → [Nested Loops] → [Index Seek] → [Hash Aggregate] → [Stream Aggregate] → [SELECT]

When Hash Aggregate spills to tempdb:
[Clustered Index Scan] → ... → [Hash Aggregate] → [Table Spill (tempdb)] → [Stream Aggregate] → [SELECT]
```

### Failure Modes

**Failure 1 — COUNT(DISTINCT multiple columns incorrectly):**

```sql
-- ❌ Wrong: COUNT(DISTINCT col1, col2) is not valid in SQL Server
SELECT COUNT(DISTINCT CustomerId, ProductId) FROM Orders;
-- Error: "COUNT(DISTINCT ...) is not supported with multiple columns"
```

The fix is to use a subquery or concatenation (though concatenation can cause false matches):

```sql
-- ✅ Correct approaches:
-- Option 1: Subquery
SELECT COUNT(*) FROM (SELECT DISTINCT CustomerId, ProductId FROM Orders) d;

-- Option 2: Windowed COUNT (SQL Server 2022+)
SELECT COUNT(DISTINCT CAST(CustomerId AS NVARCHAR(10)) + ':' + CAST(ProductId AS NVARCHAR(10)))
FROM Orders;
-- ⚠️ Beware: concatenation may create collisions (CustomerId=1, ProductId=23 vs CustomerId=12, ProductId=3)
```

**Failure 2 — EF Core correlated subquery explosion:**

EF Core generates a correlated subquery for each COUNT(DISTINCT) in the SELECT list. With multiple aggregate columns using COUNT(DISTINCT), this multiplies the I/O cost.

```sql
-- EF Core generates for 3 COUNT(DISTINCT) calls:
SELECT
    (SELECT COUNT(DISTINCT [o0].[CustomerId]) FROM ...) AS [UniqueCustomers],
    (SELECT COUNT(DISTINCT [o0].[ProductId]) FROM ...) AS [UniqueProducts],
    (SELECT COUNT(DISTINCT [o0].[OrderId]) FROM ...) AS [UniqueOrders],
    ...
```

**Detection:** Check EF Core logs for repeated subqueries with identical FROM clauses.

**Failure 3 — Memory grant spill for high-cardinality COUNT(DISTINCT):**

On a column with 5M distinct values out of 10M rows, the Sort or Hash Aggregate memory grant can exceed available memory, causing a spill to tempdb. Spills are visible in the execution plan as a "Table Spill" operator and in sys.dm_exec_query_stats with `spill_to_tempdb_count > 0`.

**Detection DMV:**

```sql
SELECT TOP 10
    qs.total_elapsed_time / 1000000.0 AS total_seconds,
    qs.total_logical_reads,
    qs.max_grant_kb / 1024 AS max_grant_mb,
    CAST(qs.max_reserved_grant_kb AS BIGINT) / 1024 AS max_reserved_mb,
    qs.total_spills_to_tempdb,
    SUBSTRING(st.text, (qs.statement_start_offset/2)+1, 200) AS query_text
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
WHERE st.text LIKE '%COUNT(DISTINCT%'
ORDER BY qs.total_spills_to_tempdb DESC;
```

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- Production pattern: count unique customers per product category
-- Table: Orders with ~10M rows, ~100K distinct customers, ~500 categories

-- Scalable approach with APPROX_COUNT_DISTINCT for high cardinality
SELECT
    p.CategoryId,
    p.CategoryName,
    COUNT(DISTINCT o.CustomerId) AS ExactUniqueCustomers,
    APPROX_COUNT_DISTINCT(o.CustomerId) AS ApproxUniqueCustomers,
    COUNT(DISTINCT o.CustomerId) - APPROX_COUNT_DISTINCT(o.CustomerId) AS Difference,
    ROUND(
        ABS(COUNT(DISTINCT o.CustomerId) - APPROX_COUNT_DISTINCT(o.CustomerId))
        * 100.0 / NULLIF(COUNT(DISTINCT o.CustomerId), 0), 2
    ) AS ErrorPct
FROM Orders o
INNER JOIN Products p ON o.ProductId = p.ProductId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY p.CategoryId, p.CategoryName
ORDER BY ExactUniqueCustomers DESC;
```

### APPROX_COUNT_DISTINCT (SQL Server 2019+)

For scenarios where exact precision is not required and performance matters:

```sql
-- APPROX_COUNT_DISTINCT uses HyperLogLog algorithm
-- Returns ~97-98% accuracy with <2% error rate typically
-- Memory grant: ~2.5KB per group (vs. potentially GB for COUNT(DISTINCT))
SELECT
    p.CategoryName,
    APPROX_COUNT_DISTINCT(o.CustomerId) AS ApproxUniqueCustomers
FROM Orders o
INNER JOIN Products p ON o.ProductId = p.ProductId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY p.CategoryName;

-- Typical error: 0.1-0.5% for 1M distinct values
-- Performance: 10-50x faster than COUNT(DISTINCT) for high cardinality
```

### EF Core Implementation

```csharp
public record CategoryStats
{
    public int CategoryId { get; init; }
    public string CategoryName { get; init; } = "";
    public int ExactUniqueCustomers { get; init; }
    public int? ApproxUniqueCustomers { get; init; }
}

public async Task<IReadOnlyList<CategoryStats>> GetUniqueCustomersByCategoryAsync(
    DateTime startDate,
    CancellationToken cancellationToken = default)
{
    // Option 1: Raw SQL (recommended for COUNT(DISTINCT))
    const string sql = @"
        SELECT
            p.CategoryId,
            p.CategoryName,
            COUNT(DISTINCT o.CustomerId) AS ExactUniqueCustomers,
            APPROX_COUNT_DISTINCT(o.CustomerId) AS ApproxUniqueCustomers
        FROM Orders o
        INNER JOIN Products p ON o.ProductId = p.ProductId
        WHERE o.OrderDate >= @StartDate
        GROUP BY p.CategoryId, p.CategoryName
        ORDER BY ExactUniqueCustomers DESC";

    await using var context = _dbContextFactory.CreateDbContext();
    return await context.Database
        .SqlQuery<CategoryStats>(FormattableStringFactory.Create(sql,
            new SqlParameter("@StartDate", startDate)))
        .ToListAsync(cancellationToken);
}

// Option 2: Pure LINQ (only if count of groups is small and cardinality is low)
public async Task<List<CategoryStats>> GetUniqueCustomersByCategoryLinqAsync(
    DateTime startDate,
    CancellationToken cancellationToken = default)
{
    await using var context = _dbContextFactory.CreateDbContext();
    var query = from o in context.Orders
                join p in context.Products on o.ProductId equals p.ProductId
                where o.OrderDate >= startDate
                group o by new { p.CategoryId, p.CategoryName } into g
                select new CategoryStats
                {
                    CategoryId = g.Key.CategoryId,
                    CategoryName = g.Key.CategoryName,
                    ExactUniqueCustomers = g.Select(o => o.CustomerId).Distinct().Count()
                    // ⚠️ EF Core generates correlated subquery per group!
                };

    return await query.ToListAsync(cancellationToken);
}
```

### SQL Server vs PostgreSQL Differences

PostgreSQL supports COUNT(DISTINCT) identically but also provides:

```sql
-- PostgreSQL alternative: array_length + uniq on array_agg
SELECT
    p.category_name,
    COUNT(DISTINCT o.customer_id) AS exact_unique,
    -- PostgreSQL approximate distinct via extension
    -- (no built-in approx_count_distinct)
    -- Third-party: hyperloglog extension
    card(hyperloglog_add(hyperloglog_init(), o.customer_id::text)) AS approx_unique
FROM orders o
INNER JOIN products p ON o.product_id = p.product_id
WHERE o.order_date >= '2024-01-01'
GROUP BY p.category_id, p.category_name;

-- PostgreSQL is typically slower for COUNT(DISTINCT) due to
-- different hash aggregate implementation (less memory for hash table)
```

PostgreSQL does not have APPROX_COUNT_DISTINCT built-in. Use the `hyperloglog` or `tdigest` extensions for approximate distinct counts. The HLL (HyperLogLog) approach is common in PostgreSQL analytical workloads.

## Gotchas and Production Pitfalls

### 1. COUNT(DISTINCT col) with Large Cardinality Causing Memory Spill

**Pitfall:** Using COUNT(DISTINCT CustomerId) on a 50M row Orders table where CustomerId has 20M distinct values. The Hash Aggregate memory grant (20M * ~40 bytes = 800 MB) exceeds the available memory, causing a tempdb spill.

```sql
-- ❌ Problem: Hash Aggregate spills to tempdb
SELECT COUNT(DISTINCT CustomerId) FROM Orders;
-- Memory grant: 800 MB, spills 3x to tempdb
```

**Symptom:** Query runs 10-50x slower than expected. Execution plan shows "Table Spill" or "Warning: Operator used tempdb to spill data" in the actual plan. sys.dm_exec_query_stats shows high total_spills_to_tempdb.

**Fix:**

```sql
-- ✅ Fix 1: APPROX_COUNT_DISTINCT (if exact precision not needed)
SELECT APPROX_COUNT_DISTINCT(CustomerId) FROM Orders;
-- Memory grant: 2.5 KB, no spill, 97-99.9% accuracy

-- ✅ Fix 2: COUNT(*) FROM (SELECT DISTINCT CustomerId FROM Orders) d;
-- May be more efficient than COUNT(DISTINCT) in some optimizer versions

-- ✅ Fix 3: Batch mode (requires columnstore index or SQL 2019+)
CREATE CLUSTERED COLUMNSTORE INDEX CCI_Orders ON Orders;
-- Batch mode Hash Aggregate handles large cardinality better
```

**Cost of not fixing:** Query that runs in 15 seconds instead of 200ms. With 100 concurrent users hitting the report, the server runs out of memory grants, causing RESOURCE_SEMAPHORE waits and query timeouts at scale.

### 2. EF Core Generates Correlated Subqueries for COUNT(DISTINCT)

**Pitfall:** Using LINQ's `.Select().Distinct().Count()` inside a GroupBy, expecting EF Core to generate a single efficient query.

```csharp
// ❌ Expensive EF Core pattern
var stats = context.Orders
    .GroupBy(o => o.Category)
    .Select(g => new {
        Category = g.Key,
        UniqueCustomers = g.Select(o => o.CustomerId).Distinct().Count(),
        UniqueProducts = g.Select(o => o.ProductId).Distinct().Count(),
        UniqueRegions = g.Select(o => o.Region).Distinct().Count()
    })
    .ToList();
```

**Symptom:** The query generates 3 correlated subqueries per group, each scanning the entire Orders table. With 500 groups and 10M rows, this runs 1,500 scans. SQL Server's optimizer may not fold them into one scan.

**Fix:**

```csharp
// ✅ Fix: Use raw SQL via FromSqlRaw
var sql = @"
    SELECT
        Category,
        COUNT(DISTINCT CustomerId) AS UniqueCustomers,
        COUNT(DISTINCT ProductId) AS UniqueProducts,
        COUNT(DISTINCT Region) AS UniqueRegions
    FROM Orders
    GROUP BY Category";

var stats = await dbContext.Database
    .SqlQuery<StatsDto>(FormattableStringFactory.Create(sql))
    .ToListAsync(cancellationToken);
```

**Cost of not fixing:** A reporting endpoint that takes 30 seconds instead of 1 second. Under load, the SQL Server CPU spikes to 100% and connection pool exhaustion occurs.

### 3. DISTINCT in Multiple Aggregates (One Distinct Per Aggregate)

**Pitfall:** COUNT(DISTINCT col1) and COUNT(DISTINCT col2) in the same query each require their own deduplication pass. SQL Server cannot share the deduplication across different columns.

```sql
-- Each COUNT(DISTINCT) needs its own Sort/Hash
SELECT
    COUNT(DISTINCT CustomerId) AS UniqueCustomers,
    COUNT(DISTINCT ProductId) AS UniqueProducts,
    COUNT(DISTINCT Region) AS UniqueRegions
FROM Orders;
```

**Symptom:** Three separate Sort or Hash Aggregate operations in the execution plan. The memory grant is the sum of all three, potentially exceeding available memory.

**Fix:** If only one distinct column has high cardinality, use APPROX_COUNT_DISTINCT for that column while keeping COUNT(DISTINCT) for low-cardinality columns. Or restructure as separate queries:

```sql
-- ✅ Better: use APPROX_COUNT_DISTINCT for high-cardinality columns
SELECT
    APPROX_COUNT_DISTINCT(CustomerId) AS UniqueCustomers,  -- 20M distinct
    COUNT(DISTINCT ProductId) AS UniqueProducts,            -- 500 distinct
    COUNT(DISTINCT Region) AS UniqueRegions                 -- 50 distinct
FROM Orders;
```

**Cost of not fixing:** Three large memory grants competing for the same Resource Semaphore. Under concurrent load, queries queue up waiting for memory.

### 4. COUNT(DISTINCT NULLable Column) with High NULL Density

**Pitfall:** COUNT(DISTINCT col) on a column that is 99% NULL. The engine still scans all rows and builds a hash table for the distinct non-NULL values.

```sql
-- 50M rows, 99.5% NULL in OptionalSurveyResponseId
SELECT COUNT(DISTINCT OptionalSurveyResponseId) FROM Orders;
-- Still builds hash table for 250K distinct values
```

**Symptom:** The query takes 10 seconds even though there are only 250K non-NULL distinct values. The scan reads 50M rows (all columns) and the hash table fragment is significant.

**Fix:**

```sql
-- ✅ Better: filter first to reduce scan + hash
SELECT COUNT(DISTINCT OptionalSurveyResponseId)
FROM Orders
WHERE OptionalSurveyResponseId IS NOT NULL;
-- Or even better if only need the count:
SELECT COUNT(*) FROM (
    SELECT DISTINCT OptionalSurveyResponseId
    FROM Orders
    WHERE OptionalSurveyResponseId IS NOT NULL
) d;
```

**Cost of not fixing:** 50M-row scan every time this query runs. With filtered access (index on OptionalSurveyResponseId), the scan becomes a seek.

### 5. SUM(DISTINCT col) or AVG(DISTINCT col) Semantics Misunderstanding

**Pitfall:** Engineers use SUM(DISTINCT col) thinking it means "sum of unique values per group" without realizing the semantic implications.

```sql
-- Given: values = {10, 10, 20, 30, 30}
-- SUM(DISTINCT col) = 10 + 20 + 30 = 60 (each distinct value counted once)
-- NOT 10 + 20 + 30 = 60 in this case (coincidentally same)
-- If values = {10, 10, 10}, SUM(DISTINCT col) = 10, not 30
```

**Symptom:** SUM(DISTINCT) produces unexpected totals. Managers question why the sum of unique prices (treating the same price only once) differs from total revenue. This is almost always a logical error — SUM(DISTINCT) is rarely the correct business measure.

**Fix:**

```sql
-- ✅ If you need to deduplicate before summing, use a subquery:
SELECT SUM(Amount)
FROM (
    SELECT DISTINCT OrderId, Amount  -- dedup by business key
    FROM OrderItems
) d;
```

**Cost of not fixing:** Incorrect financial reporting. SUM(DISTINCT Price) understates total revenue when multiple items have the same price.

### 6. COUNT(DISTINCT) on NVARCHAR(MAX) or Large Types

**Pitfall:** Using COUNT(DISTINCT) on large string or binary columns causes the Sort/Hash Aggregate to sort or hash large values, increasing memory grant and CPU cost.

```sql
-- COUNT(DISTINCT) on NVARCHAR(MAX) with large strings
SELECT COUNT(DISTINCT Notes) FROM Orders;
-- Notes can be up to 2GB each! Sorting these is catastrophic.
```

**Symptom:** Massive memory grant request (could be GB), Sort operator spills to tempdb, query takes minutes or runs out of memory.

**Fix:**

```sql
-- ✅ Compute hash first, then distinct count
SELECT COUNT(DISTINCT CHECKSUM(Notes)) FROM Orders;
-- ⚠️ But CHECKSUM has collisions! Use HASHBYTES for better accuracy
SELECT COUNT(DISTINCT HASHBYTES('SHA2_256', Notes)) FROM Orders;
-- Or simply don't use COUNT(DISTINCT) on large types
```

**Cost of not fixing:** Query timeouts and out-of-memory exceptions. The application becomes unavailable during peak hours when this query executes.

### Spoken Answers

**Q1: What is COUNT(DISTINCT col) and how does it differ from COUNT(col)?**

> **Average answer:** "COUNT(DISTINCT col) counts the number of unique non-NULL values in a column. COUNT(col) counts all non-NULL values including duplicates."

> **Great answer:** "COUNT(col) counts every non-NULL row — if the same value appears 100 times, it contributes 100 to the count. COUNT(DISTINCT col) deduplicates first, so each distinct value contributes at most 1. The engine accomplishes this by either sorting the data and using a Stream Aggregate to skip duplicates, or by building a hash table and counting distinct hash buckets. The cost difference is enormous: COUNT(col) uses a trivial Stream Aggregate that just increments a counter — O(N) with negligible memory. COUNT(DISTINCT col) requires either a Sort (O(N log N) with memory ~N * row_width) or Hash Aggregate (O(N) with memory ~D * key_size where D is distinct count). For a 50M row table with 20M distinct CustomerIds, COUNT(DISTINCT) can request an 800 MB memory grant and spill to tempdb, running for 25 seconds. COUNT(CustomerId) on the same table completes in 200ms. The distinction also affects NULL handling: COUNT(col) excludes NULLs, and COUNT(DISTINCT col) also excludes NULLs — but DISTINCT first filters NULLs before counting, so the 'no NULL' property is preserved."

**Q5: COUNT(DISTINCT) vs APPROX_COUNT_DISTINCT — when would you use each?**

> **Average answer:** "APPROX_COUNT_DISTINCT is faster but less accurate. Use it when you don't need exact numbers."

> **Great answer:** "I use COUNT(DISTINCT) when: (1) the distinct cardinality is low (<10K), (2) an exact count is legally required for financial reporting, or (3) the query is part of an ETL pipeline where precision propagates downstream. I use APPROX_COUNT_DISTINCT when: (1) the distinct cardinality is high (>100K), (2) the use case is a dashboard or trend chart where 99.9% accuracy is sufficient, or (3) the query is ad-hoc exploration. APPROX_COUNT_DISTINCT uses the HyperLogLog algorithm, which uses about 2.5 KB per group regardless of distinct count — compared to 800 MB for COUNT(DISTINCT) on 20M distinct values. In production, we had a monthly active users dashboard that used COUNT(DISTINCT UserId) over 200M events and ran for 40 minutes with 1.5 GB memory grant. Replacing it with APPROX_COUNT_DISTINCT dropped it to 4 minutes with 2.5 KB memory grant. The error rate was consistently below 0.3%. For trend analysis, that is more than sufficient — you are looking for direction and magnitude, not the 145th decimal place."

**Q7: How does COUNT(DISTINCT) perform on a 100M-row table with 80M distinct values?**

> **Great answer:** "This is the worst-case scenario for COUNT(DISTINCT). The Hash Aggregate requires a hash table with 80M entries. At roughly 40-60 bytes per entry (hash key + pointer + overhead), that is 3.2-4.8 GB of memory. SQL Server's query memory grant is limited to 25% of the server's max server memory by default, so on a 64 GB server, the maximum grant is ~16 GB — but no single query should consume that much. The Hash Aggregate will spill to tempdb repeatedly. Each spill writes a partition of the hash table to disk during the build phase and reads it back during the probe phase. With 80M entries spilling 3-4 times, you are looking at gigabytes of tempdb I/O. The query will run for minutes. The execution plan shows 'Hash Aggregate (Spill Level: 3)' with a Table Spill operator. The fix is: (1) APPROX_COUNT_DISTINCT reduces memory to 2.5 KB, (2) batch mode with columnstore indexes improves hash efficiency, or (3) pre-aggregate the distinct values in an indexed view or materialized table that refreshes periodically."

### COUNT(DISTINCT) with GROUPING SETS — Multi-Dimensional Unique Counts

One of the most expensive patterns in data warehousing is counting distinct values across multiple dimensions simultaneously. Combining COUNT(DISTINCT) with GROUPING SETS requires careful planning:

```sql
-- High-cost pattern: distinct customer count by multiple dimensions
SELECT
    CASE WHEN GROUPING(Year) = 1 THEN 'All Years' ELSE CAST(Year AS NVARCHAR(10)) END AS Year,
    CASE WHEN GROUPING(Country) = 1 THEN 'All Countries' ELSE Country END AS Country,
    CASE WHEN GROUPING(ProductCategory) = 1 THEN 'All Categories' ELSE ProductCategory END AS Category,
    COUNT(DISTINCT CustomerId) AS UniqueCustomers,
    COUNT(DISTINCT OrderId) AS UniqueOrders
FROM Orders o
JOIN Customers c ON o.CustomerId = c.CustomerId
JOIN Products p ON o.ProductId = p.ProductId
GROUP BY GROUPING SETS (
    (Year, Country, ProductCategory),   -- detail: unique customers per cell
    (Year, Country),                    -- unique customers per country per year
    (Year),                             -- unique customers per year
    ()                                  -- total unique customers ever
);
```

Each GROUPING SET requires its own DISTINCT deduplication. With 4 grouping sets and 2 COUNT(DISTINCT) columns, the Hash Aggregate builds 8 internal hash tables. On a 100M row fact table with 10M distinct customers, this query can request 4 GB+ memory grant and spill catastrophically.

**Optimization strategies for multi-dimensional distinct counting:**

```sql
-- Strategy 1: Pre-compute distinct values in a factless fact table
CREATE TABLE dw.FactCustomerOrders (
    CustomerId INT NOT NULL,
    OrderId INT NOT NULL,
    OrderDate DATE NOT NULL,
    ProductId INT NOT NULL
);
-- Load once, then count from this narrow table
-- The narrow table is ~10% the size of the full fact table

-- Strategy 2: Use APPROX_COUNT_DISTINCT for high-cardinality dimensions
SELECT
    CASE WHEN GROUPING(Year) = 1 THEN 'All Years' ELSE CAST(Year AS NVARCHAR(10)) END AS Year,
    APPROX_COUNT_DISTINCT(CustomerId) AS ApproxUniqueCustomers,
    COUNT(DISTINCT OrderId) AS ExactOrders  -- lower cardinality, safer for COUNT(DISTINCT)
FROM Orders
GROUP BY GROUPING SETS ((Year), ());

-- Strategy 3: Break into separate queries and UNION
-- Each query only needs one distinct hash table
SELECT Year, Country, COUNT(DISTINCT CustomerId) AS UniqueCustomers
FROM Orders JOIN Customers ON ...
GROUP BY Year, Country
UNION ALL
SELECT Year, NULL, COUNT(DISTINCT CustomerId)
FROM Orders
GROUP BY Year;
```

### COUNT(DISTINCT) with Window Functions — Running Distinct Count

SQL Server does not natively support COUNT(DISTINCT) as a window function, but you can approximate it with other techniques:

```sql
-- Attempt: COUNT(DISTINCT CustomerId) OVER (PARTITION BY Year ORDER BY Month)
-- Error: "DISTINCT is not allowed with window functions"

-- Workaround: DENSE_RANK to simulate running distinct count
SELECT
    Year,
    Month,
    CustomerId,
    DENSE_RANK() OVER (PARTITION BY Year ORDER BY CustomerId) AS RunningDistinctCount
FROM Orders;
-- This gives a rank, not a count. For the actual running distinct count:
-- (Requires SQL Server 2022+ with LAST_VALUE or subquery)

-- Better: pre-aggregate in a subquery
SELECT
    Year,
    Month,
    (
        SELECT COUNT(DISTINCT CustomerId)
        FROM Orders o2
        WHERE o2.Year = o.Year
          AND o2.Month <= o.Month
    ) AS RunningUniqueCustomers
FROM (SELECT DISTINCT Year, Month FROM Orders) o
ORDER BY Year, Month;
-- ⚠️ Correlated subquery per month — expensive for many months
```

For true running distinct counts at scale, consider using HyperLogLog in the application layer or pre-computing with batch jobs.

### COUNT(DISTINCT) in Index Maintenance and Fragmentation

When you create an index on a column that frequently uses COUNT(DISTINCT), the index maintenance overhead must be weighed against the query benefit:

```sql
-- Index for COUNT(DISTINCT CustomerId) queries
CREATE INDEX IX_Orders_CustomerId ON Orders(CustomerId);
-- This index helps COUNT(DISTINCT CustomerId) by:
-- 1. Providing a narrower index than the clustered index (fewer pages to scan)
-- 2. Enabling a Scan on a ~50MB index instead of a ~2GB clustered index

-- But the index has write costs:
-- INSERT: One additional leaf page insert per new row
-- UPDATE CustomerId: Delete + Insert in the index (worst case)
-- DELETE: One additional leaf page delete per row

-- For tables with >1M rows and high write volume (>100 writes/sec):
-- The index may cause 15-30% write performance degradation
```

To measure the tradeoff:

```sql
-- Estimate index size vs scan benefit
SELECT
    OBJECT_NAME(i.object_id) AS TableName,
    i.name AS IndexName,
    s.used_page_count * 8 / 1024 AS IndexSizeMB,
    s.row_count,
    s.row_count * 100.0 / (SELECT SUM(row_count) FROM sys.dm_db_partition_stats
                           WHERE object_id = i.object_id AND index_id = 1) AS PctOfTable
FROM sys.indexes i
JOIN sys.dm_db_partition_stats s ON i.object_id = s.object_id AND i.index_id = s.index_id
WHERE i.name = 'IX_Orders_CustomerId';
```

### COUNT(DISTINCT) for Data Quality — Detecting Duplicates

A common data quality check uses COUNT(DISTINCT) to detect duplicate or missing values:

```sql
-- Duplicate detection: count of rows vs distinct count of business keys
SELECT
    COUNT(*) AS TotalRows,
    COUNT(DISTINCT OrderId) AS DistinctOrders,
    COUNT(*) - COUNT(DISTINCT OrderId) AS DuplicateOrderRows,
    COUNT(DISTINCT CustomerId) AS UniqueCustomers,
    COUNT(DISTINCT EmailAddress) AS UniqueEmails,
    COUNT(*) - COUNT(DISTINCT EmailAddress) AS DuplicateEmails
FROM Orders o
JOIN Customers c ON o.CustomerId = c.CustomerId;
-- If DuplicateOrderRows > 0: OrderId is not unique (data integrity issue)
-- If UniqueCustomers << UniqueEmails: customers have multiple accounts

-- Finding the actual duplicates:
SELECT
    OrderId,
    COUNT(*) AS Occurrences,
    STRING_AGG(CustomerId, ', ') AS Customers
FROM Orders
GROUP BY OrderId
HAVING COUNT(*) > 1;
```

This pattern is used in ETL validation stages to catch data quality issues before loading into the data warehouse.

### COUNT(DISTINCT) in Differential Privacy and Anonymization

In systems that need to protect user privacy, COUNT(DISTINCT) plays a role in k-anonymity checks:

```sql
-- k-anonymity check: ensure no group has fewer than k records
-- If COUNT(DISTINCT UserId) < k, the data is redacted
SELECT
    Country,
    AgeGroup,
    COUNT(DISTINCT UserId) AS UniqueUsers,
    CASE WHEN COUNT(DISTINCT UserId) < 10 THEN 'REDACTED'
         ELSE CAST(COUNT(*) AS NVARCHAR(10))
    END AS SafeRowCount
FROM UserEvents
GROUP BY Country, AgeGroup;

-- This prevents re-identification attacks on small groups.
-- COUNT(DISTINCT) is used because same UserId may appear in multiple rows.
```

This is increasingly important for GDPR, CCPA, and other privacy regulations. The COUNT(DISTINCT) here is typically on a small subset of data (filtered to a specific country/age group), so cardinality is low and performance is not a concern.

-- This query runs in 45 seconds on a 100M row Events table.
-- The table has a clustered index on (EventId).

<details>
<summary>Solution</summary>

**Root cause:** Two COUNT(DISTINCT) calls on high-cardinality columns (50M distinct VisitorIds, 100K distinct PageUrls) require two separate deduplication passes. The Hash Aggregate for VisitorIds alone needs ~2 GB memory and spills to tempdb (Spill Level 3). The PageUrl distinct count adds another hash table.

**Fixed query:**

```sql
-- Option 1: APPROX_COUNT_DISTINCT for high-cardinality VisitorId
SELECT
    CAST(EventTimestamp AS DATE) AS EventDay,
    APPROX_COUNT_DISTINCT(VisitorId) AS UniqueVisitors,
    COUNT(DISTINCT PageUrl) AS UniquePages
FROM Events
WHERE EventTimestamp >= '2024-01-01'
GROUP BY CAST(EventTimestamp AS DATE)
ORDER BY EventDay DESC;

-- Option 2: Pre-aggregate VisitorIds in a CTE
WITH DailyVisitors AS (
    SELECT DISTINCT
        CAST(EventTimestamp AS DATE) AS EventDay,
        VisitorId
    FROM Events
    WHERE EventTimestamp >= '2024-01-01'
)
SELECT
    d.EventDay,
    COUNT(DISTINCT d.VisitorId) AS UniqueVisitors,
    e.UniquePages
FROM DailyVisitors d
LEFT JOIN (
    SELECT
        CAST(EventTimestamp AS DATE) AS EventDay,
        COUNT(DISTINCT PageUrl) AS UniquePages
    FROM Events
    WHERE EventTimestamp >= '2024-01-01'
    GROUP BY CAST(EventTimestamp AS DATE)
) e ON d.EventDay = e.EventDay
GROUP BY d.EventDay, e.UniquePages
ORDER BY d.EventDay DESC;
```

**Index to create:**

```sql
CREATE NONCLUSTERED INDEX IX_Events_Timestamp_VisitorId_PageUrl
    ON Events(EventTimestamp)
    INCLUDE (VisitorId, PageUrl)
    WITH (DATA_COMPRESSION = PAGE);
```

**After fix — logical reads:** ~50,000 (from 350,000), no spills, memory grant ~2.5 KB for APPROX_COUNT_DISTINCT

<details>
<summary>Solution</summary>

**Why Hash Aggregate:** The optimizer estimated 100K distinct CustomerIds (based on statistics). With this estimate, the hash table needs ~4 MB (100K * ~40 bytes). This fits comfortably in memory, and Hash Aggregate avoids the O(N log N) cost of a Sort. The optimizer chose correctly for the estimate.

**Why it spilled:** The actual distinct count was 5M (50x the estimate). The hash table needed ~200 MB, exceeding the memory grant of ~40 MB (based on the underestimated cardinality). The Hash Aggregate spilled level 2, writing hash partitions to tempdb during build and reading them back during probe.

**What to change:**
1. **Update statistics:** `UPDATE STATISTICS Orders WITH FULLSCAN;` — gives the optimizer accurate distinct count
2. **Force different plan:** Add `OPTION (ORDER GROUP)` to force Sort + Stream Aggregate if the Sort fits in memory
3. **Use APPROX_COUNT_DISTINCT:** Eliminates the spill entirely
4. **Increase memory grant:** `OPTION (MIN_GRANT_PERCENT = 10, MAX_GRANT_PERCENT = 20)` — gives more memory for the hash table

</details>

**Challenge 4 — Diagnose the concurrency problem**

A reporting application queries `SELECT Category, COUNT(DISTINCT CustomerId) ... GROUP BY Category` every 30 seconds. The query uses a Sort + Stream Aggregate with a 500 MB memory grant. During peak hours (200 concurrent users), the query time increases from 2 seconds to 30 seconds. The wait stats show RESOURCE_SEMAPHORE as the dominant wait type.

<details>
<summary>Solution</summary>

**Root cause:** RESOURCE_SEMAPHORE indicates queries waiting for memory grants. Each COUNT(DISTINCT) query requests 500 MB. With 200 concurrent users, the total memory demand is 100 GB. The server has 64 GB RAM, so queries queue up waiting for memory to become available. The default query memory grant limit is 25% of max server memory (16 GB), meaning only ~32 queries can execute simultaneously. The remaining 168 queries wait.

**Detection query:**

```sql
SELECT
    wait_type,
    wait_time_ms,
    waiting_tasks_count,
    wait_time_ms / waiting_tasks_count AS avg_wait_ms
FROM sys.dm_os_wait_stats
WHERE wait_type = 'RESOURCE_SEMAPHORE'
ORDER BY wait_time_ms DESC;
```

**Fix:**
1. **Replace COUNT(DISTINCT) with APPROX_COUNT_DISTINCT** — reduces memory grant from 500 MB to 2.5 KB
2. **Pre-compute in an indexed view** — eliminates runtime aggregation entirely
3. **Implement resource governor** — limit query memory for this workload
4. **Reduce concurrency** — implement server-side caching (Redis) for the dashboard

**In .NET:**
```csharp
// Add output caching for the dashboard endpoint
builder.Services.AddOutputCache(options =>
{
    options.AddPolicy("DashboardCache", builder =>
        builder.Expire(TimeSpan.FromMinutes(1)).SetVaryByQuery("*"));
});

// Or use Redis for distributed caching
builder.Services.AddStackExchangeRedisCache(options =>
    options.Configuration = redisConnectionString);
```

**Challenge 5 — Design the index**

**Scenario:** A SaaS analytics platform has a PageViews table with 500M rows. The most common query is: `SELECT Date, COUNT(DISTINCT VisitorId) AS UniqueVisitors FROM PageViews GROUP BY Date ORDER BY Date`. The table has a clustered columnstore index. The query runs in 30 seconds for a single day. The team needs it to run in < 3 seconds for real-time dashboards. The write volume is 10K inserts/second (batch insert every 5 seconds).

<details>
<summary>Solution</summary>

```sql
-- Strategy 1: Materialized/Indexed View (pre-compute daily)
-- SQL Server Enterprise Edition
CREATE VIEW dbo.DailyUniqueVisitors WITH SCHEMABINDING
AS
    SELECT
        CAST(ViewTimestamp AS DATE) AS ViewDate,
        COUNT(DISTINCT VisitorId) AS UniqueVisitors,
        COUNT_BIG(*) AS TotalViews
    FROM dbo.PageViews
    GROUP BY CAST(ViewTimestamp AS DATE);
GO

CREATE UNIQUE CLUSTERED INDEX IX_DailyUniqueVisitors
    ON dbo.DailyUniqueVisitors(ViewDate);
-- ⚠️ Indexed views with COUNT(DISTINCT) are not supported!
-- Need a different approach for exact counts.

-- Strategy 2: Pre-aggregation table (recommended)
CREATE TABLE dbo.PageViewsDailySummary (
    ViewDate DATE NOT NULL,
    UniqueVisitors INT NOT NULL,
    TotalViews BIGINT NOT NULL,
    CONSTRAINT PK_PageViewsDailySummary PRIMARY KEY (ViewDate)
);

-- Load data periodically (every 5 seconds, batch of recent rows)
MERGE dbo.PageViewsDailySummary AS target
USING (
    SELECT
        CAST(ViewTimestamp AS DATE) AS ViewDate,
        COUNT(DISTINCT VisitorId) AS UniqueVisitors,
        COUNT(*) AS TotalViews
    FROM dbo.PageViews
    WHERE ViewTimestamp >= DATEADD(HOUR, -1, GETUTCDATE())
    GROUP BY CAST(ViewTimestamp AS DATE)
) AS source ON target.ViewDate = source.ViewDate
WHEN MATCHED THEN
    UPDATE SET
        UniqueVisitors = source.UniqueVisitors,
        TotalViews = source.TotalViews
WHEN NOT MATCHED THEN
    INSERT (ViewDate, UniqueVisitors, TotalViews)
    VALUES (source.ViewDate, source.UniqueVisitors, source.TotalViews);

-- Query becomes trivially fast:
SELECT ViewDate, UniqueVisitors FROM dbo.PageViewsDailySummary ORDER BY ViewDate;
```

**Tradeoffs:**
- Pre-aggregation adds 5-second latency (acceptable for "real-time" dashboard)
- Batch merge of ~50K rows every 5 seconds has minimal write overhead
- The summary table stores 365 rows/year regardless of PageViews size
- Query cost: 2 logical reads (clustered index seek on PK) vs 150,000+ without summary

**What NOT to index:**
- Do not index (VisitorId) individually for this query — the columnstore already handles aggregation patterns
- Do not create a rowstore index on (ViewTimestamp) INCLUDE (VisitorId) — it would be 2-3x the size of the columnstore and add 15-20% write overhead to the 10K inserts/second

### COUNT(DISTINCT) on Compressed Columnstore Indexes

Columnstore indexes change the performance characteristics of COUNT(DISTINCT) significantly:

```sql
-- On a rowstore table (50M rows, 10M distinct CustomerId):
SELECT COUNT(DISTINCT CustomerId) FROM Sales;
-- Logical reads: 150,000 | CPU: 25s | Memory: 800 MB | Spills: 3

-- On the same table with a clustered columnstore index:
SELECT COUNT(DISTINCT CustomerId) FROM Sales;
-- Logical reads: 1,200 | CPU: 8s | Memory: 200 MB | Spills: 0
-- Columnstore compression reduces page reads by 125x
-- Batch mode Hash Aggregate is more memory-efficient
-- Accuracy check: APPROX_COUNT_DISTINCT still faster:
SELECT APPROX_COUNT_DISTINCT(CustomerId) FROM Sales;
-- Logical reads: 1,200 | CPU: 1.5s | Memory: 2.5 KB | Spills: 0
```

Columnstore compression reduces the data size dramatically, which reduces both I/O and memory grant requirements for the Hash Aggregate. However, APPROX_COUNT_DISTINCT still outperforms COUNT(DISTINCT) by 5x even on columnstore due to the HyperLogLog memory advantage. For maximum performance on COUNT(DISTINCT) workloads, combine columnstore indexes with APPROX_COUNT_DISTINCT.

</details>
