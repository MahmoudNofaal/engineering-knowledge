### Where This Fits

APPROX_COUNT_DISTINCT is an aggregate function introduced in SQL Server 2019 that returns an approximate distinct count using the HyperLogLog algorithm. A .NET backend engineer encounters this when building real-time dashboards (unique visitors per page), monitoring systems (distinct IP addresses hitting an endpoint), cardinality estimation (distinct products viewed per session), or any scenario where the exact count is unnecessary and the performance of COUNT(DISTINCT) is unacceptable. The key tradeoff: APPROX_COUNT_DISTINCT returns results within ~2% error rate with 97% confidence, but runs 10-100x faster than COUNT(DISTINCT) using a fixed 2.5 KB memory footprint regardless of cardinality. The interview signal is strong: candidates who know APPROX_COUNT_DISTINCT demonstrate awareness that not every business question requires exact precision and understand the algorithmic tradeoffs (HyperLogLog, memory vs accuracy). Candidates who cannot name a single scenario where approximate counting is acceptable show inexperience with large-scale data.

## Core Mental Model

APPROX_COUNT_DISTINCT estimates the number of distinct values in a column without materializing the distinct set. Instead of storing every unique value (as COUNT(DISTINCT) does, requiring O(N) memory proportional to distinct count), it uses the HyperLogLog (HLL) sketch algorithm: each value is hashed, the hash's leading-zero count is tracked in a fixed-size array of registers, and the final estimate is derived from the harmonic mean of these registers. The invariant is that memory is constant (~2.5 KB) and time is O(N) per scan, regardless of whether the column has 1,000 distinct values or 1 billion. The tradeoff is that the result is approximate — within ~2% of the true value with 97% probability. The recognition pattern: any dashboard or monitoring query that uses COUNT(DISTINCT) on a large dataset (>1M rows) and does not require exact precision. The database engine implements APPROX_COUNT_DISTINCT as a Hash Aggregate that builds the HLL sketch during the build phase instead of a full hash set — no sort required, no memory spill risk.

### Classification

APPROX_COUNT_DISTINCT is an aggregate function in the SELECT or HAVING clause (SQL Server 2019+). It is not SARGable — it requires a full scan of the column (or index scan of a covering index). However, the scan is significantly more efficient than COUNT(DISTINCT) because it eliminates the Sort or Hash Match operator that COUNT(DISTINCT) requires for deduplication. The function replaces the Sort or Hash Match with a much lighter HLL sketch update.

```mermaid
flowchart TD
    A[APPROX_COUNT_DISTINCT(col) in query] --> B[Parser identifies HLL aggregate]
    B --> C[Query optimizer chooses Hash Aggregate]
    C --> D[No Sort required - HLL is streaming]
    D --> E[Scan: read each row, hash the column value]
    E --> F{HLL sketch: 2.5 KB fixed array of 2^14 registers}
    F --> G[For each hash: count leading zeros]
    G --> H[Update register with max leading-zero count]
    H --> I{More rows?}
    I -->|Yes| E
    I -->|No| J[Compute harmonic mean of registers]
    J --> K[Apply bias correction for small/large cardinalities]
    K --> L[Return approximate distinct count]
    
    M[COUNT(DISTINCT) for comparison] --> N[Hash Set: memory = O(distinct count)]
    M --> O[Sort required or Hash Match Aggregate]
    O --> P[Memory grant proportional to distinct count]
    P --> Q[Risks spill at high cardinality]
    Q --> R[Lower throughput, higher latency]
```

## Deep Mechanics

### How the Engine Executes This

1. **Parsing** — The parser identifies `APPROX_COUNT_DISTINCT(col)` as a distinct aggregate. Unlike `COUNT(DISTINCT col)`, the `DISTINCT` keyword is not specified — the function implies distinct counting.

2. **Binding (Algebrizer)** — The algebrizer verifies the argument type. APPROX_COUNT_DISTINCT accepts any data type except:
   - `text`, `ntext`, `image` (deprecated types)
   - `sql_variant`
   - `xml`
   - `hierarchyid`, `geometry`, `geography` (spatial types)
   
   For supported types, the algebrizer registers the function as an HLL aggregate.

3. **Optimization** — The optimizer recognizes APPROX_COUNT_DISTINCT as a streaming-friendly aggregate. No Sort operator is added (COUNT(DISTINCT) requires a Sort or Hash Match to remove duplicates). The optimizer still considers Hash Aggregate vs Stream Aggregate for any GROUP BY columns, but the HLL sketch portion requires no sort or hash set. The memory grant for the aggregate is minimal — only the 2.5 KB HLL sketch buffer plus GROUP BY key storage.

4. **Execution — HLL Sketch Update Loop:**
   - For each row, the engine reads the column value.
   - The value is hashed using a 64-bit hash function (internal implementation detail, not documented — likely uses a variant of MurmurHash3 or xxHash).
   - The hash is split into two parts: the first 14 bits determine the register index (0 to 16,383), and the remaining 50 bits are used to count leading zeros.
   - For each hash, the leading-zero count (+1) is computed. If this value exceeds the current register value at that index, the register is updated.
   - The register array (16,384 × 6 bits = 98,304 bits ≈ 12 KB raw, compressed to ~2.5 KB in the SQL Server implementation) is updated in memory.

5. **Execution — Final Estimation:**
   - After all rows are processed, the engine computes the harmonic mean of all register values (2^(-register_value) summed, then inverted).
   - The raw HLL estimate is: `alpha_16 * 16384^2 / sum(2^(-reg[i]))` where alpha_16 ≈ 0.7213.
   - Bias correction is applied for small cardinalities (n < 5 * 16384) and large cardinalities (n > 2^32) using precomputed correction tables.

6. **Return** — The corrected estimate is returned as a `bigint`. For zero input rows, it returns 0 (not NULL, unlike COUNT(DISTINCT) which returns 0 as well, but consistent behavior).

### SQL Visibility

```sql
-- Basic APPROX_COUNT_DISTINCT: approximate count of distinct customers
SELECT 
    COUNT(DISTINCT o.CustomerId) AS ExactCustomerCount,
    APPROX_COUNT_DISTINCT(o.CustomerId) AS ApproximateCustomerCount
FROM dbo.Orders o
WHERE o.OrderDate >= '2024-01-01';

-- APPROX_COUNT_DISTINCT with GROUP BY (SQL Server 2022+)
SELECT 
    c.Region,
    COUNT(DISTINCT o.CustomerId) AS ExactCustomers,
    APPROX_COUNT_DISTINCT(o.CustomerId) AS ApproxCustomers,
    COUNT(*) AS TotalOrders
FROM dbo.Orders o
INNER JOIN dbo.Customers c ON o.CustomerId = c.CustomerId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY c.Region
ORDER BY c.Region;

-- Dashboard: unique page views per hour
SELECT 
    DATEADD(hour, DATEDIFF(hour, 0, w.LogDate), 0) AS HourBucket,
    COUNT(*) AS TotalRequests,
    APPROX_COUNT_DISTINCT(w.IPAddress) AS UniqueIPs,
    APPROX_COUNT_DISTINCT(w.UserAgent) AS UniqueUserAgents,
    APPROX_COUNT_DISTINCT(w.Url) AS UniqueUrls
FROM dbo.WebLog w
WHERE w.LogDate >= DATEADD(hour, -24, GETUTCDATE())
GROUP BY DATEDIFF(hour, 0, w.LogDate)
ORDER BY HourBucket;

-- Monitoring: distinct products viewed per session
SELECT 
    w.SessionId,
    COUNT(*) AS PageViews,
    APPROX_COUNT_DISTINCT(w.ProductId) AS DistinctProductsViewed
FROM dbo.WebLog w
WHERE w.LogDate >= DATEADD(day, -7, GETUTCDATE())
  AND w.ProductId IS NOT NULL
GROUP BY w.SessionId
HAVING COUNT(*) >= 5

SELECT
```

### Execution Plan Analysis

For the APPROX_COUNT_DISTINCT query with GROUP BY:

```
Clustered Index Scan (Orders)  -- 15% cost
  |-- Hash Match (Inner Join)  -- 20% cost
      |-- Clustered Index Scan (Customers)  -- 10% cost
          |-- Hash Match Aggregate (GROUP BY + HLL)  -- 55% cost
              |-- Compute Scalar (HLL final estimate)  -- <1% cost
                  |-- SELECT
```

**Key difference from COUNT(DISTINCT) plan:**
- COUNT(DISTINCT) plan: `[Scan] → [Sort (for dedup)] → [Stream Aggregate] → [Compute Scalar]`
- APPROX_COUNT_DISTINCT plan: `[Scan] → [Hash Aggregate with HLL] → [Compute Scalar]`

The Sort operator (memory-intensive, O(N log N), may spill) is completely eliminated. The Hash Aggregate with HLL uses the 2.5 KB sketch instead of a hash set proportional to distinct count.

**Estimated vs actual row counts:**
- Input: ~500K estimated, ~485K actual
- After GROUP BY: ~10 regions, ~10 actual
- HLL sketch: no row count estimate needed — fixed memory

**Without supporting index:**
```
Table 'Orders'. Scan count 1, logical reads 8,500
Table 'Customers'. Scan count 1, logical reads 150
```

**With covering index on Orders(OrderDate) INCLUDE (CustomerId):**
```
Table 'Orders'. Scan count 1, logical reads 1,200
Table 'Customers'. Scan count 1, logical reads 150
```

### Failure Modes

1. **SQL Server 2019 only supports scalar aggregate (no GROUP BY):**
   ```sql
   -- ❌ SQL Server 2019: Error if used with GROUP BY
   SELECT Region, APPROX_COUNT_DISTINCT(CustomerId)
   FROM Orders GROUP BY Region;
   -- Msg 10752, The function 'APPROX_COUNT_DISTINCT' is not supported with GROUP BY in this version.
   
   -- ✅ SQL Server 2022+ supports GROUP BY
   -- Legacy workaround for 2019: run separate queries per group
   ```
   Check `SELECT SERVERPROPERTY('ProductVersion')` — version 15.x is 2019, version 16.x is 2022.

2. **Database compatibility level must be 130 or higher (2019+) for scalar, 160 for GROUP BY:**
   ```sql
   -- ❌ Compatibility level < 130
   ALTER DATABASE Current SET COMPATIBILITY_LEVEL = 120;  -- SQL Server 2014
   SELECT APPROX_COUNT_DISTINCT(CustomerId) FROM Orders;
   -- Msg 208: Invalid object name 'APPROX_COUNT_DISTINCT'
   
   -- ✅ Set to 130+ for scalar, 160+ for GROUP BY
   ALTER DATABASE Current SET COMPATIBILITY_LEVEL = 160;
   ```

3. **APPROX_COUNT_DISTINCT does not accept DISTINCT keyword:**
   ```sql
   -- ❌ Syntax error: DISTINCT not allowed
   SELECT APPROX_COUNT_DISTINCT(DISTINCT CustomerId) FROM Orders;
   
   -- ✅ Correct: function is inherently distinct
   SELECT APPROX_COUNT_DISTINCT(CustomerId) FROM Orders;
   ```

4. **Insufficient accuracy for financial or legal reporting:**
   ```
   APPROX_COUNT_DISTINCT guarantees ~2% error with 97% confidence.
   For a true count of 100,000 distinct customers, results may be
   98,000 to 102,000. If exact counts are required for compliance,
   auditing, or billing, use COUNT(DISTINCT) instead.
   ```

5. **No support for multiple distinct aggregates in one query (pre-2022 limitations):**
   ```sql
   -- ✅ Supported in SQL Server 2022+
   SELECT 
       APPROX_COUNT_DISTINCT(CustomerId),
       APPROX_COUNT_DISTINCT(ProductId)
   FROM Orders;
   -- Both use independent HLL sketches, each 2.5 KB
   ```

6. **Covering index still beneficial but not for HLL itself:**
   The HLL sketch update does not benefit from an index seek (it must read all values). However, a covering index on the filtered column reduces logical reads by avoiding the clustered index scan. The HLL sketch update per row is the same cost regardless of the index choice.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ======================================================
-- Schema setup for APPROX_COUNT_DISTINCT examples
-- ======================================================

CREATE TABLE dbo.WebLog (
    LogId BIGINT IDENTITY(1,1) PRIMARY KEY,
    LogDate DATETIME2 NOT NULL,
    SessionId UNIQUEIDENTIFIER NOT NULL,
    IPAddress VARCHAR(45) NOT NULL,  -- IPv6 compatible
    Url NVARCHAR(2000) NOT NULL,
    UserAgent NVARCHAR(500) NULL,
    ProductId INT NULL,
    CustomerId INT NULL,
    PageLoadTimeMs INT NULL
);

CREATE INDEX IX_WebLog_LogDate ON dbo.WebLog(LogDate) 
    INCLUDE (SessionId, IPAddress, Url, UserAgent, ProductId, CustomerId);

-- ======================================================
-- Pattern 1: Real-time dashboard — unique visitors per hour
-- ======================================================
-- Business question: How many unique IPs and sessions per hour?
-- Exact count is unnecessary — trend detection needs speed, not precision
SELECT 
    DATEADD(hour, DATEDIFF(hour, 0, w.LogDate), 0) AS TimeBucket,
    COUNT(*) AS TotalRequests,
    APPROX_COUNT_DISTINCT(w.IPAddress) AS UniqueIPs,
    APPROX_COUNT_DISTINCT(w.SessionId) AS UniqueSessions,
    APPROX_COUNT_DISTINCT(w.CustomerId) AS UniqueCustomers,
    -- Compare with exact for verification (expensive — use sparingly)
    COUNT(DISTINCT w.IPAddress) AS ExactUniqueIPs
FROM dbo.WebLog w
WHERE w.LogDate >= DATEADD(hour, -24, GETUTCDATE())
GROUP BY DATEDIFF(hour, 0, w.LogDate)
ORDER BY TimeBucket;

-- ======================================================
-- Pattern 2: Cardinality monitoring — distinct products per day
-- ======================================================
SELECT 
    CAST(w.LogDate AS DATE) AS Day,
    APPROX_COUNT_DISTINCT(w.ProductId) AS ApproxDistinctProductsViewed,
    COUNT(*) AS TotalViews
FROM dbo.WebLog w
WHERE w.LogDate >= DATEADD(day, -90, GETUTCDATE())
  AND w.ProductId IS NOT NULL
GROUP BY CAST(w.LogDate AS DATE)
ORDER BY Day;

-- ======================================================
-- Pattern 3: Large-scale distinct counting — 100M+ rows
-- ======================================================
-- WebLog has 500M rows. COUNT(DISTINCT IPAddress) would sort 500M rows.
-- APPROX_COUNT_DISTINCT scans 500M rows with 2.5 KB memory.
DECLARE @StartTime DATETIME2 = SYSDATETIME();

SELECT 
    APPROX_COUNT_DISTINCT(w.IPAddress) AS ApproxUniqueIPs,
    APPROX_COUNT_DISTINCT(w.SessionId) AS ApproxUniqueSessions,
    APPROX_COUNT_DISTINCT(w.Url) AS ApproxUniqueUrls
FROM dbo.WebLog w
WHERE w.LogDate >= DATEADD(day, -30, GETUTCDATE());

SELECT DATEDIFF(ms, @StartTime, SYSDATETIME()) AS ElapsedMs;

-- ======================================================
-- Pattern 4: Error budget — SLA monitoring
-- ======================================================
-- Count distinct users who experienced errors vs total
SELECT 
    DATEADD(hour, DATEDIFF(hour, 0, w.LogDate), 0) AS TimeBucket,
    APPROX_COUNT_DISTINCT(CASE WHEN w.StatusCode >= 500 
        THEN w.CustomerId END) AS DistinctUsersWithErrors,
    APPROX_COUNT_DISTINCT(w.CustomerId) AS DistinctUsersTotal,
    COUNT(*) AS TotalRequests
FROM dbo.WebLog w
WHERE w.LogDate >= DATEADD(hour, -48, GETUTCDATE())
GROUP BY DATEDIFF(hour, 0, w.LogDate)
ORDER BY TimeBucket;

-- ======================================================
-- Pattern 5: Batch validation — APPROX vs EXACT comparison
-- ======================================================
-- Run periodically to validate APPROX accuracy against exact count
-- Uses sampling to keep the exact count cost manageable
WITH SampleWebLog AS (
    SELECT TOP (100000) w.IPAddress, w.SessionId
    FROM dbo.WebLog w
    WHERE w.LogDate >= DATEADD(day, -1, GETUTCDATE())
    ORDER BY NEWID()  -- random sample
)
SELECT 
    COUNT(DISTINCT sw.IPAddress) AS ExactUniqueIPs_Sample,
    APPROX_COUNT_DISTINCT(sw.IPAddress) AS ApproxUniqueIPs_Sample,
    ABS(
        COUNT(DISTINCT sw.IPAddress) - APPROX_COUNT_DISTINCT(sw.IPAddress)
    ) / NULLIF(COUNT(DISTINCT sw.IPAddress), 0) * 100.0 AS PercentError
FROM SampleWebLog sw;
```

```csharp
    }

    public async Task<IReadOnlyList<HourlyTraffic>> GetHourlyTrafficAsync(
        int lookbackHours, CancellationToken cancellationToken = default)
    {
        // SQL Server 2022+ GROUP BY with APPROX_COUNT_DISTINCT
        var results = await _dbContext.Database
            .SqlQueryRaw<HourlyTraffic>(@"
                SELECT 
                    DATEADD(hour, DATEDIFF(hour, 0, LogDate), 0) AS TimeBucket,
                    COUNT(*) AS TotalRequests,
                    APPROX_COUNT_DISTINCT(IPAddress) AS UniqueIPs,
                    APPROX_COUNT_DISTINCT(SessionId) AS UniqueSessions,
                    APPROX_COUNT_DISTINCT(Url) AS UniqueUrls
                FROM WebLog
                WHERE LogDate >= DATEADD(hour, -@LookbackHours, GETUTCDATE())
                GROUP BY DATEDIFF(hour, 0, LogDate)
                ORDER BY TimeBucket",
                new SqlParameter("@LookbackHours", lookbackHours))
            .ToListAsync(cancellationToken);

        return results.AsReadOnly();
    }

    public async Task<long> GetApproximateDistinctCountAsync<T>(
        string tableName, string columnName,
        DateTime? startDate = null, CancellationToken cancellationToken = default)
    {
        // Generic helper for ad-hoc approximate distinct counts
        var sql = startDate.HasValue
            ? $"SELECT APPROX_COUNT_DISTINCT([{columnName}]) FROM [{tableName}] WITH (NOLOCK) WHERE LogDate >= @StartDate"
            : $"SELECT APPROX_COUNT_DISTINCT([{columnName}]) FROM [{tableName}] WITH (NOLOCK)";

        var parameters = new List<SqlParameter>();
        if (startDate.HasValue)
            parameters.Add(new SqlParameter("@StartDate", startDate.Value));

        var result = await _dbContext.Database
            .SqlQueryRaw<long>(sql, parameters.ToArray())
            .FirstOrDefaultAsync(cancellationToken);

        return result;
    }
}
```

## Gotchas and Production Pitfalls

### 1. Using APPROX_COUNT_DISTINCT When Exact Counts Are Required

**Pitfall:** Using APPROX_COUNT_DISTINCT for billing, compliance, or auditing where exact counts are mandatory.

```sql
-- ❌ Wrong: Approximate count for billing
SELECT APPROX_COUNT_DISTINCT(CustomerId) AS BillableCustomers
FROM UsageLog WHERE Month = '2024-01';
-- Result: 99,847 when true count is 101,233 (1.4% error)
-- Customer is underbilled by 1,386 customers

-- ✅ Correct: Use COUNT(DISTINCT) for billing
SELECT COUNT(DISTINCT CustomerId) AS BillableCustomers
FROM UsageLog WHERE Month = '2024-01';
```

**Symptom:** Business reports show discrepancies with financial systems. Customers are billed incorrectly (either over or under by ~2%). Auditors flag the inconsistency.

**Fix:** Always use COUNT(DISTINCT) for financial, legal, or compliance reporting. APPROX_COUNT_DISTINCT is for dashboards, monitoring, and internal estimates where ~2% error is acceptable.

**Cost of not fixing:** Financial reconciliation fails. Regulatory non-compliance. Customer disputes over billing amounts. Internal dashboards show trends correctly but absolute values are untrustworthy.

### 2. APPROX_COUNT_DISTINCT with GROUP BY on SQL Server 2019

**Pitfall:** Writing GROUP BY queries in a codebase that targets SQL Server 2019, which does not support APPROX_COUNT_DISTINCT with GROUP BY.

```sql
-- ❌ Wrong: GROUP BY + APPROX_COUNT_DISTINCT on SQL Server 2019
SELECT Region, APPROX_COUNT_DISTINCT(CustomerId) AS UniqueCustomers
FROM Orders GROUP BY Region;
-- Msg 10752: 'APPROX_COUNT_DISTINCT' is not supported with GROUP BY

-- ✅ Correct 1: Use scalar aggregate without GROUP BY (one value at a time)
SELECT 'North' AS Region, APPROX_COUNT_DISTINCT(CustomerId) FROM Orders WHERE Region = 'North'
UNION ALL
SELECT 'South', APPROX_COUNT_DISTINCT(CustomerId) FROM Orders WHERE Region = 'South';

-- ✅ Correct 2: Upgrade to SQL Server 2022
-- ✅ Correct 3: Use COUNT(DISTINCT) despite performance cost
```

**Symptom:** Deployment to a SQL Server 2019 environment fails with error 10752. Rollback required.

**Fix:** Check `SELECT SERVERPROPERTY('ProductMajorVersion')` — version 15 = 2019, version 16 = 2022. For 2019, only scalar APPROX_COUNT_DISTINCT is supported (no GROUP BY). For 2022+, GROUP BY is supported.

**Cost of not fixing:** Production deployment fails. Database migration script aborts mid-execution. Rollback procedures must run under pressure.

### 3. Overestimating Accuracy — 2% Error Can Mask Problems

**Pitfall:** Assuming APPROX_COUNT_DISTINCT is accurate enough for all non-financial use cases.

**Reality:** The 2% error rate is a bound, not a guarantee. For small cardinalities (< 1,000), the error can be higher (up to 5-10%). For very large cardinalities (> 1 billion), the HLL algorithm can saturate.

```sql
-- Check actual error on your data
SELECT 
    COUNT(DISTINCT CustomerId) AS Exact,
    APPROX_COUNT_DISTINCT(CustomerId) AS Approx,
    ABS(COUNT(DISTINCT CustomerId) - APPROX_COUNT_DISTINCT(CustomerId)) AS AbsoluteError,
    (ABS(COUNT(DISTINCT CustomerId) - APPROX_COUNT_DISTINCT(CustomerId)) * 100.0) 
        / NULLIF(COUNT(DISTINCT CustomerId), 0) AS PercentError
FROM Orders
WHERE OrderDate >= '2024-01-01';
```

**Symptom:** Dashboard spike alerts trigger falsely because APPROX_COUNT_DISTINCT reports a 3% change when the true value changed 1% during a re-hash of data. The error is magnified for small time buckets.

**Fix:** For time-series dashboards, validate APPROX_COUNT_DISTINCT against exact COUNT(DISTINCT) on a sample periodically. Apply error smoothing (rolling averages) to reduce the impact of the approximation.

**Cost of not fixing:** Ops team chases false alarms from monitoring. Degraded trust in dashboard data. Engineers begin to distrust the metrics system.

### 4. EF Core Cannot Translate — Hidden Full Table Scan

**Pitfall:** Materializing all rows in EF Core and computing distinct count in C# (or via LINQ's `.Distinct().Count()`).

```csharp
// ❌ Wrong: Pulls all rows into memory
var ipCount = await dbContext.WebLog
    .Where(w => w.LogDate >= startDate)
    .Select(w => w.IPAddress)
    .Distinct()
    .CountAsync();  // EF Core translates this to COUNT(DISTINCT...) on SQL Server
// Actually, EF Core does translate .Distinct().Count() to COUNT(DISTINCT)
// But the point: COUNT(DISTINCT) is expensive

// ❌ Worse: Materializes all distinct values
var ips = await dbContext.WebLog
    .Where(w => w.LogDate >= startDate)
    .Select(w => w.IPAddress)
    .Distinct()
    .ToListAsync();  // All distinct IPs in memory — O(N) memory
var count = ips.Count;

// ✅ Correct: Use raw SQL with APPROX_COUNT_DISTINCT
var approxCount = await dbContext.Database
    .SqlQueryRaw<long>(@"SELECT APPROX_COUNT_DISTINCT(IPAddress) 
        FROM WebLog WHERE LogDate >= @p0", startDate)
    .FirstOrDefaultAsync();
```

**Symptom:** Dashboard endpoint returns in 8 seconds instead of 200ms. Memory usage spikes from 200MB to 2GB. Page load timeout from the browser.

**Fix:** Never use LINQ for APPROX_COUNT_DISTINCT — always use raw SQL via `FromSql` or `SqlQueryRaw`. Add a read replica connection for dashboard queries.

**Cost of not fixing:** Dashboard API endpoint times out under load. Application pool recycles due to memory pressure. Monitoring system goes down because it consumed too many resources to monitor itself.

### 5. NULL Handling Is Inconsistent with COUNT(DISTINCT)

**Pitfall:** Assuming APPROX_COUNT_DISTINCT handles NULLs the same way as COUNT(DISTINCT) in edge cases.

```sql
-- COUNT(DISTINCT col): NULLs are NOT counted, but if ALL values are NULL, returns 0
-- APPROX_COUNT_DISTINCT(col): NULLs are NOT counted, same behavior

SELECT COUNT(DISTINCT NULLIF(CustomerId, CustomerId)) FROM Orders;  -- 0
SELECT APPROX_COUNT_DISTINCT(NULLIF(CustomerId, CustomerId)) FROM Orders;  -- 0

-- Both behave the same for NULL — but APPROX_COUNT_DISTINCT cannot combine
-- with CASE expressions in some versions
```

**Symptom:** No actual inconsistency — but developers unfamiliar with both functions assume APPROX_COUNT_DISTINCT might be non-deterministic. It is deterministic for a given dataset and hash function.

**Fix:** Understand that both COUNT(DISTINCT) and APPROX_COUNT_DISTINCT ignore NULLs. The approximate nature is in the count, not in the inclusion/exclusion of NULLs.

**Cost of not fixing:** Minimal — but a developer might add `WHERE col IS NOT NULL` redundantly, slightly increasing the scan cost. Not a production bug, but a code readability issue.

### 6. APPROX_COUNT_DISTINCT Cannot Be Used in Window Functions

**Pitfall:** Attempting to use APPROX_COUNT_DISTINCT in a window function context.

```sql
-- ❌ Wrong: APPROX_COUNT_DISTINCT does not support OVER()
SELECT 
    LogDate,
    APPROX_COUNT_DISTINCT(IPAddress) OVER(PARTITION BY LogDate) AS UniqueIPs
FROM WebLog;
-- SQL Server does not support OVER() with APPROX_COUNT_DISTINCT

-- ✅ Correct: Use GROUP BY instead
SELECT 
    LogDate,
    APPROX_COUNT_DISTINCT(IPAddress) AS UniqueIPs
FROM WebLog
GROUP BY LogDate;
```

**Symptom:** Syntax error when trying to use OVER() with APPROX_COUNT_DISTINCT.

**Fix:** Use GROUP BY for windowed approximate distinct counts. For running totals of distinct counts, consider pre-aggregating with GROUP BY and then using SUM() OVER() on the pre-aggregated results.

**Cost of not fixing:** Query fails to compile. Developer wastes time looking for non-existent window function support.

### Benchmark: Before and After


### Spoken Answers

**Q1: What is APPROX_COUNT_DISTINCT and what problem does it solve?**

> **Average answer:** It's a function that counts distinct values approximately, faster than COUNT(DISTINCT). It was added in SQL Server 2019.

> **Great answer:** APPROX_COUNT_DISTINCT solves the problem of counting distinct values at scale without the memory and CPU cost of exact deduplication. Internally, it implements the HyperLogLog algorithm: each value is hashed, and only the maximum leading-zero count per hash register is tracked in a fixed 2.5 KB array of 16,384 6-bit registers. It does not store the actual values — it only stores a probabilistic sketch. This means the function uses 2.5 KB of memory regardless of whether there are 1,000 distinct values or 1 billion. The cost is accuracy: it returns approximately 2% error with 97% confidence. The performance difference is dramatic — on a 10M row table with 1M distinct IPs, COUNT(DISTINCT) requires either a 45 MB hash set or a sort of all 10M rows, running in about 5 seconds. APPROX_COUNT_DISTINCT scans the same 10M rows but only updates the sketch, running in about 150ms — a 30x improvement. I use it in production for real-time dashboards where trend direction is more important than exact precision — unique visitors per hour, distinct IPs per endpoint, cardinality of product views.

**Q4: What is the accuracy guarantee and when is it insufficient?**

> **Average answer:** It's about 97% accurate with 2% error. Not for financial reporting.

> **Great answer:** The formal guarantee is 97% confidence that the result is within 2% of the true value. Practically, for a true count of 100,000, you'll see results between 98,000 and 102,000 in 97 out of 100 runs. The error distribution is not uniform — it's higher for very small cardinalities (under 1,000, the error can reach 5-10% due to register sparsity) and for very large cardinalities (over 2^32, the registers saturate). SQL Server applies bias correction for these edge cases using precomputed correction tables. APPROX_COUNT_DISTINCT is insufficient for: billing and invoicing where exact counts determine charges, compliance reporting where regulators require exact numbers, audit trails where discrepancies must be explainable, and scientific research requiring exact statistics. It is appropriate for: real-time dashboards (unique visitors), monitoring systems (distinct error types per minute), capacity planning (distinct IPs over time), and any scenario where trend detection matters more than absolute values. The key interview insight is that APPROX_COUNT_DISTINCT is a tradeoff decision — you trade exactness for speed and memory. The candidate who knows when to make that tradeoff understands the distinction between operational reporting and analytical reporting.

**Q8: How do EF Core and Dapper handle APPROX_COUNT_DISTINCT?**

> **Average answer:** EF Core doesn't support it in LINQ. Dapper can run the SQL directly.

> **Great answer:** Neither EF Core nor Dapper provide LINQ-level abstraction for APPROX_COUNT_DISTINCT because it's a SQL Server-specific function (not part of the SQL standard) added in SQL Server 2019. The approach differs between the two:

For **EF Core**, there is zero LINQ translation. Calling `.Distinct().CountAsync()` generates `COUNT(DISTINCT ...)` — the exact version. To use APPROX_COUNT_DISTINCT, you must use raw SQL via `FromSql` or `SqlQueryRaw`. The anti-pattern is materializing all rows into memory with `ToListAsync()` and computing distinct count in C#, which defeats the entire purpose. The correct pattern: `await dbContext.Database.SqlQueryRaw<long>(@"SELECT APPROX_COUNT_DISTINCT(IPAddress) FROM WebLog WHERE LogDate >= @p0", startDate).FirstOrDefaultAsync()`.

For **Dapper**, it's straightforward: write the T-SQL query with APPROX_COUNT_DISTINCT, map to a DTO or use `ExecuteScalarAsync<long>`. Dapper does not care about what SQL you write — it only maps results. The Dapper approach is cleaner because there's no ORM abstraction layer to fight against. The key Dapper consideration: APPROX_COUNT_DISTINCT returns `bigint`, so map to `long` in C#.

For **PostgreSQL**, there is no built-in approximate distinct count. You need the `hll` extension for HyperLogLog support, which exposes `hll_add_agg`, `hll_cardinality`, `hll_count_distinct`. The API is different from SQL Server's, requiring explicit hash function calls.

<details>
<summary>Answers</summary>

1. **APPROX_COUNT_DISTINCT** is an aggregate function introduced in SQL Server 2019 that returns an approximate distinct count using the HyperLogLog algorithm. It uses a fixed 2.5 KB memory buffer (16,384 6-bit registers) to maintain a probabilistic sketch of the distinct values, resulting in ~2% error with 97% confidence.

2. **HyperLogLog sketch:** Each value is hashed with a 64-bit hash function. The first 14 bits determine the register index (0-16383). The remaining 50 bits are scanned for the position of the first 1 bit (leading-zero count + 1). Each register stores the maximum leading-zero count seen for hashes mapped to its index. The final estimate uses the harmonic mean of all register values with bias correction. No original values are stored — only the sketch.

3. **Performance measurement:** `SET STATISTICS TIME ON` shows CPU time and elapsed time — APPROX_COUNT_DISTINCT will show ~95% less CPU time. `sys.dm_exec_query_stats` shows `total_worker_time`, `total_logical_reads`, and `total_spills` — APPROX_COUNT_DISTINCT should show zero spills while COUNT(DISTINCT) may show spills for large cardinalities.

4. **GROUP BY failure:** Using APPROX_COUNT_DISTINCT with GROUP BY on SQL Server 2019 (version 15.x) fails with error 10752. GROUP BY support requires SQL Server 2022 (version 16.x) with compatibility level 160.

5. **EF Core and APPROX_COUNT_DISTINCT:** EF Core has no LINQ translation for APPROX_COUNT_DISTINCT. Must use raw SQL via `FromSql`, `SqlQueryRaw`, or `ExecuteSqlRaw`. The `.Distinct().CountAsync()` pattern generates `COUNT(DISTINCT ...)` — not the approximate version.

6. **Dapper implementation:**
   ```csharp
   const string sql = @"SELECT APPROX_COUNT_DISTINCT(IPAddress) 
       FROM WebLog WHERE LogDate >= @StartDate";
   var count = await connection.ExecuteScalarAsync<long>(
       new CommandDefinition(sql, new { StartDate = startDate },
           cancellationToken: cancellationToken));
   ```

7. **Execution plan difference:** COUNT(DISTINCT) produces either a Sort operator followed by Stream Aggregate or a Hash Match Aggregate (for deduplication). APPROX_COUNT_DISTINCT replaces these with a lighter HLL sketch update within the Hash Aggregate operator — no Sort, no full hash set. The final Compute Scalar applies the HLL bias correction formula.

8. **Scale threshold:** APPROX_COUNT_DISTINCT becomes significantly beneficial above ~1M rows. At 10M rows with 1M distinct values, it's ~30x faster. At 100M rows with 10M distinct values, COUNT(DISTINCT) often spills to tempdb while APPROX_COUNT_DISTINCT uses 2.5 KB and completes in seconds.

9. **Supporting index:** A narrow covering index on the filtered column (`CREATE INDEX IX_WebLog_LogDate ON WebLog(LogDate) INCLUDE (IPAddress)`) reduces logical reads but cannot eliminate the scan. APPROX_COUNT_DISTINCT must read every row — it cannot use an index seek for the HLL update.

10. **60-second explanation:** "HyperLogLog is a probabilistic algorithm that estimates the number of distinct elements in a set using very little memory. It works by hashing each element and tracking the maximum number of leading zeros seen across 16,384 buckets. If we see a hash starting with 10 zeros, it's likely the first time we've encountered that pattern — and this gives us information about how many distinct elements we've seen. The registers store only the max leading-zero count per bucket, not the values themselves. The final estimate uses the harmonic mean of all bucket values, which handles outliers well. SQL Server implements this in APPROX_COUNT_DISTINCT with a fixed 2.5 KB memory budget, regardless of whether we're counting 1,000 or 1 billion distinct values. The tradeoff is accuracy: we get within 2% of the true value 97% of the time."

</details>

**Challenge 1 — Write the SQL**

The SRE team needs a real-time monitoring dashboard showing, for the last 24 hours broken into hourly buckets: total HTTP requests, approximate distinct IP addresses, approximate distinct user sessions, and approximate distinct URLs requested. The WebLog table has 500M rows. Results must return in under 2 seconds.

<details>
<summary>Solution</summary>

```sql
SELECT 
    DATEADD(hour, DATEDIFF(hour, 0, w.LogDate), 0) AS TimeBucket,
    COUNT(*) AS TotalRequests,
    APPROX_COUNT_DISTINCT(w.IPAddress) AS UniqueIPs,
    APPROX_COUNT_DISTINCT(w.SessionId) AS UniqueSessions,
    APPROX_COUNT_DISTINCT(w.Url) AS UniqueUrls
FROM dbo.WebLog w
WHERE w.LogDate >= DATEADD(hour, -24, GETUTCDATE())
GROUP BY DATEDIFF(hour, 0, w.LogDate)
ORDER BY TimeBucket;
```

```sql
-- This dashboard query runs in 45 seconds on a 200M row WebLog table.
-- SET STATISTICS IO: logical reads = 8500
-- SET STATISTICS TIME: CPU time = 45200 ms, elapsed time = 48700 ms
SELECT 
    CAST(w.LogDate AS DATE) AS Day,
    COUNT(DISTINCT w.IPAddress) AS UniqueIPs,
    COUNT(DISTINCT w.SessionId) AS UniqueSessions,
```

```sql
-- Fixed query: use APPROX_COUNT_DISTINCT

    APPROX_COUNT_DISTINCT(w.IPAddress) AS UniqueIPs,
    APPROX_COUNT_DISTINCT(w.SessionId) AS UniqueSessions,
    APPROX_COUNT_DISTINCT(w.CustomerId) AS UniqueCustomers
FROM dbo.WebLog w

GROUP BY CAST(w.LogDate AS DATE)
ORDER BY Day;
```

**Index to create:**

```sql
CREATE INDEX IX_WebLog_LogDate_INCLUDES 
    ON dbo.WebLog(LogDate) 
    INCLUDE (IPAddress, SessionId, CustomerId);
```

**After fix — logical reads:** ~8,500 (same scan, but each row read once for three HLL updates instead of three separate sorts) **Elapsed time:** ~1,500 ms (from 48,700 ms) — 32x improvement.

**Challenge 3 — Explain the execution plan**

Your query uses APPROX_COUNT_DISTINCT with GROUP BY. The execution plan shows a Hash Match Aggregate operator with a warning: "Operator used tempdb to spill data during execution." Approximate aggregates use 2.5 KB fixed memory — why would there be a spill?

<details> <summary>Solution</summary>

**Why the spill:** The APPROX_COUNT_DISTINCT HLL sketch itself does not spill — it's 2.5 KB. However, the GROUP BY columns are processed with a Hash Aggregate operator. If the GROUP BY cardinality estimate is inaccurate (e.g., 100 groups estimated but 1M groups actual), the hash table for the GROUP BY keys can exceed the memory grant, causing the Hash Aggregate to spill to tempdb. The HLL sketches are stored within each group's hash table entry — each entry contains the HLL registers plus the group key. With 1M groups, this is 1M × ~2.5 KB = ~2.5 GB for the HLL sketches alone, plus the group key overhead.

**To fix:** Update statistics on the GROUP BY columns. Consider reducing the number of groups (e.g., aggregate to a higher level). If the data is ordered by the GROUP BY columns, a Stream Aggregate eliminates the hash table entirely.

**Detection query:**
```sql
SELECT qs.total_spills, qs.total_worker_time, 
       SUBSTRING(st.text, 1, 200) AS query_text
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
WHERE qs.total_spills > 0
ORDER BY qs.total_spills DESC;
```

</details>

**Challenge 4 — Diagnose the concurrency problem**

A monitoring system runs APPROX_COUNT_DISTINCT queries on WebLog every 5 seconds for 50 different metrics. After 2 hours of runtime, the queries start timing out with RESOURCE_SEMAPHORE waits. The execution plans show Hash Match Aggregate with HLL. The database has 64 GB RAM available.

<details> <summary>Solution</summary>

**Root cause:** Each APPROX_COUNT_DISTINCT execution requests a memory grant for the Hash Aggregate. Although the HLL sketch is 2.5 KB, the Hash Aggregate memory grant request also accounts for the GROUP BY key hash table. With 50 concurrent queries, each requesting ~100 MB of memory grant (based on GROUP BY cardinality estimates), the total request can reach 5 GB. If the resource semaphore limits total memory grant to 25% of the buffer pool (16 GB), 50 concurrent queries each requesting 100 MB can exceed available grant memory, causing RESOURCE_SEMAPHORE waits.

**Detection query:**
```sql
SELECT 
    session_id, request_id, grant_time,
    requested_memory_kb, granted_memory_kb, 
    required_memory_kb, dop -- degree of parallelism
FROM sys.dm_exec_query_memory_grants
WHERE session_id > 50
ORDER BY requested_memory_kb DESC;
```

**Fix:**
1. Reduce concurrency: use a single batch job that collects all 50 metrics in one query with multiple APPROX_COUNT_DISTINCT calls
2. Reduce memory grant: `OPTION (MIN_GRANT_PERCENT = 1)` limits each query's memory grant
3. Use a read replica for monitoring queries to isolate from OLTP workload
4. Increase resource semaphore: `ALTER RESOURCE GOVERNOR` to allow more concurrent memory grants

**In .NET:** Use a semaphore to limit concurrent monitoring queries to 3-5. Batch metric collection into a single database round trip.

</details>

**Challenge 5 — Design the index**

A WebLog table with 500M rows is queried by a real-time dashboard every 30 seconds. The dashboard queries show: total requests, unique IPs, unique sessions, and unique URLs for the last hour, grouped by 5-minute buckets. The current plan does a full clustered index scan (42,000 logical reads) because there is no index on LogDate. Write overhead is minimal because logs are append-only. Response time must be under 500ms.

<details> <summary>Solution</summary>

```sql
-- Index 1: Covering index for all dashboard columns
CREATE INDEX IX_WebLog_LogDate_AllMetrics
    ON dbo.WebLog(LogDate)
    INCLUDE (IPAddress, SessionId, Url, CustomerId);
```

**Why this index:**
- `LogDate` as the leading column enables a range seek for the last hour filter
- All four columns used in APPROX_COUNT_DISTINCT are included, making the index covering for the query
- The clustered index (LogId clustered) is not scanned — only the narrow index pages are read
- For append-only logs, index maintenance is sequential writes to the end of the index — low fragmentation

**Tradeoffs accepted:**
- ~30% more storage (LogDate in both clustered and nonclustered index)
- INSERT overhead: +1 index page write per row (sequential, low cost)
- For 500M rows, index size is ~15 GB (narrow index: 8 bytes LogId + 8 bytes LogDate + 45 bytes IP + 16 bytes SessionId + 500 bytes Url + 4 bytes CustomerId ≈ 600 bytes per row × 500M ≈ 300 GB uncompressed, but page compression reduces this significantly — expect ~50 GB)

**Expected improvement:**
- Logical reads: 42,000 → ~200 (range seek on LogDate scans only the last hour's index pages, which is 1/365th of the data)
- Elapsed time: ~1,200 ms → ~150 ms
- Dashboard refresh drops from 1.2 seconds to 150ms — below the perceived 200ms threshold for real-time

**What NOT to index:**
- No need for a filtered index (WHERE LogDate > ...) because the query pattern requires the last hour only — but the maintenance overhead of a filtered index is not justified for append-only data
- No need for a columnstore index — the query is point-in-time, not analytical over the full dataset
- No need for included columns on IPAddress/SessionId to be indexed (they are included, not key columns — no seeks on them, but no sort cost either)

</details>
