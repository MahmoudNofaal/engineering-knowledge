### Where This Fits

`CONTAINSTABLE` is a table-valued function that performs the same full-text search as CONTAINS but returns ranked results — each matching document gets a `RANK` value (0-1000) representing its relevance to the search query. A .NET backend engineer uses CONTAINSTABLE whenever search results must be ordered by relevance: product search showing most relevant products first, document search that surfaces the best-matching documents at the top, or any "top-N by relevance" pattern. The interview signal is strong because CONTAINSTABLE combines the TF-IDF ranking understanding with the JOIN mechanics and the top-N optimization pattern.

### Classification

CONTAINSTABLE is a **table-valued function** that belongs to the full-text search family. It takes: (1) a table name, (2) a column list or `*`, (3) a full-text predicate (same syntax as CONTAINS), and optionally (4) a `topN` integer. It returns a two-column result: `[KEY]` (the table's full-text key column value, with datatype matching the key column) and `RANK` (an INT, 0-1000). The rank is computed by the FTS engine using a TF-IDF-based formula with document length normalization. The function must be JOINed with the base table using `ON base.PK = ct.[KEY]` to retrieve the actual data columns. The `topN` parameter limits the number of results to the N highest-ranked documents — this is the critical optimization that reduces key lookups to only the most relevant matches.

|---|---|---|

## Deep Mechanics

### How the Engine Computes the Rank

**Step-by-step for: `CONTAINSTABLE(Products, ProductDescription, '"wireless" AND "mouse"', 20)`**

1. **Predicate processing** — The FTS engine receives the predicate `'"wireless" AND "mouse"'`. It word-breaks each term, checks the stoplist, and identifies two required terms: 'wireless' and 'mouse'.

2. **Inverted index lookups** — The engine looks up 'wireless' in the inverted index, retrieving the document frequency (DF = the number of documents containing 'wireless') and the postings list (document keys + term frequency per document + position data). It does the same for 'mouse'.

3. **Document intersection** — The engine intersects the postings lists to find documents containing BOTH 'wireless' and 'mouse'. Only documents in the intersection are candidates for ranking.

4. **Term Frequency (TF) extraction** — For each candidate document, the engine extracts the term frequency for each matched term: how many times does 'wireless' appear in this document? How many times does 'mouse' appear? These TF values come from the inverted index postings.

5. **Inverse Document Frequency (IDF) calculation** — The engine computes IDF for each term:
   - IDF(wireless) = log10(N / DF_wireless) where N = total number of documents in the index
   - IDF(mouse) = log10(N / DF_mouse)
   - Rarer terms get higher IDF. If 'wireless' appears in 10,000 out of 100,000 documents, IDF = log10(100000/10000) = 1.0. If 'mouse' appears in only 500 documents, IDF = log10(100000/500) = 2.3.

6. **Raw rank calculation** — For each candidate document D:
   - Rank(D) = (log10(TF_wireless_in_D + 1) * IDF_wireless) + (log10(TF_mouse_in_D + 1) * IDF_mouse)
   - Using log10(TF + 1) dampens the effect of very high term frequencies (a term appearing 100 times is not 100x more relevant than appearing once).

7. **Document length normalization** — The raw rank is multiplied by a normalization factor: shorter documents get a slight boost, longer documents get a slight penalty. The formula is roughly: `norm_rank = raw_rank * (0.5 + 0.5 * avg_doc_length / doc_length)`. This prevents very long documents from dominating results just because they contain more words total.

8. **Top-N selection** — The documents are sorted by rank descending. With `topN = 20`, only the 20 highest-ranked documents are kept. The engine returns only these 20 rows to the relational engine — dramatically reducing key lookups compared to returning all matches.

9. **Rank normalization to 0-1000** — The top-ranked document in this query's result set gets rank 1000. Other documents get proportionally lower ranks. This normalization is per-query: the same document with the same content would get different ranks for different queries.

10. **Result returned** — The FTS engine returns a table with columns `[KEY]` (ProductId values of the top 20 documents) and `RANK` (normalized to 0-1000). The relational engine then does 20 key lookups into the base table.

**Rank Formula Summary:**

```
For each document D and each matched term T:
  contribution(T, D) = log10(TF(T, D) + 1) * IDF(T)
  where IDF(T) = log10(N / DF(T))

Total rank = SUM(contribution(term, D)) * length_normalization(D)

Final rank = (total_rank / max_total_rank_in_results) * 1000
```

### Execution Plan Analysis

```sql
SELECT p.ProductId, p.ProductName, p.UnitPrice, ct.RANK
FROM dbo.Products p
INNER JOIN CONTAINSTABLE(Products, ProductDescription,
    '"wireless" AND "bluetooth"',
    20) ct ON p.ProductId = ct.[KEY]
WHERE p.IsActive = 1
ORDER BY ct.RANK DESC;
```

**Expected plan shape:**
```
[FullTextMatch — CONTAINSTABLE predicate]
    → [Sort — ORDER BY RANK DESC (top 20 already sorted by FTS engine)]
    → [Nested Loops — INNER JOIN on p.ProductId = ct.[KEY]]
        → [Clustered Index Seek — key lookup for each of top 20]
        → [Concatenation — the 20 rows from FTS engine]
```

- The `FullTextMatch` operator for CONTAINSTABLE returns 20 rows (because of topN=20). Each row has [KEY] and RANK.
- The `Sort` may not be necessary if the FTS engine returns rows in rank order — the engine already sorts by rank for topN selection.
- The `Nested Loops` join iterates over the 20 rows from CONTAINSTABLE. For each row, it does a `Clustered Index Seek` (or non-clustered if covered) on the base table.
- The total logical reads = 20 × 2-3 pages per seek = ~40-60 for the key lookups.

**Without topN:**
- Without topN, the FTS engine would return ALL 1,200+ matching documents.
- The Nested Loops join would do 1,200+ key lookups.
- The Sort would be expensive, sorting 1,200 rows by RANK.
- Logical reads = 1,200+ × 2-3 = 2,400+.

**Improvement:** topN reduces key lookups from 1,200+ to 20 — a 60x reduction in IO.

### Failure Modes

**Failure Mode 1: Forgetting the JOIN — using RANK as a column**

```sql
-- ❌ CONTAINSTABLE must be joined; it is not a scalar function
SELECT ProductId, RANK FROM CONTAINSTABLE(Products, ProductDescription, '"wireless"');
-- This works but returns only [KEY] and RANK, not ProductName or other columns
```

**Failure Mode 2: Mismatched key column type in JOIN**

```sql
-- ❌ If [KEY] is INT but join column is BIGINT, implicit conversion
SELECT p.ProductId, p.ProductName, ct.RANK
FROM dbo.Products p
INNER JOIN CONTAINSTABLE(Products, ProductDescription, '"wireless"') ct
    ON CAST(p.ProductId AS BIGINT) = ct.[KEY];  -- Non-SARGable join
```

**Failure Mode 3: Comparing ranks across queries**

```sql
-- ❌ Rank of 800 in one query is NOT comparable to rank of 800 in another
SELECT RANK FROM CONTAINSTABLE(Products, *, '"wireless"', 1);  -- Rank: 800
SELECT RANK FROM CONTAINSTABLE(Products, *, '"bluetooth"', 1); -- Rank: 600
-- The first result with rank 800 is NOT more relevant than the second with 600
```

### Primary SQL Implementation

```sql
-- ============================================================
-- Production Product Search with CONTAINSTABLE
-- ============================================================

-- Schema: Products with full-text index on multiple columns

-- Scenario 1: E-commerce product search — top 20 by relevance
CREATE PROCEDURE dbo.SearchProducts
    @SearchTerm     NVARCHAR(200),
    @MaxResults     INT = 20,
    @CategoryName   NVARCHAR(100) = NULL,
    @MinPrice       DECIMAL(18,2) = NULL,
    @MaxPrice       DECIMAL(18,2) = NULL,
    @BrandName      NVARCHAR(100) = NULL
AS
BEGIN
    SET NOCOUNT ON;

    SELECT TOP (@MaxResults)
        p.ProductId,
        p.ProductName,
        p.ShortDescription,
        p.UnitPrice,
        p.BrandName,
        p.CategoryName,
        ct.RANK AS Relevance
    FROM dbo.Products p
    INNER JOIN CONTAINSTABLE(Products,
        (ProductName, ShortDescription, LongDescription,
         CategoryName, BrandName, Tags),
        @SearchTerm,
        @MaxResults) ct ON p.ProductId = ct.[KEY]
    WHERE p.IsActive = 1
      AND (@CategoryName IS NULL OR p.CategoryName = @CategoryName)
      AND (@MinPrice IS NULL OR p.UnitPrice >= @MinPrice)
      AND (@MaxPrice IS NULL OR p.UnitPrice <= @MaxPrice)
      AND (@BrandName IS NULL OR p.BrandName = @BrandName)
    ORDER BY ct.RANK DESC;
END;
GO

-- Scenario 2: Weighted column search with ISABOUT
-- ProductName matches weighted more heavily than Description
DECLARE @SearchTerm NVARCHAR(200) = 'wireless mouse';

SELECT p.ProductId, p.ProductName, p.UnitPrice, ct.RANK
FROM dbo.Products p
INNER JOIN CONTAINSTABLE(Products,
    (ProductName, ShortDescription, LongDescription),
    'ISABOUT("wireless" WEIGHT(0.7), "mouse" WEIGHT(0.3))',
    20) ct ON p.ProductId = ct.[KEY]
WHERE p.IsActive = 1
ORDER BY ct.RANK DESC;

-- Scenario 3: Document search with proximity ranking
SELECT d.DocumentId, d.DocumentName, ct.RANK
FROM dbo.Documents d
INNER JOIN CONTAINSTABLE(Documents, DocumentContent,
    'NEAR(("confidential", "agreement"), 10)',
    10) ct ON d.DocumentId = ct.[KEY]
ORDER BY ct.RANK DESC;

-- Scenario 4: Article search with inflectional forms
SELECT a.ArticleId, a.Title, a.PublishedDate, ct.RANK
FROM dbo.Articles a
INNER JOIN CONTAINSTABLE(Articles, (Title, Body),
    'FORMSOF(INFLECTIONAL, optimize) AND "database"',
    20) ct ON a.ArticleId = ct.[KEY]
WHERE a.IsPublished = 1
ORDER BY ct.RANK DESC;

-- Scenario 5: Search with multiple weighted ISABOUT terms
-- across different columns using a combined predicate
SELECT p.ProductId, p.ProductName, p.UnitPrice, ct.RANK
FROM dbo.Products p
INNER JOIN CONTAINSTABLE(Products, *,
    'ISABOUT("wireless" WEIGHT(0.5), "bluetooth" WEIGHT(0.3),

    30) ct ON p.ProductId = ct.[KEY]
WHERE p.IsActive = 1
ORDER BY ct.RANK DESC;
```

```csharp
// ============================================================
// EF Core — CONTAINSTABLE Search
// ============================================================

public class RankedSearchResult
{
    public int ProductId { get; set; }
    public string ProductName { get; set; } = string.Empty;
    public string? ShortDescription { get; set; }
    public decimal UnitPrice { get; set; }
    public string? BrandName { get; set; }
    public int RANK { get; set; }
}

public class RankedSearchService
{
    private readonly ApplicationDbContext _dbContext;

    public RankedSearchService(ApplicationDbContext dbContext)
    {
        _dbContext = dbContext;
    }

    public async Task<IReadOnlyList<RankedSearchResult>> SearchProductsAsync(
        string searchTerm,
        int maxResults = 20,
        string? categoryName = null,
        decimal? maxPrice = null,
        CancellationToken cancellationToken = default)
    {
        if (string.IsNullOrWhiteSpace(searchTerm))
            return Array.Empty<RankedSearchResult>();

        // Build CONTAINSTABLE predicate from search term
        var terms = searchTerm
            .Split(' ', StringSplitOptions.RemoveEmptyEntries)
            .Take(6)  // Limit to 6 terms for performance
            .Select(t => $"\"{t.Replace("\"", "\\\"")}\"");
        var predicate = string.Join(" AND ", terms);

        // EF Core cannot generate CONTAINSTABLE in LINQ — use raw SQL
        var sql = $@"
            SELECT p.ProductId, p.ProductName, p.ShortDescription,
                   p.UnitPrice, p.BrandName, ct.RANK
            FROM dbo.Products p
            INNER JOIN CONTAINSTABLE(Products,
                (ProductName, ShortDescription, LongDescription,
                 CategoryName, BrandName, Tags),
                {{0}},
                {{1}}) ct
                ON p.ProductId = ct.[KEY]
            WHERE p.IsActive = 1
            {(categoryName != null ? "AND p.CategoryName = {2}" : "")}
            {(maxPrice.HasValue ? "AND p.UnitPrice <= {3}" : "")}
            ORDER BY ct.RANK DESC";

        var results = await _dbContext.Database
            .SqlQueryRaw<RankedSearchResult>(sql,
                predicate,
                maxResults,
                categoryName,
                maxPrice)
            .ToListAsync(cancellationToken);

        return results;
    }

    public async Task<IReadOnlyList<RankedSearchResult>> SearchWithWeightedTermsAsync(
        string primaryTerm,
        string secondaryTerm,
        int maxResults = 20,
        CancellationToken cancellationToken = default)
    {
        // Use ISABOUT with weighted terms
        var predicate = $"ISABOUT(\"{primaryTerm}\" WEIGHT(0.7), \"{secondaryTerm}\" WEIGHT(0.3))";

        var sql = $@"
            SELECT p.ProductId, p.ProductName, p.ShortDescription,
                   p.UnitPrice, p.BrandName, ct.RANK
            FROM dbo.Products p
            INNER JOIN CONTAINSTABLE(Products,
                (ProductName, ShortDescription, LongDescription,
                 CategoryName, BrandName, Tags),
                {{0}},
                {{1}}) ct
                ON p.ProductId = ct.[KEY]
            WHERE p.IsActive = 1
            ORDER BY ct.RANK DESC";

        var results = await _dbContext.Database
            .SqlQueryRaw<RankedSearchResult>(sql,
                predicate,
                maxResults)
            .ToListAsync(cancellationToken);

        return results;
    }
}
```

### SQL Server vs PostgreSQL Differences

## Gotchas and Production Pitfalls

### 1. [KEY] Column Name Must Be Bracketed

**Pitfall:** `[KEY]` is a reserved word in T-SQL. The CONTAINSTABLE result column is literally named `KEY` (or `[KEY]`). Many developers forget the brackets and get a syntax error.

```sql
-- ❌ Syntax error — KEY is a reserved word
SELECT ProductId, RANK
FROM CONTAINSTABLE(Products, ProductDescription, '"wireless"') ct
INNER JOIN dbo.Products p ON p.ProductId = ct.KEY;
-- Msg 156: Incorrect syntax near the keyword 'KEY'.

-- ✅ Correct — bracket the KEY column
SELECT p.ProductId, p.ProductName, ct.RANK
FROM CONTAINSTABLE(Products, ProductDescription, '"wireless"') ct
INNER JOIN dbo.Products p ON p.ProductId = ct.[KEY];
```

**Symptom:** Syntax error at JOIN clause. The keyword `KEY` is interpreted as a T-SQL keyword, not a column name.

**Fix:** Always use `[KEY]` (with brackets) when referencing the CONTAINSTABLE key column.

**Cost of not fixing:** Query compilation failure. Developer confusion — the column name is `KEY` but it needs brackets.

### 2. Ranks Are Not Comparable Across Queries

**Pitfall:** Comparing rank values from different CONTAINSTABLE invocations is meaningless. Rank is normalized per query — a rank of 800 in one query does not mean the same as 800 in another.

```sql
-- ❌ Meaningless comparison
-- Query 1: 'wireless mouse' returns rank 842 for Product A
-- Query 2: 'bluetooth keyboard' returns rank 215 for Product A
-- Product A is NOT "less relevant" to query 2 than to query 1
-- The rank normalization is per-query

-- ✅ Correct: ranks are only meaningful within a single query
SELECT p.ProductId, p.ProductName, ct.RANK
FROM dbo.Products p
INNER JOIN CONTAINSTABLE(Products, ProductDescription,
    '"wireless" AND "mouse"',
    10) ct ON p.ProductId = ct.[KEY]
ORDER BY ct.RANK DESC;
-- Product with rank 1000 is the most relevant match for THIS query
```

**Symptom:** Product managers compare ranks across different searches and think the ranking is broken because "wireless mouse" always gets higher ranks than other queries.

**Fix:** Document that ranks are per-query and only used for ordering within a result set. Never compare ranks across queries.

**Cost of not fixing:** Incorrect A/B testing conclusions, confused stakeholders, wasted hours investigating "ranking inconsistencies."

### 3. topN Silently Drops Results

**Pitfall:** Using topN means only the top N documents by internal FTS rank are returned. If you have additional WHERE filters (IsActive, CategoryName), the JOIN + filter may eliminate some of the top-N results, potentially returning fewer than N results even though more matching documents exist.

```sql
-- ❌ May return < 20 results even though more exist
SELECT p.ProductId, p.ProductName, ct.RANK
FROM dbo.Products p
INNER JOIN CONTAINSTABLE(Products, ProductDescription,
    '"wireless"',
    20) ct ON p.ProductId = ct.[KEY]
WHERE p.IsActive = 1
  AND p.CategoryName = 'Clothing';  -- Only 8 products in Clothing
-- Returns 8 results instead of 20 — the other 12 top-ranked products
-- were in different categories
```

**Symptom:** The search returns fewer results than expected. The developer assumed topN=20 guarantees 20 results.

**Fix:** Either increase topN to account for post-filter removal, or apply filters before the CONTAINSTABLE join using a subquery or CTE.

```sql
-- ✅ Option 1: Increase topN to account for filtered-out results
SELECT p.ProductId, p.ProductName, ct.RANK
FROM dbo.Products p
INNER JOIN CONTAINSTABLE(Products, ProductDescription,
    '"wireless"',
    200) ct ON p.ProductId = ct.[KEY]  -- Higher topN
WHERE p.IsActive = 1
  AND p.CategoryName = 'Clothing'      -- Filter removes most matches
ORDER BY ct.RANK DESC;

-- ✅ Option 2: Apply non-FTS filters first (may be different performance)
WITH FilteredProducts AS (
    SELECT ProductId, ProductName
    FROM dbo.Products
    WHERE IsActive = 1
      AND CategoryName = 'Clothing'
)
SELECT p.ProductId, p.ProductName, ct.RANK
FROM FilteredProducts p
INNER JOIN CONTAINSTABLE(Products, ProductDescription,
    '"wireless"',
    20) ct ON p.ProductId = ct.[KEY]
ORDER BY ct.RANK DESC;
```

**Cost of not fixing:** Search returns few or zero results for some category + term combinations, leading users to think products do not exist.

### 4. CONTAINSTABLE with ISABOUT — Weights Affect Ranking in Non-Intuitive Ways

**Pitfall:** The ISABOUT weights are applied to the term's contribution to the rank, but the final rank normalization (0-1000) can mask the effect of weights. A term with WEIGHT(0.1) can still contribute significantly if it has high TF-IDF in a document.

```sql
-- ❌ Weights are not absolute — they are relative to each other
SELECT p.ProductId, p.ProductName, ct.RANK
FROM dbo.Products p
INNER JOIN CONTAINSTABLE(Products, ProductDescription,
    'ISABOUT("wireless" WEIGHT(0.9), "the" WEIGHT(0.1))',
    20) ct ON p.ProductId = ct.[KEY]
ORDER BY ct.RANK DESC;
-- If 'the' appears 50 times in a document (high TF) but has low IDF
-- (appears in almost every document), its contribution may be lower
-- than 'wireless' appearing once but with high IDF (rarer term).
-- The weight is a multiplier within the rank formula, not a filter.
```

**Symptom:** Documents with the heavily weighted term do not always rank highest.

**Fix:** Understand that ISABOUT weights multiply the TF-IDF contribution of each term. The actual rank depends on both the weight AND the term's frequency across the document set. Use ISABOUT to give preference, not to enforce filtering.

**Cost of not fixing:** Frustration with "broken" weighted search. Product managers expect WEIGHT(0.9) to mean "90% of the rank comes from this term."

### 5. JOIN Order Affects Performance with TopN

**Pitfall:** Writing the CONTAINSTABLE JOIN with the base table on the left vs right side of the JOIN can affect the query plan, especially when topN is small.

```sql
-- ❌ Suboptimal join order — base table scanned before FTS results
SELECT p.ProductId, p.ProductName, ct.RANK
FROM dbo.Products p
LEFT HASH JOIN CONTAINSTABLE(Products, ProductDescription,
    '"wireless"', 20) ct ON p.ProductId = ct.[KEY]
WHERE ct.[KEY] IS NOT NULL;
-- The optimizer may choose to scan Products first (2M rows) then probe FTS
```

**Symptom:** The execution plan shows a scan of the base table instead of the expected 20 key lookups.

**Fix:** Always use INNER JOIN with CONTAINSTABLE on the LEFT side (logically the driving table) and a Nested Loops join hint if needed:

```sql
-- ✅ INNER JOIN with topN drives the plan
SELECT p.ProductId, p.ProductName, ct.RANK
FROM CONTAINSTABLE(Products, ProductDescription,
    '"wireless"', 20) ct
INNER JOIN dbo.Products p ON p.ProductId = ct.[KEY]
WHERE p.IsActive = 1
ORDER BY ct.RANK DESC;
```

**Cost of not fixing:** The optimizer may scan the full base table instead of doing 20 key lookups, negating the benefit of topN.

### CONTAINSTABLE vs CONTAINS for COUNT

For counting matches, CONTAINS is more efficient than CONTAINSTABLE:

```sql
-- COUNT: CONTAINS is faster (no rank computation)
SELECT COUNT(*)
FROM dbo.Products
WHERE CONTAINS(ProductDescription, '"wireless"');
-- Logical reads: 4

-- COUNT via CONTAINSTABLE — rank is computed unnecessarily
SELECT COUNT(*)
FROM CONTAINSTABLE(Products, ProductDescription, '"wireless"') ct;
-- Logical reads: 4 (same FTS engine work but rank computed for each match)
```

**Rule:** Use CONTAINS in WHERE for membership-only queries. Use CONTAINSTABLE only when you need rank for ORDER BY.

CONTAINSTABLE reads the inverted index — it does NOT write. Write amplification is entirely in the full-text index population (see 8.247).

### Question Bank

1. **What is CONTAINSTABLE and how does it differ from CONTAINS?** — CONTAINSTABLE is a table-valued function that returns [KEY] and RANK; CONTAINS is a WHERE clause predicate that only determines membership.

2. **How does the FTS engine compute the rank value for CONTAINSTABLE?** — TF-IDF with log frequency dampening and document length normalization, normalized to 0-1000.

3. **What is the topN parameter and why is it important for performance?** — Limits results to top N by rank, reducing key lookups from all matches to only N.

4. **What is the difference between RANK values from different CONTAINSTABLE queries?** — Ranks are normalized per query and NOT comparable across queries.

5. **CONTAINSTABLE vs FREETEXTTABLE — when would you use each?** — CONTAINSTABLE for explicit operators (AND, OR, NEAR, FORMSOF) with ranking; FREETEXTTABLE for natural language queries with automatic stemming and ranking.

6. **What does the execution plan look like for CONTAINSTABLE with topN?** — FullTextMatch returns N rows → Nested Loops → Clustered Index Seek (N executions). No scan of the base table.

7. **How does CONTAINSTABLE scale with the number of matches?** — Without topN, scales linearly with match count (each match = 1 key lookup). With topN, constant-time key lookups regardless of total match count.

8. **How do you implement CONTAINSTABLE in EF Core and Dapper?** — Both require raw SQL. JOIN the base table with CONTAINSTABLE on `base.PK = ct.[KEY]`. Map RANK as a property.

### Spoken Answers

**Q1: What is CONTAINSTABLE and how does it differ from CONTAINS?**

> **Average answer:** CONTAINSTABLE is like CONTAINS but it also returns a rank.

> **Great answer:** CONTAINSTABLE is a table-valued function while CONTAINS is a WHERE clause predicate — this is the fundamental difference. CONTAINS evaluates whether a row matches the search predicate and returns TRUE or FALSE for each row — it is a membership test. CONTAINSTABLE, on the other hand, returns a result set with two columns: `[KEY]` (the full-text key value, typically the primary key of the matching row) and `RANK` (an integer from 0 to 1000 representing relevance). The RANK is computed by the FTS engine using a TF-IDF formula: for each matched term, it calculates term frequency (TF) in the document, the inverse document frequency (IDF) across all documents, applies log dampening to TF, and normalizes by document length. The final rank is normalized so the best match gets 1000 and others are proportional. Because CONTAINSTABLE is a function that returns a table, it must be JOINed with the base table using `ON base.PK = ct.[KEY]`. The other critical difference is the `topN` parameter — an optional integer that tells the FTS engine to return only the N highest-ranked results. This is the key performance optimization: without topN, a search matching 10,000 documents returns all 10,000 keys and requires 10,000 key lookups. With topN=20, only 20 keys are returned, requiring only 20 key lookups — a 500x reduction in IO.

**Q3: What is the topN parameter and why is it important for performance?**

> **Average answer:** topN limits the number of results. It makes queries faster by returning fewer rows.

> **Great answer:** The topN parameter in CONTAINSTABLE tells the full-text engine to compute ranks for ALL matching documents, sort them by rank descending, and then RETURN only the top N rows. This is a critical performance optimization because it limits the number of key lookups into the base table. Without topN, a query matching 10,000 documents requires the relational engine to do 10,000 clustered index seeks (each 2-3 page reads) to retrieve the requested columns — that is 20,000-30,000 logical reads and potentially tens of millions of page reads per second at high query rates. With topN=20, the FTS engine does the same amount of work internally (it must still evaluate all 10,000 documents to know which 20 rank highest), but only 20 keys are returned to the relational engine, requiring only 40-60 logical reads for key lookups. The catch is that topN can interact badly with additional WHERE filters: if you specify topN=20 but then apply a filter like `AND CategoryName = 'Clothing'` that eliminates 15 of the 20 results, you get only 5 results even though more matches exist in other categories. The fix is to either increase topN to account for filter selectivity or apply filters differently. In practice, I always use topN with CONTAINSTABLE in production — typically topN=20 for user-facing search and topN=100 for admin search — because the IO reduction is enormous and the top N results are the ones users actually see.

**Q8: How do you implement CONTAINSTABLE in EF Core and Dapper?**

> **Average answer:** Both require raw SQL. You JOIN the base table with CONTAINSTABLE on the key.

> **Great answer:** Neither EF Core nor Dapper has any built-in support for CONTAINSTABLE — it is a SQL Server-specific table-valued function that has no LINQ equivalent. In EF Core, you use `SqlQuery` (EF Core 7+) or `FromSqlRaw` to execute the raw SQL, passing the search predicate and topN as parameters. The result type must include a `RANK` property to capture the rank value. The JOIN pattern is: `FROM dbo.Products p INNER JOIN CONTAINSTABLE(Products, ..., @predicate, @topN) ct ON p.ProductId = ct.[KEY]`. In Dapper, the same SQL pattern is used with `QueryAsync<T>`. One key implementation detail is that the `[KEY]` column must be bracketed because `KEY` is a T-SQL reserved word. Another is that the CONTAINS predicate string must be built by the application — you split the user input into terms, sanitize each term (escape double quotes), and join with AND, OR, or ISABOUT depending on the search mode. I typically encapsulate this predicate-building logic in a dedicated service or extension method so it can be reused across both EF Core and Dapper implementations. For weighted search, I use ISABOUT with WEIGHT parameters in the predicate string — this is transparent to both EF Core and Dapper since the predicate is just a string parameter.

How many key lookups occur? How many logical reads?" The candidate who articulates that the FTS engine evaluates all 50,000 documents internally to compute and sort ranks, but only 20 key lookups occur (40-60 logical reads), understands the optimization deeply.

### Application Checklist

- [ ] Full-text index is populated and available for queries
- [ ] CONTAINSTABLE is used instead of CONTAINS when ORDER BY relevance is needed

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Relevance ordering (RANK 0-1000)|FTS engine must compute rank for all matches|
|topN optimization reduces IO|topN + filters may return fewer results than expected|
|Explicit operator control (AND, OR, NEAR)|Predicate building logic in application code|
|ISABOUT weighted terms|Weights are relative, not absolute — unintuitive|

### Scale Thresholds

- **Relevant when table exceeds ~10K rows** — Below this, simple ORDER BY on other columns may be acceptable.
- **Critical when match count exceeds ~100 per query** — Without topN, 100 matches = 100 key lookups = ~200-300 logical reads per query. At 100 qps, that is 20,000-30,000 logical reads/second.
- **Required when match count exceeds ~1,000 per query** — Always use topN. At 1,000 matches without topN, each query does 2,000-3,000 logical reads. At 50 qps, that is 100,000-150,000 reads/second.
- **Degraded when ISABOUT weights are used with >10 terms** — Each term adds rank computation cost. Keep weighted terms to 3-5.
- **Prohibitive when catalog size exceeds ~50M documents** — FTS engine rank computation across all matches becomes CPU-bound at this scale. Consider Elasticsearch.

**Challenge 1 — Write the SQL**

Create a stored procedure that searches the `dbo.Products` table for products matching the word "wireless" in the `ProductDescription`. Return the top 10 results ranked by relevance, including `ProductId`, `ProductName`, `UnitPrice`, and `RANK`. Only return active products (`IsActive = 1`) with a price under $500.

<details>
<summary>Solution</summary>

```sql
CREATE PROCEDURE dbo.SearchTopWirelessProducts
AS
BEGIN
    SET NOCOUNT ON;

    SELECT p.ProductId, p.ProductName, p.UnitPrice, ct.RANK
    FROM dbo.Products p
    INNER JOIN CONTAINSTABLE(Products, ProductDescription,
        '"wireless"',
        10) ct ON p.ProductId = ct.[KEY]
    WHERE p.IsActive = 1
      AND p.UnitPrice < 500.00
    ORDER BY ct.RANK DESC;
END;
GO
```

**Logical reads:** ~22 (2 for index root + leaf + 20 for 10 key lookups × 2) + FTS engine work (opaque).

**Execution plan:** `FullTextMatch` (topN=10) → `Nested Loops` → `Clustered Index Seek` (10 executions) → `Filter` (IsActive, UnitPrice) → `Sort` (RANK DESC)

**Challenge 2 — Fix the performance problem**

```sql
-- This query runs in 8 seconds on a 1M row Products table.
-- There are ~50,000 products containing 'wireless'.
-- The query returns the top 20, but it is slow.

SELECT p.ProductId, p.ProductName, p.UnitPrice, p.CategoryName, ct.RANK
FROM CONTAINSTABLE(Products, ProductDescription, '"wireless"') ct
INNER JOIN dbo.Products p ON p.ProductId = ct.[KEY]
WHERE p.IsActive = 1
  AND p.CategoryName = 'Electronics'
ORDER BY ct.RANK DESC;
```

<details>
<summary>Solution</summary>

**Root cause:** No topN parameter! CONTAINSTABLE returns ALL 50,000 matching document keys. The INNER JOIN does 50,000 clustered index seeks to retrieve ProductName, UnitPrice, CategoryName, and IsActive. Then the WHERE clause filters out most results — only ~5,000 belong to the 'Electronics' category and are active. 45,000 key lookups were wasted. Then the Sort by RANK operates on 5,000 rows.

```sql
-- ✅ Fix: Add topN to limit key lookups
SELECT TOP 20
    p.ProductId, p.ProductName, p.UnitPrice, p.CategoryName, ct.RANK
FROM CONTAINSTABLE(Products, ProductDescription,
    '"wireless"',
    200) ct                       -- higher topN to account for filter removal
INNER JOIN dbo.Products p ON p.ProductId = ct.[KEY]
WHERE p.IsActive = 1
  AND p.CategoryName = 'Electronics'
ORDER BY ct.RANK DESC;
```

**Index to create:**

```sql
CREATE NONCLUSTERED INDEX IX_Products_SearchCovering
    ON dbo.Products (ProductId)
    INCLUDE (ProductName, UnitPrice, CategoryName, IsActive);
```

**After fix — logical reads:** ~402 (200 key lookups × 2 pages). From 50,000+ key lookups to 200 — 250x reduction. Elapsed time from 8 seconds to ~200ms.

</details>

**Challenge 3 — Explain the execution plan**

```sql
SELECT p.ProductId, p.ProductName, ct.RANK
FROM dbo.Products p
INNER JOIN CONTAINSTABLE(Products, ProductDescription, '"wireless"', 20) ct
    ON p.ProductId = ct.[KEY]
WHERE p.IsActive = 1
ORDER BY ct.RANK DESC;
```

The execution plan shows: `Clustered Index Scan` (1.2M rows, 97% cost) → `Merge Join` → `FullTextMatch` (20 rows). Why is there a Clustered Index Scan costing 97% when you expected 20 key lookups? What changed?

<details>
<summary>Solution</summary>

**Why Clustered Index Scan:** The optimizer has chosen to SCAN the Products table and MERGE JOIN with the 20 rows from CONTAINSTABLE. This typically happens when:
1. The table has a small number of pages (it is small or recently truncated).
2. Statistics are out of date — the optimizer thinks the Products table has only a few pages.
3. There is an index hint or the optimizer estimates that scanning the full table is cheaper than doing 20 key lookups (possible on very small tables or when the key lookup cost estimate is very high due to fragmentation).

**To get the expected plan (20 key lookups):**
- Update statistics: `UPDATE STATISTICS dbo.Products;`
- Add a covering non-clustered index to reduce key lookup cost: `CREATE NONCLUSTERED INDEX IX_Products_Covering ON dbo.Products (ProductId) INCLUDE (ProductName, IsActive);`
- Force the desired plan with a join hint: `INNER LOOP JOIN`

```sql
-- Force Nested Loops (key lookup pattern)
SELECT p.ProductId, p.ProductName, ct.RANK
FROM CONTAINSTABLE(Products, ProductDescription, '"wireless"', 20) ct
INNER LOOP JOIN dbo.Products p ON p.ProductId = ct.[KEY]
WHERE p.IsActive = 1
ORDER BY ct.RANK DESC;
```

**Tradeoff:** The Merge Join + Scan may be correct if the Products table is small. For large tables, the Nested Loops + Key Lookup pattern is better. The optimizer's choice depends on statistics accuracy and the relative cost estimates.

</details>

**Challenge 4 — Diagnose the concurrency problem**

Your search query uses CONTAINSTABLE with topN=20. It runs at 100 qps on an 8-core server. The average response time is 15ms. Suddenly, response time spikes to 500ms. You check `sys.dm_os_waiting_tasks` and see `PAGEIOLATCH_SH` waits on the Products clustered index. The Products table has 2M rows. What is happening?

<details>
<summary>Solution</summary>

**Root cause:** The CONTAINSTABLE topN=20 returns 20 keys, which should only require 20 key lookups = ~60 logical reads per query. At 100 qps, that is 6,000 logical reads/second — well within normal range. The `PAGEIOLATCH_SH` waits on the Products clustered index suggest the key lookups are reading from disk (cold cache) or that the pages are being evicted between queries. Three possible causes:
1. The buffer pool is under pressure — other queries are evicting the Products pages.
2. The covering index was dropped or is not being used.
3. The clustered index key is wide (e.g., includes large columns), causing each key lookup to read more pages.

**Detection query:**

```sql
-- Check buffer pool hit ratio
SELECT
    (COUNT(CASE WHEN is_modified = 0 THEN 1 END) * 1.0 / COUNT(*)) * 100 AS BufferHitPercent
FROM sys.dm_os_buffer_descriptors
WHERE database_id = DB_ID();

-- Check page life expectancy
SELECT
    [object_name],
    [counter_name],
    [cntr_value] AS PageLifeExpectancySeconds
FROM sys.dm_os_performance_counters
WHERE [object_name] LIKE '%Manager%'
  AND [counter_name] = 'Page life expectancy';
```

**Fix:**
```sql
-- Ensure the covering index exists and is used
CREATE NONCLUSTERED INDEX IX_Products_SearchCovering
    ON dbo.Products (ProductId)
    INCLUDE (ProductName, IsActive)
    WITH (FILLFACTOR = 100);
GO

-- Increase server memory if buffer pool is small
-- EXEC sp_configure 'max server memory (MB)', 16384;
-- RECONFIGURE;
```

**Challenge 5 — Design the index**

**Scenario:** A job board's `dbo.Jobs` table has 200K job postings. Columns: `JobId` (INT PK), `Title` (NVARCHAR(300)), `Description` (NVARCHAR(MAX) — average 2KB), `CompanyName` (NVARCHAR(200)), `Location` (NVARCHAR(200)), `SalaryRange` (NVARCHAR(100)), `PostedDate` (DATETIME2), `IsActive` (BIT). Users search for jobs using keywords like "senior software engineer C# .NET remote". Results must be ordered by relevance AND posted date (most relevant + most recent first). There are 50 queries/second and 500 new jobs/day.

<details>
<summary>Solution</summary>

```sql
-- Full-text catalog
CREATE FULLTEXT CATALOG JobSearchCatalog;
GO

-- Full-text index on Title and Description
CREATE FULLTEXT INDEX ON dbo.Jobs
(
    Title       LANGUAGE 1033,
    Description LANGUAGE 1033
)
KEY INDEX PK_Jobs
ON JobSearchCatalog
WITH
(
    CHANGE_TRACKING MANUAL,        -- 500 new jobs/day — batch updates
    STOPLIST = SYSTEM
);
GO

-- Schedule population every 30 minutes:
-- ALTER FULLTEXT INDEX ON dbo.Jobs START UPDATE POPULATION;

-- Covering index for key lookups with all SELECT columns
CREATE NONCLUSTERED INDEX IX_Jobs_SearchCovering
    ON dbo.Jobs (JobId)
    INCLUDE (Title, CompanyName, Location, SalaryRange, PostedDate, IsActive);
GO

-- Search procedure with combined relevance + date ordering
CREATE PROCEDURE dbo.SearchJobs
    @SearchPhrase   NVARCHAR(500),
    @MaxResults     INT = 20
AS
BEGIN
    SET NOCOUNT ON;

    SELECT TOP (@MaxResults)
        j.JobId,
        j.Title,
        j.CompanyName,
        j.Location,
        j.SalaryRange,
        j.PostedDate,
        ct.RANK AS Relevance
    FROM dbo.Jobs j
    INNER JOIN CONTAINSTABLE(Jobs, (Title, Description),
        @SearchPhrase,
        @MaxResults * 3) ct    -- fetch 3x to account for date ordering
        ON j.JobId = ct.[KEY]
    WHERE j.IsActive = 1
    ORDER BY
        ct.RANK * 0.7 + DATEDIFF(DAY, j.PostedDate, GETUTCDATE()) * (-1) * 0.3 DESC;
        -- Combined score: 70% relevance, 30% recency
END;
GO
```

**Why these choices:**
- CHANGE_TRACKING MANUAL: 500 jobs/day is low volume; batch every 30 minutes
- Covering index: avoids clustered index seeks entirely — all SELECT columns are in the INCLUDE
- topN × 3: because the combined ORDER BY (relevance + recency) may reorder results; fetching extra ensures top N by combined score
- Combined score: `ct.RANK * 0.7 + recency_score * 0.3` balances relevance and freshness

**What NOT to index:**
- Do NOT add a full-text index on CompanyName or Location — these are exact-match filters, not full-text search columns
- Do NOT add a B-tree index on Description — it is too large; FTS handles it

</details>
