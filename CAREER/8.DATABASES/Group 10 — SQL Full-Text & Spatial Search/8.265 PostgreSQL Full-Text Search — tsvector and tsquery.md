### Where This Fits

PostgreSQL Full-Text Search (FTS) provides built-in inverted index search using `tsvector` (the precomputed tokenized document) and `tsquery` (the search expression). Unlike SQL Server FTS, which hides the internal token representation, PostgreSQL exposes the full text processing pipeline — the developer can inspect the tsvector, configure the text search configuration, and use multiple dictionaries (Ispell, Snowball, Thesaurus). A .NET backend engineer encounters this when using PostgreSQL as the primary database and needing built-in search without adding Elasticsearch infrastructure. The interview signal is whether a candidate understands that PostgreSQL FTS is more powerful than SQL Server FTS (it supports multiple languages per document, configurable ranking, dictionary-based stemming, and phrase search with `tsquery` `<>` operator) but still lacks fuzzy search and distributed scaling — making it the middle ground between SQL Server FTS and Elasticsearch.

## Core Mental Model

PostgreSQL Full-Text Search operates on two exposed data types: `tsvector` (the document's text broken into lexemes sorted by position) and `tsquery` (the search query parsed into lexemes with operators). The invariant is: the `@@` match operator evaluates whether a `tsquery` matches a `tsvector`, enabling the GIN index to accelerate this check from O(N * word_count) to O(log V * match_count). The text processing pipeline is: raw text → parser (tokenization) → dictionary (stemming, stop words) → `tsvector` (sorted lexemes with positions). Unlike SQL Server FTS where the indexing is opaque, PostgreSQL allows `SELECT to_tsvector('english', 'Running foxes')` to see the actual lexemes (`'fox':2 'run':1`), and `SELECT to_tsquery('english', 'running & foxes')` to see the query tree (`'fox' & 'run'`). The recognition pattern for PostgreSQL FTS queries: `WHERE to_tsvector('english', col) @@ to_tsquery('english', 'word & phrase')` with a GIN index on the tsvector expression.

### Classification

- **Data types:** tsvector (text search vector), tsquery (query expression)
- **Operators:** `@@` (match), `||` (concatenate tsvector), `&&` (AND tsquery), `!!` (OR tsquery), `!!` (prefix tsquery), `<>` (followed by — phrase search)
- **Functions:** to_tsvector(config, text) → tsvector; to_tsquery(config, text) → tsquery; plainto_tsquery(config, text) → tsquery; phraseto_tsquery(config, text) → tsquery; ts_rank(tsvector, tsquery) → float4; ts_headline(config, text, tsquery) → text
- **Index:** GIN (Generalized Inverted Index) on tsvector column or expression; GiST (less common, slower for updates)
- **SARGable:** Yes — `to_tsvector(col) @@ to_tsquery('term')` can use GIN index seek; `col @@ to_tsquery('term')` on a tsvector column also uses GIN

## Deep Mechanics

### How PostgreSQL Executes a Full-Text Search Query

1. **Query parsing:** PostgreSQL parses `WHERE to_tsvector('english', p.description) @@ to_tsquery('english', 'running & foxes')`. The query tree is built with the `@@` operator node.

2. **to_tsquery evaluation:** First, `to_tsquery('english', 'running & foxes')` is evaluated. PostgreSQL's text search parser breaks the query into tokens: `running` and `foxes`. The Snowball stemmer converts them to lexemes: `run`, `fox`. The stop word filter removes any stop words. The result is a `tsquery` value: `'run' & 'fox'`.

3. **Index selection:** If a GIN index exists on `to_tsvector('english', p.description)`, the optimizer checks whether the expression matches the index expression. If it does (exact match of the expression), the optimizer generates a Bitmap Index Scan on the GIN index.

4. **GIN index scan:** The GIN index is an inverted index — for each lexeme, it stores a posting list of (row_id, positions). The index seeks to the lexeme `run`, retrieves its posting list, then seeks to `fox` and retrieves its posting list. The Bitmap Index Scan ORs these posting lists (for OR queries) or ANDs them (for AND queries).

5. **Bitmap Heap Scan:** The resulting bitmap of matching heap pages is used to fetch rows from the table. Each row that had both `run` and `fox` passes the `@@` filter. If the query uses phrase search (`<->` operator), positions within the posting list are also checked.

6. **Ranking (optional):** If `ts_rank` is used in ORDER BY, PostgreSQL computes the rank per row based on term frequency, document length normalization, and query term weight.

```sql
-- ============================================================
-- PostgreSQL Full-Text Search — complete examples
-- ============================================================

-- 1. See what to_tsvector produces:
SELECT to_tsvector('english', 'The quick brown foxes jump over the lazy dogs');
-- Result: 'brown':3 'dog':9 'fox':4 'jump':5 'lazi':8 'quick':2
-- Note: 'the' and 'over' are stop words (removed)
-- Note: 'foxes' → 'fox', 'lazy' → 'lazi', 'dogs' → 'dog' (stemming)

-- 2. See what to_tsquery produces:
SELECT to_tsquery('english', 'foxes & dogs');
-- Result: 'fox' & 'dog'

-- 3. Basic full-text search:
SELECT p.product_id, p.product_name
FROM products p
WHERE to_tsvector('english', p.description) @@ to_tsquery('english', 'ergonomic & keyboard');
-- Returns products where description contains both 'ergonomic' and 'keyboard'

-- 4. With ranking:
SELECT p.product_id, p.product_name,
       ts_rank(to_tsvector('english', p.description), to_tsquery('english', 'ergonomic & keyboard')) AS rank
FROM products p
WHERE to_tsvector('english', p.description) @@ to_tsquery('english', 'ergonomic & keyboard')
ORDER BY rank DESC;

-- 5. plainto_tsquery (simpler — no operators needed):
SELECT p.product_id, p.product_name
FROM products p
WHERE to_tsvector('english', p.description) @@ plainto_tsquery('english', 'ergonomic keyboard');
-- plainto_tsquery converts 'ergonomic keyboard' to 'ergonomic & keyboard'
-- Automatically ANDs terms

-- 6. phraseto_tsquery (phrase search):
SELECT p.product_id, p.product_name
FROM products p
WHERE to_tsvector('english', p.description) @@ phraseto_tsquery('english', 'ergonomic keyboard');
-- Uses <-> operator: 'ergonomic' <-> 'keyboard'
-- Both terms must appear and be adjacent in the correct order

-- 7. Phrase search with proximity:
SELECT p.product_id, p.product_name
FROM products p
WHERE to_tsvector('english', p.description) @@ to_tsquery('english', 'ergonomic <2> keyboard');
-- <2> means within 2 positions: 'ergonomic wireless keyboard' matches
-- 'ergonomic keyboard' (adjacent) also matches

-- 8. Prefix matching (autocomplete-like):
SELECT p.product_id, p.product_name
FROM products p
WHERE to_tsvector('english', p.description) @@ to_tsquery('english', 'ergo:*');
-- Matches any lexeme starting with 'ergo': ergonomic, ergo, ergonomic

-- 9. Multiple language support:
-- Store text in multiple languages and query accordingly:
SELECT p.product_id, p.product_name
FROM products p
WHERE to_tsvector('english', p.description_en) @@ to_tsquery('english', 'keyboard')
   OR to_tsvector('german', p.description_de) @@ to_tsquery('german', 'tastatur');

-- 10. Combined search with multiple fields:
SELECT p.product_id, p.product_name,
       ts_rank(to_tsvector('english', p.product_name || ' ' || p.description),
               to_tsquery('english', 'ergonomic & keyboard')) AS rank
FROM products p
WHERE to_tsvector('english', p.product_name || ' ' || p.description)
      @@ to_tsquery('english', 'ergonomic & keyboard')
ORDER BY rank DESC;

-- 11. Create GIN index on tsvector expression:
CREATE INDEX idx_products_fts ON products
    USING GIN (to_tsvector('english', description));

-- 12. Create GIN index on a stored tsvector column (for write optimization):
-- Add a tsvector column that updates on INSERT/UPDATE via trigger
ALTER TABLE products ADD COLUMN fts tsvector;

CREATE OR REPLACE FUNCTION products_fts_trigger()
RETURNS trigger AS $$
BEGIN
    NEW.fts := to_tsvector('english', NEW.product_name || ' ' || NEW.description);
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_products_fts
    BEFORE INSERT OR UPDATE ON products
    FOR EACH ROW EXECUTE FUNCTION products_fts_trigger();

CREATE INDEX idx_products_fts_column ON products USING GIN (fts);

-- Query using the stored column:
SELECT p.product_id, p.product_name
FROM products p
WHERE p.fts @@ to_tsquery('english', 'ergonomic & keyboard');
-- No function call in WHERE — the index is used directly
-- Faster than expression-based index because the tsvector is precomputed
```

```csharp
// ============================================================
// Dapper with PostgreSQL Full-Text Search
// ============================================================

public class PostgresFtsRepository
{
    private readonly IDbConnectionFactory _connectionFactory;

    public PostgresFtsRepository(IDbConnectionFactory connectionFactory)
        => _connectionFactory = connectionFactory;

    // Basic FTS with ranking
    public async Task<IReadOnlyList<ProductSearchResult>> SearchProductsAsync(
        string searchTerm,
        string language = "english",
        int maxResults = 50,
        CancellationToken ct = default)
    {
        const string sql = @"
            SELECT p.product_id AS ProductId,
                   p.product_name AS ProductName,
                   ts_rank(to_tsvector(@Language, p.description),
                           to_tsquery(@Language, @SearchTerm)) AS Rank
            FROM products p
            WHERE to_tsvector(@Language, p.description) @@ to_tsquery(@Language, @SearchTerm)
            ORDER BY Rank DESC
            LIMIT @MaxResults;";

        await using var connection = _connectionFactory.Create();
        var results = await connection.QueryAsync<ProductSearchResult>(
            new CommandDefinition(sql,
                new
                {
                    Language = language,
                    SearchTerm = searchTerm,
                    MaxResults = maxResults
                },
                cancellationToken: ct));
        return results.AsList();
    }

    // Phrase search with proximity

        int proximity = 1,

            .Split(' ', StringSplitOptions.RemoveEmptyEntries)

                   p.product_name AS ProductName,
                   ts_rank(to_tsvector(@Language, p.description),
                           to_tsquery(@Language, @TsQuery)) AS Rank
            FROM products p
            WHERE to_tsvector(@Language, p.description) @@ to_tsquery(@Language, @TsQuery)
            ORDER BY Rank DESC
            LIMIT @MaxResults;";

        await using var connection = _connectionFactory.Create();
        var results = await connection.QueryAsync<ProductSearchResult>(
            new CommandDefinition(sql,
                new
                {
                    Language = language,
                    TsQuery = tsquery,
                    MaxResults = maxResults
                },
                cancellationToken: ct));
        return results.AsList();
    }

    // Search across multiple fields with weighted ranking
    public async Task<IReadOnlyList<ProductSearchResult>> SearchMultiFieldAsync(
        string searchTerm,
        string language = "english",
        int maxResults = 50,
        CancellationToken ct = default)
    {
        const string sql = @"
            SELECT p.product_id AS ProductId,
                   p.product_name AS ProductName,
                   -- Weighted rank: title (A=1.0) + description (D=0.1)
                   (ts_rank(to_tsvector(@Language, p.product_name),
                            to_tsquery(@Language, @SearchTerm)) * 1.0
                    + ts_rank(to_tsvector(@Language, p.description),
                              to_tsquery(@Language, @SearchTerm)) * 0.1
                   ) AS Rank
            FROM products p
            WHERE to_tsvector(@Language, p.product_name || ' ' || p.description)
                  @@ to_tsquery(@Language, @SearchTerm)
            ORDER BY Rank DESC
            LIMIT @MaxResults;";

        await using var connection = _connectionFactory.Create();
        var results = await connection.QueryAsync<ProductSearchResult>(
            new CommandDefinition(sql,
                new
                {
                    Language = language,
                    SearchTerm = searchTerm,
                    MaxResults = maxResults
                },
                cancellationToken: ct));
        return results.AsList();
    }

    // Prefix search (autocomplete)
    public async Task<IReadOnlyList<string>> AutocompleteAsync(
        string prefix,
        string language = "english",
        int maxResults = 10,
        CancellationToken ct = default)
    {
        const string sql = @"
            SELECT DISTINCT ts_lexize(@Language, word) AS lexeme
            FROM (
                SELECT regexp_split_to_table(lower(description), E'\\W+') AS word
                FROM products
                WHERE description ILIKE @Prefix || '%'
            ) words
            WHERE word IS NOT NULL

                new

                    Language = language,
                    Prefix = prefix,
                    MaxResults = maxResults
                },
                cancellationToken: ct));
        return results.AsList();
    }

    // Check if GIN index exists
    public async Task<bool> GinIndexExistsAsync(CancellationToken ct = default)
    {
        const string sql = @"
            SELECT COUNT_BIG(*)
            FROM pg_indexes
            WHERE tablename = 'products'
              AND indexdef LIKE '%GIN%fts%';";

        await using var connection = _connectionFactory.Create();
        return await connection.QuerySingleAsync<int>(sql) > 0;
    }
}
```

### Execution Plan Analysis

**PostgreSQL FTS with GIN index (optimal):**

```
Bitmap Heap Scan on products p
  Recheck Cond: (to_tsvector('english'::regconfig, description) @@ '''ergonomic'' & ''keyboard'''::tsquery)
  → Bitmap Index Scan on idx_products_fts
     Index Cond: (to_tsvector('english'::regconfig, description) @@ '''ergonomic'' & ''keyboard'''::tsquery)
```

- `Bitmap Index Scan` on the GIN index — this reads the GIN posting lists for the lexemes.
- `Bitmap Heap Scan` — fetches rows from the heap using the bitmap. The "Recheck Cond" re-evaluates the `@@` condition for each row (the bitmap is approximate for GIN if the index uses pending list entries).
- The plan shows the function call in the Index Condition — PostgreSQL evaluates `to_tsvector('english', description)` once per row in the index? No — the expression-based index stores the precomputed tsvector, so the index lookup uses the stored values. The `to_tsquery` is evaluated first (result is constant).

**PostgreSQL FTS without GIN index (sequential scan):**

```
Seq Scan on products p
  Filter: (to_tsvector('english'::regconfig, description) @@ '''ergonomic'' & ''keyboard'''::tsquery)
```

- Sequential scan — `to_tsvector('english', description)` is computed for EVERY row, then checked against the query.
- This is the same cost as SQL Server's LIKE `%term%` — O(N * avg_words).

**PostgreSQL FTS with stored tsvector column (better GIN performance):**

```
Bitmap Heap Scan on products p
  Recheck Cond: (fts @@ '''ergonomic'' & ''keyboard'''::tsquery)
  → Bitmap Index Scan on idx_products_fts_column
     Index Cond: (fts @@ '''ergonomic'' & ''keyboard'''::tsquery)
```

- The index is on the stored `fts` column, not on the expression. The GIN index stores the tsvector directly.
- The condition references the column directly (`fts @@ ...`), not the function expression.
- No `to_tsvector` call at all — the tsvector is precomputed.
- This is faster because the index lookup does not need to compute the function.

### Failure Modes

**Expression mismatch prevents index usage:** If the WHERE clause uses `to_tsvector('english', col)` but the GIN index is on `to_tsvector('english', col)` (identical), PostgreSQL matches them. But if the configuration differs (`'simple'` vs `'english'`), the index cannot be used. If a concatenation like `to_tsvector('english', col1 || ' ' || col2)` differs from the index expression, the index is not used.

**GIN index pending list overhead for frequent writes:** GIN indexes handle fast inserts via a pending list. When the pending list exceeds `gin_pending_list_limit` (default 4 MB), it is merged into the main index. During merge, write performance degrades. For high-write tables, use a stored tsvector column and schedule the merge.

**No stop word handling in plainto_tsquery:** `plainto_tsquery('english', 'to be or not to be')` returns an empty query because all words are stop words. The query matches nothing.

**Language configuration mismatch:** If the document is English but the query uses `'german'` configuration, stemming and stop words differ. `to_tsvector('german', 'keyboards')` produces `'keyboard'` (English stemmer not applied), so the query `to_tsquery('english', 'keyboard')` may not match.

**GIN index size explosion:** For tables with highly distinct text (e.g., UUIDs, unique codes), the GIN index can be very large because each distinct lexeme gets its own posting list.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Production PostgreSQL Full-Text Search setup
-- ============================================================

-- 1. Table with stored tsvector column (optimal for write-heavy workloads)
CREATE TABLE products (
    product_id SERIAL PRIMARY KEY,
    product_name TEXT NOT NULL,
    description TEXT NOT NULL,
    category TEXT,
    price NUMERIC(10,2),
    created_at TIMESTAMPTZ DEFAULT NOW(),
    modified_at TIMESTAMPTZ DEFAULT NOW(),

    -- Stored tsvector column for FTS
    fts tsvector GENERATED ALWAYS AS (
        to_tsvector('english', product_name || ' ' || description)
    ) STORED
);

-- 2. GIN index on the stored tsvector column
CREATE INDEX idx_products_fts ON products USING GIN (fts);

-- 3. Search function with ranking and pagination
CREATE OR REPLACE FUNCTION search_products(
    search_term TEXT,
    language TEXT DEFAULT 'english',
    page_number INT DEFAULT 1,
    page_size INT DEFAULT 20
)
RETURNS TABLE(
    product_id INT,
    product_name TEXT,
    price NUMERIC(10,2),
    rank FLOAT4,
    total_count BIGINT
)

    tsq tsquery;
    total BIGINT;
BEGIN
    tsq := to_tsquery(language, search_term);

    -- Total matching count
    SELECT COUNT(*) INTO total
    FROM products p
    WHERE p.fts @@ tsq;

    -- Ranked, paginated results
    RETURN QUERY
    SELECT p.product_id, p.product_name, p.price,
           ts_rank(p.fts, tsq) AS rank,
           total AS total_count
    FROM products p
    WHERE p.fts @@ tsq
    ORDER BY rank DESC
    OFFSET (page_number - 1) * page_size
    LIMIT page_size;
END;
$$;
```

### SQL Server vs PostgreSQL Differences

```sql
-- ============================================================
-- SQL Server FTS vs PostgreSQL FTS — syntax comparison
-- ============================================================

-- SQL Server: CONTAINS
SELECT ProductId FROM Products WHERE CONTAINS(Description, 'ergonomic');
-- PostgreSQL: @@
SELECT product_id FROM products WHERE to_tsvector('english', description) @@ to_tsquery('english', 'ergonomic');

-- SQL Server: CONTAINS with AND
SELECT ProductId FROM Products WHERE CONTAINS(Description, '"ergonomic" AND "keyboard"');
-- PostgreSQL: @@ with &
SELECT product_id FROM products WHERE to_tsvector('english', description) @@ to_tsquery('english', 'ergonomic & keyboard');

-- SQL Server: CONTAINS with OR
SELECT ProductId FROM Products WHERE CONTAINS(Description, '"ergonomic" OR "keyboard"');
-- PostgreSQL: @@ with |
SELECT product_id FROM products WHERE to_tsvector('english', description) @@ to_tsquery('english', 'ergonomic | keyboard');

-- SQL Server: CONTAINSTABLE with RANK
SELECT p.ProductId, kt.RANK FROM Products p
INNER JOIN CONTAINSTABLE(Products, Description, 'ergonomic', 100) kt ON p.ProductId = kt.[KEY];
-- PostgreSQL: ts_rank
SELECT product_id, ts_rank(to_tsvector('english', description), to_tsquery('english', 'ergonomic')) AS rank
FROM products WHERE to_tsvector('english', description) @@ to_tsquery('english', 'ergonomic')
ORDER BY rank DESC LIMIT 100;

-- SQL Server: FREETEXT (loose meaning match)
SELECT ProductId FROM Products WHERE FREETEXT(Description, 'comfortable typing');
-- PostgreSQL: plainto_tsquery (ANDs all non-stop words)
SELECT product_id FROM products WHERE to_tsvector('english', description) @@ plainto_tsquery('english', 'comfortable typing');

-- SQL Server: No explicit phrase proximity operator
-- PostgreSQL: <-> operator for phrase proximity
SELECT product_id FROM products WHERE to_tsvector('english', description) @@ to_tsquery('english', 'ergonomic <-> keyboard');
-- <-> means adjacent; <2> means within 2 words

-- SQL Server: Full-text index on table
CREATE FULLTEXT INDEX ON Products (Description LANGUAGE 1033) KEY INDEX PK_Products;
-- PostgreSQL: GIN index on expression or column
CREATE INDEX idx_products_fts ON products USING GIN (to_tsvector('english', description));

-- SQL Server: No tsvector visibility — internal only
-- PostgreSQL: tsvector is visible and inspectable
SELECT to_tsvector('english', 'Running foxes');  -- 'fox':2 'run':1

-- Key differences summary:
-- 1. PostgreSQL exposes the tokenization (tsvector) — SQL Server hides it
-- 2. PostgreSQL has phrase search (<->) — SQL Server has NEAR but not fine-grained proximity
-- 3. PostgreSQL GIN index is expression-based — SQL Server FTS requires separate crawl
-- 4. PostgreSQL supports multiple languages in a single query — SQL Server uses one language per column
-- 5. PostgreSQL FTS is ACID-compliant (same transaction) — SQL Server FTS has crawl lag
```

## Gotchas and Production Pitfalls

### Expression Mismatch Disables GIN Index

**Pitfall:** Creating a GIN index on `to_tsvector('english', description)` but the query uses a slightly different expression — the index cannot be used.

```sql
-- ✅ Index created on:
CREATE INDEX idx_products_fts ON products USING GIN (to_tsvector('english', description));

-- ❌ Query uses coalesce — index NOT used:
SELECT * FROM products
WHERE to_tsvector('english', coalesce(description, '')) @@ to_tsquery('english', 'term');
-- Seq Scan! The expression "coalesce(description, '')" differs from "description"

-- ❌ Query uses different language — index NOT used:
SELECT * FROM products
WHERE to_tsvector('simple', description) @@ to_tsquery('simple', 'term');
-- Different text search configuration!

-- ✅ Query matches the index expression exactly:
SELECT * FROM products
WHERE to_tsvector('english', description) @@ to_tsquery('english', 'term');
```

**Symptom:** Despite having a GIN index, `EXPLAIN (ANALYZE)` shows a `Seq Scan`. The index expression must exactly match the query expression (including the text search configuration).

**Fix:** Use a stored `tsvector` column (GENERATED AS ... STORED) and query the column directly. The GIN index on the stored column is used regardless of query expression variations.

```sql
ALTER TABLE products ADD COLUMN fts tsvector
    GENERATED ALWAYS AS (to_tsvector('english', coalesce(description, ''))) STORED;

CREATE INDEX idx_products_fts ON products USING GIN (fts);

-- Query uses the stored column — always matches the index:
SELECT * FROM products WHERE fts @@ to_tsquery('english', 'term');
```

**Cost of not fixing:** GIN index exists but is never used. All FTS queries do sequential scans. On a 1M row table, each query takes 3-4 seconds instead of 5-8ms.

---

### GIN Pending List Causes Write Slowdown Under High INSERT Rates

**Pitfall:** High-frequency INSERTs on a table with a GIN index cause the pending list (fast-insert buffer) to grow beyond `gin_pending_list_limit` (default 4 MB). When the limit is reached, PostgreSQL must merge the pending list into the main index — a blocking operation that slows all concurrent writes and can cause deadlocks.

```sql
-- Check current pending list size:
SELECT gin_pending_list_limit();  -- Default: 4096 (4 MB)

-- Check pending list usage for a GIN index:
SELECT
    (pending_pages * current_setting('block_size')::int / 1024) AS pending_kb
FROM gin_metapage_info('idx_products_fts');
```

**Symptom:** Under high INSERT load (e.g., 100 INSERTs/second), writes to the table with the GIN index periodically stall for 100-500ms. The application sees occasional write timeouts.

**Fix:** Increase `gin_pending_list_limit` for larger batches, or use a stored tsvector column with a separate sync schedule.

```sql
-- Option 1: Increase pending list limit (per session or globally)
SET gin_pending_list_limit = 16384;  -- 16 MB

-- Option 2: Manual cleanup
-- Schedule during low-write periods:
SELECT gin_clean_pending_list('idx_products_fts');

-- Option 3: Use stored tsvector with trigger and batch insert
-- (avoid frequent single-row INSERTs — batch them)
```

**Cost of not fixing:** Write performance degrades unpredictably under load. The GIN index merge causes periodic latency spikes, visible in application P99 write latency.

---

### Stop Words in plainto_tsquery Return Empty Results

**Pitfall:** Using `plainto_tsquery('english', 'to be or not to be')` — all words are English stop words (to, be, or, not). The function returns an empty tsquery, which matches zero rows.

```sql
-- ❌ Returns no results:
SELECT * FROM products
WHERE to_tsvector('english', description) @@ plainto_tsquery('english', 'to be or not to be');
-- plainto_tsquery returns empty tsquery because all words are stop words

-- ✅ Fix: Use to_tsquery with explicit operators or change the query
SELECT * FROM products
WHERE to_tsvector('english', description) @@ to_tsquery('english', 'to <-> be <-> or <-> not <-> to <-> be');
-- Now each word is treated as a lexeme search (but 'to', 'be', 'or', 'not' may still be stop words)

-- ✅ Better: Use phrase search if the exact phrase matters
SELECT * FROM products
WHERE to_tsvector('english', description) @@ phraseto_tsquery('english', 'to be or not to be');
-- phraseto_tsquery still removes stop words for individual lexemes
```

**Symptom:** Searching for common phrases (movie quotes, idioms) returns zero results. Users think the search is broken.

**Fix:** Use `websearch_to_tsquery()` (PostgreSQL 11+) which handles common words better:

```sql
-- websearch_to_tsquery preserves quoted phrases:
SELECT websearch_to_tsquery('english', '"to be or not to be"');
```

**Cost of not fixing:** Users cannot search using common phrases. Search appears broken for a significant subset of queries.

---

### GIN Index Does Not Support Prefix Search with Wildcard

**Pitfall:** Using `LIKE 'term%'` on a text column alongside FTS. The GIN index supports prefix matching on tsquery with `:*` suffix, but only for lexemes — not raw text. Raw text prefix searches (LIKE) still need a separate trigram index.

```sql
-- ❌ GIN index does NOT accelerate this:
SELECT * FROM products WHERE product_name LIKE 'ergo%';
-- Seq Scan — GIN on tsvector is for lexemes, not raw text

-- ✅ GIN index DOES accelerate this (prefix search on lexemes):
SELECT * FROM products
WHERE to_tsvector('english', description) @@ to_tsquery('english', 'ergo:*');
-- Matches 'ergonomic', 'ergo', 'ergonomics' (any lexeme starting with 'ergo')

-- ✅ For raw text prefix, use a trigram index:
CREATE INDEX idx_products_name_trgm ON products USING GIN (product_name gin_trgm_ops);
SELECT * FROM products WHERE product_name LIKE 'ergo%';  -- Uses trigram index
```

**Symptom:** Raw text prefix searches (autocomplete, "starts with") do not use the FTS index. The developer assumes the GIN index covers all text search patterns.

**Fix:** Use `:*` prefix in tsquery for lexeme prefix matching, or add a trigram GIN index for raw text prefix/LIKE matching.

**Cost of not fixing:** Autocomplete queries run as sequential scans. On 1M rows, a `LIKE 'prefix%'` is a sequential scan of all rows (unless trigram index exists).

---

### setweight() Does Not Persist in Stored tsvector

**Pitfall:** Using `setweight()` in a GENERATED tsvector column — the weight is evaluated at computation time but not stored as separate metadata. `setweight` is ignored when the tsvector is stored in a column and later used in `ts_rank`.

```sql
-- ❌ setweight in GENERATED column — weights are NOT persisted
ALTER TABLE products ADD COLUMN fts tsvector
    GENERATED ALWAYS AS (
        setweight(to_tsvector('english', product_name), 'A') ||
        setweight(to_tsvector('english', description), 'D')
    ) STORED;
-- The weights 'A' and 'D' are applied at GENERATION time but
-- the tsvector column stores the combined result WITHOUT preserving
-- which lexemes came from which weight set!

-- ✅ Correct: compute weights at query time
SELECT p.product_id,
       ts_rank(
           setweight(to_tsvector('english', p.product_name), 'A') ||
           setweight(to_tsvector('english', p.description), 'D'),
           to_tsquery('english', 'term')
       ) AS rank
FROM products p
WHERE (
    to_tsvector('english', p.product_name) ||
    to_tsvector('english', p.description)
) @@ to_tsquery('english', 'term');
```

**Symptom:** Weighted search (title weight > description weight) does not work as expected. The stored tsvector has all lexemes with the same weight, so `ts_rank` applies equal weighting.

**Fix:** Do not use `setweight` in stored columns. Compute weights at query time in the `ts_rank` call, or use separate tsvector columns per field.

```sql
-- Alternative: Two stored tsvector columns
ALTER TABLE products ADD COLUMN fts_title tsvector
    GENERATED ALWAYS AS (to_tsvector('english', product_name)) STORED;
ALTER TABLE products ADD COLUMN fts_body tsvector
    GENERATED ALWAYS AS (to_tsvector('english', description)) STORED;

-- Then weight at query time:
SELECT ts_rank(
    setweight(p.fts_title, 'A') || setweight(p.fts_body, 'D'),
    to_tsquery('english', 'term')
) AS rank
FROM products p
WHERE p.fts_title || p.fts_body @@ to_tsquery('english', 'term');
```

**Cost of not fixing:** Weighted search relevance is wrong. Title matches should rank higher than body matches but the stored tsvector loses this distinction.

### Spoken Answers

**Q: What is tsvector and tsquery in PostgreSQL — how are they different from SQL Server's full-text search types?**

> **Average answer:** tsvector is the searchable text format and tsquery is the query format. SQL Server uses CONTAINS and FREETEXT.

> **Great answer:** tsvector is a PostgreSQL data type that represents a preprocessed document — a sorted list of lexemes (stemmed, stop-word-removed tokens) with positional information for each occurrence. For example, `to_tsvector('english', 'Running foxes')` produces `'fox':2 'run':1` — the lexeme 'fox' at position 2, 'run' at position 1. tsquery is the search expression data type, also broken into lexemes with boolean operators: `to_tsquery('english', 'running & foxes')` produces `'fox' & 'run'`. The key difference from SQL Server FTS is that PostgreSQL exposes these types — the developer can inspect the tsvector, debug why a match fails, configure per-language dictionaries, and use positional operators like `<->` for phrase search. SQL Server FTS hides all of this — you pass text to CONTAINS and get results, but you cannot see how the text was tokenized or what lexemes were generated. This transparency makes PostgreSQL FTS more debuggable and more flexible. Additionally, PostgreSQL's GIN index on a tsvector expression is an expression-based index that updates within the same transaction as the data change — there is no crawl lag like SQL Server's full-text index population. The visibility tradeoff is that the developer must manage the tsvector column (or stored generated column) and ensure the index expression matches the query expression.

8. At what row count and write frequency does GIN index maintenance become a performance concern?

10. Explain how you would design FTS for a PostgreSQL table with 20M product descriptions and 500 INSERTs/second, in 60 seconds.

<details>
<summary>Answers</summary>

1. `tsvector` (document — precomputed sorted lexemes with positions) and `tsquery` (search query — lexemes with boolean operators). Unlike SQL Server FTS which hides these types, PostgreSQL exposes them directly.

2. The `@@` operator checks if a tsquery matches a tsvector. With a GIN index, PostgreSQL performs a Bitmap Index Scan: the GIN index contains posting lists for each lexeme. The index looks up each query term's lexeme in the posting list, combines them (AND = intersection, OR = union), and produces a bitmap of matching heap page locations. A Bitmap Heap Scan then fetches the rows.

3. `EXPLAIN (ANALYZE, BUFFERS)` shows the access method. Without GIN: `Seq Scan`. With GIN: `Bitmap Index Scan` followed by `Bitmap Heap Scan`. The `Buffers` output shows buffer reads (e.g., 42,000 for seq scan vs 18 for GIN index scan).

4. The GIN index expression must exactly match the WHERE clause expression. Common mismatches: different text search configuration (`'english'` vs `'simple'`), different column reference (`col` vs `coalesce(col, '')`), or concatenation differences (`col1 || col2` vs `col1 || ' ' || col2`).

5. Yes. `EF.Functions.ToTsVector("english", p.Description).Matches(EF.Functions.ToTsQuery("english", "term"))` generates `to_tsvector('english', p.description) @@ to_tsquery('english', 'term')` which can use a GIN index with the matching expression.

6. Use `ts_rank(p.fts, to_tsquery('english', @SearchTerm)) AS rank` in the SELECT and `p.fts @@ to_tsquery('english', @SearchTerm)` in the WHERE. Use `LIMIT @MaxResults` for pagination. Map results with Dapper's `QueryAsync<T>`.

7. `ts_rank` considers term frequency in the document (TF-IDF based). `ts_rank_cd` uses coverage density — it considers how much of the document is covered by the query terms. `ts_rank_cd` is better for long documents where term frequency may be diluted; `ts_rank` is better for shorter fields like titles.

8. GIN index maintenance becomes a concern above ~100 INSERTs/second where the pending list (default 4 MB limit) fills frequently, triggering blocking merges. Above 10M rows, GIN index rebuilds take minutes. Use stored tsvector columns and batch inserts to reduce the impact.

9. A GIN (Generalized Inverted Index) supports the @@ operator. Unlike a B-tree (maps key → rows), a GIN stores a posting list per lexeme (lexeme → rows + positions). This enables the inverse lookup: "which rows contain this word?" where B-trees would need a full scan.

10. "Use a stored generated tsvector column: `ALTER TABLE products ADD COLUMN fts tsvector GENERATED ALWAYS AS (to_tsvector('english', coalesce(product_name, '') || ' ' || coalesce(description, ''))) STORED`. Create a GIN index on it. Set `gin_pending_list_limit` to 16 MB for the high write rate. Batch INSERTs into chunks of 100-500 rows to reduce pending list merges. For search, use `fts @@ to_tsquery('english', @term)` with `ts_rank` for ordering. For weighted search (title > description), use `setweight(to_tsvector('english', product_name), 'A') || setweight(to_tsvector('english', description), 'D')` in the query. This gives <10ms search on 20M rows with consistent write performance at 500 INS/s."

**Challenge 2 — Fix the performance problem**

```sql
-- This query searches for products matching "ergonomic keyboard".
-- It runs in 3.5 seconds on a 500K row table.

SELECT p.product_id, p.product_name
FROM products p
WHERE p.description LIKE '%ergonomic%' AND p.description LIKE '%keyboard%';

-- Seq Scan on products  (cost=0.00..45000.00 rows=10 width=68)
```

```sql
-- 2. Create GIN index
CREATE INDEX idx_products_fts ON products USING GIN (fts);

-- 3. Rewrite the query
SELECT p.product_id, p.product_name
FROM products p
WHERE p.fts @@ to_tsquery('english', 'ergonomic & keyboard');
```

**After fix — buffers:** ~15 (GIN index bitmap scan) from 42,000. **Execution time:** ~5ms from 3,500ms.

**Challenge 3 — Explain the execution plan**

Given this query:

```sql
SELECT p.product_id, p.product_name,
       ts_rank(p.fts, to_tsquery('english', 'ergonomic')) AS rank
FROM products p
WHERE p.fts @@ to_tsquery('english', 'ergonomic')
ORDER BY rank DESC
LIMIT 20;
```

The execution plan shows:
- `Limit` (cost=25.00..30.00 rows=20 width=72)
- `→ Sort` (cost=25.00..27.00 rows=800 width=72) Sort Key: (ts_rank(p.fts, '''ergonomic''::tsquery'))
- `→ Bitmap Heap Scan` (cost=12.00..20.00 rows=800 width=72) Recheck Cond: (fts @@ '''ergonomic''::tsquery')
- `→ Bitmap Index Scan` (cost=0.00..4.00 rows=800 width=0) Index Cond: (fts @@ '''ergonomic''::tsquery')

Why is there a Sort after the Bitmap Heap Scan? Could the GIN index provide ordered results by rank?

<details> <summary>Solution</summary>

**Why Sort exists:** The GIN index is an inverted index — it stores posting lists per lexeme. It can efficiently find which rows contain the term "ergonomic" (the Bitmap Index Scan produces the bitmap). But the index does not store rank information (ts_rank is computed from the tfvector positions and term frequency). The Bitmap Heap Scan fetches the rows and the tsvector from each row, then `ts_rank` computes the rank. The optimizer must then sort by this computed rank value to return the top 20.

**Could GIN index provide ordered results by rank?** No — GIN indexes do not support ordered output. They support existence checks (which rows contain the term?). Even if the index stored tf-idf information (which it does not), the query's ordering is by `ts_rank(fts, query)` which depends on the specific query's term frequency in each document — a runtime-computed value.

**Is there a way to avoid the sort?** Not for ranking by relevance score. If the query simply ordered by `product_id` or another indexed column, the sort could be avoided. But relevance ranking inherently requires computing the score for each matching row and sorting by it.

**Cost:** At 800 matching rows, the sort is fast (~1-2ms). The sort would be a concern at 100K+ matches (requires memory grant, possible temp spill). For large match sets, consider limiting the match count (`LIMIT 1000` before sort).

</details>

**Challenge 4 — Design the FTS index strategy**

**Scenario:** You have a PostgreSQL table with 50M product records. Each product has `product_name` (TEXT, ~50 words), `description` (TEXT, ~500 words), and `specifications` (JSONB). The search workload is:
- 200 searches/second (peak)
- 500 INSERTs/second (streaming product feed)
- Requirements: keyword search, phrase search, multi-language (English + German)
- Results must include relevance ranking with title weight = 3x description weight
- Write latency must stay under 20ms P99

Design the FTS index strategy. Show the table schema, indexes, and query pattern.

<details> <summary>Solution</summary>

```sql
-- 1. Table with stored tsvector columns per language
CREATE TABLE products (
    product_id SERIAL PRIMARY KEY,
    product_name TEXT NOT NULL,
    description TEXT NOT NULL,
    specifications JSONB,
    lang TEXT NOT NULL DEFAULT 'english',  -- 'english' or 'german'

    -- Separate tsvector columns per language (for GIN index matching)
    fts_english tsvector
        GENERATED ALWAYS AS (
            CASE WHEN lang = 'english'
            THEN to_tsvector('english',
                coalesce(product_name, '') || ' ' || coalesce(description, ''))
            ELSE NULL END
        ) STORED,

    fts_german tsvector
        GENERATED ALWAYS AS (
            CASE WHEN lang = 'german'
            THEN to_tsvector('german',
                coalesce(product_name, '') || ' ' || coalesce(description, ''))
            ELSE NULL END
        ) STORED,

    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 2. GIN indexes on each language column
CREATE INDEX idx_products_fts_english ON products USING GIN (fts_english);
CREATE INDEX idx_products_fts_german ON products USING GIN (fts_german);

-- 3. Increase GIN pending list for high write rate
SET gin_pending_list_limit = 32768;  -- 32 MB (for the session or in postgresql.conf)

-- 4. Query with language-specific routing and weighted ranking
-- Application passes the language from product/lang field
-- For English:
SELECT p.product_id, p.product_name,
       ts_rank(
           setweight(to_tsvector('english', coalesce(p.product_name, '')), 'A') ||
           setweight(to_tsvector('english', coalesce(p.description, '')), 'D'),
           to_tsquery('english', @SearchTerm)
       ) AS rank
FROM products p
WHERE p.lang = 'english'
  AND (
      to_tsvector('english', coalesce(p.product_name, '')) ||
      to_tsvector('english', coalesce(p.description, ''))
  ) @@ to_tsquery('english', @SearchTerm)
ORDER BY rank DESC
LIMIT 50;

-- 5. Combined search across both languages (fallback):
SELECT p.product_id, p.product_name,
       ts_rank(
           CASE p.lang
               WHEN 'english' THEN
                   setweight(to_tsvector('english', coalesce(p.product_name, '')), 'A') ||
                   setweight(to_tsvector('english', coalesce(p.description, '')), 'D')
               WHEN 'german' THEN
                   setweight(to_tsvector('german', coalesce(p.product_name, '')), 'A') ||
                   setweight(to_tsvector('german', coalesce(p.description, '')), 'D')
           END,
           CASE p.lang
               WHEN 'english' THEN to_tsquery('english', @SearchTerm)
               WHEN 'german' THEN to_tsquery('german', @SearchTerm)
           END
       ) AS rank
FROM products p
WHERE (
    (p.lang = 'english' AND p.fts_english @@ to_tsquery('english', @SearchTerm)) OR
    (p.lang = 'german' AND p.fts_german @@ to_tsquery('german', @SearchTerm))
)
ORDER BY rank DESC
LIMIT 50;
```

**Write cost:** GIN index inserts ~2x overhead. At 500 INS/s, the pending list fills in ~65 seconds (32 MB / (500 × ~1 KB per tsvector)). The merge takes ~50ms, happening every ~65 seconds — acceptable for 20ms P99.

**Read cost:** Bitmap Index Scan on the GIN index — ~15-20 buffers per query.

</details>

**Challenge 5 — Diagnose the FTS ranking bug**

**Scenario:** You deployed a product search feature using PostgreSQL FTS. The business team reports that search ranking is wrong — products with the search term in the title are not ranking higher than products with the term deep in the description. For example, searching "laptop" returns a product named "Premium Laptop Pro" at position 15, while a product named "Desk for Laptop Users" is at position 3. Both have relevant descriptions but the title match should clearly rank higher.

The query:

```sql
SELECT p.product_id, p.product_name,
       ts_rank(p.fts, to_tsquery('english', 'laptop')) AS rank
FROM products p
WHERE p.fts @@ to_tsquery('english', 'laptop')
ORDER BY rank DESC
LIMIT 50;
```

The `fts` column is defined as:

```sql
fts tsvector GENERATED ALWAYS AS (
    to_tsvector('english', coalesce(product_name, '') || ' ' || coalesce(description, ''))
) STORED;
```

<details> <summary>Solution</summary>

**Root cause:** The stored `fts` column combines `product_name` and `description` into a single tsvector WITHOUT using `setweight`. All lexemes have the same weight (default 'D'). `ts_rank` computes the score based on term frequency and document length — if "laptop" appears 5 times in a long description but only once in the title, the description match scores higher. The title match is not distinguished from body matches.

**Fix — Use setweight at query time:**

```sql
SELECT p.product_id, p.product_name,
       ts_rank(
           setweight(to_tsvector('english', coalesce(p.product_name, '')), 'A') ||
           setweight(to_tsvector('english', coalesce(p.description, '')), 'D'),
           to_tsquery('english', 'laptop')
       ) AS rank
FROM products p
WHERE (
    to_tsvector('english', coalesce(p.product_name, '')) ||
    to_tsvector('english', coalesce(p.description, ''))
) @@ to_tsquery('english', 'laptop')
ORDER BY rank DESC
LIMIT 50;
```

**Better fix — Maintain separate tsvector columns:**

```sql
ALTER TABLE products ADD COLUMN fts_title tsvector
    GENERATED ALWAYS AS (to_tsvector('english', coalesce(product_name, ''))) STORED;
ALTER TABLE products ADD COLUMN fts_body tsvector
    GENERATED ALWAYS AS (to_tsvector('english', coalesce(description, ''))) STORED;

CREATE INDEX idx_products_fts_title ON products USING GIN (fts_title);
CREATE INDEX idx_products_fts_body ON products USING GIN (fts_body);

-- Query with setweight at query time:
SELECT p.product_id, p.product_name,
       ts_rank(
           setweight(p.fts_title, 'A') || setweight(p.fts_body, 'D'),
           to_tsquery('english', 'laptop')
       ) AS rank
FROM products p
WHERE p.fts_title || p.fts_body @@ to_tsquery('english', 'laptop')
ORDER BY rank DESC
LIMIT 50;
```

**After fix:** "Premium Laptop Pro" (title match, weight A) ranks above "Desk for Laptop Users" (description only, weight D). The A-weight term receives ~3x the rank contribution of D-weight terms in the default ts_rank normalization.

</details>

---
