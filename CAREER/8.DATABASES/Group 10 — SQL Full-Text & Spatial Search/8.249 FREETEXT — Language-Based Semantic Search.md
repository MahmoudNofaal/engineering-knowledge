### Where This Fits

`FREETEXT` is a full-text predicate designed for natural language, user-facing search. Unlike CONTAINS (which requires explicit operators), FREETEXT takes a natural language phrase, breaks it into individual words, applies stemming and thesaurus expansion to each word, and returns rows where any of the derived words match — using implicit OR with internal relevance weighting. A .NET backend engineer uses FREETEXT for search bars where users type natural queries like "affordable wireless mouse for travel" — FREETEXT automatically extracts the meaningful terms (affordable, wireless, mouse, travel), stems them, and returns products matching any of them, ranked by relevance. The interview signal comes from understanding the tradeoff between the simplicity of FREETEXT and the precision of CONTAINS, and knowing when each is appropriate.

### Classification

FREETEXT is a **full-text predicate** that belongs to the same family as CONTAINS but with fundamentally different predicate interpretation. FREETEXT does not accept boolean operators, proximity terms, or explicit inflectional/thesaurus forms — instead, it applies these transformations automatically to every term in the search phrase. The classification is **semantic search**: FREETEXT aims to match the meaning of the search phrase, not the exact words. The predicate is evaluated by the full-text engine using the same inverted index as CONTAINS. The FTS engine word-breaks the search phrase, stems each term, expands through the thesaurus, filters stopwords, and then performs an implicit OR of all resulting terms. Results are internally ranked so that documents matching more terms or rarer terms appear first (even though FREETEXT itself does not return rank — use FREETEXTTABLE for ranking).

```mermaid

    Q -.-> S[CONTAINS result: narrower, no ranking]
```

## Deep Mechanics

### How the Engine Executes a FREETEXT Query

**Step-by-step for: `FREETEXT(ProductDescription, 'affordable wireless mouse for travel')`**

1. **Predicate identification** — The relational engine parser identifies `FREETEXT` as a full-text predicate. Unlike CONTAINS, the parser does not attempt to interpret the predicate content — it is treated as an opaque string that will be entirely processed by the FTS engine.

2. **Word breaking** — The FTS engine takes the search phrase 'affordable wireless mouse for travel' and passes it through the language-specific word breaker. The word breaker splits the phrase into individual tokens at word boundaries: `[affordable]`, `[wireless]`, `[mouse]`, `[for]`, `[travel]`. The word breaker handles punctuation, hyphens, and other non-alphabetic characters as token separators.

3. **Stopword filtering** — Each token is checked against the configured stoplist (default: system stoplist). The word `[for]` is in the English system stoplist and is removed. Remaining tokens: `[affordable, wireless, mouse, travel]`.

4. **Stemming** — Each remaining token is reduced to its root form by the language stemmer. For English (Porter stemmer): `affordable → afford`, `wireless → wireless` (no change — already root), `mouse → mouse` (no change), `travel → travel` (no change). NOTE: Not all words have inflectional variants — the stemmer only reduces words that have morphological variants.

5. **Thesaurus expansion** — Each stemmed root is looked up in the language-specific thesaurus XML file. For 'afford', the thesaurus may contain `<expansion><sub>afford</sub><sub>cheap</sub><sub>inexpensive</sub></expansion>`. For 'travel': `<expansion><sub>travel</sub><sub>trip</sub><sub>journey</sub></expansion>`. The expanded term set becomes: `{afford, cheap, inexpensive, wireless, mouse, travel, trip, journey}`.

6. **Inverted index lookups** — Each term in the expanded set is looked up in the inverted index. The engine retrieves the document key list for each term:
   - afford → [Doc_1, Doc_45, Doc_302]
   - cheap → [Doc_12, Doc_45, Doc_88]
   - inexpensive → [Doc_45, Doc_201]
   - wireless → [Doc_2, Doc_45, Doc_67, Doc_302]
   - mouse → [Doc_45, Doc_67, Doc_302]
   - travel → [Doc_45, Doc_302, Doc_401]
   - trip → [Doc_88, Doc_302]
   - journey → [Doc_45, Doc_401]

7. **Union with ranking** — The engine computes the union of all document lists. For each document in the union, it computes an internal rank value based on:
   - **Term Frequency (TF):** How many times each matched term appears in the document. Higher TF = higher rank.
   - **Inverse Document Frequency (IDF):** How rare the matched term is across all documents. Rarer terms contribute more to rank. "Affordable" appearing in only 3 documents contributes more than "wireless" appearing in 50.
   - **Document length normalization:** Longer documents get a slight penalty — a match in a short document is more relevant than a match in a long one.
   - **Proximity bonus (implicit):** Terms appearing closer together within the document get a small rank bonus.

8. **Document ordering** — The results are ordered by internal rank in descending order, but the rank value itself is NOT exposed to the query (use FREETEXTTABLE to see ranks). The ORDER BY is implied — FREETEXT results are returned in relevance order within the result set.

9. **Key lookup** — The relational engine receives the ordered list of document keys and performs key lookups into the base table for the requested columns.

**Key difference from CONTAINS:** With CONTAINS, you write `'"wireless" AND "mouse"'` and only documents containing both terms match. With FREETEXT, `'wireless mouse'` expands to `{wireless, mouse}` via word breaking and stemming, and any document containing EITHER term (or synonyms) matches. FREETEXT is more permissive but produces more results, many of which may be less relevant.

### SQL Visibility

```sql
-- ============================================================
-- FREETEXT — All Usage Patterns
-- ============================================================

-- Setup: assume full-text index on dbo.Products, dbo.Articles, dbo.Documents

-- 1. Basic FREETEXT — natural language search
SELECT ProductId, ProductName, UnitPrice
FROM dbo.Products
WHERE FREETEXT(ProductDescription, 'affordable wireless mouse for travel');
-- Internally expands to: afford, cheap, inexpensive, wireless, mouse, travel, trip, journey
-- Matches products containing ANY of these terms
-- Results ordered by internal relevance

-- 2. FREETEXT with multiple columns
SELECT ProductId, ProductName, UnitPrice
FROM dbo.Products
WHERE FREETEXT(
    (ProductName, ShortDescription, LongDescription),
    'affordable wireless mouse'
);

-- 3. FREETEXT on all indexed columns
SELECT ProductId, ProductName
FROM dbo.Products
WHERE FREETEXT(*, 'wireless mouse');

-- 4. FREETEXT with language parameter
SELECT ProductId, ProductName
FROM dbo.Products
WHERE FREETEXT(ProductDescription, N'förskott betalning', LANGUAGE 1033);
-- Specify language for word breaker selection
-- 1033 = English, 1031 = German, 1036 = French

-- 5. FREETEXT with additional WHERE filters
SELECT p.ProductId, p.ProductName, p.UnitPrice
FROM dbo.Products p
WHERE FREETEXT(p.ProductDescription, 'wireless mouse')
  AND p.IsActive = 1
  AND p.UnitPrice BETWEEN 10.00 AND 200.00
ORDER BY p.UnitPrice;

-- 6. FREETEXT for Article search (user-facing)
SELECT a.ArticleId, a.Title, a.PublishedDate, a.AuthorName
FROM dbo.Articles a
WHERE FREETEXT((a.Title, a.Body), 'database optimization techniques')
  AND a.IsPublished = 1
ORDER BY a.PublishedDate DESC;

-- 7. FREETEXT for Document search
SELECT d.DocumentId, d.DocumentName, d.UploadedDate
FROM dbo.Documents d
WHERE FREETEXT(d.DocumentContent, 'confidential agreement terms')
ORDER BY d.UploadedDate DESC;

-- 8. FREETEXT vs CONTAINS — compare results
-- FREETEXT: broad, any term matches
SELECT COUNT(*) AS FreetextMatchCount
FROM dbo.Products
WHERE FREETEXT(ProductDescription, 'wireless bluetooth mouse');
-- Returns: 1,247 (any product mentioning wireless, bluetooth, OR mouse)

-- CONTAINS: narrow, all terms must match
SELECT COUNT(*) AS ContainsMatchCount
FROM dbo.Products
WHERE CONTAINS(ProductDescription, '"wireless" AND "bluetooth" AND "mouse"');
-- Returns: 83 (only products mentioning ALL THREE terms)

-- 9. FREETEXT with table alias and schema
SELECT p.ProductId, p.ProductName
FROM dbo.Products p
WHERE FREETEXT(p.ProductDescription, @UserSearchInput);

-- 10. FREETEXT for "I'm feeling lucky" — top 1 result
SELECT TOP 1 ProductId, ProductName, UnitPrice
FROM dbo.Products
WHERE FREETEXT(ProductDescription, 'best wireless mouse under 50 dollars')
  AND IsActive = 1;
```

### Execution Plan Analysis

```sql
SELECT p.ProductId, p.ProductName, p.UnitPrice
FROM dbo.Products p
WHERE FREETEXT(p.ProductDescription, 'affordable wireless mouse')
  AND p.IsActive = 1;
```

**Expected plan shape:**
```
[FullTextMatch — FREETEXT(p.ProductDescription, 'affordable wireless mouse')]
    → [Filter: p.IsActive = 1]
    → [Clustered Index Seek — key lookup]
```

- `FullTextMatch` — Same operator as CONTAINS. The plan shows 0% cost because the FTS engine cost is opaque to the relational optimizer.
- `Filter` — Applied after key lookups. Without a covering index, every matched row requires a clustered index seek to retrieve `IsActive`.
- `Clustered Index Seek` — One seek per matched document. FREETEXT typically matches more documents than CONTAINS (because of implicit OR and thesaurus expansion), so this operator may have more executions.

**Cost comparison with CONTAINS:**
- FREETEXT 'affordable wireless mouse': ~1,200 matches → 1,200 key lookups
- CONTAINS '"affordable" AND "wireless" AND "mouse"': ~50 matches → 50 key lookups

FREETEXT produces more key lookups because the implicit OR returns a broader result set.

### Cost Visibility

```sql
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

-- ============================================================
-- FREETEXT — logical reads
-- ============================================================
SELECT p.ProductId, p.ProductName
FROM dbo.Products p
WHERE FREETEXT(p.ProductDescription, 'affordable wireless mouse')
  AND p.IsActive = 1;

-- Expected output (100K row table, 1,200 FREETEXT matches):
-- Table 'Products'. Scan count 1, logical reads 2402
--   (2 for clustered index scan root + leaf + 2400 for 1,200 key lookups × 2 pages each)
-- Table 'Products' (Full-text). Scan count 1, logical reads 0
--   (FTS engine reads — not captured)
-- SQL Server Execution Times:
--   CPU time = 65ms, elapsed time = 180ms

-- Compare with CONTAINS:
SELECT p.ProductId, p.ProductName
FROM dbo.Products p
WHERE CONTAINS(p.ProductDescription, '"affordable" AND "wireless" AND "mouse"')
  AND p.IsActive = 1;

-- Expected output (same table, 50 CONTAINS matches):
-- Table 'Products'. Scan count 1, logical reads 102
--   (2 + 100 for 50 key lookups)
-- SQL Server Execution Times:
--   CPU time = 18ms, elapsed time = 45ms
```

**Key insight:** FREETEXT produces ~24x more logical reads than CONTAINS in this example because the implicit OR matches many more documents. The logical read cost is proportional to the number of matches, not the complexity of the predicate.

### Failure Modes

**Failure Mode 1: Using CONTAINS operators in FREETEXT**

```sql
-- ❌ FREETEXT does not support boolean operators
SELECT ProductId FROM dbo.Products
WHERE FREETEXT(ProductDescription, '"wireless" AND "mouse"');
-- No syntax error! FREETEXT treats this as a phrase:
-- '"wireless" AND "mouse"' is word-broken into tokens:
-- [wireless, AND, mouse]
-- 'AND' is a stopword (removed).
-- Result: matches products containing wireless OR mouse.
-- The AND has no effect — FREETEXT uses implicit OR.
```

**Symptom:** The query runs without error but returns different results than expected. The developer used CONTAINS-like syntax but FREETEXT ignored the operators.

**Fix:** Use CONTAINS for boolean queries, or accept that FREETEXT always uses implicit OR.

**Cost of not fixing:** The search returns much broader results than expected. A search for "wireless AND mouse" returns all products with wireless OR mouse — potentially millions of rows. Users see irrelevant results.

**Failure Mode 2: FREETEXT with very long phrases**

```sql
-- ❌ FREETEXT with 50+ word query
SELECT ProductId FROM dbo.Products
WHERE FREETEXT(ProductDescription, 'wireless mouse bluetooth keyboard charger cable adapter stand cover screen protector laptop bag backpack sleeve ...');
-- Each word is stemmed and expanded through the thesaurus.
-- With 50 terms + synonym expansion, the expanded term set may have
-- 100-200 terms. Each term requires an inverted index lookup.
-- The union of all document lists may include nearly every row.
```

**Symptom:** Very slow query returning most rows in the table.

**Fix:** Limit search phrase length at the application level to ~10 words.

**Cost of not fixing:** Search performance degrades linearly with phrase length. A 50-word FREETEXT query may take 5-10 seconds on a large table.

**Failure Mode 3: FREETEXT on columns with different languages**

```sql
-- ❌ FREETEXT with wrong language parameter
SELECT ProductId FROM dbo.Products
WHERE FREETEXT(ProductDescription, N'guten Morgen', LANGUAGE 1033);
-- The English word breaker (1033) does not understand German word boundaries.
-- "guten Morgen" is correctly split by English word breaker (space-separated),
-- but stemming uses English rules which do not apply to German.
```

**Symptom:** Poor search quality for multi-language content.

**Fix:** Ensure LANGUAGE parameter matches the content language, or use separate columns per language with appropriate LANGUAGE LCID in the full-text index definition.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Product Search with FREETEXT
-- ============================================================

-- Schema: Products with full-text index (see 8.247)

-- Scenario 1: User-facing product search bar
-- User types: 'affordable wireless mouse for travel'
CREATE PROCEDURE dbo.SearchProducts
    @SearchPhrase   NVARCHAR(500),
    @MaxResults     INT = 50,
    @MinPrice       DECIMAL(18,2) = NULL,
    @MaxPrice       DECIMAL(18,2) = NULL,
    @CategoryName   NVARCHAR(100) = NULL
AS
BEGIN
    SET NOCOUNT ON;

    SELECT TOP (@MaxResults)
        p.ProductId,
        p.ProductName,
        p.ShortDescription,
        p.UnitPrice,
        p.BrandName,
        p.CategoryName
    FROM dbo.Products p
    WHERE FREETEXT((p.ProductName, p.ShortDescription, p.LongDescription,
                    p.CategoryName, p.BrandName, p.Tags), @SearchPhrase)
      AND p.IsActive = 1
      AND (@MinPrice IS NULL OR p.UnitPrice >= @MinPrice)
      AND (@MaxPrice IS NULL OR p.UnitPrice <= @MaxPrice)
      AND (@CategoryName IS NULL OR p.CategoryName = @CategoryName)
    ORDER BY p.UnitPrice;  -- Note: FREETEXT does not expose rank for ordering
    -- Use FREETEXTTABLE if rank-based ordering is needed
END;
GO

-- Scenario 2: Article search for CMS
CREATE PROCEDURE dbo.SearchArticles
    @SearchPhrase   NVARCHAR(1000),
    @MaxResults     INT = 20,
    @PublishedAfter DATE = NULL,
    @AuthorName     NVARCHAR(200) = NULL
AS
BEGIN
    SET NOCOUNT ON;

    SELECT TOP (@MaxResults)
        a.ArticleId,
        a.Title,
        a.Excerpt,
        a.PublishedDate,
        a.AuthorName
    FROM dbo.Articles a
    WHERE FREETEXT((a.Title, a.Body, a.Tags), @SearchPhrase)
      AND a.IsPublished = 1
      AND (@PublishedAfter IS NULL OR a.PublishedDate >= @PublishedAfter)
      AND (@AuthorName IS NULL OR a.AuthorName = @AuthorName)
    ORDER BY a.PublishedDate DESC;
END;
GO

-- Scenario 3: Document search
CREATE PROCEDURE dbo.SearchDocuments
    @SearchPhrase   NVARCHAR(500),
    @Department     NVARCHAR(100) = NULL,
    @MaxResults     INT = 50
AS
BEGIN
    SET NOCOUNT ON;

    SELECT TOP (@MaxResults)
        d.DocumentId,
        d.DocumentName,
        d.ContentType,
        d.UploadedDate,
        d.UploadedBy
    FROM dbo.Documents d
    WHERE FREETEXT((d.DocumentName, d.DocumentContent), @SearchPhrase)
      AND (@Department IS NULL OR d.Department = @Department)
    ORDER BY d.UploadedDate DESC;
END;
GO

-- ============================================================
-- FREETEXT with Ranking via FREETEXTTABLE
-- ============================================================

-- When ordering by relevance is needed, use FREETEXTTABLE:
SELECT p.ProductId, p.ProductName, p.UnitPrice, ft.RANK
FROM dbo.Products p
INNER JOIN FREETEXTTABLE(Products,
    (ProductName, ShortDescription, LongDescription),
    'affordable wireless mouse',
    20) ft ON p.ProductId = ft.[KEY]
WHERE p.IsActive = 1
ORDER BY ft.RANK DESC;

-- ============================================================
-- Monitoring FREETEXT Performance
-- ============================================================

-- Check which queries use FREETEXT
SELECT
    qs.execution_count,
    qs.total_worker_time / 1000 AS TotalCpuMs,
    qs.total_elapsed_time / 1000 AS TotalElapsedMs,
    qs.total_logical_reads,
    SUBSTRING(st.text, (qs.statement_start_offset / 2) + 1,
        ((CASE WHEN qs.statement_end_offset = -1
               THEN DATALENGTH(st.text)
               ELSE qs.statement_end_offset
          END - qs.statement_start_offset) / 2) + 1) AS QueryText
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
WHERE st.text LIKE '%FREETEXT%'
ORDER BY qs.total_worker_time DESC;
```

```csharp
public class FreetextSearchService
```

```csharp
// Dapper — FREETEXT Search

public class DapperFreetextSearchService
{
    private readonly IDbConnectionFactory _connectionFactory;

    public DapperFreetextSearchService(IDbConnectionFactory connectionFactory)
    {
        _connectionFactory = connectionFactory;
    }

            )
            AND p.IsActive = 1");

            sql.Append(" AND p.UnitPrice <= @MaxPrice");

        sql.Append(" ORDER BY p.ProductName");

        await using var connection = _connectionFactory.CreateConnection();
        var results = await connection.QueryAsync<ProductSearchResult>(
            new CommandDefinition(
                sql.ToString(),

        int topN = 20,
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            SELECT p.ProductId, p.ProductName, p.ShortDescription,
                   p.UnitPrice, p.CategoryName, ft.RANK
            FROM dbo.Products p
            INNER JOIN FREETEXTTABLE(Products,
                (ProductName, ShortDescription, LongDescription,

                @SearchPhrase,
                @TopN) ft

            WHERE p.IsActive = 1
            ORDER BY ft.RANK DESC";

        await using var connection = _connectionFactory.CreateConnection();

            new CommandDefinition(
                sql,
                new { SearchPhrase = searchPhrase, TopN = topN },
                cancellationToken: cancellationToken));

        return results.AsList();
    }
}

public record RankedSearchResult : ProductSearchResult
{
    public int RANK { get; init; }
}
```

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL has no direct FREETEXT equivalent.
-- The closest is to_tsquery with plainto_tsquery (parses phrase into terms with &):

-- PostgreSQL: plainto_tsquery — similar to FREETEXT
-- It parses the phrase, stems each term, and joins with AND (not OR)
SELECT product_id, product_name
FROM products
WHERE search_vector @@ plainto_tsquery('english', 'affordable wireless mouse');
-- Equivalent to: to_tsquery('english', 'affordable & wireless & mouse')

-- PostgreSQL: to_tsquery with OR operator — closer to FREETEXT
SELECT product_id, product_name
FROM products
```

## Gotchas and Production Pitfalls

### 1. FREETEXT Does Not Expose Rank — Cannot ORDER BY Relevance

**Pitfall:** Using FREETEXT in a WHERE clause does not return a rank value. `ORDER BY` on relevance is not possible without using FREETEXTTABLE or FREETEXT in a subquery.

```sql
-- ❌ Cannot order by relevance with FREETEXT alone
SELECT ProductId, ProductName
FROM dbo.Products
WHERE FREETEXT(ProductDescription, 'wireless mouse')
ORDER BY ???  -- No rank column available
```

**Symptom:** Results are returned in an arbitrary order (typically by clustered index key) — not by relevance. Users see less relevant results first.

**Fix:** Use FREETEXTTABLE instead:

```sql
-- ✅ Use FREETEXTTABLE for ranking
SELECT p.ProductId, p.ProductName, ft.RANK
FROM dbo.Products p
INNER JOIN FREETEXTTABLE(Products, ProductDescription, 'wireless mouse', 50) ft
    ON p.ProductId = ft.[KEY]
WHERE p.IsActive = 1
ORDER BY ft.RANK DESC;
```

**Cost of not fixing:** Poor user experience. Users see random results instead of the most relevant ones. This is the most common complaint about FREETEXT-based search.

### 2. Implicit OR Returns Too Many Results

**Pitfall:** FREETEXT with multiple terms matches documents that contain ANY of the stemmed/expanded terms. A 5-term FREETEXT query can match 80%+ of the table.

```sql
-- ❌ Very broad match
SELECT COUNT(*) FROM dbo.Products
WHERE FREETEXT(ProductDescription, 'wireless bluetooth mouse keyboard');
-- Returns 45,000 out of 100,000 rows
-- Most products contain at least one of these common words
```

**Symptom:** Search returns thousands of results even for specific-sounding queries. Users cannot find what they need.

**Fix:** Use CONTAINS with AND for precision, or combine FREETEXT with additional WHERE filters (category, price range, date range):

```sql
-- ✅ Add filters to narrow results
SELECT p.ProductId, p.ProductName, p.UnitPrice
FROM dbo.Products p
WHERE FREETEXT(p.ProductDescription, 'wireless bluetooth mouse keyboard')
  AND p.IsActive = 1
  AND p.CategoryName = 'Computer Peripherals'
  AND p.UnitPrice BETWEEN 20.00 AND 150.00;
```

**Cost of not fixing:** Heavy IO from thousands of key lookups. Slow queries. Poor user experience due to irrelevant results.

### 3. Stopwords Strip Meaning from Short Queries

**Pitfall:** Very short FREETEXT queries (1-3 words) can be reduced to nothing by stopword filtering and stemming.

```sql
-- ❌ FREETEXT query stripped to nothing
SELECT ArticleId FROM dbo.Articles
WHERE FREETEXT(Body, 'the and or');
-- Stopwords: [the, and, or] → all removed → empty term set
-- Returns: ALL rows? Or NO rows?
-- Answer: NO rows. An empty term set matches nothing.
```

**Symptom:** A search for "the and or" returns zero results. This looks like a bug but is correct behavior — all terms were stopwords.

**Fix:** At the application level, check if the query after stopword removal is empty:

```csharp
var stopwords = new HashSet<string>(StringComparer.OrdinalIgnoreCase)
{
    "the", "a", "an", "and", "or", "in", "on", "at", "to", "for",
    "of", "by", "with", "is", "it", "as", "be", "but", "not"
};

var terms = searchPhrase.Split(' ', StringSplitOptions.RemoveEmptyEntries)
    .Where(t => !stopwords.Contains(t))
    .ToArray();

if (terms.Length == 0)
    return Array.Empty<SearchResult>();  // Or return recent items
```

**Cost of not fixing:** Search returns zero results for queries that seem valid to users. Support tickets: "I searched for 'to be or not to be' and got nothing."

### 4. Thesaurus Expansion Can Match Unexpected Synonyms

**Pitfall:** The thesaurus may contain synonyms that match unintended documents. For example, the English thesaurus shipped with SQL Server has expansions that may not fit your domain.

```sql
-- ❌ Thesaurus expands 'fast' to unexpected synonyms
SELECT ProductId FROM dbo.Products
WHERE FREETEXT(ProductDescription, 'fast');
-- The thesaurus for 'fast' may expand to:
--   fast, quick, rapid, speedy, swift, fleet, fixed
-- If 'fixed' is a thesaurus entry for 'fast', products with
-- 'fixed' in the description also match — even if 'fixed' means
-- "not broken" in the product context.
```

**Symptom:** Search returns documents that contain synonyms unrelated to the original intent. "Fast" search returns products with "fixed price" because the thesaurus considers "fast" and "fixed" as related.

**Fix:** Either customize the thesaurus XML file (located in the SQL Server FTDATA directory) to remove inappropriate synonyms, or use CONTAINS with explicit `FORMSOF(THESAURUS, word)` to control which terms are expanded.

```sql
-- ✅ CONTAINS with explicit thesaurus control
SELECT ProductId FROM dbo.Products
WHERE CONTAINS(ProductDescription,
    'FORMSOF(THESAURUS, fast)');
-- Still uses thesaurus but you know exactly which terms are being expanded
```

**Cost of not fixing:** Irrelevant search results that confuse users and erode trust in search. Debugging these issues is difficult because the thesaurus expansion is invisible to the user.

### 5. Performance Collapse with FREETEXT on Large Text Columns

**Pitfall:** Using FREETEXT on an NVARCHAR(MAX) column that stores very large text (50KB+ per row) with a long search phrase.

```sql
-- ❌ FREETEXT on large text column with long phrase
SELECT DocumentId FROM dbo.Documents
WHERE FREETEXT(DocumentContent, 'confidential agreement non-disclosure employee termination severance package benefits');
-- DocumentContent: average 50KB per row
-- 10 terms → stemmed → thesaurus expanded → ~30 lookup terms
-- Each of the 30 lookups scans the inverted index for that term
-- The union can match most documents
```

**Symptom:** Slow query, high FTS engine CPU, many key lookups.

**Fix:** Limit the search phrase at the application level. Use CONTAINSTABLE with `topN` for better performance.

**Cost of not fixing:** Search queries take 5-10 seconds on large document collections. IO and CPU saturation from the FTS engine.

## Performance Implications

### Benchmark: FREETEXT vs CONTAINS vs LIKE

```sql
-- ============================================================
-- Performance Comparison
-- ============================================================
-- 100K row Products table, 2KB average ProductDescription

-- LIKE '%wireless%' — full scan
SET STATISTICS IO ON;
SELECT COUNT(*) FROM dbo.Products
WHERE ProductDescription LIKE '%wireless%';
-- Logical reads: 45,832
-- CPU: 312ms, elapsed: 480ms

-- CONTAINS — single term
SELECT COUNT(*) FROM dbo.Products
WHERE CONTAINS(ProductDescription, '"wireless"');
-- Logical reads: 4
-- CPU: 5ms, elapsed: 12ms

-- FREETEXT — single word
SELECT COUNT(*) FROM dbo.Products
WHERE FREETEXT(ProductDescription, 'wireless');
-- Logical reads: 4
-- CPU: 6ms, elapsed: 14ms

-- FREETEXT — multi-word (implicit OR)
SELECT COUNT(*) FROM dbo.Products
WHERE FREETEXT(ProductDescription, 'wireless bluetooth mouse');
-- Logical reads: 4
-- CPU: 15ms, elapsed: 35ms
-- More CPU than single-term because of stemming + thesaurus + union

-- CONTAINS — multi-word (boolean AND)
SELECT COUNT(*) FROM dbo.Products
WHERE CONTAINS(ProductDescription, '"wireless" AND "bluetooth" AND "mouse"');
-- Logical reads: 4
-- CPU: 12ms, elapsed: 25ms
```

**Improvement:** FREETEXT is ~35x faster than LIKE (480ms → 14ms for single word). Multi-word FREETEXT is slightly slower than multi-word CONTAINS because of the implicit OR (more terms matched → more key lookups in the data retrieval phase).

### Question Bank

### Spoken Answers

**Q1: What is FREETEXT and how does it differ from CONTAINS?**

> **Average answer:** FREETEXT is like CONTAINS but it does fuzzy matching. CONTAINS is for exact matches.

> **Great answer:** FREETEXT and CONTAINS are both full-text predicates backed by the same inverted index and the same full-text engine, but they differ fundamentally in how they interpret the search phrase and combine results. CONTAINS gives the developer explicit control: you specify exact terms, boolean operators (AND, OR, AND NOT), proximity (NEAR), and linguistic forms (FORMSOF). Every term must match as you specify — it is a precise tool for programmatic search. FREETEXT, by contrast, treats the search phrase as natural language input. It word-breaks the phrase into individual tokens, removes stopwords, stems each token to its root form using the language stemmer, expands through the thesaurus for synonyms, and then returns rows where ANY of the expanded terms match — using implicit OR with internal relevance weighting. The key difference in practice is breadth: CONTAINS with `'"wireless" AND "mouse"'` returns only products mentioning both terms; FREETEXT with `'wireless mouse'` returns products mentioning wireless, mouse, or synonyms of either — typically 10-50x more results. CONTAINS is for precision; FREETEXT is for recall. I use FREETEXT for user-facing search bars where I want maximum recall and the user expects "Google-like" behavior. I use CONTAINS for faceted search, advanced search, or any query where boolean logic is required.

**Q4: Why does FREETEXT not have an ORDER BY for relevance?**

> **Average answer:** Because FREETEXT does not return a rank value. You need to use FREETEXTTABLE for ranking.

> **Great answer:** FREETEXT is designed as a membership predicate — it determines whether a row matches the search phrase, not how well it matches. The full-text engine does compute an internal relevance score for each matching document using term frequency (TF), inverse document frequency (IDF), and document length normalization, but this score is NOT exposed through the FREETEXT WHERE clause predicate. The internal score is used by the engine to order results internally within the engine, but the relational engine does not capture it. To access the rank value, you must use FREETEXTTABLE, which returns a two-column result set: `[KEY]` (the document's primary key value) and `RANK` (an integer 0-1000 representing relevance). You JOIN this result set with the base table and use `ORDER BY RANK DESC` to get relevance-ordered results. The rank is calculated using a formula similar to: `RANK = Σ (log10(TF) * IDF)` for each matched term in each document, normalized by document length. This is the same calculation used internally by the FREETEXT engine, but FREETEXTTABLE makes it visible. In production, I almost never use bare FREETEXT in the WHERE clause for search — I always use FREETEXTTABLE so that results can be ordered by relevance.

**Q8: How do you implement FREETEXT in EF Core and Dapper?**

> **Average answer:** You use raw SQL because EF Core does not support FREETEXT in LINQ.

> **Great answer:** Neither EF Core nor Dapper has built-in support for FREETEXT because it is a SQL Server-specific extension that does not exist in the LINQ expression tree model. In EF Core, you use `SqlQuery` (EF Core 7+) or `FromSqlRaw` to execute a raw SQL query and map results to a POCO. For ranking, you JOIN with FREETEXTTABLE and map the RANK column as a property on the result type. In Dapper, you use `QueryAsync` with the same raw SQL pattern. One key consideration is that FREETEXT does not rank results — if you need relevance ordering, you must use FREETEXTTABLE. Another consideration is that the search phrase is passed as a parameter, not concatenated into the SQL string, to prevent SQL injection. I also filter stopwords at the application level before passing the phrase to FREETEXT, to prevent the "empty term set" issue where all terms are stopwords and the query returns zero results.

**Challenge 1 — Write the SQL**

You have a `dbo.KnowledgeBase` table with `ArticleId`, `Title`, `Content`, `Category`, and `IsPublished` columns. The full-text index covers `Title` and `Content`. Write a search procedure that accepts a user's natural language query (e.g., "how to reset my password"), finds the top 10 most relevant articles, and returns the results ordered by relevance.

<details>
<summary>Solution</summary>

```sql
CREATE PROCEDURE dbo.SearchKnowledgeBase
    @SearchPhrase NVARCHAR(500),
    @Category NVARCHAR(100) = NULL
AS
BEGIN
    SET NOCOUNT ON;

    SELECT TOP 10
        a.ArticleId,
        a.Title,
        a.Category,
        ft.RANK AS Relevance
    FROM dbo.KnowledgeBase a
    INNER JOIN FREETEXTTABLE(KnowledgeBase,
        (Title, Content),
        @SearchPhrase,
        10) ft ON a.ArticleId = ft.[KEY]
    WHERE a.IsPublished = 1
      AND (@Category IS NULL OR a.Category = @Category)
    ORDER BY ft.RANK DESC;
END;
GO
```

**Logical reads:** ~20 (10 key lookups × 2 pages each) + FTS engine lookups (opaque).

**Execution plan:** `FullTextMatch` → `Nested Loops` (join on ArticleId) → `Filter` (IsPublished, Category) → `Sort` (rank DESC) → `Top`

**Challenge 3 — Explain the execution plan**

```sql
-- Query:
SELECT p.ProductId, p.ProductName
FROM dbo.Products p
WHERE FREETEXT(p.ProductDescription, 'wireless mouse')
  AND p.IsActive = 1
  AND p.CategoryName = 'Electronics';
```

The execution plan shows: `FullTextMatch` (1,200 rows returned) → `Clustered Index Seek` (1,200 executions) → `Filter` (IsActive=1 AND CategoryName='Electronics') → `SELECT` (247 rows). Why is the Clustered Index Seek executed 1,200 times when only 247 rows are returned? How would you fix this?

<details>
<summary>Solution</summary>

**Why 1,200 executions:** FREETEXT with 'wireless mouse' (implicit OR) matched 1,200 documents containing "wireless", "mouse", or synonyms. The relational engine performs a clustered index seek for EACH of the 1,200 matched documents to retrieve the ProductName, IsActive, and CategoryName values. After all 1,200 key lookups, the Filter operator eliminates 953 rows that do not satisfy `IsActive = 1` or `CategoryName = 'Electronics'`. The 953 key lookups were wasted.

**Fix:** Create a covering index that includes the filtered columns, so the key lookup reads from a narrow non-clustered index instead of the full clustered index:

```sql
CREATE NONCLUSTERED INDEX IX_Products_SearchCovering
    ON dbo.Products (ProductId)
    INCLUDE (ProductName, IsActive, CategoryName);
```

With this covering index, each key lookup reads a single page from the narrow covering index instead of 2-3 pages from the wide clustered index. This reduces IO but does not eliminate the wasted lookups.

**Better fix:** Push the non-FTS filters into the query differently by using EXISTS with FREETEXTTABLE:

```sql
-- Key lookups are done only after both filters pass
```

**Challenge 4 — Diagnose the concurrency problem**

Your e-commerce site uses FREETEXT for product search. The Products table has 2M rows with a full-text index on ProductName and Description. During Black Friday, search traffic peaks at 200 queries/second. Users report that search is slow or timing out. You check `sys.dm_os_waiting_tasks` and see many tasks waiting on `SOS_SCHEDULER_YIELD` with high `signal_wait_time_ms`.

<details>
<summary>Solution</summary>

**Root cause:** The FTS engine is CPU-bound from processing 200 FREETEXT queries/second. Each query word-breaks the search phrase, stems each term, expands through the thesaurus, performs inverted index lookups for each expanded term, and unions the results. At 200 qps, the FTS engine saturates the CPU, causing `SOS_SCHEDULER_YIELD` waits as threads exceed their quantum.

**Detection query:**

```sql
-- Check for signal waits (CPU pressure)
SELECT
    wait_type,
    wait_time_ms / 1000.0 AS WaitTimeSec,
    signal_wait_time_ms / 1000.0 AS SignalWaitSec,
    waiting_tasks_count,
    (signal_wait_time_ms * 1.0 / wait_time_ms) * 100 AS SignalPercent
FROM sys.dm_os_wait_stats
WHERE wait_type = 'SOS_SCHEDULER_YIELD'
ORDER BY signal_wait_time_ms DESC;

-- Check FTS query stats
SELECT
    OBJECT_NAME(st.objectid) AS TableName,
    qs.execution_count,
    qs.total_worker_time / 1000 AS TotalCpuMs,
    qs.total_elapsed_time / 1000 AS TotalElapsedMs
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
WHERE st.text LIKE '%FREETEXT%';
```

**Fix:**
1. Implement application-level caching of search results for frequent queries (e.g., dictionary cache with 5-minute TTL)
2. Use FREETEXTTABLE with a smaller `topN` to reduce FTS engine work
3. Add rate limiting on the search endpoint
4. Scale up (more CPU) or scale out (read replicas)
5. Consider Elasticsearch for production-scale search at this volume

**In .NET:**

```csharp
public async Task<IReadOnlyList<ProductSearchResult>> SearchWithCacheAsync(
    string searchPhrase, CancellationToken cancellationToken = default)
{
    var cacheKey = $"search:freetext:{searchPhrase.ToLowerInvariant()}";

    return await _cache.GetOrCreateAsync(cacheKey, async entry =>
    {
        entry.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5);
        entry.SlidingExpiration = TimeSpan.FromMinutes(1);
        return await SearchProductsAsync(searchPhrase, cancellationToken);
    }) ?? Array.Empty<ProductSearchResult>();
}
```

**Challenge 5 — Design the index**

**Scenario:** You are building search for a recipe website. The `dbo.Recipes` table has 500K recipes with columns: `RecipeId` (INT PK), `Title` (NVARCHAR(300)), `Ingredients` (NVARCHAR(MAX) — comma-separated list), `Instructions` (NVARCHAR(MAX) — multi-paragraph text), `CuisineType` (NVARCHAR(50)), `DifficultyLevel` (NVARCHAR(20)), `PrepTimeMinutes` (INT), `IsPublished` (BIT). Users search by typing natural language queries like "quick chicken pasta dinner under 30 minutes". Response time must be <1 second for 50 qps.

<details>
<summary>Solution</summary>

```sql
-- Full-text index covering all searchable text columns
CREATE FULLTEXT CATALOG RecipeCatalog;
GO

CREATE FULLTEXT INDEX ON dbo.Recipes
(
    Title           LANGUAGE 1033,
    Ingredients     LANGUAGE 1033,
    Instructions    LANGUAGE 1033
)
KEY INDEX PK_Recipes
ON RecipeCatalog
WITH
(
    CHANGE_TRACKING MANUAL,       -- Recipes updated infrequently (~500/day)
    STOPLIST = SYSTEM
);
GO

-- Schedule population every 15 minutes:
-- ALTER FULLTEXT INDEX ON dbo.Recipes START UPDATE POPULATION;

-- Covering index for filter columns + key lookup avoidance
CREATE NONCLUSTERED INDEX IX_Recipes_SearchCovering
    ON dbo.Recipes (RecipeId)
    INCLUDE (Title, CuisineType, DifficultyLevel, PrepTimeMinutes, IsPublished)
    WHERE IsPublished = 1;
GO

-- Search procedure
CREATE PROCEDURE dbo.SearchRecipes
    @SearchPhrase   NVARCHAR(500),
    @CuisineType    NVARCHAR(50) = NULL,
    @MaxPrepTime    INT = NULL,
    @Difficulty     NVARCHAR(20) = NULL,
    @MaxResults     INT = 20
AS
BEGIN
    SET NOCOUNT ON;

    SELECT TOP (@MaxResults)
        r.RecipeId,
        r.Title,
        r.CuisineType,
        r.DifficultyLevel,
        r.PrepTimeMinutes,
        ft.RANK AS Relevance
    FROM dbo.Recipes r
    INNER JOIN FREETEXTTABLE(Recipes,
        (Title, Ingredients, Instructions),
        @SearchPhrase,
        @MaxResults) ft ON r.RecipeId = ft.[KEY]
    WHERE r.IsPublished = 1
      AND (@CuisineType IS NULL OR r.CuisineType = @CuisineType)
      AND (@MaxPrepTime IS NULL OR r.PrepTimeMinutes <= @MaxPrepTime)
      AND (@Difficulty IS NULL OR r.DifficultyLevel = @Difficulty)
    ORDER BY ft.RANK DESC;
END;
GO
```

**Why these choices:**
- FREETEXTTABLE with topN ensures only the top @MaxResults recipes incur key lookups
- CHANGE_TRACKING MANUAL (500 writes/day is low; schedule every 15 minutes)
- Covering filtered index makes key lookups fast and narrow
- The WHERE clause filters (cuisine, prep time, difficulty) are applied after JOIN — true, but with topN=20 only 20 key lookups happen

**What NOT to index:**
- Do NOT add a B-tree index on Ingredients or Instructions — too large, FTS handles them
- Do NOT create separate full-text indexes for each column — a single FTS index covering all columns is sufficient

</details>
