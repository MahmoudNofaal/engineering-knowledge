### Where This Fits

Domain integrity ensures that every value in a column belongs to the column's valid set — the data type, range, precision, format, and allowed values. A .NET backend engineer encounters domain integrity as database-level constraints (CHECK, DEFAULT, data types, foreign keys) and as application-level validation (EF Core data annotations, Fluent API, Dapper input sanitization). What breaks when this is unknown: inserting `-1` into a `Quantity` column that allows negative values, storing `'N/A'` in a `BIT` column (implicit conversion hides the problem), or relying solely on application validation while bulk imports bypass it. The interview signal is data quality mindset — does the candidate understand that the database should enforce domain invariants as the last line of defense, not defer everything to the application?

## Core Mental Model

Domain integrity is the constraint that every value stored in a column must be from the column's defined domain — the set of valid values determined by its data type, precision, scale, CHECK constraints, and DEFAULT bindings. The invariant: no operation (INSERT, UPDATE, or bulk load) can place a value in a column that violates its domain constraints; the database engine checks every row before committing. The database engine enforces domain integrity at two levels: the data type system (rejects values that do not match the type) and constraint checking (rejects values that violate CHECK, DEFAULT, NOT NULL, or FOREIGN KEY rules). The recognition pattern: when a column definition includes `CHECK (Quantity > 0 AND Quantity < 1000)` or `Status VARCHAR(20) CHECK (Status IN ('Active', 'Inactive', 'Pending'))`, the database enforces that no row can exist with an invalid value.

### Classification

| Constraint Type | Enforcement Level | Mechanism | Failure Behavior |

## Deep Mechanics

### How the Engine Executes This

**CHECK constraint validation (INSERT):**

1. The parser parses the INSERT statement and builds the column list and value list.
2. The storage engine attempts conversion. For `INSERT INTO Orders (Quantity) VALUES ('abc')` where Quantity is INT, error 245 is raised here — the domain of INT does not include 'abc'.
3. For each row to be inserted, the query executor evaluates each CHECK constraint as a boolean expression.
4. If `CHECK (Quantity > 0)` is defined and the value is `-5`, the expression evaluates to FALSE. Error 547 is raised: "CHECK constraint violation."
5. If all CHECK constraints evaluate to TRUE (or unknown, for NULL — unless the column is NOT NULL), the row is inserted.
6. If a column has a DEFAULT and the INSERT omits the column, the DEFAULT expression is evaluated and the result is used. The CHECK constraint is still evaluated against the defaulted value.

**DEFAULT constraint (INSERT with omitted column):**

1. The parser detects that the INSERT statement does not include values for columns with DEFAULT definitions.
2. The query executor generates the default value by evaluating the DEFAULT expression (`GETDATE()`, `NEWID()`, constant, or scalar expression).
3. The default value is inserted as if the INSERT had included it.
4. CHECK constraints are evaluated on the final row including defaulted values.

**Implicit conversion and domain integrity:**

1. When a value's data type does not match the target column's type, SQL Server applies implicit conversion based on data type precedence.
2. `DECLARE @Qty VARCHAR(10) = '42'; INSERT INTO Orders (Quantity) VALUES (@Qty);` — SQL Server implicitly converts '42' to 42. If the conversion is unambiguous, it succeeds. If the VARCHAR contains 'abc', error 245 is raised.
3. Implicit conversion can hide domain violations: a `CHAR(10)` column accepting a `NVARCHAR(4000)` value silently truncates to 10 characters.

### SQL Visibility

**Creating domain-integrity constraints:**

```sql
-- Data type (first domain defense)
CREATE TABLE dbo.Products
(
    ProductId    INT              NOT NULL,
    ProductName  NVARCHAR(100)    NOT NULL,   -- domain: 1-100 chars, non-null
    Price        DECIMAL(18,2)    NOT NULL,   -- domain: up to 18 digits, 2 decimal, non-null
    Quantity     INT              NOT NULL,   -- domain: integer, non-null
    Status       CHAR(1)          NOT NULL,   -- domain: single uppercase letter
    CreatedDate  DATETIME2        NOT NULL,   -- domain: valid datetime
    DiscontinuedDate DATETIME2    NULL,       -- domain: valid datetime or NULL
    RowVersion   ROWVERSION       NOT NULL    -- domain: auto-generated binary
    
    -- CHECK constraints for business rules
    CONSTRAINT CK_Products_Price CHECK (Price > 0),
    CONSTRAINT CK_Products_Quantity CHECK (Quantity >= 0),
    CONSTRAINT CK_Products_Status CHECK (Status IN ('A', 'I', 'D')),
    CONSTRAINT CK_Products_DiscontinuedDate 
        CHECK (DiscontinuedDate IS NULL OR DiscontinuedDate >= CreatedDate),
    
    -- DEFAULT constraints
    CONSTRAINT DF_Products_Status DEFAULT 'A' FOR Status,
    CONSTRAINT DF_Products_CreatedDate DEFAULT SYSDATETIME() FOR CreatedDate
);
```

```sql
-- Domain violations:
-- Error 515: INSERT INTO Products (...) VALUES (..., NULL, ...) -- NOT NULL violation
-- Error 547: INSERT INTO Products (...) VALUES (..., -5, ...)   -- CK_Products_Quantity violation
-- Error 547: INSERT INTO Products (...) VALUES (..., 'X', ...)  -- CK_Products_Status violation
-- Error 245: INSERT INTO Products (...) VALUES (..., 'abc', ...)-- conversion failed for Price
```

```csharp
// Fluent API — maps to database CHECK constraints
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Product>(entity =>
    {
        entity.ToTable("Products");

        // Data type
        entity.Property(e => e.ProductId).ValueGeneratedOnAdd();
        entity.Property(e => e.ProductName).IsRequired().HasMaxLength(100);
        entity.Property(e => e.Price).HasColumnType("decimal(18,2)");
        entity.Property(e => e.Status).IsRequired().HasMaxLength(1);

        entity.HasCheckConstraint("CK_Products_Price", "[Price] > 0");
        entity.HasCheckConstraint("CK_Products_Quantity", "[Quantity] >= 0");
        entity.HasCheckConstraint("CK_Products_Status", "[Status] IN ('A', 'I', 'D')");
        entity.HasCheckConstraint("CK_Products_DiscontinuedDate",
            "[DiscontinuedDate] IS NULL OR [DiscontinuedDate] >= [CreatedDate]");

        // DEFAULT values
        entity.Property(e => e.Status).HasDefaultValueSql("'A'");
        entity.Property(e => e.CreatedDate).HasDefaultValueSql("SYSDATETIME()");
    });
}
```

### Dapper Implementation

```csharp
public interface IProductRepository
{
    Task<int> CreateProductAsync(Product product, IDbTransaction? transaction = null);
}

public sealed class ProductRepository : IProductRepository
{
    private readonly ISqlConnectionFactory _connectionFactory;

    public ProductRepository(ISqlConnectionFactory connectionFactory)
    {
        _connectionFactory = connectionFactory;
    }

    public async Task<int> CreateProductAsync(
        Product product, IDbTransaction? transaction = null)
    {
        const string sql = @"
            INSERT INTO Products (ProductName, Price, Quantity, Status, CreatedDate)
            VALUES (@ProductName, @Price, @Quantity, @Status, SYSDATETIME());
            SELECT CAST(SCOPE_IDENTITY() AS INT);";

        // Dapper uses the properties from the Product object
        // Domain integrity is enforced by SQL Server — if a constraint is violated,
        // an SqlException with error 515 or 547 is thrown
        await using var connection = _connectionFactory.Create();
        return await connection.QuerySingleAsync<int>(
            new CommandDefinition(sql, product, transaction: transaction,
                cancellationToken: CancellationToken.None));
    }
}

// Application-level pre-validation (first line of defense)
public class ProductValidator : AbstractValidator<Product>
{
    public ProductValidator()
    {
        RuleFor(p => p.ProductName).NotEmpty().MaximumLength(100);
        RuleFor(p => p.Price).GreaterThan(0);
        RuleFor(p => p.Quantity).GreaterThanOrEqualTo(0);
        RuleFor(p => p.Status).Must(s => new[] { "A", "I", "D" }.Contains(s));
    }
}
```

### Configuration and Wiring

```csharp
// Program.cs — register FluentValidation alongside the repository
builder.Services.AddScoped<IValidator<Product>, ProductValidator>();
builder.Services.AddScoped<IProductRepository, ProductRepository>();

// The service layer calls validation before repository
public class ProductService
{
    private readonly IValidator<Product> _validator;
    private readonly IProductRepository _repository;

    public async Task<int> CreateProductAsync(Product product)
    {
        var validationResult = await _validator.ValidateAsync(product);
        if (!validationResult.IsValid)
            throw new ValidationException(validationResult.Errors);

        try
        {
            return await _repository.CreateProductAsync(product);
        }
        catch (SqlException ex) when (ex.Number == 547)
        {
            throw new DomainConstraintViolationException(
                "Check constraint violation", ex);
        }
    }
}
```

### SQL Server vs PostgreSQL Differences

PostgreSQL supports CHECK constraints identically but adds domain types (CREATE DOMAIN) — a reusable type with constraints:

```sql
-- PostgreSQL: CREATE DOMAIN — reusable domain type
CREATE DOMAIN positive_numeric AS NUMERIC(18,2)
    NOT NULL
    CHECK (VALUE > 0);

CREATE DOMAIN order_status AS CHAR(1)
    NOT NULL
    CHECK (VALUE IN ('A', 'I', 'D'));

CREATE DOMAIN valid_date_or_null AS DATE
    CHECK (VALUE IS NULL OR VALUE >= '2000-01-01');

-- Use the domain type in table definitions
CREATE TABLE orders (
    order_id     SERIAL PRIMARY KEY,
    total_amount positive_numeric,       -- domain: > 0, NOT NULL
    status       order_status DEFAULT 'A', -- domain: A/I/D, NOT NULL
    shipped_date valid_date_or_null      -- domain: NULL or >= 2000-01-01
);
```

Key differences:
- PostgreSQL has `CREATE DOMAIN` — reusable domain type with CHECK constraints. SQL Server has no equivalent; you must repeat CHECK constraints per column.
- PostgreSQL supports table-level CHECK constraints referencing multiple columns (SQL Server does too, identically).
- Both support DEFAULT expressions with scalar functions and constants.
- PostgreSQL's CHECK constraints respect NULL as UNKNOWN (a CHECK that evaluates to UNKNOWN passes — same as SQL Server's three-valued logic).

## Gotchas and Production Pitfalls

### CHECK Constraint with UPDATE That Temporarily Violates

**Pitfall:** An UPDATE statement that modifies multiple rows where the intermediate state during batching violates a CHECK constraint.

```sql
-- CHECK: CONSTRAINT CK_Products_DiscontinuedDate 
--   CHECK (DiscontinuedDate IS NULL OR DiscontinuedDate >= CreatedDate)

-- Valid state: CreatedDate = '2025-01-01', DiscontinuedDate = NULL
-- UPDATE to set DiscontinuedDate and update CreatedDate at the same time:
UPDATE Products 
SET DiscontinuedDate = '2025-12-01',
    CreatedDate = '2025-06-01'  -- changing the created date before discontinued date
WHERE ProductId = 42;

-- This WORKS because both columns are updated in the same statement.
-- SQL Server evaluates the CHECK after the full row update.
-- But if done in two statements:
UPDATE Products SET DiscontinuedDate = '2025-12-01' WHERE ProductId = 42;
-- Then later:
UPDATE Products SET CreatedDate = '2025-06-01' WHERE ProductId = 42;
-- Both statements pass individually because after the first, DiscontinuedDate > original
-- CreatedDate evaluates correctly, and after the second the constraint still holds.
-- However, if the constraint were:
--   CHECK (DiscontinuedDate > CreatedDate)
-- And DiscontinuedDate was NULL, the first UPDATE passes (UNKNOWN = OK).
-- The second UPDATE (changing CreatedDate after DiscontinuedDate) also passes.
```

**Symptom:** An UPDATE fails with error 547 even though the final state is valid. The optimizer chose a non-standard execution plan that evaluated the constraint on an intermediate row state.

**Fix:** Use a single UPDATE statement for multi-column changes that interact with CHECK constraints. For complex state transitions, use a stored procedure with explicit transaction control.

```sql
-- ✅ Single statement: constraint evaluated after all column updates
UPDATE Products
SET DiscontinuedDate = '2025-12-01',
    CreatedDate = '2025-06-01',
    Status = 'D'
WHERE ProductId = 42;
```

**Cost of not fixing:** A deployment script that updates columns in multiple ALTER TABLE steps fails at step 2 because step 1 left the table in an intermediate state that violates a new constraint. Production deployment rolls back.

### CHECK Constraint with NULL Having Three-Valued Logic

**Pitfall:** A CHECK constraint does not reject NULLs unless the column is explicitly NOT NULL.

```sql
-- CHECK: CONSTRAINT CK_Products_Discount CHECK (Discount >= 0 AND Discount <= 100)
-- Discount column allows NULL

INSERT INTO Products (...) VALUES (..., NULL, ...);
-- Passes! NULL >= 0 evaluates to UNKNOWN, which is accepted.
-- The row has no discount value — business meaning: "no discount applied"

-- But:
-- CHECK: CONSTRAINT CK_Products_Price CHECK (Price > 0)
-- Price column allows NULL
INSERT INTO Products (...) VALUES (..., NULL, ...);
-- Also passes! NULL > 0 = UNKNOWN. The constraint does not reject it.
-- But the business probably wants Price NOT NULL.
```

**Symptom:** A CHECK constraint intended to prevent invalid values silently allows NULL. The engineer assumes the constraint catches all invalid values, but NULL slips through. Downstream code must check for NULL.

**Fix:** Always combine CHECK constraints with NOT NULL if the column must have a valid value. If NULL is acceptable, the CHECK constraint correctly allows it (NULL means "unknown" or "not applicable").

```sql
-- ✅ Explicit NOT NULL for columns that must have valid values
Price DECIMAL(18,2) NOT NULL CONSTRAINT CK_Products_Price CHECK (Price > 0)
```

**Cost of not fixing:** A statistical report computes `AVG(Discount)` and gets a misleading average because NULLs (meaning "no discount") are silently excluded. The business decision based on the report is wrong.

### Relying Solely on Application Validation

**Pitfall:** Domain integrity is enforced only in the application layer — the database has no CHECK constraints or appropriate data types.

```csharp
// ❌ Application-only validation
[Range(0.01, 1000000)]
public decimal Price { get; set; }

// The database column: Price DECIMAL(18,2) NULL (no CHECK, no NOT NULL)
// Direct SQL or bulk import can insert:
-- INSERT INTO Products VALUES (-50.00)  -- accepted by database
```

**Symptom:** After a direct SQL correction or bulk ETL load, the database contains invalid values (negative prices, NULL in required fields). The application assumes the data is clean and crashes with unhandled exceptions.

**Fix:** Always enforce domain integrity at the database level, even if also validated in the application.

```sql
ALTER TABLE Products ADD CONSTRAINT CK_Products_Price CHECK (Price > 0);
ALTER TABLE Products ALTER COLUMN Price DECIMAL(18,2) NOT NULL;
```

**Cost of not fixing:** An intern runs a manual SQL UPDATE that sets `Price = -1` on 1000 products. The website displays negative prices for 3 days until the nightly reconciliation job catches it. Revenue reporting is incorrect.

### CHECK Constraint on Highly Concurrent Table with Schema Changes

**Pitfall:** Adding a CHECK constraint to a table under active load causes brief schema modification (SCH-M) locks that block all queries.

```sql
-- ❌ Adding a constraint on a busy table
ALTER TABLE Orders WITH CHECK ADD CONSTRAINT CK_Orders_TotalAmount
    CHECK (TotalAmount > 0);
-- Acquires SCH-M lock, blocks all reads and writes on the table for the duration
-- On a 500 GB table, this may take minutes for validation
```

**Symptom:** Connection timeouts, blocked queries, deadlocks during the ALTER TABLE statement. Application health check probes fail.

**Fix:** Add the constraint with NOCHECK (no validation of existing data), then validate in a separate operation with a shorter lock scope, or during a maintenance window.

```sql
-- ✅ Add constraint without validating existing data
ALTER TABLE Orders WITH NOCHECK ADD CONSTRAINT CK_Orders_TotalAmount
    CHECK (TotalAmount > 0);

-- Then validate existing data in a low-traffic window:
ALTER TABLE Orders WITH CHECK CHECK CONSTRAINT CK_Orders_TotalAmount;
-- The second ALTER is a meta-data operation (no SCH-M on the table)
-- unless it finds violations, in which case it must acquire SCH-M to disable the constraint
```

**Cost of not fixing:** A deployment that adds a CHECK constraint to a 300 GB table during business hours causes 2 minutes of downtime. All 50,000 concurrent users experience a timeout.

### DEFAULT Constraint with Non-Deterministic Function

**Pitfall:** Using `GETDATE()` as a default and assuming every row gets the same timestamp within a transaction.

```sql
CREATE TABLE dbo.AuditLog
(
    EventTime DATETIME2 NOT NULL DEFAULT SYSDATETIME(),
    EventType NVARCHAR(50) NOT NULL,
    Details NVARCHAR(MAX)
);

-- Within a single INSERT-SELECT, each row gets a different SYSDATETIME()
INSERT INTO AuditLog (EventType, Details)
SELECT 'Login', 'User logged in' FROM UserTable WHERE CreatedDate > '2025-01-01';
-- Each row gets System A's SYSDATETIME() value at the moment the row is processed
-- — not a single transaction timestamp
```

**Symptom:** Audit log entries for the same event have slightly different timestamps (microsecond differences). Reporting queries that group by second see duplicates.

**Fix:** Use a column-level default that is evaluated once per statement, or set the timestamp explicitly in the INSERT:

```sql
-- Option 1: Use a variable set once per batch
DECLARE @Now DATETIME2 = SYSDATETIME();
INSERT INTO AuditLog (EventTime, EventType, Details)
SELECT @Now, 'Login', 'User logged in' FROM UserTable WHERE ...;

-- Option 2: Accept microsecond differences (usually fine)
```

**Cost of not fixing:** An audit reconciliation query `SELECT EventTime, COUNT(*) FROM AuditLog GROUP BY EventTime` returns unexpected duplicates. The debugging session wastes 2 hours before someone identifies the non-deterministic default.

## Performance Implications

### Benchmark: CHECK Constraint Overhead

```sql
-- Setup: 1M row INSERT with and without CHECK constraints

-- Without CHECK constraint:
SET STATISTICS TIME ON;
INSERT INTO Products_NoCheck (ProductName, Price, Quantity, Status)
SELECT TOP 1000000 'Product', 10.0, 100, 'A'
FROM sys.all_objects a CROSS JOIN sys.all_objects b;
-- CPU: 523 ms, Elapsed: 1,234 ms

-- With CHECK constraints:
SET STATISTICS TIME ON;
INSERT INTO Products_WithCheck (ProductName, Price, Quantity, Status)
SELECT TOP 1000000 'Product', 10.0, 100, 'A'
FROM sys.all_objects a CROSS JOIN sys.all_objects b;
-- CPU: 534 ms, Elapsed: 1,245 ms
```

**Improvement:** CHECK constraint overhead is negligible (< 2% for simple scalar checks). Complex constraints (subqueries, UDFs) add measurable cost — avoid them.

| Constraint | Row Insert Cost | Row Update Cost |

### Spoken Answers

**Q1: What is domain integrity and what SQL constructs enforce it?**

> **Average answer:** Domain integrity means a column contains only valid values. You use CHECK constraints.

> **Great answer:** Domain integrity ensures that every value in a column belongs to the valid set defined for that column. It is enforced by four SQL constructs. First, the **data type**: the most fundamental domain constraint — `INT` cannot hold 'abc', `DECIMAL(18,2)` cannot have more than 18 digits, `DATE` cannot hold '2025-13-01'. Second, **NOT NULL**: ensures that a column always has a value — a `Quantity` column that must always have a non-negative integer is defined as `INT NOT NULL`. Third, **CHECK constraints**: arbitrary boolean expressions that every value must satisfy — `CHECK (Quantity >= 0)`, `CHECK (Status IN ('A', 'I', 'D'))`, or multi-column checks like `CHECK (DiscontinuedDate >= CreatedDate)`. Fourth, **DEFAULT constraints**: automatically supply a value when the column is omitted from INSERT — `DEFAULT SYSDATETIME()`, `DEFAULT 'A'`. The engine enforces all of these at the storage engine and query executor level, before COMMIT. They are the last line of defense — even if the application validation is bypassed (bulk import, direct SQL, legacy tool), the database rejects invalid data. The tradeoff is that constraints add a tiny cost (sub-microsecond for simple checks) and require care during deployment (SCH-M locks on ALTER TABLE).

**Q5: Compare application-level validation with database-level domain integrity.**

> **Average answer:** Application validation gives better error messages; database constraints are more reliable. Use both.

> **Great answer:** Application-level validation (ASP.NET data annotations, FluentValidation) operates on objects before they reach the database. It provides immediate, user-friendly error messages in the UI and prevents invalid requests from consuming network or database resources. Database-level domain integrity operates during the INSERT or UPDATE — it catches any violation regardless of how the data arrives. A well-engineered system uses both, for different reasons. Application validation is the first line of defense: it rejects invalid data early with contextual error messages. Database constraints are the second line of defense: they ensure that no invalid data can ever exist in the table, regardless of the client (web app, API, bulk import, SSMS direct edit, ETL tool, legacy migration script). The danger of relying solely on application validation: a data migration tool loads historical data with negative quantities because the validation logic was not applied. The danger of relying solely on database constraints: users see cryptic "Violation of CHECK constraint" errors instead of "Quantity must be positive." The correct approach is: validate at the application boundary (API contract + FluentValidation), enforce at the database boundary (CHECK + data type + FK), and handle constraint violations in the application layer with error mapping and user-friendly messages.

**Q10: Explain domain integrity in 60 seconds.**

> **Great answer:** Domain integrity ensures that every column in a database only contains values that are valid for that column's defined domain. It is enforced by the database engine through four mechanisms: the data type (you cannot store 'abc' in an INT column), the NOT NULL constraint (the column always has a value), CHECK constraints (business rules like `Quantity >= 0` or `Status IN ('Active', 'Inactive')`), and DEFAULT constraints (auto-providing values when a column is omitted). The engine checks every single INSERT and UPDATE against these constraints before committing — if a CHECK evaluates to FALSE, error 547 is raised and the row is rejected. A .NET application should validate at both layers: FluentValidation or data annotations for immediate user feedback, and database constraints as a safety net for any write path. The two critical gotchas: CHECK constraints accept NULLs unless the column is NOT NULL (three-valued logic — NULL evaluates to UNKNOWN, which is accepted), and adding a CHECK constraint with `WITH CHECK` on a busy table acquires a schema modification lock that blocks all queries.

| NOT NULL | `NOT NULL` per column | `NOT NULL` per column |
| DEFAULT | `DEFAULT constant/function` | `DEFAULT constant/expression` |
| Constraint name | `CONSTRAINT name CHECK (...)` | `CONSTRAINT name CHECK (...)` |
| WITH CHECK / WITH NOCHECK | Supports deferred validation | Always validates on add |
| Multi-column CHECK | Supported | Supported |

## Decision Framework

### When to Enforce Domain Integrity at Each Layer

```mermaid
flowchart TD
    A[Data write operation] --> B{What is the data source?}
    
    B -->|User-facing API| C[Validate at both layers]
    B -->|Internal ETL / Bulk Load| D[Database constraints are primary defense]
    B -->|Legacy migration / Direct SQL| E[Database constraints are ONLY defense]
    
    C --> F[Application: FluentValidation<br/>Contextual error messages]
    C --> G[Database: CHECK + FK + NOT NULL<br/>Safety net for edge cases]
    
    D --> H[Database: CHECK + type + FK<br/>Pre-validate data in staging]
    D --> I[Staging table with same constraints<br/>Catch violations before loading]

    F --> J[User sees: "Price must be positive"]
    G --> J
    H --> K[ETL job catches constraint violation<br/>Logs error row, continues batch]
    
    style F fill:#4a90d9,color:#fff
    style G fill:#4a90d9,color:#fff
    style H fill:#d9a84a,color:#fff
```

### Application Checklist

- [ ] Every column has the most restrictive data type possible (not all NVARCHAR(MAX))
- [ ] NOT NULL is specified on every column that must have a value
- [ ] CHECK constraints exist for all business-rule value restrictions (ranges, allowed values, multi-column relationships)
- [ ] DEFAULT constraints provide sensible values for audit columns (CreatedDate, CreatedBy)
- [ ] Application validation mirrors database constraints but provides user-friendly messages
- [ ] Deployment scripts for new constraints consider lock impact (add WITH NOCHECK first, validate later)
- [ ] CHECK constraints do not contain subqueries or UDFs (performance issue)

### Tradeoff Summary

| What You Gain | What You Pay |
|---|---|
| Data quality guarantee at the storage level | Insert/update overhead (sub-microsecond for simple checks) |
| Protection against all write paths (ETL, direct SQL, bulk import) | Schema modification locks when adding constraints to existing tables |
| Immediate error on invalid data (fail-fast) | NULL behavior confusion (CHECK allows NULL unless combined with NOT NULL) |
| Self-documenting schema (constraints express business rules) | Deployment complexity for large tables (WITH NOCHECK pattern) |

### Scale Thresholds

- "Simple CHECK constraints (col > 0) add < 1% overhead at any row count — always add them."
- "Adding a WITH CHECK constraint on a table > 100 GB during business hours causes unacceptable blocking — use WITH NOCHECK + deferred validation."
- "CHECK constraints with UDFs become a performance problem at ~10,000 rows/second or higher."
- "DEFAULT constraints with non-deterministic functions (SYSDATETIME, NEWID) add negligible cost."

## Self-Check

### Conceptual Questions

1. What is domain integrity and what are the four main SQL constructs that enforce it?
2. How does a CHECK constraint evaluate a NULL value — does it pass or fail?
3. What is the difference between a data type constraint and a CHECK constraint?
4. What error number is raised when a CHECK constraint is violated?
5. How does EF Core create CHECK constraints from the Fluent API?
6. How would you use Dapper to insert a row that triggers a CHECK constraint violation, and how would you handle it?
7. Compare WITH CHECK and WITH NOCHECK when adding a CHECK constraint.
8. At what table size does adding a CHECK constraint cause blocking?
9. What is the difference between a DEFAULT constraint and a CHECK constraint?
10. Explain the three-valued logic behavior of CHECK constraints in 60 seconds.

<details>
<summary>Answers</summary>

1. Domain integrity ensures that every column value belongs to the column's valid domain. It is enforced by: (1) data type — the most fundamental constraint, (2) NOT NULL — excludes the absence of a value, (3) CHECK constraint — boolean expression that must be true, and (4) DEFAULT constraint — provides a value when the column is omitted from INSERT.

2. A CHECK constraint evaluates NULL as UNKNOWN, not FALSE. Since UNKNOWN is accepted (the constraint passes), NULL values are allowed even if the CHECK constraint would reject them. For example, `CHECK (Quantity > 0)` allows NULL. To reject NULL, combine with `NOT NULL`: `Quantity INT NOT NULL CHECK (Quantity > 0)`.

3. A data type constraint restricts the set of valid values implicitly (INT accepts integers, DATE accepts dates, VARCHAR(50) accepts up to 50 characters). Conversion errors (error 245) occur when an incompatible value is supplied. A CHECK constraint adds an explicit boolean expression that every value must satisfy. The data type is the first filter; CHECK is a second filter applied after type conversion.

4. Error 547: "The INSERT statement conflicted with the CHECK constraint." The error message includes the constraint name. The transaction is rolled back if the constraint violation occurs within an implicit transaction, or the statement is aborted within an explicit transaction.

5. `entity.HasCheckConstraint("CK_Products_Price", "[Price] > 0")` in the `OnModelCreating` method. EF Core 5+ supports `HasCheckConstraint` which generates a `CHECK` constraint in the migration. Prior versions require manual SQL in migrations.

6. Dapper throws an `SqlException` with error 547 when a CHECK constraint is violated. Catch the exception, inspect `ex.Number == 547`, extract the constraint name from `ex.Message`, and map it to a user-friendly error.

7. `WITH CHECK` validates all existing rows against the constraint at the time of addition. It acquires an SCH-M lock on the table, blocking all reads and writes for the validation duration. `WITH NOCHECK` adds the constraint without validating existing rows (schema metadata only — fast, no blocking). Existing data may violate the constraint. Use `ALTER TABLE ... WITH CHECK CHECK CONSTRAINT name` later to validate existing rows.

8. Adding a `WITH CHECK` constraint to a table > 100 GB causes blocking proportional to the table scan time. On a 500 GB table with NVMe, this is ~2–5 minutes. On HDD, 10–30 minutes. Use `WITH NOCHECK` for such tables and validate in a maintenance window.

9. A DEFAULT constraint automatically supplies a value when the column is not included in an INSERT statement. It fires only on INSERT (never on UPDATE). A CHECK constraint validates that a supplied value meets a condition — it fires on both INSERT and UPDATE. They serve complementary roles: DEFAULT provides values, CHECK validates values.

10. [60-second spoken answer]: CHECK constraints use SQL's three-valued logic: TRUE, FALSE, and UNKNOWN. When the CHECked expression evaluates to TRUE, the value is accepted. When it evaluates to FALSE, the value is rejected with error 547. When it evaluates to UNKNOWN — which happens when any operand is NULL — the value is accepted. This means `CHECK (Quantity > 0)` does NOT reject NULL because `NULL > 0` is UNKNOWN, not FALSE. If the column should never have NULL, it must be defined as NOT NULL. This behavior catches many developers off guard: they write `CHECK (Quantity > 0)` thinking it also ensures a value is present. It does not. Always combine NOT NULL with CHECK when both are required. This is consistent across SQL Server, PostgreSQL, and MySQL — all standard SQL databases use three-valued logic for CHECK constraints.

</details>

**Challenge 1 — Write the SQL**

Create a table `Invoices` with the following domain integrity rules:
- InvoiceId: INT, identity, primary key
- InvoiceDate: DATE, NOT NULL, default to today
- DueDate: DATE, NOT NULL, must be >= InvoiceDate
- TotalAmount: DECIMAL(18,2), NOT NULL, must be > 0
- Status: CHAR(1), NOT NULL, default 'P', must be 'P' (Pending), 'S' (Sent), 'R' (Received), or 'C' (Cancelled)
- CurrencyCode: CHAR(3), NOT NULL, must be one of 'USD', 'EUR', 'GBP', 'JPY'

<details>
<summary>Solution</summary>

```sql
CREATE TABLE dbo.Invoices
(
    InvoiceId     INT             NOT NULL IDENTITY(1,1),
    InvoiceDate   DATE            NOT NULL DEFAULT CAST(SYSDATETIME() AS DATE),
    DueDate       DATE            NOT NULL,
    TotalAmount   DECIMAL(18,2)   NOT NULL,
    Status        CHAR(1)         NOT NULL DEFAULT 'P',
    CurrencyCode  CHAR(3)         NOT NULL,

    CONSTRAINT PK_Invoices PRIMARY KEY (InvoiceId),
    CONSTRAINT CK_Invoices_DueDate CHECK (DueDate >= InvoiceDate),
    CONSTRAINT CK_Invoices_TotalAmount CHECK (TotalAmount > 0),
    CONSTRAINT CK_Invoices_Status CHECK (Status IN ('P', 'S', 'R', 'C')),
    CONSTRAINT CK_Invoices_CurrencyCode CHECK (CurrencyCode IN ('USD', 'EUR', 'GBP', 'JPY'))
);
```

**Challenge 2 — Fix the performance problem**

```sql
-- A table has this CHECK constraint that calls a scalar UDF:
CREATE FUNCTION dbo.fn_IsValidProductCategory(@CategoryId INT)
RETURNS BIT
AS
BEGIN
    DECLARE @Result BIT = 0;
    IF EXISTS (SELECT 1 FROM dbo.ValidCategories WHERE CategoryId = @CategoryId)
        SET @Result = 1;
    RETURN @Result;
END;

ALTER TABLE dbo.Products ADD CONSTRAINT CK_Products_CategoryId
    CHECK (dbo.fn_IsValidProductCategory(CategoryId) = 1);

-- Inserts into Products are slow. A 10,000 row INSERT takes 30 seconds.
```

<details>
<summary>Solution</summary>

**Root cause:** The CHECK constraint calls a scalar UDF, which runs once per row. The UDF executes a SELECT on `ValidCategories` for each row — 10,000 rows = 10,000 individual queries. The UDF also prevents parallelism for the entire INSERT statement and forces serial execution.

**Fix: Replace the UDF-based CHECK constraint with a FOREIGN KEY constraint — that is what foreign keys are designed for:**

```sql
-- Remove the UDF-based CHECK
ALTER TABLE dbo.Products DROP CONSTRAINT CK_Products_CategoryId;

-- Add a proper foreign key constraint
ALTER TABLE dbo.Products ADD CONSTRAINT FK_Products_ValidCategories
    FOREIGN KEY (CategoryId) REFERENCES dbo.ValidCategories(CategoryId);

-- Foreign key validation uses an index seek — O(log N) instead of O(N) per row
```

**Expected improvement:** 10,000 row INSERT goes from 30 seconds to < 1 second. The foreign key constraint uses an index on `ValidCategories.CategoryId` for efficient validation.

**If a FK is not possible** (e.g., the valid list is dynamic and cannot be an FK target), rewrite the CHECK as a table-level constraint with an inline EXISTS (but note: this still has per-row overhead):

```sql
ALTER TABLE dbo.Products ADD CONSTRAINT CK_Products_CategoryId
    CHECK (EXISTS (SELECT 1 FROM dbo.ValidCategories vc WHERE vc.CategoryId = CategoryId));
```

**Lesson:** Never use scalar UDFs in CHECK constraints. The cost multiplies by the number of rows and disables parallelism.

</details>

**Challenge 3 — Explain the execution plan**

A CHECK constraint `CHECK (TotalAmount > 0)` on a 500 GB table was added with `WITH CHECK`. The ALTER TABLE statement took 8 minutes and caused 500 timeout errors across the application. Explain what happened and how to avoid it.

<details>
<summary>Solution</summary>

**What happened:** `ALTER TABLE ... WITH CHECK ADD CONSTRAINT` acquires a schema modification (SCH-M) lock on the table. This lock conflicts with every other query accessing the table — SELECT requires SCH-S (compatibility with SCH-S only). During the 8-minute validation (500 GB table scan), every query requiring any lock on Products was blocked. Applications with 30-second command timeouts timed out and raised errors.

**How to avoid it:** Use WITH NOCHECK to add the constraint without validation (metadata only, acquires SCH-M for milliseconds, not minutes). Then validate in a separate operation with a shorter lock scope:

```sql
-- Step 1: Add constraint without validation (fast, brief SCH-M)
ALTER TABLE dbo.Orders WITH NOCHECK
ADD CONSTRAINT CK_Orders_TotalAmount CHECK (TotalAmount > 0);

-- Step 2: Validate existing data in batches (low lock impact)
-- This does not acquire SCH-M — it checks the constraint bit
ALTER TABLE dbo.Orders WITH CHECK CHECK CONSTRAINT CK_Orders_TotalAmount;
-- If violations exist, the constraint is marked as untrusted
-- (is_not_trusted = 1 in sys.check_constraints)
```

If violations are expected:

```sql
-- Step 2 alternative: find and fix violations first
SELECT COUNT_BIG(*) FROM Orders WHERE TotalAmount <= 0;

-- Fix violations
UPDATE Orders SET TotalAmount = 0.01 WHERE TotalAmount <= 0;

-- Then validate
ALTER TABLE dbo.Orders WITH CHECK CHECK CONSTRAINT CK_Orders_TotalAmount;
```

**Cost of not knowing:** An 8-minute ALTER TABLE during business hours causes 500 application timeouts. Incident escalated to VP. Post-mortem identifies "no-maintenance-window deployment" as root cause.

</details>

**Challenge 4 — Diagnose the concurrency problem**

An application has a stored procedure that inserts an Order and then inserts OrderItems. The Order has a CHECK constraint `CK_Orders_TotalAmount CHECK (TotalAmount > 0)`. The application first inserts the Order with `TotalAmount = 0`, then updates it after computing item totals. The second INSERT into OrderItems triggers the Order update, and the CHECK constraint fails. Why?

<details>
<summary>Solution</summary>

**Root cause:** The CHECK constraint evaluates at statement time, not transaction commit time. The intermediate state (TotalAmount = 0) passes the INSERT because the CONSTRAINT at that point has NOT yet been checked.... Actually, `TotalAmount > 0` would FAIL when inserting 0.

Wait — re-read: "first inserts the Order with TotalAmount = 0" — this would fail the CHECK constraint `TotalAmount > 0` immediately. The INSERT would error 547.

But perhaps the constraint was added later, or the application expected the CHECK to be deferred. Let me re-phrase:

**Scenario:** The stored procedure sets TotalAmount = 0 initially with plans to update it later. But the database does not defer CHECK constraints — they are evaluated on every INSERT and UPDATE. The initial INSERT with TotalAmount = 0 fails.

**Fix:** Insert with a valid intermediate value, or set TotalAmount after OrderItems are inserted:

```sql
-- ❌ Fails: INSERT with TotalAmount = 0 violates CHECK
INSERT INTO Orders (TotalAmount) VALUES (0);

-- ✅ Works: Insert with minimum valid value, then update
INSERT INTO Orders (TotalAmount) VALUES (0.01);
-- Later: UPDATE Orders SET TotalAmount = @ComputedTotal WHERE OrderId = @OrderId;

-- ✅ Better: Defer the computation — use DEFAULT 0 but defer the CHECK
-- (not possible — CHECK is always evaluated)
-- Instead: allow 0 during insertion, but enforce > 0 at finalization
ALTER TABLE Orders DROP CONSTRAINT CK_Orders_TotalAmount;
ALTER TABLE Orders ADD CONSTRAINT CK_Orders_TotalAmount
    CHECK (TotalAmount >= 0);  -- allow 0 as intermediate state

-- Add a separate constraint for finalized orders
ALTER TABLE Orders ADD CONSTRAINT CK_Orders_FinalizedTotal
    CHECK (Status != 'F' OR TotalAmount > 0);
```

**Lesson:** CHECK constraints cannot be deferred in SQL Server. Every intermediate state of a row must satisfy all CHECK constraints at the time of writing. Design your data flow with this in mind.

</details>

**Challenge 5 — Design the domain integrity strategy**

**Scenario:** A SaaS platform allows customers to define custom fields for their products. Each custom field has a name, data type, and optional validation rules. The core `Products` table has 20 fixed columns with domain integrity constraints. The custom fields are stored in a JSON column in the `Products` table and in a parallel `CustomFieldValues` EAV table. Design a domain integrity strategy that ensures:
- Built-in columns are protected by database-level CHECK constraints
- Custom fields are validated at the application layer (the JSON schema varies per customer)
- Bulk imports (from customer CSV uploads) cannot insert invalid data
- Performance for the 20 fixed columns remains optimal

<details>
<summary>Solution</summary>

**Design: Two-layer domain integrity**

```sql
-- Layer 1: Database-level constraints on fixed columns
CREATE TABLE dbo.Products
(
    ProductId       INT             NOT NULL IDENTITY(1,1),
    ProductName     NVARCHAR(200)   NOT NULL,
    Price           DECIMAL(18,2)   NOT NULL,
    Quantity        INT             NOT NULL,
    CategoryId      INT             NOT NULL,
    Status          CHAR(1)         NOT NULL DEFAULT 'A',
    CustomFields    NVARCHAR(MAX)   NULL,  -- JSON for customer-defined fields
    CreatedDate     DATETIME2       NOT NULL DEFAULT SYSDATETIME(),

    CONSTRAINT PK_Products PRIMARY KEY (ProductId),
    CONSTRAINT CK_Products_Price CHECK (Price > 0),
    CONSTRAINT CK_Products_Quantity CHECK (Quantity >= 0),
    CONSTRAINT CK_Products_Status CHECK (Status IN ('A', 'I', 'D')),
    CONSTRAINT FK_Products_Category FOREIGN KEY (CategoryId)
        REFERENCES dbo.ProductCategories(CategoryId),

    -- JSON validation: ensure CustomFields is valid JSON
    CONSTRAINT CK_Products_CustomFields 
        CHECK (CustomFields IS NULL OR ISJSON(CustomFields) = 1)
);
```

```sql
-- Layer 2: Staging table for bulk imports with OPENJSON validation
-- (Customers upload CSV → staged in import table → validated → merged)

CREATE TABLE dbo.Products_Import
(
    ImportId        INT             NOT NULL IDENTITY(1,1),
    ProductName     NVARCHAR(200)   NOT NULL,
    Price           DECIMAL(18,2)   NOT NULL,
    Quantity        INT             NOT NULL,
    CategoryId      INT             NOT NULL,
    Status          CHAR(1)         NOT NULL DEFAULT 'A',
    CustomFieldsJson NVARCHAR(MAX)  NULL,
    ImportStatus    CHAR(1)         NOT NULL DEFAULT 'P',  -- P=pending, V=validated, E=error
    ValidationError NVARCHAR(MAX)   NULL,
    CreatedDate     DATETIME2       NOT NULL DEFAULT SYSDATETIME(),

    -- Same constraints as the target table
    CONSTRAINT CK_Products_Import_Price CHECK (Price > 0),
    CONSTRAINT CK_Products_Import_Quantity CHECK (Quantity >= 0),
    CONSTRAINT CK_Products_Import_Status CHECK (Status IN ('A', 'I', 'D')),
    CONSTRAINT CK_Products_Import_CustomFields
        CHECK (CustomFieldsJson IS NULL OR ISJSON(CustomFieldsJson) = 1)
);

-- Stored procedure validates and moves data
CREATE PROCEDURE dbo.usp_ValidateAndMergeProducts
AS
BEGIN
    SET NOCOUNT ON;

    -- Validate custom fields (application-specific JSON schema)
    UPDATE p
    SET ImportStatus = 'E',
        ValidationError = 'Custom field validation failed: ' + 
            dbo.fn_ValidateCustomFields(p.CustomFieldsJson)
    FROM dbo.Products_Import p
    WHERE p.ImportStatus = 'P'
        AND dbo.fn_ValidateCustomFields(p.CustomFieldsJson) IS NOT NULL;

    -- Merge validated rows
    INSERT INTO dbo.Products (ProductName, Price, Quantity, CategoryId, Status, CustomFields)
    SELECT ProductName, Price, Quantity, CategoryId, Status, CustomFieldsJson
    FROM dbo.Products_Import
    WHERE ImportStatus = 'P';

    -- Mark as validated
    UPDATE dbo.Products_Import
    SET ImportStatus = 'V'
    WHERE ImportStatus = 'P';
END;
```

**In .NET — FluentValidation for custom fields:**

```csharp
public class ProductImportValidator : AbstractValidator<ProductImportRow>
{
    private readonly ICustomFieldSchemaProvider _schemaProvider;

    public ProductImportValidator(ICustomFieldSchemaProvider schemaProvider)
    {
        _schemaProvider = schemaProvider;
        
        RuleFor(p => p.ProductName).NotEmpty().MaximumLength(200);
        RuleFor(p => p.Price).GreaterThan(0);
        RuleFor(p => p.Quantity).GreaterThanOrEqualTo(0);
        RuleFor(p => p.Status).Must(s => new[] { "A", "I", "D" }.Contains(s));
        
        // Custom field validation — per-tenant schema
        When(p => !string.IsNullOrEmpty(p.CustomFieldsJson), () =>
        {
            RuleFor(p => p.CustomFieldsJson).Must(json =>
            {
                var schema = _schemaProvider.GetSchema();
                return schema.Validate(json);
            }).WithMessage("Custom field validation failed");
        });
    }
}
```

**Tradeoffs:**
- Database constraints protect against ALL write paths (direct SQL, bulk import, API)
- Application validation provides better error messages and custom-field schema validation
- The staging table pattern prevents invalid bulk imports from corrupting the main table
- JSON schema validation is in the application, not the database (databases are bad at JSON validation at scale)

</details>
