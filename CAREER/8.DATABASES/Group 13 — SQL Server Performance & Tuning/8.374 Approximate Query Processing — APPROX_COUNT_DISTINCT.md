### Where This Fits

APPROX_COUNT_DISTINCT is an approximate query processing function introduced in SQL Server 2019 (compatibility level 130+). It implements the HyperLogLog algorithm to estimate the number of distinct values in a column with ~97% accuracy (1-2% error rate) using a fraction of the memory and CPU required by exact COUNT(DISTINCT column).

**Why it matters:** Exact distinct counts require sorting or hashing all values — memory proportional to the number of distinct values. For columns with millions of distinct values (clickstream user IDs, session IDs, IP addresses, product SKUs), this is expensive and can spill to TempDB. APPROX_COUNT_DISTINCT uses a fixed, small memory footprint (~1.5 KB for 64-register HyperLogLog) regardless of distinct count, with bounded error. It is ideal for dashboards, monitoring, and exploratory analytics where small imprecision is acceptable.

### Mental Model Analogy

Think of HyperLogLog like counting distinct people in a stadium by asking everyone to flip a coin and report the longest streak of heads they got. If someone says "10 heads in a row," there are probably a LOT of people. If the longest streak is 3, there are fewer. You lose individual identity but get a statistical estimate. APPROX_COUNT_DISTINCT does this with hashed values — each value's hash determines both which "coin-flip game" (register) it plays and the result.

## Section 3 — Deep Mechanics

### 3.1 HyperLogLog Algorithm (Step-by-Step)

**Step 1 — Hash Function:** Each distinct value is hashed to a 64-bit integer using a high-quality hash function.

**Step 2 — Register Selection:** The first p bits of the hash determine which register (bucket) the value belongs to. With p = 16, there are 2^16 = 65,536 registers.

**Step 3 — Leading Zero Count:** The remaining bits count the number of leading zeros (+1). For example:
- Hash = 0b0000 0101 1010 ... -> 5 leading zeros -> rank = 5
- Hash = 0b1011 0000 ... -> 0 leading zeros -> rank = 0

**Step 4 — Register Update:** For register i, maintain M[i] = max(M[i], rank/value). Each register stores a single value (max rank observed).

**Step 5 — Estimate Calculation:**
RawEstimate = alpha_m * m^2 * (1 / sum 2^(-M[j])) for j = 1...m
Where: m = number of registers (65536), alpha_m = bias correction constant (~0.7213 for m > 128)

**Step 6 — Bias Correction:** Apply small/large range corrections for edge cases (very small or very large cardinalities).

### 3.2 Execution Plan Analysis

\\\sql
-- Compare plans:
SET SHOWPLAN_XML ON;
GO
-- Exact plan
SELECT COUNT(DISTINCT ProductID) FROM Sales.SalesOrderDetail;
GO
-- Approximate plan
SELECT APPROX_COUNT_DISTINCT(ProductID) FROM Sales.SalesOrderDetail;
GO
SET SHOWPLAN_XML OFF;
\\\

**Exact plan operators:** Sort (distinct sort) or Hash Match (aggregate) -> Stream Aggregate -> Compute Scalar

**Approximate plan operators:** Hash Match (approximate distinct count) -> Compute Scalar

The approximate plan has NO Sort operator — the single biggest difference. No sort means no memory grant for sorting, no TempDB spills from sort overflow.

## Section 5 — Gotchas

### Gotcha 1: Low Accuracy for Small Distinct Counts

**Pitfall:** When a column has fewer than ~1,000 distinct values, APPROX_COUNT_DISTINCT can have error rates well above the advertised 1-2%. HyperLogLog is optimized for large cardinalities.

**Symptom:** APPROX_COUNT_DISTINCT(ProductCategoryID) returns 6 instead of 4 (50% error). The error is technically within algorithm bounds but practically unacceptable.

**Fix:** Don't use APPROX_COUNT_DISTINCT for columns with expected distinct counts < 1,000. Use COUNT(DISTINCT column) for small sets.

**Cost:** Low — conditional logic in queries.

### Gotcha 2: Error Is Random, Not Systematic

**Pitfall:** The error is random — it can overestimate or underestimate equally. A daily dashboard may show unique users fluctuating by 2% due to approximation, not actual behavior.

**Symptom:** Chart of daily unique users shows noise that masks real trends when actual count is stable but APPROX_COUNT_DISTINCT varies +/-2%.

**Fix:** Use a rolling average or apply smoothing to dashboard queries. Document that the metric is approximate.

**Cost:** Low — presentation layer change.

### Gotcha 3: APPROX_COUNT_DISTINCT Compared to COUNT(DISTINCT) with Filters

**Pitfall:** COUNT(DISTINCT CASE WHEN condition THEN col END) cannot be directly replaced with APPROX_COUNT_DISTINCT on conditional expressions.

**Symptom:** Syntax error when trying APPROX_COUNT_DISTINCT(CASE WHEN ...).

**Fix:** Use APPROX_COUNT_DISTINCT(col) with WHERE clause filtering, or use a subquery with WHERE.

**Cost:** Low — query rewrite.

### Gotcha 4: APPROX_COUNT_DISTINCT With GROUP BY and HAVING

**Pitfall:** HAVING APPROX_COUNT_DISTINCT(col) > N can be nondeterministic across executions since the approximation may fluctuate near the boundary.

**Symptom:** A daily report may include or exclude groups inconsistently.

**Fix:** For HAVING filters, use a larger safety margin or use exact count for the filter condition.

**Cost:** Medium — may need dual approach (exact filter, approximate display).

### Gotcha 5: Data Type Limitations

**Pitfall:** APPROX_COUNT_DISTINCT does not support all data types (e.g., text, ntext, image, xml, sql_variant).

**Symptom:** Error: "Argument data type text is not supported for APPROX_COUNT_DISTINCT."

**Fix:** Cast the column to varchar/nvarchar or use COUNT(DISTINCT CAST(col AS NVARCHAR(MAX))).

**Cost:** Low — CAST/convert.

### 7.2 Spoken Answers (Two-Tier)

**Q1: What is APPROX_COUNT_DISTINCT and what algorithm does it use?**

*Junior/Mid:* "APPROX_COUNT_DISTINCT is a SQL Server 2019 function that estimates distinct counts using the HyperLogLog algorithm. It's much faster and uses way less memory than COUNT(DISTINCT) because it doesn't need to sort or hash all values. It's about 97-99% accurate with a 1-2% error rate. You just replace COUNT(DISTINCT col) with APPROX_COUNT_DISTINCT(col) in your query."

*Senior/Lead:* "APPROX_COUNT_DISTINCT implements a 64-register variant of HyperLogLog, a probabilistic cardinality estimation algorithm. The key insight is that exact distinct counting is O(N) in both memory and CPU — SQL Server must sort or hash every distinct value. APPROX_COUNT_DISTINCT uses a fixed 1.5 KB hash table of 65,536 registers, mapping each value's hash to a register and tracking the maximum leading-zero count per register. The estimate is derived via harmonic mean with bias correction. For sets >100K distinct values, the error is consistently 1-2% with 97% confidence bounds, while memory savings are orders of magnitude. For a column with 10M distincts, we're talking ~41 GB potentially vs 1.5 KB fixed. The tradeoff is clear: for any high-cardinality distinct count in dashboards, monitoring, or analytics, APPROX_COUNT_DISTINCT is the right choice. For financial reporting or audit trails where exact counts are mandatory, use COUNT(DISTINCT)."

**Q2: Compare exact vs approximate distinct count in terms of execution plans.**

*Junior/Mid:* "With COUNT(DISTINCT), the plan shows a Sort operator or Hash Match aggregate. With APPROX_COUNT_DISTINCT, there's no sort at all — just a Hash Match that estimates distinct values. No sort means no TempDB spills."

*Senior/Lead:* "The execution plan differences are dramatic. COUNT(DISTINCT) typically produces a Sort -> Stream Aggregate pipeline, or a Hash Match (Aggregate) depending on the optimizer's choice. Both require workspace memory proportional to the distinct count. The Sort operator alone can require multiple times the distinct count in memory and spills to TempDB when exceeded. APPROX_COUNT_DISTINCT produces a Hash Match (Approximate Distinct Count) operator with a fixed memory footprint — approximately 1.5 KB plus minimal overhead. No Sort operator. No spill risk. The CPU difference is equally stark: I've measured 92% CPU reduction on large tables. The plan XML for APPROX_COUNT_DISTINCT contains a special element <ApproximateDistinctCount> with the algorithm type and register count."

HyperLogLog's error is proportional to 1/sqrt(m) where m = registers. With 65K registers, the algorithm needs sufficient values to populate them. With <1K distinct values, most registers are empty, reducing estimate quality.
</details>

<details>
<summary>10. In what production scenario would you NOT use APPROX_COUNT_DISTINCT?</summary>
Financial reporting, legal/audit compliance, inventory counts, regulatory reporting, any scenario requiring exact numbers.
</details>

-- Nightly validation procedure
CREATE PROCEDURE dbo.ValidateApproxCountAccuracy
AS
BEGIN
    WITH Comparison AS (
        SELECT
            COUNT(DISTINCT UserId) AS ExactCount,
            APPROX_COUNT_DISTINCT(UserId) AS ApproxCount
        FROM dbo.UserActivityLog
        WHERE EventDate >= CAST(GETUTCDATE() AS DATE)
    )
    SELECT
        ExactCount,
        ApproxCount,
        ABS(ExactCount - ApproxCount) AS AbsError,
        CASE
            WHEN ExactCount = 0 THEN 0
            ELSE (ABS(ExactCount - ApproxCount) * 1.0 / ExactCount) * 100
        END AS PctError,
        CASE
            WHEN ExactCount > 1000
             AND (ABS(ExactCount - ApproxCount) * 1.0 / ExactCount) * 100 > 5
            THEN 'ALERT: Error exceeds 5% threshold'
            ELSE 'OK'
        END AS AlertStatus
    FROM Comparison;
END;
