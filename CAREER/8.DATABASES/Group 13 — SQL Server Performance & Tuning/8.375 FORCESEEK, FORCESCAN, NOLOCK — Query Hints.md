### Where This Fits

Query hints are directives that override the SQL Server query optimizer's default behavior. They exist for cases where the optimizer makes a suboptimal choice due to inaccurate statistics, parameter sniffing, or complex query patterns. This file covers three categories:

1. **Access method hints** — FORCESEEK, FORCESCAN (FORCESEEK forces index seek; FORCESCAN forces index/table scan)
2. **Isolation level hints** — NOLOCK (READUNCOMMITTED), READCOMMITTED, READCOMMITTEDLOCK, READUNCOMMITTED, REPEATABLEREAD, SERIALIZABLE, SNAPSHOT
3. **Locking hints** — ROWLOCK, PAGLOCK, TABLOCK, TABLOCKX, UPDLOCK, XLOCK, HOLDLOCK, NOWAIT, READPAST

**Why it matters:** Query hints are double-edged swords. Used correctly, they fix specific performance problems. Used incorrectly, they cause regressions, blocking, or data corruption. Understanding each hint's behavior, side effects, and appropriate use cases is essential for senior-level troubleshooting.

|---|---|

| **FORCESCAN** | Forces index/table scan regardless of selectivity |

## Section 3 — Deep Mechanics

### 3.1 FORCESEEK Mechanics

FORCESEEK forces the optimizer to choose an index seek operator instead of a scan. It was introduced in SQL Server 2008.

**Syntax variants:**

\\\sql
-- Basic FORCESEEK (no index specified)
SELECT *
FROM Sales.SalesOrderDetail WITH (FORCESEEK)
WHERE ProductID = 870;

-- FORCESEEK with specific index
SELECT *
FROM Sales.SalesOrderDetail WITH (INDEX = IX_SalesOrderDetail_ProductID, FORCESEEK)
WHERE ProductID = 870;

-- FORCESEEK with specific index and key columns (SQL 2012+)
SELECT *
FROM Sales.SalesOrderDetail WITH (FORCESEEK, INDEX (IX_SalesOrderDetail_ProductID))
WHERE ProductID = 870
  AND CarrierTrackingNumber = '4911-403C-92';

-- FORCESEEK with column specification
SELECT *
FROM Sales.SalesOrderDetail WITH (FORCESEEK (IX_SalesOrderDetail_ProductID (ProductID)))
WHERE ProductID = 870;
\\\

**When FORCESEEK fails:**
- The specified index does not exist (error)
- The index cannot support the query predicates
- The hint conflicts with other query plans

\\\sql
-- This will fail if no index on Status exists
SELECT * FROM Sales.SalesOrderHeader WITH (FORCESEEK) WHERE Status = 5;
-- Error: Query optimizer could not produce a query plan because a FORCESEEK hint
-- was used on table without a suitable index.
\\\

**Plan impact:**
- FORCESEEK forces a Nonclustered Index Seek + possibly Key Lookup
- If the lookup is expensive, the seek may be slower than a scan

### 3.2 FORCESCAN Mechanics

FORCESCAN forces an index or table scan even if the optimizer would prefer a seek.

\\\sql
-- Force table scan (clustered index scan)
SELECT *
FROM Sales.SalesOrderDetail WITH (FORCESCAN)
WHERE ProductID = 870;

-- Force index scan with specific index
SELECT *
FROM Sales.SalesOrderDetail WITH (INDEX = IX_SalesOrderDetail_ProductID, FORCESCAN)
WHERE ProductID = 870;
\\\

**When FORCESCAN is useful:**
- Very large result sets (> 30% of rows)
- Queries where key lookups from seeks dominate the cost
- Parameter sniffing where sniffed value leads to seek but actual values should scan

### 3.3 NOLOCK / READUNCOMMITTED Mechanics

NOLOCK is the most misunderstood and overused hint in SQL Server. It applies the READ UNCOMMITTED isolation level to a specific table reference.

\\\sql
-- NOLOCK hint (table level)
SELECT *
FROM Sales.SalesOrderHeader WITH (NOLOCK)
WHERE OrderDate >= '2013-01-01';

-- READUNCOMMITTED hint (same as NOLOCK)
SELECT *
FROM Sales.SalesOrderHeader WITH (READUNCOMMITTED)
WHERE OrderDate >= '2013-01-01';
\\\

**What NOLOCK actually does:**
- Does NOT take shared (S) locks when reading
- Can read uncommitted data from in-progress transactions
- May skip pages, read pages twice, or encounter page splits mid-read
- Can cause 823 errors (data corruption at the read level, not persistent corruption)

**Isolation level comparison:**

| Level | Dirty Reads | Non-Repeatable Reads | Phantom Reads |
|---|---|---|---|
| READ UNCOMMITTED | Yes | Yes | Yes |
| READ COMMITTED (default) | No | Yes | Yes |
| READ COMMITTED (RCSI) | No | No | Yes |
| REPEATABLE READ | No | No | Yes |
| SERIALIZABLE | No | No | No |
| SNAPSHOT | No | No | No |

### 3.4 Locking Hints Mechanics

\\\sql
-- ROWLOCK: Take row-level locks (finer granularity)
SELECT *
FROM Sales.SalesOrderDetail WITH (ROWLOCK)
WHERE ProductID = 870;

-- PAGLOCK: Take page-level locks (medium granularity)
SELECT *
FROM Sales.SalesOrderDetail WITH (PAGLOCK)
WHERE ProductID = 870;

-- TABLOCK: Take table-level shared lock (coarsest granularity, minimal overhead)
SELECT *
FROM Sales.SalesOrderDetail WITH (TABLOCK)
WHERE ProductID = 870;

-- TABLOCKX: Take table-level exclusive lock
UPDATE Sales.SalesOrderDetail WITH (TABLOCKX)
SET ModifiedDate = GETDATE()
WHERE ProductID = 870;

-- UPDLOCK: Take update locks (prevents conversion deadlocks)
SELECT *
FROM Sales.SalesOrderDetail WITH (UPDLOCK)
WHERE ProductID = 870;

-- XLOCK: Take exclusive locks
SELECT *
FROM Sales.SalesOrderDetail WITH (XLOCK)
WHERE ProductID = 870;

-- HOLDLOCK: Hold locks until transaction end (equivalent to SERIALIZABLE)
SELECT *
FROM Sales.SalesOrderHeader WITH (HOLDLOCK)
WHERE SalesOrderID = 43659;

-- NOWAIT: Do not wait for locks, immediately error
SELECT *
FROM Sales.SalesOrderDetail WITH (NOWAIT)
WHERE ProductID = 870;

-- READPAST: Skip locked rows, return only available data
SELECT *
FROM Sales.SalesOrderHeader WITH (READPAST)
WHERE OrderDate >= '2013-01-01';
\\\

### 3.5 Combined Hints

\\\sql
-- Multiple hints combined (comma-separated)
SELECT *
FROM Sales.SalesOrderDetail WITH (NOLOCK, INDEX = IX_SalesOrderDetail_ProductID, FORCESEEK)
WHERE ProductID = 870;

-- UPDATE with multiple hints
UPDATE Sales.SalesOrderHeader WITH (ROWLOCK, UPDLOCK)
SET OrderDate = GETDATE()
WHERE SalesOrderID = 43659;

-- DELETE with hints
DELETE FROM Sales.SalesOrderDetail WITH (TABLOCK, XLOCK)
WHERE ProductID < 100;
\\\

\\\sql
-- Find queries using NOLOCK in plan cache
SELECT

### 3.7 Failure Modes

| Hint | Failure Mode | Symptom | Diagnosis |
|---|---|---|---|
| FORCESEEK | No suitable index | Error 8622 | Check index exists on predicate column |
| FORCESEEK | Wrong plan | Key lookup storm | Compare actual rows vs estimated |
| FORCESCAN | Scan is worse | High logical reads | Check selectivity |
| NOLOCK | Dirty read | Wrong results | Read uncommitted transaction |
| NOLOCK | 823 error | Database consistency check error | Page split during read |
| NOWAIT | Lock contention | Error 1222 (Lock request time out) | Frequent blocking |
| TABLOCK | Concurrency drop | Blocking chains | Many concurrent writers |

## Section 4 — Production Patterns

### 4.1 FORCESEEK for Known Selective Predicate

\\\sql
-- When statistics are stale but you know the predicate is highly selective
-- Bad optimizer choice: Clustered Index Scan (240,000 logical reads)
-- Forced fix: Index Seek + Key Lookup (1,200 logical reads)

SELECT
    SalesOrderID,
    OrderDate,
    TotalDue,
    CustomerID
FROM Sales.SalesOrderHeader WITH (FORCESEEK)
WHERE SalesOrderID BETWEEN 43659 AND 43700;
\\\

### 4.2 FORCESEEK with Specific Index Columns

\\\sql
-- Force seek on specific index columns to avoid key lookup if all needed columns covered
SELECT
    SalesOrderID,
    ProductID,
    OrderQty,
    UnitPrice
FROM Sales.SalesOrderDetail WITH (FORCESEEK, INDEX (IX_SalesOrderDetail_ProductID))
WHERE ProductID = 870
  AND UnitPrice > 100;
\\\

### 4.3 FORCESCAN for Low Selectivity Queries

\\\sql
-- When optimizer chooses a seek + lookup that's worse than a scan
-- Bad: Seek returns 30% of rows -> 30% lookups -> expensive
-- Fix: Force scan

SELECT
    ProductID,
    SUM(LineTotal) AS TotalSales
FROM Sales.SalesOrderDetail WITH (FORCESCAN)
WHERE ProductID IN (707, 708, 709, 710, 711)
GROUP BY ProductID;
\\\

### 4.4 NOLOCK for Reporting / Historical Queries

\\\sql
-- WARNING: NOLOCK can read uncommitted data, skip rows, or cause 823 errors
-- Use only for historical reporting where approximate results are acceptable

SELECT
    CustomerID,
    COUNT(*) AS OrderCount,
    SUM(TotalDue) AS TotalSpent
FROM Sales.SalesOrderHeader WITH (NOLOCK)
WHERE OrderDate < '2014-01-01'
GROUP BY CustomerID
ORDER BY TotalSpent DESC;
\\\

### 4.5 READPAST for Queue Processing

\\\sql
-- Skip locked rows in queue processing pattern
-- Each worker picks the next available (non-locked) row

BEGIN TRANSACTION;

SELECT TOP 1
    QueueId,
    Payload,
    CreatedDate
FROM dbo.ProcessingQueue WITH (READPAST, UPDLOCK, ROWLOCK)
WHERE Status = 'Pending'
ORDER BY CreatedDate ASC;

-- Process the item...
-- UPDATE Status = 'Processing', LockToken = @token

COMMIT TRANSACTION;
\\\

### 4.6 UPDLOCK for Deadlock Prevention

\\\sql
-- Prevent deadlock by taking update lock instead of shared -> update conversion
BEGIN TRANSACTION;

-- Step 1: Read with UPDLOCK (no shared lock)
SELECT Quantity, BinLocation
FROM dbo.Inventory WITH (UPDLOCK, ROWLOCK)
WHERE ProductID = 870;

-- Step 2: Update (already have UPDLOCK, no conversion deadlock)
UPDATE dbo.Inventory
SET Quantity = Quantity - 1
WHERE ProductID = 870;

COMMIT TRANSACTION;
\\\

### 4.7 READCOMMITTEDLOCK for RCSI Databases

\\\sql
-- In a database with READ_COMMITTED_SNAPSHOT = ON,
-- force the old-style READ COMMITTED (blocking) behavior
-- for a specific statement

SELECT OrderDate, TotalDue
FROM Sales.SalesOrderHeader WITH (READCOMMITTEDLOCK)
WHERE SalesOrderID = 43659;
\\\

### 4.8 Entity Framework Core / Dapper Hints

**EF Core Interceptor for NOLOCK or other hints:**

\\\csharp
public class QueryHintInterceptor : DbCommandInterceptor
{
    private readonly string _hint;

    public QueryHintInterceptor(string hint = "READUNCOMMITTED")
    {
        _hint = hint;
    }

    public override ValueTask<InterceptionResult<DbDataReader>> ReaderExecutingAsync(
        DbCommand command,
        CommandEventData eventData,
        InterceptionResult<DbDataReader> result,
        CancellationToken cancellationToken = default)
    {
        // Inject table hints for queries with specific table names
        foreach (SqlParameter param in command.Parameters)
        {
            if (param.ParameterName == "@QueryHint")
            {
                // Use query tags to mark queries for hint injection
                // e.g., EF.Functions.TagWith("-- NOLOCK")
                if (command.CommandText.Contains("-- NOLOCK"))
                {
                    // Inject WITH (NOLOCK) after each table reference
                    command.CommandText = System.Text.RegularExpressions.Regex.Replace(
                        command.CommandText,
                        @"\b(Sales\.\w+)\b(?!\s*WITH)",
                        " WITH (NOLOCK)",
                        System.Text.RegularExpressions.RegexOptions.IgnoreCase);
                }
            }
        }
        return base.ReaderExecutingAsync(command, eventData, result, cancellationToken);
    }
}
\\\

## Section 5 — Gotchas

### Gotcha 1: NOLOCK Can Cause 823 Errors (Not Just Dirty Reads)

**Pitfall:** NOLOCK doesn't just read uncommitted data — it can cause 823 I/O errors when reading a page that is being split by another transaction.

**Symptom:** Intermittent error 823 in error log. You check database integrity (DBCC CHECKDB) and find no corruption. Restarting doesn't help.

**Fix:** Remove NOLOCK hints. Use RCSI (READ_COMMITTED_SNAPSHOT) instead for reporting workloads. RCSI provides consistent reads without blocking writers.

**Cost:** Medium — requires application change and enabling RCSI (which increases TempDB usage).

### Gotcha 2: FORCESEEK Can Cause Massive Key Lookup Storm

**Pitfall:** FORCESEEK forces a seek even when the optimizer would correctly choose a scan. If the seek returns many rows (e.g., 30%+ of the table), each row needs a key lookup, causing massive logical reads.

**Symptom:** Query that ran in 2 seconds now runs in 2 minutes after adding FORCESEEK. Logical reads skyrocket.

**Fix:** Remove FORCESEEK. Consider covering index or included columns to eliminate lookups if seek is genuinely beneficial.

**Cost:** Low — remove the hint.

### Gotcha 3: NOLOCK + INSERT/UPDATE/DELETE = Data Corruption

**Pitfall:** Writing with NOLOCK is irrelevant — NOLOCK only affects reads. But using NOLOCK in a transaction that also writes can cause logical data inconsistencies.

**Symptom:** A report that aggregates data with NOLOCK while an ETL process is running shows different totals on page 1 vs page 2.

**Fix:** Never use NOLOCK for any query that feeds into a financial/operational report. Use RCSI or SNAPSHOT isolation for consistency.

**Cost:** High — data reconciliation effort.

### Gotcha 4: FORCESEEK + Incorrect Index = Error

**Pitfall:** FORCESEEK with INDEX hint specifying a non-existent or inappropriate index causes error 8622.

**Symptom:** Query fails after index rename or rebuild with a different structure.

**Fix:** Use FORCESEEK without a specific index name, or verify index exists before deploying hints.

**Cost:** Low — deployment-time validation.

### Gotcha 5: TABLOCK Hurts Concurrency More Than You Think

**Pitfall:** Adding TABLOCK to a SELECT in a busy OLTP system blocks all writers for the duration of the read.

**Symptom:** Application timeouts, blocked processes showing wait type LCK_M_S.

**Fix:** Use NOLOCK (if acceptable) or RCSI instead of TABLOCK for reporting. For bulk operations, use TABLOCK intentionally but briefly.

**Cost:** Medium — concurrency redesign.

### Gotcha 6: READUNCOMMITTED vs NOLOCK — Exactly the Same

**Pitfall:** Some DBAs think READUNCOMMITTED is "safer" than NOLOCK. They are identical synonyms.

**Symptom:** No difference — same dirty reads, same 823 risks.

**Fix:** Use whichever is clearer in your codebase. Neither is safer.

**Cost:** None.

### 6.1 SET STATISTICS IO / TIME Comparison

\\\sql

### 7.2 Spoken Answers (Two-Tier)

**Q1: What does FORCESEEK do and when would you use it?**

*Junior/Mid:* "FORCESEEK tells SQL Server to use an index seek operation instead of a scan. I'd use it when I know a query should use a seek but the optimizer is choosing a scan — usually because of stale statistics. I just add WITH (FORCESEEK) to the table reference."

*Senior/Lead:* "FORCESEEK overrides the optimizer's access method choice, forcing it to produce an Index Seek operator. I use it sparingly and with a specific index and column specification. The primary use case is when statistics are stale or cannot capture correlation — for example, a compound predicate where column A = X AND column B = Y, and the optimizer underestimates the combined selectivity. FORCESEEK with index specification avoids the key lookup overhead by ensuring the index covers the query. However, I always test both FORCESEEK and the default plan because a seek + lookup on 30%+ of rows is worse than a clustered index scan. The real fix is usually index tuning or statistics maintenance, not hints."

**Q2: Why is NOLOCK dangerous?**

*Junior/Mid:* "NOLOCK lets you read data without taking locks, so you avoid blocking. But it can read uncommitted data (dirty reads). It's okay for rough estimates or historical data."

*Senior/Lead:* "NOLOCK is dangerous because it causes three categories of issues: (1) Dirty reads — reading uncommitted data that may be rolled back. (2) Data movement errors — 823 errors when reading a page mid-split, which looks like corruption but isn't. And 829 errors (move/read consistency error). (3) Non-repeatable reads within a single scan — since NOLOCK doesn't maintain read stability, a page read twice may show different data. The most insidious issue is the 823 error: DBAs panic, run DBCC CHECKDB (which passes), but the application got an error. Users see 823 in the error log and assume corruption. The real fix is to enable RCSI (Read Committed Snapshot Isolation) for reporting databases, which gives you statement-level read consistency without blocking writers. RCSI uses TempDB row versioning and is much safer than NOLOCK."

### 7.3 Comparison Table

| Hint | Purpose | Performance Impact | Risk Level |
|---|---|---|---|
| **FORCESEEK** | Force index seek | Mixed (good for selective, bad for unselective) | Medium |
| **FORCESCAN** | Force scan | Good for low selectivity, bad for high | Low |
| **NOLOCK** | Avoid read locks | No blocking | High (data integrity) |
| **READCOMMITTED** | Default behavior | Standard blocking | None |
| **READPAST** | Skip locked rows | Concurrency without errors | Low |
| **NOWAIT** | Immediate error on lock | Best effort | Medium (errors) |
| **UPDLOCK** | Prevent conversion deadlock | Slightly more locking | Low |
| **XLOCK** | Exclusive lock | Blocks all | High |
| **TABLOCK** | Table-level lock | Minimal lock overhead | High (concurrency) |
| **ROWLOCK** | Row-level lock | Max concurrency | Low (more lock memory) |

    D -->|Small result set| J[Optimizer probably correct<br/>Check for lookups]

    E -->|Read-only reporting| K{RCSI Available?}
    E -->|Queue processing| L[READPAST + UPDLOCK + ROWLOCK]
    E -->|Deadlock prevention| M[UPDLOCK for read-before-write]

    K -->|Yes| N[Use RCSI at DB level<br/>No hints needed]
    K -->|No| O{Approximate results OK?}
    O -->|Yes| P[NOLOCK (document risks)]
    O -->|No| Q[READCOMMITTED with<br/>timeout handling]

    F --> R[Monitor key lookups]
    R -->|Too many lookups| S[Add covering index<br/>or included columns]
    R -->|OK| T[Keep FORCESEEK]

    P --> U[WARNING: 823 errors possible]
    L --> V[Monitor for blocking]
\\\

### Decision Checklist

- [ ] Have statistics been updated recently? (Outdated stats are the #1 reason for wrong access methods)
- [ ] Is the predicate selective enough for FORCESEEK? (Generally < 10% of rows)
- [ ] Will FORCESEEK cause excessive key lookups? (Check the execution plan)
- [ ] Is the covering index available? (Index with included columns reduces lookup cost)
- [ ] Is NOLOCK absolutely necessary? (RCSI is safer — enable at database level)
- [ ] Are you aware of 823 error risk with NOLOCK?
- [ ] Have you tested with AND without the hint?
- [ ] Is there a schema-bound plan for hint removal when the root cause is fixed?
- [ ] For locking hints: is there a documented pattern (queue, bulk, etc.)?
- [ ] Are hints documented with rationale in comments?

### Scale Thresholds

| Factor | Threshold | Action |
|---|---|---|
| Predicate selectivity | < 5% | FORCESEEK beneficial |
| Predicate selectivity | 5-30% | Test both, update stats first |
| Predicate selectivity | > 30% | FORCESEEK likely harmful |
| Key lookup count | > 50% of rows | FORCESEEK harmful without covering index |
| Blocking duration | > 1 second per query | Consider RCSI or NOLOCK (with docs) |
| NOLOCK error rate | Any 823 error | Remove NOLOCK immediately, enable RCSI |
| Deadlock frequency | > 5/day | Review locking order, use UPDLOCK |
| Queue length | > 10,000 items | READPAST + UPDLOCK pattern |
| Concurrency | > 100 simultaneous | Avoid TABLOCK, use ROWLOCK |

### Tradeoff Summary

**For hints:** Precise control over optimizer decisions; immediate fix for specific regressions; no schema changes needed.

**Against hints:** Bypasses the optimizer's cost-based decisions; can become stale if data distribution changes; hard to maintain across versions; masks underlying issues (stats, indexing, query design).

## Section 9 — Self-Check

### 9.1 Conceptual Questions

<details>
<summary>1. What does FORCESEEK force the optimizer to do?</summary>
Force an index seek operator instead of a scan. If no suitable index exists, the query fails with error 8622.
</details>

<details>
<summary>2. What is the difference between NOLOCK and READUNCOMMITTED?</summary>
They are identical synonyms. Both apply the READ UNCOMMITTED isolation level to the table.
</details>

<details>
<summary>3. Can NOLOCK cause errors besides dirty reads?</summary>
Yes. NOLOCK can cause 823 I/O errors (page split during read) and 829 errors (move/read consistency). These are not persistent corruption but appear in logs.
</details>

<details>
<summary>4. What is the READPAST hint used for?</summary>
Queue processing patterns. READPAST skips locked rows and returns only available (unlocked) data, enabling multiple workers to consume from a queue simultaneously.
</details>

<details>
<summary>5. How does UPDLOCK prevent deadlocks?</summary>
UPDLOCK takes an update lock during the read phase. This prevents a lock conversion deadlock where two transactions hold shared locks and both try to upgrade to exclusive.
</details>

<details>
<summary>6. What happens when TABLOCK is used on a SELECT?</summary>
A shared (S) lock is taken at the table level, preventing any concurrent DML (INSERT/UPDATE/DELETE) on the table until the transaction completes.
</details>

<details>
<summary>7. What is the correct syntax for FORCESEEK with a specific index?</summary>
WITH (FORCESEEK, INDEX (index_name)) or WITH (INDEX = index_name, FORCESEEK).
</details>

<details>
<summary>8. Can multiple table hints be used together?</summary>
Yes, comma-separated: WITH (NOLOCK, INDEX = IX_Name, FORCESEEK).
</details>

<details>
<summary>9. What is the difference between NOWAIT and READPAST?</summary>
NOWAIT immediately errors (1222) on lock conflict. READPAST silently skips locked rows and returns only available data.
</details>

<details>
<summary>10. Which hint prevents lock escalation?</summary>
ROWLOCK hints SQL Server to take row-level locks, reducing the likelihood of escalation to page or table locks. However, SQL Server may still escalate under memory pressure.
</details>

<summary>Challenge 4: Write a query that uses FORCESCAN and explain when this is the right choice despite higher logical reads.</summary>

\\\sql
-- FORCESCAN is right when the query returns a large percentage of rows
-- and the optimizer's seek + lookup is more expensive than a full scan
SELECT ProductID, SUM(LineTotal) AS TotalSales

<details>
<summary>Challenge 5: Design a stored procedure monitoring query that identifies all queries in the plan cache using NOLOCK or FORCESEEK and returns their execution stats.</summary>

\\\sql
SELECT
    qs.execution_count,
    qs.total_worker_time / 1000000.0 AS total_worker_sec,
    qs.total_logical_reads / 1000.0 AS total_logical_reads_k,
    qs.total_elapsed_time / 1000000.0 AS total_elapsed_sec,
    qs.last_execution_time,
    CASE
        WHEN st.text LIKE '%NOLOCK%' THEN 'NOLOCK'
        WHEN st.text LIKE '%READUNCOMMITTED%' THEN 'READUNCOMMITTED'
        WHEN st.text LIKE '%FORCESEEK%' THEN 'FORCESEEK'
        WHEN st.text LIKE '%FORCESCAN%' THEN 'FORCESCAN'
        WHEN st.text LIKE '%TABLOCK%' THEN 'TABLOCK'
        WHEN st.text LIKE '%READPAST%' THEN 'READPAST'
        WHEN st.text LIKE '%UPDLOCK%' THEN 'UPDLOCK'
        ELSE 'OTHER'
    END AS hint_type,
    SUBSTRING(st.text, 1, 500) AS query_text
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
WHERE st.text LIKE '%WITH (%'
   OR st.text LIKE '%NOLOCK%'
   OR st.text LIKE '%FORCESEEK%'
   OR st.text LIKE '%FORCESCAN%'
   OR st.text LIKE '%TABLOCK%'
   OR st.text LIKE '%READPAST%'
   OR st.text LIKE '%UPDLOCK%'
ORDER BY qs.total_worker_time DESC;
\\\
