**Where This Fits:** The graphical execution plan is the primary diagnostic tool for query performance. SSMS renders the internal ShowPlanXML document as an interactive tree of icons and tooltips. Reading plans right-to-left, top-to-bottom, interpreting arrow thickness, recognizing operator icons, and spotting warning icons are foundational skills for every performance tuning exercise. This note covers the visual language of graphical plans — how to read them, what to look for, and how to identify common performance issues at a glance.

## Section 2 — Core Mental Model

`mermaid
graph TD
    subgraph "Plan Reading Direction"
        direction LR
        R1[Rightmost: Leaf Ops] --> R2[Index/Table Scans]
        R2 --> R3[Join Ops]
        R3 --> R4[Compute Scalar / Filter]
        R4 --> R5[Top / Sort]
        R5 --> R6[Leftmost: SELECT]
    end
    subgraph "Operator Icons Quick Ref"
        I1[🔍 Seek] --> I2[📄 Scan]
        I2 --> I3[🔗 NL Join]
        I3 --> I4[△ Hash Join]
        I4 --> I5[‖ Merge Join]
        I5 --> I6[🔑 Key Lookup]
    end
`

**Explanation:** Execution plans are read from right to left — data flows from leaf operators (scans/ seeks) upward and leftward through join, filter, and aggregation operators to the final SELECT. Each operator icon encodes the physical algorithm used. Arrow thickness between operators is proportional to row count relative to other arrows in the same plan. Tooltips contain estimated vs actual rows, costs, and execution counts.

### Mental Model — The Three Checks

| Check | What to Look For | Why It Matters |
|-------|-----------------|----------------|
| Arrow thickness | Thick arrows entering Nested Loops = outer estimate may be wrong | Under-estimated outer input causes 1000x more inner executions |
| Cost percentages | Single operator consuming >70% of plan cost | Focus tuning effort on highest-cost operator |
| Warning icons (yellow triangle) | Missing stats, spills, implicit conversions, no join predicate | These indicate sub-optimal conditions the optimizer identified |

**Key Insight:** The graphical plan is a visualization of the cost model. The optimizer chose this plan because it estimated these operators would be cheapest. If the estimates are wrong (bad stats or CE), the plan will be wrong — and that is visible in the arrow thickness mismatch.

### RelOp Mapping in the Visual Tree

Every box in the graphical plan corresponds to a `<RelOp>` element in ShowPlanXML. The icon is determined by the `@PhysicalOp` attribute:

| PhysicalOp | Icon (SSMS) | Visual Shape |
|---|---|---|
| Index Seek | Magnifying glass over data page | Single page with magnifier |
| Clustered Index Scan | Data pages with key icon | Three cylinders + key |
| Table Scan | Three cylinders (no key) | Plain cylinders |
| Nested Loops | Two circles joined by arrow | Circles with arrow |
| Hash Match | Interleaved triangles | Venn overlap |
| Merge Join | Parallel vertical lines | Two bars with arrows |
| Key Lookup | Key with arrow pointing right | Key + right arrow |
| Sort | A-Z with arrow | Letters with arrow |
| Stream Aggregate | Single cylinder with sigma | Cylinder with Σ |
| Compute Scalar | Calculator icon | Small calculator |
| Filter | Funnel icon | Funnel |
| Spool (Table/Index) | Spool/scroll icon | Scroll |
| Top | Top N icon | "Top" label with number |
| Sequence Project | Windows logo (window pane) | Segmented window |
| Assert | Green checkmark / shield | Shield / check |
| Parallelism (Distribute Streams) | Two arrows splitting | Fork icon |
| Parallelism (Gather Streams) | Two arrows merging | Merge icon |
| Bitmap | Honeycomb pattern | Grid dots |
| Adaptive Join | Three vertical dots + join | Dots + join shape |

### Tooltip Field Reference

Hovering over an operator shows the tooltip with these fields (title case as shown in SSMS):

| Field | Meaning | Example Value |
|---|---|---|
| Physical Operation | Physical algorithm used | Index Seek (NonClustered) |
| Logical Operation | Logical operation | Index Seek |

| Estimated Data Size | Total data size | 52340 * 67 = 3.5 MB |
| Estimated Operator Cost | This operator's cost | 0.150 (cost units) |
| Estimated Subtree Cost | Cumulative cost of this op + children | 1.234 |

| Ordered | Output preserves sort order? | True / False |
| Parallel | Executes in parallel? | False |
| Partitioned | Scans partitions? | False |
| Actual Rebinds | Number of full resets (NL inner) | 0 |
| Actual Rewinds | Re-executions without reset | 0 |

### Arrow Thickness & Color Coding

- **Thickness** is relative to the entire plan — a thin arrow may still represent 10K rows if the plan processes millions
- **Green / Blue arrows** = row mode (row-by-row processing)
- **Red arrows** = presence of warnings or issues
- **Dashed arrows** = estimated rows only (estimated plan)
- **Solid arrows** = actual rows (actual plan)

### Warning Icons Reference

| Warning | Icon | Cause |
|---|---|---|
| Missing Column Statistics | Yellow triangle on operator | No statistics on predicate column |
| Spill To TempDb | Yellow triangle on Sort/Hash | Memory insufficient for sort/join |
| Implicit Conversion | Yellow triangle on Scan/Seek | Data type mismatch on predicate |
| No Join Predicate | Yellow triangle on Join | Cartesian product join |
| Excessive Memory Grant | Yellow triangle on QueryPlan | Memory grant >> actual usage |
| Convert Issue | Yellow triangle on Scan | Non-SARGable predicate |

## Section 4 — Production Patterns

### Pattern 1: Spotting the Root Cause in 30 Seconds

When given a plan you have never seen, follow this visual scan:

1. **Rightmost operators first** — Identify scan/seek operators. A Clustered Index Scan or Table Scan with a filter is your first target.
2. **Find the thickest arrows** — The thickest arrow between any two operators indicates the largest data flow. If this data flows into a Nested Loops join, that join is executing many times.
3. **Find the highest cost percentage** — SSMS highlights the most expensive operator. Right-click → Properties to drill in.
4. **Look for warning triangles** — Yellow triangles immediately tell you something is sub-optimal.
5. **Check the Nested Loops inner** — If Nested Loops is present, check the inner side. If the inner side shows a scan (not seek), the join is not supported by an index.

### Pattern 2: Missing Index Detection from Visual Plan

The graphical plan shows a green text banner at the top: "Missing Index (Impact X%)". The plan XML also contains the `<MissingIndexes>` element. When you see:

- A **Key Lookup** operator (key icon with arrow) followed by a Nested Loop to the clustered index
- The Key Lookup shows "Actual Number of Rows" in the thousands or more
- The non-clustered index seek is narrow (few columns)

**Action:** Create an index with INCLUDE columns to cover the SELECT list.

### Pattern 3: Detecting Parameter Sniffing from Plan Operator Mismatch

If the estimated plan (Ctrl+L) shows different operators than the actual plan (Ctrl+M), the optimizer made different decisions during compilation vs what was actually needed:

1. Estimated plan might show Nested Loops (low estimate)
2. Actual plan shows Hash Join (high actual rows)
3. Root cause: Parameter sniffing — the plan was compiled for a parameter value returning few rows, but the actual parameter value returns many rows

### Pattern 4: Identifying Plan Instability Across Executions

Compare two actual plans from the same query at different times:
- Different join types → CE instability or parameter sniffing
- Same join but different order → statistics changes or data distribution changes
- Different memory grant → CE estimate changed between executions
- Different parallelism degree → plan recompilation with different DOP

### Pattern 5: EF Core — Mapping LINQ to Plan Operators

When debugging an EF Core query that appears slow:

1. Capture the actual plan from the SQL output in SSMS
2. Look for **Table Spool** operators — these often appear when EF's query pipeline produces nested subqueries
3. Look for multiple **Compute Scalar** operators producing the same expressions — indicates repeated materialization
4. Look for **Key Lookup** on tables that should have covering indexes
5. High number of **Nested Loops** joins compared to the number of Include() calls in LINQ

`csharp
// In EF Core, add an interceptor to capture plans:
public class PlanCaptureInterceptor : DbCommandInterceptor
{
    private static int _queryId = 0;
    
    public override ValueTask<InterceptionResult<DbDataReader>> ReaderExecutingAsync(
        DbCommand command, CommandEventData eventData, 
        InterceptionResult<DbDataReader> result, CancellationToken ct = default)
    {
        var id = Interlocked.Increment(ref _queryId);
        Debug.WriteLine($"--- Query #{id}: {command.CommandText}");
        return base.ReaderExecutingAsync(command, eventData, result, ct);
    }
}
`

### Pattern 6: Operator Count as Complexity Metric

Count the operators in a plan to assess tuning priority:

| Operator Count | Complexity | Tuning Approach |
|---|---|---|
| 1-10 | Simple | Full manual plan review |
| 11-30 | Moderate | Focus on top 5 by cost, check for spills |
| 31-100 | Complex | Use XML analysis, focus on cardinality mismatches |
| 100+ | Very complex | Consider query rewrite; extract subqueries into temp tables |

### Pattern 7: Visual Detection of Memory Grant Issues

When hovering over the root SELECT operator:
- **MemoryGrant** > 1GB and plan is OLTP → likely over-allocation
- **MemoryGrant** > 1GB with spills → under-allocation (estimate too low)
- **GrantWaitTime** > 0ms → query waited for memory (concurrency issue)

The root operator tooltip shows these values. A red X icon on the root operator indicates memory grant wait.

## Section 5 — Gotchas

### Gotcha 1 — Cost Percentages Are Estimated, Not Actual
**Pitfall:** You focus optimization on an operator showing 80% cost but the actual bottleneck is elsewhere.
**Symptom:** Adding an index for the high-cost operator has no effect on actual duration.
**Fix:** Always verify with SET STATISTICS PROFILE ON for actual row counts.
**Cost:** Hours of wasted optimization.

### Gotcha 2 — Parallelism Costs Are Misleading in Serial Plans
**Pitfall:** A plan showing low CPU cost may be serial when a parallel plan would be faster.
**Symptom:** Table scan with 0.5% cost but query takes 30 seconds.
**Fix:** Check DOP in plan properties. Test with OPTION (MAXDOP 0).
**Cost:** Production queries running serially that would benefit from parallelism.

### Gotcha 3 — Arrow Thickness Is Relative to the Whole Plan
**Pitfall:** A thin arrow might still represent 10K rows if the plan processes millions.
**Symptom:** You ignore an operator because its arrow is thin, but it is actually expensive due to multiple executions.
**Fix:** Check the actual tooltip row counts and Number of Executions.
**Cost:** Missed tuning opportunity costing 1000+ logical reads per execution.

### Gotcha 4 — Estimated Plan Shows Different Operators Than Actual Plan
**Pitfall:** The estimated plan (Ctrl+L) may never match the actual plan due to parameter sniffing.
**Symptom:** You optimize the estimated plan, deploy, and see no improvement.
**Fix:** Always use the actual execution plan (Ctrl+M) from a representative execution.
**Cost:** Deploying ineffective optimizations.

### Gotcha 5 — Nested Loops Inner Side Cost Is Misleading
**Pitfall:** The inner side of a Nested Loops join shows low per-execution cost but is executed 1M+ times.
**Symptom:** An Index Seek with 0.01% cost on a nested loop inner side is actually 1M * 0.01%.
**Fix:** Check Number of Executions. Multiply operator cost by executions.
**Cost:** Missed 90% optimization opportunities.

### Gotcha 6 — Warnings in Plans Are Easy to Miss
**Pitfall:** SSMS shows warnings as yellow triangles on operators, but they are small.
**Symptom:** A Column No Stats warning on a join predicate causes a 0-row estimate.
**Fix:** Enable Show Warnings in SSMS plan settings. Run sys.dm_db_stats_properties.
**Cost:** Wrong join type chosen causing 100x degradation.

## Section 6 — Performance Implications

### Demo: Reading a Plan with a Sub-Optimal Join

`sql
CREATE TABLE #LargeOrderHistory (OrderId INT PRIMARY KEY, CustomerId INT, TotalAmount DECIMAL(10,2), OrderDate DATETIME);
CREATE TABLE #CustomerPreferences (CustomerId INT PRIMARY KEY, PreferredCategory VARCHAR(50), LastPurchaseDate DATETIME);

WITH CTE AS (SELECT TOP 500000 ROW_NUMBER() OVER (ORDER BY (SELECT NULL)) AS N FROM sys.all_columns a CROSS JOIN sys.all_columns b)
INSERT INTO #LargeOrderHistory (OrderId, CustomerId, TotalAmount, OrderDate)
SELECT N, 1 + (N % 50000), RAND(CHECKSUM(NEWID())) * 1000, DATEADD(DAY, -N % 365, '2026-06-28') FROM CTE;

WITH CTE AS (SELECT TOP 50000 ROW_NUMBER() OVER (ORDER BY (SELECT NULL)) AS N FROM sys.all_columns)
INSERT INTO #CustomerPreferences (CustomerId, PreferredCategory, LastPurchaseDate)
SELECT N, CASE WHEN N % 3 = 0 THEN 'Electronics' WHEN N % 3 = 1 THEN 'Clothing' ELSE 'Books' END,
       DATEADD(DAY, -N % 30, '2026-06-28') FROM CTE;

CREATE STATISTICS s_CustomerId ON #LargeOrderHistory(CustomerId) WITH FULLSCAN;

SET STATISTICS TIME ON; SET STATISTICS IO ON;
SELECT lo.OrderId, lo.TotalAmount, cp.PreferredCategory
FROM #LargeOrderHistory lo
LEFT JOIN #CustomerPreferences cp ON lo.CustomerId = cp.CustomerId
WHERE lo.TotalAmount > 500 AND cp.PreferredCategory IS NOT NULL;
SET STATISTICS IO OFF; SET STATISTICS TIME OFF;
`

**Reading the plan:** Right-to-left: scans of both tables. Join operator: likely Hash Match (both large, unsorted). Check filter placement: is TotalAmount > 500 a seek predicate or residual?

**Performance data:**

| Operator | Estimated Rows | Actual Rows | Logical Reads | Cost % |
|---|---|---|---|---|
| Clustered Index Scan (Orders) | 500,000 | 500,000 | 2,500 | 15% |
| Table Scan (Preferences) | 50,000 | 50,000 | 150 | 2% |
| Hash Match (Inner Join) | 250,000 | 249,500 | 0 (CPU) | 78% |

The Hash Match dominates. Adding an index on TotalAmount or CustomerId could enable Nested Loops.

## Section 7 — Interview Arsenal

### Interview Questions

**Q1: How do you read a graphical execution plan?**
*Spoken answer (tier 1):* Right-to-left, top-to-bottom for binary operators. Data flows from leaf operators upward to root. Arrow thickness shows relative row volume. Cost percentages identify expensive operators.

*Spoken answer (tier 2):* I start at the rightmost operator and trace left. For each join, I check which input is outer (top) vs inner (bottom). I look for three things: (1) Is arrow thickness consistent? Thin in, thick out = under-estimate. (2) Warning icons? (3) Most expensive operator by cost % AND actual execution count. The Nested Loops inner side may be cheap per execution but expensive when executed 1M times.

**Q2: What do you look for first in a graphical plan?**
*Spoken answer:* The ratio of actual vs estimated rows. If any operator has 10x+ discrepancy, that is the root cause. Then join type - Nested Loops with a full clustered index scan on inner side is a red flag. Then warnings: missing stats, spills, implicit conversions.

**Q3: What does arrow thickness tell you?**
*Spoken answer:* Arrow thickness is proportional to row count between operators. In estimated plans, based on CE. In actual plans, based on actual rows. Hairline into Nested Loops but thick out = under-estimated iterations.

**Q4: How do you distinguish Table Scan from Clustered Index Scan?**
*Spoken answer:* Table Scan icon shows three cylinders without a key symbol; physical operator is Table Scan. Clustered Index Scan shows three cylinders with a key icon; physical operator is Clustered Index Scan. Table Scan = heap. Clustered Index Scan = table with clustered index. Matters for fragmentation and forwarding pointers.

**Q5: What tooltip fields are most important?**
*Spoken answer:* Actual vs Estimated rows, Number of Executions, Node ID, Estimated Operator Cost %, Ordered. The pair of Actual vs Estimated rows is the single most important diagnostic metric.

**Q6: How do you identify parameter sniffing from a graphical plan?**
*Spoken answer:* Operators where Estimated Rows differ significantly from Actual Rows, AND the query uses parameters. Hash Join for large data used when only 10 rows return = classic sniffing. Fix: RECOMPILE, OPTIMIZE FOR UNKNOWN, or Query Store plan forcing.

**Q7: What is a Key Lookup and why is it expensive?**
*Spoken answer:* A Key Lookup appears when a non-clustered index covers WHERE and JOIN columns but not all SELECT columns. For each row found in the non-clustered index, SQL Server issues a clustered index seek to fetch missing columns. 100K rows = 100K random I/Os. Fix: add INCLUDE columns to the non-clustered index.

**Q8: What does a Sort operator tell you about indexing?**
*Spoken answer:* A Sort operator indicates no index provides the requested sort order (for ORDER BY, GROUP BY, or Merge Join). Sort is a blocking operator - must consume all rows before producing output. For large inputs, sorts spill to tempdb. Fix: index matching the sort order, or remove ORDER BY if client handles sorting.

### Comparison Table

| Operator | When Selected | Memory | Spills | Best Index |
|---|---|---|---|---|
| Nested Loops | Small outer (<100 rows), inner has seek | None | No | Index on inner join column |
| Hash Match | Large unsorted inputs | High | Yes if insufficient | Index on join columns |
| Merge Join | Sorted inputs on join column | Low | No | Index on join column in sort order |
| Index Seek | SARGable predicate | None | No | Matching index on predicate columns |
| Clustered Index Scan | No useful index or all columns needed | None | No | Missing index on WHERE columns |
| Key Lookup | Non-clustered covers predicates but not SELECT | None | No | INCLUDE columns on non-clustered index |

## Section 8 — Decision Framework

`mermaid
flowchart TD
    A[Open Plan in SSMS] --> B{Actual or Estimated?}
    B -->|Estimated| C[Enable Actual Plan + Capture Execution]
    B -->|Actual| D[Read R->L]
    D --> E[Check EstimateRows vs ActualRows per operator]
    E -->|OK <2x| F[Check cost distribution]
    E -->|>10x| G[Root cause: stats/CE/sniffing]
    F -->|Single op >70%| H[Focus on that operator]
    F -->|Even| I[Check all for small gains]
    G --> J{Diagnose operator}
    J -->|Scan not seek| K[Add index on predicates]
    J -->|Key Lookup| L[Add INCLUDE columns]
    J -->|Sort| M[Add index matching sort order]
    J -->|Hash memory spill| N[Increase memory or add index]
    J -->|Nested Loops inner scan| O[Add index on join column]
    K --> P[Implement and verify]
    L --> P; M --> P; N --> P; O --> P
    P --> Q{Reads reduced?}
    Q -->|Yes| R[Done]
    Q -->|No| S[Re-analyze plan]
`

### Decision Checklist

1. Get actual plan from production workload
2. Start at rightmost operator, note each operator type
3. Compare Estimated vs Actual rows per operator
4. Identify operators where ratio > 10x
5. Check warnings: missing stats, spills, implicit conversions
6. Check join order: Nested Loops with thick arrow on outer?
7. Prioritize: fix biggest discrepancy first
8. Productionize: create index, update stats, or add query hint

### Tradeoffs

| Optimization | Benefit | Risk | When |
|---|---|---|---|
| Add index | Eliminates scan | Write overhead | OLTP, selective predicates |
| Include columns | Eliminates key lookup | Index size | SELECT * on wide tables |
| Rewrite query | Avoids spool | Maintenance burden | Complex views or TVFs |
| Add hint | Forces plan | Blocks better plans | Emergency fix |
| Update statistics | Fixes estimate | CPU spike | After large data changes |

### Scale Thresholds

- Simple (<10 ops): Scan visually in 5 seconds
- Moderate (10-30 ops): Use plan XML search or DMVs
- Complex (30+ ops): Use plan analysis tools, focus on top 5
- Very large (100+ ops): Use SET STATISTICS PROFILE and parse row output

## Section 9 — Self-Check

### Conceptual Questions

**Q1:** What is the reading direction for a graphical execution plan?
<details>Right-to-left (data flows from leaf to root). For binary operators: top-to-bottom, outer (build) input on top.</details>

**Q2:** What does a thick arrow between operators indicate?
<details>Large number of rows flowing between operators. Thickness proportional to estimated (or actual) row count relative to other operators.</details>

**Q3:** How do you find the Number of Executions for a Nested Loops inner operator?
<details>Hover over the operator tooltip. Number of Executions shows how many times the inner ran (once per outer row). High executions + thin outer arrow = under-estimated outer.</details>

**Q4:** What is the difference between Table Scan and Clustered Index Scan?
<details>Table Scan operates on a heap (no clustered index). Clustered Index Scan reads the clustered index B-tree. Both read all rows.</details>

**Q5:** What does the Ordered property in Index Seek tooltip mean?
<details>True means output maintains index key order, potentially feeding a Merge Join or avoiding Sort.</details>

**Q6:** What icon represents a Hash Match Join in SSMS?
<details>Two interleaved triangles (Venn diagram intersection).</details>

**Q7:** What does a yellow triangle warning on an operator indicate?
<details>Warning: missing stats, implicit conversion, spill, excessive memory grant, or no join predicate.</details>

**Q8:** How do you generate estimated vs actual plans in SSMS?
<details>Estimated: Ctrl+L. Actual: Ctrl+M then execute. Actual plan includes runtime statistics.</details>

**Q9:** What XML element contains runtime information?
<details>RunTimeInformation inside RelOp with RunTimeCountersPerThread elements.</details>

**Q10:** What is subtree cost and its use?
<details>Cumulative cost of an operator and all its children. Compare subtree costs between two plans for the same query.</details>

**Challenge 6:** Write a procedure that accepts a .sqlplan file path and returns all operators with their subtree cost, ranked.
<details>`sql
CREATE OR ALTER PROCEDURE dbo.AnalyzePlanFile @file_path NVARCHAR(500)
AS
BEGIN
    DECLARE @plan_xml XML = (SELECT CAST(BulkColumn AS XML) FROM OPENROWSET(BULK @file_path, SINGLE_BLOB) AS x);
    WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
    SELECT ROW_NUMBER() OVER (ORDER BY t.c.value('@EstimatedTotalSubtreeCost','FLOAT') DESC) AS rank,
        t.c.value('@NodeId','INT') AS node_id,
        t.c.value('@PhysicalOp','VARCHAR(100)') AS physical_op,
        t.c.value('@EstimateRows','FLOAT') AS est_rows,
        t.c.value('@EstimatedTotalSubtreeCost','FLOAT') AS subtree_cost,
        t.c.value('@EstimateIO','FLOAT') AS est_io,
        t.c.value('@EstimateCPU','FLOAT') AS est_cpu,
        t2.r.value('@ActualRows','BIGINT') AS actual_rows
    FROM @plan_xml.nodes('//p:RelOp') t(c)
    OUTER APPLY t.c.nodes('p:RunTimeInformation/p:RunTimeCountersPerThread') t2(r)
    ORDER BY subtree_cost DESC;
END;
`</details>

**Challenge 7:** Write a query that returns all plans in cache that contain a Table Spool operator (often indicates complex subqueries from ORMs).
<details>`sql
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
SELECT qs.query_hash, qs.execution_count,
    SUBSTRING(st.text, 1, 200) AS query_text,
    qp.query_plan.value('(//p:RelOp[@PhysicalOp="Table Spool"]/@EstimateRows)[1]','FLOAT') AS spool_est_rows,
    qp.query_plan.value('count(//p:RelOp[@PhysicalOp="Table Spool"])','INT') AS spool_count
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
WHERE qp.query_plan.exist('//p:RelOp[@PhysicalOp="Table Spool"]') = 1
ORDER BY qs.total_elapsed_time DESC;
`</details>

**Challenge 8:** Create a script that simulates the optimizer NOT having a missing index and shows the plan difference.
<details>`sql
-- Create a table with and without a covering index to see plan differences
CREATE TABLE ##PlanDemo (Id INT IDENTITY(1,1), KeyCol INT, DataCol VARCHAR(200), CreatedDate DATETIME);
INSERT INTO ##PlanDemo (KeyCol, DataCol, CreatedDate)
SELECT ABS(CHECKSUM(NEWID()))%1000, 'Row_' + CAST(ROW_NUMBER() OVER (ORDER BY (SELECT NULL)) AS VARCHAR(20)), GETDATE()
FROM (SELECT TOP 10000 ROW_NUMBER() OVER (ORDER BY (SELECT NULL)) AS N FROM sys.all_columns a CROSS JOIN sys.all_columns b) t;

-- Plan without covering index
SET STATISTICS XML ON;
SELECT KeyCol, DataCol, CreatedDate FROM ##PlanDemo WHERE KeyCol = 500;
SET STATISTICS XML OFF;

-- Create non-clustered index (missing the INCLUDE)
CREATE INDEX IX_KeyCol ON ##PlanDemo(KeyCol);

### Case Study: Real-World Plan Reading

**Scenario:** A customer reported that a simple order lookup page timed out after a data migration. The query was:

`sql
SELECT o.OrderId, o.OrderDate, o.Status, c.CustomerName, c.Email
FROM Orders o
JOIN Customers c ON o.CustomerId = c.CustomerId
WHERE o.OrderDate >= '2026-06-01' AND o.Status = 'Pending'
ORDER BY o.OrderDate;
`

**Plan Analysis (Step by Step):**

1. **Rightmost operator:** Clustered Index Scan on Orders (PK_Orders). Estimated rows: 10M. Cost: 12%. Arrow thickness: thick.
   - This is a red flag — the query filters on OrderDate and Status, but there is no index to support it.

2. **Left of scan:** Filter operator. Predicate: `o.Status = 'Pending'`. The scan reads all 10M rows, then filters.
   - The optimizer is doing a full scan because it estimates that most rows match the date range AND it has no statistics on Status.

3. **Join operator:** Hash Match (Inner Join). Estimated rows from scan: 10M. Actual rows after filter: 500K.
   - Hash Match is appropriate for 500K rows, but the memory grant was sized for 10M rows (8GB granted, 400MB used).

4. **Customers side:** Clustered Index Seek on PK_Customers with Key Lookup for Email column.
   - Estimated rows: 1 (per lookup, via Nested Loops inner) but actual: 500K seeks.

5. **Root SELECT:** MemoryGrant=8GB, GrantWaitTime=3200ms, Warnings=Excessive MemoryGrant.
   - The query waited 3.2 seconds for memory grant. This was the actual bottleneck, not the scan itself.

**Resolution:**
- Created: `CREATE INDEX IX_Orders_OrderDate_Status_INCLUDE ON Orders(OrderDate, Status) INCLUDE (CustomerId)`
- Created: `CREATE INDEX IX_Customers_CustomerId_INCLUDE ON Customers(CustomerId) INCLUDE (CustomerName, Email)`
- After: Index Seek (12ms), Nested Loops join, Memory Grant = 1MB, Logical reads reduced from 52,000 to 340.

**Lesson:** The graphical plan showed the scan as only 12% cost, but the memory grant wait (not visible as a cost %) was the real problem. Arrow thickness and cost percentages are guides, not absolute truths.

## Section 9 — Advanced Self-Check

### Additional Conceptual Questions

**Q16:** What does a "Bitmap" operator look like in a graphical plan and what does it do?
<details>The Bitmap operator appears as a honeycomb/grid icon. It creates a bitmap filter from one side of a join to reduce rows on the other side before the join. This is a runtime optimization — the bitmap is created from the build input and applied to the probe input to eliminate non-matching rows early. It does not appear in the estimated plan, only in the actual plan.</details>

**Q17:** How do you distinguish an Index Scan from an Index Seek in the graphical plan?
<details>Index Scan has an arrow pointing down into the index pages (all pages read). Index Seek has an arrow pointing to a subset of pages with a B-tree traversal icon. The tooltip shows: Index Scan = Scan, Index Seek = Seek. The text also differs: "Scan: scanning a nonclustered index" vs "Seek: seeking in a nonclustered index".</details>

**Q18:** What does "Actual Rebinds" > 0 in a Nested Loops inner operator indicate?
<details>Rebinds indicate that the outer row changed the correlated parameter, forcing the inner operator to re-initialize. Each rebind resets any state in the inner operator. High rebinds with a Nested Loops join means the outer input has many rows, and each one triggers a new execution of the inner side. If Rebinds = Number of Executions, every execution is a fresh start — no parameter reuse from prior iterations.</details>

**Q19:** In a graphical plan, how do you see the exact memory grant per operator?
<details>Memory grant is shown at the root SELECT operator level, not per operator. To see per-operator memory, examine the RunTimeInformation in the plan XML: `//p:RunTimeCountersPerThread/@ActualMemoryKB`. Memory-consuming operators (Sort, Hash Match) show their actual memory usage here. Compare to TotalMemoryGrant at the query level to find which operator consumed the grant.</details>

**Q20:** What does it mean when a compute scalar shows "DEFINE: [Expr1002] = [...]" in the tooltip?
<details>It defines a computed expression with an internal column alias (Expr1002). This expression may be referenced by operators further up the tree. Each Compute Scalar defines one or more expressions. Multiple Compute Scalars in sequence can indicate repeated evaluation of the same expression — a common ORM-generated query issue. The fix is to simplify the query or add a computed column.</details>

**Q21:** How do you tell if a spool operator is a performance problem or benign?
<details>Check the Actual Rows vs Estimated Rows on the spool. A spool with Actual Rows matching Estimated Rows and low actual reads is benign — it cached a small intermediate result. A spool with Actual Rows >> Estimated Rows or high Actual Reads is a performance problem because it materialized a large intermediate result to tempdb. Look for the "Lazy Spool" vs "Eager Spool" distinction: Lazy Spool reads on demand; Eager Spool materializes fully before proceeding.</details>

**Q22:** What is the difference between "Clustered Index Scan" and "Clustered Index Seek" in the plan?
<details>Clustered Index Scan reads all rows (or all pages) from the clustered index — the entire table. Clustered Index Seek reads only the rows matching a predicate using the B-tree structure of the clustered index. A Clustered Index Seek typically appears only when the clustered key columns are used in a WHERE predicate (e.g., WHERE Id = 5). All other predicates on non-clustered columns result in a Clustered Index Scan.</details>

**Q23:** How do you detect a non-SARGable predicate from the graphical plan?
<details>Look for a Filter operator immediately after an Index Seek or Clustered Index Scan. The Seek Predicate in the tooltip will show the SARGable part (e.g., OrderDate BETWEEN ...), while the Predicate section shows the residual non-SARGable filter (e.g., WHERE YEAR(OrderDate) = 2026). The presence of both Seek Predicate and Predicate on the same Index Seek operator indicates a partially non-SARGable predicate.</details>

CREATE OR ALTER PROCEDURE dbo.ComparePlanFiles
    @file1 NVARCHAR(500), @file2 NVARCHAR(500)
AS
BEGIN
    DECLARE @xml1 XML = (SELECT CAST(BulkColumn AS XML) FROM OPENROWSET(BULK @file1, SINGLE_BLOB) AS x);
    DECLARE @xml2 XML = (SELECT CAST(BulkColumn AS XML) FROM OPENROWSET(BULK @file2, SINGLE_BLOB) AS x);

    WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
    SELECT COALESCE(p1.node_id, p2.node_id) AS node_id,
        p1.phys_op AS plan1_op, p2.phys_op AS plan2_op,
        p1.est_rows AS plan1_rows, p2.est_rows AS plan2_rows,
        p1.subtree_cost AS plan1_cost,
        p2.subtree_cost AS plan2_cost,
        p1.actual_rows AS plan1_actual,
        p2.actual_rows AS plan2_actual,
        CASE WHEN p1.subtree_cost > p2.subtree_cost * 1.5 THEN 'Plan1 Worse'
             WHEN p2.subtree_cost > p1.subtree_cost * 1.5 THEN 'Plan2 Worse'
             ELSE 'Similar' END AS comparison
    FROM (SELECT t.c.value('@NodeId','INT') AS node_id, t.c.value('@PhysicalOp','VARCHAR(100)') AS phys_op,
                 t.c.value('@EstimateRows','FLOAT') AS est_rows,
                 t.c.value('@EstimatedTotalSubtreeCost','FLOAT') AS subtree_cost,
                 t2.r.value('@ActualRows','BIGINT') AS actual_rows
          FROM @xml1.nodes('//p:RelOp') t(c)
          OUTER APPLY t.c.nodes('p:RunTimeInformation/p:RunTimeCountersPerThread') t2(r)) p1
    FULL OUTER JOIN
        (SELECT t.c.value('@NodeId','INT') AS node_id, t.c.value('@PhysicalOp','VARCHAR(100)') AS phys_op,
                 t.c.value('@EstimateRows','FLOAT') AS est_rows,
                 t.c.value('@EstimatedTotalSubtreeCost','FLOAT') AS subtree_cost,
                 t2.r.value('@ActualRows','BIGINT') AS actual_rows
          FROM @xml2.nodes('//p:RelOp') t(c)
          OUTER APPLY t.c.nodes('p:RunTimeInformation/p:RunTimeCountersPerThread') t2(r)) p2
    ON p1.node_id = p2.node_id
    ORDER BY COALESCE(p2.subtree_cost, p1.subtree_cost) DESC;
END;

## Section 6 — Performance Implications (Extended)

### Demo: Reading a Real-World Plan Step by Step

Consider this query on a 10M row Orders table with a non-clustered index on OrderDate:

`sql
SELECT OrderId, OrderDate, CustomerId, TotalAmount
FROM Orders
WHERE OrderDate BETWEEN '2026-01-01' AND '2026-03-31'
  AND TotalAmount > 100
ORDER BY OrderDate;
`

**Step-by-step plan reading:**

1. **Rightmost operator:** Clustered Index Scan on Orders (PK_Orders). EstimateRows = 10,000,000. Cost = 85%.
   - Red flag: Full scan with filter. If OrderDate has an index, why scan?

2. **Left of scan:** Filter operator. Predicate: TotalAmount > 100.
   - Red flag: The index on OrderDate covers the range but does not include TotalAmount. The optimizer chose scan + filter instead of seek + key lookup because it estimated fewer rows from the range.

3. **Ordered property:** If the Clustered Index Scan shows Ordered=True, the output is naturally sorted by OrderId (clustered key). But the ORDER BY is on OrderDate, so a Sort operator may appear.

4. **Sort:** If Sort appears, check EstimatedRows vs ActualRows. If the estimate says 100K but 2M rows are sorted, the sort spills to tempdb.

5. **Diagnosis:** The missing index is: CREATE INDEX IX_Orders_OrderDate_INCLUDE ON Orders(OrderDate) INCLUDE (CustomerId, TotalAmount). This would eliminate both the scan and the need for a key lookup.

## Section 7 — Interview Arsenal (Extended)

### Q9: How do you identify a missing index from a graphical plan?
*Spoken answer:* SSMS shows green text at the top of the plan: "Missing Index (Impact 95.2%)". The plan XML has a MissingIndexes element. The graphical plan also shows a Key Lookup operator followed by a Nested Loop with the clustered index - this is the classic missing INCLUDE columns pattern.

### Q10: What is the difference between a Hash Match join and a Merge Join operator?
*Spoken answer:* Hash Match builds a hash table on one input (build side) and probes with the other (probe side). It requires no sorted input but uses memory proportional to the build side. Merge Join requires both inputs sorted on the join column - it walks through both in order, matching rows. Merge Join uses no memory and is faster for large sorted inputs. The graphical plan shows hash with interleaved triangles, merge with parallel lines.

### Q11: When does the optimizer choose a Stream Aggregate vs Hash Match Aggregate?
*Spoken answer:* Stream Aggregate requires sorted input (usually from an index or a preceding Sort). It processes one group at a time. Hash Match Aggregate builds a hash table of groups in memory. Stream Aggregate is more efficient for few groups with sorted input; Hash Match is better for many groups or unsorted input. In the plan, Stream Aggregate shows with a single cylinder icon; Hash Match Aggregate shows as a hash table icon.

### Q12: What does the Actual Rebinds and Actual Rewinds mean in a plan?
*Spoken answer:* These appear in the tooltip for operators inside Nested Loops. Rebinds count how many times the operator was fully reset (outer row changes a correlated parameter). Rewinds count re-executions without reset (parameter unchanged). If Rewinds dominate, the operator benefits from parameter sniffing across iterations. High Rebinds suggest poor join order or missing index.

## Section 8 — Decision Framework (Extended)

### Detailed Escalation Path

When reading a plan and identifying the root cause, follow this priority:

1. **Cardinality mismatch > 10x:** Fix the estimate first. Nothing else matters if the optimizer is working with wrong data.
2. **Key Lookup with high executions:** Add INCLUDE columns to existing non-clustered index.
3. **Table/Clustered Index Scan with selective predicate:** Create an index on the predicate columns.
4. **Sort with large input:** Either remove ORDER BY (if client sorts) or add an index matching the sort order.
5. **Nested Loops with large outer:** Wrong join type caused by under-estimated outer. Fix estimate or add join hint.
6. **Hash Match spill:** Increase memory grant or add index to enable Merge Loop.
7. **Parallelism bottleneck:** Check DOP, cost threshold for parallelism, or thread skew.

### Query Store Integration

`sql
-- Find regressed plans by comparing two time periods
DECLARE @old_period_start DATETIME2 = '2026-01-01';
DECLARE @old_period_end DATETIME2 = '2026-02-01';
DECLARE @new_period_start DATETIME2 = '2026-05-01';
DECLARE @new_period_end DATETIME2 = '2026-06-01';

SELECT qsq.query_id, qsqt.query_sql_text,
    old_stats.avg_duration/1000 AS old_duration_ms,
    new_stats.avg_duration/1000 AS new_duration_ms,
    (new_stats.avg_duration - old_stats.avg_duration) / old_stats.avg_duration * 100 AS regress_pct
FROM sys.query_store_query qsq
INNER JOIN sys.query_store_query_text qsqt ON qsq.query_text_id = qsqt.query_text_id
CROSS APPLY (
    SELECT AVG(rs.avg_duration) AS avg_duration
    FROM sys.query_store_plan qsp
    INNER JOIN sys.query_store_runtime_stats rs ON qsp.plan_id = rs.plan_id
    WHERE qsp.query_id = qsq.query_id
    AND rs.first_execution_time BETWEEN @old_period_start AND @old_period_end
) old_stats
CROSS APPLY (
    SELECT AVG(rs.avg_duration) AS avg_duration
    FROM sys.query_store_plan qsp
    INNER JOIN sys.query_store_runtime_stats rs ON qsp.plan_id = rs.plan_id
    WHERE qsp.query_id = qsq.query_id
    AND rs.first_execution_time BETWEEN @new_period_start AND @new_period_end
) new_stats
WHERE old_stats.avg_duration > 0
AND (new_stats.avg_duration - old_stats.avg_duration) / old_stats.avg_duration * 100 > 50
ORDER BY regress_pct DESC;
`

## Section 9 — Self-Check (Extended)

### Additional Conceptual Questions

**Q11:** What is the significance of the "Estimated Execution Mode" property in a plan?
<details>Shows Row or Batch mode. Row mode = row-by-row processing (OLTP). Batch mode = batch processing for columnstore/DW queries. Batch mode is more efficient for large scans but requires columnstore indexes or specific SQL editions.</details>

**Q12:** How do you find the actual number of rows read from disk (physical reads) vs cache (logical reads)?
<details>SET STATISTICS IO ON shows both logical reads (from buffer pool) and physical reads (from disk). Read-ahead reads are also shown. High physical reads relative to logical reads indicates buffer pool pressure or cold cache.</details>

**Q13:** What does a "Warning: No Join Predicate" mean in a plan?
<details>Two tables in the FROM clause have no join condition, producing a Cartesian product. This is almost always a bug. The warning appears as a yellow triangle on the join operator. The fix is to add a proper join predicate.</details>

**Q14:** What does "Implicit Conversion" warning indicate?
<details>A column is compared to a value of a different data type (e.g., VARCHAR column compared to NVARCHAR parameter, or INT compared to VARCHAR). SQL Server converts one side, which disables index SARGability. The warning appears on the scan/seek operator. Fix: align data types between column and parameter.</details>

**Q15:** How do you differentiate between a Filter and a Seek Predicate in the plan?
<details>Seek Predicate (in the Index Seek tooltip) is used directly by the B-tree to navigate to rows - it is SARGable and efficient. Filter (in the Predicate section) is applied as a residual check after the seek/scan - it is non-SARGable. Pushing a Filter into a Seek Predicate via index tuning is a common optimization.</details>
