### Where This Fits

Parameter sniffing is the problem; this note is the armory. There are six canonical solutions, each with specific tradeoffs between plan quality, CPU cost, and maintenance burden. The right choice depends on the query's frequency, data skew profile, and performance requirements. For .NET engineers, the most practical solutions are OPTIMIZE FOR (when you know a representative parameter value) and query separation (splitting hot paths into distinct queries with their own cached plans). OPTION (RECOMPILE) is the nuclear option — effective but CPU-expensive — and should be reserved for low-frequency, high-skew queries. The local variable trick is a legacy approach that defeats sniffing by hiding the value but falls back to density-based estimates. Plan guides and Query Store plan forcing are DBA-level tools for vendor applications that cannot be modified. At interview, the ability to enumerate all six solutions and explain the tradeoffs (with specific logical read numbers) separates senior engineers from those who know one or two hints.

## Section 2 — Core Mental Model

There is no single "best" solution to parameter sniffing because the problem is fundamentally about information: the optimizer needs to know the parameter value to estimate cardinality, but if it uses that value for plan caching, the plan may be suboptimal for other values. Every solution trades off one of three dimensions: (1) **plan reuse** — how many executions share the same plan, (2) **plan quality** — how well the plan matches the actual data for each execution, and (3) **CPU cost** — how much compile time is spent. The six solutions form a spectrum:

- **OPTIMIZE FOR UNKNOWN:** Uses average density (no sniffing). Medium plan quality, high reuse, low CPU.
- **OPTIMIZE FOR (@p = value):** Sniffs a specific value. Medium plan quality, high reuse, low CPU.
- **OPTION (RECOMPILE):** No plan reuse — sniffs fresh each time. High plan quality, no reuse, high CPU.
- **DISABLE_PARAMETER_SNIFFING hint:** New in SQL Server 2016 (USE HINT). No sniffing at compile; uses density. Same as OPTIMIZE FOR UNKNOWN but applies to all parameters.
- **Local variable trick (DECLARE @local = @param):** Hides parameter value from optimizer. Uses density. Low CPU, high reuse, but potentially inaccurate.
- **Plan guide / Query Store forcing:** Pins a specific plan. Fixed plan quality (good if the forced plan is good), high reuse, very low CPU, but manual maintenance.

## Section 3 — Deep Mechanics

### Solution 1 — OPTIMIZE FOR UNKNOWN

**How it works:**
The `OPTIMIZE FOR UNKNOWN` hint tells the optimizer to use average column density (from statistics) instead of peeking at the actual parameter value. The optimizer estimates cardinality as if the parameter value is unknown — it uses `DENSITY * table_cardinality` instead of a histogram lookup for the specific sniffed value.

```sql
CREATE PROCEDURE Sales.GetOrdersByStatus_Unknown
    @Status tinyint
AS
    SELECT OrderId, OrderDate, CustomerId, OrderTotal
    FROM Sales.Orders
    WHERE Status = @Status
    OPTION (OPTIMIZE FOR UNKNOWN);
```

**When it helps:** When data is roughly evenly distributed, density ≈ actual selectivity. E.g., Status 0-3 each have ~25% of rows → density-based estimate predicts 25% of rows, which is close enough. The plan (scan vs seek) is reasonable for all values.

**When it hurts:** When data is highly skewed. E.g., Status 0 = 2%, Status 2 = 65%. Density = (1 / distinct_count) = 0.25. The estimate is 25% of rows. For Status 0 (2%): overestimates by 12.5x, may choose scan instead of seek. For Status 2 (65%): underestimates by 2.6x, may choose seek instead of scan.

### Solution 2 — OPTIMIZE FOR (@p = value)

**How it works:**
Specifies a specific parameter value for the optimizer to use during compilation, ignoring the actual runtime value. The plan is optimized for this chosen value and reused for all executions.

```sql
CREATE PROCEDURE Sales.GetOrdersByStatus_OptimizeFor
    @Status tinyint
AS
    SELECT OrderId, OrderDate, CustomerId, OrderTotal
    FROM Sales.Orders
    WHERE Status = @Status
    OPTION (OPTIMIZE FOR (@Status = 2));  -- sniff Status=2 at compile
```

**How to choose the value:** Use the most common parameter value. In the statuses example, if Status = 2 (Active) represents 65% of queries and is the most common, optimize for Status = 2. The plan will be excellent for the 65% case and acceptable (or at least not catastrophic) for others.

**Compromise:** The plan is suboptimal for the 2% edge case (Status 0) but good for the 65% common case. This is acceptable if the edge case is also infrequent — the total performance impact across all executions is minimized.

### Solution 3 — OPTION (RECOMPILE)

**How it works:**
Forces recompilation on every execution. The compiler sniffs the current parameter values, optimizes for them, and the plan is used for that execution only. No plan is cached.

```sql
CREATE PROCEDURE Sales.GetOrdersByStatus_Recompile
    @Status tinyint
AS
    SELECT OrderId, OrderDate, CustomerId, OrderTotal
    FROM Sales.Orders
    WHERE Status = @Status
    OPTION (RECOMPILE);
```

**Cost:** Each execution includes compile time (5-15ms for a simple query, more for complex). At 1000 QPS, that's 5-15 seconds of CPU per second — the CPU is pegged. Only suitable for low-frequency queries (< 100 exec/hour) or queries where the plan quality difference is enormous (10x+ read reduction).

### Solution 4 — DISABLE_PARAMETER_SNIFFING (USE HINT)

**How it works:**
A query hint introduced in SQL Server 2016 (compatibility level 130+) that disables parameter sniffing for this query. The optimizer treats all parameters as unknown, using density-based estimates.

```sql
CREATE PROCEDURE Sales.GetOrdersByStatus_NoSniff
    @Status tinyint
AS
    SELECT OrderId, OrderDate, CustomerId, OrderTotal
    FROM Sales.Orders
    WHERE Status = @Status
    OPTION (USE HINT('DISABLE_PARAMETER_SNIFFING'));
```

**Difference from OPTIMIZE FOR UNKNOWN:** DISABLE_PARAMETER_SNIFFING applies to ALL parameters in the query, not just the ones you specify with UNKNOWN. It also affects the cardinality estimation for joins, GROUP BY, and other operations.

### Solution 5 — Local Variable Trick

**How it works:**
Assign the parameter to a local variable inside the stored procedure. The optimizer cannot sniff local variable values at compile time — only parameters are sniffed. Without a sniffed value, the optimizer falls back to density-based estimates (same as OPTIMIZE FOR UNKNOWN or DISABLE_PARAMETER_SNIFFING).

```sql
CREATE PROCEDURE Sales.GetOrdersByStatus_LocalVar
    @Status tinyint
AS
BEGIN
    DECLARE @LocalStatus tinyint = @Status;
    
    SELECT OrderId, OrderDate, CustomerId, OrderTotal
    FROM Sales.Orders
    WHERE Status = @LocalStatus;  -- uses local var, no sniffing
END;
```

**Key nuance:** This trick works because the optimizer sees a variable in the predicate, not a parameter. It cannot peek at the variable's value because the variable assignment occurs at runtime (after compilation). The estimate uses `DENSITY * table_cardinality`.

**Downside:** The execution plan NEVER sees the actual value, even on first execution. There is no scenario where the optimizer gets a good cardinality estimate. This is a hammer that treats all parameter values as equally unknown. Use only when OPTIMIZE FOR or RECOMPILE are not viable.

### Solution 6 — Plan Guide / Query Store Plan Forcing

**How it works:**
Create a plan guide that applies a hint (like OPTIMIZE FOR or RECOMPILE) to a query without modifying its source code. Or use Query Store to force a specific plan for a query.

```sql
-- Plan guide approach
EXEC sp_create_plan_guide 
    @name = N'PG_GetOrdersByStatus',
    @stmt = N'SELECT OrderId, OrderDate, CustomerId, OrderTotal
              FROM Sales.Orders WHERE Status = @Status',
    @type = N'OBJECT',
    @module_or_batch = N'Sales.GetOrdersByStatus',
    @hints = N'OPTION (OPTIMIZE FOR (@Status = 2))';

-- Query Store approach
EXEC sys.sp_query_store_force_plan 
    @query_id = 42, 
    @plan_id = 10;
```

```sql
-- 1. Find queries with RECOMPILE hint (high compile CPU)
SELECT 
    qs.execution_count,
    qs.total_worker_time / 1000000 AS TotalCPU_sec,
    qs.total_elapsed_time / 1000000 AS TotalDuration_sec,
    qs.total_worker_time / NULLIF(qs.execution_count, 0) / 1000 AS AvgCPU_ms,

CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
WHERE qs.plan_generation_num = qs.execution_count  -- compiles every time = RECOMPILE
```

### Cost Visibility — Comparing Solutions

```sql
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

-- Demo database: Sales.Orders with skewed Status
-- Status 0: 2%, Status 2: 65%

-- Baseline: No fix (sniffing causes regression)
PRINT '=== Baseline: No fix ===';
EXEC Sales.GetOrdersByStatus_Demo @Status = 0;   -- first exec: sniffs 0, seek plan
EXEC Sales.GetOrdersByStatus_Demo @Status = 2;   -- reuses seek plan, BAD for 65%
-- Status 0: logical reads ~500 (seek, optimal)
-- Status 2: logical reads ~810,000 (seek+lookup for 650K rows, TERRIBLE)

-- Solution 1: OPTIMIZE FOR UNKNOWN
PRINT '=== OPTIMIZE FOR UNKNOWN ===';
EXEC Sales.GetOrdersByStatus_Unknown @Status = 0;
EXEC Sales.GetOrdersByStatus_Unknown @Status = 2;
-- Both: logical reads ~15,000 (scan, acceptable for both)

-- Solution 2: OPTIMIZE FOR (@Status = 2)
PRINT '=== OPTIMIZE FOR (@Status = 2) ===';
EXEC Sales.GetOrdersByStatus_OptimizeFor @Status = 0;
EXEC Sales.GetOrdersByStatus_OptimizeFor @Status = 2;
-- Status 0: logical reads ~15,000 (scan, suboptimal for 2% but acceptable)
-- Status 2: logical reads ~15,000 (scan, optimal for 65%)

-- Solution 3: OPTION (RECOMPILE)
PRINT '=== OPTION (RECOMPILE) ===';
EXEC Sales.GetOrdersByStatus_Recompile @Status = 0;
-- logical reads ~500, CPU includes compile (~10ms)
EXEC Sales.GetOrdersByStatus_Recompile @Status = 2;
-- logical reads ~15,000, CPU includes compile (~10ms)

-- Summary:
-- Solution best for high-frequency Status=2 workload: OPTIMIZE FOR (@Status=2)
-- Solution best for mixed workload: OPTIMIZE FOR UNKNOWN or local variable
-- Solution best for low-frequency: OPTION (RECOMPILE)
```

### EF Core Query Interception for Hints

```csharp
// For adding hints to LINQ queries, implement an IDbCommandInterceptor
// EF Core 5+:

public class SniffingMitigationInterceptor : DbCommandInterceptor
{
    private readonly string _hint;

    public SniffingMitigationInterceptor(string hint = "OPTION (RECOMPILE)")
    {
        _hint = hint;
    }

    public override ValueTask<InterceptionResult<DbDataReader>> ReaderExecutingAsync(
        DbCommand command,
        CommandEventData eventData,
        InterceptionResult<DbDataReader> result,
        CancellationToken cancellationToken = default)
    {
        // Only apply to specific query patterns
        if (command.CommandText.Contains("Orders") && 
            command.CommandText.Contains("Status ="))
        {
            command.CommandText += Environment.NewLine + _hint;
        }
        return base.ReaderExecutingAsync(command, eventData, result, cancellationToken);
    }
}

// Registration in Program.cs:
builder.Services.AddDbContext<ApplicationDbContext>(options =>
    options.UseSqlServer(connectionString))
    .AddSingleton<SniffingMitigationInterceptor>();

builder.Services.AddSingleton<DbCommandInterceptor, SniffingMitigationInterceptor>();
```

## Section 5 — Gotchas and Production Pitfalls

### Gotcha 1 — OPTION (RECOMPILE) on High-Frequency Queries Causes CPU Meltdown

**Pitfall:** Developer adds OPTION (RECOMPILE) to a query that runs 5000 times per second as a "quick fix" for parameter sniffing.

```sql
-- ❌ Applied to high-frequency OLTP query
SELECT * FROM Sales.Orders 
WHERE OrderId = @OrderId 
OPTION (RECOMPILE);  -- compiles 5000 times per second!
```

**Symptom:** CPU goes from 20% to 95% immediately after deployment. Each execution's compile CPU (5-15ms) overwhelms the server. sys.dm_exec_query_stats shows `plan_generation_num = execution_count` (every execution compiles).

**Fix:** Remove RECOMPILE. Use OPTIMIZE FOR UNKNOWN or OPTIMIZE FOR a representative value instead. RECOMPILE is for queries that run < 100 times per hour.

**Cost of not fixing:** At 5000 QPS × 10ms compile = 50 seconds of CPU per second. The server is completely CPU-bound. Application times out. Production incident.

### Gotcha 2 — OPTIMIZE FOR UNKNOWN Does Not Fix Skewed Distributions

**Pitfall:** Applying OPTIMIZE FOR UNKNOWN to a query with highly skewed data, expecting a good plan for all values.

**Symptom:** The query runs moderately slow for ALL parameter values — it is never optimal for any value. Status 0 (2%) gets a scan (should be seek). Status 2 (65%) gets a scan (acceptable). But Status 0 could have been 230x faster with a seek. The compromise plan is suboptimal for everyone.

**Fix:** For skewed data, use OPTIMIZE FOR the most common value, not UNKNOWN. If Status = 2 represents 65% of queries, optimize for that. Accept that the 2% edge case runs suboptimally.

**Cost of not fixing:** The 2% edge case (Status = 0) runs 230x slower than necessary. If that edge case happens to be the CEO's report (Cancelled Orders report), the CEO complains. The fix is to identify the common value and optimize for it.

### Gotcha 3 — Local Variable Trick Prevents All Sniffing, Even the Good Kind

**Pitfall:** Applying the local variable trick universally in all stored procedures "to be safe."

```sql
CREATE PROCEDURE Sales.GetCustomerById
    @CustomerId int
AS
BEGIN
    DECLARE @LocalId int = @CustomerId;
    SELECT * FROM Customers WHERE CustomerId = @LocalId;  -- no sniffing
END;
```

**Symptom:** The procedure always uses a density-based estimate. If CustomerId is highly selective (each value ≈ 0.001% of rows), the optimizer estimates `1/distinct_count * table_rows` = ~100 rows instead of the actual 1 row. The plan may choose a scan or hash join instead of a seek + nested loops. Performance is consistently "ok" but never optimal.

**Fix:** Remove the local variable trick. The parameter sniffing problem does NOT affect all queries — only those with skewed distributions. For a CustomerId lookup (highly selective, roughly uniform), sniffing is always beneficial. The local variable trick prevents the optimizer from ever seeing the selective value.

**Cost of not fixing:** A point lookup that should use an Index Seek (2-3 logical reads) instead uses a Clustered Index Scan (10,000+ logical reads) because the density estimate was too high. Every single execution is suboptimal.

### Gotcha 4 — Plan Guide Can Become Stale After Schema Change

**Pitfall:** A plan guide forces a specific plan via OPTIMIZE FOR hint. Six months later, the data distribution has changed (new Status values, different skew), but the plan guide still forces the old hint.

**Symptom:** The plan starts performing poorly. A DBA investigates and finds a plan guide from a former team member that forces OPTIMIZE FOR (@Status = 0). But now Status 0 represents 1% of rows while Status 2 represents 80%. The plan is optimized for the wrong value.

**Fix:** Review plan guides quarterly. Use Query Store plan forcing instead, which shows performance metrics and allows easy rollback.

**Cost of not fixing:** A plan guide that was set up correctly 18 months ago now causes the exact problem it was meant to solve. The team doesn't think to check plan guides because "we haven't changed anything."

### Gotcha 5 — DISABLE_PARAMETER_SNIFFING Only Works on SQL Server 2016+

**Pitfall:** Using the DISABLE_PARAMETER_SNIFFING hint in an environment still running SQL Server 2014 or earlier, or on an Azure SQL Database with compatibility level < 130.

**Symptom:** The query compiles and runs, but the hint is silently ignored. SQL Server 2014 does not recognize the hint and falls back to default sniffing behavior. No error is raised.

**Fix:** Check compatibility level: `SELECT compatibility_level FROM sys.databases WHERE name = DB_NAME()`. Must be >= 130 (SQL Server 2016). For older versions, use OPTIMIZE FOR UNKNOWN instead.

**Cost of not fixing:** The team believes they have disabled parameter sniffing, but the problem continues. Months of debugging follow before someone checks the SQL Server version.

## Section 6 — Performance Implications

### Benchmark: All Solutions Compared

```sql
-- Performance comparison of all 6 solutions
-- Table: Sales.Orders, 1M rows, Status skewed (0=2%, 1=13%, 2=65%, 3=20%)
-- Measure logical reads for Status = 0 (selective) and Status = 2 (non-selective)

SET STATISTICS IO ON;
SET NOCOUNT ON;

-- Clear cache
ALTER DATABASE SCOPED CONFIGURATION CLEAR PLAN CACHE;

-- Test each solution
PRINT '=== Baseline (no fix) — first exec Status=0, second Status=2 ===';
EXEC Sales.GetOrdersByStatus_Baseline @Status = 0;
-- Logical reads: ~500 (Index Seek + Key Lookup — good for 2%)
EXEC Sales.GetOrdersByStatus_Baseline @Status = 2;
-- Logical reads: ~810,000 (reuses seek plan — TERRIBLE for 65%)

ALTER DATABASE SCOPED CONFIGURATION CLEAR PLAN CACHE;

PRINT '=== SOLUTION 1: OPTIMIZE FOR UNKNOWN ===';
EXEC Sales.GetOrdersByStatus_Unknown @Status = 0;
```

### Spoken Answers

**Q: Name all six solutions to parameter sniffing, and explain when you would use each.**

> **Average answer:** "You can use RECOMPILE, OPTIMIZE FOR, or a local variable. RECOMPILE compiles every time, OPTIMIZE FOR tells the optimizer what value to use, and the local variable trick hides the value from the optimizer."

> **Great answer:** "There are six solutions, each with different tradeoffs. First, OPTIMIZE FOR UNKNOWN tells the optimizer to use density-based estimates instead of sniffing the parameter value. Use this when data is roughly uniformly distributed. Second, OPTIMIZE FOR (@p = value) tells the optimizer to sniff a specific value that you choose. This is the best compromise for skewed data: pick the most common parameter value and optimize for it. The edge case queries use a suboptimal plan, but the majority benefit. Third, OPTION (RECOMPILE) forces compilation on every execution, producing an optimal plan each time. Use this only for low-frequency queries (under 100 executions/hour) because compile CPU adds 5-15ms per call — at 1000 QPS, that's 5-15 seconds of CPU per second. Fourth, DISABLE_PARAMETER_SNIFFING, available in SQL Server 2016+, is a USE HINT that disables sniffing for all parameters. It's functionally equivalent to OPTIMIZE FOR UNKNOWN but applies to everything. Fifth, the local variable trick (DECLARE @local = @param) hides the parameter value from the optimizer, forcing density estimates. This works only in stored procedures and prevents all sniffing, even the beneficial kind. Sixth, plan guides or Query Store plan forcing allow you to apply hints without modifying application code — essential for vendor applications. My go-to approach: determine the data distribution and query frequency first. For high-frequency skewed data, I use OPTIMIZE FOR the most common value. For uniform distributions, OPTIMIZE FOR UNKNOWN. For low-frequency skewed queries, RECOMPILE. I never use the local variable trick in new code because it prevents beneficial sniffing with no way to opt-in."

**Q: Compare OPTIMIZE FOR UNKNOWN with DISABLE_PARAMETER_SNIFFING.**

> **Average answer:** "They do the same thing — they make the optimizer not use the parameter value."

> **Great answer:** "Both OPTIMIZE FOR UNKNOWN and DISABLE_PARAMETER_SNIFFING prevent the optimizer from peeking at parameter values, forcing density-based cardinality estimates. The practical differences are subtle but important. OPTIMIZE FOR UNKNOWN is a query hint that applies only to the parameters specified, and it has been available since SQL Server 2005. DISABLE_PARAMETER_SNIFFING is a USE HINT introduced in SQL Server 2016 (compatibility level 130) that applies to all parameters in the query. USE HINT is the newer, more consistent syntax for hinting. Additionally, OPTIMIZE FOR UNKNOWN has a quirk: it disables sniffing but the optimizer still uses the parameter metadata (like length and type) for cardinality estimation. DISABLE_PARAMETER_SNIFFING completely ignores the parameter values. In practice, the cardinality estimates are nearly identical for most queries. I would use OPTIMIZE FOR UNKNOWN for SQL Server 2008-2014 compatibility, and DISABLE_PARAMETER_SNIFFING for new development on SQL Server 2016+ because the USE HINT syntax is cleaner and applies to all parameters consistently."

**Q: How would you handle a query that runs 5000 times/second with a parameter that has 3 possible values?**

> **Average answer:** "Use OPTIMIZE FOR the most common value. That way most queries get a good plan."

> **Great answer:** "At 5000 QPS, RECOMPILE is off the table — it would add 25-75 seconds of CPU per second. OPTIMIZE FOR UNKNOWN would give a compromise plan that is acceptable for all values but optimal for none. The best approach depends on the query and data distribution. If the three values have different optimal access paths — for example, value A should use Index Seek, value B should use Index Scan, and value C should use Hash Join — then no single plan can serve all three optimally. In that case, I would split the query into three separate endpoint queries, each with a literal value hardcoded: one query for value A, one for value B, one for value C. Each gets its own cached plan optimized for its specific value. The application routes to the correct endpoint based on the parameter. This is a common pattern: in EF Core, I would create three separate repository methods (GetOrdersByStatusA, GetOrdersByStatusB, GetOrdersByStatusC) instead of one method with a parameter. For the stored procedure approach, I would use OPTIMIZE FOR the most frequent value and accept that the other two values get a suboptimal but acceptable plan. I would also measure: if the suboptimal plan for value C adds 200ms but value C represents only 5% of traffic, the overall impact is 10ms average — acceptable."

<summary>Answers</summary>

1. OPTIMIZE FOR UNKNOWN, OPTIMIZE FOR (@p = value), OPTION (RECOMPILE), DISABLE_PARAMETER_SNIFFING (USE HINT), local variable trick (DECLARE @local = @param), and plan guide / Query Store plan forcing.

2. It uses the column density from the statistics object: `DENSITY * table_cardinality = estimated rows`. Density is approximately `1 / COUNT(DISTINCT column)` for equality predicates. This produces an average selectivity estimate — not as accurate as a histogram lookup for a specific value, but stable across all parameter values.

3. Each execution includes 5-15ms of CPU time for compilation (on modern hardware for a typical query). RECOMPILE becomes prohibitive above ~100-200 executions per hour. At 1000 QPS (3.6M/hour), RECOMPILE adds 5-15 seconds of CPU per second — the server will be 100% CPU-bound.

4. Local variables are assigned at runtime, AFTER the stored procedure plan is compiled. The optimizer compiles the plan before executing the variable assignment, so it cannot peek at the variable's value. In ad hoc SQL (not in a stored procedure), the entire batch is compiled as one unit, so the optimizer CAN see the variable assignment and the parameter value. The trick only works inside stored procedures or EXEC blocks.

5. OPTIMIZE FOR UNKNOWN is a query hint (available since SQL Server 2005) that disables sniffing for specified parameters. DISABLE_PARAMETER_SNIFFING is a USE HINT (SQL Server 2016+, compat level 130) that disables sniffing for ALL parameters in the query. USE HINT syntax is the newer, preferred approach. Functionally, both result in density-based estimates and are almost identical in behavior.

**Challenge 2 — Diagnose and fix a production sniffing problem**

```sql
-- Production alert: A stored procedure is slow for one parameter value

CREATE PROCEDURE Sales.GetShipmentReport
    @RegionId int,
    @StartDate date,
    @EndDate date
AS

           o.OrderTotal, c.CustomerName

    INNER JOIN Sales.Orders o ON s.OrderId = o.OrderId
    INNER JOIN Sales.Customers c ON o.CustomerId = c.CustomerId
    WHERE s.RegionId = @RegionId
      AND s.ShipDate >= @StartDate
      AND s.ShipDate < @EndDate;

-- Data:
-- RegionId 1: 500,000 shipments (50%)
-- RegionId 5: 100 shipments (0.01%)
-- RegionId 10: 200,000 shipments (20%)

-- Symptom: When users pick RegionId = 5, the report runs in 3 seconds.
-- When they pick RegionId = 1, it takes 4 minutes.
-- What is happening and how do you fix it?
```

<details> <summary>Solution</summary>

**Root cause:** The first execution (or the most recent plan cache refresh) sniffed RegionId = 5 (or another selective region). The optimizer chose a plan using Index Seek on IX_Shipments_RegionId + Nested Loops joining to Orders and Customers. This plan is excellent for RegionId = 5 (100 rows, 100 seeks + lookups) but catastrophic for RegionId = 1 (500,000 rows, 500,000 seeks + lookups). The 500,000 key lookups cause the 4-minute runtime.

**Fix option 1 (best for this case): OPTIMIZE FOR the most common value**
```sql
ALTER PROCEDURE Sales.GetShipmentReport
    @RegionId int,
    @StartDate date,
    @EndDate date
AS
    SELECT ...
    FROM ...
    WHERE s.RegionId = @RegionId
      AND s.ShipDate >= @StartDate
      AND s.ShipDate < @EndDate
    OPTION (OPTIMIZE FOR (@RegionId = 1));
```
This optimizes for the most common region (50% of rows), choosing a scan plan. Region 5 gets a scan (suboptimal but for 100 rows, a scan is ~50 pages, so maybe 50ms instead of 3 seconds — acceptable tradeoff).

Create separate endpoints for "small region report" and "large region report" with different hints.

**Challenge 3 — Explain why the local variable trick doesn't work everywhere**

```sql
-- Why does this NOT prevent parameter sniffing?
DECLARE @LocalStatus tinyint = @Status;
SELECT * FROM Sales.Orders WHERE Status = @LocalStatus;

-- But this does?
-- (inside a stored procedure)
CREATE PROCEDURE Sales.TestLocalVar
    @Status tinyint
AS
BEGIN
    DECLARE @LocalStatus tinyint = @Status;
    SELECT * FROM Sales.Orders WHERE Status = @LocalStatus;
END;
```

<details> <summary>Solution</summary>

The difference is the compilation boundary. In the ad hoc SQL batch, the entire script is compiled as one unit before execution. The optimizer sees the `DECLARE @LocalStatus = @Status` and the subsequent `WHERE Status = @LocalStatus` in the same compilation scope. It can track the value of @LocalStatus from the parameter @Status (which IS sniffable). So sniffing still occurs.

In the stored procedure, the plan is compiled when the procedure is first executed (or recompiled). At that point, the procedure has parameters (@Status) but the `DECLARE @LocalStatus = @Status` is an assignment that happens at runtime, AFTER compilation. The optimizer cannot peek into runtime variable assignments. It sees `@LocalStatus` as an unknown variable and falls back to density-based estimates.

**Exception:** If the stored procedure uses OPTION (RECOMPILE), the local variable trick is unnecessary because RECOMPILE handles sniffing re-evaluation differently.

</details>

**Challenge 4 — Design a sniffing mitigation strategy for a multi-tenant SaaS application**

**Scenario:** Your .NET SaaS application has a shared SQL Server database with per-tenant data identified by TenantId. 2. **Medium tenants:** Use the default parameterized query with OPTIMIZE FOR UNKNOWN or a density-based approach. This provides a reasonable compromise plan for tenants with 1K-100K rows.

3. **Small tenants (< 1K rows):** A separate query path with OPTIMIZE FOR (@TenantId = <small_tenant_id>) or even OPTION (RECOMPILE) since small-tenant queries are fast anyway.

```sql
            WHEN N % 100 < 15 THEN 1   -- 13%
            WHEN N % 100 < 80 THEN 2   -- 65%
            ELSE 3                      -- 20%
        END,
        ROUND(RAND(CHECKSUM(NEWID())) * 1000, 2)
    FROM Numbers;
    
    CREATE INDEX IX_SniffingDemo_CategoryId ON Sales.SniffingDemo(CategoryId);
END
```
