# 8.344 — Execution Plans: Estimated vs Actual

## Section 1 — Navigation

**Breadcrumb:** Domain 8 — Databases > Group 13 — SQL Server Performance & Tuning > 8.344

**Previous:** [[8.343 — Execution Plans — Reading Graphical Plans]]  
**Next:** [[8.345 — Execution Plans — XML Plan Analysis]]

**Where This Fits:** The estimated plan is the optimizer's prediction before execution. The actual plan captures what really happened — real row counts, execution counts, logical reads, and spills. Comparing these two is the most powerful diagnostic technique in query tuning: every operator where EstimatedRows deviates from ActualRows by more than 10x points to a root cause. This note covers capture methods, comparison techniques, and the diagnostic value of discrepancies.

## Section 2 — Core Mental Model

`mermaid

    end
    subgraph "Execution Time"
        F[Actual Execution] --> G[Runtime Statistics]
        G --> H[ActualRows, ActualReads, ActualEndOfScans]
        G --> I[Number of Executions]
        G --> J[Spills, Warnings, Memory Usage]
    end
    C -.->|Compare| G
    G --> K[Cardinality Mismatch >10x = Root Cause]
`

**Core Principle:** The estimated plan is a prediction. The actual plan is ground truth. Every operator where the two differ significantly represents a failure of the optimizer's cost model — and therefore a tuning opportunity. The most critical comparison is Estimated Number of Rows vs Actual Number of Rows per operator.

### What Each Plan Type Tells You

| Aspect | Estimated Plan | Actual Plan |
|---|---|---|
| Row counts | Optimizer's prediction (from stats + CE) | Real row counts from execution |
| Operator costs | Cost model estimate | Real I/O and CPU (not directly shown) |
| Plan shape | What optimizer intended | What actually ran (may differ) |
| Executions | Always 1 per operator | Actual count (Nested Loops inner) |
| Memory grant | Compile-time allocation | Actual memory used |
| Spills | Not predicted | Captured if they occur |
| Warnings | Compile-time (missing stats, etc.) | Runtime (spills, memory) |
| Per-thread stats | Not available | Row distribution per thread (parallel) |

### The Central Diagnostic Question

For every operator in the plan, ask: **Is EstimatedRows close to ActualRows?**

- Within 2x → CE and stats are working well. Look elsewhere for performance issues.
- 2-10x → Moderate concern. May not change plan shape but worth investigating.
- 10-100x → Significant mismatch. The optimizer likely chose the wrong operator.
- 100x+ → Critical. The plan is fundamentally misoptimized.

## Section 3 — Deep Mechanics

### Capture Method Comparison

| Method | Syntax / Action | Plan Type | Overhead | Runtime Data | Best For |
|---|---|---|---|---|---|
| Ctrl+L (SSMS) | Query → Display Estimated Execution Plan | Estimated | None | No | What-if analysis |
| Ctrl+M (SSMS) | Query → Include Actual Execution Plan | Actual | Minimal (UI) | Yes | Ad-hoc debugging |
| SET SHOWPLAN_XML ON | SET SHOWPLAN_XML ON; query; SET SHOWPLAN_XML OFF; | Estimated | None | No | Scripting, comparison |
| SET STATISTICS XML ON | SET STATISTICS XML ON; query; SET STATISTICS XML OFF; | Actual | 1-3% | Yes | Scripted diagnostics |
| SET STATISTICS PROFILE ON | SET STATISTICS PROFILE ON; query; SET STATISTICS PROFILE OFF; | Actual | 3-8% | Yes | Tabular row-by-row analysis |
| sys.dm_exec_query_plan | SELECT query_plan FROM sys.dm_exec_query_plan(handle) | Estimated | None | No | Production monitoring |
| Query Store | sys.query_store_plan + sys.query_store_runtime_stats | Both | <1% | Yes (aggregated) | Continuous monitoring |
| Extended Events | XEvent with query_post_execution_plan_showplan | Actual | 1-5% | Yes | Deep diagnostics |

### What Each Output Looks Like

**SET SHOWPLAN_XML ON:**
Output: Single row with XML column. The XML has StmtSimple with StatementOptmLevel and CardinalityEstimationModelVersion, but NO RunTimeInformation elements. Each RelOp has EstimateRows, EstimateIO, EstimateCPU, EstimatedTotalSubtreeCost but no actual counters.

**SET STATISTICS XML ON:**
Output: Regular query result sets PLUS a final result set with the XML plan. The XML includes RunTimeInformation with RunTimeCountersPerThread containing ActualRows, ActualEndOfScans, ActualReads, ActualMemoryKB. Best format for automated analysis.

**SET STATISTICS PROFILE ON:**
Output: Tabular rowset with columns: Rows (actual), Executes, StmtText (operator tree with indentation), EstimateRows, EstimateIO, EstimateCPU, TotalSubtreeCost.  No XML — pure tabular. Good for quick visual comparison without XML parsing.

### The Underlying Schema

In the ShowPlanXML, the distinction between estimated and actual plans is:
- **Estimated:** `<RelOp>` elements have `@EstimateRows`, `@EstimateIO`, `@EstimateCPU`, `@EstimatedTotalSubtreeCost`
- **Actual:** All the above PLUS `<RunTimeInformation>` child element with `<RunTimeCountersPerThread>` containing `@ActualRows`, etc.
- The XPath `.exist('//p:RunTimeInformation')` returns 1 for actual plans, 0 for estimated.

### Key Differences in Behavior

1. **Cached vs Fresh:** Ctrl+L (estimated) always generates a fresh plan. Ctrl+M (actual) may use a previously cached plan. This is why the two can show different operators — the cached plan was compiled with different statistics or parameter values.

2. **Multiple Executions:** In estimated plans, each operator appears once. In actual plans, the Number of Executions may be > 1 (Nested Loops inner side). EstimatedRows is the per-execution estimate; multiply by Number of Executions to get total estimated rows processed.

3. **Batch Mode:** Batch mode operators (columnstore) may appear only in the actual plan because they are runtime-optimized. The estimated plan may show row mode operators that are upgraded to batch mode during execution.

4. **Bitmaps:** Similarly, bitmap filters may appear only in actual plans — they are created at runtime from the build side of a hash join.

## Section 4 — Production Patterns

### Pattern 1: Quick Cardinality Mismatch Detection

Run this script on any production system to find queries with >10x cardinality mismatches:

`sql
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
SELECT TOP 20 qs.query_hash,
    qs.execution_count,
    qs.total_logical_reads / qs.execution_count AS avg_reads,
    qs.total_elapsed_time / qs.execution_count / 1000 AS avg_ms,
    (SELECT MAX(CASE 
        WHEN t.c.value('@EstimateRows','FLOAT') > 0
        THEN t2.r.value('@ActualRows','BIGINT') / t.c.value('@EstimateRows','FLOAT')
        ELSE NULL END)
     FROM qp.query_plan.nodes('//p:RelOp') t(c)
     OUTER APPLY t.c.nodes('p:RunTimeInformation/p:RunTimeCountersPerThread') t2(r)) AS max_mismatch_ratio
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
WHERE qp.query_plan IS NOT NULL
    AND qp.query_plan.exist('//p:RunTimeInformation') = 0  -- Only plans without runtime info
ORDER BY qs.total_elapsed_time DESC;
`

> **Note:** `sys.dm_exec_query_plan` stores estimated plans only. To get actual plans from cache, use Query Store or capture via Extended Events.

### Pattern 2: Detecting Compile vs Runtime Plan Differences

When the same query_hash has multiple plan_ids in Query Store, compare them:

`sql
SELECT q.query_id, p.plan_id, p.query_plan_hash,
    p.initial_compile_start_time,
    p.last_compile_start_time,
    rs.avg_duration / 1000 AS avg_ms,
    rs.avg_logical_reads,
    rs.avg_rows_actual,
    rs.avg_rows_estimated,
    CASE WHEN rs.avg_rows_actual > rs.avg_rows_estimated * 10 
         THEN 'UNDER-ESTIMATE'
         WHEN rs.avg_rows_estimated > rs.avg_rows_actual * 10
         THEN 'OVER-ESTIMATE'
         ELSE 'OK' END AS cardinality_status
FROM sys.query_store_query q
INNER JOIN sys.query_store_plan p ON q.query_id = p.query_id
INNER JOIN sys.query_store_runtime_stats rs ON p.plan_id = rs.plan_id
WHERE q.query_id = @query_id
ORDER BY p.last_compile_start_time DESC;
`

### Pattern 3: Parameter Sniffing Diagnosis via Plan Comparison

Capture the same query with two different parameter values and compare plans:

`sql
-- Create procedure with sniffing
CREATE OR ALTER PROCEDURE dbo.GetOrdersByStatus @Status VARCHAR(20)
AS
    SELECT o.OrderId, o.OrderDate, o.CustomerId
    FROM Orders o
    WHERE o.Status = @Status
    ORDER BY o.OrderDate;

-- Execute with selective and non-selective values
EXEC dbo.GetOrdersByStatus 'Pending';    -- few rows
EXEC dbo.GetOrdersByStatus 'Shipped';    -- many rows

-- Compare plans in Query Store
SELECT qsq.query_id, qsp.plan_id,
    qsqt.query_sql_text,
    qsp.cardinality_estimation_model_version,
    qrs.avg_duration / 1000 AS avg_ms,
    qrs.avg_logical_reads,
    qrs.avg_rows_actual,
    qrs.avg_rows_estimated
FROM sys.query_store_query qsq
JOIN sys.query_store_query_text qsqt ON qsq.query_text_id = qsqt.query_text_id
JOIN sys.query_store_plan qsp ON qsq.query_id = qsp.query_id
JOIN sys.query_store_runtime_stats qrs ON qsp.plan_id = qrs.plan_id
WHERE qsqt.query_sql_text LIKE '%GetOrdersByStatus%'
ORDER BY qrs.last_execution_time DESC;
`

### Pattern 4: Identifying False Low Estimates from ASCENDING Keys

When a query filters on a column with increasing values (OrderDate, Identity), the histogram may not cover new values:

`sql
CREATE PROCEDURE dbo.CheckAscendingKeyEstimate
    @table_name NVARCHAR(128), @column_name NVARCHAR(128)
AS
BEGIN
    DECLARE @sql NVARCHAR(MAX);
    DECLARE @max_val SQL_VARIANT;
    
    -- Get the current maximum value
    SET @sql = N'SELECT @max_val = MAX(' + QUOTENAME(@column_name) + ') FROM ' + @table_name;
    EXEC sp_executesql @sql, N'@max_val SQL_VARIANT OUTPUT', @max_val = @max_val OUTPUT;
    
    -- Get the histogram max value from statistics
    SET @sql = N'
    SELECT STATS_DATE(OBJECT_ID(''' + @table_name + '''), 
           (SELECT stats_id FROM sys.stats WHERE object_id = OBJECT_ID(''' + @table_name + ''') 
            AND name LIKE ''_WA_Sys_%' + @column_name + '%'')) AS stats_last_updated,
           sp.rows, sp.modification_counter,
           CONVERT(NVARCHAR(100), @max_val) AS actual_max
    FROM sys.dm_db_stats_properties(OBJECT_ID(''' + @table_name + '''), 
         (SELECT stats_id FROM sys.stats WHERE object_id = OBJECT_ID(''' + @table_name + ''') 
          AND name LIKE ''_WA_Sys_%' + @column_name + '%'')) sp';
    
    EXEC sp_executesql @sql, N'@max_val SQL_VARIANT', @max_val;
END;
`

### Pattern 5: EF Core — Plan Cache Pollution Detection

When EF Core generates many similar-but-not-identical queries, each gets a cached plan. The estimated plan for each may be fresh (no cache hit), but the actual plan uses a cached version of a similar query. This causes plan cache bloat:

`csharp
public class PlanCacheInterceptor : DbCommandInterceptor
{
    private readonly HashSet<string> _seenQueries = new();
    
    public override InterceptionResult<DbDataReader> ReaderExecuting(
        DbCommand command, CommandEventData eventData, 
        InterceptionResult<DbDataReader> result)
    {
        if (!_seenQueries.Contains(command.CommandText))
        {
            _seenQueries.Add(command.CommandText);
            // Log first-time queries to identify non-parameterized SQL
            Debug.WriteLine($"NEW PLAN: {command.CommandText}");
        }
        return base.ReaderExecuting(command, eventData, result);
    }
}
`

### Pattern 6: Estimating the Cost of Plan Capture in Production

To decide whether capture overhead is acceptable for a given query:

`sql
-- Measure overhead of STATISTICS XML ON for a representative query
DECLARE @start DATETIME2, @baseline_ms INT, @with_stats_ms INT;

SET @start = SYSDATETIME();
EXEC dbo.YourRepresentativeQuery;  -- without capture
SET @baseline_ms = DATEDIFF(MS, @start, SYSDATETIME());

SET @start = SYSDATETIME();
SET STATISTICS XML ON;
EXEC dbo.YourRepresentativeQuery;
SET STATISTICS XML OFF;
SET @with_stats_ms = DATEDIFF(MS, @start, SYSDATETIME());

SELECT @baseline_ms AS baseline_ms, @with_stats_ms AS with_stats_ms,
    (1.0 * @with_stats_ms - @baseline_ms) / NULLIF(@baseline_ms, 0) * 100 AS overhead_pct;
`

### Pattern 7: STATISTICS PROFILE Tabular Analysis

The tabular output of STATISTICS PROFILE is underused. Each row shows:
- **Rows** (column 1): Actual rows flowing through this operator
- **Executes** (column 2): Number of executions
- **StmtText** (column 3): Operator name with indentation
- **EstimateRows** (column 4): Estimated rows
- **EstimateIO** (column 5): Estimated I/O cost
- **EstimateCPU** (column 6): Estimated CPU cost

To find the problematic operator, sort the output by `|Rows - EstimateRows|` manually — the largest discrepancy is the root cause. Multiple rows with the same StmtText but different indentation represent the same operator executed multiple times (Nested Loops inner).

## Section 5 — Gotchas

### Gotcha 1 — Actual Plan Shows Different Operators Than Estimated
**Pitfall:** You optimize based on estimated plan (Ctrl+L), but actual execution uses different operators because the plan was cached.
**Fix:** Always capture actual plans from production. Use DBCC FREEPROCCACHE or RECOMPILE.
**Cost:** Wasted hours redesigning indexes that don't help.

### Gotcha 2 — SET STATISTICS PROFILE Adds Overhead
**Pitfall:** Running in production adds 5-10% profiling overhead.
**Fix:** Use Query Store for continuous monitoring (low overhead). Use STATISTICS XML ON instead.
**Cost:** 5-10% performance degradation.

### Gotcha 3 — Multiple Executions of the Same Operator
**Pitfall:** Nested Loops tooltip shows Actual Rows=1 per execution, but total is 50,000. Multiply Actual Rows by Number of Executions.
**Fix:** Look for high Number of Executions with thin outer arrow.
**Cost:** Missed optimization of wrong join type.

### Gotcha 4 — Actual Plan Captures Only Last Execution
**Pitfall:** SET STATISTICS XML ON shows runtime for that specific execution only, which may not be typical.
**Fix:** Use Query Store for average statistics across many executions.
**Cost:** Optimizing for a rare parameter value.

### Gotcha 5 — Parallel Plans Show Aggregated Actual Rows
**Pitfall:** Tooltip sums across all threads. Individual threads may have imbalanced distribution.
**Fix:** Examine RunTimeInformation in XML for per-thread counts.
**Cost:** Parallelism inefficiency invisible in summary view.

### Gotcha 6 — Estimated Rows Can Be Scientific Notation
**Pitfall:** Tooltip shows 2.3456e-005, rounds to 0 in UI.
**Fix:** Hover carefully or read plan XML with full precision.
**Cost:** False assumption about missing statistics.

## Section 6 — Performance Implications

### Demo: Parameter Sniffing Impact

`sql
CREATE TABLE #Transactions (TransactionId INT IDENTITY(1,1), AccountId INT, Amount DECIMAL(10,2), TransactionDate DATETIME, Type VARCHAR(20));
INSERT INTO #Transactions (AccountId, Amount, TransactionDate, Type)
SELECT 1+ABS(CHECKSUM(NEWID()))%1000,
    CASE WHEN RAND(CHECKSUM(NEWID()))<0.9 THEN RAND()*100 ELSE RAND()*10000 END,
    DATEADD(DAY, -ABS(CHECKSUM(NEWID()))%365, GETDATE()),
    CASE WHEN RAND(CHECKSUM(NEWID()))<0.9 THEN 'Regular' ELSE 'Bulk' END
FROM (SELECT TOP 100000 ROW_NUMBER() OVER (ORDER BY (SELECT NULL)) AS N FROM sys.all_columns a CROSS JOIN sys.all_columns b) t;
CREATE STATISTICS s_Type ON #Transactions(Type) WITH FULLSCAN;

SET STATISTICS PROFILE ON;
SELECT COUNT(*), SUM(Amount) FROM #Transactions WHERE Type = 'Regular' OPTION (RECOMPILE);
SELECT COUNT(*), SUM(Amount) FROM #Transactions WHERE Type = 'Bulk' OPTION (RECOMPILE);
SET STATISTICS PROFILE OFF;
`

**Sniffing Scenario:** Procedure compiled for Regular (90K rows, Hash Join). When called with Bulk (10K rows), reused cached plan with Hash Join (overkill for 10K rows). Over-estimate leads to unnecessary memory grant.

| Scenario | Estimated | Actual | Mismatch | Join Type | Logical Reads | Duration |
|---|---|---|---|---|---|---|
| NL under-estimate | 10 | 100,000 | 10,000x | NL (should be Hash) | 1,500,000 | 45s |
| Hash over-estimate | 500,000 | 50 | 10,000x | Hash (should be NL) | 50,000 | 2s |
| Correct (baseline) | 100,000 | 100,000 | 1x | Hash | 2,500 | 800ms |

## Section 7 — Interview Arsenal

### Interview Questions

**Q1: What is the difference between an estimated and actual execution plan?**
*Tier 1:* Estimated is the optimizer's prediction before execution. Actual has runtime statistics after execution.

*Tier 2:* The estimated plan is the output of cost-based optimization using statistics and CE. The actual plan captures real runtime data - rows, executions, spills, reads. The most critical comparison is EstimatedRows vs ActualRows per operator. Over 10x discrepancy means the cost model was wrong. I use SET STATISTICS PROFILE ON for quick analysis and Query Store for continuous monitoring.

**Q2: How do you detect parameter sniffing from estimated vs actual?**
*Spoken answer:* If the compiled plan shows estimated rows matching the sniffed parameter value but actual execution processes different rows, that indicates sniffing. Plan XML shows compiled value. Query Store tracks both plan_id and runtime stats.

**Q3: What tools capture actual execution plans?**
*Spoken answer:* SSMS Ctrl+M, SET STATISTICS XML ON, SET STATISTICS PROFILE ON, sys.dm_exec_query_stats + sys.dm_exec_query_plan, Query Store, Extended Events.

**Q4: What does a 1000x cardinality mismatch mean?**
*Spoken answer:* The optimizer's cost model was wrong by three orders of magnitude. Under-estimate in Nested Loop outer makes inner execute 1000x more times. Fix depends on cause: missing stats, stale stats, correlation, ASCENDING key.

**Q5: When would you trust estimated but not actual plan?**
*Spoken answer:* Estimated plans are useful for what-if analysis - testing what indexes would help without running the query. But I never trust estimated costs for performance numbers. For production diagnosis, always use actual plans.

**Q6: How does Rows column in STATISTICS PROFILE differ from EstimateRows?**
*Spoken answer:* Rows = actual rows flowed through operator. EstimateRows = optimizer's prediction. Rows consistently exceeding EstimateRows = wrong stats or CE model.

**Q7: What causes estimated vs actual operator differences?**
*Spoken answer:* (1) Plan cached from earlier compilation. (2) Batch mode or bitmap adds runtime-only ops. (3) Plan forced via Query Store.

**Q8: Relationship between SET STATISTICS PROFILE and XML plan?**
*Spoken answer:* PROFILE produces tabular rowset with Rows, Executes, StmtText, EstimateRows, EstimateIO, EstimateCPU. XML plan has same info plus CE version, memory grant, warnings, per-thread runtime.

## Section 8 — Decision Framework

`mermaid
flowchart TD
    A[Capture Actual Plan] --> B{Check Estimate vs Actual}
    B -->|Within 2x| C[Stats and CE accurate]
    C --> D{Still slow?}
    D -->|Yes| E[Check join types, parallelism, memory]
    D -->|No| F[Done]
    B -->|>10x mismatch| G{Root cause}
    G -->|Estimate << Actual| H[Under-estimate]
    G -->|Estimate >> Actual| I[Over-estimate]
    H --> J{Operator type}
    J -->|Scan low estimate| K[Missing/stale stats]
    J -->|NL high executions| L[Wrong join type]
    J -->|Sort low estimate| M[Sort spill to tempdb]
    I --> N{Operator type}
    N -->|Hash Join| O[Over-estimated rows]
    N -->|Memory Grant| P[Excessive reservation]
    K --> Q[Update stats or multi-column stats]
    L --> R[Add join hint or update stats]
    M --> S[Add index matching sort order]
    O --> T[Add OPTIMIZE FOR UNKNOWN or RECOMPILE]
`

### Decision Checklist

1. Capture actual plan with runtime statistics
2. For each operator, compare Actual Rows vs Estimated Rows
3. Mark operators with mismatch ratio > 10x or < 0.1x
4. Determine cause: missing stats? stale stats? correlation? ASCENDING key? sniffing?
5. Apply fix: update stats, add index, change CE, add hint
6. Re-run and verify improvement
7. Use Query Store to monitor fix persistence

### Tradeoffs

| Action | Benefit | Risk | When |
|---|---|---|---|
| Update stats FULLSCAN | Most accurate | CPU/IO spike | After bulk loads |
| Multi-column stats | Fixes correlation | Maintenance | Known correlated pairs |
| RECOMPILE | Fresh plan per execution | CPU increase | Parameter-sensitive queries |
| OPTIMIZE FOR UNKNOWN | Average plan | No best plan | Balanced distribution |
| Plan forcing | Stability | Masks issues | Critical SLAs |
| CE version change | Different model | Regressions | After specific analysis |

### Scale Thresholds

- <1M rows: Mismatches rarely cause problems
- 1M-100M: Single 10x mismatch can turn 100ms into 30s
- >100M: Any mismatch over 5x is critical
- >1B: Even 2x mismatches matter. Use batch mode and CE150 feedback.

## Section 6 — Performance Implications (Extended)

### Deep Dive: Cost of Cardinality Mismatch

When the optimizer under-estimates by 10x, the consequences cascade:

1. **Memory Grant:** If the optimizer expects 100 rows but actual is 10,000, the memory grant for a hash join is sized for 100 rows. The hash spills to tempdb, causing physical I/O.
2. **Join Type:** A Nested Loop (good for small outer) iterates 10,000 times instead of 100. Each iteration does an index seek. Total: 10,000 seeks vs 100 seeks.
3. **Parallelism:** If the estimate drops below the cost threshold for parallelism, the plan becomes serial. A serial plan on a multi-core server wastes 90% of available CPU.
4. **Spooling:** If the estimate is too low, SQL Server may not choose a spool to protect from Halloween protection, causing incorrect results or re-scans.

### Benchmark Script: Nested Loop Under-Estimate

`sql

CREATE TABLE #Large (Id INT IDENTITY(1,1), SmallId INT, Value DECIMAL(10,2));

### Interpreting STATISTICS IO Output

`
Table: #Large. Scan count 1, logical reads 4523, physical reads 0, read-ahead reads 0
Table: #Small. Scan count 1, logical reads 1, physical reads 0, read-ahead reads 0
`

- **Scan count:** Number of times the table was scanned. 1 = single scan. >1 = re-scan (Nested Loop inner side).
- **Logical reads:** Number of pages read from the buffer pool. Each page is 8KB. This is the primary performance metric.
- **Physical reads:** Pages read from disk (cache miss). Should be near 0 for warm cache.
- **Read-ahead reads:** Pages prefetched by the read-ahead mechanism.

## Section 7 — Interview Arsenal (Extended)

### Q9: How do you use STATISTICS PROFILE to find the most expensive operator?
*Spoken answer:* Look at the Rows column (actual) vs EstimateRows column. The operator with the largest ratio of EstimateRows to Rows (or vice versa) is the root cause. Then check the Executes column - if any operator has Executes > 1 and high actual rows, that is the performance bottleneck. The StmtText column shows the operator tree with indentation indicating parent-child relationships.

### Q10: What is the difference between elapsed time and CPU time in STATISTICS TIME?
*Spoken answer:* CPU time is the total CPU processing time (sum across all threads). Elapsed time is wall-clock time from start to finish. If elapsed time > CPU time, the query waited on something (I/O, locks, memory grant). If CPU time > elapsed time, the query used parallelism (multiple CPUs processing simultaneously). High elapsed with low CPU = blocking or I/O bottleneck.

### Q11: How do you detect when a query is spilling to tempdb from STATISTICS PROFILE?
*Spoken answer:* Look for a Sort or Hash Match operator with a yellow warning icon. In the profile output, these operators will show significant logical reads but few actual rows (because most rows went to tempdb, not through the operator). You can also check sys.dm_exec_query_stats.total_spills.

### Q12: What does it mean when the actual plan shows a different number of operators than the estimated plan?
*Spoken answer:* This is rare but can happen with:
1. Bitmap filters added at runtime (hash join optimization)
2. Batch mode operators added for columnstore scans
3. Runtime statistics showing spools that were compile-time decisions
4. Adaptive joins where the threshold operator appears only at runtime
In all cases, the core operator tree is the same; only optimization operators are added or removed at runtime.

## Section 8 — Decision Framework (Extended)

### Root Cause Decision Tree Detailed

When you find a mismatch > 10x, follow this detailed path:

`
Mismatch > 10x
├── Under-estimate (Actual >> Estimated)
│   ├── Check if predicate value is at histogram edge -> ASCENDING key issue
│   │   └── Fix: Update stats with FULLSCAN or enable CE150 feedback
│   ├── Check if multiple predicates -> possible correlation
│   │   └── Fix: Multi-column statistics or CE70/CE120 switch
│   ├── Check if join -> missing statistics on foreign key
│   │   └── Fix: Create statistics on join column
│   └── Check if parameterized -> parameter sniffing
│       └── Fix: RECOMPILE or OPTIMIZE FOR UNKNOWN
├── Over-estimate (Estimated >> Actual)
│   ├── Check if multiple independent predicates -> CE over-correlation
│   │   └── Fix: ASSUME_MIN_SELECTIVITY hint or CE70
│   ├── Check if join with filter -> base containment under-estimate
│   │   └── Fix: ASSUME_JOIN_PREDICATE_DEPENDENCE hint
│   └── Check if TVF -> 1-row estimate bug (pre-CE140)
│       └── Fix: Upgrade compatibility level or inline TVF
`

### Proactive Monitoring Setup

`sql
-- Set up daily job to check for new cardinality mismatches
CREATE PROCEDURE dbo.CheckCardinalityMismatches
    @threshold DECIMAL(10,2) = 10.0
AS
INSERT INTO dbo.CardinalityMismatchLog (check_date, query_hash, mismatch_ratio, plan_handle)
SELECT GETDATE(), qs.query_hash,

) qs
GROUP BY qs.query_hash, qs.plan_handle
HAVING CASE WHEN MAX(qs.total_rows) > MAX(qs.total_estimated_rows)
            THEN MAX(qs.total_rows) / NULLIF(MAX(qs.total_estimated_rows), 1)
            ELSE MAX(qs.total_estimated_rows) / NULLIF(MAX(qs.total_rows), 1)

### Additional Conceptual Questions

**Q11:** Can SET STATISTICS PROFILE ON cause incorrect results?
<details>No. It captures runtime statistics but does not change query semantics. However, it adds overhead that may cause timeouts for near-threshold queries. It never changes result sets.</details>

**Q12:** What is the difference between STATISTICS XML and SHOWPLAN XML output?
<details>SHOWPLAN XML returns the estimated plan as a single row with an XML column. STATISTICS XML returns the actual plan in the same format but also executes the query and returns the query result set along with the plan. STATISTICS XML output has the plan in the last result set.</details>

**Q13:** How do you preserve the actual plan XML for later analysis?
<details>Save it from SSMS: right-click on the plan tab > Save Execution Plan As > .sqlplan file. Or capture it to a table: INSERT INTO PlanCache (plan_xml) SELECT query_plan FROM sys.dm_exec_query_plan(handle).</details>

**Q14:** What is the impact of RECOMPILE on estimated vs actual plan alignment?
<details>RECOMPILE forces a new compilation before every execution, so the estimated plan matches the actual execution. This eliminates plan caching effects but increases CPU usage. For frequently executed queries, the compilation overhead can outweigh the benefits.</details>

**Q15:** How do you compare the same query across two different environments (dev vs prod) using actual plans?
<details>Capture actual plans from both environments using SET STATISTICS XML ON. Compare: (1) Estimated rows per operator - are stats similar? (2) Operators chosen - same join types? (3) Actual rows per operator - same data distribution? (4) Logical reads - hardware differences or missing indexes? (5) CE version - could be different compatibility levels.</details>

### Case Study: Estimated vs Actual in a Data Warehouse Query

**Scenario:** A nightly ETL query that aggregates 50M fact rows joined to 3 dimension tables. After a data volume increase, query duration went from 4 minutes to 45 minutes.

**Initial Estimated Plan (Ctrl+L):**
- Hash Match joins on all 3 dimensions
- MemoryGrant: 4GB
- Estimated rows: 12M after first join
- Operators: 18 total

**Actual Plan (SET STATISTICS XML ON):**
- Actual rows: 48M after first join (4x estimate)
- Hash Match spills to tempdb (ActualRows 48M vs EstimateRows 12M = 4x mismatch)
- MemoryGrant: 4GB granted, but 12GB needed → spill
- Spill writes 8GB to tempdb
- Actual I/O: 450K logical reads (estimated: 120K)

**Root Cause:** The statistics on the fact table were last updated before a bulk insert added 3x more rows. The modification_counter was 45M, exceeding the auto-update threshold, but the auto-update sampled only 20% of the rows, producing a stale density vector.

**Resolution:**
1. UPDATE STATISTICS FactTable WITH FULLSCAN
2. After fix: EstimateRows = 46M, ActualRows = 48M (within 5%)
3. No spill: 4GB memory grant was sufficient for correct estimate
4. Duration: 4 minutes 12 seconds

**Key Learning:** A 4x estimate mismatch caused a spill that added 40 minutes to the query. Fixing the statistics alignment eliminated the mismatch entirely. The estimated plan alone would never have shown this problem — only the actual plan revealed the spill.

## Section 9 — Advanced Scenarios

### Scenario-Based Questions

**SQ1:** You run a query with SET STATISTICS XML ON and get a plan. You run the same query again with Ctrl+M and get a different plan. Why?
<details>SET STATISTICS XML ON forces a new compilation (the STATISTICS settings are reset per execution). Ctrl+M uses the cached plan from the first execution (if any). The first execution compiles and caches a plan; Ctrl+M reuses the cached plan. To get consistent results, run SET STATISTICS XML ON first (to capture the fresh plan), then Ctrl+M (to capture the cached plan reuse).</details>

**SQ2:** A stored procedure has Avg Duration = 500ms in Query Store but occasionally spikes to 30 seconds. Estimated vs actual analysis shows the plan is the same. What else could cause the spike?
<details>The same plan with wildly different duration indicates a runtime issue, not a cardinality issue: (1) Blocking — another transaction holds locks on the queried rows, (2) Memory grant wait — concurrent queries exhaust memory, (3) I/O subsystem latency spike — other workloads saturating the disk, (4) CPU pressure — other queries consuming CPU on the same server, (5) Waits — check sys.dm_exec_requests.wait_type for the spike period. The plan is stable; the environment is not.</details>

**SQ3:** You see an operator with EstimateRows = 1 and ActualRows = 1,000,000. The operator is a Nested Loops join. The query uses a multi-statement TVF. What is the root cause?
<details>Multi-statement TVFs are estimated as 1 row under CE70 and CE120. The TVF actually returns 1M rows, but the optimizer thinks it returns 1 row. The Nested Loops join then executes the inner side 1M times (once per actual TVF row). Fix: (1) Inline the TVF if possible, (2) Upgrade compatibility level to 130+ (CE140+ fixes TVF estimation), (3) Split into a temp table population step.</details>

**SQ4:** A query shows EstimatedRows = 500,000 and ActualRows = 5,000. The plan uses Hash Match Join. What is the symptom and fix?
<details>This is an over-estimate (optimizer expected 500K, actual is 5K). Hash Match is overkill for 5K rows — Nested Loops would be faster (no memory grant, no hash build). The over-estimate causes excessive memory grant (sized for 500K rows), potentially causing memory grant wait. Fix: (1) Check if statistics are stale — update with FULLSCAN, (2) If estimates are always too high, use OPTIMIZE FOR UNKNOWN, (3) For independent predicates, try ASSUME_MIN_SELECTIVITY hint, (4) Consider using join hint INNER LOOP JOIN to force Nested Loops.</details>

**SQ5:** How do you verify that a plan forcing via Query Store is actually being used? Compare estimated vs actual.
<details>Check sys.query_store_plan.is_forced_plan = 1. Additionally, in the actual plan (Ctrl+M), check the Root operator tooltip: the "Statement" section shows the plan_id and whether it was forced. If the plan was forced, the forced_plan_id will match the plan_id from Query Store. Compare the actual plan operators to the forced plan — if they match, the forcing succeeded. If different, check last_force_failure_reason in sys.query_store_plan.</details>

### Extended Self-Check Questions

**Q16:** What is the difference between SET STATISTICS XML ON and SET SHOWPLAN XML ON in terms of query execution?
<details>SET SHOWPLAN XML ON does not execute the query — it returns only the estimated plan. The query never runs; no results returned. SET STATISTICS XML ON executes the query AND returns the actual plan. The query runs, returns results, AND appends the plan XML as a final result set. Use SHOWPLAN_XML for safe what-if analysis; use STATISTICS XML for actual performance diagnosis.</details>

**Q17:** How do you find queries where the estimated plan differs from the actual plan in terms of operator count?
<details>This indicates a plan cache change between compilation and execution. Likely causes: (1) Plan was invalidated between estimated and actual capture, (2) The estimated plan was manually generated fresh (Ctrl+L) while the actual plan uses a cached version, (3) Runtime plan modifications (bitmaps, batch mode) added operators at execution time. The operator count should be close (± 1-2 ops for runtime additions). A difference > 5 ops suggests a different plan altogether.</details>

**Q18:** Can SET STATISTICS PROFILE ON cause query timeout for long-running queries?
<details>Yes. The overhead of capturing profile data (3-8%) can push a near-threshold query over the timeout. For a query that normally runs in 28 seconds with a 30-second timeout, the 3% overhead adds ~0.84 seconds, potentially causing a timeout. Always test with PROFILE in a non-production environment first, or use STATISTICS XML ON (lower overhead) for production diagnostics.</details>

**Q19:** What does STMT text indentation in STATISTICS PROFILE output represent?
<details>Indentation represents the operator tree hierarchy. The SELECT operator is flush left. Each child operator is indented deeper. The most deeply indented operators are leaf operators (scans/seeks). The parent of an operator is the one directly above it with one less indentation level. This mirrors the relational operator tree structure in the graphical plan.</details>

**Q20:** How do you use sys.dm_exec_query_profiles for live estimated vs actual comparison?
<details>sys.dm_exec_query_profiles provides per-operator, per-thread live statistics during query execution. It shows physical_reads, logical_reads, rows, and elapsed_time_ms for each operator in real time. Enable with SET STATISTICS PROFILE ON (or SET STATISTICS XML ON with trace flag 7412). Useful for observing plan behavior mid-execution, identifying which operator is slowest in real time, and detecting spills as they happen.</details>

**Challenge 9:** Write a T-SQL script that re-executes a query N times with and without plan capture and computes the average overhead percentage.
<details>`sql
CREATE OR ALTER PROCEDURE dbo.MeasureCaptureOverhead
    @query NVARCHAR(MAX), @iterations INT = 50
AS
BEGIN
    SET NOCOUNT ON;
    DECLARE @i INT = 0, @baseline_start DATETIME2, @with_stats_start DATETIME2;
    DECLARE @baseline_ms INT = 0, @with_stats_ms INT = 0;
    DECLARE @sql NVARCHAR(MAX);
    
    -- Baseline
    SET @i = 0;
    SET @baseline_start = SYSDATETIME();
    WHILE @i < @iterations
    BEGIN
        SET @sql = 'EXEC sp_executesql N''' + REPLACE(@query, '''', '''''') + '''';
        EXEC sp_executesql @sql;
        SET @i = @i + 1;
    END
    SET @baseline_ms = DATEDIFF(MS, @baseline_start, SYSDATETIME());
    
    -- With STATISTICS XML
    SET @i = 0;
    SET @with_stats_start = SYSDATETIME();
    WHILE @i < @iterations
    BEGIN
        SET @sql = 'SET STATISTICS XML ON; EXEC sp_executesql N''' + REPLACE(@query, '''', '''''') + '''; SET STATISTICS XML OFF;';
        EXEC sp_executesql @sql;
        SET @i = @i + 1;
    END
    SET @with_stats_ms = DATEDIFF(MS, @with_stats_start, SYSDATETIME());
    
    SELECT @baseline_ms AS baseline_ms, @with_stats_ms AS with_stats_ms,
        @baseline_ms / @iterations AS avg_baseline_ms,
        @with_stats_ms / @iterations AS avg_with_stats_ms,
        (1.0 * @with_stats_ms - @baseline_ms) / NULLIF(@baseline_ms, 0) * 100 AS overhead_pct;
END;
`</details>

**Challenge 10:** Create a procedure that captures actual plans for all currently executing queries by session_id.
<details>`sql
CREATE OR ALTER PROCEDURE dbo.CaptureActiveQueryPlans
AS
BEGIN
    SET NOCOUNT ON;
    CREATE TABLE #ActivePlans (
        session_id INT, request_id INT,
        sql_text NVARCHAR(MAX), plan_xml XML
    );
    
    DECLARE @session_id INT, @request_id INT, @sql_handle VARBINARY(64), @plan_handle VARBINARY(64);

    SELECT session_id, request_id, sql_handle, plan_handle
    FROM sys.dm_exec_requests
    WHERE status = 'running';
    
    OPEN cur;
    FETCH NEXT FROM cur INTO @session_id, @request_id, @sql_handle, @plan_handle;

        INSERT INTO #ActivePlans (session_id, request_id, sql_text, plan_xml)

            (SELECT query_plan FROM sys.dm_exec_query_plan(@plan_handle));
        FETCH NEXT FROM cur INTO @session_id, @request_id, @sql_handle, @plan_handle;
    END
    
    CLOSE cur; DEALLOCATE cur;
    SELECT * FROM #ActivePlans;
    DROP TABLE #ActivePlans;
END;
