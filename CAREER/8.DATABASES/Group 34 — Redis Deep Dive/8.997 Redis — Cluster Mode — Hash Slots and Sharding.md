# 8.997 Redis — Cluster Mode — Hash Slots and Sharding

## Section 1 — Overview

Redis Cluster provides a way to run Redis across multiple nodes, automatically sharding data across the cluster and providing a degree of availability through automatic failover. Unlike a standalone Redis instance, a cluster distributes keys across 16,384 hash slots, each assigned to a specific master node. Clients can connect to any node and be transparently redirected to the correct node for a given key, or they can use smart client libraries like StackExchange.Redis that maintain a local slot map and route requests directly.

The core of Redis Cluster's sharding mechanism is hash slot assignment. For every key, Redis computes CRC16(key) % 16384 to determine which of the 16,384 slots the key belongs to. The cluster maintains a mapping from slot ranges to master nodes. Each master node can have zero or more replica nodes that replicate its data and can take over if the master fails.

Redis Cluster provides:
- **Scalability** — adding more nodes increases total memory capacity and throughput
- **Automatic sharding** — data is distributed across nodes without manual partitioning
- **High availability** — automatic failover when master nodes fail (replicas are promoted)
- **Decentralized architecture** — no single point of failure; all nodes communicate via gossip protocol
- **Smart client support** — clients can route requests directly to the correct node

Cluster topology consists of master nodes that own hash slot ranges and replica nodes that replicate from masters. The cluster uses a gossip protocol to exchange configuration information, detect node failures, and propagate slot assignments. Configuration is maintained as a cluster configuration epoch, ensuring that all nodes eventually agree on the state.

When a client sends a request to a node that does not own the key's slot, the node responds with a MOVED redirect containing the address of the correct node. StackExchange.Redis intercepts MOVED responses, updates its internal slot mapping, and retries the request on the correct node — all transparently to the application code.

During resharding (moving slots between nodes), Redis Cluster uses ASK redirects. ASK differs from MOVED in that it is temporary — the slot is being migrated and the client should retry on the target node. ASK redirects are single-request; the client must use ASKING before the request and does not update its permanent slot mapping.

### Hash Slot Calculation

The hash slot for a key is calculated as:

```
slot = CRC16(key) & 16383  (equivalent to CRC16(key) % 16384)
```

For keys with hash tags (denoted by `{...}`), only the part inside the braces is used for slot calculation. This allows related keys to be co-located on the same node:

```
Key: "user:123"                  → slot = CRC16("user:123") % 16384
Key: "{user:123}:profile"        → slot = CRC16("user:123") % 16384
Key: "{user:123}:settings"       → slot = CRC16("user:123") % 16384
Key: "{user:123}:orders"         → slot = CRC16("user:123") % 16384
```

All four keys hash to the same slot because only the hash tag content `user:123` is used.

### Slot Distribution

In a 3-master cluster, slots are typically distributed as:

| Node | Slot Range | Slot Count |
|------|-----------|------------|
| Node A | 0-5460 | 5461 |
| Node B | 5461-10922 | 5462 |
| Node C | 10923-16383 | 5461 |

The distribution is not always perfectly even, but it can be rebalanced using the `redis-cli --cluster rebalance` command.

### Cluster Minimum Requirements

A Redis Cluster requires:
- **At least 3 master nodes** for a production deployment (for quorum-based failure detection)
- **At least 3 replica nodes** (recommended: 1 replica per master) for high availability
- **Port configuration**: each node needs a client port (6379) and a cluster bus port (16379)
- **Network connectivity**: all nodes must be able to communicate via TCP

### Cluster vs Sentinel Decision

| Feature | Cluster | Sentinel |
|---------|---------|----------|
| Sharding | Automatic (16384 slots) | None (single instance) |
| High availability | Automatic master failover | Automatic failover |
| Write scaling | Yes (N masters) | No (single write master) |
| Read scaling | Yes (replicas + replicas) | Yes (read replicas) |
| Max nodes | Up to 1000 (practical: ~50-100) | Typically 3-10 |
| Multi-key ops | Same slot only | Any keys |
| Transactions | Same slot only | Any keys |
| Lua scripts | Same slot keys | Any keys |
| Pub/Sub | Broadcast to all nodes | Single instance |
| Client complexity | Higher (handles redirects) | Lower |

### Cluster Capabilities and Limitations

**Capabilities:**
- Automatic failover with replica promotion
- Scalability to hundreds of nodes
- Online resharding (add/remove nodes without downtime)
- Smart client routing (MOVED/ASK handling)
- Partial read scalability (read from replicas)

**Limitations:**
- Multi-key operations limited to same hash slot
- Transactions limited to same hash slot
- Lua scripts limited to same hash slot keys
- Pub/Sub messages broadcast to all nodes (not sharded by key)
- No multi-database support (only database 0)
- Cross-slot joins or aggregations require client-side logic
- Resharding can be complex to manage

### ASK and MOVED Redirects

When a client requests a key from a node that does not own it:

**MOVED redirect (permanent):**
```bash
GET key
# Output: -MOVED 12345 127.0.0.1:7001
# Client should update its slot map and retry at 127.0.0.1:7001
```

**ASK redirect (temporary — during resharding):**
```bash
GET key
# Output: -ASK 12345 127.0.0.1:7001
# Client should send ASKING then retry the command at 127.0.0.1:7001
# Client does NOT update its slot map permanently
```

### READONLY and READWRITE Commands

```bash
# Connect to replica and switch to read-only mode

# Now reads are allowed from this replica
GET user:123
# Output: ... (value from replica)

# Switch back to read-write (redirect writes to master)
```

### Connecting to a Redis Cluster

```csharp
using StackExchange.Redis;

/// <summary>
/// Demonstrates StackExchange.Redis connection to a Redis Cluster.
/// SE.Redis handles cluster routing, MOVED redirects, and slot mapping automatically.
/// </summary>
public class ClusterConnectionService
{
    private readonly ConnectionMultiplexer _muxer;
    private readonly IDatabase _db;

    /// <summary>
    /// Creates a ConnectionMultiplexer configured for cluster mode.
    /// Provide multiple seed endpoints — SE.Redis discovers the full cluster topology.
    /// </summary>
    public ClusterConnectionService()
    {
        var options = new ConfigurationOptions
        {
            // Seed endpoints — SE.Redis discovers all nodes
            EndPoints =
            {
                { "node1.cluster.example.com", 6379 },
                { "node2.cluster.example.com", 6379 },
                { "node3.cluster.example.com", 6379 }
            },
            // Cluster configuration
            AbortOnConnectFail = false,
            ConnectTimeout = 5000,
            SyncTimeout = 5000,
            Password = "your-password",
            Ssl = true,
            SslProtocols = System.Security.Authentication.SslProtocols.Tls12,
            // Cluster-specific options
            CommandMap = CommandMap.Create(new HashSet<string>
            {
                "INFO", "CONFIG", "CLUSTER", "KEYS"
            }, available: false), // Block dangerous commands on client
            ReconnectRetryPolicy = new ExponentialRetry(1000, 8000),
            KeepAlive = 60,
            // Allow reads from replicas for read scaling
            // PreserveAsyncOrder = true  // default: commands execute in order
        };

        // Add event handlers for cluster events
        options.HashSlotMoved += (sender, args) =>
        {
            Console.WriteLine($"[Cluster] Slot {args.HashSlot} moved from " +
                $"{args.OldEndPoint} to {args.NewEndPoint}");
        };

        _muxer = ConnectionMultiplexer.Connect(options);
        _db = _muxer.GetDatabase();

        Console.WriteLine("[Cluster] Connected. Cluster topology discovered.");
    }

    public IDatabase Database => _db;
    public ConnectionMultiplexer Multiplexer => _muxer;
}
```

### Reading and Writing with Cluster Awareness

```csharp
public class ClusterDataService
{
    private readonly ConnectionMultiplexer _muxer;

    public ClusterDataService(ConnectionMultiplexer muxer)
    {
        _muxer = muxer;
    }

    /// <summary>
    /// Basic read/write in cluster mode.
    /// StackExchange.Redis automatically routes to the correct node.
    /// </summary>
    public async Task SetAndGetAsync(string key, string value)
    {
        var db = _muxer.GetDatabase();

        // Write — SE.Redis computes the slot, finds the owning node, sends the command
        await db.StringSetAsync(key, value);

        // Read — same routing logic
        var result = await db.StringGetAsync(key);

        Console.WriteLine($"Key '{key}' = '{result}'");
    }

    /// <summary>
    /// Demonstrates that keys with hash tags are co-located.
    /// Both keys will hash to the same slot.
    /// </summary>
    public async Task HashTagExampleAsync(string userId)
    {
        var db = _muxer.GetDatabase();
        var hashTag = $"{{user:{userId}}}";

        // All these keys are in the SAME slot
        await db.StringSetAsync($"{hashTag}:name", "Alice");
        await db.StringSetAsync($"{hashTag}:email", "alice@example.com");
        await db.StringSetAsync($"{hashTag}:preferences", "dark_mode=true");

        // Because they share the same hash tag, we can use multi-key ops
        var keys = new RedisKey[]
        {
            $"{hashTag}:name",
            $"{hashTag}:email",
            $"{hashTag}:preferences"
        };
        var values = await db.StringGetAsync(keys);

        foreach (var value in values)
        {
            Console.WriteLine($"  -> {value}");
        }
    }

    /// <summary>
    /// Demonstrates CROSSSLOT error in cluster mode.
    /// This operation will fail because keys hash to different slots.
    /// </summary>
    public async Task CrossSlotExampleAsync()
    {
        var db = _muxer.GetDatabase();

        try
        {
            // These keys have different hash tags — different slots
            var keys = new RedisKey[] { "{user:1}:name", "{user:2}:name" };
            var values = await db.StringGetAsync(keys);
        }
        catch (RedisServerException ex) when (ex.Message.Contains("CROSSSLOT"))
        {
            Console.WriteLine("Cross-slot operation blocked: " + ex.Message);
        }
    }
}
```

### Using Hash Tags for Multi-Key Operations

```csharp
public class HashTagTransactionService
{
    private readonly ConnectionMultiplexer _muxer;

    public HashTagTransactionService(ConnectionMultiplexer muxer)
    {
        _muxer = muxer;
    }

    /// <summary>
    /// Executes a transaction with keys sharing the same hash tag.
    /// This works in cluster mode because all keys are on the same slot.
    /// </summary>
    public async Task<bool> AtomicUserUpdateAsync(string userId, string name, string email)
    {
        var db = _muxer.GetDatabase();
        var hashTag = $"{{user:{userId}}}";

        var tran = db.CreateTransaction();

        // All keys share the same hash tag — same slot
        tran.StringSetAsync($"{hashTag}:name", name);
        tran.StringSetAsync($"{hashTag}:email", email);
        tran.HashSetAsync($"{hashTag}:meta", new HashEntry[]
        {
            new("updated_at", DateTimeOffset.UtcNow.ToUnixTimeSeconds().ToString())
        });

        return await tran.ExecuteAsync();
    }

    /// <summary>
    /// Lua script with hash-tagged keys works in cluster mode.
    /// </summary>
    public async Task<RedisResult> AtomicLuaScriptAsync(string userId, int incrementBy)
    {
        var db = _muxer.GetDatabase();
        var hashTag = $"{{user:{userId}}}";

        var script = @"
            local score = redis.call('GET', KEYS[1])
            local counter = redis.call('INCRBY', KEYS[2], ARGV[1])
            return {score, counter}
        ";

        return await db.ScriptEvaluateAsync(script,
            new RedisKey[] { $"{hashTag}:score", $"{hashTag}:counter" },
            new RedisValue[] { incrementBy.ToString() });
    }
}
```

### Reading from Replicas in Cluster Mode

```csharp
public class ClusterReadScalingService
{
    private readonly ConnectionMultiplexer _muxer;

    public ClusterReadScalingService(ConnectionMultiplexer muxer)
    {
        _muxer = muxer;
    }

    /// <summary>
    /// Reads from replica nodes for read-heavy workloads.
    /// StackExchange.Redis supports reading from replicas when configured.
    /// </summary>
    public async Task<string> ReadFromReplicaAsync(string key)
    {
        // Get a database handle that prefers replicas for read operations
        var db = _muxer.GetDatabase();

        // Configure the multiplexer to allow reads from replicas
        // This must be done at connection time:
        //   config.AllowAdmin = true;
        // Then use: db.StringGetAsync(key, CommandFlags.PreferReplica);
        
        var value = await db.StringGetAsync(key, CommandFlags.PreferReplica);

        if (value.HasValue)
        {
            Console.WriteLine($"[Read from replica] {key} = {value}");
            return value.ToString();
        }

        // Fall back to primary read
        Console.WriteLine("[Read from replica] Miss, trying primary...");
        value = await db.StringGetAsync(key, CommandFlags.PreferMaster);
        return value.ToString();
    }

    /// <summary>
    /// Batch read with replica preference.
    /// </summary>
    public async Task<Dictionary<string, string>> BatchReadFromReplicasAsync(params string[] keys)
    {
        var db = _muxer.GetDatabase();
        var redisKeys = keys.Select(k => (RedisKey)k).ToArray();
        var values = await db.StringGetAsync(redisKeys, CommandFlags.PreferReplica);

        var result = new Dictionary<string, string>();
        for (int i = 0; i < keys.Length; i++)
        {
            result[keys[i]] = values[i].HasValue ? values[i].ToString() : null;
        }

        return result;
```

### Cluster Failover Handling

```csharp
public class ClusterFailoverService
{
    private readonly ConnectionMultiplexer _muxer;
    private readonly ILogger _logger;

    public ClusterFailoverService(ConnectionMultiplexer muxer, ILogger logger)
    {
        _muxer = muxer;
        _logger = logger;
    }

    /// <summary>
    /// Demonstrates how to handle cluster failover events.
    /// StackExchange.Redis raises events when cluster topology changes.
    /// </summary>
    public void RegisterFailoverHandlers()
    {
        _muxer.HashSlotMoved += (sender, args) =>
        {
            _logger.LogInformation(
                "Hash slot {Slot} moved from {OldEndPoint} to {NewEndPoint}",
                args.HashSlot, args.OldEndPoint, args.NewEndPoint);
        };

        _muxer.ConnectionFailed += (sender, args) =>
        {
            _logger.LogWarning(
                "Connection failed to {EndPoint}: {Exception}",
                args.EndPoint, args.Exception?.Message);
        };

        _muxer.ConnectionRestored += (sender, args) =>
        {
            _logger.LogInformation(
                "Connection restored to {EndPoint}",
                args.EndPoint);
        };

        _muxer.ErrorMessage += (sender, args) =>
        {
            _logger.LogError(
                "Redis error: {Message} from {EndPoint}",
                args.Message, args.EndPoint);
        };
    }

    /// <summary>
    /// Retry logic for operations during cluster failover.
    /// If a MOVED redirect occurs, SE.Redis retries automatically.
    /// If the cluster is unavailable, we retry with exponential backoff.
    /// </summary>
    public async Task<T> ExecuteWithFailoverRetryAsync<T>(
        Func<IDatabase, Task<T>> operation, int maxRetries = 5)
    {
        for (int attempt = 0; attempt < maxRetries; attempt++)
        {
            try
            {
                var db = _muxer.GetDatabase();
                return await operation(db);
            }
            catch (RedisConnectionException ex) when (attempt < maxRetries - 1)
            {
                _logger.LogWarning(ex,
                    "Cluster failover detected, retrying ({Attempt}/{MaxRetries})",
                    attempt + 1, maxRetries);
                await Task.Delay(TimeSpan.FromMilliseconds(
                    200 * Math.Pow(2, attempt)));
            }
            catch (RedisServerException ex) when (
                ex.Message.Contains("CLUSTERDOWN") && attempt < maxRetries - 1)
            {
                _logger.LogWarning(ex,
                    "Cluster is down, retrying ({Attempt}/{MaxRetries})",
                    attempt + 1, maxRetries);
                await Task.Delay(TimeSpan.FromMilliseconds(
                    500 * Math.Pow(2, attempt)));
            }
        }

        throw new InvalidOperationException(
            "Operation failed after cluster failover retries.");
    }
}
```

## Section 6 — Performance Considerations

### Cluster Routing Overhead

StackExchange.Redis maintains a local slot-to-node mapping that it uses to route commands directly to the correct node. This avoids the MOVED redirect round-trip for most commands. The overhead of slot mapping lookup is negligible (hash map lookup in memory).

When a MOVED redirect does occur (due to cluster topology change), SE.Redis:
1. Receives the MOVED error
2. Updates its local slot mapping
3. Retries the command on the correct node
4. Returns the result to the caller

The retry adds one additional round-trip. During resharding or failover, MOVED redirects are more common but still typically a small percentage of total commands.

### Network Overhead

Cluster mode adds network overhead compared to a single instance:
- **Gossip traffic**: nodes exchange cluster state information (small but constant)
- **Client redirection**: MOVED/ASK redirects add latency
- **Cross-node operations**: Pub/Sub messages are broadcast to all nodes
- **Pipeline complexity**: pipelining to multiple nodes requires multiple connections

In practice, the network overhead is negligible for most workloads. The benefits of horizontal scaling far outweigh the overhead.

### Slot Distribution Balance

Uneven slot distribution causes some nodes to be overloaded. The `redis-cli --cluster rebalance` command redistributes slots evenly. StackExchange.Redis does not perform rebalancing — it's a cluster management operation.

Monitor slot distribution with:
```bash
redis-cli --cluster info 127.0.0.1:7000
```

### Read Scaling with Replicas

In cluster mode, each master can have one or more replicas. Read scaling can be achieved by reading from replicas:

```csharp
// Configure SE.Redis to allow replica reads
var options = new ConfigurationOptions
{
    // ... other options
};
// Replica reads are configured per-operation with CommandFlags
var db = _muxer.GetDatabase();
var value = await db.StringGetAsync("key", CommandFlags.PreferReplica);
```

Read scaling trade-offs:
- **Stale data**: replicas may lag behind the master (typically < 1ms in normal operation)
- **Consistency**: reads from replicas are eventually consistent
- **Connection count**: more connections = more memory on client and server
- **Complexity**: must decide per-operation whether to use replica reads

### Write Throughput Scaling

Write throughput scales linearly with the number of master nodes, assuming writes are evenly distributed across slots. For example:

| Master Nodes | Total Write Throughput (approx) |
|-------------|-------------------------------|
| 3 | 150,000 ops/sec |
| 6 | 300,000 ops/sec |
| 12 | 600,000 ops/sec |
| 24 | 1,200,000 ops/sec |

Actual throughput depends on hardware, network, and data size. The key insight is that cluster sharding provides near-linear write scaling.

### Batch Operations Across Nodes

Batch operations (MGET, MSET) can span multiple nodes in cluster mode. StackExchange.Redis handles this by:
1. Grouping keys by their hash slot/node
2. Sending separate MGET/MSET commands to each node
3. Combining the results

This means a 100-key MGET across 3 masters becomes 3 MGET commands (one per node). The combined latency is determined by the slowest node.

### Multi-Key Operation Costs

Multi-key operations (MGET, MSET, DEL, EXISTS) are more expensive in cluster mode because:
- Keys may be on different nodes, requiring multiple network round-trips
- Results must be combined on the client
- Cross-slot operations (MGET with keys from different slots) still work but are less efficient

For maximum performance, use hash tags to co-locate related keys and minimize cross-node operations.

### Connection Management

Each node in the cluster maintains a separate TCP connection from the client. For a 6-node cluster with one master and one replica per master, StackExchange.Redis maintains:
- 3 master connections (writes)
- 3 replica connections (optional reads)
- 1 cluster bus connection (internal)

Total: 6-7 connections per ConnectionMultiplexer. Each connection consumes memory on the Redis server (client output buffer).

## Section 7 — Production Considerations

### Minimum Node Requirements

A production Redis Cluster requires:
- **At least 6 nodes** (3 masters + 3 replicas) for high availability
- **At least 3 masters** for quorum-based failure detection
- **Each on separate hardware** (or at least separate failure domains)

The minimum cluster that can survive a single node failure is 3 masters + 3 replicas. With 2 masters, you lose quorum if one fails.

### Network Requirements

Cluster nodes must communicate over:
- **Client port** (default 6379): for client commands
- **Cluster bus port** (default 16379): for gossip protocol (client port + 10000)

The cluster bus requires:
- Low latency (ideally < 1ms between nodes)
- Reliable TCP connectivity
- All nodes on the same network (or low-latency WAN)

Do not place cluster nodes across high-latency WAN links — the gossip protocol will fail.

### Cluster Node Failure Detection

Redis Cluster uses a gossip-based failure detection:
1. **PFAIL** (possible failure): one node suspects another is down (node-timeout without response)
2. **FAIL** (confirmed failure): majority of masters agree that PFAIL is real

The `cluster-node-timeout` configuration (default 5000ms) controls:
- How long before a node is marked PFAIL
- How quickly failover occurs
- How long the cluster is unavailable during partition

Tune this based on your network reliability requirements:
- `cluster-node-timeout 5000` (5 seconds) — good for most environments
- `cluster-node-timeout 15000` (15 seconds) — tolerant of network hiccups
- `cluster-node-timeout 1000` (1 second) — aggressive failover

### Graceful Shutdown and Restart

```bash
# Gracefully shut down a node — triggers replica promotion
redis-cli -p 7000 SHUTDOWN

# Restart the node — it will rejoin as a replica
redis-server /path/to/node-7000.conf

# Check that it rejoined correctly
redis-cli -p 7000 CLUSTER NODES
```

### Online Resharding

```bash
# Move 100 slots to a new node
redis-cli --cluster reshard 127.0.0.1:7000 \
    --cluster-from <source-node-id> \
    --cluster-to <target-node-id> \
    --cluster-slots 100 \
    --cluster-yes
```

During resharding:
- Source node marks slots as MIGRATING
- Target node marks slots as IMPORTING
- Clients receive ASK redirects for migrating slots
- StackExchange.Redis handles ASK redirects transparently
- Performance may degrade during resharding

### Backup and Restore in Cluster

Backing up a cluster is more complex than a single instance:

**Approach 1: Backup each node individually**
```bash
# On each node, trigger BGSAVE
redis-cli -p 7000 BGSAVE
redis-cli -p 7001 BGSAVE
redis-cli -p 7002 BGSAVE

# Copy the RDB files from each node
cp /var/lib/redis/dump.rdb /backup/node-7000.rdb
```

**Approach 2: Use redis-cli --cluster backup**
```bash
# Create a backup of the entire cluster
redis-cli --cluster backup 127.0.0.1:7000 /backup/cluster
```

Restore requires rebuilding the cluster and reloading data into each node.

## Section 8 — Gotchas & Pitfalls

### CROSSSLOT Error on Multi-Key Operations

The most common cluster pitfall: operations spanning multiple keys in different hash slots fail with CROSSSLOT error.

```bash
# BAD — keys in different slots
MGET user:1 user:2
# Output: (error) CROSSSLOT Keys in request don't hash to the same slot

# GOOD — use hash tags to co-locate
MGET {user:1}:name {user:1}:email
# Both keys share the {user:1} hash tag → same slot → works
```

StackExchange.Redis handles MGET/MSET across slots by splitting into separate commands per node. However, transactions and Lua scripts still require same-slot keys.

### Transactions and Lua Scripts Are Slot-Bound

In cluster mode, MULTI/EXEC transactions and Lua scripts can only operate on keys within the same hash slot.

```csharp
// BAD — transaction with keys from different slots
var tran = db.CreateTransaction();
tran.StringSetAsync("{user:1}:name", "Alice");   // slot A
tran.StringSetAsync("{user:2}:name", "Bob");      // slot B — CROSSSLOT!
await tran.ExecuteAsync();

// GOOD — same slot via hash tag
var tran = db.CreateTransaction();
tran.StringSetAsync("{user:1}:name", "Alice");    // same slot
tran.StringSetAsync("{user:1}:email", "@");       // same slot
await tran.ExecuteAsync();
```

### Only Database 0 Is Available

Redis Cluster only supports database 0. The SELECT command is not available. Attempting to select a different database returns an error.

```bash
SELECT 1
# Output: (error) ERR SELECT is not allowed in cluster mode
```

StackExchange.Redis respects this — GetDatabase(db) with db != 0 in cluster mode may fail or redirect to database 0.

### Pub/Sub Is Not Sharded

In cluster mode, Pub/Sub messages are broadcast to ALL nodes in the cluster, not sharded by channel name. This means:
- Every node receives every publish message
- Nodes forward messages to all connected subscribers
- Network traffic increases linearly with cluster size
- Pub/Sub does NOT scale horizontally in cluster mode

For scalable messaging, use Redis Streams instead of Pub/Sub in cluster deployments.

### Cluster Bus Uses Separate Port

Each Redis Cluster node opens two TCP ports:
1. **Client port** (default 6379): for client commands
2. **Cluster bus port** (default 16379 = 6379 + 10000): for node-to-node communication

Firewalls must allow both ports. The cluster bus uses a custom binary protocol (not Redis protocol).

### MOVED/ASK Redirects During Resharding

During online resharding, clients may receive ASK and MOVED redirects:
- **ASK**: temporary — the slot is being migrated; send ASKING command before retrying
- **MOVED**: permanent — the slot has moved; update your slot map

StackExchange.Redis handles both transparently, but the extra round-trips add latency. During large resharding operations, monitor for increased latency.

### Client-Side Slot Map May Be Stale

StackExchange.Redis maintains a local slot-to-node mapping. If the cluster topology changes (failover, resharding), the client's mapping may be stale. SE.Redis updates its mapping when it receives a MOVED redirect, but there is a brief window where the client routes to the wrong node.

```csharp
// SE.Redis refreshes slot mapping periodically
// ConfigCheckSeconds (default 60s) controls refresh interval
// MOVED redirects also trigger local mapping updates
```

### Cluster Mode and Connection Multiplexing

Each ConnectionMultiplexer maintains connections to all nodes in the cluster. If the cluster has 10 nodes, SE.Redis maintains 10 TCP connections (plus replica connections). This is normal but can use significant file descriptors on the client machine for large clusters.

```csharp
// Each multiplexer connects to all cluster nodes
var muxer1 = ConnectionMultiplexer.Connect(options); // ~N connections
var muxer2 = ConnectionMultiplexer.Connect(options); // ~N more connections

// For large clusters, minimize the number of multiplexers
```

### Replica Lag in Cluster Mode

Replicas in cluster mode may lag behind their master, especially during high write loads. If your application requires strong consistency:

- Always read from the master (CommandFlags.PreferMaster)
- Accept that replica reads may return stale data
- Monitor replica lag with `INFO REPLICATION`:
  ```bash
  redis-cli -p 7003 INFO REPLICATION
  # Look for: master_repl_offset, slave_repl_offset
  ```

### Node Addition and Removal Requires Care

Adding or removing nodes in a cluster requires explicit management:
- **Adding**: start node, CLUSTER MEET, reshard some slots to it
- **Removing**: reshard all slots away from the node, CLUSTER FORGET
- **Data loss**: if you remove a node without resharding, data in its slots is lost

Always reshard slots before removing nodes. Use the `redis-cli --cluster rebalance` and `redis-cli --cluster reshard` commands.

### Cluster Configuration File

Each node stores its cluster state in a configuration file (default: `nodes-<port>.conf`). This file is automatically updated by Redis. Deleting or corrupting this file causes the node to lose its cluster membership.

```bash
# Do NOT edit this file manually while Redis is running
# If a node cannot join the cluster, check:
# - nodes.conf is intact
# - node-id matches what other nodes expect
# - cluster-config-file path is writable
```

### Network Partitions and Split-Brain

Redis Cluster is designed to handle network partitions:
- **Majority side**: continues operating normally
- **Minority side**: stops accepting writes to prevent split-brain
- **After partition heals**: minority nodes rejoin as replicas

If the minority side has replicas that were promoted during the partition, the old masters will be killed when they rejoin (rollback). This can cause data loss for writes that occurred on the promoted replica.

The `cluster-node-timeout` setting controls how quickly the cluster detects partitions and reacts. Too short = unnecessary failovers. Too long = extended unavailability.

### Client Bandwidth to Multiple Nodes

Each connection to a cluster node consumes bandwidth on the client. With many nodes and high throughput, the client's network interface can become a bottleneck. Monitor:
- Client network utilization
- Per-connection buffer sizes
- Total file descriptor count

### Monitoring Tools Compatibility

Some Redis monitoring tools have limited cluster support:
- `redis-cli --stat` shows only the connected node
- `redis-cli --bigkeys` scans only the connected node
- MONITOR only shows commands on the connected node

Use cluster-aware tools or run monitoring commands on each node.
