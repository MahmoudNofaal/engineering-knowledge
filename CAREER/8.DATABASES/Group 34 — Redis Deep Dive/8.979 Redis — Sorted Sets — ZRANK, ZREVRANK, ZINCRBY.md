# Redis — Sorted Sets — ZRANK, ZREVRANK, ZINCRBY

---

## Section 1 — Overview

Sorted sets in Redis provide commands to query and modify the position and score of individual members. ZRANK returns the 0-based position of a member in ascending score order (lowest score = rank 0). ZREVRANK returns the position in descending order (highest score = rank 0). ZINCRBY atomically increments (or decrements) the score of a member, returning the new score.

These three commands form the backbone of interactive leaderboard and scoring systems. ZRANK and ZREVRANK answer the question "where do I stand?" — essential for showing players their current rank. ZINCRBY allows real-time score updates without the need for a read-modify-write cycle (which would require WATCH/MULTI/EXEC transactions).

Together with ZADD, ZREM, ZRANGE, and ZREVRANGE, these commands cover the full sorted set workflow: insert, update, score increment, rank query, and range retrieval.

---

### 1.1 — Core Concept

```
Sorted Set — rank and score relationship:

Scores Ascending:
  Rank 0: playerA (score 500)   ← lowest score = best rank (ascending)
  Rank 1: playerB (score 1000)
  Rank 2: playerC (score 2000)
  Rank 3: playerD (score 3000)  ← highest score = worst rank (ascending)

Scores Descending (most common for leaderboards):
  Rank 0: playerD (score 3000)  ← highest score = rank 0
  Rank 1: playerC (score 2000)
  Rank 2: playerB (score 1000)
  Rank 3: playerA (score 500)   ← lowest score = worst rank
```

---

### 1.2 — When to Use

| Command | Use Case |
|---|---|
| ZRANK | Show "your rank" when sorted ascending (e.g., low score wins — golf) |
| ZREVRANK | Show "your rank" when sorted descending (e.g., high score wins — leaderboard) |
| ZINCRBY | Live score updates, point awards, penalty deductions |

## Section 2 — Command Reference

---

### 2.1 — ZRANK

Returns the 0-based rank of a member in a sorted set, ordered by score ascending (low to high).

**Syntax:**
```
ZRANK key member [WITHSCORE]
```

**Parameters:**
- `key` — the sorted set key
- `member` — the member whose rank to look up
- `WITHSCORE` — (Redis 7.2+) also return the member's score

**Complexity:** O(log N)

**Return:** Integer rank (0-based), or nil if member does not exist.

**Redis CLI Example:**
```
> ZADD leaderboard 100 player1 200 player2 50 player3 300 player4
(integer) 4

> ZRANK leaderboard "player1"
(integer) 1         # player1 has rank 1 (second lowest score)

> ZRANK leaderboard "player3"
(integer) 0         # player3 has rank 0 (lowest score)

> ZRANK leaderboard "player4"
(integer) 3         # player4 has rank 3 (highest score)

> ZRANK leaderboard "nonexistent"
(nil)               # member does not exist
```

---

### 2.2 — ZREVRANK

Returns the 0-based rank of a member in a sorted set, ordered by score descending (high to low).

**Syntax:**
```
ZREVRANK key member [WITHSCORE]
```

**Parameters:**
- `key` — the sorted set key
- `member` — the member whose rank to look up
- `WITHSCORE` — (Redis 7.2+) also return the member's score

**Complexity:** O(log N)

**Return:** Integer rank (0-based), or nil if member does not exist.

**Redis CLI Example:**
```
> ZREVRANK leaderboard "player1"
(integer) 2         # player1 has rank 2 (third highest score)

> ZREVRANK leaderboard "player4"
(integer) 0         # player4 has rank 0 (highest score)

> ZREVRANK leaderboard "player3"
(integer) 3         # player3 has rank 3 (lowest score, worst in descending)
```

---

### 2.3 — ZINCRBY

Atomically increments the score of a member by a given delta. If the member does not exist, it is created with score = delta.

**Syntax:**
```
ZINCRBY key increment member
```

**Parameters:**
- `key` — the sorted set key
- `increment` — the amount to add (can be negative, integer or double)
- `member` — the member whose score to increment

**Complexity:** O(log N)

**Return:** The new score as a bulk string (double representation).

**Redis CLI Example:**
```
> ZADD leaderboard 100 player1
(integer) 1

> ZINCRBY leaderboard 50 "player1"
"150"               # player1 score is now 150

> ZINCRBY leaderboard -20 "player1"
"130"               # player1 score is now 130 (decremented)

> ZINCRBY leaderboard 200 "newplayer"
"200"               # member created with score 200
```

---

### 2.4 — ZSCORE (Related)

While not the primary focus, ZSCORE is closely related to ZRANK/ZREVRANK — it returns the score of a member without its rank.

**Syntax:**
```
ZSCORE key member
```

**Redis CLI Example:**
```
> ZSCORE leaderboard "player1"
"130"
```

---

### 2.5 — ZCARD (Related)

ZCOUNT can be used in combination with ranks to compute a "position from bottom" or "position from top" percentage.

**Syntax:**
```
ZCARD key
```

**Redis CLI Example:**
```
> ZCARD leaderboard
(integer) 4

# Top percentile: rank / total * 100
# For player1 with ZREVRANK = 2 and ZCARD = 4:
# Percentile = 2 / 4 * 100 = top 50%
```

---

### 2.6 — ZCOUNT (Related)

Counts members with scores in a given range. Useful for understanding how many players are ahead or behind.

**Syntax:**
```
ZCOUNT key min max
```

**Redis CLI Example:**
```
> ZCOUNT leaderboard 100 200
(integer) 2          # 2 players have scores between 100 and 200
```

---

### 2.7 — WITHSCORE Option (Redis 7.2+)

Starting from Redis 7.2, ZRANK and ZREVRANK support the `WITHSCORE` option to return both rank and score in a single round trip:

```bash
> ZRANK leaderboard "player1" WITHSCORE
1) (integer) 1
2) "130"

> ZREVRANK leaderboard "player1" WITHSCORE
1) (integer) 2
2) "130"
```

In older Redis versions or when the client library does not support this, you need separate ZRANK and ZSCORE calls.

|---|---|---|

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;

public class RankAndScoreService

    {

    }

    {
        try
        {
            // ZREVRANK — 0 = highest score, null = not found
            return await _db.SortedSetRankAsync(
                leaderboardKey,

                Order.Descending);
        }
        catch (RedisConnectionException ex)
        {
            Console.Error.WriteLine($"Connection failed: {ex.Message}");
            return null;
        }
        catch (RedisTimeoutException ex)
        {
            Console.Error.WriteLine($"Timeout: {ex.Message}");
            return null;
        }
        catch (RedisServerException ex) when (ex.Message.Contains("WRONGTYPE"))
        {
            Console.Error.WriteLine($"Key is not a sorted set: {ex.Message}");
            return null;
        }
    }

    public async Task<double> IncrementScoreAsync(
        string leaderboardKey, string playerId, double points)
    {
        try
        {
            return await _db.SortedSetIncrementAsync(
                leaderboardKey, playerId, points);
        }
        catch (RedisException ex)
        {
            Console.Error.WriteLine($"Failed to increment score: {ex.Message}");
            throw; // caller should handle and potentially retry
        }
    }
}
```

### 4.1 — Player Rank Service with 1-Based Conversion

Most users expect rank to start at 1 (not 0). This service converts from Redis's 0-based to 1-based rank and handles the "player not found" case.

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;

public class PlayerRankService
{
    private readonly IDatabase _db;
    private const string DefaultLeaderboard = "leaderboard:global";

    public PlayerRankService(IDatabase db)
    {
        _db = db;
    }

    /// <summary>
    /// Get a player's rank (1-based, 1 = top/highest score).
    /// Returns null if player not in leaderboard.
    /// </summary>
    public async Task<int?> GetPlayerRankAsync(
        string playerId,
        string leaderboardKey = DefaultLeaderboard)
    {
        // ZREVRANK returns 0-based rank (null if not found)
        long? zeroBasedRank = await _db.SortedSetRankAsync(
            leaderboardKey,
            playerId,
            Order.Descending);

        if (zeroBasedRank == null)
        {
            return null; // player not in leaderboard
        }

        // Convert to 1-based for display
        return (int)(zeroBasedRank.Value + 1);
    }

    /// <summary>
    /// Get both rank and score for a player in one method.
    /// </summary>
    public async Task<PlayerRankInfo?> GetPlayerRankWithScoreAsync(
        string playerId,
        string leaderboardKey = DefaultLeaderboard)
    {
        // Get rank and score in parallel (2 round trips)
        Task<long?> rankTask = _db.SortedSetRankAsync(
            leaderboardKey, playerId, Order.Descending);
        Task<double?> scoreTask = _db.SortedSetScoreAsync(
            leaderboardKey, playerId);

        await Task.WhenAll(rankTask, scoreTask);

        long? rank = rankTask.Result;
        double? score = scoreTask.Result;

        if (rank == null || score == null)
        {
            return null; // player not found
        }

        // Also get total player count for positioning
        long totalPlayers = await _db.SortedSetLengthAsync(leaderboardKey);

        return new PlayerRankInfo
        {
            PlayerId = playerId,
            Rank = (int)(rank.Value + 1),   // 1-based
            Score = (long)score.Value,
            TotalPlayers = totalPlayers,
            TopPercentile = totalPlayers > 0
                ? Math.Round((double)(rank.Value + 1) / totalPlayers * 100, 1)
                : 0
        };
    }
}

public class PlayerRankInfo
{
    public string PlayerId { get; set; } = string.Empty;
    public int Rank { get; set; }
    public long Score { get; set; }
    public long TotalPlayers { get; set; }
    public double TopPercentile { get; set; }
}
```

### 4.2 — Live Score Increment with ZINCRBY

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;

public class ScoreIncrementService
{
    private readonly IDatabase _db;

    public ScoreIncrementService(IDatabase db)
    {
        _db = db;
    }

    /// <summary>
    /// Award points to a player atomically.
    /// ZINCRBY creates the member if it doesn't exist.
    /// Returns the new total score.
    /// </summary>
    public async Task<double> AwardPointsAsync(
        string leaderboardKey,
        string playerId,
        double points)
    {
        double newScore = await _db.SortedSetIncrementAsync(
            leaderboardKey, playerId, points);

        return newScore;
    }

    /// <summary>
    /// Deduct points from a player (penalty).
    /// Uses ZINCRBY with negative value.
    /// </summary>
    public async Task<double> DeductPointsAsync(
        string leaderboardKey,
        string playerId,
        double points)
    {
        double newScore = await _db.SortedSetDecrementAsync(
            leaderboardKey, playerId, points);

        return newScore;
    }

    /// <summary>
    /// Batch award points to multiple players using a pipeline.
    /// </summary>
    public async Task<double[]> BatchAwardPointsAsync(
        string leaderboardKey,
        (string playerId, double points)[] awards)
    {
        IBatch batch = _db.CreateBatch();
        var tasks = new Task<double>[awards.Length];

        for (int i = 0; i < awards.Length; i++)
        {
            tasks[i] = batch.SortedSetIncrementAsync(
                leaderboardKey,
                awards[i].playerId,
                awards[i].points);
        }

        batch.Execute();

        double[] results = new double[awards.Length];
        for (int i = 0; i < awards.Length; i++)
        {
            results[i] = await tasks[i];
        }

        return results;
    }
}
```

### 4.3 — Rank Change Detection

Track how a player's rank changed since their last check by comparing current rank against a stored previous rank.

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;

public class RankChangeTracker
{
    private readonly IDatabase _db;

    public RankChangeTracker(IDatabase db)
    {
        _db = db;
    }

    /// <summary>
    /// Get current rank and compare with stored previous rank.
    /// Returns rank change (positive = improved, negative = dropped, 0 = same).
    /// </summary>
    public async Task<RankChangeResult> GetRankChangeAsync(
        string leaderboardKey,
        string playerId)
    {
        string previousRankKey = $"rank:prev:{leaderboardKey}:{playerId}";

        // Get current rank
        long? currentRank = await _db.SortedSetRankAsync(
            leaderboardKey, playerId, Order.Descending);

        if (currentRank == null)
        {
            // Player not in leaderboard
            return new RankChangeResult { Change = null, CurrentRank = null };
        }

        // Get previous rank from separate key
        RedisValue previousRankValue = await _db.StringGetAsync(previousRankKey);

        int? previousRank = previousRankValue.IsNull
            ? null
            : (int)(int.Parse(previousRankValue!) + 1); // stored 0-based, convert to 1-based

        int rank = (int)(currentRank.Value + 1); // 1-based for display

        // Store current rank as previous for next check
        await _db.StringSetAsync(previousRankKey, currentRank.Value.ToString());

        int? change = null;
        if (previousRank.HasValue)
        {
            change = previousRank.Value - rank; // positive = improved
        }

        return new RankChangeResult
        {
            CurrentRank = rank,
            PreviousRank = previousRank,
            Change = change
        };
    }
}

public class RankChangeResult
{
    public int? CurrentRank { get; set; }
    public int? PreviousRank { get; set; }
    public int? Change { get; set; }

    public string ChangeIndicator =>
        Change switch
        {
            > 0 => $"↑ {Change}",
            < 0 => $"↓ {Math.Abs(Change.Value)}",
            0 => "—",
            null => "NEW"
        };
}
```

### 4.4 — Periodic Top-K with ZREVRANK for "Rank Me" Button

Complete leaderboard page: show top 10 and the requesting player's rank.

```csharp
using StackExchange.Redis;
using System;
using System.Collections.Generic;
using System.Threading.Tasks;

public class LeaderboardWithPlayerRankService
{
    private readonly IDatabase _db;

    public LeaderboardWithPlayerRankService(IDatabase db)
    {
        _db = db;
    }

    public async Task<LeaderboardWithPlayerResponse> GetLeaderboardWithPlayerRankAsync(
        string leaderboardKey,
        string requestingPlayerId,
        int topCount = 10)
    {
        // Parallel queries: top N and requesting player's rank + score
        Task<SortedSetEntry[]> topTask = _db.SortedSetRangeByRankWithScoresAsync(
            leaderboardKey,
            start: 0,
            stop: topCount - 1,
            order: Order.Descending);

        Task<long?> rankTask = _db.SortedSetRankAsync(
            leaderboardKey,
            requestingPlayerId,
            Order.Descending);

        Task<double?> scoreTask = _db.SortedSetScoreAsync(
            leaderboardKey,
            requestingPlayerId);

        Task<long> totalTask = _db.SortedSetLengthAsync(leaderboardKey);

        await Task.WhenAll(topTask, rankTask, scoreTask, totalTask);

        SortedSetEntry[] topPlayers = topTask.Result;
        long? playerRank = rankTask.Result;
        double? playerScore = scoreTask.Result;
        long totalPlayers = totalTask.Result;

        // Build top players list
        var top = new List<LeaderboardEntry>();
        for (int i = 0; i < topPlayers.Length; i++)
        {
            top.Add(new LeaderboardEntry
            {
                Rank = i + 1,
                PlayerId = topPlayers[i].Element.ToString(),
                Score = (long)topPlayers[i].Score,
                IsRequestingPlayer = topPlayers[i].Element == requestingPlayerId
            });
        }

        // Build player's own position
        PlayerPositionInfo? playerPosition = null;
        if (playerRank.HasValue && playerScore.HasValue)
        {
            playerPosition = new PlayerPositionInfo
            {
                Rank = (int)(playerRank.Value + 1),
                Score = (long)playerScore.Value,
                TotalPlayers = totalPlayers
            };
        }

        return new LeaderboardWithPlayerResponse
        {
            TopPlayers = top,
            PlayerPosition = playerPosition,
            TotalPlayers = totalPlayers
        };
    }
}

public class LeaderboardEntry
{
    public int Rank { get; set; }
    public string PlayerId { get; set; } = string.Empty;
    public long Score { get; set; }
    public bool IsRequestingPlayer { get; set; }
}

public class PlayerPositionInfo
{
    public int Rank { get; set; }
    public long Score { get; set; }
    public long TotalPlayers { get; set; }
    public double Percentile => TotalPlayers > 0
        ? Math.Round((double)Rank / TotalPlayers * 100, 1)
        : 0;
}

public class LeaderboardWithPlayerResponse
{
    public List<LeaderboardEntry> TopPlayers { get; set; } = new();
    public PlayerPositionInfo? PlayerPosition { get; set; }
    public long TotalPlayers { get; set; }
}
```

### 4.5 — ZINCRBY Atomicity Demo

Demonstrate that ZINCRBY is atomic — no race conditions in concurrent scenarios.

```csharp
using StackExchange.Redis;
using System;
using System.Diagnostics;
using System.Threading.Tasks;

public class AtomicIncrementDemo
{
    private readonly IDatabase _db;

    public AtomicIncrementDemo(IDatabase db)
    {
        _db = db;
    }

    /// <summary>
    /// Demonstrate atomic increment under concurrency.
    /// Without ZINCRBY, you'd need WATCH + MULTI + EXEC or Lua.
    /// With ZINCRBY, it's a single atomic command.
    /// </summary>
    public async Task RunConcurrencyDemoAsync(string key)
    {
        // Reset
        await _db.KeyDeleteAsync(key);
        await _db.SortedSetAddAsync(key, "counter", 0);

        // Simulate 10 concurrent increments
        int concurrentTasks = 10;
        var tasks = new Task[concurrentTasks];

        for (int i = 0; i < concurrentTasks; i++)
        {
            tasks[i] = Task.Run(async () =>
            {
                for (int j = 0; j < 100; j++)
                {
                    // This is atomic — no race condition
                    await _db.SortedSetIncrementAsync(key, "counter", 1);
                }
            });
        }

        await Task.WhenAll(tasks);

        double finalScore = (await _db.SortedSetScoreAsync(key, "counter")) ?? 0;
        Debug.Assert(finalScore == 1000, $"Expected 1000, got {finalScore}");
        Console.WriteLine($"Final score: {finalScore} (expected: 1000)");
    }
}
```

### 4.6 — Composite Score with ZINCRBY

When using ZINCRBY with a composite score (score + tiebreaker), you cannot increment the composite directly. Instead, read the current score, decompose, increment, and write back using Lua or WATCH.

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;

public class CompositeScoreIncrementService
{
    private readonly IDatabase _db;
    private const double TiebreakerMax = 999_999.0;

    public CompositeScoreIncrementService(IDatabase db)
    {
        _db = db;
    }

    /// <summary>
    /// Encode score: score * (TiebreakerMax + 1) + tiebreaker
    /// </summary>
    public double EncodeScore(int score, double tiebreaker)
    {
        return score * (TiebreakerMax + 1) + tiebreaker;
    }

    /// <summary>
    /// Decode into (score, tiebreaker)
    /// </summary>
    public (int score, double tiebreaker) DecodeScore(double encoded)
    {
        int score = (int)(encoded / (TiebreakerMax + 1));
        double tiebreaker = encoded % (TiebreakerMax + 1);
        return (score, tiebreaker);
    }

    /// <summary>
    /// Increment a composite score atomically using Lua.
    /// The Lua script handles read-decompose-increment-recompose in one atomic step.
    /// </summary>
    public async Task<double> IncrementCompositeScoreAsync(
        string key,
        string member,
        int incrementBy,
        double newTiebreaker)
    {
        string script = @"
            local key = KEYS[1]
            local member = ARGV[1]
            local increment = tonumber(ARGV[2])
            local newTiebreaker = tonumber(ARGV[3])
            local tiebreakerMax = tonumber(ARGV[4])

            local current = redis.call('ZSCORE', key, member)
            local newScore

            if current then
                local scorePart = math.floor(current / (tiebreakerMax + 1))
                scorePart = scorePart + increment
                newScore = scorePart * (tiebreakerMax + 1) + newTiebreaker
            else
                newScore = increment * (tiebreakerMax + 1) + newTiebreaker
            end

            redis.call('ZADD', key, newScore, member)
            return tostring(newScore)
        ";

        var result = await _db.ScriptEvaluateAsync(script,
            new RedisKey[] { key },
            new RedisValue[] {
                member,
                incrementBy.ToString(),
                newTiebreaker.ToString(),
                TiebreakerMax.ToString()
            });

        return (double)result;
    }
}
```

### 5.4 — Scoreboard with "Around Me" View

```csharp
public class AroundMeLeaderboardService
{
    private readonly IDatabase _db;

    public AroundMeLeaderboardService(IDatabase db) => _db = db;

    public async Task<LeaderboardAroundPlayer> GetAroundPlayerAsync(
        string key, string playerId, int radius = 5)
    {
        long? rank = await _db.SortedSetRankAsync(
            key, playerId, Order.Descending);

        if (rank == null)
        {
            return new LeaderboardAroundPlayer
            {
                PlayerFound = false,
                Entries = Array.Empty<LeaderboardEntry>()
            };
        }

        long start = Math.Max(0, rank.Value - radius);
        long stop = rank.Value + radius;

        SortedSetEntry[] entries = await _db.SortedSetRangeByRankWithScoresAsync(
            key, start, stop, Order.Descending);

        var result = new List<LeaderboardEntry>();
        for (int i = 0; i < entries.Length; i++)
        {
            result.Add(new LeaderboardEntry
            {
                Rank = (int)(start + i + 1),
                PlayerId = entries[i].Element.ToString(),
                Score = (long)entries[i].Score,
                IsRequestingPlayer = entries[i].Element == playerId
            });
        }

        return new LeaderboardAroundPlayer
        {
            PlayerFound = true,
            PlayerRank = (int)(rank.Value + 1),
            Entries = result.ToArray()
        };
    }
}

public class LeaderboardAroundPlayer
{
    public bool PlayerFound { get; set; }
    public int? PlayerRank { get; set; }
    public LeaderboardEntry[] Entries { get; set; } = Array.Empty<LeaderboardEntry>();
}

public class LeaderboardEntry
{
    public int Rank { get; set; }
    public string PlayerId { get; set; } = string.Empty;
    public long Score { get; set; }

}
```

## Section 6 — Performance and Complexity

---

### 6.1 — Time Complexity

| Command | Complexity | Notes |
|---|---|---|
| ZRANK | O(log N) | Uses skiplist to locate member |
| ZREVRANK | O(log N) | Same as ZRANK |
| ZINCRBY | O(log N) | Update score + reposition in skiplist |
| ZSCORE | O(1) | Uses hash table for direct lookup |

---

### 6.2 — ZINCRBY vs Read-Modify-Write

```
Approach 1: Read-Modify-Write (NOT atomic — race condition)
  score = GET key member     ← Thread 1 and 2 both read "100"
  score += 50                ← Thread 1: 150, Thread 2: 150
  SET key member score       ← Both write 150 (lost increment!)
  Result: 150 instead of 200

Approach 2: WATCH + MULTI + EXEC (atomic but retry on conflict)
  WATCH key member           ← Watch for changes
  score = GET key member     ← Read
  score += 50                ← Compute
  MULTI                      ← Begin transaction
  SET key member score       ← Write
  EXEC                       ← Execute; if key changed, retry

Approach 3: ZINCRBY (atomic, single command, no retry)
  ZINCRBY key member 50      ← Single atomic command
  Result: 200 guaranteed
```

ZINCRBY is both simpler and more performant than approaches 1 and 2.

---

### 6.3 — ZRANK/ZREVRANK vs Scanning

Finding a member's rank by scanning the sorted set on the client would be O(N). ZRANK uses the skiplist to find the member in O(log N).

```
Sorted set with 1,000,000 members:
  Client scan:       1,000,000 operations (infeasible)
  ZRANK:             ~20 skiplist level traversals (log₂ 1M ≈ 20)
```

---

### 6.4 — Single vs Batched Queries

```csharp
// Sequential (3 round trips)
long? rank = await db.SortedSetRankAsync(key, member, Order.Descending);
double? score = await db.SortedSetScoreAsync(key, member);
long total = await db.SortedSetLengthAsync(key);

// Parallel (1 network round trip with 3 multiplexed operations)
Task<long?> rankTask = db.SortedSetRankAsync(key, member, Order.Descending);
Task<double?> scoreTask = db.SortedSetScoreAsync(key, member);
Task<long> totalTask = db.SortedSetLengthAsync(key);
await Task.WhenAll(rankTask, scoreTask, totalTask);
```

When using StackExchange.Redis's multiplexer, concurrent tasks are automatically pipelined — they share a single connection and are sent together. Always issue independent queries concurrently.

---

### 6.5 — Benchmark Estimates

```
Single node, localhost, 100,000 members in sorted set:

ZRANK (existing member):    ~0.05ms
ZRANK (nonexistent):        ~0.05ms (same, must traverse to end)
ZINCRBY (increment):        ~0.08ms (read + update + reposition)
ZSCORE:                     ~0.01ms (O(1) hash lookup)
```

With network latency ~1ms, expect ~1-2ms per command.

## Section 7 — Gotchas and Pitfalls

---

### 7.1 — ZRANK Returns Null for Nonexistent Members

In StackExchange.Redis, `SortedSetRankAsync` returns `long?` (nullable). Always check for null:

```csharp
long? rank = await db.SortedSetRankAsync(key, member, Order.Descending);

if (rank == null)
{
    // Member not in leaderboard
    Console.WriteLine("Player not ranked yet");
}
else
{
    // rank.Value is 0-based
    Console.WriteLine($"Player rank: {rank.Value + 1}"); // 1-based display
}
```

---

### 7.2 — Rank is 0-Based

Redis returns 0-based ranks. Most users expect 1-based ranks. Always convert:

```csharp
// Display rank
int displayRank = (int)(redisRank.Value + 1);

// Wrong: using 0-based rank directly
Console.WriteLine($"Rank: {redisRank}"); // Shows "Rank: 0" for #1
```

---

### 7.3 — ZINCRBY Creates Member If Not Exist

If you call ZINCRBY on a member that doesn't exist, Redis creates it with score = increment:

```bash
> ZINCRBY leaderboard 100 "newbie"
"100"               # Member "newbie" created with score 100
```

This is usually desired, but be aware: it means ZINCRBY cannot be used to test for member existence. If you need to check first, use ZSCORE:

```csharp
double? existing = await db.SortedSetScoreAsync(key, member);
if (existing == null)
{
    // Handle new member (e.g., apply initial score logic)
    await db.SortedSetAddAsync(key, member, initialScore);
}
else
{
    await db.SortedSetIncrementAsync(key, member, delta);
}
```

---

### 7.4 — ZINCRBY with Negative Delta (Decrement)

Use negative values with ZINCRBY or use `SortedSetDecrementAsync`:

```csharp
// Both are equivalent:
double newScore = await db.SortedSetIncrementAsync(key, member, -50);
double newScore = await db.SortedSetDecrementAsync(key, member, 50);
```

Both return the new score. The score can become negative.

---

### 7.5 — Floating Point Precision

ZRANK, ZREVRANK, and ZINCRBY all use double-precision scores. Be careful with very large scores or fractional increments:

```csharp
// Precision loss for scores > 2^53
double large = 9007199254740993.0; // > 2^53
await db.SortedSetIncrementAsync(key, "member", large);
// Score may be rounded to nearest representable double

// Fractional increments accumulate rounding error
for (int i = 0; i < 1000; i++)
    await db.SortedSetIncrementAsync(key, "member", 0.1);
// Result: ~100.0 (might be 99.9999999999986 due to IEEE 754)
```

---

### 7.6 — ZRANK on Empty Set or Missing Key

When the key does not exist or is empty:
- ZRANK returns nil (null)
- ZINCRBY creates the sorted set and the member

```csharp
long? rank = await db.SortedSetRankAsync(
    "nonexistent:key", "player1");
// rank == null

double newScore = await db.SortedSetIncrementAsync(
    "nonexistent:key", "player1", 100);
// newScore == 100, sorted set created
```

---

### 7.7 — Race Condition with ZRANK + ZINCRBY Pattern

A common anti-pattern: read rank, then increment. The rank is stale by the time you use it:

```csharp
// ANTI-PATTERN — race condition
long? rank = await db.SortedSetRankAsync(key, member, Order.Descending);
int displayRank = (int)(rank.Value + 1);

await db.SortedSetIncrementAsync(key, member, 50);
// rank is now stale — another thread may have incremented too
```

**Solution:** If you need "rank before increment" and "score after increment" atomically, use a Lua script:

```csharp
string script = @"
    local rank = redis.call('ZREVRANK', KEYS[1], ARGV[1])
    local newScore = redis.call('ZINCRBY', KEYS[1], ARGV[2], ARGV[1])
    return {rank, newScore}
";

var result = (RedisResult[])await db.ScriptEvaluateAsync(script,
    new RedisKey[] { key },
    new RedisValue[] { member, increment.ToString() });

long? rank = (long?)result[0];
double newScore = (double)result[1];
```

---

### 7.8 — ZRANK with Lexicographic Tiebreakers

When scores are equal, Redis uses lexicographic ordering. This affects ZRANK:

```bash
> ZADD test 100 alpha 100 beta 100 gamma
> ZRANK test "beta"
(integer) 1          # lexicographic order: alpha(0), beta(1), gamma(2)

# With descending order:
> ZREVRANK test "beta"
(integer) 1          # lexicographic reversed: gamma(0), beta(1), alpha(2)
```

If tiebreaker order matters, encode it explicitly in the score rather than relying on member names.

---

### 7.9 — ZINCRBY Return Type

ZINCRBY returns the new score as a string representation of a double. In StackExchange.Redis, this is automatically converted to a .NET `double`. For integer scores, cast to `long`:

```csharp
double result = await db.SortedSetIncrementAsync(key, member, 1);
long score = (long)result; // safe if scores are integers < 2^53
```

---

### 7.10 — WRONGTYPE Error

If the key exists but is not a sorted set:
```bash
> SET mystring "hello"
> ZRANK mystring "member"
(error) WRONGTYPE Operation against a key holding the wrong kind of value
```

Handle in .NET:
```csharp
try
{
    long? rank = await db.SortedSetRankAsync(key, member);
}
catch (RedisServerException ex) when (ex.Message.Contains("WRONGTYPE"))
{
    // Key is not a sorted set
}
```

## Section 8 — Related Patterns and Comparisons

---

### 8.1 — ZRANK vs ZSCORE

| Aspect | ZRANK | ZSCORE |
|---|---|---|
| Returns | Position (0-based) | Score value |
| Complexity | O(log N) | O(1) |
| Use | "What place am I?" | "How many points do I have?" |
| Null behavior | nil if missing | nil if missing |

Use ZRANK for relative positioning (leaderboard display). Use ZSCORE for raw score queries (progress bars, thresholds).

---

### 8.2 — ZINCRBY vs Transactional Increment

```
ZINCRBY (single command):
  - Atomic by design
  - O(log N) complexity
  - No overhead for WATCH/MULTI/EXEC
  - Always succeeds

WATCH + GET + MULTI + SET + EXEC:
  - Atomic (with retry on conflict)
  - Requires handling TransactionAbortedException
  - Higher latency due to multiple round trips
  - Better for complex multi-key operations
```

For simple score increments, ZINCRBY is always preferred.

---

### 8.3 — ZRANK vs ZRANGE for Rank Discovery

| Method | Use | Complexity |
|---|---|---|
| ZRANK member | Find rank of known member | O(log N) |
| ZRANGE | List members by rank | O(log N + M) |
| ZRANGEBYSCORE | List members by score | O(log N + M) |

If you know the member and need its position, ZRANK is the right tool. ZRANGE is for enumerating a range of members.

---

### 8.4 — Reversed Score Trick for ZRANK "High Score Wins" Default

ZRANK is ascending (low score = rank 0). For leaderboards where high score wins, use ZREVRANK instead. Alternatively, store negative scores:

```bash
# Store negative scores so ZRANK returns "high score wins"
ZADD leaderboard -3000 playerD
ZADD leaderboard -2500 playerC
ZADD leaderboard -1000 playerB
ZADD leaderboard -500 playerA

# Now ZRANK works like ZREVRANK would with positive scores
ZRANK leaderboard "playerD"
(integer) 0  # highest score (most negative) = rank 0
```

This is a legacy trick; using ZREVRANK is clearer.

---

### 8.5 — ZINCRBY in Lua Scripting for Multi-Key Operations

```csharp
/// <summary>
/// Transfer points from one player to another atomically
/// </summary>
public async Task TransferPointsAsync(
    string leaderboardKey,
    string fromPlayer,
    string toPlayer,
    double points)
{
    string script = @"
        local fromPlayer = ARGV[1]
        local toPlayer = ARGV[2]
        local points = tonumber(ARGV[3])

        local fromScore = redis.call('ZSCORE', KEYS[1], fromPlayer)
        if not fromScore or fromScore < points then
            return redis.error_reply('Insufficient points')
        end

        redis.call('ZINCRBY', KEYS[1], -points, fromPlayer)
        redis.call('ZINCRBY', KEYS[1], points, toPlayer)

        return redis.status_reply('OK')
    ";

    await _db.ScriptEvaluateAsync(script,
        new RedisKey[] { leaderboardKey },
        new RedisValue[] { fromPlayer, toPlayer, points.ToString() });
}
```

### 9.1 — Quick Reference Card

| Count in score range | `ZCOUNT key min max` | `SortedSetLengthAsync(key, min, max)` |

### 9.2 — Command Cheatsheet

```bash
# Setup test data
ZADD leaderboard 1000 alice 2500 bob 500 charlie 3000 diana

# Get ranks
ZRANK leaderboard "alice"       # 1 (0-based: 1, ascending)
ZREVRANK leaderboard "alice"    # 2 (0-based: 2, descending)

# Increment/decrement
ZINCRBY leaderboard 500 "alice"    # alice: 1500
ZINCRBY leaderboard -200 "bob"     # bob: 2300

# Check scores
ZSCORE leaderboard "diana"     # 3000

# Combine rank and score (Redis 7.2+)
ZRANK leaderboard "alice" WITHSCORE     # rank + score in one call
ZREVRANK leaderboard "alice" WITHSCORE

# Get total players and count in range
ZCARD leaderboard              # 4
ZCOUNT leaderboard 1000 2000   # 2
```

### 9.3 — Key Design Principles

1. **Always handle null** — `SortedSetRankAsync` returns nullable `long?`
2. **Convert to 1-based** for user-facing display
3. **Use ZINCRBY for atomic increments** — never read-modify-write
4. **Pipeline independent queries** — ZRANK + ZSCORE + ZCARD can run in parallel
5. **Lua for multi-step atomic operations** — rank-change detection, transfers
6. **Mind precision** — integer scores < 2^53 are safe; fractional scores accumulate error
7. **ZINCRBY creates members** — use ZSCORE first if you need to check existence
