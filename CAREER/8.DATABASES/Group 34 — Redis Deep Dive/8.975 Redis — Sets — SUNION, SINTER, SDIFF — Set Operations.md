# 1 — Overview

Redis Set operations extend the basic Set commands to perform mathematical set theory operations across multiple Sets. They transform Redis from a simple key-value store into a computational engine capable of executing set logic server-side, eliminating the need to transfer data to the client for processing.

The three primary Set operations are:

| Command | Mathematical Operation | Description | Time Complexity |
| ------- | ---------------------- | ----------- | --------------- |
| SUNION key [key ...] | A ∪ B (Union) | Returns members present in ANY of the Sets | O(N) where N = total members across all Sets |
| SINTER key [key ...] | A ∩ B (Intersection) | Returns members present in ALL of the Sets | O(N * M) worst case |
| SDIFF key [key ...] | A ∖ B (Difference) | Returns members present in the first Set but NOT in any subsequent Set | O(N) where N = total members across all Sets |

Each operation has a corresponding *STORE variant that stores the result into a new key:

| Variant | Purpose | Return Value |
| ------- | ------- | ------------ |
| SUNIONSTORE dest key [key ...] | Store union result | Number of members in the resulting Set |
| SINTERSTORE dest key [key ...] | Store intersection result | Number of members in the resulting Set |
| SDIFFSTORE dest key [key ...] | Store difference result | Number of members in the resulting Set |

These operations are the foundation for many real-world features: mutual friends, common interests, permission overlap, recommendation exclusions, and analytics segmentation.

## 3.8 — Mutual Friends Service

`csharp
public class MutualFriendsService
{
    private readonly IDatabase _db;
    private const string FriendsPrefix = "friends:";

    public MutualFriendsService(IDatabase db) => _db = db;

    public async Task AddFriendAsync(long userId, long friendId)
    {
        var tran = _db.CreateTransaction();
        tran.SetAddAsync($"{FriendsPrefix}{userId}", friendId);
        tran.SetAddAsync($"{FriendsPrefix}{friendId}", userId);
        await tran.ExecuteAsync();
    }

    public async Task<string[]> GetMutualFriendsAsync(long userIdA, long userIdB)
    {
        RedisKey keyA = $"{FriendsPrefix}{userIdA}";
        RedisKey keyB = $"{FriendsPrefix}{userIdB}";

        RedisValue[] mutual = await _db.SetCombineAsync(
            SetOperation.Intersect, keyA, keyB);

        return mutual.Select(m => m.ToString()).ToArray();
    }

    public async Task<long> GetMutualFriendCountAsync(long userIdA, long userIdB)
    {
        RedisKey tempKey = $"temp:mutual:{userIdA}:{userIdB}:{Guid.NewGuid():N}";

        try
        {
            long count = await _db.SetCombineAndStoreAsync(
                SetOperation.Intersect, tempKey,
                $"{FriendsPrefix}{userIdA}",
                $"{FriendsPrefix}{userIdB}");

            await _db.KeyExpireAsync(tempKey, TimeSpan.FromSeconds(30));
            return count;
        }
        catch
        {
            await _db.KeyDeleteAsync(tempKey);
            throw;
        }
    }

    public async Task<string[]> GetFriendSuggestionsAsync(long userId)
    {
        string userFriendsKey = $"{FriendsPrefix}{userId}";

        RedisValue[] friendIds = await _db.SetMembersAsync(userFriendsKey);
        if (friendIds.Length == 0) return Array.Empty<string>();

        var friendOfFriendKeys = friendIds
            .Select(id => (RedisKey)$"{FriendsPrefix}{id}")
            .ToArray();

        RedisValue[] friendsOfFriends = await _db.SetCombineAsync(
            SetOperation.Union, friendOfFriendKeys);

        return friendsOfFriends.Select(f => f.ToString()).ToArray();
    }
}

## 3.9 — Set Operations with Error Handling

`csharp
public async Task<string[]> SafeSetCombineAsync(
    SetOperation operation,
    RedisKey[] keys,
    int retryCount = 3)
{
    int attempt = 0;
    TimeSpan delay = TimeSpan.FromMilliseconds(200);

    while (attempt < retryCount)
    {
        try
        {
            long existsCount = 0;
            foreach (var key in keys)
            {
                bool exists = await _db.KeyExistsAsync(key);
                if (exists) existsCount++;
            }

            if (existsCount == 0)
                return Array.Empty<string>();

            var result = await _db.SetCombineAsync(operation, keys);
            return result.Select(r => r.ToString()).ToArray();
        }
        catch (RedisTimeoutException ex) when (attempt < retryCount - 1)
        {
            attempt++;
            await Task.Delay(delay);
            delay *= 2;
        }
        catch (RedisConnectionException ex) when (attempt < retryCount - 1)
        {
            attempt++;
            await Task.Delay(delay);
            delay *= 2;
        }
        catch (RedisException ex)
        {
            Console.WriteLine($"Unexpected Redis error: {ex.Message}");
            throw;
        }
    }
    throw new TimeoutException("Max retries exceeded");
}
`

## 3.10 — Client-Side Intersection for Large Sets

`csharp
public async Task<string[]> ClientSideIntersectAsync(
    RedisKey smallSetKey,
    RedisKey largeSetKey,
    int smallSetThreshold = 1000)
{
    long setSize = await _db.SetLengthAsync(smallSetKey);

    if (setSize > smallSetThreshold)
        return await IntersectSetsAsync(smallSetKey, largeSetKey);

    RedisValue[] smallMembers = await _db.SetMembersAsync(smallSetKey);

    var common = new List<string>();
    foreach (RedisValue member in smallMembers)
    {
        bool exists = await _db.SetContainsAsync(largeSetKey, member);
        if (exists)
            common.Add(member.ToString());
    }

    return common.ToArray();
}
`

## 3.12 — Progressive Intersection

`csharp
public async IAsyncEnumerable<string> ProgressiveIntersectAsync(
    RedisKey[] keys, int batchSize = 100)
{
    var keyCounts = new List<(RedisKey Key, long Count)>();
    foreach (var key in keys)
    {
        long count = await _db.SetLengthAsync(key);
        keyCounts.Add((key, count));
    }

    keyCounts.Sort((a, b) => a.Count.CompareTo(b.Count));

    RedisKey smallestKey = keyCounts[0].Key;
    int cursor = 0;

    do
    {
        var scanResult = await _db.SetScanAsync(smallestKey, cursor: cursor, pageSize: batchSize);
        cursor = scanResult.Cursor;

        foreach (RedisValue candidate in scanResult.Values)
        {
            bool inAll = true;
            for (int i = 1; i < keyCounts.Count; i++)
            {
                if (!await _db.SetContainsAsync(keyCounts[i].Key, candidate))
                {
                    inAll = false;
                    break;
                }
            }
            if (inAll)
                yield return candidate.ToString();
        }
    }
    while (cursor != 0);
}
`

## 3.13 — Caching Set Operation Results

`csharp
public class CachedSetOperations
{
    private readonly IDatabase _db;
    private static readonly TimeSpan DefaultCacheTtl = TimeSpan.FromMinutes(5);

    public CachedSetOperations(IDatabase db) => _db = db;

    public async Task<string[]> GetOrComputeIntersectionAsync(
        string cacheKey,
        RedisKey[] sourceKeys,
        TimeSpan? ttl = null)
    {
        bool cached = await _db.KeyExistsAsync(cacheKey);
        if (cached)
        {
            RedisValue[] cachedResult = await _db.SetMembersAsync(cacheKey);
            return cachedResult.Select(r => r.ToString()).ToArray();
        }

        long count = await _db.SetCombineAndStoreAsync(
            SetOperation.Intersect, cacheKey, sourceKeys);

        if (count > 0)
            await _db.KeyExpireAsync(cacheKey, ttl ?? DefaultCacheTtl);

        RedisValue[] result = await _db.SetMembersAsync(cacheKey);
        return result.Select(r => r.ToString()).ToArray();
    }

    public async Task InvalidateCachedResultAsync(string cacheKey)
    {
        await _db.KeyDeleteAsync(cacheKey);
    }
}
`

## 3.14 — Multi-Operation Pipeline

`csharp
public async Task<SetOperationResult> ComputeAllOperationsAsync(
    RedisKey keyA, RedisKey keyB)
{
    var batch = _db.CreateBatch();

    Task<RedisValue[]> unionTask = batch.SetCombineAsync(SetOperation.Union, keyA, keyB);
    Task<RedisValue[]> interTask = batch.SetCombineAsync(SetOperation.Intersect, keyA, keyB);
    Task<RedisValue[]> diffATask = batch.SetCombineAsync(SetOperation.Difference, keyA, keyB);
    Task<RedisValue[]> diffBTask = batch.SetCombineAsync(SetOperation.Difference, keyB, keyA);

    await batch.ExecuteAsync();

    return new SetOperationResult
    {
        Union = (await unionTask).Select(r => r.ToString()).ToArray(),
        Intersection = (await interTask).Select(r => r.ToString()).ToArray(),
        DiffAminusB = (await diffATask).Select(r => r.ToString()).ToArray(),
        DiffBminusA = (await diffBTask).Select(r => r.ToString()).ToArray()
    };
}

public class SetOperationResult
{
    public string[] Union { get; set; }
    public string[] Intersection { get; set; }
    public string[] DiffAminusB { get; set; }
    public string[] DiffBminusA { get; set; }
}

using StackExchange.Redis;

public class RedisSetOperationsService : IDisposable
{

        }

            throw;

    public async Task<long> DifferenceAndStoreAsync(RedisKey destination, params RedisKey[] keys)
    {

        {

        {
            _logger?.LogError(ex, "SDIFFSTORE failed");
            throw;
        }
    }

    public async Task<bool> AreDisjointAsync(RedisKey keyA, RedisKey keyB)
    {
        long intersectionSize = await _db.SetCombineAndStoreAsync(
            SetOperation.Intersect,
            (RedisKey)$"__check_disjoint:{Guid.NewGuid():N}",
            keyA, keyB);
        return intersectionSize == 0;
    }

    public async Task<bool> IsSubsetAsync(RedisKey subsetKey, RedisKey supersetKey)
    {
        long subsetSize = await _db.SetLengthAsync(subsetKey);
        if (subsetSize == 0) return true;

        long intersectionSize = await _db.SetCombineAndStoreAsync(
            SetOperation.Intersect,
            (RedisKey)$"__check_subset:{Guid.NewGuid():N}",
            subsetKey, supersetKey);
        return subsetSize == intersectionSize;
    }

    public async Task<double> GetJaccardSimilarityAsync(RedisKey keyA, RedisKey keyB)
    {
        string tempInter = $"__jaccard_inter:{Guid.NewGuid():N}";
        string tempUnion = $"__jaccard_union:{Guid.NewGuid():N}";

        var batch = _db.CreateBatch();
        Task<long> interCount = batch.SetCombineAndStoreAsync(
            SetOperation.Intersect, tempInter, keyA, keyB);
        Task<long> unionCount = batch.SetCombineAndStoreAsync(
            SetOperation.Union, tempUnion, keyA, keyB);
        await batch.ExecuteAsync();

        long inter = await interCount;
        long union = await unionCount;

        await _db.KeyDeleteAsync(tempInter);
        await _db.KeyDeleteAsync(tempUnion);

        if (union == 0) return 1.0;
        return (double)inter / union;
    }

    private void ValidateKeys(RedisKey[] keys, int minimum)

    }

# 4 — Performance Characteristics

## 4.1 — Time Complexity Details

| Operation | Time Complexity | Worst-Case Scenario |
| --------- | --------------- | ------------------- |
| SUNION | O(N) where N = sum of all set sizes | All members unique |
| SINTER | O(N × M) worst case | Many sets, large smallest set |
| SDIFF | O(N) where N = sum of all set sizes | Large first set, no overlap |
| SINTERSTORE | Same as SINTER + O(K) | + memory allocation for result |
| SUNIONSTORE | Same as SUNION + O(K) | + memory allocation for result |
| SDIFFSTORE | Same as SDIFF + O(K) | + memory allocation for result |

## 4.2 — SINTER Optimization Strategy

Redis optimizes SINTER by sorting input sets by size (smallest first) and iterating the smallest set while checking membership in the others. Whether you write SINTER set:a set:b or SINTER set:b set:a, Redis internally uses the same strategy.

## 4.3 — Memory Impact of *STORE Operations

SINTERSTORE, SUNIONSTORE, and SDIFFSTORE create new Sets. For large operations:
- Redis must allocate memory for the result Set
- Memory usage peaks during computation (source + temp + result)
- May double memory usage temporarily

## 4.4 — Network Bandwidth

Non-store variants transfer the full result to the client. For large results, use *STORE variants and iterate with SSCAN server-side.

## 4.5 — Benchmark Data

Approximate performance on modern single-instance Redis (6-core, 32 GB RAM, localhost):

| Operation | Set Sizes | Time (ms) |
| --------- | --------- | --------- |
| SINTER (2 sets) | 1K each | < 1 ms |
| SINTER (5 sets) | 1K each | 2–3 ms |
| SINTER (2 sets) | 100K each | 10–50 ms |
| SINTER (2 sets) | 1M each | 100–500 ms |
| SUNION (2 sets) | 100K each | 5–10 ms |
| SUNION (2 sets) | 1M each | 50–200 ms |
| SDIFF (2 sets) | 100K each | 5–10 ms |
| SINTERSTORE (2 sets) | 1M each | 150–600 ms |
| SINTERSTORE (5 sets) | 100K each | 50–200 ms |

## 4.6 — Server Load During Set Operations

Set operations are CPU-intensive. While they execute:
- Redis is single-threaded — no other commands can be processed
- Long-running operations cause latency spikes for all clients
- Schedule large *STORE operations during low-traffic periods or use a replica

# 6 — Comparison with Other Approaches

## 6.1 — Server-Side vs Client-Side Computation

| Aspect | Server-Side (Redis) | Client-Side |
| ------ | ------------------- | ----------- |
| Data transfer | Only result | All source sets |
| Network usage | Minimal | High |
| CPU load | Redis server | Application server |
| Concurrency | Atomic | Race conditions possible |
| Memory on client | Result only | All source sets + result |
| Latency | 1–2 round trips | N round trips + computation |

## 6.2 — Set Operations vs SQL Joins

| Aspect | Redis Set Operations | SQL Joins |
| ------ | ------------------- | --------- |
| Data model | Key-value, denormalized | Relational, normalized |
| Performance | In-memory, sub-ms | Disk-based, variable |
| Atomicity | Single command | Transaction |
| Complexity | Simple, limited | Full relational algebra |
| Indexes | Always O(1) membership | Requires index tuning |

## 6.3 — Sets vs Bloom Filters

| Aspect | Redis Sets | Bloom Filters |
| ------ | ---------- | ------------- |
| Membership | Exact | Probabilistic (false positives) |
| Intersection | Exact | Not supported |
| Memory | High per member | Fixed, small |
| Operations | Full set math | Add + Check only |

## 6.4 — When to Use *STORE Variants

Use *STORE when:
- The result will be queried multiple times
- The result is too large to transfer to the client
- You need to apply TTL to the result
- You want to chain operations

Use non-store when:
- The result is small (a few hundred members)
- The result is used once
- Memory on Redis is constrained

## 6.5 — SUNION vs Multiple SADD

SUNION reads existing sets and creates a result on the fly. Multiple SADD calls read all members from source sets and add them to a new set. SUNION is better for one-time queries; SADD is better if the union needs to persist.

# 8 — Gotchas and Pitfalls

## 8.1 — *STORE Operations Are Blocking

The *STORE variants block the Redis server during computation. Since Redis is single-threaded, no other commands can execute while a SINTERSTORE on two 10-million-member sets is running. This causes latency spikes for all other clients.

**Mitigation:**
- Check set sizes before running large operations
- Schedule during low-traffic periods
- Use a dedicated Redis replica for expensive operations
- Set timeouts and handle failures gracefully

`csharp
long sizeA = await _db.SetLengthAsync("huge:set:a");
long sizeB = await _db.SetLengthAsync("huge:set:b");
if (sizeA > 1_000_000 || sizeB > 1_000_000)
{
    await QueueBackgroundJobAsync("intersect", "huge:set:a", "huge:set:b");
}
else
{
    await _db.SetCombineAndStoreAsync(
        SetOperation.Intersect, "result", "huge:set:a", "huge:set:b");
}
`

## 8.2 — Set Operations Are O(N) — Plan Accordingly

Running large intersections in web request handlers blocks Redis. Use caching or background processing.

## 8.3 — SDIFF Order Matters

SDIFF set:a set:b returns members in set:a NOT in set:b. Reversing the sets produces a completely different result. Always verify order.

## 8.4 — Empty and Non-Existent Set Behavior

Non-existent keys are treated as empty sets. SDIFF existing:set non:existent:key returns all members of existing:set.

## 8.5 — Key Existence Check Is Not Atomic

Checking if keys exist before a Set operation introduces race conditions. Run the operation and handle exceptions instead.

## 8.6 — WRONGTYPE Error

If any key in a Set operation is not a Set, Redis returns WRONGTYPE Operation against a key holding the wrong kind of value. Handle this exception in code.

`csharp
try
{
    await _db.SetCombineAsync(SetOperation.Intersect, "key1", "key2");
}
catch (RedisException ex) when (ex.Message.Contains("WRONGTYPE"))
{
    // Handle wrong type
}
`

## 8.7 — Memory Fragmentation After Large SINTERSTORE

Creating and deleting large temporary Sets causes memory churn. StackExchange.Redis uses UNLINK (non-blocking) by default for KeyDeleteAsync.

## 8.8 — Large Result Sets and Client Memory

Non-store variants transfer the full result to the client. For results with millions of members, use *STORE + SSCAN iteration.

## 8.9 — No Indexes on Sets

Unlike databases, Redis Sets have no secondary indexes. SINTER iterates one set and does hash lookups on the rest. The only optimization is choosing the smallest set for iteration.

## 8.10 — Variadic Argument Limits

SUNION, SINTER, and SDIFF accept multiple keys but are limited by proto-max-bulk-len (default 512 MB). For hundreds of keys, batch processing is necessary.

`csharp
public async Task<string[]> IntersectManyKeysAsync(IEnumerable<RedisKey> keys)
{
    var keyList = keys.ToList();
    if (keyList.Count == 0) return Array.Empty<string>();

    // Start with first set
    var result = await _db.SetMembersAsync(keyList[0]);
    var resultSet = new HashSet<string>(result.Select(r => r.ToString()));

    // Check each subsequent set
    for (int i = 1; i < keyList.Count && resultSet.Count > 0; i++)
    {
        var members = await _db.SetMembersAsync(keyList[i]);
        resultSet.IntersectWith(members.Select(m => m.ToString()));
    }

    return resultSet.ToArray();
}
`

# 9 — Summary and Related Notes

## 9.1 — Key Takeaways

1. **SUNION**: All unique members from any input Set (A ∪ B)
2. **SINTER**: Members present in ALL input Sets (A ∩ B)
3. **SDIFF**: Members in first Set NOT in subsequent Sets (A ∖ B)
4. **SINTERSTORE/SUNIONSTORE/SDIFFSTORE**: Store results server-side
5. All operations are O(N) and block Redis during execution
6. For large Sets, prefer *STORE + SSCAN over transferring to client
7. Redis optimizes SINTER by iterating smallest set first
