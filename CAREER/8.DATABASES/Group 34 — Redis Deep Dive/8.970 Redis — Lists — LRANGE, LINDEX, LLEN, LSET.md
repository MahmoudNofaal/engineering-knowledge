LRANGE reads what LPUSH wrote; LINDEX plumbs the linked-list internals; LLEN queries the length counter; LSET mutates an element at a specific position. Without the push/pop muscle memory, the range and index operations lack context about which end is head vs tail and how indices shift on insert/delete.

### Where This Fits

LRANGE, LINDEX, LLEN, and LSET are the read-and-mutate operations that make Redis lists usable beyond simple queues and stacks. A .NET backend engineer uses LRANGE to paginate a list of recent notifications, LINDEX to peek at a specific position without popping, LLEN to check queue depth before scheduling work, and LSET to replace an element at a known index (e.g., retry count on a failed job). When these operations are unknown, engineers either pop elements to inspect them (destroying the list in the process) or fall back to polling with LLEN to approximate queue depth. The interview signal is the "how do you get the third element from a list without removing it" question — the candidate who says LINDEX without hesitation understands the distinction between read and destructive operations.

## 2. Core Mental Model — Overview & Classification

LRANGE returns a contiguous slice of the list — O(N) in the number of returned elements, but O(N) in traversal distance as well (Redis walks from the nearest end to the start offset, then continues to the end offset). LINDEX returns a single element at the specified index — Redis traverses from the nearest end (head or tail, whichever is closer) to the target position, making it O(N) in list length for arbitrary indices but O(1) for index 0 or -1 (head or tail). LLEN returns the list length in O(1) time because Redis stores the length as an atomic counter on the list object. LSET replaces the element at a given index — Redis must first traverse to the index (O(N)), then mutate the ziplist entry within the quicklist node (O(1) once at the node).

The mental model: a Redis list is an array-like interface on a linked-list implementation. Operations at the edges (0, -1) are fast; operations in the middle are slow. LLEN is the only operation that is always O(1) regardless of position. This asymmetry between the abstract interface (seemingly array-like with index access) and the concrete implementation (linked list) is the most common source of performance surprises.

### Classification

**For Redis data structures:** LRANGE/LINDEX/LLEN/LSET are the read and in-place mutation commands that complement the push/pop commands. They belong to the "non-destructive" family — LRANGE and LINDEX return data without modifying the list; LSET mutates in place without adding or removing elements. LLEN is a metadata query that does not touch the list data at all.

### Key Properties

| Property | Value | Notes |
|---|---|---|
| LRANGE time complexity | O(S + N) | S = offset distance from nearest end, N = returned element count |
| LINDEX time complexity | O(N) | N = index to traverse; index 0 or -1 is effectively O(1) (head/tail pointer) |
| LLEN time complexity | O(1) | List stores length as an integer counter, updated atomically on every push/pop/trim |
| LSET time complexity | O(N) | Must traverse to index first, then O(1) ziplist mutation within the quicklist node |
| LRANGE memory | O(N) | Returns N elements — full elements serialized in RESP protocol |
| LSET safety | Index must exist | LSET returns error if index is out of range — no auto-extend behavior |
| Negative indexing | Supported | -1 is last, -2 is second-to-last, etc. |
| Zero-based indices | Yes | Like C# arrays — index 0 is the first element (LPUSH end) |
| .NET (SE.Redis) | Async methods | ListRangeAsync, ListGetByIndexAsync, ListLengthAsync, ListSetByIndexAsync |
| Return type SE.Redis | RedisValue[] / RedisValue | Null check IsNull for LINDEX out-of-range |
| Overflow behavior | Index out of range returns nil (LINDEX) or error (LSET) | LINDEX returns nil for out-of-range; LSET throws WRONGTYPE or execution error |

## 3. Deep Mechanics — How Redis Executes Range, Index, Length, Set

### How Redis Executes LRANGE, LINDEX, LLEN, LSET

**LRANGE execution:**

**Step 1 — Command parsing:** Redis matches LRANGE to `lrangeCommand`. It parses the key, start index, and stop index.

**Step 2 — Index normalization:** Redis converts negative indices to positive: -1 becomes length - 1, -2 becomes length - 2, etc. If start > stop after normalization (e.g., LRANGE key 5 3 on a 10-element list), Redis returns an empty array immediately — no traversal.

**Step 3 — Quicklist iteration:** Redis seeks to the start position. From the nearest end (head for indices near 0, tail for indices near length), it traverses quicklist nodes and ziplist entries until reaching the start index.

**Step 4 — Element collection:** From the start position, Redis continues traversing forward, collecting elements into a response array, until it reaches the stop index or the end of the list, whichever comes first.

**Step 5 — Response:** Returns the array of elements as RESP bulk strings. An empty range returns an empty array.

**LINDEX execution:**

**Step 1 — Command parsing:** Redis matches to `lindexCommand`.

**Step 2 — Index normalization:** If the index is negative, converts to positive: `index = length + index`. If the normalized index is out of range (negative or >= length), returns nil immediately.

**Step 3 — Quicklist traversal:** Redis determines the nearest end to the target index. It walks from that end through quicklist nodes, counting ziplist entries until reaching the target index.

**Step 4 — Element extraction:** Redis reads the element at that position from the ziplist.

**Step 5 — Response:** Returns the element as a bulk string, or nil if out of range.

**LLEN execution:**

**Step 1 — Command parsing:** Redis matches to `llenCommand`.

**Step 2 — Key lookup:** Redis looks up the key. If the key does not exist, returns 0 (length of an empty list, even though the key does not exist — Redis treats missing keys as empty lists for LLEN).

**Step 3 — Length read:** Redis reads the `length` field from the list object header — this is an `unsigned long` maintained by every modifying operation (LPUSH, RPUSH, LPOP, RPOP, LINSERT, LREM, LTRIM).

**Step 4 — Response:** Returns the length as an integer.

**LSET execution:**

**Step 1 — Command parsing:** Redis matches to `lsetCommand`. Parses key, index, and value.

**Step 2 — Index normalization:** Converts negative indices to positive. Returns `ERR index out of range` if the normalized index is out of range — LSET does NOT auto-extend the list.

**Step 3 — Quicklist traversal:** Redis traverses to the target index (same mechanics as LINDEX).

**Step 4 — Ziplist mutation:** At the target position, Redis replaces the ziplist entry with the new value. This is an in-place update — it does not change the list length or shift any other elements.

**Step 5 — Response:** Returns `OK`.

```csharp
public class RedisListReaderService
{
    private readonly IDatabase _db;

    public RedisListReaderService(IDatabase db)
    {
        _db = db;
    }

    /// <summary>
    /// LRANGE — returns a range of elements.
    /// Start 0, stop -1 returns all elements.
    /// Returns empty array if list does not exist.
    /// </summary>
    public async Task<RedisValue[]> GetRangeAsync(
        string listKey, long start = 0, long stop = -1)
    {
        try
        {
            var results = await _db.ListRangeAsync(listKey, start, stop);
            return results;
        }
        catch (RedisConnectionException ex)
        {
            throw new RedisOperationException($"Failed LRANGE on {listKey}", ex);
        }
    }

    /// <summary>
    /// LINDEX — returns element at specific index.
    /// Returns null if index out of range.
    /// </summary>
    public async Task<RedisValue?> GetByIndexAsync(string listKey, long index)
    {
        try
        {
            var result = await _db.ListGetByIndexAsync(listKey, index);
            if (result.IsNull)
                return null;
            return result;
        }
        catch (RedisConnectionException ex)
        {
            throw new RedisOperationException($"Failed LINDEX on {listKey} at {index}", ex);
        }
    }

    /// <summary>
    /// LLEN — returns list length. 0 if list does not exist.
    /// </summary>
    public async Task<long> GetLengthAsync(string listKey)
    {
        try
        {
            return await _db.ListLengthAsync(listKey);
        }
        catch (RedisConnectionException ex)
        {
            throw new RedisOperationException($"Failed LLEN on {listKey}", ex);
        }
    }

    /// <summary>
    /// LSET — set element at index.
    /// Throws if index out of range.
    /// </summary>
    public async Task SetByIndexAsync(string listKey, long index, RedisValue value)
    {
        try
        {
            await _db.ListSetByIndexAsync(listKey, index, value);
        }
        catch (RedisServerException ex) when (ex.Message.Contains("index out of range"))
        {
            throw new ArgumentOutOfRangeException(
                $"Index {index} is out of range for list {listKey}", ex);
        }
        catch (RedisConnectionException ex)
        {
            throw new RedisOperationException($"Failed LSET on {listKey}", ex);
        }
    }

    /// <summary>
    /// Try LSET with index validation — check length first.
    /// </summary>
    public async Task<bool> TrySetByIndexAsync(string listKey, long index, RedisValue value)
    {
        var length = await _db.ListLengthAsync(listKey);
        var normalizedIndex = index >= 0 ? index : length + index;
        if (normalizedIndex < 0 || normalizedIndex >= length)
            return false;

        await _db.ListSetByIndexAsync(listKey, index, value);
        return true;
    }
}
```

### StackExchange.Redis — Pagination Pattern with LRANGE

```csharp
public class RedisPaginatedList
{
    private readonly IDatabase _db;

    public RedisPaginatedList(IDatabase db) => _db = db;

    /// <summary>
    /// Paginate through list contents with LRANGE.
    /// Page size = |stop - start| + 1.
    /// Returns (items, totalLength).
    /// </summary>
    public async Task<(RedisValue[] Items, long TotalLength)> GetPageAsync(
        string listKey, int pageNumber, int pageSize)
    {
        var start = (pageNumber - 1) * pageSize;
        var stop = start + pageSize - 1;

        // Execute both commands in a batch for single round trip
        var batch = _db.CreateBatch();
        var rangeTask = batch.ListRangeAsync(listKey, start, stop);
        var lengthTask = batch.ListLengthAsync(listKey);
        batch.Execute();

        await Task.WhenAll(rangeTask, lengthTask);

        return (rangeTask.Result, lengthTask.Result);
    }

    /// <summary>
    /// Peek at head element without popping.
    /// </summary>
    public async Task<RedisValue?> PeekHeadAsync(string listKey)
    {
        var result = await _db.ListGetByIndexAsync(listKey, 0);
        if (result.IsNull) return null;
        return result;
    }

    /// <summary>
    /// Peek at tail element without popping.
    /// </summary>
    public async Task<RedisValue?> PeekTailAsync(string listKey)
    {
        var result = await _db.ListGetByIndexAsync(listKey, -1);
        if (result.IsNull) return null;
        return result;
    }

    /// <summary>
    /// Peek at top 5 elements (newest first for LPUSH lists).
    /// </summary>
    public async Task<RedisValue[]> PeekTopAsync(string listKey, int count = 5)
    {
        return await _db.ListRangeAsync(listKey, 0, count - 1);
    }
}
```

### StackExchange.Redis — Type-Safe Wrapper with Serialization

```csharp
public class RedisListReader<T>
{
    private readonly IDatabase _db;
    private readonly IRedisSerializer _serializer;

    public RedisListReader(IDatabase db, IRedisSerializer serializer)
    {
        _db = db;
        _serializer = serializer;
    }

    /// <summary>
    /// LRANGE with deserialization — returns typed elements.
    /// </summary>
    public async Task<IReadOnlyList<T>> GetRangeAsync(string listKey, long start = 0, long stop = -1)
    {
        var results = await _db.ListRangeAsync(listKey, start, stop);
        return results
            .Where(r => !r.IsNull)
            .Select(r => _serializer.Deserialize<T>(r))
            .ToList();
    }

    /// <summary>
    /// LINDEX with deserialization.
    /// </summary>
    public async Task<T?> GetByIndexAsync(string listKey, long index) where T : class
    {
        var result = await _db.ListGetByIndexAsync(listKey, index);
        if (result.IsNull) return null;
        return _serializer.Deserialize<T>(result);
    }

    /// <summary>
    /// LSET with serialization.
    /// </summary>
    public async Task SetByIndexAsync(string listKey, long index, T item)
    {
        var serialized = _serializer.Serialize(item);
        await _db.ListSetByIndexAsync(listKey, index, serialized);
    }
}
```

### Execution Plan Analysis — Redis Complexity

```
LRANGE products 0 99
List: quicklist with 25 nodes (ziplists of ~40 elements each)
Start: 0 (head) → immediate, within first node
Stop: 99 → traverse 3 nodes (0, 1, 2), collecting 100 elements
Complexity: O(3 traversal + 100 response) ≈ O(103) → ~0.05ms on localhost
Memory: 100 strings in RESP protocol → ~5KB response (for 50-byte elements)

LINDEX products 50000
List: quicklist with 12500 nodes (100K elements, ~8 elements per node)
Traversal: from head, walk 1250 nodes to reach element 50000
Complexity: O(50000) → ~2.5ms on localhost
Compare to head: LINDEX products 0 → O(1) → ~0.005ms

LSET products 50000 "newvalue"
Same traversal as LINDEX (O(50000)) → ~2.5ms
Plus ziplist mutation within the node → O(1) → negligible

LLEN products
Counter read → O(1) → ~0.001ms
```

### Failure Modes

**LRANGE on non-existent key:** Returns empty array — no error. The caller receives zero elements, which may be silently interpreted as "no data" when the actual condition is "key not created yet." Mitigation: check `KeyExistsAsync` before LRANGE if empty results and missing keys need different handling.

**LINDEX out of range:** Returns nil (null in SE.Redis). Unlike C# array indexing, Redis does not throw — it returns a null response. Callers must check `IsNull`. Using the result as a string without checking produces an empty string.

**LSET out of range:** Returns `ERR index out of range`. SE.Redis throws `RedisServerException`. The caller must handle this exception or validate the index with LLEN first. LSET does NOT auto-extend the list — it is a mutation-only operation, not an insertion.

**LRANGE on huge list:** Requesting LRANGE 0 -1 on a list with 5M elements returns 5M elements in a single response. This consumes 5M * element-size bytes of Redis output buffer memory and network bandwidth. Redis may disconnect the client if the output buffer exceeds `client-output-buffer-limit`.

**Type mismatch:** Any of these commands on a non-list key returns `WRONGTYPE`. Except LLEN — LLEN returns `WRONGTYPE` on non-list keys because the command is type-specific (unlike EXISTS or TYPE which work on any key).

### Pattern 1 — Paginated Queue Inspection

```csharp
public class QueueInspector
{
    private readonly IDatabase _db;

    public QueueInspector(IDatabase db) => _db = db;

    /// <summary>
    /// Paginate through queue contents without consuming.
    /// Useful for dashboards and monitoring.
    /// </summary>
    public class QueuePageResult
    {
        public List<string> Items { get; set; } = new();
        public long TotalCount { get; set; }
        public int Page { get; set; }
        public int PageSize { get; set; }
        public int TotalPages => (int)Math.Ceiling((double)TotalCount / PageSize);
        public bool HasNext => Page < TotalPages;
        public bool HasPrevious => Page > 1;
    }

    public async Task<QueuePageResult> InspectQueueAsync(
        string queueKey, int page = 1, int pageSize = 50)
    {
        var start = (page - 1) * pageSize;
        var stop = start + pageSize - 1;

        var batch = _db.CreateBatch();
        var rangeTask = batch.ListRangeAsync(queueKey, start, stop);
        var lengthTask = batch.ListLengthAsync(queueKey);
        batch.Execute();

        await Task.WhenAll(rangeTask, lengthTask);

        var items = rangeTask.Result
            .Where(r => !r.IsNull)
            .Select(r => r.ToString())
            .ToList();

        return new QueuePageResult
        {
            Items = items,
            TotalCount = lengthTask.Result,
            Page = page,
            PageSize = pageSize
        };
    }

    /// <summary>
    /// Monitor queue depth over time.
    /// </summary>
    public async Task<long> GetQueueDepthAsync(string queueKey)
    {
        return await _db.ListLengthAsync(queueKey);
    }
}
```

### Pattern 2 — Job Retry Tracker with LSET

```csharp
public class JobRetryTracker
{
    private readonly IDatabase _db;
    private const string RetryListKey = "jobs:retry";

    public JobRetryTracker(IDatabase db) => _db = db;

    /// <summary>
    /// Enqueue a job with initial retry count.
    /// Each element is JSON: { "id": "guid", "data": "...", "retries": 0 }
    /// </summary>
    public async Task EnqueueJobAsync(string jobId, string jobData)
    {
        var entry = JsonSerializer.Serialize(new
        {
            Id = jobId,
            Data = jobData,
            Retries = 0,
            MaxRetries = 3
        });
        await _db.ListRightPushAsync(RetryListKey, entry);
    }

    /// <summary>
    /// Get job at index and inspect its retry count.
    /// </summary>
    public async Task<int> GetRetryCountAsync(long index)
    {
        var result = await _db.ListGetByIndexAsync(RetryListKey, index);
        if (result.IsNull)
            throw new ArgumentOutOfRangeException($"Index {index} out of range");

        var job = JsonDocument.Parse(result.ToString());
        return job.RootElement.GetProperty("Retries").GetInt32();
    }

    /// <summary>
    /// Increment retry count in place using LSET.
    /// This modifies the element without changing the list.
    /// </summary>
    public async Task IncrementRetryAsync(long index)
    {
        var result = await _db.ListGetByIndexAsync(RetryListKey, index);
        if (result.IsNull)
            throw new ArgumentOutOfRangeException($"Index {index} out of range");

        var job = JsonDocument.Parse(result.ToString());
        var retries = job.RootElement.GetProperty("Retries").GetInt32();
        var maxRetries = job.RootElement.GetProperty("MaxRetries").GetInt32();

        if (retries >= maxRetries)
        {
            // Remove the job — exceeded max retries
            // Using LSET with a tombstone marker or separate dead-letter queue
            await MarkAsDeadLetterAsync(index, result.ToString());
            return;
        }

        var updated = JsonSerializer.Serialize(new
        {
            Id = job.RootElement.GetProperty("Id").GetString(),
            Data = job.RootElement.GetProperty("Data").GetString(),
            Retries = retries + 1,
            MaxRetries = maxRetries
        });

        await _db.ListSetByIndexAsync(RetryListKey, index, updated);
    }

    private async Task MarkAsDeadLetterAsync(long index, string jobJson)
    {
        // LSET to mark as dead
        var deadLetter = JsonSerializer.Serialize(new
        {
            Dead = true,
            Original = jobJson,
            MovedAt = DateTime.UtcNow
        });
        await _db.ListSetByIndexAsync(RetryListKey, index, deadLetter);

        // Also push to dead-letter queue for later analysis
        await _db.ListRightPushAsync("jobs:deadletter", deadLetter);
    }
}
```

```csharp
public class ActivityFeed
{
    private readonly IDatabase _db;

    public ActivityFeed(IDatabase db) => _db = db;

    /// <summary>
    /// Add activity and trim to max size.
    /// </summary>
    public async Task AddActivityAsync(string feedKey, string activityJson)
    {
        await _db.ListLeftPushAsync(feedKey, activityJson);
        await _db.ListTrimAsync(feedKey, 0, 999); // Keep 1000 max
    }

    /// <summary>
    /// Get paginated feed — newest first.
    /// Page 1 returns items [0..19], page 2 returns [20..39], etc.
    /// </summary>
    public async Task<(List<string> Items, long Total)> GetFeedPageAsync(
        string feedKey, int page = 1, int pageSize = 20)
    {
        var start = (page - 1) * pageSize;
        var stop = start + pageSize - 1;

        var batch = _db.CreateBatch();
        var rangeTask = batch.ListRangeAsync(feedKey, start, stop);
        var lengthTask = batch.ListLengthAsync(feedKey);
        batch.Execute();

        await Task.WhenAll(rangeTask, lengthTask);

        var items = rangeTask.Result
            .Where(r => !r.IsNull)
            .Select(r => r.ToString())
            .ToList();

        return (items, lengthTask.Result);
    }
```

### Pattern 4 — Batch LRANGE with Pipelining

```csharp
public class BatchListReader
{
    private readonly IDatabase _db;

    public BatchListReader(IDatabase db) => _db = db;

    /// <summary>
    /// Read multiple ranges from a list in a single round trip.
    /// Uses batching to pipeline all LRANGE commands.
    /// </summary>
    public async Task<List<RedisValue[]>> ReadMultipleRangesAsync(
        string listKey, params (long Start, long Stop)[] ranges)
    {
        if (ranges.Length == 0)
            return new List<RedisValue[]>();

        var batch = _db.CreateBatch();
        var tasks = ranges.Select(r => batch.ListRangeAsync(listKey, r.Start, r.Stop)).ToArray();
        batch.Execute();

        await Task.WhenAll(tasks);
        return tasks.Select(t => t.Result).ToList();
    }

    /// <summary>
    /// Read the first N, middle N, and last N elements in one round trip.
    /// </summary>
    public async Task<(RedisValue[] Head, RedisValue[] Middle, RedisValue[] Tail)> ReadHeadMiddleTailAsync(
        string listKey, int headCount = 5, int middleCount = 5, int tailCount = 5)
    {
        var length = await _db.ListLengthAsync(listKey);
        if (length == 0) return (Array.Empty<RedisValue>(), Array.Empty<RedisValue>(), Array.Empty<RedisValue>());

        var middleStart = length / 2 - middleCount / 2;
        var middleStop = middleStart + middleCount - 1;

        var batch = _db.CreateBatch();
        var headTask = batch.ListRangeAsync(listKey, 0, headCount - 1);
        var middleTask = batch.ListRangeAsync(listKey, middleStart, middleStop);
        var tailTask = batch.ListRangeAsync(listKey, length - tailCount, length - 1);
        batch.Execute();

        await Task.WhenAll(headTask, middleTask, tailTask);
        return (headTask.Result, middleTask.Result, tailTask.Result);
    }
}
```

## 5. Gotchas — Production Pitfalls

### Pitfall 1 — LRANGE 0 -1 on Million-Element Lists

**Pitfall:** The developer loads an entire list into memory without considering the size:

```csharp
// ❌ Loads ALL elements — 100K items × 1KB each = 100MB response
var allItems = await _db.ListRangeAsync("logs:audit", 0, -1);
```

**Symptom:** The SE.Redis call takes 2+ seconds because Redis must serialize and transmit all elements. The `ConnectionMultiplexer`'s output buffer grows to 100MB+. The Redis server logs `Client connection lost due to output buffer limit exceeded`. The application receives an incomplete or failed response.

**Fix:** Always paginate with known page sizes:

```csharp
// ✅ Paginate — max 100 items at a time
var page = await _db.ListRangeAsync("logs:audit", 0, 99);
```

**Cost of not fixing:** Redis disconnects the client. The application's next Redis call throws `RedisConnectionException` because the multiplexer enters a faulted state. All Redis-dependent features fail until the multiplexer reconnects.

---

### Pitfall 2 — LINDEX on Large Lists Without Understanding O(N)

**Pitfall:** The developer uses LINDEX repeatedly in a loop on a large list:

```csharp
// ❌ O(N) per iteration → total O(N²)
for (int i = 0; i < 100000; i++)
{
    var item = await _db.ListGetByIndexAsync("large", i);
    Process(item);
}
```

**Symptom:** The first few iterations are fast (index 0 is O(1)), but each subsequent iteration gets slower as the traversal distance increases. By index 50000, each LINDEX takes ~2.5ms. The total loop takes 125 seconds of Redis CPU time, blocking all other Redis operations.

**Fix:** Use LRANGE to fetch the entire segment once:

```csharp
// ✅ Single LRANGE — O(N) once, not O(N²)
var allItems = await _db.ListRangeAsync("large", 0, -1);
foreach (var item in allItems)
{
    Process(item);
}
```

**Cost of not fixing:** Redis CPU at 100% for 2+ minutes. All other clients experience timeout exceptions. The application is effectively down for the duration of the loop. SE.Redis may trip `SyncTimeout` (default 3s) and throw `TimeoutException`.

---

### Pitfall 3 — LSET Without Validating Index Causes Runtime Error

**Pitfall:** The developer assumes an index is valid without checking:

```csharp
// ❌ LSET throws if index is out of range
await _db.ListSetByIndexAsync("queue", 100, "new-value");
```

**Symptom:** `RedisServerException: ERR index out of range` at runtime. If the method is inside a transaction (`MULTI/EXEC`), the entire transaction is discarded — all operations in the transaction are rolled back.

**Fix:** Validate index with LLEN first, or use a try-catch:

```csharp
// ✅ Validate with LLEN
var length = await _db.ListLengthAsync("queue");
if (100 < length)
{
    await _db.ListSetByIndexAsync("queue", 100, "new-value");
}
else
{
    // Handle gracefully — log, extend, or skip
}

// ✅ Or try-catch for atomicity
try
{
    await _db.ListSetByIndexAsync("queue", 100, "new-value");
}
catch (RedisServerException ex) when (ex.Message.Contains("index out of range"))
{
    // Handle out of range
}
```

**Cost of not fixing:** If the LSET is in a transaction with other operations, an index-out-of-range error aborts the entire transaction. For example, a MULTI block that does LSET + LPUSH + LREM — if LSET fails, the LPUSH and LREM are also rolled back. The caller gets `EXECABORT` and must retry the entire transaction.

---

### Pitfall 4 — Negative Index Confusion with LINDEX

**Pitfall:** The developer uses LINDEX with negative indices but does not understand the wrap behavior:

```csharp
// ❌ LINDEX -0 is same as LINDEX 0, not the last element
var notTheLast = await _db.ListGetByIndexAsync("list", -0); // Returns FIRST element, not last

// ❌ LINDEX with index == -length returns nil (out of range for position -length)
var list = new[] { "a", "b", "c" }; // length = 3
var outOfRange = await _db.ListGetByIndexAsync("list", -4); // -4 normalized to 3-4 = -1 → nil
```

**Symptom:** Off-by-one errors in pagination. Page 1 gets elements [0..19], page 2 gets [20..39], but negative-index math for "last page" wraps incorrectly, returning an empty range or elements from the wrong section.

**Fix:** Always normalize negative indices away from user-facing code. Use LLEN to convert to positive indices before using them:

```csharp
// ✅ Normalize to positive indices
var length = await _db.ListLengthAsync("list");
var positiveIndex = index >= 0 ? index : length + index;
if (positiveIndex < 0 || positiveIndex >= length)
    throw new ArgumentOutOfRangeException(nameof(index));

var result = await _db.ListGetByIndexAsync("list", index); // Redis handles negative, but caller logic uses positive
```

**Cost of not fixing:** Pagination shows duplicate items on page boundaries (because the last element of page N is also the first element of page N+1 due to off-by-one). Or the last page returns an empty result, making the user think there are no more items.

---

### Pitfall 5 — LRANGE Returns Elements in List Order, Not Insertion Order

**Pitfall:** The developer assumes LRANGE returns elements in the order they were inserted (oldest first):

```csharp
// Producer pushes to queue
await _db.ListRightPushAsync("queue", "first");
await _db.ListRightPushAsync("queue", "second");
await _db.ListRightPushAsync("queue", "third");

// Consumer reads all
var items = await _db.ListRangeAsync("queue", 0, -1);
// items[0] is "first", items[1] is "second", items[2] is "third" ✅ — correct for RPUSH
```

But if the producer used LPUSH:

```csharp
await _db.ListLeftPushAsync("stack", "first");
await _db.ListLeftPushAsync("stack", "second");
await _db.ListLeftPushAsync("stack", "third");

var items = await _db.ListRangeAsync("stack", 0, -1);
// items[0] is "third", items[1] is "second", items[2] is "first" — most recent first
```

**Symptom:** The consumer interprets LRANGE results as chronological when the list was populated with LPUSH (LIFO), getting items in reverse-chronological order. This causes a "most recent action processed first" behavior that may violate business ordering requirements.

**Fix:** Document the access pattern. If chronological order is required, always RPUSH (tail). If reverse-chronological is required, LPUSH (head). LRANGE 0 -1 always returns from head (index 0) to tail (index -1), regardless of access pattern:

```csharp
/// <summary>
/// Returns items in chronological order (oldest first).
/// REQUIRES: producer uses RPUSH (append to tail).
/// </summary>
public async Task<RedisValue[]> GetChronologicalAsync(string queueKey)
{
    return await _db.ListRangeAsync(queueKey, 0, -1);
}

/// <summary>
/// Returns items in reverse-chronological order (newest first).
/// REQUIRES: producer uses LPUSH (insert at head).
/// </summary>
public async Task<RedisValue[]> GetReverseChronologicalAsync(string feedKey)
{
    return await _db.ListRangeAsync(feedKey, 0, -1);
}
```

**Cost of not fixing:** A job processing system that processes jobs in reverse order of submission — the newest job runs first, the oldest job runs last. If job ordering matters (e.g., "cancel order" must run after "place order"), the system produces incorrect state.

---

### Pitfall 6 — LLEN on Non-Existent Key Returns 0 (Not Error)

**Pitfall:** The developer uses LLEN to check if a list exists and interprets 0 as "list exists but is empty":

```csharp
// ❌ Cannot distinguish between "list exists, empty" and "list does not exist"
var length = await _db.ListLengthAsync("key");
if (length == 0)
{
    // Is this a new key that was never created?
    // Or an existing list that was drained?
}
```

**Symptom:** A monitoring system that alerts when queue depth is 0. After the queue is drained and the list key is deleted by Redis (when the last element is popped, the key is removed), LLEN still returns 0. The monitoring system shows "0 items, all good" when the reality is "key does not exist, no queue processing is happening."

**Fix:** Use EXISTS to distinguish:

```csharp
// ✅ Distinguish between "empty" and "does not exist"
var exists = await _db.KeyExistsAsync("queue");
var length = await _db.ListLengthAsync("queue"); // 0 if not exists too

if (!exists)
    _logger.LogWarning("Queue key does not exist — has the consumer crashed?");
else if (length == 0)
    _logger.LogInformation("Queue is empty but key exists — waiting for new items");
else
    _logger.LogInformation("Queue has {Length} items", length);
```

**Cost of not fixing:** Silent queue failure. The producer has a bug and is not creating jobs. LLEN returns 0, the monitoring dashboard shows green, but no work is being done. The incident is only discovered when a customer complains 8 hours later.

The middle-100 range takes ~3ms despite returning only 100 elements, because Redis must traverse ~50K elements to reach the start offset. The last-100 is fast (0.3ms) because it traverses backwards from the tail pointer — only ~100 elements back.
- **LINDEX head/tail vs middle:** Head and tail are essentially O(1) via the quicklist pointers. Middle (50,000) requires traversing 50K elements — ~2.5ms of Redis CPU time.
- **LLEN:** Always O(1) — the fastest operation at ~0.03ms.
- **LSET:** Same pattern as LINDEX for traversal (O(N) to find position, O(1) to mutate). LSET middle is ~2.6ms — 0.1ms more than LINDEX middle due to the ziplist update overhead.

### When LRANGE/LINDEX Become Expensive

| Scenario | Time Complexity | Real Cost (100K list, localhost) |
|---|---|---|
| LRANGE 0 -1 (full list) | O(N) | ~45ms, ~6MB network |
| LRANGE first N (N small) | O(N) ~ O(N traversal) | ~0.3ms for N=100, if at head |
| LRANGE after offset M (N small) | O(M + N) | ~3ms for M=50000, N=100 |
| LINDEX at position P | O(P) or O(L-P) | ~2.5ms for P=50000 |
| LLEN | O(1) | ~0.03ms |
| LSET at position P | O(P) or O(L-P) | ~2.6ms for P=50000 |

### Memory Considerations

- LRANGE creates a complete serialized copy of every returned element in Redis's output buffer. For a list with 100K elements of 1KB each, LRANGE 0 -1 allocates ~100MB in the output buffer.
- Redis's `client-output-buffer-limit normal` default is 256MB / 64MB / 60s. If the output buffer exceeds the soft limit (64MB) for 60s, the connection is closed.
- For LINDEX, only the single element is serialized — memory per call is bounded by the element size.

### Spoken Answers

**Q: Why is LINDEX O(N) while LLEN is O(1)?**

> **Average answer:** "LINDEX has to find the element by walking through the list, and LLEN just reads a stored number."
>
> **Great answer:** "Redis lists are implemented as quicklists — a doubly-linked list of ziplist nodes. LINDEX must traverse the linked list from whichever end is closer to the target index, counting elements as it goes. For a list with 1M elements, LINDEX at position 500K traverses ~500K elements — O(N) in the index position. LLEN, by contrast, reads a simple `unsigned long` counter stored in the list object header. Every append (RPUSH), prepend (LPUSH), pop (LPOP/RPOP), insert (LINSERT), and remove (LREM) updates this counter atomically. Reading it is a memory dereference — no traversal needed. This is the most important performance characteristic of Redis lists to understand: edge operations are O(1), counter reads are O(1), but indexed access is O(N). If you need O(log N) indexed access, use a sorted set."

**Q: What does LRANGE 0 -1 return and when should you avoid it?**

> **Average answer:** "It returns all elements in the list. You should not use it on large lists because it's slow."
>
> **Great answer:** "LRANGE 0 -1 returns every element from the head (index 0) through the tail (index -1). Redis traverses the entire quicklist from head to tail, serializing every element into the output buffer — O(N) time and O(N) memory where N is the list length. For a list with 100K 1KB elements, this allocates ~100MB in Redis's output buffer and takes ~45ms of Redis CPU time. The network transfer for the response is another ~100MB. I avoid LRANGE 0 -1 on any list larger than ~10K elements. Instead, I use pagination with LRANGE offset stop (e.g., LRANGE 0 99 for the first page) to control both CPU and network cost. For monitoring dashboards that show queue contents, I use LRANGE 0 19 for a preview and display the total count from LLEN. If I absolutely must process all elements, I use a Lua script or client-side iteration with batched LRANGE calls (e.g., 500 elements per batch) to keep individual response sizes manageable."

**Q: How does StackExchange.Redis expose LRANGE methods?**

> **Average answer:** "There's a method called ListRangeAsync."
>
> **Great answer:** "StackExchange.Redis provides `ListRangeAsync(RedisKey key, long start = 0, long stop = -1, CommandFlags flags = CommandFlags.None)` which returns `Task<RedisValue[]>`. The start and stop parameters map directly to Redis's LRANGE start and stop — they support negative indices. The method returns `RedisValue[]`, where each element is a `RedisValue` struct. For deserialization, I map `RedisValue[]` to my typed objects: `results.Select(r => JsonSerializer.Deserialize<T>(r.ToString())).ToList()`. The method is asynchronous and supports the SE.Redis multiplexer architecture — it uses one of the multiplexer's connections to send the LRANGE command and awaits the response without blocking. For high-throughput scenarios, I batch multiple LRANGE calls using `IDatabase.CreateBatch()` to pipeline them in a single round trip."

### Interview Trigger

The interviewer asks: "How would you inspect the contents of a Redis list without modifying it?" The senior candidate lists LRANGE for ranges, LINDEX for single-element access, and LLEN for length. They immediately qualify with "but I limit LRANGE to small ranges on large lists because of the O(N) traversal and output buffer cost." The follow-up is: "How do you handle a list where you need to read all 500K elements?" The separating factor is whether the candidate knows the quicklist traversal mechanics and can describe the tradeoff between a single LRANGE (45ms, 100MB buffer) vs batched LRANGE (multiple 2ms calls, 5MB per call) vs client-side iteration using LINDEX (terrible idea — O(N²)).

## 8. Decision Framework — When & Why

### When to Apply LRANGE, LINDEX, LLEN, LSET

```mermaid

    B -->|Read| C{Batch or single?}
    B -->|Mutate| D{Update existing element?}

    C -->|Single element| F{At known position?}
    
    E -->|Yes| G[LRANGE — O(S+N)]
    E -->|No — need to find| H[Consider Sorted Set<br/>or scan client-side]
    
    F -->|Yes — position known| I[LINDEX — O(N) for middle,<br/>O(1) for edges]
    F -->|No — need position| J{Only need length?}
    
    J -->|Yes| K[LLEN — O(1)]
    J -->|No| L[LRANGE to search<br/>client-side or use<br/>different data structure]
    
    D -->|Yes — position known| M{Position is at edge?}
    D -->|No — new element| N[Use LPUSH/RPUSH]
    
    M -->|Edge (0 or -1)| O[LSET — effectively O(1)]
    M -->|Middle| P[LSET — O(N) traversal<br/>+ O(1) mutation<br/>Consider if O(N) is acceptable]
    
    G --> Q[Output: RedisValue[]]
    I --> R[Output: RedisValue]
    K --> S[Output: long]
    M --> T[Output: OK]
```

### Application Checklist

- [ ] List size is known — LRANGE 0 -1 is only used on lists < 10K elements
- [ ] LINDEX is not called in a loop — one LRANGE is preferred over N LINDEX calls
- [ ] LLEN is used for queue depth monitoring with EXISTS to distinguish "empty" from "not exist"
- [ ] LSET index validity is checked with LLEN before calling, or wrapped in try-catch
- [ ] Negative indices are normalized to positive in caller logic (not passed raw from user input)
- [ ] Batch reading uses `IDatabase.CreateBatch()` for pipeline efficiency
- [ ] Element serialization is consistent — all LRANGE results are deserialized with the same format
- [ ] Output buffer limits are configured if LRANGE may return large responses
- [ ] Monitoring alerts on list length thresholds use LLEN

### Tradeoff Summary

| What You Gain | What You Pay |
|---|---|
| LRANGE — slice access without popping | O(N) traversal + O(N) response memory for full range |
| LINDEX — random access by position | O(N) traversal for middle positions — not truly random-access |
| LLEN — instant length query | O(1) but cannot distinguish non-existent from empty without EXISTS |
| LSET — in-place mutation without length change | O(N) traversal + error if index out of range |
| All operations are atomic on single-threaded Redis | Cannot combine with other operations atomically — use MULTI for transaction |

### Scale Thresholds

- **LRANGE relevant when:** List exceeds ~10K elements — LRANGE 0 -1 takes 5ms+ and returns 500KB+ response.
- **LRANGE critical when:** List exceeds ~100K elements — LRANGE 0 -1 takes 50ms+ and may trip output buffer limits. Always paginate.
- **LINDEX critical when:** List exceeds ~50K elements and LINDEX is called at middle positions — each call takes 1ms+ and in a loop creates O(N²) cost.
- **LLEN always O(1):** No scale threshold — LLEN on a 10-element list and a 10M-element list both take ~0.03ms.
- **LSET critical when:** List exceeds ~100K elements and LSET is called at middle positions — each call takes 2.5ms+. At 100 calls/second, this uses 250ms of Redis CPU per second.

## 9. Self-Check — Review & Challenges

### Conceptual Questions

1. What does LRANGE return when called on a non-existent key — error or empty array?
2. How does Redis compute the time complexity of LINDEX for index 0 vs index 500,000 on a 1M-element list?
3. Why is LLEN O(1) while LINDEX is O(N)? What does the quicklist store differently?
4. What happens when you call LSET with an index equal to the list length (e.g., list of 5 elements, LSET 5)?
5. How do negative indices work in LRANGE and LINDEX? What does -1 mean? What does -0 mean?
6. What is the SE.Redis method for LRANGE, and what does it return — array, list, or enumerable?
7. How would you paginate through a Redis list of 500K elements without loading all at once?
8. What is the difference between LRANGE and LINDEX in terms of time complexity for reading element at position 0?
9. How do you handle the case where LSET returns an index-out-of-range error in SE.Redis?
10. What happens to Redis's output buffer when LRANGE 0 -1 is called on a 1GB list?

<details>
<summary>Answers</summary>

1. LRANGE on a non-existent key returns an empty array (zero elements). It does not return an error. The caller receives `RedisValue[]` of length 0.

2. LINDEX 0 is O(1) — the quicklist maintains a direct pointer to the head node, so no traversal is needed. LINDEX 500,000 is O(N) = O(500,000) — Redis must traverse from the nearest end (head or tail) counting 500K elements through quicklist nodes and ziplist entries.

3. LLEN reads a counter field stored directly in the list object header (`unsigned long length`). This counter is updated by every push/pop/trim/insert/remove operation. LINDEX must traverse the linked list because there is no index-to-node mapping — the quicklist only knows its head and tail pointers.

4. LSET with an index equal to the list length (or any index ≥ length) returns `ERR index out of range`. LSET does NOT auto-extend the list — it only mutates existing elements. To add elements, use LPUSH, RPUSH, LINSERT, or LSET in combination with RPUSH for missing positions.

5. Negative indices count from the tail: -1 is the last element, -2 is second-to-last, etc. `LINDEX key -0` is treated as 0 (the first element, because -0 === 0 in Redis's parsing). For LINDEX, if the normalized index (length + negative) is still negative, it returns nil (out of range).

6. StackExchange.Redis: `Task<RedisValue[]> ListRangeAsync(RedisKey key, long start = 0, long stop = -1, CommandFlags flags = CommandFlags.None)`. It returns a `RedisValue[]` array — all elements in the requested range are materialized into the array before the task completes.

7. Use LRANGE with offset and limit in a loop:
   ```csharp
   var pageSize = 500;
   for (int offset = 0; ; offset += pageSize)
   {
       var batch = await _db.ListRangeAsync(key, offset, offset + pageSize - 1);
       if (batch.Length == 0) break;
       ProcessBatch(batch);
   }
   ```

8. Both LRANGE and LINDEX for position 0 are effectively O(1) with respect to traversal distance. LRANGE 0 0 returns a single-element array in ~0.05ms. LINDEX 0 returns a single value in ~0.05ms. However, LRANGE has slightly more overhead because it creates an array response vs LINDEX's single-value response.

9. Catch `RedisServerException` and check the message for "index out of range". Alternatively, validate with LLEN before calling LSET:
   ```csharp
   var length = await _db.ListLengthAsync(key);
   if (index >= 0 && index < length)
       await _db.ListSetByIndexAsync(key, index, value);
   ```

10. Redis's output buffer grows to contain the full serialized list. With `client-output-buffer-limit normal 256mb 64mb 60`, if the response exceeds 64MB for 60 seconds, Redis closes the connection. For a 1GB list, the buffer would quickly exceed the hard limit (256MB), causing an immediate disconnect. The client receives an incomplete response. The solution is to never call LRANGE 0 -1 on a 1GB list — always paginate.

</details>

**Challenge 1 — Paginated Queue Monitor**

You maintain a Redis list `queue:orders` that contains 200K order-processing jobs. Implement a monitoring endpoint that returns page 42 of the queue (items 2050–2099, 0-indexed) along with total count and pagination metadata. Show both the Redis CLI command and SE.Redis C#.

<details>
<summary>Solution</summary>

**Redis CLI:**

```bash
127.0.0.1:6379> LLEN queue:orders
(integer) 200000
127.0.0.1:6379> LRANGE queue:orders 2050 2099
1) "{\"orderId\":2051,\"status\":\"pending\"}"
2) "{\"orderId\":2052,\"status\":\"processing\"}"
...
50) "{\"orderId\":2100,\"status\":\"pending\"}"
```

**SE.Redis C#:**

```csharp
public class PaginatedQueueMonitor
{
    private readonly IDatabase _db;
    private const int PageSize = 50;

    public PaginatedQueueMonitor(IDatabase db) => _db = db;

    public async Task<QueuePage> GetPageAsync(string queueKey, int pageNumber)
    {
        var start = (pageNumber - 1) * PageSize;
        var stop = start + PageSize - 1;

        // Pipeline LLEN and LRANGE in one round trip
        var batch = _db.CreateBatch();
        var lengthTask = batch.ListLengthAsync(queueKey);
        var rangeTask = batch.ListRangeAsync(queueKey, start, stop);
        batch.Execute();

        await Task.WhenAll(lengthTask, rangeTask);

        var totalItems = lengthTask.Result;
        var items = rangeTask.Result
            .Where(r => !r.IsNull)
            .Select(r => r.ToString())
            .ToList();

        var totalPages = (int)Math.Ceiling((double)totalItems / PageSize);

        return new QueuePage
        {
            PageNumber = pageNumber,
            PageSize = PageSize,
            TotalItems = totalItems,
            TotalPages = totalPages,
            Items = items,
            HasNext = pageNumber < totalPages,
            HasPrevious = pageNumber > 1
        };
    }

    public class QueuePage
    {
        public int PageNumber { get; set; }
        public int PageSize { get; set; }
        public long TotalItems { get; set; }
        public int TotalPages { get; set; }
        public List<string> Items { get; set; } = new();
        public bool HasNext { get; set; }
        public bool HasPrevious { get; set; }
    }
}
```

**Cost:** 2 commands in 1 round trip — ~0.5ms total on localhost. LRANGE traversal from start=2050 traverses ~2050 elements (O(2050)) then returns 50 elements.

</details>

**Challenge 2 — Fix the LINDEX Loop**

```csharp
// This method is called frequently to process the first 1000 items in a queue.
// The queue has 100K items. The method takes 3 seconds to run.
public async Task ProcessFirstThousandAsync(string queueKey)
{
    for (int i = 0; i < 1000; i++)
    {
        var item = await _db.ListGetByIndexAsync(queueKey, i);
        if (item.IsNull) break;
        await ProcessItemAsync(item.ToString());
    }
}
```

Why is it slow? Fix it.

<details>
<summary>Solution</summary>

**Root cause:** The loop calls LINDEX 1000 times — each is a separate round trip to Redis. LINDEX at i=0 is O(1) but LINDEX at i=999 traverses 999 elements. Total traversed elements = sum(0..999) ≈ 500K — half a million element traversals. With 1000 round trips at ~0.4ms each, the network overhead alone is 400ms. Redis CPU adds the traversal cost. Total = ~3 seconds.

**Fix — use a single LRANGE call:**

```csharp
public async Task ProcessFirstThousandAsync(string queueKey)
{
    // Single LRANGE returns all 1000 elements in one round trip
    var items = await _db.ListRangeAsync(queueKey, 0, 999);

    foreach (var item in items)
    {
        if (item.IsNull) break;
        await ProcessItemAsync(item.ToString());
    }
}
```

**After fix:** 1 round trip instead of 1000. LRANGE 0 999 traverses 1000 elements (not 500K) and returns them all. Total time: ~0.5ms (network) + 0.2ms (traversal) = ~0.7ms. Improvement: 4,000x faster.

**If LRANGE is not possible (need to process and pop):** Use LPOP with count (Redis 6.2+):

```csharp
var items = await _db.ListLeftPopAsync(queueKey, 1000);
```

**Challenge 3 — Safe LSET with Validation**

Implement a method that updates the retry count of a job at a given index in a processing list. Ensure no exception if the index is out of range. Show both the safe way (with validation) and the atomic way (with Lua), and explain the tradeoff.

<details>
<summary>Solution</summary>

**Safe way — validate with LLEN first:**

```csharp
public async Task<bool> TryUpdateRetryCountAsync(string listKey, long index, int newRetryCount)
{
    var length = await _db.ListLengthAsync(listKey);
    if (index < 0)
        index = length + index;
    if (index < 0 || index >= length)
        return false; // Index out of range — no exception

    var current = await _db.ListGetByIndexAsync(listKey, index);
    if (current.IsNull)
        return false;

    // Deserialize, update, serialize back
    var job = JsonDocument.Parse(current.ToString());
    var updated = JsonSerializer.Serialize(new
    {
        Id = job.RootElement.GetProperty("Id").GetString(),
        Data = job.RootElement.GetProperty("Data").GetString(),
        Retries = newRetryCount
    });

    await _db.ListSetByIndexAsync(listKey, index, updated);
    return true;
}
```

**Atomic way — Lua script (one round trip, all server-side):**

```csharp
public async Task<bool> TryUpdateRetryCountAtomicAsync(string listKey, long index, int newRetryCount)
{
    var lua = @"
        local length = redis.call('LLEN', KEYS[1])
        local idx = tonumber(ARGV[1])
        if idx < 0 then
            idx = length + idx
        end
        if idx < 0 or idx >= length then
            return 0
        end
        local current = redis.call('LINDEX', KEYS[1], idx)
        if not current then
            return 0
        end
        local job = cjson.decode(current)
        job['Retries'] = tonumber(ARGV[2])
        redis.call('LSET', KEYS[1], idx, cjson.encode(job))
        return 1
    ";

    var result = await _db.ScriptEvaluateAsync(lua,
        new RedisKey[] { listKey },
        new RedisValue[] { index, newRetryCount });

    return (int)result == 1;
}
```

**Tradeoff:** The safe way uses 3 round trips (LLEN, LINDEX, LSET) and is subject to race conditions (another consumer modifies the list between LLEN and LSET). The Lua script uses 1 round trip and is atomic — no other command can execute between LLEN, LINDEX, and LSET. For high-contention queues, Lua is preferred. For low-contention inspection tools, the safe way is simpler.

</details>

**Challenge 4 — Diagnose the Output Buffer Disconnect**

Your monitoring dashboard calls `LRANGE queue:events 0 -1` to display all items. The queue has 50K items of ~2KB each. The dashboard works on small queues but disconnects when the queue exceeds 30K items. The Redis log shows: `Client id=42 addr=127.0.0.1:54321 addr=10.0.0.1:6379 closed due to output buffer limit`. Identify and fix.

<details>
<summary>Solution</summary>

**Root cause:** LRANGE 0 -1 on a 50K-item queue with 2KB elements produces a 100MB response. Redis's default `client-output-buffer-limit normal` is 256MB hard limit, 64MB soft limit for 60 seconds. At ~50K items × 2KB = ~100MB, the response exceeds the 64MB soft limit for more than 60 seconds (the serialization takes time), so Redis closes the connection.

**Detection:**
```bash
127.0.0.1:6379> CONFIG GET client-output-buffer-limit
1) "client-output-buffer-limit"
2) "normal 268435456 67108864 60 replica 268435456 16777216 60 pubsub 33554432 8388608 60"
```

**Fix — paginate the LRANGE call:**

```csharp
public async Task<List<string>> GetDashboardItemsSafeAsync(string listKey, int maxItems = 100)
{
    var batch = _db.CreateBatch();
    var countTask = batch.ListLengthAsync(listKey);
    var itemsTask = batch.ListRangeAsync(listKey, 0, maxItems - 1);
    batch.Execute();

    await Task.WhenAll(countTask, itemsTask);
    return itemsTask.Result
        .Where(r => !r.IsNull)
        .Select(r => r.ToString())
        .ToList();
}
```

**Or increase the output buffer limit if pagination is not an option:**

```bash
127.0.0.1:6379> CONFIG SET client-output-buffer-limit "normal 512mb 128mb 120"
OK
```

**Prevention:** Never use LRANGE 0 -1 in application code. Always paginate. The monitoring dashboard should use `LLEN` for the count and `LRANGE 0 99` for a preview.

**Challenge 5 — Consistent Pagination Across Producers**

Your team uses both LPUSH and RPUSH in different services to add items to the same list. The pagination code uses LRANGE start stop and assumes elements are in chronological order. The QA team reports that page 2 sometimes shows items that also appear on page 1. Diagnose and fix.

<details>
<summary>Solution</summary>

**Root cause:** Mixing LPUSH and RPUSH on the same list creates unpredictable ordering. A producer that LPUSHes inserts at index 0, shifting all existing items right by 1. A producer that RPUSHes appends at the tail. If producer A uses RPUSH (chronological) and producer B uses LPUSH (reverse-chronological), the list has a schizophrenic ordering — some items are at the correct chronological position, but items from producer B appear in front of them.

When items are inserted at the head via LPUSH, existing items shift right. An item that was at index 49 (last item of page 1) becomes index 50 (first item of page 2) after an LPUSH. The next call to LRANGE 0 49 shows the new item and items that were previously at indices 0–48, but the item that was at index 49 is now at index 50 — it appears on both pages from the consumer's perspective.

**Fix — standardize on one push direction:**

```csharp
// All producers must use RPUSH for chronological order
public async Task EnqueueAsync(string queueKey, string item)
{
    await _db.ListRightPushAsync(queueKey, item);
}

// Pagination assumes RPUSH ordering
public async Task<RedisValue[]> GetPageAsync(string queueKey, int page, int pageSize)
{
    var start = (page - 1) * pageSize;
    var stop = start + pageSize - 1;
    return await _db.ListRangeAsync(queueKey, start, stop);
}
```

**If LPUSH must be used (recent-activity-style):**
```csharp
// LPUSH — newest first
// Paginate with LPUSH: newest is at index 0, page 1 gets items 0-19 (most recent)
public async Task<RedisValue[]> GetRecentPage(string queueKey, int page, int pageSize)

    var stop = start + pageSize - 1;
```
