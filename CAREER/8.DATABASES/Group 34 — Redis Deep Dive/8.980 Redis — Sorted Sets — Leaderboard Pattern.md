# Redis — Sorted Sets — Leaderboard Pattern

---

## Section 1 — Overview

The leaderboard pattern is one of the most common and powerful use cases for Redis sorted sets. It leverages the natural score-based ordering of sorted sets to build real-time ranking systems for games, competitions, engagement metrics, and any scenario where you need to rank entities by a numeric value.

A leaderboard built on Redis sorted sets provides:

- **Real-time updates** — scores change instantly with ZINCRBY
- **Top-N queries** — ZREVRANGE fetches the highest-ranked members
- **Rank lookups** — ZREVRANK answers "where do I stand?"
- **Pagination** — ZREVRANGE with offset/limit for "load more"
- **Time-window rotation** — daily/weekly keys with TTL
- **Aggregation** — ZUNIONSTORE to combine multiple leaderboards

This note covers the complete leaderboard pattern: from a simple top-N list to a production-grade service with tiebreakers, time windows, caching, and error handling.

### 1.1 — Core Concept

```
Player Scores             Redis Sorted Set              Leaderboard Display
                        "leaderboard:global"
  playerA: 2500         ┌─────────────────────┐        ┌─────────────────────┐
  playerB: 1800   ──►   │ playerA: 2500       │   ──►  │ 1. playerA   2500  │
  playerC: 3200         │ playerC: 3200       │        │ 2. playerC   3200  │
  playerD: 1500         │ playerB: 1800       │        │ 3. playerB   1800  │
  playerE: 2900         │ playerE: 2900       │        │ 4. playerE   2900  │
                        │ playerD: 1500       │        │ 5. playerD   1500  │
                        └─────────────────────┘        └─────────────────────┘
                          Ordered by score               Sorted by rank
                          (skiplist + hash)              (ZREVRANGE: 0 4)
```

### 1.2 — When to Use

| Scenario | Redis Key Pattern | Update Pattern | Query Pattern |
|---|---|---|---|
| Daily game leaderboard | `leaderboard:daily:2024-01-15` | ZINCRBY on score | ZREVRANGE top 100 |
| Weekly competition | `leaderboard:weekly:2024-W03` | ZINCRBY on score | ZREVRANGE top 50 |
| All-time high scores | `leaderboard:alltime` | ZADD (overwrite on new high) | ZREVRANGE top 10 |
| Achievement progress | `achievement:kills:playerA` | ZINCRBY increment | ZSCORE check threshold |
| Tournament bracket | `tournament:round1:groupA` | ZADD single scores | ZRANGE all by rank |

|---|---|
| ZINCRBY | Atomically update a player's score |

| ZREMRANGEBYRANK | Trim leaderboard to a max size |

| ZINTERSTORE | Find players common across leaderboards |

### 2.4 — Aggregation with ZUNIONSTORE

Combine daily leaderboards into a weekly leaderboard:

```bash
# Sum scores across 7 daily leaderboards
ZUNIONSTORE leaderboard:weekly:2024-W03 7
    leaderboard:daily:2024-01-15
    leaderboard:daily:2024-01-16
    leaderboard:daily:2024-01-17
    leaderboard:daily:2024-01-18
    leaderboard:daily:2024-01-19
    leaderboard:daily:2024-01-20
    leaderboard:daily:2024-01-21
    AGGREGATE SUM
```

Or with WEIGHTS for partial credit:
```bash
# Weekend days count double
ZUNIONSTORE leaderboard:weekly:2024-W03 7
    leaderboard:daily:2024-01-15   leaderboard:daily:2024-01-16
    leaderboard:daily:2024-01-17   leaderboard:daily:2024-01-18
    leaderboard:daily:2024-01-19   leaderboard:daily:2024-01-20
    leaderboard:daily:2024-01-21
    WEIGHTS 1 1 1 1 1 2 2
    AGGREGATE SUM
```

### 2.5 — Score Tiebreaker Strategies

When two players have the same score, Redis sorts them lexicographically by member name. For controlled tiebreaking, use one of these strategies:

**Strategy 1: Composite Score (score + tiebreaker in a single double)**

```bash
# Encode: score * 1_000_000 + (999_999 - timestamp_suffix)
# This gives higher rank to earlier submissions at same score level
ZADD leaderboard:global 1500999998 "playerA"   # score 1500, submitted late
ZADD leaderboard:global 1500000001 "playerB"   # score 1500, submitted early
# playerB ranks higher (lower tiebreaker value = submitted earlier)
```

**Strategy 2: Lexicographic with prefixed member names**

```bash
# Prefix member name with tiebreaker string
# Lower lexicographic = higher rank at same score
ZADD leaderboard:global 1500 "0001:playerA"   # first submission
ZADD leaderboard:global 1500 "0002:playerB"   # second submission
# playerA (0001...) ranks higher than playerB (0002...)
```

**Strategy 3: Separate tiebreaker sorted set (not recommended)**

```bash
# Two sorted sets: one for score, one for timestamp
ZADD scores 1500 "playerA"
ZADD scores 1500 "playerB"
ZADD tiebreakers 1704067200 "playerA"   # timestamp as score
ZADD tiebreakers 1704067300 "playerB"
# Query both and merge on client side (O(N) and complex)
```

Strategy 1 (composite score) is the most common and recommended approach.

### 3.3 — Error Handling Foundation

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;
using Microsoft.Extensions.Logging;

public class RedisErrorHandler
{
    private readonly ILogger<RedisErrorHandler> _logger;

    public RedisErrorHandler(ILogger<RedisErrorHandler> logger)
    {
        _logger = logger;
    }

    public async Task<T> ExecuteWithRetryAsync<T>(
        Func<Task<T>> operation,
        string operationName,
        int maxRetries = 3)
    {
        for (int attempt = 1; attempt <= maxRetries; attempt++)
        {
            try
            {
                return await operation();
            }
            catch (RedisConnectionException ex)
            {
                _logger.LogWarning(ex,
                    "Redis connection error on {Op} (attempt {Attempt}/{Max})",
                    operationName, attempt, maxRetries);

                if (attempt == maxRetries)
                {
                    _logger.LogError(ex,
                        "Failed {Op} after {Max} attempts",
                        operationName, maxRetries);
                    throw;
                }

                await Task.Delay(TimeSpan.FromMilliseconds(100 * attempt));
            }
            catch (RedisTimeoutException ex)
            {
                _logger.LogWarning(ex,
                    "Redis timeout on {Op} (attempt {Attempt}/{Max})",
                    operationName, attempt, maxRetries);

                if (attempt == maxRetries)
                    throw;

                await Task.Delay(TimeSpan.FromMilliseconds(200 * attempt));
            }
            catch (RedisServerException ex) when (ex.Message.Contains("WRONGTYPE"))
            {
                _logger.LogError(ex,
                    "Type mismatch on {Op}: key is not a sorted set",
                    operationName);
                throw;
            }
        }

        throw new InvalidOperationException($"Unexpected exit from retry loop for {operationName}");
    }
}
```

### 4.1 — Complete LeaderboardService

A comprehensive leaderboard service with error handling, tiebreaker support, and all common operations.

```csharp
using StackExchange.Redis;
using System;
using System.Collections.Generic;
using System.Threading.Tasks;
using Microsoft.Extensions.Logging;

public class LeaderboardService
{
    private readonly IDatabase _db;
    private readonly ILogger<LeaderboardService> _logger;
    private readonly RedisErrorHandler _errorHandler;
    private const long TiebreakerMax = 999_999;

    public LeaderboardService(
        IDatabase db,
        ILogger<LeaderboardService> logger,
        RedisErrorHandler errorHandler)
    {
        _db = db;
        _logger = logger;
        _errorHandler = errorHandler;
    }

    /// <summary>
    /// Submit a score for a player. Uses ZINCRBY for additive scoring.
    /// Returns the player's new total score.
    /// </summary>
    public async Task<double> SubmitScoreAsync(
        string leaderboardKey,
        string playerId,
        long scoreToAdd)
    {
        try
        {
            double newScore = await _db.SortedSetIncrementAsync(
                leaderboardKey, playerId, scoreToAdd);

            _logger.LogInformation(
                "Player {Player} score updated to {Score} on {Key}",
                playerId, newScore, leaderboardKey);

            return newScore;
        }
        catch (RedisException ex)
        {
            _logger.LogError(ex,
                "Failed to submit score for player {Player} on {Key}",
                playerId, leaderboardKey);
            throw;
        }
    }

    /// <summary>
    /// Submit a score with composite tiebreaker.
    /// Higher score wins; earlier submission wins ties.
    /// </summary>
    public async Task SubmitScoreWithTiebreakerAsync(
        string leaderboardKey,
        string playerId,
        long score)
    {
        long timestampMs = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();

        // Encode: score * (TiebreakerMax + 1) + (TiebreakerMax - timestamp % TiebreakerMax)
        // Later timestamp = higher tiebreaker = lower rank (good for "earlier = better")
        long tiebreaker = TiebreakerMax - (timestampMs % (TiebreakerMax + 1));
        double encodedScore = (double)(score * (TiebreakerMax + 1) + tiebreaker);

        await _db.SortedSetAddAsync(leaderboardKey, playerId, encodedScore);
    }

    /// <summary>
    /// Get top N players with their scores.
    /// Returns a list ordered by score descending.
    /// </summary>
    public async Task<List<LeaderboardEntry>> GetTopPlayersAsync(
        string leaderboardKey,
        int count = 10,
        int offset = 0)
    {
        SortedSetEntry[] entries = await _errorHandler.ExecuteWithRetryAsync(
            () => _db.SortedSetRangeByRankWithScoresAsync(
                leaderboardKey,
                start: offset,
                stop: offset + count - 1,
                order: Order.Descending),
            nameof(GetTopPlayersAsync));

        var result = new List<LeaderboardEntry>();
        for (int i = 0; i < entries.Length; i++)
        {
            result.Add(new LeaderboardEntry
            {
                Rank = offset + i + 1,
                PlayerId = entries[i].Element.ToString(),
                Score = DecodeScoreForDisplay(entries[i].Score)
            });
        }

        return result;
    }

    /// <summary>
    /// Get a player's current rank (1-based) and score.
    /// Returns null if the player is not in the leaderboard.
    /// </summary>
    public async Task<PlayerRankInfo?> GetPlayerRankAsync(
        string leaderboardKey,
        string playerId)
    {
        // Fetch rank, score, and total count in parallel
        Task<long?> rankTask = _db.SortedSetRankAsync(
            leaderboardKey, playerId, Order.Descending);
        Task<double?> scoreTask = _db.SortedSetScoreAsync(
            leaderboardKey, playerId);
        Task<long> totalTask = _db.SortedSetLengthAsync(leaderboardKey);

        await Task.WhenAll(rankTask, scoreTask, totalTask);

        long? rank = rankTask.Result;
        double? score = scoreTask.Result;
        long total = totalTask.Result;

        if (rank == null || score == null)
        {
            return null;
        }

        return new PlayerRankInfo
        {
            PlayerId = playerId,
            Rank = (int)(rank.Value + 1),
            Score = DecodeScoreForDisplay(score.Value),
            TotalPlayers = total,
            TopPercentile = total > 0
                ? Math.Round((double)(rank.Value + 1) / total * 100, 1)
                : 0
        };
    }

    /// <summary>
    /// Get players around a specific player (e.g., 5 above, 5 below).
    /// </summary>
    public async Task<List<LeaderboardEntry>> GetScoresAroundPlayerAsync(
        string leaderboardKey,
        string playerId,
        int radius = 5)
    {
        long? rank = await _db.SortedSetRankAsync(
            leaderboardKey, playerId, Order.Descending);

        if (rank == null)
        {
            return new List<LeaderboardEntry>();
        }

        long start = Math.Max(0, rank.Value - radius);
        long stop = rank.Value + radius;

        SortedSetEntry[] entries = await _db.SortedSetRangeByRankWithScoresAsync(
            leaderboardKey, start, stop, Order.Descending);

        var result = new List<LeaderboardEntry>();
        for (int i = 0; i < entries.Length; i++)
        {
            result.Add(new LeaderboardEntry
            {
                Rank = (int)(start + i + 1),
                PlayerId = entries[i].Element.ToString(),
                Score = DecodeScoreForDisplay(entries[i].Score),
                IsCurrentPlayer = entries[i].Element == playerId
            });
        }

        return result;
    }

    /// <summary>
    /// Trim the leaderboard to only keep the top N players.
    /// </summary>
    public async Task<long> TrimLeaderboardAsync(
        string leaderboardKey,
        int keepTopN)
    {
        // ZREMRANGEBYRANK removes by ascending rank.
        // To keep top N (highest scores), remove all with rank >= N
        // in ascending order (i.e., the bottom players).
        // Since ZREMRANGEBYRANK uses ascending order:
        // Rank 0 = lowest score.
        // We want to keep the top N highest, so remove from 0 to total-N-1.

        long total = await _db.SortedSetLengthAsync(leaderboardKey);
        if (total <= keepTopN)
        {
            return 0;
        }

        long removed = await _db.SortedSetRemoveRangeByRankAsync(
            leaderboardKey,
            start: 0,
            stop: total - keepTopN - 1);

        _logger.LogInformation(
            "Trimmed {Key}: removed {Removed} players, keeping top {Keep}",
            leaderboardKey, removed, keepTopN);

        return removed;
    }

    /// <summary>
    /// Extract the score component from a composite encoded score.
    /// </summary>
    private static long DecodeScoreForDisplay(double encodedScore)
    {
        // If using composite encoding, extract just the score part
        if (encodedScore > TiebreakerMax * 10)
        {
            return (long)(encodedScore / (TiebreakerMax + 1));
        }
        return (long)encodedScore;
    }
}

public class LeaderboardEntry
{
    public int Rank { get; set; }
    public string PlayerId { get; set; } = string.Empty;
    public long Score { get; set; }
    public bool IsCurrentPlayer { get; set; }
}

public class PlayerRankInfo
{
    public string PlayerId { get; set; } = string.Empty;
    public int Rank { get; set; }
    public long Score { get; set; }
    public long TotalPlayers { get; set; }
    public double TopPercentile { get; set; }
}
```

### 4.2 — Time-Based Leaderboard Manager

Manages daily, weekly, and monthly leaderboards with automatic key rotation.

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;
using Microsoft.Extensions.Logging;

public class TimeBasedLeaderboardManager
{
    private readonly IDatabase _db;
    private readonly ILogger<TimeBasedLeaderboardManager> _logger;

    public TimeBasedLeaderboardManager(
        IDatabase db,
        ILogger<TimeBasedLeaderboardManager> logger)
    {
        _db = db;
        _logger = logger;
    }

    public string GetDailyKey(DateTime? date = null)
    {
        DateTime d = date ?? DateTime.UtcNow;
        return $"leaderboard:daily:{d:yyyy-MM-dd}";
    }

    public string GetWeeklyKey(DateTime? date = null)
    {
        DateTime d = date ?? DateTime.UtcNow;
        int weekNum = System.Globalization.ISOWeek.GetWeekOfYear(d);
        int year = d.Year;
        return $"leaderboard:weekly:{year}-W{weekNum:D2}";
    }

    public string GetMonthlyKey(DateTime? date = null)
    {
        DateTime d = date ?? DateTime.UtcNow;
        return $"leaderboard:monthly:{d:yyyy-MM}";
    }

    /// <summary>
    /// Submit a score to the daily leaderboard and set TTL.
    /// </summary>
    public async Task<double> SubmitDailyScoreAsync(
        string playerId,
        double points,
        DateTime? date = null)
    {
        string key = GetDailyKey(date);
        double newScore = await _db.SortedSetIncrementAsync(key, playerId, points);

        // Set TTL to expire 24 hours after end of day (with 1 hour buffer)
        DateTime endOfDay = (date ?? DateTime.UtcNow).Date.AddDays(1).AddHours(1);
        TimeSpan ttl = endOfDay - DateTime.UtcNow;
        await _db.KeyExpireAsync(key, ttl);

        return newScore;
    }

    /// <summary>
    /// Aggregate daily leaderboards into a weekly leaderboard.
    /// </summary>
    public async Task<long> AggregateDailyToWeeklyAsync(DateTime weekStart)
    {
        string weeklyKey = GetWeeklyKey(weekStart);
        string[] dailyKeys = new string[7];

        for (int i = 0; i < 7; i++)
        {
            dailyKeys[i] = GetDailyKey(weekStart.AddDays(i));
        }

        long result = await _db.SortedSetCombineAndStoreAsync(
            SetOperation.Union,
            weeklyKey,
            Array.ConvertAll(dailyKeys, k => (RedisKey)k),
            weights: null,
            Aggregate.Sum);

        // Set TTL on weekly to expire 7 days after end of week + buffer
        DateTime endOfWeek = weekStart.AddDays(7).AddHours(1);
        TimeSpan ttl = endOfWeek - DateTime.UtcNow;
        await _db.KeyExpireAsync(weeklyKey, ttl);

        _logger.LogInformation(
            "Aggregated {Count} players into weekly leaderboard {Key}",
            result, weeklyKey);

        return result;
    }

    /// <summary>
    /// Get the top players from the current daily leaderboard.
    /// Falls back to the weekly if daily is empty.
    /// </summary>
    public async Task<SortedSetEntry[]> GetTopPlayersAsync(
        TimePeriod period,
        int count = 10,
        int offset = 0)
    {
        string key = period switch
        {
            TimePeriod.Daily => GetDailyKey(),
            TimePeriod.Weekly => GetWeeklyKey(),
            TimePeriod.Monthly => GetMonthlyKey(),
            _ => GetDailyKey()
        };

        SortedSetEntry[] entries = await _db.SortedSetRangeByRankWithScoresAsync(
            key,
            start: offset,
            stop: offset + count - 1,
            order: Order.Descending);

        // If daily is empty, fall back to weekly
        if (entries.Length == 0 && period == TimePeriod.Daily)
        {
            string weeklyKey = GetWeeklyKey();
            entries = await _db.SortedSetRangeByRankWithScoresAsync(
                weeklyKey,
                start: offset,
                stop: offset + count - 1,
                order: Order.Descending);
        }

        return entries;
    }
}

public enum TimePeriod
{
    Daily,
    Weekly,
    Monthly
}
```

### 4.3 — Paginated Leaderboard with Caching

```csharp
using StackExchange.Redis;
using System;
using System.Collections.Generic;
using System.Threading.Tasks;
using Microsoft.Extensions.Logging;
using Microsoft.Extensions.Caching.Memory;

public class CachedLeaderboardService
{
    private readonly IDatabase _db;
    private readonly IMemoryCache _cache;
    private readonly ILogger<CachedLeaderboardService> _logger;
    private readonly TimeSpan _cacheDuration = TimeSpan.FromSeconds(30);

    public CachedLeaderboardService(
        IDatabase db,
        IMemoryCache cache,
        ILogger<CachedLeaderboardService> logger)
    {
        _db = db;
        _cache = cache;
        _logger = logger;
    }

    public async Task<List<LeaderboardEntry>> GetTopPlayersCachedAsync(
        string leaderboardKey,
        int count = 10,
        int offset = 0)
    {
        string cacheKey = $"lb:{leaderboardKey}:{offset}:{count}";

        if (_cache.TryGetValue(cacheKey, out List<LeaderboardEntry>? cached))
        {
            if (cached != null)
            {
                _logger.LogDebug("Cache hit for {CacheKey}", cacheKey);
                return cached;
            }
        }

        _logger.LogDebug("Cache miss for {CacheKey}", cacheKey);

        SortedSetEntry[] entries = await _db.SortedSetRangeByRankWithScoresAsync(
            leaderboardKey, offset, offset + count - 1, Order.Descending);

        var result = new List<LeaderboardEntry>();
        for (int i = 0; i < entries.Length; i++)
        {
            result.Add(new LeaderboardEntry
            {
                Rank = offset + i + 1,
                PlayerId = entries[i].Element.ToString(),
                Score = (long)entries[i].Score
            });
        }

        _cache.Set(cacheKey, result, _cacheDuration);

        return result;
    }

    /// <summary>
    /// Invalidate cache when a score is submitted.
    /// </summary>
    public void InvalidateLeaderboardCache(string leaderboardKey)
    {
        // Remove all cache entries for this leaderboard
        string pattern = $"lb:{leaderboardKey}:";
        // Note: In-memory cache doesn't support pattern removal well.
        // In production, consider Redis cache or a distributed cache.
        _logger.LogInformation("Cache invalidated for {Key}", leaderboardKey);
    }
}
```

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;

public class TiebreakerLeaderboardService
{
    private readonly IDatabase _db;
    private const long TiebreakerMax = 999_999;
    private const long Multiplier = TiebreakerMax + 1;

    public TiebreakerLeaderboardService(IDatabase db)
    {
        _db = db;
    }

    /// <summary>
    /// Encode score and tiebreaker into a single double.
    /// score * Multiplier + (Multiplier - 1 - timestampMs % Multiplier)
    /// Higher score wins. For equal scores, earlier submission wins.
    /// </summary>
    public static double EncodeScore(long score, long timestampMs)
    {
        long tiebreaker = (TiebreakerMax - 1) - (timestampMs % TiebreakerMax);
        return score * Multiplier + tiebreaker;
    }

    /// <summary>
    /// Submit score with automatic timestamp-based tiebreaker.
    /// </summary>
    public async Task<double> SubmitScoreAsync(
        string key, string playerId, long score)
    {
        long nowMs = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        double encoded = EncodeScore(score, nowMs);

        await _db.SortedSetAddAsync(key, playerId, encoded);
        return encoded;
    }

    /// <summary>
    /// Get decoded leaderboard entries.
    /// </summary>
    public async Task<List<DecodedLeaderboardEntry>> GetTopPlayersAsync(
        string key, int count = 10)
    {
        SortedSetEntry[] entries = await _db.SortedSetRangeByRankWithScoresAsync(
            key, 0, count - 1, Order.Descending);

        var result = new List<DecodedLeaderboardEntry>();
        for (int i = 0; i < entries.Length; i++)
        {
            double encoded = entries[i].Score;
            long fullValue = (long)encoded;
            long score = fullValue / Multiplier;
            long tiebreaker = fullValue % Multiplier;

            result.Add(new DecodedLeaderboardEntry
            {
                Rank = i + 1,
                PlayerId = entries[i].Element.ToString(),
                DisplayScore = score,
                Tiebreaker = tiebreaker,
                SubmittedAt = DateTimeOffset.FromUnixTimeMilliseconds(
                    (TiebreakerMax - 1 - tiebreaker)).UtcDateTime
            });
        }

        return result;
    }
}

public class DecodedLeaderboardEntry
{
    public int Rank { get; set; }
    public string PlayerId { get; set; } = string.Empty;
    public long DisplayScore { get; set; }
    public long Tiebreaker { get; set; }
    public DateTime SubmittedAt { get; set; }
}
```

### 4.5 — Multi-Criteria Leaderboard (Lua Script)

For complex ranking formulas that combine multiple metrics:

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;

public class MultiCriteriaLeaderboardService
{
    private readonly IDatabase _db;

    public MultiCriteriaLeaderboardService(IDatabase db)
    {
        _db = db;
    }

    /// <summary>
    /// Calculate a weighted score: wins * weight1 + kills * weight2 - deaths * weight3
    /// Uses a Lua script for atomic read-write of multiple sorted sets.
    /// </summary>
    public async Task<double> UpdateWeightedScoreAsync(
        string leaderboardKey,
        string playerId,
        int winsChange,
        int killsChange,
        int deathsChange,
        double winsWeight = 3.0,
        double killsWeight = 1.0,
        double deathsWeight = -0.5)
    {
        string script = @"
            local key = KEYS[1]
            local player = ARGV[1]

            local winsKey = key .. ':wins'
            local killsKey = key .. ':kills'
            local deathsKey = key .. ':deaths'

            -- Update individual metrics
            redis.call('ZINCRBY', winsKey, tonumber(ARGV[2]), player)
            redis.call('ZINCRBY', killsKey, tonumber(ARGV[3]), player)
            redis.call('ZINCRBY', deathsKey, tonumber(ARGV[4]), player)

            -- Read current metrics
            local wins = redis.call('ZSCORE', winsKey, player) or 0
            local kills = redis.call('ZSCORE', killsKey, player) or 0
            local deaths = redis.call('ZSCORE', deathsKey, player) or 0

            -- Calculate weighted score
            local w1 = tonumber(ARGV[5])
            local w2 = tonumber(ARGV[6])
            local w3 = tonumber(ARGV[7])
            local total = wins * w1 + kills * w2 + deaths * w3

            -- Update leaderboard
            redis.call('ZADD', key, total, player)

            return tostring(total)
        ";

        var result = await _db.ScriptEvaluateAsync(script,
            new RedisKey[] { leaderboardKey },
            new RedisValue[] {
                playerId,
                winsChange.ToString(), killsChange.ToString(), deathsChange.ToString(),
                winsWeight.ToString(), killsWeight.ToString(), deathsWeight.ToString()
            });

        return (double)result;
    }
}
```

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;

public class SelfCleaningLeaderboardService
{
    private readonly IDatabase _db;

    public SelfCleaningLeaderboardService(IDatabase db)
    {
        _db = db;
    }

    /// <summary>
    /// Submit a score and set/refresh TTL on the leaderboard.
    /// </summary>
    public async Task<double> SubmitScoreWithTtlAsync(
        string leaderboardKey,
        string playerId,
        double score,
        TimeSpan? ttl = null)
    {
        ttl ??= TimeSpan.FromDays(7);

        double newScore = await _db.SortedSetIncrementAsync(
            leaderboardKey, playerId, score);

        // Set TTL — this refreshes on every score submission
        await _db.KeyExpireAsync(leaderboardKey, ttl);

        return newScore;
    }

    /// <summary>
    /// Remove inactive players (score = 0 or very old entries).
    /// Assumes player scores are reset to 0 when they become inactive.
    /// </summary>
    public async Task<long> RemoveInactivePlayersAsync(string leaderboardKey)
    {
        // Remove all players with score <= 0
        long removed = await _db.SortedSetRemoveRangeByScoreAsync(
            leaderboardKey,
            start: double.NegativeInfinity,
            stop: 0,
            exclude: Exclude.None);

        return removed;
    }

    /// <summary>
    /// Keep only the top N players (trim inactive).
    /// </summary>
    public async Task<long> KeepOnlyTopPlayersAsync(
        string leaderboardKey, int topN)
    {
        long total = await _db.SortedSetLengthAsync(leaderboardKey);
        if (total <= topN) return 0;

        // Remove the lowest (total - topN) players
        long removed = await _db.SortedSetRemoveRangeByRankAsync(
            leaderboardKey,
            start: 0,
            stop: total - topN - 1);

        return removed;
    }
}
```

## Section 6 — Performance and Complexity

---

### 6.1 — Time Complexity Per Operation

| Operation | Complexity | Notes |
|---|---|---|
| ZINCRBY (submit score) | O(log N) | Update + reposition |
| ZREVRANGE (top N) | O(log N + M) | N = size, M = returned |
| ZREVRANK (player rank) | O(log N) | Skiplist traversal |
| ZSCORE (player score) | O(1) | Hash table lookup |
| ZCARD (total players) | O(1) | Cached count |
| ZUNIONSTORE (aggregate) | O(N log N) | N = total elements across all sets |
| ZREMRANGEBYRANK (trim) | O(log N + M) | M = removed elements |

---

### 6.2 — Memory Usage

```
Per element in sorted set (skiplist encoding):
  Member string: variable (e.g., "player_abc123" = ~16 bytes)
  Score (double): 8 bytes
  Skiplist overhead: ~32 bytes (forward pointers, level array)
  Hash table entry: ~16 bytes
  Total: ~72-100 bytes per element

For a leaderboard with:
  10,000 players:  ~720 KB  — very small
  100,000 players: ~7.2 MB  — comfortable
  1,000,000 players: ~72 MB — reasonable
  10,000,000 players: ~720 MB — needs memory planning
```

---

### 6.3 — Caching Strategy

| Scenario | Cache Strategy | Benefit |
|---|---|---|
| Top-N (no change) | Cache for 30-60s | Redundant reads eliminated |
| Top-N (frequent updates) | Short cache (5-10s) | Fresh enough, fewer queries |
| Player rank (own) | No cache | Fast single query, cache adds complexity |
| Aggregate leaderboards | Cache until next aggregation | Prevents repeated UNION |

## Section 7 — Gotchas and Pitfalls

---

### 7.1 — Double Precision for Large Scores

Redis uses IEEE 754 double-precision for scores. Precise integers up to 2^53 (9,007,199,254,740,991). Beyond that, precision loss occurs.

```csharp
// SAFE: integer scores up to ~9 quadrillion
await db.SortedSetAddAsync("lb", "player", 9_007_199_254_740_991L);

// UNSAFE: beyond 2^53
double large = 9_007_199_254_740_993.0; // will be rounded
await db.SortedSetAddAsync("lb", "player", large);
```

**Mitigation:** For composite scores, ensure the total encoded value stays below 2^53.

---

### 7.2 — Tiebreaker Strategy

When scores are equal, Redis uses lexicographic ordering of member names. This can cause unexpected rank order.

**Problem:**
```bash
# Both players score 1500, but "aaa" comes first lexicographically
ZADD leaderboard 1500 "aaa"
ZADD leaderboard 1500 "zzz"
ZREVRANK "aaa"  # 0
ZREVRANK "zzz"  # 1  (aaa wins tie, which may be wrong)
```

**Solutions:**
1. Composite score: encode time-based tiebreaker in the score
2. Prefix member names with tiebreaker values
3. Accept lexicographic and document the behavior

---

### 7.3 — ZREVRANK Returns -0 vs nil

In older Redis versions, ZREVRANK on an empty set returned `-0` (negative zero). Modern Redis returns `nil` consistently. StackExchange.Redis handles this transparently — it returns `long?` with `null` for missing members.

---

### 7.4——Leaderboard Key Naming Conventions

Poor naming leads to collisions and confusion.

```
Bad: "leaderboard1", "board", "scores"
Good: "leaderboard:daily:2024-01-15:game1:ranked"
      "leaderboard:weekly:2024-W03:global"
      "leaderboard:monthly:2024-01:referrals"
```

Convention: `leaderboard:<period>:<date>:<scope>:<type>`

---

### 7.5 — ZUNIONSTORE Memory Spike

ZUNIONSTORE creates a new sorted set. For large aggregations, this temporarily doubles memory usage for the keys involved. Schedule aggregations during low-traffic periods.

```csharp
// Consider the memory impact
await db.SortedSetCombineAndStoreAsync(
    SetOperation.Union,
    "leaderboard:weekly:2024-W03",      // destination — new sorted set created
    dailyKeys,                           // source — 7 daily sorted sets
    weights: null,
    Aggregate.Sum);
```

---

### 7.6 — ZREMRANGEBYRANK Ascending Semantics

ZREMRANGEBYRANK uses ascending rank order (lowest score first). To remove the lowest-ranked players:

```bash
# Remove the bottom 100 players (lowest scores)
ZREMRANGEBYRANK leaderboard 0 99

# Remove the top 10 players (highest scores) — use score range instead
ZREMRANGEBYSCORE leaderboard -inf +inf LIMIT 0 10  # Not supported directly
# Better: Use ZREVRANGE to find the 10th highest score, then ZREMRANGEBYSCORE
```

---

### 7.7 — EXPIRE Resets on Every ZADD/ZINCRBY

Setting EXPIRE on a leaderboard key does NOT auto-extend on every write. You must explicitly set or refresh the TTL each time you modify the set.

```csharp
// Correct: Set TTL after every score submission
await db.SortedSetIncrementAsync(key, playerId, score);
await db.KeyExpireAsync(key, expiry); // Refresh TTL

// Wrong: TTL set once, expires regardless of activity
await db.KeyExpireAsync(key, TimeSpan.FromDays(7));
// ...later...
await db.SortedSetIncrementAsync(key, playerId, score);
// TTL is NOT refreshed unless you call KeyExpireAsync again
```

---

### 7.8 — Network Overhead for Top-N Queries

Fetching top 100 with WITHSCORES returns 200 values. Each is a separate network payload. For very large leaderboard queries, consider:

1. **Reducing column count** — skip WITHSCORES if scores aren't needed
2. **Compression** — enable RESP3 or compression in the transport
3. **Caching** — cache top-N results server-side
4. **Batching** — don't fetch more than needed for the current view

---

### 7.9 — Leaderboard Page Size Mismatch

When requesting a page near the end of the leaderboard, the returned count may be less than the page size:

```csharp
// Request page 10 of 10, page size 10
SortedSetEntry[] entries = await db.SortedSetRangeByRankWithScoresAsync(
    key, 90, 99, Order.Descending);
// entries.Length may be < 10 if total is less than 100

// Always check actual count
int actualCount = entries.Length;
```

---

### 7.10 — Thread Safety in Score Submission

ZINCRBY is atomic and thread-safe at the Redis level. However, if your application uses composite scoring with read-write, you need Lua for atomicity:

```csharp
// NOT ATOMIC — two concurrent calls can interleave
double current = await db.SortedSetScoreAsync(key, player);
double encoded = EncodeScore((long)current + 100, nowMs);
await db.SortedSetAddAsync(key, player, encoded);

// ATOMIC — use Lua
string script = "redis.call('ZINCRBY', KEYS[1], ARGV[1], ARGV[2]); " +
    "local s = redis.call('ZSCORE', KEYS[1], ARGV[2]); " +
    "return s";
await db.ScriptEvaluateAsync(script, new RedisKey[] { key },
    new RedisValue[] { "100", player });
```

## Section 8 — Related Patterns and Comparisons

---

### 8.1 — Leaderboard vs Rate Limiter

| Aspect | Leaderboard | Rate Limiter |
|---|---|---|
| Score meaning | Points | Timestamp |
| Key per | Game/period | User/action |
| Cleanup | ZREMRANGEBYRANK (top N) | ZREMRANGEBYSCORE (outside window) |
| Query | ZREVRANGE top N | ZCARD (count in window) |
| Update | ZINCRBY (add points) | ZADD (add timestamp) |

Both use sorted sets but with opposite score semantics.

---

### 8.2 — Leaderboard vs Priority Queue (using sorted sets)

| Aspect | Leaderboard | Priority Queue |
|---|---|---|
| Order | By score descending | By score ascending (earliest deadline) |
| Insert | ZADD | ZADD |
| Pop | ZREVRANGE (top) | ZRANGE (bottom) + ZREMRANGEBYRANK |
| Update | ZINCRBY (change priority) | ZINCRBY or ZADD |

Both are sorted sets but prioritize different ends.

---

### 8.3 — Leaderboard vs Hash for Player Stats

| Aspect | Sorted Set | Hash |
|---|---|---|
| Ranking | Built-in | Requires external sort |
| Score lookup by member | O(log N) or O(1) | O(1) |
| Top-N query | O(log N + M) | O(N) (scan all) |
| Pagination | Native (ZRANGE) | Requires manual implementation |

**Best practice:** Use sorted set for the leaderboard index, hash for rich player metadata:

```bash
ZADD leaderboard:global 2500 "player_abc"
HSET player:player_abc name "Alice" avatarURL "https://..." level 42
```

---

### 8.4 — Leaderboard with Fallback to Relational Database

For enterprise leaderboards, consider a hybrid approach:

1. **Redis** — real-time top-N, rank queries (low latency)
2. **PostgreSQL/MySQL** — persistent storage, historical analysis, auditing

```csharp
public async Task SubmitScoreWithPersistenceAsync(
    string leaderboardKey,
    string playerId,
    int score,
    IDatabase redis,
    DbContext dbContext)
{
    // 1. Update Redis (fast, volatile)
    await redis.SortedSetIncrementAsync(leaderboardKey, playerId, score);

    // 2. Persist to database (durable, slow)
    var record = await dbContext.ScoreRecords
        .FirstOrDefaultAsync(r => r.PlayerId == playerId && r.Key == leaderboardKey);
    if (record == null)
    {
        record = new ScoreRecord { PlayerId = playerId, Key = leaderboardKey };
        dbContext.ScoreRecords.Add(record);
    }
    record.Score += score;
    await dbContext.SaveChangesAsync();

    // 3. Rebuild Redis from DB if Redis data is lost (RESTORE)
}
```

### 8.5 — Leaderboard vs Redis Stream for Event-Driven Ranking

| Pattern | Pros | Cons |
|---|---|---|
| Direct sorted set ZINCRBY | Low latency, simple | No event history, can't replay |
| Stream + Consumer | Full event log, replayable, audit | Higher latency, more complex |
| Stream → Sorted Set (bridge) | Both audit trail and fast queries | Two systems to maintain |

The stream→sorted set pattern is used when you need both an event log and a real-time leaderboard:

```
Game Event → Stream (XADD) → Consumer (XREADGROUP) → ZINCRBY → Sorted Set
                                                              ↓
                                                     Leaderboard Display
```

```bash
# Players around rank 2 (rank 0-4)
ZREVRANGE leaderboard 0 4 WITHSCORES

# Aggregate two daily boards into weekly
ZUNIONSTORE leaderboard:weekly 2 leaderboard:daily:1 leaderboard:daily:2

# Trim to top 100
# First get total, then remove bottom (total-100) players
ZREMRANGEBYRANK leaderboard 0 (ZCARD leaderboard)-100-1
```

### 9.3 — Key Design Principles

1. **Use ZINCRBY for additive scores** — atomic, no race conditions
2. **Always convert to 1-based ranks** for user-facing display
3. **Plan tiebreakers** — composite scores for controlled ordering
4. **Rotate keys by time period** — daily/weekly with TTL for automatic cleanup
5. **Aggregate with ZUNIONSTORE** — combine daily into weekly leaderboards
6. **Cache top-N with short TTL** — reduces Redis load for popular queries
7. **Keep scores as integers** — avoid floating point precision issues
8. **Use Lua for multi-step atomic operations** — composite score increments

### 9.4 — Common Pitfalls Checklist

- [ ] Score exceeds 2^53 causing precision loss
- [ ] ZREVRANK returns null but code assumes non-null
- [ ] Rank shown as 0-based (should be 1-based for users)
- [ ] TTL not refreshed on score update (leaderboard expires early)
- [ ] Tiebreaker not considered (lexicographic order surprises)
- [ ] ZUNIONSTORE memory spike during aggregation
- [ ] Page offset exceeds total players (empty results)
- [ ] WRONGTYPE error when key exists as different type
- [ ] Race condition in read-score-modify-score pattern
