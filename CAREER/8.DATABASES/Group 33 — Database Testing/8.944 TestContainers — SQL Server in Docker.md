# 8.944 — TestContainers — SQL Server in Docker

## Section 1 — Overview and Motivation

Integration testing against a real SQL Server database has historically been painful for .NET developers. The traditional options were all flawed:

1. **LocalDB** — Installed on developer machines, but not available in CI. Different behavior from full SQL Server. Version mismatches between developers.
2. **Full SQL Server install** — Heavy, requires admin rights, ties up ports, conflicts between projects, not available in CI.
3. **Shared integration database** — State leaking between tests, data corruption, CI pipeline conflicts, manual cleanup.
4. **SQLite in-memory** — Different SQL dialect, missing features, different behavior for transactions, concurrency, and data types.
5. **Mock everything** — Zero confidence in SQL correctness, false positives, bugs reach production.

TestContainers solves all of these problems by providing disposable, programmatic SQL Server instances via Docker containers.

### 1.1 — What TestContainers Provides

TestContainers for .NET is a library that manages the lifecycle of Docker containers from within test code. It handles:

- **Image pulling** — Downloads the SQL Server Docker image on first use
- **Container creation** — Configures and starts the container
- **Health checks** — Waits for SQL Server to be ready to accept connections
- **Connection string generation** — Provides the correct connection string pointing to the container
- **Cleanup** — Stops and removes the container when tests complete

### 1.2 — When to Use TestContainers

Use TestContainers when:

- You need a real SQL Server database for integration tests
- Your CI environment supports Docker (GitHub Actions, Azure DevOps, GitLab CI, Jenkins)
- You want disposable, isolated database instances per test run
- You need to test against the same database engine as production
- You want to avoid manual database setup and maintenance

Do NOT use TestContainers when:

- Your CI does not support Docker (some restricted environments)
- You need sub-second test startup (container takes 10-30 seconds)
- You are running a large number of test suites that all need separate containers (memory/disk constraints)
- You are on a constrained machine with limited Docker resources

### 1.3 — How It Fits in the Testing Strategy

TestContainers is the infrastructure layer that enables Level 2 (Integration) and Level 3 (Contract) tests from the database testing pyramid:

```
TestContainers provides disposable SQL Server instances via Docker.
         │
         ▼
┌─────────────────────────────────────┐
│  Integration Tests (Real DB)        │
│  Schema Contract Tests              │
│  Migration Tests                    │
│  Dapper Query Tests                 │
│  EF Core Query Translation Tests    │
└─────────────────────────────────────┘
```

## Section 2 — Core Concepts

### 2.1 — The TestContainers Lifecycle

The TestContainers lifecycle follows a consistent pattern: Build, Start, Use, Dispose.

**Build stage:** The MsSqlBuilder configures the container image, password, port mappings, environment variables, and cleanup behavior. No Docker operations happen at this stage.

**Start stage:** StartAsync() performs the actual Docker operations. On first run, Docker downloads the SQL Server image (~2GB on disk, ~1.5GB compressed). It then creates and starts the container. The library waits for SQL Server to be ready by polling the SQL Server error log or TCP port. This stage takes 10-30 seconds on first run, 3-10 seconds on subsequent runs when the image is cached locally.

**Use stage:** After StartAsync() completes, GetConnectionString() returns a connection string pointing to the container's SQL Server instance. Tests use this connection string to create DbContext instances (EF Core) or IDbConnection instances (Dapper).

**Dispose stage:** DisposeAsync() stops and removes the container. With WithCleanUp(true), Docker also removes the container's anonymous volumes.

### 2.2 — Key Classes in the Library

| Class | Purpose |
|-------|---------|
| MsSqlBuilder | Fluent builder for SQL Server container configuration |
| MsSqlContainer | Represents the running SQL Server container instance |
| TestcontainersSettings | Global settings like image registry, timeouts, and labels |
| IContainer | Interface implemented by all container types |
| WaitStrategy | Configures how the library waits for container readiness |

## Section 3 — Setup and Configuration

### 3.1 — Minimal Container Setup

```csharp
using Testcontainers.MsSql;

var msSqlContainer = new MsSqlBuilder()
    .WithImage("mcr.microsoft.com/mssql/server:2022-latest")
    .WithPassword("Your_Str0ng_P@ssw0rd!")
    .WithCleanUp(true)
    .Build();

await msSqlContainer.StartAsync();

var connectionString = msSqlContainer.GetConnectionString();
// Use connectionString to create DbContext or SqlConnection

await msSqlContainer.DisposeAsync();
```

### 3.2 — Builder Configuration Options

**Image selection:**
```csharp
new MsSqlBuilder().WithImage("mcr.microsoft.com/mssql/server:2022-latest")
new MsSqlBuilder().WithImage("mcr.microsoft.com/mssql/server:2019-latest")
new MsSqlBuilder().WithImage("mcr.microsoft.com/azure-sql-edge:latest") // ARM64 compatible
```

**Password (8+ characters with upper, lower, digit, special):**
```csharp
new MsSqlBuilder().WithPassword("My_Str0ng_P@ssw0rd!")
```

**Port mapping:**
```csharp
new MsSqlBuilder()                                   // Random host port (recommended)
new MsSqlBuilder().WithPortBinding(1433, 1433)       // Fixed port (risk of conflict)
new MsSqlBuilder().WithPortBinding(11433, 1433)       // Specific host port
```

**Cleanup behavior:**
```csharp
new MsSqlBuilder().WithCleanUp(true)   // Auto-remove on dispose (recommended)
new MsSqlBuilder().WithCleanUp(false)  // Keep for debugging
```

**Container naming:**
```csharp
new MsSqlBuilder().WithName("my-test-sqlserver")
```

**Environment variables:**
```csharp
new MsSqlBuilder()
    .WithEnvironment("MSSQL_PID", "Developer")
    .WithEnvironment("MSSQL_MEMORY_LIMIT_MB", "2048")
    .WithEnvironment("MSSQL_LCID", "1033")
```

### 3.3 — Complete xUnit Fixture

```csharp
public class SqlServerContainerFixture : IAsyncLifetime
{
    private MsSqlContainer _container;
    private Respawner _respawner;

    public string ConnectionString { get; private set; }

    public async Task InitializeAsync()
    {
        _container = new MsSqlBuilder()
            .WithImage("mcr.microsoft.com/mssql/server:2022-latest")
            .WithPassword("Test_P@ssw0rd!")
            .WithCleanUp(true)
            .Build();

        await _container.StartAsync();
        ConnectionString = _container.GetConnectionString();

        _respawner = await Respawner.CreateAsync(ConnectionString, new RespawnerOptions
        {
            SchemasToInclude = new[] { "dbo" },
            TablesToIgnore = new[] { "__EFMigrationsHistory" }
        });
    }

    public async Task DisposeAsync()
    {
        if (_container != null)
            await _container.DisposeAsync();
    }

    public IDbConnection CreateConnection() => new SqlConnection(ConnectionString);

    public AppDbContext CreateDbContext()
    {
        var options = new DbContextOptionsBuilder<AppDbContext>()
            .UseSqlServer(ConnectionString).Options;
        return new AppDbContext(options);
    }

    public async Task ResetDatabaseAsync()
        => await _respawner.ResetAsync(ConnectionString);
}
```

[Fact]
public async Task JoinQuery_AggregatesCorrectly()
{
    using var conn = _fixture.CreateConnection();
    await _fixture.ResetDatabaseAsync();

    var customerId = Guid.NewGuid();
    await conn.ExecuteAsync(
        "INSERT INTO Customers (Id, Name, Status) VALUES (@Id, @Name, @Status)",

    var orderId = Guid.NewGuid();
    await conn.ExecuteAsync(
        "INSERT INTO Orders (Id, CustomerId, Total, Status, CreatedAt) VALUES (@Id, @CustomerId, @Total, @Status, @CreatedAt)",

        "INSERT INTO OrderItems (Id, OrderId, ProductName, Quantity, UnitPrice) VALUES (@Id, @OrderId, @ProductName, @Quantity, @UnitPrice)",

        JOIN Orders o ON c.Id = o.CustomerId

### 5.4 — Multi-Mapping (One-to-Many)

```csharp
[Fact]
public async Task MultiMapping_OneToMany()
{
    using var conn = _fixture.CreateConnection();
    await _fixture.ResetDatabaseAsync();

    var orderId = Guid.NewGuid();
    await conn.ExecuteAsync(
        "INSERT INTO Orders (Id, CustomerId, Total, Status, CreatedAt) VALUES (@Id, @CustomerId, @Total, @Status, @CreatedAt)",
        new { Id = orderId, CustomerId = Guid.NewGuid(), Total = 250m, Status = "Pending", CreatedAt = DateTime.UtcNow });

    await conn.ExecuteAsync(
        "INSERT INTO OrderItems (Id, OrderId, ProductName, Quantity, UnitPrice) VALUES (@Id, @OrderId, @ProductName, @Quantity, @UnitPrice)",
        new[] {
            new { Id = Guid.NewGuid(), OrderId = orderId, ProductName = "A", Quantity = 2, UnitPrice = 50m },
            new { Id = Guid.NewGuid(), OrderId = orderId, ProductName = "B", Quantity = 1, UnitPrice = 150m }
        });

    var lookup = new Dictionary<Guid, Order>();
    await conn.QueryAsync<Order, OrderItem, Order>(
        "SELECT o.Id, o.Total, oi.Id, oi.ProductName, oi.Quantity, oi.UnitPrice FROM Orders o LEFT JOIN OrderItems oi ON o.Id = oi.OrderId WHERE o.Id = @OrderId",
        (order, item) =>
        {
            if (!lookup.TryGetValue(order.Id, out var existing))
                lookup[order.Id] = existing = order;
            if (item != null) existing.Items.Add(item);
            return existing;
        },
        new { OrderId = orderId }, splitOn: "Id");

    Assert.Equal(2, lookup[orderId].Items.Count);
}
```

## Section 7 — CI/CD and Production Considerations

### 7.1 — Docker Requirements

TestContainers requires Docker to be available. In CI environments:

| Provider | Docker Support | Configuration |
|----------|---------------|---------------|
| GitHub Actions | ✅ Built-in | ubuntu-latest includes Docker |
| Azure DevOps | ✅ Available | Self-hosted or DockerInstaller task |
| GitLab CI | ✅ Available | dind (Docker-in-Docker) service |
| Jenkins | ✅ Available | Agent with Docker installed |
| AWS CodeBuild | ✅ Available | Enable Docker in build spec |
| TeamCity | ✅ Available | Agent with Docker installed |

**GitHub Actions example:**

`yaml
jobs:
  integration:
    runs-on: ubuntu-latest
    services:
      docker:
        image: docker:24.0.5
        options: --privileged
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'
      - run: dotnet test ./tests/IntegrationTests
`

**GitLab CI example:**

`yaml
integration:
  image: mcr.microsoft.com/dotnet/sdk:8.0
  services:
    - docker:24.0.5-dind
  variables:
    DOCKER_HOST: tcp://docker:2375
  script:
    - dotnet test ./tests/IntegrationTests
`

### 7.2 — Optimizing CI Pipeline for TestContainers

1. **Cache Docker images** — Use Docker layer caching to avoid re-downloading the SQL Server image on every run.
2. **Share containers across suites** — Use collection fixtures to share a single container across all tests in a suite.
3. **Parallelize test suites** — Each collection gets its own container; collections can run in parallel.
4. **Set appropriate timeouts** — Docker pull + container start can take 2-5 minutes on cold CI cache.
5. **Monitor disk space** — SQL Server image is ~2GB. Multiple containers consume significant disk.

### 7.3 — Dealing with CI Environments Without Docker

If your CI does not support Docker:

1. **Use a managed database service** — Provision a SQL Server instance (Azure SQL, AWS RDS, etc.) and use its connection string in tests.
2. **Use LocalDB if on Windows** — Limited to Windows-based CI agents. Not the same as full SQL Server.
3. **Use SQLite as a fallback** — Not ideal but better than no integration tests. Acknowledge the limitations.
4. **Use a standalone Docker host** — Set up a dedicated Docker host and configure TestContainers to connect remotely via DOCKER_HOST environment variable.

`csharp
// Connect to remote Docker host
Environment.SetEnvironmentVariable("DOCKER_HOST", "tcp://remote-docker-host:2375");
`

### 7.4 — Performance Considerations

| Factor | Impact | Mitigation |
|--------|--------|------------|
| Image pull (first time) | 2-5 minutes | Cache image in CI; use smaller images |
| Container startup | 10-30 seconds | Share container via collection fixture |
| Migration application | 5-30 seconds | Run once per collection, not per test |
| Respawn cleanup | 50-200ms | Acceptable per test |
| Data seeding | 10-500ms | Deterministic, minimal data per test |

### 7.5 — Security Considerations

1. **Use strong passwords** — SQL Server requires complex passwords. Use environment variables or secure configuration.
2. **Don't reuse production passwords** — Test container passwords should be different from production.
3. **Container network isolation** — TestContainers by default uses Docker's bridge network, isolating from other containers.
4. **Don't expose container ports to the internet** — TestContainers binds to localhost by default.
5. **Clean up containers** — WithCleanUp(true) ensures containers are removed even if tests fail.

### 7.6 — Troubleshooting Common CI Issues

**Problem: Docker not found**
`
System.InvalidOperationException: Docker is not available.
`
**Solution:** Verify Docker is installed and the Docker daemon is running. In GitHub Actions, use ubuntu-latest which includes Docker.

**Problem: Port conflicts**
`
Docker.DotNet.DockerApiException: Conflict. The container name ... is already in use.
`
**Solution:** Use WithCleanUp(true) to ensure containers are removed. If tests are interrupted, run docker rm  manually.

**Problem: Out of disk space**
`
No space left on device
`
**Solution:** Clean up Docker images and containers regularly. Use docker system prune -af in CI cleanup step.

**Problem: SQL Server not ready**
`
Timeout waiting for container to be ready.
`
**Solution:** Increase the wait timeout or check Docker resource allocation (RAM, CPU). SQL Server requires at least 2GB RAM.

        builder.ConfigureTestServices(services =>
        {
            // Remove the real DbContext registration
            var descriptor = services.SingleOrDefault(
                d => d.ServiceType == typeof(DbContextOptions<AppDbContext>));
            if (descriptor != null) services.Remove(descriptor);

            // Add DbContext pointing to the test container
            services.AddDbContext<AppDbContext>(options =>
                options.UseSqlServer(_container.GetConnectionString()));
        });
    }

    public async Task InitializeAsync()
    {
        await _container.StartAsync();

        // Apply migrations
        using var scope = Services.CreateScope();
        var context = scope.ServiceProvider.GetRequiredService<AppDbContext>();
        await context.Database.MigrateAsync();
    }

    public async Task DisposeAsync()
    {
        await _container.DisposeAsync();
    }
}

## Section 9 — Summary, Gotchas, and Checklist

### 9.1 — Key Principles

1. **Use TestContainers for disposable, real database instances.** They provide the same SQL Server engine as production in a CI-compatible package.
2. **Share containers across test suites.** Collection fixtures reduce startup overhead from 30s to near-zero.
3. **Always clean up.** WithCleanUp(true) ensures containers are removed after tests, preventing resource leaks.
4. **Apply schema via migrations or scripts.** TestContainers provides the database engine; you must set up the schema.
5. **Combine with Respawn for per-test isolation.** Start the container once, reset the data between each test.
6. **Use deterministic test data.** Fixed GUIDs, dates, and values make tests repeatable.

### 9.2 — Gotchas

- **Docker must be available.** TestContainers cannot work without Docker. CI agents must have Docker installed and the daemon running.
- **Container startup is slow.** 10-30 seconds for SQL Server. Use collection fixtures to pay this cost once per suite.
- **SQL Server image is ~2GB.** First pull takes time. Cache the image in CI to avoid re-downloading.
- **M1 Macs require platform flag.** SQL Server image is x86_64. Use --platform linux/amd64 or use Azure SQL Edge which supports ARM64.
- **Port conflicts.** If multiple containers try to bind to the same host port, they fail. Use random port binding (default).
- **Memory requirements.** SQL Server requires at least 2GB RAM. Smaller CI agents may fail. Consider Azure SQL Edge or limit memory with MSSQL_MEMORY_LIMIT_MB.
- **Container cleanup on test abort.** If tests are killed mid-run, containers may leak. Add a cleanup script: docker rm -f .
- **Timeouts.** Default wait strategy may timeout on slow machines. Configure longer wait times or use UntilMessageIsLogged.
- **Running as non-root.** Some CI environments restrict Docker access. Ensure the CI user is in the docker group or use rootless Docker.
- **Windows containers.** TestContainers for .NET supports Linux containers. Windows containers have limited support.
- **Networking.** Some CI environments restrict container networking. TestContainers uses bridge network by default; adjust if needed.
- **Environment variable inheritance.** TestContainers may inherit proxy settings from the host, causing connection issues.

### 9.3 — Checklist

- [ ] TestContainers.MsSql NuGet package is installed in the test project
- [ ] MsSqlContainer is configured with the correct image and password
- [ ] Container cleanup is enabled (WithCleanUp(true))
- [ ] Fixture implements IAsyncLifetime for async setup/teardown
- [ ] Container is shared via ICollectionFixture (not per-test)
- [ ] Respawn is configured for per-test data cleanup
- [ ] EF Core migrations are applied during fixture setup
- [ ] Dapper schema scripts are executed during fixture setup
- [ ] Connection string is retrieved from GetConnectionString()
- [ ] Tests use deterministic data (fixed GUIDs, dates)
- [ ] Docker is available in the CI environment
- [ ] CI pipeline has sufficient RAM (2GB+) for SQL Server
- [ ] CI pipeline caches Docker images to reduce pull time
- [ ] Port conflicts are avoided (use random port binding)
- [ ] M1 Mac workaround is documented (use Azure SQL Edge or --platform)
- [ ] Container lifecycle is managed: StartAsync, GetConnectionString, DisposeAsync
- [ ] Tests have appropriate timeouts (30s per test, longer for container startup)
- [ ] Orphaned container cleanup is configured (docker system prune or CI cleanup step)
- [ ] Custom wait strategies are configured if default timeout is insufficient
- [ ] Test collections are configured with DisableParallelization for shared containers
- [ ] Multiple container support is tested if application uses multiple databases

[Fact]
public async Task ComplexJoin_ThreeTables_ReturnsCorrectAggregation()
{
    using var conn = _fixture.CreateConnection();
    await _fixture.ResetDatabaseAsync();

    var customerId = Guid.NewGuid();
    await conn.ExecuteAsync(
        "INSERT INTO Customers (Id, Name, Status) VALUES (@Id, @Name, @Status)",
        new { Id = customerId, Name = "Join Test Customer", Status = "Active" });

    var orderId = Guid.NewGuid();
    await conn.ExecuteAsync(
        "INSERT INTO Orders (Id, CustomerId, Total, Status, CreatedAt) VALUES (@Id, @CustomerId, @Total, @Status, @CreatedAt)",
        new { Id = orderId, CustomerId = customerId, Total = 500m, Status = "Shipped", CreatedAt = DateTime.UtcNow });

    await conn.ExecuteAsync(
        "INSERT INTO OrderItems (Id, OrderId, ProductName, Quantity, UnitPrice) VALUES (@Id, @OrderId, @ProductName, @Quantity, @UnitPrice)",
        new[] {
            new { Id = Guid.NewGuid(), OrderId = orderId, ProductName = "Widget X", Quantity = 3, UnitPrice = 100m },
            new { Id = Guid.NewGuid(), OrderId = orderId, ProductName = "Widget Y", Quantity = 2, UnitPrice = 100m }
        });

    var result = await conn.QuerySingleAsync<CustomerAggregate>(@"
        SELECT
            c.Id,
            c.Name,
            COUNT(DISTINCT o.Id) AS TotalOrders,
            SUM(oi.Quantity * oi.UnitPrice) AS TotalRevenue,
            COUNT(oi.Id) AS TotalItems
        FROM Customers c
        INNER JOIN Orders o ON c.Id = o.CustomerId
        INNER JOIN OrderItems oi ON o.Id = oi.OrderId
        WHERE c.Id = @CustomerId
        GROUP BY c.Id, c.Name",
        new { CustomerId = customerId });

    Assert.Equal("Join Test Customer", result.Name);
    Assert.Equal(1, result.TotalOrders);
    Assert.Equal(500m, result.TotalRevenue);
    Assert.Equal(5, result.TotalItems);

[Fact]

### 9.11 — Configuration Cheat Sheet

| Configuration | Builder Method | Default | Notes |
|--------------|---------------|---------|-------|
| SQL Server image | WithImage() | 2022-latest | Also: 2019-latest, azure-sql-edge |
| SA password | WithPassword() | (required) | 8+ chars, complex |
| Port mapping | WithPortBinding() | Random | Avoid fixed ports for CI |
| Auto-cleanup | WithCleanUp() | true | Set false for debugging |
| Container name | WithName() | Auto-generated | Useful for identification |
| Memory limit | WithEnvironment() | Unlimited | Set for constrained CI |
| Edition | WithEnvironment() | Developer | Express, Standard, Enterprise |
