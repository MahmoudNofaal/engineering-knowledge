# Bulk Insert — SqlBulkCopy

## 1 — Overview

\SqlBulkCopy\ is a .NET class in \Microsoft.Data.SqlClient\ (formerly \System.Data.SqlClient\) that writes a \DataTable\ or \IDataReader\ directly to a SQL Server table via the **TDS (Tabular Data Stream) bulk load protocol**. It is the fastest way to insert 1,000+ rows into SQL Server, often achieving 10x–100x speedup over individual \INSERT\ statements.

### How it works

Unlike individual \INSERT\ statements that go through the full query pipeline (parse, compile, execute, log each row), \SqlBulkCopy\:

1. Opens a **bulk copy session** on the server
2. Streams rows as raw data packets via TDS
3. Uses **minimal logging** (if the target table is a heap or has \SIMPLE\/\BULK_LOGGED\ recovery model)
4. Writes directly to the data pages, bypassing most constraint checks

### Core API

\\\csharp
using Microsoft.Data.SqlClient;

// Minimal example
var dataTable = new DataTable();
// ... populate DataTable ...

using var bulkCopy = new SqlBulkCopy(connectionString);
bulkCopy.DestinationTableName = "Orders";
await bulkCopy.WriteToServerAsync(dataTable);
\\\

### Supported sources

| Source Type | Interface | When to Use |
|---|---|---|
| \DataTable\ | — | In-memory collection, small/medium batches, unit tests |
| \DataRow[]\ | — | Subset of a DataTable |
| \DbDataReader\ | \IDataReader\ | Streaming from another query (e.g., Dapper \ExecuteReaderAsync\) |
| \ITableMapping\ | — | Column name mismatch correction |

### Key properties

| Property | Purpose |
|---|---|
| \DestinationTableName\ | Target table name (required) |
| \BatchSize\ | Rows per network round-trip (default = all) |
| \ColumnMappings\ | Maps source columns → destination columns |
| \BulkCopyTimeout\ | Command timeout in seconds (default 30) |
| \EnableStreaming\ | For \IDataReader\ — streams rows without buffering |
| \NotifyAfter\ + \SqlRowsCopied\ | Progress notification |

## 2 — When to Use SqlBulkCopy

### Use SqlBulkCopy when

| Scenario | Reason |
|---|---|
| Loading 1,000+ rows in one go | Row-by-row INSERT is ~10–100x slower |
| ETL / data import from CSV, JSON, or another DB | SqlBulkCopy is the fastest ingestion path |
| Initial seed of a table (reference data, migrations) | Minimize import time |
| Data warehouse / staging table loads | Minimal logging reduces transaction log pressure |
| Streaming query results from one DB to another | \IDataReader\ avoids materializing all rows in memory |

### Don't use SqlBulkCopy when

| Scenario | Alternative |
|---|---|
| Inserting fewer than 100 rows | Use regular \INSERT\ (overhead of setup exceeds benefit) |
| Need constraint enforcement (FK, check, trigger) | Use normal INSERT or disable/re-enable constraints manually |
| Target is not SQL Server | Use provider-specific bulk insert (Npgsql \BinaryCopy\, \MySqlBulkLoader\) |
| Every row must be individually validated | Insert through TVP or staging → validate → merge |

### Typical threshold

\\\
Rows  |  INSERT (loop)  |  TVP  |  SqlBulkCopy
------|-----------------|-------|---------------
100   |  120ms          |  80ms |  150ms (setup overhead)
1K    |  1.2s           |  400ms|  120ms
10K   |  14s            |  2s   |  350ms
100K  |  ~2min          |  18s  |  2.5s
1M    |  ~20min         |  ~3min|  ~20s
\\\

SqlBulkCopy becomes the clear winner above ~1,000 rows.

## 3 — DataTable Setup and Schema Matching

### Creating a DataTable that matches the target table

\\\csharp
public DataTable BuildOrderDataTable()
{
    var dt = new DataTable("Orders");

    dt.Columns.Add("Id", typeof(Guid));
    dt.Columns.Add("CustomerId", typeof(Guid));
    dt.Columns.Add("OrderDate", typeof(DateTime));
    dt.Columns.Add("TotalAmount", typeof(decimal));
    dt.Columns.Add("Status", typeof(string));
    dt.Columns.Add("CreatedAt", typeof(DateTime));

    return dt;
}
\\\

### Populating the DataTable

\\\csharp
var dt = BuildOrderDataTable();

for (int i = 0; i < 10_000; i++)
{
    dt.Rows.Add(
        Guid.NewGuid(),
        customerId,
        DateTime.UtcNow.AddDays(-Random.Shared.Next(0, 365)),
        Random.Shared.Next(100, 5000) + 0.99m,
        "Pending",
        DateTime.UtcNow
    );
}
\\\

### Schema matching — critical!

SqlBulkCopy matches columns **by ordinal position or by name** (with \ColumnMappings\). If the ordinal order doesn't match, or column names differ, you get a runtime error or wrong data.

\\\csharp
// Option 1: Column names must match the target table exactly
// DataTable column "Id" → SQL column "Id"
// DataTable column "CustomerId" → SQL column "CustomerId"

// Option 2: Use ColumnMappings when names differ
bulkCopy.ColumnMappings.Add("Id", "OrderId");
bulkCopy.ColumnMappings.Add("CustomerId", "CustomerId");
bulkCopy.ColumnMappings.Add("OrderDate", "OrderDate");
bulkCopy.ColumnMappings.Add("Total", "TotalAmount"); // different name
bulkCopy.ColumnMappings.Add("Status", "Status");
\\\

### Automatic mapping with reflection

\\\csharp
public static DataTable ToDataTable<T>(this IEnumerable<T> items)
{
    var dt = new DataTable();
    var props = typeof(T).GetProperties();

    foreach (var prop in props)
    {
        dt.Columns.Add(prop.Name, Nullable.GetUnderlyingType(prop.PropertyType) ?? prop.PropertyType);
    }

    foreach (var item in items)
    {
        var row = dt.NewRow();
        foreach (var prop in props)
        {
            row[prop.Name] = prop.GetValue(item) ?? DBNull.Value;
        }
        dt.Rows.Add(row);
    }

    return dt;
}

// Usage
var orders = GetOrders(10000);
var dt = orders.ToDataTable();

using var bulk = new SqlBulkCopy(connectionString);
bulk.DestinationTableName = "Orders";
await bulk.WriteToServerAsync(dt);
\\\

### Performance note on DataTable

\DataTable\ materializes all rows in memory. For 100K+ rows, prefer \IDataReader\ streaming to avoid high memory pressure.

## 4 — SqlBulkCopy with DataTable

### Full example — DataTable to SqlBulkCopy

\\\csharp
using Microsoft.Data.SqlClient;

public class BulkInsertService
{
    private readonly string _connectionString;

    public BulkInsertService(string connectionString)
    {
        _connectionString = connectionString;
    }

    public async Task BulkInsertOrdersAsync(List<Order> orders)
    {
        var dt = orders.ToDataTable();

        using var conn = new SqlConnection(_connectionString);
        await conn.OpenAsync();

        using var bulk = new SqlBulkCopy(conn)
        {
            DestinationTableName = "Orders",
            BatchSize = 5000,
            BulkCopyTimeout = 120,
            EnableStreaming = false // DataTable — no streaming benefit
        };

        // Optional: column mappings if names differ
        bulk.ColumnMappings.Add("Id", "Id");
        bulk.ColumnMappings.Add("CustomerId", "CustomerId");
        bulk.ColumnMappings.Add("OrderDate", "OrderDate");
        bulk.ColumnMappings.Add("TotalAmount", "TotalAmount");
        bulk.ColumnMappings.Add("Status", "Status");

        await bulk.WriteToServerAsync(dt);
    }
}

public record Order
{
    public Guid Id { get; init; }
    public Guid CustomerId { get; init; }
    public DateTime OrderDate { get; init; }
    public decimal TotalAmount { get; init; }
    public string Status { get; init; } = "Pending";
    public DateTime CreatedAt { get; init; } = DateTime.UtcNow;
}
\\\

### With progress notification

\\\csharp
using var bulk = new SqlBulkCopy(connectionString);
bulk.DestinationTableName = "Orders";
bulk.NotifyAfter = 1000;
bulk.SqlRowsCopied += (sender, args) =>
{
    Console.WriteLine($"Copied {args.RowsCopied} rows...");
};
await bulk.WriteToServerAsync(dt);
// Output:
// Copied 1000 rows...
// Copied 2000 rows...
// ...
\\\

### With transaction

\\\csharp
using var conn = new SqlConnection(_connectionString);
await conn.OpenAsync();
using var tx = conn.BeginTransaction();

using var bulk = new SqlBulkCopy(conn, SqlBulkCopyOptions.Default, tx);
bulk.DestinationTableName = "Orders";
await bulk.WriteToServerAsync(dt);

tx.Commit(); // or tx.Rollback()
\\\

### SQL that runs on the server

\\\sql
-- SqlBulkCopy generates and executes a BULK INSERT operation internally.
-- If you enable SQL Server Profiler, you'll see:
BULK INSERT [Orders] FROM '...' WITH (DATAFILETYPE = 'native');
-- Or equivalently, rows are written via TDS INSERT BULK protocol:
INSERT BULK [Orders] (
    [Id]           UniqueIdentifier,
    [CustomerId]   UniqueIdentifier,
    [OrderDate]    DateTime2,
    [TotalAmount]  Decimal(18,2),
    [Status]       NVarchar(50)
)
-- No individual INSERT statements are generated.

## 5 — SqlBulkCopy with IDataReader (from Dapper)

Using \IDataReader\ avoids materializing all rows in a \DataTable\. You can stream directly from a Dapper query or from any \DbDataReader\.

### Option A: Copy from one DB to another via Dapper

\\\csharp
using Microsoft.Data.SqlClient;
using Dapper;

public async Task CopyOrdersAsync(
    string sourceConnectionString,
    string destConnectionString,
    DateOnly cutoffDate)
{
    await using var sourceConn = new SqlConnection(sourceConnectionString);
    await sourceConn.OpenAsync();

    // Dapper ExecuteReaderAsync returns a DbDataReader (implements IDataReader)
    await using var reader = await sourceConn.ExecuteReaderAsync(
        "SELECT Id, CustomerId, OrderDate, TotalAmount, Status FROM Orders WHERE OrderDate >= @Cutoff",
        new { Cutoff = cutoffDate.ToDateTime(TimeOnly.MinValue) });

    await using var destConn = new SqlConnection(destConnectionString);
    await destConn.OpenAsync();

    using var bulk = new SqlBulkCopy(destConn)
    {
        DestinationTableName = "Orders",
        BatchSize = 10000,
        EnableStreaming = true // Important for IDataReader!
    };

    // Column mappings match the SELECT column order
    await bulk.WriteToServerAsync(reader);
}
\\\

### Option B: Custom IDataReader from in-memory list

You can implement \IDataReader\ yourself or use a library like \FastMember\'s \ObjectReader\:

\\\csharp
using FastMember;

var orders = new List<Order> { /* ... 100K rows ... */ };
using var reader = ObjectReader.Create(orders, "Id", "CustomerId", "OrderDate", "TotalAmount", "Status");
// FastMember.ObjectReader implements IDataReader over IEnumerable<T>

using var bulk = new SqlBulkCopy(connectionString);
bulk.DestinationTableName = "Orders";
await bulk.WriteToServerAsync(reader);
\\\

### Option C: Streaming from a CSV via IDataReader

\\\csharp
public class CsvDataReader : IDataReader
{
    private readonly StreamReader _reader;
    private readonly string[] _headers;
    private string[]? _currentRow;
    private bool _disposed;

    public CsvDataReader(string filePath)
    {
        _reader = new StreamReader(filePath);
        _headers = _reader.ReadLine()!.Split(',');
    }

    public int FieldCount => _headers.Length;
    public string GetName(int i) => _headers[i];
    public Type GetFieldType(int i) => typeof(string);
    public int GetOrdinal(string name) => Array.IndexOf(_headers, name);

    public bool Read()
    {
        var line = _reader.ReadLine();
        if (line == null) return false;
        _currentRow = line.Split(',');
        return true;
    }

    public object GetValue(int i) => _currentRow![i];

    public void Dispose()
    {
        if (_disposed) return;
        _disposed = true;
        _reader.Dispose();
    }

    public void Close() => Dispose();
    public bool IsClosed => _disposed;
    public int Depth => 0;
    public long RecordsAffected => -1;
}

// Usage
using var csvReader = new CsvDataReader("orders.csv");
using var bulk = new SqlBulkCopy(connectionString);
bulk.DestinationTableName = "Orders";
await bulk.WriteToServerAsync(csvReader);
\\\

### IDataReader vs DataTable memory comparison

| Aspect | DataTable | IDataReader |
|---|---|---|
| Memory | All rows buffered in memory | Streamed, 1 row at a time |
| 100K rows | ~200–500 MB | ~1 MB |
| Random access | Yes (by index) | No (forward-only) |
| Ease of use | Simple | Requires reader implementation |
| Best for | Small batches (< 10K) | Large batches (10K+) |

## 6 — SqlBulkCopy from EF Core (DataTable Bridge)

EF Core does not natively support \SqlBulkCopy\. You bridge by materializing entities and writing them to a \DataTable\ or by using \FromSqlRaw\ to get an \IDataReader\.

### Approach 1: Entities → DataTable → SqlBulkCopy

\\\csharp
using Microsoft.EntityFrameworkCore;

public async Task BulkInsertFromEfCoreAsync(AppDbContext db, List<Order> orders)
{
    var dt = orders.ToDataTable();

    var connectionString = db.Database.GetConnectionString();
    await using var conn = new SqlConnection(connectionString);
    await conn.OpenAsync();

    using var bulk = new SqlBulkCopy(conn);
    bulk.DestinationTableName = "Orders";
    await bulk.WriteToServerAsync(dt);
}
\\\

### Approach 2: Copy from one DB to another using EF Core query

\\\csharp
public async Task CopyOrdersViaEfCoreAsync(AppDbContext sourceDb, string destConnectionString)
{
    var orders = await sourceDb.Orders
        .Where(o => o.OrderDate >= DateTime.UtcNow.AddDays(-7))
        .AsNoTracking()
        .ToListAsync();

    var dt = orders.ToDataTable();

    await using var conn = new SqlConnection(destConnectionString);
    await conn.OpenAsync();
    using var bulk = new SqlBulkCopy(conn);
    bulk.DestinationTableName = "Orders";
    await bulk.WriteToServerAsync(dt);
}
\\\

### Approach 3: FromSqlRaw → IDataReader → SqlBulkCopy (no materialization)

\\\csharp
public async Task CopyDirectlyAsync(AppDbContext db, string destConnectionString)
{
    var connection = db.Database.GetDbConnection();
    await connection.OpenAsync();

    await using var command = connection.CreateCommand();
    command.CommandText = "SELECT Id, CustomerId, OrderDate, TotalAmount, Status FROM Orders WHERE OrderDate >= @Cutoff";
    command.Parameters.Add(new SqlParameter("@Cutoff", DateTime.UtcNow.AddDays(-7)));

    await using var reader = await command.ExecuteReaderAsync();

    await using var destConn = new SqlConnection(destConnectionString);
    await destConn.OpenAsync();
    using var bulk = new SqlBulkCopy(destConn);
    bulk.DestinationTableName = "Orders";
    await bulk.WriteToServerAsync(reader);
}
\\\

### Approach 4: Extension method on DbContext

\\\csharp
public static class DbContextBulkExtensions
{
    public static async Task BulkInsertAsync<T>(
        this DbContext context,
        IReadOnlyCollection<T> entities,
        string? tableName = null) where T : class
    {
        if (entities.Count == 0) return;

        var entityType = context.Model.FindEntityType(typeof(T))!;
        tableName ??= entityType.GetTableName();

        var properties = entityType.GetProperties()
            .Where(p => !p.IsShadowProperty() || p.Name != "Id")
            .ToList();

        var dt = new DataTable();
        foreach (var prop in properties)
        {
            dt.Columns.Add(prop.GetColumnName(), prop.ClrType);
        }

        foreach (var entity in entities)
        {
            var row = dt.NewRow();
            foreach (var prop in properties)
            {
                row[prop.GetColumnName()] = prop.PropertyInfo?.GetValue(entity) ?? DBNull.Value;
            }
            dt.Rows.Add(row);
        }

        var connStr = context.Database.GetConnectionString();
        await using var conn = new SqlConnection(connStr);
        await conn.OpenAsync();
        using var bulk = new SqlBulkCopy(conn);
        bulk.DestinationTableName = tableName;
        await bulk.WriteToServerAsync(dt);
    }
}

// Usage
await db.BulkInsertAsync(orders);
\\\

## 9 — Gotchas and Best Practices

### Gotcha 1: Column names must match (or be mapped)

SqlBulkCopy matches columns **by ordinal position** if no \ColumnMappings\ are specified. If the DataTable has columns in a different order than the target table, data goes into wrong columns.

\\\csharp
// DataTable columns: [Id, Name, Amount]
// SQL columns:       [Id, Amount, Name]
// Without mapping, Amount from DataTable goes into Name column!

// Fix: always use ColumnMappings
bulk.ColumnMappings.Add("Id", "Id");
bulk.ColumnMappings.Add("Name", "Name");
bulk.ColumnMappings.Add("Amount", "Amount");
\\\

### Gotcha 2: Constraints are NOT checked by default

SqlBulkCopy does **not** validate foreign keys, check constraints, or fire triggers unless you explicitly enable them:

\\\csharp
// CHECK_CONSTRAINTS — validates check constraints
var options = SqlBulkCopyOptions.CheckConstraints;
using var bulk = new SqlBulkCopy(conn, options, tx);

// FIRE_TRIGGERS — fires INSERT triggers
var options = SqlBulkCopyOptions.FireTriggers;
using var bulk = new SqlBulkCopy(conn, options, tx);
\\\

\\\sql
-- Equivalent T-SQL with constraints and triggers:
BULK INSERT Orders FROM '...'
WITH (
    CHECK_CONSTRAINTS,
    FIRE_TRIGGERS,
    DATAFILETYPE = 'native'
);
\\\

Both options slow down the bulk insert significantly. Only enable them if necessary.

### Gotcha 3: Lock escalation

A bulk insert can escalate from row/page locks to a table lock, blocking other sessions.

\\\csharp
// Mitigation: use batch size
bulk.BatchSize = 1000; // Forces a commit (on default transaction) every 1000 rows
\\\

\\\sql
-- Or use TABLOCK hint:
BULK INSERT Orders FROM '...' WITH (TABLOCK);
-- TABLOCK enables minimal logging but blocks concurrent access
\\\

\\\csharp
// In SqlBulkCopy, TableLock option:
var options = SqlBulkCopyOptions.TableLock;
using var bulk = new SqlBulkCopy(conn, options, tx);
\\\

### Gotcha 4: KeepIdentity — preserve identity values

By default, SqlBulkCopy ignores identity values and generates new ones.

\\\csharp
// To preserve identity values from source:
var options = SqlBulkCopyOptions.KeepIdentity;
using var bulk = new SqlBulkCopy(conn, options, tx);
\\\

\\\sql
-- Equivalent: SET IDENTITY_INSERT [Orders] ON;
-- Then BULK INSERT with KEEPIDENTITY:
BULK INSERT Orders FROM '...' WITH (KEEPIDENTITY);
\\\

This is essential when copying data between databases or migrating data.

### Gotcha 5: Minimal logging prerequisites

For minimal logging with SqlBulkCopy:

\\\sql
-- 1. Target table is a HEAP (no clustered index) OR
-- 2. Recovery model is SIMPLE or BULK_LOGGED
-- 3. TABLOCK hint is used (or TableLock option)

-- Check recovery model:
SELECT name, recovery_model_desc FROM sys.databases WHERE name = 'YourDb';

-- Set to bulk-logged for import:
ALTER DATABASE YourDb SET RECOVERY BULK_LOGGED;
\\\

Without minimal logging, the bulk operation is fully logged and can fill the transaction log.

### Gotcha 6: FireTriggers — unexpected behavior

When \FireTriggers\ is enabled, triggers fire **once per batch**, not once per row. This can cause logic issues if your trigger expects row-by-row processing.

### Gotcha 7: Timeout with large batches

\\\csharp
// Default timeout is 30 seconds — increase for large imports
bulk.BulkCopyTimeout = 0; // 0 = no timeout (not recommended for production)
bulk.BulkCopyTimeout = 600; // 10 minutes
\\\

### Gotcha 8: Data type mismatches

SqlBulkCopy is strict about data types. A \decimal(18,2)\ column will reject a value with more decimal places. A \string\ column with length > \NVARCHAR(50)\ will truncate or fail.

\\\csharp
// Fix: ensure DataTable column types match SQL types exactly
// Use SqlMetaData to define precise schema:
var columnMetadata = new[]
{
    new SqlMetaData("Id", SqlDbType.UniqueIdentifier),
    new SqlMetaData("TotalAmount", SqlDbType.Decimal, 18, 2),
    // ...
};
\\\

### Gotcha 9: SqlBulkCopy in a transaction

SqlBulkCopy participates in transactions. If the transaction rolls back, the bulk insert is undone. However, each \BatchSize\ chunk is internally committed within the transaction — so a rollback after partial batches is still possible.

\\\csharp
using var conn = new SqlConnection(ConnString);
await conn.OpenAsync();
using var tx = conn.BeginTransaction();

try
{
    using var bulk = new SqlBulkCopy(conn, SqlBulkCopyOptions.Default, tx);
    bulk.DestinationTableName = "Orders";
    bulk.BatchSize = 5000;
    await bulk.WriteToServerAsync(dt);

    // If WriteToServerAsync fails mid-way, the tx can still rollback entirely
    tx.Commit();
}
catch
{
    tx.Rollback();
    throw;
}
\\\

### Gotcha 10: Dapper ExecuteReaderAsync — connection must stay open

When using Dapper's \ExecuteReaderAsync\ with SqlBulkCopy, the source connection must remain open for the duration of \WriteToServerAsync\:

\\\csharp
// CORRECT — both connections open simultaneously
await using var sourceConn = new SqlConnection(sourceCs);
await using var destConn = new SqlConnection(destCs);
await sourceConn.OpenAsync();
await destConn.OpenAsync();

await using var reader = await sourceConn.ExecuteReaderAsync(sql);
using var bulk = new SqlBulkCopy(destConn);
await bulk.WriteToServerAsync(reader);
// Both connections are open during the entire transfer
\\\

### Best practices checklist

| # | Practice | Why |
|---|---|
| 1 | Use \ColumnMappings\ even when names match | Self-documenting, prevents silent data corruption |
| 2 | Use \IDataReader\ for > 10K rows | Avoids materializing all rows in memory |
| 3 | Enable \EnableStreaming\ with \IDataReader\ | Streams without buffering |
| 4 | Set \BatchSize\ to 1000–5000 | Avoids lock escalation |
| 5 | Increase \BulkCopyTimeout\ for large imports | Prevents timeout errors |
| 6 | Use \TableLock\ for single-user import | Enables minimal logging |
| 7 | Validate data before bulk insert | SqlBulkCopy doesn't validate constraints by default |
| 8 | Wrap in explicit transaction | Enables full rollback on failure |
| 9 | Test with production-sized data in staging | Benchmarks vary by schema, indexes, and hardware |
| 10 | Monitor transaction log size | Bulk inserts can grow the log even with minimal logging |

### Summary

\\\
SqlBulkCopy is the fastest way to insert large volumes of data into SQL Server from .NET.

Key facts:
  • Uses TDS bulk load protocol — bypasses query engine
  • Minimal logging with heap + TABLOCK + SIMPLE/BULK_LOGGED
  ~45x faster than individual INSERTs for 10K rows
  • Column mappings required for schema matching
  • Constraints and triggers OFF by default
  • Works with DataTable, DataRow[], or IDataReader

Use with Dapper:
  • ExecuteReaderAsync → IDataReader → SqlBulkCopy
  • FastMember ObjectReader for IEnumerable<T>

Use with EF Core:
  • Bridge via DataTable (entities → dt → bulk copy)
  • Or FromSqlRaw → IDataReader
  • Better: use Dapper connection alongside EF Core
