### Where This Fits

The DISTINCT clause removes duplicate rows from the result set by comparing every column in the SELECT list. Every .NET backend engineer encounters it in reporting queries, dropdown population, API filters, and data cleaning operations. The critical insight is that DISTINCT is not free — it is a blocking operator that must either sort all rows (O(N log N)) or hash all rows (O(N) with O(unique) memory) to identify duplicates. The most expensive mistake in production is using DISTINCT as a band-aid for a missing JOIN predicate or an unintended CROSS JOIN: the join multiplies rows by orders of magnitude and DISTINCT then tries to unsort the mess — a query that should return 500 rows in 40ms instead processes 250B rows and takes 12 seconds. Interviewers probe this topic to determine whether a candidate understands the three execution strategies (Stream Aggregate, Hash Match Aggregate, Sort + Distinct Sort), when DISTINCT is the right tool vs GROUP BY or EXISTS, and how to verify the cost using SET STATISTICS IO and the execution plan. Engineers who know this topic can predict the plan operator from the cardinality ratio and spot a DISTINCT band-aid just by reading the plan.

## Core Mental Model

The DISTINCT clause is the deduplication operator: it removes rows that are identical across every column in the SELECT list. Logically, DISTINCT executes after SELECT (step 6) and before ORDER BY (step 7) and TOP (step 8). The SQL Server query engine implements DISTINCT using one of three strategies chosen by the optimizer based on input sortedness and cardinality estimates. **Stream Aggregate** is the cheapest — when the input is already sorted by all SELECT columns (typically from an index scan/seek), the operator scans sequentially and emits a row only when the current row differs from the previous row; zero additional memory, O(N) comparisons. **Hash Match Aggregate (Distinct)** builds a hash table of unique row values in memory — O(N) hashing operations with O(unique_rows) memory; ideal when the number of distinct values is small relative to the total rows (e.g., 3 statuses out of 1M orders). **Sort + Distinct Sort** sorts the entire input by all SELECT columns then applies a Stream Aggregate on the sorted output — O(N log N) with a memory grant proportional to total rows; used when unique values are numerous (e.g., 800K distinct customer IDs out of 1M orders). NULLs are considered equal for deduplication: two rows that are both NULL in the same column are treated as duplicates and one is eliminated. The single most important performance rule: **DISTINCT on a column that already has an index is nearly free** (Stream Aggregate via ordered scan); DISTINCT without a supporting index forces a Sort or Hash that can dominate the query cost.

### Classification

This is a **deduplication operator** in the `SELECT` clause family. DISTINCT is semantically equivalent to `GROUP BY <all SELECT columns>` without aggregate functions — both produce the same execution plan for deduplication. There is no "DISTINCT on one column" in standard SQL: `SELECT DISTINCT col1, col2` deduplicates on the **pair** (col1, col2), not on col1 alone. PostgreSQL extends this with `DISTINCT ON (col1) col1, col2` which keeps one row per distinct col1 value.

```mermaid
flowchart TD
    A[SELECT DISTINCT received] --> B{Input already sorted by all SELECT columns?}
    B -->|Yes — from index seek/scan matching column list| C[Stream Aggregate — O(N), zero additional memory]
    B -->|No| D{Estimated unique rows << total rows?}
    D -->|Yes — few unique values, many duplicates| E[Hash Match Aggregate — O(N), O(unique) memory]
    D -->|No — unique count approaches total rows| F[Sort + Distinct Sort — O(N log N), O(total) memory]
    C --> G[Index scan in sort order + adjacent comparison]
    E --> H[Build hash table of unique values, discard duplicates]
    F --> I[Sort all rows by SELECT columns, then adjacent comparison]
    G --> J[Optimal: near-zero CPU beyond I/O]
    H --> J
    I --> J
```

## Deep Mechanics

### How the Engine Executes This

1. **Parsing** — The parser identifies DISTINCT in the SELECT clause. If both DISTINCT and GROUP BY appear in the same query, T-SQL raises error 156 — they are semantically incompatible because GROUP BY already deduplicates.

2. **Binding** — The algebrizer resolves every column reference in the SELECT list. DISTINCT applies to the entire projected row — every column is part of the deduplication key. If `SELECT DISTINCT *` is used, all columns of all tables participate, including IDENTITY columns, timestamps, and LOBs.

3. **Optimization** — The optimizer evaluates the three strategies. The decision tree:
   - **Is the input already sorted by all SELECT columns?** If an index provides ordered access on exactly the DISTINCT columns, the optimizer chooses Stream Aggregate (zero-cost dedup).
   - **If not sorted: what is the estimated number of unique rows vs total rows?** The optimizer uses the statistics histogram for each column to estimate distinct count. If the product of column-level distinct estimates is small (roughly < 30% of total rows), it chooses Hash Match Aggregate.
   - **Otherwise:** Sort + Distinct Sort is chosen. The sort is a blocking operator that requires a memory grant proportional to estimated row size × row count.

4. **Execution** — The chosen strategy executes. Stream Aggregate: maintain a "current row" buffer; for each incoming row, compare against the buffer; if different, emit the buffer and replace it. Hash Match: hash each row by its column values; probe the hash table; if not found, insert. Sort: sort all rows in TempDB if necessary, then Stream Aggregate.

5. **Output** — Deduplicated rows pass to the next operator (ORDER BY, TOP, or SELECT output).

### Execution Plan Analysis

**Stream Aggregate — input already sorted by index:**

- Index: `IX_Orders_Status` on `(Status)`
- Query: `SELECT DISTINCT o.Status FROM dbo.Orders AS o`
- Plan: `[Index Scan IX_Orders_Status] → [Stream Aggregate] → [SELECT]`
- The index returns rows in Status order (e.g., Cancelled, Delivered, Pending, Shipped). The Stream Aggregate keeps a buffer of the current Status; when the value changes, it emits the previous one. No sort, no hash.
- Logical Reads: ~22 (narrow index leaf pages)
- Estimated Cost: ~0.05 — essentially free beyond the index scan

**Hash Match Aggregate — few unique values, unsorted input:**

- Query: `SELECT DISTINCT o.Status FROM dbo.Orders AS o` (no index on Status)
- Plan: `[Clustered Index Scan] → [Hash Match Aggregate (Distinct)] → [SELECT]`
- The Hash Match builds a hash table with 3 entries (for 3 unique statuses). Each of the 1M rows is hashed, the hash table is probed. 3 entries × ~40 bytes = ~120 bytes of memory.
- Logical Reads: ~11,400 (full clustered index scan)
- Estimated Cost: ~2.8 — dominated by the scan, not the aggregation

**Sort + Distinct Sort — many unique values, unsorted input:**

- Query: `SELECT DISTINCT o.CustomerId FROM dbo.Orders AS o` (800K unique, 1M total)
- Plan: `[Clustered Index Scan] → [Sort] → [Stream Aggregate (Distinct Sort)] → [SELECT]`
- The Sort orders all 1M rows by CustomerId (O(N log N)). Then the Stream Aggregate scans the sorted output, removing adjacent duplicate CustomerIds.
- Logical Reads: ~11,400 (scan) + Sort (CPU/memory, no I/O)
- Memory Grant: ~30 MB
- Estimated Cost: ~6.3 — Sort dominates (~3.5 of 6.3)

```

[Index Scan IX_Orders_Status] → [Stream Aggregate] → [SELECT]
Logical Reads: ~22  |  Memory: 0  |  CPU: ~3ms  |  Cost: 0.05

[Clustered Index Scan] → [Hash Match Aggregate] → [SELECT]
Logical Reads: ~11,400  |  Memory: ~120 B  |  CPU: ~451ms  |  Cost: 2.8

Sort + Distinct Sort (many unique, no index):
[Clustered Index Scan] → [Sort] → [Stream Aggregate] → [SELECT]
Logical Reads: ~11,400  |  Memory: ~30 MB  |  CPU: ~1,820ms  |  Cost: 6.3
```

### Failure Modes

**DISTINCT as band-aid for missing join predicate:** The worst DISTINCT anti-pattern. A missing or incorrect JOIN ON condition causes row multiplication (often a Cartesian product or unintended 1:N multiplication). The developer adds DISTINCT to "fix" the duplicate rows instead of correcting the join. The deduplication operator must then process the multiplied rows — exponentially more expensive than fixing the join.

Detect with:

```sql
-- Find DISTINCT queries with high row-processing ratios
SELECT TOP 20
    qs.total_logical_reads / NULLIF(qs.execution_count, 0) AS avg_logical_reads,
    qs.total_worker_time / NULLIF(qs.execution_count, 0) AS avg_cpu,
    qs.execution_count,
    SUBSTRING(st.text, 1, 500) AS query_text,
    qp.query_plan
FROM sys.dm_exec_query_stats AS qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) AS st
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) AS qp
WHERE st.text LIKE '%DISTINCT%'
  AND qs.total_logical_reads / NULLIF(qs.execution_count, 0) > 50000
ORDER BY avg_logical_reads DESC;
```

**DISTINCT on LOB columns:** Deduplication on NVARCHAR(MAX), VARCHAR(MAX), or VARBINARY(MAX) forces the engine to compare large off-row values. Hash-based DISTINCT must hash the entire LOB value; Sort-based DISTINCT must sort by LOB values. Both are expensive because LOB comparison does not short-circuit on prefix — the engine reads the full value.

**DISTINCT with ORDER BY on non-selected columns:** SQL Server requires that ORDER BY columns appear in the SELECT list when DISTINCT is specified. `SELECT DISTINCT col1 FROM t ORDER BY col2` fails with `Msg 145`. This is because DISTINCT removes rows, and the ORDER BY column may have multiple values for a deduplicated row — ordering would be ambiguous.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Schema context
-- ============================================================
CREATE TABLE dbo.Orders
(
    OrderId      INT           NOT NULL IDENTITY(1,1),
    CustomerId   INT           NOT NULL,
    OrderDate    DATETIME2(0)  NOT NULL,
    Status       VARCHAR(20)   NOT NULL,
    TotalAmount  DECIMAL(18,2) NOT NULL,
    ShippingAddr NVARCHAR(500) NULL,
    Notes        NVARCHAR(MAX) NULL,
    CreatedAt    DATETIME2(0)  NOT NULL DEFAULT SYSUTCDATETIME(),
    CONSTRAINT PK_Orders PRIMARY KEY CLUSTERED (OrderId)
);

-- Indexes to enable Stream Aggregate for common DISTINCT queries
CREATE NONCLUSTERED INDEX IX_Orders_Status ON dbo.Orders (Status);
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId ON dbo.Orders (CustomerId);

-- ============================================================
-- Pattern 1: DISTINCT on a single column — dropdown population
-- ============================================================
-- Returns unique status values (e.g., 'Pending', 'Shipped', 'Delivered')
-- With IX_Orders_Status: Stream Aggregate, ~22 logical reads
SELECT DISTINCT o.Status
FROM dbo.Orders AS o;

-- ============================================================
-- Pattern 2: DISTINCT with WHERE — filter then deduplicate
-- ============================================================
-- Returns unique customers who have orders in a specific status
SELECT DISTINCT o.CustomerId
FROM dbo.Orders AS o
WHERE o.Status = 'Shipped';

-- ============================================================
-- Pattern 3: DISTINCT on multiple columns — composite dedup
-- ============================================================
-- Returns unique (CustomerId, Status) pairs
-- Equivalent to GROUP BY CustomerId, Status
SELECT DISTINCT o.CustomerId, o.Status
FROM dbo.Orders AS o;

-- ============================================================
-- Pattern 4: DISTINCT with aggregation via GROUP BY alternative
-- ============================================================
-- DISTINCT is semantically identical to GROUP BY without aggregates
-- Both produce the same execution plan:
SELECT DISTINCT o.CustomerId, o.Status FROM dbo.Orders AS o;
SELECT o.CustomerId, o.Status FROM dbo.Orders AS o GROUP BY o.CustomerId, o.Status;

-- ============================================================
-- Pattern 5: DISTINCT with ORDER BY — must include ORDER BY columns in SELECT
-- ============================================================
SELECT DISTINCT o.CustomerId, o.Status, o.OrderDate
FROM dbo.Orders AS o
ORDER BY o.OrderDate DESC;

-- ============================================================
-- Pattern 6: EXISTS instead of DISTINCT for existence checks (more efficient)
-- ============================================================
-- ❌ DISTINCT on join result — processes all joined rows
SELECT DISTINCT c.CustomerId, c.FirstName, c.LastName
FROM dbo.Customers AS c
INNER JOIN dbo.Orders AS o ON c.CustomerId = o.CustomerId
WHERE o.OrderDate >= '2024-01-01';

-- ✅ EXISTS — short-circuits per customer (no join multiplication)
SELECT c.CustomerId, c.FirstName, c.LastName
FROM dbo.Customers AS c
WHERE EXISTS (
    SELECT 1 FROM dbo.Orders AS o
    WHERE o.CustomerId = c.CustomerId
      AND o.OrderDate >= '2024-01-01'
);

-- ============================================================
-- Anti-pattern 1: DISTINCT used to mask a missing join predicate
-- ============================================================
-- ❌ ON 1=1 creates a cross join; DISTINCT tries to undo it
SELECT DISTINCT o.OrderId, o.TotalAmount, p.PaymentMethod
FROM dbo.Orders AS o
INNER JOIN dbo.Payments AS p ON o.OrderId = p.OrderId;

-- ============================================================
-- Anti-pattern 2: DISTINCT * on a table with a unique column
-- ============================================================
-- ❌ OrderId is IDENTITY — every row is already unique, DISTINCT is a no-op
SELECT DISTINCT * FROM dbo.Orders;

-- ============================================================
-- Anti-pattern 3: DISTINCT on LOB columns
-- ============================================================
-- ❌ Notes is NVARCHAR(MAX) — forces LOB comparison
SELECT DISTINCT o.Notes FROM dbo.Orders AS o;
```

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL: standard DISTINCT — identical syntax
SELECT DISTINCT status FROM orders;

-- PostgreSQL: DISTINCT ON (non-standard, extremely useful)
-- Returns the "first" row per distinct value in the specified column(s)
-- ORDER BY controls WHICH row is "first" for each group
SELECT DISTINCT ON (customer_id)
    order_id,
    customer_id,
    total_amount,
    order_date
FROM orders
ORDER BY customer_id, order_date DESC;
-- Returns the most recent order for each customer
-- Only one row per customer_id

-- SQL Server does NOT support DISTINCT ON
-- SQL Server equivalent: ROW_NUMBER() with PARTITION BY
SELECT OrderId, CustomerId, TotalAmount, OrderDate
FROM (
    SELECT *,
        ROW_NUMBER() OVER (PARTITION BY CustomerId ORDER BY OrderDate DESC) AS rn
    FROM dbo.Orders
) AS sub
WHERE rn = 1;

-- PostgreSQL: NULL behavior is identical — NULL = NULL for dedup
-- PostgreSQL: ORDER BY columns must appear in SELECT with DISTINCT (same as SQL Server)

-- PostgreSQL: can use GROUPING SETS, CUBE, ROLLUP with DISTINCT
-- SQL Server: DISTINCT cannot be combined with GROUP BY (error 156)
```

## Gotchas and Production Pitfalls

### DISTINCT as a Band-Aid for a Missing Join Predicate

**Pitfall:** Writing a query that joins two tables without a proper ON condition (or with `ON 1=1`), producing a Cartesian product or unintended row multiplication. Adding DISTINCT to "fix" the duplicate rows instead of correcting the join.

```sql
-- ❌ ON 1=1 is a cross join — every Customer × every Order = 250B rows
-- DISTINCT then tries to collapse 250B rows back to 500K customers
SELECT DISTINCT c.CustomerId, c.FirstName, c.LastName
FROM dbo.Customers AS c
INNER JOIN dbo.Orders AS o ON 1=1
WHERE o.OrderDate >= '2024-01-01';
```

**Symptom:** The execution plan shows `[Nested Lops Cross Join]` followed by `[Sort]` or `[Hash Match Aggregate]` processing billions of rows. `SET STATISTICS IO` shows hundreds of thousands of logical reads (480,000+). The query takes 12+ seconds on a 2M row database. DMV `sys.dm_exec_query_stats` shows `total_spills` in the hundreds of MB because the Sort spills to TempDB.

**Fix:**

```sql
-- ✅ Fix the join predicate — then DISTINCT may not even be needed
SELECT DISTINCT c.CustomerId, c.FirstName, c.LastName
FROM dbo.Customers AS c
INNER JOIN dbo.Orders AS o ON c.CustomerId = o.CustomerId
WHERE o.OrderDate >= '2024-01-01';

-- ✅ Even better: use EXISTS — avoids the join entirely, short-circuits per customer
SELECT c.CustomerId, c.FirstName, c.LastName
FROM dbo.Customers AS c
WHERE EXISTS (
    SELECT 1 FROM dbo.Orders AS o
    WHERE o.CustomerId = c.CustomerId
      AND o.OrderDate >= '2024-01-01'
);
```

**Cost of not fixing:** On a 500K customer table with 500K matching orders, the cross join produces 250B intermediate rows. The Sort or Hash Aggregate must process those 250B rows — not feasible in any practical memory grant. The query spills to TempDB, writes gigabytes of data, consumes all I/O bandwidth, and takes 45+ seconds. Other queries block on TempDB contention. The application times out. Every hour this query runs, it causes a 1-minute production incident.

### DISTINCT * on a Table With a Unique Column

**Pitfall:** Using `SELECT DISTINCT *` on a table that has a naturally unique column (IDENTITY, GUID primary key, uniqueidentifier). Every row is already unique, so DISTINCT does nothing — but the engine still pays the full cost of deduplication.

```sql
-- ❌ OrderId is IDENTITY(1,1) — every row has a unique OrderId
-- DISTINCT * is a complete no-op but still sorts/hashes ALL columns
SELECT DISTINCT * FROM dbo.Orders;
```

**Symptom:** The execution plan shows `[Clustered Index Scan] → [Sort]` or `[Hash Match Aggregate]` on all columns including `OrderId`. If the table has an `NVARCHAR(MAX) Notes` column, the Sort must compare LOB values for every row — unnecessary. The query runs 10–50x slower than `SELECT *` with no DISTINCT, and returns identical results.

**Fix:**

```sql
-- ✅ Remove DISTINCT — the table already has unique rows
SELECT * FROM dbo.Orders;

-- ✅ If only a subset of columns needs deduplication, select them explicitly
SELECT DISTINCT o.CustomerId, o.Status FROM dbo.Orders AS o;
```

**Cost of not fixing:** 1M rows sorted by all columns including `Notes NVARCHAR(MAX) (avg 500 bytes)`. The sort requires ~500 MB memory grant, likely spills to TempDB, takes 30+ seconds — all for a result set that is identical to `SELECT *`. If this runs as part of a frequently-used view or stored procedure, it wastes server resources continuously.

### DISTINCT on LOB Columns

**Pitfall:** Running DISTINCT on NVARCHAR(MAX), VARCHAR(MAX), VARBINARY(MAX), or other large object columns. LOB values are stored off-row and require full-value comparison for deduplication — extremely expensive.

```sql
-- ❌ Notes is NVARCHAR(MAX) — forces LOB comparison for dedup
SELECT DISTINCT o.Notes FROM dbo.Orders AS o;
```

**Symptom:** The execution plan shows a `Hash Match Aggregate` or `Sort` on the LOB column. Memory grant spikes to hundreds of MB because the hash table stores LOB values or the sort needs LOB comparison space. Query may spill to TempDB. CPU time is dominated by LOB hashing or comparison logic. On a 1M row table with 500-byte average Notes, the hash table could reach 500 MB.

**Fix:**

```sql
-- ✅ Option 1: Use CHECKSUM for approximate deduplication (fast, 4-byte hash)
SELECT DISTINCT CHECKSUM(o.Notes) AS NotesHash
FROM dbo.Orders AS o;
-- ⚠️ CHECKSUM collisions are possible — different values can produce the same 4-byte checksum
-- Acceptable for approximate reporting, not for exact dedup

-- ✅ Option 2: Create a persisted computed column with a strong hash
ALTER TABLE dbo.Orders
    ADD NotesHash AS CAST(HASHBYTES('SHA2_256', Notes) AS BINARY(32)) PERSISTED;

CREATE NONCLUSTERED INDEX IX_Orders_NotesHash ON dbo.Orders (NotesHash);

-- Then query distinct Notes by joining on the hash
SELECT o.Notes
FROM dbo.Orders AS o
WHERE o.NotesHash IN (
    SELECT DISTINCT o2.NotesHash
    FROM dbo.Orders AS o2
    WHERE o2.Notes IS NOT NULL
);
-- BINARY(32) hash: 32 bytes per entry instead of 500 bytes — 15x memory reduction

-- ✅ Option 3: If exact dedup is not required, accept the cost and schedule during off-peak
```

**Cost of not fixing:** 1M rows with 500-byte average Notes — 500 MB of LOB data to hash or sort. Hash Match Aggregate on LOB values requires reading each LOB from its off-row storage into memory for hashing. At 500 MB hash table size, the memory grant (~25% of max server memory) is insufficient, causing repeated TempDB spills. A query that should take 2 seconds takes 5+ minutes.

### Client-Side DISTINCT After Loading All Rows

**Pitfall:** Loading all rows from the database into the application, then applying `.Distinct()` using LINQ-to-Objects. The database transfers every row over the network, and the application discards duplicates in memory.

```csharp
// ❌ Loads ALL 1M rows from database, transfers over network, then deduplicates
var allStatuses = await _dbContext.Orders
    .Select(o => o.Status)
    .ToListAsync(cancellationToken);
var distinctStatuses = allStatuses.Distinct(StringComparer.OrdinalIgnoreCase).ToList();
```

**Symptom:** The query returns 1M rows over the network (10+ MB of data) when only 3 unique values are needed. Application memory spikes as 1M strings are allocated. Response time is 2+ seconds for what should be a 5ms query. The database SQL log shows no DISTINCT keyword — the SELECT is unbounded.

**Fix:**

```csharp
// ✅ Database does the deduplication — only 3 rows transferred
var distinctStatuses = await _dbContext.Orders
    .Select(o => o.Status)
    .Distinct()
    .AsNoTracking()
    .ToListAsync(cancellationToken);
// Generated SQL: SELECT DISTINCT [o].[Status] FROM [Orders] AS [o]
```

**Cost of not fixing:** API endpoint "get distinct order statuses" returns 1M rows (10 MB) at 100 requests/minute = 1 GB/minute of unnecessary data transfer. Network bandwidth saturates at peak traffic. Application servers run out of memory. The database server is idle (it never receives the DISTINCT instruction) while the application does all the work. This is the most common EF Core DISTINCT mistake.

### Assuming DISTINCT Applies to a Single Column

**Pitfall:** Writing `SELECT DISTINCT CustomerId, Status` expecting one row per distinct CustomerId, with an arbitrary Status. In standard SQL, DISTINCT applies to the entire row — deduplication is on the pair (CustomerId, Status).

```sql
-- ❌ Developer expects: one row per CustomerId (1042 → 'Shipped')
-- Reality: one row per (CustomerId, Status) pair
-- If CustomerId 1042 has orders with 3 different statuses, 3 rows are returned
SELECT DISTINCT o.CustomerId, o.Status
FROM dbo.Orders AS o;
```

**Symptom:** A dropdown intended to show each customer once shows the same customer multiple times (once per status). Application logic that assumes a 1:1 mapping between customers and rows breaks silently. In a reporting context, aggregate counts are inflated because distinct rows are counted instead of distinct customers.

**Fix:**

```sql
-- ✅ Option 1: Select only the column you want distinct
SELECT DISTINCT o.CustomerId
FROM dbo.Orders AS o;

-- ✅ Option 2: Use GROUP BY with an aggregate if you need other columns
SELECT o.CustomerId, MAX(o.Status) AS AnyStatus
FROM dbo.Orders AS o
GROUP BY o.CustomerId;

-- ✅ Option 3: Use ROW_NUMBER() for per-group ordering (SQL Server)
SELECT CustomerId, Status
FROM (
    SELECT CustomerId, Status,
        ROW_NUMBER() OVER (PARTITION BY CustomerId ORDER BY OrderDate DESC) AS rn
    FROM dbo.Orders
) AS sub
WHERE rn = 1;

-- ✅ PostgreSQL: DISTINCT ON (non-standard)
SELECT DISTINCT ON (customer_id) customer_id, status
FROM orders
ORDER BY customer_id, order_date DESC;
```

**Cost of not fixing:** A customer dropdown intended for order assignment shows each customer 3–5 times (once per order status). Users cannot find customers. The application team adds client-side deduplication (wasting memory) or the UX team adds "ignore duplicates" logic (masking the real problem). In batch processing, order counts per customer are inflated by the number of distinct statuses — a 3x inflation that silently corrupts downstream analytics.

### ORDER BY With DISTINCT on Columns Not in SELECT

**Pitfall:** Writing `SELECT DISTINCT col1, col2 FROM t ORDER BY col3` — SQL Server rejects this with a compile-time error.

```sql
-- ❌ Msg 145: ORDER BY items must appear in the select list if SELECT DISTINCT is specified
SELECT DISTINCT o.CustomerId, o.Status
FROM dbo.Orders AS o
ORDER BY o.OrderDate DESC;
```

**Symptom:** `Msg 145, Level 15, State 1, Line X: ORDER BY items must appear in the select list if SELECT DISTINCT is specified.` The query fails to compile. This is because DISTINCT removes rows, and if the ORDER BY column (OrderDate) is not in the output, the engine cannot determine which of possibly multiple OrderDate values should determine the ordering for a deduplicated row.

**Fix:**

```sql
-- ✅ Option 1: Include the ORDER BY column in the SELECT
SELECT DISTINCT o.CustomerId, o.Status, o.OrderDate
FROM dbo.Orders AS o
ORDER BY o.OrderDate DESC;
-- Note: this changes deduplication semantics! Now dedup is on (CustomerId, Status, OrderDate)

-- ✅ Option 2: Use GROUP BY with an aggregate for ordering
SELECT o.CustomerId, o.Status
FROM dbo.Orders AS o
GROUP BY o.CustomerId, o.Status
ORDER BY MAX(o.OrderDate) DESC;
-- Now ordering is deterministic: by the most recent OrderDate per group

-- ✅ Option 3: Use a subquery to separate dedup from ordering
SELECT CustomerId, Status
FROM (
    SELECT DISTINCT CustomerId, Status
    FROM dbo.Orders
) AS sub
ORDER BY CustomerId;
```

**Cost of not fixing:** Compile-time error — the application crashes at startup or the stored procedure fails to create. A developer unfamiliar with the DISTINCT + ORDER BY restriction may "fix" it by adding all ORDER BY columns to the SELECT list, inadvertently changing the deduplication semantics and producing duplicate rows that should have been collapsed.

### Spoken Answers

**Q: What are the three execution strategies for DISTINCT in SQL Server, and what does the optimizer consider when choosing between them?**

> **Average answer:** The database either sorts the data or uses a hash. If there's an index, it can use that. The optimizer picks the fastest one.

> **Great answer:** SQL Server has three strategies, and the optimizer chooses based on sortedness and cardinality estimates. **Stream Aggregate** is the cheapest: when the input is already sorted by all SELECT columns — typically from an index scan/seek — the operator scans sequentially and emits a row only when the current row differs from the previous. This costs O(N) with zero additional memory, just N row comparisons. For example, a DISTINCT on `Status` with an index on `(Status)` scans the index leaf pages in status order and compares adjacent rows — it never needs to hold more than one row in memory. **Hash Match Aggregate** is chosen when the input is unsorted and the estimated number of unique rows is small relative to the total. It builds a hash table of unique values in memory. Each input row is hashed by its column values; if the hash is already in the table, the row is discarded. This is O(N) hashing operations with O(unique_rows) memory. For "distinct statuses from 1M orders" — 3 unique values out of 1M — the hash table is 3 entries, far cheaper than sorting 1M rows. **Sort + Distinct Sort** is chosen when unique values are numerous relative to the total. It sorts all rows by the SELECT columns, then applies a Stream Aggregate to remove adjacent duplicates. This is O(N log N) with a memory grant proportional to total row count. For "distinct CustomerIds from 1M orders" where 800K of 1M are unique, the optimizer picks Sort because the hash table would be nearly as large as the input itself. The decision threshold is roughly: if `estimated_unique_rows < total_rows / 3`, prefer Hash; otherwise prefer Sort. And if any index provides sorted input, Stream Aggregate always wins. I verify the choice by opening the execution plan — the physical operator name (`Stream Aggregate`, `Hash Match`, `Sort`) tells me immediately which strategy was chosen and why.

---

**Q: DISTINCT vs GROUP BY vs EXISTS: when would you choose each approach to eliminate duplicates?**

> **Average answer:** DISTINCT removes duplicates. GROUP BY groups things. EXISTS checks if something exists. You can use any of them for dedup.

> **Great answer:** These three are often interchangeable for simple deduplication, but each has a specific strength. **DISTINCT** is the most concise and readable for simple value lists. `SELECT DISTINCT Status FROM Orders` clearly communicates intent: "give me every status that exists, one row each." No aggregation, no subquery. I use this for dropdowns and filter lists. **GROUP BY** is the right choice when you need aggregation alongside deduplication — `SELECT CustomerId, COUNT(*) FROM Orders GROUP BY CustomerId` cannot be expressed with DISTINCT. GROUP BY also gives you more flexibility with ordering: `ORDER BY COUNT(*) DESC`. When I need dedup plus a summary metric, GROUP BY is the natural fit. **EXISTS** (or an anti-semi-join) is the right choice when you're deduplicating to filter rows in an outer query based on existence in a related table. For example, "find customers who have placed an order in the last 30 days." Using EXISTS avoids the row multiplication of a DISTINCT + JOIN entirely because the semi-join short-circuits per outer row: the engine scans the inner table (Orders) for each outer row (Customer) and stops as soon as it finds a match. The performance difference is significant at scale. `SELECT c.Id FROM Customers c WHERE EXISTS (SELECT 1 FROM Orders o WHERE o.CustomerId = c.CustomerId AND o.OrderDate > @date)` is more efficient than `SELECT DISTINCT c.Id FROM Customers c INNER JOIN Orders o ON ...` because the latter must JOIN all matching rows before deduplicating. On a table with 500K customers and 2M orders, the EXISTS version processes 500K semi-join probes (each short-circuits after the first match), while the DISTINCT + JOIN version joins 2M rows then deduplicates 500K. I choose EXISTS when the question is "does at least one row exist?" rather than "what are all the unique values?"

---

**Q: On a 1M row table with 3 unique statuses, what is the cost difference between DISTINCT with and without an index on that column?**

> **Average answer:** With an index, DISTINCT is faster. Without an index, it's slower.

> **Great answer:** The difference is dramatic because the execution strategy changes entirely. Without an index on `Status`, the optimizer has no sorted input and must choose between Hash Match and Sort. With only 3 unique values out of 1M rows, it chooses Hash Match Aggregate — build a 3-entry hash table. The plan is `[Clustered Index Scan] → [Hash Match Aggregate] → [SELECT]`. The scan reads all 11,400 pages of the clustered index (1M rows at 200 bytes/row), and each row is hashed into the 3-entry table. I measured 11,400 logical reads, 451 ms CPU, and ~468 ms elapsed time for this. With an index on `(Status)`, the plan changes to `[Index Scan IX_Orders_Status] → [Stream Aggregate] → [SELECT]`. The index scan reads only the narrow index leaf pages — approximately 22 logical reads. The Stream Aggregate compares each row's Status against the previous row's Status and emits a row when it changes. No sort, no hash, no memory grant. I measured 22 logical reads, 3 ms CPU, and 5 ms elapsed time. That's a 150x reduction in CPU and a 500x reduction in logical reads. The index is only 2–3 pages wide per 1,000 rows, adds negligible write overhead, and pays for itself if this query runs even a few hundred times per day.

### Interview Trigger

The most diagnostic DISTINCT question: "Your query returns duplicate rows. What do you do?" A junior candidate says "add DISTINCT." A senior candidate says "first I find out why duplicates are occurring — usually a missing join predicate or an unintended CROSS JOIN. Adding DISTINCT is the band-aid that masks the real problem but makes the query slower. I check the execution plan: if I see a Sort or Hash Aggregate processing many more rows than the largest table, the join is multiplying rows. I fix the join predicate instead." The follow-up: "You need a list of distinct customer cities where at least one order was placed in 2024. Write three versions: DISTINCT, GROUP BY, and EXISTS. Which is fastest and why?" The senior answer: "EXISTS is fastest because the semi-join short-circuits per customer; DISTINCT must join all orders then dedup the entire result; GROUP BY also joins all orders then groups. The EXISTS version uses a semi-join plan with a `Left Anti Semi Join` or `Right Semi Join` operator that stops scanning the inner table as soon as it finds a match per customer."

## Decision Framework

### When to Apply

```mermaid
flowchart TD
    A[Need to eliminate duplicate rows] --> B{Why are duplicates occurring?}
    B -->|Missing or incorrect join predicate| C[Fix the join - do NOT use DISTINCT]
    B -->|Legitimate need - data genuinely has duplicates| D{What is the use case?}
    D -->|Simple list of unique values for a dropdown or filter| E[SELECT DISTINCT - most concise]
    D -->|Need to filter outer query based on existence| F[EXISTS or semi-join - avoids row multiplication]
    D -->|Need aggregation alongside dedup (COUNT, SUM)| G[GROUP BY + aggregate functions]
    D -->|Need one row per group with specific ordering| H[ROW_NUMBER OVER PARTITION BY]
    E --> I{Number of unique rows vs total rows?}
    I -->|Few unique values (eg 3 of 1M)| J[Hash Match Aggregate expected - O(N), little memory]
    I -->|Many unique values (eg 800K of 1M)| K[Sort + Distinct Sort expected - O(N log N)]
    I -->|Index on SELECT columns exists| L[Stream Aggregate expected - optimal, zero extra cost]
    J --> M[Verify plan: no Sort operator present]
    K --> M
    L --> M
    F --> N[Verify plan: semi-join operator, no Sort]
    G --> O[Verify plan: aggregate operator]
```

### Application Checklist

- [ ] DISTINCT is NOT used to fix a missing join predicate — join condition is confirmed correct
- [ ] No `SELECT DISTINCT *` on tables with unique columns (IDENTITY, GUID) or LOB columns
- [ ] DISTINCT column(s) have a supporting index to enable Stream Aggregate (or index is planned)
- [ ] No LOB columns (NVARCHAR(MAX)) in DISTINCT — CHECKSUM or HASHBYTES alternative evaluated
- [ ] DISTINCT is executed in the database, not client-side (no `.ToListAsync()` before `.Distinct()`)
- [ ] EF Core `.Distinct()` is paired with `.Select()` for narrow column list — no full entity load then client-side dedup
- [ ] ORDER BY columns appear in the SELECT list when DISTINCT is used
- [ ] DISTINCT ON columns match the actual deduplication intent — single-column assumption not made
- [ ] Execution plan reviewed — Stream Aggregate preferred over Sort or Hash if index exists
- [ ] EXISTS evaluated as alternative to DISTINCT + JOIN for existence-based filtering

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Clean, deduplicated result set|Sort or Hash operator CPU + memory cost (when no index)|
|Stream Aggregate is zero-cost if index supports order|Index maintenance overhead for the supporting index (~2–3 pages per DML)|
|Hash Aggregate handles large unsorted inputs efficiently|Hash table memory proportional to unique row count|
|EXISTS avoids row multiplication entirely|Requires correlated subquery — less intuitive than JOIN|

### Scale Thresholds

- DISTINCT on a few unique values (e.g., 3 statuses) is cheap even at **100M rows** — Hash Aggregate handles it.
- DISTINCT on high-cardinality column (e.g., CustomerId) becomes expensive above **~1M rows** — Sort operator can dominate query cost, memory grant reaches 30+ MB.
- DISTINCT on LOB columns is expensive at **any row count above ~10K** — LOB comparison blows up memory and CPU by 100x+.
- DISTINCT as a band-aid for missing join predicate becomes critical when the join multiplies rows beyond **~100K** — the multiplied rows make dedup exponentially more expensive than fixing the join.
- Stream Aggregate via index is worthwhile when the DISTINCT query runs more than **~500 times/hour**.

<summary>Answers</summary>

1. DISTINCT is the **deduplication operator** — it removes rows where every column value is identical to another row. It executes at **step 6**, after SELECT (projection) and before ORDER BY (step 7) and TOP (step 8). This is why ORDER BY columns must appear in the SELECT list when DISTINCT is used.

2. (a) **Stream Aggregate** — when input is already sorted by SELECT columns (index provides order). O(N), zero memory. Chosen when an index matches the DISTINCT column list. (b) **Hash Match Aggregate** — when input is unsorted and estimated unique rows are few relative to total (roughly < 30% of total rows). O(N) hashing, O(unique) memory. (c) **Sort + Distinct Sort** — when input is unsorted and unique rows are numerous. O(N log N), O(total) memory. Chosen when estimated unique > ~30% of total.

3. `SET STATISTICS IO ON` shows logical reads (e.g., 22 vs 11,400). `SET STATISTICS TIME ON` shows CPU time (e.g., 3 ms vs 451 ms). For historical analysis: `sys.dm_exec_query_stats` with `total_logical_reads / execution_count` and `total_worker_time / execution_count`.

4. The anti-pattern is a **missing or incorrect join predicate** that causes row multiplication (Cartesian product or unintended 1:N). The developer adds DISTINCT instead of fixing `ON 1=1` or adding the correct join condition. The correct response: determine root cause of duplicates, fix the join predicate, and remove DISTINCT. Use EXISTS instead of JOIN if only existence is checked.

5. EF Core's `.Distinct()` translates to `SELECT DISTINCT` SQL when used **before `.ToListAsync()`** or `.AsEnumerable()`. If called after those methods, it runs client-side via LINQ-to-Objects — all rows are fetched from the database first. Always verify by enabling SQL logging at `Information` level for `Microsoft.EntityFrameworkCore.Database.Command`.

6. `await connection.QueryAsync<string>("SELECT DISTINCT Status FROM dbo.Orders")`. Dapper maps each result row to a string by column name. For multiple columns, Dapper maps to a DTO or anonymous type by property/column name matching.

7. **3 unique values (no index):** `[Clustered Index Scan] → [Hash Match Aggregate (3-entry hash table)] → [SELECT]` — CPU ~468 ms, memory ~8 KB. **800K unique (no index):** `[Clustered Index Scan] → [Sort] → [Stream Aggregate] → [SELECT]` — CPU ~2,150 ms, memory ~30 MB. With an index on the column, both become `[Index Scan] → [Stream Aggregate]` — no Sort or Hash, CPU 3–52 ms.

8. Above **~100K rows** the Sort/Hash cost becomes measurable. Above **~1M rows** it becomes critical (Sort dominates query cost). An index for Stream Aggregate is worthwhile when the query runs more than **~500 times/hour** and the read/write ratio exceeds **~10:1**.

9. A composite non-clustered index on `(CustomerId, Status)` — the index key columns match both the DISTINCT columns (CustomerId, Status) and the ORDER BY (CustomerId). The optimizer can scan this index in sorted order and use Stream Aggregate. If the table also has a WHERE clause, include those columns as leading key columns.

10. "DISTINCT is a blocking operator that removes duplicate rows by sorting or hashing the entire result set. When it's used to fix a missing join predicate, the join first multiplies rows. On a 500K customer table joined to a 500K order table with `ON 1=1` (cross join), the join produces 250B rows. Adding DISTINCT to 'fix' the duplicates forces the engine to sort or hash those 250B rows to collapse them back to 500K. That's processing 250B rows instead of 500K — a 500,000x overhead. The correct fix is to add the join predicate: `ON c.CustomerId = o.CustomerId`. With the correct join, the query processes 500K rows without duplication, and no DISTINCT is needed. I always check: if the query uses DISTINCT and joins multiple tables, I look at the execution plan for a `Nested Loops` or `Hash Match` join processing more rows than the largest table — that tells me the join is the problem, not the deduplication."

</details>

**Challenge 1 — Write the SQL**

The customer service team needs a dropdown listing all unique order statuses that a specific customer (CustomerId = 1042) has used across their orders. Return only the distinct statuses, ordered alphabetically.

<details>
<summary>Solution</summary>

```sql
SELECT DISTINCT o.Status
FROM dbo.Orders AS o
WHERE o.CustomerId = 1042
ORDER BY o.Status ASC;
```

**Logical reads:** ~4–6 (with index on CustomerId + Status — seek to CustomerId 1042, then Stream Aggregate on Status within that customer's rows). Without index: ~50–200 (filter 1M rows, then DISTINCT on small set).

**Execution plan:** `[Index Seek IX_Orders_CustomerId_Status] → [Stream Aggregate] → [SELECT]` — the index on `(CustomerId, Status)` returns rows sorted by Status within each CustomerId, so Stream Aggregate compares adjacent rows.


**Challenge 2 — Fix the performance problem**

```sql
-- This query returns 500 rows but runs in 12 seconds on a 2M row database.
-- Identify why and fix it.
SELECT DISTINCT
    c.CustomerId,
    c.FirstName,
    c.LastName,
    c.Email,
    c.City,
    c.Region,
    c.Phone
FROM dbo.Customers AS c
INNER JOIN dbo.Orders AS o ON 1=1
WHERE o.OrderDate >= '2024-01-01';
-- SET STATISTICS IO: logical reads = 480,000
-- Actual execution plan: [Nested Loops Cross Join] → [Sort] → [Stream Aggregate] → [SELECT]
-- Estimated rows going into Sort: 250,000,000,000
```

<details>
<summary>Solution</summary>

**Root cause:** Two critical problems:
1. **Missing join predicate:** `INNER JOIN dbo.Orders AS o ON 1=1` is a cross join. Every customer is joined to every order that matches the WHERE clause. If 500K customers exist and 500K orders are from 2024, the join produces 500K × 500K = 250B rows.
2. **DISTINCT as band-aid:** DISTINCT then tries to collapse those 250B rows back to the 500 distinct customers that placed orders — an impossible task. The Sort must process 250B rows; it spills to TempDB immediately, writing hundreds of GB of intermediate data.

```sql
-- ✅ Fix: proper join predicate — eliminates row multiplication
SELECT DISTINCT
    c.CustomerId,
    c.FirstName,
    c.LastName,
    c.Email,
    c.City,
    c.Region,
    c.Phone
FROM dbo.Customers AS c
INNER JOIN dbo.Orders AS o ON c.CustomerId = o.CustomerId
WHERE o.OrderDate >= '2024-01-01';
```

**Logical reads after fix:** ~1,200 (from 480,000). **Execution time:** ~120 ms (from 12 seconds).

**Better alternative — EXISTS eliminates the join entirely:**

```sql
-- ✅ EXISTS: semi-join short-circuits per customer — no row multiplication at all
SELECT c.CustomerId, c.FirstName, c.LastName, c.Email, c.City, c.Region, c.Phone
FROM dbo.Customers AS c
WHERE EXISTS (
    SELECT 1 FROM dbo.Orders AS o
    WHERE o.CustomerId = c.CustomerId
      AND o.OrderDate >= '2024-01-01'
);
```

**With EXISTS:** The plan shows `[Index Scan Customers] → [Nested Loops Semi Join]` or `[Merge Semi Join]`. The semi-join scans Customers and, for each customer, seeks into the Orders index. The seek stops as soon as it finds the first 2024 order — no need to find all orders, no grouping, no dedup. Logical reads: ~300–600 (customer scan + 500 index seeks). Execution time: ~40–80 ms.

</details>

**Challenge 3 — Explain the execution plan**

```sql
-- Query:
SELECT DISTINCT o.CustomerId, o.Status
FROM dbo.Orders AS o;

-- Actual plan: [Clustered Index Scan] → [Hash Match Aggregate] → [SELECT]
-- Estimated unique rows: 300,000 (out of 1,000,000 total)
-- Actual rows returned: 300,000
```

The optimizer chose Hash Match Aggregate over Sort + Distinct Sort. Why? And what would make it choose Sort instead?

<details>
<summary>Solution</summary>

**Why Hash Match Aggregate:** The optimizer estimated 300K unique rows out of 1M total — a 30% uniqueness ratio. At this ratio, both Hash and Sort are viable, but the optimizer's cost model estimated that Hash is cheaper for three reasons:

1. **CPU:** Hash is O(N) = 1M hashing operations, each hashing a (CustomerId, Status) pair. Sort is O(N log N) = 1M × ~20 comparisons ≈ 20M comparison operations. Hash requires 20x fewer operations.

2. **Memory:** Hash Aggregate builds a hash table of 300K entries. At roughly 40 bytes per entry (hash code + pointer + key values), that's ~12 MB. Sort requires a memory grant proportional to the full row size × row count. At 100 bytes per row × 1M rows, that's ~100 MB — 8x more memory.

3. **TempDB risk:** With a 100 MB memory grant needed for Sort vs 12 MB for Hash, the Sort is much more likely to exceed the server's available memory and spill to TempDB.

**What would make it choose Sort instead:**
- If the estimated unique rows exceeded ~50% of total (500K+ unique rows). At that point the hash table size approaches the sort memory grant, and Sort's constant factors become cheaper.
- If the query had `ORDER BY CustomerId, Status` — the optimizer must sort for ORDER BY anyway, so it would use the sort output for Stream Aggregate, combining the Sort and DISTINCT into one operation.
- If the statistics showed very high uniqueness for one column and moderate for the other, producing a combined estimate > 500K unique rows.

**Index that eliminates both:** A composite index on `(CustomerId, Status)`. The plan would become `[Index Scan] → [Stream Aggregate] → [SELECT]` — no Hash table, no Sort, just adjacent row comparison. Logical reads drop from ~11,400 (clustered scan) to ~2,000 (narrower index). CPU drops from ~900 ms (Hash) to ~100 ms (Stream Aggregate).

</details>

**Challenge 4 — Diagnose the concurrency problem**

A reporting query runs every 30 minutes:

```sql
SELECT DISTINCT o.Notes
FROM dbo.Orders AS o
WHERE o.OrderDate >= DATEADD(DAY, -30, GETUTCDATE());
```

The Orders table has 10M rows, 500K of which are from the last 30 days. The Notes column is NVARCHAR(MAX) with an average length of 2,000 characters. The query runs for 45 minutes and causes TempDB to grow by 15 GB. The server reports `PAGELATCH_CONTENTION` on TempDB allocations during and after the query execution.

<details>
<summary>Solution</summary>

**Root cause:** DISTINCT on an NVARCHAR(MAX) column. The Hash Match Aggregate (or Sort) must compare 2,000-byte LOB values for 500K rows. Each row's Notes value is stored off-row and must be read into memory for hashing. If most Notes are unique, the hash table reaches 500K × 2,000 bytes ≈ 1 GB. The memory grant is insufficient (default max ~500 MB), so the hash spills to TempDB — writing 15 GB of overflow data. The TempDB contention comes from 500K concurrent hash bucket allocations spilling to disk.

```sql
-- Fix Option 1: Use CHECKSUM for approximate deduplication
-- CHECKSUM is 4 bytes per entry instead of 2,000 — hash table fits in memory
SELECT DISTINCT CHECKSUM(o.Notes) AS NotesHash
FROM dbo.Orders AS o
WHERE o.OrderDate >= DATEADD(DAY, -30, GETUTCDATE());
-- ⚠️ CHECKSUM has collisions (different Notes → same 4-byte checksum)
-- Acceptable for approximate reporting, not for exact dedup

-- Fix Option 2: Persisted computed column with SHA2_256 hash
ALTER TABLE dbo.Orders
    ADD NotesHash AS CAST(HASHBYTES('SHA2_256', Notes) AS BINARY(32)) PERSISTED;
CREATE INDEX IX_Orders_NotesHash ON dbo.Orders (NotesHash);

SELECT DISTINCT o.Notes
FROM dbo.Orders AS o
WHERE o.OrderDate >= DATEADD(DAY, -30, GETUTCDATE())
  AND o.NotesHash IN (
    SELECT DISTINCT o2.NotesHash
    FROM dbo.Orders AS o2
    WHERE o2.OrderDate >= DATEADD(DAY, -30, GETUTCDATE())
  );
-- BINARY(32) hash: 32 bytes per entry — 60x less memory than 2,000-byte LOB

-- Fix Option 3: Schedule during maintenance window with MIN_GRANT_PERCENT
SELECT DISTINCT o.Notes
FROM dbo.Orders AS o
WHERE o.OrderDate >= DATEADD(DAY, -30, GETUTCDATE())
OPTION (MIN_GRANT_PERCENT = 20);
```

**Expected improvement with CHECKSUM:** The hash table stores 4-byte values instead of 2,000-byte LOBs — a 500x memory reduction. No TempDB spill. Query completes in ~30–60 seconds instead of 45 minutes. The tradeoff (hash collisions) is acceptable for approximate "how many distinct Notes are there?" reporting.

</details>

**Challenge 5 — Design the index**

**Scenario:** The application has a frequently-used API endpoint that returns distinct `City + Region + PostalCode` combinations for customer address autocomplete:

```sql
SELECT DISTINCT c.City, c.Region, c.PostalCode
FROM dbo.Customers AS c
WHERE c.City LIKE @prefix + '%'
ORDER BY c.Region, c.City, c.PostalCode;
```

The Customers table has 10M rows with 150K distinct (City, Region, PostalCode) combinations. The query runs ~20,000 times/hour during business hours. The read/write ratio is 100:1. The current plan shows a Clustered Index Scan + Sort + Stream Aggregate with 112,000 logical reads and 3,200 ms CPU.

Design the optimal index strategy. Consider that the WHERE clause has a prefix `LIKE` on `City`.

<details>
<summary>Solution</summary>

```sql
-- Composite index matching DISTINCT columns + WHERE prefix + ORDER BY
-- Key order: City first (for the LIKE prefix seek), then Region, PostalCode (for ORDER BY)
CREATE NONCLUSTERED INDEX IX_Customers_City_Region_PostalCode
    ON dbo.Customers (City, Region, PostalCode);
```

**Why this index:**

1. **Key order = `(City, Region, PostalCode)`** — The WHERE clause has `City LIKE @prefix + '%'` (SARGable prefix match on City). The index enables an Index Seek on the City prefix range. Within each City, the rows are ordered by (Region, PostalCode), which matches the ORDER BY. The Stream Aggregate then deduplicates adjacent rows that have the same (City, Region, PostalCode) combination.

2. **Covering index** — all three SELECT columns (City, Region, PostalCode) are key columns. No INCLUDE needed because the query selects only those columns. The index fully covers the query.

3. **Key order matches sort order** — ORDER BY `(Region, City, PostalCode)` does NOT match the index key order `(City, Region, PostalCode)`. The optimizer may still need a Sort to reorder from (City, Region) to (Region, City). An alternative index `(Region, City, PostalCode)` would match the ORDER BY but would not support the `City LIKE @prefix` WHERE clause efficiently (City is second key column, not first).

**Alternative: Include a Sort is acceptable** if the reorder Sort is cheap (150K unique rows × 3 small VARCHAR columns). The Stream Aggregate eliminates duplicates from the ordered City scan, then a lightweight Sort reorders for the final output.

**Alternative index for ORDER BY match:**

```sql
-- If the WHERE clause is not selective enough, consider this order:
CREATE NONCLUSTERED INDEX IX_Customers_Region_City_PostalCode
    ON dbo.Customers (Region, City, PostalCode);
-- ORDER BY (Region, City, PostalCode) matches exactly — no Sort needed
-- But City LIKE prefix may not be seekable (City is second key column)
-- Only useful if the City prefix filter is not selective or is absent
```

**Expected improvement:**

|Metric|Before (no index)|After (IX_Customers_City_Region_PostalCode)|
|---|---|---|
|Execution plan|Clustered Index Scan → Sort → Stream Aggregate|Index Seek → Stream Aggregate → Sort (lightweight)|
|Logical reads|~112,000|~600 (index leaf pages for 150K unique rows)|
|CPU time|~3,200 ms|~45 ms|
|Memory grant|~2 MB (Sort on 10M rows)|~0 (Stream Aggregate)|

**Write cost:** The non-clustered index adds ~2–3 pages per INSERT/UPDATE on the Customers table. With a 100:1 read/write ratio and 20,000 reads/hour, the write overhead (~200 extra page writes/hour) is negligible compared to the read performance gain.

</details>
