### Where This Fits

ISNULL, COALESCE, and NULLIF are the three T-SQL functions purpose-built for NULL handling. ISNULL replaces NULL with a specified replacement value (two arguments). COALESCE returns the first non-NULL value from a list (ANSI standard, two or more arguments). NULLIF returns NULL when two expressions are equal. Every .NET backend engineer uses these daily: replacing NULL shipping addresses with 'N/A', coalescing discount percentages, preventing NULL from propagating in arithmetic expressions, and guarding against NULL in WHERE clause comparisons. The most expensive mistakes are: assuming ISNULL and COALESCE are interchangeable (they differ in type precedence and subquery evaluation), using COALESCE in WHERE on a nullable column (non-SARGable when the replacement value is on a different column), misunderstanding that NULL in a WHERE predicate with `= NULL` returns UNKNOWN not TRUE, and forgetting that COALESCE short-circuits but ISNULL does not (though this is a logical rather than performance distinction). Interviewers ask about these to evaluate understanding of three-valued logic, the ISNULL vs COALESCE debate, and practical experience with NULL-induced bugs in production.

## Core Mental Model

NULL is not a value — it is the absence of a value. Any expression involving NULL (arithmetic, comparison, concatenation) produces NULL unless NULL-handling functions are used. ISNULL(check_expression, replacement) returns replacement if check_expression is NULL, otherwise returns check_expression. The result type is determined by the type of check_expression. COALESCE(expression1, expression2, ..., expressionN) returns the first expression that is not NULL, evaluating left to right. The result type follows T-SQL type precedence across all arguments — the highest-precedence type wins. NULLIF(expression1, expression2) returns NULL if the two expressions are equal, otherwise returns expression1. The critical performance rule: ISNULL and COALESCE can be SARGable when used in a WHERE clause if the function is on the **parameter side** (e.g., `WHERE TotalAmount = ISNULL(@Param, TotalAmount)`), but are non-SARGable when applied to a **column** (e.g., `WHERE ISNULL(TotalAmount, 0) = 0`). The three-valued logic rule: any comparison with NULL using =, <, >, BETWEEN, LIKE, or IN returns UNKNOWN, not FALSE. Queries return only rows where the WHERE predicate evaluates to TRUE — rows with UNKNOWN are excluded.

### Classification

ISNULL is a T-SQL-specific function (not ANSI). COALESCE and NULLIF are ANSI standard. All three are scalar functions. ISNULL and COALESCE can be SARGable or non-SARGable depending on usage pattern.

## Deep Mechanics

### How the Engine Executes This

**ISNULL:**

1. The engine evaluates check_expression.
2. If check_expression is NULL, the replacement_value is evaluated and returned.
3. If check_expression is not NULL, check_expression is returned (replacement_value is still evaluated — no short-circuit).
4. The result type is the type of check_expression. If replacement_value has a higher data type precedence, it is implicitly converted to the type of check_expression. This can cause truncation or conversion errors.
5. For varchar arguments: if check_expression is VARCHAR(10) and replacement is VARCHAR(20), the result is VARCHAR(10) — trailing characters are truncated.

**COALESCE:**

1. The engine evaluates expressions left to right.
2. The first expression that evaluates to non-NULL is returned.
3. If all expressions are NULL, NULL is returned.
4. COALESCE does short-circuit — subsequent expressions are not evaluated after a non-NULL expression is found.
5. The result type is determined by data type precedence across **all** arguments. If expression1 is VARCHAR(10) and expression2 is VARCHAR(20), the result is VARCHAR(20) — no truncation.
6. Under the hood, the query optimizer often expands COALESCE into a CASE expression: `CASE WHEN expr1 IS NOT NULL THEN expr1 WHEN expr2 IS NOT NULL THEN expr2 ... END`.

**NULLIF:**

1. The engine evaluates both expressions.
2. If expression1 equals expression2, NULL is returned.
3. If expression1 does not equal expression2, expression1 is returned.
4. NULLIF can be expressed as `CASE WHEN expr1 = expr2 THEN NULL ELSE expr1 END`.

**Three-valued logic in WHERE:**

- `WHERE Column = NULL` evaluates to UNKNOWN (not FALSE). The row is not returned.
- `WHERE Column IS NULL` evaluates to TRUE if Column is NULL — this is the correct way to test for NULL.
- `WHERE Column <> NULL` also evaluates to UNKNOWN.
- `WHERE Column = @Param` when @Param is NULL also returns UNKNOWN — to match NULL parameters, use `(Column = @Param OR (Column IS NULL AND @Param IS NULL))`.

### SQL Visibility

```sql
-- ISNULL: basic null replacement
SELECT
    o.OrderId,
    ISNULL(o.ShippingAddr, 'No Address') AS ShippingAddress,
    ISNULL(o.Notes, '') AS Notes,
    ISNULL(o.TotalAmount, 0.00) AS TotalAmount
FROM dbo.Orders AS o;

-- COALESCE: multiple fallback values
SELECT
    o.OrderId,
    COALESCE(o.DeliveryDate, o.ShippedDate, o.OrderDate) AS EffectiveDate,
    COALESCE(o.Notes, o.ShippingAddr, 'No Info') AS BestDescription
FROM dbo.Orders AS o;

-- NULLIF: guard against zero/blank
SELECT
    o.OrderId,
    o.TotalAmount / NULLIF(o.Quantity, 0) AS UnitPrice,  -- avoids divide-by-zero
    NULLIF(o.Notes, '') AS NotesOrNull                     -- blank → NULL
FROM dbo.Orders AS o;

-- ISNULL in WHERE — non-SARGable (on column side)
-- ❌ Non-SARGable: ISNULL on column
SELECT OrderId, TotalAmount, ShippingAddr
FROM dbo.Orders
WHERE ISNULL(ShippingAddr, '') = '';

-- ✅ SARGable: IS NULL predicate
SELECT OrderId, TotalAmount, ShippingAddr
FROM dbo.Orders
WHERE ShippingAddr IS NULL;

-- ISNULL in WHERE — SARGable (on parameter side)
-- ✅ SARGable: ISNULL on parameter
SELECT OrderId, TotalAmount, Status
FROM dbo.Orders
WHERE Status = ISNULL(@StatusFilter, Status);
-- When @StatusFilter is NULL: WHERE Status = Status (always true)
-- When @StatusFilter is 'Shipped': WHERE Status = 'Shipped'

-- COALESCE in WHERE — non-SARGable
-- ❌ Non-SARGable: COALESCE on column
SELECT OrderId, TotalAmount
FROM dbo.Orders
WHERE COALESCE(DeliveryDate, ShippedDate) >= '2026-06-01';

-- ✅ Alternative: ((DeliveryDate >= '2026-06-01') OR (DeliveryDate IS NULL AND ShippedDate >= '2026-06-01'))

-- NULLIF in WHERE — non-SARGable
-- ❌ Non-SARGable: NULLIF on column
SELECT OrderId, TotalAmount
FROM dbo.Orders
WHERE NULLIF(Status, 'Pending') IS NOT NULL;

-- Three-valued logic demonstration
DECLARE @Val INT = NULL;
SELECT
    CASE WHEN @Val = NULL THEN 'Equal' ELSE 'Not Equal' END AS EqResult,       -- 'Not Equal'
    CASE WHEN @Val IS NULL THEN 'Is Null' ELSE 'Is Not Null' END AS IsResult;   -- 'Is Null'

-- Proper NULL-safe comparison pattern
SELECT OrderId, Status
FROM dbo.Orders
WHERE Status = @Status OR (Status IS NULL AND @Status IS NULL);
```

### Execution Plan Analysis

**ISNULL/COALESCE in SELECT:**
- Plan: `[Index Scan / Seek] → [Compute Scalar] → [SELECT]`
- Compute Scalar evaluates the null-handling function. CPU cost: negligible (~0.0001 per row).

**ISNULL/COALESCE on column in WHERE (non-SARGable):**
- Plan: `[Clustered Index Scan] → [Filter] → [SELECT]`
- No index seek possible. The function result is computed per row.
- On a 1M row table: scan of ~12,000 logical reads.

**ISNULL on parameter side in WHERE (SARGable):**
- Plan: `[Index Seek] → [SELECT]`
- `WHERE Status = ISNULL(@Filter, Status)` — the optimizer simplifies this to a seek if it generates a startup expression predicate.
- In practice: the optimizer may still scan if it cannot simplify the expression. Performance depends on the complexity.

```
ISNULL/COALESCE on column in WHERE (1M rows):
[Clustered Index Scan] → [Compute Scalar: ISNULL] → [Filter] → [SELECT]
Cost: ~12  |  Logical Reads: ~12,000

ISNULL on parameter in WHERE (SARGable):
[Index Seek: Status = @Filter] → [SELECT]
Cost: 0.03  |  Logical Reads: ~12
```

### Failure Modes

**ISNULL type precedence truncation:** ISNULL returns the type of the first argument. If replacement_value has a higher precision or longer length, it is silently truncated: `ISNULL(VARCHAR(10) column, 'too long value')` returns only 10 characters.

**COALESCE unexpected type promotion:** COALESCE determines the result type by type precedence across all arguments. `COALESCE(DECIMAL(18,2) column, 0)` returns DECIMAL(18,2). But `COALESCE(INT column, DECIMAL(18,2) column)` returns DECIMAL(18,2) because DECIMAL has higher precedence than INT. This can cause implicit conversion in joins or unexpected precision.

**NULL in IN list:** `WHERE Column NOT IN (1, 2, NULL)` returns UNKNOWN for every row — the entire query returns zero rows because `NOT IN` with NULL in the list is equivalent to `Column <> 1 AND Column <> 2 AND Column <> NULL`, where `Column <> NULL` is always UNKNOWN.

**COALESCE subquery evaluation:** COALESCE evaluates subqueries in order, but the optimizer may expand it to a CASE expression. Side effects in subqueries (e.g., modifying data) may execute differently than expected.

**ISNULL vs COALESCE with subqueries:** If a subquery returns NULL, ISNULL and COALESCE handle it the same way. But COALESCE short-circuits — if an earlier expression is non-NULL, later subqueries are not executed. ISNULL always evaluates both arguments.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Schema context
-- ============================================================
CREATE TABLE dbo.Orders
(
    OrderId        INT             NOT NULL IDENTITY(1,1),
    CustomerId     INT             NOT NULL,
    ShippingAddr   VARCHAR(200)    NULL,
    Notes          VARCHAR(500)    NULL,
    DiscountPct    DECIMAL(5,4)    NULL,
    TotalAmount    DECIMAL(18,2)   NOT NULL DEFAULT 0,
    Quantity       INT             NOT NULL DEFAULT 1,
    Status         VARCHAR(20)     NOT NULL DEFAULT 'Pending',
    DeliveryDate   DATETIME2(0)    NULL,
    ShippedDate    DATETIME2(0)    NULL,
    CreatedAt      DATETIME2(0)    NOT NULL DEFAULT SYSUTCDATETIME(),
    CONSTRAINT PK_Orders PRIMARY KEY CLUSTERED (OrderId)
);

CREATE INDEX IX_Orders_ShippingAddr ON dbo.Orders (ShippingAddr);
CREATE INDEX IX_Orders_Status ON dbo.Orders (Status);

-- ============================================================
-- Pattern 1: ISNULL for single default replacement
-- ============================================================
SELECT
    o.OrderId,
    ISNULL(o.ShippingAddr, 'No Address') AS DisplayAddress,
    ISNULL(o.Notes, '') AS Notes
FROM dbo.Orders AS o;

-- ============================================================
-- Pattern 2: COALESCE for priority-based fallback
-- ============================================================
SELECT
    o.OrderId,
    COALESCE(o.DeliveryDate, o.ShippedDate, o.OrderDate) AS EffectiveDate,
    COALESCE(o.Notes, o.ShippingAddr, 'No Info') AS BestDescription
FROM dbo.Orders AS o;

-- ============================================================
-- Pattern 3: NULLIF for divide-by-zero guard
-- ============================================================
SELECT
    o.OrderId,
    o.TotalAmount / NULLIF(o.Quantity, 0) AS UnitPriceCalculated,
    NULLIF(o.TotalAmount, 0) AS NonZeroAmount
FROM dbo.Orders AS o;

-- ============================================================
-- Pattern 4: SARGable optional filter (ISNULL on parameter)
-- ============================================================
-- Filter by Status, but return all if @Status is NULL
DECLARE @StatusFilter VARCHAR(20) = NULL;

SELECT OrderId, Status, TotalAmount
FROM dbo.Orders
WHERE Status = ISNULL(@StatusFilter, Status)
ORDER BY OrderId;
-- When @StatusFilter is NULL, the predicate becomes Status = Status (always true)
-- When @StatusFilter is 'Shipped', the predicate becomes Status = 'Shipped' (SARGable seek)

-- ============================================================
-- Pattern 5: COALESCE for computed column
-- ============================================================
ALTER TABLE dbo.Orders
ADD EffectiveDate AS COALESCE(DeliveryDate, ShippedDate, OrderDate);

-- ============================================================
-- Pattern 6: NULL-safe WHERE clause (handles NULL parameter matching)
-- ============================================================
DECLARE @SearchAddr VARCHAR(200) = NULL;

SELECT OrderId, ShippingAddr, Status
FROM dbo.Orders
WHERE (ShippingAddr = @SearchAddr OR (ShippingAddr IS NULL AND @SearchAddr IS NULL));
-- Matches rows where ShippingAddr = @SearchAddr,
-- AND also matches rows where both are NULL

-- ============================================================
-- Pattern 7: NULLIF for blank-to-NULL conversion
-- ============================================================
-- Convert empty strings to NULL for consistent NULL handling
UPDATE dbo.Orders
SET Notes = NULLIF(Notes, '')
WHERE Notes = '';  -- SARGable: direct comparison, not on function

-- ============================================================
-- Pattern 8: ISNULL/COALESCE for aggregate safety
-- ============================================================
SELECT
    CustomerId,
    COUNT(*) AS OrderCount,
    ISNULL(SUM(TotalAmount), 0) AS TotalRevenue,     -- SUM returns NULL if no rows
    ISNULL(AVG(DiscountPct), 0) AS AvgDiscount        -- AVG returns NULL if no rows
FROM dbo.Orders
GROUP BY CustomerId;

-- ============================================================
-- Anti-pattern: ISNULL on column in WHERE
-- ============================================================
-- ❌ Non-SARGable
-- SELECT * FROM Orders WHERE ISNULL(ShippingAddr, '') = '';
-- ✅ SARGable
SELECT * FROM Orders WHERE ShippingAddr IS NULL;
SELECT * FROM Orders WHERE ShippingAddr = '';
```

```csharp
    CancellationToken cancellationToken = default)

    if (statusFilter is not null)
```

## Gotchas and Production Pitfalls

### ISNULL Truncates to First Argument's Type

**Pitfall:** Using ISNULL where the replacement value has a longer string or higher precision than the first argument. ISNULL returns the type of the first argument, silently truncating the replacement.

```sql
-- ❌ Truncation: ISNULL returns VARCHAR(10)
DECLARE @ShortCol VARCHAR(10) = NULL;
SELECT ISNULL(@ShortCol, 'Very long replacement text');
-- Returns 'Very long ' (truncated to 10 characters)
```

**Symptom:** A report shows truncated address data after applying ISNULL. Shipping addresses that are NULL are replaced with 'No Address' correctly, but when the replacement value exceeds the column's length, the output is silently truncated.

**Fix:** Use COALESCE (which uses the highest-precedence type across all arguments) or CAST/convert the first argument:

```sql
-- ✅ COALESCE uses the wider type
DECLARE @ShortCol VARCHAR(10) = NULL;
SELECT COALESCE(@ShortCol, 'Very long replacement text');  -- Full text (VARCHAR(29))

-- ✅ Explicit CAST to match the replacement
DECLARE @ShortCol VARCHAR(10) = NULL;
SELECT ISNULL(CAST(@ShortCol AS VARCHAR(100)), 'Very long replacement text');
```

**Cost of not fixing:** A CRM system uses ISNULL(CompanyName, 'Unknown Company') where CompanyName is VARCHAR(50). A merger creates companies with names up to 100 characters. The ISNULL result truncates to 50 chars. The "Unknown Company" is stored as 'Unknown Company...' (truncated). Data export to the ERP system fails because the truncated name doesn't match.

---

### COALESCE Type Precedence Surprise

**Pitfall:** COALESCE determines its result type by data type precedence across **all** arguments. Mixing INT and DECIMAL (or DATETIME and DATETIME2) causes implicit type promotion that may be unexpected.

```sql
-- ❌ Type promotion: COALESCE returns DECIMAL(18,0), not INT
DECLARE @IntCol INT = NULL;
DECLARE @DecCol DECIMAL(18,2) = 100.50;
SELECT COALESCE(@IntCol, @DecCol) AS Result;
-- Returns 100.50 as DECIMAL(18,2) — INT was promoted to DECIMAL
```

**Symptom:** A join on the COALESCE result fails or performs poorly due to type mismatch. The COALESCE output type differs from the column type in the join predicate, causing an implicit conversion that prevents index seek.

**Fix:** Ensure consistent types across all COALESCE arguments, or CAST to the desired type:

```sql
-- ✅ Consistent types — cast to desired output type
DECLARE @IntCol INT = NULL;
DECLARE @DecCol DECIMAL(18,2) = 100.50;
SELECT COALESCE(CAST(@IntCol AS DECIMAL(18,2)), @DecCol) AS Result;
```

**Cost of not fixing:** A COALESCE in a JOIN predicate causes an implicit CONVERT_IMPLICIT in the execution plan, preventing an index seek. The query scans 500K rows instead of seeking 12. The production issue goes unnoticed until the table grows to 10M rows and the query times out.

---

### NOT IN With NULL in the Subquery — Empty Result Set

**Pitfall:** Using `WHERE Column NOT IN (SELECT NullableColumn FROM Table)` where NullableColumn contains any NULL values. The entire query returns zero rows.

```sql
-- ❌ Returns zero rows if any Status in the subquery is NULL
SELECT OrderId, Status
FROM dbo.Orders
WHERE Status NOT IN (
    SELECT DISTINCT Status FROM dbo.Orders WHERE Status IS NOT NULL  -- missing IS NOT NULL
);
-- Equivalent to: Status <> 'Shipped' AND Status <> 'Pending' AND Status <> NULL
-- Status <> NULL is UNKNOWN — entire predicate is UNKNOWN

-- ✅ Fix: add IS NOT NULL or use NOT EXISTS
SELECT OrderId, Status
FROM dbo.Orders
WHERE Status NOT IN (
    SELECT DISTINCT Status FROM dbo.Orders WHERE Status IS NOT NULL
);

-- ✅ Better: use NOT EXISTS (handles NULL correctly)
SELECT o.OrderId, o.Status
FROM dbo.Orders o
WHERE NOT EXISTS (
    SELECT 1 FROM dbo.Orders sub
    WHERE sub.Status = o.Status
    AND sub.Status IS NOT NULL
);
```

**Symptom:** A "find unmatched records" query that used NOT IN for months suddenly returns zero rows. The status table (used in the subquery) was modified to allow NULL. The query silently breaks — no error, no warning. The business reports "no orders needing attention" and misses a critical processing batch.

**Fix:** Always add `WHERE NullableColumn IS NOT NULL` to the subquery in NOT IN, or switch to NOT EXISTS:

```sql
-- ✅ NOT EXISTS is NULL-safe
SELECT o.OrderId, o.Status
FROM dbo.Orders o
WHERE NOT EXISTS (
    SELECT 1 FROM dbo.ExcludedStatuses es
    WHERE es.Status = o.Status
);

-- ✅ Or explicitly filter NULL from IN subquery
SELECT OrderId, Status
FROM dbo.Orders
WHERE Status NOT IN (
    SELECT Status FROM dbo.ExcludedStatuses WHERE Status IS NOT NULL
);
```

**Cost of not fixing:** A compliance report that filters out sanctioned countries using NOT IN against a country exclusion table suddenly returns zero rows because one country code in the exclusion table was entered as NULL. The compliance team approves all orders (including sanctioned ones) because the "not sanctioned" filter returns nothing. The oversight is caught by an auditor 6 months later.

---

### ISNULL on Column in WHERE — Non-SARGable Predicate

**Pitfall:** Using `ISNULL(Column, Default)` in the WHERE clause. The function is applied to the column, preventing index seek.

```sql
-- ❌ Non-SARGable: ISNULL on column
SELECT OrderId, ShippingAddr
FROM dbo.Orders
WHERE ISNULL(ShippingAddr, '') = '';
```

**Symptom:** The execution plan shows a Clustered Index Scan. The ISNULL function is applied to every row's ShippingAddr before the comparison. Even with an index on ShippingAddr, SQL Server cannot seek.

**Fix:** Use the direct IS NULL or range pattern:

```sql
-- ✅ SARGable alternatives
SELECT OrderId, ShippingAddr
FROM dbo.Orders
WHERE ShippingAddr IS NULL OR ShippingAddr = '';
```

**Cost of not fixing:** A nightly cleanup job that removes orders with no shipping address uses `DELETE FROM Orders WHERE ISNULL(ShippingAddr, '') = ''`. The query scans the entire 50M row table every night. The scan takes 8 minutes and generates 620,000 logical reads. Replacing with `WHERE ShippingAddr IS NULL OR ShippingAddr = ''` enables two seeks and reduces the execution to 2 seconds.

---

### COALESCE with Subqueries — Unexpected Multiple Evaluations

**Pitfall:** Assuming COALESCE evaluates subqueries at most once. The query optimizer may expand COALESCE into a CASE expression, potentially evaluating subqueries multiple times or in unexpected order.

```sql
-- ❌ Potential double evaluation
SELECT
    COALESCE(
        (SELECT TOP 1 Notes FROM dbo.OrderNotes WHERE OrderId = o.OrderId ORDER BY CreatedAt DESC),
        (SELECT TOP 1 ShippingAddr FROM dbo.OrderAddr WHERE OrderId = o.OrderId ORDER BY UpdatedAt DESC),
        'No Info'
    ) AS BestInfo
FROM dbo.Orders o;
```

**Symptom:** The subqueries appear multiple times in the execution plan (as separate index scans). The optimizer may evaluate both subqueries even when the first subquery returns a non-NULL value. This doubles the I/O and CPU cost.

**Fix:** Use a CTE or derived table to materialize subquery results before COALESCE:

```sql
-- ✅ Materialize subqueries first
WITH OrderInfo AS (
    SELECT
        o.OrderId,
        (SELECT TOP 1 Notes FROM dbo.OrderNotes WHERE OrderId = o.OrderId ORDER BY CreatedAt DESC) AS Notes,
        (SELECT TOP 1 ShippingAddr FROM dbo.OrderAddr WHERE OrderId = o.OrderId ORDER BY UpdatedAt DESC) AS Addr
    FROM dbo.Orders o
)
SELECT OrderId, COALESCE(Notes, Addr, 'No Info') AS BestInfo
FROM OrderInfo;
```

**Cost of not fixing:** A report query processes 100K orders with COALESCE containing two correlated subqueries per row. The execution plan shows 200K subquery executions instead of 100K. The query takes 45 seconds instead of 25 seconds. At 1M orders, the query times out.

---

### NULL in Arithmetic Expressions — Silent NULL Propagation

**Pitfall:** Forgetting that any arithmetic with NULL produces NULL. `NULL + 100 = NULL`. `NULL * 5 = NULL`.

```sql
-- ❌ NULL propagation: if DiscountPct is NULL, DiscountedAmount is NULL
SELECT
    OrderId,
    TotalAmount,
    DiscountPct,
    TotalAmount * (1 - DiscountPct) AS DiscountedAmount   -- NULL if DiscountPct is NULL
FROM dbo.Orders;
```

**Symptom:** A financial report shows NULL for discounted amounts on orders without a discount. The NULL propagates up: SUM(DiscountedAmount) also returns NULL. The report shows "No Data" for total discounted revenue.

**Fix:** Use ISNULL or COALESCE to provide a default for the nullable operand:

```sql
-- ✅ Guard against NULL in arithmetic
SELECT
    OrderId,
    TotalAmount,
    DiscountPct,
    TotalAmount * (1 - ISNULL(DiscountPct, 0)) AS DiscountedAmount
FROM dbo.Orders;
```

**Cost of not fixing:** A revenue report uses `SUM(TotalAmount * (1 - DiscountPct))` where DiscountPct is nullable. If any order has a NULL discount, the entire SUM returns NULL (not just that order's contribution). The finance team sees "Total Revenue: NULL" and assumes the ETL pipeline is broken. It takes 3 hours to identify the NULL propagation.

---

### ISNULL vs COALESCE with Subquery Side Effects

**Pitfall:** Relying on short-circuit evaluation in COALESCE with subqueries or function calls that have side effects. While COALESCE logically short-circuits, the optimizer may expand it into CASE and evaluate expressions in unexpected order.

```sql
-- ❌ Side effect in COALESCE subquery
SELECT
    COALESCE(
        dbo.GetCachedValue(@OrderId),          -- function with side effect (logging)
        dbo.ComputeExpensiveValue(@OrderId),   -- expensive computation
        0
    ) AS FinalValue;
```

**Symptom:** The GetCachedValue function is called, and when it returns NULL, ComputeExpensiveValue is called. But if the optimizer rewrites the COALESCE and evaluates both expressions, both functions execute for every row — doubling the cost.

**Fix:** Materialize the function results before COALESCE, or use explicit CASE:

```sql
-- ✅ Explicit CASE guarantees short-circuit
SELECT
    CASE
        WHEN dbo.GetCachedValue(@OrderId) IS NOT NULL
            THEN dbo.GetCachedValue(@OrderId)
        WHEN dbo.ComputeExpensiveValue(@OrderId) IS NOT NULL
            THEN dbo.ComputeExpensiveValue(@OrderId)
        ELSE 0
    END AS FinalValue;
```

**Cost of not fixing:** A stored procedure that generates order reports calls two functions in a COALESCE — one fast cache lookup and one expensive computation. The optimizer evaluates both for every row. The report that should take 2 seconds takes 2 minutes. The cache layer is effectively bypassed.

### Spoken Answers

**Q: What is the difference between ISNULL and COALESCE? When do you use each?**

> **Great answer:** ISNULL and COALESCE both return the first non-NULL value, but there are four critical differences. First, ISNULL takes exactly two arguments; COALESCE takes two or more. Second, ISNULL returns the type of the first argument — if the replacement has a longer string or higher precision, it gets truncated to the first argument's type. COALESCE returns the type with the highest data type precedence across all arguments, so no truncation occurs. Third, COALESCE is ANSI standard and portable; ISNULL is T-SQL-specific. Fourth, COALESCE short-circuits (stops evaluating after finding the first non-NULL), while ISNULL always evaluates both arguments, though in practice the optimizer may handle this similarly. I use ISNULL for simple two-argument cases where I'm certain the types match, because it's marginally faster. I use COALESCE for any case with three or more arguments, when I need ANSI portability, or when I need the wider type from a later argument. I never use ISNULL when the replacement value has a different type or length than the check expression — that's asking for silent truncation.

---

**Q: Why does `WHERE Column NOT IN (1, 2, NULL)` return zero rows?**

> **Great answer:** This is a classic three-valued logic trap. The predicate `Column NOT IN (1, 2, NULL)` is logically equivalent to `Column <> 1 AND Column <> 2 AND Column <> NULL`. The comparison `Column <> NULL` always evaluates to UNKNOWN because NULL is not a value — any comparison with NULL using any operator (=, <>, <, >, LIKE, IN, BETWEEN) returns UNKNOWN, not TRUE or FALSE. For a row to be included in the result set, the WHERE predicate must evaluate to TRUE. When one of the AND conditions is UNKNOWN, the entire AND expression is either FALSE or UNKNOWN — never TRUE. So no rows match. The fix is to add `WHERE NullableColumn IS NOT NULL` to the IN subquery, or better, use NOT EXISTS which handles NULL correctly because EXISTS uses two-valued logic. This is why production code should always use NOT EXISTS instead of NOT IN for subqueries that might return NULL values.

---

**Q: How do you make an optional filter parameter SARGable when the parameter can be NULL?**

> **Great answer:** There are two approaches. The first is application-level: only add the WHERE clause when the parameter is not NULL. In EF Core, this means building the query conditionally with `if (statusFilter is not null) query = query.Where(o => o.Status == statusFilter)`. When the parameter is NULL, no WHERE clause is generated — the query returns all rows efficiently. When the parameter is provided, the WHERE clause uses an equality predicate that seeks on the index. The second approach is the SQL pattern `WHERE Status = ISNULL(@Filter, Status)`, which works logically but may still cause a scan if the optimizer cannot simplify it to a startup expression predicate. The first approach (application-level conditional filtering) is always preferred because it gives the optimizer the simplest possible predicate and avoids any ambiguity. In Dapper, I use `WHERE (@Status IS NULL OR Status = @Status)` which is SARGable — when @Status is not NULL, the OR is optimized away and the equality predicate seeks on the index.

### Interview Trigger

The defining NULL question: "I have a table with 100M rows. I need to find all rows where the nullable column 'Status' matches a parameter, including the case where both are NULL. Write the query. Now make it fast." A candidate who writes `WHERE ISNULL(Status, '') = ISNULL(@Param, '')` fails SARGability. A candidate who writes `WHERE Status = @Param OR (Status IS NULL AND @Param IS NULL)` passes the correctness test. The follow-up: "How do you make this SARGable when @Param is 'Shipped'?" — the strong candidate uses dynamic SQL or application-level conditional filtering: `WHERE Status = @Param` when @Param is non-NULL; `WHERE Status IS NULL` when the intent is to match only NULLs; or no WHERE clause when returning all rows.

### When to Apply

```mermaid
flowchart TD

    I -->|COALESCE on parameter| K[OK - but evaluate alternatives]

    M -->|Same type| N[ISNULL is fine]
    M -->|Different types| O[Use COALESCE or explicit CAST]

```

### Application Checklist

- [ ] ISNULL used only when first argument's type matches the replacement type (no truncation)
- [ ] COALESCE used for 3+ arguments or when type precedence matters
- [ ] COALESCE arguments have consistent data types (avoid implicit conversion)
- [ ] NULL handled in arithmetic: `TotalAmount * (1 - ISNULL(DiscountPct, 0))`
- [ ] NOT IN never used with a subquery that can return NULL — use NOT EXISTS
- [ ] WHERE clauses use IS NULL / IS NOT NULL instead of `= NULL` or `ISNULL(col, '') = ''`
- [ ] NULL-safe equality patterns handle the `(col = @p OR (col IS NULL AND @p IS NULL))` case
- [ ] Optional filter parameters built at the application level (dynamic WHERE) for optimal SARGability
- [ ] NULLIF used for divide-by-zero guard and blank-to-NULL conversion
- [ ] ISNULL/COALESCE around aggregates prevents NULL-on-empty-group (ISNULL(SUM(Amount), 0))

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|ISNULL: simple, fast 2-arg NULL replacement|Truncation risk; only 2 args; not ANSI|
|COALESCE: flexible, ANSI, no truncation|Slightly slower; type precedence surprises|
|NULLIF: elegant divide-by-zero guard|Evaluates both args; NULL if equal (may surprise)|
|IS NULL/IS NOT NULL: SARGable|Only checks NULL/non-NULL — no replacement|
|App-level conditional filter: optimal SARGability|More C# code; dynamic query generation|

### Scale Thresholds

- ISNULL/COALESCE on column in WHERE is critical above **~10K rows** — scan cost dominates.
- NOT IN vs NOT EXISTS matters at **any scale** when NULL values exist — correctness issue, not performance.
- NULL propagation in aggregates (SUM returning NULL) matters at **any scale** — a single NULL input to SUM returns NULL for the whole group.
- ISNULL/COALESCE on parameter side becomes material above **~100K rows** when the optimizer fails to simplify the expression into a seek.

Or use `EF.Functions.Equals()` in EF Core 6+.

7. In SELECT: `[Compute Scalar]` operator evaluates ISNULL/COALESCE after data access. In WHERE on column: `[Filter]` operator applies the condition after a table/index scan. In WHERE on parameter: can be optimized into an `[Index Seek]` with a startup expression predicate.

8. SQL uses three-valued logic: TRUE, FALSE, UNKNOWN. C# uses two-valued logic: true, false. In SQL, any comparison with NULL returns UNKNOWN. WHERE clauses include only rows where the predicate is TRUE — UNKNOWN rows are excluded. In C#, `null == null` returns true (reference equality). The SQL equivalent is `IS NULL`. This difference causes the NOT IN NULL trap and the `WHERE Column = NULL` bug.

9. `TotalAmount / NULLIF(Quantity, 0)` returns NULL when Quantity = 0 (because NULLIF(0, 0) = NULL). Division by NULL returns NULL instead of raising a divide-by-zero error. The application handles NULL gracefully rather than crashing with an error. This is a NULL-based guard, not an error-based guard.

10. "In production SQL Server, NULL handling requires constant vigilance. I never use `= NULL` — always IS NULL. I never use NOT IN with a subquery that can return NULL — always NOT EXISTS or filter NULLs explicitly. For arithmetic, I wrap nullable columns in ISNULL to prevent NULL propagation. For type safety, I prefer COALESCE over ISNULL when the replacement value has a different type — COALESCE uses the widest type and avoids silent truncation. In WHERE, I never apply ISNULL or COALESCE to a column — that's non-SARGable and forces a scan. Instead, I use IS NULL / IS NOT NULL or range predicates. For optional filter parameters, I build the query dynamically in application code — only adding the WHERE clause when the parameter is provided. This gives the optimizer the cleanest predicate and guarantees a seek on the index. In EF Core, I use the `??` operator which translates to COALESCE, and I use EF.Functions.Equals for null-safe equality. In Dapper, I use explicit `(@p IS NULL OR col = @p)` patterns. The three-valued logic distinction is the most important concept: SQL's UNKNOWN is not FALSE. Forgetting this causes the NOT IN NULL trap, the `WHERE Column = NULL` bug, and silent NULL propagation in expressions."

</details>

### Query Challenges

**Challenge 1 — Write the NULL-safe filter**

Write a query that returns orders where ShippingAddr matches @SearchAddr. If @SearchAddr is NULL, return orders where ShippingAddr IS NULL. The query must be SARGable when @SearchAddr is not NULL.

<details>
<summary>Solution</summary>

```sql
DECLARE @SearchAddr VARCHAR(200) = '123 Main St';

SELECT OrderId, ShippingAddr, Status, TotalAmount
FROM dbo.Orders
WHERE (ShippingAddr = @SearchAddr OR (ShippingAddr IS NULL AND @SearchAddr IS NULL));
```

**SARGability:** When @SearchAddr is non-NULL, the predicate `ShippingAddr = @SearchAddr` seeks on IX_Orders_ShippingAddr. When @SearchAddr is NULL, the predicate `ShippingAddr IS NULL` seeks on the same index for NULL rows.

**Alternative application-level approach (preferred for pure SARGability):**
```csharp
var query = dbContext.Orders.AsQueryable();
if (searchAddr is not null)
    query = query.Where(o => o.ShippingAddr == searchAddr);
else
    query = query.Where(o => o.ShippingAddr == null);
```

</details>

**Challenge 2 — Fix the performance problem**

```sql
-- This query takes 8 seconds on a 50M row table.
SET STATISTICS TIME ON;

SELECT COUNT(*)
FROM dbo.Orders
WHERE ISNULL(ShippingAddr, 'No Address') = 'No Address';

-- SET STATISTICS IO: Table 'Orders'. Scan count 1, logical reads 620,000
```

Identify why it is slow and fix it.

<details>
<summary>Solution</summary>

**Root cause:** ISNULL(ShippingAddr, 'No Address') on the column side — non-SARGable. The function is applied to every row, preventing index seek on IX_Orders_ShippingAddr.

**Fix:** The predicate matches rows where ShippingAddr is NULL (ISNULL replaces NULL with 'No Address', then compares). Also matches rows where ShippingAddr = 'No Address' literally (but this is unlikely — addresses are real addresses). The intent is nearly always to find rows with NULL ShippingAddr:

```sql
SELECT COUNT(*)
FROM dbo.Orders
WHERE ShippingAddr IS NULL;
```

If the intent includes matching literal 'No Address':
```sql
SELECT COUNT(*)
FROM dbo.Orders
WHERE ShippingAddr IS NULL OR ShippingAddr = 'No Address';
```

**After fix — logical reads:** ~12 (seek on IX_Orders_ShippingAddr for IS NULL) from 620,000. **Execution time:** ~3 ms from 8 seconds.

**EF Core:**
```csharp
var count = await dbContext.Orders
    .Where(o => o.ShippingAddr == null)  // or (o.ShippingAddr == null || o.ShippingAddr == "No Address")
    .CountAsync(cancellationToken);
```

</details>

**Challenge 3 — Explain the execution plan**

A query produces this plan:
`[Clustered Index Scan] → [Compute Scalar] → [Filter] → [SELECT]`

The query is:
```sql
SELECT OrderId, TotalAmount, DiscountPct
FROM dbo.Orders
WHERE COALESCE(DiscountPct, 0) > 0.05;
```

Why does the plan show a Clustered Index Scan? What would you change to get a SARGable plan?

<details>
<summary>Solution</summary>

**Why Clustered Index Scan:** COALESCE(DiscountPct, 0) is applied to the DiscountPct column. The index on DiscountPct stores the raw decimal values, not the COALESCE output. The optimizer cannot seek on `COALESCE(column, 0) > constant`.

**SARGable alternative:** `COALESCE(DiscountPct, 0) > 0.05` matches rows where DiscountPct > 0.05 OR (DiscountPct IS NULL AND 0 > 0.05). Since 0 > 0.05 is FALSE, the NULL case never matches. Therefore the predicate simplifies to:

```sql
SELECT OrderId, TotalAmount, DiscountPct
FROM dbo.Orders
WHERE DiscountPct > 0.05;
```

This seeks on an index on DiscountPct. If the intent was to include NULL as matching (> 0.05), use:

```sql
SELECT OrderId, TotalAmount, DiscountPct
FROM dbo.Orders
WHERE DiscountPct > 0.05 OR DiscountPct IS NULL;
```

**Plan after fix:** `[Index Seek (IX_Orders_DiscountPct)] → [SELECT]` (or with a scan for NULL if included).

</details>

**Challenge 4 — Diagnose the NULL aggregation bug**

A financial report aggregates total revenue:

```sql
SELECT
    CustomerId,
    COUNT(*) AS OrderCount,
    SUM(TotalAmount * (1 - DiscountPct)) AS TotalRevenue
FROM dbo.Orders
GROUP BY CustomerId;
```

Some customers show NULL for TotalRevenue even though they have orders. Diagnose the bug.

<details>
<summary>Solution</summary>

**Root cause:** NULL propagation. `DiscountPct` is NULL for some orders. `TotalAmount * (1 - NULL)` = NULL. `SUM(NULL)` returns NULL for the entire group — not 0, and not the sum of non-NULL values. A single NULL DiscountPct in a customer's orders causes the entire TotalRevenue for that customer to be NULL.

**Detection:**
```sql
-- Find customers with NULL revenue
SELECT CustomerId, SUM(TotalAmount * (1 - DiscountPct)) AS Revenue
FROM dbo.Orders
GROUP BY CustomerId
HAVING SUM(TotalAmount * (1 - DiscountPct)) IS NULL;
```

**Fix:**
```sql
SELECT
    CustomerId,
    COUNT(*) AS OrderCount,
    ISNULL(SUM(TotalAmount * (1 - ISNULL(DiscountPct, 0))), 0) AS TotalRevenue
FROM dbo.Orders
GROUP BY CustomerId;
```

**Better approach — separate concerns:**
```sql
SELECT
    CustomerId,
    COUNT(*) AS OrderCount,
    SUM(TotalAmount) - SUM(TotalAmount * ISNULL(DiscountPct, 0)) AS TotalRevenue
FROM dbo.Orders
GROUP BY CustomerId;
```

**In EF Core:**
```csharp
var summaries = await dbContext.Orders
    .GroupBy(o => o.CustomerId)
    .Select(g => new
    {
        CustomerId = g.Key,
        OrderCount = g.Count(),
        TotalRevenue = g.Sum(o => o.TotalAmount * (1 - (o.DiscountPct ?? 0m)))
    })
    .ToListAsync(cancellationToken);
```

</details>

**Challenge 5 — Design the NULL handling strategy**

**Scenario:** A CRM database has these requirements:
1. PhoneNumber can be NULL (no phone), empty string (phone not provided), or a valid number
2. Email can be NULL (no email), empty string (not provided), or valid email
3. ContactPreference: 'Phone', 'Email', 'Mail', or NULL (no preference)
4. Reports must show "Preferred Contact: Phone / Email / Mail / No Preference"
5. Queries must find contacts by phone or email efficiently, supporting partial matches

Design the column types, constraints, and query patterns.

<details>
<summary>Solution</summary>

**Schema:**

```sql
CREATE TABLE dbo.Contacts
(
    ContactId         INT            NOT NULL IDENTITY(1,1),
    Name              VARCHAR(100)   NOT NULL,
    PhoneNumber       VARCHAR(20)    NULL,
    Email             VARCHAR(200)   NULL,
    ContactPreference VARCHAR(5)     NULL
        CHECK (ContactPreference IN ('Phone', 'Email', 'Mail')),
    CONSTRAINT PK_Contacts PRIMARY KEY CLUSTERED (ContactId)
);

-- Normalize empty strings to NULL for consistent handling
-- Use a trigger or application logic on insert/update
CREATE TRIGGER TR_Contacts_NormalizeNulls
ON dbo.Contacts
AFTER INSERT, UPDATE
AS
BEGIN
    SET NOCOUNT ON;
    UPDATE c
    SET PhoneNumber = NULLIF(PhoneNumber, ''),
        Email = NULLIF(Email, '')
    FROM dbo.Contacts c
    INNER JOIN inserted i ON c.ContactId = i.ContactId
    WHERE NULLIF(i.PhoneNumber, '') IS NULL OR NULLIF(i.Email, '') IS NULL;
END;

-- Indexes for efficient search
CREATE INDEX IX_Contacts_PhoneNumber ON dbo.Contacts (PhoneNumber)
    WHERE PhoneNumber IS NOT NULL;  -- filtered index: only non-NULL
CREATE INDEX IX_Contacts_Email ON dbo.Contacts (Email)
    WHERE Email IS NOT NULL;
```

**Display query:**
```sql
SELECT
    ContactId,
    Name,
    PhoneNumber,
    Email,
    COALESCE(
        CASE ContactPreference
            WHEN 'Phone' THEN 'Phone: ' + ISNULL(PhoneNumber, '')
            WHEN 'Email' THEN 'Email: ' + ISNULL(Email, '')
            WHEN 'Mail' THEN 'Mail'
        END,
        'No Preference'
    ) AS PreferredContactDisplay
FROM dbo.Contacts;
```

**Search query (SARGable):**
```sql
-- Search by exact phone (including NULL)
DECLARE @SearchPhone VARCHAR(20) = NULL;
DECLARE @SearchEmail VARCHAR(200) = 'user@example.com';

SELECT ContactId, Name, PhoneNumber, Email, ContactPreference
FROM dbo.Contacts
WHERE (@SearchPhone IS NULL OR PhoneNumber = @SearchPhone)
  AND (@SearchEmail IS NULL OR Email = @SearchEmail);
-- SARGable: when @SearchPhone is provided, seeks on IX_Contacts_PhoneNumber
-- SARGable: when @SearchEmail is provided, seeks on IX_Contacts_Email
```

**Partial match (non-SARGable but correct):**
```sql
SELECT ContactId, Name, PhoneNumber, Email
FROM dbo.Contacts
WHERE PhoneNumber LIKE '%' + @SearchPattern + '%'
   OR Email LIKE '%' + @SearchPattern + '%';
-- Non-SARGable but acceptable for low-volume admin search
-- For high-performance partial search, use Full-Text Search
```

**Tradeoffs:**
- Normalizing empty strings to NULL via trigger ensures consistent state but adds write overhead.
- Filtered indexes on non-NULL phone/email reduce index size (no NULL rows in the index) and speed up non-NULL searches.
- COALESCE for display is acceptable in SELECT (final projection) — never in WHERE.
- The search query uses `(@p IS NULL OR col = @p)` pattern which is SARGable when the parameter is provided.

**What NOT to do:**
- Do not use `ISNULL(PhoneNumber, '') = ''` in WHERE — non-SARGable.
- Do not create a single index on both PhoneNumber and Email — they are queried separately.
- Do not store empty strings and NULL as different "no value" states — pick one (NULL) and enforce consistently.

</details>
