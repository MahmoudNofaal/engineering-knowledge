### Where This Fits

DELETE, TRUNCATE, and DROP are the three mechanisms for removing data in SQL Server. Every backend engineer must understand the operational and performance differences: DELETE is fully logged, row-by-row, trigger-firing DML — appropriate for removing specific rows but pathological for bulk removal because it generates N log records and holds row/ page locks. TRUNCATE is minimally logged DDL that deallocates entire extents — it removes all rows instantly (O(extents) instead of O(rows)) but cannot have a WHERE clause and requires higher permissions. DROP removes the table metadata entirely — the data pages are deallocated and the schema is gone. The wrong choice causes production incidents: using DELETE to clear a 100M row staging table instead of TRUNCATE fills the transaction log and blocks replication; using TRUNCATE where a WHERE clause is needed silently removes all data; dropping a table instead of truncating requires point-in-time recovery. Interviewers ask this to determine whether a candidate understands the logging, locking, and rollback behavior that distinguishes these three operations.

## Core Mental Model

DELETE, TRUNCATE, and DROP operate at different layers of the SQL Server storage engine with fundamentally different logging, locking, and rollback characteristics. **DELETE** is a DML statement that logs every row removal individually — each deleted row generates a log record containing the full row image for rollback. It fires triggers, respects foreign key constraints, increments the identity counter in reverse (actually it doesn't reset it), and acquires row or page locks. A DELETE of 100M rows generates 100M log records. **TRUNCATE** is a DDL statement that deallocates entire extents (8 pages = 64 KB at a time) by modifying the IAM (Index Allocation Map) and PFS (Page Free Space) pages. It logs only the extent deallocations — O(extents) instead of O(rows). TRUNCATE resets the identity seed, does NOT fire triggers, and requires ALTER permission on the table. Crucially, TRUNCATE IS transaction-rollbackable — if executed inside a transaction, ROLLBACK restores all extents from the deallocation records. **DROP** is a DDL statement that removes the table's metadata from the system catalog (sys.objects, sys.columns, sys.indexes). For a heap, it deallocates all extents. For a clustered index, it deallocates the B-tree leaf pages and intermediate pages. DROP is also transaction-rollbackable. The choice among them is driven by: (1) whether a row filter is needed (DELETE only), (2) how many rows are being removed (TRUNCATE for bulk, DELETE for targeted), and (3) whether the table or its data needs to persist (DROP removes the object).

### Classification

DELETE is **DML** (Data Manipulation Language), operates at the row level, and is fully logged. TRUNCATE and DROP are **DDL** (Data Definition Language), operate at the extent and metadata level respectively, and are minimally logged.

```mermaid
flowchart TD
    A[Need to remove data] --> B{Keep the table structure?}
    B -->|No| C[DROP TABLE - removes schema + data]
    B -->|Yes| D{Remove all rows or specific rows?}
    D -->|All rows| E{Need trigger firing?}
    D -->|Specific rows| F[DELETE with WHERE - row-by-row, fully logged]
    E -->|Yes - triggers required| G[DELETE without WHERE - fully logged, DML]
    E -->|No - triggers not needed| H[TRUNCATE TABLE - minimally logged, extent deallocation]
    C --> I{Will the schema be recreated later?}
    I -->|Yes - same schema| J[TRUNCATE preserves schema - preferred for staging]
    I -->|No - table is obsolete| K[DROP is correct - frees metadata]
    F --> L[Acquire X-locks on qualifying rows/pages]
    G --> M[Acquire X-locks on every page - all rows scanned]
    H --> N[Schema modification lock - brief]
    L --> O[Log: 1 log record per row - fully logged]
    M --> O
    N --> P[Log: extent deallocations only - O(extents)]
```

## Deep Mechanics

### How the Engine Executes This

**DELETE:**

1. **Row Resolution** — The WHERE clause is evaluated (SARGable rules apply — function on column = scan). Each qualifying row is identified.
2. **Row Lock** — An exclusive (X) lock is acquired on the row (if row-level locking is enabled) or the page. Higher isolation levels (SERIALIZABLE) acquire range locks (RangeS-S).
3. **Ghost Cleanup** — The row is marked as ghost-deleted: a bit in the row header is flipped. The row remains physically on the page. The ghost cleanup process (running every 5 seconds in the background) will reclaim the space when no active transaction references the page.
4. **Index Maintenance** — For each non-clustered index, the index row is also ghost-deleted. Each NC index delete is logged individually.
5. **Log Write** — The full row image (all columns) is written to the transaction log. This includes the logical operation code (LOP_DELETE_ROWS), the page ID, and the row data. For a 200-byte row, each DELETE log record is approximately 220–250 bytes.
6. **Constraint Validation** — FOREIGN KEY constraints are checked: any referencing rows in child tables will block the DELETE or cascade.
7. **Space Reclamation** — The ghost cleanup process runs in the background. It uses the ghost token count on each page: when all ghost records on a page are cleaned, the page may be deallocated (heap) or remain empty in the B-tree (clustered index — empty pages are deallocated only by `ALTER INDEX REORGANIZE`).

**TRUNCATE:**

1. **Schema Modification Lock** — TRUNCATE acquires a brief Sch-M (schema modification) lock on the table. This blocks all concurrent operations on the table.
2. **Extent Deallocation** — The storage engine iterates the IAM (Index Allocation Map) pages for the table and marks each allocated extent as deallocated in GAM (Global Allocation Map) and SGAM (Shared Global Allocation Map) pages. The data pages are NOT overwritten — they are simply returned to the free extent pool.
3. **Identity Reset** — If the table has an IDENTITY column, the seed is reset to the initial value (the seed value specified in CREATE TABLE).
4. **Log Write** — Only the extent deallocations are logged: LOP_MODIFY_HEADER (for GAM/SGAM page changes) and LOP_TRUNCATE_TABLE. This is approximately 8–12 log records for the entire operation, regardless of row count.
5. **Page Deallocation** — The pages are immediately available for allocation to other objects. No ghost cleanup is needed — the extents are returned to the free pool instantly.

**DROP:**

1. **Schema Modification Lock** — DROP acquires a Sch-M lock. This blocks all concurrent access.
2. **Metadata Removal** — The table's row in sys.objects is marked as dropped. sys.columns, sys.indexes, and other catalog views are updated. If the table has a dependency chain (views, procedures referencing it), the DROP fails unless objects are dropped first.
3. **Extent Deallocation** — For a heap, all extents are deallocated via GAM/SGAM updates. For a clustered index, the B-tree pages are deallocated. For a heap with NC indexes, each NC index's extents are also deallocated.
4. **Log Write** — Only the schema change and extent deallocations are logged. LOP_DROP_OBJECT for the metadata, plus LOP_MODIFY_HEADER for extent management.
5. **Object Removal** — The table no longer appears in catalog views. The space is returned to the database.

### SQL Visibility

```sql
-- DELETE with WHERE — row-by-row, fully logged
DELETE FROM dbo.Orders
WHERE OrderId = 10248;

-- DELETE without WHERE — all rows, fully logged, fires triggers
DELETE FROM dbo.Orders;

-- DROP — removes table entirely
DROP TABLE dbo.Orders;
```

### Execution Plan Analysis

**DELETE with WHERE on PK:**

- Plan: `[Clustered Index Seek] → [Clustered Index Delete]`
- The Clustered Index Delete operator marks the row as ghost-deleted.
- Logical Reads: ~5 (seek + delete mark)
- Plan cost: ~0.01

**DELETE without WHERE on 10M row table:**

- Plan: `[Clustered Index Scan] → [Clustered Index Delete]`
- The scan touches every page in the table. Each row is ghost-deleted.
- Logical Reads: ~120,000 (for a 10M row table at ~80 rows/page)
- Plan cost: ~70+ (scan) + proportional delete cost

**TRUNCATE TABLE:**

- Plan: No query plan is generated — TRUNCATE does not go through the optimizer. It executes at the storage engine level directly.
- There is no operator tree in the execution plan cache. The only visible artifact is the Sch-M lock acquisition in sys.dm_tran_locks.

**DROP TABLE:**

- Plan: No query plan. Similar to TRUNCATE — operates at metadata + storage engine level.

```
DELETE with WHERE (1 row, PK seek):
[Clustered Index Seek] → [Clustered Index Delete]
Cost: 0.008  |  Logical Reads: 5  |  Log: ~250 bytes

DELETE without WHERE (10M rows, full scan):
[Clustered Index Scan] → [Clustered Index Delete]
Cost: ~70  |  Logical Reads: ~120,000  |  Log: ~2.5 GB

TRUNCATE (10M rows):
No query plan  |  Logical Reads: ~12 (IAM + GAM/SGAM)  |  Log: ~2 KB
```

### Failure Modes

**DELETE with missing WHERE:** `DELETE FROM Orders` removes every row. No error. The transaction log grows by the full row size × row count. If executed inside a transaction without commit, rollback may take as long as the DELETE itself.

**TRUNCATE with FK references:** `TRUNCATE TABLE Orders` fails if a FOREIGN KEY references Orders. TRUNCATE cannot be used when FK relationships exist — SQL Server raises `Msg 4712` even if the child table is empty. Must DELETE or drop and recreate FKs first.

**DROP TABLE with dependency:** `DROP TABLE Orders` fails if a view, stored procedure, or function references Orders. The error lists the dependent object. Must DROP dependencies first.

**DELETE causing transaction log blowup:** A DELETE of 10M rows in a single transaction generates ~2.5 GB of log. In full recovery, if the log is not backed up during the operation, the log grows until it fills the disk.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Schema context
-- ============================================================
CREATE TABLE dbo.Orders
(
    OrderId      INT           NOT NULL IDENTITY(1,1),
    CustomerId   INT           NOT NULL,
    OrderDate    DATETIME2(0)  NOT NULL,
    Status       VARCHAR(20)   NOT NULL DEFAULT 'Pending',
    TotalAmount  DECIMAL(18,2) NOT NULL,
    ShippingAddr NVARCHAR(500) NULL,
    Notes        NVARCHAR(MAX) NULL,
    CreatedAt    DATETIME2(0)  NOT NULL DEFAULT SYSUTCDATETIME(),
    CONSTRAINT PK_Orders PRIMARY KEY CLUSTERED (OrderId)
);

CREATE TABLE dbo.OrderItems
(
    OrderItemId INT            NOT NULL IDENTITY(1,1),
    OrderId     INT            NOT NULL,
    ProductId   INT            NOT NULL,
    Quantity    INT            NOT NULL,
    UnitPrice   DECIMAL(18,2) NOT NULL,
    CONSTRAINT PK_OrderItems PRIMARY KEY CLUSTERED (OrderItemId),
    CONSTRAINT FK_OrderItems_Orders
        FOREIGN KEY (OrderId) REFERENCES dbo.Orders(OrderId)
);

-- ============================================================
-- Pattern 1: DELETE with WHERE — targeted row removal
-- ============================================================
-- Remove a single order
DELETE FROM dbo.Orders
WHERE OrderId = 10248;

-- Remove orders older than 2 years (batched — safe for production)
DECLARE @BatchSize INT = 1000;
DECLARE @RowsAffected INT = @BatchSize;

WHILE @RowsAffected = @BatchSize
BEGIN
    DELETE TOP (@BatchSize) FROM dbo.Orders
    WHERE Status = 'Archived' AND OrderDate < '2024-01-01';

    SET @RowsAffected = @@ROWCOUNT;
    CHECKPOINT;  -- in simple recovery — truncate log between batches
END

-- ============================================================
-- Pattern 2: DELETE using CTE for complex filtering
-- ============================================================
WITH OrdersToDelete AS (
    SELECT OrderId
    FROM dbo.Orders
    WHERE Status = 'Cancelled' AND OrderDate < '2023-01-01'
)
DELETE FROM o
FROM dbo.Orders AS o
INNER JOIN OrdersToDelete AS d ON o.OrderId = d.OrderId;

-- ============================================================
-- Pattern 3: DELETE with OUTPUT — audit trail
-- ============================================================
DELETE FROM dbo.Orders
OUTPUT DELETED.OrderId, DELETED.Status, GETUTCDATE() AS DeletedAt
INTO dbo.OrdersAudit (OrderId, Status, DeletedAt)
WHERE OrderId = 10248;

-- ============================================================
-- Pattern 4: TRUNCATE — clear all rows, reset identity
-- ============================================================
TRUNCATE TABLE dbo.Orders;
-- All rows removed. Identity reset to seed (1,1).
-- Cannot use with FK references or WHERE clause.

-- ============================================================
-- Pattern 5: DROP TABLE — remove schema and data
-- ============================================================
DROP TABLE IF EXISTS dbo.Orders;  -- SQL Server 2016+
DROP TABLE dbo.Orders;           -- pre-2016

-- Drop with dependency check
IF OBJECT_ID(N'dbo.Orders', N'U') IS NOT NULL
    DROP TABLE dbo.Orders;

-- ============================================================
-- Anti-pattern: DELETE all rows instead of TRUNCATE
-- ============================================================
-- ❌ Slow, fully logged, fires triggers, does not reset identity
-- DELETE FROM dbo.Orders;

-- ✅ Fast, minimally logged, resets identity, no trigger overhead
-- TRUNCATE TABLE dbo.Orders;

-- ============================================================
-- Anti-pattern: TRUNCATE when you need a WHERE clause
-- ============================================================
-- ❌ TRUNCATE has no WHERE — this removes all rows
-- TRUNCATE TABLE dbo.Orders;

-- ✅ Use DELETE with WHERE (batched if many rows)
-- DELETE FROM dbo.Orders WHERE OrderDate < '2024-01-01';
```

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL: DELETE syntax is identical
DELETE FROM orders WHERE order_id = 10248;
DELETE FROM orders WHERE status = 'Archived' AND order_date < '2024-01-01';

-- PostgreSQL: DELETE with RETURNING (equivalent to OUTPUT)
DELETE FROM orders
WHERE order_id = 10248
RETURNING order_id, status, deleted_at;

-- PostgreSQL: TRUNCATE — identical statement
TRUNCATE TABLE orders;

-- PostgreSQL: TRUNCATE with CASCADE (SQL Server does not support)

-- CASCADE truncates all tables with FK references to orders

-- PostgreSQL: DROP TABLE with CASCADE
DROP TABLE IF EXISTS orders CASCADE;
-- SQL Server: requires manual DROP of dependent objects

-- PostgreSQL: TRUNCATE can be used with FK if CASCADE is specified
-- SQL Server: TRUNCATE fails with FK regardless — must DELETE first

-- PostgreSQL: TRUNCATE does not reset GENERATED AS IDENTITY by default
-- Use RESTART IDENTITY:
TRUNCATE TABLE orders RESTART IDENTITY;
```

## Gotchas and Production Pitfalls

### DELETE Without WHERE — Full Table Delete

**Pitfall:** Running a DELETE without a WHERE clause, wiping every row from the table.

```sql
-- ❌ Deletes every row — no error, no confirmation
DELETE FROM dbo.Orders;
```

**Symptom:** All rows are immediately ghost-deleted. The transaction log grows by the size of every row (full before-image for rollback). All queries on the table return empty result sets. If the transaction is not committed immediately, the rollback takes as long as the original DELETE. If committed, every row is lost.

**Fix:**

```sql
-- ✅ Always write SELECT COUNT(*) first to verify row count
SELECT COUNT(*) FROM dbo.Orders;
-- Then DELETE with WHERE (even if you intend to delete all,
-- use TRUNCATE instead)

-- ✅ Use BEGIN TRAN + ROLLBACK pattern
BEGIN TRAN;
DELETE FROM dbo.Orders;
SELECT @@ROWCOUNT AS DeletedRows;
ROLLBACK TRAN;
-- Verify @@ROWCOUNT before committing
```

**Cost of not fixing:** An engineer connected to production runs `DELETE FROM Orders WHERE Status = 'Pending'` but accidentally runs it on the wrong database (a copy with identical table names but no WHERE clause). Every row in all tables is deleted. The 2-hour restore from backup causes a 4-hour outage for the entire ordering system.

### Using DELETE Instead of TRUNCATE for Bulk Removal

**Pitfall:** Using DELETE (without WHERE) to clear a large staging table instead of TRUNCATE.

```sql
-- ❌ DELETE on 50M row staging table — 50M log records, 50M ghost-deletes
DELETE FROM dbo.Orders_Staging;
```

**Symptom:** The DELETE runs for 15 minutes. The transaction log grows by gigabytes. The log reuse wait is `LOG_BACKUP` (the log cannot be truncated because the DELETE is still active). Ghost cleanup processes 50M rows in the background, causing I/O pressure. The SSIS/ETL pipeline that needs to repopulate the staging table is blocked for the duration.

**Fix:**

```sql
-- ✅ TRUNCATE clears all rows in ~100ms regardless of row count
TRUNCATE TABLE dbo.Orders_Staging;
```

**Cost of not fixing:** An ETL job that runs hourly uses DELETE to clear the staging table. Each hour: 15 minutes of DELETE, 5 minutes of ghost cleanup overhead, and 2 GB of log that must be backed up. The total ETL window stretches from 20 minutes to 45 minutes. During peak hours, the ETL overlaps with OLTP load, causing log disk contention and blocking.

### TRUNCATE Fails on FK-Referenced Table

**Pitfall:** Attempting to TRUNCATE a table that is referenced by a FOREIGN KEY — even if the child table is empty.

```sql
-- ❌ Fails with Msg 4712 even if OrderItems is empty
TRUNCATE TABLE dbo.Orders;

-- ERROR: Cannot truncate table 'Orders' because it is being referenced
-- by a FOREIGN KEY constraint 'FK_OrderItems_Orders'.
```

**Symptom:** The TRUNCATE statement fails at compile time with error 4712. The developer discovers the FK and either (a) tries to DELETE instead (slow) or (b) manually drops and recreates FKs (risky in production).

**Fix:**

```sql
-- ✅ Option 1: Use DELETE (with TOP + WHILE batch for large tables)
DELETE FROM dbo.Orders WHERE OrderId IN (SELECT OrderId FROM dbo.Orders);

-- ✅ Option 2: Drop FK, TRUNCATE, recreate FK (requires downtime window)
ALTER TABLE dbo.OrderItems DROP CONSTRAINT FK_OrderItems_Orders;
TRUNCATE TABLE dbo.Orders;
ALTER TABLE dbo.OrderItems ADD CONSTRAINT FK_OrderItems_Orders
    FOREIGN KEY (OrderId) REFERENCES dbo.Orders(OrderId);

-- ✅ Option 3: Disable FK check (not recommended)
ALTER TABLE dbo.OrderItems NOCHECK CONSTRAINT FK_OrderItems_Orders;
-- TRUNCATE still fails — NOCHECK does not bypass TRUNCATE restriction
-- Must use DELETE if FK exists
```

**Cost of not fixing:** A data archival job fails silently every night because an FK was added to the Orders table in a recent migration. The job tried to TRUNCATE — it now writes an error to the job log and exits. The Orders table grows unbounded. Disk space is exhausted after 3 weeks.

### TRUNCATE Cannot Be Partially Rolled Back

**Pitfall:** Assuming TRUNCATE within a transaction can be selectively applied or that it fires triggers (triggers that audit deletions are not triggered).

```sql
-- ❌ Developer expects DELETE triggers to fire
BEGIN TRAN;
TRUNCATE TABLE dbo.Orders;
-- No ON DELETE triggers fire — the audit table has zero records
-- Even though 5M orders were removed
ROLLBACK TRAN;  -- Orders are restored, but no audit trail was created
```

**Symptom:** The compliance audit shows zero deletion events, but the Orders table was temporarily empty. The trigger-based audit system did not capture the TRUNCATE because TRUNCATE does not fire DML triggers. If the TRUNCATE is inside a transaction and rolled back, the data returns but the trigger-based audit still has no record.

**Fix:**

```sql
-- ✅ Use DELETE with OUTPUT for auditable bulk removal
DECLARE @BatchSize INT = 1000;
DECLARE @RowsAffected INT = @BatchSize;

WHILE @RowsAffected = @BatchSize
BEGIN
    DELETE TOP (@BatchSize) FROM dbo.Orders
    OUTPUT DELETED.OrderId, DELETED.Status, GETUTCDATE() AS DeletedAt
    INTO dbo.OrdersAudit (OrderId, Status, DeletedAt)
    WHERE Status = 'Archived';

    SET @RowsAffected = @@ROWCOUNT;
END

-- ✅ Or use a TRUNCATE-only table and log deletions separately
TRUNCATE TABLE dbo.Orders;  -- fast, no audit
-- But this requires a separate logging mechanism outside the DB
```

**Cost of not fixing:** A GDPR right-to-erasure script uses TRUNCATE to clear user data quickly. The DPO (Data Protection Officer) requires an audit trail of all deletions for regulatory compliance. The TRUNCATE leaves no audit trail. During a regulatory audit, the company cannot demonstrate that deletions occurred properly, resulting in a compliance violation fine.

### DROP TABLE Without Double-Checking Database Context

**Pitfall:** Running `DROP TABLE Orders` on the wrong database or pointing at the wrong table name.

```sql
-- ❌ Connected to Production instead of Development
DROP TABLE dbo.Orders;
```

**Symptom:** The table and all its data are gone. The schema is removed. The table no longer appears in sys.objects. All stored procedures, views, and functions that reference Orders will fail on execution. Recovery requires a point-in-time restore from the last full backup.

**Fix:**

```sql
-- ✅ Always verify database context before DROP
SELECT DB_NAME() AS CurrentDatabase;
SELECT SCHEMA_NAME(schema_id) AS SchemaName, name AS TableName
FROM sys.tables WHERE name = 'Orders';

-- ✅ Use DROP TABLE IF EXISTS (SQL Server 2016+) — same risk but more explicit
DROP TABLE IF EXISTS dbo.Orders;

-- ✅ Use Object_ID check for idempotent scripts
IF OBJECT_ID(N'dbo.Orders', N'U') IS NOT NULL
    DROP TABLE dbo.Orders;

-- ✅ Never use DROP in application code — use migration tools
-- (EF Core migrations, DbUp, Flyway)
```

**Cost of not fixing:** A CI/CD pipeline runs a migration script on the wrong environment. The script contains `DROP TABLE Orders` followed by a CREATE TABLE with a different schema. The production Orders table with 5M active orders is dropped. Data recovery takes 6 hours from the last full backup. 6 hours of transactions (orders, payments, shipments) are lost.

### DELETE with Non-SARGable WHERE on Large Table

**Pitfall:** Using a non-SARGable WHERE clause in a DELETE on a table with millions of rows, causing a full table scan and blocking all concurrent access.

```sql
-- ❌ Non-SARGable — function on filtered column
DELETE FROM dbo.Orders
WHERE YEAR(OrderDate) = 2024 AND MONTH(OrderDate) = 1;
```

**Symptom:** The DELETE scans the entire table (120,000 logical reads for 10M rows). Every page scanned is X-locked. Concurrent SELECT queries block. The DELETE takes 45 seconds.

**Fix:**

```sql
-- ✅ SARGable range predicate
DELETE FROM dbo.Orders
WHERE OrderDate >= '2024-01-01' AND OrderDate < '2024-02-01';
-- Uses Index Seek on IX_Orders_OrderDate
-- Locks only the qualifying pages
```

**Cost of not fixing:** A daily data retention job uses `DELETE FROM Logs WHERE YEAR(CreatedAt) = 2024`. The Logs table has 50M rows. The DELETE scans every page, locks the entire table for 90 seconds, and blocks all application activity that writes to or reads from the Logs table. The on-call engineer investigates an "application unresponsive" alert and finds a massive blocking chain rooted in the DELETE statement.

### Spoken Answers

**Q: Why does TRUNCATE not fire triggers, and what are the implications for audit compliance?**

> **Great answer:** TRUNCATE does not fire triggers because it is a DDL statement, not a DML statement. DML triggers (AFTER DELETE, INSTEAD OF DELETE) are designed to fire per-row — but TRUNCATE does not operate at the row level. TRUNCATE deallocates entire extents by modifying the IAM, GAM, and SGAM pages — it never reads or processes individual rows. The engine cannot logically fire a per-row trigger because it does not iterate rows. From a compliance and auditing perspective, this is critical: if your audit system depends on a DELETE trigger to log row removals, TRUNCATE bypasses it completely. A developer who uses TRUNCATE for a GDPR right-to-erasure workflow will leave zero audit trail. The fix is to either: (a) use a batched DELETE with OUTPUT INTO an audit table to maintain the audit trail while achieving acceptable performance, or (b) use a server-level or database-level DDL trigger (which fires on TRUNCATE as a DDL event) but capturing the row-level detail requires a different architecture. This is a common interview trap — I always verify the trigger behavior before choosing TRUNCATE for any compliance-sensitive operation.

---

**Q: Can TRUNCATE be rolled back? Explain the mechanism.**

> **Great answer:** Yes, TRUNCATE is fully rollbackable within a transaction. This is a common misconception — many engineers believe TRUNCATE cannot be rolled back because it is a DDL statement. In SQL Server, all DDL statements are transaction-scoped. When TRUNCATE executes, it logs extent deallocations to the transaction log using LOP_MODIFY_HEADER records for GAM and SGAM pages. These records include the page IDs of the deallocated extents. If a ROLLBACK is issued, the recovery process reads these log records in reverse, walks the extent allocation structures, and re-marks the extents as allocated — effectively restoring the data. The data pages themselves are never overwritten by TRUNCATE (only the allocation bitmaps change), so rollback is fast — proportional to the number of extents, not the row count. The key difference from DELETE rollback is mechanism: DELETE rollback reads the full row images from log records and re-inserts them row by row (O(rows)); TRUNCATE rollback flips bits in allocation pages (O(extents)). For a 100M row table, TRUNCATE rollback takes milliseconds; DELETE rollback takes minutes.

### Interview Trigger

The distinguishing question: "You're clearing a 200GB staging table nightly. DELETE takes 45 minutes and fills the transaction log. TRUNCATE takes 2 seconds. What do you choose?" A junior candidate says TRUNCATE without caveats. A mid-level candidate says TRUNCATE but mentions the limitations (no FK, no triggers, no WHERE). A senior candidate says "it depends on whether I need an audit trail, whether FKs exist, and whether I'm in a transactional replication topology where TRUNCATE is not replicated by default. If I need fast cleanup and audit, I'd use a batched DELETE with OUTPUT captured to a separate audit table, or I'd use TRUNCATE inside a transaction with a DDL trigger to log the event. If there are FKs, I either drop and recreate them or use batched DELETE. If the table is replicated, I'd verify whether TRUNCATE is supported by the replication type." The follow-up: "What about ghost cleanup after DELETE?" — a candidate who mentions the ghost cleanup process, its 5-second cycle, and its impact on read I/O during heavy DELETE operations demonstrates depth.

### Application Checklist

- [ ] Correct operation chosen: DELETE (filtered), TRUNCATE (all rows), DROP (remove schema)
- [ ] DELETE WHERE clause is SARGABLE — no functions on filtered columns
- [ ] Large DELETE is batched with TOP + WHILE loop (if >10K rows)
- [ ] TRUNCATE confirmed to have no FK references — SELECT OBJECT_ID referenced by sys.foreign_keys
- [ ] Triggers considered — if audit relies on DELETE triggers, do not use TRUNCATE
- [ ] Identity reset behavior confirmed — TRUNCATE resets, DELETE does not
- [ ] Transaction log growth monitored — DELETE of large table may fill the log
- [ ] Rollback plan in place — BEGIN TRAN + ROLLBACK test before production execution
- [ ] DROP TABLE only in migration scripts — never in application code
- [ ] EF Core: ExecuteDeleteAsync preferred over tracked entity loop for batch removal

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|TRUNCATE: O(extents) vs O(rows) performance|No WHERE clause, no triggers, no FK support|
|DELETE with WHERE: precise row targeting|Fully logged — 1 log record per row|
|Batced DELETE: limits lock duration to ~50 ms|More complex code (WHILE loop + @@ROWCOUNT)|
|DROP TABLE: instant schema + data removal|Schema lost — must recreate from migration script|
|OUTPUT clause: built-in audit trail|Extra log writes proportional to output rows|

### Scale Thresholds

- DELETE without WHERE becomes unacceptable above **~10K rows** — the transaction log grows visibly (~5 MB for 10K rows) and blocking becomes measurable.
- TRUNCATE is always O(extents) — relevant at **any row count above 0**, but the difference is dramatic above **~100K rows** (1800 ms for DELETE vs 25 ms for TRUNCATE).
- Batched DELETE with TOP becomes necessary above **~10K rows** to prevent long X-lock durations.
- DROP TABLE should never be executed in application code — only in migration tools with environment-specific verification.
- TRUNCATE with FK replication topologies: transactional replication does not replicate TRUNCATE by default; merge replication does. Verify before using.

### Conceptual Questions

1. What SQL categories do DELETE, TRUNCATE, and DROP belong to, and what determines each classification?
2. Why does TRUNCATE log only extent deallocations instead of row deletions, and which allocation map pages are modified?
3. Can TRUNCATE be rolled back inside an explicit transaction? What mechanism enables this?
4. Why does TRUNCATE fail with error 4712 when a FOREIGN KEY references the table, but DELETE succeeds?
5. Does EF Core's `ExecuteDeleteAsync` use DELETE or TRUNCATE internally?
6. How would you use Dapper to capture deleted row data for an audit log using TRUNCATE?
7. What is the execution plan for a DELETE without a WHERE clause on a 10M row table?
8. At what row count does TRUNCATE become meaningfully faster than DELETE, and what is the throughput ratio at 1M rows?
9. What index supports a filtered DELETE with WHERE on Status and OrderDate, and how does the index choice affect the DELETE performance?
10. Explain in 60 seconds, for a senior interviewer, when you would choose TRUNCATE over DELETE and what operational risks you verify before doing so.

<details>
<summary>Answers</summary>

1. **DELETE** is DML (Data Manipulation Language) — operates on individual rows, fires triggers, fully logged. **TRUNCATE** is DDL (Data Definition Language) — operates on extents via allocation map pages. **DROP** is DDL — removes table metadata from system catalog. The DML/DDL distinction determines: logging behavior, trigger firing, permissions required, and transaction scope behavior.

2. TRUNCATE modifies GAM (Global Allocation Map), SGAM (Shared GAM), and IAM (Index Allocation Map) pages to mark extents as deallocated. It never reads individual rows. Only the bitmap changes in these pages are logged (LOP_MODIFY_HEADER). The number of log records is proportional to the number of extents (pages ÷ 8), not the number of rows (potentially rows ÷ 800).

3. **Yes, TRUNCATE is fully rollbackable.** It logs the extent deallocation operations. On ROLLBACK, the recovery process re-marks the extents as allocated by reversing the GAM/SGAM bit changes. The data pages are never overwritten, so rollback is O(extents) — milliseconds for any table size. This is a common interview trick — many candidates incorrectly believe TRUNCATE cannot be rolled back.

4. SQL Server's TRUNCATE operation does not evaluate individual rows, so it cannot check whether child rows exist in FK-referenced tables. The engine raises error 4712 at compile time to prevent data integrity violations. DELETE evaluates rows one by one and can check FKs per row. Even if the child table is empty, TRUNCATE fails — the engine does not check FK existence vs data existence.

5. `ExecuteDeleteAsync` generates a **DELETE** statement (the `DELETE FROM [o] FROM [Orders] AS [o] WHERE ...` T-SQL syntax). EF Core does not support TRUNCATE directly — only via `ExecuteSqlRawAsync("TRUNCATE TABLE ...")`. Using `ExecuteDeleteAsync` is safe and transactional; using raw TRUNCATE bypasses the change tracker and cannot be part of EF Core's automatic migration pipeline.

6. TRUNCATE does not support OUTPUT — it's DDL, not DML, so there is no `deleted` virtual table. To audit a TRUNCATE with Dapper, you must either: (a) use a DDL trigger that captures the TRUNCATE event metadata (who, when, which table) but not the row data; (b) capture row data via SELECT before the TRUNCATE; or (c) use a batched DELETE with OUTPUT instead of TRUNCATE when row-level audit is required.

7. The execution plan is: `[Clustered Index Scan] → [Clustered Index Delete]`. The scan reads every data page (for a 10M row table at 80 rows/page, that's ~125,000 logical reads). The Delete operator ghost-deletes each qualifying row. Without a WHERE clause, every row qualifies. The plan does NOT include a spool — Halloween Protection is not needed because DELETE does not read a modified value (there are no SET operations to create the feedback loop).

8. TRUNCATE is meaningfully faster than DELETE above **~1000 rows**. At 100K rows: DELETE ~1800 ms vs TRUNCATE ~25 ms (72x). At 1M rows: DELETE ~18,000 ms vs TRUNCATE ~30 ms (600x). The ratio increases with row count because DELETE scales O(rows) while TRUNCATE scales O(extents) — at 10M rows the ratio exceeds 1000x.

9. A filtered DELETE with `WHERE Status = 'Archived' AND OrderDate < @Cutoff` benefits from: `CREATE INDEX IX_Orders_DeleteFilter ON dbo.Orders (OrderDate, Status)`. This enables a SARGable Index Seek to locate only the qualifying rows without scanning the entire table. If this index is missing, the DELETE forces a Clustered Index Scan (125K logical reads for 10M rows). With the index: ~145 logical reads.

10. "I choose TRUNCATE over DELETE when I need to remove all rows from a table and the speed difference is enormous — 25 ms for TRUNCATE vs 18 seconds for DELETE at 1M rows. Before using TRUNCATE, I verify three things: (1) No FOREIGN KEY constraints reference the table — TRUNCATE fails with error 4712 if they exist; (2) I don't need DELETE triggers to fire for audit or compliance — TRUNCATE does not fire per-row triggers; (3) The identity column reset is acceptable for the use case — TRUNCATE resets the identity seed, DELETE does not. I also check whether transactional replication is in use, since TRUNCATE is not replicated by default in transactional replication. If I need audit, I use a batched DELETE with OUTPUT capturing the deleted rows. If FKs exist but I must clear all data, I drop the FKs, TRUNCATE, and recreate them in a maintenance window. At a concrete performance level: a 50M row staging table that takes 45 minutes to DELETE and produces 12 GB of log takes 30 ms to TRUNCATE with 2 KB of log."

</details>

**Challenge 2 — Fix the performance problem**

```sql
-- This runs every hour. It takes 8 minutes on a 25M row table.
SET STATISTICS IO ON;
DELETE FROM dbo.AuditLog
WHERE YEAR(CreatedAt) = 2024 AND MONTH(CreatedAt) = 1;
```

<details>
<summary>Solution</summary>

**Root cause:** Two performance problems. (1) The WHERE clause is non-SARGable — `YEAR(CreatedAt) = 2024 AND MONTH(CreatedAt) = 1` cannot use an index seek because the function wraps the column.     WHERE CreatedAt >= '2024-01-01' AND CreatedAt < '2024-02-01';

```sql
    SET @RowsAffected = @@ROWCOUNT;
    CHECKPOINT;  -- in simple recovery
END

-- Index to create:
CREATE INDEX IX_AuditLog_CreatedAt ON dbo.AuditLog (CreatedAt);
```

**After fix — logical reads:** ~5 per batch (Index Seek + Delete) vs 320,000. **Execution time:** ~5 seconds (batched) from 8 minutes.

**Challenge 3 — Explain the execution plan behavior**

A DELETE of all rows from a 50M row table produces the plan:
`[Clustered Index Scan] → [Clustered Index Delete]`

The DELETE takes 3 minutes. A TRUNCATE of the same table takes 40 ms. What is the difference in the storage engine operations that makes TRUNCATE so much faster?

<details>
<summary>Solution</summary>

**DELETE path:** The Clustered Index Scan reads every data page in the index — approximately 625,000 pages for 50M rows at 80 rows/page. For each qualifying row (all rows, since no WHERE), the Clustered Index Delete operator ghost-deletes the row by flipping a bit in the row header. Each ghost-delete generates a full row image log record (~250 bytes per row, ~12.5 GB total). The rows remain on the pages; the ghost cleanup process reclaims them in the background. During the DELETE, the scan holds shared locks, and the delete holds exclusive locks. All 625,000 pages are touched.

**TRUNCATE path:** TRUNCATE reads the IAM pages for the table to identify which extents are allocated. It then modifies the GAM and SGAM bitmaps to mark those extents as free. It does NOT read or modify any data pages. The number of pages touched is O(extents) — for 50M rows, approximately 78,125 extents ÷ 64 pages per GAM page = ~1220 GAM pages. Total logical reads: ~2000 vs 625,000. The log writes are ~8–12 records vs 50 million.

**Key difference:** DELETE operates at the row level (O(rows)), TRUNCATE at the extent level (O(pages/8)). The gap widens with more rows because DELETE's log scales O(rows) and TRUNCATE's log scales O(extents).

</details>

**Challenge 4 — Diagnose the concurrency problem**

A scheduled job runs `DELETE FROM Orders WHERE Status = 'Archived'` every hour. It deletes ~50K rows. During the DELETE, all application read queries time out. `sys.dm_exec_requests` shows `LCK_M_S` waits on the Orders table with `wait_duration_ms` of 30,000+. Propose a fix.

<details>
<summary>Solution</summary>

**Root cause:** Even with a WHERE clause, the DELETE scans the clustered index to find 'Archived' rows (unless a covering index on Status exists). Every page scanned is locked with an X lock. The scan acquires locks on ALL pages it reads, not just pages with qualifying rows. Concurrent SELECT queries need Sch-S (schema stability) locks, which are incompatible with X locks.

**Fix:**

```sql
-- 1. Add a covering index on Status so only qualifying pages are locked
CREATE INDEX IX_Orders_Status ON dbo.Orders (Status) INCLUDE (OrderDate, TotalAmount);

-- 2. Batch the DELETE to release locks between batches
DECLARE @BatchSize INT = 1000;
DECLARE @RowsAffected INT = @BatchSize;

WHILE @RowsAffected = @BatchSize
BEGIN
    DELETE TOP (@BatchSize) FROM dbo.Orders
    WHERE Status = 'Archived';

    SET @RowsAffected = @@ROWCOUNT;
    WAITFOR DELAY '00:00:00:100';
END
```

**Detection query:**
```sql
SELECT wait_type, wait_duration_ms, blocking_session_id, session_id
FROM sys.dm_exec_requests
WHERE wait_type LIKE 'LCK_M_%';
```

**In EF Core:**
```csharp
int rowsAffected;
do
{
    rowsAffected = await dbContext.Orders
        .Where(o => o.Status == "Archived")
        .Take(1000)
        .ExecuteDeleteAsync(cancellationToken);
    await Task.Delay(100, cancellationToken);
}
while (rowsAffected >= 1000);
```

**Challenge 5 — Design the cleanup strategy**

**Scenario:** An Orders table (200 GB, 100M rows) has a compliance requirement to retain data for 7 years. Monthly archival deletes rows older than 7 years: `DELETE FROM Orders WHERE OrderDate < DATEADD(YEAR, -7, GETUTCDATE())`. The DELETE currently takes 4 hours and causes blocking. The system is OLTP with 5000 requests/second. Design a cleanup strategy that completes within a 1-hour maintenance window with minimal blocking.

<details>
<summary>Solution</summary>

```sql
-- Strategy: Partitioned table + SWITCH + TRUNCATE
--
-- 1. Create a partitioning function and scheme by year
-- 2. Each month: create a new partition for the coming year
-- 3. For cleanup: SWITCH the oldest partition to a staging table
-- 4. TRUNCATE the staging table

-- Partition function (annual)
CREATE PARTITION FUNCTION PF_Orders_Year (DATETIME2(0))
AS RANGE RIGHT FOR VALUES (
    '2020-01-01', '2021-01-01', '2022-01-01',
    '2023-01-01', '2024-01-01', '2025-01-01',
    '2026-01-01', '2027-01-01'
);

CREATE PARTITION SCHEME PS_Orders_Year
AS PARTITION PF_Orders_Year ALL TO ([PRIMARY]);

-- Create partitioned table
CREATE TABLE dbo.Orders (
    OrderId      INT           NOT NULL,
    CustomerId   INT           NOT NULL,
    OrderDate    DATETIME2(0)  NOT NULL,
    Status       VARCHAR(20)   NOT NULL,
    TotalAmount  DECIMAL(18,2) NOT NULL,
    CreatedAt    DATETIME2(0)  NOT NULL DEFAULT SYSUTCDATETIME(),
    CONSTRAINT PK_Orders PRIMARY KEY CLUSTERED (OrderId, OrderDate)
) ON PS_Orders_Year (OrderDate);

-- Monthly cleanup: switch old partition to staging, truncate staging
-- Step 1: Create staging table with same schema on filegroup
CREATE TABLE dbo.Orders_Staging (
    OrderId      INT           NOT NULL,
    CustomerId   INT           NOT NULL,
    OrderDate    DATETIME2(0)  NOT NULL,
    Status       VARCHAR(20)   NOT NULL,
    TotalAmount  DECIMAL(18,2) NOT NULL,
    CreatedAt    DATETIME2(0)  NOT NULL
) ON [PRIMARY];

-- Step 2: SWITCH (metadata-only operation — ~100 ms)
ALTER TABLE dbo.Orders SWITCH PARTITION 1 TO dbo.Orders_Staging;

-- Step 3: TRUNCATE staging (instant — no data reading)
TRUNCATE TABLE dbo.Orders_Staging;

-- If partitioning is not feasible, use batched DELETE with Read Committed Snapshot:
DECLARE @BatchSize INT = 5000;
DECLARE @RowsDeleted INT = @BatchSize;
DECLARE @CutoffDate DATETIME2(0) = DATEADD(YEAR, -7, GETUTCDATE());

WHILE @RowsDeleted = @BatchSize
BEGIN
    DELETE TOP (@BatchSize) FROM dbo.Orders WITH (READCOMMITTEDLOCK)
    WHERE OrderDate < @CutoffDate;

    SET @RowsDeleted = @@ROWCOUNT;
    WAITFOR DELAY '00:00:00:200';
END
```

**Tradeoffs:** Partitioning adds schema complexity but allows instant data removal with zero blocking during the SWITCH. Batched DELETE is simpler but still generates log and may take 30–60 minutes. For a 200 GB table, partitioning is the correct long-term solution.

</details>
