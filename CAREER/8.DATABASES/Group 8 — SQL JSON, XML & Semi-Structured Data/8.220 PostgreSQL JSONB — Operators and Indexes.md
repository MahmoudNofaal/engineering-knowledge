### Where This Fits

PostgreSQL JSONB is a binary JSON storage format with native indexing, a rich operator set (containment, existence, concatenation, path access), and deep .NET integration via Npgsql. A .NET backend engineer encounters JSONB when using PostgreSQL as the primary database with Npgsql and EF Core, when migrating from SQL Server (which has weaker JSON support), or when building applications that need flexible schema evolution without EAV tables. JSONB's GIN index provides the equivalent of a full-text-search-style inverted index over JSON keys and values — enabling path-agnostic querying that SQL Server JSON cannot match without computed columns. The critical distinctions from SQL Server JSON are: PostgreSQL stores JSONB in a decomposed binary format (not NVARCHAR), supports the `@>` containment operator with GIN index seeks, and allows expression indexes on specific JSON paths. The interview signal is high for PostgreSQL roles: the @> operator, GIN index internals, and Npgsql's owned entity mapping for JSONB are frequent topics.

## Core Mental Model

JSONB stores JSON data in a **decomposed binary format** — the input JSON text is parsed, sorted by key, and stored as a tree of key-value pairs where each value is typed (string, number, boolean, null, array, object). This decomposition eliminates duplicate keys, normalises whitespace, and enables key-level indexing. The storage format is similar to the XML binary format but optimised for key-value access rather than path navigation. JSONB operators operate on this decomposed tree: `->` extracts a field and returns JSONB (preserving the type), `->>` extracts a field as SQL text, `@>` checks if the left JSONB contains the right JSONB (structural containment), `?` checks key existence, `?|` checks if any key exists, `?&` checks if all keys exist, and `||` concatenates two JSONB values. The GIN (Generalized Inverted Index) on JSONB decomposes each JSONB value into a set of `(key, value)` tokens and creates an inverted index mapping each token to the rows that contain it. This enables indexed searches with `@>`, `?`, `?|`, and `?&` without knowing the exact path in advance. The critical invariant: **JSONB indexes are path-agnostic — a GIN index on a JSONB column indexes every key at every path level. An expression index on a specific path `(data->'field')` provides a traditional B-tree for range queries on that specific field.** This is fundamentally different from SQL Server's JSON model, where indexing requires a computed column with a specific JSON_VALUE expression.

### Classification

JSONB is a **native PostgreSQL data type** (not an add-on) in the `postgres` schema. It belongs to the **binary storage engine** layer — the JSON text is decomposed at insert time and stored as a tree of typed nodes in the heap. Operators belong to the **expression evaluation** engine and are designed for index utilisation: the GIN index operator classes support `@>`, `?`, `?|`, `?&` for indexed access. The `->` and `->>` operators are not directly indexable by GIN but can be indexed via expression B-tree indexes. JSONB is **always SARGable with the right index**: the `@>` operator with a GIN index performs an inverted index seek. The `->` operator with an expression B-tree index performs a standard B-tree seek. Without indexes, all JSONB operators require sequential scans with per-row JSONB tree evaluation.

## Deep Mechanics

### How the Engine Executes This

1. **JSONB parsing and decomposition** — When a JSON string is cast to JSONB (or inserted into a JSONB column), the PostgreSQL parser validates the JSON and decomposes it into a binary tree. Each object becomes a sorted array of key-value pairs (sorted by key for efficient binary search). Each value is internally typed: string, number (stored as numeric or int8 depending on precision), boolean, null, array, or object. Arrays are stored as sequential element lists. The binary format is ~10-20% smaller than the original JSON text because of whitespace removal and integer encoding.

2. **GIN index tokenisation** — When a GIN index exists on a JSONB column, every INSERT or UPDATE triggers tokenisation of the JSONB tree into a set of `(key, value)` tokens. Each token consists of: the full path (e.g., `"Address.State"`), the value type (string, number, boolean, null), and the value itself (for scalar types). For `{"Address": {"State": "WA", "City": "Seattle"}}`, the tokens include: `"Address"` (key existence), `"Address.State"` (path), `"Address.State.WA"` (path + value), `"Address.City"` (path), `"Address.City.Seattle"` (path + value). Each token is hashed and inserted into the GIN posting list for that token.

3. **GIN index structure** — A GIN index consists of: (a) an entry tree (B-tree) mapping each unique token to a posting list location, and (b) posting lists (stored in separate pages) containing the TIDs (tuple IDs) of rows containing that token. The posting lists can be large for common tokens (e.g., `"type"` in a collection where every document has a `type` field). PostgreSQL uses a pending list for fast inserts (temporary storage that is batch-merged into the main index).

4. **@> containment evaluation** — When the `@>` operator is used, PostgreSQL: (a) decomposes the right-hand operand (the "contained" JSONB) into tokens, (b) for each token, looks up the posting list in the GIN index, (c) intersects the posting lists to find rows that contain all tokens. The containment check is: "does the left document contain all tokens that exist in the right document?" This is structural containment — the right document's structure must be a subset of the left document's structure.

5. **? key existence evaluation** — The `?` operator checks if a top-level key exists in the JSONB. With a GIN index, PostgreSQL looks up the token for that key name in the GIN entry tree and reads the posting list. This is a single B-tree lookup + TID scan.

6. **Expression B-tree index evaluation** — For `CREATE INDEX ON tab USING BTREE ((data->>'field'))`, PostgreSQL creates a standard B-tree on the extracted text value. Queries like `WHERE data->>'field' = 'value'` or `WHERE data->>'field' > 'threshold'` can use this index for equality or range seeks. The expression is evaluated once at index time and the result stored in the B-tree.

7. **-> (JSONB field access)** — The `->` operator accesses a field by key in the decomposed binary tree. Because the keys are sorted within each object, PostgreSQL uses binary search (O(log k) where k = number of keys) to locate the field. This is the fastest JSONB operation — no parsing, no token lookup.

8. **Write path** — On INSERT of a JSONB value:
   - The JSONB binary tree is stored in the heap
   - For GIN index: each distinct token is inserted into the pending list (fast mode) or directly into the entry tree and posting lists
   - For expression B-tree: the expression is evaluated and the result inserted into the B-tree
   On UPDATE of a JSONB column (which is a new tuple in MVCC):
   - Old GIN index entries are marked dead (vacuum cleans later)
   - New tokens are inserted for the new JSONB value
   On DELETE: tokens are marked dead

```sql
-- JSONB Operators: -> and ->>
-- ============================================================
-- -> returns JSONB (preserves type, includes quotes for strings)
SELECT
    OrderId,
    Metadata->'source' AS source_jsonb,       -- "web" (JSONB string)
    Metadata->'priority' AS priority_jsonb,    -- "high" (JSONB string)
    Metadata->'shipping' AS shipping_jsonb,    -- {"carrier": "UPS", ...} (JSONB object)
    Metadata->'items' AS items_jsonb,          -- [{...}, {...}] (JSONB array)
    Metadata->'flags' AS flags_jsonb           -- ["fragile", "rush"] (JSONB array)
FROM dbo.Orders;

-- ->> returns SQL TEXT (unquoted string)
SELECT
    OrderId,
    Metadata->>'source' AS source_text,         -- web (TEXT)
    Metadata->>'priority' AS priority_text,     -- high (TEXT)
    Metadata->'shipping'->>'carrier' AS carrier, -- UPS (TEXT)
    Metadata->'shipping'->'address'->>'city' AS city,
    Metadata->'shipping'->'address'->>'state' AS state
FROM dbo.Orders;

-- ============================================================
-- JSONB Operators: @> (containment)
-- ============================================================
-- Find orders where metadata contains specified structure
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE Metadata @> '{"priority": "high"}'::JSONB;

SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE Metadata @> '{"shipping": {"carrier": "UPS"}}'::JSONB;

SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE Metadata @> '{"shipping": {"address": {"state": "WA"}}}'::JSONB;

-- Array containment: find orders with a specific item
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE Metadata @> '{"items": [{"sku": "A100"}]}'::JSONB;

-- Multiple conditions in one @> expression
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE Metadata @> '{"priority": "high", "source": "web"}'::JSONB;

-- ============================================================
-- JSONB Operators: ? (key exists), ?| (any key), ?& (all keys)
-- ============================================================
-- ?: does top-level key exist?
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE Metadata ? 'flags';               -- Orders with "flags" key

-- ?|: does any of these keys exist?
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE Metadata ?| ARRAY['flags', 'notes'];  -- Orders with "flags" OR "notes"

-- ?&: do all these keys exist?
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE Metadata ?& ARRAY['source', 'priority', 'shipping'];  -- All three keys exist

-- ============================================================
-- JSONB Operators: || (concatenation/merge)
-- ============================================================
-- Merge new fields into existing JSONB
UPDATE dbo.Orders
SET Metadata = Metadata || '{"updated": true, "reviewed_by": "system"}'::JSONB
WHERE OrderId = 1;

-- Override existing field
UPDATE dbo.Orders
SET Metadata = Metadata || '{"priority": "critical"}'::JSONB
WHERE OrderId = 1;

-- ============================================================
-- JSONB Array Access
-- ============================================================
-- Access array elements by index (0-based)
SELECT
    OrderId,
    Metadata->'items'->0 AS first_item_jsonb,           -- First item as JSONB
    Metadata->'items'->0->>'sku' AS first_item_sku,      -- SKU of first item
    Metadata->'items'->>-1 AS last_item_jsonb,           -- Last item (negative index)
    Metadata->'items'->>2 AS nonexistent_item              -- NULL if index out of bounds
FROM dbo.Orders;

-- Unnest JSONB array to rows
SELECT
    o.OrderId,
    jsonb_array_elements(o.Metadata->'items')->>'sku' AS sku,
    (jsonb_array_elements(o.Metadata->'items')->>'qty')::INT AS qty
FROM dbo.Orders o
WHERE o.OrderId = 1;

-- ============================================================
-- JSONB Aggregation
-- ============================================================
-- Aggregate rows to JSONB
SELECT
    CustomerId,
    JSONB_AGG(OrderCode ORDER BY OrderDate) AS order_codes,
    JSONB_OBJECT_AGG(OrderCode::TEXT, TotalAmount) AS order_amounts
FROM dbo.Orders
GROUP BY CustomerId;

-- ============================================================
-- JSONB Indexes
-- ============================================================
-- GIN index on the entire JSONB column
CREATE INDEX IX_Orders_Metadata_GIN ON dbo.Orders USING GIN (Metadata);

-- GIN index with jsonb_path_ops (smaller, faster for @>)
CREATE INDEX IX_Orders_Metadata_GIN_Path ON dbo.Orders USING GIN (Metadata jsonb_path_ops);

-- Expression B-tree index on a specific field
CREATE INDEX IX_Orders_Source ON dbo.Orders (((Metadata->>'source')));
CREATE INDEX IX_Orders_Priority ON dbo.Orders (((Metadata->>'priority')));

-- Expression index with partial index for sparse data
CREATE INDEX IX_Orders_Flags ON dbo.Orders USING GIN (Metadata)
    WHERE Metadata ? 'flags';

-- ============================================================
-- Using indexes effectively
-- ============================================================
-- GIN index seek for @> (containment)
EXPLAIN ANALYZE
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE Metadata @> '{"priority": "high"}'::JSONB;
-- Bitmap Index Scan on ix_orders_metadata_gin_path
--   Index Cond: (metadata @> '{"priority": "high"}'::jsonb)
-- → Bitmap Heap Scan on orders

-- Expression B-tree seek for ->>
EXPLAIN ANALYZE
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE Metadata->>'source' = 'web';
-- Index Scan using ix_orders_source on orders
--   Index Cond: ((metadata ->> 'source'::text) = 'web'::text)

-- Combined index usage
EXPLAIN ANALYZE
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE Metadata @> '{"priority": "high"}'::JSONB
  AND Metadata->>'source' = 'web';
```

```csharp
// EF Core with Npgsql — JSONB has first-class support via owned entities
public class ApplicationDbContext : DbContext
{
    public DbSet<Order> Orders => Set<Order>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Order>(entity =>
        {
            entity.ToTable("Orders");
            entity.HasKey(o => o.OrderId);
            entity.Property(o => o.OrderId).ValueGeneratedOnAdd();
            entity.Property(o => o.OrderCode).HasMaxLength(20).IsRequired();
            entity.Property(o => o.TotalAmount).HasColumnType("numeric(18,2)");

            // JSONB owned entity mapping
            entity.OwnsOne(o => o.Metadata, metadata =>
            {
                metadata.ToJson();  // Maps as JSONB column

                metadata.Property(m => m.Source).HasJsonPropertyName("source");
                metadata.Property(m => m.Priority).HasJsonPropertyName("priority");

                metadata.OwnsOne(m => m.Shipping, shipping =>
                {
                    shipping.Property(s => s.Carrier).HasJsonPropertyName("carrier");
                    shipping.Property(s => s.Method).HasJsonPropertyName("method");
                    shipping.Property(s => s.Cost).HasJsonPropertyName("cost");

                    shipping.OwnsOne(s => s.Address, address =>
                    {
                        address.Property(a => a.City).HasJsonPropertyName("city");
                        address.Property(a => a.State).HasJsonPropertyName("state");
                        address.Property(a => a.Zip).HasJsonPropertyName("zip");
                    });
                });

                metadata.OwnsMany(m => m.Items, items =>
                {
                    items.Property(i => i.Sku).HasJsonPropertyName("sku");
                    items.Property(i => i.Qty).HasJsonPropertyName("qty");
                    items.Property(i => i.Price).HasJsonPropertyName("price");
                });
            });
        });
    }
}

public sealed class OrderService
{
    private readonly ApplicationDbContext _dbContext;

    public OrderService(ApplicationDbContext dbContext)
        => _dbContext = dbContext;

    // JSONB queries via owned entity navigation — Npgsql translates to ->
    public async Task<List<Order>> GetHighPriorityOrdersAsync(
        CancellationToken cancellationToken = default)
    {
        return await _dbContext.Orders
            .Where(o => o.Metadata.Priority == "high")
            .OrderByDescending(o => o.OrderDate)
            .ToListAsync(cancellationToken);
        // Generated SQL:
        // SELECT o.* FROM "Orders" AS o
        // WHERE o."Metadata"->>'priority' = 'high'
        // ORDER BY o."OrderDate" DESC
    }

    // Nested JSONB access
    public async Task<List<Order>> GetOrdersByStateAsync(
        string state,
        CancellationToken cancellationToken = default)
    {
        return await _dbContext.Orders
            .Where(o => o.Metadata.Shipping != null
                        && o.Metadata.Shipping.Address != null
                        && o.Metadata.Shipping.Address.State == state)
            .ToListAsync(cancellationToken);
        // Generated SQL:
        // SELECT o.* FROM "Orders" AS o
        // WHERE o."Metadata" #>> '{shipping,address,state}' = @state
    }

    // JSONB array containment
    public async Task<List<Order>> GetOrdersWithSkuAsync(
        string sku,
        CancellationToken cancellationToken = default)
    {
        return await _dbContext.Orders
            .Where(o => o.Metadata.Items.Any(i => i.Sku == sku))
            .ToListAsync(cancellationToken);
        // Generated SQL:
        // SELECT o.* FROM "Orders" AS o
        // WHERE o."Metadata" @> ('[{"sku": "' || @sku || '"}]')::jsonb
        // Note: EF Core generates a @> containment check
    }

    // Raw JSONB via FromSql for complex operators
    public async Task<List<Order>> GetOrdersWithFlagsAsync(
        CancellationToken cancellationToken = default)
    {
        return await _dbContext.Orders
            .FromSqlRaw(@"
                SELECT * FROM ""Orders""
                WHERE ""Metadata"" ? 'flags'")
            .ToListAsync(cancellationToken);
    }
}
```

```csharp
// Dapper — full control over JSONB operators
public sealed class OrderDapperRepository
{
    private readonly IDbConnectionFactory _connectionFactory;

    public OrderDapperRepository(IDbConnectionFactory connectionFactory)
        => _connectionFactory = connectionFactory;

    // JSONB @> containment
    public async Task<IReadOnlyList<Order>> GetHighPriorityOrdersAsync(
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            SELECT *
            FROM ""Orders""
            WHERE ""Metadata"" @> '{"priority": "high"}'::jsonb
            ORDER BY ""OrderDate"" DESC";

        await using var connection = _connectionFactory.Create();

        var results = await connection.QueryAsync<Order>(
            new CommandDefinition(sql,
                cancellationToken: cancellationToken));

        return results.AsList();
    }

    // JSONB ? (key existence) with parameterised value
    public async Task<IReadOnlyList<Order>> GetOrdersWithKeyAsync(
        string key,
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            SELECT *
            FROM ""Orders""
            WHERE ""Metadata"" ? @Key";

        await using var connection = _connectionFactory.Create();

        var results = await connection.QueryAsync<Order>(
            new CommandDefinition(sql,
                new { Key = key },
                cancellationToken: cancellationToken));

        return results.AsList();
    }

    // JSONB array elements with jsonb_array_elements
    public async Task<IReadOnlyList<OrderItemFlat>> GetOrderItemsFlatAsync(
        int customerId,
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            SELECT
                o.""OrderId"",
                o.""OrderCode"",
                item->>'sku' AS SKU,
                (item->>'qty')::INT AS Qty,
                (item->>'price')::NUMERIC(18,2) AS Price
            FROM ""Orders"" o
            CROSS JOIN LATERAL jsonb_array_elements(o.""Metadata""->'items') AS item
            WHERE o.""CustomerId"" = @CustomerId
            ORDER BY o.""OrderId"", SKU";

        await using var connection = _connectionFactory.Create();

        var results = await connection.QueryAsync<OrderItemFlat>(
            new CommandDefinition(sql,
                new { CustomerId = customerId },
                cancellationToken: cancellationToken));

        return results.AsList();
    }

    // JSONB aggregation
    public async Task<List<CustomerOrderSummary>> GetCustomerSummariesAsync(
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            SELECT
                o.""CustomerId"",
                JSONB_AGG(o.""OrderCode"" ORDER BY o.""OrderDate"") AS OrderCodes,
                JSONB_OBJECT_AGG(o.""OrderCode""::TEXT, o.""TotalAmount"") AS OrderAmounts,
                COUNT(*) AS OrderCount,
                SUM(o.""TotalAmount"") AS TotalSpent
            FROM ""Orders"" o
            GROUP BY o.""CustomerId""
            ORDER BY o.""CustomerId""";

        await using var connection = _connectionFactory.Create();

        return (await connection.QueryAsync<CustomerOrderSummary>(
            new CommandDefinition(sql,
                cancellationToken: cancellationToken))).AsList();
    }

    public async Task UpdateMetadataAsync(
        int orderId,
        string key,
        string value,
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            UPDATE ""Orders""
            SET ""Metadata"" = ""Metadata"" || JSONB_BUILD_OBJECT(@Key, @Value::JSONB)

        await connection.ExecuteAsync(
            new CommandDefinition(sql,
                new { OrderId = orderId, Key = key, Value = value },
                cancellationToken: cancellationToken));
    }
```

### Execution Plan Analysis

**For @> containment with GIN index (jsonb_path_ops):**

```
[Bitmap Index Scan (IX_Orders_Metadata_GIN_Path)]
  Index Cond: (Metadata @> '{"priority": "high"}'::jsonb)
  GIN scan finds matching tokens in entry tree
  Returns TID list (posting list)
→ [Bitmap Heap Scan (Orders)]
  Reads heap pages by TID from bitmap
  Rechecks visibility (MVCC) and applies any remaining filters
Estimated Cost: ~M pages (proportional to matches)
Actual: seeks on GIN entry tree (O(log tokens)) + heap scan
```

**For ->> with expression B-tree index:**

```
[Index Scan (IX_Orders_Source)]
  Index Cond: ((Metadata ->> 'source') = 'web'::text)
  Standard B-tree seek on extracted value
→ [Heap Fetch (Orders)]
  Fetch full row by TID from index leaf
Estimated Cost: O(log n) B-tree seek + heap fetch per row
```

**For ? key existence with GIN index:**

```
[Bitmap Index Scan (IX_Orders_Metadata_GIN)]
  Index Cond: (Metadata ? 'flags')
  Looks up 'flags' key token in GIN entry tree
→ [Bitmap Heap Scan (Orders)]
```

### SQL Server vs PostgreSQL Differences

| | SQL Server JSON | PostgreSQL JSONB |
|---|---|---|
| Storage type | NVARCHAR(MAX) | Binary decomposed tree |
| Indexing | Computed column + standard index | GIN (full JSONB) or expression B-tree |
| Containment operator | None — use JSON_VALUE in WHERE | `@>` — structural containment |
| Key existence | None — use JSON_VALUE(..., 'lax') | `?` — direct key existence |
| Concatenation | No native operator | `||` — merge two JSONB values |
| Array elements | OPENJSON | jsonb_array_elements |
| Nested path | JSON_VALUE(data, '$.a.b.c') | `data #>> '{a,b,c}'` |
| .NET support | Raw SQL only | Npgsql owned entities with query translation |
| Index on path | Computed column with PERSISTED | Expression B-tree index |
| Write performance | Fast (NVARCHAR text update) | Medium (binary decomposition + GIN tokens) |

## Gotchas and Production Pitfalls

### 1. GIN Index Pending List Insert Stalls

**Pitfall:** The GIN index uses a pending list for fast inserts — an in-memory buffer that accumulates recently inserted tokens. When the pending list exceeds `gin_pending_list_limit` (default 4MB or 10% of shared_buffers), a background worker merges it into the main index. During the merge, INSERT latency spikes.

```sql
-- ❌ Default pending list limit may cause spikes under high insert rates
SHOW gin_pending_list_limit;
-- 4MB (default)

-- ✅ Increase pending list limit for write-heavy workloads
ALTER SYSTEM SET gin_pending_list_limit = 65536;  -- 64MB
-- Or per-session:
SET gin_pending_list_limit = 65536;
```

**Symptom:** INSERT latency spikes from 5ms to 200ms every few minutes during bulk load. The spike corresponds to pending list merge.

**Cost of not fixing:** Unpredictable INSERT latency under load, application timeouts.

### 2. jsonb_path_ops Does Not Support ? Operator

**Pitfall:** The `jsonb_path_ops` operator class for GIN indexes is faster (smaller index, faster `@>` queries) but does not support the `?`, `?|`, and `?&` key existence operators.

```sql
-- ❌ jsonb_path_ops index — query with ? fails
CREATE INDEX IX_Orders_Metadata_GIN_Path ON dbo.Orders USING GIN (Metadata jsonb_path_ops);

SELECT * FROM dbo.Orders WHERE Metadata ? 'flags';
-- ERROR: operator does not exist: jsonb ? unknown
-- HINT: No operator matches the given name and argument type.

-- ✅ Use default GIN operator class for ? queries
CREATE INDEX IX_Orders_Metadata_GIN ON dbo.Orders USING GIN (Metadata);

-- ✅ Or create both: jsonb_path_ops for @>, default GIN for ?
CREATE INDEX IX_Orders_Metadata_GIN_Path ON dbo.Orders USING GIN (Metadata jsonb_path_ops);
CREATE INDEX IX_Orders_Metadata_GIN ON dbo.Orders USING GIN (Metadata);
```

**Symptom:** Query fails with operator does not exist error.

**Cost of not fixing:** Production query failure, requiring index schema change.

### 3. Expression B-tree Index Not Used Due to COALESCE or Wrapper

**Pitfall:** An expression index on `((Metadata->>'field'))` is only used when the query has the exact same expression. Wrapping it in COALESCE, CAST (to a different type), or function prevents index usage.

```sql
-- Index on extracted text
CREATE INDEX ON dbo.Orders (((Metadata->>'priority')));

-- ❌ Wrapped in COALESCE — no index usage
SELECT * FROM dbo.Orders
WHERE COALESCE(Metadata->>'priority', 'normal') = 'high';
-- Seq Scan — COALESCE is a different expression

-- ✅ Exact match — index seek
SELECT * FROM dbo.Orders
WHERE Metadata->>'priority' = 'high';

-- ✅ Expression index on COALESCE if needed
CREATE INDEX ON dbo.Orders ((COALESCE(Metadata->>'priority', 'normal')));

-- Query matches: uses the COALESCE index
SELECT * FROM dbo.Orders
WHERE COALESCE(Metadata->>'priority', 'normal') = 'high';
```

**Symptom:** Sequential scan despite matching index existing. EXPLAIN shows no index usage.

**Cost of not fixing:** Full table scans on 10M+ row tables, queries taking seconds instead of milliseconds.

### 4. JSONB Duplicate Key Handling (Last Value Wins)

**Pitfall:** JSONB removes duplicate keys from JSON objects, keeping only the last value. This is different from SQL Server's JSON, which preserves duplicate keys.

```sql
-- ❌ JSONB silently drops duplicate keys
SELECT '{"a": 1, "a": 2}'::JSONB;
-- {"a": 2}  -- Only the last value is kept!

-- In SQL Server:
SELECT JSON_QUERY('{"a": 1, "a": 2}');
-- {"a": 1, "a": 2}  -- Both keys preserved
```

**Symptom:** Data loss when importing JSON with duplicate keys. The first value is silently discarded.

**Cost of not fixing:** Silent data corruption — the first value is lost without warning.

### 5. GIN Index Size on High-Cardinality JSONB

**Pitfall:** Each unique key-value pair becomes an entry in the GIN index. If the JSONB documents have many unique keys (e.g., user-defined fields), the GIN index can grow larger than the table itself.

```sql
-- Check index size ratio
SELECT
    pg_size_pretty(pg_relation_size('Orders')) AS table_size,
    pg_size_pretty(pg_relation_size('IX_Orders_Metadata_GIN')) AS gin_index_size,
    pg_size_pretty(pg_total_relation_size('Orders')) AS total_size;
```

**Symptom:** GIN index 3-5x larger than the table on high-cardinality JSONB. Index maintenance (autovacuum) becomes expensive.

**Cost of not fixing:** Autovacuum pressure, storage costs, slower vacuum operations.

### 6. Type Mismatch in JSONB Queries

**Pitfall:** JSONB preserves the type of values. The `@>` operator does type-sensitive comparison. A string "123" does not match a numeric 123.

```sql
-- ❌ Type mismatch: string vs numeric
SELECT '{"value": "123"}'::JSONB @> '{"value": 123}'::JSONB;
-- false — "123" (string) != 123 (number)

-- ✅ Correct type
SELECT '{"value": 123}'::JSONB @> '{"value": 123}'::JSONB;
-- true

SELECT '{"value": "123"}'::JSONB @> '{"value": "123"}'::JSONB;
-- true
```

**Symptom:** Containment queries return fewer results than expected because of type mismatches between stored data and query predicates.

**Cost of not fixing:** Invisible data filtering — queries that should match rows do not.

### Spoken Answers

**Q: What is JSONB and how does it differ from the JSON data type in PostgreSQL?**

> **Average answer:** "JSONB stores JSON in binary format, while JSON stores it as text. JSONB is faster for querying because it's indexed."

> **Great answer:** "JSONB stores JSON data in a decomposed binary format — the input text is parsed and converted to an internal tree of key-value pairs where keys are sorted alphabetically and values are typed (string, number, boolean, null, array, object). This decomposition happens at insert time, so INSERT is slightly slower than JSON, but subsequent queries are much faster because: (1) key access uses binary search on the sorted key array — O(log k) per access, (2) the GIN index decomposes the JSONB further into tokens that enable inverted index lookups for containment and existence queries, (3) duplicate keys are removed at parse time (last value wins). The plain JSON type stores the original text verbatim and requires re-parsing on every access. JSONB is always preferred for production use. The tradeoff: JSONB takes slightly more storage (~5-10%) than the original text due to the binary overhead of key sorting and type encoding, but is vastly more queryable."

**Q: How does JSONB in PostgreSQL compare to JSON in SQL Server?**

> **Average answer:** "PostgreSQL JSONB has more operators and better indexing. SQL Server JSON is not a real data type — it's just NVARCHAR."

> **Great answer:** "JSONB vs SQL Server JSON is a fundamental architectural difference. PostgreSQL JSONB is a native binary data type with its own storage format, operator class, and dedicated GIN index support. It supports structural containment (`@>`), key existence (`?`, `?|`, `?&`), concatenation (`||`), and has path extraction operators (`->`, `->>`, `#>`, `#>>`). The GIN index indexes every key and value in the document at every level, enabling path-agnostic queries. SQL Server JSON is stored as NVARCHAR(MAX) — it is parsed at query time by JSON_VALUE/JSON_QUERY functions. There is no native JSON type. Indexing requires creating a computed column with a specific JSON_VALUE expression and putting a standard index on it. SQL Server has no containment operator, no key existence operator, and no GIN index equivalent. The only advantage of SQL Server JSON is that NVARCHAR storage is very efficient for write-heavy workloads — there is no binary decomposition cost on INSERT. For query-heavy workloads, PostgreSQL JSONB is significantly more capable. For .NET developers: Npgsql provides owned entity mapping for JSONB with LINQ query translation, while EF Core for SQL Server requires raw SQL for all JSON queries."

**Q: How does EF Core Npgsql translate LINQ queries on JSONB owned entities?**

> **Average answer:** "EF Core maps owned entities to JSONB columns using .ToJson(). LINQ queries on the owned properties are translated to JSONB operators."

> **Great answer:** "Npgsql's EF Core provider has first-class JSONB support via owned entity mapping using `entity.OwnsOne(o => o.Metadata, m => m.ToJson())`. This tells EF Core to store the owned entity as a JSONB column instead of creating a separate table. The provider translates LINQ queries on owned entity properties to the appropriate JSONB operators: `o.Metadata.Priority == "high"` becomes `o."Metadata"->>'priority' = 'high'` (using `->>` for text comparison). Nested navigation like `o.Metadata.Shipping.Address.State` generates the `#>>` path operator: `o."Metadata" #>> '{shipping,address,state}'`. Array containment with `Any()` generates the `@>` operator: `o."Metadata" @> '[{"sku": "A100"}]'::jsonb`. Updates to owned entities are sent as full JSONB column updates (not partial JSONB_SET), which means EF Core replaces the entire Metadata column on any property change. For partial updates, you need raw SQL with JSONB_SET. The generated SQL uses parameterised values, so expression indexes on specific paths are used when the query matches the expression exactly."

## Self-Check

### Conceptual Questions

1. What is the difference between JSON and JSONB in PostgreSQL?
2. How does the @> (containment) operator work with a GIN index?
3. What is the difference between jsonb_path_ops and the default GIN operator class?
4. How does a JSONB expression B-tree index differ from a GIN index on JSONB?
5. Can EF Core Npgsql translate LINQ queries on JSONB owned entities to JSONB operators?
6. How would you query for a specific key's existence in JSONB using Dapper?
7. What is the difference between PostgreSQL JSONB and SQL Server JSON for indexing?
8. At what write rate does the GIN pending list merge become a concern?
9. What index type supports a WHERE clause with `Metadata @> '{"priority":"high"}'`?
10. Explain how EF Core Npgsql translates `o.Metadata.Shipping.Address.State == "WA"` to SQL.

<details>
<summary>Answers</summary>

1. **JSON vs JSONB:** JSON stores the text verbatim (re-parsed on each access). JSONB stores a decomposed binary tree (parsed once on insert, keys sorted, duplicates removed, values typed). JSONB supports indexing (GIN, expression B-tree); JSON does not.
2. **@> with GIN:** The @> (contains) operator checks if the left JSONB structurally contains the right JSONB. The GIN index decomposes the left operand into tokens. For the right operand, PostgreSQL generates candidate tokens, looks them up in the GIN entry tree (B-tree of unique tokens), and intersects the posting lists (TID lists) to find rows containing all tokens.
3. **jsonb_path_ops vs default:** The default GIN operator class indexes `(key, value)` pairs and supports @>, ?, ?|, ?&. The jsonb_path_ops class indexes hashed path-value pairs, is smaller and faster for @>, but does NOT support the ?, ?|, ?& key existence operators.
4. **Expression B-tree vs GIN:** Expression B-tree indexes a specific extracted value: `((data->>'field'))` — standard B-tree with equality/range/order support. GIN indexes the entire JSONB document decomposed into tokens — path-agnostic but limited to @>, ?, ?|, ?& operators.
5. **EF Core Npgsql:** Yes. EF Core Npgsql translates owned entity navigation on JSONB columns to ->, ->>, #>>, and @> operators. For example, `o.Metadata.Priority == "high"` translates to `o."Metadata"->>'priority' = 'high'`.
6. **Dapper key existence:** `SELECT * FROM Orders WHERE Metadata ? @Key` with `new { Key = "flags" }`.
7. **JSONB vs SQL Server JSON:** PostgreSQL JSONB is a native binary type with GIN indexing (path-agnostic), @> containment, ? key existence, || concatenation. SQL Server JSON is NVARCHAR(MAX) with no native type, no containment operator, no key existence operator, and indexing via computed columns only.
8. **GIN pending list concern:** Above ~1000 inserts/second, the default 4MB pending list fills frequently (every few minutes), triggering merges that cause INSERT latency spikes. Increase `gin_pending_list_limit` or batch inserts.
9. **Index for @>:** GIN index with either default operator class (supports @> and ?) or jsonb_path_ops (faster @>, no ?).
10. **EF Core translation:** `o."Metadata" #>> '{shipping,address,state}' = 'WA'::text`. Npgsql uses the #>> path operator for deeply nested JSONB access.

</details>

**Challenge 1 — Write the SQL**

Your `Orders` table has a JSONB `Metadata` column. Write a query that finds all orders where:
- The source is "web"
- The shipping carrier is "UPS" or "FedEx"
- The shipping method is "Express"
- There is at least one item with qty > 3

Return OrderId, OrderCode, and the shipping address city.

<details>
<summary>Solution</summary>

```sql
SELECT
    o.OrderId,
    o.OrderCode,
    o.Metadata #>> '{shipping,address,city}' AS ShipCity
FROM dbo.Orders o
WHERE o.Metadata @> '{"source": "web"}'::JSONB
  AND o.Metadata @> '{"shipping": {"method": "Express"}}'::JSONB
  AND o.Metadata #>> '{shipping,carrier}' IN ('UPS', 'FedEx')
  AND o.Metadata @> '{"items": [{"qty": 4}]}'::JSONB
  -- Alternative: check any item with qty > 3 using jsonb_array_elements
  AND EXISTS (
      SELECT 1
      FROM jsonb_array_elements(o.Metadata->'items') AS item
      WHERE (item->>'qty')::INT > 3
  )
ORDER BY o.OrderDate DESC;
```

**Index strategy:**
```sql
CREATE INDEX IX_Orders_Metadata_GIN ON dbo.Orders USING GIN (Metadata jsonb_path_ops);
CREATE INDEX IX_Orders_Source ON dbo.Orders (((Metadata->>'source')));
CREATE INDEX IX_Orders_Carrier ON dbo.Orders (((Metadata #>> '{shipping,carrier}')));
```

**Challenge 3 — Explain the execution plan**

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE Metadata ? 'flags'
  AND Metadata @> '{"flags": ["fragile"]}'::JSONB;
```

The execution plan shows:
```
Bitmap Heap Scan on orders (actual rows=5000)
  Recheck Cond: ((metadata ? 'flags'::text) AND (metadata @> '{"flags": ["fragile"]}'::jsonb))
  → Bitmap Index Scan on ix_orders_metadata_gin (actual rows=5000)
      Index Cond: ((metadata ? 'flags'::text) AND (metadata @> '{"flags": ["fragile"]}'::jsonb))
```

Why is there only one Bitmap Index Scan for both conditions? What would change if the GIN index used jsonb_path_ops instead?

<details>
<summary>Solution</summary>

**Why one scan:** The GIN index (default operator class) can handle both `?` and `@>` in a single index scan because the GIN entry tree contains tokens for both key existence and path-value pairs. The index condition evaluates both operators by looking up the token for key 'flags' (for the ? condition) and the token for flags array containing "fragile" (for the @> condition). The posting lists are intersected during the bitmap scan.

**With jsonb_path_ops:** The `?` operator would NOT use the index — jsonb_path_ops does not support key existence. The query would fail with "operator does not exist" error, or PostgreSQL would fall back to a sequential scan with a filter. The @> condition would still use the jsonb_path_ops index, but the ? condition would require separate evaluation.

**Recommendation:** If both `?` and `@>` are needed, use the default GIN operator class. If only `@>` is needed, jsonb_path_ops is faster and smaller.

</details>

**Challenge 4 — Diagnose the concurrency problem**

A production system inserts 2000 orders/second into an Orders table with a JSONB Metadata column. The Metadata column has a GIN index (default operator class). Every few minutes, INSERT latency spikes from 3ms to 500ms for 2-3 seconds. Application queries against the Orders table experience timeouts during these spikes. PostgreSQL logs show "GIN pending list" messages.

<details>
<summary>Solution</summary>

**Root cause:** The GIN index's pending list (in-memory buffer for fast inserts) fills to the default 4MB limit every ~30 seconds under 2000 inserts/second. A background worker merges the pending list into the main GIN index. During the merge, the GIN index is partially locked, causing INSERT latency spikes. The merge also generates I/O pressure on the GIN index pages.

**Detection query:**
```sql
-- Check pending list size
SELECT * FROM pg_stat_user_tables WHERE relname = 'orders';

-- Check GIN index pending merges
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch
FROM pg_stat_user_indexes
WHERE indexname LIKE '%metadata%gin%';
```

**Fix:**

```sql
-- Option 1: Increase pending list limit to reduce merge frequency
ALTER SYSTEM SET gin_pending_list_limit = 262144;  -- 256MB

-- Option 2: Use jsonb_path_ops (smaller index, less pending list contention)
-- Only if ? operator is not needed
DROP INDEX IX_Orders_Metadata_GIN;
CREATE INDEX IX_Orders_Metadata_GIN ON dbo.Orders USING GIN (Metadata jsonb_path_ops);

-- Option 3: Batch inserts to reduce GIN index churn
-- Instead of 2000 individual INSERTs, batch into 100-row INSERTs
INSERT INTO dbo.Orders (CustomerId, OrderCode, OrderDate, TotalAmount, Metadata)
SELECT * FROM UNNEST(@CustomerIds, @OrderCodes, @OrderDates, @Amounts, @Metadatas);

-- Option 4: Remove GIN index and use expression indexes on specific paths only
-- if most queries filter by specific known fields
CREATE INDEX IX_Orders_Priority ON dbo.Orders (((Metadata->>'priority')));
CREATE INDEX IX_Orders_Source ON dbo.Orders (((Metadata->>'source')));
```

**In .NET:**

**Challenge 5 — Design the JSONB indexing strategy**

**Scenario:** An e-commerce platform stores product metadata as JSONB in a 20M-row `Products` table. The JSONB has the structure:
```json
{
    "brand": "Acme",
    "category": "Electronics",
    "tags": ["wireless", "bluetooth"],
    "specs": {
        "weight": "0.5kg",
        "color": "black",
        "dimensions": {"width": 10, "height": 5}
    },
    "reviews": [
        {"user": "alice", "rating": 5, "text": "Great!"},
        {"user": "bob", "rating": 3, "text": "Okay"}
    ],
    "attributes": {
        "connectivity": "bluetooth 5.0",
        "battery": "20h"
    }
}
```

Query patterns:
- 500K queries/day: filter by brand and category: `Metadata @> '{"brand": "X", "category": "Y"}'`
- 200K queries/day: filter by specs attributes: `Metadata @> '{"specs": {"color": "black"}}'`
- 100K queries/day: range query on price (a regular column, not JSONB)
- 10K queries/day: check existence of tags: `Metadata ? 'tags'`
- 50K inserts/day (product updates), each updating Metadata

Design the optimal indexing strategy. Show CREATE INDEX statements.

<details>
<summary>Solution</summary>

```sql
-- 1. GIN index with jsonb_path_ops for the 700K/day @> queries
-- jsonb_path_ops is optimal because @> is the dominant pattern
-- It does NOT support ? but that's only 10K queries/day (low frequency)
CREATE INDEX IX_Products_Metadata_GIN
    ON dbo.Products USING GIN (Metadata jsonb_path_ops);

-- 2. For the 10K/day ? 'tags' queries that jsonb_path_ops cannot serve,
-- they can use a sequential scan (10K/day on 20M rows, ~10ms per scan
-- with a partial index on rows that have tags, or just accept seq scan)
-- OR create a second GIN index with default operator class:
CREATE INDEX IX_Products_Metadata_GIN_Default
    ON dbo.Products USING GIN (Metadata)
    WHERE Metadata ? 'tags';  -- Partial index, only covers rows with tags

-- 3. Expression indexes for commonly-queried paths (brand and category)
-- to support equality queries with B-tree efficiency
CREATE INDEX IX_Products_Brand ON dbo.Products (((Metadata->>'brand')));
CREATE INDEX IX_Products_Category ON dbo.Products (((Metadata->>'category')));

-- 4. Price column index (regular column, not JSONB) for 100K range queries/day
CREATE INDEX IX_Products_Price ON dbo.Products (Price);

-- 5. Composite index for the most common query pattern (brand + category + price range)
CREATE INDEX IX_Products_Brand_Category_Price
    ON dbo.Products (((Metadata->>'brand'), (Metadata->>'category'), Price));
```

**Why this strategy:**
- jsonb_path_ops for the bulk of @> queries — smaller index, faster containment checks
- Partial default GIN for the ? queries — only indexes rows that actually have tags (sparse index)
- Expression B-tree indexes for brand and category — used when queries filter on these exact paths with equality
- The composite index covers the most common pattern: brand + category + price range

**Tradeoffs:**
- Two GIN indexes: write overhead doubles on UPDATE (both indexes must maintain entries)
- Expression indexes: exact expression matching required in queries
- 50K inserts/day is moderate — GIN pending list should be tuned to ~64MB

**What NOT to index:**
- Do NOT create expression indexes on reviews or specs sub-fields unless those specific paths are queried with equality/range — the GIN index already covers @> containment for any path
- Do NOT create a GIN index on the entire table with default operator class (would be 3x larger than jsonb_path_ops)

</details>
</details>
