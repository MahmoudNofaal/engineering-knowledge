### Where This Fits

JSON_MODIFY is the server-side function for updating JSON data stored in SQL Server — it lets you modify a JSON document in place within an UPDATE statement without pulling the entire document to the application tier, deserializing, modifying, and writing it back. A .NET backend engineer encounters this when storing flexible-schema data like customer attributes, workflow state, integration payloads, or audit metadata in NVARCHAR(MAX) columns with JSON content. Production systems updating individual JSON fields without JSON_MODIFY read the full column content (potentially 100+ KB), modify it in application memory, and write it back — a read-modify-write round trip that creates UPDATE concurrency issues (lost updates, higher lock duration) and wastes bandwidth. JSON_MODIFY performs the modification as a string manipulation inside SQL Server with a single UPDATE statement, reducing network round trips from 2 to 1 and eliminating the lost-update window between read and write within the same transaction. The interview signal here is whether a candidate understands that JSON_MODIFY is a string function — it reconstructs the entire NVARCHAR(MAX) column on every call, it is NOT an in-place JSON patch, and executing 100 JSON_MODIFY calls in sequence on the same document in a single statement creates 100 full document rewrites.

## Core Mental Model

JSON_MODIFY is SQL Server's string-manipulation function for JSON NVARCHAR(MAX) columns — it takes an existing JSON string, a JSON path expression, and a new value, and returns a new JSON string with the specified field updated, inserted, or deleted. The critical mental model: JSON_MODIFY is NOT a document-store atomic update operation. It performs a full string rebuild of the JSON document on every invocation. When you call `JSON_MODIFY(@json, '$.field', 'value')`, SQL Server parses the input JSON string, locates the path, replaces the value at that path (or inserts or deletes it), and serializes the entire document back as a new NVARCHAR(MAX) string. The original document is not mutated — a new string is created. This means: (1) the cost of JSON_MODIFY scales with the document size, not just the field being modified, (2) multiple JSON_MODIFY calls on the same document in a single SET clause cascade — each call creates a new full document, (3) the function is deterministic for a given input but the path resolution uses lax mode by default (no error for missing paths — it inserts instead), and (4) deleting a property requires either strict mode (which errors if the path is missing) or explicitly setting the value to NULL in lax mode with specific syntax. The `append` keyword enables array element insertion. JSON_MODIFY always returns NVARCHAR(MAX), even if the input is smaller, which has implications for storage and indexing.

### Classification

- **Function family:** JSON modification function in T-SQL — part of the built-in JSON function set (JSON_VALUE, JSON_QUERY, JSON_MODIFY, ISJSON, OPENJSON).
- **Engine behavior:** Runs as a scalar function during query execution. The function is evaluated per row and produces a new string value. Inside an UPDATE SET clause, the full document is read from the column, modified by JSON_MODIFY, and written back as the new column value.
- **SARGability:** JSON_MODIFY is not used in WHERE clauses directly, so SARGability does not apply. However, if JSON_MODIFY is used in a computed column or a trigger, it affects performance of those operations.
- **Atomicity:** JSON_MODIFY is not atomic at the JSON field level — the row-level UPDATE is atomic (transactional), but the JSON_MODIFY function itself is a string builder, not an in-place patcher.

## Deep Mechanics

### How the Engine Executes This

1. **Parsing and binding:** When SQL Server encounters JSON_MODIFY in a T-SQL batch or within an UPDATE SET clause, the parser validates the function signature — three required arguments (expression, path, newValue) plus an optional mode keyword ('lax' or 'strict' embedded in the path). The algebrizer resolves the expression argument to determine whether it is a column reference (runtime binding per row) or a constant or variable (compile-time binding).

2. **Path resolution mode:** The path argument is parsed for the mode prefix. `'$.field'` defaults to lax mode. `'strict$.field'` forces strict mode. In strict mode, JSON_MODIFY returns NULL if the path does not exist. In lax mode, if the path does not exist, JSON_MODIFY attempts to insert the new value at the path — creating intermediate objects if the path implies nested structure (e.g., `$.metadata.attributes.color` creates `metadata` and `attributes` if they do not exist).

3. **Value type detection from the third argument:** JSON_MODIFY examines the third argument to determine whether to treat it as a scalar or a JSON fragment. If the value is a valid JSON fragment (object or array) returned by JSON_QUERY or constructed as a string with proper JSON syntax, JSON_MODIFY inserts it as structured JSON. If the value is a scalar (string, number, boolean, NULL), it inserts it as a scalar JSON value — this means the value is escaped and quoted as a JSON string. The critical distinction: `JSON_MODIFY(@j, '$.key', '{"nested":1}')` inserts the literal string `'{"nested":1}'` as a JSON string (with escaped quotes), while `JSON_MODIFY(@j, '$.key', JSON_QUERY('{"nested":1}'))` inserts the JSON object as a nested JSON fragment.

4. **String construction:** Internally, SQL Server parses the JSON input into a lightweight Document Object Model (DOM), navigates to the path node, applies the modification (insert, replace, or delete), and serializes the entire DOM back to a string. This serialization step creates a new NVARCHAR(MAX) value. For small documents (< 8 KB), this is fast. For large documents (100 KB+), this allocates new memory, copies the entire content, and the resulting string is written to the column in the UPDATE operation.

5. **UPDATE integration:** When JSON_MODIFY is used in an UPDATE SET clause like `SET Attributes = JSON_MODIFY(Attributes, '$.color', 'red')`, the storage engine reads the current value of Attributes from the data page, passes it to JSON_MODIFY, receives the new NVARCHAR(MAX) string, and writes the new value back to the page. If the new string exceeds the current page space, a page split or row forwarding may occur. The entire row is locked for the duration of the UPDATE.

6. **Multiple modifications in one statement:** A common pattern is `SET col = JSON_MODIFY(JSON_MODIFY(col, '$.a', 1), '$.b', 2)`. This chains two JSON_MODIFY calls. The inner call produces a new full document. The outer call parses that new document and produces another new full document. For N modifications on the same document in a single SET, the document is rebuilt N times. For a 50 KB document with 10 modifications, this processes 500 KB of JSON data in one UPDATE.

### SQL Visibility

```sql
-- Basic JSON_MODIFY: update a scalar field
DECLARE @json NVARCHAR(MAX) = N'{"color":"red","size":"M","quantity":10}';

SELECT JSON_MODIFY(@json, '$.color', 'blue') AS UpdatedJson;

-- Result: {"color":"blue","size":"M","quantity":10}

-- JSON_MODIFY with append: add to array
DECLARE @tags NVARCHAR(MAX) = N'{"tags":["sql","json"]}';
SELECT JSON_MODIFY(@tags, 'append$.tags', 'performance') AS WithNewTag;

-- Result: {"tags":["sql","json","performance"]}

-- JSON_MODIFY to add a new property
DECLARE @config NVARCHAR(MAX) = N'{"theme":"dark"}';
SELECT JSON_MODIFY(@config, '$.fontSize', 14) AS WithFontSize;

-- Result: {"theme":"dark","fontSize":14}

-- JSON_MODIFY to delete a property (lax mode, value = NULL)
DECLARE @product NVARCHAR(MAX) = N'{"id":1,"name":"Widget","temporary":true}';
SELECT JSON_MODIFY(@product, '$.temporary', NULL) AS WithoutTemp;

-- Result: {"id":1,"name":"Widget"}

-- JSON_MODIFY update nested value
DECLARE @settings NVARCHAR(MAX) = N'{"user":{"name":"Alice","prefs":{"theme":"light","lang":"en"}}}';
SELECT JSON_MODIFY(@settings, '$.user.prefs.theme', 'dark') AS WithNestedUpdate;

-- Result: {"user":{"name":"Alice","prefs":{"theme":"dark","lang":"en"}}}

-- JSON_MODIFY with JSON_QUERY to insert a JSON object as nested structure
DECLARE @doc NVARCHAR(MAX) = N'{"order":1}';
DECLARE @address NVARCHAR(MAX) = N'{"city":"NYC","zip":"10001"}';
SELECT JSON_MODIFY(@doc, '$.shipping', JSON_QUERY(@address)) AS WithAddressObject;

-- Result: {"order":1,"shipping":{"city":"NYC","zip":"10001"}}

-- Without JSON_QUERY — inserts as escaped string, not as object
SELECT JSON_MODIFY(@doc, '$.shipping', @address) AS WithAddressString;

-- Result: {"order":1,"shipping":"{\"city\":\"NYC\",\"zip\":\"10001\"}"}
```

### Execution Plan Analysis

```sql
-- Example: UPDATE with JSON_MODIFY
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.Color', 'Blue')
WHERE ProductId = 1042;
```

**Expected plan shape (single row by PK):**

```
[Clustered Index Seek (PK_Products)] → [Compute Scalar (JSON_MODIFY)] → [Clustered Index Update]
```

**Operator breakdown:**

- **Clustered Index Seek (PK_Products)** — seeks to ProductId = 1042, retrieves the row (including Attributes column). Logical reads: ~3-4.
- **Compute Scalar (JSON_MODIFY)** — evaluates the JSON_MODIFY function. No new I/O here — this is CPU-bound string processing. The estimated cost depends on document size: for a 10 KB document, ~0.001 cost units; for 500 KB, ~0.05-0.1 cost units.
- **Clustered Index Update** — writes the modified row back. If the new Attributes string is larger than the previous value and the row no longer fits on the same page, a page split may occur. Logical writes: 1 (page) + possibly 1 (index page split) + 1 (log write).

**Plan shape for bulk update (no filter):**

```sql
-- Bulk update — applies to all rows
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.processed', 'true');
```

```
[Table Scan] → [Compute Scalar (JSON_MODIFY)] → [Clustered Index Update]
```

**Logical reads (bulk):** ~N pages for scan + ~N page writes for update (where N = number of pages in the table).

### Failure Modes

**Non-JSON input returns NULL:** If the input expression is NULL or contains invalid JSON (ISJSON returns 0), JSON_MODIFY returns NULL. An UPDATE that sets a column to NULL can cascadingly set other values to NULL in chained calls.

**Lost concurrent updates:** Two concurrent transactions reading and then updating the same JSON column using application-side read-modify-write can lose one update. Using JSON_MODIFY in a single UPDATE statement avoids this because the read and modify happen atomically in the same statement.

```sql
-- ❌ Application-side read-modify-write (lost update risk)
BEGIN TRAN;
DECLARE @attrs NVARCHAR(MAX) = (SELECT Attributes FROM dbo.Products WHERE ProductId = 1);
SET @attrs = JSON_MODIFY(@attrs, '$.Color', 'Blue');
UPDATE dbo.Products SET Attributes = @attrs WHERE ProductId = 1;
COMMIT;
-- Between SELECT and UPDATE, another session may have modified Attributes

-- ✅ Single UPDATE with JSON_MODIFY (atomic)
BEGIN TRAN;
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.Color', 'Blue')
WHERE ProductId = 1;
COMMIT;
-- Read and modify happen atomically in one statement
```

**Escaped string vs JSON object trap:** Passing a string literal that looks like JSON as the third argument without wrapping it in JSON_QUERY inserts it as an escaped string, not a JSON fragment.

**Document size growth:** Repeated updates that add properties without deleting old ones cause the document to grow. At some point, the document exceeds the page size (8,060 bytes) and causes row-overflow storage, which increases read cost.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Schema context: Products table with JSON Attributes column
-- ============================================================
CREATE TABLE dbo.Products
(
    ProductId       INT            NOT NULL IDENTITY(1,1),
    ProductName     NVARCHAR(200)  NOT NULL,
    CategoryId      INT            NOT NULL,
    Price           DECIMAL(18,2)  NOT NULL,
    Attributes      NVARCHAR(MAX)  NULL,  -- JSON column for flexible attributes
    CreatedAt       DATETIME2(0)   NOT NULL DEFAULT SYSUTCDATETIME(),
    UpdatedAt       DATETIME2(0)   NOT NULL DEFAULT SYSUTCDATETIME(),
    CONSTRAINT PK_Products PRIMARY KEY CLUSTERED (ProductId),
    CONSTRAINT CK_Products_Attributes CHECK (ISJSON(Attributes) = 1),
    CONSTRAINT FK_Products_Category FOREIGN KEY (CategoryId)
        REFERENCES dbo.ProductCategories(CategoryId)
);

-- Sample JSON in Attributes:
-- {"Color":"Red","Size":"M","Weight":1.5,"Tags":["sale","featured"],"Dimensions":{"W":10,"H":20,"D":5}}

-- ============================================================
-- Pattern 1: Update a scalar property
-- SARGability: N/A (UPDATE with PK seek)
-- ============================================================
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.Color', 'Blue'),
    UpdatedAt = SYSUTCDATETIME()
WHERE ProductId = 1042;

-- ============================================================
-- Pattern 2: Append to an array
-- JSON_MODIFY with 'append' keyword adds to end of array
-- ============================================================
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, 'append$.Tags', 'clearance')
WHERE ProductId = 1042;

-- ============================================================
-- Pattern 3: Delete a property (lax mode, value = NULL)
-- ============================================================
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.Weight', NULL)
WHERE ProductId = 1042;

-- ============================================================
-- Pattern 4: Insert a JSON object as nested property
-- Must use JSON_QUERY to prevent escaping
-- ============================================================
DECLARE @warranty NVARCHAR(MAX) = N'{"type":"extended","years":3,"cost":49.99}';

UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.Warranty', JSON_QUERY(@warranty))
WHERE ProductId = 1042;

-- ============================================================
-- Pattern 5: Update a nested value at depth
-- ============================================================
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.Dimensions.W', 12)
WHERE ProductId = 1042;

-- ============================================================
-- Pattern 6: Multiple modifications in one statement
-- Each JSON_MODIFY call rebuilds the full document
-- ============================================================
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(
    JSON_MODIFY(
        JSON_MODIFY(Attributes, '$.Color', 'Green'),
        '$.Size', 'L'
    ),
    'append$.Tags', 'summer'
)
WHERE ProductId = 1042;
-- Note: three full document rebuilds in one UPDATE

-- ============================================================
-- Pattern 7: Conditional update with CASE and JSON_VALUE
-- ============================================================
UPDATE dbo.Products
SET Attributes = CASE
    WHEN JSON_VALUE(Attributes, '$.Color') = 'Red'
        THEN JSON_MODIFY(Attributes, '$.Color', 'Blue')
    WHEN JSON_VALUE(Attributes, '$.Color') = 'Blue'
        THEN JSON_MODIFY(Attributes, '$.Color', 'Green')
    ELSE Attributes
END
WHERE ProductId = 1042;

-- ============================================================
-- Pattern 8: Bulk update all rows with a new property
-- ============================================================
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.auditTimestamp',
    FORMAT(SYSUTCDATETIME(), 'yyyy-MM-ddTHH:mm:ssZ'))
WHERE JSON_VALUE(Attributes, '$.auditTimestamp') IS NULL;

-- ============================================================
-- Anti-pattern: Application-side read-modify-write
-- ============================================================
-- ❌ DO NOT DO THIS (in T-SQL or application code):
BEGIN TRAN;
DECLARE @attrs NVARCHAR(MAX);
SELECT @attrs = Attributes FROM dbo.Products WHERE ProductId = 1042;
SET @attrs = JSON_MODIFY(@attrs, '$.Color', 'Red');
UPDATE dbo.Products SET Attributes = @attrs WHERE ProductId = 1042;
COMMIT;

-- ✅ DO THIS instead:
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.Color', 'Red')
WHERE ProductId = 1042;
```

```csharp
                    Attributes,
                    '$.Warranty',
                    JSON_QUERY(@WarrantyJson)),
                UpdatedAt = SYSUTCDATETIME()
            WHERE ProductId = @ProductId;";

        await using var connection = _connectionFactory.Create();
        await connection.ExecuteAsync(
            new CommandDefinition(sql,
                new { ProductId = productId, WarrantyJson = warrantyJson },
                cancellationToken: cancellationToken));
    }

    // Multiple modifications in a single UPDATE
    public async Task BulkUpdateColorForCategoryAsync(
        int categoryId,
        string oldColor,
        string newColor,
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            UPDATE dbo.Products
            SET Attributes = JSON_MODIFY(Attributes, '$.Color', @NewColor),
                UpdatedAt = SYSUTCDATETIME()
            WHERE CategoryId = @CategoryId
              AND JSON_VALUE(Attributes, '$.Color') = @OldColor;";

        await using var connection = _connectionFactory.Create();
        await connection.ExecuteAsync(
            new CommandDefinition(sql,
                new
                {
                    CategoryId = categoryId,
                    OldColor = oldColor,
                    NewColor = newColor
                },

    }
```

## Gotchas and Production Pitfalls

### Full Document Rebuild on Every JSON_MODIFY Call

**Pitfall:** Chaining multiple JSON_MODIFY calls on a large JSON document in a single UPDATE, assuming each call is cheap because it only modifies a small field.

```sql
-- ❌ Three full document rebuilds on a 500 KB document
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(
    JSON_MODIFY(
        JSON_MODIFY(Attributes, '$.field1', 'val1'),
        '$.field2', 'val2'
    ),
    '$.field3', 'val3'
)
WHERE ProductId = 1042;
```

**Symptom:** UPDATE on a 500 KB document with 10 chained modifications takes 50+ ms vs 5 ms for a single modification. CPU time is high relative to logical reads. Extended events show high `sql_statement_completed` duration.

**Fix:**

```sql
-- ✅ Option 1: Reduce chaining — use one modification per statement
UPDATE dbo.Products SET Attributes = JSON_MODIFY(Attributes, '$.field1', 'val1') WHERE ProductId = 1042;
UPDATE dbo.Products SET Attributes = JSON_MODIFY(Attributes, '$.field2', 'val2') WHERE ProductId = 1042;

-- ✅ Option 2: Build the full document in the application tier and do one write
-- Deserialize → modify → serialize → single UPDATE without JSON_MODIFY
```

**Cost of not fixing:** At 100 updates/second on a 500 KB document with 5 chained modifications, the server processes 250 MB/s of JSON string data through JSON_MODIFY — consuming significant CPU and causing buffer pool pressure from large NVARCHAR(MAX) allocations.

---

### Escaped String Instead of JSON Fragment

**Pitfall:** Passing a JSON-formatted string as the third argument without wrapping it in JSON_QUERY — the JSON is inserted as an escaped string literal.

**Symptom:** After the update, `JSON_VALUE(Attributes, '$.nestedProp')` returns NULL because the value is a string `'{"key":"val"}'` not an object. `JSON_QUERY(Attributes, '$.nestedProp')` also returns NULL because the value is not a JSON fragment — it is a JSON string containing what looks like JSON but is escaped.

```sql
-- ❌ This inserts the string, not the JSON object
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.metadata', '{"source":"web","version":2}')
WHERE ProductId = 1042;
-- Result: {"metadata":"{\"source\":\"web\",\"version\":2}"}

-- ✅ Use JSON_QUERY to treat the string as a JSON fragment
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.metadata', JSON_QUERY('{"source":"web","version":2}'))
WHERE ProductId = 1042;
-- Result: {"metadata":{"source":"web","version":2}}
```

**Fix:**
```sql
-- Always wrap JSON object/array strings in JSON_QUERY for the value parameter
JSON_MODIFY(col, '$.path', JSON_QUERY(@jsonString))
-- Scalar values do not need JSON_QUERY
JSON_MODIFY(col, '$.path', 'scalar string')
JSON_MODIFY(col, '$.path', 42)
```

**Cost of not fixing:** Broken application logic that reads Attributes expecting `$.metadata.source` to return 'web' instead gets NULL. This bug is subtle because the JSON is still valid — it just has a different structure.

---

### Concurrent Update Lost with Application-Side Read-Modify-Write

**Pitfall:** Reading a JSON column into the application, modifying it with JSON_MODIFY (or any deserialization), and writing it back — the classic non-atomic read-modify-write.

**Symptom:** Intermittently, property updates are lost. Session A reads Attributes, Session B reads same Attributes, Session A modifies Color and writes, Session B modifies Size and writes — Session A's Color change is overwritten by Session B's write of the old Attributes value.

```sql
-- ❌ Application code does this implicitly when using EF Core < 8
-- without concurrency tokens:
-- var p = db.Products.Find(1);
-- p.Attributes.Color = "Blue";
-- db.SaveChanges();
-- If two requests do this concurrently, one overwrites the other
```

**Fix:**
```sql
-- ✅ Always use JSON_MODIFY in a single UPDATE when only modifying one path
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.Color', 'Blue')
WHERE ProductId = 1042;

-- ✅ In EF Core 8+, the generated SQL is a single UPDATE with JSON_MODIFY
-- as long as the tracked entity was loaded within the same DbContext instance
-- Use a RowVersion or high concurrency token if needed
```

**Cost of not fixing:** Silent data corruption — customer attribute changes disappear in production. May take weeks to discover because the bug is race-condition dependent.

---

### Document Size Growth and Page Splits

**Pitfall:** Repeatedly adding properties to a JSON document without pruning, causing the document to grow beyond 8,060 bytes and triggering row-overflow storage.

**Symptom:** Initial INSERT/UPDATE performance is fast. Over months of incremental additions, the Attributes column grows to 20 KB, 50 KB, or more. Updates become slower. `ALTER TABLE ... REBUILD` operations take longer. Page splits occur frequently during UPDATE operations.

```sql
-- Monitoring query to find large JSON documents
SELECT
    ProductId,
    DATALENGTH(Attributes) / 1024.0 AS JsonSizeKB,
    ProductName
FROM dbo.Products
WHERE DATALENGTH(Attributes) > 8000  -- > 8 KB triggers row overflow
ORDER BY JsonSizeKB DESC;
```

**Fix:**
```sql
-- ✅ Strategy 1: Archive old properties to a separate table
-- Create an audit log table for historical attribute values

-- ✅ Strategy 2: Set a maximum document size in application logic
-- Enforce < 8 KB per document to keep data in-row

-- ✅ Strategy 3: Use a schema with dedicated columns for frequently-queried properties
-- Store only truly variable/flexible attributes in JSON
```

**Cost of not fixing:** 50 KB documents on a table with 1M rows consume 50 GB of storage (vs 8 GB if in-row). UPDATE performance degrades 10-50x. Backup/restore times increase proportionally.

---

### Strict Mode Errors When Path Is Missing

**Pitfall:** Using strict mode in JSON_MODIFY path and expecting it to insert the property when the path does not exist, similar to lax mode.

```sql
-- ❌ Strict mode errors when path is missing
-- This JSON document has $.Color but $.Warranty does not exist
DECLARE @json NVARCHAR(MAX) = N'{"Color":"Red","Size":"M"}';

-- THIS THROWS AN ERROR:
SELECT JSON_MODIFY(@json, 'strict$.Warranty.type', 'extended');
-- Error: JSON path "strict$.Warranty.type" is not found
```

**Symptom:** An INSERT or UPDATE fails with error code 13624 — "JSON path is not found in strict mode." The transaction rolls back if unhandled.

**Fix:**
```sql
-- ✅ Use lax mode (default) when you want to insert new properties
SELECT JSON_MODIFY(@json, '$.Warranty.type', 'extended');
-- Works: inserts $.Warranty.type with value "extended"

-- ✅ Use strict mode only when you want to ensure the path exists before modification
SELECT JSON_MODIFY(@json, 'strict$.Color', 'Blue');
-- Works: $.Color exists

-- ✅ Use NULL result handling to catch strict mode failures
SELECT JSON_MODIFY(@json, 'strict$.Warranty', 'extended');  -- errors
-- Use ISNULL or COALESCE to handle:
SELECT ISNULL(JSON_MODIFY(@json, 'strict$.Warranty', 'extended'), @json);
```

**Cost of not fixing:** A production job fails at 3 AM because a JSON structure changed slightly (a property was removed in a different code path but the JSON_MODIFY strict statement was not updated). Monitoring alerts on unhandled SQL exceptions wake up the on-call engineer.

---

### JSON_MODIFY with NULL Loses the Entire Row

**Pitfall:** Using JSON_MODIFY with a NULL first argument (the JSON document is NULL for the row) causes the entire UPDATE to set the column to NULL.

```sql
-- If ProductId = 9999 has Attributes = NULL:
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.Color', 'Blue')
WHERE ProductId = 9999;
-- Result: Attributes remains NULL (JSON_MODIFY(NULL, ...) returns NULL)
```

**Symptom:** No error occurs. The row is updated (UpdatedAt changes) but Attributes remains NULL. Code that reads Attributes later expects a JSON document and breaks with JSON parsing errors.

**Fix:**
```sql
-- ✅ Provide a default JSON document if the column is NULL
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(
    ISNULL(Attributes, N'{}'),
    '$.Color',
    'Blue'
)
WHERE ProductId = 9999;

-- ✅ Or add a CHECK constraint to prevent NULL
ALTER TABLE dbo.Products
    ADD CONSTRAINT CK_Products_Attributes_NotNull
    CHECK (Attributes IS NOT NULL);
```

**Cost of not fixing:** Silent data loss — attributes are not saved because the column was NULL. Customer-facing features (product filters, attribute search) return incorrect results.

### Benchmark: Before and After

### Spoken Answers

**Q: What does JSON_MODIFY do, and how is it different from application-side JSON manipulation?**

> **Average answer:** JSON_MODIFY updates JSON in SQL Server. It changes a value at a path. Application-side means reading the data into C#, deserializing, modifying, and writing it back.

> **Great answer:** JSON_MODIFY is a T-SQL function that takes a JSON string, a JSON path, and a new value, and returns a new JSON string with the modification applied. Critical distinction: it is NOT an in-place mutation — it parses the entire input JSON, modifies the target path, and serializes the entire document back to a new NVARCHAR(MAX) string. Every single call rebuilds the full document. The performance implication is that a 100 KB document is fully processed on every JSON_MODIFY call, regardless of whether you are changing a single character or a complex subdocument. Application-side read-modify-write has the same rebuilding cost in the application layer but adds an extra network round trip and, critically, creates a lost-update window — between the SELECT that reads the JSON and the UPDATE that writes it back, another session can modify the same column, and one of the updates is silently lost. JSON_MODIFY in a single UPDATE statement eliminates this window because the read and write are atomic within the same statement. The pattern I always recommend: if you only need to change one field, use JSON_MODIFY in a single UPDATE. If you need to change many fields, consider whether chaining JSON_MODIFY calls is acceptable (for small documents, it is fine; for 50 KB+ documents, consider building the new document in the application and doing a single direct write — but accept the concurrency risk and use row versioning to detect conflicts).

---

**Q: JSON_MODIFY in lax mode vs strict mode: compare behavior when the path does not exist.**

> **Average answer:** Lax mode creates the path if it doesn't exist. Strict mode throws an error.

> **Great answer:** Lax mode is the default. When JSON_MODIFY operates in lax mode and the specified path does not exist in the JSON document, the function inserts the new property at that path, creating intermediate object structures as needed. For example, if you run `JSON_MODIFY(@doc, '$.metadata.audit.user', 'admin')` and `$.metadata` does not exist, lax mode creates the `metadata` object, then the `audit` object inside it, and sets `user` to `"admin"`. The resulting document has the full nested structure. Strict mode, on the other hand, requires the path to exist before modification. If the path is not found, JSON_MODIFY returns NULL and, if the error is not handled, raises error 13624: "JSON path is not found in strict mode." This is not an error in lax mode — it is a structural difference. I use strict mode specifically for defensive validation: if I am updating a property that should exist (like a required configuration field), strict mode catches structural inconsistencies. I use lax mode for adding optional properties. The NULL behavior also differs by mode: in lax mode, setting the value to NULL deletes the property. In strict mode, setting the value to NULL sets the property value to the JSON null literal `null` — it does not delete it. This is a critical behavioral difference that I have seen cause production bugs: `JSON_MODIFY(col, 'strict$.path', NULL)` sets the value to `null` (still exists in document), while `JSON_MODIFY(col, '$.path', NULL)` in lax mode removes the property entirely.

---

**Q: How does EF Core 8+ translate JSON column updates to T-SQL?**

> **Average answer:** EF Core 8 supports JSON columns and generates JSON_MODIFY calls.

> **Great answer:** EF Core 8 introduced native JSON column support for SQL Server. When you configure an owned entity with `.ToJson()`, EF Core maps it as an NVARCHAR(MAX) column and tracks changes to individual properties within the JSON document. When you call `SaveChangesAsync`, EF Core generates individual JSON_MODIFY calls for each changed property. If you change both `Color` and `Size`, EF Core generates a chained UPDATE: `UPDATE [Products] SET [Attributes] = JSON_MODIFY(JSON_MODIFY([Attributes], '$.Color', @p0), '$.Size', @p1) WHERE [Id] = @p2`. This is efficient because it happens in a single database round trip, but the document is rebuilt for each changed property. For a typical JSON document under 10 KB with 2-3 field changes, this is fine. For large documents with many changes, the chaining cost adds up. I always profile the generated SQL using EF Core's SQL logging to verify the exact JSON_MODIFY calls. An important caveat: if you replace the entire JSON document (set the property to a new instance), EF Core writes the new document directly without JSON_MODIFY — this is actually cheaper for large documents with many changes because it is just a single NVARCHAR(MAX) write. The key EF Core 7- limitation: without the JSON column support, developers used value converters that serialize the entire object to JSON and back — every SaveChangesAsync wrote the entire document regardless of what changed. EF Core 8 is strictly better for partial updates.

### Application Checklist

- [ ] JSON_MODIFY used in a single UPDATE — no separate SELECT followed by UPDATE in application code
- [ ] If inserting a JSON object/array, the third argument is wrapped in JSON_QUERY
- [ ] Chained JSON_MODIFY calls on large documents (> 50 KB) avoided — consider alternative
- [ ] Lax vs strict mode chosen deliberately for each JSON_MODIFY call
- [ ] NULL handling: ISNULL used to provide default JSON document if column can be NULL
- [ ] Document size monitored — no single document exceeds ~8 KB for in-row storage
- [ ] EF Core 8+ JSON column updates verified via SQL logging — correct JSON_MODIFY generation
- [ ] Dapper queries use parameterized JSON_MODIFY with correct data types
- [ ] RowVersion or concurrency token used when application-side document replacement is necessary
- [ ] Execution plan reviewed — expects Clustered Index Seek + Compute Scalar + Update

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Atomic single-statement field update — no lost update|Full document rebuild on every JSON_MODIFY call|
|Reduced network round trips (1 vs 2)|String serialization CPU cost in SQL Server|
|Simpler application code — no manual JSON patching|Complex chaining logic for multi-field updates|
|Compatible with EF Core 8+ JSON column feature|Not all ORM operations translate to optimized JSON_MODIFY|
|Transaction-safe within the UPDATE statement|Page splits if document grows significantly|

### Scale Thresholds

- JSON_MODIFY cost becomes noticeable when documents exceed **~50 KB** — the full-string rebuild time starts to dominate CPU.
- Chained JSON_MODIFY calls become problematic at **3+ modifications on documents > 50 KB** — each chain doubles the effective data processed.
- The page split threshold is **~8,060 bytes** — documents larger than this trigger row-overflow storage, increasing read/write cost by 3-10x.
- Application-side build becomes more efficient than chained JSON_MODIFY when modifying **5+ fields on documents > 100 KB** — because the app builds the document once instead of SQL Server rebuilding it N times.
- Concurrency conflicts from application-side read-modify-write become a real problem when the same row is updated **> 10 times/second** — the lost-update probability exceeds 1%.

4. What happens when you pass a JSON object string as the third argument without wrapping it in JSON_QUERY?
5. In EF Core 8+, does SaveChangesAsync on a JSON column property generate a full column rewrite or a JSON_MODIFY call?
6. How would you implement a field update on a JSON column using Dapper?
7. Compare lax mode vs strict mode in JSON_MODIFY — what happens when the path is missing?
8. At what document size does JSON_MODIFY performance become a concern?
9. What index is used when updating a JSON column via the primary key?
10. Explain in 60 seconds why application-side read-modify-write on JSON columns is problematic and how JSON_MODIFY solves it.

<details>
<summary>Answers</summary>

1. JSON_MODIFY updates a JSON string at a specified path and returns a new NVARCHAR(MAX) string with the modification applied. It takes three arguments: the input JSON expression, the path with optional mode prefix, and the new value.

2. No — JSON_MODIFY does NOT mutate the original string. It parses the entire input JSON into an internal DOM, applies the modification at the target path, and serializes the entire document back to a new NVARCHAR(MAX) string. The original string is not modified. This means the cost is proportional to the document size, not the field size.

3. `SET STATISTICS IO ON` shows the logical reads for the table scan/seek and logical writes for the update. `SET STATISTICS TIME ON` shows the CPU time, which reflects the string processing cost of JSON_MODIFY.

4. The JSON object string is inserted as an escaped JSON string literal, not as a nested JSON object. For example, `'{"key":"val"}'` becomes `"{\"key\":\"val\"}"` — a JSON string, not a JSON object. Always wrap JSON fragments with `JSON_QUERY()` to insert them as structured JSON.

5. EF Core 8+ generates individual JSON_MODIFY calls for each changed property of the JSON column. If you change one property, one JSON_MODIFY call. If you change three properties, three chained JSON_MODIFY calls in a single UPDATE statement. The full document is NOT rewritten unless you replace the entire navigation property with a new instance.

6. With Dapper, execute a parameterized UPDATE using `JSON_MODIFY`:

```csharp
const string sql = @"
    UPDATE dbo.Products
    SET Attributes = JSON_MODIFY(Attributes, '$.Color', @NewColor)
    WHERE ProductId = @ProductId;";
await connection.ExecuteAsync(new CommandDefinition(sql,
    new { ProductId = 1042, NewColor = "Blue" },
    cancellationToken: cancellationToken));
```

7. In **lax mode** (default), if the path does not exist, JSON_MODIFY creates intermediate objects and inserts the property. Setting the value to NULL deletes the property. In **strict mode**, if the path does not exist, JSON_MODIFY returns NULL and raises error 13624. Setting the value to NULL in strict mode sets the property value to JSON `null` — it does NOT delete the property.

8. JSON_MODIFY performance becomes noticeable when documents exceed ~50 KB and the update runs more than 100x/hour. At 50 KB, each JSON_MODIFY rebuilds 50 KB of string. At 500 KB, each call processes 500 KB. Chained calls multiply this: 5 calls on a 500 KB document processes 2.5 MB of string data in one UPDATE.

9. The UPDATE uses the clustered index (PK_Products) to find the row via Clustered Index Seek when filtering by ProductId. The JSON_MODIFY function runs in a Compute Scalar operator. The update operation writes the new value back to the clustered index data page.

10. Application-side read-modify-write does this: SELECT Attributes to app → deserialize → modify → serialize → UPDATE set Attributes = newValue. Between the SELECT and UPDATE, another session can modify the same row, and the second UPDATE silently overwrites the first — a lost update. JSON_MODIFY eliminates this because the SELECT and modify happen atomically within the SQL statement: `UPDATE SET col = JSON_MODIFY(col, '$.path', val)` reads the current value, modifies it, and writes it back in one statement, under the same row lock, with no window for another session to interleave a conflicting write.

</details>

```sql
UPDATE dbo.Products
```

**Challenge 2 — Fix the lost-update problem**

```sql
-- This application code runs in a high-traffic microservice.
-- Attribute updates are intermittently lost in production.
-- Identify why and fix it.

-- Application code (simplified):
-- var product = db.Products.Find(productId);
-- product.Attributes.Color = newColor;
```

**Challenge 3 — Explain the execution plan**

```sql
-- Given this query:
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(
    JSON_MODIFY(Attributes, '$.Color', 'Green'),
    '$.Size', 'L'
)
WHERE ProductId = 1042;

-- And this execution plan:
-- |--Clustered Index Update(OBJECT:([Products].[PK_Products]))
--    |--Compute Scalar(DEFINE:([Products].[Attributes]=json_modify(json_modify(...))))
--       |--Clustered Index Seek(OBJECT:([Products].[PK_Products]))
```

Why does the plan show two JSON_MODIFY calls in a single Compute Scalar? What happens if the Attributes column is 200 KB and we chain 10 JSON_MODIFY calls?

<details> <summary>Solution</summary>

**Why two JSON_MODIFY in one Compute Scalar:** The Compute Scalar operator evaluates all expressions in the SET clause. The outer expression is `JSON_MODIFY(JSON_MODIFY(...))`. SQL Server evaluates the inner JSON_MODIFY first (producing a new full document string), then passes that result to the outer JSON_MODIFY (which parses it and produces another new full document). This all happens within the Compute Scalar — no additional I/O, just CPU-bound string processing.

**At 200 KB with 10 chained calls:** SQL Server processes approximately 10 * 200 KB = 2 MB of JSON string data within the Compute Scalar. This takes 20-50 ms of CPU time for the single row update. The Clustered Index Seek and Update each take < 1 ms. The bottleneck is CPU, not I/O. The logical read count stays the same (~4 for seek + writes), but CPU time increases linearly with document size and chain count.

**How to optimize:** For large documents with many field changes, build the final document in the application and do a single direct NVARCHAR(MAX) write — or restructure the schema to move large JSON documents to a separate table.

</details>

**Challenge 4 — Diagnose the document growth problem**

A table `dbo.Products` has been in production for 18 months. The `Attributes` NVARCHAR(MAX) column contains JSON that has grown from an average of 2 KB to 45 KB per row due to repeated property additions by various microservices. The nightly maintenance job that rebuilds all product attributes now takes 4 hours. UPDATE statements on individual products that used to take 2 ms now take 35 ms. Identify the root cause and propose a remediation plan.

<details> <summary>Solution</summary>

**Root cause:** Document bloat — the Attributes column has grown past the in-row limit (8,060 bytes) into LOB/row-overflow storage. Each JSON_MODIFY call now processes a 45 KB string instead of 2 KB, and the LOB storage format requires additional page reads for both read and write operations. The nightly rebuild job is slow because it deserializes and re-serializes 45 MB of JSON data per 1000 products.

**Detection query:**
```sql
SELECT
    AVG(DATALENGTH(Attributes) / 1024.0) AS AvgSizeKB,
    MAX(DATALENGTH(Attributes) / 1024.0) AS MaxSizeKB,
    COUNT(*) AS TotalRows
FROM dbo.Products
WHERE DATALENGTH(Attributes) > 8000;
```

**Fix:**
```sql
-- Option 1: Archive historical attributes to audit table
-- Move old property snapshots to dbo.ProductAttributeHistory
INSERT INTO dbo.ProductAttributeHistory (ProductId, AttributeSnapshot, CreatedAt)
SELECT ProductId, Attributes, SYSUTCDATETIME()
FROM dbo.Products
WHERE JSON_VALUE(Attributes, '$.lastUpdated') < DATEADD(MONTH, -6, SYSUTCDATETIME());

-- Option 2: Extract frequently-queried properties to dedicated columns
ALTER TABLE dbo.Products ADD
    Color NVARCHAR(50) NULL,
    Size NVARCHAR(20) NULL,
    Weight DECIMAL(10,2) NULL;

UPDATE dbo.Products
SET Color = JSON_VALUE(Attributes, '$.Color'),
    Size = JSON_VALUE(Attributes, '$.Size'),
    Weight = TRY_CAST(JSON_VALUE(Attributes, '$.Weight') AS DECIMAL(10,2));

-- Option 3: Set a soft cap on document size in middleware
-- Reject updates that would make the document exceed 50 KB
```

**In .NET:**
```csharp
public async Task<bool> TryUpdateAttributeAsync<T>(
    int productId,
    string propertyName,
    T value,
    int maxDocSizeKB = 50)
{
    const string checkSql = @"
        SELECT DATALENGTH(Attributes)
        FROM dbo.Products
        WHERE ProductId = @ProductId;";

    var currentSize = await _connection.ExecuteScalarAsync<int>(checkSql, new { ProductId = productId });
    if (currentSize > maxDocSizeKB * 1024)
        return false;

    const string updateSql = @"
        UPDATE dbo.Products
        SET Attributes = JSON_MODIFY(Attributes, @Path, @Value)
        WHERE ProductId = @ProductId;";

    await _connection.ExecuteAsync(updateSql,
        new { ProductId = productId, Path = $"$.{propertyName}", Value = value });
    return true;
}
```

**Challenge 5 — Design the JSON update strategy**

**Scenario:** An e-commerce platform uses a `dbo.Products` table with 5 million rows. Each product has an `Attributes` NVARCHAR(MAX) JSON column (avg size: 15 KB, max: 200 KB). Three microservices update this column:

1. **Catalog service:** Updates `$.Price`, `$.StockLevel`, `$.Status` — 10,000 updates/second
2. **Recommendation service:** Updates `$.PopularityScore`, `$.RelatedProductIds` — 500 updates/second
3. **CMS service:** Updates `$.Description`, `$.Images`, `$.SeoKeywords` — 100 updates/second, documents often rewritten entirely

Design the update strategy. Should services use JSON_MODIFY, full document replacement, or something else? What concurrency mechanism prevents lost updates?

<details> <summary>Solution</summary>

**Recommended strategy:**

1. **Catalog service (10K/sec, high concurrency, specific fields):** Use individual JSON_MODIFY calls per field — one UPDATE per property. With 15 KB documents, each JSON_MODIFY rebuilds 15 KB. At 10K/sec, this is 150 MB/sec of JSON string processing. Acceptable on modern hardware. Add a RowVersion column and include it in the UPDATE WHERE clause to detect conflicts. If conflict rate exceeds 5%, consider queue-based writes.

2. **Recommendation service (500/sec, single field):** Same approach — JSON_MODIFY on `$.PopularityScore`. Low enough volume that chaining is unnecessary.

3. **CMS service (100/sec, full rewrites):** Build the full JSON document in the application and do a single UPDATE with the complete document. Use RowVersion for concurrency. At 100/sec, the full rewrite is cheaper than 10+ chained JSON_MODIFY calls on a 200 KB document.

```sql
-- Catalog service pattern (10K/sec)
UPDATE dbo.Products
SET Attributes = JSON_MODIFY(Attributes, '$.StockLevel', @StockLevel),
    UpdatedAt = SYSUTCDATETIME()
WHERE ProductId = @ProductId
  AND RowVersion = @ExpectedRowVersion;

-- On conflict (0 rows affected): retry with fresh read
```

```sql
-- CMS service pattern (100/sec, full rewrite)
UPDATE dbo.Products
SET Attributes = @FullJsonDocument,
    UpdatedAt = SYSUTCDATETIME()
WHERE ProductId = @ProductId
  AND RowVersion = @ExpectedRowVersion;
```

**Concurrency mechanism:** Add a `RowVersion ROWVERSION NOT NULL` column. Every service includes `RowVersion = @ExpectedRowVersion` in the UPDATE WHERE clause. If the row was modified since the service read it, the UPDATE affects 0 rows and the application retries (read the fresh row, apply changes, retry UPDATE). For the Catalog service's 10K/sec volume, retry with exponential backoff (50ms, 100ms, 200ms). If retry rate exceeds 10%, investigate whether the RowVersion granularity is too fine.

**Tradeoff summary:**

|Service|Approach|Throughput|Conflict Handling|
|---|---|---|---|
|Catalog|JSON_MODIFY per field|10K/sec|RowVersion + retry (50ms backoff)|
|Recommendation|JSON_MODIFY per field|500/sec|RowVersion + retry|
|CMS|Full document replacement|100/sec|RowVersion + retry (full re-read of large doc)|

</details>
