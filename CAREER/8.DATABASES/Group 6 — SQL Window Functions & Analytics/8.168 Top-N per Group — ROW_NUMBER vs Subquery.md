### Where This Fits

Top-N per Group is one of the most common SQL patterns in production: "Get the top 3 products by revenue per category," "Find the most recent order per customer," "Retrieve the highest-paid employee per department." The pattern uses ROW_NUMBER() OVER(PARTITION BY group_col ORDER BY value DESC) to assign ranks within each group, then filters WHERE rn <= N to keep only the top rows. A .NET backend engineer encounters this in dashboards, reporting feeds, leaderboards, and any "per-group top-N" requirement. The interview signal is critical: candidates must know the three main approaches (ROW_NUMBER, CROSS APPLY, correlated subquery), understand their performance profiles, and recognise that ROW_NUMBER is typically 5-50x faster than the correlated subquery approach at scale.

## Core Mental Model

Top-N per Group solves the problem: "For each group, give me the N rows with the highest (or lowest) values." The ROW_NUMBER approach assigns a sequential rank (1, 2, 3, ...) to each row within its group based on the specified ORDER BY, then filters to rows where the rank <= N. The key invariant: ROW_NUMBER restarts at 1 for each partition, so the top N rows in each group are independently identified. The execution plan uses a single Sort (ordered by PARTITION BY columns + ORDER BY columns) followed by a Window Spool or Segment Top operator. The alternative CROSS APPLY approach uses a correlated subquery with TOP N for each group, which translates to a Nested Loops join — one subquery execution per group. ROW_NUMBER is typically faster because it processes all groups in a single sorted pass, while CROSS APPLY may win when the number of groups is small and the index supports efficient seeks per group.

### Classification

**For SQL topics:** ROW_NUMBER is a ranking window function. The Top-N per Group pattern belongs to the ranking/filtering family. The optimiser can use either a Window Spool (needs Sort) or a Segment Top operator (if the index provides the required order). The pattern is SARGable only in the WHERE clause that filters the base data before the window function; the `WHERE rn <= N` filter on the window function output is a post-computation filter. The three implementation approaches have different execution plan shapes — ROW_NUMBER uses a single data scan, CROSS APPLY uses Nested Loops with one scan per group, and correlated subquery also uses Nested Loops but evaluates the subquery per outer row.

## Deep Mechanics

### How the Engine Executes This

**ROW_NUMBER approach execution:**

1. **Data retrieval:** SQL Server reads rows from the base table, applying any WHERE filters. For efficiency, a covering index on (PARTITION BY column, ORDER BY column) INCLUDE (other selected columns) avoids key lookups.

2. **Sort phase:** The Window Spool requires rows sorted by PARTITION BY columns (groups), then by ORDER BY columns (rank). If no index provides this order, SQL Server inserts a Sort operator. The sort order must match the OVER clause exactly: `ORDER BY group_col, value DESC`.

3. **Segment Top vs Window Spool:** SQL Server may choose a Segment Top operator instead of Window Spool if the data is already sorted by the PARTITION BY + ORDER BY columns. Segment Top is a non-blocking operator that passes through only the top N rows per group as it reads them — it is more efficient because it does not require a spool. The optimiser chooses Segment Top when it detects the Top-N pattern and the Sort is already satisfied by an index.

4. **Filter phase:** The WHERE clause `rn <= N` filters out non-top rows. The `rn` column is computed in the CTE or subquery, not available in the outer WHERE of the same SELECT (hence the CTE or derived table).

**CROSS APPLY approach execution:**

1. SQL Server gets the distinct list of group values (e.g., distinct CategoryIds). This list may come from an index on the group column.

2. For each group value, SQL Server executes the inner query with TOP N. If a covering index on (group, order, include) exists, the inner query is an index seek + scan of N rows — extremely efficient.

3. The results are concatenated via a Nested Loops Apply operator.

**Correlated subquery approach execution (worst):**

1. SQL Server scans the outer table. For each row, it evaluates the correlated subquery: `SELECT COUNT(*) FROM table t2 WHERE t2.group = t1.group AND t2.value > t1.value`. This subquery must be executed for every row in the outer result.

2. If N=5, `WHERE (SELECT COUNT(*) ...) < 5` keeps the top 5 per group. Each subquery execution requires an index seek on (group, value). For 50,000 rows, the subquery runs 50,000 times — 50,000 index seeks.

```sql
-- ============================================================
-- Approach 1: ROW_NUMBER in CTE (recommended — best general case)
-- ============================================================
WITH ranked AS (
    SELECT
        p.ProductId,
        p.ProductName,
        p.CategoryId,
        p.UnitPrice,
        ROW_NUMBER() OVER (
            PARTITION BY p.CategoryId
            ORDER BY p.UnitPrice DESC
        ) AS rn
    FROM dbo.Products p
)
SELECT r.ProductId, r.ProductName, r.CategoryId, r.UnitPrice
FROM ranked r
WHERE r.rn <= 3
ORDER BY r.CategoryId, r.UnitPrice DESC;

-- Approach 2: CROSS APPLY (best with few groups + index)
-- ============================================================
SELECT c.CategoryId, c.CategoryName, p.ProductId, p.ProductName, p.UnitPrice
FROM dbo.Categories c
CROSS APPLY (
    SELECT TOP 3
        p.ProductId, p.ProductName, p.UnitPrice
    FROM dbo.Products p
    WHERE p.CategoryId = c.CategoryId
    ORDER BY p.UnitPrice DESC
) p
ORDER BY c.CategoryId, p.UnitPrice DESC;

-- Approach 3: Correlated subquery (NOT recommended)
-- ============================================================
SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice
FROM dbo.Products p
WHERE (
    SELECT COUNT(*)
    FROM dbo.Products p2
    WHERE p2.CategoryId = p.CategoryId
      AND p2.UnitPrice > p.UnitPrice
) < 3
ORDER BY p.CategoryId, p.UnitPrice DESC;
```

### Execution Plan Analysis

**ROW_NUMBER approach (covering index exists):**

```
[Index Scan (NonClustered) IX_Products_CategoryId_UnitPrice — ordered]
  → [Segment Top]
      Partition By: [Products].CategoryId
      Order By: [Products].UnitPrice DESC
      Top: 3
  → [SELECT]
Estimated Cost: ~2.5  |  Logical Reads: ~45
No Sort needed — index provides the required order
```

**ROW_NUMBER approach (no covering index):**

```
[Clustered Index Scan Products]
  → [Sort — ORDER BY CategoryId, UnitPrice DESC] ← 70% of cost
  → [Window Spool — computes ROW_NUMBER]
  → [Filter — rn <= 3]
  → [SELECT]
Estimated Cost: ~18  |  Logical Reads: ~450
Sort is the dominant cost — 50,000 rows sorted
```

**CROSS APPLY approach (index on group column):**

```
[Index Scan (or Seek) Categories] — 10 groups
  → [Nested Loops (Apply)]
      → [Index Seek IX_Products_CategoryId_UnitPrice] — per group
      → [Top — 3 rows per execution]
  → [SELECT]
Estimated Cost: ~1.5  |  Logical Reads: ~32
Highly efficient for few groups
```

**Correlated subquery (worst plan):**

```
[Clustered Index Scan Products] — 50,000 rows
  → [Nested Loops (Left Semi Join)]
      → [Index Seek IX_Products_CategoryId_UnitPrice] — 50,000 times!
      → [Compute Scalar — COUNT(*)]
      → [Filter — COUNT < 3]
  → [SELECT]
Estimated Cost: ~850  |  Logical Reads: ~450,000
Catastrophic at scale!
```

### Failure Modes

**Ties produce inconsistent results:** ROW_NUMBER assigns unique ranks even for equal values. If two products have the same UnitPrice, one gets rank 3 and the other rank 4. For N=3, one is included and the other excluded arbitrarily. Use RANK() or DENSE_RANK() if ties should be included.

**Missing covering index:** Without a covering index on (group, order DESC), SQL Server must scan the clustered index and sort. For a 10M row table, this Sort can spill to tempdb.

**CROSS APPLY with no Categories table:** When there is no Groups/Categories table, the distinct list of groups must come from a `SELECT DISTINCT group FROM table`. This DISTINCT query itself requires a scan and sort, adding cost.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Schema context
-- ============================================================
CREATE TABLE dbo.Categories (
    CategoryId   INT           NOT NULL IDENTITY(1,1),
    CategoryName NVARCHAR(100) NOT NULL,
    Description  NVARCHAR(500) NULL,
    CONSTRAINT PK_Categories PRIMARY KEY CLUSTERED (CategoryId)
);

CREATE TABLE dbo.Products (
    ProductId    INT           NOT NULL IDENTITY(1,1),
    ProductName  NVARCHAR(200) NOT NULL,
    CategoryId   INT           NOT NULL,
    UnitPrice    DECIMAL(18,2) NOT NULL,
    StockQty     INT           NOT NULL DEFAULT 0,
    CreatedAt    DATETIME2(0)  NOT NULL DEFAULT SYSUTCDATETIME(),
    Status       VARCHAR(20)   NOT NULL DEFAULT 'Active',
    CONSTRAINT PK_Products PRIMARY KEY CLUSTERED (ProductId),
    CONSTRAINT FK_Products_Categories FOREIGN KEY (CategoryId)
        REFERENCES dbo.Categories(CategoryId)
);

-- Covering index for ROW_NUMBER Top-N per Category
CREATE INDEX IX_Products_CategoryId_UnitPrice
    ON dbo.Products (CategoryId, UnitPrice DESC)
    INCLUDE (ProductName, StockQty, Status);

-- Index for per-group seeks (supports CROSS APPLY)
CREATE INDEX IX_Products_CategoryId
    ON dbo.Products (CategoryId)
    INCLUDE (ProductName, UnitPrice, StockQty);

-- ============================================================
-- Pattern 1: Top-N per Category — ROW_NUMBER (general case)
-- ============================================================
-- Get top 3 most expensive products per category
DECLARE @TopN INT = 3;

WITH ranked AS (
    SELECT
        p.ProductId,
        p.ProductName,
        p.CategoryId,
        c.CategoryName,
        p.UnitPrice,
        p.StockQty,
        ROW_NUMBER() OVER (
            PARTITION BY p.CategoryId
            ORDER BY p.UnitPrice DESC
        ) AS rn
    FROM dbo.Products p
    INNER JOIN dbo.Categories c ON p.CategoryId = c.CategoryId
    WHERE p.Status = 'Active'
)
SELECT
    r.CategoryId,
    r.CategoryName,
    r.ProductId,
    r.ProductName,
    r.UnitPrice,
    r.StockQty,
    r.rn AS Rank
FROM ranked r
WHERE r.rn <= @TopN
ORDER BY r.CategoryName, r.rn;

-- ============================================================
-- Pattern 2: Top-N per Category — CROSS APPLY
-- ============================================================
-- Best when Categories table is small (< 100) and Products has an index
SELECT
    c.CategoryId,
    c.CategoryName,
    p.ProductId,
    p.ProductName,
    p.UnitPrice,
    p.StockQty
FROM dbo.Categories c
CROSS APPLY (
    SELECT TOP 3
        p.ProductId, p.ProductName, p.UnitPrice, p.StockQty
    FROM dbo.Products p
    WHERE p.CategoryId = c.CategoryId
      AND p.Status = 'Active'
    ORDER BY p.UnitPrice DESC, p.ProductId  -- Tie-breaker
) p
ORDER BY c.CategoryName, p.UnitPrice DESC;

-- ============================================================
-- Pattern 3: Top-N with ties (use RANK instead of ROW_NUMBER)
-- ============================================================
-- If 3 products tie for 2nd place, all 3 are included (total 5)
WITH ranked AS (
    SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice,
        RANK() OVER (PARTITION BY p.CategoryId ORDER BY p.UnitPrice DESC) AS rnk
    FROM dbo.Products p
    WHERE p.Status = 'Active'
)
SELECT r.ProductId, r.ProductName, r.CategoryId, r.UnitPrice, r.rnk
FROM ranked r
WHERE r.rnk <= 3
ORDER BY r.CategoryId, r.rnk;

-- ============================================================
-- Pattern 4: Bottom-N per Group (ORDER BY ASC)
-- ============================================================
-- Cheapest products per category
WITH ranked AS (
    SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice,
        ROW_NUMBER() OVER (
            PARTITION BY p.CategoryId
            ORDER BY p.UnitPrice ASC  -- ASC for cheapest
        ) AS rn
    FROM dbo.Products p
    WHERE p.Status = 'Active'
)
SELECT r.ProductId, r.ProductName, r.CategoryId, r.UnitPrice
FROM ranked r
WHERE r.rn <= 3
ORDER BY r.CategoryId, r.UnitPrice;

-- ============================================================
-- Pattern 5: Top-N by aggregated value (subquery + window)
-- ============================================================
-- Top 3 categories by total revenue
WITH category_revenue AS (
    SELECT
        p.CategoryId,
        c.CategoryName,
        SUM(oi.Quantity * oi.UnitPrice) AS TotalRevenue,
        COUNT(DISTINCT o.OrderId) AS OrderCount
    FROM dbo.Orders o
    INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
    INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
    INNER JOIN dbo.Categories c ON p.CategoryId = c.CategoryId
    WHERE o.OrderDate >= DATEADD(month, -6, GETUTCDATE())
    GROUP BY p.CategoryId, c.CategoryName
),
ranked AS (
    SELECT CategoryId, CategoryName, TotalRevenue, OrderCount,
        ROW_NUMBER() OVER (ORDER BY TotalRevenue DESC) AS rn
    FROM category_revenue
)
SELECT CategoryId, CategoryName, TotalRevenue, OrderCount
FROM ranked
WHERE rn <= 3
ORDER BY rn;

-- ============================================================
-- Pattern 6: Top-N with pagination per group
-- ============================================================
-- Get products ranked 4-6 per category (pages of 3)
WITH ranked AS (
    SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice,
        ROW_NUMBER() OVER (
            PARTITION BY p.CategoryId
            ORDER BY p.UnitPrice DESC
        ) AS rn
    FROM dbo.Products p
)
SELECT r.ProductId, r.ProductName, r.CategoryId, r.UnitPrice
FROM ranked r
WHERE r.rn BETWEEN 4 AND 6
ORDER BY r.CategoryId, r.rn;

-- ============================================================
-- Pattern 7: Top-N with multiple sort criteria per group
-- ============================================================
-- Top 3 products by price, with recency as tie-breaker
WITH ranked AS (
    SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice, p.CreatedAt,
        ROW_NUMBER() OVER (
            PARTITION BY p.CategoryId
            ORDER BY p.UnitPrice DESC, p.CreatedAt DESC  -- Multi-criteria
        ) AS rn
    FROM dbo.Products p
)
SELECT r.ProductId, r.ProductName, r.CategoryId, r.UnitPrice, r.CreatedAt
FROM ranked r
WHERE r.rn <= 3
ORDER BY r.CategoryId, r.rn;

-- ============================================================
-- Pattern 8: Top-N per dynamic group (derived group column)
-- ============================================================
-- Top 3 orders per month (grouped by derived year_month)
WITH monthly_orders AS (
    SELECT
        o.OrderId,
        o.CustomerId,
        o.TotalAmount,
        o.OrderDate,
        YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate) AS year_month,
        ROW_NUMBER() OVER (
            PARTITION BY YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate)
            ORDER BY o.TotalAmount DESC
        ) AS rn
    FROM dbo.Orders o
    WHERE o.OrderDate >= '2024-01-01'
)
SELECT m.year_month, m.OrderId, m.CustomerId, m.TotalAmount, m.OrderDate
FROM monthly_orders m
WHERE m.rn <= 3
ORDER BY m.year_month DESC, m.rn;

-- ============================================================
-- Pattern 9: Delete duplicates per group (keep top 1)
-- ============================================================
-- Keep only the most recent order per customer
WITH ranked AS (
    SELECT o.OrderId, o.CustomerId, o.OrderDate,
        ROW_NUMBER() OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate DESC
        ) AS rn
    FROM dbo.Orders o
)
DELETE FROM dbo.Orders
WHERE OrderId IN (
    SELECT OrderId FROM ranked WHERE rn > 1
);

-- ============================================================
-- Pattern 10: Top-N with Dapper-friendly pagination
-- ============================================================
-- Parameterised Top-N per category
CREATE PROCEDURE dbo.GetTopProductsByCategory
    @TopN INT = 3,
    @CategoryId INT = NULL  -- NULL = all categories
AS
BEGIN
    SET NOCOUNT ON;

    WITH ranked AS (
        SELECT
            p.ProductId, p.ProductName, p.CategoryId,
            c.CategoryName, p.UnitPrice, p.StockQty,
            ROW_NUMBER() OVER (
                PARTITION BY p.CategoryId
                ORDER BY p.UnitPrice DESC
            ) AS rn
        FROM dbo.Products p
        INNER JOIN dbo.Categories c ON p.CategoryId = c.CategoryId
        WHERE (p.CategoryId = @CategoryId OR @CategoryId IS NULL)
          AND p.Status = 'Active'
    )
    SELECT CategoryId, CategoryName, ProductId, ProductName, UnitPrice, StockQty, rn AS Rank
    FROM ranked
    WHERE rn <= @TopN
    ORDER BY CategoryName, rn;
END;
```

```csharp
public class TopNService : ITopNService
{
    private readonly ApplicationDbContext _dbContext;

    public TopNService(ApplicationDbContext dbContext) => _dbContext = dbContext;

    public async Task<List<ProductRank>> GetTopProductsByCategoryAsync(
        int topN = 3, CancellationToken ct = default)
    {
        var sql = @"
            WITH ranked AS (
                SELECT
                    p.CategoryId, c.CategoryName, p.ProductId, p.ProductName,
                    p.UnitPrice, p.StockQty,
                    ROW_NUMBER() OVER (
                        PARTITION BY p.CategoryId
                        ORDER BY p.UnitPrice DESC
                    ) AS rn
                FROM dbo.Products p
                INNER JOIN dbo.Categories c ON p.CategoryId = c.CategoryId
                WHERE p.Status = 'Active'
            )

                   UnitPrice, StockQty, rn AS [Rank]

            WHERE rn <= @TopN
            ORDER BY CategoryName, rn";

        return await _dbContext.Database
            .SqlQueryRaw<ProductRank>(sql, new SqlParameter("@TopN", topN))
            .ToListAsync(ct);
    }

    public async Task<List<ProductRank>> GetTopProductsForCategoryAsync(
        int categoryId, int topN = 3, CancellationToken ct = default)
    {
        var sql = @"
            SELECT TOP (@TopN)
                p.CategoryId, c.CategoryName, p.ProductId, p.ProductName,
                p.UnitPrice, p.StockQty, ROW_NUMBER() OVER (ORDER BY p.UnitPrice DESC) AS [Rank]
            FROM dbo.Products p
            INNER JOIN dbo.Categories c ON p.CategoryId = c.CategoryId
            WHERE p.CategoryId = @CategoryId AND p.Status = 'Active'
            ORDER BY p.UnitPrice DESC";

        return await _dbContext.Database
            .SqlQueryRaw<ProductRank>(sql,
                new SqlParameter("@CategoryId", categoryId),
                new SqlParameter("@TopN", topN))
            .ToListAsync(ct);
    }

    {
        var sql = @"
            WITH category_revenue AS (
                SELECT
                    p.CategoryId, c.CategoryName,
                    SUM(oi.Quantity * oi.UnitPrice) AS TotalRevenue,
                    COUNT(DISTINCT o.OrderId) AS OrderCount
                FROM dbo.Orders o
                INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
                INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
                INNER JOIN dbo.Categories c ON p.CategoryId = c.CategoryId
                WHERE o.OrderDate >= DATEADD(month, -6, GETUTCDATE())
                GROUP BY p.CategoryId, c.CategoryName
            ),
            ranked AS (
                SELECT CategoryId, CategoryName, TotalRevenue, OrderCount,
                    ROW_NUMBER() OVER (ORDER BY TotalRevenue DESC) AS rn
                FROM category_revenue
            )
            SELECT CategoryId, CategoryName, TotalRevenue, OrderCount, rn AS [Rank]
            FROM ranked
            WHERE rn <= @TopN
            ORDER BY rn";

        return await _dbContext.Database
            .SqlQueryRaw<CategoryRevenue>(sql, new SqlParameter("@TopN", topN))
            .ToListAsync(ct);
    }

    public async Task DeleteDuplicateOrdersAsync(CancellationToken ct = default)
    {
        var sql = @"
            WITH ranked AS (
                SELECT o.OrderId, o.CustomerId, o.OrderDate,
                    ROW_NUMBER() OVER (
                        PARTITION BY o.CustomerId
                        ORDER BY o.OrderDate DESC
                    ) AS rn
                FROM dbo.Orders o
            )
            DELETE FROM dbo.Orders
            WHERE OrderId IN (
                SELECT OrderId FROM ranked WHERE rn > 1
            )";
```

## Gotchas and Production Pitfalls

### Tie Handling — ROW_NUMBER Excludes Rows Arbitrarily

**Pitfall:** Using ROW_NUMBER for Top-N when tied values exist. ROW_NUMBER assigns unique sequential numbers even when ORDER BY values are equal. For N=3, the 3rd and 4th rows may have the same value but one is arbitrarily excluded.

```sql
-- ❌ Ties broken arbitrarily — may not match business expectations
WITH ranked AS (
    SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice,
        ROW_NUMBER() OVER (PARTITION BY p.CategoryId ORDER BY p.UnitPrice DESC) AS rn
    FROM dbo.Products p
)
SELECT r.ProductId, r.ProductName, r.UnitPrice
FROM ranked r
WHERE r.rn <= 3;
-- If 4 products have UnitPrice = 99.99, which 3 are selected is undefined
```

**Symptom:** A product category with 4 products at $99.99 shows only 3 of them in the "Top 3" list. The excluded product changes each time the query runs (if no tie-breaker in ORDER BY). Users report "missing" products.

**Fix:**

```sql
-- ✅ Use RANK to include all ties
WITH ranked AS (
    SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice,
        RANK() OVER (PARTITION BY p.CategoryId ORDER BY p.UnitPrice DESC) AS rnk
    FROM dbo.Products p
)
SELECT r.ProductId, r.ProductName, r.UnitPrice
FROM ranked r
WHERE r.rnk <= 3;
-- Returns all products tied for ranks 1-3 (may be more than 3 rows)

-- ✅ Or add a tie-breaker column to ROW_NUMBER
ROW_NUMBER() OVER (PARTITION BY CategoryId ORDER BY UnitPrice DESC, ProductId) AS rn
-- ProductId ensures deterministic ordering
```

**Cost of not fixing:** An e-commerce "Top 3 Best Sellers" widget on the category page shows different products each time it loads. The product team cannot reproduce the bug because it depends on tie values. Customer trust erodes.

---

### Covering Index Order Must Match the OVER Clause

**Pitfall:** Creating an index on (CategoryId, UnitPrice) but the ORDER BY in ROW_NUMBER uses DESC. SQL Server can scan the index backward to handle DESC, but the optimiser may not always choose this plan.

```sql
-- ❌ Index order doesn't match ORDER BY direction
CREATE INDEX IX_Products_CategoryId_UnitPrice
    ON dbo.Products (CategoryId, UnitPrice ASC);  -- ASC default

-- ROW_NUMBER uses DESC — may still work (backward scan) but suboptimal
ROW_NUMBER() OVER (PARTITION BY CategoryId ORDER BY UnitPrice DESC)
```

**Symptom:** The execution plan shows a Sort operator instead of a Segment Top, even though the index seems to cover the query. The Sort adds O(n log n) cost.

**Fix:**

```sql
-- ✅ Create index matching the ORDER BY direction
CREATE INDEX IX_Products_CategoryId_UnitPrice_Desc
    ON dbo.Products (CategoryId, UnitPrice DESC)
    INCLUDE (ProductName, StockQty);
-- Enables Segment Top — no Sort needed
```

**Cost of not fixing:** A Top-5 products per category query runs in 8 seconds on a 1M row Products table instead of 50ms. The missing DESC direction forces a full sort.

---

### CROSS APPLY Without a Groups Table Causes Extra Scan

**Pitfall:** Using CROSS APPLY when there is no Categories/Countries/etc. table. The APPLY needs a driver table of distinct group values — if one doesn't exist, you must SELECT DISTINCT, which adds a full scan.

```sql
-- ❌ No Categories table — must SELECT DISTINCT first
SELECT DISTINCT CategoryId
INTO #Categories
FROM dbo.Products;  -- Extra full scan + sort

SELECT c.CategoryId, p.ProductId, p.ProductName, p.UnitPrice
FROM #Categories c
CROSS APPLY (
    SELECT TOP 3 ProductId, ProductName, UnitPrice
    FROM dbo.Products p
    WHERE p.CategoryId = c.CategoryId
    ORDER BY UnitPrice DESC
) p;
```

**Symptom:** The CROSS APPLY now costs: one scan for DISTINCT + one scan for each group. Performance is worse than ROW_NUMBER.

**Fix:**

```sql
-- ✅ Use ROW_NUMBER instead — only one scan needed
WITH ranked AS (
    SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice,
        ROW_NUMBER() OVER (PARTITION BY p.CategoryId ORDER BY p.UnitPrice DESC) AS rn
    FROM dbo.Products p
)
SELECT r.CategoryId, r.ProductId, r.ProductName, r.UnitPrice
FROM ranked r
WHERE r.rn <= 3;
```

**Cost of not fixing:** A query that runs in 200ms with ROW_NUMBER takes 5 seconds with CROSS APPLY because 500 categories force 500 index seeks instead of one scan.

---

### ROW_NUMBER in WHERE Without CTE or Subquery

**Pitfall:** Trying to use ROW_NUMBER directly in the WHERE clause. Window functions cannot be referenced in WHERE of the same SELECT because they are evaluated after WHERE.

```sql
-- ❌ Invalid — ROW_NUMBER not available in WHERE
SELECT ProductId, ProductName, CategoryId, UnitPrice,
       ROW_NUMBER() OVER (PARTITION BY CategoryId ORDER BY UnitPrice DESC) AS rn
FROM dbo.Products
WHERE rn <= 3;  -- Error: Invalid column name 'rn'
```

**Symptom:** `Msg 207, Level 16, State 1: Invalid column name 'rn'.`

**Fix:**

```sql
-- ✅ Use CTE or derived table
WITH ranked AS (
    SELECT ProductId, ProductName, CategoryId, UnitPrice,
        ROW_NUMBER() OVER (PARTITION BY CategoryId ORDER BY UnitPrice DESC) AS rn
    FROM dbo.Products
)
SELECT ProductId, ProductName, CategoryId, UnitPrice
FROM ranked
WHERE rn <= 3;

-- Or derived table:
SELECT r.ProductId, r.ProductName, r.CategoryId, r.UnitPrice
FROM (
    SELECT ProductId, ProductName, CategoryId, UnitPrice,
        ROW_NUMBER() OVER (PARTITION BY CategoryId ORDER BY UnitPrice DESC) AS rn
    FROM dbo.Products
) r
WHERE r.rn <= 3;
```

**Cost of not fixing:** Developer spends 30 minutes debugging why the query fails, then resorts to a non-sargable workaround that returns wrong results.

---

### Correlated Subquery Performance Catastrophe

**Pitfall:** Using the correlated subquery approach for Top-N on tables larger than 1,000 rows. Each row in the outer query triggers a subquery execution — O(n) index seeks.

```sql
-- ❌ Catastrophic at scale — subquery runs for every row
SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice
FROM dbo.Products p
WHERE (
    SELECT COUNT(*)
    FROM dbo.Products p2
    WHERE p2.CategoryId = p.CategoryId
      AND p2.UnitPrice > p.UnitPrice
) < 3;
```

**Symptom:** On a 50K row Products table, the query runs 50,000 subqueries. Execution time: 8 seconds vs 50ms for ROW_NUMBER. The execution plan shows a Nested Loops Semi Join with 50,000 inner executions.

**Fix:** Use ROW_NUMBER or CROSS APPLY instead.

**Cost of not fixing:** A reporting API endpoint that serves Top-N per category responds in 10 seconds. The page times out. Users refresh repeatedly, increasing server load. The DBA finds 500,000 logical reads per execution.

### Spoken Answers

**Q: What is the Top-N per Group pattern and what SQL approaches can implement it?**

> **Average answer:** It's getting the top N rows per group. You can use ROW_NUMBER, CROSS APPLY, or a correlated subquery.

> **Great answer:** Top-N per Group solves "for each group, give me the N highest (or lowest) rows." There are three main SQL approaches. ROW_NUMBER with a CTE or derived table is the most common and generally the best performer: it scans data once, assigns ranks within each partition, and filters to rn <= N. CROSS APPLY with TOP N is competitive when the number of groups is small (typically < 100) and each group has many rows — it does one index seek per group. The correlated subquery approach (WHERE (SELECT COUNT(*) ...) < N) is catastrophically slow at scale because it executes the subquery for every outer row — I've seen 450,000 logical reads vs 45 for the ROW_NUMBER approach on the same 50K-row table. The execution plan for ROW_NUMBER shows a Segment Top operator (when an index provides sorted input) or a Sort + Window Spool. Segment Top is a non-blocking operator that streams only the top N rows per partition — it's the most efficient possible implementation. The covering index must match the OVER clause order: (PARTITION BY columns, ORDER BY columns DESC) INCLUDE (selected columns). In EF Core, this requires raw SQL — ROW_NUMBER in OVER is not translatable. Dapper handles it natively with direct T-SQL.

---

**Q: When would CROSS APPLY outperform ROW_NUMBER for Top-N per Group?**

> **Great answer:** CROSS APPLY can outperform ROW_NUMBER in two scenarios. First, when the number of groups is very small — say 5 categories with 10,000 products each. CROSS APPLY does 5 index seeks (each returns 3 rows) = 15 rows read. ROW_NUMBER must scan and sort all 50,000 rows even though it only returns 15. The break-even point is typically around 50-100 groups: below that, CROSS APPLY is faster; above that, ROW_NUMBER wins. Second, when a Groups/Categories table already exists as a driver, so there's no extra DISTINCT scan. If you already have `FROM Categories c CROSS APPLY (TOP N ...)`, this is very efficient because Categories is already indexed and the APPLY does per-row seeks. However, if there is no Categories table and you must SELECT DISTINCT to get the groups, ROW_NUMBER is usually better. The key metric: CROSS APPLY cost = O(groups × log N) where N is the table size in each group; ROW_NUMBER cost = O(total rows × log total rows) for the sort. When groups is small and total rows is large, CROSS APPLY wins.

---

**Q: How does Top-N per Group behave at 100M rows with 1M groups?**

> **Great answer:** At this scale, ROW_NUMBER is the only viable approach. CROSS APPLY would need to do 1M index seeks — even at 1ms per seek, that's 1,000 seconds (16 minutes). The correlated subquery would execute 100M subqueries — infeasible. ROW_NUMBER scans the data once (100M rows) and sorts by (group, order columns). The Sort on 100M rows requires ~6 GB of memory if fully in-memory; more likely it spills to tempdb. The covering index is essential: (GroupId, OrderValue DESC) INCLUDE (all other selected columns). With this index, the scan is an ordered Index Scan of ~100M index rows, followed by a Segment Top that streams only the top N per group. If the index is covering, this is about 350 MB of logical reads (at 8KB per page, ~44,000 pages for 100M rows at ~36 bytes per row). The key production concern at this scale is the single Sort if the index doesn't provide the exact order. Partitioning the table by group could help, but the Segment Top operator already handles partition boundaries efficiently. Realistically, at 100M rows with 1M groups, this query should complete in 10-30 seconds with a covering index and adequate memory. For sub-second performance, materialise the Top-N results in a summary table updated by a nightly job.

```mermaid
    D --> I[ROW_NUMBER + CTE]
    I --> J{Index covers OVER clause?}
    J -->|Yes| K[Segment Top — optimal]
    J -->|No| L[Sort + Window Spool — acceptable]
    K --> M[Result: Top-N per group]
    L --> M
    H --> M
    M --> N{Ties possible?}
    N -->|Yes| O[Use RANK or add tie-breaker]
    N -->|No| P[ROW_NUMBER is fine]
```

### Application Checklist

- [ ] The PARTITION BY column(s) in ROW_NUMBER matches the group definition
- [ ] The ORDER BY direction is correct (DESC for top, ASC for bottom)
- [ ] Ties are handled — either by tie-breaker column or by using RANK/DENSE_RANK
- [ ] A covering index exists on (PARTITION BY columns, ORDER BY columns DESC) INCLUDE (selected columns)
- [ ] The index ORDER BY direction matches the ROW_NUMBER ORDER BY
- [ ] The query uses a CTE or derived table (no ROW_NUMBER in WHERE directly)
- [ ] For CROSS APPLY: a Groups table exists and is small (< 100 rows)
- [ ] The correlated subquery approach is not used in production code

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Single-pass processing (ROW_NUMBER)|Sort of all rows (if no covering index)|
|Per-group index seek efficiency (CROSS APPLY)|Requires small group count|
|Correct tie handling (RANK, DENSE_RANK)|RANK may return more than N rows|
|Segment Top — streams only N rows per group|Requires covering index with correct order|

### Scale Thresholds

- **< 1K rows, 10 groups**: Any approach works. ROW_NUMBER is simplest.
- **10K-100K rows, 10-100 groups**: ROW_NUMBER with covering index. Sub-millisecond per row.
- **100K-1M rows, 100-1K groups**: ROW_NUMBER with Segment Top. Sort is avoidable with index.
- **1M-10M rows, 1K-10K groups**: Covering index essential. Query completes in 100ms-2s.
- **> 10M rows, > 10K groups**: Consider materialised Top-N summary table for frequent queries.

### Query Challenges

**Challenge 1 — Write the Top-N per Group query**

Given `Products(ProductId, ProductName, CategoryId, UnitPrice, CreatedAt)` and `Categories(CategoryId, CategoryName)`, write a query that returns the top 3 most recently created products per category. Include the category name and rank. Order by category name then rank.

<details>
<summary>Solution</summary>

```sql
WITH ranked AS (
    SELECT
        p.ProductId, p.ProductName, p.CategoryId,
        c.CategoryName, p.CreatedAt,
        ROW_NUMBER() OVER (
            PARTITION BY p.CategoryId
            ORDER BY p.CreatedAt DESC
        ) AS rn
    FROM dbo.Products p
    INNER JOIN dbo.Categories c ON p.CategoryId = c.CategoryId
    WHERE p.Status = 'Active'
)
SELECT
    r.CategoryId, r.CategoryName,
    r.ProductId, r.ProductName,
    r.CreatedAt, r.rn AS Rank
FROM ranked r
WHERE r.rn <= 3
ORDER BY r.CategoryName, r.rn;
```

**Logical reads:** ~45 (with covering index on CategoryId, CreatedAt DESC)
**Execution plan:** [Index Scan (covering)] → [Segment Top] → [SELECT]
**EF Core equivalent:** Raw SQL via `SqlQueryRaw`.

</details>

**Challenge 2 — Fix the performance problem**

```sql
-- This query returns the top 5 most expensive products per category.
-- On a 100K row Products table with 500 categories, it takes 12 seconds.
SET STATISTICS IO ON;

SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice
FROM dbo.Products p
WHERE (
    SELECT COUNT(*)
    FROM dbo.Products p2
    WHERE p2.CategoryId = p.CategoryId
      AND p2.UnitPrice > p.UnitPrice
) < 5
ORDER BY p.CategoryId, p.UnitPrice DESC;

-- SET STATISTICS IO: Table 'Products'. Scan count 100001, logical reads 1,125,000
```

Identify the problem and fix it.

<details>
<summary>Solution</summary>

**Root cause:** The correlated subquery approach executes the COUNT(*) subquery for every row in the Products table — 100,000 subquery executions. Each subquery does an index seek on (CategoryId, UnitPrice) and counts rows with higher prices. 100,000 index seeks = 1,125,000 logical reads.

**Fix:** Use ROW_NUMBER approach:

```sql
-- Step 1: Create covering index
CREATE INDEX IX_Products_CategoryId_UnitPrice_Desc
    ON dbo.Products (CategoryId, UnitPrice DESC)
    INCLUDE (ProductName);

-- Step 2: Rewrite with ROW_NUMBER
WITH ranked AS (
    SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice,
        ROW_NUMBER() OVER (PARTITION BY p.CategoryId ORDER BY p.UnitPrice DESC) AS rn
    FROM dbo.Products p
)
SELECT r.ProductId, r.ProductName, r.CategoryId, r.UnitPrice
FROM ranked r
WHERE r.rn <= 5
ORDER BY r.CategoryId, r.UnitPrice DESC;
```

**After fix — logical reads:** ~450 (from 1,125,000) — 2,500x reduction. **Runtime:** ~30ms (from 12 seconds).

</details>

**Challenge 3 — Explain the execution plan**

```sql
-- Query A (ROW_NUMBER):
WITH ranked AS (
    SELECT p.ProductId, p.CategoryId, p.UnitPrice,
        ROW_NUMBER() OVER (PARTITION BY p.CategoryId ORDER BY p.UnitPrice DESC) AS rn
    FROM dbo.Products p
)
SELECT r.ProductId, r.CategoryId, r.UnitPrice FROM ranked r WHERE r.rn <= 3;

-- Plan A:
-- [Index Scan (NonClustered) IX_Products_CategoryId_UnitPrice_Desc]
-- → [Segment Top] (2%) → [SELECT]
-- Estimated cost: 1.8, Logical reads: 45

-- Query B (CROSS APPLY):
SELECT c.CategoryId, p.ProductId, p.UnitPrice
FROM (SELECT DISTINCT CategoryId FROM dbo.Products) c
CROSS APPLY (
    SELECT TOP 3 ProductId, UnitPrice
    FROM dbo.Products p
    WHERE p.CategoryId = c.CategoryId
    ORDER BY p.UnitPrice DESC
) p;

-- Plan B:
-- [Index Scan (for DISTINCT)] → [Sort (DISTINCT)] → [Nested Loops (Apply)]
-- → [Index Seek (per category)] → [Top] → [SELECT]
-- Estimated cost: 3.2, Logical reads: 102
```

Why does Plan A use Segment Top while Plan B uses Nested Loops? What would change if the query returned all columns from Products (not just indexed ones)?

<details>
<summary>Solution</summary>

**Why Plan A uses Segment Top:** The covering index `IX_Products_CategoryId_UnitPrice_Desc` provides sorted input in exactly the order required by the OVER clause: CategoryId for PARTITION BY, UnitPrice DESC for ORDER BY. The Segment Top operator reads the ordered index stream and passes through only the top 3 rows per CategoryId, discarding the rest. It never materialises the full 50K rows — it streams only 30 rows (10 categories × 3). This is the most efficient possible approach.

**Why Plan B uses Nested Loops:** CROSS APPLY conceptually iterates over each distinct CategoryId and executes a TOP 3 query. The Nested Loops Apply operator performs one index seek per category. With 10 categories, that's 10 seeks. Each seek returns 3 rows = 30 total.

**What changes if all columns are selected:** The covering index may not include all columns (e.g., Description, StockQty). SQL Server would need to add a Key Lookup to fetch the missing columns for each of the 30 rows returned. With ROW_NUMBER, the Segment Top streams only 30 rows, so only 30 key lookups. Logical reads increase from 45 to ~75 (30 lookups × 1 read each). The plan would show `[Segment Top] → [Key Lookup] → [SELECT]`. The solution is to add the missing columns to the INCLUDE clause of the covering index.

</details>

**Challenge 4 — Diagnose the tie-handling problem**

A product catalogue uses this query to show "Top 3 Products" per category:

```sql
WITH ranked AS (
    SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice,
        ROW_NUMBER() OVER (PARTITION BY p.CategoryId ORDER BY p.UnitPrice DESC) AS rn
    FROM dbo.Products p
)
SELECT r.ProductId, r.ProductName, r.CategoryId, r.UnitPrice
FROM ranked r
WHERE r.rn <= 3
ORDER BY r.CategoryId, r.rn;
```

The product manager reports: "In the Electronics category, there are 4 products priced at $999.99. The widget shows only 3 of them, and the one shown changes every time I refresh." Fix the query.

<details>
<summary>Solution</summary>

**Root cause:** ROW_NUMBER assigns unique ranks 1, 2, 3, 4 to the 4 tied products arbitrarily. Each query execution (or plan recompilation) may assign different ROW_NUMBER values to the same products because there is no tie-breaker in the ORDER BY.

**Fix:**

```sql
-- Option A: Add a tie-breaker column to make ordering deterministic
WITH ranked AS (
    SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice,
        ROW_NUMBER() OVER (
            PARTITION BY p.CategoryId
            ORDER BY p.UnitPrice DESC, p.ProductId  -- ProductId as tie-breaker
        ) AS rn
    FROM dbo.Products p
)
SELECT r.ProductId, r.ProductName, r.CategoryId, r.UnitPrice
FROM ranked r
WHERE r.rn <= 3
ORDER BY r.CategoryId, r.rn;
-- Always returns products with lowest ProductIds in tied groups

-- Option B: Use RANK to include all ties (may show more than 3)
WITH ranked AS (
    SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice,
        RANK() OVER (PARTITION BY p.CategoryId ORDER BY p.UnitPrice DESC) AS rnk
    FROM dbo.Products p
)
SELECT r.ProductId, r.ProductName, r.CategoryId, r.UnitPrice
FROM ranked r
WHERE r.rnk <= 3
ORDER BY r.CategoryId, r.rnk;
-- Returns all 4 products tied at $999.99 (ranks 1, 1, 1, 1 — all <= 3)

-- Option C: Use DENSE_RANK + ROW_NUMBER for "top 3 values, then pick first by ID"
WITH ranked AS (
    SELECT p.ProductId, p.ProductName, p.CategoryId, p.UnitPrice,
        DENSE_RANK() OVER (PARTITION BY p.CategoryId ORDER BY p.UnitPrice DESC) AS dr,
        ROW_NUMBER() OVER (PARTITION BY p.CategoryId ORDER BY p.UnitPrice DESC, p.ProductId) AS rn
    FROM dbo.Products p
)
SELECT r.ProductId, r.ProductName, r.CategoryId, r.UnitPrice
FROM ranked r
WHERE r.dr <= 3  -- All distinct price tiers
   OR r.rn <= 3  -- At least 3 products
ORDER BY r.CategoryId, r.rn;
```

</details>

**Challenge 5 — Design the Top-N infrastructure**

**Scenario:** A news aggregator site shows the top 10 most-read articles per category every hour. The `ArticleViews` table logs every view: `ArticleViews(Id, ArticleId, CategoryId, ViewedAt)`. The table grows by 2M rows per day (20M views/day, 600M/month). The top-10 query must complete in under 500ms. The site has 50 categories and 500K articles.

Design the SQL, index, and materialization strategy.

<details>
<summary>Solution</summary>

**Strategy: Materialised hourly summary + ROW_NUMBER top-10**

```sql
-- Step 1: Hourly materialised summary
CREATE TABLE dbo.HourlyArticleViews (
    ArticleId      INT           NOT NULL,
    CategoryId     INT           NOT NULL,
    HourBucket     DATETIME2(0)  NOT NULL,  -- Rounded to hour
    ViewCount      INT           NOT NULL DEFAULT 0,
    CONSTRAINT PK_HourlyArticleViews PRIMARY KEY (HourBucket, CategoryId, ArticleId)
);

-- Step 2: Incremental update every hour (runs at :05 past each hour)
INSERT INTO dbo.HourlyArticleViews (ArticleId, CategoryId, HourBucket, ViewCount)
SELECT
    av.ArticleId,
    av.CategoryId,
    DATEADD(HOUR, DATEDIFF(HOUR, 0, av.ViewedAt), 0) AS HourBucket,
    COUNT(*) AS ViewCount
FROM dbo.ArticleViews av
WHERE av.ViewedAt >= DATEADD(HOUR, -1, GETUTCDATE())  -- Last hour only
  AND NOT EXISTS (  -- Idempotent: skip already-processed
      SELECT 1 FROM dbo.HourlyArticleViews h
      WHERE h.HourBucket = DATEADD(HOUR, DATEDIFF(HOUR, 0, av.ViewedAt), 0)
        AND h.ArticleId = av.ArticleId
        AND h.CategoryId = av.CategoryId
  )
GROUP BY av.ArticleId, av.CategoryId,
         DATEADD(HOUR, DATEDIFF(HOUR, 0, av.ViewedAt), 0);

-- Step 3: Rolling 24-hour top-10 per category query
WITH rolling_views AS (
    SELECT
        h.ArticleId,
        h.CategoryId,
        a.Title,
        SUM(h.ViewCount) AS TotalViewsLast24h
    FROM dbo.HourlyArticleViews h
    INNER JOIN dbo.Articles a ON h.ArticleId = a.ArticleId
    WHERE h.HourBucket >= DATEADD(HOUR, -24, GETUTCDATE())
    GROUP BY h.ArticleId, h.CategoryId, a.Title
),
ranked AS (
    SELECT ArticleId, CategoryId, Title, TotalViewsLast24h,
        ROW_NUMBER() OVER (
            PARTITION BY CategoryId
            ORDER BY TotalViewsLast24h DESC
        ) AS rn
    FROM rolling_views
)
SELECT ArticleId, CategoryId, Title, TotalViewsLast24h
FROM ranked
WHERE rn <= 10
ORDER BY CategoryId, rn;
```

**Indexes:**

```sql
-- For the hourly aggregation query:
CREATE INDEX IX_ArticleViews_ViewedAt
    ON dbo.ArticleViews (ViewedAt)
    INCLUDE (ArticleId, CategoryId);

-- For the top-10 query (covering):
CREATE INDEX IX_HourlyArticleViews_CategoryId_ViewCount
    ON dbo.HourlyArticleViews (CategoryId, ViewCount DESC)
    INCLUDE (ArticleId, HourBucket);
```

**Expected performance:**
- Hourly summary insert: processes ~83K views (2M/day / 24), completes in < 5 seconds
- Top-10 query: reads 50 categories × ~500 articles × 24 hours = ~600K rows from HourlyArticleViews, groups by ArticleId, then ranks. With the index, executes in ~200ms
- Rolling window: the 24-hour filter on HourBucket uses the PK for efficient range scan
- Total storage: HourlyArticleViews = 500K articles × 24 hours × ~30 bytes = ~360 MB per day

**Why not query ArticleViews directly:** The raw table has 600M rows/month. GROUP BY on article + category for 24 hours would scan 2M rows and do a Hash Match Aggregate — taking 5-10 seconds. The materialised hourly summary reduces this to 600K rows and enables the covering index.

</details>
