### Where This Fits

Window functions require sorted data. The SQL Server query optimizer inserts a Sort operator whenever the input data is not already ordered by the PARTITION BY + ORDER BY columns. A single query with multiple window functions can generate multiple Sort operators, each sorting the same data independently. This is the single most common performance problem with window function queries. A .NET backend engineer running analytics queries through EF Core or Dapper sees this as slow API responses, high CPU usage, and memory grant errors. The solution involves: (1) understanding when the optimizer can share a single sort across multiple window functions, (2) creating indexes that provide ordered access and eliminate the Sort, (3) using trace flags to enable spool elimination, (4) reducing the number of window functions by combining or rewriting queries. The interview signal is the ability to read an execution plan, identify redundant Sort operators, and describe the fix (usually an index) without looking at the query.

## Core Mental Model

The Sort operator sorts rows by the PARTITION BY + ORDER BY columns before the Window Aggregate computes the window function. If multiple window functions share the same PARTITION BY and ORDER BY, the optimizer can compute them all in a single pass over the same sorted input — one Sort feeds multiple Window Aggregates. If the PARTITION BY or ORDER BY differs, each unique combination requires its own Sort. The invariant: each unique sort requirement (distinct set of PARTITION BY + ORDER BY columns) generates at most one Sort operator, but the optimizer does not always recognize when two window function specifications are logically equivalent (e.g., ORDER BY ASC vs DESC on the same columns, or one with a frame and one without). The key optimization technique is to align all window function specifications to the same PARTITION BY + ORDER BY and to create an index that provides the sort order, eliminating the Sort entirely. When the Sort cannot be eliminated, the memory grant must be sized correctly to avoid a spill to TempDB.

### Classification

- **Optimizer behavior:** The optimizer analyzes window function sort requirements and determines if they can share a sort. It does NOT merge sorts with different ORDER BY directions.
- **Data access path:** Index seek/scan with Ordered:true property eliminates the Sort; scan with Ordered:false requires a Sort.
- **SARGable:** The Sort itself is not a predicate; however, the WHERE clause that reduces rows before the window function is independently SARGable.

## Deep Mechanics

### How the Engine Executes This

**Step-by-step processing of multiple window functions:**

1. **Query parsing:** SQL Server parses the query and identifies each window function specification. For each unique PARTITION BY + ORDER BY combination, it creates a separate sort requirement.
2. **Optimization — sort merging:** The optimizer groups window functions by sort requirement. Window functions with the same PARTITION BY and ORDER BY columns (exact match including direction) are assigned to the same Sort + Window Aggregate operator. The optimizer does NOT merge sorts where ORDER BY ASC vs DESC differs, even if one could be derived from the other.
3. **Index selection:** The optimizer checks for non-clustered indexes that match the sort requirement. An index on (PARTITION BY columns, ORDER BY columns) with the same direction can provide an ordered scan (Index Scan with Ordered:true property), eliminating the Sort operator.
4. **Window Aggregate computation:** On the sorted stream, the Segment operator detects partition boundaries. The Sequence Project (Window Aggregate) computes the window function value for each row.
5. **Frame spool for RANGE:** If the frame uses RANGE BETWEEN (as opposed to ROWS), the optimizer adds a blocking Spool operator that stores all rows of the partition before computing the window function, because RANGE requires knowing the total set of peer rows.

### Multiple Window Functions — Sort Sharing

```sql
-- Query with 3 window functions, all sharing the same PARTITION BY + ORDER BY
-- The optimizer produces ONE Sort for all three
SELECT
    OrderId,
    CustomerId,
    OrderDate,
    TotalAmount,
    ROW_NUMBER() OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate DESC, OrderId DESC
    ) AS RowNum,
    SUM(TotalAmount) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate DESC, OrderId DESC
        ROWS UNBOUNDED PRECEDING
    ) AS RunningTotal,
    COUNT(*) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate DESC, OrderId DESC
        ROWS UNBOUNDED PRECEDING
    ) AS OrderCount
FROM Orders;
```

Expected plan (indexed):
```
[Index Scan (IX_Orders_CustomerId_OrderDate)]
  → [Segment (partition on CustomerId)]
    → [Sequence Project (ROW_NUMBER, SUM OVER, COUNT OVER)]
      → [SELECT]
```
No Sort operator — the index provides the required order.

Expected plan (unindexed):
```
[Clustered Index Scan]
  → [Sort (ORDER BY CustomerId, OrderDate DESC, OrderId DESC)]
    → [Segment]
      → [Sequence Project]
        → [SELECT]
```
One Sort for all three window functions.

### SQL Visibility — The Problem

```sql
-- PROBLEM: Two different ORDER BY specifications → TWO SORTS
SELECT
    OrderId,
    CustomerId,
    OrderDate,
    TotalAmount,
    -- Sort 1: CustomerId, OrderDate ASC
    ROW_NUMBER() OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate ASC, OrderId ASC
    ) AS RowNumAsc,
    -- Sort 2: CustomerId, TotalAmount DESC (different ORDER BY)
    RANK() OVER(
        PARTITION BY CustomerId
        ORDER BY TotalAmount DESC
    ) AS AmountRank
FROM Orders;
```

Expected plan:
```
[Clustered Index Scan]
  → [Sort 1: CustomerId, OrderDate ASC, OrderId ASC]
    → [Segment + Sequence Project: ROW_NUMBER]
      → [Sort 2: CustomerId, TotalAmount DESC]
        → [Segment + Sequence Project: RANK]
          → [SELECT]
```

Two Sort operators. The second Sort re-sorts the already-sorted-by-CustomerId stream because the ORDER BY columns differ (OrderDate ASC vs TotalAmount DESC).

### Execution Plan Analysis

**Plan with Sort operator (unindexed):**

```
|--Sort (ORDER BY: CustomerId, OrderDate DESC, OrderId DESC)
    |--Clustered Index Scan (Table: Orders)
        Logical Reads: 12,345  |  Estimated I/O Cost: 5.4
        Operator Cost: 70% of batch
```

**Plan with Sort eliminated (indexed):**

```
|--Index Scan (IX_Orders_CustomerId_OrderDate, ORDERED FORWARD)
    |--Clustered Index Scan (Table: Orders, Key Lookup if not covering)
        Logical Reads: 234  |  Estimated I/O Cost: 0.8
        Operator Cost: 30% of batch
```

The Sort operator's cost is typically 20-40% of the total query cost. Eliminating it can cut total query time in half.

### Failure Modes

**Failure 1: Non-covering index causes Key Lookup explosion.**

```sql
-- If index on (CustomerId, OrderDate) does not INCLUDE TotalAmount
-- The optimizer may choose: Index Seek + Key Lookup (expensive) or Clustered Index Scan + Sort
-- For large row counts, Scan + Sort is often cheaper than 10K Key Lookups
```

**Failure 2: ASC vs DESC mismatch prevents sort sharing.**

```sql
CREATE INDEX IX_Orders_CustomerId_OrderDate
    ON Orders(CustomerId, OrderDate ASC);  -- ASC index

-- Window function uses DESC:
ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC)
-- The Sort operator still appears because the index provides ASC but the window needs DESC
-- Fix: create index with same direction as the window function
CREATE INDEX IX_Orders_CustomerId_OrderDate_Desc
    ON Orders(CustomerId, OrderDate DESC);
```

**Failure 3: RANGE BETWEEN forces a blocking Spool.**

```sql
-- RANGE BETWEEN + UNBOUNDED PRECEDING = Spool + Sort
-- ROWS BETWEEN would use running window without spool
SUM(TotalAmount) OVER(
    PARTITION BY CustomerId
    ORDER BY OrderDate
    RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
)
-- vs:
SUM(TotalAmount) OVER(
    PARTITION BY CustomerId
    ORDER BY OrderDate
    ROWS UNBOUNDED PRECEDING
)
```
RANGE requires knowing all peer rows (rows with same ORDER BY value) before computing, which adds a blocking Spool. ROWS does not — it computes for each row incrementally.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- Schema for optimization examples
CREATE TABLE #Orders (
    OrderId INT IDENTITY(1,1) PRIMARY KEY,
    CustomerId INT NOT NULL,
    OrderDate DATETIME2 NOT NULL,
    TotalAmount DECIMAL(10,2) NOT NULL,
    Status VARCHAR(20) NOT NULL DEFAULT 'Pending'
);

-- Scenario 1: Multiple window functions with the same ORDER BY
-- Optimizer shares one sort — but only if ORDER BY is identical

-- ✅ GOOD: Same ORDER BY for all three → ONE sort
SELECT
    OrderId,
    CustomerId,
    OrderDate,
    TotalAmount,
    ROW_NUMBER() OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate DESC, OrderId DESC
    ) AS RowNum,
    SUM(TotalAmount) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate DESC, OrderId DESC
        ROWS UNBOUNDED PRECEDING
    ) AS RunningTotal,
    AVG(TotalAmount) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate DESC, OrderId DESC
        ROWS BETWEEN 3 PRECEDING AND 1 PRECEDING
    ) AS MovingAvgExcludingCurrent
FROM #Orders;

-- ❌ BAD: Different ORDER BY → TWO sorts (or more)
SELECT
    OrderId,
    CustomerId,
    OrderDate,
    TotalAmount,
    ROW_NUMBER() OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate DESC  -- Sort 1
    ) AS RowNum,
    RANK() OVER(
        PARTITION BY CustomerId
        ORDER BY TotalAmount DESC  -- Sort 2
    ) AS AmountRank,
    DENSE_RANK() OVER(
        ORDER BY TotalAmount DESC  -- Sort 3 (no PARTITION BY = all rows is one partition)
    ) AS OverallRank
FROM #Orders;

-- Scenario 2: Eliminate Sort with a covering index
-- Step 1: Without index
SELECT OrderId, CustomerId, OrderDate, TotalAmount,
    ROW_NUMBER() OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate DESC, OrderId DESC
    ) AS RowNum,
    SUM(TotalAmount) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate DESC, OrderId DESC
        ROWS UNBOUNDED PRECEDING
    ) AS RunningTotal
FROM #Orders;
-- Logical reads: Full scan  |  Sort operator present

-- Step 2: Create matching index
CREATE INDEX IX_Orders_CustomerId_OrderDate_Desc
    ON #Orders(CustomerId, OrderDate DESC, OrderId DESC)
    INCLUDE (TotalAmount);

-- Step 3: After index — Sort eliminated, ordered scan used
SELECT OrderId, CustomerId, OrderDate, TotalAmount,
    ROW_NUMBER() OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate DESC, OrderId DESC
    ) AS RowNum,
    SUM(TotalAmount) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate DESC, OrderId DESC
        ROWS UNBOUNDED PRECEDING
    ) AS RunningTotal
FROM #Orders;
-- Logical reads: Non-clustered index scan ~10% of full scan  | No Sort

-- Scenario 3: ROWS vs RANGE impact on Spool
-- ROWS: no spool (running window)
SELECT CustomerId, OrderDate, TotalAmount,
    SUM(TotalAmount) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate
        ROWS UNBOUNDED PRECEDING
    ) AS RunningTotal
FROM #Orders;

-- RANGE: blocking spool (needs to identify peer rows)
SELECT CustomerId, OrderDate, TotalAmount,
    SUM(TotalAmount) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate
        RANGE UNBOUNDED PRECEDING
    ) AS RunningTotal
FROM #Orders;
-- The RANGE version shows a Window Spool operator in the plan

DROP TABLE #Orders;
```

### Configuration — Index Design for Window Functions

```sql
-- Rule of thumb for window function index design:
-- 1. PARTITION BY columns go FIRST in index key
-- 2. ORDER BY columns go NEXT — same direction as the window function
-- 3. INCLUDE all other columns referenced in the SELECT (value columns, frame columns)

-- Index for: ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC, OrderId DESC)
-- Plus: SUM(TotalAmount) OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC, OrderId DESC)
CREATE INDEX IX_Orders_WindowFunction_Covering
    ON Orders(CustomerId, OrderDate DESC, OrderId DESC)
    INCLUDE (TotalAmount);

-- Index for: RANK() OVER(PARTITION BY CustomerId ORDER BY TotalAmount DESC)
CREATE INDEX IX_Orders_CustomerId_TotalAmount_Desc
    ON Orders(CustomerId, TotalAmount DESC)
    INCLUDE (OrderId, OrderDate);

-- Index for: PARTITION BY CategoryId, ORDER BY SalesAmount DESC
CREATE INDEX IX_Products_CategoryId_SalesAmount_Desc
    ON Products(CategoryId, SalesAmount DESC)
    INCLUDE (ProductId, ProductName);
```

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL: Sort elimination works the same way (index matching ORDER BY)
CREATE INDEX ON Orders(CustomerId, OrderDate DESC, OrderId DESC);

-- PostgreSQL: ROWS vs RANGE behavior is the same
-- ROWS = running window, RANGE = peer-based window (no spool but slower)

-- PostgreSQL: Can use WINDOW clause to define reusable window
-- This helps avoid redundant sorts by declaring the window once
SELECT
    OrderId,
    CustomerId,
    OrderDate,
    TotalAmount,
    ROW_NUMBER() OVER w AS RowNum,
    SUM(TotalAmount) OVER w AS RunningTotal,
    AVG(TotalAmount) OVER w AS MovingAvg
FROM Orders
WINDOW w AS (
    PARTITION BY CustomerId
    ORDER BY OrderDate DESC, OrderId DESC
    ROWS UNBOUNDED PRECEDING
);

-- No equivalent WINDOW clause in SQL Server — must repeat OVER clause

-- PostgreSQL: No trace flag 9358 equivalent
-- Instead, avoid RANGE frame if not needed (use ROWS)
```

## Gotchas and Production Pitfalls

### Pitfall 1: ASC vs DESC Mismatch with Index

**Pitfall:** Creating an index with ASC order but the window function uses DESC, or vice versa. The Sort operator still appears.

```sql
-- Index created:
CREATE INDEX IX_Orders_CustomerId_OrderDate
    ON Orders(CustomerId, OrderDate ASC);

-- Window function uses DESC:
ROW_NUMBER() OVER(
    PARTITION BY CustomerId
    ORDER BY OrderDate DESC  -- Does NOT match index order
)
```

**Symptom:** Execution plan shows a Sort operator even though there is an index on the exact columns. The optimizer does not reverse the scan direction for window functions in all cases — it requires the exact order.

**Fix:** Create an index matching the window function's ORDER BY direction:

```sql
CREATE INDEX IX_Orders_CustomerId_OrderDate_Desc
    ON Orders(CustomerId, OrderDate DESC);
```

**Cost of not fixing:** Full sort of the data, consuming memory and CPU. For 10M rows, this adds 2-5 seconds per query.

### Pitfall 2: RANGE Frame Forces Blocking Spool

**Pitfall:** Using RANGE BETWEEN instead of ROWS BETWEEN, causing a blocking Spool operator that prevents sort sharing.

```sql
-- ❌ RANGE BETWEEN adds a Window Spool (blocking operator)
SUM(TotalAmount) OVER(
    PARTITION BY CustomerId
    ORDER BY OrderDate
    RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
) AS RunningTotal
```

**Symptom:** The execution plan shows a Window Spool operator between the Sort and the Window Aggregate. The spool reads all rows into a worktable before computing the aggregate, doubling memory usage and adding I/O.

**Fix:** Use ROWS BETWEEN unless RANGE semantics (peer rows) are specifically required:

```sql
-- ✅ ROWS BETWEEN eliminates the spool (running window)
SUM(TotalAmount) OVER(
    PARTITION BY CustomerId
    ORDER BY OrderDate
    ROWS UNBOUNDED PRECEDING
) AS RunningTotal
```

**Cost of not fixing:** Each RANGE window adds a blocking spool. For queries with 3-4 RANGE windows, the spool writes millions of rows to TempDB, causing 10-50x slowdown.

### Pitfall 3: Non-Covering Index Causes Key Lookup That Is More Expensive Than Sort

**Pitfall:** Creating a narrow index that matches the ORDER BY but does not INCLUDE all columns. The optimizer may choose Key Lookup + Sort instead of Scan + Sort, and Key Lookup may be more expensive.

```sql
-- Index does not INCLUDE TotalAmount:
CREATE INDEX IX_Orders_CustomerId_OrderDate
    ON Orders(CustomerId, OrderDate DESC)
    INCLUDE (OrderId);  -- Missing TotalAmount!

-- Query references TotalAmount:
SELECT OrderId, CustomerId, OrderDate, TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC) AS RowNum
FROM Orders;
```

**Symptom:** Execution plan shows Key Lookup (Clustered) for each row — potentially millions of lookups. The optimizer may choose Clustered Index Scan + Sort instead, which is slower than Index Seek + Key Lookup for small result sets but faster for large ones.

**Fix:** INCLUDE all referenced columns in the index:

```sql
CREATE INDEX IX_Orders_CustomerId_OrderDate_Covering
    ON Orders(CustomerId, OrderDate DESC)
    INCLUDE (OrderId, TotalAmount);  -- All SELECT columns
```

**Cost of not fixing:** Suboptimal plan choice. At 10M rows, Key Lookups cause 10M random I/Os, which is 5-10x slower than sequential scan.

### Pitfall 4: Same Logical Sort, Different Column Aliases

**Pitfall:** The optimizer treats two ORDER BY specifications as different if they use different expressions, even if they produce the same sort order.

```sql
-- ❌ These look the same to the developer but generate TWO sorts:
ROW_NUMBER() OVER(
    PARTITION BY CustomerId
    ORDER BY OrderDate ASC, OrderId ASC
) AS RowNum,
SUM(TotalAmount) OVER(
    PARTITION BY CustomerId
    ORDER BY OrderDate ASC, OrderId ASC
    ROWS UNBOUNDED PRECEDING
) AS RunningTotal
```

Actually, these DO share the same sort — the optimizer merges them. The failure occurs when:
- One uses `OrderDate ASC, OrderId ASC` and another uses `OrderDate, OrderId` (missing direction — but both default to ASC, so this is fine).
- One uses a computed column: `ORDER BY YEAR(OrderDate)` vs `ORDER BY OrderDate` (different expressions, different sort keys).

**Symptom:** Execution plan shows multiple Sort operators when the developer expected one.

**Fix:** Ensure the ORDER BY clause is identical for all window functions that should share the sort. Use column references, not expressions.

**Cost of not fixing:** Double the memory and CPU for sorting the same data.

### Pitfall 5: Memory Grant Underestimation for Sort

**Pitfall:** The optimizer underestimates the number of rows flowing into the Sort, causing a memory grant too small for an in-memory sort. The sort spills to TempDB.

```sql
-- Optimizer estimates 100K rows (based on outdated statistics)
-- Actual row count: 5M rows
-- Memory grant: 256MB (sufficient for 100K, insufficient for 5M)
-- Sort spills to TempDB → 10x slower
```

**Symptom:** Execution plan shows "Sort Warnings" with spill level 1, 2, or 3. Query runs 5-10x longer than expected. TempDB data file grows.

**Detection:**
```sql
SELECT
    qs.sql_handle,
    qs.statement_start_offset,
    qs.statement_end_offset,
    qs.sort_warning_count
FROM sys.dm_exec_query_stats qs
WHERE qs.sort_warning_count > 0;
```

**Fix:**
```sql
-- Update statistics for accurate row estimation
UPDATE STATISTICS Orders WITH FULLSCAN;

-- Or provide a query hint for larger memory grant
SELECT ...
FROM Orders
OPTION (MIN_GRANT_PERCENT = 10);

-- Or use MAX_GRANT_PERCENT to cap memory (prevent excessive grant)
SELECT ...
FROM Orders
OPTION (MAX_GRANT_PERCENT = 20);
```

**Cost of not fixing:** Sort spills write to TempDB, consuming disk I/O. Spill level 1 doubles query time; level 3 quadruples it. For concurrent queries, TempDB contention causes additional waits (PAGELATCH_EX on TempDB allocation pages).

### Pitfall 6: PARTITION BY on High-Cardinality Column Causes Many Small Sorts

**Pitfall:** PARTITION BY on a column with very high cardinality (e.g., OrderId, TransactionId) creates as many partitions as there are distinct values. Each partition triggers a sort reset.

```sql
-- ❌ PARTITION BY OrderId (each partition has 1 row — wasteful)
ROW_NUMBER() OVER(
    PARTITION BY OrderId
    ORDER BY OrderItemId
) AS ItemNum
```

**Symptom:** The Segment operator fires for every row, and the Sort operator has no work to do (each partition is 1 row), but the overhead of partition switching for every row still costs CPU.

**Fix:** Remove the PARTITION BY if each value is unique. Use simple ORDER BY:

```sql
-- ✅ No PARTITION BY needed for unique OrderId
ROW_NUMBER() OVER(
    ORDER BY OrderItemId
) AS ItemNum
```

**Cost of not fixing:** Higher CPU for partition detection. With 10M partitions (one per row), the Segment operator overhead adds 100-200ms.

### Spoken Answers

**Q1: Why do window functions require sorted data? How does the database engine get the data in order?**

> **Average answer:** "Window functions need sorted data because they process rows in order within a partition. The database adds a Sort operator if the data is not already sorted."

> **Great answer:** "Window functions compute values based on the position of each row relative to its partition and the ordering within that partition. ROW_NUMBER assigns 1 to the first row, 2 to the second — this requires a defined order. The database engine gets data sorted in two ways: either it reads from an index that already provides the required order (the Index Scan operator shows 'Ordered: true' in the plan), or it inserts a Sort operator that sorts the data before the Window Aggregate. The Sort is a blocking operator — it must consume all input rows before emitting the first row. This is typically the single most expensive operator in a window function query, costing 20-40% of total query execution. The optimizer decides whether to sort or use an index based on cost estimation: if there is no ordered index, it always sorts. If there is an ordered index, it evaluates whether the index is covering or if key lookups make it cheaper to scan + sort instead. The key to performance is creating a covering, ordered index that matches the exact PARTITION BY + ORDER BY of the window function, which sets Ordered: true on the Index Scan and eliminates the Sort entirely."

**Q4: How do you eliminate the Sort operator for a window function query?**

> **Average answer:** "Create an index on the partition and order columns. That way SQL Server can read the data in order without sorting."

> **Great answer:** "To eliminate the Sort operator, you need an index that provides the data in exactly the order required by the window function's PARTITION BY and ORDER BY clauses. The index key columns must be: first, the PARTITION BY columns; second, the ORDER BY columns in the same direction (ASC or DESC). The index must also be covering: any column referenced in the SELECT (value columns, frame columns) must be included in the index as INCLUDE columns to avoid Key Lookups. If the window function is 'ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC, OrderId DESC)', the index should be 'CREATE INDEX IX_Window ON Orders(CustomerId, OrderDate DESC, OrderId DESC) INCLUDE (TotalAmount, Status)'. With this index, the execution plan shows an Index Scan with Ordered: true, and the Sort operator is gone. If there are multiple window functions with different ORDER BY specifications, you need multiple indexes — one per unique sort requirement. The tradeoff is write overhead: each index adds ~3-5ms per INSERT and ~5-10ms per UPDATE on the indexed columns. For a table with 100K writes/day and a window function query running 1000x/day, the index is worth it — the read benefit (eliminating a 100ms Sort) far outweighs the write cost."

**Q7: How does ASC vs DESC in the ORDER BY affect sort sharing?**

> **Average answer:** "The optimizer treats ASC and DESC as different sort orders. They don't share a sort."

> **Great answer:** "The optimizer performs exact matching on the ORDER BY clause, including the direction of each column. A window function with ORDER BY OrderDate ASC and another with ORDER BY OrderDate DESC have different sort requirements, so they get separate Sort operators. SQL Server does not reverse an ASC scan to serve a DESC requirement for window functions in all cases — unlike a simple ORDER BY with an index, where the optimizer can do a backward scan. This is a known optimizer limitation. The fix is twofold: first, align all window functions to use the same ORDER BY direction where possible. For example, if one window needs ascending and another needs descending, consider whether you can derive the descending value from the ascending rank (by subtracting from the total count). Second, if you cannot align them, create an index that matches the less common direction. For the common direction, the optimizer can use the index. For the uncommon one, it must sort, but the sort operates on the filtered/restricted data. The performance hit is the cost of the minority sort, not the majority."

<details>
<summary>Answers</summary>

1. **One Sort operator.** The optimizer groups window functions by sort requirement. If all three have the same PARTITION BY and ORDER BY, they share one Sort (or zero if an ordered index exists).

2. **Ordered: true.** The Index Scan (or Index Seek) operator shows Ordered: true when the index provides rows in the required order for the window function. Ordered: false means the optimizer must add a Sort.

3. **ROWS** uses a running window — it adds a Segment and Sequence Project but no spool. **RANGE** requires a blocking Window Spool because it must identify all peer rows (rows with the same ORDER BY value) before computing the window function.

4. **Sort Warnings** appear in the XML execution plan or in the execution plan tooltip. The warning shows spill level (1, 2, or 3). Level 1 writes to TempDB once; each subsequent level roughly doubles the I/O. Also check `sys.dm_exec_query_stats.sort_warning_count`.

5. **No.** The optimizer requires exact match of ORDER BY directions for sort sharing. ASC and DESC are considered different sort requirements. However, if the index provides one direction, the optimizer may use it if you reverse the ORDER BY in the OVER clause — but this changes the result.

6. **Index key:** PARTITION BY columns first, then ORDER BY columns with the same direction. **INCLUDE:** all other columns referenced in the SELECT. This makes the index covering and ordered, eliminating both the Sort and any Key Lookups.

7. **Trace Flag 9358** (SQL Server 2019+, compat level 150+) eliminates the blocking Segment Spool that RANGE BETWEEN introduces. Use with `OPTION (QUERYTRACEON 9358)`.

8. **The optimizer may choose Key Lookup + Index Ordered Scan** (non-covering index) vs **Clustered Index Scan + Sort** (no index). The optimizer picks the cheaper plan based on row count estimates. For row counts where Key Lookup is more expensive than Scan + Sort, the Sort remains despite the ordered index.

**Challenge 5 — Design the index strategy**

**Scenario:** A real-time dashboard queries the Orders table (20M rows, 1M new rows/month) with these window function queries:

1. `ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC, OrderId DESC) AS RowNum` — paginated order list
2. `SUM(TotalAmount) OVER(PARTITION BY CustomerId ORDER BY OrderDate ASC, OrderId ASC ROWS UNBOUNDED PRECEDING) AS RunningTotal` — customer spending timeline
3. `LAG(TotalAmount) OVER(PARTITION BY CustomerId ORDER BY OrderDate ASC, OrderId ASC) AS PreviousAmount` — order comparison

The dashboard is accessed by 200 concurrent users, each querying their own customer data. The query runs ~5000x/day. Write operations are 500 INSERTs/day.

<details>
<summary>Solution</summary>

```sql
-- Key observation: Queries 2 and 3 share the same ORDER BY (ASC),
-- but Query 1 uses DESC. We need ONE index that serves both directions.

-- Optimal index: ASC for queries 2 and 3; DESC can use backward scan
CREATE INDEX IX_Orders_CustomerId_OrderDate_OrderId
    ON Orders(CustomerId, OrderDate ASC, OrderId ASC)
    INCLUDE (TotalAmount);
```

**How this serves all three queries:**

- **Query 2 (ASC, ROWS):** The index provides ordered data in ASC — no Sort needed. ROWS UNBOUNDED PRECEDING without spool.
- **Query 3 (ASC, LAG):** Same ordered data — LAG computes without Sort.
- **Query 1 (DESC):** The optimizer may use a backward scan of the same index (Ordered: true, Scan Direction: BACKWARD). This is supported for window functions in SQL Server 2019+. If backward scan is not used, a Sort is required — but the index covers the ORDER BY columns, making the sort cheaper.

**Alternative if backward scan does not work for the window function:**
```sql
-- Create a second index explicitly for DESC order
CREATE INDEX IX_Orders_CustomerId_OrderDate_OrderId_Desc
    ON Orders(CustomerId, OrderDate DESC, OrderId DESC)
    INCLUDE (TotalAmount);
```

**Tradeoff:** The single ASC index is 30% narrower than two indexes. Write overhead is one index maintenance instead of two. If backward scan works for Query 1, this is the ideal solution. If not, the second index adds ~50MB storage and ~3ms per INSERT.

</details>

### Additional Optimization Techniques

#### Sort Sharing via Common Subexpression Elimination

When multiple window functions have different ORDER BY specifications, check if one ORDER BY is a prefix of another. If both start with the same column, the optimizer can sort by the longer specification and derive the shorter one from the same sorted stream — but only in specific cases:

```sql
-- ❌ Different ORDER BY, no prefix relationship
ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC, OrderId DESC) AS RowNum,
RANK() OVER(PARTITION BY CustomerId ORDER BY TotalAmount DESC) AS AmountRank

-- ✅ Prefix relationship possible (same PARTITION BY, ORDER BY starts with CustomerId)
-- But PARTITION BY is not the same as ORDER BY prefix — this doesn't apply
```

The optimizer does not share sorts when PARTITION BY differs or when the first ORDER BY column differs. Each unique sort requirement gets its own Sort operator.

#### Batch Sorting with TOP N Sort

When using TOP N per group (WHERE RowNum <= N), SQL Server may use a TopN Sort instead of a full Sort. The TopN Sort maintains only the top N rows per partition, significantly reducing memory and CPU:

```sql
-- Full Sort (without WHERE RowNum <= N)
SELECT CustomerId, OrderId, TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY TotalAmount DESC) AS RowNum
FROM Orders;

-- TopN Sort (with WHERE RowNum <= N)
SELECT CustomerId, OrderId, TotalAmount, RowNum
FROM (
    SELECT CustomerId, OrderId, TotalAmount,
        ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY TotalAmount DESC) AS RowNum
    FROM Orders
) Ranked
WHERE RowNum <= 3;
```

The inner query still does a full sort. However, when using `TOP 1 WITH TIES`, the TopN Sort operator appears instead of the full Sort, which is more efficient.

#### Hash Match Aggregate as Sort Alternative

Some window function patterns can be rewritten to use Hash Match Aggregate, which may use less memory than a Sort + Window Aggregate:

```sql
-- Window function approach: Sort + Window Aggregate required
SELECT CustomerId, OrderDate,
    SUM(TotalAmount) OVER(PARTITION BY CustomerId ORDER BY OrderDate) AS RunningTotal
FROM Orders;

-- Rewrite as self-join (uses Hash Match Join instead of Sort)
SELECT o1.CustomerId, o1.OrderDate, SUM(o2.TotalAmount) AS RunningTotal
FROM Orders o1
INNER JOIN Orders o2
    ON o2.CustomerId = o1.CustomerId
    AND o2.OrderDate <= o1.OrderDate
GROUP BY o1.CustomerId, o1.OrderDate;
```

The self-join approach uses a Hash Match (Join) and Hash Match (Aggregate) instead of Sort + Window Aggregate. For tables without a matching index, this can be more efficient because Hash Match spills are less expensive than Sort spills in some scenarios. However, the self-join is O(n²) in the worst case and should only be used when the partition size is known to be small.

#### Parallel Plan for Window Functions

SQL Server can parallelize window functions in specific circumstances. The window function must be a running aggregate (SUM, COUNT, AVG, MIN, MAX) with ROWS or RANGE frame, and the partition size must be large enough to justify parallelism:

```sql
-- This query may get a parallel plan
SELECT CustomerId, OrderId, TotalAmount,
    SUM(TotalAmount) OVER(PARTITION BY CustomerId ORDER BY OrderDate) AS RunningTotal
FROM Orders
ORDER BY CustomerId, OrderDate;
```

The parallel plan shows a Repartition Streams operator that distributes rows by partition key, followed by parallel Window Aggregate operators. Each thread processes a subset of partitions. The Gather Streams operator then merges the results.

To encourage parallel execution for window functions (SQL Server 2019+):

```sql
SELECT ...
FROM Orders
OPTION (USE HINT('ENABLE_PARALLEL_PLAN_WINDOW_AGG'));
```

#### Memory Grant Feedback for Window Functions

SQL Server 2019+ includes Memory Grant Feedback (MGF) for sort operations. After the first execution, SQL Server adjusts the memory grant for subsequent executions based on actual usage:

```sql
-- First execution: estimated rows = 100K, actual = 5M → spill
-- MGF remembers the actual memory needed
-- Second execution: grant is larger, no spill
```

MGF is enabled by default in SQL Server 2019+ (database compatibility level 150+). For window function queries with Sort operators, MGF automatically corrects underestimation after the first execution. This is the primary mitigation for sort spill without query hints.

Monitor MGF adjustments:

```sql
SELECT
    qs.query_id,
    qs.count_compiles,
    qs.avg_memory_grant_kb,
    qs.last_dop,
    qs.last_grant_kb,
    qs.last_used_grant_kb,
    qs.last_ideal_grant_kb
FROM sys.dm_exec_query_stats qs
WHERE qs.last_ideal_grant_kb > qs.last_grant_kb * 1.2;
```

#### Query Store for Window Function Regression Detection

Query Store tracks execution plan changes for window function queries. If an index is added or removed, or if statistics change cause the optimizer to choose a different plan (e.g., from Index Ordered Scan to Sort), Query Store can force the previous plan:

```sql
-- Enable Query Store on the database
ALTER DATABASE YourDatabase SET QUERY_STORE = ON;

-- Force the plan that had no Sort operator
EXEC sp_query_store_force_plan @query_id = 42, @plan_id = 7;
```

This is particularly valuable for window function queries because the optimizer's plan choice (Index Ordered Scan vs Sort) can change abruptly when indexes are added or removed during deployments.
