### Where This Fits

Median calculation is a fundamental statistical operation: the middle value of an ordered dataset, or the interpolated value between the two middle values when the dataset has an even number of rows. SQL Server provides PERCENTILE_CONT(0.5) WITHIN GROUP(ORDER BY col) OVER(PARTITION BY group) to compute the median per group. A .NET backend engineer encounters this in salary surveys, performance percentile analysis, A/B test result aggregation, price analysis, and any reporting where the median better represents central tendency than the average (which is skewed by outliers). The interview signal is strong: candidates must understand that PERCENTILE_CONT interpolates (returns values that may not exist in the data) while PERCENTILE_DISC returns actual values, that both require a full sort of all values per partition, and that the ROW_NUMBER-based median alternative can be 2-5x faster on large datasets.

## Core Mental Model

PERCENTILE_CONT(0.5) computes the median by: sorting all values in the partition, computing the fractional rank of the 50th percentile using linear interpolation between adjacent values when the median falls between two data points, and returning that interpolated value as a continuous result. For an odd number of rows, the median is the middle value. For an even number, it is the average of the two middle values (interpolation). The key invariant: PERCENTILE_CONT always returns a result that lies within the range of the data but may not be an actual observed value — for example, salaries of $50,000 and $60,000 produce a median of $55,000, which is interpolated. PERCENTILE_CONT requires a WITHIN GROUP clause (for ORDER BY) and an OVER clause (for PARTITION BY). The execution plan shows a Stream Aggregate over a Sort — every row in each partition must be sorted before the median can be computed. This full sort is O(n log n) per partition and is the dominant cost.

### Classification

**For SQL topics:** PERCENTILE_CONT is an analytic (window) function in the PERCENTILE family. It is an ordered set function that requires WITHIN GROUP (ORDER BY col) syntax. The OVER clause must include PARTITION BY (optional without PARTITION BY for a single global median). The function is NOT SARGable — it requires a full scan and sort of the partition data. The execution plan uses a Sort followed by a Stream Aggregate. For large partitions, this Sort can spill to tempdb. The alternative ROW_NUMBER-based median avoids the PERCENTILE-specific overhead by computing row counts and picking the middle value(s) with a simple filter, often 2-5x faster on datasets with large partitions.

## Deep Mechanics

### How the Engine Executes This

1. **Parse and bind:** SQL Server recognises PERCENTILE_CONT as an ordered set function. The WITHIN GROUP (ORDER BY col) clause specifies the value column for median computation. The OVER (PARTITION BY group) clause specifies the partitioning.

2. **Sort phase:** SQL Server sorts the input data by the PARTITION BY columns followed by the WITHIN GROUP ORDER BY column. For `PARTITION BY DepartmentId ORDER BY Salary`, the sort key is `DepartmentId, Salary ASC`. This sort is mandatory — PERCENTILE_CONT cannot compute without ordered input. The sort is the dominant cost.

3. **Stream Aggregate phase:** SQL Server walks the sorted data group by group. For each partition:
   - It counts the number of rows (N).
   - It computes the median position: for odd N, the (N+1)/2-th row; for even N, it interpolates between the N/2-th and (N/2+1)-th rows.
   - For PERCENTILE_CONT with 0.5: the fractional rank is computed as ROW_NUMBER / (total_rows - 1). The function then linearly interpolates between the two adjacent rows that bracket the 0.5 mark.
   - The interpolated value = value_at_lower_row + (value_at_upper_row - value_at_lower_row) * fractional_part.

4. **Output:** The median value is returned for each row in the partition (duplicated across all rows in the same partition). This duplication is a quirk — every row in the partition gets the same median value. Use DISTINCT or GROUP BY to get one median per group.

**Detailed interpolation example:**

```
Partition: Salaries = [30000, 40000, 50000, 60000, 70000] (N=5, odd)
  Median position: (5+1)/2 = 3rd row → 50000
  PERCENTILE_CONT(0.5) returns 50000 (exact value)

Partition: Salaries = [30000, 40000, 50000, 60000] (N=4, even)
  Median: between 2nd (40000) and 3rd (50000) row
  PERCENTILE_CONT interpolates: 40000 + (50000-40000) * 0.5 = 45000
  PERCENTILE_DISC(0.5) returns 40000 (the value at or below the median)

Partition: Salaries = [30000, 50000, 100000] (N=3, odd but sparse)
  Median position: (3+1)/2 = 2nd row → 50000
  PERCENTILE_CONT(0.5) returns 50000
```

### Execution Plan Analysis

**PERCENTILE_CONT plan (standard):**

```
[Clustered Index Scan Employees]
  → [Clustered Index Scan Departments]  -- or Merge Join if indexed
  → [Merge Join (Inner Join)]  -- on DepartmentId, preserves order
  → [Sort — ORDER BY DepartmentId, Salary ASC]  ← 70% of cost
  → [Stream Aggregate — compute median per partition]
      Partition By: DepartmentId
      Aggregate: PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY Salary)
  → [SELECT — DISTINCT removes duplicated median rows]
Estimated Cost: ~32 (Sort: ~22, Aggregate: ~6, Join: ~4)
Logical Reads: ~4,500 (table scans) + Sort spill I/O if memory insufficient
```

**PERCENTILE_CONT on large single partition:**

```
[Clustered Index Scan Orders]
  → [Sort — ORDER BY TotalAmount ASC]  ← 80% of cost, may spill
  → [Stream Aggregate — global median]
Estimated Cost: ~65 (Sort: ~52)
Logical Reads: ~12,500
Memory Grant: ~500 MB (for the Sort of 5M rows)
```

**ROW_NUMBER median alternative plan:**

```
[Index Scan (NonClustered) IX_Orders_CustomerId_TotalAmount — ordered]
  → [Segment Top — count rows, identify middle]  ← much cheaper
  → [Compute Scalar — AVG of two middle values for even counts]
Estimated Cost: ~8 (no sort, no spill)
Logical Reads: ~450
```

### Cost Visibility

```sql
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

-- PERCENTILE_CONT median (5M rows, 500 departments)
SELECT DISTINCT
    e.DepartmentId,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary
FROM dbo.Employees e;

-- Expected output:
-- Table 'Employees'. Scan count 1, logical reads 15200
-- Worktable (sort). Scan count 1, logical reads 8500
-- SQL Server Execution Times: CPU time = 3200ms, elapsed time = 6800ms
-- Plan: Clustered Index Scan → Sort (spills to tempdb) → Stream Aggregate

-- ROW_NUMBER-based median (same data)
WITH employee_ranks AS (
    SELECT e.DepartmentId, e.Salary,
        COUNT(*) OVER (PARTITION BY e.DepartmentId) AS cnt,
        ROW_NUMBER() OVER (PARTITION BY e.DepartmentId ORDER BY e.Salary) AS rn
    FROM dbo.Employees e
)
SELECT
    r.DepartmentId,
    AVG(CAST(r.Salary AS DECIMAL(18,2))) AS MedianSalary
FROM employee_ranks r
WHERE r.rn IN ((r.cnt + 1) / 2, (r.cnt + 2) / 2)
GROUP BY r.DepartmentId;

-- Expected output:
-- Table 'Employees'. Scan count 1, logical reads 15200
-- SQL Server Execution Times: CPU time = 1200ms, elapsed time = 2500ms
-- Plan: Index Scan → Window Spool (COUNT + ROW_NUMBER) → Filter → Hash Match Aggregate
```

**Improvement:** ROW_NUMBER approach: 2,500ms vs 6,800ms (2.7x faster). No Sort spill to tempdb.

### Failure Modes

**DISTINCT required to deduplicate:** PERCENTILE_CONT returns the same median value for every row in the partition. Without DISTINCT or GROUP BY, the result set has one row per input row, all with the same median. Developers often forget DISTINCT and then wonder why the query never finishes (because the output set is 5M rows instead of 500).

**Even number of rows produces interpolated value:** For an even number of rows, PERCENTILE_CONT returns a value that may not exist in the data. This subtlety matters for salary data: the median of [50K, 60K] is 55K, which is not an actual salary. PERCENTILE_DISC would return 50K.

**Sort spill on large partitions:** PERCENTILE_CONT must sort all values in each partition. For a partition with 1M rows (e.g., global median of 5M orders), the Sort requires ~500 MB memory. If insufficient, it spills to tempdb, dramatically increasing I/O.

### Primary SQL Implementation

```sql
-- ============================================================
-- Schema context
-- ============================================================
CREATE TABLE dbo.Employees (
    EmployeeId   INT           NOT NULL IDENTITY(1,1),
    FirstName    NVARCHAR(100) NOT NULL,
    LastName     NVARCHAR(100) NOT NULL,
    DepartmentId INT           NOT NULL,
    Salary       DECIMAL(18,2) NOT NULL,
    HireDate     DATE          NOT NULL,
    Status       VARCHAR(20)   NOT NULL DEFAULT 'Active',
    CONSTRAINT PK_Employees PRIMARY KEY CLUSTERED (EmployeeId)
);

CREATE TABLE dbo.Departments (
    DepartmentId   INT           NOT NULL IDENTITY(1,1),
    DepartmentName NVARCHAR(100) NOT NULL,
    Location       VARCHAR(100)  NULL,
    Budget         DECIMAL(18,2) NOT NULL DEFAULT 0,
    CONSTRAINT PK_Departments PRIMARY KEY CLUSTERED (DepartmentId)
);

-- Index for median queries: covers PARTITION BY and ORDER BY
CREATE INDEX IX_Employees_DepartmentId_Salary
    ON dbo.Employees (DepartmentId, Salary)
    INCLUDE (EmployeeId, FirstName, LastName);

CREATE INDEX IX_Employees_Salary
    ON dbo.Employees (Salary)
    INCLUDE (DepartmentId);

-- ============================================================
-- Pattern 1: Median per department — PERCENTILE_CONT
-- ============================================================
SELECT DISTINCT
    e.DepartmentId,
    d.DepartmentName,
    e.Salary AS MedianSalary,  -- Note: this is the median value, not the employee's salary
    COUNT(*) OVER (PARTITION BY e.DepartmentId) AS EmployeeCount
FROM (
    SELECT DepartmentId,
        PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY Salary)
            OVER (PARTITION BY DepartmentId) AS MedianSalary
    FROM dbo.Employees
    WHERE Status = 'Active'
) med
INNER JOIN dbo.Departments d ON med.DepartmentId = d.DepartmentId
CROSS APPLY (
    SELECT TOP 1 Salary FROM dbo.Employees
    WHERE DepartmentId = med.DepartmentId AND Status = 'Active'
    ORDER BY Salary
) e  -- Just to show the column is not the employee salary
ORDER BY d.DepartmentName;

-- Better: clean median query
SELECT DISTINCT
    e.DepartmentId,
    d.DepartmentName,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary,
    COUNT(*) OVER (PARTITION BY e.DepartmentId) AS EmployeeCount
FROM dbo.Employees e
INNER JOIN dbo.Departments d ON e.DepartmentId = d.DepartmentId
WHERE e.Status = 'Active'
ORDER BY d.DepartmentName;
-- Note: COUNT(*) OVER gives employee count per dept, WITH DISTINCT deduplicates

-- ============================================================
-- Pattern 2: Global median (single value for all data)
-- ============================================================
SELECT DISTINCT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY Salary)
        OVER () AS GlobalMedianSalary
FROM dbo.Employees
WHERE Status = 'Active';

-- ============================================================
-- Pattern 3: Median + Quartiles in one query
-- ============================================================
SELECT DISTINCT
    e.DepartmentId,
    PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Q1,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Median,
    PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Q3,
    COUNT(*) OVER (PARTITION BY e.DepartmentId) AS EmployeeCount
FROM dbo.Employees e
INNER JOIN dbo.Departments d ON e.DepartmentId = d.DepartmentId
WHERE e.Status = 'Active'
ORDER BY d.DepartmentName;

-- ============================================================
-- Pattern 4: ROW_NUMBER-based median (faster on large data)
-- ============================================================
-- For datasets with large partitions, this approach is 2-5x faster
WITH employee_ranks AS (
    SELECT
        e.EmployeeId,
        e.DepartmentId,
        e.Salary,
        COUNT(*) OVER (PARTITION BY e.DepartmentId) AS TotalCount,
        ROW_NUMBER() OVER (
            PARTITION BY e.DepartmentId
            ORDER BY e.Salary
        ) AS RowNum
    FROM dbo.Employees e
    WHERE e.Status = 'Active'
)
SELECT
    r.DepartmentId,
    d.DepartmentName,
    AVG(CAST(r.Salary AS DECIMAL(18,2))) AS MedianSalary,
    MAX(r.TotalCount) AS EmployeeCount
FROM employee_ranks r
INNER JOIN dbo.Departments d ON r.DepartmentId = d.DepartmentId
WHERE r.RowNum IN ((r.TotalCount + 1) / 2, (r.TotalCount + 2) / 2)
GROUP BY r.DepartmentId, d.DepartmentName
ORDER BY d.DepartmentName;

-- ============================================================
-- Pattern 5: Median of aggregated values
-- ============================================================
-- Median order value per customer
WITH customer_orders AS (
    SELECT
        o.CustomerId,
        SUM(o.TotalAmount) AS CustomerLifetimeValue
    FROM dbo.Orders o
    WHERE o.OrderDate >= '2024-01-01'
    GROUP BY o.CustomerId
)
SELECT DISTINCT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY CustomerLifetimeValue)
        OVER () AS MedianCustomerValue
FROM customer_orders;

-- ============================================================
-- Pattern 6: Median with filtering and multiple percentiles
-- ============================================================
SELECT DISTINCT
    e.DepartmentId,
    d.DepartmentName,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary,
    PERCENTILE_CONT(0.1) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Bottom10Pct,
    PERCENTILE_CONT(0.9) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Top90Pct,
    AVG(e.Salary) OVER (PARTITION BY e.DepartmentId) AS AvgSalary,
    STDEV(e.Salary) OVER (PARTITION BY e.DepartmentId) AS SalaryStdDev,
    COUNT(*) OVER (PARTITION BY e.DepartmentId) AS EmployeeCount
FROM dbo.Employees e
INNER JOIN dbo.Departments d ON e.DepartmentId = d.DepartmentId
WHERE e.Status = 'Active' AND e.Salary >= 30000
ORDER BY d.DepartmentName;

-- ============================================================
-- Pattern 7: Median difference from average (skew analysis)
-- ============================================================
WITH department_stats AS (
    SELECT DISTINCT
        e.DepartmentId,
        PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS MedianSalary,
        AVG(e.Salary) OVER (PARTITION BY e.DepartmentId) AS AvgSalary,
        COUNT(*) OVER (PARTITION BY e.DepartmentId) AS EmployeeCount
    FROM dbo.Employees e
    WHERE e.Status = 'Active'
)
SELECT
    d.DepartmentName,
    d.MedianSalary,
    d.AvgSalary,
    ROUND(d.AvgSalary - d.MedianSalary, 2) AS MeanMinusMedian,
    CASE
        WHEN d.MedianSalary = 0 THEN NULL
        ELSE ROUND((d.AvgSalary - d.MedianSalary) / d.MedianSalary * 100, 2)
    END AS SkewPct,
    d.EmployeeCount
FROM department_stats d
ORDER BY SkewPct DESC;

-- ============================================================
-- Pattern 8: Stored procedure for median with multiple granularities
-- ============================================================
CREATE PROCEDURE dbo.GetSalaryMedians
    @DepartmentId INT = NULL,       -- NULL = all departments
    @Percentile DECIMAL(3,2) = 0.5  -- Default median; 0.25 for Q1, 0.75 for Q3
AS
BEGIN
    SET NOCOUNT ON;

    SELECT DISTINCT
        e.DepartmentId,
        d.DepartmentName,
        PERCENTILE_CONT(@Percentile) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS PercentileValue,
        COUNT(*) OVER (PARTITION BY e.DepartmentId) AS RowCount,
        @Percentile AS PercentileRequested
    FROM dbo.Employees e
    INNER JOIN dbo.Departments d ON e.DepartmentId = d.DepartmentId
    WHERE (e.DepartmentId = @DepartmentId OR @DepartmentId IS NULL)
      AND e.Status = 'Active'
    ORDER BY d.DepartmentName;
END;
```

## Gotchas and Production Pitfalls

### DISTINCT Is Required to Avoid Row Duplication

**Pitfall:** PERCENTILE_CONT returns the median value for every row in the partition, not one row per partition. Without DISTINCT, the result set has N rows per partition (one per employee) instead of 1 row per department.

```sql
-- ❌ Returns one row per employee, all with the same median
SELECT e.DepartmentId, e.EmployeeId, e.Salary,
       PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
           OVER (PARTITION BY e.DepartmentId) AS MedianSalary
FROM dbo.Employees e;
-- 10,000 employees → 10,000 rows with 500 distinct median values
```

**Symptom:** The query returns 10,000 rows instead of 500. The application tries to display 10,000 data points on a chart, using 200 MB of memory. The web page renders slowly and crashes the browser.

**Fix:**

```sql
-- ✅ Use DISTINCT to get one row per department
SELECT DISTINCT
    e.DepartmentId,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary
FROM dbo.Employees e;

-- ✅ Or wrap in a subquery:
SELECT DISTINCT DepartmentId, MedianSalary
FROM (
    SELECT e.DepartmentId,
        PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS MedianSalary
    FROM dbo.Employees e
) med;
```

**Cost of not fixing:** An API endpoint designed to return 50 department medians returns 10,000 rows. The response size is 2 MB instead of 10 KB. Mobile clients time out. The server CPU spikes at 80% serialising the JSON.

---

### Sort Spill for Large Partitions

**Pitfall:** Computing median on a large table without a covering index. PERCENTILE_CONT requires a full sort of the value column within each partition. For 5M rows in a single partition (global median), the Sort needs ~500 MB of memory and spills to tempdb if the memory grant is insufficient.

```sql
-- ❌ Global median on 5M rows — Sort spills to tempdb
SELECT DISTINCT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY TotalAmount)
        OVER () AS MedianOrderValue
FROM dbo.Orders;
```

**Symptom:** Execution plan shows Sort with "Operator used tempdb to spill data" warning. The query runs for 30 seconds instead of 5 seconds. Tempdb I/O saturates at 4,000 IOPS.

**Fix:**

```sql
-- ✅ Use ROW_NUMBER-based median for large partitions
WITH order_ranks AS (
    SELECT o.TotalAmount,
        COUNT(*) OVER () AS TotalCount,
        ROW_NUMBER() OVER (ORDER BY o.TotalAmount) AS RowNum
    FROM dbo.Orders o
)
SELECT AVG(CAST(TotalAmount AS DECIMAL(18,2))) AS MedianOrderValue
FROM order_ranks
WHERE RowNum IN ((TotalCount + 1) / 2, (TotalCount + 2) / 2);

-- ✅ Or increase memory grant for PERCENTILE_CONT
SELECT DISTINCT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY TotalAmount)
        OVER () AS MedianOrderValue
FROM dbo.Orders
OPTION (MAX_GRANT_PERCENT = 20);  -- Default is 25%, capped at 12 GB
```

**Cost of not fixing:** A median calculation for a salary survey report runs for 2 minutes on a 10M row table. The report generation script times out. Users manually export data to Excel to compute medians, introducing errors.

---

### Interpolated Value Doesn't Exist in Data

**Pitfall:** Assuming the median returned by PERCENTILE_CONT is an actual data value. For even-sized partitions, PERCENTILE_CONT interpolates between the two middle values.

```sql
-- ❌ Median of [100, 200, 300, 400] = 250, which is not an actual value
SELECT DISTINCT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY Price)
        OVER () AS MedianPrice
FROM dbo.Products;
-- Returns 250 when prices are [100, 200, 300, 400]
```

**Symptom:** The median price reported is $52.50 when every product is priced at $50 or $55. Users question data integrity. The product manager insists on seeing actual product prices.

**Fix:**

```sql
-- ✅ Use PERCENTILE_DISC for discrete (actual) values
SELECT DISTINCT
    PERCENTILE_DISC(0.5) WITHIN GROUP (ORDER BY Price)
        OVER () AS MedianPrice
FROM dbo.Products;
-- Returns 200 for [100, 200, 300, 400] — the value at or below 50th percentile

-- ✅ Or use ROW_NUMBER approach that can return AVG of two actual values
WITH price_ranks AS (
    SELECT Price,
        COUNT(*) OVER () AS cnt,
        ROW_NUMBER() OVER (ORDER BY Price) AS rn
    FROM dbo.Products
)
SELECT AVG(CAST(Price AS DECIMAL(18,2))) AS MedianPrice
FROM price_ranks
WHERE rn IN ((cnt + 1) / 2, (cnt + 2) / 2);
-- Returns 250 for [100, 200, 300, 400] — AVG(200, 300)
```

**Cost of not fixing:** A salary report shows a median of $55,234.50 when the company only pays salaries in round $1,000 increments. HR questions the accuracy of the entire analytics system. Trust takes months to rebuild.

---

### PERCENTILE_CONT Ignores NULLs — But Only in ORDER BY

**Pitfall:** PERCENTILE_CONT ignores NULL values in the WITHIN GROUP ORDER BY column (it does not include NULLs in the median calculation). However, if PARTITION BY column has NULLs, those rows form their own NULL partition. This subtle distinction causes unexpected results when the grouping column is nullable.

```sql
-- ❌ NULL DepartmentId rows form a 'NULL' department
SELECT DISTINCT
    e.DepartmentId,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary
FROM dbo.Employees e;
-- Employees with DepartmentId = NULL get their own median
```

**Symptom:** A department shows as "Unknown" with a median salary. The developer expected NULL departments to be excluded. The report shows an extra row that confuses stakeholders.

**Fix:**

```sql
-- ✅ Exclude NULL grouping values explicitly
SELECT DISTINCT
    e.DepartmentId,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary
FROM dbo.Employees e
WHERE e.DepartmentId IS NOT NULL;  -- Exclude NULL group

-- ✅ Per ANSI, PERCENTILE_CONT ignores NULLs in ORDER BY column
-- This is correct: NULL salaries don't affect the median
```

**Cost of not fixing:** A salary report shows an "Unknown" department with 200 employees and a median of $45,000. These are actually terminated employees whose DepartmentId was set to NULL. The VP questions her own department's median.

---

### Multiple PERCENTILE_CONT Calls Cause Redundant Sorts

**Pitfall:** Using three separate PERCENTILE_CONT functions for Q1, median, and Q3. Each call triggers a separate sort and aggregate pass, tripling the cost.

```sql
-- ❌ Three PERCENTILE_CONT calls = three sorts
SELECT DISTINCT
    e.DepartmentId,
    PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Q1,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Median,
    PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Q3
FROM dbo.Employees e;
```

**Symptom:** Execution plan shows three Stream Aggregate operators, each with a Sort. The query takes 3x longer than necessary.

**Fix:**

```sql
-- ✅ SQL Server may optimise multiple PERCENTILE_CONT into a single
-- sort if the WITHIN GROUP ORDER BY and PARTITION BY are identical.
-- Verify with the actual execution plan:
-- If there is 1 Sort + 1 Stream Aggregate, SQL Server folded them.
-- If there are 3 Sorts, force a single pass with a subquery:

WITH ranked AS (
    SELECT e.DepartmentId, e.Salary,
        ROW_NUMBER() OVER (PARTITION BY e.DepartmentId ORDER BY e.Salary) AS rn,
        COUNT(*) OVER (PARTITION BY e.DepartmentId) AS cnt
    FROM dbo.Employees e
),
medians AS (
    SELECT
        r.DepartmentId,
        AVG(CASE WHEN r.rn = FLOOR((r.cnt + 1) / 2) THEN r.Salary END) AS Q1,
        AVG(CASE WHEN r.rn IN ((r.cnt + 1) / 2, (r.cnt + 2) / 2) THEN r.Salary END) AS Median,
        AVG(CASE WHEN r.rn = FLOOR((r.cnt * 3 + 1) / 2) THEN r.Salary END) AS Q3
    FROM ranked r
    GROUP BY r.DepartmentId
)
SELECT m.DepartmentId, d.DepartmentName, m.Q1, m.Median, m.Q3
FROM medians m
INNER JOIN dbo.Departments d ON m.DepartmentId = d.DepartmentId;
```

**Cost of not fixing:** A quarterly compensation report that computes Q1, median, Q3, P10, and P90 for 50 departments across 50K employees runs for 45 seconds. With one sort, it would run in 15 seconds.

### Spoken Answers

**Q: What is the median and how does PERCENTILE_CONT(0.5) compute it?**

> **Average answer:** The median is the middle value of an ordered dataset. PERCENTILE_CONT(0.5) returns the 50th percentile.

> **Great answer:** The median is the 50th percentile — the value below which 50% of the data falls. PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY col) computes the median by sorting all values in the partition, calculating the exact fractional rank for the 0.5 percentile, and linearly interpolating between the two bracketing values if the median falls between data points. For example, with values [10, 20, 30, 40], the median position is at rank 0.5 which falls between the 2nd and 3rd values, so PERCENTILE_CONT returns (20 + 30) / 2 = 25. This interpolation means the returned value may not exist in the actual data — an important distinction when exact values matter (salaries, prices). The execution plan requires a Sort of all values per partition (the dominant cost) followed by a Stream Aggregate. For a partition of N rows, the sort is O(N log N). The key production insight is that multiple PERCENTILE_CONT calls in the same query may (or may not) be folded into a single sort — verify with the actual plan. The ROW_NUMBER-based alternative is typically 2-5x faster because it avoids the interpolation computation and the Stream Aggregate overhead, and it returns only the middle 1-2 rows instead of outputting the median for every input row.

---

**Q: Compare PERCENTILE_CONT to the ROW_NUMBER-based median approach — which is faster and why?**

> **Great answer:** The ROW_NUMBER approach is typically 2-5x faster than PERCENTILE_CONT on datasets with large partitions. The reasons are: PERCENTILE_CONT is a generalised ordered-set function that computes the value at an arbitrary percentile using linear interpolation. It always sorts all values in the partition (O(N log N)), then the Stream Aggregate runs a full pass over all rows outputting the median for every row (which is later deduplicated with DISTINCT). The ROW_NUMBER approach computes COUNT(*) OVER and ROW_NUMBER() OVER, which requires a single pass of the data, then filters to only the middle 1-2 rows per partition — it never materialises the full sorted output. For a 50K row partition, PERCENTILE_CONT writes 50K rows to the sort workfile and then reads all 50K back through the Stream Aggregate. The ROW_NUMBER approach writes 50K rows to the Window Spool but only reads back 2 rows per department. At 500 departments, PERCENTILE_CONT reads 50K rows from the spool, while ROW_NUMBER reads 1,000 rows (500 departments × 2 middle rows). Additionally, PERCENTILE_CONT must compute the interpolation for even-sized partitions, adding CPU overhead. The ROW_NUMBER approach uses AVG of the two middle values for even partitions, which is a simpler calculation. The tradeoff is that PERCENTILE_CONT is simpler to write and understand — just one function call vs a CTE with window functions. For small datasets (< 10K rows total), the difference is negligible.

---

**Q: How does median calculation behave at 100M rows? What breaks?**

> **Great answer:** At 100M rows, PERCENTILE_CONT is effectively unusable without careful tuning. The single-partition global median would require sorting 100M values — that's ~6 GB of memory for the Sort alone (at 8 bytes per value + row overhead). If the memory grant is insufficient (default max is 25% of server memory, capped at 12 GB per query), the Sort spills to tempdb. Each spill level requires a full read-write cycle of the spilled data, turning a 10-second sort into a 10-minute sort. The ROW_NUMBER approach is better but still expensive: the Window Spool must count and rank 100M rows. At this scale, the production solution is almost never to compute median on-the-fly from the raw table. Instead, pre-compute a histogram or use a sampling approach. For exact median, partition the data into manageable chunks (e.g., compute per-day medians, then aggregate). For approximate median, use APPROX_PERCENTILE_CONT (SQL Server 2022+) which uses a sketch algorithm and runs in O(N) without a sort, with ~1% error. The only case where exact median at 100M rows is feasible is with a covering index that provides sorted input, adequate memory (set MAX_GRANT_PERCENT appropriately), and a patient reporting system that can tolerate 30-60 second execution times.

### Application Checklist

- [ ] DISTINCT is used with PERCENTILE_CONT to deduplicate the per-row output
- [ ] For PERCENTILE_DISC: the returned value is guaranteed to exist in the data
- [ ] For even-sized partitions: you have decided whether interpolation is acceptable
- [ ] The covering index includes the PARTITION BY column and the ORDER BY column
- [ ] Multiple PERCENTILE_CONT calls are verified to fold into a single sort
- [ ] For large partitions (> 100K rows): ROW_NUMBER approach or APPROX_PERCENTILE_CONT is used
- [ ] NULL handling: PARTITION BY columns with NULLs create a NULL group (exclude if unwanted)
- [ ] The memory grant for the Sort is sufficient to avoid tempdb spills

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|PERCENTILE_CONT: simple syntax, one function call|Interpolation for even N; requires DISTINCT|
|PERCENTILE_CONT: works with any percentile|Expensive Sort for large partitions|
|ROW_NUMBER: returns actual values (or AVG of two)|More complex SQL (CTE + window functions)|
|ROW_NUMBER: 2-5x faster on large partitions|Cannot handle arbitrary percentiles without more CASE expressions|
|APPROX_PERCENTILE_CONT: fast, memory-efficient|~1% error; SQL Server 2022+ only|

### Scale Thresholds

- **< 1K rows**: Any approach works. PERCENTILE_CONT is simplest.
- **1K-100K rows**: PERCENTILE_CONT is acceptable. Sort fits in memory.
- **100K-1M rows per partition**: Sort may spill. Use ROW_NUMBER approach.
- **1M-10M rows per partition**: ROW_NUMBER approach required. Covering index essential.
- **> 10M rows per partition**: Pre-compute or use APPROX_PERCENTILE_CONT. On-the-fly exact median is impractical.

### Query Challenges

**Challenge 1 — Write the median salary per department query**

Given `Employees(EmployeeId, DepartmentId, Salary, Status)` and `Departments(DepartmentId, DepartmentName)`, write a query that returns the median salary per department for active employees. Show the department name, median salary, and employee count.

<details>
<summary>Solution</summary>

```sql
SELECT DISTINCT
    e.DepartmentId,
    d.DepartmentName,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary,
    COUNT(*) OVER (PARTITION BY e.DepartmentId) AS EmployeeCount
FROM dbo.Employees e
INNER JOIN dbo.Departments d ON e.DepartmentId = d.DepartmentId
WHERE e.Status = 'Active'
ORDER BY d.DepartmentName;
```

**Logical reads:** ~15,200 (with index on DepartmentId, Salary)
**Execution plan:** [Index Scan] → [Sort] → [Stream Aggregate] → [SELECT]
**EF Core equivalent:** Raw SQL via `SqlQueryRaw<DepartmentMedian>`.

Alternatively, the ROW_NUMBER-based approach:

```sql
WITH employee_ranks AS (
    SELECT e.DepartmentId, e.Salary,
        COUNT(*) OVER (PARTITION BY e.DepartmentId) AS TotalCount,
        ROW_NUMBER() OVER (PARTITION BY e.DepartmentId ORDER BY e.Salary) AS RowNum
    FROM dbo.Employees e
    WHERE e.Status = 'Active'
)
SELECT r.DepartmentId, d.DepartmentName,
    AVG(CAST(r.Salary AS DECIMAL(18,2))) AS MedianSalary,
    MAX(r.TotalCount) AS EmployeeCount
FROM employee_ranks r
INNER JOIN dbo.Departments d ON r.DepartmentId = d.DepartmentId
WHERE r.RowNum IN ((r.TotalCount + 1) / 2, (r.TotalCount + 2) / 2)
GROUP BY r.DepartmentId, d.DepartmentName
ORDER BY d.DepartmentName;
```

**Logical reads:** ~15,200 (same scan). **Runtime:** ~2,500ms vs ~6,800ms for PERCENTILE_CONT.

</details>

**Challenge 2 — Fix the performance problem**

```sql
-- This query takes 45 seconds on an Orders table with 10M rows.
-- SET STATISTICS IO: Table 'Orders'. Scan count 1, logical reads 152000
-- Plan shows Sort with SpillToTempDb warning.

SELECT DISTINCT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY TotalAmount)
        OVER () AS MedianOrderValue
FROM dbo.Orders
WHERE OrderDate >= '2024-01-01';
```

Identify the problems and fix them.

<details>
<summary>Solution</summary>

**Problems:**
1. **Sort spill on 10M rows** — The Sort needs ~800 MB memory for 10M values. Default memory grant may be insufficient, causing tempdb spill.
2. **No covering index** — The Clustered Index Scan reads all columns, using more memory bandwidth than needed.
3. **PERCENTILE_CONT on large single partition is expensive** — Requires sorting all 10M rows and outputting median for each.

**Fixes:**

```sql
-- Option A: ROW_NUMBER-based median (2-5x faster)
WITH order_ranks AS (
    SELECT o.TotalAmount,
        COUNT(*) OVER () AS TotalCount,
        ROW_NUMBER() OVER (ORDER BY o.TotalAmount) AS RowNum
    FROM dbo.Orders o
    WHERE o.OrderDate >= '2024-01-01'
)
SELECT AVG(CAST(TotalAmount AS DECIMAL(18,2))) AS MedianOrderValue
FROM order_ranks
WHERE RowNum IN ((TotalCount + 1) / 2, (TotalCount + 2) / 2);

-- Option B: Increase memory grant for PERCENTILE_CONT
SELECT DISTINCT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY TotalAmount)
        OVER () AS MedianOrderValue
FROM dbo.Orders
WHERE OrderDate >= '2024-01-01'
OPTION (MAX_GRANT_PERCENT = 15);

-- Option C: Create covering index for the sort
CREATE INDEX IX_Orders_TotalAmount ON dbo.Orders (TotalAmount)
    WHERE OrderDate >= '2024-01-01';
-- Filtered index reduces index size and sort cost
```

**After fix — logical reads:** ~45,000 (filtered index scan, from 152,000). **Runtime:** ~5 seconds (from 45 seconds).

**Challenge 3 — Explain the execution plan**

```sql
-- Query:
SELECT DISTINCT
    e.DepartmentId,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary
FROM dbo.Employees e;

-- Plan:
-- [Clustered Index Scan] → [Sort] (75%) → [Stream Aggregate] (15%) → [SELECT]
-- Cost: 45, Logical reads: ~23,700
```

Why does the plan show a Sort when there is an index on (DepartmentId, Salary)? What would change if the index had (DepartmentId, Salary DESC)?

<details>
<summary>Solution</summary>

**Why the Sort exists even with the index:** PERCENTILE_CONT requires the ORDER BY column (Salary) sorted in ASC order for the median computation (interpolation works on ascending order). If the index is created as `(DepartmentId, Salary)` with default ASC, it matches exactly. But the execution plan may still show a Sort if the index is not covering (missing INCLUDE columns) or if the scan direction doesn't match. Check the index definition: if it has `(DepartmentId, Salary ASC)`, the Sort should be eliminated and replaced with an ordered Index Scan. If the Sort persists, the index may not be covering (it needs to include all columns referenced in the SELECT or used by the Stream Aggregate).

**What changes with (DepartmentId, Salary DESC):** The index order is descending. PERCENTILE_CONT needs ascending order for the WITHIN GROUP ORDER BY. SQL Server can scan a DESC index backward to get ASC order, but the optimiser may still insert a Sort if the backward scan is not estimated as cheaper. In practice, create the index with ASC direction matching the ORDER BY to give the optimiser the clearest path.

**To fix:** Create a covering index that exactly matches the required sort order:
```sql
CREATE INDEX IX_Employees_DepartmentId_Salary_Median
    ON dbo.Employees (DepartmentId, Salary)
    INCLUDE (EmployeeId, FirstName, LastName, Status);
-- Or if the query only needs DepartmentId and Salary:
CREATE INDEX IX_Employees_DepartmentId_Salary_Covering
    ON dbo.Employees (DepartmentId, Salary);
-- Still a clustered index scan if the index is too wide; consider filtered index
```

</details>

**Challenge 4 — Diagnose the logical problem**

A compensation report computes median salary per department:

```sql
SELECT DISTINCT
    e.DepartmentId,
    d.DepartmentName,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary,
    AVG(e.Salary) OVER (PARTITION BY e.DepartmentId) AS AvgSalary
FROM dbo.Employees e
INNER JOIN dbo.Departments d ON e.DepartmentId = d.DepartmentId;
```

The HR team reports: "The median for department 5 is $52,500, but every employee in that department earns either $50,000 or $55,000. How can the median be $52,500? That's not anyone's salary!"

<details>
<summary>Solution</summary>

**Root cause:** PERCENTILE_CONT interpolates for even-sized partitions. Department 5 has an even number of employees. The two middle salaries are $50,000 and $55,000, and PERCENTILE_CONT returns their average: ($50,000 + $55,000) / 2 = $52,500. This is the mathematically correct median (the value at the 50th percentile, linearly interpolated between the two bracketing values), but it does not correspond to any actual employee's salary.

**Fix:** Use PERCENTILE_DISC for discrete (actual) values, or use the ROW_NUMBER approach:

```sql
-- Option A: PERCENTILE_DISC — returns actual value at or below 50th percentile
SELECT DISTINCT
    e.DepartmentId,
    d.DepartmentName,
    PERCENTILE_DISC(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary
FROM dbo.Employees e
INNER JOIN dbo.Departments d ON e.DepartmentId = d.DepartmentId;
-- Returns $50,000 for dept 5 (value at or below 50th percentile)

-- Option B: ROW_NUMBER-based median (returns AVG of two middle actual values)
WITH employee_ranks AS (
    SELECT e.DepartmentId, e.Salary,
        COUNT(*) OVER (PARTITION BY e.DepartmentId) AS TotalCount,
        ROW_NUMBER() OVER (PARTITION BY e.DepartmentId ORDER BY e.Salary) AS RowNum
    FROM dbo.Employees e
)
SELECT r.DepartmentId, AVG(CAST(r.Salary AS DECIMAL(18,2))) AS MedianSalary
FROM employee_ranks r
WHERE r.RowNum IN ((r.TotalCount + 1) / 2, (r.TotalCount + 2) / 2)
GROUP BY r.DepartmentId;
-- Returns ($50,000 + $55,000) / 2 = $52,500
-- But explicitly averaging two actual values is more transparent
```

**Explanation to HR:** "PERCENTILE_CONT computes the median mathematically — for an even number of employees, it averages the two middle salaries. This gives a precise statistical median but may not match any individual's salary. If you need a median that always equals an actual salary, we use a different function."

</details>

**Challenge 5 — Design the salary analytics infrastructure**

**Scenario:** A company with 50,000 employees across 500 departments needs a salary analytics dashboard that shows:
- Median, average, P10, P25, P75, P90 salaries per department
- Salary range (min, max) per department
- Year-over-year change in median per department
- All queries must return in under 2 seconds

Design the SQL, indexing, and materialisation strategy.

<details>
<summary>Solution</summary>

**Strategy: Pre-computed summary table refreshed nightly.**

```sql
-- Step 1: Materialised per-department salary statistics
CREATE TABLE dbo.DepartmentSalaryStats (
    DepartmentId       INT           NOT NULL,
    RecordDate         DATE          NOT NULL DEFAULT CAST(GETUTCDATE() AS DATE),
    EmployeeCount      INT           NOT NULL,
    MinSalary          DECIMAL(18,2) NOT NULL,
    MaxSalary          DECIMAL(18,2) NOT NULL,
    AvgSalary          DECIMAL(18,2) NOT NULL,
    MedianSalary       DECIMAL(18,2) NOT NULL,
    P10Salary          DECIMAL(18,2) NOT NULL,
    P25Salary          DECIMAL(18,2) NOT NULL,
    P75Salary          DECIMAL(18,2) NOT NULL,
    P90Salary          DECIMAL(18,2) NOT NULL,
    TotalPayroll       DECIMAL(18,2) NOT NULL,
    CONSTRAINT PK_DepartmentSalaryStats PRIMARY KEY (DepartmentId, RecordDate)
);

-- Step 2: Nightly refresh stored procedure
CREATE PROCEDURE dbo.RefreshSalaryStats
AS
BEGIN
    SET NOCOUNT ON;

    INSERT INTO dbo.DepartmentSalaryStats (
        DepartmentId, EmployeeCount,
        MinSalary, MaxSalary, AvgSalary, MedianSalary,
        P10Salary, P25Salary, P75Salary, P90Salary,
        TotalPayroll
    )
    SELECT
        e.DepartmentId,
        COUNT(*) AS EmployeeCount,
        MIN(e.Salary) AS MinSalary,
        MAX(e.Salary) AS MaxSalary,
        AVG(e.Salary) AS AvgSalary,
        -- ROW_NUMBER-based median (fast)
        (
            SELECT AVG(CAST(Salary AS DECIMAL(18,2)))
            FROM (
                SELECT Salary, RowNum, TotalCount
                FROM (
                    SELECT Salary,
                        ROW_NUMBER() OVER (ORDER BY Salary) AS RowNum,
                        COUNT(*) OVER () AS TotalCount
                    FROM dbo.Employees e2
                    WHERE e2.DepartmentId = e.DepartmentId AND e2.Status = 'Active'
                ) r
                WHERE r.RowNum IN ((r.TotalCount + 1) / 2, (r.TotalCount + 2) / 2)
            ) m
        ) AS MedianSalary,
        -- PERCENTILE_CONT for exact percentiles (slower but only runs once per day)
        PERCENTILE_CONT(0.1) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS P10Salary,
        PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS P25Salary,
        PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS P75Salary,
        PERCENTILE_CONT(0.9) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS P90Salary,
        SUM(e.Salary) AS TotalPayroll
    FROM dbo.Employees e
    WHERE e.Status = 'Active'
    GROUP BY e.DepartmentId;

    -- Keep only last 3 years of stats
    DELETE FROM dbo.DepartmentSalaryStats
    WHERE RecordDate < DATEADD(YEAR, -3, GETUTCDATE());
END;
```

**Indexes:**
```sql
-- For the nightly refresh query:
CREATE INDEX IX_Employees_DepartmentId_Salary_Status
    ON dbo.Employees (DepartmentId, Salary)
    INCLUDE (Status)
    WHERE Status = 'Active';
```

**Dashboard queries (sub-second):**

```sql
-- Current month stats
SELECT
    d.DepartmentName,
    s.EmployeeCount,
    s.MedianSalary,
    s.AvgSalary,
    s.P10Salary,
    s.P90Salary,
    s.MinSalary,
    s.MaxSalary,
    s.TotalPayroll
FROM dbo.DepartmentSalaryStats s
INNER JOIN dbo.Departments d ON s.DepartmentId = d.DepartmentId
WHERE s.RecordDate = (SELECT MAX(RecordDate) FROM dbo.DepartmentSalaryStats)
ORDER BY d.DepartmentName;

-- YoY median change
SELECT
    s.DepartmentId,
    d.DepartmentName,
    s.MedianSalary AS CurrentMedian,
    s_prev.MedianSalary AS PrevYearMedian,
    ROUND((s.MedianSalary - s_prev.MedianSalary) / s_prev.MedianSalary * 100, 2) AS YoYChange
FROM dbo.DepartmentSalaryStats s
INNER JOIN dbo.Departments d ON s.DepartmentId = d.DepartmentId
INNER JOIN dbo.DepartmentSalaryStats s_prev
    ON s.DepartmentId = s_prev.DepartmentId
    AND s_prev.RecordDate = DATEADD(YEAR, -1, s.RecordDate)
WHERE s.RecordDate = (SELECT MAX(RecordDate) FROM dbo.DepartmentSalaryStats);
```

**Expected performance:**
- Nightly refresh: ~30 seconds (50K employees, per-dept percentiles)
- Dashboard queries: < 10ms (PK lookups on summary table)
- Storage: 500 departments × 3 years × ~365 days = 547K rows, ~50 MB

</details>
