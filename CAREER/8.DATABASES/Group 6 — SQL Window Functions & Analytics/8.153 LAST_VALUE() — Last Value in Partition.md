### Where This Fits

LAST_VALUE(col) OVER(...) returns the value of a column from the last row within the window frame. This is one of the most misused functions in SQL because of its default frame behavior. Every .NET backend engineer encounters LAST_VALUE when computing the most recent status in a workflow, the latest order value per customer, or the current inventory level from a history table. The single most expensive mistake made here is using LAST_VALUE without an explicit frame and expecting it to return the last value of the partition. With the default frame (RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW), LAST_VALUE returns the current row's value for every row — completely useless for most analytics use cases. Interviewers use LAST_VALUE to test whether candidates truly understand window frame semantics: anyone who writes LAST_VALUE without UNBOUNDED FOLLOWING is immediately flagged as not understanding frames. This is one of the most common SQL interview trap questions.

## Core Mental Model

LAST_VALUE returns the value from the last row of the window frame as defined by the frame specification. The "last" row is the row with the highest ORDER BY value within the frame boundaries. The critical and counterintuitive detail: with only PARTITION BY and ORDER BY specified (no explicit frame), the default frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. This means the frame ends at the current row — LAST_VALUE returns the current row's value, not the partition's last value. For every row except the very last row of the partition, LAST_VALUE returns the wrong value (it returns that row's own value). To get the true last value of the partition, you must specify a frame that extends beyond the current row: ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING (the entire partition), or ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING (from current to end), or ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW (if you actually want the running last value up to the current row). The symmetry with FIRST_VALUE is instructive: FIRST_VALUE with the default frame happens to work correctly because its frame starts at UNBOUNDED PRECEDING (the beginning of the partition). LAST_VALUE's frame ends at CURRENT ROW by default, which is almost never what you want for "last value of the partition."

### Classification

LAST_VALUE is a **frame-based window function** in the analytic function family. It belongs to the set of value functions (FIRST_VALUE, LAST_VALUE, NTH_VALUE) that return values from specific positions within the window frame. The query optimizer uses a Window Spool + Segment + Window Aggregate operator sequence. Unlike FIRST_VALUE which can be optimized to compute once per partition, LAST_VALUE with UNBOUNDED FOLLOWING must read the end of the spool for each row, which requires either a full spool scan or a backward read pattern. This makes LAST_VALUE slightly more expensive than FIRST_VALUE in some implementations. LAST_VALUE is not SARGable.

## Deep Mechanics

### How the Engine Executes This

1. **Row retrieval** — The storage engine reads rows from FROM tables, applies WHERE filters, and performs joins.

2. **Partition segmentation** — The Segment operator divides the input into partition groups. Each partition boundary resets the frame calculation.

3. **Sort** — The Sort operator orders rows by the ORDER BY clause within each partition. The last row in sorted order is the row with the highest ORDER BY values.

4. **Window Spool (materialization)** — The sorted rows are written to a spool. For LAST_VALUE with UNBOUNDED FOLLOWING, the spool must be fully materialized before the function can be computed because the "last" row for any given row may be at the end of the partition.

5. **Window Aggregate — LAST_VALUE resolution:**

   - **With default frame (RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW):** The frame ends at the current row's position (or the current peer group for RANGE). LAST_VALUE reads the value from the last row within that truncated frame — which is the current row itself (for ROWS) or the last peer (for RANGE). This is almost never the intended result.

   - **With UNBOUNDED FOLLOWING (ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING):** The frame covers the entire partition. LAST_VALUE reads the value from the last row of the spool (position N-1 in a partition with N rows). The engine may optimize this by reading the last row of each partition once and returning that value for all rows in the partition.

   - **With ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING:** The frame starts at the current row and extends to the end. LAST_VALUE returns the last value from the end of the partition — the same value for all rows, but the engine may compute it differently.

6. **Key performance insight:** For LAST_VALUE with UNBOUNDED FOLLOWING, the engine must either:
   - Read backward from the end of the spool for each row (potentially expensive), or
   - Scan the spool forward, track the "last seen" value, and at the end of the partition, back-fill all rows with that value.
   
   SQL Server uses the back-fill approach: it scans the partition, caches rows, computes the last value at the partition end, and rewinds to fill cached rows.

**Detailed algorithm (default frame — WRONG for most use cases):**

```
For each partition:
  1. Sort rows by ORDER BY cols
  2. Materialize sorted rows to spool
  3. For i = 0 to (num_rows - 1):
       frame_end = i (CURRENT ROW for ROWS)
       last_value = spool[frame_end].col  -- which is spool[i].col — the current row!
       output(current_row, last_value)
```

**Detailed algorithm (UNBOUNDED FOLLOWING — correct):**

```
For each partition:
  1. Sort rows by ORDER BY cols
  2. Materialize sorted rows to spool
  3. Determine absolute_last_index = num_rows - 1
  4. actual_last_value = spool[absolute_last_index].col
  5. For i = 0 to (num_rows - 1):
       output(current_row, actual_last_value)
```

### SQL Visibility

```sql
-- ❌ WRONG: LAST_VALUE with default frame (TRAP!)
-- Returns the CURRENT ROW's value, not the last of partition!
SELECT
    o.OrderId,
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        -- Default frame: RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS LastAmount_WRONG
FROM dbo.Orders o
WHERE o.CustomerId = 1001
ORDER BY o.OrderDate;
-- Row 1: LastAmount = Row 1's amount (correct by coincidence)
-- Row 2: LastAmount = Row 2's amount (WRONG — should be last row's amount)
-- Row 3: LastAmount = Row 3's amount (WRONG — should be last row's amount)
-- Last row: LastAmount = Last row's amount (correct by coincidence)

-- ✅ CORRECT: LAST_VALUE with UNBOUNDED FOLLOWING

    o.OrderId,
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING

FROM dbo.Orders o
WHERE o.CustomerId = 1001
ORDER BY o.OrderDate;

-- LAST_VALUE comparison: wrong vs correct
WITH OrderData AS (
    SELECT
        o.CustomerId,
        o.OrderId,
        o.OrderDate,
        o.TotalAmount,
        -- WRONG: default frame
        LAST_VALUE(o.TotalAmount) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate
        ) AS LastAmount_Wrong,
        -- CORRECT: explicit frame to end of partition
        LAST_VALUE(o.TotalAmount) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS LastAmount_Correct,
        ROW_NUMBER() OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
        ) AS OrderSeq,
        COUNT(*) OVER (PARTITION BY o.CustomerId) AS CustomerOrderCount
    FROM dbo.Orders o
)
SELECT
    CustomerId,
    OrderSeq,
    OrderId,
    OrderDate,
    TotalAmount,
    LastAmount_Wrong,
    LastAmount_Correct,
    CASE
        WHEN OrderSeq = 1 AND LastAmount_Wrong <> LastAmount_Correct
            THEN 'FIRST_VALUE trap: wrong for rows before last'
        WHEN OrderSeq = CustomerOrderCount AND LastAmount_Wrong = LastAmount_Correct
            THEN 'Correct only for last row'
        ELSE ''
    END AS Note
FROM OrderData
ORDER BY CustomerId, OrderSeq;

-- LAST_VALUE with ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING
-- Returns the last value from current row to end of partition
SELECT
    o.CustomerId,
    o.OrderId,
    o.OrderDate,
    o.TotalAmount,

        PARTITION BY o.CustomerId

FROM dbo.Orders o
ORDER BY o.CustomerId, o.OrderDate;
-- Row 1: RemainingLastAmount = last of partition (includes rows 2-N)
-- Row 2: RemainingLastAmount = last of partition (includes rows 3-N)
-- Last row: RemainingLastAmount = its own value (only row in frame)
```

### Execution Plan Analysis

For `SELECT CustomerId, OrderDate, LAST_VALUE(TotalAmount) OVER(PARTITION BY CustomerId ORDER BY OrderDate ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING) FROM Orders`:

```
Expected plan shape:
[Clustered Index Scan / Index Seek]
→ [Segment (partition by CustomerId)]
→ [Sort (by OrderDate within each partition)]
→ [Window Spool (materialize sorted rows)]
→ [Window Aggregate (compute LAST_VALUE with UNBOUNDED FOLLOWING)]
→ [SELECT]

Estimated Cost: Scan 10%, Segment 2%, Sort 55%, Spool 18%, Aggregate 15%
```

- The Window Aggregate for LAST_VALUE with UNBOUNDED FOLLOWING is slightly more expensive than FIRST_VALUE because the engine must either back-fill values after reaching the partition end, or scan the spool backward.
- With the default frame (no UNBOUNDED FOLLOWING), the plan is similar but the Window Aggregate is cheaper (trivially returns current row value) — yet the results are wrong.
- With a covering index on (CustomerId, OrderDate) INCLUDE (TotalAmount), the Sort is eliminated because rows arrive already ordered. The Segment still operates, but the Index Scan provides order-preserving access.

**Execution plan with default frame (WRONG results, cheaper but useless):**

```
[Clustered Index Scan]
→ [Segment]
→ [Sort]
→ [Window Spool]
→ [Window Aggregate (trivial — returns spool[current_row])]
→ [SELECT]
```

### Failure Modes

**Failure Mode 1: Default frame — the single most common LAST_VALUE mistake.**

```sql
-- ❌ WRONG: LAST_VALUE without explicit UNBOUNDED FOLLOWING
SELECT
    o.CustomerId,
    o.OrderId,
    o.OrderDate,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
    ) AS LastOrderAmount
FROM dbo.Orders o;

-- ✅ CORRECT: Add UNBOUNDED FOLLOWING
SELECT
    o.CustomerId,
    o.OrderId,
    o.OrderDate,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastOrderAmount
FROM dbo.Orders o;
```

**Symptom:** LAST_VALUE returns the current row's value for every row except the last row of the partition (where it is accidentally correct). Engineers test on small datasets and think it works, then deploy to production where analytics are wrong.

**Fix:** Always specify ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING when you want the true last value of the partition.

**Cost of not fixing:** All analytics that depend on the last value are wrong. Status reports show the current status instead of the final status. "Latest order value" is actually each row's own value. This is a silent data corruption bug — no error message, just wrong numbers.

**Failure Mode 2: Assuming LAST_VALUE and FIRST_VALUE have symmetric default frame behavior.**

```sql
-- FIRST_VALUE with default frame happens to work:
FIRST_VALUE(col) OVER (PARTITION BY p ORDER BY o)
-- Returns first value of partition ✓

-- Last VALUE with default frame is BROKEN:
LAST_VALUE(col) OVER (PARTITION BY p ORDER BY o)
-- Returns current row value ✗
```

**Symptom:** The engineer tested FIRST_VALUE and it worked, so they assume LAST_VALUE works the same way. The underlying reason — FIRST_VALUE's frame starts at UNBOUNDED PRECEDING (always correct), LAST_VALUE's frame ends at CURRENT ROW (always wrong) — is subtle.

**Fix:** Never assume frame symmetry between FIRST_VALUE and LAST_VALUE. Always specify the frame explicitly for LAST_VALUE.

**Cost of not fixing:** Same as Failure Mode 1 — silent data corruption.

**Failure Mode 3: LAST_VALUE without ORDER BY — "last" is undefined.**

```sql
-- ❌ WRONG: No ORDER BY
SELECT
    o.CustomerId,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastAmount
FROM dbo.Orders o;

-- ✅ Correct: Specify ORDER BY
SELECT
    o.CustomerId,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastAmount
FROM dbo.Orders o;
```

**Symptom:** Even with the correct frame, "last" is ambiguous without ORDER BY. The engine uses physical row order, which changes with index reorganizations or data modifications.

**Fix:** Always specify ORDER BY to define which row is "last."

**Cost of not fixing:** Non-reproducible results. The "latest" order changes depending on which scan order the engine uses.

**Failure Mode 4: LAST_VALUE with RANGE frame and ties — incorrect "last" with peers.**

```sql
-- ❌ Risky: RANGE frame with non-unique ORDER BY
SELECT
    o.CustomerId,
    o.OrderDate,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastAmount
FROM dbo.Orders o;
-- With ROWS, frame is row-precise — correct
-- With RANGE (default), ties create peer groups
```

**Symptom:** If ORDER BY is not unique, RANGE frame peer groups may cause LAST_VALUE to return a value from a non-deterministic "last" peer.

**Fix:** Use ROWS (not RANGE) for precise row-based boundaries. Add a unique tiebreaker to ORDER BY.

**Cost of not fixing:** Deterministic results when ORDER BY includes ties.

**Failure Mode 5: LAST_VALUE with ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING — semantic confusion.**

```sql
-- LAST_VALUE with frame starting at CURRENT ROW:
SELECT
    o.CustomerId,
    o.OrderId,
    o.OrderDate,
    o.TotalAmount,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING
    ) AS RemainingLastAmount
FROM dbo.Orders o;
-- For first row: returns last of partition ✓
-- For second row: returns last of partition ✓
-- For last row: returns its own value (only row in frame)
```

**Symptom:** The engineer expects the same value for all rows, but the last row gets its own value instead of... itself (which happens to be correct but for a different reason).

**Fix:** Use ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING for the absolute last value regardless of row position.

**Cost of not fixing:** Confusion during code review. The frame behavior is correct but unnecessarily complex.

**Failure Mode 6: Performance regression from LAST_VALUE with large partitions.**

```sql
-- LAST_VALUE with UNBOUNDED FOLLOWING requires the engine to
-- either scan backward or cache all rows until partition end
SELECT
    o.CustomerId,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastAmount
FROM dbo.Orders o;
```

**Symptom:** For very large partitions (millions of rows), the back-fill strategy requires caching all rows until the engine reaches the partition end. This increases memory grant requirements.

**Fix:** Consider using FIRST_VALUE with descending ORDER BY as an alternative — it uses the same spool but can compute forward without back-fill:

```sql
SELECT
    o.CustomerId,
    FIRST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate DESC, o.OrderId DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS LastAmount
FROM dbo.Orders o;
```

```sql
-- Pattern 1: Current (last) status per order from history table
WITH CurrentStatus AS (
    SELECT
        osh.OrderId,
        osh.Status,
        osh.StatusDate,
        osh.ChangedBy,
        LAST_VALUE(osh.Status) OVER (
            PARTITION BY osh.OrderId
            ORDER BY osh.StatusDate, osh.StatusHistoryId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS CurrentStatusValue,
        LAST_VALUE(osh.StatusDate) OVER (
            PARTITION BY osh.OrderId
            ORDER BY osh.StatusDate, osh.StatusHistoryId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS CurrentStatusDate,
        ROW_NUMBER() OVER (
            PARTITION BY osh.OrderId
            ORDER BY osh.StatusDate DESC, osh.StatusHistoryId DESC
        ) AS rn
    FROM dbo.OrderStatusHistory osh
)
SELECT
    OrderId,
    Status AS StatusAtThisPoint,
    StatusDate,
    CurrentStatusValue,
    CurrentStatusDate,
    CASE
        WHEN Status = CurrentStatusValue THEN 'Current'
        ELSE 'Historical'
    END AS IsCurrentStatus
FROM CurrentStatus
WHERE rn = 1  -- Only the most recent status entry
   OR Status <> CurrentStatusValue
ORDER BY OrderId, StatusDate;

-- Pattern 2: Latest order value per customer (as a column on every row)
SELECT
    o.CustomerId,
    o.OrderId,
    o.OrderDate,
    o.TotalAmount,

    LAST_VALUE(o.OrderDate) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LatestOrderDate
FROM dbo.Orders o
ORDER BY o.CustomerId, o.OrderDate;

-- Pattern 3: Inventory — most recent quantity per product
WITH LatestInventory AS (
    SELECT
        i.ProductId,
        i.WarehouseId,
        i.LastUpdated,
        i.QuantityOnHand,
        LAST_VALUE(i.QuantityOnHand) OVER (
            PARTITION BY i.ProductId, i.WarehouseId
            ORDER BY i.LastUpdated
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS CurrentQuantity,
        LAST_VALUE(i.LastUpdated) OVER (
            PARTITION BY i.ProductId, i.WarehouseId
            ORDER BY i.LastUpdated
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS CurrentUpdatedTime
    FROM dbo.InventoryItems i
)
SELECT DISTINCT
    ProductId,
    WarehouseId,
    CurrentQuantity,
    CurrentUpdatedTime
FROM LatestInventory
ORDER BY ProductId, WarehouseId;

-- Pattern 4: Running last value (progressive — last value up to current row)
-- This is one case where the DEFAULT frame behavior is actually useful
SELECT
    o.CustomerId,
    o.OrderId,
    o.OrderDate,
    o.TotalAmount,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningLastAmount
FROM dbo.Orders o
ORDER BY o.CustomerId, o.OrderDate;
-- Row 1: RunningLastAmount = Row 1 value (only row seen so far)
-- Row 2: RunningLastAmount = Row 2 value (last of rows 1-2)
-- Row N: RunningLastAmount = Row N value (last of rows 1-N)
-- This is essentially the same as the current row's value — USELESS
-- But if you wanted the last seen distinct status:

SELECT
    osh.OrderId,
    osh.StatusDate,
    osh.Status,
    LAST_VALUE(osh.Status) OVER (
        PARTITION BY osh.OrderId
        ORDER BY osh.StatusDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS LastKnownStatus
FROM dbo.OrderStatusHistory osh
ORDER BY osh.OrderId, osh.StatusDate;
-- Row 1: LastKnownStatus = 'New'
-- Row 2: LastKnownStatus = 'Shipped' (most recent status so far)
-- Row 3: LastKnownStatus = 'Delivered' (most recent status so far)
-- This gives you the "current status up to this point in time"

-- Pattern 5: FIRST_VALUE vs LAST_VALUE for customer lifetime metrics
WITH CustomerLifetime AS (
    SELECT
        o.CustomerId,
        o.OrderId,
        o.OrderDate,
        o.TotalAmount,
        FIRST_VALUE(o.OrderDate) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS FirstOrderDate,
        FIRST_VALUE(o.TotalAmount) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS FirstOrderAmount,
        LAST_VALUE(o.OrderDate) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS LastOrderDate,
        LAST_VALUE(o.TotalAmount) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS LastOrderAmount
    FROM dbo.Orders o
)
SELECT
    CustomerId,
    MIN(OrderDate) AS AcqDate,
    MAX(OrderDate) AS LatestDate,
    COUNT(*) AS TotalOrders,
    SUM(TotalAmount) AS LifetimeRevenue,
    -- These are the same for all rows in the partition
    MAX(FirstOrderDate) AS AcquisitionDate,
    MAX(FirstOrderAmount) AS FirstOrderValue,
    MAX(LastOrderDate) AS MostRecentDate,
    MAX(LastOrderAmount) AS MostRecentOrderValue,
    DATEDIFF(day, MAX(FirstOrderDate), MAX(LastOrderDate)) AS CustomerLifetimeDays,
    CASE
        WHEN MAX(LastOrderAmount) > MAX(FirstOrderAmount) THEN 'Spending Increased'
        WHEN MAX(LastOrderAmount) < MAX(FirstOrderAmount) THEN 'Spending Decreased'
        ELSE 'Stable'
    END AS SpendingTrend
FROM CustomerLifetime
GROUP BY CustomerId
ORDER BY LifetimeRevenue DESC;

SELECT
    e.EmployeeId,
    e.FirstName + ' ' + e.LastName AS EmployeeName,
    e.Salary,
    e.HireDate,
    LAST_VALUE(e.Salary) OVER (
        PARTITION BY e.EmployeeId
        ORDER BY e.HireDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS MostRecentSalary,
    LAST_VALUE(e.HireDate) OVER (
        PARTITION BY e.EmployeeId
        ORDER BY e.HireDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS MostRecentSalaryDate
FROM dbo.Employees e
ORDER BY e.EmployeeId, e.HireDate;

-- Pattern 7: LAST_VALUE for gap analysis — compare current to last known good
-- Find inventory items where current quantity differs from last known good
WITH InventoryChanges AS (
    SELECT
        i.ProductId,
        i.WarehouseId,
        i.LastUpdated,
        i.QuantityOnHand,
        LAST_VALUE(i.QuantityOnHand) OVER (
            PARTITION BY i.ProductId, i.WarehouseId
            ORDER BY i.LastUpdated
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS CurrentQuantity,
        LAST_VALUE(i.QuantityOnHand) OVER (
            PARTITION BY i.ProductId, i.WarehouseId
            ORDER BY i.LastUpdated
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) AS QuantityAtThisPoint
    FROM dbo.InventoryItems i
)
SELECT
    ProductId,
    WarehouseId,
    LastUpdated,
    QuantityOnHand,
    CurrentQuantity,
    QuantityAtThisPoint,
    CASE
        WHEN QuantityOnHand <> CurrentQuantity THEN 'Not Latest'
        ELSE 'Latest'
    END AS IsLatest
FROM InventoryChanges
ORDER BY ProductId, WarehouseId, LastUpdated;
```

## Gotchas and Production Pitfalls

### 1. Default Frame Trap — The Most Common Window Function Bug

**Pitfall:** Using LAST_VALUE without specifying UNBOUNDED FOLLOWING. The default frame ends at CURRENT ROW, so LAST_VALUE returns the current row's value for every row.

```sql
-- ❌ WRONG: Default frame — returns current row's value
SELECT
    o.CustomerId,
    o.OrderId,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
    ) AS LastAmount
FROM dbo.Orders o;

-- ✅ CORRECT: Explicit frame to end of partition
SELECT
    o.CustomerId,
    o.OrderId,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastAmount
FROM dbo.Orders o;
```

**Symptom:** The "last" amount is the same as the current row's amount. Only the final row of each partition gets the correct value. This is a silent data corruption bug — no warnings, no errors.

**Fix:** Always specify ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING for the true last value of the partition.

**Cost of not fixing:** Complete analytics corruption. All "last value" calculations are wrong for every non-terminal row. At 10M rows, 9,999,999 rows have the wrong value.

### 2. Symmetry Assumption — FIRST_VALUE Works, So LAST_VALUE Works

**Pitfall:** Engineers test FIRST_VALUE (which works with the default frame) and assume LAST_VALUE works identically.

```sql
-- FIRST_VALUE works with default frame (frame starts at partition beginning):
FIRST_VALUE(col) OVER (PARTITION BY p ORDER BY o)  -- ✓ Correct

-- LAST_VALUE with default frame FAILS (frame ends at current row):
LAST_VALUE(col) OVER (PARTITION BY p ORDER BY o)  -- ✗ Wrong
```

**Symptom:** The engineer's test query appears to work for simple cases (single-row partitions, or when the "last" row is the same as the first row). Production data with multiple rows per partition reveals the bug.

**Fix:** Always explicitly specify the frame for LAST_VALUE. Never rely on default frame behavior.

**Cost of not fixing:** Deployed to production with wrong analytics. Retroactive data fix required.

### 3. LAST_VALUE with RANGE Frame and Non-Unique ORDER BY

**Pitfall:** Using LAST_VALUE with RANGE (explicit or default) and non-unique ORDER BY. RANGE treats ties as peers, and the "last" peer can be non-deterministic.

```sql
-- ❌ Risky: RANGE frame with non-unique ORDER BY
SELECT
    o.CustomerId,
    o.OrderDate,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        RANGE BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastAmount
FROM dbo.Orders o;

-- ✅ Correct: ROWS frame with unique ORDER BY
SELECT
    o.CustomerId,
    o.OrderDate,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastAmount
FROM dbo.Orders o;
```

**Symptom:** When multiple rows share the same ORDER BY value, LAST_VALUE may return different values across executions. RANGE frame considers all ties as a single group.

**Fix:** Use ROWS (not RANGE) and add a unique tiebreaker to ORDER BY.

**Cost of not fixing:** Non-reproducible last values. The "latest" order changes between report runs.

### 4. LAST_VALUE Without ORDER BY (Empty Partition Ordering)

**Pitfall:** Using LAST_VALUE without ORDER BY. The "last" row is determined by physical order, which is non-deterministic.

```sql
-- ❌ Wrong: No ORDER BY
SELECT
    o.CustomerId,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastAmount
FROM dbo.Orders o;

-- ✅ Correct: ORDER BY defines "last"
SELECT
    o.CustomerId,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate DESC, o.OrderId DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastAmount
FROM dbo.Orders o;
```

**Symptom:** The "last" value changes when the physical data layout changes (index rebuild, data modification).

**Fix:** Always specify ORDER BY. Use DESC ordering if you want the most recent row first.

**Cost of not fixing:** Non-reproducible reports. Data integrity issues in downstream systems.

### 5. Memory Grant for LAST_VALUE with Large Partitions

**Pitfall:** LAST_VALUE with UNBOUNDED FOLLOWING on very large partitions (millions of rows) requires the engine to cache all rows until it reaches the partition end, increasing memory pressure.

```sql
-- Memory-intensive: Large partition with LAST_VALUE
SELECT
    o.CustomerId,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastAmount
FROM dbo.Orders o;
```

**Symptom:** The Window Spool memory grant request is larger for LAST_VALUE with UNBOUNDED FOLLOWING than for FIRST_VALUE, because the engine cannot release rows until the partition end is reached.

**Fix:** Use FIRST_VALUE with descending ORDER BY as an alternative — it avoids the back-fill cost:

```sql
SELECT
    o.CustomerId,
    FIRST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate DESC, o.OrderId DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS LastAmount
FROM dbo.Orders o;
```

**Cost of not fixing:** Higher memory grant, potential TempDB spill, RESOURCE_SEMAPHORE waits.

### 6. LAST_VALUE in WHERE Clause (Logical Processing Order)

**Pitfall:** Attempting to filter on LAST_VALUE in the WHERE clause. Window functions are evaluated after WHERE.

```sql
-- ❌ Wrong: LAST_VALUE in WHERE
SELECT OrderId, OrderDate
FROM dbo.Orders
WHERE LAST_VALUE(TotalAmount) OVER (
    PARTITION BY CustomerId
    ORDER BY OrderDate
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
) > 100;
-- Error 4108: Windowed functions cannot be used in the WHERE clause

-- ✅ Correct: CTE or subquery
WITH OrderWithLast AS (
    SELECT
        OrderId,
        OrderDate,
        TotalAmount,
        LAST_VALUE(TotalAmount) OVER (
            PARTITION BY CustomerId
            ORDER BY OrderDate, OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS LastAmount
    FROM dbo.Orders
)
SELECT OrderId, OrderDate
FROM OrderWithLast
WHERE LastAmount > 100;
```

**Symptom:** Error 4108 at compile time.

**Fix:** Wrap in CTE or subquery and filter in the outer scope.

**Cost of not fixing:** Query compilation error.

### Spoken Answers

**Q1: What is LAST_VALUE and what is the critical frame requirement?**

> **Average answer:** "LAST_VALUE returns the last value in a partition. You use it with OVER and ORDER BY. It's like FIRST_VALUE but for the last row."

> **Great answer:** "LAST_VALUE(col) OVER(PARTITION BY ... ORDER BY ...) is a frame-based window function that returns the value of col from the last row within the window frame. The single most critical requirement for correct results is specifying ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING. Without that explicit frame, the default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, which means the frame ends at the current row — LAST_VALUE returns the current row's own value, not the partition's last value. This is the most common window function bug in production SQL. I have personally debugged a production issue where a dashboard showed the current order value as the 'latest order value' for every row, completely corrupting customer lifetime value calculations. The fix was adding three words: AND UNBOUNDED FOLLOWING. The alternative approach — FIRST_VALUE with descending ORDER BY — avoids the frame trap entirely and has slightly better performance because it uses forward-only spool access instead of the back-fill logic LAST_VALUE requires."

**Q5: Compare LAST_VALUE to MAX().**

> **Average answer:** "LAST_VALUE returns the last row while MAX returns the maximum value. They're different things."

> **Great answer:** "LAST_VALUE and MAX are fundamentally different. MAX is an aggregate function that returns the maximum numeric/date value from all rows in a group, regardless of position. LAST_VALUE is a window function that returns the value from the last row in the ordered frame, regardless of the value itself. For example, to get the most recent order date per customer: MAX(OrderDate) and LAST_VALUE(OrderDate) with correct frame give the same result — because the latest date is also the last row in date order. But to get the most recent order amount (which might not be the maximum amount), MAX(TotalAmount) gives the highest amount ever spent, while LAST_VALUE(TotalAmount) gives the amount from the most recent order. These are different questions. The execution plan difference: MAX is a Stream Aggregate or Hash Aggregate operator. LAST_VALUE is a Window Aggregate with Window Spool. MAX scans and aggregates, LAST_VALUE sorts and positions. For large datasets, MAX is faster if it can use an index — it just needs the maximum value from the index's last page. LAST_VALUE requires a full sort of the partition. The performance difference on a table with an index on (CustomerId, OrderDate): MAX can do an Index Seek + TOP 1 (virtually instant), while LAST_VALUE still needs to scan the partition to read the last row's value."

**Q8: How do you ensure LAST_VALUE is correctly implemented in Dapper queries?**

> **Average answer:** "You write the SQL with LAST_VALUE and the correct frame, then Dapper maps the results."

> **Great answer:** "In Dapper, LAST_VALUE is implemented as raw SQL in the const string passed to QueryAsync. The critical code review items for any LAST_VALUE usage are: (1) Does the OVER clause include ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING? (2) Is the ORDER BY unique — does it include a tiebreaker column like the primary key? (3) Is the default value handling correct if the target column can be NULL? I always include a comment above the SQL explaining why the frame is specified. In production, I wrap the LAST_VALUE query in a CTE with ROW_NUMBER() to deduplicate, because LAST_VALUE with UNBOUNDED FOLLOWING returns the same value for every row in the partition. The pattern is: LAST_VALUE in inner window CTE, SELECT DISTINCT or ROW_NUMBER filter in outer query to get one row per partition. For the DTO mapping, Dapper handles the LAST_VALUE column as a regular property — no special handling needed. The key is getting the SQL right."

**Challenge 3 — Explain the execution plan**

```sql
-- Query A (runs but returns wrong results):
SELECT
    o.CustomerId,
    o.OrderId,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
    ) AS LastAmount
FROM dbo.Orders o;

-- Query B (correct):
SELECT
    o.CustomerId,
    o.OrderId,
    LAST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastAmount
FROM dbo.Orders o;
```

Why do both queries have similar execution plans and similar cost, even though Query A produces wrong results?

<details>
<summary>Solution</summary>

**Why plans are similar:** Both queries require the same operators — Clustered Index Scan, Segment, Sort (based on ORDER BY), Window Spool, and Window Aggregate. The difference is in the Window Aggregate's internal logic:

- **Query A (default frame):** The frame ends at CURRENT ROW. The Window Aggregate reads the current row from the spool and returns its value. This is a trivial operation — no back-fill, no additional spool scans.
- **Query B (UNBOUNDED FOLLOWING):** The frame extends to the end of the partition. The Window Aggregate must cache all rows until the partition end is reached, then back-fill the last value to all cached rows. This adds CPU for caching and back-fill, and may increase memory usage for the row cache.

The cost difference between A and B is small (5-15%) because the dominant cost is the Sort (55-60%) and the Spool (15-20%). The Window Aggregate is only 10-15% of total cost. Even doubling the aggregate cost adds only ~10% to total query time.

**Why this is dangerous:** Engineers profile Query A, see it runs in 3.6 seconds, and think it's performing well. They don't notice the wrong results. The fix (Query B) costs 4.1 seconds — only 14% slower — but produces correct results.

**To get a different plan:** Use FIRST_VALUE with descending ORDER BY as the alternative. The plan is identical but the Window Aggregate uses forward-only logic (no back-fill needed).

**Tradeoff:** Default frame is cheaper but wrong. UNBOUNDED FOLLOWING is slightly more expensive but correct. The small performance difference should never be a reason to use the default frame — correct results always trump marginal performance gains.

</details>

**Challenge 5 — Design the index**

**Scenario:** An OrderStatusHistory table has 200M rows. The table records every status change for orders. The primary query is:

```sql
SELECT
    osh.OrderId,
    osh.Status,
    osh.StatusDate,
    LAST_VALUE(osh.Status) OVER (
        PARTITION BY osh.OrderId
        ORDER BY osh.StatusDate, osh.StatusHistoryId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS CurrentStatus
FROM dbo.OrderStatusHistory osh
WHERE osh.OrderId IN (SELECT OrderId FROM @ActiveOrders);
```

The query runs for a set of 1,000 active orders. Each order has an average of 15 status changes. Current execution time is 8 seconds. Read/write ratio is 90/10.

<details>
<summary>Solution</summary>

```sql
-- Primary index: Covering index for the window function query
CREATE INDEX IX_OrderStatusHistory_OrderId_StatusDate
ON dbo.OrderStatusHistory(OrderId, StatusDate, StatusHistoryId)
INCLUDE (Status, ChangedBy);
```

**Why this index:**
- OrderId as the leading column enables an Index Seek for each active order in the filter
- StatusDate + StatusHistoryId provides the sorted order required by LAST_VALUE's ORDER BY, eliminating the Sort operator
- INCLUDE (Status, ChangedBy) makes the index covering — the query can be satisfied entirely from the index, no clustered index lookups

**Without this index:**
- Clustered index scan + Sort: 200M rows sorted by (OrderId, StatusDate)
- Memory grant: ~4-6GB for sort
- Execution time: 8 seconds (as reported)

**With this index:**
- Index Seek per OrderId (1,000 seeks, each returning ~15 rows = 15K total rows)
- No Sort — rows arrive already ordered by (OrderId, StatusDate)
- Window Spool: 15 rows per partition — trivial, in-memory
- Estimated: < 100ms

**Tradeoffs:**
- Size: OrderId (4) + StatusDate (8) + StatusHistoryId (8) = 20 bytes key + Status (1) + ChangedBy (200 avg) = ~221 bytes per row × 200M = ~44GB. This is large but justified by the 90/10 read/write ratio.
- Write overhead: Each status change INSERT writes to this index. Estimated 20-30% increase in insert time.

**What NOT to index:**
- Do not create a filtered index for specific statuses — the query filters by OrderId list, not status
- Do not include columns not used in the query (EventData, Notes, etc.)
- Consider partitioning the table by OrderId ranges if the 200M rows grow further

**After index — logical reads:** ~15-30 per OrderId seek (15-30K total from >128K)
**Execution plan:** Index Seek (per OrderId) → Segment → Window Spool (15 rows) → Window Aggregate → SELECT
**Sort eliminated:** Yes — index provides order-preserving access

</details>
