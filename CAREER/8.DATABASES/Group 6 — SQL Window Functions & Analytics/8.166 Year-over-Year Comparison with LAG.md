### Where This Fits

Year-over-Year (YoY) comparison is the most common time-series analytical pattern in business reporting: "How did revenue this month compare to the same month last year?" The pattern uses LAG(amount, 12) OVER(PARTITION BY category ORDER BY year_month) for monthly data — the offset of 12 skips exactly one year back in a monthly series. A .NET backend engineer encounters this daily in financial dashboards, KPI reports, product analytics, and any system that tracks metrics over time with period-over-period comparisons. The interview signal is strong: knowing how to handle missing periods, choosing between LAG and self-join, understanding that ORDER BY must use a proper date or period key rather than a string, and knowing that EF Core cannot generate LAG natively so raw SQL is required.

## Core Mental Model

YoY comparison with LAG works by partitioning data into groups (categories, products, regions) and ordering each partition by the time period (year_month, quarter, etc.). LAG(value, N) retrieves the value from N rows earlier in the ordered partition — for monthly data, LAG(amount, 12) goes back exactly 12 months. The growth rate is computed as (current - previous) / previous * 100. The key invariant: the ORDER BY column must be a proper period identifier that guarantees exactly N rows between periods — using a date string like '2024-01' works only if every month has data; missing months cause incorrect comparisons because LAG simply skips N rows regardless of whether they represent the correct time offset. The production fix is either a date dimension table that generates all periods via a join (guaranteeing contiguous rows) or using the default parameter of LAG to handle the first row where no prior period exists.

### Classification

**For SQL topics:** LAG is a window function in the analytic function family. It accesses rows preceding the current row within the window partition. The query optimiser executes LAG via a Window Spool operator that sorts the data by PARTITION BY and ORDER BY columns, then scans the spool to compute the lag values. LAG is not SARGable — it is a post-sort computation that does not filter rows. The OVER clause is evaluated after WHERE, GROUP BY, and HAVING but before ORDER BY in the final SELECT list, so any filters in WHERE reduce the rows before LAG computes.

## Deep Mechanics

### How the Engine Executes This

1. **Data retrieval phase:** SQL Server reads rows from the base table(s) applying WHERE filters. For example, `SELECT YEAR(order_date) * 100 + MONTH(order_date) AS year_month, category, SUM(amount) AS revenue` groups data by year_month and category.

2. **Sort phase:** The Window Spool operator reads the aggregated input and sorts it by the PARTITION BY columns (e.g., category) and then by the ORDER BY columns (e.g., year_month ASC). This sort is required — Window Spool cannot compute LAG without ordered data. The sort is the dominant cost of this query.

3. **Window Spool phase:** The sorted data is written to a spool (workfile in tempdb). SQL Server then reads the spool sequentially. For each row, LAG(amount, 12) looks back 12 rows in the spool (within the same partition). If the partition changes between current and target row, LAG returns NULL — this handles the boundary automatically.

4. **Computation phase:** SQL Server computes the expression `(amount - prior_amount) / NULLIF(prior_amount, 0) * 100.0` for each row. Rows without a prior period (first 12 months of data per partition) return NULL.

5. **Output phase:** The SELECT list includes all computed columns. Any WHERE clause filtering on the computed YoY value (e.g., `WHERE yoy_growth > 50`) must use a CTE or subquery because LAG is evaluated after WHERE.

6. **Alternative — self-join execution:** The self-join approach does two full scans: one for the current period data, one for the prior period data (with a date offset in the JOIN clause). The database engine then does a hash match or merge join on (category, year_month = year_month - 12). Both scans do the same aggregation, so total work is roughly 2x the scan + join cost vs 1x scan + spool for the window function approach.

### SQL Visibility

```sql
-- The core YoY pattern: LAG with offset 12 for monthly data
-- Requires a date dimension or a guarantee that every month has data
SELECT
    d.year_month,
    d.category,
    d.monthly_revenue,
    LAG(d.monthly_revenue, 12) OVER (
        PARTITION BY d.category
        ORDER BY d.year_month
    ) AS prior_year_revenue,
    ROUND(
        (d.monthly_revenue - LAG(d.monthly_revenue, 12) OVER (
            PARTITION BY d.category
            ORDER BY d.year_month
        ))
        / NULLIF(LAG(d.monthly_revenue, 12) OVER (
            PARTITION BY d.category
            ORDER BY d.year_month
        ), 0) * 100,
        2
    ) AS yoy_growth_pct
FROM (
    SELECT
        YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate) AS year_month,
        p.Category AS category,
        SUM(oi.Quantity * oi.UnitPrice) AS monthly_revenue
    FROM dbo.Orders o
    INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
    INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
    WHERE o.OrderDate >= '2022-01-01'
    GROUP BY YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate), p.Category
) d
ORDER BY d.category, d.year_month;
```

### Execution Plan Analysis

**Window function plan (LAG approach):**

```
[Clustered Index Scan Orders] → [Filter (OrderDate >= '2022-01-01')]
  → [Hash Match (Inner Join) — Orders ↔ OrderItems]
  → [Hash Match (Inner Join) — result ↔ Products]
  → [Hash Match Aggregate — GROUP BY year_month, category]
  → [Sort — ORDER BY category, year_month]
  → [Window Spool — computes LAG(monthly_revenue, 12)]
      Partition By: category
      Order By: year_month
  → [SELECT — computes yoy_growth_pct]
  → [Sort — ORDER BY category, year_month]
Estimated Cost: ~28 (Sort is ~40%, Window Spool ~25%, Aggregation ~20%)
Logical Reads: ~4,500 (table scans + spool writes/reads)
```

**Self-join plan:**

```
  → [Clustered Index Scan Orders (current)] → [Group by month/category]
  → [Clustered Index Scan Orders (prior)]   → [Group by month/category]
  → [Hash Match (Inner Join) — current.year_month = prior.year_month + 12
      AND current.category = prior.category]
  → [SELECT — computes growth]
  → [Sort]
Estimated Cost: ~42 (two full scans + hash join)
Logical Reads: ~9,000 (2x the window approach)
```

### Failure Modes

**Missing period — LAG returns wrong prior value:** When a month has no data (e.g., no orders in January 2023), LAG(value, 12) skips to the 13th row back instead of the correct January 2022 row. The query silently computes incorrect YoY. Fix: join to a date dimension table that guarantees all months exist:

```sql
-- Detect missing months
WITH monthly_data AS (
    SELECT YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate) AS year_month,
           p.Category AS category,
           SUM(oi.Quantity * oi.UnitPrice) AS monthly_revenue
    FROM dbo.Orders o
    INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
    INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
    GROUP BY YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate), p.Category
)
SELECT d.year_month, d.category, COALESCE(m.monthly_revenue, 0) AS monthly_revenue
FROM dbo.DateDimension d
CROSS JOIN (SELECT DISTINCT category FROM dbo.Products) cats
LEFT JOIN monthly_data m ON d.year_month = m.year_month AND cats.category = m.category
WHERE d.date >= '2022-01-01' AND d.date < '2025-01-01'
ORDER BY cats.category, d.year_month;
```

**ORDER BY string date column:** ORDER BY year_month as VARCHAR '2024-01' sorts lexicographically: '2024-01', '2024-02', ..., '2024-10', '2024-11', '2024-12' is correct by luck, but '2024-1' sorts as '2024-1', '2024-10', '2024-11', '2024-12', '2024-2'. Use INTEGER year_month (202401) or DATE type.

**NULLIF division by zero:** When prior_year_revenue is 0, the growth calculation divides by zero. Wrap with NULLIF(prior, 0) or COALESCE(NULLIF(...), 0).

**First period NULL handling:** The first 12 months of data per partition have no prior year. These rows show NULL for prior_year_revenue. Applications must handle NULL display (show "N/A" or exclude via WHERE prior IS NOT NULL).

```sql
    SELECT @start AS dt
    UNION ALL

-- ============================================================
-- Pattern 1: YoY Monthly Comparison with Date Dimension
-- ============================================================
-- Guarantees all months appear even if no sales occurred
WITH monthly_revenue AS (
    SELECT
        dd.YearMonth,
        dd.Year,
        dd.Month,
        p.Category,
        COALESCE(SUM(oi.Quantity * oi.UnitPrice), 0) AS monthly_revenue
    FROM dbo.DateDimension dd
    CROSS JOIN (SELECT DISTINCT Category FROM dbo.Products) p
    LEFT JOIN dbo.Orders o
        ON o.OrderDate >= DATEFROMPARTS(dd.Year, dd.Month, 1)
        AND o.OrderDate < DATEADD(month, 1, DATEFROMPARTS(dd.Year, dd.Month, 1))
    LEFT JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
        AND oi.ProductId IN (SELECT ProductId FROM dbo.Products WHERE Category = p.Category)
    WHERE dd.Year >= 2022 AND dd.Year <= 2025
    GROUP BY dd.YearMonth, dd.Year, dd.Month, p.Category
)
SELECT
    m.YearMonth,
    m.Year,
    m.Month,
    m.Category,
    m.monthly_revenue,
    LAG(m.monthly_revenue, 12) OVER (
        PARTITION BY m.Category
        ORDER BY m.YearMonth
    ) AS prior_year_revenue,
    CASE
        WHEN LAG(m.monthly_revenue, 12) OVER (PARTITION BY m.Category ORDER BY m.YearMonth) IS NULL
            THEN NULL
        WHEN LAG(m.monthly_revenue, 12) OVER (PARTITION BY m.Category ORDER BY m.YearMonth) = 0
            THEN NULL
        ELSE ROUND(
            (m.monthly_revenue - LAG(m.monthly_revenue, 12) OVER (PARTITION BY m.Category ORDER BY m.YearMonth))
            / NULLIF(LAG(m.monthly_revenue, 12) OVER (PARTITION BY m.Category ORDER BY m.YearMonth), 0) * 100,
            2
        )
    END AS yoy_growth_pct
FROM monthly_revenue m
ORDER BY m.Category, m.YearMonth;

-- Pattern 2: YoY Quarterly Comparison
-- ============================================================
-- LAG(value, 4) for quarterly data (4 quarters back = 1 year)
WITH quarterly_revenue AS (
    SELECT
        dd.YearQuarter,
        dd.Year,
        dd.Quarter,
        p.Category,
        COALESCE(SUM(oi.Quantity * oi.UnitPrice), 0) AS quarterly_revenue
    FROM dbo.DateDimension dd
    CROSS JOIN (SELECT DISTINCT Category FROM dbo.Products) p
    LEFT JOIN dbo.Orders o
        ON YEAR(o.OrderDate) = dd.Year
        AND DATEPART(quarter, o.OrderDate) = dd.Quarter
    LEFT JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
    WHERE dd.Year >= 2021 AND dd.Year <= 2025
    GROUP BY dd.YearQuarter, dd.Year, dd.Quarter, p.Category
)
SELECT
    q.YearQuarter,
    q.Year,
    q.Quarter,
    q.Category,
    q.quarterly_revenue,
    LAG(q.quarterly_revenue, 4) OVER (
        PARTITION BY q.Category
        ORDER BY q.YearQuarter
    ) AS prior_year_revenue,
    ROUND(
        (q.quarterly_revenue - LAG(q.quarterly_revenue, 4) OVER (PARTITION BY q.Category ORDER BY q.YearQuarter))
        / NULLIF(LAG(q.quarterly_revenue, 4) OVER (PARTITION BY q.Category ORDER BY q.YearQuarter), 0) * 100,
        2
    ) AS yoy_growth_pct

-- ============================================================
-- Pattern 5: YoY with running total comparison
-- ============================================================
-- Compare cumulative revenue year-to-date vs same period last year
WITH ytd_revenue AS (
    SELECT
        YEAR(o.OrderDate) AS year,
        MONTH(o.OrderDate) AS month,
        SUM(oi.Quantity * oi.UnitPrice) AS monthly_revenue,
        SUM(SUM(oi.Quantity * oi.UnitPrice)) OVER (
            PARTITION BY YEAR(o.OrderDate)
            ORDER BY MONTH(o.OrderDate)
            ROWS UNBOUNDED PRECEDING
        ) AS ytd_revenue
    FROM dbo.Orders o
    INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
    WHERE o.OrderDate >= '2022-01-01'
    GROUP BY YEAR(o.OrderDate), MONTH(o.OrderDate)
)
SELECT
    y.year,
    y.month,
    y.ytd_revenue,
    LAG(y.ytd_revenue, 12) OVER (
        ORDER BY y.year, y.month
    ) AS prior_year_ytd,
    ROUND(
        (y.ytd_revenue - LAG(y.ytd_revenue, 12) OVER (ORDER BY y.year, y.month))
        / NULLIF(LAG(y.ytd_revenue, 12) OVER (ORDER BY y.year, y.month), 0) * 100,
        2
    ) AS ytd_yoy_growth_pct
FROM ytd_revenue y
ORDER BY y.year, y.month;

-- ============================================================
-- Pattern 6: YoY with total and subtotal categories
-- ============================================================
-- Use GROUP BY ROLLUP + LAG for category subtotals + grand total
WITH monthly_data AS (
    SELECT
        YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate) AS year_month,
        p.Category,
        SUM(oi.Quantity * oi.UnitPrice) AS revenue
    FROM dbo.Orders o
    INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
    INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
    WHERE o.OrderDate >= '2022-01-01'
    GROUP BY ROLLUP (YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate), p.Category)
)
SELECT
    m.year_month,
    m.Category,
    m.revenue,
    CASE WHEN m.Category IS NOT NULL AND m.year_month IS NOT NULL
        THEN LAG(m.revenue, 12) OVER (
            PARTITION BY m.Category
            ORDER BY m.year_month
        )
        ELSE NULL
    END AS prior_year_revenue
FROM monthly_data m
ORDER BY m.year_month, m.Category;
```

## Gotchas and Production Pitfalls

### Missing Periods Cause Incorrect Prior Values

**Pitfall:** Using LAG(amount, 12) without ensuring all 12 months exist for every year in every category. If January 2023 has no data, the row is absent from the result set, and LAG for February 2023 looks back 12 rows — which may be January 2022 or February 2021 depending on which rows are missing.

```sql
-- ❌ Missing months cause wrong LAG offsets
-- If no orders in Jan 2023 for Category 'Electronics', the Feb 2023 row
-- will LAG back to... whatever row is 12 positions back in the sorted result
SELECT
    YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate) AS year_month,
    p.Category,
    SUM(oi.Quantity * oi.UnitPrice) AS revenue,
    LAG(SUM(oi.Quantity * oi.UnitPrice), 12) OVER (
        PARTITION BY p.Category
        ORDER BY YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate)
    ) AS wrong_prior_year
FROM dbo.Orders o
INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
GROUP BY YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate), p.Category;
```

**Symptom:** YoY comparison shows impossible values — a category shows 200% growth when the business knows revenue was flat. The prior year value does not match any known monthly number. Data integrity is compromised silently.

**Fix:**

```sql
-- ✅ Join to date dimension to ensure all months exist
WITH all_months AS (
    SELECT DISTINCT dd.YearMonth, p.Category
    FROM dbo.DateDimension dd
    CROSS JOIN (SELECT DISTINCT Category FROM dbo.Products) p
    WHERE dd.Year >= 2022 AND dd.Year <= 2025
),
monthly_data AS (
    SELECT
        YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate) AS year_month,
        p.Category,
        SUM(oi.Quantity * oi.UnitPrice) AS revenue
    FROM dbo.Orders o
    INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
    INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
    GROUP BY YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate), p.Category
)
SELECT
    am.YearMonth,
    am.Category,
    COALESCE(md.revenue, 0) AS revenue,
    LAG(COALESCE(md.revenue, 0), 12) OVER (
        PARTITION BY am.Category
        ORDER BY am.YearMonth
    ) AS prior_year_revenue
FROM all_months am
LEFT JOIN monthly_data md ON am.YearMonth = md.year_month AND am.Category = md.Category
ORDER BY am.Category, am.YearMonth;
```

**Cost of not fixing:** A financial dashboard shows YoY growth of 450% for a product category because LAG skipped over 3 missing months and compared against a value from 15 months ago. The CFO makes an incorrect budget decision based on this data. The error is caught 2 weeks later during audit, requiring a restatement.

### ORDER BY String Date Sorts Lexicographically

**Pitfall:** Using VARCHAR year_month like '2024-01' in ORDER BY. Lexicographic order works by accident for YYYY-MM but fails for YYYY-M or MM/YYYY or other formats.

```sql
-- ❌ String ordering — wrong for non-standard formats
-- '2024-1', '2024-10', '2024-11', '2024-12', '2024-2' — February after December!
SELECT FORMAT(o.OrderDate, 'yyyy-M') AS year_month, /* wrong format */
       SUM(oi.Quantity * oi.UnitPrice) AS revenue,
       LAG(SUM(oi.Quantity * oi.UnitPrice), 12) OVER (ORDER BY FORMAT(o.OrderDate, 'yyyy-M'))
FROM dbo.Orders o
INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
GROUP BY FORMAT(o.OrderDate, 'yyyy-M');
```

**Symptom:** LAG retrieves the wrong prior row because ORDER BY sorts '2024-10' before '2024-2'. The YoY values are completely wrong.

**Fix:**

```sql
-- ✅ Use integer year_month: 202401, 202402, ..., 202412
SELECT YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate) AS year_month,
       SUM(oi.Quantity * oi.UnitPrice) AS revenue,
       LAG(SUM(oi.Quantity * oi.UnitPrice), 12) OVER (ORDER BY YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate))
FROM dbo.Orders o
INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
GROUP BY YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate);

-- ✅ Or use the first day of month as DATE type
SELECT DATEFROMPARTS(YEAR(o.OrderDate), MONTH(o.OrderDate), 1) AS month_start,
       SUM(oi.Quantity * oi.UnitPrice) AS revenue
FROM dbo.Orders o
INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
GROUP BY DATEFROMPARTS(YEAR(o.OrderDate), MONTH(o.OrderDate), 1);
```

**Cost of not fixing:** A report showing YoY growth by month sorts February after December. The Q1 report shows impossible growth numbers. The error takes 3 hours to debug because the sorting looks correct in the first few rows (Jan, Feb, Mar, Apr, May).

### Division by Zero When Prior Is Zero

**Pitfall:** Computing `(current - prior) / prior * 100` without handling the case where prior is zero. SQL Server throws a divide-by-zero error.

```sql
-- ❌ Division by zero when prior_year_revenue = 0
SELECT revenue,
       LAG(revenue, 12) OVER (ORDER BY year_month) AS prior,
       (revenue - LAG(revenue, 12) OVER (ORDER BY year_month))
       / LAG(revenue, 12) OVER (ORDER BY year_month) * 100 AS yoy  -- Error!
FROM monthly_data;
```

**Symptom:** `Divide by zero error encountered.` The entire query fails.

**Fix:**

```sql
-- ✅ Use NULLIF to convert zero to NULL, avoiding division by zero
SELECT revenue,
       LAG(revenue, 12) OVER (ORDER BY year_month) AS prior,
       ROUND(
           (revenue - LAG(revenue, 12) OVER (ORDER BY year_month))
           / NULLIF(LAG(revenue, 12) OVER (ORDER BY year_month), 0) * 100,
           2
       ) AS yoy_growth_pct
FROM monthly_data;

-- ✅ Or use CASE for explicit handling
SELECT revenue,
       prior,
       CASE
           WHEN prior IS NULL OR prior = 0 THEN NULL
           ELSE ROUND((revenue - prior) / prior * 100, 2)
       END AS yoy_growth_pct
FROM monthly_data;
```

**Cost of not fixing:** A nightly ETL job that computes YoY growth fails at 3 AM because a category had zero revenue in a month. The failure cascades to downstream reports, and the morning executive dashboard shows stale data. The data team spends 2 hours diagnosing.

### LAG Default Parameter Assumes Contiguous Data

**Pitfall:** Using LAG(amount, 12, 0) to return 0 for missing prior periods assumes the only reason for NULL is the first year boundary. If a category is new and has only 6 months of data, LAG returns 0 for all months, making it appear that prior year had zero revenue when really there is no comparison possible.

```sql
-- ❌ LAG default = 0 masks missing data vs new product launch
SELECT year_month, category, revenue,
       LAG(revenue, 12, 0) OVER (PARTITION BY category ORDER BY year_month) AS prior,
       ROUND((revenue - 0) / NULLIF(0, 0) * 100, 2) AS yoy  -- Division by zero even with 0 default!
FROM monthly_data;
```

**Symptom:** A new product category launched in June 2024 shows infinite or NULL growth for all months because LAG(..., 12, 0) returns 0 and NULLIF(0, 0) returns NULL.

**Fix:**

```sql
-- ✅ NULL is correct — it means "not applicable" not "zero"
-- Handle NULL at the application layer as "New" or "N/A"
SELECT year_month, category, revenue,
       LAG(revenue, 12) OVER (PARTITION BY category ORDER BY year_month) AS prior_year_revenue,
       CASE
           WHEN LAG(revenue, 12) OVER (PARTITION BY category ORDER BY year_month) IS NULL
               THEN NULL  -- No prior data available
           WHEN LAG(revenue, 12) OVER (PARTITION BY category ORDER BY year_month) = 0
               THEN NULL  -- Avoid division by zero
           ELSE ROUND(
               (revenue - LAG(revenue, 12) OVER (PARTITION BY category ORDER BY year_month))
               / LAG(revenue, 12) OVER (PARTITION BY category ORDER BY year_month) * 100,
               2
           )
       END AS yoy_growth_pct
FROM monthly_data;
```

**Cost of not fixing:** A new product line shows 9999% growth on the dashboard because prior year was NULL/null/0. The sales team celebrates incorrectly. When the actual data is reviewed, trust in the reporting system is eroded.

### LAG Across Multiple Partitions Is Expensive

**Pitfall:** Running LAG(value, 12) OVER (PARTITION BY category, subcategory, product ORDER BY year_month) on a query with thousands of small partitions. Each partition requires the Window Spool to manage separate sort orders. The spool grows with the number of partitions.

```sql
-- ❌ 10K product-level partitions — Window Spool must manage each separately
SELECT year_month, category, subcategory, product_id, revenue,
       LAG(revenue, 12) OVER (
           PARTITION BY category, subcategory, product_id  -- 10K+ partitions
           ORDER BY year_month
       ) AS prior_year_revenue
FROM monthly_data;
```

**Symptom:** The execution plan shows a Window Spool with 10,000+ sort runs. The spool writes and reads hundreds of MB to tempdb. Query runs for 5+ minutes.

**Fix:**

```sql
-- ✅ Compute at a higher granularity, then join back to product level
-- Or use a single partition (no PARTITION BY) if all products are in the same domain
SELECT year_month, category, subcategory, product_id, revenue,
       LAG(revenue, 12) OVER (
           PARTITION BY category  -- coarser partition, fewer spool runs
           ORDER BY year_month
       ) AS prior_year_revenue
FROM monthly_data;

-- ✅ Or batch by category in separate queries
```

**Cost of not fixing:** A product-level YoY report takes 15 minutes to run. The report is abandoned and users resort to exporting raw data to Excel, which takes even longer.

### Spoken Answers

**Q: What is the YoY comparison pattern with LAG, and what problem does it solve?**

> **Average answer:** LAG lets you access the previous row in a result set. For YoY, you use LAG(amount, 12) to go back 12 rows for monthly data. You compute the percentage difference between current and previous values.

> **Great answer:** The YoY pattern solves the business need of comparing a metric in the current period to the same period one year ago — answering "are we growing?" The core SQL uses LAG(value, N) OVER(PARTITION BY group ORDER BY period) where N is the number of periods in one year: 12 for monthly, 4 for quarterly, 1 for yearly. The critical engineering detail is that LAG is a positional offset — it goes back N rows in the sorted result, not N time units. This means every period must have a row in the dataset, or the offset lands on the wrong row. The fix is a date dimension table that guarantees contiguous periods via a CROSS JOIN + LEFT JOIN. The growth computation needs NULLIF(prior, 0) to avoid division by zero, and the first year of data will always have NULL for prior (no comparison possible). Performance-wise, LAG is better than a self-join because it scans the data once instead of twice — the Window Spool writes the aggregated data to tempdb and reads it back, which is typically 1.5-2x faster than re-scanning the base tables. In my production experience, adding a date dimension join to fill missing months added about 5% overhead to the query but eliminated the class of bugs where LAG returned wrong values for months that followed missing data.

---

**Q: Compare LAG(value, 12) for monthly data vs LAG(value, 4) for quarterly — what changes?**

> **Great answer:** The pattern is structurally identical but the offset changes: monthly data has 12 periods per year, so LAG(value, 12) goes back one year; quarterly data has 4 periods per year, so LAG(value, 4) goes back one year. The engineering implications are: quarterly data is less likely to have missing periods (only 4 data points per year vs 12), so the date dimension safety net is less critical. Quarterly data also sorts faster — fewer rows to spool. The ORDER BY column changes from year_month (YYYYMM) to year_quarter (YYYYQ). If you have both monthly and quarterly queries, consider a date dimension with both YearMonth and YearQuarter columns so the ORDER BY is always on a proper integer key. In terms of execution cost, quarterly LAG is about 3x faster than monthly because there are 3x fewer rows to sort and spool. For a data set spanning 5 years: 60 monthly rows vs 20 quarterly rows per group.

---

**Q: How does this pattern behave at scale — 10 years of daily data across 1,000 product categories?**

> **Great answer:** At this scale, there are several issues that must be addressed. First, the offset changes: daily data requires LAG(value, 365) for standard year comparison, but this breaks on leap years (366 days). The correct approach is either LAG(value, 365) and accept slight drift on leap years, or use a date dimension and a different pattern entirely (JOIN on date = DATEADD(year, -1, date)). Second, 1,000 categories × 3,650 days = 3.65M rows in the spool — the Window Spool writes all of these to tempdb. The sort required for the LAG computation can spill if memory grant is insufficient. The production fix is to batch by category — process 50 categories per batch — to keep the spool manageable. Third, at this scale, consider indexed views or materialised aggregates rather than computing YoY on the fly from the raw Orders table. A monthly materialised table reduces 3.65M rows to 12,000 (1,000 categories × 12 months). The fourth observation: at this cardinality, the self-join approach may actually be competitive if both sides are pre-aggregated and indexed — the optimiser can use a Merge Join on the ordered year_month keys. I've seen cases where self-join outperforms LAG at very large scale because the Window Spool's single-threaded sort becomes a bottleneck. The key is to test both approaches at the target data volume with realistic statistics.

### Application Checklist

- [ ] Data is aggregated by a period column (year_month, year_quarter) that is a proper integer or date
- [ ] Every period has a row for every group (use date dimension join if not)
- [ ] ORDER BY uses an integer period key, not a formatted string
- [ ] NULLIF handles division by zero for prior-year revenue of 0
- [ ] First year of data correctly shows NULL for prior year (not 0)
- [ ] PARTITION BY matches the business grouping (category, region, product)
- [ ] The offset value matches the number of periods per year (12 monthly, 4 quarterly, 1 yearly)
- [ ] Missing periods are eliminated via date dimension or COALESCE

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Single table scan vs self-join's two scans|Window Spool writes intermediate results to tempdb|
|Cleaner, more maintainable SQL|OFFSET is positional — missing periods break correctness|
|Faster execution on most datasets|Requires a date dimension for production reliability|
|Single pass aggregation + LAG|Sort cost dominates the execution plan|

### Scale Thresholds

- **< 12 months, single group**: Simple LAG works trivially. Negligible cost.
- **12-60 months, < 50 groups**: LAG is ideal. Spool fits in memory. Query < 100ms.
- **60-120 months, 50-500 groups**: Spool may hit tempdb if memory grant is small. Create a covering index for the aggregation.
- **> 5 years, > 1,000 groups**: Consider materialising monthly aggregates to a summary table. LAG over 1M+ spool rows can be expensive.
- **Daily data, long range**: LAG(value, 365) is unreliable — use date-based JOIN instead.

### Query Challenges

**Challenge 1 — Write the YoY monthly comparison query**

The Sales team needs a report showing total revenue by category for each month since January 2022, comparing each month to the same month in the prior year. The report must show: YearMonth, Category, Revenue, PriorYearRevenue, and YoY growth percentage. Some categories have missing months (no sales). Write the query.

<details>
<summary>Solution</summary>

```sql
WITH all_months AS (
    SELECT DISTINCT dd.YearMonth, p.Category
    FROM dbo.DateDimension dd
    CROSS JOIN (SELECT DISTINCT Category FROM dbo.Products) p
    WHERE dd.Year >= 2022 AND dd.Year <= 2025
),
monthly_revenue AS (
    SELECT
        YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate) AS year_month,
        p.Category,
        SUM(oi.Quantity * oi.UnitPrice) AS revenue
    FROM dbo.Orders o
    INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
    INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
    WHERE o.OrderDate >= '2022-01-01'
    GROUP BY YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate), p.Category
)
SELECT
    am.YearMonth,
    am.Category,
    COALESCE(mr.revenue, 0) AS Revenue,
    LAG(COALESCE(mr.revenue, 0), 12) OVER (
        PARTITION BY am.Category
        ORDER BY am.YearMonth
    ) AS PriorYearRevenue,
    CASE
        WHEN LAG(COALESCE(mr.revenue, 0), 12) OVER (PARTITION BY am.Category ORDER BY am.YearMonth) IS NULL THEN NULL
        WHEN LAG(COALESCE(mr.revenue, 0), 12) OVER (PARTITION BY am.Category ORDER BY am.YearMonth) = 0 THEN NULL
        ELSE ROUND(
            (COALESCE(mr.revenue, 0)
                - LAG(COALESCE(mr.revenue, 0), 12) OVER (PARTITION BY am.Category ORDER BY am.YearMonth))
            / LAG(COALESCE(mr.revenue, 0), 12) OVER (PARTITION BY am.Category ORDER BY am.YearMonth) * 100,
            2
        )
    END AS YoYGrowthPct
FROM all_months am
LEFT JOIN monthly_revenue mr ON am.YearMonth = mr.year_month AND am.Category = mr.Category
ORDER BY am.Category, am.YearMonth;
```

**Execution plan:** [Index Scan DateDimension] → [Hash Match Cross Join] → [Hash Match Aggregate] → [Sort] → [Window Spool (LAG)] → [SELECT]
**EF Core equivalent:** Raw SQL via `SqlQueryRaw<YoYResult>` — LAG is not translatable.

```sql
    GROUP BY YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate), p.Category
)
SELECT m.year_month, m.Category, m.revenue,
    LAG(m.revenue, 12) OVER (PARTITION BY m.Category ORDER BY m.year_month) AS prior
FROM monthly_revenue m
ORDER BY m.Category, m.year_month;

-- Execution plan:
-- [Clustered Index Scan Orders] → [Hash Match (Inner Join)] → [Hash Match (Inner Join)]
-- → [Hash Match Aggregate] → [Sort] (40%) → [Window Spool] (25%) → [SELECT]
-- Estimated cost: 28.5  |  Logical reads: ~8,050
```

Why does the plan have a Sort before the Window Spool even though the data comes from a Hash Aggregate? What would eliminate the Sort?

<details>
<summary>Solution</summary>

**Why the Sort:** The Hash Aggregate produces output in hash-bucket order, not in the ORDER BY order required by the OVER clause (PARTITION BY Category, ORDER BY year_month). The Window Spool requires sorted input — it must be able to process rows sequentially within each partition. Since the Hash Aggregate does not preserve order, the Sort operator is inserted to reorganise the data.

**To eliminate the Sort:** Create indexes on the base tables that allow the optimiser to use a Stream Aggregate instead of Hash Aggregate. The Stream Aggregate preserves input order. The required index is:
```sql
CREATE INDEX IX_OrderItems_OrderId_ProductId ON dbo.OrderItems (OrderId, ProductId)
    INCLUDE (Quantity, UnitPrice);
```
With this index, the join to OrderItems can use a Merge Join instead of Hash Match, preserving the order from Orders (if scanned in OrderDate order). However, since the GROUP BY is on year_month (derived from OrderDate), the Sort may still be needed unless the date parts can be ordered.

In practice, the Sort on aggregated data (e.g., 3,000 rows: 36 months × 80 categories) is trivial — the Sort accounts for 40% of cost but the total cost is only 28.5. The Sort is on ~3,000 rows, which takes ~5ms. Eliminating it is not worth the index overhead.

</details>

**Challenge 4 — Diagnose the performance problem**

A YoY report runs for 15 minutes on a 200M row Orders table with 5,000 product categories and 10 years of daily data. The query uses:

```sql
SELECT
    o.OrderDate,
    p.Category,
    SUM(oi.Quantity * oi.UnitPrice) AS daily_revenue,
    LAG(SUM(oi.Quantity * oi.UnitPrice), 365) OVER (
        PARTITION BY p.Category
        ORDER BY o.OrderDate
    ) AS prior_year_daily_revenue,
    ...
FROM dbo.Orders o
INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
WHERE o.OrderDate >= '2015-01-01'
GROUP BY o.OrderDate, p.Category
ORDER BY p.Category, o.OrderDate;
```

What makes this query slow, and how would you fix it?

<details>
<summary>Solution</summary>

**Root cause:** Three problems compound:
1. **LAG(value, 365) for daily data** — Daily data has 365 rows per year, but LAG(value, 365) is positional. Leap years (366 days) break the offset. Even ignoring that, sorting 200M rows × 5,000 categories is astronomical.
2. **5,000 categories × ~3,650 days = 18.25M aggregated rows** — The Window Spool must sort all 18.25M rows in a single sort. This spills massively to tempdb.
3. **GROUP BY on date** — Aggregating 200M orders to 18.25M daily rows is already expensive, then the Window Spool sorts all 18.25M rows.

**Fixes:**

```sql
-- Option A: Monthly aggregation (reduce rows 30x)
WITH monthly_revenue AS (
    SELECT
        YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate) AS year_month,
        p.Category,
        SUM(oi.Quantity * oi.UnitPrice) AS monthly_revenue
    FROM dbo.Orders o
    INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
    INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
    WHERE o.OrderDate >= '2015-01-01'
    GROUP BY YEAR(o.OrderDate) * 100 + MONTH(o.OrderDate), p.Category
)
SELECT year_month, Category, monthly_revenue,
       LAG(monthly_revenue, 12) OVER (PARTITION BY Category ORDER BY year_month) AS prior
FROM monthly_revenue
ORDER BY Category, year_month;

-- Option B: Batch by year (reduce sort partitions)
-- Run one query per year, process 1/10th the data at a time

-- Option C: Materialised monthly summary table
CREATE TABLE dbo.MonthlyRevenue (
    YearMonth INT NOT NULL,
    Category VARCHAR(50) NOT NULL,
    Revenue DECIMAL(18,2) NOT NULL DEFAULT 0,
    CONSTRAINT PK_MonthlyRevenue PRIMARY KEY (YearMonth, Category)
);

-- Populate via nightly job, then LAG over 2,160 rows (12 months × 180 categories)
-- instead of 18.25M rows
```

**Expected improvement:** Monthly aggregation reduces rows from 18.25M to ~60K (5,000 categories × 12 months). The Window Spool sorts 60K rows instead of 18.25M. Execution time drops from 15 minutes to ~30 seconds.

</details>

**Challenge 5 — Design the YoY reporting infrastructure**

**Scenario:** A SaaS company needs a customer-facing dashboard that shows monthly active users (MAU), revenue, and churn rate — each with YoY comparison. The data warehouse has:
- `FactUsage`: 500M rows, columns: DateKey, CustomerId, SessionCount
- `FactRevenue`: 50M rows, columns: DateKey, CustomerId, Amount
- `DimDate`: 20 years of daily data, columns: DateKey, YearMonth, YearQuarter, etc.
- `DimCustomer`: 100K customers, columns: CustomerId, Plan, CohortMonth

The dashboard must show 3 years of monthly data with YoY for each metric. Queries must complete in under 3 seconds. Design the solution.

<details>
<summary>Solution</summary>

**Architecture:**

```sql
-- Step 1: Materialise monthly aggregates into a summary table
CREATE TABLE dbo.MonthlyMetrics (
    YearMonth INT NOT NULL,
    Plan VARCHAR(50) NOT NULL,
    MAU INT NOT NULL DEFAULT 0,
    Revenue DECIMAL(18,2) NOT NULL DEFAULT 0,
    ChurnedCustomers INT NOT NULL DEFAULT 0,
    CONSTRAINT PK_MonthlyMetrics PRIMARY KEY (YearMonth, Plan)
);

-- Step 2: Nightly ETL to populate the summary
INSERT INTO dbo.MonthlyMetrics (YearMonth, Plan, MAU, Revenue, ChurnedCustomers)
SELECT
    dd.YearMonth,
    dc.Plan,
    COUNT(DISTINCT fu.CustomerId) AS MAU,
    COALESCE(SUM(fr.Amount), 0) AS Revenue,
    COUNT(DISTINCT CASE WHEN ... THEN dc.CustomerId END) AS ChurnedCustomers
FROM dbo.DimDate dd
CROSS JOIN (SELECT DISTINCT Plan FROM dbo.DimCustomer) plans
LEFT JOIN dbo.FactUsage fu ON dd.DateKey = fu.DateKey
    AND fu.CustomerId IN (SELECT CustomerId FROM dbo.DimCustomer WHERE Plan = plans.Plan)
LEFT JOIN dbo.FactRevenue fr ON dd.DateKey = fr.DateKey
    AND fr.CustomerId IN (SELECT CustomerId FROM dimCustomer WHERE Plan = plans.Plan)
WHERE dd.Year >= 2022
GROUP BY dd.YearMonth, dc.Plan;

-- Step 3: YoY query over the summary table (sub-second performance)
SELECT
    mm.YearMonth,
    mm.Plan,
    mm.MAU,
    mm.Revenue,
    LAG(mm.MAU, 12) OVER (PARTITION BY mm.Plan ORDER BY mm.YearMonth) AS PriorMAU,
    LAG(mm.Revenue, 12) OVER (PARTITION BY mm.Plan ORDER BY mm.YearMonth) AS PriorRevenue,
    ROUND((mm.MAU - LAG(mm.MAU, 12) OVER (PARTITION BY mm.Plan ORDER BY mm.YearMonth))
        / NULLIF(LAG(mm.MAU, 12) OVER (PARTITION BY mm.Plan ORDER BY mm.YearMonth), 0) * 100, 1) AS MAUYoY
FROM dbo.MonthlyMetrics mm
ORDER BY mm.Plan, mm.YearMonth;
```

**Expected performance:** The summary table has ~360 rows (3 years × 10 plans × 12 months). The LAG query completes in < 50ms with 2 logical reads. The nightly ETL takes 30 minutes but the dashboard queries are instant. This trades ETL complexity for dashboard performance.

**Index strategy:**
- Primary key `(YearMonth, Plan)` on MonthlyMetrics covers the LAG query (ordered scan by Plan, YearMonth).
- Clustered index on `(DateKey, CustomerId)` for FactUsage and FactRevenue for fast aggregation.
- Non-clustered index on `DimCustomer(Plan)` INCLUDE (CustomerId, CohortMonth) for the CROSS JOIN plan filtering.

</details>
