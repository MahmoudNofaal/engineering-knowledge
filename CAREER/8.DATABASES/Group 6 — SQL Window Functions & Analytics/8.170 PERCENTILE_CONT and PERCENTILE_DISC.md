### Where This Fits

PERCENTILE_CONT and PERCENTILE_DISC are ordered-set functions that compute percentiles — values below which a given percentage of the data falls. PERCENTILE_CONT interpolates (returns a continuous value that may not exist in the data), while PERCENTILE_DISC returns an actual discrete value from the dataset. A .NET backend engineer encounters these in salary analysis, A/B test result reporting, performance benchmarking (P50, P95, P99 latencies), price distribution analysis, and any statistical reporting where percentiles better represent data distribution than averages. The interview signal is strong: candidates must understand the CONT vs DISC difference, know that both require a full sort per partition, recognise that multiple percentiles may be folded into a single sort by the optimiser, and know that the ROW_NUMBER alternative can compute percentiles faster for large datasets.

## Core Mental Model

PERCENTILE_CONT and PERCENTILE_DISC compute the value at a specified percentile within a partition. The difference is in how they handle the boundary between two rows when the percentile falls between data points. PERCENTILE_CONT linearly interpolates: for percentile p, it finds the two rows at fractional ranks bracketing p and returns value_low + (value_high - value_low) * fractional_part. PERCENTILE_DISC returns the value of the row with the smallest cumulative distribution value that is >= p — i.e., the actual data value at or above the requested percentile. Both functions require WITHIN GROUP (ORDER BY col) for the value column and OVER (PARTITION BY group) for the grouping. The key invariant: PERCENTILE_DISC always returns a value that exists in the data, while PERCENTILE_CONT may return a value between two data points. When adjacent values are equal (e.g., multiple rows with the same salary), both functions return the same result because interpolation between equal values produces that same value.

### Classification

**For SQL topics:** PERCENTILE_CONT and PERCENTILE_DISC are ordered-set aggregate functions that can be used as window functions via the OVER clause. They belong to the inverse distribution function family — given a percentile, they return the corresponding value from the distribution. The execution plan uses a Sort + Stream Aggregate (or multiple aggregates folded into one). Neither function is SARGable — both require a full scan and sort of the partition data. The WITHIN GROUP clause is unique to these functions and cannot be used with other window functions. Both functions return the same result for every row in the partition (requiring DISTINCT for de-duplication).

## Deep Mechanics

### How the Engine Executes This

1. **Parse and bind:** SQL Server recognises PERCENTILE_CONT/DISC as an ordered-set function. The WITHIN GROUP (ORDER BY col) specifies the value column. The OVER (PARTITION BY group) specifies grouping. The percentile argument p is a numeric literal or expression between 0.0 and 1.0.

2. **Sort phase:** SQL Server sorts input by the PARTITION BY columns followed by the WITHIN GROUP ORDER BY column. For `PARTITION BY DepartmentId ORDER BY Salary`, the sort key is `DepartmentId, Salary ASC`. This sort is mandatory.

3. **Stream Aggregate phase:** For each partition:
   - **Per-row computation:** PERCENTILE_CONT computes `ROW_NUMBER / (COUNT(*) - 1)` as the fractional rank for each row. For percentile p, it finds where p falls between two ranks. If p matches a row exactly, it returns that row's value. If p falls between two rows, it interpolates.
   - **PERCENTILE_DISC:** Finds the first row whose cumulative distribution (CUME_DIST) >= p. Since CUME_DIST is computed as `ROW_NUMBER / COUNT(*)`, the function returns the value of the row at position `CEILING(p * COUNT(*))`.
   - Both functions output the percentile value for every row in the partition.

4. **Multiple percentile optimisation:** When multiple PERCENTILE_CONT/DISC calls share the same WITHIN GROUP ORDER BY and PARTITION BY, SQL Server 2016+ may fold them into a single Sort + Stream Aggregate. Each percentile is computed in the same pass over the sorted data.

**Detailed example:**

```
Partition: Salaries = [30000, 40000, 50000, 60000] (N=4)

PERCENTILE_CONT(0.5):
  Rank positions: 0, 0.333, 0.667, 1.0 (ROW_NUMBER / (N-1))
  0.5 falls between rank 0.333 (value 40000) and 0.667 (value 50000)
  Interpolation: 40000 + (50000 - 40000) * ((0.5 - 0.333) / (0.667 - 0.333))
  = 40000 + 10000 * 0.5 = 45000

PERCENTILE_DISC(0.5):
  CUME_DIST values: 0.25, 0.50, 0.75, 1.0 (RN / N)
  First value with CUME_DIST >= 0.5 is the 2nd row: 40000
  Returns 40000

PERCENTILE_CONT(0.25):
  0.25 falls between rank 0 (value 30000) and 0.333 (value 40000)
  Interpolation: 30000 + (40000 - 30000) * (0.25 / 0.333)
  = 30000 + 10000 * 0.75 = 37500

PERCENTILE_DISC(0.25):
  CUME_DIST: 0.25, 0.50, 0.75, 1.0
  First with CUME_DIST >= 0.25 is the 1st row: 30000
  Returns 30000
```

### Execution Plan Analysis

**Single PERCENTILE_CONT plan:**

```
[Index Scan IX_Employees_DepartmentId_Salary — ordered]
  → [Stream Aggregate — PERCENTILE_CONT(0.5)]
      Partition By: DepartmentId
      Order By: Salary
  → [SELECT — DISTINCT dedup]
Estimated Cost: ~28 (Sort: ~20, Aggregate: ~6)
Logical Reads: ~23,700
```

**Multiple percentiles plan (optimised — single sort):**

```
[Index Scan] → [Sort] → [Stream Aggregate]
    Stream Aggregate computes:
      PERCENTILE_CONT(0.1), PERCENTILE_CONT(0.25),
      PERCENTILE_CONT(0.5), PERCENTILE_CONT(0.75),
      PERCENTILE_CONT(0.9)
      all in ONE pass over sorted data
Estimated Cost: ~30 (only slightly more than single percentile)
Logical Reads: ~24,500
```

**Multiple percentiles plan (unoptimised — separate sorts — SQL Server < 2016):**

```
[Index Scan]
  → [Sort 1] → [Stream Aggregate — P10]
  → [Sort 2] → [Stream Aggregate — Q1]
  → [Sort 3] → [Stream Aggregate — Median]
  → [Sort 4] → [Stream Aggregate — Q3]
  → [Sort 5] → [Stream Aggregate — P90]
  → [SELECT — DISTINCT]
Estimated Cost: ~140 (5x the optimised cost!)
Logical Reads: ~120,000
```

**PERCENTILE_DISC vs CONT plan shape:**

```
Both produce identical plan shapes:
  [Sort] → [Stream Aggregate]
The difference is only in the interpolation vs discrete logic,
which is handled inside the Stream Aggregate operator.
```

### Failure Modes

**PERCENTILE_CONT returns non-existent value:** For even-sized partitions, CONT interpolates. If the business requires actual values (e.g., salary medians), use DISC instead.

**Multiple sorts for multiple percentiles:** On SQL Server versions before 2016 (or certain compatibility levels), multiple PERCENTILE_CONT calls cause separate sorts. Verify with the execution plan.

**DISTINCT required:** Both functions return the percentile value for every input row. DISTINCT is mandatory for per-group results.

**NULL handling:** PERCENTILE_CONT and PERCENTILE_DISC ignore NULLs in the WITHIN GROUP ORDER BY column. NULLs in the PARTITION BY column form their own NULL group.

-- ============================================================
-- Schema context
-- ============================================================
CREATE TABLE dbo.Employees (
    EmployeeId   INT           NOT NULL IDENTITY(1,1),
    FirstName    NVARCHAR(100) NOT NULL,
    LastName     NVARCHAR(100) NOT NULL,
    DepartmentId INT           NOT NULL,
    Salary       DECIMAL(18,2) NOT NULL,
    HireDate     DATE          NOT NULL,
    Status       VARCHAR(20)   NOT NULL DEFAULT 'Active',
    CONSTRAINT PK_Employees PRIMARY KEY CLUSTERED (EmployeeId)
);

CREATE TABLE dbo.Departments (
    DepartmentId   INT           NOT NULL IDENTITY(1,1),
    DepartmentName NVARCHAR(100) NOT NULL,
    Location       VARCHAR(100)  NULL,
    Budget         DECIMAL(18,2) NOT NULL DEFAULT 0,
    CONSTRAINT PK_Departments PRIMARY KEY CLUSTERED (DepartmentId)
);

CREATE INDEX IX_Employees_DepartmentId_Salary
    ON dbo.Employees (DepartmentId, Salary)
    INCLUDE (Status);

-- ============================================================
-- Pattern 1: CONT vs DISC comparison
-- ============================================================
SELECT DISTINCT
    e.DepartmentId,
    d.DepartmentName,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Median_Continuous,
    PERCENTILE_DISC(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Median_Discrete,

WHERE e.Status = 'Active'

-- Pattern 2: Full percentile distribution (P1-P99)
-- ============================================================
-- Compute deciles, quartiles, and key percentiles
SELECT DISTINCT
    e.DepartmentId,
    d.DepartmentName,
    COUNT(*) OVER (PARTITION BY e.DepartmentId) AS EmployeeCount,
    PERCENTILE_CONT(0.01) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS P01,
    PERCENTILE_CONT(0.05) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS P05,
    PERCENTILE_CONT(0.10) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS P10,
    PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Q1,
    PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Median,
    PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Q3,
    PERCENTILE_CONT(0.90) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS P90,
    PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS P95,
    PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS P99
FROM dbo.Employees e
INNER JOIN dbo.Departments d ON e.DepartmentId = d.DepartmentId
WHERE e.Status = 'Active'
ORDER BY d.DepartmentName;

-- ============================================================
-- Pattern 3: Interquartile Range (IQR) and outlier detection
-- ============================================================
WITH dept_percentiles AS (
    SELECT DISTINCT
        e.DepartmentId,
        PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS Q1,
        PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS Q3
    FROM dbo.Employees e
    WHERE e.Status = 'Active'
),
dept_iqr AS (
    SELECT DepartmentId, Q1, Q3,
        Q3 - Q1 AS IQR,
        Q1 - 1.5 * (Q3 - Q1) AS LowerFence,
        Q3 + 1.5 * (Q3 - Q1) AS UpperFence
    FROM dept_percentiles
)
SELECT
    e.EmployeeId, e.FirstName, e.LastName,
    e.DepartmentId, d.DepartmentName,
    e.Salary,
    i.Q1, i.Q3, i.IQR,
    CASE
        WHEN e.Salary < i.LowerFence THEN 'Below Lower Fence (Outlier)'
        WHEN e.Salary > i.UpperFence THEN 'Above Upper Fence (Outlier)'
        ELSE 'Within Range'
    END AS OutlierStatus
FROM dbo.Employees e
INNER JOIN dbo.Departments d ON e.DepartmentId = d.DepartmentId
INNER JOIN dept_iqr i ON e.DepartmentId = i.DepartmentId
WHERE e.Status = 'Active'
ORDER BY d.DepartmentName, e.Salary;

-- ============================================================
-- Pattern 4: P50, P95, P99 latency analysis
-- ============================================================
-- For API response time analysis
SELECT DISTINCT
    r.Endpoint,
    r.DateBucket,
    COUNT(*) OVER (PARTITION BY r.Endpoint, r.DateBucket) AS RequestCount,
    PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY r.DurationMs)
        OVER (PARTITION BY r.Endpoint, r.DateBucket) AS P50_Latency,
    PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY r.DurationMs)
        OVER (PARTITION BY r.Endpoint, r.DateBucket) AS P95_Latency,

        OVER (PARTITION BY r.Endpoint, r.DateBucket) AS P99_Latency,
    AVG(r.DurationMs) OVER (PARTITION BY r.Endpoint, r.DateBucket) AS AvgLatency
FROM dbo.ApiRequestLog r
WHERE r.DateBucket >= DATEADD(day, -1, GETUTCDATE())
ORDER BY r.Endpoint, r.DateBucket;

-- Pattern 5: PERCENTILE_DISC for exact percentiles
-- ============================================================
-- For salary band analysis — always return actual values
SELECT DISTINCT
    e.DepartmentId,
    d.DepartmentName,
    PERCENTILE_DISC(0.1) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Bottom10Pct,

        OVER (PARTITION BY e.DepartmentId) AS Q1,
    PERCENTILE_DISC(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Median,
    PERCENTILE_DISC(0.75) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Q3,
    PERCENTILE_DISC(0.9) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Top10Pct
FROM dbo.Employees e
INNER JOIN dbo.Departments d ON e.DepartmentId = d.DepartmentId
WHERE e.Status = 'Active'
ORDER BY d.DepartmentName;

-- ============================================================
-- Pattern 6: CONT vs DISC with GROUP BY (no DISTINCT)
-- ============================================================
-- Use a subquery to avoid DISTINCT
SELECT DepartmentId,
       PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY Salary) AS MedianSalary
FROM dbo.Employees
WHERE Status = 'Active'
GROUP BY DepartmentId;
-- Wait — this does NOT work! PERCENTILE_CONT cannot be used with GROUP BY directly.
-- It MUST be used with OVER() or as an ordered-set aggregate (PostgreSQL only).

-- Correct approach for per-group median:
SELECT DISTINCT
    e.DepartmentId,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary
FROM dbo.Employees e
WHERE e.Status = 'Active';

-- ============================================================
-- Pattern 7: ROW_NUMBER-based percentile alternative
-- ============================================================
-- Faster for large datasets; computes exact percentiles using discrete values
WITH employee_ranks AS (
    SELECT e.DepartmentId, e.Salary,
        COUNT(*) OVER (PARTITION BY e.DepartmentId) AS cnt,
        ROW_NUMBER() OVER (PARTITION BY e.DepartmentId ORDER BY e.Salary) AS rn
    FROM dbo.Employees e
    WHERE e.Status = 'Active'
)
SELECT
    r.DepartmentId,
    d.DepartmentName,
    -- P25: row at CEILING(0.25 * cnt)
    AVG(CASE WHEN r.rn = CEILING(0.25 * r.cnt) THEN r.Salary END) AS P25,
    -- Median: middle 1-2 rows
    AVG(CASE WHEN r.rn IN ((r.cnt + 1) / 2, (r.cnt + 2) / 2) THEN r.Salary END) AS Median,
    -- P75: row at CEILING(0.75 * cnt)
    AVG(CASE WHEN r.rn = CEILING(0.75 * r.cnt) THEN r.Salary END) AS P75,
    -- P90: row at CEILING(0.90 * cnt)
    AVG(CASE WHEN r.rn = CEILING(0.90 * r.cnt) THEN r.Salary END) AS P90,
    MAX(r.cnt) AS EmployeeCount
FROM employee_ranks r
INNER JOIN dbo.Departments d ON r.DepartmentId = d.DepartmentId
WHERE r.rn IN (
    CEILING(0.25 * r.cnt), (r.cnt + 1) / 2, (r.cnt + 2) / 2,
    CEILING(0.75 * r.cnt), CEILING(0.90 * r.cnt)
)
GROUP BY r.DepartmentId, d.DepartmentName
ORDER BY d.DepartmentName;

-- ============================================================
-- Pattern 8: Stored procedure for flexible percentile analysis
-- ============================================================
CREATE PROCEDURE dbo.GetDepartmentPercentiles
    @DepartmentId INT = NULL,
    @Percentiles NVARCHAR(100) = '0.25,0.5,0.75'  -- Comma-separated
AS
BEGIN
    SET NOCOUNT ON;

    -- Parse percentiles and build dynamic SQL
    -- (Simplified: fixed percentiles for production safety)
    SELECT DISTINCT
        e.DepartmentId,
        d.DepartmentName,
        PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS Q1,
        PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS Median,
        PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS Q3,
        PERCENTILE_DISC(0.50) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS Median_Discrete,
        COUNT(*) OVER (PARTITION BY e.DepartmentId) AS EmployeeCount
    FROM dbo.Employees e
    INNER JOIN dbo.Departments d ON e.DepartmentId = d.DepartmentId
    WHERE (e.DepartmentId = @DepartmentId OR @DepartmentId IS NULL)
      AND e.Status = 'Active'
    ORDER BY d.DepartmentName;
END;

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL: PERCENTILE_CONT and PERCENTILE_DISC are ordered-set aggregates
-- They use GROUP BY instead of OVER(PARTITION BY)
-- No DISTINCT needed — GROUP BY returns one row per group

-- PostgreSQL: PERCENTILE_CONT
SELECT department_id,
       PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) AS median
FROM employees
WHERE status = 'Active'
GROUP BY department_id;

-- PostgreSQL: PERCENTILE_DISC
SELECT department_id,
       PERCENTILE_DISC(0.5) WITHIN GROUP (ORDER BY salary) AS median
FROM employees
WHERE status = 'Active'
GROUP BY department_id;

-- PostgreSQL: Multiple percentiles — one GROUP BY
SELECT
    department_id,
    PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY salary) AS q1,
    PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY salary) AS median,
    PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY salary) AS q3
FROM employees
WHERE status = 'Active'
GROUP BY department_id;

-- PostgreSQL: Can also use as window function (same as SQL Server)
SELECT DISTINCT
    department_id,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary)
        OVER (PARTITION BY department_id) AS median
FROM employees
WHERE status = 'Active';
```

## Gotchas and Production Pitfalls

### CONT Returns Non-Existent Values for Even Partitions

**Pitfall:** Using PERCENTILE_CONT when only actual data values are meaningful. For even-sized partitions, CONT interpolates.

```sql
-- ❌ Median = 45000 for salaries [30000, 40000, 50000, 60000]
-- 45000 is not an actual salary
SELECT DISTINCT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY Salary)
        OVER () AS MedianSalary
FROM dbo.Employees;
```

**Symptom:** A compensation report shows median salary as $45,234.50. Every employee earns in $1,000 increments. HR cannot identify which employee has that salary.

**Fix:**

```sql
-- ✅ Use PERCENTILE_DISC for actual values
SELECT DISTINCT
    PERCENTILE_DISC(0.5) WITHIN GROUP (ORDER BY Salary)
        OVER () AS MedianSalary
FROM dbo.Employees;
-- Returns 40000 for [30000, 40000, 50000, 60000]
```

**Cost of not fixing:** VP of Engineering asks "which employee earns $45,234.50?" The data team cannot answer. The report is labelled "unreliable" and stakeholders lose trust.

---

### Forgetting DISTINCT Causes Massive Duplication

**Pitfall:** PERCENTILE_CONT/DISC returns the percentile value for every row in the partition. Without DISTINCT, the result set has one row per input row.

```sql
-- ❌ Returns 50,000 rows (one per employee)
SELECT e.EmployeeId, e.DepartmentId,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary
FROM dbo.Employees e;
```

**Symptom:** Query returns 50K rows when 500 are expected. Application serialises 50K records, using 50 MB of memory. Browser crashes.

**Fix:**

```sql
-- ✅ DISTINCT gives one row per department
SELECT DISTINCT e.DepartmentId,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS MedianSalary
FROM dbo.Employees e;
```

**Cost of not fixing:** API endpoint that should return 500 department medians returns 50K records. Network transfer time goes from 50ms to 2 seconds. Mobile clients time out.

---

### Multiple Percentile Calls May Not Fold Into One Sort

**Pitfall:** Assuming the optimiser folds multiple PERCENTILE_CONT calls into a single sort. On older SQL Server versions (< 2016) or with different PARTITION BY/ORDER BY clauses, each call gets its own Sort.

```sql
-- ❌ May cause 3 separate sorts if ORDER BY columns differ
SELECT DISTINCT
    e.DepartmentId,
    PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Q1,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.HireDate)  -- Different ORDER BY!
        OVER (PARTITION BY e.DepartmentId) AS MedianByHireDate,
    PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Q3
FROM dbo.Employees e;
```

**Symptom:** Execution plan shows three Sort operators. Query takes 3x longer than expected.

**Fix:**

```sql
-- ✅ Verify plan has one Sort (if ORDER BY columns match)
-- If ORDER BY differs, split into separate queries or use ROW_NUMBER approach
EXPLAIN -- or SET STATISTICS PROFILE ON
SELECT DISTINCT
    e.DepartmentId,
    PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Q1,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Median,
    PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Q3
FROM dbo.Employees e;
-- Check plan for 1 Sort (good) or 3 Sorts (bad)
```

**Cost of not fixing:** A salary analytics query with P10, P25, Median, P75, P90 runs for 30 seconds instead of 10 seconds. The reporting dashboard times out.

---

### PERCENTILE_CONT on Very Large Partitions Spills to Tempdb

**Pitfall:** Computing percentiles on a single large partition (e.g., global percentile on 10M rows). The Sort must order all 10M values, requiring ~800 MB of memory.

```sql
-- ❌ 10M rows sorted for global P50 — likely spills to tempdb
SELECT DISTINCT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY DurationMs)
        OVER () AS P50
FROM dbo.ApiRequestLog
WHERE DateBucket >= DATEADD(hour, -24, GETUTCDATE());
```

**Symptom:** Sort spill warning in execution plan. Query runs for 60 seconds instead of 5 seconds. Tempdb I/O spikes.

**Fix:**

```sql
-- ✅ Use APPROX_PERCENTILE_CONT (SQL Server 2022+) for large data
SELECT DISTINCT
    APPROX_PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY DurationMs)
        OVER () AS P50
FROM dbo.ApiRequestLog
WHERE DateBucket >= DATEADD(hour, -24, GETUTCDATE());
-- Approximate, ~1% error, O(N) without sort

-- ✅ Or use ROW_NUMBER-based approach (2-5x faster)
WITH ranked AS (
    SELECT DurationMs,
        COUNT(*) OVER () AS cnt,
        ROW_NUMBER() OVER (ORDER BY DurationMs) AS rn
    FROM dbo.ApiRequestLog
    WHERE DateBucket >= DATEADD(hour, -24, GETUTCDATE())
)
SELECT AVG(CAST(DurationMs AS DECIMAL(18,2))) AS P50
FROM ranked
WHERE rn IN ((cnt + 1) / 2, (cnt + 2) / 2);
```

**Cost of not fixing:** A P99 latency monitoring query runs for 2 minutes every hour. The monitoring dashboard shows "No Data" because the query times out at 60 seconds. The team misses a production latency regression.

### NULLs in the ORDER BY Column Are Silently Ignored

**Pitfall:** PERCENTILE_CONT and PERCENTILE_DISC ignore NULLs in the WITHIN GROUP ORDER BY column. The percentile is computed over non-NULL values only. This is usually correct, but if NULLs represent missing data that should affect the percentile, the result may be misleading.

```sql
-- ❌ Employees with NULL Salary are excluded from percentile
-- If 50% of employees have NULL salary (missing data), the median
-- is computed over only the remaining 50% — misleading!
SELECT DISTINCT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY Salary)
        OVER () AS MedianSalary
FROM dbo.Employees;
```

**Symptom:** Only 5,000 of 10,000 employees have non-NULL salaries. The median is computed over 5,000 records. The report shows "Median Salary: $55,000" but the true median for all employees (treating NULL as 0) would be different.

**Fix:**

```sql
-- ✅ Handle NULLs explicitly based on business rules
-- Option A: Exclude NULLs with WHERE (current behaviour)
SELECT DISTINCT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY Salary)
        OVER () AS MedianSalary
FROM dbo.Employees
WHERE Salary IS NOT NULL;  -- Explicit exclusion

-- Option B: Replace NULLs with COALESCE (if business rules require)
SELECT DISTINCT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY COALESCE(Salary, 0))
        OVER () AS MedianSalary
FROM dbo.Employees;
-- This includes NULLs as zeros — changes the distribution significantly

-- Option C: Report both counts
SELECT DISTINCT
    COUNT(*) OVER () AS TotalEmployees,
    COUNT(Salary) OVER () AS EmployeesWithSalary,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY Salary)
        OVER () AS MedianSalary
FROM dbo.Employees;
```

**Cost of not fixing:** A salary survey shows median of $65K based on 80% response rate. Leadership believes the company pays above market. The actual median including non-respondents (who may have lower salaries) is $55K. Budget decisions are based on incorrect data.

**Q: What is the difference between PERCENTILE_CONT and PERCENTILE_DISC?**

> **Average answer:** CONT interpolates, DISC returns an actual value.

> **Great answer:** Both compute percentile values — the value below which a given percentage of the data falls. The difference is in handling the boundary between two data points when the percentile falls between them. PERCENTILE_CONT performs linear interpolation: for a percentile p, it finds the two rows at fractional ranks bracketing p and returns a value proportionally between them. For example, the median of [10, 20, 30, 40] is (20+30)/2 = 25 — a value that does not exist in the data. PERCENTILE_DISC returns the actual data value at or above the percentile by using the CUME_DIST function: it finds the first row whose cumulative distribution is >= p. For the same data, PERCENTILE_DISC(0.5) returns 20 (the value at or above the 50th percentile). They return the same value when adjacent values at the percentile boundary are equal (e.g., [10, 20, 20, 30] — both return 20 for median), or when the exact percentile falls directly on a data point (odd-sized dataset for median). The choice between them is business-driven: PERCENTILE_CONT is better for continuous metrics like latency averages (where interpolation produces a more accurate distribution estimate), while PERCENTILE_DISC is better for discrete values like salary bands (where you need actual values).

---

**Q: When do PERCENTILE_CONT and PERCENTILE_DISC return the same value?**

> **Great answer:** They return identical values in two scenarios. First, when the percentile falls exactly on a data point — for example, median of an odd-sized dataset: [10, 20, 30] has median at the 2nd row regardless of function. Second, when adjacent values at the percentile boundary are equal — for example, [10, 20, 20, 30] where the two middle values are both 20: CONT computes (20+20)/2 = 20, DISC returns 20 (first at or above 0.5). In practice, they differ only when the dataset has an even number of distinct values at the boundary. For salary data where no two salaries are identical, CONT and DISC will always differ for even-sized departments. For product prices where multiple products share the same price, they may agree. The key production insight: if you need the result to be reproducible and verifiable against the actual data, use PERCENTILE_DISC. If you need a statistically precise estimate (e.g., for regression analysis), use PERCENTILE_CONT.

---

**Q: What is the performance profile of these functions, and how can you optimise multiple percentiles?**

> **Great answer:** Both functions have identical performance characteristics — they require a full Sort of all values in each partition (O(N log N)), followed by a Stream Aggregate that computes the percentile value. The Sort is 70-80% of the total cost. For 50K rows across 500 departments, the query takes ~7 seconds on NVMe storage. The critical optimisation is for multiple percentiles: when you need P10, P25, P50, P75, P90 in the same query, the SQL Server 2016+ optimiser may fold all PERCENTILE_CONT calls into a single Sort + Stream Aggregate if the WITHIN GROUP ORDER BY and PARTITION BY are identical across all calls. I've verified this reduces cost from 5 separate sorts (35 seconds) to 1 sort (7.5 seconds) — a 4.7x improvement. You can verify the fold by checking the execution plan for a single Sort operator. If you see multiple Sorts, you can rewrite using a ROW_NUMBER-based approach with CASE expressions that compute all percentiles in one pass. For very large partitions (> 1M rows), the ROW_NUMBER approach is 2-5x faster than PERCENTILE_CONT because it filters to only the target rows rather than outputting the percentile for every input row. For global percentiles on > 10M rows, APPROX_PERCENTILE_CONT (SQL Server 2022+) uses a sketch algorithm and runs in O(N) without a sort, with ~1% error.

**Challenge 1 — Write the CONT vs DISC comparison query**

Given `Employees(DepartmentId, Salary)` with 100 employees across 5 departments, write a query that shows for each department: the department name, employee count, median using PERCENTILE_CONT, median using PERCENTILE_DISC, and average salary. Include a column showing whether CONT and DISC agree for that department.

<details>
<summary>Solution</summary>

```sql
WITH dept_percentiles AS (
    SELECT DISTINCT
        e.DepartmentId,
        d.DepartmentName,
        COUNT(*) OVER (PARTITION BY e.DepartmentId) AS EmployeeCount,
        PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS Median_CONT,
        PERCENTILE_DISC(0.5) WITHIN GROUP (ORDER BY e.Salary)
            OVER (PARTITION BY e.DepartmentId) AS Median_DISC,
        AVG(e.Salary) OVER (PARTITION BY e.DepartmentId) AS AvgSalary
    FROM dbo.Employees e
    INNER JOIN dbo.Departments d ON e.DepartmentId = d.DepartmentId
)
SELECT
    DepartmentName,
    EmployeeCount,
    Median_CONT,
    Median_DISC,
    AvgSalary,
    CASE WHEN Median_CONT = Median_DISC THEN 'Yes' ELSE 'No' END AS FunctionsAgree,
    CASE WHEN Median_DISC IS NOT NULL THEN ROUND((Median_CONT - Median_DISC) / Median_DISC * 100, 2) END AS DiffPct
FROM dept_percentiles
ORDER BY DepartmentName;
```

**Logical reads:** ~15,200 (with index on DepartmentId, Salary)
**Execution plan:** [Index Scan] → [Sort] → [Stream Aggregate (both CONT and DISC)] → [SELECT]

**Challenge 2 — Fix the performance problem**

```sql
-- This query returns P10, P25, P50, P75, P90 salaries per department.
-- On a 100K row Employees table with 1,000 departments, it takes 35 seconds.
SET STATISTICS IO ON;

SELECT DISTINCT
    e.DepartmentId,
    PERCENTILE_CONT(0.10) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS P10,
    PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS P25,
    PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS P50,
    PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS P75,
    PERCENTILE_CONT(0.90) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS P90
FROM dbo.Employees e;

-- SET STATISTICS IO: Table 'Employees'. Scan count 1, logical reads 15200
-- Worktable (sort). Scan count 5, logical reads 42500
-- Plan: Index Scan → 5 Sorts → 5 Stream Aggregates
```

Identify the problem and fix it.

<details>
<summary>Solution</summary>

**Root cause:** The execution plan shows 5 separate Sorts, one for each PERCENTILE_CONT call. This happens on SQL Server versions before 2016, or when the optimiser does not recognise the opportunity to fold them. Each sort processes 100K rows, causing 5x the necessary I/O.

**Fix:**

```sql
-- Option A: Upgrade to SQL Server 2016+ or verify compatibility level
-- If already on 2016+, check if plan has one sort. The issue may be
-- different ORDER BY or PARTITION BY (all are identical here).

-- Option B: Use ROW_NUMBER approach — single pass for all percentiles
WITH ranked AS (
    SELECT e.DepartmentId, e.Salary,
        COUNT(*) OVER (PARTITION BY e.DepartmentId) AS cnt,
        ROW_NUMBER() OVER (PARTITION BY e.DepartmentId ORDER BY e.Salary) AS rn
    FROM dbo.Employees e
)
SELECT
    r.DepartmentId,
    AVG(CASE WHEN r.rn = CEILING(0.10 * r.cnt) THEN r.Salary END) AS P10,
    AVG(CASE WHEN r.rn = CEILING(0.25 * r.cnt) THEN r.Salary END) AS P25,
    AVG(CASE WHEN r.rn IN ((r.cnt + 1) / 2, (r.cnt + 2) / 2) THEN r.Salary END) AS P50,
    AVG(CASE WHEN r.rn = CEILING(0.75 * r.cnt) THEN r.Salary END) AS P75,
    AVG(CASE WHEN r.rn = CEILING(0.90 * r.cnt) THEN r.Salary END) AS P90
FROM ranked r
WHERE r.rn IN (
    CEILING(0.10 * r.cnt), CEILING(0.25 * r.cnt),
    (r.cnt + 1) / 2, (r.cnt + 2) / 2,
    CEILING(0.75 * r.cnt), CEILING(0.90 * r.cnt)
)
GROUP BY r.DepartmentId
ORDER BY r.DepartmentId;
```

**After fix — logical reads:** ~19,400 (single pass). **Runtime:** ~8 seconds (from 35 seconds). **4.4x improvement.**

</details>

**Challenge 3 — Explain the execution plan**

```sql
-- Query:
SELECT DISTINCT
    e.DepartmentId,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Median_CONT,
    PERCENTILE_DISC(0.5) WITHIN GROUP (ORDER BY e.Salary)
        OVER (PARTITION BY e.DepartmentId) AS Median_DISC
FROM dbo.Employees e
WHERE e.Status = 'Active';

-- Plan:
-- [Clustered Index Scan Employees] → [Sort] (75%) → [Stream Aggregate] (15%) → [SELECT]
-- Cost: 45, Logical reads: ~23,700
```

Why does the plan show one Sort when there are two percentile functions? How would the plan differ if the ORDER BY columns were different (e.g., Salary for CONT and HireDate for DISC)?

<details>
<summary>Solution</summary>

**Why one Sort:** Both PERCENTILE_CONT(0.5) and PERCENTILE_DISC(0.5) share the same WITHIN GROUP ORDER BY (Salary) and the same PARTITION BY (DepartmentId). The SQL Server 2016+ optimiser recognises this and folds both into a single Sort + Stream Aggregate. The Stream Aggregate computes both percentiles in the same pass over the sorted data. The cost of two percentiles is nearly identical to one — the Sort dominates at 75%.

**With different ORDER BY columns:** If CONT used ORDER BY Salary and DISC used ORDER BY HireDate, the optimiser cannot fold them. The plan would show:
```
[Clustered Index Scan]
  → [Sort 1 (DepartmentId, Salary)] → [Stream Aggregate — CONT]
  → [Sort 2 (DepartmentId, HireDate)] → [Stream Aggregate — DISC]
  → [SELECT]
```
Estimated cost: ~90 (2x the sort). Two sorts process 100K rows each.

**To verify folding in production:**
- Check the execution plan for number of Sort operators
- Use `SET STATISTICS TIME ON` — one Sort shows one set of CPU/elapsed times
- Query `sys.dm_exec_query_stats` for plan comparison

-- Option A: Use PERCENTILE_DISC for discrete (actual) latencies
SELECT DISTINCT
    r.Endpoint,
    PERCENTILE_DISC(0.50) WITHIN GROUP (ORDER BY r.DurationMs)
        OVER (PARTITION BY r.Endpoint) AS P50,
    PERCENTILE_DISC(0.95) WITHIN GROUP (ORDER BY r.DurationMs)

        OVER (PARTITION BY r.Endpoint) AS P99

WHERE r.DateBucket >= DATEADD(hour, -1, GETUTCDATE());

-- Option B: Keep CONT and add a note that values may be interpolated
-- P95 = 5023ms (interpolated — no single request took exactly this time)

This is standard for latency monitoring — it produces a smoother distribution estimate than picking one actual request's latency. If you need to know the actual latency of the request at the 95th percentile boundary, we can use PERCENTILE_DISC instead, but that will show 4,900ms, which under-reports the true 95th percentile."

**Challenge 5 — Design the latency percentile infrastructure**

**Scenario:** A SaaS platform handles 500M API requests per day across 200 endpoints. The SRE team needs a real-time dashboard showing P50, P95, and P99 latencies per endpoint, updated every minute. The raw `ApiRequestLog` table grows by 5,000 rows/second. Queries on raw data take > 30 seconds. Design the solution.

<details>
<summary>Solution</summary>

**Architecture: Pre-computed minute-level latency histograms + streaming aggregation.**

```sql
-- Step 1: Minute-level aggregated latency stats
CREATE TABLE dbo.EndpointLatencyMinute (
    Endpoint      VARCHAR(200)  NOT NULL,
    MinuteBucket  DATETIME2(0)  NOT NULL,  -- Rounded to minute
    RequestCount  INT           NOT NULL,
    AvgLatencyMs  FLOAT         NOT NULL,
    MinLatencyMs  FLOAT         NOT NULL,
    MaxLatencyMs  FLOAT         NOT NULL,
    -- Pre-computed percentiles using ROW_NUMBER-based approach
    P50LatencyMs  FLOAT         NOT NULL,
    P95LatencyMs  FLOAT         NOT NULL,
    P99LatencyMs  FLOAT         NOT NULL,
    CONSTRAINT PK_EndpointLatencyMinute PRIMARY KEY (Endpoint, MinuteBucket)
);

-- Step 2: Every minute, aggregate the last minute's data
CREATE PROCEDURE dbo.AggregateLatencyMinute
AS
BEGIN
    SET NOCOUNT ON;

    DECLARE @MinuteBucket DATETIME2(0) = DATEADD(MINUTE,
        DATEDIFF(MINUTE, 0, GETUTCDATE()) - 1, 0);

    WITH ranked AS (
        SELECT
            r.Endpoint,
            r.DurationMs,
            COUNT(*) OVER (PARTITION BY r.Endpoint) AS cnt,
            ROW_NUMBER() OVER (PARTITION BY r.Endpoint ORDER BY r.DurationMs) AS rn
        FROM dbo.ApiRequestLog r
        WHERE r.DateBucket = @MinuteBucket
    ),
    percentiles AS (
        SELECT
            Endpoint,
            COUNT(*) AS RequestCount,
            AVG(DurationMs) AS AvgLatencyMs,
            MIN(DurationMs) AS MinLatencyMs,
            MAX(DurationMs) AS MaxLatencyMs,
            AVG(CASE WHEN rn IN ((cnt + 1) / 2, (cnt + 2) / 2) THEN DurationMs END) AS P50LatencyMs,
            AVG(CASE WHEN rn = CEILING(0.95 * cnt) THEN DurationMs END) AS P95LatencyMs,
            AVG(CASE WHEN rn = CEILING(0.99 * cnt) THEN DurationMs END) AS P99LatencyMs
        FROM ranked
        WHERE rn IN (
            (cnt + 1) / 2, (cnt + 2) / 2,
            CEILING(0.95 * cnt), CEILING(0.99 * cnt)
        )
        GROUP BY Endpoint
    )
    INSERT INTO dbo.EndpointLatencyMinute (
        Endpoint, MinuteBucket, RequestCount,
        AvgLatencyMs, MinLatencyMs, MaxLatencyMs,
        P50LatencyMs, P95LatencyMs, P99LatencyMs
    )
    SELECT
        Endpoint, @MinuteBucket, RequestCount,
        AvgLatencyMs, MinLatencyMs, MaxLatencyMs,
        P50LatencyMs, P95LatencyMs, P99LatencyMs
    FROM percentiles;
END;
```

**Dashboard query (sub-10ms):**

```sql

    MinuteBucket,

    P95LatencyMs,
    P99LatencyMs
FROM dbo.EndpointLatencyMinute
WHERE MinuteBucket >= DATEADD(MINUTE, -60, GETUTCDATE())

-- Current status (latest minute)

```
