## 🧠 PART 1 — The Core Mental Model

### The Fundamental Rule

> **`string` is an immutable reference type. Every method that appears to "modify" a string actually allocates a new string object and returns it. The original is untouched. The practical consequence is that naive concatenation in a loop is O(n²) in both time and allocations.**

### The Plain-Language Analogy

Think of a `string` as a **laminated document** — once printed and sealed, nothing can change it. When a colleague asks you to "add a paragraph," you don't peel the lamination and insert text; you photocopy the whole document, add the paragraph to the copy, and hand them the new copy. The original laminated version still exists in the filing cabinet until the cleaner (the garbage collector) disposes of it.

This analogy holds under edge cases. Calling `.ToUpper()` gives you a new laminated document — the original lowercase version is still in the filing cabinet. Calling `.Replace("old", "new")` gives you a third laminated document. If you're in a loop doing this a thousand times, you're creating a thousand documents and throwing away 999 of them — that's what `StringBuilder` solves: it's a **draft pad** where you write and erase freely, and only when you call `ToString()` do you laminate the final result into an immutable string.

> [!IMPORTANT] The Immutability Guarantee
> Every `System.String` instance is sealed against modification after construction. There is no legal C# operation that changes the characters inside an existing string. `Replace`, `ToUpper`, `Trim`, `Insert`, `Remove` — all return **new** string objects. This is not a convention. It is enforced by the runtime.

## 🔬 PART 2 — Deep Mechanics

### 2.1 String Memory Layout — What Lives Where

Strings are reference types but their character data is stored **inline inside the object**, not in a separate array. This is a special CLR optimization for string specifically.

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
string s = "Order-12345";   // 10 chars
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

HEAP — string object layout (x64):
┌───────────────────────────────────────────────────┐
│  Object header      (8 bytes: sync block index)   │
│  MethodTable ptr    (8 bytes: → string type info)  │
│  _stringLength      (4 bytes: value = 10)          │
│  _firstChar         (2 bytes: 'O' — UTF-16)        │
│  [chars inline]     'r','d','e','r','-',           │
│                     '1','2','3','4','5'             │
│  null terminator    (2 bytes: '\0' — C compat)     │
└───────────────────────────────────────────────────┘
  Total: 8 + 8 + 4 + (10+1)*2 = 42 bytes → padded to 48

KEY POINTS:
• Characters are UTF-16 (2 bytes each)
• Length field is stored INLINE — Length is O(1), no scan
• null terminator exists for native interop — Length is authoritative, NOT '\0'
• There is NO separate char[] array — no indirection cost
• string.Empty is a single shared instance — "".Length is O(1), zero allocation
```

**Cost of common operations:**

| Operation | Cost | Why |
|---|---|---|
| `s.Length` | O(1), ~0 ns | Length field read, inline in object |
| `s[i]` (indexer) | O(1), ~1 ns | Bounds check + pointer arithmetic |
| `s1 == s2` | O(n) worst, O(1) best | Ordinal char-by-char; short-circuits on length mismatch |
| `s.Contains(sub)` | O(n·m) worst | Boyer-Moore variant in .NET 5+ |
| `s + t` (concat) | O(n+m), 1 alloc | New string allocated, both copied |
| `string.Concat(a,b,c)` | O(n), 1 alloc | Computes final length first, allocates once |
| `s.ToUpper()` | O(n), 1 alloc | New string, each char mapped |
| `s.Trim()` | O(n), 0–1 alloc | May return `this` if no whitespace |

### 2.2 String Concatenation — The O(n²) Trap and What the Compiler Does

The `+` operator on strings is the most common performance hazard in C# codebases.

```csharp
// What you write:
string result = firstName + " " + lastName + " <" + email + ">";

// What the compiler generates for EXACTLY this form (all on one line/expression):
// Compiler uses string.Concat overload — ONE allocation for the full result
string result = string.Concat(firstName, " ", lastName, " <", email, ">");
// Cost: one heap allocation sized to the sum of all parts. ~O(n) total. ✅

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// BUT: accumulation in a loop is DIFFERENT
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

// ⚠️ WRONG: Classic O(n²) loop
string report = "";
foreach (var order in orders)         // 10,000 orders
{
    report += order.ToString() + "\n"; // each += allocates a NEW string
                                       // copying ALL previous content
}
// Iteration 1: alloc 20 chars
// Iteration 2: alloc 40 chars (copies the 20 + new 20)
// Iteration 3: alloc 60 chars (copies the 40 + new 20)
// ...
// Iteration n: alloc n*20 chars
// Total allocations: n, Total bytes copied: n*(n+1)/2 * 20 = O(n²)
// For 10,000 orders: ~1 billion char copies. This is a production incident.

// What the compiler generates for +=:
// report = string.Concat(report, order.ToString(), "\n");
// Each call: allocates new string, copies all of report + new content, old report is garbage
```

**Memory picture after 3 iterations:**

```
After iteration 1:  HEAP: ["line1\n"]                        ← report points here
After iteration 2:  HEAP: ["line1\n"] ["line1\nline2\n"]     ← report points here, iter1 garbage
After iteration 3:  HEAP: ["line1\n"] ["line1\nline2\n"] ["line1\nline2\nline3\n"]
                    First two objects are unreachable — waiting for GC
                    New string copies ALL previous content every time
```

### 2.3 StringBuilder Internals — The Mutable Buffer

`StringBuilder` solves the loop allocation problem by maintaining a mutable char buffer internally and only sealing it into an immutable string when `ToString()` is called.

```csharp
// What StringBuilder looks like internally (simplified):
// System.Text.StringBuilder maintains a linked list of char[] chunks
// Each chunk has a default capacity of 16 chars; doubles on overflow

// Initial state after new StringBuilder(capacity: 256):
// ┌───────────────────────────────────────────────────┐
// │ StringBuilder                                      │
// │   _chunkChars:   char[256] (allocated, mostly ∅)  │
// │   _chunkLength:  0                                 │
// │   _chunkOffset:  0                                 │
// │   _chunkPrevious: null                             │
// └───────────────────────────────────────────────────┘

// After sb.Append("Order-") and sb.Append("12345"):
// ┌───────────────────────────────────────────────────┐
// │ StringBuilder                                      │
// │   _chunkChars:   ['O','r','d','e','r','-','1','2',│
// │                   '3','4','5', ∅, ∅, ∅, ...]      │
// │   _chunkLength:  11                                │
// └───────────────────────────────────────────────────┘
// Cost of Append: O(1) amortized — copies chars into existing array
// No new heap object created unless the buffer must grow

// ToString() seals the result — ONE allocation:
// Allocates a string of length 11, copies chars, returns it
// Cost: O(n) for n chars, 1 heap allocation
```

**StringBuilder method costs:**

```csharp
var sb = new StringBuilder(initialCapacity: 256); // 1 heap alloc: ~550 bytes

sb.Append("line");              // O(1) amortized — no alloc if capacity sufficient
sb.AppendLine("line");          // O(1) amortized — Append + Append(NewLine)
sb.AppendFormat("{0:C}", 99m);  // O(n) — formats then appends; avoids extra string alloc
sb.AppendJoin(", ", items);     // O(total) — joins without intermediate strings
sb.Insert(0, "PREFIX: ");       // O(n) — shifts existing chars right, then inserts
sb.Remove(0, 7);                // O(n) — shifts chars left
sb.Replace("old", "new");       // O(n·m) — scans and replaces all occurrences

string result = sb.ToString();  // O(n) — 1 alloc: final immutable string
```

> [!WARNING] StringBuilder is NOT Always Faster
> `StringBuilder` has an upfront cost: allocating the `StringBuilder` object plus its internal `char[]`. For joining 2–3 strings, `string.Concat()` or `$"..."` is faster because you avoid that overhead. The crossover point is roughly **4–5 concatenations or any loop**. Below that threshold, `string.Concat` wins.

### 2.4 String Comparison — The Culture Trap

String equality in C# has multiple modes, and using the wrong one is a production security and correctness hazard.

```csharp
// The six StringComparison values and what they mean:
//
// Ordinal                — byte-for-byte comparison. No culture rules. Fastest.
//                          "Straße" != "Strasse" (no case folding)
//
// OrdinalIgnoreCase      — Ordinal but ASCII letters A-Z folded to a-z. Fast.
//                          "ORDER" == "order"
//                          DOES NOT fold ß → ss. Only ASCII case folding.
//
// CurrentCulture         — Uses the thread's current locale. SLOW. Non-deterministic.
//                          Results differ on machines with different locale settings.
//
// CurrentCultureIgnoreCase — Same but case-insensitive per locale.
//
// InvariantCulture       — Uses English-like rules, deterministic across machines.
//                          Still slower than Ordinal.
//
// InvariantCultureIgnoreCase — As above, case-insensitive.

// RULES:
// • Identifiers, keys, protocol tokens, file paths, enum names → Ordinal or OrdinalIgnoreCase
// • User-visible display text that will be sorted/displayed → CurrentCulture
// • Cross-machine-consistent text (config files, serialized data) → InvariantCulture

// ⚠️ The Turkish I problem — why Ordinal matters for security:
string httpMethod = "GET";
bool isGet_wrong  = httpMethod.ToUpper() == "GET";            // ⚠️ CurrentCulture ToUpper
bool isGet_correct = string.Equals(httpMethod, "GET",
    StringComparison.OrdinalIgnoreCase);                       // ✅ Safe

// On a Turkish locale machine:
// "i".ToUpper() == "İ" (dotted capital I), NOT "I"
// "GET".ToLower() == "get" (fine), but "get".ToUpper() can differ
// The Ordinal comparison is immune to all locale-specific transformations

// == operator: uses Ordinal for string, NOT culture-aware
bool eq = "hello" == "HELLO"; // false — ordinal, case-sensitive
// This is the correct behavior for most production code
```

**Cost of comparison modes:**

| Mode | Relative Cost | Use case |
|---|---|---|
| `Ordinal` | 1× (baseline) | Keys, IDs, protocol tokens |
| `OrdinalIgnoreCase` | 1.2× | Case-insensitive keys |
| `InvariantCulture` | 3–5× | Serialized data, config |
| `CurrentCulture` | 5–20× | UI display, user-visible sort |

### 2.5 Format Specifiers — The Full Reference

Format specifiers control how values are converted to string. Every numeric type, `DateTime`, and `TimeSpan` supports them.

```csharp
// ─────────────────────────────────────────────────
// NUMERIC FORMAT SPECIFIERS
// ─────────────────────────────────────────────────
decimal price    = 1234567.89m;
int     quantity = 42;
double  ratio    = 0.8765;

// Standard specifiers:
Console.WriteLine($"{price:C}");       // $1,234,567.89  — Currency (locale-aware)
Console.WriteLine($"{price:C2}");      // $1,234,567.89  — Currency, 2 decimal places
Console.WriteLine($"{price:N}");       // 1,234,567.89   — Number with thousand separators
Console.WriteLine($"{price:N0}");      // 1,234,568       — Number, 0 decimal places (rounds)
Console.WriteLine($"{price:F2}");      // 1234567.89      — Fixed-point, 2 decimals
Console.WriteLine($"{price:E2}");      // 1.23E+006       — Scientific notation
Console.WriteLine($"{quantity:D5}");   // 00042           — Decimal, zero-padded to width 5
Console.WriteLine($"{quantity:X}");    // 2A              — Hexadecimal uppercase
Console.WriteLine($"{quantity:x4}");   // 002a            — Hex lowercase, padded to 4
Console.WriteLine($"{ratio:P1}");      // 87.7%           — Percent, 1 decimal
Console.WriteLine($"{ratio:G4}");      // 0.8765          — General (shorter of F or E)

// Custom patterns:
Console.WriteLine($"{price:#,##0.00}");          // 1,234,567.89  — explicit grouping
Console.WriteLine($"{0.005m:0.##}");              // 0.01          — suppress trailing zeros
Console.WriteLine($"{quantity:000}");             // 042           — zero-fill to width 3

// ─────────────────────────────────────────────────
// DATE/TIME FORMAT SPECIFIERS
// ─────────────────────────────────────────────────
var dt = new DateTime(2026, 6, 5, 14, 30, 0);

Console.WriteLine($"{dt:yyyy-MM-dd}");            // 2026-06-05    ISO 8601 date
Console.WriteLine($"{dt:yyyy-MM-ddTHH:mm:ssZ}");  // 2026-06-05T14:30:00Z  full ISO 8601
Console.WriteLine($"{dt:dd/MM/yyyy}");            // 05/06/2026    UK format
Console.WriteLine($"{dt:MMM dd, yyyy}");          // Jun 05, 2026
Console.WriteLine($"{dt:HH:mm}");                 // 14:30         24-hour
Console.WriteLine($"{dt:hh:mm tt}");              // 02:30 PM      12-hour with AM/PM
Console.WriteLine($"{dt:D}");                     // Thursday, June 5, 2026 (long date, locale)
Console.WriteLine($"{dt:s}");                     // 2026-06-05T14:30:00  sortable format
Console.WriteLine($"{dt:O}");                     // 2026-06-05T14:30:00.0000000 round-trip

// ─────────────────────────────────────────────────
// TIMESPAN FORMAT SPECIFIERS
// ─────────────────────────────────────────────────
var duration = TimeSpan.FromSeconds(3725.5);

Console.WriteLine($"{duration:c}");              // 1:02:05.5000000  constant (invariant)
Console.WriteLine($"{duration:hh\\:mm\\:ss}");  // 01:02:05         custom (escape colons!)
Console.WriteLine(duration.TotalSeconds);        // 3725.5
```

### 2.6 String Literal Forms — From Verbatim to Raw

```csharp
// Regular literal: escape sequences active
string path1 = "C:\\Users\\alice\\Documents\\report.pdf";   // works but noisy

// Verbatim literal (@): no escape processing (except "" for a literal quote)
string path2 = @"C:\Users\alice\Documents\report.pdf";      // cleaner

// Verbatim with embedded quote:
string json1 = @"{ ""name"": ""Alice"" }";  // "" = one quote inside verbatim

// Interpolated literal ($): embed expressions
string name = "Alice";
string greeting = $"Hello, {name}! It is {DateTime.Now:yyyy-MM-dd}.";

// Verbatim interpolated ($@): both features — common for dynamic paths
string folderPath = @"C:\Reports";
string filePath = $@"{folderPath}\{DateTime.Now:yyyyMMdd}_report.pdf";

// Raw string literals (C# 11) — no escapes, no doubling quotes
// Open with 3+ quotes, close with same count
string rawJson = """
    {
        "name": "Alice",
        "path": "C:\\Users\\alice"
    }
    """;
// Content is the text between the delimiters, dedented by leading whitespace
// Backslashes are LITERAL — perfect for JSON, regex, SQL

// Interpolated raw string:
string tableName = "Orders";
string sql = $"""
    SELECT *
    FROM {tableName}
    WHERE CreatedAt > '{DateTime.Now.AddDays(-7):yyyy-MM-dd}'
    """;

// UTF-8 string literals (C# 11): for network protocols and binary formats
// Returns ReadOnlySpan<byte>, NOT string — no allocation at runtime
ReadOnlySpan<byte> httpGet = "GET / HTTP/1.1\r\n"u8;  // ~0 ns, data is in the binary
ReadOnlySpan<byte> contentType = "Content-Type: application/json"u8;
```

**Cost comparison:**

| Literal type | Allocation at runtime | Use case |
|---|---|---|
| `"regular"` | One heap string (interned for literals) | General text |
| `@"verbatim"` | One heap string (interned for literals) | Paths, regex patterns |
| `$"interpolated"` | One heap allocation per use | Dynamic formatting |
| `$@"both"` | One heap allocation per use | Dynamic paths |
| `"""raw"""` | One heap string (interned for literals) | JSON, SQL, multi-line |
| `"text"u8` | **Zero** — data in binary segment | Binary protocols, UTF-8 output |

## 💻 PART 3 — Production Code Patterns

### Pattern 1: The Loop Accumulator — StringBuilder for Report Generation

Generating any string from a variable-length sequence of items must never use `+=` in a loop.

```csharp
// Scenario: generating a CSV export of invoices in a billing service

// ⚠️ WRONG: O(n²) concatenation in a loop
public static string BuildCsvExport_Wrong(IEnumerable<Invoice> invoices)
{
    string csv = "InvoiceId,CustomerId,Amount,Currency,Date\n";
    foreach (var inv in invoices)
    {
        // Each += allocates a new string copying ALL previous content
        csv += $"{inv.Id},{inv.CustomerId},{inv.Amount:F2},{inv.Currency},{inv.Date:yyyy-MM-dd}\n";
    }
    return csv; // For 10,000 invoices: ~500 million char copies
}

// ✅ CORRECT: StringBuilder — O(n) total, one final allocation
public static string BuildCsvExport(IEnumerable<Invoice> invoices)
{
    // Pre-size estimate: avoids mid-loop buffer doublings
    // 80 chars/row * estimated count = reasonable initial capacity
    var sb = new StringBuilder(capacity: 80 * 1024);

    sb.AppendLine("InvoiceId,CustomerId,Amount,Currency,Date");

    foreach (var inv in invoices)
    {
        // AppendFormat avoids creating an intermediate interpolated string
        sb.Append(inv.Id).Append(',')
          .Append(inv.CustomerId).Append(',')
          .AppendFormat("{0:F2}", inv.Amount).Append(',')
          .Append(inv.Currency).Append(',')
          .AppendFormat("{0:yyyy-MM-dd}", inv.Date)
          .AppendLine();
        // Each Append is O(1) amortized — appends to existing buffer
    }

    return sb.ToString(); // ONE heap allocation: the final string
}
```

### Pattern 2: The Ordinal Gate — Safe String Comparison at Service Boundaries

Any string comparison used for routing, authorization, or filtering must specify `StringComparison` explicitly. Never rely on the overload that omits it.

```csharp
// Scenario: API gateway routing by HTTP method and content type

// ⚠️ WRONG: Omitting StringComparison is ambiguous and potentially culture-sensitive
public static bool IsJsonPost_Wrong(HttpRequest request)
{
    return request.Method == "POST"                                         // ok — == is ordinal
        && request.ContentType.StartsWith("application/json");              // ⚠️ uses CurrentCulture
}

// ✅ CORRECT: All comparisons are explicit and documented
public static bool IsJsonPost(HttpRequest request)
{
    // Method token comparison: protocol-level, must be exact bytes — Ordinal
    if (!string.Equals(request.Method, "POST", StringComparison.Ordinal))
        return false;

    // Content-Type is case-insensitive per RFC 2045 — OrdinalIgnoreCase
    // We do NOT want Turkish locale converting i → İ on a server with tr-TR locale
    return request.ContentType?.StartsWith(
        "application/json", StringComparison.OrdinalIgnoreCase) == true;
}

// Pattern for building a routing table that survives locale changes:
private static readonly Dictionary<string, Func<HttpContext, Task>> _routes
    = new(StringComparer.OrdinalIgnoreCase)  // key comparison is OrdinalIgnoreCase
    {
        ["/api/orders"]  = HandleOrders,
        ["/api/payments"] = HandlePayments,
        ["/health"]       = HandleHealth,
    };

// Lookup: "/API/ORDERS" will match "/api/orders" because the comparer is OrdinalIgnoreCase
public static bool TryGetHandler(string path, out Func<HttpContext, Task>? handler)
    => _routes.TryGetValue(path, out handler);
```

### Pattern 3: The Format Envelope — Isolating Culture-Sensitive Formatting

Payment amounts, dates, and numbers must be formatted for display in the user's culture, but stored and parsed in invariant culture. Mixing these is a data corruption bug.

```csharp
// Scenario: payment service that displays amounts to users but persists them as strings

public static class MoneyFormatter
{
    // ✅ User-facing display: use user's culture so they see their local currency format
    public static string FormatForDisplay(decimal amount, string currencyCode, CultureInfo userCulture)
    {
        // Format amount in user locale: $1,234.56 (en-US) or €1.234,56 (de-DE)
        return amount.ToString("C", userCulture);
    }

    // ✅ Storage / serialization / logging: use InvariantCulture — deterministic across all servers
    public static string FormatForStorage(decimal amount)
    {
        // InvariantCulture: always "1234.56", never "1234,56" (which is fr-FR's decimal separator)
        return amount.ToString("F2", CultureInfo.InvariantCulture);
    }

    // ✅ Parsing from storage: must match the storage format — InvariantCulture
    public static decimal ParseFromStorage(string stored)
    {
        if (!decimal.TryParse(stored, NumberStyles.Any,
                CultureInfo.InvariantCulture, out var amount))
            throw new FormatException(
                $"Cannot parse '{stored}' as a decimal amount. Expected invariant format e.g. '1234.56'");
        return amount;
    }

    // ✅ Parsing user input: use user's culture (they typed it in their locale)
    public static bool TryParseUserInput(string input, CultureInfo userCulture,
        out decimal amount)
        => decimal.TryParse(input, NumberStyles.Currency, userCulture, out amount);
}

// ISO 8601 dates: always use "O" or "s" round-trip format for storage
public static class DateTimeFormatter
{
    public static string FormatForStorage(DateTime dt)
        => dt.ToString("O", CultureInfo.InvariantCulture); // "2026-06-05T14:30:00.0000000"

    public static DateTime ParseFromStorage(string s)
        => DateTime.ParseExact(s, "O", CultureInfo.InvariantCulture,
            DateTimeStyles.RoundtripKind);
}
```

### Pattern 4: The Conditional Allocator — Avoiding Unnecessary String Builds

Allocation should be conditional on whether the string is actually needed. Logging, diagnostics, and debug strings should not be built when the output channel is disabled.

```csharp
// Scenario: structured logging in an order processing service

// ⚠️ WRONG: String interpolation ALWAYS runs — even if the log level is disabled
public void ProcessOrder_Wrong(Order order)
{
    // If Debug logging is off, this string is built and immediately discarded
    _logger.LogDebug($"Processing order {order.Id} for customer {order.CustomerId} " +
                     $"with {order.Lines.Count} lines totalling {order.Total:C}");
    // ~200–500 ns of string building and ~200 bytes allocation — every call, always
}

// ✅ CORRECT: Use structured logging with deferred evaluation
public void ProcessOrder(Order order)
{
    // ASP.NET Core's ILogger checks the level BEFORE building the string
    // If Debug is disabled, the format string is never processed and args are never boxed
    _logger.LogDebug(
        "Processing order {OrderId} for customer {CustomerId} with {LineCount} lines totalling {Total}",
        order.Id,           // passed as object — but only used if level is enabled
        order.CustomerId,
        order.Lines.Count,
        order.Total);
    // Cost when Debug disabled: ~5 ns (level check only). Zero string allocation.
    // Cost when Debug enabled: normal formatting path.
}

// ✅ ALTERNATIVE: IsEnabled guard for expensive message construction
public void ProcessPaymentBatch(IEnumerable<Payment> payments)
{
    if (_logger.IsEnabled(LogLevel.Debug))
    {
        // Only enter this block when Debug logging is on
        var summary = string.Join(", ", payments.Select(p => p.Id));
        _logger.LogDebug("Processing payment batch: [{PaymentIds}]", summary);
    }
}
```

### Pattern 5: The Identifier Normalizer — Consistent Key Canonicalization

Any string used as a dictionary key or database lookup identifier must be normalized consistently. Inconsistent normalization creates duplicate data and lookup failures.

```csharp
// Scenario: customer account service where email addresses are used as identifiers

// ⚠️ WRONG: No normalization — "Alice@Example.com" and "alice@example.com" 
//           create duplicate accounts and failed logins
public class CustomerRepository_Wrong
{
    private readonly Dictionary<string, Customer> _customers = new();

    public void Register(string email, string name)
        => _customers[email] = new Customer(email, name); // raw email as key

    public Customer? FindByEmail(string email)
        => _customers.TryGetValue(email, out var c) ? c : null;
}

// ✅ CORRECT: Normalize at the boundary — one canonical form everywhere
public class CustomerRepository
{
    // OrdinalIgnoreCase comparer: "Alice@EXAMPLE.COM" == "alice@example.com"
    // No culture rules — email addresses use ASCII comparison per RFC 5321
    private readonly Dictionary<string, Customer> _customers
        = new(StringComparer.OrdinalIgnoreCase);

    public void Register(string email, string name)
    {
        // Normalize: trim whitespace, lowercase, canonical form
        // Do it ONCE at registration — all subsequent lookups work naturally
        var normalizedEmail = NormalizeEmail(email);
        if (_customers.ContainsKey(normalizedEmail))
            throw new InvalidOperationException($"Email already registered: {normalizedEmail}");
        _customers[normalizedEmail] = new Customer(normalizedEmail, name);
    }

    public Customer? FindByEmail(string email)
        => _customers.TryGetValue(email.Trim(), out var c) ? c : null;
        // OrdinalIgnoreCase on the dictionary handles case; we only need to Trim

    // Normalization logic centralized in one place — no duplication
    private static string NormalizeEmail(string email)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(email, nameof(email));
        return email.Trim().ToLowerInvariant();
        // ToLowerInvariant: ASCII lowercasing, immune to Turkish locale
        // Do NOT use ToLower() — it uses CurrentCulture and may differ by machine
    }
}
```

### Pattern 6: The Raw SQL Builder — Safe Dynamic Query Construction

Raw string literals (C# 11) eliminate the escape noise in SQL, JSON, and regex strings. Combined with interpolation, they produce readable, maintainable query code.

```csharp
// Scenario: reporting service building dynamic analytics queries

// ⚠️ WRONG: Regular string — escape-heavy, hard to maintain
public static string BuildOrderSummaryQuery_Wrong(string schema, DateTime fromDate)
{
    return $"SELECT o.Id, o.CustomerId, SUM(l.UnitPrice * l.Quantity) AS Total\r\n" +
           $"FROM [{schema}].[Orders] o\r\n" +
           $"JOIN [{schema}].[OrderLines] l ON l.OrderId = o.Id\r\n" +
           $"WHERE o.CreatedAt >= '{fromDate:yyyy-MM-dd}'\r\n" +
           $"GROUP BY o.Id, o.CustomerId\r\n" +
           $"ORDER BY Total DESC";
}

// ✅ CORRECT: Raw interpolated string — readable, no escaping needed
public static string BuildOrderSummaryQuery(string schema, DateTime fromDate)
{
    // """ delimiters: backslashes are literal, quotes don't need doubling
    // Leading whitespace before closing """ is stripped from each line
    return $"""
        SELECT o.Id, o.CustomerId, SUM(l.UnitPrice * l.Quantity) AS Total
        FROM [{schema}].[Orders] o
        JOIN [{schema}].[OrderLines] l ON l.OrderId = o.Id
        WHERE o.CreatedAt >= '{fromDate:yyyy-MM-dd}'
        GROUP BY o.Id, o.CustomerId
        ORDER BY Total DESC
        """;
    // Note: if schema comes from user input, use parameterized queries instead.
    // This pattern is safe ONLY for values that are validated/controlled internally.
}

// ✅ ALSO CORRECT: Raw string for regex patterns
// Regex patterns with backslashes are notoriously hard to read in regular strings
// ⚠️ WRONG: @"^\d{4}-\d{2}-\d{2}$" — works but verbatim still needs care
// ✅ CORRECT: raw string
private static readonly Regex DatePattern = new("""^\d{4}-\d{2}-\d{2}$""",
    RegexOptions.Compiled | RegexOptions.CultureInvariant);
```

### Pattern 7: The StringBuilder Reuse — Object Pooling for Hot-Path Formatting

For services formatting many strings in rapid succession, allocating and discarding a `StringBuilder` per operation adds measurable GC pressure. Rent from a pool or reuse across operations.

```csharp
// Scenario: notification service that formats hundreds of push notification payloads per second

// ⚠️ WRONG: New StringBuilder per notification — overhead adds up at volume
public IEnumerable<string> FormatNotifications_Wrong(IEnumerable<Notification> notifications)
{
    foreach (var n in notifications)
    {
        var sb = new StringBuilder(256); // alloc ~550 bytes per notification
        sb.Append(n.Title).Append(": ").Append(n.Body);
        if (n.ActionUrl != null) sb.Append(" → ").Append(n.ActionUrl);
        yield return sb.ToString();
        // StringBuilder and its internal char[] become garbage immediately
    }
}

// ✅ CORRECT: Reuse a single StringBuilder by calling .Clear() between uses
public IEnumerable<string> FormatNotifications(IEnumerable<Notification> notifications)
{
    // One StringBuilder allocation for the entire batch — shared across all iterations
    var sb = new StringBuilder(256);

    foreach (var n in notifications)
    {
        sb.Clear(); // Resets length to 0; does NOT release the internal buffer
                    // The buffer is reused — zero allocation after the first use

        sb.Append(n.Title).Append(": ").Append(n.Body);
        if (n.ActionUrl != null) sb.Append(" → ").Append(n.ActionUrl);

        yield return sb.ToString(); // ONE allocation per notification: the result string
    }
    // Total: 1 StringBuilder alloc + 1 string alloc per notification (unavoidable)
    // vs WRONG: 1 StringBuilder alloc + 1 string alloc per notification (same) PLUS
    //           StringBuilder GC churn (additional GC pressure from short-lived objects)
}

// ✅ BEST for extremely hot paths: Microsoft.Extensions.ObjectPool
// (See 2.41 — Performance: Zero-Allocation Patterns for the full pattern)
```

## ⚠️ PART 4 — Gotchas & Anti-Patterns

### Gotcha 1: `string.IsNullOrEmpty` vs `string.IsNullOrWhiteSpace` — Silent Whitespace-Only Bugs

Engineers use `IsNullOrEmpty` when they mean "no meaningful content" but whitespace-only strings pass the check and reach downstream logic, causing silent data corruption or confusing errors.

```csharp
// ⚠️ WRONG: Whitespace-only customer name passes the guard and gets stored
public void CreateCustomer_Wrong(string name, string email)
{
    if (string.IsNullOrEmpty(name))
        throw new ArgumentException("Name required");
    // A name of "   " (3 spaces) passes this check
    // Gets stored as "   " in the database
    // User sees a blank name in the UI
    _db.InsertCustomer(name, email);
}

// ✅ CORRECT: Use IsNullOrWhiteSpace for user-provided text fields
public void CreateCustomer(string name, string email)
{
    if (string.IsNullOrWhiteSpace(name))
        throw new ArgumentException("Name cannot be empty or whitespace.", nameof(name));

    // Also: trim before storing — "  Alice  " → "Alice"
    _db.InsertCustomer(name.Trim(), email.Trim().ToLowerInvariant());
}

// WHY: IsNullOrEmpty returns true only for null or "" (Length == 0).
// IsNullOrWhiteSpace additionally returns true for any string where all
// characters are whitespace (space, tab, newline, \r, non-breaking space, etc.).
// User-facing text fields almost always need the latter.
```

### Gotcha 2: `string.Format` Argument Index Mismatch — Silent Wrong Output

`string.Format` with positional arguments will compile and run with wrong argument order. The format string is a runtime-evaluated pattern, not a compile-time binding.

```csharp
// ⚠️ WRONG: Arguments are in the wrong order — compiles, runs, produces incorrect output
public static string FormatShipmentLabel_Wrong(string trackingId, string destination, DateTime eta)
{
    // Intended: "TRACK-001 → London, ETA: 2026-06-10"
    // Actual:   "London → TRACK-001, ETA: 2026-06-10"  — swapped 0 and 1
    return string.Format("{{0}} → {1}, ETA: {2:yyyy-MM-dd}", destination, trackingId, eta);
    //                     ^^^                ^^^
    //                     {0} = destination  {1} = trackingId  (swapped)
    // No exception. No warning. Wrong label gets printed.
}

// ✅ CORRECT: Use string interpolation — names in the format prevent argument-order bugs
public static string FormatShipmentLabel(string trackingId, string destination, DateTime eta)
{
    // Interpolation: expressions are positionally tied to their source — reordering is visible
    return $"{trackingId} → {destination}, ETA: {eta:yyyy-MM-dd}";
    // Compiler evaluates each {expr} in order — if you swap names, refactoring tools catch it
}

// WHY: string.Format uses integer indices that have no semantic connection to variable names.
// String interpolation ($"...") embeds the actual expression, so the variable name
// documents intent and refactoring tools can track renames.
// Reserve string.Format for cases where the format string is loaded from external configuration.
```

### Gotcha 3: `ToUpper()` / `ToLower()` for Comparison — The Culture-Sensitivity Trap

Converting to upper/lower before comparing looks safe but is culture-dependent. The fix is not to stop comparing case-insensitively — it's to use `StringComparison.OrdinalIgnoreCase` directly.

```csharp
// ⚠️ WRONG: ToLower() uses CurrentCulture — results differ by machine locale
public bool IsAdminUser_Wrong(string role)
{
    return role.ToLower() == "admin"; // On tr-TR locale: "I".ToLower() == "ı" (dotless i)
                                      // "ADMIN" → "admın" — does NOT equal "admin"
                                      // Your auth check silently fails for Turkish users
}

// ⚠️ ALSO WRONG: ToUpperInvariant() before comparison — creates an unnecessary allocation
public bool IsAdminUser_AlsoWrong(string role)
{
    return role.ToUpperInvariant() == "ADMIN"; // Allocates a new string just for comparison
}

// ✅ CORRECT: Pass StringComparison to the comparison method — zero allocation, correct behavior
public bool IsAdminUser(string role)
{
    return string.Equals(role, "admin", StringComparison.OrdinalIgnoreCase);
    // No allocation. No culture rules. Correct everywhere.
}

// ✅ ALSO CORRECT for StartsWith/EndsWith/Contains:
public bool IsHealthCheckPath(string path)
    => path.StartsWith("/health", StringComparison.OrdinalIgnoreCase);

// WHY: Ordinal comparison treats strings as sequences of UTF-16 code units.
// It applies no language-specific rules. "A" is always 0x0041, "a" is always 0x0061.
// OrdinalIgnoreCase folds only the ASCII range (A-Z to a-z), nothing else.
// This is deterministic across every locale, OS, and CLR version.
```

### Gotcha 4: `Split` Returns Empty Entries — Silent Parsing Errors

`string.Split` by default returns empty strings when delimiters are adjacent or at the start/end of the input. Processing those empty entries as meaningful data causes subtle parsing bugs.

```csharp
// Input from a log parsing service: CSV line with trailing comma
string csvLine = "CUST-001,Alice,alice@example.com,";

// ⚠️ WRONG: Default Split — includes trailing empty entry
string[] fields_wrong = csvLine.Split(',');
// Result: ["CUST-001", "Alice", "alice@example.com", ""]
//                                                     ^^ empty string at index 3
// Downstream code: fields_wrong[3].Trim() == "" — looks valid but means nothing

// ⚠️ ALSO WRONG: Multi-space delimiters without removing empties
string logLine = "ERROR  : Payment  failed";
string[] words_wrong = logLine.Split(' ');
// Result: ["ERROR", "", ":", "Payment", "", "failed"]
// Two empty strings from the double spaces

// ✅ CORRECT: Use StringSplitOptions.RemoveEmptyEntries
string[] fields = csvLine.Split(',', StringSplitOptions.RemoveEmptyEntries);
// Result: ["CUST-001", "Alice", "alice@example.com"]
// Trailing empty entry removed

// ✅ CORRECT: Also TrimEntries (C# 8+) to handle surrounding whitespace
string[] words = logLine.Split(' ',
    StringSplitOptions.RemoveEmptyEntries | StringSplitOptions.TrimEntries);
// Result: ["ERROR", ":", "Payment", "failed"]

// ✅ ALSO: Limit the number of parts for structured formats
string headerLine = "Content-Type: application/json; charset=utf-8";
string[] parts = headerLine.Split(':', count: 2, StringSplitOptions.TrimEntries);
// Result: ["Content-Type", "application/json; charset=utf-8"]
// count: 2 prevents splitting on the colon inside the value

// WHY: Split's default behavior preserves all parts including empty ones,
// which is technically correct but almost never what you want when parsing
// real-world user or system-generated text.
```

### Gotcha 5: `StringBuilder.ToString()` Called Multiple Times — Unexpected Behavior

`StringBuilder` is commonly misunderstood as "accumulate, then read once." Calling `ToString()` multiple times on an unmodified `StringBuilder` is actually safe and returns the same string value — but the mental model that it "empties" the builder causes incorrect code when engineers add `Clear()` in the wrong place.

```csharp
// The misunderstood mental model: "ToString() empties the StringBuilder"
// This is WRONG. ToString() copies the current content into a new string
// and leaves the StringBuilder unchanged.

var sb = new StringBuilder();
sb.Append("Invoice #").Append(12345);

string s1 = sb.ToString(); // "Invoice #12345"
string s2 = sb.ToString(); // "Invoice #12345" — same value, DIFFERENT string object

Console.WriteLine(object.ReferenceEquals(s1, s2)); // false — two allocations!
// Calling ToString() twice allocates two strings — unnecessary if s1 is sufficient

// ⚠️ WRONG: Engineer thinks Clear() is needed to "reset" ToString output
public string BuildAndReturnLabel_Wrong(string orderId)
{
    var sb = new StringBuilder();
    sb.Append("Order: ").Append(orderId);
    string result = sb.ToString();
    sb.Clear(); // ⚠️ Pointless here — sb is local, discarded anyway
                // But in shared-builder scenarios, premature Clear is a bug
    return result;
}

// ⚠️ THE ACTUAL BUG — shared builder, Clear at wrong point:
private static readonly StringBuilder _sharedBuilder = new(); // ⚠️ static! not thread-safe

public static string FormatOrderId_Broken(string orderId)
{
    _sharedBuilder.Clear();
    _sharedBuilder.Append("ORD-").Append(orderId.ToUpperInvariant());
    string result = _sharedBuilder.ToString();
    // If another thread calls this concurrently, _sharedBuilder is corrupted
    // Result may be garbage: "ORD-ORD-CUST-001" or worse
    return result;
}

// ✅ CORRECT: Never share a StringBuilder across threads without synchronization.
// Use a local StringBuilder or rent from ObjectPool<StringBuilder>.
// Clear() is correct only when REUSING a local builder across multiple items in a loop.
public static string FormatOrderId(string orderId)
{
    // Local: no sharing, no threading issues
    var sb = new StringBuilder(32);
    sb.Append("ORD-").Append(orderId.ToUpperInvariant());
    return sb.ToString(); // called once, result used, builder discarded
}

// WHY: StringBuilder is not thread-safe. Static or shared instances without
// locking are a race condition. The correct sharing strategy is ObjectPool<StringBuilder>,
// which handles thread-local acquisition and return. (See 2.41.)
```

### 5.1 Allocation Characteristics Table

| Scenario | Allocation Behavior | Approx Cost |
|---|---|---|
| `string` literal | Zero at runtime (interned by CLR) | 0 ns |
| `$"..."` interpolation, no expressions | One heap alloc — the string | ~50 ns |
| `$"Hello {name}"` one short expr | One heap alloc | ~80–150 ns |
| `"a" + "b" + "c"` (compile-time const) | Zero — compiler folds to one literal | 0 ns |
| `a + b + c` (3 variables, one statement) | One alloc — compiler uses `string.Concat` | ~100 ns |
| `s += value` in a loop, n iterations | n allocations, O(n²) total bytes copied | O(n²) |
| `new StringBuilder(cap).ToString()` | Two allocs: StringBuilder obj + result string | ~150 ns |
| `sb.Append(...)` within capacity | Zero alloc — writes to existing buffer | ~5–10 ns |
| `sb.Append(...)` exceeding capacity | One alloc — new larger `char[]` | ~50 ns |
| `sb.Clear()` | Zero alloc — sets length to 0, keeps buffer | ~1 ns |
| `string.Join(sep, items)` | One alloc — pre-computes length, allocates once | ~O(n) |

| `s.Trim()` | Zero or one alloc — may return `this` if no whitespace | ~O(n) |
| `s.Contains(sub)` no alloc | Zero alloc — pure scan, no new string | ~O(n) |

### 5.3 When to Care / When to Ignore

**When string operations cost you in production:**

- **Report generation and export pipelines** — any code that builds large strings from collections must use `StringBuilder` or `string.Join`. The O(n²) trap with `+=` is a production incident waiting to happen for n > 1,000.
- **High-throughput API serialization** — every `JsonSerializer.Serialize` produces a string. In services handling 10,000 req/sec, that's 10,000 string allocations per second from serialization alone. This is where `Utf8JsonWriter` and `ISpanFormattable` (covered in 2.35 and 2.38) earn their complexity.
- **String comparison in hot routing logic** — HTTP request routing that calls `ToLower()` before comparing allocates a new string on every request. At 50,000 req/sec that's 50,000 allocations per second just for routing. Use `StringComparison.OrdinalIgnoreCase` directly — zero allocation.
- **Log message construction** — formatting log strings that are never emitted (because the log level is off) is pure waste. Structured logging with deferred evaluation (ILogger's pattern) eliminates this entirely.
- **Parsing fixed-format text** — tokenizing CSV, splitting log lines, and parsing protocol headers in hot paths benefit from `ReadOnlySpan<char>` operations (covered in 2.35/2.38) that avoid allocating substrings.

**When string performance doesn't matter:**

- **Any code path that executes less than ~1,000 times per second** — at that frequency, even the most wasteful string concatenation adds less than 1 ms/sec overhead. Don't optimize.
- **Startup and configuration parsing** — reading config files and parsing connection strings at startup is a one-time cost. `string.Split` and `+=` are perfectly appropriate here.
- **Error message construction** — error paths are by definition not hot paths. Building detailed, readable exception messages with `$"..."` interpolation is always correct. Never compromise error message quality for performance.
- **Test code** — benchmark your tests for correctness, not speed. Use whatever string construction is most readable.

## 🎤 PART 6 — Interview Arsenal

### A. The Question Bank

---

**Q: "Why is `string` immutable in C#, and what are the practical consequences?"**

**Average Answer:** "Once a string is created you can't change it. Methods like Replace return a new string."

**Why That's Insufficient:** Doesn't explain WHY immutability was chosen or what the second-order consequences are for equality, hashing, and thread safety.

**Great Answer:**
> "Immutability in `string` is a deliberate design choice that buys three things simultaneously. First, strings are safe to share across threads with zero synchronization — because nobody can mutate them, there are no race conditions. Second, string equality can be value-based rather than reference-based — `==` compares content, not pointer identity, and this is only sensible because the content can't change after you observe it. Third, interning is possible — the CLR can deduplicate identical literals into a single heap object because there's no risk of one holder's 'change' corrupting another's view. The practical cost is that every 'modification' allocates a new string. The production implication is that the `+=` operator in a loop is O(n²) — each iteration allocates a new string copying all previous content. That's why `StringBuilder` exists: it's a mutable buffer that you seal into an immutable string via `ToString()` only when the construction is complete."

---

**Q: "When would you use `StringBuilder` over string concatenation?"**

**Average Answer:** "Use `StringBuilder` in loops because it's more efficient for lots of concatenations."

**Why That's Insufficient:** Doesn't quantify the threshold, doesn't explain why single-statement concatenation is already optimized by the compiler, and doesn't address the StringBuilder overhead for small cases.

**Great Answer:**
> "There are two scenarios where `StringBuilder` is unambiguously correct. First, any loop — because `+=` in a loop is O(n²) and `StringBuilder.Append` is O(1) amortized. This matters even for a few hundred iterations. Second, when building a string from more than about four to five dynamic parts and those parts aren't all on the same expression line. For the other case — a single expression like `a + b + c + d` — the C# compiler already converts that to `string.Concat(a, b, c, d)` which pre-computes the total length and allocates exactly once, so `StringBuilder` would actually be slower there because you're paying for the `StringBuilder` object allocation plus its buffer allocation before you get to the same result. The crossover is roughly: loop or dynamic accumulation → `StringBuilder`; fixed small number of parts in one expression → `+` or `$''`."

---

**Q: "What is the difference between `StringComparison.Ordinal` and `StringComparison.OrdinalIgnoreCase` vs the culture-aware comparisons? When does it matter in production?"**

**Average Answer:** "Ordinal compares byte-by-byte, culture-aware uses language rules. Use culture-aware for user-facing text."

**Why That's Insufficient:** Doesn't mention the Turkish I problem, doesn't address the performance difference, and doesn't give a concrete decision rule for when each applies.

**Great Answer:**
> "Ordinal comparison treats strings as sequences of UTF-16 code units — 'A' is 0x0041, 'a' is 0x0061, and that's it. No language rules, no locale, completely deterministic on every machine on earth. OrdinalIgnoreCase folds only ASCII A-Z to a-z before comparing — nothing else. Culture-aware comparisons apply language-specific rules: ligatures, accented characters, case folding that varies by locale. The production rule I follow is: protocol tokens, identifiers, URLs, dictionary keys, file paths, and anything from a database or config file → Ordinal or OrdinalIgnoreCase. User-visible text that will be sorted or searched → CurrentCulture. Serialized and cross-machine data → InvariantCulture. The classic failure mode is the Turkish I problem — on a machine with the tr-TR locale, `'I'.ToLower()` returns the dotless 'ı', not 'i'. If your authorization code calls `role.ToLower() == 'admin'`, it silently breaks for Turkish-locale servers. The fix is `string.Equals(role, 'admin', StringComparison.OrdinalIgnoreCase)` which has no allocation cost and is immune to all locale rules."

**Q: "Explain the format specifier system in C#. How does `$\"{value:C2}\"` actually work at runtime?"**

**Average Answer:** "Format specifiers control how numbers and dates are displayed. C is currency, F is fixed-point."

**Why That's Insufficient:** Doesn't explain the mechanism (IFormattable), doesn't address the allocation cost, and doesn't connect to production concerns about culture-sensitive formatting.

**Great Answer:**
> "When the compiler sees `$\"{value:C2}\"`, it generates a call to `string.Format` or, in .NET 6+, uses an interpolated string handler that calls `value.ToString('C2', provider)`. The 'C2' part is passed as a format string to the `ToString` overload defined on the type. Types that support format specifiers implement `IFormattable`, which has a `ToString(string? format, IFormatProvider? provider)` method. The `IFormatProvider` is how culture gets wired in — if you pass `CultureInfo.InvariantCulture`, the output is always the same regardless of the server locale. The production concern is separating display formatting from storage formatting. When I format a payment amount for display I use the user's culture — `amount.ToString('C', userCulture)`. When I serialize it to a database or a JSON payload I always use InvariantCulture — `amount.ToString('F2', CultureInfo.InvariantCulture)` — so the string '1234.56' is always parseable back to the same value regardless of which server reads it."

**Q: "What are raw string literals in C# 11 and when are they better than verbatim strings?"**

**Average Answer:** "Raw string literals use triple quotes and don't need escape characters."

**Why That's Insufficient:** Doesn't explain the indentation stripping behavior, the interpolation rules (number of `$` signs matches brace count), or the specific scenarios where they're superior.

**Great Answer:**
> "Raw string literals delimit content with three or more double quotes on each side. The compiler strips leading whitespace based on the indentation of the closing delimiter, so the string content doesn't inherit your code's indentation level. That alone solves a real problem with verbatim strings in methods — they either push content all the way to column 0, which is ugly, or they include unwanted indentation in the string value. For SQL, JSON, and regex patterns, raw strings are strictly better than verbatim strings because you truly never need to escape anything. The interpolation form uses `$"""..."""` and expressions are wrapped in `{expr}`, but if your string contains literal braces — like JSON — you use `$$"""..."""` where `{{expr}}` marks an expression, so a lone brace in the JSON content is not interpreted as an interpolation hole. This makes embedding JSON templates with dynamic values completely readable without any escaping or doubling."

### B. The Trick Questions

> [!WARNING] Common Interview Traps

**"Is `string.Empty` the same as `""`?"**
Trap: Many candidates say "they're the same." Correct: They're guaranteed to be the same object (`object.ReferenceEquals(string.Empty, "")` is `true` due to interning). Both have Length 0. But the idiomatic preference is `string.Empty` in code to signal intent, and `""` in switch cases and attribute arguments where `string.Empty` is not a compile-time constant.

**"Does `string.Concat("a", "b")` allocate one or two strings?"**
Trap: Candidates say "one — the result." Correct: One — `string.Concat` allocates exactly one string sized to the total length. There's no intermediate. But `"a" + "b"` where both are *literals* allocates **zero** — the compiler folds them to `"ab"` at compile time.

**"What does `"hello".Replace("l", "l")` do? Does it allocate?"**
Trap: Candidates say "returns the same string." Correct: The BCL implementation actually returns `this` (the original instance) when the replacement would produce an identical string — **but only in some .NET versions**. In .NET 5+, `Replace` detects no-op replacements and returns the original reference. In older versions it allocates a new string. Don't rely on this optimization — it's an implementation detail.

**"Can you modify a string character by using `unsafe` code?"**
Trap: Candidates say "no, it's immutable." Correct: Technically yes — with `fixed (char* p = myString)` you can write through the pointer. The CLR doesn't stop you at the memory level. But this violates the fundamental contract, breaks interning (you'd silently corrupt every reference to the same interned string), and is completely wrong in production. The right answer is: don't.

**"What is the output of `string.Format("{0} {0}", "hello")`?"**
Trap: Candidates hesitate or say it's an error. Correct: `"hello hello"` — the same argument at index 0 is used twice. `string.Format` positional arguments are indices into the args array, not consumed one-by-one.

### C. Red Flags to Avoid

```
❌ "string is a value type" — string is a reference type. It has VALUE EQUALITY
   semantics for == but it is allocated on the heap and copied by pointer.

❌ "StringBuilder is always faster than string concatenation" — for 2-3 pieces in
   one expression, string.Concat (which + compiles to) is faster. StringBuilder
   has upfront allocation cost.

❌ "I always use ToLower() before comparing strings" — ToLower() allocates a new
   string. Use StringComparison.OrdinalIgnoreCase for zero-allocation case-insensitive
   comparison.

❌ "String interning means strings are stored on the stack" — interning is a heap
   pool of deduplicated string objects. All strings are on the heap. Interning means
   multiple identical literals share the SAME heap object.

❌ "string.Format and $"..." are the same thing" — $"..." in .NET 6+ uses
   interpolated string handlers (DefaultInterpolatedStringHandler) that can avoid
   intermediate allocations via ISpanFormattable. string.Format always boxes value-type
   arguments and always allocates the result string.

❌ Omitting StringComparison from any production comparison — it signals you haven't
   thought about cultural correctness. Always be explicit about Ordinal vs culture.

❌ "Verbatim strings (@) are raw strings" — verbatim strings (C# 1.0) still require
   "" for a literal quote. Raw string literals (C# 11, """) need no escaping at all.
   They are different features.
```

### B. Code Puzzles

**Puzzle 1 — What is printed?**
```csharp
string a = "hello";
string b = "hel" + "lo";  // compile-time constant folding?
string c = "hel";
string d = c + "lo";       // runtime concatenation

Console.WriteLine(object.ReferenceEquals(a, b)); // Line 1
Console.WriteLine(object.ReferenceEquals(a, d)); // Line 2
Console.WriteLine(a == d);                        // Line 3
```

<details>
<summary>Answer and Explanation</summary>

**Line 1:** `True`
`"hel" + "lo"` is a constant expression — both operands are string literals. The compiler folds this to the literal `"hello"` at compile time. The CLR then interns string literals, so `a` and `b` both point to the same interned instance. `ReferenceEquals` returns `true`.

**Line 2:** `False`
`c + "lo"` is a runtime concatenation. `c` is a variable (even though it holds a literal value, the compiler doesn't know that at the concatenation site without constant folding). The `+` compiles to `string.Concat(c, "lo")`, which allocates a new string object at runtime. That new object is not the interned `"hello"` instance. `ReferenceEquals` returns `false`.

**Line 3:** `True`
`==` on strings uses `string.Equals`, which compares content. Both strings contain `"hello"`, so `==` returns `true` regardless of reference identity.

**Key lesson:** Reference equality and value equality are different for strings. The compiler does constant-fold string literal concatenation, but runtime-computed strings are always new objects unless explicitly interned with `string.Intern(d)`.

</details>

**Puzzle 2 — What is the output and how many allocations?**
```csharp
var invoiceIds = new[] { "INV-001", "INV-002", "INV-003" };

// Approach A:
string resultA = "";
foreach (var id in invoiceIds)
    resultA += id + ", ";

// Approach B:
string resultB = string.Join(", ", invoiceIds);

Console.WriteLine(resultA);
Console.WriteLine(resultB);
Console.WriteLine(resultA == resultB);
```

<details>
<summary>Answer and Explanation</summary>

**resultA output:** `"INV-001, INV-002, INV-003, "` (note: trailing `", "`)
**resultB output:** `"INV-001, INV-002, INV-003"` (no trailing separator)
**`resultA == resultB`:** `False` — trailing separator makes them different.

**Allocations in Approach A:**
- Iteration 1: `id + ", "` → 1 alloc (`"INV-001, "`), `resultA += ...` → 1 alloc (`"INV-001, "`)
- Iteration 2: `id + ", "` → 1 alloc (`"INV-002, "`), `resultA += ...` → 1 alloc (`"INV-001, INV-002, "`)
- Iteration 3: `id + ", "` → 1 alloc (`"INV-003, "`), `resultA += ...` → 1 alloc (`"INV-001, INV-002, INV-003, "`)
- **Total: 6 allocations**

**Allocations in Approach B:**
- `string.Join` pre-computes the total length, allocates exactly one result string.
- **Total: 1 allocation**

**Functional difference:** `string.Join` does NOT append a trailing separator after the last element. The `+=` approach appends `", "` after every element including the last. This is a common off-by-one in report generation — the trailing separator looks wrong to users.

</details>

**Puzzle 3 — Find the bug**
```csharp
// Payment authorization service
public static bool IsAuthorizedRegion(string countryCode)
{
    var authorized = new[] { "US", "GB", "DE", "FR", "JP" };
    return authorized.Any(c => c == countryCode);  // Line A
}

// Called from request handler:
bool allowed = IsAuthorizedRegion(request.Headers["X-Country-Code"].ToString().ToUpper());
```

<details>
<summary>Answer and Explanation</summary>

**The bug:** There are two issues.

**Issue 1 (allocation waste):** `ToString().ToUpper()` allocates two strings for every request — once for `ToString()` (if the header value is not already a string) and once for `ToUpper()`. For a service handling 10,000 req/sec, that's 10,000 unnecessary allocations per second just for this lookup.

**Issue 2 (the real bug):** `ToUpper()` uses `CurrentCulture`. On a machine with Turkish locale, the letter `i` in a country code would be incorrectly uppercased. While country codes are all ASCII uppercase, the `ToUpper()` before comparison is the wrong pattern — it's calling `ToUpper()` on caller-provided input (the header value) which might not be ASCII.

**The correct fix:**
```csharp
public static bool IsAuthorizedRegion(string countryCode)
{
    var authorized = new[] { "US", "GB", "DE", "FR", "JP" };
    // Zero allocation — OrdinalIgnoreCase handles case-insensitivity without ToUpper()
    return authorized.Any(c => string.Equals(c, countryCode, StringComparison.OrdinalIgnoreCase));
}

// Caller: no ToUpper() needed — the comparison handles it
bool allowed = IsAuthorizedRegion(request.Headers["X-Country-Code"]);
```

**Better still:** Put `authorized` in a `HashSet<string>(StringComparer.OrdinalIgnoreCase)` as a static field — `O(1)` lookup instead of `O(n)` scan.

</details>

**Puzzle 4 — What is printed?**
```csharp
var sb = new StringBuilder();
sb.Append("Invoice");
string s1 = sb.ToString();
sb.Append("-001");
string s2 = sb.ToString();
sb.Clear();
string s3 = sb.ToString();

Console.WriteLine(s1);
Console.WriteLine(s2);
Console.WriteLine(s3);
Console.WriteLine(s3 == string.Empty);
Console.WriteLine(object.ReferenceEquals(s3, string.Empty));
```

<details>
<summary>Answer and Explanation</summary>

```
Invoice
Invoice-001
                   ← empty line (empty string)
True
True               ← (implementation detail — may vary)
```

**Explanation:**

`ToString()` does NOT consume or reset the `StringBuilder`. It creates a snapshot of the current content as an immutable string. So `s1` captures the state after `Append("Invoice")`, and `s2` captures the state after `Append("-001")` — they are independent strings. Appending after a `ToString()` call continues from where the builder left off.

`sb.Clear()` sets the builder's length to 0 but retains its internal buffer. `sb.ToString()` on a zero-length builder returns `string.Empty`.

`s3 == string.Empty` is `True` because string equality is value-based and both have length 0.

`object.ReferenceEquals(s3, string.Empty)` is typically `True` in modern .NET because `StringBuilder.ToString()` on an empty builder returns the shared `string.Empty` instance as an optimization — but this is an implementation detail, not a language guarantee. The interesting observation is that `s3` is the same object reference as `string.Empty`.

**Key lesson:** `ToString()` is a snapshot — it doesn't affect the builder state. `Clear()` resets the builder but the internal buffer is retained for reuse.

</details>

**Puzzle 5 — The most common misunderstanding: does this code have a bug?**
```csharp
// User search service
public List<User> SearchByName(string searchTerm)
{
    if (string.IsNullOrEmpty(searchTerm))
        return new List<User>();

    string normalizedTerm = searchTerm.ToLower();

    return _users
        .Where(u => u.FullName.ToLower().Contains(normalizedTerm))
        .ToList();
}
```

<details>
<summary>Answer and Explanation</summary>

**Yes — there are two bugs and one design issue.**

**Bug 1 (correctness): `IsNullOrEmpty` instead of `IsNullOrWhiteSpace`.**
A search term of `"   "` (spaces only) passes the null/empty guard, gets lowercased to `"   "`, and then `Contains("   ")` will match any user whose name contains three consecutive spaces. This likely returns no results silently, confusing the caller. Use `IsNullOrWhiteSpace` and also call `searchTerm.Trim()`.

**Bug 2 (performance and correctness): `ToLower()` in a loop.**
`u.FullName.ToLower()` is called for **every user in the dataset, on every search request**. For a dataset of 100,000 users, that's 100,000 string allocations per query — each one garbage-collected immediately. The fix is `string.Contains(normalizedTerm, StringComparison.OrdinalIgnoreCase)` which performs case-insensitive comparison in-place with zero allocation.

**Design issue:** `ToLower()` uses `CurrentCulture`, which is non-deterministic across servers. If a user's name contains Turkish, German, or other non-ASCII characters, lowercasing and searching may produce different results depending on the server locale.

**Corrected version:**
```csharp
public List<User> SearchByName(string searchTerm)
{
    if (string.IsNullOrWhiteSpace(searchTerm))
        return new List<User>();

    string trimmedTerm = searchTerm.Trim(); // normalize whitespace

    return _users
        .Where(u => u.FullName.Contains(trimmedTerm, StringComparison.OrdinalIgnoreCase))
        .ToList();
    // Contains(string, StringComparison) — zero extra allocation, correct behavior
}
```

</details>
