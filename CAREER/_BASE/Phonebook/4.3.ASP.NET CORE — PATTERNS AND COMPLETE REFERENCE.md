## 1. DELEGATINGHANDLER — HTTPCLIENT CROSS-CUTTING CONCERNS

### What It Actually Is
`DelegatingHandler` is the HttpClient equivalent of middleware. It wraps an `HttpMessageHandler` and intercepts outbound HTTP requests and inbound responses. The factory creates a chain: your `DelegatingHandler` → next handler → ... → `HttpClientHandler` (actual socket). Use it for: authentication token injection, request/response logging, retry correlation, and API-specific header propagation.

### How It Works — Mechanically

```
HttpClient.SendAsync(request)
         │
         ▼
[LoggingHandler]        ← outermost, logs request start
         │
[AuthTokenHandler]      ← injects Authorization header
         │
[CorrelationIdHandler]  ← forwards X-Correlation-Id
         │
[HttpClientHandler]     ← actual TCP socket, makes the call
         │
         ← response flows back up through handlers in reverse
```

### The Real-World Code

```csharp
// ── Auth token injection handler ──────────────────────────────────────────────
public class BearerTokenHandler : DelegatingHandler
{
    private readonly ITokenProvider _tokenProvider;

    public BearerTokenHandler(ITokenProvider tokenProvider)
        => _tokenProvider = tokenProvider;

    protected override async Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request,
        CancellationToken ct)
    {
        // Get token (cached internally by ITokenProvider)
        var token = await _tokenProvider.GetTokenAsync(ct);
        request.Headers.Authorization = new AuthenticationHeaderValue("Bearer", token);

        var response = await base.SendAsync(request, ct);

        // If 401, token may have expired — refresh and retry once
        if (response.StatusCode == HttpStatusCode.Unauthorized)
        {
            response.Dispose();
            var freshToken = await _tokenProvider.RefreshTokenAsync(ct);
            request.Headers.Authorization = new AuthenticationHeaderValue("Bearer", freshToken);
            response = await base.SendAsync(request, ct);
        }

        return response;
    }
}


// ── Correlation ID propagation handler ───────────────────────────────────────
public class CorrelationIdHandler : DelegatingHandler
{
    private readonly IHttpContextAccessor _httpContextAccessor;

    public CorrelationIdHandler(IHttpContextAccessor accessor)
        => _httpContextAccessor = accessor;

    protected override Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request,
        CancellationToken ct)
    {
        // Forward the incoming correlation ID to outgoing calls
        // This links traces across service boundaries
        var correlationId = _httpContextAccessor.HttpContext?
            .Items["X-Correlation-Id"]?.ToString()
            ?? Guid.NewGuid().ToString("N");

        request.Headers.TryAddWithoutValidation("X-Correlation-Id", correlationId);
        request.Headers.TryAddWithoutValidation("traceparent",
            Activity.Current?.Id ?? correlationId);

        return base.SendAsync(request, ct);
    }
}


// ── Outbound request/response logging ────────────────────────────────────────
public class HttpLoggingHandler : DelegatingHandler
{
    private readonly ILogger<HttpLoggingHandler> _logger;

    public HttpLoggingHandler(ILogger<HttpLoggingHandler> logger)
        => _logger = logger;

    protected override async Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request,
        CancellationToken ct)
    {
        _logger.LogDebug(
            "→ {Method} {Uri}",
            request.Method,
            request.RequestUri);

        var sw = Stopwatch.StartNew();
        HttpResponseMessage response;

        try
        {
            response = await base.SendAsync(request, ct);
        }
        catch (Exception ex)
        {
            sw.Stop();
            _logger.LogError(ex,
                "✗ {Method} {Uri} failed after {ElapsedMs}ms",
                request.Method, request.RequestUri, sw.ElapsedMilliseconds);
            throw;
        }

        sw.Stop();

        var level = response.IsSuccessStatusCode
            ? LogLevel.Debug
            : LogLevel.Warning;

        _logger.Log(level,
            "← {Method} {Uri} → {StatusCode} in {ElapsedMs}ms",
            request.Method,
            request.RequestUri,
            (int)response.StatusCode,
            sw.ElapsedMilliseconds);

        return response;
    }
}


// ── Registration: chain handlers in correct order ─────────────────────────────
builder.Services.AddTransient<BearerTokenHandler>();
builder.Services.AddTransient<CorrelationIdHandler>();
builder.Services.AddTransient<HttpLoggingHandler>();

builder.Services.AddHttpClient<IInventoryApiClient, InventoryApiClient>(client =>
{
    client.BaseAddress = new Uri(configuration["InventoryApi:BaseUrl"]!);
})
// Handlers execute in registration order (first added = outermost)
.AddHttpMessageHandler<HttpLoggingHandler>()     // outermost — logs everything
.AddHttpMessageHandler<CorrelationIdHandler>()   // middle
.AddHttpMessageHandler<BearerTokenHandler>()     // innermost — sets auth header
.AddStandardResilienceHandler();                 // Polly wraps the innermost handler
```

## 2. REQUEST TIMEOUTS (.NET 8)

### What It Actually Is
`RequestTimeoutMiddleware` (introduced in .NET 8) enforces a deadline on how long a request can take to complete, killing it with a `504 Gateway Timeout` if the deadline passes. Previously you'd have to manage this with `CancellationTokenSource.CreateLinkedTokenSource`, which is error-prone. This is separate from Kestrel's request body timeout — this cancels your entire endpoint handler.

### How It Works — Mechanically

1. `UseRequestTimeouts()` middleware wraps the downstream pipeline.
2. When the timeout expires, it cancels `HttpContext.RequestAborted` and returns 504.
3. Your handler must observe `CancellationToken` for this to work — the timeout can't force-abort synchronous code.

### The Real-World Code

```csharp
// ── Registration ──────────────────────────────────────────────────────────────
builder.Services.AddRequestTimeouts(options =>
{
    // Default: applies to all endpoints without a specific policy
    options.DefaultPolicy = new RequestTimeoutPolicy
    {
        Timeout = TimeSpan.FromSeconds(30),
        TimeoutStatusCode = StatusCodes.Status504GatewayTimeout
    };

    // Tight timeout for simple lookups
    options.AddPolicy("fast", new RequestTimeoutPolicy
    {
        Timeout = TimeSpan.FromSeconds(5)
    });

    // Generous timeout for report generation
    options.AddPolicy("report", new RequestTimeoutPolicy
    {
        Timeout = TimeSpan.FromMinutes(5),
        // Custom response when timeout fires
        WriteTimeoutResponse = async ctx =>
        {
            ctx.Response.ContentType = "application/json";
            await ctx.Response.WriteAsJsonAsync(new ProblemDetails
            {
                Status = 504,
                Title = "Report generation timed out",
                Detail = "The report took too long to generate. Try a smaller date range."
            });
        }
    });
});

app.UseRequestTimeouts();


// ── Apply to endpoints ────────────────────────────────────────────────────────
app.MapGet("/api/products/{id}", GetProduct)
    .WithRequestTimeout("fast");

app.MapPost("/api/reports/generate", GenerateReport)
    .WithRequestTimeout("report");

// Disable timeout for streaming endpoints
app.MapGet("/api/orders/stream", StreamOrders)
    .DisableRequestTimeout();


// ── Handler must observe CancellationToken ────────────────────────────────────
static async Task<IResult> GetProduct(
    int id,
    IProductRepository repo,
    CancellationToken ct) // This token is cancelled when the request times out
{
    // If the operation exceeds "fast" policy (5 seconds), ct is cancelled
    // SqlClient, HttpClient, EF Core all honour ct — so they'll abort
    var product = await repo.GetByIdAsync(id, ct);
    return product is null ? TypedResults.NotFound() : TypedResults.Ok(product);
}
```

## 3. IHOSTEDLIFECYCLESERVICE (.NET 8)

### What It Actually Is
`.NET 8` extended `IHostedService` with `IHostedLifecycleService` — adding four new hooks that fire before and after the existing `StartAsync`/`StopAsync`. This lets you sequence initialization precisely: run migrations before routes are registered, warm up caches before accepting traffic, drain queues before disconnecting from the message bus.

### How It Works — Mechanically

```
Startup order:
  1. StartingAsync   ← new in .NET 8 — runs before StartAsync
  2. StartAsync      ← existing IHostedService.StartAsync
  3. StartedAsync    ← new in .NET 8 — runs after all services have started
  4. [Application running, accepting traffic]

Shutdown order (reverse of registration):
  5. StoppingAsync   ← new in .NET 8 — runs before StopAsync
  6. StopAsync       ← existing IHostedService.StopAsync
  7. StoppedAsync    ← new in .NET 8 — runs after all services have stopped
```

### The Real-World Code

```csharp
// ── Full lifecycle service ─────────────────────────────────────────────────────
public class ApplicationInitializationService : IHostedLifecycleService
{
    private readonly IServiceScopeFactory _scopeFactory;
    private readonly ILogger<ApplicationInitializationService> _logger;

    public ApplicationInitializationService(
        IServiceScopeFactory scopeFactory,
        ILogger<ApplicationInitializationService> logger)
    {
        _scopeFactory = scopeFactory;
        _logger = logger;
    }

    // Runs BEFORE StartAsync — perfect for database migrations
    public async Task StartingAsync(CancellationToken ct)
    {
        _logger.LogInformation("Running database migrations...");
        await using var scope = _scopeFactory.CreateAsyncScope();
        var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
        await db.Database.MigrateAsync(ct);
        _logger.LogInformation("Migrations complete.");
    }

    // Standard startup — start background loops
    public async Task StartAsync(CancellationToken ct)
    {
        _logger.LogInformation("Application services starting...");
        await Task.CompletedTask;
    }

    // Runs AFTER all services have started — perfect for cache warm-up
    // At this point, traffic may already be arriving
    public async Task StartedAsync(CancellationToken ct)
    {
        _logger.LogInformation("Warming up product catalog cache...");
        await using var scope = _scopeFactory.CreateAsyncScope();
        var cache = scope.ServiceProvider.GetRequiredService<IProductCacheService>();
        await cache.WarmUpAsync(ct);
        _logger.LogInformation("Cache warm-up complete. Ready for traffic.");
    }

    // Runs BEFORE StopAsync — stop accepting new work
    public async Task StoppingAsync(CancellationToken ct)
    {
        _logger.LogInformation("Stopping: draining message queue...");
        await using var scope = _scopeFactory.CreateAsyncScope();
        var queue = scope.ServiceProvider.GetRequiredService<IMessageQueueDrainer>();
        await queue.DrainAsync(ct);
    }

    public Task StopAsync(CancellationToken ct)
    {
        _logger.LogInformation("Services stopped.");
        return Task.CompletedTask;
    }

    // Runs AFTER all services have stopped — final cleanup
    public async Task StoppedAsync(CancellationToken ct)
    {
        _logger.LogInformation("Final cleanup complete. Goodbye.");
        await Task.CompletedTask;
    }
}

builder.Services.AddHostedService<ApplicationInitializationService>();
```

## 4. FORM HANDLING & FILE UPLOADS

### What It Actually Is
Form data in HTTP arrives in two encodings: `application/x-www-form-urlencoded` (URL-encoded key-value pairs, for simple forms) and `multipart/form-data` (for file uploads — each part is a separate MIME segment with its own headers). ASP.NET Core parses both into `IFormCollection`. For large file uploads, stream directly to storage rather than buffering in memory.

### How It Works — Mechanically

**Multipart parsing:**
1. Kestrel reads the boundary from `Content-Type: multipart/form-data; boundary=----WebKitBoundary...`
2. The parser splits the body into sections at each boundary.
3. Each section's headers are parsed: `Content-Disposition: form-data; name="file"; filename="report.pdf"`.
4. Sections become `IFormFile` objects in `IFormCollection.Files`.

### The Real-World Code

```csharp
// ── Simple file upload ────────────────────────────────────────────────────────
[HttpPost("upload")]
[RequestSizeLimit(50 * 1024 * 1024)]     // 50 MB — override Kestrel global limit
[RequestFormLimits(MultipartBodyLengthLimit = 50 * 1024 * 1024)]
public async Task<IActionResult> UploadFile(
    [FromForm] FileUploadRequest request,
    CancellationToken ct)
{
    if (request.File.Length == 0)
        return BadRequest("File is empty");

    // Validate content type — don't trust client-supplied Content-Type
    var allowedTypes = new[] { "image/jpeg", "image/png", "image/webp", "application/pdf" };
    if (!allowedTypes.Contains(request.File.ContentType))
        return BadRequest($"File type not allowed: {request.File.ContentType}");

    // Validate file signature (magic bytes) — not just extension or Content-Type
    if (!await IsValidFileSignatureAsync(request.File, ct))
        return BadRequest("File content does not match declared type");

    // Stream directly to storage — never buffer in memory for large files
    var blobName = $"{Guid.NewGuid():N}{Path.GetExtension(request.File.FileName)}";
    await using var stream = request.File.OpenReadStream();
    var url = await _storage.UploadAsync(blobName, stream, request.File.ContentType, ct);

    return Ok(new { Url = url, FileName = blobName });
}

public record FileUploadRequest
{
    [Required]
    public IFormFile File { get; init; } = null!;

    [MaxLength(500)]
    public string? Description { get; init; }
}

// File signature validation — prevents content-type spoofing
private static async Task<bool> IsValidFileSignatureAsync(IFormFile file, CancellationToken ct)
{
    await using var stream = file.OpenReadStream();
    var header = new byte[8];
    var read = await stream.ReadAsync(header.AsMemory(0, 8), ct);
    if (read < 4) return false;

    // JPEG: FF D8 FF
    if (header[0] == 0xFF && header[1] == 0xD8 && header[2] == 0xFF)
        return file.ContentType == "image/jpeg";

    // PNG: 89 50 4E 47
    if (header[0] == 0x89 && header[1] == 0x50 && header[2] == 0x4E && header[3] == 0x47)
        return file.ContentType == "image/png";

    // PDF: 25 50 44 46
    if (header[0] == 0x25 && header[1] == 0x50 && header[2] == 0x44 && header[3] == 0x46)
        return file.ContentType == "application/pdf";

    return false;
}


// ── Multi-file upload with progress ──────────────────────────────────────────
[HttpPost("upload-batch")]
public async Task<IActionResult> UploadBatch(
    List<IFormFile> files,
    CancellationToken ct)
{
    if (files.Count > 10)
        return BadRequest("Maximum 10 files per batch");

    var results = new List<UploadResult>();

    foreach (var file in files)
    {
        try
        {
            await using var stream = file.OpenReadStream();
            var url = await _storage.UploadAsync(
                $"{Guid.NewGuid():N}{Path.GetExtension(file.FileName)}",
                stream,
                file.ContentType,
                ct);
            results.Add(new UploadResult(file.FileName, url, null));
        }
        catch (Exception ex)
        {
            // Partial success — continue with other files, report individual errors
            results.Add(new UploadResult(file.FileName, null, ex.Message));
        }
    }

    // 207 Multi-Status when some succeeded and some failed
    var statusCode = results.All(r => r.Error is null) ? 200 :
                     results.All(r => r.Error is not null) ? 400 : 207;

    return StatusCode(statusCode, results);
}


// ── Minimal API file upload ───────────────────────────────────────────────────
app.MapPost("/api/files", async (
    IFormFile file,           // ASP.NET Core binds from multipart form
    IStorageService storage,
    CancellationToken ct) =>
{
    await using var stream = file.OpenReadStream();
    var url = await storage.UploadAsync(file.FileName, stream, file.ContentType, ct);
    return TypedResults.Created(url, new { Url = url });
})
.DisableAntiforgery()        // Multipart forms from SPAs don't need antiforgery
.RequireAuthorization();
```

## 5. CONTENT NEGOTIATION

### What It Actually Is
Content negotiation is the mechanism by which the client tells the server what format it wants (`Accept: application/xml`) and the server either obliges or returns `406 Not Acceptable`. ASP.NET Core's `ObjectResult` (returned by `Ok()`, `BadRequest()`, etc.) runs through content negotiation before writing the response. By default, only JSON is supported. Adding `AddXmlSerializerFormatters()` enables XML.

### The Real-World Code

```csharp
// ── Enable multiple output formats ────────────────────────────────────────────
builder.Services.AddControllers(options =>
{
    // Return 406 if no supported format matches — don't silently default to JSON
    options.RespectBrowserAcceptHeader = true;
    options.ReturnHttpNotAcceptable = true;
})
.AddJsonOptions(opts =>
{
    opts.JsonSerializerOptions.PropertyNamingPolicy = JsonNamingPolicy.CamelCase;
    opts.JsonSerializerOptions.WriteIndented = false; // Never indent in production
    opts.JsonSerializerOptions.DefaultIgnoreCondition =
        JsonIgnoreCondition.WhenWritingNull;
    opts.JsonSerializerOptions.Converters.Add(new JsonStringEnumConverter());
})
.AddXmlSerializerFormatters();  // Adds XML support for Accept: application/xml


// ── Custom formatter: CSV output ──────────────────────────────────────────────
public class CsvOutputFormatter : TextOutputFormatter
{
    public CsvOutputFormatter()
    {
        SupportedMediaTypes.Add("text/csv");
        SupportedEncodings.Add(Encoding.UTF8);
        SupportedEncodings.Add(Encoding.Unicode);
    }

    protected override bool CanWriteType(Type? type)
        => type is not null &&
           (typeof(IEnumerable).IsAssignableFrom(type) && type != typeof(string));

    public override async Task WriteResponseBodyAsync(
        OutputFormatterWriteContext context,
        Encoding selectedEncoding)
    {
        var response = context.HttpContext.Response;
        response.Headers.ContentDisposition = "attachment; filename=\"export.csv\"";

        await using var writer = new StreamWriter(response.Body, selectedEncoding);

        if (context.Object is IEnumerable items)
        {
            bool headerWritten = false;
            PropertyInfo[]? properties = null;

            foreach (var item in items)
            {
                if (!headerWritten)
                {
                    properties = item.GetType().GetProperties(
                        BindingFlags.Public | BindingFlags.Instance);
                    await writer.WriteLineAsync(
                        string.Join(",", properties.Select(p => p.Name)));
                    headerWritten = true;
                }

                var values = properties!.Select(p =>
                {
                    var val = p.GetValue(item)?.ToString() ?? "";
                    return val.Contains(',') || val.Contains('"')
                        ? $"\"{val.Replace("\"", "\"\"")}\""
                        : val;
                });
                await writer.WriteLineAsync(string.Join(",", values));
            }
        }
    }
}

// Register
builder.Services.AddControllers(opts =>
    opts.OutputFormatters.Add(new CsvOutputFormatter()));

// Usage: client sends Accept: text/csv → gets a CSV download
```

## 6. CUSTOM JSON CONVERTERS

### What It Actually Is
`JsonConverter<T>` lets you control exactly how a type is serialized/deserialized by `System.Text.Json`. The built-in converters handle primitive types, collections, and POCOs. Custom converters handle: discriminated unions, polymorphism by type discriminator, strongly-typed IDs, custom date formats, value objects, and enums-as-integers-with-string-fallback.

### The Real-World Code

```csharp
// ── Strongly-typed ID converter (prevents primitive obsession) ────────────────
public readonly record struct OrderId(int Value);

public class StronglyTypedIdJsonConverter<TId, TValue>
    : JsonConverter<TId>
    where TId : struct
{
    private readonly Func<TValue, TId> _factory;
    private readonly Func<TId, TValue> _valueSelector;

    public StronglyTypedIdJsonConverter(
        Func<TValue, TId> factory,
        Func<TId, TValue> valueSelector)
    {
        _factory = factory;
        _valueSelector = valueSelector;
    }

    public override TId Read(ref Utf8JsonReader reader, Type typeToConvert, JsonSerializerOptions options)
    {
        var value = JsonSerializer.Deserialize<TValue>(ref reader, options)!;
        return _factory(value);
    }

    public override void Write(Utf8JsonWriter writer, TId value, JsonSerializerOptions options)
    {
        JsonSerializer.Serialize(writer, _valueSelector(value), options);
    }
}

// Registration
builder.Services.AddControllers().AddJsonOptions(opts =>
{
    opts.JsonSerializerOptions.Converters.Add(
        new StronglyTypedIdJsonConverter<OrderId, int>(
            value => new OrderId(value),
            id => id.Value));
});


// ── Polymorphic JSON (discriminated union) ────────────────────────────────────
// .NET 7+ has built-in polymorphic support via attributes
[JsonPolymorphic(TypeDiscriminatorPropertyName = "$type")]
[JsonDerivedType(typeof(EmailNotification), "email")]
[JsonDerivedType(typeof(SmsNotification), "sms")]
[JsonDerivedType(typeof(PushNotification), "push")]
public abstract class NotificationRequest
{
    public string Message { get; set; } = null!;
}

public class EmailNotification : NotificationRequest
{
    public string To { get; set; } = null!;
    public string Subject { get; set; } = null!;
}

public class SmsNotification : NotificationRequest
{
    public string PhoneNumber { get; set; } = null!;
}

// JSON: { "$type": "email", "message": "Hello", "to": "a@b.com", "subject": "Hi" }
// Deserializes to EmailNotification automatically


// ── Custom DateTimeOffset converter (epoch seconds) ───────────────────────────
public class EpochSecondsConverter : JsonConverter<DateTimeOffset>
{
    public override DateTimeOffset Read(
        ref Utf8JsonReader reader, Type typeToConvert, JsonSerializerOptions options)
    {
        if (reader.TokenType == JsonTokenType.Number)
        {
            var epoch = reader.GetInt64();
            return DateTimeOffset.FromUnixTimeSeconds(epoch);
        }
        // Fallback: try ISO 8601
        return DateTimeOffset.Parse(reader.GetString()!);
    }

    public override void Write(Utf8JsonWriter writer, DateTimeOffset value, JsonSerializerOptions options)
    {
        writer.WriteNumberValue(value.ToUnixTimeSeconds());
    }
}


// ── NullableConverter for value types ─────────────────────────────────────────
// Handles: null → default, "N/A" → null, etc.
public class MoneyJsonConverter : JsonConverter<decimal?>
{
    public override decimal? Read(
        ref Utf8JsonReader reader, Type typeToConvert, JsonSerializerOptions options)
    {
        if (reader.TokenType == JsonTokenType.Null) return null;
        if (reader.TokenType == JsonTokenType.String)
        {
            var str = reader.GetString();
            if (str is null or "N/A" or "") return null;
            return decimal.Parse(str, CultureInfo.InvariantCulture);
        }
        return reader.GetDecimal();
    }

    public override void Write(Utf8JsonWriter writer, decimal? value, JsonSerializerOptions options)
    {
        if (value is null)
            writer.WriteNullValue();
        else
            writer.WriteNumberValue(value.Value);
    }
}
```

## 7. API BEHAVIOR OPTIONS — CUSTOMIZING [APICONTROLLER]

### What It Actually Is
`[ApiController]` attribute enables several automatic behaviors. `ApiBehaviorOptions` lets you override each one. The most important: the automatic 400 response. By default, if ModelState is invalid, ASP.NET Core returns a `ValidationProblemDetails` response before your action method even runs. You can replace this behavior, customize the response factory, or suppress it entirely.

### The Real-World Code

```csharp
// ── Customize the automatic 400 response factory ──────────────────────────────
builder.Services.Configure<ApiBehaviorOptions>(options =>
{
    // Replace the default ValidationProblemDetails factory
    options.InvalidModelStateResponseFactory = context =>
    {
        var errors = context.ModelState
            .Where(e => e.Value?.Errors.Count > 0)
            .ToDictionary(
                kvp => kvp.Key,
                kvp => kvp.Value!.Errors
                    .Select(e => e.ErrorMessage)
                    .ToArray());

        var problemDetails = new HttpValidationProblemDetails(errors)
        {
            Status = StatusCodes.Status400BadRequest,
            Title = "Validation failed",
            Type = "https://api.example.com/errors/validation",
            // Add trace ID so the client can reference it in support tickets
            Extensions =
            {
                ["traceId"] = context.HttpContext.TraceIdentifier,
                ["timestamp"] = DateTimeOffset.UtcNow
            }
        };

        return new BadRequestObjectResult(problemDetails)
        {
            ContentTypes = { "application/problem+json" }
        };
    };

    // Suppress the auto-400 for a specific area (not recommended globally)
    // options.SuppressModelStateInvalidFilter = true;

    // Suppress 415 Unsupported Media Type for missing Content-Type
    // options.SuppressConsumesConstraintForFormFileParameters = true;

    // Suppress the binding sources inference
    // options.SuppressInferBindingSourcesForParameters = true;
});


// ── ApiBehaviorOptions complete defaults reference ────────────────────────────
// SuppressModelStateInvalidFilter = false         Auto-400 on invalid ModelState
// SuppressInferBindingSourcesForParameters = false Infer [FromBody] for complex types
// SuppressConsumesConstraintForFormFileParameters = false Require Content-Type
// SuppressMapClientErrors = false                 Map 4xx to ProblemDetails
// ClientErrorMapping = { 400→"rfc", 401→"rfc", ... } Maps status to ProblemDetails type
```

## 8. IENDPOINTCONVENTIONBUILDER — ENDPOINT METADATA PATTERNS

### What It Actually Is
Every `app.MapGet(...)` call returns an `IEndpointConventionBuilder`. It's a fluent API for attaching metadata to endpoints. Middleware (like `UseAuthorization`) reads this metadata via `HttpContext.GetEndpoint()?.Metadata` to make decisions. You can add standard metadata (auth policies, rate limit policies, CORS policies, OpenAPI tags) or custom metadata for your own middleware.

### The Real-World Code

```csharp
// ── Custom metadata and convention extensions ─────────────────────────────────

// Marker interface for endpoints that require audit logging
public interface IAuditableEndpoint { }
public class AuditableEndpoint : IAuditableEndpoint { }

// Custom extension method — cleaner than calling AddEndpointFilter everywhere
public static class EndpointConventionBuilderExtensions
{
    // Mark as auditable — consumed by AuditLoggingMiddleware
    public static TBuilder RequireAuditLogging<TBuilder>(this TBuilder builder)
        where TBuilder : IEndpointConventionBuilder
    {
        builder.WithMetadata(new AuditableEndpoint());
        return builder;
    }

    // Mark as internal — block from public-facing proxy
    public static TBuilder InternalOnly<TBuilder>(this TBuilder builder)
        where TBuilder : IEndpointConventionBuilder
    {
        builder.WithMetadata(new InternalOnlyAttribute());
        return builder;
    }

    // Apply full production conventions: auth + logging + rate limit
    public static TBuilder WithProductionDefaults<TBuilder>(this TBuilder builder)
        where TBuilder : IEndpointConventionBuilder
    {
        return builder
            .RequireAuthorization()
            .RequireRateLimiting("default")
            .RequireAuditLogging()
            .WithOpenApi();
    }
}


// ── Reading metadata in middleware ────────────────────────────────────────────
public class AuditLoggingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly IAuditService _audit;

    public AuditLoggingMiddleware(RequestDelegate next, IAuditService audit)
    {
        _next = next;
        _audit = audit;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        // Check if the matched endpoint requires audit logging
        var endpoint = context.GetEndpoint();
        var requiresAudit = endpoint?.Metadata.GetMetadata<IAuditableEndpoint>() is not null;

        await _next(context);

        if (requiresAudit && context.User.Identity?.IsAuthenticated == true)
        {
            await _audit.LogAsync(new AuditEntry
            {
                UserId = context.User.FindFirstValue(ClaimTypes.NameIdentifier)!,
                Action = $"{context.Request.Method} {context.Request.Path}",
                StatusCode = context.Response.StatusCode,
                Timestamp = DateTime.UtcNow
            });
        }
    }
}
```

## 9. LOCALIZATION & GLOBALIZATION

### What It Actually Is
Localization adapts your API's string responses (error messages, validation messages) to the client's language. The client signals language preference via `Accept-Language: fr-FR, en;q=0.9`. ASP.NET Core's request localization middleware reads this, determines the culture, and sets `CultureInfo.CurrentCulture` for the request thread. Resource files (`.resx`) hold translations.

### The Real-World Code

```csharp
// ── Registration ──────────────────────────────────────────────────────────────
builder.Services.AddLocalization(options =>
    options.ResourcesPath = "Resources"); // Folder containing .resx files

builder.Services.Configure<RequestLocalizationOptions>(options =>
{
    var supportedCultures = new[]
    {
        new CultureInfo("en-US"),
        new CultureInfo("fr-FR"),
        new CultureInfo("ar-SA"),
        new CultureInfo("es-ES")
    };

    options.DefaultRequestCulture = new RequestCulture("en-US");
    options.SupportedCultures = supportedCultures;
    options.SupportedUICultures = supportedCultures;

    // Culture resolution order: query → cookie → Accept-Language header
    options.RequestCultureProviders = new List<IRequestCultureProvider>
    {
        new QueryStringRequestCultureProvider { QueryStringKey = "lang" },
        new CookieRequestCultureProvider(),
        new AcceptLanguageHeaderRequestCultureProvider()
    };
});

app.UseRequestLocalization();


// ── Using IStringLocalizer in services ────────────────────────────────────────
// Resources/OrderService.en-US.resx → OrderNotFound = "Order {0} was not found"
// Resources/OrderService.fr-FR.resx → OrderNotFound = "La commande {0} n'a pas été trouvée"
```

## 10. CANCELLATION TOKEN PATTERNS — THE RIGHT MODEL

### What It Actually Is
`CancellationToken` is the mechanism for cooperative cancellation — the caller signals that it no longer needs the result, and the callee checks the token and stops work. In ASP.NET Core, `HttpContext.RequestAborted` fires when the client disconnects. `IHostApplicationLifetime.ApplicationStopping` fires on graceful shutdown. Understanding when each fires and how to compose them is the difference between a server that drains cleanly and one that corrupts data on pod restart.

### The Real-World Code

```csharp
// ── The three tokens in ASP.NET Core ─────────────────────────────────────────
//
// 1. CancellationToken (action parameter) = HttpContext.RequestAborted
//    Fires when: client disconnects, request times out
//    Use for: ALL I/O in request handlers (DB, HTTP, Redis)
//
// 2. BackgroundService.stoppingToken (ExecuteAsync parameter)
//    Fires when: IHostApplicationLifetime.ApplicationStopping fires
//    Use for: loop control in background services
//
// 3. IHostApplicationLifetime.ApplicationStopping
//    Fires when: SIGTERM received
//    Use for: one-time cleanup registration


// ── Correct pattern: compose request abort + shutdown ─────────────────────────
// Scenario: background service processing queue messages
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    await foreach (var message in _queue.ReadAllAsync(stoppingToken))
    {
        // Create a linked token: cancel if EITHER the app is stopping
        // OR if the individual message processing takes too long
        using var processTimeout = CancellationTokenSource.CreateLinkedTokenSource(stoppingToken);
        processTimeout.CancelAfter(TimeSpan.FromSeconds(30));

        try
        {
            await ProcessMessageAsync(message, processTimeout.Token);
        }
        catch (OperationCanceledException) when (stoppingToken.IsCancellationRequested)
        {
            // App is shutting down — requeue the message and exit cleanly
            await _queue.NackAsync(message);
            break;
        }
        catch (OperationCanceledException)
        {
            // Message processing timed out — log and nack
            _logger.LogWarning("Message {Id} processing timed out", message.Id);
            await _queue.NackAsync(message);
        }
    }
}


// ── Correct pattern: request handler ─────────────────────────────────────────
[HttpGet("{id:int}")]
public async Task<ActionResult<OrderDto>> GetById(
    int id,
    CancellationToken ct) // = HttpContext.RequestAborted
{
    // Always pass ct to ALL async operations
    // If client disconnects, all these will throw OperationCanceledException
    var order = await _db.Orders
        .Where(o => o.Id == id)
        .FirstOrDefaultAsync(ct);    // EF Core honours ct

    if (order is null) return NotFound();

    var inventory = await _inventoryClient.GetStockAsync(id, ct); // HttpClient honours ct
    var pricing = await _pricingClient.GetPriceAsync(id, ct);

    return Ok(new OrderDto(order, inventory, pricing));
    // If client disconnected, OperationCanceledException propagates out of here
    // ExceptionHandler middleware catches it — returns nothing (client is gone anyway)
}


// ── Anti-pattern: ignoring cancellation on DB writes ─────────────────────────
// WRONG: if client disconnects during a POST, we want to commit the write
// (order is created even if client doesn't see the response)
public async Task<Order> CreateOrderAsync(CreateOrderCommand cmd, CancellationToken ct)
{
    var order = Order.Create(cmd);
    _db.Orders.Add(order);

    // DON'T pass ct here — if client disconnects, we still want to commit
    // The order was created; the client just didn't get the response
    await _db.SaveChangesAsync(CancellationToken.None); // Intentional

    // DO pass ct here — sending email is optional, can be abandoned
    await _emailService.SendConfirmationAsync(order, ct);

    return order;
}
```

## 11. PROBLEMDETAILS — COMPLETE CUSTOMIZATION

### What It Actually Is
Problem Details (RFC 7807) is the standard JSON format for HTTP API error responses. The factory produces these objects; you customize it to add fields (trace ID, timestamp, error codes), map status codes to types, and suppress sensitive information in production. `IProblemDetailsService` is the interface; `ProblemDetailsFactory` is the abstract base class for MVC controllers.

### The Real-World Code

```csharp
// ── Custom ProblemDetailsFactory ─────────────────────────────────────────────
public class ApiProblemDetailsFactory : ProblemDetailsFactory
{
    private readonly ApiBehaviorOptions _options;
    private readonly IOptions<RouteOptions> _routeOptions;

    public ApiProblemDetailsFactory(
        IOptions<ApiBehaviorOptions> options,
        IOptions<RouteOptions> routeOptions)
    {
        _options = options.Value;
        _routeOptions = routeOptions.Value;
    }

    public override ProblemDetails CreateProblemDetails(
        HttpContext httpContext,
        int? statusCode = null,
        string? title = null,
        string? type = null,
        string? detail = null,
        string? instance = null)
    {
        statusCode ??= 500;

        var problemDetails = new ProblemDetails
        {
            Status = statusCode,
            Title = title ?? GetDefaultTitle(statusCode.Value),
            Type = type ?? GetRfcTypeUri(statusCode.Value),
            Detail = detail,
            Instance = instance ?? httpContext.Request.Path
        };

        ApplyExtensions(problemDetails, httpContext);
        return problemDetails;
    }

    public override ValidationProblemDetails CreateValidationProblemDetails(
        HttpContext httpContext,
        ModelStateDictionary modelStateDictionary,
        int? statusCode = null,
        string? title = null,
        string? type = null,
        string? detail = null,
        string? instance = null)
    {
        var problemDetails = new ValidationProblemDetails(modelStateDictionary)
        {
            Status = statusCode ?? 400,
            Title = title ?? "Validation failed",
            Type = type ?? "https://api.example.com/errors/validation",
            Detail = detail,
            Instance = instance ?? httpContext.Request.Path
        };

        ApplyExtensions(problemDetails, httpContext);
        return problemDetails;
    }

    private static void ApplyExtensions(ProblemDetails problemDetails, HttpContext ctx)
    {
        // Always include trace ID — correlates with logs
        problemDetails.Extensions["traceId"] = ctx.TraceIdentifier;
        problemDetails.Extensions["timestamp"] = DateTimeOffset.UtcNow;

        // Tenant context if available
        var tenantFeature = ctx.Features.Get<ITenantFeature>();
        if (tenantFeature is not null)
            problemDetails.Extensions["tenantId"] = tenantFeature.Tenant.Id;
    }

    private static string GetDefaultTitle(int statusCode) => statusCode switch
    {
        400 => "Bad Request",
        401 => "Unauthorized",
        403 => "Forbidden",
        404 => "Not Found",
        409 => "Conflict",
        422 => "Unprocessable Entity",
        429 => "Too Many Requests",
        500 => "Internal Server Error",
        502 => "Bad Gateway",
        503 => "Service Unavailable",
        504 => "Gateway Timeout",
        _ => "An error occurred"
    };

    private static string GetRfcTypeUri(int statusCode) => statusCode switch
    {
        400 => "https://tools.ietf.org/html/rfc7231#section-6.5.1",
        401 => "https://tools.ietf.org/html/rfc7235#section-3.1",
        403 => "https://tools.ietf.org/html/rfc7231#section-6.5.3",
        404 => "https://tools.ietf.org/html/rfc7231#section-6.5.4",
        409 => "https://tools.ietf.org/html/rfc7231#section-6.5.8",
        500 => "https://tools.ietf.org/html/rfc7231#section-6.6.1",
        _ => "https://tools.ietf.org/html/rfc7231"
    };
}

// Register
builder.Services.AddSingleton<ProblemDetailsFactory, ApiProblemDetailsFactory>();
```

## 12. RESPONSE CACHING (HTTP CACHE HEADERS)

### What It Actually Is
Response caching (`[ResponseCache]` and `UseResponseCaching()`) sets HTTP cache headers that instruct clients and proxies (CDN, Nginx, Varnish) to cache the response. Unlike Output Caching (server-side), this is *client/proxy-side* caching — the server doesn't store anything. The two work together: Output Cache stores the response on your server; ResponseCache headers tell CDNs to also cache downstream. They are complementary, not alternatives.

### The Real-World Code

```csharp
// ── Registration ──────────────────────────────────────────────────────────────
builder.Services.AddResponseCaching(options =>
{
    options.MaximumBodySize = 64 * 1024 * 1024; // 64 MB max response to cache
    options.UseCaseSensitivePaths = false;
});

app.UseResponseCaching(); // Must be before routing


// ── Cache profiles — reusable named configurations ────────────────────────────
builder.Services.AddControllers(options =>
{
    options.CacheProfiles.Add("Static", new CacheProfile
    {
        Duration = 86400,      // 1 day
        Location = ResponseCacheLocation.Any, // Client + CDN
    });
    options.CacheProfiles.Add("Short", new CacheProfile
    {
        Duration = 60,         // 1 minute
        Location = ResponseCacheLocation.Any,
        VaryByQueryKeys = new[] { "page", "pageSize" }
    });
    options.CacheProfiles.Add("Private", new CacheProfile
    {
        Duration = 30,
        Location = ResponseCacheLocation.Client, // Browser only, NOT CDN
        VaryByHeader = "Authorization"
    });
    options.CacheProfiles.Add("NoCache", new CacheProfile
    {
        NoStore = true
    });
});


// ── Applying to endpoints ─────────────────────────────────────────────────────
[HttpGet("countries")]
[ResponseCache(CacheProfileName = "Static")]
// Sends: Cache-Control: public, max-age=86400
public async Task<IActionResult> GetCountries() { /* ... */ return Ok(); }

[HttpGet]
[ResponseCache(CacheProfileName = "Short")]
// Sends: Cache-Control: public, max-age=60
// Varies response by page + pageSize query params
public async Task<IActionResult> GetProducts([FromQuery] int page = 1) { /* ... */ return Ok(); }

[HttpGet("my-orders")]
[ResponseCache(CacheProfileName = "Private")]
// Sends: Cache-Control: private, max-age=30
// Each user's browser caches their own orders; CDN does NOT cache
[Authorize]
public async Task<IActionResult> GetMyOrders() { /* ... */ return Ok(); }


// ── HTTP cache headers reference ──────────────────────────────────────────────
// Cache-Control: public             → CDN and browser can cache
// Cache-Control: private            → Browser only (not CDN)
// Cache-Control: no-store           → Never cache (sensitive data)
// Cache-Control: no-cache           → Revalidate with server before using
// Cache-Control: max-age=N          → Cache for N seconds
// Cache-Control: s-maxage=N         → CDN cache for N seconds (overrides max-age for proxies)
// Cache-Control: stale-while-revalidate=N → Serve stale while refreshing in background
// Cache-Control: must-revalidate    → Must check server when stale (no serving stale)
// ETag: "abc123"                    → Fingerprint of response; client sends If-None-Match
// Last-Modified: <date>             → Client sends If-Modified-Since
// Vary: Accept-Language             → Cache separately per language
// Vary: Authorization               → NEVER do this — security risk, each token gets own cache
```

## 13. SECURITY HEADERS — COMPLETE REFERENCE

### What It Actually Is
HTTP response headers are one of the cheapest security controls available. Most XSS, clickjacking, and MIME-sniffing attacks can be mitigated or eliminated with correct headers. Setting them globally in middleware means no endpoint can forget them.

### The Real-World Code

```csharp
// ── Security headers middleware ────────────────────────────────────────────────
app.Use(async (ctx, next) =>
{
    // Prevents browsers from MIME-sniffing the Content-Type
    // Without this, browsers may execute a text/plain file as script
    ctx.Response.Headers.Append("X-Content-Type-Options", "nosniff");

    // Prevents the page from being embedded in an iframe (clickjacking)
    ctx.Response.Headers.Append("X-Frame-Options", "DENY");

    // Tells browser to enforce HTTPS (only over HTTPS — HSTS handles the redirect)
    // max-age=31536000 = 1 year; includeSubDomains = all subdomains too
    // preload = can be submitted to browser HSTS preload lists
    if (ctx.Request.IsHttps)
        ctx.Response.Headers.Append(
            "Strict-Transport-Security",
            "max-age=31536000; includeSubDomains; preload");

    // Controls how much referrer info to send when navigating away
    ctx.Response.Headers.Append("Referrer-Policy", "strict-origin-when-cross-origin");

    // Restricts browser features (camera, microphone, geolocation)
    ctx.Response.Headers.Append("Permissions-Policy",
        "camera=(), microphone=(), geolocation=(), payment=()");

    // Content Security Policy — restricts script/style/image sources
    // This is complex; tune based on your actual sources
    ctx.Response.Headers.Append("Content-Security-Policy",
        "default-src 'self'; " +
        "script-src 'self' https://cdn.example.com; " +
        "style-src 'self' 'unsafe-inline'; " +
        "img-src 'self' data: https:; " +
        "connect-src 'self' https://api.example.com; " +
        "frame-ancestors 'none'; " +
        "base-uri 'self'; " +
        "form-action 'self'");

    // Remove headers that reveal tech stack
    ctx.Response.Headers.Remove("Server");       // Don't advertise Kestrel
    ctx.Response.Headers.Remove("X-Powered-By"); // Don't advertise .NET

    await next(ctx);
});

// Also configure Kestrel to not send Server header:
builder.WebHost.ConfigureKestrel(opts =>
    opts.AddServerHeader = false);
```

## 14. CONFIGURATION & SECRETS — PRODUCTION PATTERNS

### What It Actually Is
Secrets management is the practice of keeping sensitive values (connection strings, API keys, certificates) out of source control and deployable artifacts. In development, **User Secrets** (stored outside the repo on your local machine) is correct. In production, **Azure Key Vault** (or AWS Secrets Manager, HashiCorp Vault) is correct. Never put production secrets in `appsettings.Production.json`. Never put ANY secrets in committed files.

### The Real-World Code

```csharp
// ── Development: User Secrets ─────────────────────────────────────────────────
// Stored in: %APPDATA%\Microsoft\UserSecrets\{userSecretsId}\secrets.json
// NOT in the repository — per-developer, per-machine

// .csproj:
// <UserSecretsId>my-app-unique-id-here</UserSecretsId>

// CLI commands:
// dotnet user-secrets set "ConnectionStrings:Default" "Server=localhost;..."
// dotnet user-secrets set "Jwt:Secret" "dev-secret-min-32-chars-long!!"
// dotnet user-secrets list
// dotnet user-secrets remove "Jwt:Secret"


// ── Production: Azure Key Vault ───────────────────────────────────────────────
// Key Vault secrets use -- as hierarchy separator (not : — Key Vault doesn't allow :)
// "ConnectionStrings--Default" → configuration["ConnectionStrings:Default"]

if (builder.Environment.IsProduction())
{
    var keyVaultUri = new Uri(builder.Configuration["KeyVault:Uri"]!);

    // DefaultAzureCredential: tries Managed Identity → Visual Studio → CLI → Environment
    builder.Configuration.AddAzureKeyVault(keyVaultUri, new DefaultAzureCredential());
}

// Azure Managed Identity setup (no secrets needed for Key Vault access itself):
// 1. Enable Managed Identity on the App Service / AKS pod
// 2. Grant Key Vault "Key Vault Secrets User" role to the Managed Identity
// 3. That's it — no client ID, no secret, no password


// ── Environment variable overrides ───────────────────────────────────────────
// In Kubernetes / Docker Compose, override config without touching files:
// MYAPP_ConnectionStrings__Default=Server=prod-db;...
// MYAPP_Jwt__Secret=prod-secret

// In docker-compose.yml:
// environment:
//   - MYAPP_ConnectionStrings__Default=${DB_CONNECTION_STRING}
//   - MYAPP_Jwt__Secret=${JWT_SECRET}
// (Values come from .env file or CI/CD pipeline secrets)


// ── Validate all secrets exist at startup ────────────────────────────────────
// Use ValidateOnStart() with required options to fail fast if secrets are missing
builder.Services.AddOptions<JwtOptions>()
    .BindConfiguration("Jwt")
    .Validate(opts =>
    {
        if (string.IsNullOrWhiteSpace(opts.Secret))
            return false;
        if (opts.Secret.Length < 32)
            return false;
        if (opts.Secret == "development-secret-please-change") // Common mistake
            return builder.Environment.IsDevelopment(); // Allow in dev, reject in prod
        return true;
    }, "JWT secret is missing or too short (minimum 32 characters)")
    .ValidateOnStart();


// ── Secret rotation pattern ───────────────────────────────────────────────────
// Key Vault supports multiple versions of a secret
// Rotate without downtime:
// 1. Add new secret version in Key Vault
// 2. IOptionsMonitor<JwtOptions> detects change if using AddAzureKeyVault with reload
// 3. New tokens signed with new key; old tokens still validated during transition
//    (keep both keys in validation list during rotation window)
public class JwtTokenService
{
    // IOptionsMonitor reads the latest value on each access — picks up rotated secrets
    private JwtOptions Options => _monitor.CurrentValue;
    private readonly IOptionsMonitor<JwtOptions> _monitor;
    public JwtTokenService(IOptionsMonitor<JwtOptions> monitor) => _monitor = monitor;
}
```

## 16. PERFORMANCE CHECKLIST & DIAGNOSTICS

### The Checklist

```
REQUEST PIPELINE
 □ CancellationToken passed to ALL async calls
 □ No synchronous I/O (no .Result, no .GetAwaiter().GetResult())
 □ No unnecessary middleware registered
 □ Static files served before auth middleware
 □ Output cache on read-heavy endpoints
 □ Response compression enabled

DATABASE
 □ AsNoTracking() on read-only queries
 □ Select only needed columns (.Select(x => new { x.Id, x.Name }))
 □ Pagination on all list endpoints (no SELECT * without LIMIT)
 □ Index on filtered/sorted columns verified in query plan
 □ N+1 queries eliminated (use .Include() or split queries)
 □ Connection pool size configured (default is 100, tune per load)
 □ EF Core query compilation cached (use compiled queries for hot paths)

HTTPCLIENT
 □ Using IHttpClientFactory (never new HttpClient())
 □ Resilience pipeline configured (retry + circuit breaker)
 □ Timeouts configured per client (not just the global Kestrel timeout)
 □ JSON deserialization using source generation (not reflection) for hot paths

MEMORY
 □ Large object heap pressure monitored (arrays > 85 KB go to LOH)
 □ ArrayPool used for temporary byte buffers
 □ MemoryStream replaced with RecyclableMemoryStream for large streams
 □ Avoid string concatenation in hot paths (use StringBuilder or interpolation once)
 □ IAsyncEnumerable for streaming large result sets (not List<T> materializations)

SERIALIZATION
 □ System.Text.Json (not Newtonsoft) for hot paths
 □ JsonSerializerOptions cached as static field (not new per call)
 □ Source generation enabled for frequently serialized types
```

## 17. THE REQUEST CANCELLATION ANTI-PATTERNS

A reference of what NOT to do — these bugs are subtle and show up under load.

```csharp
// ══════════════════════════════════════════════════════════════════
// ANTI-PATTERN 1: Not passing CancellationToken
// ══════════════════════════════════════════════════════════════════
// WRONG
public async Task<Order?> GetOrderAsync(int id)
    => await _db.Orders.FindAsync(id); // Query continues after client disconnects

// RIGHT
public async Task<Order?> GetOrderAsync(int id, CancellationToken ct)
    => await _db.Orders.FindAsync(new object[] { id }, ct);


// ══════════════════════════════════════════════════════════════════
// ANTI-PATTERN 2: Catching and swallowing OperationCanceledException
// ══════════════════════════════════════════════════════════════════
// WRONG — logs an error for normal client disconnect
try { await _next(ctx); }
catch (Exception ex) { _logger.LogError(ex, "Request failed"); throw; }

// RIGHT — distinguish expected cancellation from real errors
try { await _next(ctx); }
catch (OperationCanceledException) when (ctx.RequestAborted.IsCancellationRequested)
{
    // Client disconnected — normal, not an error
    _logger.LogDebug("Request cancelled by client");
}
catch (Exception ex)
{
    _logger.LogError(ex, "Request failed unexpectedly");
    throw;
}


// ══════════════════════════════════════════════════════════════════
// ANTI-PATTERN 3: Using CancellationToken on SaveChangesAsync for write operations
// ══════════════════════════════════════════════════════════════════
// WRONG — if client disconnects during POST, data is NOT saved
public async Task<Order> CreateOrderAsync(CreateOrderCommand cmd, CancellationToken ct)
{
    var order = Order.Create(cmd);
    _db.Add(order);
    await _db.SaveChangesAsync(ct); // WRONG: cancelled if client disconnects
    return order;
}

// RIGHT — always commit writes regardless of client state
public async Task<Order> CreateOrderAsync(CreateOrderCommand cmd, CancellationToken ct)
{
    var order = Order.Create(cmd);
    _db.Add(order);
    await _db.SaveChangesAsync(CancellationToken.None); // Always commit
    // NOW use ct for optional post-write operations
    await _eventBus.PublishAsync(new OrderCreatedEvent(order.Id), ct);
    return order;
}


// ══════════════════════════════════════════════════════════════════
// ANTI-PATTERN 4: Blocking on async code (deadlock territory)
// ══════════════════════════════════════════════════════════════════
// WRONG — can deadlock in certain synchronization contexts
public Order GetOrder(int id)
    => _repo.GetOrderAsync(id, CancellationToken.None).Result; // .Result blocks

// RIGHT — async all the way down
public async Task<Order> GetOrderAsync(int id, CancellationToken ct)
    => await _repo.GetOrderAsync(id, ct);


// ══════════════════════════════════════════════════════════════════
// ANTI-PATTERN 5: Returning Task without await (exception swallowing)
// ══════════════════════════════════════════════════════════════════
// WRONG — exceptions thrown after the first await are swallowed
public Task<IActionResult> GetOrder(int id)
{
    // If GetOrderAsync throws, the exception propagates correctly here
    return _service.GetOrderAsync(id, CancellationToken.None)
        .ContinueWith(t => (IActionResult)Ok(t.Result));
}

// RIGHT — using await catches all exceptions
public async Task<IActionResult> GetOrder(int id)
{
    var order = await _service.GetOrderAsync(id, CancellationToken.None);
    return Ok(order);
}
```

## 18. COMPLETE ASPNET CORE GOTCHAS

| Gotcha | Effect | Fix |
|---|---|---|
| `UseAuthentication` after `UseAuthorization` | All endpoints return 401 even for public ones | Always: Auth then Authz |
| `app.Map` branches don't rejoin | Middleware added after `Map` doesn't run for mapped paths | Use `app.UseWhen` instead |
| `IOptions<T>` in a singleton | Config never updates even with `reloadOnChange: true` | Use `IOptionsMonitor<T>` in singletons |
| Injecting scoped into singleton | Scoped service lives forever (captive dependency) | Use `IServiceScopeFactory` |
| `async void` in middleware or event handlers | Exceptions are unobserved → crash with no stack trace | Always `async Task`, never `async void` |
| `HttpContext` access outside request | `IHttpContextAccessor.HttpContext` is null | Only access in request-scoped code |
| Not awaiting `SaveChangesAsync` | Changes are not persisted; no exception thrown | Always await persistence calls |
| Logging with string interpolation | Properties are embedded as strings, not searchable | Use `LogInformation("{Name}", value)` |
| `new JsonSerializerOptions()` in hot path | Reflection metadata not cached → massive perf hit | Cache options as a `static readonly` field |
| Forgetting `.AllowAnonymous()` with fallback policy | All endpoints require auth when `FallbackPolicy` is set | Explicitly mark public endpoints |
| `ConfigureAwait(false)` in ASP.NET Core | Not needed — ASP.NET Core has no synchronization context | Don't add noise; it's a no-op in ASP.NET Core |
| Reading `Request.Body` twice without `EnableBuffering()` | Second read returns empty | Call `EnableBuffering()` before first read |
| Setting headers after `Response.HasStarted` | `InvalidOperationException` at runtime | Use `OnStarting` callback instead |
| `DateTimeOffset` vs `DateTime` in JSON | Timezone information lost with `DateTime` | Always use `DateTimeOffset` for timestamps |
| `CancellationToken` in `SaveChangesAsync` for writes | Write aborted if client disconnects before DB commits | Use `CancellationToken.None` for writes |
