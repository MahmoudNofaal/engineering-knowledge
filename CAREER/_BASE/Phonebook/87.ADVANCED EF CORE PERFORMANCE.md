## 87.01 — EF Core Performance Patterns

**Query compilation caching**
```csharp
// EF Core caches compiled queries automatically
// But: complex dynamic queries create many cache entries (memory leak)

// Use Compiled Queries for hot paths (pre-compiled, zero translation overhead)
private static readonly Func<AppDbContext, int, Task<Order?>> GetOrderByIdCompiled =
    EF.CompileAsyncQuery((AppDbContext ctx, int id) =>
        ctx.Orders.Include(o => o.Items).FirstOrDefault(o => o.Id == id));

// Usage — no translation overhead on each call
var order = await GetOrderByIdCompiled(_db, orderId);

// Another compiled query with projection
private static readonly Func<AppDbContext, int, IAsyncEnumerable<OrderSummaryDto>> GetOrderSummariesCompiled =
    EF.CompileAsyncQuery((AppDbContext ctx, int customerId) =>
        ctx.Orders
            .Where(o => o.CustomerId == customerId)
            .Select(o => new OrderSummaryDto
            {
                Id = o.Id,
                Status = o.Status.ToString(),
                Total = o.Total,
                CreatedAt = o.CreatedAt
            }));

await foreach (var dto in GetOrderSummariesCompiled(_db, customerId))
    yield return dto;
```

**Batch operations (EF Core 7+)**
```csharp
// Bulk updates without loading entities (single SQL UPDATE)
await _db.Orders
    .Where(o => o.Status == OrderStatus.Pending && o.CreatedAt < DateTime.UtcNow.AddDays(-7))
    .ExecuteUpdateAsync(setters =>
        setters.SetProperty(o => o.Status, OrderStatus.Expired)
               .SetProperty(o => o.ExpiredAt, DateTime.UtcNow));
// Generates: UPDATE Orders SET Status = 'Expired', ExpiredAt = @now
//            WHERE Status = 'Pending' AND CreatedAt < @cutoff
// Single round trip, no tracking, no loading all entities

// Bulk delete
await _db.Notifications
    .Where(n => n.IsRead && n.CreatedAt < DateTime.UtcNow.AddDays(-30))
    .ExecuteDeleteAsync();
// Generates: DELETE FROM Notifications WHERE IsRead = 1 AND CreatedAt < @cutoff
```

**Split queries for complex includes**
```csharp
// Problem: Single query with multiple includes → Cartesian explosion
// Order with 5 items, each item with 3 tags = 15 rows for 1 order!

// WRONG for large collections:
var orders = await _db.Orders
    .Include(o => o.Items)
    .ThenInclude(i => i.Tags)
    .Include(o => o.Addresses)    // Multiple collection includes = huge result set
    .ToListAsync();

// RIGHT: Split into multiple SQL queries
var orders = await _db.Orders
    .Include(o => o.Items)
    .ThenInclude(i => i.Tags)
    .Include(o => o.Addresses)
    .AsSplitQuery()  // EF Core executes as separate JOINed queries
    .ToListAsync();
// Generates 3 queries: orders, items+tags, addresses
// No Cartesian explosion

// Or globally:
optionsBuilder.UseSqlServer(connectionString, opts =>
    opts.UseQuerySplittingBehavior(QuerySplittingBehavior.SplitQuery));
```

**NoTracking with identity resolution**
```csharp
// NoTracking: fastest reads, no change tracking, but loses identity map
// Problem: Same entity loaded twice = two different objects (breaks reference equality)

var orders = await _db.Orders
    .AsNoTracking()
    .Include(o => o.Customer)
    .ToListAsync();
// orders[0].Customer and orders[1].Customer may be same customer
// but are DIFFERENT objects in memory

// AsNoTrackingWithIdentityResolution: still fast, but entities with same PK = same object
var orders = await _db.Orders
    .AsNoTrackingWithIdentityResolution()  // New in EF Core 5
    .Include(o => o.Customer)
    .ToListAsync();
// Now orders[0].Customer == orders[1].Customer (same object reference)
// Useful when you care about reference equality or navigation through shared entities
```

**Interceptors for cross-cutting DB concerns**
```csharp
// Query interceptor — modify queries globally, log slow queries
public class QueryLoggingInterceptor : DbCommandInterceptor
{
    private readonly ILogger<QueryLoggingInterceptor> _logger;
    private readonly TimeSpan _slowQueryThreshold = TimeSpan.FromSeconds(1);
    
    public override async ValueTask<DbDataReader> ReaderExecutedAsync(
        DbCommand command,
        CommandExecutedEventData eventData,
        DbDataReader result,
        CancellationToken ct)
    {
        if (eventData.Duration > _slowQueryThreshold)
        {
            _logger.LogWarning(
                "Slow query detected ({Duration}ms):\n{Sql}",
                eventData.Duration.TotalMilliseconds,
                command.CommandText);
        }
        return result;
    }
}

// Register
optionsBuilder.AddInterceptors(new QueryLoggingInterceptor(logger));
```
