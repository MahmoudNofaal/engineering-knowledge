## 12.01 — ACID Properties

**Atomicity**
A transaction either completes entirely or not at all. If any step fails, all changes are rolled back. No partial updates.

**Consistency**
A transaction takes the database from one valid state to another. All integrity constraints, foreign keys, and rules are maintained.

**Isolation**
Concurrent transactions execute as if they were sequential. One transaction's intermediate states are not visible to others (degree depends on isolation level).

**Durability**
Once committed, a transaction survives system failures. WAL (Write-Ahead Log) ensures data is on disk before acknowledging commit.

## 12.02 — Normalization

**What it is**
Organizing database tables to reduce redundancy and improve data integrity.

**1NF — First Normal Form**
- Each column contains atomic (indivisible) values
- No repeating groups
- Has a primary key

Violation: `tags = "backend, dotnet, azure"` — not atomic, comma-separated.
Fix: separate Tags table with one tag per row.

**2NF — Second Normal Form**
- Must be in 1NF
- Every non-key attribute is fully functionally dependent on the entire primary key (no partial dependencies)

Violation: Table with composite key (OrderId, ProductId) and CustomerName. CustomerName depends only on OrderId, not the full key.
Fix: Move CustomerName to the Orders table.

**3NF — Third Normal Form**
- Must be in 2NF
- No transitive dependencies (non-key column determines another non-key column)

Violation: Employee table with DepartmentId and DepartmentName. DepartmentName depends on DepartmentId, not EmployeeId.
Fix: Separate Departments table.

**BCNF — Boyce-Codd Normal Form**
Stronger version of 3NF. Every determinant must be a candidate key.

**When to denormalize**
Normalization improves write consistency and reduces storage but hurts read performance (joins are expensive). Denormalization intentionally introduces redundancy for read performance:
- Pre-computed aggregates (TotalOrderValue stored on Order)
- Duplicated columns to avoid joins
- Materialized views
- Read model in CQRS

## 12.03 — Indexing

**Clustered Index**
The physical order of rows in the table matches the index order. Only one per table. In SQL Server, the primary key creates a clustered index by default.

```sql
-- The table is physically sorted by OrderId
CREATE TABLE Orders (
    OrderId INT PRIMARY KEY,  -- clustered index
    CustomerId INT,
    OrderDate DATE
);
```

**Non-clustered Index**
A separate B+ tree structure with key values and pointers to the actual rows. Multiple per table.

```sql
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId
ON Orders (CustomerId);
-- For: SELECT * FROM Orders WHERE CustomerId = 42
```

**Covering Index**
A non-clustered index that includes all columns needed by a query — eliminates the need to look up the actual row.

```sql
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId_Date
ON Orders (CustomerId)
INCLUDE (OrderDate, Status);  -- Covered columns
-- Query: SELECT OrderDate, Status FROM Orders WHERE CustomerId = 42
-- No key lookup needed - all columns in the index
```

**Composite Index**
Multiple columns in one index. Column order matters — the index is useful for queries on the leading columns.

```sql
CREATE NONCLUSTERED INDEX IX_Orders_Status_Date
ON Orders (Status, OrderDate);
-- Useful for: WHERE Status = 'Pending' ORDER BY OrderDate
-- NOT useful for: WHERE OrderDate = '2026-01-01' (leading column missing)
```

**When indexes hurt**
- Every write (INSERT, UPDATE, DELETE) must update all relevant indexes
- Indexes use disk space
- Too many indexes → worse write performance
- The optimizer may choose the wrong index if statistics are stale

**Index fragmentation**
As rows are inserted/deleted, pages become fragmented (internal: page not full, external: logical order ≠ physical order).
```sql
-- Check fragmentation
SELECT avg_fragmentation_in_percent
FROM sys.dm_db_index_physical_stats(DB_ID(), OBJECT_ID('Orders'), NULL, NULL, 'SAMPLED');

-- Rebuild (heavy: offline option, resets fragmentation to 0)
ALTER INDEX IX_Orders_CustomerId ON Orders REBUILD;

-- Reorganize (light: online, partial defrag)
ALTER INDEX IX_Orders_CustomerId ON Orders REORGANIZE;
```

## 12.04 — Transaction Isolation Levels

**Read Uncommitted**
Can read rows being modified by uncommitted transactions. Risk: **dirty reads**.
```sql
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
-- Or: SELECT * FROM Orders WITH (NOLOCK)
```
Use only when approximate data is acceptable (analytics dashboards).

**Read Committed** (SQL Server default)
Only reads committed data. Prevents dirty reads. Still allows non-repeatable reads and phantoms.

**Read Committed Snapshot Isolation (RCSI)**
MVCC-based version of Read Committed. Readers don't block writers. Version store in tempdb provides consistent reads.
```sql
ALTER DATABASE MyDb SET READ_COMMITTED_SNAPSHOT ON;
-- No code changes needed - transparent to application
```

**Repeatable Read**
Prevents dirty and non-repeatable reads. Rows read in a transaction cannot be modified by others until the transaction completes. Allows phantom reads.

**Snapshot Isolation**
Entire transaction sees a consistent snapshot of the database as it was when the transaction started. Prevents all anomalies. Optimistic: conflicts detected at commit time.
```sql
SET TRANSACTION ISOLATION LEVEL SNAPSHOT;
```

**Serializable**
Highest isolation. Transactions execute as if serial. Prevents all anomalies including phantom reads. Most restrictive — range locks.

## 12.05 — Query Execution Plans

**What it is**
The strategy the query optimizer chooses for executing a query. Reading plans is essential for query optimization.

**Access patterns**
- **Index Seek**: The optimizer uses the index to jump directly to matching rows. Efficient.
- **Index Scan**: The optimizer scans the entire index. Needed when selectivity is low.
- **Table Scan**: No usable index — scans the entire table. Expensive on large tables.
- **Key Lookup (RID Lookup)**: Non-clustered index found the row, but needed columns not in index → extra lookup to the clustered index. Fix: use covering index.

**Join algorithms**
- **Nested Loops**: For each outer row, scan inner table. Best for small outer input and indexed inner.
- **Hash Match**: Build a hash table from smaller input, probe with larger. Best for large, unsorted inputs.
- **Merge Join**: Both inputs sorted on join key. Very fast. Requires sorted inputs.

**Reading a plan**
- Read right-to-left, bottom-to-top
- Thick arrows = large row counts = potential bottlenecks
- Look for table scans, key lookups, sort operations

**Enabling statistics**
```sql
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

-- Run your query here

-- Output example:
-- Table 'Orders'. Scan count 1, logical reads 1234 (= 1234 × 8KB pages read)
-- SQL Server Execution Times: CPU time = 250 ms, elapsed time = 380 ms
```

**Parameter sniffing**
SQL Server compiles a query plan based on the first parameter values. If subsequent calls use very different parameters, the cached plan may be suboptimal.

```sql
-- Solutions:
-- 1. OPTION (RECOMPILE) -- recompile each time, no caching
SELECT * FROM Orders WHERE CustomerId = @id OPTION (RECOMPILE);

-- 2. OPTIMIZE FOR UNKNOWN -- use average statistics
SELECT * FROM Orders WHERE CustomerId = @id OPTION (OPTIMIZE FOR (@id UNKNOWN));

-- 3. Local variable -- breaks parameter sniffing
DECLARE @localId INT = @id;
SELECT * FROM Orders WHERE CustomerId = @localId;
```
