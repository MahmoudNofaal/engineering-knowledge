## 10.01 — System Design Process

**The structured approach (for interviews and real design)**

**Step 1 — Clarify requirements**
```
Functional requirements: What does the system do?
  - Users can post messages
  - Users can follow other users
  - Timeline shows followed users' posts

Non-functional requirements: How well does it do it?
  - 100M daily active users
  - < 200ms latency for timeline
  - 99.9% availability
  - Strong consistency vs. eventual consistency?
```

**Step 2 — Scale estimation**
```
Daily Active Users: 100M
Tweets per user per day: 1
Total tweets/day: 100M
Tweets/second: 100M / 86,400 ≈ 1,200 RPS write

Timeline reads per user per day: 50
Timeline reads/second: 100M × 50 / 86,400 ≈ 58,000 RPS read
Read:write ratio ≈ 50:1 — heavily read-dominant

Tweet size: 140 chars UTF-8 = ~280 bytes + metadata ≈ 1 KB
Storage/day: 100M × 1 KB = 100 GB/day
5-year storage: 100 GB × 365 × 5 ≈ 182 TB
```

**Step 3 — High-level design**
Draw the major components: clients, load balancer, API servers, databases, caches, message queues.

**Step 4 — Deep dive**
Pick the interesting/bottleneck component and design it in detail.

**Step 5 — Identify bottlenecks**
- What if one database gets too many writes?
- What happens if the cache is cold?
- What if a service goes down?

**Step 6 — Address failure scenarios**
- How does the system handle database failure?
- How does it recover from network partition?
- What is the consistency model?

## 10.02 — URL Shortener Design

**Functional**: Given a long URL, return a short URL. Redirect from short URL to long URL.

**Scale**: 100M URLs shortened/day, 10B redirects/day.

**Key design decisions**

**ID generation**: How to generate short codes?
- Option 1: Auto-increment ID → Base62 encode. Problem: predictable, requires central counter.
- Option 2: Random string. Problem: collision checking required.
- Option 3: MD5/SHA hash of URL → take first 7 chars. Problem: collision possible, but rare.
- Option 4: Snowflake ID (timestamp + machine ID + sequence). Best: unique, sortable, distributed.

**Base62 encoding**
7 characters in Base62 = 62^7 = 3.5 trillion possible URLs — sufficient.

**Database schema**
```sql
CREATE TABLE urls (
    id BIGINT PRIMARY KEY,
    short_code VARCHAR(7) UNIQUE NOT NULL,
    original_url TEXT NOT NULL,
    created_at TIMESTAMP NOT NULL,
    expires_at TIMESTAMP,
    user_id BIGINT
);
CREATE INDEX idx_short_code ON urls(short_code);
```

**Caching strategy**
80-20 rule: 20% of URLs account for 80% of redirects. Cache hot short codes in Redis.
- On redirect: check Redis first. Cache hit → return. Cache miss → query DB → cache → return.
- TTL: 24 hours. Auto-extend on access.

**Architecture**
```
Client → CDN → Load Balancer → API Servers → Redis Cache → Database
                                                  ↓
                                           DB (on miss)
```

**Write path**
1. Receive long URL
2. Generate short code
3. Check collision (Redis/DB)
4. Store in DB
5. Cache in Redis
6. Return short URL

**Read path**
1. Receive short code
2. Check Redis: hit → return long URL (302 redirect)
3. Redis miss → query DB → cache in Redis → return

## 10.03 — Rate Limiter Design

**Algorithms**

**Fixed Window**
```
Window: [0:00 - 1:00] → allow 100 requests
Window: [1:00 - 2:00] → allow 100 requests (counter resets)
Problem: 100 requests at 0:59, 100 at 1:01 = 200 in 2 seconds
```

**Sliding Window Log**
Store timestamp of each request. Count requests in last 60 seconds. Accurate but memory-intensive.

**Sliding Window Counter**
```
Current window count + (previous window count × overlap percentage)
More accurate than fixed window, less memory than log
```

**Token Bucket**
- Bucket holds N tokens. Refills at rate R tokens/second.
- Each request takes 1 token. If no tokens, reject.
- Allows controlled bursts.

**Leaky Bucket**
- Requests queue up. Process at fixed rate.
- Smooths out bursts. Good for downstream rate limiting.

**Distributed rate limiter (Redis)**
```csharp
// Redis sorted set approach — sliding window
public async Task<bool> IsAllowedAsync(string userId, string endpoint)
{
    var key = $"rate:{userId}:{endpoint}";
    var now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
    var windowStart = now - 60_000; // 60 seconds
    
    var db = _redis.GetDatabase();
    
    await db.SortedSetRemoveRangeByScoreAsync(key, 0, windowStart);
    var count = await db.SortedSetLengthAsync(key);
    
    if (count >= _limit)
        return false;
    
    await db.SortedSetAddAsync(key, Guid.NewGuid().ToString(), now);
    await db.KeyExpireAsync(key, TimeSpan.FromMinutes(2));
    return true;
}
```

## 10.04 — Notification Service Design

**Functional**
- Send push notifications, emails, SMS
- Millions of users
- User preferences (opt-out)
- Delivery guarantees

**Architecture**
```
Trigger Services (Order, Payment, etc.)
    ↓ publish events
Message Queue (Kafka/Azure Service Bus)
    ↓ consume
Notification Service
    ├── User preference check
    ├── Template rendering
    ├── Rate limiting (don't spam users)
    └── Channel routing
         ├── Push → FCM/APNs
         ├── Email → SendGrid/SES
         └── SMS → Twilio
    ↓
Delivery tracking table
```

**Key design decisions**
- Async delivery: don't block the triggering service
- Retry with exponential backoff: delivery services fail
- Deduplication: same event processed twice = duplicate notification
- User preferences: store per-channel, per-event-type opt-out
- Template engine: store templates in DB, render server-side

## 10.05 — Chat System Design (WhatsApp scale)

**Functional**
- 1:1 messaging
- Group messaging (up to 500 members)
- Message delivery status (sent, delivered, read)
- Online presence

**Protocol**: WebSockets for bi-directional real-time communication.

**Message flow**
```
User A types message
    → WebSocket to Chat Server A
    → Message stored in DB (persistent)
    → User B online on Server B? → push via WebSocket
    → User B offline? → push notification (FCM/APNs)
```

**Message storage**
- Messages: append-only. Use Cassandra (or similar) partitioned by conversation_id.
- Schema: conversation_id (partition key), message_id (clustering key, timestamp-based), sender_id, content, status

**Cassandra schema**
```sql
CREATE TABLE messages (
    conversation_id UUID,
    message_id TIMEUUID,
    sender_id UUID,
    content TEXT,
    message_type TEXT,
    status TEXT,
    PRIMARY KEY (conversation_id, message_id)
) WITH CLUSTERING ORDER BY (message_id DESC);
```

**Presence service**
- Store last_active timestamp in Redis (TTL = 30 seconds)
- Heartbeat from client every 15 seconds keeps presence alive
- On WebSocket disconnect → mark offline

**Group messages**
- Fanout to all group members: write one message, fan out to all members' inboxes
- For large groups (>500): fanout on read (user fetches messages from conversation)

## 10.06 — News Feed Design (Twitter/Instagram)

**Two approaches**

**Push model (write fanout)**
When User A posts, push the post to all followers' feed caches immediately.
- Read: O(1) — feed is precomputed
- Write: expensive for celebrities (100M followers = 100M writes)

**Pull model (read fanout)**
When User A fetches feed, pull recent posts from all followed users.
- Write: O(1)
- Read: expensive — query N users' posts and merge

**Hybrid** (what large platforms use)
- Ordinary users: push model (fanout on write)
- Celebrities: pull model (fanout on read)
- Threshold: followers > 1M → pull model

**Feed generation with push model**
```
User posts tweet:
  1. Store tweet in tweets table
  2. Publish event to message queue
  3. Fanout service reads event
  4. For each follower: push tweet_id to feed cache (Redis sorted set, score = timestamp)
  
User reads feed:
  1. Read tweet_ids from Redis feed (sorted by score)
  2. Fetch tweet details from tweet cache / DB
  3. Return hydrated feed
```

**Redis sorted set as feed cache**
```
Key: feed:{user_id}
Sorted set: score = unix_timestamp, member = tweet_id
```

## 10.07 — Distributed Cache Design

**What it is**
A cache shared across multiple application instances, providing consistent data to all nodes.

**Redis as distributed cache**
```csharp
// In ASP.NET Core
builder.Services.AddStackExchangeRedisCache(options =>
{
    options.Configuration = builder.Configuration["Redis:ConnectionString"];
    options.InstanceName = "MyApp:";
});

// IDistributedCache usage
public class ProductService
{
    private readonly IDistributedCache _cache;
    
    public async Task<Product?> GetAsync(int id, CancellationToken ct)
    {
        var cacheKey = $"product:{id}";
        var cached = await _cache.GetStringAsync(cacheKey, ct);
        
        if (cached != null)
            return JsonSerializer.Deserialize<Product>(cached);
        
        var product = await _db.Products.FindAsync(id);
        if (product != null)
        {
            await _cache.SetStringAsync(
                cacheKey,
                JsonSerializer.Serialize(product),
                new DistributedCacheEntryOptions
                {
                    AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(10)
                },
                ct);
        }
        return product;
    }
}
```

**Cache patterns**
- **Cache-aside (lazy loading)**: App checks cache, fetches from DB on miss, updates cache. Most common.
- **Write-through**: Write to cache and DB simultaneously. Always consistent, always in cache.
- **Write-behind**: Write to cache, async write to DB. Fast writes, risk of data loss.
- **Read-through**: Cache sits between app and DB, fetches automatically on miss.

**Cache invalidation**
- TTL expiry (simplest)
- Event-based: invalidate cache when DB changes
- Version tags: tag cache entries, invalidate by tag

**Cache stampede prevention**
When cache expires, many concurrent requests miss and all hit the database.
```csharp
// Probabilistic early expiration
// Or: use a lock to let only one request refresh
private static readonly SemaphoreSlim _lock = new(1, 1);

public async Task<Product?> GetWithLockAsync(int id, CancellationToken ct)
{
    var cached = await _cache.GetAsync<Product>($"product:{id}", ct);
    if (cached != null) return cached;
    
    await _lock.WaitAsync(ct);
    try
    {
        // Double-check inside lock
        cached = await _cache.GetAsync<Product>($"product:{id}", ct);
        if (cached != null) return cached;
        
        var product = await _db.Products.FindAsync(id);
        await _cache.SetAsync($"product:{id}", product, TimeSpan.FromMinutes(10), ct);
        return product;
    }
    finally
    {
        _lock.Release();
    }
}
```
