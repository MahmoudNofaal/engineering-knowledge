## 78.01 — Technical Leadership Without Authority

**You don't need a title to lead technically**

**Practices of technical leaders**
1. **Write things down.** Architecture decisions, debugging notes, runbooks. The impact of written knowledge compounds.

2. **Pull up, not just push down.** Help junior engineers grow. Review their code thoughtfully. Pair with them.

3. **Make decisions stick.** A decision not documented is not a decision. Use Architecture Decision Records (ADRs).

4. **Say no thoughtfully.** Not "we can't" but "here's what we trade if we do that."

5. **Simplify.** The senior instinct is often to make things more complex. The senior skill is to resist that instinct.

6. **Own reliability.** When production breaks, senior engineers run toward it, not away.

**Architecture Decision Records (ADRs)**
```markdown
# ADR-001: Use PostgreSQL instead of SQL Server

## Status: Accepted

## Context
We are starting a new project. We need a relational database.
Our team has experience with both. We need to choose.

## Decision
Use PostgreSQL.

## Rationale
- Open source: no licensing cost at scale
- Excellent JSON support (jsonb) for flexible schemas
- LISTEN/NOTIFY for simple event notifications
- Strong cloud support on all major providers
- Team has existing knowledge of both

## Consequences
- No SSMS (use pgAdmin or DataGrip)
- No SQL Server-specific features (SSIS, SSRS)
- EF Core provider requires Npgsql package (not built-in)

## Alternatives Considered
- SQL Server: Better tooling on Windows, but licensing cost at scale
- MySQL: Less feature-rich, team has no experience
```

## 78.02 — Managing Technical Debt

**Types of technical debt**
```
Reckless/Deliberate: "We'll cut corners now, fix later"
  → Most visible, least forgivable

Prudent/Deliberate: "We know this isn't perfect, ship now"
  → Legitimate, must be tracked and paid back

Reckless/Inadvertent: "We didn't know better when we wrote this"
  → Unavoidable, discovered through experience

Prudent/Inadvertent: "Now we know what we should have done"
  → Normal, happens as understanding grows
```

**Making debt visible**
```csharp
// Technical debt tracking in code
// TODO: [DEBT-2026-06-01] This should use a queue instead of synchronous processing
// See: https://jira.example.com/TECH-1234
// Impact: Under load, this will block requests for 2-3 seconds
// Fix: Replace with Hangfire background job

// Or use a custom attribute for searchable debt
[TechnicalDebt("TECH-1234", Description = "Should be async", Priority = DebtPriority.High)]
private void ProcessOrderSynchronously(Order order) { ... }

// Track in backlog: debt/feature ratio should not exceed 20% of sprint capacity
// Without this, debt grows faster than it's paid
```

**When to pay debt**
```
Boy Scout Rule: pay small debt while in the area (< 30 min fix)
Sprint allocation: 15-20% of every sprint for debt
Dedicated sprints: when debt prevents new feature development
Refactoring: paid with new features (restructure + add)

Never: pay debt right before a release
Always: measure impact before spending time (is this debt actually hurting us?)
```

## 78.03 — Communicating Technical Decisions

**The technical proposal template**
```markdown
## Problem
What specific problem are we solving? (One sentence)

## Current State
How does the system behave today?

## Proposal
What specific change do we propose?

## Why This Option
Why this specific solution and not the alternatives?

## Alternatives Considered
1. Option A: Why we didn't choose this
2. Option B: Why we didn't choose this

## Risks and Mitigations
Risk 1: What could go wrong? How do we mitigate?
Risk 2: ...

## Success Metrics
How will we know this worked?

## Rollback Plan
How do we undo this if it fails?

## Implementation Estimate
Time: N days / sprints
Who: Which team members
Dependencies: What else must happen first
```

**Talking to non-technical stakeholders**
```
BAD: "We need to refactor the codebase to reduce cyclomatic complexity"
GOOD: "We need to simplify the code to reduce bug rates and enable faster feature delivery"

BAD: "We're seeing O(n²) algorithm performance in the order list endpoint"
GOOD: "The order list page becomes noticeably slower when customers have more than ~500 orders. We have a fix."

BAD: "We have to pay technical debt"
GOOD: "We need to invest in code quality so that new features take half the time they take now"

KEY INSIGHT: Business language = outcomes, not processes
```
