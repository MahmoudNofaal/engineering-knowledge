## 91.01 — System Design Framework

**How to approach any system design problem**

**Step 1: Clarify requirements (5 min)**
```
Functional requirements:
- What EXACTLY does it do?
- Who are the users?
- What are the key user journeys?

Non-functional requirements:
- Scale: how many users? QPS? Data volume?
- Latency requirements: P99 < 200ms?
- Availability: 99.9%? 99.99%?
- Consistency: strong? eventual?
- Durability: can we lose any data?
- Read:write ratio?
```

**Step 2: Estimate scale (2 min)**
```
Users: 10M DAU
Requests: 10M × 100 actions/day = 1B requests/day
QPS: 1B / 86400 ≈ 12,000 QPS (peak: 3x = 36,000 QPS)
Storage: 10M users × 1KB profile = 10 GB user data
          1B events/day × 1KB = 1 TB/day
```

**Step 3: High-level design (10 min)**
```
Draw:
- Clients (web, mobile)
- Load balancer
- API servers (horizontally scaled)
- Cache (Redis)
- Primary database
- Message queue (if async needed)
- CDN (for static content)

Talk through the critical path of the main user journey
```

**Step 4: Deep dive (15 min)**
```
Pick the hardest 2-3 components and go deep:
- Database schema
- Caching strategy
- How to handle the specific scale challenge
- Data partitioning/sharding strategy
- Handling failures
```

**Step 5: Bottlenecks and improvements (5 min)**
```
- What breaks first at 10x scale?
- What would you do next?
- Trade-offs you made
```

## 91.02 — Design URL Shortener (like bit.ly)

**Requirements**
- Shorten: POST /shorten → short URL
- Redirect: GET /{code} → redirect to original
- Analytics: track click count
- 100M URLs stored
- 500M redirects/day (5,800 QPS, peak ~20,000)
- Short code: 7 characters (62^7 = 3.5T combinations)

**Core design**
```
Client → CDN/Load Balancer → API Servers → Redis Cache → PostgreSQL

Short code generation:
- Take MD5 of long URL → first 8 chars as hex → 16^8 = 4B combinations
- Problem: collisions for same URL → detect and retry with next 8 chars
- Better: base62 counter (incrementing ID → base62 encode)
  1 → "1", 100 → "1C", 3,521,614,606,207 → "zzzzzzz"

Schema:
urls table:
  id BIGINT PRIMARY KEY (auto-increment)
  short_code VARCHAR(10) UNIQUE
  long_url TEXT NOT NULL
  user_id INT
  created_at TIMESTAMP
  expires_at TIMESTAMP (nullable)

clicks table: (for analytics, separate)
  id BIGINT
  short_code VARCHAR(10)
  clicked_at TIMESTAMP
  ip_address INET
  country VARCHAR(2)
  referer VARCHAR(500)

Caching:
  Redis: short_code → long_url (most redirects served from cache)
  TTL: 24 hours (sliding)
  Hit rate: ~95% after warm-up (power law: small set of codes get most traffic)
  
Redirect flow:
  1. Check Redis cache
  2. Cache miss → query DB, cache result
  3. 301 redirect (cached by browsers — reduces server load)
     or 302 redirect (analytics-accurate — every request hits server)
  4. Async: publish click event to Kafka → analytics consumers
```

## 91.03 — Design a Notification System

**Requirements**
- Push notifications: FCM (Android), APNs (iOS), Web Push
- Email notifications
- SMS notifications
- In-app notifications
- Template system
- Delivery tracking
- 10M users, 1M notifications/day

**Architecture**
```
Events → Message Queue (Kafka/Service Bus) → Notification Workers → Delivery Channels
                                            → Notification Log (DB)

Producers:
  OrderService → "order.created" event
  PaymentService → "payment.failed" event
  
Coordinator (consumes events):
  1. Look up user notification preferences
  2. Determine which channels to use (email? push? both?)
  3. Hydrate template with event data
  4. Publish to channel-specific queues

Channel Workers (one per channel):
  Email Worker: sends via SendGrid/SES
  Push Worker: sends via FCM/APNs
  SMS Worker: sends via Twilio
  In-App Worker: stores in Redis, delivered via WebSocket/SSE

Delivery tracking:
  notifications table: id, userId, type, channel, templateId, status, sentAt, deliveredAt
  Status: Pending → Sent → Delivered/Failed
  Webhook callbacks from FCM, APNs, Twilio update status

Failure handling:
  Per channel: retry queue with exponential backoff
  Dead letter queue: notifications that failed all retries
  Alerting: DLQ size > threshold → alert on-call

User preferences:
  preferences table: userId, channel (email/push/sms), notificationType, enabled
  Respect quiet hours: don't send push between 10pm-8am (user's timezone)
```

## 91.04 — Common Coding Interview Patterns

**The meta-pattern for solving problems**
```
1. Read the problem twice
2. Think of the simplest possible solution (even brute force)
3. Think about its time and space complexity
4. Think: can we do better?
5. Usually: what data structure makes lookups O(1)?
   → HashMap / HashSet for most problems
6. For sequences: two pointers, sliding window
7. For optimization: dynamic programming
8. For trees/graphs: DFS or BFS
9. Code the solution cleanly
10. Test with examples and edge cases (empty, single element, all same)
```

**The HashMap pattern**
```csharp
// 90% of "find duplicates", "two sum", "anagram", "frequency count" problems
// need a HashMap

// Two sum
int[] TwoSum(int[] nums, int target)
{
    var seen = new Dictionary<int, int>();  // value → index
    for (int i = 0; i < nums.Length; i++)
    {
        int complement = target - nums[i];
        if (seen.ContainsKey(complement))
            return new[] { seen[complement], i };
        seen[nums[i]] = i;
    }
    return Array.Empty<int>();
}

// Anagram groups
IList<IList<string>> GroupAnagrams(string[] strs)
{
    var groups = new Dictionary<string, List<string>>();
    foreach (var str in strs)
    {
        var key = new string(str.OrderBy(c => c).ToArray());  // Sorted = canonical form
        if (!groups.ContainsKey(key)) groups[key] = new();
        groups[key].Add(str);
    }
    return groups.Values.ToList<IList<string>>();
}
```

**The "Monotonic Stack" pattern**
```csharp
// Next greater element
int[] NextGreaterElement(int[] nums)
{
    var result = new int[nums.Length];
    Array.Fill(result, -1);
    var stack = new Stack<int>();  // Stack of indices
    
    for (int i = 0; i < nums.Length; i++)
    {
        while (stack.Count > 0 && nums[stack.Peek()] < nums[i])
        {
            result[stack.Pop()] = nums[i];  // Found the next greater
        }
        stack.Push(i);
    }
    return result;
}

// Largest rectangle in histogram
int LargestRectangle(int[] heights)
{
    var stack = new Stack<int>();  // Monotonically increasing indices
    int maxArea = 0;
    
    for (int i = 0; i <= heights.Length; i++)
    {
        int h = i == heights.Length ? 0 : heights[i];
        
        while (stack.Count > 0 && heights[stack.Peek()] > h)
        {
            int height = heights[stack.Pop()];
            int width = stack.Count == 0 ? i : i - stack.Peek() - 1;
            maxArea = Math.Max(maxArea, height * width);
        }
        stack.Push(i);
    }
    return maxArea;
}
```
