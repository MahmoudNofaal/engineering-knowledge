## 9.01 — Clean Architecture

**What it is**
An architectural pattern that enforces a strict dependency rule: dependencies always point inward. The inner layers know nothing about the outer layers.

**The layers (outside to inside)**

1. **Presentation** (controllers, minimal API endpoints, Blazor components)
2. **Infrastructure** (databases, external APIs, file systems, email services)
3. **Application** (use cases, commands, queries, DTOs)
4. **Domain** (entities, value objects, domain events, domain services)

**The dependency rule**
- Presentation depends on Application
- Infrastructure depends on Application
- Application depends on Domain
- Domain depends on nothing

```
Domain ← Application ← Infrastructure
Domain ← Application ← Presentation
```

**In .NET project structure**
```
Solution/
  src/
    YourApp.Domain/          # Entities, VOs, domain events, interfaces
    YourApp.Application/     # Commands, queries, DTOs, validators, handlers
    YourApp.Infrastructure/  # EF Core, Redis, external APIs, email
    YourApp.Api/             # Controllers/Minimal APIs, DI registration
  tests/
    YourApp.UnitTests/
    YourApp.IntegrationTests/
```

**Domain layer has no dependencies**
```csharp
// Domain/Entities/Order.cs — no NuGet packages, no framework references
public class Order
{
    private readonly List<OrderItem> _items = new();
    
    public int Id { get; private set; }
    public int CustomerId { get; private set; }
    public OrderStatus Status { get; private set; }
    public IReadOnlyList<OrderItem> Items => _items.AsReadOnly();
    public decimal Total => _items.Sum(i => i.Price * i.Quantity);
    
    public static Order Create(int customerId)
    {
        if (customerId <= 0) throw new DomainException("Invalid customer");
        return new Order { CustomerId = customerId, Status = OrderStatus.Pending };
    }
    
    public void AddItem(int productId, decimal price, int quantity)
    {
        if (Status != OrderStatus.Pending)
            throw new DomainException("Cannot add items to a non-pending order");
        _items.Add(new OrderItem(productId, price, quantity));
    }
    
    public void Ship()
    {
        if (Status != OrderStatus.Confirmed)
            throw new DomainException("Can only ship confirmed orders");
        Status = OrderStatus.Shipped;
        AddDomainEvent(new OrderShippedEvent(Id));
    }
}
```

**Application layer — use cases as commands/queries**
```csharp
// Application/Commands/CreateOrderCommand.cs
public record CreateOrderCommand(int CustomerId, List<OrderItemDto> Items)
    : IRequest<int>;  // Returns orderId

// Application/Handlers/CreateOrderHandler.cs
public class CreateOrderHandler : IRequestHandler<CreateOrderCommand, int>
{
    private readonly IOrderRepository _orders;
    private readonly IUnitOfWork _unitOfWork;
    
    public async Task<int> Handle(CreateOrderCommand cmd, CancellationToken ct)
    {
        var order = Order.Create(cmd.CustomerId);
        foreach (var item in cmd.Items)
            order.AddItem(item.ProductId, item.Price, item.Quantity);
        
        _orders.Add(order);
        await _unitOfWork.SaveChangesAsync(ct);
        return order.Id;
    }
}
```

## 9.02 — Domain-Driven Design (DDD)

**What it is**
An approach to software development that centers the design around the domain model — the business logic and rules — and uses a shared language between developers and domain experts.

**Ubiquitous Language**
A common vocabulary agreed upon by developers and domain experts. Every concept has one name used everywhere: in code, in conversations, in documentation. No translation layer.

**Bounded Context**
A clear boundary within which a domain model applies. "Customer" means different things in Sales (a prospect), Billing (a debtor), and Shipping (an address). Each context has its own model.

**Entities**
Objects defined by identity, not by their attributes. Two Users with the same name are different if they have different IDs.

```csharp
public class User  // Entity
{
    public UserId Id { get; private set; }  // Identity
    public string Email { get; private set; }
    public string Name { get; private set; }
    
    // Identity equality
    public override bool Equals(object? obj) =>
        obj is User user && Id == user.Id;
    
    public override int GetHashCode() => Id.GetHashCode();
}
```

**Value Objects**
Objects defined by their attributes, with no identity. Two Money objects with the same amount and currency are equal.

```csharp
public record Money(decimal Amount, string Currency)
{
    public Money Add(Money other)
    {
        if (Currency != other.Currency)
            throw new DomainException("Cannot add different currencies");
        return new Money(Amount + other.Amount, Currency);
    }
    
    public static Money operator +(Money a, Money b) => a.Add(b);
}

// Value objects as strongly-typed IDs
public record UserId(int Value);
public record OrderId(int Value);
// Prevents mixing up IDs of different entity types
```

**Aggregates**
A cluster of domain objects treated as a single unit. Has a root entity (aggregate root) that controls access to all inner objects. Only the aggregate root can be referenced from outside.

```csharp
public class Order  // Aggregate root
{
    private readonly List<OrderItem> _items = new();  // Part of aggregate
    
    // External code cannot add items directly — must go through root
    public void AddItem(ProductId productId, Money price, int qty)
    {
        // Invariant enforcement happens here
        if (_items.Count >= 100)
            throw new DomainException("Order cannot exceed 100 items");
        _items.Add(new OrderItem(productId, price, qty));
    }
}

public class OrderItem  // Part of Order aggregate — not accessible from outside
{
    internal OrderItem(ProductId productId, Money price, int qty) { ... }
}
```

**Domain Events**
Something that happened in the domain that other parts of the system need to know about.

```csharp
public abstract class DomainEvent
{
    public DateTime OccurredOn { get; } = DateTime.UtcNow;
}

public class OrderShippedEvent : DomainEvent
{
    public int OrderId { get; }
    public string TrackingNumber { get; }
    
    public OrderShippedEvent(int orderId, string trackingNumber)
    {
        OrderId = orderId;
        TrackingNumber = trackingNumber;
    }
}

// Base class for aggregates with domain events
public abstract class AggregateRoot
{
    private readonly List<DomainEvent> _events = new();
    
    protected void Raise(DomainEvent evt) => _events.Add(evt);
    
    public IReadOnlyList<DomainEvent> GetDomainEvents() => _events.AsReadOnly();
    public void ClearDomainEvents() => _events.Clear();
}
```

**Repository**
An abstraction over the persistence layer. The domain defines the interface; infrastructure implements it.

```csharp
// Domain defines the interface (no EF Core, no SQL references)
public interface IOrderRepository
{
    Task<Order?> GetByIdAsync(OrderId id, CancellationToken ct = default);
    Task<IReadOnlyList<Order>> GetByCustomerAsync(CustomerId customerId, CancellationToken ct = default);
    void Add(Order order);
}

// Infrastructure implements it
public class EfCoreOrderRepository : IOrderRepository
{
    private readonly AppDbContext _db;
    public EfCoreOrderRepository(AppDbContext db) => _db = db;
    
    public async Task<Order?> GetByIdAsync(OrderId id, CancellationToken ct)
        => await _db.Orders
            .Include(o => o.Items)
            .FirstOrDefaultAsync(o => o.Id == id.Value, ct);
    
    public void Add(Order order) => _db.Orders.Add(order);
}
```

**Specifications**
Encapsulate query logic as composable, reusable objects.

```csharp
public abstract class Specification<T>
{
    public abstract Expression<Func<T, bool>> ToExpression();
    
    public bool IsSatisfiedBy(T entity) =>
        ToExpression().Compile()(entity);
    
    public Specification<T> And(Specification<T> other) =>
        new AndSpecification<T>(this, other);
}

public class ActiveOrdersSpecification : Specification<Order>
{
    public override Expression<Func<Order, bool>> ToExpression() =>
        o => o.Status != OrderStatus.Cancelled && o.Status != OrderStatus.Completed;
}

public class HighValueOrdersSpecification : Specification<Order>
{
    private readonly decimal _threshold;
    public HighValueOrdersSpecification(decimal threshold) => _threshold = threshold;
    
    public override Expression<Func<Order, bool>> ToExpression() =>
        o => o.Total >= _threshold;
}

// Usage
var spec = new ActiveOrdersSpecification().And(new HighValueOrdersSpecification(1000));
var orders = await _repo.GetBySpecificationAsync(spec);
```

## 9.03 — CQRS

**What it is**
Command Query Responsibility Segregation — separating the write model (commands) from the read model (queries).

**Why**
- Write side: complex domain logic, validation, event publishing
- Read side: optimized for display, often needs denormalized data, no domain logic
- They have different scaling requirements and performance characteristics

**Simple CQRS with MediatR (same database)**
```csharp
// Command — changes state, returns minimal data
public record PlaceOrderCommand(int CustomerId, List<OrderItemDto> Items)
    : IRequest<PlaceOrderResult>;

public record PlaceOrderResult(int OrderId, string Status);

public class PlaceOrderHandler : IRequestHandler<PlaceOrderCommand, PlaceOrderResult>
{
    // Complex domain logic here
    public async Task<PlaceOrderResult> Handle(PlaceOrderCommand cmd, CancellationToken ct)
    {
        var order = Order.Create(cmd.CustomerId);
        // ... domain logic
        return new PlaceOrderResult(order.Id, order.Status.ToString());
    }
}

// Query — reads state, no side effects
public record GetOrderQuery(int OrderId) : IRequest<OrderDetailDto>;

public record OrderDetailDto(int Id, string Status, decimal Total, List<OrderItemDto> Items);

public class GetOrderHandler : IRequestHandler<GetOrderQuery, OrderDetailDto>
{
    private readonly AppDbContext _db;
    
    public async Task<OrderDetailDto> Handle(GetOrderQuery query, CancellationToken ct)
    {
        // Direct database read, no domain model, optimized for display
        return await _db.Orders
            .Where(o => o.Id == query.OrderId)
            .Select(o => new OrderDetailDto(
                o.Id,
                o.Status.ToString(),
                o.Items.Sum(i => i.Price * i.Quantity),
                o.Items.Select(i => new OrderItemDto(i.ProductId, i.Price, i.Quantity)).ToList()
            ))
            .FirstOrDefaultAsync(ct)
            ?? throw new NotFoundException($"Order {query.OrderId} not found");
    }
}
```

**Full CQRS with separate read/write databases**
```
Write side: SQL Server with domain model
    ↓ domain events
Event bus (RabbitMQ / Azure Service Bus)
    ↓ event handlers project to read model
Read side: Optimized read database (another SQL table, MongoDB, Elasticsearch)
```

## 9.04 — Event Sourcing

**What it is**
Storing state as a sequence of events rather than the current state. The current state is derived by replaying events.

**Traditional state storage**
```
Order table: { Id, CustomerId, Status: "Shipped", Total: 250 }
```

**Event sourcing storage**
```
Events table:
  1. OrderCreated    { OrderId: 1, CustomerId: 42, At: "2026-01-01" }
  2. ItemAdded       { OrderId: 1, ProductId: 5, Qty: 2, Price: 50 }
  3. ItemAdded       { OrderId: 1, ProductId: 8, Qty: 1, Price: 150 }
  4. OrderConfirmed  { OrderId: 1, At: "2026-01-02" }
  5. OrderShipped    { OrderId: 1, Tracking: "ABC123", At: "2026-01-03" }
```

**Advantages**
- Complete audit history — every state change recorded
- Temporal queries — "what was the state at any point in time?"
- Event replay — can rebuild any derived view from events
- Decoupled projections — multiple read models from same event stream

**Disadvantages**
- Complex to implement correctly
- Eventual consistency between write and read models
- Event schema evolution is hard
- Querying current state requires replaying or maintaining projections

**Snapshots**
For aggregates with thousands of events, replay is slow. Snapshots periodically save current state so replay only needs to cover events since the last snapshot.

**When to use event sourcing**
- Audit requirements (financial systems, healthcare, legal)
- Complex domain with temporal queries
- Multiple read models needed
- Not for simple CRUD applications

## 9.05 — Microservices

**What it is**
An architectural style where an application is built as a collection of small, independently deployable services, each responsible for a specific business capability.

**Service decomposition strategies**
- **By business capability**: Inventory, Ordering, Shipping, Billing — each a service
- **By subdomain (DDD)**: Bounded contexts map to services
- **By team structure (Conway's Law)**: The system architecture mirrors the team structure

**When microservices are appropriate**
- Large teams that need to deploy independently
- Different scaling requirements per service
- Different technology requirements per service
- The organizational structure is already distributed

**When NOT to use microservices**
- Small team (< 8 people) — complexity overhead not worth it
- Early-stage product — requirements change, boundaries shift
- Monolith-first: start with a well-structured monolith, extract services when the pain becomes real

**Service communication**
```
Synchronous (request-response):
  REST (HTTP/JSON) — simple, familiar, good tooling
  gRPC — binary, fast, strict contracts, streaming support

Asynchronous (event-driven):
  Message broker (RabbitMQ, Azure Service Bus, Kafka)
  Better for: cross-service workflows, loose coupling, resilience
```

**gRPC in .NET**
```protobuf
// product.proto
syntax = "proto3";
service ProductService {
  rpc GetProduct (GetProductRequest) returns (ProductResponse);
  rpc ListProducts (ListProductsRequest) returns (stream ProductResponse);
}
message GetProductRequest { int32 id = 1; }
message ProductResponse { int32 id = 1; string name = 2; double price = 3; }
```

```csharp
// Server
public class ProductGrpcService : ProductService.ProductServiceBase
{
    public override async Task<ProductResponse> GetProduct(
        GetProductRequest request, ServerCallContext context)
    {
        var product = await _repo.GetByIdAsync(request.Id);
        return new ProductResponse { Id = product.Id, Name = product.Name };
    }
}

// Client
var channel = GrpcChannel.ForAddress("https://product-service:443");
var client = new ProductService.ProductServiceClient(channel);
var response = await client.GetProductAsync(new GetProductRequest { Id = 1 });
```

**Circuit Breaker with Polly**
```csharp
builder.Services.AddHttpClient<IProductClient, ProductHttpClient>()
    .AddTransientHttpErrorPolicy(policy =>
        policy.WaitAndRetryAsync(3, attempt =>
```

**API Gateway**
Single entry point for clients. Handles:
- Routing requests to services
- Authentication/authorization offloading
- Rate limiting
- Request aggregation
- Protocol translation

In .NET: YARP (Yet Another Reverse Proxy) — Microsoft's built-in reverse proxy library.

## 9.06 — The Outbox Pattern

**What it is**
A pattern for reliably publishing domain events/messages without distributed transactions.

**The problem**
You need to save an order to the database AND publish an "OrderCreated" event. If you do these separately, either can fail:
- Save succeeds, publish fails → event lost
- Publish succeeds, save fails → inconsistent state

**The solution**
Write both the domain object and the outbox message in the same database transaction. A background process reads the outbox and publishes the messages.

```csharp
// Outbox table: Id, EventType, EventData, OccurredOn, ProcessedOn

// In the same transaction as the order:
await _db.Orders.AddAsync(order);
await _db.OutboxMessages.AddAsync(new OutboxMessage
{
    EventType = nameof(OrderCreatedEvent),
    EventData = JsonSerializer.Serialize(new OrderCreatedEvent(order.Id)),
    OccurredOn = DateTime.UtcNow
});
await _db.SaveChangesAsync();  // Atomic: order + message

// Background worker (Hangfire, IHostedService)
public class OutboxProcessor : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            var messages = await _db.OutboxMessages
                .Where(m => m.ProcessedOn == null)
                .OrderBy(m => m.OccurredOn)
                .Take(100)
                .ToListAsync(ct);
            
            foreach (var message in messages)
            {
                await _messageBus.PublishAsync(message.EventType, message.EventData, ct);
                message.ProcessedOn = DateTime.UtcNow;
            }
            
            await _db.SaveChangesAsync(ct);
            await Task.Delay(TimeSpan.FromSeconds(5), ct);
        }
    }
}
```

## 9.07 — The Saga Pattern

**What it is**
A pattern for managing distributed transactions across multiple services without 2PC (two-phase commit).

**The problem**
A business transaction spans multiple services: deduct inventory, charge payment, create shipment. If payment succeeds but shipment fails, inventory must be restored.

**Choreography-based saga**
Services listen to events and react independently. No central coordinator.
```
OrderService → publishes OrderCreated
InventoryService → listens → reserves inventory → publishes InventoryReserved
PaymentService → listens → charges payment → publishes PaymentProcessed
ShippingService → listens → creates shipment → publishes ShipmentCreated

On failure:
PaymentService → publishes PaymentFailed
InventoryService → listens → releases reservation → publishes InventoryReleased
OrderService → listens → marks order as failed
```

**Orchestration-based saga**
A central orchestrator tells each service what to do.
```csharp
public class CreateOrderSaga : ISaga
{
    public async Task HandleAsync(CreateOrderCommand cmd)
    {
        await _inventory.ReserveAsync(cmd.Items);
        try
        {
            await _payment.ChargeAsync(cmd.PaymentMethod, cmd.Total);
            try
            {
                await _shipping.CreateShipmentAsync(cmd.Address);
            }
            catch
            {
                await _payment.RefundAsync(transactionId);
                await _inventory.ReleaseAsync(reservationId);
                throw;
            }
        }
        catch
        {
            await _inventory.ReleaseAsync(reservationId);
            throw;
        }
    }
}
```
