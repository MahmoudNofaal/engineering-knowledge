## 11.01 — CAP Theorem

**What it is**
A theorem stating that a distributed system can provide at most two of the following three guarantees simultaneously:

**C — Consistency**
Every read receives the most recent write or an error. All nodes see the same data at the same time.

**A — Availability**
Every request receives a response (not an error). The system is always operational.

**P — Partition Tolerance**
The system continues operating when network partitions (node communication failures) occur.

**The real implication**
Network partitions happen in any distributed system. You cannot choose to not have partitions. So the real choice is: **when a partition occurs, do you sacrifice Consistency or Availability?**

- **CP systems** (prioritize Consistency): Return an error or wait until consistency is restored. MongoDB (with strong reads), HBase, ZooKeeper.
- **AP systems** (prioritize Availability): Return potentially stale data. Cassandra, DynamoDB (default), CouchDB.

**PACELC — the more nuanced model**
- **During Partition**: Choose Consistency (C) or Availability (A) — same as CAP
- **Else (no partition)**: Choose Latency (L) or Consistency (C) — there's still a tradeoff

Most databases are PA/EL or PC/EL.

## 11.02 — Consistency Models

**Strong Consistency**
After a write completes, all subsequent reads return the new value. Feels like a single machine. Expensive: requires coordination between all nodes.

**Eventual Consistency**
After a write, eventually all replicas converge to the same value. Reads may return stale data temporarily. Much cheaper — no coordination needed.

**Causal Consistency**
Operations causally related are seen in the same order by all nodes. "If A caused B, then everyone sees A before B." More useful than eventual, cheaper than strong.

**Read-Your-Writes**
After writing, you always read your own writes. Others may still see old data.

**Monotonic Reads**
Once you read a value, you never read an older value in subsequent reads.

**In SQL Server**
| Isolation Level | Dirty Read | Non-repeatable | Phantom |
|---|---|---|---|
| Read Uncommitted | ✓ possible | ✓ | ✓ |
| Read Committed | ✗ | ✓ possible | ✓ |
| Repeatable Read | ✗ | ✗ | ✓ possible |
| Serializable | ✗ | ✗ | ✗ |
| Snapshot (RCSI) | ✗ | ✗ | ✗ |

## 11.03 — Message Brokers

**RabbitMQ**
A traditional message broker with rich routing capabilities.

Concepts:
- **Exchange**: Receives messages from producers and routes to queues
  - `direct`: routes by exact routing key
  - `fanout`: routes to all bound queues (broadcast)
  - `topic`: routes by pattern matching on routing key
  - `headers`: routes by message headers
- **Queue**: Stores messages until consumed
- **Binding**: Link between exchange and queue with optional routing key
- **Consumer acknowledgment**: Consumer tells broker when message is processed
- **Dead Letter Exchange (DLX)**: Where failed/expired messages go

```csharp
// Publishing (with EasyNetQ)
await _bus.PubSub.PublishAsync(new OrderCreatedEvent(orderId));

// Subscribing
await _bus.PubSub.SubscribeAsync<OrderCreatedEvent>("subscription-id",
    async evt => await HandleOrderCreated(evt));
```

**Azure Service Bus**
Microsoft's managed message broker.

Key features:
- **Queues**: Point-to-point, FIFO with sessions
- **Topics/Subscriptions**: Pub-sub with filter rules per subscription
- **Sessions**: Ordered processing for correlated messages
- **Dead letter queue**: Built-in dead lettering with reason
- **Message deferral**: Defer processing while keeping message in queue
- **Scheduled messages**: Deliver at a specific time

```csharp
// Sender
var client = new ServiceBusClient(connectionString);
var sender = client.CreateSender("orders");
await sender.SendMessageAsync(new ServiceBusMessage(
    BinaryData.FromObjectAsJson(new OrderCreatedEvent(orderId))));

// Processor
var processor = client.CreateProcessor("orders", new ServiceBusProcessorOptions
{
    MaxConcurrentCalls = 10,
    AutoCompleteMessages = false
});

processor.ProcessMessageAsync += async args =>
{
    var evt = args.Message.Body.ToObjectFromJson<OrderCreatedEvent>();
    await HandleAsync(evt, args.CancellationToken);
    await args.CompleteMessageAsync(args.Message);
};

processor.ProcessErrorAsync += args =>
{
    _logger.LogError(args.Exception, "Message processing error");
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
```

**Apache Kafka**
A distributed log for high-throughput event streaming.

Concepts:
- **Topic**: A named stream of records (like a category)
- **Partition**: A topic is split into ordered partitions for parallelism
- **Offset**: Position within a partition. Consumers track their own offset.
- **Consumer group**: Multiple consumers sharing a topic's partitions
- **Retention**: Messages kept for a configurable period (not deleted on consumption)
- **Log compaction**: Keep only the latest value per key

```
Topic: orders (3 partitions)
  Partition 0: msg1, msg4, msg7 ...
  Partition 1: msg2, msg5, msg8 ...
  Partition 2: msg3, msg6, msg9 ...
  
Consumer Group A: 3 consumers — each reads one partition
Consumer Group B: 1 consumer — reads all partitions
```

Kafka vs. RabbitMQ/Service Bus:
| Aspect | Kafka | RabbitMQ/Service Bus |
|---|---|---|
| Message replay | Yes (up to retention) | No (consumed = gone) |
| Throughput | Millions/sec | Hundreds of thousands/sec |
| Consumer model | Pull | Push |
| Ordering | Per partition | Per queue/session |
| Use case | Event streaming, audit log | Task queues, RPC |

## 11.04 — Idempotency

**What it is**
An operation that produces the same result whether executed once or many times. Critical for safe retries.

**Why it matters**
Networks fail. Requests time out. Your service might not know if a request was received. If not idempotent, retry = duplicate.

**Making operations idempotent**

**Idempotency key**
Client generates a unique key per logical operation. Server checks if key was seen before.

```csharp
public async Task<OrderResult> CreateOrderAsync(
    CreateOrderRequest request,
    string idempotencyKey,
    CancellationToken ct)
{
    // Check if we've seen this key before
    if (await _idempotencyStore.ExistsAsync(idempotencyKey, ct))
        return await _idempotencyStore.GetResultAsync<OrderResult>(idempotencyKey, ct);
    
    var result = await ProcessCreateOrderAsync(request, ct);
    
    await _idempotencyStore.StoreAsync(idempotencyKey, result,
        expiry: TimeSpan.FromDays(7), ct);
    
    return result;
}
```

**Database-level idempotency**
```sql
INSERT INTO orders (idempotency_key, customer_id, ...)
VALUES (@key, @customerId, ...)
ON CONFLICT (idempotency_key) DO NOTHING
RETURNING *;
```

**At-most-once vs. at-least-once vs. exactly-once**
| Semantic | Description | How |
|---|---|---|
| At-most-once | Message may be lost, never duplicated | Fire and forget, no ack |
| At-least-once | Message delivered at least once, may be duplicated | Ack after processing, retry on failure |
| Exactly-once | Delivered exactly once | Idempotent consumer + transactional messaging (Kafka transactions) |

## 11.05 — Consistent Hashing

**What it is**
A technique for distributing data across nodes such that when nodes are added or removed, only a minimal fraction of keys need to be remapped.

**The problem with simple modulo hashing**
With 3 nodes: `server = hash(key) % 3`
When adding a 4th node: `server = hash(key) % 4`
Nearly every key is remapped — cache invalidated, massive load on databases.

**Consistent hashing**
- Arrange all possible hash values (0 to 2^32 - 1) in a circle (ring)
- Map each server to a point on the ring using its hash
- Map each key to a point on the ring using its hash
- Assign each key to the next server clockwise on the ring

Adding/removing a server: only keys between the new server and its predecessor need to be remapped. ~N/M keys where N = total keys, M = total servers.

**Virtual nodes**
One physical server maps to multiple points on the ring (virtual nodes). Creates more uniform distribution and better load balancing.

**Used in**
- Amazon DynamoDB
- Apache Cassandra
- Memcached (with ketama hashing)
- Redis Cluster
