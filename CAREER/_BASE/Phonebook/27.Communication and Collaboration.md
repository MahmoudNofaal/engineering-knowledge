## 27.01 — Coding Interview Protocol

**The communication sequence**
1. **Restate**: "So I need to find... given that... with the constraint that..."
2. **Clarify**: "Can the input be empty? Can values be negative? Is there a space constraint?"
3. **Design before coding**: "I'm thinking of using a sliding window approach because..."
4. **Implement with narration**: Talk through what you're doing as you write
5. **Trace an example**: Run through a test case manually
6. **Analyze complexity**: "This is O(n) time and O(1) space because..."
7. **Consider edge cases**: Empty input, single element, duplicates, max/min values

**When stuck**
```
Don't: Sit silently and stare at the problem

Do:
1. State what you understand and what you're confused about specifically
2. Try a brute force approach first — any working solution beats none
3. Identify what's inefficient — "This nested loop is O(n²), I need to bring it to O(n)"
4. Ask: "Would it help if I could look up values in O(1)?" (hint: use a hash map)
5. Try a small example by hand — trace through it manually
```

**C# interview idioms**
```csharp
// Know these cold:
var dict = new Dictionary<int, int>();
dict[key] = dict.GetValueOrDefault(key, 0) + 1;  // Frequency count

// Swap without temp variable (in .NET)

// Modulo for circular index
int next = (current + 1) % length;

// Max/min initialization
int max = int.MinValue;

// Two-sum pattern

{
    if (seen.Contains(target - n)) return true;
    seen.Add(n);
}
```

## 27.02 — System Design Interview

**The RADFY framework**
- **R**equirements: functional + non-functional
- **A**ssumptions and scale estimation
- **D**ata model and storage
- **F**low: the happy path step by step
- **Y**our tradeoffs and failure handling

**Scale estimation cheat sheet**
```
Daily Active Users → RPS:
  1M DAU × 10 actions/day / 86,400 sec ≈ 115 RPS

Storage estimation:
  1M users × 1 KB profile = 1 GB
  100 RPS × 1 KB × 86,400 sec × 365 days = ~3 TB/year

Bandwidth:
  100 RPS × 10 KB response = 1 MB/s = 8 Mbps

Memory (for 20% hot data in cache):
  10 GB data × 20% = 2 GB cache
```

## 27.03 — Behavioral Interview (STAR Method)

**STAR format**
- **S**ituation: The context. When and where.
- **T**ask: What you were responsible for.
- **A**ction: What YOU specifically did. Use "I", not "we".
- **R**esult: Quantified outcome. What changed.

**Stories to prepare from your background**

**Wrong technical decision**
"In the InterVision graduation project, I initially chose to store session transcripts as raw text in a VARCHAR column, assuming the text would always be short. In production, session transcripts grew to 20KB+, causing performance issues. I had to migrate to TEXT columns and add a dedicated search table. If I had done a capacity estimate upfront, I would have chosen the right type initially."

**Disagreement resolved**
"During InterVision development, I disagreed with my supervisor's suggestion to handle all AI processing synchronously within the HTTP request. I believed async processing via Hangfire was necessary for UX. I documented both approaches with estimated response times and tradeoffs, and we agreed to test both. The async approach was 3x more responsive for users."

**Learning something quickly**
"When a client needed Firebase push notifications integrated in 2 weeks and I had never used FCM, I read the official documentation, built a minimal prototype in 2 days, integrated it with the existing Hangfire pipeline in 3 more days, and shipped on time."
