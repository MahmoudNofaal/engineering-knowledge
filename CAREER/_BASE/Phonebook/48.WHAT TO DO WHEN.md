# CATEGORY 48 — WHAT TO DO WHEN

---

## 48.01 — Decision Trees

**When something is slow**
```
1. Measure first — use profiling, not intuition
   - dotnet-trace / dotnet-counters for .NET
   - Application Insights for end-to-end
   - SQL Server Profiler / query plan for DB

2. Is it the database?
   - Enable EF Core query logging
   - Check for N+1 queries
   - Check for missing indexes (sys.dm_db_missing_index_details)
   - Check query execution plan for table scans
   
3. Is it network I/O?
   - Are external API calls slow?
   - Add caching if data doesn't change frequently
   - Use connection pooling for external services
   - Consider async streaming for large responses

4. Is it CPU?
   - Is this truly CPU-bound or just blocking threads?
   - Profile with dotnet-trace --providers Microsoft-DotNETCore-SampleProfiler
   - Check for unnecessary serialization/deserialization
   - Check for excessive allocations causing GC pressure

5. Is it memory?
   - Check Gen2 GC frequency (GC.CollectionCount(2))
   - Check LOH allocations (large objects)
   - Use BenchmarkDotNet with MemoryDiagnoser
   - Use dotnet-gcdump for heap analysis
```

**When you have a bug**
```
1. Reproduce it reliably
   - Can you write a test that fails?
   - What are the exact conditions?

2. Isolate it
   - Binary search the call stack
   - Disable code paths until bug disappears
   - Check if it's data-dependent

3. Hypothesize
   - What MUST be true for this bug to occur?
   - What changed recently? (git log, deployment history)

4. Prove your hypothesis
   - Add logging/breakpoints that would ONLY trigger if your hypothesis is right
   - Do they trigger?

5. Fix and verify
   - Write a test that catches the bug
   - Fix the bug
   - Test passes
   - Deploy
```

**When choosing a database**
```
1. Do you need ACID transactions across multiple records?
   → SQL database (SQL Server, PostgreSQL)

2. Is your data highly relational (many joins)?
   → SQL database

3. Is your schema fixed and well-understood?
   → SQL database (normalization is a feature)

4. Do you need massive horizontal write scalability?
   → Cassandra, DynamoDB, MongoDB

5. Is your data document-oriented (nested, variable schema)?
   → MongoDB, Cosmos DB

6. Do you need full-text search?
   → Elasticsearch, or SQL FTS for simpler cases

7. Do you need time-series data?
   → InfluxDB, TimescaleDB, or Azure Data Explorer

8. Do you need key-value with sub-millisecond latency?
   → Redis

9. Do you need graph traversal?
   → Neo4j, Cosmos DB Gremlin API
```

**When you receive a failing system**
```
1. Don't touch anything until you understand what's happening
   - Check recent deployments
   - Check error rate trends (was it gradual or sudden?)
   - Check resource utilization (CPU, memory, disk, network)

2. Is it a complete outage or degraded service?
   - Complete: prioritize restoration first, understanding second
   - Degraded: investigate while system runs

3. Is it getting worse or stable?
   - Worse: act now (rollback, scale up, disable feature)
   - Stable: take time to understand before acting

4. What changed most recently?
   - Last deployment (roll it back)
   - Traffic spike (scale up)
   - Data change (check for corruption)
   - Config change (revert)

5. Communicate to stakeholders before diving deep
```

## 48.02 — Code Review Checklist

**Correctness**
- [ ] Does it handle null/empty inputs?
- [ ] Does it handle the boundary cases (first item, last item, empty collection)?
- [ ] Are concurrent modifications handled correctly?
- [ ] Are exceptions handled appropriately?
- [ ] Are resources (connections, streams, handles) disposed correctly?

**Security**
- [ ] Is user input validated before use?
- [ ] Are SQL queries parameterized?
- [ ] Are authorization checks in place?
- [ ] Are sensitive values not logged?
- [ ] Are new secrets stored safely (not hardcoded)?

**Performance**
- [ ] Are database queries using appropriate indexes?
- [ ] Is there potential for N+1 queries?
- [ ] Are expensive operations cached appropriately?
- [ ] Are large collections paginated?
- [ ] Are any synchronous operations blocking async contexts?

**Maintainability**
- [ ] Are names clear and intention-revealing?
- [ ] Is the code testable (dependencies injected, not created)?
- [ ] Is the complexity appropriate for the problem?
- [ ] Are tests included that verify the behavior?
- [ ] Is error handling consistent with the codebase?

**Observability**
- [ ] Are important operations logged (with appropriate level)?
- [ ] Are logs structured (no string concatenation)?
- [ ] Are correlation IDs passed through?
- [ ] Are metrics emitted for important operations?

## 48.03 — Pre-Deploy Checklist

```
Code quality:
  [ ] All tests pass (unit + integration)
  [ ] No new compiler warnings
  [ ] No TODO/FIXME left that should be resolved before deploy
  [ ] Code reviewed and approved

Configuration:
  [ ] Environment variables updated in prod config
  [ ] New secrets added to Key Vault / secrets manager
  [ ] Feature flags configured correctly
  [ ] No development-only config will run in production

Database:
  [ ] Migrations tested on a copy of production data
  [ ] Migration is backward compatible (old code can run with new schema)
  [ ] Indexes created (if any) concurrently/online, not locking tables
  [ ] Rollback plan for migration if needed

Deployment:
  [ ] Staging deployed and tested
  [ ] Health checks pass on staging
  [ ] Rollback procedure documented and ready
  [ ] On-call person knows about the deploy

Observability:
  [ ] Dashboards ready to watch during and after deploy
  [ ] Alert thresholds reasonable (not firing pre-deploy)
  [ ] Log level appropriate for production
```
