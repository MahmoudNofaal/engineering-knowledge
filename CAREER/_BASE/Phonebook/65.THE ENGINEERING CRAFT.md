## 65.01 — Code Readability Principles

**The newspaper test**
A good function is like a newspaper article. The headline (name) tells you what it does. The first paragraph (first few lines) gives you the key information. Details come later if you need them.

**The principle of least surprise**
Code should behave as the caller expects it to. No hidden side effects. A method named `GetOrder` should not delete anything.

**Command-Query Separation (Bertrand Meyer)**
A method either does something (command) or returns something (query) — not both.
```csharp
// WRONG: Does two things — queries and modifies state
int GetAndIncrementCounter()
{
    var current = _counter;
    _counter++;
    return current;
}

// RIGHT: Separate operations
int GetCounter() => _counter;
void IncrementCounter() => _counter++;

// Exception: ConcurrentStack.TryPop — pragmatic exception to the rule
```

**Names that communicate intent**
```csharp
// WRONG names:
void Process(List<int> data) { }
bool Check(Order o) { }
string Handle(string s) { }

// RIGHT names:
void ApplyDiscount(List<OrderItem> items) { }
bool IsEligibleForFreeShipping(Order order) { }
string NormalizePhoneNumber(string rawPhoneNumber) { }

// Common naming patterns:
// Is/Has/Can/Should → boolean returning methods
// Get → retrieves value
// Set/Update → modifies value
// Create/Build/Make → creates new object
// Find → might return null
// GetOrThrow → never returns null
// Calculate/Compute → performs calculation
// Send/Publish/Dispatch → sends to external system
// Validate → checks validity
// Parse/Format → converts string
```

**Function length and complexity**
```
Guideline (not rule): Functions < 20 lines are easier to understand.
When a function exceeds this, look for opportunities to extract.

Cyclomatic complexity: number of independent paths through code.
- 1-5: Simple. No problems.
- 6-10: Moderate. Consider refactoring.
- 11-20: Complex. Refactor.
- > 20: Very complex. Refactor urgently.

Visual Studio, SonarQube, and Roslyn can report this automatically.
```

## 65.02 — Naming Conventions in .NET

```csharp
// Classes, methods, properties, public fields: PascalCase
public class OrderService
{
    public string CustomerName { get; set; }
    public void ProcessOrder() { }
}

// Private fields: _camelCase (underscore prefix)
private readonly IOrderRepository _repository;
private string _currentStatus;

// Local variables, method parameters: camelCase
var orderId = 42;
void CreateOrder(int customerId, decimal total) { }

// Constants: PascalCase (Microsoft convention) or ALL_CAPS (some teams)
private const int MaxRetryCount = 3;
public const string DefaultCurrency = "USD";

// Interfaces: IPascalCase
public interface IOrderRepository { }
public interface IPaymentService { }

// Type parameters: T or TPurpose
public class Repository<TEntity> { }
public class Result<TValue, TError> { }

// Async methods: MethodNameAsync
public async Task<Order> GetOrderAsync(int id) { }

// Extension methods: in static class named TypeExtensions
public static class OrderExtensions
{
    public static OrderDto ToDto(this Order order) { }
}

// Test classes: ClassNameTests
public class OrderServiceTests { }

// Test methods: MethodName_Scenario_ExpectedOutcome
[Fact]
public void CreateOrder_WithNegativeTotal_ThrowsArgumentException() { }
```

## 65.03 — The Boy Scout Rule in Practice

**Leave it better than you found it — specific actions**

**While reading unfamiliar code:**
- Add a comment explaining WHY if the WHY is non-obvious
- Rename a confusing variable to something clearer
- Extract a magic number to a named constant

**While writing a bug fix:**
- Write a test that proves the bug exists and the fix works
- While in the method, fix any obvious style inconsistency

**While adding a feature:**
- Refactor any obviously duplicated code you touch
- Add missing null checks you spot
- But: don't refactor code you're NOT touching (risk vs. benefit)

**Limits of the rule:**
- Don't do a full refactor "while you're here" — scope creep
- Don't break working code in the name of cleanup without tests
- Large cleanups → their own PR with their own review
