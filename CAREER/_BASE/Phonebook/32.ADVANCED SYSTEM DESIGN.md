## 32.01 — Video Streaming System (YouTube Scale)

**Scale**
- 500 hours of video uploaded per minute
- 1 billion hours watched per day
- 2 billion monthly active users

**Upload flow**
```
1. Client requests upload URL from API
2. API generates a pre-signed URL (Azure Blob / S3)
3. Client uploads directly to blob storage (not through API servers)
4. Blob storage triggers an event on completion
5. Event triggers a transcoding job

Upload API → Blob Storage
                ↓ event
         Transcoding Queue
                ↓
    Transcoding Workers (FFmpeg)
                ↓
    Multiple resolutions: 1080p, 720p, 480p, 360p, 144p
    Multiple formats: MP4 (H.264), WebM (VP9)
                ↓
    Output Blob Storage → CDN
```

**Transcoding architecture**
```
Master job: Split video into 2-minute segments
  → Each segment: transcode in parallel across workers
  → Reassemble: stitch segments back together
  
Benefits: 30-minute video → 15 parallel 2-minute jobs → 15x faster
```

**Streaming (HLS / DASH)**
```
HLS (HTTP Live Streaming):
  master.m3u8 (playlist of quality variants)
    ├── 1080p/index.m3u8 (list of 6-second .ts segments)
    ├── 720p/index.m3u8
    └── 360p/index.m3u8

Client (video player):
  - Downloads master.m3u8
  - Estimates bandwidth
  - Selects quality level
  - Downloads segments sequentially
  - Adapts quality on bandwidth change (adaptive bitrate streaming)
```

**Storage**
```
Video storage: Azure Blob / S3 — terabytes to petabytes
  - Videos never deleted (legal, historical)
  - Lifecycle policies: move cold content to archive storage

Metadata: Relational DB (PostgreSQL / SQL Server)
  - Video ID, title, description, tags
  - View count, like count

Comments: MongoDB or Cassandra (high write volume)
Search: Elasticsearch (full text search on title, description, tags)
```

## 32.02 — Search Autocomplete System

**Functional requirements**
- As user types, show top 5 suggestions in < 100ms
- Suggestions based on query frequency (popular queries first)
- Handle 10M DAU, peak 2,000 queries/second

**Core data structure: Trie**
Each node = one character. Each terminal node has the count of completions.

**Scaled approach: Offline trie + serving layer**
```
Data collection:
  User queries → Kafka → Aggregation job → Frequency store
  
  Every hour: rebuild trie from top queries
  
Serving:
  Browser → CDN cache (top completions by 2-letter prefix, pre-computed)
          → If cache miss → Trie service → Redis (trie serialized in memory)
```

**Redis sorted set for each prefix**
```
For prefix "ca":
  ZADD autocomplete:ca 10234 "car insurance"
  ZADD autocomplete:ca 8921  "car rental"
  ZADD autocomplete:ca 7654  "candy"
  
ZREVRANGE autocomplete:ca 0 4  → top 5 suggestions
```

**CDN-level caching**
Pre-compute completions for all 2-character prefixes (26² = 676 possible prefixes in English).
Store in CDN. Only 3-character+ queries hit origin servers.

## 32.03 — Designing for High Availability

**Redundancy at every layer**
```
DNS: Multiple A records, health-checked by DNS provider
Load Balancer: Active-Active pair, health checks on backends
Application: Multiple instances across AZs (Availability Zones)
Database: Primary + 2 synchronous replicas across AZs
Cache: Redis Cluster with 3 masters + 3 replicas
Message Broker: Kafka with 3+ brokers, replication factor 3
```

**Availability zones**
```
Region: East US
  AZ 1: App Server A, DB Primary, Redis Master 1
  AZ 2: App Server B, DB Replica 1, Redis Master 2
  AZ 3: App Server C, DB Replica 2, Redis Master 3

AZ failure → the other two AZs absorb traffic
```

**Circuit Breaker States**
```
CLOSED (normal):
  Requests flow through normally
  Failure count tracked
  If failures > threshold → OPEN

OPEN (failing):
  Requests immediately rejected (fail-fast)
  After timeout → HALF-OPEN

HALF-OPEN (testing):
  Limited requests allowed through
  If succeed → CLOSED
  If fail → OPEN again
```

```csharp
// Polly circuit breaker
builder.Services.AddHttpClient<IPaymentClient, PaymentHttpClient>()
    .AddResilienceHandler("payment-pipeline", pipeline =>
    {
        // Retry
        pipeline.AddRetry(new HttpRetryStrategyOptions
        {
            MaxRetryAttempts = 3,
            Delay = TimeSpan.FromMilliseconds(500),
            BackoffType = DelayBackoffType.Exponential,
            UseJitter = true  // Avoids thundering herd on retry
        });
        
        // Circuit breaker
        pipeline.AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions
        {
            SamplingDuration = TimeSpan.FromSeconds(30),
            FailureRatio = 0.5,  // Open when 50% of requests fail
            MinimumThroughput = 10,  // Need at least 10 requests to evaluate
            BreakDuration = TimeSpan.FromSeconds(30)
        });
        
        // Timeout
        pipeline.AddTimeout(TimeSpan.FromSeconds(10));
    });
```

**Graceful degradation**
Design for partial failure. If a non-critical service is down, the core experience still works.

```csharp
// Recommendation service is non-critical
public async Task<ProductPageDto> GetProductPageAsync(int productId)
{
    var product = await _productService.GetAsync(productId);  // Required
    
    List<ProductSummary> recommendations = new();
    try
    {
        recommendations = await _recommendationService.GetSimilarAsync(productId);
    }
    catch (Exception ex)
    {
        // Log but don't fail the request — just show empty recommendations
        _logger.LogWarning(ex, "Recommendation service unavailable");
    }
    
    return new ProductPageDto(product, recommendations);
}
```

## 32.04 — API Versioning

**Why**
APIs are contracts. Breaking changes break clients. Versioning allows evolution without forcing all clients to upgrade simultaneously.

**Versioning strategies**

**URL versioning** (most common, explicit)
```
/api/v1/orders
/api/v2/orders
```

**Header versioning**
```
GET /api/orders
API-Version: 2.0
```

**Query string versioning**
```
GET /api/orders?api-version=2.0
```

**In ASP.NET Core**
```csharp
// NuGet: Asp.Versioning.Mvc / Asp.Versioning.Http

// Registration
builder.Services.AddApiVersioning(options =>
{
    options.DefaultApiVersion = new ApiVersion(1, 0);
    options.AssumeDefaultVersionWhenUnspecified = true;
    options.ReportApiVersions = true;  // X-Api-Supported-Versions header
    options.ApiVersionReader = ApiVersionReader.Combine(
        new UrlSegmentApiVersionReader(),
        new HeaderApiVersionReader("X-Api-Version"),
        new QueryStringApiVersionReader("api-version"));
});

// Controller versioning
[ApiVersion("1.0")]
[ApiVersion("2.0")]
[Route("api/v{version:apiVersion}/orders")]
public class OrdersController : ControllerBase
{
    [HttpGet]
    [MapToApiVersion("1.0")]
    public async Task<IActionResult> GetOrdersV1() { ... }
    
    [HttpGet]
    [MapToApiVersion("2.0")]
    public async Task<IActionResult> GetOrdersV2() { ... }  // New response shape
}

// Minimal API versioning
var v1 = app.NewApiVersionSet()
    .HasApiVersion(new ApiVersion(1, 0))
    .HasApiVersion(new ApiVersion(2, 0))
    .ReportApiVersions()
    .Build();

app.MapGet("/api/v{version:apiVersion}/orders", GetOrders)
    .WithApiVersionSet(v1)
    .MapToApiVersion(1.0);
```

**Versioning deprecation**
```csharp
[ApiVersion("1.0", Deprecated = true)]  // Returns Sunset header
[ApiVersion("2.0")]
```
