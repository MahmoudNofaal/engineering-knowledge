## 25.01 — Architecture Decision Records (ADRs)

**What it is**
A short document recording a significant architectural decision: the context, the options considered, the decision made, and the consequences.

**Template**
```markdown
# ADR-001: Use CQRS Pattern for Read/Write Separation

## Status
Accepted

## Context
The InterVision application has heavy read load for interview session display
while writes (creating sessions, recording answers) happen less frequently.
The same domain model is used for both, causing EF Core to track unnecessary
entities during reads and slowing down response times.

## Decision
Implement CQRS using MediatR:
- Commands go through the domain layer with full validation and events
- Queries bypass the domain and query the database directly using projections

## Options Considered
1. Single model: simpler, but EF Core tracking overhead on reads
2. CQRS with MediatR: moderate complexity, clear separation
3. Separate read database: high complexity, eventual consistency overhead

## Consequences
Positive:
- Read queries are 40-60% faster (no EF Core tracking)
- Domain logic is protected from read-model concerns
- Easier to optimize each side independently

Negative:
- More files per feature (command + query + handlers)
- Developers must remember to update read models when write model changes
- Slight learning curve for new team members

## Date
2026-06-01
```

## 25.02 — Debugging Discipline

**The scientific method for bugs**
1. **Observe**: What is the actual behavior? What is the expected behavior?
2. **Hypothesize**: What could cause this specific discrepancy?
3. **Predict**: If my hypothesis is correct, what should happen when I do X?
4. **Test**: Do X. Does what happened match the prediction?
5. **Conclude**: Was the hypothesis correct? If not, form a new one.

**Practical debugging techniques**
```csharp
// Binary search debugging: if you can't isolate, divide the problem space
// If the bug is between steps A and Z, check step M
// If M works, check step S. If M fails, check step G. etc.

// Git bisect: find which commit introduced a regression
// git bisect start
// git bisect bad           (current commit is bad)
// git bisect good v1.0.0   (this version was good)

// Data breakpoints: break when a variable's value changes
// Useful for tracking mutation bugs

// Differential debugging: what changed between "working" and "broken"?
// Environment variables, dependency versions, data, configuration
```

**Reading stack traces**
```
System.NullReferenceException: Object reference not set to an instance of an object.
   at MyApp.Services.OrderService.ProcessAsync(Int32 orderId)  ← Start here
     in D:\projects\myapp\src\MyApp.Services\OrderService.cs:line 87

```
- Read from top (the actual error site) not bottom
- The innermost frame is where the exception actually occurred
- The context frames show how you got there

## 25.03 — Tradeoff Thinking

**The tradeoffs you must know**

**Consistency vs. Availability**: In a distributed system under partition, choose one.

**Performance vs. Simplicity**: Optimize after profiling, not before. Premature optimization adds complexity without measured benefit.

**Flexibility vs. Simplicity**: Every abstraction layer adds flexibility but costs readability and debugging complexity.

**Build vs. Buy**: Building gives control; buying gives speed. Total cost of ownership includes maintenance.

**Now vs. Later**: Ship fast to learn; build carefully to avoid rework. The right balance depends on how much you know.

**Normalization vs. Denormalization**: Normalized = consistent, slower reads. Denormalized = faster reads, consistency risks.

**Monolith vs. Microservices**: Monolith = simpler, faster to develop. Microservices = independent scaling, deployment, complexity.

**Strong vs. Eventual Consistency**: Strong = correct, slow. Eventual = fast, complex, sometimes stale.
