# CATEGORY 46 — THE MENTAL MODELS

---

## 46.01 — First Principles Thinking

**What it is**
Breaking down a problem to its fundamental truths and reasoning up from there, rather than reasoning by analogy ("we do it this way because we've always done it this way").

**Applied to system design**
```
Problem: Our API is slow at peak load.

By analogy (weak): "Other companies use microservices when they scale."
→ We should use microservices.
→ This might be completely wrong for our situation.

First principles (strong):
- WHY is the API slow? Profiling shows: database queries.
- WHY are database queries slow? Missing indexes, N+1 queries.
- WHAT is the fundamental bottleneck? Disk I/O.
- WHAT can reduce disk I/O? Caching, better indexes, denormalization.
- WHAT is the cheapest solution? Add indexes and fix N+1.
- Microservices would not have helped at all.
```

**The 5 Whys**
```
Problem: Checkout is failing.
Why 1: Because the payment service is returning 500.
Why 2: Because the payment service can't connect to the database.
Why 3: Because the database connection pool is exhausted.
Why 4: Because connections are being held open during slow external API calls.
Why 5: Because the code was written that way and nobody measured connection hold time.

Root cause: Code holds DB connections during slow external calls.
Fix: Release connection before external call; re-acquire after.
```

## 46.02 — The Abstraction Ladder

**What it is**
A mental model for choosing the right level of abstraction for a given problem.

```
High abstraction:  "The user wants to track their work"
                   ↑
Medium:            "The user wants a TODO list"
                   ↑
Low abstraction:   "The user wants to CRUD tasks with due dates"
                   ↑
Very low:          "INSERT INTO tasks (title, due_date) VALUES (?, ?)"
```

**Pitfalls at each level**

Too high: Requirements so vague that any solution is valid, but no direction.
Too low: Optimizing bytes when the user just needs the feature to exist.

**The right level** depends on: your current knowledge, the stage of the project, and what question you're answering.

Design discussions → higher abstraction
Implementation discussions → lower abstraction
Debugging → lowest abstraction (what exactly is the CPU doing?)

## 46.03 — The Iceberg Model

**What it is**
A systems thinking model. Only 10% of a system's behavior is visible (events). Beneath that is: patterns, structures, and mental models.

```
Visible:       Events (bugs, slowness, outages)
               ↓
Patterns:      Recurring problems, trends over time
               ↓
Structures:    Architecture, incentives, processes that cause the patterns
               ↓
Mental models: Assumptions, beliefs, cultural norms that created the structures
```

**Applied to tech debt**
```
Event:        "There's a production bug every Friday deploy"
Pattern:      "We've had deploy issues 8 of the last 12 Fridays"
Structure:    "No staging environment, direct to prod, no rollback procedure"
Mental model: "Moving fast is more important than stability"
```

Fixing only events (the bug) is whack-a-mole.
Fixing the structure (add staging, rollback) is durable.
Fixing the mental model (change the culture) is transformational.

## 46.04 — Inversion

**What it is**
Instead of asking "how do I succeed?", ask "what causes failure? How do I avoid that?"

**Applied to software**
```
Instead of: "How do I build reliable software?"
Ask:        "What causes software to be unreliable?"

Causes of unreliability:
- Untested code paths
- External dependencies failing
- State corruption
- Race conditions
- Insufficient logging to diagnose problems
- Lack of rollback ability

Fix all of those → reliable software emerges.
```

**Applied to career**
```
Instead of: "How do I become a great engineer?"
Ask:        "What makes engineers mediocre or stuck?"

Causes of stagnation:
- Not writing code regularly
- Working only on familiar problems
- Not reading production code of good systems
- Not getting feedback on code
- Not teaching what you know

Avoid those → growth emerges.
```
