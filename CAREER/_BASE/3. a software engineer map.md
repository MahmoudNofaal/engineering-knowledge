## Section 01 — Computer Science Fundamentals

_The floor beneath everything. Before language, before frameworks, before patterns._

### How Computers Execute Code

- CPU architecture: cores, clock speed, instruction cycle (fetch, decode, execute)
- Registers and the CPU cache hierarchy (L1, L2, L3)
- Cache hits and cache misses — why data locality matters in code
- Branch prediction — what it is and why mispredictions cost cycles
- SIMD — processing multiple data elements in a single instruction

### Memory

- The memory hierarchy: registers → cache → RAM → disk
- Stack vs. heap — allocation, lifetime, and ownership
- Virtual memory — what it is, why it exists, and how paging works
- Memory-mapped files
- How a program's memory layout looks: code, data, stack, heap segments

### Operating System Concepts

- Processes vs. threads — what each is, how they differ, what they share
- Context switching — what it costs and why it matters
- Scheduling algorithms (FIFO, round-robin, priority-based)
- Synchronization primitives: mutex, semaphore, spinlock, monitor
- Deadlock — conditions that cause it and how to prevent it
- File system concepts: inodes, blocks, directories, file descriptors
- System calls — how user space talks to kernel space
- I/O models: blocking, non-blocking, multiplexed (select/poll/epoll), async

### Networking

- The OSI model — each layer and its responsibility
- TCP vs. UDP — guarantees, use cases, tradeoffs
- The TCP handshake (3-way) and teardown (4-way)
- IP addressing, subnets, CIDR notation
- DNS resolution — the full journey of a domain name to an IP address
- HTTP/1.1 vs. HTTP/2 vs. HTTP/3 — what changed and why
- HTTPS and TLS — the handshake, certificates, symmetric vs. asymmetric encryption in context
- WebSockets — full-duplex communication over a persistent connection
- Long polling vs. server-sent events vs. WebSockets — when each is appropriate
- CDN — how content delivery networks work and what they cache
- Load balancer — Layer 4 vs. Layer 7, round-robin vs. least connections vs. IP hash

### How Databases Store Data

- B-tree and B+ tree structures — why databases use them for indexes
- Pages and buffer pools — how data is read from disk into memory
- Write-ahead logging (WAL) — how databases guarantee durability
- MVCC (Multi-Version Concurrency Control) — how databases allow concurrent reads and writes

### Encoding and Data Representation

- Binary and hexadecimal — reading memory addresses and bit flags
- ASCII, UTF-8, UTF-16 — character encoding and why it matters for strings
- Endianness — big-endian vs. little-endian and where it appears in practice
- Serialization formats: JSON, XML, Protocol Buffers, MessagePack — tradeoffs

## Section 04 — Design Principles and Patterns

_After you can write code, learn to design it._

### SOLID Principles

- Single Responsibility Principle — one reason to change, not one method
- Open/Closed Principle — open for extension, closed for modification; how interfaces enable it
- Liskov Substitution Principle — behavioral subtyping, not just type hierarchy
- Interface Segregation Principle — client-specific interfaces vs. fat interfaces
- Dependency Inversion Principle — depending on abstractions, not implementations; how DI containers implement this

### General Principles

- DRY (Don't Repeat Yourself) — knowledge duplication vs. code duplication (they are different)
- KISS (Keep It Simple) — the cost of complexity, when abstraction is premature
- YAGNI (You Aren't Gonna Need It) — the discipline of not building what is not required
- Composition over inheritance — why inheritance hierarchies become liabilities
- Principle of least surprise — code should behave the way a reader expects
- Fail fast — surface errors as early as possible, as loudly as possible

### Clean Code

- Naming: meaningful, pronounceable, searchable, intention-revealing names
- Functions: do one thing, one level of abstraction per function, argument count discipline
- Comments: explain why, not what; dead code, noise comments, mandated comments are all liabilities
- Formatting: vertical density, horizontal alignment, team agreement over personal preference
- Error handling: exceptions vs. return values, the `Result<T>` pattern, not using exceptions for control flow
- Boundaries: third-party code, adapter pattern, do not scatter vendor APIs throughout your codebase

### Design Patterns

**Creational**

- Singleton — thread-safe implementation in .NET, when it is appropriate, when it is not
- Factory Method — deferring instantiation to subclasses
- Abstract Factory — families of related objects
- Builder — constructing complex objects step by step, fluent builder in C#
- Prototype — cloning objects, `ICloneable` and its limitations

**Structural**

- Adapter — making incompatible interfaces work together
- Decorator — adding behavior without modifying the original class; how ASP.NET Core middleware is a decorator chain
- Facade — simplifying a complex subsystem behind a clean interface
- Proxy — controlling access to an object; lazy loading, caching, access control
- Composite — treating individual objects and compositions uniformly (tree structures)
- Flyweight — sharing common state across many fine-grained objects

**Behavioral**

- Strategy — interchangeable algorithms behind a common interface
- Observer — publish-subscribe, event systems, `IObservable<T>` / `IObserver<T>`
- Command — encapsulating a request as an object; undo/redo systems
- Chain of Responsibility — passing a request through a chain of handlers; ASP.NET Core middleware
- Template Method — defining the skeleton of an algorithm, letting subclasses fill in the steps
- Iterator — sequential access without exposing the underlying structure; `IEnumerable<T>`
- Mediator — centralizing complex communication; MediatR in .NET
- State — changing behavior when internal state changes
- Visitor — adding operations to a class hierarchy without modifying it

### Refactoring

- Code smell catalog: long method, large class, feature envy, data clumps, primitive obsession, switch statements, parallel inheritance hierarchies, lazy class, speculative generality, temporary field, message chains, middle man, inappropriate intimacy, alternative classes with different interfaces
- Refactoring techniques: extract method, extract class, inline method, rename, move method, introduce parameter object, replace conditional with polymorphism, replace magic number with symbolic constant, decompose conditional

## Section 05 — System Design and Architecture

_How to design systems that serve many users, survive failures, and can be changed._

### Architectural Patterns

**Clean Architecture**

- The dependency rule — dependencies point inward, never outward
- Layers: domain, application, infrastructure, presentation
- Domain entities and their invariants
- Application use cases (interactors)
- Interface adapters — controllers, presenters, gateways
- Infrastructure — databases, external APIs, file systems

**Domain-Driven Design (DDD)**

- Ubiquitous language — shared vocabulary between engineers and domain experts
- Bounded contexts — explicit boundaries where a model applies
- Context mapping — relationships between bounded contexts
- Entities — identity-defined objects, lifecycle, invariant enforcement
- Value objects — equality by value, immutability
- Aggregates — consistency boundaries, the aggregate root rule
- Domain services — stateless operations that do not belong to an entity
- Domain events — something that happened in the domain, communicated across boundaries
- Repositories — abstracting the persistence layer from the domain
- Specifications — encapsulating query logic as composable objects

**CQRS (Command Query Responsibility Segregation)**

- Commands vs. queries — the segregation principle
- Separate read and write models
- MediatR in .NET — `IRequest<T>`, `IRequestHandler<TRequest,TResponse>`, `INotification`
- Pipeline behaviors — cross-cutting concerns (validation, logging, caching) as behaviors
- When CQRS adds value and when it adds complexity without benefit

**Event Sourcing**

- Storing events, not state — the log as the source of truth
- Event replay — rebuilding state from the event log
- Projections — building read models from events
- Snapshots — optimizing event replay for large event streams
- When event sourcing is appropriate — audit requirements, temporal queries, complex domain logic

### Scalability Patterns

- Horizontal vs. vertical scaling — what each means and their limits
- Stateless services — why statelessness enables horizontal scaling
- Sticky sessions — what they are and why they undermine scalability
- Database read replicas — distributing read load, replication lag implications
- Database sharding — horizontal partitioning, partition key selection, cross-shard queries
- Consistent hashing — distributing data across nodes without full redistribution on node changes
- The Outbox pattern — reliable message publishing within a database transaction
- The Inbox pattern — idempotent message consumption
- The Saga pattern — managing distributed transactions across services
- Event-driven architecture — loose coupling through events, eventual consistency implications

### Distributed Systems Concepts

- CAP theorem — consistency, availability, partition tolerance — pick two (and the nuances of that claim)
- PACELC — an extension of CAP that accounts for latency trade-offs even without partitions
- Consistency models: strong consistency, eventual consistency, causal consistency, read-your-writes, monotonic reads
- Consensus algorithms — what they solve: Raft (conceptually), Paxos (conceptually)
- Leader election — how distributed systems agree on a primary
- Distributed transactions — two-phase commit (2PC), its failure modes, and why sagas are often preferred
- Idempotency — designing operations that can be safely retried
- At-most-once vs. at-least-once vs. exactly-once delivery — what each means and what each costs

### Caching

- Cache-aside (lazy loading) — load from cache, populate on miss
- Write-through — write to cache and database simultaneously
- Write-behind (write-back) — write to cache, flush to database asynchronously
- Read-through — cache sits in front of the database transparently
- TTL (Time To Live) — expiry strategy, TTL jitter to prevent cache stampede
- Cache stampede (thundering herd) — prevention with probabilistic early expiration or locking
- Cache eviction policies: LRU, LFU, FIFO, MRU
- Cache invalidation strategies — the hardest problem in computer science and why

### Message Brokers and Async Communication

- Point-to-point vs. publish-subscribe
- RabbitMQ: exchanges (direct, fanout, topic, headers), queues, bindings, consumer acknowledgments, dead letter exchanges
- Azure Service Bus: queues vs. topics, message sessions, dead letter queue, message deferral, scheduled messages
- Apache Kafka: partitions, consumer groups, offsets, log compaction, exactly-once semantics
- Outbox pattern with a message broker — ensuring reliability without distributed transactions
- Backpressure — what happens when consumers are slower than producers, and how to handle it

### Microservices

- Monolith vs. microservices — not a default choice, a deliberate trade-off
- Service decomposition strategies — by business capability, by subdomain, by team structure (Conway's Law)
- Service communication: synchronous (HTTP, gRPC) vs. asynchronous (message broker)
- gRPC — Protocol Buffers, service definition, streaming, advantages over REST for internal services
- API gateway — single entry point, routing, authentication offloading, rate limiting, request aggregation
- Service discovery — client-side vs. server-side, DNS-based, Consul, Kubernetes Services
- Circuit breaker — Polly in .NET: `AddPolicyHandler`, retry, circuit breaker, timeout, bulkhead isolation, hedging
- Sidecar pattern — co-located helper processes (logging, service mesh proxy, config agent)
- Service mesh — Istio, Linkerd — mutual TLS, observability, traffic management at the infrastructure level

### API Design

- REST constraints — uniform interface, statelessness, client-server, cacheability, layered system
- REST API design: resource naming (nouns not verbs), HTTP method semantics, status codes, sub-resources
- API versioning: URL path (`/v1/`), query string, header — tradeoffs of each
- Idempotency keys — client-generated keys to make non-idempotent operations safe to retry
- Pagination: offset-based, cursor-based, keyset pagination — performance implications at scale
- Error response design — RFC 9457 Problem Details, machine-readable error codes, actionable messages
- OpenAPI / Swagger — documentation as code, Swashbuckle in .NET, NSwag
- GraphQL — queries, mutations, subscriptions, N+1 problem (DataLoader), when to choose GraphQL over REST
- gRPC vs. REST — binary vs. text, strict contracts vs. flexible JSON, streaming support

## Section 06 — Databases

_Where most application performance lives._

### Relational Database Fundamentals

- The relational model — relations, tuples, attributes, keys
- Normalization: 1NF, 2NF, 3NF, BCNF — what each eliminates, when to stop normalizing
- Denormalization — when read performance justifies consistency risk
- Primary keys: natural vs. surrogate, UUIDs vs. sequential IDs (and the index fragmentation tradeoff)
- Foreign keys — referential integrity, cascade behaviors (delete, update, restrict)
- ACID: Atomicity, Consistency, Isolation, Durability — what each guarantees
- Transaction isolation levels: Read Uncommitted, Read Committed, Repeatable Read, Serializable — what anomaly each prevents
    - Dirty read, non-repeatable read, phantom read — definitions and which isolation levels prevent each
- Snapshot isolation and Read Committed Snapshot Isolation (RCSI) in SQL Server — optimistic concurrency

### Indexing

- Clustered index — the physical ordering of the table, only one per table
- Non-clustered index — a separate structure with a pointer back to the row
- Covering index — all columns needed for a query are in the index
- Included columns — extending a non-clustered index without adding to the key
- Composite index — multi-column index, column order matters for query matching
- Filtered index — index over a subset of rows
- Index on computed column
- Fill factor — pre-allocating space for future inserts to reduce page splits
- Index fragmentation — internal vs. external fragmentation, rebuild vs. reorganize
- Index statistics — how the query optimizer uses them, why stale statistics cause bad plans
- When indexes hurt — write amplification, storage cost

### SQL Mastery

- Joins: INNER, LEFT OUTER, RIGHT OUTER, FULL OUTER, CROSS, SELF — when each is appropriate
- Subqueries vs. JOINs — correlated subqueries, when one is more readable and when one is faster
- CTEs (Common Table Expressions) — readability, recursive CTEs for hierarchical data
- Window functions: `ROW_NUMBER()`, `RANK()`, `DENSE_RANK()`, `NTILE()`, `LEAD()`, `LAG()`, `FIRST_VALUE()`, `LAST_VALUE()`, `SUM() OVER(...)`, `AVG() OVER(...)`, `COUNT() OVER(...)`, partition and order clauses
- `MERGE` statement — upsert operations, source and target matching
- `EXCEPT` and `INTERSECT` — set operations
- Pivot and unpivot — reshaping tabular data
- String functions: `STRING_AGG`, `STRING_SPLIT`, `STUFF`, `PATINDEX`, `CHARINDEX`
- Date and time: `DATEADD`, `DATEDIFF`, `DATEPART`, `FORMAT`, `AT TIME ZONE`, `DATETIMEOFFSET`
- JSON support in SQL Server: `FOR JSON`, `OPENJSON`, `JSON_VALUE`, `JSON_QUERY`
- Full-text search: full-text indexes, `CONTAINS`, `FREETEXT`, `CONTAINSTABLE`
- Temporal tables — system-versioned tables for point-in-time queries

### Query Optimization

- Reading execution plans in SSMS — seek vs. scan, nested loops vs. hash match vs. merge join
- Index seeks vs. index scans — what causes a scan and how to convert it to a seek
- Key lookups (RID lookups) — what they are and how to eliminate them with covering indexes
- `SET STATISTICS IO ON` and `SET STATISTICS TIME ON` — measuring query cost honestly
- Query hints — `NOLOCK`, `FORCESEEK`, `USE INDEX`, `OPTION (RECOMPILE)` — when to use and the risks

- Table variables vs. temp tables — statistics, scope, and performance differences
- `sp_WhoIsActive` and `sys.dm_exec_requests` — identifying long-running queries in production

### Entity Framework Core

- DbContext lifecycle — scoped lifetime in ASP.NET Core, Unit of Work pattern
- Change tracking — added, modified, deleted, unchanged, detached — and what `SaveChanges()` does with each
- The N+1 problem — recognizing it in EF Core logs, resolving with `Include()`, `ThenInclude()`, explicit loading
- `AsNoTracking()` — when to use it and the performance difference
- Split queries (`AsSplitQuery()`) — avoiding cartesian explosion on multiple includes

- Global query filters — soft delete, multi-tenancy
- Owned entities — value objects in EF Core, `OwnsOne`, `OwnsMany`

- Bulk operations — EF Core 7+ `ExecuteUpdateAsync`, `ExecuteDeleteAsync` — when to bypass the ORM

### Redis

- Data types: strings, hashes, lists, sets, sorted sets, bitmaps, hyperloglogs, streams, geospatial
- Use cases: caching, session storage, rate limiting, leaderboards, pub/sub, distributed locks

## Section 08 — Testing

_Code without tests is incomplete._

### Testing Principles

- The test pyramid — unit tests (many, fast), integration tests (fewer, slower), end-to-end tests (fewest, slowest)
- The testing trophy — a more nuanced view: static analysis → unit → integration → E2E
- F.I.R.S.T. principles: Fast, Independent, Repeatable, Self-validating, Timely
- Arrange-Act-Assert (AAA) — the standard structure for a test
- One assertion per test — and when it is reasonable to have more
- Test doubles taxonomy: dummy (unused), stub (returns canned data), spy (records calls), mock (verifies expectations), fake (working implementation)
- Test naming conventions: `MethodUnderTest_Scenario_ExpectedBehavior`, BDD-style `should_do_X_when_Y`

### Unit Testing

- xUnit — `[Fact]`, `[Theory]`, `[InlineData]`, `[MemberData]`, `[ClassData]`, `IClassFixture<T>`, `ICollectionFixture<T>`
- NUnit — `[Test]`, `[TestCase]`, `[SetUp]`, `[TearDown]`, `[OneTimeSetUp]`
- FluentAssertions — readable assertions, `Should().Be()`, `Should().Contain()`, `Should().Throw<T>()`
- Shouldly — alternative assertion library
- NSubstitute — `Substitute.For<T>()`, `Returns()`, `Received()`, `Arg.Any<T>()`, `Arg.Is<T>()`
- Moq — `Mock<T>`, `Setup()`, `Returns()`, `Verify()`, `It.IsAny<T>()`, `It.Is<T>()`
- AutoFixture — generating test data automatically, customizations, auto-mocking
- Bogus — generating realistic fake data (names, emails, addresses, IDs)

### Integration Testing

- `WebApplicationFactory<TProgram>` — hosting the real application in memory for tests
- Replacing registrations for tests — `WithWebHostBuilder`, `ConfigureTestServices`
- `HttpClient` from `WebApplicationFactory` — making real HTTP requests against the test server
- TestContainers for .NET — spinning up SQL Server, Redis, RabbitMQ in Docker containers for tests
- Database setup and teardown — respawn library for resetting databases between tests
- `IAsyncLifetime` — async setup and teardown in xUnit

### Testing Patterns

- Object Mother — factory methods that create domain objects for tests
- Test Data Builder — fluent builders for complex domain objects
- Shared test context — `IClassFixture` for expensive setup shared across tests in a class
- Collection fixtures — shared context across multiple test classes
- Parameterized tests — data-driven testing with `[Theory]` and `[InlineData]` / `[MemberData]`

### Specialized Testing

- BDD with SpecFlow — feature files, step definitions, scenario outlines, living documentation
- Snapshot testing — capturing and comparing rendered output (useful for complex JSON responses)
- Load testing with NBomber — scenarios, steps, reporting, threshold assertions
- Load testing with k6 — JavaScript-based load tests, virtual users, ramping stages
- Mutation testing with Stryker.NET — verifying test suite effectiveness by introducing mutations
- Contract testing with Pact — consumer-driven contracts for microservice API compatibility
- Property-based testing with FsCheck — generating inputs to find edge cases automatically

### Test-Driven Development (TDD)

- The red-green-refactor cycle
- Starting with the simplest failing test
- Incremental design through tests
- The effect of TDD on API design — tests reveal awkward interfaces before they are shipped
- When TDD is most valuable: complex domain logic, algorithmic code, behavior-heavy use cases
- When TDD is less natural: integration tests, infrastructure code, exploratory code

## Section 09 — AI Engineering

_The layer being added to every serious system._

### LLM Fundamentals

- Transformer architecture — attention mechanism, self-attention, what "tokens" are
- Context window — what it means, how it limits conversation and document processing
- Temperature — controlling output randomness; when to set it low (structured tasks) and high (creative tasks)
- Top-p (nucleus sampling) and top-k — alternative sampling strategies
- System prompts — setting persistent instructions, personas, constraints
- Few-shot prompting — providing examples within the prompt to guide output format and style
- Chain-of-thought prompting — instructing the model to reason before answering
- Structured output — JSON mode, function calling as a structured output mechanism
- Token limits and cost — tokens in, tokens out, pricing models, why efficiency matters
- Model selection by task: small fast cheap (Haiku, GPT-4o-mini) vs. large smart expensive (Opus, GPT-4o) — routing by complexity

### Prompt Engineering

- Prompt structure: role, context, task, constraints, output format
- Delimiter patterns — separating instructions from content with XML tags, triple backticks
- Negative constraints — telling the model what not to do
- Self-consistency prompting — generating multiple responses and selecting the majority answer
- Prompt injection attacks — what they are, how to mitigate them in production
- Prompt versioning — treating prompts as code, storing them in version control
- Prompt testing — writing test cases against prompt outputs

### RAG (Retrieval-Augmented Generation)

- Why RAG — solving the knowledge cutoff problem and hallucination on private data
- Document ingestion pipeline: load → clean → chunk → embed → store
- Chunking strategies: fixed-size, sentence-based, paragraph-based, recursive character splitting, semantic chunking
- Overlap — maintaining context across chunk boundaries
- Embedding models — `text-embedding-3-small`, `text-embedding-3-large`, open-source alternatives (BAAI/bge, E5)
- Cosine similarity vs. dot product vs. Euclidean distance — what each measures
- Vector databases: Qdrant, Weaviate, Pinecone, pgvector — capability and performance comparison
- HNSW (Hierarchical Navigable Small World) — the indexing algorithm most vector DBs use
- Hybrid search — combining vector similarity with BM25 keyword search (reciprocal rank fusion for merging)
- Reranking — using a cross-encoder to reorder retrieved results by relevance
- Metadata filtering — narrowing retrieval by document attributes before or after vector search
- Evaluation of RAG: faithfulness (does the answer match the retrieved context?), answer relevancy, context relevancy

### Agents

- The ReAct pattern (Reasoning + Acting) — think, act, observe, repeat
- Tool use / function calling — defining tools, the model deciding which to call, parsing results
- Planning — breaking complex tasks into subtasks before executing
- Memory patterns: short-term (conversation history), long-term (vector store), episodic (event log)
- Multi-agent systems — orchestrator agents, specialist agents, handoff patterns
- Agent failure modes — infinite loops, hallucinated tool calls, unrecoverable error states
- Human-in-the-loop — approval steps, escalation, graceful degradation

### .NET AI Frameworks

- Semantic Kernel — kernel, plugins (native functions, prompt functions), planners, memory, connectors
- `KernelPlugin` and `[KernelFunction]` — exposing .NET methods as tools
- `AzureOpenAIChatCompletionService` and `OpenAIChatCompletionService` — pluggable LLM backends
- Semantic Kernel memory — `ISemanticTextMemory`, `VolatileMemoryStore`, vector store connectors
- `Microsoft.Extensions.AI` — the unified abstraction for AI services in .NET (IChatClient, IEmbeddingGenerator)
- Kernel memory (formerly Semantic Memory) — full RAG pipeline as a library

### Production AI Engineering

- Latency management: streaming responses (Server-Sent Events in ASP.NET Core), timeouts, async pipelines
- Caching LLM responses: semantic caching (cache by embedding similarity), exact match caching
- Fallback chains — primary model fails → fallback to cheaper model → fallback to cached response
- Retry with exponential backoff — rate limits from LLM APIs are common and expected
- Cost tracking — logging token usage per request, per user, per feature; budget alerts
- Evaluation pipelines — automated quality scoring, LLM-as-judge, golden dataset regression testing
- Output validation — JSON schema validation, structured output retry on failure
- Responsible AI — content filtering, bias detection, output moderation with Azure AI Content Safety

## Security

### OWASP Top 10

- A01 Broken Access Control — IDOR, privilege escalation, missing authorization checks, CORS misconfiguration
- A02 Cryptographic Failures — sensitive data in plaintext, weak algorithms, improper key management
- A03 Injection — SQL injection, NoSQL injection, command injection, LDAP injection — detection and prevention
- A04 Insecure Design — threat modeling, secure design patterns, abuse cases
- A05 Security Misconfiguration — default credentials, verbose error messages, open cloud storage, missing headers
- A06 Vulnerable and Outdated Components — dependency scanning, `dotnet list package --vulnerable`, Dependabot
- A07 Identification and Authentication Failures — weak passwords, missing MFA, session fixation, credential stuffing
- A08 Software and Data Integrity Failures — deserialization attacks, CI/CD pipeline integrity, update verification
- A09 Security Logging and Monitoring Failures — what to log, log tampering prevention, alert thresholds
- A10 Server-Side Request Forgery (SSRF) — what it enables, validation of user-supplied URLs

### Authentication

- JWT anatomy — header, payload, signature; signing algorithms (HS256, RS256, ES256) — symmetric vs. asymmetric
- Access tokens vs. refresh tokens — lifetime, storage (where to keep them in a browser), rotation
- Refresh token rotation — detecting reuse attacks (refresh token family)
- OAuth 2.0 grant types: Authorization Code (with PKCE), Client Credentials, Implicit (deprecated), Device Code
- OpenID Connect — ID token, UserInfo endpoint, discovery document
- Session-based authentication — server-side sessions, cookie flags (`HttpOnly`, `Secure`, `SameSite`)
- Multi-factor authentication — TOTP (Time-based One-Time Password), WebAuthn
- ASP.NET Core identity — `UserManager<T>`, `SignInManager<T>`, `RoleManager<T>`, password hashing, lockout

### Authorization

- Role-Based Access Control (RBAC) — roles assigned to users, permissions assigned to roles
- Policy-Based Authorization in ASP.NET Core — `IAuthorizationRequirement`, `IAuthorizationHandler`, `[Authorize(Policy = "...")]`
- Claims-Based Authorization — inspecting claims within policies
- Resource-Based Authorization — authorizing against a specific resource instance, `IAuthorizationService`
- Attribute-Based Access Control (ABAC) — policies evaluated against attributes of subject, resource, and environment

### Cryptography

- Symmetric encryption: AES — key sizes, modes of operation (CBC, GCM), IV/nonce requirements
- Asymmetric encryption: RSA, ECC — public/private key pairs, signing vs. encrypting
- Hashing: SHA-256, SHA-3, BLAKE3 — one-way, collision resistance, use cases
- Password hashing: bcrypt, Argon2, PBKDF2 — why regular hashing is wrong for passwords
- ASP.NET Core Data Protection API — key management, purpose strings, key rotation
- TLS internals — the TLS 1.3 handshake: key exchange, certificate validation, cipher suites

### API Security

- CORS — same-origin policy, preflight requests, allowed origins, credentials — why wildcards are dangerous
- Security headers: `Content-Security-Policy`, `Strict-Transport-Security`, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`
- Input validation and sanitization — never trust client input, validate at the boundary
- SQL injection in EF Core — `FromSqlRaw` vs. `FromSqlInterpolated` — when you can still be vulnerable
- Secrets management: Azure Key Vault with managed identity, HashiCorp Vault, `dotnet user-secrets` for development
- API key management — hashing stored keys, key rotation, scoped permissions
- Rate limiting as a defense — against credential stuffing, brute force, scraping

## Engineering Thinking and Judgment

### Decision Frameworks

- Build vs. buy — total cost of ownership, maintenance burden, flexibility, time to market
- Abstraction threshold — when the cost of an abstraction is less than the cost of repetition
- The boring technology principle — choosing proven tools over novel ones, and when to break the rule
- Reversibility — prefer reversible decisions, be more careful with irreversible ones (Jeff Bezos's one-way/two-way door)
- Tradeoff documentation — writing down what was traded and why before the context is lost

### Architecture Decision Records (ADRs)

- ADR format — title, status, context, decision, consequences
- When to write an ADR — any non-trivial technical decision that will affect the team for months or years
- ADR storage — next to the code they describe, in a dedicated `docs/decisions` directory
- Superseding ADRs — when a previous decision is reversed, record why

### Post-Mortem Analysis

- The blameless post-mortem — focusing on systems, not people
- Timeline reconstruction — the sequence of events, what was known when
- Contributing factors — the real causes, usually multiple and systemic
- Action items — concrete, assigned, time-bounded
- Reading public post-mortems: Cloudflare, GitHub, Stripe, Atlassian — building intuition for failure modes

### Debugging Discipline

- The scientific method applied to bugs: observe → hypothesize → predict → test → conclude
- Binary search debugging (git bisect) — isolating the commit that introduced a regression
- Rubber duck debugging — explaining the problem precisely often reveals the solution
- Reproducing production issues locally — environment parity, feature flags, log replay
- Differential debugging — what changed between when it worked and when it stopped
- Reading stack traces completely — not just the first line
- Using the debugger deliberately — breakpoints, conditional breakpoints, data breakpoints, step-over vs. step-into

### Estimation

- Back-of-envelope calculations for system design — QPS, storage, bandwidth
- PERT estimation — best case, worst case, most likely
- Story points and their failure modes — why they measure complexity, not time
- The planning fallacy — why estimates are almost always optimistic, and how to compensate

### Technical Writing

- RFC (Request for Comments) — proposing changes before implementing them
- Design documents — problem statement, proposed solution, alternatives considered, open questions
- Runbooks — step-by-step operational procedures for recurring tasks and incident response
- Incident reports — factual, blameless, focused on systemic issues

## Performance and Optimization

### Profiling Tools

- BenchmarkDotNet — microbenchmarks, statistical analysis, memory diagnostics, disassembly
- `dotnet-trace` — event-based profiling from the command line, trace collection
- `dotnet-dump` — collecting and analyzing process memory dumps
- `dotnet-counters` — real-time performance counter monitoring
- JetBrains dotTrace — CPU profiling, timeline profiling, call tree analysis
- JetBrains dotMemory — heap snapshot, memory traffic, retained objects
- Visual Studio Diagnostic Tools — CPU usage, memory usage, events
- PerfView — Windows-native, deep CLR and kernel event analysis

### Memory Optimization

- Allocation profiling — finding the code paths that allocate the most
- `Span<T>` — eliminating allocations in parsing, formatting, slicing
- `ArrayPool<T>.Shared` — renting arrays for temporary use
- `ObjectPool<T>` — pooling expensive-to-create objects (parsers, builders, clients)
- String operations — `StringBuilder` for concatenation, `string.Create()` for zero-copy string construction, `ReadOnlySpan<char>` for parsing
- Avoiding LINQ in hot paths — LINQ has allocation overhead from iterators and closures
- `stackalloc` — allocating small buffers on the stack for performance-critical paths

### CPU Optimization

- Eliminating unnecessary work — the best optimization is not doing the thing
- SIMD intrinsics via `System.Runtime.Intrinsics` — processing 128/256/512 bits at once
- Auto-vectorization — writing loops that the JIT can vectorize automatically
- Cache-friendly data structures — sequential access patterns, struct-of-arrays vs. array-of-structs
- Branch elimination — using lookup tables or bitwise operations instead of conditional branches

### I/O Optimization

- Async I/O — non-blocking calls, the I/O completion port model on Windows
- Buffered vs. unbuffered I/O — reading and writing in large chunks
- `System.IO.Pipelines` — high-performance I/O pipeline for parsing network protocols
- Connection pooling — `SqlConnection` pooling, `HttpClient` reuse via `IHttpClientFactory`
- Compression — choosing Brotli vs. Gzip, compression level vs. CPU tradeoff

### Database Performance

- N+1 query elimination — EF Core includes, split queries, projection
- Bulk operations — `EFCore.BulkExtensions`, `SqlBulkCopy`, `BULK INSERT`
- Dapper — a lightweight ORM for performance-critical read paths where EF Core is too heavy
- Connection pooling configuration — min pool size, max pool size, connection lifetime
- Read replicas — routing read-heavy operations to replicas
- Query result caching — caching expensive query results in Redis with appropriate TTL

## Interview Preparation

### Coding Interview

- The communication protocol — restate, clarify, design before coding, explain while coding, test
