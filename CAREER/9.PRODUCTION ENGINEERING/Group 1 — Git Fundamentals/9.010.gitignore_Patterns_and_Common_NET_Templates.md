### Where This Fits

You just cloned the `PaymentGateway` repo, opened it in Rider, and now `git status` is showing 400 changed files — all IDE metadata, build artifacts, and local user settings. None of it should ever touch the remote. Or a new engineer joins and accidentally commits their `bin/` directory, bloating the repo by 80 MB and triggering a history-rewrite cleanup that costs the team an afternoon. Both scenarios are prevented by a well-crafted `.gitignore` written on day one of the repo. This note covers how to write one correctly for a .NET solution, how the pattern syntax actually works, and what to do when a file that should be ignored is already tracked.

## What You're Doing and Why

`.gitignore` is a plain-text file that tells Git which files to pretend don't exist. Files matching its patterns are excluded from `git status`, `git add`, and `git diff` — they're invisible to Git's tracking machinery. For .NET solutions this means: build outputs (`bin/`, `obj/`), NuGet package caches, IDE-specific metadata (`.vs/`, `.idea/`, `*.user`), secrets (`appsettings.Development.json`, `*.pfx`), and environment files (`.env`, `launchSettings.json` in some workflows). The `.gitignore` file itself gets committed and shared — meaning the rules apply to every contributor on every machine the moment they clone.

### Mental Model

Think of `.gitignore` as a doorman with a blacklist. Files on the list never get into the building (the Git index). Files already inside — already tracked — are ignored by the doorman entirely; you need to explicitly evict them with `git rm --cached` before the doorman's rules apply. This is the single most important operational fact about `.gitignore`: **it does not un-track files that are already tracked.**

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`dotnet new gitignore`|Generate official .NET `.gitignore` from the SDK template|Starting a new .NET repo|
|`git check-ignore -v <file>`|Show which rule is ignoring a file (and from which `.gitignore`)|Debugging why a file is or isn't being ignored|
|`git status --ignored`|Show ignored files in addition to tracked/untracked|Auditing what's being filtered|
|`git rm --cached <file>`|Un-track a file without deleting it from disk|Removing a file that was committed but should be ignored|
|`git rm --cached -r <dir>/`|Un-track an entire directory recursively|Removing `bin/` or `.vs/` that snuck into history|
|`git ls-files --others --ignored --exclude-standard`|List all ignored files|Full audit of what's being suppressed|
|`git add -f <file>`|Force-add a file that matches an ignore pattern|Deliberately committing something that's in `.gitignore` (rare)|
|`echo "pattern" >> .gitignore`|Append a pattern from the shell|Quickest way to add a one-off ignore rule|

### The Scenario

You've just run `dotnet new webapi -n OrderService` and `git init` inside the solution directory. Before writing a single line of business code, you need to set up the `.gitignore` so the first `git add .` doesn't pollute the repository with build artifacts and IDE files. You're using VS Code on one machine and a teammate uses Rider on another.

```bash
# Step 2: Inspect what it generated
$ cat .gitignore | head -40

```

> Notice `[Bb]in/` and `[Oo]bj/` with bracket patterns — this handles both `Bin/` and `bin/` on case-sensitive filesystems (Linux CI agents).

```bash
# Step 3: Add IDE-specific rules not in the SDK template
# The SDK template covers .vs/ but not JetBrains Rider — add those
$ cat >> .gitignore << 'EOF'

## JetBrains Rider / IntelliJ
.idea/
*.iml
*.ipr
*.iws

## VS Code (team uses .editorconfig — workspace settings don't belong in source)
.vscode/

## Local environment secrets — NEVER commit these
.env
.env.local
appsettings.Development.json
appsettings.Local.json
*.pfx
*.p12
secrets.json

## OS artifacts
.DS_Store
Thumbs.db
desktop.ini
EOF
```

> The `!` prefix is a negation rule — `.vscode/extensions.json` and `launch.json` are team-shared and should be committed even though the rest of `.vscode/` is ignored.

```bash
# Step 4: Verify the first git add won't pull in artifacts
# Build first to generate bin/obj, then check what git sees
$ dotnet build -c Release > /dev/null 2>&1
$ git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be tracked)
        .gitignore
        OrderService.sln
        src/
        tests/

nothing added to commit but untracked files present
```

> `bin/` and `obj/` are completely absent from `git status` — the ignore rules are working before the first commit.

```bash
# Step 5: Confirm a specific file is being ignored and why
$ git check-ignore -v src/OrderService/bin/Release/net8.0/OrderService.dll
.gitignore:68:[Bb]in/	src/OrderService/bin/Release/net8.0/OrderService.dll
```

> `git check-ignore -v` prints `<file>:<line>:<pattern>\t<matched-path>` — tells you exactly which rule in which file triggered. Essential for debugging.

```bash
# Step 6: Commit the gitignore before anything else
$ git add .gitignore
$ git commit -m "chore: add .gitignore for .NET solution with IDE rules"
[main (root-commit) c3a1b2d] chore: add .gitignore for .NET solution with IDE rules
 1 file changed, 312 insertions(+)
 create mode 100644 .gitignore

# Step 7: Now safely add everything else
$ git add .
$ git status
On branch main
Changes to be committed:
  new file:   OrderService.sln
  new file:   src/OrderService/OrderService.csproj
  new file:   src/OrderService/Program.cs
  new file:   src/OrderService/appsettings.json
  new file:   tests/OrderService.Tests/OrderService.Tests.csproj
  new file:   tests/OrderService.Tests/UnitTest1.cs
```

> `git add .` after the gitignore is committed is safe — the rules are in effect and `bin/`, `obj/`, and IDE files are all invisible.

### What Could Go Wrong Mid-Walkthrough

**If you run `git add .` BEFORE committing `.gitignore`:**

The ignore rules are still applied (`.gitignore` is read from the working tree, not from history), but it's a fragile moment. If you accidentally committed artifacts before writing `.gitignore`, the rules won't help — already-tracked files are immune.

```bash
# Symptom: bin/ somehow got staged
$ git status
Changes to be committed:
  new file:   src/OrderService/bin/Release/net8.0/OrderService.dll
  new file:   src/OrderService/bin/Release/net8.0/OrderService.runtimeconfig.json
  ... (hundreds more)

# Fix: unstage everything and try again
$ git reset HEAD .
$ git status
# bin/ is now back to untracked — and .gitignore will suppress it on the next add
```

## Production Scenario

### Realistic Context

`InventoryApi` has been in development for six months. Three engineers committed their local `bin/` directories and `.vs/` folders before a `.gitignore` was added — the repo is 220 MB when it should be under 10 MB. New clones are slow, CI pipelines hit GitHub's LFS warning thresholds, and `git status` on any branch is noisy with phantom changes. The team needs to retroactively clean the tracked artifacts from history without losing any code. This is the most common `.gitignore` emergency on real teams.

## Gotchas and Troubleshooting

### File is in `.gitignore` but `git status` still shows it as modified

**Root cause:** The file is already tracked. `.gitignore` only prevents untracked files from being staged. Once a file is in Git's index, the ignore rules have no effect on it — Git tracks its changes regardless.

```bash
# Symptom
$ git status
	modified:   src/OrderService/appsettings.Development.json

# Confirm it's in .gitignore
$ grep "appsettings.Development" .gitignore
appsettings.Development.json

# Confirm it's still tracked
$ git ls-files src/OrderService/appsettings.Development.json
src/OrderService/appsettings.Development.json
# (non-empty output = it's tracked)
```

**Fix:**

```bash
$ git rm --cached src/OrderService/appsettings.Development.json
rm 'src/OrderService/appsettings.Development.json'

$ git commit -m "chore: stop tracking appsettings.Development.json (now in .gitignore)"
```

**Prevention:** Add secret-adjacent config files to `.gitignore` before writing them. If using `dotnet user-secrets`, `appsettings.Development.json` should never hold secrets — use `secrets.json` via `dotnet user-secrets` ([[9.472 — dotnet user-secrets — Local Development Workflow]]) instead.

---

### `git check-ignore -v <file>` returns nothing — but the file IS being ignored

**Root cause:** The file is in a nested `.gitignore` (project-level or directory-level), not the root one. Or it's being excluded by a global gitignore (`~/.gitignore_global` or the path set in `core.excludesFile`). `git check-ignore -v` checks all of them but you need to look at all layers.

```bash
# Check all levels explicitly
$ git check-ignore -v --no-index src/OrderService/.vs/config/applicationhost.config
# If still empty, check the global excludes file
$ git config --global core.excludesFile
/home/tarek/.gitignore_global

$ cat /home/tarek/.gitignore_global | grep "\.vs"
.vs/
# Found — it's the global file, not the repo-level .gitignore
```

**Fix:** Document in the team README whether the team relies on global gitignore — it's invisible to other contributors and causes "works on my machine" ignore behavior.

**Prevention:** Keep all team-shared ignore rules in the repo's `.gitignore`. Global `~/.gitignore_global` is for personal IDE preferences that shouldn't be imposed on the team (e.g., `.DS_Store` for macOS users, `Thumbs.db` for Windows users).

---

### Pattern added to `.gitignore` is silently not working — file still appears in `git status`

**Root cause:** Four common causes — (1) pattern has a trailing space; (2) the path separator is wrong (`\` instead of `/` on Windows editors); (3) the pattern is meant to match a directory but is missing the trailing `/`; (4) a negation rule `!pattern` later in the file is re-including the file.

```bash
# Diagnose: check-ignore gives you the winning rule
$ git check-ignore -v src/OrderService/bin/
# If empty, the file is tracked (see gotcha #1 above)
# If it shows a rule, that rule is the one winning

# Check for negation rules that might be overriding your ignore
$ grep "^!" .gitignore
!.vscode/extensions.json
!.vscode/launch.json
# If you see !src/OrderService/bin/ here, that's the problem
```

**Fix:** Rules in `.gitignore` are processed top-to-bottom; a `!` negation rule at the bottom beats an ignore rule above it. Put specific negation rules immediately after their parent ignore rule, not at the bottom of the file.

**Prevention:** Use `git check-ignore -v <path>` as the first diagnostic for any ignore mystery — it shows the exact line and file that matched. Never guess.

---

### `git rm --cached -r .` accidentally stages deletions of source files

**Root cause:** If `.gitignore` has an overly broad pattern — e.g., `*.cs` accidentally added, or a `src/` rule — then `git rm --cached -r .` followed by `git add .` will exclude source files from the re-add, making Git think they were deleted.

```bash
# Symptom after git rm --cached -r . && git add .
$ git status | grep "deleted:" | head -5
deleted:    src/OrderService/Controllers/OrdersController.cs
deleted:    src/OrderService/Models/Order.cs
# DANGER — these are source files, not artifacts
```

**Fix:**

```bash
# DO NOT COMMIT — abort immediately
$ git checkout -- src/
# Or if you need to reset the whole index:
$ git reset HEAD .
# Then fix the overly-broad pattern in .gitignore before retrying
```

**Prevention:** Always run `git rm --cached -r --dry-run .` and review the output before the real run. Pipe through `grep -v -E "\.(dll|pdb|exe|nupkg|user|suo)$"` to catch anything that isn't an artifact.

---

### `.gitignore` committed to the repo but a teammate's IDE files still show up in `git status` on their machine

**Root cause:** The teammate has their IDE files in a directory path that doesn't match the pattern. For example, `.gitignore` has `.idea/` but Rider created its files at `InventoryApi.sln.iml` in the root (not inside `.idea/`). Or they're on Windows and the path case doesn't match the pattern on a case-sensitive CI host.

```bash
# Teammate runs:
$ git status
Untracked files:
        InventoryApi.iml
        InventoryApi.sln.iml

# These match *.iml, not .idea/
$ grep "iml" .gitignore
.idea/
*.iml
# *.iml is there — check if it's missing
```

**Fix:** Confirm that `*.iml` is in `.gitignore` at the root (not just inside a `.idea/` block). If it's missing:

```bash
$ echo "*.iml" >> .gitignore
$ git add .gitignore
$ git commit -m "chore: add *.iml to .gitignore for Rider users"
```

**Prevention:** When adding `.gitignore` rules, cover both the directory (`/.idea/`) and the file extensions (`*.iml`) that the IDE generates outside that directory. Test on the actual IDE before committing.

---

### Committed a secret (`appsettings.Development.json` with a connection string) — `git rm --cached` isn't enough

**Root cause:** `git rm --cached` removes the file from the current commit, but the secret is still in the repo's full history. Anyone who clones the repo or checks out an older commit can still read it.

```bash
# Wrong assumption: git rm --cached removes it from history
$ git rm --cached src/OrderService/appsettings.Development.json
$ git commit -m "remove secret"
# Secret is still in commit abc1234 — anyone with the repo can see it
$ git show abc1234:src/OrderService/appsettings.Development.json
# Output: the connection string is right there
```

**Fix:** Rotate the secret immediately — treat it as compromised. Then rewrite history using `git filter-repo` ([[9.037 — Rewriting History — filter-branch and filter-repo]]) to purge it from all commits, then force-push. This requires team coordination and all contributors to re-clone.

**Prevention:** Use `dotnet user-secrets` ([[9.472 — dotnet user-secrets — Local Development Workflow]]) for local secrets. Never put credentials in any `appsettings.*.json` file. Add a pre-commit hook ([[9.021 — Git Hooks — pre-commit and pre-push Basics]]) that scans staged files for connection string patterns before allowing the commit.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# Shell aliases for .gitignore work
# Add to ~/.bashrc or ~/.zshrc

# Generate official .NET gitignore instantly
alias gi-dotnet='dotnet new gitignore'

# Check what rule is ignoring a file
alias gci='git check-ignore -v'

# List everything currently being ignored in the repo
alias gignored='git ls-files --others --ignored --exclude-standard'

# Quick audit: see if any dll/pdb/exe snuck into tracking
alias gartifacts='git ls-files | grep -E "\.(dll|pdb|exe|nupkg|suo|user|iml)$"'

# Un-track a file from index without deleting it
guncache() { git rm --cached "$@" && echo "Removed from index. Add to .gitignore if not already there."; }
```

### Automation Opportunity

A pre-commit safety check that blocks artifact commits before they happen — far cheaper than cleaning them up after:

```bash
#!/usr/bin/env bash
# .git/hooks/pre-commit (or hooks/pre-commit for Husky / git hooks manager)
# Blocks commits containing binary build artifacts

set -euo pipefail

ARTIFACT_PATTERN='\.(dll|pdb|exe|nupkg|snupkg|cache|suo|user)$'

STAGED_ARTIFACTS=$(git diff --cached --name-only | grep -E "${ARTIFACT_PATTERN}" || true)

if [[ -n "${STAGED_ARTIFACTS}" ]]; then
  echo "❌ Pre-commit: build artifacts detected in staged files:"
  echo "${STAGED_ARTIFACTS}"
  echo ""
  echo "Add them to .gitignore and run: git rm --cached <file>"
  exit 1
fi

# Also block known secret file names
SECRET_PATTERN='(appsettings\.(Development|Local|Staging)\.json|\.env\.local|secrets\.json|\.pfx|\.p12)$'
STAGED_SECRETS=$(git diff --cached --name-only | grep -E "${SECRET_PATTERN}" || true)

if [[ -n "${STAGED_SECRETS}" ]]; then
  echo "❌ Pre-commit: possible secret file detected in staged files:"
  echo "${STAGED_SECRETS}"
  echo ""
  echo "If this file doesn't contain secrets, force-add with: git add -f <file>"
  echo "If it does, use dotnet user-secrets instead."
  exit 1
fi

exit 0
```

```bash
# Install it
$ cp hooks/pre-commit .git/hooks/pre-commit
$ chmod +x .git/hooks/pre-commit

# Test it
$ touch src/OrderService/bin/Debug/net8.0/OrderService.dll
$ git add src/OrderService/bin/Debug/net8.0/OrderService.dll
$ git commit -m "test"
❌ Pre-commit: build artifacts detected in staged files:
src/OrderService/bin/Debug/net8.0/OrderService.dll
```

For team-wide distribution, use [Husky.Net](https://alirezanet.github.io/Husky.Net/) which installs hooks via `dotnet tool install` and stores them in source control alongside the `.gitignore`.

### Keyboard / CLI Efficiency Notes

Five habits that separate engineers with clean repos from those doing history rewrites:

1. **`dotnet new gitignore` on the first command after `git init`** — before `dotnet new`, before opening the IDE. Order matters: init → gitignore → first commit → project files.
2. **`git check-ignore -v <file>` before filing a bug** — every "why is this file ignored/not ignored" question is answered by this command in under 3 seconds. Never guess.
3. **`git ls-files | grep -E "\.(dll|pdb|exe)$"` in CI** — add this as a step in the build pipeline to fail the build if artifacts are ever tracked. Catches mistakes that slip past local hooks.
4. **`gartifacts` alias from the Aliases section** — run it after onboarding a new developer or merging a large refactor PR. Takes 2 seconds, prevents 2-hour cleanup sessions.
5. **Keep `.gitignore` in the first commit, alone** — `git log --oneline .gitignore` should show one commit at the very root of history. It signals to every future contributor that the repo was set up with discipline.

### Spoken Answers

**Q: Walk me through how you'd set up a `.gitignore` for a brand-new .NET solution from scratch.**

> **Average answer:** "I'd copy a template from GitHub or gitignore.io and put it in the root."

> **Great answer:** "First command after `git init` — before I even create the project files — is `dotnet new gitignore`. That generates the official .NET template maintained by the SDK team, which covers `bin/`, `obj/`, NuGet caches, `.vs/`, and most Visual Studio artifacts. Then I append whatever it doesn't cover for our specific setup: Rider's `.idea/` directory and `*.iml` files, any `appsettings.Development.json` or `.env` files that hold local credentials, and OS junk like `.DS_Store`. I commit the `.gitignore` as its own first commit before running `git add .` on anything else — that way the rules are in effect before any artifacts exist. Finally, I check the result with `git check-ignore -v` on a couple of expected-ignored files to confirm it's working. The thing I explicitly don't do is commit first and add `.gitignore` later — once artifacts are tracked, you need `git rm --cached` to clean them up, which is a whole extra step that can go wrong."

---

**Q: You've added a pattern to `.gitignore` but `git status` still shows the file as modified. What's happening and how do you fix it?**

> **Average answer:** "Maybe the pattern is wrong — I'd check the syntax."

> **Great answer:** "The pattern syntax is almost never the issue here. `.gitignore` only prevents untracked files from being staged — it has zero effect on files Git is already tracking. If a file is in Git's index, it's tracked, and ignore rules don't apply to it. I confirm this with `git ls-files <file>` — if it returns output, the file is tracked. The fix is `git rm --cached <file>`, which removes it from the index without touching the disk copy. After that commit, the ignore rule kicks in and the file disappears from `git status`. I also verify with `git check-ignore -v <file>` to make sure the pattern actually matches — because if it doesn't match, even `git rm --cached` won't help long-term; it'll reappear as an untracked file on the next `git add .`."

**Q: A junior developer accidentally committed `appsettings.Development.json` containing a database connection string. What do you do?**

> **Average answer:** "`git rm --cached` the file, add it to `.gitignore`, and commit the fix."

> **Great answer:** "Two things happen in parallel. First, I rotate the credential immediately — treat it as compromised from the moment it was pushed. If it's an Azure SQL connection string, I change the password via the portal right now, before doing anything else. The git history is a secondary concern. `git rm --cached` only removes the file from future commits; the secret is still in every historical commit and anyone who has cloned the repo can see it. The proper remediation for the git side is `git filter-repo` to rewrite every commit that contains the file, followed by a force-push and a team-wide re-clone. That's disruptive, which is why the credential rotation happens first and independently. For prevention: add a pre-commit hook that blocks files matching `appsettings.*.json` from being staged, and switch local development to `dotnet user-secrets` so secrets never go near the filesystem path Git is watching."

### War Story Prompt

A new `NotificationWorker` service was bootstrapped by copy-pasting an older repo's structure — the older repo had no `.gitignore`. On day two, three developers pushed commits and CI started reporting a 340 MB repository size that cloned in over 90 seconds. Investigation with `git ls-files | grep "\.dll$" | wc -l` returned 1,847. Every developer's local build output had been committed. The team spent four hours running `git rm --cached -r .`, re-committing, and coordinating force-pushes while keeping the main branch deployable. The incident was entirely preventable by one command — `dotnet new gitignore` — on the first day. The team added a CI step that fails the build if `git ls-files | grep -E '\.(dll|pdb|exe)$'` returns any output, and it has never fired since.

## Decision Framework

### When to Reach for This

- [ ] Starting a new repo — add `.gitignore` as the very first commit
- [ ] Onboarding a new developer — verify `git status` is clean after first build
- [ ] After a new IDE or tool is introduced to the team — add its artifact patterns
- [ ] `git status` is showing IDE or build files as modified/untracked
- [ ] CI clone time has grown unexpectedly — audit for tracked artifacts with `gartifacts`
- [ ] A secret file was accidentally staged — immediate `git rm --cached` + credential rotation

### When NOT To

- [ ] The file is IDE-specific to one developer (`.DS_Store`, `Thumbs.db`) — use `~/.gitignore_global` instead so you don't impose macOS/Windows noise suppression on the whole team
- [ ] You want to exclude a file from a single build or directory only — create a local `.gitignore` in that subdirectory rather than polluting the root ignore file
- [ ] You need to remove a secret from history (not just current tracking) — `git rm --cached` is insufficient; use `git filter-repo` ([[9.037 — Rewriting History — filter-branch and filter-repo]])

### Tool Comparison

||`.gitignore` in repo root|`~/.gitignore_global`|Directory-level `.gitignore`|
|---|---|---|---|
|Scope|Whole repo, all contributors|One machine, all repos|One directory subtree|
|Committed to repo?|Yes|No|Yes|
|When it's the right call|Team-wide rules (build artifacts, IDE files)|Personal tools (your specific IDE, OS junk)|Monorepo subproject with unique artifacts|
|Invisible to teammates?|No — everyone sees it|Yes — silent for others|No — committed alongside code|

## Self-Check

### Conceptual Questions

1. You add `*.log` to `.gitignore`. A file `logs/app.log` already exists and is tracked. What happens to it — is it now ignored?
2. What does the pattern `[Bb]in/` match that `bin/` alone would not, and why does it matter for .NET repos?
3. What is the difference between `/bin/` and `bin/` in a `.gitignore` pattern?
4. You have these two lines in `.gitignore`. What is the result for the file `src/config/appsettings.Development.json`?
    
    ```
    appsettings.Development.json!src/config/appsettings.Development.json
    ```
    
5. What command tells you which specific rule and which file is responsible for ignoring a given path?
6. A teammate says "I can see my `.env` file in `git status` as untracked even though we have `.env` in `.gitignore`." What are two possible explanations?
7. What's the relationship between `.gitignore` and `.dockerignore`, and should they be identical? (Link to [[9.123 — .dockerignore — Writing One for a .NET Solution]].)

<details> <summary>Answers</summary>

1. No — nothing changes. `.gitignore` does not affect already-tracked files. `logs/app.log` will continue to show up as modified whenever its contents change. You must `git rm --cached logs/app.log` and commit the deletion before the ignore rule takes effect on it.
    
2. `[Bb]in/` matches both `bin/` and `Bin/` (and `BIn/`, `bIn/` etc.). On case-sensitive filesystems (Linux — which includes virtually all CI agents), MSBuild can produce output in `Bin/` on certain project types or configurations. The bracket pattern ensures the ignore rule fires regardless of case, making it portable between developers' macOS/Windows machines and Linux CI hosts.
    
3. `/bin/` (with a leading slash) only matches a `bin/` directory at the root of the repository — i.e., `<repo-root>/bin/`. `bin/` (without a leading slash) matches any `bin/` directory anywhere in the tree — `src/OrderService/bin/`, `tests/OrderService.Tests/bin/`, etc. For .NET repos, `bin/` (no leading slash) is correct because each project has its own `bin/` subdirectory.
    
4. The file is **not ignored** — it will appear in `git status` as untracked (if not tracked) or modified (if tracked). In `.gitignore`, rules are processed top-to-bottom and later rules override earlier ones. The negation rule `!src/config/appsettings.Development.json` comes after the ignore rule and re-includes that specific file. This is the correct pattern to use when you want to ignore a pattern broadly but allow one specific instance — common for `!.vscode/extensions.json` inside a broader `.vscode/` ignore.
    
5. `git check-ignore -v <path>`. Output format: `<gitignore-file>:<line-number>:<pattern>\t<matched-path>`. It shows the exact file and line number of the winning rule, traversing all `.gitignore` files (repo-level, directory-level, and global).
    
6. Two explanations: (a) The `.env` file is already tracked — it was committed before `.gitignore` was written or before this pattern was added. `git ls-files .env` will confirm; fix with `git rm --cached .env`. (b) The `.gitignore` pattern is `.env` but the file is `.env.local` or `.env.development` — the pattern doesn't match the filename. Confirm with `git check-ignore -v .env.local`.
    
7. `.gitignore` and `.dockerignore` serve different engines (Git vs Docker build context) and have different purposes, so they should not be identical. `.gitignore` excludes files from Git tracking — IDE metadata, secrets, build artifacts. `.dockerignore` excludes files from the Docker build context sent to the daemon — it should exclude `bin/`, `obj/`, `.git/`, `*.md`, test projects, and IDE files, but for performance reasons (smaller context = faster builds) rather than security. They often overlap but `.dockerignore` needs to be independently maintained and is usually more aggressive about excluding anything not needed to compile the app. See [[9.123 — .dockerignore — Writing One for a .NET Solution]] for the complete `.dockerignore` treatment.
    

</details>

**Drill 2 — Fix the broken thing**

A developer ran the following and is now seeing this:

```bash
$ git status
On branch main
Changes not staged for commit:
  modified:   src/NotificationWorker/bin/Debug/net8.0/NotificationWorker.dll

$ cat .gitignore | grep bin
[Bb]in/
```

The pattern is there. The file is still showing up. Fix it.

<details> <summary>Solution</summary>

**Diagnosis:** The file `src/NotificationWorker/bin/Debug/net8.0/NotificationWorker.dll` is already tracked in Git's index. `.gitignore` doesn't affect tracked files — the pattern `[Bb]in/` would prevent it from being staged for the first time, but has no power over files already in the index.

```bash
# Confirm it's tracked
$ git ls-files src/NotificationWorker/bin/Debug/net8.0/NotificationWorker.dll
src/NotificationWorker/bin/Debug/net8.0/NotificationWorker.dll
# Non-empty output = tracked

# Remove from index only (keep file on disk)
$ git rm --cached src/NotificationWorker/bin/Debug/net8.0/NotificationWorker.dll
rm 'src/NotificationWorker/bin/Debug/net8.0/NotificationWorker.dll'

# Better: remove the whole bin/ directory from tracking
$ git rm --cached -r src/NotificationWorker/bin/
rm 'src/NotificationWorker/bin/Debug/net8.0/NotificationWorker.dll'
rm 'src/NotificationWorker/bin/Debug/net8.0/NotificationWorker.pdb'
# ... etc

$ git commit -m "chore: remove tracked bin/ artifacts from NotificationWorker"

# Verify — build and check status
$ dotnet build > /dev/null 2>&1
$ git status
On branch main
nothing to commit, working tree clean
# Fixed
```

</details>

**Drill 3 — Under time pressure**

A security alert fires: `git log --all --full-history -- "**/*.pfx"` returns a commit hash — someone committed a `.pfx` certificate file to `InventoryApi` three weeks ago. You have 10 minutes before the security team escalates. Walk through exactly what you do.

<details> <summary>Solution</summary>

```bash
# FIRST: Find and rotate the credential — do this before any git operations
# (Assume the cert is for a staging Azure service — revoke it in the Azure portal NOW)
# This is non-negotiable: the certificate is compromised from the moment it was pushed

# Step 1: Identify the file path and commit
$ git log --all --full-history --oneline -- "**/*.pfx"
4f3a91c chore: add cert for staging deployment
# Found: commit 4f3a91c, 3 weeks ago

$ git show 4f3a91c --stat
 devops/certs/staging-intervision.pfx | Bin 0 -> 2847 bytes added

# Step 2: Confirm it's still in history (not just the working tree)
$ git show 4f3a91c:devops/certs/staging-intervision.pfx | wc -c
2847
# Still there

# Step 3: Remove from current tracking (immediate, low-disruption step)
$ git rm --cached devops/certs/staging-intervision.pfx
$ echo "*.pfx" >> .gitignore
$ echo "*.p12" >> .gitignore
$ git add .gitignore
$ git commit -m "security: remove pfx from tracking, add to .gitignore — cert being rotated"

# Step 4: Post in incident channel
# "PFX certificate committed in 4f3a91c has been removed from HEAD.
#  Cert is being rotated NOW. Full history rewrite (git filter-repo)
#  is scheduled for [time] — all contributors will need to re-clone.
#  Do not use the old cert after [rotation time]."

# Step 5: (after credential rotation, schedule with team)
# Full history purge with git filter-repo — see [[9.037 — Rewriting History]]
$ pip install git-filter-repo --break-system-packages
$ git filter-repo --path devops/certs/staging-intervision.pfx --invert-paths
# Force-push all branches, notify team to re-clone
```

Why each step in this order:

- Credential rotation is instant and independent of git operations — it eliminates the security risk immediately
- `git rm --cached` + commit tells future `git log` that the file was removed, even though history still has it
- The incident channel post sets expectations and prevents teammates from pulling during the rewrite
- `git filter-repo` is scheduled rather than run immediately — it's disruptive and the security risk is already mitigated by the rotation

</details>
