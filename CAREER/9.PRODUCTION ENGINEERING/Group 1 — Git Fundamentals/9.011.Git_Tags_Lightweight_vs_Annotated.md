### Where This Fits

Your CI pipeline just ran green on `main` and it's time to cut a release. Before you push to production or create a GitHub Release, you need a tag — a permanent marker on the exact commit that shipped. You'll also reach for tags when a hotfix deployment targets "the last known-good release" (see [[9.005 — Git Branch — Create, Switch, Delete]]'s production scenario), when Kubernetes rollbacks need a pinned image tag derived from a Git tag, or when your changelog tooling queries `git describe` to generate version numbers automatically.

## What You're Doing and Why

Tags are named pointers to commits — like branches, but they don't move. A branch pointer advances with every new commit on that branch; a tag stays pinned to the exact commit it was placed on, forever. That permanence is what makes them useful for releases: `v2.14.1` always means the same commit, regardless of how many more commits land on `main` afterward.

The lightweight vs annotated distinction matters operationally: a **lightweight tag** is just the pointer file (identical to a branch, but static). An **annotated tag** is a full Git object — it stores a tagger name, email, timestamp, and a message, and can be GPG-signed. For releases, always use annotated tags: they carry the metadata your changelog tools, `git describe`, and GitHub Releases depend on. Lightweight tags are fine for personal bookmarks and ephemeral CI markers you'll never ship from.

### Mental Model

Think of commits as frames in a film reel. Branches are the director's finger pointing at the current frame — it moves forward as new frames are added. Tags are a sticky label glued onto a specific frame: "this is the version we shipped to cinemas." The label never moves. You can always rewind to that exact frame by name.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git tag v2.14.1`|Creates a lightweight tag at HEAD|Ephemeral local bookmark; never for releases|
|`git tag -a v2.14.1 -m "Release v2.14.1"`|Creates an annotated tag at HEAD|Every production release|
|`git tag -a v2.14.1 <commit-sha> -m "..."`|Creates an annotated tag at a specific commit|Tagging a past commit you missed at release time|
|`git tag`|Lists all tags (alphabetical)|Checking what tags exist locally|
|`git tag -l 'v2.*'`|Lists tags matching a glob|Finding all v2.x releases|
|`git show v2.14.1`|Shows tag metadata + the tagged commit|Inspecting who tagged it and when|
|`git push origin v2.14.1`|Pushes a single tag to origin|After creating a release tag locally|
|`git push origin --tags`|Pushes **all** local tags to origin|Syncing all missing tags in one shot|
|`git push origin --follow-tags`|Pushes current branch + any annotated tags reachable from it|The recommended default for release pushes|
|`git tag -d v2.14.1`|Deletes a tag locally|Fixing a bad tag before it's pushed|
|`git push origin --delete v2.14.1`|Deletes a tag on origin|Retracting a pushed tag (rare; communicate with team first)|
|`git fetch --tags`|Fetches all remote tags not present locally|Syncing tags after a teammate pushed a release|
|`git describe --tags --abbrev=0`|Prints the most recent tag reachable from HEAD|Getting the base version for `git describe`|
|`git describe --tags`|Prints nearest tag + commit count + short SHA|Generating build version strings|
|`git checkout v2.14.1`|Checks out the tagged commit (detached HEAD)|Reproducing the exact build from a release|

## Step-by-Step Walkthrough

### The Scenario

The `PaymentGateway` service has passed all CI checks on `main`. Commit `f3c8a1e` is the release candidate for `v2.15.0`. You need to tag it as an annotated release tag, push it to origin, and verify the tag is visible on the remote so the deployment pipeline can resolve it.

### Full Walkthrough

```bash
$ git log --oneline -5
f3c8a1e (HEAD -> main, origin/main) feat: add retry budget enforcement to idempotency layer
a1f9c3d fix: correct tax rounding on international orders
7e2b1a0 feat: add order export endpoint
3c9d5f2 chore: bump EF Core to 8.0.4
9d2f3c0 fix: correct gateway timeout on retries
```

Confirm `HEAD` is exactly where you want the tag — `f3c8a1e` is the commit that passed CI and will ship.

```bash
$ git tag -l 'v2.*'
v2.13.0
v2.13.1
v2.14.0
v2.14.1
```

Check existing tags first to confirm your new version number doesn't already exist and follows the team's numbering sequence.

```bash
$ git tag -a v2.15.0 -m "Release v2.15.0

- Add retry budget enforcement to idempotency layer (PAY-988)
- Fix tax rounding on international orders (PAY-1001)
- Bump EF Core to 8.0.4"
```

Annotated tag with `-a` and a multiline message via `-m`. The message becomes the release notes body in `git show` and GitHub Releases. Write it like a changelog entry, not a commit message.

```bash
$ git show v2.15.0
tag v2.15.0
Tagger: Alice Chen <alice.chen@acme.com>
Date:   Fri Jun 26 14:32:07 2026 +0000

Release v2.15.0

- Add retry budget enforcement to idempotency layer (PAY-988)
- Fix tax rounding on international orders (PAY-1001)
- Bump EF Core to 8.0.4

commit f3c8a1eab2c7d9e04f15a836b29c1d73e8f042a1 (HEAD -> main, tag: v2.15.0, origin/main)
Author: Alice Chen <alice.chen@acme.com>
Date:   Fri Jun 26 11:15:22 2026 +0000

    feat: add retry budget enforcement to idempotency layer
...
```

Verify the tagger identity, date, message, and the commit SHA before pushing. Once pushed, retracting is a team-coordination event.

```bash
$ git push origin v2.15.0
Enumerating objects: 1, done.
Counting objects: 100% (1/1), done.
Writing objects: 100% (1/1), 276 bytes | 276.00 KiB/s, done.
Total 1 (delta 0), reused 0 (delta 0), pack-reused 0
To https://github.com/acme/PaymentGateway.git
 * [new tag]         v2.15.0 -> v2.15.0
```

Tags are not pushed with `git push origin main` — they require an explicit push. Use `--follow-tags` going forward to avoid forgetting (see Aliases section).

```bash
$ git ls-remote --tags origin | grep v2.15
b9f3a2c1d4e7f8a0b1c2d3e4f5a6b7c8d9e0f1a2	refs/tags/v2.15.0
b9f3a2c1d4e7f8a0b1c2d3e4f5a6b7c8d9e0f1a2	refs/tags/v2.15.0^{}
```

The `^{}` entry (called the "dereferenced" object) confirms this is an annotated tag — it points to the commit object behind the tag object. Lightweight tags produce only one line here.

### What Could Go Wrong Mid-Walkthrough

**You tagged the wrong commit — but haven't pushed yet:**

```bash
# Check what HEAD was when you ran the tag command
$ git show v2.15.0 | head -10
tag v2.15.0
Tagger: Alice Chen <alice.chen@acme.com>
Date:   Fri Jun 26 14:32:07 2026 +0000
...
commit a1f9c3d...   # ← wrong! this is the commit BEFORE the one you wanted
```

```bash
# Delete locally and recreate at the correct commit
$ git tag -d v2.15.0
Deleted tag 'v2.15.0' (was b9f3a2c)

$ git tag -a v2.15.0 f3c8a1e -m "Release v2.15.0 ..."
$ git show v2.15.0 | head -5
tag v2.15.0
...
commit f3c8a1e...   # ← correct
```

**You tagged the wrong commit and already pushed it:**

Stop. Don't silently overwrite. Coordinate with the team first (see Gotcha: `! [rejected] v2.15.0 -> v2.15.0 (already exists)`), then follow the force-push procedure with explicit team notification.

## Production Scenario

### Realistic Context

The `InventoryApi` service is on a two-week release cadence. Every release goes through three steps: tag on `main`, build triggered by the tag in GitHub Actions (which pushes a container image tagged with the Git tag to ACR), and deploy to AKS with that image tag pinned in the Helm values. The CI pipeline reads the Git tag via `git describe` to stamp the container image and inject into `ASPNETCORE_ENVIRONMENT`-aware version endpoints. A missing or malformed tag breaks the entire release chain.

### Full Implementation / Resolution

The full tagging sequence for a production release, with the CI integration pattern:

```bash
# 1. Confirm you're on the right commit, main is clean and pushed
$ git status
On branch main
Your branch is up to date with 'origin/main'.
nothing to commit, working tree clean

$ git log --oneline -3
c9e3f7a (HEAD -> main, origin/main) feat: add stock reservation timeout (INV-204)
b2d1e4c fix: correct reorder threshold calculation (INV-198)
9a1f0e2 chore: update integration test timeouts

# 2. Verify no tag collision
$ git tag -l 'v3.2.*'
v3.2.0
v3.2.1
v3.2.2
# Next should be v3.2.3

# 3. Create the annotated release tag with a structured message
$ git tag -a v3.2.3 -m "Release v3.2.3

Changes:
- feat: add stock reservation timeout (INV-204)
- fix: correct reorder threshold calculation (INV-198)

Breaking changes: none
Migration required: none"

# 4. Push the commit AND the tag together
$ git push origin main --follow-tags
Enumerating objects: 1, done.
Counting objects: 100% (1/1), done.
Writing objects: 100% (1/1), 302 bytes | 302.00 KiB/s, done.
Total 1 (delta 0), reused 0 (delta 0), pack-reused 0
To https://github.com/acme/InventoryApi.git
   b2d1e4c..c9e3f7a  main -> main
 * [new tag]         v3.2.3 -> v3.2.3
```

The GitHub Actions release pipeline reads the tag via `git describe`:

```yaml
# .github/workflows/release.yml (relevant extract)
# Triggered by: push of tags matching v*.*.*
on:
  push:
    tags:
      - 'v[0-9]+.[0-9]+.[0-9]+'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0        # Required — shallow clone has no tag history

      - name: Derive version from tag
        id: version
        run: |
          VERSION=$(git describe --tags --abbrev=0)
          echo "VERSION=${VERSION}" >> $GITHUB_OUTPUT
          echo "Building image for ${VERSION}"

      - name: Build and push container image
        run: |
          docker build -t acmecr.azurecr.io/inventoryapi:${{ steps.version.outputs.VERSION }} .
          docker push acmecr.azurecr.io/inventoryapi:${{ steps.version.outputs.VERSION }}
```

Helm deploy step in the same pipeline pins the image tag:

```bash
helm upgrade inventoryapi ./charts/inventoryapi \
  --set image.tag=${{ steps.version.outputs.VERSION }} \
  --set image.repository=acmecr.azurecr.io/inventoryapi \
  --namespace production \
  --atomic \
  --timeout 5m
```

## Gotchas and Troubleshooting

### "! [rejected] v2.15.0 -> v2.15.0 (already exists)"

**Root cause:** You're trying to push a tag that already exists on origin with the same name. Git refuses to silently overwrite remote tags.

```bash
$ git push origin v2.15.0
To https://github.com/acme/PaymentGateway.git
 ! [rejected]        v2.15.0 -> v2.15.0 (already exists)
error: failed to push some refs to 'https://github.com/acme/PaymentGateway.git'
hint: Updates were rejected because the tag already exists in the remote.
```

**Fix:**

```bash
# Step 1: Communicate with the team FIRST — don't silently rewrite history
# Step 2: Delete the remote tag
$ git push origin --delete v2.15.0
To https://github.com/acme/PaymentGateway.git
 - [deleted]         v2.15.0

# Step 3: Recreate locally at the correct commit
$ git tag -d v2.15.0
$ git tag -a v2.15.0 f3c8a1e -m "Release v2.15.0 (corrected tag)"

# Step 4: Push with force (only way to overwrite — use sparingly)
$ git push origin v2.15.0
```

**Prevention:** Always run `git show <tag>` and verify the commit SHA before pushing. Treat pushed annotated tags as immutable — the cost of coordinating a retraction is high enough that a 10-second pre-push check is always worth it.

---

### Tags not visible after `git fetch` — `git fetch` doesn't pull tags by default in all scenarios

**Root cause:** `git fetch origin` fetches branch refs and tags reachable from fetched branches, but if a tag points to a commit not on any branch you fetched, it won't appear. Also, some older Git configs suppress tag fetching.

```bash
$ git tag -l 'v2.15.*'
# (empty — but teammate says v2.15.0 exists on origin)
```

**Fix:**

```bash
# Explicit tag fetch
$ git fetch --tags
From https://github.com/acme/PaymentGateway
 * [new tag]         v2.15.0    -> v2.15.0

$ git tag -l 'v2.15.*'
v2.15.0
```

**Prevention:** Add `fetch.tags = true` to your global Git config so `git fetch` always retrieves tags:

```bash
$ git config --global fetch.tags true
```

Or use `git fetch --tags` in any CI step that needs to resolve version information — the shallow-clone problem below is the more common CI culprit.

---

### `git describe` returns "fatal: No names found, cannot describe anything."

**Root cause:** The CI checkout is a shallow clone (`--depth 1`), which doesn't fetch tag history. `git describe` walks back through commit ancestry to find the nearest tag — with depth 1, there is no ancestry to walk.

```bash
$ git describe --tags --abbrev=0
fatal: No names found, cannot describe anything.
```

**Fix:**

For GitHub Actions, set `fetch-depth: 0` in the checkout step:

```yaml
- uses: actions/checkout@v4
  with:
    fetch-depth: 0   # Fetches full history including all tags
```

For a pipeline where you can't change the checkout depth, fetch tags explicitly after checkout:

```bash
$ git fetch --tags --unshallow
# or, if it's already a full clone:
$ git fetch --tags
```

**Prevention:** Any pipeline that uses `git describe`, GitVersion, or tag-based versioning **must** use `fetch-depth: 0`. Add a comment in the workflow YAML next to that setting explaining why, or the next person will "optimize" it back to a shallow clone.

---

### Lightweight tag used instead of annotated — `git show` shows only the commit, no tagger metadata

**Root cause:** `git tag v2.15.0` (without `-a`) creates a lightweight tag. GitHub Releases, `git describe`, and most changelog tools work correctly only with annotated tags. Lightweight tags don't carry tagger name, date, or message.

```bash
$ git show v2.15.0
commit f3c8a1eab2c7d9e04f15a836b29c1d73e8f042a1 (tag: v2.15.0, HEAD -> main)
Author: Alice Chen <alice.chen@acme.com>
Date:   Fri Jun 26 11:15:22 2026 +0000

    feat: add retry budget enforcement to idempotency layer
# ← No "tag object" header, no tagger, no release message
```

**Fix:**

```bash
# Delete the lightweight tag locally (and remotely if already pushed)
$ git tag -d v2.15.0
Deleted tag 'v2.15.0' (was f3c8a1e)

# Push delete if it was already on origin
$ git push origin --delete v2.15.0

# Recreate as annotated
$ git tag -a v2.15.0 f3c8a1e -m "Release v2.15.0"
$ git push origin v2.15.0
```

**Prevention:** Add a pre-push hook or CI gate that rejects lightweight tags with names matching your release pattern (e.g., `v*.*.*`). Shell one-liner to detect lightweight tags:

```bash
# In a pre-push hook or CI check:
TAG_TYPE=$(git cat-file -t v2.15.0)
if [ "$TAG_TYPE" != "tag" ]; then
  echo "ERROR: v2.15.0 is a lightweight tag (type: $TAG_TYPE). Use git tag -a."
  exit 1
fi
```

### "error: tag 'v2.15.0' not found." when trying to delete or checkout

**Root cause:** The tag exists on origin but hasn't been fetched to the local repo, or you mistyped the tag name.

```bash
$ git tag -d v2.15.0
error: tag 'v2.15.0' not found.

$ git checkout v2.15.0
error: pathspec 'v2.15.0' did not match any file(s) known to git
```

**Fix:**

```bash
# Fetch tags first
$ git fetch --tags

# Verify it exists now
$ git tag -l 'v2.15.*'
v2.15.0

# Now the checkout or delete works
$ git checkout v2.15.0
Note: switching to 'v2.15.0'.
You are in 'detached HEAD' state.
HEAD is now at f3c8a1e feat: add retry budget enforcement to idempotency layer
```

**Prevention:** Always run `git fetch --tags` before any tag-dependent operation in a fresh clone or CI environment.

---

### Pushed tags don't trigger the GitHub Actions release workflow

**Root cause:** The workflow trigger uses `tags:` with a pattern, but you pushed the tag in a separate `git push origin <tag>` after the commit was already on the remote. GitHub Actions fires the `push` event for the tag push — but only if the workflow file already exists at that tag's commit. If your workflow was added after the tagged commit, or the tag pattern doesn't match, no run fires.

```bash
# Tag pushed, but no Actions run triggered
$ git push origin v2.15.0
 * [new tag]         v2.15.0 -> v2.15.0
# Expected a release pipeline run — nothing appeared in Actions tab
```

**Fix:**

```bash
# Check your workflow trigger pattern matches
# In .github/workflows/release.yml:
on:
  push:
    tags:
      - 'v[0-9]+.[0-9]+.[0-9]+'   # ← must match your tag format exactly

# If the workflow file didn't exist at the tagged commit,
# delete the tag, add the workflow, commit, then retag:
$ git push origin --delete v2.15.0
$ git tag -d v2.15.0
# (add workflow file, commit it)
$ git tag -a v2.15.0 HEAD -m "Release v2.15.0"
$ git push origin main --follow-tags
```

**Prevention:** Use `--follow-tags` on the `git push origin main` call rather than a separate tag push. This pushes the commit and the reachable annotated tags in one atomic push, ensuring the tag is tied to the same push event as the commit that carries the workflow file.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# ~/.gitconfig — add under [alias]

# Create an annotated release tag and push it in one command
# Usage: git release v2.15.0 "Release message here"
release = "!f() { git tag -a \"$1\" -m \"${2:-Release $1}\" && git push origin \"$1\"; }; f"

# List tags sorted by version (semver-ish, using version sort)
tls = tag -l --sort=-version:refname

# Show the most recent tag on current branch
latest-tag = describe --tags --abbrev=0

# Delete a tag both locally and on origin
# Usage: git tagdel v2.15.0-bad
tagdel = "!f() { git tag -d \"$1\" && git push origin --delete \"$1\"; }; f"
```

```bash
# Shell aliases — ~/.bashrc or ~/.zshrc
alias gtl='git tag -l --sort=-version:refname | head -10'   # last 10 tags by version
alias gta='git tag -a'                                        # annotated tag shorthand
```

### Automation Opportunity

For release-driven teams, a release script that validates and tags in one step prevents the most common errors (wrong commit, lightweight tag, forgotten push):

```bash
#!/usr/bin/env bash
# scripts/release-tag.sh
# Usage: ./scripts/release-tag.sh v2.15.0 "Release notes here"
# Safe to re-run — fails fast if tag already exists.

set -euo pipefail

VERSION="${1:?Usage: release-tag.sh <version> <message>}"
MESSAGE="${2:?Usage: release-tag.sh <version> <message>}"

# 1. Validate semver format
if ! echo "$VERSION" | grep -qE '^v[0-9]+\.[0-9]+\.[0-9]+$'; then
  echo "ERROR: Version must match vX.Y.Z format. Got: $VERSION"
  exit 1
fi

# 2. Refuse to tag if working tree is dirty
if ! git diff --quiet || ! git diff --cached --quiet; then
  echo "ERROR: Working tree is dirty. Commit or stash changes before tagging."
  exit 1
fi

# 3. Refuse to tag if not on main
CURRENT_BRANCH=$(git branch --show-current)
if [ "$CURRENT_BRANCH" != "main" ]; then
  echo "ERROR: Must be on main to create a release tag. Currently on: $CURRENT_BRANCH"
  exit 1
fi

# 4. Fetch to check for collision
git fetch --tags
if git tag -l | grep -qx "$VERSION"; then
  echo "ERROR: Tag $VERSION already exists locally or on origin."
  exit 1
fi

# 5. Create annotated tag
git tag -a "$VERSION" -m "$MESSAGE"
echo "Created annotated tag: $VERSION"

# 6. Push commit + tag together
git push origin main --follow-tags
echo "Pushed: $VERSION is now on origin"

# 7. Confirm
git show "$VERSION" | head -8
```

### Keyboard / CLI Efficiency Notes

- `git push origin main --follow-tags` is the release-day push alias you should use instead of `git push` everywhere — it pushes the branch and any annotated tags reachable from it in one command, eliminating the separate `git push origin <tag>` step.
- `git tag -l --sort=-version:refname` sorts tags in descending semantic version order (`v3.2.3` before `v3.2.10`). The default `--sort=-creatordate` sorts by date, which breaks for backdated tags.
- `git describe --tags` in a dev build's startup log is the fastest way to surface "which release commit is running" in a non-production environment — wire it into your health endpoint during local dev.
- `git log v2.14.1..v2.15.0 --oneline` generates the commit list between two release tags — your changelog source of truth without any external tooling.

**Q: A teammate tagged `v2.15.0` on the wrong commit and already pushed it. How do you handle it?**

> **Average answer:** "Delete the tag and re-push it."

> **Great answer:** "The first thing I'd do is NOT silently overwrite it — I'd check whether anyone has already deployed from that tag or pulled it into their pipeline. If a container image is already built and tagged `v2.15.0`, we have an image-tagging problem to sort out separately. Assuming we catch it early, I'd post in the incident or release channel to let the team know the tag is being rewritten. Then: `git push origin --delete v2.15.0` to remove the remote, `git tag -d v2.15.0` locally, recreate the annotated tag at the correct commit SHA, verify with `git show v2.15.0`, and repush. The force-overwrite approach (`git push --force` on a tag) is an alternative but I prefer the explicit delete-and-recreate because it makes the history of the tag object itself clear. After the fix, I'd check if CI already triggered a release build from the bad tag and cancel or retract that run."

**Q: `fatal: No names found, cannot describe anything.` — what's wrong?**

> **Average answer:** "Fetch the tags."

> **Great answer:** "That error means `git describe` can't find any tag in the commit ancestry — almost always because the checkout is a shallow clone with `--depth 1`. With a depth-1 clone, Git has no commit history to walk back through, so it can never find the nearest tag. The fix in GitHub Actions is `fetch-depth: 0` on the checkout step — that fetches the full history including all tags. If I'm in a pipeline where I can't change the depth, I add `git fetch --tags --unshallow` as an explicit step right after checkout. I also add a comment next to that `fetch-depth: 0` setting in the YAML so the next engineer doesn't 'optimize' it back to a shallow clone and spend an afternoon debugging a silent versioning failure."

### War Story Prompt

A team shipped `v3.1.0` of `InventoryApi` using a lightweight tag — `git tag v3.1.0` instead of `git tag -a v3.1.0 -m "..."`. Everything deployed fine. Six weeks later, a new engineer set up a changelog automation tool that used `git log <prev_tag>..<current_tag>` and `git show <tag>` to generate release notes. The tool silently produced empty release metadata for every historical release because lightweight tags don't carry a tagger object or message — the tool's `git show` output didn't match the expected format. Two sprints of release notes had to be reconstructed manually from Jira. The fix was a one-line CI gate that ran `git cat-file -t <tag>` and failed the build if the result wasn't `tag` (the object type for annotated tags). The lesson: the difference between lightweight and annotated doesn't matter until it does — and when it does, it's always retroactive.

## Decision Framework

### When to Reach for This

- [ ] Marking a commit that will be deployed to staging or production — use annotated tag
- [ ] Publishing a GitHub Release — the tag is the anchor; annotated message becomes release notes body
- [ ] Any pipeline that derives a version string via `git describe` — needs annotated tags to function
- [ ] Pinning a container image tag to a Git tag — `image:v2.15.0` ↔ `git tag v2.15.0`
- [ ] Creating a hotfix branch from a specific past release — `git switch -c hotfix/X v2.14.1`
- [ ] Generating a changelog between two releases — `git log v2.14.1..v2.15.0 --oneline`
- [ ] Marking a "last known good" commit before a risky migration or infrastructure change

### When NOT To

- [ ] Bookmarking a local commit during development — use a branch name or `git stash` with a descriptive message; tags are global state once pushed
- [ ] Marking every PR merge — that produces tag noise that overwhelms `git describe`; use branches and CI run numbers for that
- [ ] Using a tag name that encodes mutable state (e.g., `latest`, `stable`) — tags don't move; use branches or container registry aliases for floating references

### Tool Comparison

||Annotated Tag (`git tag -a`)|Lightweight Tag (`git tag`)|
|---|---|---|
|Stores metadata|Yes — tagger, date, message, optional GPG signature|No — just a pointer to the commit|
|Works with `git describe`|Yes|Yes, but less reliably (no sort guarantee)|
|Works with GitHub Releases|Yes — message populates release notes|Yes, but no release body|
|Works with changelog tooling|Yes|Often no — tools expect tag objects|
|GPG signing|Yes (`git tag -s`)|No|
|When it's the right call|Every production release, every release candidate|Personal local bookmarks, ephemeral CI markers never used downstream|

## Self-Check

### Conceptual Questions

1. What is the fundamental difference between a Git branch and a Git tag at the object level?
2. You run `git describe --tags` on a commit that is 4 commits past the tag `v2.14.1`. What does the output look like?
3. What flag do you add to `git tag` to create an annotated tag instead of a lightweight one?
4. You see `fatal: No names found, cannot describe anything.` in a CI pipeline. What is the most likely cause and fix?
5. What does `git push origin --follow-tags` do differently from `git push origin main`?
6. What is the single most dangerous thing you can do wrong with tags in a shared repo, and why?
7. How does the tagging workflow connect to [[9.005 — Git Branch — Create, Switch, Delete]] in a hotfix scenario?

<details> <summary>Answers</summary>

1. A branch is a movable pointer — it advances to the latest commit every time you commit on that branch. A tag is a static pointer — once placed, it never moves regardless of future commits. At the filesystem level, both are files in `.git/refs/` (branches under `heads/`, tags under `tags/`). Annotated tags are full Git objects stored in the object store; lightweight tags are identical in structure to branches (just a file containing a SHA).
    
2. `v2.14.1-4-g9a1f0e2` — the format is `<nearest-tag>-<commits-since-tag>-g<short-sha>`. The `g` prefix stands for "git". This is the standard output of `git describe --tags` when HEAD is not exactly on a tag.
    
3. `-a` (or `--annotate`). Full command: `git tag -a v2.15.0 -m "Release message"`. Without `-a`, the tag is lightweight. You can also use `-s` (`--sign`) for a GPG-signed annotated tag, which implies `-a`.
    
4. Most likely cause: the pipeline uses a shallow clone (`--depth 1` or similar), which fetches only the latest commit without any history. `git describe` needs to walk back through ancestry to find a tag — with depth 1, there is no ancestry. Fix: set `fetch-depth: 0` in the GitHub Actions checkout step, or run `git fetch --tags --unshallow` after a shallow checkout.
    
5. `git push origin main` pushes the branch ref only. Tags are never pushed implicitly. `git push origin main --follow-tags` pushes the branch AND any annotated tags that are reachable from the branch tip and not yet on the remote — in one atomic push. It only pushes annotated tags (not lightweight tags), which is the desired behavior for release workflows.
    
6. Force-pushing a rewrite to a tag that has already been pulled and deployed by teammates or CI systems — `git push --force origin v2.15.0`. If any pipeline has already built an artifact from that tag (container image, NuGet package, deployment manifest), the tag now points to a different commit than the artifact, breaking the traceability chain. The safe approach is always: delete the old tag (remote + local), recreate, repush — and communicate before doing it.
    
7. In a hotfix scenario, you branch off a tag rather than off `main`: `git switch -c hotfix/PAY-999 v2.14.1`. The tag gives you a stable, named starting point that is guaranteed to match exactly what's running in production. Without the tag, you'd have to look up the production commit SHA from deployment logs or container image metadata — slower and more error-prone. After the hotfix is written and reviewed, it gets merged to `main` and cherry-picked to the release branch, then a new patch tag (`v2.14.2`) is created at that hotfix commit.
    

</details>

**Drill 2 — Fix the broken thing**

```bash
# A CI pipeline just failed with:
$ git describe --tags --abbrev=0
fatal: No names found, cannot describe anything.

# Context: this is a GitHub Actions job. The workflow checkout step is:
# - uses: actions/checkout@v4
#   (no additional parameters)
#
# There ARE tags on origin — running `git ls-remote --tags origin` from your
# local machine shows v2.14.1 and v2.15.0 exist.
```

Diagnose why `git describe` fails in CI but works locally, and provide the exact fix.

<details> <summary>Solution</summary>

**Diagnosis:** GitHub Actions defaults to a shallow clone (`--depth 1`) when no `fetch-depth` is specified. A depth-1 clone has exactly one commit and zero history — `git describe` cannot walk back to find any tag. Locally you have the full clone with all history, so `git describe` works fine. This is the most common cause of this error in CI.

```bash
# Proof: in the CI job, run this to confirm shallow clone
$ git rev-list HEAD --count
1
# Only 1 commit in history = definitely a shallow clone

# Fix: update the checkout step in the workflow YAML
```

```yaml
# .github/workflows/release.yml — before (broken)
- uses: actions/checkout@v4

# After (fixed)
- uses: actions/checkout@v4
  with:
    fetch-depth: 0   # Full history required for git describe to find tags
```

```bash
# If you cannot modify the checkout step, add this immediately after it:
- name: Fetch all tags
  run: git fetch --tags --unshallow

# Verify the fix works in the next run:
$ git describe --tags --abbrev=0
v2.15.0
```

Add a comment in the YAML:

```yaml
- uses: actions/checkout@v4
  with:
    fetch-depth: 0   # DO NOT change to depth 1 — git describe needs full tag history
```

</details>
