### Where This Fits

You sit down in the morning and your local `main` is behind `origin/main`. A teammate pushed a hotfix overnight. You need to get current before you branch off. Or you're mid-feature and your PR is blocked because `main` has moved — you need to decide whether to merge or rebase in the upstream changes. Both situations start with the same question: fetch or pull?

---

## What You're Doing and Why

`git fetch` downloads commits, branches, and tags from a remote and updates your remote-tracking references (e.g. `origin/main`) without touching your working tree or current branch at all. `git pull` does a fetch _and then immediately_ merges (or rebases, if configured) the fetched changes into your current branch. The distinction matters because `git pull` makes an irreversible local change — it either creates a merge commit or fast-forwards your branch — while `git fetch` just brings information and lets you decide what to do with it.

The professional default is `git fetch` first, inspect, then act. `git pull` is a shortcut that's fine on a clean, up-to-date branch but dangerous when your working tree is dirty or when you haven't looked at what's arriving.

### Mental Model

`git fetch` is checking your mailbox — you see what arrived, but you haven't opened or acted on anything yet. `git pull` is checking your mailbox _and_ immediately filing everything it contains directly into your desk without looking at it first. Most of the time that's fine. Occasionally it's not, and that's when the distinction saves you.

```
Before fetch:

  Local:           A---B---C  (main, HEAD)
  Remote-tracking: A---B---C  (origin/main)   ← same as local, stale

After git fetch origin:

  Local:           A---B---C  (main, HEAD)    ← unchanged
  Remote-tracking: A---B---C---D---E  (origin/main)  ← updated

After git merge origin/main (= what pull did automatically):

  Local:           A---B---C---D---E  (main, HEAD, fast-forward)
```

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git fetch origin`|Downloads all remote branches/tags; updates `origin/*` refs|Every morning; before branching; before any merge decision|
|`git fetch origin main`|Downloads only the `main` branch from origin|When you only care about one branch|
|`git fetch --all`|Fetches from all configured remotes|Multi-remote repos; fork-based workflows|
|`git fetch --prune`|Fetches and removes stale remote-tracking refs for deleted branches|After teammates delete merged branches|
|`git pull`|Fetch + merge into current branch (default)|Fast-forward update on a clean branch with no divergence|
|`git pull --rebase`|Fetch + rebase current branch onto the fetched tip|When you want a linear history; preferred over merge pull on feature branches|
|`git pull --ff-only`|Fetch + fast-forward only; aborts if a merge would be needed|Safe default — fails loudly if the branch has diverged|
|`git pull --no-rebase`|Explicitly forces merge behavior regardless of config|When config has `pull.rebase=true` but you need a merge this once|
|`git log origin/main..main`|Show commits on local `main` not yet on `origin/main`|"What have I got that remote doesn't?"|
|`git log main..origin/main`|Show commits on `origin/main` not yet on local `main`|"What did the team push that I haven't pulled?"|

## Step-by-Step Walkthrough

### The Scenario

You're on `feature/order-refunds` in the `OrderService` repo. Overnight a teammate merged a hotfix directly to `main`. You need to update your local `main`, inspect what changed, and then bring those changes into your feature branch — without accidentally creating a messy merge commit on top of your in-progress work.

### Full Walkthrough

```bash
$ git status
On branch feature/order-refunds
nothing to commit, working tree clean
```

Clean working tree before fetching — always confirm this first. Fetching with dirty state is safe, but pulling with dirty state can cause unnecessary conflicts.

```bash
$ git fetch origin
remote: Enumerating objects: 9, done.
remote: Counting objects: 100% (9/9), done.
remote: Compressing objects: 100% (5/5), done.
remote: Total 5 (delta 3), reused 0 (delta 0), pack-reused 0
Unpacking objects: 100% (5/5), done.
From github.com:intervision/OrderService
   4a3f9c2..d7e8b31  main       -> origin/main
```

The output tells you exactly what arrived: the `main` branch moved from commit `4a3f9c2` to `d7e8b31`. Your local `main` and your working tree are untouched. `origin/main` is now updated.

```bash
$ git log main..origin/main --oneline
d7e8b31 fix: prevent null ref in RefundHandler when order not found
c91f3a4 chore: bump Microsoft.EntityFrameworkCore to 8.0.4
```

Two commits arrived. Reading this before merging is the whole point of fetching first — you know what you're about to integrate.

```bash
$ git diff main...origin/main --stat
 src/OrderService/RefundHandler.cs          |  8 ++++++--
 src/OrderService/OrderService.csproj       |  2 +-
 2 files changed, 8 insertions(+), 3 deletions(-)
```

Two files. Scoped entirely to `RefundHandler.cs` and the project file — no surprises, no unrelated migrations or config changes that could conflict with your feature branch.

```bash
$ git checkout main
Switched to branch 'main'
Your branch is behind 'origin/main' by 2 commits, and can be fast-forwarded.
  (use "git pull" to update your local branch)
```

Git already knows local `main` is behind `origin/main` by exactly 2 commits — because the fetch updated `origin/main` and now the divergence is visible locally.

```bash
$ git merge --ff-only origin/main
Updating 4a3f9c2..d7e8b31
Fast-forward
 src/OrderService/RefundHandler.cs    | 8 ++++++--
 src/OrderService/OrderService.csproj | 2 +-
 2 files changed, 8 insertions(+), 3 deletions(-)
```

`--ff-only` ensures this either fast-forwards (no merge commit, clean linear history) or fails loudly if the branch has diverged. It did fast-forward — local `main` is now identical to `origin/main`.

```bash
$ git checkout feature/order-refunds
Switched to branch 'feature/order-refunds'

$ git rebase main
Successfully rebased and updated refs/heads/feature/order-refunds.
```

Rebase the feature branch onto the updated `main`. This replays your commits on top of the two hotfix commits — linear history, no merge bubble, clean PR diff. See [[9.027 — Rebase vs Merge — Choosing the Workflow]] for when this is the right call vs. merging.

### What Could Go Wrong Mid-Walkthrough

**If `git merge --ff-only origin/main` fails:**

```bash
$ git merge --ff-only origin/main
fatal: Not possible to fast-forward, aborting.
```

Your local `main` has diverged from `origin/main` — someone else pushed to `main` while you had local commits on it that haven't been pushed. This is the situation `--ff-only` is designed to catch before it becomes a problem.

```bash
# Option 1: you have local commits on main that should be rebased
$ git rebase origin/main

# Option 2: your local commits on main were mistakes — reset to remote
$ git reset --hard origin/main    # WARNING: destroys local commits on main
```

See [[9.046 — Resolving Diverged Branches Cleanly]] for the full decision tree.

**If `git rebase main` reports conflicts:**

```bash
$ git rebase main
Auto-merging src/OrderService/RefundHandler.cs
CONFLICT (content): Merge conflict in src/OrderService/RefundHandler.cs
error: could not apply 7b2c1d3... feat: add refund validation
hint: Resolve all conflicts manually, mark them as resolved with
hint: "git add/rm <conflicted_files>", then run "git rebase --continue".
```

See [[9.007 — Resolving Merge Conflicts — Manual Workflow]] for the resolution procedure. After resolving: `git add src/OrderService/RefundHandler.cs && git rebase --continue`.

## Gotchas and Troubleshooting

### "Your branch is behind 'origin/main' by N commits, and can be fast-forwarded" — but `git pull` created a merge commit

**Root cause:** `pull.rebase` is not set (or is set to `false`), so `git pull` defaults to `git merge`. Even a fast-forward-eligible update creates a merge commit if the merge strategy is not `--ff-only`.

```bash
$ git log --oneline --graph
*   c3d4e5f (HEAD -> main) Merge branch 'main' of github.com:intervision/OrderService
|\
| * d7e8b31 fix: prevent null ref in RefundHandler
|/
* 4a3f9c2 feat: add order export endpoint
```

**Fix for next time:**

```bash
$ git config --global pull.rebase true     # rebase instead of merge on pull
# OR
$ git config --global pull.ff only         # fail loudly if can't fast-forward
```

**Fix for the already-created merge commit (if not pushed):**

```bash
$ git reset --hard HEAD~1                  # undo the merge commit, keep remote changes
$ git merge --ff-only origin/main          # fast-forward cleanly
```

**Prevention:** Set `pull.rebase true` or `pull.ff only` globally on every machine. Never use bare `git pull` on a branch with uncommitted divergence.

---

### "`git fetch` shows nothing but `git log main..origin/main` shows commits I don't have"

**Root cause:** `origin/main` is stale — the last fetch was some time ago and you're looking at cached remote-tracking state, not the current remote. `git log main..origin/main` compares local `main` to the _last known_ state of `origin/main`, not the live remote.

```bash
$ git log main..origin/main --oneline
d7e8b31 fix: prevent null ref in RefundHandler   ← this is from a stale origin/main ref
$ git fetch origin
# (same commits now arrive for real)
```

**Fix:** There's nothing to fix — run `git fetch origin` to get current. The symptom reveals you have stale remote-tracking refs that looked like "new commits" but were actually from a previous fetch you hadn't yet merged.

**Prevention:** `git fetch origin` before reading `git log main..origin/main`. The log command is only meaningful immediately after a fetch.

---

### "`fatal: refusing to merge unrelated histories`" on `git pull`

**Root cause:** The local branch and the remote branch have no common ancestor — this happens when a repo was force-initialized locally (`git init` + first commit) and then a remote was added that has its own independent commit history.

```bash
$ git pull origin main
From github.com:intervision/OrderService
 * branch            main       -> FETCH_HEAD
fatal: refusing to merge unrelated histories
```

**Fix:**

```bash
$ git pull origin main --allow-unrelated-histories
# Then resolve any conflicts and commit
```

**Prevention:** This almost never happens in normal team workflows — it indicates a repo setup error. Clone the remote repo directly (`git clone`) rather than initializing locally and adding a remote afterward.

---

### "`git pull --rebase` left me in a detached HEAD with conflicts mid-rebase"

**Root cause:** A rebase conflict was hit during `git pull --rebase`. Git paused the rebase and left you in the middle of it.

```bash
$ git pull --rebase
Auto-merging src/OrderService/RefundHandler.cs
CONFLICT (content): Merge conflict in src/OrderService/RefundHandler.cs
error: could not apply 7b2c1d3... feat: add refund validation
hint: Resolve all conflicts manually, mark them as resolved with
hint: "git add/rm <conflicted_files>", then run "git rebase --continue".
hint: You can instead skip this commit: run "git rebase --skip".
hint: To abort and get back to the state before "git rebase", run "git rebase --abort".
```

**Fix:**

```bash
# Option A: resolve and continue
$ git status                                         # see which files conflict
$ code src/OrderService/RefundHandler.cs             # resolve markers
$ git add src/OrderService/RefundHandler.cs
$ git rebase --continue

# Option B: abort and return to pre-pull state
$ git rebase --abort
```

**Prevention:** Before `git pull --rebase`, run `git fetch origin` and `git log HEAD..origin/main --stat` to see what's coming and mentally flag likely conflict zones before they hit.

---

### "`git fetch --prune` deleted a remote-tracking ref I still needed"

**Root cause:** `--prune` removes remote-tracking refs for branches that no longer exist on the remote. If a teammate deleted a branch from GitHub that you were still working against locally, your `origin/their-branch` ref disappears.

```bash
$ git fetch --prune
From github.com:intervision/OrderService
 - [deleted]         (none)     -> origin/feature/payment-retry
```

**Fix:** Your _local_ branch `feature/payment-retry` is unaffected — pruning only removes remote-tracking refs (`origin/*`). The commits are still there.

```bash
$ git branch feature/payment-retry   # local branch still exists
$ git log feature/payment-retry      # commits still intact
```

**Prevention:** Understand that `--prune` only removes `origin/*` tracking refs, never local branches. Local branches survive pruning entirely.

---

### "`git pull` says 'Already up to date' but I can see new commits on GitHub"

**Root cause:** You ran `git pull` while on a branch that is tracking a _different_ remote branch than the one you were looking at on GitHub. Or you haven't fetched and `origin/main` is stale so Git has nothing new to pull from its local cache.

```bash
$ git branch -vv
* feature/order-refunds  4a3f9c2 [origin/feature/order-refunds] feat: add validation
  main                   4a3f9c2 [origin/main]
```

```bash
$ git pull
Already up to date.
# But GitHub's main clearly has new commits...
```

**Fix:**

```bash
$ git fetch origin             # update origin/main from the live remote
$ git log main..origin/main    # now the new commits appear
$ git checkout main && git merge --ff-only origin/main
```

**Prevention:** `git pull` on your feature branch only updates _that branch's_ tracking remote — it does not update `origin/main`. Always `git fetch origin` explicitly to refresh all remote-tracking refs.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# ~/.gitconfig — add under [alias]
[alias]
  f      = fetch origin --prune              # the only fetch you ever need
  fall   = fetch --all --prune              # for multi-remote repos
  up     = pull --ff-only                   # safe pull — aborts if diverged
  upr    = pull --rebase                    # pull with rebase (linear history)
  incoming = log main..origin/main --oneline --graph   # what's on remote not locally
  outgoing = log origin/main..main --oneline --graph   # what's local not on remote
```

Daily workflow as a one-liner with aliases:

```bash
$ git f && git incoming    # fetch + see what arrived, before touching anything
```

### Automation Opportunity

Shell function that does the full safe morning sync in one command — fetch, show incoming, fast-forward main, rebase current branch:

```bash
# ~/.zshrc or ~/.bashrc
gsync() {
  local current_branch
  current_branch=$(git rev-parse --abbrev-ref HEAD)

  echo "=== Fetching origin ==="
  git fetch origin --prune

  echo ""
  echo "=== Incoming on main ==="
  git log main..origin/main --oneline --graph

  echo ""
  echo "=== Updating main ==="
  git checkout main
  git merge --ff-only origin/main

  echo ""
  echo "=== Rebasing $current_branch onto main ==="
  git checkout "$current_branch"
  git rebase main

  echo ""
  echo "=== Done. $(git log --oneline -1) ==="
}
```

Usage:

```bash
$ gsync
=== Fetching origin ===
From github.com:intervision/OrderService
   4a3f9c2..d7e8b31  main -> origin/main

=== Incoming on main ===
* d7e8b31 fix: prevent null ref in RefundHandler
* c91f3a4 chore: bump EF Core to 8.0.4

=== Updating main ===
Updating 4a3f9c2..d7e8b31
Fast-forward
...

=== Rebasing feature/order-refunds onto main ===
Successfully rebased and updated refs/heads/feature/order-refunds.

=== Done. 7b2c1d3 feat: add refund validation ===
```

### Keyboard / CLI Efficiency Notes

- **`git fetch origin` is essentially free** — it's read-only, never destructive, and takes under a second on most repos. Run it compulsively. You can never over-fetch.
- **`git log A..B --oneline`** — the double-dot range is the fastest way to answer "what's new between these two points?" without looking at GitHub. Works with commit hashes, branch names, and `origin/` refs equally.
- **`ORIG_HEAD` after a pull** — after `git pull` executes a merge or fast-forward, Git sets `ORIG_HEAD` to where you were before. `git diff ORIG_HEAD` shows everything that just arrived in the pull — useful for quick review immediately after.
- **`git fetch origin main:main`** — fetches `origin/main` and updates your local `main` branch directly, even if you're on a different branch. Safe if local `main` is behind; fails if it has diverged.

### Spoken Answers

**Q: What is the exact difference between `git fetch` and `git pull`?**

> **Average answer:** "`git fetch` downloads changes and `git pull` downloads and merges them."

> **Great answer:** "`git fetch` updates your remote-tracking references — things like `origin/main` — without touching your working tree, your index, or any local branch. It's purely informational: it says 'here is the current state of the remote.' Your local `main` stays exactly where it was. `git pull` does a fetch first, then immediately merges or rebases the result into whatever branch you're currently on — that's the local-state-changing part. The practical consequence is that `git fetch` is always safe to run at any time, even with a dirty working tree. `git pull` is a shortcut that skips the review step — which is fine when you already know what's coming, but risky when you don't. My default is fetch + inspect + explicit merge or rebase, because the review step has saved me more than once."

---

**Q: Your `git pull` created a `Merge branch 'main' of github.com:...' commit on your feature branch. How did that happen?**

> **Average answer:** "It merges when there's divergence. You should use rebase instead."

> **Great answer:** "It happened because the local feature branch and `origin/main` had diverged — both had commits the other didn't — and `git pull` defaulted to merge behavior (either `pull.rebase` was false or not set). Git created a three-way merge commit to reconcile them. On a feature branch, that merge commit is noise — it shows up in the PR diff and makes the history harder to read. The fix going forward is `git config --global pull.rebase true`, which makes `git pull` rebase instead of merge. Or better, I'd replace bare `git pull` on feature branches entirely with the explicit sequence: `git fetch origin`, inspect with `git log`, then `git rebase origin/main`. If I'm already stuck with the merge commit and haven't pushed it yet, I can undo it with `git reset --hard HEAD~1` and then rebase cleanly."

---

**Q: Tell me about a time `git pull` caused you a problem that `git fetch` would have avoided.**

> **Average answer:** "I got merge conflicts from a pull once. It was annoying."

> **Great answer:** "We were mid-sprint on `InventoryApi` — three engineers on overlapping changes to `InventoryService.cs`. I ran `git pull` to grab a dependency update my teammate had just pushed. Because I hadn't checked what was coming, I didn't know the pull would also drag in a partial refactor they'd committed directly to `main` that morning — a class rename that touched the same file I was editing. The merge conflict wasn't large, but the wrong assumption — 'it's just a dependency bump' — meant I resolved it carelessly and introduced a subtle bug where the old class name was used in one path. It made it through review and broke a nightly job. If I'd run `git fetch origin` first and read `git log main..origin/main`, I'd have seen 'refactor: rename InventoryRepository' and approached the conflict with appropriate attention."

### War Story Prompt

A `PaymentGateway` engineer ran `git pull` to grab a production hotfix before demoing a new feature. They didn't notice that `git pull` also fast-forwarded their local `main` branch — but they were on a branch that was tracking `main` with two uncommitted local changes. Git pulled, merged, and left them with a dirty working tree they didn't expect. The demo started in 3 minutes. The fix — `git stash`, then `git pull --rebase`, then `git stash pop` — took 90 seconds. The lesson: run `git status` before `git pull` every single time, and set `pull.ff only` so you at least get a hard stop instead of a silent merge when something unexpected is about to happen.

## Decision Framework

### When to Reach for This

- [ ] Every morning before you start work — `git fetch origin --prune` first, always
- [ ] Before creating a new branch off `main` — confirm `main` is current with a fetch
- [ ] Before opening a PR — fetch and check `git log origin/main..HEAD` to make sure your branch is current
- [ ] After a teammate tells you they pushed something you need — fetch to get it, inspect before merging
- [ ] When `git status` says "Your branch is behind" — fetch was already done by a previous command; now use `--ff-only` merge or `--rebase` pull to update

### When NOT To

- [ ] Don't use bare `git pull` on a feature branch with local commits unless `pull.rebase true` is set — you'll get a noisy merge commit
- [ ] Don't use `git pull --rebase` if you've already pushed your local commits to a shared remote branch — rebasing rewrites history, which breaks other people's copies
- [ ] Don't use `--allow-unrelated-histories` as a normal workflow — if you're hitting that flag, the repo setup is wrong and needs fixing at the root

### Tool Comparison

||`git fetch`|`git pull`|`git pull --rebase`|`git pull --ff-only`|
|---|---|---|---|---|
|Touches working tree|Never|Yes (merge/FF)|Yes (rebase)|Yes (FF only)|
|Creates merge commit|Never|Maybe|Never|Never|
|Safe with dirty tree|Yes|Risky|Risky|Risky|
|History shape|N/A|Divergent (merge bubble)|Linear|Linear|
|When it's the right call|Default first step|Greenfield, no divergence|Feature branch updates|Protected branch updates|

5. What's the difference between `git fetch --prune` deleting `origin/feature/old-branch` and `git branch -d feature/old-branch`?
6. What flag do you add to `git pull` to guarantee it will never create a merge commit — and instead fail loudly if a merge would be needed?
7. How does this note connect to [[9.046 — Resolving Diverged Branches Cleanly]]? What scenario bridges them?

<details> <summary>Answers</summary>

1. After `git fetch origin`: your **remote-tracking refs** (e.g. `origin/main`, `origin/feature/*`) are updated to reflect the remote's current state. Your **local branches**, **working tree**, and **index** are completely unchanged.
    
2. Git updates **`feature/order-refunds`** — `git pull` always applies to your _current_ branch's tracking remote. It does not update `main` or any other branch.
    
3. `git config --global pull.rebase true`. After this, bare `git pull` always rebases instead of merging.
    
4. It shows commits that are on `origin/main` but not yet on local `main` — i.e. what arrived in the last fetch that you haven't merged yet. It is only meaningful **immediately after a `git fetch`**; if you run it on a stale `origin/main`, the output reflects the last fetch, not the live remote.
    
5. `git fetch --prune` deletes **remote-tracking refs** (`origin/feature/old-branch`) — the local pointers to what the remote had. Your **local branch** `feature/old-branch` is untouched. `git branch -d feature/old-branch` deletes the **local branch**. They clean up different things.
    
6. `git pull --ff-only`. If the branch has diverged and a merge would be needed, Git aborts with `fatal: Not possible to fast-forward, aborting.` — a loud, safe failure rather than a silent merge commit.
    
7. When `git pull --ff-only` (or `git merge --ff-only origin/main`) aborts with "Not possible to fast-forward," you've discovered a diverged branch. That's exactly the starting state of [[9.046 — Resolving Diverged Branches Cleanly]] — the fetch/pull workflow hands off to the divergence-resolution workflow when a clean fast-forward isn't possible.
    

</details>

```bash
# The merge commit has NOT been pushed yet.
# Fix the history so the branch is linear: your two commits on top of d7e8b31.
```

<details> <summary>Solution</summary>

**Diagnosis:** A merge commit was created by `git pull` (merge mode) when the branch diverged. Since it hasn't been pushed, history can be rewritten safely.

```bash
# Undo the merge commit, keeping the working tree at its current state
$ git reset --hard HEAD~1
HEAD is now at 4a3f9c2 feat: add webhook retry logic

# Now rebase onto the current main tip
$ git rebase main
Successfully rebased and updated refs/heads/feature/webhook-retry.

# Verify linear history
$ git log --oneline --graph
* 8f7e6d5 (HEAD -> feature/webhook-retry) feat: add webhook retry logic
* 7e6d5c4 feat: scaffold webhook service
* d7e8b31 (origin/main, main) fix: prevent null ref in ChargeProcessor
```

Set `pull.rebase true` globally so this doesn't happen again:

```bash
$ git config --global pull.rebase true
```

</details>

**Drill 3 — Under time pressure**

Production alert: `PaymentGateway` is returning 500s. Your on-call lead says a colleague just pushed a hotfix to `origin/main` 2 minutes ago. You're on `feature/refund-api`. You need to get that hotfix into your local environment and run it locally in under 60 seconds. Walk through exactly what you type — and explain why each command is safe to run fast.

<details> <summary>Solution</summary>

```bash
# 1. Fetch — safe, never destructive, gets the hotfix into origin/main (~3s)
$ git fetch origin main
From github.com:intervision/PaymentGateway
   a1b2c3d..f4e5d6c  main       -> origin/main

# 2. Confirm the hotfix is what you expect — one line, ~1s
$ git log main..origin/main --oneline
f4e5d6c fix: prevent null response crash in ChargeProcessor on timeout

# 3. Fast-forward local main to include the hotfix — safe because main is protected
#    (no local commits should ever be on main directly)
$ git fetch origin main:main
Updating a1b2c3d..f4e5d6c

# 4. Checkout main to run the hotfix locally
$ git checkout main
Switched to branch 'main'

# 5. Run the service locally against the hotfix
$ dotnet run --project src/PaymentGateway
# Verify the 500s are gone in local logs
```

Why each step is safe under pressure: `git fetch` is always read-only. `git log` is read-only. `git fetch origin main:main` only fast-forwards (fails safely if it can't). `git checkout main` is safe because working tree was clean. No rebase, no merge, no decisions required — the whole sequence is deterministic.

</details>
