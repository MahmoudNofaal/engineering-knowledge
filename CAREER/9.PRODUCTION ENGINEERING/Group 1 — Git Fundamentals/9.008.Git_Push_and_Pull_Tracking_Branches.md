### Where This Fits

You've committed your fix to `feature/payment-retry` and you need to get it onto the remote so your teammate can review it — or you need to pull their latest changes before you continue. These two commands (`git push` and `git pull`) are the daily handshake between your local repo and the team's shared remote. When they work, you don't think about them. When they don't — rejected push, diverged branch, wrong upstream — they block everything.

---

## What You're Doing and Why

`git push` uploads your local commits to a remote repository and advances the remote branch pointer to match yours. `git pull` does the opposite: it downloads remote commits and integrates them into your current branch (by default via a merge, or optionally via rebase). Both commands depend on **tracking relationships** — a local branch knowing which remote branch it corresponds to. Without a tracking relationship Git doesn't know where to push to or pull from, and it tells you so explicitly. Setting up that relationship correctly once means every subsequent push and pull on that branch is a one-word command.

For the team workflow policy that governs _when_ to push and _which_ branches to push to, see [[7.917 — Trunk-Based Development]].

### Mental Model

Think of your local branch and the remote branch as two clipboards. `git push` copies your clipboard to the server's clipboard — but only if the server's clipboard hasn't changed since you last synced (fast-forward). If someone else wrote to the server's clipboard in the meantime, Git refuses to overwrite their work and tells you to sync first. `git pull` is "fetch what's on the server's clipboard and paste it onto mine." The tracking relationship is the label on each clipboard saying which remote clipboard it corresponds to.

```
Local branch                   Remote branch (origin)
──────────────                 ──────────────────────
feature/payment-retry   ─────▶  origin/feature/payment-retry
       │                               │
  [your commits]              [shared commits]

git push  ──▶  copies local commits → remote
git pull  ──▶  fetches remote commits → merges into local
```

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git push`|Push current branch to its tracked remote branch|Daily — after committing on a branch with tracking set|
|`git push -u origin <branch>`|Push and set upstream tracking in one step|First push of a new local branch|
|`git push origin <branch>`|Push to a specific remote branch (no tracking set)|One-off push without changing tracking config|
|`git push --force-with-lease`|Force push but abort if remote changed since last fetch|After a rebase or amend on a personal branch|
|`git push origin --delete <branch>`|Delete a branch on the remote|After a PR is merged; cleaning up stale remote branches|
|`git pull`|Fetch + merge remote branch into current|Before starting work; after a teammate signals they've pushed|
|`git pull --rebase`|Fetch + rebase local commits on top of remote|Keeping a clean linear history instead of merge commits|
|`git pull origin <branch>`|Pull from a specific remote/branch explicitly|When tracking isn't set or you're pulling from a different branch|
|`git branch -vv`|Show all local branches with their upstream tracking info|Diagnosing "where does this branch push/pull?"|
|`git branch --set-upstream-to=origin/<branch>`|Set tracking on an existing branch|When you forgot `-u` on first push and now pull doesn't know where to go|

### What Could Go Wrong Mid-Walkthrough

**Scenario: Push is rejected because a teammate pushed to the same branch while you were working**

```bash
$ git push origin feature/stripe-idempotency-key
To github.com:company/PaymentGateway.git
 ! [rejected]        feature/stripe-idempotency-key -> feature/stripe-idempotency-key (non-fast-forward)
error: failed to push some refs to 'github.com:company/PaymentGateway.git'
hint: Updates were rejected because the remote contains work that you do not have locally.
hint: Integrate the remote changes (e.g. 'git pull ...') before pushing again.
hint: See the 'Note about fast-forwards' in 'git push --help' for details.
```

The remote has commits your local branch doesn't. You must integrate them first:

```bash
# Option A — pull with rebase (preferred for feature branches: clean linear history)
$ git pull --rebase origin feature/stripe-idempotency-key
From github.com:company/PaymentGateway
 * branch            feature/stripe-idempotency-key -> FETCH_HEAD
Successfully rebased and updated refs/heads/feature/stripe-idempotency-key.

$ git push origin feature/stripe-idempotency-key
To github.com:company/PaymentGateway.git
   c8d1a42..f3e9b21  feature/stripe-idempotency-key -> feature/stripe-idempotency-key

# Option B — pull with merge (creates a merge commit; less clean but no history rewrite)
$ git pull origin feature/stripe-idempotency-key
# Resolve any conflicts, then:
$ git push origin feature/stripe-idempotency-key
```

See [[9.046 — Resolving Diverged Branches Cleanly]] for the full decision tree on which integration strategy to use.

### Realistic Context

The `InventoryApi` team runs three feature branches in flight simultaneously: `feature/batch-restock`, `feature/supplier-webhook`, and `feature/restock-threshold-alerts`. A junior engineer created `feature/batch-restock` locally but forgot `-u` on their first push. Now `git pull` on that branch fails with "no tracking information." Meanwhile, after a rebase of `feature/supplier-webhook` to clean it up before PR review, the engineer needs to push with force — but must not clobber a teammate who might have pulled the old version. Both situations need resolving before the afternoon code review.

## Gotchas and Troubleshooting

### "error: failed to push some refs ... (non-fast-forward)"

**Root cause:** The remote branch has commits your local branch doesn't. Git refuses to push because doing so would silently overwrite the remote work — it's not a fast-forward.

```bash
$ git push origin feature/batch-restock
To github.com:company/InventoryApi.git
 ! [rejected]        feature/batch-restock -> feature/batch-restock (non-fast-forward)
error: failed to push some refs to 'github.com:company/InventoryApi.git'
hint: Updates were rejected because the remote contains work that you do not have locally.
```

**Fix:**

```bash
# Integrate the remote work first, then push:
$ git pull --rebase origin feature/batch-restock
# Resolve any rebase conflicts, then:
$ git push origin feature/batch-restock
```

**Prevention:** Always `git fetch` or `git pull` before starting a work session on a shared branch. On personal feature branches you own alone, `git pull --rebase` keeps history clean.

---

### "There is no tracking information for the current branch."

**Root cause:** The local branch was created without `-u` on first push (or created locally and never pushed), so Git doesn't know which remote branch to pull from.

```bash
$ git pull
There is no tracking information for the current branch.
Please specify which branch you want to merge with.
See git-pull(1) for details.

    git pull <remote> <branch>

If you wish to set tracking information for this branch you can do so with:

    git branch --set-upstream-to=origin/<branch> feature/batch-restock
```

**Fix:**

```bash
# If the remote branch already exists:
$ git branch --set-upstream-to=origin/feature/batch-restock feature/batch-restock
branch 'feature/batch-restock' set up to track 'origin/feature/batch-restock'.

# If the remote branch doesn't exist yet (first push of a new branch):
$ git push -u origin feature/batch-restock
```

**Prevention:** Always use `git push -u origin <branch>` on the first push of any new branch. Add `push.autoSetupRemote = true` to your global Git config to have Git do this automatically on every first push:

```bash
$ git config --global push.autoSetupRemote true
```

---

### "Your branch and 'origin/main' have diverged, and have X and Y different commits each."

**Root cause:** Both your local branch and the remote branch have new commits that the other doesn't. This happens when you commit locally while someone else also commits to the same remote branch.

```bash
$ git status
On branch main
Your branch and 'origin/main' have diverged,
and have 2 and 3 different commits each, respectively.
  (use "git pull" to merge the remote branch into yours)
```

**Fix:**

```bash
# Option A — rebase your local commits on top of the remote (linear history):
$ git pull --rebase
Successfully rebased and updated refs/heads/main.

# Option B — merge (creates a merge commit):
$ git pull
# If there are conflicts, resolve them, then:
$ git add <resolved-files>
$ git merge --continue
```

**Prevention:** On long-running branches, pull frequently. On short-lived feature branches, keep the branch lifespan short (hours to 1–2 days) so divergence rarely accumulates. See [[9.046 — Resolving Diverged Branches Cleanly]] for the full merge-vs-rebase decision.

---

### "fatal: 'origin/feature/batch-restock' is not a commit and thus cannot be pulled"

**Root cause:** The remote branch doesn't exist yet — you're trying to pull from a branch that was never pushed, or it was deleted on the remote.

```bash
$ git pull
fatal: 'origin/feature/batch-restock' is not a commit and thus cannot be pulled
```

**Fix:**

```bash
# Check what actually exists on the remote:
$ git fetch --prune
$ git branch -r
  origin/HEAD -> origin/main
  origin/main
  origin/feature/supplier-webhook
# feature/batch-restock is gone from the remote

# If the branch was deleted (e.g., after a merged PR) and you're done with it:
$ git branch -d feature/batch-restock

# If the branch should exist but doesn't (someone deleted it in error),
# re-push your local version:
$ git push -u origin feature/batch-restock
```

**Prevention:** After a PR is merged, delete the remote branch in GitHub and run `git fetch --prune` locally to update your remote-tracking refs. Running `git remote prune origin` periodically also cleans up stale remote-tracking branches.

---

### "error: src refspec main does not match any" on `git push`

**Root cause:** Your local branch is named differently from what you're trying to push to. Most common cause: the repo's default branch is `main` but you're on a branch called `master` (or vice versa), or you have a typo in the branch name.

```bash
$ git push origin main
error: src refspec main does not match any
error: failed to push some refs to 'github.com:company/PaymentGateway.git'
```

**Fix:**

```bash
# Check what branch you're actually on:
$ git branch --show-current
master

# Push your actual branch to the correct remote branch:
$ git push -u origin master:main
# or rename your local branch first:
$ git branch -m master main
$ git push -u origin main
```

**Prevention:** After cloning a repo, run `git branch --show-current` to confirm the local default branch name before making any commits. Configure `init.defaultBranch = main` globally to avoid the mismatch on new repos:

```bash
$ git config --global init.defaultBranch main
```

---

### "`git push --force` was rejected: remote contains work you do not have"

**Root cause:** Between your last fetch and your force-push attempt, someone else pushed to that branch. Plain `--force` would silently overwrite their commits; `--force-with-lease` detects this and aborts.

```bash
$ git push --force-with-lease origin feature/supplier-webhook
To github.com:company/InventoryApi.git
 ! [rejected]        feature/supplier-webhook -> feature/supplier-webhook (stale info)
error: failed to push some refs to 'github.com:company/InventoryApi.git'
hint: Updates were rejected because the remote contains work that you do not have locally. Suggest fetching first.
```

**Fix:**

```bash
# Fetch to see what they pushed:
$ git fetch origin feature/supplier-webhook
$ git log --oneline HEAD..origin/feature/supplier-webhook
c4a9f11 fix: handle duplicate webhook delivery

# Incorporate their commit, then re-evaluate whether force-push is still needed:
$ git rebase origin/feature/supplier-webhook
# If your rebase result still needs force-pushing:
$ git push --force-with-lease origin feature/supplier-webhook
```

**Prevention:** Never use `--force` (without `--lease`) on any branch another engineer might have pulled. `--force-with-lease` is the safe default for all forced pushes. See [[9.036 — Force Push — Risks and Safer Alternatives]] for when force-pushing is appropriate vs. when it's the wrong tool entirely.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# Add to ~/.gitconfig under [alias]
[alias]
  p    = push
  pu   = push -u origin HEAD         # push current branch + set upstream in one command
  pfl  = push --force-with-lease     # safe force push
  pl   = pull
  plr  = pull --rebase
  bvv  = branch -vv                  # show all branches with upstream tracking info
  gone = "!git fetch --prune && git branch -vv | grep ': gone]' | awk '{print $1}'"
  # 'git gone' lists local branches whose remote tracking branch has been deleted

# Shell aliases (add to ~/.bashrc or ~/.zshrc)
alias gp='git push'
alias gpu='git push -u origin HEAD'
alias gpl='git pull --rebase'
alias gbvv='git branch -vv'
```

Usage:

```bash
$ gpu                     # push new branch + set upstream, regardless of branch name
$ gpl                     # pull with rebase — keeps history linear
$ git gone                # see which local branches have stale/deleted remotes
```

### Automation Opportunity

A post-merge hook that automatically deletes stale local branches after a PR merge prevents the local branch list from growing unbounded:

```bash
#!/usr/bin/env bash
# scripts/prune-merged-branches.sh
# Run after a PR merge to clean up local branches whose remote is gone.
# Safe to run anytime — only deletes fully-merged branches.

set -euo pipefail

echo "Fetching and pruning remote refs..."
git fetch --prune

echo "Branches with deleted remotes:"
gone_branches=$(git branch -vv | grep ': gone]' | awk '{print $1}')

if [[ -z "$gone_branches" ]]; then
  echo "  None. All clean."
  exit 0
fi

echo "$gone_branches"
echo ""
read -rp "Delete these local branches? [y/N] " confirm
if [[ "$confirm" =~ ^[Yy]$ ]]; then
  echo "$gone_branches" | xargs git branch -d
  echo "Done."
else
  echo "Skipped."
fi
```

```bash
$ chmod +x scripts/prune-merged-branches.sh
$ ./scripts/prune-merged-branches.sh
Fetching and pruning remote refs...
Branches with deleted remotes:
  feature/batch-restock
  feature/supplier-webhook

Delete these local branches? [y/N] y
Deleted branch feature/batch-restock (was e2d9c14).
Deleted branch feature/supplier-webhook (was 9f2c3d1).
Done.
```

Add to your Makefile as a convenience target:

```makefile
.PHONY: prune
prune:
	@bash scripts/prune-merged-branches.sh
```

### Keyboard / CLI Efficiency Notes

- **`git push -u origin HEAD`** — `HEAD` always refers to the current branch name. Using `HEAD` instead of typing the branch name eliminates the most common push typo and works identically regardless of the branch you're on.
- **`git config --global push.autoSetupRemote true`** — after this one-time config change, every first push automatically sets upstream tracking. Eliminates the need to remember `-u` ever again.
- **`git config --global pull.rebase true`** — makes `git pull` always rebase instead of merge, giving you linear history without having to type `--rebase` every time.
- **`git branch -vv`** in your prompt or as a pre-push habit — shows `ahead N` (uncommitted pushes), `behind N` (unpulled remote commits), and `[gone]` (remote branch deleted). One command to understand the complete state of all your branches vs their remotes.
- **`git fetch --prune` vs `git remote prune origin`** — `--prune` on fetch removes stale remote-tracking refs in one step. Running it at the start of every work session keeps your remote branch list accurate.

### Spoken Answers

**Q: Walk me through what happens under the hood when you run `git push origin feature/payment-retry`.**

> **Average answer:** "It sends my commits to GitHub and updates the remote branch."

> **Great answer:** "Git first resolves which remote `origin` maps to — it checks `.git/config` for the remote URL. Then it compares the local branch tip (`HEAD` of `feature/payment-retry`) against the remote-tracking ref `origin/feature/payment-retry` stored in `.git/refs/remotes/origin/`. If the remote tip is an ancestor of the local tip — meaning it's a fast-forward — Git transfers only the delta objects the remote doesn't have yet, then updates the remote branch pointer to your local tip. If the remote tip is _not_ an ancestor — meaning someone else pushed in the meantime — Git rejects the push with 'non-fast-forward' rather than silently overwriting the remote history. The last thing it does locally is update `origin/feature/payment-retry` in your local remote-tracking refs to reflect what the remote now points to."

---

**Q: What's the difference between `--force` and `--force-with-lease`?**

> **Average answer:** "`--force-with-lease` is safer because it checks first."

> **Great answer:** "`--force` unconditionally overwrites whatever is on the remote branch — it doesn't check whether anyone else has pushed since you last fetched. `--force-with-lease` compares your local remote-tracking ref (`origin/feature/supplier-webhook`) against what's actually on the remote. If someone pushed after your last fetch, your local ref is stale — it doesn't match — and Git aborts with 'stale info' rather than silently deleting their commits. The practical rule is: `--force-with-lease` is always the right choice when you need to force-push; plain `--force` is only ever justified in automation scripts where you've just fetched and are certain no concurrent push is possible. On any branch another engineer might have pulled — even a personal feature branch during a PR review cycle — I'd always use `--force-with-lease` and always communicate in Slack before doing it."

---

**Q: A teammate's push is rejected with "non-fast-forward." Walk me through what they should do.**

> **Average answer:** "They should pull first, then push again."

> **Great answer:** "First, I'd tell them not to panic and definitely not to use `--force`. The rejection means the remote has commits their local branch doesn't. The fix depends on the branch type. If it's a personal feature branch they own, I'd have them run `git pull --rebase origin <branch>` — this takes their local commits, temporarily sets them aside, fast-forwards to the remote tip, then replays their commits on top. The result is a linear history with no merge commit, and then `git push` succeeds normally. If it's a shared branch like `develop` where a merge commit is acceptable, `git pull` (merge) is fine, then push. The one thing to stress: before pulling, they should run `git log --oneline -5` so they know exactly what commits they're working with and can verify the result afterward. `git log --oneline --graph -8` after the rebase lets them confirm their commits landed on top correctly."

### War Story Prompt

A senior engineer on the `NotificationWorker` team did a large rebase to clean up 12 commits before a code review — hours of work. They pushed with `git push --force` (not `--force-with-lease`) without checking first. A junior engineer had pulled the branch 20 minutes earlier to run the tests locally and had pushed one small fix to the same branch during that window. The force-push silently overwrote that fix. It wasn't discovered until the PR was merged and the fix was missing in staging. The recovery required `git reflog` on the junior engineer's laptop, cherry-picking the lost commit onto a hotfix branch, and re-merging. After that, the team added a branch protection rule on all non-main branches requiring `--force-with-lease` via a pre-receive hook, and established a Slack message convention before any force-push: "⚠️ force-pushing to `feature/X` in 2 min — don't pull."

## Decision Framework

### When to Reach for `git push`

- [ ] You've committed work locally and a teammate needs to see it (PR review, pairing)
- [ ] A CI pipeline needs to run against your branch
- [ ] You're creating a remote backup of local work before a risky operation
- [ ] A feature is complete and you're opening a PR

### When to Reach for `git pull`

- [ ] Starting a work session on any shared branch
- [ ] A teammate has signalled they've pushed changes you need
- [ ] Before branching off `main` for a new task
- [ ] Before merging or rebasing to make sure your base is current

### When NOT To

- [ ] Don't `git pull` (with merge) on a feature branch if you want linear history — use `git pull --rebase` instead
- [ ] Don't `git push --force` (without `--lease`) on any branch where someone else might have pulled
- [ ] Don't push directly to `main` or `develop` — use a PR unless the team has explicitly agreed to trunk-based commit access; see [[7.921 — Branch Protection Rules]]
- [ ] Don't `git push` before running tests locally — CI is not a substitute for a fast local sanity check

### Tool Comparison

||`git pull` (merge)|`git pull --rebase`|
|---|---|---|
|History shape|Creates a merge commit — diverged then converged|Linear — your commits move to the tip|
|Safety|Safe; preserves original commit timestamps|Safe on personal branches; avoid on shared branches (rewrites hashes)|
|When it's the right call|Shared branches (`develop`, `release/*`) where history of integration matters|Personal feature branches where a clean, reviewable history matters|

## Self-Check

### Conceptual Questions

1. What does `git push -u origin feature/payment-retry` do that `git push origin feature/payment-retry` doesn't?
2. You run `git branch -vv` and see `[origin/feature/batch-restock: ahead 2, behind 1]`. What does this mean, and what should you do before pushing?
3. What flag do you add to `git pull` to rebase your local commits on top of the remote instead of merging?
4. You see `fatal: 'origin/feature/old-branch' is not a commit and thus cannot be pulled`. What is the most likely cause?
5. What is the safer alternative to `git push --force`, and what does it check before overwriting the remote?
6. What should you _never_ do with `git push --force` on a branch that another engineer has already pulled?
7. How does `git fetch --prune` differ from `git fetch`, and why does it matter for daily branch hygiene? See [[9.009 — Git Fetch vs Pull — What Actually Happens]].

<details> <summary>Answers</summary>

1. `-u` (or `--set-upstream`) establishes a tracking relationship between the local branch `feature/payment-retry` and the remote branch `origin/feature/payment-retry`. After this, subsequent `git push` and `git pull` on that branch need no arguments — Git knows where to send and receive commits. Without `-u`, the push still works but no tracking is set.
    
2. `ahead 2` means your local branch has 2 commits the remote doesn't have yet (unpushed). `behind 1` means the remote has 1 commit your local branch doesn't have (unpulled). You must pull first — either `git pull --rebase` (to keep linear history) or `git pull` (merge) — before your push will be accepted. Pushing now will fail with "non-fast-forward."
    
3. `git pull --rebase` (or the short form `git pull -r`). You can also set this as the default for all pulls: `git config --global pull.rebase true`.
    
4. The remote-tracking branch `origin/feature/old-branch` no longer exists on the remote — most likely because the branch was deleted after a merged PR. Your local remote-tracking ref is stale. Fix: `git fetch --prune` to remove stale refs, then `git branch -d feature/old-branch` to delete the local branch if the work is merged.
    
5. `git push --force-with-lease`. Before overwriting the remote, it checks whether your local remote-tracking ref (`origin/<branch>`) matches the actual remote tip. If someone pushed between your last fetch and your force-push, the remote-tracking ref will be stale and Git aborts with "stale info" instead of silently overwriting the new commits.
    
6. Never force-push over another engineer's commits without coordinating first. Plain `--force` doesn't check — it overwrites silently. Their commits are not truly gone (they're in the remote's reflog for a window), but recovering them requires finding the lost SHA, `git cherry-pick`, and re-merging — all under stress. Use `--force-with-lease` and communicate before any force-push on a shared branch.
    
7. `git fetch` downloads new remote commits and updates remote-tracking refs but does not delete refs for remote branches that no longer exist. `git fetch --prune` additionally removes any local remote-tracking refs (e.g., `origin/feature/old-branch`) whose corresponding remote branch has been deleted. Without `--prune`, `git branch -r` accumulates stale entries indefinitely, making it harder to see what's actually live on the remote.
    

</details>

### Hands-On Drills

**Drill 1 — Do it from a clean state**

Starting from a fresh clone of the `OrderService` repo, create a new local branch `feature/discount-engine`, make one commit, and push it to `origin` with tracking set up. Then verify the tracking relationship is correctly established using `git branch -vv`.

**Drill 2 — Fix the broken thing**

```bash
$ git switch feature/batch-restock
$ git pull
There is no tracking information for the current branch.
Please specify which branch you want to merge with.
See git-pull(1) for details.

    git pull <remote> <branch>

If you wish to set tracking information for this branch you can do so with:

    git branch --set-upstream-to=origin/<branch> feature/batch-restock

$ git push
fatal: The current branch feature/batch-restock has no upstream branch.
To push the current branch and set the remote as upstream, use

    git push --set-upstream origin feature/batch-restock
```

The branch exists on the remote (a teammate pushed it) but no tracking is configured locally.

<details> <summary>Solution</summary>

**Diagnosis:** The local branch `feature/batch-restock` was created without `-u` (or was checked out locally without tracking being established). The remote branch exists — we just need to link them.

```bash
# Step 1 — confirm the remote branch actually exists
$ git fetch origin
$ git branch -r | grep batch-restock
  origin/feature/batch-restock
# It's there — safe to set tracking

# Step 2 — set the upstream tracking relationship
$ git branch --set-upstream-to=origin/feature/batch-restock feature/batch-restock
branch 'feature/batch-restock' set up to track 'origin/feature/batch-restock'.

# Step 3 — confirm, then pull
$ git branch -vv
* feature/batch-restock e2d9c14 [origin/feature/batch-restock] feat: add batch restock endpoint

$ git pull
Already up to date.

# Step 4 — prevent this in future: enable auto-setup for all new branches
$ git config --global push.autoSetupRemote true
```

</details>

**Drill 3 — Under time pressure**

A critical hotfix is merged to `main` on the remote. Your local `main` is 3 commits behind. You need to pull `main`, verify the hotfix commit is present, and immediately push a one-line config change you've already committed locally on `main` (your team has trunk-based direct commit access for hotfixes). You have 90 seconds before the deployment window closes.

<details> <summary>Solution</summary>

```bash
# 1 — Pull with rebase to land remote hotfix commits cleanly under your local commit
$ git pull --rebase
# Rebase replays your local commit on top of the 3 remote commits
# Output ends with: "Successfully rebased and updated refs/heads/main."

# 2 — Verify the hotfix is in history and your commit is on top
$ git log --oneline -5
f9c2d11 (HEAD -> main) config: increase payment timeout to 30s
e3a1b77 (origin/main) fix: handle null merchant ID in charge processor   ← hotfix
c9d4f22 fix: guard against empty order list in batch endpoint
b7e8a31 feat: add discount engine scaffold
d9e1b33 chore: bump version to 2.4.1
# Your commit (f9c2d11) is on top. Hotfix (e3a1b77) is directly below it. Correct.

# 3 — Push (fast-forward, will succeed immediately)
$ git push
Enumerating objects: 5, done.
...
To github.com:company/PaymentGateway.git
   e3a1b77..f9c2d11  main -> origin/main
# Deployment pipeline picks up f9c2d11 automatically
```

The key under pressure: `--rebase` not `--merge` — no merge commit to explain in the deployment log, and no chance of a "merge commit" landing in `main` history that confuses the rollback path.

</details>
