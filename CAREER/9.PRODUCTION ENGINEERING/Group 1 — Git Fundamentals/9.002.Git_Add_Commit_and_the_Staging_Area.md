### Where This Fits

You've just finished a chunk of work on `OrderService` — maybe a bug fix, maybe a feature slice — and you need to save a meaningful snapshot to the branch before pushing. The question is never "do I commit?" but "what exactly goes into this commit?" You sit down, run `git status`, see five changed files, and now you have to decide which changes belong together and how to describe them.

---

## What You're Doing and Why

`git add` moves changes from your working directory into the **staging area** (the index), and `git commit` turns whatever is in the staging area into a permanent, immutable snapshot in the repository's history. These two commands are intentionally separated so you can craft a commit from a subset of your changes — you may have touched six files but only three of them belong in this logical unit of work. The staging area is where you compose before you record.

If you need a deeper understanding of the commit graph and what "snapshot" means architecturally, see [[7.917 — Trunk-Based Development]] for how this fits into team workflow policy.

### Mental Model

Think of your working directory as a whiteboard — scribbles everywhere. The staging area is a photocopier tray: you pick up only the pages you want reproduced. `git commit` is pressing Print. The whiteboard doesn't change. The tray empties. The printed document is permanent.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git add <file>`|Stage a specific file|You want to commit one file out of many changed|
|`git add .`|Stage all changes in current directory tree|You want everything in scope|
|`git add -p`|Stage interactively, hunk by hunk|One file has two unrelated changes; you want to split them across commits|
|`git add -u`|Stage only modifications and deletions (no untracked)|You don't want to accidentally stage new scratch files|
|`git commit -m "message"`|Commit staged changes with an inline message|Short, clear message; no multi-line body needed|
|`git commit`|Open editor for message|Message requires a body/explanation|
|`git commit --amend`|Rewrite the most recent commit|Typo in message, or forgot one file — _before pushing only_|
|`git commit --amend --no-edit`|Add staged changes to last commit, keep message|Forgot to stage one small file|
|`git restore --staged <file>`|Unstage a file without discarding working directory changes|Staged the wrong file; pull it back|
|`git diff --staged`|Show what's staged vs last commit|Sanity-check before committing|

## Gotchas and Troubleshooting

### "nothing to commit, working tree clean" (but you just saved a file)

**Root cause:** The file you saved is listed in `.gitignore`, so Git is intentionally not tracking it.

```bash
$ git status
On branch main
nothing to commit, working tree clean
# You just saved appsettings.Development.json — which is in .gitignore
```

**Fix:**

```bash
$ git check-ignore -v appsettings.Development.json
.gitignore:14:appsettings*.Development.json    appsettings.Development.json
# Confirms the pattern that's suppressing it
```

If it legitimately needs tracking, remove the pattern from `.gitignore` (or add an exception `!appsettings.Development.json`) and then `git add` the file.

**Prevention:** Always run `git check-ignore -v <file>` before wondering if Git has a bug.

---

### "Changes not staged for commit" — staged file reappears after `git add`

**Root cause:** Line-ending normalization. `core.autocrlf` is converting CRLF → LF (or vice versa) on every read, making the file look perpetually dirty.

```bash
$ git add src/OrderService/Handlers/RefundHandler.cs
$ git status
Changes not staged for commit:
        modified:   src/OrderService/Handlers/RefundHandler.cs
# The file you just staged is showing as modified again immediately
```

**Fix:**

```bash
$ git config --global core.autocrlf input      # macOS/Linux
# or
$ git config --global core.autocrlf true       # Windows
$ git add --renormalize .
$ git commit -m "chore: normalize line endings"
```

**Prevention:** Add a `.gitattributes` file to the repo root that pins `* text=auto eol=lf` — see [[9.018 — .gitattributes — Line Endings and Diff Behavior]] for the full setup.

---

### `git commit --amend` after `git push` causes "rejected (non-fast-forward)"

**Root cause:** `--amend` rewrites the commit hash. If the original commit is already on the remote, the amended commit and the remote's commit have diverged histories.

```bash
$ git push origin feature/order-refunds
To github.com:company/OrderService.git
 ! [rejected]        feature/order-refunds -> feature/order-refunds (non-fast-forward)
error: failed to push some refs to 'github.com:company/OrderService.git'
hint: Updates were rejected because the tip of your current branch is behind its remote counterpart.
```

**Fix:** If you're the only person on this branch and you're certain nobody has pulled it:

```bash
$ git push --force-with-lease origin feature/order-refunds
# --force-with-lease refuses to overwrite if someone else pushed since your last fetch
```

Never `--force` (without `--lease`) on a shared branch. See [[9.036 — Force Push — Risks and Safer Alternatives]].

**Prevention:** Only amend commits that haven't been pushed yet. Make `--amend` a pre-push habit, not a post-push cleanup.

---

### Staged the wrong file entirely and already committed

**Root cause:** `git add .` or muscle memory — the wrong file ended up in a commit you didn't intend.

```bash
$ git log --oneline -1
c3e9a1b fix: update refund threshold
# You realize debug_notes.txt is in this commit
$ git show --stat c3e9a1b
 debug_notes.txt                         | 47 +++++++++++
 src/OrderService/Handlers/RefundHandler.cs | 12 ++-
 2 files changed, 59 insertions(+), 0 deletions(-)
```

**Fix (commit not yet pushed):**

```bash
$ git reset HEAD~1              # soft by default — returns files to staged
# or
$ git reset --mixed HEAD~1     # explicit — returns files to unstaged working directory
$ git restore --staged debug_notes.txt
$ git commit -m "fix: update refund threshold"
```

**Prevention:** `git diff --staged` before every commit. Make it a muscle-memory step like reading a code diff before merging a PR. See [[9.014 — Git Reset — Soft, Mixed, Hard]] for when to use which reset mode.

---

### "Author identity unknown" on first commit

**Root cause:** `user.email` and `user.name` are not configured globally or locally.

```bash
$ git commit -m "fix: initial handler fix"
Author identity unknown

*** Please tell me who you are.

Run

  git config --global user.email "you@example.com"
  git config --global user.name "Your Name"
```

**Fix:**

```bash
$ git config --global user.email "alex.chen@company.com"
$ git config --global user.name "Alex Chen"
$ git commit -m "fix: initial handler fix"
```

**Prevention:** Run `git config --global user.email` and `git config --global user.name` immediately after installing Git on any new machine. See [[9.019 — Git Config — Global, Local, and System Scope]] for managing multiple identities per repo.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# Add to ~/.gitconfig under [alias]
[alias]
  a    = add
  ap   = add -p
  au   = add -u
  c    = commit
  cm   = commit -m
  ca   = commit --amend
  can  = commit --amend --no-edit
  ds   = diff --staged
  st   = status -sb            # compact status: branch line + one-char codes per file

# Usage examples
$ git ap src/PaymentGateway/Processors/StripeProcessor.cs    # interactive patch
$ git cm "fix: null check on refund amount"                  # inline commit
$ git can                                                     # amend without touching message
$ git ds                                                      # diff the staging area
```

### Automation Opportunity

A pre-commit hook that runs `dotnet format --verify-no-changes` before every commit prevents style-only fixup commits from polluting history:

```bash
#!/usr/bin/env bash
# .git/hooks/pre-commit  (chmod +x this file)
# Aborts the commit if dotnet format would make changes — force clean code in.

set -euo pipefail

echo "Running dotnet format check..."
if ! dotnet format --verify-no-changes --no-restore 2>&1; then
  echo ""
  echo "❌ Commit blocked: dotnet format found issues."
  echo "   Run 'dotnet format' and stage the changes, then commit again."
  exit 1
fi

echo "✅ Format check passed."
```

Install for the repo:

```bash
$ cp scripts/pre-commit .git/hooks/pre-commit
$ chmod +x .git/hooks/pre-commit
```

Or with [Husky.NET](https://alirezanet.github.io/Husky.Net/) for team-wide enforcement — see [[9.021 — Git Hooks — pre-commit and pre-push Basics]].

### Keyboard / CLI Efficiency Notes

- **`git add -p` navigation keys:** `y` stage hunk, `n` skip, `s` split into smaller hunks, `e` manually edit hunk boundaries, `q` quit. `s` and `e` are the two most powerful and least-used.
- **`git commit` in editor:** Your `$GIT_EDITOR` / `$EDITOR` opens. First line = subject (keep under 72 chars). Blank line. Body starts line 3. `git log --oneline` only shows the subject.
- **`git status -sb`:** The `-s` short format + `-b` branch line is half the width of normal status — fits in a narrow terminal pane while you have a second pane open.
- **Shell completion:** Both bash and zsh completions for Git ship with most Git installs — enables Tab-completion on branch names, remote names, and flags. Worth confirming it's active: `type __git_wrap__git_main` (bash) or `type _git` (zsh) should return a function.

### Spoken Answers

**Q: Walk me through how you'd stage only part of a changed file.**

> **Average answer:** "I'd use `git add -p` and it shows you hunks you can accept or skip."

> **Great answer:** "I'd run `git add -p <filename>`. Git splits the file into hunks — contiguous changed regions — and prompts me for each one. I press `y` to stage a hunk, `n` to skip it. If a hunk is too large and contains two unrelated changes, I press `s` to split it into smaller hunks. In rare cases where `s` can't split fine enough, I press `e` to open the hunk in my editor and manually remove lines I don't want staged by prefixing them with a space instead of `+`. After staging, I always run `git diff --staged` to verify exactly what's going into the commit before I press Enter on `git commit`."

---

**Q: A colleague commits a debug file and then pushes — what are their options?**

> **Average answer:** "They can revert the commit or amend it."

> **Great answer:** "There are three realistic paths, each with a different risk profile. If they're alone on that branch, they can `git reset --mixed HEAD~1`, unstage the debug file with `git restore --staged`, recommit, and then `git push --force-with-lease`. The `--force-with-lease` is critical — it refuses to overwrite if anyone else pushed since they last fetched, preventing silent data loss. If the branch is shared, rewriting history is off the table — the cleanest option is a new commit that deletes the file: `git rm --cached debug_notes.txt && git commit -m 'chore: remove accidentally committed debug file'`. The third option — `git revert` — creates an explicit inverse commit, which is the safest for a main/trunk branch but leaves the original commit permanently visible in history. Which path you choose depends entirely on whether the branch is shared."

---

**Q: Your commit shows a file as both staged and unstaged simultaneously in `git status`. What happened?**

> **Average answer:** "That's weird, probably a Git bug."

> **Great answer:** "That's actually expected and it's not a bug — it means you staged the file, then made additional changes to it before committing. Git's staging area holds a snapshot of the file at the moment you ran `git add`. Any edits after that add appear as unstaged modifications to the same file. If you commit now, the commit will only contain the changes that were staged at `git add` time — the newer edits won't be included. You either run `git add <file>` again to re-stage including the new changes, or you use `git diff` to see what's unstaged and `git diff --staged` to see what's staged — they'll show different diffs for the same file."

### War Story Prompt

A service had been throwing a `NullReferenceException` in production for three days before anyone noticed — it was silently swallowed by a logging middleware change. Running `git bisect` revealed the offending commit. When the team opened it, the commit contained 14 files, an unrelated refactor, and two dependency version bumps — all squashed together under a message that read "various fixes." There was no way to revert just the middleware change without taking everything else with it. The fix was to manually cherry-pick the relevant lines into a hotfix branch. After that, the team added a PR template requiring a single logical change per PR and enforced it at review. The lesson wasn't "don't make mistakes" — it was "atomic commits are your future self's insurance policy."

## Decision Framework

### When to Reach for This

- [ ] You've finished a logical, self-contained unit of work (a fix, a feature slice, a refactor step)
- [ ] You want to checkpoint before switching tasks or branches
- [ ] You're about to do something risky (a big refactor) and want a safe rollback point
- [ ] Your changes span multiple files but all serve one purpose and belong together

### When NOT To

- [ ] You have mixed concerns in your working tree — use `git add -p` to split first; don't bundle unrelated changes into one commit
- [ ] You're mid-refactor with broken tests — don't commit red tests to a shared branch (commit to a local WIP branch or stash instead)
- [ ] You're committing to `main`/`trunk` without a CI gate in place — the commit belongs in a feature branch first

### Tool Comparison

||`git add <file>` / `git add .`|`git add -p`|
|---|---|---|
|Speed|Fast — whole files|Slower — interactive per-hunk|
|Precision|File granularity|Line-group granularity|
|When it's the right call|Changes in a file are coherent and fully intended|A single file contains two unrelated changes you want in separate commits|

## Self-Check

### Conceptual Questions

1. What is the staging area, and where does it sit in the workflow between your working directory and the repository?
2. You run `git add src/OrderService/Handlers/RefundHandler.cs`, then edit that file again. What does `git status` show? What does the commit capture?
3. What flag makes `git add` stage hunks interactively rather than whole files?
4. You see this error: `nothing to commit, working tree clean` — but you definitely saved a file. What's the most likely cause?
5. What's the difference between `git commit --amend` and `git commit --amend --no-edit`?
6. What should you _never_ do with `git commit --amend` once a commit is pushed to a shared branch, and why?
7. How does the staging area relate to `git stash` — does stash include staged changes? See [[9.016 — Git Stash — Save, Pop, Apply, Drop]].

<details> <summary>Answers</summary>

1. The staging area (index) is a buffer between the working directory and the repository. It holds a snapshot of what the next commit will look like. `git add` writes to it; `git commit` reads from it and clears it.
    
2. `git status` shows the file as both `Changes to be committed` (the staged snapshot) and `Changes not staged for commit` (the new edits after `git add`). The commit captures only the staged snapshot — the post-add edits are not included unless you run `git add` again.
    
3. `git add -p` (`--patch`). Inside the interactive session, `s` splits a hunk into smaller pieces, and `e` opens the hunk in an editor for manual line-by-line control.
    
4. The file is likely matched by a pattern in `.gitignore`. Confirm with `git check-ignore -v <filename>`.
    
5. Both rewrite the last commit. `--amend` opens the editor so you can change the commit message. `--amend --no-edit` keeps the existing message and just updates the commit's contents (e.g., to add a newly staged file).
    
6. Never amend a commit that's already been pushed to a branch others are using. Amend changes the commit hash, creating a diverged history. Anyone who has pulled the original will have a conflict, and `git push --force` would silently overwrite their work if they push before you.
    
7. Yes — by default `git stash` (or `git stash push`) saves both staged and unstaged changes, then resets both areas. The `--keep-index` flag preserves staged changes so only unstaged changes are stashed.
    

</details>

**Drill 3 — Under time pressure**

Production is down. You have a one-line hotfix ready in `src/PaymentGateway/Processors/StripeProcessor.cs`. A second unrelated change in that same file must NOT go out. You have 90 seconds to commit just the fix to `hotfix/stripe-null-charge` and push it. Walk through exactly what you type.

<details> <summary>Solution</summary>

```bash
# 1 — Stage only the hotfix hunk, not the entire file
$ git add -p src/PaymentGateway/Processors/StripeProcessor.cs
# Press 'y' for the null-check hunk, 'n' for everything else
# If the hunk is too large, press 's' to split; stage only the fix hunk

# 2 — Confirm only the right hunk is staged
$ git diff --staged
# Read it. One hunk. One null check. Nothing else.

# 3 — Commit with a clear hotfix message (reviewers will read this under pressure too)
$ git commit -m "fix: guard against null charge object in StripeProcessor

Stripe webhook can deliver a charge.updated event with a null Amount
field during dispute creation. Null reference was crashing the processor.
Fixes #P1-2026-06-26"

# 4 — Push to the hotfix branch (not main — CI gate runs first)
$ git push origin hotfix/stripe-null-charge
# Output shows the branch pushed; open the PR from here, not by typing
```

The key under pressure: `git add -p` first, `git diff --staged` second, _always_. Skipping the diff check is how the wrong change ships during an incident.

</details>
