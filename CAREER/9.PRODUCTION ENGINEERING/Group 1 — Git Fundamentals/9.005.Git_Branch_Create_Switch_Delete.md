### Where This Fits

You've been asked to pick up a Jira ticket for a new `OrderService` feature. Before you write a single line of code, you need a branch — a named pointer to your starting commit that keeps your work isolated from `main` until it's ready to merge. This is the muscle-memory operation every engineer repeats a dozen times a week: branch off, do the work, switch back, clean up. Get it wrong (working directly on `main`, or switching branches mid-edit with unrelated staged changes) and you'll be untangling history instead of shipping.

---

## What You're Doing and Why

A Git branch is just a movable pointer to a commit. Creating one costs almost nothing — it's a 41-byte file in `.git/refs/heads/`. The value is isolation: your commits accumulate on the branch without touching `main` or anyone else's work. When you switch branches, Git swaps your working tree to match the target branch's snapshot; when you delete a merged branch, you're only deleting the pointer, not the commits.

The modern CLI surface is `git switch` (introduced in Git 2.23) for changing branches and `git branch` for management. Older `git checkout` still works everywhere and is what you'll see in older docs, CI scripts, and colleagues' muscle memory — know both.

### Mental Model

Think of branches as sticky notes on a timeline. Every commit is a bead on a string. A branch is just a sticky note that says "I'm here" — it moves forward with each new commit you make on that branch. Switching branches moves which sticky note you're _standing on_. Deleting a branch pulls off the note; the beads (commits) stay unless nothing else points to them and Git GCs.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git branch feature/order-refunds`|Creates a new branch at current HEAD|You want a branch but stay on current one|
|`git switch feature/order-refunds`|Switches to an existing branch|Changing to a branch that already exists|
|`git switch -c feature/order-refunds`|Creates **and** switches in one step|Most common — new branch + immediately work on it|
|`git checkout -b feature/order-refunds`|Old equivalent of `switch -c`|Older Git, scripts, colleagues' aliases|
|`git branch`|Lists all local branches|Quick sanity check on what branches you have|
|`git branch -a`|Lists local **and** remote-tracking branches|Checking what exists on origin before creating|
|`git branch -d feature/order-refunds`|Deletes a **merged** branch (safe)|Post-merge cleanup|
|`git branch -D feature/order-refunds`|Force-deletes regardless of merge status|Abandoning work you don't want|
|`git branch -m old-name new-name`|Renames a branch|Typo in the branch name, or naming convention change|
|`git push origin --delete feature/order-refunds`|Deletes the remote branch|Post-PR merge cleanup on origin|
|`git branch --merged main`|Lists branches already merged into main|Identifying safe-to-delete candidates|
|`git branch --no-merged main`|Lists branches not yet merged|Checking what work is still in flight|

## Step-by-Step Walkthrough

### The Scenario

You're starting work on ticket `ORD-412` — adding refund validation to `OrderService`. The team's convention is `feature/<ticket>-<short-description>`. `main` is the integration branch. You need to branch off, verify you're in the right place, and know how to switch back if a hotfix interrupts you mid-task.

### Full Walkthrough

```bash
$ git status
On branch main
Your branch is up to date with 'origin/main'.

nothing to commit, working tree clean
```

Confirm you're starting from a clean state on `main` — never branch off uncommitted changes you don't intend to carry forward.

```bash
$ git pull origin main
From https://github.com/acme/OrderService
 * branch            main       -> FETCH_HEAD
Already up to date.
```

Pull latest before branching so your starting point is current, not yesterday's `main`.

```bash
$ git switch -c feature/ORD-412-refund-validation
Switched to a new branch 'feature/ORD-412-refund-validation'
```

`-c` creates the branch and immediately switches to it. One command, zero footguns.

```bash
$ git branch
* feature/ORD-412-refund-validation
  main
```

The `*` confirms you're on the new branch. `main` is still there — untouched.

```bash
$ git log --oneline -3
a1f9c3d (HEAD -> feature/ORD-412-refund-validation, origin/main, main) fix: correct tax rounding on international orders
7e2b1a0 feat: add order export endpoint
3c9d5f2 chore: bump EF Core to 8.0.4
```

`HEAD` points to the new branch, which points to the same commit as `main` — they're identical right now. That's expected.

```bash
# (... you write code, add commits ...)
$ git commit -m "feat: add refund eligibility validator"
[feature/ORD-412-refund-validation 58f0e1b] feat: add refund eligibility validator
 2 files changed, 47 insertions(+), 3 deletions(-)
```

Your branch has now diverged from `main` by one commit.

```bash
# Hotfix paged in — need to jump to main immediately
$ git status
On branch feature/ORD-412-refund-validation
nothing to commit, working tree clean
```

Working tree is clean, so switching is safe without stashing.

```bash
$ git switch main
Switched to branch 'main'
Your branch is up to date with 'origin/main'.
```

Back on `main`. Your feature work is preserved on `feature/ORD-412-refund-validation` — not lost, just not visible until you switch back.

```bash
# Hotfix done and merged. Return to feature work:
$ git switch feature/ORD-412-refund-validation
Switched to branch 'feature/ORD-412-refund-validation'
```

```bash
# Feature merged via PR. Post-merge cleanup:
$ git switch main
$ git pull origin main
$ git branch -d feature/ORD-412-refund-validation
Deleted branch feature/ORD-412-refund-validation (was 58f0e1b).
```

`-d` succeeds because Git confirms the branch's commits are reachable from `main`. The commit `58f0e1b` still exists in history — you only deleted the pointer.

```bash
$ git push origin --delete feature/ORD-412-refund-validation
To https://github.com/acme/OrderService.git
 - [deleted]         feature/ORD-412-refund-validation
```

Deletes the remote tracking branch too. GitHub often does this automatically on PR merge, but do it manually when it doesn't.

### What Could Go Wrong Mid-Walkthrough

**You have uncommitted changes when you try to switch:**

```bash
$ git switch main
error: Your local changes to the following files would be overwritten by checkout:
        src/OrderService/RefundValidator.cs
Please commit your changes or stash them before you switch branches.
Aborting
```

Git refuses to overwrite your work-in-progress. Two options:

Option A — stash it (see [[9.016 — Git Stash]]):

```bash
$ git stash push -m "WIP refund validator — switching for hotfix"
Saved working directory and index state On feature/ORD-412-refund-validation: WIP refund validator — switching for hotfix
$ git switch main
Switched to branch 'main'
# ... do hotfix work ...
$ git switch feature/ORD-412-refund-validation
$ git stash pop
```

Option B — commit the WIP (preferable if it won't break CI):

```bash
$ git commit -m "wip: refund validator in progress — do not merge"
$ git switch main
```

## Production Scenario

### Realistic Context

The `PaymentGateway` service has a P1 bug in production — a race condition in the idempotency check that causes duplicate charges on retry. While one engineer investigates on `main`'s history, you need to cut a hotfix branch from the last known-good release tag (`v2.14.1`), not from the current tip of `main` (which has two unreleased features). You then need to push the branch to origin so the team can review it and CI can run against it.

### Full Implementation / Resolution

```bash
# Confirm the release tag exists
$ git fetch --tags
$ git tag --list 'v2.14.*'
v2.14.0
v2.14.1

# Create hotfix branch FROM the tag, not from current main
$ git switch -c hotfix/duplicate-charge-idempotency v2.14.1

Fixes: PAY-988"
[hotfix/duplicate-charge-idempotency 9b3f2c1] fix: prevent duplicate charge on concurrent idempotency check

# Push to origin so CI runs and the team can review
$ git push -u origin hotfix/duplicate-charge-idempotency
```

```bash
e9d7b04 fix: correct gateway timeout on retries
```

The fix commit is on `main`, the branch pointer is gone, history is clean.

## Gotchas and Troubleshooting

### "fatal: A branch named 'feature/ORD-412-refund-validation' already exists."

**Root cause:** You (or a teammate via a shared push) already created this branch name in the local repo. `switch -c` fails if the branch exists.

```bash
$ git switch -c feature/ORD-412-refund-validation
fatal: A branch named 'feature/ORD-412-refund-validation' already exists.
```

**Fix:**

```bash
# If you want to go TO the existing branch:
$ git switch feature/ORD-412-refund-validation

# If it's stale and you want to recreate from current HEAD:
$ git branch -D feature/ORD-412-refund-validation
$ git switch -c feature/ORD-412-refund-validation
```

**Prevention:** Run `git branch | grep ORD-412` before creating. Add the ticket number to every branch name so collisions are obvious before you type the full create command.

---

### "error: The branch 'feature/ORD-412-refund-validation' is not fully merged."

**Root cause:** You used `git branch -d` (safe delete) on a branch whose commits aren't reachable from `HEAD` or `main`. Git is protecting you from silently discarding work.

```bash
$ git branch -d feature/ORD-412-refund-validation
error: The branch 'feature/ORD-412-refund-validation' is not fully merged.
If you are sure you want to delete it, run 'git branch -D feature/ORD-412-refund-validation'.
```

**Fix:**

```bash
# First, confirm you actually want to lose this work:
$ git log main..feature/ORD-412-refund-validation --oneline
58f0e1b feat: add refund eligibility validator
# There IS unmerged work here — do you really want to delete?

# If the PR was merged and you're confused, pull main first:
$ git pull origin main
$ git branch -d feature/ORD-412-refund-validation
Deleted branch feature/ORD-412-refund-validation (was 58f0e1b).

# If you genuinely want to abandon the work:
$ git branch -D feature/ORD-412-refund-validation
```

**Prevention:** Always pull `main` before running post-PR cleanup. The squash-merge case (common on GitHub) means the _commits_ won't match even though the _changes_ are in main — in that case `-D` is correct after confirming the PR merged.

---

### "Your local changes to the following files would be overwritten by checkout"

**Root cause:** You have unstaged or staged-but-uncommitted changes to files that differ between your current branch and the target branch. Git won't clobber your edits.

```bash
$ git switch main
error: Your local changes to the following files would be overwritten by checkout:
        src/OrderService/RefundValidator.cs
Please commit your changes or stash them before you switch branches.
Aborting
```

**Fix:**

```bash
# Option A: stash
$ git stash push -m "WIP refund validator"
$ git switch main
# later:
$ git switch feature/ORD-412-refund-validation
$ git stash pop

# Option B: commit (even a WIP commit)
$ git commit -am "wip: refund validator incomplete"
$ git switch main
```

**Prevention:** Run `git status` before switching. Treat uncommitted changes as a liability — commit or stash before any context switch. An alias like `alias gsw='git status && git switch'` makes this automatic.

---

### "git: 'switch' is not a git command"

**Root cause:** `git switch` was added in Git 2.23 (August 2019). Older systems — particularly some CI agents, Docker base images, and long-lived dev VMs — may still run 2.17–2.22.

```bash
$ git switch -c feature/new-work
git: 'switch' is not a git command. See 'git --help'.
```

**Fix:**

```bash
# Check your version:
$ git --version
git version 2.17.1

# Use the old syntax instead:
$ git checkout -b feature/new-work

# Or upgrade git (Ubuntu):
$ sudo apt-get update && sudo apt-get install --only-upgrade git
```

**Prevention:** Pin your CI base image's Git version explicitly and run `git --version` in your environment setup script. For local dev, keep Git current — `brew upgrade git` or `winget upgrade Git.Git`.

---

### Remote branch exists but `git switch` doesn't find it

**Root cause:** Remote-tracking references are stale. If a teammate pushed a branch after your last `fetch`, your local Git doesn't know it exists yet.

```bash
$ git switch feature/ORD-500-notifications
fatal: invalid reference: feature/ORD-500-notifications
```

**Fix:**

```bash
$ git fetch origin
From https://github.com/acme/OrderService
 * [new branch]      feature/ORD-500-notifications -> origin/feature/ORD-500-notifications

$ git switch feature/ORD-500-notifications
Branch 'feature/ORD-500-notifications' set up to track 'origin/feature/ORD-500-notifications'.
Switched to a new branch 'feature/ORD-500-notifications'
```

Git is smart enough to auto-create a local tracking branch when you switch to a name that matches exactly one remote branch.

**Prevention:** Run `git fetch origin` before switching to any branch you didn't create yourself. An alias like `alias gf='git fetch --prune'` keeps remote tracking refs fresh.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# ~/.gitconfig — add under [alias]

# Create and switch (replaces switch -c)
cob = switch -c

# Switch to previous branch (like `cd -` for directories)
sw- = switch -

# List branches sorted by last commit date — shows which are most recent
brl = branch --sort=-committerdate --format='%(committerdate:relative)%09%(refname:short)'

# Delete merged branches (local cleanup in one command)
# Usage: git bclean main
bclean = "!f() { git branch --merged ${1:-main} | grep -v \" ${1:-main}\" | xargs git branch -d; }; f"
```

```bash
# Shell aliases — ~/.bashrc or ~/.zshrc

alias gsw='git switch'
alias gswc='git switch -c'
alias gb='git branch'
alias gba='git branch -a'
alias gbD='git branch -D'
```

### Automation Opportunity

Post-PR cleanup is repetitive. This script deletes all local branches already merged to `main` and their corresponding remote branches:

```bash
#!/usr/bin/env bash
# scripts/git-cleanup-merged.sh
# Usage: ./scripts/git-cleanup-merged.sh
# Safe to re-run — only touches merged branches.

set -euo pipefail

BASE="${1:-main}"

echo "Fetching and pruning origin..."
git fetch origin --prune

echo "Pulling latest ${BASE}..."
git switch "${BASE}"
git pull origin "${BASE}"

echo "Deleting local branches merged into ${BASE}..."
git branch --merged "${BASE}" \
  | grep -v "^\*\|${BASE}" \
  | xargs -r git branch -d

echo "Done. Remaining branches:"
git branch -a
```

### Keyboard / CLI Efficiency Notes

- `git switch -` is the single most underused shortcut — switches back to the previous branch. Same muscle memory as `cd -` in the shell.
- `git switch -c feature/X origin/main` lets you branch off a specific remote ref without switching to it first. Avoids the pull step when you know origin/main is current.
- `git branch --merged main | grep -v main` produces a clean list for pipe-to-delete — build the `bclean` alias above once, use it forever.
- Tab completion for branch names works in Bash and Zsh with the Git completion scripts. If it's not working, install `bash-completion` or `zsh-completions` — branch name typos go to near-zero.

### Spoken Answers

**Q: Walk me through exactly how you'd start work on a new feature ticket.**

> **Average answer:** "I'd run `git checkout -b feature/my-feature` and start coding."

> **Great answer:** "First I'd make sure I'm on a clean state — `git status` to confirm nothing's staged. Then `git pull origin main` to make sure I'm branching off the latest, not yesterday's commit. Then `git switch -c feature/ORD-412-refund-validation` — I use `switch -c` over `checkout -b` because the intent is clearer, but I know both. I include the ticket number in the branch name so it links back to the Jira issue and shows up in PR titles automatically. After the first commit, I'd push with `-u origin` so the remote tracking branch is set up and CI starts running against it."

---

**Q: `git branch -d` gives you "not fully merged." Walk me through how you decide if it's safe to force-delete.**

> **Average answer:** "I'd use `git branch -D` to force delete it."

> **Great answer:** "I wouldn't reach for `-D` immediately. I'd first run `git log main..feature/branch --oneline` to see exactly which commits aren't in `main`. If the PR was squash-merged, those exact commits won't be there even though the changes are — so I'd cross-check the PR on GitHub to confirm it actually merged. If it did, `-D` is correct. If those commits genuinely aren't in `main` yet, I'd stop — that's a sign the PR isn't merged or someone's about to lose work. I've seen engineers force-delete a branch thinking cleanup was safe, only to find the PR had been closed without merging by mistake."

---

**Q: A teammate pushed a branch but `git switch` says it doesn't exist. What happened?**

> **Average answer:** "I'd try fetching first."

> **Great answer:** "Classic stale remote-tracking refs. My local Git doesn't automatically know about branches pushed after my last `fetch`. I'd run `git fetch origin`, which downloads the new remote ref, and then `git switch feature/their-branch` — Git is smart enough to detect that the branch name matches exactly one remote and will auto-create a local tracking branch. It's also worth setting `git config --global fetch.prune true` so deleted remote branches get cleaned up locally on every fetch, not just new ones."

### War Story Prompt

During a P1 incident, an engineer cut a hotfix branch with `git switch -c hotfix/payment-fix` — but forgot to first switch to `main` and pull latest. They were sitting on a three-day-old commit from a feature branch. The hotfix was written, deployed, and tagged — but it was missing a dependency update that had landed on `main` two days earlier. The fix "worked" in staging (which had the old build cached) but caused a `DllNotFoundException` in production thirty seconds after deploy. The real root cause wasn't the code — it was the branching point. `git log --oneline hotfix/payment-fix..origin/main` would have shown the gap immediately. The lesson: always confirm your starting commit before writing a single line of a hotfix.

## Decision Framework

### When to Reach for This

- [ ] Starting any new piece of work — feature, bugfix, hotfix, spike, chore
- [ ] Switching context between two in-progress tasks (stash or commit first, then switch)
- [ ] Creating a hotfix from a specific tag or known-good SHA rather than HEAD
- [ ] Post-PR cleanup — deleting the local and remote branch after merge
- [ ] Renaming a branch you mistyped or that no longer reflects the ticket

### When NOT To

- [ ] Committing directly on `main` for trivial one-liner fixes — most teams use branch protection that prevents this anyway; even when allowed, a branch creates a PR which creates a review record
- [ ] Creating a branch just to "save your work" when you don't intend to merge — use `git stash` instead (see [[9.016 — Git Stash]])
- [ ] Creating dozens of long-lived branches — this is a workflow smell pointing to incomplete work, not a branching problem to solve here (see [[7.917 — Trunk-Based Development]])

### Tool Comparison

||`git switch -c` (modern)|`git checkout -b` (classic)|
|---|---|---|
|Availability|Git 2.23+|All Git versions|
|Intent clarity|Explicit — switch only does branch switching|Overloaded — checkout also restores files|
|CI/scripting compatibility|Check agent Git version first|Safe everywhere|
|When it's the right call|Local development on any modern machine|Legacy CI agents, older teammates' scripts, pre-2.23 containers|

## Self-Check

### Conceptual Questions

1. What does `git branch` actually create at the filesystem level inside `.git/`?
2. You run `git switch -c feature/X` from commit `abc123`. What does `git log --oneline -1` show on the new branch immediately after creation?
3. What flag makes `git switch` create the branch instead of just switching to an existing one?
4. You see this: `error: Your local changes to the following files would be overwritten by checkout`. What does this mean and what are your two resolution options?
5. What's the difference between `git branch -d` and `git branch -D`?
6. After a squash-merge PR on GitHub, why does `git branch -d` often fail even though the feature is in `main`?
7. How does `git switch feature/X` (no `-c`) behave when `feature/X` doesn't exist locally but does exist on `origin`?

<details> <summary>Answers</summary>

1. A 41-byte file in `.git/refs/heads/<branch-name>` containing the full SHA of the commit the branch points to. That's the entire branch — one file with one hash.
    
2. It shows `abc123` — the same commit you were on. The new branch starts as an identical pointer to HEAD. No new commits are created by branching.
    
3. `-c` (or `--create`). Full form: `git switch -c <branch-name>`. The old equivalent in `checkout` is `-b`.
    
4. You have modifications to a file that differs between your current branch and the target branch, and Git won't overwrite your edits to perform the switch. Resolution A: `git stash push` to temporarily shelve the changes, switch, then `git stash pop` later. Resolution B: `git commit` the work-in-progress (even a WIP commit) before switching.
    
5. `-d` is a safety check — it refuses to delete if the branch contains commits not reachable from the current HEAD (i.e., unmerged work). `-D` is a force delete with no safety check. Use `-d` for post-PR cleanup. Use `-D` deliberately, after confirming with `git log main..branch` that you genuinely don't need those commits.
    
6. Squash merge collapses all your branch commits into a single new commit on `main`. That new commit has a _different SHA_ from any commit on your feature branch. So from Git's perspective, your branch commits are still "not reachable from main" — even though the changes are there. The fix is to pull `main` first and then use `-D`, or verify the PR merged on GitHub before force-deleting.
    
7. Git performs what's called "DWIM" (Do What I Mean): it checks remote-tracking refs, finds exactly one match (`origin/feature/X`), and automatically creates a local tracking branch. Output: `Branch 'feature/X' set up to track 'origin/feature/X'. Switched to a new branch 'feature/X'`. This only works when the name matches exactly one remote.
    

</details>

### Hands-On Drills

**Drill 1 — Do it from a clean state**

Starting from a fresh clone of `https://github.com/acme/OrderService` (substitute any real repo), create a branch named `feature/drill-001-branch-practice`, make one commit on it, switch back to `main`, verify the commit isn't visible on `main`, then delete the branch both locally and would-be-remotely.

**Drill 3 — Under time pressure**

Production is down. `PaymentGateway` is returning 500s. A colleague has identified the bad commit as `e3b1f29` on `main`. You need to cut a hotfix branch from the commit _before_ it (`e3b1f29~1`), in under 90 seconds, and push it to origin so CI can validate the revert. Go.

<details> <summary>Solution</summary>

```bash
# 1. Confirm the bad commit and its parent SHA (5 seconds)
$ git log --oneline -3
e3b1f29 (HEAD -> main, origin/main) feat: add dynamic pricing tier
a7c4e1b fix: correct VAT calculation on EU orders
9d2f3c0 chore: update appsettings for staging

# 2. Create hotfix branch from the parent of the bad commit (10 seconds)
$ git switch -c hotfix/revert-dynamic-pricing e3b1f29~1
Switched to a new branch 'hotfix/revert-dynamic-pricing'
# Safe: this targets the specific parent SHA, not a name that could move

# 3. Push immediately so CI starts and teammates can see the branch (15 seconds)
$ git push -u origin hotfix/revert-dynamic-pricing
Branch 'hotfix/revert-dynamic-pricing' set up to track 'origin/hotfix/revert-dynamic-pricing'.
# CI is now running against the pre-bad-commit codebase

# 4. Verify you're at the right commit (5 seconds)
$ git log --oneline -2
a7c4e1b (HEAD -> hotfix/revert-dynamic-pricing) fix: correct VAT calculation on EU orders
9d2f3c0 chore: update appsettings for staging
# ✓ e3b1f29 is gone from this branch's history

# Total: ~35 seconds. Open PR, ping #incident channel with branch URL.
```
