### Where This Fits

You run `git merge` (or pull, which is fetch-plus-merge) and instead of a clean fast-forward, Git stops and tells you it couldn't automatically combine two branches' changes to the same lines. Your terminal is now sitting mid-merge, the file has `<<<<<<<` markers in it, and the rest of the team is waiting on this PR. This is the single most common moment a working engineer freezes — not because the fix is hard, but because the markers look alarming the first dozen times you see them.

---

## What You're Doing and Why

A merge conflict means Git found two branches that changed the _same lines_ of the _same file_ in incompatible ways and can't pick a winner automatically — your job is to manually decide what the final content should be, then tell Git you've resolved it. This is purely mechanical once you know the markers: `<<<<<<< HEAD` through `=======` is "your side," `=======` through `>>>>>>> branch-name` is "their side," and the fix is editing the file to the version you actually want and removing the markers entirely. See [[7.917 — Trunk-Based Development]] for why frequent small merges keep conflicts like this rare and small.

### Mental Model

Think of a merge conflict as Git handing you a redline document and saying "I can't tell which edit wins here — you decide." It's not broken, and it's not dangerous — the markers are just text Git inserted into the file so you can see both versions side by side. Nothing is lost; both versions exist in the file right now, between the markers, waiting for you to pick, blend, or rewrite.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git status`|Shows which files are conflicted (`both modified`)|Immediately after a merge stops with conflicts|
|`git diff`|Shows conflict markers inline with context|Reading exactly what each side changed|
|`git checkout --ours <file>`|Discards conflict, keeps your branch's version entirely|One side is simply correct, no blending needed|
|`git checkout --theirs <file>`|Discards conflict, keeps the incoming branch's version entirely|The incoming change should fully win|
|`git add <file>`|Marks a conflicted file as resolved|After manually editing out the markers|
|`git merge --abort`|Cancels the merge, returns to pre-merge state|You realize mid-resolution this isn't the right time/approach|
|`git merge --continue`|Completes the merge commit after all conflicts are resolved|Final step once every conflicted file is staged|
|`git diff --name-only --diff-filter=U`|Lists only the files still in conflict|Large merges with many files — narrowing focus|
|`git log --merge`|Shows commits from both branches that touched the conflicted lines|Understanding _why_ each side made its change|

## Step-by-Step Walkthrough

### The Scenario

You're merging `main` into your branch `feature/order-refunds` on `OrderService` before opening a PR. Both branches modified `RefundHandler.cs` — your branch added validation logic, and `main` independently added logging to the same method in the time since you branched off.

### Full Walkthrough

```bash
$ git merge main
Auto-merging src/OrderService/RefundHandler.cs
CONFLICT (content): Merge conflict in src/OrderService/RefundHandler.cs
Automatic merge failed; fix conflicts and then commit the result.
```

_Git tried to combine both sides automatically and couldn't — the merge is now paused, not failed._

```bash
$ git status
On branch feature/order-refunds
You have unmerged paths.
  (fix conflicts and run "git commit")
  (use "git merge --abort" to abort the merge)

Unmerged paths:
  (use "git add <file>..." to mark resolution)
        both modified:   src/OrderService/RefundHandler.cs
```

_Confirms exactly one file is conflicted — `both modified` is the signal to open it next._

```bash
$ cat src/OrderService/RefundHandler.cs
public RefundResult ProcessRefund(RefundRequest request)
{
<<<<<<< HEAD
    if (request.Amount <= 0)
        throw new ArgumentException("Refund amount must be positive.");

    if (request.OrderId == Guid.Empty)
        throw new ArgumentException("Order ID is required.");
=======
    _logger.LogInformation("Processing refund for order {OrderId}", request.OrderId);
>>>>>>> main

    return _refundService.Execute(request);
}
```

_`HEAD` (your branch's validation logic) is above `=======`; `main`'s logging line is below it, up to `>>>>>>> main`._

```bash
# Edit the file by hand to combine both — neither side is wrong, they're complementary
$ cat src/OrderService/RefundHandler.cs
public RefundResult ProcessRefund(RefundRequest request)
{
    _logger.LogInformation("Processing refund for order {OrderId}", request.OrderId);

    if (request.Amount <= 0)
        throw new ArgumentException("Refund amount must be positive.");

    if (request.OrderId == Guid.Empty)
        throw new ArgumentException("Order ID is required.");

    return _refundService.Execute(request);
}
```

_All three marker lines (`<<<<<<<`, `=======`, `>>>>>>>`) are gone — this is the part people forget, leaving a marker behind silently breaks the build._

```bash
$ git add src/OrderService/RefundHandler.cs
$ git status
All conflicts fixed but you are still merging.
  (use "git commit" to conclude merge)

$ git commit
[feature/order-refunds 8b2f1a3] Merge branch 'main' into feature/order-refunds
```

_`git add` tells Git this file's conflict is resolved; `git commit` with no `-m` opens the pre-filled merge commit message — accept it as-is unless you have a reason to change it._

### What Could Go Wrong Mid-Walkthrough

If you open the file and realize the two changes are genuinely incompatible — not complementary — and you're not confident you understand both sides well enough to resolve it correctly right now, abort and regroup instead of guessing:

```bash
$ git merge --abort
$ git status
On branch feature/order-refunds
nothing to commit, working tree clean
```

This returns you to exactly the state before you ran `git merge main` — no harm done, no partial resolution left lying around. Use `git log --merge` first next time to see what each side's commit was actually trying to do before re-attempting.

## Production Scenario

### Realistic Context

`InventoryApi` has two long-running feature branches that both touched `StockReservationService.cs` — one team added a retry policy, another team independently refactored the same method's error handling. The PR merging both into `main` shows 3 conflicted files and a tight deadline before a release freeze. You need a workflow that doesn't just resolve markers blindly but actually verifies the merged result still works.

### Full Implementation / Resolution

```bash
$ git checkout main
$ git pull
$ git checkout feature/retry-policy
$ git merge main

Auto-merging src/InventoryApi/Tests/StockReservationServiceTests.cs

CONFLICT (content): Merge conflict in src/InventoryApi/Tests/StockReservationServiceTests.cs
```

```bash

src/InventoryApi/Tests/StockReservationServiceTests.cs
```

```csharp
// src/InventoryApi/StockReservationService.cs — after manual resolution
public async Task<ReservationResult> ReserveStockAsync(string sku, int quantity)
```

## Gotchas and Troubleshooting

### "error: Pulling is not possible because you have unmerged files."

**Root cause:** A previous merge was left half-resolved — conflict markers are still in a file or it's still staged as conflicted — and Git refuses to start a new pull on top of an unfinished one.

```bash
$ git pull
error: Pulling is not possible because you have unmerged files.
hint: Fix them up in the work tree, and then use 'git add/rm <file>'
hint: as appropriate to mark resolution and make a commit.
fatal: Exiting because of an unresolved conflict.
```

**Fix:**

```bash
$ git status
$ # resolve remaining conflicted files, then:
$ git add <file>
$ git commit
$ git pull
```

**Prevention:** Never walk away from a terminal mid-merge for "later" — finish or `git merge --abort` before switching context.

### Build succeeds locally but CI fails with a CS error pointing at conflict markers

**Root cause:** A `<<<<<<<`, `=======`, or `>>>>>>>` line was left in the file because `git add` was run before fully removing all markers, or a marker was missed in a large multi-hunk conflict.

```bash
$ dotnet build
error CS1525: Invalid expression term '<'
```

**Fix:**

```bash
$ grep -rn "<<<<<<<\|=======\|>>>>>>>" src/
src/OrderService/RefundHandler.cs:14:>>>>>>> main
$ # remove the stray marker line, then:
$ git add src/OrderService/RefundHandler.cs
$ git commit --amend --no-edit
```

**Prevention:** Always run a `grep` for marker patterns across the repo right before the final `git add`/`commit` on any multi-file conflict — don't rely on visual scanning alone.

### "CONFLICT (modify/delete): file.cs deleted in HEAD and modified in main."

**Root cause:** One branch deleted a file while the other branch kept editing it — Git can't auto-decide whether the file should still exist.

```bash
$ git merge main
CONFLICT (modify/delete): src/OrderService/LegacyRefundCalculator.cs deleted in HEAD and modified in main. Version main of src/OrderService/LegacyRefundCalculator.cs left in tree.
```

**Fix:**

```bash
$ # decide: should the file stay deleted, or keep main's edits?
$ git rm src/OrderService/LegacyRefundCalculator.cs   # confirm deletion
$ # OR
$ git add src/OrderService/LegacyRefundCalculator.cs  # keep main's version
$ git commit
```

**Prevention:** Before deleting a file on a long-lived branch, check `git log --all -- <file>` to see if another active branch is still actively modifying it.

### Resolved everything correctly but `git commit` opens an editor you don't know how to exit

**Root cause:** `git commit` with no `-m` after a merge drops you into your configured editor (often Vim by default) to confirm the auto-generated merge message, and Vim's modal interface is unfamiliar if you haven't used it.

```bash
$ git commit
# opens Vim showing "Merge branch 'main' into feature/order-refunds"
```

**Fix:** In Vim, press `Esc` then type `:wq` and press `Enter` to save and exit, accepting the message as-is. **Prevention:** Set a more familiar default editor once: `git config --global core.editor "code --wait"` (VS Code) or `git config --global core.editor "nano"`.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# ~/.gitconfig — conflict-resolution shortcuts
[alias]
    conflicts = diff --name-only --diff-filter=U
    mc = merge --continue
    ma = merge --abort
```

### Automation Opportunity

```bash
#!/usr/bin/env bash
# scripts/check-for-markers.sh — pre-commit safety net that catches leftover conflict markers
#!/usr/bin/env bash
set -euo pipefail

if git diff --cached | grep -qE '^\+(<{7}|={7}|>{7})'; then
    echo "ERROR: Staged changes contain unresolved conflict markers." >&2
    exit 1
fi
```

### Keyboard / CLI Efficiency Notes

- VS Code's built-in merge editor shows "Accept Current Change / Accept Incoming Change / Accept Both" buttons directly above each conflict block — far faster than hand-editing markers for simple one-sided resolutions.
- `git diff --diff-filter=U --name-only` is worth aliasing (`conflicts` above) since typing it from memory under pressure is error-prone.
- `git log --merge -p -- <file>` shows the actual diffs from both sides that caused the conflict, faster than reconstructing intent from the markers alone.

### Spoken Answers

**Q: Walk me through what you actually do, step by step, the moment `git merge` reports a conflict.**

> **Average answer:** "I open the file, find the markers, pick the right code, and remove the markers." Mechanically correct but skips verification and doesn't mention checking which files are actually conflicted first.

> **Great answer:** Starts with `git status` to see the full list of conflicted files rather than assuming it's just the one Git mentioned in the merge output. For each file, reads both sides of the marker block, and — critically — considers whether the right resolution is "pick one side" or "blend both," since complementary changes (like validation plus logging) need merging, not choosing. After editing, greps for leftover marker characters before staging, then runs the relevant tests before committing — not just trusting that the file compiles.

**Q: What's the difference between `git checkout --ours` and manually editing the conflict markers, and when is each the right call?**

> **Average answer:** "`--ours` keeps your version, `--theirs` keeps the other version." Correct but doesn't explain when blind acceptance is actually appropriate versus dangerous.

> **Great answer:** Explains that `--ours`/`--theirs` are correct when one side is unambiguously right for the _entire file_ — e.g., a generated lockfile where regenerating beats merging — but dangerous for source files where both sides likely made meaningful, complementary changes. In that more common case, manual marker editing is required because Git genuinely can't know the two changes were meant to coexist.

**Q: Tell me about a real merge conflict that was genuinely hard to resolve correctly, not just mechanically.**

> **Average answer:** "We had a big conflict and I just picked one side." No insight into the difficulty.

> **Great answer:** Describes a case where both sides looked plausible in isolation but combining them naively reintroduced a bug one side had specifically fixed — for example, one branch added a null check that the other branch's refactor accidentally bypassed. Explains using `git log --merge` to understand the _intent_ behind each change before resolving, not just the diff text, and running tests post-resolution specifically because the markers being gone didn't guarantee correctness.

### War Story Prompt

Two engineers on `PaymentGateway` both fixed the same race condition independently on separate branches, using different approaches. The merge conflict looked trivial — pick one side — but blending them naively left a partially-applied lock that caused an intermittent double-charge in staging two days later, traced back to a resolved conflict nobody had re-tested against the original race-condition repro steps.

## Decision Framework

### When to Reach for This

- [ ] `git merge` (or `git pull`) stops with `CONFLICT` in the output
- [ ] `git status` shows `both modified` or `both added` on one or more files
- [ ] You're combining two branches that touched overlapping code and need a human decision on the final result

### When NOT To

- [ ] You're mid-rebase, not mid-merge — the resolution mechanics are similar but the continue/abort commands differ; see [[9.028 — Resolving Rebase Conflicts Step-by-Step]]
- [ ] The conflict is purely in a generated/lock file you can safely regenerate — often faster to delete and regenerate than to hand-merge
- [ ] You don't understand either side's intent yet — pause and read `git log --merge` before editing anything

### Tool Comparison

||Manual marker resolution|`--ours` / `--theirs`|
|---|---|---|
|Speed|Slower — requires reading and judgment|Instant — no review needed|
|Safety/reversibility|Safer for source code with complementary changes|Risky if the discarded side had meaningful changes|
|When it's the right call|Both sides made real, possibly-overlapping changes|One side is fully authoritative (generated files, clear reverts)|

<details> <summary>Answers</summary>

1. `<<<<<<< HEAD` marks the start of your current branch's version; `=======` separates the two sides; `>>>>>>> branch-name` marks the end of the incoming branch's version.
2. `git add <file>` to mark the conflict resolved, then `git commit` to complete the merge commit.
3. `git diff --name-only --diff-filter=U`.
4. Finish resolving any remaining conflicted files with `git add`/`git commit` (or run `git merge --abort` to cancel entirely) before attempting `git pull` again.
5. Run `git log --merge` to see the actual commits from both sides that touched the conflicted lines, understanding intent before editing.
6. Don't guess and commit anyway — run `git merge --abort` to return to the pre-merge state and re-approach once you understand both sides.

Starting from a fresh clone of `OrderService` with two branches — `main` and `feature/discount-codes` — that both modified the same line of `PricingEngine.cs` in different ways, merge `main` into the feature branch and resolve the conflict by combining both changes.

**Drill 3 — Under time pressure**

A release is freezing in 10 minutes and merging `feature/inventory-sync` into `main` produces 2 conflicted files in `InventoryApi`. You don't have time to deeply reason through both — but you also can't ship a broken merge. Walk through exactly what you'd type to resolve fast but safely.

<details> <summary>Solution</summary>

```bash
$ git status
# both modified: lists exactly which 2 files — confirms scope before touching anything
$ git log --merge --oneline
# quick read of both sides' commit messages to understand intent in under a minute, faster than reading full diffs
$ # resolve each file by hand, keeping both sides' logic where they don't truly conflict
$ grep -rn "<<<<<<<\|=======\|>>>>>>>" src/
# zero output required before proceeding — non-negotiable even under time pressure, a missed marker breaks the build instantly
$ git add <file1> <file2>
$ dotnet build && dotnet test --filter "Category=Smoke"
# fastest meaningful verification — full suite would blow the time budget, smoke tests catch a bad blend
$ git commit
```

</details>
