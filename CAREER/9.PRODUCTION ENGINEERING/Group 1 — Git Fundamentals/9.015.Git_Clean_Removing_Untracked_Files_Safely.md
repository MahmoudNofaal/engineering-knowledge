Or a teammate's build script crashed mid-run and left generated artifacts scattered through the working tree, and `dotnet build` now fails with confusing stale-file errors. You want the working tree back to exactly what's tracked in Git — nothing more — without nuking files you haven't committed yet but actually care about.

## What You're Doing and Why

`git clean` deletes files that Git isn't tracking and that aren't covered by `.gitignore` (unless you explicitly ask it to include ignored files too) — it's the tool for "make my working tree match what a fresh clone would look like." Unlike `git reset` or `git checkout`, which operate on tracked files and are always recoverable through reflog or the index, `git clean` permanently deletes from disk with no undo — there's no Git mechanism that remembers a file it never tracked. See [[9.014 — Git Reset — Soft, Mixed, Hard]] for the tracked-file equivalent of "throw away changes."

### Mental Model

Think of your working tree as having three buckets: tracked-and-committed, tracked-and-modified, and untracked. `git reset`/`git checkout` clean up the first two buckets because Git has a record of them. `git clean` is the only command that touches the third bucket — and because Git never had a copy, deletion there is final. Always run `git clean -n` (dry run) first; it's the "are you sure" step Git doesn't ask for you.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git clean -n`|Dry run — lists what _would_ be deleted, deletes nothing|Always run this first, every time|
|`git clean -f`|Deletes untracked files (not directories, not ignored files)|Cleaning stray untracked files after a dry run looks correct|
|`git clean -fd`|Deletes untracked files _and_ untracked directories|Removing whole leftover folders like a stray `scripts/temp/`|
|`git clean -fx`|Deletes untracked files including ones matched by `.gitignore`|Nuking `bin/`/`obj/` build output along with everything else|
|`git clean -fX`|Deletes _only_ ignored files, leaves other untracked files alone|Clearing build artifacts without touching untracked source files|
|`git clean -fdx`|The "nuclear" combo — untracked files, directories, and ignored files|Resetting a working tree to exactly match a fresh clone|
|`git clean -i`|Interactive mode — choose what to delete file by file|Mixed working tree where some untracked files are precious|
|`git clean -n -d -x`|Dry run for the nuclear combo — see exactly what `-fdx` would wipe|Sanity check before the irreversible nuclear version|

## Step-by-Step Walkthrough

### The Scenario

You're on `feature/order-refunds` in `OrderService`. You switched away from a half-finished spike branch where you'd been experimenting with a new caching layer, and `git status` now shows a pile of untracked files: a stray `notes.txt` you actually want to keep, plus a full `bin/` and `obj/` directory from a build that ran on the old branch. You want the build artifacts gone but the notes file kept.

### Full Walkthrough

```bash
$ git status
On branch feature/order-refunds
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        bin/
        obj/
        notes.txt
        scripts/temp-cache-test/

nothing added to commit but untracked files present (use "git add" to track)
```

_Confirms exactly what's untracked before touching anything — never skip straight to deletion._

```bash
$ git clean -n -d
Would remove bin/
Would remove obj/
Would remove notes.txt
Would remove scripts/temp-cache-test/
```

_The dry run shows `notes.txt` would be caught too — this is the moment that saves you from losing it._

```bash
$ git clean -f -d -e notes.txt
Removing bin/
Removing obj/
Removing scripts/temp-cache-test/
```

_`-e` excludes a specific pattern from deletion — `notes.txt` survives, everything else untracked goes._

```bash
$ git status
On branch feature/order-refunds
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        notes.txt

nothing added to commit but untracked files present (use "git add" to track)
```

_Working tree is clean except for the one file you deliberately kept._

### What Could Go Wrong Mid-Walkthrough

If `bin/` and `obj/` are already listed in `.gitignore` (as they should be for any .NET project — see [[9.010 — .gitignore — Patterns and Common .NET Templates]]), plain `git clean -fd` won't touch them at all — ignored files are skipped by default:

```bash
$ git clean -fd
Removing scripts/temp-cache-test/
$ git status
Untracked files:
        bin/
        obj/
        notes.txt
```

Here you'd need `-x` to also remove ignored files: `git clean -fdx -e notes.txt`. Always dry-run with `-n` before adding `-x` — it deletes ignored files too, which sometimes includes intentionally-ignored local config you didn't mean to lose.

## Production Scenario

### Realistic Context

`InventoryApi`'s CI agent pool runs self-hosted runners that reuse the same disk between jobs for speed. A bad build left orphaned `obj/` cache files with a stale NuGet lock reference, and the next job intermittently fails with confusing restore errors that don't reproduce locally. The fix is to make the pipeline always start from a truly clean working tree — matching what a fresh clone would look like — before every build, not just rely on `dotnet clean`.

### Full Implementation / Resolution

```yaml
# .github/workflows/build.yml
name: build-and-test

on:
  push:
    branches: [main]
  pull_request:

jobs:
  build:
    runs-on: self-hosted
    steps:
      - uses: actions/checkout@v4

      - name: Reset working tree to clean state
        run: |
          git clean -fdx -e "*.user" -e ".vs/"
          git status --porcelain

      - name: Restore
        run: dotnet restore InventoryApi.sln

      - name: Build
        run: dotnet build InventoryApi.sln --configuration Release --no-restore

      - name: Test
        run: dotnet test InventoryApi.sln --configuration Release --no-build
```

```bash
$ git clean -fdx -e "*.user" -e ".vs/"
Removing bin/
Removing obj/
Removing InventoryApi.Tests/bin/
Removing InventoryApi.Tests/obj/
Removing artifacts/stale-restore-cache/
```

_Runs before every job on the self-hosted runner, leaving editor-specific files alone via `-e`, so the next job never inherits stale build output from the previous one._

### Verifying It Worked

```bash
$ git status --porcelain
$ echo $?
0
```

An empty `git status --porcelain` output with exit code `0` confirms the working tree exactly matches the checked-out commit — no leftover files from the prior job exist to interfere with the new build.

## Gotchas and Troubleshooting

### "fatal: clean.requireForce defaults to true and neither -i, -n, nor -f given; refusing to clean"

**Root cause:** Git's safety default refuses to delete anything unless you explicitly pass `-f` (force), `-n` (dry run), or `-i` (interactive) — this is intentional friction for a destructive command.

```bash
$ git clean -d
fatal: clean.requireForce defaults to true and neither -i, -n, nor -f given; refusing to clean
```

**Fix:**

```bash
$ git clean -n -d   # see what would happen first
$ git clean -f -d   # then actually do it
```

**Prevention:** Never set `clean.requireForce = false` globally just to skip typing `-f` — the friction is the safety net.

### Deleted a file you actually needed and `git status` no longer shows it anywhere

**Root cause:** The file was untracked, so Git never had a copy to fall back on — `git clean` deletion is not reflog-recoverable like a tracked-file reset.

```bash
$ git clean -fd
Removing notes.txt
$ git show HEAD:notes.txt
fatal: path 'notes.txt' does not exist in 'HEAD'
```

**Fix:** There usually isn't a Git-level fix. Check your editor's local history/backup feature, OS-level file recovery (Recycle Bin/Trash don't catch CLI deletions), or a recent IDE auto-save snapshot. **Prevention:** Always run `git clean -n` first, and use `-e <pattern>` to explicitly protect files you know you want to keep, or `git add` the file temporarily (even unstaged is enough to make it tracked) before cleaning.

### `git clean -fdx` removed a local secrets file you needed but never committed

**Root cause:** `.gitignore` correctly excluded the file from version control (e.g., `appsettings.Development.json` with local secrets), but `-x` tells `git clean` to remove ignored files too — which includes exactly the files `.gitignore` was protecting from commits, not from deletion.

```bash
$ git clean -fdx
Removing appsettings.Development.json
Removing bin/
Removing obj/
```

**Fix:** Restore from wherever the value originally came from — Key Vault, a teammate, or your password manager (see [[9.483 — Setting Up Per-Developer Local Secrets Safely]]). There is no Git-level recovery for an ignored file. **Prevention:** Use `-fdX` (capital X) instead, which deletes _only_ ignored build artifacts that match patterns like `bin/`/`obj/`, or always pass explicit `-e` excludes for known local-secret filenames before running `-x`.

### `git clean -fd` reports nothing to remove but `dotnet build` still fails with stale-reference errors

**Root cause:** The stale files are _tracked_, not untracked — they were committed accidentally at some point — so `git clean` correctly ignores them since it only ever touches untracked content.

```bash
$ git clean -n -d
$ # empty output — nothing untracked found
$ dotnet build
error MSB3030: Could not copy "obj/Debug/net8.0/OrderService.dll" because it was not found.
```

**Fix:**

```bash
$ git rm -r --cached bin/ obj/
$ echo -e "bin/\nobj/" >> .gitignore
$ git commit -m "chore: stop tracking build output"
```

**Prevention:** Set up `.gitignore` correctly before the first commit on a new repo — see [[9.010 — .gitignore — Patterns and Common .NET Templates]].

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# ~/.gitconfig — clean aliases that bake in the dry-run safety habit
[alias]
    cln = clean -n -d
    clf = clean -f -d
    clall = clean -fdx
```

### Automation Opportunity

```bash
#!/usr/bin/env bash
# scripts/reset-workspace.sh — full reset to a fresh-clone-equivalent state, with a confirmation gate
set -euo pipefail

echo "The following will be permanently deleted:"
git clean -n -d -x

read -p "Proceed? (y/N) " confirm
if [[ "$confirm" == "y" ]]; then
    git clean -f -d -x
    echo "Workspace reset."
else
    echo "Aborted — nothing deleted."
fi
```

### Keyboard / CLI Efficiency Notes

- `git clean -n` and `git clean -i` should become muscle memory before `-f` on any repo you don't personally fully trust the contents of.
- `-e <pattern>` can be passed multiple times in one command (`git clean -fd -e "*.local" -e "notes.txt"`) — no need to chain multiple invocations.
- `git clean -fdx -- path/to/subfolder` scopes the nuclear option to a single directory instead of the whole repo, useful when only one project's `bin/`/`obj/` is misbehaving.

### Spoken Answers

**Q: Walk me through safely removing all untracked build artifacts from a working tree without losing anything important.**

> **Average answer:** "Run `git clean -fd` to remove untracked files." Gets the core command right but skips the dry-run step and doesn't mention the risk of catching files you wanted to keep.

> **Great answer:** Starts with `git status` to see what's actually untracked, then `git clean -n -d` as a dry run to confirm exactly what would be removed before anything is deleted. Calls out that ignored files like `bin/`/`obj/` need `-x` to be included, and that `-e <pattern>` protects specific files from an otherwise-broad clean — treats the dry run as non-negotiable, not optional.

**Q: How is `git clean` different from `git reset --hard` in terms of what's recoverable afterward?**

> **Average answer:** "They both undo changes." Doesn't distinguish tracked vs untracked, or recoverability.

> **Great answer:** Explains that `git reset --hard` only ever touches _tracked_ files, and even then the previous state is still findable via `git reflog` for a window of time. `git clean` only touches _untracked_ files — files Git never had a copy of — so there is no reflog, no stash, no Git-level undo once they're deleted. Names this as the reason `git clean` defaults to refusing to run without `-f`/`-n`/`-i`.

**Q: Tell me about a time `git clean` (or a teammate running it) caused a real problem.**

> **Average answer:** "Someone ran clean and deleted some files we needed." Vague, no specifics.

> **Great answer:** Describes a specific incident — e.g., a CI agent running `git clean -fdx` wiped a local `appsettings.Development.json` that wasn't in source control, breaking a developer's local secrets setup, or a teammate running the nuclear combo on a repo with an untracked scratch file containing unsaved work. Names the fix (restoring from Key Vault, or recovering from IDE local history) and the prevention habit adopted afterward (explicit `-e` excludes, or switching CI to `-fdX` instead of `-fdx`).

## Decision Framework

### When to Reach for This

- [ ] `git status` shows a pile of untracked files/directories you want gone, not just hidden
- [ ] You need the working tree to match exactly what a fresh clone would produce (e.g., before CI builds, before a demo)
- [ ] Stale build artifacts (`bin/`, `obj/`) are causing confusing build errors that `dotnet clean` alone doesn't fully resolve

### When NOT To

- [ ] The files in question are tracked — use [[9.014 — Git Reset — Soft, Mixed, Hard]] or `git checkout -- <file>` instead, since `git clean` won't touch them anyway
- [ ] You're not fully sure what's untracked and haven't run `-n` yet — never jump straight to `-f`
- [ ] The "mess" includes uncommitted work you might still want — stash it with [[9.016 — Git Stash — Save, Pop, Apply, Drop]] instead of deleting it outright

### Tool Comparison

||`git clean -fd`|`git reset --hard`|
|---|---|---|
|Speed|Fast, single pass over untracked files|Fast, single pass over tracked files|
|Safety/reversibility|Permanent — no reflog, no undo|Recoverable via `git reflog` for a window of time|
|When it's the right call|Untracked clutter, build artifacts, ignored files|Tracked changes you want to discard back to a known commit|

## Self-Check

### Conceptual Questions

1. What category of files does `git clean` actually delete, and what does it leave alone by default?
2. You run `git clean -n -d` on a repo with `bin/`, `obj/`, and a stray `scratch.cs` all untracked, and `bin/`/`obj/` are in `.gitignore`. What does the output list?
3. Which flag makes `git clean` include files matched by `.gitignore`?
4. You see `fatal: clean.requireForce defaults to true` — what's the actual fix, and what should you run immediately before it?
5. What's the safer alternative to `-x` when you only want to remove ignored build artifacts, not all ignored files?
6. What should you NOT do the moment before running `git clean -fdx` on a repo you didn't set up yourself?
7. Which Domain 9 note covers stashing uncommitted work instead of deleting it outright?

<details> <summary>Answers</summary>

1. It deletes untracked files and directories that are _not_ matched by `.gitignore`. Ignored files are left alone unless `-x` (or `-X`) is explicitly passed.
2. Just `Would remove scratch.cs` — `bin/` and `obj/` are skipped because they're ignored and `-x`/`-X` wasn't passed.
3. `-x` (lowercase) includes ignored files alongside other untracked files; `-X` (uppercase) removes _only_ ignored files.
4. Run `git clean -n` (or `-i`) to see what would be deleted, then `git clean -f` to actually force it — there's no config-level "fix," the message is the intended safety gate.
5. `git clean -fdX` (capital X) — removes only files matched by `.gitignore`, leaving other untracked-but-not-ignored files alone.
6. Skip running `git clean -n -d -x` first — never run the nuclear combo without a dry run on a repo whose untracked/ignored contents you haven't personally verified.
7. [[9.016 — Git Stash — Save, Pop, Apply, Drop]].

### Hands-On Drills

**Drill 1 — Do it from a clean state**

Starting from a repo where `bin/`, `obj/`, and `.gitignore` already correctly ignore those two folders, plus an untracked file `draft-notes.md` you want to keep, remove only the build artifacts.

<details> <summary>Solution</summary>

```bash
$ git status
Untracked files:
        bin/
        obj/
        draft-notes.md

$ git clean -n -d -x
Would remove bin/
Would remove obj/

$ git clean -f -d -x
Removing bin/
Removing obj/

$ git status
Untracked files:
        draft-notes.md
```

_`-x` without touching `draft-notes.md` works here because that file was never matched by `.gitignore` in the first place — `bin/`/`obj/` were the only ignored entries._
