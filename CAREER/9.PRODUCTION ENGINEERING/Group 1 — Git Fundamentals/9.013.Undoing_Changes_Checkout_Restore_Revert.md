### Where This Fits

You just broke a test suite by editing the wrong file. Or your teammate asks you to revert a commit that's already been merged to `main`. Or you staged a file you didn't mean to. In every case, the question is the same: "how do I get back to a working state without making things worse?" These three commands — `git restore`, `git revert`, and (old-style) `git checkout` — are your undo toolkit. Picking the right one depends entirely on _where_ the change lives and _whether the history has been shared_.

---

## What You're Doing and Why

`git restore` discards working-directory changes or unstages files — it operates below the commit level and never touches history. `git revert` creates a new commit that is the inverse of a previous one — it's the only safe undo operation for commits that have already been pushed to a shared branch, because it adds to history rather than rewriting it. `git checkout -- <file>` is the old syntax for what `git restore` now does — you'll still see it in documentation and scripts, so you need to recognise it, but modern Git (2.23+) prefers `git restore` because the intent is explicit.

The critical dividing line: **working tree / staging area changes** → `git restore`. **Committed but not pushed** → `git reset` (see [[9.014 — Git Reset — Soft, Mixed, Hard]]). **Committed and pushed to a shared branch** → `git revert`. Crossing these lines in the wrong direction (e.g., `git reset` on a pushed commit) causes history divergence that affects everyone on the branch.

### Mental Model

Think of your repo as a stack of printed pages (commits) in a filing cabinet (the remote), with a notepad (staging area) and a whiteboard (working directory) on your desk. `git restore` erases what's on the whiteboard or wipes the notepad — the filing cabinet is untouched. `git revert` prints a new page that cancels out a previous page and files it on top — the cabinet grows but the old page stays in the record. `git reset` reaches into the cabinet and pulls pages back out — dangerous when others have photocopied those pages already. This note covers the first two. The third is [[9.014 — Git Reset — Soft, Mixed, Hard]].

```
WHERE IS THE CHANGE?          COMMAND TO USE
─────────────────────────────────────────────────────────
Working directory (unsaved)   git restore <file>
Staging area (git add done)   git restore --staged <file>
Staged + working              git restore --staged --worktree <file>
Committed, not pushed         git reset (→ see 9.014)
Committed and pushed          git revert <commit>
```

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git restore <file>`|Discard working-directory changes to a file|You edited a file and want the last-committed version back|
|`git restore .`|Discard ALL working-directory changes|Nuclear option — reset entire working tree to HEAD|
|`git restore --staged <file>`|Unstage a file (keep working-directory changes)|Accidentally staged the wrong file before committing|
|`git restore --staged --worktree <file>`|Unstage and discard working-directory changes|Staged and edited a file; want both gone|
|`git restore --source=<commit> <file>`|Restore a file to its state at a specific commit|You want one file as it was 3 commits ago|
|`git revert <commit>`|Create a new inverse commit|Undoing a pushed commit on a shared branch|
|`git revert <commit> --no-commit`|Stage the inverse changes without committing yet|You want to inspect or modify the revert before committing|
|`git revert <old>..<new>`|Revert a range of commits (newest first)|Undoing a series of merged commits without rewriting history|
|`git revert -m 1 <merge-commit>`|Revert a merge commit|Undoing a merged PR on a shared branch|
|`git checkout -- <file>`|Old syntax: discard working-directory changes|Legacy scripts; prefer `git restore` in new work|

### The Scenario

You're working on `OrderService`, branch `feature/invoice-export`. You've made three changes in your working session: (1) you modified `InvoiceExporter.cs` experimentally and now want to discard it, (2) you staged `PdfRenderer.cs` accidentally and want to unstage it without losing your edits, and (3) a commit from yesterday (`a3c9f12`) introduced a bug in `OrderController.cs` and was already pushed — you need to undo it safely on the shared branch.


_(No output on success — Git's way of saying "done." The file is now identical to the last committed version.)_

### What Could Go Wrong Mid-Walkthrough

**Scenario: `git revert` hits a conflict**

When the commits between the one you're reverting and `HEAD` touched the same lines, Git can't automatically invert the change:

```bash
$ git revert a3c9f12
Auto-merging src/OrderService/Controllers/OrderController.cs
CONFLICT (content): Merge conflict in src/OrderService/Controllers/OrderController.cs
error: could not revert a3c9f12... fix: update invoice total calculation
hint: After resolving the conflicts, mark them with
hint: "git add/rm <pathspec>", then run "git revert --continue".
hint: You can instead skip this commit with "git revert --skip".
hint: To abort and get back to the state before "git revert",
hint: run "git revert --abort".
```

Resolution:

```bash
# Open the file and resolve conflict markers manually:
$ code src/OrderService/Controllers/OrderController.cs

# After resolving:
$ git add src/OrderService/Controllers/OrderController.cs
$ git revert --continue
# Editor opens for the commit message — save and close
[feature/invoice-export 9c4d2a1] Revert "fix: update invoice total calculation"
 1 file changed, 2 deletions(-)
```

If the conflict is too complex to untangle safely under pressure, abort and reach for `--no-commit` to inspect manually:

```bash
$ git revert --abort
# Back to pre-revert state

$ git revert a3c9f12 --no-commit
# Conflict markers appear; resolve them, inspect with git diff --staged, then:
$ git commit -m "Revert \"fix: update invoice total calculation\" — manual conflict resolution"
```

## Gotchas and Troubleshooting

### "error: Your local changes to the following files would be overwritten by checkout"

**Root cause:** You tried to `git restore <file>` (or the old `git checkout -- <file>`) while the file had uncommitted changes that would be silently destroyed. Git refuses rather than overwrite without warning.

```bash
$ git restore src/OrderService/Exporters/InvoiceExporter.cs
error: Your local changes to the following files would be overwritten by checkout:
        src/OrderService/Exporters/InvoiceExporter.cs
Please commit your changes or stash them before you switch branches.
Aborting
```

**Root cause:** This error surfaces most often when switching branches with `git checkout <branch>`, not when using `git restore` directly — `git restore` by design discards without warning. If you see this, you're likely using `git checkout` for branch switching and have unsaved work.

**Fix:**

```bash
# Option A — stash the work, do what you need, pop it back
$ git stash push -m "WIP: invoice exporter experiment"
$ git checkout main
$ git stash pop

# Option B — if you genuinely want to discard, git restore does it without ceremony:
$ git restore src/OrderService/Exporters/InvoiceExporter.cs
# (no prompt — the change is gone immediately)
```

**Prevention:** Use `git status` before any `git checkout <branch>` call. If switching branches is frequent, `git stash` or committing a WIP commit before switching is the cleaner habit. See [[9.016 — Git Stash — Save, Pop, Apply, Drop]].

---

### `git restore <file>` silently discarded changes I wanted to keep

**Root cause:** `git restore` on the working directory has no confirmation prompt and no undo. Once run, the working-directory changes are gone — they were never committed, so there's no reflog entry for them.

```bash
$ git restore src/PaymentGateway/Processors/StripeProcessor.cs
# No output. Your 2 hours of work are gone.
# git reflog cannot help here — reflog tracks commits, not working-directory state.
```

**Fix:** If the file was open in your editor and the editor keeps a history (VS Code, Rider), check the local file history. In VS Code: right-click the file → "Timeline" → "Local History." This is your only recovery path.

**Prevention:** Before `git restore` on anything important, run `git diff <file>` to read what you're about to destroy. For experiments you're unsure about, `git stash push <file>` first — stash is reversible, `git restore` is not.

---

### "error: commit <sha> is a merge commit" when running `git revert`

**Root cause:** You tried to revert a merge commit without specifying which parent to treat as the "mainline." Git doesn't know which branch's perspective to revert from.

```bash
$ git revert e7b3c91
error: commit e7b3c91 is a merge commit.
hint: Use -m to specify the mainline. For example, 'git revert -m 1 e7b3c91'.
```

**Fix:**

```bash
# -m 1 means "keep parent 1" — i.e., revert the changes introduced by parent 2 (the merged branch)
# Parent 1 is almost always the branch you merged *into* (e.g., main)
$ git revert -m 1 e7b3c91
[main 3f9a2c1] Revert "Merge pull request #314 from feature/remove-dedup-check"
 1 file changed, 6 insertions(+)
```

To confirm which parent is which before running the revert:

```bash
$ git show e7b3c91 --no-patch
commit e7b3c91
Merge: f2d9c14 c7a3b19     ← parent 1 is f2d9c14 (main), parent 2 is c7a3b19 (feature branch)
Author: Alex Chen <alex.chen@company.com>
Date:   Fri Jun 26 14:22:11 2026 +0000

    Merge pull request #314 from feature/remove-dedup-check
```

Parent 1 (`f2d9c14`) is `main` — so `-m 1` reverts the merge from the perspective of `main`, effectively undoing everything the feature branch introduced.

**Prevention:** When reverting a merged PR, always use `-m 1`. Make it a checklist item in your incident runbook: "Is the target commit a merge commit? If yes: `git revert -m 1 <sha>`."

---

### After `git revert`, the reverted code reappears when the branch is re-merged

**Root cause:** This is a known Git behaviour. If you revert a merge commit on `main` and later re-merge the same feature branch (with the bug fixed), Git sees the commits as "already in history" and skips them — only the net-new commits after the revert land. The original bad code doesn't come back, but neither do the good parts of the original branch.

```bash
# Symptom: you reverted PR #314, fixed the bug on the feature branch, re-opened PR #315,
# merged it — and now some features from the original PR are missing in main.
```

**Fix:** After reverting a merge commit on `main`, you must revert the revert before re-merging the fixed feature branch:

```bash
# On the feature branch, revert the revert (restores the original commits):
$ git revert 3f9a2c1           # SHA of the revert commit on main
# Apply your actual fix commits on top
# Then merge/PR the feature branch back to main
```

This is counter-intuitive but correct. The revert of the revert re-exposes the original commits so Git's merge machinery includes them again.

**Prevention:** Document this pattern in your team's incident runbook. When a PR is reverted on `main`, the PR description of the fix branch should explicitly note: "Includes revert-of-revert of commit `3f9a2c1`."

---

### `git restore --staged` doesn't exist on older Git versions

**Root cause:** `git restore` was introduced in Git 2.23 (August 2019). On servers or CI agents pinned to older Git, the command isn't available.

```bash
$ git restore --staged src/OrderService/Controllers/OrderController.cs
git: 'restore' is not a git command. See 'git --version' and 'git help'.
```

**Fix:**

```bash
$ git --version
git version 2.17.1   ← too old for git restore

# Use the old syntax instead:
$ git reset HEAD src/OrderService/Controllers/OrderController.cs
# (equivalent to git restore --staged <file>)
```

**Prevention:** On CI agents, pin Git to 2.23+ in your runner base image. For local dev, enforce a minimum Git version in your `scripts/setup.sh` onboarding script:

```bash
required_git="2.23"
current_git=$(git --version | awk '{print $3}')
if [[ "$(printf '%s\n' "$required_git" "$current_git" | sort -V | head -n1)" != "$required_git" ]]; then
  echo "ERROR: Git $required_git or higher required. Found $current_git."
  exit 1
fi
```

---

### `git revert` on a range applies reverts in the wrong order

**Root cause:** When reverting a range like `git revert A..B`, Git reverts commits from oldest to newest by default. But to maintain a coherent history, you usually want to revert newest-first (so each revert applies cleanly on top of the previous one).

```bash
$ git revert 2c1d9f3..d9f2c11
# Reverts 2c1d9f3 first — but 2c1d9f3 is the oldest commit in the range.
# If later commits depend on it, each revert may conflict with the previous.
```

**Fix:**

```bash
# Use --no-commit with a manual loop, newest-first:
$ git revert --no-commit d9f2c11 a3c9f12 7b4e8a1 2c1d9f3
# Resolve any conflicts across the combined staging area
$ git commit -m "Revert commits d9f2c11..2c1d9f3 — rolling back invoice export feature"

# Or: use the range with the order flag explicitly reversed via git log:
$ for sha in $(git log --oneline d9f2c11..HEAD --format="%H" | head -4); do
    git revert --no-commit $sha
  done
$ git commit -m "Revert invoice export feature (4 commits)"
```

**Prevention:** For multi-commit reverts, always use `--no-commit` and revert in newest-first order to minimise conflicts. Test the combined staging area with `git diff --staged` before committing.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# Add to ~/.gitconfig under [alias]
[alias]
  # Discard working-directory changes to a file
  discard   = restore
  # Unstage a file (keep working-directory changes)
  unstage   = restore --staged
  # Discard staged + working directory (nuclear for a single file)
  nuke-file = restore --staged --worktree
  # Restore a file to a specific commit's version
  restore-at = "!f() { git restore --source=\"$1\" \"$2\"; }; f"

# Usage:
# git discard src/OrderService/Exporters/InvoiceExporter.cs
# git unstage src/OrderService/Renderers/PdfRenderer.cs
# git restore-at HEAD~3 src/PaymentGateway/Processors/StripeProcessor.cs
```

### Automation Opportunity

A `revert-pr` shell function that looks up the merge commit for a given PR number and runs the revert with correct `-m 1` automatically — eliminates the manual SHA lookup step during incidents:

```bash
#!/usr/bin/env bash
# scripts/revert-pr.sh
# Usage: ./scripts/revert-pr.sh 314 "INC-2026-06-26-001"
# Finds the merge commit for PR #314 and reverts it on the current branch.

set -euo pipefail

PR_NUMBER="${1:?Usage: revert-pr.sh <PR_NUMBER> <INCIDENT_ID>}"
INCIDENT_ID="${2:?Usage: revert-pr.sh <PR_NUMBER> <INCIDENT_ID>}"

echo "Looking up merge commit for PR #${PR_NUMBER}..."
MERGE_SHA=$(git log --oneline --merges | grep "Merge pull request #${PR_NUMBER}" | awk '{print $1}')

if [[ -z "$MERGE_SHA" ]]; then
  echo "ERROR: Could not find merge commit for PR #${PR_NUMBER} in current branch history."
  exit 1
fi

echo "Found merge commit: ${MERGE_SHA}"
echo "Reverting with -m 1..."

git revert -m 1 --no-commit "${MERGE_SHA}"

git commit -m "Revert \"Merge pull request #${PR_NUMBER}\"

Emergency revert for incident ${INCIDENT_ID}.
Original merge commit: ${MERGE_SHA}

See incident channel for context."

echo "Done. Revert committed. Push when ready."
```

```bash
$ chmod +x scripts/revert-pr.sh
$ ./scripts/revert-pr.sh 314 INC-2026-06-26-001
Looking up merge commit for PR #314...
Found merge commit: e7b3c91
Reverting with -m 1...
Done. Revert committed. Push when ready.
```

### Keyboard / CLI Efficiency Notes

- **`git restore .`** — restores the entire working tree to HEAD in one command. Fastest nuclear option when an experiment has gone sideways and you want a clean slate. No prompt — run it only when you mean it.
- **`git diff <file>` before `git restore <file>`** — makes this a two-keystroke sequence in your shell history (Up arrow twice). The habit saves you from accidentally discarding intended work.
- **`git revert HEAD`** — shorthand for "revert the most recent commit." Useful when you committed something wrong and haven't pushed yet, though `git reset HEAD~1` (see [[9.014 — Git Reset — Soft, Mixed, Hard]]) is usually cleaner for unpushed commits.
- **`git revert --no-commit`** — stage the inverse changes without committing, giving you time to inspect with `git diff --staged` before the commit is permanent. Under pressure, this extra step is worth it.
- **Shell history search:** `Ctrl+R` then type `revert` brings back your last revert command including the SHA — faster than re-looking up the commit log.

### Spoken Answers

**Q: Walk me through the difference between `git restore`, `git revert`, and `git reset`.**

> **Average answer:** "`git restore` undoes file changes, `git revert` undoes a commit, `git reset` moves HEAD back."

> **Great answer:** "The key axis is: has the change been committed, and has it been shared? `git restore` operates below the commit level — it discards working-directory changes or unstages files without touching history at all. It's the right tool when you've edited a file and changed your mind before committing. `git revert` creates a new commit that is the logical inverse of a previous one — it's additive, so it's safe on any branch that others have pulled because history isn't rewritten. I reach for it any time I need to undo something that's already been pushed. `git reset` moves the branch pointer backwards — it's powerful but it rewrites history, which makes it unsafe on shared branches. I use `git reset --soft` or `--mixed` only for local, unpushed commits where I want to restructure what I'm about to push. The rule I follow: if the commit is already in `origin`, revert it. If it's only local, reset it."

---

**Q: A commit was pushed to `main` two hours ago and is causing a production incident. Walk me through exactly what you type.**

> **Average answer:** "I'd run `git revert <hash>` and push."

> **Great answer:** "First, I pull `main` to make sure I'm at the latest: `git switch main && git pull`. Then I find the exact SHA: `git log --oneline -10` — I look for the commit by message and timestamp, confirm it with `git show <sha> --stat` to see which files it touched. I check whether it's a regular commit or a merge commit using `git show <sha> --no-patch` — if I see `Merge:` in the header, I need `-m 1`. Then I run `git revert <sha> --no-commit`, which stages the inverse changes without committing. I run `git diff --staged` to confirm the staged diff looks like the right inverse — it should show the bad changes being removed. If it looks correct, `git commit` with a message that references the incident ticket. Then `git push origin main`. The whole sequence takes under two minutes if the revert is clean. If there's a conflict, I run `git revert --abort`, take a breath, and do `--no-commit` again to resolve the conflict markers manually before committing."

---

**Q: After reverting a bad merge commit on `main` and fixing the bug, what must you do before re-merging the feature branch?**

> **Average answer:** "Just merge the fixed branch back in."

> **Great answer:** "You have to revert the revert first — this is the most common trap with merge reverts and it's caused at least one production issue I've seen. Here's what happens: when you revert the merge commit on `main`, Git records that those commits have been 'seen.' When you re-merge the same feature branch — even with a fix — Git skips the commits it considers already integrated. The original bad code doesn't return, but neither does anything good from the original branch that wasn't part of the fix. The correct sequence is: on the feature branch, run `git revert <sha-of-the-revert-commit>`, which re-exposes the original commits. Then apply your actual fix on top. Then merge or PR back to `main`. The re-merge now sees commits that aren't in `main`'s history and includes them correctly. I put this in our incident runbook explicitly — it's too easy to miss under pressure."

### War Story Prompt

During a late-night incident on the `InventoryApi` service, an engineer ran `git reset --hard HEAD~2` on a branch they thought was personal — but it had been pushed to `origin` and two other engineers had pulled it to run integration tests. The next morning, both engineers' `git pull` produced diverged-branch errors and one of them, confused, ran `git push --force` which pushed the old state of the branch back to the reset version — overwriting the night-shift engineer's subsequent fix commits. The recovery took 45 minutes of `git reflog` archaeology across three laptops. Afterward, the team added a pre-receive hook on the remote that rejected `--force` (without `--lease`) and documented a one-rule policy: if a commit has been pushed, `git revert` — never `git reset`.

## Decision Framework

### When to Reach for `git restore`

- [ ] You've edited a file and want to throw the changes away (not yet staged)
- [ ] You staged a file accidentally and want to unstage it
- [ ] You want to bring a single file back to its state at a specific past commit
- [ ] The change has never been committed — it exists only in your working directory or staging area

### When to Reach for `git revert`

- [ ] The commit has been pushed to a remote branch others may have pulled
- [ ] You need to undo a commit on a protected branch (`main`, `develop`) where force-push is blocked
- [ ] You want the undo recorded as an explicit, auditable event in history (compliance, incident trail)
- [ ] You're reverting a merged PR (use `-m 1`)

### When NOT To

- [ ] Don't use `git restore` if you're not certain you want those changes gone — there is no undo
- [ ] Don't use `git revert` for unpushed local commits where `git reset` gives you a cleaner result — see [[9.014 — Git Reset — Soft, Mixed, Hard]]
- [ ] Don't use `git reset --hard` in place of `git revert` on any commit that has been pushed — this causes diverged histories for everyone who pulled
- [ ] Don't use `git checkout -- <file>` in new scripts — use `git restore <file>`; the old syntax is ambiguous and deprecated

### Tool Comparison

||`git restore`|`git revert`|`git reset`|
|---|---|---|---|
|Scope|Working dir / staging|Commits|Commits|
|Rewrites history?|No|No|Yes|
|Safe after push?|N/A (pre-commit)|Yes — always|No — dangerous|
|Leaves audit trail?|No|Yes (new commit)|No|
|When it's the right call|Discard unsaved/unstaged work|Undo anything already pushed|Restructure local unpushed commits|

## Self-Check

### Conceptual Questions

1. What is the difference between `git restore <file>` and `git restore --staged <file>`? Which parts of the repo does each touch?
2. You run `git revert HEAD` and Git opens your editor. You save and close. What does `git log --oneline -3` show immediately after?
3. What flag does `git revert` require when the target commit is a merge commit, and what does `-m 1` specifically mean?
4. You see this error: `error: commit e7b3c91 is a merge commit. hint: Use -m to specify the mainline.` What command do you run to proceed?
5. `git restore` discards working-directory changes without a confirmation prompt. What is the only reliable recovery path if you ran it on a file you needed?
6. What must you do to a reverted merge commit before re-merging the (now-fixed) feature branch, and why?
7. How does `git revert` differ from `git reset --hard` in terms of safety on a shared branch? See [[9.014 — Git Reset — Soft, Mixed, Hard]] for the reset side.

<details> <summary>Answers</summary>

1. `git restore <file>` discards working-directory changes — it reverts the file's content to what's in the index (staging area), or HEAD if nothing is staged. It doesn't touch the staging area. `git restore --staged <file>` removes the file from the index (unstages it) and restores the staging area to match HEAD, but leaves the working-directory content untouched. To do both at once: `git restore --staged --worktree <file>`.
    
2. `git log --oneline -3` would show:
    
    ```
    f3a9c21 (HEAD -> feature/invoice-export) Revert "feat: add PDF header logo"
    d9f2c11 feat: add PDF header logo
    a3c9f12 fix: update invoice total calculation
    ```
    
    A new commit is added at HEAD with the message "Revert "<original message>"". The original commit remains in history immediately below it.
    
3. `-m <parent-number>`. `-m 1` means "treat parent 1 as the mainline" — i.e., revert the changes introduced by the other parent (the feature branch). Parent 1 is almost always the branch you merged into (e.g., `main`), and parent 2 is the tip of the feature branch that was merged. Check `git show <sha> --no-patch` to see the `Merge: <p1> <p2>` line and confirm which is which.
    
4. `git revert -m 1 e7b3c91`. The `-m 1` tells Git to keep parent 1 (the base branch) as the mainline and revert the changes that parent 2 (the merged feature branch) introduced.
    
5. If the file was open in an editor with local history (VS Code "Timeline" → "Local History", or JetBrains' local history), you can recover from there. Git has no record of working-directory state that was never committed — `git reflog` tracks commits only. If the editor has no history, the changes are unrecoverable.
    
6. You must run `git revert <sha-of-the-revert-commit>` on the feature branch before re-merging. When you reverted the merge commit on `main`, Git recorded those commits as "seen." Re-merging without reverting the revert causes Git to skip those original commits (it considers them already integrated), meaning the good parts of the feature branch may not land. Reverting the revert re-exposes the original commits so the merge machinery includes them correctly.
    
7. `git revert` adds a new commit that is the inverse of the target — history grows forward, every engineer who pulls gets the undo automatically, and no one's local branch diverges. `git reset --hard` moves the branch pointer backward, rewriting history. Anyone who pulled the commits you reset now has commits that no longer exist on the remote — their next `git pull` will show diverged branches and they may accidentally push the old state back. On shared branches, `git reset --hard` is never the right tool; it creates cascading problems for the whole team.
    

</details>

### Hands-On Drills

**Drill 1 — Do it from a clean state**

Clone the `PaymentGateway` repo. On branch `feature/refund-validation`, create and commit a file `src/PaymentGateway/Validators/RefundValidator.cs`. Then, without pushing, make a further working-directory edit to that file. Use `git restore` to discard the working-directory edit. Then stage the file again with a small change, and use `git restore --staged` to unstage it. Confirm with `git status` at each step that the working tree state is exactly what you expect.

<details> <summary>Solution</summary>

```bash
$ git clone git@github.com:company/PaymentGateway.git
$ cd PaymentGateway
$ git switch -c feature/refund-validation

# Create and commit the file
$ mkdir -p src/PaymentGateway/Validators
$ echo "public class RefundValidator { }" > src/PaymentGateway/Validators/RefundValidator.cs
$ git add src/PaymentGateway/Validators/RefundValidator.cs
$ git commit -m "feat: scaffold RefundValidator"
[feature/refund-validation 3b9c1f2] feat: scaffold RefundValidator
 1 file changed, 1 insertion(+)

# Make a working-directory change
$ echo "// EXPERIMENT" >> src/PaymentGateway/Validators/RefundValidator.cs

$ git status
On branch feature/refund-validation
Changes not staged for commit:
  modified:   src/PaymentGateway/Validators/RefundValidator.cs

# Discard the working-directory change
$ git diff src/PaymentGateway/Validators/RefundValidator.cs
# Shows the "// EXPERIMENT" line — confirm what's being discarded
$ git restore src/PaymentGateway/Validators/RefundValidator.cs

$ git status
On branch feature/refund-validation
nothing to commit, working tree clean
# Working directory clean — the EXPERIMENT line is gone

# Now stage a change and unstage it
$ echo "// intentional comment" >> src/PaymentGateway/Validators/RefundValidator.cs

$ git status

$ git status

  modified:   src/PaymentGateway/Validators/RefundValidator.cs
# Unstaged — the comment is still in the file (working dir intact), just not staged
```

**Drill 2 — Fix the broken thing**

```bash
$ git log --oneline -5
9e4b2d1 (HEAD -> main, origin/main) feat: add webhook signature validation
c5a8f31 fix: normalize phone number before SMS dispatch   ← this commit broke SMS in prod
3b7c9e2 feat: add retry queue for failed notifications
1a4d8f7 chore: update appsettings schema
4a3f9c2 fix: prevent double refund on concurrent requests

# You need to revert c5a8f31 on main. You try:
$ git revert c5a8f31
# Git opens your editor, you save the message, and get:
CONFLICT (content): Merge conflict in src/NotificationWorker/Dispatchers/SmsDispatcher.cs
error: could not revert c5a8f31... fix: normalize phone number before SMS dispatch
hint: After resolving the conflicts, mark them with
hint: "git add/rm <pathspec>", then run "git revert --continue".
hint: To abort and get back to the state before "git revert",
hint: run "git revert --abort".
```

<details> <summary>Solution</summary>

**Diagnosis:** A later commit (`9e4b2d1 — webhook signature validation`) also touched `SmsDispatcher.cs` on the same lines that `c5a8f31` modified. Git can't automatically invert `c5a8f31` because the file no longer looks the way it did when that commit was made.

```bash
# Step 1 — open the conflicted file and resolve the conflict markers
$ code src/NotificationWorker/Dispatchers/SmsDispatcher.cs
# Conflict markers look like:
# <<<<<<< HEAD
#     var phone = input.PhoneNumber;   ← current state (after c5a8f31 AND 9e4b2d1)
# =======
#     var phone = rawPhoneNumber;      ← what the revert wants to restore
# >>>>>>> parent of c5a8f31

# The revert wants to remove the normalisation that c5a8f31 added.
# Keep the version WITHOUT the normalisation (the ======= side), also keeping
# any changes from 9e4b2d1 that are on different lines.
# Edit the file manually to represent "the state c5a8f31 should not have introduced"

# Step 2 — mark as resolved
$ git add src/NotificationWorker/Dispatchers/SmsDispatcher.cs

# Step 3 — confirm staged diff looks correct
$ git diff --staged
# Should show the normalisation logic being removed without touching webhook validation code

# Step 4 — complete the revert
$ git revert --continue
# Editor opens with the pre-filled revert message — augment with incident reference, save
[main 7f3c9b2] Revert "fix: normalize phone number before SMS dispatch"
 1 file changed, 3 deletions(-)

# Step 5 — push
$ git push origin main
To github.com:company/NotificationWorker.git
   9e4b2d1..7f3c9b2  main -> origin/main
```

</details>

**Drill 3 — Under time pressure**

Production alert: `OrderService` is returning 500s on all invoice exports. A teammate identifies commit `b8d3e14` — pushed to `main` 45 minutes ago — as the cause. The commit is a normal (non-merge) commit. You have 3 minutes to revert it and push before the SLA breach. Walk through exactly what you type.

<details> <summary>Solution</summary>

```bash
# 1 — Get to latest main immediately
$ git switch main && git pull
# Fast-forward — "Updating xxxxx..yyyyy"

# 2 — Confirm the target commit is present and is NOT a merge commit
$ git show b8d3e14 --no-patch
commit b8d3e14
Author: Jordan Lee <jordan.lee@company.com>
Date:   Fri Jun 26 11:38:04 2026 +0000

    feat: restructure invoice PDF layout
# No "Merge:" line — regular commit, no -m flag needed

# 3 — Revert without editor (--no-edit accepts the default message, saving ~10 seconds)
$ git revert b8d3e14 --no-edit
[main 4c9a2f1] Revert "feat: restructure invoice PDF layout"
 1 file changed, 18 deletions(-)
# Clean revert — no conflict

# 4 — Sanity-check the new HEAD
$ git log --oneline -3
4c9a2f1 (HEAD -> main) Revert "feat: restructure invoice PDF layout"
b8d3e14 feat: restructure invoice PDF layout
9e4b2d1 feat: add webhook signature validation

# 5 — Push
$ git push origin main
To github.com:company/OrderService.git
   b8d3e14..4c9a2f1  main -> origin/main
# Deployment pipeline triggers automatically — 500s will clear once the new build deploys
```

Key decisions under pressure: `--no-edit` skips the editor entirely (saves the 10-second round-trip); checking `--no-patch` first takes 2 seconds but ensures you don't accidentally need `-m 1`; `git pull` before reverting ensures your revert applies cleanly to the current state of `main`, not a stale local copy.

</details>
