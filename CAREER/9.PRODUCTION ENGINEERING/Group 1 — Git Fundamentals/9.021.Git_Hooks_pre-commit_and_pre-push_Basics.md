### Where This Fits

You've just merged a PR where `dotnet format` wasn't run — the diff is littered with whitespace noise, and the reviewer wasted twenty minutes on style comments instead of logic. Or a developer pushed directly to `main` with a failing unit test because the CI pipeline takes four minutes to surface it. Both situations have the same fix: a hook that runs automatically at the right moment in Git's workflow and blocks the operation if the check fails. `pre-commit` and `pre-push` are the two hooks that catch the most problems before they reach the remote.

## What You're Doing and Why

Git hooks are executable scripts that Git invokes automatically at defined points in its workflow. They live in `.git/hooks/` and are named after the event they respond to. A hook that exits with a non-zero status aborts the operation that triggered it — that's the entire enforcement mechanism. Nothing else is required.

`pre-commit` fires before Git writes the commit object — it's the right place to run formatters, linters, and secret scanners that operate on staged files. `pre-push` fires after you run `git push` but before any data is sent to the remote — it's the right place to run tests, since you want to catch failures before they hit CI rather than after.

Because `.git/hooks/` is not tracked by version control, sharing hooks across a team requires either committing them to a tracked directory (e.g., `scripts/hooks/`) and installing them via a setup script, or using a hook management tool like `pre-commit` (the Python tool, distinct from the hook name) or `husky` (for Node.js projects). This note covers the raw hook authoring and installation approach — the foundation both tools build on.

### Mental Model

Think of hooks as bouncers stationed at specific doors. `pre-commit` stands at the "I want to commit" door. `pre-push` stands at the "I want to push" door. Each bouncer runs a check. If the check passes (exit code 0), the person goes through. If the check fails (exit code non-zero), the door stays shut and Git prints whatever the hook wrote to stderr. You write the bouncer's rulebook — a shell script — and drop it in `.git/hooks/`.

```
git commit                    git push
     │                             │
     ▼                             ▼
[pre-commit hook]            [pre-push hook]
     │                             │
  exit 0? ──Yes──► write commit    exit 0? ──Yes──► send to remote
     │                             │
    No                            No
     │                             │
     ▼                             ▼
  abort + print stderr         abort + print stderr
```

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`ls .git/hooks/`|List available hook samples|Exploring what hooks are possible|
|`cat .git/hooks/pre-commit.sample`|Read Git's sample hook for this event|Understanding the hook contract before writing one|
|`chmod +x .git/hooks/pre-commit`|Make a hook executable|Required — Git ignores non-executable hooks silently|
|`git commit --no-verify`|Bypass all hooks for this commit|Emergency use only — never routine|
|`git push --no-verify`|Bypass all hooks for this push|Emergency use only — never routine|
|`git config core.hooksPath scripts/hooks`|Point Git to a tracked directory for hooks|Sharing hooks across the team via version control|
|`git config --global core.hooksPath ~/.git-hooks`|Set a global hooks directory for all repos|Applying personal hooks to every repo you work in|

## Step-by-Step Walkthrough

### The Scenario

You're working on the `OrderService` repo at InterVision. The team has agreed on two rules: every commit must pass `dotnet format --verify-no-changes` (no unformatted code), and every push to a branch must pass `dotnet test --no-build` against the already-built solution. You need to wire both checks as hooks that run automatically, share them with the team through version control, and make the install step part of the repo's standard setup.

### Full Walkthrough

**Part A — Inspect the default hooks directory**

```bash
$ ls .git/hooks/
applypatch-msg.sample  pre-applypatch.sample  pre-rebase.sample
commit-msg.sample      pre-commit.sample      pre-receive.sample
post-update.sample     pre-merge-commit.sample  prepare-commit-msg.sample
pre-push.sample        update.sample
```

Every file here ends in `.sample` — Git ignores `.sample` files. A hook becomes active the moment you create a file with the exact hook name (no extension) and make it executable.

---

**Part B — Write the `pre-commit` hook**

Create a tracked directory for hooks so they live in version control:

```bash
$ mkdir -p scripts/hooks
```

```bash
$ cat > scripts/hooks/pre-commit << 'EOF'
#!/usr/bin/env bash
# pre-commit: enforce dotnet format on staged .cs files
# Exits non-zero to abort the commit if formatting violations are found.

set -euo pipefail

# Only run if there are staged .cs files — skip the check for docs-only commits
STAGED_CS=$(git diff --cached --name-only --diff-filter=ACMR | grep '\.cs$' || true)

if [[ -z "$STAGED_CS" ]]; then
  exit 0   # nothing to check
fi

echo "pre-commit: running dotnet format --verify-no-changes..."

if ! dotnet format --verify-no-changes --verbosity diagnostic 2>&1; then
  echo ""
  echo "pre-commit: FAILED — formatting violations found."
  echo "Run 'dotnet format' to fix them, then stage the changes and commit again."
  echo "To skip this check in an emergency: git commit --no-verify"
  exit 1
fi

echo "pre-commit: formatting check passed."
exit 0
EOF

$ chmod +x scripts/hooks/pre-commit
```

---

**Part C — Write the `pre-push` hook**

```bash
$ cat > scripts/hooks/pre-push << 'EOF'
#!/usr/bin/env bash
# pre-push: run the test suite before sending anything to the remote.
# Git passes the remote name and URL as arguments $1 and $2.
# Reads from stdin: <local-ref> <local-sha> <remote-ref> <remote-sha>
# Exits non-zero to abort the push if tests fail.

set -euo pipefail

REMOTE_NAME="$1"
REMOTE_URL="$2"

# Skip the test run when pushing to a personal fork (not origin or upstream)
if [[ "$REMOTE_URL" != *"intervision"* ]]; then
  echo "pre-push: skipping tests for non-InterVision remote ($REMOTE_URL)."
  exit 0
fi

echo "pre-push: running dotnet test --no-build against $REMOTE_NAME ($REMOTE_URL)..."

if ! dotnet test --no-build --logger "console;verbosity=minimal" 2>&1; then
  echo ""
  echo "pre-push: FAILED — test suite did not pass."
  echo "Fix the failing tests before pushing."
  echo "To skip this check in an emergency: git push --no-verify"
  exit 1
fi

echo "pre-push: all tests passed."
exit 0
EOF

$ chmod +x scripts/hooks/pre-push
```

---

**Part D — Wire the tracked directory as the hooks source**

Tell Git to use `scripts/hooks/` instead of `.git/hooks/`:

```bash
$ git config core.hooksPath scripts/hooks
```

Verify:

```bash
$ git config --get core.hooksPath
scripts/hooks
```

This setting is written to `.git/config` (local scope), so it applies only to this repo. It is **not** committed automatically — which is why a setup script is needed (Part E).

---

**Part E — Create the install script so the team can wire this in one command**

```bash
$ cat > scripts/install-hooks.sh << 'EOF'
#!/usr/bin/env bash
# scripts/install-hooks.sh
# Wires the tracked hooks directory for this repo.
# Run once after cloning: bash scripts/install-hooks.sh

set -euo pipefail

REPO_ROOT="$(git rev-parse --show-toplevel)"
HOOKS_DIR="$REPO_ROOT/scripts/hooks"

if [[ ! -d "$HOOKS_DIR" ]]; then
  echo "Error: $HOOKS_DIR not found. Are you running this from the repo root?"
  exit 1
fi

git config core.hooksPath "$HOOKS_DIR"
echo "Hooks installed. Active hooks directory: $(git config --get core.hooksPath)"
echo "Hooks available:"
ls -1 "$HOOKS_DIR" | grep -v '\.sh$'
EOF

$ chmod +x scripts/install-hooks.sh
```

Add to the repo README under "Getting Started":

````markdown
## Getting Started

After cloning, run the hook installer:

```bash
bash scripts/install-hooks.sh
````

This wires `pre-commit` (dotnet format check) and `pre-push` (test suite) locally.

````

$ git commit -m "chore: add pre-commit format check and pre-push test gate"
````

```bash
$ git add src/OrderService/OrderValidator.cs
$ git commit -m "chore: add pre-commit format check and pre-push test gate"
```

### What Could Go Wrong Mid-Walkthrough

**If the hook fires but does nothing — the check never runs:**

```bash
$ git commit -m "test commit"
[main 9a3c21b] test commit   # committed immediately — hook didn't run
```

The two most likely causes:

1. The hook file is not executable:

```bash
$ ls -la scripts/hooks/pre-commit
-rw-r--r--  1 amercer  staff  612 Jun 26 09:14 scripts/hooks/pre-commit
# No 'x' bit — Git skipped it silently
$ chmod +x scripts/hooks/pre-commit
```

2. `core.hooksPath` is not set:

```bash
$ git config --get core.hooksPath
# (no output — not configured)
$ git config core.hooksPath scripts/hooks
```

## Production Scenario

### Realistic Context

The `PaymentGateway` service at InterVision handles card processing. The team has had two incidents in the past quarter caused by secrets (API keys and connection strings) accidentally committed in plain text — once in a `.json` config file, once in a test fixture. They've also had recurring CI failures from unformatted code that waste reviewer time. The tech lead wants three automated gates: a secret scanner on every commit (using `trufflehog` for the staged diff), a format check on `.cs` files, and a test run on every push to `origin`. All three must be shareable via version control and skippable with `--no-verify` only in documented emergencies.

### Full Implementation / Resolution

```bash
$ mkdir -p scripts/hooks
```

```bash
$ cat > scripts/hooks/pre-commit << 'EOF'
#!/usr/bin/env bash
# pre-commit for PaymentGateway
# Gate 1: secret scan on staged diff
# Gate 2: dotnet format check on staged .cs files

set -euo pipefail

PASS=true

# ── Gate 1: Secret scan ────────────────────────────────────────────────────
echo "pre-commit [1/2]: scanning staged diff for secrets..."

# Pipe the staged diff into trufflehog's stdin scanner
if ! git diff --cached | trufflehog stdin --fail --no-update 2>&1; then
  echo ""
  echo "pre-commit [1/2]: FAILED — potential secret detected in staged diff."
  echo "Remove the secret, rotate any exposed credentials, then re-stage."
  PASS=false
fi

# ── Gate 2: Format check ───────────────────────────────────────────────────
STAGED_CS=$(git diff --cached --name-only --diff-filter=ACMR | grep '\.cs$' || true)

if [[ -n "$STAGED_CS" ]]; then
  echo "pre-commit [2/2]: running dotnet format --verify-no-changes..."

  if ! dotnet format --verify-no-changes --verbosity quiet 2>&1; then
    echo ""
    echo "pre-commit [2/2]: FAILED — formatting violations found."
    echo "Run 'dotnet format', stage the result, then commit again."
    PASS=false
  else
    echo "pre-commit [2/2]: formatting check passed."
  fi
else
  echo "pre-commit [2/2]: no .cs files staged — skipping format check."
fi

# ── Final result ───────────────────────────────────────────────────────────
if [[ "$PASS" == "false" ]]; then
  echo ""
  echo "pre-commit: one or more checks FAILED. Commit aborted."
  echo "Emergency bypass (document the reason): git commit --no-verify"
  exit 1
fi

echo "pre-commit: all checks passed."
exit 0
EOF

chmod +x scripts/hooks/pre-commit
```

```bash
$ cat > scripts/hooks/pre-push << 'EOF'
#!/usr/bin/env bash
# pre-push for PaymentGateway
# Gate 3: full test suite must pass before anything goes to origin

set -euo pipefail

REMOTE_NAME="$1"
REMOTE_URL="$2"

echo "pre-push: remote is $REMOTE_NAME ($REMOTE_URL)"

# Read stdin to find out which refs are being pushed
# Format: <local-ref> <local-sha> <remote-ref> <remote-sha>
# A remote-sha of 0000000000000000000000000000000000000000 means new branch
while IFS=' ' read -r LOCAL_REF LOCAL_SHA REMOTE_REF REMOTE_SHA; do
  if [[ "$LOCAL_SHA" == "0000000000000000000000000000000000000000" ]]; then
    # Branch deletion — no tests to run
    continue
  fi

  echo "pre-push: checking $LOCAL_REF -> $REMOTE_REF"
done

echo "pre-push: running dotnet build --no-incremental..."
if ! dotnet build --no-incremental --verbosity quiet 2>&1; then
  echo "pre-push: FAILED — build did not succeed."
  exit 1
fi

echo "pre-push: running dotnet test --no-build..."
if ! dotnet test --no-build \
     --logger "console;verbosity=minimal" \
     --results-directory /tmp/payment-gateway-test-results 2>&1; then
  echo ""
  echo "pre-push: FAILED — test suite did not pass."
  echo "Fix failing tests before pushing to $REMOTE_NAME."
  echo "Emergency bypass (document the reason): git push --no-verify"
  exit 1
fi

echo "pre-push: build and tests passed. Sending to $REMOTE_NAME."
exit 0
EOF

chmod +x scripts/hooks/pre-push
```

```bash
$ cat > scripts/install-hooks.sh << 'EOF'
#!/usr/bin/env bash
# scripts/install-hooks.sh
# Idempotent. Run once after cloning: bash scripts/install-hooks.sh

set -euo pipefail

REPO_ROOT="$(git rev-parse --show-toplevel)"
HOOKS_DIR="$REPO_ROOT/scripts/hooks"

# Ensure all hooks are executable
chmod +x "$HOOKS_DIR"/pre-commit "$HOOKS_DIR"/pre-push

# Point git at the tracked hooks directory
git config core.hooksPath "$HOOKS_DIR"

echo ""
echo "Git hooks installed successfully."
echo "  Hooks directory : $(git config --get core.hooksPath)"
echo "  Active hooks    :"
ls -1 "$HOOKS_DIR" | grep -v '\.sh$' | sed 's/^/    /'
echo ""
echo "To bypass a hook in a documented emergency:"
echo "  git commit --no-verify   (skips pre-commit)"
echo "  git push --no-verify     (skips pre-push)"
EOF

chmod +x scripts/install-hooks.sh
```

Commit everything:

```bash
$ git add scripts/
$ git commit -m "chore: add pre-commit secret scan + format gate and pre-push test gate"
pre-commit [1/2]: scanning staged diff for secrets...
pre-commit [1/2]: secret scan passed.
pre-commit [2/2]: no .cs files staged — skipping format check.
pre-commit: all checks passed.
[main 7c3f91a] chore: add pre-commit secret scan + format gate and pre-push test gate
 3 files changed, 89 insertions(+)
```

## Gotchas and Troubleshooting

### Hook is present but never fires — commit or push proceeds without running it

**Root cause:** The hook file is missing the executable bit. Git checks `access(hook_path, X_OK)` before running a hook — if the file isn't executable, Git silently skips it. There is no warning or error from Git itself.

```bash
$ git commit -m "test"
[main 4a2b19c] test    # hook never ran
```

**Fix:**

```bash
$ ls -la .git/hooks/pre-commit
-rw-r--r--  1 amercer  staff  421  Jun 26 09:14 .git/hooks/pre-commit
#            ↑ no x bit

$ chmod +x .git/hooks/pre-commit
# or, if using a tracked directory:
$ chmod +x scripts/hooks/pre-commit
```

**Prevention:** Always include `chmod +x` in the install script and in the same shell session where you create the hook file. Add a check to `scripts/install-hooks.sh` that verifies the executable bit is set.

---

### "pre-commit: command not found" or hook exits with 126/127

**Root cause:** The shebang line is wrong or missing, the interpreter isn't available at the path specified, or the hook references a tool (`dotnet`, `trufflehog`) that isn't on `$PATH` in the shell Git spawns.

```bash
$ git commit -m "test"
error: cannot run .git/hooks/pre-commit: No such file or directory
# or:
pre-commit: line 3: dotnet: command not found
```

**Fix — verify shebang and PATH:**

```bash
# Confirm the shebang is the first line:
$ head -1 scripts/hooks/pre-commit
#!/usr/bin/env bash

# Confirm dotnet is findable from a non-interactive shell:
$ env -i PATH="$PATH" bash -c 'dotnet --version'
9.0.5   # if this fails, dotnet isn't on the PATH Git will use

# Add the dotnet path explicitly at the top of the hook if needed:
export PATH="$PATH:/usr/local/share/dotnet"
```

**Prevention:** Always use `#!/usr/bin/env bash` (not `#!/bin/bash`) — it resolves `bash` through `$PATH`, which is more portable. Explicitly export the `PATH` at the top of any hook that calls tools installed in non-standard locations (e.g., `.NET SDK` on macOS via installer, `nvm`-managed Node).

---

### Hook fires but `git commit --no-verify` is being used to bypass it routinely

**Root cause:** The hook is too slow (running the full test suite in `pre-commit` instead of just a format check) or too aggressive (blocking commits with warnings instead of only hard failures). Developers start using `--no-verify` as their default commit command, defeating the purpose entirely.

```bash
# In shell history:
$ git commit --no-verify -m "wip"
$ git commit --no-verify -m "fix tests"
$ git commit --no-verify -m "real fix"
```

**Fix — move slow checks to `pre-push` and keep `pre-commit` fast:**

- `pre-commit`: only run checks that complete in under 3 seconds on staged files (format, lint, secret scan on the diff — not the full tree)
- `pre-push`: run tests, builds, and anything that takes more than a few seconds

**Prevention:** Time your hook explicitly. If `pre-commit` takes more than 5 seconds, it will be bypassed. A useful heuristic: `pre-commit` should feel instant; `pre-push` can take up to 2 minutes.

---

### Hooks work locally but don't run in CI

**Root cause:** This is expected and correct behaviour. Hooks are local-only — they live in `.git/hooks/` (or `core.hooksPath`) and are never transmitted to the remote. CI pipelines do not run Git hooks; they run explicit pipeline steps instead.

```bash
# CI log shows no mention of pre-commit or pre-push:
Run dotnet test
  Starting test execution...
```

**Fix — the CI pipeline must duplicate the checks the hooks enforce:**

```yaml
# .github/workflows/ci.yml
- name: Check formatting
  run: dotnet format --verify-no-changes --verbosity diagnostic

- name: Run tests
  run: dotnet test --no-build --logger "trx" --results-directory test-results
```

**Prevention:** Treat hooks as a developer convenience (fast feedback, catches issues before they reach CI) and CI as the authoritative gate (hooks can be bypassed; CI cannot). Both layers are necessary — hooks reduce CI noise, CI enforces the rules.

---

### `core.hooksPath` setting is lost after a fresh clone

**Root cause:** `core.hooksPath` is a local config setting written to `.git/config`, which is not tracked by version control. Every new clone starts without it.

```bash
# New clone:
$ git clone git@github.com:intervision/order-platform.git
$ git config --get core.hooksPath
# (no output — not set in fresh clone)
```

**Fix — the install script handles this; it just needs to be run:**

```bash
$ bash scripts/install-hooks.sh
```

**Prevention:** Add `bash scripts/install-hooks.sh` as a step in the repo's onboarding doc, the `Makefile`'s `setup` target, and/or a `dotnet tool` post-restore script. Some teams add it to their `devcontainer.json` `postCreateCommand` so it runs automatically in dev containers.

---

### Hook blocks a commit to `main` on a hotfix when there's no time to fix formatting

**Root cause:** The `pre-commit` hook has no escape valve configured — or the developer doesn't know about `--no-verify`.

```bash
$ git commit -m "hotfix: fix null ref in PaymentProcessor"
pre-commit: FAILED — formatting violations found.
# (but it's 23:00 and the site is down)
```

**Fix — bypass with documented justification:**

```bash
$ git commit --no-verify -m "hotfix: fix null ref in PaymentProcessor [format-skip: hotfix under incident INC-4821]"
```

**Prevention:** Document `--no-verify` clearly in the hook's own error output (as in the examples above) and in the team runbook. The bypass is a feature, not a flaw — the goal is for it to be used rarely and with a documented reason, not never used at all.

---

### Hook on Windows fails with "'\r': command not found" or similar

**Root cause:** The hook script was authored on Windows (or committed without correct line-ending handling) and has CRLF line endings. When Git invokes the script on Linux/Mac (or in WSL), the `\r` on each line is treated as part of the command name.

```bash
$ git commit -m "test"
.git/hooks/pre-commit: line 2: $'\r': command not found
.git/hooks/pre-commit: line 4: $'\r': command not found
```

**Fix:**

```bash
# Convert line endings in the hook file:
$ sed -i 's/\r//' scripts/hooks/pre-commit
$ sed -i 's/\r//' scripts/hooks/pre-push

# Or with dos2unix:
$ dos2unix scripts/hooks/pre-commit scripts/hooks/pre-push
```

**Prevention:** Add a `.gitattributes` rule to ensure hook scripts always use LF line endings regardless of the author's OS:

```
scripts/hooks/* text eol=lf
```

See [[9.018 — .gitattributes — Line Endings and Diff Behavior]] for the full line-ending configuration.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# In ~/.gitconfig [alias] section

[alias]
  # Commit bypassing all hooks — for emergencies only; paste the reason in the message
  cnv   = commit --no-verify

  # Push bypassing all hooks — for emergencies only
  pnv   = push --no-verify

  # Show which hooks directory is active for this repo
  hooks = "!echo \"Hooks directory: $(git config --get core.hooksPath 2>/dev/null || echo '.git/hooks (default)')\""

  # List active hooks in the configured directory
  hooksls = "!HDIR=$(git config --get core.hooksPath 2>/dev/null || echo '.git/hooks'); ls -la \"$HDIR\" | grep -v '\\.sample'"
```

Usage:

```bash
$ git hooks
Hooks directory: scripts/hooks

$ git hooksls
-rwxr-xr-x  1 amercer  staff  721 Jun 26 09:14 scripts/hooks/pre-commit
-rwxr-xr-x  1 amercer  staff  893 Jun 26 09:31 scripts/hooks/pre-push
```

### Automation Opportunity

Add hook installation as a target in the project `Makefile` so it's part of the standard `make setup` flow every developer runs after cloning:

```makefile
# Makefile

.PHONY: setup hooks test format

setup: hooks
	dotnet restore
	dotnet build
	@echo "Setup complete."

hooks:
	@echo "Installing git hooks..."
	@chmod +x scripts/hooks/pre-commit scripts/hooks/pre-push
	@git config core.hooksPath scripts/hooks
	@echo "  Hooks active: $$(ls scripts/hooks | grep -v '\.sh$$' | tr '\n' ' ')"

format:
	dotnet format

test:
	dotnet test --no-build --logger "console;verbosity=minimal"
```

Now every developer runs `make setup` once after cloning and hooks are wired automatically. No separate `bash scripts/install-hooks.sh` step to remember.

### Keyboard / CLI Efficiency Notes

- The single most important thing to know about `pre-commit` hooks: **they only see staged files** — `git diff --cached --name-only` is the correct command to scope a hook to what's actually being committed, rather than running the check against the entire working tree
- `git stash --include-untracked` before running format in a hook avoids false positives — without stashing, `dotnet format` may format unstaged files and make the check fail for files that aren't part of the commit
- Test hooks in isolation before installing: `bash -x scripts/hooks/pre-commit` runs the hook in debug mode, printing every command as it executes — the fastest way to diagnose why a hook isn't working as expected
- On CI: `git config core.hooksPath /dev/null` disables all hooks for the CI process without modifying the repo — useful if CI accidentally picks up a hooks directory

### Spoken Answers

**Q: Walk me through exactly what happens when `git commit` runs a `pre-commit` hook.**

> **Average answer:** "Git runs the script before writing the commit. If it exits non-zero, the commit is aborted."

> **Great answer:** "When you run `git commit`, Git resolves the hooks directory — either `.git/hooks/` or whatever `core.hooksPath` points to — and looks for a file named exactly `pre-commit`. If the file exists and is executable, Git invokes it as a subprocess with the same working directory as the repo root. The hook receives no arguments; it reads the staged index itself if it needs to know what's being committed — typically via `git diff --cached`. If the hook exits with code 0, Git proceeds to write the commit object and open the editor for the commit message (if `-m` wasn't provided). If the hook exits with any non-zero code, Git aborts the entire commit, prints whatever the hook wrote to stderr, and leaves the staging area exactly as it was. No commit object is written. The developer has to fix whatever the hook complained about and run `git commit` again — or use `--no-verify` to bypass, which is an explicit, traceable override rather than a silent failure."

---

**Q: How do you share Git hooks across a team when `.git/hooks/` is not tracked by version control?**

> **Average answer:** "Put the hooks in a folder like `scripts/hooks/` and tell people to copy them."

> **Great answer:** "The cleanest approach is `core.hooksPath`. You commit your hook scripts to a tracked directory — I usually use `scripts/hooks/` — and then configure Git to look there instead of `.git/hooks/` by running `git config core.hooksPath scripts/hooks`. That config is local to the repo but it's a one-liner, so you write a setup script that runs it — `scripts/install-hooks.sh` — and make it part of the standard `make setup` target every developer runs after cloning. The key points are: first, the hook files in `scripts/hooks/` must be executable — the install script sets `chmod +x` on them. Second, `core.hooksPath` is written to `.git/config`, which is not committed, so every new clone needs to run the install script. Third, CI does not run hooks — it runs explicit pipeline steps — so you duplicate the same checks in the CI YAML. Hooks are fast feedback for developers; CI is the authoritative gate."

---

**Q: A developer routinely uses `git commit --no-verify`. What does that tell you about the hook design?**

> **Average answer:** "The hook is probably too slow or too strict."

> **Great answer:** "Routine `--no-verify` use is a signal that the hook is broken as a developer tool — not because of what it checks, but because of _when_ or _how_. The most common cause is that `pre-commit` is running the full test suite — something that takes 30-60 seconds — when it should only be running fast static checks like formatting and linting. Tests belong in `pre-push`, not `pre-commit`. The second common cause is that the hook blocks on warnings rather than only on hard failures, so developers bypass it because it keeps stopping them for things they can't or won't fix right now. The fix is to audit the hook for runtime and scope: `pre-commit` should finish in under 3 seconds and only check what's staged, not the whole tree. If a check genuinely can't be made fast, move it to `pre-push`. The goal is for `--no-verify` to feel like an unusual override used in genuine emergencies — not a habit developers build because the hook is in their way."

### War Story Prompt

A developer at InterVision committed a Stripe live secret key into the `PaymentGateway` repo — `sk_live_...` — buried inside a JSON test fixture. The commit was pushed to a feature branch, opened as a PR, and sat there for six hours before a reviewer spotted it. The key had to be rotated immediately, the commit had to be force-purged from history (including all forks), and GitHub support had to be contacted to remove the content from cached views. The total incident cost was four hours of engineering time and a 20-minute payment processing degradation while the new key was provisioned and deployed. The root cause was a blank `pre-commit` hook directory — the repo had `pre-commit.sample` in `.git/hooks/` but the actual hook was never created. After the incident, a `trufflehog` secret scan was added to the `pre-commit` hook and to the CI pipeline as a redundant gate. The engineering manager noted in the post-mortem that the CI scan would have caught it too — but it was configured to run only on merge to `main`, not on PR push. Both gaps were closed.

## Decision Framework

### When to Reach for This

- [ ] The team has recurring CI failures from issues that could be caught locally before pushing (formatting, obvious syntax errors, accidentally committed secrets)
- [ ] Code review time is being spent on style comments rather than logic — a format hook eliminates this noise
- [ ] You want developers to get feedback in under 5 seconds rather than waiting 4+ minutes for CI
- [ ] A specific class of mistake has occurred more than once (leaked secret, broken build pushed to `main`) and needs a local gate
- [ ] You're onboarding new engineers who aren't yet fluent in the team's tooling — hooks encode the rules they'd otherwise miss

### When NOT To

- [ ] Do not put slow operations (full build, integration tests, database migrations) in `pre-commit` — put them in `pre-push` or CI only
- [ ] Do not use hooks as a substitute for CI — hooks can always be bypassed with `--no-verify`; CI cannot
- [ ] Do not write hooks that modify staged files silently and then succeed — this causes confusing diffs where the commit content differs from what the developer staged; always fail with an actionable message instead of auto-fixing
- [ ] Do not rely on hooks for security enforcement — they are a convenience gate for developers, not a security boundary; all security checks must also run in CI

### Tool Comparison

||Raw bash hook|`pre-commit` tool (Python)|`husky` (Node)|
|---|---|---|---|
|Setup complexity|Low — one file, one `chmod`|Medium — requires Python, `pip install pre-commit`, `.pre-commit-config.yaml`|Medium — requires Node, `npm install husky`|
|Ecosystem|Any tool you can shell out to|Large registry of pre-built hooks; language-agnostic|npm-centric; good for JS/TS frontends|
|Shareability|Via `core.hooksPath` + install script|Via `.pre-commit-config.yaml` (committed)|Via `package.json` `prepare` script|
|Flexibility|Total — write any bash you want|High — write custom hooks or use registry hooks|High for Node-based checks; awkward for .NET|
|Best for|.NET teams with a few specific checks|Polyglot teams or teams wanting a standard hook registry|Frontend teams with existing Node toolchain|
|Bypass|`--no-verify`|`--no-verify` or `SKIP=hook-name git commit`|`--no-verify`|

## Self-Check

### Conceptual Questions

1. What is the only mechanism by which a Git hook aborts the operation that triggered it?
2. You create a file at `.git/hooks/pre-commit` with a valid bash script. You run `git commit`. The hook doesn't fire. What's the single most likely cause?
3. What is `core.hooksPath` and why is it important for team hook sharing?
4. A `pre-push` hook receives data on stdin. What is the format of that data and what does each field mean?
5. Why do Git hooks in `.git/hooks/` not run on CI, and how do you ensure CI enforces the same rules?
6. What is the difference between a hook exiting with code 1 versus code 0 in terms of Git's behaviour?
7. How does `git stash` relate to running `dotnet format` inside a `pre-commit` hook, and why does it matter? (Connect to [[9.016 — Git Stash — Save, Pop, Apply, Drop]])

<details> <summary>Answers</summary>

1. The hook exits with a **non-zero exit code**. Any non-zero value aborts the triggering operation. Exit code 0 allows it to proceed. This is the entire enforcement mechanism — nothing else is checked.
    
2. The hook file is **not executable**. Git checks the executable bit (`access(path, X_OK)`) before invoking a hook. If the file is not executable, Git silently skips it — no error, no warning. Fix: `chmod +x .git/hooks/pre-commit`.
    
3. `core.hooksPath` is a Git config setting (written to `.git/config` at local scope, or `~/.gitconfig` at global scope) that tells Git to look for hook scripts in a specified directory instead of the default `.git/hooks/`. This is important for team sharing because `.git/hooks/` is not tracked by version control — you can't commit files into it. By pointing `core.hooksPath` at a directory like `scripts/hooks/` that _is_ tracked, the hook scripts themselves are committed and versioned, and all developers share the same hooks once they run the install script that sets `core.hooksPath`.
    
4. `pre-push` receives one line per ref being pushed on stdin, in the format:
    
    ```
    <local-ref> <local-sha> <remote-ref> <remote-sha>
    ```
    
    - `local-ref`: the local branch name being pushed (e.g., `refs/heads/feature/payment-retry`)
    - `local-sha`: the SHA of the local commit being pushed
    - `remote-ref`: the remote branch it's being pushed to
    - `remote-sha`: the current SHA at the remote (all zeros if the remote branch doesn't exist yet — i.e., a new branch being pushed for the first time)
5. Git hooks live in `.git/hooks/` (or `core.hooksPath`), which is a local filesystem path on the developer's machine. When CI clones a repo, it gets a fresh `.git/hooks/` directory containing only the `.sample` files — none of the installed hooks are transmitted by the remote. To ensure CI enforces the same rules, you must duplicate the checks as explicit pipeline steps in the CI YAML (e.g., `dotnet format --verify-no-changes`, `dotnet test`). Hooks are fast feedback for developers; CI is the authoritative, un-bypassable gate.
    
6. **Exit code 0**: the hook succeeded; Git proceeds with the operation (writes the commit, sends data to the remote). **Exit code non-zero (any value except 0)**: the hook failed; Git aborts the operation, prints any output the hook wrote to stderr, and leaves the repository state exactly as it was before the operation was attempted. The specific non-zero value doesn't matter to Git — only zero vs non-zero is evaluated.
    
7. When `dotnet format` runs inside a `pre-commit` hook, it operates on files in the working tree — not just the staged index. If you have unstaged changes in a file that's also staged, `dotnet format` may format the unstaged content and report a violation even though the commit would not have included that content. The fix is to `git stash --keep-index` before running the format check (stashing only the unstaged changes, preserving the staged index), run the check, then `git stash pop` to restore the unstaged work. Without this, the hook can produce false positives: it fails on content that wasn't going to be committed anyway. See [[9.016 — Git Stash — Save, Pop, Apply, Drop]] for the full `--keep-index` stash pattern.
    

</details>

### Hands-On Drills

**Drill 1 — Do it from a clean state**

Starting from a fresh clone of any Git repo (or a new `git init`), create a `pre-commit` hook at `.git/hooks/pre-commit` that prints `"pre-commit: hook is running"` and exits 0. Verify it fires on `git commit`. Then change the exit code to 1, attempt another commit, and confirm the commit is aborted. Finally, use `git commit --no-verify` and confirm the commit proceeds despite the failing hook.

<details> <summary>Solution</summary>

```bash
$ mkdir /tmp/drill-hooks && cd /tmp/drill-hooks
$ git init
Initialized empty Git repository in /tmp/drill-hooks/.git/

# Create a passing hook
$ cat > .git/hooks/pre-commit << 'EOF'
#!/usr/bin/env bash
echo "pre-commit: hook is running"
exit 0
EOF
$ chmod +x .git/hooks/pre-commit

# Test that it fires
$ touch test.txt && git add test.txt
$ git commit -m "test: hook should fire"
pre-commit: hook is running
[main (root-commit) 7a3f21c] test: hook should fire
 1 file changed, 0 insertions(+), 0 deletions(-)
 create mode 100644 test.txt
# Hook fired and commit succeeded (exit 0)

# Change to failing hook
$ cat > .git/hooks/pre-commit << 'EOF'
#!/usr/bin/env bash
echo "pre-commit: hook is running — FAILING"
exit 1
EOF
$ chmod +x .git/hooks/pre-commit

$ touch test2.txt && git add test2.txt
$ git commit -m "test: this commit should be blocked"
pre-commit: hook is running — FAILING
# Commit was aborted — no new commit SHA printed

$ git log --oneline
7a3f21c (HEAD -> main) test: hook should fire
# Still only one commit — the second was blocked

# Bypass with --no-verify
$ git commit --no-verify -m "test: bypassed hook"
[main 4b1e93d] test: bypassed hook
 1 file changed, 0 insertions(+), 0 deletions(-)
# Hook did not run — commit succeeded

$ git log --oneline
4b1e93d (HEAD -> main) test: bypassed hook
7a3f21c test: hook should fire

# Cleanup
$ cd ~ && rm -rf /tmp/drill-hooks
```

```bash
$ git commit -m "feat: add inventory sync to OrderService"
[main 9c3f81a] feat: add inventory sync to OrderService
 1 file changed, 28 insertions(+)
# The commit went through instantly — but the repo has a scripts/hooks/pre-commit
# that should have run dotnet format --verify-no-changes.

$ ls scripts/hooks/
pre-commit  pre-push  install-hooks.sh

$ bash -x scripts/hooks/pre-commit
+ set -euo pipefail
+ git diff --cached --name-only --diff-filter=ACMR
+ grep '\.cs$'
+ STAGED_CS=
+ [[ -z '' ]]
+ exit 0
# The hook exits 0 — but it should have caught formatting issues.
# Diagnose why the hook isn't blocking bad commits and fix the configuration.
```

<details> <summary>Solution</summary>

**Diagnosis step 1 — check if core.hooksPath is configured:**

```bash
$ git config --get core.hooksPath
# (no output — not set)
```

`core.hooksPath` was never set after cloning. Git is using the default `.git/hooks/` directory, which has no `pre-commit` file (only `pre-commit.sample`). The `scripts/hooks/pre-commit` exists but Git doesn't know to look there.

**Diagnosis step 2 — verify the default hooks directory:**

```bash
$ ls .git/hooks/
applypatch-msg.sample  pre-commit.sample  pre-push.sample  ...
# No executable pre-commit file — only the sample
```

**Fix:**

```bash
# Run the install script (which sets core.hooksPath)
$ bash scripts/install-hooks.sh
Git hooks installed successfully.
  Hooks directory : scripts/hooks
  Active hooks    :
    pre-commit
    pre-push

# Verify
$ git config --get core.hooksPath
scripts/hooks

# Now test the hook fires on a new commit with a formatting issue
$ echo "public class Test{public void Method(){}}" > src/OrderService/Test.cs
$ git add src/OrderService/Test.cs
$ git commit -m "test: hook should fire now"
pre-commit: running dotnet format --verify-no-changes...
  Formatted code file 'src/OrderService/Test.cs'.

pre-commit: FAILED — formatting violations found.
Run 'dotnet format' to fix them, then stage the changes and commit again.
# Hook is now blocking correctly

# Clean up the test file
$ git restore --staged src/OrderService/Test.cs
$ rm src/OrderService/Test.cs
```

**Root cause summary:** `core.hooksPath` was not set — the `scripts/install-hooks.sh` step was skipped after cloning. The hook scripts were present and executable in `scripts/hooks/` but Git had no instruction to look there. Add `bash scripts/install-hooks.sh` to the onboarding checklist and the `make setup` target.

**Drill 3 — Under time pressure**

It's an active incident. `PaymentGateway` is returning 500s. You've identified the fix — a one-line null check in `PaymentProcessor.cs` — and you've staged it. But your `pre-commit` hook is failing because `dotnet format` is complaining about an unrelated file (`LegacyAdapter.cs`) that was already in that state before you touched anything. The format fix would take 10 minutes you don't have. You need to commit the fix, push it, and get it deployed in under 90 seconds. Walk through exactly what you type, including how you document the bypass.

<details> <summary>Solution</summary>

```bash
# 1 — Confirm the fix is staged and only the intended file is included
$ git diff --cached --name-only
src/PaymentGateway/PaymentProcessor.cs   # correct — only the fix

# 2 — Commit with --no-verify, documenting the bypass reason inline
$ git commit --no-verify -m "hotfix: null check in PaymentProcessor prevents 500 on empty order

format-skip: pre-commit bypassed under incident INC-5032
LegacyAdapter.cs formatting pre-existing; tracked in TECH-8821
Pair: amercer + dpatel"
[main 2c8f91a] hotfix: null check in PaymentProcessor prevents 500 on empty order

# 3 — Push with --no-verify to skip the pre-push test gate
#    (tests can't run without a build; we need speed over safety here)
$ git push --no-verify origin main
Enumerating objects: 5, done.
To github.com:intervision/payment-gateway.git
   7c3f91a..2c8f91a  main -> main

# 4 — Immediately open a follow-up ticket for the bypassed checks
# (mental note: after incident is resolved)
# - Run dotnet format on LegacyAdapter.cs
# - Push a formatting-only commit
# - Verify CI passes on the next build

# 5 — Confirm CI triggered (GitHub Actions)
# Check: https://github.com/intervision/payment-gateway/actions
# Look for run on SHA 2c8f91a — CI will still run the format and test checks
# even though the local hooks were bypassed
```

**Key principle:** `--no-verify` bypasses hooks but does not bypass CI. The CI pipeline will still run `dotnet format --verify-no-changes` and `dotnet test` — if the null check itself breaks a test, CI will catch it. The bypass only removes the _local_ gates; the remote authoritative gates remain active. Always verify CI passes after an emergency bypass, and always create a follow-up ticket for the thing you skipped.

</details>
