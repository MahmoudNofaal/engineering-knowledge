## PART 1 — The Core Mental Model

ASP.NET Core’s built-in Dependency Injection container (the `Microsoft.Extensions.DependencyInjection` or MEDI container) is designed specifically as a "conforming container." It provides a foundational, lowest-common-denominator feature set: Transient/Scoped/Singleton lifetimes, constructor injection, and basic `IEnumerable<T>` resolution. It was explicitly built to be *just enough* for the framework itself and average line-of-business applications, but it was also designed with a critical integration point (`IServiceProviderFactory<TContainerBuilder>`) allowing it to be swapped out for more powerful, mature third-party Inversion of Control (IoC) containers.

Think of the built-in DI as a standard automatic transmission—perfect for 95% of driving. Replacing it with Autofac or Lamar is like installing a specialized manual transmission with a dual-clutch and custom gear ratios. You gain advanced capabilities like property injection, native decorators, advanced generic covariance/contravariance handling, module-based registrations, and granular lifetime scopes, but you take on the overhead of managing the adapter boundary between ASP.NET Core's expectations and the third-party container's reality.

**Autofac** acts as a heavy-duty, highly extensible container. It brings a massive ecosystem of advanced registration APIs (Assembly scanning, Modules, Delegate factories) and granular control over disposal and lifetime scopes (`InstancePerMatchingLifetimeScope`), but at the cost of slightly higher resolution overhead.
**Lamar** (the spiritual successor to StructureMap) focuses heavily on lightning-fast resolution. It uses dynamic code generation via Roslyn to compile object instantiation graphs into plain C# at startup, making it significantly faster than Autofac while retaining advanced features like offline container validation (`AssertConfigurationIsValid`).

Replacing the container isn't an "all or nothing" override. The ASP.NET Core framework still registers its own services (like MVC routing, authentication) using the standard `IServiceCollection`. The third-party container *consumes* this collection, merges it with its own advanced registrations, and then takes over the actual job of object instantiation (`IServiceProvider` fulfillment) for the rest of the application's lifecycle.

## PART 2 — Deep Mechanics

### 2.1. The `IServiceProviderFactory` Abstraction
To replace the default container, you must provide an implementation of `IServiceProviderFactory<TContainerBuilder>`. This interface enforces a two-step handshake:
1. `CreateBuilder(IServiceCollection services)`: The host passes the fully populated `IServiceCollection` (containing all framework registrations) to the factory. The factory converts these standard `ServiceDescriptor` objects into the third-party container's specific registration format (e.g., Autofac's `ContainerBuilder` or Lamar's `ServiceRegistry`).
2. `CreateServiceProvider(TContainerBuilder containerBuilder)`: The host asks the factory to finalize the container and return an `IServiceProvider`. Once this returns, the built-in `DefaultServiceProvider` is completely abandoned, and all framework requests for dependencies route through the third-party container.

### 2.2. The ASP.NET Core Host Integration Pipeline
In .NET 6+ minimal hosting (`WebApplicationBuilder`), replacing the container is done via `builder.Host.UseServiceProviderFactory()`. The sequence of events is crucial:
1. `builder.Services.Add...()` registers framework dependencies into MEDI.
2. `builder.Host.ConfigureContainer<TContainerBuilder>()` allows you to register third-party specific services (like Autofac Modules).
3. Upon calling `builder.Build()`, the host invokes `CreateBuilder` on your factory, merging MEDI and third-party registrations.
4. The host invokes `CreateServiceProvider`, finalizing the container.
5. The request pipeline is built using the new container. 

### 2.3. Adapting Lifetimes and Scopes
ASP.NET Core has three rigid lifetimes: `Transient`, `Scoped`, and `Singleton`. When adapting to Autofac, these are mapped precisely:
- `Transient` $\rightarrow$ `InstancePerDependency()`
- `Scoped` $\rightarrow$ `InstancePerLifetimeScope()` (bound to the HTTP Request scope)
- `Singleton` $\rightarrow$ `SingleInstance()`

However, Autofac allows deeper scopes (e.g., `InstancePerMatchingLifetimeScope("my-background-task")`). You must be exceedingly careful: ASP.NET Core framework components (like `IHttpContextAccessor`) assume standard HTTP request scoping. If you resolve them inside custom deep scopes without proper nesting, you can cause lifecycle leaks or `ObjectDisposedException`s.

### 2.4. Roslyn Compilation in Lamar
Unlike Autofac which evaluates dependencies dynamically or via reflection at resolution time, Lamar takes a radical approach: it converts your IoC configuration into a raw C# source code syntax tree representing the exact instantiation logic (e.g., `new MyController(new MyService(), new MyRepo())`). It then invokes the Roslyn compiler at startup to compile this into a dynamic assembly.
This means:
1. Startup is slightly slower as compilation occurs.
2. Resolution is exceptionally fast (equivalent to handwritten `new` statements).
3. You can actually inspect the generated code using `container.GenerateCode()`.

### 2.5. Autofac Modules and Architectural Boundaries
One of the most compelling reasons to switch to Autofac is `Module`. Unlike C# extension methods on `IServiceCollection` which can become cluttered, an Autofac `Module` is a class that encapsulates a set of related registrations. This allows you to deploy assemblies with their own IoC wiring embedded inside them. The application host simply scans the assemblies and loads the modules, achieving true plugin-style decoupled architecture without the host needing to reference every implementation project directly.

## PART 3 — Production Code Patterns

### Pattern 1: Wiring up Autofac in .NET 8 `Program.cs`
This is the standard baseline for injecting Autofac into the modern minimal hosting model. Note how we transition from `builder.Services` to `builder.Host.ConfigureContainer`.

```csharp
using Autofac;
using Autofac.Extensions.DependencyInjection;

var builder = WebApplication.CreateBuilder(args);

// 1. Tell the Host to use Autofac as the factory
builder.Host.UseServiceProviderFactory(new AutofacServiceProviderFactory());

// 2. Standard ASP.NET Core registrations
builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();

// 3. Configure Autofac-specific registrations
builder.Host.ConfigureContainer<ContainerBuilder>(containerBuilder =>
{
    // Native Autofac module registration
    containerBuilder.RegisterModule(new OrderingDomainModule());
    
    // Advanced Autofac registration: Property Injection
    containerBuilder.RegisterType<EmailSender>()
                    .As<IEmailSender>()
                    .PropertiesAutowired();
});

var app = builder.Build();

app.MapControllers();
app.Run();
```

### Pattern 2: Wiring up Lamar in .NET 8 `Program.cs`
Lamar uses a slightly different factory but follows the exact same logical pipeline.

```csharp
using Lamar.Microsoft.DependencyInjection;

var builder = WebApplication.CreateBuilder(args);

// 1. Use Lamar's factory
builder.Host.UseLamar();

// 2. Lamar specific configuration
builder.Host.ConfigureContainer<Lamar.ServiceRegistry>(services =>
{
    // Assembly scanning - Lamar's superpower
    services.Scan(s =>
    {
        s.TheCallingAssembly();
        s.WithDefaultConventions(); // IMyService -> MyService automatically
        s.Singletons(); // Make everything found a singleton (optional)
    });
    
    services.For<IPaymentProcessor>().Use<StripePaymentProcessor>();
});

var app = builder.Build();

// Optional: Validate configuration at startup (fails fast on missing dependencies)
app.Services.GetRequiredService<Lamar.IContainer>().AssertConfigurationIsValid();

app.Run();
```

### Pattern 3: Autofac Modules for Architectural Boundaries
A core Clean Architecture practice. The Web project references the `Infrastructure` project, but instead of the Web project knowing about `SqlCustomerRepository`, the `Infrastructure` project registers its own internals.

```csharp
// Inside Infrastructure Project
using Autofac;

public class InfrastructureModule : Module
{
    protected override void Load(ContainerBuilder builder)
    {
        // Assembly scanning within the specific architectural layer
        builder.RegisterAssemblyTypes(ThisAssembly)
               .Where(t => t.Name.EndsWith("Repository"))
               .AsImplementedInterfaces()
               .InstancePerLifetimeScope(); // Scoped

        // Register a singleton connection factory
        builder.RegisterType<SqlConnectionFactory>()
               .As<IDbConnectionFactory>()
               .SingleInstance();
    }
}
```

### Pattern 4: Advanced Native Decorators with Autofac
The built-in container requires third-party libraries like Scrutor for decorators. Autofac does this natively and robustly, especially for open generics (e.g., MediatR handlers).

```csharp
builder.Host.ConfigureContainer<ContainerBuilder>(containerBuilder =>
{
    // Register the base implementation
    containerBuilder.RegisterType<OrderProcessor>()
                    .Named<IOrderProcessor>("processor");

    // Register the decorator, pointing to the base implementation
    containerBuilder.RegisterDecorator<IOrderProcessor>(
        (context, inner) => new ResilientOrderProcessor(inner),
        fromKey: "processor"
    );
    
    // Open generic decorator (e.g., logging for all IRequestHandler<T, R>)
    containerBuilder.RegisterGenericDecorator(
        typeof(LoggingRequestHandler<,>),
        typeof(IRequestHandler<,>)
    );
});
```

### Pattern 5: Property Injection in Action Filters (Autofac)
Action Filters in ASP.NET Core often suffer from injection issues because attributes cannot take services via constructors. Autofac allows injecting properties into the filter after instantiation.

```csharp
// 1. The Filter Attribute
public class AuditLogFilter : ActionFilterAttribute
{
    // Property injection used here
    public IAuditLogger Logger { get; set; } = null!;

    public override void OnActionExecuting(ActionExecutingContext context)
    {
        Logger.Log("Action executed");
    }
}

// 2. Wiring it up in Program.cs
builder.Host.ConfigureContainer<ContainerBuilder>(containerBuilder =>
{
    containerBuilder.RegisterType<AuditLogFilter>().PropertiesAutowired();
    
    // Crucial: Tell MVC to let Autofac resolve the filter
    builder.Services.AddControllers(options =>
    {
        options.Filters.AddService<AuditLogFilter>();
    });
});
```

### Pattern 6: Lamar's `AssertConfigurationIsValid()`
Lamar can traverse your entire application's dependency graph at startup and throw an aggregate exception if any class cannot be resolved (e.g., missing constructor dependency), rather than waiting for a runtime crash.

```csharp
var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    var container = app.Services.GetRequiredService<Lamar.IContainer>();
    
    // Will throw ContainerValidationException listing ALL missing dependencies
    container.AssertConfigurationIsValid();
    
    // Bonus: Output the generated C# code for debugging
    Console.WriteLine(container.GenerateCode());
}
```

## PART 4 — Gotchas & Anti-Patterns

### 1. The "Two Containers" Trap
**Anti-Pattern:** Registering services in `builder.Services` *after* calling `builder.Build()`, or trying to build a secondary `new ContainerBuilder()` and using it alongside the host's container.
**Gotcha:** Once `builder.Build()` executes, the adapter creates the finalized third-party container. Any further registrations to `IServiceCollection` are silently ignored because the factory has already consumed them.
**Resolution:** Always keep framework registrations in `builder.Services` and third-party registrations inside the `builder.Host.ConfigureContainer` callback. Never build a standalone container unless writing a console app without generic host.

### 2. Disposing the Container Prematurely
**Anti-Pattern:** Manually calling `.Dispose()` on an Autofac `ILifetimeScope` or `IContainer` injected into a scoped service.
**Gotcha:** The ASP.NET Core host manages the root container's lifecycle. If you manually dispose of the container or the HTTP request scope, subsequent requests or background services will crash with `ObjectDisposedException`.
**Resolution:** Let the framework handle scope disposal. If you create manual scopes (e.g., via `ILifetimeScope.BeginLifetimeScope()`), you are responsible for disposing *that specific scope*, usually via a `using` block, but never the injected root scope.

### 3. Misunderstanding the Request Scope Boundary
**Anti-Pattern:** Spawning a background thread (`Task.Run`) from an API controller and passing an injected `Scoped` service (like an EF `DbContext`) to it.
**Gotcha:** Autofac tracks the request scope. When the HTTP request completes, Autofac disposes the scope and all `Scoped` services within it. The background thread will crash mid-execution because its injected dependency was abruptly disposed by the web framework.
**Resolution:** Inject `IServiceProvider` or `ILifetimeScope` into the controller, pass it to the background thread, and create a brand new, independent scope using `BeginLifetimeScope()` inside the thread.

### 4. Over-using Property Injection
**Anti-Pattern:** Slapping `PropertiesAutowired()` on all controllers and services to avoid writing constructors.
**Gotcha:** Property injection hides dependencies, making classes harder to test (you have to remember to set the properties manually) and masking God-classes that have 15 dependencies. Furthermore, property injection bypasses standard `null` reference checks at instantiation.
**Resolution:** Use property injection *strictly* for framework limitations (e.g., Action Filters, SignalR hubs in older versions) or massive base classes where constructor chaining is untenable. Default to constructor injection 99% of the time.

### 5. Capturing Scoped Services in Autofac Singletons (Captive Dependencies)
**Anti-Pattern:** Registering a `SingleInstance()` caching service that takes an `InstancePerLifetimeScope()` `DbContext` in its constructor.
**Gotcha:** Autofac will resolve the `DbContext` once, when the singleton is instantiated. That `DbContext` is now captive. It will be shared across all HTTP requests (violating thread safety) and eventually disposed, crashing the app.
**Resolution:** Autofac provides captive dependency checking in .NET Core if enabled, but you should always inject `Func<Owned<T>>` or `IServiceScopeFactory` into singletons when they need to resolve transient or scoped dependencies on demand.

## PART 5 — Performance Implications

Replacing the built-in container introduces overhead, specifically in how the adapter bridges Microsoft's `IServiceProvider` with the third-party container.

| Operation | Built-In (MEDI) | Lamar | Autofac | Performance Impact / Notes |
| :--- | :--- | :--- | :--- | :--- |
| **Startup / Build Time** | Blazing Fast | Slow | Fast | Lamar invokes Roslyn to compile dynamic assemblies at startup. |
| **Transient Resolution** | Extremely Fast | Extremely Fast | Moderate | Lamar compiles direct `new()` calls. Autofac relies heavily on expressions/reflection. |
| **Scoped Resolution** | Extremely Fast | Fast | Moderate | Autofac incurs overhead managing tracking in complex scope hierarchies. |
| **Singleton Resolution** | Fastest (Cached) | Fastest (Cached) | Fastest (Cached) | All containers cache singletons efficiently; difference is negligible. |
| **IEnumerable`<T>` (Many)** | Fast | Very Fast | Moderate | Lamar optimizes array creation. Autofac has custom metadata wrappers slowing it slightly. |
| **Open Generic Resolution**| Fast | Fast | Moderate | Autofac supports complex covariance/contravariance rules, adding lookup cost. |
| **Memory Allocation** | Lowest | Low | Medium | Autofac tracks disposables aggressively in `LifetimeScope`, increasing GC pressure on heavy requests. |
| **Deep Object Graphs** | Fast | Blazing Fast | Moderate | Lamar's generated C# code excels at multi-level nested resolution graphs. |

### BenchmarkDotNet Code
This benchmark simulates resolving a controller with a moderately deep dependency graph.

```csharp
using BenchmarkDotNet.Attributes;
using Microsoft.Extensions.DependencyInjection;
using Autofac;
using Autofac.Extensions.DependencyInjection;
using Lamar.Microsoft.DependencyInjection;

[MemoryDiagnoser]
public class ContainerBenchmarks
{
    private IServiceProvider _mediProvider;
    private IServiceProvider _autofacProvider;
    private IServiceProvider _lamarProvider;

    [GlobalSetup]
    public void Setup()
    {
        var services = new ServiceCollection();
        // Register a deep graph: Controller -> Service -> Repo -> DbContext
        services.AddTransient<IDbContext, MyDbContext>();
        services.AddTransient<IRepository, MyRepository>();
        services.AddTransient<IService, MyService>();
        services.AddTransient<MyController>();

        // 1. Built-in
        _mediProvider = services.BuildServiceProvider();

        // 2. Autofac
        var builder = new ContainerBuilder();
        builder.Populate(services);
        _autofacProvider = new AutofacServiceProvider(builder.Build());

        // 3. Lamar
        var registry = new Lamar.ServiceRegistry(services);
        _lamarProvider = new Lamar.Container(registry);
    }

    [Benchmark(Baseline = true)]
    public MyController ResolveMEDI() => _mediProvider.GetRequiredService<MyController>();

    [Benchmark]
    public MyController ResolveAutofac() => _autofacProvider.GetRequiredService<MyController>();

    [Benchmark]
    public MyController ResolveLamar() => _lamarProvider.GetRequiredService<MyController>();
}
```

### When to Care
- **High-Throughput APIs (10k+ RPS):** Autofac's per-request allocation overhead and slower resolution *can* become a bottleneck. You will see garbage collection pauses due to tracking `IDisposable` instances in the scope. If performance is paramount but you need advanced DI, use **Lamar**.
- **Standard Enterprise APIs:** You will likely not notice the difference. A database query takes 10-50ms; Autofac takes 0.05ms to resolve the graph. The architectural benefits of Modules outweigh the microscopic CPU cost.

## PART 6 — Interview Arsenal

**Q: Why would you replace the built-in ASP.NET Core DI container?**
A: The built-in container is a basic conforming container. I would replace it to gain advanced features: auto-discovery of services (Assembly Scanning), explicit architectural boundaries (IoC Modules), native decorator support (without Scrutor), property injection (for attributes/filters), or robust container validation at startup (`AssertConfigurationIsValid`).

**Q: Explain how the adapter pattern works when adding Autofac to ASP.NET Core.**
A: The host builder uses `UseServiceProviderFactory`. This factory intercepts the framework's populated `IServiceCollection`, converts all Microsoft `ServiceDescriptor` objects into Autofac registrations, allows appending custom Autofac-specific configurations, and finally calls `.Build()` to generate an `IServiceProvider` backed entirely by Autofac's resolution engine.

**Q: How does Lamar differ architecturally from Autofac?**
A: Autofac uses reflection and expression trees at runtime to resolve dependencies. Lamar leverages the Roslyn compiler at startup to literally generate and compile C# source code that acts like handwritten `new` statements. This makes Lamar's startup slower but its per-request resolution significantly faster and closer to the built-in container.

**Q: What is a captive dependency, and how does Autofac handle it?**
A: A captive dependency occurs when a longer-lived component (Singleton) holds a reference to a shorter-lived component (Transient/Scoped), keeping it alive incorrectly and potentially causing thread-safety or memory leak issues. Autofac has built-in diagnostic analyzers that can be enabled to throw an exception if this graph mismatch is detected.

**Q: How do you handle `IDisposable` tracking in Autofac?**
A: By default, any resolved component implementing `IDisposable` is tracked by the `ILifetimeScope` that created it. When the scope (e.g., the HTTP request) is disposed, Autofac automatically calls `Dispose()` on all tracked objects. You can opt out using `.ExternallyOwned()` during registration if you intend to manage disposal manually.

## PART 7 — Decision Framework

| Scenario | Recommendation | Rationale |
| :--- | :--- | :--- |
| **Greenfield Microservice / Minimal API** | **Built-in DI** | Keep it simple. The built-in DI is fast, native, and well-understood. Use Scrutor if you need basic assembly scanning. |
| **Large Modular Monolith** | **Autofac** | Autofac `Module`s allow each bounded context to encapsulate its own dependency registrations, drastically reducing the size of `Program.cs` and decoupling projects. |
| **Heavy Decorator Usage (CQRS)** | **Autofac** | Built-in DI struggles with open generic decorators natively. Autofac makes intercepting and decorating handlers seamless. |
| **Legacy StructureMap Migration** | **Lamar** | Lamar is the direct successor to StructureMap. The APIs (`Registry`, `Scan`) are nearly identical, making migration trivial. |
| **Ultra-High Performance API needing advanced DI** | **Lamar** | You get the advanced features (scanning, assertions) without the runtime reflection cost of Autofac, thanks to Roslyn compilation. |
| **Blazor WebAssembly** | **Built-in DI** | Avoid Autofac/Lamar here. Custom containers add significant payload size to the WASM download and suffer performance hits in the constrained browser sandbox. |
