### Why This Matters at Production Scale

SQL injection is the highest-severity data breach vector in .NET applications not because EF Core is unsafe by default — it is not — but because teams intermittently drop to raw SQL for performance or stored procedure integration, and those escape hatches become injection points the moment string interpolation replaces parameterized input. A single injectable query in a payment API or patient record service can exfiltrate an entire database in under ten seconds.

## PART 1 — The Core Mental Model

### The Fundamental Rule

> **EF Core LINQ queries are always parameterized and cannot produce SQL injection; every raw SQL surface in EF Core (`FromSqlRaw`, `ExecuteSqlRaw`, `Database.ExecuteSqlRaw`) is injectable the moment you concatenate user input into the SQL string. The practical consequence is that the safety boundary in your application is not the ORM — it is the specific API method you call.**

### The Plain-Language Analogy

Think of your database as a bank vault. EF Core LINQ is the bank teller window: you hand a teller a slip that says "give me balance for account 42", and the teller looks it up safely — the slip is never parsed as an instruction, only as a value. Raw SQL with string interpolation is like handing the teller a note that says "give me balance for account 42, then wire all funds to account 99". The vault does not know where the instruction ends and the data begins, so it executes the whole note.

The analogy holds when you ask "but what about parameterized raw SQL?" — that is still the teller window, because you hand over the slip and the value separately: the teller processes the instruction template first, then slots in the value without re-parsing. It also holds under concurrent requests: every request hands its own note independently; injection in one request does not interfere with others, but each compromised note is its own breach.

## PART 2 — Deep Mechanics

### 2.1 — How EF Core LINQ Prevents SQL Injection

The safety of EF Core LINQ is not magic — it is a deliberate two-pass translation architecture that separates query structure from runtime values at compile time.

**Pipeline Position:**

```
HTTP Request
    │
    ▼
[Model Binding] — binds string orderId from route
    │
    ▼
[Your Controller/Endpoint] — passes orderId to service
    │
    ▼
[EF Core LINQ Translation] — builds Expression tree; values become @p0, @p1...
    │
    ▼
[SQL Server Driver / Npgsql] — executes parameterized SQL; SQL structure is fixed
    │
    ▼
[Database] — executes with parameters bound separately from SQL text
    │
    ▼
[HTTP Response] ← data returned, user input never touched SQL grammar
```

**How EF Core Internally Generates Parameterized SQL (approximate):**

```csharp
// EF Core internally (approximate) — ExpressionVisitor in QueryCompilationContext:
// When you write:
//   dbContext.Orders.Where(o => o.CustomerId == customerId && o.Status == status)
//
// EF Core's SqlExpressionFactory produces:
//   SELECT o.Id, o.Total, o.Status
//   FROM Orders AS o
//   WHERE o.CustomerId = @customerId AND o.Status = @status
//
// The SqlParameter objects are constructed SEPARATELY from the SQL string:
//   new SqlParameter("@customerId", customerId)   // value bound, not interpolated
//   new SqlParameter("@status", status)           // value bound, not interpolated
//
// The DbCommand.CommandText is fixed; DbCommand.Parameters holds the values.
// The database driver NEVER parses user values as SQL grammar.
```

**HTTP Wire Format (safe path):**

```http
// HTTP request:
GET /api/orders?customerId=42&status=pending HTTP/1.1
Authorization: Bearer eyJhbGci...

// EF Core executes this to the database (wire protocol, approximate):
// CommandText = "SELECT o.Id, o.Total FROM Orders o WHERE o.CustomerId = @p0 AND o.Status = @p1"
// Parameters  = [ @p0=42, @p1="pending" ]

// HTTP response:
HTTP/1.1 200 OK
Content-Type: application/json
[{"id":101,"total":299.99,"status":"pending"}]
```

**Runtime cost:** ~1 allocation for the `DbCommand` parameters collection; LINQ translation is cached per query shape — O(1) cache lookup after warm-up.

**The edge case that bites engineers:** EF Core caches the query plan by shape (the structure of the expression tree). The parameter values are substituted at execution time. This is why switching from `Where(o => o.Id == id)` to `Where(o => ids.Contains(o.Id))` generates different SQL — the collection size affects the SQL structure — but neither is injectable. The compiler prevents structural changes from user data.

### 2.2 — The Raw SQL Attack Surface: FromSqlRaw and ExecuteSqlRaw

`FromSqlRaw` and `ExecuteSqlRaw` accept a literal SQL string and an optional `params object[]` for parameters. They are injectable only when you assemble the SQL string using user data before passing it to the method.

**Pipeline Position:**

```
HTTP Request ──► [Model Binding] ──► [Your Handler]
                                          │
                          ┌───────────────┴───────────────────┐
                          │ WRONG: string concat before call   │
                          │ CORRECT: value passed as param     │
                          └───────────────┬───────────────────┘
                                          ▼
                            dbContext.Orders.FromSqlRaw(sql, params)
                                          │
                          ┌───────────────┴───────────────────┐
                          │ WRONG path: SQL grammar is         │
                          │ determined at runtime by user      │
                          │ CORRECT path: SQL grammar is       │
                          │ fixed; user value is a parameter   │
                          └───────────────┬───────────────────┘
                                          ▼
                                     [Database]
```

**The injection mechanism (wrong path):**

```csharp
// ASP.NET Core internally does NOT protect you here.
// FromSqlRaw passes the string verbatim to DbCommand.CommandText.
// If the string contains user input, the DB parses it as grammar.

// ATTACK SCENARIO — Payment API:
// HTTP request: GET /api/payments?status=active' OR '1'='1
// customerId = "active' OR '1'='1"   ← attacker-controlled

string sql = $"SELECT * FROM Payments WHERE CustomerId = '{customerId}'";
// sql is now: SELECT * FROM Payments WHERE CustomerId = 'active' OR '1'='1'
// The WHERE clause evaluates TRUE for every row.
// The attacker receives the entire Payments table.
```

**HTTP wire format (injection attack):**

```http
// HTTP request (attack):
GET /api/payments?status=active%27%20OR%20%271%27%3D%271 HTTP/1.1

// What EF Core sends to the database (decoded):
// SELECT * FROM Payments WHERE Status = 'active' OR '1'='1'
// This returns ALL payments, not just "active" ones.

// HTTP response (breach):
HTTP/1.1 200 OK
Content-Type: application/json
[{"id":1,"cardNumber":"4111...","amount":9999.99},{"id":2,...}]
// ↑ ENTIRE payments table returned to attacker
```

**The DROP TABLE attack:**

```http
// HTTP request:
POST /api/reports/generate HTTP/1.1
Content-Type: application/json
{"reportType":"summary; DROP TABLE Orders; --"}

// ExecuteSqlRaw generates:
// EXEC GenerateReport 'summary'; DROP TABLE Orders; --'
// SQL Server executes both statements. Orders table is deleted.
```

**Runtime cost of injection:** Zero additional cost to the attacker — the same code path executes; the database just interprets more grammar than intended.

**The edge case that bites engineers:** `FromSqlRaw` accepts `params object[]` — the parameterized overload. Engineers who have used `string.Format` for years reflexively write `FromSqlRaw("WHERE Id = {0}", userId)` thinking `{0}` is a format specifier. It is. EF Core replaces `{0}` with `@p0` and binds the value safely. But `$"WHERE Id = {userId}"` is C# interpolation — the value is baked into the string before EF Core ever sees it, and the `{0}` syntax is never invoked. This is the most common injection bug in EF Core codebases.

### 2.3 — EF Core 7+: FromSql and ExecuteSql with FormattableString

EF Core 7 introduced `FromSql` and `ExecuteSql` (without the `Raw` suffix) that accept `FormattableString` — a C# type that preserves the interpolation holes as separate parameters rather than substituting them into the string.

**How FormattableString safety works:**

```csharp
// C# compiler produces different types depending on context:

// This is a string (UNSAFE):
string sql = $"SELECT * FROM Orders WHERE Id = {orderId}";
// typeof(sql) == string — orderId is baked in, injection possible

// This is a FormattableString (SAFE when EF uses the FormattableString overload):
FormattableString sql = $"SELECT * FROM Orders WHERE Id = {orderId}";
// typeof(sql) == FormattableString
// sql.Format      == "SELECT * FROM Orders WHERE Id = {0}"
// sql.GetArguments() == [ orderId ]

// EF Core's FromSql(FormattableString) extracts .Format and .GetArguments()
// and creates SqlParameters — identical to explicit FromSqlRaw with params.
```

**The overload resolution trap:**

```csharp
// ⚠️ WRONG — this calls FromSqlRaw(string, params object[]) NOT FromSql(FormattableString):
var orders = dbContext.Orders.FromSqlRaw($"SELECT * FROM Orders WHERE Id = {orderId}");
// The $ interpolation is evaluated BEFORE the method call.
// By the time FromSqlRaw sees the string, orderId is already embedded.

// ✅ CORRECT — explicitly use the FormattableString overload:
var orders = dbContext.Orders.FromSql($"SELECT * FROM Orders WHERE Id = {orderId}");
// FormattableString is passed; EF Core extracts {0} = orderId as a parameter.
```

**Failure mode diagram — overload resolution:**

```
FormattableString overloads in EF Core:
  FromSql(FormattableString)         → SAFE   → EF extracts parameters
  FromSqlRaw(string, object[])       → SAFE   → you supply parameters explicitly
  FromSqlInterpolated(FormattableString) → SAFE (EF 6.x alias, still works)

DANGER ZONE — where string is inferred:
  FromSqlRaw($"...{userInput}...")   → UNSAFE → $ creates string, not FormattableString
  FromSqlRaw(string.Format("..."))   → UNSAFE → string, not FormattableString
  ExecuteSqlRaw($"...{userInput}...") → UNSAFE
```

**Runtime cost:** `FormattableString` allocates the argument array once per call — ~1 small heap allocation. Identical to explicit `params object[]`. No meaningful overhead.

### 2.4 — ADO.NET and Dapper: The Fully Manual Surface

Some teams bypass EF Core entirely for maximum performance queries — reporting endpoints, bulk operations, stored procedures with complex output. ADO.NET with `SqlCommand` and Dapper are the two common paths, and both expose the same injection risk.

**Pipeline Position:**

```
HTTP Request ──► [Endpoint] ──► [Service] ──► [SqlCommand] ──► [SQL Server]
                                                  │
                              ┌───────────────────┴──────────────────────┐
                              │ ADO.NET: SqlCommand.CommandText is the    │
                              │ raw SQL. Parameters are added to          │
                              │ SqlCommand.Parameters separately.         │
                              │ Dapper: QueryAsync(sql, new { Param })   │
                              │ maps anonymous object properties to       │
                              │ @Param placeholders — always safe.        │
                              └───────────────────┬──────────────────────┘
                                                  ▼
                              SAFE: CommandText never modified by user data
                              UNSAFE: CommandText += userInput (injection)
```

**ADO.NET safe pattern:**

```csharp
// ✅ CORRECT — ADO.NET parameterized:
using var cmd = new SqlCommand(
    "SELECT * FROM Shipments WHERE TrackingNumber = @trackingNumber AND CarrierId = @carrierId",
    connection);
cmd.Parameters.AddWithValue("@trackingNumber", trackingNumber);   // value, not grammar
cmd.Parameters.AddWithValue("@carrierId", carrierId);
```

**Dapper pattern:**

```csharp
// ✅ CORRECT — Dapper anonymous object maps to @placeholders:
var shipments = await connection.QueryAsync<Shipment>(
    "SELECT * FROM Shipments WHERE TrackingNumber = @TrackingNumber",
    new { TrackingNumber = trackingNumber });  // property name must match @placeholder

// ⚠️ WRONG — Dapper with string interpolation:
var sql = $"SELECT * FROM Shipments WHERE TrackingNumber = '{trackingNumber}'";
var shipments = await connection.QueryAsync<Shipment>(sql);  // INJECTABLE
```

**Dynamic ORDER BY — the hardest injection case:**

```
ORDER BY clauses CANNOT be parameterized. The column name and direction
must be literal SQL. This is the #1 source of injection in "otherwise safe"
EF Core applications.

User input: ?sortBy=Name&direction=ASC
Attack:     ?sortBy=Name; DROP TABLE Shipments; --
```

**Runtime cost:** ADO.NET `SqlCommand` with parameters: ~2 allocations (command + parameter collection). Dapper anonymous object: ~1 allocation for anonymous type + IL-generated mapper. Both are the correct cost for safe queries.

### 2.5 — The Dynamic Query Problem: Filters, Sorting, and Search

The real-world injection risk is rarely a naive `"SELECT * FROM X WHERE id = '" + id + "'"`. The hardest cases are dynamic queries: APIs where column selection, ordering, and filter predicates are user-controlled at runtime.

**Failure Mode Diagram — Dynamic ORDER BY:**

```
User input arrives:
  GET /api/orders?sortBy=TotalAmount&direction=DESC   ← legitimate
  GET /api/orders?sortBy=1;DROP TABLE Orders--&direction=ASC  ← attack

If you write:
  ORDER BY {sortBy} {direction}     ← injectable — column names can't be parameterized

HTTP consequence (wrong path):
  HTTP/1.1 200 OK                   ← or 500 if DROP succeeds and query fails
  (Orders table now deleted)

HTTP consequence (correct path — allowlist):
  HTTP/1.1 400 Bad Request          ← if sortBy not in allowlist
  {"type":"...","title":"Invalid sort column"}
```

**The allowlist pattern (correct):**

```csharp
// Pipeline position: controller/endpoint → before any SQL construction
// This validation runs BEFORE any database interaction.

private static readonly HashSet<string> AllowedSortColumns = new(StringComparer.OrdinalIgnoreCase)
{
    "TotalAmount", "CreatedAt", "Status", "CustomerId"
};

// In the handler (logistics tracking endpoint):
if (!AllowedSortColumns.Contains(request.SortBy))
{
    return TypedResults.BadRequest(new ProblemDetails
    {
        Title = "Invalid sort column",
        Detail = $"'{request.SortBy}' is not a sortable column.",
        Status = 400
    });
}

// Now safe to use in raw SQL:
var sql = $"SELECT * FROM Shipments ORDER BY {request.SortBy} {direction}";
// request.SortBy is validated; direction must also be validated ("ASC" or "DESC" only)
```

## PART 3 — Production Code Patterns

### Pattern 1: The LINQ Firewall — All User Input Stays in Parameters

Every filter an external API caller can specify goes through LINQ, never through string-building. This is the default EF Core posture for all user-facing query endpoints.

```csharp
// Domain: E-commerce order management service
// Pipeline position: after auth/authz middleware, inside endpoint handler
// EF Core generates fully parameterized SQL — no injection surface.

app.MapGet("/api/orders", async (
    [FromQuery] int? customerId,
    [FromQuery] string? status,
    [FromQuery] decimal? minTotal,
    [FromQuery] int page,
    [FromQuery] int pageSize,
    OrderDbContext db,
    ClaimsPrincipal user) =>
{
    // All filter values become @p0, @p1, @p2, etc. in EF-generated SQL.
    // No user value ever touches SQL grammar.
    var query = db.Orders
        .Where(o => customerId == null || o.CustomerId == customerId)
        .Where(o => status == null || o.Status == status)
        .Where(o => minTotal == null || o.TotalAmount >= minTotal)
        .OrderByDescending(o => o.CreatedAt)
        .Skip((page - 1) * pageSize)
        .Take(pageSize);

    var orders = await query
        .Select(o => new OrderSummaryDto(o.Id, o.TotalAmount, o.Status, o.CreatedAt))
        .ToListAsync();

    return TypedResults.Ok(orders);
})
.RequireAuthorization();

// HTTP wire effect (approximate EF-generated SQL):
// SELECT TOP(@pageSize) o.Id, o.TotalAmount, o.Status, o.CreatedAt
// FROM Orders o
// WHERE (@customerId IS NULL OR o.CustomerId = @customerId)
//   AND (@status IS NULL OR o.Status = @status)
//   AND (@minTotal IS NULL OR o.TotalAmount >= @minTotal)
// ORDER BY o.CreatedAt DESC
// OFFSET @skip ROWS
//
// ALL values are @parameters. Attacker-controlled strings cannot alter this structure.
```

### Pattern 2: The Safe Raw SQL Gateway — Stored Procedure with Explicit Parameters

When a stored procedure is required (DBA mandate, performance, complex business logic), use `FromSqlRaw` with explicit `SqlParameter` objects — never string interpolation.

```csharp
// Domain: Payment API — complex reporting stored procedure
// Anti-pattern first:

// ⚠️ WRONG: String interpolation before calling FromSqlRaw
public async Task<IReadOnlyList<PaymentReport>> GetPaymentReportWrong(
    string merchantId, DateTime from, DateTime to)
{
    // INJECTABLE: merchantId is embedded in the string before EF sees it
    var sql = $"EXEC sp_GetMerchantPaymentReport '{merchantId}', '{from:yyyy-MM-dd}', '{to:yyyy-MM-dd}'";
    return await _db.PaymentReports.FromSqlRaw(sql).ToListAsync();
}
// HTTP consequence (wrong path):
// merchantId = "x'; EXEC xp_cmdshell('net user hacker hacker123 /add'); --"
// SQL Server executes the stored procedure AND the xp_cmdshell command.

// ✅ CORRECT: Explicit SqlParameter objects
public async Task<IReadOnlyList<PaymentReport>> GetPaymentReport(
    string merchantId, DateTime from, DateTime to)
{
    // Why SqlParameter and not the {0} syntax: SqlParameter is explicit and clear.
    // The {0} syntax in FromSqlRaw also works but is easy to confuse with C# interpolation.
    var merchantIdParam = new SqlParameter("@merchantId", merchantId);
    var fromParam = new SqlParameter("@from", from);
    var toParam = new SqlParameter("@to", to);

    return await _db.PaymentReports
        .FromSqlRaw(
            "EXEC sp_GetMerchantPaymentReport @merchantId, @from, @to",
            merchantIdParam, fromParam, toParam)
        .AsNoTracking()
        .ToListAsync();
}

// HTTP wire effect (correct path):
// CommandText = "EXEC sp_GetMerchantPaymentReport @merchantId, @from, @to"
// Parameters  = [ @merchantId='acme-corp', @from=2026-01-01, @to=2026-03-31 ]
// Attacker-controlled merchantId cannot inject SQL — it is always a value, never grammar.
```

### Pattern 3: The FormattableString Gateway — EF Core 7+ Safe Interpolation

Use `FromSql` (not `FromSqlRaw`) with `$""` when the query is constructed inline and the FormattableString overload resolves correctly. This is the ergonomic safe path.

```csharp
// Domain: Healthcare patient portal — patient record search
// Requires EF Core 7+ for the FromSql(FormattableString) overload.

public async Task<PatientRecord?> GetPatientRecord(int patientId, string facilityCode)
{
    // ✅ CORRECT: FromSql accepts FormattableString — holes become @p0, @p1 parameters
    // The $ string is NOT evaluated to a plain string before this call.
    // C# passes it as FormattableString because the method signature demands it.
    return await _db.PatientRecords
        .FromSql($"""
            SELECT pr.*
            FROM PatientRecords pr
            INNER JOIN FacilityAccess fa ON fa.PatientId = pr.Id
            WHERE pr.Id = {patientId}
              AND fa.FacilityCode = {facilityCode}
              AND pr.IsActive = 1
            """)
        .AsNoTracking()
        .FirstOrDefaultAsync();
}

// ⚠️ CRITICAL WARNING: The overload you call determines safety.
// FromSql($"...{patientId}...")         → SAFE  (FormattableString overload)
// FromSqlRaw($"...{patientId}...")      → UNSAFE (string overload, $ already resolved)
// FromSqlRaw("...{0}...", patientId)    → SAFE  (explicit positional parameter)

// HTTP wire effect (safe path):
// CommandText = "SELECT pr.* FROM PatientRecords pr WHERE pr.Id = @p0 AND fa.FacilityCode = @p1 ..."
// Parameters  = [ @p0=42, @p1="FAC-001" ]
```

### Pattern 4: The Dynamic Sort Allowlist — The Hardest Case Done Right

User-controlled ordering is the most common source of injection in applications that otherwise use EF Core correctly. This pattern shows the complete safe implementation.

```csharp
// Domain: Logistics shipment tracker — sortable, filterable shipment list
// Why this is hard: ORDER BY column names cannot be SQL parameters.
// They must be literal SQL, validated against an allowlist before use.

public static class ShipmentSortColumns
{
    // The allowlist is the security boundary.
    // If a column is not here, it cannot be sorted by, period.
    // Use a dictionary to map safe API names to actual column names (defense in depth).
    public static readonly IReadOnlyDictionary<string, string> Allowed =
        new Dictionary<string, string>(StringComparer.OrdinalIgnoreCase)
        {
            ["estimatedDelivery"] = "EstimatedDeliveryDate",
            ["createdAt"]         = "CreatedAt",
            ["weight"]            = "WeightKg",
            ["carrier"]           = "CarrierCode",
        };
}

app.MapGet("/api/shipments", async (
    [FromQuery] string? sortBy,
    [FromQuery] string? direction,
    ShipmentDbContext db) =>
{
    // Step 1: Validate sort direction — only "asc" or "desc" allowed.
    // Never put direction directly in SQL without this check.
    var sqlDirection = direction?.ToUpperInvariant() == "DESC" ? "DESC" : "ASC";

    // Step 2: Validate sort column against allowlist.
    // Map from user-friendly API name to actual column name.
    if (sortBy != null && !ShipmentSortColumns.Allowed.TryGetValue(sortBy, out var sqlColumn))
    {
        return Results.Problem(
            title: "Invalid sort field",
            detail: $"'{sortBy}' is not a valid sort field. Valid values: {string.Join(", ", ShipmentSortColumns.Allowed.Keys)}",
            statusCode: 400);
    }

    // Step 3: Column name is now either null (use default) or a validated literal.
    // User-supplied value is NEVER embedded directly.
    sqlColumn ??= "CreatedAt";

    // Step 4: Combine LINQ (for safe parameterized filtering) with validated literal column.
    // EF Core does not support parameterized ORDER BY — we use raw SQL for that part only.
    var shipments = await db.Shipments
        .Where(s => s.IsActive)
        .FromSql($"SELECT * FROM Shipments WHERE IsActive = 1 ORDER BY {sqlColumn} {sqlDirection}")
        // ↑ sqlColumn and sqlDirection are allowlist-validated literals, not user input.
        .AsNoTracking()
        .ToListAsync();

    return TypedResults.Ok(shipments);
});

// HTTP wire effect:
// GET /api/shipments?sortBy=weight&direction=DESC
// CommandText = "SELECT * FROM Shipments WHERE IsActive = 1 ORDER BY WeightKg DESC"
// (No parameters needed — sqlColumn and sqlDirection are hardcoded-safe values)

// HTTP consequence (invalid sortBy):
// GET /api/shipments?sortBy=1;DROP TABLE Shipments--
// HTTP/1.1 400 Bad Request
// {"title":"Invalid sort field","detail":"'1;DROP TABLE Shipments--' is not a valid sort field..."}
```

### Pattern 5: The Dapper Safe Pattern with QueryTag for Tracing

When using Dapper for high-performance bulk reads, always use the anonymous object parameter syntax. Add `QueryTag` for observability (correlate app traces to DB query logs).

```csharp
// Domain: Inventory management service — high-volume SKU availability check
// Why Dapper here: bulk lookup of 500 SKUs; EF Core's IN clause with large collections
//                 generates suboptimal SQL for some query shapes.

public class InventoryRepository
{
    private readonly IDbConnection _connection;

    public InventoryRepository(IDbConnection connection)
    {
        _connection = connection;
    }

    // ✅ CORRECT: Dapper anonymous object maps property names to @placeholders
    public async Task<IReadOnlyList<InventoryItem>> GetAvailability(
        IEnumerable<string> skuCodes, string warehouseId)
    {
        // Why a table-valued parameter instead of IN (@skuList): prevents injection
        // AND avoids the "query plan explosion" problem with large IN lists.
        var skuTable = new DataTable();
        skuTable.Columns.Add("Sku", typeof(string));
        foreach (var sku in skuCodes) skuTable.Rows.Add(sku);

        // ✅ CORRECT: TVP + parameterized warehouseId
        var results = await _connection.QueryAsync<InventoryItem>(
            """
            SELECT i.SkuCode, i.QuantityOnHand, i.ReservedQuantity
            FROM Inventory i
            INNER JOIN @skus s ON s.Sku = i.SkuCode
            WHERE i.WarehouseId = @warehouseId
              AND i.IsActive = 1
            """,
            new
            {
                skus = skuTable.AsTableValuedParameter("dbo.SkuList"),
                warehouseId = warehouseId    // ← bound as @warehouseId, never grammar
            });

        return results.AsList();
    }
}

// ⚠️ WRONG — Dapper with string join (injectable and causes plan pollution):
// var inClause = string.Join(",", skuCodes.Select(s => $"'{s}'"));
// var sql = $"SELECT * FROM Inventory WHERE SkuCode IN ({inClause})";
// INJECTABLE if sku contains: ', 'x') OR 1=1--

// HTTP consequence (correct path): parameterized batch lookup, no injection surface
// HTTP consequence (wrong path): attacker with sku="x') OR 1=1; DROP TABLE Inventory--"
//   causes data breach or table deletion depending on DB permissions
```

### Pattern 7: The Exception-to-4xx Firewall for Database Errors

When injection attempts fail (because you parameterized), the database sometimes returns errors (invalid data type cast, constraint violation). Never leak database error details to clients.

```csharp
// Domain: Order management — preventing database error information leakage
// Pipeline position: exception handling middleware (must be first in pipeline)

// In Program.cs — first middleware registered:
app.UseExceptionHandler(exceptionHandler =>
{
    exceptionHandler.Run(async context =>
    {
        var exceptionFeature = context.Features.Get<IExceptionHandlerFeature>();
        var exception = exceptionFeature?.Error;

        // Log the full exception internally (includes SQL error details)
        var logger = context.RequestServices.GetRequiredService<ILogger<Program>>();
        logger.LogError(exception, "Unhandled exception for {RequestPath}", context.Request.Path);

        // Return a generic error to the client — NEVER include exception.Message
        // for database errors (it may contain table names, column names, query fragments)
        context.Response.StatusCode = 500;
        context.Response.ContentType = "application/problem+json";
        await context.Response.WriteAsJsonAsync(new ProblemDetails
        {
            Type = "https://tools.ietf.org/html/rfc7807",
            Title = "An unexpected error occurred",
            Status = 500,
            // Why NOT exception.Message here:
            // SqlException.Message: "Invalid column name 'Orders'." reveals schema.
            // SqlException for injection attempt may contain user's payload in message.
            Detail = "Please contact support with your request ID.",
            Instance = context.Request.Path
        });
    });
});

// HTTP consequence (correct path — injection attempt that fails):
// Attacker gets:
// HTTP/1.1 500 Internal Server Error
// {"title":"An unexpected error occurred","detail":"Please contact support..."}
// NOT:
// {"detail":"Incorrect syntax near 'OR 1=1'"}  ← reveals that injection was attempted
```

## PART 4 — Gotchas & Anti-Patterns

### Gotcha 1: FromSqlRaw with C# Interpolation Is Not the Same as FromSql

The most common EF Core injection bug comes from engineers who know about `FromSqlRaw`'s safe `{0}` parameter syntax but use C# string interpolation (`$""`) instead of the positional placeholder.

```csharp
// ⚠️ WRONG CODE:
// Engineer thinks: "I'm using EF Core's parameterization"
// Reality: C# interpolates the string before the method call
var orders = dbContext.Orders
    .FromSqlRaw($"SELECT * FROM Orders WHERE CustomerId = {customerId}");
// customerId is embedded in the string by C# before FromSqlRaw executes.
// FromSqlRaw receives: "SELECT * FROM Orders WHERE CustomerId = 42" (if customerId=42)
// If customerId = "42 OR 1=1": "SELECT * FROM Orders WHERE CustomerId = 42 OR 1=1"

// HTTP consequence (wrong path):
// GET /api/orders?customerId=42%20OR%201%3D1
// HTTP/1.1 200 OK
// [all orders from all customers returned]

// ✅ CORRECT CODE:
var orders = dbContext.Orders
    .FromSql($"SELECT * FROM Orders WHERE CustomerId = {customerId}");
// This calls the FormattableString overload. EF Core extracts {0}=customerId.
// OR:
var orders = dbContext.Orders
    .FromSqlRaw("SELECT * FROM Orders WHERE CustomerId = {0}", customerId);

// HTTP consequence (correct path):
// CommandText = "SELECT * FROM Orders WHERE CustomerId = @p0"
// Parameters  = [ @p0=42 ]

// WHY: The C# compiler resolves method overloads at compile time.
// FromSqlRaw has two overloads: (string, params object[]) and nothing for FormattableString.
// When you write FromSqlRaw($"...{var}..."), the compiler sees a string — not FormattableString.
// FromSql has a FormattableString overload. The compiler correctly routes to it.
```

---

### Gotcha 2: Dynamic Column Names in ORDER BY Bypass All EF Core Safety

Parameterization only works for values, not for SQL structural elements (column names, table names, direction keywords). Putting `{sortColumn}` in an EF Core `FromSql` expression does not parameterize the column name — it injects it verbatim.

```csharp
// ⚠️ WRONG CODE:
// Engineer thinks: "I'm using FromSql which is safe"
// Reality: column names are never parameters — they are always literal SQL
string sortColumn = request.SortBy; // user-controlled
var payments = await dbContext.Payments
    .FromSql($"SELECT * FROM Payments ORDER BY {sortColumn}")  // INJECTABLE
    .ToListAsync();
// sortColumn = "Amount; DROP TABLE Payments; --"
// SQL: "SELECT * FROM Payments ORDER BY Amount; DROP TABLE Payments; --"

// HTTP consequence (wrong path):
// POST /api/payments/list with body {"sortBy":"Amount; DROP TABLE Payments; --"}
// HTTP/1.1 500 Internal Server Error (Payments table deleted)
// OR: attacker uses "1" which returns rows ordered by first column — leaks order info

// ✅ CORRECT CODE:
private static readonly HashSet<string> _allowedColumns =
    new(StringComparer.OrdinalIgnoreCase) { "Amount", "CreatedAt", "Status" };

if (!_allowedColumns.Contains(request.SortBy))
    return TypedResults.BadRequest("Invalid sort column");

// Now safe: sortColumn is a validated literal, not user-controlled grammar
var payments = await dbContext.Payments
    .FromSql($"SELECT * FROM Payments ORDER BY {request.SortBy}")
    .ToListAsync();

// HTTP consequence (correct path):
// Invalid sort column → HTTP/1.1 400 Bad Request
// Valid sort column   → parameterized query with safe literal ORDER BY
// WHY: SQL parameterization is a feature of values in WHERE/SELECT.
// Column names in ORDER BY, GROUP BY, table names, and schema names are SQL grammar —
// they must be whitelisted literals, never user-controlled strings.
```

---

### Gotcha 3: EF Core's `ExecuteSql` with Raw String Literal in .NET 8+

In .NET 8, C#'s raw string literals (`"""..."""`) look safe but behave identically to `$""` for injection purposes. The danger is amplified because raw strings are often used for multi-line SQL where injection points are visually buried.

```csharp
// ⚠️ WRONG CODE:
// Engineer writes clean, readable SQL with raw string literal
// and forgets that ExecuteSqlRaw does NOT accept FormattableString
await dbContext.Database.ExecuteSqlRaw(
    $"""
    UPDATE Orders
    SET Status = '{newStatus}'
    WHERE CustomerId = {customerId}
    """);
// newStatus = "cancelled'; UPDATE Orders SET Status = 'cancelled' WHERE '1' = '1"
// Both statements execute. All orders cancelled.

// HTTP consequence (wrong path):
// PATCH /api/orders/bulk with {"newStatus":"cancelled'..."}
// All orders in the database set to "cancelled"

// ✅ CORRECT CODE:
await dbContext.Database.ExecuteSql(
    $"""
    UPDATE Orders
    SET Status = {newStatus}
    WHERE CustomerId = {customerId}
    """);
// FormattableString overload — newStatus and customerId become @p0, @p1

// HTTP consequence (correct path):
// CommandText = "UPDATE Orders SET Status = @p0 WHERE CustomerId = @p1"
// Parameters  = [ @p0="pending", @p1=42 ]

// WHY: In .NET 8, ExecuteSql (without Raw) accepts FormattableString and parameterizes
// the holes. ExecuteSqlRaw (with Raw) always accepts string — C# interpolation runs first.
// The method name suffix "Raw" is your signal: "Raw" = no automatic parameterization.
```

### Gotcha 4: Logging and Error Responses That Leak the Injected Payload

Experienced engineers parameterize their queries correctly but then write the exception message (which may contain the attacker's payload echoed back) into the HTTP response.

```csharp
// ⚠️ WRONG CODE:
// Looks like safe error handling, but leaks schema and possibly the attack payload
try
{
    var result = await dbContext.Invoices.Where(i => i.Id == invoiceId).FirstAsync();
    return TypedResults.Ok(result);
}
catch (SqlException ex)
{
    // SqlException.Message for an error from parameterized query might say:
    // "The conversion of a nvarchar value '1 OR 1=1' to data type int failed."
    // This CONFIRMS to the attacker that injection was attempted and reveals the column type.
    return TypedResults.Problem(ex.Message);  // ⚠️ leaks internal details
}

// HTTP consequence (wrong path):
// HTTP/1.1 500 Internal Server Error
// {"detail":"The conversion of a nvarchar value '1 OR 1=1' to data type int failed."}
// Attacker learns: column is int, injection attempt reached SQL Server, parameter name

// ✅ CORRECT CODE:
catch (SqlException ex)
{
    _logger.LogError(ex, "Database error for invoice {InvoiceId}", invoiceId); // log internally
    return TypedResults.Problem(
        title: "An internal error occurred",
        statusCode: 500); // generic — reveals nothing
}

// HTTP consequence (correct path):
// HTTP/1.1 500 Internal Server Error
// {"title":"An internal error occurred","status":500}
// WHY: SqlException.Message contains SQL Server's error text, which may include
// the parameter value (if type conversion failed) or schema information.
// Logging is correct; returning to client is an information disclosure vulnerability.
```

### Gotcha 5: Dapper's QueryMultiple with User-Controlled SQL Shape

Dapper's `QueryMultiple` accepts a single SQL string for multiple result sets. Teams using it for reporting dashboards sometimes build the SQL by concatenating user-selected sections.

```csharp
// ⚠️ WRONG CODE:
// Dashboard API lets users select which metrics to load
// Engineer builds the SQL by concatenating "safe" blocks based on user selection
public async Task<DashboardData> GetDashboard(string[] selectedMetrics)
{
    var sqlParts = selectedMetrics.Select(m => m switch
    {
        "revenue"    => "SELECT SUM(Amount) FROM Payments;",
        "orders"     => "SELECT COUNT(*) FROM Orders;",
        _ => $"SELECT * FROM {m};"  // ← INJECTABLE — m is user-controlled table name
    });
    var sql = string.Join("\n", sqlParts);
    // If selectedMetrics = ["revenue", "Users; DROP TABLE Payments; --"]
    // sql becomes: "SELECT SUM(Amount) FROM Payments;\nSELECT * FROM Users; DROP TABLE Payments; --;"

    using var multi = await _connection.QueryMultipleAsync(sql);
    // ...
}

// HTTP consequence (wrong path):
// POST /api/dashboard with {"metrics":["revenue","Users; DROP TABLE Payments; --"]}
// HTTP/1.1 500 (or 200 if DROP succeeded before query failed)

// ✅ CORRECT CODE:
private static readonly Dictionary<string, string> _metricQueries =
    new(StringComparer.OrdinalIgnoreCase)
    {
        ["revenue"] = "SELECT SUM(Amount) AS Value FROM Payments;",
        ["orders"]  = "SELECT COUNT(*) AS Value FROM Orders;",
        ["users"]   = "SELECT COUNT(*) AS Value FROM Users;"
    };

public async Task<DashboardData> GetDashboard(string[] selectedMetrics)
{
    // Allowlist: only known, hardcoded SQL fragments are used.
    var validatedParts = selectedMetrics
        .Where(m => _metricQueries.ContainsKey(m))
        .Select(m => _metricQueries[m])
        .ToList();

    if (validatedParts.Count == 0)
        throw new ArgumentException("No valid metrics selected");

    var sql = string.Join("\n", validatedParts);
    // sql is now composed of hardcoded strings only — user input only controlled WHICH
    // hardcoded block was selected, never the SQL text itself.
    using var multi = await _connection.QueryMultipleAsync(sql);
    // ...
}

// HTTP consequence (correct path):
// Unknown metric silently excluded; only allowlisted SQL blocks used.
// WHY: User input must never determine SQL grammar — only which pre-validated
// grammar block to include. The distinction: user selects from a menu (safe),
// vs user provides the menu item text (unsafe).
```

### 5.3 — When to Care / When to Ignore

**When this costs you:**

The parameterization pattern itself has negligible overhead — under 0.2ms per request for the extra parameter objects. What actually costs you is getting injection prevention wrong:

- **Schema information leakage** in error responses — zero latency cost but causes a security incident that costs the company far more than any performance optimization.
- **Large IN clauses** built from user input — even if parameterized, 500-element IN clauses cause query plan cache pollution, forcing SQL Server to recompile for each unique parameter count. Use table-valued parameters instead.
- **Repeated LINQ expression tree compilation** — EF Core caches by shape but not by closure variable names; using `string.Format` to build LINQ predicates inside loops defeats the query cache.
- **Missing `AsNoTracking()`** on read-only endpoints — change tracking allocates the entity graph and has ~0.3ms overhead per query. Every read-only endpoint that uses EF Core for security-safe queries should also be using `AsNoTracking()`.

**When this doesn't matter:**

- **Admin-only internal endpoints** — still parameterize for correctness, but the security exposure is lower and the performance difference is zero.
- **Single-record lookups by primary key** — `FindAsync(id)` uses EF Core's identity map; the parameterization is automatic and the overhead of the safety check is unmeasurable.
- **Low-traffic management APIs** (`< 10 req/s`) — the overhead of parameterization and allowlist validation is measured in microseconds and is irrelevant at this traffic level.
- **CI/CD and test environments** — use real databases with real parameterization in integration tests; do not accept "it works in test" as evidence of injection safety.

## PART 6 — Interview Arsenal

### A. The Question Bank

**Question 1: "Is EF Core immune to SQL injection?"**

**Average Answer:** "Yes, EF Core uses parameterized queries so SQL injection isn't possible."

**Why That's Insufficient:** It ignores that EF Core has multiple raw SQL APIs that are not automatically parameterized, and conflates "LINQ is safe" with "all EF Core is safe."

> **Great Answer:** "LINQ queries in EF Core are immune because the expression tree compilation and SQL generation happen separately from value binding — user-controlled values always become `@p0`, `@p1` parameters, and the SQL grammar is fixed at compile time. However, EF Core also exposes `FromSqlRaw` and `ExecuteSqlRaw`, and those are injectable the moment you concatenate or interpolate user input into the SQL string before the call. I've seen teams get burned by writing `FromSqlRaw($"... {userInput} ...")` — the C# `$` interpolation runs before the method call, so EF Core receives a plain string with the user's value already embedded. The safe alternative is `FromSql` with `$""`, which EF Core 7+ routes to the `FormattableString` overload and extracts the values as separate parameters. The security boundary is not the ORM — it's the specific API method you call."

---

**Question 2: "Why can't you just parameterize ORDER BY columns?"**

**Average Answer:** "You can't use parameters for column names in SQL, so you have to use string concatenation."

**Why That's Insufficient:** It identifies the constraint but doesn't explain why it exists or how to safely handle it.

> **Great Answer:** "Parameterization works because the database separates SQL grammar from data: the query plan is compiled against the SQL text, and then parameter values are bound separately at execution time. Column names are structural SQL — they're part of the grammar, not values, so the database parser needs them to understand the query shape. You can't pass a column name as a parameter for the same reason you can't pass a keyword as a parameter. This means dynamic ORDER BY is genuinely a different security problem than dynamic WHERE values. The correct solution is an allowlist — a `HashSet<string>` or dictionary in your application code that maps user-facing sort field names to actual column names. The user's input controls which pre-validated literal gets used, never the literal itself. This is the pattern I enforce in code review: any raw SQL that includes a sort column must have a corresponding allowlist validation above it."

---

**Question 3: "What's the difference between `FromSqlRaw` and `FromSql` in EF Core 7+?"**

**Average Answer:** "FromSql is newer and handles interpolation automatically."

**Why That's Insufficient:** It doesn't explain the mechanism — the `FormattableString` type — or why it matters for security, and doesn't address the overload resolution trap.

> **Great Answer:** "The key difference is the method signature. `FromSqlRaw` accepts `string` — and that's it. If you write `FromSqlRaw($"...{variable}...")`, C# evaluates the interpolation before the call and the method receives a plain string with the value already embedded. `FromSql` (introduced in EF Core 7) has an overload that accepts `FormattableString`, which is a C# type that preserves the interpolation holes as separate arguments rather than substituting them into the string. EF Core extracts those arguments and creates proper `SqlParameter` objects. The practical rule I use is: `FromSqlRaw` with `{0}` positional placeholders is safe; `FromSqlRaw` with `$""` is injectable. `FromSql` with `$""` is safe because the compiler routes it to the `FormattableString` overload. The method name 'Raw' is the signal — it means 'I give you the string exactly as you give it to me'."

---

**Question 4: "How do you prevent SQL injection in a reporting API where the columns and filters are fully dynamic?"**

**Average Answer:** "Use an ORM like EF Core to avoid writing raw SQL."

**Why That's Insufficient:** Reporting queries almost always require dynamic SQL that LINQ cannot express, so "use an ORM" is not actionable advice for this scenario.

> **Great Answer:** "Dynamic reporting is the hardest case because LINQ's expression trees don't cover all SQL constructs — grouping by user-selected columns, conditional aggregations, pivot-style queries. My pattern is to separate the dynamic parts into two categories: values (which can be parameterized) and SQL structural elements like column names and sort direction (which cannot). For values, I always use parameterization — Dapper's anonymous object syntax or `FromSqlRaw` with explicit `SqlParameter` objects. For structural elements, I use an allowlist dictionary that maps client-facing names to actual database identifiers. The user's input selects a key from the dictionary; the dictionary value is a hardcoded string that goes into the SQL. If the client sends a column name that's not in the allowlist, the API returns a 400 immediately before any SQL is constructed. I also add a code review policy: any file containing `FromSqlRaw` or `ExecuteSqlRaw` requires a second approval from a senior engineer, because those are the only places injection can enter."

---

### B. Trick Questions

**Trick 1: "Can you get SQL injection from a route parameter like `/api/orders/{orderId}` where `orderId` is an `int`?"**

The trap is thinking "it's a route parameter, so it's safe." The correct answer: **No**, because ASP.NET Core model binding validates that the route value can be parsed as `int` before it reaches your handler. If `orderId` is declared as `int`, non-integer values (like `42; DROP TABLE Orders`) cause a 400 Bad Request at the model binding layer, before your code executes. The injection never reaches the database layer. This is why using strongly-typed parameters (int, Guid, enum) is a defense-in-depth measure — even if your query had an injection point, the attacker can't inject a non-integer value through an int parameter.

**Trick 2: "Your team says you're protected because you validate all input with DataAnnotations before it reaches the database. Is that sufficient?"**

The trap is thinking input validation replaces parameterization. The correct answer: **No**. Input validation and parameterization are independent, complementary layers. DataAnnotations validates shape and format (email looks like an email, length is within bounds) but does not prevent injection. A 100-character string that passes `[StringLength(100)]` can still contain SQL injection characters. Parameterization prevents injection regardless of what characters the string contains. The correct posture is: validate input (reject malformed data early) AND parameterize all queries (structural safety for whatever passes validation).

**Trick 3: "A developer uses `EF.Functions.Like` for a search query: `Where(o => EF.Functions.Like(o.Description, $"%{searchTerm}%"))`. Is this safe?"**

The trap: `EF.Functions.Like` is called inside an EF Core LINQ expression, so the developer assumes EF Core parameterizes `searchTerm`. The correct answer: **Yes, it is safe** — `searchTerm` is passed as a parameter `@p0` in the SQL `LIKE '%' + @p0 + '%'` pattern. EF Core handles the concatenation in SQL, not in C#. However, this does NOT protect against LIKE wildcard abuse — if `searchTerm = "%"`, it matches everything and becomes a performance attack. Parameterization prevents injection; input length and wildcard limits prevent abuse.

**Trick 4: "You call `dbContext.Database.ExecuteSqlRaw("EXEC sp_GetUserDetails @userId", userId)` where `userId` is a `string`. Is this safe?"**

The trap is the ambiguity in the API: `ExecuteSqlRaw(string, params object[])`. The correct answer: **Yes, this specific call is safe** — `userId` is passed as `params object[]`, not interpolated into the string. EF Core creates a `SqlParameter("@userId", userId)`. The confusion arises because both `ExecuteSqlRaw("EXEC sp ... @p", param)` (safe) and `ExecuteSqlRaw($"EXEC sp ... '{param}'")` (unsafe) are valid C# syntax — only the pattern of use determines safety.

---

### C. Red Flags to Avoid

1. **"EF Core prevents SQL injection"** — this is dangerously incomplete. EF Core LINQ prevents it; raw SQL APIs do not automatically.
    
2. **"We validate all user input so we're safe"** — validation and parameterization are independent layers. Validation rejects bad data; parameterization prevents injection from any data that passes validation. Saying one replaces the other will get you failed.
    
3. **"ORDER BY can be parameterized, you just need the right syntax"** — it cannot. Column names and SQL structural elements cannot be parameterized in any SQL database. An interviewer who is a security-aware engineer knows this and will catch you.
    
4. **"We use stored procedures so we're safe"** — stored procedures called with parameterized inputs are safe. Stored procedures that internally concatenate strings and execute dynamic SQL have the same injection risk as any other dynamic SQL.
    
5. **"The WAF handles SQL injection"** — WAF rules catch known patterns but can be bypassed by encoding, obfuscation, or novel injection techniques. Parameterization is the only reliable defense.
    
6. **"We encode/escape the user input before putting it in the SQL"** — escaping is fragile and database-specific. It fails on Unicode edge cases, encoding differences, and novel injection vectors. Parameterization is always correct; escaping is always fragile.
    
7. **"I tested it with SQLMap and it didn't find anything"** — automated scanners miss application-logic injection paths, time-based blind injection in some configurations, and injection in non-obvious parameters like headers and cookies. A clean scan is not a guarantee.

### B. Code Puzzles

**Puzzle 1 — What is the HTTP response?**

```csharp
app.MapGet("/api/products/search", async (string query, ProductDbContext db) =>
{
    var results = await db.Products
        .FromSqlRaw($"SELECT * FROM Products WHERE Name LIKE '%{query}%'")
        .ToListAsync();
    return TypedResults.Ok(results);
});
// Request: GET /api/products/search?query=Widget%27%3B%20DROP%20TABLE%20Products%3B%20--
```

<details> <summary>Answer</summary>

**HTTP consequence:** This is a SQL injection vulnerability. The `$""` C# interpolation runs before `FromSqlRaw` executes, embedding the decoded value `Widget'; DROP TABLE Products; --` directly into the SQL string.

The SQL sent to the database is:

```sql
SELECT * FROM Products WHERE Name LIKE '%Widget'; DROP TABLE Products; --'%'
```

SQL Server executes both statements. The `Products` table is deleted. The HTTP response is likely `HTTP/1.1 500 Internal Server Error` (because the subsequent `ToListAsync()` fails if the table was dropped, or the query returns an empty/incorrect result set).

**The bug:** `FromSqlRaw` must never receive a C# `$""` string that contains user input. The correct fix is `FromSql($"SELECT * FROM Products WHERE Name LIKE '%{query}%'")` (FormattableString overload, EF Core 7+) or `FromSqlRaw("SELECT * FROM Products WHERE Name LIKE {0}", $"%{query}%")`.

</details>

**Puzzle 2 — Is this safe?**

```csharp
public async Task<List<Order>> GetOrdersByStatus(string[] allowedStatuses, int customerId)
{
    var statusList = string.Join(",", allowedStatuses.Select(s => $"'{s}'"));
    var sql = $"SELECT * FROM Orders WHERE CustomerId = @customerId AND Status IN ({statusList})";
    return await _db.Orders.FromSqlRaw(sql, new SqlParameter("@customerId", customerId)).ToListAsync();
}
// Called with: allowedStatuses = ["active", "pending"], customerId = 42
```

<details> <summary>Answer</summary>

**This is UNSAFE.** `customerId` is correctly parameterized via `SqlParameter`. However, `allowedStatuses` is concatenated into the SQL string before the call. Even though the caller controls `allowedStatuses` internally (it may look "safe" because it's not directly from user input), this pattern is:

1. **Potentially injectable** if `allowedStatuses` ever contains user-provided values.
2. **Structurally wrong** because it causes SQL Server query plan cache pollution — a new query plan is generated for every unique combination of statuses and their lengths.

The SQL becomes:

```sql
SELECT * FROM Orders WHERE CustomerId = @customerId AND Status IN ('active','pending')
```

**The correct pattern** for IN clauses with EF Core: use LINQ `.Where(o => statuses.Contains(o.Status))` which EF Core translates to a properly parameterized `IN (@p0, @p1, ...)`. For Dapper, use a table-valued parameter.

**HTTP consequence:** Currently works but is fragile. If this method is ever called with user-provided statuses (a future code change), it becomes injectable. The correct fix eliminates the structural risk now.

</details>

**Puzzle 3 — What status code does the client receive?**

```csharp
app.MapGet("/api/orders/{orderId}", async (
    int orderId,          // declared as int
    OrderDbContext db) =>
{
    var order = await db.Orders.FindAsync(orderId);
    return order is null ? TypedResults.NotFound() : TypedResults.Ok(order);
});
// Request: GET /api/orders/42;DROP TABLE Orders-- HTTP/1.1
```

<details> <summary>Answer</summary>

**HTTP/1.1 400 Bad Request**

The route parameter `orderId` is declared as `int`. ASP.NET Core's model binding attempts to parse the URL segment `42;DROP TABLE Orders--` (URL-decoded from the route) as an `int`. This fails because the string is not a valid integer.

The model binding failure returns a `400 Bad Request` with a `ValidationProblemDetails` body (assuming `[ApiController]` is present or using Minimal API default validation). The injection payload never reaches the database or even the handler method.

**Why this matters:** Strongly typing route parameters to their actual domain types (`int`, `Guid`, `enum`) provides defense-in-depth against injection. Even if the handler had an injection vulnerability, many injection payloads cannot pass through an `int` type constraint.

**HTTP response (approximate):**

```http
HTTP/1.1 400 Bad Request
Content-Type: application/problem+json
{"type":"...","title":"One or more validation errors occurred.","status":400,"errors":{"orderId":["The value '42;DROP TABLE Orders--' is not valid."]}}
```

</details>

**Puzzle 4 — Which middleware handles this, and what is the response?**

```csharp
// In Program.cs:
app.UseExceptionHandler(options =>
{
    options.Run(async context =>
    {
        var feature = context.Features.Get<IExceptionHandlerFeature>();
        context.Response.StatusCode = 500;
        await context.Response.WriteAsJsonAsync(new
        {
            error = feature?.Error.Message
        });
    });
});

// Endpoint:
app.MapGet("/api/reports", async (string reportType, ReportDbContext db) =>
{
    // An injection attempt reaches a correctly parameterized query that fails type conversion
    var sql = "SELECT * FROM Reports WHERE Type = @type AND ActiveYear = @year";
    var results = await db.Database.SqlQueryRaw<ReportRow>(sql,
        new SqlParameter("@type", reportType),
        new SqlParameter("@year", "not-a-year")); // always a bug, not from user input
    return TypedResults.Ok(results);
});
// Request: GET /api/reports?reportType=annual%27%20OR%201%3D1%20--
```

<details> <summary>Answer</summary>

**HTTP/1.1 500 Internal Server Error**

The `@type` parameter is correctly parameterized — `reportType = "annual' OR 1=1 --"` is bound as a `varchar` value, not injected into SQL grammar. However, `@year = "not-a-year"` causes a SQL Server type conversion error when the database tries to convert it to an integer (or date) column type.

The `SqlException` is thrown, travels back up the middleware pipeline, and is caught by `UseExceptionHandler`.

**The critical problem:** The response body includes `feature?.Error.Message`. For a `SqlException` thrown because of a type conversion failure, the message may contain:

```
"Conversion failed when converting the varchar value 'not-a-year' to data type int."
```

This **leaks schema information**: the column type (`int`), and depending on the error, column names. For an injection attempt that fails with a type mismatch, the message might include the attacker's payload echoed back.

**Correct fix:** Log `feature.Error.Message` internally; return a generic message to the client:

```csharp
await context.Response.WriteAsJsonAsync(new { error = "An internal error occurred" });
```

</details>

**Puzzle 5 — The Most Common EF Core Injection Bug**

```csharp
// Production code in an order management service:
public async Task<IReadOnlyList<Order>> SearchOrders(
    string customerName,
    string sortField,
    string sortDirection)
{
    if (!new[] { "ASC", "DESC" }.Contains(sortDirection.ToUpper()))
        sortDirection = "ASC";

    return await _db.Orders
        .FromSql($"""
            SELECT * FROM Orders
            WHERE CustomerName LIKE {customerName}
            ORDER BY {sortField} {sortDirection}
            """)
        .AsNoTracking()
        .ToListAsync();
}
// Called with: customerName = "%", sortField = "1; DROP TABLE Orders; --", sortDirection = "ASC"
```

<details> <summary>Answer</summary>

**Two injection vulnerabilities in this single method:**

**Vulnerability 1 (fixed by FormattableString):** `customerName` appears in a `{...}` hole in the `FromSql` call. Because `FromSql` accepts `FormattableString`, `customerName` IS correctly parameterized as `@p0`. So `customerName = "%"` becomes a safe `LIKE @p0` query — it returns all orders (a LIKE wildcard abuse / DoS, but not injection).

**Vulnerability 2 (NOT fixed by FormattableString):** `sortField` and `sortDirection` are also in `{...}` holes in the `FromSql` call. While `sortDirection` is validated against `["ASC", "DESC"]`, `sortField` is passed directly without an allowlist check.

Even though `FromSql` uses `FormattableString`, **column names in ORDER BY cannot be parameterized**. EF Core detects that `sortField` is in an `ORDER BY` position (structural SQL, not a value position) and inserts it as a literal, not as `@p2`. This is an EF Core behavior specific to `ORDER BY`, `GROUP BY`, and table name positions — they are always literals.

**HTTP consequence:**

```
SQL sent to database:
SELECT * FROM Orders WHERE CustomerName LIKE @p0
ORDER BY 1; DROP TABLE Orders; -- ASC

Result: Orders table deleted.
HTTP/1.1 500 Internal Server Error
```

**The fix:** Add `sortField` to an allowlist before calling `FromSql`. The `sortDirection` validation is correct but insufficient alone.

**Why this is the most common bug:** Engineers correctly use `FromSql` for the value positions, see that `customerName` is parameterized, and conclude the query is safe. They don't realize that structural positions (ORDER BY column names) behave differently.

</details>
