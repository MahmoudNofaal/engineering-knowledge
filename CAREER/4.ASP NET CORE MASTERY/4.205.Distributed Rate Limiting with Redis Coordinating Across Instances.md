# 4.205 — Distributed Rate Limiting with Redis: Coordinating Across Instances

---

## PART 0 — Navigation & Context

**Prerequisites:** [[4.202]], [[4.203]], [[4.188 — Redis]]

**Why it matters:** In-memory `PartitionedRateLimiter` is **per pod** — 100 req/min × 20 pods = **2000 req/min** effective limit; Redis centralizes counters so HTTP **429** fires at the true global budget.

---

## PART 1 — Core Mental Model

> **Built-in ASP.NET Core rate limiting is in-process — distributed limiting requires a Redis-backed `RateLimiter` implementation (community packages like `RedisRateLimiting` or custom `PartitionedRateLimiter` using atomic INCR + EXPIRE) so all Kestrel instances share partition state.**

### Approaches

| Approach | Mechanism |
|---|---|
| Redis INCR + TTL | Fixed window per key |
| Redis sliding window (sorted sets) | Smoother limits |
| API Gateway | Envoy/Kong limits before app |
| AspNetCoreRateLimit package | Middleware + Redis store |

---

## PART 2 — Conceptual Redis Fixed Window

```csharp
// Pseudocode per request:
// key = $"rl:{partitionKey}:{windowId}"
// count = await redis.StringIncrementAsync(key)
// if (count == 1) await redis.KeyExpireAsync(key, window)
// if (count > permitLimit) reject → HTTP 429
```

**Cost:** +1–3ms Redis RTT per request.

## PART 3 — Production Patterns

### Pattern 1: Gateway-level limiting + app-level per-user

Edge CDN limits IP; app limits userId in Redis.

### Pattern 2: Fintech — strict global payment endpoint

All pods share `payments:create:{userId}` counter.

### Pattern 3: ⚠️ WRONG — In-memory only with 20 replicas

Effective limit = configured × pod count.

---

## PART 4 — Gotchas

1. Per-pod in-memory limits ineffective horizontally.
2. Redis clock skew on window boundaries.
3. Hot key on single partition — Redis single-thread per key still fast but monitor.
4. Race without atomic INCR.
5. Sticky sessions don't replace distributed limits.

---

## PART 6 — Interview

> Built-in rate limiting is per instance. For Kubernetes I use Redis-backed counters with the same partition keys as 4.203 so 100/min is global. Tradeoff is Redis latency and availability. Alternative is API gateway rate limiting.
