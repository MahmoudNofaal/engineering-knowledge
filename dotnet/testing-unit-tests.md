# Unit Testing in .NET

> A unit test verifies that a single, isolated piece of logic produces the correct output for a given input — with no real database, network, or file system involved.

---

## When To Use It
Use unit tests for any logic that makes decisions — calculations, validations, branching conditions, domain rules, mapping logic. They're the fastest tests to run and the most precise at pinpointing failures. Don't unit test code that has no logic — a method that calls `_repo.GetByIdAsync(id)` and returns the result has nothing to assert without also testing the repository, which becomes an integration test. Don't reach for unit tests when the thing you care about is whether two real components work together correctly — that's integration territory.

---

## Core Concept
A unit test has three parts: arrange (set up the inputs and dependencies), act (call the thing under test), assert (verify the output). The key constraint is isolation — the class under test must not touch anything external. You achieve isolation by injecting fakes: either hand-written stub/fake classes, or mock objects generated by a library like Moq or NSubstitute. The test should only fail because the logic in the class under test is wrong — not because the database is down or a dependency changed its behavior. In .NET, xUnit is the standard test framework; Moq and NSubstitute handle mocking; FluentAssertions makes assertions readable. The file structure mirrors the production project: `MyApp.Tests/` with one test class per production class.

```csharp
// 2. Simple unit test — pure logic, no dependencies
public class PriceCalculator
{
    public decimal ApplyDiscount(decimal price, int customerAge)
    {
        if (customerAge >= 65) return price * 0.8m;   // 20% senior discount
        if (price > 100m)      return price * 0.9m;   // 10% bulk discount
        return price;
    }
}

public class PriceCalculatorTests
{
    private readonly PriceCalculator _sut = new();    // sut = system under test

    [Fact]
    public void ApplyDiscount_SeniorCustomer_Returns20PercentOff()
    {
        var result = _sut.ApplyDiscount(100m, customerAge: 65);
        result.Should().Be(80m);
    }

    [Fact]
    public void ApplyDiscount_PriceOver100_Returns10PercentOff()
    {
        var result = _sut.ApplyDiscount(200m, customerAge: 30);
        result.Should().Be(180m);
    }

    [Theory]
    [InlineData(50, 30, 50)]     // under 100, not senior — no discount
    [InlineData(100, 65, 80)]    // senior — 20% off
    [InlineData(150, 40, 135)]   // over 100 — 10% off
    public void ApplyDiscount_VariousInputs_ReturnsExpected(
        decimal price, int age, decimal expected)
    {
        _sut.ApplyDiscount(price, age).Should().Be(expected);
    }
}
```

```csharp
// 3. Mocking a dependency with Moq
public class OrderService
{
    private readonly IOrderRepository _repo;
    private readonly IEmailService _email;

    public OrderService(IOrderRepository repo, IEmailService email)
    {
        _repo = repo;
        _email = email;
    }

    public async Task<bool> CancelOrderAsync(int orderId)
    {
        var order = await _repo.GetByIdAsync(orderId);
        if (order is null || order.Status == OrderStatus.Shipped)
            return false;

        order.Status = OrderStatus.Cancelled;
        await _repo.SaveChangesAsync();
        await _email.SendCancellationAsync(order.CustomerEmail);
        return true;
    }
}

public class OrderServiceTests
{
    private readonly Mock<IOrderRepository> _repo = new();
    private readonly Mock<IEmailService>    _email = new();
    private readonly OrderService           _sut;

    public OrderServiceTests() =>
        _sut = new OrderService(_repo.Object, _email.Object);

    [Fact]
    public async Task CancelOrder_OrderNotFound_ReturnsFalse()
    {
        _repo.Setup(r => r.GetByIdAsync(99))
             .ReturnsAsync((Order?)null);

        var result = await _sut.CancelOrderAsync(99);

        result.Should().BeFalse();
        _email.Verify(e => e.SendCancellationAsync(It.IsAny<string>()), Times.Never);
    }

    [Fact]
    public async Task CancelOrder_ShippedOrder_ReturnsFalse()
    {
        _repo.Setup(r => r.GetByIdAsync(1))
             .ReturnsAsync(new Order { Id = 1, Status = OrderStatus.Shipped });

        var result = await _sut.CancelOrderAsync(1);

        result.Should().BeFalse();
    }

    [Fact]
    public async Task CancelOrder_ValidOrder_CancelsAndSendsEmail()
    {
        var order = new Order { Id = 1, Status = OrderStatus.Pending,
                                CustomerEmail = "x@y.com" };

        _repo.Setup(r => r.GetByIdAsync(1)).ReturnsAsync(order);

        var result = await _sut.CancelOrderAsync(1);

        result.Should().BeTrue();
        order.Status.Should().Be(OrderStatus.Cancelled);
        _email.Verify(e => e.SendCancellationAsync("x@y.com"), Times.Once);
    }
}
```

```csharp
public class FakeOrderRepository : IOrderRepository
{
    private readonly List<Order> _store = new();

    public Task<Order?> GetByIdAsync(int id) =>
        Task.FromResult(_store.FirstOrDefault(o => o.Id == id));

    public Task SaveChangesAsync() => Task.CompletedTask;

    public void Seed(Order order) => _store.Add(order);   // test helper method
```

## Gotchas
- **Testing implementation, not behavior, makes tests brittle.** If your test asserts that `_repo.GetByIdAsync()` was called exactly once, it breaks whenever you refactor the method — even if the behavior is identical. Assert on outputs and observable state, not on which internal methods were called. Reserve `Verify()` for side effects that have no return value to assert on (like sending an email).
- **`[Theory]` with `[InlineData]` doesn't replace distinct named tests for important edge cases.** Parameterized tests are great for exhaustive coverage of similar cases, but a failing `[Theory]` row shows up with its data values — not a meaningful test name. Add `[Fact]` tests with descriptive names for the cases that matter most to document intent.
- **Moq's `Setup()` returns `default` if the call doesn't match.** If you set up `GetByIdAsync(1)` but the code calls `GetByIdAsync(2)`, Moq silently returns `null` rather than throwing. Use `MockBehavior.Strict` to catch unexpected calls: `new Mock<IRepo>(MockBehavior.Strict)`.
- **Async tests must be `async Task`, not `async void`.** `async void` test methods swallow exceptions — the test runner sees a passing test while the assertion fails silently. Always declare async tests as `async Task`.
- **Shared mutable state in test class fields causes flaky tests.** If you reset a `Mock<T>` between tests by calling `.Reset()` in a constructor, you're fine — xUnit creates a new test class instance per test. But if you use a static field or a shared fake with mutable state, tests affect each other. Keep all test state instance-scoped.

## Interview Angle
**What they're really testing:** Whether you understand what makes a test a *unit* test — isolation — and whether you know what's worth testing vs what's just noise.

**Common question form:** *"How do you test a service that depends on a database?"* or *"What's the difference between a mock and a stub?"* or *"How do you structure your tests?"*

**The depth signal:** A junior says "use Moq to mock the repository" and writes a test that verifies every method call. A senior distinguishes between mocks (verify interactions), stubs (return canned data), and fakes (working implementations) — and knows when each is appropriate. They assert on behavior and output, not on call counts, use `[Theory]` for data coverage without losing meaningful test names, know the `MockBehavior.Strict` trick for catching silent mismatches, and can articulate the line between a unit test and an integration test: if real EF Core or a real HTTP client is involved, it's integration.
