# Unit Testing in .NET

> A unit test verifies that a single, isolated piece of logic produces the correct output for a given input — with no real database, network, or file system involved.

---

## Quick Reference

| | |
|---|---|
| **What it is** | Isolated logic test, no real dependencies |
| **Use when** | Testing decisions, calculations, domain rules, branching logic |
| **Avoid when** | Verifying two real components work together — use integration tests |
| **Framework** | xUnit (standard), NUnit (legacy), MSTest (legacy) |
| **Key packages** | `xunit`, `Moq` or `NSubstitute`, `FluentAssertions` |
| **Key types** | `[Fact]`, `[Theory]`, `[InlineData]`, `Mock<T>`, `IClassFixture<T>` |

---

## When To Use It
Use unit tests for any logic that makes decisions — calculations, validations, branching conditions, domain rules, mapping logic. They're the fastest tests to run and the most precise at pinpointing failures. Don't unit test code that has no logic — a method that calls `_repo.GetByIdAsync(id)` and returns the result has nothing to assert without also testing the repository, which becomes an integration test. Don't reach for unit tests when the thing you care about is whether two real components work together correctly — that's integration territory.

---

## Core Concept
A unit test has three parts: arrange (set up the inputs and dependencies), act (call the thing under test), assert (verify the output). The key constraint is isolation — the class under test must not touch anything external. You achieve isolation by injecting fakes: either hand-written stub/fake classes, or mock objects generated by a library like Moq or NSubstitute.

The three terms people conflate are worth separating clearly. A **stub** provides canned return values so the code under test can run — you don't verify calls on it. A **mock** is a stub where you also assert after the fact that specific methods were called with specific arguments. A **fake** is a lightweight working implementation — like an in-memory list standing in for a database — that you write by hand when setup repetition across tests makes mocks impractical. In practice Moq objects do all three depending on how you use them.

The test should only fail because the logic in the class under test is wrong — not because the database is down or a dependency changed its behavior. In .NET, xUnit is the standard test framework; Moq and NSubstitute handle mocking; FluentAssertions makes assertions readable. The file structure mirrors the production project: `MyApp.Tests/` with one test class per production class, named `{ClassName}Tests`.

The rule of thumb for when to use `Verify()` vs asserting on output: assert on return values and observable state first — `Verify()` is only for fire-and-forget side effects like sending an email or publishing an event where there is no return value to assert on. Over-using `Verify()` creates brittle tests that break on refactors even when behavior is correct.

## Version History

| Package | Version | What changed |
|---|---|---|
| xUnit | 1.x → 2.x | Constructor/`IAsyncLifetime` replaced `Setup`/`Teardown` attributes |
| xUnit | 2.x | Parallel test execution on by default — shared state between tests now causes flakiness |
| xUnit | 2.4+ | `IAsyncLifetime` for async setup/teardown |
| Moq | 4.8+ | `ReturnsAsync()` and `ThrowsAsync()` added for async interface methods |
| Moq | 4.16+ | `Setup` on protected members via `Protected()` |
| NSubstitute | 4.x+ | Async-safe `Returns()` and `ThrowsAsync()` |
| FluentAssertions | 6.x → 7.x | Breaking: `AndWhich` and some chain methods renamed — check changelog when upgrading |

*Before xUnit 2, `[SetUp]` and `[TearDown]` attributes were used (NUnit style). xUnit replaced them with constructor injection and `IDisposable`/`IAsyncLifetime` — there is no `[Setup]` attribute in xUnit.*

## Performance

| Aspect | Typical range | Notes |
|---|---|---|
| Single test execution | < 1ms | Pure logic tests with no I/O |
| Moq mock creation | ~0.1–0.5ms | Cost per `new Mock<T>()` — negligible but noticeable at scale |
| Full unit test suite (1,000 tests) | 1–5 seconds | With parallelisation on |
| Full unit test suite (1,000 tests) | 5–20 seconds | With parallelisation off or on a slow machine |

**Allocation behaviour:** Unit tests themselves allocate normally. Moq uses runtime code generation (via `Castle.DynamicProxy`) to create mock objects — each `new Mock<T>()` triggers dynamic proxy generation, which is heap-allocated. For 10,000+ mocks per run this becomes measurable; in practice suites of a few hundred tests see no meaningful overhead.

**Benchmark notes:** Unit tests should not be the slow part of your build. If your unit test suite takes more than 30 seconds, the likely culprit is tests that are secretly integration tests — they're hitting a real database or real HTTP endpoint and have been miscategorised. Running `dotnet test --logger "console;verbosity=normal"` and sorting by duration exposes them quickly.

```csharp
// 1. Simple unit test — pure logic, no dependencies
public class PriceCalculator
{
    public decimal ApplyDiscount(decimal price, int customerAge)
    {
        if (customerAge >= 65) return price * 0.8m;   // 20% senior discount
        if (price > 100m)      return price * 0.9m;   // 10% bulk discount
        return price;
    }
}

public class PriceCalculatorTests
{
    private readonly PriceCalculator _sut = new();    // sut = system under test

    [Fact]
    public void ApplyDiscount_SeniorCustomer_Returns20PercentOff()
    {
        var result = _sut.ApplyDiscount(100m, customerAge: 65);
        result.Should().Be(80m);
    }

    [Theory]
    [InlineData(50,  30, 50)]    // under 100, not senior — no discount
    [InlineData(100, 65, 80)]    // senior — 20% off
    [InlineData(150, 40, 135)]   // over 100 — 10% off
    public void ApplyDiscount_VariousInputs_ReturnsExpected(
        decimal price, int age, decimal expected)
    {
        _sut.ApplyDiscount(price, age).Should().Be(expected);
    }
}
```

```csharp
// 2. Mocking a dependency — stub return values
public class OrderService
{
    private readonly IOrderRepository _repo;
    private readonly IEmailService _email;

    public OrderService(IOrderRepository repo, IEmailService email)
    {
        _repo = repo;
        _email = email;
    }

    public async Task<bool> CancelOrderAsync(int orderId)
    {
        var order = await _repo.GetByIdAsync(orderId);
        if (order is null || order.Status == OrderStatus.Shipped)
            return false;

        order.Status = OrderStatus.Cancelled;
        await _repo.SaveChangesAsync();
        await _email.SendCancellationAsync(order.CustomerEmail);
        return true;
    }
}

public class OrderServiceTests
{
    private readonly Mock<IOrderRepository> _repo = new();
    private readonly Mock<IEmailService>    _email = new();
    private readonly OrderService           _sut;

    public OrderServiceTests() =>
        _sut = new OrderService(_repo.Object, _email.Object);

    [Fact]
    public async Task CancelOrder_OrderNotFound_ReturnsFalse()
    {
        _repo.Setup(r => r.GetByIdAsync(99))
             .ReturnsAsync((Order?)null);

        var result = await _sut.CancelOrderAsync(99);

        result.Should().BeFalse();
        _email.Verify(e => e.SendCancellationAsync(It.IsAny<string>()), Times.Never);
    }

    [Fact]
    public async Task CancelOrder_ValidOrder_CancelsAndSendsEmail()
    {
        var order = new Order { Id = 1, Status = OrderStatus.Pending,
                                CustomerEmail = "x@y.com" };

        _repo.Setup(r => r.GetByIdAsync(1)).ReturnsAsync(order);

        var result = await _sut.CancelOrderAsync(1);

        result.Should().BeTrue();
        order.Status.Should().Be(OrderStatus.Cancelled);
        // Verify only for the side effect — email has no return value to assert on
        _email.Verify(e => e.SendCancellationAsync("x@y.com"), Times.Once);
    }
}
```

```csharp
// 4. Hand-written fake — prefer over mocks when reused across many tests
public class FakeOrderRepository : IOrderRepository
{
    private readonly List<Order> _store = new();
    public int SaveCallCount { get; private set; }

    public Task<Order?> GetByIdAsync(int id) =>
        Task.FromResult(_store.FirstOrDefault(o => o.Id == id));

    public Task SaveChangesAsync()
    {
        SaveCallCount++;
        return Task.CompletedTask;
    }

    public void Seed(params Order[] orders) => _store.AddRange(orders);
}

// Usage — no Setup() repetition across ten tests
public class OrderServiceFakeTests
{
    private readonly FakeOrderRepository _repo = new();
    private readonly Mock<IEmailService> _email = new();
    private readonly OrderService        _sut;

    public OrderServiceFakeTests()
    {
        _sut = new OrderService(_repo, _email.Object);
        _repo.Seed(new Order { Id = 1, Status = OrderStatus.Pending,
                               CustomerEmail = "x@y.com" });
    }

    [Fact]
    public async Task CancelOrder_SavesChangesExactlyOnce()
    {
        await _sut.CancelOrderAsync(1);
        _repo.SaveCallCount.Should().Be(1);
    }
}
```

```csharp
// 5. What NOT to do — and why

// BAD: verifying every call makes tests brittle and couples them to implementation
[Fact]
public async Task BAD_CancelOrder_VerifiesEverything()
{
    _repo.Setup(r => r.GetByIdAsync(1)).ReturnsAsync(new Order { Id = 1 });

    await _sut.CancelOrderAsync(1);

    // These verify internal plumbing — break on refactors even when behavior is correct
    _repo.Verify(r => r.GetByIdAsync(1), Times.Once);         // ← don't do this
    _repo.Verify(r => r.SaveChangesAsync(), Times.Once);       // ← don't do this
    _email.Verify(e => e.SendCancellationAsync(It.IsAny<string>()), Times.Once);
}

// GOOD: assert on output, verify only the non-observable side effect
[Fact]
public async Task GOOD_CancelOrder_ReturnsTrueAndNotifiesCustomer()
{
    var order = new Order { Id = 1, Status = OrderStatus.Pending,
                            CustomerEmail = "x@y.com" };
    _repo.Setup(r => r.GetByIdAsync(1)).ReturnsAsync(order);

    var result = await _sut.CancelOrderAsync(1);

    result.Should().BeTrue();
    order.Status.Should().Be(OrderStatus.Cancelled);           // ← assert observable state
    _email.Verify(e => e.SendCancellationAsync("x@y.com"), Times.Once); // ← only side effect
}
```

## Real World Example

An e-commerce platform runs a nightly job that applies promotional discounts to eligible orders. The `PromotionEngine` evaluates each order against a set of rules — minimum spend thresholds, customer tier, days since last order — and returns a discount percentage. This logic is pure calculation with multiple branching conditions and no database involvement, making it the ideal unit test target: fast feedback, precise failure location, no infrastructure setup.

```csharp
public class PromotionEngine
{
    private readonly IClock _clock;

    public PromotionEngine(IClock clock) => _clock = clock;

    public decimal CalculateDiscount(Order order, CustomerProfile customer)
    {
        ArgumentNullException.ThrowIfNull(order);
        ArgumentNullException.ThrowIfNull(customer);

        if (order.Total < 50m)
            return 0m;

        var daysSinceLastOrder = (_clock.UtcNow - customer.LastOrderDate).TotalDays;

        // Win-back promotion: hasn't ordered in 90+ days
        if (daysSinceLastOrder >= 90)
            return customer.Tier == CustomerTier.Gold ? 0.20m : 0.15m;

        // Loyalty discount: Gold customers with order over £200
        if (customer.Tier == CustomerTier.Gold && order.Total >= 200m)
            return 0.10m;

        // Standard bulk discount
        return order.Total >= 100m ? 0.05m : 0m;
    }
}

public class PromotionEngineTests
{
    private readonly Mock<IClock> _clock = new();
    private readonly PromotionEngine _sut;
    private readonly DateTime _now = new DateTime(2026, 1, 15);

    public PromotionEngineTests()
    {
        _clock.Setup(c => c.UtcNow).Returns(_now);
        _sut = new PromotionEngine(_clock.Object);
    }

    [Fact]
    public void CalculateDiscount_OrderUnder50_ReturnsZero()
    {
        var order    = new Order { Total = 49.99m };
        var customer = new CustomerProfile { Tier = CustomerTier.Gold,
                                             LastOrderDate = _now.AddDays(-10) };

        var result = _sut.CalculateDiscount(order, customer);

        result.Should().Be(0m);
    }

    [Theory]
    [InlineData(CustomerTier.Gold,     20, 0.20)]   // Gold win-back
    [InlineData(CustomerTier.Standard, 20, 0.15)]   // Standard win-back
    public void CalculateDiscount_WinBackEligible_AppliesTierDiscount(
        CustomerTier tier, decimal daysAgo, decimal expected)
    {
        var order    = new Order { Total = 100m };
        var customer = new CustomerProfile { Tier = tier,
                                             LastOrderDate = _now.AddDays(-(double)daysAgo) };

        // Act with different clock values to test the 90-day boundary
        _clock.Setup(c => c.UtcNow).Returns(_now);
        var customerInactive = customer with { LastOrderDate = _now.AddDays(-90) };

        var result = _sut.CalculateDiscount(order, customerInactive);

        result.Should().Be(expected);
    }

    [Fact]
    public void CalculateDiscount_GoldCustomerLargeOrder_AppliesLoyaltyDiscount()
    {
        var order    = new Order { Total = 250m };
        var customer = new CustomerProfile { Tier = CustomerTier.Gold,
                                             LastOrderDate = _now.AddDays(-30) };

        var result = _sut.CalculateDiscount(order, customer);

        result.Should().Be(0.10m);
    }

    [Fact]
    public void CalculateDiscount_NullOrder_ThrowsArgumentNullException()
    {
        var customer = new CustomerProfile();
        var act = () => _sut.CalculateDiscount(null!, customer);

        act.Should().Throw<ArgumentNullException>()
           .WithParameterName("order");
    }
}
```

*The key insight here is the `IClock` abstraction — injecting the clock instead of calling `DateTime.UtcNow` directly is what makes the 90-day boundary deterministically testable. Without it, a test asserting win-back behavior would start failing 90 days after it was written.*

## Common Misconceptions

**"Unit tests should test every method in a class."**
Unit tests should test every *behavior* — which does not map one-to-one to methods. A private helper method that's called by three public methods doesn't get its own tests; the public method tests cover it. Testing private methods directly through reflection is a sign the class has too many responsibilities and should be split, not a sign you need smarter test tooling.

**"The more `Verify()` calls I have, the more thorough my tests are."**
The opposite is closer to the truth. Every `Verify()` call on an internal implementation detail is a test that will break when you refactor — even when the behavior is still correct. A test that breaks on a valid refactor is a test with a negative value. Assert on the observable outputs (return values, observable state changes); use `Verify()` only for side effects that have no return value to assert on, like sending an email or publishing an event.

**"Mocking an interface is always better than a hand-written fake."**
Moq shines for one-off setups in a single test. When the same mock setup appears five or more times across different tests, a hand-written fake with a `Seed()` method is less code, more readable, and easier to evolve. The question to ask is: if this interface grows a new method, which approach requires fewer changes — updating every `Setup()` call in every test class, or adding one method to the fake?

```csharp
// When this interface gets a third method, the fake needs one addition.
// Every Moq-based test class needs nothing — until MockBehavior.Strict catches the gap.
public class FakeOrderRepository : IOrderRepository
{
    // ...existing implementation...

    // New method — add once here, all fake-based tests still compile and work
    public Task<IReadOnlyList<Order>> GetByCustomerAsync(int customerId) =>
        Task.FromResult<IReadOnlyList<Order>>(
            _store.Where(o => o.CustomerId == customerId).ToList());
}
```

## Gotchas

- **Testing implementation instead of behavior makes tests brittle.** If your test asserts that `_repo.GetByIdAsync()` was called exactly once, it breaks whenever you refactor the method — even if the behavior is identical. Assert on outputs and observable state first; only use `Verify()` for side effects that have no return value to assert on.

- **`[Theory]` with `[InlineData]` doesn't replace named tests for important edge cases.** Parameterized tests are great for exhaustive coverage of similar cases, but a failing `[Theory]` row shows up with its data values, not a meaningful test name. Add `[Fact]` tests with descriptive names for the cases that matter most to document intent.

- **Moq's `Setup()` returns `default` silently if the call doesn't match.** If you set up `GetByIdAsync(1)` but the code calls `GetByIdAsync(2)`, Moq returns `null` instead of throwing. Use `MockBehavior.Strict` to catch unexpected calls: `new Mock<IRepo>(MockBehavior.Strict)`.

- **Async tests must be `async Task`, not `async void`.** `async void` test methods swallow exceptions — the test runner sees a passing test while the assertion fails silently. Always declare async tests as `async Task`.

- **Shared mutable state in test class fields causes flaky tests.** xUnit creates a new test class instance per test — constructor injection is safe. But static fields and shared fake instances with mutable internal state bleed between tests. Keep all test state instance-scoped and never use `static` fields for test data.

- **`DateTime.UtcNow` called directly inside production code makes time-sensitive logic untestable.** Inject an `IClock` abstraction (or use `TimeProvider` from .NET 8+). Any code that computes "90 days ago" or "has this expired" needs a seam for the test to control time — otherwise the test becomes a time bomb.

## Interview Angle

**What they're really testing:** Whether you understand what makes a test a *unit* test — isolation — and whether you know what's worth testing vs what's just noise.

**Common question forms:**
- *"How do you test a service that depends on a database?"*
- *"What's the difference between a mock and a stub?"*
- *"How do you structure your tests?"*
- *"How do you test code that depends on the current time?"*

**The depth signal:** A junior says "use Moq to mock the repository" and writes a test that verifies every method call. A senior distinguishes between mocks (verify interactions), stubs (return canned data), and fakes (working implementations) — and knows when each is appropriate. They assert on behavior and output, not on call counts, use `[Theory]` for data coverage without losing meaningful test names, know the `MockBehavior.Strict` trick for catching silent mismatches, inject `IClock` or use `TimeProvider` instead of calling `DateTime.UtcNow` directly, and can articulate the line between a unit test and an integration test: if real EF Core or a real HTTP client is involved, it's integration.

**Follow-up questions to expect:**
- *"When would you use a hand-written fake instead of Moq?"* — When the same setup repeats across many tests or when the fake needs real logic (e.g. maintaining a list of seeded items across calls).
- *"How do you test private methods?"* — You don't, directly. If a private method has complex logic worth testing, it's a sign it should be extracted to a separate class with its own public interface.
