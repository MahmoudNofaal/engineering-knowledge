## 1. Overview — The Five Pillars of SQL Server Health

SQL Server performance monitoring centers around five key metric categories. Together they provide a complete picture of database engine health. Monitoring just one category in isolation can lead to misdiagnosis — for example, high IO waits might actually be caused by memory pressure forcing page swaps rather than a slow storage subsystem.

### 1.1 The Five Categories

| Category | What It Measures | Key DMVs | Typical Thresholds |
|---|---|---|---|
| CPU | Scheduler pressure, batch requests, plan cache churn | `sys.dm_os_schedulers`, `sys.dm_os_performance_counters` | Avg runnable task count < 10, Batch Reqs/sec trending stable |
| IO | Disk throughput, latency, page read/write patterns | `sys.dm_io_virtual_file_stats`, `sys.dm_os_wait_stats` | Avg read ms < 20, Avg write ms < 10 |
| Memory | Buffer pool health, cache efficiency, page life expectancy | `sys.dm_os_performance_counters`, `sys.dm_os_sys_info` | PLE > 300s (baseline-dependent), Cache Hit Ratio > 95% |
| Concurrency | Blocking chains, deadlocks, lock wait duration | `sys.dm_tran_locks`, `sys.dm_exec_requests`, `sys.dm_os_wait_stats` | LCK_M waits < 5% of total waits |
| Throughput | Batch requests, compilations, recompilations, user connections | `sys.dm_os_performance_counters` | SQL Recompilations/sec < 10% of Batch Reqs/sec |

### 1.2 Why These Five Categories

Each category maps to a specific subsystem inside SQL Server. When a query is slow, the bottleneck will manifest in one or more of these five areas. The art of performance monitoring is using wait statistics to determine _which_ subsystem is the bottleneck, then drilling into that category's detailed metrics.

### 1.3 Monitoring Cadence

| Timeframe | Action | Purpose |
|---|---|---|
| Real-time (every 30-60s) | Poll key perf counters | Detect acute spikes, blocking |
| Hourly | Snapshot wait stats, file stats | Track trends intra-day |
| Daily | Capture baseline snapshot | Compare day-over-day |
| Weekly | Analyze wait categorization | Identify emerging bottlenecks |
| Monthly | Review trend data, adjust thresholds | Capacity planning |

## 2. Core Concepts — Performance Counter Architecture

SQL Server exposes internal metrics through two primary mechanisms: Dynamic Management Views (DMVs) and Performance Counters. Understanding how each works is essential for accurate monitoring.

### 2.1 sys.dm_os_performance_counters

This DMV mirrors the SQL Server perfmon counters. It returns three counter types:

- **Cumulative counters** — Value increases monotonically from server start (e.g., `Batch Requests/sec` is actually total batch requests since startup; you compute a rate yourself).
- **Instantaneous counters** — Current value at query time (e.g., `Page Life Expectancy`).
- **Ratio counters** — Two counters you divide (e.g., `Buffer cache hit ratio` and `Buffer cache hit ratio base`).

```sql
-- Counter type detection
SELECT
    object_name,
    counter_name,
    instance_name,
    cntr_value,
    cntr_type,
    CASE cntr_type
        WHEN 272696576 THEN 'Cumulative — divide by base counter'
        WHEN 1073939712 THEN 'Cumulative — compute rate (delta / time)'
        WHEN 537003264 THEN 'Instantaneous — point-in-time value'
        WHEN 65792 THEN 'Ratio base — use as denominator'
        WHEN 1073874176 THEN 'Cumulative — large value, compute rate'
        ELSE 'Unknown: ' + CAST(cntr_type AS VARCHAR(20))
    END AS counter_type_description
FROM sys.dm_os_performance_counters
WHERE counter_name IN (
    'Batch Requests/sec',
    'SQL Compilations/sec',
    'SQL Re-Compilations/sec',
    'Page Life Expectancy',
    'Buffer cache hit ratio',
    'Buffer cache hit ratio base',
    'Checkpoint pages/sec',
    'Page reads/sec',
    'Page writes/sec',
    'Lazy writes/sec',
    'Lock Waits/sec',
    'Deadlocks/sec'
)
ORDER BY object_name, counter_name;
```

### 2.2 sys.dm_os_sys_info

Provides server-level hardware information and SQL Server internal state. Key columns:

| Column | Meaning |
|---|---|
| cpu_count | Number of logical CPUs visible to SQL Server |
| hyperthread_ratio | Physical vs logical core ratio |
| physical_memory_kb | Total physical memory on the machine |
| virtual_memory_kb | Virtual address space available |
| committed_kb | Memory committed to SQL Server |
| committed_target_kb | Target memory for SQL Server |
| max_workers_count | Maximum worker threads |
| scheduler_count | Number of schedulers (usually = cpu_count) |
| scheduler_total_count | Including hidden/system schedulers |
| sqlserver_start_time | Last restart time — crucial for baseline context |

```sql
SELECT
    cpu_count,
    hyperthread_ratio,
    cpu_count / hyperthread_ratio AS physical_cores,
    physical_memory_kb / 1024 AS physical_memory_mb,
    committed_kb / 1024 AS committed_mb,
    committed_target_kb / 1024 AS committed_target_mb,
    max_workers_count,
    scheduler_count,
    sqlserver_start_time,
    DATEDIFF(DAY, sqlserver_start_time, GETDATE()) AS days_since_restart
FROM sys.dm_os_sys_info;
```

### 2.3 sys.dm_os_schedulers

Shows per-scheduler health. Each CPU gets a scheduler. Important columns:

| Column | Meaning |
|---|---|
| scheduler_id | ID of the scheduler |
| cpu_id | Associated CPU |
| status | VISIBLE ONLINE, VISIBLE OFFLINE, HIDDEN |
| current_tasks_count | Currently running tasks |
| runnable_tasks_count | Tasks waiting in the runnable queue (CPU pressure) |
| pending_disk_io_count | IO waiting for this scheduler |
| load_factor | How loaded the scheduler is |
| yield_count | How many times this scheduler yielded |

```sql
SELECT
    scheduler_id,
    cpu_id,
    status,
    current_tasks_count,
    runnable_tasks_count,
    pending_disk_io_count,
    load_factor,
    yield_count
FROM sys.dm_os_schedulers
WHERE status = 'VISIBLE ONLINE'
ORDER BY scheduler_id;
```

### 2.4 How They Fit Together

```
[Query Execution]
      │
      ▼
[sys.dm_os_schedulers] ── runnable_tasks_count > 0 = CPU pressure
      │
      ▼
[sys.dm_os_wait_stats] ── SOS_SCHEDULER_YIELD = CPU waits
                                PAGEIOLATCH_* = IO waits
                                LCK_* = Lock waits
      │
      ▼
[sys.dm_io_virtual_file_stats] ── Actual disk latency per file
      │
      ▼
[sys.dm_os_performance_counters] ── Point-in-time rates from perf counters
```

## 3. Deep Mechanics — DMV Queries by Metric Category

### 3.1 CPU Pressure Metrics

CPU pressure occurs when the runnable queue has sustained backlogs. This is one of the most frequently misunderstood areas — a high total CPU percentage does not automatically mean CPU pressure.

```sql
-- CPU pressure check: runnable tasks count
-- Values > 10 sustained indicate CPU pressure
-- Values > 50 sustained indicate severe CPU pressure
SELECT
    scheduler_id,
    cpu_id,
    runnable_tasks_count,
    current_tasks_count,
    pending_disk_io_count,
    load_factor,
    'CPU Pressure: ' +
    CASE
        WHEN runnable_tasks_count > 50 THEN 'SEVERE — Run queries below immediately'
        WHEN runnable_tasks_count > 20 THEN 'HIGH — Investigate top queries'
        WHEN runnable_tasks_count > 10 THEN 'MODERATE — Monitor trend'
        ELSE 'NORMAL'
    END AS assessment
FROM sys.dm_os_schedulers
WHERE status = 'VISIBLE ONLINE'
    AND scheduler_id < 255
ORDER BY runnable_tasks_count DESC;
```

```sql
-- Batch requests and compilations (rate since last restart)
-- Compute per-second rate by capturing, waiting, capturing again
DECLARE @br1 BIGINT, @br2 BIGINT, @sc1 BIGINT, @sc2 BIGINT;
DECLARE @sr1 BIGINT, @sr2 BIGINT, @time1 DATETIME2, @time2 DATETIME2;

SELECT
    @br1 = cntr_value
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:SQL Statistics%'
    AND counter_name = 'Batch Requests/sec';

SELECT
    @sc1 = cntr_value
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:SQL Statistics%'
    AND counter_name = 'SQL Compilations/sec';

SELECT
    @sr1 = cntr_value
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:SQL Statistics%'
    AND counter_name = 'SQL Re-Compilations/sec';

SET @time1 = SYSDATETIME();

WAITFOR DELAY '00:00:10';

SELECT
    @br2 = cntr_value
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:SQL Statistics%'
    AND counter_name = 'Batch Requests/sec';

SELECT
    @sc2 = cntr_value
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:SQL Statistics%'
    AND counter_name = 'SQL Compilations/sec';

SELECT
    @sr2 = cntr_value
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:SQL Statistics%'
    AND counter_name = 'SQL Re-Compilations/sec';

SET @time2 = SYSDATETIME();

SELECT
    DATEDIFF(SECOND, @time1, @time2) AS elapsed_seconds,
    (@br2 - @br1) / DATEDIFF(SECOND, @time1, @time2) AS batch_req_per_sec,
    (@sc2 - @sc1) / DATEDIFF(SECOND, @time1, @time2) AS compilations_per_sec,
    (@sr2 - @sr1) / DATEDIFF(SECOND, @time1, @time2) AS recompilations_per_sec,
    CASE
        WHEN (@sc2 - @sc1) * 1.0 / NULLIF(@br2 - @br1, 0) > 0.20
            THEN 'HIGH — Compilation overhead > 20% of batch requests. Check for parameterization issues.'
        WHEN (@sc2 - @sc1) * 1.0 / NULLIF(@br2 - @br1, 0) > 0.10
            THEN 'MODERATE — Monitor plan cache eviction.'
        ELSE 'NORMAL'
    END AS compilation_assessment;
```

### 3.2 IO Metrics

IO metrics tell you about storage subsystem health. Two complementary views are needed: `sys.dm_io_virtual_file_stats` (actual file-level latency) and `sys.dm_os_performance_counters` (throughput rates).

```sql
-- File stats with latency calculation
-- Focus on files with high read/write stall times
-- < 10ms = excellent, 10-20ms = good, 20-50ms = moderate, > 50ms = poor
SELECT
    DB_NAME(database_id) AS database_name,
    file_id,
    CASE
        WHEN database_id = 2 THEN 'tempdb'
        ELSE DB_NAME(database_id)
    END + CASE WHEN file_id = 1 THEN ' (data)' ELSE ' (log)' END AS file_description,
    num_of_reads,
    num_of_writes,
    CAST(num_of_reads + num_of_writes AS DECIMAL(38,0)) AS total_ios,
    CAST(io_stall_read_ms / NULLIF(num_of_reads, 0) AS DECIMAL(10,2)) AS avg_read_latency_ms,
    CAST(io_stall_write_ms / NULLIF(num_of_writes, 0) AS DECIMAL(10,2)) AS avg_write_latency_ms,
    CAST(io_stall / NULLIF(num_of_reads + num_of_writes, 0) AS DECIMAL(10,2)) AS avg_total_latency_ms,
    size_on_disk_bytes / 1048576 AS size_mb,
    CASE
        WHEN num_of_reads = 0 AND num_of_writes = 0 THEN 'No activity'
        WHEN io_stall_read_ms / NULLIF(num_of_reads, 0) > 50 OR io_stall_write_ms / NULLIF(num_of_writes, 0) > 50
            THEN 'CRITICAL — Storage subsystem latency too high'
        WHEN io_stall_read_ms / NULLIF(num_of_reads, 0) > 20 OR io_stall_write_ms / NULLIF(num_of_writes, 0) > 20
            THEN 'WARNING — Elevated latency, investigate storage'
        WHEN io_stall_read_ms / NULLIF(num_of_reads, 0) > 10 OR io_stall_write_ms / NULLIF(num_of_writes, 0) > 10
            THEN 'MODERATE — Monitor trend'
        ELSE 'HEALTHY'
    END AS io_health
FROM sys.dm_io_virtual_file_stats(NULL, NULL)
WHERE database_id > 4  -- Exclude system DBs for user focus
    OR database_id IN (1, 2, 3, 4)  -- Still include system DBs selectively
ORDER BY avg_total_latency_ms DESC;
```

```sql
-- IO throughput rates from perf counters
-- Capture rates over a 30-second window
DECLARE @reads1 BIGINT, @reads2 BIGINT;
DECLARE @writes1 BIGINT, @writes2 BIGINT;
DECLARE @t1 DATETIME2, @t2 DATETIME2;

SELECT @reads1 = cntr_value
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:Buffer Manager%'
    AND counter_name = 'Page reads/sec';

SELECT @writes1 = cntr_value
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:Buffer Manager%'
    AND counter_name = 'Page writes/sec';

SET @t1 = SYSDATETIME();
WAITFOR DELAY '00:00:30';
SET @t2 = SYSDATETIME();

SELECT @reads2 = cntr_value
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:Buffer Manager%'
    AND counter_name = 'Page reads/sec';

SELECT @writes2 = cntr_value
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:Buffer Manager%'
    AND counter_name = 'Page writes/sec';

SELECT
    DATEDIFF(SECOND, @t1, @t2) AS elapsed_seconds,
    (@reads2 - @reads1) / DATEDIFF(SECOND, @t1, @t2) AS page_reads_per_sec,
    (@writes2 - @writes1) / DATEDIFF(SECOND, @t1, @t2) AS page_writes_per_sec,
    CASE
        WHEN (@reads2 - @reads1) / DATEDIFF(SECOND, @t1, @t2) > 5000
            THEN 'HIGH IO READ — Check for missing indexes or memory pressure'
        WHEN (@reads2 - @reads1) / DATEDIFF(SECOND, @t1, @t2) > 2000
            THEN 'ELEVATED IO — Investigate query patterns'
        ELSE 'NORMAL'
    END AS read_assessment,
    CASE
        WHEN (@writes2 - @writes1) / DATEDIFF(SECOND, @t1, @t2) > 5000
            THEN 'HIGH IO WRITE — Check for logging issues or ETL operations'
        ELSE 'NORMAL'
    END AS write_assessment;
```

### 3.3 Memory Metrics

```sql
-- Page Life Expectancy (PLE): how long pages stay in buffer pool
-- Lower values = memory pressure. Thresholds depend on total memory.
SELECT
    cntr_value AS page_life_expectancy_seconds,
    cntr_value / 60 AS page_life_expectancy_minutes,
    CASE
        WHEN cntr_value < 300 THEN 'CRITICAL — Severe memory pressure'
        WHEN cntr_value < 600 THEN 'WARNING — Significant memory pressure'
        WHEN cntr_value < 1800 THEN 'MODERATE — Monitor trend'
        ELSE 'HEALTHY'
    END AS ple_assessment
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:Buffer Manager%'
    AND counter_name = 'Page Life Expectancy';
```

```sql
-- Buffer cache hit ratio: percentage of page requests served from cache
-- Should be > 95% for OLTP workloads
SELECT
    CAST(a.cntr_value * 100.0 / NULLIF(b.cntr_value, 0) AS DECIMAL(5,1)) AS buffer_cache_hit_ratio_pct,
    a.cntr_value AS cache_hits,
    b.cntr_value AS cache_hits_base,
    CASE
        WHEN a.cntr_value * 100.0 / NULLIF(b.cntr_value, 0) < 90
            THEN 'CRITICAL — Excessive physical reads, memory pressure'
        WHEN a.cntr_value * 100.0 / NULLIF(b.cntr_value, 0) < 95
            THEN 'WARNING — Below healthy threshold, investigate'
        ELSE 'HEALTHY'
    END AS cache_hit_assessment
FROM sys.dm_os_performance_counters a
JOIN sys.dm_os_performance_counters b
    ON a.object_name = b.object_name
WHERE a.object_name LIKE '%:Buffer Manager%'
    AND a.counter_name = 'Buffer cache hit ratio'
    AND b.counter_name = 'Buffer cache hit ratio base';
```

```sql
-- Memory grants pending — queries waiting for memory
-- Should be 0 under normal conditions
SELECT
    cntr_value AS memory_grants_pending,
    CASE
        WHEN cntr_value > 10 THEN 'CRITICAL — Many queries waiting for memory grants'
        WHEN cntr_value > 0 THEN 'WARNING — Some queries waiting for memory'
        ELSE 'NORMAL — No memory grant waits'
    END AS grant_assessment
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:Memory Manager%'
    AND counter_name = 'Memory Grants Pending';
```

```sql
-- Memory clerk breakdown — top consumers
SELECT TOP 20
    type AS clerk_type,
    memory_node_id,
    pages_kb / 1024 AS pages_mb,
    virtual_memory_committed_kb / 1024 AS virtual_committed_mb,
    virtual_memory_reserved_kb / 1024 AS virtual_reserved_mb,
    CASE
        WHEN type LIKE '%BUFFER%' THEN 'Buffer Pool'
        WHEN type LIKE '%CACHESTORE%' THEN 'Plan Cache'
        WHEN type LIKE '%USERSTORE%' THEN 'User Store'
        WHEN type LIKE '%QUERY%' THEN 'Query Optimizer'
        WHEN type LIKE '%OBJECTSTORE%' THEN 'Object Store'
        WHEN type LIKE '%MEMORYCLERK%' THEN 'Internal'
        ELSE 'Other'
    END AS memory_category
FROM sys.dm_os_memory_clerks
ORDER BY pages_kb DESC;
```

### 3.4 Concurrency Metrics

```sql
-- Current blocking chains
-- Returns session-level blocking info
SELECT
    blocked.session_id AS blocked_session_id,
    blocked.blocking_session_id AS blocking_session_id,
    blocked.wait_type AS blocked_wait_type,
    blocked.wait_time AS blocked_wait_time_ms,
    blocked.wait_resource AS blocked_wait_resource,
    blocked.database_id AS blocked_db_id,
    DB_NAME(blocked.database_id) AS database_name,
    blocked.command AS blocked_command,
    DATEDIFF(SECOND, blocked.last_wait_type, GETDATE()) AS blocked_seconds,
    blocker.host_name AS blocker_host,
    blocker.program_name AS blocker_program,
    blocker.login_name AS blocker_login,
    blocker.command AS blocker_command,
    blocker_status.text AS blocker_sql_text,
    blocked_status.text AS blocked_sql_text
FROM sys.dm_exec_requests blocked
INNER JOIN sys.dm_exec_requests blocker
    ON blocked.blocking_session_id = blocker.session_id
    AND blocked.blocking_session_id > 0
    AND blocked.session_id != @@SPID
CROSS APPLY sys.dm_exec_sql_text(blocker.sql_handle) AS blocker_status
CROSS APPLY sys.dm_exec_sql_text(blocked.sql_handle) AS blocked_status
WHERE blocked.blocking_session_id > 0
ORDER BY blocked.wait_time DESC;
```

## 5. Monitoring & Alerting — Threshold Configuration

### 5.1 Tiered Alert Strategy

| Alert Level | Response Time | Example Thresholds | Notification Channel |
|---|---|---|---|
| Critical | < 5 minutes | PLE < 100, Avg Runnable > 50, Disk < 5% free | PagerDuty, SMS, Teams |
| Warning | < 15 minutes | PLE < 300, Avg Runnable > 20, Latency > 20ms | Email, Teams |
| Informational | Daily digest | Trend changes, baseline deviations | Dashboard, email report |

### 5.2 Baseline Methodology

Thresholds must be derived from historical baselines, not generic values. A production OLTP server with 256GB RAM will have a very different healthy PLE than a 16GB development server.

```sql
-- Calculate baseline from historical snapshot data
-- Run this after at least 2 weeks of data collection
SELECT
    MetricCategory,
    MetricName,
    COUNT(*) AS sample_count,
    CAST(AVG(MetricValue) AS DECIMAL(10,2)) AS avg_value,
    CAST(STDEV(MetricValue) AS DECIMAL(10,2)) AS stddev_value,
    CAST(MIN(MetricValue) AS DECIMAL(10,2)) AS min_value,
    CAST(MAX(MetricValue) AS DECIMAL(10,2)) AS max_value,
    CAST(AVG(MetricValue) - 2 * STDEV(MetricValue) AS DECIMAL(10,2)) AS lower_alert_bound,
    CAST(AVG(MetricValue) + 2 * STDEV(MetricValue) AS DECIMAL(10,2)) AS upper_alert_bound
FROM dbo.HealthSnapshotLog
WHERE CaptureTime >= DATEADD(DAY, -14, GETDATE())
GROUP BY MetricCategory, MetricName
ORDER BY MetricCategory, MetricName;
```

```sql
    MetricValue,
    LAG(MetricValue) OVER (
        PARTITION BY MetricCategory, MetricName
        ORDER BY CaptureTime
    ) AS previous_value,
    MetricValue - LAG(MetricValue) OVER (
        PARTITION BY MetricCategory, MetricName
        ORDER BY CaptureTime
    ) AS change_from_previous,
    CASE
        WHEN AVG(MetricValue) OVER (
            PARTITION BY MetricCategory, MetricName
            ORDER BY CaptureTime
            ROWS BETWEEN 13 PRECEDING AND 1 PRECEDING
        ) > 0
        THEN (MetricValue - AVG(MetricValue) OVER (
            PARTITION BY MetricCategory, MetricName
            ORDER BY CaptureTime
            ROWS BETWEEN 13 PRECEDING AND 1 PRECEDING
        )) / AVG(MetricValue) OVER (
            PARTITION BY MetricCategory, MetricName
            ORDER BY CaptureTime
            ROWS BETWEEN 13 PRECEDING AND 1 PRECEDING
        ) * 100
        ELSE 0
    END AS pct_change_from_14day_avg
FROM dbo.HealthSnapshotLog;
```

## 6. Troubleshooting — Common Issues and Resolutions

### 6.1 High CPU — Diagnosis Flow

```
High CPU detected (task manager / perfmon)
        │
        ▼
Check sys.dm_os_schedulers.runnable_tasks_count
        │
        ├── > 20 sustained → CPU pressure confirmed
        │       │
        │       ▼
        │   Check sys.dm_exec_query_stats for high CPU queries
        │       │
        │       ▼
        │   Look for: missing indexes (table scans), 
        │   non-SARGable predicates, implicit conversions,
        │   excessive parallelism (CXPACKET waits)
        │
        └── < 10 runnable → Not CPU pressure
                │
                ▼
            Check signal_wait_time_ms in sys.dm_os_wait_stats
                │
                ├── Signal waits > 25% → CPU pressure from signal queue
                └── Signal waits < 15% → Look elsewhere (IO, blocking)
```

### 6.2 High IO — Diagnosis Flow

```
High IO detected (Avg Read > 20ms, PAGEIOLATCH > 30%)
        │
        ▼
Check Page Life Expectancy
        │
        ├── PLE < 300 → MEMORY PRESSURE causing IO
        │       │
        │       ▼
        │   Solution: Add more RAM or reduce buffer pool consumption
        │
        └── PLE > 600 → STORAGE SUBSYSTEM issue
                │
                ▼
            Check avg read/write latency per file
                │
                ├── One file slow → File placement or disk contention
                ├── All files slow → Storage array issue
                └── Log file slow → Check log growth patterns, VLFs
```

### 6.3 High Blocking — Diagnosis Flow

```
High lock waits detected (LCK_* > 10% of total waits)
        │
        ▼
Run the blocking query from Section 3.4
        │
        ├── Blocking head found
        │       │
        │       ▼
        │   Check blocker's last_wait_type, open_tran, query text
        │   Determine: long-running transaction? Missing index?
        │   Application pattern (SELECT then UPDATE in loop)?
        │
        └── No current blocking
                │
                ▼
            Check sys.dm_tran_locks for lock escalation
            Check for implicit transactions
            Review deadlock graphs from system_health session
```

### 6.4 Low Memory — Diagnosis Flow

```
Low PLE / Memory Grants Pending > 0
        │
        ▼
Check sys.dm_os_memory_clerks for top consumers
        │
        ├── Buffer pool is largest → Normal for OLTP
        │   Check if max server memory is set appropriately
        │
        ├── Plan cache is large → Check for ad-hoc queries
        │   Solution: Enable forced parameterization or optimize for ad-hoc
        │
        └── CACHESTORE_* are large → Check object cache
            Solution: Check for unparameterized queries with many variants
```

### 7.1 Application Insights SQL Tracking

```csharp
// Custom SQL Server metrics reporter for Application Insights
// Sends key DMV metrics as custom telemetry
public class SqlServerMetricsReporter : BackgroundService
{
    private readonly TelemetryClient _telemetryClient;
    private readonly string _connectionString;
    private readonly ILogger<SqlServerMetricsReporter> _logger;

    public SqlServerMetricsReporter(
        TelemetryClient telemetryClient,
        IConfiguration configuration,
        ILogger<SqlServerMetricsReporter> logger)
    {
        _telemetryClient = telemetryClient;
        _connectionString = configuration.GetConnectionString("AdminConnection");
        _logger = logger;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            try
            {
                await ReportMetricsAsync(stoppingToken);
            }
            catch (Exception ex)
            {
                _logger.LogError(ex, "Failed to report SQL Server metrics");
            }

            await Task.Delay(TimeSpan.FromMinutes(5), stoppingToken);
        }
    }

    private async Task ReportMetricsAsync(CancellationToken ct)
    {
        using var connection = new Microsoft.Data.SqlClient.SqlConnection(_connectionString);
        await connection.OpenAsync(ct);

        using var command = connection.CreateCommand();
        command.CommandText = @"
            SELECT
                (SELECT AVG(runnable_tasks_count * 1.0)
                 FROM sys.dm_os_schedulers
                 WHERE status = 'VISIBLE ONLINE' AND scheduler_id < 255) AS AvgRunnableTasks,
                (SELECT cntr_value
                 FROM sys.dm_os_performance_counters
                 WHERE object_name LIKE '%:Buffer Manager%'
                   AND counter_name = 'Page Life Expectancy') AS PageLifeExpectancy,
                (SELECT cntr_value
                 FROM sys.dm_os_performance_counters
                 WHERE object_name LIKE '%:General Statistics%'
                   AND counter_name = 'User Connections') AS UserConnections;
        ";

        using var reader = await command.ExecuteReaderAsync(ct);
        if (await reader.ReadAsync(ct))
        {
            _telemetryClient.TrackMetric("SQL-Server-AvgRunnableTasks",
                Convert.ToDouble(reader["AvgRunnableTasks"]));
            _telemetryClient.TrackMetric("SQL-Server-PageLifeExpectancy",
                Convert.ToDouble(reader["PageLifeExpectancy"]));
            _telemetryClient.TrackMetric("SQL-Server-UserConnections",
                Convert.ToDouble(reader["UserConnections"]));
        }
    }
}
```

```csharp
                data["PageLifeExpectancy"] = ple;
                data["BufferCacheHitRatio"] = cacheHit;
                data["MemoryGrantsPending"] = memGrants;

                var issues = new List<string>();

                if (avgRunnable > 20)
                    issues.Add($"CPU pressure: AvgRunnableTasks = {avgRunnable}");

                if (ple < 300)
                    issues.Add($"Memory pressure: PLE = {ple}");

                if (cacheHit < 95)
                    issues.Add($"Buffer cache hit ratio low: {cacheHit}%");
```

## 9. Gotchas — Common Pitfalls and Misconceptions

### 9.1 Counters Reset on Restart

```sql
-- ALWAYS check how long since SQL Server started before interpreting counters
SELECT
    sqlserver_start_time,
    DATEDIFF(DAY, sqlserver_start_time, GETDATE()) AS uptime_days
FROM sys.dm_os_sys_info;
```

**Why this matters:** Batch Requests/sec counter shows total since startup. If the server restarted 2 hours ago, the value will be low. You must either:
- Divide by uptime to get an average rate
- Or capture snapshots over time and compute deltas

### 9.2 VM vs Physical — Thresholds Differ

| Metric | Physical | VM (Properly Configured) | VM (Over-committed) |
|---|---|---|---|
| Avg Read Latency | < 10ms | < 20ms (VM overhead) | > 50ms |
| Avg Write Latency | < 5ms | < 10ms | > 30ms |
| PLE (with 64GB) | > 3000s | > 1000s | < 300s |

VM monitoring requires adjusted thresholds. The hypervisor adds latency that is invisible from inside the guest. If you suspect VM issues, check the host-side metrics.

### 9.3 100% CPU Is Not Always Bad

SQL Server is designed to use available CPU. 100% CPU is only a problem if:
- **Is is sustained** (hours, not minutes)
- **There is a runnable queue backlog** (runnable_tasks_count > 10)
- **Signal waits are high** (> 25%)
- **User complaints correlate** with high CPU periods

During a data warehouse ETL load, 100% CPU is expected and acceptable. During normal OLTP hours, 50-70% CPU might be normal. The key is baseline comparison.

### 9.4 PAGEIOLATCH vs PAGEIOLATCH_SH

```sql
-- Common confusion: PAGEIOLATCH_SH is not the same as PAGEIOLATCH_EX
SELECT
    wait_type,
    wait_time_ms,
    waiting_tasks_count,
    CASE wait_type
        WHEN 'PAGEIOLATCH_SH' THEN 'Page IO — Shared (read) — Most common IO wait'
        WHEN 'PAGEIOLATCH_EX' THEN 'Page IO — Exclusive (write) — Less common'
        WHEN 'PAGEIOLATCH_UP' THEN 'Page IO — Update'
        ELSE ''
    END AS description
FROM sys.dm_os_wait_stats
WHERE wait_type LIKE 'PAGEIOLATCH%'
ORDER BY wait_time_ms DESC;
```

PAGEIOLATCH_SH (shared) dominates most workloads. High PAGEIOLATCH_EX might indicate tempdb contention or index maintenance storms.

### 9.5 PLE Is Not a Universal Number

Page Life Expectancy varies dramatically based on:
- Total server memory (256GB server should have PLE > 10,000)
- Workload type (DW workloads naturally have lower PLE)
- Number of databases sharing the buffer pool

**The rule of thumb:** PLE should be at least 300 seconds _per 4GB of buffer pool_. A server with 64GB allocated to SQL should have PLE > 4,800.

### 9.6 Buffer Cache Hit Ratio Can Be Misleading

99% cache hit ratio sounds great, but consider:
- If you read 10,000 pages/sec at 99% hit ratio, you're still doing 100 physical reads/sec
- 100 physical reads/sec × 8KB = 800KB/sec from disk — still significant
- The 1% that misses the cache is what users experience as slow queries

Always pair cache hit ratio with Page Reads/sec to get the full picture.

### 9.7 Lazy Writes vs Checkpoint Pages

```sql
-- Distinguishing lazy writes from checkpoint activity
SELECT
    counter_name,
    cntr_value
FROM sys.dm_os_performance_counters
WHERE object_name LIKE '%:Buffer Manager%'
    AND counter_name IN (
        'Lazy writes/sec',
        'Checkpoint pages/sec',
        'Page writes/sec'
    );
```

Lazy writes indicate memory pressure (forced page eviction). Checkpoint pages are normal write activity. If lazy writes are a significant percentage of page writes, you have memory pressure.

### 9.8 Don't Chase Symptom Metrics

The most common mistake: fixing the metric instead of the root cause.
- Low PLE → Add more memory (might work)
- Low PLE → Missing indexes causing excessive reads (better fix)
- High CPU → Add more CPUs (might work if parallelism)
- High CPU → Non-SARGable predicates causing table scans (better fix)

Always ask **why** the metric is off baseline, not just how to make it look better.

### 9.9 DMV Query Impact

```sql
-- DMV queries are lightweight but can still cause contention
-- Avoid joining multiple large DMVs without filtering
-- Use TOP and WHERE clauses to limit rows
-- Test your monitoring queries during peak load before deploying

-- Bad: joining all DMVs for a "super query"
SELECT *
FROM sys.dm_exec_query_stats qs
CROSS JOIN sys.dm_exec_requests r  -- Don't do this — huge cross product
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle);

-- Good: targeted, filtered queries
SELECT TOP 10
    qs.total_worker_time / qs.execution_count AS avg_cpu,
    qs.total_logical_reads / qs.execution_count AS avg_reads
FROM sys.dm_exec_query_stats qs
WHERE qs.execution_count > 100
ORDER BY avg_cpu DESC;
```

### 9.10 Correlation Is Not Causation

Just because two metrics change at the same time doesn't mean one causes the other. The classic example:
- PLE drops at the same time as Page Reads/sec increases
- Is it memory pressure (low PLE) causing the reads? Or heavy reads (missing index) causing pages to flush from cache?
- The answer requires examining query patterns, not just the metrics.

Always validate DMV-based hypotheses with query-level analysis (`sys.dm_exec_query_stats`) before making configuration changes.
