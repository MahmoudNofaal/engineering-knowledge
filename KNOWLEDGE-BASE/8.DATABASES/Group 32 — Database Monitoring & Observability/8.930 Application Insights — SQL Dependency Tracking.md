# 8.930 Application Insights — SQL Dependency Tracking

Priority: 2 | Domain: Databases | Group: Database Monitoring & Observability

---

## Overview — Introduction & Purpose

Application Insights (AI) automatically collects SQL dependency telemetry from .NET applications that use Microsoft.Data.SqlClient (or System.Data.SqlClient). Every SQL query, stored procedure call, and batch command becomes a dependency telemetry item that captures duration, success/failure status, target server, and (when enabled) full query text. This provides application-aware SQL performance monitoring without requiring any database-level configuration, additional agents, or changes to SQL Server itself.

The value proposition is unique: unlike server-side monitoring (DMV-based tools, Extended Events, or Prometheus exporters), Application Insights traces SQL calls back to specific application operations, requests, and user contexts. This enables teams to answer questions like "Which API endpoint is causing the most SQL Server CPU?" or "What page load generates the slowest database query?" — questions that server-side monitoring simply cannot answer.

Key capabilities include:

- **Automatic instrumentation**: No code changes required for SQL execution in .NET applications using standard ADO.NET patterns. The AI SDK hooks into SqlClient at the connection level via diagnostic listeners.
- **End-to-end correlation**: SQL dependencies appear as child telemetry items under the parent request, enabling distributed tracing across service boundaries.
- **Query-level detail**: SQL Server name, database name, command text, duration, and result code are captured for each dependency call.
- **Performance analysis**: The AI Performance blade aggregates SQL dependency data to show slowest queries, most frequent queries, and failure rates.
- **Smart detection and alerts**: Built-in anomaly detection identifies unusual patterns in SQL dependency duration, failure rate, and volume.

This note covers how Application Insights SQL dependency tracking works, how to enable and configure it, how to interpret the data in the Azure Portal, and the production considerations and limitations to be aware of.

It is part of the Database Monitoring & Observability group and relates to other SQL monitoring approaches including [[8.928 — SQL Server Dashboards — Grafana Setup]] (server-side Prometheus monitoring) and [[8.931 — EF Core Logging — SQL Output Configuration]] (ORM-level logging).

## Architecture — How Dependency Tracking Works

Application Insights SQL dependency tracking relies on the .NET DiagnosticSource infrastructure. When the AI SDK is configured, it subscribes to diagnostic events emitted by Microsoft.Data.SqlClient (and previously System.Data.SqlClient). Each SQL command execution emits begin and end events that the SDK captures, transforms into dependency telemetry, and sends to Application Insights.

### Telemetry Flow

1. A user request enters the application (e.g., HTTP GET /api/orders).
2. ASP.NET Core middleware creates an AI `RequestTelemetry` item and starts tracking the request duration.
3. During request processing, business logic calls EF Core or Dapper to execute a SQL query.
4. EF Core/Dapper internally uses Microsoft.Data.SqlClient, which emits a diagnostic event when the SQL command begins execution.
5. The AI SDK's dependency tracking module (DependencyTrackingTelemetryModule) catches this event.
6. The SDK creates a `DependencyTelemetry` object with:
   - `type`: "SQL" (or "SQL Server")
   - `target`: server name and database name
   - `name`: command text (or first 20K characters)
   - `data`: the full command text (when enabled)
   - `duration`: measured timing of the SQL execution
   - `success`: true if the SQL command completed without exception
   - `resultCode`: SQL Server error code (0 = success, else error number)
   - `timestamp`: start time of the SQL call
7. The dependency is automatically associated as a child of the current request telemetry (parent-child relationship via `operation_Id` and `operation_parentId`).
8. Telemetry items are batched and sent to the Application Insights ingestion endpoint every 30 seconds (configurable) or when the batch reaches 500 items.
9. In Application Insights, the dependency telemetry can be viewed in the Performance blade, Failures blade, Transaction Search, and via Log Analytics / Kusto queries.

### Dependency Correlation

The critical architectural feature is **distributed tracing**. Each dependency telemetry item is correlated with its parent request:

- The request has a unique `operation_Id` (same across all child operations for the same request)
- The dependency has an `id` that matches the request's `operation_parentId`
- This allows AI to construct a tree view of operations: Request → Service Dependency → SQL Dependency

This correlation works across service boundaries when HTTP calls are instrumented. For example, Service A calls Service B via HTTP, and Service B calls SQL Server. The complete chain is visible as a single transaction in Application Insights:

```
GET /api/orders (Service A)
  └─ HTTP POST /api/data (Service B)
       └─ SQL SELECT * FROM Orders (Service B → SQL Server)
```

## Setup — Enabling SQL Dependency Tracking

### Step 1: Add Application Insights to ASP.NET Core

The simplest way to enable SQL dependency tracking is via the Microsoft.ApplicationInsights.AspNetCore NuGet package with the AddApplicationInsightsTelemetry extension method.

**Program.cs (minimal API or startup):**

```csharp
// .NET 6+ minimal API or Program.cs
WebApplicationBuilder builder = WebApplication.CreateBuilder(args);

// Add Application Insights to the DI container
builder.Services.AddApplicationInsightsTelemetry(builder.Configuration["ApplicationInsights:ConnectionString"]);

WebApplication app = builder.Build();
await app.RunAsync();
```

**appsettings.json:**

```json
{
  "ApplicationInsights": {
    "ConnectionString": "InstrumentationKey=00000000-0000-0000-0000-000000000000;IngestionEndpoint=https://your-region.in.applicationinsights.azure.com/"
  }
}
```

The connection string (preferred over instrumentation key) contains both the instrumentation key and the ingestion endpoint URL. It can be configured via environment variable `APPLICATIONINSIGHTS_CONNECTION_STRING` for runtime configuration without code changes.

**For versions older than .NET 6 (Startup.cs):**

```csharp
public void ConfigureServices(IServiceCollection services)
{
    services.AddApplicationInsightsTelemetry(Configuration["ApplicationInsights:ConnectionString"]);
}
```

### Step 2: Verify SQL Dependency Tracking is Enabled

By default, `DependencyTrackingTelemetryModule` is automatically registered when you call `AddApplicationInsightsTelemetry()`. It includes `SqlClientDependencyTracking` for tracking SQL Server (and other SQL databases) calls.

To verify SQL dependency tracking is active:

1. Deploy your application and run some operations that execute SQL.
2. Go to Azure Portal > Application Insights > Transaction Search.
3. Filter by `Dependency` type and look for dependency items with type "SQL".
4. If no SQL dependencies appear, check the troubleshooting section below.

### Step 3: Disable or Configure SQL Capabilities

In some cases, you may want to disable SQL dependency tracking (e.g., sensitive data concerns, overhead). Use `ApplicationInsightsServiceOptions`:

```csharp
builder.Services.ConfigureTelemetryModule<DependencyTrackingTelemetryModule>((module, options) =>
{
    // Disable SQL command text collection
    module.EnableSqlCommandTextInstrumentation = false;
});

// Or disable SQL dependency tracking entirely
builder.Services.AddApplicationInsightsTelemetry(options =>
{
    options.EnableDependencyTracking = false;
    // Re-enable other telemetry modules as needed
});
```

### Step 4: Enable SQL Text Collection (with Caution)

By default, the AI SDK does **not** collect SQL command text in recent versions due to cost and PII concerns. To enable it:

```csharp
builder.Services.ConfigureTelemetryModule<DependencyTrackingTelemetryModule>((module, options) =>
{
    module.EnableSqlCommandTextInstrumentation = true;
});
```

When this flag is enabled, the `DependencyTelemetry.Data` property contains the SQL command text. This is visible in Transaction Search individual items but **not aggregated** in Performance blade queries.

### Framework Compatibility

| Framework | SQL Client | Auto-instrumentation | Notes |
|-----------|-----------|---------------------|-------|
| .NET 8+ | Microsoft.Data.SqlClient 5.x | Full support | Recommended, best performance |
| .NET 6/7 | Microsoft.Data.SqlClient 4.x | Full support | Will work, upgrade if possible |
| .NET Core 3.1 | Microsoft.Data.SqlClient 2.x | Full support | Extended support ending |
| .NET Framework 4.7.2+ | System.Data.SqlClient | Supported via Status Monitor | Requires host on Azure VM/on-prem agent |
| .NET Framework 4.6.1 | System.Data.SqlClient | Limited | Use Status Monitor v2 |

For .NET Framework applications not running in Azure, use Application Insights Status Monitor (an IIS module that attaches to the runtime) or the Azure Monitor OpenTelemetry distro.

## Viewing — Analyzing SQL Dependency Data in Azure Portal

Application Insights provides several views for analyzing SQL dependency performance. Each serves a different purpose and audience.

### Performance Blade — Dependencies Tab

The primary tool for SQL performance analysis in Application Insights.

**Navigation**: Azure Portal → Application Insights resource → Performance → Dependencies

This blade shows:

- **Dependency operations table**: Lists distinct SQL commands (truncated to first ~500 characters) with aggregate statistics:
  - Count (number of executions)
  - Median, 95th percentile, 99th percentile duration
  - Failure rate (%)
  - Impact (weighted contribution to overall request latency)

- **Dependency duration distribution**: Histogram of SQL call durations, showing how execution times are distributed

- **Top 3 dependency impact**: Pie or horizontal bar chart showing which SQL calls contribute most to total request duration

**Workflow for identifying slow SQL queries**:

1. Open the Performance blade and select the Dependencies tab
2. Sort by 95th percentile duration descending — this shows the slowest SQL calls
3. Click on a dependency to view:
   - **Call duration over time**: Time-series chart showing how the SQL call's duration varies (useful for identifying intermittent slowness)
   - **Samples**: Actual individual telemetry samples with full details (server name, database, command text)
   - **Related requests**: Which application operations triggered this SQL call
   - **Suggested actions**: AI may suggest indexing or query tuning if patterns match known issues

4. To investigate further, open the Samples tab for a specific dependency and click on any sample to see the end-to-end transaction details

### Failures Blade — SQL Dependency Failures

**Navigation**: Azure Portal → Application Resources → Failures → Dependencies

This blade surfaces SQL dependency errors categorized by:

- **Exception type**: TimeoutException, SqlException, InvalidOperationException, etc.
- **Result code**: SQL Server error numbers (e.g., 1205 = deadlock, 1222 = lock request timeout)
- **Failed dependency count** over time

Common SQL Server errors visible here:

| Error Code | Meaning | Common Cause |
|-----------|---------|-------------|
| -2 | Timeout expired | Query exceeds CommandTimeout, blocking |
| 1205 | Deadlock victim | Concurrent transactions with conflicting resource access orders |
| 1222 | Lock request time-out | Query waiting too long for a lock |
| 18456 | Login failed | Connection string issues, credential problems |
| 4060 | Cannot open database | Wrong database name in connection string |
| 2812 | Could not find stored procedure | Deployments without synchronizing schemas |

### Transaction Search — Deep Dive

**Navigation**: Azure Portal → Application Insights → Transaction Search

This is the raw data explorer for Application Insights. Filter by:

- **Event type**: Dependency
- **Dependency type**: SQL (or SQL Server)
- **Duration**: Custom range (e.g., > 5000ms for slow queries)
- **Success**: False to see failed SQL calls

Use Transaction Search when you need to:

- Verify that SQL dependency tracking is working (do a quick search after deployment)
- Examine individual SQL calls with full detail (target server, command text, duration, timestamp)
- Correlate a specific slow SQL call with its parent request and other dependencies
- Export data for custom analysis (CSV export available)

### Log Analytics (Kusto Queries)

For programmatic analysis, use Log Analytics with Kusto Query Language. The dependency data is stored in the `dependencies` table.

**Basic SQL dependency status:**

```kusto
dependencies
| where timestamp > ago(1d)
| where type == "SQL"
| summarize
    TotalCalls = count(),
    AvgDuration = avg(duration),
    P95Duration = percentile(duration, 95),
    P99Duration = percentile(duration, 99),
    FailureRate = (countif(success == false) * 100.0) / count()
    by name
| top 20 by P95Duration desc
| project name, TotalCalls, AvgDuration, P95Duration, P99Duration, FailureRate
```

**Slowest queries by duration:**

```kusto
dependencies
| where timestamp > ago(6h)
| where type == "SQL"
| where duration > 5000  // slower than 5 seconds
| project timestamp, name, duration, success, resultCode, target, cloud_RoleName
| order by duration desc
| take 100
```

**Correlate slow SQL with operations:**

```kusto
dependencies
| where timestamp > ago(1d)
| where type == "SQL"
| where duration > 2000
| join kind=inner (
    requests
    | where timestamp > ago(1d)
    | project operation_Id, operationName, url, resultCode
) on operation_Id
| project timestamp, duration, name, operationName, url, target
| order by duration desc
```

**Identify top databases by load:**

```kusto
dependencies
| where timestamp > ago(7d)
| where type == "SQL"
| extend Database = extract("\\| ([^ ]+)", 1, target)
| summarize
    TotalCalls = count(),
    AvgDuration = avg(duration),
    TotalDuration = sum(duration)
    by Database
| top 10 by TotalDuration desc
```

**Deadlock detection:**

```kusto
dependencies
| where timestamp > ago(1d)
| where type == "SQL"
| where resultCode == "1205"
| summarize DeadlockCount = count() by bin(timestamp, 15m)
| render timechart
```

## Alerts — Setting Up SQL Performance Alerts

Application Insights alerting enables proactive notification when SQL dependency patterns deviate from normal behavior.

### Types of Alerts

**Metric Alert** — Fires when a specific metric crosses a threshold:

- Duration: P95 or average SQL dependency duration above threshold
- Failure rate: percentage of failed SQL calls above threshold
- Count: unusual spike or drop in SQL dependency count

**Log Alert** — Fires when a Kusto query returns results matching a condition:

- Specific error codes: deadlock (1205), lock timeout (1222), etc.
- Slowest queries exceeding custom threshold
- Dependency failures from a specific server or database

**Smart Detection** — AI automatically detects anomalies without manual configuration:

- Anomalous rise in dependency duration (e.g., one service tier suddenly experiencing 3x normal latency)
- Failed dependencies (unusual increase in failure rate)
- Degradation in dependency response time (sustained increase over baseline)

### Setting Up a Metric Alert for SQL Duration

**Azure Portal**: Monitor → Alerts → Create → Alert rule

1. **Scope**: Select your Application Insights resource
2. **Condition**: Custom log search with the following query:

```kusto
dependencies
| where type == "SQL"
| summarize AggregatedValue = percentile(duration, 95) by bin(timestamp, 5m)
```

3. **Measurement**: AggregatedValue (P95 duration)
4. **Threshold**: > 5000 (5 seconds) — adjust based on baseline
5. **Evaluation frequency**: Every 5 minutes
6. **Lookback period**: Last 30 minutes
7. **Actions**: Create action group with email/SMS/PagerDuty/Slack

### Setting Up a Log Alert for SQL Failures

**Alert logic**:

```kusto
dependencies
| where timestamp > ago(5m)
| where type == "SQL"
| where success == false
| summarize FailureCount = count()
| where FailureCount > 10  // more than 10 failures in 5 minutes
```

**Threshold**: Result count > 0

**Evaluation**: Every 5 minutes, look at last 5 minutes.

### Setting Up Smart Detection Alerts

Smart Detection rules are partially pre-configured. To enable or configure:

1. Navigate to Application Insights > Smart Detection
2. Locate the rules:
   - **Degradation in dependency response time**: Detects when response times degrade compared to the last 7 days
   - **Failed dependencies**: Detects abnormal increase in failed dependency calls (requires 10+ failures/day baseline)
3. Enable the rules and configure action groups

### Alert Action Groups

Configure action groups to route alerts to appropriate channels:

- **Severity 0 (Critical)**: PagerDuty/OpsGenie + SMS to DBA on-call for production databases
- **Severity 1 (Error)**: Email to DBA team + Slack #dba-alerts for non-critical production databases
- **Severity 2 (Warning)**: Slack #database-monitoring for informational alerts and trending concerns
- **Severity 3 (Informational)**: No direct notification, logged for dashboard/reporting

### Best Practices for Alert Configuration

1. **Baseline first**: Run the application for 1-2 weeks in production to establish baseline SQL performance before setting alert thresholds. Use the Metric Explorer to view natural P95 and P99 latency distributions.

2. **Avoid alert fatigue**: Start with high thresholds (P99 duration > 10s) and tighten over time. Not every slow query needs an alert — only queries that indicate a real problem.

3. **Tag alerts with context**: Include service name, database name, and environment in alert payloads so responders can quickly triage without navigating multiple dashboards.

4. **Use dynamic thresholds** (Preview): AI Metric Alerts can calculate thresholds based on historical patterns rather than static values. Enable "Dynamic threshold" in alert rule configuration.

5. **Correlate with deployments**: Alert on SQL duration spikes within 15 minutes of a new deployment. Configure separate alerts for "deployment windows" with higher thresholds.

## Sampling — Impact on SQL Dependency Data

Application Insights uses sampling to control the volume of telemetry data sent to the service. Sampling directly affects SQL dependency data — when a request is sampled out, its child dependencies are also sampled out.

### Types of Sampling

**Fixed-rate sampling**: A consistent percentage of all telemetry is kept. For example, 25% sampling keeps every 4th request and all its dependencies. This is predictable but does not adapt to traffic variations.

**Adaptive sampling** (default for ASP.NET Core): Automatically adjusts the sampling rate based on incoming telemetry volume. During high-traffic periods, sampling drops more telemetry to stay within the configured maximum rate (default 5 telemetry items per second). During low-traffic periods, sampling is reduced or disabled entirely.

**Ingestion sampling**: The Application Insights service can drop telemetry at ingestion time based on its own sampling rules. This is visible in the Azure Portal as a difference between "data ingested" and "data analyzed."

### Sampling Impact on SQL Dependency Visibility

- **Individual query tracing**: When a request is sampled out, all its SQL dependencies are also dropped. You cannot trace an individual slow query if its parent request was not sampled.
- **Aggregated metrics**: The Performance blade uses a separate aggregation pipeline that is not affected by client-side sampling. The duration percentiles and counts shown in the Dependencies tab are accurate, even if individual telemetry items are sampled.
- **Log Analytics queries**: Queries against the `dependencies` table only contain sampled data. For accurate counts, use the `itemCount` field which accounts for sampling weight.

**Example — querying with sampling weight:**

```kusto
dependencies
| where timestamp > ago(1d)
| where type == "SQL"
| summarize
    TotalExecutions = sum(itemCount),  // items + sampling weight
    SampledCount = count()              // actual telemetry items
    by name
| project name, TotalExecutions, SampledCount
| order by TotalExecutions desc
```

### Configuring Sampling for SQL Visibility

To ensure SQL dependency data is useful, configure adaptive sampling to allow higher rates during critical periods:

```csharp
// Program.cs — configure adaptive sampling
builder.Services.Configure<TelemetryConfiguration>((config) =>
{
    // Increase max telemetry rate for better SQL visibility
    config.DefaultTelemetrySink.TelemetryProcessorChainBuilder
        .UseAdaptiveSampling(
            excludedTypes: "Exception",  // always capture exceptions
            maxTelemetryItemsPerSecond: 10,  // default is 5
            samplingPercentage: 100  // start at 100% and adapt downward
        );
    config.DefaultTelemetrySink.TelemetryProcessorChainBuilder.Build();
});
```

**Key settings:**

- `maxTelemetryItemsPerSecond`: Higher values preserve more dependency data. Default is 5 items/s. For heavy SQL monitoring, consider 10-20.
- `excludedTypes`: Never sample "Exception" (always capture SQL errors). Optionally add "Dependency" to exclude from sampling entirely (but will increase cost).
- `samplingPercentage`: Starting point (100% = no sampling until traffic volume triggers reduction).

### Cost Implications

Sampling is Application Insights' primary cost control mechanism. More sampling = lower cost but less data. The trade-off:

| Sampling Rate | Monthly AI Cost (estimate) | SQL Visibility |
|--------------|---------------------------|----------------|
| 100% (no sampling) | High | Full detail on every query |
| 25% | Low | Aggregate data only, no per-query tracing |
| Adaptive (default) | Medium | Good aggregate data, some per-query tracing during off-peak |

For most production applications, adaptive sampling with `maxTelemetryItemsPerSecond` of 10-15 provides a good balance between cost and visibility.

## Production — Operational Considerations

### Cloud Role Name Configuration

When monitoring multi-service applications, configure `cloud_RoleName` to identify which service generates each SQL dependency. This is essential for distinguishing between dependencies from different microservices.

```csharp
// Program.cs
builder.Services.AddApplicationInsightsTelemetry(options =>
{
    options.ConnectionString = configuration["ApplicationInsights:ConnectionString"];
});

// Set cloud role name
builder.Services.AddSingleton<ITelemetryInitializer, CloudRoleNameInitializer>();

// Telemetry initializer implementation
public class CloudRoleNameInitializer : ITelemetryInitializer
{
    public void Initialize(ITelemetry telemetry)
    {
        if (telemetry is DependencyTelemetry dependency && dependency.Type == "SQL")
        {
            telemetry.Context.Cloud.RoleName = "OrderService";
        }
    }
}
```

### SQL Server Security Configuration

**SQL Server firewall**: If the application runs outside Azure or in a different VNet, configure SQL Server firewall to accept traffic from Application Insights IP ranges (not needed — Application Insights does not connect to SQL Server; the application does).

**SQL Client IP tracking**: Application Insights logs the target server and database but **not** the client IP address. If you need client IP for security auditing, add it as a custom property on the dependency telemetry.

**PII and GDPR**: SQL dependency telemetry may contain personal data (customer names, emails, account numbers) if full query text is captured. See the Gotchas section for PII handling.

### Dependency Telemetry Overhead

The AI SDK adds minimal overhead to SQL execution — approximately 0.1-1ms per SQL call in most benchmarks. This overhead comes from:

1. DiagnosticSource event subscription and dispatching (sub-microsecond dispatch, negligible)
2. Telemetry item creation and property population (~0.01-0.1ms)
3. Telemetry channel batching and serialization (asynchronous, not on hot path)
4. Network transmission (asynchronous batched, 30s interval, out-of-process)

In practice, the overhead is imperceptible for most applications. For ultra-low-latency applications (sub-millisecond SQL calls), consider:

- Disabling dependency tracking in the critical path
- Sampling to reduce volume
- Using a custom lightweight instrumentation approach (e.g., only tracking calls that exceed a duration threshold)

### Retention and Data Management

Application Insights retains telemetry according to the workspace retention policy:

- Default: 90 days for Log Analytics workspace-based (newer) applications
- Extended: up to 730 days with additional cost
- Continuous Export (classic): export to storage for custom retention (being deprecated in favor of the Log Analytics Workspace export feature)

For long-term SQL performance trend analysis, export dependency data to a SQL Database, Azure Data Explorer, or Azure Storage for custom retention beyond 90 days.

### Managing High Volume of SQL Telemetry

High-volume applications can generate millions of SQL dependency telemetry items per day. Strategies to manage this:

1. **Increase adaptive sampling rate**: Accept lower sample rate for dependency data.
2. **Exclude known-fast queries**: Use a telemetry processor to filter out trivial SQL calls (duration < 5ms):

```csharp
public class FastSqlFilter : ITelemetryProcessor
{
    private readonly ITelemetryProcessor _next;

    public FastSqlFilter(ITelemetryProcessor next) => _next = next;

    public void Process(ITelemetry item)
    {
        if (item is DependencyTelemetry dep && dep.Type == "SQL" && dep.Duration.TotalMilliseconds < 5)
        {
            return; // drop trivial SQL calls
        }
        _next.Process(item);
    }
}
```

3. **Focus on outlier endpoints**: Only track SQL dependencies for specific API endpoints or services rather than the entire application.
4. **Use separate AI resources**: Split application telemetry by component, with dedicated AI resources for high-volume services.

## Multi-Service — Tracking Across Microservices

In a microservices architecture, SQL dependencies may be generated by multiple services that share the same database. Application Insights provides two mechanisms for correlating these: cloud role name and distributed tracing.

### Distributed Tracing with W3C TraceContext

When Service A calls Service B via HTTP, and both are instrumented with AI SDK:

1. Service A's outgoing HTTP request propagates `traceparent` and `tracestate` headers (W3C TraceContext format).
2. Service B's incoming request automatically picks up these headers to continue the trace.
3. When Service B calls SQL Server, the dependency telemetry inherits the propagated operation context.
4. In Transaction Search, the complete chain is visible: Service A Request → Service B Request → SQL Dependency.

This requires:

- Both services instrumented with AI SDK version 2.12+ (TraceContext support added in 2.11+)
- HTTP call headers propagated (automatic with HttpClient instrumentation)
- `ApplicationInsightsServiceOptions.EnableW3CCorrelation = true` (default in recent SDK versions)

### Managing Cloud Role Names

Set `cloud_RoleName` in each service to distinguish SQL dependencies by service origin:

- OrderService → `cloud_RoleName = "OrderService"` → SQL dependency tagged with "OrderService"
- PaymentService → `cloud_RoleName = "PaymentService"` → SQL dependency tagged with "PaymentService"

This allows queries like:

```kusto
dependencies
| where timestamp > ago(7d)
| where type == "SQL"
| summarize
    AvgDuration = avg(duration),
    TotalCalls = count()
    by cloud_RoleName, name
| order by TotalCalls desc
```

### Shared Database Monitoring Challenges

When multiple services target the same database:

1. **Server-level monitoring** (Prometheus/DMV) shows total load but cannot attribute it to a specific service.
2. **AI dependency tracking** shows per-service load but may miss queries executed outside the application (SSMS ad-hoc queries, scheduled jobs, ETL processes).
3. **Combined approach**: Use both AI dependency tracking (per-service) and Prometheus/DMV monitoring (per-server) for full visibility.

A dashboard combining both data sources (e.g., Grafana with Prometheus + AI data) provides the most comprehensive picture.

## Troubleshooting — Common Issues & Gotchas

### Missing SQL Dependencies in Application Insights

**Problem**: Application Insights is configured but no SQL dependencies appear.

**Checklist**:

1. **Verify SQL dependency tracking is enabled**: Confirmed default behavior of `AddApplicationInsightsTelemetry()`. Check `ApplicationInsightsServiceOptions.EnableDependencyTracking` is not explicitly set to false.

2. **Check SDK version**: AI SDK versions prior to 2.7.0 had incomplete SQL dependency support. Update to the latest version (2.21+ recommended).

3. **Verify SQL Client version**: Microsoft.Data.SqlClient 2.0+ supports DiagnosticSource events. Older versions may not emit events AI can capture.

4. **Check for custom SqlClient wrappers**: If the application wraps SqlClient in a custom abstraction that does not use the standard ADO.NET patterns, AI may not be able to instrument the calls. Ensure EF Core or Dapper uses the standard SqlClient internally.

5. **Check firewall / network**: Dependency telemetry is sent via HTTPS to the AI ingestion endpoint. If the application network blocks outbound HTTPS traffic, telemetry will not reach AI.

6. **Check Application Map**: Go to Application Insights > Application Map. If the application appears as an isolated node without SQL connection, dependencies are not being tracked.

7. **Check sampling configuration**: Overly aggressive sampling may drop most dependency telemetry. Check the sampling percentage in live metrics.

**Debug with custom tracking:**

```csharp
// Temporarily add explicit dependency tracking to verify pipeline
using (var client = new TelemetryClient())
{
    var dep = new DependencyTelemetry("SQL", "test-server", "test-command", DateTimeOffset.UtcNow, TimeSpan.FromMilliseconds(1), true);
    client.TrackDependency(dep);
}
```

If this appears in Transaction Search but real SQL dependencies don't, the issue is with SqlClient instrumentation specifically.

### SQL Text Truncation at 20K Characters

Application Insights truncates SQL command text at 20,000 characters. For very long queries, stored procedures, or batch commands:

- The `name` field contains the truncated command text
- The `data` field (when command text collection is enabled) also truncates at 20K
- No option exists to increase this limit in the current SDK

For queries that exceed 20K, identify them by query hash or a truncated prefix. Store the full query text in a separate logging system (e.g., file logs, EventLog) linked by a correlation ID.

### PII and Sensitive Data in SQL Telemetry

**Risk**: SQL dependency telemetry full query text may contain PII (personally identifiable information) such as customer names, email addresses, social security numbers, password hashes, or credit card data.

**Mitigation options**:

1. **Disable SQL command text collection** (default in recent SDKs): `EnableSqlCommandTextInstrumentation = false`. This prevents any SQL text from being sent to AI.

2. **Custom telemetry processor** to sanitize SQL text:

```csharp
public class SqlSanitizationProcessor : ITelemetryProcessor
{
    private readonly ITelemetryProcessor _next;
    private static readonly Regex EmailRegex = new Regex(@"\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}\b", RegexOptions.Compiled);
    private static readonly Regex SsnRegex = new Regex(@"\b\d{3}-\d{2}-\d{4}\b", RegexOptions.Compiled);

    public SqlSanitizationProcessor(ITelemetryProcessor next) => _next = next;

    public void Process(ITelemetry item)
    {
        if (item is DependencyTelemetry dep && dep.Type == "SQL" && dep.Data != null)
        {
            string sanitized = dep.Data;
            sanitized = EmailRegex.Replace(sanitized, "***@***.com");
            sanitized = SsnRegex.Replace(sanitized, "***-**-****");
            dep.Data = sanitized;
        }
        _next.Process(item);
    }
}
```

3. **Database-level data masking**: Use SQL Server's Dynamic Data Masking to mask sensitive columns. This prevents the data from appearing in query results at the application level.

4. **Use parameterized queries** (always, for security): Parameters appear in the SQL text as `@p0`, `@p1`, etc., not as actual values. However, if the SDK captures the parameter values separately, they may appear in telemetry.

**Compliance implications**:

- **GDPR**: If SQL text captured by AI contains personal data of EU residents, the data processor agreement between Microsoft and the customer applies. The customer (application owner) remains responsible for ensuring only necessary data is captured.
- **HIPAA**: Azure Application Insights is not HIPAA-eligible when collecting full SQL text with PHI. Configure sanitization or disable text collection.
- **PCI-DSS**: Full credit card numbers must not appear in SQL telemetry. Use tokenization or disable text collection.

### Adaptive Sampling Drops Dependency Data

**Problem**: Applications under heavy load show gaps in SQL dependency data, and some slow queries are only detected server-side (via DMVs) but not in Application Insights.

**Cause**: Adaptive sampling reduces telemetry volume during high traffic. It uses a sliding window of the past 60 seconds to estimate current traffic and applies sampling percentage based on `maxTelemetryItemsPerSecond`. When requests are sampled out, all their dependencies (including slow SQL queries) are also dropped.

**Mitigation**:

1. Increase `maxTelemetryItemsPerSecond` to preserve more dependency data
2. Set `samplingPercentage` floor to ensure at least ~10% sampling even under max load
3. Exclude dependency telemetry from sampling:

```csharp
config.DefaultTelemetrySink.TelemetryProcessorChainBuilder
    .UseAdaptiveSampling(
        excludedTypes: "Exception;Dependency",  // exclude dependency from sampling
        maxTelemetryItemsPerSecond: 5
    );
```

4. Combine with server-side monitoring (Prometheus + Grafana) for guaranteed SQL performance data

### Inconsistent Dependency Duration Measurements

**Problem**: SQL dependency durations reported by Application Insights are consistent but differ from SQL Server's own measurements (DMV query stats, Profiler, Extended Events).

**Expected differences**:

- Application Insights measures **client-side elapsed time**: time from when SqlClient sends the command to when it receives the first result. This includes network latency, SQL Server processing time, and client-side serialization.
- SQL Server DMVs (`sys.dm_exec_query_stats`) measure **server-side CPU/duration**: time spent within SQL Server only.
- Network latency (typically 1-10ms in Azure, 10-50ms cross-region) is included in AI dependency duration but not in SQL Server's DMV measurements.

**When to investigate significant discrepancies**:

- If AI dependency duration is consistently 2x+ longer than DMV-reported duration, check for network latency, client-side processing, or serialization bloat.
- If AI dependency duration shows intermittent spikes not visible in DMV data, suspect client-side blocking (connection pool exhaustion, thread pool starvation) or resource contention on the application server.

### Telemetry Lost During High Volume

**Problem**: During traffic spikes, AI dependency telemetry is incomplete or shows gaps.

**Causes**:

1. **Channel capacity exceeded**: The in-memory buffer (default 500 items) overflows if the telemetry channel cannot send data fast enough.
2. **Ingestion throttling**: Application Insights has per-resource ingestion limits (varies by pricing tier).
3. **Sampling**: Adaptive sampling drops telemetry.

**Solutions**:

1. Increase telemetry channel buffer size and batch interval:
```csharp
builder.Services.Configure<TelemetryConfiguration>((config) =>
{
    config.TelemetryChannel = new InMemoryChannel
    {
        MaxTelemetryBufferCapacity = 2000,  // default 500
        SendingInterval = TimeSpan.FromSeconds(10)  // default 30
    };
});
```

2. Upgrade to a higher Application Insights pricing tier with higher ingestion limits.
3. Consider splitting telemetry across multiple AI resources for very high-volume services.

### SQL Client IP Not Tracked

Application Insights does not capture the client IP address for SQL connections. The `target` property contains the SQL Server hostname and database name, not the client IP. If client IP auditing is required (for security compliance), add it as a custom property:

```csharp
public class SqlClientIpInitializer : ITelemetryInitializer
{
    private readonly IHttpContextAccessor _httpContextAccessor;

    public SqlClientIpInitializer(IHttpContextAccessor httpContextAccessor)
    {
        _httpContextAccessor = httpContextAccessor;
    }

    public void Initialize(ITelemetry telemetry)
    {
        if (telemetry is DependencyTelemetry dep && dep.Type == "SQL")
        {
            var context = _httpContextAccessor.HttpContext;
            if (context?.Connection?.RemoteIpAddress != null)
            {
                dep.Properties["ClientIp"] = context.Connection.RemoteIpAddress.ToString();
            }
        }
    }
}
```
