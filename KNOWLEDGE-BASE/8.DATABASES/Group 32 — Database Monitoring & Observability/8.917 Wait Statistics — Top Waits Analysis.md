## 1. Overview - What Wait Statistics Tell You

Wait statistics are the single most important diagnostic tool in SQL Server performance analysis. Every time a query cannot immediately execute - because it needs a page from disk, waits for a lock, or waits for CPU time - SQL Server records that delay as a wait. The `sys.dm_os_wait_stats` DMV aggregates all waits since the last server restart, providing a cumulative picture of where the database engine spends its time waiting.

### 1.1 Wait Stats Philosophy

The core premise of wait-based tuning is simple: find out what queries are waiting on, address the biggest contributors, and performance improves proportionally. This is often called the wait-based or waits and queues approach to performance tuning.

### 1.2 Why Priority 1 (Critical)

Wait statistics is classified as Priority 1 because:
- It is the first thing to check when any performance issue arises
- It points directly to the bottleneck subsystem (CPU, IO, Memory, Locks)
- It requires no special tools - just T-SQL access to a DMV
- It is available on every edition of SQL Server, including Express

### 1.3 What This Note Covers

- The anatomy of wait types and how they are recorded
- The default wait stats query and how to filter noise
- Signal vs resource waits and what each means
- Cumulative vs delta analysis
- Production-ready wait stat snapshots
- Common pitfalls and misinterpretations

## 2. Core Concepts - Wait Types, Signal Waits, and Resource Waits

### 2.1 The Three States of a Query Thread

A query thread passes through three states during execution:
- **Runnable**: Thread is ready to execute but waiting for a CPU to become available
- **Running**: Thread is actively executing on a CPU
- **Waiting**: Thread is blocked waiting for a resource (IO, lock, memory)

```
Thread Lifecycle:  Runnable -> Running -> Waiting -> Runnable -> Running -> Done
                        |                        |          |
                    CPU queue              Resource wait    Signal wait
                                          (IO, lock, etc.) (CPU queue)
```

### 2.2 Two Components of Every Wait

Every wait has two components captured in `sys.dm_os_wait_stats`:

| Component | Column | Description |
|---|---|---|
| Wait time | wait_time_ms | Total time the thread spent waiting (signal + resource) |
| Signal wait time | signal_wait_time_ms | Time spent in the runnable queue waiting for CPU |
| Resource wait time | (computed) | wait_time_ms - signal_wait_time_ms = time waiting for the resource |

```sql
-- Understanding the wait time split
SELECT
    wait_type,
    wait_time_ms,
    signal_wait_time_ms,
    (wait_time_ms - signal_wait_time_ms) AS resource_wait_time_ms,
    waiting_tasks_count,
    wait_time_ms / NULLIF(waiting_tasks_count, 0) AS avg_wait_ms_per_task,
    signal_wait_time_ms / NULLIF(waiting_tasks_count, 0) AS avg_signal_wait_ms,
    CAST(signal_wait_time_ms * 100.0 / NULLIF(wait_time_ms, 0) AS DECIMAL(5,2)) AS signal_wait_pct
FROM sys.dm_os_wait_stats
WHERE wait_time_ms > 0
    AND waiting_tasks_count > 0
    AND wait_type NOT IN (
        'BROKER_EVENT_HANDLER', 'BROKER_INTERNAL', 'BROKER_RECEIVE_WAIT',
        'BROKER_TO_FLUSH', 'BROKER_TRANSMITTER', 'CHECKPOINT_QUEUE',
```

### 2.3 Signal Wait vs Resource Wait Interpretation

When signal waits are more than 25% of total waits, the bottleneck is CPU - threads are waiting in the runnable queue. When signal waits are low (under 15%), the bottleneck is a resource (IO, lock, memory).

```sql
-- Signal wait ratio analysis
SELECT
    SUM(wait_time_ms) AS total_wait_ms,
    SUM(signal_wait_time_ms) AS total_signal_wait_ms,
    SUM(wait_time_ms) - SUM(signal_wait_time_ms) AS total_resource_wait_ms,
    CAST(SUM(signal_wait_time_ms) * 100.0 / NULLIF(SUM(wait_time_ms), 0) AS DECIMAL(5,2)) AS signal_wait_pct,
    CASE
        WHEN SUM(signal_wait_time_ms) * 100.0 / NULLIF(SUM(wait_time_ms), 0) > 25
            THEN 'CRITICAL -- CPU pressure, threads waiting in runnable queue'
        WHEN SUM(signal_wait_time_ms) * 100.0 / NULLIF(SUM(wait_time_ms), 0) > 15
            THEN 'ELEVATED -- Monitor CPU subsystem'
        ELSE 'NORMAL -- Resource waits dominate, CPU is not the bottleneck'
    END AS signal_wait_assessment
FROM sys.dm_os_wait_stats
WHERE wait_type NOT IN (
    'BROKER_EVENT_HANDLER', 'BROKER_INTERNAL', 'BROKER_RECEIVE_WAIT',
```

### 3.2 What Each Column Returns

The `sys.dm_os_wait_stats` DMV returns these key columns:

- **wait_type**: Name of the wait (PAGEIOLATCH_SH, LCK_M_S, SOS_SCHEDULER_YIELD, etc.)
- **wait_time_ms**: Total accumulated wait time in milliseconds since server restart
- **waiting_tasks_count**: Number of individual wait events that occurred
- **signal_wait_time_ms**: Portion of wait_time_ms spent in the CPU runnable queue
- **max_wait_time_ms**: (SQL 2017+) Maximum single wait duration for this type
- **max_waiting_tasks_count**: (SQL 2017+) Maximum concurrent tasks waiting simultaneously

The resource wait time is computed: `wait_time_ms - signal_wait_time_ms`

### 3.3 How SQL Server Records Waits - Event Flow

When a thread needs a page from disk:
1. Thread enters PAGEIOLATCH_SH wait, wait_time_ms starts counting
2. IO subsystem starts reading the page from disk
3. Thread is in WAITING state - signal_wait_time_ms stays at 0, wait_time_ms accumulates
4. IO completes, page is in buffer pool - thread becomes RUNNABLE
5. Now signal_wait_time_ms starts counting - thread is waiting for CPU scheduler
6. Scheduler picks up thread, it becomes RUNNING - both counters stop

Example: If a query spends 200ms reading from disk and 50ms in CPU queue:
- wait_time_ms = 250ms (total)
- signal_wait_time_ms = 50ms (CPU queue)
- resource_wait_time_ms = 200ms (IO wait time)
- signal_pct = 50 / 250 = 20%

### 3.5 Cumulative vs Delta Analysis

Wait stats accumulate from server startup. To measure a specific time window, use delta analysis:

```sql
-- Create snapshot table
IF NOT EXISTS (SELECT * FROM sys.tables WHERE name = 'WaitStatsSnapshot')
BEGIN
    CREATE TABLE dbo.WaitStatsSnapshot (
        SnapshotId INT IDENTITY(1,1) PRIMARY KEY,
        CaptureTime DATETIME2 NOT NULL DEFAULT SYSDATETIME(),
        wait_type NVARCHAR(120) NOT NULL,
        wait_time_ms BIGINT NOT NULL,
        waiting_tasks_count BIGINT NOT NULL,
        signal_wait_time_ms BIGINT NOT NULL
    );
END
GO

-- Capture first snapshot
INSERT INTO dbo.WaitStatsSnapshot (CaptureTime, wait_type, wait_time_ms, waiting_tasks_count, signal_wait_time_ms)
SELECT SYSDATETIME(), wait_type, wait_time_ms, waiting_tasks_count, signal_wait_time_ms
FROM sys.dm_os_wait_stats;
GO
```

```sql
-- Delta analysis between two snapshots
WITH S1 AS (
    SELECT wait_type, wait_time_ms, waiting_tasks_count, signal_wait_time_ms, CaptureTime
    FROM dbo.WaitStatsSnapshot WHERE SnapshotId = 1
),
S2 AS (
    SELECT wait_type, wait_time_ms, waiting_tasks_count, signal_wait_time_ms, CaptureTime
    FROM dbo.WaitStatsSnapshot WHERE SnapshotId = 2
)
SELECT
    COALESCE(s1.wait_type, s2.wait_type) AS wait_type,
    DATEDIFF(SECOND, s1.CaptureTime, s2.CaptureTime) AS elapsed_seconds,
    s2.wait_time_ms - ISNULL(s1.wait_time_ms, 0) AS delta_wait_time_ms,
    s2.waiting_tasks_count - ISNULL(s1.waiting_tasks_count, 0) AS delta_tasks,
    s2.signal_wait_time_ms - ISNULL(s1.signal_wait_time_ms, 0) AS delta_signal_ms,
    (s2.wait_time_ms - ISNULL(s1.wait_time_ms, 0))
        / NULLIF(DATEDIFF(SECOND, s1.CaptureTime, s2.CaptureTime), 0) AS wait_ms_per_sec,
    CAST((s2.wait_time_ms - ISNULL(s1.wait_time_ms, 0)) * 100.0
        / NULLIF(SUM(s2.wait_time_ms - ISNULL(s1.wait_time_ms, 0)) OVER (), 0) AS DECIMAL(5,2)) AS pct_of_total
FROM S2 s2 LEFT JOIN S1 s1 ON s2.wait_type = s1.wait_type
WHERE s2.wait_time_ms - ISNULL(s1.wait_time_ms, 0) > 0
ORDER BY delta_wait_time_ms DESC;
```

### 3.6 Per-Second Wait Rates

```sql
-- Current wait rates per second since server start
SELECT TOP 15
    w.wait_type,
    w.wait_time_ms,
    w.waiting_tasks_count,
    w.wait_time_ms / NULLIF(DATEDIFF(SECOND, s.sqlserver_start_time, GETDATE()), 0) AS wait_ms_per_sec,
    w.waiting_tasks_count / NULLIF(DATEDIFF(SECOND, s.sqlserver_start_time, GETDATE()), 0) AS waits_per_sec,
    CAST(w.signal_wait_time_ms * 100.0 / NULLIF(w.wait_time_ms, 0) AS DECIMAL(5,2)) AS signal_pct,
    CASE
        WHEN w.wait_type LIKE 'PAGEIOLATCH%' THEN 'IO'
        WHEN w.wait_type LIKE 'LCK%' THEN 'LOCK'
        WHEN w.wait_type = 'SOS_SCHEDULER_YIELD' THEN 'CPU'
        WHEN w.wait_type LIKE 'LATCH%' THEN 'LATCH'
        WHEN w.wait_type = 'WRITELOG' THEN 'LOG'
        WHEN w.wait_type IN ('CXPACKET','CXCONSUMER') THEN 'PARALLEL'
        WHEN w.wait_type IN ('RESOURCE_SEMAPHORE','RESOURCE_SEMAPHORE_QUERY_COMPILE') THEN 'MEMORY'
        WHEN w.wait_type = 'THREADPOOL' THEN 'WORKER'
        WHEN w.wait_type = 'ASYNC_NETWORK_IO' THEN 'NETWORK'
        ELSE 'OTHER'
    END AS category
FROM sys.dm_os_wait_stats w
CROSS JOIN sys.dm_os_sys_info s
ORDER BY wait_time_ms DESC;
```

### 4.2 Weekly Trend Analysis View

```sql
CREATE OR ALTER VIEW monitoring.vw_WeeklyWaitTrend
AS
WITH SnapshotRanked AS (
    SELECT CaptureTime, wait_type, wait_time_ms, waiting_tasks_count, signal_wait_time_ms,
           DENSE_RANK() OVER (ORDER BY CaptureTime DESC) AS snapshot_rank
    FROM monitoring.WaitStatsHistory
),
Current AS (SELECT * FROM SnapshotRanked WHERE snapshot_rank = 1),
Previous AS (SELECT * FROM SnapshotRanked WHERE snapshot_rank = 2)
SELECT
    c.wait_type,
    c.CaptureTime AS current_time,
    p.CaptureTime AS previous_time,
    DATEDIFF(HOUR, p.CaptureTime, c.CaptureTime) AS hours_between,
    c.wait_time_ms - ISNULL(p.wait_time_ms, 0) AS delta_wait_time_ms,
    c.waiting_tasks_count - ISNULL(p.waiting_tasks_count, 0) AS delta_tasks,
    c.signal_wait_time_ms - ISNULL(p.signal_wait_time_ms, 0) AS delta_signal_ms,
    CAST((c.wait_time_ms - ISNULL(p.wait_time_ms, 0)) * 100.0
        / NULLIF(SUM(c.wait_time_ms - ISNULL(p.wait_time_ms, 0)) OVER (), 0) AS DECIMAL(5,2)) AS pct_of_total
FROM Current c
LEFT JOIN Previous p ON c.wait_type = p.wait_type
WHERE c.wait_time_ms - ISNULL(p.wait_time_ms, 0) > 0
HAVING hours_between > 0;
GO
```

### 4.3 Wait Stats Alerting Procedure

```sql
CREATE OR ALTER PROCEDURE monitoring.CheckWaitStatThresholds
AS
BEGIN
    SET NOCOUNT ON;

    DECLARE @LatestId INT, @PreviousId INT;
    SELECT @LatestId = MAX(SnapshotId) FROM dbo.WaitStatsSnapshot;
    SELECT @PreviousId = MAX(SnapshotId) FROM dbo.WaitStatsSnapshot WHERE SnapshotId < @LatestId;

    IF @PreviousId IS NULL BEGIN SELECT 'Insufficient snapshots' AS Status; RETURN; END;

    ;WITH Delta AS (
        SELECT s2.wait_type,
               s2.wait_time_ms - ISNULL(s1.wait_time_ms, 0) AS delta_wait_ms,
               s2.waiting_tasks_count - ISNULL(s1.waiting_tasks_count, 0) AS delta_tasks
        FROM dbo.WaitStatsSnapshot s2
        LEFT JOIN dbo.WaitStatsSnapshot s1
            ON s2.wait_type = s1.wait_type AND s1.SnapshotId = @PreviousId
        WHERE s2.SnapshotId = @LatestId
    ),
    Total AS (SELECT SUM(delta_wait_ms) AS total_ms FROM Delta)
    SELECT
        d.wait_type, d.delta_wait_ms, d.delta_tasks,
        CAST(d.delta_wait_ms * 100.0 / NULLIF(t.total_ms, 0) AS DECIMAL(5,2)) AS pct_of_total,
        CASE
            WHEN d.wait_type LIKE 'PAGEIOLATCH%' AND d.delta_wait_ms * 100.0 / NULLIF(t.total_ms, 0) > 50
                THEN 'ALARM: IO subsystem bottleneck'
            WHEN d.wait_type LIKE 'LCK%' AND d.delta_wait_ms * 100.0 / NULLIF(t.total_ms, 0) > 20
                THEN 'ALARM: Severe lock contention'
            WHEN d.wait_type = 'SOS_SCHEDULER_YIELD' AND d.delta_wait_ms * 100.0 / NULLIF(t.total_ms, 0) > 30
                THEN 'ALARM: CPU pressure'
            WHEN d.wait_type IN ('RESOURCE_SEMAPHORE','RESOURCE_SEMAPHORE_QUERY_COMPILE')
                AND d.delta_wait_ms * 100.0 / NULLIF(t.total_ms, 0) > 15
                THEN 'ALARM: Memory pressure'
            WHEN d.wait_type = 'THREADPOOL' AND d.delta_tasks > 10
                THEN 'ALARM: Worker thread exhaustion'
            WHEN d.wait_type IN ('CXPACKET','CXCONSUMER')
                AND d.delta_wait_ms * 100.0 / NULLIF(t.total_ms, 0) > 20
                THEN 'WARNING: Excessive parallelism waits'
            ELSE 'INFO: Within normal parameters'
        END AS alert
    FROM Delta d CROSS JOIN Total t
    WHERE d.delta_wait_ms > 0
    ORDER BY d.delta_wait_ms DESC;
END
GO
```

### 4.4 .NET Integration - Wait Stats Reader

```csharp
                signal_wait_time_ms,
                CASE WHEN wait_time_ms > 0 AND waiting_tasks_count > 0
                    THEN wait_time_ms / waiting_tasks_count ELSE 0 END AS avg_wait_ms,
                CASE WHEN wait_time_ms > 0
                    THEN CAST(signal_wait_time_ms * 100.0 / wait_time_ms AS DECIMAL(5,2))
                    ELSE 0 END AS signal_pct
            FROM sys.dm_os_wait_stats
            WHERE wait_type NOT IN ('BROKER_EVENT_HANDLER','SLEEP_TASK','WAITFOR','XE_TIMER_EVENT')
            ORDER BY wait_time_ms DESC;";
        command.Parameters.AddWithValue("@Top", top);

        using var reader = await command.ExecuteReaderAsync(ct);
        while (await reader.ReadAsync(ct))
        {
            results.Add(new WaitStatEntry
            {
                WaitType = reader.GetString(0),
                WaitTimeMs = reader.GetInt64(1),
                WaitingTasksCount = reader.GetInt64(2),
                SignalWaitTimeMs = reader.GetInt64(3),
                AvgWaitMs = reader.GetInt64(4),
                SignalPct = reader.GetDecimal(5)
            });
        }
        return results;
    }
}

public class WaitStatEntry
{
    public string WaitType { get; set; }
    public long WaitTimeMs { get; set; }
    public long WaitingTasksCount { get; set; }
    public long SignalWaitTimeMs { get; set; }
    public long AvgWaitMs { get; set; }
    public decimal SignalPct { get; set; }

    public string Assessment => SignalPct > 25 ? "CPU Pressure"
        : WaitType switch
        {
            string w when w.StartsWith("PAGEIOLATCH") => "IO Pressure",
            string w when w.StartsWith("LCK_") => "Lock Contention",
            "WRITELOG" => "Log Write Pressure",
            "CXPACKET" or "CXCONSUMER" => "Parallelism",
            "RESOURCE_SEMAPHORE" => "Memory Pressure",
            _ => "Resource Contention"
        };
}
```

## 5. Monitoring & Alerting - Thresholds

### 5.1 Signal Wait Threshold Reference

| Signal % | Interpretation | Action |
|---|---|---|
| 0-10% | Resource waits dominate | Focus on top resource wait |
| 10-15% | Normal OLTP | No action |
| 15-25% | Elevated CPU queue | Investigate if sustained 1hr+ |
| 25-50% | CPU pressure | Check top CPU queries, MAXDOP |
| Over 50% | Severe CPU starvation | Immediate action needed |

### 5.2 Wait Type Thresholds for Alerting

| Wait Type/Category | Warning | Critical | Action |
|---|---|---|---|
| PAGEIOLATCH_* | Over 30% of total | Over 50% of total | Check disk latency, memory pressure |
| LCK_M_* | Over 10% of total | Over 20% of total | Check blocking chains |
| SOS_SCHEDULER_YIELD | Over 20% of total | Over 30% of total | Check CPU queries, MAXDOP |
| RESOURCE_SEMAPHORE | Over 5% of total | Over 15% of total | Check memory grants |
| THREADPOOL | Any non-zero | Over 10 tasks | Worker thread exhaustion |
| CXPACKET+CXCONSUMER | Over 10% of total | Over 20% of total | Check parallelism settings |
| WRITELOG | Over 15% of total | Over 30% of total | Check log disk speed |

### 5.3 Baseline-Based Alerting

Rather than using fixed thresholds, establish baselines from historical data:

```sql
-- Calculate baseline thresholds from 30 days of data
SELECT
    wait_type,
    COUNT(*) AS samples,
    AVG(CaptureTime) AS avg_capture_time,
    AVG(delta_wait_time_ms) AS avg_delta,
    STDEV(delta_wait_time_ms) AS stddev_delta,
    AVG(delta_wait_time_ms) + 2 * STDEV(delta_wait_time_ms) AS alert_upper_boundary
FROM (
    SELECT
        wait_type,
        CaptureTime,
        wait_time_ms - LAG(wait_time_ms) OVER (
            PARTITION BY wait_type ORDER BY CaptureTime
        ) AS delta_wait_time_ms
    FROM monitoring.WaitStatsHistory
) AS deltas
WHERE CaptureTime >= DATEADD(DAY, -30, GETDATE())
    AND delta_wait_time_ms > 0
GROUP BY wait_type
ORDER BY avg_delta DESC;
```

## 6. Troubleshooting - Common Patterns

### 6.1 IO Bottleneck Pattern

PAGEIOLATCH_SH dominates (over 40% of total waits), avg read latency over 20ms.

Investigation steps:
1. Check Page Life Expectancy - low PLE means memory pressure causing IO
2. Check missing indexes causing table scans
3. Check file latency from sys.dm_io_virtual_file_stats
4. Check if tempdb is causing IO contention

```sql
-- IO bottleneck investigation
SELECT
    DB_NAME(database_id) AS db,
    file_id,
    io_stall_read_ms / NULLIF(num_of_reads, 0) AS avg_read_latency_ms,
    io_stall_write_ms / NULLIF(num_of_writes, 0) AS avg_write_latency_ms
FROM sys.dm_io_virtual_file_stats(NULL, NULL)
WHERE num_of_reads > 0
ORDER BY avg_read_latency_ms DESC;
```

### 6.2 CPU Pressure Pattern

Signal waits over 25%, runnable tasks over 10, SOS_SCHEDULER_YIELD in top waits.

Investigation steps:
1. Find top CPU queries from sys.dm_exec_query_stats
2. Look for implicit conversions, non-SARGable predicates
3. Check for excessive parallelism (high CXPACKET alongside CPU)
4. Review MAXDOP and cost threshold settings

```sql
-- Top CPU consuming queries
SELECT TOP 10
    total_worker_time / execution_count / 1000 AS avg_cpu_ms,
    total_worker_time / 1000 AS total_cpu_ms,
    execution_count,
    SUBSTRING(st.text, (qs.statement_start_offset/2)+1,
        (CASE WHEN qs.statement_end_offset = -1 THEN LEN(st.text)
         ELSE qs.statement_end_offset - qs.statement_start_offset END)/2) AS query_text
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) AS st
WHERE execution_count > 10
ORDER BY total_worker_time / execution_count DESC;
```

### 6.3 Blocking Pattern

LCK_M_* waits over 10% of total, users report timeouts.

Investigation steps:
1. Run blocking chain query from sys.dm_exec_requests
2. Identify the blocker session
3. Check for implicit transactions holding locks
4. Consider Read Committed Snapshot Isolation (RCSI)

```sql
-- Current blocking chains
SELECT
    blocked.session_id AS blocked_id,
    blocked.blocking_session_id AS blocker_id,
    blocked.wait_type,
    blocked.wait_time,
    DB_NAME(blocked.database_id) AS db,
    blocker_status.text AS blocker_sql
FROM sys.dm_exec_requests blocked
INNER JOIN sys.dm_exec_requests blocker
    ON blocked.blocking_session_id = blocker.session_id
CROSS APPLY sys.dm_exec_sql_text(blocker.sql_handle) AS blocker_status
WHERE blocked.blocking_session_id > 0;
```

### 6.4 Memory Pressure Pattern

RESOURCE_SEMAPHORE or RESOURCE_SEMAPHORE_QUERY_COMPILE in top waits, PLE dropping.

Investigation steps:
1. Check Memory Grants Pending counter
2. Check max server memory setting
3. Look for large memory grants from hash joins
4. Check plan cache size for ad-hoc query bloat

```sql
-- Memory grant sizes by query hash
SELECT TOP 10
    query_hash,
    SUM(total_grant_kb) / 1024 AS total_grant_mb,
    AVG(avg_grant_kb) / 1024 AS avg_grant_mb,
    COUNT(*) AS variants
FROM sys.dm_exec_query_stats
GROUP BY query_hash
ORDER BY total_grant_mb DESC;
```

### 6.5 Worker Thread Exhaustion Pattern

THREADPOOL wait type appears, new connections fail.

This is critical. SQL Server cannot process new work until threads free up.

Checklist:
1. Check max_workers_count from sys.dm_os_sys_info
2. Identify long-running queries holding threads
3. Look for blocking chains consuming threads
4. Review application connection pool settings

```csharp
public class WaitStatsHealthCheck : IHealthCheck
{
    private readonly string _connectionString;

    public WaitStatsHealthCheck(IConfiguration configuration)
    {
        _connectionString = configuration.GetConnectionString("DefaultConnection");
    }

    public async Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext context,
        CancellationToken ct = default)
    {
        var data = new Dictionary<string, object>();
        try
        {
            using var connection = new SqlConnection(_connectionString);
            await connection.OpenAsync(ct);

            using var command = connection.CreateCommand();

                FROM sys.dm_os_wait_stats
                WHERE wait_type NOT IN ('BROKER_EVENT_HANDLER','SLEEP_TASK','WAITFOR')
                ORDER BY wait_time_ms DESC;";

            using var reader = await command.ExecuteReaderAsync(ct);
            while (await reader.ReadAsync(ct))
            {
                var wt = reader.GetString(0);
                data[wt] = reader.GetInt64(1);

                return HealthCheckResult.Unhealthy("Worker thread exhaustion", data: data);
            if (data.ContainsKey("RESOURCE_SEMAPHORE"))
                return HealthCheckResult.Degraded("Memory pressure", data: data);

            return HealthCheckResult.Healthy("No critical waits", data);
        }
        catch (Exception ex)
        {
            return HealthCheckResult.Unhealthy($"Check failed: {ex.Message}", exception: ex);
        }
    }
```

### 7.2 Application Insights Background Reporter

```csharp
public class WaitStatsReporter : BackgroundService
{
    private readonly TelemetryClient _telemetry;
    private readonly string _connectionString;

    public WaitStatsReporter(TelemetryClient telemetry, IConfiguration config)
    {
        _telemetry = telemetry;
        _connectionString = config.GetConnectionString("DefaultConnection");
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        var lastSnapshot = new Dictionary<string, long>();
        var lastTime = DateTime.UtcNow;

        while (!stoppingToken.IsCancellationRequested)
        {
            try
            {
                using var connection = new SqlConnection(_connectionString);
                await connection.OpenAsync(stoppingToken);

                using var cmd = connection.CreateCommand();
                cmd.CommandText = "SELECT wait_type, wait_time_ms FROM sys.dm_os_wait_stats;";

                var current = new Dictionary<string, long>();
                using var reader = await cmd.ExecuteReaderAsync(stoppingToken);
                while (await reader.ReadAsync(stoppingToken))
                    current[reader.GetString(0)] = reader.GetInt64(1);

                var now = DateTime.UtcNow;
                var elapsedSeconds = (now - lastTime).TotalSeconds;

                if (lastSnapshot.Count > 0 && elapsedSeconds > 0)
                {
                    foreach (var kvp in current)
                    {
                        if (lastSnapshot.TryGetValue(kvp.Key, out var lastVal))
                        {
                            var delta = kvp.Value - lastVal;
                            if (delta > 0)
                                _telemetry.TrackMetric($"SQL-Wait-{kvp.Key}", delta / elapsedSeconds);
                        }
                    }
                }

                lastSnapshot = current;
                lastTime = now;
            }
            catch { /* retry next cycle */ }

            await Task.Delay(TimeSpan.FromMinutes(5), stoppingToken);
        }
    }
}
```

## 9. Gotchas - Common Pitfalls

### 9.1 Waits Reset on Restart

`sys.dm_os_wait_stats` is cleared on SQL Server restart. Always check uptime before interpreting values.

```sql
SELECT sqlserver_start_time, DATEDIFF(DAY, sqlserver_start_time, GETDATE()) AS uptime_days
FROM sys.dm_os_sys_info;
```

### 9.2 Cumulative Data Requires Delta Analysis

Raw values are totals since startup, not rates. Always capture two snapshots and subtract for per-period analysis. Without this, you cannot distinguish stale history from current activity.

### 9.3 Signal Waits Over 25% = CPU Pressure

The most important threshold in wait stats: if signal wait time exceeds 25% of total wait time, the CPU is the bottleneck. Do not chase resource waits when CPU is the real issue.

### 9.4 Harmless Waits Inflate Totals

Failure to exclude background waits (BROKER_EVENT, SLEEP_TASK, WAITFOR, XE_TIMER_EVENT) inflates total wait time and dilutes the percentage of actionable waits. Always filter them.

### 9.5 CXPACKET Is Normal - Don't Set MAXDOP=1

CXPACKET is coordination overhead for parallel queries. Some amount is normal and healthy. Only worry if CXPACKET exceeds 20% of total waits. Setting MAXDOP=1 disables parallel execution entirely and often makes other problems worse.

### 9.6 High Wait Time Is Not Always Bad

High total wait time on a busy server is expected. The diagnostic insight comes from the distribution (which waits dominate) and the trend (is a category growing as a percentage over time).

### 9.7 WRITELOG Can Be Application Pattern

High WRITELOG waits may be caused by many small transactions, not just a slow log disk. Check for application patterns that issue many individual INSERTs within explicit transactions.

### 9.8 THREADPOOL Is an Emergency

Any non-zero THREADPOOL wait is serious. It means SQL Server has exhausted its worker thread pool and cannot process new requests. Investigate immediately by checking for long-running queries and blocking.

### 9.9 Correlation Is Not Causation

High PAGEIOLATCH might be caused by memory pressure (low PLE), not slow storage. High SOS_SCHEDULER_YIELD might be caused by parallelism skew, not query inefficiency. Always investigate the chain of causation before acting.

### 9.10 Use SET STATISTICS IO for Query-Level Validation

```sql
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

-- Your query here
SELECT * FROM Orders WHERE OrderDate >= '2026-01-01';

SET STATISTICS IO OFF;
SET STATISTICS TIME OFF;
```

Interpret STATISTICS IO output:
- logical reads: pages read from buffer pool
- physical reads: pages read from disk (should be near 0)
- scan count: number of table/index scans (1 = seeks, high = scans)
- CPU time vs elapsed time: large difference = waits
