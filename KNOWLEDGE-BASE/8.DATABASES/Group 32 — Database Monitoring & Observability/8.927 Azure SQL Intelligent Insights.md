# 8.927 Azure SQL Intelligent Insights

## 1. Overview — Intelligent Insights for Azure SQL

Azure SQL Intelligent Insights is a built-in AI-powered performance monitoring and diagnostics service for Azure SQL Database. It continuously analyzes telemetry data across 30+ performance metrics — including DTU/CPU consumption, IO throughput, log write rates, wait statistics, tempdb contention, and deadlock frequency — and automatically identifies root causes of performance degradation without manual configuration.

Unlike traditional monitoring approaches that require setting thresholds and defining alert rules, Intelligent Insights uses machine learning models trained on Azure SQL telemetry from thousands of databases. It establishes behavioral baselines for each database and detects anomalies that deviate from those baselines. When an anomaly is detected, it produces a natural-language root cause analysis (RCA) with specific recommendations for remediation.

The key differentiators of Intelligent Insights are:
- Zero configuration — No thresholds to set, no baseline to capture
- AI-driven detection — Anomaly detection based on learned behavior patterns
- Root cause analysis — Pinpoints the likely cause, not just the symptom
- Actionable recommendations — Provides specific T-SQL or configuration changes
- Native Azure integration — Works with Azure Monitor, alerts, and runbooks

Intelligent Insights is available for Azure SQL Database (both DTU and vCore models) and is being extended to Azure SQL Managed Instance. It is not available for SQL Server on Azure VMs (IaaS).

The service has some latency — insights are typically available 15 to 60 minutes after the observed behavior. This makes it ideal for post-incident analysis and trend monitoring rather than real-time alerting.

### 2.1 Metrics Analyzed

Intelligent Insights analyzes the following categories of metrics:

Resource Consumption:
- DTU / vCore utilization (percentage)
- CPU percentage
- Data IO percentage
- Log write percentage
- Memory consumption
- Session count

IO Performance:
- Data IO latency (avg, max)
- Log write latency (avg, max)
- Physical IO throughput
- IO queue depth
- Storage throughput

Query Performance:
- Average query duration
- Query execution count
- Compilation count
- Recompilation count
- Query wait time

Database Internals:
- Tempdb utilization (version store, allocation)
- Deadlock frequency
- Lock wait time
- Log growth rate
- Checkpoint duration

Wait Statistics:
- Wait time by wait category
- Top wait types
- Total wait time vs signal wait time

### 2.2 Machine Learning Models

The ML models that power Intelligent Insights are:

- Seasonal decomposition — Detects trends and seasonal patterns in metrics
- Anomaly detection — Identifies points where metric values deviate significantly from the learned baseline
- Change point detection — Detects sudden shifts in metric behavior
- Correlation analysis — Correlates anomalies across multiple metrics to identify root cause

These models are pre-trained and run automatically. There is no model training or tuning required from the user.

### 2.3 Insight Categories

Intelligent Insights categorizes detected issues into:

| Category | Description |
|----------|-------------|
| Resource Limitation | DTU/CPU/IO/Log write reached 100% |
| Query Regression | A query duration increased significantly |
| IO Bottleneck | Data or log IO latency exceeded threshold |
| Wait Bottleneck | Specific wait type dominates wait time |
| Deadlock Pattern | Deadlock frequency exceeds learned baseline |
| Tempdb Contention | Allocation or version store contention |
| Log Rate Limiting | Log write throughput is throttled |

Each insight includes severity level, time period affected, root cause analysis, and recommended actions.

## 3. Implementation — Querying Intelligent Insights Data

### 3.1 Viewing Insights in the Azure Portal

In the Azure portal, navigate to: Azure SQL Database, Monitoring, Intelligent Insights.

The blade shows:
- Timeline of insights with severity indicators
- Insight details with RCA text
- Recommended actions
- Related metrics

### 3.2 Querying Resource Metrics with T-SQL

Intelligent Insights data is accessible via system DMVs within the database:

```sql
-- Resource consumption per 15-second interval
-- Available in Azure SQL DB only
SELECT
    end_time,
    avg_cpu_percent,
    avg_data_io_percent,
    avg_log_write_percent,
    max_worker_percent,
    max_session_percent,
    dtu_limit,
    dtu_consumption_percent,
    avg_instance_cpu_percent,
    avg_instance_memory_percent,
    avg_login_rate_percent,
    avg_app_cpu_percent
FROM sys.dm_db_resource_stats
ORDER BY end_time DESC;
```

The sys.dm_db_resource_stats DMV returns one row per 15-second interval. It retains data for the last 15 minutes in Basic tiers and up to 2 hours for higher tiers. This is the most granular resource data available directly in T-SQL.

### 3.3 Resource Governance Information

```sql
-- View resource governance configuration and limits
SELECT
```

### 3.4 Intelligent Insights Queries

Intelligent Insights stores its analysis results in the sys.dm_intelligent_insights DMV:

```sql
-- Query all available insights
SELECT
    insight_id,

    end_time,
    severity,

    metric_details,

```

### 3.5 Querying Performance History with Query Store

Intelligent Insights correlates with Query Store data for query-level analysis:

```sql
-- Queries that may be causing IO pressure
SELECT TOP 20
    qt.query_sql_text,
    qsq.query_id,
    qsrs.avg_logical_io_reads,
    qsrs.avg_physical_io_reads,
    qsrs.avg_duration,
    qsrs.count_executions,
    qsrs.avg_cpu_time
FROM sys.query_store_runtime_stats qsrs
JOIN sys.query_store_plan qsqp ON qsrs.plan_id = qsqp.plan_id
JOIN sys.query_store_query qsq ON qsqp.query_id = qsq.query_id
JOIN sys.query_store_query_text qt ON qt.query_text_id = qsq.query_text_id
JOIN sys.query_store_runtime_stats_interval qsrsi
    ON qsrs.runtime_stats_interval_id = qsrsi.runtime_stats_interval_id
WHERE qsrsi.start_time > DATEADD(HOUR, -24, GETUTCDATE())
    AND qsrs.execution_type_desc = 'Regular'
ORDER BY qsrs.avg_logical_io_reads DESC;
```

### 3.6 Identifying Resource Bottlenecks

```sql
-- Recent resource consumption
DECLARE @HighCpuThreshold DECIMAL(5,2) = 80.0;

-- CPU bottleneck detection
SELECT
    end_time,
    avg_cpu_percent,
    avg_instance_cpu_percent,
    CASE
        WHEN avg_cpu_percent > @HighCpuThreshold
            OR avg_instance_cpu_percent > @HighCpuThreshold
        THEN 'CPU Bottleneck'
        ELSE 'Normal'
    END AS Status
FROM sys.dm_db_resource_stats
WHERE end_time > DATEADD(MINUTE, -30, GETUTCDATE())
    AND (avg_cpu_percent > @HighCpuThreshold
         OR avg_instance_cpu_percent > @HighCpuThreshold)
ORDER BY end_time DESC;

-- IO bottleneck via Query Store
SELECT TOP 10
    qt.query_sql_text,
    qsrs.avg_duration,
    qsrs.avg_logical_io_reads,
    qsrs.avg_physical_io_reads,
    qsrs.count_executions
FROM sys.query_store_runtime_stats qsrs
JOIN sys.query_store_plan qsqp ON qsrs.plan_id = qsqp.plan_id
JOIN sys.query_store_query qsq ON qsqp.query_id = qsq.query_id
JOIN sys.query_store_query_text qt ON qt.query_text_id = qsq.query_text_id
JOIN sys.query_store_runtime_stats_interval qsrsi
    ON qsrs.runtime_stats_interval_id = qsrsi.runtime_stats_interval_id
WHERE qsrsi.start_time > DATEADD(HOUR, -4, GETUTCDATE())
    AND qsrs.avg_physical_io_reads > 1000
    AND qsrs.execution_type_desc = 'Regular'
ORDER BY qsrs.avg_physical_io_reads DESC;
```

### 3.7 Log Write Performance Analysis

```sql
-- Log write percentage and throttling
SELECT
    end_time,
    avg_log_write_percent,
    CASE
        WHEN avg_log_write_percent >= 100 THEN 'Throttling active'
        WHEN avg_log_write_percent >= 80 THEN 'Near throttling'
        ELSE 'Normal'
    END AS LogWriteStatus
FROM sys.dm_db_resource_stats
WHERE end_time > DATEADD(HOUR, -2, GETUTCDATE())
    AND avg_log_write_percent > 80
ORDER BY end_time DESC;

-- Log write throttle events
SELECT
    name,
    value,
    value_data
FROM sys.dm_user_db_resource_governance
WHERE name LIKE '%log_write%' OR name LIKE '%throttle%';
```

## 4. Configuration — Setting Up Azure Alerts

### 4.1 Creating Alerts Based on Intelligent Insights

Alerts can be configured in the Azure portal or via ARM templates:

```powershell
# Azure CLI command to create an alert
az monitor metrics alert create `
    --name "HighDTUAlert" `
    --resource-group "MyResourceGroup" `
    --scopes "/subscriptions/.../providers/Microsoft.Sql/servers/myserver/databases/mydb" `
    --condition "avg dtu_consumption_percent > 90" `
    --window-size 15m `
    --evaluation-frequency 5m `
    --action-groups "/subscriptions/.../actionGroups/MyActionGroup"

# PowerShell equivalent
New-AzMetricAlertRuleV2 `
    -Name "HighDTUAlert" `
    -ResourceGroupName "MyResourceGroup" `
    -TargetResourceId "/subscriptions/.../databases/mydb" `
    -WindowSize (New-Timespan -Minutes 15) `
    -Frequency (New-Timespan -Minutes 5) `
    -Condition (New-AzMetricAlertRuleV2Criteria -MetricName "dtu_consumption_percent" -Operator GreaterThan -Threshold 90) `
    -ActionGroupId "/subscriptions/.../actionGroups/MyActionGroup"
```

### 4.2 Key Metrics for Alerting

The most important metrics to alert on:

| Metric | Threshold | Window | Description |
|--------|-----------|--------|-------------|
| dtu_consumption_percent | > 90% | 15 min | Resource exhaustion |
| cpu_percent | > 90% | 15 min | CPU bottleneck |
| data_io_percent | > 90% | 15 min | IO bottleneck |
| log_write_percent | > 90% | 15 min | Log write bottleneck |
| deadlock | > 0 | 5 min | Deadlocks occurring |
| sessions_percent | > 80% | 15 min | Connection saturation |
| workers_percent | > 80% | 15 min | Worker thread exhaustion |
| physical_data_read_percent | > 90% | 15 min | Cache miss rate high |
| avg_data_io_latency_ms | > 10 ms | 15 min | IO latency |
| avg_log_write_latency_ms | > 10 ms | 15 min | Log write latency |

### 4.3 Integration with Azure Monitor

Configure diagnostic settings to stream Intelligent Insights data to Log Analytics, Event Hub, or Azure Storage for long-term retention and cross-resource querying.

### 4.4 Action Groups for Notification

Configure action groups to trigger on Intelligent Insights alerts:
- Email/SMS/Push — Notify the DBA team
- Webhook — Trigger a Logic App or automation
- ITSMC — Integrate with ServiceNow or other ITSM tools
- Runbook — Execute an automated response

## 5. Query Patterns — Resource Analysis

### 5.1 DTU Consumption Trend Analysis

```sql
-- DTU consumption over the last 2 hours
SELECT
    end_time,
    avg_cpu_percent,
    avg_data_io_percent,
    avg_log_write_percent,
    dtu_consumption_percent,
    CASE
        WHEN dtu_consumption_percent >= 100 THEN 'THROTTLED'
        WHEN dtu_consumption_percent >= 80 THEN 'HIGH'
        WHEN dtu_consumption_percent >= 50 THEN 'MODERATE'
        ELSE 'LOW'
    END AS DtuStatus
FROM sys.dm_db_resource_stats
ORDER BY end_time DESC;
```

### 5.2 Correlating Slow Queries with Resource Spikes

```sql
-- Combine resource stats with Query Store
WITH ResourceHigh AS (
    SELECT
        end_time,
        avg_cpu_percent,
        dtu_consumption_percent
    FROM sys.dm_db_resource_stats
    WHERE avg_cpu_percent > 80
        OR dtu_consumption_percent > 80
),
RecentQueries AS (
    SELECT
        qsrsi.start_time,
        qsrsi.end_time,
        qsq.query_id,
        qt.query_sql_text,
        qsrs.avg_cpu_time,
        qsrs.avg_duration,
        qsrs.count_executions,
        qsrs.avg_logical_io_reads
    FROM sys.query_store_runtime_stats qsrs
    JOIN sys.query_store_runtime_stats_interval qsrsi
        ON qsrs.runtime_stats_interval_id = qsrsi.runtime_stats_interval_id
    JOIN sys.query_store_plan qsqp ON qsrs.plan_id = qsqp.plan_id
    JOIN sys.query_store_query qsq ON qsqp.query_id = qsq.query_id
    JOIN sys.query_store_query_text qt ON qt.query_text_id = qsq.query_text_id
    WHERE qsrs.execution_type_desc = 'Regular'
)
SELECT
    rq.query_id,
    LEFT(rq.query_sql_text, 200) AS SqlText,
    rq.avg_cpu_time,
    rq.avg_duration,
    rq.count_executions,
    rq.avg_logical_io_reads,
    rh.avg_cpu_percent,
    rh.dtu_consumption_percent,
    rh.end_time AS ResourcePeakTime
FROM RecentQueries rq
JOIN ResourceHigh rh ON
    rq.start_time <= rh.end_time
    AND rq.end_time >= DATEADD(MINUTE, -1, rh.end_time)
ORDER BY rh.end_time DESC, rq.avg_cpu_time DESC;
```

### 5.3 Waits Analysis in Azure SQL

```sql
-- Wait statistics in Azure SQL (database-scoped)
SELECT TOP 20
    wait_type,
    waiting_tasks_count,
    wait_time_ms,
    max_wait_time_ms,
    signal_wait_time_ms,
    wait_time_ms - signal_wait_time_ms AS resource_wait_time_ms,
    CASE
        WHEN wait_time_ms = 0 THEN 0
        ELSE (signal_wait_time_ms * 100.0 / wait_time_ms)
    END AS signal_wait_pct
FROM sys.dm_db_wait_stats
WHERE wait_type NOT IN (
    'BROKER_EVENTHANDLER', 'BROKER_RECEIVE_WAITFOR',
    'BROKER_TASK_STOP', 'BROKER_TO_FLUSH',
    'CHECKPOINT_QUEUE', 'DBMIRROR_DBM_EVENT',
    'DBMIRROR_EVENTS_QUEUE', 'DBMIRROR_WORKER_QUEUE',
    'HADR_CLUSAPI_CALL', 'HADR_FILESTREAM_IOMGR_IOCOMPLETION',
    'HADR_LOGCAPTURE_WAIT', 'HADR_NOTIFICATION_DEQUEUE',
    'HADR_TIMER_TASK', 'HADR_WORK_QUEUE',
    'LAZYWRITER_SLEEP', 'LOGMGR_QUEUE',
    'ONDEMAND_TASK_QUEUE',
    'QDS_CLEANUP_STALE_QUERIES_TASK_MAIN_LOOP_SLEEP',
    'REQUEST_FOR_DEADLOCK_SEARCH', 'SLEEP_BPOOL_FLUSH',
    'SLEEP_DBSTARTUP', 'SLEEP_DCOMSTARTUP',
    'SLEEP_MASTERDBREADY', 'SLEEP_MASTERMDREADY',
    'SLEEP_MASTERUPGRADED', 'SLEEP_MSDBSTARTUP',
    'SLEEP_SYSTEMTASK', 'SLEEP_TASK', 'SLEEP_TEMPDBSTARTUP',
    'SQLTRACE_BUFFER_FLUSH', 'SQLTRACE_INCREMENTAL_FLUSH_SLEEP',
    'SQLTRACE_WAIT_ENTRIES', 'WAIT_FOR_RESULTS',
    'WAITFOR', 'XE_DISPATCHER_JOIN', 'XE_DISPATCHER_WAIT',
    'XE_TIMER_EVENT'
)
ORDER BY wait_time_ms DESC;
```

## 6. Production Deployment — Integration with Azure Monitor

### 6.1 Diagnostic Settings

Configure diagnostic settings to stream Intelligent Insights to Log Analytics, Event Hub, or Azure Storage:

```powershell
# PowerShell: Configure diagnostic settings
$settings = @{
    workspaceId = "/subscriptions/.../workspaces/MyWorkspace"
    logs = @(
        @{ category = "SQLInsights"; enabled = $true },
        @{ category = "QueryStoreRuntimeStatistics"; enabled = $true },
        @{ category = "QueryStoreWaitStatistics"; enabled = $true },
        @{ category = "Errors"; enabled = $true },
        @{ category = "DatabaseWaitStatistics"; enabled = $true },
        @{ category = "Timeouts"; enabled = $true },
        @{ category = "Blocks"; enabled = $true },
        @{ category = "Deadlocks"; enabled = $true }
    )
    metrics = @(
        @{ category = "Basic"; enabled = $true }
    )
}

Set-AzDiagnosticSetting `
    -ResourceId "/subscriptions/.../databases/mydb" `
    -Name "SQLInsightsDiagnostics" `
    -Setting $settings
```

### 6.4 Automated Response Runbooks

```powershell
# Azure Automation Runbook: Respond to High DTU
param(
    [string]$ResourceId,
    [string]$AlertName
)

$connection = Get-AutomationConnection -Name AzureRunAsConnection
Connect-AzAccount -ServicePrincipal `
    -Tenant $connection.TenantID `
    -ApplicationId $connection.ApplicationID `
    -CertificateThumbprint $connection.CertificateThumbprint

$db = Get-AzSqlDatabase -DatabaseId $ResourceId

if ($db.Edition -eq "GeneralPurpose" -or $db.Edition -eq "BusinessCritical") {
    $currentCores = $db.Capacity
    $newCores = $currentCores * 2
    Set-AzSqlDatabase `
        -ResourceGroupName $db.ResourceGroupName `
        -ServerName $db.ServerName `
        -DatabaseName $db.DatabaseName `
        -RequestedServiceObjectiveName "${newCores}_${db.Edition}"
    Write-Output "Scaled $($db.DatabaseName) from $currentCores to $newCores vCores"
} else {
    Write-Output "Manual intervention required for DTU-based database"
}
```

## 7. Monitoring and Maintenance — Managing Insights

### 7.1 Monitoring Insight Health

```sql
-- Check when insights were last generated
SELECT
    MAX(end_time) AS LastInsightTime,
    COUNT(*) AS TotalInsights,
    COUNT(CASE WHEN severity = 'High' THEN 1 END) AS HighSeverityCount,
    COUNT(CASE WHEN severity = 'Medium' THEN 1 END) AS MediumSeverityCount,
    MIN(start_time) AS EarliestInsight
FROM sys.dm_intelligent_insights;
```

### 7.2 Insight Retention

Intelligent Insights data is retained for 30 days by default. Data is automatically purged by the Azure platform with no user configuration required.

### 7.3 Dismissing and Archiving Insights

In the Azure portal, insights can be dismissed, archived, or marked as resolved. These actions are portal operations, but you can filter insights by status in T-SQL:

```sql
-- Filter insights by status
SELECT
    insight_id,
    insight_type,
    start_time,
    end_time,
    severity,
    status,
    root_cause,
    recommendation
FROM sys.dm_intelligent_insights
WHERE status = 'Active'
ORDER BY severity DESC, start_time DESC;
```

### 7.4 Proactive Monitoring with Alerts

Set up the following alert rules for proactive monitoring:

Critical Alerts (Respond immediately):
- DTU consumption > 95% for 15 minutes
- CPU > 95% for 15 minutes
- Deadlocks > 5 per minute
- Log write throttling active

Warning Alerts (Investigate during business hours):
- DTU consumption > 80% for 30 minutes
- IO latency > 10ms average over 15 minutes
- Query Store size > 80% of max
- Connection count > 80% of limit

Informational Alerts (Review weekly):
- Query regression detected
- Index maintenance recommended
- Storage approaching limit

### 7.5 Cross-Database Monitoring

For environments with multiple Azure SQL databases, use Azure CLI or PowerShell to aggregate insights across all databases:

```powershell
# Get resource metrics for all databases in a server
$server = Get-AzSqlServer -ResourceGroupName "MyRG" -ServerName "myserver"
$results = @()

foreach ($db in Get-AzSqlDatabase -Server $server) {
    $metrics = Get-AzMetric `
        -ResourceId $db.ResourceId `
        -MetricName "dtu_consumption_percent" `
        -TimeGrain "00:05:00" `
        -StartTime (Get-Date).AddHours(-24)

    $results += [PSCustomObject]@{
        DatabaseName = $db.DatabaseName
        AvgDTU = ($metrics.Data | Measure-Object Average -Average).Average
        MaxDTU = ($metrics.Data | Measure-Object Maximum -Maximum).Maximum
    }
}

$results | Sort-Object AvgDTU -Descending | Format-Table -AutoSize
```

## 8. Troubleshooting — Common Issues

### 8.1 Intelligent Insights Not Available

If Intelligent Insights is not showing data:
- Verify the database is Azure SQL Database (not Managed Instance for basic Insights)
- Check the service tier (Basic tier may have limited insights)
- Ensure the database has been running for at least 24 hours for ML model training
- Check if the feature is enabled in the region

```sql
-- Verify we are in Azure SQL
SELECT
    SERVERPROPERTY('EngineEdition') AS EngineEdition,
    CASE SERVERPROPERTY('EngineEdition')
        WHEN 5 THEN 'Azure SQL Database'
        WHEN 8 THEN 'Azure SQL Managed Instance'
        ELSE 'SQL Server On-Premises'
    END AS Environment,
    DB_NAME() AS CurrentDatabase;
```

### 8.2 High Latency for Insights

Intelligent Insights has inherent latency of 15-60 minutes by design. The ML models need sufficient data to make accurate assessments. For real-time monitoring use:
- sys.dm_db_resource_stats for 15-second resource data
- sys.dm_exec_query_stats for current query performance
- Query Store for recent query metrics

### 8.3 DTU Model vs vCore Model Differences

Understanding the metrics available in each model:

| Metric | DTU | vCore |
|--------|-----|-------|
| dtu_consumption_percent | Yes | No |
| avg_cpu_percent | Yes | Yes |
| avg_data_io_percent | Yes | Yes |
| avg_log_write_percent | Yes | Yes |
| max_worker_percent | Yes | Yes |
| max_session_percent | Yes | Yes |
| avg_instance_cpu_percent | No | Yes |
| avg_instance_memory_percent | No | Yes |
| avg_login_rate_percent | No | Yes |
| avg_app_cpu_percent | No | Yes |

### 8.4 Insight Not Catching Application-Level Patterns

Intelligent Insights analyzes database-level metrics. It cannot correlate performance issues with application code, user behavior, or external dependencies to capture application-level patterns, use Application Insights SDK in the application to track SQL dependency calls with duration and correlate application errors with database performance.

### 8.5 Insight Suggests Scaling But Problem Persists

If Intelligent Insights recommends scaling up but the issue continues:
- The bottleneck may have shifted from CPU to IO
- Query-level issues may need index tuning or query rewriting
- Parameter sniffing may be causing inconsistent performance
- Check if the scaling actually took effect

### 8.6 False Positives in Insights

Intelligent Insights may produce false positives during:
- Known maintenance windows
- Deployment activities (new code releases)
- Unusual but legitimate workload spikes (end-of-month processing)
- Initial baseline training period (first 24-48 hours)

To handle false positives, mark the insight as dismissed, configure maintenance windows in Azure Advisor, or add custom alert suppression logic in Action Groups.

## 9. Summary — Key Takeaways

Azure SQL Intelligent Insights provides a unique AI-driven approach to database monitoring that eliminates the need for manual threshold configuration and baseline capture. It is best used as a complement to traditional monitoring practices.

| Component | Best Practice |
|-----------|---------------|
| Primary Use | Post-incident analysis, trend detection |
| Latency | 15-60 minutes (not real-time) |
| Configuration | Zero configuration required |
| Alerting | Use Azure Monitor alerts for real-time metrics |
| Integration | Stream to Log Analytics for long-term analysis |
| Automation | Use runbooks for auto-scaling and response |
| Complementary | Use with Query Store and DMVs for complete view |
| Limitations | Not available on all tiers, database-level only |

The strengths of Intelligent Insights are:
- Zero setup — Enable and get insights immediately
- AI-driven — Learns your workload baseline automatically
- Root cause — Not just what but why
- Actionable — Clear recommendations with T-SQL snippets

The weaknesses to be aware of:
- Latency — Not suitable for real-time alerting
- Scope — Database-level only, no application context
- Availability — Not on Managed Instance or SQL Server on VMs
- Control — Limited ability to tune the ML models

The complete Azure SQL monitoring stack includes:
1. sys.dm_db_resource_stats — Real-time 15-second resource metrics
2. sys.dm_db_wait_stats — Wait statistics (database-scoped)
3. Query Store — Per-query plan and runtime tracking
4. Intelligent Insights — AI-powered root cause analysis
5. Azure Monitor and Log Analytics — Long-term retention and cross-resource queries
6. Application Insights — End-to-end application dependency tracking

Intelligent Insights fits into the observability pyramid at the top, providing the why by correlating signals from the lower layers. When an alert fires from real-time metrics, Intelligent Insights provides context and root cause. When no alert fires but performance seems off, checking Intelligent Insights may reveal gradual regression that thresholds did not catch.

The most effective monitoring strategy combines all layers: real-time alerts for immediate response, Query Store for query-level plan tracking, DMV baselines for trend analysis, Extended Events for deep-dive investigation, and Intelligent Insights for automated root cause analysis. Together these tools give complete observability over Azure SQL Database performance.
