# 8.924 Baseline Capture — DMV Snapshot Strategy

## 1. Overview — Fundamentals of DMV Baseline Capture

Dynamic Management Views (DMVs) are the foundation of SQL Server performance troubleshooting. They expose internal state of the database engine — wait statistics, query execution metrics, index usage patterns, and operating system counters. However, DMVs have a critical architectural property: they are cumulative since the last SQL Server restart. This means a single snapshot in time tells you what has happened since the server came online, but it cannot tell you what changed in the last hour, the last day, or between two deployment windows.

Baseline capture solves this problem by taking periodic snapshots of key DMVs and storing them in a dedicated baseline database. By comparing consecutive snapshots, you can compute deltas — the change in a metric over a specific interval. This transforms raw cumulative counters into actionable rate-of-change data.

The core strategy involves three layers:
- **Capture Layer** — A scheduled job runs every 60 minutes, executing a stored procedure that reads from the target DMVs and inserts the results into baseline fact tables.
- **Storage Layer** — A baseline database with a schema designed for efficient insert and range-scan queries. Data is partitioned or indexed by SnapshotTime for fast retrieval.
- **Analysis Layer** — Delta queries that join consecutive snapshots, compute differences, and surface anomalies — unexpected spikes in wait times, regressed query durations, or index usage changes.

This approach is production-proven and scales from a single instance to hundreds of servers when combined with a centralized monitoring repository. The storage overhead is modest — roughly 100–200 MB per month per instance for a 60-minute interval, depending on the number of queries in the plan cache.

The key insight is that DMVs are free — they carry no licensing cost and minimal overhead. The snapshot queries themselves are lightweight reads against system views. The only cost is storage and the SQL Agent job runtime, which is typically sub-second.

Baseline data enables:
- **Trend analysis** — Is wait time growing week over week?
- **Anomaly detection** — Did a specific query regress after a code deployment?
- **Capacity planning** — Is logical IO growing faster than physical IO?
- **RCA (Root Cause Analysis)** — What changed in the 60 minutes before the incident?

This note covers the complete end-to-end implementation: schema design, capture scripts, delta queries, SQL Agent job configuration, data retention, and dashboard integration.

## 2. Architecture — Snapshot Collection Pipeline

The baseline capture pipeline is designed as a series of discrete stages that can be monitored, restarted, and tuned independently.

```
flowchart TD
    A[SQL Server Online] --> B[SQL Agent Job: BaselineCapture]
    B --> C{Job Step 1}
    C --> D[Capture sys.dm_os_wait_stats]
    C --> E[Capture sys.dm_exec_query_stats]
    C --> F[Capture sys.dm_os_performance_counters]
    C --> G[Capture sys.dm_db_index_usage_stats]
    D --> H[Insert into BaselineDB.dbo.WaitStatsSnapshot]
    E --> I[Insert into BaselineDB.dbo.QueryStatsSnapshot]
    F --> J[Insert into BaselineDB.dbo.PerfCountersSnapshot]
    G --> K[Insert into BaselineDB.dbo.IndexUsageSnapshot]
    H --> L[Commit]
    I --> L
    J --> L
    K --> L
    L --> M[Job Step 2: Purge > 90 days]
    M --> N[Job Step 3: Log completion]
    N --> O[Reporting Layer]
    O --> P[Delta Analysis Reports]
    O --> Q[Trend Dashboards]
    O --> R[Alert on Anomaly]
```

### 2.1 Capture Frequency Considerations

Choosing the snapshot interval requires balancing granularity against storage and performance:

| Interval | Storage per Month | Use Case |
|----------|-------------------|----------|
| 15 min | ~800 MB | High-frequency analysis, short-lived queries |
| 30 min | ~400 MB | General purpose, most deployments |
| 60 min | ~200 MB | Standard, low overhead |
| 4 hours | ~50 MB | Trend-only, minimal storage |

A 60-minute interval is the recommended starting point. It provides sufficient resolution to correlate performance changes with deployments, maintenance windows, and incident timestamps.

### 2.2 DMV Selection Rationale

Not all DMVs need baselining. The four selected DMVs provide the highest diagnostic value:

- **sys.dm_os_wait_stats** — Cumulative wait times by wait type. Deltas show what the server is waiting on right now.
- **sys.dm_exec_query_stats** — Per-query execution metrics (CPU, duration, reads, writes). Deltas identify regressed queries.
- **sys.dm_os_performance_counters** — SQL Server performance counters (buffer cache hit ratio, page life expectancy, checkpoint pages/sec). Deltas reveal system-level trends.
- **sys.dm_db_index_usage_stats** — Index seek/scan/lookup/update counts per index. Deltas show which indexes are actually being used or neglected.

### 2.3 Baseline Database Design

The baseline database should be on a separate disk from user databases to avoid IO contention. Use SIMPLE recovery model — baseline data is disposable and does not require point-in-time recovery.

A single baseline database can serve multiple monitored instances by adding a ServerName column to each snapshot table. This enables cross-server reporting.

### 3.2 Capture Procedure

The capture procedure reads from the DMVs and inserts into the baseline tables:

```sql
CREATE OR ALTER PROCEDURE dbo.usp_CaptureBaseline
    @ServerName NVARCHAR(128) = NULL
AS
BEGIN
    SET NOCOUNT ON;
    
    IF @ServerName IS NULL
        SET @ServerName = CAST(SERVERPROPERTY('MachineName') AS NVARCHAR(128)) 
                        + N'\' + CAST(SERVERPROPERTY('InstanceName') AS NVARCHAR(128));
    
    DECLARE @SnapshotTime DATETIME2 = SYSUTCDATETIME();
    
    -- Capture Wait Stats
    INSERT INTO dbo.WaitStatsSnapshot (
        SnapshotTime, ServerName, WaitType, 
        WaitingTasksCount, WaitTimeMs, SignalWaitTimeMs, MaxWaitTimeMs
    )
    SELECT 
        @SnapshotTime,
        @ServerName,
        wait_type,
        waiting_tasks_count,
        wait_time_ms,
        signal_wait_time_ms,
        max_wait_time_ms
    FROM sys.dm_os_wait_stats
    WHERE wait_type NOT IN ('BROKER_EVENTHANDLER', 'BROKER_RECEIVE_WAITFOR', 
                           'BROKER_TASK_STOP', 'BROKER_TO_FLUSH', 'BROKER_TRANSMITTER',
                           'CHECKPOINT_QUEUE', 'DBMIRROR_DBM_EVENT', 'DBMIRROR_EVENTS_QUEUE',
                           'DBMIRROR_WORKER_QUEUE', 'DBMIRRORING_CMD', 'DISPATCHER_QUEUE_SEMAPHORE',
                           'FT_IFTS_SCHEDULER_IDLE_WAIT', 'FT_IFTSHC_MUTEX', 'HADR_CLUSAPI_CALL',
                           'HADR_FILESTREAM_IOMGR_IOCOMPLETION', 'HADR_LOGCAPTURE_WAIT',
                           'HADR_NOTIFICATION_DEQUEUE', 'HADR_TIMER_TASK', 'HADR_WORK_QUEUE',
                           'LAZYWRITER_SLEEP', 'LOGMGR_QUEUE', 'ONDEMAND_TASK_QUEUE',
                           'PREEMPTIVE_OS_LIBRARYOPS', 'PREEMPTIVE_OS_COMOPS', 
                           'PREEMPTIVE_OS_CRYPTOPS', 'PREEMPTIVE_OS_PIPEOPS',
                           'PREEMPTIVE_OS_VERIFYTRUST', 'QDS_CLEANUP_STALE_QUERIES_TASK_MAIN_LOOP_SLEEP',
                           'QDS_PERSIST_TASK_MAIN_LOOP_SLEEP', 'REQUEST_FOR_DEADLOCK_SEARCH',
                           'SLEEP_BPOOL_FLUSH', 'SLEEP_DBSTARTUP', 'SLEEP_DCOMSTARTUP',
                           'SLEEP_MASTERDBREADY', 'SLEEP_MASTERMDREADY', 'SLEEP_MASTERUPGRADED',
                           'SLEEP_MSDBSTARTUP', 'SLEEP_SYSTEMTASK', 'SLEEP_TASK',
                           'SLEEP_TEMPDBSTARTUP', 'SNI_HTTP_ACCEPT', 'SP_SERVER_DIAGNOSTICS_SLEEP',
                           'SQLTRACE_BUFFER_FLUSH', 'SQLTRACE_INCREMENTAL_FLUSH_SLEEP',
                           'SQLTRACE_WAIT_ENTRIES', 'WAIT_FOR_RESULTS', 'WAITFOR',
                           'XE_DISPATCHER_JOIN', 'XE_DISPATCHER_WAIT', 'XE_TIMER_EVENT');
    
    -- Capture Query Stats
    INSERT INTO dbo.QueryStatsSnapshot (
        SnapshotTime, ServerName, DatabaseID, ObjectID, 
        QueryHash, PlanHandle, SqlHandle,
        ExecutionCount, TotalWorkerTime, TotalElapsedTime,
        TotalLogicalReads, TotalLogicalWrites, TotalPhysicalReads,
        TotalCLRTime, TotalGrantKB
    )
    SELECT 
        @SnapshotTime,
        @ServerName,
        qs.dbid,
        qs.objectid,
        qs.query_hash,
        qs.plan_handle,
        qs.sql_handle,
        qs.execution_count,
        qs.total_worker_time,
        qs.total_elapsed_time,
        qs.total_logical_reads,
        qs.total_logical_writes,
        qs.total_physical_reads,
        qs.total_clr_time,
        qs.total_grant_kb
    FROM sys.dm_exec_query_stats qs
    WHERE qs.total_elapsed_time > 0;
    
    -- Capture Performance Counters
    INSERT INTO dbo.PerfCountersSnapshot (
        SnapshotTime, ServerName,
        ObjectName, CounterName, InstanceName, CntrValue, CntrType
    )
    SELECT 
        @SnapshotTime,
        @ServerName,
        pc.object_name,
        pc.counter_name,
        pc.instance_name,
        pc.cntr_value,
        pc.cntr_type
    FROM sys.dm_os_performance_counters pc
    WHERE pc.cntr_value > 0;
    
    -- Capture Index Usage Stats
    INSERT INTO dbo.IndexUsageSnapshot (
        SnapshotTime, ServerName, DatabaseID, ObjectID, IndexID,
        UserSeeks, UserScans, UserLookups, UserUpdates,
        LastUserSeek, LastUserScan, LastUserLookup, LastUserUpdate
    )
    SELECT 
        @SnapshotTime,
        @ServerName,
        ius.database_id,
        ius.object_id,
        ius.index_id,
        ius.user_seeks,
        ius.user_scans,
        ius.user_lookups,
        ius.user_updates,
        ius.last_user_seek,
        ius.last_user_scan,
        ius.last_user_lookup,
        ius.last_user_update
    FROM sys.dm_db_index_usage_stats ius
    WHERE ius.database_id > 4;  -- Skip system databases
    
    RETURN 0;
END;
GO
```

### 3.3 Scheduling the Capture

Create a SQL Agent job that runs the capture procedure every hour:

```sql
-- Create SQL Agent job (requires SQLAgent running)
USE msdb;
GO

DECLARE @jobId BINARY(16);

EXEC dbo.sp_add_job
    @job_name = N'BaselineCapture - DMV Snapshot',
    @enabled = 1,
    @description = N'Captures DMV snapshots every 60 minutes for baseline analysis',
    @job_id = @jobId OUTPUT;

EXEC dbo.sp_add_jobstep
    @job_id = @jobId,
    @step_name = N'Capture Baseline Data',
    @command = N'EXEC BaselineDB.dbo.usp_CaptureBaseline;',
    @database_name = N'BaselineDB';

EXEC dbo.sp_add_jobstep
    @job_id = @jobId,
    @step_name = N'Purge Old Data',
    @command = N'EXEC BaselineDB.dbo.usp_PurgeBaseline @RetentionDays = 90;',
    @database_name = N'BaselineDB';

EXEC dbo.sp_add_jobschedule
    @job_id = @jobId,
    @name = N'Hourly',
    @freq_type = 4,
    @freq_interval = 1,
    @freq_subday_type = 8,
    @freq_subday_interval = 60;

EXEC dbo.sp_add_jobserver @job_id = @jobId, @server_name = N'(local)';
GO
```

## 4. Configuration — Baseline Table Schema Details

The baseline table schema must balance insert performance with query efficiency. The design choices above are deliberate:

### 4.1 Indexing Strategy

The snapshot tables use narrow, covering indexes optimized for the two primary access patterns:
- **Range scan by SnapshotTime** — Used by delta queries that join consecutive snapshots.
- **Lookup by key column** — WaitType for wait stats, QueryHash for query stats, CounterName for perf counters, ObjectID+IndexID for index usage.

The SnapshotID identity column provides a clustered key that minimizes page splitting during inserts.

### 4.2 Data Retention

A purge procedure removes data older than the retention window:

```sql
CREATE OR ALTER PROCEDURE dbo.usp_PurgeBaseline
    @RetentionDays INT = 90
AS
BEGIN
    SET NOCOUNT ON;
    
    DECLARE @CutoffDate DATETIME2 = DATEADD(DAY, -@RetentionDays, SYSUTCDATETIME());
    
    DELETE FROM dbo.WaitStatsSnapshot WHERE SnapshotTime < @CutoffDate;
    DELETE FROM dbo.QueryStatsSnapshot WHERE SnapshotTime < @CutoffDate;
    DELETE FROM dbo.PerfCountersSnapshot WHERE SnapshotTime < @CutoffDate;
    DELETE FROM dbo.IndexUsageSnapshot WHERE SnapshotTime < @CutoffDate;
    
    -- Rebuild indexes to maintain performance
    ALTER INDEX ALL ON dbo.WaitStatsSnapshot REORGANIZE;
    ALTER INDEX ALL ON dbo.QueryStatsSnapshot REORGANIZE;
    ALTER INDEX ALL ON dbo.PerfCountersSnapshot REORGANIZE;
    ALTER INDEX ALL ON dbo.IndexUsageSnapshot REORGANIZE;
END;
GO
```

### 4.3 Partitioning for Large Deployments

For environments with hundreds of servers or sub-15-minute intervals, consider partitioning by date:

```sql
-- Create partition function and scheme
CREATE PARTITION FUNCTION PF_BaselineByDate (DATETIME2)
AS RANGE RIGHT FOR VALUES (
    '2026-01-01', '2026-02-01', '2026-03-01', '2026-04-01',
    '2026-05-01', '2026-06-01', '2026-07-01', '2026-08-01',
    '2026-09-01', '2026-10-01', '2026-11-01', '2026-12-01'
);

CREATE PARTITION SCHEME PS_BaselineByDate
AS PARTITION PF_BaselineByDate ALL TO ([PRIMARY]);

-- Use partition scheme in table creation
CREATE TABLE dbo.WaitStatsSnapshotPartitioned (
    SnapshotID INT IDENTITY(1,1),
    SnapshotTime DATETIME2 NOT NULL,
    ServerName NVARCHAR(128) NOT NULL,
    WaitType NVARCHAR(60) NOT NULL,
    WaitingTasksCount BIGINT NOT NULL,
    WaitTimeMs BIGINT NOT NULL,
    SignalWaitTimeMs BIGINT NOT NULL,
    MaxWaitTimeMs BIGINT NOT NULL
) ON PS_BaselineByDate(SnapshotTime);
```

## 5. Query Patterns — Delta Analysis

The heart of baseline analysis is computing deltas — the difference between two consecutive snapshots.

### 5.1 Wait Stats Delta

Compare the two most recent snapshots to find wait types with the largest increase:

```sql
WITH SnapshotRank AS (
    SELECT 
        SnapshotID,
        SnapshotTime,
        ServerName,
        WaitType,
        WaitingTasksCount,
        WaitTimeMs,
        SignalWaitTimeMs,
        ROW_NUMBER() OVER (PARTITION BY WaitType, ServerName ORDER BY SnapshotTime DESC) AS rn
    FROM dbo.WaitStatsSnapshot
    WHERE ServerName = @ServerName
),
CurrentWait AS (
    SELECT * FROM SnapshotRank WHERE rn = 1
),
PreviousWait AS (
    SELECT * FROM SnapshotRank WHERE rn = 2
)
SELECT 
    cw.WaitType,
    pw.WaitTimeMs AS PreviousWaitTimeMs,
    cw.WaitTimeMs AS CurrentWaitTimeMs,
    (cw.WaitTimeMs - pw.WaitTimeMs) AS DeltaWaitTimeMs,
    CASE 
        WHEN pw.WaitTimeMs = 0 THEN 100.0
        ELSE ((cw.WaitTimeMs - pw.WaitTimeMs) * 100.0) / pw.WaitTimeMs
    END AS PctChange,
    cw.WaitingTasksCount - pw.WaitingTasksCount AS DeltaWaitingTasks,
    cw.SignalWaitTimeMs - pw.SignalWaitTimeMs AS DeltaSignalWaitMs
FROM CurrentWait cw
JOIN PreviousWait pw ON cw.WaitType = pw.WaitType AND cw.ServerName = pw.ServerName
WHERE (cw.WaitTimeMs - pw.WaitTimeMs) > 0
    AND cw.WaitTimeMs > 10000  -- Filter noise
ORDER BY DeltaWaitTimeMs DESC;
```

### 5.2 Query Stats Delta — Regression Detection

Identify queries whose average duration increased significantly between snapshots:

```sql
WITH SnapshotRank AS (
    SELECT 
        SnapshotID,
        SnapshotTime,
        ServerName,
        QueryHash,
        PlanHandle,
        ExecutionCount,
        (TotalElapsedTime / NULLIF(ExecutionCount, 0)) AS AvgDuration,
        (TotalWorkerTime / NULLIF(ExecutionCount, 0)) AS AvgCPU,
        (TotalLogicalReads / NULLIF(ExecutionCount, 0)) AS AvgReads,
        ROW_NUMBER() OVER (PARTITION BY QueryHash, PlanHandle, ServerName ORDER BY SnapshotTime DESC) AS rn
    FROM dbo.QueryStatsSnapshot
    WHERE ServerName = @ServerName
),
CurrentStat AS (
    SELECT * FROM SnapshotRank WHERE rn = 1 AND ExecutionCount > 100
),
PreviousStat AS (
    SELECT * FROM SnapshotRank WHERE rn = 2
)
SELECT 
    cs.QueryHash,
    cs.AvgDuration AS CurrentAvgDuration,
    ps.AvgDuration AS PreviousAvgDuration,
    (cs.AvgDuration - ps.AvgDuration) AS DeltaAvgDuration,
    CASE 
        WHEN ps.AvgDuration = 0 THEN 100.0
        ELSE ((cs.AvgDuration - ps.AvgDuration) * 100.0) / ps.AvgDuration
    END AS PctChange,
    cs.AvgCPU,
    cs.AvgReads,
    cs.ExecutionCount,
    cs.PlanHandle
FROM CurrentStat cs
JOIN PreviousStat ps ON cs.QueryHash = ps.QueryHash AND cs.ServerName = ps.ServerName
WHERE (cs.AvgDuration - ps.AvgDuration) > 50000  -- > 50ms increase
    AND cs.AvgDuration > ps.AvgDuration
    AND ps.ExecutionCount > 10  -- Ensure previous had enough samples
ORDER BY DeltaAvgDuration DESC;
```

### 5.3 Performance Counter Delta

Compute rate-of-change for key performance counters:

```sql
WITH SnapshotRank AS (
    SELECT 
        SnapshotTime,
        ServerName,
        CounterName,
        InstanceName,
        CntrValue,
        ROW_NUMBER() OVER (PARTITION BY CounterName, InstanceName, ServerName ORDER BY SnapshotTime DESC) AS rn
    FROM dbo.PerfCountersSnapshot
    WHERE ObjectName LIKE '%Buffer Manager%'
        OR ObjectName LIKE '%SQL Statistics%'
),
CurrentVal AS (
    SELECT * FROM SnapshotRank WHERE rn = 1
),
PreviousVal AS (
    SELECT * FROM SnapshotRank WHERE rn = 2
)
SELECT 
    cv.CounterName,
    cv.InstanceName,
    cv.CntrValue AS CurrentValue,
    pv.CntrValue AS PreviousValue,
    (cv.CntrValue - pv.CntrValue) AS Delta
FROM CurrentVal cv
JOIN PreviousVal pv ON cv.CounterName = pv.CounterName 
                    AND ISNULL(cv.InstanceName, '') = ISNULL(pv.InstanceName, '')
                    AND cv.ServerName = pv.ServerName
ORDER BY Delta DESC;
```

### 5.4 Index Usage Delta

Identify indexes that are not being used or suddenly seeing high activity:

```sql
WITH SnapshotRank AS (
    SELECT 
        SnapshotTime,
        ServerName,
        DatabaseID,
        ObjectID,
        IndexID,
        UserSeeks,
        UserScans,
        UserLookups,
        UserUpdates,
        ROW_NUMBER() OVER (PARTITION BY DatabaseID, ObjectID, IndexID, ServerName ORDER BY SnapshotTime DESC) AS rn
    FROM dbo.IndexUsageSnapshot
),
CurrentUsage AS (
    SELECT * FROM SnapshotRank WHERE rn = 1
),
PreviousUsage AS (
    SELECT * FROM SnapshotRank WHERE rn = 2
)
SELECT 
    OBJECT_SCHEMA_NAME(cu.ObjectID, cu.DatabaseID) AS SchemaName,
    OBJECT_NAME(cu.ObjectID, cu.DatabaseID) AS TableName,
    i.name AS IndexName,
    (cu.UserSeeks - pu.UserSeeks) AS DeltaSeeks,
    (cu.UserScans - pu.UserScans) AS DeltaScans,
    (cu.UserLookups - pu.UserLookups) AS DeltaLookups,
    (cu.UserUpdates - pu.UserUpdates) AS DeltaUpdates
FROM CurrentUsage cu
JOIN PreviousUsage pu ON cu.DatabaseID = pu.DatabaseID 
                      AND cu.ObjectID = pu.ObjectID 
                      AND cu.IndexID = pu.IndexID
                      AND cu.ServerName = pu.ServerName
JOIN sys.indexes i ON i.object_id = cu.ObjectID AND i.index_id = cu.IndexID
WHERE (cu.UserSeeks + cu.UserScans + cu.UserLookups) = 0  -- Unused indexes
    AND cu.UserUpdates > 0                                  -- But being maintained
ORDER BY cu.UserUpdates DESC;
```

### 6.3 Concurrency Handling

Ensure only one capture runs at a time:

```sql
CREATE OR ALTER PROCEDURE dbo.usp_CaptureBaseline
AS
BEGIN
    SET NOCOUNT ON;
    
    -- Acquire application lock
    DECLARE @LockResult INT;
    EXEC @LockResult = sp_getapplock 
        @Resource = 'DMV_Capture',
        @LockMode = 'Exclusive',
        @LockTimeout = 1000;  -- 1 second timeout
    
    IF @LockResult < 0
    BEGIN
        RAISERROR('Could not acquire lock. Another capture may be running.', 16, 1);
        RETURN -1;
    END
    
    BEGIN TRY
        -- Capture logic here
        
        -- Release lock
        EXEC sp_releaseapplock @Resource = 'DMV_Capture';
    END TRY
    BEGIN CATCH
        EXEC sp_releaseapplock @Resource = 'DMV_Capture';
        THROW;
    END CATCH
END;
GO
```

## 7. Monitoring & Maintenance — Keeping Baselines Healthy

### 7.1 Monitoring the Capture Job

Key metrics to track:
- **Job duration** — Should be < 5 seconds. If longer, investigate blocking or DMV contention.
- **Row counts** — Sudden drop may indicate plan cache flush or server restart.
- **Error rate** — Capture failures should be zero.

Create a job log table to track execution history:

```sql
CREATE TABLE dbo.JobLog (
    LogID INT IDENTITY(1,1) PRIMARY KEY,
    JobName NVARCHAR(128) NOT NULL,
    RunTime DATETIME2 NOT NULL DEFAULT SYSUTCDATETIME(),
    Status NVARCHAR(20) NOT NULL,  -- Success, Failed
    RowsCaptured INT NULL,
    DurationMs INT NULL,
    ErrorMessage NVARCHAR(4000) NULL
);

CREATE OR ALTER PROCEDURE dbo.usp_CheckCaptureHealth
AS
BEGIN
    SELECT 
        ServerName,
        COUNT(DISTINCT CAST(SnapshotTime AS DATE)) AS DaysWithData,
        MIN(SnapshotTime) AS EarliestSnapshot,
        MAX(SnapshotTime) AS LatestSnapshot,
        DATEDIFF(HOUR, MIN(SnapshotTime), MAX(SnapshotTime)) AS HoursOfData
    FROM dbo.WaitStatsSnapshot
    GROUP BY ServerName
    ORDER BY ServerName;
END;
```

### 7.2 Data Integrity Checks

Run periodic validation:

```sql
CREATE OR ALTER PROCEDURE dbo.usp_ValidateSnapshotIntegrity
AS
BEGIN
    -- Check for gaps in snapshot schedule
    WITH SnapshotGrid AS (
        SELECT 
            ServerName,
            SnapshotTime,
            LAG(SnapshotTime) OVER (PARTITION BY ServerName ORDER BY SnapshotTime) AS PrevSnapshot,
            DATEDIFF(MINUTE, LAG(SnapshotTime) OVER (PARTITION BY ServerName ORDER BY SnapshotTime), SnapshotTime) AS GapMinutes
        FROM dbo.WaitStatsSnapshot
    )
    SELECT 
        ServerName,
        SnapshotTime,
        GapMinutes,
        CASE 
            WHEN GapMinutes > 90 THEN 'WARNING: Gap > 90 minutes'
            WHEN GapMinutes IS NULL THEN 'First snapshot'
            ELSE 'OK'
        END AS Status
    FROM SnapshotGrid
    WHERE GapMinutes > 90 OR GapMinutes IS NULL
    ORDER BY SnapshotTime DESC;
END;
```

### 7.3 Storage Management

Monitor baseline database size:

```sql
SELECT 
    s.name AS SchemaName,
    t.name AS TableName,
    p.rows AS RowCounts,
    CAST(ROUND(a.total_pages * 8.0 / 1024, 2) AS DECIMAL(18,2)) AS TotalSizeMB,
    CAST(ROUND(a.used_pages * 8.0 / 1024, 2) AS DECIMAL(18,2)) AS UsedSizeMB,
    CAST(ROUND((a.total_pages - a.used_pages) * 8.0 / 1024, 2) AS DECIMAL(18,2)) AS FreeSpaceMB
FROM sys.tables t
INNER JOIN sys.schemas s ON s.schema_id = t.schema_id
INNER JOIN sys.indexes i ON t.object_id = i.object_id
INNER JOIN sys.partitions p ON i.object_id = p.object_id AND i.index_id = p.index_id
CROSS APPLY (
    SELECT SUM(a.total_pages) AS total_pages, SUM(a.used_pages) AS used_pages
    FROM sys.allocation_units a WHERE p.partition_id = a.container_id
) a
WHERE t.name LIKE '%Snapshot%'
ORDER BY TotalSizeMB DESC;
```

## 8. Troubleshooting — Common Issues and Solutions

### 8.1 DMVs Reset on Restart

When SQL Server restarts, all cumulative DMVs reset to zero. The first post-restart snapshot becomes the new baseline.

Detection query:
```sql
-- Check if DMVs were recently reset
SELECT 
    sqlserver_start_time,
    DATEDIFF(HOUR, sqlserver_start_time, GETDATE()) AS HoursSinceRestart
FROM sys.dm_os_sys_info;
```

Mitigation: Schedule an immediate capture on SQL Server startup using a startup stored procedure:
```sql
EXEC sp_procoption @ProcName = 'dbo.usp_CaptureBaseline', @OptionName = 'startup', @OptionValue = 'true';
```

### 8.2 Query Stats Eviction

The plan cache evicts plans under memory pressure. This causes query stats to disappear between snapshots. The delta query handles this naturally — if a plan is missing from the current snapshot, it simply wasn't executed recently.

To detect eviction events:
```sql
-- Check plan cache count trend
SELECT 
    COUNT(*) AS PlanCount,
    SUM(size_in_bytes) / 1048576 AS CacheSizeMB
FROM sys.dm_exec_cached_plans;
```

### 8.3 Snapshot Frequency vs Storage Tradeoff

Higher frequency captures provide more granular data but increase storage and IO. The baseline capture typically runs in under 2 seconds for each DMV query, so the IO impact is minimal even at 15-minute intervals. However, storage grows linearly with frequency.

Storage projection formula:
```
Monthly Size = (SnapshotSize * 730 hours * SnapshotsPerHour) * NumberOfServers
```

Where SnapshotSize is approximately:
- 80 KB for wait stats
- 300 KB for query stats (varies by cache size)
- 200 KB for perf counters
- 40 KB for index usage

### 8.4 Blocking During Capture

The capture queries against DMVs use schema-stability locks (SCH-S) which are compatible with most operations. However, under extreme blocking scenarios, even DMV queries may experience delays.

Monitor with:
```sql
SELECT 
    blocking_session_id,
    wait_type,
    wait_time,
    wait_resource,
    session_id
FROM sys.dm_exec_requests
WHERE blocking_session_id > 0;
```

### 8.5 Time Zone Consistency

Always store SnapshotTime in UTC (using SYSUTCDATETIME) to avoid DST issues and enable cross-server comparisons. Convert to local time in reporting.

### 8.6 Server Name Changes

If a server is renamed, the baseline data under the old name will be orphaned. Handle with an update script:
```sql
UPDATE dbo.WaitStatsSnapshot SET ServerName = @NewName WHERE ServerName = @OldName;
```

## 9. Summary — Key Takeaways

Baseline capture via DMV snapshots is a low-cost, high-value monitoring strategy that every SQL Server deployment should implement. The cumulative nature of DMVs makes periodic snapshots essential for understanding performance trends over time.

| Component | Best Practice |
|-----------|---------------|
| Capture Interval | 60 minutes (default), 15 minutes for critical systems |
| Retention Period | 90 days rolling |
| Storage Format | SIMPLE recovery, separate database, UTC timestamps |
| Key DMVs | wait_stats, exec_query_stats, performance_counters, index_usage_stats |
| Delta Analysis | Compare consecutive snapshots for rate-of-change |
| Alerting | Spike in wait times, query regression > 50ms, storage > 80% |
| High Availability | Capture on every replica (each has independent DMVs) |
| Security | Schema-level permissions to DMVs for the agent account |

The complete implementation — capture procedure, delta queries, job setup, and dashboards — provides a robust foundation for SQL Server performance analysis. Combined with Query Store and Extended Events, it forms the observability stack that enables proactive database administration.

When deployed correctly, baseline data answers the most common performance question in any DBA's career: "What changed?"
