## Section 5 — Gotchas

### Gotcha 1 — New CE Does Not Fix All Under-Estimates
**Pitfall:** Assuming all cardinality problems vanish with CE150.
**Symptom:** Multi-column correlation across more than 4-5 predicates still under-estimates because exponential backoff is a heuristic, not a perfect model.
**Fix:** Create multi-column statistics: CREATE STATISTICS s_CatPriceStock ON Products(Category, Price, InStock) WITH FULLSCAN.
**Cost:** Ongoing maintenance; must identify correlated column groups first.

### Gotcha 2 — Exponential Backoff Can Over-Estimate
**Pitfall:** The New CE assumes some correlation between predicates. If columns are truly independent, the exponential backoff produces estimates that are too high.
**Symptom:** Memory grant is too large; query waits for memory grant; concurrent queries under memory pressure.
**Fix:** OPTION (USE HINT('ASSUME_MIN_SELECTIVITY_FOR_FILTER_ESTIMATES')) or revert to CE70 for the specific query.
**Cost:** Up to 10x memory over-allocation when predicates are independent.

### Gotcha 3 — TVF Correction Only in CE140+
**Pitfall:** CE120 still estimates multi-statement TVF output as 1 row (same as CE70).
**Symptom:** Queries joining TVFs get nested loop joins that iterate millions of times, assuming a 1-row outer input.
**Fix:** Upgrade to SQL 2016+ (compatibility level 130+) or restructuring queries to avoid TVFs.
**Cost:** 1000x performance difference if a TVF returning 10K rows is estimated as 1 row.

### Gotcha 4 — Feedback Loops Take Multiple Executions
**Pitfall:** CE150's join CE feedback does not help the first execution. It learns over at least 2-3 executions.
**Symptom:** Ad-hoc queries with RECOMPILE never benefit from feedback because the plan is never re-used.
**Fix:** For critical paths, use parameterized queries that allow plan reuse, or use Query Store to force plans once they converge.
**Cost:** The first few executions may use a bad plan.

### Gotcha 5 — ASCENDING Key Fix Does Not Work for All Data Types
**Pitfall:** The 200-row ASCENDING key sampling works for numeric and date columns but has limitations with VARCHAR or GUID ASCENDING keys.
**Symptom:** WHERE OrderGuid > 'some-new-guid' still estimates 0 rows even under New CE.
**Fix:** Partitioned table with partitioning aligned to the GUID range or use OPTION (OPTIMIZE FOR UNKNOWN).
**Cost:** Complete under-estimate leading to nested loop when merge join is needed.

### Gotcha 6 — Database Scoped Config CAN Override New CE
**Pitfall:** ALTER DATABASE SCOPED CONFIGURATION SET LEGACY_CARDINALITY_ESTIMATION = ON silently reverts the entire database to CE70 regardless of compatibility level.
**Symptom:** You set compatibility level 150 but your plans show CardinalityEstimationModelVersion = 70.
**Fix:** Check sys.database_scoped_configurations for LEGACY_CARDINALITY_ESTIMATION.
**Cost:** Months of sub-optimal plans without knowing why.

## Section 7 — Interview Arsenal

### Interview Questions

**Q1: Explain the base containment assumption vs simple containment. Why does it matter?**
*Spoken answer (tier 1):* Base containment uses actual row frequencies from both sides of a join instead of assuming all values in the smaller side exist in the larger side. This prevents over-estimating join cardinalities, especially in star schemas.

*Spoken answer (tier 2):* In a star join where a 10K dimension table joins a 10M fact table, CE70's simple containment assumes every dimension row matches at least one fact row. In reality, some dimension rows may have no facts yet (new products). Base containment uses the frequency vector to estimate how many dimension rows actually participate, producing a more accurate join cardinality. This changes the join type decision: simple containment might over-estimate and still choose hash join, but base containment could correctly identify that a nested loop is sufficient.

**Q2: How does exponential backoff work in detail?**
*Spoken answer:* The CE sorts predicates by selectivity descending: s1 >= s2 >= ... >= sn. Then: estimated selectivity = s1 * s2^(1/2) * s3^(1/4) * ... * sn^(1/2^(n-1)). If all selectivities are identical at 0.1, CE70 produces 0.1^n which approaches 0 quickly. Exponential backoff produces: 0.1 * 0.316 * 0.562 * 0.749 which converges to about 0.01 for 4 predicates - still aggressive but far more realistic.

**Q3: What was wrong with the ASCENDING key handling in CE70 and how did CE120+ fix it?**
*Spoken answer:* CE70 estimated 0 rows for WHERE OrderDate > '2026-06-28' if the histogram's maximum value was older. A 0-row estimate causes the optimizer to choose operators that fail when millions of rows appear. CE120+ detects that the key is ASCENDING and performs a partial stats update. It introduces up to 200 new rows into the statistics, giving a non-zero estimate.

**Q4: When might the New CE produce WORSE estimates than CE70?**
*Spoken answer:* When predicates are genuinely independent - exponential backoff assumes some correlation exists. For completely independent columns, the backoff formula deliberately over-estimates compared to pure multiplication. Also, the base containment assumption can under-estimate if there is no filtering on dimensions. The New CE is not strictly better in every scenario.

**Q5: What CE150 feedback mechanisms exist and how do they work?**
*Spoken answer:* Three feedback mechanisms: (1) Join CE feedback compares actual join cardinality to estimated and adjusts future compilations via a correction factor. (2) Memory grant feedback reduces memory over/under-grants by tracking actual memory usage. (3) UDF inlining feedback helps decide whether to inline scalar UDFs. All three are persisted across recompiles of the same query.

**Q6: How does the Number of Rows in a Multi-Statement TVF affect New CE behavior?**
*Spoken answer:* CE70 and CE120 estimate 1 row regardless of actual TVF output. CE140+ changed this to estimate the actual number of rows the TVF returns. This is critical because a 1-row estimate pushes the optimizer to choose nested loop joins, which fail catastrophically when the TVF returns 100K rows.

**Q7: What is the ASSUME_JOIN_PREDICATE_DEPENDENCE hint and when would you use it?**
*Spoken answer:* This hint forces SQL Server to use base containment even under CE70. It is useful when you are stuck on CE70 for other reasons but want better join estimates. Conversely, ASSUME_JOIN_PREDICATE_INDEPENDENCE forces simple containment under the New CE.

**Q8: How does plan cache invalidation interact with CE version changes?**
*Spoken answer:* Changing the CE version via compatibility level or LEGACY_CARDINALITY_ESTIMATION flushes the entire plan cache for that database. This causes a recompilation storm - every query must be re-optimized with the new CE. On large production systems, this can cause CPU spikes for 5-30 minutes. Use Query Store to preserve plan history.

## Section 8 — Decision Framework

`mermaid
flowchart TD
    A[Start: Query with plan regression] --> B{CE Version in Plan XML?}
    B -->|70| C[Check if new CE would help]
    B -->|120+| D[Check estimate accuracy]

    C --> E{Problem pattern?}
    E -->|Correlated predicates| F[Enable New CE via compatibility level 120+]
    E -->|ASCENDING key| G[Update stats FULLSCAN or keep CE70 with manual maintenance]
    E -->|Join estimate| H[New CE base containment likely helps]

    D --> I{Estimate vs Actual?}
    I -->|Close| J[CE is fine. Look elsewhere.]
    I -->|Under-estimate| K{Which operator?}
    I -->|Over-estimate| L{Which operator?}

    K -->|Filter| M[Add missing stats or multi-column stats]
    K -->|Join| N[Check containment assumption, try base containment hint]

    L -->|Filter| O[Check for independent predicates, try ASSUME_MIN_SELECTIVITY]
    L -->|Join| P[Check for simple containment over-estimate, try base containment hint]
`

### Decision Checklist

1. Identify the CE version: plan XML -> CardinalityEstimationModelVersion
2. Check estimate accuracy: SET STATISTICS PROFILE ON -> compare EstimateRows vs ActualRows
3. Classify the problem: predicate correlation? join containment? ASCENDING key? TVF?
4. Choose solution: force different CE for query, update stats, create multi-column stats, or add hint
5. Validate: compare logical reads and duration between CE versions
6. Productionize: use Query Store, plan guides, or hints to lock in the fix
7. Monitor: watch for regressions after next stats update or version upgrade

### Tradeoffs

| Decision | Upside | Downside | Best For |
|---|---|---|---|
| Force New CE for query | Better correlation handling | May over-estimate independent predicates | OLTP with many WHERE clauses |
| Force CE70 for query | Safe, predictable for simple predicates | Misses ASCENDING key fix | Simple lookups with known statistics |
| Multi-column statistics | Fixes correlation for both CEs | Storage cost, maintenance | Known correlated column groups |
| Feedback loop reliance | Self-tuning, no manual work | Only helps repeated queries | Stable OLTP workloads |
| Query Store forced plan | Guarantees plan stability | Masked CE issues; manual effort | Critical queries that must not regress |

### Scale Thresholds

- <10 GB: CE version barely matters. Most queries are simple enough.
- 10-100 GB: CE differences appear for complex reports. Query Store essential.
- 100 GB-1 TB: CE version is critical. Plan regressions from wrong CE can cause outages.
- >1 TB: CE feedback loops (CE150) are important. Join estimates dominate plan quality.

**Q1:** What is the formula for exponential backoff with 5 predicates of selectivities 0.5, 0.4, 0.3, 0.2, 0.1?
<details>
s1 = 0.5, s2 = 0.4^(1/2) = 0.6325, s3 = 0.3^(1/4) = 0.7401, s4 = 0.2^(1/8) = 0.8123, s5 = 0.1^(1/16) = 0.8660
Combined selectivity = 0.5 * 0.6325 * 0.7401 * 0.8123 * 0.8660 = 0.2055
CE70 would give: 0.5 * 0.4 * 0.3 * 0.2 * 0.1 = 0.0012
Ratio: New CE estimate is ~171x higher.
</details>

**Q2:** What is the difference between CE120 and CE140 for TVF joins?
<details>
CE120 estimates multi-statement TVF joins as 1 row (same as CE70). CE140+ uses the actual cardinality of the TVF, which can be thousands or millions of rows.
</details>

**Q3:** How does base containment improve join cardinality estimation?
<details>
Base containment uses frequency vectors (actual row counts per value) rather than assuming all values in the smaller input match values in the larger input. CE70's simple containment over-estimates joins by assuming perfect overlap.
</details>

**Q4:** What three feedback mechanisms does CE150 introduce?
<details>
(1) Join cardinality feedback, (2) Memory grant feedback, (3) UDF inlining feedback.
</details>

**Q5:** What is the ASSUME_MIN_SELECTIVITY_FOR_FILTER_ESTIMATES hint used for?
<details>
It forces the optimizer to use the minimum selectivity (most restrictive predicate) for the entire filter, ignoring other predicates. This counters the New CE's tendency to over-estimate when predicates are independent.
</details>

**Q6:** How do you detect if a query is using ASCENDING key correction?
<details>
Check if the estimated rows for an out-of-range value is exactly 200 (or slightly less) - that indicates the ASCENDING key sampling kicked in.
</details>

**Q7:** Can a single query use both CE70 and New CE features?
<details>
No. The CE version is a binary choice per query compilation - either CE70 or the New CE. However, some USE HINT options can modify the New CE behavior to mimic certain CE70 characteristics.
</details>

**Q8:** What database scoped configuration silently reverts to CE70?
<details>
ALTER DATABASE SCOPED CONFIGURATION SET LEGACY_CARDINALITY_ESTIMATION = ON.
</details>

**Q9:** How many executions does CE150 feedback typically need to converge?
<details>
Join CE feedback can apply a correction after a single execution if the estimate ratio exceeds ~10x. Memory grant feedback typically converges within 2-3 executions.
</details>

**Q10:** What is the risk of combining Query Store forced plans with CE feedback loops?
<details>
A forced plan bypasses re-optimization, so CE150's feedback loop never triggers. The plan stays frozen with the original estimate.
</details>

## Section 9 — Extended Case Studies

### Case Study 1: Join Containment in a Star Schema

**Scenario:** A data warehouse with 50M row fact table and 10K row dimension table. Query: SELECT * FROM Fact f JOIN Dim d ON f.DimId = d.DimId WHERE d.Category = 'Active'.

**CE70 (Simple Containment):** Estimated 50M * (3K/10K) = 15M rows. Actual: 12M rows. Close enough.

**CE120+ (Base Containment):** Estimated using frequency vectors. Noticed that only 8K of 10K DimId values have any matching Fact rows. Estimate: 50M * (8K/10K) * (3K/8K) = 12M rows. Almost exact.

**Impact:** CE120+ chose a more efficient plan because the estimate was accurate. CE70 over-estimated by 25%, still choosing Hash Join (same result, but memory grant was slightly oversized).

### Case Study 2: Multiple TVF Join in Reporting

**Scenario:** A report joins two multi-statement TVFs. Under CE70 and CE120, each TVF was estimated as 1 row. Actual: TVF1 returns 5K rows, TVF2 returns 10K rows.

**CE70/120:** 1 * 1 = 1 row estimate. Nested Loops chosen for the join. 5K * 10K = 50M loop iterations. Query: 45 minutes.

**CE140+:** 5K * 10K = 50M row estimate. Hash Join chosen. Query: 3 seconds.

**Resolution:** Upgrade compatibility level to 130+ (CE140+) for the reporting database. If upgrade not possible, rewrite TVFs as stored procedures populating temp tables.

### Case Study 3: Exponential Backoff Over-Correction

**Scenario:** A search page with 8 filter criteria, all independent. Query: 500ms under CE70, 8 seconds under CE150.

**Root Cause:** CE150's exponential backoff for 8 predicates: 0.5^(1/2) * 0.5^(1/4) * ... = converging to ~0.67 combined selectivity. So 1M * 0.67 = 670K estimated rows. CE70: 1M * 0.5^8 = 3.9K. Actual: 4.2K.

**Resolution:** CE70 was correct (independent columns). Applied OPTION (USE HINT('ASSUME_MIN_SELECTIVITY_FOR_FILTER_ESTIMATES')) to make CE150 use the minimum selectivity instead of exponential backoff.

### Exercise: Correlation Detection Script

Write a query that detects whether columns in a table are correlated, helping you predict which CE will estimate better.

<details>`sql
CREATE PROCEDURE dbo.CheckColumnCorrelation
    @SchemaName NVARCHAR(128),
    @TableName NVARCHAR(128),
    @Col1 NVARCHAR(128),
    @Col2 NVARCHAR(128)
AS
BEGIN
    DECLARE @sql NVARCHAR(MAX);
    DECLARE @count INT, @distinct1 INT, @distinct2 INT, @distinct12 INT;
    
    -- Get counts
    SET @sql = N'SELECT @c = COUNT(*), @d1 = COUNT(DISTINCT ' + QUOTENAME(@Col1) + '), @d2 = COUNT(DISTINCT ' + QUOTENAME(@Col2) + '),
        @d12 = COUNT(DISTINCT CAST(' + QUOTENAME(@Col1) + ' AS NVARCHAR(100)) + '':' + ' + CAST(' + QUOTENAME(@Col2) + ' AS NVARCHAR(100)))
        FROM ' + QUOTENAME(@SchemaName) + '.' + QUOTENAME(@TableName);
    
    EXEC sp_executesql @sql, N'@c INT OUTPUT, @d1 INT OUTPUT, @d2 INT OUTPUT, @d12 INT OUTPUT',
        @c = @count OUTPUT, @d1 = @distinct1 OUTPUT, @d2 = @distinct2 OUTPUT, @d12 = @distinct12 OUTPUT;
    
    -- Compute correlation indicator
    SELECT
        @count AS total_rows,
        @distinct1 AS distinct_col1,
        @distinct2 AS distinct_col2,
        @distinct12 AS distinct_pair,
        CASE 
            WHEN @distinct1 * @distinct2 = 0 THEN 0
            ELSE 1.0 * @distinct12 / (@distinct1 * @distinct2)
        END AS correlation_ratio,
        CASE
            WHEN 1.0 * @distinct12 / NULLIF(@distinct1 * @distinct2, 0) < 0.1 THEN 'STRONG CORRELATION (CE150 better)'
            WHEN 1.0 * @distinct12 / NULLIF(@distinct1 * @distinct2, 0) < 0.5 THEN 'MODERATE CORRELATION (CE150 slightly better)'
            ELSE 'WEAK OR NO CORRELATION (CE70 likely better)'
        END AS ce_recommendation;
END;

## Section 9 — Extended: Additional Case Studies

### Case Study 4: CE150 Feedback Loop Converging on Wrong Plan

**Scenario:** An e-commerce product search query with 6 filter parameters. CE150's join cardinality feedback initially corrected a 20x under-estimate, but the correction factor overshot on the third execution, causing the plan to flip between Hash Join and Nested Loops every 2-3 executions.

**Diagnosis:**
1. Execution 1: NL (under-estimate: 500 rows → actual 50K). Feedback correction factor = 100.
2. Execution 2: Hash Join (correction applied: 50K → actual 50K). Correct. Feedback = 1.0.
3. Execution 3: Hash Join (but stats change slightly → 55K). Feedback correction = 0.9. Over-corrects.
4. Execution 4: NL again (under-estimate again). Cycle repeats.

**Resolution:**
- Identified the correlated column set (Category, PriceRange, InStock).
- Created multi-column statistics: `CREATE STATISTICS s_Cat_Price_Stock ON Products(Category, PriceRange, InStock) WITH FULLSCAN;`
- After multi-column stats: estimates within 2x on all executions. Feedback loop stabilized at correction factor ~1.2.
- Duration stable at 800ms (was oscillating between 400ms and 6s).

**Key Learning:** Feedback loops cannot fix what the CE fundamentally cannot model. Multi-column statistics provide the missing correlation information that the CE needs.

### Case Study 5: Base Containment vs Simple Containment in Financial Reporting

**Scenario:** A financial reporting query joining 3 fact tables (Transactions, Positions, CashFlows) to 5 dimension tables (Account, Product, Currency, Region, Date).

**Before (CE70 Simple Containment):**
- Join cardinality estimates were 2-5x actual for most joins
- Plan used Hash Match for all joins (sized for over-estimated rows)
- Memory grant: 12GB (8GB actually used) → 4GB wasted
- Duration: 8 minutes

**After (CE150 Base Containment):**
- Base containment reduced join estimates to within 30% of actual
- Two joins switched from Hash Match to Merge Join (sorted inputs from covering indexes)
- Memory grant: 6GB (5.8GB actually used) → much tighter
- Duration: 3 minutes 20 seconds

**Metrics:**

| Metric | CE70 (Simple) | CE150 (Base) | Improvement |
|---|---|---|---|
| Join output estimate | 12.5M rows | 4.2M rows | 3x more accurate |
| Memory grant | 12 GB | 6 GB | 50% reduction |
| Actual memory used | 8 GB | 5.8 GB | 27% reduction |
| Duration | 8 min 12 sec | 3 min 20 sec | 59% faster |
| Logical reads | 1,240,000 | 520,000 | 58% reduction |

### Case Study 6: ASSUME_MIN_SELECTIVITY Fix for Dashboard Over-Estimate

**Scenario:** A real-time dashboard query with 8 independent filter criteria (Region, ProductCategory, SalesChannel, CustomerSegment, DateRange, PaymentMethod, PromotionCode, ShippingMethod). CE150 estimated 340K rows; actual was 4,200 rows. The 80x over-estimate caused the dashboard to use a Hash Join when Nested Loops was sufficient.

**Fix Applied:**
`sql
SELECT Region, SUM(SalesAmount) AS TotalSales
FROM Sales
WHERE Region = @Region
  AND ProductCategory = @Category
  AND SalesChannel = @Channel
  AND CustomerSegment = @Segment
  AND OrderDate BETWEEN @StartDate AND @EndDate
  AND PaymentMethod = @Payment
  AND PromotionCode = @Promo
  AND ShippingMethod = @Shipping
GROUP BY Region
OPTION (USE HINT('ASSUME_MIN_SELECTIVITY_FOR_FILTER_ESTIMATES'));
`

**After Fix:** CE150 with ASSUME_MIN_SELECTIVITY estimated 5,100 rows (close to actual 4,200). Plan used Nested Loops. Duration: from 12 seconds to 200ms.

**Key Learning:** The ASSUME_MIN_SELECTIVITY hint is specifically designed for this scenario — many WHERE predicates that are genuinely independent. It tells the optimizer to use the most restrictive predicate's selectivity rather than applying exponential backoff.

### Scenario: Diagnosing a Plan Regression with CE150

A query joins 4 tables with 6 WHERE predicates. Under CE70: 500ms. Under CE150: 8 seconds.

<details>
**Step 1:** Check CE version in plan XML → CardinalityEstimationModelVersion = 150.

**Step 2:** Compare estimate rows:
- CE70 plan: EstimateRows = 250 at root → Nested Loops → 500ms
- CE150 plan: EstimateRows = 85,000 at root → Hash Match (spills) → 8 seconds

**Step 3:** Actual rows = 320. CE70 estimate (250) is much closer than CE150 (85,000).

**Step 4:** The predicates are on independent columns (Status, Type, Region, Channel, PaymentMethod, Source). CE150's exponential backoff over-estimates because it assumes some correlation. CE70's multiplication is more accurate for independent predicates.

**Step 5:** Fix: `OPTION (USE HINT('ASSUME_MIN_SELECTIVITY_FOR_FILTER_ESTIMATES'))` — forces CE150 to use the minimum selectivity (most restrictive predicate) instead of exponential backoff. After fix: EstimateRows = 380, Nested Loops, 450ms.
</details>

### Advanced: Query Store CE Version Tracking

`sql
-- Query Store tracks CE version per plan
SELECT qsq.query_id, qsp.plan_id, qsqt.query_sql_text,
    qsp.cardinality_estimation_model_version,
    qrs.avg_duration / 1000 AS avg_duration_ms,
    qrs.avg_logical_reads,
    qrs.avg_rows_actual,
    qrs.avg_rows_estimated,
    CASE WHEN qrs.avg_rows_actual > qrs.avg_rows_estimated * 10 THEN 'UNDER'
         WHEN qrs.avg_rows_estimated > qrs.avg_rows_actual * 10 THEN 'OVER'
         ELSE 'OK' END AS estimate_quality,
    qsp.last_compile_start_time,
    qsp.is_forced_plan,
    qsp.last_force_failure_reason_desc
FROM sys.query_store_query qsq
JOIN sys.query_store_query_text qsqt ON qsq.query_text_id = qsqt.query_text_id
JOIN sys.query_store_plan qsp ON qsq.query_id = qsp.query_id
JOIN sys.query_store_runtime_stats qrs ON qsp.plan_id = qrs.plan_id
WHERE qsp.cardinality_estimation_model_version IS NOT NULL
    AND qsp.last_compile_start_time > DATEADD(DAY, -7, GETDATE())
ORDER BY qrs.avg_duration DESC;
`

| USE HINT | What It Does | Best For |

| FORCE_DEFAULT_CARDINALITY_ESTIMATION | Use New CE regardless of database setting | Testing CE150 on CE70 database |
| FORCE_LEGACY_CARDINALITY_ESTIMATION | Use CE70 regardless of database setting | Fixing regression from CE150 |

### Reference: CE150 vs CE70 Cost Model Differences

| Model Component | CE70 | CE150 | Impact |
|---|---|---|---|
| Filter selectivity | Multiplication | Exponential backoff | 5+ predicates: CE150 estimates up to 100x higher |
| Join containment | Simple (assumes overlap) | Base (frequency vector) | Star joins: CE150 20-40% lower |
| ASCENDING key | 0 row estimate | 200-row sampling | New data: CE150 avoids 0-row plan |
| TVF cardinality | 1 row | 1 row (CE120) / actual (CE140+) | TVFs: CE140+ necessary |
| Join to join correlation | Not modeled | Not modeled | Still inaccurate for complex join chains |
| Multi-column correlation | Not modeled (unless multi-col stats) | Not modeled (unless multi-col stats) | Multi-col stats still essential |
| Feedback loops | None | Join CE + Memory Grant + UDF | Converges over 2-3 executions |
