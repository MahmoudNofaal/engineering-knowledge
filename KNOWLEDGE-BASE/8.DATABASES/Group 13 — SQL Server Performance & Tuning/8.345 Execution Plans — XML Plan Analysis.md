# 8.345 — Execution Plans: XML Plan Analysis

## Section 1 — Navigation

**Breadcrumb:** Domain 8 — Databases > Group 13 — SQL Server Performance & Tuning > 8.345

**Where This Fits:** The graphical plan in SSMS is a visualization of an XML document. Every piece of information - row estimates, costs, CE version, parameter values, missing indexes, runtime statistics - is stored in the XML. Programmatic access via XPath/XQuery enables automated plan analysis at scale. This note covers the XML structure, key elements (RelOp, RunTimeInformation, QueryPlan, ParameterList, MissingIndexes), and querying the plan cache using XPath.

**Classification:** The ShowPlanXML schema (http://schemas.microsoft.com/sqlserver/2004/07/showplan) is the canonical representation of all plan data. The SSMS graphical plan is just a rendering.

| XML Element | Contains | Use Case |
|---|---|---|
| StmtSimple | Query text, statement type, CE version, optimization level | Identify query and CE version |
| QueryPlan | DOP, memory grant, cached plan size | Memory grant tuning |
| RelOp | Physical/logical operator, row estimates, costs | Operator-level analysis |
| RunTimeInformation | Actual rows per thread, executions, reads | Post-execution validation |
| SeekPredicates | SARGable predicates used by seek operators | Index usage analysis |
| ParameterList | Compiled parameter values, data types | Parameter sniffing detection |
| MissingIndexes | Index recommendation with impact % | Index tuning automation |
| Warnings | Spills, missing stats, no join predicate | Alert automation |
| MemoryGrantInfo | Grant size, requested/used memory | Memory tuning |
| CardinalityEstimationFeedback | Feedback type, correction factor | CE150 feedback monitoring |

**Key Properties:**
1. Namespace-aware: All XPath queries must declare the namespace
2. NodeId: Every operator has a unique NodeId linking it to RunTimeInformation
3. Statement-specific: Each statement in a batch has its own StmtSimple
4. Runtime vs Compile: RunTimeInformation only appears in actual plans

### XPath Queries to Extract Plan Info

`sql
-- Setup: enable XPath namespace shorthand
WITH XMLNAMESPACES(
    'http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p
)

-- Extract CE version
SELECT query_plan.value('(//p:StmtSimple/@CardinalityEstimationModelVersion)[1]', 'INT') AS ce_version;

-- Extract optimization level
SELECT query_plan.value('(//p:StmtSimple/@StatementOptmLevel)[1]', 'VARCHAR(10)') AS opt_level;

-- Extract memory grant
SELECT query_plan.value('(//p:QueryPlan/@MemoryGrant)[1]', 'BIGINT') AS memory_grant_kb;

-- Extract missing index details
SELECT query_plan.query('//p:MissingIndexes/p:MissingIndexGroup') AS missing_index;

-- Extract all operators with estimated rows
SELECT t.c.value('@NodeId', 'INT') AS node_id,
       t.c.value('@PhysicalOp', 'VARCHAR(100)') AS physical_op,
       t.c.value('@EstimateRows', 'FLOAT') AS estimate_rows,
       t.c.value('@EstimatedTotalSubtreeCost', 'FLOAT') AS subtree_cost
FROM query_plan.nodes('//p:RelOp') t(c);

-- Extract runtime actual rows per operator
SELECT t.c.value('@NodeId', 'INT') AS node_id,
       t.c.value('@PhysicalOp', 'VARCHAR(100)') AS physical_op,
       t2.r.value('@ActualRows', 'BIGINT') AS actual_rows,
       t2.r.value('@ActualReads', 'BIGINT') AS actual_reads
FROM query_plan.nodes('//p:RelOp') t(c)
CROSS APPLY t.c.nodes('p:RunTimeInformation/p:RunTimeCountersPerThread') t2(r)
WHERE t.c.exist('p:RunTimeInformation') = 1;
`

### Querying Plan Cache with XPath

`sql
-- Find all queries with warnings (spills, missing stats)
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
SELECT SUBSTRING(st.text, 1, 200) AS query_text, qs.execution_count,
    qs.total_elapsed_time / qs.execution_count / 1000 AS avg_ms,
    qp.query_plan.value('(//p:QueryPlan/p:Warnings/@SpillToTempDb)[1]', 'INT') AS spill_to_tempdb,
    qp.query_plan.value('count(//p:RelOp[p:Warnings])', 'INT') AS operator_warnings
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
WHERE qp.query_plan.exist('//p:Warnings') = 1
ORDER BY qs.total_elapsed_time DESC;
`

### Failure Modes in XML Plan Parsing

| Failure Mode | Symptom | Fix |
|---|---|---|
| Empty XPath result | .value returns NULL | Check namespace declaration |
| Multiple matches error | XPath returns error | Add [1] or use .nodes |
| Invalid XML | Cast to XML fails | Use SET STATISTICS XML ON |
| Slow XPath queries | Minutes to run | Filter plans before shredding |
| Namespace mismatch | XPath never matches | Check @Version and adjust |
| RunTimeInformation missing | Actual rows NULL | Use actual plan, not estimated |

## Section 4 — Production Patterns

### Pattern 1: Automated Missing Index Detection

`sql
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
SELECT qs.query_hash, qs.total_elapsed_time/qs.execution_count/1000 AS avg_ms,
    qs.total_logical_reads/qs.execution_count AS avg_reads, qs.execution_count,
    mid.value('@Database', 'VARCHAR(100)') AS db,
    mid.value('@Schema', 'VARCHAR(100)') AS schema_name,
    mid.value('@Table', 'VARCHAR(100)') AS table_name,
    mid.value('(p:ColumnGroup[@Usage="EQUALITY"]/p:Column/@Name)[1]', 'VARCHAR(100)') AS eq_cols,
    mid.value('(p:ColumnGroup[@Usage="INEQUALITY"]/p:Column/@Name)[1]', 'VARCHAR(100)') AS ineq_cols,
    (SELECT c.value('@Name','VARCHAR(100)')+', ' FROM mid.nodes('p:ColumnGroup[@Usage="INCLUDE"]/p:Column') t(c) FOR XML PATH('')) AS inc_cols,
    mig.value('@Impact', 'FLOAT') AS impact_pct
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
CROSS APPLY qp.query_plan.nodes('//p:MissingIndexGroup') t(mig)
CROSS APPLY mig.nodes('p:MissingIndex') s(mid)
ORDER BY impact_pct DESC;
`

### Pattern 2: Parameter Sniffing Detection via ParameterList

`sql
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
SELECT qs.query_hash, qs.plan_handle, qs.execution_count,
    param.value('@Column', 'VARCHAR(100)') AS parameter_name,
    param.value('@ParameterCompiledValue', 'VARCHAR(100)') AS compiled_value,
    param.value('@ParameterRuntimeValue', 'VARCHAR(100)') AS runtime_value,
    CASE WHEN param.value('@ParameterCompiledValue','VARCHAR(100)') != param.value('@ParameterRuntimeValue','VARCHAR(100)')
         THEN 'SNIFFED DIFFERENT' ELSE 'same' END AS sniffing_check
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
CROSS APPLY qp.query_plan.nodes('//p:ParameterList/p:ColumnReference') t(param)
WHERE param.value('@ParameterCompiledValue','VARCHAR(100)') IS NOT NULL
ORDER BY qs.total_elapsed_time DESC;
`

### Pattern 3: Memory Grant Analysis

`sql
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
SELECT qs.query_hash, qs.execution_count,
    qp.query_plan.value('(//p:QueryPlan/@MemoryGrant)[1]', 'BIGINT') AS grant_kb,
    qp.query_plan.value('(//p:MemoryGrantInfo/@MaxUsedMemory)[1]', 'BIGINT') AS used_kb,
    qp.query_plan.value('(//p:MemoryGrantInfo/@GrantWaitTime)[1]', 'BIGINT') AS wait_ms,
    CASE WHEN 1.0*qp.query_plan.value('(//p:MemoryGrantInfo/@MaxUsedMemory)[1]','BIGINT')/NULLIF(qp.query_plan.value('(//p:MemoryGrantInfo/@GrantedMemory)[1]','BIGINT'),0) < 0.3
         THEN 'OVER-GRANT' ELSE 'OK' END AS efficiency
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
WHERE qp.query_plan.value('(//p:QueryPlan/@MemoryGrant)[1]','BIGINT') > 1024*100
ORDER BY CAST(qp.query_plan.value('(//p:QueryPlan/@MemoryGrant)[1]','BIGINT') AS BIGINT) DESC;
`

### Pattern 4: CE150 Feedback Loop Monitoring

`sql
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
SELECT qs.query_hash, qs.execution_count, qs.last_execution_time,
    feedback.value('@FeedbackType', 'VARCHAR(50)') AS feedback_type,
    feedback.value('@CorrectionFactor', 'FLOAT') AS correction_factor,
    feedback.value('@LastUsed', 'DATETIME') AS last_used
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
CROSS APPLY qp.query_plan.nodes('//p:CardinalityEstimationFeedback/p:JoinedTable/p:EstimationInfo') t(feedback)
WHERE feedback.exist('@CorrectionFactor') = 1
ORDER BY qs.last_execution_time DESC;
`

### Pattern 5: EF Core — Serialize Plan XML for Analysis

`csharp

### Pattern 6: Extracting Operator Tree as Relational Rows

`sql
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
SELECT qs.query_hash, t.c.value('@NodeId','INT') AS node_id,
    t.c.value('@PhysicalOp','VARCHAR(100)') AS physical_op,
    t.c.value('@LogicalOp','VARCHAR(100)') AS logical_op,
    t.c.value('@EstimateRows','FLOAT') AS estimated_rows,
    t.c.value('@EstimateIO','FLOAT') AS estimated_io,
    t.c.value('@EstimateCPU','FLOAT') AS estimated_cpu,
    t.c.value('@AvgRowSize','INT') AS avg_row_size,
    t.c.value('@EstimatedTotalSubtreeCost','FLOAT') AS subtree_cost,
    t2.r.value('@ActualRows','BIGINT') AS actual_rows,
    t2.r.value('@ActualReads','BIGINT') AS actual_reads
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
CROSS APPLY qp.query_plan.nodes('//p:RelOp') t(c)
OUTER APPLY t.c.nodes('p:RunTimeInformation/p:RunTimeCountersPerThread') t2(r)
ORDER BY qs.total_elapsed_time DESC, t.c.value('@NodeId','INT');
`

## Section 5 — Gotchas

### Gotcha 1 — Namespace Must Match Exactly
**Pitfall:** Case-sensitive namespace. Single character mismatch = NULL for all XPath.
**Fix:** Copy namespace directly from plan XML. Use reusable declaration.
**Cost:** Hours debugging what appears to be broken XML.

### Gotcha 2 — RunTimeInformation Only in Actual Plans
**Pitfall:** Querying ActualRows from sys.dm_exec_query_plan always returns NULL (it stores estimated plans).
**Fix:** Use sys.dm_exec_query_stats for runtime stats, or Query Store. Only SET STATISTICS XML ON produces plans with RunTimeInformation.
**Cost:** Incorrect assumption that actual rows are 0.

### Gotcha 3 — Multiple Statements in One Batch
**Pitfall:** XPath with [1] only reads first statement. Batch with DECLARE+SELECT+UPDATE has multiple StmtSimple.
**Fix:** Use .nodes('//p:StmtSimple') to shred all statements, filter by StatementId.
**Cost:** Missing plan regression on UPDATE because you only analyzed SELECT.

### Gotcha 4 — ParameterCompiledValue May Be Truncated
**Pitfall:** Long parameter values (JSON, ID lists) get truncated in plan XML.
**Fix:** Use string comparison with LEFT, or hash for comparison.
**Cost:** False negatives in parameter sniffing detection.

### Gotcha 5 — MissingIndexes Order Does Not Imply Priority
**Pitfall:** First missing index in XML is not necessarily most impactful.
**Fix:** Always check @Impact attribute on MissingIndexGroup. Highest impact wins.
**Cost:** Wasted maintenance on low-impact indexes.

### Gotcha 6 — XML Plan Size Limitation (2MB)
**Pitfall:** sys.dm_exec_query_plan returns NULL for plans > 2MB.
**Fix:** Use sys.dm_exec_text_query_plan which returns NVARCHAR(MAX) without size limit.
**Cost:** Incomplete plan analysis for large queries.

## Section 6 — Performance Implications

### Cost of XML Plan Parsing

`sql
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
SELECT COUNT(*) AS plans, AVG(LEN(CONVERT(NVARCHAR(MAX), qp.query_plan))) AS avg_plan_size,
    COUNT(DISTINCT t.c.value('@NodeId','INT')) AS total_operators
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
CROSS APPLY qp.query_plan.nodes('//p:RelOp') t(c)
WHERE qp.query_plan IS NOT NULL;
`

**Performance characteristics:**
- Simple .value() on 1000 plans: ~50ms
- .nodes() shredding all RelOps on 1000 plans: ~200-500ms
- .exist() with XPath filter on 1000 plans: ~30ms

**Best practices:** Filter plans before parsing, use selective XPath, consider external tools for large-scale analysis.

### Benchmark

| Operation | Plans/sec | CPU per plan |
|---|---|---|
| Simple .value() (1 attribute) | 20,000 | 0.05ms |
| .nodes() shredding all RelOps | 2,500 | 0.4ms |
| Full XML cast to NVARCHAR | 50,000 | 0.02ms |
| .exist() with warn filter | 15,000 | 0.07ms |

## Section 7 — Interview Arsenal

### Interview Questions

**Q1: What is ShowPlanXML and why is it important?**
*Tier 1:* The underlying XML schema for all execution plans. SSMS graphical plan is a visualization of this XML.

*Tier 2:* The schema defines the canonical plan representation. I use it for automated analysis: extracting CE versions, detecting parameter sniffing, identifying missing indexes by impact, and monitoring CE150 feedback loops. XML scales to hundreds of plans while manual SSMS review does not.

**Q2: How do you extract CE version from plan XML?**
*Spoken answer:* CardinalityEstimationModelVersion on StmtSimple. XPath: //p:StmtSimple/@CardinalityEstimationModelVersion. Values: 70, 120, 130, 140, 150. I query this across the plan cache to find databases running legacy CE.

**Q3: Difference between sys.dm_exec_query_plan and sys.dm_exec_text_query_plan?**
*Spoken answer:* query_plan returns XML with 2MB limit. text_query_plan returns NVARCHAR(MAX) with no limit. Use text version for large plans or when XML returns NULL.

**Q4: How do you detect memory grant over-allocation from XML?**
*Spoken answer:* Compare GrantedMemory vs MaxUsedMemory from MemoryGrantInfo. If MaxUsedMemory < 30% of GrantedMemory, the query is over-allocating. Flag queries with GrantedMemory > 100MB and poor utilization.

**Q5: What is RunTimeInformation and when does it appear?**
*Spoken answer:* Inside each RelOp in actual plans. Contains RunTimeCountersPerThread with ActualRows, ActualEndOfScans, ActualReads. Only appears with SET STATISTICS XML ON or Query Store. NOT in estimated plans or sys.dm_exec_query_plan.

**Q6: How do you automate missing index detection?**
*Spoken answer:* Query MissingIndexes element with Impact percentage. Each MissingIndex has EQUALITY, INEQUALITY, INCLUDE ColumnGroups. Join with sys.dm_exec_query_stats to prioritize by execution count and impact.

**Q7: What does RetrievedFromCache tell you?**
*Spoken answer:* Whether plan was retrieved from cache (true) or freshly compiled (false). Most production plans show true. Helps distinguish ad-hoc analysis vs actual cached behavior.

**Q8: How do you identify compile-time vs runtime memory grant differences?**
*Spoken answer:* MemoryGrantInfo has SerialRequiredMemory, DesiredMemory (compile-time) and GrantedMemory, MaxUsedMemory (runtime). DesiredMemory > GrantedMemory = memory pressure. MaxUsedMemory << GrantedMemory = over-allocation.

### Comparison Table

| DMV / View | Plan Type | Size Limit | Runtime Stats | Use Case |
|---|---|---|---|---|
| sys.dm_exec_query_plan | Estimated | 2MB | No | Quick retrieval |
| sys.dm_exec_text_query_plan | Estimated | None | No | Large plans |
| sys.dm_exec_query_stats | N/A | N/A | Yes | Perf metrics |
| SET STATISTICS XML ON | Actual | None | Yes | Single query |
| Query Store | Both | 2MB | Yes | Continuous monitoring |
| SET SHOWPLAN_XML ON | Estimated | None | No | What-if analysis |

## Section 8 — Decision Framework

`mermaid
flowchart TD
    A[Need to analyze plans at scale?] -->|Manual| B[Use SSMS graphical plan]
    A -->|Automated| C[Choose XML analysis method]
    C --> D{Data source?}

    D -->|Extended Events| G[XEvent plan capture]
    D -->|Ad-hoc| H[SET STATISTICS XML ON]
    E --> I{What to extract?}
    I -->|CE version| J[CardinalityEstimationModelVersion]
    I -->|Parameter sniffing| K[ParameterList Compiled vs Runtime]
    I -->|Memory| L[MemoryGrantInfo ratios]
    I -->|Missing indexes| M[MissingIndexes with Impact]
    I -->|Warnings| N[Spills, missing stats]
    I -->|Operator tree| O[RelOp shredding via .nodes]

    N --> T[Alert on spills]
    O --> U[Detect NL with high execution count]
`

### Decision Checklist

1. Determine source: plan cache DMVs, Query Store, XEvent, or ad-hoc
2. For plan cache: start with sys.dm_exec_query_stats to find expensive queries, then join to sys.dm_exec_query_plan
3. For XPath: declare namespace, test with .exist() before extracting values
4. Extract: CE version, ParameterList, MissingIndexes, MemoryGrant, Warnings
5. For each operator (.nodes): compare EstimateRows vs actual, check PhysicalOp, check Parallel
6. Generate output: index scripts, hint recommendations, stats updates
7. Validate: implement top-3, re-run analysis

### Tradeoffs

| Approach | Benefit | Risk | Best For |
|---|---|---|---|
| Plan cache DMVs | Always available, zero overhead | Estimated only | Regular monitoring |
| Query Store | Runtime+compile, history | Must enable | Long-term trends |
| Extended Events | Complete capture | Overhead | Deep debugging |
| SSMS graphical | Visual, intuitive | Manual, doesn't scale | First-line |
| External tools | Flexible, complex | Setup overhead | Large-scale analysis |

### Scale Thresholds

- <100 queries: Manual XML review in SSMS
- 100-1000: Use T-SQL XPath queries
- 1000-10000: Filter by execution stats first. Consider external tools
- >10000: Query Store with targeted policies. Automated scripts with thresholds

**Challenge 2:** Create stored procedure that returns operator tree from plan XML.
<details>`sql
CREATE PROC dbo.ParsePlanXml @plan_xml XML AS
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
SELECT t.c.value('@NodeId','INT') AS node_id, t.c.value('@PhysicalOp','VARCHAR(100)') AS phys_op,
       t.c.value('@EstimateRows','FLOAT') AS est_rows, t.c.value('@EstimateIO','FLOAT') AS est_io,
       t.c.value('@EstimatedTotalSubtreeCost','FLOAT') AS subtree_cost,
       t2.r.value('@ActualRows','BIGINT') AS actual_rows, t2.r.value('@ActualReads','BIGINT') AS actual_reads

## Section 9 — Extended: Advanced XPath & FLWOR Patterns

### FLWOR Expressions for Plan Analysis

FLWOR (For-Let-Where-Order-Return) expressions in XQuery enable complex plan analysis beyond simple .value() extraction:

`sql
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
SELECT qs.query_hash,
    qp.query_plan.query('
        for $relop in //p:RelOp
        let $actual := $relop/p:RunTimeInformation/p:RunTimeCountersPerThread
        where $relop/@EstimateRows > 0
            and exists($actual)
            and ($actual/@ActualRows div $relop/@EstimateRows > 10
                 or $relop/@EstimateRows div $actual/@ActualRows > 10)
        order by ($actual/@ActualRows div $relop/@EstimateRows) descending
        return <op mismatch="{if ($actual/@ActualRows > $relop/@EstimateRows) then "under" else "over"}"
                   nodeId="{$relop/@NodeId}"
                   physOp="{$relop/@PhysicalOp}"
                   estRows="{$relop/@EstimateRows}"
                   actRows="{$actual/@ActualRows}"
                   ratio="{if ($actual/@ActualRows > $relop/@EstimateRows)
                           then $actual/@ActualRows div $relop/@EstimateRows
                           else $relop/@EstimateRows div $actual/@ActualRows}"/>
    ') AS mismatches
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
WHERE qp.query_plan.exist('//p:RunTimeInformation') = 1
ORDER BY qs.total_elapsed_time DESC;
`

### Using .query() to Extract Sub-Plans

Extract only the portion of the plan tree relevant to a specific NodeId:

`sql
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)

    qp.query_plan.query('

FROM sys.dm_exec_query_stats qs

`

## Section 9 — Extended: Advanced Self-Check

### Additional Conceptual Questions

**Q11:** Why does .value() with an XPath returning multiple nodes require [1]?
<details>XPath may match multiple nodes; .value() requires a singleton. Adding [1] takes the first matching node. To handle multiple matches, use .nodes() to shred into rows, then apply .value() per row.</details>

**Q12:** How do you check if a plan XML contains a specific operator type?
<details>Use .exist('//p:RelOp[@PhysicalOp="Index Seek"]'). Returns 1 if any operator is Index Seek. Combine with different PhysicalOp values to classify plan shapes.</details>

**Q13:** What is the difference between .value() and .query() in T-SQL XML methods?
<details>.value() extracts a single scalar value. .query() returns an XML fragment. Use .value() for attribute extraction (rows, costs, versions). Use .query() for extracting sub-trees (a specific operator with its children).</details>

**Q14:** How to handle plans with multiple statements (batches) when extracting CE version?
<details>Each statement has its own StmtSimple. Use .nodes('//p:StmtSimple') to capture each statement separately, then extract per-statement CE versions. A DECLARE + SELECT + UPDATE batch may have different CE versions per statement.</details>

**Q15:** What causes .exist() to return NULL instead of 0 or 1?
<details>.exist() returns NULL when the XML is NULL (plan >2MB or plan handle invalid). Always check for NULL input before calling .exist(). Use COALESCE(.exist(...), 0) to handle NULL.</details>

**Q16:** How do you extract per-thread row distribution from parallel plans?
<details>Shred RunTimeCountersPerThread — each thread is a separate element with Thread attribute. Compare ActualRows per thread to detect skew:</details>

`sql
WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
SELECT qs.query_hash, qs.plan_handle,
    t.c.value('@NodeId','INT') AS node_id,
    t2.r.value('@Thread','INT') AS thread,
    t2.r.value('@ActualRows','BIGINT') AS actual_rows
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
CROSS APPLY qp.query_plan.nodes('//p:RelOp') t(c)
CROSS APPLY t.c.nodes('p:RunTimeInformation/p:RunTimeCountersPerThread') t2(r)
WHERE qp.query_plan.exist('//p:RelOp[@Parallel="true"]') = 1
ORDER BY qs.total_elapsed_time DESC, node_id, thread;
`

**Q17:** How do you use XPath to find plans with both missing indexes AND spills?
<details>Combine exist() checks: .exist('//p:MissingIndexes') = 1 AND .exist('//p:Warnings[@SpillToTempDb="1"]') = 1. These are the highest-value tuning targets — they have a recommended index AND a current spill.</details>

**Q18:** How to detect compile-time memory grant vs actual usage ratio from XML?
<details>Compare MemoryGrantInfo/GrantedMemory vs MaxUsedMemory. Efficiency = MaxUsedMemory * 100 / NULLIF(GrantedMemory, 0). Values < 30% indicate over-grant; values > 90% indicate tight grant (risk of spill if estimate is off).</details>

**Q19:** What is the XPath to extract the query text from a plan XML?
<details>(//p:StmtSimple/@StatementText)[1] — but this is truncated in the XML. For full text, use sys.dm_exec_sql_text(sql_handle) from DMVs, or look at the raw query batch.</details>

**Q20:** How to find plans with adaptive joins and see which join type was selected at runtime?
<details>Check for PhysicalOp="Adaptive Join" in RelOp. The RunTimeInformation includes which join type was actually used (Nested Loops or Hash Match). Also check @AdaptiveThresholdValue and @Isadaptive in the RelOp attributes.</details>
