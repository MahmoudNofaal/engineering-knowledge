## Section 5 — Gotchas

### Gotcha 1 — Database Upgrade Changes CE Silently
**Pitfall:** Restoring a database from SQL 2012 (CE70) to SQL 2022 (CE150) changes the active CE without changing the query text.
**Symptom:** Previously fast queries time out after migration. Plan cache is invalidated on compatibility level change.
**Fix:** Run the Query Tuning Assistant (QTA) in SSMS or test with QUERYTRACEON 9481 before committing to the upgrade.
**Cost:** Hours of outage if a critical query regresses in production.

### Gotcha 2 — ASCENDING Key Still Under-Estimates
**Pitfall:** CE120+ adds a 200-row sampling for out-of-range dates, but for extremely large tables with bulk inserts, this is still too small.
**Symptom:** Queries filtering on OrderDate > Today estimate 200 rows but return 1M rows. Spills to tempdb.
**Fix:** Use OPTION (USE HINT('ENABLE_HISTORICAL_CARDINALITY_ESTIMATION')) or manually update stats with FULLSCAN.
**Cost:** 10x-100x logical reads; potential timeout for the first execution until plan is recompiled.

### Gotcha 3 — Legacy CE Can Be Better for Simple Queries
**Pitfall:** Assuming CE150 is always better because it is newer.
**Symptom:** CE150 produces a hash join for a simple lookup returning 10 rows because it over-estimates cardinality due to the base containment assumption.
**Fix:** Compare both CEs on the specific query workload before forcing a version.
**Cost:** Sub-optimal plan shape causes 2-5x more logical reads.

### Gotcha 4 — CE Version Is Not the Same as Compatibility Level
**Pitfall:** Setting COMPATIBILITY_LEVEL = 150 does not guarantee CE150 if database scoped configurations override it.
**Symptom:** ALTER DATABASE SCOPED CONFIGURATION SET LEGACY_CARDINALITY_ESTIMATION = ON silently reverts to CE70.
**Fix:** Verify with SELECT * FROM sys.database_scoped_configurations WHERE name = 'LEGACY_CARDINALITY_ESTIMATION'.
**Cost:** Months of undiagnosed performance issues.

### Gotcha 5 — Trace Flags Override Database Configuration
**Pitfall:** QUERYTRACEON 9481 at the query level or DBCC TRACEON(9481, -1) at the instance level overrides all database-level settings.
**Symptom:** Even with the correct compatibility level, CE70 is used for all databases on the instance.
**Fix:** Check DBCC TRACESTATUS(-1) during investigation.
**Cost:** Wasted investigation time chasing wrong assumptions.

### Gotcha 6 — Parameter Sniffing Interacts with CE Version
**Pitfall:** CE70 and CE120+ respond differently to parameter sniffing because their density calculations differ, causing plan variability across parameter values.
**Symptom:** Same query, same parameters, different CE versions produce drastically different plans.
**Fix:** Use OPTION (OPTIMIZE FOR UNKNOWN) combined with the preferred CE version.
**Cost:** Hard-to-diagnose intermittent performance degradation.

## Section 7 — Interview Arsenal

### Interview Questions

**Q1: How does cardinality estimation affect query performance?**
*Spoken answer (tier 1):* The CE predicts row counts at each operator. The optimizer uses those counts to cost operators - hash join, merge join, nested loops, memory grant, index selection. If estimates are off by 10x or more, the wrong plan is chosen, leading to spills, excessive logical reads, or timeouts.

*Spoken answer (tier 2):* CE is the foundation of cost-based optimization. A 1-row estimate for an ASCENDING key join forces a nested loop that iterates millions of times. CE70 under-estimates correlated predicates because it assumes independence. CE120+ uses exponential backoff: sqrt(sel1 * sel2) instead of sel1 * sel2. This is why upgrading from 2012 to 2016+ causes plan regressions - queries that ran fine under CE70 might suddenly get large memory grants or hash joins under CE120+.

**Q2: What is the difference between CE70 and CE120 in how they handle multiple predicate selectivity?**
*Spoken answer:* CE70 multiplies selectivities assuming independence. For a=1 AND b=2 with 1% selectivity each: 0.01 * 0.01 = 0.0001. CE120 uses exponential backoff: higher of the two selectivities * lower^(1/2). For correlated columns, CE120 produces higher estimates, which is more accurate. This is the single biggest difference between the CEs.

**Q3: When would you force CE70 for a specific query?**
*Spoken answer:* When after an upgrade, a specific query shows plan regression - where the new CE's estimate pushes the optimizer to choose a hash join or a larger memory grant that causes spills. I use OPTION (QUERYTRACEON 9481) for a single query or USE HINT('FORCE_LEGACY_CARDINALITY_ESTIMATION') for a cleaner approach. I monitor with Query Store to identify regressions.

**Q4: Explain the ASCENDING KEY issue and how CE120+ attempts to fix it.**
*Spoken answer:* When a column has strictly increasing values (like OrderDate or Identity), the histogram may not cover new values. CE70 estimates 0 rows for WHERE OrderDate > Today. CE120+ detects ASCENDING keys during compilation and updates a portion of the statistics (200 rows) on-the-fly, producing a non-zero estimate. However, if 200 rows is far too small, manual stats maintenance is needed.

**Q5: What is the base containment assumption?**
*Spoken answer:* CE70 uses simple containment: for a join A JOIN B ON A.x = B.y, it assumes every distinct value in the smaller table exists in the larger one. CE120+ uses base containment: it respects the actual density and distribution, producing more accurate join cardinalities. This matters most for star-joins where a large fact table joins multiple small dimensions.

**Q6: How do you detect which CE version a query used?**
*Spoken answer:* Check the plan XML - the CardinalityEstimationModelVersion attribute in StmtSimple shows the version (70, 120, 130, 140, 150). Also check sys.dm_exec_query_stats combined with sys.dm_exec_query_plan. The database compatibility level is the primary gating factor, but trace flags and database scoped configurations can override it.

**Q7: What is the impact of CE70 on modern hardware?**
*Spoken answer:* CE70 was designed when CPUs were single-core and memory was measured in MB. Modern hardware handles more complex optimization. CE70's simpler model can miss plans that CE120+ would find. However, CE70's simpler costing can also produce faster compilation times for very simple queries. The tradeoff is compilation speed vs plan quality.

**Q8: How does CE150's feedback loop differ from earlier versions?**
*Spoken answer:* CE150 introduces the first feedback mechanisms: join cardinality feedback adjusts future estimates based on actual rows returned; memory grant feedback adjusts memory allocations based on actual usage. Earlier CEs had no learning - every execution used the same stale estimates until recompile. CE150 converges toward correct estimates over multiple executions.

### Comparison Table

| Feature | CE70 | CE120+ (120/140/150) |
|---|---|---|
| Independence assumption | Yes - multiplies | No - exponential backoff |
| Ascending key fix | None | 200-row auto stats update |
| Join containment | Simple | Base |
| Multiple TVFs | Estimates 1 row | Estimates actual (CE140+) |
| Feedback loops | None | Memory grant, join (CE150) |
| Plan complexity handled | Simple OLTP | Mixed OLTP/DW |
| Regression risk after upgrade | Baseline | Moderate (known pattern) |

## Section 8 — Decision Framework

`mermaid
flowchart TD
    A[Query Regression After Upgrade?] -->|Yes| B[Check Plan XML CardinalityEstimationModelVersion]
    B --> C{CE Version}
    C -->|CE70| D[No regression risk from CE. Investigate stats, indexes]
    C -->|CE120+| E[Is estimate accurate?]
    E -->|Actual Rows approx Estimated| F[CE is fine. Investigate other causes]
    E -->|Actual >> Estimated| G[CE under-estimating]
    E -->|Actual << Estimated| H[CE over-estimating]
    G --> I{Correlation or ASCENDING key?}
    I -->|Correlated predicates| J[Try CE70 via QUERYTRACEON 9481]
    I -->|ASCENDING key| K[Update stats FULLSCAN or tune stats maintenance]
    H --> L{Multiple predicates?}
    L -->|Yes| M[Exponential backoff too aggressive. Try OPTIMIZE FOR UNKNOWN]
    L -->|No| N[Look at join estimates, try FORCE_LEGACY for this query]
    J --> O[Compare plans side-by-side. Pick lower logical reads]
    K --> O
`

### Decision Checklist

1. Identify regression: Query Store, plan cache, or extended events
2. Check CE version used in the plan XML: CardinalityEstimationModelVersion
3. Compare actual vs estimated rows: SET STATISTICS PROFILE ON
4. Test with alternative CE: OPTION (USE HINT('FORCE_LEGACY_CARDINALITY_ESTIMATION'))
5. If CE70 fixes: keep it for this query, investigate why CE120+ misestimates
6. If CE120+ is better: update stats, consider indexed views for correlated columns
7. Long-term: create multi-column statistics (CREATE STATISTICS ON (col1, col2))
8. Monitor: set up alerts for plan changes via Query Store

### Tradeoffs

| Approach | Benefit | Risk | When to Use |
|---|---|---|---|
| CE150 (default) | Latest features, feedback loops | Regression risk on edge cases | Most databases on SQL 2019+ |
| CE70 via TF 9481 | Proven for OLTP | Missing new CE improvements | Specific query regression only |
| CE120 via COMPATIBILITY_LEVEL 120 | Stable for 2014 | Lacks feedback loops | Before upgrading past 2016 |
| Multi-column stats | Fixes correlation for both CEs | Maintenance overhead, storage | When column correlation is known |

### Scale Thresholds

- 0-50 GB databases: CE version rarely matters; simple queries dominated by I/O
- 50-500 GB: CE version differences visible; plan regressions possible
- 500 GB+: CE version choice can be the difference between 5-second and 5-minute queries; test thoroughly
- Enterprise with Resource Governor: CE feedback loops (CE150) help converge on stable plans for critical workloads

## Section 9 — Self-Check

### Conceptual Questions

**Q1:** What compatibility level introduced the new Cardinality Estimator?
<details>
SQL Server 2014, compatibility level 120. The new CE is available from 120 onward. CE70 covers compatibility levels 70 through 110.
</details>

**Q2:** How does CE70 calculate selectivity for WHERE A=1 AND B=2 vs CE120?
<details>
CE70: sel(A=1) * sel(B=2). CE120: sel(B=2) * sel(A=1)^(1/2) where the second predicate gets the exponential backoff treatment using the higher selectivity as the base.
</details>

**Q3:** What is the exponential backoff formula?
<details>
For n predicates with selectivity s1 >= s2 >= ... >= sn, CE120+ uses: s1 * s2^(1/2) * s3^(1/4) * ... * sn^(1/2^(n-1)). The most selective predicate gets full weight; less selective predicates get diminishing weight.
</details>

**Q4:** What is the ASCENDING key fix in CE120+?
<details>
When the optimizer detects an ASCENDING key column (via column value distribution or statistics modification counter), it performs a partial stats update by sampling up to 200 new rows, producing a non-zero estimate for out-of-range values.
</details>

**Q5:** What trace flag forces legacy CE?
<details>
TF 9481. At query level: OPTION (QUERYTRACEON 9481). Globally: DBCC TRACEON(9481, -1). Equivalent to USE HINT('FORCE_LEGACY_CARDINALITY_ESTIMATION').
</details>

**Q6:** What plan XML attribute identifies the CE version used?
<details>
CardinalityEstimationModelVersion on the StmtSimple element. Values: 70, 120, 130, 140, 150.
</details>

**Q7:** How does CE120 handle multiple-table TVF joins differently from CE70?
<details>
CE70 estimated multi-statement TVF joins as 1 row regardless of actual cardinality. CE140+ (not CE120) corrected this to estimate the actual number of rows the TVF returns. CE120 still uses the 1-row assumption.
</details>

**Q8:** What is the difference between simple containment and base containment for joins?
<details>
Simple containment (CE70) assumes every distinct value in the build side is contained within the probe side. Base containment (CE120+) uses the actual density and distribution, accounting for values that may not exist in both sides of the join.
</details>

**Q9:** Can you have CE150 with compatibility level 130?
<details>
No. CE150 requires compatibility level 150. With compatibility level 130, the database uses CE140. The CE version tracks the compatibility level, not the SQL Server version. However, database scoped configurations or trace flags can override the default CE.
</details>

**Q10:** How does sys.dm_db_stats_properties help diagnose CE behavior?
<details>
It shows the actual modification counter (modification_counter), which the CE uses to determine if statistics are stale. Combined with the histogram and density vector, it explains why the CE produced a particular estimate and whether auto-stats update is triggered.
</details>

## Section 9 — Extended Case Studies

### Case Study 1: CE Upgrade from SQL 2012 to SQL 2022

**Scenario:** A financial services application upgraded from SQL Server 2012 (CE70, compatibility level 110) to SQL Server 2022 (CE150, compatibility level 160).

**Observed:** After upgrade, one critical overnight batch process went from 15 minutes to 2+ hours and timing out.

**Diagnosis:**
1. Checked CE version on the plan XML: CardinalityEstimationModelVersion = 150 (new CE)
2. Found the problem query had 6 joined tables with 12 WHERE predicates
3. Under CE70: estimate was ~500 rows (independent multiplication: 0.5^12 = 0.0002)
4. Under CE150: estimate was ~250,000 rows (exponential backoff: more realistic for correlated financial data)
5. The 500-row CE70 plan used Nested Loops throughout (finishing in 15 minutes despite being wrong)
6. The 250K-row CE150 plan used Hash Joins with large memory grants, causing memory pressure and tempdb spills

**Resolution:**
- Option A (used): Forced CE70 for this specific query: OPTION (QUERYTRACEON 9481) — restored 15-minute runtime
- Option B (long-term): Created multi-column statistics on correlated predicate groups to improve CE150 estimates
- Option C (rejected): Forced CE70 database-wide — would lose CE150 benefits for all other queries

**Lessons Learned:**
- Always test critical queries under both CEs in a pre-production environment
- CE150 is not universally better; correlated predicates can be over-estimated
- Query Store captures both plan variants for comparison
- Plan guides are the safest way to force CE per query in production

### Case Study 2: ASCENDING Key Crisis

**Scenario:** An e-commerce site's order lookup query started timing out after a Black Friday sale inserted 2M orders in 24 hours.

**Diagnosis:**
1. Query: SELECT * FROM Orders WHERE OrderDate > '2026-11-29' ORDER BY OrderDate
2. CE70 estimate: 0 rows (histogram max was '2026-11-28')
3. CE120+ estimate: ~200 rows (ASCENDING key fix kicked in)
4. Actual rows returned: 2,100,000
5. The optimizer chose a narrow non-clustered index seek (expecting 0 or 200 rows) + key lookup for each row
6. 2.1M key lookups = catastrophic performance

**Resolution:**
- Immediate: UPDATE STATISTICS Orders WITH FULLSCAN to get accurate histogram
- Short-term: Scheduled stats update every 6 hours during high-volume periods
- Long-term: Implemented incremental statistics for partition-switched data

**Metrics:**
- Before: 42,000 logical reads, 180 seconds duration
- After stats update: 2,500 logical reads, 3 seconds duration
- Improvement: 16x reads, 60x duration

### Case Study 3: Parameter Sniffing Across CE Versions

**Scenario:** A parameterized stored procedure performed well under CE70 but regressed under CE150.

**Root Cause:** CE150's exponential backoff amplified the effect of parameter sniffing. When the sniffed parameter had high selectivity (few rows), CE70's multiplication produced a very small estimate, correctly choosing Nested Loops. CE150's backoff produced a larger estimate, sometimes crossing the threshold to Hash Join.

**Demonstration:**
`sql
CREATE PROC dbo.SearchOrders @Status INT, @CustomerId INT
AS
SELECT * FROM Orders WHERE Status = @Status AND CustomerId = @CustomerId;

-- Under CE70: sel(Status=1)=0.33 * sel(CustomerId=9999)=0.001 = 0.00033 -> 330 rows -> NL
-- Under CE150: sel(Status=1)=0.33 * sel(CustomerId=9999)^0.5=0.0316 = 0.0104 -> 10,400 rows -> Hash
`

**Fix:** Applied OPTION (OPTIMIZE FOR UNKNOWN) to produce a consistent plan across both CEs.

## Section 9 — Extended Self-Check: Practical Scenarios

### Scenario Questions

**SQ1:** You see an execution plan with a Key Lookup operator showing 500,000 actual rows and 5 logical reads per execution. The Number of Executions is 500,000. What is the total cost in logical reads, and what is the fix?
<details>
Total logical reads = 500,000 * 5 = 2,500,000 reads. The Key Lookup is executed once per outer row. The fix is to add the missing columns to the non-clustered index as INCLUDE columns, eliminating the lookup entirely. Expected improvement: 99% reduction in logical reads.
</details>

**SQ2:** A query under CE70 estimates 100 rows but returns 100,000 actual rows. Under CE150, it estimates 50,000 rows. Which CE should you use and why?
<details>
CE150 is better (50K estimate vs 100K actual = 2x mismatch vs 1000x mismatch). However, if the 100x under-estimate under CE70 doesn't change the plan shape (Nested Loops is fine for 100K rows with an index seek), keeping CE70 may be acceptable. Test both and choose based on actual logical reads, not estimate accuracy alone.
</details>

**SQ3:** Your production SQL Server suddenly starts using CE70 even though compatibility level is 150. What happened?
<details>
Check sys.database_scoped_configurations for LEGACY_CARDINALITY_ESTIMATION = ON. This can be set inadvertently by a migration script, a DBA action, or a deployment tool. Also check DBCC TRACESTATUS() for global trace flag 9481. The fix: ALTER DATABASE SCOPED CONFIGURATION SET LEGACY_CARDINALITY_ESTIMATION = OFF.
</details>

**SQ4:** A query plan shows actual rows of 10,000 but estimated rows of 1 for a TVF. What CE version is being used and what is the fix?
<details>
This is characteristic of CE70 or CE120, which estimate multi-statement TVFs as 1 row. CE140+ corrects this. Fix: upgrade compatibility level to 130+, or rewrite the TVF as an inline TVF, or add OPTION (RECOMPILE) to get a fresh estimate each time (not ideal).
</details>

**SQ5:** You have a query with 5 WHERE predicates on unrelated columns. CE150 estimates 10x more rows than CE70. Which estimate is more accurate and why?
<details>
If the columns are truly independent (no correlation), CE70's multiplication is more accurate. CE150's exponential backoff assumes some correlation, leading to over-estimation. Test with actual row counts. If CE70 is closer, use OPTION (USE HINT('ASSUME_MIN_SELECTIVITY_FOR_FILTER_ESTIMATES')) or force CE70 for this query.
</details>

**SQ6:** A plan shows Warning: SpillToTempDb on a Hash Match operator. The estimated rows are 10,000 but actual rows are 1,000,000. What is the root cause and the fix?
<details>
The hash join's memory grant was sized for 10,000 rows (based on estimate), but 1,000,000 rows were processed. The hash table overflowed to tempdb. Fix: (1) Update statistics so the estimate is accurate, (2) If estimate is already accurate, use OPTION (MIN_GRANT_PERCENT = 10) or increase memory for the query.
</details>

**SQ7:** You are migrating from SQL 2012 to SQL 2022. How do you safely test CE150?
<details>
(1) Enable Query Store before the upgrade. (2) Upgrade, then use Query Store to identify regressed queries. (3) For each regression, test with OPTION (USE HINT('FORCE_LEGACY_CARDINALITY_ESTIMATION')). (4) Use plan guides to fix individual queries rather than forcing CE70 database-wide. (5) For databases that absolutely cannot regress, keep compatibility level at 110 and gradually raise it for individual databases.
</details>

**SQ8:** You are analyzing an XML plan and the XPath query returns NULL for every attribute. What are the three most likely causes?
<details>
(1) Wrong namespace declaration (case mismatch in 'showplan'). (2) Plan is NULL (over 2MB, use sys.dm_exec_text_query_plan). (3) Using [1] when the plan has no matching elements (check with .exist() first).
</details>

**SQ9:** During a query tuning session, you notice the estimated plan (Ctrl+L) shows different operators than the actual plan (Ctrl+M). What is the most likely explanation?
<details>
The estimated plan is freshly compiled and may use updated statistics. The actual plan may be using a cached plan from an earlier compilation with different parameter values or older statistics. Always use the actual plan from a production execution for diagnosis. Use DBCC FREEPROCCACHE (carefully!) or RECOMPILE to force a fresh plan.
</details>

**SQ10:** Your query shows 50,000 logical reads and 2 seconds duration. After adding an index, it shows 500 logical reads and 50ms duration. Was the index the right fix? How do you verify the plan changed?
<details>
The 100x reduction in reads and 40x reduction in duration strongly suggest the index was the right fix. Verify by comparing the old and new plan XML: the old plan showed a Clustered Index Scan or Key Lookup; the new plan shows an Index Seek. Use Query Store to confirm the new plan is stable across parameter values.
</details>

-- Create a stored procedure that compares plan XML from two CE versions
CREATE OR ALTER PROCEDURE dbo.CompareCEPlans
    @query NVARCHAR(MAX)
AS
BEGIN
    SET NOCOUNT ON;
    
    DECLARE @ce70_plan XML, @ce150_plan XML;
    DECLARE @sql NVARCHAR(MAX);
    
    -- Capture CE70 plan
    SET @sql = 'SET SHOWPLAN_XML ON; ' + @query + ' OPTION (QUERYTRACEON 9481); SET SHOWPLAN_XML OFF;';
    EXEC sp_executesql @sql;
    -- (Capture XML from result set)
    
    -- Capture CE150 plan
    SET @sql = 'SET SHOWPLAN_XML ON; ' + @query + ' OPTION (USE HINT(''FORCE_DEFAULT_CARDINALITY_ESTIMATION'')); SET SHOWPLAN_XML OFF;';
    EXEC sp_executesql @sql;
    -- (Capture XML from result set)
    
    -- Compare operator costs
    WITH XMLNAMESPACES('http://schemas.microsoft.com/sqlserver/2004/07/showplan' AS p)
    SELECT 
        COALESCE(p1.physical_op, p2.physical_op) AS operator_name,
        p1.est_rows AS ce70_est_rows,
        p2.est_rows AS ce150_est_rows,
        p1.subtree_cost AS ce70_cost,
        p2.subtree_cost AS ce150_cost,
        CASE WHEN p1.subtree_cost > 0 
             THEN (p2.subtree_cost - p1.subtree_cost) / p1.subtree_cost * 100 
             ELSE NULL END AS cost_change_pct
    FROM 
        (SELECT t.c.value('@PhysicalOp','VARCHAR(100)') AS physical_op,
                t.c.value('@EstimateRows','FLOAT') AS est_rows,
                t.c.value('@EstimatedTotalSubtreeCost','FLOAT') AS subtree_cost
         FROM @ce70_plan.nodes('//p:RelOp') t(c)) p1
    FULL OUTER JOIN
        (SELECT t.c.value('@PhysicalOp','VARCHAR(100)') AS physical_op,
                t.c.value('@EstimateRows','FLOAT') AS est_rows,
                t.c.value('@EstimatedTotalSubtreeCost','FLOAT') AS subtree_cost
         FROM @ce150_plan.nodes('//p:RelOp') t(c)) p2
    ON p1.physical_op = p2.physical_op
    ORDER BY COALESCE(p2.subtree_cost, p1.subtree_cost) DESC;
END;
`

### Reference: Common Cardinality Mismatch Patterns

| Pattern | CE70 Estimate | CE120+ Estimate | Actual | Typical Fix |
|---|---|---|---|---|
| 2 independent predicates | s1 * s2 (low) | s1 * sqrt(s2) (high) | Close to CE70 | Use CE70 or ASSUME_MIN_SELECTIVITY |
| 2 correlated predicates | s1 * s2 (very low) | s1 * sqrt(s2) | Close to CE120+ | Use CE120+ or multi-column stats |
| ASCENDING key range > max | 0 | ~200 | Varies widely | Update stats FULLSCAN |
| Star join (fact + dim) | |A| * |B| / max(distinct) | Frequency vector based | Close to CE120+ | Use CE120+ |
| TVF multi-join | 1 | 1 (CE120) / actual (CE140+) | Actual TVF rows | Use CE140+ or inline TVF |
| Multi-table join (4+ tables) | Multiplicative (very low) | Exponential backoff | Varies | Multi-column stats or plan guide |

## Section 9 — Extended Case Studies

### Case Study 1: CE70 Over-Estimation on Simple Lookup

**Scenario:** A customer lookup page queries WHERE CustomerId = @Id AND IsActive = 1. Under CE150, the query uses a Hash Join between Customers and Orders (6-second response). Under CE70, it uses Nested Loops (50ms response).

**Root Cause:** The two predicates (CustomerId, IsActive) are independent, but CE150's exponential backoff assumes some correlation. CE150 estimates 5,000 rows; CE70 estimates 50 rows. Actual is 1 row.

**Resolution:** Forced CE70 for the query. Later created a filtered index WHERE IsActive = 1 to provide better statistics.

### Case Study 2: TVF Join Regression

**Scenario:** An ETL process joining a TVF with a staging table took 2 hours after migrating from SQL 2012 to SQL 2016.

**Root Cause:** CE120 (SQL 2014) still estimated TVF output as 1 row. The plan used Nested Loops with the TVF as the outer input. Under CE70, the optimizer also estimated 1 row but chose a different join order. CE140+ would fix this, but the database was at CE120.

**Resolution:** Rewrote the TVF as an inline TVF, which eliminated the 1-row estimate. Duration dropped to 4 minutes.

### Case Study 3: Memory Grant Over-Allocation

**Scenario:** A reporting query used 8GB of memory grant but only used 200MB. Concurrent queries were waiting for memory.

**Root Cause:** CE150's exponential backoff over-estimated row counts for 6 independent predicates, leading to a 40x larger memory grant than needed.

**Resolution:** Enabled memory grant feedback (CE150) which reduced the grant to 500MB after 3 executions. Also added ASSUME_MIN_SELECTIVITY_FOR_FILTER_ESTIMATES hint for immediate fix.

## Section 9 — Extended Scenarios and Verifications

### Scenario-Based Questions

**Q1:** You are monitoring a production system and notice a query that previously used Hash Join is now using Nested Loops. Response time went from 500ms to 30 seconds. What happened?

<details>
The most likely cause is a cardinality under-estimate that made Nested Loops appear cheaper. Check: (1) Are statistics up to date? (2) Did the CE version change? (3) Is parameter sniffing causing a different plan? (4) Did data distribution change? Fix: Update statistics, consider plan forcing in Query Store, or add a join hint (INNER LOOP JOIN or INNER HASH JOIN) to stabilize the plan.
</details>

**Q2:** You execute a query with SET STATISTICS XML ON and the resulting XML shows RunTimeInformation but all ActualRows are 0. The query returned 1M rows. What happened?

<details>
This is a known issue with very large XML plans where per-thread statistics are truncated. Check if the XML is complete (might be cut off). Try using SET STATISTICS PROFILE ON instead, or use sys.dm_exec_query_stats for aggregated runtime statistics. For parallel plans, ActualRows may be aggregated only at the root operator.
</details>

**Q3:** The same query, same database, same compatibility level, but on Server A (SQL 2019) it uses Nested Loops and on Server B (SQL 2022) it uses Hash Match. Both servers have the same data and indexes. What could be different?

<details>
Even with the same compatibility level, different builds have different cumulative updates that may fix or change CE behavior. Check: (1) Exact build numbers. (2) Trace flags active on each server (DBCC TRACESTATUS). (3) Database scoped configurations. (4) Query Store settings (plan forcing). (5) Resource Governor settings. (6) MAXDOP and cost threshold for parallelism settings. The CE version in the plan XML should tell you if theyre truly the same.
</details>

CREATE TABLE dbo.CardinalityBaseline (
    BaselineDate DATETIME2,
    QueryHash BINARY(8),
    MaxMismatchRatio DECIMAL(10,2),

-- Capture baseline
INSERT INTO dbo.CardinalityBaseline

        THEN 1.0 * qs.total_rows / NULLIF(qs.total_estimated_rows, 1)

    MAX(qs.plan_handle)

-- Alert on new mismatches

        ELSE 1.0 * qs.total_estimated_rows / NULLIF(qs.total_rows, 1) END) AS current_ratio

WHERE qs.query_hash NOT IN (SELECT QueryHash FROM dbo.CardinalityBaseline WHERE BaselineDate > DATEADD(HOUR, -24, GETDATE()))

HAVING MAX(CASE WHEN qs.total_rows > qs.total_estimated_rows
    THEN 1.0 * qs.total_rows / NULLIF(qs.total_estimated_rows, 1)

CREATE PROCEDURE dbo.ValidateCEParity
    @query NVARCHAR(MAX),
    @expected_rows INT = NULL OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    CREATE TABLE #CE70 (result SQL_VARIANT);
    CREATE TABLE #CE150 (result SQL_VARIANT);
    
    DECLARE @sql70 NVARCHAR(MAX) = 'INSERT INTO #CE70 ' + @query + ' OPTION (QUERYTRACEON 9481)';
    DECLARE @sql150 NVARCHAR(MAX) = 'INSERT INTO #CE150 ' + @query + ' OPTION (USE HINT(''FORCE_DEFAULT_CARDINALITY_ESTIMATION''))';
    
    EXEC sp_executesql @sql70;
    EXEC sp_executesql @sql150;
    
    -- Compare results
    IF EXISTS (SELECT * FROM #CE70 EXCEPT SELECT * FROM #CE150)
       OR EXISTS (SELECT * FROM #CE150 EXCEPT SELECT * FROM #CE70)
    BEGIN
        RAISERROR('CE70 and CE150 produce different results!', 16, 1);
        RETURN;
    END
    
    SELECT @expected_rows = COUNT(*) FROM #CE70;
    
    DROP TABLE #CE70;
    DROP TABLE #CE150;
END;
