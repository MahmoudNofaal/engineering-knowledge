### Where This Fits

NTH_VALUE(col, N) OVER(...) returns the value of a column from the Nth row within the window frame. Every .NET backend engineer encounters NTH_VALUE when computing second-purchase behavior ("what did the customer buy on their second order?"), cohort comparisons ("compare first vs third order value"), or finding specific positional events in a workflow. The most expensive mistake made here is the same as LAST_VALUE: the default frame (RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) does not include future rows, so NTH_VALUE(col, 3) returns NULL for rows where the partition has not yet reached 3 rows within the current frame. The frame must explicitly extend to UNBOUNDED FOLLOWING to see the Nth row from the full partition. An additional SQL Server limitation: NTH_VALUE supports FROM FIRST but NOT FROM LAST — you cannot get the Nth value from the end of the partition in a single call. Interviewers use NTH_VALUE to test whether candidates understand how frame boundaries interact with positional access, and whether they know the FROM FIRST/FROM LAST difference between SQL Server and PostgreSQL.

## Core Mental Model

NTH_VALUE(col, N) returns the value of a column from the Nth row within the window frame, where N is a positive integer starting from 1. The "first" row (N=1) is determined by the ORDER BY clause — it is the row with the lowest ORDER BY value within the frame boundaries. The critical detail: NTH_VALUE is frame-relative. The frame determines which rows are visible; the Nth row is the Nth row within those frame boundaries, not the Nth row of the entire partition. With the default frame (RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW), the frame ends at the current row. For NTH_VALUE(col, 3), if the current row is the 2nd row in the partition, the frame only contains 2 rows, so NTH_VALUE returns NULL (there is no 3rd row in the frame). To access the 3rd row of the entire partition from any row position, the frame must end at UNBOUNDED FOLLOWING. NTH_VALUE differs from FIRST_VALUE and LAST_VALUE because it accesses an arbitrary position, not just the boundaries. The execution plan is similar to FIRST_VALUE/LAST_VALUE: Sort + Window Spool + Window Aggregate, with additional logic to count rows to the Nth position.

### Classification

NTH_VALUE is a **frame-based positional window function** in the analytic function family. It belongs to the set of value functions (FIRST_VALUE, LAST_VALUE, NTH_VALUE). NTH_VALUE(col, N) with N=1 is equivalent to FIRST_VALUE. SQL Server supports FROM FIRST (the default) but NOT FROM LAST. This means NTH_VALUE(col, 2) gives the 2nd row from the start, but there is no built-in way to get the 2nd row from the end in a single NTH_VALUE call. The function is not SARGable — it operates on ordered in-memory rows. The Window Aggregate operator must track row positions within the frame to identify the Nth row.

## Deep Mechanics

### How the Engine Executes This

1. **Row retrieval** — The storage engine reads rows from FROM tables, applies WHERE, and performs joins.

2. **Partition segmentation** — The Segment operator divides the input into partition groups. Each partition boundary resets the position counter.

3. **Sort** — The Sort operator orders rows by the ORDER BY clause within each partition. This defines the row order that determines which row is position 1, 2, 3, etc.

4. **Window Spool (materialization)** — The sorted rows are written to a spool. For NTH_VALUE with UNBOUNDED FOLLOWING, the spool must be fully materialized before the function can compute because the Nth row may be at a position beyond the current row.

5. **Window Aggregate — NTH_VALUE resolution:**

   - **With default frame (RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW):** The frame ends at the current row. The engine counts rows within the frame. If the frame contains at least N rows, NTH_VALUE reads the value from the Nth row in the frame (which is the Nth row from the start of the partition). If the frame has fewer than N rows, NTH_VALUE returns NULL.

   - **With UNBOUNDED FOLLOWING (ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING):** The frame covers the entire partition. The engine reads the value from the Nth row (1-indexed) of the spool. For all rows in the same partition, this is the same value.

   - **Position tracking:** The engine maintains a row counter. For each partition, row index 0 is the first row (ORDER BY ASC), row index 1 is the second row, etc. NTH_VALUE(col, N) reads spool[N-1] within the frame boundaries.

6. **SQL Server limitation — FROM LAST not supported:** Unlike PostgreSQL, SQL Server does not support NTH_VALUE(col, N FROM LAST). To get the Nth value from the end, you must use FIRST_VALUE with descending ORDER BY (e.g., for 2nd from last: FIRST_VALUE(col) OVER (ORDER BY col DESC) with N=2).

**Detailed algorithm (UNBOUNDED FOLLOWING — correct):**

```
For each partition:
  1. Sort rows by ORDER BY cols
  2. Materialize sorted rows to spool (length = partition_size)
  3. If N > partition_size:
       for all rows: output(NULL)  -- not enough rows
  4. Else:
       nth_value = spool[N-1].col
       for i = 0 to (partition_size - 1):
           output(current_row, nth_value)
```

### Execution Plan Analysis

For `SELECT CustomerId, OrderDate, NTH_VALUE(TotalAmount, 3) OVER(PARTITION BY CustomerId ORDER BY OrderDate ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING) FROM Orders`:

```
Expected plan shape:
[Clustered Index Scan / Index Seek]
→ [Segment (partition by CustomerId)]
→ [Sort (by OrderDate within each partition)]
→ [Window Spool (materialize sorted rows)]
→ [Window Aggregate (compute NTH_VALUE at position 3)]
→ [SELECT]

Estimated Cost: Scan 10%, Segment 2%, Sort 55%, Spool 18%, Aggregate 15%
Logical Reads: ~128,342 (scan) + 0 (spool in memory)
```

- The Window Aggregate for NTH_VALUE(3) reads position 2 (0-indexed) from the spool. This is a direct positional read — the same cost as FIRST_VALUE (position 0) or LAST_VALUE (position N-1).
- With the default frame, the Window Aggregate must check that the frame contains at least N rows before reading. This adds a comparison per row. If the frame has fewer rows, NULL is returned.
- With UNBOUNDED FOLLOWING, there is no frame-size check needed (the frame always contains all rows), so the positional read is unconditional once the spool is built.
- NTH_VALUE with N=1 is optimized identically to FIRST_VALUE — the engine may recognize the equivalence and use the same code path.
- NTH_VALUE with a large N (e.g., N=1000) still costs the same as N=1 in the spool read (positional access is O(1)), but the spool must contain at least N rows — requiring the partition to be large enough or returning NULL.

### Failure Modes

**Failure Mode 1: Default frame causes NULL for early rows.**

```sql
-- ❌ WRONG: Default frame — returns NULL for rows 1 and 2
SELECT
    o.CustomerId,
    o.OrderId,
    NTH_VALUE(o.TotalAmount, 3) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        -- Default: RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS ThirdAmount
FROM dbo.Orders o;

-- ✅ Correct: UNBOUNDED FOLLOWING — same value for all rows
SELECT
    o.CustomerId,
    o.OrderId,
    NTH_VALUE(o.TotalAmount, 3) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS ThirdAmount
FROM dbo.Orders o;
```

**Symptom:** NTH_VALUE returns NULL for the first N-1 rows of each partition. Engineers test with customers that have many orders and don't notice the NULLs.

**Fix:** Specify ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING to ensure the frame includes all rows.

**Cost of not fixing:** NULL values in analytics. Second-purchase analysis fails for customers on their first order (but appears to work for repeat customers).

**Failure Mode 2: Assuming NTH_VALUE works like LAG with offset N.**

```sql
-- NTH_VALUE(3) gives the 3rd row FROM THE START, not 3 rows ago
SELECT
    o.CustomerId,
    o.OrderId,
    o.OrderDate,
    o.TotalAmount,
    NTH_VALUE(o.TotalAmount, 3) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS ThirdOrder,   -- Same for ALL rows — the absolute 3rd order
    LAG(o.TotalAmount, 2) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
    ) AS TwoOrdersAgo  -- Changes per row — relative offset
FROM dbo.Orders o
ORDER BY o.CustomerId, o.OrderDate;
```

**Symptom:** Engineers expect NTH_VALUE to return a different value per row (like LAG), but it returns the same value for all rows (with UNBOUNDED FOLLOWING).

**Fix:** Use LAG for relative offsets. Use NTH_VALUE only for absolute positional queries.

**Cost of not fixing:** Wrong analytics. "Second order value" computed as the same value for all rows of a customer.

**Failure Mode 3: FROM LAST not supported in SQL Server.**

```sql
-- ❌ Not valid in SQL Server:
-- NTH_VALUE(o.TotalAmount, 2 FROM LAST) OVER (...)

-- ✅ SQL Server alternative: FIRST_VALUE with descending ORDER BY
SELECT
    o.CustomerId,
    o.OrderId,
    FIRST_VALUE(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate DESC, o.OrderId DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastValue,
    -- For 2nd from last, use ROW_NUMBER:
    MAX(CASE WHEN rn_desc = 2 THEN TotalAmount END) OVER (PARTITION BY CustomerId) AS SecondToLast
FROM (
    SELECT *,
        ROW_NUMBER() OVER (PARTITION BY CustomerId ORDER BY OrderDate DESC, OrderId DESC) AS rn_desc
    FROM dbo.Orders
) o
ORDER BY o.CustomerId, o.OrderDate;
```

**Symptom:** SQL compilation error when attempting FROM LAST syntax.

**Fix:** Use FIRST_VALUE with descending ORDER BY for the last value, or ROW_NUMBER for the Nth-from-last.

**Cost of not fixing:** Query compilation error. Attempting to use PostgreSQL-compatible syntax that fails on SQL Server.

**Failure Mode 4: NTH_VALUE with N larger than partition size.**

```sql
-- N is larger than any partition
SELECT
    o.CustomerId,
    NTH_VALUE(o.TotalAmount, 999999) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS NthAmount
FROM dbo.Orders o;
-- Returns NULL for all rows — no partition has 999,999 rows
```

**Symptom:** All rows return NULL. No error message. The query runs successfully but produces no useful data.

**Fix:** Ensure N is less than or equal to the smallest partition size. Use COALESCE to provide a default:

```sql
SELECT
    o.CustomerId,
    COALESCE(NTH_VALUE(o.TotalAmount, 3) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ), 0) AS ThirdAmount
FROM dbo.Orders o;
```

**Cost of not fixing:** Silent NULL propagation in calculations.

**Failure Mode 5: Non-unique ORDER BY makes Nth row non-deterministic.**

```sql
-- Multiple orders on same date — which is "2nd"?
SELECT
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    NTH_VALUE(o.TotalAmount, 2) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS SecondAmount
FROM dbo.Orders o;
```

**Symptom:** The 2nd order value is non-deterministic when multiple orders share the same OrderDate.

**Fix:** Add a unique tiebreaker to ORDER BY:

```sql
NTH_VALUE(o.TotalAmount, 2) OVER (
    PARTITION BY o.CustomerId
    ORDER BY o.OrderDate, o.OrderId  -- OrderId breaks ties
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
)
```

**Cost of not fixing:** Non-reproducible results across query executions.

```sql
-- Pattern 1: First, second, and third order comparison for cohorts
WITH CustomerCohort AS (
    SELECT
        o.CustomerId,
        o.OrderId,
        o.OrderDate,
        o.TotalAmount,
        FIRST_VALUE(o.TotalAmount) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS FirstOrderAmount,
        NTH_VALUE(o.TotalAmount, 2) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS SecondOrderAmount,
        NTH_VALUE(o.TotalAmount, 3) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS ThirdOrderAmount,
        ROW_NUMBER() OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
        ) AS OrderSeq,
        COUNT(*) OVER (PARTITION BY o.CustomerId) AS CustomerOrderCount
    FROM dbo.Orders o
)
SELECT
    CustomerId,
    OrderSeq,
    OrderId,
    OrderDate,
    TotalAmount,
    FirstOrderAmount,
    SecondOrderAmount,
    ThirdOrderAmount,
    CASE
        WHEN OrderSeq = 1 THEN 'First'
        WHEN OrderSeq = 2 THEN 'Second'
        WHEN OrderSeq = 3 THEN 'Third'
        ELSE 'Post-Third'
    END AS OrderTier,
    CASE
        WHEN CustomerOrderCount >= 3 THEN 'Has 3+ Orders'
        WHEN CustomerOrderCount = 2 THEN 'Churned After 2'
        ELSE 'Churned After 1'
    END AS CohortSegment
FROM CustomerCohort
ORDER BY CustomerId, OrderSeq;

-- Pattern 2: Nth order product category — what did customer buy on 2nd order
WITH SecondPurchaseCategory AS (
    SELECT
        c.CustomerId,
        c.FirstName + ' ' + c.LastName AS CustomerName,
        o.OrderId,
        o.OrderDate,
        oi.ProductId,
        p.CategoryId,
        ROW_NUMBER() OVER (
            PARTITION BY c.CustomerId
            ORDER BY o.OrderDate, o.OrderId
        ) AS OrderSeq
    FROM dbo.Customers c
    INNER JOIN dbo.Orders o ON c.CustomerId = o.CustomerId
    INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
    INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
)
SELECT
    CustomerId,
    CustomerName,
    OrderId,
    OrderDate,
    ProductId,
    CategoryId
FROM SecondPurchaseCategory
WHERE OrderSeq = 2
ORDER BY CustomerId;

-- Pattern 3: Nth salary for employee (salary progression tracking)
WITH SalaryProgression AS (
    SELECT
        e.EmployeeId,
        e.FirstName + ' ' + e.LastName AS EmployeeName,
        e.HireDate,
        e.Salary,
        ROW_NUMBER() OVER (
            PARTITION BY e.EmployeeId
            ORDER BY e.HireDate
        ) AS SalarySeq,
        NTH_VALUE(e.Salary, 1) OVER (
            PARTITION BY e.EmployeeId
            ORDER BY e.HireDate
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS StartingSalary,
        NTH_VALUE(e.Salary, 2) OVER (
            PARTITION BY e.EmployeeId
            ORDER BY e.HireDate
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS SecondSalary,
        NTH_VALUE(e.Salary, 3) OVER (
            PARTITION BY e.EmployeeId
            ORDER BY e.HireDate
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS ThirdSalary
    FROM dbo.Employees e
)
SELECT
    EmployeeId,
    EmployeeName,
    SalarySeq,
    HireDate,
    Salary,
    StartingSalary,
    SecondSalary,
    ThirdSalary,
    CASE
        WHEN SalarySeq = 1 THEN 'Initial'
        WHEN SalarySeq = 2 THEN 'First Raise'
        WHEN SalarySeq = 3 THEN 'Second Raise'
        ELSE 'Later'
    END AS SalaryEvent
FROM SalaryProgression
ORDER BY EmployeeId, SalarySeq;

-- Pattern 4: NTH_VALUE for customer loyalty tiers
-- Find first, second, third, and most recent order per customer
WITH OrderMetrics AS (
    SELECT
        o.CustomerId,
        o.OrderId,
        o.OrderDate,
        o.TotalAmount,
        NTH_VALUE(o.TotalAmount, 1) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS FirstAmount,
        NTH_VALUE(o.TotalAmount, 2) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS SecondAmount,
        NTH_VALUE(o.TotalAmount, 3) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS ThirdAmount,
        FIRST_VALUE(o.TotalAmount) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate DESC, o.OrderId DESC
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS MostRecentAmount
    FROM dbo.Orders o
)
SELECT DISTINCT
    CustomerId,
    FirstAmount,
    SecondAmount,
    ThirdAmount,
    MostRecentAmount,
    CASE
        WHEN SecondAmount IS NULL THEN 'One-Time Buyer'
        WHEN ThirdAmount IS NULL THEN 'Repeat Buyer (2 orders)'
        WHEN MostRecentAmount > FirstAmount THEN 'Growing Spender'
        ELSE 'Established Buyer'
    END AS BuyerSegment
FROM OrderMetrics
ORDER BY CustomerId;

-- Pattern 5: NTH_VALUE with dynamic N (using CASE or parameter)
-- Note: N must be a literal, not a column reference
DECLARE @N INT = 3;
SELECT
    o.CustomerId,
    o.OrderId,
    o.TotalAmount,
    -- N must be literal — cannot use variable directly
    CASE @N
        WHEN 1 THEN NTH_VALUE(o.TotalAmount, 1) OVER (
            PARTITION BY o.CustomerId ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        )
        WHEN 2 THEN NTH_VALUE(o.TotalAmount, 2) OVER (
            PARTITION BY o.CustomerId ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        )
        WHEN 3 THEN NTH_VALUE(o.TotalAmount, 3) OVER (
            PARTITION BY o.CustomerId ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        )
        ELSE NULL
    END AS NthAmount
FROM dbo.Orders o
ORDER BY o.CustomerId, o.OrderDate;

-- Pattern 6: Cohort retention — first purchase vs second purchase time gap
WITH CohortAnalysis AS (
    SELECT
        o.CustomerId,
        o.OrderId,
        o.OrderDate,
        o.TotalAmount,
        FIRST_VALUE(o.OrderDate) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS FirstOrderDate,
        NTH_VALUE(o.OrderDate, 2) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        ) AS SecondOrderDate
    FROM dbo.Orders o
)
SELECT DISTINCT
    CustomerId,
    FirstOrderDate,
    SecondOrderDate,
    DATEDIFF(day, FirstOrderDate, SecondOrderDate) AS DaysToSecondOrder,
    CASE
        WHEN SecondOrderDate IS NULL THEN 'No_Return'
        WHEN DATEDIFF(day, FirstOrderDate, SecondOrderDate) <= 7 THEN 'Within_Week'
        WHEN DATEDIFF(day, FirstOrderDate, SecondOrderDate) <= 30 THEN 'Within_Month'
        WHEN DATEDIFF(day, FirstOrderDate, SecondOrderDate) <= 90 THEN 'Within_Quarter'
        ELSE 'Beyond_Quarter'
    END AS ReturnTiming
FROM CohortAnalysis
ORDER BY CustomerId;
```

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL supports FROM LAST (SQL Server does not)
-- PostgreSQL:
SELECT
    o.customer_id,
    o.order_date,
    NTH_VALUE(o.total_amount, 2 FROM LAST) OVER (
        PARTITION BY o.customer_id
        ORDER BY o.order_date
    ) AS second_to_last_amount
FROM orders o;

-- SQL Server alternative (no FROM LAST):
SELECT
    o.customer_id,
    o.order_date,
    FIRST_VALUE(o.total_amount) OVER (
        PARTITION BY o.customer_id
        ORDER BY o.order_date DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS last_amount,
    -- For 2nd to last, use ROW_NUMBER approach
    MAX(CASE WHEN rn_desc = 2 THEN total_amount END) OVER (PARTITION BY customer_id) AS second_to_last
FROM (
    SELECT *,
        ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC, order_id DESC) AS rn_desc
    FROM orders
) o;

-- PostgreSQL default frame is the same (RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
-- Same trap applies: always specify ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
```

## Gotchas and Production Pitfalls

### 1. Default Frame Causes NULL for Early Rows

**Pitfall:** Using NTH_VALUE with the default frame. For the first N-1 rows of each partition, the frame does not contain N rows, so NTH_VALUE returns NULL.

```sql
-- ❌ Wrong: Default frame — rows 1-2 return NULL for NTH_VALUE(3)
SELECT
    o.CustomerId,
    o.OrderId,
    NTH_VALUE(o.TotalAmount, 3) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
    ) AS ThirdAmount
FROM dbo.Orders o;

-- ✅ Correct: UNBOUNDED FOLLOWING
SELECT
    o.CustomerId,
    o.OrderId,
    NTH_VALUE(o.TotalAmount, 3) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS ThirdAmount
FROM dbo.Orders o;
```

**Symptom:** For customers with 3+ orders, rows 1 and 2 show NULL for the 3rd order value. For customers with fewer than 3 orders, all rows show NULL.

**Fix:** Specify ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING.

**Cost of not fixing:** Silent NULL propagation. Analytics that compute averages or deltas with NTH_VALUE produce incorrect results.

### 2. FROM LAST Not Supported in SQL Server

**Pitfall:** Trying to use PostgreSQL's NTH_VALUE(col, N FROM LAST) syntax in SQL Server. SQL Server does not support the FROM LAST clause.

```sql
-- ❌ SQL Server: Syntax error
NTH_VALUE(o.TotalAmount, 2 FROM LAST) OVER (...)  -- Not valid

-- ✅ SQL Server: Use FIRST_VALUE with DESC order or ROW_NUMBER
FIRST_VALUE(o.TotalAmount) OVER (
    ORDER BY o.OrderDate DESC, o.OrderId DESC
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
) AS LastAmount
```

**Symptom:** SQL compilation error. Attempting to run PostgreSQL-compatible SQL on SQL Server.

**Fix:** Use FIRST_VALUE with descending ORDER BY for the last value, or ROW_NUMBER for Nth-from-last.

**Cost of not fixing:** Query compilation failure. Portability issues when migrating between databases.

### 3. Non-Unique ORDER BY Makes Nth Position Non-Deterministic

**Pitfall:** Using NTH_VALUE with non-unique ORDER BY. Ties in the ordering make the Nth row ambiguous.

```sql
-- Non-deterministic: Which order is "2nd" when two orders have the same date?
SELECT
    o.CustomerId,
    o.OrderDate,
    NTH_VALUE(o.TotalAmount, 2) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS SecondAmount
FROM dbo.Orders o;
```

**Symptom:** The 2nd order value changes across executions for customers with tied order dates.

**Fix:** Add a unique tiebreaker to ORDER BY:

```sql
NTH_VALUE(o.TotalAmount, 2) OVER (
    PARTITION BY o.CustomerId
    ORDER BY o.OrderDate, o.OrderId
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
)
```

**Cost of not fixing:** Non-reproducible cohort analysis. Second-purchase values change unpredictably.

### 4. NTH_VALUE with N Larger Than Partition Size Returns NULL

**Pitfall:** Using a fixed N without accounting for partitions that may be smaller than N.

```sql
-- Returns NULL for customers with < 3 orders
SELECT
    o.CustomerId,
    NTH_VALUE(o.TotalAmount, 3) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS ThirdAmount
FROM dbo.Orders o;
```

**Symptom:** Customers with 1-2 orders get NULL for the 3rd order value. No error, just NULLs.

**Fix:** Use COALESCE for a default, or filter partitions that are large enough:

```sql
SELECT
    o.CustomerId,
    COALESCE(NTH_VALUE(o.TotalAmount, 3) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ), 0) AS ThirdAmount
FROM dbo.Orders o
WHERE o.CustomerId IN (
    SELECT CustomerId FROM dbo.Orders GROUP BY CustomerId HAVING COUNT(*) >= 3
);
```

**Cost of not fixing:** Silent NULLs in computed columns. Averages that include NULLs produce unexpected results.

### 5. NTH_VALUE with Variable N (Cannot Parameterize)

**Pitfall:** NTH_VALUE requires a literal integer for N. You cannot use a variable, parameter, or column reference.

```sql
-- ❌ Wrong: Cannot use variable for N
DECLARE @N INT = 3;
SELECT
    o.OrderId,
    NTH_VALUE(o.TotalAmount, @N) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS NthAmount
FROM dbo.Orders o;
-- Error: argument 2 of NTH_VALUE must be a constant

-- ✅ Alternative: Dynamic SQL
DECLARE @Sql NVARCHAR(MAX) = '
SELECT
    o.OrderId,
    NTH_VALUE(o.TotalAmount, ' + CAST(@N AS NVARCHAR) + ') OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS NthAmount
FROM dbo.Orders o';
EXEC sp_executesql @Sql;
```

**Symptom:** SQL compilation error: "argument 2 of NTH_VALUE must be a constant."

**Fix:** Use dynamic SQL to build the query with the literal N, or use a CASE expression with multiple NTH_VALUE calls.

**Cost of not fixing:** Cannot make N dynamic. Must hard-code N for each query.

### 6. Assuming NTH_VALUE Returns Same Value as LAG(N-1) with Correct Frame

**Pitfall:** Engineers think NTH_VALUE(3) returns the value from 3 rows ago (like LAG(2)), but NTH_VALUE gives the absolute 3rd row from the start of the partition.

```sql
-- NTH_VALUE(3) = absolute 3rd row (same for all rows)
-- LAG(2) = 2 rows before current (changes per row)
SELECT
    o.OrderId,
    o.OrderDate,
    NTH_VALUE(o.TotalAmount, 3) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS AbsoluteThird,  -- Same for all rows: value from the 3rd order
    LAG(o.TotalAmount, 2) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
    ) AS TwoBeforeCurrent  -- Changes per row
FROM dbo.Orders o;
```

**Symptom:** The engineer expects NTH_VALUE to change per row, but it returns a constant value for the entire partition.

**Fix:** Use LAG for relative offsets. Use NTH_VALUE only for absolute position queries.

**Cost of not fixing:** Wrong analytics. Misunderstood function behavior leads to incorrect business decisions.

### Spoken Answers

**Q1: What is NTH_VALUE and how is it different from FIRST_VALUE and LAST_VALUE?**

> **Average answer:** "NTH_VALUE returns the Nth row in a partition. FIRST_VALUE returns the first row, LAST_VALUE returns the last. NTH_VALUE is more flexible."

> **Great answer:** "NTH_VALUE(col, N) is a frame-based positional window function that returns the value of a column from the Nth row within the window frame, where N is a 1-based literal integer. The fundamental difference from FIRST_VALUE and LAST_VALUE is that NTH_VALUE accesses an arbitrary position, not just the boundaries. With N=1, it is equivalent to FIRST_VALUE. However, there is no SQL Server equivalent for 'Nth from last' — FROM LAST is PostgreSQL-only. The critical implementation detail shared with LAST_VALUE is the default frame trap: NTH_VALUE's default frame ends at CURRENT ROW, so for N=3, rows 1 and 2 see a frame with fewer than 3 rows and get NULL. The fix is always: ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING. Unlike LAG and LEAD which are relative to the current row, NTH_VALUE with UNBOUNDED FOLLOWING returns the same absolute position for all rows in the partition. This makes it ideal for cohort analysis — 'compare every order to the 2nd order value' — but wrong for row-by-row offset comparisons. The performance is identical to FIRST_VALUE: one sort plus one spool read, because the spool supports O(1) positional access regardless of N."

**Q5: Compare NTH_VALUE and LAG.**

> **Average answer:** "NTH_VALUE gets the Nth row, LAG gets the previous row. They're for different purposes."

> **Great answer:** "NTH_VALUE and LAG serve fundamentally different positioning models. NTH_VALUE is absolute position: it returns the value from position N from the start of the partition, and with UNBOUNDED FOLLOWING, that value is the same for every row. LAG is relative position: it returns the value from N rows before the current row, and the value changes for each row. For example, NTH_VALUE(TotalAmount, 3) with UNBOUNDED FOLLOWING gives the 3rd order's amount for every row of a customer. LAG(TotalAmount, 2) gives the amount from 2 orders ago, which is different for the 3rd order (1st order's amount), the 4th order (2nd order's amount), etc. The execution plan difference: NTH_VALUE reads position N-1 from the Window Spool and returns it for all rows. LAG reads position (current - offset) from the spool, which changes per row. Both use the same spool operator, but LAG's per-row position calculation is slightly more CPU-intensive. In practice, the cost difference is negligible — the sort dominates both. The choice is purely semantic: use NTH_VALUE for 'compare to the Nth event' and LAG for 'compare to N events ago.'"

**Q8: How do you handle NTH_VALUE with dynamic N in .NET?**

> **Average answer:** "You can't use a parameter. You build dynamic SQL."

> **Great answer:** "NTH_VALUE requires a literal constant for the N parameter. You cannot pass a SQL parameter. In EF Core and Dapper, you have two options. First, dynamic SQL: build the query string with the N value interpolated. This means the query plan changes with each N value, but that's usually acceptable. In Dapper: var sql = $"SELECT NTH_VALUE(col, {n}) OVER (...)" — but use SqlCommand with parameterized ORDER BY to protect against injection. Second, use a CASE expression with multiple NTH_VALUE calls for a limited set of N values. For example, handle N=1 through N=5 with separate NTH_VALUE calls in CASE branches. This gives plan stability at the cost of complexity. For EF Core, use SqlQueryRaw with the N value interpolated into the string. Always validate N is a positive integer before building the query. In code review, the key concern with dynamic N is SQL injection if the query is built from user input — but since N must be an integer literal, the risk is limited."

### Application Checklist

- [ ] The problem requires the value from the Nth row in absolute position (not relative)
- [ ] The frame is explicitly specified as ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
- [ ] N is a literal constant (not a variable/parameter) — or dynamic SQL is planned
- [ ] ORDER BY includes a unique tiebreaker for deterministic position
- [ ] N is less than or equal to the minimum partition size (or NULL handling is in place)

**Challenge 3 — Explain the execution plan**

```sql
-- Query A:
SELECT
    CustomerId,
    OrderId,
    NTH_VALUE(TotalAmount, 3) OVER (
        PARTITION BY CustomerId
        ORDER BY OrderDate, OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS ThirdAmount
FROM dbo.Orders;

-- Query B:
SELECT
    CustomerId,
    OrderId,
    NTH_VALUE(TotalAmount, 3) OVER (
        PARTITION BY CustomerId
        ORDER BY OrderDate, OrderId
    ) AS ThirdAmount
FROM dbo.Orders;
```

Both queries have similar execution plans and similar cost. Why does Query B return different (wrong) results for early rows?

<details>
<summary>Solution</summary>

**Why plans are similar:** Both queries require the same operators: Clustered Index Scan, Segment, Sort (based on ORDER BY), Window Spool, and Window Aggregate. The Window Aggregate's internal logic differs:

- **Query A (UNBOUNDED FOLLOWING):** The frame covers the entire partition. The aggregate reads from spool position 2 (0-indexed for N=3) and returns that value for every row in the partition. No frame-size check needed.

- **Query B (default frame):** The frame ends at CURRENT ROW. The aggregate checks: "does the current frame contain at least 3 rows?" For row 1 (frame has 1 row), returns NULL. For row 2 (frame has 2 rows), returns NULL. For row 3+ (frame has 3+ rows), returns spool position 2.

The cost difference is negligible because the Sort and Spool dominate. The frame check in Query B (comparing row position to N) is CPU-trivial.

**Why this is dangerous:** Query B runs quickly and returns data — it just returns NULLs for the first 2 rows of each partition. Engineers who only test on large partitions (where most rows have 3+ rows) miss the NULLs. The first two rows per customer are the most important — they represent the first and second purchases.

**To get a different plan for Query B:** Add an explicit frame. The plan becomes identical to Query A's plan, but the results are correct.

**Tradeoff:** Default frame is marginally faster but semantically wrong for absolute-position NTH_VALUE. Always specify UNBOUNDED FOLLOWING.

</details>

```sql
    NTH_VALUE(o.TotalAmount, 3) OVER (PARTITION BY o.CustomerId ORDER BY o.OrderDate, o.OrderId ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING) AS ThirdAmount
FROM dbo.Orders o;
```

Current execution time is 35 seconds. Read/write ratio is 80/20.

<details>
<summary>Solution</summary>

```sql
-- Covering index to eliminate Sort and make the query index-only
CREATE INDEX IX_Orders_CustomerId_OrderDate_OrderId
ON dbo.Orders(CustomerId, OrderDate, OrderId)
INCLUDE (TotalAmount);
```

**Why this index:**
- CustomerId is the PARTITION BY column
- OrderDate + OrderId provides the ordered sequence (OrderId added as tiebreaker)
- TotalAmount in INCLUDE makes the index covering for the three NTH_VALUE calls
- All three NTH_VALUE calls share the same sort order — one index serves all three

**Without this index:**
- Clustered index scan + Sort: 100M rows
- Memory grant: ~6-8GB for sort
- Execution time: 35 seconds

**With this index:**
- Index Scan (order-preserving) — eliminates Sort
- Window Spool: minimal (rows pre-sorted)
- Estimated: 3-5 seconds

**Tradeoffs:**
- Size: CustomerId (4) + OrderDate (8) + OrderId (4) = 16 bytes key × 100M = 1.6GB + INCLUDE TotalAmount (9) = 0.9GB + overhead ≈ 3-4GB total
- Write overhead: Each INSERT/UPDATE must maintain this index (~20% overhead for writes)
- The index also accelerates any query filtering by CustomerId with order by OrderDate

**What NOT to index:**
- Do not include other columns (Status, ShipDate, etc.) — they are not used in this query
- Do not create separate indexes for each NTH_VALUE — they all use the same sort order
- A filtered index is not appropriate because the query scans all customers

**After index — logical reads:** ~30,000-50,000 (from 128,342)
**Execution plan:** Index Scan (order-preserving) → Segment → Window Spool → Window Aggregate (3 NTH_VALUE calls share spool) → SELECT
**Sort eliminated:** Yes
