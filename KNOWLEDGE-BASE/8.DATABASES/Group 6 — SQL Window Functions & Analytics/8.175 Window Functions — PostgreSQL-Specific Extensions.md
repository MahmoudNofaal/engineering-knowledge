### Where This Fits

PostgreSQL provides several window function extensions beyond the ANSI standard and beyond what SQL Server offers: the FILTER clause for conditional aggregation within windows, RANGE with intervals for temporal windows, GROUPS frame mode for tick-based grouping, the WINDOW clause for reusable window definitions, array_agg as a window function for ordered list aggregation, and DISTINCT within aggregate window functions. A .NET backend engineer encounters these when migrating from SQL Server to PostgreSQL (or supporting both), when building analytics on PostgreSQL, or when using PostgreSQL-specific features in a multi-database environment with EF Core or Dapper. The challenge is that these features have no SQL Server equivalents in most cases — migration requires query rewrites using subqueries, CTEs, or client-side logic. The interview signal is cross-database knowledge: knowing both what PostgreSQL can do that SQL Server cannot and how to bridge the gap with standard SQL.

## Core Mental Model

PostgreSQL extends the ANSI SQL window function specification with five major additions that SQL Server does not support (or supports incompletely). The invariant: each extension solves a specific limitation in standard window functions. FILTER allows conditional aggregation within a window without CASE expressions. RANGE with intervals enables temporal windows with DATE/TIMESTAMP boundaries (e.g., "last 7 days") without needing to calculate row counts. GROUPS frame mode groups rows by ORDER BY value equality (like RANGE) but with frame boundaries defined in tick units rather than value ranges — a hybrid between ROWS (by count) and RANGE (by value). The WINDOW clause eliminates repeated window definitions by declaring the window once and referencing it by name. array_agg as a window function collects ordered values into an array per partition, enabling list aggregation without GROUP BY. DISTINCT in aggregate window functions (PostgreSQL 15+) allows counting distinct values within a window. These features make PostgreSQL significantly more expressive for analytics queries but create migration pain when moving to SQL Server.

### Classification

- **SQL dialect extension:** PostgreSQL-specific — not available in SQL Server (or available with different syntax/semantics)
- **Optimizer behavior:** PostgreSQL translates these extensions into appropriate plan nodes; the query planner handles them natively
- **Portability:** Low — these features require significant rewrite for SQL Server
- **Interface:** EF Core (Npgsql) translates only standard window functions; LINQ does not expose FILTER, GROUPS, or WINDOW clause

## Deep Mechanics

### How PostgreSQL Executes FILTER in Window Functions

PostgreSQL's FILTER clause applies a condition before the aggregate function runs, within the window context. The aggregate computes only over rows matching the FILTER condition. The engine handles this in the WindowAgg plan node, which evaluates the FILTER predicate per row before passing the row to the aggregate accumulator. Unlike a CASE expression (which computes a value per row that the aggregate then processes), FILTER avoids computing the expression for non-matching rows entirely — a subtle performance advantage for expensive expressions.

```sql
-- PostgreSQL FILTER inside window aggregate:
SELECT
    OrderId,
    CustomerId,
    OrderDate,
    TotalAmount,
    COUNT(*) FILTER(WHERE TotalAmount > 100)
        OVER(PARTITION BY CustomerId) AS HighValueOrderCount,
    COUNT(*) FILTER(WHERE TotalAmount <= 100)
        OVER(PARTITION BY CustomerId) AS LowValueOrderCount
FROM Orders;
```

Execution plan:
```
WindowAgg (FILTER: TotalAmount > 100), (FILTER: TotalAmount <= 100)
  -> Sort (ORDER BY CustomerId)
    -> Seq Scan on Orders
```

### How PostgreSQL Executes RANGE with INTERVAL

PostgreSQL's RANGE BETWEEN INTERVAL uses the ORDER BY column's data type semantics to determine frame boundaries. For ORDER BY TIMESTAMP, each row's frame includes all rows whose ORDER BY value falls within the specified interval. The engine evaluates the frame boundaries by comparing the ORDER BY value of the current row to each qualifying row's ORDER BY value — this requires a blocking Spool (materializing the partition) because frame membership cannot be determined incrementally.

```sql
-- PostgreSQL: 7-day rolling window
SELECT
    SaleDate,
    Amount,
    SUM(Amount) OVER(
        ORDER BY SaleDate
        RANGE BETWEEN INTERVAL '7 days' PRECEDING AND CURRENT ROW
    ) AS WeeklyRunningTotal
FROM Sales;
```

Execution plan:
```
WindowAgg (RANGE BETWEEN INTERVAL '7 days' PRECEDING AND CURRENT ROW)
  -> Sort (ORDER BY SaleDate)
    -> Seq Scan on Sales
```

### How GROUPS Frame Mode Works

GROUPS is a frame mode introduced in SQL:2011 that sits between ROWS and RANGE. Like RANGE, it considers all rows with the same ORDER BY value as peers (a group/tick). Like ROWS, frame boundaries are specified in counts of groups (not value ranges). `GROUPS BETWEEN 1 PRECEDING AND 1 FOLLOWING` means: include the current row's group, one group before, and one group after. This avoids the problem with RANGE where boundaries depend on the actual data values (if values are non-uniform, the RANGE window size varies).

```sql
-- PostgreSQL: GROUPS frame — 1 group before and after
SELECT
    CategoryId,
    SaleAmount,
    AVG(SaleAmount) OVER(
        ORDER BY CategoryId
        GROUPS BETWEEN 1 PRECEDING AND 1 FOLLOWING
    ) AS SmoothedAvg
FROM Sales;
```

### SQL Visibility — All PostgreSQL Extensions

```sql
-- 1. FILTER clause
SELECT
    DepartmentId,
    EmployeeName,
    Salary,
    AVG(Salary) FILTER(WHERE Status = 'Active')
        OVER(PARTITION BY DepartmentId) AS AvgActiveSalary,
    AVG(Salary) FILTER(WHERE Status = 'Inactive')
        OVER(PARTITION BY DepartmentId) AS AvgInactiveSalary
FROM Employees;

-- SQL Server equivalent: CASE inside aggregate
SELECT
    DepartmentId,
    EmployeeName,
    Salary,
    AVG(CASE WHEN Status = 'Active' THEN Salary END)
        OVER(PARTITION BY DepartmentId) AS AvgActiveSalary,
    AVG(CASE WHEN Status = 'Inactive' THEN Salary END)
        OVER(PARTITION BY DepartmentId) AS AvgInactiveSalary
FROM Employees;

-- 2. RANGE with INTERVAL
SELECT
    MeasurementTime,
    SensorValue,
    AVG(SensorValue) OVER(
        ORDER BY MeasurementTime
        RANGE BETWEEN INTERVAL '1 hour' PRECEDING AND CURRENT ROW
    ) AS HourlyMovingAvg
FROM SensorReadings
WHERE SensorId = 101;

-- SQL Server equivalent: ROWS with calculated row count
-- (requires knowing rows per hour, which varies)
-- Alternative: subquery with BETWEEN dates
SELECT
    sr.MeasurementTime,
    sr.SensorValue,
    (SELECT AVG(sr2.SensorValue)
     FROM SensorReadings sr2
     WHERE sr2.SensorId = sr.SensorId
       AND sr2.MeasurementTime >= DATEADD(HOUR, -1, sr.MeasurementTime)
       AND sr2.MeasurementTime <= sr.MeasurementTime
    ) AS HourlyMovingAvg
FROM SensorReadings sr
WHERE sr.SensorId = 101;

-- 3. GROUPS frame mode
SELECT
    CategoryId,
    CategoryName,
    SaleAmount,
    AVG(SaleAmount) OVER(
        ORDER BY SaleAmount
        GROUPS BETWEEN 1 PRECEDING AND 1 FOLLOWING
    ) AS GroupSmoothedAvg
FROM Sales;

-- 4. WINDOW clause
SELECT
    DepartmentId,
    EmployeeName,
    Salary,
    ROW_NUMBER() OVER w AS RowNum,
    RANK() OVER w AS SalaryRank,
    SUM(Salary) OVER w AS DeptTotal
FROM Employees
WINDOW w AS (PARTITION BY DepartmentId ORDER BY Salary DESC);

-- SQL Server equivalent: no WINDOW clause, repeat OVER
SELECT
    DepartmentId,
    EmployeeName,
    Salary,
    ROW_NUMBER() OVER(PARTITION BY DepartmentId ORDER BY Salary DESC) AS RowNum,
    RANK() OVER(PARTITION BY DepartmentId ORDER BY Salary DESC) AS SalaryRank,
    SUM(Salary) OVER(PARTITION BY DepartmentId ORDER BY Salary DESC) AS DeptTotal
FROM Employees;

-- 5. array_agg as window function
SELECT
    DepartmentId,
    EmployeeName,
    Salary,
    array_agg(EmployeeName) OVER(
        PARTITION BY DepartmentId
        ORDER BY Salary DESC
    ) AS OrderedEmployees
FROM Employees;

-- SQL Server equivalent: STRING_AGG (SQL Server 2017+)
-- STRING_AGG cannot be used as a window function in SQL Server
-- Must use subquery with FOR XML PATH or STRING_AGG with GROUP BY

-- 6. DISTINCT in window aggregate (PostgreSQL 15+)
SELECT
    CustomerId,
    OrderDate,
    ProductId,
    COUNT(DISTINCT ProductId) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate
        ROWS UNBOUNDED PRECEDING
    ) AS DistinctProductsCumulative
FROM OrderItems;
```

### Execution Plan Analysis — PostgreSQL

PostgreSQL's execution plans differ from SQL Server. Key differences:

- **WindowAgg** node is the PostgreSQL equivalent of SQL Server's Sequence Project + Segment
- **Sort** appears when no matching index exists — same as SQL Server
- **GroupAggregate** + **WindowAgg** for window functions with GROUP BY
- PostgreSQL EXPLAIN ANALYZE shows actual row counts and timing

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT DepartmentId, EmployeeName, Salary,
    AVG(Salary) FILTER(WHERE Status = 'Active')
        OVER(PARTITION BY DepartmentId) AS AvgActiveSalary
FROM Employees;

-- Output:
-- WindowAgg (cost=1234.56..1500.00 rows=10000 width=100)
--   Filter: (Status = 'Active')  -- FILTER evaluated here
--   -> Sort (cost=1234.56..1300.00 rows=10000 width=60)
--     Sort Key: DepartmentId
--     -> Seq Scan on Employees (cost=0.00..200.00 rows=10000 width=60)
--         Buffers: shared hit=500
```

### Failure Modes

**Failure 1: FILTER in window with LEFT JOIN produces wrong results.**

```sql
-- If the FILTER condition references a LEFT JOINed table with NULLs,
-- the FILTER treats NULL = FALSE (not UNKNOWN), which is correct
-- but can surprise developers used to WHERE clause semantics
```

**Failure 2: RANGE with INTERVAL on non-unique ORDER BY values duplicates rows in the frame.**

```sql
-- Multiple rows with same MeasurementTime are all peer rows
-- RANGE includes ALL peers, potentially duplicating values in SUM
```

**Failure 3: GROUPS frame misapplied as ROWS-equivalent.**

```sql
-- GROUPS BETWEEN 2 PRECEDING AND CURRENT ROW includes all rows
-- from the current group's peers PLUS all rows from the previous
-- 2 groups — this can be many more rows than expected when groups are large
```

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- PostgreSQL-specific window function patterns with production data

-- Schema
CREATE TABLE sensor_readings (
    reading_id BIGSERIAL PRIMARY KEY,
    sensor_id INT NOT NULL,
    measurement_time TIMESTAMPTZ NOT NULL,
    temperature NUMERIC(5,2),
    humidity NUMERIC(5,2),
    pressure NUMERIC(6,2)
);

-- Insert sample data
INSERT INTO sensor_readings (sensor_id, measurement_time, temperature, humidity, pressure)
SELECT
    s.sensor_id,
    m.measurement_time,
    20 + random() * 15,
    40 + random() * 40,
    1000 + random() * 30
FROM generate_series(1, 10) AS s(sensor_id)
CROSS JOIN generate_series(
    '2024-01-01'::timestamptz,
    '2024-01-07'::timestamptz,
    '5 minutes'::interval
) AS m(measurement_time);

-- 1. FILTER in window: conditional aggregates in one pass
-- Daily summary with active vs inactive sensor counts
SELECT
    date_trunc('day', measurement_time) AS reading_day,
    sensor_id,
    AVG(temperature) FILTER(WHERE humidity > 60)
        OVER(PARTITION BY sensor_id ORDER BY date_trunc('day', measurement_time))
        AS avg_temp_high_humidity,
    AVG(temperature) FILTER(WHERE humidity <= 60)
        OVER(PARTITION BY sensor_id ORDER BY date_trunc('day', measurement_time))
        AS avg_temp_normal_humidity
FROM sensor_readings
ORDER BY reading_day, sensor_id;

-- 2. RANGE with INTERVAL: 1-hour rolling average
SELECT
    measurement_time,
    sensor_id,
    temperature,
    AVG(temperature) OVER(
        PARTITION BY sensor_id
        ORDER BY measurement_time
        RANGE BETWEEN INTERVAL '1 hour' PRECEDING AND CURRENT ROW
    ) AS hourly_moving_avg_temperature,
    COUNT(*) OVER(
        PARTITION BY sensor_id
        ORDER BY measurement_time
        RANGE BETWEEN INTERVAL '1 hour' PRECEDING AND CURRENT ROW
    ) AS readings_in_window
FROM sensor_readings
WHERE sensor_id = 1
ORDER BY measurement_time;

-- 3. GROUPS frame: smooth over adjacent reading groups
SELECT
    measurement_time,
    sensor_id,
    temperature,
    AVG(temperature) OVER(
        PARTITION BY sensor_id
        ORDER BY measurement_time
        GROUPS BETWEEN 1 PRECEDING AND 1 FOLLOWING
    ) AS group_smoothed_temperature
FROM sensor_readings
WHERE sensor_id = 1
ORDER BY measurement_time;

-- 4. WINDOW clause: reusable window definition
SELECT
    date_trunc('day', measurement_time) AS reading_day,
    sensor_id,
    AVG(temperature) OVER w AS avg_temperature,
    MIN(temperature) OVER w AS min_temperature,
    MAX(temperature) OVER w AS max_temperature,
    COUNT(*) OVER w AS reading_count
FROM sensor_readings
WINDOW w AS (
    PARTITION BY sensor_id, date_trunc('day', measurement_time)
    ORDER BY measurement_time
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
)
ORDER BY reading_day, sensor_id;

-- 5. array_agg as window function: ordered list per partition
SELECT
    sensor_id,
    measurement_time,
    temperature,
    array_agg(temperature) OVER(
        PARTITION BY sensor_id
        ORDER BY measurement_time
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS last_3_temperatures
FROM sensor_readings
WHERE sensor_id = 1
ORDER BY measurement_time;

-- 6. DISTINCT in window: cumulative distinct count (PostgreSQL 15+)
SELECT
    customer_id,
    order_date,
    product_category,
    COUNT(DISTINCT product_category) OVER(
        PARTITION BY customer_id
        ORDER BY order_date
        ROWS UNBOUNDED PRECEDING
    ) AS distinct_categories_so_far
FROM orders
ORDER BY customer_id, order_date;
```

            SELECT
                measurement_time,
                sensor_id,
                temperature,
                AVG(temperature) OVER(

                    ORDER BY measurement_time
                    RANGE BETWEEN INTERVAL '1 hour' PRECEDING AND CURRENT ROW

                    RANGE BETWEEN INTERVAL '1 hour' PRECEDING AND CURRENT ROW
                ) AS readings_in_window
            FROM sensor_readings
            WHERE sensor_id = @SensorId
            ORDER BY measurement_time";

        return await _dbContext.Database
            .SqlQueryRaw<HourlyAverage>(sql,
                new NpgsqlParameter("@SensorId", sensorId))

### SQL Server Equivalents — Migration Reference

```sql
-- ============================================================
-- Migration Reference: PostgreSQL → SQL Server
-- ============================================================

-- 1. FILTER clause → CASE inside aggregate (identical performance)
-- PostgreSQL:
AVG(salary) FILTER(WHERE status = 'Active') OVER(PARTITION BY dept_id)

-- SQL Server equivalent:
AVG(CASE WHEN status = 'Active' THEN salary END) OVER(PARTITION BY dept_id)

-- 2. RANGE with INTERVAL → subquery or ROWS with calculated frame
-- PostgreSQL:
SUM(amount) OVER(ORDER BY date RANGE BETWEEN INTERVAL '7 days' PRECEDING AND CURRENT ROW)

-- SQL Server equivalent (subquery approach):
SELECT
    date,
    amount,
    (SELECT SUM(amount)
     FROM sales s2
     WHERE s2.date BETWEEN DATEADD(DAY, -6, s1.date) AND s1.date
    ) AS weekly_sum
FROM sales s1;

-- Or (if row count per interval is known/stable):
ROW_NUMBER() OVER(ORDER BY date) AS rn,
SUM(amount) OVER(ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)

-- 3. GROUPS frame → not supported; use self-join or ROWS with fixed count
-- PostgreSQL:
AVG(value) OVER(ORDER BY category_id GROUPS BETWEEN 1 PRECEDING AND 1 FOLLOWING)

-- SQL Server (simulate with self-join):
SELECT a.category_id, a.value,
    AVG(b.value * 1.0) AS group_smoothed_avg
FROM categories a
LEFT JOIN categories b
    ON b.category_id BETWEEN a.category_id - 1 AND a.category_id + 1
GROUP BY a.category_id, a.value;

-- 4. WINDOW clause → repeat OVER clause
-- PostgreSQL:
WINDOW w AS (PARTITION BY dept_id ORDER BY salary DESC)

-- SQL Server: no WINDOW clause; repeat the full OVER specification

-- 5. array_agg as window function → STRING_AGG (not a window function in SQL Server)
-- PostgreSQL:
array_agg(employee_name) OVER(PARTITION BY dept_id ORDER BY salary DESC)

-- SQL Server: subquery with FOR XML PATH (pre-2017) or STRING_AGG with correlated subquery
SELECT department_id,
    STUFF((
        SELECT ',' + employee_name
        FROM employees e2
        WHERE e2.department_id = e1.department_id
        ORDER BY salary DESC
        FOR XML PATH('')
    ), 1, 1, '') AS employee_list
FROM employees e1
GROUP BY department_id;

-- 6. DISTINCT in window aggregate → subquery with ROW_NUMBER pre-aggregation
-- PostgreSQL 15+:
COUNT(DISTINCT product_id) OVER(PARTITION BY order_id)

-- SQL Server: pre-aggregate distinct values in a subquery, then apply window
SELECT order_id, COUNT(*) OVER(PARTITION BY order_id) AS distinct_product_count
FROM (
    SELECT DISTINCT order_id, product_id
    FROM order_items
) AS distinct_products;
```

## Gotchas and Production Pitfalls

### Pitfall 1: FILTER Clause Performance — Expression vs Predicate

**Pitfall:** Assuming FILTER in a window aggregate has the same performance characteristics as a WHERE clause.

```sql
-- ❌ FILTER is evaluated AFTER the scan — it does not reduce scan rows
SELECT AVG(temperature) FILTER(WHERE sensor_id = 1)
    OVER(PARTITION BY date_trunc('day', measurement_time))
FROM sensor_readings;
```

**Symptom:** The query scans the entire table even though only sensor_id = 1 is filtered. FILTER in a window aggregate does not push the condition to the scan — it is evaluated in the WindowAgg node.

**Fix:** Push the filter into a subquery WHERE clause before the window function:

```sql
-- ✅ Filter before window function (reduces scan)
SELECT date_trunc('day', measurement_time) AS reading_day,
    AVG(temperature) OVER(PARTITION BY date_trunc('day', measurement_time))
FROM sensor_readings
WHERE sensor_id = 1;  -- Filter applied at scan level
```

**Cost of not fixing:** Full table scan on 100M rows when sensor_id=1 is only 1% of the data. Query runs 10 seconds instead of 100ms.

### Pitfall 2: RANGE with INTERVAL on Non-Key Columns

**Pitfall:** Using RANGE with INTERVAL on a column that is not indexed. The Sort operator sorts by the ORDER BY column, which may be expensive.

```sql
-- No index on measurement_time:
SELECT measurement_time, temperature,
    AVG(temperature) OVER(
        ORDER BY measurement_time
        RANGE BETWEEN INTERVAL '1 hour' PRECEDING AND CURRENT ROW
    )
FROM sensor_readings;
```

**Symptom:** Full sequential scan + Sort on measurement_time. Sort may spill to disk if memory is insufficient.

**Fix:** Create an index on the ORDER BY column:

```sql
CREATE INDEX idx_sensor_readings_measurement_time
    ON sensor_readings(measurement_time);
```

**Cost of not fixing:** Query time dominated by Sort operator at high row counts.

### Pitfall 3: GROUPS Frame Misunderstood as ROWS

**Pitfall:** Assuming GROUPS BETWEEN 2 PRECEDING AND CURRENT ROW includes exactly 3 rows (like ROWS).

```sql
-- If CategoryId has 1000 rows with the same value:
AVG(value) OVER(ORDER BY category_id GROUPS BETWEEN 2 PRECEDING AND CURRENT ROW)
-- This includes ALL rows from the current group + ALL rows from the previous 2 groups
-- If each group has 1000 rows, the window includes 3000 rows, not 3
```

**Symptom:** Moving average is much smoother (more rows in the window) than expected. Window size varies with group size.

**Fix:** Use ROWS if a fixed row count is needed. Use GROUPS only when whole-group semantics are required.

**Cost of not fixing:** Incorrect analytics. A "3-point moving average" that actually averages 3000 rows per point hides signal variation.

### Pitfall 4: WINDOW Clause Does Not Support All Frame Types Uniformly

**Pitfall:** Defining a WINDOW clause with ROWS frame, then applying a window function that implicitly uses RANGE.

```sql
WINDOW w AS (PARTITION BY dept_id ORDER BY salary ROWS UNBOUNDED PRECEDING)

-- This works for ROW_NUMBER and SUM (ROWS frame accepted)
ROW_NUMBER() OVER w,
SUM(salary) OVER w,

-- But this ignores the ROWS frame and uses default RANGE:
-- (Because RANK uses default frame, which is RANGE)
RANK() OVER w  -- frame from WINDOW clause is overwritten by RANK's default
```

**Symptom:** RANK uses the default RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, not the ROWS frame from WINDOW clause. The window definition's frame is only applied when the window function does not have its own default frame.

**Fix:** Define separate WINDOW clauses for different frame requirements:

```sql
WINDOW
    w_rows AS (PARTITION BY dept_id ORDER BY salary ROWS UNBOUNDED PRECEDING),
    w_range AS (PARTITION BY dept_id ORDER BY salary)
```

**Cost of not fixing:** Unexpected results with RANK/DENSE_RANK if the frame matters for the calculation (it doesn't for ranking functions — frame is irrelevant — but the principle applies for aggregate functions over the window).

### Pitfall 5: DISTINCT in Window Functions (PostgreSQL 15+) — Performance Cost

**Pitfall:** Using COUNT(DISTINCT col) OVER() on a high-cardinality column without understanding the performance cost.

```sql
-- Expensive: must track distinct values per row within the window
COUNT(DISTINCT product_id) OVER(
    PARTITION BY customer_id
    ORDER BY order_date
    ROWS UNBOUNDED PRECEDING
) AS distinct_products_so_far
```

**Symptom:** The query maintains a hash set of distinct values for each partition. As the partition grows, the hash set memory and CPU cost grow linearly. For customers with 10K orders, each row's COUNT(DISTINCT) must check 10K previous values.

**Fix:** Pre-aggregate distinct values in a subquery, then apply the window function:

```sql
WITH distinct_products AS (
    SELECT DISTINCT customer_id, order_date, product_id
    FROM order_items
)
SELECT customer_id, order_date,
    COUNT(*) OVER(
        PARTITION BY customer_id
        ORDER BY order_date
        ROWS UNBOUNDED PRECEDING
    ) AS distinct_products_so_far
FROM distinct_products;
```

**Cost of not fixing:** O(n²) complexity for COUNT(DISTINCT) in a window. At 10M rows, this query may take hours.

### Pitfall 6: FILTER with LEFT JOIN NULLs

**Pitfall:** FILTER condition on a column from a LEFT JOINed table may include rows unexpectedly.

```sql
SELECT o.order_id, o.total_amount,
    AVG(o.total_amount) FILTER(
        WHERE p.category_id = 5  -- NULL when no matching product
    ) OVER() AS avg_amount_for_category_5
FROM orders o
LEFT JOIN products p ON o.product_id = p.product_id;
```

If a row has no matching product (p.product_id IS NULL), then p.category_id IS NULL, which is UNKNOWN in the comparison `= 5`, so UNKNOWN — the row is excluded from the FILTER. This is correct behavior, but developers may expect the LEFT JOIN to produce NULL and the FILTER to fail differently.

**Symptom:** avg_amount_for_category_5 excludes orders without a matching product, even though the intent might have been to include them as "not matching."

**Fix:** Be explicit about NULL handling in FILTER:

```sql
AVG(o.total_amount) FILTER(
    WHERE p.category_id IS NOT NULL AND p.category_id = 5
) OVER() AS avg_amount_for_category_5
```

**Cost of not fixing:** Subtle wrong numbers in analytics that depend on the presence of related rows.

## Performance Implications

### Benchmark: FILTER vs CASE in Window Aggregates

**PostgreSQL — FILTER vs CASE expression (functionally equivalent):**

```sql
-- Approach 1: FILTER (PostgreSQL only)
SELECT
    department_id,
    AVG(salary) FILTER(WHERE status = 'Active') OVER(PARTITION BY department_id) AS avg_active,
    AVG(salary) FILTER(WHERE status = 'Inactive') OVER(PARTITION BY department_id) AS avg_inactive
FROM employees;

-- Approach 2: CASE (portable)
SELECT
    department_id,
    AVG(CASE WHEN status = 'Active' THEN salary END) OVER(PARTITION BY department_id) AS avg_active,
    AVG(CASE WHEN status = 'Inactive' THEN salary END) OVER(PARTITION BY department_id) AS avg_inactive
FROM employees;
```

**Expected performance:** FILTER is marginally faster (~5-10%) because it avoids the CASE expression evaluation for non-matching rows. The difference is in the Compute Scalar (CASE) vs native aggregate filtering.

### Write Amplification

PostgreSQL-specific window function features are read-only — no write amplification. However, the indexes that support these features (for ORDER BY + PARTITION BY) add write overhead similar to SQL Server:

- B-tree index on (partition_col, order_by_col): ~3ms per INSERT/UPDATE

PostgreSQL's MVCC means index updates on UPDATE are DELETE + INSERT, doubling the write cost compared to SQL Server's in-place updates.

### Spoken Answers

**Q1: What PostgreSQL window function extensions are not available in SQL Server?**

> **Average answer:** "FILTER, GROUPS, WINDOW clause, array_agg. I know there are several differences."

> **Great answer:** "PostgreSQL provides six major window function extensions that SQL Server does not support. First, the FILTER clause — `AVG(salary) FILTER(WHERE status = 'Active') OVER(...)` — allows conditional aggregation within a window without CASE expressions. Second, RANGE with INTERVAL — `RANGE BETWEEN INTERVAL '7 days' PRECEDING AND CURRENT ROW` — enables temporal windows with date/time boundaries directly in the frame specification. Third, GROUPS frame mode — `GROUPS BETWEEN 1 PRECEDING AND 1 FOLLOWING` — which groups by ORDER BY value equality (like RANGE) but specifies boundaries in group counts (like ROWS). Fourth, the WINDOW clause — `WINDOW w AS (PARTITION BY ... ORDER BY ...)` — lets you define a window once and reference it by name. Fifth, array_agg as a window function — `array_agg(employee_name) OVER(PARTITION BY dept_id ORDER BY salary)` — builds an ordered array per row within each partition. Sixth, DISTINCT in aggregate window functions (PostgreSQL 15+) — `COUNT(DISTINCT product_id) OVER(...)` — which SQL Server cannot do. Each of these requires a different workaround in SQL Server: CASE expressions for FILTER, subqueries for RANGE with INTERVAL, repeated OVER clauses for WINDOW, and FOR XML PATH or STRING_AGG with GROUP BY for array_agg. The DISTINCT window workaround requires pre-aggregating distinct values in a subquery before applying the window function."

**Q3: What is the difference between ROWS, RANGE, and GROUPS frame modes?**

> **Average answer:** "ROWS is by row count, RANGE is by value, and GROUPS is by groups of equal values."

> **Great answer:** "In ROWS mode, frame boundaries are specified as a fixed number of rows — `ROWS BETWEEN 5 PRECEDING AND 5 FOLLOWING` always includes exactly 11 rows (assuming enough rows exist). In RANGE mode, boundaries are specified as a value interval relative to the current row's ORDER BY value — `RANGE BETWEEN '7 days' PRECEDING AND CURRENT ROW` includes all rows whose ORDER BY value is within 7 days before the current row's value, regardless of how many rows that is. In GROUPS mode, introduced in SQL:2011, boundaries are specified as a number of groups (ticks) — `GROUPS BETWEEN 1 PRECEDING AND 1 FOLLOWING` includes all rows from the current group (rows with the same ORDER BY value), plus all rows from the previous group and the next group. The key distinction: ROWS counts rows, RANGE measures value distance, and GROUPS counts distinct ORDER BY value groups. ROWS is the most predictable and performant because it is purely positional — no frame re-evaluation needed per row. RANGE requires comparing ORDER BY values, which is more expensive. GROUPS is a middle ground: the frame boundary is defined in group counts (like ROWS) but the frame content is all rows in those groups (like RANGE). SQL Server supports ROWS and RANGE but not GROUPS."

**Q7: What version of PostgreSQL added DISTINCT in window function aggregates? How do you achieve this in SQL Server?**

> **Average answer:** "PostgreSQL 15 added DISTINCT in window aggregates. SQL Server does not support it, so you use a subquery."

> **Great answer:** "PostgreSQL 15 introduced support for `COUNT(DISTINCT col) OVER(...)` and other DISTINCT aggregate variants in window functions. This was a long-requested feature — before PostgreSQL 15, you had to use subqueries or array_agg + unnest workarounds. SQL Server still does not support DISTINCT in any window aggregate. The error message is 'DISTINCT is not allowed with the OVER clause.' The workaround in SQL Server is to pre-aggregate distinct values in a subquery and then apply the window function: `SELECT order_id, COUNT(*) OVER(PARTITION BY customer_id) FROM (SELECT DISTINCT order_id, customer_id FROM order_items) AS distinct_orders`. This works because once you have deduplicated the rows, a regular COUNT(*) OVER gives you the distinct count. The performance tradeoff is that the subquery must materialize the distinct rows, which requires a Sort or Hash Aggregate on the deduplication columns. For high-cardinality columns, this is expensive — but the alternative (a custom CLR aggregate) is impractical for most teams."

<details>
<summary>Answers</summary>

1. **FILTER clause** adds a condition directly to the aggregate within the window: `AVG(salary) FILTER(WHERE status = 'Active') OVER(PARTITION BY dept_id)`. The equivalent with CASE is `AVG(CASE WHEN status = 'Active' THEN salary END) OVER(PARTITION BY dept_id)`. FILTER is slightly faster because it avoids computing the expression for non-matching rows.

2. **RANGE BETWEEN INTERVAL '1 day' PRECEDING AND CURRENT ROW** includes all rows whose ORDER BY value (e.g., a DATE or TIMESTAMP) is >= (current row's value - 1 day) and <= current row's value. The number of rows in the frame varies depending on the density of the data.

3. **GROUPS** groups rows by equality of the ORDER BY value. `GROUPS BETWEEN 1 PRECEDING AND 1 FOLLOWING` includes: all rows from the current group (same ORDER BY value), all rows from the previous group (previous distinct value), and all rows from the next group (next distinct value). **ROWS** counts individual rows. **RANGE** considers value distance.

4. **WINDOW clause** declares a window definition (PARTITION BY + ORDER BY + frame) once and references it by name: `WINDOW w AS (PARTITION BY dept_id ORDER BY salary ROWS UNBOUNDED PRECEDING)`. It reduces repetition but has no performance benefit — the optimizer would merge identical OVER clauses anyway.

5. **array_agg as a window function** collects values into an ordered array for each row: `array_agg(employee_name) OVER(PARTITION BY dept_id ORDER BY salary DESC)`. The result is an array column where each row contains an array of employee names up to and including the current row (or the full partition depending on frame).

6. **PostgreSQL 15** (released October 2022) added support for DISTINCT in aggregate window functions. Before 15, `COUNT(DISTINCT x) OVER(...)` produced a syntax error.

7. **SQL Server:** Pre-aggregate distinct values in a subquery, then apply the window function: `SELECT order_id, COUNT(*) OVER(PARTITION BY customer_id) FROM (SELECT DISTINCT order_id, customer_id FROM order_items) AS distinct_orders`. The DISTINCT in the subquery deduplicates, then COUNT(*) counts the remaining rows per partition.

8. **RANGE with INTERVAL** is 2-3x slower than ROWS because it cannot be evaluated incrementally. Each row's frame boundaries must be determined by comparing the ORDER BY value to all candidate rows. PostgreSQL uses a blocking spool to materialize the partition before computing the range. ROWS is purely positional and computed incrementally.

9. **No.** Npgsql's EF Core provider translates standard window functions (ROW_NUMBER, RANK, LAG, LEAD, SUM OVER) when using the appropriate LINQ patterns or within raw SQL. FILTER, GROUPS, RANGE with INTERVAL, WINDOW clause, and array_agg OVER are PostgreSQL-specific extensions that require `FromSqlRaw` or `FromSqlInterpolated`.

10. **60-second spoken answer — FILTER clause:** "PostgreSQL's FILTER clause lets you add a condition directly to an aggregate inside a window function: `AVG(salary) FILTER(WHERE status = 'Active') OVER(PARTITION BY dept_id)`. SQL Server does not support FILTER — you must use CASE. The performance difference is small, typically 5-10%, so the choice is mostly about code clarity. ### Query Challenges

**Challenge 1 — Write the PostgreSQL-Specific Query for 7-Day Rolling Average**

You have a `website_visits` table: `visit_id, page_url, visit_timestamp, duration_seconds`. Write a PostgreSQL query that returns each visit with a 7-day rolling average of daily visit count. Use RANGE with INTERVAL. Include the daily count in the output.

<details>
<summary>Solution</summary>

```sql
WITH daily_visits AS (
    SELECT
        visit_timestamp::date AS visit_date,
        COUNT(*) AS daily_count
    FROM website_visits
    GROUP BY visit_timestamp::date
)
SELECT
    visit_date,
    daily_count,
    AVG(daily_count) OVER(
        ORDER BY visit_date
        RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW
    ) AS rolling_7_day_avg,
    COUNT(*) OVER(
        ORDER BY visit_date
        RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW
    ) AS days_in_window
FROM daily_visits
ORDER BY visit_date;
```

**Note:** Use `INTERVAL '6 days' PRECEDING` because RANGE includes the current row. To get 7 days total (today + 6 previous), the interval is 6 days.

**SQL Server equivalent:**
```sql
WITH daily_visits AS (
    SELECT CAST(visit_timestamp AS DATE) AS visit_date,
        COUNT(*) AS daily_count
    FROM website_visits
    GROUP BY CAST(visit_timestamp AS DATE)
)
SELECT dv.visit_date, dv.daily_count,
    (SELECT AVG(dv2.daily_count * 1.0)
     FROM daily_visits dv2
     WHERE dv2.visit_date BETWEEN DATEADD(DAY, -6, dv.visit_date) AND dv.visit_date
    ) AS rolling_7_day_avg
FROM daily_visits dv
ORDER BY dv.visit_date;
```

**Challenge 2 — Fix the performance problem**

```sql
-- This PostgreSQL query runs 10x slower than expected on 5M sensor readings.
-- The execution plan shows a large Sort and a WindowAgg with "RANGE BETWEEN"
-- consuming most of the time.

SELECT
    measurement_time,
    sensor_id,
    temperature,
    AVG(temperature) OVER(
        PARTITION BY sensor_id
        ORDER BY measurement_time
        RANGE BETWEEN INTERVAL '1 hour' PRECEDING AND CURRENT ROW
    ) AS hourly_avg
FROM sensor_readings
WHERE sensor_id = 1
ORDER BY measurement_time;
```

<details>
<summary>Solution</summary>

**Root cause:** RANGE BETWEEN INTERVAL requires a blocking WindowAgg — it must compare each row's measurement_time to all other rows in the partition to determine frame membership. With 5M rows and a 1-hour window at 1-minute intervals (~60 rows per window), the frame is small, but the frame evaluation cost is still O(n * avg_window_size) comparisons.

**Fix 1 — Use ROWS with calculated frame size (if interval is stable):**
```sql
-- If readings are exactly 1 minute apart, 1 hour = 60 rows
-- ROWS is O(n) — evaluated incrementally
SELECT
    measurement_time,
    sensor_id,
    temperature,
    AVG(temperature) OVER(
        PARTITION BY sensor_id
        ORDER BY measurement_time
        ROWS BETWEEN 59 PRECEDING AND CURRENT ROW
    ) AS hourly_avg
FROM sensor_readings
WHERE sensor_id = 1
ORDER BY measurement_time;
```

**Fix 2 — Create a covering index:**
```sql
CREATE INDEX idx_sensor_readings_sensor_time_covering
    ON sensor_readings(sensor_id, measurement_time)
    INCLUDE (temperature);
```
This eliminates the Sort operator (data is read in order from the index).

**Fix 3 — Use a subquery approach with join (if ROWS approximation is not acceptable):**
```sql
SELECT sr.measurement_time, sr.sensor_id, sr.temperature,
    AVG(sr2.temperature) AS hourly_avg
FROM sensor_readings sr
LEFT JOIN sensor_readings sr2
    ON sr2.sensor_id = sr.sensor_id
    AND sr2.measurement_time > sr.measurement_time - INTERVAL '1 hour'
    AND sr2.measurement_time <= sr.measurement_time
WHERE sr.sensor_id = 1
GROUP BY sr.measurement_time, sr.sensor_id, sr.temperature
ORDER BY sr.measurement_time;
```
The self-join approach allows a nested loop join with index seek on (sensor_id, measurement_time). For a small window size (60 rows), this is efficient.

**Optimized query logical reads:** After index, the index scan reads ~14,000 rows (14K rows for sensor_id=1 over ~10 days at 1-minute intervals). The Sort is eliminated. Query time drops from 3 seconds to 200ms.

</details>

**Challenge 3 — Explain the PostgreSQL execution plan**

```
WindowAgg (cost=12345..15678 rows=50000 width=100)
  Filter: ((temperature > 50) AND (sensor_id = 1))
  -> Sort (cost=12345..13000 rows=50000 width=60)
    Sort Key: measurement_time
    -> Seq Scan on sensor_readings (cost=0..5000 rows=100000 width=60)
```

This plan has two problems. Identify them.

<details>
<summary>Solution</summary>

**Problem 1: Filter in WindowAgg, not in Seq Scan.** The `sensor_id = 1` filter is applied after the window function processes all rows. The Seq Scan reads all 100K rows, the Sort sorts them all, and only then does the WindowAgg filter to sensor_id = 1. The fix is to push the filter down into a WHERE clause on the Seq Scan.

**Problem 2: Full Seq Scan instead of index scan.** There is no index on (sensor_id, measurement_time). Adding a B-tree index would enable an index seek for sensor_id=1, reading only 50K rows (assuming 1 out of 2 sensors) instead of 100K, and the index provides ordered data, eliminating the Sort.

**Fixed query:**
```sql
SELECT measurement_time, sensor_id, temperature,
    AVG(temperature) OVER(
        PARTITION BY sensor_id ORDER BY measurement_time
        ROWS BETWEEN 59 PRECEDING AND CURRENT ROW
    ) AS hourly_avg
FROM sensor_readings
WHERE sensor_id = 1   -- Filter pushed to scan level
ORDER BY measurement_time;
```

**Fixed plan:**
```
WindowAgg (cost=500..1500 rows=50000 width=100)
  -> Index Scan using idx_sensor_readings_sensor_time
      on sensor_readings (cost=0..500 rows=50000 width=60)
      Index Cond: (sensor_id = 1)
```

No Sort, no scan of irrelevant sensor data, no Filter node in WindowAgg.

</details>

**Challenge 4 — Diagnose the concurrency problem**

A PostgreSQL batch job uses a window function with `array_agg OVER` on a table with 10M rows, partitioned by customer_id (100K distinct customers). The query runs for 30 minutes and the PostgreSQL server's memory usage spikes to 32GB, causing OOM killer to terminate the process. The query:

```sql
SELECT customer_id, order_date, total_amount,
    array_agg(order_id) OVER(
        PARTITION BY customer_id
        ORDER BY order_date
        ROWS UNBOUNDED PRECEDING
    ) AS all_order_ids_so_far
FROM orders;
```

<details>
<summary>Solution</summary>

**Root cause:** `array_agg OVER` with ROWS UNBOUNDED PRECEDING builds an array for each row that contains ALL preceding order IDs for that customer. For a customer with 10K orders, the 10,000th row has an array of 10,000 order IDs. The total memory is sum(customer_order_count²). With 100K customers, this is the sum of squares — easily 10s of gigabytes.

**Fix 1 — Limit the array size:**
```sql
SELECT customer_id, order_date, total_amount,
    array_agg(order_id) OVER(
        PARTITION BY customer_id
        ORDER BY order_date
        ROWS BETWEEN 99 PRECEDING AND CURRENT ROW  -- Only last 100
    ) AS last_100_order_ids
FROM orders;
```

**Fix 2 — Remove the array_agg if not needed for every row (compute only at partition end).**

**Fix 3 — Split into batch by customer range.**

**Fix 4 — Use a subquery with array_agg and GROUP BY instead of window function (if per-customer list is the final goal).**

```sql
SELECT customer_id, array_agg(order_id ORDER BY order_date) AS all_order_ids
FROM orders
GROUP BY customer_id;
```
This builds one array per customer, not one per row — same data, vastly less memory.

**Monitoring:** Check `pg_stat_activity` for queries with high `temp_files` and `temp_bytes`:
```sql
SELECT query, temp_files, temp_bytes
FROM pg_stat_activity
WHERE state = 'active';
```

**Challenge 5 — Design the migration from PostgreSQL to SQL Server**

**Scenario:** A .NET application is being migrated from PostgreSQL to SQL Server. The application uses these PostgreSQL-specific window function patterns:

1. `AVG(value) FILTER(WHERE category = 'X') OVER(PARTITION BY region ORDER BY date)`
2. `SUM(amount) OVER(ORDER BY date RANGE BETWEEN INTERVAL '30 days' PRECEDING AND CURRENT ROW)`
3. `WINDOW w AS (PARTITION BY dept ORDER BY salary ROWS UNBOUNDED PRECEDING)`
4. `array_agg(product_name) OVER(PARTITION BY order_id ORDER BY line_number)`

Design the SQL Server equivalents and describe the migration strategy.

<details>
<summary>Solution</summary>

```sql
-- 1. FILTER → CASE expression
-- PostgreSQL:
AVG(value) FILTER(WHERE category = 'X') OVER(PARTITION BY region ORDER BY date)

-- SQL Server:
AVG(CASE WHEN category = 'X' THEN value END) OVER(PARTITION BY region ORDER BY date)

-- 2. RANGE with INTERVAL → correlated subquery
-- PostgreSQL:
SUM(amount) OVER(ORDER BY date RANGE BETWEEN INTERVAL '30 days' PRECEDING AND CURRENT ROW)

-- SQL Server:
SELECT date, amount,
    (SELECT SUM(amount)
     FROM sales s2
     WHERE s2.date BETWEEN DATEADD(DAY, -29, s1.date) AND s1.date
    ) AS rolling_30_day_sum
FROM sales s1;

-- 3. WINDOW clause → repeat OVER specification
-- PostgreSQL:
SELECT ..., ROW_NUMBER() OVER w, SUM(salary) OVER w, AVG(salary) OVER w
FROM employees
WINDOW w AS (PARTITION BY dept ORDER BY salary ROWS UNBOUNDED PRECEDING)

-- SQL Server:
SELECT ...,
    ROW_NUMBER() OVER(PARTITION BY dept ORDER BY salary ROWS UNBOUNDED PRECEDING),
    SUM(salary) OVER(PARTITION BY dept ORDER BY salary ROWS UNBOUNDED PRECEDING),
    AVG(salary) OVER(PARTITION BY dept ORDER BY salary ROWS UNBOUNDED PRECEDING)
FROM employees;

-- 4. array_agg OVER → STRING_AGG with subquery (if string) or multiple rows
-- PostgreSQL:
array_agg(product_name) OVER(PARTITION BY order_id ORDER BY line_number)

-- SQL Server (string):
SELECT order_id,
    STUFF((
        SELECT ',' + product_name
        FROM order_items oi2
        WHERE oi2.order_id = oi1.order_id
        ORDER BY line_number
        FOR XML PATH('')
    ), 1, 1, '') AS product_list
FROM order_items oi1
GROUP BY order_id;
```

**Migration strategy:**

1. **Identify all PostgreSQL-specific window function usages** — search for FILTER, RANGE INTERVAL, GROUPS, WINDOW, array_agg OVER, DISTINCT in aggregate OVER.
2. **Mechanically rewrite FILTER to CASE** — this is a simple search-and-replace with validated equivalence.
3. **Rewrite RANGE with INTERVAL to correlated subqueries** — verify functional equivalence (consider edge cases: NULL dates, duplicate dates).
4. **Expand WINDOW clause references** — find all OVER w and replace with full OVER specification.
5. **Replace array_agg OVER with STRING_AGG or subquery FOR XML PATH** — for numeric arrays, use a JSON workaround or multiple rows.
6. **Remove DISTINCT from window aggregates** — use pre-aggregation subquery approach.
7. **Test with integration tests** — each rewritten query must produce identical results for the same data.

**In .NET:** Update raw SQL in Dapper queries and EF Core FromSqlRaw calls. Npgsql-specific parameter types may need to change to SqlClient equivalents.

</details>
