### Where This Fits

Dapper does not generate SQL — it executes the SQL you provide and maps the result rows to .NET objects. Window functions in Dapper are therefore a SQL concern, not a Dapper concern: you write the T-SQL with OVER clauses, Dapper maps the flat result rows to POCOs by column name. A .NET backend engineer encounters this in every analytics endpoint, reporting page, or dashboard that uses window functions with Dapper: top-N per group, running totals, paginated lists, year-over-year comparisons. The simplicity is both the strength (no ORM overhead, no translation surprises) and the weakness (no LINQ composition, no compile-time SQL checking). The Dapper-specific challenge is result mapping: window functions produce a flat rowset with one row per base row (when used with ranking or value functions) or one row per group (when used with aggregate window functions). Dapper maps each row to a POCO. For partitioned results, Dapper's multi-mapping (splitOn) can reconstruct parent-child relationships from a single query with window functions. The interview signal is the understanding that Dapper is the right tool for window function queries because it adds zero overhead between your SQL and the database engine.

## Core Mental Model

Dapper is a thin wrapper over ADO.NET's IDbConnection. It takes your SQL string, parameterizes it, executes it, and maps the result set to .NET objects using fast IL generation. The invariant: any SQL that produces a result set — including SQL with window functions — maps directly to a flat POCO. Each row becomes one POCO instance. The window function columns (RowNum, RunningTotal, LaggedValue, etc.) become properties on that POCO. Dapper does not interpret, transform, or optimize the SQL — it is a pass-through. This is the critical difference from EF Core: there is no LINQ provider to get in the way, no client evaluation risk, no translation failures. For window functions, Dapper's model is simpler and more predictable. The tradeoff: you write all the SQL yourself, and you must design the POCO to match the exact shape of the result set. For multi-result set scenarios (e.g., a page of ranked results plus total count), Dapper's QueryMultipleAsync executes a batch of SQL and maps multiple result sets independently.

### Classification

- **ORM pattern:** Micro-ORM — raw SQL pass-through with automatic result mapping
- **Translation responsibility:** Zero — developer writes SQL, Dapper maps results
- **Result mapping:** Column name to property name matching (case-insensitive, exact name)
- **Multi-mapping:** splitOn parameter for splitting a single row into multiple POCOs
- **Multi-result:** GridReader for executing multiple queries and reading each result set

## Deep Mechanics

### How Dapper Executes a Window Function Query

1. **SQL provisioning:** The developer provides a T-SQL string containing window functions. For example: `SELECT o.OrderId, o.CustomerId, o.TotalAmount, ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.TotalAmount DESC) AS RowNum FROM Orders o`.
2. **Command creation:** Dapper's `QueryAsync<T>` method creates a `SqlCommand` from the provided `IDbConnection`. It wraps the SQL and parameters into a `CommandDefinition`. No SQL transformation occurs — the SQL is passed verbatim.
3. **Execution:** The `DbCommand` executes against SQL Server. SQL Server compiles the query, including the window function operators (Sort, Segment, Sequence Project/Window Aggregate, Filter).
4. **Row materialization:** Dapper's `SqlMapper` reads each row from `DbDataReader`. For each row, it creates a `Func<DbDataReader, T>` delegate using IL generation (Emit) that reads each column by name and sets the corresponding property. This delegate is cached per type and column set — subsequent calls with the same shape skip IL generation.
5. **Column matching:** Dapper matches columns to properties by name (case-insensitive). The window function's alias becomes the column name. If the POCO has a property named `RowNum`, Dapper maps the `RowNum` column to it. If no matching property exists, the column is ignored (no exception in default behavior).
6. **Buffered vs non-buffered:** By default, `QueryAsync<T>` buffers results into a `List<T>`. For large result sets, `QueryAsync<T>(buffered: false)` streams rows one at a time, reducing memory pressure but keeping the connection open.

### SQL Visibility — Dapper Window Function Patterns

```sql
-- Pattern 1: Top-N per group (ROW_NUMBER with PARTITION BY)
SELECT CustomerId, OrderId, TotalAmount, OrderDate, RowNum
FROM (
    SELECT CustomerId, OrderId, TotalAmount, OrderDate,
        ROW_NUMBER() OVER(
            PARTITION BY CustomerId
            ORDER BY TotalAmount DESC, OrderDate DESC
        ) AS RowNum
    FROM Orders
) AS RankedOrders
WHERE RowNum <= @TopN
ORDER BY CustomerId, RowNum;

-- Pattern 2: Running total (SUM OVER)
SELECT OrderId, CustomerId, OrderDate, TotalAmount,
    SUM(TotalAmount) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate, OrderId
        ROWS UNBOUNDED PRECEDING
    ) AS RunningTotal,
    AVG(TotalAmount) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate, OrderId
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS MovingAvg3
FROM Orders
WHERE CustomerId = @CustomerId
ORDER BY OrderDate, OrderId;

-- Pattern 3: Year-over-year comparison (LAG)
WITH YearlySales AS (
    SELECT YEAR(OrderDate) AS SalesYear,
        SUM(TotalAmount) AS TotalSales
    FROM Orders
    GROUP BY YEAR(OrderDate)
)
SELECT SalesYear, TotalSales,
    LAG(TotalSales) OVER(ORDER BY SalesYear) AS PreviousYearSales,
    (TotalSales - LAG(TotalSales) OVER(ORDER BY SalesYear)) * 1.0
        / NULLIF(LAG(TotalSales) OVER(ORDER BY SalesYear), 0) * 100 AS YoYGrowthPct
FROM YearlySales
ORDER BY SalesYear;

-- Pattern 4: Running count per partition (COUNT OVER)
SELECT OrderId, CustomerId, OrderDate,
    COUNT(*) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate, OrderId
        ROWS UNBOUNDED PRECEDING
    ) AS OrderNumberPerCustomer
FROM Orders
WHERE CustomerId = @CustomerId
ORDER BY OrderDate, OrderId;
```

### Execution Plan Analysis

The execution plan for a Dapper-executed window function query is identical to the same query executed in SSMS. Dapper adds zero plan overhead.

```sql
-- SQL passed to Dapper
SELECT CustomerId, OrderId, TotalAmount, RowNum
FROM (
    SELECT CustomerId, OrderId, TotalAmount,
        ROW_NUMBER() OVER(
            PARTITION BY CustomerId
            ORDER BY TotalAmount DESC, OrderId DESC
        ) AS RowNum
    FROM Orders
) AS RankedOrders
WHERE RowNum <= 3
ORDER BY CustomerId, RowNum;
```

Expected plan:
```
[Clustered Index Scan (Orders)]
  → [Sort (ORDER BY CustomerId, TotalAmount DESC, OrderId DESC)]
    → [Segment (partition boundary detection)]
      → [Sequence Project (ROW_NUMBER computation)]
        → [Filter (WHERE RowNum <= 3)]
          → [Sort (ORDER BY CustomerId, RowNum)]
            → [SELECT]
```

The Dapper call adds no operators. The plan is determined entirely by the SQL and the indexes present.

### Failure Modes

**Failure 1: Column name mismatch — property silently remains default value.**

```csharp
// SQL returns "RowNum" but POCO has "Rank"
public class CustomerTopOrder
{
    public int Rank { get; init; }  // Always 0 — no matching column
}
```
Dapper does not throw or warn when a column has no matching property or a property has no matching column. The property `Rank` remains at its default value (0 for int, null for nullable).

**Fix:** Ensure column aliases in SQL exactly match POCO property names. Use `[Column("RowNum")]` attribute to map mismatched names.

**Failure 2: Buffered mode on large result sets causes OOM.**

```csharp
// ❌ Buffered mode (default) — all rows in memory
var results = await connection.QueryAsync<OrderWithRunningTotal>(sql);

// For 1M rows at 100 bytes each = 100MB in memory
```

**Fix:** Use `buffered: false` for streaming:
```csharp
var results = await connection.QueryAsync<OrderWithRunningTotal>(
    new CommandDefinition(sql, buffered: false));
// Process rows one at a time via IEnumerable<T>
```

**Failure 3: QueryMultipleAsync result set order mismatched.**

```csharp
var grid = await connection.QueryMultipleAsync(sql);
// SQL returns: Result set 1 = ranked rows, Result set 2 = total count
var orders = await grid.ReadAsync<OrderRanking>();    // Must read in order
var totalCount = await grid.ReadSingleAsync<int>();   // Second
```
If the SQL returns result sets in a different order than the C# `ReadAsync` calls, type mismatches occur (e.g., trying to read a row as `int` when it's actually a complex type).

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- Production schema for window function examples
CREATE TABLE #Orders (
    OrderId INT IDENTITY(1,1) PRIMARY KEY,
    CustomerId INT NOT NULL,
    OrderDate DATETIME2 NOT NULL,
    TotalAmount DECIMAL(10,2) NOT NULL,
    Status VARCHAR(20) NOT NULL DEFAULT 'Pending'
);

-- Seed 100K rows
WITH Numbers AS (
    SELECT TOP 100000 ROW_NUMBER() OVER(ORDER BY (SELECT NULL)) AS N
    FROM sys.columns a CROSS JOIN sys.columns b
)
INSERT INTO #Orders (CustomerId, OrderDate, TotalAmount, Status)
SELECT
    N % 1000 + 1,
    DATEADD(DAY, N, '2023-01-01'),
    CAST(ABS(CHECKSUM(NEWID())) % 10000 + 1 AS DECIMAL(10,2)),
    CASE WHEN N % 5 = 0 THEN 'Shipped' ELSE 'Pending' END
FROM Numbers;

-- 1. Top-3 orders per customer (ROW_NUMBER with PARTITION BY)
SELECT CustomerId, OrderId, TotalAmount, OrderDate, RowNum
FROM (
    SELECT CustomerId, OrderId, TotalAmount, OrderDate,
        ROW_NUMBER() OVER(
            PARTITION BY CustomerId
            ORDER BY TotalAmount DESC, OrderDate DESC
        ) AS RowNum
    FROM #Orders
) AS RankedOrders
WHERE RowNum <= 3
ORDER BY CustomerId, RowNum;

-- 2. Running total per customer (SUM OVER with frame)
SELECT OrderId, CustomerId, OrderDate, TotalAmount,
    SUM(TotalAmount) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate, OrderId
        ROWS UNBOUNDED PRECEDING
    ) AS RunningTotal
FROM #Orders
ORDER BY CustomerId, OrderDate, OrderId;

-- 3. Moving average (3-order window)
SELECT OrderId, CustomerId, OrderDate, TotalAmount,
    AVG(TotalAmount) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate, OrderId
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS MovingAvg3,
    COUNT(*) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate, OrderId
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS MovingCount3
FROM #Orders
ORDER BY CustomerId, OrderDate, OrderId;

-- 4. Year-over-year comparison
WITH YearlySales AS (
    SELECT YEAR(OrderDate) AS SalesYear,
        COUNT(*) AS OrderCount,
        SUM(TotalAmount) AS TotalSales
    FROM #Orders
    GROUP BY YEAR(OrderDate)
)
SELECT SalesYear, OrderCount, TotalSales,
    LAG(TotalSales) OVER(ORDER BY SalesYear) AS PreviousSales,
    (TotalSales - LAG(TotalSales) OVER(ORDER BY SalesYear)) * 1.0
        / NULLIF(LAG(TotalSales) OVER(ORDER BY SalesYear), 0) * 100 AS YoYGrowthPct,
    LAG(OrderCount) OVER(ORDER BY SalesYear) AS PreviousOrderCount,
    OrderCount - LAG(OrderCount) OVER(ORDER BY SalesYear) AS OrderCountChange
FROM YearlySales
ORDER BY SalesYear;

-- 5. Dense ranking with distribution (percentile-like)
SELECT OrderId, CustomerId, TotalAmount,
    RANK() OVER(ORDER BY TotalAmount DESC) AS RankDesc,
    DENSE_RANK() OVER(ORDER BY TotalAmount DESC) AS DenseRankDesc,
    ROW_NUMBER() OVER(ORDER BY TotalAmount DESC) AS RowNumDesc,
    COUNT(*) OVER() AS TotalOrders,
    CAST(ROW_NUMBER() OVER(ORDER BY TotalAmount DESC) AS FLOAT)
        / COUNT(*) OVER() * 100 AS PercentileRank
FROM #Orders;

DROP TABLE #Orders;
```

### Dapper Multi-Mapping for Partitioned Data

```csharp
/// <summary>
/// Uses Dapper multi-mapping to reconstruct an Order → OrderItems hierarchy
/// from a single query that includes a window function for ranking.
/// The splitOn parameter tells Dapper where the Order columns end and the OrderItems columns begin.
/// </summary>
public async Task<IReadOnlyList<OrderWithTopItems>> GetOrdersWithTopItemsAsync(
    int topItemsPerOrder,
    CancellationToken cancellationToken = default)
{
    const string sql = @"
        SELECT
            o.OrderId, o.CustomerId, o.OrderDate, o.TotalAmount,
            oi.OrderItemId, oi.ProductId, oi.Quantity, oi.UnitPrice,
            ROW_NUMBER() OVER(
                PARTITION BY oi.OrderId
                ORDER BY oi.Quantity * oi.UnitPrice DESC
            ) AS ItemRank
        FROM Orders o
        INNER JOIN OrderItems oi ON o.OrderId = oi.OrderId
        ORDER BY o.OrderId, ItemRank";

    await using var connection = _connectionFactory.Create();

    var orderDictionary = new Dictionary<int, OrderWithTopItems>();

    var results = await connection.QueryAsync<OrderWithTopItems, OrderItemRanked, OrderWithTopItems>(
        sql,
        (order, orderItem) =>
        {
            if (!orderDictionary.TryGetValue(order.OrderId, out var existingOrder))
            {
                existingOrder = order;
                existingOrder.TopItems = new List<OrderItemRanked>();
                orderDictionary.Add(existingOrder.OrderId, existingOrder);
            }

            if (orderItem.ItemRank <= topItemsPerOrder)
            {
                existingOrder.TopItems.Add(orderItem);
            }

            return existingOrder;
        },
        splitOn: "OrderItemId",  // Dapper splits after OrderItemId — columns before map to Order, columns after to OrderItem
        commandDefinition: new CommandDefinition(sql, cancellationToken: cancellationToken));

    return orderDictionary.Values
        .Where(o => o.TopItems.Count > 0)
        .ToList()
        .AsReadOnly();
}

public class OrderWithTopItems
{
    public int OrderId { get; init; }
    public int CustomerId { get; init; }
    public DateTime OrderDate { get; init; }
    public decimal TotalAmount { get; init; }
    public List<OrderItemRanked> TopItems { get; set; } = new();
}

public class OrderItemRanked
{
    public int OrderItemId { get; init; }
    public int ProductId { get; init; }
    public int Quantity { get; init; }
    public decimal UnitPrice { get; init; }
    public long ItemRank { get; init; }
}
```

### Dapper GridReader — Multiple Result Sets

```csharp
/// <summary>
/// Uses QueryMultipleAsync to return both the window function result
/// and the total count in a single database round-trip.
/// </summary>
public async Task<(IReadOnlyList<OrderRanking> Items, int TotalCount)> GetPagedOrdersAsync(
    int pageNumber,
    int pageSize,
    CancellationToken cancellationToken = default)
{
    const string sql = @"
        SELECT OrderId, CustomerId, OrderDate, TotalAmount,
            ROW_NUMBER() OVER(
                ORDER BY OrderDate DESC, OrderId DESC
            ) AS RowNum
        FROM Orders
        ORDER BY RowNum
        OFFSET @Offset ROWS FETCH NEXT @PageSize ROWS ONLY;

        SELECT COUNT(*) AS TotalCount FROM Orders;";

    await using var connection = _connectionFactory.Create();
    await using var grid = await connection.QueryMultipleAsync(
        new CommandDefinition(
            sql,
            new
            {
                Offset = (pageNumber - 1) * pageSize,
                PageSize = pageSize
            },
            cancellationToken: cancellationToken));

    var items = (await grid.ReadAsync<OrderRanking>()).AsList();
    var totalCount = await grid.ReadSingleAsync<int>();

    return (items, totalCount);
}

    public int CustomerId { get; init; }
```

### Dapper with Window Function + Pagination (Keyset)

```csharp
/// <summary>
/// Keyset pagination (seek method) using window function.
/// More efficient than OFFSET-FETCH for deep pages.
/// </summary>
public async Task<IReadOnlyList<OrderRanking>> GetOrdersKeysetAsync(
    DateTime? lastOrderDate,
    int? lastOrderId,
    int pageSize,
    CancellationToken cancellationToken = default)
{
    const string sql = @"
        SELECT TOP (@PageSize) OrderId, CustomerId, OrderDate, TotalAmount,
            ROW_NUMBER() OVER(
                ORDER BY OrderDate DESC, OrderId DESC
            ) AS RowNum
        FROM Orders
        WHERE (OrderDate < @LastOrderDate
            OR (OrderDate = @LastOrderDate AND OrderId < @LastOrderId))
        ORDER BY OrderDate DESC, OrderId DESC";

    await using var connection = _connectionFactory.Create();
    var results = await connection.QueryAsync<OrderRanking>(
        new CommandDefinition(
            sql,
            new
            {
                PageSize = pageSize,
                LastOrderDate = lastOrderDate ?? DateTime.MaxValue,
                LastOrderId = lastOrderId ?? int.MaxValue
            },
            cancellationToken: cancellationToken));

    return results.AsList();
}
```

## Gotchas and Production Pitfalls

### Pitfall 1: Dapper Buffered Mode Causes OOM on Large Window Function Results

**Pitfall:** Using `QueryAsync<T>` (default buffered=true) on a window function query that returns millions of rows.

```csharp
// ❌ Buffered — loads all rows into memory
var allOrders = await connection.QueryAsync<OrderWithRunningTotal>(sql);
// With 2M rows at 80 bytes each = 160MB allocation
```

**Symptom:** High memory usage on the application server. GC pressure causes high % Time in GC (perf counter). OOM exceptions under concurrent usage.

**Fix:**

```csharp
// ✅ Streaming mode
var orders = await connection.QueryAsync<OrderWithRunningTotal>(
    new CommandDefinition(sql, buffered: false));

// Process as a stream
await foreach (var order in orders.AsAsyncEnumerable())
{
    ProcessOrder(order);  // One row at a time
}
```

**Cost of not fixing:** OOM kills the application process. On Azure App Service, the process is recycled, losing in-flight requests.

### Pitfall 2: Column Name Mismatch — No Error, Wrong Data

**Pitfall:** SQL alias does not match POCO property name. Dapper silently skips columns that don't match.

```csharp
// SQL: SELECT ..., COUNT(*) OVER() AS TotalCount
// POCO:
public class OrderWithStats
{
    public int TotalCount { get; init; }  // ❌ Property name doesn't match column "TotalCount"
}
```
Wait — this matches! But if the property were `TotalRowCount` and the column is `TotalCount`, Dapper keeps `TotalRowCount` at 0.

**Symptom:** A property is always 0, null, or false. Developers waste hours debugging "why is running total always 0?"

**Fix:** Use Dapper's `[Column]` attribute or match names exactly.

```csharp
public class OrderWithRunningTotal
{
    public int OrderId { get; init; }
    [Column("RunningTotal")]  // Explicit mapping
    public decimal TotalSoFar { get; init; }
}
```

**Cost of not fixing:** Incorrect analytics displayed to users. A dashboard showing "Top 3 products" actually shows products with RowNum always 0 (the first partition order).

### Pitfall 3: QueryMultipleAsync Misordered Read Calls

**Pitfall:** Calling `ReadAsync<T>` in a different order than the result sets appear in the SQL.

```csharp
const string sql = @"
    SELECT OrderId, CustomerId, TotalAmount,
        ROW_NUMBER() OVER(ORDER BY TotalAmount DESC) AS RowNum
    FROM Orders
    WHERE CustomerId = @CustomerId;   -- Result set 1

    SELECT COUNT(*) FROM Orders;       -- Result set 2
";

var grid = await connection.QueryMultipleAsync(sql, new { CustomerId = 1 });
// ❌ Wrong order: trying to read total count first
var totalCount = await grid.ReadSingleAsync<int>();       // Reads result set 1 — fails
var orders = await grid.ReadAsync<OrderRanking>();        /// Reads result set 2 — wrong type
```

**Symptom:** InvalidCastException or wrong data mapped to the POCO. The second ReadAsync gets the wrong result set shape.

**Fix:** Read in the exact order of result sets in the SQL.

```csharp
// ✅ Correct order
var orders = await grid.ReadAsync<OrderRanking>();         // Result set 1
var totalCount = await grid.ReadSingleAsync<int>();        // Result set 2
```

**Cost of not fixing:** Runtime exceptions in production that are hard to reproduce because they depend on result set ordering.

### Pitfall 4: Dapper Cannot Map Window Function to Nested Objects

**Pitfall:** Assuming Dapper will automatically nest window function results into child objects.

```csharp
// ❌ This does NOT work — window function + JOIN produces flat rows
public class CustomerWithTopOrders
{
    public int CustomerId { get; init; }
    public string CustomerName { get; init; }
    public List<Order> TopOrders { get; init; }  // ❌ Dapper won't populate this
}
```
Dapper maps flat rows to flat POCOs. It does not automatically detect nested collections.

**Fix:** Use multi-mapping with `splitOn`:

```csharp
var customerDict = new Dictionary<int, CustomerWithTopOrders>();
var results = await connection.QueryAsync<CustomerWithTopOrders, OrderRanking, CustomerWithTopOrders>(
    sql,
    (customer, order) =>
    {
        if (!customerDict.TryGetValue(customer.CustomerId, out var existing))
        {
            existing = customer;
            existing.TopOrders = new List<OrderRanking>();
            customerDict.Add(existing.CustomerId, existing);
        }
        existing.TopOrders.Add(order);
        return existing;
    },
    splitOn: "OrderId",
    commandDefinition: new CommandDefinition(sql, cancellationToken: ct));
```

**Cost of not fixing:** Developers write N+1 queries to get data that could be returned in one trip.

### Pitfall 5: LAG/LEAD with NULL Default — Type Mismatch

**Pitfall:** LAG/LEAD returns NULL for the first/last row, but the POCO property is a non-nullable type.

```csharp
// SQL: LAG(TotalAmount) OVER(ORDER BY OrderDate) AS PreviousAmount
// First row: PreviousAmount is NULL

public class YoYComparison
{
    public decimal PreviousAmount { get; init; }  // ❌ Non-nullable — Dapper sets 0
}
```

**Symptom:** First row shows `PreviousAmount = 0` instead of NULL. This looks like a valid value (zero sales) rather than "no previous row."

**Fix:**

```csharp
public class YoYComparison
{
    public decimal? PreviousAmount { get; init; }  // ✅ Nullable — matches LAG behavior
}
```

**Cost of not fixing:** Year-over-year growth calculation for the first year shows -100% (comparing first year sales to 0) instead of NULL.

### Pitfall 6: Performance — Window Function Sort Not Indexed

**Pitfall:** Assuming Dapper's fast mapping compensates for a missing index that causes a Sort spill.

```csharp
// SQL with ROW_NUMBER OVER(PARTITION BY CustomerId ORDER BY OrderDate)
// If no index on (CustomerId, OrderDate), SQL Server sorts 1M rows
var results = await connection.QueryAsync<OrderRanking>(sql); // 3-second query
```

**Symptom:** The query runs slow regardless of Dapper's efficiency. Dapper's mapping takes 1-2µs per row (2 seconds for 1M rows), but SQL Server's Sort takes 10+ seconds.

**Fix:** Create the covering index:

```sql
CREATE INDEX IX_Orders_CustomerId_OrderDate
    ON Orders(CustomerId, OrderDate)
    INCLUDE (TotalAmount);
```

The query then uses an Index Scan (ordered) instead of a Sort, eliminating the bottleneck.

**Cost of not fixing:** 15-second API calls. Application timeouts. The Sort in the execution plan spills to TempDB, causing disk I/O.

### Spoken Answers

**Q1: Does Dapper support window functions? How do you use them?**

> **Average answer:** "Yes, Dapper supports window functions. You just write the SQL with the window function and Dapper maps the result."

> **Great answer:** "Dapper does not generate any SQL itself — it is a pass-through micro-ORM. So the question is not whether Dapper supports window functions, but whether the SQL you write with window functions can be mapped by Dapper. And the answer is yes: any SQL result set can be mapped by Dapper, including those with ROW_NUMBER, RANK, LAG, LEAD, SUM OVER, or any other window function. You write the T-SQL with the OVER clause just as you would in SSMS, pass it to QueryAsync<T>, and Dapper maps each row to a POCO by column name. The window function column aliases (e.g., 'RowNum', 'RunningTotal') become properties on the POCO. There is no special API or configuration required for window functions. The key advantage over EF Core is that there is no LINQ provider that could fail to translate or worse, evaluate client-side. With Dapper, you own the SQL entirely — you are responsible for writing it correctly, but you have full control over the execution plan. The only Dapper-specific consideration is that the POCO must have properties matching the column names, and for large result sets, you should use buffered: false to stream rows instead of loading everything into memory."

**Q2: Compare Dapper and EF Core for executing queries with window functions.**

> **Average answer:** "Dapper is faster because it's lighter. EF Core requires raw SQL too. I'd use Dapper for performance."

> **Great answer:** "Both Dapper and EF Core require raw SQL for window functions — neither can translate LINQ to OVER clauses. The difference is in the surrounding plumbing. With Dapper, you write the SQL, create an IDbConnection, call QueryAsync<T>, and get results. There is no DbContext setup, no OnModelCreating, no registration of keyless entities. The connection is created and disposed explicitly. With EF Core's FromSqlRaw, you must register a keyless entity type with HasNoKey and ToView(null), define a DbSet property, and ensure column name matching. EF Core then supports LINQ composition on top of the raw SQL — you can chain Where, OrderBy, and Select after FromSqlRaw. Dapper has no composition layer — every query variation requires modifying the SQL string. The performance difference is small: Dapper is typically 5-10% faster and allocates less memory because it skips EF Core's state manager and change tracking. For a service that primarily does analytics with window functions, I would use Dapper for simplicity. For a service that needs both EF Core CRUD and occasional window function queries, I would use EF Core's FromSqlRaw to stay within the same data access pattern. The critical factor is not performance (both are sub-millisecond overhead compared to network and query time) but development complexity and team familiarity."

**Q5: How would you use Dapper's multi-mapping to reconstruct a parent-child relationship from a window function query?**

> **Average answer:** "You can use splitOn to split a row into multiple objects. I've used it for Order → OrderItems."

> **Great answer:** "Dapper's multi-mapping with splitOn is very useful when a window function query includes a JOIN and you want to reconstruct the parent-child hierarchy on the client side. For example: I want each Order with its top 3 OrderItems by line total. The SQL joins Orders to OrderItems, uses ROW_NUMBER() OVER(PARTITION BY OrderItems.OrderId ORDER BY Quantity * UnitPrice DESC) to rank items within each order, and filters to RowNum <= 3. The result is one row per item, with Order columns repeated. Calling QueryAsync<Order, OrderItem, Order> with splitOn: 'OrderItemId' tells Dapper: 'columns up to OrderItemId map to Order, columns from OrderItemId onward map to OrderItem.' But Dapper still returns one row per item — it does not automatically group them into Order objects. You provide a callback function that collects OrderItems into a Dictionary<int, Order>, adding each OrderItem to its parent Order. This gives you the completed hierarchy from a single database round trip with the window function filtering already applied. The keys: use splitOn at the column boundary between parent and child, and use a dictionary to track parent instances so you don't create duplicates for each child row."

<details>
<summary>Answers</summary>

1. **Dapper does not generate any SQL** — it is a pass-through micro-ORM. You write the T-SQL with window functions, pass it to `QueryAsync<T>`, and Dapper maps the result rows to POCOs.

2. **Dapper matches columns to properties by name** (case-insensitive). If the SQL column is aliased `AS RowNum`, the POCO must have a property named `RowNum` (or mapped via `[Column("RowNum")]` attribute).

3. **The column is silently ignored.** Dapper reads only columns that match POCO properties. Unmatched columns have no effect, but data from window functions may be lost.

4. **The property retains its default value** (0 for int, null for nullable types, false for bool). No error is thrown.

5. **Buffered mode** (default: true) reads all rows into a `List<T>` in memory before returning. **Non-buffered mode** (buffered: false) returns a lazy `IEnumerable<T>` that streams rows one at a time, keeping the connection open.

6. **Write SQL with ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY TotalAmount DESC) AS RowNum** in a subquery, filter WHERE RowNum <= @TopN in the outer query. Map the result to a POCO with a `RowNum` property. No Dapper-specific configuration needed.

7. **splitOn** tells Dapper where one object's columns end and the next object's columns begin in a flat row. Dapper splits the row columns at the splitOn column name — everything before maps to the first type, everything after to the second type.

8. **QueryMultipleAsync** executes a batch of SQL statements (semicolon-separated) and returns a `GridReader` that reads each result set independently via `ReadAsync<T>()`. Use it with window functions when you need both the ranked data and a total count in one round trip.

9. **Dapper maps NULL to null for nullable types** and to default(T) for non-nullable types. If the POCO property is `decimal` (non-nullable), the first row's `PreviousAmount` from LAG will be 0 instead of NULL. Use `decimal?` for LAG/LEAD columns.

10. **60-second spoken answer:** "Dapper is a pass-through micro-ORM — it does not generate SQL or translate LINQ. To use window functions with Dapper, you write the T-SQL with OVER clauses and call QueryAsync<T>. Dapper maps each row to a POCO by matching column aliases to property names. There is no client evaluation risk because Dapper never interprets the SQL. For large results, use buffered: false to stream rows. For parent-child reconstruction, use splitOn in multi-mapping. For pagination, use QueryMultipleAsync with separate SELECT statements for data and count. Dapper's advantage over EF Core is simplicity: no DbSet registration, no OnModelCreating, no keyless entity configuration. Just SQL, connection, and POCO."

**Challenge 2 — Fix the performance problem**

```csharp
// This Dapper query runs slowly (8 seconds) on a 5M row Orders table.
// It returns the top 5 orders per customer.

public async Task<IReadOnlyList<OrderRanking>> GetTopOrdersAsync(
    CancellationToken ct)
{
    const string sql = @"
        SELECT CustomerId, OrderId, TotalAmount, OrderDate,
            ROW_NUMBER() OVER(
                PARTITION BY CustomerId
                ORDER BY TotalAmount DESC, OrderDate DESC
            ) AS RowNum
        FROM Orders";

    await using var connection = new SqlConnection(connectionString);
    var results = await connection.QueryAsync<OrderRanking>(
        new CommandDefinition(sql, cancellationToken: ct));
    return results.AsList();
}
```

<details>
<summary>Solution</summary>

**Root cause:** The query computes ROW_NUMBER for ALL orders (scanning 5M rows) but only top 5 per customer are needed. The WHERE RowNum <= 5 filter is missing — the query transfers all 5M rows to Dapper. The Sort operator sorts all 5M rows by CustomerId + TotalAmount DESC.

```csharp
// ✅ Fixed: add outer filter to limit rows
public async Task<IReadOnlyList<OrderRanking>> GetTopOrdersAsync(
    int topN, CancellationToken ct)
{
    const string sql = @"
        SELECT CustomerId, OrderId, TotalAmount, OrderDate, RowNum
        FROM (
            SELECT CustomerId, OrderId, TotalAmount, OrderDate,
                ROW_NUMBER() OVER(
                    PARTITION BY CustomerId
                    ORDER BY TotalAmount DESC, OrderDate DESC
                ) AS RowNum
            FROM Orders
        ) AS RankedOrders
        WHERE RowNum <= @TopN
        ORDER BY CustomerId, RowNum";

    await using var connection = new SqlConnection(connectionString);
    var results = await connection.QueryAsync<OrderRanking>(
        new CommandDefinition(sql, new { TopN = topN }, cancellationToken: ct));
    return results.AsList();
}
```

**After fix — logical reads:** Same scan (must still read all rows to rank them) but only returns 5 * (number of customers) rows instead of 5M. **Network transfer:** Reduced from ~400MB to ~200KB. **Elapsed time:** ~800ms (from 8 seconds).

**Additional optimization:** Add index on (CustomerId, TotalAmount DESC, OrderDate DESC) INCLUDE (OrderId). This provides an ordered scan for the window function, eliminating the Sort operator.

```sql
CREATE INDEX IX_Orders_CustomerId_TotalAmount
    ON Orders(CustomerId, TotalAmount DESC, OrderDate DESC)
    INCLUDE (OrderId);
```

**Challenge 3 — Explain the execution plan**

```sql
SELECT OrderId, CustomerId, TotalAmount,
    SUM(TotalAmount) OVER(
        PARTITION BY CustomerId
        ORDER BY OrderDate, OrderId
        ROWS UNBOUNDED PRECEDING
    ) AS RunningTotal
FROM Orders
WHERE CustomerId = 123
ORDER BY OrderDate, OrderId;
```

Why does this query show a Sort operator in the execution plan even when there is an index on (CustomerId, OrderDate, OrderId)?

<details>
<summary>Solution</summary>

**Why Sort appears:** The index is `IX_Orders_CustomerId_OrderDate(CustomerId, OrderDate, OrderId)`. This should provide sorted data for the PARTITION BY CustomerId and ORDER BY OrderDate, OrderId. However, if the index does not include `TotalAmount` as an INCLUDE column, the optimizer may choose a Clustered Index Scan (which requires a Sort) or a Key Lookup (which is expensive for 500 rows). If it chooses the Clustered Index Scan + Sort, the Sort appears even though the data could be read in order from the index — because the optimizer estimates the Key Lookup cost is higher than scanning + sorting.

**What to change:** Create a covering index:

```sql
CREATE INDEX IX_Orders_CustomerId_OrderDate_Covering
    ON Orders(CustomerId, OrderDate, OrderId)
    INCLUDE (TotalAmount);
```

With this index, the Index Scan is ordered (eliminating the Sort operator) and the index is covering (no Key Lookup).

**Tradeoff:** The covering index adds TotalAmount to the leaf level, increasing index width by ~9 bytes (DECIMAL(10,2) = 9 bytes). For a 5M row table, the index grows by ~45MB.

</details>

**Challenge 4 — Diagnose the concurrency problem**

A Dapper query with ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC) runs in the background service. During execution, it holds Shared (S) locks on the Orders table for 20 seconds. INSERT queries into Orders start to block after 5 seconds, timing out after 30 seconds.

<details>
<summary>Solution</summary>

**Root cause:** The window function's Sort operator holds S locks on pages as it scans. The Sort is a blocking operator — it reads all qualifying pages and holds the S locks until the Sort completes and the connection is closed. Concurrent INSERTs need IX (Intent Exclusive) locks, which are incompatible with S locks.

**Detection query:**
```sql
SELECT
    blocking_session_id,
    wait_type,
    wait_time,
    wait_resource
FROM sys.dm_exec_requests
WHERE blocking_session_id > 0;
```

**Fix 1 — READ UNCOMMITTED (if dirty reads acceptable):**
```csharp
using var tx = connection.BeginTransaction(IsolationLevel.ReadUncommitted);
var results = await connection.QueryAsync<OrderRanking>(sql, transaction: tx);
```

**Fix 2 — SNAPSHOT ISOLATION LEVEL (recommended):**
```sql
ALTER DATABASE YourDb SET ALLOW_SNAPSHOT_ISOLATION ON;
```
```csharp
using var tx = connection.BeginTransaction(IsolationLevel.Snapshot);
var results = await connection.QueryAsync<OrderRanking>(sql, transaction: tx);
```

**Fix 3 — ROWLOCK hint on the query:**
```sql
SELECT ... FROM Orders WITH (ROWLOCK, READPAST)
```

But READPAST skips locked rows silently — not suitable for analytics.

**Fix 4 — Reduce execution time with a covering index (shortens lock duration).**

**In .NET:**
```csharp
// Use snapshot isolation via transaction
await using var connection = _connectionFactory.Create();
await connection.OpenAsync(ct);
using var transaction = connection.BeginTransaction(IsolationLevel.Snapshot);
var results = await connection.QueryAsync<OrderRanking>(
    new CommandDefinition(sql, transaction: transaction, cancellationToken: ct));
transaction.Commit();
```

**Challenge 5 — Design the Dapper data access layer**

**Scenario:** You are building a high-throughput (.NET 9) API that serves a sales dashboard for 500 concurrent users. The API needs:
- Top 10 products per category (cacheable for 5 minutes)
- Running total per order for a given customer (user-specific, not cached)
- Paginated order list with total count
- Year-over-year comparison (monthly)

All queries require window functions. The database is SQL Server 2022 on a 16-core, 64GB RAM VM. Design the Dapper data access layer.

<details>
<summary>Solution</summary>

```csharp
// Architecture: Repository pattern with Dapper, connection pooling
// Caching: IDistributedCache for stale-tolerant analytics

public interface ISalesAnalyticsRepository
{
    Task<IReadOnlyList<TopProductDto>> GetTopProductsPerCategoryAsync(
        int topN, CancellationToken ct);

    Task<IReadOnlyList<CustomerRunningTotalDto>> GetCustomerRunningTotalAsync(
        int customerId, CancellationToken ct);

    Task<PagedOrderResult> GetPagedOrdersAsync(
        int page, int pageSize, CancellationToken ct);

    Task<IReadOnlyList<YearOverYearDto>> GetYearOverYearAsync(CancellationToken ct);
}

public class SalesAnalyticsRepository : ISalesAnalyticsRepository
{
    private readonly IDapperConnectionFactory _connectionFactory;
    private readonly IDistributedCache _cache;

    public SalesAnalyticsRepository(
        IDapperConnectionFactory connectionFactory,
        IDistributedCache cache)
    {
        _connectionFactory = connectionFactory;
        _cache = cache;
    }

    // Cache stale-tolerant analytics (top products per category)
    public async Task<IReadOnlyList<TopProductDto>> GetTopProductsPerCategoryAsync(
        int topN, CancellationToken ct)
    {
        var cacheKey = $"top_products_{topN}";
        var cached = await _cache.GetAsync(cacheKey, ct);
        if (cached is not null)
            return Deserialize<List<TopProductDto>>(cached);

        const string sql = @"
            SELECT CategoryId, ProductId, TotalSales, RowNum
            FROM (
                SELECT
                    p.CategoryId,
                    p.ProductId,
                    SUM(oi.Quantity * oi.UnitPrice) AS TotalSales,
                    ROW_NUMBER() OVER(
                        PARTITION BY p.CategoryId
                        ORDER BY SUM(oi.Quantity * oi.UnitPrice) DESC
                    ) AS RowNum
                FROM OrderItems oi
                INNER JOIN Products p ON oi.ProductId = p.ProductId
                GROUP BY p.CategoryId, p.ProductId
            ) AS RankedProducts
            WHERE RowNum <= @TopN
            ORDER BY CategoryId, RowNum";

        await using var connection = _connectionFactory.Create();
        var results = await connection.QueryAsync<TopProductDto>(
            new CommandDefinition(sql, new { TopN = topN }, cancellationToken: ct));

        var list = results.AsList();
        await _cache.SetAsync(cacheKey, Serialize(list),
            new DistributedCacheEntryOptions
            {
                AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5)
            }, ct);
        return list;
    }

    // User-specific, no caching
    public async Task<IReadOnlyList<CustomerRunningTotalDto>> GetCustomerRunningTotalAsync(
        int customerId, CancellationToken ct)
    {
        const string sql = @"
            SELECT OrderId, OrderDate, TotalAmount,
                SUM(TotalAmount) OVER(
                    PARTITION BY CustomerId
                    ORDER BY OrderDate, OrderId
                    ROWS UNBOUNDED PRECEDING
                ) AS RunningTotal
            FROM Orders
            WHERE CustomerId = @CustomerId
            ORDER BY OrderDate, OrderId";

        await using var connection = _connectionFactory.Create();
        var results = await connection.QueryAsync<CustomerRunningTotalDto>(
            new CommandDefinition(sql, new { CustomerId = customerId },
                cancellationToken: ct));
        return results.AsList();
    }

    // Paginated with total count — single round trip
    public async Task<PagedOrderResult> GetPagedOrdersAsync(
        int page, int pageSize, CancellationToken ct)
    {
        const string sql = @"
            SELECT OrderId, CustomerId, OrderDate, TotalAmount, Status, RowNum
            FROM (
                SELECT OrderId, CustomerId, OrderDate, TotalAmount, Status,
                    ROW_NUMBER() OVER(
                        ORDER BY OrderDate DESC, OrderId DESC
                    ) AS RowNum
                FROM Orders
            ) AS RankedOrders
            WHERE RowNum BETWEEN @Offset + 1 AND @Offset + @PageSize
            ORDER BY RowNum;

            SELECT COUNT(*) FROM Orders;";

        await using var connection = _connectionFactory.Create();
        await using var grid = await connection.QueryMultipleAsync(
            new CommandDefinition(sql,
                new { Offset = (page - 1) * pageSize, PageSize = pageSize },
                cancellationToken: ct));

        var items = (await grid.ReadAsync<OrderListItem>()).AsList();
        var totalCount = await grid.ReadSingleAsync<int>();

        return new PagedOrderResult(items, totalCount, page, pageSize);
    }

    // Year-over-year — monthly, infrequent, no caching needed
    public async Task<IReadOnlyList<YearOverYearDto>> GetYearOverYearAsync(
        CancellationToken ct)
    {
        const string sql = @"
            WITH MonthlySales AS (
                SELECT YEAR(OrderDate) AS SalesYear,
                    MONTH(OrderDate) AS SalesMonth,
                    SUM(TotalAmount) AS TotalSales
                FROM Orders
                GROUP BY YEAR(OrderDate), MONTH(OrderDate)
            )
            SELECT SalesYear, SalesMonth, TotalSales,
                LAG(TotalSales) OVER(
                    PARTITION BY SalesMonth
                    ORDER BY SalesYear
                ) AS PreviousYearSales,
                (TotalSales - LAG(TotalSales) OVER(
                    PARTITION BY SalesMonth ORDER BY SalesYear
                )) * 1.0 / NULLIF(LAG(TotalSales) OVER(
                    PARTITION BY SalesMonth ORDER BY SalesYear
                ), 0) * 100 AS YoYGrowthPct
            FROM MonthlySales
            ORDER BY SalesYear, SalesMonth";

        await using var connection = _connectionFactory.Create();
        var results = await connection.QueryAsync<YearOverYearDto>(
            new CommandDefinition(sql, cancellationToken: ct));
        return results.AsList();
    }
}

// Output DTOs
public record TopProductDto(
    int CategoryId, int ProductId, decimal TotalSales, long RowNum);

public record CustomerRunningTotalDto(
    int OrderId, DateTime OrderDate, decimal TotalAmount, decimal RunningTotal);

public record OrderListItem(
    int OrderId, int CustomerId, DateTime OrderDate,
    decimal TotalAmount, string Status, long RowNum);

public record YearOverYearDto(
    int SalesYear, int SalesMonth, decimal TotalSales,
    decimal? PreviousYearSales, decimal? YoYGrowthPct);

public record PagedOrderResult(
    IReadOnlyList<OrderListItem> Items, int TotalCount,
    int Page, int PageSize);
```

**Index strategy:**
```sql
-- For top products per category (covers the GROUP BY + PARTITION BY)
CREATE INDEX IX_Products_CategoryId ON Products(CategoryId);

-- For running total (covers PARTITION BY + ORDER BY + WHERE)
CREATE INDEX IX_Orders_CustomerId_OrderDate
    ON Orders(CustomerId, OrderDate, OrderId)
    INCLUDE (TotalAmount);

-- For pagination (covers ORDER BY)
CREATE INDEX IX_Orders_OrderDate_OrderId
    ON Orders(OrderDate DESC, OrderId DESC)
    INCLUDE (CustomerId, TotalAmount, Status);
```
