### Where This Fits

The ORDER BY clause inside OVER() is what makes window functions dynamic — it defines the logical order of rows within each partition for ranking, running aggregates, and offset access. Without ORDER BY, ranking functions are non-deterministic and aggregate functions return the entire partition's value on every row (no running behavior). A .NET backend engineer encounters ORDER BY within OVER when writing pagination queries (`ROW_NUMBER() OVER(ORDER BY Id)`), running totals (`SUM() OVER(ORDER BY Date)`), or top-N per group queries. The most important and most misunderstood aspect is the **default frame**: when ORDER BY is present, aggregate window functions use a default frame of `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`, which produces a running aggregate — but `RANGE` (vs `ROWS`) has specific semantics for ties and duplicate values. For ranking functions like ROW_NUMBER(), ORDER BY is required and must be unique for deterministic results. Interviewers frequently ask: "What's the default frame when ORDER BY is present? What's the difference between ROWS and RANGE for the default frame? How does ORDER BY affect performance?"

## Core Mental Model

ORDER BY inside the OVER() clause specifies the logical order of rows within each window partition for the purpose of computing the window function. It does NOT affect the query's overall ORDER BY — the window's ORDER BY is independent of the query's ORDER BY. The presence or absence of ORDER BY changes the default frame: without ORDER BY, the window is the entire partition (equivalent to `ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING`); with ORDER BY, the default frame becomes `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`, which includes all rows from the start of the partition through the current row's peer group (including ties). This default frame difference is the single most common source of confusion: developers who use `SUM() OVER(PARTITION BY CustomerId ORDER BY OrderDate)` intending a partition total get a running total instead. For ranking functions, ORDER BY is mandatory — `ROW_NUMBER() OVER(PARTITION BY CustomerId)` without ORDER BY is valid but non-deterministic. For offset functions (LAG, LEAD), ORDER BY defines which row is "previous" or "next." The Sort operator in the execution plan is driven entirely by the ORDER BY (and PARTITION BY) columns in the OVER clause — the Sort orders the input by these columns so the downstream Segment and Sequence Project operators can process rows in order.

### Classification

**For SQL topics:** ORDER BY within OVER is part of the OLAP window specification (ANSI SQL:2003). It is not SARGable (it doesn't filter rows). The Sort operator it requires is the dominant cost of most window function queries — typically 60-75% of query cost. The ORDER BY columns in the window specification should match the trailing columns of an index whose leading columns match the PARTITION BY keys. Multiple ORDER BY columns are supported and have multi-level sorting behavior. ASC/DESC is supported and affects both the window computation (reversed ranking/reversed running total) and the required sort order. The `NULLS FIRST / NULLS LAST` option (SQL Server uses `ASC` = NULLs first by default) affects which rows get ROW_NUMBER=1.

## Deep Mechanics

### How the Engine Executes This

The ORDER BY within OVER drives the execution plan's Sort operator. Here is the step-by-step execution:

1. **The query's FROM/WHERE/GROUP BY/HAVING clauses are evaluated**, producing the intermediate result set that will be input to the window function stage.

2. **The Sort operator orders the intermediate result set** by the PARTITION BY columns (if present) followed by the OVER() ORDER BY columns. For `OVER(PARTITION BY DeptId ORDER BY Salary DESC)`, the Sort orders by (DeptId ASC, Salary DESC). The Sort is necessary because the Segment operator (partition detection) needs rows ordered by partition key, and the Sequence Project needs rows ordered by the window ORDER BY within each partition.

3. **The Segment operator reads the sorted rows** and detects partition boundaries by comparing the PARTITION BY column values between consecutive rows. Each boundary signals a reset.

4. **The Sequence Project (or Window Aggregate) operator** computes the window function. For aggregate functions with ORDER BY present, it maintains a running aggregate. The key behavior difference depending on default frame:
   - **Default RANGE frame**: When ORDER BY values have ties (duplicate values), the running aggregate includes ALL rows with the same ORDER BY value as peer rows. This means if three rows have the same SaleDate, the running SUM on the third row includes all three rows (not just the first and second). The RANGE frame adds rows by logical position — all rows with the same ORDER BY value are peers and are included together.
   - **ROWS frame**: When explicitly specified, ROWS uses physical offset. `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` includes exactly 1 row for each row processed — ties are NOT grouped. If three rows have the same SaleDate, the running SUM increments one row at a time.

5. **The Window Spool operator** (for LAG/LEAD only) buffers rows to allow backward/forward access. The number of buffered rows depends on the LAG/LEAD offset parameter.

**Default frame behavior demonstration:**

```sql
-- Sample data: two rows with same OrderDate
INSERT INTO Orders VALUES (1, '2025-01-01', 100.00);
INSERT INTO Orders VALUES (1, '2025-01-01', 50.00);  -- same date!

-- Default RANGE frame: SUM(TotalAmount) OVER(PARTITION BY CustomerId ORDER BY OrderDate)
-- Row 1 (100.00 on 2025-01-01): running sum = 150 (includes both rows! because RANGE includes ties)
-- Row 2 (50.00 on 2025-01-01): running sum = 150 (same — both rows are peers)

-- ROWS frame: SUM(TotalAmount) OVER(PARTITION BY CustomerId ORDER BY OrderDate
--                                ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
-- Row 1 (100.00 on 2025-01-01): running sum = 100
-- Row 2 (50.00 on 2025-01-01): running sum = 150
```

```sql
-- ============================================================
-- Setup
-- ============================================================
CREATE TABLE dbo.Transactions (
    TransactionId  INT            NOT NULL IDENTITY(1,1),
    AccountId      INT            NOT NULL,
    TransactionDate DATE          NOT NULL,
    Amount         DECIMAL(18,2)  NOT NULL,
    TransactionType VARCHAR(20)   NOT NULL,
    CONSTRAINT PK_Transactions PRIMARY KEY CLUSTERED (TransactionId)
);

INSERT INTO dbo.Transactions (AccountId, TransactionDate, Amount, TransactionType)
VALUES
    (1, '2025-01-01',  100.00, 'Deposit'),
    (1, '2025-01-01',   50.00, 'Withdrawal'),  -- same date as above — ties
    (1, '2025-01-15',  200.00, 'Deposit'),
    (1, '2025-02-01',   75.00, 'Withdrawal'),
    (1, '2025-02-01',  125.00, 'Deposit'),     -- same date — ties
    (2, '2025-01-05', 1000.00, 'Deposit'),
    (2, '2025-01-20',  500.00, 'Withdrawal'),
    (2, '2025-02-10',  300.00, 'Deposit'),
    (2, '2025-03-01',  200.00, 'Deposit');

-- ============================================================
-- ORDER BY absent vs present — default frame difference
-- ============================================================

-- Without ORDER BY: partition total (same on all rows in partition)
SELECT
    t.TransactionId,
    t.AccountId,
    t.TransactionDate,
    t.Amount,
    SUM(t.Amount) OVER(PARTITION BY t.AccountId) AS AccountTotal
FROM dbo.Transactions AS t;
-- Account 1: AccountTotal = 550 for all 5 rows
-- Account 2: AccountTotal = 2000 for all 4 rows

-- With ORDER BY: running total (increases per row)
SELECT
    t.TransactionId,
    t.AccountId,
    t.TransactionDate,
    t.Amount,
    SUM(t.Amount) OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate
    ) AS RunningTotal
FROM dbo.Transactions AS t;
-- Account 1: running total by date — first row shows 150 (100+50),
-- because RANGE default includes both rows with same TransactionDate!

-- ============================================================
-- ROWS vs RANGE with ties (duplicate ORDER BY values)
-- ============================================================

-- Default RANGE frame (includes ties as peers)
SELECT
    t.TransactionId,
    t.AccountId,
    t.TransactionDate,
    t.Amount,
    SUM(t.Amount) OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate
        -- default: RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal_RANGE
FROM dbo.Transactions AS t;
-- 2025-01-01: RunningTotal_RANGE = 150 for BOTH rows (ties grouped)
-- 2025-01-15: RunningTotal_RANGE = 350 (150 + 200)
-- 2025-02-01: RunningTotal_RANGE = 550 for BOTH rows (ties grouped: 350 + 75 + 125)

-- Explicit ROWS frame (physical rows, one at a time)
SELECT
    t.TransactionId,
    t.AccountId,
    t.TransactionDate,
    t.Amount,
    SUM(t.Amount) OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal_ROWS
FROM dbo.Transactions AS t;
-- 2025-01-01: Row 1 = 100, Row 2 = 150 (increments one row at a time)
-- 2025-01-15: RunningTotal_ROWS = 350
-- 2025-02-01: Row 4 = 425, Row 5 = 550 (increments one row at a time)

-- ============================================================
-- ORDER BY with ASC vs DESC
-- ============================================================

-- ASC: earliest transaction first, running total grows forward
SELECT
    t.TransactionId,
    t.AccountId,
    t.TransactionDate,
    t.Amount,
    SUM(t.Amount) OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate ASC
    ) AS RunningTotal_ASC
FROM dbo.Transactions AS t;

-- DESC: latest transaction first, running total grows backward
SELECT
    t.TransactionId,
    t.AccountId,
    t.TransactionDate,
    t.Amount,
    SUM(t.Amount) OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate DESC
    ) AS RunningTotal_DESC
FROM dbo.Transactions AS t;
-- Latest transactions get lower running totals (they are "first" in the window order)

-- ============================================================
-- Multiple ORDER BY columns
-- ============================================================

SELECT
    t.TransactionId,
    t.AccountId,
    t.TransactionDate,
    t.Amount,
    t.TransactionType,
    ROW_NUMBER() OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate, t.TransactionType, t.TransactionId
    ) AS RowNum,
    SUM(t.Amount) OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate, t.TransactionId
    ) AS RunningTotal
FROM dbo.Transactions AS t;
-- Both ORDER BY columns affect the ordering within each partition.
-- The Sort operator sorts by (AccountId, TransactionDate, TransactionId).

-- ============================================================
-- ORDER BY with ranking functions — determinism
-- ============================================================

-- Without ORDER BY: non-deterministic
SELECT
    t.TransactionId,
    t.AccountId,
    t.Amount,
    ROW_NUMBER() OVER(PARTITION BY t.AccountId) AS RowNum
    -- Which row gets RowNum = 1? SQL Server decides based on physical order.
FROM dbo.Transactions AS t;

-- With unique ORDER BY: deterministic
SELECT
    t.TransactionId,
    t.AccountId,
    t.Amount,
    ROW_NUMBER() OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionId  -- Unique column ensures determinism
    ) AS RowNum
FROM dbo.Transactions AS t;

-- ============================================================
-- ORDER BY with RANK() and DENSE_RANK() — ties behavior
-- ============================================================

SELECT
    t.TransactionId,
    t.AccountId,
    t.Amount,
    RANK() OVER(
        PARTITION BY t.AccountId
        ORDER BY t.Amount DESC
    ) AS AmountRank_RANK,
    DENSE_RANK() OVER(
        PARTITION BY t.AccountId
        ORDER BY t.Amount DESC
    ) AS AmountRank_DENSE
FROM dbo.Transactions AS t;
-- If two rows have same Amount, RANK gives same value but next rank has gap.
-- DENSE_RANK gives same value but next rank has NO gap.

-- ============================================================
-- ORDER BY with offset functions — defines "previous" and "next"
-- ============================================================

SELECT
    t.TransactionId,
    t.AccountId,
    t.TransactionDate,
    t.Amount,
    LAG(t.Amount, 1, 0.00) OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate, t.TransactionId
    ) AS PreviousTransactionAmount,
    LEAD(t.Amount, 1, 0.00) OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate, t.TransactionId
    ) AS NextTransactionAmount
FROM dbo.Transactions AS t;
-- "Previous" = the row that comes before the current row in the ORDER BY sequence.
-- "Next" = the row that comes after.

-- ============================================================
-- ORDER BY with FIRST_VALUE and LAST_VALUE
-- ============================================================

-- FIRST_VALUE: always the first row in the partition ORDER BY
SELECT
    t.TransactionId,
    t.AccountId,
    t.TransactionDate,
    t.Amount,
    FIRST_VALUE(t.Amount) OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate
    ) AS FirstTransactionAmount,
    LAST_VALUE(t.Amount) OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate
        -- default frame makes LAST_VALUE return current row!
    ) AS LastTransactionAmount_Wrong,
    LAST_VALUE(t.Amount) OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS LastTransactionAmount_Correct
FROM dbo.Transactions AS t;
```

### Execution Plan Analysis

**Query with ORDER BY within OVER (running total):**

```sql
SELECT
    t.TransactionId,
    t.AccountId,
    t.TransactionDate,
    t.Amount,
    SUM(t.Amount) OVER(
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate
    ) AS RunningTotal
FROM dbo.Transactions AS t;
```

**Expected execution plan:**

```
Clustered Index Scan (Transactions)
  → Sort (Order by: AccountId ASC, TransactionDate ASC)
      Estimated cost: ~68%
  → Segment (Partition by: AccountId)
      Estimated cost: ~2%
  → Sequence Project (Compute running SUM)
      Estimated cost: ~10%
  → SELECT
```

**Operator breakdown:**

1. **Clustered Index Scan** — Reads all 9 rows from Transactions. 1 logical read for this small table.

2. **Sort** — The dominant operator (~68% of query cost). Sorts 9 rows by AccountId (partition key) then TransactionDate (ORDER BY key). For the 9-row example, the sort completes in milliseconds. For 10M rows, the sort requires a memory grant proportional to row count × row size. If the memory grant is insufficient, the sort spills to TempDB, causing severe performance degradation.

3. **Segment** — Detects AccountId boundaries. When AccountId changes from 1 to 2, signals a partition reset. Cheap — ~2% of cost.

4. **Sequence Project** — Computes the running SUM of Amount. Since rows arrive sorted by TransactionDate within each AccountId partition, the Sequence Project can incrementally accumulate the sum. When Segment signals a partition boundary, the accumulator resets to 0.

**Same query without ORDER BY in OVER:**

```
Clustered Index Scan (Transactions)
  → Sequence Project (Compute partition SUM)
      → SELECT
```

No Sort, no Segment needed. The Sequence Project computes SUM over all rows in each AccountId group (but since ORDER BY is absent, the partition total is the same on every row — no running behavior).

**Same query with ORDER BY + ROWS frame:**

```
Clustered Index Scan (Transactions)
  → Sort (Order by: AccountId ASC, TransactionDate ASC)
  → Segment
  → Sequence Project (Compute running SUM with ROWS semantics)
      → SELECT
```

The plan shape is identical to the RANGE frame version. The difference is in the Sequence Project's behavior: with ROWS, each row adds its Amount to the accumulator independently (ties are NOT grouped). With RANGE, ties (same TransactionDate) are processed as peer groups — all rows with the same TransactionDate are added to the accumulator before any of them are output.

**Index that eliminates the Sort:**

```sql
CREATE INDEX IX_Transactions_AccountId_TransactionDate
ON dbo.Transactions (AccountId, TransactionDate)
INCLUDE (Amount, TransactionType);
```

Plan becomes:

```
Index Scan (ordered: AccountId ASC, TransactionDate ASC)
  → Segment
  → Sequence Project
      → SELECT
```

The Sort operator is eliminated because the index provides the required ordering. The Index Scan is typically more expensive than a Clustered Index Scan (non-clustered indexes have leaf-level pages that may need to be read), but the elimination of the Sort more than compensates.

### Failure Modes

**1. RANGE default frame causing unexpected running totals:** The most common window function bug. Adding ORDER BY to an aggregate window function changes it from a partition total to a running total. Developers who add ORDER BY for "ordering" (not realizing it changes semantics) get wrong results.

**2. RANGE default frame with ties — peer grouping:** With RANGE, all rows with the same ORDER BY value are peers. `SUM() OVER(ORDER BY Date)` with two rows on the same date will include both rows in the SUM for both rows. This is surprising to developers who expect row-by-row incrementing.

**3. Non-deterministic ROW_NUMBER without ORDER BY:** `ROW_NUMBER() OVER(PARTITION BY CustomerId)` is valid but non-deterministic. The physical order of rows in the partition (allocation order, index order) determines the numbering.

**4. ORDER BY non-unique column for ranking functions:** `ROW_NUMBER() OVER(ORDER BY Amount)` where Amount has ties gives non-deterministic results for the tied rows. Add a tiebreaker column (e.g., TransactionId) for determinism.

**5. ORDER BY with LAST_VALUE default frame:** `LAST_VALUE(x) OVER(ORDER BY y)` returns the current row's x because the default frame ends at CURRENT ROW. To get the last value in the partition, specify `ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING`.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Schema context
-- ============================================================
CREATE TABLE dbo.Inventory (
    InventoryId   INT            NOT NULL IDENTITY(1,1),
    ProductId     INT            NOT NULL,
    WarehouseId   INT            NOT NULL,
    ChangeDate    DATETIME2(0)   NOT NULL,
    QuantityChange INT           NOT NULL,  -- positive for inbound, negative for outbound
    RunningQuantity AS SUM(QuantityChange) OVER(
        PARTITION BY ProductId, WarehouseId
        ORDER BY ChangeDate, InventoryId
    ) PERSISTED -- ERROR: cannot use window functions in computed column definition
    CONSTRAINT PK_Inventory PRIMARY KEY CLUSTERED (InventoryId)
);

INSERT INTO dbo.Inventory (ProductId, WarehouseId, ChangeDate, QuantityChange)
VALUES
    (1, 100, '2025-01-01',  100),  -- inbound
    (1, 100, '2025-01-05',  -20),  -- outbound
    (1, 100, '2025-01-10',  -15),  -- outbound
    (1, 200, '2025-01-02',  200),  -- inbound (different warehouse)
    (1, 200, '2025-01-08',  -50),  -- outbound
    (2, 100, '2025-01-03',   50),  -- inbound
    (2, 100, '2025-01-07',   30),  -- inbound
    (2, 100, '2025-01-12',  -10);  -- outbound

CREATE INDEX IX_Inventory_ProductId_WarehouseId_ChangeDate
ON dbo.Inventory (ProductId, WarehouseId, ChangeDate, InventoryId)
INCLUDE (QuantityChange);

-- ============================================================
-- Pattern 1: Running inventory balance per product per warehouse
-- ============================================================
SELECT
    i.InventoryId,
    i.ProductId,
    i.WarehouseId,
    i.ChangeDate,
    i.QuantityChange,
    SUM(i.QuantityChange) OVER(
        PARTITION BY i.ProductId, i.WarehouseId
        ORDER BY i.ChangeDate, i.InventoryId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningBalance
FROM dbo.Inventory AS i
ORDER BY i.ProductId, i.WarehouseId, i.ChangeDate, i.InventoryId;

-- ============================================================
-- Pattern 2: Moving average (3-row window)
-- ============================================================
-- 8.XXX: [[8.156 — AVG() OVER() — Moving Averages]]
SELECT
    i.InventoryId,
    i.ProductId,
    i.WarehouseId,
    i.ChangeDate,
    i.QuantityChange,
    AVG(CAST(i.QuantityChange AS DECIMAL(10,2))) OVER(
        PARTITION BY i.ProductId, i.WarehouseId
        ORDER BY i.ChangeDate, i.InventoryId
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS MovingAvg3Rows,
    SUM(i.QuantityChange) OVER(
        PARTITION BY i.ProductId, i.WarehouseId
        ORDER BY i.ChangeDate, i.InventoryId
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS MovingSum3Rows
FROM dbo.Inventory AS i;

-- ============================================================
-- Pattern 3: Year-over-year comparison with LAG + ORDER BY
-- ============================================================
-- 8.XXX: [[8.166 — Year-over-Year Comparison with LAG]]
SELECT
    t.AccountId,
    YEAR(t.TransactionDate) AS TranYear,
    MONTH(t.TransactionDate) AS TranMonth,
    SUM(t.Amount) AS MonthlyTotal,
    LAG(SUM(t.Amount), 1, 0.00) OVER(
        PARTITION BY t.AccountId, MONTH(t.TransactionDate)
        ORDER BY YEAR(t.TransactionDate)
    ) AS PreviousYearSameMonth,
    SUM(t.Amount) - LAG(SUM(t.Amount), 1, 0.00) OVER(
        PARTITION BY t.AccountId, MONTH(t.TransactionDate)
        ORDER BY YEAR(t.TransactionDate)
    ) AS YearOverYearChange
FROM dbo.Transactions AS t
GROUP BY t.AccountId, YEAR(t.TransactionDate), MONTH(t.TransactionDate)
ORDER BY t.AccountId, TranYear, TranMonth;
-- Note: Window function applied AFTER GROUP BY — partitions by (AccountId, Month)
-- and orders by Year within each partition.

-- ============================================================
-- Pattern 4: ASC vs DESC for running totals
-- ============================================================
-- Forward running total (ASC)
SELECT
    i.InventoryId,
    i.ProductId,
    i.WarehouseId,
    i.ChangeDate,
    i.QuantityChange,
    SUM(i.QuantityChange) OVER(
        PARTITION BY i.ProductId, i.WarehouseId
        ORDER BY i.ChangeDate ASC, i.InventoryId ASC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS ForwardRunningBalance
FROM dbo.Inventory AS i;

-- Backward running total (DESC) — sum remaining after this transaction
SELECT
    i.InventoryId,
    i.ProductId,
    i.WarehouseId,
    i.ChangeDate,
    i.QuantityChange,
    SUM(i.QuantityChange) OVER(
        PARTITION BY i.ProductId, i.WarehouseId
        ORDER BY i.ChangeDate DESC, i.InventoryId DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS BackwardRunningBalance
FROM dbo.Inventory AS i;

-- ============================================================
-- Pattern 5: Multiple ORDER BY columns for deterministic ranking
-- ============================================================
-- Ensure deterministic row numbering even with ties
SELECT
    t.TransactionId,
    t.AccountId,
    t.Amount,
    ROW_NUMBER() OVER(
        ORDER BY t.Amount DESC, t.TransactionDate, t.TransactionId
    ) AS DeterministicRank
FROM dbo.Transactions AS t;
-- Amount DESC is the primary sort.
-- TransactionDate breaks ties for same Amount.
-- TransactionId breaks ties for same Amount + same TransactionDate.

-- ============================================================
-- Pattern 6: ORDER BY with NULL handling
-- ============================================================
-- Control NULL placement in window ordering
SELECT
    e.EmployeeId,
    e.FirstName + ' ' + e.LastName AS EmployeeName,
    e.ManagerId,
    e.Salary,
    ROW_NUMBER() OVER(
        ORDER BY
            CASE WHEN e.ManagerId IS NULL THEN 0 ELSE 1 END,
            e.Salary DESC
    ) AS RowNum
    -- NULL ManagerId first, then by salary descending
FROM dbo.Employees AS e;
```

## Gotchas and Production Pitfalls

### Default RANGE Frame Causes Unexpected Running Total with Ties

**Pitfall:** Using ORDER BY in an aggregate window function without specifying a frame, expecting row-by-row incrementing, but getting peer-group incrementing from the default RANGE frame.

```sql
-- ❌ WRONG assumption: running total increments one row at a time
-- The two transactions on 2025-01-01 are peers (RANGE default)
SELECT AccountId, TransactionDate, Amount,
       SUM(Amount) OVER(
           PARTITION BY AccountId
           ORDER BY TransactionDate
       ) AS RunningTotal
FROM dbo.Transactions;
-- Row 1 (100.00, 2025-01-01): RunningTotal = 150 ← both rows included!
-- Row 2 (50.00, 2025-01-01): RunningTotal = 150 ← same value

-- ✅ CORRECT: Specify ROWS frame for physical row-by-row
SELECT AccountId, TransactionDate, Amount,
       SUM(Amount) OVER(
           PARTITION BY AccountId
           ORDER BY TransactionDate
           ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS RunningTotal
FROM dbo.Transactions;
-- Row 1 (100.00): RunningTotal = 100
-- Row 2 (50.00): RunningTotal = 150
```

**Symptom:** A financial report shows different sums for two rows with the same date (if someone believes they should be different) or the same sum (which is correct for RANGE but the developer expected row-by-row). A QA test with distinct dates passes, but a test with duplicate dates fails.

**Fix:** Specify `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` for physical row-by-row running totals. Use `RANGE` only when peer grouping is intended (e.g., "all sales on the same day should be grouped").

**Cost of not fixing:** An inventory balance query shows incorrect running balances when multiple inventory changes occur on the same day. The warehouse manager reports discrepancies of thousands of dollars. Investigation reveals that two inbound shipments on the same date were both credited with the total of both shipments.

---

### Non-Deterministic ROW_NUMBER Without Unique ORDER BY

**Pitfall:** Using `ROW_NUMBER() OVER(ORDER BY NonUniqueColumn)` where the ORDER BY column has duplicate values, making the row numbering for tied rows non-deterministic.

```sql
-- ❌ WRONG: Amount has duplicates — which row gets RN=1?
SELECT TransactionId, AccountId, Amount,
       ROW_NUMBER() OVER(ORDER BY Amount DESC) AS RowNum
FROM dbo.Transactions;
-- If two transactions have Amount=200.00, either can get RowNum=2 or 3
-- The result depends on physical order (allocation order, page splits)

-- ✅ CORRECT: Add tiebreaker to ORDER BY
SELECT TransactionId, AccountId, Amount,
       ROW_NUMBER() OVER(
           ORDER BY Amount DESC, TransactionId
       ) AS RowNum
FROM dbo.Transactions;
```

**Symptom:** A pagination API returns different row ordering on successive calls for the same data. An ETL job using ROW_NUMBER for deduplication deletes different rows on each run.

**Fix:** Add a unique column (typically the primary key) as the last ORDER BY column. This ensures determinism without changing the primary sort order.

**Cost of not fixing:** A customer-facing paginated "top transactions" list reorders itself on page refresh. Customers report "the list keeps changing order." Support spends 3 days investigating before discovering the missing tiebreaker.

---

### ORDER BY in OVER Does NOT Affect Query ORDER BY

**Pitfall:** Assuming that ORDER BY within OVER controls the query's output order. The window's ORDER BY is independent of the query's ORDER BY.

```sql
-- ❌ WRONG: Expecting output sorted by TransactionDate
SELECT AccountId, TransactionDate, Amount,
       ROW_NUMBER() OVER(ORDER BY TransactionDate) AS RowNum
FROM dbo.Transactions;
-- Output ORDER is NOT guaranteed to be by TransactionDate!
-- The window ORDER BY affects ROW_NUMBER computation only.

-- ✅ CORRECT: Add query-level ORDER BY for output sorting
SELECT AccountId, TransactionDate, Amount,
       ROW_NUMBER() OVER(ORDER BY TransactionDate) AS RowNum
FROM dbo.Transactions
ORDER BY TransactionDate;
```

**Symptom:** Query output appears in random order. The developer assumed the window ORDER BY would sort the result set.

**Fix:** Add a separate ORDER BY at the query level. Note that SQL Server may optimize the plan to avoid a second Sort if the query ORDER BY matches the window ORDER BY, but this is not guaranteed.

**Cost of not fixing:** A report lists transactions in insertion order instead of date order. The first page shows the most recent transactions (if they were inserted last) but the business expects chronological order.

---

### Multiple Window Functions with Different ORDER BY — Redundant Sorts

**Pitfall:** Using multiple window functions with different ORDER BY clauses, causing the optimizer to add multiple Sort operators.

```sql
-- ❌ Multiple Sort operators (potentially)
SELECT
    TransactionId,
    AccountId,
    TransactionDate,
    Amount,
    ROW_NUMBER() OVER(ORDER BY Amount DESC) AS AmountRank,
    RANK() OVER(ORDER BY TransactionDate) AS DateRank,
    SUM(Amount) OVER(ORDER BY TransactionDate) AS RunningTotal
FROM dbo.Transactions;
-- If Amount DESC and TransactionDate are different orders,
-- the optimizer may need two separate Sort operations.
```

**Symptom:** The execution plan shows two Sort operators, doubling the Sort cost. The query runs slower than expected.

**Fix:** Align ORDER BY clauses where possible. If the same ORDER BY can serve multiple window functions, the optimizer can share one Sort:

```sql
-- ✅ Optimizer can share one Sort if ORDER BY matches
SELECT
    TransactionId,
    AccountId,
    TransactionDate,
    Amount,
    ROW_NUMBER() OVER(ORDER BY TransactionDate) AS DateRank,
    SUM(Amount) OVER(ORDER BY TransactionDate) AS RunningTotal
FROM dbo.Transactions;
-- One Sort (by TransactionDate) serves both window functions.
```

**Cost of not fixing:** A dashboard with 5 window functions on 10M rows shows 3 Sort operators in the plan. The query takes 12 seconds instead of 4 seconds.

---

### ORDER BY With Aggregate Window Function Changes Cardinality Intuition

**Pitfall:** Expecting an aggregate window function with ORDER BY to produce the same result as the equivalent without ORDER BY.

```sql
-- Without ORDER BY: partition total
SELECT AVG(Amount) OVER(PARTITION BY AccountId) AS AccountAvg FROM Transactions;
-- All rows for AccountId 1 show AccountAvg = 110 (550/5)

-- With ORDER BY: running average (changes per row)
SELECT AVG(Amount) OVER(PARTITION BY AccountId ORDER BY TransactionDate) AS RunningAvg
FROM Transactions;
-- Row 1: Avg = 100.00 (100/1)
-- Row 2: Avg = 75.00 (150/2) — includes both rows on 2025-01-01!
-- Row 3: Avg = 116.67 (350/3)
```

**Symptom:** A developer uses a window aggregate with ORDER BY expecting a single partition-level average but gets per-row varying values. They add ORDER BY thinking "I just want the results ordered" but accidentally change the aggregate semantics.

**Fix:** Always think: "Does this window function need ORDER BY for its semantics?" If you want a partition-level aggregate (same value for all rows), omit ORDER BY. If you want a running aggregate, add ORDER BY and explicitly choose ROWS vs RANGE.

**Cost of not fixing:** An analyst reports "average order value" as 75 (running average at row 2) instead of 110 (actual partition average). Business decisions about pricing are based on the wrong number.

### Spoken Answers

**Q1: What is the default frame for an aggregate window function when ORDER BY is present vs absent?**

> **Average answer:** Without ORDER BY, the frame is the whole partition. With ORDER BY, the default frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW.

> **Great answer:** When ORDER BY is absent from the OVER clause, the default window frame is the entire partition — equivalent to `ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING`. The aggregate function computes one value for the entire partition and returns that same value for every row. When ORDER BY is present, the default frame changes to `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`. This subtle but critical change means the aggregate becomes a running aggregate — it computes a value that includes all rows from the start of the partition through the current row's peer group. The word "RANGE" is critical: RANGE includes all rows with the same ORDER BY value as peers. If three rows share the same OrderDate, the running SUM on the first of those rows already includes all three, because RANGE groups by logical order value. This is different from ROWS, which applies the frame as physical offset — one row at a time. This default frame behavior is the single most common source of window function bugs in production. Developers adding ORDER BY to "sort the output" accidentally change a partition total into a running total with peer grouping. The fix is either: (1) omit ORDER BY if you want a partition total, or (2) explicitly specify `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` if you want physical row-by-row running behavior.

**Q2: What is the difference between ROWS and RANGE for the default frame when ORDER BY is present?**

> **Average answer:** ROWS uses physical row positions, RANGE uses logical order values and includes ties.

> **Great answer:** ROWS and RANGE differ in how they determine which rows belong to the frame. ROWS uses physical position: the frame includes exactly N rows before/after the current row based on the row's ordinal position in the ordered partition. ROWS BETWEEN 3 PRECEDING AND CURRENT ROW includes exactly 4 rows regardless of their values. RANGE uses logical order: the frame includes all rows whose ORDER BY value falls within a range relative to the current row's ORDER BY value. RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW includes every row whose ORDER BY value is less than or equal to the current row's ORDER BY value. This means if three rows have the same TransactionDate (the ORDER BY column), all three are peers and are included in the frame for any of those three rows. The practical consequence: RANGE with ties produces "jumps" in running totals where a group of rows with the same ORDER BY value all show the same running total (the sum including all of them). ROWS produces smooth row-by-row incrementing. RANGE is more expensive than ROWS because the engine must track the logical ORDER BY value boundaries and may need to spool rows. For ordered-unique columns (like Identity or DateTime2 with millisecond precision), ROWS and RANGE produce identical results because there are no ties.

**Q5: What execution plan operator does ORDER BY within OVER require, and how would you eliminate it?**

> **Average answer:** It requires a Sort operator. You eliminate it by creating an index on the ORDER BY columns.

> **Great answer:** The ORDER BY within OVER requires a Sort operator in the execution plan — this is typically 60-75% of total query cost for large tables. The Sort orders the input rows by the combination of PARTITION BY columns (if present) followed by the OVER() ORDER BY columns. To eliminate the Sort, you need a non-clustered index with leading columns matching the PARTITION BY columns and trailing columns matching the ORDER BY columns, with the ORDER BY sort direction matching the index sort direction. For `SUM(Amount) OVER(PARTITION BY AccountId ORDER BY TransactionDate DESC)`, the index should be on `(AccountId, TransactionDate DESC) INCLUDE (Amount)`. When this index exists, the optimizer can scan it in order — the rows arrive at the Segment operator already sorted by AccountId (for partition detection) and TransactionDate DESC (for window ORDER BY). The Sort operator is eliminated entirely, and the plan becomes: Index Scan (ordered) → Segment → Sequence Project. The Index Scan replaces the Clustered Index Scan + Sort, and because the non-clustered index is typically narrower than the clustered index, the I/O cost is also reduced. The tradeoff is write amplification: each INSERT/UPDATE to the base table must also maintain this non-clustered index.

|---|---|---|
| Default frame (aggregate) | `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` | Entire partition |
| Function result | Running aggregate | Partition aggregate |
| Sort required | Yes | No (unless PARTITION BY needs it) |
| Ranking functions | Required for determinism | Non-deterministic |
| Offset functions (LAG/LEAD) | Required — defines previous/next | Not allowed — must have ORDER BY |
| ROWS vs RANGE distinction | Critical — RANGE groups ties | No distinction (entire partition) |
| Performance | Sort is dominant cost (~70%) | Minimal overhead |

## Decision Framework

### When to Apply

```mermaid
flowchart TD
    A[Window function needed] --> B{ORDER BY in OVER?}
    B -->|No| C[Partition total / count / average]
    B -->|Yes| D{What function family?}
    D -->|Ranking| E[ORDER BY required for determinism]
    D -->|Aggregate| F[Running aggregate triggered]
    D -->|Offset| G[Previous/next row defined]
    E --> H[Add unique tiebreaker column]
    F --> I{Default RANGE frame OK?}
    I -->|Yes — ties should be peers| J[Use default — no frame needed]
    I -->|No — need row-by-row| K[Explicit ROWS frame]
    G --> L[Order defines what "previous" and "next" mean]
    H --> M[Deterministic ranking]
    J --> N[Running aggregate with peer grouping]
    K --> O[Running aggregate — physical rows]
    L --> P[LAG/LEAD with correct ORDER BY]
    M --> Q[Index on ORDER BY columns eliminates Sort]
    N --> Q
    O --> Q
    P --> Q
```

### Application Checklist

- [ ] The ORDER BY within OVER correctly defines the intended order for the window function
- [ ] The default frame (RANGE) is either intended OR explicitly overridden with ROWS
- [ ] Ranking functions have a deterministic ORDER BY (unique tiebreaker column included)
- [ ] The window ORDER BY is independent of the query ORDER BY (both specified if needed)
- [ ] Multiple window functions are aligned to share a Sort operator where possible
- [ ] A covering index on (PARTITION BY columns, ORDER BY columns) exists
- [ ] The Sort memory grant is sufficient for the expected row count

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Running total / moving average without self-join|Sort operator added (~60-75% query cost)|
|Deterministic ranking with tiebreaker|Additional sort key increases Sort complexity|
|Flexible RANGE vs ROWS semantics|RANGE is more expensive than ROWS for peer groups|
|Independent ordering per window function|Multiple Sorts if ORDER BY clauses differ|

### Scale Thresholds

- "Sort cost becomes measurable above ~10K rows" (under 10K, Sort completes in <1ms)
- "Sort becomes dominant cost above ~100K rows" (expect ~70% of query time in Sort)
- "Covering index eliminates Sort at any scale — create when query runs > 100x/hour"
- "RANGE frame with ties on 1M+ row partitions can cause significant CPU overhead"
- "Multiple different ORDER BY windows above 1M rows should be consolidated or indexed"

## Self-Check

### Conceptual Questions

1. What is the default frame for `SUM(x) OVER(ORDER BY y)`? How does it differ from `SUM(x) OVER()`?
2. What is the difference between ROWS and RANGE for `BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`?
3. How do you check the Sort operator's memory grant and spill status for a window function query?
4. What happens if you use `ROW_NUMBER() OVER(PARTITION BY DeptId ORDER BY Salary)` and two employees have the same salary?
5. Does EF Core support ORDER BY within OVER in its LINQ expressions? What happens if you try?
6. How would you use Dapper to map a query with `SUM() OVER(PARTITION BY DeptId ORDER BY HireDate)`?
7. What is the difference between `ORDER BY Amount DESC` in the window vs the query level?
8. At what row count does the Sort operator become the dominant cost in a window function query?
9. What index on the Transactions table eliminates the Sort for `SUM(Amount) OVER(PARTITION BY AccountId ORDER BY TransactionDate)`?
10. Explain in 60 seconds why ORDER BY within OVER changes a partition aggregate into a running aggregate.

<details>
<summary>Answers</summary>

1. `SUM(x) OVER(ORDER BY y)` has default frame `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` — a running total that includes peer rows with the same y value. `SUM(x) OVER()` has the entire result set as the window — the grand total on every row.

2. ROWS uses physical row offsets: each row adds one unit to the frame. RANGE uses logical ORDER BY values: rows with the same ORDER BY value are peers and are all included or excluded together. With unique ORDER BY values, they produce identical results. With ties (duplicate ORDER BY values), ROWS increments per row while RANGE groups peers.

3. Query `sys.dm_exec_query_stats` for the query: examine `estimated_grant_kb`, `grant_kb`, `used_grant_kb`, and `ideal_grant_kb`. The wait stat `SORT_HASH_EVENT` indicates sort spills. `RESOURCE_SEMAPHORE` indicates memory pressure from the sort grant. Use `SET STATISTICS TIME ON` to see CPU vs elapsed time (large discrepancy indicates I/O-bound sort spills).

4. ROW_NUMBER gives consecutive numbers to tied rows, but which tied row gets which number is non-deterministic. The two employees with same salary will get different row numbers but which employee gets which number depends on physical order. To make it deterministic, add a tiebreaker: `ORDER BY Salary DESC, EmployeeId`.

5. No, EF Core does not support ORDER BY within OVER. LINQ OrderBy translates to query-level ORDER BY, not window ORDER BY. EF Core GroupBy with Sum translates to SQL GROUP BY, not SUM() OVER(). Raw SQL via FromSql is the only option.

6. ``var results = await connection.QueryAsync<Result>(new CommandDefinition(@"SELECT EmployeeId, DepartmentId, HireDate, Salary, SUM(Salary) OVER(PARTITION BY DepartmentId ORDER BY HireDate) AS RunningDeptSalary FROM Employees ORDER BY DepartmentId, HireDate", cancellationToken: ct));`` Dapper maps the RunningDeptSalary column to the corresponding property.

7. ORDER BY within OVER defines the order for window function computation (ranking, running total, previous/next row). Query-level ORDER BY defines the output row order. They are independent — the window ORDER BY does NOT guarantee output order, and the query ORDER BY does NOT affect window computation. They can be the same or different.

8. The Sort becomes the dominant cost (~60-75% of query) above approximately 100K rows. Below 10K rows, the Sort completes in <1ms and is negligible. Between 10K-100K, the Sort cost grows linearly with row count. Above 1M rows, the Sort memory grant may be large enough to cause memory pressure.

9. A non-clustered index on `(AccountId, TransactionDate) INCLUDE (Amount, TransactionType)`. The leading column (AccountId) supports the PARTITION BY, the second column (TransactionDate) supports the ORDER BY, and the INCLUDE columns cover the aggregate computation.

10. In the SQL logical processing order, window functions are computed at step 6. When ORDER BY is absent, the window frame defaults to the entire partition — the aggregate computes one value over all rows and returns it on every row. When ORDER BY is present, the SQL standard defines the default frame as `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`, which includes only rows from the start of the partition through the current row's peer group. This creates a running aggregate because each subsequent row (or peer group) adds its value to the accumulator. The ORDER BY clause signals to the query processor "this window function should be computed incrementally as rows are processed in order."

</details>

### Query Challenges

**Challenge 1 — Write the SQL**

The inventory team needs a daily report that tracks inventory levels for each product in each warehouse. For each inventory change, show: (a) the running balance after this change, (b) a 3-transaction moving average of QuantityChange, (c) the difference between the current QuantityChange and the previous change. Use the Inventory table with columns ProductId, WarehouseId, ChangeDate, QuantityChange. Assume multiple changes can occur on the same date — use InventoryId as tiebreaker.

<details>
<summary>Solution</summary>

```sql
SELECT
    i.InventoryId,
    i.ProductId,
    i.WarehouseId,
    i.ChangeDate,
    i.QuantityChange,
    -- Running balance
    SUM(i.QuantityChange) OVER(
        PARTITION BY i.ProductId, i.WarehouseId
        ORDER BY i.ChangeDate, i.InventoryId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningBalance,
    -- 3-transaction moving average
    AVG(CAST(i.QuantityChange AS DECIMAL(10,2))) OVER(
        PARTITION BY i.ProductId, i.WarehouseId
        ORDER BY i.ChangeDate, i.InventoryId
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS MovingAvg3,
    -- Difference from previous change
    i.QuantityChange - LAG(i.QuantityChange, 1, 0) OVER(
        PARTITION BY i.ProductId, i.WarehouseId
        ORDER BY i.ChangeDate, i.InventoryId
    ) AS ChangeFromPrevious
FROM dbo.Inventory AS i
ORDER BY i.ProductId, i.WarehouseId, i.ChangeDate, i.InventoryId;
```

**Logical reads:** ~1 (full scan of small table) **Execution plan:** Clustered Index Scan → Sort → Segment → Sequence Project (3 window functions share Sort) → SELECT

</details>

**Challenge 2 — Fix the performance problem**

```sql
-- This query computes a running total of sales by date.
-- It runs in 22 seconds on a 5M row Sales table.
SELECT
    s.SaleDate,
    s.Amount,
    (
        SELECT SUM(s2.Amount)
        FROM dbo.Sales AS s2
        WHERE s2.SaleDate <= s.SaleDate
    ) AS RunningTotal
FROM dbo.Sales AS s
ORDER BY s.SaleDate;
-- SET STATISTICS IO: logical reads = 3,200,000
```

<details> <summary>Solution</summary>

**Root cause:** Correlated subquery creates O(n²) behavior. For each of 5M rows, the subquery scans all rows with SaleDate <= current row's SaleDate. This is effectively a triangular join.

```sql
-- Fixed query using window function with ORDER BY
SELECT
    s.SaleDate,
    s.Amount,
    SUM(s.Amount) OVER(
        ORDER BY s.SaleDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Sales AS s
ORDER BY s.SaleDate;
```

**Index to create:**

```sql
CREATE INDEX IX_Sales_SaleDate
ON dbo.Sales (SaleDate) INCLUDE (Amount);
```

**After fix — logical reads:** ~18,000 (from 3,200,000) — 178x reduction.

The window function scans the Sales table once (sorted by SaleDate using the index or a Sort). The correlated subquery scans the Sales table repeatedly — approximately (5M × 2.5M average) = 12.5T row comparisons. The window function does O(n log n) due to the Sort, compared to O(n²) for the self-join.

</details>

**Challenge 3 — Explain the execution plan**

```sql
-- Given this query and execution plan:
SELECT
    TransactionId, AccountId, TransactionDate, Amount,
    SUM(Amount) OVER(
        PARTITION BY AccountId
        ORDER BY TransactionDate
    ) AS RunningTotal
FROM dbo.Transactions;

-- Plan fragment:
--   |--Sequence Project(DEFINE:([Expr1004]=SUM([Amount])))
--       |--Segment(SEGMENT:([AccountId]))
--           |--Sort(ORDER BY:([AccountId] ASC, [TransactionDate] ASC))
--               |--Clustered Index Scan(OBJECT:([Transactions].[PK_Transactions]))
-- Estimated costs: Sort = 72%, Clustered Index Scan = 18%, Segment = 2%, Sequence Project = 8%
-- Output rows: 1,000,000
```

Why does the Sequence Project compute the correct running total without seeing a ROWS/RANGE specification? The query does not specify ROWS or RANGE — what is the default frame, and why does the plan not need a Window Spool operator for this aggregate window function?

<details> <summary>Solution</summary>

**Why no ROWS/RANGE needed:** The query uses `SUM(Amount)` with ORDER BY but no explicit frame. The default frame is `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`. However, since TransactionDate is not guaranteed unique (there can be multiple transactions on the same date), the RANGE frame groups peer rows. If TransactionDate has ties, the running total for the first row in a tie group includes all rows in that tie group.

**Why no Window Spool:** SUM() is an aggregate window function that can be computed incrementally — the Sequence Project operator maintains a running accumulator. It does NOT need to buffer rows for backward access (unlike LAG/LEAD which need a Window Spool). The accumulator resets when the Segment operator signals a new partition. The Sequence Project simply reads each row, adds its Amount to the accumulator, and outputs the accumulator value. This is O(1) memory per partition — just the accumulator value.

**What would change with LAG() instead of SUM():**
If the query used `LAG(Amount, 1) OVER(PARTITION BY AccountId ORDER BY TransactionDate)`, the plan would include a **Window Spool** operator between Segment and Sequence Project. The Window Spool buffers the previous row's values to allow backward access. For LAG with offset 1, the spool needs to hold only 1 row per partition, so spool overhead is minimal. For LAG with offset 100, the spool may need to hold 100 rows per partition and could spill to TempDB if memory is insufficient.

**Default frame behavior on this data:**
If Transaction has two rows on 2025-01-01 for AccountId 1:
- Row 1 (100.00): RunningTotal = 100 + 50 = 150 (both peers already included!)
- Row 2 (50.00): RunningTotal = 150 (same — peer group already complete)

This is the RANGE default frame behavior with ties. To avoid this peer grouping, the query should specify:

```sql
SUM(Amount) OVER(
    PARTITION BY AccountId
    ORDER BY TransactionDate, TransactionId
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
) AS RunningTotal
```

Adding TransactionId to ORDER BY makes the ordering unique, so ROWS and RANGE produce identical results (no ties).

</details>

**Challenge 4 — Diagnose the concurrency problem**

A financial reporting application runs the following query every hour:

```sql
SELECT
    AccountId,
    TransactionDate,
    Amount,
    SUM(Amount) OVER(
        PARTITION BY AccountId
        ORDER BY TransactionDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningBalance
FROM dbo.Transactions
ORDER BY AccountId, TransactionDate;
```

During the hour-end reporting run, a `RESOURCE_SEMAPHORE` wait type appears. The query waits for 30+ seconds before executing. The Transactions table has 10M rows. The server has 64GB RAM. The query runs under `READ COMMITTED` isolation level. What is causing the wait and how do you fix it?

<details> <summary>Solution</summary>

**Root cause:** The Sort operator requires a memory grant to sort 10M rows by (AccountId, TransactionDate). The estimated memory grant for this sort is approximately 60-80MB (10M rows × ~8 bytes per sort key). Under `RESOURCE_SEMAPHORE`, the query waits for this memory grant to become available. If the server has many concurrent queries with large memory grants, the 80MB grant may be gated by the "resource semaphore" — SQL Server limits total memory grants to avoid exhausting buffer pool memory. The query cannot start executing until the memory grant is available.

**Detection query:**

```sql
SELECT
    session_id,
    wait_type,
    wait_duration_ms,
    grant_time,
    requested_memory_kb,
    granted_memory_kb,
    required_memory_kb,
    used_memory_kb,
    max_used_memory_kb,
    is_small
FROM sys.dm_exec_query_memory_grants
WHERE session_id > 50;
```

**Fix:**

1. **Create covering index to eliminate Sort:** `CREATE INDEX IX_Transactions_AccountId_Date ON dbo.Transactions (AccountId, TransactionDate) INCLUDE (Amount);` With this index, the Sort operator is eliminated, and the memory grant drops from 80MB to near zero.

2. **Increase `max degree of parallelism` memory:** If the Sort uses parallelism, each parallel worker gets a memory grant. Reducing DOP for this query can reduce total grant.

3. **Use `OPTION (MIN_GRANT_PERCENT = 10)` hint:** Reduce the required memory grant (risk: the sort spills if grant is too small).

4. **Schedule the report during low-concurrency periods:** If the memory grant cannot be eliminated, run the report when fewer concurrent queries compete for memory.

**Recommended fix:** Create the covering index. The query plan changes from:
- `Clustered Index Scan → Sort (memory grant: 80MB) → Segment → Sequence Project`
to:
- `Index Scan (ordered) → Segment → Sequence Project (memory grant: 0MB)`

The `RESOURCE_SEMAPHORE` wait disappears because the query no longer requests a large memory grant.

</details>

**Challenge 5 — Design the index**

**Scenario:** A banking application runs the following transaction history query frequently:

```sql
SELECT
    AccountId,
    TransactionDate,
    Reference,
    Amount,
    SUM(Amount) OVER(
        PARTITION BY AccountId
        ORDER BY TransactionDate DESC, TransactionId DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningBalance
FROM dbo.Transactions
WHERE AccountId = @AccountId
  AND TransactionDate >= @StartDate
ORDER BY TransactionDate DESC, TransactionId DESC;
```

The Transactions table has 100M rows across 1M accounts (100 transactions per account average). The WHERE clause filters to approximately 5,000 rows per query (one account, last 3 months). The query currently performs a clustered index scan + Sort and takes 8 seconds for a single account. The read/write ratio is 90/10 (mostly reads, few inserts). Design the optimal index strategy.

<details> <summary>Solution</summary>

```sql
-- Primary index: seek for AccountId + Date filter
CREATE INDEX IX_Transactions_AccountId_Date
ON dbo.Transactions (AccountId, TransactionDate DESC, TransactionId DESC)
INCLUDE (Reference, Amount);
```

**Why this index eliminates both the scan and the Sort:**

1. **Leading columns (AccountId, TransactionDate DESC, TransactionId DESC)** — The WHERE clause filters by `AccountId = @AccountId AND TransactionDate >= @StartDate`. The index supports a seek on AccountId, then a seek on TransactionDate to the @StartDate boundary. Only the 5,000 relevant rows are read. The index is already sorted in DESC order matching the window's ORDER BY, so no Sort is needed.

2. **INCLUDE (Reference, Amount)** — The SELECT list and the SUM aggregate need Reference and Amount. With INCLUDE, the index is covering — no key lookups to the clustered index are required.

3. **Leading with AccountId** — Since the query always filters by AccountId (WHERE AccountId = @AccountId), making AccountId the leading column enables a highly selective seek (1 account out of 1M = 0.0001% of rows). The DESC order on TransactionDate matches the window ORDER BY, eliminating the Sort.

**Execution plan before:**

```
Clustered Index Scan (100M rows) → Sort (AccountId, TransactionDate DESC, TransactionId DESC)
  → Segment → Sequence Project → Filter (AccountId + Date) → SELECT
-- Logical reads: ~450,000
-- Elapsed: ~8s
```

**Execution plan after:**

```
Index Seek (5,000 rows) → Segment → Sequence Project → SELECT
-- Logical reads: ~20 (seek for one account + INCLUDE columns)
-- Elapsed: ~5ms (1600x faster)
```

**Tradeoffs:**

|What You Gain|What You Pay|
|---|---|
|1600x read performance (8s → 5ms)|Additional write cost: INSERT must update this non-clustered index|
|Sort eliminated (no memory grant needed)|Index storage: ~1.2GB for 100M rows|
|Covering index (no key lookups)|Index must be rebuilt with correct DESC order|
|Highly selective seek (0.0001% of rows)|Maintenance overhead for index fragmentation|

**Write amplification:** Each INSERT to Transactions must write to:
- Clustered PK index (1 write)
- This non-clustered index (1 write)

Total: 2 index writes per INSERT (acceptable for 90/10 read/write ratio).

Index fragmentation: Since TransactionId is ever-increasing, inserts are mostly appends at the right edge of the index. Fragmentation is minimal. Reorganize weekly with `ALTER INDEX IX_Transactions_AccountId_Date ON Transactions REORGANIZE;`

</details>
