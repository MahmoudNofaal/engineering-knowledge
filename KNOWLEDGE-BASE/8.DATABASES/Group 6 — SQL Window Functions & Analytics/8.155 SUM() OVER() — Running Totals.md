### Where This Fits

SUM(col) OVER(ORDER BY col) is the most commonly used window aggregate — it computes a running total (cumulative sum) within a partition. Every .NET backend engineer encounters SUM() OVER() when building financial reports (running account balance), inventory analysis (cumulative stock movements), or any ordered accumulation (cumulative revenue by month). The three distinct patterns are: SUM() OVER(ORDER BY) for running totals, SUM() OVER(PARTITION BY) for per-partition totals repeated on every row, and SUM() OVER() for grand total on every row. The most expensive mistake made here is the default RANGE frame: SUM() OVER(ORDER BY col) defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, which with non-unique ORDER BY includes all peers (ties) in the running total, not just up to the current row. This produces unexpectedly large running totals when ties exist. Interviewers use SUM() OVER() to test whether the candidate understands the three distinct OVER patterns, the default frame behavior, and when to use ROWS vs RANGE.

## Core Mental Model

SUM(col) OVER(...) applies the SUM aggregation over a window defined by PARTITION BY, ORDER BY, and frame specification. Three distinct behaviors arise from the combination:

1. **SUM(col) OVER()** — No PARTITION BY, no ORDER BY. The entire rowset is one window. Every row gets the grand total of col. This is equivalent to SUM(col) FROM the entire table, repeated on every row.

2. **SUM(col) OVER(PARTITION BY p)** — Partitioned, no ORDER BY. Each row gets the total of col within its partition. Every row in the same partition gets the same value. This is equivalent to SUM(col) GROUP BY p, joined back to every row.

3. **SUM(col) OVER(ORDER BY o)** — Ordered, no explicit frame. The default frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. This computes a running total: the sum of col from the first row of the partition up to (and including) the current row's peers (for RANGE) or the exact current row (for ROWS). This is the running total pattern.

The engine evaluates SUM() OVER() after the FROM/WHERE/GROUP BY/HAVING phase. For running totals (pattern 3), the engine sorts by the ORDER BY columns, then scans the sorted rows sequentially, accumulating the running total. The accumulation is a streaming operator: it does not require a Window Spool (unlike LEAD/LAG/FIRST_VALUE) because it only needs the current running total value, not access to other rows. This makes SUM() OVER(ORDER BY) one of the most efficient window functions.

### Classification

SUM() OVER() is a **window aggregate function** — it takes the familiar SUM aggregate and applies it over a window frame. The query optimizer can use either a Stream Aggregate (if input is sorted) or a Window Aggregate operator. The running total pattern (SUM OVER ORDER BY) can be computed with minimal memory — just one accumulator per partition — making it more memory-efficient than positional window functions that need spools. SUM() OVER() is not SARGable — it operates on rows after the WHERE and JOIN phases. The three patterns (grand total, partition total, running total) have different execution plan shapes and performance characteristics.

## Deep Mechanics

### How the Engine Executes This

#### Pattern 1: SUM(col) OVER() — Grand Total

1. The engine computes the total SUM of col across all rows that survive the WHERE/JOIN/GROUP BY filtering.
2. This total is computed by a Stream Aggregate or Hash Aggregate operator (same as a regular SUM without GROUP BY).
3. The total is then repeated on every row in the result set. This is implemented as a Compute Scalar that reads the aggregate value and outputs it for each row.
4. No sort is needed. No spool is needed. The engine just keeps a running accumulator.

#### Pattern 2: SUM(col) OVER(PARTITION BY p) — Partition Total

1. The engine partitions the rows by the PARTITION BY columns. If the input is not already ordered by those columns, a Sort operator is needed.
2. For each partition, a Stream Aggregate computes the SUM of col for that partition.
3. The partition total is then repeated for every row in the partition. The Segment operator detects partition boundaries.
4. This is equivalent to: SUM(col) GROUP BY p, then JOIN back to all rows. But the window function does it in one pass.

#### Pattern 3: SUM(col) OVER(ORDER BY o) — Running Total

1. The engine sorts the rows by the ORDER BY columns within each partition (or globally if no PARTITION BY).
2. The Window Aggregate operator scans the sorted rows sequentially.
3. For each row, it evaluates the window frame (default: RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW).
4. With the default RANGE frame, the engine adds the value of col for all rows from the start of the partition up to and including all peers of the current row's ORDER BY value.
5. With ROWS frame, the engine adds from the start up to exactly the current row position.
6. The accumulator is a single variable per partition — no spool needed. This makes SUM OVER ORDER BY much more memory-efficient than LAG/LEAD or FIRST_VALUE, which require spools.

**Key performance insight:** SUM() OVER(ORDER BY) uses a STREAMING accumulator. It does NOT need a Window Spool (unless the optimizer chooses to spool for complex plans). The accumulator is maintained in memory as the scan progresses. This means the memory grant is dominated by the Sort operator, not by the window function itself.

### Execution Plan Analysis

#### Pattern 1: SUM() OVER() — Grand Total

```
Expected plan shape:
[Clustered Index Scan / Index Seek]
→ [Stream Aggregate (compute SUM of col)]
→ [Compute Scalar (repeat grand total on every row)]
→ [SELECT]

Estimated Cost: Scan 80%, Aggregate 15%, Compute Scalar 5%
Logical Reads: ~128,342 (scan)
```

- The Stream Aggregate computes the total in a single pass. No sort needed.
- The Compute Scalar repeats the same constant value on every output row.

#### Pattern 2: SUM() OVER(PARTITION BY p) — Partition Total

```
Expected plan shape:
[Clustered Index Scan / Index Seek]
→ [Sort (by partition column)] — if not already ordered
→ [Segment (partition boundaries)]
→ [Stream Aggregate (compute SUM per partition)]
→ [Window Aggregate (repeat per-row)]
→ [SELECT]

Estimated Cost: Scan 30%, Sort 50%, Segment 5%, Aggregate 10%, Window Aggregate 5%
```

- With a covering index on the partition column, the Sort is eliminated.
- The Segment operator detects when the partition column changes, resetting the aggregate.

#### Pattern 3: SUM() OVER(ORDER BY o) — Running Total

```
Expected plan shape:
[Clustered Index Scan / Index Seek]
→ [Sort (by ORDER BY columns)]
→ [Window Aggregate (running sum accumulator)]
→ [SELECT]

Estimated Cost: Scan 20%, Sort 60%, Window Aggregate 20%
Logical Reads: ~128,342 (scan) + 0 (no spool)
```

- The Window Aggregate uses a streaming accumulator — it does NOT use a Window Spool. This is the critical difference from LEAD/LAG/FIRST_VALUE. The accumulator is a single variable.
- For each row in sorted order: `running_total = running_total + current_row_value`.
- At partition boundaries (if PARTITION BY is present), the accumulator resets to 0.
- The Sort is the dominant cost (60%+). With a covering index on the ORDER BY columns, the Sort is eliminated.
- No TempDB spool is needed (unless the Sort spills).

### Failure Modes

**Failure Mode 1: Default RANGE frame with non-unique ORDER BY — inflated running total.**

```sql
-- ❌ WRONG: Default RANGE frame includes all peers (ties)
SELECT
    o.CustomerId,
    o.OrderDate,
    SUM(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        -- Default: RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o;

-- ✅ Correct: ROWS frame with unique ORDER BY
SELECT
    o.CustomerId,
    o.OrderDate,
    SUM(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o;
```

**Symptom:** When multiple orders have the same OrderDate, the running total for those rows includes ALL peers, not just the rows up to and including the current row. The running total for row 1 and row 2 (same date) are identical and include both amounts.

**Fix:** Specify ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW and add a unique tiebreaker to ORDER BY.

**Cost of not fixing:** Inflated running totals. Financial reports show incorrect cumulative balances. The running total jumps by multiple rows' values at each tie boundary.

**Failure Mode 2: Self-join for running total — catastrophic performance.**

```sql
-- ❌ WRONG: Self-join approach
SELECT
    o1.CustomerId,
    o1.OrderId,
    SUM(o2.TotalAmount) AS RunningTotal
FROM dbo.Orders o1
INNER JOIN dbo.Orders o2
    ON o1.CustomerId = o2.CustomerId
    AND o2.OrderDate <= o1.OrderDate
GROUP BY o1.CustomerId, o1.OrderId, o1.OrderDate;

-- ✅ Correct: SUM OVER ORDER BY
SELECT
    o.CustomerId,
    o.OrderId,
    SUM(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o;
```

**Symptom:** The self-join runs for minutes on 1M rows. The execution plan shows two scans of the Orders table + a complex join + GROUP BY. This is 100-1000x slower than SUM OVER ORDER BY.

**Fix:** Always use SUM OVER ORDER BY for running totals. Never use the self-join approach.

**Cost of not fixing:** 100-1000x slower queries. Query timeouts on large tables.

**Failure Mode 3: Overflow — SUM exceeds data type range.**

```sql
-- ❌ WRONG: DECIMAL(18,2) may overflow for large cumulative sums
SELECT
    o.OrderId,
    SUM(o.TotalAmount) OVER (
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o;
-- Error: Arithmetic overflow error converting numeric to data type decimal(18,2)

-- ✅ Correct: Use larger data type
SELECT
    o.OrderId,
    SUM(CAST(o.TotalAmount AS DECIMAL(38,2))) OVER (
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o;
```

**Symptom:** "Arithmetic overflow error" when the running total exceeds the data type's maximum value. DECIMAL(18,2) max is ~10^16. A running total of 10M rows × $10^6 each overflows.

**Fix:** Use DECIMAL(38,2) for the accumulator, or use CONVERT to a larger type.

**Cost of not fixing:** Query failure at runtime. The overflow occurs at an unpredictable row count.

**Failure Mode 4: SUM OVER ORDER BY without PARTITION BY — global running total when per-group intended.**

```sql
-- Probably wrong: Global running total (no PARTITION BY)
SELECT
    o.CustomerId,
    o.OrderDate,
    SUM(o.TotalAmount) OVER (
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS GlobalRunningTotal
FROM dbo.Orders o;
-- Total accumulates across ALL customers, not per customer

-- Usually intended: Per-customer running total
SELECT
    o.CustomerId,
    o.OrderDate,
    SUM(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS CustomerRunningTotal
FROM dbo.Orders o;
```

**Symptom:** The running total is much larger than expected because it accumulates across all customers instead of resetting per customer.

**Fix:** Always consider whether PARTITION BY is needed. For per-customer running totals, include PARTITION BY.

**Cost of not fixing:** Wrong running totals. Financial reports show cumulative revenue across all customers instead of per customer.

**Failure Mode 5: SUM OVER(ORDER BY col DESC) — descending running total.**

```sql
-- Descending running total (accumulates from the end)
SELECT
    o.OrderId,
    o.OrderDate,
    o.TotalAmount,
    SUM(o.TotalAmount) OVER (
        ORDER BY o.OrderDate DESC, o.OrderId DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS DescendingRunningTotal
FROM dbo.Orders o
ORDER BY o.OrderDate;
-- The running total accumulates from the most recent order backward
-- This is a valid but often surprising pattern
```

**Symptom:** The engineer expects a forward-running total but the accumulator starts from the end of the ordering.

**Fix:** Always order ASC for forward-running totals unless the DESC behavior is explicitly intended.

**Cost of not fixing:** Confusion during debugging. The running total decreases as the query progresses forward.

```sql
    SELECT
        CAST(o.OrderDate AS DATE) AS OrderDay,
        SUM(o.TotalAmount) AS DayRevenue
    FROM dbo.Orders o
    WHERE o.Status NOT IN (5, 6) -- Exclude cancelled/returned
    GROUP BY CAST(o.OrderDate AS DATE)
)
SELECT
    OrderDay,
    DayRevenue,
    SUM(DayRevenue) OVER (
        ORDER BY OrderDay
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS CumulativeRevenue,
    SUM(DayRevenue) OVER (
        ORDER BY OrderDay
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS Trailing7DayRevenue
FROM DailyRevenue
ORDER BY OrderDay;

-- Pattern 2: Customer running balance with financial transactions
SELECT
    t.TransactionId,
    t.AccountId,
    t.TransactionDate,
    t.Amount,
    t.TransactionType,
    t.Description,
    SUM(CASE
        WHEN t.TransactionType = 'Deposit' THEN t.Amount
        WHEN t.TransactionType IN ('Withdrawal', 'Fee') THEN -t.Amount
        ELSE 0
    END) OVER (
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate, t.TransactionId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningBalance
FROM dbo.Transactions t
ORDER BY t.AccountId, t.TransactionDate, t.TransactionId;

-- Pattern 3: Running total per customer with % of total
WITH CustomerStats AS (
    SELECT
        o.CustomerId,
        o.OrderId,
        o.OrderDate,
        o.TotalAmount,
        SUM(o.TotalAmount) OVER (
            PARTITION BY o.CustomerId
            ORDER BY o.OrderDate, o.OrderId
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) AS RunningTotal,
        SUM(o.TotalAmount) OVER (
            PARTITION BY o.CustomerId
        ) AS CustomerTotal
    FROM dbo.Orders o
)
SELECT
    CustomerId,
    OrderId,
    OrderDate,
    TotalAmount,
    RunningTotal,
    CustomerTotal,
    CAST(RunningTotal * 100.0 / NULLIF(CustomerTotal, 0) AS DECIMAL(5,1)) AS PctOfCustomerRunning,
    CAST(RunningTotal * 100.0 / NULLIF(SUM(CustomerTotal) OVER(), 0) AS DECIMAL(5,2)) AS PctOfOverall
FROM CustomerStats
ORDER BY CustomerId, OrderDate;

-- Pattern 4: Inventory running quantity
SELECT
    i.ProductId,
    i.WarehouseId,
    i.LastUpdated,
    i.QuantityOnHand AS ChangeAmount,
    SUM(i.QuantityOnHand) OVER (
        PARTITION BY i.ProductId, i.WarehouseId
        ORDER BY i.LastUpdated
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningQuantity
FROM dbo.InventoryItems i
ORDER BY i.ProductId, i.WarehouseId, i.LastUpdated;

-- Pattern 5: Running total with 3-frame average
SELECT
    o.OrderId,
    o.OrderDate,
    o.TotalAmount,
    SUM(o.TotalAmount) OVER (
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal,
    AVG(o.TotalAmount) OVER (
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS MovingAvg3Orders,
    COUNT(*) OVER (
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS MovingCount3
FROM dbo.Orders o
ORDER BY o.OrderDate, o.OrderId;

-- Pattern 6: Running total with percentage of grand total (pareto)
WITH OrderValues AS (
    SELECT
        o.OrderId,
        o.TotalAmount,
        SUM(o.TotalAmount) OVER() AS GrandTotal
    FROM dbo.Orders o
)
SELECT
    OrderId,
    TotalAmount,
    SUM(TotalAmount) OVER (
        ORDER BY TotalAmount DESC, OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS CumulativeTotal,
    GrandTotal,
    CAST(SUM(TotalAmount) OVER (
        ORDER BY TotalAmount DESC, OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) * 100.0 / NULLIF(GrandTotal, 0) AS DECIMAL(5,2)) AS CumulativePct
FROM OrderValues
ORDER BY TotalAmount DESC, OrderId;
-- This is a Pareto analysis: what % of total revenue comes from top N orders

-- Pattern 7: Running total reset per month (compound running total)
SELECT
    o.CustomerId,
    YEAR(o.OrderDate) AS OrderYear,
    MONTH(o.OrderDate) AS OrderMonth,
    SUM(o.TotalAmount) AS MonthlySpend,
    SUM(SUM(o.TotalAmount)) OVER (
        PARTITION BY o.CustomerId, YEAR(o.OrderDate)
        ORDER BY MONTH(o.OrderDate)
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningYearTotal,
    SUM(SUM(o.TotalAmount)) OVER (
        PARTITION BY o.CustomerId
        ORDER BY YEAR(o.OrderDate), MONTH(o.OrderDate)
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotalAllTime
FROM dbo.Orders o
GROUP BY o.CustomerId, YEAR(o.OrderDate), MONTH(o.OrderDate)
ORDER BY o.CustomerId, OrderYear, OrderMonth;

-- Pattern 8: Running total with overflow-safe accumulator
SELECT
    o.OrderId,
    o.OrderDate,
    o.TotalAmount,
    SUM(CONVERT(DECIMAL(38,2), o.TotalAmount)) OVER (
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotalSafe
FROM dbo.Orders o
ORDER BY o.OrderDate, o.OrderId;
```

```sql
-- PostgreSQL SUM OVER syntax is identical

-- Both databases default to RANGE — same tie-related pitfalls

-- PostgreSQL supports GROUPS frame mode (SQL Server 2022+ does too):
SUM(amount) OVER (ORDER BY date GROUPS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)

-- PostgreSQL supports EXCLUDE CURRENT ROW (SQL Server does not):
-- PostgreSQL:

-- SQL Server: must use ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING instead
```

## Gotchas and Production Pitfalls

### 1. Default RANGE Frame with Ties Inflates Running Total

**Pitfall:** Using SUM OVER ORDER BY without specifying ROWS frame. The default RANGE frame includes all peers (rows with same ORDER BY value) in the running total, not just up to the current row.

```sql
-- ❌ Wrong: RANGE frame includes all peers
SELECT
    o.CustomerId,
    o.OrderDate,
    SUM(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
    ) AS RunningTotal
FROM dbo.Orders o;
-- If 3 orders on same date, each row's running total includes all 3 orders

-- ✅ Correct: ROWS with unique ORDER BY
SELECT
    o.CustomerId,
    o.OrderDate,
    SUM(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o;
```

**Symptom:** The running total jumps by more than one row's value at each tie boundary. Multiple rows share the same running total value.

**Fix:** Specify ROWS frame explicitly and add a unique tiebreaker to ORDER BY.

**Cost of not fixing:** Financial reports show incorrect cumulative balances. Running total for the 2nd and 3rd order on the same date are identical and both include all 3 orders.

### 2. Self-Join for Running Total — Catastrophic Performance

**Pitfall:** Using a self-join to compute running totals instead of SUM OVER ORDER BY. This is 100-1000x slower.

```sql
-- ❌ Wrong: Self-join (catastrophic performance)
SELECT o1.OrderId, SUM(o2.TotalAmount) AS RunningTotal
FROM dbo.Orders o1
INNER JOIN dbo.Orders o2 ON o2.OrderDate <= o1.OrderDate
GROUP BY o1.OrderId, o1.OrderDate;

-- ✅ Correct: SUM OVER ORDER BY
SELECT o.OrderId,
    SUM(o.TotalAmount) OVER (
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o;
```

**Symptom:** The self-join runs for minutes on 1M rows. At 10M rows, it may never complete.

**Fix:** Always use SUM OVER ORDER BY for running totals. Never use self-joins.

**Cost of not fixing:** 100-1000x slower queries. Query timeouts on any table larger than 100K rows.

### 3. Arithmetic Overflow for Large Running Totals

**Pitfall:** The running total accumulator exceeds the maximum value of the data type.

```sql
-- ❌ Wrong: DECIMAL(10,2) will overflow
SELECT
    o.OrderId,
    SUM(o.TotalAmount) OVER (
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o;
-- Error if total exceeds 99999999.99

-- ✅ Correct: Use larger data type
SELECT
    o.OrderId,
    SUM(CONVERT(DECIMAL(38,2), o.TotalAmount)) OVER (
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o;
```

**Symptom:** "Arithmetic overflow error" when the running total exceeds the column's data type range.

**Fix:** Cast/convert the column to a larger data type before the SUM OVER. DECIMAL(38,2) supports up to 10^38 — safe for virtually any running total.

**Cost of not fixing:** Query failure at runtime. The overflow is unpredictable — it depends on the cumulative sum reaching the limit.

### 4. Missing PARTITION BY — Global Running Total Instead of Per-Group

**Pitfall:** Omitting PARTITION BY when a per-group running total is needed. The running total accumulates across all rows instead of resetting per group.

```sql
-- Probably wrong: Global running total
SELECT
    o.CustomerId,
    o.OrderDate,
    SUM(o.TotalAmount) OVER (
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o;

-- Intended: Per-customer running total
SELECT
    o.CustomerId,
    o.OrderDate,
    SUM(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o;
```

**Symptom:** The running total is much larger than expected. It accumulates across all customers without resetting.

**Fix:** Include PARTITION BY for per-group running totals.

**Cost of not fixing:** Wrong cumulative calculations. Financial reports show total company revenue instead of per-customer cumulative.

### 5. SUM OVER with Frame Starting After UNBOUNDED PRECEDING

**Pitfall:** Using a frame start other than UNBOUNDED PRECEDING creates a non-standard running total that only includes recent rows.

```sql
-- Running total of only last 5 rows (not a full running total)
SELECT
    o.OrderId,
    o.OrderDate,
    o.TotalAmount,
    SUM(o.TotalAmount) OVER (
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN 5 PRECEDING AND CURRENT ROW
    ) AS Last5Sum  -- Not a running total! Only 6 rows max
FROM dbo.Orders o;
```

**Symptom:** The "running total" stops growing after the first 6 rows. The engineer expected an unbounded cumulative sum.

**Fix:** Use ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW for full running totals.

**Cost of not fixing:** Confusion and incorrect reports. The running total is actually a moving sum, not a cumulative sum.

### 6. SUM OVER with NULL Values

**Pitfall:** Assuming NULL values contribute to the running total. SUM skips NULLs (treats them as 0 in the window aggregate).

```sql
-- NULLs are skipped in the running total
SELECT
    o.OrderId,
    o.DiscountAmount,
    SUM(ISNULL(o.DiscountAmount, 0)) OVER (
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningDiscounts
FROM dbo.Orders o;
-- Without ISNULL, NULL DiscountAmount rows contribute 0 to the running total
-- But the running total might not increment as expected

-- If you need NULLs to be 0 explicitly:
SUM(COALESCE(o.DiscountAmount, 0)) OVER (...)
```

**Symptom:** The running total does not increase for rows where the target column is NULL. This is correct behavior (SUM skips NULLs) but can be surprising.

**Fix:** Use ISNULL/COALESCE to replace NULLs with 0 if you want the running total to be continuous.

**Cost of not fixing:** Running total appears to skip rows. Not an error but may be confusing in financial reports.

### Benchmark: SUM OVER vs Self-Join

```sql
-- Baseline: Self-join running total (10K rows)
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

SELECT
    o1.CustomerId,
    o1.OrderId,
    SUM(o2.TotalAmount) AS RunningTotal
FROM dbo.Orders o1
INNER JOIN dbo.Orders o2
    ON o1.CustomerId = o2.CustomerId
    AND o2.OrderDate <= o1.OrderDate
GROUP BY o1.CustomerId, o1.OrderId, o1.OrderDate
ORDER BY o1.CustomerId, o1.OrderDate;
-- 10K rows: ~2,500ms, 256K logical reads
-- 100K rows: ~timeout (~250,000ms+)

    o.CustomerId,
    o.OrderId,
    SUM(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o
ORDER BY o.CustomerId, o.OrderDate;
-- 10K rows: ~35ms, 128K logical reads
-- 100K rows: ~250ms, 128K logical reads
-- 1M rows: ~3,200ms, 128K logical reads

-- Improvement: 100-1000x faster
```

### Spoken Answers

**Q1: What are the three patterns of SUM() OVER()?**

> **Average answer:** "SUM OVER does running totals, partition totals, and grand totals. You can use it with ORDER BY, PARTITION BY, or nothing."

> **Great answer:** "SUM() OVER() has three distinct patterns. First, SUM(col) OVER() — no PARTITION BY, no ORDER BY — gives the grand total of col on every row. The engine computes this with a single Stream Aggregate pass and repeats the constant value. Second, SUM(col) OVER(PARTITION BY p) gives the total of col per partition, repeated on every row within that partition. This is equivalent to SUM GROUP BY p joined back, but done in one scan with a Segment operator. Third, SUM(col) OVER(ORDER BY o) gives a running total — a cumulative sum that increases with each row. This is the most commonly used pattern and the one with the most subtle frame behavior. The critical implementation detail: SUM OVER ORDER BY does NOT use a Window Spool. It uses a streaming accumulator — a single variable updated as the sorted rows are scanned. This makes it much more memory-efficient than LEAD/LAG/FIRST_VALUE, which require spools. The memory grant for SUM OVER ORDER BY is dominated entirely by the Sort operator, not by the window function itself. The default frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, which with non-unique ORDER BY includes all peer rows — always specify ROWS for precise running totals."

**Q5: Compare SUM() OVER(ORDER BY) and SUM() OVER(PARTITION BY).**

> **Average answer:** "SUM OVER ORDER BY is a running total that increases row by row. SUM OVER PARTITION BY is the same total repeated for all rows in the same partition."

> **Great answer:** "SUM() OVER(ORDER BY) and SUM() OVER(PARTITION BY) are fundamentally different. SUM OVER PARTITION BY gives a static value — the total for the entire partition, repeated identically on every row. It does not change as you move from row to row within the partition. SUM OVER ORDER BY gives a dynamic value — the cumulative sum up to and including the current row, which increases as you progress through the ordered partition. The execution plan difference: SUM OVER PARTITION BY uses Segment + Stream Aggregate (or Hash Aggregate) — it computes the total per partition and repeats it. SUM OVER ORDER BY uses Sort + Window Aggregate with a streaming accumulator — it computes progressively. The partition total pattern is useful for percentage-of-total calculations: 'what % of the customer's total is this order?' The running total pattern is useful for cumulative calculations: 'what is the running balance after this transaction?' The two can be combined: SUM OVER PARTITION BY ... ORDER BY gives a running total that resets at each partition boundary."

**Q8: How do you implement running totals in EF Core and Dapper?**

> **Average answer:** "Both require raw SQL since EF Core can't translate SUM OVER."

> **Great answer:** "EF Core does not translate any LINQ expression to SUM() OVER(). You must use SqlQueryRaw or FromSqlRaw with the complete SQL including the frame specification. For running totals, I always specify ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW explicitly, never relying on the default frame. I also add a unique tiebreaker to ORDER BY and consider overflow with CONVERT(DECIMAL(38,2), col). In Dapper, the pattern is the same: raw SQL in a const string passed to QueryAsync. For both frameworks, I wrap the running total query in a dedicated service method with a clear XML comment explaining the frame behavior. For very large running totals, I consider capping with MAX_GRANT_PERCENT and using a covering index to eliminate the Sort. One pattern I use in production: for financial running balances that must be absolutely correct, I add a CHECKSUM or validation query that compares the SUM OVER result with a manual aggregation to detect overflow or frame issues."

**Challenge 1 — Write the SQL**

You are building a financial ledger system. For each account, show every transaction with: the running balance after each transaction, the total deposits for the account, the total withdrawals, and the net change (deposits - withdrawals). Use the `Transactions` table with columns: TransactionId (BIGINT), AccountId (INT), TransactionDate (DATETIME2), Amount (DECIMAL(18,2)), TransactionType (NVARCHAR — 'Deposit', 'Withdrawal', 'Fee'). <details>
<summary>Solution</summary>

```sql
SELECT
    t.TransactionId,
    t.AccountId,
    t.TransactionDate,
    t.Amount,
    t.TransactionType,
    t.Description,
    SUM(CASE WHEN t.TransactionType = 'Deposit' THEN t.Amount ELSE 0 END) OVER (
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate, t.TransactionId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningDeposits,
    SUM(CASE WHEN t.TransactionType IN ('Withdrawal', 'Fee') THEN t.Amount ELSE 0 END) OVER (
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate, t.TransactionId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningWithdrawals,
    SUM(CASE
        WHEN t.TransactionType = 'Deposit' THEN t.Amount
        WHEN t.TransactionType IN ('Withdrawal', 'Fee') THEN -t.Amount
        ELSE 0
    END) OVER (
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate, t.TransactionId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningBalance,
    SUM(CASE WHEN t.TransactionType = 'Deposit' THEN t.Amount ELSE 0 END) OVER (
        PARTITION BY t.AccountId
    ) AS TotalDeposits,
    SUM(CASE WHEN t.TransactionType IN ('Withdrawal', 'Fee') THEN t.Amount ELSE 0 END) OVER (
        PARTITION BY t.AccountId

FROM dbo.Transactions t
ORDER BY t.AccountId, t.TransactionDate, t.TransactionId;
```

**Challenge 3 — Explain the execution plan**

```sql
-- Query A:
SELECT
    o.CustomerId,
    o.OrderId,
    SUM(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders o;

-- Query B:
SELECT
    o.CustomerId,
    o.OrderId,
    LEAD(o.TotalAmount) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate, o.OrderId
    ) AS NextAmount
FROM dbo.Orders o;
```

Why does Query A not show a Window Spool operator in its execution plan, while Query B does?

<details>
<summary>Solution</summary>

**Why Query A has no spool:** SUM() OVER(ORDER BY) uses a streaming accumulator. The engine sorts the rows by (CustomerId, OrderDate, OrderId), then scans the sorted result sequentially. As it scans, it maintains a single variable per partition: `running_total = running_total + current_row.TotalAmount`. This variable is all it needs — it does not need to look back at previous rows or forward at future rows. No spool is required.

**Why Query B has a spool:** LEAD() is an offset function that needs to read a row at a specific offset ahead of the current row. To do this, the engine must materialize the sorted rows in a Window Spool that supports positional reads: "read the row at position current + offset." The spool allows random access by position, which is necessary for offset-based functions.

**Plan for Query A:** Clustered Index Scan → Sort → Window Aggregate (streaming accumulator) → SELECT
**Plan for Query B:** Clustered Index Scan → Sort → Window Spool → Window Aggregate (offset read) → SELECT

**Memory impact:** Query A's memory grant is dominated by the Sort operator. Query B's memory grant includes both the Sort AND the Window Spool. For a 1M row table with 10M total bytes of data in the window columns, Query A needs ~200MB for the sort; Query B needs ~200MB for the sort + ~80MB for the spool.

**To get a different plan for Query A:** Add a covering index on (CustomerId, OrderDate) INCLUDE (TotalAmount) — eliminates the Sort entirely. Query A then becomes a simple Index Scan + streaming accumulator with zero memory grant for sorting.

</details>

**Challenge 5 — Design the index**

**Scenario:** A financial dashboard runs this query every 5 minutes against the 200M-row Transactions table:

```sql
SELECT
    t.AccountId,
    t.TransactionId,
    t.TransactionDate,
    t.Amount,
    t.TransactionType,
    SUM(CASE
        WHEN t.TransactionType = 'Deposit' THEN t.Amount
        WHEN t.TransactionType IN ('Withdrawal', 'Fee') THEN -t.Amount
        ELSE 0
    END) OVER (
        PARTITION BY t.AccountId
        ORDER BY t.TransactionDate, t.TransactionId
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningBalance
FROM dbo.Transactions t
WHERE t.AccountId = @AccountId
ORDER BY t.TransactionDate, t.TransactionId;
```

The query runs for a single account (returning ~500 rows). Current execution time is 50ms. However, with 10 concurrent users, the overall system performance degrades. Read/write ratio is 60/40.

<details>
<summary>Solution</summary>

```sql
-- Primary index: Supporting the running total query
CREATE INDEX IX_Transactions_AccountId_TransactionDate
ON dbo.Transactions(AccountId, TransactionDate, TransactionId)
INCLUDE (Amount, TransactionType);
```

**Why this index:**
- AccountId as the leading column enables an Index Seek for the single account — extremely fast
- TransactionDate + TransactionId provides the sorted order required by the running total's ORDER BY — eliminates the Sort operator entirely
- INCLUDE (Amount, TransactionType) makes the index covering — all columns needed for the CASE expression and SUM are in the index
- The query returns ~500 rows per execution — the index seek reads 500 rows with minimal I/O

**Without this index:**
- Clustered index scan on 200M rows (even filtering to one AccountId — partition elimination happens at the storage level but seek is faster)
- Sort on 200M rows (even though only 500 match the filter — the Sort happens before the filter in some plan shapes)

**With this index:**
- Index Seek on AccountId — reads exactly 500 rows
- No Sort — rows arrive in correct order
- Window Aggregate with streaming accumulator — 500 accumulator operations
- Estimated: <5ms per query

**Tradeoffs:**
- Size: AccountId (4) + TransactionDate (8) + TransactionId (8) = 20 bytes key × 200M = 4GB + INCLUDE columns (Amount 9 + TransactionType avg 10) = 19 × 200M = 3.8GB + overhead ≈ 10GB total. This is significant but justified by the 60/40 read/write ratio and the criticality of the dashboard.
- Write overhead: Each INSERT to Transactions must maintain this index. Estimated 25-30% increase in insert time. Since transactions are inserted in chronological order and by account, the index insert is mostly sequential (appending to the end of each account's partition).

**What NOT to index:**
- A filtered index for specific account IDs is not practical — the query handles any account
- Do not include Description or other large columns — they bloat the index without benefit
- The clustered index on TransactionId is still needed for point lookups and foreign keys

**After index — logical reads:** ~4-8 per execution (seek + range scan of 500 rows)
**Execution plan:** Index Seek → Window Aggregate (streaming accumulator) → SELECT
**Sort eliminated:** Yes — index provides order-preserving access
**No Sort memory grant needed — zero wait impact on concurrent users**

</details>
