### Where This Fits

Every window function query adds a Sort operator to the execution plan unless the required ordering is already provided by an index. A .NET backend engineer shipping a query with `ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC)` may unknowingly trigger a 200 MB sort on a 10M row table, consuming server memory and potentially spilling to tempdb. The interview signal here distinguishes engineers who understand that window functions are not free — they require ordering, and ordering requires a sort (or an index). The candidate who can describe the sort memory grant formula, identify redundant sorts from multiple window functions, and create indexes to eliminate sorts demonstrates senior-level query performance knowledge.

## Core Mental Model

A window function's OVER clause specifies a partitioning and ordering of rows. The database engine must arrange rows in that order before it can compute the function. SQL Server uses a Sort operator in the execution plan to achieve this ordering unless an existing index already provides rows in the correct order. The Sort operator reads all input rows, allocates a memory grant (estimated rows × row width × overhead factor), sorts the data using the specified ORDER BY columns, and writes the sorted result to a workfile in memory. If the memory grant is insufficient to hold the full sort, the sort spills to tempdb — it writes intermediate sort runs to disk and merges them, causing 10-100x performance degradation. The memory grant is computed based on cardinality estimates from statistics: `memory_grant = estimated_rows × (row_width + sort_overhead) × safety_factor`. Multiple window functions in the same query can share a single Sort if they have identical PARTITION BY and ORDER BY clauses — but different specifications require separate Sort operators. The recognition pattern: in the execution plan, each Sort operator before a Sequence Project (window compute) represents one sort needed by one or more window functions.

### Classification

Sort is a physical operator in the execution plan. It is not SARGable — it cannot use an index seek. It requires a memory grant proportional to the number of rows and their width. SQL Server has two sort modes: an in-memory sort (fits in the memory grant) and a two-pass sort that spills to tempdb. The Sort operator appears in the plan as a blocking operator — it must consume all input rows before emitting the first output row. Sort is CPU-bound for in-memory operations and I/O-bound for spill operations.

## Deep Mechanics

### How the Engine Executes This

**Sort operator execution for window functions:**

1. **Memory grant computation** — The optimizer estimates the number of rows entering the Sort based on cardinality estimates from statistics. It computes the required memory grant: `grant = estimated_rows × (fixed_row_size + variable_overhead) × spill_factor`. The spill_factor accounts for the internal sort data structure overhead (typically 1.5-2x of the raw row size).

2. **Input consumption** — The Sort operator reads all rows from its child operator (table scan, index scan, or join output). It stores each row in the sort work area in memory. The memory area is pre-allocated based on the memory grant.

3. **In-memory sort (no spill):** — If all rows fit in the memory grant, the sort performs a standard comparison-based sort (typically a variation of quicksort or merge sort) on the input rows. It writes the sorted output to a work table in memory. The rows are then emitted one by one to the parent operator (Sequence Project).

4. **Spill to tempdb (insufficient memory):** — If the rows exceed the memory grant, SQL Server uses a two-phase merge sort:
   - **Phase 1 (run generation):** Reads as many rows as fit in the memory grant, sorts them, and writes the sorted run to tempdb. Repeats until all rows are processed. Each run is a sorted subset of data stored as a tempdb worktable.
   - **Phase 2 (merge):** Reads the sorted runs from tempdb and merges them into a single sorted output. The merge may require additional memory for the merge buffer.
   - Each spill level adds one full read + write cycle to tempdb. Level 1 spill writes all data once. Level 2 spill writes it twice.

5. **Sequence Project consumes sorted input** — The Sequence Project operator reads rows in sorted order. It detects partition boundaries by comparing the PARTITION BY column values of consecutive rows. When a boundary is detected, it resets the window function state (running aggregate, row counter, etc.). It computes the window function value for each row and passes it to the SELECT clause.

6. **Multiple window functions — sort sharing:**
   - If two window functions have identical `PARTITION BY` and `ORDER BY` clauses, the optimizer can use a single Sort and a single Sequence Project for both. The Sequence Project computes multiple window functions in one pass.
   - If the OVER clauses differ in PARTITION BY or ORDER BY, each unique specification gets its own Sort + Sequence Project. This is visible in the execution plan as multiple Sort operators before Sequence Project operators.

**Sort properties in the execution plan (actual vs estimated):**

The execution plan XML exposes sort properties:
- `<Sort Ordered="false">` — indicates the Sort is required for window function ordering (ordered:false means the output needs sort, not that input is unsorted). Wait — actually `Ordered: false` in the sort input (the child of sort) means the input is not ordered. The Sort itself has properties like `DISTINCT` and `ORDER BY` columns.
- In the plan XML: `Ordered="true"` on the input to Sequence Project means the Sequence Project requires ordered input.

**Memory grant feedback (SQL Server 2017+):**

For queries with Sort operators, SQL Server 2017+ can adjust the memory grant on subsequent executions:
- First execution: if the query spills, the actual sort size is recorded.
- Second execution: the memory grant is increased if the previous execution spilled.
- Third and subsequent executions: the grant converges to the actual size needed.
- The feedback is visible in `sys.dm_exec_query_stats.used_grant_kb` and `ideal_grant_kb`.

### SQL Visibility

```sql
-- Example 1: Single window function — one Sort
SELECT
    o.OrderId,
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01'
ORDER BY o.CustomerId, o.OrderDate DESC;
-- Plan: Clustered Index Scan → Sort → Sequence Project → SELECT
-- Sort orders by CustomerId, OrderDate DESC for the window function

-- Example 2: Two window functions with same OVER — one Sort (shared)
SELECT
    o.OrderId,
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn,
    SUM(o.TotalAmount) OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS RunningTotal
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01'
ORDER BY o.CustomerId, o.OrderDate DESC;
-- Plan: Clustered Index Scan → Sort (ONE sort) → Sequence Project (both functions) → SELECT
-- Both window functions have identical OVER — optimizer merges into one sort

-- Example 3: Two window functions with different OVER — two Sorts (red flag)
SELECT
    o.OrderId,
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn,
    SUM(o.TotalAmount) OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate ASC) AS RunningTotalAsc
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01'
ORDER BY o.CustomerId, o.OrderDate DESC;
-- Plan: Clustered Index Scan → Sort1 (DESC) → Sequence Project1 → Sort2 (ASC) → Sequence Project2 → SELECT
-- TWO Sort operators — DESC and ASC are different orderings

-- Example 4: Window function with index support — no Sort
-- Requires: CREATE INDEX IX_Orders_CustomerId_OrderDate
--     ON dbo.Orders(CustomerId, OrderDate DESC) INCLUDE (TotalAmount);
SELECT
    o.OrderId,
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn
FROM dbo.Orders AS o WITH (INDEX(IX_Orders_CustomerId_OrderDate))
WHERE o.OrderDate >= '2024-01-01'
ORDER BY o.CustomerId, o.OrderDate DESC;
-- Plan: Index Scan (ordered) → Sequence Project → SELECT
-- NO Sort operator! Index provides rows in correct order.

-- Example 5: Trace flag 9358 for sort elimination
-- Requires TF 9358 enabled globally or at query level
SELECT
    o.OrderId,
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    SUM(o.TotalAmount) OVER(PARTITION BY o.CustomerId) AS CustTotal,
    AVG(o.TotalAmount) OVER(PARTITION BY o.CustomerId) AS CustAvg
FROM dbo.Orders AS o
OPTION (QUERYTRACEON 9358);
-- With TF 9358, if the optimizer determines that ORDER BY is not needed for
-- the window aggregate computation (e.g., no frame specification, default RANGE),
-- it may eliminate the sort even without an index.
```

### Failure Modes

**1. Sort spill to tempdb from under-estimated memory grant:**

When statistics are stale or the cardinality estimate is too low, the memory grant is insufficient and the Sort spills:

```sql
-- Detect sort spills from the plan cache
SELECT TOP 10
    qs.total_elapsed_time / qs.execution_count AS avg_elapsed_us,
    qs.total_logical_reads / qs.execution_count AS avg_logical_reads,
    qs.total_worker_time / qs.execution_count AS avg_cpu_us,
    CAST(qp.query_plan AS XML).value(
        'declare namespace p="http://schemas.microsoft.com/sqlserver/2004/07/showplan";
         count(//p:SpillToTempDb[1])', 'int') AS spill_count,
    SUBSTRING(st.text, 1, 200) AS query_text
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
ORDER BY avg_elapsed_us DESC;
```

**2. Multiple Sort operators from different window function specifications:**

If a query has `ROW_NUMBER() OVER(PARTITION BY A ORDER BY B)` and `SUM() OVER(PARTITION BY A ORDER BY C)`, the different ORDER BY columns force two separate Sort operators. Each sort consumes its own memory grant. Fix: homogenize the ORDER BY if possible, or create indexes covering both orderings.

**3. Sort with ordered:false on the input to Sequence Project:**

In the execution plan XML, if the input to Sequence Project has `Ordered="false"`, the Sequence Project is not receiving rows in the required partition order. This typically means the Sort is missing or the optimizer chose a different plan shape (e.g., hash aggregate then window function without proper ordering). This is a sign that the window function may not be computed correctly in all cases.

**4. Grant memory feedback not helping on first execution:**

Memory grant feedback requires at least two executions to adjust. The first execution always uses the estimated grant. If the estimate is wrong, the first execution spills. For one-off queries (parameterized queries with rare parameter values), feedback never helps.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Schema context
-- ============================================================
CREATE TABLE dbo.Orders
(
    OrderId      INT            NOT NULL IDENTITY(1,1),
    CustomerId   INT            NOT NULL,
    OrderDate    DATETIME2(0)   NOT NULL,
    TotalAmount  DECIMAL(18,2)  NOT NULL,
    Status       VARCHAR(20)    NOT NULL DEFAULT 'Pending',
    ShipCity     VARCHAR(100)   NULL,
    CONSTRAINT PK_Orders PRIMARY KEY CLUSTERED (OrderId)
);

CREATE TABLE dbo.SalesTransactions
(
    TransactionId   INT            NOT NULL IDENTITY(1,1),
    SaleDate        DATE           NOT NULL,
    ProductId       INT            NOT NULL,
    Quantity        INT            NOT NULL,
    UnitPrice       DECIMAL(18,2)  NOT NULL,
    StoreId         INT            NOT NULL,
    CashierId       INT            NOT NULL,
    CONSTRAINT PK_SalesTransactions PRIMARY KEY CLUSTERED (TransactionId)
);

-- ============================================================
-- Pattern 1: Avoid redundant sorts — match window function OVER clauses
-- ============================================================
-- ❌ Two Sorts: different ORDER BY directions
SELECT
    o.OrderId,
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn,
    SUM(o.TotalAmount) OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate ASC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS RunningTotal
FROM dbo.Orders AS o;

-- ✅ One Sort: make both window functions use the same ORDER BY
-- Running total with ORDER BY DESC can be computed as window aggregate
-- (the frame ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW works in DESC order too)
-- Or compute running total differently to match the ORDER BY
SELECT
    o.OrderId,
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn,
    SUM(o.TotalAmount) OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS RunningTotal
FROM dbo.Orders AS o;
-- Both use ORDER BY OrderDate DESC — one Sort for both window functions

-- ============================================================
-- Pattern 2: Create index to eliminate Sort for window function
-- ============================================================
-- Before: Clustered Index Scan → Sort (200 MB grant) → Sequence Project
-- After:  Index Scan (ordered) → Sequence Project

CREATE INDEX IX_Orders_CustomerId_OrderDate_Window
    ON dbo.Orders(CustomerId, OrderDate DESC)
    INCLUDE (TotalAmount, Status);

-- Verify index usage:
SELECT
    o.OrderId,
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01'
ORDER BY o.CustomerId, o.OrderDate DESC;
-- Plan: Index Seek (on CustomerId for covering) or Index Scan (ordered)
-- No Sort operator

-- ============================================================
-- Pattern 3: Index for window function with filter
-- ============================================================
-- If the window function is always filtered by Status, create a filtered index
CREATE INDEX IX_Orders_CustomerId_OrderDate_Delivered
    ON dbo.Orders(CustomerId, OrderDate DESC)
    INCLUDE (TotalAmount)
    WHERE Status = 'Delivered';

SELECT
    o.OrderId,
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn
FROM dbo.Orders AS o
WHERE o.Status = 'Delivered'
ORDER BY o.CustomerId, o.OrderDate DESC;
-- Filtered index is narrower (fewer rows), faster scan, no sort

-- ============================================================
-- Pattern 4: Trace flag 9358 for sort elimination (when ORDER BY not needed)
-- ============================================================
-- For window aggregate functions without frame specification (default RANGE),
-- ORDER BY may not be needed for the aggregate value — TF 9358 can eliminate the sort
-- SQL Server 2019+ with specific compatibility level
SELECT
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    SUM(o.TotalAmount) OVER(PARTITION BY o.CustomerId) AS CustTotal,
    AVG(o.TotalAmount) OVER(PARTITION BY o.CustomerId) AS CustAvg
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01'
OPTION (QUERYTRACEON 9358);
-- TF 9358 tells the optimizer: "window aggregate functions don't need ORDER BY
-- because the default frame (RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
-- gives the same result as the full partition for SUM and AVG"
-- This can only be enabled globally, not per-query, in most environments

-- ============================================================
-- Pattern 5: Memory grant tuning for Sort-heavy queries
-- ============================================================
-- Increase minimum memory grant per query to avoid spills (but may cause RESOURCE_SEMAPHORE)
-- Query hint to control grant:
SELECT
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    SUM(o.TotalAmount) OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS RunningTotal
FROM dbo.Orders AS o
OPTION (MIN_GRANT_PERCENT = 5, MAX_GRANT_PERCENT = 20);
-- MIN_GRANT_PERCENT: minimum 5% of ideal grant
-- MAX_GRANT_PERCENT: maximum 20% of ideal grant
-- Useful when the estimated grant is too large (causing MEMORY_GRANT waits)
-- or too small (causing spills)

-- ============================================================
-- Pattern 6: Check for sort spills in actual execution plan
-- ============================================================
-- Enable actual execution plan and look for:
-- Sort operator with yellow triangle warning ⚠
-- Warning text: "Operator used tempdb to spill data during execution"
-- Spill level: 1, 2, etc. (higher = more severe)

-- Use DMV to see recent sort spill queries:
SELECT TOP 10
    qs.execution_count,
    qs.total_elapsed_time / qs.execution_count / 1000 AS avg_elapsed_ms,
    qs.total_worker_time / qs.execution_count / 1000 AS avg_cpu_ms,
    qs.total_logical_reads / qs.execution_count AS avg_logical_reads,
    CAST(qp.query_plan AS XML).value('
        declare namespace p="http://schemas.microsoft.com/sqlserver/2004/07/showplan";
        sum(//p:SpillToTempDb/@SpillLevel)', 'int') AS total_spill_level,
    SUBSTRING(st.text, 1, 200) AS query_text
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
ORDER BY total_spill_level DESC;
```

## Gotchas and Production Pitfalls

### Two Window Functions With Different ORDER BY = Two Sorts

**Pitfall:** Adding a second window function with a different ORDER BY direction (ASC vs DESC) or different columns, not realizing the optimizer creates separate Sort operators.

```sql
-- ❌ Two different ORDER BY directions — two Sort operators
SELECT
    o.OrderId, o.CustomerId, o.OrderDate, o.TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn,
    SUM(o.TotalAmount) OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate ASC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS RunningTotal
FROM dbo.Orders AS o;
-- Plan: Sort1 (DESC) on 10M rows + Sort2 (ASC) on 10M rows = ~400 MB total memory

-- ✅ Homogenize ORDER BY direction — one shared Sort
SELECT
    o.OrderId, o.CustomerId, o.OrderDate, o.TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn,
    SUM(o.TotalAmount) OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS RunningTotal
FROM dbo.Orders AS o;
-- Plan: One Sort (DESC) — both window functions use the same sorted input
-- Memory grant: ~200 MB total
```

**Symptom:** Execution plan shows two Sort operators with similar costs. Memory grant doubles. Query CPU time doubles. If the server has memory pressure, both sorts may spill independently.

**Fix:** Rewrite window function ORDER BY clauses to match. If the running total must be in ASC order but the rank needs DESC, compute the running total using a different frame (ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING for DESC order) or compute the running total in the application layer.

**Cost of not fixing:** A 10M row query with two different ORDER BY directions uses 400 MB memory grant instead of 200 MB. At 20 concurrent queries, that is 8 GB vs 4 GB — the difference between fitting in memory or encountering RESOURCE_SEMAPHORE waits. Query time goes from 500ms to 1.2s due to sort overhead.

### Sort Spill to Tempdb From Stale Statistics

**Pitfall:** The memory grant for the Sort is based on estimated row count from statistics. If statistics are stale (e.g., after a large data load without updating statistics), the estimate may be far below actual, causing the sort to spill.

```sql
-- ❌ Statistics show 1M rows but actual is 10M rows
-- Sort memory grant: 20 MB (for 1M rows)
-- Actual sort needs: 200 MB (for 10M rows)
-- Spill level 1: writes 200 MB to tempdb
SELECT o.OrderId, o.TotalAmount,
       ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC) AS rn
FROM dbo.Orders AS o;

-- ✅ Update statistics before running heavy window queries
UPDATE STATISTICS dbo.Orders;
-- Or auto-update: ensure AUTO_UPDATE_STATISTICS is ON (default)
-- Check: SELECT is_auto_update_stats_on FROM sys.databases WHERE name = 'YourDb';
```

**Symptom:** The query runs fine during development (small dataset) but spills in production (large dataset). SET STATISTICS TIME shows elapsed time >> CPU time (waiting on tempdb I/O). The execution plan shows a Sort with a spill warning.

**Fix:** Keep statistics up to date. Use a maintenance plan to update statistics on tables with window function queries. For critical window queries, consider manual UPDATE STATISTICS before the ETL window closes.

**Cost of not fixing:** A nightly sales report that processes 50M rows with a window function spills 1 GB to tempdb. The report takes 45 minutes instead of 2 minutes. Tempdb grows by 10 GB, consuming disk space and causing I/O contention for all other queries.

---

### Assuming Index on PARTITION BY Alone Eliminates the Sort

**Pitfall:** Creating an index on only the PARTITION BY column (e.g., `CustomerId`) and assuming the window function's Sort is eliminated. The Sort needs the PARTITION BY AND ORDER BY columns together.

```sql
-- ❌ Index on CustomerId alone does NOT eliminate the sort
-- The window function needs ORDER BY OrderDate DESC
CREATE INDEX IX_Orders_CustomerId ON dbo.Orders(CustomerId)
    INCLUDE (TotalAmount, OrderDate);

SELECT o.OrderId, o.TotalAmount,
       ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn
FROM dbo.Orders AS o;
-- Plan: Index Scan → Sort (still needs to sort by OrderDate DESC within each CustomerId)
-- The index provides CustomerId order, but OrderDate is not in the key — it's INCLUDE'd
-- Sort sorts by OrderDate DESC within each CustomerId

-- ✅ Index on (CustomerId, OrderDate DESC) as KEY columns — eliminates the Sort
CREATE INDEX IX_Orders_CustomerId_OrderDate_Key
    ON dbo.Orders(CustomerId, OrderDate DESC)
    INCLUDE (TotalAmount);

SELECT o.OrderId, o.TotalAmount,
       ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn
FROM dbo.Orders AS o;
-- Plan: Index Scan (ordered) → Sequence Project
-- No Sort! Index provides rows in CustomerId, OrderDate DESC order
```

**Symptom:** Execution plan shows a Sort operator even though an index on the PARTITION BY column exists. The Sort is still sorting by the ORDER BY column.

**Fix:** Include the ORDER BY column as a key column in the index (not just as an INCLUDE column). The index key order must match the window function's ORDER BY: leading column = PARTITION BY, second column = ORDER BY column, direction must match.

**Cost of not fixing:** The Sort operator remains, consuming CPU and memory. At 10M rows, the Sort uses ~200 MB grant and adds 300ms CPU time. With the correct index, the Sort is eliminated, reducing grant to 0 MB and CPU to ~100ms.

---

### Memory Grant Feedback Does Not Prevent First-Execution Spills

**Pitfall:** Relying on SQL Server 2017+ memory grant feedback to fix sort spills. Feedback adjusts the grant on the SECOND execution — the first execution still spills.

```sql
-- SQL Server 2017+ with memory grant feedback enabled (default)
-- First execution: Sort on 10M rows with estimated 1M rows → spills
-- Feedback records: actual sort size = 200 MB
-- Second execution: Sort with grant upgraded to ~220 MB → no spill

-- ❌ For queries that run infrequently, the spill happens every time
-- A monthly report executed once per month always spills (first execution)
```

**Symptom:** Monitoring shows that the query spills on its first execution after a server restart or plan recompile. Subsequent executions within the plan cache lifetime do not spill.

**Fix:** Update statistics before the query runs. Pre-warm the plan cache with realistic parameter values. Consider a covering index to eliminate the Sort entirely.

**Cost of not fixing:** A monthly financial report that runs once per month always takes 45 minutes instead of 2 minutes because the sort spills on every execution (plan cache cleared between months due to server maintenance).

---

### Redundant Sort From Implicit Column Conversion in ORDER BY

**Pitfall:** The ORDER BY column in the window function has a different data type than the column in the table due to implicit conversion, causing the Sort to use a different sort key than the index provides.

```sql
-- ❌ OrderDate is DATETIME2(0) but the parameter/variable is DATETIME
-- The DECLARE creates a DATETIME variable that compares differently to DATETIME2
DECLARE @StartDate DATETIME = '2024-01-01';

SELECT o.OrderId, o.TotalAmount,
       ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn
FROM dbo.Orders AS o
WHERE o.OrderDate >= @StartDate;
-- Even with an index on (CustomerId, OrderDate DESC), the sort may not be eliminated
-- if the implicit conversion between DATETIME2 and DATETIME causes the sort key to differ

-- ✅ Match data types exactly
DECLARE @StartDate DATETIME2(0) = '2024-01-01';
-- Or use proper parameterization in .NET
```

**Symptom:** The execution plan shows a Sort operator even though a seemingly correct index exists. The index is not used for ordering because the column types don't match exactly in the sort key.

**Fix:** Ensure data types match between the table column and the parameter/variable. In .NET, use `DateTime` for `DATETIME2` columns (EF Core maps C# `DateTime` to `DATETIME2` by default in modern versions). Use `SqlParameter` with `SqlDbType.DateTime2`.

**Cost of not fixing:** The Sort operator persists despite the index, adding 200 MB memory grant and 300ms CPU per execution. Hard to diagnose because the index IS used for the scan, but the sort IS NOT eliminated.

### Spoken Answers

**Q1: What Sort operator properties does SQL Server report in an execution plan, and how do you read them?**

> **Average answer:** The Sort operator shows ORDER BY columns and memory grant.

> **Great answer:** In the execution plan, the Sort operator exposes the following key properties: (1) ORDER BY columns — the columns being sorted and their direction (ASC/DESC); (2) Memory Grant — estimated memory required for the sort, shown in KB or MB in the plan properties; (3) Spill Level — if the sort spills to tempdb, the warning shows the spill level (1, 2, etc.) where level 1 indicates one write pass and one read pass of tempdb data; (4) Estimated Number of Rows — the cardinality estimate driving the memory grant; (5) Distinct — whether the Sort eliminates duplicates (not relevant for window functions). In the plan XML, the Sort operator is `<Sort>` with child `<OrderBy>` listing the sort columns. The preceding operator's `Ordered` property on the input to the Sequence Project tells you whether the Sequence Project receives ordered rows. If `Ordered="false"` on the Sequence Project input, something is wrong — the window function may not compute correctly. To read the plan: open the actual execution plan in SSMS, hover over the Sort operator, and check the "Memory Grant" and "Warnings" sections. A yellow triangle with "SpillToTempDb" is the key warning. You can also query `sys.dm_exec_query_stats` to see sort spill counts: the `total_spills` column indicates how many times the query spilled across all executions.

**Q3: How do you calculate the memory grant for a Sort in a window function query? What happens when the grant is insufficient?**

> **Average answer:** The memory grant is based on row count and row size. If it's too small, the sort spills to tempdb.

> **Great answer:** The optimizer estimates the memory grant for a Sort as: `estimated_rows × avg_row_width × sort_overhead_factor`. The avg_row_width includes all columns needed by the sort key and any columns propagated through the operator. The sort_overhead_factor is approximately 1.5-2x of the raw row size because the sort data structure stores additional metadata per row (row pointers, sort key extracted values). For example, sorting 10M rows with a 40-byte average row width gives an estimated 400 MB of raw data, and the memory grant would be approximately 600 MB with overhead. If the grant is insufficient, SQL Server uses a two-phase merge sort: it writes sorted runs to tempdb (each run as large as the memory grant), then merges the runs. This is called a "spill" and is visible in the execution plan as a warning. The performance impact is dramatic: a sort that takes 500ms in memory can take 5-15 seconds when spilling, because each row is written to tempdb and read back at least once per spill level. Level 1 spill = write all rows once, read all rows once. Level 2 spill = write all rows twice, read all rows twice. The detection DMV is `sys.dm_exec_query_stats` where `total_spills` is non-zero, or the execution plan XML contains `<SpillToTempDb SpillLevel="1"/>`. The fix is either (a) create a covering index that provides sorted input and eliminates the Sort entirely, (b) update statistics so the memory grant is accurate, or (c) use query hints like `MIN_GRANT_PERCENT` to increase the minimum grant for critical queries.

**Q5: How does SQL Server share a single Sort between multiple window functions? When does it create separate Sort operators?**

> **Average answer:** If window functions have the same OVER, they share one sort. Different OVER means different sorts.

> **Great answer:** SQL Server's optimizer detects window functions with identical `PARTITION BY` and `ORDER BY` clauses and groups them into a single Sequence Project operator. That operator uses one sorted input to compute all matching window functions. The sort is performed once on the combined key (PARTITION BY columns + ORDER BY columns). The Sequence Project then iterates over the sorted rows, detecting partition boundaries by comparing consecutive rows, and computes all window functions in a single pass. This is visible in the execution plan as one Sort feeding one Sequence Project that shows multiple window function definitions. When the OVER clauses differ, separate Sort + Sequence Project pairs appear. The differences that force separate sorts include: (1) different PARTITION BY columns, (2) different ORDER BY columns, (3) different ORDER BY direction (ASC vs DESC), (4) different frame specifications (ROWS vs RANGE, different window size). However, there is an important nuance: if the ORDER BY columns are a subset or the same but with different directions, the optimizer still treats them as different. For example, `ORDER BY OrderDate DESC` and `ORDER BY OrderDate ASC` require two sorts because the sort order is reversed. The only exception is when one sort's output can be reversed by reading it backwards — but SQL Server's Sequence Project does not read sorted input backwards, so both directions require separate sorts. To optimize: homogenize window function OVER clauses so they share a single sort. This reduces memory grant by half and reduces CPU by ~40% for window-function-heavy queries.

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Sort elimination (index approach) → 0 MB memory grant, no spill risk|Index adds write overhead (~66% per INSERT)|
|Homogenized OVER clauses → half the memory grant|Requires careful SQL design — may need frame adjustment|
|Updated statistics → accurate grant, fewer spills|Statistics maintenance CPU cost (minimal, 1-5% of server)|

### Scale Thresholds

- **Sort becomes relevant at ~100K rows** — below this, sort time is negligible (<10ms)
- **Sort memory matters at ~1M rows** — memory grant ~20-100 MB depending on row width
- **Sort spill becomes critical at ~10M rows** — memory grant ~200-1000 MB; spill risk if stats are stale
- **Concurrency bottleneck at ~50 concurrent sort queries** — total grant demand exceeds 8 GB, RESOURCE_SEMAPHORE waits begin
- **Index elimination becomes mandatory at ~50M rows** — sort on this scale without index support spills on most servers

**Challenge 1 — Write the SQL**

Write a query that returns each order with its running total (cumulative sum of TotalAmount ordered by OrderDate within each CustomerId) and a row number ordered by OrderDate DESC within each CustomerId. Ensure the query uses only a single Sort operator in the execution plan. The Orders table has columns: OrderId, CustomerId, OrderDate, TotalAmount.

<details>
<summary>Solution</summary>

```sql
SELECT
    o.OrderId,
    o.CustomerId,
    o.OrderDate,
    o.TotalAmount,
    ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn,
    SUM(o.TotalAmount) OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS RunningTotal
FROM dbo.Orders AS o
ORDER BY o.CustomerId, o.OrderDate DESC;
```

Both window functions use `PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC` — identical OVER clauses. The optimizer creates one Sort (by CustomerId, OrderDate DESC) and one Sequence Project that computes both functions. Note: the running total in DESC order still computes correctly because ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW in DESC order covers from the most recent row backwards.

**Logical reads:** ~12,450 **Execution plan:** Clustered Index Scan → Sort (CustomerId ASC, OrderDate DESC) → Segment → Sequence Project (rn + RunningTotal) → SELECT

```sql
-- This query takes 8 seconds on a 10M row Orders table.
-- Logical reads: 12,450 on Orders, 2 x 450 on Worktable (2 sorts)
-- Memory grant: 400 MB
-- It returns each order with two window function values.

SELECT o.OrderId, o.CustomerId, o.OrderDate, o.TotalAmount,
       ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS RnDesc,
       DENSE_RANK() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate ASC) AS RnAsc
```

<summary>Solution</summary>

**Root cause:** Two window functions with opposite ORDER BY directions (DESC vs ASC). The optimizer creates two separate Sort operators — each sorting 10M rows, each with ~200 MB memory grant. Total memory: 400 MB.

```sql
-- Fixed: homogenize ORDER BY direction to DESC for both

       ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS RnDesc,
       DENSE_RANK() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS RnAscReversed
```

```sql
CREATE INDEX IX_Orders_CustomerId_OrderDate_Desc
    ON dbo.Orders(CustomerId, OrderDate DESC)
    INCLUDE (TotalAmount);

-- With this index, the DESC sort is eliminated (index provides rows in DESC order)
-- Only the ASC sort remains — memory grant drops from 400 MB to 200 MB
```

**After fix — logical reads:** ~12,450 (unchanged) + Worktable: ~450 (one sort)
**Memory grant:** 200 MB (from 400 MB)
**Query time:** ~4.5 seconds (from 8 seconds)

**Challenge 3 — Explain the execution plan**

Given this query:

```sql
SELECT o.OrderId, o.TotalAmount,
       ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn,
       SUM(o.TotalAmount) OVER(PARTITION BY o.CustomerId) AS CustTotal
FROM dbo.Orders AS o;
```

The execution plan shows: `Clustered Index Scan → Sort1 → Segment → Sequence Project (ROW_NUMBER) → Sort2 → Segment → Sequence Project (SUM)`. Why are there two Sort operators when the second window function (SUM OVER PARTITION BY CustomerId) has no ORDER BY clause?

<details>
<summary>Solution</summary>

**Why two Sorts:** The first window function `ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC)` requires sorting by `CustomerId, OrderDate DESC`. The second window function `SUM(TotalAmount) OVER(PARTITION BY CustomerId)` — even without an ORDER BY — has a default frame specification. In SQL Server, the default frame for aggregate window functions is `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`. The RANGE default requires ordered input because it needs to know the logical position of the current row within the partition. Without an explicit ORDER BY, the default frame still applies — and ROWS/RANGE semantics WITH an implicit ordering cause the optimizer to request sorted input. However, without any ORDER BY clause, the window function sum is constant across the partition (same value for every row), and the optimizer SHOULD recognize this and avoid the second sort. In practice, SQL Server may not always optimize this case, especially in older compatibility levels.

**To fix:** Add an explicit ORDER BY that matches the first window function, or add `ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING` (which does not require ordering):

```sql
-- Fix: specify ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
-- This tells the optimizer no ordering needed — constant aggregate per partition
SELECT o.OrderId, o.TotalAmount,
       ROW_NUMBER() OVER(PARTITION BY o.CustomerId ORDER BY o.OrderDate DESC) AS rn,
       SUM(o.TotalAmount) OVER(PARTITION BY o.CustomerId
           ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING) AS CustTotal
FROM dbo.Orders AS o;
-- Plan: Clustered Index Scan → Sort1 → Segment → Sequence Project (both functions)
-- One Sort shared between both window functions

-- Or use a subquery with GROUP BY and join for the constant aggregate
```

**Challenge 4 — Diagnose the concurrency problem**

At peak hours (100 concurrent users), window function queries on a 50M row SalesTransactions table start timing out. The queries use `ROW_NUMBER() OVER(PARTITION BY StoreId ORDER BY SaleDate DESC)` to get recent transactions per store. `sys.dm_os_wait_stats` shows `RESOURCE_SEMAPHORE` as the top wait type with 500,000 ms total wait time. What is happening and what are three possible fixes?

<details>
<summary>Solution</summary>

**Root cause:** Each window function query allocates a memory grant for the Sort operator. On a 50M row table, the Sort memory grant is approximately 1-2 GB per query. With 100 concurrent queries, the total memory grant demand is 100-200 GB — far exceeding the server's available memory. Queries queue on `RESOURCE_SEMAPHORE` waiting for memory grants. The wait time accumulates.

**Detection query:**
```sql
SELECT wait_type, waiting_tasks_count, wait_time_ms, max_wait_time_ms,
       signal_wait_time_ms
FROM sys.dm_os_wait_stats
WHERE wait_type = 'RESOURCE_SEMAPHORE'
ORDER BY wait_time_ms DESC;

-- See also:
SELECT session_id, grant_time, requested_memory_kb, granted_memory_kb,
       required_memory_kb, query_cost, timeout_sec
FROM sys.dm_exec_query_memory_grants
WHERE grant_time IS NULL  -- queries waiting for grant
ORDER BY required_memory_kb DESC;
```

**Fix 1 — Index elimination (best):** Create a covering index on `(StoreId, SaleDate DESC) INCLUDE (all other SELECT columns)`. This eliminates the Sort operator entirely. Memory grant drops to 0 MB per query. No more RESOURCE_SEMAPHORE waits.

```sql
CREATE INDEX IX_SalesTransactions_StoreId_SaleDate
    ON dbo.SalesTransactions(StoreId, SaleDate DESC)
    INCLUDE (Quantity, UnitPrice, ProductId);
```

**Fix 2 — Reduce concurrency with paging:** Instead of all 100 users running the full query simultaneously, limit to 10 concurrent queries using a SemaphoreSlim in the application layer.

**Fix 3 — Resource Governor (SQL Server Enterprise):** Limit memory grant per query or per workload group:

```sql
CREATE WORKLOAD GROUP WindowFunctionQueries
    WITH (REQUEST_MAX_MEMORY_GRANT_PERCENT = 5);
CREATE RESOURCE POOL WindowFunctionPool
    WITH (MAX_MEMORY_PERCENT = 30);
ALTER WORKLOAD GROUP WindowFunctionQueries
    USING WindowFunctionPool;
```

**In .NET:** Use a connection pool with a limited number of connections for analytics queries. Use `SemaphoreSlim(10, 10)` in the repository to limit concurrent window function executions.

</details>

**Challenge 5 — Design the index**

**Scenario:** An e-commerce analytics dashboard runs three window function queries on a 100M row OrderItems table:

```sql
-- Q1: Rank products by total quantity sold per month (1M executions/day)
SELECT ProductId, SaleMonth,
       SUM(Quantity) AS TotalSold,
       RANK() OVER(PARTITION BY SaleMonth ORDER BY SUM(Quantity) DESC) AS Rank
FROM OrderItems
GROUP BY ProductId, SaleMonth;

-- Q2: Running total of sales per product over time (500K executions/day)
SELECT ProductId, SaleDate,
       SUM(Quantity * UnitPrice) AS DailyRevenue,
       SUM(SUM(Quantity * UnitPrice)) OVER(
           PARTITION BY ProductId ORDER BY SaleDate
           ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS RunningRevenue
FROM OrderItems
GROUP BY ProductId, SaleDate;

-- Q3: Top 10 most recent orders per customer (200K executions/day)
SELECT CustomerId, OrderId, OrderDate, TotalAmount
FROM (
    SELECT CustomerId, OrderId, OrderDate, TotalAmount,
           ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC) AS rn
    FROM Orders
) AS sub WHERE rn <= 10;
```

The table has 80% reads, 20% writes. Design the optimal index strategy.

<details>
<summary>Solution</summary>

**Index 1 — Covering index for Q1 and Q2 (both aggregate + window):**
```sql
CREATE INDEX IX_OrderItems_ProductId_SaleDate
    ON dbo.OrderItems(ProductId, SaleDate)
    INCLUDE (Quantity, UnitPrice, CustomerId);
```

**Why:** Q1 groups by ProductId and SaleMonth; Q2 groups by ProductId and SaleDate. Both need ProductId as the leading column for PARTITION BY. The index provides ordered input for the window function in Q2 (ORDER BY SaleDate). The Sort for Q1's `RANK() OVER(PARTITION BY SaleMonth ORDER BY SUM(Quantity) DESC)` cannot use this index because SaleMonth is derived — the Sort is unavoidable for Q1 unless a computed column is used. The index reduces scan cost by being narrower than the clustered index.

**Index 2 — For Q3 (ROW_NUMBER per customer):**
```sql
CREATE INDEX IX_Orders_CustomerId_OrderDate
    ON dbo.Orders(CustomerId, OrderDate DESC)
    INCLUDE (TotalAmount);
```

**Why:** Q3 needs `ROW_NUMBER() OVER(PARTITION BY CustomerId ORDER BY OrderDate DESC)`. This index provides the exact ordering needed, eliminating the Sort. The index is covering (includes TotalAmount), so no key lookups. Estimated memory grant reduction: from 200 MB to 0 MB.

**Index 3 — Computed column for Q1's SaleMonth (optional, for sort elimination):**
```sql
ALTER TABLE dbo.OrderItems ADD SaleMonth AS DATETIME2(0)
    (DATEADD(day, 1 - DAY(SaleDate), CAST(SaleDate AS DATE))) PERSISTED;

CREATE INDEX IX_OrderItems_SaleMonth_ProductId_Qty
    ON dbo.OrderItems(SaleMonth, ProductId)
    INCLUDE (Quantity);
```
With this computed column and index, Q1's GROUP BY ProductId, SaleMonth can use Stream Aggregate (if index provides ordering), and the RANK window function can avoid the Sort because SaleMonth is ordered.

**Tradeoffs:**
- Index 1 adds ~2 GB storage for 100M rows (key: ProductId 4B + SaleDate 8B; included: Quantity 4B + UnitPrice 9B + CustomerId 4B + overhead ~10B ≈ 40 bytes/row).
- Index 2 adds ~1.5 GB for 50M Orders rows.
- Index 3 adds significant storage (computed column persisted adds 8B per row) and may not be worth it if Q1 is only 1M executions/day but not the top bottleneck.
- Write overhead: 3 additional indexes on OrderItems add ~300% more I/O per INSERT/UPDATE/DELETE (each index leaf insert). With 20% writes, this is significant. Consider removing Index 3 if Q1 performance is acceptable with the Sort.

**What NOT to index:**
- Do not create a separate index for each query. Index 1 covers both Q1 and Q2 (Q1 uses it for scan + stream aggregate; Q2 uses it for sorted range scan + running total).
- Do not create an index on SaleDate alone — the leading column must be ProductId for both Q1 and Q2.

</details>
