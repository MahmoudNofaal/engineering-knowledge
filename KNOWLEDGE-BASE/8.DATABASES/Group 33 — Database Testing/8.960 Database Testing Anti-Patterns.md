# 8.960 — Database Testing Anti-Patterns

---

## 1 — Mocking the Database Layer — False Confidence

### Anti-Pattern 1: Using InMemory Provider for Integration Tests

❌ **Bad Code:**
```csharp
// ❌ Using EF Core InMemory provider to test SQL behavior
public class UserRepositoryTests
{
    [Fact]
    public async Task GetActiveUsers_ShouldReturnOnlyActive()
    {
        var options = new DbContextOptionsBuilder<AppDbContext>()
            .UseInMemoryDatabase("TestDb")
            .Options;

        await using var context = new AppDbContext(options);
        context.Users.AddRange(
            new User { Name = "Alice", IsActive = true },
            new User { Name = "Bob", IsActive = false }
        );
        context.SaveChanges();

        var repo = new UserRepository(context);
        var result = await repo.GetActiveUsersAsync();

        Assert.Equal(1, result.Count());
    }
}
```

**Symptom:** The test passes with InMemory but fails against real SQL Server because the LINQ query uses a method that EF Core cannot translate (e.g., `DateTime.DayOfWeek`, `string.Split`, `LastOrDefault()`). InMemory executes all LINQ in-memory, masking provider-specific SQL translation failures.

✅ **Fix:** Use a real database (SQL Server container or LocalDB) for integration tests. Keep InMemory only for unit tests that do not test query behavior.

```csharp
// ✅ Use TestContainers or SQL Server LocalDB
public class UserRepositoryTests : IClassFixture<DatabaseFixture>
{
    private readonly AppDbContext _context;
    private readonly UserRepository _repository;

    public UserRepositoryTests(DatabaseFixture fixture)
    {
        _context = fixture.Context;
        _repository = new UserRepository(_context);
    }

    [Fact]
    public async Task GetActiveUsers_ShouldReturnOnlyActive()
    {
        _context.Users.AddRange(
            new User { Name = "Alice", IsActive = true },
            new User { Name = "Bob", IsActive = false }
        );
        await _context.SaveChangesAsync();

        var result = await _repository.GetActiveUsersAsync();

        Assert.Equal(1, result.Count());
    }
}
```

**Cost:** Using InMemory for SQL behavior tests leads to production failures that could have been caught in CI. The cost of debugging a SQL translation issue in production is 100x the cost of a proper integration test.

### Anti-Pattern 2: Mocking DbSet / IDbConnection

❌ **Bad Code:**
```csharp
// ❌ Mocking DbSet<T> — recreating EF Core internals
[Fact]
public async Task GetUserById_ShouldReturnUser()
{
    var data = new List<User>
    {
        new User { Id = 1, Name = "Alice" }
    }.AsQueryable();

    var mockSet = new Mock<DbSet<User>>();
    mockSet.As<IQueryable<User>>().Setup(m => m.Provider).Returns(data.Provider);
    mockSet.As<IQueryable<User>>().Setup(m => m.Expression).Returns(data.Expression);
    mockSet.As<IQueryable<User>>().Setup(m => m.ElementType).Returns(data.ElementType);
    mockSet.As<IQueryable<User>>().Setup(m => m.GetEnumerator()).Returns(data.GetEnumerator());

    var mockContext = new Mock<AppDbContext>();
    mockContext.Setup(c => c.Users).Returns(mockSet.Object);

    var repo = new UserRepository(mockContext.Object);
    var result = await repo.GetByIdAsync(1);

    Assert.NotNull(result);
    Assert.Equal("Alice", result.Name);
}
```

**Symptom:** The mock setup is fragile — every change to how `DbSet` is accessed breaks the mock, not because the code is wrong but because the mock is incomplete. The mock also does not test the actual LINQ query translation.

✅ **Fix:** Use a real `DbContext` with a real or SQLite database. For pure unit tests that do not test query logic, inject the dependency at a higher level (e.g., inject `IUserRepository` and mock that).

```csharp
// ✅ Mock the repository interface, not DbSet
var mockRepo = new Mock<IUserRepository>();
mockRepo.Setup(r => r.GetByIdAsync(1))
    .ReturnsAsync(new User { Id = 1, Name = "Alice" });

var service = new UserService(mockRepo.Object);
var result = await service.GetUserDisplayNameAsync(1);
Assert.Equal("Alice", result);
```

**Cost:** Mocking `DbSet` adds ~30 lines of fragile setup per test. Each refactoring of the data access layer requires updating all mocks. Mocking `IDbConnection` for Dapper is similarly fragile — you end up mocking `QueryAsync` return values, which tests nothing about your SQL.

## 2 — Fake Databases and Missing SQL Fidelity

### Anti-Pattern 3: Not Testing Against a Real Database

❌ **Bad Code:**
```csharp
// ❌ All tests use SQLite InMemory — misses SQL Server-specific behavior
public class ProductRepositoryTests
{
    private readonly AppDbContext _context;

    public ProductRepositoryTests()
    {
        var options = new DbContextOptionsBuilder<AppDbContext>()
            .UseSqlite("DataSource=:memory:")
            .Options;
        _context = new AppDbContext(options);
        _context.Database.OpenConnection();
        _context.Database.EnsureCreated();
    }
}
```

**Symptom:** SQLite does not support SQL Server features used by the application: schema-bound views, `OUTPUT INSERTED`, table-valued parameters, `FORMAT()`, `STRING_AGG()`, `STUFF()`, full-text search, temporal tables, sequences, and many data types (`datetime2`, `hierarchyid`, `geography`). Tests pass against SQLite but fail against SQL Server.

✅ **Fix:** Run a subset of tests against SQL Server (via TestContainers or LocalDB). Use SQLite only for fast unit-level tests that do not depend on SQL Server-specific features. Categorize tests with `[Trait("Category", "SQLServerRequired")]`.

```csharp
// ✅ Run critical queries against real SQL Server
public class ProductRepositoryIntegrationTests : IClassFixture<DatabaseFixture>
{
    private readonly AppDbContext _context;

    public ProductRepositoryIntegrationTests(DatabaseFixture fixture)
    {
        _context = fixture.Context; // Real SQL Server
    }

    [Fact]
    [Trait("Category", "SQLServerRequired")]
    public async Task SearchProducts_ShouldUseFullTextIndex()
    {
        var repo = new ProductRepository(_context);
        var results = await repo.SearchAsync("wireless mouse");
        Assert.NotEmpty(results);
    }
}
```

**Cost:** Deploying untested SQL Server-specific queries to production causes runtime exceptions. The cost of a hotfix deployment is typically 10-100x the cost of a proper test environment.

### Anti-Pattern 4: SQLite as a "Good Enough" SQL Server Stand-In

❌ **Bad Code:**
```csharp
// ❌ Assuming SQLite behaves identically to SQL Server
optionsBuilder.UseSqlite("DataSource=:memory:");

// Queries that work on SQLite but fail on SQL Server:
var users = await context.Users
    .Where(u => u.CreatedAt.Date == today)
    .ToListAsync(); // SQLite: works. SQL Server: DateTime.Date not translated

var counts = await context.Orders
    .GroupBy(o => o.Status)
    .Select(g => new { Status = g.Key, Count = g.Count() })
    .ToListAsync(); // Works on both, but SQL Server may generate different SQL
```

**Symptom:** The application works fine in development (SQLite) but fails in production (SQL Server) because:
- SQLite supports `DateTime.Date` in LINQ; SQL Server does not
- SQLite has different string comparison semantics (case-sensitive by default vs case-insensitive for SQL Server)
- SQLite type affinity means any type of data can be stored in any column
- SQLite does not enforce length constraints on `NVARCHAR(50)`

✅ **Fix:** Use the same database provider in tests as in production. If that is not possible, maintain a known list of unsupported operations and validate tests against it.

**Cost:** Debugging production-only failures is the most expensive type of bug — no repro in development, no repro in staging, only in production under load.

## 3 — Shared State and Flaky Tests

### Anti-Pattern 5: Shared Mutable State Between Tests

❌ **Bad Code:**
```csharp
// ❌ Tests depend on data left by previous tests
public class OrderTests
{
    private static int _nextOrderId = 1;
    private static readonly List<Order> SharedOrders = new();

    [Fact]
    public void CreateOrder_ShouldAddOrder()
    {
        var order = new Order { Id = _nextOrderId++, Total = 100 };
        SharedOrders.Add(order);
        Assert.Equal(1, SharedOrders.Count); // ❌ Fails if run second
    }

    [Fact]
    public void GetOrder_ShouldReturnOrder()
    {
        // Depends on CreateOrder having run first
        var order = SharedOrders.FirstOrDefault(o => o.Id == 1);
        Assert.NotNull(order); // ❌ Fails if run in isolation
    }
}
```

**Symptom:** Tests pass when run in a specific order but fail in random order. xUnit, NUnit, and MSTest all randomize test order by default. The test suite becomes unreliable — developers stop trusting the tests.

✅ **Fix:** Each test must set up its own data and clean up after itself. Use database transactions that roll back (Per-Test isolation) or Respawn between tests (Per-Suite isolation). Never share mutable state between tests.

```csharp
// ✅ Each test creates and owns its data
public class OrderTests : IDisposable
{
    private readonly PerTestFixture _fixture;

    public OrderTests()
    {
        _fixture = new PerTestFixture("conn string");
    }

    [Fact]
    public async Task CreateOrder_ShouldAddOrder()
    {
        var repo = new OrderRepository(_fixture.Context);
        var order = new Order { Total = 100 };
        await repo.CreateAsync(order);

        var saved = await _fixture.Context.Orders.FindAsync(order.Id);
        Assert.NotNull(saved);
    }

    [Fact]
    public async Task GetOrder_ShouldReturnOrder_WhenExists()
    {
        var repo = new OrderRepository(_fixture.Context);
        var order = new Order { Total = 200 };
        await repo.CreateAsync(order);

        var result = await repo.GetByIdAsync(order.Id);
        Assert.NotNull(result);
        Assert.Equal(200, result.Total);
    }

    public void Dispose() => _fixture.Dispose(); // Rollback
}
```

**Cost:** Flaky tests waste developer time (10-30 minutes per investigation), reduce trust in the test suite, and eventually lead to tests being disabled entirely.

### Anti-Pattern 6: Non-deterministic Test Data

❌ **Bad Code:**
```csharp
// ❌ Tests produce different results on every run
[Fact]
public async Task CreateOrder_ShouldGenerateOrderNumber()
{
    var order = new Order
    {
        Total = 100m,
        CreatedAt = DateTime.UtcNow, // ❌ Different every time
        OrderGuid = Guid.NewGuid()    // ❌ Different every time
    };
    await _repository.CreateAsync(order);

    Assert.True(order.OrderNumber > 0);
}
```

**Symptom:** A test that uses `DateTime.UtcNow`, `Guid.NewGuid()`, or `Random` in seed data may pass or fail depending on timing or random values. A test that creates a user with `Guid.NewGuid()` as the email will pass every time individually but makes assertions difficult.

✅ **Fix:** Use deterministic, semantically meaningful test data. Use fixed dates like `new DateTime(2024, 1, 1)` and fixed GUIDs like `Guid.Parse("A1B2C3D4-...")`.

```csharp
// ✅ Deterministic test data
[Fact]
public async Task CreateOrder_ShouldGenerateOrderNumber()
{
    var order = new Order
    {
        Total = 100m,
        CreatedAt = new DateTime(2024, 6, 15, 10, 30, 0, DateTimeKind.Utc),
        OrderGuid = Guid.Parse("11111111-1111-1111-1111-111111111111")
    };
    await _repository.CreateAsync(order);

    Assert.True(order.OrderNumber > 0);
}
```

**Cost:** Non-deterministic tests are flaky by definition. Each failure requires investigation to determine whether it is a real bug or a test artifact.

## 4 — Test Structure and Scope

### Anti-Pattern 7: Testing Too Much in One Test

❌ **Bad Code:**
```csharp
// ❌ One test that does everything — unclear failure
[Fact]
public async Task FullOrderLifecycle_ShouldWork()
{
    // Seed
    var user = new User { Name = "Alice" };
    _context.Users.Add(user);
    await _context.SaveChangesAsync();

    // Create order
    var order = new Order { UserId = user.Id, Total = 100 };
    _context.Orders.Add(order);
    await _context.SaveChangesAsync();

    // Add order line
    var line = new OrderLine { OrderId = order.Id, Product = "Widget", Quantity = 1, Price = 100 };
    _context.OrderLines.Add(line);
    await _context.SaveChangesAsync();

    // Update order total
    order.Total = 150;
    await _context.SaveChangesAsync();

    // Cancel order
    order.Status = "Cancelled";
    await _context.SaveChangesAsync();

    // Assert everything
    Assert.Equal(150, order.Total);
    Assert.Equal("Cancelled", order.Status);
    var dbOrder = await _context.Orders.Include(o => o.Lines).FirstAsync(o => o.Id == order.Id);
    Assert.Single(dbOrder.Lines);
    Assert.Equal("Widget", dbOrder.Lines[0].Product);
}
```

**Symptom:** When this test fails, it is unclear which step caused the failure. The test mixes seed, query, update, and assertion in one long script. Debugging requires stepping through the entire test.

✅ **Fix:** Each test should verify one behavior. Use Arrange-Act-Assert strictly. Create helper methods for common setup.

```csharp
// ✅ Each test verifies one thing — clear failure signal
[Fact]
public async Task CreateOrder_ShouldPersistOrder()
{
    var order = new Order { UserId = 1, Total = 100 };
    _context.Orders.Add(order);
    await _context.SaveChangesAsync();

    var saved = await _context.Orders.FindAsync(order.Id);
    Assert.NotNull(saved);
    Assert.Equal(100, saved.Total);
}

[Fact]
public async Task UpdateOrderTotal_ShouldModifyTotal()
{
    var order = new Order { UserId = 1, Total = 100 };
    _context.Orders.Add(order);
    await _context.SaveChangesAsync();

    order.Total = 200;
    await _context.SaveChangesAsync();

    var saved = await _context.Orders.FindAsync(order.Id);
    Assert.Equal(200, saved.Total);
}
```

**Cost:** A single failing test that covers multiple behaviors requires manual investigation. Multiply by 50 such tests and you lose half a day per CI failure.

### Anti-Pattern 8: Not Testing Transaction Rollback

❌ **Bad Code:**
```csharp
// ❌ Only test the happy path — never verify rollback
[Fact]
public async Task TransferFunds_ShouldSucceed()
{
    var from = new Account { Balance = 500 };
    var to = new Account { Balance = 100 };
    _context.Accounts.AddRange(from, to);
    await _context.SaveChangesAsync();

    await _service.TransferAsync(from.Id, to.Id, 200);

    Assert.Equal(300, from.Balance);
    Assert.Equal(300, to.Balance);
}
```

**Symptom:** The transfer logic is not wrapped in a transaction. If the credit to account B succeeds but the debit from account A fails, the money is created out of thin air. The test never catches this because it only tests the success path.

✅ **Fix:** Write a test that forces the transaction to fail and verify that no partial changes persist.

```csharp
// ✅ Verify rollback actually reverts changes
[Fact]
public async Task TransferFunds_ShouldRollback_WhenCreditFails()
{
    var from = new Account { Balance = 500 };
    var to = new Account { Balance = 100 };
    _context.Accounts.AddRange(from, to);
    await _context.SaveChangesAsync();

    // Force failure in the credit step
    try
    {
        await _service.TransferWithFailureAsync(from.Id, to.Id, 200);
    }
    catch
    {
        // Expected
    }

    // Assert both accounts are unchanged
    var fromReloaded = await _context.Accounts.FindAsync(from.Id);
    var toReloaded = await _context.Accounts.FindAsync(to.Id);
    Assert.Equal(500, fromReloaded.Balance);
    Assert.Equal(100, toReloaded.Balance);
}
```

**Cost:** Missing rollback tests lead to data corruption bugs in production. A single data corruption incident can cost thousands in recovery and reputational damage.

## 5 — Performance and Respawn Overuse

### Anti-Pattern 9: Over-relying on Respawn

❌ **Bad Code:**
```csharp
// ❌ Respawn after every test — 500ms overhead even when not needed
public class UserTests : IAsyncLifetime
{
    private static readonly Checkpoint _checkpoint = new();

    public async Task InitializeAsync()
    {
        await _checkpoint.Reset(_connection); // 500ms per test
    }

    [Fact]
    public async Task GetUser_ShouldReturnUser() { /* single connection, no DDL */ }
    [Fact]
    public async Task GetUsers_ShouldReturnAll() { /* single connection, no DDL */ }
    // 20 tests × 500ms = 10 seconds just for cleanup
}
```

**Symptom:** The test suite is 10x slower than necessary because every test uses Respawn even though 90% of tests only need a single connection and no DDL. Developers stop running the tests locally because they take too long.

✅ **Fix:** Use transaction rollback for tests that do not need Respawn. Only use Respawn for tests that require multiple connections or execute DDL.

```csharp
// ✅ Transaction rollback for single-connection tests (50ms vs 500ms)
public class UserTests : IDisposable
{
    private readonly PerTestFixture _fixture;

    public UserTests()
    {
        _fixture = new PerTestFixture("conn string");
    }

    [Fact]
    public async Task GetUser_ShouldReturnUser()
    {
        // Test logic — rollback on dispose
    }

    public void Dispose() => _fixture.Dispose(); // ~50ms
}
```

**Cost:** A test suite with 500 tests that takes 5 minutes with rollback takes 50 minutes with Respawn. That 45-minute difference means developers either skip tests or context-switch while waiting.

### Anti-Pattern 10: Testing with Too-Small Data

❌ **Bad Code:**
```csharp
// ❌ Performance tests with 100 rows — doesn't test the query optimizer
[Fact]
public async Task SearchOrders_ShouldBeFast()
{
    for (int i = 0; i < 100; i++)
    {
        _context.Orders.Add(new Order { /* ... */ });
    }
    await _context.SaveChangesAsync();

    var sw = Stopwatch.StartNew();
    var results = await _service.SearchAsync("test");
    sw.Stop();

    Assert.True(sw.ElapsedMilliseconds < 100, "Query too slow");
}
```

**Symptom:** The query uses an index scan on 100 rows and passes the test. In production with 10M rows, the query uses a table scan and takes 30 seconds. The small-data test gives false confidence about performance.

✅ **Fix:** Use data volumes proportional to production. For performance tests, use at least 10% of expected production volume or use a scaling factor. If the query is expected to handle 1M rows, test with 100K–1M rows.

```csharp
// ✅ Test with production-scale data
[Fact]
public async Task SearchOrders_ShouldUseIndex()
{
    // Seed 100K orders
    var orders = Enumerable.Range(1, 100_000).Select(i => new Order
    {
        OrderNumber = $"ORD-{i:D6}",
        Total = i * 10m,
        Status = i % 3 == 0 ? "Shipped" : "Pending"
    });
    _context.Orders.AddRange(orders);
    await _context.SaveChangesAsync();

    // Act — measure
    var sw = Stopwatch.StartNew();
    var results = await _service.SearchAsync("ORD-050000");
    sw.Stop();

    Assert.Single(results);
    Assert.True(sw.ElapsedMilliseconds < 500, "Index seek should be < 500ms");
}
```

**Cost:** A query that performs well with 100 rows but poorly with 10M rows will cause a production incident. The cost of discovering this during a traffic spike is far higher than catching it in a pre-production performance test.

## 6 — CI Integration and Developer Workflow

### Anti-Pattern 11: No CI Pipeline Database Tests

❌ **Bad Code:**
```csharp
// ❌ "Works on my machine" — SQL Server container tests never run in CI
// .csproj has no CI-specific test configuration
// Tests are marked [Fact] but never executed in CI pipeline
```

**Symptom:** Developers run database tests locally against their LocalDB instance. The tests pass. The PR is merged. In CI, the tests are either not included in the pipeline or rely on a SQL Server instance that does not exist in the CI environment. A schema change that breaks the build is only discovered when another developer pulls the latest code and runs the tests.

✅ **Fix:** Include SQL Server container in the CI pipeline. Use Docker services in GitHub Actions, Azure Pipelines, or whatever CI system is in use.

```yaml
# ✅ GitHub Actions with SQL Server container
jobs:
  test:
    runs-on: ubuntu-latest
    services:
      sqlserver:
        image: mcr.microsoft.com/mssql/server:2022-latest
        env:
          SA_PASSWORD: "TestPassword123!"
          ACCEPT_EULA: "Y"
        ports:
          - 1433:1433
    steps:
      - uses: actions/checkout@v4
      - run: dotnet test
        env:
          ConnectionStrings__Default: "Server=localhost;Database=TestDb;User Id=sa;Password=TestPassword123!;TrustServerCertificate=True;"
```

**Cost:** Database tests that never run in CI are not tests — they are local scripts. The gap between local and CI environments ensures that integration issues are discovered late, often at deployment time.

### Anti-Pattern 12: Database Tests So Slow They Don't Run

❌ **Bad Code:**
```csharp
// ❌ Every test sets up the full schema from scratch
public class SlowDatabaseTests
{
    public SlowDatabaseTests()
    {
        var options = new DbContextOptionsBuilder<AppDbContext>()
            .UseSqlServer("Server=.;Database=TestDb;...")
            .Options;
        var context = new AppDbContext(options);
        context.Database.EnsureDeleted();  // Drops entire DB
        context.Database.Migrate();         // Runs all migrations — 30s
    }
}
```

**Symptom:** Running the full migration for each test class takes 30 seconds. With 20 test classes, the suite takes 10 minutes. Developers run the tests once, then skip them for the rest of the day. Bugs introduced in the second commit are not caught until the CI pipeline runs.

✅ **Fix:** Use a shared database setup (migrations run once per test run) with per-test transactions. Categorize tests so the slowest tests can be run selectively.

```csharp
// ✅ Shared database setup — migrations run once
public class DatabaseFixture : IAsyncLifetime
{
    private static readonly object _lock = new();
    private static bool _initialized;

    public AppDbContext Context { get; private set; }

    public async Task InitializeAsync()
    {
        var options = new DbContextOptionsBuilder<AppDbContext>()
            .UseSqlServer("Server=.;Database=TestDb;...")
            .Options;

        Context = new AppDbContext(options);

        lock (_lock)
        {
            if (!_initialized)
            {
                Context.Database.EnsureDeleted();
                Context.Database.Migrate();
                _initialized = true;
            }
        }
    }

    public async Task DisposeAsync()
    {
        await Context.DisposeAsync();
    }
}
```

**Cost:** A test suite that takes 10 minutes to run locally is a barrier to development. Developers will avoid running it, meaning bugs slip through that could have been caught with a 2-minute test suite.

## 7 — Migration and Deployment Testing

### Anti-Pattern 13: Migrations Not Tested

❌ **Bad Code:**
```csharp
// ❌ Migration applies in production for the first time
// Developer adds a migration:
// dotnet ef migrations add AddIsPremiumColumn
// No test verifies that the migration actually applies
```

**Symptom:** The migration works on the developer's local database (which has 100 rows and no production data patterns). In production, the migration:
- Times out because it tries to update all 10M rows in a single transaction
- Fails because there is a check constraint that the developer's database does not have
- Runs for hours because it acquires schema locks that block all other queries

✅ **Fix:** Write a migration test that applies the migration to a copy of the production schema (or a close approximation).

```csharp
// ✅ Migration test — applies migration and verifies outcome
public class MigrationTests
{
    private const string ConnectionString = "Server=localhost;Database=MigrationTest;User Id=sa;Password=TestPassword123!;TrustServerCertificate=True;";

    [Fact]
    public async Task AddIsPremiumColumn_ShouldApplySuccessfully()
    {
        // Arrange — start from previous migration
        var options = new DbContextOptionsBuilder<AppDbContext>()
            .UseSqlServer(ConnectionString)
            .Options;

        var context = new AppDbContext(options);
        await context.Database.EnsureDeletedAsync();
        await context.Database.MigrateAsync(); // Applies ALL migrations

        // Assert — verify the new column exists
        var hasColumn = await context.Database.SqlQueryRaw<bool>(
            "SELECT COUNT(*) FROM INFORMATION_SCHEMA.COLUMNS WHERE TABLE_NAME='Users' AND COLUMN_NAME='IsPremium'"
        ).FirstOrDefaultAsync();

        Assert.True(hasColumn, "Migration did not create IsPremium column");
    }
}
```

**Cost:** A failed migration in production requires a rollback, causes downtime, and may leave the database in an inconsistent state. The cost of a production migration failure is typically measured in hours of engineering time and minutes of downtime.

### Anti-Pattern 14: Deadlock / Concurrency Tests Disabled

❌ **Bad Code:**
```csharp
// ❌ "They're flaky" — deadlock tests disabled
public class ConcurrencyTests
{
    [Fact(Skip = "Flaky — need to investigate")]
    public async Task ConcurrentOrderUpdate_ShouldNotDeadlock()
    {
        // Test that simulates two concurrent order updates
    }
}
```

**Symptom:** The concurrency test is flaky because it actually tests real race conditions. The team disables it because it "fails randomly." The real problem is that the application has a deadlock bug that only manifests under load. Disabling the test does not fix the bug — it just hides it.

✅ **Fix:** Flaky concurrency tests are testing real issues. Run them multiple times in CI to get statistical confidence. Fix the underlying concurrency issue instead of disabling the test.

```csharp
// ✅ Run concurrency tests multiple times
[Fact]
public async Task ConcurrentOrderUpdate_ShouldNotDeadlock()
{
    var tasks = new List<Task>();
    for (int i = 0; i < 10; i++)
    {
        tasks.Add(Task.Run(async () =>
        {
            await using var context = CreateContext();
            var repo = new OrderRepository(context);
            // Use retry logic for deadlock victims
            await repo.UpdateOrderWithRetryAsync(orderId, 100);
        }));
    }

    await Task.WhenAll(tasks);
    // If this fails, there is a real deadlock issue
}
```

```yaml
# CI — run flaky tests multiple times
- name: Run concurrency tests 5 times
  run: |
    for i in 1..5; do
      dotnet test --filter "Category=Concurrency" || true
    done
```

**Cost:** Disabling concurrency tests because they are flaky is like removing the check engine light because it is annoying. The underlying problem still exists and will manifest in production under the right conditions.

## 8 — Configuration and Environment

### Anti-Pattern 15: Connection String Hardcoded in Tests

❌ **Bad Code:**
```csharp
// ❌ Different devs connect to different databases
public class DatabaseTests
{
    private const string ConnectionString = "Server=.;Database=MyApp;Trusted_Connection=True;";
    // Developer A: LocalDB instance named "MyApp"
    // Developer B: SQL Express instance named "MyApp"
    // Developer C: Full SQL Server instance with different auth
}
```

**Symptom:** Tests that pass on Developer A's machine fail on Developer B's machine because they have different SQL Server configurations (different versions, different collation, different instance names). Tests that pass locally fail in CI because the CI connection string is different.

✅ **Fix:** Use environment variables or a `appsettings.Test.json` for the connection string. Use a default that works for local development and override in CI.

```csharp
// ✅ Connection string from configuration
public static class TestConfig
{
    public static string ConnectionString =>
        Environment.GetEnvironmentVariable("TEST_CONNECTION_STRING")
        ?? "Server=(localdb)\\MSSQLLocalDB;Database=TestDb;Trusted_Connection=True;";
}

public class DatabaseTests
{
    private readonly string _connectionString = TestConfig.ConnectionString;
    // ...
}
```

```json
// appsettings.Test.json
{
  "ConnectionStrings": {
    "Default": "Server=(localdb)\\MSSQLLocalDB;Database=TestDb;Trusted_Connection=True;"
  }
}
```

**Cost:** Connection string inconsistencies cause "works on my machine" failures that waste hours of debugging across the team.

### Anti-Pattern 16: No Cleanup After Failed Tests

❌ **Bad Code:**
```csharp
// ❌ Failed test leaves data, subsequent tests fail
[Fact]
public async Task InsertUser_ShouldSucceed()
{
    var user = new User { Email = "test@example.com", Name = "Test" };
    await _repository.InsertAsync(user);
    Assert.NotNull(user.Id);
    // No cleanup — if this assert fails, the user remains in the database
}

[Fact]
public async Task InsertUser_DuplicateEmail_ShouldThrow()
{
    // Depends on previous test having completed successfully AND cleaned up
    var user = new User { Email = "test@example.com", Name = "Test" };
    await _repository.InsertAsync(user); // May throw if previous test left data
}
```

**Symptom:** Test B fails because test A left data behind after failing. The developer thinks test B is the problem, debugs it for 30 minutes, only to discover that test A did not clean up. The suite is unreliable — tests must pass in a specific order.

✅ **Fix:** Use transactions (rollback on dispose) or ensure cleanup always runs (try/finally or `IAsyncLifetime`).

```csharp
// ✅ Always clean up — even on failure
public class UserTests : IAsyncLifetime
{
    private readonly AppDbContext _context;
    private readonly UserRepository _repository;

    public async Task InitializeAsync()
    {
        // Set up clean state
        await CleanupAsync();
    }

    public async Task DisposeAsync()
    {
        // Clean up even if test failed
        await CleanupAsync();
    }

    private async Task CleanupAsync()
    {
        _context.Users.RemoveRange(_context.Users);
        await _context.SaveChangesAsync();
    }

    [Fact]
    public async Task InsertUser_ShouldSucceed()
    {
        var user = new User { Email = "test@example.com", Name = "Test" };
        await _repository.InsertAsync(user);
        Assert.NotNull(user.Id);
    }
}
```

**Cost:** Debugging cascading test failures is one of the most common time-wasters in test maintenance. A single uncleaned record can cause a cascade of 20 failures that all appear unrelated.

## 9 — Advanced Anti-Patterns

### Anti-Pattern 17: Testing Through the Web Layer (Skip the Database Layer)

❌ **Bad Code:**
```csharp
// ❌ Testing database behavior through HTTP endpoints
[Fact]
public async Task CreateUser_ApiTest()
{
    var client = _factory.CreateClient();
    var response = await client.PostAsJsonAsync("/api/users", new { Name = "Alice" });
    response.EnsureSuccessStatusCode();

    // Indirectly verifying database state
    var getResponse = await client.GetAsync("/api/users/1");
    var user = await getResponse.Content.ReadFromJsonAsync<User>();
    Assert.Equal("Alice", user.Name);
}
```

**Symptom:** Testing database behavior through the web API layer is slow (HTTP overhead), fragile (any change to the API surface breaks the test), and tests too many concerns at once (routing, serialization, auth, business logic, database).

✅ **Fix:** Test database behavior at the repository/service layer. Keep API tests focused on API contract concerns (status codes, headers, serialization).

```csharp
// ✅ Test database logic directly at the repository layer
[Fact]
public async Task UserRepository_InsertAndRetrieve()
{
    var repo = new UserRepository(_context);
    var user = new User { Name = "Alice", Email = "alice@example.com" };
    await repo.InsertAsync(user);

    var saved = await repo.GetByEmailAsync("alice@example.com");
    Assert.NotNull(saved);
    Assert.Equal("Alice", saved.Name);
}
```

### Anti-Pattern 18: Using Shared Databases Between Development and Testing

❌ **Bad Code:**
```csharp
// ❌ Tests run against the developer's local database
private const string ConnectionString = "Server=.;Database=MyApp_Dev;Trusted_Connection=True;";
```

**Symptom:** The developer's local database contains a mix of test data, manual testing data, and application seeding data. Tests fail unpredictably because the state is never clean. Running tests requires the developer to manually reset the database.

✅ **Fix:** Always use a dedicated test database that is created/destroyed by the test framework. Use TestContainers or a uniquely named database per test run.

```csharp
// ✅ Dedicated test database
[CollectionDefinition("Database Tests")]
public class DatabaseTestCollection : ICollectionFixture<DatabaseFixture> { }

public class DatabaseFixture : IAsyncLifetime
{
    private const string BaseConnectionString = "Server=.;Trusted_Connection=True;";
    private readonly string _dbName = $"TestDb_{Guid.NewGuid():N}";

    public async Task InitializeAsync()
    {
        var masterConnection = new SqlConnection(BaseConnectionString);
        await masterConnection.OpenAsync();
        var cmd = masterConnection.CreateCommand();
        cmd.CommandText = $"CREATE DATABASE [{_dbName}]";
        await cmd.ExecuteNonQueryAsync();
        masterConnection.Close();
    }

    public async Task DisposeAsync()
    {
        var masterConnection = new SqlConnection(BaseConnectionString);
        await masterConnection.OpenAsync();
        var cmd = masterConnection.CreateCommand();
        cmd.CommandText = $"DROP DATABASE IF EXISTS [{_dbName}]";
        await cmd.ExecuteNonQueryAsync();
        masterConnection.Close();
    }
}
```

### Anti-Pattern 19: Forgetting to Dispose DbContext / Connections

❌ **Bad Code:**
```csharp
[Fact]
public async Task LeakTest()
{
    var context = new AppDbContext(options);
    var users = await context.Users.ToListAsync();
    // context not disposed — connection pool exhaustion
}
```

**Symptom:** Each test creates a new `DbContext` (or `SqlConnection`) without disposing it. After 100 tests, the connection pool is exhausted and subsequent tests hang or timeout.

✅ **Fix:** Always use `await using` for `DbContext` and `SqlConnection`.

```csharp
[Fact]
public async Task NoLeakTest()
{
    await using var context = new AppDbContext(options);
    var users = await context.Users.ToListAsync();
    // Automatically disposed
}
```

### Anti-Pattern 20: Ignoring Database Version Differences

❌ **Bad Code:**
```csharp
// ❌ Tests pass on SQL Server 2022 but queries run on SQL Server 2017 in production
// Using STRING_AGG which was introduced in SQL Server 2017
var result = await context.Orders
    .GroupBy(o => o.CustomerId)
    .Select(g => new { CustomerId = g.Key, Products = string.Join(",", g.Select(o => o.Product)) })
    .ToListAsync();
```

**Symptom:** `STRING_AGG` works on SQL Server 2017+ but the production database is SQL Server 2016. The test passes in CI (which uses SQL Server 2022) but the query fails in production.

✅ **Fix:** Match the CI database version to the production database version. Test against the minimum supported version.

```yaml
services:
  sqlserver:
    image: mcr.microsoft.com/mssql/server:2017-latest  # Match production
```

### Anti-Pattern 21: Tests That Pass with No Data

❌ **Bad Code:**
```csharp
[Fact]
public async Task GetOrders_ShouldReturnOrders()
{
    var orders = await _repository.GetOrdersAsync();
    Assert.NotNull(orders);
    // No assertion on count — passes even if the table is empty!
}
```

**Symptom:** The test passes trivially because it does not assert meaningful state. An empty database returns an empty list, which is "not null" and therefore passes.

✅ **Fix:** Always seed the data you need and assert specific values. A test that passes with no data in the database is a false positive.

```csharp
[Fact]
public async Task GetOrders_ShouldReturnAllOrders()
{
    _context.Orders.AddRange(
        new Order { Total = 100 },
        new Order { Total = 200 }
    );
    await _context.SaveChangesAsync();

    var orders = await _repository.GetOrdersAsync();
    Assert.Equal(2, orders.Count());
}
```
