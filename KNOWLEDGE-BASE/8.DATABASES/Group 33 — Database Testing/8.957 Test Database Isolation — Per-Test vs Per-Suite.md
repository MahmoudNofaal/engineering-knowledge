# 8.957 — Test Database Isolation — Per-Test vs Per-Suite

---

## 1 — Core Concept — The Three Levels of Isolation

Test database isolation is the mechanism by which each test (or group of tests) starts with a known, clean database state. Without proper isolation, tests interfere with each other: test A inserts a record, test B expects that record to be absent, and the result is a flaky, order-dependent test suite. Three levels of isolation form a spectrum:

| Level | Scope | Cleanup Mechanism | Typical Speed | Isolation Guarantee |
|-------|-------|-------------------|---------------|---------------------|
| Per-Test | Single test | Transaction rollback | ~50ms | Strongest |
| Per-Suite | Test class | Respawn / schema reset | ~500ms | Strong |
| Per-Assembly | Entire test run | Global fixture + cleanup | ~1000ms | Moderate |

**Per-Test** isolation wraps every test in a database transaction that is rolled back at the end. This is the fastest approach because no data is actually written to disk — the rollback simply discards the transaction log. It works only for tests that use a single database connection and do not issue DDL statements (CREATE/ALTER/DROP), because DDL auto-commits in most databases.

**Per-Suite** isolation uses a tool like Respawn to delete all data from all tables between tests. This is slower than rollback (it runs DELETE statements or TRUNCATE operations), but works across multiple connections and handles DDL. It is appropriate for tests that verify schema changes or use connection pooling with multiple open connections.

**Per-Assembly** isolation uses a xUnit collection fixture that is shared across all test classes in the assembly. The database is set up once (migrations, seed data) and cleaned up once. Tests must cooperate to avoid state pollution — each test must clean up after itself. This is the fastest for total execution time but the most error-prone.

The key insight is that your test suite should mix all three levels. Use Per-Test for the bulk of your tests (they are the fastest and safest), Per-Suite for test classes that need multiple connections or test DDL, and Per-Assembly only for read-only tests or carefully managed integration suites.

Choosing the wrong isolation level leads to:
- Flaky tests (isolation too weak)
- Slow test suites (isolation too strong for every test)
- Tests that pass in isolation but fail in CI (order dependency from per-suite or per-assembly sharing)

## 2 — Per-Test Isolation — Transaction Rollback with EF Core

Per-test isolation using EF Core relies on wrapping each test in an `IDbContextTransaction` that is never committed. At the end of the test, `Dispose()` is called on the transaction, which issues a ROLLBACK. Any data inserted, updated, or deleted during the test is invisible to other tests and is discarded.

```csharp
// PerTestFixture.cs — wraps each test in a transaction
using Microsoft.EntityFrameworkCore.Storage;

public class PerTestFixture : IDisposable
{
    private readonly IDbContextTransaction _transaction;
    private readonly AppDbContext _context;

    public PerTestFixture(string connectionString)
    {
        var options = new DbContextOptionsBuilder<AppDbContext>()
            .UseSqlServer(connectionString)
            .Options;

        _context = new AppDbContext(options);
        _context.Database.EnsureCreated();

        _transaction = _context.Database.BeginTransaction();
    }

    public AppDbContext Context => _context;

    public void Dispose()
    {
        _transaction?.Rollback();
        _transaction?.Dispose();
        _context?.Dispose();
    }
}
```

```csharp
// Usage in xUnit test
public class UserRepositoryTests : IDisposable
{
    private readonly PerTestFixture _fixture;
    private readonly UserRepository _repository;

    public UserRepositoryTests()
    {
        _fixture = new PerTestFixture("Server=.;Database=TestDb;...");
        _repository = new UserRepository(_fixture.Context);
    }

    [Fact]
    public async Task InsertUser_ShouldPersistUser()
    {
        var user = new User { Name = "Alice", Email = "alice@example.com" };
        await _repository.InsertAsync(user);

        var saved = await _fixture.Context.Users
            .FirstOrDefaultAsync(u => u.Email == "alice@example.com");
        Assert.NotNull(saved);
        Assert.Equal("Alice", saved.Name);
    }

    [Fact]
    public async Task InsertUser_ShouldAssignId()
    {
        var user = new User { Name = "Bob", Email = "bob@example.com" };
        await _repository.InsertAsync(user);

        Assert.NotEqual(0, user.Id);
    }

    public void Dispose()
    {
        _fixture?.Dispose();
    }
}
```

The two tests above share the same `PerTestFixture` — each one runs against a fresh database state because the transaction from the previous test was rolled back in `Dispose()`. The first test inserts Alice and asserts; the second test inserts Bob and asserts. Neither test sees the other's data.

**Important:** Call `_transaction.Rollback()` before `Dispose()` to be explicit. In some ADO.NET providers, `Dispose()` alone may commit if the transaction was created with `TransactionScope` or certain connection string flags.

**EF Core-specific considerations:**
- `EnsureCreated()` is called once per fixture. If the fixture is created per test (not shared), this adds overhead. Use a shared schema setup + per-test transaction instead.
- Some migrations use DDL that auto-commits (e.g., CREATE INDEX). Do not run EF Core migrations inside a transaction that you intend to roll back — the DDL will persist.
- Use `context.Database.UseTransaction()` if you need to share an existing transaction with other operations.

## 3 — Per-Test Isolation — Transaction Rollback with Dapper

Dapper does not have a context or unit-of-work abstraction, so per-test isolation requires managing `SqlConnection` and `SqlTransaction` manually. The pattern is the same: open a connection, begin a transaction, pass the transaction to every Dapper operation, and roll back at the end.

```csharp
// DapperPerTestFixture.cs
using System.Data;
using Microsoft.Data.SqlClient;

public class DapperPerTestFixture : IDisposable
{
    private readonly SqlConnection _connection;
    private readonly SqlTransaction _transaction;

    public DapperPerTestFixture(string connectionString)
    {
        _connection = new SqlConnection(connectionString);
        _connection.Open();
        _transaction = _connection.BeginTransaction();
    }

    public IDbConnection Connection => _connection;
    public IDbTransaction Transaction => _transaction;

    public void Dispose()
    {
        _transaction?.Rollback();
        _transaction?.Dispose();
        _connection?.Close();
        _connection?.Dispose();
    }
}
```

```csharp
// Dapper-based repository test with transaction rollback
public class DapperUserRepositoryTests : IDisposable
{
    private readonly DapperPerTestFixture _fixture;
    private readonly DapperUserRepository _repository;

    public DapperUserRepositoryTests()
    {
        _fixture = new DapperPerTestFixture("Server=.;Database=TestDb;...");
        _repository = new DapperUserRepository(_fixture.Connection, _fixture.Transaction);
    }

    [Fact]
    public async Task GetUserById_ShouldReturnCorrectUser()
    {
        // Arrange — insert test data within the same transaction
        const string insertSql = "INSERT INTO Users (Name, Email) VALUES (@Name, @Email); SELECT CAST(SCOPE_IDENTITY() AS INT);";
        var userId = await _fixture.Connection.QuerySingleAsync<int>(
            insertSql,
            new { Name = "Charlie", Email = "charlie@example.com" },
            transaction: _fixture.Transaction);

        // Act
        const string querySql = "SELECT Id, Name, Email FROM Users WHERE Id = @Id";
        var user = await _fixture.Connection.QuerySingleOrDefaultAsync<User>(
            querySql,
            new { Id = userId },
            transaction: _fixture.Transaction);

        // Assert
        Assert.NotNull(user);
        Assert.Equal("Charlie", user.Name);
    }

    [Fact]
    public async Task DeleteUser_ShouldRemoveUser()
    {
        // Arrange
        const string insertSql = "INSERT INTO Users (Name, Email) VALUES (@Name, @Email); SELECT CAST(SCOPE_IDENTITY() AS INT);";
        var userId = await _fixture.Connection.QuerySingleAsync<int>(
            insertSql,
            new { Name = "Delta", Email = "delta@example.com" },
            transaction: _fixture.Transaction);

        // Act
        const string deleteSql = "DELETE FROM Users WHERE Id = @Id";
        await _fixture.Connection.ExecuteAsync(
            deleteSql,
            new { Id = userId },
            transaction: _fixture.Transaction);

        // Assert
        var deleted = await _fixture.Connection.QuerySingleOrDefaultAsync<User>(
            "SELECT Id, Name, Email FROM Users WHERE Id = @Id",
            new { Id = userId },
            transaction: _fixture.Transaction);
        Assert.Null(deleted);
    }

    public void Dispose()
    {
        _fixture?.Dispose();
    }
}
```

**Dapper-specific considerations:**
- You must pass the `transaction` parameter to every Dapper method (`QueryAsync`, `ExecuteAsync`, etc.). If you forget, the operation runs outside the transaction and data persists across tests.
- The `SqlTransaction` is bound to a single `SqlConnection`. If you open a second connection inside a test, that second connection will not see the uncommitted data from the first connection (it would see the database state before the test started). This is correct behavior — it means each test gets a consistent snapshot.
- For Dapper + transaction rollback to work reliably, all database operations in the test must go through the same connection and transaction object. Consider using a `UnitOfWork` wrapper that exposes both `IDbConnection` and `IDbTransaction`.

## 4 — Per-Suite Isolation — Respawn with EF Core

Per-suite isolation uses Respawn to delete all data from all tables between tests. Respawn intelligently handles foreign key constraints: it disables foreign keys, truncates all tables, then re-enables foreign keys. This is necessary for tests that cannot use transaction rollback — for example, tests that verify EF Core migrations (which issue DDL that auto-commits) or tests that open multiple connections.

```csharp
// DatabaseFixture.cs — shared across all tests in a class
using Respawn;

public class DatabaseFixture : IAsyncLifetime
{
    private static readonly Checkpoint _checkpoint = new Checkpoint
    {
        SchemasToInclude = new[] { "dbo" },
        DbAdapter = DbAdapter.SqlServer,
        TablesToIgnore = new[] { "__EFMigrationsHistory" }
    };

    private readonly string _connectionString;
    public AppDbContext Context { get; private set; }

    public DatabaseFixture(string connectionString)
    {
        _connectionString = connectionString;
    }

    public async Task InitializeAsync()
    {
        var options = new DbContextOptionsBuilder<AppDbContext>()
            .UseSqlServer(_connectionString)
            .Options;

        Context = new AppDbContext(options);
        await Context.Database.EnsureCreatedAsync();
        await ResetAsync();
    }

    public async Task ResetAsync()
    {
        await using var connection = new SqlConnection(_connectionString);
        await connection.OpenAsync();
        await _checkpoint.Reset(connection);
    }

    public async Task DisposeAsync()
    {
        await Context.DisposeAsync();
    }
}
```

```csharp
// Test class using per-suite isolation with Respawn
public class OrderServiceTests : IClassFixture<DatabaseFixture>
{
    private readonly DatabaseFixture _fixture;
    private readonly OrderService _service;

    public OrderServiceTests(DatabaseFixture fixture)
    {
        _fixture = fixture;
        _service = new OrderService(fixture.Context);
    }

    [Fact]
    public async Task CreateOrder_ShouldAddOrderLine()
    {
        var order = new Order { CustomerId = 1, Total = 100m };
        await _service.CreateOrderAsync(order);

        var line = new OrderLine { OrderId = order.Id, Product = "Widget", Quantity = 1, Price = 100m };
        await _service.AddOrderLineAsync(order.Id, line);

        var saved = await _fixture.Context.OrderLines
            .Where(ol => ol.OrderId == order.Id)
            .ToListAsync();
        Assert.Single(saved);
        Assert.Equal("Widget", saved[0].Product);

        await _fixture.ResetAsync(); // Clean up for next test
    }

    [Fact]
    public async Task CancelOrder_ShouldSetStatus()
    {
        // Database is clean because ResetAsync was called in previous test
        var order = new Order { CustomerId = 1, Total = 50m };
        await _service.CreateOrderAsync(order);

        await _service.CancelOrderAsync(order.Id);

        var saved = await _fixture.Context.Orders.FindAsync(order.Id);
        Assert.Equal(OrderStatus.Cancelled, saved.Status);

        await _fixture.ResetAsync(); // Clean up for next test
    }
}
```

**Key points about Respawn + EF Core:**
- Respawn resets the *entire* database (all tables). This means seed data is lost after each reset. If you need seed data, re-seed after every `ResetAsync()` call, or use `TablesToIgnore` to skip seed tables.
- The `__EFMigrationsHistory` table should be in `TablesToIgnore` — you do not want Respawn to delete migration history.
- Respawn is slower than transaction rollback because it actually issues DELETE/TRUNCATE statements. On a typical developer machine, expect ~300–800ms per reset depending on table count and row count.
- Respawn does not reseed identity columns by default. Use `Checkpoint.WithReseed = true` to reset identity seeds back to their starting values.

## 5 — Per-Suite Isolation — Respawn with Dapper

When using Dapper in a per-suite isolation scenario, Respawn is the recommended approach for the same reasons: it handles multi-connection scenarios and DDL operations. The test fixture manages the connection lifecycle and calls Respawn between tests.

```csharp
// DapperDatabaseFixture.cs
using Respawn;

public class DapperDatabaseFixture : IAsyncLifetime
{
    private static readonly Checkpoint _checkpoint = new Checkpoint
    {
        SchemasToInclude = new[] { "dbo" },
        DbAdapter = DbAdapter.SqlServer,
        TablesToIgnore = new[] { "__MigrationHistory" }
    };

    private readonly string _connectionString;
    private SqlConnection _connection;

    public DapperDatabaseFixture(string connectionString)
    {
        _connectionString = connectionString;
    }

    public async Task InitializeAsync()
    {
        _connection = new SqlConnection(_connectionString);
        await _connection.OpenAsync();
        await _checkpoint.Reset(_connection);
    }

    public IDbConnection Connection => _connection;

    public async Task ResetDatabaseAsync()
    {
        await _checkpoint.Reset(_connection);
    }

    public async Task DisposeAsync()
    {
        await _connection.CloseAsync();
        await _connection.DisposeAsync();
    }
}
```

```csharp
// Dapper repository test with per-suite isolation
public class DapperOrderRepositoryTests : IClassFixture<DapperDatabaseFixture>
{
    private readonly DapperDatabaseFixture _fixture;
    private readonly DapperOrderRepository _repository;

    public DapperOrderRepositoryTests(DapperDatabaseFixture fixture)
    {
        _fixture = fixture;
        _repository = new DapperOrderRepository(fixture.Connection);
    }

    [Fact]
    public async Task GetOrdersByCustomer_ShouldReturnAllOrders()
    {
        // Arrange — seed some orders
        const string insertOrder = "INSERT INTO Orders (CustomerId, Total, Status) VALUES (@CustomerId, @Total, @Status); SELECT CAST(SCOPE_IDENTITY() AS INT);";

        var order1Id = await _fixture.Connection.QuerySingleAsync<int>(
            insertOrder, new { CustomerId = 1, Total = 100m, Status = "Active" });
        var order2Id = await _fixture.Connection.QuerySingleAsync<int>(
            insertOrder, new { CustomerId = 1, Total = 200m, Status = "Active" });
        var order3Id = await _fixture.Connection.QuerySingleAsync<int>(
            insertOrder, new { CustomerId = 2, Total = 300m, Status = "Active" });

        // Act
        var customerOrders = await _repository.GetOrdersByCustomerAsync(1);

        // Assert
        Assert.Equal(2, customerOrders.Count());
        Assert.Contains(customerOrders, o => o.Id == order1Id);
        Assert.Contains(customerOrders, o => o.Id == order2Id);

        // Clean
        await _fixture.ResetDatabaseAsync();
    }

    [Fact]
    public async Task GetOrdersByCustomer_ShouldReturnEmpty_WhenNoOrders()
    {
        // Database is clean from previous ResetDatabaseAsync call

        var customerOrders = await _repository.GetOrdersByCustomerAsync(99);

        Assert.Empty(customerOrders);

        await _fixture.ResetDatabaseAsync();
    }

    [Fact]
    public async Task UpdateOrderTotal_ShouldModifyTotal()
    {
        // Arrange
        const string insertSql = "INSERT INTO Orders (CustomerId, Total, Status) VALUES (1, 100, 'Active'); SELECT CAST(SCOPE_IDENTITY() AS INT);";
        var orderId = await _fixture.Connection.QuerySingleAsync<int>(insertSql);

        // Act
        await _repository.UpdateOrderTotalAsync(orderId, 250m);

        // Assert
        var updatedTotal = await _fixture.Connection.QuerySingleAsync<decimal>(
            "SELECT Total FROM Orders WHERE Id = @Id", new { Id = orderId });
        Assert.Equal(250m, updatedTotal);

        await _fixture.ResetDatabaseAsync();
    }
}
```

**Dapper + Respawn differences from EF Core:**
- You do not need to worry about EF Core's change tracker — Dapper has no first-level cache. Every query hits the database, so the state is always fresh.
- Dapper tests using Respawn should explicitly call `ResetDatabaseAsync()` at the end of each test (or use xUnit's `IAsyncLifetime` on the collection fixture to reset after every test).
- If your Dapper tests use multiple connections (e.g., you test a service that opens its own connections internally), Respawn is essential — transaction rollback cannot span multiple connections.

## 6 — Per-Assembly Isolation — Global Fixture with xUnit Collection Fixtures

Per-assembly isolation uses xUnit's `ICollectionFixture<T>` to create a single fixture instance that is shared across all test classes in the assembly. The fixture sets up the database once (runs migrations, seeds reference data) and cleans up once after all tests complete. Individual tests must manage their own data — they should only add the data they need and should clean up after themselves.

```csharp
// GlobalDatabaseFixture.cs — shared across all test classes
public class GlobalDatabaseFixture : IDisposable
{
    private static readonly object _lock = new();
    private static bool _initialized;

    public AppDbContext Context { get; }
    public string ConnectionString { get; }

    public GlobalDatabaseFixture()
    {
        ConnectionString = "Server=.;Database=GlobalTestDb;...";

        var options = new DbContextOptionsBuilder<AppDbContext>()
            .UseSqlServer(ConnectionString)
            .Options;

        Context = new AppDbContext(options);

        lock (_lock)
        {
            if (!_initialized)
            {
                Context.Database.EnsureDeleted();
                Context.Database.Migrate();
                SeedReferenceData(Context);
                _initialized = true;
            }
        }
    }

    private static void SeedReferenceData(AppDbContext context)
    {
        // Seed countries, currencies, status codes — data that is read-only
        if (!context.Countries.Any())
        {
            context.Countries.AddRange(
                new Country { Code = "US", Name = "United States" },
                new Country { Code = "CA", Name = "Canada" }
            );
            context.SaveChanges();
        }
    }

    public void Dispose()
    {
        // Optional: clean up only if needed
        // Context?.Dispose();
    }
}
```

```csharp
// Collection definition — links the fixture to all test classes in this collection
[CollectionDefinition("Global Database Collection")]
public class GlobalDatabaseCollection : ICollectionFixture<GlobalDatabaseFixture>
{
    // This class has no code, it only serves as a marker
}
```

```csharp
// Test class using the global fixture
[Collection("Global Database Collection")]
public class UserRepositoryGlobalTests
{
    private readonly GlobalDatabaseFixture _fixture;
    private readonly UserRepository _repository;

    public UserRepositoryGlobalTests(GlobalDatabaseFixture fixture)
    {
        _fixture = fixture;

        // Each test must clean up after itself — start by clearing any leftover data
        CleanupTestData();

        _repository = new UserRepository(fixture.Context);
    }

    [Fact]
    public async Task InsertUser_ShouldSucceed()
    {
        var user = new User { Name = "Eve", Email = "eve@example.com" };
        await _repository.InsertAsync(user);
        Assert.NotEqual(0, user.Id);
    }

    [Fact]
    public async Task GetUser_ShouldReturnNull_WhenMissing()
    {
        var user = await _repository.GetByIdAsync(9999);
        Assert.Null(user);
    }

    private void CleanupTestData()
    {
        _fixture.Context.Users.RemoveRange(_fixture.Context.Users);
        _fixture.Context.SaveChanges();
    }
}
```

```csharp
// Another test class sharing the same global fixture
[Collection("Global Database Collection")]
public class OrderRepositoryGlobalTests
{
    private readonly GlobalDatabaseFixture _fixture;
    private readonly OrderRepository _repository;

    public OrderRepositoryGlobalTests(GlobalDatabaseFixture fixture)
    {
        _fixture = fixture;
        CleanupTestData();
        _repository = new OrderRepository(fixture.Context);
    }

    [Fact]
    public async Task CreateOrder_ShouldSuccess()
    {
        var order = new Order { CustomerId = 1, Total = 100m };
        await _repository.CreateAsync(order);
        Assert.NotEqual(0, order.Id);
    }

    private void CleanupTestData()
    {
        _fixture.Context.Orders.RemoveRange(_fixture.Context.Orders);
        _fixture.Context.SaveChanges();
    }
}
```

**Per-assembly guidelines:**
- Only use this for test suites where setup cost is prohibitively high (e.g., a database that takes 30 seconds to migrate).
- Each test class must clean up its own data in the constructor (or in `IAsyncLifetime.InitializeAsync`). Do not rely on other tests to clean up.
- Never run per-assembly tests in parallel — use `[Collection("...")]` which makes xUnit run them sequentially.
- The global fixture is a singleton for the test run. Be careful with static state in the fixture itself.

## 7 — Speed Benchmark Comparison

The following table shows approximate timings for each isolation strategy on a typical developer machine (SQL Server on localhost, 20 tables, ~100 total rows of reference data):

| Isolation Strategy | Setup Time | Per-Test Overhead | Total for 100 Tests | Parallel-Safe |
|-------------------|-----------|-------------------|---------------------|---------------|
| Transaction Rollback | ~50ms (open+begin) | ~5ms | ~0.5s | Yes (with different connections) |
| Respawn | ~100ms (schema setup) | ~400ms | ~40s | Yes (unique DB per parallel thread) |
| Respawn + Skip Seed Tables | ~100ms (schema setup) | ~200ms | ~20s | Yes |
| TestContainers (new container) | ~15s (docker pull + start) | ~15s per test class | ~15–60s per class | Yes (each container is isolated) |
| TestContainers (reused container) | ~5s (first start) | ~500ms (Respawn) | ~55s | Limited (shared container) |

**Key takeaways from the benchmark:**

1. **Transaction rollback is 10x faster than Respawn** for per-test isolation. Use it for the 80% of tests that only need a single connection and do not issue DDL.

2. **Respawn overhead is per-test, not per-suite.** If you have 100 tests in a class, Respawn after each test adds ~40 seconds total. If you batch cleanup less frequently (e.g., after every 10 tests), you save time but risk test interference.

3. **TestContainers overhead is incurred once per test class** (container creation). If you use a fresh container per test class, a suite with 10 test classes pays ~150s just in container startup. Reuse a single container across classes and use Respawn for cleanup.

4. **Parallel execution multiplies the cost** of isolation. With 4 parallel threads, you need 4 separate databases (or 4 containers). Each thread's database must be independently isolated.

5. **The total test suite time is dominated by the weakest isolation strategy.** If 95% of tests use rollback (50ms each) and 5% use Respawn (500ms each), the Respawn tests contribute disproportionately to total time.

```csharp
// Benchmark helper to measure isolation overhead
// Not a production test — run as a console app to measure timings
public class IsolationBenchmark
{
    private const string ConnectionString = "Server=.;Database=BenchmarkDb;Trusted_Connection=True;";

    public static async Task RunAsync()
    {
        // Measure transaction rollback
        var rollbackTotal = 0L;
        for (int i = 0; i < 100; i++)
        {
            var sw = Stopwatch.StartNew();
            using var fixture = new PerTestFixture(ConnectionString);
            fixture.Context.Users.Add(new User { Name = "Test" });
            await fixture.Context.SaveChangesAsync();
            sw.Stop();
            rollbackTotal += sw.ElapsedMilliseconds;
        }
        Console.WriteLine($"Rollback avg: {rollbackTotal / 100}ms");

        // Measure Respawn
        var respawnTotal = 0L;
        var checkpoint = new Checkpoint { DbAdapter = DbAdapter.SqlServer };
        await using var setupConn = new SqlConnection(ConnectionString);
        await setupConn.OpenAsync();
        await new AppDbContext(new DbContextOptionsBuilder<AppDbContext>()
            .UseSqlServer(ConnectionString).Options).Database.EnsureCreatedAsync();

        for (int i = 0; i < 100; i++)
        {
            var sw = Stopwatch.StartNew();
            await using var conn = new SqlConnection(ConnectionString);
            await conn.OpenAsync();
            await checkpoint.Reset(conn);
            sw.Stop();
            respawnTotal += sw.ElapsedMilliseconds;
        }
        Console.WriteLine($"Respawn avg: {respawnTotal / 100}ms");
    }
}
```

## 8 — Decision Flowchart — Choosing the Right Isolation Level

The following decision tree helps select the appropriate isolation level for a given test or test class:

```mermaid
flowchart TD
    A[Start: How many tests in this class?] --> B{Less than 5?}
    B -->|Yes| C{Need multiple<br/>connections?}
    B -->|No| D{Need parallel<br/>execution?}

    C -->|Yes| E[Per-Suite: Respawn]
    C -->|No| F{DDL operations<br/>(migrations)?}
    F -->|Yes| E
    F -->|No| G[Per-Test: Transaction Rollback]

    D -->|Yes| H{Can use unique<br/>database names?}
    D -->|No| I{Shared fixture<br/>acceptable?}

    H -->|Yes| J[Create unique DB<br/>per parallel thread<br/>+ Rollback per test]
    H -->|No| K[Serial execution<br/>+ Respawn per test]

    I -->|Yes| L[Per-Assembly:<br/>Global Collection Fixture]
    I -->|No| E

    style G fill:#90EE90,color:#000
    style E fill:#FFD700,color:#000
    style L fill:#FF9999,color:#000
    style J fill:#90EE90,color:#000
    style K fill:#FFD700,color:#000
```

**Decision tree walkthrough:**

1. **Small test class (fewer than 5 tests):** Per-Test rollback is ideal unless the tests need multiple connections or DDL. If they do, use Respawn.

2. **Large test class (5+ tests):** Consider parallel execution. If you can run tests in parallel with unique databases, use per-test rollback. If you cannot use unique databases, run serially with Respawn or use a global fixture.

3. **Parallel execution requires unique databases:** Each parallel thread needs its own database (or at least its own schema). The simplest approach is to append a GUID to the database name. TestContainers gives each thread its own container, which provides full isolation but is slower to start.

4. **Per-assembly fixtures are for read-only or carefully managed tests:** If the test suite takes 30 seconds to migrate and has 500 tests that all read from the same reference data, a global fixture makes sense. For most tests, the risk of state pollution is not worth the speed gain.

## 9 — Production Guidance and Gotchas

### Production Guidance

1. **Default to transaction rollback.** For the majority of tests, this is the right choice. It is fast, safe, and well-understood. Use it for any test that operates on a single connection and does not issue DDL.

2. **Use Respawn for the exceptions.** When a test needs multiple connections (e.g., you test a service that internally opens a second connection) or tests DDL (migrations, index creation), switch to Respawn.

3. **Respawn is also correct for test classes that mix patterns.** If one test in a class uses multiple connections, the entire class must use Respawn (or rollback per-test with a warning on the multi-connection test).

4. **Always run database tests in CI.** Use TestContainers in CI to spin up a real SQL Server. Do not use SQLite InMemory for integration tests — it has different locking, type mapping, and SQL dialect behavior.

5. **Use parallel-safe isolation for CI.** If your CI runs tests in parallel (which it should), ensure each parallel worker has its own database or container. The simplest pattern: `Database_TestRun_{WorkerId}`.

6. **Consider a hybrid approach.** Use a base class that wraps each test in a transaction, and override the cleanup method for tests that need Respawn. Most test frameworks support this via `IAsyncLifetime` or base class `Dispose`.

### Gotchas

1. **Parallel test execution with per-suite isolation:** If two test classes share the same database and run in parallel, Respawn in one class will delete data that the other class is currently using. Solution: use unique databases per parallel thread, or disable parallelization for database tests with `[Collection]`.

2. **Static mutable state across tests:** A static `DatabaseFixture` that stores state between tests is dangerous. Tests run in non-deterministic order in xUnit (unless `[Collection]` is used). Never store test-specific data in static fields.

3. **Test order dependency with per-suite fixtures:** If test A inserts data and forgets to clean up, test B may fail or pass unexpectedly. Always call `ResetAsync()` (or equivalent) at the end of each test, even if the test fails. Use try/finally or `IAsyncLifetime` to guarantee cleanup.

4. **Rollback only works for single-connection tests:** If your test opens a second `SqlConnection` (even to the same database), the second connection will not see the uncommitted data from the first connection's transaction. This is correct SQL Server isolation behavior — the second connection sees the state as of the last committed transaction. If you need multi-connection testing, use Respawn.

5. **Respawn does not reset sequences/identity columns by default:** After Respawn resets the data, `INSERT` operations will continue from the last identity value. This can cause tests to fail if they expect identity values like 1, 2, 3. Use `checkpoint.WithReseed = true` to reset identity columns.

6. **DDL auto-commit:** `CREATE TABLE`, `CREATE INDEX`, `ALTER TABLE`, and other DDL statements auto-commit the current transaction in SQL Server. Running migrations inside a rollback transaction will not roll back the schema changes. Use Respawn for test classes that run migrations.

7. **Dispose ordering:** If you dispose the `DbContext` before the transaction, EF Core may attempt to commit or roll back the transaction during disposal, causing errors. Always dispose the transaction first, then the context. In C# `using` blocks: inner `using` disposes first.

8. **TransactionScope vs IDbContextTransaction:** `TransactionScope` promotes to a distributed transaction if multiple connections are involved, which requires MSDTC. For test isolation, use `IDbContextTransaction` (EF Core) or `SqlTransaction` (Dapper) to avoid this complication.

9. **xUnit fixture lifecycle:** `IClassFixture<T>` creates one instance of `T` per test class and injects it into every test method. The fixture is disposed after all tests in the class complete. `ICollectionFixture<T>` creates one instance per collection. `IClassFixture` does NOT guarantee that the fixture is created before each test — it is created once and shared. If you need per-test setup, implement `IAsyncLifetime` on the fixture or call a reset method at the start of each test.

10. **Mixing isolation levels in the same test class:** If you have one test that uses transaction rollback and another that uses Respawn in the same class, they will interfere. The rollback test opens a transaction; the Respawn test truncates the transaction's data mid-flight, causing errors. Pick one isolation level per test class.

11. **Seed data loss with Respawn:** Respawn deletes data from all tables (except those in `TablesToIgnore`). If you have seed data (countries, status codes), you must either add those tables to `TablesToIgnore` or re-seed after every `Reset()` call. The former is faster but means seed data can accumulate test-specific rows over time. The latter is safer but adds overhead.

12. **Database name collision in parallel runs:** When running tests in parallel, use a unique database name per parallel worker. The pattern: `TestDb_{ProcessId}_{ManagedThreadId}`. This prevents one worker from dropping another worker's database during cleanup.
