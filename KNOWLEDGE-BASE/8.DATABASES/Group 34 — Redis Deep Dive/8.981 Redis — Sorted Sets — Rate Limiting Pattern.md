# Redis — Sorted Sets — Rate Limiting Pattern

---

## Section 1 — Overview

Rate limiting is a critical pattern for protecting APIs, services, and resources from abuse or excessive usage. Redis sorted sets provide an elegant solution for **sliding window rate limiting**, which is more accurate than fixed-window counters but more memory-efficient than keeping full per-request logs.

The core idea: use a sorted set where each request is stored as a member, with its timestamp as the score. When checking a rate limit, remove entries that fall outside the time window, count the remaining entries, and compare against the limit.

```
Sliding Window Rate Limiter (sorted set approach):

Timeline:  [────── Window: 60 seconds ──────]
           │                                  │
    Request 1    Request 2    ...   Request N  New Request
    (removed)    (in window)       (in window)  (added if allowed)

    Each request = ZADD key timestamp_ms memberId
    Cleanup     = ZREMRANGEBYSCORE key -inf (now - window)
    Count       = ZCARD key
    Decision    = count < limit → ALLOW, else → REJECT
```

---

### 1.1 — Sliding Window vs Fixed Window

| Aspect | Sliding Window (Sorted Set) | Fixed Window (INCR + EXPIRE) |
|---|---|---|
| Accuracy | Precise — any 60s window | Approximate — resets at clock boundary |
| Memory | Higher — stores each request | Lower — single counter |
| Complexity | O(log N) per request | O(1) per request |
| Thundering herd | Handles naturally | Burst at window boundary |

The sorted set approach prevents the "boundary burst" problem where a fixed window allows 2x traffic at the reset point:

```
Fixed Window:
  12:00:59 — 100 requests (all allowed)
  12:01:00 — 100 requests (all allowed, counter reset)
  Total: 200 requests in 2 seconds — defeats rate limiting!

Sliding Window:
  12:00:59 — request counted
  12:01:00 — request counted, old entries evicted
  Total: 100 requests in any rolling 60-second window
```

---

### 1.2 — When to Use Sorted Set Rate Limiting

| Scenario | Window Size | Limit | Key Pattern |
|---|---|---|---|
| API rate limit per user | 1 minute | 60 req/min | `ratelimit:api:{userId}` |
| Login attempts | 15 minutes | 5 attempts | `ratelimit:login:{ip}` |
| SMS sending | 1 hour | 10 messages | `ratelimit:sms:{phoneNumber}` |
| Password reset | 24 hours | 3 requests | `ratelimit:pwdreset:{email}` |
| Webhook calls | 1 second | 10 calls/sec | `ratelimit:webhook:{apiKey}` |

## Section 2 — Command Reference

---

### 2.1 — Commands Used in Rate Limiting

| Command | Role in Pattern |
|---|---|
| ZADD | Add current request timestamp as score |
| ZREMRANGEBYSCORE | Remove entries outside the time window |
| ZCARD | Count entries remaining in the window |
| EXPIRE | Set TTL for auto-cleanup of the key |
| MULTI/EXEC | Transactional execution of cleanup + add |
| EVAL (Lua) | Atomic rate check + cleanup + add |

---

### 2.2 — Basic Redis CLI Example

```bash
# Rate limit: max 5 requests per 10 seconds for user:42

# On each request:
# 1. Remove entries older than 10 seconds ago
ZREMRANGEBYSCORE ratelimit:user:42 -inf (1704067210

# 2. Count remaining entries
ZCARD ratelimit:user:42

# 3. If count < 5, allow the request and add entry
ZADD ratelimit:user:42 1704067220 "req:abc123"

# 4. Set TTL (auto-cleanup after window + buffer)
EXPIRE ratelimit:user:42 15
```

### 2.3 — Full CLI Session

```bash
# Initialize: user makes 3 requests
> ZADD ratelimit:user:1 1704067200 "req:001"
> ZADD ratelimit:user:1 1704067205 "req:002"
> ZADD ratelimit:user:1 1704067209 "req:003"

# After 5 more seconds, old request (req:001, ts=1704067200) is now 14s old
# Window = 10 seconds, so remove anything older than (now - 10)
> ZREMRANGEBYSCORE ratelimit:user:1 -inf (1704067204
(integer) 1  # removed req:001

# Count remaining (2 entries within window)
> ZCARD ratelimit:user:1
(integer) 2

# Allow request #4
> ZADD ratelimit:user:1 1704067214 "req:004"

# Now count = 3, still under limit of 5
# ... user continues ...

# At request #6, count = 5 which hits the limit → reject
> ZCARD ratelimit:user:1
(integer) 5
# Count >= limit, reject request
```

### 2.4 — Lua Script for Atomic Rate Check

For production, use a Lua script to make the entire check + cleanup + add atomic:

```lua
-- KEYS[1]: ratelimit key (e.g., "ratelimit:user:42")
-- ARGV[1]: current timestamp in milliseconds
-- ARGV[2]: window size in milliseconds
-- ARGV[3]: max requests (limit)
-- ARGV[4]: unique member ID for this request
-- ARGV[5]: TTL in seconds

-- 1. Remove expired entries
redis.call('ZREMRANGEBYSCORE', KEYS[1], '-inf', '(' .. (ARGV[1] - ARGV[2]))

-- 2. Count remaining
local count = redis.call('ZCARD', KEYS[1])

-- 3. Check limit
if count >= tonumber(ARGV[3]) then
    return 0  -- REJECTED
end

-- 4. Add current request
redis.call('ZADD', KEYS[1], ARGV[1], ARGV[4])

-- 5. Set TTL for auto-cleanup
redis.call('EXPIRE', KEYS[1], ARGV[5])

return 1  -- ALLOWED
```

### 3.3 — Error Handling for Rate Limiter

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;
using Microsoft.Extensions.Logging;

public class RateLimiterErrorHandler
{
    private readonly ILogger<RateLimiterErrorHandler> _logger;

    public RateLimiterErrorHandler(ILogger<RateLimiterErrorHandler> logger)
    {
        _logger = logger;
    }

    public async Task<T> HandleRateLimitOperationAsync<T>(
        Func<Task<T>> operation,
        string operationName,
        T fallbackValue)
    {
        try
        {
            return await operation();
        }
        catch (RedisConnectionException ex)
        {
            _logger.LogWarning(ex,
                "Redis connection error in {Op}. Falling back to allow.",
                operationName);
            return fallbackValue; // Fail open: allow the request
        }
        catch (RedisTimeoutException ex)
        {
            _logger.LogWarning(ex,
                "Redis timeout in {Op}. Falling back to allow.",
                operationName);
            return fallbackValue; // Fail open: allow the request
        }
        catch (RedisServerException ex)
        {
            _logger.LogError(ex,
                "Redis server error in {Op}. Falling back to allow.",
                operationName);
            return fallbackValue;
        }
        catch (Exception ex)
        {
            _logger.LogError(ex,
                "Unexpected error in {Op}. Falling back to allow.",
                operationName);
            return fallbackValue; // Fail open: allow on unexpected errors
        }
    }
}
```

### 4.1 — Basic Sliding Window Rate Limiter

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;
using Microsoft.Extensions.Logging;

public class SlidingWindowRateLimiter
{
    private readonly IDatabase _db;
    private readonly ILogger<SlidingWindowRateLimiter> _logger;
    private readonly RateLimiterErrorHandler _errorHandler;

    public SlidingWindowRateLimiter(
        IDatabase db,
        ILogger<SlidingWindowRateLimiter> logger,
        RateLimiterErrorHandler errorHandler)
    {
        _db = db;
        _logger = logger;
        _errorHandler = errorHandler;
    }

    /// <summary>
    /// Check if a request is allowed under the rate limit.
    /// Uses sorted set sliding window.
    /// </summary>
    public async Task<RateLimitResult> IsAllowedAsync(
        string userId,
        string action,
        int maxRequests,
        int windowSeconds)
    {
        return await _errorHandler.HandleRateLimitOperationAsync(
            () => CheckRateLimitAsync(userId, action, maxRequests, windowSeconds),
            nameof(IsAllowedAsync),
            new RateLimitResult { IsAllowed = true, Reason = "Error fallback" });
    }

    private async Task<RateLimitResult> CheckRateLimitAsync(
        string userId,
        string action,
        int maxRequests,
        int windowSeconds)
    {
        string key = $"ratelimit:{userId}:{action}";
        long nowMs = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        long windowMs = windowSeconds * 1000L;
        long cutoffMs = nowMs - windowMs;
        string memberId = $"req:{userId}:{nowMs}:{Guid.NewGuid():N}";

        // Step 1: Remove entries outside the window
        await _db.SortedSetRemoveRangeByScoreAsync(
            key,
            start: double.NegativeInfinity,
            stop: cutoffMs,
            exclude: Exclude.None);

        // Step 2: Count remaining entries
        long currentCount = await _db.SortedSetLengthAsync(key);

        // Step 3: Check limit
        if (currentCount >= maxRequests)
        {
            return new RateLimitResult
            {
                IsAllowed = false,
                CurrentCount = currentCount,
                MaxRequests = maxRequests,
                WindowSeconds = windowSeconds,
                Reason = "Rate limit exceeded",
                RetryAfterMs = GetRetryAfterMs(key, nowMs, windowMs)
            };
        }

        // Step 4: Add current request
        await _db.SortedSetAddAsync(key, memberId, nowMs);

        // Step 5: Set TTL (window + 10% buffer)
        int ttlSeconds = (int)(windowSeconds * 1.1) + 1;
        await _db.KeyExpireAsync(key, TimeSpan.FromSeconds(ttlSeconds));

        return new RateLimitResult
        {
            IsAllowed = true,
            CurrentCount = currentCount + 1,
            MaxRequests = maxRequests,
            WindowSeconds = windowSeconds,
            Remaining = maxRequests - (currentCount + 1)
        };
    }

    /// <summary>
    /// Estimate when the oldest request in the window will expire.
    /// </summary>
    private long GetRetryAfterMs(
        RedisKey key, long nowMs, long windowMs)
    {
        // Get the oldest entry's score (first entry in ascending order)
        var oldest = _db.SortedSetRangeByRankWithScores(
            key, 0, 0, Order.Ascending);

        if (oldest.Length > 0)
        {
            long oldestMs = (long)oldest[0].Score;
            long expiresAtMs = oldestMs + windowMs;
            return Math.Max(0, expiresAtMs - nowMs);
        }

        return windowMs; // fallback
    }
}

public class RateLimitResult
{
    public bool IsAllowed { get; set; }
    public long CurrentCount { get; set; }
    public int MaxRequests { get; set; }
    public int WindowSeconds { get; set; }
    public int? Remaining { get; set; }
    public long RetryAfterMs { get; set; }
    public string? Reason { get; set; }
}
```

### 4.2 — Atomic Rate Limiter with Lua Script

The recommended production approach — a single Lua script handles the entire check atomically.

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;
using Microsoft.Extensions.Logging;

public class AtomicRateLimiter
{
    private readonly IDatabase _db;
    private readonly ILogger<AtomicRateLimiter> _logger;
    private readonly RateLimiterErrorHandler _errorHandler;

    // Lua script: atomic rate check + cleanup + add
    private const string RateLimitScript = @"
        local key = KEYS[1]
        local nowMs = tonumber(ARGV[1])
        local windowMs = tonumber(ARGV[2])
        local maxRequests = tonumber(ARGV[3])
        local memberId = ARGV[4]
        local ttlSeconds = tonumber(ARGV[5])

        -- Remove expired entries
        redis.call('ZREMRANGEBYSCORE', key, '-inf', '(' .. (nowMs - windowMs))

        -- Count remaining
        local count = redis.call('ZCARD', key)

        -- Check limit
        if count >= maxRequests then
            return { 0, count }
        end

        -- Add current request
        redis.call('ZADD', key, nowMs, memberId)

        -- Set TTL
        redis.call('EXPIRE', key, ttlSeconds)

        return { 1, count + 1 }
    ";

    public AtomicRateLimiter(
        IDatabase db,
        ILogger<AtomicRateLimiter> logger,
        RateLimiterErrorHandler errorHandler)
    {
        _db = db;
        _logger = logger;
        _errorHandler = errorHandler;
    }

    public async Task<RateLimitResult> IsAllowedAsync(
        string userId,
        string action,
        int maxRequests,
        int windowSeconds)
    {
        return await _errorHandler.HandleRateLimitOperationAsync(
            () => ExecuteRateCheckAsync(userId, action, maxRequests, windowSeconds),
            nameof(IsAllowedAsync),
            new RateLimitResult { IsAllowed = true, Reason = "Error fallback" });
    }

    private async Task<RateLimitResult> ExecuteRateCheckAsync(
        string userId,
        string action,
        int maxRequests,
        int windowSeconds)
    {
        string key = $"ratelimit:{userId}:{action}";
        long nowMs = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        long windowMs = windowSeconds * 1000L;
        string memberId = $"{userId}:{nowMs}:{Guid.NewGuid():N}";
        int ttlSeconds = (int)(windowSeconds * 1.1) + 1;

        RedisResult result = await _db.ScriptEvaluateAsync(
            RateLimitScript,
            new RedisKey[] { key },
            new RedisValue[]
            {
                nowMs.ToString(),
                windowMs.ToString(),
                maxRequests.ToString(),
                memberId,
                ttlSeconds.ToString()
            });

        var values = (RedisResult[])result;
        bool allowed = (int)values[0] == 1;
        long count = (long)values[1];

        return new RateLimitResult
        {
            IsAllowed = allowed,
            CurrentCount = count,
            MaxRequests = maxRequests,
            WindowSeconds = windowSeconds,
            Remaining = allowed ? maxRequests - (int)count : 0,
            Reason = allowed ? null : "Rate limit exceeded"
        };
    }
}
```

### 4.3 — Fixed Window Rate Limiter (INCR + EXPIRE)

Simpler alternative when perfect sliding window accuracy isn't required.

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;
using Microsoft.Extensions.Logging;

public class FixedWindowRateLimiter
{
    private readonly IDatabase _db;
    private readonly ILogger<FixedWindowRateLimiter> _logger;
    private readonly RateLimiterErrorHandler _errorHandler;

    public FixedWindowRateLimiter(
        IDatabase db,
        ILogger<FixedWindowRateLimiter> logger,
        RateLimiterErrorHandler errorHandler)
    {
        _db = db;
        _logger = logger;
        _errorHandler = errorHandler;
    }

    public async Task<RateLimitResult> IsAllowedAsync(
        string userId,
        string action,
        int maxRequests,
        int windowSeconds)
    {
        return await _errorHandler.HandleRateLimitOperationAsync(
            () => CheckFixedWindowAsync(userId, action, maxRequests, windowSeconds),
            nameof(IsAllowedAsync),
            new RateLimitResult { IsAllowed = true, Reason = "Error fallback" });
    }

    private async Task<RateLimitResult> CheckFixedWindowAsync(
        string userId,
        string action,
        int maxRequests,
        int windowSeconds)
    {
        // Key includes the window bucket: "ratelimit:fixed:{userId}:{action}:{windowStart}"
        string windowBucket = GetWindowBucket(windowSeconds);
        string key = $"ratelimit:fixed:{userId}:{action}:{windowBucket}";

        long count = await _db.StringIncrementAsync(key, 1);

        // Set TTL on first increment (returns true only if key was just created)
        bool ttlSet = await _db.KeyExpireAsync(key, TimeSpan.FromSeconds(windowSeconds));

        if (ttlSet)
        {
            _logger.LogDebug(
                "New fixed window bucket created: {Key}", key);
        }

        if (count > maxRequests)
        {
            return new RateLimitResult
            {
                IsAllowed = false,
                CurrentCount = count,
                MaxRequests = maxRequests,
                WindowSeconds = windowSeconds,
                Reason = "Fixed window rate limit exceeded",
                Remaining = 0
            };
        }

        return new RateLimitResult
        {
            IsAllowed = true,
            CurrentCount = count,
            MaxRequests = maxRequests,
            WindowSeconds = windowSeconds,
            Remaining = maxRequests - (int)count
        };
    }

    private static string GetWindowBucket(int windowSeconds)
    {
        long nowSeconds = DateTimeOffset.UtcNow.ToUnixTimeSeconds();
        long bucket = nowSeconds / windowSeconds;
        return bucket.ToString();
    }
}
```

### 4.4 — Multi-Tier Rate Limiter

Apply different rate limits at different tiers (e.g., per-second, per-minute, per-hour).

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;
using Microsoft.Extensions.Logging;

public class MultiTierRateLimiter
{
    private readonly IDatabase _db;
    private readonly ILogger<MultiTierRateLimiter> _logger;

    public MultiTierRateLimiter(
        IDatabase db,
        ILogger<MultiTierRateLimiter> logger)
    {
        _db = db;
        _logger = logger;
    }

    public async Task<MultiTierResult> IsAllowedAsync(
        string userId,
        string action)
    {
        // Define tiers: (maxRequests, windowSeconds)
        var tiers = new (int max, int window)[]
        {
            (10, 1),      // 10 requests per second
            (200, 60),    // 200 requests per minute
            (5000, 3600)  // 5000 requests per hour
        };

        var result = new MultiTierResult { IsAllowed = true };

        foreach (var (max, window) in tiers)
        {
            bool allowed = await CheckTierAsync(
                userId, action, max, window);

            if (!allowed)
            {
                result.IsAllowed = false;
                result.LimitBreached = (max, window);
                result.Reason = $"Rate limit exceeded: {max} req/{window}s";
                _logger.LogWarning(
                    "Rate limit breached for {User}/{Action}: {Max} req/{Window}s",
                    userId, action, max, window);
                break;
            }
        }

        return result;
    }

    private async Task<bool> CheckTierAsync(
        string userId,
        string action,
        int maxRequests,
        int windowSeconds)
    {
        string key = $"ratelimit:{userId}:{action}:tier:{windowSeconds}s";
        long nowMs = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        long windowMs = windowSeconds * 1000L;
        long cutoffMs = nowMs - windowMs;
        string memberId = $"{userId}:{nowMs}:{Guid.NewGuid():N}";

        string script = @"
            local key = KEYS[1]
            local cutoff = tonumber(ARGV[1])
            local now = tonumber(ARGV[2])
            local max = tonumber(ARGV[3])
            local member = ARGV[4]
            local ttl = tonumber(ARGV[5])

            redis.call('ZREMRANGEBYSCORE', key, '-inf', '(' .. cutoff)
            local count = redis.call('ZCARD', key)

            if count >= max then
                return 0
            end

            redis.call('ZADD', key, now, member)
            redis.call('EXPIRE', key, ttl)
            return 1
        ";

        int ttlSeconds = (int)(windowSeconds * 1.1) + 1;

        RedisResult result = await _db.ScriptEvaluateAsync(script,
            new RedisKey[] { key },
            new RedisValue[]
            {
                cutoffMs.ToString(),
                nowMs.ToString(),
                maxRequests.ToString(),
                memberId,
                ttlSeconds.ToString()
            });

        return (int)result == 1;
    }
}

public class MultiTierResult
{
    public bool IsAllowed { get; set; }
    public (int max, int window)? LimitBreached { get; set; }
    public string? Reason { get; set; }
}
```

### 4.5 — Distributed Rate Limiter with Consistent Key Hashing

For multi-instance deployments, ensure all instances use the same Redis instance/cluster for the same user's rate limit keys.

```csharp
using StackExchange.Redis;
using System;
using System.Security.Cryptography;
using System.Text;
using System.Threading.Tasks;

public class DistributedRateLimiter
{
    private readonly ConnectionMultiplexer _muxer;
    private readonly int _databaseCount;

    public DistributedRateLimiter(
        ConnectionMultiplexer muxer,
        int databaseCount = 16)
    {
        _muxer = muxer;
        _databaseCount = databaseCount;
    }

    /// <summary>
    /// Route rate limit keys to specific database slots consistently.
    /// All instances route the same user to the same slot.
    /// </summary>
    private IDatabase GetDatabaseForKey(string key)
    {
        using var sha256 = SHA256.Create();
        byte[] hash = sha256.ComputeHash(Encoding.UTF8.GetBytes(key));
        int slot = Math.Abs(BitConverter.ToInt32(hash, 0)) % _databaseCount;
        return _muxer.GetDatabase(slot);
    }

    public async Task<RateLimitResult> IsAllowedAsync(
        string userId,
        string action,
        int maxRequests,
        int windowSeconds)
    {
        string key = $"ratelimit:d:{userId}:{action}";
        IDatabase db = GetDatabaseForKey(key);

        long nowMs = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        long windowMs = windowSeconds * 1000L;
        long cutoffMs = nowMs - windowMs;
        string memberId = $"{userId}:{nowMs}:{Guid.NewGuid():N}";
        int ttlSeconds = (int)(windowSeconds * 1.1) + 1;

        string script = @"
            redis.call('ZREMRANGEBYSCORE', KEYS[1], '-inf', '(' .. ARGV[1])
            local c = redis.call('ZCARD', KEYS[1])
            if c >= tonumber(ARGV[2]) then return {0, c} end
            redis.call('ZADD', KEYS[1], ARGV[3], ARGV[4])
            redis.call('EXPIRE', KEYS[1], ARGV[5])
            return {1, c + 1}
        ";

        var result = (RedisResult[])await db.ScriptEvaluateAsync(script,
            new RedisKey[] { key },
            new RedisValue[]
            {
                cutoffMs.ToString(),
                maxRequests.ToString(),
                nowMs.ToString(),
                memberId,
                ttlSeconds.ToString()
            });

        bool allowed = (int)result[0] == 1;
        long count = (long)result[1];

        return new RateLimitResult
        {
            IsAllowed = allowed,
            CurrentCount = count,
            MaxRequests = maxRequests,
            WindowSeconds = windowSeconds,
            Remaining = allowed ? maxRequests - (int)count : 0
        };
    }
}
```

### 4.6 — Rate Limiter Middleware for ASP.NET Core

```csharp
using Microsoft.AspNetCore.Http;
using Microsoft.Extensions.Logging;
using System;
using System.Threading.Tasks;

public class RateLimitingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly AtomicRateLimiter _rateLimiter;
    private readonly ILogger<RateLimitingMiddleware> _logger;

    public RateLimitingMiddleware(
        RequestDelegate next,
        AtomicRateLimiter rateLimiter,
        ILogger<RateLimitingMiddleware> logger)
    {
        _next = next;
        _rateLimiter = rateLimiter;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        // Identify the caller (API key, IP, authenticated user)
        string callerId = GetCallerId(context);
        string path = context.Request.Path.ToString().ToLowerInvariant();

        // Apply rate limit: 100 requests per minute per caller
        var result = await _rateLimiter.IsAllowedAsync(
            callerId, path, 100, 60);

        if (!result.IsAllowed)
        {
            context.Response.StatusCode = 429; // Too Many Requests
            context.Response.Headers["Retry-After"] =
                (result.CurrentCount > 0 ? "60" : "1");
            context.Response.Headers["X-RateLimit-Limit"] = "100";
            context.Response.Headers["X-RateLimit-Remaining"] = "0";
            context.Response.Headers["X-RateLimit-Reset"] =
                DateTimeOffset.UtcNow.AddSeconds(60)
                    .ToUnixTimeSeconds().ToString();

            _logger.LogWarning(
                "Rate limit exceeded for {Caller} on {Path}",
                callerId, path);

            await context.Response.WriteAsync(
                "{\"error\":\"rate_limit_exceeded\",\"message\":\"Too many requests\"}");
            return;
        }

        context.Response.Headers["X-RateLimit-Limit"] = "100";
        context.Response.Headers["X-RateLimit-Remaining"] =
            (result.Remaining ?? 0).ToString();
        context.Response.Headers["X-RateLimit-Reset"] =
            DateTimeOffset.UtcNow.AddSeconds(60)
                .ToUnixTimeSeconds().ToString();

        await _next(context);
    }

    private static string GetCallerId(HttpContext context)
    {
        // Prefer authenticated user ID
        if (context.User?.Identity?.IsAuthenticated == true)
        {
            return context.User.FindFirst("sub")?.Value
                   ?? context.User.Identity.Name
                   ?? "anonymous";
        }

        // Fall back to IP address
        return context.Connection.RemoteIpAddress?.ToString()
               ?? "unknown-ip";
    }
}
```

### 4.8 — Rate Limiter with Burst Allowance

Allow brief bursts above the sustained limit using a two-tier approach.

```csharp
using StackExchange.Redis;
using System;
using System.Threading.Tasks;

public class BurstRateLimiter
{
    private readonly IDatabase _db;

    public BurstRateLimiter(IDatabase db)
    {
        _db = db;
    }

    /// <summary>
    /// Check rate limit with burst allowance.
    /// Sustained: 10 req/s. Burst: 20 req/s for up to 3 seconds.
    /// </summary>
    public async Task<RateLimitResult> IsAllowedBurstAsync(
        string userId,
        string action,
        int sustainedLimit = 10,
        int burstLimit = 20,
        int windowSeconds = 1)
    {
        string key = $"ratelimit:burst:{userId}:{action}";
        long nowMs = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        long windowMs = windowSeconds * 1000L;
        string memberId = $"{userId}:{nowMs}:{Guid.NewGuid():N}";
        int ttlSeconds = (int)(windowSeconds * 1.1) + 1;

        string script = @"
            local key = KEYS[1]
            local now = tonumber(ARGV[1])
            local windowMs = tonumber(ARGV[2])
            local burstMax = tonumber(ARGV[3])
            local sustainedMax = tonumber(ARGV[4])
            local member = ARGV[5]
            local ttl = tonumber(ARGV[6])

            redis.call('ZREMRANGEBYSCORE', key, '-inf', '(' .. (now - windowMs))
            local count = redis.call('ZCARD', key)

            if count >= burstMax then
                return {0, count, 0}
            end

            local burstUsed = 0
            if count >= sustainedMax then
                burstUsed = 1
            end

            redis.call('ZADD', key, now, member)
            redis.call('EXPIRE', key, ttl)

            return {1, count + 1, burstUsed}
        ";

        var result = (RedisResult[])await _db.ScriptEvaluateAsync(script,
            new RedisKey[] { key },
            new RedisValue[]
            {
                nowMs.ToString(),
                windowMs.ToString(),
                burstLimit.ToString(),
                sustainedLimit.ToString(),
                memberId,
                ttlSeconds.ToString()
            });

        bool allowed = (int)result[0] == 1;
        long count = (long)result[1];
        bool isBurst = (int)result[2] == 1;

        return new RateLimitResult
        {
            IsAllowed = allowed,
            CurrentCount = count,
            MaxRequests = isBurst ? burstLimit : sustainedLimit,
            WindowSeconds = windowSeconds,
            IsBurst = isBurst,
            Remaining = allowed
                ? (isBurst ? burstLimit : sustainedLimit) - (int)count
                : 0
        };
    }
}
```

```csharp
    /// Check if a login attempt should be allowed.

    /// </summary>

        string ipKey = $"ratelimit:login:ip:{ipAddress}";
        string ipMember = $"{ipAddress}:{nowMs}";

            new RedisValue[]

                nowMs.ToString(), windowMs.ToString(),
                (MaxAttempts * 2).ToString(), ipMember, ttlSec.ToString()
            });

            return (false, "Too many login attempts from this IP address.");

        return (true, null);
    }
```

### 6.1 — Time Complexity

| Operation | Complexity | Notes |
|---|---|---|
| ZREMRANGEBYSCORE | O(log N + M) | N = size, M = removed entries |
| ZCARD | O(1) | Cached length |
| ZADD | O(log N) | Insert into skiplist |
| EXPIRE | O(1) | Metadata update |
| Lua script (all above) | O(log N + M) | Combined |

---

### 6.2 — Memory Usage

```
Sorted set rate limiter — per request stored:
  Member string (~36 bytes): "user:123:1704067200000:a1b2c3d4"
  Score (double): 8 bytes
  Skiplist + hash overhead: ~56 bytes
  Total per request: ~100 bytes

For a rate limit of 100 req/min per user:
  Memory per user at peak: ~10 KB (100 entries × 100 bytes)
  Memory for 10,000 active users: ~100 MB
  Memory for 100,000 active users: ~1 GB
```

**Comparison with fixed window:**
```
Fixed window (INCR + EXPIRE):
  Memory per user: ~1 key = ~50 bytes
  Memory for 100,000 active users: ~5 MB

Sliding window (sorted set):
  Memory per user at peak: ~10 KB
  Memory for 100,000 active users: ~1 GB
```

Sliding window uses more memory but provides better accuracy.

---

### 6.3 — Cleanup Efficiency

On every request, ZREMRANGEBYSCORE removes expired entries. Over time, the sorted set never grows beyond `maxRequests` entries per user.

```
Sorted set size over time:
  t=0:   1 entry  (after first request)
  t=30s: 5 entries (at limit)
  t=60s: 5 entries (oldest expired, newest added)
  t=120s: 0 entries (all expired, TTL may have cleaned up)
```

```
With network latency (~1ms):
```

## Section 7 — Gotchas and Pitfalls

---

### 7.1 — Clock Skew in Distributed Systems

If application servers have different clocks, timestamp-based rate limiting can be inconsistent:

```
Server A (clock = 12:00:00.000):
  Request at 12:00:00.050 → stored with score = 1704067200050

Server B (clock = 11:59:59.900, 100ms behind):
  Request at true time 12:00:00.100 → stored with score = 1704067199100
  → 100ms too early → may be incorrectly evicted from window
```

**Mitigations:**
- Use NTP on all servers
- Accept small clock skew (±1s is usually acceptable)
- Use Redis TIME command to get a consistent timestamp

```csharp
// Use Redis server time for consistency
public async Task<DateTime> GetRedisTimeAsync()
{
    var result = await _db.ScriptEvaluateAsync("return redis.call('TIME')");
    var values = (RedisResult[])result;
    long unixSeconds = (long)values[0];
    long micros = (long)values[1];
    return DateTimeOffset.FromUnixTimeSeconds(unixSeconds)
        .AddTicks(micros * 10).UtcDateTime;
}
```

---

### 7.2 — EXPIRE After ZADD

Always set EXPIRE after ZADD. If you set EXPIRE first and the request fails after ZADD, the sorted set may never be cleaned up:

```csharp
// WRONG: EXPIRE before ZADD
await db.KeyExpireAsync(key, TTL); // TTL starts now
await db.SortedSetAddAsync(key, member, score); // Key might expire before this

// CORRECT: ZADD first, then EXPIRE (or both in Lua)
await db.SortedSetAddAsync(key, member, score);
await db.KeyExpireAsync(key, TTL); // TTL refreshes
```

Best practice: use Lua to ensure both happen atomically.

---

### 7.3 — Sliding Window Memory vs Fixed Window Tradeoff

| Factor | Sliding Window (Sorted Set) | Fixed Window (INCR + EXPIRE) |
|---|---|---|
| Accuracy | Exact | ±1 window at boundary |
| Memory | O(maxRequests) per user | O(1) per user |
| Complexity | O(log N) | O(1) |
| Burst at boundary | None | Up to 2× limit |

Choose based on:
- **High accuracy required**: Sliding window (sorted set)
- **Low memory / high scale**: Fixed window (INCR + EXPIRE)
- **Medium accuracy, medium memory**: Hybrid approach

---

### 7.4 — Sorted Set Grows During Window

Without cleanup, the sorted set grows unbounded for active users. ZREMRANGEBYSCORE on every request keeps it bounded, but if a user stops sending requests, the set persists until TTL expires.

```
Request pattern:
  User sends 100 req/min for 10 minutes → 1000 entries at peak
  User stops → TTL = 66 seconds (window × 1.1)
  After 66s, key is auto-deleted
```

If a user sends requests intermittently, the TTL refreshes each time. Very active users may keep the key alive indefinitely.

---

### 7.5 — Unique Member IDs for Each Request

Each request needs a unique member in the sorted set. Using the same member ID twice would overwrite the score (not add a new entry):

```csharp
// WRONG: same member ID for all requests
string member = "request";
await db.SortedSetAddAsync(key, member, nowMs); // Overwrites previous score!

// CORRECT: unique member per request
string member = $"request:{userId}:{nowMs}:{Guid.NewGuid():N}";
await db.SortedSetAddAsync(key, member, nowMs);
```

---

### 7.6 — Lua Script Cache (EVALSHA)

For frequently used Lua scripts, use EVALSHA to avoid sending the script source on every call:

```csharp
private string? _scriptHash;

public async Task<bool> CheckRateLimitWithEvalshaAsync(
    string key, int max, int windowSec)
{
    if (_scriptHash == null)
    {
        // Load script and get SHA1 hash
        _scriptHash = await _db.ScriptLoadAsync(RateLimitScript);
    }

    try
    {
        var result = await _db.ScriptEvaluateAsync(
            _scriptHash,
            new RedisKey[] { key },
            new RedisValue[] { /* args */ });
        return (int)result == 1;
    }
    catch (RedisServerException ex) when (ex.Message.Contains("NOSCRIPT"))
    {
        // Script was evicted from cache, reload and retry
        _scriptHash = null;
        return await CheckRateLimitWithEvalshaAsync(key, max, windowSec);
    }
}
```

StackExchange.Redis handles EVALSHA caching internally — it will fall back to EVAL automatically if the script is not cached.

### 7.7 — Rate Limiter Key Collisions

Different rate limit scopes should use distinct key patterns to avoid collisions:

```
Good key patterns:
  ratelimit:api:{userId}:{endpoint}
  ratelimit:login:ip:{ipAddress}
  ratelimit:login:user:{username}
  ratelimit:sms:{phoneNumber}

Bad key pattern (collisions possible):
```

### 7.8 — Fail-Open vs Fail-Closed

When Redis is unavailable, decide whether to allow or deny requests:

| Strategy | Behavior | Use When |
|---|---|---|
| Fail-open | Allow request | Availability > rate limiting |
| Fail-closed | Deny request | Security > availability |

Most APIs should fail-open (allow traffic when Redis is down) to avoid a total outage. Use fail-closed only for security-critical rate limits (e.g., login attempts).

```csharp
public async Task<RateLimitResult> IsAllowedWithFailStrategyAsync(
    bool failOpen = true)
{
    try
    {
        return await CheckRateLimitAsync();
    }
    catch (RedisException)
    {
        if (failOpen)
        {
            return new RateLimitResult { IsAllowed = true, Reason = "Redis unavailable — fail open" };
        }
        return new RateLimitResult { IsAllowed = false, Reason = "Redis unavailable — fail closed" };
    }
}
```

### 7.9 — ZREMRANGEBYSCORE Exclusive Bounds

When removing entries outside the window, use an exclusive upper bound to avoid removing the entry at the exact cutoff boundary:

```csharp
// Correct: remove entries with score < cutoff (exclusive max)
await db.SortedSetRemoveRangeByScoreAsync(
    key,
    double.NegativeInfinity,
    cutoffMs,
    Exclude.None); // Stop is exclusive in this context

// Equivalent Redis: ZREMRANGEBYSCORE key -inf (cutoffMs
// The '(' prefix makes the upper bound exclusive
```

StackExchange.Redis's `SortedSetRemoveRangeByScoreAsync` by default treats `stop` as inclusive. To make it exclusive, you need to handle the precision carefully. The Lua script approach using `'(..cutoff)'` is more reliable.

### 7.10 — Large Window Sizes and High Limits

For windows like "10,000 requests per day", the sorted set stores 10,000 entries per user. This is memory-intensive. Consider alternatives:

1. **Token bucket algorithm** — constant memory, less precise
2. **Hybrid approach** — sorted set for short windows, counters for long windows
3. **Redis with more memory** — 10,000 entries × 100 bytes = 1 MB per active user

```csharp
// Hybrid: use sorted set for small window, counter for large window
public async Task<bool> HybridRateCheckAsync(
    string userId, string action)
{
    // Per-second: sliding window (accurate)
    bool perSecond = await CheckTierAsync(userId, action, 10, 1);

    // Per-day: fixed window counter (memory efficient)
    bool perDay = await CheckDailyCounterAsync(userId, action, 10000);

    return perSecond && perDay;
}
```

## Section 8 — Related Patterns and Comparisons

---

### 8.1 — Sorted Set Rate Limiter vs Token Bucket

| Aspect | Sorted Set Sliding Window | Token Bucket |
|---|---|---|
| Algorithm | Log of timestamps | Bucket of tokens |
| Memory | O(maxRequests) per key | O(1) per key |
| Accuracy | Exact (any time window) | Approximate (refill rate) |
| Burst handling | Natural (burst = window size burst) | Configurable (burst = bucket size) |
| Complexity | O(log N) | O(1) |

**Token bucket implementation in Redis:**

```bash
# Token bucket uses sorted set or string + Lua
# Key: "tokenbucket:{user}:{action}"
# Hash fields: tokens (current count), last_refill (timestamp)

-- Lua script for token bucket
local key = KEYS[1]
local now = tonumber(ARGV[1])
local maxTokens = tonumber(ARGV[2])
local refillRate = tonumber(ARGV[3]) -- tokens per second
local refillInterval = tonumber(ARGV[4]) -- seconds between refills

local data = redis.call('HGETALL', key)
local tokens = maxTokens
local lastRefill = now

if #data > 0 then
    tokens = tonumber(data[2])
    lastRefill = tonumber(data[4])
    local elapsed = now - lastRefill
    local refillAmount = math.floor(elapsed / refillInterval) * refillRate
    tokens = math.min(maxTokens, tokens + refillAmount)
    lastRefill = lastRefill + math.floor(elapsed / refillInterval) * refillInterval
end

if tokens >= 1 then
    tokens = tokens - 1
    redis.call('HSET', key, 'tokens', tokens, 'last_refill', lastRefill)
    redis.call('EXPIRE', key, math.ceil(refillInterval * (maxTokens / refillRate + 1)))
    return {1, tokens}
else
    return {0, tokens}
end
```

---

### 8.2 — Sorted Set Rate Limiter vs INCR + EXPIRE

| Aspect | Sorted Set | INCR + EXPIRE |
|---|---|---|
| Window type | Sliding | Fixed |
| Memory per user | ~100 bytes per entry | ~50 bytes total |
| Accuracy at boundary | Exact | ±window burst |
| Implementation complexity | Medium | Low |
| Best for | High accuracy requirements | High scale, low memory |

---

### 8.3 — Sorted Set Rate Limiter vs Redis Cell (Generic Cell Rate Algorithm)

Redis 7.0+ includes the `redis-cell` module (or you can use RESP3 with GCRA):

| Aspect | Sorted Set | redis-cell / GCRA |
|---|---|---|
| Algorithm | Sliding window log | Generic Cell Rate Algorithm |
| Memory | O(limit) per key | O(1) per key |
| Accuracy | Exact | Exact |
| Module dependency | None | Requires redis-cell module |
| Throughput | ~10k/s per instance | ~100k/s per instance |

If you need maximum performance and can install the module, GCRA is superior. The sorted set approach works on any Redis instance with no dependencies.

---

### 8.4 — Rate Limiting with Lua Script vs Client-Side Logic

```
Client-side (multiple round trips):
  1. ZREMRANGEBYSCORE → cleanup
  2. ZCARD → count
  3. if count < limit: ZADD + EXPIRE → add request
  Risk: race condition between steps 2 and 3

Server-side (Lua, single round trip):
  1. EVAL "cleanup + count + check + add" → atomic
  No race condition, fewer network round trips
```

Always prefer the Lua script approach for production.

### 8.5 — Rate Limiting vs Debouncing vs Throttling

| Pattern | Behavior | Redis Implementation |
|---|---|---|
| Rate limiting | Deny if too many in window | Sorted set (this note) |
| Debouncing | Only accept if no request in N seconds | GET key (check last timestamp) |
| Throttling | Enforce minimum interval between requests | GET + SET with TTL |

```csharp
// Debounce example: only allow if 5 seconds since last request
public async Task<bool> CanProceedDebouncedAsync(
    string userId, string action, int cooldownSeconds)
{
    string key = $"debounce:{userId}:{action}";
    RedisValue lastTime = await _db.StringGetAsync(key);

    if (!lastTime.IsNull)
    {
        long lastMs = (long)lastTime;
        long elapsed = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds() - lastMs;
        if (elapsed < cooldownSeconds * 1000L)
        {
            return false; // Still in cooldown
        }
    }

    await _db.StringSetAsync(key,
        DateTimeOffset.UtcNow.ToUnixTimeMilliseconds().ToString(),
        TimeSpan.FromSeconds(cooldownSeconds));
    return true;
}
```

### 9.2 — Command Cheatsheet

```bash
# Setup: user makes 3 requests in a 10-second window
ZADD ratelimit:user:42 1704067200 "req:001"
ZADD ratelimit:user:42 1704067205 "req:002"
ZADD ratelimit:user:42 1704067209 "req:003"

# On 4th request at ts=1704067215:
# Step 1: Remove entries older than (1704067215 - 10000) = 1704067205
ZREMRANGEBYSCORE ratelimit:user:42 -inf (1704067205
# Removes "req:001" (ts=1704067200), keeps "req:002" and "req:003"

# Step 2: Count remaining
ZCARD ratelimit:user:42
# Returns 2

# Step 3: If count < limit (e.g., 5), allow and add
ZADD ratelimit:user:42 1704067215 "req:004"
EXPIRE ratelimit:user:42 15
```

### 9.3 — Lua Script Template

```lua
-- rate_limit_check.lua
-- KEYS[1]: rate limit key
-- ARGV[1]: current timestamp (ms)
-- ARGV[2]: window size (ms)
-- ARGV[3]: max requests
-- ARGV[4]: unique member id
-- ARGV[5]: TTL (seconds)

redis.call('ZREMRANGEBYSCORE', KEYS[1], '-inf', '(' .. (ARGV[1] - ARGV[2]))
local count = redis.call('ZCARD', KEYS[1])

if count >= tonumber(ARGV[3]) then
    return {0, count}
end

redis.call('ZADD', KEYS[1], ARGV[1], ARGV[4])
redis.call('EXPIRE', KEYS[1], ARGV[5])
return {1, count + 1}
```

### 9.4 — Key Design Principles

1. **Always use Lua** for atomic rate check + cleanup + add
2. **Fail open by default** — allow requests when Redis is unavailable
3. **Set TTL after ZADD** — prevent orphaned keys
4. **Use unique member IDs** — each request must be distinct in the sorted set
5. **Choose window type wisely** — sliding (sorted set) vs fixed (INCR + EXPIRE)
6. **Consistent timestamps** — use Redis TIME or NTP-synchronized clocks
7. **Monitor memory** — sorted set grows with maxRequests per active user
8. **Multi-tier for defense in depth** — per-second + per-minute + per-hour

---

### 9.5 — Common Pitfalls Checklist

- [ ] Race condition between ZREMRANGEBYSCORE and ZCARD (use Lua)
- [ ] EXPIRE set before ZADD (key may expire prematurely)
- [ ] Member ID not unique (scores overwritten instead of added)
- [ ] Clock skew between app servers in distributed deployment
- [ ] Sorted set memory grows unbounded for high-limit windows
- [ ] Fail-closed causes total outage when Redis is down
- [ ] Fixed window boundary burst at reset time
- [ ] Too many keys with long TTLs consuming memory
- [ ] ZREMRANGEBYSCORE with inclusive/exclusive bound confusion
- [ ] Missing TTL leads to permanent keys for active users
