### Where This Fits

A .NET backend engineer encounters composite primary keys in junction tables (`OrderId`, `ProductId`), multi-tenant systems where `TenantId` is part of every key, and temporal tables where `(Id, ValidFrom)` forms the PK. The choice between a composite key and a single surrogate key determines JOIN complexity, index size, and whether foreign key references require one column or many. Production systems suffer when composite keys with 4+ columns are used as foreign key targets — every related table must include all columns, and every JOIN requires 4 equality predicates. The interview signal tests whether the candidate knows when a composite key is the right design (junction tables, natural key of the domain) versus when it is an anti-pattern (deep FK graphs, high-volume child tables).

## Core Mental Model

A composite primary key uses two or more columns together to uniquely identify each row. The database engine concatenates the columns into a single key for B-tree ordering — the first column is the leading key, the second breaks ties within the first column's values, and so on. The storage engine does not distinguish between a composite key and a single-column key; both are stored as a concatenated key in the index pages. The difference is in how the key interacts with foreign keys and queries: a 3-column composite PK means every FK reference must include all 3 columns, every JOIN must specify all 3 equality predicates, and every non-clustered index stores all 3 columns as the clustering key (if the table is clustered). The decision hinges on whether the composite key is the target of foreign keys — if yes, prefer a surrogate single-column key; if no (the table is a leaf in the FK graph), a natural composite key is fine.

### Classification

**For junction tables (many-to-many):** The composite PK `(ParentId, ChildId)` is the standard and recommended pattern. The FK graph is shallow — no child table references the junction table.

**For dimension/lookup tables:** Natural composite keys like `(CountryCode, TaxYear)` are appropriate when the combination is provably unique and stable.

**For parent tables with deep FK graphs:** Avoid composite keys. Use a surrogate single-column key and add a UNIQUE constraint on the composite natural key.

**For .NET/EF Core:** Composite keys require `HasKey(e => new { e.Key1, e.Key2 })`. EF Core supports composite foreign keys, but the configuration is verbose and error-prone.

### Key Properties

|Property|Composite PK (2 columns)|Composite PK (4+ columns)|Surrogate + UNIQUE composite|
|---|---|---|---|
|FK column count|2|4+|1|

|Clustering key width|Width of both columns|Width of all 4+ columns|4 bytes (INT)|
|Non-clustered index size|Wide (both columns per row)|Very wide|Narrow (4 bytes per row)|

## Deep Mechanics

### How the Engine Executes This

**Composite PK B-tree structure:**
The storage engine concatenates the key columns into a single sort key. For a PK on `(TenantId, OrderId)`, the sort order is: first by `TenantId`, then by `OrderId` within each `TenantId`. The B-tree pages store the concatenated value — there is no separate structure for each column.

**Composite PK insert:**
1. The storage engine builds the concatenated key from the column values. For `(TenantId = 5, OrderId = 100)`, the sort key is `(5, 100)`.
2. The B-tree is navigated to find the correct position. Rows with lower `TenantId` values sort first; within the same `TenantId`, rows sort by `OrderId`.
3. If most inserts use increasing `OrderId` values but varying `TenantId` values, inserts target different pages — one per active tenant. This can reduce last-page contention compared to a single sequential key.
4. The page split behavior depends on the leading column's insert pattern. If `TenantId` is sequential (new tenants are rare), inserts cluster at the rightmost page for the highest `TenantId`. If `TenantId` is random (e.g., a hash), inserts scatter across pages.

**Composite FK in child tables:**
When another table references a composite PK, the FK must include all PK columns. The child table's clustered index (if it has one) stores the composite FK as part of the clustering key. Every non-clustered index on the child table also stores the composite FK. For a 3-column PK with average width of 20 bytes per column, that is 60 bytes added to every non-clustered index row.

### SQL Visibility

**Composite PK on junction table:**

```sql
CREATE TABLE OrderProducts (
    OrderId      INT NOT NULL,
    ProductId    INT NOT NULL,
    Quantity     INT NOT NULL,
    UnitPrice    DECIMAL(19,4) NOT NULL,
    CONSTRAINT PK_OrderProducts PRIMARY KEY CLUSTERED (OrderId, ProductId),
    CONSTRAINT FK_OrderProducts_Orders FOREIGN KEY (OrderId)
        REFERENCES Orders(OrderId),
    CONSTRAINT FK_OrderProducts_Products FOREIGN KEY (ProductId)
        REFERENCES Products(ProductId)
);

-- Query uses both PK columns
SELECT op.OrderId, op.ProductId, p.ProductName, op.Quantity
FROM OrderProducts op
INNER JOIN Products p ON op.ProductId = p.ProductId
WHERE op.OrderId = 1001;
```

**Problematic composite PK with deep FK graph:**

```sql
-- Parent: composite PK
CREATE TABLE InvoiceLineItems (
    InvoiceId    INT NOT NULL,
    LineNumber   INT NOT NULL,
    ProductId    INT NOT NULL,
    Quantity     INT NOT NULL,
    CONSTRAINT PK_InvoiceLineItems PRIMARY KEY CLUSTERED (InvoiceId, LineNumber),
    CONSTRAINT FK_InvoiceLineItems_Invoice FOREIGN KEY (InvoiceId)
        REFERENCES Invoices(InvoiceId)
);

-- Child: must include both PK columns as FK
CREATE TABLE Shipments (
    ShipmentId   INT IDENTITY(1,1) NOT NULL,
    InvoiceId    INT NOT NULL,
    LineNumber   INT NOT NULL,     -- extra column from composite FK
    ShipDate     DATE NOT NULL,
    Quantity     INT NOT NULL,
    CONSTRAINT PK_Shipments PRIMARY KEY CLUSTERED (ShipmentId),
    CONSTRAINT FK_Shipments_LineItems FOREIGN KEY (InvoiceId, LineNumber)
        REFERENCES InvoiceLineItems(InvoiceId, LineNumber)
);

-- Grandchild: includes 3 FK columns
CREATE TABLE ShipmentItems (
    ShipmentItemId INT IDENTITY(1,1) NOT NULL,
    ShipmentId     INT NOT NULL,
    InvoiceId      INT NOT NULL,    -- 2nd FK column from grandparent
    LineNumber     INT NOT NULL,    -- 3rd FK column from grandparent
    ...
);
```

```csharp
// EF Core — composite FK configuration is verbose
public class Shipment
{
    public int ShipmentId { get; set; }
    public int InvoiceId { get; set; }
    public int LineNumber { get; set; }  // part of composite FK
    public DateTime ShipDate { get; set; }
    public int Quantity { get; set; }
    public InvoiceLineItem LineItem { get; set; } = null!;
}

// OnModelCreating:
modelBuilder.Entity<Shipment>(e =>
{
    e.HasKey(s => s.ShipmentId);
    e.HasOne(s => s.LineItem)
        .WithMany(li => li.Shipments)
        .HasForeignKey(s => new { s.InvoiceId, s.LineNumber });
});
```

**Fix: Add surrogate PK, keep composite as UNIQUE:**

```sql
ALTER TABLE InvoiceLineItems ADD LineItemId INT IDENTITY(1,1);
ALTER TABLE InvoiceLineItems ADD CONSTRAINT PK_InvoiceLineItems PRIMARY KEY CLUSTERED (LineItemId);
ALTER TABLE InvoiceLineItems ADD CONSTRAINT UQ_InvoiceLineItems UNIQUE (InvoiceId, LineNumber);

-- Now child tables only need LineItemId as FK
ALTER TABLE Shipments ADD LineItemId INT NOT NULL;
ALTER TABLE Shipments ADD CONSTRAINT FK_Shipments_LineItems FOREIGN KEY (LineItemId)
    REFERENCES InvoiceLineItems(LineItemId);
```

### Execution Plan Analysis

**Composite PK seek (both columns):**

```sql
SELECT * FROM OrderProducts
WHERE OrderId = 1001 AND ProductId = 500;
```

```
Clustered Index Seek — PK_OrderProducts (OrderId = 1001, ProductId = 500)
  Table 'OrderProducts'. Scan count 0, logical reads 3
```

The seek uses both key columns. The optimizer navigates the B-tree using the full concatenated key. Logical reads are identical to a single-column seek (3 reads for INT keys).

**Composite PK seek (leading column only):**

```sql
SELECT * FROM OrderProducts
WHERE OrderId = 1001;
```

```
Clustered Index Seek — PK_OrderProducts (OrderId = 1001)
  Table 'OrderProducts'. Scan count 1, logical reads 5
```

The optimizer can use the leading column alone as a seek predicate because the B-tree is sorted by the leading column first. This is equivalent to an index on `(OrderId)` with an implicit `INCLUDE (ProductId)`. Logical reads are slightly higher because the range covers multiple `ProductId` values.

**Composite PK seek (non-leading column only):**

```sql
SELECT * FROM OrderProducts
WHERE ProductId = 500;
```

```
Clustered Index Scan — PK_OrderProducts
  Table 'OrderProducts'. Scan count 1, logical reads ~Full table pages
```

Without the leading column, the optimizer cannot seek — it must scan. This is a common performance trap. For this query pattern, add a non-clustered index on `ProductId`.

### Cost Visibility

```sql
SET STATISTICS IO ON;

-- Composite PK: seek on leading column (OrderId)
SELECT * FROM OrderProducts
WHERE OrderId = 1001;
-- Table 'OrderProducts'. Scan count 0, logical reads 3

-- Composite PK: full table scan on non-leading column (ProductId)
SELECT * FROM OrderProducts
WHERE ProductId = 500;
-- Table 'OrderProducts'. Scan count 1, logical reads 8500 (full table)

-- With non-clustered index on ProductId:
-- CREATE INDEX IX_OrderProducts_ProductId ON OrderProducts(ProductId);
SELECT * FROM OrderProducts
WHERE ProductId = 500;
-- Table 'OrderProducts'. Scan count 0, logical reads 4 (non-clustered seek + key lookup)
```

### Failure Modes

**1. Deep FK chains with composite keys.** A 3-column PK in a grandparent table propagates to every child and grandchild table. Each FK reference stores all 3 columns. The JOIN between child and grandchild requires 3 equality predicates.

**2. Non-leading column queries cause scans.** A composite PK on `(A, B, C)` does not support a seek on `B` alone or `C` alone. Queries filtering on non-leading columns scan the entire table.

**3. Composite clustering key width.** Every non-clustered index stores the full composite clustering key. For a 3-column VARCHAR(100) PK, that is 300+ bytes per non-clustered index row — 75x the storage of an INT clustering key.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- Scenario 1: Junction table (composite PK is the right choice)
CREATE TABLE OrderProducts (
    OrderId    INT NOT NULL,
    ProductId  INT NOT NULL,
    Quantity   INT NOT NULL,
    UnitPrice  DECIMAL(19,4) NOT NULL,
    CONSTRAINT PK_OrderProducts PRIMARY KEY CLUSTERED (OrderId, ProductId),
    CONSTRAINT FK_OrderProducts_Orders FOREIGN KEY (OrderId)
        REFERENCES Orders(OrderId),
    CONSTRAINT FK_OrderProducts_Products FOREIGN KEY (ProductId)
        REFERENCES Products(ProductId)
);

-- Scenario 2: Multi-tenant table with tenant isolation (composite leading with TenantId)
CREATE TABLE TenantOrders (
    TenantId   INT NOT NULL,
    OrderId    INT IDENTITY(1,1) NOT NULL,
    CustomerId INT NOT NULL,
    OrderTotal DECIMAL(19,4) NOT NULL,
    CONSTRAINT PK_TenantOrders PRIMARY KEY CLUSTERED (TenantId, OrderId)
);
-- Leading with TenantId scatters inserts across tenants, reducing last-page contention.
-- Query pattern: WHERE TenantId = @tid AND OrderId = @oid

-- Scenario 3: Natural composite key for a lookup table
CREATE TABLE TaxRates (
    CountryCode  CHAR(2) NOT NULL,
    TaxYear      INT NOT NULL,
    TaxRate      DECIMAL(5,4) NOT NULL,
    CONSTRAINT PK_TaxRates PRIMARY KEY CLUSTERED (CountryCode, TaxYear)
);

-- Scenario 4: Deep FK graph — use surrogate instead
CREATE TABLE InvoiceLineItems (
    LineItemId   INT IDENTITY(1,1) NOT NULL,  -- surrogate PK
    InvoiceId    INT NOT NULL,
    LineNumber   INT NOT NULL,
    ProductId    INT NOT NULL,
    Quantity     INT NOT NULL,
    CONSTRAINT PK_InvoiceLineItems PRIMARY KEY CLUSTERED (LineItemId),
    CONSTRAINT UQ_InvoiceLineItems UNIQUE (InvoiceId, LineNumber),
    CONSTRAINT FK_InvoiceLineItems_Invoice FOREIGN KEY (InvoiceId)
        REFERENCES Invoices(InvoiceId)
);
```

        // Multi-tenant

        {
            e.HasKey(to => new { to.TenantId, to.OrderId });
            e.Property(to => to.OrderId).ValueGeneratedOnAdd();
        });

        // Lookup
        modelBuilder.Entity<TaxRate>(e =>
        {
            e.HasKey(tr => new { tr.CountryCode, tr.TaxYear });
            e.Property(tr => tr.CountryCode).HasMaxLength(2).IsFixedLength();
        });

        // Surrogate
        modelBuilder.Entity<InvoiceLineItem>(e =>
        {
            e.HasKey(li => li.LineItemId);
            e.Property(li => li.LineItemId).ValueGeneratedOnAdd();
            e.HasIndex(li => new { li.InvoiceId, li.LineNumber }).IsUnique();
        });
    }
}

public class OrderRepository
{
    private readonly IDbConnectionFactory _connectionFactory;

    public OrderRepository(IDbConnectionFactory connectionFactory)
    {
        _connectionFactory = connectionFactory;
    }

    // Composite key query
    public async Task<OrderProduct?> GetOrderProductAsync(
        int orderId, int productId, CancellationToken ct = default)
    {
        const string sql = @"
            SELECT OrderId, ProductId, Quantity, UnitPrice
            FROM OrderProducts
            WHERE OrderId = @OrderId AND ProductId = @ProductId";

        await using var connection = _connectionFactory.Create();
        return await connection.QueryFirstOrDefaultAsync<OrderProduct>(
            new CommandDefinition(sql,
                new { OrderId = orderId, ProductId = productId },
                cancellationToken: ct));
    }

    // Query by leading column only
    public async Task<IReadOnlyList<OrderProduct>> GetOrderProductsAsync(
        int orderId, CancellationToken ct = default)
    {
        const string sql = @"
            SELECT op.OrderId, op.ProductId, p.ProductName,
                   op.Quantity, op.UnitPrice
            FROM OrderProducts op
            INNER JOIN Products p ON op.ProductId = p.ProductId
            WHERE op.OrderId = @OrderId";

        await using var connection = _connectionFactory.Create();
        var results = await connection.QueryAsync<OrderProduct>(
            new CommandDefinition(sql, new { OrderId = orderId },
                cancellationToken: ct));
        return results.AsList();
    }

    // Insert with composite key
    public async Task InsertOrderProductAsync(
        OrderProduct op, CancellationToken ct = default)
    {
        const string sql = @"
            INSERT INTO OrderProducts (OrderId, ProductId, Quantity, UnitPrice)
            VALUES (@OrderId, @ProductId, @Quantity, @UnitPrice)";

        await using var connection = _connectionFactory.Create();
        await connection.ExecuteAsync(
            new CommandDefinition(sql, op, cancellationToken: ct));
    }

    // Insert with surrogate key — returns generated key
    public async Task<int> InsertLineItemAsync(
        InvoiceLineItem item, CancellationToken ct = default)
    {
        const string sql = @"
            INSERT INTO InvoiceLineItems (InvoiceId, LineNumber, ProductId, Quantity)
            OUTPUT INSERTED.LineItemId
            VALUES (@InvoiceId, @LineNumber, @ProductId, @Quantity)";

        await using var connection = _connectionFactory.Create();
        return await connection.ExecuteScalarAsync<int>(
            new CommandDefinition(sql, item, cancellationToken: ct));
    }
}

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL: same composite PK syntax
CREATE TABLE OrderProducts (
    OrderId   INT NOT NULL,
    ProductId INT NOT NULL,
    Quantity  INT NOT NULL,
    PRIMARY KEY (OrderId, ProductId)
);

-- PostgreSQL: no clustered index by default
-- The composite PK creates a unique B-tree index.
-- No clustering key is stored in other indexes.

-- PostgreSQL: INCLUDE columns in unique constraint
CREATE TABLE InvoiceLineItems (
    LineItemId INT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    InvoiceId  INT NOT NULL,
    LineNumber INT NOT NULL,
    ProductId  INT NOT NULL,
    Quantity   INT NOT NULL,
    UNIQUE (InvoiceId, LineNumber)
);
```

PostgreSQL's heap storage eliminates the clustering-key-in-non-clustered-indexes problem. The composite PK is just another index — child tables store their own references as regular columns. This makes composite keys in PostgreSQL less harmful than in SQL Server.

## Gotchas and Production Pitfalls

### 1. Composite PK on a table that becomes the target of FKs

**Pitfall:** The engineer starts with a composite PK on `InvoiceLineItems` and later adds `Shipments` referencing it.

```sql
-- ❌ Composite PK that propagates
CREATE TABLE InvoiceLineItems (
    InvoiceId  INT NOT NULL,
    LineNumber INT NOT NULL,
    PRIMARY KEY (InvoiceId, LineNumber)
);
```

**Symptom:** By the time `ShipmentItems` (grandchild) is added, the FK path requires 6 columns. JOINs between `Shipments` and `InvoiceLineItems` require 2 equality predicates. Every non-clustered index on `Shipments` and `ShipmentItems` stores the 2-column composite FK.

**Fix:** Use a surrogate PK on `InvoiceLineItems` from the start. Add UNIQUE on `(InvoiceId, LineNumber)` for business key enforcement.

**Cost of not fixing:** Schema migration to add the surrogate key later requires updating all FK references — an expensive DDL operation on tables with 100M+ rows.

### 2. Non-leading column query causes full scan

**Pitfall:** The engineer queries a composite PK on a non-leading column without adding a secondary index.

```sql
-- PK is (TenantId, OrderId)
-- This query scans the entire table:
SELECT * FROM TenantOrders WHERE OrderId = 1001;
```

**Symptom:** The query performs a Clustered Index Scan on `TenantOrders` — 50,000 logical reads for a single-row lookup. Execution time: 300ms instead of 1ms.

**Fix:** Add a non-clustered index on the non-leading column:

```sql
CREATE INDEX IX_TenantOrders_OrderId ON TenantOrders(OrderId);
```

**Cost of not fixing:** Every query by `OrderId` alone scans the full table. At 100 requests/second, this generates 5M logical reads/second — saturating the buffer pool.

### 3. Order of columns in composite PK matters for performance

**Pitfall:** The engineer chooses column order based on logical grouping rather than query patterns.

```sql
-- ❌ Wrong order: ProductId first, OrderId second
-- Query pattern: 95% by OrderId
CREATE TABLE OrderProducts (
    ProductId INT NOT NULL,
    OrderId   INT NOT NULL,
    PRIMARY KEY (ProductId, OrderId)
);
```

**Symptom:** The 95% of queries that filter by `OrderId` cannot seek — they scan the entire table.

**Fix:** Leading column should match the most common query predicate:

```sql
-- ✅ Correct order: OrderId first (matches query pattern)
CREATE TABLE OrderProducts (
    OrderId   INT NOT NULL,
    ProductId INT NOT NULL,
    PRIMARY KEY (OrderId, ProductId)
);
```

**Cost of not fixing:** The most common query pattern performs a table scan instead of an index seek. This is a silent performance killer — the query works but is 100x slower than necessary.

### 4. EF Core composite FK configuration errors

**Pitfall:** The engineer defines a composite FK in EF Core but misorders the foreign key properties.

```csharp
modelBuilder.Entity<Shipment>(e =>
{
    e.HasOne(s => s.LineItem)
        .WithMany(li => li.Shipments)
        .HasForeignKey(s => new { s.InvoiceId, s.LineNumber });
        // Order must match the PK order: (InvoiceId, LineNumber)
});
```

**Symptom:** EF Core throws `InvalidOperationException: The number of properties in the foreign key does not match the number of properties in the principal key` or (worse) silently creates a mismatched FK that violates referential integrity.

**Fix:** Ensure the anonymous type order in `HasForeignKey` exactly matches the principal key order.

**Cost of not fixing:** Runtime exception on first query, or silent data corruption if the FK is mismatched.

### 5. Composite key width causes non-clustered index bloat

**Pitfall:** The engineer uses a composite clustered PK with 3+ wide VARCHAR columns.

```sql
-- PK is (CountryCode CHAR(2), TaxCode VARCHAR(50), EffectiveDate DATE = 8 bytes)
-- Total clustered key: ~60 bytes per row
```

**Symptom:** A non-clustered index on `(CountryCode, TaxYear)` stores the 60-byte clustering key in every row. Index size: 60 bytes × 50M rows = 3 GB just for the clustering key overhead.

**Fix:** Add a surrogate INT identity as the clustered PK. Keep the composite business key as a UNIQUE non-clustered index.

**Cost of not fixing:** Index maintenance windows grow longer. Buffer pool memory is consumed by oversized non-clustered index pages. Every index seek reads more pages because the B-tree is deeper (fewer 60-byte entries per page).

## Performance Implications

### Benchmark: Composite PK vs Surrogate PK with Composite UNIQUE

```sql
SET STATISTICS IO ON;

-- Composite PK: insert into junction table
INSERT INTO OrderProducts (OrderId, ProductId, Quantity, UnitPrice)
VALUES (1001, 500, 2, 49.99);
-- Table 'OrderProducts'. Scan count 0, logical reads 1

-- Composite PK: seek on both columns
SELECT * FROM OrderProducts
WHERE OrderId = 1001 AND ProductId = 500;
-- Table 'OrderProducts'. Scan count 0, logical reads 3

-- Composite PK: seek on leading column only
SELECT * FROM OrderProducts
WHERE OrderId = 1001;
-- Table 'OrderProducts'. Scan count 0, logical reads 3

-- Surrogate PK + composite UNIQUE: insert
INSERT INTO InvoiceLineItems (InvoiceId, LineNumber, ProductId, Quantity, UnitPrice)
VALUES (1001, 1, 500, 2, 49.99);
-- Table 'InvoiceLineItems'. Scan count 0, logical reads 2
-- (1 for clustered insert + 1 for UNIQUE index insert)

-- Surrogate PK: seek by surrogate
SELECT * FROM InvoiceLineItems WHERE LineItemId = 500;
-- Table 'InvoiceLineItems'. Scan count 0, logical reads 3

-- Surrogate PK: seek by composite business key (via UNIQUE index)
SELECT * FROM InvoiceLineItems
WHERE InvoiceId = 1001 AND LineNumber = 1;
-- Table 'InvoiceLineItems'. Scan count 0, logical reads 4
-- (3 for UNIQUE index seek + 1 for clustered key lookup)
```

**Improvement:** Surrogate PK adds 1 extra logical read per insert (for the UNIQUE index) but reduces FK-related reads across all child tables by 30–50% (narrower FK key).

### Spoken Answers

**Q: When is a composite primary key the right design choice?**

> **Average answer:** Composite keys are good for junction tables in many-to-many relationships and for tables where the natural key has multiple columns.

> **Great answer:** A composite primary key is the right choice in three specific situations. First, junction tables for many-to-many relationships — `OrderProducts` with `(OrderId, ProductId)` — because the junction table is a leaf in the FK graph (no other table references it), the composite key enforces uniqueness at the business level without a separate unique constraint, and both columns are foreign keys anyway so no extra storage is introduced. Second, multi-tenant tables where `TenantId` must be part of every key — leading with `TenantId` provides implicit partitioning: queries are naturally scoped to a tenant, and inserts scatter across tenant pages instead of contending on a single last page. Third, time-series or versioned tables where the natural unique combination is multi-column, like `(CountryCode, TaxYear)` for tax rates, and the table is small or has no child FK references. The common thread: composite keys are correct when the table is a leaf in the foreign key graph. As soon as another table needs to reference this table, the composite key becomes a burden — every child table must store all PK columns, every JOIN requires multiple predicates, and non-clustered index sizes balloon. In those cases, add a surrogate single-column PK and enforce the business key with a UNIQUE constraint.

**Q: How does column order in a composite PK affect query performance?**

> **Great answer:** The leading column of a composite PK determines the B-tree sort order and whether a query can seek or must scan. A composite PK on `(A, B)` is stored in the B-tree sorted first by `A`, then by `B`. Queries that filter on `A` alone can seek — the optimizer navigates to the `A` value and scans within that range. Queries that filter on `B` alone cannot seek — they must scan the entire index. Queries that filter on both `A` and `B` seek using both columns. The practical rule: put the column with the most selective, most frequently filtered predicate first. For a junction table, if 95% of queries start from `OrderId` and look up products, put `OrderId` first. If the table is queried equally from both directions, create a non-clustered index on the second column. In the execution plan, a seek on the leading column alone shows `Seek Keys[1]: Prefix: (A = @a)` with a range scan on `B`. A seek on both columns shows `Seek Keys[1]: Start: (A, B) >= (@a, @b) AND End: (A, B) <= (@a, @b)`.

### Application Checklist

- [ ] Is the table a junction or leaf table with no child FK references?
- [ ] Is the composite key composed of 2–3 narrow columns (INT, SMALLINT, CHAR(2–3))?
- [ ] Does the most common query predicate match the leading column?
- [ ] Is a non-clustered index on non-leading columns in place if needed?
- [ ] If the table has children, is there a surrogate PK strategy instead?

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|No separate UNIQUE index needed (PK enforces uniqueness)|Composite key propagates to all child FK tables|
|Leading column queries can seek without a secondary index|Non-leading column queries scan without a secondary index|
|Multi-tenant key ensures partition isolation|Wider clustering key inflates non-clustered index sizes|
|Semantically clear primary key|FK references require multiple columns|

### Scale Thresholds

- **Composite PK fine at any scale for leaf tables** — junction tables with no children
- **Composite PK hurts at > 10 FK children** — each child stores the composite width × row count
- **Leading column decision matters at > 1M rows** — non-leading scans become expensive
- **Composite clustering key width matters at > 10M rows** — non-clustered index size doubles per extra 4 bytes

**Challenge 1 — Write the SQL**

Design a schema for a product catalog where each product belongs to multiple categories, and each category can have multiple products. The system runs at 10M products and 1M categories. Show the table DDL with appropriate PK choices.

<details>
<summary>Solution</summary>

```sql
-- Junction table: composite PK is correct (leaf table)
CREATE TABLE ProductCategories (
    ProductId    INT NOT NULL,
    CategoryId   INT NOT NULL,
    IsPrimary    BIT NOT NULL DEFAULT 0,
    DisplayOrder INT NOT NULL DEFAULT 0,
    CONSTRAINT PK_ProductCategories PRIMARY KEY CLUSTERED (ProductId, CategoryId),
    CONSTRAINT FK_ProductCategories_Products FOREIGN KEY (ProductId)
        REFERENCES Products(ProductId),
    CONSTRAINT FK_ProductCategories_Categories FOREIGN KEY (CategoryId)
        REFERENCES Categories(CategoryId)
);

-- Query by product (leading column seek):
SELECT c.CategoryName, pc.IsPrimary
FROM ProductCategories pc
INNER JOIN Categories c ON pc.CategoryId = c.CategoryId
WHERE pc.ProductId = 1001
ORDER BY pc.DisplayOrder;

-- Query by category (non-leading column — needs secondary index):
CREATE INDEX IX_ProductCategories_CategoryId ON ProductCategories(CategoryId);

SELECT p.ProductName, pc.IsPrimary
FROM ProductCategories pc
INNER JOIN Products p ON pc.ProductId = p.ProductId
WHERE pc.CategoryId = 500
ORDER BY pc.DisplayOrder;
```

**Challenge 2 — Fix the performance problem**

```sql
-- This query runs in 8 seconds on a 50M row table.
-- PK: (TenantId, OrderId, LineNumber)
SELECT * FROM InvoiceLineItems
WHERE OrderId = 1001 AND LineNumber = 1;
-- SET STATISTICS IO: logical reads = 450,000 (full table scan)
```

Identify why and fix it.

<details> <summary>Solution</summary>

**Root cause:** The query filters on `OrderId` and `LineNumber` but not on `TenantId`. The leading column `TenantId` is missing from the WHERE clause, so the optimizer cannot seek on the composite PK — it scans the entire clustered index.

**Fix (two options):**

Option A — Add TenantId to the query (if available):
```sql
SELECT * FROM InvoiceLineItems
WHERE TenantId = @tenantId AND OrderId = 1001 AND LineNumber = 1;
-- Seek on PK — 3 logical reads
```

Option B — Add a non-clustered index on (OrderId, LineNumber):
```sql
CREATE INDEX IX_InvoiceLineItems_OrderId_LineNumber
ON InvoiceLineItems(OrderId, LineNumber)
INCLUDE (ProductId, Quantity, UnitPrice);
-- Seek on new index — 4 logical reads
```

**Index to create:** The non-clustered index above.

**After fix — logical reads:** ~4 (from 450,000) — a 100,000x reduction.

</details>

**Challenge 3 — Explain the execution plan**

```sql
SELECT op.OrderId, op.ProductId, p.ProductName, op.Quantity
FROM OrderProducts op
INNER JOIN Products p ON op.ProductId = p.ProductId
WHERE op.OrderId = 1001;
```

PK on `OrderProducts` is `(OrderId, ProductId)`. The query returns 50 rows. Explain the execution plan.

<details> <summary>Solution</summary>

**Execution plan:**

```
Clustered Index Seek — PK_OrderProducts (OrderId = 1001)
  |-- 3 logical reads (seek to OrderId 1001, range scan within ProductId)
  |-- Estimated rows: 50
  |-- Actual rows: 50

Nested Loops (Inner Join)
  |-- Outer: Clustered Index Seek results (50 rows)
  |-- Inner: Clustered Index Seek — PK_Products (ProductId = @pid)
  |   |-- 3 logical reads per execution
  |   |-- 50 executions × 3 reads = 150 total logical reads

Total logical reads: ~153
```

**Why Seek on leading column only:** The optimizer recognizes that `OrderId` is the leading column of the composite PK. It seeks to the first row where `OrderId = 1001` and then scans forward until `OrderId` changes. This is a range scan, not a point seek — but it is much cheaper than a full table scan.

**Why Nested Loops:** The outer input (OrderProducts) produces 50 rows. For each row, a seek into Products by ProductId is optimal. The optimizer estimates 50 rows, which is well below the Nested Loops threshold (~10,000 rows for a Hash Match).

**What would change without the leading column:** If the WHERE clause were `WHERE ProductId = 500` without `OrderId`, the plan would show a Clustered Index Scan (full table) instead of a seek, with 450,000+ logical reads.

</details>

**Challenge 4 — Diagnose the concurrency problem**

A multi-tenant SaaS system uses `(TenantId, OrderId)` as the clustered PK on a `TenantOrders` table. The `OrderId` is an `INT IDENTITY(1,1)` per tenant (reset each tenant). After adding 50 new tenants in one day, the system experiences increased `PAGELATCH_EX` waits and page splits. What is happening?

<details> <summary>Solution</summary>

**Root cause:** The composite PK `(TenantId, OrderId)` with `OrderId` as IDENTITY means inserts from existing tenants go to the page for their highest `TenantId` + the latest `OrderId`. But new tenants have a higher `TenantId`. When a new tenant inserts its first order, it goes to a new page — the rightmost page for the highest `TenantId`. All 50 new tenants insert into the same rightmost page range, causing last-page latch contention similar to a single INT IDENTITY key. The page splitting is normal sequential-page behavior, but the latch contention is caused by all new tenants hitting the same page.

**Detection query:**

```sql
SELECT wait_type, wait_time_ms, waiting_tasks_count
FROM sys.dm_os_wait_stats
WHERE wait_type = 'PAGELATCH_EX';
```

**Fix:** Add `OPTIMIZE_FOR_SEQUENTIAL_KEY = ON` to reduce last-page contention:

```sql
ALTER INDEX PK_TenantOrders ON TenantOrders SET (OPTIMIZE_FOR_SEQUENTIAL_KEY = ON);
```

Alternatively, use a hash-distributed leading key or reverse the column order if the IDENTITY is the primary insert pattern (though this loses tenant-partitioned order).

**In .NET:** No code change needed. The optimizer hint is transparent to EF Core and Dapper.

</details>

**Challenge 5 — Design the index**

An e-commerce system has a `Shipments` table that references `InvoiceLineItems` — which has a composite PK `(InvoiceId, LineNumber)`. The `Shipments` table has 200M rows and is queried by `(InvoiceId, LineNumber)` and also by `ShipmentId` (surrogate PK). Design the index strategy.

<details> <summary>Solution</summary>

```sql
-- Keep surrogate PK on Shipments (narrow, fast)
-- Add covering index for the composite FK lookup

CREATE TABLE Shipments (
    ShipmentId    INT IDENTITY(1,1) NOT NULL,
    InvoiceId     INT NOT NULL,
    LineNumber    INT NOT NULL,
    ShipDate      DATE NOT NULL,
    Quantity      INT NOT NULL,
    TrackingCode  VARCHAR(100) NOT NULL,
    CONSTRAINT PK_Shipments PRIMARY KEY CLUSTERED (ShipmentId),
    CONSTRAINT FK_Shipments_LineItems FOREIGN KEY (InvoiceId, LineNumber)
        REFERENCES InvoiceLineItems(InvoiceId, LineNumber)
);

-- Covering index for the most common query pattern:
CREATE INDEX IX_Shipments_InvoiceId_LineNumber
ON Shipments(InvoiceId, LineNumber)
INCLUDE (ShipmentId, ShipDate, Quantity, TrackingCode);
-- This is a covering index: the query never needs to touch the clustered PK.
-- The 2-column FK key is stored only in this index, not in the clustered PK.

-- Alternatively, convert InvoiceLineItems to a surrogate PK:
-- Then Shipments only needs LineItemId INT as FK.
-- This is the long-term fix.
```

**Why this works:** The covering index on `(InvoiceId, LineNumber)` supports the FK lookup with a non-clustered index seek (3–4 reads) and includes all columns needed for the query — no key lookup needed. The clustered PK on `ShipmentId` keeps the table's clustering key narrow (4 bytes).

**Tradeoffs:** The covering index adds write overhead (~1.3x logical reads per insert). The index stores `(InvoiceId, LineNumber)` at 8+ bytes per row = ~1.6 GB for 200M rows. Acceptable for the query performance gain.

**Better long-term fix:** Migrate `InvoiceLineItems` to surrogate PK `LineItemId`, change `Shipments.LineItemId` to a single INT FK, and drop the composite FK. This eliminates the composite FK overhead entirely.

</details>
