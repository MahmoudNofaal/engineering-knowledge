### Where This Fits

A .NET backend engineer building a SaaS product must choose how to isolate tenant data at the database level. The three approaches — shared database/shared schema, shared database/separate schema, and separate database — determine the ceiling for scalability, the cost of tenant isolation, and the complexity of cross-tenant queries. Production systems fail when a missing `WHERE TenantId = @tid` exposes Tenant A's data to Tenant B, when a shared-schema approach hits row-count limits for a single large tenant, or when the schema-per-tenant approach creates tens of thousands of identical tables that make schema migrations a nightmare. The interview signal tests whether the candidate can articulate the tradeoffs in terms of isolation, performance at scale, and operational complexity.

## Core Mental Model

Multi-tenancy at the database level is a spectrum of isolation. Shared database/shared schema — every tenant's rows live in the same tables, distinguished by a `TenantId` column. Shared database/separate schema — each tenant has its own schema (`tenant1.Orders`, `tenant2.Orders`) within the same database. Separate database — each tenant has its own database, possibly on separate servers. The database engine does not distinguish between these approaches; a `WHERE TenantId = @tid` filter on a shared table is the same predicate as a cross-database query. The difference is in operational properties: shared schema offers the simplest operations (one table to manage, one index to maintain) but requires every query to include `TenantId` and offers no physical isolation. Separate databases offer full isolation (backup one tenant, tune one tenant's indexes independently) but multiply operational overhead by the tenant count.

### Classification

**For isolation:** Separate database > separate schema > shared schema. Separate databases provide hardware-level isolation; separate schemas provide schema-level name isolation; shared schema relies on query filters.

**For performance at scale:** Shared schema with `TenantId` as the leading clustered index column provides excellent per-tenant query performance (seek on `TenantId`). Separate databases allow per-tenant resource governance (resource governor, pool sizing). Separate schemas offer no performance advantage over shared schema — they add overhead for cross-schema queries.

**For operational complexity:** Shared schema is simplest (one migration, one backup, one index maintenance job). Separate schemas complicate migrations (must apply to N schemas). Separate databases multiply every operational task by tenant count.

**For .NET/EF Core:** Shared schema uses global query filters (`HasQueryFilter(e => e.TenantId == _currentTenantId)`). Separate schemas use `HasDefaultSchema()` per tenant. Separate databases use separate `DbContext` instances or connection strings.

## Deep Mechanics

### How the Engine Executes This

**Shared schema — clustered index with TenantId leading:**
1. The PK is `(TenantId, OrderId)` where `OrderId` is `INT IDENTITY` per tenant. The clustered index sorts all rows first by `TenantId`, then by `OrderId`.
2. A query `WHERE TenantId = @tid AND OrderId = @oid` performs a clustered index seek on the composite key. The seek navigates to the `TenantId` value, then within that to the `OrderId` value.
3. A query `WHERE TenantId = @tid ORDER BY OrderId DESC` seeks to the last `OrderId` for the given `TenantId` — only the pages for that tenant are touched.
4. Because rows for different tenants are stored in separate B-tree ranges, a large tenant with 100M rows does not affect the B-tree depth for a small tenant with 100 rows — the small tenant's rows are in a compact range of pages.

**Separate schema — tenant-specific tables:**
1. Each tenant's table `tenant1.Orders`, `tenant2.Orders` is a separate B-tree. The PK is just `OrderId INT IDENTITY` (no `TenantId` column).
2. Queries do not need `TenantId` — the schema name provides the isolation.
3. Schema-level operations (backup, restore, index rebuild) affect all tenants together because they are in the same database.

**Separate database — tenant-specific database:**
1. Each tenant's database is on the same or different server. The connection string is selected per tenant.
2. Full physical isolation: one tenant's backup, restore, or index maintenance does not affect others.
3. Cross-tenant queries (admin reporting) require linked servers or application-level aggregation.

### SQL Visibility

**Shared schema:**

```sql
CREATE TABLE Orders (
    TenantId     INT NOT NULL,
    OrderId      INT IDENTITY(1,1) NOT NULL,
    CustomerId   INT NOT NULL,
    OrderTotal   DECIMAL(19,4) NOT NULL,
    OrderDate    DATETIME2 NOT NULL DEFAULT SYSUTCDATETIME(),
    CONSTRAINT PK_Orders PRIMARY KEY CLUSTERED (TenantId, OrderId)
);

-- Query for a specific tenant
SELECT OrderId, OrderTotal, OrderDate
FROM Orders
WHERE TenantId = @TenantId
ORDER BY OrderDate DESC;

-- Important: TenantId must be in WHERE clause of EVERY query
```

```csharp
public class Order
{
    public int TenantId { get; set; }
    public int OrderId { get; set; }
    public int CustomerId { get; set; }
    public decimal OrderTotal { get; set; }
    public DateTime OrderDate { get; set; }
}

public class AppDbContext : DbContext
{
    private readonly int _tenantId;

    public AppDbContext(DbContextOptions<AppDbContext> options,
        ITenantService tenantService) : base(options)
    {
        _tenantId = tenantService.CurrentTenantId;
    }

    public DbSet<Order> Orders => Set<Order>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Order>(e =>
        {
            e.HasKey(o => new { o.TenantId, o.OrderId });
            e.Property(o => o.OrderId).ValueGeneratedOnAdd();
            e.HasQueryFilter(o => o.TenantId == _tenantId);
            -- global query filter — automatically added to every LINQ query
        });
    }
}

-- Generated SQL by EF Core:
-- SELECT [o].[TenantId], [o].[OrderId], [o].[CustomerId], ...
-- FROM [Orders] AS [o]
-- WHERE [o].[TenantId] = @__tenantId_0
--   AND [o].[OrderDate] >= ...
```

**Separate schema:**

```csharp
// Each tenant gets a dedicated DbContext with a specific schema
public class TenantDbContext : DbContext
{
    private readonly string _schema;

    public TenantDbContext(DbContextOptions<TenantDbContext> options,
        string schema) : base(options)
    {
        _schema = schema;
    }

    public DbSet<Order> Orders => Set<Order>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.HasDefaultSchema(_schema);
        modelBuilder.Entity<Order>(e =>
        {
            e.HasKey(o => o.OrderId);
            e.Property(o => o.OrderId).ValueGeneratedOnAdd();
        });
    }
}

    .UseSqlServer(connectionString)
    .Options;

-- Generated SQL:
-- SELECT [o].[OrderId], [o].[CustomerId], ...
-- FROM [tenant_42].[Orders] AS [o]
-- WHERE [o].[OrderDate] >= @p0
```

### Execution Plan Analysis

**Shared schema — per-tenant query with TenantId leading:**

```
Clustered Index Seek — PK_Orders (TenantId = 42, OrderId >= @min)
  |-- 3 logical reads (seek to tenant 42's range, scan forward)
  |-- Only pages for Tenant 42 are touched
```

**Shared schema — cross-tenant query (missing WHERE TenantId):**

```
Clustered Index Scan — PK_Orders
  |-- 450,000 logical reads (full table, all tenants)
```

Without `TenantId`, the query cannot seek — it scans all tenants' data.

### Cost Visibility

```sql
SET STATISTICS IO ON;

-- Shared schema: per-tenant query (TenantId = 42, 50K rows in tenant)
SELECT OrderId, OrderTotal FROM Orders
WHERE TenantId = 42 AND OrderDate >= '2026-01-01';
-- Table 'Orders'. Scan count 0, logical reads 45 (tenant's data only)

-- Shared schema: cross-tenant query (no TenantId — 10M rows total)
SELECT OrderId, OrderTotal FROM Orders
WHERE OrderDate >= '2026-01-01';
-- Table 'Orders'. Scan count 1, logical reads 45,000 (full table, all tenants)
```

### Failure Modes

**1. Missing TenantId in WHERE clause.** A developer writes `SELECT * FROM Orders WHERE OrderId = 5` without `TenantId = @tid`. If Tenant A's OrderId=5 and Tenant B's OrderId=5 both exist, the query returns one row based on lock or scan order — potentially exposing Tenant B's data to Tenant A.

**2. TenantId not leading in PK.** The PK is `(OrderId IDENTITY)` with `TenantId` as a non-key column. Queries by `TenantId` perform a full clustered index scan.

**3. No filtered index per tenant.** An index on `(TenantId, OrderDate)` is shared across all tenants. A single large tenant can dominate the index. Separate databases avoid this.

**4. Schema-per-tenant migration failures.** A migration adds a column to 10,000 tenant schemas. If the 7,503rd migration fails, the database is in a partially migrated state.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- Shared schema pattern (recommended for most SaaS)
CREATE TABLE Orders (
    TenantId     INT NOT NULL,
    OrderId      INT IDENTITY(1,1) NOT NULL,
    CustomerId   INT NOT NULL,
    OrderTotal   DECIMAL(19,4) NOT NULL,
    OrderStatus  VARCHAR(50) NOT NULL DEFAULT 'Pending',
    OrderDate    DATETIME2 NOT NULL DEFAULT SYSUTCDATETIME(),
    CreatedBy    VARCHAR(200) NOT NULL,
    CONSTRAINT PK_Orders PRIMARY KEY CLUSTERED (TenantId, OrderId),
    CONSTRAINT FK_Orders_Tenants FOREIGN KEY (TenantId) REFERENCES Tenants(TenantId)
);

-- Tenant-partitioned index for common per-tenant queries
CREATE INDEX IX_Orders_TenantId_OrderDate
ON Orders(TenantId, OrderDate DESC)
INCLUDE (CustomerId, OrderTotal, OrderStatus);

-- Row-level security (SQL Server 2016+):
CREATE SECURITY POLICY TenantAccessPolicy
ADD FILTER PREDICATE dbo.fn_TenantAccessPredicate(TenantId)
ON dbo.Orders;
```

### EF Core Implementation

```csharp
public interface ITenantService
{
    int CurrentTenantId { get; }
    string CurrentTenantSchema { get; }
    string GetConnectionString(int tenantId);
}

// Shared schema with global query filter
public class AppDbContext : DbContext
{
    private readonly int _tenantId;

    public AppDbContext(DbContextOptions<AppDbContext> options,
        ITenantService tenantService) : base(options)
    {
        _tenantId = tenantService.CurrentTenantId;
    }

    public DbSet<Order> Orders => Set<Order>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Order>(e =>
        {
            e.HasKey(o => new { o.TenantId, o.OrderId });
            e.Property(o => o.OrderId).ValueGeneratedOnAdd();
            e.Property(o => o.TenantId).IsRequired();
            e.HasQueryFilter(o => o.TenantId == _tenantId);
            e.HasIndex(o => new { o.TenantId, o.OrderDate })
                .IsDescending(false, true);
        });

        // Apply to all entities via IMutableEntityType
        foreach (var entity in modelBuilder.Model.GetEntityTypes()
            .Where(e => typeof(ITenantScoped).IsAssignableFrom(e.ClrType)))
        {
            entity.AddQueryFilter<ITenantScoped>(
                e => e.TenantId == _tenantId);
        }
    }
}

public interface ITenantScoped
{
    int TenantId { get; set; }
}

// Usage — TenantId is transparently set on INSERT:
public override async Task<int> SaveChangesAsync(
    bool acceptAllChangesOnSuccess, CancellationToken ct = default)
{
    foreach (var entry in ChangeTracker.Entries<ITenantScoped>())
    {
        if (entry.State == EntityState.Added)
        {
            entry.Entity.TenantId = _tenantId;
        }
    }
    return await base.SaveChangesAsync(acceptAllChangesOnSuccess, ct);
}
```

### Dapper Implementation

```csharp
public class OrderRepository
{
    private readonly IDbConnectionFactory _connectionFactory;
    private readonly ITenantService _tenantService;

    public OrderRepository(IDbConnectionFactory connectionFactory,
        ITenantService tenantService)
    {
        _connectionFactory = connectionFactory;
        _tenantService = tenantService;
    }

    // Every Dapper query must include TenantId
    public async Task<IReadOnlyList<Order>> GetOrdersAsync(
        DateTime cutoff, CancellationToken ct = default)
    {
        const string sql = @"
            SELECT OrderId, CustomerId, OrderTotal, OrderStatus, OrderDate
            FROM Orders
            WHERE TenantId = @TenantId
              AND OrderDate >= @Cutoff
            ORDER BY OrderDate DESC";

        await using var connection = _connectionFactory.Create();
        var results = await connection.QueryAsync<Order>(
            new CommandDefinition(sql, new
            {
                TenantId = _tenantService.CurrentTenantId,
                Cutoff = cutoff
            }, cancellationToken: ct));
        return results.AsList();
    }

    // INSERT with TenantId
    public async Task<int> CreateOrderAsync(
        Order order, CancellationToken ct = default)
    {
        const string sql = @"
            INSERT INTO Orders (TenantId, CustomerId, OrderTotal, OrderStatus, CreatedBy)
            OUTPUT INSERTED.OrderId
            VALUES (@TenantId, @CustomerId, @OrderTotal, @OrderStatus, @CreatedBy)";

        order.TenantId = _tenantService.CurrentTenantId;
        await using var connection = _connectionFactory.Create();
        return await connection.ExecuteScalarAsync<int>(
            new CommandDefinition(sql, order, cancellationToken: ct));
    }
}
```

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL: Row-Level Security (RLS) for shared schema
CREATE POLICY tenant_isolation ON Orders
    USING (TenantId = current_setting('app.tenant_id')::INT);

ALTER TABLE Orders ENABLE ROW LEVEL SECURITY;

-- PostgreSQL: schema-per-tenant
CREATE SCHEMA IF NOT EXISTS tenant_42;
CREATE TABLE tenant_42.Orders ( ... );
```

PostgreSQL's RLS is more mature than SQL Server's security predicate — it is built into the query pipeline and applies automatically to all queries including DML. SQL Server's security policy requires a scalar function that is evaluated for every row.

## Gotchas and Production Pitfalls

### 1. Missing TenantId in query — data leak

**Pitfall:** A developer forgets `WHERE TenantId = @tid` in a query.

```csharp
// ❌ Missing TenantId — returns data from ALL tenants
var order = await dbContext.Orders
    .FirstOrDefaultAsync(o => o.OrderId == 5, ct);
```

**Symptom:** EF Core's global query filter automatically adds `WHERE TenantId = @tid` — so this particular case is safe with EF Core. But raw SQL, Dapper, or ad-hoc queries without the filter leak data across tenants.

**Fix:** Enforce tenant isolation at every level:
- EF Core: global query filters
- Dapper: template or base repository that injects `TenantId`
- SQL Server: row-level security policy as a safety net
- Testing: automated tests that verify cross-tenant data isolation

**Cost of not fixing:** Data breach. Tenant A sees Tenant B's orders. PCI/HIPAA/SOC2 violation.

### 2. TenantId not in clustered PK

**Pitfall:** The PK is `(OrderId IDENTITY)` and `TenantId` is a non-clustered index.

```sql
CREATE TABLE Orders (
    OrderId    INT IDENTITY PRIMARY KEY CLUSTERED,
    TenantId   INT NOT NULL
);
CREATE INDEX IX_Orders_TenantId ON Orders(TenantId);
```

**Symptom:** A per-tenant query `WHERE TenantId = 42` performs a non-clustered index seek on `IX_Orders_TenantId` followed by a key lookup for each row — 10K key lookups for a 10K-row tenant. The non-clustered index is large because every row stores the 4-byte clustering key.

**Fix:** Make `TenantId` the leading column of the clustered PK: `PRIMARY KEY (TenantId, OrderId)`.

**Cost of not fixing:** Every per-tenant query performs a key lookup per row. At 10,000 rows per tenant, that is 10,000 extra logical reads.

### 3. Tenant IDENTITY namespace is shared across tenants

**Pitfall:** `OrderId` is `INT IDENTITY(1,1)` on a shared-schema table. All tenants share the same identity sequence.

**Symptom:** `OrderId` values are globally sequential, not per-tenant sequential. Tenant A's first order might be `OrderId = 1000`. Tenant B's first order is `OrderId = 1001`. Business users ask why order numbers are not sequential per tenant.

**Fix:** Accept it — order numbers are not meaningful across tenants. Or use `(TenantId, TenantOrderId)` where `TenantOrderId` is a sequential number per tenant (managed by application logic or a `SEQUENCE` per tenant).

```sql
CREATE SEQUENCE Seq_Orders_42 START WITH 1 INCREMENT BY 1;
CREATE SEQUENCE Seq_Orders_99 START WITH 1 INCREMENT BY 1;
-- (Requires managing a sequence per tenant)
```

**Cost of not fixing:** None technically — the `OrderId` is still unique and functional. Cosmetic issue for per-tenant reporting.

### 4. Schema-per-tenant migration failures at scale

**Pitfall:** The team chooses separate schemas for 5,000 tenants. A migration adds a column:

```sql
ALTER TABLE tenant_1.Orders ADD DiscountPercent DECIMAL(5,2) NULL;
ALTER TABLE tenant_2.Orders ADD DiscountPercent DECIMAL(5,2) NULL;
-- ... 4,998 more ALTER TABLE statements
```

**Symptom:** The 3,201st `ALTER TABLE` takes a schema modification lock that blocks queries on that tenant. If the operation fails mid-way, 3,200 schemas have the column and 1,800 do not. Rolling back is complex.

**Fix:** Use a script to generate migration SQL dynamically, execute with error handling and checkpointing, and verify each schema after migration. Consider shared schema for tenants < 10K.

**Cost of not fixing:** Partially migrated state. Application code must handle both schemas during the migration window.

### 5. Shared schema hot tenant contention

**Pitfall:** A single tenant with 100M rows (a "heavy tenant") causes page splits and index maintenance that affects other tenants' queries.

**Symptom:** The heavy tenant's nightly index rebuild takes 2 hours and generates 95% fragmentation during the day on that tenant's page range. Other tenants' queries slow down because the heavy tenant's pages compete for buffer pool space.

**Fix:** Isolate heavy tenants to separate databases. Monitor per-tenant page reads and data size. Define a threshold (e.g., 5% of total data) above which a tenant is moved to its own database.

**Cost of not fixing:** All tenants experience performance degradation because of one heavy tenant. Support tickets from small tenants.

### 6. Row-level security impact on query performance

**Pitfall:** SQL Server row-level security adds a scalar function invocation per row.

**Symptom:** A query that normally runs in 100ms takes 500ms with RLS enabled because the security predicate function is evaluated for every row in the result set.

**Fix:** Benchmark with and without RLS. For high-throughput tables, prefer application-level tenant filtering (EF Core query filter, Dapper WHERE clause) and use RLS only as a defense-in-depth layer.

**Cost of not fixing:** All queries are 5x slower with RLS. The application spends most of its time in the security predicate.

## Performance Implications

### Benchmark: Shared Schema vs Separate Schema vs Separate Database

```sql
SET STATISTICS IO ON;

-- Shared schema: clustered on (TenantId, OrderId)
-- Per-tenant query (TenantId = 42, 50K rows)
SELECT OrderId, OrderTotal FROM Orders
WHERE TenantId = 42 AND OrderId BETWEEN 1 AND 1000;
-- Table 'Orders'. Scan count 0, logical reads 4 (seek to tenant range)

-- Separate schema: tenant_42.Orders
-- Same query (no TenantId needed)
SELECT OrderId, OrderTotal FROM tenant_42.Orders
WHERE OrderId BETWEEN 1 AND 1000;
-- Table 'tenant_42.Orders'. Scan count 0, logical reads 3 (seek on PK)
```

The difference is 3 vs 4 logical reads — negligible. The shared schema adds one level of B-tree depth because the composite key `(TenantId, OrderId)` has one more leading column than `(OrderId)` alone.

### Spoken Answers

**Q: What are the three approaches to multi-tenant database schema, and what are their tradeoffs?**

> **Average answer:** Shared schema — same tables with TenantId column. Separate schema — each tenant has their own schema. Separate database — each tenant gets their own database. Shared is cheapest but least isolated.

> **Great answer:** The three approaches form a spectrum of isolation versus operational complexity. Shared database, shared schema — every tenant's data lives in the same tables with a `TenantId` column distinguishing them. The primary benefit is operational simplicity: one set of tables to manage, one migration to run, one index maintenance job. The clustered PK is `(TenantId, IDENTITY)` so each tenant's rows are stored in a compact B-tree range — querying a single tenant's data costs only ~4 logical reads regardless of the total tenant count. The risk is that a missing `WHERE TenantId` filter exposes cross-tenant data, and a single heavy tenant can degrade performance for everyone. Shared database, separate schema — each tenant gets its own schema (`tenant1.Orders`, `tenant2.Orders`). This provides name isolation: queries do not need a `TenantId` column because the schema itself scopes the data. The downside is schema migration: a migration that adds a column must be applied to every tenant's schema. At 5,000 tenants, this is a significant operational burden. Separate database — each tenant gets its own database, possibly on its own server. This is the maximum isolation: you can back up, tune, and scale each tenant independently. The cost is operational complexity: managing N databases, N connection pools, and N migration pipelines. The choice depends on the application's tenant count, isolation requirements, and operational capacity. For most SaaS applications under 1,000 tenants, shared schema is the pragmatic default. For enterprise SaaS with strict compliance requirements, separate databases are the standard. Separate schemas are rarely the best choice — they combine the worst of both: no physical isolation with high operational overhead.

**Q: Why should TenantId be the leading column in the clustered PK for a shared schema?**

> **Great answer:** The leading column of a clustered index determines the physical sort order and the B-tree's first-level seek. With `(TenantId, OrderId)` as the clustered PK, every tenant's rows are stored in a contiguous page range. A query `WHERE TenantId = @tid` becomes a clustered index seek to the start of that tenant's range, followed by a scan of only that tenant's pages. The B-tree depth is determined by the total row count, but the seek cost per tenant query is only the depth of the tree (approximately `log_200(totalRows)` levels) — the same as a single-column PK on `OrderId`. The alternative — `(OrderId)` as the PK with a non-clustered index on `TenantId` — would require a key lookup for every row returned, because the non-clustered index on `TenantId` does not include the business columns. For a tenant query returning 10,000 rows, that is 10,000 key lookups instead of a single range scan within the tenant's page range. The composite key also eliminates the need for a separate unique constraint on `(TenantId, OrderId)` — it is the PK itself. The one cost is that non-clustered indexes on this table store both `TenantId` and `OrderId` as the clustering key (8 bytes instead of 4), but this is negligible compared to the query performance gain.

|---|---|

### Scale Thresholds

- **Shared schema fine up to ~10K tenants** — each tenant's rows are in a compact B-tree range; per-tenant queries cost ~4 reads
- **Heavy tenant threshold at ~5% of total rows** — isolate to separate database to prevent performance degradation
- **Schema-per-tenant becomes painful at > 100 tenants** — migrations require looping over schemas
- **Separate database needed for compliance** — HIPAA, PCI, SOC2 often require physical isolation

**Challenge 1 — Write the SQL**

Design a shared-schema `Invoices` table for a multi-tenant SaaS billing system. The most common query is "get the last 30 days of invoices for the current tenant, sorted by invoice date descending." Write the table DDL, the index, and the query.

<details>
<summary>Solution</summary>

```sql
CREATE TABLE Invoices (
    TenantId      INT NOT NULL,
    InvoiceId     INT IDENTITY(1,1) NOT NULL,
    CustomerId    INT NOT NULL,
    InvoiceNumber VARCHAR(50) NOT NULL,
    TotalAmount   DECIMAL(19,4) NOT NULL,
    InvoiceDate   DATE NOT NULL,
    DueDate       DATE NOT NULL,
    PaidAt        DATETIME2 NULL,
    CreatedAt     DATETIME2 NOT NULL DEFAULT SYSUTCDATETIME(),
    CONSTRAINT PK_Invoices PRIMARY KEY CLUSTERED (TenantId, InvoiceId),
    CONSTRAINT FK_Invoices_Tenants FOREIGN KEY (TenantId) REFERENCES Tenants(TenantId)
);

-- Covering index for the common query pattern
CREATE INDEX IX_Invoices_TenantId_InvoiceDate
ON Invoices(TenantId, InvoiceDate DESC)
INCLUDE (CustomerId, InvoiceNumber, TotalAmount, DueDate, PaidAt);

-- Query (TenantId is injected by application)
SELECT InvoiceId, InvoiceNumber, TotalAmount, InvoiceDate, DueDate, PaidAt
FROM Invoices
WHERE TenantId = @TenantId
  AND InvoiceDate >= DATEADD(DAY, -30, GETUTCDATE())
ORDER BY InvoiceDate DESC;

-- Execution plan:
-- Index Seek (backward) — IX_Invoices_TenantId_InvoiceDate
--   Seek Keys: TenantId = @tid, InvoiceDate >= @cutoff
--   Ordered: true (descending)
--   No key lookups (covering index)
```

**Challenge 2 — Fix the performance problem**

```sql
-- A multi-tenant shared-schema Orders table has 50M rows across 500 tenants.
-- This query runs for 15 seconds:
SELECT OrderId, CustomerId, OrderTotal, OrderDate
FROM Orders
WHERE TenantId = 42
ORDER BY OrderDate DESC;
-- SET STATISTICS IO: logical reads = 225,000
```

The PK is `(OrderId INT IDENTITY PRIMARY KEY CLUSTERED)`. `TenantId` has a non-clustered index. Identify why and fix it.

<details> <summary>Solution</summary>

**Root cause:** `TenantId` is not the leading column of the clustered PK. The clustered PK is `(OrderId)`, so `WHERE TenantId = 42` uses the non-clustered index on `TenantId`. For 50K rows in tenant 42, this means:
1. 50K non-clustered index seeks to find the rows
2. 50K key lookups on the clustered PK to retrieve `CustomerId`, `OrderTotal`, `OrderDate`
3. A Sort operator to order by `OrderDate DESC`

Total: ~150K logical reads instead of ~50.

**Fix:** Change the clustered PK to `(TenantId, OrderId)`:

```sql
ALTER TABLE Orders DROP CONSTRAINT PK_Orders;
ALTER TABLE Orders ADD CONSTRAINT PK_Orders PRIMARY KEY CLUSTERED (TenantId, OrderId);
DROP INDEX IX_Orders_TenantId;  -- no longer needed (TenantId is leading in PK)
```

**After fix — logical reads:** ~50 (from 225,000). Execution time: < 50ms.

</details>

**Challenge 3 — Explain the execution plan**

```sql
SELECT o.OrderId, o.OrderTotal, c.CustomerName
FROM Orders o
INNER JOIN Customers c ON o.TenantId = c.TenantId
    AND o.CustomerId = c.CustomerId
WHERE o.TenantId = 42
  AND o.OrderDate >= '2026-06-01';
```

The PK is `(TenantId, OrderId)` on Orders and `(TenantId, CustomerId)` on Customers. Explain the execution plan.

<details> <summary>Solution</summary>

**Execution plan:**

```
Clustered Index Seek — PK_Orders (TenantId = 42, OrderDate >= '2026-06-01')
  |-- 45 logical reads (seek to tenant 42, range scan on OrderDate range)
  |-- 500 rows

Nested Loops (Inner Join)
  |-- Outer: Clustered Index Seek results (500 rows)
  |-- Inner: Clustered Index Seek — PK_Customers (TenantId = 42, CustomerId = @cid)
  |   |-- 3 logical reads per execution
  |   |-- 500 executions × 3 reads = 1500 total

Total logical reads: ~1545
```

**Why two seeks on TenantId:** Both tables use `(TenantId, ...)` as the leading columns. The join predicate includes both `TenantId` and `CustomerId` for each side. The optimizer seeks to `TenantId = 42` on Customers for each outer row — only 3 logical reads per seek because the TenantId is part of the seek key.

**Why Nested Loops:** The outer input (500 orders) is small. Each probe into Customers seeks on the composite key `(TenantId, CustomerId)` — optimized by the leading TenantId.

**What would be different without TenantId in both PKs:** If the join predicate was just `ON o.CustomerId = c.CustomerId` without `TenantId`, the Customers table would need a full scan to find matching rows — the PK seek on `(TenantId, CustomerId)` cannot be used because TenantId is not specified in the join.

</details>

**Challenge 4 — Diagnose the concurrency problem**

A shared-schema multi-tenant system has a heavy tenant with 80M rows (80% of total data). During the heavy tenant's nightly batch job (updating order statuses), all other tenants experience query timeouts. `sys.dm_os_wait_stats` shows high `PAGEIOLATCH_SH` and `LCK_M_S` waits. What is happening and how do you fix it?

<details> <summary>Solution</summary>

**Root cause:** The heavy tenant's batch job scans and updates 10M rows. This generates:
1. High I/O — the batch reads the heavy tenant's pages into the buffer pool, evicting other tenants' pages (`PAGEIOLATCH_SH` waits on other tenants' queries).
2. Lock escalation — the batch crosses the 5,000-lock threshold, escalating row locks to a table lock (`LCK_M_S` waits as other tenants' queries block on the table lock).

**Detection query:**

```sql
SELECT wait_type, wait_time_ms, waiting_tasks_count
FROM sys.dm_os_wait_stats
WHERE wait_type IN ('PAGEIOLATCH_SH', 'LCK_M_S', 'LCK_M_U', 'LCK_M_X')
ORDER BY wait_time_ms DESC;
```

**Fix:** Isolate the heavy tenant:

1. Move the heavy tenant to a separate database (short-term: copy data, redirect traffic).
2. Or batch the heavy tenant's updates in smaller chunks.
3. Or use `LOCK_ESCALATION = AUTO` with a threshold higher than 5,000 (not recommended — can cause deadlocks).

**In .NET:** Use batched updates:

```csharp
int offset = 0, batchSize = 1000;
int affected;
do
{
    affected = await dbContext.Orders
        .Where(o => o.TenantId == heavyTenantId
            && o.OrderStatus == "Pending"
            && o.OrderDate < cutoff)
        .Take(batchSize)
        .ExecuteUpdateAsync(setters => setters
            .SetProperty(o => o.OrderStatus, "Archived")
            .SetProperty(o => o.ModifiedAt, DateTime.UtcNow));
    offset += batchSize;
} while (affected > 0);
```

**Long-term fix:** Implement a tenant data size threshold. When a tenant exceeds 5% of total data, provision a separate database and migrate the tenant automatically.

</details>

**Challenge 5 — Design the index**

A multi-tenant SaaS application has 200 tenants, each with 500K–5M rows in the `Orders` table (total 500M rows). The most common query patterns are:
1. Get orders for the current tenant by date range (95% of queries).
2. Get a single order by OrderId for the current tenant (4% of queries).
3. Cross-tenant admin report: total orders per day across all tenants (1% of queries).

Design the PK and index strategy.

<details> <summary>Solution</summary>

```sql
CREATE TABLE Orders (
    TenantId     INT NOT NULL,
    OrderId      INT IDENTITY(1,1) NOT NULL,
    CustomerId   INT NOT NULL,
    OrderTotal   DECIMAL(19,4) NOT NULL,
    OrderDate    DATETIME2 NOT NULL DEFAULT SYSUTCDATETIME(),
    OrderStatus  VARCHAR(50) NOT NULL DEFAULT 'Pending',
    CONSTRAINT PK_Orders PRIMARY KEY CLUSTERED (TenantId, OrderId)
);

-- Pattern 1: per-tenant date range queries (95%)
-- The clustered PK on (TenantId, OrderId) supports seek on TenantId,
-- but the query filters by OrderDate, not OrderId.
-- Add a covering index on (TenantId, OrderDate DESC):
CREATE INDEX IX_Orders_TenantId_OrderDate
ON Orders(TenantId, OrderDate DESC)
INCLUDE (CustomerId, OrderTotal, OrderStatus);

-- Pattern 2: single order by OrderId (4%)
-- Without an index on (OrderId), this would scan all tenants.
CREATE INDEX IX_Orders_OrderId
ON Orders(OrderId)
INCLUDE (TenantId, CustomerId, OrderTotal);

-- Pattern 3: cross-tenant admin report (1%)
-- Uses a filtered index idea, but admin queries can scan the clustered index
-- since they run infrequently and can tolerate a full scan.
-- Alternatively, maintain a separate aggregated reporting table.
```

**Why this works:**
- The PK on `(TenantId, OrderId)` provides clustered storage per tenant — each tenant's rows in a contiguous page range
- `IX_Orders_TenantId_OrderDate` covers 95% of queries: seek on `TenantId`, range scan on `OrderDate DESC`, covering all returned columns
- `IX_Orders_OrderId` supports single-order lookup by `OrderId` (needed for order detail pages). Since `OrderId` is globally unique (IDENTITY), this index is small (500M entries × key width ≈ 4GB) and fast (3–4 reads per seek)

**Tradeoffs:** `IX_Orders_TenantId_OrderDate` is large (500M rows × key width ≈ 24 GB). Consider making it a filtered index if many tenants are small — but with 200 tenants averaging 2.5M rows each, the index is justified.

</details>
