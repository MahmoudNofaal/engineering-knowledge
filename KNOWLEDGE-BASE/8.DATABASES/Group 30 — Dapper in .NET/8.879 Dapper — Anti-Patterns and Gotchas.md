# Dapper — Anti-Patterns and Gotchas

## Overview

Dapper is a fast, lightweight micro-ORM that sits directly atop ADO.NET. Its simplicity is its greatest strength, but that same simplicity makes it easy to fall into patterns that silently degrade performance, kill scalability, or introduce hard-to-find bugs. This note catalogs the most common anti-patterns and gotchas encountered when using Dapper in production .NET applications.

Each anti-pattern is presented with a consistent structure:

- **Name** — what the community calls it
- **Bad Code** — the problematic snippet
- **Symptom** — what you observe at runtime or in production
- **Fix** — the corrected approach
- **Cost** — the impact of leaving it unfixed (maintainability, performance, correctness, or security)

## Anti-Pattern 1 — Not Disposing the `IDbConnection`

### ❌ Bad Code

```csharp
public IEnumerable<Order> GetOrders(int customerId)
{
    var connection = new SqlConnection(_connectionString);
    return connection.Query<Order>(
        "SELECT * FROM Orders WHERE CustomerId = @Id",
        new { Id = customerId });
}
```

The `SqlConnection` implements `IDisposable`, but this code never calls `Dispose()` or `Close()`. The connection is not wrapped in a `using` block.

### Symptom

- **Connection pool exhaustion** under load.
- `System.InvalidOperationException: Timeout expired. The timeout period elapsed prior to obtaining a connection from the pool. This may have occurred because all pooled connections were in use and max pool size was reached.`
- Connections stay `Sleeping` in SQL Server `sys.dm_exec_sessions` until GC finalizes them (non-deterministic).
- Under steady load, the application becomes completely unavailable after the pool hits `Max Pool Size` (default = 100).

### ✅ Fix

```csharp
public IEnumerable<Order> GetOrders(int customerId)
{
    using var connection = new SqlConnection(_connectionString);
    return connection.Query<Order>(
        "SELECT * FROM Orders WHERE CustomerId = @Id",
        new { Id = customerId });
}
```

Or the classic block form:

```csharp
public IEnumerable<Order> GetOrders(int customerId)
{
    using (var connection = new SqlConnection(_connectionString))
    {
        return connection.Query<Order>(
            "SELECT * FROM Orders WHERE CustomerId = @Id",
            new { Id = customerId });
    }
}
```

### Cost

**Critical** — Production outages under any moderate load. Connection pool exhaustion is one of the most common Dapper-related production incidents. Each leaked connection occupies a SQL Server worker thread and a pool slot until GC intervenes (which may take minutes or never happen under memory pressure).

## Anti-Pattern 2 — Sync-over-Async (`.Result` / `.Wait()`)

### ❌ Bad Code

```csharp
public IEnumerable<Order> GetOrdersSync(int customerId)
{
    using var connection = new SqlConnection(_connectionString);
    var task = connection.QueryAsync<Order>(
        "SELECT * FROM Orders WHERE CustomerId = @Id",
        new { Id = customerId });
    return task.Result; // BAD
}
```

```csharp
public IEnumerable<Order> GetOrdersSync(int customerId)
{
    using var connection = new SqlConnection(_connectionString);
    return connection.QueryAsync<Order>(
        "SELECT * FROM Orders WHERE CustomerId = @Id",
        new { Id = customerId }).Result; // BAD
}
```

```csharp
public IEnumerable<Order> GetOrdersSync(int customerId)
{
    using var connection = new SqlConnection(_connectionString);
    connection.QueryAsync<Order>(
        "SELECT * FROM Orders WHERE CustomerId = @Id",
        new { Id = customerId }).Wait(); // BAD
}
```

```csharp
// The worst variant in ASP.NET / UI contexts:
public IEnumerable<Order> GetOrdersSync(int customerId)
{
    using var connection = new SqlConnection(_connectionString);
    return Task.Run(async () =>
        await connection.QueryAsync<Order>(
            "SELECT * FROM Orders WHERE CustomerId = @Id",
            new { Id = customerId })).Result; // DOUBLY BAD
}
```

### Symptom

- **Deadlocks** — In ASP.NET Core (which has `SynchronizationContext`), `.Result` or `.Wait()` on an incomplete task causes the calling thread to block while the async continuation waits for that same context to be free. This is a classic deadlock.
- **Thread pool starvation** — Blocking a thread while an async operation is in-flight consumes a thread that could otherwise serve requests. Under load, the thread pool injects more threads, leading to high CPU context switching and degraded throughput.
- **Erratic performance** — The application works fine in dev (fast queries, task completes before blocking matters) but deadlocks or slows to a crawl under production load.
- `System.AggregateException` with an inner `TaskCanceledException` or deadlock hang.

### ✅ Fix

```csharp
// Go async all the way up:
public async Task<IEnumerable<Order>> GetOrdersAsync(int customerId)
{
    using var connection = new SqlConnection(_connectionString);
    return await connection.QueryAsync<Order>(
        "SELECT * FROM Orders WHERE CustomerId = @Id",
        new { Id = customerId });
}
```

Or use Dapper's synchronous API properly (don't call the async variant):

```csharp
public IEnumerable<Order> GetOrders(int customerId)
{
    using var connection = new SqlConnection(_connectionString);
    return connection.Query<Order>(
        "SELECT * FROM Orders WHERE CustomerId = @Id",
        new { Id = customerId });
}
```

### Gotcha: The "Just This Once" Trap

Even a single call to `.Result` in a callstack that's otherwise async can cause deadlocks. The golden rule: **once async, always async**.

### Cost

**Critical** — Production outages, app hangs, and thread pool starvation. This is arguably the deadliest anti-pattern in ASP.NET applications using Dapper.

## Anti-Pattern 3 — `SELECT *` Returning More Columns Than Needed

### ❌ Bad Code

```csharp
public record OrderSummary(int Id, string CustomerName, decimal Total);

public IEnumerable<OrderSummary> GetOrderSummaries()
{
    using var connection = new SqlConnection(_connectionString);
    return connection.Query<OrderSummary>(
        "SELECT * FROM Orders"); // Returns 25 columns, needs 3
}
```

### Symptom

- **Excess memory pressure** — Every row materializes 25 column values over the network when only 3 are needed. For large result sets (thousands of rows), this multiplies memory allocation on both SQL Server and the application.
- **DataReader overhead** — Dapper's `SqlMapper` must iterate over every column in the result set to map them to the target type. Unneeded columns are still read and discarded, adding CPU cycles per row.
- **Network bandwidth** — More data crosses the wire, increasing latency for every query.
- **Brittle code** — Adding a column to the `Orders` table changes the query result shape. If the new column has a name that conflicts with `OrderSummary` properties (or introduces a `DBNull` for a non-nullable property), the query breaks at runtime.
- **Index inefficiency** — `SELECT *` prevents covering-index-only plans. SQL Server may need a clustered index scan (or key lookup) when a non-covering index would have sufficed for the few columns actually needed.

### Table: Memory Waste by Column Count

| Columns in Table | Columns Needed | Waste per Row (~100 byte avg) | 10k Rows | 100k Rows |
|:-:|:-:|:-:|:-:|:-:|
| 5 | 3 | 40% | ~2 MB | ~20 MB |
| 25 | 3 | 88% | ~22 MB | ~220 MB |
| 50 | 3 | 94% | ~47 MB | ~470 MB |

### ✅ Fix

```csharp
public IEnumerable<OrderSummary> GetOrderSummaries()
{
    using var connection = new SqlConnection(_connectionString);
    return connection.Query<OrderSummary>(
        "SELECT Id, CustomerName, Total FROM Orders");
}
```

### Cost

**Medium to High** — Performance degrades proportionally to row count and column bloat. In high-throughput OLTP systems, the CPU and memory waste from discarded columns can be significant.

## Anti-Pattern 4 — Missing `CancellationToken` on `CommandDefinition`

### ❌ Bad Code

```csharp
public async Task<IEnumerable<Order>> GetOrdersAsync(int customerId)
{
    using var connection = new SqlConnection(_connectionString);
    return await connection.QueryAsync<Order>(
        "SELECT * FROM Orders WHERE CustomerId = @Id",
        new { Id = customerId });
    // No CancellationToken — cannot be cancelled on shutdown
}
```

### Symptom

- **Application hangs during graceful shutdown** — When the host triggers `IHostApplicationLifetime.StopAsync()`, in-flight database queries continue running. The application waits for them to complete before shutting down.
- **Orphaned queries in SQL Server** — A long-running query continues executing on the database server even though the application is shutting down.
- **Resource leak during rollout** — During blue/green deployments, old instances stay alive waiting for SQL queries to finish, delaying the rollout.

### ✅ Fix

```csharp
public async Task<IEnumerable<Order>> GetOrdersAsync(
    int customerId,
    CancellationToken cancellationToken = default)
{
    using var connection = new SqlConnection(_connectionString);
    var command = new CommandDefinition(
        "SELECT * FROM Orders WHERE CustomerId = @Id",
        new { Id = customerId },
        cancellationToken: cancellationToken);
    return await connection.QueryAsync<Order>(command);
}
```

Or use the overload that accepts `CancellationToken` directly:

```csharp
public async Task<IEnumerable<Order>> GetOrdersAsync(
    int customerId,
    CancellationToken cancellationToken = default)
{
    using var connection = new SqlConnection(_connectionString);
    return await connection.QueryAsync<Order>(
        "SELECT * FROM Orders WHERE CustomerId = @Id",
        new { Id = customerId },
        cancellationToken: cancellationToken);
}
```

### Cancellation Chain

```csharp
// In the controller / endpoint:
[HttpGet("orders/{customerId}")]
public async Task<IActionResult> GetOrders(
    int customerId,
    CancellationToken cancellationToken)
{
    // ASP.NET Core passes the RequestAborted token automatically
    var orders = await _orderService.GetOrdersAsync(customerId, cancellationToken);
    return Ok(orders);
}
```

### Cost

**Medium** — Causes deployment delays and resource waste during shutdown. In systems with long-running queries (reporting, ETL, batch operations), this can add significant downtime during rollouts.

## Anti-Pattern 5 — Parameter Name Mismatch

### ❌ Bad Code

```csharp
public IEnumerable<Order> GetOrders(int customerId)
{
    using var connection = new SqlConnection(_connectionString);
    return connection.Query<Order>(
        "SELECT * FROM Orders WHERE CustomerId = @CustomerId",
        new { Id = customerId }); // Parameter name "Id" does not match "@CustomerId"
}
```

### Common Variants

```csharp
// Typo in the SQL parameter:
connection.Query<Order>(
    "SELECT * FROM Orders WHERE CustomerId = @CustmerId", // typo
    new { CustomerId = id });

// Case mismatch (SQL Server is case-insensitive for params, PostgreSQL is NOT):
// Works on SQL Server, fails silently or with exception on PostgreSQL
connection.Query<Order>(
    "SELECT * FROM Orders WHERE CustomerId = @customerid",
    new { CustomerId = id });

// Extra parameters not used in SQL — Dapper ignores them silently
connection.Query<Order>(
    "SELECT * FROM Orders WHERE CustomerId = @Id",
    new { Id = id, Unused = "blah" }); // No error, but confusing

// Missing parameter entirely:
connection.Query<Order>(
    "SELECT * FROM Orders WHERE CustomerId = @Id AND Status = @Status",
    new { Id = id }); // @Status is never provided
```

### Symptom

- **No compile-time error** — Dapper uses dynamic `ExpandoObject` or anonymous types, so there is zero compile-time checking of parameter names.
- **Runtime exception** — `System.Data.SqlClient.SqlException: Must declare the scalar variable "@CustomerId".`
- **Wrong data returned** — If the parameter is silently ignored (e.g., the SQL doesn't use it, or the name matches a different parameter), the query executes with unexpected bindings.
- **Case-sensitivity surprises** — The code works in local dev against SQL Server (case-insensitive) but breaks when deployed against PostgreSQL or Azure SQL with a case-sensitive collation.

### ✅ Fix

```csharp
// Match parameter names exactly:
public IEnumerable<Order> GetOrders(int customerId)
{
    using var connection = new SqlConnection(_connectionString);
    return connection.Query<Order>(
        "SELECT * FROM Orders WHERE CustomerId = @CustomerId",
        new { CustomerId = customerId }); // ✅ Name matches
}

// Or alias the SQL parameter to match the C# parameter name:
return connection.Query<Order>(
    "SELECT * FROM Orders WHERE CustomerId = @customerId",
    new { customerId });
```

### Prevention Pattern

```csharp
// Use a named record/builder to avoid anonymous type drift:
public record OrderQueryParams(int CustomerId, string? Status = null);

public IEnumerable<Order> GetOrders(OrderQueryParams p)
{
    using var connection = new SqlConnection(_connectionString);
    return connection.Query<Order>(
        "SELECT * FROM Orders WHERE CustomerId = @CustomerId " +
        "AND (@Status IS NULL OR Status = @Status)",
        p);
}
```

### Cost

**High** — Runtime failures that often bypass testing because they depend on exact parameter binding. PostgreSQL users suffer this more than SQL Server users due to case sensitivity.

## Anti-Pattern 6 — Storing Connection Strings in Code

### ❌ Bad Code

```csharp
public class OrderRepository
{
    private const string ConnectionString =
        "Server=tcp:prod-db.database.windows.net,1433;" +
        "Database=Orders;" +
        "User Id=sqladmin;" +
        "Password=P@ssw0rd123!"; // HARDCODED CREDENTIALS

    public IEnumerable<Order> GetOrders(int customerId)
    {
        using var connection = new SqlConnection(ConnectionString);
        return connection.Query<Order>("...", new { customerId });
    }
}
```

### Symptom

- **Security breach** — Connection strings in source code are exposed in: Git history, compiled assemblies (decompilable with ILSpy/dnSpy), CI/CD logs, error messages that dump stack traces with embedded strings, and developer screen shares.
- **No rotation** — Changing the database password requires a code change, recompile, and redeploy.
- **Environment inflexibility** — Different connection strings for dev, staging, and production cannot be managed independently without conditional compilations or manual edits.

### ✅ Fix

```csharp
// appsettings.json:
// {
//   "ConnectionStrings": {
//     "OrdersDb": "Server=tcp:prod-db.database.windows.net,1433;..."
//   }
// }

public class OrderRepository
{
    private readonly string _connectionString;

    public OrderRepository(IConfiguration configuration)
    {
        _connectionString = configuration.GetConnectionString("OrdersDb")
            ?? throw new InvalidOperationException("OrdersDb connection string is required");
    }

    public IEnumerable<Order> GetOrders(int customerId)
    {
        using var connection = new SqlConnection(_connectionString);
        return connection.Query<Order>("...", new { customerId });
    }
}
```

For secrets (passwords, tokens), use **Azure Key Vault**, **AWS Secrets Manager**, **User Secrets** (dev), or environment variables:

```csharp
// Using User Secrets (development):
// dotnet user-secrets set "ConnectionStrings:OrdersDb:Password" "..."

// Using environment variables (production):
// $env:CONNECTIONSTRINGS__ORDERSDB="Server=...;..."
```

### Cost

**Critical** — Security vulnerability. Hardcoded credentials in source code are a PCI-DSS, SOC2, and HIPAA violation. They are one of the most common findings in security audits.

## Anti-Pattern 7 — Using Dapper for CRUD That EF Core Would Handle Better

### ❌ Bad Code

```csharp
// A repository that manually writes every CRUD operation for a simple entity:
public class ProductRepository
{
    private readonly string _cs;
    public ProductRepository(string cs) => _cs = cs;

    public Product GetById(int id)
    {
        using var conn = new SqlConnection(_cs);
        return conn.QueryFirstOrDefault<Product>(
            "SELECT * FROM Products WHERE Id = @Id", new { id });
    }

    public IEnumerable<Product> GetAll()
    {
        using var conn = new SqlConnection(_cs);
        return conn.Query<Product>("SELECT * FROM Products");
    }

    public void Insert(Product p)
    {
        using var conn = new SqlConnection(_cs);
        conn.Execute(
            "INSERT INTO Products (Name, Price, CategoryId) VALUES (@Name, @Price, @CategoryId)",
            p);
    }

    public void Update(Product p)
    {
        using var conn = new SqlConnection(_cs);
        conn.Execute(
            "UPDATE Products SET Name = @Name, Price = @Price, CategoryId = @CategoryId WHERE Id = @Id",
            p);
    }

    public void Delete(int id)
    {
        using var conn = new SqlConnection(_cs);
        conn.Execute("DELETE FROM Products WHERE Id = @Id", new { id });
    }
}
```

### Symptom

- **Massive boilerplate** — Every entity requires 30+ lines of handwritten SQL and mapping. A typical 50-entity application means 1500 lines of tedious, error-prone code.
- **No change tracking** — You must manually track which properties changed. No unit-of-work pattern.
- **No navigation property loading** — Loading related entities (e.g., `Product.Category`) requires hand-written JOINs or N+1-prone lazy calls.
- **No migrations** — Schema changes are managed separately (if at all). No `dotnet ef migrations add`.
- **Developer overhead** — Every new entity needs a full repository, tests, and SQL review.

### ✅ Fix — Use EF Core for Standard CRUD

```csharp
// EF Core DbContext handles all of the above automatically:
public class AppDbContext : DbContext
{
    public DbSet<Product> Products => Set<Product>();

    protected override void OnConfiguring(DbContextOptionsBuilder options)
        => options.UseSqlServer(_connectionString);
}

// Usage:
using var db = new AppDbContext();
var product = await db.Products
    .Include(p => p.Category)
    .FirstAsync(p => p.Id == id);
```

### Decision Framework

| Scenario | Tool |
|---|---|
| Simple CRUD, no complex queries | EF Core |
| Complex reporting, read-only queries | Dapper |
| High-throughput writes (bulk insert) | Dapper + SqlBulkCopy |
| Mixed workload (CRUD + reports) | EF Core for writes, Dapper for reads |
| Prototype / small app | Either (EF Core is faster to build) |

### Cost

**Medium** — Wasted developer time and increased defect rate. For every hour spent writing Dapper CRUD, a developer could have written the same functionality in EF Core in 10 minutes. Over a project lifetime, this adds weeks of unnecessary effort.

## Anti-Pattern 8 — Not Using `buffered = false` for Large Result Sets

### ❌ Bad Code

```csharp
public IEnumerable<Order> GetAllOrders()
{
    using var connection = new SqlConnection(_connectionString);
    // Default buffered=true — loads ALL rows into memory at once
    return connection.Query<Order>("SELECT * FROM Orders");
}
```

### Symptom

- **OutOfMemoryException** — When the result set is large (millions of rows), Dapper buffers the entire result set into a `List<T>` in memory. A 10-million-row query with 500 bytes per row = 5 GB allocation.
- **High GC pressure** — Even moderately large result sets (100k+ rows) cause frequent Gen-2 collections, stalling all threads.
- **Slow first-row latency** — Dapper must read ALL rows into memory before returning the first one to the caller, increasing time-to-first-result.
- **IIS/Azure App Service recycling** — Memory thresholds trigger process recycling, causing request failures.
- **SQL Server memory grant** — Large buffered reads often require large memory grants on SQL Server, reducing concurrency.

### ✅ Fix

```csharp
// Use buffered: false to stream rows one at a time:
public IEnumerable<Order> GetAllOrders()
{
    using var connection = new SqlConnection(_connectionString);
    return connection.Query<Order>(
        "SELECT * FROM Orders",
        buffered: false); // Streams results
}
```

### Important: Lifetime Management

When using `buffered: false`, Dapper returns an iterator (`IEnumerable<T>`) that holds the `SqlDataReader` open. The connection must remain open until the enumeration is complete:

```csharp
// CORRECT — connection stays open while iterating:
public IEnumerable<Order> GetAllOrders()
{
    var connection = new SqlConnection(_connectionString);
    connection.Open();
    return connection.Query<Order>(
        "SELECT * FROM Orders",
        buffered: false);
    // Caller must Dispose the enumerator to close the reader
}
```

### Streaming vs Buffering Comparison

| Aspect | Buffered (default) | Unbuffered (streaming) |
|---|---|---|
| Memory per 100k rows | ~50 MB | ~1 KB (per row) |
| First-row latency | Slow (waits for all data) | Instant (as data arrives) |
| Connection lifetime | Released after buffering | Held until enumeration completes |
| Multiple active result sets (MARS) | Not needed | May need MARS=true in connection string |
| LINQ composition | Safe (materialized list) | Dangerous (reader still open) |

### Gotcha: LINQ After Streaming

```csharp
// DANGEROUS — enumerating the reader twice:
var orders = connection.Query<Order>("SELECT * FROM Orders", buffered: false);
var highValue = orders.Where(o => o.Total > 1000); // First enumeration
var count = orders.Count(); // Second enumeration — reader is closed! 💥
```

Materialize if you need multiple passes:

```csharp
var orders = connection.Query<Order>("SELECT * FROM Orders", buffered: false).ToList();
var highValue = orders.Where(o => o.Total > 1000);
var count = orders.Count();
```

### Cost

**High** — Applications processing large datasets are at risk of `OutOfMemoryException`, process recycling, or severe GC pauses. The fix is trivial (one parameter change) but the failure mode is catastrophic.

## Anti-Pattern 9 — Using a Transaction Without Passing It to Every Dapper Call

### ❌ Bad Code

```csharp
public void TransferFunds(int fromAccountId, int toAccountId, decimal amount)
{
    using var connection = new SqlConnection(_connectionString);
    connection.Open();
    using var transaction = connection.BeginTransaction();

    connection.Execute(
        "UPDATE Accounts SET Balance = Balance - @Amount WHERE Id = @Id",
        new { Amount = amount, Id = fromAccountId });
    // ⚠️ Transaction not passed — uses ambient transaction or none

    connection.Execute(
        "UPDATE Accounts SET Balance = Balance + @Amount WHERE Id = @Id",
        new { Amount = amount, Id = toAccountId });
    // ⚠️ Also not in the transaction

    transaction.Commit();
    // 😱 Both updates are auto-committed independently!
}
```

### Symptom

- **Partial updates** — If the first `Execute` succeeds but the second fails (or vice versa), the transaction is not rolled back. The database is left in an inconsistent state.
- **Transaction useless** — The `transaction.Commit()` commits nothing because no commands were enrolled in the transaction.
- **Ambient transaction confusion** — If `TransactionScope` is active elsewhere, Dapper might auto-enlist, masking the bug. The code works in some call paths but not others.
- **Difficult to debug** — Funds are transferred out of one account but never arrive in another. Data integrity is violated silently.

### ✅ Fix

```csharp
public void TransferFunds(int fromAccountId, int toAccountId, decimal amount)
{
    using var connection = new SqlConnection(_connectionString);
    connection.Open();
    using var transaction = connection.BeginTransaction();

    connection.Execute(
        "UPDATE Accounts SET Balance = Balance - @Amount WHERE Id = @Id",
        new { Amount = amount, Id = fromAccountId },
        transaction: transaction); // ✅

    connection.Execute(
        "UPDATE Accounts SET Balance = Balance + @Amount WHERE Id = @Id",
        new { Amount = amount, Id = toAccountId },
        transaction: transaction); // ✅

    transaction.Commit();
}
```

### Helper Pattern

```csharp
// Extension method to reduce repetition:
public static class DapperTransactionExtensions
{
    public static async Task<int> ExecuteAsync(
        this IDbConnection cnn,
        string sql,
        object? param,
        IDbTransaction tx)
    {
        return await cnn.ExecuteAsync(sql, param, transaction: tx);
    }
}
```

### Cost

**Critical** — Data corruption. Partial updates in finance, inventory, or any domain with invariants lead to unrecoverable data inconsistencies. This anti-pattern defeats the entire purpose of using transactions.

## Anti-Pattern 10 — Mixing Dapper and EF Core in the Same Transaction Without Sharing the Connection

### ❌ Bad Code

```csharp
public async Task PlaceOrder(int customerId, Order order)
{
    // Dapper connection (independent)
    using var dapperConn = new SqlConnection(_connectionString);
    await dapperConn.ExecuteAsync(
        "INSERT INTO Orders (CustomerId, Total) VALUES (@CustomerId, @Total)",
        order);

    // EF Core context (different connection — same database, different object)
    using var dbContext = new AppDbContext();
    dbContext.OrderItems.AddRange(order.Items);
    await dbContext.SaveChangesAsync(); // ⚠️ Different connection, no shared transaction!
}
```

### Symptom

- **No atomicity** — If the Dapper insert succeeds but EF Core's `SaveChangesAsync` fails (or vice versa), there is no rollback. Partial data is committed.
- **Distributed transaction escalation** — If both use the same connection string but different `SqlConnection` instances, wrapping them in a `TransactionScope` promotes to a MSDTC distributed transaction (slow, often blocked by firewall).
- **Mystery deadlocks** — Two separate connections to the same database can deadlock each other without a shared transaction scope.
- **Inconsistent reads** — The EF Core context may read stale data because the Dapper insert hasn't been committed yet from EF Core's isolation level perspective.

### ✅ Fix

```csharp
public async Task PlaceOrder(int customerId, Order order)
{
    // Share the same connection
    using var connection = new SqlConnection(_connectionString);
    await connection.OpenAsync();
    using var transaction = connection.BeginTransaction();

    try
    {
        // Dapper uses the shared connection + transaction
        await connection.ExecuteAsync(
            "INSERT INTO Orders (CustomerId, Total) VALUES (@CustomerId, @Total)",
            order,
            transaction: transaction);

        // EF Core uses the SAME connection and transaction
        using var dbContext = new AppDbContext(
            new DbContextOptionsBuilder<AppDbContext>()
                .UseSqlServer(connection) // Reuse the connection
                .Options);
        dbContext.Database.UseTransaction(transaction); // Reuse the transaction
        dbContext.OrderItems.AddRange(order.Items);
        await dbContext.SaveChangesAsync();

        transaction.Commit();
    }
    catch
    {
        transaction.Rollback();
        throw;
    }
}
```

### ✅ Alternative — TransactionScope

```csharp
public async Task PlaceOrder(int customerId, Order order)
{
    using var scope = new TransactionScope(TransactionScopeAsyncFlowOption.Enabled);
    using var dapperConn = new SqlConnection(_connectionString);
    using var dbContext = new AppDbContext();

    await dapperConn.ExecuteAsync(
        "INSERT INTO Orders (CustomerId, Total) VALUES (@CustomerId, @Total)",
        order);

    dbContext.OrderItems.AddRange(order.Items);
    await dbContext.SaveChangesAsync();

    scope.Complete(); // Enlists both in a single ambient transaction
}
```

Note: `TransactionScope` may promote to MSDTC if the two connections use different connection strings or different database instances.

### Cost

**High** — Data integrity violations. This anti-pattern is especially common in "we're migrating from Dapper to EF Core" codebases where both coexist temporarily.

## Anti-Pattern 11 — Not Handling `DBNull` in POCO Properties

### ❌ Bad Code

```csharp
public class Order
{
    public int Id { get; set; }
    public string CustomerName { get; set; }  // ❌ Non-nullable string
    public DateTime? ShippedDate { get; set; } // This one is fine
    public decimal Discount { get; set; }     // ❌ Non-nullable decimal — but DB allows NULL
}

// SQL: SELECT Id, CustomerName, ShippedDate, Discount FROM Orders
// Some rows have CustomerName = NULL and/or Discount = NULL
```

### Symptom

- `InvalidCastException: Specified cast is not valid` — Dapper tries to assign `DBNull` to a non-nullable `string` or `decimal`.
- `NullReferenceException` — If Dapper succeeds (e.g., with a nullable `int?` mapped correctly) but the non-nullable `Discount` gets a default `0` instead of signaling "no discount set."
- **Silent bad data** — With `int` (non-nullable), Dapper maps `DBNull` to `0` by default. You lose the semantic distinction between "discount = $0" and "no discount set."
- **Intermittent failures** — Queries work when all columns are non-null but fail when data contains nulls. Dev databases rarely have nulls; production databases always do.

### ✅ Fix

```csharp
public class Order
{
    public int Id { get; set; }
    public string? CustomerName { get; set; }      // ✅ Nullable
    public DateTime? ShippedDate { get; set; }     // ✅ Already nullable
    public decimal? Discount { get; set; }          // ✅ Nullable decimal
}
```

### Default Value Mapping Table

| DB Column Type | C# Property Type | `DBNull` Behavior | Correct Type |
|---|---|---|---|
| `int NULL` | `int` | Throws `InvalidCastException` | `int?` |
| `nvarchar NULL` | `string` | Throws `NullReferenceException` (pre-.NET 6) or `invalid cast` | `string?` |
| `decimal NULL` | `decimal` | Returns `0.0m` (silent data loss) | `decimal?` |
| `datetime NULL` | `DateTime` | Throws `InvalidCastException` | `DateTime?` |
| `bit NULL` | `bool` | Throws `InvalidCastException` | `bool?` |
| `uniqueidentifier NULL` | `Guid` | Returns `Guid.Empty` (silent data loss) | `Guid?` |
| `int NOT NULL` | `int` | Works fine | `int` |

### Gotcha: NRT (Nullable Reference Types) Warning

With C# 8+ nullable reference types enabled, mapping a `string` (non-nullable) to an `nvarchar NULL` column produces a compiler warning, but many teams suppress or ignore these warnings. Enable NRTs to catch these at compile time:

```csharp
#nullable enable
public class Order
{
    public int Id { get; set; }
    public string CustomerName { get; set; } = ""; // CS8618 — uninitialized
    // Fix: make it nullable
    public string? CustomerName { get; set; }
}
```

### Cost

**Medium** — Runtime crashes and silent data corruption. The symptom is often intermittent, making it hard to reproduce in development environments.

## Anti-Pattern 12 — Dynamic Return Type Abuse

### ❌ Bad Code

```csharp
public IEnumerable<dynamic> GetOrders(int customerId)
{
    using var connection = new SqlConnection(_connectionString);
    var result = connection.Query(
        "SELECT * FROM Orders WHERE CustomerId = @Id",
        new { Id = customerId });
    // ⚠️ Non-generic Query() returns IEnumerable<dynamic>
    return result;
}

// Usage — no compiler checking:
var orders = repo.GetOrders(42);
foreach (var order in orders)
{
    Console.WriteLine(order.CustomerName); // Runtime bind — no intellisense
    Console.WriteLine(order.NonExistent);   // 🔥 No compiler error! Runtime: RuntimeBinderException
}
```

### Variants

```csharp
// Implicit typing hides the dynamic:
var result = connection.Query("SELECT * FROM Orders");
// 'result' is IEnumerable<dynamic>, not IEnumerable<Order>

// Passing dynamic around:
public dynamic GetOrder(int id) => connection.QueryFirst("SELECT * FROM Orders WHERE Id = @Id", new { id });
```

### Symptom

- **No Intellisense** — IDE cannot provide autocomplete for properties of `dynamic` objects. Developers must know all column names by heart or constantly refer to the schema.
- **RuntimeBinderException** — Accessing a property that doesn't exist (because of a column rename or typo) only fails at runtime.
- **Reflection overhead** — Every property access on a `dynamic` object uses DLR (Dynamic Language Runtime) binding, which is slower than static dispatch by 10–100×.
- **Hidden column mapping** — When the SQL changes (adding/removing columns), there is no compile-time feedback. All callers must be manually checked.
- **Serialization issues** — JSON serialization of `dynamic` objects (e.g., returning from an API) can produce inconsistent casing, extra null fields, or circular references that are hard to debug.

### ✅ Fix — Always Use the Generic Variant

```csharp
public record OrderSummary(int Id, string CustomerName, decimal Total);

public IEnumerable<OrderSummary> GetOrderSummaries(int customerId)
{
    using var connection = new SqlConnection(_connectionString);
    return connection.Query<OrderSummary>(
        "SELECT Id, CustomerName, Total FROM Orders WHERE CustomerId = @Id",
        new { Id = customerId });
}
```

### When Dynamic Is Acceptable

- **Ad-hoc scripting** (LINQPad, one-off migration scripts)
- **Truly dynamic schema** (key-value stores, EAV tables where columns are not known at compile time)
- **Rapid prototyping** — but convert to typed classes before production

### Cost

**Medium to High** — Loss of type safety eliminates the primary benefit of C#. Applications with heavy `dynamic` usage have higher defect rates and are harder to refactor.

## Anti-Pattern 13 — Opening the Connection Too Early

### ❌ Bad Code

```csharp
public async Task<IEnumerable<Order>> GetExpensiveOrders(decimal minTotal)
{
    using var connection = new SqlConnection(_connectionString);
    await connection.OpenAsync(); // 🔓 Opened immediately

    // ... some CPU-bound work or delay ...
    await SomeExpensiveValidationAsync(); // ⏳ 200ms of work

    // Connection has been open and idle for 200ms
    return await connection.QueryAsync<Order>(
        "SELECT * FROM Orders WHERE Total >= @MinTotal",
        new { MinTotal = minTotal });
}
```

```csharp
// A more insidious variant — opening in constructor:
public class OrderRepository
{
    private readonly IDbConnection _connection;

    public OrderRepository()
    {
        _connection = new SqlConnection(_connectionString);
        _connection.Open(); // 🔓 Opened at construction, held for object lifetime
    }
}
```

### Symptom

- **Connection pool exhaustion** — Each connection is held open while the application does non-database work. If the validation takes 200ms and the query takes 50ms, the connection is idle 80% of the time.
- **Unnecessary load on SQL Server** — Each open connection consumes a SQL Server worker thread, even when idle. At 100 concurrent requests with early-open connections, all 100 pool slots are occupied, and additional requests queue.
- **Timeout spikes** — Under load, requests that need a connection find the pool exhausted and wait up to `Connect Timeout` (default 15s) before failing.
- **Connection pool fragmentation** — Early-open connections with different `Context Connection` settings or `Application Intent` values fragment the pool, making exhaustion worse.

### ✅ Fix — Open Connection Just Before the Query

```csharp
public async Task<IEnumerable<Order>> GetExpensiveOrders(decimal minTotal)
{
    using var connection = new SqlConnection(_connectionString);
    // Do NOT open yet

    await SomeExpensiveValidationAsync(); // CPU-bound work while connection is closed

    // Open + query together:
    // Dapper's QueryAsync opens the connection automatically if it's closed
    return await connection.QueryAsync<Order>(
        "SELECT * FROM Orders WHERE Total >= @MinTotal",
        new { MinTotal = minTotal });
}
```

### Dapper Auto-Open Behavior

Dapper automatically opens the connection if it's closed when you call `Query`, `Execute`, `QueryAsync`, or `ExecuteAsync`. There is almost never a reason to call `Open()` or `OpenAsync()` explicitly unless you need:

- Multiple operations on the same connection (transaction scope)
- `buffered: false` (streaming) where the connection must be open
- Manually controlling the `SqlConnection` state for advanced pooling scenarios

```csharp
// Explicit open is only needed for these patterns:
await connection.OpenAsync();
using var tx = connection.BeginTransaction();
await connection.ExecuteAsync(sql1, param1, tx);
await connection.ExecuteAsync(sql2, param2, tx);
tx.Commit();
```

### Cost

**Medium** — Connection pool contention under load. For applications with significant business logic between opening the connection and executing the query, the waste is substantial.

## Anti-Pattern 14 — `CommandDefinition` Without Explicit Command Timeout

### ❌ Bad Code

```csharp
public async Task<IEnumerable<Order>> GetOrdersReport(DateTime start, DateTime end)
{
    using var connection = new SqlConnection(_connectionString);
    // Uses default command timeout of 30 seconds
    return await connection.QueryAsync<Order>(
        "sp_GenerateReport @Start, @End",
        new { Start = start, End = end });
    // If the report takes >30s, this throws a SqlException
}
```

### Symptom

- `System.Data.SqlClient.SqlException: Execution Timeout Expired. The timeout period elapsed prior to completion of the operation or the server is not responding.`
- **Intermittent failures** — The same query works during off-peak hours (fast) but fails during peak hours (slow).
- **False positives in monitoring** — Timeout exceptions get flagged as database health issues when the real problem is an inappropriate default timeout.
- **User-facing errors** — Long-running reports, batch operations, or maintenance queries fail, leaving users with no results and an error message.
- **Retry storms** — If a circuit breaker or retry policy catches the timeout, it retries the query, which also times out, amplifying database load.

### ✅ Fix — Set an Appropriate Command Timeout

```csharp
public async Task<IEnumerable<Order>> GetOrdersReport(
    DateTime start, DateTime end,
    CancellationToken cancellationToken = default)
{
    using var connection = new SqlConnection(_connectionString);
    var command = new CommandDefinition(
        "sp_GenerateReport @Start, @End",
        new { Start = start, End = end },
        commandTimeout: 120, // 2 minutes for this specific query
        cancellationToken: cancellationToken);
    return await connection.QueryAsync<Order>(command);
}
```

### Timeout Guidelines

| Query Type | Typical Timeout | Rationale |
|---|---|---|
| Simple CRUD (by PK) | Default (30s) | Should never take >1s |
| Reporting query (simple) | 60s | May scan moderate tables |
| Reporting query (complex) | 120–300s | Aggregations over large tables |
| ETL / batch operation | 600s+ | Long-running bulk work |
| Stored procedure (unknown duration) | 300s | Safety margin for first run |

### Setting Global Default

```csharp
// In your startup / DI configuration:
services.AddSingleton(new CommandDefinition
{
    CommandTimeout = 60 // Default for all commands in this app
});

// Or set at the connection string level:
// "Server=...;Connection Timeout=15;Command Timeout=60"
```

**Important**: Connection timeout (`Connect Timeout`) and command timeout (`Command Timeout`) are different. Connection timeout is how long to wait for a pool connection; command timeout is how long to wait for query execution.

### Cost

**Medium** — Production outages for long-running queries. The default 30-second timeout is appropriate for OLTP but completely unsuitable for reporting, batch, or maintenance operations.

## Anti-Pattern 15 — Single Active Result Set (MARS) Violations with `buffered: false`

### ❌ Bad Code

```csharp
public async Task ProcessOrdersAsync()
{
    using var connection = new SqlConnection(_connectionString);
    await connection.OpenAsync();

    // First un-buffered reader (streaming):
    var orders = connection.Query<Order>(
        "SELECT * FROM Orders WHERE Processed = 0",
        buffered: false);

    foreach (var order in orders)
    {
        // 🔥 Trying to execute another query while the first reader is open:
        var details = await connection.QueryAsync<OrderDetail>(
            "SELECT * FROM OrderDetails WHERE OrderId = @Id",
            new { order.Id });
        // SqlException: "There is already an open DataReader associated with this Command
        //                which must be closed first."
    }
}
```

### Symptom

- `System.InvalidOperationException: There is already an open DataReader associated with this Command which must be closed first.`
- The exception occurs only when `buffered: false` is used. With buffered (default), the reader is closed before iteration begins, so nested queries work.
- **Intermittent** — Without MARS enabled, this always fails. With MARS enabled, it works but may have subtle concurrency issues.

### ✅ Fix Options

**Option 1 — Enable MARS in the connection string** (use with caution):

```csharp
// "Server=...;Database=...;MultipleActiveResultSets=True"
// Then the nested queries work, but MARS has overhead and can cause deadlocks.
```

**Option 2 — Buffer the first result set** (preferred):

```csharp
public async Task ProcessOrdersAsync()
{
    using var connection = new SqlConnection(_connectionString);
    // Buffer the orders list first:
    var orders = (await connection.QueryAsync<Order>(
        "SELECT * FROM Orders WHERE Processed = 0")).ToList();

    // Then iterate with nested queries — reader is closed:
    foreach (var order in orders)
    {
        var details = await connection.QueryAsync<OrderDetail>(
            "SELECT * FROM OrderDetails WHERE OrderId = @Id",
            new { order.Id });
    }
}
```

**Option 3 — Use a JOIN to avoid the N+1** (best for performance):

```csharp
public async Task ProcessOrdersAsync()
{
    using var connection = new SqlConnection(_connectionString);
    var orderDictionary = new Dictionary<int, Order>();

    var results = await connection.QueryAsync<Order, OrderDetail, Order>(
        "SELECT o.*, od.* FROM Orders o " +
        "LEFT JOIN OrderDetails od ON o.Id = od.OrderId " +
        "WHERE o.Processed = 0",
        (order, detail) =>
        {
            if (!orderDictionary.TryGetValue(order.Id, out var existing))
            {
                existing = order;
                existing.Details = new List<OrderDetail>();
                orderDictionary.Add(order.Id, existing);
            }
            if (detail is not null)
                existing.Details.Add(detail);
            return existing;
        },
        splitOn: "Id");
}
```

### MARS Connection String Tradeoffs

| Aspect | MARS Enabled | MARS Disabled |
|---|---|---|
| Nested readers | Allowed | Throws exception |
| Performance overhead | Small (~5-10%) | None |
| Deadlock risk | Higher (shared locks held longer) | Lower |
| Complexity | More concurrency edge cases | Simpler |

### Cost

**Medium** — Runtime exceptions that only appear in production when result sets are large enough to justify un-buffered queries. Usually caught late because dev databases are small enough that buffering works fine.

## Anti-Pattern 16 — Ignoring `SqlConnection` Pooling Settings

### ❌ Bad Code

```csharp
// Default connection string with no pooling considerations:
private const string ConnectionString =
    "Server=prod-db;Database=Orders;User Id=app;Password=pwd;";

// Application creates and destroys thousands of connections per minute:
public async Task<IEnumerable<Order>> GetOrdersAsync(int customerId)
{
    using var connection = new SqlConnection(ConnectionString);
    return await connection.QueryAsync<Order>("...", new { customerId });
}
```

### Symptom

- **Pool fragmentation** — If multiple connection strings differ by even one character (trailing space, different `Application Name`), each gets its own pool. The application may have 40 pools with 100 connections each (4000 total connections) instead of one pool of 100.
- **Pool bloat under load** — Default `Min Pool Size = 0` means connections are created on demand. Under a sudden burst of traffic, the pool grows rapidly, causing SQL Server to handle hundreds of new login requests simultaneously.
- **Login bottleneck** — Each new connection requires a SQL Server login handshake (SSL/TLS negotiation, authentication). With `Min Pool Size = 0` and many concurrent connections, login storms degrade database performance.
- **Connection leak accumulation** — If the connection string has `Pooling=false`, every `Dispose` physically closes the TCP connection instead of returning it to the pool. TCP TIME_WAIT sockets accumulate on the server.

### ✅ Fix — Explicit Pooling Configuration

```csharp
private const string ConnectionString =
    "Server=prod-db;Database=Orders;User Id=app;Password=pwd;" +
    "Min Pool Size=10;Max Pool Size=200;" +
    "Application Name=OrderService;"; // Consistent Application Name avoids pool fragmentation
```

### Recommended Pool Settings by Scenario

| Scenario | Min Pool | Max Pool | Connection Lifetime | Notes |
|---|---|---|---|---|
| Standard web app | 10 | 200 | 0 (infinite) | Balance between startup latency and pool size |
| High-throughput API | 50 | 500 | 300s | Rotate connections to avoid stale query plans |
| Batch processor | 1 | 50 | 0 | Few concurrent connections expected |
| Azure SQL Serverless | 5 | 100 | 600s | Avoid startup latency from serverless pause/resume |
| Microservice (low traffic) | 2 | 50 | 0 | Minimal resource usage |

### Gotcha: Pool Fragmentation by `Application Name`

```csharp
// Different modules use different Application Names:
var conn1 = new SqlConnection("Server=...;Application Name=OrderService");
var conn2 = new SqlConnection("Server=...;Application Name=ReportingService");
// These create DIFFERENT pools even though they connect to the same database!
```

Use a consistent `Application Name` (or omit it) to avoid fragmentation:

```csharp
private const string BaseConnectionString =
    "Server=prod-db;Database=Orders;User Id=app;Password=pwd;";

// All connections share the same pool
```

### Cost

**Medium** — Login storms, connection pool fragmentation, and TCP socket exhaustion. These issues typically manifest during traffic spikes or after deployments.

## Anti-Pattern 17 — Using Dapper with `SqlConnection` That Has Different `ConnectionString` per Call

### ❌ Bad Code

```csharp
public class MultiTenantRepository
{
    public async Task<IEnumerable<Order>> GetOrdersAsync(string tenantDbName)
    {
        var cs = $"Server=prod-db;Database={tenantDbName};User Id=app;Password=pwd;";
        using var connection = new SqlConnection(cs);
        return await connection.QueryAsync<Order>("SELECT * FROM Orders");
    }
}
```

### Symptom

- **Out-of-control pool count** — If there are 1000 tenants, each with a different database name, this creates 1000 separate connection pools. Each pool defaults to `Max Pool Size = 100`, so SQL Server could be configured for 100,000 connections.
- **Memory pressure from unused pools** — Each pool maintains internal state even when idle (event handles, timers, lists). 1000 idle pools consume significant memory on the application server.
- **Slow first-request latency** — Every tenant's first request after an app restart pays the connection establishment cost (TCP + SSL + login). With 1000 tenants, the first request wave is extremely slow.
- **SQL Server connection limit exceeded** — `USER CONNECTIONS` setting may be hit, causing login failures for all new connections.

### ✅ Fix — Connection Pool Sharing via Catalog

```csharp
// Use a single connection pool and switch database with USE statement:
public async Task<IEnumerable<Order>> GetOrdersAsync(string tenantDbName)
{
    var cs = "Server=prod-db;Database=master;User Id=app;Password=pwd;"; // Single pool
    using var connection = new SqlConnection(cs);
    await connection.ExecuteAsync($"USE [{tenantDbName}]"); // Switch catalog
    return await connection.QueryAsync<Order>("SELECT * FROM Orders");
}
```

### Better Fix — Connection Pool per Tenant Group

If `USE` has permissions issues (the app principal may not have `master` access), group tenants into a few pools:

```csharp
// Connection string per tenant with explicit pool sizing:
var cs = $"Server=prod-db;Database={tenantDbName};" +
         $"User Id=app;Password=pwd;" +
         $"Min Pool Size=1;Max Pool Size=20;";
```

### Cost

**High** — Connection pool explosion in multi-tenant systems. A 1000-tenant app with per-tenant connection strings will exhaust either the application's memory or SQL Server's connection limit.

## Anti-Pattern 18 — Ignoring Dapper's `CommandDefinition` for `DbType` Mismatches

### ❌ Bad Code

```csharp
// Stored procedure expects a specific SQL type:
// CREATE PROCEDURE GetOrdersByStatus @StatusIds dbo.IntList READONLY

public async Task<IEnumerable<Order>> GetOrdersByStatusAsync(int[] statusIds)
{
    using var connection = new SqlConnection(_connectionString);
    // Dapper should infer the table-valued parameter, but:
    return await connection.QueryAsync<Order>(
        "EXEC GetOrdersByStatus @StatusIds",
        new { StatusIds = statusIds });
    // ⚠️ Dapper may pass the array as NVARCHAR (comma-separated) instead of TVP
    // or raise a type mismatch error.
}
```

### Symptom

- `SqlException: Operand type clash: nvarchar is incompatible with IntList`
- **Wrong results** — Dapper silently converts the array to a comma-separated string, and the stored procedure performs a string match instead of a set-based lookup.
- **Performance disaster** — Passing a comma-separated string that's parsed inside the stored procedure with `STRING_SPLIT` can work functionally but has terrible performance compared to TVP.

### ✅ Fix — Explicit `DbType`

```csharp
public async Task<IEnumerable<Order>> GetOrdersByStatusAsync(int[] statusIds)
{
    using var connection = new SqlConnection(_connectionString);
    var table = new DataTable();
    table.Columns.Add("Id", typeof(int));
    foreach (var id in statusIds)
        table.Rows.Add(id);

    return await connection.QueryAsync<Order>(
        "EXEC GetOrdersByStatus @StatusIds",
        new { StatusIds = table.AsTableValuedParameter("dbo.IntList") });
    // ✅ Explicitly passes as TVP
});
```

### Common Type Mismatch Cases

| C# Type | SQL Type | Default Behavior | Fix |
|---|---|---|---|
| `int[]` | TVP | Fails or coerces to string | `AsTableValuedParameter()` |
| `string` | `char` (fixed-length) | Passes as `varchar`; query uses LTRIM/RTRIM | Set `DbType = DbType.AnsiStringFixedLength` |
| `decimal` | `money` | Works but loses 4th decimal place rounding | Ensure scale matches |
| `DateTime` | `date` (no time) | Works but truncation warning | Set `DbType = DbType.Date` |
| `bool` | `bit` | Works fine | N/A |
| `Guid` | `uniqueidentifier` | Works fine | N/A |
| `long` (`int64`) | `int` | `OverflowException` | Match types exactly |

### Explicit `DbType` via `CommandDefinition`

```csharp
var command = new CommandDefinition(
    "SELECT * FROM Orders WHERE ShippedDate = @Date",
    new { Date = new DbParameter
    {
        Value = someDate,
        DbType = DbType.Date  // Explicit
    }});
```

### Cost

**Medium** — Runtime exceptions or silent performance degradation. TVP mismatches are especially painful because the error message often doesn't clearly indicate the root cause.

## Anti-Pattern 19 — Not Handling Mapping of Reserved SQL Keywords / Special Columns

### ❌ Bad Code

```csharp
public class User
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public string Group { get; set; } = ""; // "Group" is SQL reserved word
    public string Order { get; set; } = "";  // "Order" is SQL reserved word
}

return connection.Query<User>(
    "SELECT Id, Name, Group, Order FROM Users");
// ⚠️ "Group" and "Order" are SQL reserved keywords — causes syntax error
```

### Symptom

- `SqlException: Incorrect syntax near the keyword 'Group'.`
- `SqlException: Incorrect syntax near the keyword 'Order'.`
- Works in some databases (PostgreSQL with unquoted identifiers is case-normalized) but fails in others.

### ✅ Fix — Quote Reserved Words in SQL

```csharp
// SQL Server uses square brackets:
return connection.Query<User>(
    "SELECT Id, Name, [Group], [Order] FROM Users");

// PostgreSQL uses double quotes:
return connection.Query<User>(
    "SELECT Id, Name, \"Group\", \"Order\" FROM Users");

// MySQL uses backticks:
return connection.Query<User>(
    "SELECT Id, Name, `Group`, `Order` FROM Users");
```

### Better Fix — Use Column Aliases

```csharp
// Rename in SQL to avoid keyword conflicts:
return connection.Query<User>(
    "SELECT Id, Name, [Group] AS UserGroup, [Order] AS UserOrder FROM Users");
```

But then the POCO properties must match the aliases:

```csharp
public class User
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public string UserGroup { get; set; } = "";  // Maps to alias
    public string UserOrder { get; set; } = "";   // Maps to alias
}
```

### Gotcha: Column Name Conflict with Dapper's Internal Fields

```csharp
// Dapper uses "Key" internally for multi-mapping. Avoid naming columns "Key":
return connection.Query<User, Role, User>(
    "SELECT u.Id, u.Name, r.* FROM Users u JOIN Roles r ON u.RoleId = r.Id",
    map: (user, role) => { user.Role = role; return user; },
    splitOn: "Key"); // ⚠️ If a column is named "Key", Dapper splits there!
```

### Cost

**Low to Medium** — SqlException on deployment. Usually caught in integration tests but easily missed when adding new columns to existing queries.

## Anti-Pattern 20 — `QueryMultiple` Without Proper Disposal of `GridReader`

### ❌ Bad Code

```csharp
public (IEnumerable<Order>, IEnumerable<Customer>) GetOrdersAndCustomers()
{
    using var connection = new SqlConnection(_connectionString);
    using var multi = connection.QueryMultiple(
        "SELECT * FROM Orders; SELECT * FROM Customers");

    // Forgetting to read one of the result sets:
    var orders = multi.Read<Order>();
    // multi.Read<Customer>() is never called
    // ⚠️ GridReader holds the connection until ALL results are consumed
}
```

### Symptom

- **Connection not returned to pool** — `GridReader` implements `IDisposable`, but if a caller reads only the first result set and forgets the second, the underlying `SqlDataReader` stays open until `GridReader` is disposed or finalized.
- **Resource leak** — The `SqlConnection` is held open because `SqlDataReader` is still associated with it.
- **Deferred exceptions** — If the second result set has an error (e.g., invalid column name), it's never observed because the result set is never read.
- **Incorrect Dispose pattern** — Wrapping `GridReader` in `using` ensures disposal, but the common mistake is storing `multi` in a field and not disposing it.

### ✅ Fix

```csharp
public (IEnumerable<Order>, IEnumerable<Customer>) GetOrdersAndCustomers()
{
    using var connection = new SqlConnection(_connectionString);
    using var multi = connection.QueryMultiple(
        "SELECT * FROM Orders; SELECT * FROM Customers");

    var orders = multi.Read<Order>();
    var customers = multi.Read<Customer>();
    // ✅ Both result sets consumed. GridReader disposed by 'using'.
    return (orders, customers);
}
```

### Lazy Evaluation Gotcha

```csharp
// 🔥 DANGER — the GridReader is disposed before result materialization:
IEnumerable<Order> GetOrders()
{
    using var connection = new SqlConnection(_connectionString);
    using var multi = connection.QueryMultiple("SELECT * FROM Orders; SELECT * FROM Customers");
    return multi.Read<Order>(); // Returns IEnumerable<Order> (lazy!)
    // GridReader is disposed at the end of 'using', but the enumerable hasn't been enumerated yet!
}
```

Fix: Materialize eagerly:

```csharp
return multi.Read<Order>().ToList(); // ✅ Enumerated immediately
```

### Cost

**Medium** — Connection leaks and resource leaks. The lazy evaluation variant is a particularly nasty bug because it silently returns an empty collection or throws `ObjectDisposedException` at enumeration time, far from the dispose site.

## Anti-Pattern 21 — Overusing `Execute` for Single-Row Operations in a Loop

### ❌ Bad Code

```csharp
public async Task BulkDiscontinueProducts(IEnumerable<int> productIds)
{
    using var connection = new SqlConnection(_connectionString);
    await connection.OpenAsync();
    using var tx = connection.BeginTransaction();

    foreach (var id in productIds) // 🔥 10,000 separate round-trips
    {
        await connection.ExecuteAsync(
            "UPDATE Products SET Discontinued = 1 WHERE Id = @Id",
            new { Id = id },
            transaction: tx);
    }

    tx.Commit();
}
```

### Symptom

- **Network round-trip per row** — Each `ExecuteAsync` sends a TDS packet to SQL Server and waits for the response. With 10,000 products, that's 10,000 round-trips. At 2ms latency (same datacenter), that's 20 seconds of network overhead alone.
- **Transaction log bloat** — Each update generates a log record. If no explicit transaction is used, each update creates an implicit transaction with a log flush. The log must grow to accommodate.
- **High SQL Server CPU** — Each statement must be parsed, compiled (or cached-plan looked up), executed, and committed. This overhead dominates for small operations.
- **Application threading** — If `ExecuteAsync` is called without `ConfigureAwait(false)` in ASP.NET, context switches add further overhead.

### ✅ Fix — Use Table-Valued Parameters or Batch Update

```csharp
// Option 1: TVP bulk update
public async Task BulkDiscontinueProducts(IEnumerable<int> productIds)
{
    var idsTable = new DataTable();
    idsTable.Columns.Add("Id", typeof(int));
    foreach (var id in productIds)
        idsTable.Rows.Add(id);

    using var connection = new SqlConnection(_connectionString);
    await connection.ExecuteAsync(
        "UPDATE p SET Discontinued = 1 " +
        "FROM Products p " +
        "INNER JOIN @Ids i ON p.Id = i.Id",
        new { Ids = idsTable.AsTableValuedParameter("dbo.IntList") });
}
```

```csharp
// Option 2: Batch with SqlBulkCopy
public async Task BulkDiscontinueProducts(IEnumerable<int> productIds)
{
    var table = new DataTable();
    table.Columns.Add("Id", typeof(int));
    table.Columns.Add("Discontinued", typeof(bool));
    foreach (var id in productIds)
        table.Rows.Add(id, true);

    using var bulkCopy = new SqlBulkCopy(_connectionString);
    bulkCopy.DestinationTableName = "Products";
    await bulkCopy.WriteToServerAsync(table);
}
```

```csharp
// Option 3: Single multi-row UPDATE (if all same value)
public async Task BulkDiscontinueProducts(IEnumerable<int> productIds)
{
    using var connection = new SqlConnection(_connectionString);
    var idsList = string.Join(",", productIds);
    await connection.ExecuteAsync(
        $"UPDATE Products SET Discontinued = 1 WHERE Id IN ({idsList})");
    // ⚠️ Use with caution — SQL injection risk if IDs come from user input
}
```

### Performance Comparison (10,000 rows)

| Approach | Time (ms) | Round-trips | Transaction Log |
|---|---|---|---|
| Loop with `ExecuteAsync` | ~20,000 | 10,000 | High (10k log records) |
| TVP with JOIN update | ~150 | 1 | Low (1 set-based operation) |
| `SqlBulkCopy` | ~100 | 1 | Low |
| `IN (...)` clause | ~50 | 1 | Low |

### Cost

**High** — Severe performance degradation for any batch operation. The "iterate and execute" pattern is the #1 performance killer in Dapper-based batch processing.

## Anti-Pattern 22 — Assuming Dapper Maps Private Properties or Fields

### ❌ Bad Code

```csharp
public class Order
{
    public int Id { get; private set; }      // Private setter — Dapper won't set it
    private decimal _total;                    // Private field — Dapper ignores it
    public decimal Total => _total;            // Read-only property — Dapper ignores it

    public Order(int id, decimal total)
    {
        Id = id;
        _total = total;
    }
}

// Dapper's Query<Order> returns objects with Id=0 and Total=0m
```

### Symptom

- **Default values everywhere** — Dapper returns objects with all properties set to their default values (`0`, `null`, `false`), but no exception is thrown.
- **Bugs hidden in production** — The query succeeds and returns the right number of rows, but the data is silently wrong. Developers trust Dapper to have populated the objects.
- **No compile-time warning** — This is not a compile error. Immutable POCOs look correct in code reviews but produce entirely empty objects.

### ✅ Fix

**Option 1 — Parameterized constructor matching column names** (Dapper supports this):

```csharp
public class Order
{
    public int Id { get; }
    public decimal Total { get; }

    public Order(int id, decimal total)
    {
        Id = id;
        Total = total;
    }
}
```

Dapper can construct objects using a parameterized constructor if parameter names match column names (case-insensitive). The constructor must accept all column values.

**Option 2 — Use `[SqlMapper.ITypeMap]` or custom type handler**:

```csharp
public class Order
{
    [Key]
    public int Id { get; private set; }
    public decimal Total { get; private set; }
}

// Dapper uses the parameterless constructor + property setters
// But private setters require Dapper's default access:
SqlMapper.Settings.IncludePrivateSetters = true; // Dapper 2.1+
```

**Option 3 — Record types** (cleanest):

```csharp
public record Order(int Id, decimal Total);
// Dapper handles records perfectly — positional constructor matches columns
```

### Dapper Property Resolution Order

1. Parameterized constructor (if parameter names match column names)
2. Parameterless constructor + public property setters
3. Fields (opt-in: `SqlMapper.Settings.IncludePrivateFields = true`)

### Gotcha: Init-only Setters

```csharp
public record Order
{
    public int Id { get; init; }  // Init-only — Dapper 2.0.78+ supports this
    public decimal Total { get; init; }
}
```

Dapper 2.0.78+ supports init-only setters. If you're on an older version, init-only properties are silently ignored.

### Cost

**High** — Silent data corruption. Entire rows appear to load successfully but contain no actual data. This is especially dangerous in reporting or data export scenarios where missing data looks like "no results" rather than an error.

## Anti-Pattern 23 — Forgetting `OrderColumnNameSplit` in Multi-Mapping (Default `Id`)

### ❌ Bad Code

```csharp
public IEnumerable<Order> GetOrdersWithDetails()
{
    using var connection = new SqlConnection(_connectionString);
    return connection.Query<Order, OrderDetail, Order>(
        "SELECT o.*, od.* FROM Orders o " +
        "LEFT JOIN OrderDetails od ON o.Id = od.OrderId",
        map: (order, detail) =>
        {
            order.Details ??= new List<OrderDetail>();
            order.Details.Add(detail);
            return order;
        },
        splitOn: "Id" // ⚠️ Default splitOn value
    );
}
```

### Symptom

- **Wrong split point** — Dapper splits on the first column named `Id`. But both `Orders` and `OrderDetails` have an `Id` column. Dapper splits at the Order's `Id` and assigns the rest (including OrderDetail's `Id` and all other details columns) to `OrderDetail`.
- **Missing details** — `OrderDetail` object gets incorrect values. Its `Id` may receive the `OrderId` value (first column after the split).
- **Incorrect data in results** — The Orders may appear to have no details, or the details have wrong IDs, corruption that silently flows into business logic.

### ✅ Fix — Explicit `splitOn` with the Boundary Column

```csharp
public IEnumerable<Order> GetOrdersWithDetails()
{
    using var connection = new SqlConnection(_connectionString);
    return connection.Query<Order, OrderDetail, Order>(
        "SELECT o.*, od.* FROM Orders o " +
        "LEFT JOIN OrderDetails od ON o.Id = od.OrderId",
        map: (order, detail) =>
        {
            order.Details ??= new List<OrderDetail>();
            if (detail is not null)
                order.Details.Add(detail);
            return order;
        },
        splitOn: "OrderId" // Dapper splits when it sees "OrderId" column from OrderDetails
    );
}
```

### Alias-Based Approach to Control Split

```csharp
public IEnumerable<Order> GetOrdersWithDetails()
{
    using var connection = new SqlConnection(_connectionString);
    return connection.Query<Order, OrderDetail, Order>(
        "SELECT o.Id, o.CustomerName, o.Total, " +        // Order columns
        "od.Id AS DetailId, od.ProductId, od.Quantity, " + // OrderDetail columns (aliased)
        "od.UnitPrice " +
        "FROM Orders o LEFT JOIN OrderDetails od ON o.Id = od.OrderId",
        map: (order, detail) =>
        {
            order.Details ??= new List<OrderDetail>();
            if (detail is not null)
                order.Details.Add(detail);
            return order;
        },
        splitOn: "DetailId" // ✅ Split on the aliased column boundary
    );
}
```

### Multi-Mapping Split Rules

- Dapper reads columns left to right.
- When it encounters a column matching the `splitOn` name, it begins populating the next type in the generic parameter list.
- Default `splitOn` is `"Id"`. Use a different `splitOn` if the column boundary is not named `Id`.

### Cost

**Medium** — Subtle data corruption in related entity collections. The symptom can appear as "some orders have details but most don't" depending on the order of columns in `SELECT *`.

## Anti-Pattern 24 — Not Using `AsList()` or `AsTableValuedParameter()` for IN Clauses

### ❌ Bad Code

```csharp
public async Task<IEnumerable<Order>> GetOrdersByIdsAsync(IEnumerable<int> ids)
{
    using var connection = new SqlConnection(_connectionString);
    // Dapper expands the IEnumerable into @id1, @id2, @id3, ... dynamically
    return await connection.QueryAsync<Order>(
        "SELECT * FROM Orders WHERE Id IN @Ids",
        new { Ids = ids });
}
```

### What Dapper Does Internally

Dapper's parameter expansion rewrites `WHERE Id IN @Ids` to `WHERE Id IN (@id1, @id2, @id3, ...)`. It generates `@id1 = ids[0]`, `@id2 = ids[1]`, etc. This is convenient but has several problems:

- **SQL injection surface** — Dapper handles escaping correctly, but the dynamic SQL prevents query plan caching. Each unique set of IDs generates a new SQL string and a new execution plan.
- **Plan cache bloat** — A query with 10 IDs and a query with 11 IDs are different SQL strings, so they get different plans. Over time, the plan cache fills with near-identical plans.
- **Parameter limit** — SQL Server supports a maximum of 2100 parameters. With 2000 IDs, other parameters in the query may push it over the limit, causing `SqlException`.
- **Plan guide / forced parameterization issues** — Dynamic parameter expansion defeats forced parameterization, which some DBAs use to stabilize query plans.

### ✅ Fix

**Option 1 — TVP (recommended)**:

```csharp
public async Task<IEnumerable<Order>> GetOrdersByIdsAsync(int[] ids)
{
    using var connection = new SqlConnection(_connectionString);
    var idsTable = new DataTable();
    idsTable.Columns.Add("Id", typeof(int));
    foreach (var id in ids) idsTable.Rows.Add(id);

    return await connection.QueryAsync<Order>(
        "SELECT o.* FROM Orders o " +
        "INNER JOIN @Ids i ON o.Id = i.Id",
        new { Ids = idsTable.AsTableValuedParameter("dbo.IntList") });
}
```

**Option 2 — Batch in chunks** (for very large ID lists):

```csharp
public async Task<IEnumerable<Order>> GetOrdersByIdsAsync(int[] ids, int chunkSize = 1000)
{
    using var connection = new SqlConnection(_connectionString);
    var results = new List<Order>();
    foreach (var chunk in ids.Chunk(chunkSize))
    {
        var chunkResults = await connection.QueryAsync<Order>(
            "SELECT * FROM Orders WHERE Id IN @Ids",
            new { Ids = chunk });
        results.AddRange(chunkResults);
    }
    return results;
}
```

### Cost

**Medium** — Plan cache bloat and risk of hitting the 2100 parameter limit. In high-throughput systems, the plan cache pollution from parameter expansion causes plan eviction for other queries, degrading overall performance.

## Anti-Pattern 25 — Sharing `SqlConnection` Across Threads Without Synchronization

### ❌ Bad Code

```csharp
// A singleton repository that shares a single connection:
public class OrderRepository
{
    private readonly SqlConnection _connection = new(_connectionString);

    public async Task<IEnumerable<Order>> GetOrdersAsync()
    {
        await _connection.OpenAsync(); // ⚠️ Thread 1 opens
        return await _connection.QueryAsync<Order>("...");
    }

    public async Task<IEnumerable<Customer>> GetCustomersAsync()
    {
        await _connection.OpenAsync(); // ⚠️ Thread 2 opens concurrently
        return await _connection.QueryAsync<Customer>("...");
    }
}

// Called from multiple threads:
// Thread 1: await repo.GetOrdersAsync();
// Thread 2: await repo.GetCustomersAsync();
```

### Symptom

- `InvalidOperationException: The connection is already open.` — Thread 1 opens the connection, Thread 2 tries to open the same connection before it's closed.
- **Corrupted reads** — If Thread 1 is reading data while Thread 2 issues a command on the same connection, the data reader gets corrupted. Results from both queries mix together.
- **Deadlocks** — Thread 1 opens connection → Thread 2 waits for open → Thread 1 waits for Thread 2's result. Hang.
- **Mystery behavior** — The issue is non-deterministic. It may work for days in development (low concurrency) and fail immediately under production load.

### ✅ Fix — Create Connections Per Operation

```csharp
public class OrderRepository
{
    private readonly string _connectionString;

    public OrderRepository(string connectionString)
    {
        _connectionString = connectionString;
    }

    public async Task<IEnumerable<Order>> GetOrdersAsync()
    {
        using var connection = new SqlConnection(_connectionString);
        return await connection.QueryAsync<Order>("...");
    }

    public async Task<IEnumerable<Customer>> GetCustomersAsync()
    {
        using var connection = new SqlConnection(_connectionString);
        return await connection.QueryAsync<Customer>("...");
    }
}
```

### When Shared Connection Is Acceptable

- **Synchronous, single-threaded batch jobs** that run a sequence of commands.
- **Transaction scopes** where the connection is used by a single logical operation.
- **Integration tests** where a single connection is opened and reused for test setup/teardown (with careful synchronization).

### Verified: Connection Pooling Makes Per-Operation Connection Cheap

Each `new SqlConnection()` from the same connection string returns an already-pooled connection. The "create per operation" pattern does not create a new TCP connection each time; it borrows from the pool. The overhead is minimal (object allocation + pool lookup).

### Cost

**Critical** — Data corruption, race conditions, and thread safety violations. This anti-pattern is often introduced by developers trying to "optimize" by reusing connections, not realizing that `SqlConnection` is not thread-safe.

## Prevention Checklist

- [ ] All `SqlConnection` instances are wrapped in `using` (or declared with `using var`)
- [ ] No `async void` methods using Dapper — always `async Task`
- [ ] No `.Result`, `.Wait()`, or `.GetAwaiter().GetResult()` on Dapper calls
- [ ] All `SELECT` statements list specific columns (no `SELECT *` in production code)
- [ ] All async Dapper methods accept and forward `CancellationToken`
- [ ] Parameter names in anonymous types match SQL parameter names exactly
- [ ] Connection strings come from configuration (never hardcoded)
- [ ] Dapper is used for queries that benefit from its speed; EF Core handles standard CRUD
- [ ] Large result sets use `buffered: false` or are explicitly paginated
- [ ] Explicit transactions are passed to every Dapper call within the transaction scope
- [ ] Dapper and EF Core transactions share the same `SqlConnection` when both are used
- [ ] All POCO properties that may receive `NULL` from the database are nullable
- [ ] Generic `Query<T>` is used instead of non-generic `Query()`
- [ ] Connections are opened just-in-time, not at construction or before non-DB work
- [ ] Appropriately large commands use explicit `commandTimeout: N`
- [ ] Batch operations use TVP, `SqlBulkCopy`, or set-based SQL instead of per-row loops
- [ ] POCO constructors and setters are compatible with Dapper's mapping rules
- [ ] Multi-mapping queries specify the correct `splitOn` column
- [ ] Connection string pooling settings (`Min Pool Size`, `Max Pool Size`) are tuned for the workload
- [ ] Multi-tenant systems share pools where possible (avoid dynamic per-tenant connection strings)
