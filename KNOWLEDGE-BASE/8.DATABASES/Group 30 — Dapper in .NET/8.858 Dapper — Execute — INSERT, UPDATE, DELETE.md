### Where This Fits

`Execute` is Dapper's method for statements that return no result set — INSERT, UPDATE, DELETE, and DDL commands. It wraps ADO.NET's `Command.ExecuteNonQuery` and returns the number of affected rows. A .NET backend engineer encounters this everywhere Dapper replaces EF Core for write operations: inserting orders, updating inventory counts, deleting expired sessions. When this is unknown or misused, teams either: (a) call `Query` for DML and ignore the result set (wasteful), (b) use EF Core's SaveChanges with full change tracking when they only need a simple INSERT (unnecessary overhead), or (c) write raw ADO.NET ExecuteNonQuery calls without Dapper's parameterization (SQL injection risk). The interview signal is moderate: it tests whether a candidate knows the simplest Dapper write method and its return value semantics.

## Core Mental Model

`Execute` sends any SQL command that does not return rows — INSERT, UPDATE, DELETE, CREATE TABLE, TRUNCATE TABLE — and returns the number of rows affected as an `int`. Dapper creates an `IDbCommand`, sets the command text and parameters, calls `Command.ExecuteNonQuery()`, and returns the result. The method does NO result mapping — it is pure command execution with parameter binding. For INSERT with identity column, use `Execute` plus `ExecuteScalar` for `SCOPE_IDENTITY()`. For batch operations, pass a collection of parameter objects to `Execute` with the same command text — Dapper executes the command multiple times in a single method call. The invariant: **Execute returns the row count from `ExecuteNonQuery` — for DDL (CREATE, DROP, TRUNCATE), this is -1.** The recognition pattern: a SQL command string with parameter placeholders, an anonymous object or DynamicParameters for values, and the result assigned to an `int` variable.

### Classification

`Execute` is a **Dapper extension method on `IDbConnection`** that wraps **ADO.NET `IDbCommand.ExecuteNonQuery()`**. It belongs to the **Dapper command execution layer** — no IL Emit, no result deserialization, no mapping overhead. The method supports **single execution** (one set of parameters) and **batch execution** (multiple parameter sets). The batch execution is NOT a single database round trip — Dapper iterates and calls `ExecuteNonQuery` for each parameter set in a loop. For true bulk operations, use `SqlBulkCopy` or Dapper's `BulkExtensions`. The performance characteristic is **direct ADO.NET speed** — the overhead above raw `ExecuteNonQuery` is parameter binding only.

## Deep Mechanics

### How the Engine Executes This

1. **Dapper receives `connection.ExecuteAsync(sql, param)`.** The `sql` parameter is any valid T-SQL DML or DDL statement. The `param` parameter is an anonymous object, `DynamicParameters`, or a dictionary.

2. **Dapper creates an `IDbCommand`.** Using the connection's `CreateCommand()` factory method, Dapper instantiates a `SqlCommand`. It sets the `CommandText` to the provided SQL. For each property on the parameter object, Dapper calls `IDbCommand.Parameters.Add()` with the property name and value, inferring the `DbType` from the .NET type.

3. **Dapper calls `IDbCommand.ExecuteNonQueryAsync()`.** This sends the command to SQL Server via TDS. SQL Server parses the SQL, compiles an execution plan (if not cached), and executes it.

4. **SQL Server executes the DML.** For INSERT, the storage engine allocates a new data page (or finds space in an existing page), writes the row, updates any affected indexes, writes to the transaction log. For UPDATE, the engine reads the row, applies changes, writes the updated row, updates indexes, writes to the transaction log. For DELETE, the engine marks the row as deleted (or actually removes it depending on the table's design), updates indexes, writes to the transaction log.

5. **SQL Server returns the affected row count.** The TDS response includes `DONE_IN_PROC` tokens with the row count. ADO.NET surfaces this as the return value of `ExecuteNonQuery`.

6. **Dapper returns the int.** No further processing. The connection remains open (unless `CommandBehavior.CloseConnection` was implied).

### Batch Execution

```csharp
// Batch INSERT — multiple orders in one ExecuteAsync call
public async Task<int> InsertOrdersBatchAsync(IEnumerable<Order> orders, CancellationToken ct)
{
    const string sql = @"
        INSERT INTO Orders (CustomerId, OrderDate, Status, TotalAmount)
        VALUES (@CustomerId, @OrderDate, @Status, @TotalAmount);";

    await using var connection = _connectionFactory.Create();
    await connection.OpenAsync(ct);

    // Dapper loops through each order and calls ExecuteNonQuery
    var totalAffected = await connection.ExecuteAsync(
        new CommandDefinition(sql, orders, cancellationToken: ct));

    return totalAffected; // sum of all rows affected
}
```

Important: This batch execution is NOT a single database round trip. Dapper iterates the parameter collection and calls `ExecuteNonQuery` for each item. For 1000 orders, this sends 1000 individual TDS batches. For true bulk insert, use `SqlBulkCopy` or `Dapper.BulkExtensions`.

### Execution Plan Analysis

For `INSERT INTO Orders ... VALUES (...)`:

- **Plan shape:** `[Clustered Index Insert]` → `[Top]` (for the OUTPUT clause if present)
- **Operators:** `Clustered Index Insert` (one row into the clustered index), plus `Index Insert` for each non-clustered index
- **Logical reads:** ~1 per index (clustered index write + NCI writes)
- **Without OUTPUT:** plan is `Clustered Index Insert` only
- **With OUTPUT:** plan adds a `Compute Scalar` for the OUTPUT expression + `Top` to cap at 1 row

For `UPDATE Orders SET ... WHERE OrderId = @Id`:

- **Plan shape:** `[Clustered Index Seek]` → `[Compute Scalar]` → `[Clustered Index Update]`
- **Logical reads:** ~2 (1 seek, 1 update write)
- **If updating a non-clustered index key column:** additional `Index Update` operators per NCI

For `DELETE FROM UserSessions WHERE ExpiresAt < @Cutoff`:

- **Plan shape:** `[Clustered Index Scan]` (if no index on ExpiresAt) → `[Clustered Index Delete]`
- **Without index on ExpiresAt:** full scan of UserSessions — 50,000 logical reads on 50K rows
- **With index on ExpiresAt:** `[Index Seek]` → `[Clustered Index Delete]` per row — ~1 logical read per deleted row

### Failure Modes

- **Constraint violation:** INSERT fails with `SqlException` (FK violation, unique constraint, NULL violation). Dapper propagates the exception. No transaction rollback unless caller manages it.
- **Concurrent UPDATE conflict:** Two concurrent UPDATEs on the same row. The second succeeds (last writer wins). Use `ROWVERSION` or `WHERE` clause with original values for optimistic concurrency.
- **DELETE with WHERE clause that matches no rows:** Returns 0. Not an error, but may indicate a logic bug.
- **Parameter type mismatch:** SQL Server casts implicitly (e.g., string to int). If cast fails, `SqlException` with conversion error. Dapper does not validate parameter types.
- **Transaction not committed:** If `ExecuteAsync` is called within a transaction but the transaction is never committed, the DML is rolled back on connection close/dispose.

## Production Patterns and Implementation

### Primary Dapper Implementation — CRUD Repository

```csharp
public sealed class OrderRepository : IOrderRepository
{
    private readonly IDbConnectionFactory _connectionFactory;
    private readonly ILogger<OrderRepository> _logger;

    public OrderRepository(IDbConnectionFactory connectionFactory, ILogger<OrderRepository> logger)
    {
        _connectionFactory = connectionFactory;
        _logger = logger;
    }

    public async Task<int> CreateOrderAsync(Order order, CancellationToken ct)
    {
        const string sql = @"
            INSERT INTO Orders (CustomerId, OrderDate, Status, TotalAmount, ShippingAddressId)
            OUTPUT INSERTED.OrderId
            VALUES (@CustomerId, @OrderDate, @Status, @TotalAmount, @ShippingAddressId);";

        await using var connection = _connectionFactory.Create();
        await connection.OpenAsync(ct);

        var newId = await connection.QuerySingleAsync<int>(
            new CommandDefinition(sql, order, cancellationToken: ct));

        _logger.LogInformation("Created order {OrderId}", newId);
        return newId;
    }

    public async Task<bool> UpdateOrderStatusAsync(int orderId, string status, CancellationToken ct)
    {
        const string sql = @"
            UPDATE Orders
            SET Status = @Status, ModifiedAt = SYSDATETIME()
            WHERE OrderId = @OrderId;";

        await using var connection = _connectionFactory.Create();
        await connection.OpenAsync(ct);

        var rows = await connection.ExecuteAsync(
            new CommandDefinition(sql, new { OrderId = orderId, Status = status },
                cancellationToken: ct));

        return rows == 1;
    }

    public async Task<bool> DeleteOrderAsync(int orderId, CancellationToken ct)
    {
        const string sql = @"
            DELETE FROM OrderItems WHERE OrderId = @OrderId;
            DELETE FROM Orders WHERE OrderId = @OrderId;";

        await using var connection = _connectionFactory.Create();
        await connection.OpenAsync(ct);
        await using var tx = connection.BeginTransaction();

        var itemsDeleted = await connection.ExecuteAsync(
            new CommandDefinition("DELETE FROM OrderItems WHERE OrderId = @OrderId",
                new { OrderId = orderId }, cancellationToken: ct), tx);
        var orderDeleted = await connection.ExecuteAsync(
            new CommandDefinition("DELETE FROM Orders WHERE OrderId = @OrderId",
                new { OrderId = orderId }, cancellationToken: ct), tx);

        if (orderDeleted == 0)
        {
            await tx.RollbackAsync(ct);
            return false;
        }

        await tx.CommitAsync(ct);
        _logger.LogInformation("Deleted order {OrderId} with {ItemCount} items",
            orderId, itemsDeleted);
        return true;
    }

    public async Task<int> InsertOrderItemsAsync(
        int orderId, IEnumerable<OrderItem> items, CancellationToken ct)
    {
        const string sql = @"
            INSERT INTO OrderItems (OrderId, ProductId, Quantity, UnitPrice)
            VALUES (@OrderId, @ProductId, @Quantity, @UnitPrice);";

        await using var connection = _connectionFactory.Create();
        await connection.OpenAsync(ct);

        var itemsWithOrderId = items.Select(i => new
        {
            OrderId = orderId,
            i.ProductId,
            i.Quantity,
            i.UnitPrice
        });

        var totalAffected = await connection.ExecuteAsync(
            new CommandDefinition(sql, itemsWithOrderId, cancellationToken: ct));

        return totalAffected;
    }
}
```

## Gotchas and Production Pitfalls

### 1 — Execute Returns -1 for DDL, Not Affected Rows

**Pitfall:** Developer checks return value of `Execute` for a `CREATE TABLE` or `TRUNCATE TABLE` command and expects the number of affected rows.

```csharp
// ❌ Wrong: checking affected rows on DDL
var rows = await connection.ExecuteAsync("TRUNCATE TABLE AuditLog;");
if (rows == 0) // ❌ -1 != 0, so this never triggers
    _logger.LogWarning("No rows truncated?");
```

**Symptom:** `Execute` returns -1 for DDL statements (this is ADO.NET behavior: `ExecuteNonQuery` returns -1 for statements that don't affect rows). The developer's logic checks for `rows == 0` or `rows > 0` and gets wrong results.

**Fix:**

```csharp
// ✅ Correct: don't check return value for DDL
await connection.ExecuteAsync("TRUNCATE TABLE AuditLog;");
_logger.LogInformation("AuditLog truncated");

// ✅ Or: use separate handling for DDL vs DML
if (sql.TrimStart().StartsWith("TRUNCATE", StringComparison.OrdinalIgnoreCase))
{
    await connection.ExecuteAsync(sql);
    // Return value is meaningless (-1)
}
```

**Cost of not fixing:** Silent logic bugs in migration scripts. Migration reports "failed" when DDL succeeded.

### 2 — Not Wrapping Multiple Writes in a Transaction

**Pitfall:** Two related `Execute` calls — one for parent INSERT, one for child INSERT — without a transaction.

```csharp
// ❌ Wrong: no transaction — partial write on failure
await connection.ExecuteAsync(
    "INSERT INTO Orders (CustomerId, TotalAmount) VALUES (@C, @T);", order);
await connection.ExecuteAsync(
    "INSERT INTO OrderItems (OrderId, ProductId, Quantity) VALUES (@O, @P, @Q);", items);
// If second INSERT fails, the order is orphaned
```

**Symptom:** Orphaned parent rows when child INSERT fails. Data integrity violation.

**Fix:**

```csharp
// ✅ Correct: explicit transaction
await using var tx = connection.BeginTransaction();
try
{
    order.OrderId = await connection.QuerySingleAsync<int>(
        "INSERT INTO Orders... OUTPUT INSERTED.OrderId...", order, tx);
    await connection.ExecuteAsync(
        "INSERT INTO OrderItems (OrderId, ProductId, Quantity) VALUES (@O, @P, @Q);",
        items.Select(i => new { O = order.OrderId, i.ProductId, i.Quantity }), tx);
    await tx.CommitAsync(ct);
}
catch
{
    await tx.RollbackAsync(ct);
    throw;
}
```

**Cost of not fixing:** Data corruption. Referential integrity violations. Manual cleanup required.

### 3 — Batch Execute Is Not a Single Database Round Trip

**Pitfall:** Developer thinks passing 10,000 parameter objects to `ExecuteAsync` sends one SQL batch.

```csharp
// ❌ Wrong: this is 10,000 individual round trips
var orders = Get10000Orders(); // IEnumerable<Order>
var total = await connection.ExecuteAsync(sql, orders); // 10,000 ExecuteNonQuery calls
```

**Symptom:** 10,000 round trips at 2ms each = 20 seconds. Debug output shows 10,000 individual `sp_executesql` calls.

**Fix:**

```csharp
// ✅ Option A: SqlBulkCopy for > 1000 rows
using var bulkCopy = new SqlBulkCopy(connectionString);
bulkCopy.DestinationTableName = "Orders";
var dataTable = OrdersToDataTable(orders);
await bulkCopy.WriteToServerAsync(dataTable, ct);

// ✅ Option B: Dapper BulkExtensions
// Install-Package Dapper.BulkExtensions
await connection.BulkInsertAsync(orders);

// ✅ Option C: TVP for 100-5000 rows (SQL Server)
// Pass DataTable as Structured parameter
var total = await connection.ExecuteAsync(sql,
    new { Orders = orderTable.AsTableValuedParameter("dbo.OrderType") });
```

**Cost of not fixing:** 20+ second INSERT time for 10K rows. Connection pool exhaustion. API timeout.

### 4 — Execute Does Not Populate Output Parameters Automatically

**Pitfall:** Developer uses `Execute` for a stored procedure that has OUTPUT parameters and expects them to be populated.

```csharp
// ❌ Wrong: Execute does not read output parameters
var p = new DynamicParameters();
p.Add("@NewId", dbType: DbType.Int32, direction: ParameterDirection.Output);
await connection.ExecuteAsync("sp_InsertOrder", p,
    commandType: CommandType.StoredProcedure);
var id = p.Get<int>("@NewId"); // ❌ May be null or wrong — Execute may not populate outputs
```

**Symptom:** OUTPUT parameters are `null` or have default values. This depends on Dapper version — some versions do populate outputs on `Execute`, but it's inconsistent.

**Fix:**

```csharp
// ✅ Correct: use Query or QueryMultiple for procs with OUTPUT parameters
var id = await connection.QuerySingleAsync<int>("sp_InsertOrder", p,
    commandType: CommandType.StoredProcedure);
// OUTPUT parameters are populated after the query completes
var newId = p.Get<int>("@NewId");
```

**Cost of not fixing:** Silent data loss — new ID is 0, insert goes through but tracking ID is lost.

### 5 — Optimistic Concurrency Without ROWVERSION Check

**Pitfall:** UPDATE overwrites another transaction's changes because there's no concurrency check.

```csharp
// ❌ Wrong: last writer wins — no concurrency check
await connection.ExecuteAsync(
    "UPDATE Orders SET Status = @Status WHERE OrderId = @OrderId",
    new { Status = "Shipped", OrderId = 42 });
// Another concurrent request may have changed the status to "Cancelled"
// This UPDATE overwrites it silently
```

**Symptom:** Lost updates. Business rules violated (e.g., order shipped after cancellation).

**Fix:**

```csharp
// ✅ Correct: use ROWVERSION for optimistic concurrency
// Table schema: Add RowVersion column (TIMESTAMP in SQL Server)

public async Task<bool> UpdateOrderWithConcurrencyAsync(Order order, CancellationToken ct)
{
    const string sql = @"
        UPDATE Orders
        SET Status = @Status, ModifiedAt = SYSDATETIME()
        WHERE OrderId = @OrderId
          AND RowVersion = @RowVersion;";

    await using var connection = _connectionFactory.Create();
    await connection.OpenAsync(ct);

    var rows = await connection.ExecuteAsync(
        new CommandDefinition(sql, new
        {
            order.OrderId,
            Status = "Shipped",
            RowVersion = order.RowVersion
        }, cancellationToken: ct));

    if (rows == 0)
    {
        _logger.LogWarning("Concurrency conflict on order {OrderId}", order.OrderId);
        throw new DbUpdateConcurrencyException(
            $"Order {order.OrderId} was modified by another user.");
    }
    return true;
}
```

**Cost of not fixing:** Inventory shipped for cancelled orders. Financial reconciliation errors. Customer disputes.

### 6 — Parameter Sniffing with Execute

**Pitfall:** Repeated `Execute` calls with different parameter values reuse a cached plan that was optimized for the first call's parameters.

**Symptom:** Intermittent slow DELETE/UPDATE when parameter values change significantly. Plan compiled for a selectivity of 1 row but now being used for 100K rows.

**Fix:**

```csharp
// ✅ Option A: Use WITH RECOMPILE hint
const string sql = @"
    DELETE FROM AuditLog
    WHERE CreatedAt < @Cutoff
    OPTION (RECOMPILE);";

// ✅ Option B: Use DynamicParameters with explicit DbType
// (less effective than OPTION RECOMPILE for parameter sensitivity)
```

**Cost of not fixing:** DELETE of 100K rows uses a plan optimized for 10 rows — 10,000x more logical reads than necessary.

### Spoken Answers

**Q1: What does Dapper's Execute method return, and how does it differ from Query?**

> **Average answer:** "Execute returns the number of rows affected. Query returns the actual data."

> **Great answer:** "Execute wraps ADO.NET's Command.ExecuteNonQuery and returns the number of rows affected by the DML statement — an int. For DDL like CREATE TABLE or TRUNCATE TABLE, it returns -1 because that's what ExecuteNonQuery returns for non-row-returning commands. The key difference from Query is that Execute does zero mapping — no IL Emit, no type deserialization — it's just parameter binding, execution, and returning the row count. This makes it the fastest Dapper method for any write operation. The other critical difference is parameter semantics: Execute can take a single parameter object or an IEnumerable of parameter objects. When you pass a list, Dapper loops through each item and calls ExecuteNonQuery individually — it is NOT a bulk operation. I've seen production incidents where someone passed 50,000 items to Execute and wondered why it took 45 seconds. For bulk inserts, you need SqlBulkCopy."

**Q5: How does Dapper Execute compare to EF Core SaveChanges for a batch of 1000 inserts?**

> **Average answer:** "Dapper is faster because it doesn't have change tracking overhead."

> **Great answer:** "For 1000 individual INSERTs, Dapper's Execute with a parameter collection sends 1000 individual commands — one round trip per row. At 2ms per round trip, that's 2 seconds minimum. EF Core's SaveChanges, by default, also sends individual INSERTs in 2.1 — so similar performance. However, EF Core 8+ can batch multiple INSERTs into a single round trip when you add multiple entities and call SaveChanges once — reducing 1000 round trips to ~30-50. The real winner for bulk inserts is SqlBulkCopy, which loads data in a streaming fashion and completes in ~40ms for 10K rows. For Dapper specifically, I use Execute for single-row writes or small batches (<50 rows), TVP for 50-5000 rows, and SqlBulkCopy for >5000 rows. EF Core's batch insert in 8+ is competitive for 100-500 rows but still has higher memory allocation due to change tracking."

**Q8: How do you implement a transactional Dapper write that rolls back on failure?**

> **Average answer:** "Use BeginTransaction and call Rollback in the catch block."

> **Great answer:** "The production pattern is: open the connection, call BeginTransaction, wrap all Execute calls in a try-catch, commit on success, rollback on failure, and dispose the transaction in a finally block. I use async versions everywhere: OpenAsync, BeginTransactionAsync, CommitAsync, RollbackAsync. The transaction object must be passed to every Execute call as the transaction parameter — Dapper's extension methods accept it. One common mistake is forgetting to pass the transaction to ALL Execute calls — if a call misses the transaction and fails, the earlier writes (which were enrolled) remain uncommitted and hold locks. For high-throughput scenarios, I avoid long-running transactions — keep them under 1 second. Any transaction holding locks for longer creates blocking chains. I also use TransactionScope for ambient transactions when coordinating across multiple connections or resources, but that requires MSDTC which has its own performance cost."

### Application Checklist

- [ ] For single-row writes: `ExecuteAsync` with anonymous parameter object
- [ ] For identity retrieval: `OUTPUT INSERTED.Id` with `QuerySingleAsync<int>`
- [ ] For multi-row writes: use TVP, SqlBulkCopy, or BulkExtensions — not `ExecuteAsync` with IEnumerable for > 50 rows
- [ ] For transactional writes: explicit `BeginTransactionAsync` + `CommitAsync` + `RollbackAsync`
- [ ] For optimistic concurrency: `ROWVERSION` column in WHERE clause of UPDATE
- [ ] For DDL: ignore return value (-1) — do not use it for logic
- [ ] For stored procedures: use `Query<T>` or `QueryMultiple`, not `Execute`, to get output parameters
- [ ] CancellationToken passed through CommandDefinition for all async calls
- [ ] Connection factory pattern in place — no raw new SqlConnection in application code

|What You Gain|What You Pay|
|---|---|

|No change tracking overhead|Must manage concurrency manually|
|Direct control over SQL|No LINQ compile-time safety|
|Single-row writes at ~120μs|Batch writes require extra tooling (SqlBulkCopy)|

### Scale Thresholds

- **Execute is optimal for:** single-row writes and small batches (<50 rows)
- **Execute becomes problematic at:** >100 rows per batch (loop overhead dominates)
- **Must use SqlBulkCopy at:** >5000 rows per batch

**Challenge 2 — Fix the transaction bug**

```csharp
public async Task CreateOrderWithItemsAsync(int customerId, List<OrderItem> items, CancellationToken ct)
{
    await using var connection = new SqlConnection(connectionString);
    await connection.OpenAsync(ct);

    await connection.ExecuteAsync(

```

Identify all bugs and fix them.

<details>
<summary>Solution</summary>

**Bugs:**
1. No transaction — if the second INSERT fails, the order is orphaned.
2. The first INSERT doesn't return the new OrderId, but the second INSERT needs it.
3. No `await using` for connection (only `await` on OpenAsync but the dispose pattern is correct).

**Fix:**

```csharp
public async Task<int> CreateOrderWithItemsAsync(int customerId, List<OrderItem> items, CancellationToken ct)
{
    const string orderSql = @"
        INSERT INTO Orders (CustomerId, TotalAmount)
        OUTPUT INSERTED.OrderId
        VALUES (@CustomerId, @TotalAmount);";

    const string itemSql = @"
        INSERT INTO OrderItems (OrderId, ProductId, Quantity, UnitPrice)
        VALUES (@OrderId, @ProductId, @Quantity, @UnitPrice);";

    await using var connection = _connectionFactory.Create();
    await connection.OpenAsync(ct);
    await using var tx = await connection.BeginTransactionAsync(ct);

    try
    {
        var totalAmount = items.Sum(i => i.Quantity * i.UnitPrice);
        var orderId = await connection.QuerySingleAsync<int>(
            new CommandDefinition(orderSql, new { CustomerId = customerId, TotalAmount = totalAmount },
                transaction: tx, cancellationToken: ct));

        await connection.ExecuteAsync(
            new CommandDefinition(itemSql,
                items.Select(i => new { OrderId = orderId, i.ProductId, i.Quantity, i.UnitPrice }),
                transaction: tx, cancellationToken: ct));

        await tx.CommitAsync(ct);
        return orderId;
    }
    catch
    {
        await tx.RollbackAsync(ct);
        throw;
    }
}
```

**After fix:** Atomic write — either all or nothing. OrderId is returned. Items are correctly associated.

</details>

**Challenge 3 — Fix the UPDATE concurrency bug**

```csharp
// Two users simultaneously update the same order:
// User A: UPDATE Orders SET Status = 'Shipped' WHERE OrderId = 42
// User B: UPDATE Orders SET Status = 'Cancelled' WHERE OrderId = 42
// Both succeed. Order ends up as whichever ran last.
```

Design the solution using optimistic concurrency with a RowVersion column.

<details>
<summary>Solution</summary>

**Table schema:**

```sql
CREATE TABLE Orders (
    OrderId INT IDENTITY PRIMARY KEY,
    CustomerId INT NOT NULL,
    Status NVARCHAR(20) NOT NULL DEFAULT 'Pending',
    TotalAmount DECIMAL(18,2) NOT NULL,
    RowVersion ROWVERSION NOT NULL
);
```

**Dapper implementation:**

```csharp
public class Order
{
    public int OrderId { get; set; }
    public int CustomerId { get; set; }
    public string Status { get; set; } = "Pending";
    public decimal TotalAmount { get; set; }
    public byte[]? RowVersion { get; set; }
}

public async Task<bool> UpdateOrderStatusAsync(Order order, string newStatus, CancellationToken ct)
{
    const string sql = @"
        UPDATE Orders
        SET Status = @NewStatus, ModifiedAt = SYSDATETIME()
        WHERE OrderId = @OrderId
          AND RowVersion = @OriginalRowVersion;";

    await using var connection = _connectionFactory.Create();
    await connection.OpenAsync(ct);

    var rows = await connection.ExecuteAsync(
        new CommandDefinition(sql, new
        {
            OrderId = order.OrderId,
            NewStatus = newStatus,
            OriginalRowVersion = order.RowVersion
        }, cancellationToken: ct));

    if (rows == 0)
    {
        // The RowVersion doesn't match — someone else modified this order
        _logger.LogWarning("Concurrency conflict on order {OrderId}", order.OrderId);
        // Option: re-fetch the order, present current state to user
        // Option: throw DbUpdateConcurrencyException
        throw new InvalidOperationException(
            $"Order {order.OrderId} was modified by another user. Refresh and retry.");
    }

    // Fetch the new RowVersion for subsequent updates
    var newVersion = await connection.QuerySingleAsync<byte[]>(
        "SELECT RowVersion FROM Orders WHERE OrderId = @OrderId",
        new { order.OrderId });
    order.RowVersion = newVersion;

    return true;
}
```

**After fix:** Second UPDATE fails with `rows == 0`. The application detects the concurrency conflict and can retry after re-fetching the current state.

</details>
