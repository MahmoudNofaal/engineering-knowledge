### Where This Fits

`QueryMultiple` is Dapper's solution for sending multiple SQL batches in a single round trip and reading each result set independently through a `GridReader`. A .NET backend engineer reaches for this when building dashboard summary endpoints ("give me order count, revenue total, top products, and recent orders in one call"), implementing CQRS read models that aggregate from multiple tables, or reducing N+1 round trips in high-throughput APIs. When this is unknown, teams execute separate `QueryAsync` calls for each result set — multiplying network latency by the number of queries, turning a 15ms API into a 75ms API under load. The interview signal is strong: it tests whether a candidate knows that Dapper sits directly on ADO.NET's `NextResult()` and can consume multiple result sets from a single command, something EF Core cannot do natively without raw SQL. The deeper signal is understanding `GridReader` disposal order, cursor position semantics, and the `CommandBehavior.SequentialAccess` implications.

## Core Mental Model

`QueryMultiple` sends one SQL batch containing multiple `SELECT` statements separated by semicolons and returns a single `SqlMapper.GridReader` that iterates through the result sets sequentially. The `GridReader` holds an open `SqlDataReader` positioned before the first result set. Each call to `Read<T>()` advances the `SqlDataReader` through the current result set, materializing rows into `T`, and each call to `ReadAsync<T>()` does the same asynchronously. When the current result set is fully consumed, the next call automatically advances to the next result set via `SqlDataReader.NextResultAsync()`. The invariant: **result sets must be consumed in the order they appear in the SQL batch — you cannot skip a result set or read them out of order.** The recognition pattern: a single SQL string with multiple `SELECT` statements, a single `QueryMultiple` call, and a sequence of `Read<T>` calls that correspond 1:1 with the `SELECT` statements.

### Classification

`QueryMultiple` is a **Dapper extension method on `IDbConnection`** that wraps **ADO.NET `Command.ExecuteReader()`** with `CommandBehavior.SequentialAccess` and iterates through multiple result sets via **`SqlDataReader.NextResult()`**. It belongs to the **Dapper mapping layer** — the SQL execution is pure ADO.NET; Dapper adds only the type materialization per result set. The `GridReader` is **disposable** — it must be disposed to close the underlying reader and connection. The method is **not buffered by default** — each `Read<T>()` call streams rows from the active result set. The performance characteristic is **I/O-bound per round trip**: one network call instead of N.

```mermaid
sequenceDiagram
    participant App as .NET App
    participant Dapper as Dapper<br/>SqlMapper
    participant ADO as ADO.NET<br/>SqlCommand
    participant S as SQL Server

    App->>Dapper: QueryMultipleAsync(sql, param)
    Dapper->>ADO: new SqlCommand(sql, connection)
    ADO->>S: sp_executesql N'SELECT...; SELECT...; SELECT...'
    S-->>ADO: Multiple result sets (3 result sets)
    ADO-->>Dapper: SqlDataReader positioned at Result Set 0
    Dapper-->>App: GridReader

    App->>Dapper: reader.ReadAsync<T0>()
    Dapper->>ADO: SqlDataReader.ReadAsync()<br/>consume rows from RS 0
    Dapper->>Dapper: Map rows to T0
    Dapper-->>App: IEnumerable<T0>

    App->>Dapper: reader.ReadAsync<T1>()
    Dapper->>ADO: SqlDataReader.NextResultAsync()<br/>advance to RS 1
    ADO->>S: (no network call — buffered on client)
    Dapper->>ADO: SqlDataReader.ReadAsync()<br/>consume rows from RS 1
    Dapper->>Dapper: Map rows to T1
    Dapper-->>App: IEnumerable<T1>

    App->>Dapper: reader.ReadAsync<T2>()
    Dapper->>ADO: SqlDataReader.NextResultAsync()<br/>advance to RS 2
    Dapper->>ADO: SqlDataReader.ReadAsync()
    Dapper->>Dapper: Map rows to T2
    Dapper-->>App: IEnumerable<T2>

    App->>Dapper: reader.DisposeAsync()
    Dapper->>ADO: SqlDataReader.Close()
    ADO-->>S: Connection returned to pool
```

## Deep Mechanics

### How the Engine Executes This

1. **Dapper receives the SQL and parameters.** The application calls `connection.QueryMultipleAsync(sql, param)` where `sql` is a string like `"SELECT * FROM Orders WHERE CustomerId = @Id; SELECT * FROM OrderItems WHERE OrderId IN (SELECT OrderId FROM Orders WHERE CustomerId = @Id)"`.

2. **Dapper calls `IDbCommand.ExecuteReaderAsync()`.** Dapper creates a `SqlCommand` with the provided SQL and parameters, sets `CommandType.Text`, and calls `ExecuteReaderAsync(CommandBehavior.SequentialAccess | CommandBehavior.CloseConnection)`. The `SequentialAccess` flag tells ADO.NET to stream column data sequentially — no buffering of the entire row in memory — which is critical for large BLOB columns but also affects how Dapper reads column values (forward-only, cannot seek backward within a row).

3. **SQL Server parses, compiles, and executes the batch.** The `sp_executesql` call (or simple batch execution) processes all `SELECT` statements in order. SQL Server builds a single execution plan for the batch. Each `SELECT` produces its own result set. The network protocol (TDS) streams result sets sequentially — the server sends all rows of result set 0, followed by a result set end token, then all rows of result set 1, etc.

4. **Dapper wraps the `SqlDataReader` in a `GridReader`.** The `GridReader` stores a reference to the `SqlDataReader` and tracks the current result set index (starting at 0). No rows have been read yet. The `GridReader` also stores the `CommandDefinition` including cancellation token and whether buffering is requested.

5. **The first `Read<T>()` call.** `GridReader.Read<T>()` calls `SqlDataReader.Read()` in a loop, mapping each row to `T` via Dapper's IL-generated deserializer, and adds the mapped object to a `List<T>`. When `SqlDataReader.Read()` returns `false` (no more rows in current result set), the list is complete. `Read<T>()` returns the list as `IEnumerable<T>`. The `SqlDataReader` is positioned after the last row of result set 0.

6. **The next `Read<T>()` or `ReadAsync<T>()` call.** Before reading, `GridReader` calls `SqlDataReader.NextResultAsync()` to advance to result set 1. `NextResult()` returns `false` if there are no more result sets, indicating the batch is complete. If there are more, `GridReader` resets its internal state and begins the read loop on result set 1.

7. **`GridReader.Dispose()` closes the reader.** When the `GridReader` is disposed (typically via `using`), it calls `SqlDataReader.Dispose()` which closes the connection if `CommandBehavior.CloseConnection` was set, or leaves it open if the caller managed connection lifetime.

FROM Orders

```csharp
// Dapper QueryMultipleAsync — both result sets from one round trip
public async Task<(IReadOnlyList<Order> Orders, IReadOnlyList<OrderItem> Items)>
    GetCustomerOrdersAsync(int customerId, CancellationToken ct)
{
    const string sql = @"
        SELECT OrderId, CustomerId, OrderDate, Status, TotalAmount
        FROM Orders
        WHERE CustomerId = @CustomerId
        ORDER BY OrderDate DESC;

        SELECT oi.OrderId, oi.OrderItemId, oi.ProductId, oi.Quantity, oi.UnitPrice
        FROM OrderItems oi
        INNER JOIN Orders o ON oi.OrderId = o.OrderId
        WHERE o.CustomerId = @CustomerId
        ORDER BY oi.OrderId;";

    await using var connection = _connectionFactory.Create();
    await connection.OpenAsync(ct);

    await using var reader = await connection.QueryMultipleAsync(
        new CommandDefinition(sql, new { CustomerId = customerId }, cancellationToken: ct));

    var orders = (await reader.ReadAsync<Order>(ct)).AsList();
    var items  = (await reader.ReadAsync<OrderItem>(ct)).AsList();

    return (orders, items);
}
```

### Execution Plan Analysis

For the QueryMultiple batch above:

- **Result set 0 (Orders):** `[Clustered Index Scan on PK_Orders]` → `[Sort (Order By OrderDate DESC)]` — or `[Index Seek on IX_Orders_CustomerId]` → `[Key Lookup]` → `[Sort]` if a non-clustered index on `CustomerId` exists. Without an index on `CustomerId`, full clustered index scan on Orders (all rows for that customer check).
- **Result set 1 (OrderItems):** `[Clustered Index Scan on PK_OrderItems]` → `[Nested Loops]` → `[Clustered Index Seek on PK_Orders]` — or `[Index Seek on IX_OrderItems_OrderId]` → `[Nested Loops]` → `[Clustered Index Seek on PK_Orders]`.

```
Result Set 0 plan shape:
  Index Seek (IX_Orders_CustomerId) → Key Lookup (PK_Orders) → Sort (Order By OrderDate DESC)
  Estimated Cost: ~35%  |  Logical Reads: ~15

Result Set 1 plan shape:
  Index Seek (IX_OrderItems_OrderId) → Nested Loops (Inner Join) → Index Seek (PK_Orders)
  Estimated Cost: ~65%  |  Logical Reads: ~50
```

**Without index on Orders.CustomerId:** Table Scan → Sort — logical reads jump to ~500+ for a 100K row Orders table.

### Failure Modes

- **Reading result sets out of order:** Calling `reader.Read<T1>()` then `reader.Read<T0>()` without consuming `T1` first throws `InvalidOperationException` — the `GridReader` enforces sequential consumption.
- **Disposing GridReader before consuming all result sets:** The unread result sets are discarded. The underlying `SqlDataReader` is closed. Any later `Read<T>()` call throws `ObjectDisposedException`.
- **Forgetting `await using` on the GridReader:** The reader and connection stay open until garbage collection, causing connection pool starvation.
- **Large result sets without buffering:** `Read<T>()` buffers the entire result set into memory. For 500K rows, this is ~100MB. Use unbuffered mode (`buffered: false`) and stream via `IAsyncEnumerable<T>` for large result sets.
- **MARS not being available:** QueryMultiple does not require Multiple Active Result Sets. It uses `NextResult()` which is sequential — only one result set is active at a time. MARS is only needed when you want to interleave reads from multiple result sets on the same connection, which QueryMultiple does not support.

public sealed class DashboardRepository : IDashboardRepository

    private readonly IDbConnectionFactory _connectionFactory;

    public DashboardRepository(IDbConnectionFactory connectionFactory)
    {
        _connectionFactory = connectionFactory;

    public async Task<DashboardSummary> GetDashboardSummaryAsync(
        DateTime fromDate, DateTime toDate, CancellationToken ct)
    {
        const string sql = @"
            SELECT COUNT(*) AS TotalOrders, ISNULL(SUM(TotalAmount), 0) AS TotalRevenue
            FROM Orders
            WHERE OrderDate >= @FromDate AND OrderDate < @ToDate;

            SELECT YEAR(OrderDate) AS [Year], MONTH(OrderDate) AS [Month],
                   COUNT(*) AS OrderCount, ISNULL(SUM(TotalAmount), 0) AS Revenue
            FROM Orders
            WHERE OrderDate >= @FromDate AND OrderDate < @ToDate
            GROUP BY YEAR(OrderDate), MONTH(OrderDate)
            ORDER BY [Year] DESC, [Month] DESC;

            FROM Orders o
            INNER JOIN Customers c ON o.CustomerId = c.CustomerId
            WHERE o.OrderDate >= @FromDate AND o.OrderDate < @ToDate
            ORDER BY o.OrderDate DESC;";

        await using var connection = _connectionFactory.Create();
        await connection.OpenAsync(ct);

        await using var reader = await connection.QueryMultipleAsync(
            new CommandDefinition(sql, new { FromDate = fromDate, ToDate = toDate },
                cancellationToken: ct));

        var summary       = await reader.ReadSingleAsync<DashboardSummary>(ct);
        var monthlyTrends = (await reader.ReadAsync<MonthlyTrend>(ct)).AsList();
        var recentOrders  = (await reader.ReadAsync<RecentOrderRow>(ct)).AsList();

        return new DashboardSummary(
            summary.TotalOrders,
            summary.TotalRevenue,
            monthlyTrends,
            recentOrders);
    }
}

## Gotchas and Production Pitfalls

### 1 — Disposing GridReader Before Consuming All Result Sets

**Pitfall:** The developer calls `QueryMultipleAsync`, reads one result set, and disposes the reader (or forgets to read the remaining ones).

```csharp
// ❌ Wrong: GridReader disposed after first Read; second Read throws
await using var reader = await connection.QueryMultipleAsync(sql, param);
var orders = await reader.ReadAsync<Order>(ct);
// GridReader disposed here (end of using block) — remaining result sets lost
var items = await reader.ReadAsync<OrderItem>(ct); // ❌ ObjectDisposedException
```

**Symptom:** `ObjectDisposedException` on the second `ReadAsync` call. Or, if the first `Read` is the only one consumed, the remaining result sets are silently discarded — no error but data is missing.

**Fix:**

```csharp
// ✅ Correct: consume all result sets before disposal
await using var reader = await connection.QueryMultipleAsync(sql, param);
var orders = await reader.ReadAsync<Order>(ct);
var items  = await reader.ReadAsync<OrderItem>(ct);
// Reader disposed here — all result sets consumed
```

**Cost of not fixing:** Production dashboard shows zero recent orders or zero monthly trends because the second result set was never read — a silent data loss that goes unnoticed until a user complains.

### 2 — Out-of-Order Result Set Consumption

**Pitfall:** The developer assumes result sets can be read in any order, like a dictionary.

```csharp
// ❌ Wrong: Reading result set 1 before result set 0
await using var reader = await connection.QueryMultipleAsync(sql, param);
var items  = await reader.ReadAsync<OrderItem>(ct); // ❌ InvalidOperationException
var orders = await reader.ReadAsync<Order>(ct);
```

**Symptom:** `InvalidOperationException: "The reader must be read in order — result set at index 1 cannot be consumed before result set at index 0"`.

**Fix:** Always read result sets in the exact order they appear in the SQL batch. If you need a different consumption order, reorder the `SELECT` statements in the SQL.

**Cost of not fixing:** Runtime exception in production. Crash loops on startup if consuming dashboard data.

### 3 — Not Calling NextResult Between Typed Reads (Implicit)

**Pitfall:** The developer tries to call `Read<T>` for the same result set multiple times, expecting it to return the next batch of rows.

```csharp
// ❌ Wrong: calling Read twice for same result set
var firstBatch  = await reader.ReadAsync<Order>(ct);
var secondBatch = await reader.ReadAsync<Order>(ct); // ❌ Empty — already consumed RS 0
```

**Symptom:** `secondBatch` is empty. The developer expects pagination behavior but `Read<T>` consumes the entire result set in one call (when buffered).

**Fix:** Each `Read<T>` call consumes one whole result set. If you need pagination, use `LIMIT`/`OFFSET` in SQL or use unbuffered reading with `IAsyncEnumerable<T>`.

**Cost of not fixing:** Silent data loss — second page of data never appears.

### 4 — Connection Closed Before GridReader Disposal

**Pitfall:** The developer opens the connection explicitly and closes it before the `GridReader` is disposed.

```csharp
// ❌ Wrong: connection closed before GridReader
await connection.OpenAsync(ct);
var reader = await connection.QueryMultipleAsync(sql, param);
connection.Close(); // ❌ GridReader still holds open data reader
var orders = await reader.ReadAsync<Order>(ct); // ❌ InvalidOperationException
```

**Symptom:** `InvalidOperationException: "Invalid attempt to read when data reader is closed"`.

**Fix:** Let the `GridReader` manage connection lifetime via `CommandBehavior.CloseConnection`, or dispose the connection after the reader.

```csharp
// ✅ Correct: using block on reader, connection auto-closed
await using var reader = await connection.QueryMultipleAsync(
    new CommandDefinition(sql, param, flags: CommandBehavior.CloseConnection));
var orders = await reader.ReadAsync<Order>(ct);
// Both reader and connection disposed when reader's using block exits
```

**Cost of not fixing:** Connection leak or runtime exception. Connection pool exhaustion at high throughput.

### 5 — Large Result Set Memory Blow

**Pitfall:** Using QueryMultiple to fetch a very large result set without buffering disabled.

```csharp
// ❌ Wrong: default buffered mode loads 500K rows into memory
await using var reader = await connection.QueryMultipleAsync(sql, param);
var allRows = (await reader.ReadAsync<LargeRow>(ct)).AsList();
// 500K rows × 500 bytes = 250MB in memory
```

**Symptom:** High memory usage, LOH fragmentation, GC pressure, `OutOfMemoryException` at scale.

**Fix:**

```csharp
// ✅ Correct: unbuffered mode streams rows
await using var reader = await connection.QueryMultipleAsync(
    new CommandDefinition(sql, param, flags: CommandBehavior.SequentialAccess));
var results = reader.ReadUnbufferedAsync<LargeRow>(ct);
await foreach (var row in results)
{
    Process(row); // one row at a time — memory is O(1)
}
```

**Cost of not fixing:** 250MB+ heap allocation per request at 100 concurrent requests = 25GB memory pressure, leading to OOM kills and process restarts.

### 6 — Missing SELECT in the Batch Causes Wrong Result Count

**Pitfall:** The SQL batch has N `SELECT` statements but the code makes N-1 or N+1 `Read<T>` calls.

**Symptom:** If fewer `Read<T>` calls than result sets — remaining result sets are silently discarded. If more `Read<T>` calls than result sets — the extra call returns an empty enumeration (or throws `InvalidOperationException` depending on Dapper version).

**Fix:** Maintain strict 1:1 mapping between `SELECT` statements in SQL and `Read<T>` calls in code. Document the expected result set order in comments near the SQL.

**Cost of not fixing:** Dashboard shows stale/empty data sections. Unnoticeable without monitoring.

## Performance Implications

### Benchmark: QueryMultiple vs N Separate Queries

The primary performance advantage of QueryMultiple is **network round-trip elimination**. Each `QueryAsync` call requires:
1. TCP connection acquisition from pool (sub-millisecond if pool is warm)
2. Network send of TDS pre-login + SQL batch (~1-5μs on local network, ~1-5ms across datacenters)
3. SQL Server parse/compile/execute
4. Network receive of TDS response
5. Row materialization

QueryMultiple consolidates steps 1-2 and 4 into a single round trip.

```sql
-- Round trips: 1 vs 3
-- Query 1:
SELECT COUNT(*) AS TotalOrders FROM Orders;
-- Query 2:
SELECT SUM(TotalAmount) AS TotalRevenue FROM Orders;
-- Query 3:
SELECT TOP 10 * FROM Orders ORDER BY OrderDate DESC;
```

**Logical reads comparison:**

|Approach|Logical Reads|Network Round Trips|Latency (100Mbps, 1ms RTT)|
|---|---|---|---|
|3x QueryAsync|~150|3|~5ms + overhead|
|1x QueryMultiple|~150|1|~2ms + overhead|

### Spoken Answers

**Q1: What is Dapper's QueryMultiple and what problem does it solve?**

> **Average answer:** "QueryMultiple lets you run multiple SQL queries in one go and get multiple result sets back. It returns a GridReader that you can call Read on to get each set."

> **Great answer:** "QueryMultiple is a Dapper extension method that sends multiple SELECT statements in a single SQL batch to the database and returns a GridReader that wraps ADO.NET's SqlDataReader. The problem it solves is network round-trip overhead — instead of N separate QueryAsync calls that each require a TCP send, server parse, and receive cycle, you get all result sets in one round trip. For a dashboard that needs four aggregated queries, this turns four 2ms round trips into one 3ms round trip. Under the hood, Dapper calls SqlCommand.ExecuteReaderAsync, SQL Server processes the batch sequentially and streams each result set back through TDS. The GridReader then calls SqlDataReader.NextResult between each typed Read to advance to the next result set. The critical constraint is that you must consume result sets in order — no skipping or reordering. This is a common gotcha that junior developers hit."

**Q5: How does QueryMultiple compare to EF Core's approach for multiple result sets?**

> **Average answer:** "EF Core doesn't have QueryMultiple. You have to use FromSqlRaw or run separate queries."

> **Great answer:** "EF Core has no built-in equivalent to Dapper's QueryMultiple. The closest options are: (1) executing raw SQL with FromSqlRaw for a single result set, (2) running multiple separate queries in a transaction, or (3) using a third-party package like EFCore.BulkExtensions. EF Core's LINQ pipeline generates one SELECT tree per query and sends them individually. With Dapper's QueryMultiple, you write a single SQL batch like 'SELECT SUM(Amount) FROM Orders; SELECT COUNT(*) FROM Orders;' and get both aggregates back in one round trip. The tradeoff is that QueryMultiple requires raw SQL — you lose compile-time safety and LINQ composability. In practice, I use QueryMultiple for read-model materialization in CQRS handlers where performance matters and the SQL is stable, and I use EF Core for transactional writes where I want change tracking. The interview signal is that EF Core abstracts away the database too much — candidates who only know EF Core often don't know about result sets at all."

**Q8: How do you use QueryMultiple in a CQRS read model with Dapper?**

> **Average answer:** "You create a repository method that calls QueryMultiple and maps the result sets to your read model DTOs."

> **Great answer:** "In a CQRS architecture, QueryMultiple is ideal for read model materialization in query handlers. For example, a GetDashboardQuery handler needs order counts, revenue totals, monthly trends, and recent orders — four separate result sets. With QueryMultiple, the handler sends one SQL batch, calls Read four times, and returns a DashboardSummary value object. This gives you an API response in 3-5ms instead of 15-20ms with separate queries. The key production considerations are: (1) register the connection factory as a singleton and the repository as scoped, (2) open the connection explicitly in the handler with await using, (3) never let the GridReader outlive its using block, (4) index all the WHERE and ORDER BY columns used in the batch. I also disable auto-buffering for any result set that could return thousands of rows by using the buffered: false parameter and streaming through IAsyncEnumerable<T>. This keeps memory constant regardless of result set size."

### Interview Trigger

The interviewer asks: "How do you get data from multiple tables in a single database round trip with Dapper?" A candidate who says "JOIN" gives a surface answer. The follow-up is: "What if the result sets have different shapes — one aggregate row, one list of recent records — and you need each independently?" A candidate who knows QueryMultiple will answer. The deeper follow-up is: "What happens inside GridReader.Read — does it call NextResult automatically?" The candidate who can explain the sequential consumption model and the underlying SqlDataReader.NextResult call demonstrates true understanding.

| | Dapper QueryMultiple | EF Core (Multiple FromSqlRaw) | N Separate ADO.NET Commands |
|---|---|---|---|
| Round trips | 1 | N | N |
| Result set ordering | Sequential, must read in order | N/A (one result set per call) | Sequential, must read in order |

### Application Checklist

- [ ] The multiple result sets have incompatible column shapes (cannot be combined with JOIN)
- [ ] The round-trip latency is significant enough to warrant consolidation (>1ms per call)
- [ ] Each result set is independently useful — no cross-row filtering needed
- [ ] The SQL batch total is comprehensible and maintainable (<100 lines per result set)
- [ ] The connection factory pattern is in place — no new SqlConnection scattered in code
- [ ] Indexes exist on every WHERE, ORDER BY, and JOIN column for all SELECT statements
- [ ] All result sets are consumed before the GridReader is disposed
- [ ] CancellationToken is passed through CommandDefinition for all async reads
- [ ] Large result sets use unbuffered mode to prevent OOM
- [ ] The GridReader lifecycle is managed with `await using`

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Single round trip instead of N|Must consume result sets in order|
|Consistent read snapshot (same transaction)|Cannot interleave reads from different result sets|
|Reduced connection pool pressure|More complex SQL (multiple SELECT statements)|

### Scale Thresholds

- **Relevant when** serving 3+ separate data shapes in one API response (dashboard, profile page)

- **Required when** API response must be <10ms p99 and the data comes from 3+ aggregates or lists
- **Not needed when** data can be combined with a single JOIN or UNION and requires no post-processing

8. Critical when round-trip latency dominates response time. This is typically >100 RPS across datacenters (>2ms RTT) or >500 RPS with N≥3 queries where the combined latency exceeds 5ms. For a local network (0.3ms RTT) with 2 queries, the savings are ~0.3ms — noticeable at 1000+ RPS.

**Challenge 1 — Write the Dapper QueryMultiple**

You are building a customer profile endpoint that needs: (1) the customer's basic info, (2) their 5 most recent orders with item counts, (3) their total lifetime spend. Write the Dapper code using `QueryMultiple` in a single round trip. Assume `Customers`, `Orders`, and `OrderItems` tables. The SQL must use `@CustomerId` as the parameter.

<details>
<summary>Solution</summary>

```csharp
public async Task<CustomerProfile> GetCustomerProfileAsync(int customerId, CancellationToken ct)
{
    const string sql = @"
        SELECT CustomerId, FullName, Email, CreatedAt
        FROM Customers
        WHERE CustomerId = @CustomerId;

        SELECT TOP 5 o.OrderId, o.OrderDate, o.Status, o.TotalAmount,
               (SELECT COUNT(*) FROM OrderItems oi WHERE oi.OrderId = o.OrderId) AS ItemCount
        FROM Orders o
        WHERE o.CustomerId = @CustomerId
        ORDER BY o.OrderDate DESC;

        SELECT ISNULL(SUM(TotalAmount), 0) AS LifetimeSpend
        FROM Orders
        WHERE CustomerId = @CustomerId;";

    await using var connection = _connectionFactory.Create();
    await connection.OpenAsync(ct);

    await using var reader = await connection.QueryMultipleAsync(
        new CommandDefinition(sql, new { CustomerId = customerId }, cancellationToken: ct));

    var customer     = await reader.ReadSingleAsync<CustomerBasic>(ct);
    var recentOrders = (await reader.ReadAsync<RecentOrder>(ct)).AsList();
    var spend        = await reader.ReadSingleAsync<LifetimeSpend>(ct);

    return new CustomerProfile
    {
        Customer = customer,
        RecentOrders = recentOrders,
        LifetimeSpend = spend.LifetimeSpend
    };
}

public record CustomerBasic(int CustomerId, string FullName, string Email, DateTime CreatedAt);
public record RecentOrder(int OrderId, DateTime OrderDate, string Status, decimal TotalAmount, int ItemCount);
public record LifetimeSpend(decimal LifetimeSpend);
public record CustomerProfile
{
    public CustomerBasic? Customer { get; init; }
    public IReadOnlyList<RecentOrder> RecentOrders { get; init; } = Array.Empty<RecentOrder>();
    public decimal LifetimeSpend { get; init; }
}
```

**Logical reads:** ~10 (Customers: 1 seek, Orders: 1 seek + 1 scan for COUNT, OrderItems: 1 seek per order) \
**Execution plan:** [Index Seek on PK_Customers] → [Index Seek on IX_Orders_CustomerId] → [Stream Aggregate for COUNT] → [Stream Aggregate for SUM] \
**EF Core equivalent:** 4 separate LINQ queries — impossible to combine into one round trip without raw SQL.

</details>

**Challenge 2 — Fix the QueryMultiple bug**

```csharp
await using var connection = new SqlConnection(connectionString);
var reader = await connection.QueryMultipleAsync("SELECT * FROM Customers; SELECT * FROM Orders;");
var orders = await reader.ReadAsync<Order>();  // throws
```

What is the bug? Fix the code. Explain what exception occurs.

<details>
<summary>Solution</summary>

**Root cause:** The result sets are consumed out of order. The SQL batch has `SELECT * FROM Customers` first (result set 0), but the code reads `Order` first (expecting result set 0 to be Orders). This throws `InvalidOperationException`.

**Fix:**

```csharp
await using var connection = new SqlConnection(connectionString);
await connection.OpenAsync();
await using var reader = await connection.QueryMultipleAsync(
    "SELECT * FROM Customers; SELECT * FROM Orders;");
var customers = await reader.ReadAsync<Customer>(); // result set 0
var orders    = await reader.ReadAsync<Order>();    // result set 1
```

**Alternatively**, swap the SELECT statements to match the expected consumption order:

```csharp
await using var reader = await connection.QueryMultipleAsync(
    "SELECT * FROM Orders; SELECT * FROM Customers;");
var orders    = await reader.ReadAsync<Order>();
var customers = await reader.ReadAsync<Customer>();
```

**After fix — logical reads:** Same as before, but no exception.

**Challenge 4 — Performance: QueryMultiple vs JOIN**

You need Order + OrderItems data. One developer uses QueryMultiple, another uses a JOIN. The API returns `{ order, items[] }`. Compare the two approaches in terms of logical reads, memory, and execution time.

<details>
<summary>Solution</summary>

**QueryMultiple approach:**

```sql
SELECT OrderId, CustomerId, OrderDate, TotalAmount
FROM Orders WHERE OrderId = @OrderId;

SELECT OrderItemId, ProductId, Quantity, UnitPrice
FROM OrderItems WHERE OrderId = @OrderId;
```

- Logical reads: ~2 (1 seek on Orders PK, 1 seek on OrderItems IX_OrderId)
- Memory: Order object + List of OrderItem objects (separate allocations)
- Network: 1 round trip, ~400 bytes total

**JOIN approach:**

```sql
SELECT o.OrderId, o.CustomerId, o.OrderDate, o.TotalAmount,
       oi.OrderItemId, oi.ProductId, oi.Quantity, oi.UnitPrice
FROM Orders o
INNER JOIN OrderItems oi ON o.OrderId = oi.OrderId
WHERE o.OrderId = @OrderId;
```

- Logical reads: ~2 (same seeks)
- Memory: Duplicate Order data for each OrderItem row (wasted memory, more GC pressure)
- Network: 1 round trip, more bytes due to duplicated Order columns per row
- Post-processing: Must group in application code

**Winner:** `QueryMultiple` when the parent data is large (many columns) and children are many (duplication cost is high). JOIN wins when you need all columns in one flat DTO (simpler code).

**At scale (100 RPS, 5 items per order):** JOIN approach sends ~30% more data over the network due to duplicated parent columns. QueryMultiple also avoids the client-side grouping logic.

</details>

**Challenge 5 — Design the connection management**

Given a high-throughput API (1000 RPS) that uses QueryMultiple for dashboard data, design the connection management strategy: factory pattern, pool size, command timeout, and error handling with retry.

<details>
<summary>Solution</summary>

```csharp
// Connection factory — singleton, registered once
public sealed class SqlConnectionFactory : IDbConnectionFactory
{
    private readonly string _connectionString;
    public SqlConnectionFactory(string connectionString) => _connectionString = connectionString;

    public IDbConnection Create()
    {
        var connection = new SqlConnection(_connectionString);
        // Minimize pooled connection setup cost
        return connection;
    }
}

// Program.cs
builder.Services.AddSingleton<IDbConnectionFactory>(_ =>
    new SqlConnectionFactory(connectionString));

builder.Services.AddScoped<IDashboardRepository, DashboardRepository>();

// Connection string — tuned for high concurrency
// "Server=.;Database=SalesDb;Integrated Security=True;Max Pool Size=200;Min Pool Size=50;Connect Timeout=5;Command Timeout=10;"

// Repository — scoped, explicit open with retry
public sealed class DashboardRepository : IDashboardRepository
{
    private readonly IDbConnectionFactory _connectionFactory;
    private readonly ILogger<DashboardRepository> _logger;

    public DashboardRepository(IDbConnectionFactory factory, ILogger<DashboardRepository> logger)
    {
        _connectionFactory = factory;
        _logger = logger;
    }

    public async Task<DashboardSummary> GetDashboardSummaryAsync(
        DateTime fromDate, DateTime toDate, CancellationToken ct)
    {
        const string sql = @"..."; // QueryMultiple SQL

        // Retry policy: 3 attempts with exponential backoff for transient failures
        var attempt = 0;
        while (true)
        {
            try
            {
                await using var connection = _connectionFactory.Create();
                await connection.OpenAsync(ct);

                await using var reader = await connection.QueryMultipleAsync(
                    new CommandDefinition(sql, new { FromDate = fromDate, ToDate = toDate },
                        commandTimeout: 10, cancellationToken: ct));

                var totalOrders   = await reader.ReadSingleAsync<int>(ct);
                var monthlyTrends = (await reader.ReadAsync<MonthlyTrend>(ct)).AsList();
                var recentOrders  = (await reader.ReadAsync<RecentOrderRow>(ct)).AsList();

                return new DashboardSummary(totalOrders, 0, monthlyTrends, recentOrders);
            }
            catch (SqlException ex) when (IsTransient(ex) && attempt < 3)
            {
                attempt++;
                _logger.LogWarning(ex, "Transient error on attempt {Attempt}", attempt);
                await Task.Delay(TimeSpan.FromMilliseconds(100 * Math.Pow(2, attempt)), ct);
            }
        }
    }

    private static bool IsTransient(SqlException ex) => ex.Number switch
    {
        -2 or 1205 or 1222 or 4060 or 10928 or 10929 or 10053 or 10054 or 10060 => true,
        _ => false
    };
}
```

**Pool configuration reasoning:**
- `Max Pool Size = 200`: For 1000 RPS with 50ms average query time, we need ~50 concurrent connections. 200 gives headroom for spikes.
- `Min Pool Size = 50`: Pre-warm connections to avoid connection establishment latency during traffic bursts.
- `Command Timeout = 10`: Dashboard queries must complete in <10s; longer indicates a problem.
- Retry with exponential backoff: Handles transient failures without crashing. Max 3 attempts avoids thundering herd.

</details>
