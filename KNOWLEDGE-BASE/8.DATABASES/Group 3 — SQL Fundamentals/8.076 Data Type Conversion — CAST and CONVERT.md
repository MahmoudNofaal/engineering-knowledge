### Where This Fits

CAST and CONVERT are the two functions for converting a value from one data type to another. They are fundamental to every SQL query that touches data: formatting dates for display, converting strings to numbers for arithmetic, truncating decimal values, and — most critically — ensuring type compatibility for joins and comparisons. Every .NET backend engineer uses these implicitly through EF Core's type mapping, but the engine also applies **implicit conversions** automatically when comparing columns of different types, which is the leading cause of non-SARGable predicates. The most expensive mistakes here are: implicit conversions in WHERE clauses that defeat index seeks (e.g., comparing an NVARCHAR column to a VARCHAR parameter), using CONVERT for formatting in the WHERE clause (non-SARGable), and misunderstanding the precision and scale behavior of DECIMAL conversions. Interviewers ask about CAST and CONVERT to determine whether a candidate understands the difference between explicit and implicit conversion, why implicit conversion is a performance problem, and how to detect it via execution plans.

## Core Mental Model

CAST and CONVERT are T-SQL functions that change a value's data type. **CAST** is ANSI-standard syntax: `CAST(expression AS target_type)`. **CONVERT** is T-SQL-specific and adds a **style parameter** for formatting: `CONVERT(target_type, expression [, style])`. The third parameter (style) is unique to CONVERT and provides predefined date and number formatting — DD/MM/YYYY, MM/DD/YYYY, time formats, currency format, etc. Both functions can be applied in the SELECT list (to transform output), in the WHERE clause (to filter by type), or in SET assignments (to ensure type compatibility). The critical distinction for performance is between **explicit** conversion (CAST/CONVERT that the developer writes intentionally) and **implicit** conversion (SQL Server automatically converts between types when comparing or assigning mismatched types). Implicit conversion in a WHERE clause or JOIN predicate makes the predicate **non-SARGable** — the optimizer cannot use an index seek and must scan. The type precedence rules determine which type wins in an implicit conversion: numeric types dominate string types, DATETIME dominates INT, NVARCHAR dominates VARCHAR. All conversions have a cost — CPU time for parsing and converting between representations — but the hidden cost is the lost index seek opportunity when conversion occurs on the column side of a predicate.

### Classification

This is a **scalar expression** operation in the query processing pipeline. The conversion operator appears in the execution plan as a **Compute Scalar** operator or as part of the **Index Seek/Scan predicate** evaluation.

```mermaid
flowchart TD
    A[Type conversion needed] --> B{Explicit or implicit?}
    B -->|Explicit: CAST or CONVERT| C[Developer controls: type, precision, format]
    B -->|Implicit: automatic| D{Conversion on column side?}
    D -->|Yes - WHERE col = @param of different type| E[Non-SARGable - Index Scan instead of Seek]
    D -->|No - @param side| F[SARGable - Index Seek still possible]
    C --> G{CONVERT with style?}
    G -->|Yes| H[Formatted output - VARCHAR with style code]
    G -->|No - basic type change| I[Plain type conversion]
    E --> J[Execution plan: Index Scan + Compute Scalar]
    F --> K[Execution plan: Index Seek - implicit conversion on literal/parameter]
    H --> L[Result: formatted string (e.g., '2026-06-24')]
    I --> L
```

## Deep Mechanics

### How the Engine Executes This

1. **Parsing and Binding** — The parser identifies CAST and CONVERT during algebrization. The conversion target type must be a valid system data type (INT, VARCHAR, DATETIME, DECIMAL, etc.).

2. **Type Precedence** — If the conversion is implicit, SQL Server follows type precedence: if an INT column is compared to a BIGINT parameter, the INT is promoted to BIGINT (the column is converted). If an NVARCHAR column is compared to a VARCHAR parameter, the VARCHAR parameter is promoted to NVARCHAR (the parameter is converted — the column is NOT converted, making it SARGable). The full precedence list (highest to lowest): user-defined types, SQL_VARIANT, FLOAT, DECIMAL, MONEY, BIGINT, INT, SMALLINT, TINYINT, BIT, NVARCHAR, NCHAR, VARCHAR, CHAR, DATETIMEOFFSET, DATETIME2, SMALLDATETIME, DATETIME, DATE, TIME, FLOAT (again), REAL.

3. **Conversion Execution** — The Compute Scalar operator (or the storage engine during predicate evaluation) performs the conversion. For numeric conversions, this is a CPU-bound operation (parsing string digits to binary representation, or converting between numeric precisions). For date conversions, the engine must parse the date string according to the session's date format or the CONVERT style code.

4. **Style Application** — CONVERT with a style code applies predefined formatting rules. For example, style 101 = 'MM/DD/YYYY', style 120 = 'YYYY-MM-DD HH:MI:SS(24h)', style 126 = ISO 8601 'YYYY-MM-DDThh:mi:ss.mmm'. The style is applied as a post-processing step after the base type conversion.

5. **Implicit Conversion in Predicates** — When a column is compared to a parameter of a different type, the optimizer generates a plan that converts the column value for every row. The `[Index Scan]` operator shows a `CONVERT_IMPLICIT` warning in its predicate. This is visible in the execution plan as a warning icon on the SELECT operator and in `sys.dm_db_index_usage_stats` as increased scan counts.

### SQL Visibility

```sql
-- CAST: explicit conversion
SELECT
    CAST(o.OrderId AS BIGINT) AS BigOrderId,
    CAST(o.TotalAmount AS DECIMAL(10,2)) AS Amount,
    CAST(o.OrderDate AS VARCHAR(10)) AS DateString
FROM dbo.Orders AS o;

-- CONVERT: with style for date formatting
SELECT
    CONVERT(VARCHAR(10), o.OrderDate, 101) AS US_Format,       -- 06/24/2026
    CONVERT(VARCHAR(10), o.OrderDate, 103) AS UK_Format,       -- 24/06/2026
    CONVERT(VARCHAR(23), o.OrderDate, 126) AS ISO_Format,      -- 2026-06-24T00:00:00.000
    CONVERT(VARCHAR(8), o.OrderDate, 112) AS YYYYMMDD           -- 20260624
FROM dbo.Orders AS o;

-- CONVERT: string to date with style
SELECT
    CONVERT(DATETIME2(0), '2026-06-24', 120) AS ParsedDate,    -- YYYY-MM-DD HH:MI:SS
    CONVERT(DATE, '06/24/2026', 101) AS ParsedUSDate,          -- MM/DD/YYYY
    CONVERT(DATE, '24/06/2026', 103) AS ParsedUKDate;          -- DD/MM/YYYY

-- CONVERT: money/currency format
SELECT CONVERT(VARCHAR(20), o.TotalAmount, 1) AS CurrencyFormat
FROM dbo.Orders AS o;  -- $149.99

-- Round-trip: number to string to number
SELECT
    CONVERT(INT, CONVERT(VARCHAR(10), o.TotalAmount)) AS RoundedDown
FROM dbo.Orders AS o;
```

### Execution Plan Analysis

**SELECT with CAST (no WHERE):**

- Plan: `[Clustered Index Scan] → [Compute Scalar] → [SELECT]`
- The Compute Scalar operator performs the CAST on each row.
- Cost: ~0.000001 per row (negligible CPU).

**SELECT with implicit conversion in WHERE:**

```
Explicit type match (no conversion):
WHERE OrderDate >= @StartDate
[Clustered Index Seek] → [SELECT]
Logical Reads: ~10 (seek)

Implicit conversion (NVARCHAR column, VARCHAR param):
WHERE CustomerName = @Name  -- CustomerName is NVARCHAR, @Name is VARCHAR
[Clustered Index Scan] → [SELECT]
Logical Reads: ~12,000 (full scan)
Warning: CONVERT_IMPLICIT(NVARCHAR(50),[dbo].[Orders].[CustomerName],0)
```

### Cost Visibility

```sql
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

-- SARGable: direct type match
SELECT OrderId, Status
FROM dbo.Orders
WHERE OrderDate >= '2026-06-01';
-- Table 'Orders'. Scan count 1, logical reads 12, physical reads 0
-- (Index Seek on IX_Orders_OrderDate)

-- Non-SARGable: CONVERT on column
SELECT OrderId, Status
FROM dbo.Orders
WHERE CONVERT(VARCHAR(10), OrderDate, 120) >= '2026-06-01';
-- Table 'Orders'. Scan count 1, logical reads 12,000, physical reads 0
-- (Clustered Index Scan — CONVERT on column defeats index)

-- Implicit conversion: comparing NVARCHAR to VARCHAR
-- (Assuming CustomerName is NVARCHAR(50))
SELECT OrderId, CustomerId
FROM dbo.Orders  o
INNER JOIN dbo.Customers c ON o.CustomerId = c.CustomerId
WHERE c.CustomerName = 'Acme Corp';  -- literal is VARCHAR
-- If CustomerName is NVARCHAR, literal 'Acme Corp' is implicitly converted to NVARCHAR
-- (on the literal side — SARGable)
```

### Failure Modes

**Arithmetic overflow on CAST:** `CAST(99999999999 AS INT)` — the value exceeds INT range (−2,147,483,648 to 2,147,483,647). Error 8115: "Arithmetic overflow error converting expression to data type int."

**Loss of precision:** `CAST(123.456 AS DECIMAL(5,2))` — truncates to 123.46 (rounded). Silent data loss.

**Invalid date string:** `CAST('2026-13-01' AS DATE)` — month 13 is invalid. Error 241: "Conversion failed when converting date and/or time from character string."

**Implicit conversion in JOIN:** Joining an INT column to a VARCHAR column forces the engine to convert all rows — no index seek possible.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Pattern 1: Safe number truncation with CAST
-- ============================================================
SELECT
    o.OrderId,
    o.TotalAmount,
    CAST(o.TotalAmount AS DECIMAL(10,0)) AS RoundedDollars,
    CAST(o.TotalAmount AS DECIMAL(10,1)) AS OneDecimal
FROM dbo.Orders AS o;

-- ============================================================
-- Pattern 2: Date formatting for display
-- ============================================================
SELECT
    o.OrderId,
    CONVERT(VARCHAR(10), o.OrderDate, 101) AS US_Date,        -- 06/24/2026
    CONVERT(VARCHAR(10), o.OrderDate, 103) AS UK_Date,        -- 24/06/2026
    CONVERT(VARCHAR(10), o.OrderDate, 120) AS ISO_Date,       -- 2026-06-24
    CONVERT(VARCHAR(23), o.OrderDate, 126) AS ISO8601,        -- 2026-06-24T00:00:00
    CONVERT(VARCHAR(8),  o.OrderDate, 112) AS SortableDate    -- 20260624
FROM dbo.Orders AS o;

-- ============================================================
-- Pattern 3: Safe string-to-date parsing
-- ============================================================
SELECT
    CONVERT(DATETIME2(0), '2026-06-24 14:30:00', 120) AS ParsedISO,
    CONVERT(DATE, '06/24/2026', 101) AS ParsedUS,
    CONVERT(DATE, '24/06/2026', 103) AS ParsedUK,
    CONVERT(DATETIME2(0), '20260624 14:30:00', 112) AS ParsedSortable;

-- ============================================================
-- Pattern 4: Removing implicit conversion in WHERE
-- ============================================================
-- ❌ Bad: implicit conversion on column
-- WHERE CONVERT(VARCHAR(10), OrderDate, 120) >= '2026-06-01'

-- ✅ Good: SARGable range predicate
WHERE OrderDate >= '2026-06-01' AND OrderDate < '2026-07-01';

-- ❌ Bad: comparing NVARCHAR column to VARCHAR literal
-- WHERE CustomerName = 'Acme Corp'  -- if CustomerName is NVARCHAR

-- ✅ Good: prefix N to make it NVARCHAR
WHERE CustomerName = N'Acme Corp';

-- ============================================================
-- Pattern 5: Converting for JOIN compatibility
-- ============================================================
-- ❌ Bad: mismatched types in JOIN
-- SELECT *
-- FROM dbo.Orders o
-- INNER JOIN dbo.LegacyOrders lo ON o.OrderId = CAST(lo.OrderId AS INT)
-- -- CAST on LegacyOrders.OrderId prevents index seek

-- ✅ Good: convert the legacy column type once (in staging)
-- ALTER TABLE dbo.LegacyOrders ALTER COLUMN OrderId INT NOT NULL;

-- ============================================================
-- Pattern 6: TRY_CAST and TRY_CONVERT (SQL Server 2012+)
-- ============================================================
SELECT
    TRY_CAST('abc' AS INT) AS InvalidCast,        -- Returns NULL, not error
    TRY_CONVERT(DATE, '2026-13-01') AS InvalidDate, -- Returns NULL
    TRY_CONVERT(DECIMAL(10,2), '') AS EmptyString; -- Returns NULL

-- ============================================================
-- Pattern 7: Binary/hex conversion
-- ============================================================
SELECT
    CONVERT(VARBINARY(8), 10248) AS BinaryValue,           -- INT to binary
    CONVERT(INT, 0x00002808) AS IntBack,                    -- Binary to INT
    CONVERT(VARCHAR(32), HASHBYTES('SHA2_256', 'hello'), 2) AS HexString;
    -- Style 2 = hex string without 0x prefix
```

{

    public OrderRepository(IDbConnectionFactory connectionFactory)
        => _connectionFactory = connectionFactory;

    // Pattern 1: CAST in SQL
    public async Task<IReadOnlyList<OrderSummary>> GetSummariesAsync(
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            SELECT
                o.OrderId,
                o.TotalAmount,
                CAST(o.TotalAmount AS DECIMAL(10,0)) AS AmountRounded,
                CONVERT(VARCHAR(10), o.OrderDate, 101) AS USDate
            FROM dbo.Orders AS o
            WHERE o.OrderDate >= @StartDate;";

        await using var connection = _connectionFactory.Create();

        var results = await connection.QueryAsync<OrderSummary>(
            new CommandDefinition(sql,
                new { StartDate = DateTime.UtcNow.AddDays(-30) },
                cancellationToken: cancellationToken));

        return results.AsList();
    }

    // Pattern 2: TRY_CAST for safe parsing
    public async Task<decimal?> TryParseDecimalAsync(
        string value,
        CancellationToken cancellationToken = default)
    {
        await using var connection = _connectionFactory.Create();

        var result = await connection.ExecuteScalarAsync<decimal?>(
            new CommandDefinition(
                "SELECT TRY_CAST(@Value AS DECIMAL(18,2));",
                new { Value = value },
                cancellationToken: cancellationToken));

        return result;
    }

    // Pattern 3: Explicit parameter types prevent implicit conversion
    public async Task<IReadOnlyList<Order>> GetByCustomerNameAsync(
        string customerName,
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            SELECT o.*
            FROM dbo.Orders o
            INNER JOIN dbo.Customers c ON o.CustomerId = c.CustomerId
            WHERE c.CustomerName = @CustomerName;";

        await using var connection = _connectionFactory.Create();

        var results = await connection.QueryAsync<Order>(
            new CommandDefinition(sql,
                new
                {
                    CustomerName = new DbString
                    {
                        Value = customerName,
                        IsAnsi = true,       // VARCHAR instead of NVARCHAR
                        Length = 50
                    }
                },
                cancellationToken: cancellationToken));

        return results.AsList();
    }

    // Pattern 4: Date without time for range queries
    public async Task<IReadOnlyList<Order>> GetOrdersForDateAsync(
        DateTime date,
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            SELECT *
            FROM dbo.Orders
            WHERE OrderDate >= @StartDate
              AND OrderDate < @EndDate;";

        await using var connection = _connectionFactory.Create();

        var startDate = date.Date;
        var endDate = startDate.AddDays(1);

        var results = await connection.QueryAsync<Order>(
            new CommandDefinition(sql,
                new { StartDate = startDate, EndDate = endDate },
                cancellationToken: cancellationToken));

        return results.AsList();
    }
}

## Gotchas and Production Pitfalls

### Implicit Conversion on Column in WHERE Clause

**Pitfall:** Comparing a column to a parameter of a different type, where the column has the lower type precedence — causing the column to be implicitly converted.

```sql
-- ❌ CustomerName is NVARCHAR(50), parameter is VARCHAR (ANSI string)
-- The VARCHAR parameter is implicitly promoted to NVARCHAR
-- But if the column is VARCHAR and the parameter is NVARCHAR, the column IS converted
-- Assuming CustomerName is VARCHAR(50) and @Name is NVARCHAR:
-- (Dapper defaults to NVARCHAR for string parameters without DbString)
SELECT *
FROM dbo.Customers
WHERE CustomerName = @Name;  -- @Name is NVARCHAR
```

**Symptom:** The query plan shows a Clustered Index Scan instead of an Index Seek. The execution plan has a warning icon on the SELECT operator: `CONVERT_IMPLICIT(VARCHAR(50),[dbo].[Customers].[CustomerName],0)`. Logical reads jump from ~3 to ~12,000 for a 1M row table.

**Fix:**

```sql
-- ✅ Option 1: Match the parameter type to the column type
-- If CustomerName is VARCHAR(50), pass VARCHAR parameter
-- In T-SQL:
DECLARE @Name VARCHAR(50) = 'Acme Corp';
SELECT * FROM dbo.Customers WHERE CustomerName = @Name;

-- ✅ Option 2: Use explicit CAST on the parameter (SARGable)
SELECT * FROM dbo.Customers
WHERE CustomerName = CAST(@Name AS VARCHAR(50));

-- ✅ Option 3: In Dapper, use DbString with IsAnsi = true
-- new DbString { Value = name, IsAnsi = true, Length = 50 }
```

**Cost of not fixing:** An application that searches customers by name uses Dapper without `DbString`. All customer name searches scan the entire 2M row Customer table instead of seeking on the non-clustered index. The search endpoint returns in 3–5 seconds instead of < 100 ms. The P95 latency for the API is 8 seconds. The on-call engineer investigates "slow search" for 2 days before discovering the implicit conversion.

### CONVERT on Column in WHERE Clause

**Pitfall:** Using CONVERT on a column in the WHERE clause to format it for comparison — makes the predicate non-SARGable.

```sql
-- ❌ Non-SARGable: CONVERT on column
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE CONVERT(VARCHAR(10), OrderDate, 120) = '2026-06-24';
```

**Symptom:** Full clustered index scan. All 12,000 pages of the Orders table are read and evaluated. The CONVERT is applied to every row's OrderDate before the comparison.

**Fix:**

```sql
-- ✅ SARGable: range predicate
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderDate >= '2026-06-24' AND OrderDate < '2026-06-25';
-- Uses Index Seek on IX_Orders_OrderDate if it exists
```

**Cost of not fixing:** A daily report query that filters orders by date uses `CONVERT(VARCHAR(10), OrderDate, 120) = @DateString`. The query takes 12 seconds on a 5M row table. Every day at 8 AM, this report runs and spikes CPU to 100% for the duration. The monitoring dashboard shows a daily CPU spike that is dismissed as "the report is slow."

### CAST Losing Precision on DECIMAL

**Pitfall:** CAST or CONVERT truncates or rounds DECIMAL values when the target precision or scale is smaller than the source.

```sql
-- ❌ Loss of precision
DECLARE @Amount DECIMAL(18,4) = 123.4567;
SELECT
    @Amount AS Original,               -- 123.4567
    CAST(@Amount AS DECIMAL(10,2)),    -- 123.46 (rounded)
    CAST(@Amount AS DECIMAL(10,1)),    -- 123.5 (rounded)
    CAST(@Amount AS DECIMAL(10,0));    -- 124 (rounded)
```

**Symptom:** Financial calculations are off by fractions of cents. The rounding is silent — no error is raised. A total of $1,234,567.89 becomes $1,234,568.00 when aggregated. The discrepancy is discovered during month-end reconciliation.

**Fix:**

```sql
-- ✅ Specify appropriate precision and scale
DECLARE @Amount DECIMAL(18,4) = 123.4567;
SELECT
    CAST(@Amount AS DECIMAL(18,4)) AS ExactMatch,
    CAST(@Amount AS DECIMAL(18,3)) AS ThreeDecimals,   -- 123.457 (rounded)
    CAST(@Amount AS DECIMAL(18,2)) AS TwoDecimals,      -- 123.46 (rounded)
    -- Use ROUND for explicit control
    ROUND(@Amount, 2, 0) AS StandardRound,              -- 123.46 (round)
    ROUND(@Amount, 2, 1) AS TruncateRound;              -- 123.45 (truncate)
```

**Cost of not fixing:** An invoicing system stores tax amounts as DECIMAL(18,4) but the reporting layer casts to DECIMAL(10,2). Each of 50K monthly invoices is off by up to $0.01 due to rounding. The annual discrepancy is ~$600. The finance team spends 3 days each month manually reconciling the difference.

### Implicit Conversion in JOIN Predicates

**Pitfall:** Joining two tables on columns of different types — the engine implicitly converts one column, preventing index seek on that column.

```sql
-- ❌ Join on mismatched types
-- Orders.OrderId is INT, LegacyOrders.OrderId is VARCHAR(10)
SELECT o.OrderId, o.Status, lo.LegacyReference
FROM dbo.Orders o
INNER JOIN dbo.LegacyOrders lo
    ON CAST(o.OrderId AS VARCHAR(10)) = lo.OrderId;
-- CAST on Orders.OrderId prevents seek on PK_Orders

-- Even worse — implicit conversion:
-- INNER JOIN dbo.LegacyOrders lo ON o.OrderId = lo.OrderId
-- If Orders.OrderId is INT and LegacyOrders.OrderId is VARCHAR,
-- SQL Server converts INT to VARCHAR (column is converted)
```

**Symptom:** The execution plan shows a full scan on one side of the join. The implicit conversion warning appears in the plan. The query runs 10x–100x slower than expected.

**Fix:**

```sql
-- ✅ Option 1: Convert the parameter/join column only (SARGable)
SELECT o.OrderId, o.Status, lo.LegacyReference
FROM dbo.Orders o
INNER JOIN dbo.LegacyOrders lo
    ON o.OrderId = TRY_CAST(lo.OrderId AS INT);
-- This converts LegacyOrders.OrderId (hopefully with a seek on the indexed INT column)

-- ✅ Option 2: Fix the schema — alter the legacy column type
ALTER TABLE dbo.LegacyOrders ALTER COLUMN OrderId INT NOT NULL;

-- ✅ Option 3: Create a persisted computed column
ALTER TABLE dbo.LegacyOrders
    ADD OrderIdInt AS TRY_CAST(OrderId AS INT) PERSISTED;
CREATE INDEX IX_LegacyOrders_OrderIdInt ON dbo.LegacyOrders (OrderIdInt);
```

**Cost of not fixing:** A migration script joins a legacy VARCHAR OrderId column to a modern INT OrderId column. The query that should take 200 ms takes 45 seconds. The migration ETL that runs nightly now takes 3 hours instead of 20 minutes, and the window for the next day's operations is missed twice a week.

### Using CAST/CONVERT on an Indexed Column in SELECT Does NOT Cause a Scan

**Pitfall:** Developers sometimes avoid CAST in SELECT lists because they think it defeats indexing. It does NOT — only CAST/CONVERT in WHERE or JOIN predicates affects SARGability.

```sql
-- ✅ CAST in SELECT is harmless for index usage
SELECT
    CAST(o.OrderId AS BIGINT) AS BigOrderId,
    CAST(o.TotalAmount AS DECIMAL(10,2)) AS Amount
FROM dbo.Orders AS o
WHERE o.OrderId = 10248;
-- Index Seek on PK_Orders works fine — CAST is in SELECT, not WHERE
```

**Symptom:** Developers unnecessarily avoid CAST in SELECT lists, using client-side conversion instead, which may materialize more data or require extra processing.

**Fix:**

```sql
-- ✅ CAST in SELECT is fine — use it freely
-- ❌ Avoid CAST in WHERE on the column side
```

**Cost of not fixing:** Overly cautious developers avoid CAST entirely. The application layer receives raw DATETIME2 values and formats them in C#, adding client-side CPU overhead. A report that returns 100K rows formats dates client-side, adding 500 ms of application CPU time for every request.

### TRY_CAST Return Type Mismatch

**Pitfall:** Using TRY_CAST without understanding that it returns NULL on failure — the result type is the target type, but the result may be NULL even for empty strings or whitespace.

```sql
-- ❌ TRY_CAST('' AS INT) returns NULL, not 0
-- ❌ TRY_CAST('   ' AS INT) returns NULL (whitespace is not a valid number)
SELECT
    TRY_CAST('' AS INT) AS EmptyResult,      -- NULL
    TRY_CAST(' ' AS INT) AS WhitespaceResult, -- NULL
    TRY_CAST('abc' AS INT) AS InvalidResult,  -- NULL
    TRY_CAST('123' AS INT) AS ValidResult;    -- 123
```

**Symptom:** Downstream logic that expects a numeric value receives NULL. Aggregations (SUM, AVG) skip NULLs. A data import script that uses TRY_CAST to silently handle invalid input loses rows that have empty string instead of NULL.

**Fix:**

```sql
-- ✅ Use COALESCE or ISNULL to provide a default
SELECT
    COALESCE(TRY_CAST(@Input AS INT), 0) AS SafeNumber,
    ISNULL(TRY_CAST(@Input AS DECIMAL(18,2)), 0.00) AS SafeDecimal;

-- ✅ Validate input before conversion
SELECT
    CASE
        WHEN @Input IS NULL OR @Input = '' THEN 0
        WHEN ISNUMERIC(@Input) = 1 THEN CAST(@Input AS DECIMAL(18,2))
        ELSE 0
    END AS SaferNumber;
```

**Cost of not fixing:** An ETL pipeline that loads customer balance data uses TRY_CAST on a string column that occasionally contains empty strings. The affected rows (2% of 500K) get NULL balances. The aggregated totals for the month-end report are understated by $1.2M. The error is discovered during audit — 3 weeks later.

### Spoken Answers

**Q: What is an implicit conversion, and why does it cause performance problems in WHERE clauses and JOIN predicates?**

> **Great answer:** An implicit conversion occurs when SQL Server automatically converts a value from one data type to another during comparison or assignment, without an explicit CAST or CONVERT. The performance problem is this: implicit conversion follows **type precedence**. The type with lower precedence is converted to the type with higher precedence. If the column's type has lower precedence than the parameter or other column it's being compared to, the column itself gets converted. A converted column in a WHERE clause or JOIN predicate means the index on that column cannot be used — the engine must read every row, convert the column value, and then compare. For example, if a CustomerName column is VARCHAR(50) and a Dapper query passes an NVARCHAR parameter (the default for .NET strings), the VARCHAR column is implicitly converted to NVARCHAR — the index on CustomerName is useless. The execution plan shows a Clustered Index Scan instead of an Index Seek, with a `CONVERT_IMPLICIT` warning. I've seen this cut performance from 2 ms to 4,500 ms on a 2M row table. The fix is to ensure parameter types match column types exactly: use `DbString { IsAnsi = true }` in Dapper for VARCHAR columns, or specify `SqlDbType.VarChar` for parameters. In EF Core, this is less common because EF Core's type mapping usually generates matching parameter types, but it can happen with raw SQL or custom mappings.

---

**Q: How can you detect implicit conversions in an execution plan?**

> **Great answer:** There are three ways. (1) **Execution plan XML/graphical plan**: look for a warning icon (yellow triangle) on the SELECT, INSERT, UPDATE, or DELETE operator. Hover or check the properties for a `Warnings` property containing `CONVERT_IMPLICIT`. (2) **SET STATISTICS IO**: if a query that should perform an Index Seek is showing a full table scan (high logical reads), suspect implicit conversion. Compare the actual reads to the expected reads for a seek. (3) **Query plan XML**: search for `Convert` or `CONVERT_IMPLICIT` in the plan XML. The specific format in the plan is `CONVERT_IMPLICIT(target_type, [source_column], 0)`. The Constant field contains the source column. (4) **Extended Events**: capture the `query_plan_profile` event and inspect the plan XML programmatically. In practice, I use the graphical plan in SSMS or Azure Data Studio — the warning icon is the most direct indicator. The plan also shows the actual operator name: if a column should be doing an Index Seek but the plan shows a Scan with a Compute Scalar above it that contains a CONVERT_IMPLICIT, that's the smoking gun.

### Interview Trigger

The defining conversion question: "Your query joins two tables on columns of different data types. The query is slow. What do you check?" A junior candidate says "check for missing indexes." A mid-level candidate says "look at the execution plan for index scans." A senior candidate says "I immediately open the execution plan and check for CONVERT_IMPLICIT warnings on the join predicate. I check sys.dm_db_index_usage_stats to see if the index is being used for seeks or scans. I look at the data types of both join columns in sys.columns. If they don't match, I either ALTER the column to match the other type, or I use an explicit CAST on the parameter side of the predicate — never on the column side. The follow-up: "What if you can't change the column type because of legacy dependencies?" — the answer involves creating a persisted computed column with the correct type and indexing it, or using indexed views to provide a type-consistent access path.

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|CAST in SELECT: type-safe output|Negligible CPU per row|
|CONVERT with style: formatted output without client-side parsing|T-SQL specific — not portable to PostgreSQL|
|TRY_CAST: safe parsing without errors|Returns NULL silently — may mask data quality issues|
|Type-matched parameters: SARGable predicates|Requires discipline in data access layer (DbString, explicit SqlParameter types)|
|ALTER COLUMN to matching type: permanent fix|Schema change — may require downtime or migration|

### Scale Thresholds

- Implicit conversion in WHERE matters above **~10K rows** — index seek becomes scan, and scan cost dominates.
- Implicit conversion in JOIN matters above **~100K rows** on either side — Hash Join falls back to scan.
- CAST in SELECT is negligible at any scale — Compute Scalar CPU cost is < 0.1 microseconds per row.
- CONVERT with style codes adds measurable CPU above **~1M rows per query** — consider client-side formatting for very large result sets.
- TRY_CAST is ~5–10% slower than CAST due to error handling overhead — use only when invalid input is expected.

10. Explain in 60 seconds, for a senior interviewer, the complete process for diagnosing and fixing an implicit conversion that causes a query to scan instead of seek.

<details>
<summary>Answers</summary>

1. **CAST**: `CAST(expression AS target_type)` — ANSI standard, no formatting. **CONVERT**: `CONVERT(target_type, expression [, style])` — T-SQL specific, includes a third style parameter for predefined formatting (e.g., 101 = US date, 120 = ISO date). CONVERT can both change the type AND format the output (e.g., `CONVERT(VARCHAR(10), OrderDate, 101)` produces '06/24/2026').

2. An implicit conversion is an automatic type conversion that SQL Server applies when comparing or assigning values of different types. The **type precedence** list determines which type "wins": the type with higher precedence is preserved, and the lower-precedence type is converted to it. The full list (highest to lowest): user-defined types, SQL_VARIANT, FLOAT, DECIMAL, MONEY, BIGINT, INT, SMALLINT, TINYINT, BIT, NVARCHAR, NCHAR, VARCHAR, CHAR, DATETIMEOFFSET, DATETIME2, SMALLDATETIME, DATETIME, DATE, TIME, FLOAT (again), REAL. If a VARCHAR column is compared to an NVARCHAR parameter, NVARCHAR has higher precedence → the VARCHAR column is converted → no index seek.

3. When a column is implicitly converted, the engine must evaluate the conversion for every row before comparing — this is a **Compute Scalar** applied to the column before the predicate. The index is built on the unconverted column values; the conversion output cannot be sought. Detection: (a) graphical execution plan shows a warning icon (yellow triangle) on the operator; (b) plan XML contains `CONVERT_IMPLICIT(target_type, [column], 0)`; (c) `SET STATISTICS IO` shows a full table scan when an Index Seek is expected.

4. `TRY_CAST` returns **NULL** when the conversion fails, instead of raising an error. `CAST` raises error 8114 (for string-to-number) or 241 (for string-to-date) on failure. `TRY_CONVERT` is the same as `TRY_CAST` but with the CONVERT-style parameter. `TRY_CAST` is slower than `CAST` by ~5–10% because it includes error-handling logic.

5. EF Core's type mapping system generates SQL parameters with types that match the SQL column types in most cases. However, implicit conversion can still occur when: (a) using raw SQL with `FromSqlRaw` and mismatched parameter types; (b) comparing columns of different .NET types that map to different SQL types; (c) using `ExecuteSqlRaw` with inline SQL. EF Core does not automatically warn about implicit conversions — you must check the generated SQL and the execution plan.

6. In Dapper, use `DbString` with `IsAnsi = true` for VARCHAR columns, `IsAnsi = false` (default) for NVARCHAR columns:
```csharp
new DbString { Value = "Acme Corp", IsAnsi = true, Length = 50 }
```
Without `DbString`, Dapper sends all string parameters as NVARCHAR. For a VARCHAR(50) column, this causes an implicit conversion (NVARCHAR → column gets converted) which defeats the index. Always check the column's data type and match it.

7. **CAST in SELECT list**: the plan adds a `[Compute Scalar]` operator after the data access operator. The index seek/scan is not affected. Cost: negligible (~0.000001 per row). **CAST in WHERE clause (on column side)**: the plan shows a full scan instead of a seek. The `[Compute Scalar]` is applied before the predicate, and the predicate becomes a residual predicate on a scan. Cost: full table scan cost (~12,000 logical reads for a 1M row table).

8. An implicit conversion in a JOIN predicate becomes a performance problem when either table exceeds **~100K rows**. Below this, the scan cost is acceptable. Above 100K rows, the scan dominates query time. At 1M rows, an implicit conversion in a JOIN adds 5–30 seconds to query time depending on row width and hardware.

9. If JOIN column types cannot be changed (legacy system): (a) create a **persisted computed column** with the correct type: `ALTER TABLE Legacy ADD OrderIdInt AS TRY_CAST(OrderId AS INT) PERSISTED`; (b) create an index on the computed column: `CREATE INDEX IX_Legacy_OrderIdInt ON Legacy (OrderIdInt)`; (c) join on the computed column instead of the original. This avoids the implicit conversion and enables index seeks. The persisted column stores the converted value, so the conversion cost is paid once on write, not on every read.

10. "When I suspect an implicit conversion, I follow five steps. First, I check the execution plan for CONVERT_IMPLICIT warnings — the yellow triangle on the SELECT operator in SSMS. Second, I identify which columns are involved by reading the plan XML or the warning text: `CONVERT_IMPLICIT(NVARCHAR(50), [CustomerName], 0)`. Third, I check the data types in sys.columns: `SELECT name, system_type_id FROM sys.columns WHERE object_id = OBJECT_ID('Customers')`. Fourth, I check the application data access layer — Dapper sends all strings as NVARCHAR by default; EF Core's mapping is usually correct but I verify with SQL Profiler. The fix depends on the root cause: for Dapper, I add `DbString { IsAnsi = true }` for VARCHAR columns; for raw SQL, I use explicit CAST on the parameter: `WHERE col = CAST(@Param AS VARCHAR(50))`; for schema mismatches, I ALTER the column to match. The performance difference is typically 1000x+ — I've seen queries drop from 4.5 seconds to 2 ms after fixing an implicit conversion."

**Challenge 4 — Diagnose the conversion problem**

An insurance application stores policy dates as VARCHAR(10) in a legacy table. A new query joins this to a modern DATETIME2(0) column, and runs in 30 seconds:

```sql
SELECT p.PolicyId, p.PolicyDate, c.ClaimId, c.ClaimAmount
FROM dbo.LegacyPolicies p
INNER JOIN dbo.Claims c ON p.PolicyDate = c.ClaimDate
WHERE c.ClaimAmount > 5000;
```

`SET STATISTICS IO` shows a scan on both tables. Diagnose and fix.

<details>
<summary>Solution</summary>

**Root cause:** `p.PolicyDate` is VARCHAR(10) and `c.ClaimDate` is DATETIME2(0). When comparing VARCHAR to DATETIME2 in the JOIN, SQL Server must convert one type to the other. Since DATETIME2 has higher precedence, the VARCHAR column is implicitly converted to DATETIME2 — but this conversion is applied to every row of LegacyPolicies. The index on PolicyDate (VARCHAR) is useless because the converted value is not indexed. Similarly, the Claims index on ClaimDate may still be usable (since the conversion is on the other side), but the plan shows a scan on both tables because of the conversion.

```sql
-- Fix 1: Convert the VARCHAR column to DATETIME2 permanently
ALTER TABLE dbo.LegacyPolicies
    ADD PolicyDate2 AS TRY_CONVERT(DATETIME2(0), PolicyDate, 120) PERSISTED;

CREATE INDEX IX_LegacyPolicies_PolicyDate2 ON dbo.LegacyPolicies (PolicyDate2);

-- Then join on the computed column:
SELECT p.PolicyId, p.PolicyDate2 AS PolicyDate, c.ClaimId, c.ClaimAmount
FROM dbo.LegacyPolicies p
INNER JOIN dbo.Claims c ON p.PolicyDate2 = c.ClaimDate
WHERE c.ClaimAmount > 5000;

-- Fix 2: If schema change is not possible, convert the modern column to VARCHAR
-- (but this is worse — it prevents index seek on Claims)

FROM dbo.LegacyPolicies p
INNER JOIN dbo.Claims c
    ON p.PolicyDate = CONVERT(VARCHAR(10), c.ClaimDate, 120)
WHERE c.ClaimAmount > 5000;
-- This causes a scan on Claims instead of LegacyPolicies
```

**Challenge 5 — Design the data type strategy**

**Scenario:** An application ingests financial transactions from multiple regional sources. Source data comes as VARCHAR columns (TransactionDate, Amount, CurrencyCode). The target tables use DATETIME2, DECIMAL(18,4), and CHAR(3). Design an ETL pattern that safely converts these types without data loss and without implicit conversion in subsequent queries.

<details>
<summary>Solution</summary>

```sql
-- Pattern: Staging table with VARCHAR columns, then transform during insert

-- 1. Load raw data into staging (VARCHAR storage)
CREATE TABLE dbo.Transactions_Staging (
    SourceId     INT           NOT NULL,
    TrxnDate     VARCHAR(50)   NOT NULL,  -- various formats
    Amount       VARCHAR(50)   NOT NULL,
    CurrencyCode VARCHAR(10)   NOT NULL
);

-- 2. Transform with TRY_CAST and insert into target
INSERT INTO dbo.Transactions (SourceId, TransactionDate, Amount, CurrencyCode)
SELECT
    SourceId,
    -- Normalize date: try ISO first, then US, then UK format
    COALESCE(
        TRY_CONVERT(DATETIME2(0), TrxnDate, 120),
        TRY_CONVERT(DATETIME2(0), TrxnDate, 101),
        TRY_CONVERT(DATETIME2(0), TrxnDate, 103),
        TRY_CAST(TrxnDate AS DATETIME2(0))  -- last resort
    ) AS TransactionDate,
    TRY_CAST(Amount AS DECIMAL(18,4)) AS Amount,
    -- Normalize currency: uppercase, trim, take 3 chars
    UPPER(LEFT(TRIM(CurrencyCode), 3)) AS CurrencyCode
FROM dbo.Transactions_Staging
WHERE TRY_CAST(Amount AS DECIMAL(18,4)) IS NOT NULL;  -- skip bad amounts

-- 3. Create indexes for reporting queries
CREATE INDEX IX_Transactions_Date ON dbo.Transactions (TransactionDate);
CREATE INDEX IX_Transactions_Currency ON dbo.Transactions (CurrencyCode);

-- 4. Query pattern (no implicit conversion — types match)
SELECT
    TransactionDate,
    SUM(Amount) AS TotalAmount
FROM dbo.Transactions
WHERE TransactionDate >= '2026-01-01'
  AND CurrencyCode = 'USD'
GROUP BY TransactionDate;
```

**Tradeoffs:** Staging with VARCHAR accepts any input without failure. TRY_CAST rejects bad rows silently — must log rejected rows separately for data quality. The transformation adds CPU cost on insert but ensures all subsequent queries have matching types and SARGable predicates.

**Rejection logging:**
```csharp
// Log rows that failed conversion
var badRows = await connection.QueryAsync(@"
    SELECT SourceId, TrxnDate, Amount, CurrencyCode
    FROM dbo.Transactions_Staging
    WHERE TRY_CAST(Amount AS DECIMAL(18,4)) IS NULL;",
    cancellationToken);

foreach (var row in badRows)
    logger.LogWarning("Bad transaction row: {SourceId}, Date: {Date}, Amount: {Amount}",
        row.SourceId, row.TrxnDate, row.Amount);
```
