### Where This Fits

The WHERE clause is the primary filtering mechanism in SQL — it implements the relational restriction (or selection) operator, reducing the set of rows before projection, joining, or aggregation. Every .NET backend engineer writes WHERE predicates daily, and SARGability is the single most common interview performance question at the senior level. The most expensive mistake here is writing a predicate that looks correct but forces a full table scan because the optimizer cannot use an index seek — wrapping a column in a function, applying implicit conversion, or using an operator that defeats index ordering. Interviewers use WHERE clause questions to gate on whether a candidate understands predicate SARGability, index seeks vs scans, and how the query optimizer estimates cardinality from statistics. Engineers who know this topic can look at a WHERE clause and predict within 10 logical reads whether it will seek or scan.

## Core Mental Model

The WHERE clause is the relational restriction operator: it defines which rows survive into the result set by applying a predicate (a condition that evaluates to TRUE, FALSE, or UNKNOWN for each row). Logically, WHERE executes second — immediately after FROM and before GROUP BY, HAVING, SELECT, and ORDER BY. Only rows where the predicate evaluates to TRUE (not FALSE, not UNKNOWN) pass through. The critical performance concept is **SARGability** (Search ARGument ABLE): a predicate is SARGable when the column value appears alone on one side of the operator, with no functions, wrappers, or expressions obscuring it — this allows the optimizer to use a B-tree index seek. A predicate like `YEAR(o.OrderDate) = 2024` is not SARGable because the column is wrapped in a function; the optimizer cannot reverse-engineer the function to find the matching index entries, so it must scan every row. The equivalent `o.OrderDate >= '2024-01-01' AND o.OrderDate < '2025-01-01'` IS SARGable — the column appears raw on one side of `>=` and the optimizer can seek directly to the matching rows in the index.

### Classification

This is a **restriction operator** in the `WHERE` clause family. Predicates belong to several categories by how the optimizer handles them:

- **SARGable operators:** `=`, `>`, `>=`, `<`, `<=`, `BETWEEN`, `LIKE` (prefix match only — `LIKE 'ABC%'`), `IN`, `IS NULL`
- **Non-SARGable operators:** `LIKE '%ABC'`, `LIKE '%ABC%'`, `<>`, `NOT IN`, `NOT EXISTS`, `IS NOT NULL` (often scan-promoting), function-wrapped columns, column arithmetic, implicit conversions
- **SARGability depends on the left side:** `o.OrderDate >= @date` IS SARGable; `DATEADD(DAY, 1, o.OrderDate) >= @date` is NOT SARGable
- **NULL in predicates:** Any comparison with NULL using `=` or `<>` returns UNKNOWN, not FALSE — use `IS NULL` / `IS NOT NULL` explicitly

```mermaid
flowchart TD

    B -->|Yes: column = expression| C{Operator is SARGable?}
    B -->|No: function(column) or expression| D[NOT SARGABLE — full scan or index scan required]

    C -->|<>, NOT IN, LIKE suffix, NOT| F[Index Scan or Table Scan required]
```

## Deep Mechanics

### How the Engine Executes This

1. **Parsing** — The parser tokenizes the WHERE clause into a predicate tree. `WHERE o.CustomerId = 1042 AND o.Status = 'Shipped'` becomes two comparison nodes connected by a logical AND node. Operator precedence is applied: AND binds tighter than OR.

2. **Binding (Algebrizer)** — The algebrizer resolves each column reference in the predicate to its source table and validates column existence and type. Implicit conversion rules are checked at this stage — if `o.CustomerId` is `INT` and the literal `'1042'` is `VARCHAR`, the algebrizer determines which side to convert (typically the literal to the column type). This is where the engine also checks for the `IN` list expansion and subquery decorrelation.

3. **Simplification (Query Optimizer)** — The optimizer applies logical transformations: constant folding (`1=1` → TRUE), predicate pushdown (pushing WHERE into joins), OR expansion to UNION, and `IN` list normalization. If statistics show a predicate is always FALSE (impossible condition), the optimizer may replace the entire query with a `Constant Scan` producing zero rows without touching the table.

4. **Cardinality Estimation** — The optimizer uses the histogram in the statistics object for the column(s) involved to estimate how many rows pass the predicate. For `o.CustomerId = 1042`, it uses `RANGE_HI_KEY` and `EQ_ROWS` from the CustomerId statistics histogram. For a range predicate `o.OrderDate >= '2024-01-01'`, it sums `AVG_RANGE_ROWS` across the affected histogram steps. If no statistics exist, the optimizer uses a fixed estimate (e.g., "9% of rows" for a `LIKE` predicate) — often wildly wrong.

5. **Index Selection** — The optimizer evaluates which indexes can satisfy the predicate. For a SARGable predicate on a column with a non-clustered index, it considers an Index Seek. It estimates the cost: for a seek with estimated N matching rows, the B-tree descent costs ~3–4 logical reads, plus N row lookups for columns not in the index. If the estimated matching rows exceed ~1–5% of the table (the "tipping point"), the optimizer may switch to a Clustered Index Scan because sequential reads are cheaper than many random Key Lookups.

6. **Execution** — The storage engine executes the chosen plan. For an Index Seek, it navigates the B-tree from root to leaf, scans the leaf level within the seek range, and for each matching row either returns the row (covering index) or looks up the row in the clustered index (Key Lookup). For a scan, it reads every data page and evaluates the predicate as a residual on each row.

7. **Row Filtering** — For each row, the predicate is evaluated. If the result is TRUE, the row passes to the next operator. If FALSE or UNKNOWN, the row is discarded. Note: NULL comparisons (`= NULL`, `<> NULL`) return UNKNOWN, so those rows are discarded. This is why `WHERE col <> 1` does NOT return rows where col is NULL — they are filtered out by the UNKNOWN result.

### Execution Plan Analysis

**SARGable predicate with covering index:**

- `[Index Seek (NonClustered)] IX_Orders_CustomerId_OrderDate_INCL` — seeks on `CustomerId = 1042`, retrieves matching leaf rows; `OrderDate`, `Status`, `TotalAmount` come from INCLUDE
- No Key Lookup — covering index satisfies all columns
- Estimated Cost: 0.003 to 0.009 (depends on matching row count)
- Logical Reads: ~4 to ~12

**Non-SARGable equivalent — function wrap on column:**

```sql
-- Not SARGable: YEAR wraps the column
SELECT o.OrderId, o.CustomerId, o.OrderDate, o.Status
FROM dbo.Orders AS o
WHERE YEAR(o.OrderDate) = 2024;
```

- `[Clustered Index Scan]` — every row in the table must be read and `YEAR()` evaluated
- Estimated Cost: ~4.5 to ~12 (for 1M rows); Logical Reads: ~10,000+
- The optimizer cannot use an index seek on OrderDate because the column is inside a function

**SARGable range equivalent:**

```sql
-- SARGable: column stands alone on left side
SELECT o.OrderId, o.CustomerId, o.OrderDate, o.Status
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01' AND o.OrderDate < '2025-01-01';
```

- `[Index Seek (NonClustered)] IX_Orders_OrderDate` — seeks to `2024-01-01`, scans leaf until `2025-01-01`
- Estimated Cost: ~0.15 (for 80K matching rows in 1M); Logical Reads: ~300
- Plan shape: `[Index Seek] → [Key Lookup (if needed)] → [SELECT]`

```
SARGable seek plan:
[Index Seek IX_Orders_OrderDate] → [SELECT (if covering)]
OR
[Index Seek IX_Orders_OrderDate] → [Key Lookup (Clustered)] → [Nested Loops] → [SELECT]
Estimated Cost: 0.003 (point lookup) to 0.15 (range)  |  Logical Reads: ~4 to ~300

Non-SARGable scan plan:
[Clustered Index Scan] → [Filter (compute YEAR())] → [SELECT]
Estimated Cost: ~8.5  |  Logical Reads: ~11,000 (for 1M row table)
```

### Failure Modes

**Non-SARGable predicate due to function on column:** Wrapping the column in `YEAR()`, `DATEADD()`, `UPPER()`, `SUBSTRING()`, or any function prevents index seek. The optimizer must evaluate the function for every row.

**Implicit conversion:** When the column type differs from the compare value type, SQL Server converts one side. If the column is the converted side, the predicate becomes non-SARGable:
```sql
-- If o.OrderStatus is VARCHAR and the literal is a number
WHERE o.OrderStatus = 1;   -- ❌ Implicit conversion: SqlDbType.Int -> VARCHAR on the column side
-- The column is converted (VARCHAR to INT comparison), making it non-SARGable
-- Equivalent to: WHERE CAST(o.OrderStatus AS INT) = 1

-- ✅ Correct: match the literal type to the column type
WHERE o.OrderStatus = 'Shipped';
```

**OR and IN list with mixed SARGability:** An OR between a SARGable and non-SARGable predicate often makes the entire WHERE non-SARGable:
```sql
WHERE o.CustomerId = 1042 OR YEAR(o.OrderDate) = 2024;  -- ❌ OR with non-SARGable forces scan
```

Detect non-SARGable predicates with:
```sql
-- Find queries with high scan count per execution
SELECT TOP 20
    qs.total_logical_reads / qs.execution_count AS avg_logical_reads,
    qs.execution_count,
    qs.total_worker_time / qs.execution_count AS avg_cpu,
    SUBSTRING(st.text, 1, 500) AS query_text
FROM sys.dm_exec_query_stats AS qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) AS st
WHERE qs.total_logical_reads / NULLIF(qs.execution_count, 0) > 1000
ORDER BY avg_logical_reads DESC;
```

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Schema context
-- ============================================================
CREATE TABLE dbo.Orders
(
    OrderId      INT           NOT NULL IDENTITY(1,1),
    CustomerId   INT           NOT NULL,
    OrderDate    DATETIME2(0)  NOT NULL,
    Status       VARCHAR(20)   NOT NULL,
    TotalAmount  DECIMAL(18,2) NOT NULL,
    ShippingAddr NVARCHAR(500) NULL,
    Notes        NVARCHAR(MAX) NULL,
    CreatedAt    DATETIME2(0)  NOT NULL DEFAULT SYSUTCDATETIME(),
    CONSTRAINT PK_Orders PRIMARY KEY CLUSTERED (OrderId)
);

CREATE NONCLUSTERED INDEX IX_Orders_CustomerId
    ON dbo.Orders (CustomerId);

CREATE NONCLUSTERED INDEX IX_Orders_OrderDate
    ON dbo.Orders (OrderDate)
    INCLUDE (CustomerId, Status, TotalAmount);

-- ============================================================
-- Pattern 1: Point lookup — single equality predicate
-- SARGable: column alone on left side of =
-- ============================================================
SELECT o.OrderId, o.CustomerId, o.OrderDate, o.Status, o.TotalAmount
FROM dbo.Orders AS o
WHERE o.CustomerId = @CustomerId;

-- ============================================================
-- Pattern 2: Range predicate — date range
-- SARGable: column alone on left side of >= and <
-- ============================================================
SELECT o.OrderId, o.CustomerId, o.OrderDate, o.Status, o.TotalAmount
FROM dbo.Orders AS o
WHERE o.OrderDate >= @StartDate
  AND o.OrderDate <  DATEADD(DAY, 1, @EndDate);

-- ============================================================
-- Pattern 3: Multi-column predicate — composite index seek
-- SARGable: both columns in key order match index definition
-- ============================================================
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId_Status
    ON dbo.Orders (CustomerId, Status)
    INCLUDE (OrderDate, TotalAmount);

SELECT o.OrderId, o.CustomerId, o.OrderDate, o.Status, o.TotalAmount
FROM dbo.Orders AS o
WHERE o.CustomerId = @CustomerId
  AND o.Status   = 'Shipped';
-- Seek on CustomerId, then within each CustomerId subtree, seek on Status

-- ============================================================
-- Pattern 4: IN list — SARGable but may be expanded
-- SARGable: IN is equivalent to multiple ORs of equality predicates
-- ============================================================
SELECT o.OrderId, o.CustomerId, o.OrderDate, o.Status, o.TotalAmount
FROM dbo.Orders AS o
WHERE o.Status IN ('Shipped', 'Delivered', 'Processing');
-- Plan: Index Seek per IN list value, concatenated

-- ============================================================
-- Pattern 5: Prefix LIKE — SARGable for leading wildcard only
-- ============================================================
SELECT c.CustomerId, c.FirstName, c.LastName, c.Email
FROM dbo.Customers AS c
WHERE c.LastName LIKE 'Smith%';
-- SARGable: can seek to 'Smith' prefix range in index

-- ============================================================
-- Pattern 6: NULL handling — must use IS NULL / IS NOT NULL
-- ============================================================
SELECT o.OrderId, o.CustomerId, o.OrderDate
FROM dbo.Orders AS o
WHERE o.ShippingAddr IS NULL;
-- IS NULL is SARGable and can use a filtered index on IS NULL condition

-- ============================================================
-- Anti-pattern 1: Function on column — forces scan
-- ============================================================
-- ❌ WHERE YEAR(o.OrderDate) = 2024;
-- ✅ WHERE o.OrderDate >= '2024-01-01' AND o.OrderDate < '2025-01-01';

-- ============================================================
-- Anti-pattern 2: Column arithmetic — forces scan
-- ============================================================
-- ❌ WHERE o.TotalAmount + 10 > 100;
-- ✅ WHERE o.TotalAmount > 90;

-- ============================================================
-- Anti-pattern 3: LIKE with leading wildcard — forces scan
-- ============================================================
-- ❌ WHERE c.LastName LIKE '%smith%';
-- ✅ WHERE c.LastName LIKE 'smith%';  (if prefix is sufficient)
-- OR: Full-Text Search for middle-of-word matching

-- ============================================================
-- Anti-pattern 4: NOT IN / <> — usually forces scan
-- ============================================================
-- ❌ WHERE o.Status <> 'Cancelled';
-- ✅ WHERE o.Status IN ('Shipped', 'Delivered', 'Processing');
-- Or: use a filtered index with status NOT IN ('Cancelled')
```

```csharp
                sql.ToString(),
                parameters,
                cancellationToken: cancellationToken));
        return results.AsList();
```

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL: SARGability rules are the same — column must stand alone
SELECT o.order_id, o.customer_id, o.order_date
FROM orders AS o
WHERE o.customer_id = $1;             -- SARGable, parameterized

-- PostgreSQL: Date extraction CAN be SARGable with expression indexes
-- Create an index on the expression, then the function-wrapped predicate seeks
CREATE INDEX idx_orders_order_year ON orders (EXTRACT(YEAR FROM order_date));
-- Now YEAR(o.order_date) = 2024 CAN seek (on the expression index)
-- SQL Server cannot create an index on YEAR(OrderDate) directly (needs computed column)

-- PostgreSQL: ILIKE for case-insensitive prefix matching (SARGable with varchar_pattern_ops)
SELECT customer_id, last_name FROM customers
WHERE last_name ILIKE 'smith%';       -- SARGable with proper index

-- PostgreSQL: NULL comparison is the same — use IS NULL / IS NOT NULL
-- PostgreSQL also supports IS DISTINCT FROM (treats NULL as comparable value)
SELECT order_id FROM orders
WHERE shipping_addr IS DISTINCT FROM 'PO Box';
```

## Gotchas and Production Pitfalls

### Function-Wrapped Column in WHERE

**Pitfall:** Wrapping a column in a scalar function (`YEAR`, `DATEADD`, `UPPER`, `LEN`, `SUBSTRING`, `FORMAT`) in the WHERE clause, expecting the optimizer to still use an index.

```sql
-- ❌ Not SARGable — forces 11,000+ logical reads on 1M rows
SELECT o.OrderId, o.OrderDate
FROM dbo.Orders AS o
WHERE YEAR(o.OrderDate) = 2024;
```

**Symptom:** `SET STATISTICS IO` shows `logical reads 11,400` (Clustered Index Scan) instead of `~300` (Index Seek). Execution plan shows `Clustered Index Scan` with `Filter` operator computing `YEAR()` for every row.

**Fix:**

```sql
-- ✅ SARGable — uses index seek on OrderDate, reads only matching rows
SELECT o.OrderId, o.OrderDate
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01'
  AND o.OrderDate <  '2025-01-01';
```

**Cost of not fixing:** On a 10M row table with 800K rows in 2024, the scan reads all 10M rows (~114,000 logical reads) vs the seek reading only the 800K rows (~3,000 logical reads). At 500 queries/hour, this is 57M page reads/hour wasted — enough to pin CPU at 100% on a busy server.

---

### Implicit Conversion on the Column Side

**Pitfall:** Comparing a column with a value of a different data type, where the conversion is applied to the column.

```sql
-- ❌ If o.CustomerId is VARCHAR(10), the integer literal causes implicit conversion
-- The column VARCHAR is converted to INT — equivalent to CAST(o.CustomerId AS INT) = 1042
SELECT o.OrderId, o.CustomerId
FROM dbo.Orders AS o
WHERE o.CustomerId = 1042;

-- ❌ If o.OrderDate is VARCHAR (anti-pattern), the DATETIME literal converts the column
SELECT o.OrderId, o.OrderDate
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01';
```

**Symptom:** The execution plan shows a `CONVERT_IMPLICIT` warning on the Index Seek operator (hover over the seek in SSMS). The Index Seek may still appear (if the index is on the column) but the scan range is the entire index — effectively a scan. Query runs slowly despite matching an index.

**Fix:**

```sql
-- ✅ Ensure column and literal have the same type
-- If CustomerId is VARCHAR(10), use a string literal
SELECT o.OrderId, o.CustomerId
FROM dbo.Orders AS o
WHERE o.CustomerId = '1042';

-- ✅ Better: fix the table design — use correct data types
-- ALTER TABLE dbo.Orders ALTER COLUMN CustomerId INT NOT NULL;
```

**Cost of not fixing:** An implicit conversion on the column side is invisible in code review (the SQL looks correct) but forces a full scan on every query. On a 5M row table and 1,000 queries/second, this adds 30–50 ms per query and 500–1,000 logical reads per execution — 500K logical reads/second, saturating buffer pool and storage.

### LIKE with Leading Wildcard

**Pitfall:** Using `LIKE '%searchterm%'` for middle-of-word matching, expecting index usage.

```sql
-- ❌ Not SARGable — cannot seek; must scan all rows
SELECT c.CustomerId, c.FirstName, c.LastName
FROM dbo.Customers AS c
WHERE c.LastName LIKE '%smith%';
```

**Symptom:** Table Scan or Clustered Index Scan in the execution plan, with the `LIKE` predicate evaluated as a residual filter on every row. Logical reads equal to the number of pages in the entire table.

**Fix:**

```sql
-- ✅ SARGable if prefix matching is acceptable
SELECT c.CustomerId, c.FirstName, c.LastName
FROM dbo.Customers AS c
WHERE c.LastName LIKE 'smith%';

-- ✅ For full-text search: use Full-Text Search or CONTAINS
SELECT c.CustomerId, c.FirstName, c.LastName
FROM dbo.Customers AS c
WHERE CONTAINS(c.LastName, 'smith');

-- ✅ For PostgreSQL: use ILIKE with pg_trgm GiST index
-- ✅ For simple cases: use CHARINDEX (still non-SARGable but may be faster than LIKE)
```

**Cost of not fixing:** `LIKE '%term%'` on a 5M row Customers table scans every clustered index page (~60,000 logical reads) vs. `LIKE 'term%'` seeking to ~800 rows (~20 logical reads). A 3,000x difference. At 100 queries/second, this is 6M unnecessary page reads/second.

### OR Between SARGable and Non-SARGable Predicates

**Pitfall:** Combining a SARGable equality predicate with a non-SARGable predicate via OR, which can cause the optimizer to bypass the index seek for the entire WHERE clause.

```sql
-- ❌ OR with a function-wrapped column — optimizer cannot use Index Seek on CustomerId
SELECT o.OrderId, o.CustomerId, o.OrderDate
FROM dbo.Orders AS o
WHERE o.CustomerId = 1042
   OR YEAR(o.OrderDate) = 2024;
```

**Symptom:** Execution plan shows `Clustered Index Scan` with `Filter` — even though `o.CustomerId = 1042` is SARGable in isolation, the OR makes the entire predicate non-seekable because the optimizer must find rows matching either condition and there is no index that can satisfy both.

**Fix:**

```sql
-- ✅ Option 1: Split into two queries and UNION
SELECT o.OrderId, o.CustomerId, o.OrderDate
FROM dbo.Orders AS o
WHERE o.CustomerId = 1042
UNION
SELECT o.OrderId, o.CustomerId, o.OrderDate
FROM dbo.Orders AS o
WHERE YEAR(o.OrderDate) = 2024;

-- ✅ Option 2: Convert the non-SARGable predicate to a SARGable range
SELECT o.OrderId, o.CustomerId, o.OrderDate
FROM dbo.Orders AS o
WHERE o.CustomerId = 1042
   OR (o.OrderDate >= '2024-01-01' AND o.OrderDate < '2025-01-01');
```

**Cost of not fixing:** A query that should seek 120 rows with 4 logical reads instead scans 11,000 logical reads. On a busy OLTP system doing 500 customer lookups/second, the scan adds 5.5M extra page reads/second — buffer pool churn causes cache eviction for all other queries.

---

### NOT IN, NOT EXISTS, and <> Operator

**Pitfall:** Using `NOT IN`, `NOT EXISTS`, or `<>` to exclude a small number of values, expecting the optimizer to use an index seek for the matching values.

```sql
-- ❌ <> typically forces a scan (cannot seek to "not equal")
SELECT o.OrderId, o.CustomerId, o.Status
FROM dbo.Orders AS o
WHERE o.Status <> 'Cancelled';
```

**Symptom:** Execution plan shows `Index Scan` (not Seek) even with an index on `Status`. The optimizer cannot seek to "everything except X" — it must scan the index and discard rows where `Status = 'Cancelled'`. Also, `<>` comparisons return UNKNOWN for NULL rows, so NULL Status values are excluded.

**Fix:**

```sql
-- ✅ Option 1: INCLUDE the desired values instead of excluding
SELECT o.OrderId, o.CustomerId, o.Status
FROM dbo.Orders AS o
WHERE o.Status IN ('Shipped', 'Delivered', 'Processing', 'Pending');

-- ✅ Option 2: Use a filtered index if the exclusion pattern is common
CREATE NONCLUSTERED INDEX IX_Orders_ActiveOrders
    ON dbo.Orders (CustomerId, OrderDate)
    INCLUDE (Status, TotalAmount)
    WHERE Status <> 'Cancelled';
-- Now the filtered index is a subset of rows — queries matching the filter use the index

-- ✅ Option 3: NOT EXISTS with a join may still scan, but the anti-semi-join is often optimal
```

**Cost of not fixing:** On a table where 99% of rows have Status = 'Active' and 1% = 'Cancelled', `<> 'Cancelled'` scans 100% of rows when it should only scan 1% (or use the filtered index on the 99%). A 100x overhead in logical reads.

### NULL Misunderstanding in WHERE Predicates

**Pitfall:** Assuming `= NULL` or `<> NULL` behaves like a normal equality comparison.

```sql
-- ❌ Both return UNKNOWN for every row — returns zero rows even when NULLs exist
SELECT o.OrderId, o.ShippingAddr
FROM dbo.Orders AS o
WHERE o.ShippingAddr = NULL;    -- Returns 0 rows (always UNKNOWN)
WHERE o.ShippingAddr <> NULL;   -- Also returns 0 rows (also UNKNOWN)
```

**Symptom:** The query returns zero rows despite the table containing NULL values in the column. No error is raised — the result set is simply empty. This is the most common NULL-related bug in SQL.

**Fix:**

```sql
-- ✅ Correct NULL comparison
SELECT o.OrderId, o.ShippingAddr
FROM dbo.Orders AS o
WHERE o.ShippingAddr IS NULL;       -- Returns rows where ShippingAddr IS NULL

SELECT o.OrderId, o.ShippingAddr
FROM dbo.Orders AS o
WHERE o.ShippingAddr IS NOT NULL;   -- Returns rows where ShippingAddr is NOT NULL

-- ✅ For three-valued logic awareness: IS NULL is the only way to test for NULL
-- NULL is not a value — it is the absence of a value
-- SQL uses three-valued logic: TRUE, FALSE, UNKNOWN
-- Any comparison with NULL using = or <> returns UNKNOWN, not FALSE or TRUE
```

**Cost of not fixing:** Silent data loss — queries that are supposed to find records with missing shipping addresses return zero rows, resulting in missed fulfillment actions, incorrect reporting, and hours of debugging time. The bug is invisible because no error occurs.

### Spoken Answers

**Q: What is SARGability, and why does it matter for query performance?**

> **Average answer:** SARGability means the query can use an index. If you wrap a column in a function, it can't use the index and runs slower.

> **Great answer:** SARGability is a property of a WHERE predicate that tells the query optimizer whether it can perform an index seek rather than an index or table scan. A predicate is SARGable when the indexed column appears by itself on one side of a comparison operator — like `OrderDate >= '2024-01-01'`. The optimizer can then use the B-tree structure to navigate directly to the matching rows: roughly 3–4 logical reads for the seek, plus one read per matching page. A predicate like `YEAR(OrderDate) = 2024` is NOT SARGable because the column is inside a function. The optimizer cannot reverse-engineer the function to find the matching index entries — there's no way to know what index keys map to `YEAR(OrderDate) = 2024` without computing the function for every row. This forces a full Clustered Index Scan. On a 1M row table with 80,000 matching rows for 2024, that's 11,400 logical reads for the scan vs 312 logical reads for the SARGable range version — a 36x difference. I always check SARGability by looking at the execution plan: if I see a Scan with a Filter operator computing a function, the predicate is non-SARGable. I fix it by rewriting so the column stands alone.

---

**Q: SARGable date range vs `YEAR()` function: compare the execution plan shapes.**

> **Average answer:** The date range uses an index seek, and the YEAR function uses a scan. The seek is faster.

> **Great answer:** The `YEAR(OrderDate) = 2024` predicate produces a `Clustered Index Scan` with a `Filter` operator that computes `YEAR(OrderDate)` for every row and discards rows where the result is not 2024. The optimizer has no choice — the predicate expression is `f(column) = constant`, and there is no function reverse-mapping in the storage engine. The estimated cost for this plan on a 1M row table is approximately 8.5 cost units, with 11,400 logical reads on a typical row size of 200 bytes per row. The SARGable range version `OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01'` produces an `Index Seek` on the `OrderDate` index. The optimizer uses the histogram statistics to estimate how many rows fall between those two dates — an 80K estimate gives it confidence that the seek is cheaper than the scan. The seek navigates the B-tree to the first 2024 entry in 3–4 reads, then scans the leaf level sequentially, reading only the pages that contain 2024 rows — approximately 312 logical reads. The plan shape is `Index Seek → SELECT` (if covering) or `Index Seek → Key Lookup → Nested Loops → SELECT` (if not covering). The key insight: the optimizer can use the histogram for cardinality estimation in the SARGable case but cannot use it in the function case, because it has no statistics on `YEAR(OrderDate)` — it falls back to a fixed 9% guess, which is statistically meaningless.

---

**Q: How does EF Core's LINQ translation handle `o.OrderDate.Year == 2024`?**

> **Average answer:** EF Core translates it to `YEAR(OrderDate) = 2024`, which doesn't use an index. You should use a date range instead.

> **Great answer:** EF Core translates `o.OrderDate.Year == 2024` to `DATEPART(year, [o].[OrderDate]) = 2024` (or `YEAR` in some providers). This wraps the column in a function, making the predicate non-SARGable — the generated SQL is `WHERE DATEPART(year, [o].[OrderDate]) = 2024`, which forces a Clustered Index Scan. The correct EF Core approach is to use a date range: `o.OrderDate >= new DateTime(2024, 1, 1) && o.OrderDate < new DateTime(2025, 1, 1)`. This generates SARGable SQL: `WHERE [o].[OrderDate] >= @__startDate AND [o].[OrderDate] < @__endDate`. I verify this by enabling EF Core's SQL logging at Information level for `Microsoft.EntityFrameworkCore.Database.Command` — I can see the generated SQL and paste it into SSMS with `SET STATISTICS IO ON` to confirm the logical reads. The EF Core team is aware of this trap but the `Year` property translation is intentional for developers who must use a function — it's up to us to write the date range form for performance.

### Interview Trigger

The most common SARGability interview question: "What's wrong with `WHERE YEAR(OrderDate) = 2024`?" — a simple-looking predicate that separates candidates who understand index internals from those who don't. The follow-up that deepens the probe: "The query still returns correct results, so why does it run 40x slower on 10M rows?" A candidate who answers "it can't use the index" understands the surface. A senior candidate names the specific execution operator (Clustered Index Scan + Filter), the logical read difference (11,400 vs 312), and the rewrite (date range). The next follow-up: "What if you can't change the query — is there a way to make `YEAR(OrderDate) = 2024` seek?" — the answer is a computed column with an index, or a date range index hint. The final separator: "What does EF Core generate for `o.OrderDate.Year == 2024`?" — knowing this requires understanding of LINQ translation and SQL verification via logging.

### Application Checklist

- [ ] Every WHERE predicate has been reviewed for SARGability — no function wraps, no column arithmetic
- [ ] Implicit conversion checked: column type matches the compare value type exactly
- [ ] `LIKE` patterns use prefix only — no leading wildcard unless full-text search is used
- [ ] `OR` between predicates does not mix SARGable and non-SARGable conditions
- [ ] `NOT IN` / `<>` usage justified — cannot be rewritten as `IN` with positive values
- [ ] NULL comparisons use `IS NULL` / `IS NOT NULL` — never `= NULL` or `<> NULL`

- [ ] Statistics on filtered columns are current (`STATS_DATE()` within 30 days for active tables)

- [ ] Parameterized queries used — no embedded constants in WHERE clauses

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Index Seek instead of Table Scan — 10–1,000x fewer logical reads|Must write predicates with column on one side — less natural for date/year extraction|
|Accurate cardinality estimation from statistics histograms|Requires understanding of operator SARGability — non-obvious for junior developers|
|Predictable query performance at scale|Some SARGable rewrites produce longer SQL (date range vs YEAR function)|
|EF Core LINQ generates SARGable SQL when used correctly|Must verify generated SQL — EF Core can translate to non-SARGable without warning|
|Covering index usage enabled|Additional non-clustered indexes increase write overhead ~30–50%|

### Scale Thresholds

- SARGability matters at **any table size above ~10K rows** — the difference between 4 reads and 200+ reads is immediate.
- Critical when table exceeds **1M rows** and the predicate runs more than **100x/hour** — scan vs seek difference becomes thousands of logical reads per execution.
- The tipping point where a seek becomes a scan is typically **1–5% of the table** — if the predicate matches more than that, the optimizer may choose a scan even for SARGable predicates (sequential reads cheaper than many random lookups).
- Implicit conversion impact becomes severe above **500K rows** — the scan of the entire index is unavoidable.

The column is NOT wrapped — the conversion is on the literal side, so the predicate REMAINS SARGable. However, if `CustomerId` is `VARCHAR` and the predicate is `WHERE CustomerId = 1042`, the column is implicitly converted to INT — that conversion wraps the column and makes the predicate NON-SARGable. **Challenge 2 — Fix the performance problem**

```sql
-- This query runs in 8 seconds on a 5M row Orders table.
-- Identify why and fix it.
SELECT o.OrderId, o.OrderDate, o.TotalAmount, o.Status
FROM dbo.Orders AS o
WHERE DATEPART(YEAR, o.OrderDate) = 2024
  AND UPPER(o.Status) = 'SHIPPED';
```

<details>
<summary>Solution</summary>

**Root cause:** Two non-SARGable predicates:
1. `DATEPART(YEAR, o.OrderDate) = 2024` — wraps the column in `DATEPART`, preventing index seek on OrderDate
2. `UPPER(o.Status) = 'SHIPPED'` — wraps Status in `UPPER()`, preventing index seek on Status

Both force a full clustered index scan (57,000 logical reads = every page in the 5M row table).

```sql
-- Fixed query — both predicates now SARGable
SELECT o.OrderId, o.OrderDate, o.TotalAmount, o.Status
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01'
  AND o.OrderDate <  '2025-01-01'
  AND o.Status = 'Shipped';
```

**Index to create:**

```sql
CREATE NONCLUSTERED INDEX IX_Orders_OrderDate_Status
    ON dbo.Orders (OrderDate, Status)
    INCLUDE (TotalAmount);
```

**After fix — logical reads:** ~350 (from 57,000 to ~350)

The plan changes from `[Clustered Index Scan] → [Filter (DATEPART + UPPER)] → [SELECT]` to `[Index Seek IX_Orders_OrderDate_Status] → [SELECT]`.

Note on `UPPER(Status)`: The fix assumes the application can control the casing of the `Status` column values. If case-insensitive comparison is required, use a case-insensitive collation (`Latin1_General_CI_AS`) instead of `UPPER()` — this makes `Status = 'Shipped'` match regardless of case without wrapping the column in a function.

</details>

**Challenge 3 — Explain the execution plan**

```sql
-- Query:
SELECT o.OrderId, o.CustomerId, o.OrderDate, o.Status
FROM dbo.Orders AS o
WHERE o.CustomerId = 1042;

-- Actual plan: [Clustered Index Scan] → [Filter] → [SELECT]
-- Despite a non-clustered index existing on CustomerId
```

Why does the optimizer choose a Clustered Index Scan instead of an Index Seek on IX_Orders_CustomerId?

<details>
<summary>Solution</summary>

**Why Clustered Index Scan:** The optimizer estimates that `CustomerId = 1042` matches a very large fraction of the table — likely a high-frequency customer with hundreds of thousands of orders. The statistics histogram for CustomerId shows this value with high `EQ_ROWS` (e.g., 800,000 of 1,000,000 rows). At 80% selectivity, an Index Seek followed by Key Lookup for 800,000 rows would require 800,000 random I/Os — far more expensive than a single sequential Clustered Index Scan. The optimizer's cost model chose the scan because sequential reads are cheaper than 800K random reads.

**Secondary factors:**
- Statistics may be stale — if CustomerId 1042 had only 100 rows when statistics were last updated but now has 800K, the optimizer uses the old estimate (100 rows) and would probably choose a seek. If it's choosing a scan, the statistics likely reflect the current distribution.
- The covering index does not exist — the non-clustered index on CustomerId does not INCLUDE OrderDate and Status, so every matching row requires a Key Lookup. Adding a covering index that includes all SELECT columns would make the Index Seek plan viable even at high row counts.

**To get an Index Seek:**
- Create a covering index: `CREATE INDEX IX_Orders_CustomerId_INCL ON dbo.Orders (CustomerId) INCLUDE (OrderDate, Status);`
- Or use a query hint (not recommended for production): `SELECT ... FROM dbo.Orders AS o WITH (INDEX(IX_Orders_CustomerId))`

**Tradeoff:** The covering index eliminates Key Lookups and makes the seek cheaper at high row counts, but adds ~2–3 pages of write overhead per INSERT/UPDATE.

</details>

**Challenge 4 — Diagnose the concurrency problem**

A stored procedure that runs every 5 minutes to process pending orders has this query:

```sql
SELECT o.OrderId, o.CustomerId, o.OrderDate, o.TotalAmount
FROM dbo.Orders AS o
WHERE o.Status = 'Pending'
  AND DATEDIFF(HOUR, o.CreatedAt, GETUTCDATE()) > 24
ORDER BY o.CreatedAt;
```

This query blocks inserts to the Orders table for 3–5 seconds, causing timeout exceptions in the application layer. The table has 50M rows, 10% of which have Status = 'Pending'. The index on Status is a non-clustered B-tree index. The `DATEDIFF` predicate evaluates `CreatedAt`, reading the entire clustered index.

<details>
<summary>Solution</summary>

**Root cause:** Two problems:

1. `DATEDIFF(HOUR, o.CreatedAt, GETUTCDATE()) > 24` is non-SARGable — wraps CreatedAt in `DATEDIFF`, forcing a Clustered Index Scan. This means the query reads millions of pages, holding schema stability locks longer and escalating to table-level locks.

2. The scan escalates locks, blocking concurrent INSERTs that need page-level locks on the clustered index.

**Detection query:**

```sql
-- Find blocking chains for this procedure
SELECT
    blocking.session_id AS blocking_session,
    blocked.session_id AS blocked_session,
    blocked.wait_type,
    blocked.wait_duration_ms,
    blocked.blocking_session_id
FROM sys.dm_exec_requests AS blocked
INNER JOIN sys.dm_exec_sessions AS blocking
    ON blocked.blocking_session_id = blocking.session_id
WHERE blocked.blocking_session_id > 0;
```

**Fix:**

```sql
-- SARGable rewrite — use a date range instead of DATEDIFF
-- This enables an Index Seek on a composite index
SELECT o.OrderId, o.CustomerId, o.OrderDate, o.TotalAmount
FROM dbo.Orders AS o
WHERE o.Status = 'Pending'
  AND o.CreatedAt < DATEADD(HOUR, -24, GETUTCDATE())
ORDER BY o.CreatedAt;

-- Create a covering filtered index (only 'Pending' rows are relevant)
CREATE NONCLUSTERED INDEX IX_Orders_Pending_CreatedAt
    ON dbo.Orders (Status, CreatedAt)
    INCLUDE (OrderId, CustomerId, OrderDate, TotalAmount)
    WHERE Status = 'Pending';
```

**After fix:** The filtered index contains only ~5M rows (10% of 50M). The SARGable predicate `CreatedAt < DATEADD(HOUR, -24, ...)` seeks into this index, reading only the matching rows (~50K-100K) rather than scanning all 50M rows. This completes in milliseconds instead of seconds, holds only row-level locks, and does not block concurrent INSERTs.

**In .NET (EF Core):**

```csharp
var pendingCutoff = DateTime.UtcNow.AddHours(-24);
var stalePendingOrders = await dbContext.Orders
    .Where(o => o.Status == "Pending" && o.CreatedAt < pendingCutoff)
    .OrderBy(o => o.CreatedAt)
    .Select(o => new OrderSummaryDto(
        o.OrderId, o.CustomerId, o.OrderDate, o.TotalAmount))
    .AsNoTracking()
    .ToListAsync(cancellationToken);
```

</details>

**Challenge 5 — Design the index**

**Scenario:** The e-commerce application has an `Invoices` table with 20M rows. The most frequent query pattern is:

```sql
SELECT i.InvoiceId, i.CustomerId, i.InvoiceDate, i.TotalAmount, i.Status
FROM dbo.Invoices AS i
WHERE i.CustomerId = @CustomerId
  AND i.Status = 'Unpaid'
  AND i.InvoiceDate >= @StartDate;
```

The query runs ~10,000 times/hour (every time a customer views their billing history). The read/write ratio is 20:1. The current execution plan shows a Clustered Index Scan with 95,000 logical reads. Design the optimal index strategy.

<details>
<summary>Solution</summary>

```sql
-- Composite filtered index: key columns in WHERE order, INCLUDE covers SELECT list
-- Filtered to only 'Unpaid' rows — reduces index size by ~70% (assuming 30% unpaid)
CREATE NONCLUSTERED INDEX IX_Invoices_CustomerId_Date_Unpaid
    ON dbo.Invoices (CustomerId, InvoiceDate DESC)
    INCLUDE (InvoiceId, TotalAmount, Status)
    WHERE Status = 'Unpaid';
```

**Why this index:**

1. **Key order:** `(CustomerId, InvoiceDate DESC)` — the WHERE clause has equality on `CustomerId` (seek to exact customer), then range on `InvoiceDate` (seek to `@StartDate` within that customer's subtree). `DESC` because the application likely shows most recent invoices first (verify with the actual ORDER BY).

2. **Filtered:** `WHERE Status = 'Unpaid'` — only about 30% of rows are unpaid, so the filtered index is ~70% smaller than an unfiltered equivalent. This means fewer leaf pages to read during the seek, less storage, and less write overhead.

3. **INCLUDE columns:** `TotalAmount` is included to make the index covering — the SELECT list has `InvoiceId`, `TotalAmount`, `Status`, and `InvoiceDate` is already the second key column. The Status INCLUDE is technically redundant (it's always 'Unpaid' due to the filter), but adding it avoids any Key Lookup.

**Tradeoffs:**

|What You Gain|What You Pay|
|---|---|
|Eliminates Clustered Index Scan — from 95K to ~15 logical reads|+3–4 pages write overhead per INSERT/UPDATE on unpaid invoices|
|Filtered index is ~70% smaller than full index|Queries for 'Paid' invoices cannot use this index|
|Seek to specific customer + date range|Index maintenance when Status changes from 'Unpaid' to 'Paid' (DELETE from index)|

**What NOT to index:** Do not create a separate index on just `CustomerId` — this composite index serves the same purpose for the single-column `CustomerId = @CustomerId` queries. Do not create a full index on `(Status, CustomerId, InvoiceDate)` unfiltered — the filtered index is smaller and more efficient for the 70% of rows that are not 'Unpaid'.

**Expected improvement:** Logical reads from ~95,000 to ~15–30 per execution. At 10,000 executions/hour, this is ~950M fewer page reads per hour — buffer pool pressure dramatically reduced.

</details>
