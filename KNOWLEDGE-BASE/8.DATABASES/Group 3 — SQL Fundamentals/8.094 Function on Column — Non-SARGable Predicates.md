### Where This Fits

Wrapping a column in a function inside the WHERE clause is the single most common SARGability mistake in production SQL. Every .NET backend engineer has written `WHERE YEAR(OrderDate) = 2024` or `WHERE UPPER(LastName) = 'SMITH'` — both are non-SARGable and force a full table scan. The database must read every row, apply the function, and then evaluate the predicate. The index on that column is useless for the seek. This is a top-3 performance antipattern in every SQL Server system with more than 100K rows. In interviews, non-SARGable predicates are a gatekeeper question: juniors don't know the term, mid-levels can define it, and seniors can rewrite on sight and explain why the optimiser cannot derive a seek range from a function-wrapped column.

## Core Mental Model

A predicate is **SARGable** (Search ARGument ABLE) when it has the form `Column op Value` — the column stands alone on one side of the operator, and the value (or expression) is on the other. The optimiser can derive a **seek range** from this form because the index B-tree stores raw column values, and the comparison `Column > Value` directly maps to a range in the B-tree key space. When a function wraps the column — `YEAR(Column) = 2024` — the optimiser cannot determine what raw column values correspond to "Year equals 2024" without examining every row. The function transforms the column value, and the index stores the pre-transformation value. The optimiser cannot invert most functions (YEAR, UPPER, SUBSTRING, DATEDIFF) to derive a seek range. The predicate is classified as a **residual filter** applied after a scan. The fix is always: rewrite the predicate so the column stands alone, moving the computation to the value side.

### Classification

Non-SARGable predicates are a **WHERE clause antipattern**. They apply to any predicate operator (`=`, `>`, `<`, `>=`, `<=`, `LIKE`, `BETWEEN`, `IN`). The predicate is **NOT SARGable** when the column appears inside any function, including scalar functions (YEAR, UPPER, SUBSTRING, DATEDIFF, CAST, CONVERT on column), string functions (LEFT, RIGHT on column), or mathematical functions (ABS, ROUND on column). The optimiser cannot use an index seek. The predicate is always applied as a scan + residual filter.

### Key Properties

|Property|Value|Notes|
|---|---|---|
|SARGable|Not SARGable|Any function wrapping the column breaks SARGability|
|Detection|Scan always — no seek possible|Execution plan shows Index Scan or Table Scan with residual predicate|
|Fix pattern|Move function to value side|`YEAR(col) = 2024` → `col >= '2024-01-01' AND col < '2025-01-01'`|
|Performance impact|10x–1000x|Seek (4 reads) vs Scan (60K reads) on large tables|
|Optimiser inversion|Cannot invert|SQL Server cannot derive seek range from function-wrapped column|
|Indexed computed column fix|Computed column + index|`YEAR(col) PERSISTED` + index on computed column — SARGable for YEAR queries|

## Deep Mechanics

### How the Engine Executes This

1. **Parsing and binding** — The parser tokenises the predicate. The algebrizer identifies that `YEAR(OrderDate)` is a scalar function applied to the column `OrderDate`. The output type (INT) is determined.

2. **Index matching** — The optimiser evaluates each index on `OrderDate`. For a predicate `YEAR(OrderDate) = 2024`, the optimiser checks if it can derive a seek predicate from `OrderDate`. The function `YEAR` cannot be inverted — the optimiser has no built-in knowledge that "Year equals 2024 means OrderDate is between 2024-01-01 and 2025-01-01." The predicate is classified as non-SARGable.

3. **Scan selection** — The optimiser selects a Clustered Index Scan or a non-clustered index scan (depending on coverage and cost). The predicate is placed in the **residual predicate** list, applied after all rows are read from the index.

4. **Residual filter execution** — The storage engine scans every page of the chosen index. For each row, the Query Processor evaluates `YEAR(OrderDate)` by extracting the year component from the datetime value. This is a CPU-intensive operation: 50M rows = 50M function evaluations.

5. **Comparison** — The result of `YEAR(OrderDate)` (an INT) is compared to `2024`. Matching rows are returned. The column's index on `OrderDate` is never used for seeking — only for scanning (if it covers the query).

6. **Execution plan shape** — The plan shows an Index Scan (or Clustered Index Scan) operator with the predicate in the "Predicate" section (not the "Seek Predicates" section). If you right-click → "Properties" → "Predicate" shows the function-wrapped expression.

```sql
(
    InvoiceId     INT            NOT NULL IDENTITY(1,1),
    OrderId       INT            NOT NULL,
    InvoiceNumber VARCHAR(20)    NOT NULL,
    InvoiceDate   DATETIME2(0)   NOT NULL,
    DueDate       DATETIME2(0)   NOT NULL,
    PaidDate      DATETIME2(0)   NULL,
    TotalAmount   DECIMAL(18,2)  NOT NULL,
    CONSTRAINT PK_Invoices PRIMARY KEY CLUSTERED (InvoiceId)
);

CREATE INDEX IX_Orders_OrderDate ON dbo.Orders (OrderDate) INCLUDE (CustomerId, TotalAmount, Status);
CREATE INDEX IX_Orders_LastName ON dbo.Orders (LastName);
CREATE INDEX IX_Orders_FirstName ON dbo.Orders (FirstName);
CREATE INDEX IX_Invoices_InvoiceDate ON dbo.Invoices (InvoiceDate);
CREATE INDEX IX_Invoices_PaidDate ON dbo.Invoices (PaidDate);

-- ============================================================
-- Pattern 1: YEAR(OrderDate) = 2024 — Classic non-SARGable
-- ============================================================
-- ❌ Non-SARGable: YEAR wraps OrderDate
SELECT o.OrderId, o.OrderDate, o.TotalAmount
FROM dbo.Orders AS o
WHERE YEAR(o.OrderDate) = 2024;
-- Execution plan: Clustered Index Scan
-- Predicate: YEAR(OrderDate) = 2024
-- Logical reads: ~60,450 (full scan of 50M rows)

-- ✅ SARGable: range predicate on OrderDate
SELECT o.OrderId, o.OrderDate, o.TotalAmount
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01'
  AND o.OrderDate < '2025-01-01';
-- Execution plan: Index Seek on IX_Orders_OrderDate
-- Seek Predicates: OrderDate >= '2024-01-01', OrderDate < '2025-01-01'
-- Logical reads: ~450 (only 2024 rows)

-- ============================================================
-- Pattern 2: MONTH(OrderDate) = 12 — Non-SARGable
-- ============================================================
-- Find orders placed in December
-- ❌ Non-SARGable
SELECT o.OrderId, o.OrderDate, o.TotalAmount
FROM dbo.Orders AS o
WHERE MONTH(o.OrderDate) = 12;

-- ✅ SARGable for a specific year
SELECT o.OrderId, o.OrderDate, o.TotalAmount
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-12-01'
  AND o.OrderDate < '2025-01-01';

-- ✅ SARGable for ALL Decembers (if index covers OrderDate)
-- This requires a range for each year — complex. Consider indexed computed column.
SELECT o.OrderId, o.OrderDate, o.TotalAmount
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2023-12-01' AND o.OrderDate < '2024-01-01'
   OR o.OrderDate >= '2024-12-01' AND o.OrderDate < '2025-01-01'
   OR o.OrderDate >= '2025-12-01' AND o.OrderDate < '2026-01-01';

-- ============================================================
-- Pattern 3: UPPER(LastName) = 'SMITH' — Non-SARGable
-- ============================================================
-- ❌ Non-SARGable: UPPER on column
SELECT o.OrderId, o.OrderDate, o.TotalAmount, o.LastName
FROM dbo.Orders AS o
WHERE UPPER(o.LastName) = 'SMITH';
-- Execution plan: Index Scan on IX_Orders_LastName
-- Predicate: UPPER(LastName) = 'SMITH'
-- Logical reads: ~60,450

-- ✅ SARGable: use case-insensitive collation
-- If column collation is CI (Case Insensitive), = is already case-insensitive
SELECT o.OrderId, o.OrderDate, o.TotalAmount, o.LastName
FROM dbo.Orders AS o
WHERE o.LastName = 'SMITH';
-- If collation is CI_AS, 'Smith', 'SMITH', 'smith' all match
-- Execution plan: Index Seek on IX_Orders_LastName
-- Logical reads: ~4

-- ✅ If collation is case-sensitive (CS), use COLLATE
SELECT o.OrderId, o.OrderDate, o.TotalAmount, o.LastName
FROM dbo.Orders AS o
WHERE o.LastName = 'SMITH' COLLATE Latin1_General_CI_AS;
-- COLLATE is on the value side — column is not converted, SARGable

-- ============================================================
-- Pattern 4: DATEDIFF(day, OrderDate, GETDATE()) > 30
-- ============================================================
-- ❌ Non-SARGable: DATEDIFF wraps OrderDate
SELECT o.OrderId, o.OrderDate, o.TotalAmount
FROM dbo.Orders AS o
WHERE DATEDIFF(day, o.OrderDate, GETDATE()) > 30;
-- Predicate: DATEDIFF(day, OrderDate, getdate()) > 30
-- Index scan: every row's OrderDate evaluated in DATEDIFF

-- ✅ SARGable: move the computation to GETDATE side
SELECT o.OrderId, o.OrderDate, o.TotalAmount
FROM dbo.Orders AS o
WHERE o.OrderDate < DATEADD(day, -30, GETDATE());
-- Seek Predicates: OrderDate < '2024-05-26 14:30:00' (evaluated once)
-- Execution plan: Index Seek on IX_Orders_OrderDate

-- ============================================================
-- Pattern 5: SUBSTRING(Code, 1, 3) = 'ABC' — Non-SARGable
-- ============================================================
-- ❌ Non-SARGable: SUBSTRING on column
SELECT o.OrderId, o.OrderDate, o.OrderCode
FROM dbo.Orders AS o
WHERE SUBSTRING(o.OrderCode, 1, 3) = 'ABC';

-- ✅ SARGable: use LIKE with prefix pattern
SELECT o.OrderId, o.OrderDate, o.OrderCode
FROM dbo.Orders AS o
WHERE o.OrderCode LIKE 'ABC%';
-- Seek Predicates: OrderCode >= 'ABC', OrderCode < 'ABD'
-- Execution plan: Index Seek

-- ============================================================
-- Pattern 6: LEFT(LastName, 2) = 'SM' — Non-SARGable
-- ============================================================
-- ❌ Non-SARGable
SELECT o.OrderId, o.LastName
FROM dbo.Orders AS o
WHERE LEFT(o.LastName, 2) = 'SM';

-- ✅ SARGable: use LIKE
SELECT o.OrderId, o.LastName
FROM dbo.Orders AS o
WHERE o.LastName LIKE 'SM%';

-- ============================================================
-- Pattern 7: CONVERT(VARCHAR, OrderDate, 112) = '202401'
-- ============================================================
-- ❌ Non-SARGable: convert on date column
SELECT o.OrderId, o.OrderDate, o.TotalAmount
FROM dbo.Orders AS o
WHERE CONVERT(VARCHAR(8), o.OrderDate, 112) = '20240101';

-- ✅ SARGable: range predicate
SELECT o.OrderId, o.OrderDate, o.TotalAmount
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01'
  AND o.OrderDate < '2024-01-02';

-- ============================================================
-- Pattern 8: CAST(OrderDate AS DATE) = '2024-01-15'
-- ============================================================
-- ❌ Non-SARGable: CAST on column
SELECT o.OrderId, o.OrderDate
FROM dbo.Orders AS o
WHERE CAST(o.OrderDate AS DATE) = '2024-01-15';

-- ✅ SARGable: range on the datetime2 column
SELECT o.OrderId, o.OrderDate
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-15'
  AND o.OrderDate < '2024-01-16';

-- ============================================================
-- Pattern 9: ABS(TotalAmount) > 1000 — Non-SARGable
-- ============================================================
-- ❌ Non-SARGable
SELECT o.OrderId, o.TotalAmount
FROM dbo.Orders AS o
WHERE ABS(o.TotalAmount) > 1000;

-- ✅ SARGable: split into two range predicates
SELECT o.OrderId, o.TotalAmount
FROM dbo.Orders AS o
WHERE o.TotalAmount > 1000
   OR o.TotalAmount < -1000;
-- Note: this is SARGable but uses OR — may still scan if not covered by index.
-- Better with a computed column if ABS queries are frequent.

-- ============================================================
-- Pattern 10: LEN(Notes) > 100 — Non-SARGable
-- ============================================================
-- ❌ Non-SARGable
SELECT o.OrderId, o.Notes
FROM dbo.Orders AS o
WHERE LEN(o.Notes) > 100;

-- ✅ Computed column fix (if Notes length filtering is common)
ALTER TABLE dbo.Orders ADD NotesLength AS LEN(Notes) PERSISTED;
CREATE INDEX IX_Orders_NotesLength ON dbo.Orders (NotesLength);
-- Now SARGable:
SELECT o.OrderId, o.Notes
FROM dbo.Orders AS o
WHERE o.NotesLength > 100;
```

```csharp
public async Task<List<Order>> GetOrdersByYearBadAsync(
    int year,
    CancellationToken cancellationToken = default)
{
    return await dbContext.Orders
        .Where(o => o.OrderDate.Year == year)      // YEAR(OrderDate) = @year
        .OrderByDescending(o => o.OrderDate)
        .ToListAsync(cancellationToken);
    // Generated: WHERE YEAR([o].[OrderDate]) = @__year_0
    // Non-SARGABLE — scan, not seek
}

// ✅ SARGABLE rewrite using range:
public async Task<List<Order>> GetOrdersByYearGoodAsync(
    int year,
    CancellationToken cancellationToken = default)
{
    var startDate = new DateTime(year, 1, 1);
    var endDate = startDate.AddYears(1);

    return await dbContext.Orders
        .Where(o => o.OrderDate >= startDate && o.OrderDate < endDate)
        .OrderByDescending(o => o.OrderDate)
        .ToListAsync(cancellationToken);
    // Generated: WHERE [o].[OrderDate] >= @__startDate_0 AND [o].[OrderDate] < @__endDate_1
    // SARGABLE — Index Seek on IX_Orders_OrderDate
}

    // Generated: WHERE [o].[LastName] LIKE @__prefix_0 + N'%'

}
```

```csharp
// Dapper — raw SQL means you control SARGability directly
public sealed class OrderRepository
{
    private readonly IDbConnectionFactory _connectionFactory;

    public OrderRepository(IDbConnectionFactory connectionFactory)
        => _connectionFactory = connectionFactory;

    // ❌ Non-SARGable — YEAR function on column
    public async Task<IReadOnlyList<Order>> GetOrdersByYearBadAsync(
        int year,
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            SELECT OrderId, OrderDate, TotalAmount, Status
            FROM dbo.Orders
            WHERE YEAR(OrderDate) = @Year;";  // Non-SARGABLE!

        await using var connection = _connectionFactory.Create();

        var results = await connection.QueryAsync<Order>(
            new CommandDefinition(sql,
                new { Year = year },
                cancellationToken: cancellationToken));

        return results.AsList();
    }

    // ✅ SARGABLE — range predicate
    public async Task<IReadOnlyList<Order>> GetOrdersByYearGoodAsync(
        int year,
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            SELECT OrderId, OrderDate, TotalAmount, Status
            FROM dbo.Orders
            WHERE OrderDate >= @StartDate
              AND OrderDate < @EndDate;";    // SARGABLE!

        await using var connection = _connectionFactory.Create();

        var results = await connection.QueryAsync<Order>(
            new CommandDefinition(sql,
                new
                {
                    StartDate = new DateTime(year, 1, 1),
                    EndDate = new DateTime(year + 1, 1, 1)
                },
                cancellationToken: cancellationToken));

        return results.AsList();
    }

    // ✅ SARGABLE — DATEDIFF rewritten as DATEADD

        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            SELECT OrderId, OrderDate, TotalAmount, Status
            FROM dbo.Orders
            WHERE OrderDate < DATEADD(day, -@Days, GETDATE());";  // SARGABLE!

        await using var connection = _connectionFactory.Create();

        var results = await connection.QueryAsync<Order>(
            new CommandDefinition(sql,
                new { Days = days },
                cancellationToken: cancellationToken));

        return results.AsList();
    }

    // ✅ SARGABLE — UPPER fix with collation
    public async Task<IReadOnlyList<Order>> GetOrdersByLastNameAsync(
        string lastName,
        CancellationToken cancellationToken = default)
    {
        const string sql = @"
            SELECT OrderId, LastName, FirstName, OrderDate
            FROM dbo.Orders
            WHERE LastName = @LastName;";  -- With CI collation, case-insensitive by default

        await using var connection = _connectionFactory.Create();

        var results = await connection.QueryAsync<Order>(
            new CommandDefinition(sql,
                new { LastName = lastName },
                cancellationToken: cancellationToken));

            new CommandDefinition(sql,
                new { Prefix = prefix },
```

```sql
  @__year_0 = 2024;
-- Plan: Clustered Index Scan, Predicate: YEAR(OrderDate) = 2024

-- ✅ SARGable: range on OrderDate
exec sp_executesql N'SELECT [o].[OrderId], [o].[OrderDate], [o].[TotalAmount]
FROM [Orders] AS [o]
WHERE [o].[OrderDate] >= @__startDate_0
  AND [o].[OrderDate] < @__endDate_1',
  N'@__startDate_0 datetime2(0),@__endDate_1 datetime2(0)',
  @__startDate_0 = '2024-01-01',
  @__endDate_1 = '2025-01-01';
-- Plan: Index Seek on IX_Orders_OrderDate

-- ❌ Non-SARGable: EF.Functions.DateDiffDay
exec sp_executesql N'SELECT [o].[OrderId], [o].[OrderDate], [o].[TotalAmount]
FROM [Orders] AS [o]
WHERE DATEDIFF(DAY, [o].[OrderDate], GETUTCDATE()) > 30',
  N'@__days_0 int',
  @__days_0 = 30;
-- Plan: Clustered Index Scan, Predicate: DATEDIFF(day, OrderDate, getutcdate()) > 30

-- ✅ SARGable: OrderDate < cutoffDate
exec sp_executesql N'SELECT [o].[OrderId], [o].[OrderDate], [o].[TotalAmount]
FROM [Orders] AS [o]
WHERE [o].[OrderDate] < @__cutoffDate_0',
  N'@__cutoffDate_0 datetime2(0)',
  @__cutoffDate_0 = '2024-05-26 14:30:00';
-- Plan: Index Seek on IX_Orders_OrderDate
```

### Execution Plan Analysis

**Non-SARGable: YEAR(OrderDate) = 2024:**

```
[Clustered Index Scan (PK_Orders)]
  Predicate: YEAR(OrderDate)=2024
  No seek predicates
  Warning: No index seek — function on column
→ [SELECT]
Estimated Cost: ~48  |  Logical Reads: ~60,450 (full scan)
```

**SARGable: OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01':**

```
[Index Seek (IX_Orders_OrderDate)]
  Seek Predicates:
    Seek Keys[0]: OrderDate >= Scalar Operator('2024-01-01')
    Seek Keys[0]: OrderDate <  Scalar Operator('2025-01-01')
  No residual predicate
→ [SELECT]
Estimated Cost: ~0.5  |  Logical Reads: ~450 (seek + range scan of 2024 pages)
```

```sql
-- Find cached queries with high logical reads that use function-on-column patterns
SELECT TOP 50
    qs.total_logical_reads,
    qs.execution_count,
    qs.total_logical_reads / qs.execution_count AS avg_logical_reads,
    SUBSTRING(st.text,
        (qs.statement_start_offset/2) + 1,
        ((CASE WHEN qs.statement_end_offset = -1
            THEN DATALENGTH(st.text)
            ELSE qs.statement_end_offset END
            - qs.statement_start_offset)/2) + 1) AS statement_text,
    qp.query_plan
FROM sys.dm_exec_query_stats AS qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) AS st
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) AS qp
WHERE qs.total_logical_reads > 10000  -- High reads = likely scan
  AND (st.text LIKE '%YEAR(%' OR st.text LIKE '%MONTH(%'
       OR st.text LIKE '%UPPER(%' OR st.text LIKE '%DATEDIFF(%'
       OR st.text LIKE '%SUBSTRING(%' OR st.text LIKE '%LEFT(%'
       OR st.text LIKE '%CONVERT(VARCHAR%' OR st.text LIKE '%CAST(% AS DATE)%')
ORDER BY qs.total_logical_reads DESC;
```

```sql
    CONSTRAINT PK_Invoices PRIMARY KEY CLUSTERED (InvoiceId)
);

-- Indexes
CREATE INDEX IX_Orders_OrderDate ON dbo.Orders (OrderDate) INCLUDE (CustomerId, TotalAmount, Status);
CREATE INDEX IX_Orders_LastName ON dbo.Orders (LastName);
CREATE INDEX IX_Orders_OrderCode ON dbo.Orders (OrderCode);
CREATE INDEX IX_Invoices_InvoiceDate ON dbo.Invoices (InvoiceDate) INCLUDE (OrderId, TotalAmount);
CREATE INDEX IX_Invoices_DueDate ON dbo.Invoices (DueDate);
CREATE INDEX IX_Invoices_PaidDate ON dbo.Invoices (PaidDate);

-- ============================================================
-- Pattern 1: Date range — YEAR/MONTH/DAY/DATEPART rewrites
-- ============================================================
-- Requirement: Find orders placed in Q1 2024
-- ❌ Non-SARGable
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE YEAR(OrderDate) = 2024
  AND MONTH(OrderDate) BETWEEN 1 AND 3;

-- ✅ SARGable — single range
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderDate >= '2024-01-01'
  AND OrderDate < '2024-04-01';

-- Requirement: Find orders placed on a specific date (regardless of time component)
-- ❌ Non-SARGable
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE CAST(OrderDate AS DATE) = '2024-06-15';

-- ✅ SARGable
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderDate >= '2024-06-15'
  AND OrderDate < '2024-06-16';

-- ============================================================
-- Pattern 2: Relative dates — DATEDIFF rewrites
-- ============================================================
-- Requirement: Orders not paid within 30 days of invoicing (overdue)
-- ❌ Non-SARGable: DATEDIFF on DueDate
SELECT i.InvoiceId, i.InvoiceNumber, i.DueDate, i.TotalAmount
FROM dbo.Invoices AS i
WHERE DATEDIFF(day, i.DueDate, GETDATE()) > 30
  AND i.PaidDate IS NULL;

-- ✅ SARGable: push computation to GETDATE side
SELECT i.InvoiceId, i.InvoiceNumber, i.DueDate, i.TotalAmount
FROM dbo.Invoices AS i
WHERE i.DueDate < DATEADD(day, -30, GETDATE())
  AND i.PaidDate IS NULL;
-- Seek Predicates: DueDate < '2024-05-26' (computed once), PaidDate IS NULL

-- Requirement: Invoices paid within 7 days of issue (early payers)
-- ❌ Non-SARGable
SELECT i.InvoiceId, i.InvoiceNumber, i.InvoiceDate, i.PaidDate
FROM dbo.Invoices AS i
WHERE DATEDIFF(day, i.InvoiceDate, i.PaidDate) <= 7;

-- ✅ SARGable (tricky — both columns in predicate):
-- Can't push to one side easily when both columns are involved
-- Fix with computed column:
ALTER TABLE dbo.Invoices ADD DaysToPay AS DATEDIFF(day, InvoiceDate, PaidDate) PERSISTED;
CREATE INDEX IX_Invoices_DaysToPay ON dbo.Invoices (DaysToPay) WHERE PaidDate IS NOT NULL;

SELECT i.InvoiceId, i.InvoiceNumber, i.InvoiceDate, i.PaidDate
FROM dbo.Invoices AS i
WHERE i.DaysToPay <= 7;
-- Seek on IX_Invoices_DaysToPay — SARGable

-- ============================================================
-- Pattern 3: Case-insensitive search — UPPER/LOWER rewrites
-- ============================================================
-- Requirement: Find customers by last name (case-insensitive)
-- ❌ Non-SARGable
SELECT OrderId, LastName, FirstName, TotalAmount
FROM dbo.Orders
WHERE UPPER(LastName) = 'SMITH';

-- ✅ SARGable: CI collation handles this
SELECT OrderId, LastName, FirstName, TotalAmount
FROM dbo.Orders
WHERE LastName = 'SMITH';
-- With Latin1_General_CI_AS (default), 'Smith', 'SMITH', 'smith' all match

-- If column has CS (case-sensitive) collation, use COLLATE on value side:
SELECT OrderId, LastName, FirstName, TotalAmount
FROM dbo.Orders
WHERE LastName = 'SMITH' COLLATE Latin1_General_CI_AS;
-- COLLATE on value side — column not converted, SARGable

-- ============================================================
-- Pattern 4: Code prefix search — SUBSTRING/LEFT rewrites
-- ============================================================
-- Requirement: Find orders whose order code starts with 'ORD-2024'
-- ❌ Non-SARGable
SELECT OrderId, OrderCode, OrderDate
FROM dbo.Orders
WHERE LEFT(OrderCode, 8) = 'ORD-2024';

-- ✅ SARGable: LIKE with prefix
SELECT OrderId, OrderCode, OrderDate
FROM dbo.Orders
WHERE OrderCode LIKE 'ORD-2024%';
-- Seek: OrderCode >= 'ORD-2024' AND OrderCode < 'ORD-2025'

-- ============================================================
-- Pattern 5: Computed column + index as permanent fix
-- ============================================================
-- If YEAR queries are frequent and rewriting every query is impractical,
-- create an indexed computed column:

ALTER TABLE dbo.Orders ADD OrderYear AS YEAR(OrderDate) PERSISTED;
GO
CREATE INDEX IX_Orders_OrderYear ON dbo.Orders (OrderYear) INCLUDE (TotalAmount, Status);
GO

-- Now this is SARGable:
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderYear = 2024;
-- Seek on IX_Orders_OrderYear — note: this is a separate index, adds write overhead

-- Similarly for UPPER searches:
ALTER TABLE dbo.Orders ADD LastNameUpper AS UPPER(LastName) PERSISTED;
GO
CREATE INDEX IX_Orders_LastNameUpper ON dbo.Orders (LastNameUpper);
GO

SELECT OrderId, LastName
FROM dbo.Orders
WHERE LastNameUpper = 'SMITH';
-- SARGable — but only useful if column collation can't be CI

-- ============================================================
-- Pattern 6: Indexed view as fix
-- ============================================================
-- For complex aggregates that involve functions:
CREATE VIEW dbo.OrdersByYear WITH SCHEMABINDING AS
SELECT YEAR(OrderDate) AS OrderYear,
       COUNT_BIG(*) AS OrderCount,
       SUM(ISNULL(TotalAmount, 0)) AS TotalRevenue
FROM dbo.Orders
GROUP BY YEAR(OrderDate);
GO

CREATE UNIQUE CLUSTERED INDEX IX_OrdersByYear ON dbo.OrdersByYear (OrderYear);
GO

-- Now queries by year are instant:
SELECT * FROM dbo.OrdersByYear WHERE OrderYear = 2024;
-- Seek on IX_OrdersByYear — materialised view, no scan of Orders
```

## Gotchas and Production Pitfalls

### YEAR(DateCol) = N — The Most Common Non-SARGable Pattern

**Pitfall:** Using `YEAR(DateColumn) = 2024` or `MONTH(DateColumn) = 1` in WHERE clauses. This is the single most common SARGability mistake. Every developer writes it at least once. The fix is deceptively simple — a range predicate — but the pattern propagates through codebases via copy-paste.

```sql
-- ❌ Non-SARGable: written by every developer at least once
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE YEAR(OrderDate) = 2024;

-- ✅ SARGable: range predicate
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01';
```

**Symptom:** A monthly report showing orders by year takes 30+ seconds on a 50M row table. The execution plan shows a Clustered Index Scan with `YEAR(OrderDate)=2024` as a residual predicate. Logical reads: 60,450.

**Fix:**

```sql
-- Compute the range boundaries once
DECLARE @Year INT = 2024;
DECLARE @StartDate DATETIME2(0) = DATEFROMPARTS(@Year, 1, 1);
DECLARE @EndDate DATETIME2(0) = DATEADD(YEAR, 1, @StartDate);

SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderDate >= @StartDate AND OrderDate < @EndDate;
```

**Cost of not fixing:** A reporting dashboard queries Orders by year for the last 5 years. Each query scans 60,450 pages. With 5 reports running simultaneously, the I/O subsystem handles 302,250 page reads. The dashboard takes 90 seconds to load. Users refresh impatiently, generating more scans. The database CPU is at 100% on YEAR evaluations for 50M rows per query.

### DATEDIFF(day, DateCol, GETDATE()) > N — The Drifting Scan

**Pitfall:** Using `DATEDIFF(day, DateCol, GETDATE()) > 30` for "older than 30 days" queries. This is non-SARGable because DATEDIFF wraps the column. The fix is `DateCol < DATEADD(day, -N, GETDATE())` — move the computation to the GETDATE side.

```sql
-- ❌ Non-SARGable: DATEDIFF on column
SELECT InvoiceId, InvoiceNumber, DueDate
FROM dbo.Invoices
WHERE DATEDIFF(day, DueDate, GETDATE()) > 30
  AND PaidDate IS NULL;

-- ✅ SARGable: DATEADD on GETDATE side
SELECT InvoiceId, InvoiceNumber, DueDate
FROM dbo.Invoices
WHERE DueDate < DATEADD(day, -30, GETDATE())
  AND PaidDate IS NULL;
```

**Symptom:** A recurring batch job that processes overdue invoices takes progressively longer as the table grows. After 12 months, it takes 8 minutes. The execution plan shows a scan with DATEDIFF as a residual filter.

**Fix:**

```sql
-- Always push the computation to the constant/GETDATE side
DECLARE @CutoffDate DATETIME2(0) = DATEADD(day, -30, GETDATE());

SELECT InvoiceId, InvoiceNumber, DueDate
FROM dbo.Invoices
WHERE DueDate < @CutoffDate
  AND PaidDate IS NULL;
```

**Cost of not fixing:** The overdue invoice job runs every hour. Each run scans 50M invoices. At 60,450 logical reads per run, 24 runs/day = 1.45M logical reads/day just for this one job. The index on DueDate exists but is never used for seeking. At 5 years of data, each scan takes longer.

### UPPER/LOWER for Case-Insensitive Search — Unnecessary Function

**Pitfall:** Using `UPPER(LastName) = 'SMITH'` when the column collation is already case-insensitive (CI). SQL Server's default collation (`Latin1_General_CI_AS`) is case-insensitive — `WHERE LastName = 'SMITH'` matches 'Smith', 'SMITH', 'smith', 'sMiTh' without any function.

```sql
-- ❌ Non-SARGable: UPPER on column, unnecessary with CI collation
SELECT OrderId, LastName, FirstName
FROM dbo.Orders
WHERE UPPER(LastName) = 'SMITH';

-- ✅ SARGable: CI collation handles case-insensitivity natively
SELECT OrderId, LastName, FirstName
FROM dbo.Orders
WHERE LastName = 'SMITH';
```

**Symptom:** A customer search form that accepts last name and searches case-insensitively. Every search scans the full table. The execution plan shows `UPPER([LastName]) = @name` as a residual predicate. The index on LastName is never sought.

**Fix:**

```sql
-- Check the column collation first
SELECT c.name, c.collation_name
FROM sys.columns c
WHERE c.object_id = OBJECT_ID('dbo.Orders') AND c.name = 'LastName';

-- If CI_AS (default), just remove UPPER:
SELECT OrderId, LastName, FirstName
FROM dbo.Orders
WHERE LastName = 'SMITH';

-- If CS (case-sensitive), add COLLATE on the value side:
SELECT OrderId, LastName, FirstName
FROM dbo.Orders
WHERE LastName = 'SMITH' COLLATE Latin1_General_CI_AS;
```

**Cost of not fixing:** A customer service application has a "search by last name" field. Every search — 10,000 searches/day — scans 50M orders. Each search takes 3 seconds. The aggregated scan traffic is 600M logical reads/day. Customer service agents wait 3 seconds per lookup, reducing their call throughput by 50%.

### SUBSTRING(Code, 1, N) = 'ABC' for Prefix Matching

**Pitfall:** Using `SUBSTRING(OrderCode, 1, 3) = 'ABC'` or `LEFT(OrderCode, 3) = 'ABC'` instead of `LIKE 'ABC%'`. The LIKE version is SARGable because the optimiser converts `LIKE 'ABC%'` to a range: `Code >= 'ABC' AND Code < 'ABD'`.

```sql
-- ❌ Non-SARGable: SUBSTRING on column
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE SUBSTRING(OrderCode, 1, 8) = 'ORD-2024';

-- ✅ SARGable: LIKE prefix
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE OrderCode LIKE 'ORD-2024%';
```

**Symptom:** A prefix search feature in the order management system takes 10 seconds. The table has 50M rows. The execution plan shows an Index Scan with SUBSTRING as the residual predicate.

**Fix:**

```sql
-- LIKE 'ABC%' is the SARGable equivalent
-- The optimiser expands it to: Code >= 'ABC' AND Code < 'ABD'
SELECT OrderId, OrderCode
FROM dbo.Orders
WHERE OrderCode LIKE 'ORD-2024%';
```

**Cost of not fixing:** An auto-complete/search feature queries orders by code prefix on every keystroke. Each query scans 60,450 pages. At 3 searches per user session and 500 concurrent users, the scan traffic saturates the I/O subsystem at peak hours.

---

### CAST(DateCol AS DATE) = '2024-01-15' — Date Truncation Pattern

**Pitfall:** Using `CAST(DateTimeCol AS DATE) = '2024-01-15'` to compare only the date portion. This wraps the column in CAST and makes the predicate non-SARGable.

```sql
-- ❌ Non-SARGable: CAST on column
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE CAST(OrderDate AS DATE) = '2024-06-15';

-- ✅ SARGable: range predicate
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderDate >= '2024-06-15'
  AND OrderDate < '2024-06-16';
```

**Symptom:** Daily order summaries query scans the full table instead of seeking the one day's data.

**Fix:**

```sql
-- Always use half-open range [start, end) for datetime2 columns
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderDate >= '2024-06-15'
  AND OrderDate < '2024-06-16';
```

**Cost of not fixing:** A daily reconciliation job queries orders for today's date. With CAST on the column, it scans all 50M orders instead of the ~10K orders from today. A 5-minute job runs for 45 minutes.

---

### LEN(Col) > 100 — Length Filter Without Index Support

**Pitfall:** Using `LEN(Notes) > 100` to find long notes. LEN on the column is non-SARGable. The fix is a computed column with the length stored and indexed.

```sql
-- ❌ Non-SARGable: LEN on column
SELECT OrderId, Notes
FROM dbo.Orders
WHERE LEN(Notes) > 100;

-- ✅ SARGable: computed column + index
ALTER TABLE dbo.Orders ADD NotesLength AS LEN(Notes) PERSISTED;
CREATE INDEX IX_Orders_NotesLength ON dbo.Orders (NotesLength);

SELECT OrderId, Notes
FROM dbo.Orders
WHERE NotesLength > 100;
```

**Symptom:** A data quality report that flags orders with excessively long notes takes 30 seconds. The execution plan scans the table.

**Fix:**

```sql
-- Computed column approach (requires schema change)
ALTER TABLE dbo.Orders ADD NotesLen AS CAST(LEN(Notes) AS SMALLINT) PERSISTED;
CREATE INDEX IX_Orders_NotesLen ON dbo.Orders (NotesLen);
```

**Cost of not fixing:** A nightly data quality check scans 50M rows to find notes longer than 500 characters. The scan takes 45 seconds. With the computed column and index, it takes 200ms by seeking to the relevant range.

### Spoken Answers

**Q: What is SARGability, and what makes a predicate non-SARGable?**

> **Great answer:** "SARGability comes from 'Search ARGument ABLE' — it describes whether the optimiser can use an index seek for a predicate. A predicate is SARGable when it has the form `Column op Value` — the column stands alone on one side of the operator, and the expression to evaluate is on the other side. For example, `OrderDate >= '2024-01-01'` is SARGable: the optimiser can find the first row where OrderDate >= '2024-01-01' in the B-tree and seek directly to it. A predicate becomes non-SARGable when the column is wrapped in a function: `YEAR(OrderDate) = 2024`. The optimiser cannot derive a seek range because the index stores raw OrderDate values, not the output of YEAR(). It has no way to know which raw values correspond to 'Year equals 2024' without reading every row and applying the function. The predicate is placed as a residual filter after a scan. The key insight is that the optimiser cannot invert most functions — it doesn't know that `YEAR(col) = 2024` means `col >= '2024-01-01' AND col < '2025-01-01'`. The fix is always to rewrite the predicate so the column is alone: move the computation to the value side."

---

**Q: Rewrite `DATEDIFF(day, OrderDate, GETDATE()) > 30` to be SARGable.**

> **Great answer:** "The non-SARGable form `DATEDIFF(day, OrderDate, GETDATE()) > 30` wraps OrderDate in the DATEDIFF function. The SARGable rewrite moves the computation to the GETDATE() side: `OrderDate < DATEADD(day, -30, GETDATE())`. Now OrderDate is alone on one side of `<`, and the expression `DATEADD(day, -30, GETDATE())` is evaluated once before the query starts — it's a constant for the duration of the query. The optimiser can seek on the OrderDate index: find all rows where OrderDate < the computed cutoff. The key difference is that DATEDIFF must be evaluated per row (50M evaluations), while DATEADD is evaluated once (1 evaluation). In terms of logical reads, the non-SARGable version scans all 60,450 pages; the SARGable version seeks to the cutoff point and reads only the pages with matching rows."

---

**Q: Which EF Core LINQ methods generate non-SARGable SQL, and how do you avoid them?**

> **Great answer:** "Several LINQ methods translate to function-on-column SQL. The most common ones are: `o.OrderDate.Year` → `YEAR([OrderDate])`; `EF.Functions.DateDiffDay(o.OrderDate, DateTime.UtcNow)` → `DATEDIFF(DAY, [OrderDate], GETUTCDATE())`; `o.LastName.ToUpper()` → `UPPER([LastName])`; `o.LastName.Substring(0, 2)` → `SUBSTRING([LastName], 0+1, 2)`; and `CAST(o.OrderDate AS Date)` via `EF.Property<DateTime>` tricks. To avoid them, always rewrite the LINQ to push computation to the value side. For year queries, compute `startDate` and `endDate` in C# and pass them: `db.Orders.Where(o => o.OrderDate >= startDate && o.OrderDate < endDate)`. For relative dates, compute the cutoff in C#: `var cutoff = DateTime.UtcNow.AddDays(-30)` then `db.Orders.Where(o => o.OrderDate < cutoff)`. For case-insensitive search, rely on CI collation — SQL Server's default collation is case-insensitive, so `o.LastName == "SMITH"` matches 'Smith' without any ToUpper. For prefix search, use `StartsWith()` which translates to `LIKE @p + '%'` — a SARGable prefix seek. The general principle: do as much date math and string manipulation in C# as possible, then pass the computed values to the query."

### Interview Trigger

The signature non-SARGable question: "Your query `WHERE YEAR(OrderDate) = 2024` is scanning. The column is indexed. Why?" A candidate who answers "Because YEAR is a function on the column" passes. A great candidate adds "The optimiser cannot invert YEAR to derive a seek range. The fix is `WHERE OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01'`. In EF Core, compute the start and end dates in C# before the query." The follow-up: "What about `DATEDIFF`?" — "Same problem. Move the computation to the other side: `OrderDate < DATEADD(day, -30, GETDATE())`." The deepest follow-up: "What if neither side is a constant — like `DATEDIFF(day, InvoiceDate, PaidDate) > 7`?" — "Both columns are wrapped. The fix is a computed column with index: `ALTER TABLE ... ADD DaysToPay AS DATEDIFF(day, InvoiceDate, PaidDate) PERSISTED` and create an index on it."

### Application Checklist

- [ ] Every WHERE predicate checked for function-on-column patterns
- [ ] YEAR/MONTH/DATEPART rewrites applied to range predicates
- [ ] DATEDIFF rewrites: computation pushed to GETDATE() or the parameter side
- [ ] UPPER/LOWER removed when column collation is case-insensitive

- [ ] Computed columns with indexes added for frequent non-rewritable patterns

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Index Seek instead of Scan (4 vs 60K reads)|Query rewrite effort (range predicates are more verbose)|
|Consistent sub-millisecond response times|Application-side date math (compute ranges in C#)|
|Eliminated residual filter CPU cost (50M function calls saved)|Slightly more complex SQL (range predicates vs simple YEAR)|
|Reduced I/O pressure|Computed column + index adds write overhead for persistent fixes|

### Scale Thresholds

- **< 10K rows**: Non-SARGable predicates are manageable. A scan of 10K rows takes < 10ms. May not be worth rewriting if the table stays small.
- **10K–1M rows**: Non-SARGable predicates become noticeable. A scan of 1M rows takes ~20-50ms. Fix when found during code review.
- **1M–10M rows**: Non-SARGable predicates are problematic. A scan of 10M rows with functions takes 200-500ms. Must fix every occurrence.
- **10M–100M rows**: Non-SARGable predicates are critical. A scan of 50M rows takes 1-5 seconds. Every function-on-column pattern causes a production incident at this scale.
- **High concurrency (> 500 queries/sec)**: One scan can saturate the I/O subsystem. Even a single non-SARGable query at high concurrency can cause cascading performance degradation.

In my team, we have a code review checklist item: 'No function wrapping the column in WHERE or JOIN.' We review execution plans for residual predicates in scans. For EF Core, we avoid `.Year`, `.ToUpper()`, and `EF.Functions.DateDiffDay` in LINQ Where clauses. For patterns that can't be rewritten (like comparing two columns with DATEDIFF), we add a computed column with an index. Every rewrite saves 60,000+ logical reads per query on a 50M row table."

</details>

**Challenge 1 — Write the range predicate rewrite**

A quarterly business review needs order counts by quarter for 2024. Write a SARGable query that returns the order count for each quarter of 2024. The `Orders` table has 50M rows and an index on `OrderDate`.

<details>
<summary>Solution</summary>

```sql
-- SARGable: each quarter is a range predicate on OrderDate
SELECT
    CASE
        WHEN o.OrderDate >= '2024-01-01' AND o.OrderDate < '2024-04-01' THEN 'Q1'
        WHEN o.OrderDate >= '2024-04-01' AND o.OrderDate < '2024-07-01' THEN 'Q2'
        WHEN o.OrderDate >= '2024-07-01' AND o.OrderDate < '2024-10-01' THEN 'Q3'
        WHEN o.OrderDate >= '2024-10-01' AND o.OrderDate < '2025-01-01' THEN 'Q4'
    END AS Quarter,
    COUNT(*) AS OrderCount,
    SUM(o.TotalAmount) AS TotalRevenue
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01'
  AND o.OrderDate < '2025-01-01'
GROUP BY
    CASE
        WHEN o.OrderDate >= '2024-01-01' AND o.OrderDate < '2024-04-01' THEN 'Q1'
        WHEN o.OrderDate >= '2024-04-01' AND o.OrderDate < '2024-07-01' THEN 'Q2'
        WHEN o.OrderDate >= '2024-07-01' AND o.OrderDate < '2024-10-01' THEN 'Q3'
        WHEN o.OrderDate >= '2024-10-01' AND o.OrderDate < '2025-01-01' THEN 'Q4'
    END
ORDER BY Quarter;
-- Execution plan: Index Seek on IX_Orders_OrderDate for the year range
-- Then Compute Scalar for CASE, Hash Match for GROUP BY
-- Logical reads: ~450 (range scan of 2024 data only)
```

**EF Core:**
```csharp

var endDate = startDate.AddYears(1);

var quarterlyStats = await dbContext.Orders
    .Where(o => o.OrderDate >= startDate && o.OrderDate < endDate)
    .GroupBy(o => new {
        Quarter = o.OrderDate.Month <= 3 ? "Q1" :
                  o.OrderDate.Month <= 6 ? "Q2" :
                  o.OrderDate.Month <= 9 ? "Q3" : "Q4"
    })
    .Select(g => new {
        Quarter = g.Key.Quarter,
        OrderCount = g.Count(),
        TotalRevenue = g.Sum(o => o.TotalAmount)
    })
    .OrderBy(r => r.Quarter)
    .ToListAsync(cancellationToken);
```

**Logical reads:** ~450 (range scan) **Execution plan:** `[Index Seek IX_Orders_OrderDate] → [Compute Scalar (Quarter)] → [Hash Match (Aggregate)] → [SELECT]`

**Challenge 2 — Fix the performance problem**

```sql
-- This query runs a daily report of high-value orders overdue for payment.
-- It takes 12 seconds on a 25M row Invoices table.
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

SELECT i.InvoiceId, i.InvoiceNumber, i.InvoiceDate, i.DueDate, i.TotalAmount,
       DATEDIFF(day, i.DueDate, GETDATE()) AS DaysOverdue
FROM dbo.Invoices AS i
WHERE DATEDIFF(day, i.DueDate, GETDATE()) > 60
  AND i.PaidDate IS NULL
  AND i.TotalAmount > 10000
ORDER BY i.DueDate;

-- SET STATISTICS IO:
-- Table 'Invoices'. Scan count 1, logical reads 32500
-- SQL Server Execution Times: CPU time = 210ms, elapsed time = 12s
```

Identify why it is slow and fix it.

<details>
<summary>Solution</summary>

**Root cause:** `DATEDIFF(day, i.DueDate, GETDATE()) > 60` wraps `DueDate` in DATEDIFF — non-SARGable. The index on DueDate cannot be used for a seek. The entire Invoices table is scanned (32,500 reads).

**Fix:**

```sql
-- SARGable: compute cutoff date once
DECLARE @CutoffDate DATETIME2(0) = DATEADD(day, -60, GETDATE());

SELECT i.InvoiceId, i.InvoiceNumber, i.InvoiceDate, i.DueDate, i.TotalAmount,
       DATEDIFF(day, i.DueDate, GETDATE()) AS DaysOverdue  -- DATEDIFF in SELECT is fine
FROM dbo.Invoices AS i
WHERE i.DueDate < @CutoffDate      -- SARGABLE: column alone
  AND i.PaidDate IS NULL           -- SARGABLE: IS NULL
  AND i.TotalAmount > 10000        -- SARGABLE: column alone
ORDER BY i.DueDate;
```

**Index to create:**
```sql
CREATE INDEX IX_Invoices_DueDate_PaidDate_TotalAmount
    ON dbo.Invoices (DueDate, PaidDate, TotalAmount)
    INCLUDE (InvoiceId, InvoiceNumber, InvoiceDate);
```

**After fix — logical reads:** ~120 (seek to overdue range then filter) from 32,500.

**EF Core:**
```csharp
var cutoffDate = DateTime.UtcNow.AddDays(-60);

var overdueInvoices = await dbContext.Invoices
    .Where(i => i.DueDate < cutoffDate
             && i.PaidDate == null

    .OrderBy(i => i.DueDate)
    .Select(i => new {
        i.InvoiceId,
        i.InvoiceNumber,
        i.InvoiceDate,
        i.DueDate,
        i.TotalAmount,
        DaysOverdue = EF.Functions.DateDiffDay(i.DueDate, DateTime.UtcNow)
    })
    .ToListAsync(cancellationToken);
```

**Challenge 3 — Explain the execution plan**

A query and its execution plan:

```sql
SELECT o.OrderId, o.OrderDate, o.TotalAmount, o.Status
FROM dbo.Orders AS o
WHERE MONTH(o.OrderDate) = 1
  AND YEAR(o.OrderDate) = 2024;
```

The execution plan shows:
```
[Clustered Index Scan (PK_Orders)]
  Predicate: MONTH(OrderDate)=1 AND YEAR(OrderDate)=2024
Estimated Cost: 48  |  Actual Logical Reads: 60,450
Rows Affected: 150,000
```

The OrderDate column has an index `IX_Orders_OrderDate`. Explain why the optimiser chose a scan.

<details>
<summary>Solution</summary>

**Why the scan:** Both `MONTH(OrderDate)` and `YEAR(OrderDate)` wrap the column in functions. The optimiser cannot invert MONTH or YEAR to derive a seek range. Even though the index on OrderDate exists, it cannot be used for seeking because the B-tree stores raw datetime2 values, not month/year components. The optimiser must scan all 60,450 pages, compute MONTH and YEAR for every row, and filter.

**What the optimiser would need to do a seek:** A predicate of the form `OrderDate >= '2024-01-01' AND OrderDate < '2024-02-01'`. This is a single range on the raw OrderDate values — the optimiser can seek to the start of January 2024 in the B-tree and read sequentially until February.

**Fix:**

```sql
-- SARGable: single range predicate
SELECT o.OrderId, o.OrderDate, o.TotalAmount, o.Status
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01'
  AND o.OrderDate < '2024-02-01';
```

**After fix:**
```
[Index Seek (IX_Orders_OrderDate)]
  Seek Predicates:
    Seek Keys[0]: OrderDate >= Scalar Operator('2024-01-01')
    Seek Keys[0]: OrderDate <  Scalar Operator('2024-02-01')
→ [SELECT]
Estimated Cost: 0.4  |  Actual Logical Reads: 120 (January data only)
```

**Tradeoff:** None. The range predicate is semantically identical — finding orders in January 2024 — and far more efficient. The MONTH+YEAR combination was querying the same thing but in a way the optimiser couldn't optimise.

</details>

**Challenge 4 — Diagnose the multi-function problem**

A team has a query that finds orders placed on weekends. It's scanning a 100M row table:

```sql
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE DATEPART(weekday, OrderDate) IN (1, 7);  -- Sunday = 1, Saturday = 7 (default)
```

They try to fix it with a computed column but see no improvement:

```sql
ALTER TABLE dbo.Orders ADD OrderWeekday AS DATEPART(weekday, OrderDate) PERSISTED;
CREATE INDEX IX_Orders_OrderWeekday ON dbo.Orders (OrderWeekday);
```

The query still scans. Explain why and fix it.

<details>
<summary>Solution</summary>

**Root cause 1:** The computed column uses `DATEPART(weekday, OrderDate)` but the `DATEFIRST` setting affects the result. If the `DATEFIRST` setting at query time differs from when the computed column was created, the persisted value may be wrong. SQL Server may not trust the computed column for indexing if the expression is non-deterministic with respect to `DATEFIRST`.

Actually, `DATEPART(weekday, date)` IS deterministic — it always returns the same value for the same date regardless of `DATEFIRST` when the column is PERSISTED. So that's not the issue.

**Root cause 2:** The query `WHERE DATEPART(weekday, OrderDate) IN (1, 7)` is still using `DATEPART` on `OrderDate` directly, NOT querying the computed column. The optimiser does not automatically rewrite `DATEPART(weekday, OrderDate)` to reference `OrderWeekday`. The query must explicitly use the computed column name.

**Fix:**

```sql
-- ✅ Query the computed column directly
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderWeekday IN (1, 7);
-- Index Seek on IX_Orders_OrderWeekday

-- Or rewrite as a date-based range (better — no computed column needed)
-- Weekends in January 2024:
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderDate >= '2024-01-06' AND OrderDate < '2024-01-08'  -- Sat-Sun
   OR OrderDate >= '2024-01-13' AND OrderDate < '2024-01-15'
   OR OrderDate >= '2024-01-20' AND Date < '2024-01-22'
   OR OrderDate >= '2024-01-27' AND OrderDate < '2024-01-29';
-- But this is impractical for open-ended queries — use the computed column

-- Better: function-based approach for open-ended weekend search
-- The computed column + index is the correct permanent fix:
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderWeekday IN (1, 7)
  AND OrderDate >= '2024-01-01';  -- Additional date range for partial seek
```

**After fix with computed column + querying it directly:**
```
[Index Seek (IX_Orders_OrderWeekday)]
  Seek Predicates: OrderWeekday IN (1, 7)
→ [Key Lookup (PK_Orders) for OrderId, OrderDate, TotalAmount] → [SELECT]
```

**Key lesson:** A computed column index only helps if queries reference the computed column name, not the function expression.

</details>

**Challenge 5 — Design the query rewrite strategy**

**Scenario:** An e-commerce platform with:
- `Orders` (80M rows) — `OrderId`, `CustomerId`, `OrderDate` (datetime2), `TotalAmount`, `Status` (VARCHAR(20))
- `OrderItems` (400M rows) — `OrderItemId`, `OrderId`, `ProductId`, `Quantity`, `UnitPrice`
- `Products` (500K rows) — `ProductId`, `ProductName`, `CategoryId`
- `ProductCategories` (5K rows) — `CategoryId`, `CategoryName`

You are consulting for a team that has the following slow queries. Rewrite each to be SARGable:

1. Find all orders from 2024 with total > $500: `WHERE YEAR(OrderDate) = 2024 AND TotalAmount > 500`
2. Find products ordered more than 100 times: `WHERE (SELECT COUNT(*) FROM OrderItems WHERE ProductId = Products.ProductId) > 100`
3. Find customers whose last order was more than 90 days ago: `WHERE DATEDIFF(day, (SELECT MAX(OrderDate) FROM Orders WHERE CustomerId = Customers.CustomerId), GETDATE()) > 90`
4. Search products by name prefix (case-insensitive): `WHERE UPPER(ProductName) LIKE UPPER(@search) + '%'`

<details>
<summary>Solution</summary>

**Query 1 — YEAR(OrderDate) rewrite:**
```sql
-- ❌ Original
SELECT * FROM Orders WHERE YEAR(OrderDate) = 2024 AND TotalAmount > 500;

-- ✅ SARGable
SELECT * FROM Orders
WHERE OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01'
  AND TotalAmount > 500;
-- Index: IX_Orders_OrderDate (exists), IX_Orders_TotalAmount (for seek)
```

**Query 2 — Correlated subquery to indexed view or window function:**
```sql
-- ❌ Original (correlated subquery scans OrderItems per Product row)
SELECT * FROM Products
WHERE (SELECT COUNT(*) FROM OrderItems WHERE ProductId = Products.ProductId) > 100;

-- ✅ SARGable with window function + temp table or derived table
SELECT p.ProductId, p.ProductName, oi.OrderCount
FROM Products AS p
INNER JOIN (
    SELECT ProductId, COUNT(*) AS OrderCount
    FROM OrderItems
    GROUP BY ProductId
) AS oi ON p.ProductId = oi.ProductId
WHERE oi.OrderCount > 100;
-- Index: IX_OrderItems_ProductId (for Group By seek)
```

**Query 3 — DATEDIFF with subquery:**
```sql
-- ❌ Original
SELECT * FROM Customers
WHERE DATEDIFF(day,
    (SELECT MAX(OrderDate) FROM Orders WHERE CustomerId = Customers.CustomerId),
    GETDATE()) > 90;

-- ✅ SARGable: pre-compute last order date
WITH CustomerLastOrder AS (
    SELECT CustomerId, MAX(OrderDate) AS LastOrderDate
    FROM Orders
    GROUP BY CustomerId
)
SELECT c.*
FROM Customers AS c
INNER JOIN CustomerLastOrder AS clo
    ON c.CustomerId = clo.CustomerId
WHERE clo.LastOrderDate < DATEADD(day, -90, GETDATE());
-- Index: IX_Orders_CustomerId_OrderDate (for MAX seek per customer)
```

**Query 4 — UPPER on column:**
```sql
-- ❌ Original (UPPER on column — non-SARGable)
SELECT * FROM Products
WHERE UPPER(ProductName) LIKE UPPER(@search) + '%';

-- ✅ SARGable: CI collation (default) handles case-insensitive prefix
SELECT * FROM Products
WHERE ProductName LIKE @search + '%';  -- CI collation: matches any case

-- If column has CS collation:
SELECT * FROM Products
WHERE ProductName LIKE @search + '%' COLLATE Latin1_General_CI_AS;
-- COLLATE on value side — column not converted, SARGable
-- Index: IX_Products_ProductName (prefix seek)
```

</details>
