### Where This Fits

LIKE is the primary T-SQL operator for pattern-based string filtering. Every .NET backend engineer encounters it in search, auto-complete, validation, and reporting queries. The most expensive mistake made here is using `LIKE '%term%'` (leading wildcard) in production: it forces a full index or table scan — the predicate is non-SARGable because the optimiser cannot determine which index keys to seek when the prefix is unknown. A trailing wildcard `LIKE 'term%'` is SARGable and enables an Index Seek via a range scan. Interviewers use LIKE to gate candidates on SARGability: the candidate who knows that `LIKE 'ABC%'` seeks while `LIKE '%ABC%'` scans understands how the B-tree index is structured and what the optimiser can and cannot do. Engineers who know this topic deeply also know when to escalate to full-text search (CONTAINS) for large-text pattern matching, when to use CHARINDEX as an alternative non-SARGable option, and how EF Core's `Contains`, `StartsWith`, and `EndsWith` translate to different LIKE patterns with different performance profiles.

## Core Mental Model

LIKE is a string comparison predicate that evaluates whether a character expression matches a specified pattern containing wildcard characters. The optimiser's ability to use an index on the searched column depends entirely on whether the pattern has a fixed prefix. `LIKE 'prefix%'` is internally converted to a range seek: `Column >= 'prefix' AND Column < 'prefiy'` (where 'y' is the next Unicode code point after 'x' in the last character of the prefix). This range is SARGable — the optimiser seeks to the first matching key and scans forward within the range. `LIKE '%suffix'` and `LIKE '%infix%'` have no fixed prefix — the optimiser cannot derive a starting point in the index and must scan every leaf page, checking each row with a pattern-matching operator. The wildcard `_` matches exactly one character and also breaks prefix SARGability if it appears in the leading position. The ESCAPE clause allows literal interpretation of wildcard characters in the pattern. LIKE is always evaluated using the collation of the column, which determines case sensitivity and accent sensitivity of the match.

### Classification

LIKE is a **string comparison predicate** in the `WHERE` clause operator family. It is **conditionally SARGable**: SARGable when the pattern has a fixed prefix (no leading wildcard), non-SARGable when the pattern starts with `%`, `_`, or a character class wildcard `[...]`. In execution plans, a SARGable LIKE produces an `Index Seek` with a `Seek Predicate` range; a non-SARGable LIKE produces an `Index Scan` or `Table Scan` with a `Predicate` (residual) filter applied to every row.

```mermaid
flowchart TD
    A[LIKE pattern received] --> B{Pattern starts with wildcard?}
    B -->|No - fixed prefix| C[Optimiser converts to range seek]
    B -->|Yes - % or _ or [| D[No fixed prefix - must scan]
    C --> E{Does column have an index?}
    E -->|Yes| F[Index Seek with range predicate]
    E -->|No| G[Table Scan with residual predicate]
    D --> H[Table/Clustered Index Scan]
    H --> I[Predicate: evaluate LIKE per row]
    F --> J[Seek: column >= prefix AND column < prefix+nextChar]
    J --> K[Only index pages in range are read]
    I --> L[Every page in index/table is read]
    K --> M{Pattern has additional wildcards after prefix?}
    M -->|Yes| N[Residual predicate filters false positives]
    M -->|No - literal match| O[No residual needed]
    N --> P[Output matching rows]
    O --> P
    L --> P
```

## Deep Mechanics

### How the Engine Executes This

1. **Parsing** — The parser tokenises the LIKE predicate. The pattern string is parsed for wildcard characters. The ESCAPE clause (if present) is validated for syntax.

2. **Binding (Algebrizer)** — The algebrizer resolves the column reference and the pattern expression. If the pattern is a literal string, the algebrizer can immediately determine whether the pattern starts with a wildcard and pass this information to the optimiser in the form of a `LIKE` operator with known properties. If the pattern is a parameter (`@Pattern`) or expression, the algebrizer cannot determine SARGability at bind time — the optimiser must generate a plan that works for any pattern value.

3. **Optimisation** — The optimiser examines the LIKE predicate:
   - **Literal pattern with fixed prefix**: The optimiser converts `LIKE 'ABC%'` to a range predicate in the `SeekPredicate` node: `Column >= 'ABC' AND Column < 'ABD'` (or the next code point after the last prefix character, adjusted for collation sort weights). This range is pushed into an Index Seek operator if a suitable index exists.
   - **Literal pattern with leading wildcard**: No range can be derived. The optimiser chooses a Scan operator with a residual `LIKE` predicate applied to every row.
   - **Parameterised pattern**: The optimiser cannot determine SARGability at compile time. It generates a plan with a Scan and a residual predicate. This is a parameter sniffing opportunity: if the first execution uses `'ABC%'`, the plan might include a Seek (if the optimiser guesses a prefix), but the plan is cached and re-used for `'%ABC'` where the Seek would be invalid. SQL Server handles this conservatively: parameterised LIKE with `%` anywhere in the pattern generally gets a Scan plan.
   - **Optimiser trick with leading `[...]`**: A pattern like `LIKE '[ABC]%'` can sometimes be converted to an OR of range seeks: `(Column >= 'A' AND Column < 'B') OR (Column >= 'B' AND Column < 'C') OR (Column >= 'C' AND Column < 'D')`. The optimiser may choose this only for small character classes; large ranges like `[a-z]` generally get a scan because the OR expansion cost exceeds the scan cost.

4. **Execution**:
   - **Seek path**: The Index Seek operator navigates the B-tree to the first key >= 'ABC', reads forward until key >= 'ABD' (or until no more matches). For each returned row, if the pattern contains additional wildcards after the prefix (e.g., `LIKE 'ABC%DEF'`), a residual predicate evaluates the full pattern. Without additional wildcards, the range predicate exactly matches the intended rows.
   - **Scan path**: The Scan operator reads every row from the table or index. For each row, the LIKE predicate is evaluated by the pattern-matching engine within SQL Server. This involves iterating through the string and the pattern simultaneously, handling wildcards. CPU cost per row is proportional to string length and pattern complexity.

5. **Pattern matching algorithm** — SQL Server's LIKE implementation uses a finite-state-machine-based matcher. Simple patterns (no `_`, `[...]`, or ESCAPE) with only `%` are optimised. Complex patterns with character classes and ESCAPE are evaluated with a full backtracking engine. The worst-case complexity is O(s × p) where s is string length and p is pattern length, but degenerate patterns can cause exponential backtracking in edge cases (e.g., `LIKE '%_%_%_%_%'` on long strings).

### Execution Plan Analysis

**SARGable LIKE (trailing wildcard):**

- `[Index Seek (NonClustered)] IX_Customers_LastName` — seeks on Range: `LastName >= 'Smith' AND LastName < 'Smiti'`
- No Key Lookup if the index covers the query columns
- Residual predicate only if additional wildcards after the prefix
- Estimated cost: ~0.003 per 10 matching rows, Logical Reads: ~3–8

**Non-SARGable LIKE (leading or surrounding wildcard):**

- `[Clustered Index Scan]` or `[Index Scan]` — reads every row
- `[Filter]` — residual predicate applies LIKE '%smith%' per row
- No index seek possible regardless of available indexes
- Estimated cost: proportional to table size; Logical Reads: full table/index pages

```
SARGable (LIKE 'Smith%'):
[Index Seek IX_Customers_LastName] → [Residual (if needed)] → [SELECT]
Estimated Cost: 0.003  |  Logical Reads: ~5

Non-SARGable (LIKE '%smith%'):
[Clustered Index Scan] → [Filter: LIKE '%smith%'] → [SELECT]
Estimated Cost: ~12  |  Logical Reads: ~12,000 (1M rows)
```

**Parameterised pattern — optimiser must guess:**

When the pattern is passed as a parameter (e.g., `WHERE LastName LIKE @Pattern`), the optimiser compiles a plan without knowing whether the pattern starts with a wildcard. SQL Server's behaviour:

- If the pattern parameter contains `%` anywhere in the literal value at compile time (parameter embedding for simple parameters), the optimiser sees the `%` and generates a Scan plan.
- If the pattern is a variable (`@Pattern`) with no compile-time value, the optimiser generates a Scan plan conservatively.
- The exception: `OPTION (RECOMPILE)` allows the optimiser to see the actual parameter value and choose Seek vs Scan per execution.

```
Parameterised pattern plan (any wildcard possibility):
[Clustered Index Scan] → [Filter: LIKE @Pattern] → [SELECT]
Cost: ~12 (always scan)  |  Logical Reads: ~12,000
```

### Failure Modes

**Leading wildcard in production:** `LIKE '%term%'` is the most common non-SARGable predicate in legacy code. It forces a full clustered index scan on any table size. Detect high-cost LIKE scans with:

```sql
-- Find queries with non-SARGable LIKE scans in plan cache
SELECT TOP 20
    qs.total_logical_reads / qs.execution_count AS avg_logical_reads,
    qs.execution_count,
    qs.total_worker_time / qs.execution_count AS avg_cpu_us,
    SUBSTRING(st.text, 1, 500) AS query_text
FROM sys.dm_exec_query_stats AS qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) AS st
WHERE st.text LIKE '%LIKE%'
    AND st.text LIKE '%%'
    AND qs.total_logical_reads / NULLIF(qs.execution_count, 0) > 1000
ORDER BY avg_logical_reads DESC;
```

**Parameter sniffing with LIKE:** A stored procedure that filters by `LIKE @Pattern` compiles one plan. If the first call uses `LIKE 'Smith%'`, the plan might include an Index Seek. If the second call uses `LIKE '%unknown%'`, the Seek plan returns wrong results (the Seek range doesn't match) or scans anyway (if the optimiser embeds the value and adds a scan). Always analyse the plan for parameterised LIKE:

```sql
CREATE PROCEDURE dbo.SearchCustomers
    @Pattern NVARCHAR(100)
AS
    SELECT CustomerId, LastName, Email
    FROM dbo.Customers
    WHERE LastName LIKE @Pattern
    OPTION (RECOMPILE);  -- Forces optimiser to see the actual value
```

**Collation mismatch:** When the pattern string has a different collation than the column, an implicit conversion occurs. The conversion can prevent index seeks even on trailing wildcard patterns:

```sql
-- ❌ Implicit conversion: pattern is Latin1_General_BIN, column is Latin1_General_CI_AS
SELECT CustomerId, LastName
FROM dbo.Customers
WHERE LastName LIKE 'Smith%' COLLATE Latin1_General_BIN;
```

### Primary SQL Implementation

```sql
-- ============================================================
-- Schema context
-- ============================================================
CREATE TABLE dbo.Customers
(
    CustomerId   INT            NOT NULL IDENTITY(1,1),
    FirstName    NVARCHAR(100)  NOT NULL,
    LastName     NVARCHAR(100)  NOT NULL,
    Email        NVARCHAR(256)  NOT NULL,
    Phone        VARCHAR(20)    NULL,
    City         NVARCHAR(100)  NOT NULL,
    Region       NVARCHAR(50)   NOT NULL DEFAULT 'US',
    PostalCode   VARCHAR(20)    NOT NULL,
    Notes        NVARCHAR(MAX)  NULL,
    CreatedAt    DATETIME2(0)   NOT NULL DEFAULT SYSUTCDATETIME(),
    IsActive     BIT            NOT NULL DEFAULT 1,
    CONSTRAINT PK_Customers PRIMARY KEY CLUSTERED (CustomerId)
);

CREATE INDEX IX_Customers_LastName ON dbo.Customers (LastName);
CREATE INDEX IX_Customers_Email ON dbo.Customers (Email);
CREATE INDEX IX_Customers_City ON dbo.Customers (City);
CREATE INDEX IX_Customers_PostalCode ON dbo.Customers (PostalCode);

-- ============================================================
-- Pattern 1: Auto-complete / typeahead (SARGable)
-- ============================================================
SELECT c.CustomerId, c.FirstName, c.LastName, c.Email
FROM dbo.Customers AS c
WHERE c.LastName LIKE @SearchTerm + '%'
ORDER BY c.LastName;

-- ============================================================
-- Pattern 2: Contains-style search (non-SARGable, use sparingly)
-- ============================================================
SELECT c.CustomerId, c.FirstName, c.LastName
FROM dbo.Customers AS c
WHERE c.LastName LIKE '%' + @SearchTerm + '%'
   OR c.FirstName LIKE '%' + @SearchTerm + '%'
ORDER BY c.LastName;

-- ============================================================
-- Pattern 3: Phone number format validation
-- ============================================================
-- Find phone numbers matching US format (XXX) XXX-XXXX
SELECT c.CustomerId, c.FirstName, c.LastName, c.Phone
FROM dbo.Customers AS c
WHERE c.Phone LIKE '([0-9][0-9][0-9]) [0-9][0-9][0-9]-[0-9][0-9][0-9][0-9]';

-- ============================================================
-- Pattern 4: Email domain filter (trailing suffix, non-SARGable)
-- ============================================================
-- ❌ Non-SARGable: leading wildcard
SELECT c.CustomerId, c.Email
FROM dbo.Customers AS c
WHERE c.Email LIKE '%@gmail.com';

-- ✅ Alternative: reverse the string and use prefix search
-- Requires computed column or indexed view for performance at scale
ALTER TABLE dbo.Customers ADD EmailReversed AS
    REVERSE(Email);

CREATE INDEX IX_Customers_EmailReversed ON dbo.Customers (EmailReversed);

-- Now suffix search becomes prefix search:
SELECT c.CustomerId, c.Email
FROM dbo.Customers AS c
WHERE c.EmailReversed LIKE REVERSE('@gmail.com') + '%';

-- Verify:
-- Email = 'user@gmail.com' → EmailReversed = 'moc.liamg@resu'
-- REVERSE('@gmail.com') = 'moc.liamg@'
-- LIKE 'moc.liamg@%' — SARGable!

-- ============================================================
-- Pattern 5: Case-insensitive search (collation-dependent)
-- ============================================================
-- If column uses case-insensitive collation (default), LIKE matches cases:
-- 'Smith%' matches 'Smith', 'smith', 'SMITH' all the same.

-- If column uses case-sensitive collation, use explicit COLLATE:
SELECT c.CustomerId, c.LastName
FROM dbo.Customers AS c
WHERE c.LastName LIKE 'smith%'
    COLLATE Latin1_General_CI_AI;  -- Force case-insensitive

-- ============================================================
-- Pattern 6: ESCAPE clause for literal wildcard characters
-- ============================================================
-- Products table with discount codes containing % or _
CREATE TABLE dbo.DiscountCodes
(
    DiscountCode   VARCHAR(50) NOT NULL PRIMARY KEY,
    Description    VARCHAR(200) NULL,
    DiscountPct    DECIMAL(5,4) NOT NULL,
    ValidFrom      DATE NOT NULL,
    ValidTo        DATE NOT NULL
);

-- Find codes starting with 'SALE_':
SELECT DiscountCode, Description, DiscountPct
FROM dbo.DiscountCodes
WHERE DiscountCode LIKE 'SALE\_%' ESCAPE '\';

-- ============================================================
-- Pattern 7: LIKE with date patterns (avoid — use proper date ranges)
-- ============================================================
-- ❌ Non-SARGable and unreliable
SELECT OrderId, OrderDate
FROM dbo.Orders
WHERE CONVERT(VARCHAR, OrderDate, 112) LIKE '2024%';

-- ✅ SARGable range
SELECT OrderId, OrderDate
FROM dbo.Orders
WHERE OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01';

-- ============================================================
-- Pattern 8: PATINDEX as an alternative to LIKE
-- ============================================================
-- PATINDEX returns the starting position of a pattern, or 0 if not found
-- Useful when you need the position of the match, not just existence
SELECT
    c.CustomerId,
    c.LastName,
    PATINDEX('%son%', c.LastName) AS MatchPosition
FROM dbo.Customers AS c
WHERE PATINDEX('%son%', c.LastName) > 0;

-- PATINDEX vs LIKE: same performance (both non-SARGable with leading wildcard)
-- Use PATINDEX when you need the match position; use LIKE for simple existence
```

```csharp
// Pattern 1: StartsWith — SARGable (LIKE 'term%')
public async Task<List<Customer>> SearchByLastNamePrefixAsync(
    string prefix,
    CancellationToken cancellationToken = default)
{
    return await dbContext.Customers
        .Where(c => c.LastName.StartsWith(prefix))
        .OrderBy(c => c.LastName)
        .Take(20)
        .ToListAsync(cancellationToken);

    //           FROM [Customers] AS [c]

        .OrderBy(c => c.LastName)
        .Take(50)
        .ToListAsync(cancellationToken);

    //           FROM [Customers] AS [c]

// Pattern 4: EF.Functions.Like for custom patterns with ESCAPE
public async Task<List<Customer>> SearchWithCustomPatternAsync(
    string pattern,
    string escapeChar,
    CancellationToken cancellationToken = default)
{
    return await dbContext.Customers
        .Where(c => EF.Functions.Like(c.LastName, pattern, escapeChar))
        .ToListAsync(cancellationToken);
    // Generated: WHERE [c].[LastName] LIKE @pattern ESCAPE @escapeChar
}

// Pattern 5: Composite search — FirstName OR LastName
public async Task<List<Customer>> SearchByNameAsync(
    string term,
    CancellationToken cancellationToken = default)
{
    return await dbContext.Customers
        .Where(c => c.LastName.Contains(term) || c.FirstName.Contains(term))
        .OrderBy(c => c.LastName)
        .Take(50)
        .ToListAsync(cancellationToken);
    // Generated: WHERE [c].[LastName] LIKE N'%' + @term + N'%' OR [c].[FirstName] LIKE N'%' + @term + N'%'
}

// Pattern 6: Case-insensitive search with EF Core
public async Task<List<Customer>> CaseInsensitiveSearchAsync(
    string term,
    CancellationToken cancellationToken = default)
{
    return await dbContext.Customers
        .Where(c => c.LastName.StartsWith(term) &&
                    c.LastName == c.LastName)  // EF Core adds collation if needed
        .Take(20)
        .ToListAsync(cancellationToken);
    // Note: By default EF Core uses the database collation.
    // To force case-insensitive on a case-sensitive column:
    // Use EF.Functions.Collate(c.LastName, "Latin1_General_CI_AI") with EF Core 8+
}
```

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL LIKE is case-sensitive by default (unlike SQL Server)
-- SQL Server (default case-insensitive collation):
SELECT * FROM Customers WHERE LastName LIKE 'smith%';
-- Returns: Smith, smith, SMITH

-- PostgreSQL (default case-sensitive):
SELECT * FROM customers WHERE last_name LIKE 'smith%';
-- Returns: smith only (not Smith or SMITH)

-- PostgreSQL case-insensitive LIKE alternative — ILIKE:
SELECT * FROM customers WHERE last_name ILIKE 'smith%';
-- Returns: Smith, smith, SMITH (case-insensitive)

-- PostgreSQL SIMILAR TO (SQL standard regex-like patterns):
SELECT * FROM customers WHERE last_name SIMILAR TO 'Sm(i|y)th%';

-- PostgreSQL POSIX regex (~ operator):
SELECT * FROM customers WHERE last_name ~ '^Sm.+th';

-- PostgreSQL full-text search (tsvector/tsquery) for text search:
SELECT * FROM customers
WHERE to_tsvector('english', notes) @@ to_tsquery('english', 'discount & offer');

-- PostgreSQL ESCAPE in LIKE (same syntax, backslash is default):
SELECT * FROM discount_codes
WHERE discount_code LIKE 'SALE\\_%';  -- Backslash is default ESCAPE in PostgreSQL

-- PostgreSQL pattern matching functions:
-- position('pattern' IN string) — similar to PATINDEX
SELECT position('smith' IN last_name) FROM customers;

-- PostgreSQL substring with regex:
SELECT substring(last_name FROM '^...') FROM customers;  -- First 3 chars
```

## Gotchas and Production Pitfalls

### Leading Wildcard — Full Index Scan on 50M Row Table

**Pitfall:** Using `LIKE '%term%'` for search functionality. The leading wildcard forces a full clustered index scan because the optimiser cannot derive a starting point in the B-tree without a fixed prefix.

```sql
-- ❌ Non-SARGable: leading wildcard
SELECT CustomerId, FirstName, LastName, Email
FROM dbo.Customers
WHERE LastName LIKE '%' + @SearchTerm + '%'
ORDER BY LastName;
```

**Symptom:** Execution plan shows a Clustered Index Scan with 50,000+ logical reads on a 1M row table. The query takes 2–8 seconds depending on row width. Under 50 concurrent searches, the I/O subsystem saturates and other queries queue behind the scans.

**Fix:**

```sql
-- Option A: Always use trailing wildcard where functionally acceptable
SELECT CustomerId, FirstName, LastName, Email
FROM dbo.Customers
WHERE LastName LIKE @SearchTerm + '%'  -- SARGable
ORDER BY LastName;

-- Option B: Add full-text search infrastructure for infix search
-- Requires full-text index on LastName
SELECT c.CustomerId, c.FirstName, c.LastName
FROM dbo.Customers AS c
INNER JOIN CONTAINSTABLE(Customers, LastName, @SearchTerm) AS ft
    ON c.CustomerId = ft.[KEY]
ORDER BY ft.RANK DESC;

-- Option C: Use a trigram index approach (custom solution or PostgreSQL pg_trgm)
```

**Cost of not fixing:** A customer-facing search page with an auto-complete box that fires `LIKE '%term%'` on every keystroke. At 10 keystrokes per search × 1,000 users = 10,000 LIKE queries per hour, each scanning 12,000 pages. The I/O subsystem becomes 100% busy during peak hours. Page load times for the search dropdown exceed 5 seconds. The business loses 15% of search users.

### Parameterised LIKE — Cached Plan Does Not Adapt to Pattern

**Pitfall:** A stored procedure or parameterised query that uses `LIKE @Pattern` compiles one plan. If the first execution uses a trailing-wildcard pattern, the plan may include an Index Seek. For leading-wildcard executions using the same cached plan, the Seek operator returns an empty or incomplete result set.

```sql
-- Stored procedure with parameterised LIKE — plan is cached
CREATE PROCEDURE dbo.FindCustomers
    @Pattern NVARCHAR(100)
AS
    SELECT CustomerId, FirstName, LastName
    FROM dbo.Customers
    WHERE LastName LIKE @Pattern;
GO

-- First call: plan includes Index Seek (range 'ABC%')
EXEC dbo.FindCustomers @Pattern = 'ABC%';
-- Plan cached with Index Seek

-- Second call: pattern needs scan
EXEC dbo.FindCustomers @Pattern = '%unknown%';
-- Uses cached Index Seek plan — returns NO rows or wrong rows
-- because the Seek range is ' >= %unknown AND < %unknowy' which matches nothing
```

**Symptom:** Sporadic empty results for certain search terms. Users searching for "ABC" find results; users searching for "Smith" find nothing. The developer blames the data, but the real issue is the cached plan that performs a Seek on a literal `%` prefix — no rows match the range `'%unknown%' >= '%' AND '%unknown%' < '%'` because the prefix is the wildcard itself.

**Fix:**

```sql
-- Add OPTION (RECOMPILE) to force per-execution plan
CREATE PROCEDURE dbo.FindCustomers
    @Pattern NVARCHAR(100)
AS
    SELECT CustomerId, FirstName, LastName
    FROM dbo.Customers
    WHERE LastName LIKE @Pattern
    OPTION (RECOMPILE);
GO

-- Or use dynamic SQL with IF/ELSE for known pattern types
CREATE PROCEDURE dbo.FindCustomers
    @Pattern NVARCHAR(100),
    @IsExactMatch BIT = 0,
    @IsPrefixSearch BIT = 1
AS
    IF @IsPrefixSearch = 1
        SELECT CustomerId, FirstName, LastName
        FROM dbo.Customers
        WHERE LastName LIKE @Pattern + '%';
    ELSE
        SELECT CustomerId, FirstName, LastName
        FROM dbo.Customers
        WHERE LastName LIKE '%' + @Pattern + '%';
GO
```

**Cost of not fixing:** A production incident where the customer search returns no results for 30 minutes after a deployed script warms the cache with a single trailing-wildcard call. The support team escalates to Tier 3. Root cause takes 2 hours to identify because the execution plan looks correct at first glance. The fix (adding OPTION (RECOMPILE)) takes 30 seconds once identified.

### ESCAPE Clause Missing — Literal Wildcard Matches Wrong Rows

**Pitfall:** Using LIKE on data that contains literal `%` or `_` characters (product codes, discount codes, URLs) without an ESCAPE clause. The wildcard characters are interpreted by LIKE, matching far more rows than intended.

```sql
-- ❌ No ESCAPE: _ matches any single character
SELECT DiscountCode, Description, DiscountPct
FROM dbo.DiscountCodes
WHERE DiscountCode LIKE 'SALE_2024';
-- Matches: 'SALE_2024', 'SALEA2024', 'SALE12024', 'SALE-2024' — all wrong
```

**Symptom:** A discount validation query returns 45 codes instead of the expected 1. Customers receive discounts they are not entitled to because `SALE_2024` also matches codes intended for other years. The revenue impact is immediate.

**Fix:**

```sql
-- ✅ Specify ESCAPE character
SELECT DiscountCode, Description, DiscountPct
FROM dbo.DiscountCodes
WHERE DiscountCode LIKE 'SALE\_2024' ESCAPE '\';
-- Matches only: 'SALE_2024'

-- ✅ Or use = for exact match if no wildcard needed
SELECT DiscountCode, Description, DiscountPct
FROM dbo.DiscountCodes
WHERE DiscountCode = 'SALE_2024';
```

**Cost of not fixing:** An e-commerce site runs a "New Year Sale" with code `SALE_2024`. The LIKE-based validation query also accepts `SALE!2024`, `SALEA2024`, and every other single-character substitution. Customers on social media share a glitch that generates 50% off on invalid codes. The fraud detection system flags 2,000 unauthorised discount redemptions over 48 hours. The finance team spends a week reconciling the revenue loss.

### LIKE on NULL Column — Zero Rows Returned Unexpectedly

**Pitfall:** Forgetting that `NULL LIKE 'pattern'` evaluates to NULL, not FALSE. In a WHERE clause, NULL is treated as UNKNOWN, and the row is filtered out. But when combined with NOT LIKE on a nullable column, the NULL behaviour inverts expectations.

```sql
-- ❌ Misunderstanding NULL with NOT LIKE
-- Find all customers whose Notes do NOT contain 'VIP'
SELECT CustomerId, FirstName, LastName
FROM dbo.Customers
WHERE Notes NOT LIKE '%VIP%';
-- This excludes rows WHERE Notes IS NULL!
-- NULL NOT LIKE '%VIP%' → NOT NULL → NULL → filtered out
```

**Symptom:** Customers without any notes are missing from the "non-VIP" list. A marketing campaign targeting non-VIP customers misses 30% of the audience because those customers have NULL notes.

**Fix:**

```sql
-- ✅ Explicitly handle NULL
SELECT CustomerId, FirstName, LastName
FROM dbo.Customers
WHERE (Notes NOT LIKE '%VIP%' OR Notes IS NULL);

-- ✅ Or use ISNULL
SELECT CustomerId, FirstName, LastName
FROM dbo.Customers
WHERE ISNULL(Notes, '') NOT LIKE '%VIP%';
```

**Cost of not fixing:** A CRM segmentation query sends a promotional email to "non-VIP" customers. Due to the NULL exclusion, 30,000 customers with no notes are skipped. The marketing team reports 30% lower-than-expected conversion. The data team takes 3 days to realise the NULL mismatch. The promotion's ROI analysis is invalidated.

### LIKE with Date/Number Conversion — Non-SARGable AND Unreliable

**Pitfall:** Using LIKE for date or number pattern matching by converting to a string. This is non-SARGable (function on column) and introduces format-dependent bugs.

```sql
-- ❌ Horrible: converts every date to string, then does LIKE
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE CONVERT(VARCHAR, OrderDate, 112) LIKE '2024%';
-- Non-SARGable: function on OrderDate prevents index seek
```

**Symptom:** The execution plan shows a Clustered Index Scan with 12,000+ logical reads. The query runs every hour as part of a daily sales report, consuming 144 MB of I/O per day for what should be a 4-page seek.

**Fix:**

```sql
-- ✅ SARGable date range
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01';

-- If the year is dynamic:
DECLARE @Year INT = 2024;
SELECT OrderId, OrderDate, TotalAmount
FROM dbo.Orders
WHERE OrderDate >= DATEFROMPARTS(@Year, 1, 1)
  AND OrderDate < DATEFROMPARTS(@Year + 1, 1, 1);
```

**Cost of not fixing:** A daily sales report that filters by year using `CONVERT(VARCHAR, OrderDate, 112) LIKE '2024%'` takes 8 seconds per run on a 10M row Orders table. The report is called by 50 regional managers every Monday morning. At 8:30 AM, the query blocks other transactions. The blocking chain escalates to a full system stall. The on-call DBA kills the report queries, and managers get incomplete reports.

### Collation Mismatch — Implicit Conversion Defeats Index Seek

**Pitfall:** Joining or filtering a column with one collation using a LIKE pattern with a different collation. The optimiser adds an implicit conversion that prevents index seeks even on trailing-wildcard patterns.

```sql
-- ❌ Collation mismatch: column is Latin1_General_CI_AS, pattern is Latin1_General_BIN
SELECT c.CustomerId, c.LastName
FROM dbo.Customers AS c
WHERE c.LastName LIKE 'Smith%' COLLATE Latin1_General_BIN;
-- Implicit conversion on the column side → index scan
```

**Symptom:** The execution plan shows an Index Scan on IX_Customers_LastName with a `CONVERT_IMPLICIT` warning, even though the pattern `LIKE 'Smith%'` would normally be SARGable. The plan has a `SCALAR OPERATOR` converting every LastName value before evaluating the LIKE.

**Fix:**

```sql
-- ✅ Convert the pattern side instead (no implicit conversion on column)
SELECT c.CustomerId, c.LastName
FROM dbo.Customers AS c
WHERE c.LastName LIKE 'Smith%' COLLATE Latin1_General_CI_AS;
-- If column is Latin1_General_CI_AS, no conversion needed

-- ✅ Or use DATABASE_DEFAULT on the pattern
SELECT c.CustomerId, c.LastName
FROM dbo.Customers AS c
WHERE c.LastName LIKE 'Smith%' COLLATE DATABASE_DEFAULT;
```

**Cost of not fixing:** A data migration script compares customer names across two databases with different collations. The LIKE predicate `LIKE 'Smith%' COLLATE Latin1_General_BIN` on a `Latin1_General_CI_AS` column causes a scan of 10M rows. The migration that should take 2 minutes takes 45 minutes. The deployment window is missed.

### Spoken Answers

**Q: Which LIKE patterns are SARGable and which are not? Why?**

> **Great answer:** A LIKE pattern is SARGable when it has a fixed prefix — when the first character of the pattern is a literal, not a wildcard. `LIKE 'ABC%'` is SARGable because the optimiser converts it to a range seek: it seeks to all keys where the column value is `>= 'ABC' AND < 'ABD'`. `LIKE '%ABC'` is NOT SARGable because the leading wildcard means the optimiser cannot determine a starting point for the seek — it doesn't know what prefix to look for. Similarly, `LIKE '%ABC%'` and `LIKE '_ABC%'` are non-SARGable. The one exception is a character class at the start: `LIKE '[ABC]%'` can sometimes be converted to an OR of three range seeks, but the optimiser typically only does this for small, selective character classes. The core reason is the B-tree index structure: an index seek requires knowing the first key value to search for. A leading wildcard hides this value from the optimiser, forcing a full scan with a residual predicate evaluated per row.

---

**Q: How does EF Core translate `StartsWith`, `Contains`, and `EndsWith`? Are they SARGable?**

> **Great answer:** EF Core translates `StartsWith` to `LIKE @term + '%'` — this is SARGable because it has a trailing wildcard. The optimiser can perform an Index Seek on the column for the range starting with the search term. `Contains` translates to `LIKE '%' + @term + '%'` — this is NOT SARGable because the leading wildcard prevents the optimiser from deriving a seek range. `EndsWith` translates to `LIKE '%' + @term` — also NOT SARGable for the same reason. The key takeaway: `StartsWith` is safe for indexed columns at any scale, while `Contains` and `EndsWith` will scan the entire table. If you need `Contains`-style search at scale on a table with millions of rows, you should use full-text search instead — EF Core supports `EF.Functions.Contains()` for this. For `EndsWith` at scale, a computed column with `REVERSE(column)` combined with `StartsWith(REVERSE(term))` can make suffix searches SARGable.

---

**Q: Compare `LIKE '%term%'` vs `CONTAINS(column, 'term')` — when would you use each?**

> **Great answer:** `LIKE '%term%'` is a basic string pattern match that works on any column in any edition of SQL Server. It is always non-SARGable — it forces a full table or clustered index scan. It does not understand word boundaries, stemming, thesaurus, or ranking. It simply checks if the substring exists anywhere in the string. `CONTAINS` is a full-text search predicate that requires a full-text index on the column. It is SARGable in the sense that it uses the full-text index to find matching rows without scanning the table. It supports stemming (finding 'running' when searching for 'run'), thesaurus expansions, proximity search (NEAR), and weighted ranking. I use `LIKE '%term%'` only on small tables (under 10K rows), for exact substring matching on short strings, or when full-text search is not available due to edition constraints or administrative overhead. I use `CONTAINS` on any table over 100K rows where infix text search is required, especially for large text columns like Notes or Descriptions. The performance difference is dramatic: on a 1M row table, `LIKE '%term%'` reads 12,000+ pages and takes 200+ ms, while `CONTAINS` reads only the full-text index pages (~50 pages) and the matching data pages (~10–100), taking 10–20 ms.

### Interview Trigger

The defining LIKE question: "What is the difference between `LIKE '%abc'` and `LIKE 'abc%'` in terms of performance?" A candidate who says "the first matches ending, the second matches beginning" fails — they understand the wildcard but not the performance implication. A candidate who says "the trailing wildcard `LIKE 'abc%'` can use an index seek while the leading wildcard `LIKE '%abc'` forces a scan" passes. The follow-up: "How does the optimiser convert `LIKE 'abc%'` into a range?" — the great candidate describes the `>= 'abc' AND < 'abd'` range conversion and how it maps to the B-tree structure. The next follow-up: "What if the column has a case-insensitive collation?" — "The range boundaries adjust to the collation sort weights, but the mechanism is identical."

## Decision Framework

### When to Apply

```mermaid
flowchart TD
    A[Need string pattern search] --> B{Pattern type?}
    B -->|Prefix search 'term%'| C{Column indexed?}
    B -->|Infix search '%term%'| D{Table size?}
    B -->|Suffix search '%term'| E{Table size?}
    B -->|Exact match 'term'| F[Use = not LIKE]
    C -->|Yes| G[LIKE optimizer to Index Seek - SARGable]
    C -->|No| H[Create index or accept scan]
    D -->|<100K rows| I[LIKE '%term%' acceptable scan]
    D -->|>=100K rows| J{Full-text index available?}
    D -->|>=100K rows, no full-text| K[Consider REVERSE + prefix or third-party search]
    E -->|<50K rows| L[LIKE '%term' acceptable scan]
    E -->|>=50K rows| M[REVERSE computed column + prefix LIKE]
    M --> N[REVERSE(column) LIKE REVERSE(suffix) + '%']
    J -->|Yes| O[Use CONTAINSTABLE for infix search]
    J -->|No| P[Add full-text index or optimize with filtered index]
    G --> Q[Covering index for optimal plan]
    O --> R[Create full-text catalog + index]
```

### Application Checklist

- [ ] LIKE pattern uses trailing wildcard (SARGable) wherever possible
- [ ] Leading wildcard only used on tables under ~100K rows or with full-text fallback
- [ ] ESCAPE clause used when pattern contains literal `%` or `_` characters
- [ ] NULL handling explicitly addressed for NOT LIKE queries (three-valued logic)
- [ ] Parameterised LIKE queries use OPTION (RECOMPILE) or pattern-specific branches
- [ ] Index exists on the column used in LIKE when SARGable pattern is expected
- [ ] Reverse computed column considered for suffix-search performance
- [ ] Date/numeric LIKE patterns replaced with proper range predicates
- [ ] Full-text search evaluated for infix search on large text columns
- [ ] EF Core uses `StartsWith` for SARGable search, not `Contains` for large tables
- [ ] Collation is consistent between column and pattern (no implicit conversion)

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Trailing wildcard Seek: fast prefix search (4 reads)|Cannot find strings where the match is not at the start|
|Leading wildcard Scan: matches anywhere in string|Scans entire table (12,000+ reads per 1M rows)|
|Full-text search: fast infix search, stemming, ranking|Requires full-text index setup, moderate admin overhead|
|REVERSE column trick: fast suffix search|Storage for computed column + index write overhead|
|ESCAPE clause: safe literal wildcard matching|Pattern is slightly less readable|

### Scale Thresholds

- **< 10K rows**: Leading wildcard `LIKE '%term%'` is acceptable for ad-hoc queries and admin scripts (scan cost is ~100 reads).
- **10K–100K rows**: Leading wildcard starts to hurt (~1,200 reads). Consider prefix-only search for user-facing features. Trailing wildcard `LIKE 'term%'` is still efficient with an index (3–8 reads).
- **> 100K rows**: Leading wildcard is unacceptable for production queries. Use full-text search or a REVERSE computed column trick. Trailing wildcard remains efficient with a covering index (< 10 reads).
- **> 10M rows**: Only trailing wildcard on a covering index or full-text search. Leading wildcard `LIKE '%term%'` is catastrophic (120,000+ logical reads per query). Implement Elasticsearch or Azure Cognitive Search for any infix/search-at-scale requirement.
- **Concurrent users > 100**: Even trailing wildcard scans of ~4 reads each are fine, but 100 concurrent full scans at 12,000 reads each = 1.2M logical reads in a short window. The buffer pool thrashes. Optimise every LIKE to SARGable or use full-text search.

## Self-Check

### Conceptual Questions

1. What is the range seek transformation the optimiser performs for `LIKE 'ABC%'`?
2. Why is `LIKE '%ABC'` non-SARGable while `LIKE 'ABC%'` is SARGable?
3. What does `SET STATISTICS IO` show for a trailing-wildcard LIKE query vs a leading-wildcard LIKE query?
4. What common mistake with `NOT LIKE` and NULL columns causes missing rows in query results?
5. How does EF Core translate `c.LastName.StartsWith("Smith")` — what SQL does it generate? Is it SARGable?
6. How would you implement a suffix search (e.g., find all emails ending with '@gmail.com') that is SARGable?
7. What is the difference between `LIKE '%term%'` and `CONTAINS(column, 'term')` in terms of both performance and matching semantics?
8. At what table size does a leading-wildcard LIKE become unacceptable for a user-facing search feature?
9. What index supports a trailing-wildcard LIKE on LastName? Would a covering index help?
10. Explain in 60 seconds, for a senior interviewer, why `LIKE 'ABC%'` can use an Index Seek while `LIKE '%ABC%'` cannot — include the B-tree structure, the seek predicate, and the residual predicate.

<details>
<summary>Answers</summary>

1. The optimiser converts `LIKE 'ABC%'` to a range: `Column >= 'ABC' AND Column < 'ABD'` (or more precisely, `Column >= 'ABC' AND Column < 'ABD'` where 'D' is the next Unicode code point after 'C' in the column's collation). This range is pushed into an Index Seek's SeekPredicate.

2. `LIKE 'ABC%'` has a fixed prefix 'ABC' that the optimiser can use as the starting point of a range seek in the B-tree. `LIKE '%ABC'` has no fixed prefix — the wildcard `%` at the start means the first character of the match can be anything. The optimiser cannot determine a starting key value, so it must scan every leaf page of the index.

3. Trailing wildcard (`LIKE 'Smith%'`): `Scan count 1, logical reads 4` (Index Seek, ~4 pages). Leading wildcard (`LIKE '%smith%'`): `Scan count 1, logical reads 12,450` (Clustered Index Scan, full table). The difference is 3,000x.

4. `NULL NOT LIKE '%pattern%'` evaluates to `NOT NULL` → NULL → UNKNOWN. In a WHERE clause, UNKNOWN causes the row to be filtered out. So `WHERE Notes NOT LIKE '%VIP%'` excludes rows WHERE Notes IS NULL, even though semantically those "non-VIP" customers should be included. The fix: `WHERE (Notes NOT LIKE '%VIP%' OR Notes IS NULL)`.

5. `c.LastName.StartsWith("Smith")` generates `[c].[LastName] LIKE N'Smith%'`. This IS SARGable. EF Core appends the `+ '%'` to the parameter value. The optimiser sees the trailing wildcard and can perform an Index Seek.

6. Add a computed column: `ALTER TABLE Customers ADD EmailReversed AS REVERSE(Email) PERSISTED;` with an index: `CREATE INDEX IX_Customers_EmailReversed ON Customers (EmailReversed);`. Then query: `WHERE EmailReversed LIKE REVERSE('@gmail.com') + '%'`. This converts the suffix search into a prefix search, which is SARGable.

7. Performance: `LIKE '%term%'` does a full clustered index scan (12,000+ reads for 1M rows). `CONTAINS` uses a full-text index lookup (~50 reads) + key lookups for matching rows. Matching semantics: LIKE does exact substring matching — it checks if the literal characters appear in the column value. CONTAINS does word-aware matching by default — it matches word boundaries, handles stemming (term → terms, terming), and supports thesaurus expansions. CONTAINS is also case-insensitive regardless of column collation.

8. The threshold depends on concurrency, but for a user-facing search feature: above ~10K rows, the scan time becomes noticeable (100+ ms). Above ~100K rows, the scan consistently takes 1+ second and starts blocking other queries. Above ~1M rows, the scan takes 5+ seconds and is unacceptable for interactive search. The hard threshold for "must fix" is ~100K rows for user-facing features.

9. A regular non-clustered index on LastName supports trailing-wildcard LIKE. A covering index (e.g., `CREATE INDEX IX_Customers_LastName_Cover ON Customers (LastName) INCLUDE (FirstName, Email)`) avoids key lookups for the extra columns in the SELECT list, making the plan even more efficient by keeping all required data in the index leaf pages.

10. "SQL Server indexes are B-trees, where the leaf level contains sorted key values. An index seek navigates from the root to the leaf, finding the first matching key and then scanning forward sequentially. For `LIKE 'ABC%'`, the optimiser calculates the range ['ABC', 'ABD') — it seeks to the first key >= 'ABC' and reads forward until the first key >= 'ABD'. This works because we know the starting value, and the B-tree structure supports positional access. For `LIKE '%ABC%'`, we cannot calculate a starting value — the first character of the match is unknown. The optimiser must scan every leaf page and apply the LIKE as a residual predicate, comparing the pattern against each column value. The B-tree's sorted key structure provides no benefit when the prefix is unknown. The performance difference is proportional to the data size: a seek reads 3–8 pages; a full scan reads every leaf page — thousands or tens of thousands of pages on a large table."

</details>

**Challenge 3 — Explain the execution plan**

Given this query and its execution plan:

```sql
SELECT CustomerId, LastName, Email
FROM dbo.Customers
WHERE LastName LIKE @Pattern;
```

The execution plan shows:
```
[Clustered Index Scan] → [Filter] → [SELECT]
Estimated Cost: 12.3  |  Actual Logical Reads: 62,250
```

But when the same query is run with `LIKE 'Smith%'` using OPTION (RECOMPILE):

```
[Index Seek IX_Customers_LastName (Range: >= 'Smith' AND < 'Smiti')] → [Key Lookup (Email)] → [Nested Loops] → [SELECT]
Estimated Cost: 0.006  |  Actual Logical Reads: 8
```

Why does the parameterised version use a Clustered Index Scan while the literal version uses an Index Seek?

<details>
<summary>Solution</summary>

**Why the parameterised version scans:** The optimiser compiles the plan without knowing the value of `@Pattern`. It cannot determine whether the pattern starts with a wildcard or a literal prefix. Since a leading wildcard would make the Index Seek invalid (the range `>= '%' AND < '%'` returns no rows or wrong rows), the optimiser conservatively chooses a Clustered Index Scan with a residual Filter. This plan is correct for any pattern value but is optimal for none.

**Why the literal version seeks:** With `LIKE 'Smith%'`, the optimiser sees the literal prefix 'Smith' at compile time. It can derive the range `>= 'Smith' AND < 'Smiti'` and push it into the SeekPredicate of the Index Seek operator. The Key Lookup retrieves the Email column not covered by the index.

**To get the Seek plan with parameterisation:**

```sql
-- Option A: OPTION (RECOMPILE) — optimiser sees actual value every time
SELECT CustomerId, LastName, Email
FROM dbo.Customers
WHERE LastName LIKE @Pattern
OPTION (RECOMPILE);
-- Compiles per execution: if @Pattern = 'Smith%', generates Seek; if '%smith%', generates Scan

-- Option B: Pattern-specific procedures
CREATE PROCEDURE dbo.SearchByPrefix @Prefix NVARCHAR(100) AS
    SELECT CustomerId, LastName, Email
    FROM dbo.Customers
    WHERE LastName LIKE @Prefix + '%';
```

**Tradeoff with OPTION (RECOMPILE):** Each execution pays ~1–2 ms for recompilation. On a high-frequency query (1000+ executions/sec), this CPU overhead can exceed the scan cost. On an infrequent user-facing search, the benefit of the correct plan far outweighs the recompile cost.

</details>

**Challenge 4 — Diagnose the NULL problem**

A marketing query identifies customers who have NOT opted out of email communications. The `OptOutNotes` column contains NULL for customers who have never opted out, and text descriptions for those who have (e.g., 'Opted out on 2024-03-15', 'Complaint — opt out requested'):

```sql
-- Find customers eligible for email marketing

```

The query returns only 60,000 customers, but the business knows there are 150,000 customers without opt-out notes. Diagnose the bug and fix it.

<details>
<summary>Solution</summary>

**Root cause:** `NULL NOT LIKE '%opt out%'` evaluates to `NOT NULL` → `NULL` → UNKNOWN. In a WHERE clause, UNKNOWN filters the row out. Customers with `OptOutNotes IS NULL` are excluded from the result, even though they have never opted out and should be included.

**Fix:**

```sql
-- ✅ Explicitly handle NULL
SELECT CustomerId, FirstName, LastName, Email

SELECT CustomerId, FirstName, LastName, Email
FROM dbo.Customers
WHERE ISNULL(OptOutNotes, '') NOT LIKE '%opt out%';

-- ✅ Using COALESCE
SELECT CustomerId, FirstName, LastName, Email
FROM dbo.Customers
WHERE COALESCE(OptOutNotes, '') NOT LIKE '%opt out%';
```

**Logical reads:** Scan of Clustered Index (same as before — the NOT LIKE predicate is non-SARGable regardless of the NULL fix). Full-text search would be a better long-term solution for this text-matching query.
