### Where This Fits

A GIN (Generalized Inverted Index) on a JSONB column creates index entries for every key-value pair at every nesting level in every document, enabling containment (`@>`), existence (`?`), and path-exists operators to perform index seeks instead of sequential scans. Every .NET backend engineer using PostgreSQL with JSONB must understand that a GIN index is NOT a traditional B-tree — it is an inverted index that maps individual values back to the documents that contain them, similar to a full-text search index. The index does not support range queries, ordering, or equality on extracted text values (use a B-tree expression index for those). The interview signal is advanced: GIN indexes on JSONB represent PostgreSQL's unique approach to indexing semi-structured data, and understanding their bitmap scan + recheck mechanism demonstrates deep knowledge of PostgreSQL's storage engine.

## Core Mental Model

A GIN index on a JSONB column decomposes each JSON document into a set of index entries — one entry per key-value pair, with arrays expanded element-by-element and nested objects flattened recursively. The index stores a mapping from each unique entry value to the list of heap tuples (TIDs) that contain it. When a query uses `@>` (contains), PostgreSQL calls the GIN `extractQuery` function to decompose the query JSON into the same entry format, does a bitmap index scan to find all TIDs that match every entry, and then performs a recheck pass on candidate rows to eliminate false positives. The false positives arise because the GIN index cannot distinguish between `{"a": {"b": 1}}` and `{"a": {"b": 1}, "b": 1}` — both produce index entries for `"a" -> {"b":1}` and `"b" -> 1`, so a query for `@> {"b": 1}` matches both, but the recheck correctly eliminates documents that have `"b": 1` only at the wrong nesting level. The recognition pattern: if you see `Bitmap Index Scan` + `Recheck Cond` in a PostgreSQL EXPLAIN plan for a JSONB query, you are looking at a GIN index in action.

### Classification

GIN is an access method (index type) in PostgreSQL's pluggable indexing framework. For JSONB, it enables operators to use a bitmap scan strategy rather than sequential scan + per-row eval. The operators GIN supports for JSONB are: `@>` (containment), `?` (key exists at top level), `?|` (any key exists), `?&` (all keys exist), and `jsonb_path_exists` (SQL/JSON path existence). The GIN index does NOT support: `->>` (text extract), `->` (JSON object extract), `@?` (path predicate), or `@@` (jsonpath match) — those require B-tree expression indexes or specific jsonpath indexes in PostgreSQL 16+. SARGability applies only to the operators listed above: `@>` is SARGable with a GIN index, `->>` is not SARGable through GIN.

## Deep Mechanics

### How the Engine Executes This

**Index creation (`CREATE INDEX ON table USING GIN (jsonb_col)`):**

1. PostgreSQL scans the JSONB column and calls `gin_extract_jsonb()` for each row. This function recursively walks the JSONB binary structure and produces a `Datum` array of index entries. For each unique path+value combination, it creates an entry by hashing the key-value pair into a 32-bit `int32` entry. Arrays are expanded: each array element becomes a separate entry. Nested objects are flattened: `{"a": {"b": 1}}` produces entries for both `a -> {"b":1}` and `b -> 1`.
2. Each entry is inserted into the GIN index's B-tree of posting trees. The GIN index uses a tree-of-trees structure: a search tree (B-tree) maps entry values to posting trees, and each posting tree maps key values to TID lists (ItemPointerData = page + offset). The posting tree compresses TID lists using `GinPostingList` — a run-length encoded list of TID deltas.
3. If the `fastupdate` option is enabled (default), new entries are written to an in-memory pending list rather than directly into the main index structure, trading query-time merge cost for faster INSERT performance.

**Query execution (`WHERE jsonb_col @> '{"key": "val"}'`):**

1. **GIN extractQuery:** PostgreSQL calls `gin_consistent_jsonb()` passing the query JSONB. The function extracts entries for the query document. For `@>`, the query entries represent the JSON structure to find: each key-value pair at each nesting level.
2. **Bitmap Index Scan:** PostgreSQL performs a bitmap index scan on the GIN index. For each required entry, it looks up the posting tree and retrieves the TID bitmap. It combines bitmaps with AND (for `@>` — all entries must match) or OR (for `?|` — any entry matches).
3. **Bitmap Heap Scan:** The resulting bitmap is converted to a list of heap page blocks and scanned in physical order. For each candidate row, the `Recheck Cond` evaluates the original `@>` expression against the full JSONB document. This eliminates false positives from the GIN index's permissive entry matching.
4. **False positive example:** A document with `{"status": "active", "user": {"role": "admin"}}` produces GIN entries for `status -> active` and `role -> admin`. A query for `@> {"role": "admin"}` also matches because `admin` is a valid GIN entry. The recheck correctly filters this out because the JSON path for `role` is `user.role`, not top-level `role`.

**GIN pending list:**

When `fastupdate = ON` (default), INSERTs do not immediately update the main GIN index. Instead, entries accumulate in a per-index pending list stored in memory (and backed by a separate fork file `index_name.gin`). When the pending list reaches `gin_pending_list_limit` (default 4MB, configurable), an automatic cleanup process merges the pending entries into the main index structure. During this merge, the GIN index is locked for writes, causing INSERT latency spikes. Queries must consult both the main index and the pending list — if the pending list is large, `pending` entries are checked via sequential scan, degrading query performance until cleanup completes.

```sql
-- ============================================================
-- Create GIN index on JSONB column
-- ============================================================
CREATE INDEX ix_product_catalog_attributes_gin
ON public.product_catalog USING GIN (attributes);

-- ============================================================
-- Query 1: Containment (@>) — uses GIN
-- Find all electronics with black color
-- ============================================================
EXPLAIN (ANALYZE, BUFFERS)
SELECT product_id, sku, attributes->>'name' AS product_name
FROM public.product_catalog
WHERE attributes @> '{"category": "electronics", "specs": {"color": "black"}}';
-- Expected plan:
-- Bitmap Heap Scan on product_catalog
--   Recheck Cond: (attributes @> '{"category": "electronics", "specs": {"color": "black"}}'::jsonb)
--   Buffers: shared hit=5
--   ->  Bitmap Index Scan on ix_product_catalog_attributes_gin
--         Index Cond: (attributes @> '{"category": "electronics", "specs": {"color": "black"}}'::jsonb)
--         Buffers: shared hit=2

-- ============================================================
-- Query 2: Key existence (?) — uses GIN
-- Find all products that have a 'ports' key
-- ============================================================
SELECT product_id, sku
FROM public.product_catalog
WHERE attributes ? 'ports';

-- ============================================================
-- Query 3: Any key exists (?|) — uses GIN
-- Find all products with either 'ports' or 'wattage'
-- ============================================================
SELECT product_id, sku
FROM public.product_catalog
WHERE attributes ?| ARRAY['ports', 'wattage'];

-- ============================================================
-- Query 4: All keys exist (?&) — uses GIN
-- Find products with both 'specs' AND 'brand'
-- ============================================================
SELECT product_id, sku
FROM public.product_catalog
WHERE attributes ?& ARRAY['specs', 'brand'];

-- ============================================================
-- Query 5: jsonb_path_exists — uses GIN (PostgreSQL 12+)
-- Find products where specs has a color property
-- ============================================================
SELECT product_id, sku
FROM public.product_catalog
WHERE jsonb_path_exists(attributes, '$.specs.color');

-- ============================================================
-- Query 6: Operator -> does NOT use GIN
-- ============================================================
EXPLAIN (ANALYZE, BUFFERS)
SELECT product_id, sku
FROM public.product_catalog
WHERE attributes->>'category' = 'electronics';
-- Expected plan:
-- Seq Scan on product_catalog
--   Filter: ((attributes ->> 'category'::text) = 'electronics'::text)
-- GIN does NOT support ->>. Need BTREE expression index:
CREATE INDEX ix_product_catalog_category_btree
ON public.product_catalog ((attributes->>'category'));
```

### Execution Plan Analysis

For `SELECT * FROM product_catalog WHERE attributes @> '{"category":"electronics","specs":{"color":"black"}}'`:

```
Expected plan shape with GIN index:
  Bitmap Index Scan (ix_product_catalog_attributes_gin)
    Index Cond: (attributes @> '{"category":"electronics","specs":{"color":"black"}}'::jsonb)
    → Bitmap Heap Scan (product_catalog)
        Recheck Cond: (attributes @> '{"category":"electronics","specs":{"color":"black"}}'::jsonb)
        → SELECT

Expected plan shape without GIN index:
  Seq Scan (product_catalog)
    Filter: (attributes @> '{"category":"electronics","specs":{"color":"black"}}'::jsonb)
    → SELECT

Cost comparison:
  With GIN: Bitmap Index Scan cost = ~50, Bitmap Heap Scan cost = ~430, total = ~480
  Without:  Seq Scan cost = ~48,000 (100x more)
```

The GIN bitmap scan reads only the index entries for `category -> electronics` and `color -> black`, combines the TID bitmaps, and fetches only matching heap pages. The recheck condition distinguishes between `{"specs":{"color":"black"}}` (nested correctly) and `{"color":"black"}` at the top level (wrong path).

### Failure Modes

**Failure Mode 1 — GIN index created but ->> operator still slow:** The developer creates a GIN index but still sees sequential scans because the query uses `->>` instead of `@>`. The `->>` operator returns `text` and cannot use the GIN index.

**Failure Mode 2 — Pending list growing too large:** With `fastupdate = ON` and high write volume, the pending list grows beyond `gin_pending_list_limit`, causing cleanup cycles that spike INSERT latency. Query: `SELECT * FROM gin_clean_pending_list('index_name'::regclass);`

**Failure Mode 3 — Index bloat from frequent UPDATEs:** Each UPDATE to a JSONB column generates new GIN index entries and marks old ones dead. Without aggressive autovacuum, dead entries accumulate, increasing index size and scan cost.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Production schema: Event table with JSONB + GIN index
-- ============================================================

CREATE TABLE IF NOT EXISTS public.order_events (
    event_id BIGSERIAL PRIMARY KEY,
    order_id INTEGER NOT NULL,
    event_type VARCHAR(50) NOT NULL,
    event_data JSONB NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Create GIN index for containment queries on event_data
CREATE INDEX ix_order_events_data_gin
ON public.order_events USING GIN (event_data);

-- Create GIN index with jsonb_path_ops for faster @> (no existence queries)
-- Use this if you only need @> and never use ? / ?| / ?&
CREATE INDEX ix_order_events_data_path_gin
ON public.order_events USING GIN (event_data jsonb_path_ops);

-- Create BTREE expression indexes for specific hot paths
CREATE INDEX ix_order_events_order_id
ON public.order_events (order_id, created_at DESC);

CREATE INDEX ix_order_events_event_type
ON public.order_events (event_type, created_at DESC);

-- ============================================================
-- Query: Find all events for orders with specific item sku
-- Uses GIN index on event_data
-- ============================================================
SELECT
    event_id,
    order_id,
    event_type,
    event_data->>'status' AS status,
    created_at
FROM public.order_events
WHERE event_data @> '{"items": [{"sku": "SKU-001"}]}'
ORDER BY created_at DESC
LIMIT 100;

-- ============================================================
-- Query: Find events where a specific key exists
-- Uses GIN index with ? operator
-- ============================================================
SELECT
    event_id,
    order_id,
    event_type,
    created_at
FROM public.order_events
WHERE event_data ? 'fraud_check'
ORDER BY created_at DESC;

-- ============================================================
-- Query: Find events with any of several keys
-- Uses GIN index with ?| operator
-- ============================================================
SELECT
    event_id,
    order_id,
    event_type,
    created_at
FROM public.order_events
WHERE event_data ?| ARRAY['refunded', 'cancelled', 'disputed']
ORDER BY created_at DESC;

-- ============================================================
-- Tune GIN index settings for this table
-- ============================================================
-- Increase pending list limit to reduce cleanup frequency
-- (reduces INSERT latency spikes at cost of slightly slower queries)
ALTER INDEX ix_order_events_data_gin SET (gin_pending_list_limit = 65536);  -- 64MB

-- Disable fastupdate for consistent query performance (slower INSERTs)
```

### GIN vs GiST Comparison for JSONB

```sql
-- PostgreSQL offers both GIN and GiST indexes for JSONB.
-- Key differences:

-- GIN (default choice):
CREATE INDEX ix_jsonb_gin ON public.table USING GIN (jsonb_col);
-- 2-3x faster lookups for @> and ?
-- Larger index (2-3x data size)
-- Slower writes (more index entries per document)
-- Supports @>, ?, ?|, ?&

-- GiST (alternative):
CREATE INDEX ix_jsonb_gist ON public.table USING GiST (jsonb_col jsonb_path_ops);
-- 2-3x slower lookups than GIN
-- Smaller index (~1x data size)
-- Faster writes (fewer entries per document)
-- Supports @>, @?, @@ (jsonpath match)
-- Better for infrequent queries on high-write tables
```

## Gotchas and Production Pitfalls

### Gotcha 1 — GIN Index Does Not Cover Text Extraction Operators

**Pitfall:** Creating a GIN index on a JSONB column and expecting `WHERE jsonb_col->>'field' = 'value'` to use it.

```sql
-- ❌ GIN index is created but ->> does NOT use it
CREATE INDEX ON public.products USING GIN (attributes);

SELECT * FROM public.products
WHERE attributes->>'category' = 'electronics';
-- Seq Scan — GIN cannot accelerate ->>
```

**Symptom:** Sequential scan on every query that uses `->>` for equality or range filtering. The GIN index exists but is unused, wasting storage.

**Fix:**

```sql
-- ✅ Add BTREE expression index for specific paths accessed with ->>
CREATE INDEX ON public.products ((attributes->>'category'));
```

**Cost of not fixing:** 100% sequential scans on all `->>` queries. For a 10M row table, each query reads all 10M rows instead of seeking directly.

### Gotcha 2 — GIN Index Write Amplification

**Pitfall:** Not accounting for the write overhead of GIN indexes on high-volume INSERT tables.

```sql
-- A document with 50 keys creates ~50 GIN index entries per INSERT
-- At 1000 INSERTs/sec: 50,000 GIN index writes/sec
-- Plus UPDATEs: removes old entries, adds new ones

-- Monitor write amplification:
SELECT
    n_tup_ins,
    n_tup_upd,
    n_tup_del,
    n_tup_hot_upd,
    idx_blks_read + idx_blks_hit AS total_idx_blks
FROM pg_stat_user_tables
WHERE relname = 'order_events';
```

**Symptom:** INSERT latency grows from 1ms to 15ms as the GIN index grows. Autovacuum runs constantly, consuming CPU and I/O.

**Fix:**

```sql
-- Tune GIN pending list to batch index writes
ALTER INDEX ix_order_events_data_gin SET (gin_pending_list_limit = 131072);  -- 128MB

-- Or disable fastupdate for consistent latency:
ALTER INDEX ix_order_events_data_gin SET (fastupdate = OFF);

-- Or drop GIN index and use relational columns for queried fields:
ALTER TABLE public.order_events ADD COLUMN event_category VARCHAR(50)
    GENERATED ALWAYS AS (event_data->>'category') STORED;
CREATE INDEX ON public.order_events (event_category);
```

**Cost of not fixing:** Write path degradation affects all INSERTs. At 2000 INSERTs/sec, latency spikes from pending list cleanup cause application timeouts and queue buildup.

### Gotcha 3 — jsonb_path_ops Loses Existence Operators

**Pitfall:** Using `jsonb_path_ops` operator class without realizing it drops support for `?`, `?|`, and `?&`.

```sql
-- ❌ Creating index with jsonb_path_ops:
CREATE INDEX ON public.products USING GIN (attributes jsonb_path_ops);

-- This ? query will NOT use the index:
SELECT * FROM public.products WHERE attributes ? 'price';
-- Seq Scan — jsonb_path_ops does not support ? operator
```

**Symptom:** `EXPLAIN` shows sequential scans for key existence queries that were previously using the GIN index. The DBA assumes all JSONB queries use the GIN index.

**Fix:**

```sql
-- ✅ Use default jsonb_ops (or create a second GIN with default):
CREATE INDEX ON public.products USING GIN (attributes);  -- default jsonb_ops
-- Drop the jsonb_path_ops index if ? queries are important
```

**Cost of not fixing:** Key existence queries (`?`, `?|`, `?&`) become sequential scans. If the application frequently checks for key existence (e.g., `WHERE event_data ? 'fraud_check'`), this degrades from ~15ms to ~3000ms.

### Gotcha 4 — GIN False Positives and Recheck Overhead

**Pitfall:** Assuming the GIN index returns exact results without recheck, or that the recheck is free.

```sql
-- GIN index entry flattening creates false positives:
-- Document: {"user": {"role": "admin"}}
-- GIN entries include: role -> admin (flattened from user.role)
-- Query: @> {"role": "admin"}
-- GIN bitmap scan finds the document (admin is a GIN entry)
-- RECHECK correctly eliminates it (role is not at top level)
```

**Symptom:** `EXPLAIN (ANALYZE)` shows high `Rows Removed by Index Recheck` count. For complex nested documents, the recheck can reject 50-90% of bitmap candidates, meaning the bitmap heap scan fetches many rows only to filter them out.

**Fix:**

```sql
-- Use jsonb_path_ops operator class to reduce false positives:
CREATE INDEX ON public.products USING GIN (attributes jsonb_path_ops);
-- jsonb_path_ops creates single entries per path (e.g., "user.role" -> "admin")
-- instead of separate entries per key+value. Fewer false positives.
-- Tradeoff: larger index, loses ? operator support.

-- Or restructure queries to use more specific containment:
-- Instead of: @> {"role": "admin"}
-- Use:      @> {"user": {"role": "admin"}}
```

**Cost of not fixing:** Bitmap heap scan reads many pages that the recheck discards. At low selectivity (<1%), this overhead is small. At high selectivity (>10%), the recheck becomes a significant portion of total query time.

### Gotcha 5 — GIN Index Does Not Support ORDER BY

**Pitfall:** Expecting a GIN index on JSONB to support sorting or range queries.

```sql
-- ❌ GIN cannot help with ORDER BY on extracted values:
SELECT * FROM public.products
WHERE attributes @> '{"category": "electronics"}'
ORDER BY (attributes->>'price')::numeric DESC
LIMIT 10;
-- The GIN index handles the @> filter, but the ORDER BY requires
-- a separate sort or a BTREE index on the expression.
```

**Symptom:** Despite a GIN index, the query plan includes an explicit Sort node that sorts the filtered results, potentially spilling to disk for large result sets.

**Fix:**

```sql
-- ✅ Add BTREE expression index for the sort column
CREATE INDEX ON public.products
    (((attributes->>'price')::numeric) DESC)
WHERE attributes @> '{"category": "electronics"}';
-- Filtered BTREE index covers both the filter and the sort
```

**Cost of not fixing:** Sort spill to disk for large result sets. With 500K matching rows, an in-memory sort could require 100MB+ of work_mem, and a disk sort adds temp file I/O.

### Gotcha 6 — GIN Index with TOASTed Values

**Pitfall:** Storing large JSONB values that are TOASTed (compressed/externally stored) creates GIN index entries that reference the compressed storage, causing decompression overhead during recheck.

**Symptom:** The `Recheck Cond` in `EXPLAIN ANALYZE` shows high `Buffers: shared hit` but unexpectedly high CPU time. PostgreSQL must decompress TOASTed JSONB values to perform the recheck.

**Fix:**

```sql
-- Compress large JSONB values to reduce TOAST overhead:
ALTER TABLE public.order_events
    ALTER COLUMN event_data SET STORAGE MAIN;  -- keep in main table, compress

-- Or increase toast_tuple_target to keep more data inline:
ALTER TABLE public.order_events SET (toast_tuple_target = 4080);  -- default 2040
```

**Cost of not fixing:** Recheck on TOASTed rows requires random I/O to the TOAST table plus decompression CPU. For large documents (>8KB), the recheck can be 10-50x slower than recheck on inline values.

### Spoken Answers

**Q1: What is a GIN index, and how does it work differently from a B-tree index for JSONB data?**

> **Average answer:** "GIN stands for Generalized Inverted Index. It's used for indexing composite values like JSONB or arrays. It's different from B-tree because it maps values to rows instead of rows to values."

> **Great answer:** "GIN stands for Generalized Inverted Index. Unlike a B-tree, which maps a single key value to a specific row location (like a phone book listing), a GIN index is an inverted index that maps individual atomic values to a list of all rows that contain them. For JSONB, when you create `CREATE INDEX ON table USING GIN (jsonb_col)`, PostgreSQL calls `gin_extract_jsonb()` for each row, which recursively walks the JSONB binary structure and extracts every key-value pair at every nesting level as a separate index entry. Arrays are expanded element-by-element. Nested objects are flattened. So a document like `{"name": "Alice", "items": [{"sku": "A1"}]}` produces GIN entries for `name -> Alice`, `items -> [...]`, and `sku -> A1`. The GIN index stores these entries in a tree-of-trees structure: a search B-tree maps each unique entry to a posting tree of TIDs. When a query uses `@>`, the GIN `extractQuery` function decomposes the query JSON into the same entry format, and a bitmap index scan finds all TIDs matching every required entry, combines them with AND, and passes candidate rows to a recheck that verifies the full containment — needed because GIN's flattening can cause false positives. The critical difference from B-tree: GIN supports multi-valued matching (find rows where ANY of these values appear), while B-tree supports single-value equality and range scans. GIN also does not support ORDER BY or range queries on extracted values."

**Q5: GIN vs GiST for JSONB: when would you choose each?**

> **Great answer:** "For JSONB, GIN is the default and preferred choice for most workloads. GIN is 2-3x faster than GiST for containment (`@>`) and existence (`?`) lookups because GIN's posting tree structure is optimized for finding all rows matching a given entry. GIN also uses bitmap index scans, which are very efficient for queries with moderate to high selectivity. The tradeoff is that GIN indexes are larger (2-3x the JSONB data size) and slower to maintain on writes (50-100% more overhead). GiST indexes on JSONB are smaller (~1x the data size) and faster to write (30-50% less overhead), but lookups are 2-3x slower. GiST also supports the `@?` (path predicate) and `@@` (jsonpath match) operators, which GIN does not. I choose GIN when: (1) read performance is the priority, (2) queries use `@>` and `?` primarily, (3) the write rate is moderate (<500 INSERTs/sec). I choose GiST when: (1) write volume is high (>1000 INSERTs/sec) and reads are less frequent, (2) the query pattern uses jsonpath operators (`@?`, `@@`) that GIN doesn't support, or (3) index storage budget is tight and the read performance difference is acceptable. In practice, about 80% of production JSONB indexes should be GIN, 20% GiST."

**Q6: What does the execution plan look like for `WHERE jsonb_col @> '{"key":"val"}'`?**

> **Great answer:** "The execution plan has two nodes: a Bitmap Index Scan on the GIN index, followed by a Bitmap Heap Scan with a Recheck Cond. The Bitmap Index Scan calls the GIN index's `consistent` function, which extracts entries from the query JSONB and looks up each entry in the GIN search tree. For `@>`, all query entries must match — the function combines the TID bitmaps from each matched entry using AND. The result is a bitmap of heap page blocks that contain candidate rows. The Bitmap Heap Scan reads these pages in physical order (avoiding random I/O) and for each row applies the Recheck Cond — which is the original `@>` expression evaluated against the full JSONB document. The recheck is necessary because GIN's permissive entry extraction can produce false positives: a document with `{"status": "active", "user": {"role": "admin"}}` produces GIN entries for both `status -> active` and `role -> admin`. A query for `@> {"role": "admin"}` matches both — but the recheck correctly eliminates this document because `role` is nested under `user`, not at the top level. Without the recheck, we would get wrong results. The recheck cost scales with the number of candidate rows (selectivity), not the full table size. For queries with <1% selectivity, the recheck is practically free. For >10% selectivity, the recheck becomes the dominant cost."

### Interview Trigger

The interviewer asks: "How would you index a JSONB column that's queried with the `@>` operator?" If the candidate says "create a GIN index," they know the basics. The follow-up — "What's your plan for the recheck overhead when 30% of rows match?" — separates candidates who understand the bitmap + recheck mechanism from those who know only the syntax. A senior candidate answers by discussing selectivity, the `jsonb_path_ops` operator class to reduce false positives, partial GIN indexes, or adding B-tree expression indexes for the specific hot path.

- [ ] The JSONB column is queried with `@>`, `?`, `?|`, or `?&` operators

- [ ] Write volume is manageable for GIN maintenance (< 500-1000 INSERTs/sec per table)
- [ ] The GIN index size (~2-3x data) is budgeted in storage plan
- [ ] autovacuum settings are tuned for the GIN index write rate
- [ ] `gin_pending_list_limit` is configured appropriately for the workload

|What You Gain|What You Pay|
|---|---|

### Scale Thresholds

- **"Relevant when table exceeds ~10K rows"** — below this, sequential scan is comparable to GIN bitmap scan
- **"Critical when table exceeds ~1M rows"** — sequential scan at 1M rows is ~4500ms; GIN reduces this to ~15ms

7. **Answer:** Default `jsonb_ops`: individual entries per key+value, supports `@>`, `?`, `?|`, `?&`, larger index, more false positives. `jsonb_path_ops`: single entries per path (e.g., `user.role -> admin`), smaller index, fewer false positives for `@>` but LOSES support for `?`, `?|`, `?&`.
8. **Answer:** Approximately 10K-100K rows. Below 10K, sequential scan overhead (~100ms) is acceptable. Above 100K, GIN's ~15ms vs sequential scan's ~4500ms becomes a decisive 300x advantage.
9. **Answer:** `gin_pending_list_limit` (default 4MB). When the pending list exceeds this limit, the next write transaction triggers an automatic cleanup cycle that merges pending entries into the main GIN index structure, blocking writes to that index during the merge and causing INSERT latency spikes.
10. **Answer (60-second narrative):** "When PostgreSQL executes `WHERE jsonb_col @> '{"a":1}'`, the optimizer generates a Bitmap Index Scan on the GIN index. The GIN index's `consistent` function calls `extractQuery` to decompose `{"a":1}` into entries — in this case, one entry for the path key-value pair. It searches the GIN index's B-tree for this entry, finds the posting tree of matching TIDs, and builds a bitmap of heap page blocks. The Bitmap Heap Scan reads these pages in physical order and applies the Recheck Cond: it evaluates the original `@>` expression against the full JSONB document for each candidate row. The recheck is needed because GIN's flattening can produce false positives — for example, a document with `{"a": {"b": 1}}` produces a GIN entry for `b -> 1`, and a query for `@> {"b": 1}` would match it via GIN, but the recheck correctly eliminates it because `b` is nested under `a`, not at the top level. The bitmap scan avoids random I/O by reading heap pages in order, and the recheck cost scales with the number of candidates, not the full table size."

</details>

**Challenge 1 — Write the SQL**

You are given `public.orders` table with an `order_data` JSONB column. Each order document has `{"items": [{"sku": "...", "qty": N}, ...], "customer": {"id": N, "tier": "gold/silver/bronze"}, "total": N`. Write two queries: (1) find all orders where the customer tier is "gold", using the GIN index; (2) find all orders where any item has sku "SKU-001", also using the GIN index.

<details>
<summary>Solution</summary>

```sql
-- Query 1: Find gold tier customers using @> containment
-- GIN index entry for customer.tier -> gold exists
SELECT order_id, order_data->>'total' AS total
FROM public.orders
WHERE order_data @> '{"customer": {"tier": "gold"}}';

-- Query 2: Find orders containing a specific SKU
-- GIN index entry for items.sku -> SKU-001 exists
SELECT order_id, order_data->>'total' AS total
FROM public.orders
WHERE order_data @> '{"items": [{"sku": "SKU-001"}]}';
```

**Logical reads (with GIN):** ~420 buffers for 100K row table. **Execution plan:** Bitmap Index Scan → Bitmap Heap Scan → Recheck Cond. **EF Core equivalent:** `FromSqlRaw` with parameterized JSONB filter.

**Challenge 2 — Fix the performance problem**

```sql
-- This query runs in 8 seconds on a 500K row table.
-- A GIN index exists on the event_data JSONB column.
-- Identify why it's not using the index.
EXPLAIN (ANALYZE, BUFFERS)
SELECT event_id, event_data->>'order_id' AS order_id
FROM public.order_events
WHERE event_data->>'event_type' = 'order_shipped'
  AND (event_data->>'amount')::numeric > 100;
-- Seq Scan on order_events  (cost=0.00..48230.00 rows=10000 width=64)
--   Filter: (((event_data ->> 'event_type'::text) = 'order_shipped'::text)
--            AND ((event_data ->> 'amount'::text))::numeric > '100'::numeric)
--   Buffers: shared read=84230
```

<details> <summary>Solution</summary>

**Root cause:** The query uses `->>` (text extraction) instead of `@>` (containment). GIN indexes do not support `->>`. The operators `->>` return text values, and PostgreSQL cannot use a GIN index for text equality or range comparisons. The entire query is evaluated as a sequential scan.

```sql
-- Fixed query — use @> for the event_type filter
SELECT event_id, event_data->>'order_id' AS order_id
FROM public.order_events
WHERE event_data @> '{"event_type": "order_shipped"}'
  AND (event_data->>'amount')::numeric > 100;
```

**Indexes to create:**

```sql
-- GIN index for containment queries (may already exist)
CREATE INDEX IF NOT EXISTS ix_events_data_gin
ON public.order_events USING GIN (event_data);

-- BTREE expression index for the amount range filter
CREATE INDEX IF NOT EXISTS ix_events_amount_btree
ON public.order_events (((event_data->>'amount')::numeric))
WHERE event_data @> '{"event_type": "order_shipped"}';
```

**After fix — EXPLAIN (ANALYZE, BUFFERS):** Bitmap Index Scan → Bitmap Heap Scan (from sequential scan). Execution time: ~15ms (from 8000ms). Buffers: 420 (from 84,230).

</details>

**Challenge 4 — Diagnose the concurrency problem**

A PostgreSQL table `public.event_log` with a JSONB column and GIN index receives 2000 INSERTs/sec. `pg_stat_user_tables` shows `n_dead_tup = 300000` and `last_autovacuum` was 4 hours ago. The `pg_stat_all_indexes` for the GIN index shows `idx_blks_read` growing rapidly. INSERT latency has increased from 2ms to 40ms. What is happening?

<details> <summary>Solution</summary>

**Root cause:** The GIN index's fastupdate pending list is filling up and not being cleaned because autovacuum is not keeping up. The pending list grows until it hits `gin_pending_list_limit` (default 4MB), then a cleanup is triggered as part of a write transaction, blocking INSERTs during the merge. The `n_dead_tup` at 300K indicates the GIN index has accumulated many dead index entries from UPDATE/DELETE operations, and the index is bloated. The growing `idx_blks_read` indicates the main index is serving reads because the pending list is large enough that reads of pending entries must scan them sequentially.

**Detection query:**

```sql
SELECT
    relname,
    n_dead_tup,
    n_live_tup,
    ROUND(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_pct,
    last_autovacuum,
    vacuum_count,
    autovacuum_count
FROM pg_stat_user_tables
WHERE relname = 'event_log';
```

**Fix:**

```sql
-- 1. Increase gin_pending_list_limit significantly
ALTER SYSTEM SET gin_pending_list_limit = 262144;  -- 256MB
SELECT pg_reload_conf();

-- 2. Tune autovacuum for this high-write table
ALTER TABLE public.event_log SET (
    autovacuum_vacuum_scale_factor = 0.01,
    autovacuum_vacuum_threshold = 50000,
    autovacuum_vacuum_cost_limit = 2000
);

-- 3. Schedule regular pending list cleanup outside peak hours
-- Run in a cron job or pg_cron:
SELECT gin_clean_pending_list('ix_event_log_data_gin'::regclass);

-- 4. Consider disabling fastupdate for consistent latency:
ALTER INDEX ix_event_log_data_gin SET (fastupdate = OFF);
-- This makes every INSERT update the main index directly (slower INSERTs
-- but no pending list cleanup spikes).

-- 5. As a last resort, drop the GIN index and use relational columns
-- for the fields being queried:
ALTER TABLE public.event_log ADD COLUMN event_type VARCHAR(50)
    GENERATED ALWAYS AS (event_data->>'event_type') STORED;
CREATE INDEX ON public.event_log (event_type);
```

</details>

**Challenge 5 — Design the index strategy**

**Scenario:** A `public.user_activities` table stores 50M rows of user behavior events in a JSONB column `activity_data`. The workload:
- 50% reads: `WHERE activity_data @> '{"type": "page_view"}'` — 20 distinct event types
- 25% reads: `WHERE activity_data ? 'purchase_id'` — checking if a specific key exists
- 25% reads: `WHERE (activity_data->>'duration_ms')::int > 5000` — range query on extracted value
- Writes: 500 INSERTs/sec with 10-20 keys per document
- 10% of queries combine multiple conditions

Design the optimal index strategy.

<details> <summary>Solution</summary>

```sql
-- Index 1: GIN index with default jsonb_ops for @> and ? queries (75% of reads)
CREATE INDEX ix_activities_data_gin
ON public.user_activities USING GIN (activity_data)
WITH (gin_pending_list_limit = 131072);  -- 128MB pending list

-- Index 2: Partial BTREE expression index for duration_ms range queries (25% of reads)
-- Filtered to only rows that have duration_ms
CREATE INDEX ix_activities_duration_btree
ON public.user_activities (((activity_data->>'duration_ms')::int))
WHERE activity_data ? 'duration_ms';

-- Optional: If specific event types dominate, add partial GIN indexes:
CREATE INDEX ix_activities_pageview_gin
ON public.user_activities USING GIN (activity_data)
WHERE activity_data @> '{"type": "page_view"}';
-- Smaller index for the most common query pattern.
-- PostgreSQL 12+ supports partial GIN indexes.

-- Autovacuum tuning for 500 INSERTs/sec:
ALTER TABLE public.user_activities SET (
    autovacuum_vacuum_scale_factor = 0.01,
    autovacuum_vacuum_threshold = 50000,
    autovacuum_vacuum_cost_limit = 1000,
    autovacuum_naptime = 30  -- check every 30 seconds
);

-- Monitor index health:
SELECT
    idx_scan,
    idx_tup_read,
    idx_tup_fetch,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE indexrelname LIKE 'ix_activities_%';
```

**Tradeoffs:**
- Single GIN for all `@>` and `?` queries: covers 75% of reads, ~150GB index for 50M rows (~3x 50GB data)
- Partial BTREE for duration: ~2GB filtered index (only rows with duration_ms) vs ~15GB unfiltered
- Partial GIN for page_view: smaller but only accelerates page_view queries; other types use the full GIN
- Write overhead: 500 INSERTs/sec × 15 keys = 7500 GIN index entry writes/sec; tune pending list to handle

**What NOT to index:** Individual GIN indexes per key — use a single GIN on the whole JSONB column instead. The single GIN already indexes all keys.

</details>
