### Where This Fits

ISJSON is the validation function that tests whether a string contains valid JSON — it returns 1 for valid JSON, 0 for invalid JSON, and NULL for NULL input. A .NET backend engineer encounters ISJSON when defining CHECK constraints on NVARCHAR(MAX) columns that store JSON, ensuring data integrity at the database level regardless of what the application layer does, and when building ETL pipelines that ingest JSON from external sources where input quality cannot be guaranteed. Production systems that omit ISJSON validation accept corrupt data silently — a string like `"{bad json"}` or `"{truncated"}` passes through INSERT without error and causes every downstream JSON_VALUE, JSON_QUERY, OPENJSON, or application deserialization call to fail at runtime, often with confusing errors. The interview signal is whether a candidate treats JSON columns as a typed domain (requiring validation constraints) rather than as string columns with convention-based JSON content — and whether they understand the performance characteristics of ISJSON versus a full JSON.parse in the application tier.

## Core Mental Model

ISJSON is a lightweight validation function that checks whether a string conforms to the JSON grammar as implemented by SQL Server's JSON parser. It does NOT perform a full structural parse — it validates the JSON structure at the token level, confirming that braces and brackets are balanced, keys are properly quoted, strings use double quotes, numbers follow JSON numeric format, and there are no trailing commas or unexpected tokens. The critical distinction: ISJSON is fast (microsecond-scale for typical documents) because it uses a streaming validator rather than building a DOM, making it suitable for CHECK constraints on every INSERT and UPDATE. When used as a CHECK constraint, ISJSON guarantees that every value in the column is valid JSON at the database level — the database enforces the JSON contract, and the application layer never needs to handle invalid JSON when reading from that column. ISJSON supports an optional type argument that constrains which JSON types are valid: VALUE (any JSON value), ARRAY, OBJECT, or SCALAR — enabling more precise constraints like "this column must contain a JSON array, not just any JSON." This type constraint is a SQL Server 2022 feature and is critical for columns that are expected to contain a specific JSON structure.

### Classification

- **Function family:** JSON validation function — scalar function, deterministic.
- **Engine behavior:** Executes as a lightweight tokenizer/validator. Returns INT (1 = valid, 0 = invalid, NULL = input is NULL). Does NOT build a DOM or allocate significant memory.
- **SARGability:** ISJSON is typically used in CHECK constraints or WHERE clauses. In a WHERE clause, `ISJSON(col) = 1` is SARGable if there is a filtered index or computed column index on `ISJSON(col)` — otherwise, it evaluates per row as a predicate. On a column with a CHECK constraint, the optimizer can sometimes infer that `ISJSON(col) = 1` is always true and remove the predicate entirely (contradiction detection).
- **Write cost:** When used in a CHECK constraint, ISJSON executes on every INSERT and UPDATE that affects the column. The cost is negligible for typical document sizes (sub-millisecond).

```mermaid
flowchart TD
    A[Input string value] --> B{Is the value NULL?}
    B -->|Yes| C[Return NULL]

    F -->|TYPE = VALUE| G[Accept any valid JSON value]
    F -->|TYPE = ARRAY| H[Must start with '[']

    I --> K

    L --> M{All tokens valid, balanced, no trailing garbage?}

```

## Deep Mechanics

### How the Engine Executes This

1. **Tokenization:** When ISJSON receives a string, it starts tokenizing from the first non-whitespace character. The tokenizer reads characters sequentially, identifying JSON structural elements: `{`, `}`, `[`, `]`, `:`, `,`, quoted strings (handling escape sequences like `\"`, `\\`, `\n`, `\t`, `\uXXXX`), numbers (including negative, decimal, scientific notation), and literals (`true`, `false`, `null`).

2. **Brace/ bracket matching:** The validator maintains a stack of open objects and arrays. Each `{` pushes an object context, each `[` pushes an array context. Closing `}` or `]` must match the corresponding open context. After processing all tokens, the stack must be empty (all objects and arrays closed).

3. **Comma and colon validation:** Inside objects, keys must be quoted strings followed by `:`. Values follow, optionally followed by `,` and another key-value pair. Inside arrays, values are separated by `,`. Trailing commas are invalid — `[1,2,]` returns 0. Empty objects `{}` and arrays `[]` are valid.

4. **String validation:** JSON strings must use double quotes — single quotes are invalid. Escape sequences are validated: `\"`, `\\`, `\/`, `\b`, `\f`, `\n`, `\r`, `\t`, and `\uXXXX` (where XXXX is a 4-digit hex code). Invalid escapes like `\x` or incomplete `\u` sequences cause validation failure.

5. **Type-constrained validation (3-argument form):** When the optional third argument is provided, an additional check is performed on the root JSON type after successful structural validation. If TYPE = ARRAY, the root must be `[`. If TYPE = OBJECT, the root must be `{`. If TYPE = SCALAR, the root must be a string, number, boolean, or null literal. If TYPE = VALUE, any valid JSON is accepted (same as the 2-argument form).

6. **Return value:** If validation passes, returns 1. If validation fails at any point, returns 0 immediately — the tokenizer stops at the first error. If the input is NULL, returns NULL (not 0).

### SQL Visibility

```sql
-- Basic ISJSON validation
SELECT
    ISJSON('{"valid": "json"}') AS ValidObject,        -- 1
    ISJSON('"a string"')        AS ValidString,         -- 1
    ISJSON('42')                AS ValidNumber,         -- 1
    ISJSON('true')              AS ValidBoolean,        -- 1
    ISJSON('null')              AS ValidNull,           -- 1
    ISJSON('[1,2,3]')           AS ValidArray,          -- 1
    ISJSON('{bad json}')        AS InvalidObject,       -- 0
    ISJSON('{unquoted: "val"}') AS InvalidKey,          -- 0
    ISJSON('{"trailing":1,}')   AS TrailingComma,       -- 0
    ISJSON(NULL)                AS NullInput;           -- NULL

-- ISJSON with type constraint (SQL Server 2022+)
SELECT
    ISJSON('[1,2,3]', ARRAY)    AS IsArray,             -- 1
    ISJSON('{"a":1}', ARRAY)    AS ObjectNotArray,      -- 0
    ISJSON('{"a":1}', OBJECT)   AS IsObject,            -- 1
    ISJSON('42', OBJECT)        AS NumberNotObject,     -- 0
    ISJSON('"hello"', SCALAR)   AS IsScalar,            -- 1
    ISJSON('[1,2]', SCALAR)     AS ArrayNotScalar,      -- 0
    ISJSON('true', SCALAR)      AS BooleanIsScalar,     -- 1
    ISJSON('42', VALUE)         AS IsValue,             -- 1
    ISJSON('{"a":1}', VALUE)    AS ObjectIsValue;       -- 1
```

```sql
CREATE TABLE dbo.Products
(
    ProductId  INT           NOT NULL IDENTITY(1,1),
    Attributes NVARCHAR(MAX) NULL,
    CONSTRAINT PK_Products PRIMARY KEY (ProductId),
    CONSTRAINT CK_Products_Attributes_IsJson CHECK (ISJSON(Attributes) = 1)
);
```

### Execution Plan Analysis

**ISJSON in a CHECK constraint — execution during INSERT:**

```sql
INSERT INTO dbo.Products (ProductName, Attributes)
VALUES ('Widget', N'{"Color":"Red"}');
```

**Expected plan shape:**

```
[Clustered Index Insert] → [Compute Scalar (ISJSON validation)]
```

- **Clustered Index Insert** — inserts the row into the clustered index.
- **Compute Scalar** — evaluates the CHECK constraint expression `ISJSON(Attributes) = 1`. This runs as a predicate filter on the inserted row. If ISJSON returns 0, the INSERT is rolled back with a constraint violation error.

**ISJSON in a WHERE clause:**

```sql
-- Find rows with invalid JSON
SELECT ProductId, Attributes
FROM dbo.Products
WHERE ISJSON(Attributes) = 0;
```

**Expected plan shape:**

```
[Clustered Index Scan] → [Filter (ISJSON = 0)] → [SELECT]
```

- **Clustered Index Scan** — reads every row (or index scan if a non-clustered index covering the needed columns exists but ISJSON still evaluates per row).
- **Filter** — evaluates `ISJSON(Attributes)` for each row. Since ISJSON is a scalar function, it is evaluated row-by-row.
- Logical reads: full table scan (~N pages).

**ISJSON in a filtered index:**

```sql
CREATE INDEX IX_Products_ValidJson
ON dbo.Products (ProductId)
WHERE ISJSON(Attributes) = 1;
```

The filtered index only includes rows where ISJSON = 1. Queries that include `WHERE ISJSON(Attributes) = 1` can use this index for scanning or seeking within the valid-JSON subset.

### Cost Visibility

```sql
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

-- Validate JSON string — ISJSON cost is CPU, not I/O
SELECT ISJSON('{"valid":"json","nested":{"array":[1,2,3],"deeply":true}}') AS IsValid;
-- CPU time = 0ms (sub-millisecond)
-- No table I/O — ISJSON processes the input string only

-- CHECK constraint validation on INSERT
INSERT INTO dbo.Products (ProductName, Attributes)
VALUES ('Test', N'{"Color":"Blue"}');
-- Table 'Products'. Scan count 0, logical reads 2, physical reads 0
-- SQL Server Execution Times: CPU time = 0ms, elapsed time = 1ms
-- (1 logical write for data page, 1 for log)

-- Invalid JSON in CHECK constraint — fails fast
INSERT INTO dbo.Products (ProductName, Attributes)
VALUES ('Bad', N'{invalid}');
-- Msg 547, Level 16, State 0: CHECK constraint violation
-- No row written — no I/O for the invalid row
```

### Failure Modes

**ISJSON returns NULL for NULL input — CHECK constraint allows NULLs:** A CHECK constraint `ISJSON(col) = 1` evaluates to UNKNOWN when col is NULL (because ISJSON(NULL) returns NULL, and NULL = 1 is UNKNOWN in three-valued logic). Since CHECK constraints reject only FALSE values (not UNKNOWN), NULL values are allowed through the constraint.

```sql
-- This INSERT succeeds even though Attributes is NULL:
INSERT INTO dbo.Products (ProductName, Attributes)
VALUES ('Test', NULL);
-- ISJSON(NULL) = 1 evaluates to UNKNOWN, which the CHECK constraint accepts

-- To also reject NULL:
ALTER TABLE dbo.Products
ADD CONSTRAINT CK_Products_Attributes_NotNullJson
CHECK (Attributes IS NOT NULL AND ISJSON(Attributes) = 1);
```

**ISJSON does not validate duplicate keys:** JSON allows duplicate keys per the RFC, and ISJSON does not reject them. `JSON_VALUE` returns the first value for a duplicate key, while `OPENJSON` returns all values. This can cause subtle data inconsistencies.

```sql
-- ISJSON returns 1 for this — structurally valid but semantically ambiguous:
SELECT ISJSON('{"key":"first","key":"second"}');  -- 1

-- JSON_VALUE returns the first:
SELECT JSON_VALUE('{"key":"first","key":"second"}', '$.key');  -- 'first'

-- OPENJSON returns both:
SELECT * FROM OPENJSON('{"key":"first","key":"second"}');
-- key: 'first', then key: 'second'
```

**ISJSON with type constraint in SQL Server 2019 and earlier:** The 3-argument form of ISJSON with a type parameter is only available in SQL Server 2022+. Running it on earlier versions causes a syntax error.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Schema context: Various tables with JSON columns
-- ============================================================

-- ============================================================
-- Pattern 1: CHECK constraint to ensure JSON integrity
-- ============================================================
CREATE TABLE dbo.Products
(
    ProductId       INT            NOT NULL IDENTITY(1,1),
    ProductName     NVARCHAR(200)  NOT NULL,
    CategoryId      INT            NOT NULL,
    Price           DECIMAL(18,2)  NOT NULL,
    Attributes      NVARCHAR(MAX)  NULL,
    CreatedAt       DATETIME2(0)   NOT NULL DEFAULT SYSUTCDATETIME(),
    CONSTRAINT PK_Products PRIMARY KEY CLUSTERED (ProductId),
    CONSTRAINT CK_Products_Attributes CHECK (ISJSON(Attributes) = 1)
);

-- ============================================================
-- Pattern 2: NOT NULL + CHECK constraint — no NULL, no invalid JSON
-- ============================================================
CREATE TABLE dbo.IntegrationEvents
(
    EventId         UNIQUEIDENTIFIER NOT NULL DEFAULT NEWID(),
    EventType       VARCHAR(100)     NOT NULL,
    EventPayload    NVARCHAR(MAX)    NOT NULL,
    CreatedAt       DATETIME2(0)     NOT NULL DEFAULT SYSUTCDATETIME(),
    CONSTRAINT PK_IntegrationEvents PRIMARY KEY CLUSTERED (EventId),
    CONSTRAINT CK_IntegrationEvents_Payload_IsJson CHECK (
        Attributes IS NOT NULL AND ISJSON(Attributes) = 1
    )
);

-- ============================================================
-- Pattern 3: Type-constrained JSON (SQL Server 2022+)
-- ============================================================
CREATE TABLE dbo.OrderAttachments
(
    AttachmentId    INT             NOT NULL IDENTITY(1,1),
    OrderId         INT             NOT NULL,
    AttachmentType  VARCHAR(50)     NOT NULL,
    Metadata        NVARCHAR(MAX)   NOT NULL,  -- Must be a JSON object
    FileList        NVARCHAR(MAX)   NULL,      -- Must be a JSON array
    CONSTRAINT PK_OrderAttachments PRIMARY KEY (AttachmentId),
    CONSTRAINT CK_Metadata_IsObject CHECK (ISJSON(Metadata, OBJECT) = 1),
    CONSTRAINT CK_FileList_IsArray CHECK (
        FileList IS NULL OR ISJSON(FileList, ARRAY) = 1
    )
);

-- ============================================================
-- Pattern 4: ISJSON in WHERE to filter valid JSON before processing
-- ============================================================
DECLARE @jsonInput NVARCHAR(MAX) = N'[
    {"id":1,"name":"valid"},
    {invalid},
    {"id":2,"name":"also valid"},
    "just a string",
    true
]';

SELECT value AS JsonFragment
FROM OPENJSON(@jsonInput)
WHERE ISJSON(value) = 1;
-- Returns 3 valid fragments: {"id":1,...}, {"id":2,...}, "just a string"

-- ============================================================
-- Pattern 5: ISJSON in ETL validation — reject bad rows
-- ============================================================
CREATE TABLE dbo.Staging_ProductImport
(
    RowId           INT             NOT NULL IDENTITY(1,1),
    RawData         NVARCHAR(MAX)   NOT NULL,
    ImportedAt      DATETIME2(0)    NOT NULL DEFAULT SYSUTCDATETIME(),
    IsValid         AS CAST(ISJSON(RawData) AS BIT) PERSISTED,
    CONSTRAINT PK_Staging_ProductImport PRIMARY KEY (RowId)
);

CREATE INDEX IX_Staging_IsValid ON dbo.Staging_ProductImport (IsValid)
    WHERE IsValid = 1;

-- ETL process: process only valid rows
INSERT INTO dbo.Products (ProductName, Price, Attributes)
SELECT
    JSON_VALUE(RawData, '$.name'),
    JSON_VALUE(RawData, '$.price'),
    RawData
FROM dbo.Staging_ProductImport
WHERE IsValid = 1;

-- Report invalid rows
SELECT RowId, RawData
FROM dbo.Staging_ProductImport
WHERE IsValid = 0;

-- ============================================================
-- Pattern 6: Computed column with ISJSON for indexing
-- ============================================================
CREATE TABLE dbo.AuditLog
(
    AuditId         INT             NOT NULL IDENTITY(1,1),
    EntityType      VARCHAR(100)    NOT NULL,
    EntityId        INT             NOT NULL,
    Changes         NVARCHAR(MAX)   NOT NULL,
    CreatedAt       DATETIME2(0)    NOT NULL DEFAULT SYSUTCDATETIME(),
    ChangesAreJson  AS CAST(ISJSON(Changes) AS BIT),
    CONSTRAINT PK_AuditLog PRIMARY KEY (CLUSTERED AuditId),
    CONSTRAINT CK_AuditLog_Changes CHECK (ISJSON(Changes) = 1)
);

CREATE INDEX IX_AuditLog_ChangesAreJson
    ON dbo.AuditLog (ChangesAreJson)
    WHERE ChangesAreJson = 1;

-- ============================================================
-- Pattern 7: Validate JSON in a trigger (when ALTER TABLE CHECK not possible)
-- ============================================================
CREATE TRIGGER TR_Products_ValidateJson
ON dbo.Products
AFTER INSERT, UPDATE
AS
BEGIN
    SET NOCOUNT ON;

    IF EXISTS (
        SELECT 1
        FROM inserted
        WHERE Attributes IS NOT NULL
          AND ISJSON(Attributes) = 0
    )
    BEGIN
        THROW 50001, 'Invalid JSON in Attributes column.', 1;
    END;
END;

-- ============================================================
-- Anti-pattern: No validation — trusting application layer
-- ============================================================
-- ❌ No CHECK constraint — string column with "JSON convention"
CREATE TABLE dbo.Products
(
    ProductId       INT            NOT NULL IDENTITY(1,1),
    Attributes      NVARCHAR(MAX)  NULL
    -- No constraint — relies on app to always insert valid JSON
);

-- An INSERT like this succeeds and corrupts data:
INSERT INTO dbo.Products (Attributes)
VALUES ('{not really json at all}');
-- Subsequent reads via JSON_VALUE or OPENJSON return NULL or error

-- ✅ Always add CHECK (ISJSON(col) = 1) on JSON columns
```

```csharp
// Dapper — JSON validation in ETL pipeline
public sealed class ProductEtlService
{
    private readonly IDbConnectionFactory _connectionFactory;

    public ProductEtlService(IDbConnectionFactory connectionFactory)
        => _connectionFactory = connectionFactory;

    // Validate JSON before inserting — filter invalid rows
    public async Task<int> ImportProductsAsync(
        IReadOnlyList<string> jsonDocuments,
        CancellationToken cancellationToken = default)
    {
        var connection = _connectionFactory.Create();
        await using var _ = connection.ConfigureAwait(false);

        // Use ISJSON in SQL to validate in database
        const string checkJson = @"
            SELECT COUNT(*)
            FROM (VALUES @JsonDocs) AS d(JsonValue)
            WHERE ISJSON(d.JsonValue) = 1;";

        // Alternative: validate in C# before sending to SQL
        var validDocs = jsonDocuments
            .Where(j => IsValidJson(j))
            .ToList();

        if (validDocs.Count == 0) return 0;

        const string insertSql = @"
            INSERT INTO dbo.Products (ProductName, Price, Attributes)
            SELECT
                JSON_VALUE(@JsonDoc, '$.name'),
                JSON_VALUE(@JsonDoc, '$.price'),
                @JsonDoc;";

        var inserted = 0;
        foreach (var doc in validDocs)
        {
            var result = await connection.ExecuteAsync(
                new CommandDefinition(insertSql,
                    new { JsonDoc = doc },
                    cancellationToken: cancellationToken));
            inserted += result;
        }

        return inserted;
    }

    // Validate JSON in bulk using a table-valued parameter
    public async Task<(int Valid, int Invalid)> BulkImportWithValidationAsync(
        IReadOnlyList<string> jsonDocuments,
        CancellationToken cancellationToken = default)
    {
        // Create temp table, bulk insert, validate with ISJSON, process valid rows
        var connection = _connectionFactory.Create();
        await using var _ = connection.ConfigureAwait(false);

        await connection.OpenAsync(cancellationToken);

        // Step 1: Create temp table
        await connection.ExecuteAsync(@"
            CREATE TABLE #Staging (RawJson NVARCHAR(MAX) NOT NULL);", cancellationToken: cancellationToken);

        // Step 2: Bulk insert raw data
        var stagingTable = new DataTable();
        stagingTable.Columns.Add("RawJson", typeof(string));
        foreach (var doc in jsonDocuments)
            stagingTable.Rows.Add(doc);

        using var bulkCopy = new SqlBulkCopy((SqlConnection)connection);
        bulkCopy.DestinationTableName = "#Staging";
        await bulkCopy.WriteToServerAsync(stagingTable, cancellationToken);

        // Step 3: Validate and move valid rows
        const string validateAndMove = @"
            INSERT INTO dbo.Products (ProductName, Price, Attributes)
            SELECT
                JSON_VALUE(s.RawJson, '$.name'),
                JSON_VALUE(s.RawJson, '$.price'),
                s.RawJson
            FROM #Staging s
            WHERE ISJSON(s.RawJson) = 1;

            SELECT COUNT(*) AS ValidCount
            FROM #Staging
            WHERE ISJSON(s.RawJson) = 1;";

        var valid = await connection.ExecuteScalarAsync<int>(
            new CommandDefinition(validateAndMove, cancellationToken: cancellationToken));

        return (Valid: valid, Invalid: jsonDocuments.Count - valid);
    }

    private static bool IsValidJson(string? value)
    {
        if (value is null) return false;
        try
        {
            using var doc = JsonDocument.Parse(value);
            return true;
        }
        catch (JsonException)
        {
            return false;
        }
    }
}
```

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL: JSON validation is implicit in the data type
-- When a column is JSONB, PostgreSQL automatically validates on INSERT
-- Invalid JSON causes an error — no explicit ISJSON needed

CREATE TABLE products (
    product_id   SERIAL PRIMARY KEY,

-- PostgreSQL: Validate JSON as json (not jsonb) — same behavior
-- PostgreSQL: Check constraint can enforce structure
```

## Gotchas and Production Pitfalls

### CHECK Constraint Allows NULL Through Three-Valued Logic

**Pitfall:** Creating a CHECK constraint `ISJSON(col) = 1` and assuming it rejects NULL values.

```sql
-- ❌ This constraint allows NULL:
ALTER TABLE dbo.Products
ADD CONSTRAINT CK_Products_Attributes CHECK (ISJSON(Attributes) = 1);

INSERT INTO dbo.Products (ProductName, Attributes)
VALUES ('No Attributes', NULL);  -- Succeeds! ISJSON(NULL) = 1 → UNKNOWN → CHECK passes
```

**Symptom:** The Attributes column contains NULL values even though the CHECK constraint was intended to ensure all values are valid JSON. Code that reads Attributes and expects it to be non-NULL JSON throws NullReferenceException.

**Fix:**

```sql
-- ✅ Include NOT NULL check:
ALTER TABLE dbo.Products
DROP CONSTRAINT CK_Products_Attributes;

ALTER TABLE dbo.Products
ADD CONSTRAINT CK_Products_Attributes CHECK (
    Attributes IS NOT NULL AND ISJSON(Attributes) = 1
);

-- Or make the column NOT NULL:
ALTER TABLE dbo.Products
ALTER COLUMN Attributes NVARCHAR(MAX) NOT NULL;
-- (Ensure all existing rows are updated first)
```

**Cost of not fixing:** Production application crashes at 3 AM because a JSON column is NULL, throwing NullReferenceException in the API layer. The database constraint was trusted to prevent NULL but it did not.

---

### ISJSON Is Insufficient for Structural Validation

**Pitfall:** Using only ISJSON to validate that a column contains a JSON object or array with a specific schema, when ISJSON only validates structural JSON syntax, not schema.

```sql
-- ❌ ISJSON validates syntax only, not structure:
ALTER TABLE dbo.Products
ADD CONSTRAINT CK_Products_Attributes CHECK (ISJSON(Attributes) = 1);

-- All of these are "valid JSON" and pass the constraint:
-- '{"Color":"Red"}'                    (valid object — what we want)
-- '"just a string"'                    (valid JSON string — not useful)
-- '42'                                 (valid JSON number — not useful)
-- '[1,2,3]'                            (valid JSON array — expected object)
-- '{"Color":"Red","extra":"unexpected"}'(valid JSON but extra fields not validated)
```

**Symptom:** The column contains JSON values that are technically valid but structurally wrong for the application expectations. `JSON_VALUE(Attributes, '$.Color')` works for the string case (returns NULL), but `JSON_QUERY(Attributes, '$')` returns a string instead of an object, causing confusion. Application code that calls `JsonSerializer.Deserialize<ProductAttributes>(Attributes)` on the number `42` throws a JsonException.

**Fix:**

```sql
-- ✅ Use the 3-argument form (SQL Server 2022+) to constrain JSON type:
ALTER TABLE dbo.Products
ADD CONSTRAINT CK_Products_Attributes_IsObject
CHECK (ISJSON(Attributes, OBJECT) = 1);

-- ✅ Use a CHECK constraint with JSON_VALUE for required fields:
ALTER TABLE dbo.Products
ADD CONSTRAINT CK_Products_Attributes_HasColor
CHECK (
    JSON_VALUE(Attributes, '$.Color') IS NOT NULL
);
```

**Cost of not fixing:** A data pipeline imports 1M rows where `Attributes` contains JSON numbers or strings instead of objects. The product system crashes when trying to read nested properties. Recovery requires a full data cleanup — identifying, fixing, or rejecting the bad rows.

### ISJSON in WHERE Clause Causes Scan on Large Tables

**Pitfall:** Using `WHERE ISJSON(col) = 1` as a filter on a large table without a filtered index.

```sql
-- ❌ No filtered index — full scan on 10M rows
SELECT ProductId, JSON_VALUE(Attributes, '$.Color') AS Color
FROM dbo.Products
WHERE ISJSON(Attributes) = 1
  AND JSON_VALUE(Attributes, '$.Color') = 'Red';
```

**Symptom:** The query runs a Clustered Index Scan on 10M rows, evaluating ISJSON for every row. Logical reads: ~120,000. Execution time: 8-12 seconds for a query that should return 20,000 rows in under 100ms.

**Fix:**

```sql
-- ✅ Create a filtered index on the ISJSON condition:
CREATE INDEX IX_Products_ValidJson
ON dbo.Products (ProductId)
INCLUDE (Attributes)
WHERE ISJSON(Attributes) = 1;

-- ✅ Or use a computed column:
ALTER TABLE dbo.Products
ADD IsValidJson AS CAST(ISJSON(Attributes) AS BIT) PERSISTED;

CREATE INDEX IX_Products_IsValidJson
ON dbo.Products (IsValidJson)
WHERE IsValidJson = 1;
```

**Cost of not fixing:** A 12-second scan on a 10M row table running 100 times/hour consumes ~12 GB of logical reads per hour. On a busy OLTP system, this displaces the buffer pool cache, slowing down all other queries.

### Duplicate Keys Pass ISJSON Validation

**Pitfall:** Assuming ISJSON validates that keys are unique in a JSON object.

```sql
-- ❌ This is valid JSON per RFC and passes ISJSON:
DECLARE @json NVARCHAR(MAX) = N'{"id":1,"id":2,"name":"widget"}';
SELECT ISJSON(@json);  -- 1 (valid JSON)

-- JSON_VALUE returns the first occurrence:
SELECT JSON_VALUE(@json, '$.id');  -- 1

-- OPENJSON returns both:
SELECT * FROM OPENJSON(@json);
-- key: 'id', value: '1'
-- key: 'id', value: '2'
```

**Symptom:** Application code that deserializes the JSON with `JsonSerializer.Deserialize<Dictionary<string, object>>()` — System.Text.Json throws a `JsonException` for duplicate keys by default, while Newtonsoft.Json uses the last value. Inconsistent behavior between services using different JSON parsers.

**Fix:**

```sql
-- ✅ Use OPENJSON to detect duplicate keys:
INSERT INTO dbo.JsonDuplicateAudit (JsonValue, DetectedAt)
SELECT @json, SYSUTCDATETIME()
WHERE EXISTS (
    SELECT 1
    FROM OPENJSON(@json)
    GROUP BY [key]
    HAVING COUNT(*) > 1
);
```

**Cost of not fixing:** Intermittent deserialization failures in production when services receive JSON with duplicate keys — the failure is environment-dependent (System.Text.Json throws, Newtonsoft.Json does not), making debugging extremely confusing.

### ISJSON Performance Assumption for Very Large Documents

**Pitfall:** Assuming ISJSON on a 2 MB document is as fast as on a 2 KB document.

```sql
-- ISJSON on a 2 MB JSON document:
SELECT ISJSON(@twoMbJson);  -- Takes 2-5 ms instead of < 0.01 ms
```

**Symptom:** A bulk insert of 10,000 rows with 2 MB JSON documents and an ISJSON CHECK constraint takes 20-50 seconds just for validation. The INSERT throughput is limited by the ISJSON validation CPU cost, not the I/O.

**Fix:**
```sql
-- ✅ Option 1: Sample validation — validate first 100 rows, trust the source
-- (if the JSON comes from a trusted source, skip per-row validation)

-- ✅ Option 2: Use a lighter validation — check first N bytes
-- ISJSON validates the entire document; for very large docs, consider
-- application-level validation instead of CHECK constraint

-- ✅ Option 3: Remove the CHECK constraint and validate in the ETL layer
-- for documents over a certain size threshold
```

**Cost of not fixing:** ETL jobs that import large JSON documents run 5-10x slower than necessary due to ISJSON validation overhead on every row. For a nightly batch job, this extends the processing window from 30 minutes to 3 hours, risking SLA violations.

### ISJSON Returns 0 for Valid JSON with NaN or Infinity

**Pitfall:** JSON numbers like `NaN`, `Infinity`, or `-Infinity` are valid in some JSON parsers but not in SQL Server's JSON implementation.

```sql
-- These are valid in JavaScript/Newtonsoft.Json but NOT in SQL Server:
SELECT ISJSON('{"value": NaN}');        -- 0
SELECT ISJSON('{"value": Infinity}');   -- 0
SELECT ISJSON('{"value": -Infinity}');  -- 0

-- These are valid in SQL Server:
SELECT ISJSON('{"value": null}');       -- 1
SELECT ISJSON('{"value": 1.5e10}');     -- 1 (scientific notation is supported)
```

**Symptom:** JSON documents that are valid per the application's JSON parser (e.g., Newtonsoft.Json with `FloatFormatHandling.AllowNaN`) are rejected by the SQL Server CHECK constraint with ISJSON. The INSERT fails with a constraint violation.

**Fix:**
```sql
-- ✅ Option 1: Ensure the application serializes using SQL Server-compatible JSON
-- In C#: use JsonSerializerOptions with NumberHandling = JsonNumberHandling.Strict
var options = new JsonSerializerOptions
{
    NumberHandling = JsonNumberHandling.Strict
};

-- ✅ Option 2: Replace NaN/Infinity before storing
-- In SQL: use REPLACE or a trigger
```

**Cost of not fixing:** Production ETL pipeline fails intermittently when incoming data contains NaN or Infinity values from external APIs. The error is confusing because the JSON is valid per the source system but invalid per SQL Server.

### Spoken Answers

**Q: What does ISJSON do, and why would you use it in a CHECK constraint?**

> **Average answer:** ISJSON checks if a string is valid JSON and returns 1 for valid, 0 for invalid. You use it in a CHECK constraint to make sure only valid JSON goes into the column.

> **Great answer:** ISJSON is SQL Server's lightweight JSON validation function. It operates as a streaming token-level validator — it confirms that the string contains valid JSON by checking balanced delimiters, proper quoting, valid escape sequences, and correct numeric formats, but it does NOT build a DOM or allocate significant memory. This makes it fast enough to use in a CHECK constraint on every INSERT and UPDATE with negligible overhead — typically 1-10 microseconds for a 10 KB document. Using ISJSON in a CHECK constraint is a data integrity practice that shifts validation from the application layer into the database, ensuring that every value in the column is valid JSON regardless of the application code path that wrote it. Without this constraint, a single bug in application code, a malformed API response, or a corrupt message in a queue can write invalid JSON into the column, causing every downstream read to fail with confusing JSON parsing errors. The constraint also enables the optimizer to make inferences — if all rows satisfy ISJSON = 1, query predicates that check ISJSON = 1 can be simplified away. One production incident I dealt with: a microservice had a bug that wrote XML into a JSON column for 3 hours before anyone noticed. If there had been an ISJSON CHECK constraint, the INSERT would have failed immediately with a clear constraint violation instead of silently corrupting 50,000 rows that had to be rebuilt from logs.

---

**Q: Compare ISJSON in SQL Server vs PostgreSQL's automatic JSON validation.**

> **Average answer:** SQL Server uses ISJSON explicitly; PostgreSQL validates JSON automatically when you use the JSON/JSONB data type.

> **Great answer:** The key structural difference is that PostgreSQL bakes validation into the type system while SQL Server treats JSON as a convention on NVARCHAR(MAX). In PostgreSQL, when you declare a column as JSONB, the database automatically validates the input on every INSERT and UPDATE — there is no equivalent of ISJSON because the type system enforces well-formedness. Invalid input is rejected with a type cast error. PostgreSQL also knows the column is JSONB at the schema level, enabling the query planner to use JSONB-specific statistics and index strategies (GIN indexes, @> containment operators, etc.) without any configuration. In SQL Server, JSON is stored as NVARCHAR(MAX) — the database does not know the column is supposed to contain JSON unless you explicitly tell it with a CHECK constraint using ISJSON. Without the constraint, the column is just a string column, and JSON functions will silently return NULL for invalid JSON (in lax mode) rather than raising an error. The practical consequence: in PostgreSQL, an invalid JSON write fails immediately. In SQL Server, it silently stores an invalid string, and the failure surfaces later — often hours later in a downstream process — making debugging significantly harder. This is why I consider the ISJSON CHECK constraint non-negotiable for any SQL Server JSON column in production. The difference in developer experience and data integrity is substantial: PostgreSQL's JSONB is stricter, safer, and more performant for JSON queries. SQL Server catches up a bit in SQL Server 2022 with the three-argument ISJSON (type constraint), but the fundamental approach remains convention-based rather than type-enforced.

---

**Q: Why does CHECK (ISJSON(col) = 1) allow NULL values through?**

> **Average answer:** Because ISJSON(NULL) returns NULL, and NULL = 1 is NULL, not FALSE. CHECK constraints only reject FALSE.

> **Great answer:** This is the three-valued logic trap in SQL CHECK constraints. ISJSON(NULL) returns NULL because NULL means "unknown" — the function cannot validate a non-existent value. Then `NULL = 1` evaluates to UNKNOWN, not FALSE. In SQL Server's three-valued logic (TRUE, FALSE, UNKNOWN), CHECK constraints enforce the rule that only FALSE values are rejected — rows where the constraint evaluates to TRUE or UNKNOWN are accepted. So `ISJSON(NULL) = 1` evaluates to UNKNOWN, and the constraint passes. This is consistent with SQL standard behavior for CHECK constraints: they must reject only rows where the constraint is FALSE, allowing NULLs through. The fix is either to make the column NOT NULL, or to add an explicit `Attributes IS NOT NULL` condition to the constraint. I always use the compound form: `CHECK (Attributes IS NOT NULL AND ISJSON(Attributes) = 1)` for columns that must contain valid, non-null JSON. This is a classic gotcha that I review in every code review involving JSON columns — I have seen production databases where NULL JSON columns existed for months because the team assumed ISJSON rejected them.

|What You Gain|What You Pay|
|---|---|

- ISJSON cost becomes noticeable only when **documents exceed ~1 MB** and validation runs on every write in a high-volume pipeline (> 10,000 writes/second).

- ETL validation in SQL with ISJSON is generally faster than C# `JsonDocument.Parse` for **batch sizes over ~1,000 rows** because the database can validate without network round trips.

<summary>Answers</summary>

1. ISJSON returns 1 for valid JSON, 0 for invalid JSON, and NULL for NULL input. The function never raises an error.

10. SQL Server stores JSON in NVARCHAR(MAX) columns — there is no JSON data type, so the database has no built-in validation. Without an ISJSON CHECK constraint, any code path can write invalid strings into the column. If that happens, every downstream operation that reads the column (JSON_VALUE, OPENJSON, C# JsonSerializer.Deserialize) either returns NULL or throws an error — and by the time you discover the problem, thousands of rows may be corrupted. The CHECK constraint is a single line of DDL that prevents this entire class of bugs by rejecting invalid JSON at write time. It costs ~1 microsecond per write, which is negligible compared to the cost of debugging data corruption.

<details> <summary>Solution</summary>

**Root cause:** The current CHECK constraint only validates JSON syntax, not JSON type. Some rows contain JSON arrays (`[1,2,3]`), scalars (`42`), or strings (`"hello"`) instead of objects. `JSON_VALUE` returns NULL for these because `$.Color` on a non-object returns NULL in lax mode.

**Fix:**

```sql
-- Step 1: Find non-object JSON rows
SELECT ProductId, Attributes
FROM dbo.Products
WHERE ISJSON(Attributes, OBJECT) = 0;

-- Step 2: Fix or reject non-object rows
UPDATE dbo.Products
SET Attributes = NULL  -- or a default object
WHERE ISJSON(Attributes, OBJECT) = 0;

-- Step 3: Upgrade the CHECK constraint (SQL Server 2022+)
ALTER TABLE dbo.Products
DROP CONSTRAINT CK_Products_Attributes;

ALTER TABLE dbo.Products
ADD CONSTRAINT CK_Products_Attributes
CHECK (Attributes IS NOT NULL AND ISJSON(Attributes, OBJECT) = 1);

-- For SQL Server 2019 and earlier, use a JSON_VALUE check:
ALTER TABLE dbo.Products
ADD CONSTRAINT CK_Products_Attributes_IsObject
CHECK (Attributes IS NOT NULL AND ISJSON(Attributes) = 1);
-- Can't directly constrain to object type pre-2022
```

```sql
-- Given this query:
SELECT ProductId, ProductName
FROM dbo.Products
WHERE ISJSON(Attributes) = 1
  AND JSON_VALUE(Attributes, '$.Color') = 'Red';

-- And this execution plan:
-- |--Clustered Index Scan (Products)
--    |--Filter (ISJSON(Attributes) = 1 AND JSON_VALUE(Attributes, '$.Color') = 'Red')
```

This query runs on a 10M row table in 12 seconds. Only 2M rows have ISJSON = 1. Suggest the index and constraint changes.

<details> <summary>Solution</summary>

**Why the plan shows a scan:** There is no index that the optimizer can use to seek for `ISJSON(Attributes) = 1 AND JSON_VALUE(Attributes, '$.Color') = 'Red'`. ISJSON is a function that prevents index seek on the Attributes column, and JSON_VALUE is also a function. The optimizer must scan all rows and evaluate both functions as a residual filter.

**Fix 1 — Add CHECK constraint + computed column index:**

```sql
-- Step 1: Add CHECK constraint to guarantee all rows are valid JSON objects
ALTER TABLE dbo.Products
WITH CHECK
ADD CONSTRAINT CK_Products_Attributes_IsJson
CHECK (Attributes IS NOT NULL AND ISJSON(Attributes, OBJECT) = 1);

-- Now ISJSON(Attributes) = 1 is always TRUE — remove it from the query

-- Step 2: Add computed column for Color
ALTER TABLE dbo.Products
ADD ProductColor AS CAST(JSON_VALUE(Attributes, '$.Color') AS NVARCHAR(50)) PERSISTED;

-- Step 3: Index the computed column
CREATE INDEX IX_Products_ProductColor
ON dbo.Products (ProductColor)
WHERE ProductColor IS NOT NULL;

-- Rewritten query:
SELECT ProductId, ProductName
FROM dbo.Products
WHERE ProductColor = N'Red';
-- Plan: Index Seek on IX_Products_ProductColor
```

**Fix 2 — Filtered index on JSON_VALUE expression (SQL Server 2022+ will still scan, but filtered index helps):**

```sql
-- A filtered index on JSON_VALUE can help if the selectivity is high
CREATE INDEX IX_Products_ColorFiltered
ON dbo.Products (ProductId)
INCLUDE (Attributes)
WHERE JSON_VALUE(Attributes, '$.Color') = 'Red';
```

**After fix — logical reads:** ~340 (index seek on ProductColor) from ~120,000 (full scan)

</details>

**Challenge 4 — Diagnose the batch validation failure**

An ETL job inserts 50,000 rows per batch into `dbo.Products`. Each row has a 500 KB JSON document in the `Attributes` column. The batch takes 45 seconds. Analysis shows 30 seconds is spent in `ISJSON` validation on the CHECK constraint. The JSON comes from a trusted internal API that never produces invalid JSON. What do you change?

<details> <summary>Solution</summary>

**Root cause:** The ISJSON CHECK constraint validates every 500 KB document on every INSERT — 25 GB of JSON validation per batch. For a trusted data source, this is unnecessary CPU overhead.

**Fix options:**

```sql
-- Option 1: Remove the CHECK constraint for trusted sources
ALTER TABLE dbo.Products
DROP CONSTRAINT CK_Products_Attributes_IsJson;

-- Option 2: Keep validation but sample — validate every Nth row
-- (requires application-level logic)

-- Option 3: Validate once in the ETL pipeline before bulk insert
-- (application validates, database trusts)

-- Option 4: Reduce document size — archive historical attributes
-- to an audit table, keep only recent/active attributes
```

**After fix — expected improvement:** Batch time drops from 45 seconds to ~15 seconds (removes 30 seconds of ISJSON CPU time).

**Tradeoff:** Removing the CHECK constraint means invalid JSON could enter the database if a bug is introduced in the trusted source. Mitigation: validate in the ETL application code, add monitoring to detect JSON parse failures in downstream readers, and schedule periodic validation queries.

</details>

**Challenge 5 — Design the JSON validation strategy**

**Scenario:** A financial services platform receives trade confirmations as JSON messages from 50 different broker APIs. Each API may produce slightly different JSON formats. Messages are stored in `dbo.TradeConfirmations` with a `RawPayload` NVARCHAR(MAX) column before processing. Some APIs occasionally send malformed JSON or messages in XML format by mistake. 100,000 messages arrive per hour. Design the validation strategy, including the database schema, the validation pipeline, and the error handling.

<details> <summary>Solution</summary>

**Recommended strategy:**

```sql
-- Schema with validation tiering
CREATE TABLE dbo.TradeConfirmations
(
    ConfirmationId  INT             NOT NULL IDENTITY(1,1),
    BrokerId        INT             NOT NULL,
    RawPayload      NVARCHAR(MAX)   NOT NULL,
    PayloadType     VARCHAR(20)     NOT NULL DEFAULT 'JSON',  -- JSON, XML, Unknown
    IsValid         AS CAST(ISJSON(RawPayload) AS BIT) PERSISTED,
    ValidationError NVARCHAR(500)   NULL,
    ProcessedAt     DATETIME2(0)    NULL,
    CreatedAt       DATETIME2(0)    NOT NULL DEFAULT SYSUTCDATETIME(),
    CONSTRAINT PK_TradeConfirmations PRIMARY KEY CLUSTERED (ConfirmationId)
);

CREATE INDEX IX_TradeConfirmations_IsValid
ON dbo.TradeConfirmations (IsValid)
WHERE IsValid = 1;

CREATE INDEX IX_TradeConfirmations_Invalid
ON dbo.TradeConfirmations (BrokerId)
WHERE IsValid = 0;

-- The IsValid computed column gives you fast filtering on valid/invalid
-- without a CHECK constraint that would reject invalid data.
-- The raw data is preserved regardless of validity.
```

**ETL Pipeline:**

1. Raw insert — all messages go into `RawPayload`, regardless of validity
2. Validation query — find invalid messages
3. Alerting — if any single broker has > 1% invalid rate, alert operations
4. Processing — only process messages where `IsValid = 1`

```csharp
// ETL service
public async Task<TradeConfirmation> IngestMessageAsync(
    int brokerId,
    string rawPayload,
    CancellationToken cancellationToken = default)
{
    const string sql = @"
        INSERT INTO dbo.TradeConfirmations (BrokerId, RawPayload)
        OUTPUT INSERTED.ConfirmationId, INSERTED.IsValid
        VALUES (@BrokerId, @RawPayload);";

    return await _connection.QuerySingleAsync<TradeConfirmation>(
        new CommandDefinition(sql,
            new { BrokerId = brokerId, RawPayload = rawPayload },
            cancellationToken: cancellationToken));
}

// Monitoring query — check broker health
const string monitoringSql = @"
    SELECT
        BrokerId,
        COUNT(*) AS TotalMessages,
        SUM(CASE WHEN IsValid = 0 THEN 1 ELSE 0 END) AS InvalidCount,
        CAST(SUM(CASE WHEN IsValid = 0 THEN 1.0 ELSE 0.0 END) / COUNT(*) * 100 AS DECIMAL(5,2)) AS InvalidPct
    FROM dbo.TradeConfirmations
    WHERE CreatedAt >= DATEADD(HOUR, -1, SYSUTCDATETIME())
    GROUP BY BrokerId;";
```

**Error handling:** Invalid messages are stored, not rejected. The broker is notified via a Dead Letter Queue (DLQ) pattern. Valid messages proceed to processing. This preserves data for analysis (why was it invalid?) without blocking ingestion.

**Tradeoff:** Storing invalid JSON costs storage but enables forensic analysis. A CHECK constraint that rejects invalid JSON would cause the INSERT to fail, which means the broker API error would be lost (not stored anywhere) unless captured in a try/catch.

</details>
