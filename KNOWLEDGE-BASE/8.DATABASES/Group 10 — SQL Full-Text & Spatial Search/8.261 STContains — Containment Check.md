### Where This Fits

STContains answers the spatial containment question: "Is this point inside this polygon?" or "Does this polygon fully contain this other geometry?" It is the primary operator for geofencing, delivery zone validation, electoral district membership, property boundary checks, and exclusion zone enforcement. A .NET backend engineer encounters this in production when implementing geospatial features like "find all active promotions for the user's current location" (querying which polygons contain the user's GPS coordinate), "validate that the delivery address falls within the service area" (point-in-polygon for logistics), or "detect whether a drone flight path intersects restricted airspace" (polygon-in-polygon containment). The interview signal is whether a candidate understands the STContains vs STIntersects vs STWithin triangle, knows that SQL Server geography ring orientation follows the left-hand rule (or the query silently returns wrong results), and can design a spatial index that turns a 2-second full-scan containment check into a 5-millisecond spatial index seek.

## Core Mental Model

STContains is the spatial containment predicate: `geography::STContains(@point)` returns 1 when the invoking geography/geometry instance completely contains the argument instance, with no points of the argument lying on the boundary or exterior. The invariant is: "A contains B if and only if every point of B is interior to A." This is distinct from STIntersects (which returns 1 if any point is shared, including boundary touches) and STWithin (the inverse: B is within A if and only if A contains B). The SQL Server spatial index accelerates this operation using a multi-level grid tessellation that decomposes the indexed geography into grid cells at levels 1 through 4; the query optimizer performs a primary filter (grid cell intersection — fast, approximate) followed by a secondary filter (exact STContains computation — precise, CPU-bound). The secondary filter must compute the full planar or geodesic containment test, which is computationally expensive (microseconds per row) but unavoidable for correctness. The recognition pattern for containment queries: a WHERE clause using `@polygon.STContains(l.Location) = 1` combined with a spatial index that the optimizer uses for the primary filter — the execution plan shows a Spatial Index Seek with estimated rows filtered down from the primary filter cardinality.

### Classification

- **Operator family:** Spatial containment methods (STContains, STWithin, STIntersects, STDistance)
- **SARGability:** Not applicable in the B-tree sense — spatial predicates use spatial index seek if a spatial index exists; without a spatial index, every row requires a full geometry computation (Table Scan + Filter)
- **Secondary filter cost:** O(n log n) for polygon intersection tests using the Bentley-Ottmann algorithm for planar geometry; O(n2) boundary cases for geodesic edges with geography
- **Return semantics:** NULL if either operand has a NULL SRID or the SRIDs do not match; SQL Server requires SRID equality

## Deep Mechanics

### How the Engine Executes STContains

1. **Parsing and Binding:** SQL Server parses the `WHERE @polygon.STContains(l.Location) = 1` predicate. The query processor recognizes the spatial method call against the geography/geometry column. The optimizer checks for the existence of a spatial index on `l.Location`. If no spatial index exists, the optimizer falls back to a full table scan — each row loads the geography, calls the STContains method via the in-process SQL CLR hosting of Microsoft.SqlServer.Types, and evaluates the predicate.

2. **Spatial Index Primary Filter:** If a spatial index exists, the optimizer constructs a query window (the bounding box of the instrument polygon, expanded by the grid cell size). This window is used to traverse the spatial index's grid tessellation tree. The spatial index in SQL Server uses a multi-level grid decomposition (4 levels, each with configurable cells-per-object: LOW = 4, MEDIUM = 8, HIGH = 16). For a polygon stored at grid level 2, the index records which grid cells at levels 1 through 4 the polygon covers or touches. The primary filter returns all rows whose covering grid cells intersect the query window — this is the "approximate" filter. On a table with 5M spatial rows and a typical polygon, the primary filter reduces candidates to ~500-5000 rows.

3. **Secondary Filter:** Each candidate row from the primary filter has the exact STContains computed against the instrument polygon. SQL Server hosts the SqlGeography/SqlGeometry types in the CLR and invokes the `STContains` method. For geography data, this computes a geodesic containment test: the polygon edges are treated as great-circle arcs on the ellipsoid. For geometry data, the computation is planar (Cartesian). The secondary filter returns the final matching rows.

4. **Row Return:** Only rows where the geography column is fully contained within the instrument polygon are returned.

### SQL Visibility

```sql
-- ============================================================
-- STContains — containment check scenarios
-- ============================================================

-- 1. Basic point-in-polygon: delivery zone check
DECLARE @customerLocation geography = geography::Point(47.6062, -122.3321, 4326);
DECLARE @deliveryZone geography = geography::STGeomFromText(
    'POLYGON((-122.35 47.60, -122.30 47.60, -122.30 47.62, -122.35 47.62, -122.35 47.60))',
    4326);

SELECT
    CASE
        WHEN @deliveryZone.STContains(@customerLocation) = 1
        THEN 'Address is within delivery zone'
        ELSE 'Address outside delivery zone'
    END AS Eligibility;

-- Expected: 1 if the point (47.6062, -122.3321) lies inside the polygon

-- 2. Table query: find all customers within a service area
SELECT c.CustomerId, c.FullName, c.Location.Lat AS Latitude, c.Location.Long AS Longitude
FROM dbo.Customers c
WHERE @serviceArea.STContains(c.Location) = 1;

-- 3. STContains with geometry (planar, faster for non-geographic data)
DECLARE @buildingFootprint geometry = geometry::STGeomFromText(
    'POLYGON((0 0, 100 0, 100 100, 0 100, 0 0))', 0);
DECLARE @devicePoint geometry = geometry::Point(50, 50, 0);

SELECT @buildingFootprint.STContains(@devicePoint) AS IsDeviceInside;
-- Returns 1

-- 4. STContains vs STIntersects — boundary sensitivity
DECLARE @polygon geography = geography::STGeomFromText(
    'POLYGON((-122.35 47.60, -122.30 47.60, -122.30 47.62, -122.35 47.62, -122.35 47.60))', 4326);
DECLARE @boundaryPoint geography = geography::Point(47.60, -122.35, 4326);
DECLARE @interiorPoint geography = geography::Point(47.61, -122.32, 4326);

SELECT
    @polygon.STContains(@boundaryPoint) AS ContainsBoundary,  -- 0 (on boundary = not contained)
    @polygon.STIntersects(@boundaryPoint) AS IntersectsBoundary; -- 1 (touches boundary = intersects)
    -- STContains returns 0 for points on the boundary
    -- STIntersects returns 1 for points on the boundary

-- 5. STContains for zone exclusion (find customers NOT in a restricted zone)
SELECT c.CustomerId, c.Location.ToString() AS LocationWKT
FROM dbo.Customers c
WHERE @restrictedZone.STContains(c.Location) = 0;
-- Note: this returns rows where the point is outside OR on the boundary
-- Use STIntersects for both interior and boundary inclusion
```

### Execution Plan Analysis

**Without spatial index:**

```
[Clustered Index Scan (Customers)] → [Filter (Compute Scalar — STContains)]
```

- Every row is read (clustered index scan).
- For each row, the Filter operator calls STContains via CLR invocation.
- No primary filter — 100% of rows evaluate the predicate.
- Estimated cost: ~100% on the Clustered Index Scan.
- Logical reads: full table scan (e.g., 42,000 for 1M rows).
- CPU: dominated by CLR STContains calls (~2-50 µs each × row count).

**With spatial index:**

```
[Spatial Index Seek (IX_Customers_Location)]
    → [Filter (Compute Scalar — STContains secondary filter)]
    → [Key Lookup (PK_Customers, if non-covering)]
```

- The Spatial Index Seek operator reads the spatial index's grid tessellation and returns only rows whose covering cells intersect the query window.
- The Filter operator re-evaluates STContains on only the primary-filter-passing rows.
- If the spatial index is non-clustering (non-primary), a Key Lookup retrieves additional columns.
- Estimated cost: Spatial Index Seek ~30%, Filter ~60%, Key Lookup ~10%.
- Logical reads: spatial index pages + data pages for matching rows only.
- Reduction factor: from 42,000 logical reads (full scan) to ~50-500 (spatial index seek + lookup).

**Spatial index plan without trace flag 6534 (legacy):**

```
[Spatial Index Seek]
    → [Filter] + [Compute Scalar (STContains)]
        → [Key Lookup]
```

- SQL Server 2012+ always uses the secondary filter in the Filter operator, even with a spatial index. The spatial index cannot guarantee containment — only intersection — so the secondary filter is mandatory.

### Failure Modes

**SRID mismatch returns NULL:** If the column and the query polygon have different SRIDs, STContains returns NULL instead of 0. A WHERE clause `= 1` excludes NULL rows, so the query returns zero rows even if spatial containment exists. Always check SRID parity.

**Ring orientation in geography causes inverted containment:** SQL Server geography expects rings to follow the left-hand rule (counterclockwise exterior, clockwise interior holes). If the polygon is defined clockwise, SQL Server interprets the entire globe as the polygon and the actual region as the hole. STContains returns 1 for points that are everywhere except the intended polygon.

**Very large polygons with many vertices degrade performance:** STContains must evaluate each vertex's position relative to the polygon edges. A polygon with 100,000 vertices causes the secondary filter to take milliseconds per row. Partition the polygon or simplify it with Reduce().

**Spatial index fragmentation:** Unlike B-tree indexes, spatial indexes can become fragmented when geography values are updated or inserted out of spatial order. Fragmented spatial indexes cause the primary filter to return more candidates (less selective), increasing secondary filter CPU load. Rebuild the spatial index when fragmentation exceeds 30%.

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Production: geofencing for delivery zone containment
-- ============================================================

-- Schema: delivery zones and customer locations
CREATE TABLE dbo.DeliveryZones (
    ZoneId INT NOT NULL IDENTITY(1,1),
    ZoneName NVARCHAR(50) NOT NULL,
    ServiceArea geography NOT NULL,
    CONSTRAINT PK_DeliveryZones PRIMARY KEY CLUSTERED (ZoneId)
);

CREATE TABLE dbo.Customers (
    CustomerId INT NOT NULL IDENTITY(1,1),
    FullName NVARCHAR(100) NOT NULL,
    AddressLine1 NVARCHAR(200) NOT NULL,
    City NVARCHAR(50) NOT NULL,
    PostalCode NVARCHAR(20) NOT NULL,
    Location geography NULL,
    CONSTRAINT PK_Customers PRIMARY KEY CLUSTERED (CustomerId)
);

-- Spatial index on Customers.Location
CREATE SPATIAL INDEX IX_Customers_Location
    ON dbo.Customers (Location)
    USING GEOGRAPHY_GRID
    WITH (
        GRIDS = (LEVEL_1 = MEDIUM, LEVEL_2 = MEDIUM, LEVEL_3 = HIGH, LEVEL_4 = HIGH),
        CELLS_PER_OBJECT = 16,
        MAXDOP = 4
    );

-- Spatial index on DeliveryZones.ServiceArea
CREATE SPATIAL INDEX IX_DeliveryZones_ServiceArea
    ON dbo.DeliveryZones (ServiceArea)
    USING GEOGRAPHY_GRID
    WITH (
        GRIDS = (LEVEL_1 = MEDIUM, LEVEL_2 = MEDIUM, LEVEL_3 = MEDIUM, LEVEL_4 = MEDIUM),
        CELLS_PER_OBJECT = 16
    );

-- 1. Find customers within a specific delivery zone
DECLARE @zoneId INT = 42;

SELECT c.CustomerId, c.FullName, c.Location.Lat AS Lat, c.Location.Long AS Long
FROM dbo.Customers c
INNER JOIN dbo.DeliveryZones dz ON dz.ZoneId = @zoneId
WHERE dz.ServiceArea.STContains(c.Location) = 1;

-- 2. Find the delivery zone that contains a customer's address
DECLARE @customerLat FLOAT = 47.6062;
DECLARE @customerLng FLOAT = -122.3321;
DECLARE @customerLocation geography = geography::Point(@customerLat, @customerLng, 4326);

SELECT TOP 1 dz.ZoneId, dz.ZoneName
FROM dbo.DeliveryZones dz
WHERE dz.ServiceArea.STContains(@customerLocation) = 1
ORDER BY dz.ServiceArea.STArea() ASC;
-- Returns the smallest zone containing the point (most specific)

-- 3. Bulk check: classify all customers by zone (join with STContains)
SELECT c.CustomerId, dz.ZoneName AS ContainingZone
FROM dbo.Customers c
CROSS APPLY (
    SELECT TOP 1 dz.ZoneName
    FROM dbo.DeliveryZones dz
    WHERE dz.ServiceArea.STContains(c.Location) = 1
    ORDER BY dz.ServiceArea.STArea() ASC
) dz;

-- 4. Exclusion zone enforcement
DECLARE @exclusionPolygon geography = geography::STGeomFromText(
    'POLYGON((-122.35 47.60, -122.30 47.60, -122.30 47.62, -122.35 47.62, -122.35 47.60))',
    4326);

SELECT c.CustomerId, c.FullName
FROM dbo.Customers c
WHERE @exclusionPolygon.STContains(c.Location) = 0;
-- Returns customers OUTSIDE the restricted area

-- 5. Polygon-in-polygon containment: find delivery zones fully within a superzone
DECLARE @superZone geography = geography::STGeomFromText(
    'POLYGON((-122.40 47.50, -122.25 47.50, -122.25 47.70, -122.40 47.70, -122.40 47.50))',
    4326);

SELECT dz.ZoneId, dz.ZoneName
FROM dbo.DeliveryZones dz
WHERE @superZone.STContains(dz.ServiceArea) = 1;
-- STContains with two polygons: test if the whole zone is inside the superzone

-- 6. Verify ring orientation (critical for geography STContains correctness)
SELECT
    ZoneId,
    ZoneName,
    CASE
        WHEN ServiceArea.STIsValid() = 1 THEN 'Valid'
        ELSE 'Invalid'
    END AS SpatialValid,
    ServiceArea.STAsText() AS WKT
FROM dbo.DeliveryZones
WHERE ServiceArea.STIsValid() = 0;
-- Invalid geometry yields undefined STContains results

-- 7. Fix ring orientation with MakeValid() if needed
UPDATE dbo.DeliveryZones
SET ServiceArea = ServiceArea.MakeValid()
WHERE ServiceArea.STIsValid() = 0;
```

```csharp
// DbContext configuration
public class SpatialDbContext : DbContext

        {
            entity.ToTable("DeliveryZones");
            entity.HasKey(z => z.ZoneId);

            entity.Property(z => z.ServiceArea)
                .HasColumnType("geography");  // SQL Server geography type

            entity.HasIndex(z => z.ServiceArea)
                .HasDatabaseName("IX_DeliveryZones_ServiceArea")
                .IsSpatial();
        });
    }

// Entity classes
public record Customer
{
    public int CustomerId { get; init; }
    public string FullName { get; init; } = string.Empty;
    public Point? Location { get; init; }  // NetTopologySuite Point
}

public record DeliveryZone
{
    public int ZoneId { get; init; }
    public string ZoneName { get; init; } = string.Empty;
    public Polygon? ServiceArea { get; init; }  // NetTopologySuite Polygon
}

// Service: containment queries
public class GeofencingService
{
    private readonly SpatialDbContext _dbContext;

    public GeofencingService(SpatialDbContext dbContext)
        => _dbContext = dbContext;

    // Find customers within a delivery zone
    public async Task<List<Customer>> GetCustomersInZoneAsync(
        int zoneId,
        CancellationToken ct = default)
    {
        var zone = await _dbContext.DeliveryZones
            .FirstOrDefaultAsync(z => z.ZoneId == zoneId, ct);

        if (zone?.ServiceArea is null)
            return [];

        // EF Core translates .Contains() to STContains
        return await _dbContext.Customers
            .Where(c => c.Location != null
                && zone.ServiceArea.Contains(c.Location))
            .AsNoTracking()
            .ToListAsync(ct);
    }

    // Find the smallest zone containing a point (geofencing)
    public async Task<DeliveryZone?> FindZoneContainingAsync(
        double latitude,
        double longitude,
        CancellationToken ct = default)
    {
        var point = new Point(longitude, latitude) { SRID = 4326 };

        // Note: EF Core 8 cannot translate STContains with a parameterized
        // Polygon across multiple zones in a single query efficiently.
        // Fall back to raw SQL for multi-zone containment search:
        const string sql = @"
            SELECT TOP 1 dz.ZoneId, dz.ZoneName
            FROM dbo.DeliveryZones dz
            WHERE dz.ServiceArea.STContains(geography::Point(@Lat, @Lng, 4326)) = 1
            ORDER BY dz.ServiceArea.STArea() ASC;";

        var zone = await _dbContext.DeliveryZones
            .FromSqlRaw(sql,
                new SqlParameter("@Lat", latitude),
                new SqlParameter("@Lng", longitude))
            .AsNoTracking()
            .FirstOrDefaultAsync(ct);

        return zone;
    }

    // Exclusion check: customers outside restricted area
    public async Task<List<Customer>> GetCustomersOutsideZoneAsync(
        int restrictedZoneId,
        CancellationToken ct = default)
    {
        var restricted = await _dbContext.DeliveryZones
            .FirstOrDefaultAsync(z => z.ZoneId == restrictedZoneId, ct);

        if (restricted?.ServiceArea is null)
            return [];

        // STContains = 0 returns customers on boundary OR outside
        return await _dbContext.Customers
            .Where(c => c.Location != null
                && !restricted.ServiceArea.Contains(c.Location))
            .AsNoTracking()
            .ToListAsync(ct);
    }

    // IServiceCollection registration
    public static void ConfigureServices(IServiceCollection services, string connectionString)
    {
        services.AddDbContext<SpatialDbContext>(options =>
            options.UseSqlServer(connectionString, sqlOptions =>
            {
                sqlOptions.UseNetTopologySuite();  // Enable NTS spatial support
                sqlOptions.EnableRetryOnFailure(3);
            }));
        services.AddScoped<GeofencingService>();
    }
}
```

```sql
-- PostgreSQL does not have STContains for geography in the same form.
```

## Gotchas and Production Pitfalls

### Ring Orientation Inversion (geography)

**Pitfall:** Defining a geography polygon with clockwise vertex order. SQL Server geography requires left-hand rule (counterclockwise exterior ring, clockwise interior rings for holes). A clockwise polygon is interpreted as the complement — the entire globe minus the intended region.

```sql
-- ❌ Wrong: clockwise exterior ring in SQL Server geography
DECLARE @wrong geography = geography::STGeomFromText(
    'POLYGON((-122.35 47.60, -122.35 47.62, -122.30 47.62, -122.30 47.60, -122.35 47.60))', 4326);
-- Clockwise: SQL Server inverts to "everything except the polygon"
SELECT @wrong.STContains(geography::Point(47.6062, -122.3321, 4326));
-- Returns 1 even though the point IS inside the polygon (because
-- the polygon is interpreted as the globe minus the small area)

-- ✅ Correct: counterclockwise exterior ring
DECLARE @correct geography = geography::STGeomFromText(
    'POLYGON((-122.35 47.60, -122.30 47.60, -122.30 47.62, -122.35 47.62, -122.35 47.60))', 4326);
-- Counterclockwise: the polygon is the bounded area
```

**Symptom:** STContains returns 1 for points that should be clearly outside the polygon. The entire application acts as if containment is inverted — customers in Seattle are marked as "outside" and customers in Tokyo are marked as "inside" the Seattle delivery zone.

**Fix:** Enforce counterclockwise at insertion. Use `MakeValid()` to reorient rings automatically. Validate with `STIsValid()` before using STContains.

```sql
-- ✅ Fix: force correct orientation
UPDATE dbo.DeliveryZones
SET ServiceArea = ServiceArea.MakeValid().ReorientObject()
WHERE ServiceArea.STIsValid() = 0 OR ServiceArea.EnvelopeAngle() > 90;
-- EnvelopeAngle > 90 indicates the polygon covers more than a hemisphere
-- (possible sign of ring orientation inversion)
```

**Cost of not fixing:** Wrong geofencing results in production — delivery drivers are dispatched to incorrect zones, customers are shown incorrect pricing, exclusion zones fail to exclude. This is a silent data corruption bug that no query optimizer or error log catches.

---

### STContains Returns NULL Instead of 0 on SRID Mismatch

**Pitfall:** Comparing two geography instances with different SRIDs. STContains returns NULL instead of 0 when the SRIDs differ. NULL is filtered out by `= 1` in WHERE clauses, so results appear empty.

```sql
-- ❌ SRID mismatch
DECLARE @polygon geography = geography::STGeomFromText(
    'POLYGON((...))', 4326);  -- SRID 4326 (WGS84)
DECLARE @point geography = geography::Point(47.6062, -122.3321, 4269);
-- SRID 4269 (NAD83) — different from 4326

SELECT @polygon.STContains(@point);
-- Returns NULL, not 0!

SELECT @polygon.STContains(@point) = 1;
-- Returns NULL = 1, which is UNKNOWN, row is excluded
```

**Symptom:** Queries that should return matching rows return empty result sets. The developer spends hours debugging the polygon shape, ring orientation, and spatial index before noticing the SRID mismatch.

**Fix:** Always use SRID 4326 for GPS coordinates. Validate SRIDs match in application code before querying.

```sql
-- ✅ Check SRID match before containment
DECLARE @polygon geography = geography::STGeomFromText('POLYGON((...))', 4326);
DECLARE @point geography = geography::Point(47.6062, -122.3321, 4326);

SELECT
    CASE
        WHEN @polygon.STSrid = @point.STSrid AND @polygon.STContains(@point) = 1
        THEN 1
        ELSE 0
    END AS IsContained;
```

**Cost of not fixing:** Silent data loss — containment queries return zero rows. If this is a production geofencing service, all customers appear outside all zones. The application becomes inoperable.

### Spatial Index Grid Misconfiguration Causes Primary Filter to Return Too Many Candidates

**Pitfall:** Using default spatial index grid settings (LEVEL_1 = MEDIUM) for a dataset with highly non-uniform spatial distribution. For example, all customers are clustered in a single US state, but the grid tessellation is too coarse to effectively filter.

```sql
-- ❌ Default grid (too coarse for clustered data)
CREATE SPATIAL INDEX IX_Customers_Location
    ON dbo.Customers (Location)
    USING GEOGRAPHY_GRID;
-- Default: LEVEL_1 = MEDIUM, LEVEL_2 = MEDIUM, LEVEL_3 = MEDIUM, LEVEL_4 = MEDIUM
```

**Symptom:** The spatial index Seek shows high estimated rows (primary filter returns 50% of the table instead of 0.1%). The secondary filter CPU usage is high because STContains has to compute for thousands of candidates.

**Fix:** Use higher grid density at levels that match the data granularity.

```sql
-- ✅ Fine-tuned grid for clustered point data
CREATE SPATIAL INDEX IX_Customers_Location
    ON dbo.Customers (Location)
    USING GEOGRAPHY_GRID
    WITH (
        GRIDS = (LEVEL_1 = HIGH, LEVEL_2 = HIGH, LEVEL_3 = HIGH, LEVEL_4 = HIGH),
        CELLS_PER_OBJECT = 16
    );
```

**Cost of not fixing:** Spatial queries run 10-100x slower than optimal. The spatial index primary filter is too coarse to be effective, and the secondary filter does all the work. The spatial index becomes nearly as slow as a full table scan.

### STContains on the Boundary Returns 0 (Misunderstood Semantics)

**Pitfall:** Expecting STContains to return 1 for points on the polygon boundary. STContains uses "interior" semantics — points on the boundary are not "contained."

```sql
DECLARE @polygon geography = geography::STGeomFromText(
    'POLYGON((-122.35 47.60, -122.30 47.60, -122.30 47.62, -122.35 47.62, -122.35 47.60))', 4326);
DECLARE @pointOnBoundary geography = geography::Point(47.60, -122.35, 4326);

SELECT @polygon.STContains(@pointOnBoundary);  -- 0
SELECT @polygon.STIntersects(@pointOnBoundary); -- 1
```

**Symptom:** Customers whose addresses fall exactly on the delivery zone boundary are excluded from geofencing results. This may be correct behavior for strict containment (e.g., "inside the building") but wrong for inclusive zones (e.g., "service area includes the border streets").

**Fix:** Use `STIntersects(@point) = 1` instead of `STContains(@point) = 1` when boundary points should be included.

**Cost of not fixing:** Hard-to-diagnose edge cases where customers on the boundary of a delivery zone are consistently excluded. Support tickets increase as customers complain they cannot order delivery from a zone they appear to be inside.

---

### Very Large Polygon Geometry Causes Secondary Filter Slowdown

**Pitfall:** Using STContains with a complex polygon (10,000+ vertices) against a large table. Each STContains call must evaluate the point against all polygon edges, which is O(vertex_count) per row in the secondary filter.

**Symptom:** Even with a spatial index, queries are slow because the secondary filter (exact STContains) takes ~500 µs per candidate row. With 5,000 candidate rows, that is 2.5 seconds of CPU time.

**Fix:** Simplify the polygon geometry with Reduce() or pre-filter with a bounding box:

```sql
-- ✅ Simplify polygon for STContains
DECLARE @simplifiedPolygon geography = @complexPolygon.Reduce(0.01);
-- Reduce(tolerance_in_meters) removes vertices within tolerance

-- ✅ Or use a two-step approach: bounding box pre-filter + STContains
DECLARE @minLat FLOAT = @complexPolygon.EnvelopeCenter().Lat - 0.05;
DECLARE @maxLat FLOAT = @complexPolygon.EnvelopeCenter().Lat + 0.05;
DECLARE @minLng FLOAT = @complexPolygon.EnvelopeCenter().Long - 0.05;
DECLARE @maxLng FLOAT = @complexPolygon.EnvelopeCenter().Long + 0.05;

SELECT c.CustomerId
FROM dbo.Customers c
WHERE c.Location.Lat BETWEEN @minLat AND @maxLat
  AND c.Location.Long BETWEEN @minLng AND @maxLng
  AND @complexPolygon.STContains(c.Location) = 1;
```

**Cost of not fixing:** Geofencing queries that should take <10ms take 3+ seconds. The API endpoint times out under moderate load. CPU on the SQL Server spikes to 100% during geofencing operations.

### Spoken Answers

**Q: What does STContains return and how does it differ from STIntersects and STWithin — specifically for a point on the polygon boundary?**

> **Average answer:** STContains returns 1 if the point is inside the polygon. STIntersects returns 1 if the point touches the polygon. STWithin is the inverse of STContains. A point on the boundary counts as intersecting but not containing.

> **Great answer:** STContains returns 1 if and only if the argument geometry lies entirely in the interior of the invoking geometry — no point of the argument may be on the boundary or exterior. For a point on the polygon boundary: STContains returns 0, STIntersects returns 1, and STWithin returns 0 (the point is not within the polygon, it is on the boundary). STWithin is the inverse of STContains: a.STContains(b) = 1 if and only if b.STWithin(a) = 1. The distinction between interior and boundary containment matters in geofencing: if you want to include customers on the boundary (e.g., delivery to the border street), use STIntersects, not STContains. Using STContains when you meant STIntersects causes a systematic exclusion of boundary-adjacent customers — in a city delivery zone, that could be 5-10% of addresses. In SQL Server geography, the boundary computation uses great-circle arcs, so a point exactly at a polygon vertex is on the boundary and STContains returns 0. The spatial index primary filter does not distinguish boundary vs interior — it only does grid-cell intersection — so the secondary filter is the only thing that handles this correctly.

**Challenge 1 — Write the SQL**

You work for a logistics company. The `DeliveryZones` table has a `ServiceArea` geography column defining delivery regions. The `Customers` table has a `Location` geography column. Write a query that, for each customer, returns their `CustomerId` and the name of the smallest (minimum area) delivery zone that contains their location.
```sql
SELECT
    c.CustomerId,
    dz.ZoneName AS SmallestZone
FROM dbo.Customers c
OUTER APPLY (
    SELECT TOP 1 dz2.ZoneName
    FROM dbo.DeliveryZones dz2
    WHERE dz2.ServiceArea.STContains(c.Location) = 1
    ORDER BY dz2.ServiceArea.STArea() ASC
) dz
ORDER BY c.CustomerId;
```

**Challenge 2 — Fix the performance problem**

```sql
-- This query identifies which delivery zone contains a given GPS point.
-- It runs in 3 seconds on a table with 1M customers and 50 delivery zones.
-- The DeliveryZones table has a spatial index on ServiceArea.
-- The Customers table has 1M rows with a spatial index on Location.

DECLARE @lat FLOAT = 47.6062, @lng FLOAT = -122.3321;
DECLARE @point geography = geography::Point(@lat, @lng, 4326);

SELECT c.CustomerId, c.FullName
FROM dbo.Customers c
INNER JOIN dbo.DeliveryZones dz ON 1=1
WHERE dz.ServiceArea.STContains(c.Location) = 1
  AND @point.STContains(dz.ServiceArea) = 1;

-- SET STATISTICS IO: Table 'Customers'. logical reads 42,000
--                   Table 'DeliveryZones'. logical reads 2
```

<details> <summary>Solution</summary>

**Root cause:** The query has two problems. First, the `CROSS JOIN` style `INNER JOIN ... ON 1=1` forces every customer to be joined with every zone, then filtered — this is a Cartesian product with 50M intermediate rows (1M × 50). Second, checking `@point.STContains(dz.ServiceArea)` is the wrong direction — you want to check which zone contains the point, not whether the point contains the zone.

```sql
-- Fixed query:
DECLARE @lat FLOAT = 47.6062, @lng FLOAT = -122.3321;
DECLARE @point geography = geography::Point(@lat, @lng, 4326);

SELECT c.CustomerId, c.FullName, dz.ZoneName
FROM dbo.DeliveryZones dz
CROSS APPLY (
    SELECT c.CustomerId, c.FullName
    FROM dbo.Customers c
    WHERE dz.ServiceArea.STContains(c.Location) = 1
) c
WHERE dz.ServiceArea.STContains(@point) = 1;
-- The zone containment of the @point runs once (not per row).
-- Then only customers in that zone are checked with the spatial index seek.
```

**Index to create:** Ensure spatial index on Customers.Location and DeliveryZones.ServiceArea.

**After fix — logical reads:** ~156 (spatial index on zones for @point) + ~156 (spatial index on customers for zone) = ~312 (from 42,002+).

</details>

**Challenge 3 — Explain the execution plan**

```sql
DECLARE @zone geography = geography::STGeomFromText('POLYGON((...))', 4326);

SELECT c.CustomerId, c.FullName
FROM dbo.Customers c
WHERE @zone.STContains(c.Location) = 1;
```

Execution plan shows:
- `Spatial Index Seek (IX_Customers_Location)` — Estimated rows: 5,200, Actual rows: 47
- `Filter` — Estimated rows: 47, Actual rows: 47
- `Key Lookup (PK_Customers)` — Estimated rows: 47, Actual rows: 47
- `SELECT`

Why does the Spatial Index Seek estimate 5,200 rows (0.52% of 1M) but only 47 rows are returned? What does this tell you about the spatial index and the secondary filter?

<details> <summary>Solution</summary>

**Why Spatial Index Seek estimates 5,200:** The spatial index primary filter returns all rows whose grid cells intersect the query polygon's bounding area. The query polygon's grid coverage spans many cells. The primary filter uses the grid tessellation to find candidate rows — this is a coarse approximation. The estimate of 5,200 means the optimizer estimates that ~0.52% of rows intersect the query polygon's grid cells. The actual 47 rows are the true positive matches after the exact STContains secondary filter.

**The 100x over-estimate is normal** for spatial indexes: the primary filter resolves grid-cell intersection, not actual spatial containment. A polygon can cover many grid cells but contain very few points. The secondary filter does the exact computation.

**To improve the estimate:** Higher grid density (HIGH at all levels) reduces the number of false positives from the primary filter but increases index size. The optimizer uses the primary filter estimate to choose the execution plan — if the estimate is far off, the plan shape may still be correct (spatial index seek is always better than scan for this query), but memory grants may be misallocated.

</details>

**Challenge 4 — Design the spatial index**

**Scenario:** You have a `Customers` table with 10M rows. Each row has a `Location` geography column. The table is used for geofencing queries: checking which customers are inside a given polygon. The polygons vary from small (neighborhood-sized, ~1 km²) to large (city-sized, ~1000 km²). The query runs ~500 times per second during business hours. The write rate is ~100 INSERTs/second. Reads must complete in <50ms at P95.

Design the spatial index strategy. Show the CREATE SPATIAL INDEX statement and explain each choice.

<details> <summary>Solution</summary>

```sql
-- Primary spatial index for geofencing queries
CREATE SPATIAL INDEX IX_Customers_Location
    ON dbo.Customers (Location)
    USING GEOGRAPHY_AUTO_GRID
    WITH (
        GRIDS = (LEVEL_1 = HIGH, LEVEL_2 = HIGH, LEVEL_3 = HIGH, LEVEL_4 = HIGH),
        CELLS_PER_OBJECT = 16,
        MAXDOP = 4
    );

-- Supporting B-tree index for bounding box pre-filter
CREATE INDEX IX_Customers_Lat_Lng
    ON dbo.Customers (Location.Lat, Location.Long)
    INCLUDE (CustomerId, FullName);
```

**Choices explained:**
1. `GEOGRAPHY_AUTO_GRID` (SQL Server 2012+): Automatically chooses the grid tessellation based on the spatial data's bounding box. Better than manual GEOGRAPHY_GRID for variable-sized query polygons.
2. `GRIDS = HIGH` at all levels: For neighborhood-sized polygons, HIGH grid density provides better primary filter selectivity (fewer false positives). The tradeoff is larger index size (~2x vs MEDIUM) and slightly slower index maintenance, but the read performance gain is critical at 500 queries/second.
3. `CELLS_PER_OBJECT = 16`: The maximum setting ensures that large polygons (city-sized) are well-represented at all grid levels, preventing them from appearing in too many grid cells (which would bloat the primary filter).
4. `MAXDOP = 4`: Limits parallel index creation operations.
5. **B-tree index on (Lat, Long):** Serves as a bounding box pre-filter for queries that know the approximate search area. For very large polygons, the bounding box lat/long columns can quickly eliminate 99% of rows using a standard B-tree seek, before the spatial index primary filter runs. This hybrid approach is optimal for mixed polygon sizes.

**Tradeoffs:**
- HIGH grid at all levels: ~2x index storage compared to MEDIUM
- 100 INSERTs/second × spatial index maintenance: ~800 additional logical writes/second
- Read performance: ~5-15ms P95 for bounded queries, ~30ms P95 for city-sized polygons

</details>

**Challenge 5 — Diagnose the geofencing bug**

**Scenario:** You deploy a geofencing feature that uses STContains to detect which customers are inside a newly defined "downtown promotional zone." The polygon was created by the marketing team using a web tool that exported WKT. In production, the feature is marking customers who are clearly outside the downtown area (e.g., customers in suburbs 20 km away) as "inside the zone," and marking customers in the actual downtown area as "outside." The query returns about 10x more customers than expected. No errors or exceptions are logged. The spatial index exists and the execution plan shows a Spatial Index Seek.

<details> <summary>Solution</summary>

**Root cause:** Ring orientation inversion. The WKT exported by the marketing team's web tool used clockwise vertex order for the polygon exterior ring. SQL Server geography interprets clockwise polygons as the complement — the entire globe minus the intended area. So customers in the suburbs (anywhere outside the intended polygon) are "contained" in the inverse polygon, and customers in downtown are "outside" because they are in the hole.

**Verification query:**
```sql
-- Check ring orientation
SELECT
    ZoneId,
    ServiceArea.EnvelopeAngle() AS EnvelopeAngle,
    ServiceArea.STAsText() AS WKT
FROM dbo.PromotionalZones
WHERE ServiceArea.EnvelopeAngle() > 90;
-- EnvelopeAngle > 90 means the polygon covers > 1/4 of the earth
-- This is almost certainly an inverted polygon
```

**Fix:**
```sql
UPDATE dbo.PromotionalZones
SET ServiceArea = ServiceArea.ReorientObject();
-- ReorientObject() reverses ring direction

-- Or use MakeValid() + filter on area size
UPDATE dbo.PromotionalZones
SET ServiceArea = ServiceArea.MakeValid().ReorientObject()
WHERE ServiceArea.EnvelopeAngle() > 89;
```

**Prevention:**
```sql
-- Add CHECK constraint to prevent inverted polygons
ALTER TABLE dbo.PromotionalZones
ADD CONSTRAINT CK_PromotionalZones_ValidOrientation
CHECK (
    ServiceArea.EnvelopeAngle() <= 90
    AND ServiceArea.STIsValid() = 1
);
```

**After fix:** The query returns the expected ~5% of customers in the downtown area. The spatial index seek returns accurate results.

</details>
