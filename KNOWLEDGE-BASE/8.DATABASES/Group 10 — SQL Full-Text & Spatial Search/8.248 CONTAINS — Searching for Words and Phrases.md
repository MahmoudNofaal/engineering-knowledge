### Where This Fits

`CONTAINS` is the primary T-SQL predicate for full-text search in SQL Server. It enables precise word, phrase, boolean, proximity, and linguistic-form searches against full-text-indexed columns. A .NET backend engineer uses CONTAINS when building search features that need operator-level control: "find products containing both 'wireless' and 'bluetooth' but not 'headphone'" or "find documents where 'contract' appears near 'termination'." Unlike FREETEXT, which is a black box for natural language search, CONTAINS gives the developer exact control over which terms match and how they combine. The interview signal comes from understanding the CONTAINS operator syntax, the difference between INFLECTIONAL and THESAURUS forms, and how to use CONTAINS vs FREETEXT vs LIKE.

### Classification

CONTAINS is a **full-text predicate** that belongs to the SQL Server full-text query family. It is evaluated by the full-text engine, not the relational engine. The predicate is **not SARGable** in the B-tree sense — it does not use a relational index seek. Instead, it uses the inverted index maintained by the FTS engine. The full-text engine returns a list of matching document keys, and the relational engine then performs key lookups into the base table. The predicate supports: simple term (`'word'`), phrase (`'"exact phrase"'`), boolean (`'word1 AND word2'`, `'word1 OR word2'`, `'word1 AND NOT word2'`), proximity (`'word1 NEAR word2'`), inflectional (`'FORMSOF(INFLECTIONAL, run)'`), and thesaurus (`'FORMSOF(THESAURUS, word)'`) forms.

## Deep Mechanics

### How the Engine Executes a CONTAINS Query

**Simple Word Search: `CONTAINS(col, 'wireless')`**

1. **Predicate parsing** — The relational engine parser identifies `CONTAINS` as a full-text predicate function. It extracts the column reference (`col`) and the search condition (`'wireless'`). The compiler determines that this predicate must be handed off to the full-text engine.

2. **Optimizer plan generation** — The optimizer generates an execution plan with a `FullTextMatch` operator. This operator represents the handoff to the FTS engine. The optimizer cannot estimate the cost of this operator — it shows as 0% in the plan, but the actual work is done outside the relational engine.

3. **Word breaking** — The FTS engine takes the search term 'wireless' and passes it through the word breaker for the column's configured language (e.g., English LCID 1033). The word breaker confirms 'wireless' is a single token. If the search term were 'wire-less', the word breaker might split it into 'wire' and 'less' or keep it as one token depending on the language rules.

4. **Stopword check** — The FTS engine checks the configured stoplist for the token 'wireless'. If 'wireless' is a stopword (unlikely, as it is a content word), it would be excluded and the search would return zero results. Stopwords are typically noise words like "the", "a", "and", "or", "is".

5. **Inverted index lookup** — The FTS engine looks up the normalized token 'wireless' in the inverted index. The inverted index stores: `wireless → [ProductId_1, ProductId_2, ..., ProductId_N]`. Each entry in the document list also stores the occurrence count (term frequency) and position data (for NEAR queries).

6. **Document key list** — The FTS engine returns the list of matching document keys (primary key values) to the relational engine. For a CONTAINS query (not CONTAINSTABLE), rank values are not returned — just the keys.

7. **Key lookup** — The relational engine takes each document key and performs a clustered index seek (or non-clustered index seek if a covering index exists) into the base table to retrieve any columns in the SELECT list.

8. **Filtering (if additional predicates exist)** — Any non-full-text predicates in the WHERE clause (e.g., `AND IsActive = 1`) are applied as a separate filter after the key lookups.

**Phrase Search: `CONTAINS(col, '"wireless mouse"')`**

The engine processes a phrase search similarly, but instead of looking up a single token, it looks up each token individually, then intersects the document lists and checks that the tokens appear in the specified order and adjacent to each other (or within the same word position). The inverted index stores position information for each token occurrence, enabling the engine to verify adjacency.

**Boolean Search: `CONTAINS(col, 'wireless AND bluetooth NOT headphone')`**

1. The FTS engine looks up 'wireless' → document list A
2. Looks up 'bluetooth' → document list B
3. Computes intersection A ∩ B (for AND)
4. Looks up 'headphone' → document list C
5. Computes (A ∩ B) \ C (for NOT)
6. Returns final document key list

**Inflectional Search: `CONTAINS(col, 'FORMSOF(INFLECTIONAL, run)')`**

1. The stemmer reduces 'run' to its root form.
2. The stemmer generates all inflectional forms of the root: runs, running, ran, runner.
3. The engine looks up each form in the inverted index.
4. Results are unioned. `FORMSOF(INFLECTIONAL, run)` matches any row containing run, runs, running, ran, or runner.

**Thesaurus Search: `CONTAINS(col, 'FORMSOF(THESAURUS, big)')`**

1. The FTS engine reads the thesaurus XML file for the configured language.
2. It finds <expansion> entries for 'big' — synonyms like large, huge, enormous, gigantic.
3. The engine looks up each synonym in the inverted index.
4. Results are unioned.

### SQL Visibility

```sql
-- ============================================================
-- CONTAINS — All Predicate Types
-- ============================================================

-- Setup: assume full-text index on dbo.Products exists
-- as created in 8.247

-- 1. Simple word search
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(ProductName, 'wireless');

-- 2. Phrase search (exact phrase in double quotes)
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(ProductDescription, '"wireless mouse"');

-- 3. Boolean AND — both terms must match
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(ProductDescription, '"wireless" AND "bluetooth"');

-- 4. Boolean OR — either term matches
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(ProductDescription, '"wireless" OR "bluetooth"');

-- 5. Boolean AND NOT — exclude term
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(ProductDescription, '"wireless" AND NOT "headphone"');

-- 6. Complex boolean with parentheses
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(ProductDescription,
    '("wireless" OR "bluetooth") AND ("mouse" OR "keyboard") AND NOT "headphone"');

-- 7. Inflectional search (stemming)
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(ProductDescription, 'FORMSOF(INFLECTIONAL, run)');
-- Matches: run, runs, running, ran, runner

-- 8. Thesaurus search (synonym expansion)
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(ProductDescription, 'FORMSOF(THESAURUS, big)');
-- Matches: big, large, huge, enormous (depends on thesaurus XML)

-- 9. Proximity search (NEAR)
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(ProductDescription, '"wireless" NEAR "mouse"');
-- Words within ~500 tokens of each other

-- 10. Proximity with distance and order
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(ProductDescription,
    'NEAR(("wireless", "mouse"), 10, TRUE)');
-- Within 10 tokens, in order (wireless before mouse)

-- 11. Weighted terms (ISABOUT) — for ranking with CONTAINSTABLE
-- CONTAINS ignores weights; weights are for CONTAINSTABLE ranking
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(ProductDescription,
    'ISABOUT("wireless" WEIGHT(0.8), "bluetooth" WEIGHT(0.2))');
-- ISABOUT with CONTAINS: any term can match (implicit OR)
-- Weights are ignored for CONTAINS but used for CONTAINSTABLE ranking

-- 12. Search specific columns (column list)
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(
    (ProductName, ProductDescription, ProductCategory),
    '"wireless" AND "bluetooth"');

-- 13. Search all full-text indexed columns with asterisk
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(*, '"wireless" AND "bluetooth"');

-- 14. CONTAINS with language parameter
SELECT ProductId, ProductName
FROM dbo.Products
WHERE CONTAINS(
    ProductDescription,
    'FORMSOF(INFLECTIONAL, run)',
    LANGUAGE 1033);

-- 15. CONTAINS with table alias
SELECT p.ProductId, p.ProductName
FROM dbo.Products p
WHERE CONTAINS(p.ProductDescription, '"wireless"');
```

### Execution Plan Analysis

```sql
SELECT p.ProductId, p.ProductName
FROM dbo.Products p
WHERE CONTAINS(p.ProductDescription, '"wireless" AND "bluetooth"')
  AND p.IsActive = 1
ORDER BY p.ProductName;
```

**Expected plan shape:**
```
[FullTextMatch — CONTAINS(p.ProductDescription, '"wireless" AND "bluetooth"')]
    → [Filter: p.IsActive = 1]
    → [Clustered Index Seek — key lookup]
    → [Sort — ORDER BY ProductName]
```

- `FullTextMatch` — The relational engine hands the predicate to the FTS engine. The FTS engine does the word breaking, stopword check, inverted index lookups for both 'wireless' and 'bluetooth', and the intersection of document lists. It returns the list of matching ProductId values. This operator shows 0% estimated cost.
- `Filter` — The `IsActive = 1` predicate is applied after the key lookups unless a covering index exists.
- `Clustered Index Seek` — For each matching document key, a seek into the clustered index to retrieve `ProductName` and `IsActive`. 500 matches = 500 seeks.
- `Sort` — Ordering by `ProductName`. If many rows match, this can be expensive.

**Estimated Cost:** FullTextMatch = 0%, Clustered Index Seek = ~95%, Sort = ~3%, Filter = ~2%.

**Without full-text index (LIKE alternative):**
```
[Clustered Index Scan — full table scan]
    → [Filter: ProductDescription LIKE '%wireless%' AND ProductDescription LIKE '%bluetooth%']
    → [Sort]
```
100% cost on the scan — no seeks possible. Full table scan of 100K rows × 2KB = 200MB scanned.

### Failure Modes

**Failure Mode 1: Leading wildcard in CONTAINS — invalid syntax**

```sql
-- ❌ CONTAINS does not support wildcards as LIKE does
SELECT ProductId FROM dbo.Products
WHERE CONTAINS(ProductDescription, '%wireless%');
-- Msg 7630: Syntax error near '%wireless%' in the full-text search condition '%wireless%'.
```

CONTAINS does not accept `%` wildcard characters. Use `LIKE` for substring matching or use the full-text `*` (prefix term) wildcard: `'"wireless*"'`.

**Failure Mode 2: NULL predicate returns FALSE**

```sql
-- ❌ Does not match NULL description
SELECT ProductId FROM dbo.Products
WHERE CONTAINS(ProductDescription, 'wireless');
-- If ProductDescription is NULL, the row is never returned.
-- NULL is the absence of text — no tokens to index.
```

**Failure Mode 3: Stopwords cause zero results**

```sql
-- ❌ Matches NOTHING if 'the' is a stopword
SELECT ProductId FROM dbo.Products
WHERE CONTAINS(ProductDescription, '"the"');
```

### EF Core Implementation

```csharp
// ============================================================
// EF Core — CONTAINS Search Service
// ============================================================

public class ProductSearchRequest
{
    public string? SearchTerm { get; init; }
    public bool MatchAllTerms { get; init; } = true;  // AND vs OR
    public bool UseStemming { get; init; }
    public bool UseThesaurus { get; init; }
    public int? CategoryId { get; init; }
    public decimal? MaxPrice { get; init; }
}

public class ProductSearchResult
{
    public int ProductId { get; set; }
    public string ProductName { get; set; } = string.Empty;
    public string? ShortDescription { get; set; }
    public decimal UnitPrice { get; set; }
    public string? BrandName { get; set; }
    public string? CategoryName { get; set; }
}

public interface IProductSearchService
{
    Task<IReadOnlyList<ProductSearchResult>> SearchAsync(
        ProductSearchRequest request,
        CancellationToken cancellationToken = default);
}

public class ProductSearchService : IProductSearchService
{
    private readonly ApplicationDbContext _dbContext;

    public ProductSearchService(ApplicationDbContext dbContext)
    {
        _dbContext = dbContext;
    }

    public async Task<IReadOnlyList<ProductSearchResult>> SearchAsync(
        ProductSearchRequest request,
        CancellationToken cancellationToken = default)
    {
        if (string.IsNullOrWhiteSpace(request.SearchTerm))
            return Array.Empty<ProductSearchResult>();

        // Build CONTAINS predicate from search term
        var terms = request.SearchTerm.Split(' ', StringSplitOptions.RemoveEmptyEntries);
        var booleanOp = request.MatchAllTerms ? "AND" : "OR";

        // Build predicate parts
        var predicateParts = new List<string>();
        foreach (var term in terms)
        {
            var sanitized = term.Replace("\"", "\\\"");  // escape double quotes
            if (request.UseStemming)
                predicateParts.Add($"FORMSOF(INFLECTIONAL, \"{sanitized}\")");
            else if (request.UseThesaurus)
                predicateParts.Add($"FORMSOF(THESAURUS, \"{sanitized}\")");
            else
                predicateParts.Add($"\"{sanitized}\"");
        }

        var predicate = string.Join($" {booleanOp} ", predicateParts);

        // EF Core cannot generate CONTAINS in LINQ — use raw SQL
        var sql = $@"
            SELECT p.ProductId, p.ProductName, p.ShortDescription,
                   p.UnitPrice, p.BrandName, p.CategoryName
            FROM dbo.Products p
            WHERE CONTAINS(
                (p.ProductName, p.ShortDescription, p.LongDescription,
                 p.CategoryName, p.BrandName, p.Tags),
                {{0}}
            )
            AND p.IsActive = 1
            {(request.CategoryId.HasValue ? "AND p.CategoryId = {1}" : "")}
            {(request.MaxPrice.HasValue ? "AND p.UnitPrice <= {2}" : "")}
            ORDER BY p.ProductName";

        var results = await _dbContext.Database
            .SqlQueryRaw<ProductSearchResult>(sql,
                predicate,
                request.CategoryId,
                request.MaxPrice)
            .ToListAsync(cancellationToken);

        return results;
    }
}
```

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL uses tsquery for boolean word search

-- PostgreSQL: simple term search
SELECT product_id, product_name
FROM products
WHERE search_vector @@ to_tsquery('english', 'wireless');

-- PostgreSQL: boolean AND
SELECT product_id, product_name
FROM products
WHERE search_vector @@ to_tsquery('english', 'wireless & bluetooth');

-- PostgreSQL: boolean OR
SELECT product_id, product_name
FROM products
WHERE search_vector @@ to_tsquery('english', 'wireless | bluetooth');

-- PostgreSQL: boolean NOT
SELECT product_id, product_name
FROM products
WHERE search_vector @@ to_tsquery('english', 'wireless & !bluetooth');

-- PostgreSQL: phrase search (uses <-> operator for adjacency)
SELECT product_id, product_name
FROM products
WHERE search_vector @@ to_tsquery('english', 'wireless <-> mouse');

-- PostgreSQL: stemming is automatic — to_tsquery applies the stemmer
-- to all terms. No FORMSOF equivalent needed — stemming is implicit.

-- Key difference: PostgreSQL applies stemming by default in to_tsvector
-- and to_tsquery. SQL Server CONTAINS requires explicit FORMSOF(INFLECTIONAL)
-- for stemming. PostgreSQL also does not have a separate thesaurus file
-- at the query level — synonym expansion must be done via dictionary configuration
-- or application code.
```

## Gotchas and Production Pitfalls

### 1. Leading Wildcard Is Not Supported in CONTAINS

**Pitfall:** Developers coming from LIKE syntax try to use `%` and `_` wildcards in CONTAINS, which causes a syntax error.

```sql
-- ❌ Syntax error
SELECT ProductId FROM dbo.Products
WHERE CONTAINS(ProductDescription, '%wireless%');
-- Msg 7630: Syntax error near '%wireless%' in the full-text search condition
```

**Symptom:** Parser error at query execution. The application crashes when the search term contains a `%` character.

**Fix:** CONTAINS does support prefix matching using the asterisk (`*`) at the end of a term:

```sql
-- ✅ Prefix matching: matches 'wireless', 'wirelessly', 'wirelessness'
SELECT ProductId FROM dbo.Products
WHERE CONTAINS(ProductDescription, '"wireless*"');

-- For leading wildcard behavior, use LIKE instead of CONTAINS
-- But accept that LIKE '%wireless' will scan the entire table.
```

**Cost of not fixing:** Search crashes when users type `%` in search boxes. In production, this causes HTTP 500 errors for all search queries containing wildcard characters.

### 2. Stopwords Cause "No Results" for Common Terms

**Pitfall:** Searching for common words like "the", "and", "or" that are in the system stoplist returns zero results — no error, just empty. This confuses both users and developers.

```sql
-- ❌ Returns 0 rows even if every document contains 'the'
SELECT DocumentId FROM dbo.Documents
WHERE CONTAINS(DocumentContent, '"the"');
-- 0 rows — 'the' is in the system stoplist for English
```

**Symptom:** Search for common words returns zero results. Users think the search is broken.

**Fix:** Either exclude stopwords from the predicate at the application level (filter them out before building the CONTAINS predicate), or create a custom stoplist without those stopwords. However, removing stopwords increases the full-text index size and may degrade performance for common words.

```sql
-- Option 1: Filter stopwords in application code (recommended)
-- In C#, before building the CONTAINS predicate:
var stopwords = new HashSet<string>(StringComparer.OrdinalIgnoreCase)
{
    "the", "a", "an", "and", "or", "in", "on", "at", "to", "for",
    "of", "by", "with", "is", "it", "as", "be", "but", "not"
};
terms = terms.Where(t => !stopwords.Contains(t)).ToArray();

-- Option 2: Create custom stoplist without common stopwords
CREATE FULLTEXT STOPLIST LenientStoplist;
ALTER FULLTEXT STOPLIST LenientStoplist ADD 'the' LANGUAGE 1033;
-- (This adds 'the' back as a non-stopword by removing it from the stoplist effect)
-- Note: this is counterintuitive — adding a word to a custom stoplist makes it
-- a stopword. You cannot undo this directly. Create a fresh stoplist instead.
```

**Cost of not fixing:** Search returns empty results for many common queries. Users lose trust in the search feature. Support tickets increase.

### 3. FORMSOF(INFLECTIONAL) Does Not Handle All Forms

**Pitfall:** The stemmer may not generate all expected inflectional forms. For example, `FORMSOF(INFLECTIONAL, run)` generates "run", "runs", "running", "ran", "runner" — but NOT "runny", "runoff", or other compound words.

```sql
-- ❌ May not match expected forms
SELECT ArticleId FROM dbo.Articles
WHERE CONTAINS(Body, 'FORMSOF(INFLECTIONAL, run)');
-- Does match: run, runs, running, ran, runner
-- Does NOT match: runway, runabout, run-down (these are different lexical words)
```

**Symptom:** Search misses documents with related words that the developer expected to match. The stemmer is algorithmic (Porter stemmer for English) and cannot handle irregular forms or contextual variations.

**Fix:** Combine FORMSOF(INFLECTIONAL) with explicit OR for expected variations, or use FORMSOF(THESAURUS) for known synonyms:

```sql
-- ✅ Combine inflectional with explicit terms for known variations
SELECT ArticleId FROM dbo.Articles
WHERE CONTAINS(Body,
    'FORMSOF(INFLECTIONAL, run) OR "runway" OR "runabout"');
```

**Cost of not fixing:** Documents that use related but differently-derived words are missed. For a legal document search searching for "terminate", documents containing "termination" match via inflectional (`FORMSOF(INFLECTIONAL, terminate)`), but documents containing "terminable" or "termini" (unrelated meaning) may also incorrectly match or be missed.

### 4. CONTAINS with Column List — Must Match Full-Text Index

**Pitfall:** Specifying a column list in CONTAINS that includes columns NOT in the full-text index causes an error. The column list must be a subset of the indexed columns.

```sql
-- ❌ Error: UnitPrice is not full-text indexed
SELECT ProductId FROM dbo.Products
WHERE CONTAINS((ProductName, UnitPrice), '"wireless"');
-- Msg 7605: Column 'UnitPrice' in the full-text search condition
-- is not full-text indexed.
```

**Symptom:** Parser error. The developer lists columns in the CONTAINS predicate that are not part of the full-text index definition.

**Fix:** Only use columns that are defined in the full-text index. Check with:

```sql
-- Check which columns are indexed
SELECT c.name AS ColumnName
FROM sys.fulltext_index_columns fic
INNER JOIN sys.columns c
    ON c.object_id = fic.object_id
    AND c.column_id = fic.column_id
WHERE fic.object_id = OBJECT_ID('dbo.Products');
```

**Cost of not fixing:** Application crashes when the search query includes a non-indexed column. Must add the column to the full-text index (which requires a full crawl) or remove it from the query.

### 5. Performance Collapse with Too Many Terms

**Pitfall:** Users can type very long search queries with many terms. Each term in a CONTAINS predicate requires a separate inverted index lookup. With 20 terms, the FTS engine does 20 index lookups plus the boolean intersection/union operations.

```sql
-- ❌ 10+ term search — performance degrades
SELECT ProductId FROM dbo.Products
WHERE CONTAINS(ProductDescription,
    '"wireless" AND "bluetooth" AND "mouse" AND "keyboard"
     AND "charger" AND "cable" AND "adapter" AND "stand"
     AND "cover" AND "screen"');
-- 10 inverted index lookups + boolean operations
```

**Symptom:** CONTAINS query slows down with many terms. Each additional term adds an inverted index lookup. The boolean intersection/union becomes expensive.

**Fix:** Limit the number of search terms at the application level:

```csharp
// C#: limit search terms to 6
var terms = request.SearchTerm
    .Split(' ', StringSplitOptions.RemoveEmptyEntries)
    .Take(6)
    .Select(t => $"\"{t.Replace("\"", "\\\"")}\"");
var predicate = string.Join(" AND ", terms);
```

**Cost of not fixing:** Search response times degrade linearly with the number of search terms. A 20-term search may take 5-10x longer than a 2-term search, causing timeouts in high-traffic applications.

## Interview Arsenal

### Question Bank

1. **What is the CONTAINS predicate and what problem does it solve?** — Precise word and phrase search over text columns using the inverted index, avoiding full table scans.

2. **How does SQL Server evaluate a CONTAINS predicate internally?** — Relational engine hands off to FTS engine → word breaking → stopwords → inverted index lookup → document keys returned → key lookup into base table.

3. **What is the performance difference between CONTAINS and LIKE '%word%'?** — CONTAINS uses inverted index (constant-time per token); LIKE forces full scan. At 100K rows: ~12ms vs ~480ms, ~4 vs ~45,832 logical reads.

4. **What happens when you use CONTAINS with a stopword?** — The stopword is filtered out silently. The query returns fewer results or zero results depending on the predicate.

5. **CONTAINS vs FREETEXT — when would you use each?** — CONTAINS for precise operator control (AND, OR, NEAR, FORMSOF). FREETEXT for natural language, user-facing search bars where meaning matching matters more than exact terms.

6. **What does the execution plan show for a CONTAINS query with boolean AND?** — FullTextMatch (0% cost) → Filter → Clustered Index Seek (key lookups). The boolean intersection happens inside the FTS engine.

7. **How does CONTAINS scale with multiple terms?** — Each term adds an inverted index lookup. At 6+ terms, performance degrades. Limit terms to ~6 or use FREETEXT for many-term queries.

8. **How do you implement CONTAINS in EF Core and Dapper?** — Both require raw SQL. EF Core uses FromSqlRaw or SqlQuery (EF Core 7+). Dapper uses QueryAsync with parameterized SQL.

### Spoken Answers

**Q1: What is the CONTAINS predicate and what problem does it solve?**

> **Average answer:** CONTAINS lets you search for words in text columns. It is faster than LIKE.

> **Great answer:** CONTAINS is a full-text predicate that delegates word search to the SQL Server Full-Text Engine, which uses an inverted index rather than a B-tree. The problem it solves is that B-tree indexes cannot efficiently answer "which rows contain this word?" — B-trees are designed for prefix matching and equality, not substring or word-level search. Without full-text search, you would need `LIKE '%wireless%'`, which forces a full clustered index scan because the leading wildcard prevents any index seek. On a 100K row table with 2KB average text, that is ~45,000 logical reads and ~500ms. CONTAINS reduces that to ~4 logical reads and ~12ms by looking up the word 'wireless' in the inverted index, which returns the document keys in O(1) per token. CONTAINS supports exact word matching, phrase matching with double quotes, boolean operators (AND, OR, AND NOT), proximity search with NEAR, stemming with FORMSOF(INFLECTIONAL), and thesaurus synonym expansion with FORMSOF(THESAURUS). It is the right tool whenever you need precise, operator-controlled word search within a table that has a populated full-text index.

**Q5: CONTAINS vs FREETEXT — when would you use each?**

> **Average answer:** CONTAINS is for exact matches, FREETEXT is for fuzzy matching.

> **Great answer:** CONTAINS and FREETEXT are complementary full-text predicates that use the same inverted index infrastructure — same word breaker, same stemmer, same stoplist — but differ in how they process the search predicate and combine results. CONTAINS gives the developer explicit control: every term, operator, and form is specified precisely. You use CONTAINS when you need to build a search query programmatically with known requirements: "find products where the description mentions 'wireless' AND 'bluetooth' within 10 words of each other, but exclude anything with 'headphone'." FREETEXT, by contrast, takes a natural language phrase, breaks it into individual terms using the word breaker, applies stemming to each term, expands through the thesaurus, and returns rows where any of the derived terms match — using implicit OR with internal weighting so that the best matches appear first. I use CONTAINS for programmatic search with defined criteria, such as filtering products by specific attributes, legal document search with precise term requirements, or any search where the user can use advanced operators. I use FREETEXT for user-facing free-text search bars where users type natural language queries like "cool wireless stuff for my desk" — FREETEXT handles stemming and weighting automatically, and the user does not need to know about boolean operators. The performance of both is similar because both use the same inverted index.

**Q8: How do you implement CONTAINS in EF Core and Dapper?**

> **Average answer:** You have to use raw SQL because EF Core does not support CONTAINS in LINQ.

> **Great answer:** Neither EF Core nor Dapper has built-in support for SQL Server's CONTAINS predicate, so both require raw SQL. In EF Core, you use `FromSqlRaw` or the newer `SqlQuery` method (EF Core 7+) to execute the CONTAINS query and map results to a POCO. The key consideration is that the CONTAINS predicate string must be built by the application — you join terms with AND or OR, wrap phrases in double quotes, and construct FORMSOF expressions. You must parameterize the CONTAINS predicate string, not concatenate user input, to prevent SQL injection. In Dapper, you use `QueryAsync` with a parameterized SQL string. Both approaches require you to handle search term sanitization: escape double quotes, limit the number of terms, filter stopwords, and handle empty search gracefully. For search results that include ranking, CONTAINSTABLE is preferred over CONTAINS because it returns a rank value (0-1000) in the `[KEY]` and `RANK` columns, which you JOIN to the base table and ORDER BY RANK DESC. I typically build an abstraction layer that accepts a search string and additional filter parameters, builds the appropriate CONTAINS or CONTAINSTABLE predicate internally, and returns a strongly-typed result set.

If a developer says "I used CONTAINS for search," the follow-up is: "Show me the execution plan for a CONTAINS query with a boolean AND operator. Where is the cost? How do you reduce the key lookup overhead?" The candidate who describes the FullTextMatch operator, the key lookup cost proportional to the number of matches, and suggests covering indexes to reduce IO demonstrates deep production experience.

### When to Use Each CONTAINS Predicate Type

### Application Checklist

- [ ] A full-text index exists on the column(s) and is populated
- [ ] The column list in CONTAINS matches columns in the full-text index
- [ ] Search terms are sanitized (double quotes escaped, stopwords filtered)
- [ ] Boolean operator matches requirement (AND vs OR)
- [ ] Terms are limited to 6 or fewer for performance
- [ ] FORMSOF language matches the column's language LCID
- [ ] Additional WHERE filters have supporting indexes (covering index for post-FTS filtering)
- [ ] Application handles the case where CONTAINS returns zero rows (stopword-only query)

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|O(1) word lookup via inverted index|Requires full-text index creation and maintenance|
|Precise boolean operator control|Predicate building logic in application code|
|Linguistic features (stemming, thesaurus)|Thesaurus configuration via XML files|
|Phrase and proximity search|Each additional term adds inverted index lookup cost|

### Scale Thresholds

- **Relevant when table exceeds ~10K rows** — Below this, LIKE with a trailing wildcard may be fast enough.
- **Critical when table exceeds ~100K rows** — CONTAINS reduces logical reads from 45K+ to ~4 for single-term searches.
- **Required when query runs more than ~50 queries/second** — Each LIKE query scans the full table; at 50 qps, this saturates IO.
- **Degraded when predicate exceeds ~6 terms** — Each term adds an inverted index lookup; consider FREETEXT for many-term queries.
- **Prohibitive when combined with FORMSOF on all terms for very large catalogs** — FORMSOF expands each term to multiple dictionary lookups.

**Challenge 3 — Explain the execution plan**

```sql
-- Query with CONTAINS on two columns
SELECT p.ProductId, p.ProductName, p.CategoryName
FROM dbo.Products p
WHERE CONTAINS(p.ProductName, '"wireless"')
   OR CONTAINS(p.ProductCategory, '"wireless"');
```

The execution plan shows two FullTextMatch operators followed by a Concatenation and then a Clustered Index Seek. Why is there a Concatenation? What would change if you used AND instead of OR?

<details>
<summary>Solution</summary>

**Why Concatenation:** The OR operator in CONTAINS across two different columns cannot be handled as a single FTS engine call. The FTS engine evaluates CONTAINS on a per-column basis per predicate. For `CONTAINS(ProductName, '"wireless"') OR CONTAINS(ProductCategory, '"wireless"')`, the engine must:
1. Query the FTS index for 'wireless' in ProductName → get document list A
2. Query the FTS index for 'wireless' in ProductCategory → get document list B
3. Compute the union A ∪ B
4. Deduplicate (a row matching both columns should appear once)

The Concatenation operator combines the two result sets, and a Sort (or Hash Match) is used for deduplication.

**If AND were used instead:** `CONTAINS(ProductName, '"wireless"') AND CONTAINS(ProductCategory, '"wireless"')` — The engine would:
1. Query for 'wireless' in ProductName → document list A
2. Query for 'wireless' in ProductCategory → document list B
3. Compute intersection A ∩ B
4. No deduplication needed

The plan would show two FullTextMatch operators followed by a Merge Join (Intersect) or Hash Match (Intersect) instead of Concatenation + Sort.

**Better approach:** Use the column list syntax to search both columns in a single CONTAINS call:

```sql
-- Single FullTextMatch operator — more efficient
SELECT p.ProductId, p.ProductName, p.CategoryName
FROM dbo.Products p
WHERE CONTAINS((p.ProductName, p.ProductCategory), '"wireless"');
-- The FTS engine handles both columns internally — one inverted index lookup
-- that considers all columns for this catalog.
```

</details>

**Challenge 4 — Diagnose the concurrency problem**

A production query uses CONTAINS to search a 500K row Articles table. The query runs 200 times/second during peak hours. Each query matches ~50 articles and returns ArticleId, Title, and PublishedDate. The execution plan shows 100% cost on Clustered Index Seek (50 executions per query, 10,000 seeks/second). CPU is at 80%. What is the fix?

<details>
<summary>Solution</summary>

**Root cause:** The Clustered Index Seek costs 100% because every matched document key requires a full clustered index lookup to retrieve Title and PublishedDate. At 200 qps × 50 seeks = 10,000 seeks/second, each seek reads 2-3 pages from the clustered index B-tree. This causes high CPU and IO.

**Detection query:**

```sql
-- Check index usage statistics
SELECT
    OBJECT_NAME(i.object_id) AS TableName,
    i.name AS IndexName,
    i.type_desc,
    us.user_seeks,
    us.user_scans,
    us.user_lookups,
    us.user_updates
FROM sys.dm_db_index_usage_stats us
INNER JOIN sys.indexes i
    ON i.object_id = us.object_id
    AND i.index_id = us.index_id
WHERE us.object_id = OBJECT_ID('dbo.Articles')
  AND i.type = 1;  -- Clustered index
```

**Fix:** Create a covering non-clustered index that includes the columns returned by the SELECT:

```sql
CREATE NONCLUSTERED INDEX IX_Articles_SearchCovering
    ON dbo.Articles (ArticleId)
    INCLUDE (Title, PublishedDate);
```

This changes the key lookup from the clustered index (which includes all columns — wide) to the covering non-clustered index (narrow — only 3 columns). Each lookup reads 1 page instead of 2-3, reducing IO by ~60%.

**In .NET:** Consider implementing in-memory caching for the most frequent search terms using `IMemoryCache` or `IDistributedCache` to reduce the database load:

```csharp
public async Task<IReadOnlyList<ArticleSearchResult>> SearchWithCacheAsync(
    string searchTerm, CancellationToken cancellationToken = default)
{
    var cacheKey = $"search:{searchTerm}";

    return await _cache.GetOrCreateAsync(cacheKey, async entry =>
    {
        entry.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(1);
        return await SearchAsync(searchTerm, cancellationToken);
    }) ?? Array.Empty<ArticleSearchResult>();
}
```

**Challenge 5 — Design the index**

**Scenario:** A legal document search system. The `dbo.LegalDocuments` table has 2M documents with columns: `DocumentId` (INT PK), `Title` (NVARCHAR(500)), `FullText` (NVARCHAR(MAX)), `PracticeArea` (NVARCHAR(100)), `Jurisdiction` (NVARCHAR(100)), `FilingDate` (DATE), `IsConfidential` (BIT). Users search by word and phrase with boolean operators. The most common query is: find documents in a specific practice area that contain "confidential" AND "agreement" within 5 words of "termination" but NOT "expired". Response time must be under 2 seconds. Document updates are rare (100/day), but queries run at ~30/second.

<details>
<summary>Solution</summary>

```sql
-- Full-text catalog on fast SSD drive
CREATE FULLTEXT CATALOG LegalCatalog;
GO

-- Full-text index on Title and FullText
-- Use MANUAL change tracking (100 updates/day — no need for AUTO)
CREATE FULLTEXT INDEX ON dbo.LegalDocuments
(
    Title       LANGUAGE 1033,
    FullText    LANGUAGE 1033
)
KEY INDEX PK_LegalDocuments
ON LegalCatalog
WITH
(
    CHANGE_TRACKING MANUAL,
    STOPLIST = SYSTEM
);
GO

-- Covering index for the filtered columns (PracticeArea, Jurisdiction, FilingDate)
-- used in WHERE clauses alongside CONTAINS
CREATE NONCLUSTERED INDEX IX_LegalDocuments_SearchFilter
    ON dbo.LegalDocuments (PracticeArea, Jurisdiction, FilingDate DESC)
    INCLUDE (DocumentId, Title, IsConfidential)
    WHERE IsConfidential = 0;  -- Most searches exclude confidential docs
GO

-- Schedule incremental population every 2 hours:
-- (SQL Agent job)
-- ALTER FULLTEXT INDEX ON dbo.LegalDocuments START UPDATE POPULATION;

-- Example query:
DECLARE @PracticeArea NVARCHAR(100) = 'Corporate Law';

SELECT ld.DocumentId, ld.Title, ld.FilingDate
FROM dbo.LegalDocuments ld
WHERE ld.PracticeArea = @PracticeArea
  AND ld.IsConfidential = 0
  AND CONTAINS(ld.FullText,
      'NEAR(("confidential", "agreement"), 5) AND "termination" AND NOT "expired"')
ORDER BY ld.FilingDate DESC;
```

**Why these choices:**
- CHANGE_TRACKING MANUAL: 100 updates/day is low; schedule updates every 2 hours
- Covering filtered index: The filter `IsConfidential = 0` is common; the covering index includes all non-FTS query columns
- NEAR with distance 5: The most common proximity requirement is supported natively
- The 30 qps × ~20 matches = ~600 key lookups/second, the covering index reduces each lookup to a narrow non-clustered index read

**What NOT to index:**
- Do NOT index FullText with a B-tree index — it is too large and CONTAINS handles it
- Do NOT create a separate full-text index for each practice area — the single index with CONTAINS + PracticeArea filter is sufficient

</details>
