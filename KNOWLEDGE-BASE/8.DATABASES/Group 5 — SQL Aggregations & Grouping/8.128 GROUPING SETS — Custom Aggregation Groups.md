### Where This Fits

GROUPING SETS lets the developer specify exactly which grouping combinations to aggregate — no more, no less. It is the most general and precise of the three grouping set extensions (ROLLUP, CUBE, GROUPING SETS). A .NET backend engineer building custom report layouts, partial aggregation summaries, or non-hierarchical multi-dimensional reports encounters GROUPING SETS when ROLLUP gives too many subtotals (it assumes hierarchy) or CUBE gives too many (it generates all 2^N). GROUPING SETS produces exactly the specified combinations, which makes result set size predictable and query performance controllable. The most common failure is not understanding that GROUPING SETS is implemented as a concatenation of separate aggregate operators — one per grouping set — which can share a common scan but may not be as efficient as ROLLUP's single-pass pipeline. Another misstep is using GROUPING SETS with redundant overlapping sets that produce duplicate rows. Interviewers use GROUPING SETS to distinguish candidates who only know ROLLUP and CUBE from those who understand the general grouping set framework and can design precise, efficient aggregation queries.

## Core Mental Model

GROUPING SETS specifies exactly which grouping combinations to include in the result. The syntax is `GROUP BY GROUPING SETS((col1, col2), (col3, col4), (col1), ())` — each parenthesized list is one grouping set. The result contains one row per group per specified set. This is the most general grouping set specification: ROLLUP is syntactic sugar for a specific set of hierarchical grouping sets, and CUBE is syntactic sugar for all 2^N combinations. GROUPING SETS is none of those — it is arbitrary. The execution plan for GROUPING SETS is typically a concatenation of separate aggregate operators, one per grouping set. Each aggregate reads from the same data source (shared scan via a Spool operator) and computes its GROUP BY independently. The results are concatenated (UNION ALL). The optimizer can share the scan across all grouping sets, avoiding the N-scan problem of manual UNION ALL, but cannot share the aggregation work — each set is aggregated independently. This makes GROUPING SETS less efficient per grouping set than ROLLUP (which uses a single-pass Split) but more efficient than UNION ALL (which scans N times).

### Classification

GROUPING SETS is a **GROUP BY subclause** that takes a list of grouping set specifications. It is the standard SQL-99 mechanism for specifying multiple aggregations in a single query. The SARGability of the underlying query is unaffected — the data access path is determined by the WHERE clause, same as any GROUP BY. The execution plan uses a Concatenation operator to merge the results of separate aggregate operators. The optimizer may use a shared scan (Table Spool) to read the base data once and feed all aggregates. The GROUPING() and GROUPING_ID() functions work the same as with ROLLUP and CUBE — they identify which grouping set a row belongs to. GROUPING SETS always produces exactly the specified result set size (no exponential explosion), which makes it the safest choice when the required grouping sets are known.

## Deep Mechanics

### How the Engine Executes This

1. **Parsing** — The parser recognizes the `GROUP BY GROUPING SETS(...)` specification. Each parenthesized list inside GROUPING SETS is a grouping set. Empty parentheses `()` represent the grand total set (no GROUP BY columns). The parser validates that each column reference is valid and that there are no duplicate grouping sets (SQL Server allows duplicates, but they produce duplicate rows — usually not intended).

2. **Binding (Algebrizer)** — The algebrizer creates a separate GroupBy expression for each grouping set. For GROUPING SETS((A,B), (C), ()), three GroupBy expressions are created: (A,B), (C), and (). The algebrizer also plans the GROUPING() and GROUPING_ID() function evaluations — GROUPING(A) returns 1 for the (C) and () sets, 0 for the (A,B) set.

3. **Optimization Strategy** — The optimizer evaluates:
   - **Shared scan (Table Spool):** Reads the base data once into a spool (Eager Spool). Each aggregate operator reads from the spool. This avoids scanning the data N times but uses tempdb for the spool storage.
   - **Independent scans:** Each aggregate scans the data independently. This is preferred when the spool overhead exceeds the scan cost (small tables, or very selective WHERE filters). The optimizer also considers whether the grouping sets can share sort orders (if (A,B) and (A) are both specified, sorting by A,B is sufficient for both).
   - **Split strategy (rare):** For specific cases where the grouping sets overlap heavily, the optimizer may use the same Segment/Split/Stream Aggregate pattern as ROLLUP, but this is uncommon for arbitrary GROUPING SETS.

4. **Plan Execution** — For each grouping set, a Stream Aggregate or Hash Aggregate operator computes the aggregation:
   - If the data is already sorted by the grouping set columns (from an index), Stream Aggregate is used — no sort needed.
   - If the data is not sorted, a Sort operator is added before the Stream Aggregate, or a Hash Aggregate is used.
   - For the grand total set `()`, the aggregate reads all rows and computes SUM/COUNT without a GROUP BY — this is the simplest aggregate operator.

5. **Concatenation** — A Concatenation operator merges the outputs of all aggregate operators. The concatenation is a logical UNION ALL — no deduplication. If two grouping sets produce duplicate rows (e.g., (A) and (A) both specified), the duplicates are preserved.

6. **Result Projection** — The final SELECT projects the columns with NULLs for columns not in the current grouping set. GROUPING() and GROUPING_ID() are evaluated.

### SQL Visibility

```sql
-- GROUPING SETS: custom aggregation groups
-- Report needs: totals by (Category, Region), by (Category), by (Region), and grand total
-- But NOT (Category, Region, Year) or any other combination — only these 4
SELECT
    CASE WHEN GROUPING(pc.CategoryId) = 1 THEN 'ALL CATEGORIES'
         ELSE pc.CategoryName END AS Category,
    CASE WHEN GROUPING(r.RegionId) = 1 THEN 'ALL REGIONS'
         ELSE r.RegionName END AS Region,
    SUM(oi.Quantity * oi.UnitPrice * (1 - oi.DiscountPct / 100)) AS Revenue,
    COUNT(DISTINCT o.OrderId) AS OrderCount,
    GROUPING_ID(pc.CategoryId, r.RegionId) AS GID
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
INNER JOIN dbo.ProductCategories AS pc ON p.CategoryId = pc.CategoryId
INNER JOIN dbo.Regions AS r ON o.RegionId = r.RegionId
WHERE o.OrderDate >= '2024-01-01'
  AND o.OrderDate < '2025-01-01'
GROUP BY GROUPING SETS(
    (pc.CategoryId, pc.CategoryName, r.RegionId, r.RegionName),
    (pc.CategoryId, pc.CategoryName),
    (r.RegionId, r.RegionName),
    ()
)
ORDER BY GID, Category, Region;

-- Equivalent UNION ALL (4 separate GROUP BY queries)
SELECT pc.CategoryName AS Category, r.RegionName AS Region,
       SUM(oi.Quantity * oi.UnitPrice * (1 - oi.DiscountPct / 100)) AS Revenue,
       COUNT(DISTINCT o.OrderId) AS OrderCount
FROM dbo.Orders AS o
... WHERE ...
GROUP BY pc.CategoryName, r.RegionName

UNION ALL

FROM dbo.Orders AS o
... WHERE ...
GROUP BY pc.CategoryName

UNION ALL

SELECT NULL, r.RegionName, SUM(...), COUNT(DISTINCT o.OrderId)
FROM dbo.Orders AS o
... WHERE ...
GROUP BY r.RegionName

UNION ALL

SELECT NULL, NULL, SUM(...), COUNT(DISTINCT o.OrderId)
FROM dbo.Orders AS o
... WHERE ...;
-- GROUPING SETS does this in one query with a shared scan

-- GROUPING SETS with non-overlapping sets
SELECT
    pc.CategoryName, NULL AS Region,
    NULL AS OrderYear, NULL AS OrderMonth,
    SUM(oi.Quantity) AS UnitsSold
FROM ... GROUP BY pc.CategoryName

UNION ALL  -- GROUPING SETS does this without the UNION ALL syntax

SELECT NULL, r.RegionName, NULL, NULL, SUM(oi.Quantity)
FROM ... GROUP BY r.RegionName

UNION ALL

SELECT NULL, NULL, YEAR(o.OrderDate), MONTH(o.OrderDate), SUM(oi.Quantity)
FROM ... GROUP BY YEAR(o.OrderDate), MONTH(o.OrderDate);

-- Using GROUPING SETS:
SELECT
    CASE WHEN GROUPING(pc.CategoryName) = 1 THEN 'ALL' ELSE pc.CategoryName END AS Category,
    CASE WHEN GROUPING(r.RegionName) = 1 THEN 'ALL' ELSE r.RegionName END AS Region,
    CASE WHEN GROUPING(YEAR(o.OrderDate)) = 1 THEN 'ALL' ELSE CAST(YEAR(o.OrderDate) AS VARCHAR(4)) END AS Year,
    CASE WHEN GROUPING(MONTH(o.OrderDate)) = 1 THEN 'ALL' ELSE CAST(MONTH(o.OrderDate) AS VARCHAR(2)) END AS Month,
    SUM(oi.Quantity) AS UnitsSold
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
INNER JOIN dbo.ProductCategories AS pc ON p.CategoryId = pc.CategoryId
INNER JOIN dbo.Regions AS r ON o.RegionId = r.RegionId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY GROUPING SETS(
    (pc.CategoryName),
    (r.RegionName),
    (YEAR(o.OrderDate), MONTH(o.OrderDate))
);
-- 3 non-overlapping grouping sets — each produces rows for different dimensions
-- Category rows have NULL for Region, Year, Month
-- Region rows have NULL for Category, Year, Month
-- Year+Month rows have NULL for Category, Region

-- GROUPING SETS with partial rollup (equivalent to ROLLUP but explicit)
-- ROLLUP(A, B, C) = GROUPING SETS((A,B,C), (A,B), (A), ())
-- Same thing, but GROUPING SETS makes the sets explicit:
GROUP BY GROUPING SETS(
    (CategoryId, RegionId, YEAR(OrderDate)),
    (CategoryId, RegionId),
    (CategoryId),
    ()
);

-- GROUPING SETS with partial cube (equivalent to CUBE but selective)
-- Instead of CUBE(A, B, C) = all 8 sets, pick only the useful ones:
GROUP BY GROUPING SETS(
    (CategoryId, RegionId, YEAR(OrderDate)),
    (CategoryId, RegionId),
    (CategoryId, YEAR(OrderDate)),
    (RegionId, YEAR(OrderDate)),
    (CategoryId),
    (RegionId),
    ()
);
-- 7 sets instead of 8 — excluded (YEAR) alone (not useful for this report)
```

### Execution Plan Analysis

**GROUPING SETS with 4 sets (shared scan strategy):**

```
  [Clustered Index Scan] Orders (filtered by date)
  Estimated Rows: 5,000,000
  Logical Reads: ~31,000
  
  [Nested Loops Join] × OrderItems (shared via Table Spool)
  
  [Table Spool (Eager Spool)]
  -- Spools the join result to tempdb
  -- All 4 aggregate branches read from this spool
  Estimated Rows: 15,000,000 (spooled)
  
  -- Branch 1: GROUP BY (CategoryId, RegionId)
  [Sort] BY CategoryId, RegionId → [Stream Aggregate] → [Compute Scalar]
  
  -- Branch 2: GROUP BY (CategoryId)
  [Sort] BY CategoryId → [Stream Aggregate] → [Compute Scalar]
  
  -- Branch 3: GROUP BY (RegionId)
  [Sort] BY RegionId → [Stream Aggregate] → [Compute Scalar]
  
  -- Branch 4: GROUP BY () — grand total
  [Stream Aggregate] (no GROUP BY — single group)
  
  [Concatenation]
  → [SELECT]
  Total Estimated Cost: ~95  |  Total Logical Reads: ~50,000 (31K spool write + 19K spool reads)
  Result rows: (C × R) + C + R + 1
```

**GROUPING SETS with non-overlapping sets (e.g., (Category), (Region), (Year,Month)):**

```
  [Clustered Index Scan] Orders (filtered by date)
  
  [Table Spool (Eager Spool)]
  
  -- Branch 1: GROUP BY Category — needs sort by Category
  [Sort] BY CategoryId → [Stream Aggregate]
  
  -- Branch 2: GROUP BY Region — needs sort by Region
  [Sort] BY RegionId → [Stream Aggregate]
  
  -- Branch 3: GROUP BY Year, Month — needs sort by Year, Month
  [Sort] BY YEAR(OrderDate), MONTH(OrderDate) → [Stream Aggregate]
  
  [Concatenation] → [SELECT]
  
  -- Note: 3 separate Sorts — each on a different column!
  -- This is more expensive than ROLLUP which sorts once
  -- For large tables, consider whether this is acceptable
```

**Without shared scan (independent scans for each set — rare, only for very selective queries):**

```
  -- Branch 1: scans independently
  [Clustered Index Seek] → [Nested Loops] → [Stream Aggregate]
  
  -- Branch 2: scans independently
  [Clustered Index Seek] → [Nested Loops] → [Stream Aggregate]
  
  -- Each branch does its own data access — preferred when
  -- the WHERE filter is very selective (< 1% of rows)
  -- because the spool write + read overhead exceeds scan cost
```

```sql
-- Table 'Worktable (Spool)'. Scan count 4, logical reads 76000 (1)
-- Table 'Regions'. Scan count 1, logical reads 450

-- Table 'OrderItems'. Scan count 1, logical reads 12450
```

(1) The Worktable logical reads represent the shared spool — 4 aggregate branches reading from the same spooled data.

**Improvement:** GROUPING SETS reduces logical reads by ~43% (103,200 → 101,800 base + spool) and CPU time by ~37% (7600ms → 4800ms). The savings come from the shared scan — the base data is scanned once and spooled, rather than scanned 4 times. The spool adds tempdb I/O but the total I/O is still lower than 4 separate scans.

### Failure Modes

**Redundant grouping sets (duplicate rows):** Specifying the same grouping set twice produces duplicate rows in the result. SQL Server does not deduplicate.

```sql
-- ❌ Duplicate: (CategoryId) appears twice
GROUP BY GROUPING SETS(
    (CategoryId, RegionId),
    (CategoryId),
    (CategoryId),  -- duplicate!
    ()
);
-- Result: two identical rows for each category subtotal
```

**Non-overlapping sets with very different sort orders:** Each grouping set requires its own Sort (unless a covering index matches its column order). Three non-overlapping sets on different columns = 3 separate Sort operators. On large tables, this can dominate the query cost.

```sql
-- ❌ Expensive: 3 different sort orders
GROUP BY GROUPING SETS(
    (CategoryId),         -- Sort by CategoryId
    (RegionId),           -- Sort by RegionId (different order!)
    (YEAR(OrderDate))     -- Sort by YEAR(OrderDate) (yet another order)
);
-- 3 Sort operators = 3× the sort memory and CPU
```

**Grand total set missing:** If the empty parentheses `()` are not included, the grand total row is not generated. This is often assumed to be included automatically — it is not.

```sql
-- No grand total:
GROUP BY GROUPING SETS((CategoryId), (RegionId));
-- Result: only category subtotals and region subtotals — no grand total

-- With grand total:
GROUP BY GROUPING SETS((CategoryId), (RegionId), ());
```

**GROUPING_ID() bit confusion with non-standard column ordering:** The GROUPING_ID() function uses bit positions based on the order of columns in the GROUP BY clause. If columns are listed in different orders in different grouping sets, the bit positions are confusing.

```sql
-- GROUPING_ID(RegionId, CategoryId) = bit0 for RegionId, bit1 for CategoryId
-- But grouping sets may list them in (CategoryId, RegionId) order — confusing!
SELECT
    GROUPING_ID(r.RegionId, pc.CategoryId) AS GID,
    ...
GROUP BY GROUPING SETS(
    (pc.CategoryId, r.RegionId),  -- (bit1, bit0) in GID
    (pc.CategoryId),               -- bit1 = 0, bit0 = 1 → GID = 1
    (r.RegionId),                  -- bit1 = 1, bit0 = 0 → GID = 2
    ()                             -- bit1 = 1, bit0 = 1 → GID = 3
);
```

**COUNT(DISTINCT) across grouping sets:** Same issue as ROLLUP and CUBE — the separate aggregates cannot share distinct state. Each grouping set computes its own COUNT(DISTINCT), which may double-count or produce incorrect results for overlapping sets.

```sql
    CASE WHEN GROUPING(YEAR(o.OrderDate)) = 1 THEN 'ALL YEARS'
         ELSE CAST(YEAR(o.OrderDate) AS VARCHAR(4)) END AS Year,
    CASE WHEN GROUPING(MONTH(o.OrderDate)) = 1 THEN 'ALL MONTHS'
         ELSE CAST(MONTH(o.OrderDate) AS VARCHAR(2)) END AS Month,
    SUM(oi.Quantity * oi.UnitPrice * (1 - oi.DiscountPct / 100)) AS Revenue,
    COUNT(DISTINCT o.OrderId) AS OrderCount,
    GROUPING_ID(d.DepartmentId, r.RegionId, YEAR(o.OrderDate), MONTH(o.OrderDate)) AS GID
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
INNER JOIN dbo.ProductCategories AS pc ON p.CategoryId = pc.CategoryId
INNER JOIN dbo.Departments AS d ON pc.DepartmentId = d.DepartmentId
INNER JOIN dbo.Regions AS r ON o.RegionId = r.RegionId
WHERE o.OrderDate >= '2024-01-01'
  AND o.OrderDate < '2025-01-01'
  AND o.Status IN ('Shipped', 'Delivered')
GROUP BY GROUPING SETS(
    (d.DepartmentId, d.DepartmentName),
    (r.RegionId, r.RegionName),
    (YEAR(o.OrderDate), MONTH(o.OrderDate)),
    (d.DepartmentId, d.DepartmentName, r.RegionId, r.RegionName),
    (d.DepartmentId, d.DepartmentName, YEAR(o.OrderDate), MONTH(o.OrderDate)),
    ()
)
ORDER BY GID, Department, Region, Year, Month;
-- 6 grouping sets: each dimension alone, department×region, department×year×month, grand total
-- Does NOT include: region×year×month alone, or any other combination not listed

-- ============================================================
-- Pattern 2: GROUPING SETS as precise replacement for partial CUBE
-- ============================================================
-- Instead of CUBE(Category, Region, Year) = 8 sets,
-- we only need 5 specific sets:
SELECT
    CASE WHEN GROUPING(pc.CategoryId) = 1 THEN 'ALL' ELSE pc.CategoryName END AS Category,
    CASE WHEN GROUPING(r.RegionId) = 1 THEN 'ALL' ELSE r.RegionName END AS Region,
    CASE WHEN GROUPING(YEAR(o.OrderDate)) = 1 THEN 'ALL' ELSE CAST(YEAR(o.OrderDate) AS VARCHAR(4)) END AS Year,
    SUM(oi.Quantity) AS UnitsSold
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
INNER JOIN dbo.ProductCategories AS pc ON p.CategoryId = pc.CategoryId
INNER JOIN dbo.Regions AS r ON o.RegionId = r.RegionId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY GROUPING SETS(
    (pc.CategoryId, pc.CategoryName, r.RegionId, r.RegionName, YEAR(o.OrderDate)),
    (pc.CategoryId, pc.CategoryName, r.RegionId, r.RegionName),
    (pc.CategoryId, pc.CategoryName, YEAR(o.OrderDate)),
    (r.RegionId, r.RegionName, YEAR(o.OrderDate)),
    ()
);
-- 5 sets instead of 8 — excluded single-dimension sets (Category alone, Region alone, Year alone)
-- because the report does not need them

-- ============================================================
-- Pattern 3: Non-overlapping grouping sets — most efficient case
-- Each set has completely different columns — no sort sharing possible
-- ============================================================
SELECT
    CASE WHEN GROUPING(d.DepartmentId) = 1 THEN 'ALL' ELSE d.DepartmentName END AS Department,
    NULL AS Region,
    NULL AS Month,
    SUM(oi.Quantity * oi.UnitPrice * (1 - oi.DiscountPct / 100)) AS Revenue,
    'By Department' AS Section
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
INNER JOIN dbo.ProductCategories AS pc ON p.CategoryId = pc.CategoryId
INNER JOIN dbo.Departments AS d ON pc.DepartmentId = d.DepartmentId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY d.DepartmentId, d.DepartmentName

UNION ALL

SELECT NULL, r.RegionName, NULL,
    SUM(oi.Quantity * oi.UnitPrice * (1 - oi.DiscountPct / 100)),
    'By Region'
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Regions AS r ON o.RegionId = r.RegionId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY r.RegionId, r.RegionName

UNION ALL

SELECT NULL, NULL, FORMAT(o.OrderDate, 'yyyy-MM'),
    SUM(oi.Quantity * oi.UnitPrice * (1 - oi.DiscountPct / 100)),
    'By Month'
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY FORMAT(o.OrderDate, 'yyyy-MM');
-- Note: This UNION ALL approach is actually MORE efficient here because
-- the grouping sets have completely different column sets and JOIN requirements.
-- Each branch has different JOINs and filters — GROUPING SETS cannot share the scan.
-- UNION ALL with separate tailored queries is better.

-- ============================================================
-- Pattern 4: GROUPING SETS with HAVING to filter specific grouping sets
-- ============================================================
SELECT
    CASE WHEN GROUPING(pc.CategoryId) = 1 THEN 'ALL' ELSE pc.CategoryName END AS Category,
    CASE WHEN GROUPING(r.RegionId) = 1 THEN 'ALL' ELSE r.RegionName END AS Region,
    SUM(oi.Quantity) AS UnitsSold,
    GROUPING_ID(pc.CategoryId, r.RegionId) AS GID
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
INNER JOIN dbo.ProductCategories AS pc ON p.CategoryId = pc.CategoryId
INNER JOIN dbo.Regions AS r ON o.RegionId = r.RegionId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY GROUPING SETS(
    (pc.CategoryId, pc.CategoryName, r.RegionId, r.RegionName),
    (pc.CategoryId, pc.CategoryName),
    (r.RegionId, r.RegionName),
    ()
)
HAVING GROUPING_ID(pc.CategoryId, r.RegionId) IN (0, 3)
    OR SUM(oi.Quantity) > 10000;
-- Only show detail rows (GID=0) and grand total (GID=3) regardless of quantity,
-- plus subtotal rows where units sold > 10,000
```

## Gotchas and Production Pitfalls

### Gotcha 1: Redundant Overlapping Grouping Sets

**Pitfall:** Specifying grouping sets where one is a subset of another, causing duplicate rows that look like data errors.

```sql
-- ❌ (CategoryId) is a subset of (CategoryId, RegionId) —
-- the category subtotal rows appear twice
SELECT pc.CategoryName, r.RegionName, SUM(oi.Quantity) AS UnitsSold
FROM ...
GROUP BY GROUPING SETS(
    (pc.CategoryId, pc.CategoryName, r.RegionId, r.RegionName),
    (pc.CategoryId, pc.CategoryName),  -- subset of first set!
    (r.RegionId, r.RegionName),
    ()
);
-- Category subtotal rows (CategoryName without RegionName) appear once from set 2.
-- But set 1 also produces rows with CategoryName + NULL for RegionName —
-- these are distinct rows (different NULL handling) and may appear confusing.
```

**Symptom:** The result set contains rows that look identical (from two different grouping sets that produce the same NULL pattern). The consumer counts rows and gets double the expected count for subtotals.

**Fix:** Remove overlapping sets or use GROUPING() to explicitly distinguish them:

```sql
-- ✅ Ensure each logical grouping set is specified only once
GROUP BY GROUPING SETS(
    (pc.CategoryId, pc.CategoryName, r.RegionId, r.RegionName),
    (r.RegionId, r.RegionName),
    (pc.CategoryId, pc.CategoryName),
    ()
);
-- These 4 sets are non-overlapping in terms of NULL pattern:
-- (Cat, Reg): all values present
-- (Cat): Region = NULL, Category = value
-- (Reg): Category = NULL, Region = value
-- (): both NULL
```

**Cost of not fixing:** Report shows 2× the expected rows for subtotals. Consumer assumes the data is corrupt. Engineer spends hours debugging before discovering the duplicate grouping set.

### Gotcha 2: No Grand Total — Missing `()`

**Pitfall:** The developer forgets to include the empty grouping set `()` for the grand total.

```sql
-- ❌ No grand total
GROUP BY GROUPING SETS(
    (CategoryId, RegionId),
    (CategoryId),
    (RegionId)
);
-- Result: detail + category subtotal + region subtotal — but NO grand total

-- ✅ With grand total
GROUP BY GROUPING SETS(
    (CategoryId, RegionId),
    (CategoryId),
    (RegionId),
    ()  -- grand total
);
```

**Symptom:** The report is missing the overall total. Users check the sum of all subtotals manually.

**Fix:** Always include `()` in the grouping sets list when a grand total is needed.

**Cost of not fixing:** Incomplete report. Users manually add up rows and may make arithmetic errors.

### Gotcha 3: GROUPING SETS with Overlapping Non-SARGable Expressions

**Pitfall:** Using non-SARGable expressions (function-wrapped columns) in multiple grouping sets, each requiring its own computation.

```sql
-- ❌ YEAR() and MONTH() are computed twice — once per grouping set
GROUP BY GROUPING SETS(
    (YEAR(OrderDate)),      -- computes YEAR for this set
    (MONTH(OrderDate)),     -- computes MONTH for this set
    ()                       -- no computation needed
);
-- The YEAR and MONTH functions are evaluated for each row in each grouping set
```

**Symptom:** The execution plan shows Compute Scalar operators for each function in each aggregate branch. CPU time is higher than necessary.

**Fix:** Pre-compute the expressions in a derived table or CTE, or use persisted computed columns:

```sql
-- ✅ Pre-compute in derived table
WITH OrderBase AS (
    SELECT o.OrderId, o.OrderDate, oi.Quantity, oi.UnitPrice,
           YEAR(o.OrderDate) AS OrderYear,
           MONTH(o.OrderDate) AS OrderMonth
    FROM dbo.Orders AS o
    INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
    WHERE o.OrderDate >= '2024-01-01'
)
SELECT
    CASE WHEN GROUPING(OrderYear) = 1 THEN 'ALL' ELSE CAST(OrderYear AS VARCHAR(4)) END AS Year,
    CASE WHEN GROUPING(OrderMonth) = 1 THEN 'ALL' ELSE CAST(OrderMonth AS VARCHAR(2)) END AS Month,
    SUM(Quantity * UnitPrice) AS Revenue
FROM OrderBase
GROUP BY GROUPING SETS(
    (OrderYear),
    (OrderMonth),
    ()
);
```

**Cost of not fixing:** 15% extra CPU from redundant expression evaluation. On a 50M-row table, this adds 2-3 seconds to the query.

### Gotcha 4: Shared Spool Memory and Tempdb Pressure

**Pitfall:** Large GROUPING SETS with many sets cause the shared Table Spool to write massive amounts of data to tempdb.

**Symptom:** The execution plan shows an Eager Spool that writes the entire joined result to tempdb. On a 10M-row query with wide rows (200+ bytes), the spool writes 2GB+ to tempdb. If the GROUPING SETS has 5+ sets, the spool is read 5+ times, generating 10GB+ of tempdb I/O. This causes tempdb contention, log growth, and slow query performance.

**Fix:**

1. **Reduce the number of grouping sets** — fewer sets = fewer spool reads.
2. **Add indexes to eliminate Sort** — if each branch can read ordered data from an index, the sort is eliminated and the plan may avoid the spool.
3. **Use columnstore index** — columnstore compression reduces the spool data size by 5-10×.
4. **Consider splitting into separate queries** — if the grouping sets have different JOIN requirements, separate queries may be more efficient.

```sql
-- If the spool is a problem, consider separate queries.
-- But measure first — the spool may be faster than separate scans!
```

**Cost of not fixing:** Tempdb runs out of space. Other queries on the instance fail with "Could not allocate space for object in database 'tempdb'."

### Gotcha 5: ORDER BY Without GROUPING() in GROUPING SETS

**Pitfall:** Same issue as ROLLUP and CUBE — ORDER BY on raw columns places subtotal rows in unexpected positions.

```sql
-- ❌ ORDER BY on raw columns — NULLs sort first
ORDER BY CategoryId, RegionId;
-- Grand total (NULL, NULL) appears first
-- Region subtotals (NULL, Region) appear before detail rows for that region

-- ✅ Correct: ORDER BY with GROUPING()
ORDER BY
    CASE WHEN GROUPING(CategoryId) = 1 THEN 1 ELSE 0 END,
    CategoryId,
    CASE WHEN GROUPING(RegionId) = 1 THEN 1 ELSE 0 END,
    RegionId;
```

### Gotcha 6: GROUPING SETS Cannot Share Aggregation Work Across Sets

**Pitfall:** The developer assumes GROUPING SETS is as efficient as ROLLUP because both are "grouping set extensions."

**Symptom:** A GROUPING SETS query with 4 sets on a 10M-row table takes 8 seconds, while an equivalent ROLLUP with 4 sets takes 4 seconds. The difference is that ROLLUP uses a single-pass Split/Stream Aggregate, while GROUPING SETS uses separate aggregates (Concatenation) with a shared spool. The spool write + read adds overhead.

**Fix:** If the grouping sets happen to be hierarchical (each set contains the previous), use ROLLUP instead:

```sql
-- ❌ GROUPING SETS with hierarchical sets
GROUP BY GROUPING SETS(
    (CategoryId, RegionId, Year),
    (CategoryId, RegionId),
    (CategoryId),
    ()
);

-- ✅ Better: same result, ROLLUP is more efficient
GROUP BY ROLLUP(CategoryId, RegionId, Year);
```

**Cost of not fixing:** 2× slower queries. Acceptable for small tables, but at scale the extra tempdb I/O from the spool adds up.

|Bulk insert 10K rows|~40K logical writes|~100K logical writes|+150%|

The higher write amplification (compared to ROLLUP indexes) is because GROUPING SETS may benefit from multiple indexes (one for each grouping set sort order), while ROLLUP benefits from a single ordered index.

### Spoken Answers

**Q: What is GROUPING SETS and what problem does it solve that ROLLUP and CUBE do not?**

> **Average answer:** GROUPING SETS lets you specify which subtotals you want, while ROLLUP gives hierarchical subtotals and CUBE gives all combinations.

> **Great answer:** GROUPING SETS is the most general grouping set specification in SQL. ROLLUP and CUBE are both syntactic sugar that expand to specific GROUPING SETS expressions. ROLLUP expands to N+1 hierarchical sets; CUBE expands to all 2^N combinations. GROUPING SETS lets you specify exactly which grouping combinations you need — nothing more, nothing less. The problem it solves is the inflexibility of ROLLUP and CUBE. ROLLUP assumes a hierarchy that may not exist in your data. CUBE generates all combinations, which is wasteful when you only need a subset. For example, if a report needs totals by (Category, Region), by (Category), and by (Region), but NOT by (Category, Region, Month) or any other combination, GROUPING SETS produces exactly those 3 sets plus the grand total. ROLLUP cannot do this because it assumes Category → Region hierarchy. CUBE would produce 4 sets for 2 columns (which works here) but if you need 5 out of 8 possible combinations for 3 columns, CUBE generates all 8 and wastes 3. GROUPING SETS with 5 explicit sets is the precise solution.

**Q: How does the execution plan for GROUPING SETS differ from ROLLUP?**

> **Average answer:** GROUPING SETS uses separate aggregates and a concatenation. ROLLUP uses a single pass with a Split operator.

> **Great answer:** The fundamental difference is that ROLLUP uses a single-pass pipeline with Segment, Split, and Stream Aggregate operators, while GROUPING SETS uses a Concatenation operator that merges the outputs of separate aggregate operators — one per grouping set. ROLLUP's Split operator duplicates each row N+1 times and the Stream Aggregate computes all sets in one pass. GROUPING SETS cannot share aggregation work across sets because each set has potentially different columns. Instead, the optimizer typically uses a shared scan via an Eager Table Spool: the base data is scanned once, written to a tempdb spool, and then each aggregate branch reads from the spool independently. This avoids scanning the data N times but adds tempdb I/O for the spool write and reads. The tradeoff is: ROLLUP is faster (single pass, no spool) but restricted to hierarchical sets; GROUPING SETS is more flexible but adds spool overhead. For 4-5 hierarchical sets, ROLLUP is about 2× faster. For non-hierarchical sets, GROUPING SETS is the only option (short of UNION ALL). The plan choice also depends on whether a covering index can eliminate Sort operators in each branch.

**Q: When is GROUPING SETS more efficient than UNION ALL?**

> **Average answer:** GROUPING SETS is faster because it scans the data once.

> **Great answer:** GROUPING SETS is more efficient than UNION ALL when the data access (FROM/JOIN/WHERE) is shared across all grouping sets — that is, when every grouping set needs the same base rows. In that case, the shared Table Spool writes the joined/filtered result to tempdb once and each aggregate branch reads from the spool. UNION ALL would scan the base tables N times (once per branch), doing N times the I/O. GROUPING SETS with a shared spool does 1 write + N reads of the spooled data, which is typically less total I/O than N full scans. However, GROUPING SETS is LESS efficient than UNION ALL when: (a) the grouping sets require different JOINs or WHERE filters — the spool cannot be shared because each branch accesses different columns; (b) the grouping sets have non-overlapping high-cardinality columns requiring different Sort orders — each branch sorts independently and UNION ALL's separate scans would allow tailored execution plans; (c) the WHERE filter is very selective (< 1%) — the spool write overhead exceeds the scan savings. In practice, GROUPING SETS is a net win when you have 3+ grouping sets sharing the same base query, the base query processes at least 1M rows, and the grouping sets have some column overlap. For 2 non-overlapping sets on a small table, UNION ALL is fine.

- **Relevant when table exceeds ~500K rows** — Below this, UNION ALL overhead is negligible.

### Query Challenges

**Challenge 1 — Write the SQL**

You are building a multi-section inventory report. The report needs four sections:
1. Total units sold by Category (all regions, all months combined)
2. Total units sold by Region (all categories, all months combined)
3. Total units sold by Month (all categories, all regions combined)
4. Grand total (all categories, all regions, all months combined)

Write a single GROUPING SETS query that produces all four sections. Use human-readable labels. The query should cover 2024 data with status Shipped or Delivered.

<details>
<summary>Solution</summary>

```sql
SELECT
    CASE WHEN GROUPING(pc.CategoryId) = 1 THEN 'ALL CATEGORIES'
         ELSE pc.CategoryName END AS Category,
    CASE WHEN GROUPING(r.RegionId) = 1 THEN 'ALL REGIONS'
         ELSE r.RegionName END AS Region,
    CASE WHEN GROUPING(YEAR(o.OrderDate)) = 1 THEN 'ALL YEARS'
         ELSE CAST(YEAR(o.OrderDate) AS VARCHAR(4)) END AS Year,
    CASE WHEN GROUPING(MONTH(o.OrderDate)) = 1 THEN 'ALL MONTHS'
         ELSE CAST(MONTH(o.OrderDate) AS VARCHAR(2)) END AS Month,
    SUM(oi.Quantity) AS UnitsSold,
    COUNT(DISTINCT o.OrderId) AS OrderCount,
    GROUPING_ID(pc.CategoryId, r.RegionId, YEAR(o.OrderDate), MONTH(o.OrderDate)) AS GID
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
INNER JOIN dbo.ProductCategories AS pc ON p.CategoryId = pc.CategoryId
INNER JOIN dbo.Regions AS r ON o.RegionId = r.RegionId
WHERE o.OrderDate >= '2024-01-01'
  AND o.OrderDate < '2025-01-01'
  AND o.Status IN ('Shipped', 'Delivered')
GROUP BY GROUPING SETS(
    (pc.CategoryId, pc.CategoryName),                         -- Section 1: by Category
    (r.RegionId, r.RegionName),                               -- Section 2: by Region
    (YEAR(o.OrderDate), MONTH(o.OrderDate)),                  -- Section 3: by Month
    ()                                                         -- Section 4: Grand total
)
ORDER BY GID, Category, Region, Year, Month;
```

**Logical reads:** ~30,000 (shared spool + 3 aggregate branches). **Execution plan:** [Index Seek] → [Nested Loops] × 3 → [Table Spool] → 3 branches (Sort + Stream Aggregate each) + 1 grand total branch → [Concatenation] → [SELECT]. **EF Core equivalent:** Raw SQL via `FromSqlRaw`.

</details>

**Challenge 2 — Fix the performance problem**

```sql
-- This GROUPING SETS query runs in 55 seconds on a 100M row Orders table.
-- The execution plan shows 6 separate Sort operators, each spilling to tempdb.
-- SET STATISTICS IO: logical reads = 850,000
SELECT
    CASE WHEN GROUPING(pc.CategoryId) = 1 THEN 'ALL' ELSE pc.CategoryName END AS Category,
    CASE WHEN GROUPING(r.RegionId) = 1 THEN 'ALL' ELSE r.RegionName END AS Region,
    CASE WHEN GROUPING(s.SalesOfficeId) = 1 THEN 'ALL' ELSE s.OfficeName END AS Office,
    SUM(oi.Quantity) AS UnitsSold
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
INNER JOIN dbo.ProductCategories AS pc ON p.CategoryId = pc.CategoryId
INNER JOIN dbo.Regions AS r ON o.RegionId = r.RegionId
INNER JOIN dbo.SalesOffices AS s ON r.SalesOfficeId = s.SalesOfficeId
WHERE o.OrderDate >= '2024-01-01' AND o.OrderDate < '2025-01-01'
GROUP BY GROUPING SETS(
    (pc.CategoryId, pc.CategoryName, r.RegionId, r.RegionName, s.SalesOfficeId, s.OfficeName),
    (pc.CategoryId, pc.CategoryName),
    (r.RegionId, r.RegionName),
    (s.SalesOfficeId, s.OfficeName),
    ()
);
```

<details>
<summary>Solution</summary>

**Root cause:** 5 grouping sets, each requiring a different Sort order:
- Set 1: Sort by CategoryId, RegionId, SalesOfficeId
- Set 2: Sort by CategoryId
- Set 3: Sort by RegionId
- Set 4: Sort by SalesOfficeId
- Set 5: no sort (grand total)

Each Sort requires a full pass over the 100M-row spooled data. 4 Sorts on 100M rows each = 400M rows sorted total. Each Sort spills to tempdb because the memory grant is insufficient for 100M-row sorts. The total tempdb I/O is enormous.

**Fixed approach: Split into separate targeted queries (UNION ALL) with indexes tailored to each:**

```sql
-- Query 1: by Category (with covering index on CategoryId)
-- Efficient Index Scan + Stream Aggregate — no sort needed
SELECT pc.CategoryName, NULL AS Region, NULL AS Office,
       SUM(oi.Quantity) AS UnitsSold
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
INNER JOIN dbo.ProductCategories AS pc ON p.CategoryId = pc.CategoryId
WHERE o.OrderDate >= '2024-01-01' AND o.OrderDate < '2025-01-01'
GROUP BY pc.CategoryId, pc.CategoryName

UNION ALL

-- Query 2: by Region (with covering index on RegionId)
SELECT NULL, r.RegionName, NULL, SUM(oi.Quantity)
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Regions AS r ON o.RegionId = r.RegionId
WHERE o.OrderDate >= '2024-01-01' AND o.OrderDate < '2025-01-01'
GROUP BY r.RegionId, r.RegionName

UNION ALL

-- Query 3: by Office (with covering index on SalesOfficeId)
SELECT NULL, NULL, s.OfficeName, SUM(oi.Quantity)
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Regions AS r ON o.RegionId = r.RegionId
INNER JOIN dbo.SalesOffices AS s ON r.SalesOfficeId = s.SalesOfficeId
WHERE o.OrderDate >= '2024-01-01' AND o.OrderDate < '2025-01-01'
GROUP BY s.SalesOfficeId, s.OfficeName

UNION ALL

-- Query 4: Grand total
SELECT NULL, NULL, NULL, SUM(oi.Quantity)
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
WHERE o.OrderDate >= '2024-01-01' AND o.OrderDate < '2025-01-01';

-- Note: UNION ALL with separate queries is faster here because:
-- 1. Each branch can use a covering index (no sort needed)
-- 2. Each branch can skip unnecessary JOINs (Office query joins Office, Category query does not)
-- 3. No shared spool overhead
-- 4. Each branch's Sort is eliminated by the covering index
```

**Indexes to create:**

```sql
CREATE INDEX IX_OrderItems_Category ON dbo.OrderItems (OrderId, ProductId)
    INCLUDE (Quantity);
CREATE INDEX IX_Orders_OrderDate ON dbo.Orders (OrderDate)
    INCLUDE (OrderId, RegionId);
```

**After fix — logical reads:** ~120,000 (from 850,000) — **7× reduction**. Each branch uses a tailored Index Scan with Stream Aggregate. No Sort operators. Elapsed time drops from 55 seconds to ~8 seconds.

**Lesson:** GROUPING SETS is not always the answer. When grouping sets have completely different column sets and require different JOINs, separate UNION ALL queries with tailored indexes are more efficient.

</details>

**Challenge 3 — Explain the execution plan**

```sql
-- Query A:
SELECT CategoryId, RegionId, SUM(Amount) AS Total
FROM Sales
WHERE SaleDate >= '2024-01-01'
GROUP BY GROUPING SETS(
    (CategoryId, RegionId),
    (CategoryId),
    (),
    (RegionId)
);

-- Query B:
SELECT CategoryId, RegionId, SUM(Amount) AS Total
FROM Sales
WHERE SaleDate >= '2024-01-01'
GROUP BY GROUPING SETS(
    (CategoryId, RegionId),
    (CategoryId),
    (RegionId),
    ()
);
```

These two queries produce the same result set but Query A lists the sets in a different order than Query B. Does the execution plan differ? Why or why not?

<details>
<summary>Solution</summary>

The execution plans are **identical**. The order of grouping sets in the GROUPING SETS list does not affect the execution plan — the optimizer reorders the sets internally based on estimated cardinality and sort order requirements. The optimizer evaluates each grouping set's column set and determines the cheapest way to compute each, then concatenates them in plan order (not query order).

However, the **output order** of the result set without an explicit ORDER BY may differ between Query A and Query B. The Concatenation operator outputs rows in the order the branches are evaluated in the plan, which may or may not match the query text order. To guarantee a consistent result order, always use ORDER BY with GROUPING():

```sql
ORDER BY
    CASE WHEN GROUPING(CategoryId) = 1 THEN 1 ELSE 0 END,
    CategoryId,
    CASE WHEN GROUPING(RegionId) = 1 THEN 1 ELSE 0 END,
    RegionId;
```

The order of sets in the GROUPING SETS specification is a **cosmetic concern only** — it does not affect query performance or result shape (with ORDER BY).

**The optimizer's reordering logic:** It evaluates the estimated cost of each grouping set (how many distinct groups, whether a sort is needed, whether an index exists) and orders the branches in the Concatenation operator to minimize total cost. Sets with lower cardinality are typically evaluated first (smaller sort memory grant), while the grand total (no group by) is evaluated last.

</details>

**Challenge 4 — Diagnose the concurrency problem**

A GROUPING SETS reporting query runs every hour and completes in 10 seconds. After a data volume increase (from 10M to 30M rows), it now takes 2 minutes and frequently causes "tempdb is full" errors. The execution plan shows a Table Spool (Eager Spool) that writes 3GB of data, read by 6 aggregate branches. What is the problem and how do you fix it?

<details>
<summary>Solution</summary>

**Root cause:** The shared Table Spool writes 3GB of the joined result to tempdb. Each of the 6 aggregate branches reads from this spool, generating 6 × 3GB = 18GB of total tempdb I/O per execution. At 30M rows with wide rows (multiple JOINs), the spool size exceeds tempdb capacity. The spool is written once and read 6 times, consuming tempdb log space for the spool write and data space for the spool pages.

**Detection:**

```sql
-- Check tempdb usage per session
SELECT
    session_id,
    user_objects_alloc_page_count * 8 / 1024 AS tempdb_alloc_MB,
    user_objects_dealloc_page_count * 8 / 1024 AS tempdb_dealloc_MB
FROM sys.dm_db_session_space_usage
WHERE session_id = @@SPID;

-- Check spool size in execution plan
-- Look for Table Spool operator: Estimated Spool Size = 3GB
```

**Fix:**

Option 1: **Reduce spool size** by filtering more aggressively in WHERE (narrower date range, additional filters).

Option 2: **Use columnstore index** to compress the spool data. Columnstore compression reduces the spool size by 5-10× → 300-600MB instead of 3GB.

```sql
CREATE NONCLUSTERED COLUMNSTORE INDEX IX_Sales_Reporting_CS
    ON dbo.Sales (SaleDate, CategoryId, RegionId)
    INCLUDE (Amount);
```

Option 3: **Reduce the number of grouping sets** — fewer branches = fewer spool reads. If the report only needs 3 sets instead of 6, the spool is read 3 times instead of 6.

Option 4: **Split into separate queries** — 6 separate targeted queries, each scanning only the columns it needs, with tailored indexes. This avoids the shared spool entirely.

```sql
-- Split into 6 targeted GROUP BY queries with UNION ALL
-- Each branch uses a covering index for its specific grouping set
-- No shared spool — each branch does its own Index Scan
-- Total I/O: 6 × selective scans vs 1 × full scan + 6 × spool reads
```

Option 5: **Increase tempdb size** and add tempdb files. If the data volume is expected to grow further, tempdb must be scaled accordingly:

```sql
ALTER DATABASE tempdb MODIFY FILE (NAME = tempdev, SIZE = 20GB);
ALTER DATABASE tempdb ADD FILE (NAME = tempdev2, FILENAME = '...', SIZE = 20GB);
```

**Recommended fix:** Use option 4 (separate queries with covering indexes). This avoids the spool entirely and each branch can be optimized independently. Expected elapsed time: ~20 seconds (vs 2 minutes with spool). Additional tempdb space required: 0 (no spool).

</details>

**Challenge 5 — Design the index**

**Scenario:** A reporting system runs a GROUPING SETS query every 15 minutes on a `Sales` table with 80M rows. The grouping sets are:
- (StoreId, DateId) — daily store sales
- (ProductId, DateId) — daily product sales
- (CustomerSegmentId, DateId) — daily segment sales
- (DateId) — daily total

The query filters on `DateId >= 20240101 AND DateId < 20240132` (January 2024, ~2M rows). The query must complete within 5 seconds. Currently it takes 30 seconds with a shared spool and 4 Sort operators. Design the optimal index strategy.

<details>
<summary>Solution</summary>

```sql
-- Key insight: All 4 grouping sets include DateId as the first column.
-- An index on (DateId) can be shared across all branches.
-- Additionally, each branch needs a specific secondary column for its GROUP BY.

-- Primary covering index: DateId first (for the WHERE filter and first sort column)
-- then include all possible secondary columns
CREATE NONCLUSTERED INDEX IX_Sales_GroupingSets_Covering
    ON dbo.Sales (DateId, StoreId, ProductId, CustomerSegmentId)
    INCLUDE (Amount, Quantity);
-- This index supports:
-- - WHERE DateId BETWEEN 20240101 AND 20240131 → Index Seek on DateId
-- - Grouping set (StoreId, DateId): ordered by DateId, then StoreId (partial match)
-- - Grouping set (ProductId, DateId): ordered by DateId, then ProductId (partial match)
-- - Grouping set (CustomerSegmentId, DateId): ordered by DateId, then CustomerSegmentId (partial match)
-- - Grouping set (DateId): ordered by DateId (full match)
-- INCLUDE covers aggregate columns without key lookups

-- Alternative: Multiple narrower indexes for each grouping set
-- (These are more targeted and may eliminate Sort more completely)
CREATE INDEX IX_Sales_StoreDaily ON dbo.Sales (DateId, StoreId)
    INCLUDE (Amount, Quantity);
CREATE INDEX IX_Sales_ProductDaily ON dbo.Sales (DateId, ProductId)
    INCLUDE (Amount, Quantity);
CREATE INDEX IX_Sales_SegmentDaily ON dbo.Sales (DateId, CustomerSegmentId)
    INCLUDE (Amount, Quantity);
CREATE INDEX IX_Sales_Daily ON dbo.Sales (DateId)
    INCLUDE (Amount, Quantity);
-- Each index perfectly matches one grouping set — eliminates all Sort operators.
-- The shared spool is not needed because each branch reads from its own index.

-- Alternative: Columnstore index (best for large fact tables)
CREATE NONCLUSTERED COLUMNSTORE INDEX IX_Sales_CS
    ON dbo.Sales (DateId, StoreId, ProductId, CustomerSegmentId)
    INCLUDE (Amount, Quantity);
-- Columnstore eliminates Sort operators entirely (columnstore segments are
-- scanned and aggregated in batch mode).
-- The DateId filter enables segment elimination.
```

**Index selection:**

|Strategy|Sort Eliminated|Spool Avoided|Query Time|Storage|Write Overhead|
|---|---|---|---|---|---|
|Single covering index|Partial (residual sort for non-leading columns)|Yes|~8s|~4GB|+80%|
|4 targeted indexes|Yes (each branch has perfect index)|Yes (no spool — each branch scans)|~5s|~8GB (4 × 2GB)|+160%|
|Columnstore|Yes (batch mode)|Yes|~2s|~1.5GB (compressed)|+40% (delta store)|

**Recommendation:** Use the **columnstore index** for the best performance, lowest storage, and simplest maintenance. The columnstore provides batch mode execution, segment elimination for the DateId filter, and eliminates Sort/Spool entirely. Expected logical reads: ~5,000 (compressed columnstore segments) vs ~50,000 (rowstore covering index). Elapsed time: ~2 seconds (meets the 5-second SLA with 2.5× headroom).
