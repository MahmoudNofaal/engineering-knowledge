### Where This Fits

SUM, AVG, MIN, and MAX are the core arithmetic aggregate functions that power every financial report, dashboard metric, and analytical query. Every .NET backend engineer uses these daily: SUM for order totals and revenue calculations, AVG for average order value and customer lifetime metrics, MIN for earliest dates and lowest prices, MAX for latest dates and highest quantities. The most expensive mistakes made here are: SUM integer overflow (summing INT columns without casting to BIGINT/DECIMAL — arithmetic overflow at 2.1B), AVG integer division (AVG of INT column returns INT, truncating the fractional part), and MIN/MAX string comparison behaving unexpectedly due to collation settings. Interviewers use these aggregate functions to test whether candidates understand data type precedence, implicit conversion, and execution plan implications. Engineers who know these functions deeply can predict when SUM will overflow based on data growth rates, know to cast before averaging, and understand that MIN/MAX on indexed columns can use a TOP 1 seek — the fastest possible read operation.

## Core Mental Model

SUM adds all non-NULL values in a group and returns the total. AVG divides SUM by COUNT of non-NULL values — it is logically `SUM(col) / COUNT(col)`. AVG does NOT divide by total rows; it divides by non-NULL rows only. MIN returns the smallest value in a group based on the data type's sort order — for numbers it is the lowest numeric value, for dates it is the earliest date, for strings it depends on collation (lexicographic order with case sensitivity based on the column's collation). MAX returns the largest value similarly. All four functions ignore NULLs entirely — NULL is not included in the sum, not counted in the average divisor, and not considered in min/max comparisons. The result is NULL if and only if all input values are NULL (or there are no rows). The engine implements these via Stream Aggregate (sorted input — efficient single pass) or Hash Aggregate (unsorted — builds hash table). For MIN and MAX, the optimizer can use index order to avoid scanning the entire table — a MIN on an indexed column can be satisfied by reading the first row from a non-clustered index (TOP 1 seek), making it O(log N) instead of O(N).

### Classification

SUM and AVG are **arithmetic aggregate functions** that require numeric data types. MIN and MAX are **comparison aggregate functions** that work on any data type with a defined sort order. All are **set functions** in the SELECT or HAVING clause. None are SARGable directly — they operate on filtered row sets. However, MIN/MAX on indexed columns are SARGable via an implicit TOP 1 seek: the optimizer can rewrite `SELECT MAX(OrderDate) FROM Orders` as a single Index Seek that reads the last row in the index. This is the most dramatic performance optimization for any aggregate function.

## Deep Mechanics

### How the Engine Executes This

**SUM execution:**

1. The storage engine reads rows from the access method (Clustered Index Scan, Index Seek, etc.). For each row, the SUM column value is loaded into the CPU cache.
2. The NULL check is performed — if the value is NULL, it is skipped. If the column is NOT NULL, this check is eliminated at compile time.
3. The value is added to an accumulator. For INT columns, the accumulator is an INT. For DECIMAL(10,2), the accumulator follows SQL Server's precise decimal arithmetic rules: the result precision is the input precision plus a scale-dependent amount.
4. **Overflow behavior:** For INT SUM, SQL Server uses a 32-bit accumulator. If the running sum exceeds 2,147,483,647 (for positive) or goes below -2,147,483,648 (for negative), an arithmetic overflow error 8115 is raised at runtime. For DECIMAL, the result precision expands up to the maximum (38). For BIGINT, the accumulator is 64-bit.
5. The result is returned. For grouped SUM, the accumulator is reset per group.

**AVG execution:**

1. AVG is implemented as two parallel accumulators: SUM (running total) and COUNT (number of non-NULL values).
2. After all rows are processed, the engine computes `SUM / COUNT`. The division follows the data type's division rules.
3. **Integer division trap:** If both SUM and COUNT are INT, SQL Server performs integer division. `AVG of (3, 4, 5) = (3+4+5)/3 = 12/3 = 4`. But `AVG of (3, 4, 6) = 13/3 = 4` (truncated from 4.333...). The fractional part is lost silently.
4. To get fractional results with INT input, the engine must cast one operand to DECIMAL: `AVG(CAST(col AS DECIMAL(10,2)))`.
5. For DECIMAL input, AVG returns DECIMAL(38, max(s, 4) + 4) in SQL Server — it preserves fractional precision.

**MIN execution:**

1. Without index: the storage engine scans all rows. A comparison register holds the current minimum. For each row, if the value is non-NULL and less than the current minimum, the register is updated.
2. With index: the optimizer can rewrite `SELECT MIN(OrderDate) FROM Orders` as a single Index Seek that reads the first (lowest) row from a non-clustered index on OrderDate. This is the Index Seek (TOP 1) optimization. The plan shows: `[Index Seek] → [Stream Aggregate] → [SELECT]`. The Stream Aggregate is a no-op because only one row enters it.
3. For grouped MIN, the index must be sorted by both the GROUP BY column and the MIN column to avoid scanning: `INDEX ON (GroupCol, MinCol)`.

**MAX execution:**

1. Mirror of MIN. Without index: scan all rows. With index: seek to the last row in the index (reading one leaf page row from the rightmost page).
2. `SELECT MAX(OrderDate) FROM Orders` with an index on OrderDate requires 3-4 logical reads regardless of table size (root page, intermediate page, leaf page). Without the index, it requires a full Clustered Index Scan (tens of thousands of reads).

    c.CustomerId,
    c.FirstName + ' ' + c.LastName AS CustomerName,

-- If sum of all quantities > 2,147,483,647, overflow error 8115

SELECT SUM(CAST(Quantity AS DECIMAL(18,0))) FROM dbo.OrderItems;

-- ❌ Integer division: returns INT

-- If TotalAmount is DECIMAL(10,2), this is fine (DECIMAL division)

SELECT AVG(Quantity) FROM dbo.OrderItems;
-- Returns INT — truncates fractional part

-- ✅ Cast to get fractional average
SELECT AVG(CAST(Quantity AS DECIMAL(10,2))) FROM dbo.OrderItems;

    MIN(LastName) AS FirstAlphabetically,
    MAX(LastName) AS LastAlphabetically

-- 'Adams' < 'smith' depending on collation (case-sensitive vs case-insensitive)

-- With index on OrderDate, these are O(log N):
SELECT MIN(OrderDate) FROM dbo.Orders;
SELECT MAX(OrderDate) FROM dbo.Orders;

SELECT SUM(DISTINCT TotalAmount) FROM dbo.Orders;
-- Sums each unique TotalAmount once — unusual but valid

-- Conditional aggregation — SUM/CASE pattern

    SUM(CASE WHEN Status = 1 THEN 1 ELSE 0 END) AS PendingCount,
    SUM(CASE WHEN Status = 3 THEN TotalAmount ELSE 0 END) AS ShippedRevenue
FROM dbo.Orders
WHERE OrderDate >= '2024-01-01';

-- AVG with NULL handling — demonstrate NULL exclusion
SELECT
    AVG(ShipDate) AS AvgShipDate, -- meaningless but valid
    AVG(TotalAmount) AS AvgOrderValue,
    AVG(NULLIF(TotalAmount, 0)) AS AvgNonZeroOrderValue
FROM dbo.Orders;
-- AVG ignores NULLs from NULLIF too

-- MIN/MAX with GROUP BY — per customer date range

    MIN(o.OrderDate) AS FirstOrder,
    MAX(o.OrderDate) AS LastOrder,

### Execution Plan Analysis

For `SELECT MIN(OrderDate), MAX(OrderDate) FROM dbo.Orders` with an index on OrderDate:

```
Expected plan shape:
[Index Seek (MIN: first row)] → [Index Seek (MAX: last row)] → [Stream Aggregate (no-op)] → [SELECT]
Estimated Cost: Index Seeks 50% each | Logical Reads: 3-4 per seek
```

- The plan shows TWO Index Seek operations on the same non-clustered index. One seeks to the first page (lowest value), one seeks to the last page (highest value). Each seek reads ~3-4 pages (root, intermediate, leaf). Total: 6-8 logical reads regardless of table size.

Without the index:

```
Expected plan shape:
[Clustered Index Scan] → [Stream Aggregate] → [SELECT]
Estimated Cost: Scan 100% | Logical Reads: N (all pages in the table)
```

For `SELECT SUM(TotalAmount), AVG(TotalAmount) FROM dbo.Orders`:

```
Expected plan shape:
[Clustered Index Scan] → [Stream Aggregate] → [Compute Scalar (AVG = SUM/COUNT)] → [SELECT]
Estimated Cost: Scan 95%, Aggregate 5%
Logical Reads: All pages in clustered index
```

- The Stream Aggregate maintains two registers (SUM and COUNT) simultaneously. After the scan, the Compute Scalar divides SUM by COUNT to produce AVG.
- If a non-clustered index covering TotalAmount exists, the optimizer may choose that index for a narrower scan.

For `SELECT AVG(CAST(Quantity AS DECIMAL(10,2))) FROM dbo.OrderItems`:

```
Expected plan shape:
[Clustered Index Scan] → [Compute Scalar (CAST)] → [Stream Aggregate] → [Compute Scalar (AVG)] → [SELECT]
Logical Reads: All pages in OrderItems
```

- The CAST adds a Compute Scalar operator between the scan and the aggregate. This is a per-row CPU cost but does not affect logical reads.

### Failure Modes

**Failure Mode 1: SUM integer overflow.**

When summing an INT column across enough rows, the accumulator exceeds INT's 32-bit range. SQL Server raises error 8115 and aborts the query. This is a runtime error — it only appears when data grows past the threshold.

```sql
-- Detection: Check for queries with INT SUM in the plan
SELECT
    qs.total_worker_time,
    qs.total_logical_reads,
    qs.execution_count,
    qt.text
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) qt
WHERE qt.text LIKE '%SUM(%'
AND qt.text LIKE '%INT%';

-- Prevention: Check data growth
SELECT
    SUM(CAST(Quantity AS BIGINT)) AS SafeSum,
    SUM(Quantity) AS UnsafeSum -- risks overflow
FROM dbo.OrderItems;
```

**Failure Mode 2: AVG integer truncation.**

The engineer computes average order value but the column is INT or the result is INT. The fractional part is silently truncated.

```sql
-- WRONG: AVG of INT returns INT
SELECT AVG(Quantity) FROM dbo.OrderItems;
-- If data is 1, 2, 3 → returns 1 (truncated from 1.666...)

-- RIGHT: Cast to DECIMAL
SELECT AVG(CAST(Quantity AS DECIMAL(10,2))) FROM dbo.OrderItems;
-- Returns 1.66
```

**Failure Mode 3: AVG(NULL column) returns NULL.**

When the column is all NULLs for a group, AVG returns NULL, not 0. If the engineer treats NULL as 0 in application code, calculations silently break.

```sql
-- If all ShipDates are NULL for pending orders:
SELECT Status, AVG(ShipDate) FROM dbo.Orders GROUP BY Status;
-- Returns NULL for ShipDate where Status = 1 (Pending)

-- Fix: Use COALESCE to provide default
SELECT Status, AVG(COALESCE(ShipDate, '1900-01-01')) FROM dbo.Orders GROUP BY Status;
-- Better: don't AVG dates — it's usually meaningless
```

**Failure Mode 4: MIN/MAX string comparison with unexpected collation.**

String MIN/MAX uses the column's collation. A case-insensitive collation (SQL_Latin1_General_CP1_CI_AS) treats 'ABC' and 'abc' as equal — the MIN returns whichever it encounters first. A case-sensitive collation treats them differently, with uppercase typically sorting before lowercase.

```sql
-- With CI collation:
SELECT MIN(Email) FROM dbo.Customers;
-- Returns 'admin@company.com' — case-insensitive ordering

-- With CS collation:
SELECT MIN(Email) FROM dbo.Customers;
-- Returns 'Admin@company.com' before 'admin@company.com' because 'A' < 'a'

-- Fix: Explicit COLLATE for consistent behavior
SELECT MIN(Email COLLATE SQL_Latin1_General_CP1_CS_AS) FROM dbo.Customers;
```

**Failure Mode 5: MIN/MAX on GUID columns.**

`NEWID()` generates random GUIDs. Using MIN/MAX on GUID columns is possible but usually meaningless and can be very expensive because GUID indexes are fragmented.

```sql
-- Expensive: MIN on GUID with fragmentation
SELECT MIN(SessionGuid) FROM dbo.UserSessions;
-- Plan: Clustered Index Scan (not seek) because the index is fragmented
-- Consider using sequential GUIDs (NEWSEQUENTIALID) if ordering matters
```

### Primary SQL Implementation

-- Schema context
CREATE TABLE dbo.Orders (
    OrderId INT IDENTITY(1,1) NOT NULL PRIMARY KEY,

);

CREATE TABLE dbo.InventoryTransactions (
    TransactionId BIGINT IDENTITY(1,1) NOT NULL PRIMARY KEY,
    ProductId INT NOT NULL,
    QuantityChange INT NOT NULL, -- positive for inbound, negative for outbound
    TransactionDate DATETIME2 NOT NULL
);

-- Supporting indexes
CREATE INDEX IX_Orders_OrderDate ON dbo.Orders(OrderDate);
CREATE INDEX IX_Orders_CustomerId ON dbo.Orders(CustomerId) INCLUDE (TotalAmount, OrderDate, Status);
CREATE INDEX IX_OrderItems_OrderId ON dbo.OrderItems(OrderId);
CREATE INDEX IX_Products_CategoryId ON dbo.Products(CategoryId) INCLUDE (ListPrice, ProductName);
CREATE INDEX IX_InventoryTransactions_ProductId ON dbo.InventoryTransactions(ProductId, TransactionDate);

-- Production pattern 1: Customer lifetime value with safe SUM
SELECT
    c.CustomerId,
    c.FirstName + ' ' + c.LastName AS CustomerName,
    SUM(CAST(o.TotalAmount AS DECIMAL(18,2))) AS LifetimeValue,
    AVG(o.TotalAmount) AS AverageOrderValue,
    MIN(o.OrderDate) AS FirstOrder,
    MAX(o.OrderDate) AS LastOrder,
    COUNT(*) AS OrderCount,
    DATEDIFF(day, MIN(o.OrderDate), MAX(o.OrderDate)) AS CustomerLifespanDays
FROM dbo.Customers c
INNER JOIN dbo.Orders o ON c.CustomerId = o.CustomerId
WHERE o.Status <> 5 -- exclude cancelled
GROUP BY c.CustomerId, c.FirstName, c.LastName
HAVING COUNT(*) >= 5
ORDER BY LifetimeValue DESC;

-- Production pattern 2: Safe SUM with CAST to prevent overflow
SELECT
    SUM(CAST(Quantity AS BIGINT)) AS TotalItemsSold,
    AVG(CAST(Quantity AS DECIMAL(10,2))) AS AverageItemsPerOrder,
    MIN(Quantity) AS MinItemsInOrder,
    MAX(Quantity) AS MaxItemsInOrder
FROM dbo.OrderItems;

-- Production pattern 3: MIN/MAX on indexed columns for date range
-- With index on OrderDate, these use TOP 1 seek
SELECT
    MIN(o.OrderDate) AS FirstOrderOverall,
    MAX(o.OrderDate) AS LastOrderOverall,
    DATEDIFF(day, MIN(o.OrderDate), MAX(o.OrderDate)) AS DaysOfOperation
FROM dbo.Orders o;

-- Production pattern 4: Conditional aggregation with SUM/CASE
-- Pivot order counts by status in a single scan
SELECT
    COUNT(*) AS TotalOrders,
    SUM(CASE WHEN o.Status = 1 THEN 1 ELSE 0 END) AS Pending,
    SUM(CASE WHEN o.Status = 2 THEN 1 ELSE 0 END) AS Processing,
    SUM(CASE WHEN o.Status = 3 THEN 1 ELSE 0 END) AS Shipped,
    SUM(CASE WHEN o.Status = 4 THEN 1 ELSE 0 END) AS Delivered,
    SUM(CASE WHEN o.Status = 5 THEN 1 ELSE 0 END) AS Cancelled,
    SUM(CASE WHEN o.Status IN (3, 4) THEN o.TotalAmount ELSE 0 END) AS CompletedRevenue,
    AVG(CASE WHEN o.Status NOT IN (1, 5) THEN o.TotalAmount END) AS AvgNonPendingValue
FROM dbo.Orders o
WHERE o.OrderDate >= DATEADD(day, -30, GETUTCDATE());

-- Production pattern 5: AVG with COALESCE for NULL handling
SELECT
    AVG(COALESCE(o.DiscountAmount, 0)) AS AvgDiscount,
    AVG(NULLIF(o.ShippingCost, 0)) AS AvgShippingWhenCharged
FROM dbo.Orders o;

-- Production pattern 6: MIN/MAX with GROUP BY — detect data anomalies
-- Find products with prices outside normal range
SELECT
    p.CategoryId,
    MIN(p.ListPrice) AS MinPrice,
    MAX(p.ListPrice) AS MaxPrice,
    MAX(p.ListPrice) - MIN(p.ListPrice) AS PriceSpread,
    AVG(p.ListPrice) AS AvgPrice
FROM dbo.Products p
GROUP BY p.CategoryId
HAVING MAX(p.ListPrice) > MIN(p.ListPrice) * 10
ORDER BY PriceSpread DESC;

-- Production pattern 7: Running SUM with window function
SELECT
    o.OrderDate,
    o.TotalAmount,
    SUM(o.TotalAmount) OVER(ORDER BY o.OrderDate) AS RunningTotal,
    AVG(o.TotalAmount) OVER(ORDER BY o.OrderDate ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS SevenDayMovingAvg
FROM dbo.Orders o
WHERE o.Status = 4
ORDER BY o.OrderDate;

-- Production pattern 8: AVG with DISTINCT for unique-value average
SELECT
    AVG(DISTINCT CAST(o.TotalAmount AS DECIMAL(18,2))) AS AvgDistinctOrderValue
FROM dbo.Orders o;
-- Averages each unique TotalAmount once — rare pattern but valid for specific analytics

## Gotchas and Production Pitfalls

### 1. SUM Integer Overflow on Large Tables

**Pitfall:** Summing an INT column across millions of rows causes arithmetic overflow when the accumulator exceeds 2,147,483,647.

```sql
-- ❌ Wrong: SUM of INT — will overflow at ~2.1B
SELECT SUM(Quantity) FROM dbo.OrderItems;

-- ✅ Correct: CAST to BIGINT or DECIMAL
SELECT SUM(CAST(Quantity AS BIGINT)) FROM dbo.OrderItems;
SELECT SUM(CAST(Quantity AS DECIMAL(18,0))) FROM dbo.OrderItems;
```

**Symptom:** Error 8115: "Arithmetic overflow error converting expression to data type int." The query fails at runtime, typically during a nightly batch job when data volume has grown past the threshold.

**Fix:** Always CAST INT columns to BIGINT or DECIMAL before SUMming when the total could exceed 2.1B. In EF Core, use `SumAsync(oi => (long)oi.Quantity)`.

**Cost of not fixing:** Production batch job fails at 3 AM. The data warehouse stops updating. Morning reports are empty. The overflow is silent until the query runs — testing with smaller data never catches it.

### 2. AVG Integer Truncation

**Pitfall:** AVG of an INT column returns INT, truncating the fractional part. This produces silently wrong averages.

```sql
-- ❌ Wrong: AVG returns 1, not 1.66
SELECT AVG(Quantity) FROM dbo.OrderItems WHERE OrderId = 100;
-- If items have quantities 1, 2, 3 → returns 1 (truncated)

-- ✅ Correct: CAST to DECIMAL before averaging
SELECT AVG(CAST(Quantity AS DECIMAL(10,2))) FROM dbo.OrderItems;
-- Returns 1.66
```

**Symptom:** The average is always an integer — no decimal values ever appear. For a retail system, "Average Items Per Order" shows 2 instead of 2.47. The business team notices but writes it off as a rounding issue.

**Fix:** Always CAST INT columns to DECIMAL before AVG when fractional precision matters. In EF Core, cast in the LINQ expression: `AverageAsync(oi => (decimal)oi.Quantity)`.

**Cost of not fixing:** Business metrics are off by up to 50%. The "average order value" dashboard shows $45 when the real average is $45.83. The difference compounds in revenue forecasting and bonus calculations.

### 3. AVG of All-NULL Column Returns NULL

**Pitfall:** AVG ignores NULLs, so AVG of a column where all values are NULL returns NULL, not 0. The application may crash or produce wrong results when it tries to display or use NULL as a number.

```sql
-- If all orders in a group have NULL DiscountAmount:
SELECT
    o.Status,
    AVG(o.DiscountAmount) AS AvgDiscount
FROM dbo.Orders o
GROUP BY o.Status;
-- Returns NULL for groups where DiscountAmount is always NULL

-- ✅ Fix: COALESCE to handle NULL
SELECT
    o.Status,
    AVG(COALESCE(o.DiscountAmount, 0)) AS AvgDiscount
FROM dbo.Orders o
GROUP BY o.Status;
```

**Symptom:** The dashboard shows "N/A" or blank for some groups. Chart rendering crashes with null reference exception. Engineers add null checks in application code as a band-aid.

**Fix:** Use `COALESCE` or `ISNULL` inside the AVG call, but understand the semantics: `AVG(COALESCE(col, 0))` treats NULL as 0, lowering the average. Use `AVG(col)` when you want "average of non-NULL values" and `AVG(COALESCE(col, 0))` when you want "average treating missing as zero."

**Cost of not fixing:** In financial reporting, NULL average discount for a product category silently disappears from reports. The analyst creates an incorrect report that excludes the category entirely.

### 4. MIN/MAX on Indexed Columns Creates Deceptively Fast Queries

**Pitfall:** MIN(OrderDate) with an index on OrderDate is fast (3-4 logical reads). But adding a filter changes the plan: `SELECT MIN(OrderDate) FROM Orders WHERE CustomerId = 1234` may revert to a full scan if the index is on OrderDate alone.

```sql
-- Fast: uses TOP 1 seek on IX_Orders_OrderDate (3 reads)
SELECT MIN(OrderDate) FROM dbo.Orders;

-- Slow: scans full table or full index (128K reads)
SELECT MIN(OrderDate) FROM dbo.Orders WHERE CustomerId = 1234;
-- The index on OrderDate does not help CustomerId filtering

-- ✅ Fix: Create a covering index on (CustomerId, OrderDate)
CREATE INDEX IX_Orders_CustomerId_OrderDate ON dbo.Orders(CustomerId, OrderDate);

-- Now this is fast again:
SELECT MIN(OrderDate) FROM dbo.Orders WHERE CustomerId = 1234;
-- Seeks to CustomerId=1234, reads first row
```

**Symptom:** Some MIN/MAX queries are instant (1ms), others on the same table take 10 seconds. The difference is whether the WHERE clause aligns with the index key order.

**Fix:** Create composite indexes that cover both the filter columns and the aggregate column: `INDEX ON (FilterCol, AggregateCol)`. The filter column is the leading key for seek access; the aggregate column provides ordered access for the MIN/MAX.

**Cost of not fixing:** Intermittent slow queries. A customer detail page that shows "First Order" takes 10 seconds for some customers but 1ms for others. The DBA cannot reproduce the issue because it depends on which customer is queried.

### 5. MIN/MAX on VARCHAR with Different Collations

**Pitfall:** String MIN/MAX depends on the column's collation. CI (case-insensitive) vs CS (case-sensitive) collations produce different orderings.

```sql
-- With CI collation: 'a' = 'A', MIN returns whichever appears first in scan order
-- With CS collation: 'A' < 'a', so MIN returns 'A' before 'a'

-- ❌ Unexpected: Different servers with different collations give different results
SELECT MIN(ProductName) FROM dbo.Products;
-- Server A (CI): returns 'apple'
-- Server B (CS): returns 'Apple'

-- ✅ Fix: Explicit COLLATE for consistent behavior
SELECT MIN(ProductName COLLATE SQL_Latin1_General_CP1_CS_AS) FROM dbo.Products;
```

**Symptom:** MIN/MAX results differ between development, staging, and production environments. Unit tests pass but production queries return different values.

**Fix:** Specify the collation explicitly in the query, or ensure all environments use the same collation. For columns used in MIN/MAX comparisons, choose the collation that matches business requirements.

**Cost of not fixing:** Data quality issues in reporting. An alphabetically sorted customer list has different ordering per environment. Data comparison scripts between environments fail for non-functional reasons.

### 6. SUM with JOIN Row Multiplication

**Pitfall:** SUMing a column from the primary table in a query with JOINs that multiply rows produces inflated sums.

```sql
-- ❌ Wrong: SUM(TotalAmount) is inflated because each order row is multiplied by item count
SELECT
    o.OrderId,
    SUM(o.TotalAmount) AS SumOfAmount -- inflated!
FROM dbo.Orders o
INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
GROUP BY o.OrderId;

-- ✅ Correct: Aggregate before join or use subquery
SELECT
    o.OrderId,
    o.TotalAmount
FROM dbo.Orders o
WHERE EXISTS (SELECT 1 FROM dbo.OrderItems oi WHERE oi.OrderId = o.OrderId);

-- Or aggregate in a subquery first:
SELECT o.OrderId, o.TotalAmount
FROM dbo.Orders o
INNER JOIN (
    SELECT OrderId, COUNT(*) AS ItemCount
    FROM dbo.OrderItems
    GROUP BY OrderId
) oi ON o.OrderId = oi.OrderId;
```

**Symptom:** SUM values appear wildly inflated — an order with 10 items shows TotalAmount 10x higher than expected. The engineer may not notice if TotalAmount varies per order.

**Fix:** Never SUM a column from the primary table in a query with a 1-to-many JOIN. Use aggregate-in-subquery, window functions, or process the aggregation before the join.

**Cost of not fixing:** Significant financial reporting errors. Revenue reports showing $5M when real revenue is $500K. Quarterly earnings reports are wrong, potentially affecting stock reporting for public companies.

### Spoken Answers

**Q2: How does the database engine execute MIN(OrderDate) and why can it be so fast?**

> **Average answer:** It scans the column looking for the smallest value. With an index, it's faster.

> **Great answer:** Without an index on OrderDate, the optimizer must scan the entire clustered index — every 8KB page — to find the smallest value. On a 10M row table, that's ~128,000 logical reads and 300+ ms. With a non-clustered index on OrderDate, the optimizer performs a TOP 1 indexed seek. It navigates the B-tree from root to the leftmost leaf page, reading approximately 3-4 pages total — root page, one or two intermediate pages, and the first leaf page. The first row on that leaf page IS the minimum OrderDate. That's 3-4 logical reads and < 1ms — a 40,000x improvement. MAX works identically but seeks to the rightmost leaf page. The key limitation is that this optimization only works for scalar aggregates (no GROUP BY). For `SELECT CustomerId, MIN(OrderDate) FROM Orders GROUP BY CustomerId`, the optimizer needs an index on (CustomerId, OrderDate) to apply the same seek-per-group optimization via a Nested Loops strategy, or it falls back to a full scan.

**Q4: What happens when you SUM an INT column on a table with 1 billion rows?**

> **Average answer:** It might overflow. Cast to BIGINT.

> **Great answer:** SUM of an INT column uses a 32-bit accumulator in SQL Server. The maximum INT value is 2,147,483,647. If you're summing quantities across 1 billion order items, even an average quantity of 3 would produce a sum of 3 billion, which exceeds the INT range. SQL Server raises error 8115 at query runtime — "Arithmetic overflow error converting expression to data type int." This is a runtime error, not a compilation error, so testing with small data never catches it. It only appears when production data crosses the threshold, typically at 3 AM during a batch ETL job. The fix is to CAST the column to a wider type before SUMming: `SUM(CAST(Quantity AS BIGINT))` or `SUM(CAST(Quantity AS DECIMAL(18,0)))`. In EF Core, projecting to `(long)oi.Quantity` before SUM generates a CAST in the SQL. The same issue applies to intermediate calculations — even if the final result looks small, the accumulator can overflow. I've personally debugged this in a production system where SUM of order item quantities on a 500M row OrderItems table crashed the nightly data warehouse load. The CAST to BIGINT fixed it permanently.

**Q6: How does AVG treat NULLs and why does this matter for business reporting?**

> **Average answer:** AVG ignores NULLs. Candidates know the rule but not the implication.

> **Great answer:** AVG ignores NULLs when computing both the numerator (SUM) and denominator (COUNT). If a column has 100 rows with 20 NULLs, AVG divides SUM of the 80 non-NULL values by 80, not 100. This is the correct mathematical behavior for "average of known values," but it produces a different number than most business users expect. For example, AVG(DiscountAmount) for a product category might be 15%, but the real average across all orders including those with no discount is much lower. The engineer must decide: do you want the average of discounts given (AVG without COALESCE), or the average discount per order (AVG(COALESCE(col, 0)))? In EF Core, `AverageAsync(o => o.DiscountAmount)` generates `AVG([DiscountAmount])` — NULLs are excluded. To include zeros, use `AverageAsync(o => (decimal?)o.DiscountAmount ?? 0)`. In Dapper, use `AVG(COALESCE(DiscountAmount, 0))`. I've seen a reporting bug where the marketing team thought the average customer discount was 12%, but it was actually 5% because 70% of orders had no discount (NULL). The 12% was the average of the 30% of orders that received discounts. Always clarify the business definition before writing AVG.

## Decision Framework

### When to Apply

```mermaid
flowchart TD
    A[Need to compute aggregate?] --> B{Which function?}
    B -->|Total of numeric values| C[SUM]
    B -->|Typical or central value| D[AVG]
    B -->|Smallest value| E[MIN]
    B -->|Largest value| F[MAX]
    C --> G{Column type?}
    G -->|INT — large table| H[CAST to BIGINT/DECIMAL first]
    G -->|DECIMAL — safe| I[SUM directly]
    G -->|FLOAT — precision| J[Consider rounding concerns]
    D --> K{Column type?}
    K -->|INT| L[CAST to DECIMAL for precision]
    K -->|DECIMAL| M[AVG directly — preserves fractions]
    K -->|FLOAT| N[ROUND(AVG(col), 2) for presentation]
    E --> O{Index on column?}
    O -->|Yes| P[TOP 1 seek — O(log N)]
    O -->|No| Q[Full scan — O(N) — create index]
    F --> R{Index on column?}
    R -->|Yes| S[TOP 1 seek — O(log N)]
    R -->|No| T[Full scan — O(N) — create index]
    H --> U[Safe from overflow]
    L --> V[Preserves fractional result]
    P --> W[Sub-millisecond — 3-4 logical reads]
    S --> W
```

### Application Checklist

- [ ] INT columns in SUM are CAST to BIGINT or DECIMAL to prevent overflow on large tables
- [ ] AVG of INT column is CAST to DECIMAL to preserve fractional precision
- [ ] MIN/MAX queries have an index on the target column for TOP 1 seek optimization
- [ ] Grouped MIN/MAX have a composite index: (GROUP BY column, MIN/MAX column)
- [ ] NULL behavior is understood: AVG ignores NULLs; COALESCE is used when zeros should replace NULLs
- [ ] JOINs in the query do not inflate SUM values via row multiplication
- [ ] The result type matches the expected precision (DECIMAL vs FLOAT vs INT)
- [ ] EF Core LINQ expressions generate the correct SQL with proper CASTs

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|MIN/MAX TOP 1 seek speed|Non-clustered index write overhead on INSERT/UPDATE|
|Covering index for SUM/AVG|Additional storage for included columns|
|CAST to DECIMAL precision|Slight per-row CPU cost for type conversion|
|COALESCE for NULL handling|Potentially different business semantics|

### Scale Thresholds

- "MIN/MAX index seek matters for any table over ~10,000 rows" — the difference between 4 reads and 4,000 reads.
- "SUM overflow risk becomes real when the sum exceeds ~2 billion" — for an INT column with average value 100, this is 20M rows.
- "AVG integer truncation matters when business users care about fractional values" — always for currency, often for quantities.
- "Grouped MIN/MAX composite index matters when the group count exceeds ~1,000 and the query runs frequently."

### Conceptual Questions

1. What does SUM return when all values in the group are NULL?
2. How does the database engine execute `SELECT MAX(OrderDate) FROM Orders` if an index on OrderDate exists?
3. Which SET STATISTICS option reveals the number of logical reads for a SUM query?
4. What happens when you AVG an INT column on a table with values 1, 2, 3, 4, 5?
5. Does EF Core's SumAsync generate SARGable SQL on the column being summed?
6. How would you write a safe SUM that handles both NULLs and overflow in Dapper?
7. Compare MIN/MAX with and without an index — what is the plan difference?
8. At what table size does the MIN/MAX index optimization become critical?
9. What index supports the query `SELECT CustomerId, MAX(OrderDate) FROM Orders GROUP BY CustomerId`?
10. Explain the relationship between AVG, SUM, and COUNT in 60 seconds.

<details>
<summary>Answers</summary>

1. SUM returns NULL when all values are NULL. It ignores NULLs, so if there are no non-NULL values, the result is NULL. If there are no rows at all, the result is also NULL. Use `COALESCE(SUM(col), 0)` to return 0 instead.

2. The optimizer performs a TOP 1 seek on the non-clustered index. It navigates the B-tree from the root to the rightmost leaf page (for MAX) or leftmost leaf page (for MIN). It reads the first/last row on that page. The plan shows `[Index Seek]` with the "Ordered: True" property. Approximately 3-4 logical reads are needed regardless of table size.

3. `SET STATISTICS IO ON` shows logical reads for any query, including aggregates. The output shows the number of 8KB pages read from the buffer pool for each table accessed.

4. AVG(Quantity) on INT values 1, 2, 3, 4, 5 returns INT 3. The actual average is 3.0 (15/5 = 3 exactly in this case). A more dangerous example: values 1, 2, 3, 4, 6 give sum 16, count 5, AVG = 3 (truncated from 3.2). To get 3.2, use `AVG(CAST(Quantity AS DECIMAL(10,2)))`.

5. SumAsync is not a predicate — SARGability does not directly apply. However, the WHERE clause in the query (if any) must be SARGable for the overall query to be efficient. SumAsync generates `SELECT SUM([col]) FROM [Table] WHERE ...` — the WHERE predicate's SARGability is what matters.

6. Dapper safe SUM with overflow and NULL handling:
```csharp
var total = await connection.ExecuteScalarAsync<long>(
    "SELECT ISNULL(SUM(CAST(Quantity AS BIGINT)), 0) FROM dbo.OrderItems",
    cancellationToken: ct);
```

7. Without index: Clustered Index Scan → Stream Aggregate. Reads all pages in the table. With index: Index Seek (TOP 1) → (no-op) Stream Aggregate → SELECT. Reads 3-4 pages. The cost difference is O(N) vs O(log N).

8. The MIN/MAX index optimization is critical when the table exceeds approximately 10,000 rows. At 10K rows, the Clustered Index Scan reads ~100-200 pages (~1-2ms). At 10M rows, it reads ~128,000 pages (~300+ ms). The index seek remains at 3-4 pages regardless of table size. For queries running hundreds of times per hour, even a 10K row table benefits from the optimization.

9. The optimal index is on `(CustomerId, OrderDate)`. The leading key (CustomerId) enables the optimizer to seek to each customer's rows. The secondary key (OrderDate) provides ordered access so the Stream Aggregate can get the MAX without a Sort. This eliminates both the Sort and the full scan.

10. AVG is derived from SUM and COUNT: `AVG(col) = SUM(col) / COUNT(col)`. All three ignore NULLs in the column. SUM adds all non-NULL values. COUNT counts the number of non-NULL values. AVG divides SUM by COUNT. If the column is INT, both SUM and COUNT are INT, so the division is integer division — the fractional part is truncated. To get a fractional AVG, CAST the column to DECIMAL before averaging.

</details>

**Challenge 1 — Write the SQL**

The analytics team needs a daily sales report showing: total revenue (SUM of TotalAmount), average order value, number of orders, the highest-value order, and the total discount given. All values must not overflow and NULL discounts should be treated as 0. Show only the last 7 complete days, ordered by day descending. Include days with no orders showing 0 values.

<details>
<summary>Solution</summary>

```sql
WITH DateRange AS (
    SELECT CAST(DATEADD(day, -n, GETUTCDATE()) AS DATE) AS ReportDay
    FROM (VALUES (1),(2),(3),(4),(5),(6),(7)) AS Numbers(n)
)
SELECT
    dr.ReportDay,
    COALESCE(SUM(CAST(o.TotalAmount AS DECIMAL(18,2))), 0) AS TotalRevenue,
    COALESCE(AVG(o.TotalAmount), 0) AS AverageOrderValue,
    COALESCE(COUNT(*), 0) AS OrderCount,
    COALESCE(MAX(o.TotalAmount), 0) AS HighestOrderValue,
    COALESCE(SUM(CAST(COALESCE(o.DiscountAmount, 0) AS DECIMAL(18,2))), 0) AS TotalDiscount
FROM DateRange dr
LEFT JOIN dbo.Orders o ON CAST(o.OrderDate AS DATE) = dr.ReportDay
GROUP BY dr.ReportDay
ORDER BY dr.ReportDay DESC;
```

**Logical reads:** ~N (depends on index) **Execution plan:** Compute Scalar → Nested Loops (Left Join) → Stream Aggregate **EF Core equivalent:**

```csharp
var results = Enumerable.Range(1, 7)
    .Select(d => DateTime.UtcNow.Date.AddDays(-d))
    .GroupJoin(
        dbContext.Orders.Where(o => o.OrderDate >= DateTime.UtcNow.Date.AddDays(-8)),
        day => day,
        order => order.OrderDate.Date,
        (day, orders) => new {
            ReportDay = day,
            TotalRevenue = orders.Sum(o => (decimal)o.TotalAmount),
            AverageOrderValue = orders.Any() ? orders.Average(o => o.TotalAmount) : 0,
            OrderCount = orders.Count(),
            HighestOrderValue = orders.Any() ? orders.Max(o => o.TotalAmount) : 0,
            TotalDiscount = orders.Sum(o => (decimal)(o.DiscountAmount ?? 0))
        })
    .OrderByDescending(x => x.ReportDay)
    .ToList();
```

**Challenge 2 — Fix the performance problem**

```sql
-- This query runs in 12 seconds on a 40M row Orders table.
-- SET STATISTICS IO: logical reads = 412,345
SELECT
    c.CustomerId,
    c.FirstName + ' ' + c.LastName AS CustomerName,
    SUM(CAST(o.TotalAmount AS DECIMAL(18,2))) AS LifetimeValue,
    MAX(o.OrderDate) AS LastOrderDate
FROM dbo.Customers c
INNER JOIN dbo.Orders o ON c.CustomerId = o.CustomerId
WHERE o.OrderDate >= DATEADD(year, -1, GETUTCDATE())
GROUP BY c.CustomerId, c.FirstName, c.LastName
ORDER BY LifetimeValue DESC;
```

<details> <summary>Solution</summary>

**Root cause:** Full Clustered Index Scan on Orders (412K logical reads). No index supports the WHERE filter or the GROUP BY. The Hash Aggregate is used because no order exists. The Sort for ORDER BY adds additional memory.

```sql
-- Fixed: Create supporting indexes
-- Index 1: Covering index for the filter + GROUP BY
CREATE INDEX IX_Orders_Covering_Agg ON dbo.Orders(CustomerId, OrderDate)
    INCLUDE (TotalAmount);

-- After index, the plan changes:
-- Index Seek (IX_Orders_Covering_Agg) seeking to CustomerId ranges
-- Stream Aggregate (sorted by CustomerId from index)
-- No Sort needed for GROUP BY (CustomerId is leading key)

-- Additional optimization: Pre-aggregate in a derived table
SELECT
    c.CustomerId,
    c.FirstName + ' ' + c.LastName AS CustomerName,
    oa.LifetimeValue,
    oa.LastOrderDate
FROM dbo.Customers c
INNER JOIN (
    SELECT
        o.CustomerId,
        SUM(CAST(o.TotalAmount AS DECIMAL(18,2))) AS LifetimeValue,
        MAX(o.OrderDate) AS LastOrderDate
    FROM dbo.Orders o
    WHERE o.OrderDate >= DATEADD(year, -1, GETUTCDATE())
    GROUP BY o.CustomerId
) oa ON c.CustomerId = oa.CustomerId
ORDER BY LifetimeValue DESC;
```

**Index to create:**

```sql
CREATE INDEX IX_Orders_CustomerId_OrderDate ON dbo.Orders(CustomerId, OrderDate)
    INCLUDE (TotalAmount);
```

**After fix — logical reads:** ~18,452 (from 412,345) — 22x reduction.

**Challenge 3 — Explain the execution plan**

```sql
SELECT MIN(o.OrderDate), MAX(o.OrderDate)
FROM dbo.Orders o
WHERE o.CustomerId = 1234;
```

The execution plan shows a Clustered Index Scan (80% cost) → Stream Aggregate (20% cost). Logical reads: 128,342. Why does the optimizer not use the IX_Orders_OrderDate index for a TOP 1 seek?

<details> <summary>Solution</summary>

**Why Clustered Index Scan:** The IX_Orders_OrderDate index is on (OrderDate) alone. It does not include CustomerId. The WHERE clause filters by CustomerId. To use IX_Orders_OrderDate, the optimizer would need to either: (a) scan the entire index (all 34,211 pages) and check CustomerId for every row, or (b) scan the clustered index which has both CustomerId and OrderDate. The optimizer chose the Clustered Index Scan because it avoids the key lookup cost. However, this is a full scan.

**To get an Index Seek:** Create a composite index on (CustomerId, OrderDate):

```sql
CREATE INDEX IX_Orders_CustomerId_OrderDate ON dbo.Orders(CustomerId, OrderDate);
```

With this index, the plan becomes:
`[Index Seek (seeking CustomerId=1234)] → [Stream Aggregate (reads MIN and MAX from first and last row in range)] → [SELECT]`
Logical reads: ~10-15 (seek to CustomerId=1234, read its range of rows).

The index works because: (a) the leading key CustomerId enables the seek for the WHERE filter, (b) the secondary key OrderDate provides ordered access within each customer's range, allowing the Stream Aggregate to read the first (MIN) and last (MAX) row without scanning all rows.

**Tradeoff:** This index adds write overhead on Orders (INSERT/UPDATE/DELETE must maintain the composite index). But for a 95:5 read/write ratio, the query performance gain (128K reads to 15 reads) justifies the overhead.

</details>

**Challenge 4 — Diagnose the concurrency problem**

A reporting dashboard runs `SELECT SUM(TotalAmount), AVG(TotalAmount) FROM Orders` every 30 seconds. A nightly batch process runs `UPDATE Orders SET Status = 4 WHERE Status = 3 AND OrderDate < @date` affecting 200,000 rows. During the batch update, the dashboard SUM/AVG queries deadlock with the UPDATE. The deadlock graph shows dashboard holding a SCH-S (schema stability) lock and the UPDATE holding X (exclusive) locks, with both waiting on the other.

<details> <summary>Solution</summary>

**Root cause:** This is a schema stability vs exclusive lock deadlock, or more likely an intra-query parallelism deadlock. The SUM/AVG query scans the entire Orders table, requesting shared (S) page locks. The UPDATE modifies 200,000 rows, holding exclusive (X) page locks. When the scanner and modifier hit the same page in different orders, a deadlock occurs.

**Detection query:**

```sql
SELECT
    deadlock_graph
FROM sys.dm_tran_locks
WHERE request_session_id IN (
    SELECT blocking_session_id FROM sys.dm_exec_requests WHERE blocking_session_id > 0
);

-- Or use system_health session:
SELECT event_data
FROM sys.fn_xe_file_target_read_file(
    'system_health*.xel', NULL, NULL, NULL)
WHERE event_data LIKE '%deadlock%';
```

**Fix:** Use snapshot isolation or RCSI for the reporting query:

```sql
-- Enable RCSI (no application code change needed)
ALTER DATABASE Current SET READ_COMMITTED_SNAPSHOT ON;

-- Or explicitly use SNAPSHOT isolation for the reporting session
SET TRANSACTION ISOLATION LEVEL SNAPSHOT;
SELECT SUM(TotalAmount), AVG(TotalAmount) FROM dbo.Orders;
```

**In .NET:**

```csharp
// Connection string approach (RCSI):
"Server=.;Database=MyDb;Read Committed Snapshot=True;"

// Or explicit transaction scope:
await using var scope = new TransactionScope(
    TransactionScopeAsyncFlowOption.Enabled);
await using var connection = new SqlConnection(connectionString);
await connection.OpenAsync(ct);
await connection.ExecuteAsync("SET TRANSACTION ISOLATION LEVEL SNAPSHOT", ct);
var sum = await connection.ExecuteScalarAsync<decimal>(
    "SELECT SUM(TotalAmount) FROM dbo.Orders", ct);
```

</details>

**Challenge 5 — Design the index**

A product dashboard runs the following query every 2 minutes on a 30M row Orders table:

```sql
SELECT
    p.CategoryId,
    AVG(CAST(oi.Quantity AS DECIMAL(10,2))) AS AvgQuantityPerItem,
    SUM(CAST(oi.Quantity * oi.UnitPrice AS DECIMAL(18,2))) AS TotalRevenue,
    MAX(oi.UnitPrice) AS HighestUnitPrice,
    MIN(p.ListPrice) AS MinListPrice
FROM dbo.Orders o
INNER JOIN dbo.OrderItems oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
WHERE o.OrderDate >= DATEADD(month, -3, GETUTCDATE())
GROUP BY p.CategoryId;
```

The query takes 45 seconds and does 1.2M logical reads. Read/write ratio is 90:10. Design the optimal index strategy.

<details> <summary>Solution</summary>

```sql
-- Analysis: The query joins 3 tables, filters on Orders.OrderDate,
-- groups by Products.CategoryId, and aggregates columns from OrderItems and Products.

-- The most expensive part is the join between Orders and OrderItems (1:M)
-- and then to Products, followed by GROUP BY on a column from Products.

-- Strategy 1: Index on OrderItems to support the join to Products
CREATE INDEX IX_OrderItems_ProductId_OrderId ON dbo.OrderItems(ProductId, OrderId)
    INCLUDE (Quantity, UnitPrice);
-- This index covers the OrderItems-to-Products join (ProductId leading key)
-- and includes all columns needed for aggregation

-- Strategy 2: Index on Orders for the date filter
CREATE INDEX IX_Orders_OrderDate ON dbo.Orders(OrderDate)
    INCLUDE (OrderId);
-- Narrow index for date range seek

-- Strategy 3: If the GROUP BY on CategoryId is the bottleneck,
-- consider denormalizing CategoryId into OrderItems
ALTER TABLE dbo.OrderItems ADD CategoryId INT NULL
    REFERENCES dbo.Products(ProductId);
UPDATE oi SET oi.CategoryId = p.CategoryId
FROM dbo.OrderItems oi INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId;
ALTER TABLE dbo.OrderItems ALTER COLUMN CategoryId INT NOT NULL;

CREATE INDEX IX_OrderItems_CategoryId ON dbo.OrderItems(CategoryId, OrderDate)
    INCLUDE (Quantity, UnitPrice);

-- Then rewrite the query to avoid the Products join:
SELECT
    oi.CategoryId,
    AVG(CAST(oi.Quantity AS DECIMAL(10,2))) AS AvgQuantityPerItem,
    SUM(CAST(oi.Quantity * oi.UnitPrice AS DECIMAL(18,2))) AS TotalRevenue,
    MAX(oi.UnitPrice) AS HighestUnitPrice,
    MAX(p.MinListPrice) AS MinListPrice -- adjust as needed
FROM dbo.OrderItems oi
INNER JOIN dbo.Orders o ON oi.OrderId = o.OrderId
INNER JOIN dbo.Products p ON oi.ProductId = p.ProductId
WHERE o.OrderDate >= DATEADD(month, -3, GETUTCDATE())
GROUP BY oi.CategoryId;
```

**Tradeoffs:** The denormalization approach (strategy 3) is the highest impact but introduces data redundancy. The indexing approach (strategies 1 + 2) should reduce logical reads significantly without schema changes. Start with strategies 1 and 2 — they typically reduce reads by 70-80%.

**What NOT to index:** Do not create an index on Products.CategoryId alone — it is a narrow lookup table and scanning it is negligible compared to OrderItems.

</details>
