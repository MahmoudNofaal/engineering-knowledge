### Where This Fits

ROLLUP generates hierarchical subtotals and a grand total in a single query — it replaces the manual UNION ALL of separate GROUP BY queries for each level of the hierarchy. A .NET backend engineer writing reporting queries, financial summaries, or inventory rollups encounters ROLLUP when they need to show department totals within a region total within a company total, all in one result set. The most expensive mistake is not understanding that ROLLUP-specific NULLs (subtotal placeholders) collide with real NULL data, requiring GROUPING() to distinguish them. Another common failure is using ROLLUP when the UI needs every combination of dimensions — that is CUBE or GROUPING SETS. Getting this wrong means shipping UNION ALL queries that scan the table N times (one per hierarchy level) instead of ROLLUP's single scan. Interviewers use ROLLUP to test whether a candidate understands grouping set theory and can distinguish hierarchical aggregation (ROLLUP) from complete aggregation (CUBE).

## Core Mental Model

ROLLUP generates a result set that includes the detail rows (same as GROUP BY without ROLLUP), plus subtotal rows for each level of the grouping hierarchy, plus a grand total row. For GROUP BY ROLLUP(col1, col2, col3), the output contains rows grouped by (col1, col2, col3), then by (col1, col2), then by (col1), then by () — the grand total. That is N+1 levels of aggregation for N columns. ROLLUP assumes a hierarchy: col1 is the top level, col2 is the next level, col3 is the leaf level. Subtle: ROLLUP does NOT generate all possible combinations — it only generates the hierarchical ones. For 3 columns, ROLLUP generates 4 grouping sets (3+1); CUBE generates 8 (2^3). The NULL values in subtotal rows are placeholders meaning "this column was aggregated away" — they are not real data NULLs. Use GROUPING() to distinguish: GROUPING(col1) returns 1 when the row is a subtotal that rolled up col1, 0 when col1 contains a real value. The database engine implements ROLLUP as a single scan of the data (or index) followed by a Segment operator that partitions the sorted data, a Split operator that duplicates rows for each grouping set, and a Stream Aggregate that computes each grouping set's aggregates.

### Classification

ROLLUP is a **GROUP BY subclause** (grouping set specification) in the `SELECT` statement. It produces hierarchical subtotals as additional rows in the result set. The base GROUP BY (without ROLLUP) remains SARGable depending on the WHERE predicates — the ROLLUP extension does not change SARGability. The execution plan adds Segment, Split, and Stream Aggregate operators after the data access. ROLLUP scans the data once, unlike UNION ALL of multiple GROUP BY queries which scans N+1 times. ROLLUP is part of the SQL-99 standard with `GROUP BY ROLLUP(col1, col2)` syntax; SQL Server also supports the older `WITH ROLLUP` syntax for backward compatibility.

```mermaid
flowchart TD
    A[Base table or index] --> B[WHERE filter]
    B --> C[Sort by ROLLUP columns: col1, col2, col3]
    C --> D[Segment: identify group boundaries]
    D --> E[Split: duplicate rows for each grouping set]
    E --> F[Stream Aggregate: compute per grouping set]
    F --> G{Which grouping set?}
    G -->|(col1, col2, col3)| H[Row with actual col1, col2, col3 values]
    G -->|(col1, col2)| I[Row with actual col1, col2, NULL for col3]
    G -->|(col1)| J[Row with actual col1, NULL for col2 and col3]
    G -->|()| K[Row with NULL for col1, col2, col3 - Grand Total]
    
    L[GROUPING function] --> M{Is this column rolled up?}
    M -->|GROUPING(col)=1| N[Column has subtotal NULL placeholder]
    M -->|GROUPING(col)=0| O[Column has real value]
    
    P[Without covering index] --> Q[Sort spills to tempdb - expensive]
    R[With covering index in ROLLUP column order] --> S[Ordered scan - no sort needed]
```

## Deep Mechanics

### How the Engine Executes This

1. **Parsing** — The parser recognizes the ROLLUP specification in the GROUP BY clause. Two syntax forms are supported: `GROUP BY col1, col2 WITH ROLLUP` (old, SQL Server specific) and `GROUP BY ROLLUP(col1, col2)` (standard, preferred). Both produce the same logical result.

2. **Binding (Algebrizer)** — The algebrizer expands ROLLUP into its constituent grouping sets. For `GROUP BY ROLLUP(col1, col2, col3)`, the grouping sets are: (col1, col2, col3), (col1, col2), (col1), (). The algebrizer records that NULLs will be injected for the rolled-up columns in each subtotal row.

3. **Simplification** — The optimizer evaluates whether the ROLLUP can use an ordered scan of a covering index. If an index exists with leading columns matching the ROLLUP order (col1, col2, col3), the Sort operator can be eliminated and the Segment/Split/Aggregate operates on an already-ordered stream. If no such index exists, a Sort operator is added.

4. **Query Plan Generation** — The optimizer builds a plan with these operators:
   - **Data access**: Clustered Index Scan or Index Seek (with covering index)
   - **Sort** (if needed): sorts the data by the ROLLUP column order
   - **Segment**: partitions the sorted stream into groups. For each boundary where a grouping column value changes, a new segment starts.
   - **Split**: for each input row, duplicates it for each grouping set that includes the row. A row at the (col1, col2, col3) level contributes to all four grouping sets. The Split operator outputs N+1 copies of each row.
   - **Stream Aggregate**: aggregates the split rows for each grouping set. Because the data is sorted and the Split tagged each copy with its grouping set, the Stream Aggregate can compute all aggregates in a single pass.

5. **GROUPING() Evaluation** — The GROUPING() function is evaluated per row. For a subtotal row where col1 = 'USA', col2 = NULL (subtotal for col3 rollup), `GROUPING(col2)` returns 1, `GROUPING(col1)` returns 0. For the grand total row where all columns are NULL, all GROUPING() calls return 1.

6. **Execution** — The plan executes in a single pass over the data. The Segment operator tracks column value changes. The Split operator fans out rows. The Stream Aggregate computes SUM, COUNT, AVG per grouping set. The result set contains detail rows plus subtotal rows plus grand total row — typically more rows than a simple GROUP BY but dramatically fewer rows than UNION ALL of separate GROUP BY queries.

```sql
-- ROLLUP: monthly sales by category with subtotals and grand total
-- Standard syntax (SQL-99)
SELECT
    p.CategoryId,
    YEAR(o.OrderDate) AS OrderYear,
    MONTH(o.OrderDate) AS OrderMonth,
    SUM(oi.Quantity * oi.UnitPrice * (1 - oi.DiscountPct / 100)) AS Revenue,
    COUNT(DISTINCT o.OrderId) AS OrderCount,
    GROUPING(p.CategoryId) AS IsCategoryRollup,
    GROUPING(YEAR(o.OrderDate)) AS IsYearRollup,
    GROUPING(MONTH(o.OrderDate)) AS IsMonthRollup
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi
    ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p
    ON oi.ProductId = p.ProductId
WHERE o.OrderDate >= '2024-01-01'
  AND o.OrderDate < '2025-01-01'
GROUP BY ROLLUP(p.CategoryId, YEAR(o.OrderDate), MONTH(o.OrderDate))
ORDER BY
    CASE WHEN GROUPING(p.CategoryId) = 1 THEN 1 ELSE 0 END,
    p.CategoryId,
    CASE WHEN GROUPING(YEAR(o.OrderDate)) = 1 THEN 1 ELSE 0 END,
    YEAR(o.OrderDate),
    CASE WHEN GROUPING(MONTH(o.OrderDate)) = 1 THEN 1 ELSE 0 END,
    MONTH(o.OrderDate);
-- Result rows: detail (cat × year × month) + subtotal per year per category
--   + subtotal per category + grand total = ~(C×Y×M + C×Y + C + 1) rows
```

### Execution Plan Analysis

**ROLLUP with 3 columns (Category, Year, Month) on 10M row Orders table:**

```
  [Clustered Index Scan] Orders (or Index Seek if filtered)
  Estimated Rows: 5,000,000 (filtered by date range)
  Logical Reads: ~31,000
  
  [Hash Match Join] Orders × OrderItems
  Estimated Rows: 15,000,000 (3 items per order average)
  Logical Reads: ~45,000
  
  [Hash Match Join] × Products
  Estimated Rows: 15,000,000
  Logical Reads: ~2,000
  
  [Sort] BY CategoryId, YEAR(OrderDate), MONTH(OrderDate)
  Estimated Rows: 15,000,000
  Memory Grant: ~500MB (potential tempdb spill if insufficient)
  
  [Segment]
  Identifies group boundaries: when CategoryId changes, when year changes, when month changes
  
  [Split]
  Duplicates each row for 4 grouping sets: detail × 4 = 60,000,000 rows output
  
  [Stream Aggregate]
  Groups by (CategoryId, Year, Month) UNION ALL (CategoryId, Year) UNION ALL (CategoryId) UNION ALL ()
  Computes SUM(Revenue), COUNT(DISTINCT OrderId) per grouping set
  
  → [SELECT]
  Total Estimated Cost: ~85  |  Total Logical Reads: ~78,000
  Result rows: ~15,000 (500 categories × 1 year × 12 months + 500 subtotals + 1 grand total)
```

**With covering index (no Sort needed):**

```
  [Index Scan] IX_Covering_Report (CategoryId, OrderDate) INCLUDE (Quantity, UnitPrice, DiscountPct)
  -- Already ordered by CategoryId, OrderDate — no Sort operator!
  Estimated Rows: 15,000,000
  Logical Reads: ~50,000
  
  [Segment] → [Split] → [Stream Aggregate]
  No Sort operator — saves ~500MB memory grant and potential tempdb spill
  
  Total Estimated Cost: ~45  |  Total Logical Reads: ~50,000
```

**UNION ALL equivalent (4 separate GROUP BY queries):**

```
  [Clustered Index Scan] × 4 separate scans
  Total Logical Reads: ~200,000 (4 × 50,000)
  Total Estimated Cost: ~180 (more than 2× the ROLLUP cost)
  Plus: 4× the plan compilation time, 4× the memory grant requests
```

### Failure Modes

**NULL collision between real data and ROLLUP subtotal placeholders:** If a column contains real NULL values AND a ROLLUP subtotal NULL, the two are indistinguishable in the result set. The `GROUPING()` function is the only reliable way to distinguish them.

```sql
-- ❌ Problem: CategoryId contains NULL for uncategorized products
-- The subtotal row for CategoryId = NULL (real) vs grand total CategoryId = NULL
-- look identical without GROUPING()
SELECT p.CategoryId, SUM(oi.Quantity) AS UnitsSold
FROM dbo.Products AS p
INNER JOIN dbo.OrderItems AS oi ON p.ProductId = oi.ProductId
GROUP BY ROLLUP(p.CategoryId);
-- Both real NULL and grand total show as NULL in CategoryId column

-- ✅ Fix: Use GROUPING() to distinguish
SELECT
    CASE WHEN GROUPING(p.CategoryId) = 1 THEN 'Grand Total'
         WHEN p.CategoryId IS NULL THEN 'Uncategorized'
         ELSE CAST(p.CategoryId AS VARCHAR(10))
    END AS CategoryLabel,
    SUM(oi.Quantity) AS UnitsSold
FROM dbo.Products AS p
INNER JOIN dbo.OrderItems AS oi ON p.ProductId = oi.ProductId
GROUP BY ROLLUP(p.CategoryId);
```

**HAVING filters out subtotal rows:** A HAVING clause that filters on the aggregate value may accidentally remove subtotal rows that the application expects.

```sql
-- ❌ HAVING removes subtotals with low aggregate values
SELECT p.CategoryId, YEAR(o.OrderDate) AS OrderYear,
       SUM(oi.Quantity) AS UnitsSold
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
GROUP BY ROLLUP(p.CategoryId, YEAR(o.OrderDate))
HAVING SUM(oi.Quantity) > 1000;
-- The subtotal row for a category with total > 1000 but some months < 1000
-- will be included, but the grand total row may be excluded if total < 1000

-- ✅ Fix: Use GROUPING() to preserve subtotal rows
SELECT p.CategoryId, YEAR(o.OrderDate) AS OrderYear,
       SUM(oi.Quantity) AS UnitsSold
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
GROUP BY ROLLUP(p.CategoryId, YEAR(o.OrderDate))
HAVING GROUPING(p.CategoryId) = 1
    OR GROUPING(YEAR(o.OrderDate)) = 1
    OR SUM(oi.Quantity) > 1000;
-- Always include subtotal and grand total rows
```

**Missing ORDER BY for meaningful output order:** ROLLUP returns rows in grouping set order (detail first, then subtotals, then grand total), but without an explicit ORDER BY, the order is not guaranteed. The GROUPING() function must be used in ORDER BY to place subtotals in the logical position.

**Memory grant for Sort when no covering index exists:** ROLLUP requires sorted data. Without an index in the ROLLUP column order, a Sort operator is added. On large tables (10M+ rows), this Sort can spill to tempdb, increasing I/O dramatically and potentially failing with insufficient disk space.

```sql
    p.ProductId,
    p.ProductName,
    SUM(oi.Quantity * oi.UnitPrice * (1 - oi.DiscountPct / 100)) AS Revenue,
    COUNT(DISTINCT o.OrderId) AS OrderCount,
    GROUPING(d.DepartmentId) AS IsDeptRollup,
    GROUPING(pc.CategoryId) AS IsCatRollup,
    GROUPING(p.ProductId) AS IsProductRollup
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi
    ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p
    ON oi.ProductId = p.ProductId
INNER JOIN dbo.ProductCategories AS pc
    ON p.CategoryId = pc.CategoryId
INNER JOIN dbo.Departments AS d
    ON pc.DepartmentId = d.DepartmentId
WHERE o.OrderDate >= '2024-01-01'
  AND o.OrderDate < '2025-01-01'
  AND o.Status IN ('Shipped', 'Delivered')
GROUP BY ROLLUP(d.DepartmentId, d.DepartmentName,
                pc.CategoryId, pc.CategoryName,
                p.ProductId, p.ProductName)
ORDER BY
    CASE WHEN GROUPING(d.DepartmentId) = 1 THEN 1 ELSE 0 END,
    d.DepartmentId,
    CASE WHEN GROUPING(pc.CategoryId) = 1 THEN 1 ELSE 0 END,
    pc.CategoryId,
    CASE WHEN GROUPING(p.ProductId) = 1 THEN 1 ELSE 0 END,
    p.ProductId;

-- ============================================================
-- Pattern 2: Shipping analysis with GROUPING() + COALESCE display
-- ============================================================
SELECT
    CASE WHEN GROUPING(s.ShipCountry) = 1 THEN 'ALL COUNTRIES'
         ELSE s.ShipCountry END AS Country,
    CASE WHEN GROUPING(s.ShipState) = 1 THEN 'ALL STATES'
         ELSE s.ShipState END AS State,
    CASE WHEN GROUPING(s.ShipCity) = 1 THEN 'ALL CITIES'
         ELSE s.ShipCity END AS City,
    COUNT(DISTINCT o.OrderId) AS TotalOrders,
    SUM(o.TotalAmount) AS TotalRevenue,
    AVG(o.TotalAmount) AS AvgOrderValue,
    COUNT(DISTINCT o.CustomerId) AS UniqueCustomers
FROM dbo.Orders AS o
INNER JOIN dbo.ShipmentDetails AS s
    ON o.OrderId = s.OrderId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY ROLLUP(s.ShipCountry, s.ShipState, s.ShipCity)
HAVING GROUPING(s.ShipCountry) = 1
    OR GROUPING(s.ShipState) = 1
    OR GROUPING(s.ShipCity) = 1
    OR COUNT(DISTINCT o.OrderId) > 10
ORDER BY Country, State, City;

-- ============================================================
-- Pattern 3: Partial ROLLUP — only roll up the last N columns
-- ============================================================
-- Group by (Department, Category) and ROLLUP (Year, Month)
SELECT
    d.DepartmentId, d.DepartmentName,
    pc.CategoryId, pc.CategoryName,
    YEAR(o.OrderDate) AS OrderYear,
    MONTH(o.OrderDate) AS OrderMonth,
    SUM(oi.Quantity) AS UnitsSold,
    GROUPING(YEAR(o.OrderDate)) AS IsYearRollup
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
INNER JOIN dbo.ProductCategories AS pc ON p.CategoryId = pc.CategoryId
INNER JOIN dbo.Departments AS d ON pc.DepartmentId = d.DepartmentId
GROUP BY d.DepartmentId, d.DepartmentName,
         pc.CategoryId, pc.CategoryName,
         ROLLUP(YEAR(o.OrderDate), MONTH(o.OrderDate));
-- Result: (Dept, Cat, Year, Month) + (Dept, Cat, Year) + (Dept, Cat)
-- Does NOT include (Dept) alone or grand total — those are outside ROLLUP scope

-- ============================================================
-- Pattern 4: ROLLUP with calculated columns
-- ============================================================
SELECT
    p.CategoryId,
    CASE WHEN GROUPING(p.CategoryId) = 1 THEN 'Storewide'
         ELSE CAST(p.CategoryId AS VARCHAR(10)) END AS CategoryDisplay,
    CASE
        WHEN GROUPING(YEAR(o.OrderDate)) = 1 AND GROUPING(p.CategoryId) = 0
            THEN 'Year Total'
        WHEN GROUPING(p.CategoryId) = 1 AND GROUPING(YEAR(o.OrderDate)) = 0
            THEN 'Year Total (all categories)'
        WHEN GROUPING(p.CategoryId) = 1 AND GROUPING(YEAR(o.OrderDate)) = 1
            THEN 'Grand Total'
        ELSE ''
    END AS SectionLabel,
    YEAR(o.OrderDate) AS OrderYear,
    FORMAT(SUM(oi.Quantity * oi.UnitPrice * (1 - oi.DiscountPct / 100)),
           'C', 'en-US') AS Revenue,
    SUM(oi.Quantity) AS TotalUnits
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY ROLLUP(p.CategoryId, YEAR(o.OrderDate))
ORDER BY
    GROUPING(p.CategoryId), p.CategoryId,
    GROUPING(YEAR(o.OrderDate)), YEAR(o.OrderDate);
```

## Gotchas and Production Pitfalls

### Gotcha 1: NULL Collision — Real NULLs vs ROLLUP NULLs

**Pitfall:** The developer does not use GROUPING() to distinguish real NULL data from ROLLUP subtotal NULL placeholders.

```sql
-- ❌ Wrong: Cannot distinguish real NULL from subtotal NULL
SELECT p.CategoryId, SUM(oi.Quantity) AS UnitsSold
FROM dbo.Products AS p
INNER JOIN dbo.OrderItems AS oi ON p.ProductId = oi.ProductId
GROUP BY ROLLUP(p.CategoryId);
-- If CategoryId IS NULL for uncategorized products, the subtotal row
-- for those products is indistinguishable from a ROLLUP grand total
```

**Symptom:** The application displays "Grand Total" for uncategorized products' subtotal row, or omits the grand total because it is merged with a real NULL group. Users see incorrect totals in the report.

**Fix:**

```sql
-- ✅ Correct: Use GROUPING() to classify each row
SELECT
    CASE WHEN GROUPING(p.CategoryId) = 1 THEN 'Grand Total'
         WHEN p.CategoryId IS NULL THEN 'Uncategorized'
         ELSE CAST(p.CategoryId AS VARCHAR(10)) END AS CategoryLabel,
    SUM(oi.Quantity) AS UnitsSold
FROM dbo.Products AS p
INNER JOIN dbo.OrderItems AS oi ON p.ProductId = oi.ProductId
GROUP BY ROLLUP(p.CategoryId);
```

**Cost of not fixing:** Production reports display incorrect totals. Finance team makes decisions based on wrong data. The bug is subtle and may go unnoticed for months. Once discovered, history must be recalculated.

### Gotcha 2: HAVING Eliminates Subtotal Rows

**Pitfall:** A HAVING clause on the aggregate value removes subtotal rows that have aggregate values below the threshold.

```sql
-- ❌ Wrong: HAVING removes category subtotals below threshold
SELECT p.CategoryId, YEAR(o.OrderDate) AS OrderYear,
       SUM(oi.Quantity) AS UnitsSold
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
GROUP BY ROLLUP(p.CategoryId, YEAR(o.OrderDate))
HAVING SUM(oi.Quantity) > 1000;
-- A category with 5000 total units but 800 units in January:
-- January detail row is removed (800 < 1000)
-- Category subtotal row is preserved (5000 > 1000)
-- BUT: if total category units = 900 (< 1000), the category subtotal
-- is removed even though it represents the category total!
```

**Symptom:** The report shows category subtotals only for categories above the threshold but shows all months for those categories — a confusing mix of filtered and unfiltered data.

**Fix:**

```sql
-- ✅ Correct: Use GROUPING() to preserve subtotal rows
SELECT p.CategoryId, YEAR(o.OrderDate) AS OrderYear,
       SUM(oi.Quantity) AS UnitsSold
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
GROUP BY ROLLUP(p.CategoryId, YEAR(o.OrderDate))
HAVING (GROUPING(p.CategoryId) = 1)   -- always include grand total
    OR (GROUPING(YEAR(o.OrderDate)) = 1)  -- always include category subtotals
    OR SUM(oi.Quantity) > 1000;  -- filter only detail rows
```

**Cost of not fixing:** The monthly report shows category totals that do not match the sum of their months. An auditor notices the discrepancy and flags the report as unreliable.

### Gotcha 3: Missing Covering Index Causes Sort Spill to Tempdb

**Pitfall:** The developer runs ROLLUP on a large table without an index that supports the ROLLUP column order.

```sql
-- No covering index on (CategoryId, OrderDate)
SELECT p.CategoryId, YEAR(o.OrderDate) AS OrderYear,
       SUM(oi.Quantity) AS UnitsSold
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
WHERE o.OrderDate >= '2024-01-01'
GROUP BY ROLLUP(p.CategoryId, YEAR(o.OrderDate));
```

**Symptom:** The execution plan shows a Sort operator between the data access and the Segment/Split/Aggregate. The Sort requires a memory grant proportional to the number of rows. On a 10M row result set with wide rows, the Sort needs ~2GB of memory. If the memory grant is insufficient, the Sort spills to tempdb. The query runs 10× slower and consumes tempdb I/O bandwidth, potentially affecting other queries. Watch the `sort_warnings` extended event or check `sys.dm_exec_query_stats` for `max_spills > 0`.

**Fix:**

```sql
-- ✅ Create covering index in ROLLUP column order
CREATE INDEX IX_Products_CategoryId
    ON dbo.Products (CategoryId)
    INCLUDE (ProductName);

-- Or better: create a view with pre-computed ordered data
-- or use a columnstore index for large aggregation workloads
CREATE NONCLUSTERED COLUMNSTORE INDEX IX_Orders_ColumnStore
    ON dbo.Orders (OrderDate, CustomerId, Status)
    INCLUDE (TotalAmount);
```

**Cost of not fixing:** A 10M-row ROLLUP query that completes in 5 seconds with a covering index takes 45+ seconds when spilling to tempdb. At 20 executions/hour, the extra tempdb I/O (500MB to 2GB per execution) degrades performance for all concurrent queries.

### Gotcha 4: WITH ROLLUP vs GROUP BY ROLLUP() — Old Syntax Issues

**Pitfall:** Using the deprecated `WITH ROLLUP` syntax, which does not support partial rollups and has subtle differences in behavior with compound GROUP BY columns.

```sql
-- Old syntax (WITH ROLLUP)
SELECT p.CategoryId, YEAR(o.OrderDate) AS OrderYear,
       SUM(oi.Quantity) AS UnitsSold
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
GROUP BY p.CategoryId, YEAR(o.OrderDate) WITH ROLLUP;
-- WITH ROLLUP applies to ALL GROUP BY columns — cannot do partial rollup

-- New syntax (GROUP BY ROLLUP())
SELECT p.CategoryId, YEAR(o.OrderDate) AS OrderYear,
       SUM(oi.Quantity) AS UnitsSold
FROM ...
GROUP BY p.CategoryId, ROLLUP(YEAR(o.OrderDate));
-- Partial rollup: only YEAR is rolled up, CategoryId is fixed
```

**Symptom:** The old syntax rolls up all GROUP BY columns, which may produce more subtotal rows than needed. The new syntax allows fine-grained control. The old syntax may also produce incorrect results with compound GROUP BY columns that include computed expressions — the new syntax handles these correctly.

**Fix:** Use `GROUP BY ROLLUP(col1, col2)` standard syntax instead of `WITH ROLLUP`. The old syntax is deprecated and will eventually be removed.

**Cost of not fixing:** Future SQL Server version may remove `WITH ROLLUP` support, forcing an emergency migration. The query produces incorrect subtotals with computed GROUP BY columns.

### Gotcha 5: ORDER BY Without GROUPING() Produces Non-Hierarchical Output

**Pitfall:** The developer orders by the raw columns without handling NULLs from ROLLUP, resulting in subtotal and grand total rows appearing at the top or in random positions.

```sql
-- ❌ Wrong: ORDER BY without GROUPING()
SELECT p.CategoryId, YEAR(o.OrderDate) AS OrderYear,
       MONTH(o.OrderDate) AS OrderMonth,
       SUM(oi.Quantity) AS UnitsSold
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
GROUP BY ROLLUP(p.CategoryId, YEAR(o.OrderDate), MONTH(o.OrderDate))
ORDER BY p.CategoryId, OrderYear, OrderMonth;
-- Grand total row has NULL in CategoryId → appears first (NULL sorts before values)
-- Year subtotals interleave with month details
```

**Symptom:** Inconsistent output order. The grand total appears at the top. Year subtotals are mixed with month-level rows.

**Fix:**

```sql
-- ✅ Correct: Use GROUPING() in ORDER BY to control subtotal placement
ORDER BY
    CASE WHEN GROUPING(p.CategoryId) = 1 THEN 1 ELSE 0 END,
    p.CategoryId,
    CASE WHEN GROUPING(OrderYear) = 1 THEN 1 ELSE 0 END,
    OrderYear,
    CASE WHEN GROUPING(OrderMonth) = 1 THEN 1 ELSE 0 END,
    OrderMonth;
```

**Cost of not fixing:** The report output is confusing — the grand total appears in the middle of the data. Users manually search for totals, increasing cognitive load and the risk of misinterpretation.

### Gotcha 6: ROLLUP with DISTINCT Aggregates Causes Overcounting

**Pitfall:** Using `COUNT(DISTINCT column)` in a ROLLUP query can produce incorrect subtotals because SQL Server may double-count distinct values that appear in different groups.

```sql
-- ❌ COUNT(DISTINCT) in ROLLUP may be inaccurate
SELECT p.CategoryId, YEAR(o.OrderDate) AS OrderYear,
       COUNT(DISTINCT o.CustomerId) AS UniqueCustomers
FROM dbo.Orders AS o
INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
GROUP BY ROLLUP(p.CategoryId, YEAR(o.OrderDate));
-- The category subtotal COUNT(DISTINCT CustomerId) should count
-- unique customers across all months in that category.
-- SQL Server may compute this correctly in the Stream Aggregate
-- (it stores the distinct set per grouping set), but the cost
-- is high and errors are possible with very large distinct sets
```

**Symptom:** Subtotals for COUNT(DISTINCT) are higher than the actual distinct count because the Stream Aggregate cannot deduplicate across grouping sets. The grand total may show the sum of subtotals rather than the true distinct count.

**Fix:** Pre-compute distinct counts at the detail level and apply ROLLUP only to SUM/COUNT aggregates, or compute COUNT(DISTINCT) separately:

```sql
-- ✅ Better: Pre-aggregate distinct counts
WITH CustomerCategory AS (
    SELECT DISTINCT
        p.CategoryId,
        YEAR(o.OrderDate) AS OrderYear,
        o.CustomerId
    FROM dbo.Orders AS o
    INNER JOIN dbo.OrderItems AS oi ON o.OrderId = oi.OrderId
    INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
    WHERE o.OrderDate >= '2024-01-01'
)
SELECT CategoryId, OrderYear, COUNT(CustomerId) AS UniqueCustomers
FROM CustomerCategory
GROUP BY ROLLUP(CategoryId, OrderYear);
```

**Cost of not fixing:** Reports show impossible numbers — the sum of monthly unique customers exceeds the annual unique customer count. Stakeholders distrust the reporting system.

### Spoken Answers

**Q: What is ROLLUP and what problem does it solve?**

> **Average answer:** ROLLUP is a way to get subtotals in SQL. It adds extra rows with subtotals at different levels.

> **Great answer:** ROLLUP is a grouping set extension that generates hierarchical subtotals and a grand total in a single query execution. The problem it solves is the manual UNION ALL pattern where you must write separate GROUP BY queries for each level of the hierarchy — one for the detail (GROUP BY col1, col2, col3), one for the first subtotal (GROUP BY col1, col2), one for the second subtotal (GROUP BY col1), and one for the grand total (no GROUP BY). That UNION ALL approach requires N+1 full table scans for N hierarchy levels, which is wasteful and verbose. ROLLUP does it in a single scan. Internally, the engine uses a Segment operator to identify group boundaries in the sorted data, a Split operator to duplicate each row for each grouping set level, and a Stream Aggregate to compute the aggregates for all levels in one pass. For a 3-level hierarchy on a 10M row table, ROLLUP reduces logical reads by 75% compared to UNION ALL — from ~100K to ~25K. The key insight is that the number of scans in UNION ALL equals the number of hierarchy levels plus one, while ROLLUP always scans once. This difference grows linearly with the hierarchy depth.

**Q: Compare ROLLUP vs CUBE vs GROUPING SETS.**

> **Average answer:** ROLLUP gives hierarchical subtotals, CUBE gives all combinations, and GROUPING SETS gives custom combinations. They are all grouping set extensions.

> **Great answer:** These three are variations of the grouping sets functionality in SQL. ROLLUP generates a hierarchical subset of all possible grouping sets — for N columns, it produces N+1 grouping sets representing a drill-down hierarchy. For ROLLUP(col1, col2, col3), the sets are (1,2,3), (1,2), (1), (). CUBE generates all 2^N possible combinations — every subset of the N columns. For CUBE(col1, col2, col3), that is all 8 grouping sets: (1,2,3), (1,2), (1,3), (2,3), (1), (2), (3), (). GROUPING SETS lets you specify exactly which combinations you want — you can list any subset of the possible grouping sets. The performance and result set size follow these patterns: ROLLUP produces O(N) additional rows, CUBE produces O(2^N) additional rows (exponential — dangerous with many columns), GROUPING SETS produces exactly the rows you specify. The execution plans differ: ROLLUP uses a single Segment/Split/Stream Aggregate pipeline; CUBE may use multiple aggregate passes or a more complex pipeline; GROUPING SETS is implemented as a concatenation of separate aggregate operators, one per grouping set. The optimizer can share scans across grouping sets in GROUPING SETS, but not as efficiently as ROLLUP's single-pass pipeline. The choice is simple: use ROLLUP for hierarchical drill-down reports, CUBE for multi-dimensional analysis, and GROUPING SETS for custom report layouts that don't fit either pattern.

**Q: How does SQL Server execute ROLLUP internally?**

> **Average answer:** It sorts the data and then computes the aggregates at each level.

> **Great answer:** The execution plan for ROLLUP has four key operators after data access. First, a **Sort** operator orders the data by the ROLLUP columns in hierarchy order (col1, col2, col3). If a covering index exists with the same leading column order, the Sort is eliminated and an ordered Index Scan is used. Second, a **Segment** operator scans the sorted data and identifies group boundaries — it emits a new segment every time any of the grouping columns change value. The Segment tracks the current and previous values of each column to detect boundaries. Third, a **Split** operator duplicates each input row for each grouping set that the row belongs to. A detail row at level (col1, col2, col3) is duplicated for all four grouping sets. The Split tags each copy with a grouping set identifier. Fourth, a **Stream Aggregate** groups the split rows by (grouping set + grouping columns) and computes the requested aggregates. Because the data is sorted by the hierarchy and the Split tagged each copy, the Stream Aggregate can compute SUM, COUNT, AVG for all grouping sets in a single pass. The total rows flowing through the Split is approximately (N+1) × input_rows for N hierarchy columns — a significant amplification of the row count in memory. This is why insufficient memory grant for the Split output can cause spills. The entire plan is a single pass over the data — no separate scans for each hierarchy level. The cost is concentrated in the Sort (if needed) and the Split/Stream Aggregate combination. Without a covering index, the Sort operator is typically the most expensive part of the plan, potentially requiring hundreds of megabytes of memory grant.

### Application Checklist

- [ ] The query requires hierarchical subtotals (group by A, then A+B, then A+B+C, then grand total)
- [ ] ROLLUP is preferred over UNION ALL of separate GROUP BY queries (single scan vs N+1 scans)
- [ ] GROUPING() function is used to distinguish real NULLs from subtotal NULL placeholders
- [ ] HAVING clause uses GROUPING() to preserve subtotal rows
- [ ] ORDER BY uses GROUPING() to place subtotals in logical order
- [ ] A covering index exists with leading columns matching the ROLLUP hierarchy order
- [ ] Memory grant estimate is sufficient for the Sort operator (or Sort is eliminated by ordered index)
- [ ] Result set size is within acceptable limits for the consumer (API, report, file export)
- [ ] COUNT(DISTINCT) in ROLLUP is validated for correctness (may overcount)
- [ ] EF Core uses FromSqlRaw — cannot use LINQ for ROLLUP

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|Single scan vs N+1 scans for N hierarchy levels|Result set is larger than simple GROUP BY (contains subtotals)|
|75-80% reduction in logical reads|Sort operator memory grant (if no covering index)|
|Reduced query complexity vs UNION ALL|GROUPING() function adds complexity to SELECT and ORDER BY|
|Built-in NULL placeholders for subtotals|Real NULL data collides with subtotal NULLs — must use GROUPING()|

### Scale Thresholds

- **Relevant when table exceeds ~100K rows** — Below this, UNION ALL of separate GROUP BY queries is fast enough that the optimization is not critical.
- **Critical when table exceeds ~1M rows with 3+ hierarchy levels** — The Sort operator memory grant becomes a concern. Without a covering index, the Sort may spill to tempdb.
- **Required when query runs more than ~1000x/day** — The I/O savings from a single scan justify creating a covering index for the ROLLUP hierarchy, even on moderately sized tables.
- **Exponential risk with CUBE** — Do not use ROLLUP as a CUBE substitute; if you need all combinations, use CUBE but limit to ≤5 columns (2^5 = 32 grouping sets max).

You are building a shipping analytics report. The `ShipmentDetails` table has columns `ShipCountry`, `ShipState`, `ShipCity`, `ShipDate`, and `Carrier`.
```sql
SELECT
    CASE WHEN GROUPING(s.ShipCountry) = 1 THEN 'ALL COUNTRIES'
         ELSE s.ShipCountry END AS Country,
    CASE WHEN GROUPING(s.ShipState) = 1 THEN 'ALL STATES'
         ELSE s.ShipState END AS State,
    CASE WHEN GROUPING(s.ShipCity) = 1 THEN 'ALL CITIES'
         ELSE s.ShipCity END AS City,
    COUNT(*) AS TotalShipments,
    SUM(s.ShipWeight) AS TotalWeight
FROM dbo.ShipmentDetails AS s
WHERE s.ShipDate >= '2024-01-01'
  AND s.ShipDate < '2025-01-01'
GROUP BY ROLLUP(s.ShipCountry, s.ShipState, s.ShipCity)
ORDER BY
    CASE WHEN GROUPING(s.ShipCountry) = 1 THEN 1 ELSE 0 END,
    s.ShipCountry,
    CASE WHEN GROUPING(s.ShipState) = 1 THEN 1 ELSE 0 END,
    s.ShipState,
    CASE WHEN GROUPING(s.ShipCity) = 1 THEN 1 ELSE 0 END,
    s.ShipCity;
```

<details>
<summary>Solution</summary>

**Root cause:** The Sort operator spills to tempdb because there is no covering index in the ROLLUP hierarchy order. The Sort needs to order 50M rows by 6 columns (DepartmentId, DepartmentName, CategoryId, CategoryName, Year, Month) — this requires ~2GB of memory grant. When the memory grant is insufficient, the sort spills to tempdb, causing the 35-second runtime.

**Index to create:**

```sql
-- Create a covering columnstore index for the aggregation
-- Columnstore indexes are optimized for large aggregation workloads
CREATE NONCLUSTERED COLUMNSTORE INDEX IX_Orders_Rollup_Covering
    ON dbo.OrderItems (OrderId, ProductId, Quantity, UnitPrice);

-- Or create a traditional covering index in ROLLUP order:
CREATE INDEX IX_Orders_Rollup_OrderDate
    ON dbo.Orders (OrderDate)
    INCLUDE (CustomerId, Status, TotalAmount);

-- The most impactful index would be on the fact table (OrderItems)
-- ordered by the join columns, with all aggregation columns included:
CREATE INDEX IX_OrderItems_Rollup_Covering
    ON dbo.OrderItems (OrderId, ProductId)
    INCLUDE (Quantity, UnitPrice);
```

```sql
CREATE INDEX IX_Orders_Rollup_DateCols
    ON dbo.Orders (OrderYear, OrderMonth)
    INCLUDE (CustomerId, TotalAmount, Status);
```

<summary>Solution</summary>

Both queries produce identical execution plans in SQL Server 2022. The optimizer normalizes both syntax forms to the same internal representation. However, there are subtle differences:

1. **GROUP BY ROLLUP()** (standard) treats the ROLLUP as applying only to the specified columns. If you write `GROUP BY CategoryId, ROLLUP(YEAR(OrderDate))`, the rollup applies only to `YEAR(OrderDate)`, not to `CategoryId`.

2. **WITH ROLLUP** (old) applies to ALL columns in the GROUP BY clause. You cannot do partial rollups with the old syntax. Additionally, `WITH ROLLUP` can produce unexpected results with compound GROUP BY columns or computed expressions. The old syntax also has issues with GROUPING() — `GROUPING(YEAR(OrderDate))` may return unexpected values.

The plan shape for both queries on a table with no covering index:
1. **[Clustered Index Scan]** Orders — reads all rows (filtered by date)
2. **[Compute Scalar]** — computes `YEAR(OrderDate)` for each row
3. **[Sort]** — sorts by CategoryId, YEAR(OrderDate)
4. **[Segment]** — identifies group boundaries
5. **[Split]** — duplicates for 3 grouping sets
6. **[Stream Aggregate]** — computes SUM(TotalAmount) per grouping set
7. **[SELECT]**

Estimated cost is identical. However, if the query is more complex with multiple GROUP BY columns and expressions, the old `WITH ROLLUP` syntax can cause the optimizer to fail to push predicates or use indexed views. The standard syntax is always preferred.

**Key difference:** The standard syntax allows `GROUP BY CategoryId, ROLLUP(YEAR(OrderDate))` — a partial rollup that only rolls up the date, preserving CategoryId in all rows. The old syntax `WITH ROLLUP` always applies to all GROUP BY columns and cannot produce partial rollups.

</details>

**Challenge 4 — Diagnose the concurrency problem**

A ROLLUP reporting query runs every hour and aggregates 50M rows of order data. Recently, the query started failing with error 8655 "The query processor could not produce a query plan because a worktable is needed and its minimum row size exceeds the maximum allowed." The query uses `GROUP BY ROLLUP(col1, col2, col3, col4, col5)` with 5 hierarchy levels and many VARCHAR columns in the GROUP BY. What is the problem and how do you fix it?

<details>
<summary>Solution</summary>

**Root cause:** The Split operator duplicates each input row for each grouping set. With 5 hierarchy columns, each row is duplicated 6 times (for 6 grouping sets). If the row width is large (many VARCHAR columns in GROUP BY), the combined width of all duplicates exceeds the SQL Server worktable row size limit of 8060 bytes. The Split operator needs to create a worktable in tempdb that holds the duplicated rows, and the worktable row exceeds the page size limit.

This is a known limitation of the Split operator for ROLLUP with wide rows and many hierarchy levels. The error 8655 specifically indicates that the query processor needs a worktable but the minimum row size exceeds the 8060-byte page limit.

**Fix:**

Option 1: **Reduce the number of hierarchy levels** — split into two ROLLUP queries and UNION them. Roll up 3 levels in one query and 2 levels in another.

```sql
-- First query: roll up the first 3 levels
SELECT col1, col2, col3, NULL AS col4, NULL AS col5, SUM(measure)
FROM Table
GROUP BY ROLLUP(col1, col2, col3)
UNION ALL
-- Second query: include all 5 levels but only 2 in ROLLUP
SELECT col1, col2, col3, col4, col5, SUM(measure)
FROM Table
GROUP BY col1, col2, col3, ROLLUP(col4, col5);
```

Option 2: **Use columnstore index** — columnstore compression reduces row width significantly.

Option 3: **Replace ROLLUP with separate GROUP BY queries and UNION ALL** — this avoids the Split operator entirely because each GROUP BY query uses its own aggregation pipeline.

Option 4: **Shorten the column widths** — convert VARCHAR(500) to VARCHAR(50) or use surrogate keys (INT) instead of VARCHAR grouping columns.

**Cost of not fixing:** The query fails every hour. The reporting system shows an error to users. An urgent workaround using UNION ALL is deployed, but it runs 3× slower (6 separate scans instead of 1). The root cause fix is to reduce the hierarchy depth or column widths.

</details>

**Challenge 5 — Design the index**

**Scenario:** A `Sales` fact table has 200M rows with columns `SaleDate`, `ProductId`, `StoreId`, `CustomerId`, `Quantity`, `UnitPrice`, `DiscountAmount`. The most important daily reporting query uses `GROUP BY ROLLUP(StoreId, ProductId, YEAR(SaleDate), MONTH(SaleDate))` and aggregates SUM(Quantity), SUM(Quantity * UnitPrice - DiscountAmount). The query runs 500 times per day and must complete within 5 seconds. Design the optimal index strategy.

<details>
<summary>Solution</summary>

```sql
-- Primary covering index in ROLLUP column order to eliminate Sort:
-- The index order must match the ROLLUP hierarchy: StoreId → ProductId → Year → Month
CREATE NONCLUSTERED INDEX IX_Sales_Rollup_Covering
    ON dbo.Sales (StoreId, ProductId, SaleDate)
    INCLUDE (Quantity, UnitPrice, DiscountAmount, CustomerId);

-- Why this index:
-- 1. StoreId and ProductId are the first two columns, matching ROLLUP hierarchy
-- 2. SaleDate is third — YEAR(SaleDate) and MONTH(SaleDate) are computed from this
--    (the Sort operator sorts by YEAR(SaleDate), MONTH(SaleDate) which approximates SaleDate order)
-- 3. INCLUDE covers all aggregate and measure columns without key lookups
-- 4. CustomerId is included (not in query but useful for other reporting queries)

-- Alternative: Columnstore index (best for aggregation workloads)
CREATE NONCLUSTERED COLUMNSTORE INDEX IX_Sales_ColumnStore
    ON dbo.Sales (SaleDate, StoreId, ProductId, CustomerId)
    INCLUDE (Quantity, UnitPrice, DiscountAmount);
-- Columnstore eliminates the Sort entirely and provides 5-10× compression.
-- The batch execution mode processes ROLLUP ~4× faster than row mode.

-- Supporting index for the date filter:
-- If queries also filter by SaleDate range, this helps narrow the scan
-- before the ROLLUP aggregation
CREATE NONCLUSTERED INDEX IX_Sales_DateFilter
    ON dbo.Sales (SaleDate)
    INCLUDE (StoreId, ProductId, Quantity, UnitPrice, DiscountAmount);
```

**Tradeoffs:**
- **IX_Sales_Rollup_Covering:** Adds ~8GB of storage (200M rows × ~40 bytes/row). Each INSERT adds ~40 bytes of index maintenance. At 10K inserts/hour, that's 400KB/hour write overhead — acceptable for the read benefit.
- **Columnstore index:** Adds ~2GB (compressed ~4×). Batch mode execution reduces query time from ~8 seconds (row mode) to ~1.5 seconds. Write overhead is higher (columnstore uses delta stores for incremental loads) but acceptable for daily batch loads.
- **IX_Sales_DateFilter:** Narrow index on SaleDate only — useful for date filtering before aggregation. Without this, the covering index scan must scan all StoreId/ProductId combinations even when only a date range is needed.

**What NOT to index:** Do not create separate indexes on YEAR(SaleDate) or MONTH(SaleDate) — these are non-SARGable expressions and cannot be used for seeks. Use SaleDate directly. Do not create an index on CustomerId for this query — CustomerId is not in the ROLLUP.

With the columnstore index, the expected execution plan shows:
1. **[Columnstore Index Scan]** — compressed scan of relevant segments
2. **[Segment]** — group boundary detection on ordered StoreId, ProductId, SaleDate
3. **[Split]** — row duplication for 5 grouping sets
4. **[Stream Aggregate]** — batch mode aggregation (columnstore offers batch mode)
5. **[SELECT]
- **Logical reads:** ~15,000 (compressed columnstore segment reads)
- **Elapsed time:** ~1.5 seconds (meets the 5-second SLA)

</details>
