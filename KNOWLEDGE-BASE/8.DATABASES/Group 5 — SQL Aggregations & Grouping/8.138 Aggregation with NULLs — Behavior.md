### Where This Fits

NULL is not a value — it is the absence of a value. SQL uses three-valued logic (TRUE, FALSE, UNKNOWN), and every comparison with NULL using `=` or `<>` returns UNKNOWN, not FALSE. This distinction causes production bugs in aggregation regularly: `COUNT(*)` includes NULL rows but `COUNT(col)` excludes them; `SUM of an all-NULL column` returns NULL, not 0; `AVG` ignores NULLs in both the sum and the count (the denominator is the non-NULL count). A .NET backend engineer encounters this when writing aggregate queries in EF Core — `Sum()` on a nullable column returns `null` in the database, and EF Core maps it to `null` in C#, which then propagates through arithmetic. Interviewers probe this topic because it separates engineers who understand SQL's three-valued logic from those who treat NULL as "a value that means unknown."

## Core Mental Model

In SQL, NULL is the absence of a value, not a value itself. Aggregate functions handle NULLs according to three rules. First, **ignoring NULLs**: `SUM(col)`, `AVG(col)`, `MIN(col)`, `MAX(col)`, and `COUNT(col)` all ignore NULL rows — they only consider non-NULL values in the column. `COUNT(*)` is the exception — it counts every row in the group regardless of NULLs. Second, **propagation through arithmetic**: any arithmetic operation involving NULL produces NULL (`NULL + 5 = NULL`, `NULL * 10 = NULL`). This means `SUM` of an all-NULL column returns NULL, not 0 — SQL Server does not assume a default. Third, **grouping treats NULLs as equal**: all NULLs in the GROUP BY column are placed into the same group (NULL = NULL for grouping purposes, even though `NULL = NULL` is UNKNOWN for comparisons). This means rows with NULL in the ShipCity column are all grouped together into a single group with ShipCity = NULL. COALESCE and ISNULL are used to replace NULL aggregate results with defaults, but must be applied with care — `COALESCE(SUM(col), 0)` is correct, but `SUM(COALESCE(col, 0))` changes the semantics (treats NULL as 0 for the sum, which inflates the result for AVG by including those rows in the denominator).

### Classification

**For SQL topics:** NULL handling belongs to the aggregate function semantics and the GROUP BY clause. NULL-aware aggregates are not SARGable in themselves — SARGability applies to predicates using the WHERE clause, not to aggregate functions. The COALESCE and ISNULL functions in the SELECT list are not SARGable when used in WHERE predicates because they wrap columns. The `IS NULL` and `IS NOT NULL` predicates are SARGable — they are not three-valued (they return TRUE or FALSE, never UNKNOWN). Understanding NULL behaviour in aggregation requires internalising that NULL is not a member of any domain — it is the absence of any value.

```mermaid
flowchart TD
```

## Deep Mechanics

### How the Engine Executes This

**COUNT(*) execution:**

1. The Stream Aggregate or Hash Aggregate operator initialises a counter to 0 for each group.
2. For every input row, regardless of column values, the counter increments by 1.
3. `COUNT(*)` is the only aggregate that never returns NULL for a non-empty group — it always returns the row count.

**COUNT(col) execution:**

1. For each input row, the operator checks whether `col IS NULL`.
2. If the column value is non-NULL, the counter increments.
3. If the column value is NULL, the counter does NOT increment — NULL rows are silently skipped.
4. An all-NULL column produces COUNT = 0 (the counter never incremented).

**SUM(col) execution:**

1. For each input row, the operator reads the column value.
2. If the value is non-NULL, it adds the value to the running accumulator (data type preserving).
3. If the value is NULL, the row is skipped — the accumulator does not change.
4. If no rows have non-NULL values (all-NULL input), the accumulator remains uninitialised and the function returns NULL. If at least one row has a non-NULL value, SUM returns the accumulated value.
5. This means SUM(NULL) is not the same as SUM(0): 0 contributes to the sum, NULL does not. `SUM(column)` where column has values {1, NULL, 3} = 4 (the NULL is skipped, 1 + 3 = 4).

**AVG(col) execution:**

1. For each input row, the operator checks if `col IS NOT NULL`.
2. If non-NULL: add to sum accumulator AND increment count accumulator.
3. If NULL: skip the row entirely — neither sum nor count changes.
4. After processing all rows: `result = sum_accumulator / count_accumulator`.
5. The denominator is the number of non-NULL rows, NOT the total number of rows. This is a common source of incorrect expectations — engineers often expect `AVG(col)` to divide by total rows including NULLs.
6. If the count accumulator is 0 (all-NULL input), AVG returns NULL (division by zero is not attempted).

**MIN(col) / MAX(col) execution:**

1. For each input row, the operator checks if `col IS NOT NULL`.
2. If non-NULL: compare with current min/max using `<` or `>`.
3. If NULL: skip — NULL is not comparable.
4. If all input rows have NULL, no comparison is ever made, and the result is NULL.

**GROUP BY NULL behaviour:**

1. The GROUP BY operator (Stream or Hash) treats all NULL values as equal for grouping purposes.
2. All rows where the GROUP BY column IS NULL are placed into the same group.
3. This is inconsistent with the normal `=` comparison (`NULL = NULL` returns UNKNOWN), but is specified by the SQL standard as the grouping behaviour.

### SQL Visibility

```sql
-- COUNT(*) vs COUNT(col): NULL handling difference
-- Sample: Orders table with 100 rows, 10 rows have NULL ShipCity
SELECT COUNT(*) AS TotalRows,          -- Returns 100
       COUNT(ShipCity) AS NonNullCities, -- Returns 90
       COUNT(*) - COUNT(ShipCity) AS NullRows -- Returns 10
FROM dbo.Orders;

-- SUM with NULL: ignored in sum
-- Values: {10.00, NULL, 30.00, NULL, 50.00}
SELECT SUM(o.TotalAmount) AS TotalAmount  -- Returns 90.00 (10+30+50, NULLs skipped)
FROM dbo.Orders AS o
WHERE o.OrderId IN (1, 2, 3, 4, 5);

-- SUM all-NULL returns NULL
-- If all rows in the query have NULL TotalAmount
SELECT SUM(o.TotalAmount) AS TotalSum  -- Returns NULL
FROM dbo.Orders AS o
WHERE o.CustomerId = -1;  -- No matching orders

-- AVG NULL handling: denominator is non-NULL count
-- Values: {10.00, NULL, 30.00, NULL, 50.00}
SELECT AVG(o.TotalAmount) AS AvgAmount  -- Returns 30.00 = (10+30+50)/3, not 90/5
FROM dbo.Orders AS o
WHERE o.OrderId IN (1, 2, 3, 4, 5);
-- 5 total rows, 3 non-NULL values → 90.00 / 3 = 30.00

-- MIN/MAX: NULLs ignored
-- Values: {100.00, NULL, 50.00, NULL, 75.00}
SELECT MIN(o.TotalAmount) AS MinAmount,  -- Returns 50.00
       MAX(o.TotalAmount) AS MaxAmount   -- Returns 100.00
FROM dbo.Orders AS o
WHERE o.OrderId IN (1, 2, 3, 4, 5);

-- GROUP BY with NULL: all NULLs grouped together
SELECT o.ShipCity,
       COUNT(*) AS OrderCount
FROM dbo.Orders AS o
GROUP BY o.ShipCity;
-- Returns one row per distinct city, PLUS one row where ShipCity IS NULL
-- (all orders with unknown city are aggregated together)

-- COALESCE to replace NULL aggregate results
SELECT COALESCE(SUM(o.TotalAmount), 0) AS TotalAmount  -- Returns 0 instead of NULL
FROM dbo.Orders AS o
WHERE o.CustomerId = -1;

-- ISNULL vs COALESCE for default values
SELECT ISNULL(SUM(o.TotalAmount), 0) AS TotalAmount  -- ISNULL is SQL Server specific
FROM dbo.Orders AS o
WHERE o.CustomerId = -1;
-- COALESCE is ANSI standard, evaluates arguments left to right
-- ISNULL is SQL Server specific, type precedence determined by first argument

-- Aggregation in HAVING with NULL awareness
SELECT o.CustomerId,
       SUM(o.TotalAmount) AS TotalSpent
FROM dbo.Orders AS o
GROUP BY o.CustomerId
HAVING SUM(o.TotalAmount) > 100
   OR SUM(o.TotalAmount) IS NULL;  -- Include customers with all-NULL total
```

### Execution Plan Analysis

**Plan for aggregate with NULL handling:**

```
[Clustered Index Scan Orders — 1M rows]
  → [Hash Match Aggregate]
      GROUP BY: [Orders].CustomerId
      Aggregates:
        COUNT(*)                  — includes NULL rows
        COUNT([ShippedDate])      — excludes NULL ShippedDate rows
        SUM([TotalAmount])        — excludes NULL TotalAmount rows
        AVG([TotalAmount])        — excludes NULL TotalAmount rows (separate SUM + COUNT)
        MIN([OrderDate])          — excludes NULL OrderDate rows
        MAX([OrderDate])          — excludes NULL OrderDate rows
  → [Compute Scalar]             — AVG = SUM / COUNT if needed
  → [SELECT]
```

**Key observation:** The execution plan does NOT show different operators for NULL vs non-NULL aggregates. The NULL handling is built into each aggregate function's implementation within the Stream Aggregate or Hash Aggregate operator. There is no separate filter that removes NULL rows before aggregation — the aggregate function itself checks for NULL internally.

**For AVG specifically:** SQL Server implements AVG as two separate aggregations (SUM + COUNT) within the same operator, then divides in a Compute Scalar. The NULL handling applies to both the sum and the count identically — NULL rows are excluded from both.

### Failure Modes

**COUNT(col) vs COUNT(*) confusion:** The most common NULL-aggregation bug. Engineers write `COUNT(column)` expecting it to count rows, but it excludes NULLs:
```sql
-- Bug: count of orders with a shipped date, not count of orders
SELECT COUNT(o.ShippedDate) AS OrderCount
FROM dbo.Orders AS o
WHERE o.CustomerId = 1001;
-- If CustomerId 1001 has 10 orders but only 8 have ShippedDate set,
-- returns 8, not 10.
-- Fix: use COUNT(*) and COUNT(ShippedDate) to see both
```

**AVG denominator misunderstanding:** Engineers expect `AVG(col)` to divide by total rows, but it divides by non-NULL rows only:
```sql
-- If 100 orders have TotalAmount values and 10 have NULL:
-- Engineer expects: SUM(TotalAmount) / 110
-- SQL Server does:  SUM(TotalAmount) / 100
SELECT AVG(o.TotalAmount) AS AvgAmount FROM dbo.Orders;
-- If you want (sum including NULLs / total rows), use:
SELECT SUM(o.TotalAmount) / COUNT(*) FROM dbo.Orders;
-- But NULL + number = NULL, so dividing sum by count(*) gives wrong result
-- Use COALESCE to replace NULLs with 0:
SELECT AVG(COALESCE(o.TotalAmount, 0)) AS AvgAmountIncludingNulls
FROM dbo.Orders;
-- This changes the denominator! NULL rows are now counted as 0 in AVG.
-- 10 NULL rows now contribute 0 to the sum and 1 to the count.
```

**AVG of empty set:** When the WHERE clause filters out all rows, AVG returns NULL, not 0:
```sql
SELECT AVG(o.TotalAmount) AS AvgAmount
FROM dbo.Orders AS o
WHERE o.OrderDate > '2099-01-01';  -- No matching rows
-- Returns NULL
-- Expecting 0? Add COALESCE:
SELECT COALESCE(AVG(o.TotalAmount), 0) AS AvgAmount
FROM dbo.Orders AS o
WHERE o.OrderDate > '2099-01-01';
```

**NULL in arithmetic expression inside aggregate:** If the SELECT list has an arithmetic expression involving a NULLable column, NULL propagates:
```sql
-- If Discount is NULL, UnitPrice * (1 - NULL) = NULL
-- AVG sees NULL and skips the row (for AVG, not for the row count)
SELECT AVG(oi.UnitPrice * (1 - oi.Discount)) AS AvgDiscountedPrice
FROM dbo.OrderItems AS oi;
-- To include rows with NULL Discount (treat as 0):
SELECT AVG(oi.UnitPrice * (1 - ISNULL(oi.Discount, 0))) AS AvgDiscountedPrice
FROM dbo.OrderItems AS oi;
```

## Production Patterns and Implementation

### Primary SQL Implementation

```sql
-- ============================================================
-- Schema context
-- ============================================================
CREATE TABLE dbo.Orders
(
    OrderId      INT            NOT NULL IDENTITY(1,1),
    CustomerId   INT            NOT NULL,
    OrderDate    DATETIME2(0)   NOT NULL,
    ShippedDate  DATETIME2(0)   NULL,
    Status       VARCHAR(20)    NOT NULL DEFAULT 'Pending',
    TotalAmount  DECIMAL(18,2)  NULL,     -- NULL means not yet calculated
    ShipCity     VARCHAR(100)   NULL,     -- NULL means unknown
    ShipState    VARCHAR(50)    NULL,
    Discount     DECIMAL(18,2)  NULL,     -- NULL means no discount
    Notes        NVARCHAR(MAX)  NULL,
    CONSTRAINT PK_Orders PRIMARY KEY CLUSTERED (OrderId)
);

CREATE TABLE dbo.Customers
(
    CustomerId   INT            NOT NULL IDENTITY(1,1),
    FirstName    NVARCHAR(100)  NOT NULL,
    LastName     NVARCHAR(100)  NOT NULL,
    Email        NVARCHAR(256)  NOT NULL,
    DeletedDate  DATETIME2(0)   NULL,     -- NULL means active account
    CONSTRAINT PK_Customers PRIMARY KEY CLUSTERED (CustomerId)
);

CREATE TABLE dbo.OrderItems
(
    OrderItemId  INT            NOT NULL IDENTITY(1,1),
    OrderId      INT            NOT NULL,
    ProductId    INT            NOT NULL,
    Quantity     INT            NOT NULL,
    UnitPrice    DECIMAL(18,2)  NOT NULL,
    Discount     DECIMAL(18,2)  NULL,
    CONSTRAINT PK_OrderItems PRIMARY KEY CLUSTERED (OrderItemId)
);

-- ============================================================
-- Pattern 1: COUNT(*) vs COUNT(col) — understand the difference
-- ============================================================
SELECT
    COUNT(*) AS TotalOrders,
    COUNT(o.ShippedDate) AS ShippedOrders,       -- NULL ShippedDate excluded
    COUNT(o.ShipCity) AS OrdersWithCity,          -- NULL ShipCity excluded
    COUNT(*) - COUNT(o.ShippedDate) AS UnshippedOrders,  -- difference
    COUNT(*) - COUNT(o.ShipCity) AS OrdersWithoutCity     -- difference
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01';

-- ============================================================
-- Pattern 2: SUM with NULL handling — all-NULL returns NULL
-- ============================================================
-- Safe SUM: returns NULL if no orders exist
SELECT
    o.CustomerId,
    SUM(o.TotalAmount) AS TotalSpent,          -- NULL if no non-NULL amounts
    COALESCE(SUM(o.TotalAmount), 0) AS SafeTotal  -- 0 instead of NULL
FROM dbo.Orders AS o
GROUP BY o.CustomerId;

-- ============================================================
-- Pattern 3: AVG — denominator is non-NULL count
-- ============================================================
SELECT
    o.CustomerId,
    AVG(o.TotalAmount) AS AvgOrderValue,                    -- Non-NULL orders only
    SUM(o.TotalAmount) / COUNT(*) AS AvgWithNullsAsZero,    -- SUM includes NULL? No!
    AVG(COALESCE(o.TotalAmount, 0)) AS AvgTreatingNullAsZero, -- NULL → 0, changes denominator
    SUM(COALESCE(o.TotalAmount, 0)) / COUNT(*) AS ManualAvg  -- Manual: SUM(COALESCE)/COUNT(*)
FROM dbo.Orders AS o
GROUP BY o.CustomerId;

-- ============================================================
-- Pattern 4: MIN/MAX — NULLs ignored
-- ============================================================
SELECT
    o.CustomerId,
    MIN(o.OrderDate) AS FirstOrderDate,       -- Earliest non-NULL date
    MAX(o.OrderDate) AS LastOrderDate,         -- Latest non-NULL date
    MIN(o.ShippedDate) AS FirstShipDate,       -- Earliest non-NULL shipped date
    MAX(o.ShippedDate) AS LastShipDate          -- Latest non-NULL shipped date
FROM dbo.Orders AS o
GROUP BY o.CustomerId;

-- ============================================================
-- Pattern 5: GROUP BY with NULL group key
-- ============================================================
SELECT
    o.ShipCity,
    COUNT(*) AS OrderCount,
    SUM(o.TotalAmount) AS TotalRevenue,
    AVG(o.TotalAmount) AS AvgRevenue
FROM dbo.Orders AS o
GROUP BY o.ShipCity
ORDER BY o.ShipCity;
-- All rows with NULL ShipCity are grouped into one row with ShipCity = NULL

-- ✅ Use COALESCE to name the NULL group
SELECT
    COALESCE(o.ShipCity, 'Unknown') AS ShipCity,
    COUNT(*) AS OrderCount,
    SUM(o.TotalAmount) AS TotalRevenue
FROM dbo.Orders AS o
GROUP BY COALESCE(o.ShipCity, 'Unknown')
ORDER BY ShipCity;

-- ============================================================
-- Pattern 6: NULL handling in HAVING
-- ============================================================
-- Customers who have spent nothing (NULL total) or spent less than $100
SELECT
    o.CustomerId,
    SUM(o.TotalAmount) AS TotalSpent
FROM dbo.Orders AS o
GROUP BY o.CustomerId
HAVING SUM(o.TotalAmount) IS NULL    -- Customers with only NULL TotalAmount orders
    OR SUM(o.TotalAmount) < 100;

-- ============================================================
-- Pattern 7: STRING_AGG with NULL handling
-- ============================================================
-- STRING_AGG ignores NULLs by default
SELECT
    o.CustomerId,
    STRING_AGG(o.ShipCity, ', ') AS CitiesVisited  -- NULLs excluded
FROM dbo.Orders AS o
GROUP BY o.CustomerId;

-- Include NULLs as placeholder text
SELECT
    o.CustomerId,
    STRING_AGG(COALESCE(o.ShipCity, 'Unknown'), ', ') AS CitiesVisited
FROM dbo.Orders AS o
GROUP BY o.CustomerId;

-- ============================================================
-- Pattern 8: Aggregate with empty set — known return values
-- ============================================================
-- All on empty set:
SELECT
    COUNT(*) AS CountResult,         -- 0 (always returns 0)
    COUNT(ShipCity) AS CountCol,     -- 0 (no non-NULL rows)
    SUM(TotalAmount) AS SumResult,   -- NULL (no rows)
    AVG(TotalAmount) AS AvgResult,   -- NULL (no rows)
    MIN(TotalAmount) AS MinResult,   -- NULL (no rows)
    MAX(TotalAmount) AS MaxResult    -- NULL (no rows)
FROM dbo.Orders AS o
WHERE 1 = 0;  -- Empty result set

-- ============================================================
-- Pattern 9: Aggregate in arithmetic with NULL propagation
-- ============================================================
-- If Discount is NULL, UnitPrice * (1 - NULL) = NULL
-- AVG skips NULL rows in UnitPrice * (1 - Discount)
SELECT
    oi.OrderId,
    AVG(oi.UnitPrice * (1 - oi.Discount)) AS AvgDiscountedPrice  -- NULL Discount → NULL → row skipped
FROM dbo.OrderItems AS oi
GROUP BY oi.OrderId;

-- ✅ Use ISNULL to prevent NULL propagation
SELECT
    oi.OrderId,
    AVG(oi.UnitPrice * (1 - ISNULL(oi.Discount, 0))) AS AvgDiscountedPrice
FROM dbo.OrderItems AS oi
GROUP BY oi.OrderId;
```

```csharp
public class NullAwareAggregationService : INullAwareAggregationService
{
    private readonly ApplicationDbContext _dbContext;

    public NullAwareAggregationService(ApplicationDbContext dbContext)
        => _dbContext = dbContext;

    // Pattern 1: COUNT(*) vs COUNT(col) — EF Core translation
    public async Task<List<CustomerStatsDto>> GetCustomerStatsAsync(
        CancellationToken ct = default)
    {
        return await _dbContext.Orders
            .GroupBy(o => o.CustomerId)
            .Select(g => new CustomerStatsDto
            {
                CustomerId = g.Key,
                TotalOrders = g.Count(),                              // COUNT(*)
                ShippedOrders = g.Count(o => o.ShippedDate != null),   // COUNT(CASE WHEN ...)
                UnshippedOrders = g.Count() - g.Count(o => o.ShippedDate != null),
                TotalAmount = g.Sum(o => o.TotalAmount) ?? 0M,         // COALESCE(SUM(...), 0)
                AvgAmount = g.Average(o => o.TotalAmount),             // AVG — nullable!
                FirstOrderDate = g.Min(o => o.OrderDate),
                LastOrderDate = g.Max(o => o.OrderDate)
            })
            .ToListAsync(ct);
        // Generated SQL uses COUNT(CASE WHEN [ShippedDate] IS NOT NULL THEN 1 END)
        // for conditional count, and SUM/AVG with standard NULL handling
    }

    // Pattern 2: GROUP BY with NULL group key — COALESCE the key
    public async Task<List<CitySummaryDto>> GetCitySummariesAsync(
        CancellationToken ct = default)
    {
        return await _dbContext.Orders
            .GroupBy(o => o.ShipCity ?? "Unknown")
            .Select(g => new CitySummaryDto
            {
                City = g.Key,
                OrderCount = g.Count(),
                TotalRevenue = g.Sum(o => o.TotalAmount) ?? 0M,
                AvgRevenue = g.Average(o => o.TotalAmount) ?? 0M
            })
            .OrderByDescending(s => s.TotalRevenue)
            .ToListAsync(ct);
        // Generated SQL: GROUP BY COALESCE([ShipCity], N'Unknown')
        // All NULL ShipCity rows grouped into 'Unknown'
    }

    // Pattern 3: Safe SUM with COALESCE for default
    public async Task<decimal> GetTotalRevenueAsync(
        int customerId,
        CancellationToken ct = default)
    {
        return await _dbContext.Orders
            .Where(o => o.CustomerId == customerId)
            .Select(o => o.TotalAmount)
            .SumAsync(ct);  // Returns 0 if no matching rows (SumAsync on empty set = 0)
        // EF Core's SumAsync returns 0 for empty set (client-side behaviour)
        // But SUM in T-SQL returns NULL. EF Core maps NULL → 0 for nullable sum.
    }

    // Pattern 4: AVG with COALESCE to include NULLs as 0
    public async Task<decimal> GetAvgOrderValueIncludingNullAsync(
        int customerId,
        CancellationToken ct = default)
    {
        var avg = await _dbContext.Orders
            .Where(o => o.CustomerId == customerId)
            .AverageAsync(o => (decimal?)o.TotalAmount ?? 0M);
        return avg ?? 0M;
        // Generated SQL: AVG(COALESCE([o].[TotalAmount], 0.0))
        // Treats NULL TotalAmount as 0 — changes both numerator and denominator
    }

    // Pattern 5: Manual AVG for full control
    public async Task<ManualAvgDto> GetManualAvgAsync(
        CancellationToken ct = default)
    {
        var stats = await _dbContext.Orders
            .GroupBy(o => 1)
            .Select(g => new
            {
                SumAmount = g.Sum(o => o.TotalAmount),
                NullCount = g.Count(o => o.TotalAmount == null),
                TotalRows = g.Count()
            })
            .FirstOrDefaultAsync(ct);

        if (stats == null) return new ManualAvgDto(0, 0, 0);

        return new ManualAvgDto(
            stats.SumAmount ?? 0M,
            stats.NullCount,
            stats.TotalRows);
    }

    // Pattern 6: HAVING with NULL check
    public async Task<List<InactiveCustomerDto>> GetInactiveCustomersAsync(
        CancellationToken ct = default)
    {
        return await _dbContext.Orders
            .GroupBy(o => o.CustomerId)
            .Where(g => g.Sum(o => o.TotalAmount) == null
                     || g.Sum(o => o.TotalAmount) < 100M)
            .Select(g => new InactiveCustomerDto
            {
                CustomerId = g.Key,
                TotalSpent = g.Sum(o => o.TotalAmount) ?? 0M
            })
            .ToListAsync(ct);
        // Generated SQL: HAVING (SUM([TotalAmount]) IS NULL OR SUM([TotalAmount]) < 100.0)
    }
}
```

### SQL Server vs PostgreSQL Differences

```sql
-- PostgreSQL: COUNT(*) and COUNT(col) same as SQL Server
SELECT COUNT(*) AS total_rows, COUNT(shipped_date) AS shipped_rows
FROM orders;

-- PostgreSQL: SUM/AVG/MIN/MAX NULL handling identical
SELECT SUM(total_amount) FROM orders;  -- NULL if all NULL

-- PostgreSQL: GROUP BY NULL behaviour identical
SELECT ship_city, COUNT(*) FROM orders GROUP BY ship_city;

-- PostgreSQL: FILTER clause for conditional aggregation (SQL Server lacks this)
SELECT
    COUNT(*) AS total_orders,
    COUNT(*) FILTER (WHERE shipped_date IS NOT NULL) AS shipped_orders,
    COUNT(*) FILTER (WHERE ship_city IS NULL) AS no_city_orders
FROM orders;
-- Equivalent SQL Server: COUNT(CASE WHEN ShippedDate IS NOT NULL THEN 1 END)

-- PostgreSQL: NULLS FIRST / NULLS LAST in ORDER BY
SELECT ship_city, COUNT(*)
FROM orders
GROUP BY ship_city
ORDER BY ship_city NULLS LAST;  -- NULL group at the end (SQL Server: NULLs are lowest)

-- PostgreSQL: COALESCE vs ISNULL
-- PostgreSQL does not have ISNULL. Use COALESCE.
SELECT COALESCE(SUM(total_amount), 0) AS total FROM orders;

-- PostgreSQL: aggregate on empty set
SELECT
    COUNT(*),       -- 0
    SUM(amount),    -- NULL
    AVG(amount),    -- NULL
    MIN(amount),    -- NULL
    MAX(amount)     -- NULL
FROM orders WHERE 1 = 0;
-- Identical to SQL Server behaviour
```

## Gotchas and Production Pitfalls

### COUNT(*) vs COUNT(col) in Dashboard Metrics

**Pitfall:** Using `COUNT(column)` when you intend to count rows, accidentally excluding rows where that column is NULL. This produces incorrect dashboard metrics that underestimate counts.

```sql
-- ❌ Wrong: counts only rows with non-NULL ShippedDate
-- Dashboard shows 850 orders instead of actual 1,000
SELECT COUNT(o.ShippedDate) AS TotalOrders
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01';
```

**Symptom:** A weekly operations dashboard shows 850 orders shipped, but the actual order count is 1,000. The business team makes staffing decisions based on 850. The 150 unshipped orders are invisible because their ShippedDate is NULL (they haven't shipped yet). The `COUNT(ShippedDate)` excludes them entirely.

**Fix:**

```sql
-- ✅ Correct: use COUNT(*) for total rows
SELECT
    COUNT(*) AS TotalOrders,
    COUNT(o.ShippedDate) AS ShippedOrders,
    COUNT(*) - COUNT(o.ShippedDate) AS UnshippedOrders
FROM dbo.Orders AS o
WHERE o.OrderDate >= '2024-01-01';
```

**Cost of not fixing:** The warehouse manager sees 850 orders and schedules 3 pickers. The actual count is 1,000. The backlog grows by 150 orders per week. After 4 weeks, the SLAs are missed by 600 orders. Customer complaints escalate to the VP of Operations.

---

### AVG Denominator Misunderstanding With NULLs

**Pitfall:** Expecting `AVG(col)` to divide by total rows including NULLs. SQL Server divides by non-NULL rows only, producing a higher average than expected.

```sql
-- ❌ 10 orders: 8 with TotalAmount values, 2 with NULL (pending calculation)
-- Engineer expects: sum of 8 values / 10 = average
-- SQL Server does: sum of 8 values / 8 = average (ignores NULL rows)
SELECT AVG(o.TotalAmount) AS AvgOrderValue FROM dbo.Orders AS o;
-- If sum of 8 values = $800:
-- Expected: $800 / 10 = $80.00
-- Actual: $800 / 8 = $100.00
```

**Symptom:** A pricing analyst reports the average order value is $100. The CFO cross-checks against total revenue / total orders and gets $80. The discrepancy is $20 per order. The analyst spends 3 days investigating "why the reports don't match."

**Fix:**

```sql
-- ✅ Option A: replace NULLs with 0 before averaging (changes denominator)
SELECT AVG(COALESCE(o.TotalAmount, 0)) AS AvgOrderValueIncludingNulls
FROM dbo.Orders AS o;
-- Now: $800 / 10 = $80.00

-- ✅ Option B: compute manually for clarity
SELECT
    SUM(o.TotalAmount) / COUNT(*) AS ManualAvgIncludingNulls,
    AVG(o.TotalAmount) AS AvgExcludingNulls
FROM dbo.Orders AS o;
```

**Cost of not fixing:** The CFO presents $80 average order value at the board meeting based on total revenue / total orders. The analyst insists it's $100. The CEO sees conflicting numbers. The data team is pulled into a 2-hour fire drill. The root cause: the analyst used AVG(TotalAmount), the CFO used SUM(TotalAmount)/COUNT(*).

---

### NULL in Arithmetic Expression Inside Aggregate

**Pitfall:** Writing an arithmetic expression inside an aggregate function where one operand can be NULL. The NULL propagates, and the row is excluded from the aggregate — not just the expression result, but the entire row's contribution.

```sql
-- ❌ If Discount can be NULL: UnitPrice * (1 - NULL) = NULL
-- AVG then skips the entire row (the expression is NULL)
-- The row's Quantity and UnitPrice are completely lost from the average
SELECT AVG(oi.UnitPrice * (1 - oi.Discount)) AS AvgDiscountedPrice
FROM dbo.OrderItems AS oi;
-- If 20% of rows have NULL Discount, those rows contribute nothing to the average
-- The result is biased toward rows WITH discount values
```

**Symptom:** A pricing report shows average discounted price of $45, but the actual average across all line items is $50. The report is missing the 20% of rows where Discount is NULL because the entire expression evaluated to NULL.

**Fix:**

```sql
-- ✅ Use ISNULL or COALESCE on the NULLable column inside the expression
SELECT AVG(oi.UnitPrice * (1 - ISNULL(oi.Discount, 0))) AS AvgDiscountedPrice
FROM dbo.OrderItems AS oi;
-- Now: NULL Discount → 0 → UnitPrice * 1 = UnitPrice (row included correctly)
```

**Cost of not fixing:** A subscription pricing model averages discounted revenue per user. Rows without a discount are excluded from the average because the discount column is NULL. The average is inflated by 20%. The product team sets prices 20% too high, causing a 15% drop in conversion.

---

### COALESCE in GROUP BY — Non-SARGable Group Key

**Pitfall:** Using `COALESCE(column, 'Default')` in the GROUP BY clause to handle NULL group keys. While this works logically, it prevents the optimiser from using an index on the column for grouped aggregation because the expression is not the indexed column.

```sql
-- ❌ COALESCE in GROUP BY — index on ShipCity cannot be used for grouping
SELECT COALESCE(o.ShipCity, 'Unknown') AS ShipCity,
       COUNT(*) AS OrderCount
FROM dbo.Orders AS o
GROUP BY COALESCE(o.ShipCity, 'Unknown');
-- Plan: Clustered Index Scan → Compute Scalar → Hash Match Aggregate
-- Index on ShipCity cannot be used because GROUP BY is on COALESCE(ShipCity, 'Unknown')
```

**Symptom:** Execution plan shows a full clustered index scan and Hash Match Aggregate even though an index on ShipCity exists. The Compute Scalar operator (COALESCE) is evaluated after the scan and before the aggregate.

**Fix:**

```sql
-- ✅ Option A: Keep original column in GROUP BY, COALESCE only in SELECT
SELECT COALESCE(o.ShipCity, 'Unknown') AS ShipCity,
       COUNT(*) AS OrderCount
FROM dbo.Orders AS o
GROUP BY o.ShipCity;  -- Index-friendly
-- Plan: Index Scan → Stream Aggregate → Compute Scalar (COALESCE after aggregate)
-- Index on ShipCity can now be used for ordered scan

-- ✅ Option B: Two-step approach with a computed column
ALTER TABLE dbo.Orders ADD ShipCityDisplay AS COALESCE(ShipCity, 'Unknown');
CREATE INDEX IX_Orders_ShipCityDisplay ON dbo.Orders (ShipCityDisplay);
-- Now GROUP BY ShipCityDisplay can use the index
```

**Cost of not fixing:** A weekly sales report by city processes 50M rows. The COALESCE in GROUP BY prevents index usage. The query runs Hash Aggregate with 450,000 logical reads instead of Stream Aggregate with 10,000. Runtime: 8 minutes instead of 10 seconds. The report misses the 9 AM deadline every Monday.

---

### Empty Set Aggregation — NULL vs 0 Surprise

**Pitfall:** Assuming aggregate functions return 0 for empty result sets. `COUNT` returns 0, but `SUM`, `AVG`, `MIN`, and `MAX` return NULL.

```sql
-- ❌ No orders found for this customer
-- Engineer expects SUM to return 0, but it returns NULL
-- The C# code does: decimal total = result.Sum; -- throws if sum is NULL
SELECT SUM(o.TotalAmount) AS TotalSpent
FROM dbo.Orders AS o
WHERE o.CustomerId = -1;  -- No customer with this ID
```

**Symptom:** The .NET application throws `InvalidOperationException: The cast to value type 'Decimal' failed because the materialized value is null.`. The exception is not handled, and the API returns a 500 error to the client.

**Fix:**

```sql
-- ✅ Always use COALESCE for aggregate results that may return NULL
SELECT COALESCE(SUM(o.TotalAmount), 0) AS TotalSpent
FROM dbo.Orders AS o
WHERE o.CustomerId = -1;

-- ✅ In EF Core, use ?? operator or coalesce in LINQ
var total = await _dbContext.Orders
    .Where(o => o.CustomerId == customerId)
    .Select(o => o.TotalAmount)
    .SumAsync(ct);  -- Returns 0 for empty set (EF Core client-side behaviour)
-- But EF Core 8's SumAsync on IQueryable<decimal?> returns nullable

-- ✅ In Dapper, use nullable types in DTO
public class CustomerTotalDto {
    public int CustomerId { get; set; }
    public decimal? TotalSpent { get; set; }  // NULL-safe
}
```

**Cost of not fixing:** A customer dashboard API returns 500 errors for new customers who have no orders. The frontend shows "Error loading data" instead of "$0.00 spent." The first 100 new customers on day 1 all see errors. Customer support receives 100 tickets. The bug takes 2 hours to diagnose because the developer assumed SUM never returns NULL.

### GROUP BY NULL Behaviour Inconsistency

**Pitfall:** Expecting NULLs in the GROUP BY column to be treated as distinct values (each NULL row in its own group). SQL Server groups all NULLs together because NULL = NULL for grouping purposes, even though NULL = NULL is UNKNOWN for comparisons.

```sql
-- ❌ If 5,000 orders have NULL ShipCity, they all appear in ONE group row
SELECT o.ShipCity, COUNT(*) AS OrderCount
FROM dbo.Orders AS o
GROUP BY o.ShipCity;
-- Output:
-- New York     1,200
-- Los Angeles  1,000
-- NULL         5,000   <-- ALL NULLs grouped together
-- Chicago        800
```

**Symptom:** The report shows "NULL" as a city with 5,000 orders. A business user assumes it's a data entry error for a specific city and reassigns those orders to "Unknown City." Managers in that city are confused because their orders appear under a different label.

**Fix:**

```sql
-- ✅ Use COALESCE to explicitly label the NULL group
SELECT COALESCE(o.ShipCity, 'Unknown') AS ShipCity,
       COUNT(*) AS OrderCount
FROM dbo.Orders AS o
GROUP BY o.ShipCity;  -- GROUP BY on original column, COALESCE in SELECT only

-- ✅ Or exclude the NULL group if it has no business meaning
SELECT o.ShipCity, COUNT(*) AS OrderCount
FROM dbo.Orders AS o
WHERE o.ShipCity IS NOT NULL
GROUP BY o.ShipCity;
```

**Cost of not fixing:** A logistics report shows all orders with NULL ShipCity as a single "Unknown" row with 5,000 orders. The logistics team assumes it's a single warehouse location and sends 5,000 packages to the wrong address. Customer complaints flood in.

### Spoken Answers

**Q: How does COUNT(*) differ from COUNT(col) in terms of NULL handling?**

> **Average answer:** COUNT(*) counts all rows, COUNT(col) counts non-NULL values in the column.

> **Great answer:** COUNT(*) counts every row in the result set regardless of NULLs in any column. It is implemented as a simple increment in the aggregate operator — for each row, the counter increments by 1. COUNT(col) increments its counter only when the column value is non-NULL. This is checked at the row level within the aggregate operator — the operator reads the column value, checks if it's NULL, and conditionally increments. This means COUNT(*) and COUNT(col) can return different values for the same GROUP BY group if the column has NULLs. The performance is nearly identical — the NULL check is negligible CPU cost. The critical production insight: if you write COUNT(ShippedDate) intending to count total orders, you silently lose all orders where ShippedDate is NULL (unshipped orders). The correct pattern for dashboards is: `COUNT(*) AS TotalOrders, COUNT(ShippedDate) AS ShippedOrders, COUNT(*) - COUNT(ShippedDate) AS UnshippedOrders`. In EF Core, `g.Count()` maps to COUNT(*), and `g.Count(o => o.ShippedDate != null)` maps to `COUNT(CASE WHEN [ShippedDate] IS NOT NULL THEN 1 END)`. In Dapper, the same T-SQL applies — use explicit column names in COUNT.

---

**Q: How does GROUP BY handle NULLs in the grouping column?**

> **Great answer:** GROUP BY treats all NULLs as equal for grouping purposes — all rows with NULL in the grouping column are placed into a single group. This is a deliberate exception to SQL's three-valued logic, where `NULL = NULL` evaluates to UNKNOWN (not TRUE). The SQL standard specifies that GROUP BY treats NULLs as equal because grouping requires a deterministic equivalence relation: each row must belong to exactly one group. If NULLs were treated as distinct values, each NULL row would be its own group (which SQL Server does not support; some databases like PostgreSQL offer `NULLS DISTINCT` in ordered-set aggregates, but not in GROUP BY). The production implication: if your ShipCity column has NULLs from data entry gaps, the GROUP BY report shows one "NULL" row with all those orders aggregated together. This is usually incorrect for business reporting — use `COALESCE(ShipCity, 'Unknown')` in the SELECT list only, not in the GROUP BY clause. Keeping the GROUP BY on the original column preserves index usage for Stream Aggregate, and the COALESCE only relabels the output. The generated SQL from EF Core for `GroupBy(o => o.ShipCity ?? "Unknown")` places COALESCE in the GROUP BY if the null-coalescing operator is used in the key selector — to avoid this, group by the original column and apply COALESCE in the Select.

---

**Q: What does an aggregate function return when applied to an empty set (no rows)?**

> **Great answer:** COUNT(*) returns 0 — it's the only aggregate that returns a non-NULL value for an empty set. This is because COUNT(*) counts rows, and zero rows means zero count. COUNT(col) also returns 0 for an empty set (no rows, so no non-NULL values to count). SUM, AVG, MIN, and MAX all return NULL for an empty set. The reasoning: SUM(empty) has no values to sum — NULL represents "no result" rather than "sum of nothing is 0." AVG(empty) would require division by zero if it returned 0, so it returns NULL. MIN and MAX of an empty set have no candidate values, so NULL is returned. This distinction is crucial in .NET applications: `EF Core's SumAsync on IQueryable<decimal?>` returns null for empty sets, but `SumAsync on IQueryable<decimal>` returns 0 (the client-side provider handles the mapping). Dapper maps NULL results to the target type's default if the property is non-nullable, or to null if the property is nullable. The safest pattern is `COALESCE(SUM(column), 0) AS ColumnSum` in SQL, paired with a nullable C# property `decimal? ColumnSum`. In EF Core, use `g.Sum(o => o.TotalAmount) ?? 0M` to provide a default.

### Application Checklist

- [ ] COUNT(*) used for total row count; COUNT(col) used for non-NULL subset count
- [ ] COALESCE or ISNULL wraps aggregate results that may return NULL (SUM, AVG, MIN, MAX on empty sets)
- [ ] COALESCE in SELECT only, not in GROUP BY (preserves index usage for Stream Aggregate)
- [ ] NULL propagation in arithmetic expressions is handled with ISNULL(NULLable_col, 0)
- [ ] AVG denominator understanding: non-NULL count, not total rows
- [ ] GROUP BY with NULL column: all NULLs grouped together — explicit label via COALESCE in SELECT
- [ ] Empty set handling: COUNT returns 0, SUM/AVG/MIN/MAX return NULL
- [ ] EF Core DTO properties are nullable (decimal?, DateTime?) for aggregate results that may be NULL
- [ ] Dapper DTO properties use nullable types for potentially NULL aggregate columns
- [ ] Application code handles null aggregate results (don't assume SUM always returns a value)

### Tradeoff Summary

|What You Gain|What You Pay|
|---|---|
|COALESCE in SELECT: safe defaults, clear output|COALESCE in GROUP BY: prevents index usage (full scan)|
|ISNULL in arithmetic: prevents NULL propagation|ISNULL wraps column — indexes on that column still usable in WHERE?|
|COUNT(*) + COUNT(col): complete picture of NULL distribution|Slightly more complex query (two COUNTs instead of one)|
|Nullable DTO properties: safe null mapping|More null checks in application code|

### Scale Thresholds

- **< 10K rows**: NULL handling performance impact is negligible. Focus on correctness.
- **10K–1M rows**: COALESCE in GROUP BY starts to matter — it forces full scan instead of index seek.
- **> 1M rows**: COALESCE in GROUP BY is catastrophic — forces Hash Aggregate with full scan. Always use COALESCE in SELECT only.
- **> 10M rows**: COUNT(*) vs COUNT(col) difference is ~2% CPU (the NULL check), negligible relative to scan cost.
- **Concurrent queries > 500/sec**: Avoid COALESCE in GROUP BY at all costs — it prevents index usage that would enable Stream Aggregate, causing millions of extra logical reads per second.

## Self-Check

### Conceptual Questions

1. What does COUNT(*) return vs COUNT(col) when the column has NULL rows?
2. What does AVG(col) return when the column has both NULL and non-NULL values — how is the denominator determined?
3. What does SUM(col) return when all rows in the group have NULL for that column?
4. How does GROUP BY treat NULLs — are they grouped together or separately?
5. What does a MIN or MAX function return when applied to an empty set?
6. What is the difference between COALESCE(SUM(col), 0) and SUM(COALESCE(col, 0))?
7. Why does NULL = NULL return UNKNOWN in WHERE but group together in GROUP BY?
8. How does EF Core map a SUM result that might be NULL from the database?
9. How does Dapper handle a NULL SUM result when the DTO property is non-nullable decimal?
10. Explain in 60 seconds to a senior interviewer the NULL behaviour of AVG, including the denominator.

<details>
<summary>Answers</summary>

1. COUNT(*) returns the total number of rows in the group, including rows where the column is NULL. COUNT(col) returns only the number of rows where the column value is non-NULL. For a group of 10 rows where 3 have NULL ShipCity: COUNT(*) = 10, COUNT(ShipCity) = 7.

2. AVG(col) sums only the non-NULL values and divides by the count of non-NULL rows. For values {10, NULL, 20, NULL, 30}: AVG = (10 + 20 + 30) / 3 = 20. The denominator is 3 (non-NULL count), not 5 (total rows). This differs from the common expectation that AVG divides by total rows.

3. SUM(col) returns NULL when all rows in the group have NULL for that column. This is because SUM accumulates non-NULL values, and if no values are non-NULL, the accumulator is never initialised. COALESCE(SUM(col), 0) returns 0 instead.

4. GROUP BY treats all NULLs as equal and groups them together into a single group with a NULL group key. This is a special case in the SQL standard — it violates three-valued logic (NULL = NULL is UNKNOWN in comparisons) but is required for deterministic grouping behaviour.

5. MIN(col) and MAX(col) return NULL when applied to an empty set. They have no non-NULL values to compare, so they return NULL. COUNT(*) returns 0 for empty sets; COUNT(col) also returns 0.

6. COALESCE(SUM(col), 0) first computes SUM(col) with NULL semantics (ignoring NULLs, returning NULL if all-NULL), then replaces a NULL sum result with 0. SUM(COALESCE(col, 0)) replaces each NULL column value with 0 before summing — this changes the sum (0 is added for each NULL row) and also changes AVG's denominator (NULL rows are now counted as rows with value 0). They produce different results.

7. The SQL standard makes an explicit exception for GROUP BY and other set operations: NULLs are treated as equal for grouping purposes. This is because grouping requires every row to be assigned to exactly one group — an equivalence relation. If NULLs were treated as distinct, each NULL row would form its own group (which SQL Server doesn't allow in GROUP BY). In WHERE, the three-valued logic preserves the semantics of unknown comparisons.

8. EF Core's SumAsync on IQueryable<decimal?> returns null for empty sets or all-NULL columns. SumAsync on IQueryable<decimal> returns 0 for empty sets (client-side provider handles it). The generated SQL uses T-SQL SUM which returns NULL. EF Core maps NULL from the database to null in C# for nullable types. For non-nullable types, EF Core throws InvalidOperationException.

9. Dapper attempts to cast the NULL result to the target type. For a non-nullable decimal property, Dapper throws an InvalidCastException. Always use nullable types (decimal?, DateTime?, int?) in Dapper DTOs for aggregate results that may be NULL. Example: `public decimal? TotalSpent { get; set; }`.

10. "AVG in SQL Server computes the average of non-NULL values only. It internally implements two accumulators: a sum accumulator and a count accumulator. For each row where the column value is non-NULL, it adds the value to the sum and increments the count. For NULL values, it skips both. After all rows are processed, it divides the sum by the count. This means the denominator is the number of non-NULL rows, not the total number of rows. If a column has values {1, NULL, 3, NULL, 5}, AVG returns (1+3+5)/3 = 3, not (1+3+5)/5 = 1.8. If all rows are NULL, AVG returns NULL because the count is 0 and division by zero is undefined. The practical implication: if you want the average including NULLs as zeros, use AVG(COALESCE(col, 0)), but be aware this changes the denominator — NULL rows are now counted as rows with value 0."

</details>

**Challenge 2 — Fix the NULL-aggregate bug**

```sql
-- This query is supposed to show average order value per city.
-- The result shows $120 average, but the finance team says it should be $80.
-- There are 10,000 orders; 2,000 have NULL TotalAmount (pending invoices).
-- 500 orders have NULL ShipCity (unknown origin).
```

Identify all the problems and fix them.

<details>
<summary>Solution</summary>

**Problems identified:**

1. **AVG denominator bug:** `AVG(o.TotalAmount)` divides by non-NULL count. 2,000 NULL TotalAmount rows are excluded from the denominator. If total sum of 8,000 non-NULL orders = $960,000, then AVG = $960,000 / 8,000 = $120. But including NULLs as 0: $960,000 / 10,000 = $96. The finance team expects $80? That's likely SUM/COUNT(*) with NULL handling.

2. **COALESCE in GROUP BY:** Prevents index usage on ShipCity. Forces full scan and Hash Aggregate.

3. **COUNT(o.TotalAmount):** Shows count of non-NULL amounts (8,000), not total orders. This is misleading — it looks like only 8,000 orders exist.

4. **NULL City grouped as 'Unknown':** The COALESCE in GROUP BY means 'Unknown' encompasses all 500 NULL-ShipCity orders, but separate from named cities.

```sql

    AVG(o.TotalAmount) AS AvgExcludingNulls,              -- $120 (8K denominator)

    AVG(COALESCE(o.TotalAmount, 0)) AS AvgTreatingNullAsZero, -- $96 (10K denominator)
```

**Challenge 3 — Explain the execution plan**

```sql
-- Query A:
SELECT COALESCE(ShipCity, 'Unknown'), COUNT(*)
FROM Orders
GROUP BY ShipCity;

-- Plan A:
-- [Index Scan IX_Orders_ShipCity] (ordered, 210 logical reads)
-- → [Stream Aggregate]
-- → [Compute Scalar] (COALESCE after GROUP BY)
-- → [SELECT]

-- Query B:
SELECT COALESCE(ShipCity, 'Unknown'), COUNT(*)
FROM Orders
GROUP BY COALESCE(ShipCity, 'Unknown');

-- Plan B:
-- [Clustered Index Scan Orders] (unsorted, 45,000 logical reads)
-- → [Compute Scalar] (COALESCE)
-- → [Hash Match Aggregate]
-- → [SELECT]
```

Why does Query A use Stream Aggregate with 210 reads, while Query B uses Hash Aggregate with 45,000 reads?

<details>
<summary>Solution</summary>

**Why Plan A:** The GROUP BY is on `ShipCity` directly — an indexed column with `IX_Orders_ShipCity`. The index scan provides sorted input, enabling Stream Aggregate. The COALESCE is applied AFTER aggregation in a Compute Scalar, which is a trivial post-processing step. The index is covering (only ShipCity and COUNT(*) needed — COUNT(*) doesn't need any column).

**Why Plan B:** The GROUP BY is on `COALESCE(ShipCity, 'Unknown')` — an expression, not a column. No index exists on this expression. The optimiser cannot use `IX_Orders_ShipCity` because the GROUP BY expression doesn't match the index key. It must scan the entire clustered index (45,000 logical reads), evaluate COALESCE for every row in a Compute Scalar, then build a Hash Aggregate because no sorted input is available.

**What would make Query B use Stream Aggregate:**
- Create a computed column: `ALTER TABLE Orders ADD ShipCityGroup AS COALESCE(ShipCity, 'Unknown');`
- Create an index on it: `CREATE INDEX IX_Orders_ShipCityGroup ON Orders (ShipCityGroup);`
- Then `GROUP BY ShipCityGroup` would use the index and enable Stream Aggregate.

**Tradeoff:** The computed column adds storage (~8 bytes per row for the string hash). For 50M rows, that's ~400 MB. The read savings (45,000 → 210 per query) may justify it if the query runs frequently.

**Challenge 4 — Diagnose the NULL propagation in arithmetic**

A query is supposed to show the average discounted line total per order:

```sql
SELECT oi.OrderId,
       AVG(oi.Quantity * oi.UnitPrice * (1 - oi.Discount)) AS AvgDiscountedLineTotal
FROM dbo.OrderItems AS oi
GROUP BY oi.OrderId;
```

The results are suspicious — the average for Order 1001 is $45, but manually checking the data shows it should be $50. Investigation reveals 2 out of 10 line items in Order 1001 have NULL Discount (no discount applied). Diagnose and fix.

<details>
<summary>Solution</summary>

**Root cause:** When `Discount IS NULL`, the expression `oi.Quantity * oi.UnitPrice * (1 - NULL)` evaluates to NULL. The AVG function excludes NULL rows from both the sum and the denominator. So AVG computes: sum of 8 non-NULL line items / 8 = $45, instead of sum of all 10 line items / 10 = $50. Two rows with NULL Discount are silently excluded from the average.

**Let's verify:**
- 8 line items with Discount: sum = $360, avg = $45
- 2 line items with NULL Discount: unit price $50 each, quantity 1 each, no discount = $50 × 1 = $50 each
- Expected: ($360 + $50 + $50) / 10 = $460 / 10 = $46... wait, the problem says $50.
- Actually: if the 2 NULL-discount rows have UnitPrice $50, Qty 1, then each should contribute $50 to the sum. So expected total = $360 + $50 + $50 = $460, expected avg = $46.

The reported $50 vs expected $50... Let me re-examine. If the 8 discounted rows have an average of $45 (sum $360, 8 rows), and the 2 non-discounted rows have line total $50 each, then the correct average is ($360 + $50 + $50) / 10 = $46. The incorrect average (excluding NULL discount rows) is $360 / 8 = $45. So the report shows $45 instead of $46 — a $1 difference.

**Fix:**

```sql
-- ✅ Replace NULL Discount with 0 in the expression
SELECT oi.OrderId,
       AVG(oi.Quantity * oi.UnitPrice * (1 - ISNULL(oi.Discount, 0))) AS AvgDiscountedLineTotal
FROM dbo.OrderItems AS oi
GROUP BY oi.OrderId;

-- ✅ Or use two-step: handle NULL in a subquery
SELECT OrderId, AVG(LineTotal) AS AvgDiscountedLineTotal
FROM (
    SELECT OrderId,
           Quantity * UnitPrice * (1 - ISNULL(Discount, 0)) AS LineTotal
    FROM OrderItems
) AS prep
GROUP BY OrderId;
```

**After fix:** `$46`. Logical reads: unchanged (same scan). CPU: negligible increase from ISNULL check.

```csharp
            oi.Quantity * oi.UnitPrice * (1 - (oi.Discount ?? 0)))
```

**Query 4 — Product discount (ISNULL in arithmetic):**

```sql
SELECT
    p.CategoryId,
    AVG(p.UnitPrice * (1 - ISNULL(oi.Discount, 0))) AS AvgDiscountedPrice,
    COUNT(*) AS LineItems
FROM dbo.OrderItems AS oi
INNER JOIN dbo.Products AS p ON oi.ProductId = p.ProductId
GROUP BY p.CategoryId;
```
