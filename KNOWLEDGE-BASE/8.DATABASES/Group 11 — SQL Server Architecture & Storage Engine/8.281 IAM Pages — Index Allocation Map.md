- [[8.272 — Extent Structure — Mixed and Uniform Extents]] — IAM pages map extents (64KB units), not individual pages. Understanding extent types (mixed for first 8 pages, uniform thereafter) is required to interpret IAM bitmaps.
- [[8.273 — GAM, SGAM, PFS — Space Management Pages]] — IAM pages work alongside GAM/SGAM/PFS. GAM/SGAM track which extents are free/partially used; IAM tracks which extents belong to a specific object. The distinction is critical for understanding allocation contention.
- [[8.277 — Allocation Units — IN_ROW, ROW_OVERFLOW, LOB]] — each allocation unit has its own IAM chain; a table with LOB columns has at least two IAM chains (IN_ROW and LOB). Understanding this is required to read IAM chains correctly.

### Where This Fits

IAM (Index Allocation Map) pages are the directory that maps which extents belong to which object (table, index, partition). Unlike GAM/SGAM which answer "is this extent free?", IAM answers "which extent belongs to this index's allocation unit?" Every object (heap, clustered index, non-clustered index, indexed view) has at least one IAM chain — a linked list of IAM pages covering the entire data file range. A .NET backend engineer encounters IAM indirectly when they see a Table Scan (Heap) vs an Index Scan (Clustered) in execution plans — the heap scan uses IAM-based allocation order scan, while the clustered index scan uses the leaf-level page chain. The interview signal is moderate: senior engineers must understand how SQL Server maps an object to its pages, why IAM scans skip pages not belonging to the object, and why IAM contention (SGAM contention) affects TempDB performance. The deeper signal is whether the candidate understands the IAM chain structure well enough to diagnose page allocation corruption, rebuild IAM chains, or optimize allocation-heavy workloads.

## Core Mental Model

An IAM page is an 8KB page (PageType 10) that contains a **bitmap** tracking which extents in a 4GB range belong to a specific allocation unit. The bitmap has 64,000 bits (8,000 bytes), where each bit represents one extent (64KB = 8 contiguous pages). Bit = 1 means the extent belongs to the object; bit = 0 means it does not. One IAM page covers a 4GB range of the data file: 64,000 extents × 64 KB/extent = 4,096 MB = 4 GB. If an allocation unit spans more than 4GB, SQL Server adds another IAM page linked into an **IAM chain** — the IAM pages form a doubly-linked list via `NextPage`/`PrevPage` pointers in the IAM page header. The first IAM page for an object is stored in `sys.indexes.first_iam_page` (for the index) or in `sys.allocation_units.first_iam_page` (for the allocation unit). The IAM chain is the **primary access path for heap scans** — SQL Server reads the IAM bitmap and reads each extent whose bit is set, in file order. For B-tree indexes, the IAM chain is used during index rebuild and DBCC CHECKDB operations, but query access uses the B-tree page chain.

### Classification

IAM pages are **space management metadata pages** in the **Storage Engine allocation infrastructure**. They are part of the same category as GAM (Global Allocation Map), SGAM (Shared Global Allocation Map), and PFS (Page Free Space) pages. Unlike GAM/SGAM which are database-level singletons (one GAM per ~4GB, one SGAM per ~4GB), IAM pages are **object-level** — each allocation unit of each index/table has its own IAM chain. IAM pages are **system pages** — they are tracked in `sys.system_internals_allocation_units` and are not directly part of user data. They are cached in the buffer pool and are subject to latch contention (particularly PAGELATCH_UP on the IAM page during allocation). The IAM page structure is **not a B-tree** — it is a flat bitmap + linked list. The bitmap format is fixed: 8,000 bytes, no variable-length data, no slot array (IAM pages do not use the standard offset array).

### Step 11: IAM Page Sequence Number — Detecting Allocation Pattern Changes

Each IAM page has a **sequence number** (8 bytes, stored immediately after the page header) that increments every time the IAM page's bitmap is modified (i.e., every time an extent is allocated to or deallocated from the object). By querying the sequence number over time, you can detect allocation pattern changes:

```sql
DBCC PAGE('YourDatabase', 1, 189, 3);
-- Look for "Sequence Number = 421" in the IAM HEADER section
```

A rapidly incrementing sequence number on an IAM page indicates that the object is growing quickly — new extents are being allocated frequently. This is normal during bulk loads but abnormal during steady-state operations. A spike in sequence number growth rate correlates with:

- Bulk INSERT operations (large batch inserts)
- Index rebuilds (deallocate old pages, allocate new pages)
- Table partitions being switched in/out
- Index reorganize operations (may deallocate merged pages)

### Step 12: IAM Page Deallocation During DROP and TRUNCATE

When a table is dropped or truncated, SQL Server deallocates IAM pages as follows:

1. **TRUNCATE TABLE**: Deallocates all extents belonging to the table. The IAM chain is reset — all IAM page bits are cleared (set to 0). The first IAM page remains allocated but empty. The data file space is returned to the database as free space (GAM/SGAM bits are updated). The table structure remains (schema persists) with zero pages.

2. **DROP TABLE**: Deallocates all extents AND the IAM pages themselves. The IAM chain is completely removed. The `sys.allocation_units` and `sys.indexes` entries for the table are removed. If the drop is within a transaction that rolls back, the entire IAM chain is restored from the log.

3. **DROP INDEX**: For non-clustered indexes, the IAM chain is fully deallocated. For clustered indexes, the table becomes a heap — a new IAM chain for the heap is created from the existing clustered index data pages. The old clustered index IAM chain is replaced.

During TRUNCATE, the deallocation is minimally logged (extent deallocations are logged, but individual page data is not). This is why TRUNCATE is faster than DELETE — DELETE logs every row deletion; TRUNCATE logs only the extent deallocations:

```sql
-- Compare log space used
BEGIN TRAN;
DELETE FROM Sales.OrderHeap;
-- vs
TRUNCATE TABLE Sales.OrderHeap;
-- Check log space with sys.dm_db_log_space_usage before and after
```

### Step 13: IAM and Data Compression Interaction

Data compression (ROW or PAGE level) affects how many rows fit on a data page but does NOT change the IAM page structure. The IAM page still tracks extents at the 64KB granularity — the number of rows per extent increases with compression because more rows fit per 8KB page, but the IAM bitmap is unchanged.

However, compression-related page splits (caused by compression context changes during updates) still require IAM page updates when new extents are needed. A compressed page split that cannot fit the updated row on the current page may need a new page from a new extent, which requires IAM bitmap modification:

```sql
-- Check if IAM sequence number changes during compression rebuild
-- Before rebuild:
DBCC PAGE('YourDatabase', 1, <IAMPID>, 3);  -- Note sequence number

ALTER TABLE Sales.OrderHeap REBUILD WITH (DATA_COMPRESSION = PAGE);

-- After rebuild:
DBCC PAGE('YourDatabase', 1, <NewIAMPID>, 3);  -- Different IAM page, new sequence
```

The IAM page for the compressed table will have the same structure but will map fewer extents (if compression reduces the data footprint) — the total page count decreases but the IAM chain length may also decrease as fewer extents are needed.

### Step 14: sys.dm_db_index_physical_stats vs IAM — Understanding Discrepancies

The `sys.dm_db_index_physical_stats` DMV and the IAM chain should agree on page counts for a healthy object, but discrepancies can occur:

```sql
-- Page count from index physical stats
SELECT 
    object_name(ips.object_id) AS TableName,
    ips.index_id,
    ips.index_level,
    ips.page_count
FROM sys.dm_db_index_physical_stats(DB_ID(), OBJECT_ID('Sales.OrderHeap'), 0, NULL, 'DETAILED')
WHERE ips.index_level = 0;
```

```sql
-- Page count from IAM (via allocation units)
SELECT 
    au.total_pages,
    au.used_pages,
    au.data_pages
FROM sys.allocation_units au
INNER JOIN sys.partitions p ON au.container_id = p.partition_id
WHERE p.object_id = OBJECT_ID('Sales.OrderHeap');
```

Discrepancies can indicate:
- **Pending IAM updates**: The IAM bitmap has not been updated for a recent extent allocation (rare, usually transient)
- **Ghost pages**: Pages that have been deallocated from the IAM but still appear in the object's page enumeration
- **Corruption**: The IAM bitmap no longer matches the actual page allocations — detected by `DBCC CHECKDB`
- **Deferred drop**: An index/table drop that was deferred (due to long-running snapshot transactions) may show pages in IAM that are no longer part of the active object

If `sys.dm_db_index_physical_stats` page_count does not match `sys.allocation_units.total_pages`, run `DBCC CHECKDB` with `EXTENDED_LOGICAL_CHECKS` to verify IAM consistency.

### Step 15: DMF sys.dm_db_database_page_allocations — The Modern IAM Inspection Tool

SQL Server 2012+ provides `sys.dm_db_database_page_allocations` (undocumented but production-safe) as a more convenient alternative to parsing DBCC IND output:

```sql
SELECT 
    object_id,
    index_id,
    partition_id,
    allocation_unit_id,
    allocated_page_file_id,
    allocated_page_page_id,
    page_type_code,
    page_type_description,
    allocation_unit_type_desc,
    extent_file_id,
    extent_page_id,
    is_allocated,
    is_iam_page,
    is_mixed_page_allocation,
    page_free_space,
    page_level
FROM sys.dm_db_database_page_allocations(
    DB_ID(),           -- Database ID
    OBJECT_ID('Sales.OrderHeap'),  -- Object ID (NULL for all objects)
    NULL,              -- Index ID (NULL for all)
    NULL,              -- Partition ID (NULL for all)
    'DETAILED'         -- 'LIMITED' or 'DETAILED'
)
WHERE is_allocated = 1
ORDER BY allocated_page_file_id, allocated_page_page_id;
```

Key columns for IAM analysis:
- `is_iam_page` = 1: This is an IAM page (PageType = 10)
- `page_type_code`: 1=Data, 2=Index, 3=LOB, 4=Overflow, 10=IAM
- `allocation_unit_type_desc`: IN_ROW_DATA, ROW_OVERFLOW_DATA, or LOB_DATA
- `is_mixed_page_allocation`: TRUE for pages in mixed extents (first 8 pages of an object)
- `extent_file_id` / `extent_page_id`: The extent containing this page

This DMF scans the actual page allocations from the boot page's allocation cache and is the most reliable way to enumerate pages without DBCC IND overhead. Use it in diagnostic scripts instead of DBCC IND for server automation.

## Deep Mechanics

### Step 1: View IAM Pages with DBCC IND

```sql
DBCC IND('YourDatabase', 'Sales.OrderHeap', -1);
```

Filter for PageType = 10:
```
PageFID=1, PagePID=189, PageType=10, IndexLevel=0, NextPageFID=0, NextPagePID=0
```

The IAM page's `NextPageFID`/`NextPagePID` may show 0 if this is the only IAM page. For larger tables, the IAM chain has multiple pages.

### Step 2: Inspect IAM Page with DBCC PAGE

### Step 3: Find First IAM Page from Metadata

```sql
SELECT

-- 0x00000000000001BD = PageID 189 in File 1
```

### Step 4: Parse the IAM Chain Structure

Each allocation unit has its own IAM chain. A table with LOB columns has three allocation units:

```sql
SELECT 
    au.allocation_unit_id,
    au.type_desc AS AllocUnitType,
    au.first_iam_page,
    au.total_pages,
    au.used_pages,
    au.data_pages,
    p.partition_number
FROM sys.allocation_units au
INNER JOIN sys.partitions p ON au.container_id = p.partition_id
WHERE p.object_id = OBJECT_ID('Sales.OrderHeap');
```

| allocation_unit_id | AllocUnitType | total_pages | used_pages | data_pages |
|--------------------|---------------|-------------|------------|------------|
| 72057594040549376 | IN_ROW_DATA | 1,000 | 950 | 900 |
| 72057594040614912 | ROW_OVERFLOW_DATA | 50 | 40 | 40 |
| 72057594040680448 | LOB_DATA | 200 | 180 | 180 |

Each of these three allocation units has its own IAM chain. The IAM chain tracks extents containing pages of that specific allocation unit type.

### Step 5: Walk the IAM Chain Using a Script

```sql
-- Walk IAM chain for the IN_ROW_DATA allocation unit
DECLARE @first_iam VARBINARY(6);
SELECT @first_iam = first_iam_page 
FROM sys.allocation_units 
WHERE allocation_unit_id = 72057594040549376;

-- Parse the 6-byte page pointer (2 bytes file, 4 bytes page)
DECLARE @FileID INT = CONVERT(INT, SUBSTRING(@first_iam, 1, 2));
DECLARE @PageID INT = CONVERT(INT, SUBSTRING(@first_iam, 3, 4));

-- Create temp table for IAM chain
CREATE TABLE #IAMChain (PageFID INT, PagePID INT, NextPageFID INT, NextPagePID INT);

-- Read each IAM page using DBCC PAGE and extract NextPage pointer
-- This is a simplified version — production would use a cursor or recursive CTE
INSERT INTO #IAMChain
EXEC ('DBCC IND(''YourDatabase'', ''Sales.OrderHeap'', -1)');  -- Get all pages

SELECT PageFID, PagePID, NextPageFID, NextPagePID, PageType
FROM #IAMChain
WHERE PageType = 10
ORDER BY PagePID;

DROP TABLE #IAMChain;
```

### Step 6: IAM Scan — How SQL Server Reads a Heap

For a heap table scan, SQL Server:

1. Reads the first IAM page from `sys.allocation_units.first_iam_page`.
2. Scans the 8,000-byte bitmap sequentially, checking each bit.
3. For each bit = 1, it reads the corresponding extent (all 8 pages for uniform extents, or the specific page for mixed extents, checking PFS for which pages within the extent are allocated).
4. Continues to the next IAM page in the chain.
5. Returns all data pages belonging to the heap.

This is called an **allocation order scan** (IAM scan). The key property: the scan reads pages in file order (extent order within each IAM range), which may differ from logical key order. The scan only reads pages owned by the object — it skips pages belonging to other objects in the same extent (critical for mixed extents).

```sql
SELECT
```

The `is_iam_page` filter returns all IAM pages. The `extent_file_id` and `extent_page_id` identify which extent each IAM page manages (the start page of the 4GB range).

### Step 8: IAM and Mixed Extents — First 8 Pages

For the first eight pages of an object, SQL Server uses mixed extents (extents shared by up to 7 other objects). The IAM page's bitmap handles this differently — it marks the entire extent as allocated (bit = 1), but the SGAM page tracks which pages within a mixed extent are used by which object. When scanning a mixed extent, SQL Server checks the PFS page to determine which specific pages belong to the object being scanned.

```sql
-- Verify: check if the heap uses mixed extents
SELECT 
    allocated_page_file_id,
    allocated_page_page_id,
    extent_file_id,
    extent_page_id,
    is_mixed_page_allocation
FROM sys.dm_db_database_page_allocations(DB_ID(), OBJECT_ID('Sales.OrderHeap'), NULL, NULL, 'DETAILED')
WHERE is_allocated = 1
  AND is_mixed_page_allocation = 1;
```

For the first 8 pages allocated to a new heap, `is_mixed_page_allocation = 1`. After page 8, `is_mixed_page_allocation = 0` and the extent is dedicated to this object (uniform extent).

### Step 9: IAM Page Contention — PAGELATCH_UP

IAM pages are updated during page allocation. When a new extent is allocated to an object, the IAM page's bitmap is modified (a bit is set from 0 to 1). This requires a PAGELATCH_UP (update latch) on the IAM page. In high-concurrency scenarios (especially TempDB with many temp tables), multiple sessions contend for the same IAM page, causing PAGELATCH_UP waits.

```sql
-- Check IAM latch contention
SELECT 
    DB_NAME(database_id) AS DatabaseName,
    file_id,
    page_id,
    page_type,
    page_level,
    wait_resource,
    wait_type,
    waiting_tasks_count,
    wait_time_ms
FROM sys.dm_os_wait_stats ws
CROSS APPLY sys.dm_db_page_info(DB_ID('TempDB'), 1, NULL, 'DETAILED')  -- Requires SQL Server 2019+
WHERE wait_type LIKE 'PAGELATCH%'
ORDER BY wait_time_ms DESC;
```

PAGELATCH_UP waits on IAM pages are especially pronounced in TempDB and in heaps with high INSERT concurrency.

### Step 10: IAM Corruption and DBCC CHECKDB

When an IAM page is corrupted (incorrect bitmap, broken chain link, orphaned extent reference), DBCC CHECKDB detects it:

```sql
DBCC CHECKDB('YourDatabase') WITH EXTENDED_LOGICAL_CHECKS;
```

Common IAM corruption patterns:
- **Missing IAM page**: An extent is allocated (GAM says used) but no IAM page claims it → orphaned extent.
- **Double-claimed extent**: Two IAM pages mark the same extent as belonging to different objects.
- **Broken IAM chain**: The NextPage pointer in an IAM page points to a non-existent page or a page that is not an IAM page.
- **Mismatched bitmap**: The IAM says an extent belongs to the object, but GAM says the extent is free (deallocated without updating IAM).

These corruptions require replaying from backup or using `DBCC CHECKDB REPAIR_ALLOW_DATA_LOSS`.

## Production Patterns

### Pattern 1: Enumerate All IAM Chains in a Database

```sql
SELECT 
    OBJECT_NAME(p.object_id) AS TableName,
    i.name AS IndexName,
    i.index_id,
    au.type_desc AS AllocUnitType,
    au.first_iam_page,
    au.total_pages,
    au.used_pages,
    p.partition_number,
    FILE_ID(CONVERT(VARBINARY, SUBSTRING(au.first_iam_page, 1, 2), 1)) AS IAM_FileID,
    CONVERT(INT, SUBSTRING(au.first_iam_page, 3, 4), 1) AS IAM_PageID
FROM sys.allocation_units au
INNER JOIN sys.partitions p ON au.container_id = p.partition_id
INNER JOIN sys.indexes i ON p.object_id = i.object_id AND p.index_id = i.index_id
WHERE p.object_id > 100  -- Skip system objects
ORDER BY au.total_pages DESC;
```

This gives every IAM chain and its starting page. The `first_iam_page` is a 6-byte binary (2 bytes file ID + 4 bytes page ID).

### Pattern 2: IAM Scan I/O Analysis

```sql
SET STATISTICS IO ON;

-- Compare IAM scan (heap) vs B-tree scan (clustered index)
SELECT COUNT(*) FROM Sales.OrderHeap;                   -- Heap (IAM scan)
SELECT COUNT(*) FROM Sales.OrdersClustered;             -- Clustered index (B-tree scan)

-- The logical reads for the heap should be close to page count.
-- If logical reads exceed page count by >10%, forwarded records exist.
```

Compare output:
```
Table 'OrderHeap': Scan count 1, logical reads 1420, physical reads 0
Table 'OrdersClustered': Scan count 1, logical reads 1510, physical reads 0
```

For the heap, logical reads = 1420 = number of pages scanned via IAM. For the clustered index, logical reads = 1510 = number of leaf pages + IAM overhead.

### Pattern 3: Detect IAM Contention

```sql
-- SQL Server 2019+ with sys.dm_db_page_info
SELECT 
    session_id,
    wait_type,
    wait_time,
    wait_resource,
    db_info.database_id,
    db_info.file_id,
    db_info.page_id,
    db_info.page_type,
    db_info.object_id,
    OBJECT_NAME(db_info.object_id) AS ObjectName,
    db_info.index_id,
    i.name AS IndexName
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_db_page_info(r.database_id, 
    CONVERT(INT, SUBSTRING(r.wait_resource, 
        CHARINDEX(':', r.wait_resource, CHARINDEX(':', r.wait_resource) + 1) + 1, 
        LEN(r.wait_resource) - CHARINDEX(':', r.wait_resource, CHARINDEX(':', r.wait_resource) + 1))),
    CONVERT(INT, SUBSTRING(r.wait_resource, 
        CHARINDEX(':', r.wait_resource) + 1, 
        CHARINDEX(':', r.wait_resource, CHARINDEX(':', r.wait_resource) + 1) - CHARINDEX(':', r.wait_resource) - 1)),
    'DETAILED') db_info
LEFT JOIN sys.indexes i ON db_info.object_id = i.object_id AND db_info.index_id = i.index_id
WHERE r.wait_type LIKE 'PAGELATCH%'
  AND db_info.page_type = 10;  -- IAM pages only
```

### Pattern 4: IAM Fragmentation — Extent Layout

The IAM bitmap shows which extents belong to an object, but not their physical contiguity. Use `sys.dm_db_index_physical_stats` to measure extent fragmentation:

```sql
SELECT 
    OBJECT_NAME(object_id) AS TableName,
    index_id,
    partition_number,
    avg_fragmentation_in_percent,       -- Logical fragmentation
    fragment_count,
    avg_fragment_size_in_pages,
    page_count
FROM sys.dm_db_index_physical_stats(DB_ID(), OBJECT_ID('Sales.OrderHeap'), NULL, NULL, 'LIMITED')
WHERE index_id = 0;
```

For a heap, `avg_fragmentation_in_percent` is less meaningful than for a clustered index (since heaps have no logical order). However, `avg_fragment_size_in_pages` indicates how many contiguous extents exist — larger fragments mean more efficient IAM scans (fewer I/O operations needed to read all extents).

## Gotchas

### Gotcha 1: IAM Scan Causes Unpredictable Row Order

**Pitfall:** A heap scan via IAM returns rows in extent allocation order, which is the order in which extents were allocated — not insertion order, not key order. Developers assume `SELECT * FROM HeapTable` returns rows in insertion order.

**Symptom:** A QA test asserts that the 10th row inserted is always the 10th row returned. The test passes in development (small table, single extent) and fails in production (large table, multiple extents scattered across file). The production query returns rows in a different order every time if page splits or forwarded records occur between queries.

**Fix:** Always use `ORDER BY` if row order matters. Do not rely on IAM scan order. If order is not specified, the SQL standard does not guarantee row order.

**Cost:** `ORDER BY` requires a sort operator in the execution plan if the column is not indexed. For large heaps, this can be expensive (memory grant, TempDB spill risk). Solution: add a clustered index or a covering non-clustered index on the ordering column.

### Gotcha 2: Mixed Extent Scanning Overhead

**Pitfall:** A small heap (fewer than 8 pages) shares a mixed extent with up to 7 other objects. The IAM bitmap marks the entire extent as allocated (bit = 1), but PFS determines which pages belong to the heap. The IAM scan must check PFS for each potential page in the mixed extent.

**Symptom:** Scanning a small heap (1-2 pages) costs more logical reads than expected. The IAM scan reads the IAM page (1 read) + checks the PFS page (1 read) + reads the data pages (2 reads) = 4 reads for 2 pages of data. For a clustered index of the same size, the scan reads only the 2 data pages (via leaf chain) + 1 IAM read = 3 reads.

**Fix:** For very small tables that are frequently scanned, consider adding a clustered index. The leaf chain eliminates the PFS-checking overhead during scans.

**Cost:** Adding a clustered index changes the physical structure. For a small table, the overhead is negligible. The benefit of eliminating extra PFS reads is marginal at this scale (1 read saved).

### Gotcha 3: IAM Chain Length Impacts Scan Start Time

**Pitfall:** A very large heap (100GB+) may have hundreds of IAM pages in the chain. An IAM scan must read every IAM page in the chain before it can start reading data pages — the IAM scan reads IAM pages sequentially, then reads the extents in the order they appear in each IAM bitmap.

**Symptom:** The first `SELECT *` on a large heap shows high `PAGEIOLATCH_SH` waits on IAM pages, not data pages. The scan appears to "stall" while reading IAM pages before any data is returned. For a 500GB heap with 500 IAM pages, reading the IAM chain adds 500 I/Os before any data is returned.

**Fix:** Convert the heap to a clustered index. The clustered index scan uses the leaf page chain, which does not require pre-reading all IAM pages. Alternatively, use partitioned tables to reduce per-partition IAM chain length.

**Cost:** Adding a clustered index to a 500GB heap is a significant operation (full table rebuild, 2× space, SCH-M lock). Schedule maintenance window. Consider online index create if Enterprise Edition.

### Gotcha 4: Ghost IAM Entries After Dropping Objects

**Pitfall:** When a table/index is dropped, its IAM pages are deallocated and the corresponding GAM/SGAM bits are updated. However, if the drop is rolled back (transaction rollback), the IAM pages are restored. In rare cases, deferred drop operations (e.g., `DROP TABLE` with long-running transactions) leave IAM pages in an intermediate state.

**Symptom:** `sys.allocation_units` references an IAM page for a dropped object. The object appears in `sys.objects` with `is_ms_shipped = 0` but is not visible in Object Explorer. `DBCC CHECKDB` reports orphaned IAM pages.

**Fix:** Run `DBCC CHECKDB` with `EXTENDED_LOGICAL_CHECKS` to detect orphaned IAM pages. Use `DBCC CLEANTABLE` or `ALTER TABLE ... REBUILD` if the object still exists. For fully orphaned IAM chains, restore from backup.

**Cost:** Full `DBCC CHECKDB` with extended checks on a large database can take hours. Orphaned IAM page cleanup is a one-time cost.

### Gotcha 5: Single IAM Page Hot-Spot in TempDB

**Pitfall:** TempDB uses the same IAM infrastructure. When hundreds of temp tables are created and dropped per second, they all contend for the same few IAM pages in the first 4GB range of TempDB data files. PAGELATCH_UP contention on TempDB IAM pages is the #1 cause of TempDB performance issues.

**Symptom:** High `PAGELATCH_UP` waits on TempDB with page IDs in the first few pages of the TempDB data file. `sys.dm_exec_requests` shows many sessions waiting on page `(2:xxx)` where x is a low page number. The waits are on IAM pages (page_type = 10).

**Fix:** Add multiple TempDB data files (one per CPU core, up to 8). Each file has its own GAM/SGAM/PFS and IAM pages, distributing the contention. Use `TF 1118` (uniform extent allocation for TempDB) to reduce SGAM contention. In SQL Server 2016+, `TF 1118` is default for TempDB.

**Cost:** Adding TempDB files requires a restart if changing the initial file count. After restart, TempDB files cannot be removed (only added). Start with the correct count (1:1 with cores, capped at 8).

## Interview Arsenal

### Core Questions

1. **Q: What is an IAM page and what problem does it solve?**
   **A (spoken):** "An Index Allocation Map (IAM) page is an 8KB metadata page that maps which extents belong to a specific allocation unit of a specific index or table. Each IAM page contains an 8,000-byte bitmap covering a 4GB range of the data file — each bit represents one extent (64KB). IAM pages solve the problem of tracking object ownership of pages: GAM/SGAM track which extents are free or partially free globally, but they don't track which object owns an allocated extent. IAM fills that gap. For heap scans, the IAM chain is the primary access path — SQL Server reads the IAM bitmap to find all extents belonging to the heap, then reads those extents. For B-tree indexes, IAM is used during rebuilds, DBCC CHECKDB, and backup operations, but query access uses the B-tree leaf chain directly."

2. **Q: Walk through how SQL Server scans a heap using IAM.**
   **A (spoken):** "SQL Server starts by reading the first IAM page from sys.allocation_units.first_iam_page for the heap's IN_ROW_DATA allocation unit. It scans the 8,000-byte bitmap sequentially, examining each of the 64,000 bits. For each bit set to 1, the corresponding extent belongs to the heap. For uniform extents (the object's own extents), SQL Server reads all 8 pages. For mixed extents (shared with other objects, which happens only for the first 8 pages), SQL Server checks the PFS page to see which specific pages within that extent belong to the heap. After exhausting one IAM page, it follows the NextPage pointer in the IAM page header to the next IAM page in the chain. It repeats until the chain ends (NextPage = NULL). The result is a set of unordered data pages — there is no guarantee of key order."

3. **Q: Why does TempDB suffer from IAM page contention, and how do you fix it?**
   **A (spoken):** "TempDB experiences high IAM contention because hundreds or thousands of concurrent sessions create and drop temporary tables every second. Each temp table allocation requires updating the IAM page bitmap (setting a bit from 0 to 1). In a single TempDB data file, all this contention targets the first IAM page in the first 4GB range — a single page that must be latched (PAGELATCH_UP) for each allocation. The fix is three-fold: (1) add multiple TempDB data files — each file has its own IAM chain, distributing the contention across files; (2) enable Trace Flag 1118 (default since SQL Server 2016) which forces uniform extent allocation from page 1, avoiding SGAM contention; (3) enable Trace Flag 1117 (default since 2016) for equal autogrowth across files. The number of TempDB files should equal the number of CPU cores, capped at 8. After these changes, TempDB IAM contention typically drops by 90%+."

<summary>Answers to Conceptual Questions</summary>

**1.** An IAM page contains an 8,000-byte bitmap where each bit represents one extent (64KB = 8 pages × 8KB). Bit = 1 means that extent belongs to the allocation unit; bit = 0 means it does not. With 64,000 bits (8,000 bytes × 8 bits), one IAM page covers 64,000 extents = 4,096 MB = 4 GB of data file space. The bitmap also has a header (8 bytes) and sequence number (8 bytes), but the total payload is 8,000 bytes for the bitmap.

**2.** An IAM scan reads pages in file order (extent order within each IAM page range), not in any logical key order. The IAM scan reads extents in ascending page ID order within each 4GB IAM range, then moves to the next IAM page. A clustered index scan reads pages in logical key order via the doubly-linked leaf page chain (NextPage pointers). This means a clustered index scan preserves the sort order of the clustering key, while an IAM scan does not.

**3.** SQL Server uses an IAM scan for heap tables because there is no B-tree page chain to follow. The only way to enumerate all pages belonging to a heap is through the IAM bitmap. Additionally, the IAM scan is more efficient than scanning the entire data file because it skips pages that belong to other objects (especially important in mixed extents). For a clustered index, the leaf page chain provides a more direct enumeration path.

**4.** During an IAM scan of a mixed extent (where the IAM bit is 1 but only some pages in the extent belong to the object), SQL Server must check the PFS page to determine which specific pages within the extent belong to this object. The PFS page has one byte per page tracking allocation status and free space. Without checking PFS, SQL Server would read pages belonging to other objects, which would waste I/O and potentially return incorrect data.

**5.** The IAM chain is a doubly-linked list of IAM pages connected via NextPage/PrevPage pointers in the IAM page header. Each allocation unit of a table has its own IAM chain. A non-partitioned table without LOB/overflow has 1 IAM chain (for IN_ROW_DATA). With LOB columns, it has 2+ chains (IN_ROW_DATA + LOB_DATA). With row overflow, it has IN_ROW_DATA + ROW_OVERFLOW_DATA. A partitioned table has separate IAM chains per partition per allocation unit. So a partitioned table with LOB and 100 partitions has 300 IAM chains.

**6.** When a new extent is allocated to an object, SQL Server sets the corresponding bit in the IAM page's bitmap from 0 to 1. This requires a PAGELATCH_UP (update latch) on the IAM page. The update latch is exclusive — no other session can read or write the IAM page during the update. This is the source of IAM page contention in TempDB. After updating the bitmap, SQL Server also updates the GAM page (set extent bit to 0 = allocated) and possibly the SGAM page (if mixed extent).

**7.** GAM/SGAM answer the global question "is this extent free or allocated?" — GAM bit = 1 means free, GAM bit = 0 means allocated; SGAM tracks mixed extents with at least one free page. IAM answers the per-object question "does this extent belong to object X?" During allocation, SQL Server checks GAM to find a free extent, sets the GAM bit to 0 (allocated), and sets the corresponding IAM bit to 1 (claimed by object). During deallocation, the IAM bit is cleared and the GAM bit is set back to 1.

**8.** The optimizer estimates the cost of scanning each available structure: the IAM scan (total data pages) vs scanning each non-clustered index (NC index leaf pages). If a narrow NC index has significantly fewer pages than the heap (e.g., 1,000 NC index pages vs 100,000 heap pages), the optimizer may choose to scan the NC index and use a Stream Aggregate to compute the count. This is why adding a narrow NC index can speed up COUNT queries on a heap.

**9.** ALTER TABLE REBUILD allocates new extents for the rebuilt table and creates a new IAM chain. The old IAM chain is deallocated. The new IAM chain typically has fewer extents (because data is consolidated) and the extents are more contiguous (less fragmentation). The new first_iam_page pointer is updated in sys.allocation_units. The rebuild essentially compacts the IAM bitmap — fewer bits set = smaller IAM chain.

**10.** `first_iam_page` is a 6-byte binary value stored in `sys.allocation_units`. It encodes the File ID (2 bytes, little-endian) and Page ID (4 bytes, little-endian). To decode: `FILE_ID(CONVERT(VARBINARY, SUBSTRING(first_iam_page, 1, 2), 1))` for the file and `CONVERT(INT, SUBSTRING(first_iam_page, 3, 4), 1)` for the page. For example, `0x0001000000BD` → File 1, Page 189.
