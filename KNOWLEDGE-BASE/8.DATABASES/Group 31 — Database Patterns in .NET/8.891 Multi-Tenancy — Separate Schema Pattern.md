# Multi-Tenancy — Separate Schema Pattern

## 1. Overview — The Schema-Per-Tenant Pattern

The separate schema pattern (also called schema-per-tenant) gives each tenant its own database schema within a shared database. A schema is a named container of database objects — tables, views, indexes, stored procedures — that belong to a single tenant. All tenants share the same database server and database, but their data is isolated at the schema level.

This pattern offers a middle ground between the shared-schema approach (row-level isolation via `TenantId`) and the separate-database approach (full database isolation):

- **Better isolation than shared schema** — a schema boundary prevents accidental cross-tenant joins or queries. A missing `WHERE TenantId = @id` clause in shared schema can leak data across all tenants. In schema-per-tenant, a query without schema qualification simply fails or returns empty results for the wrong schema.
- **Lower cost than separate databases** — a single database server and a single logical database serve all tenants. Connection pooling is shared. Backups and maintenance operate on one database.
- **Easier migrations than separate databases** — a single migration script can be applied to the database, creating or altering objects in each tenant's schema. However, the migration must iterate over N schemas, which adds complexity compared to a single-schema deployment.

```sql
-- Tenant A's schema
CREATE SCHEMA tenant_a;
CREATE TABLE tenant_a.Orders (
    Id          INT IDENTITY(1,1) PRIMARY KEY,
    CustomerId  VARCHAR(50) NOT NULL,
    OrderDate   DATETIME2 NOT NULL,
    Total       DECIMAL(18,2) NOT NULL,
    IsDeleted   BIT NOT NULL DEFAULT 0
);

-- Tenant B's schema (same table structure, different schema)
CREATE SCHEMA tenant_b;
CREATE TABLE tenant_b.Orders (
    Id          INT IDENTITY(1,1) PRIMARY KEY,
    CustomerId  VARCHAR(50) NOT NULL,
    OrderDate   DATETIME2 NOT NULL,
    Total       DECIMAL(18,2) NOT NULL,
    IsDeleted   BIT NOT NULL DEFAULT 0
);

-- Each tenant's schema has its own identity seed, indexes, etc.
CREATE INDEX IX_Orders_CustomerId ON tenant_a.Orders (CustomerId);
CREATE INDEX IX_Orders_CustomerId ON tenant_b.Orders (CustomerId);
```

The key difference from shared schema: there is no `TenantId` column on the tables. The schema name IS the tenant identifier. Queries are qualified with the schema name:

```sql
SELECT Id, CustomerId, OrderDate, Total
FROM tenant_a.Orders
WHERE CustomerId = 'CUST001';
```

### 1.1 Comparison: Shared Schema vs Separate Schema vs Separate Database

| Criterion | Shared Schema (TenantId) | Separate Schema | Separate Database |
|---|---|---|---|
| Isolation | Row-level | Schema-level | Database-level |
| Data leak risk | High (missing WHERE clause) | Medium (wrong schema reference) | None (wrong DB fails auth) |
| Unique constraints | Include TenantId | Per-schema, no cross-tenant concern | Per-database, fully isolated |
| Migration complexity | Single migration | Iterate N schemas | Iterate N databases |
| Connection pooling | Single pool | Single pool (same database) | N pools (N databases) |
| Backup/Restore | Single backup | Single backup (all schemas) | N backups |
| Cross-tenant reporting | Easy (query with TenantId filter) | Union across schemas | Cross-database queries (server links) |
| Customization per tenant | None (shared tables) | Schema changes per tenant possible | Full database customization |
| Infrastructure cost | Lowest | Low | High |

### 1.2 When to Use Separate Schema

- Tenants require moderate isolation but share a database server for cost efficiency.
- Each tenant has the same table structure but may have different indexes, views, or stored procedures.
- A tenant's data volume is large enough that a `WHERE TenantId = @id` scan becomes expensive without proper partitioning.
- You need to give tenants their own database objects (e.g., custom views or reporting procedures).
- Regulatory requirements demand schema-level separation (e.g., PCI DSS scope reduction).

### 1.3 When NOT to Use Separate Schema

- You have thousands of tenants — most databases have a practical limit on the number of schemas (SQL Server allows up to 2^31-1 schemas, but tooling and management become unwieldy beyond a few hundred).
- Tenants have identical needs and no customization is required — shared schema is simpler.
- You need to move a tenant to its own database later — this is possible but requires data migration and application changes.
- Your ORM or query tool lacks multi-schema support — some ORMs struggle with dynamic schema resolution.

## 2. Section 2 — Schema Resolution in ASP.NET Core

Before the data access layer can use the tenant-specific schema, the schema name must be resolved from the request context.

### 2.1 Tenant Schema Context

```csharp
public interface ITenantSchemaContext
{
    string Schema { get; }
    int TenantId { get; }
    string TenantName { get; }
}

public class TenantSchemaContext : ITenantSchemaContext
{
    public string Schema { get; internal set; } = string.Empty;
    public int TenantId { get; internal set; }
    public string TenantName { get; internal set; } = string.Empty;
}
```

### 2.2 Middleware to Resolve Schema

```csharp
public class TenantSchemaMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ITenantStore _tenantStore;

    public TenantSchemaMiddleware(RequestDelegate next, ITenantStore tenantStore)
    {
        _next = next;
        _tenantStore = tenantStore;
    }

    public async Task InvokeAsync(HttpContext context, ITenantSchemaContext schemaContext)
    {
        var tenantId = ResolveTenantId(context);
        if (tenantId is null)
        {
            context.Response.StatusCode = 401;
            await context.Response.WriteAsync("Tenant not identified.");
            return;
        }

        var tenant = await _tenantStore.GetTenantAsync(tenantId.Value);
        if (tenant is null || !tenant.IsActive)
        {
            context.Response.StatusCode = 403;
            await context.Response.WriteAsync("Tenant inactive or not found.");
            return;
        }

        // Normalize schema name (e.g., "tenant_123" or "acme_corp")
        var schema = NormalizeSchemaName(tenant.SchemaName ?? $"tenant_{tenant.Id}");

        ((TenantSchemaContext)schemaContext).Schema = schema;
        ((TenantSchemaContext)schemaContext).TenantId = tenant.Id;
        ((TenantSchemaContext)schemaContext).TenantName = tenant.Name;

        context.Items["Schema"] = schema;
        await _next(context);
    }

    private static int? ResolveTenantId(HttpContext context)
    {
        if (context.Request.Headers.TryGetValue("X-Tenant-Id", out var header)
            && int.TryParse(header, out var id))
            return id;

        var claim = context.User?.FindFirst("tenant_id");
        if (claim is not null && int.TryParse(claim.Value, out var fromClaim))
            return fromClaim;

        return null;
    }

    private static string NormalizeSchemaName(string name)
    {
        // Ensure the schema name is a valid SQL identifier
        var invalid = new Regex(@"[^a-zA-Z0-9_]");
        var normalized = invalid.Replace(name, "_").ToLowerInvariant();
        if (char.IsDigit(normalized[0]))
            normalized = "t_" + normalized;
        return normalized;
    }
}
```

### 2.3 Tenant Store

```csharp
public class Tenant
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;
    public string? SchemaName { get; set; }
    public bool IsActive { get; set; }
    public string? ConnectionString { get; set; }
}

public interface ITenantStore
{
    Task<Tenant?> GetTenantAsync(int tenantId);
    Task<IReadOnlyList<Tenant>> GetAllActiveTenantsAsync();
}

public class TenantStore : ITenantStore
{
    private readonly IDbConnection _connection;
    private readonly IMemoryCache _cache;

    public TenantStore(IDbConnection connection, IMemoryCache cache)
    {
        _connection = connection;
        _cache = cache;
    }

    public async Task<Tenant?> GetTenantAsync(int tenantId)
    {
        var cacheKey = $"tenant:{tenantId}";
        return await _cache.GetOrCreateAsync(cacheKey, async entry =>
        {
            entry.SlidingExpiration = TimeSpan.FromMinutes(15);
            var sql = "SELECT Id, Name, SchemaName, IsActive, ConnectionString FROM Tenants WHERE Id = @Id";
            return await _connection.QuerySingleOrDefaultAsync<Tenant>(sql, new { Id = tenantId });
        });
    }

    public async Task<IReadOnlyList<Tenant>> GetAllActiveTenantsAsync()
    {
        var sql = "SELECT Id, Name, SchemaName, IsActive, ConnectionString FROM Tenants WHERE IsActive = 1";
        var result = await _connection.QueryAsync<Tenant>(sql);
        return result.ToList();
    }
}
```

## 3. Section 3 — EF Core Implementation: Schema Per Tenant

EF Core supports schema mapping via the `ToTable` method, which accepts a schema parameter. The challenge is applying the schema dynamically per tenant.

### 3.1 Entity Configuration Without Fixed Schema

Entities are defined without schema annotations; the schema is applied at model-building time:

```csharp
public class Order
{
    public int Id { get; set; }
    public string CustomerId { get; set; } = string.Empty;
    public DateTime OrderDate { get; set; }
    public decimal Total { get; set; }
    public bool IsDeleted { get; set; }
    public DateTime? DeletedAt { get; set; }
    // No TenantId column — schema provides isolation

    public ICollection<OrderLine> Lines { get; set; } = new List<OrderLine>();
}

public class Customer
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;
    public string Email { get; set; } = string.Empty;
    public bool IsDeleted { get; set; }
    public DateTime? DeletedAt { get; set; }

    public ICollection<Order> Orders { get; set; } = new List<Order>();
}

public class Product
{
    public int Id { get; set; }
    public string Sku { get; set; } = string.Empty;
    public string Name { get; set; } = string.Empty;
    public decimal Price { get; set; }
    public bool IsDeleted { get; set; }
    public DateTime? DeletedAt { get; set; }
}
```

### 3.2 DbContext with Dynamic Schema

```csharp
public class SalesDbContext : DbContext
{
    private readonly ITenantSchemaContext _schemaContext;

    public SalesDbContext(
        DbContextOptions<SalesDbContext> options,
        ITenantSchemaContext schemaContext)
        : base(options)
    {
        _schemaContext = schemaContext;
    }

    public DbSet<Order> Orders => Set<Order>();
    public DbSet<Customer> Customers => Set<Customer>();
    public DbSet<Product> Products => Set<Product>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        var schema = _schemaContext.Schema;

        // Map each entity to the tenant-specific schema
        modelBuilder.Entity<Order>(entity =>
        {
            entity.ToTable("Orders", schema);
            entity.HasKey(e => e.Id);
            entity.Property(e => e.CustomerId).HasMaxLength(50).IsRequired();
            entity.Property(e => e.Total).HasColumnType("decimal(18,2)");
            entity.HasMany(e => e.Lines)
                  .WithOne()
                  .HasForeignKey("OrderId");
        });

        modelBuilder.Entity<Customer>(entity =>
        {
            entity.ToTable("Customers", schema);
            entity.HasKey(e => e.Id);
            entity.Property(e => e.Name).HasMaxLength(200).IsRequired();
            entity.Property(e => e.Email).HasMaxLength(200);
            entity.HasIndex(e => e.Email).IsUnique();
        });

        modelBuilder.Entity<Product>(entity =>
        {
            entity.ToTable("Products", schema);
            entity.HasKey(e => e.Id);
            entity.Property(e => e.Sku).HasMaxLength(50).IsRequired();
            entity.Property(e => e.Name).HasMaxLength(200).IsRequired();
            entity.Property(e => e.Price).HasColumnType("decimal(18,2)");
            entity.HasIndex(e => e.Sku).IsUnique();
        });

        // Soft delete filter — no TenantId needed
        modelBuilder.Entity<Order>().HasQueryFilter(e => !e.IsDeleted);
        modelBuilder.Entity<Customer>().HasQueryFilter(e => !e.IsDeleted);
        modelBuilder.Entity<Product>().HasQueryFilter(e => !e.IsDeleted);
    }
}
```

### 3.3 How EF Core's DbContext Pooling Affects Schema

The critical detail: `OnModelCreating` is called only once per `DbContext` type during the application's lifetime when using `AddDbContextPool`. This means the **first** tenant's schema is captured in the model, and all subsequent tenants use that same schema — which is incorrect.

**Solution**: Disable `DbContextPool` for multi-schema scenarios, or use a per-tenant model cache key.

```csharp
// DO NOT use pooling with schema-per-tenant:
// builder.Services.AddDbContextPool<SalesDbContext>(...);  // WRONG

// Use scoped lifetime instead:
builder.Services.AddDbContext<SalesDbContext>((sp, options) =>
{
    var schemaContext = sp.GetRequiredService<ITenantSchemaContext>();
    var connectionString = sp.GetRequiredService<IConfiguration>()
        .GetConnectionString("DefaultConnection");

    options.UseSqlServer(connectionString);
});
```

Each `DbContext` instance is created per-scope (per-request). `OnModelCreating` runs once for the first request of each tenant (the model is cached by EF Core per `DbContext` type *and* model cache key). Since the schema is part of the model, each tenant with a different schema triggers a separate model cache entry.

### 3.4 Model Cache Key Per Tenant

EF Core 5+ allows you to provide a model cache key via `DbContextOptionsBuilder.ModelCacheKey`. This ensures different tenants get different models:

```csharp
public class TenantModelCacheKeyFactory : IModelCacheKeyFactory
{
    public object Create(DbContext context, bool designTime)
    {
        if (context is SalesDbContext salesDb)
        {
            return (context.GetType(), salesDb._schemaContext.Schema);
        }
        return context.GetType();
    }
}
```

Registration:

```csharp
builder.Services.AddDbContext<SalesDbContext>((sp, options) =>
{
    var schemaContext = sp.GetRequiredService<ITenantSchemaContext>();
    var connection = sp.GetRequiredService<IConfiguration>()
        .GetConnectionString("DefaultConnection");

    options.UseSqlServer(connection)
           .ReplaceService<IModelCacheKeyFactory, TenantModelCacheKeyFactory>();
});
```

This way, EF Core caches one model per tenant schema. Each model has its `ToTable("Orders", "tenant_a")` baked in. The model is reused across requests for the same tenant.

### 3.5 Generated SQL

When the `TenantModelCacheKeyFactory` is in place and the current schema is `tenant_a`:

```csharp
var orders = await db.Orders
    .Where(o => o.CustomerId == "CUST001")
    .ToListAsync();
```

Generated SQL:

```sql
SELECT [o].[Id], [o].[CustomerId], [o].[OrderDate], [o].[Total],
       [o].[IsDeleted], [o].[DeletedAt]
FROM [tenant_a].[Orders] AS [o]
WHERE [o].[IsDeleted] = 0
  AND [o].[CustomerId] = @__customerId_0;
```

The schema `tenant_a` is hard-coded into the SQL query text because it was baked into the model. This is optimal — there is no runtime schema resolution overhead after the model is built.

### 3.6 Convention-Based Schema Application

For EF Core 6+, apply the schema to all entities via a convention:

```csharp
public class TenantSchemaConvention : IModelFinalizingConvention
{
    private readonly string _schema;

    public TenantSchemaConvention(string schema)
    {
        _schema = schema;
    }

    public void ProcessModelFinalizing(
        IConventionModelBuilder modelBuilder,
        IConventionContext<IConventionModelBuilder> context)
    {
        foreach (var entityType in modelBuilder.Metadata.GetEntityTypes())
        {
            if (!entityType.IsOwned() && !entityType.IsKeyless)
            {
                entityType.SetSchema(_schema);
            }
        }
    }
}
```

### 3.7 Creating Schemas for New Tenants

When a new tenant signs up, their schema must be created and populated:

```csharp
public async Task ProvisionTenantSchemaAsync(int tenantId)
{
    var schema = $"tenant_{tenantId}";

    await using var db = new SalesDbContext(_options, new TenantSchemaContext
    {
        Schema = schema,
        TenantId = tenantId
    });

    // Create schema
    await db.Database.ExecuteSqlRawAsync($"CREATE SCHEMA [{schema}]");

    // Apply migrations (creates all tables in the new schema)
    await db.Database.MigrateAsync();

    // Seed initial data if needed
    // await db.SeedAsync();
}
```

Alternatively, generate idempotent SQL scripts and execute them dynamically:

```sql
-- Idempotent schema creation
IF NOT EXISTS (SELECT * FROM sys.schemas WHERE name = 'tenant_123')
BEGIN
    EXEC('CREATE SCHEMA [tenant_123]');
END

-- Create tables in the schema
CREATE TABLE [tenant_123].[Orders] (
    Id          INT IDENTITY(1,1) PRIMARY KEY,
    CustomerId  VARCHAR(50) NOT NULL,
    OrderDate   DATETIME2 NOT NULL,
    Total       DECIMAL(18,2) NOT NULL,
    IsDeleted   BIT NOT NULL DEFAULT 0,
    DeletedAt   DATETIME2 NULL
);

CREATE TABLE [tenant_123].[Customers] ( ... );
CREATE TABLE [tenant_123].[Products] ( ... );

-- Create indexes
CREATE INDEX [IX_Orders_CustomerId] ON [tenant_123].[Orders] (CustomerId);
CREATE UNIQUE INDEX [IX_Customers_Email] ON [tenant_123].[Customers] (Email);
CREATE UNIQUE INDEX [IX_Products_Sku] ON [tenant_123].[Products] (Sku);
```

## 4. Section 4 — Dapper Implementation: Dynamic Schema in SQL

Dapper has no model-building phase, so schema resolution is purely a string interpolation concern. The schema must be injected into every SQL statement.

### 4.1 Repository Base with Schema

```csharp
public abstract class SchemaRepositoryBase<T> where T : class
{
    protected readonly IDbConnection _connection;
    protected readonly ITenantSchemaContext _schemaContext;
    protected readonly string _tableName;

    protected SchemaRepositoryBase(
        IDbConnection connection,
        ITenantSchemaContext schemaContext,
        string tableName)
    {
        _connection = connection;
        _schemaContext = schemaContext;
        _tableName = tableName;
    }

    protected string Schema => _schemaContext.Schema;
    protected string QualifiedTableName => $"[{Schema}].[{_tableName}]";

    public virtual async Task<T?> GetByIdAsync(int id)
    {
        var sql = $"SELECT * FROM {QualifiedTableName} WHERE Id = @Id AND IsDeleted = 0";
        return await _connection.QueryFirstOrDefaultAsync<T>(sql, new { Id = id });
    }

    public virtual async Task<IReadOnlyList<T>> GetAllAsync()
    {
        var sql = $"SELECT * FROM {QualifiedTableName} WHERE IsDeleted = 0";
        var result = await _connection.QueryAsync<T>(sql);
        return result.ToList();
    }

    public virtual async Task<int> CountAsync()
    {
        var sql = $"SELECT COUNT(1) FROM {QualifiedTableName} WHERE IsDeleted = 0";
        return await _connection.ExecuteScalarAsync<int>(sql);
    }

    public virtual async Task AddAsync(T entity)
    {
        var sql = GenerateInsertSql(QualifiedTableName);
        await _connection.ExecuteAsync(sql, entity);
    }

    public virtual async Task SoftDeleteAsync(int id)
    {
        var sql = $"UPDATE {QualifiedTableName} SET IsDeleted = 1, DeletedAt = @Now WHERE Id = @Id AND IsDeleted = 0";
        await _connection.ExecuteAsync(sql, new { Id = id, Now = DateTime.UtcNow });
    }

    private static string GenerateInsertSql(string qualifiedName)
    {
        // Generate INSERT statement dynamically based on T's properties
        // (simplified — use a SQL builder in production)
        return $"INSERT INTO {qualifiedName} (...) VALUES (...)";
    }
}
```

### 4.2 Concrete Repository

```csharp
public class OrderRepository : SchemaRepositoryBase<Order>
{
    public OrderRepository(IDbConnection connection, ITenantSchemaContext schemaContext)
        : base(connection, schemaContext, "Orders") { }

    public async Task<IReadOnlyList<Order>> GetOrdersByCustomerAsync(string customerId)
    {
        var sql = $@"SELECT * FROM {QualifiedTableName}
                     WHERE CustomerId = @CustomerId
                       AND IsDeleted = 0
                     ORDER BY OrderDate DESC";
        var result = await _connection.QueryAsync<Order>(sql,
            new { CustomerId = customerId });
        return result.ToList();
    }

    public async Task<Order?> GetOrderWithLinesAsync(int orderId)
    {
        var ordersTable = QualifiedTableName;
        var linesTable = $"[{Schema}].[OrderLines]";

        var sql = $@"SELECT o.*, ol.*
                     FROM {ordersTable} o
                     LEFT JOIN {linesTable} ol ON ol.OrderId = o.Id
                     WHERE o.Id = @OrderId
                       AND o.IsDeleted = 0
                       AND (ol.IsDeleted IS NULL OR ol.IsDeleted = 0)";

        using var multi = await _connection.QueryMultipleAsync(sql,
            new { OrderId = orderId });
        var order = await multi.ReadSingleOrDefaultAsync<Order>();
        if (order != null)
        {
            order.Lines = (await multi.ReadAsync<OrderLine>()).ToList();
        }
        return order;
    }

    public async Task<decimal> GetTotalRevenueAsync(DateTime from, DateTime to)
    {
        var sql = $@"SELECT COALESCE(SUM(Total), 0)
                     FROM {QualifiedTableName}
                     WHERE OrderDate >= @From
                       AND OrderDate < @To
                       AND IsDeleted = 0";
        return await _connection.ExecuteScalarAsync<decimal>(sql,
            new { From = from, To = to });
    }

    public async Task<IReadOnlyList<OrderSummary>> GetOrderSummaryAsync()
    {
        var sql = $@"SELECT
                        CustomerId,
                        COUNT(1) AS OrderCount,
                        COALESCE(SUM(Total), 0) AS TotalSpent
                     FROM {QualifiedTableName}
                     WHERE IsDeleted = 0
                     GROUP BY CustomerId
                     ORDER BY TotalSpent DESC";
        var result = await _connection.QueryAsync<OrderSummary>(sql);
        return result.ToList();
    }
}
```

### 4.3 Cross-Tenant Queries (Admin)

For admin queries that span tenants, you must either:

1. **Union across schemas** — dynamic SQL that iterates all tenant schemas.
2. **Query the default (dbo) schema** — maintain a separate set of views that union all tenant data.

```csharp
public async Task<IReadOnlyList<Order>> GetAllTenantsOrdersAsync(
    DateTime from, DateTime to)
{
    var tenants = await _tenantStore.GetAllActiveTenantsAsync();
    var queries = tenants.Select(t =>
        $"SELECT Id, CustomerId, OrderDate, Total FROM [{t.SchemaName}].Orders " +
        $"WHERE OrderDate >= @From AND OrderDate < @To AND IsDeleted = 0");

    var unionSql = string.Join(" UNION ALL ", queries);
    if (string.IsNullOrEmpty(unionSql))
        return new List<Order>();

    var result = await _connection.QueryAsync<Order>(unionSql,
        new { From = from, To = to });
    return result.ToList();
}
```

### 4.4 SQL Injection Prevention

Always bracket schema names to prevent SQL injection:

```csharp
// SAFE: brackets prevent injection
var safeSchema = $"[{Schema}]";

// DANGEROUS: direct interpolation without sanitization
// var dangerousSql = $"SELECT * FROM {Schema}.Orders";  // Schema could be "Orders; DROP TABLE..."
```

The `NormalizeSchemaName` function in section 2.2 ensures the schema name contains only alphanumeric and underscore characters, but bracketing adds defense-in-depth.

## 6. Section 6 — Migration Strategy for Schema-Per-Tenant

Migrating N schemas is more complex than a single schema but far simpler than N databases.

### 6.1 EF Core Migration Approach

EF Core generates migration snapshots that include the schema name. With dynamic schema, you have two options:

**Option A: Per-tenant migration run**

```csharp
public async Task MigrateAllTenantsAsync()
{
    var tenants = await _tenantStore.GetAllActiveTenantsAsync();

    foreach (var tenant in tenants)
    {
        var schema = NormalizeSchemaName(tenant.SchemaName ?? $"tenant_{tenant.Id}");

        await using var db = new SalesDbContext(_options, new TenantSchemaContext
        {
            Schema = schema,
            TenantId = tenant.Id
        });

        await db.Database.MigrateAsync();
    }
}
```

This runs the EF Core migrator once per tenant. Each invocation creates or alters tables in the tenant's schema. The migration history table (`__EFMigrationsHistory`) is tenant-specific (stored in each schema or as a shared table with schema discriminator).

**Option B: Raw SQL script per schema**

```csharp
public async Task MigrateAllTenantsWithRawSqlAsync()
{
    var script = await GetMigrationScriptAsync();  // Generate idempotent SQL
    var tenants = await _tenantStore.GetAllActiveTenantsAsync();

    foreach (var tenant in tenants)
    {
        var schema = NormalizeSchemaName(tenant.SchemaName ?? $"tenant_{tenant.Id}");

        // Replace schema placeholder with actual schema
        var tenantScript = script.Replace("__SCHEMA__", schema);

        await using var connection = new SqlConnection(_connectionString);
        await connection.OpenAsync();
        await connection.ExecuteAsync(tenantScript);
    }
}
```

### 6.2 Creating Schema for a New Tenant at Registration

```csharp
public async Task RegisterTenantAsync(string name, string? schemaName = null)
{
    var schema = NormalizeSchemaName(schemaName ?? $"tenant_{Guid.NewGuid():N}");

    // 1. Insert tenant record
    var tenantId = await InsertTenantRecordAsync(name, schema);

    // 2. Create database schema
    await using var connection = new SqlConnection(_connectionString);
    await connection.ExecuteAsync($"CREATE SCHEMA [{schema}]");

    // 3. Run migrations for this tenant's schema
    await MigrateTenantAsync(tenantId, schema);

    // 4. Seed default data
    await SeedTenantDataAsync(tenantId, schema);
}

private async Task MigrateTenantAsync(int tenantId, string schema)
{
    await using var db = new SalesDbContext(_options, new TenantSchemaContext
    {
        Schema = schema,
        TenantId = tenantId
    });
    await db.Database.MigrateAsync();
}
```

### 6.3 Handling Schema Drift

Over time, tenant schemas may drift if:

- A migration failed for one tenant but succeeded for others.
- A tenant requested custom schema modifications.
- A manual hotfix was applied to one schema but not propagated.

Detect drift by comparing the schema version (from `__EFMigrationsHistory`) across tenants:

```csharp
public async Task DetectSchemaDriftAsync()
{
    var tenants = await _tenantStore.GetAllActiveTenantsAsync();

    foreach (var tenant in tenants)
    {
        var schema = NormalizeSchemaName(tenant.SchemaName ?? $"tenant_{tenant.Id}");
        var sql = $"SELECT MigrationId FROM [{schema}].[__EFMigrationsHistory] ORDER BY MigrationId";

        var migrations = await _connection.QueryAsync<string>(sql);
        var expectedMigrations = await GetExpectedMigrationsAsync();

        var missing = expectedMigrations.Except(migrations).ToList();
        if (missing.Any())
        {
            _logger.LogWarning("Tenant {TenantId} ({Schema}) is missing migrations: {Missing}",
                tenant.Id, schema, string.Join(", ", missing));
        }

        var extra = migrations.Except(expectedMigrations).ToList();
        if (extra.Any())
        {
            _logger.LogWarning("Tenant {TenantId} ({Schema}) has unexpected migrations: {Extra}",
                tenant.Id, schema, string.Join(", ", extra));
        }
    }
}
```

### 7.2 Tenant-Specific Views

```sql
-- Tenant A needs a view that includes abandoned carts
CREATE VIEW [tenant_a].[vw_OrderSummary] AS
SELECT CustomerId, COUNT(1) AS OrderCount, SUM(Total) AS TotalRevenue
FROM [tenant_a].[Orders]
WHERE IsDeleted = 0
GROUP BY CustomerId;

-- Tenant B wants the same view but excluding small orders
CREATE VIEW [tenant_b].[vw_OrderSummary] AS

FROM [tenant_b].[Orders]

  AND Total > 10.00
GROUP BY CustomerId;
```

```sql
CREATE PROCEDURE [tenant_a].[usp_CalculateLoyaltyPoints]
    @CustomerId VARCHAR(50),
    @Points INT OUTPUT
AS
BEGIN
    -- Tenant A uses a simple formula
    SELECT @Points = COALESCE(SUM(Total) / 10, 0)
    FROM [tenant_a].[Orders]
```

```sql
CREATE PROCEDURE [tenant_b].[usp_CalculateLoyaltyPoints]
    @CustomerId VARCHAR(50),
    @Points INT OUTPUT

BEGIN
    -- Tenant B uses a weighted formula
    SELECT @Points = COALESCE(SUM(Total * Weight), 0)
    FROM [tenant_b].[Orders] o
    JOIN [tenant_b].[OrderWeights] w ON w.CustomerId = o.CustomerId
    WHERE o.CustomerId = @CustomerId AND o.IsDeleted = 0;
END
```

### 7.4 Schema-Level Permissions

```sql
-- Create a database user for each tenant and grant schema-level permissions
CREATE USER [app_tenant_a] FROM LOGIN [login_tenant_a];
GRANT SELECT, INSERT, UPDATE, DELETE ON SCHEMA :: [tenant_a] TO [app_tenant_a];
DENY SELECT, INSERT, UPDATE, DELETE ON SCHEMA :: [tenant_b] TO [app_tenant_a];
```

This provides database-level isolation — even if a query omits the schema qualifier, the user cannot access another tenant's schema.

## 8. Section 8 — Soft Delete in Schema-Per-Tenant

With schema-per-tenant, soft delete works the same way as in a single-tenant application because each schema is effectively its own isolated data space.

### 8.1 EF Core

```csharp
modelBuilder.Entity<Order>()
    .ToTable("Orders", schema)
    .HasQueryFilter(e => !e.IsDeleted);

modelBuilder.Entity<Customer>()
    .ToTable("Customers", schema)
    .HasQueryFilter(e => !e.IsDeleted);
```

No `TenantId` filter is needed — the schema provides isolation. The global query filter only handles soft delete.

### 8.2 Dapper

```csharp
public async Task<IReadOnlyList<Order>> GetActiveOrdersAsync()
{
    var sql = $"SELECT * FROM [{Schema}].[Orders] WHERE IsDeleted = 0";
    var result = await _connection.QueryAsync<Order>(sql);
    return result.ToList();
}
```

### 8.3 Unique Constraints Are Per-Schema

Since each schema has its own tables, unique constraints do not need `TenantId`. Each tenant's `Email` uniqueness is enforced independently:

```sql
-- Per-schema unique index — no TenantId needed
CREATE UNIQUE INDEX [IX_Customers_Email] ON [tenant_a].[Customers] (Email);
CREATE UNIQUE INDEX [IX_Customers_Email] ON [tenant_b].[Customers] (Email);
```

This is a significant advantage over shared schema: there is no `Email` conflict between tenants, and no need for filtered unique indexes.

## 9. Section 9 — Gotchas, Pitfalls, and Best Practices

### 9.1 EF Core Model Caching Problem

The single most critical issue: **EF Core caches the model per `DbContext` type**. If the schema is baked into `OnModelCreating` and the model is cached, the first tenant's schema leaks into all subsequent tenants' queries.

**Fix**: Implement `IModelCacheKeyFactory` and include the schema name in the cache key. Or disable `AddDbContextPool` and ensure each `DbContext` instance builds its own model (less efficient but safe).

### 9.2 Schema Name Collisions

Ensure schema names are globally unique. A collision between `tenant_1` and `tenant_01` (both normalizing to the same string) would merge tenant data. Always validate uniqueness at tenant registration.

### 9.3 Dynamic Schema in Raw SQL Queries (Dapper)

Every Dapper query must use the qualified `[schema].[table]` syntax. It is easy to forget the schema qualifier in a `JOIN` or subquery. Use a helper property like `QualifiedTableName` to reduce repetition.

### 9.4 Migration Complexity

- You must iterate over N schemas for each migration.
- A failed migration on one schema should not block other tenants.
- Implement per-schema migration status tracking.
- Consider downtime or blue-green deployment for large migrations across many tenants.

### 9.5 SQL Injection Through Schema Name

Always sanitize and bracket schema names. Never trust user input directly as a schema name. The `NormalizeSchemaName` function should strip all non-alphanumeric characters (except underscores).

### 9.6 Cross-Tenant Reporting

Querying across schemas requires `UNION ALL` or a federated view. This adds complexity and may perform poorly with many schemas. For frequent cross-tenant reporting, consider maintaining a separate reporting database that aggregates data from all schemas.

### 9.7 Schema Count Limits

While SQL Server supports up to 2^31-1 schemas, practical limits are much lower. Managing hundreds of schemas in tooling (SSMS, Azure Data Studio) becomes cumbersome. Consider sharding across multiple databases if you exceed 200-300 tenants.

### 9.8 Schema Creation at Tenant Registration

Creating a schema and running migrations must be transactional. If the migration fails, the schema should be removed:

```csharp
public async Task ProvisionTenantAsync(int tenantId)
{
    var schema = $"tenant_{tenantId}";

    using var transaction = await _connection.BeginTransactionAsync();
    try
    {
        await _connection.ExecuteAsync($"CREATE SCHEMA [{schema}]", transaction: transaction);
        await RunMigrationsForSchemaAsync(schema, transaction);
        await transaction.CommitAsync();
    }
    catch
    {
        await transaction.RollbackAsync();
        await _connection.ExecuteAsync($"DROP SCHEMA IF EXISTS [{schema}]");
        throw;
    }
}
```

### 9.9 Connection Pooling

Since all tenants share the same database, connection pooling is efficient — one pool per connection string. However, if you implement per-tenant connection strings (e.g., to route to different read replicas), pooling splits accordingly.

### 9.10 Backup and Restore

A single database backup includes all tenant schemas. To restore a single tenant, you must either:
- Restore the entire database and clean other tenants' data.
- Maintain per-tenant export/import procedures (e.g., `bcp` or `SqlPackage`).

### 9.11 Summary of Best Practices

| Practice | Recommendation |
|---|---|
| Schema naming | Use `tenant_{id}` or a UUID-based slug; normalize to alphanumeric + underscore |
| EF Core model cache | Implement `IModelCacheKeyFactory` with schema in the key |
| EF Core pooling | Avoid `AddDbContextPool` with multi-schema; use scoped `AddDbContext` |
| Dapper queries | Always use `[schema].[table]` — make it a habit via base class |
| Migration strategy | Run migrations per tenant, track per-schema migration history |
| Provisioning | Create schema and migrate in a transaction; roll back on failure |
| Cross-tenant queries | Use `UNION ALL` or a reporting aggregate; avoid in request path |
| Security | Sanitize and bracket schema names; use schema-level permissions |
| Soft delete | No `TenantId` needed — schema provides isolation |
| Unique constraints | Per-schema, no composite `TenantId` needed |
| Monitoring | Track per-schema size, migration status, and drift detection |
