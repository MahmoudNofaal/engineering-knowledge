# IQueryable vs IEnumerable — Database Execution

## 1. Overview — The Fundamental Difference

IQueryable<T> and IEnumerable<T> represent two fundamentally different execution models for LINQ queries. Understanding the difference between them is critical to writing performant database code.

- **IQueryable<T>** — builds an **expression tree** that is translated into SQL by the query provider (EF Core, LINQ to SQL, etc.). The query executes **on the database server**, and only the results are returned to the application. Filters, projections, and aggregations are translated to SQL WHERE, SELECT, GROUP BY, etc.

- **IEnumerable<T>** — executes **in memory** (LINQ to Objects). The entire result set must be loaded from the database first, and then all filtering, projection, and aggregation happens in the application's memory space.

`sql
-- IQueryable: filter translates to SQL WHERE
SELECT [o].[Id], [o].[CustomerId], [o].[OrderDate], [o].[Total]
FROM [Orders] AS [o]
WHERE [o].[Total] > 1000;

-- IEnumerable: loads ALL rows, then filters in memory
-- Step 1: SELECT [o].[Id], [o].[CustomerId], [o].[OrderDate], [o].[Total] FROM [Orders] AS [o];
-- Step 2: In memory: orders.Where(o => o.Total > 1000);
`

### 1.1 The Critical Performance Difference

`csharp
// GOOD: IQueryable — WHERE clause in SQL
IQueryable<Order> query = db.Orders;
var filtered = query.Where(o => o.Total > 1000);
var result = await filtered.ToListAsync();

// BAD: IEnumerable — loads entire table into memory
IEnumerable<Order> enumerable = db.Orders;
var filtered = enumerable.Where(o => o.Total > 1000);
var result = filtered.ToList();
`

The IEnumerable version loads **every row** from the Orders table into application memory, then discards most of them. For a table with 1,000,000 rows where only 100 have Total > 1000, you waste memory and network bandwidth on 999,900 rows.

### 1.2 Type System Summary

| Interface | Namespace | Execution | Translation | Deferred? |
|-----------|-----------|-----------|-------------|-----------|
| IQueryable<T> | System.Linq | Database (via SQL) | Expression tree → SQL | Yes |
| IEnumerable<T> | System.Collections.Generic | In-memory (LINQ to Objects) | Delegates / IL | Yes (for iteration) |
| IOrderedQueryable<T> | System.Linq | Database | IQueryable + ordering | Yes |
| IOrderedEnumerable<T> | System.Linq | In-memory | IEnumerable + ordering | Yes |

Both interfaces support **deferred execution** — the query is not executed until the result is iterated. However, the execution location and mechanism are completely different.

## 2. Section 2 — IQueryable: Expression Tree to SQL

### 2.1 How IQueryable Works

`IQueryable<T>` contains two critical components:

1. **Expression Tree** (`Expression` property) — a data structure representing the LINQ operations applied so far (Where, Select, OrderBy, etc.).
2. **Query Provider** (`Provider` property) — the component that translates the expression tree into a provider-specific query language (SQL for EF Core).

When EF Core's query provider receives the expression tree, it:

1. Walks the expression tree, visiting each node (method calls, lambdas, member accesses, constants).
2. Translates CLR method calls to SQL functions (e.g., `.Where(o => o.Total > 1000)` → `WHERE [o].[Total] > 1000`).
3. Translates C# expressions to SQL expressions (e.g., `o.Name.Contains("foo")` → `[o].[Name] LIKE N'%foo%'`).
4. Produces a parameterized SQL string.
5. Sends the SQL to the database and materializes results.

```csharp
// This is the IQueryable pipeline in EF Core:
IQueryable<Order> query = db.Orders
    .Where(o => o.Total > 1000)
    .OrderByDescending(o => o.OrderDate)
    .Select(o => new { o.Id, o.Total });

string sql = query.ToQueryString();
-- SELECT [o].[Id], [o].[Total]
-- FROM [Orders] AS [o]
-- WHERE [o].[Total] > 1000
-- ORDER BY [o].[OrderDate] DESC

var results = await query.ToListAsync();
```

### 2.2 Expression Tree vs Delegates

The key technical difference is that `IQueryable<T>` stores an **expression tree** (data), while `IEnumerable<T>` stores a **delegate** (code):

```csharp
// IQueryable: stores expression tree (can be inspected/translated)
Expression<Func<Order, bool>> expression = o => o.Total > 1000;

// IEnumerable: stores delegate (executable IL code)
Func<Order, bool> func = o => o.Total > 1000;
```

The `Expression<T>` can be analyzed and translated to SQL. The `Func<T>` can only be invoked — EF Core cannot translate it.

### 2.3 How the Compiler Selects IQueryable vs IEnumerable

LINQ uses **extension method resolution** based on the static type of the source:

```csharp
DbSet<Order> orders = db.Orders;
var queryable = orders.Where(o => o.Total > 1000);

IEnumerable<Order> asEnumerable = orders;
var filtered = asEnumerable.Where(o => o.Total > 1000);

List<Order> list = orders.ToList();
var memFiltered = list.Where(o => o.Total > 1000);
```

### 2.4 ToQueryString() — See the SQL Difference

```csharp
var query = db.Orders
    .Where(o => o.Total > 1000)
    .OrderBy(o => o.OrderDate)
    .Take(10);

var sql = query.ToQueryString();
-- SELECT TOP(10) [o].[Id], [o].[CustomerId], [o].[OrderDate], [o].[Total]
-- FROM [Orders] AS [o]
-- WHERE [o].[Total] > 1000
-- ORDER BY [o].[OrderDate]

// Compare with IEnumerable-equivalent:
var enumerable = (IEnumerable<Order>)db.Orders;
var filtered = enumerable.Where(o => o.Total > 1000).Take(10).OrderBy(o => o.OrderDate);
// No ToQueryString() — the SQL would be: SELECT * FROM Orders (full table scan!)
```

## 3. Section 3 — IEnumerable: In-Memory Execution

### 3.1 How IEnumerable Works with EF Core

```csharp
// Step 1: db.Orders cast to IEnumerable — EF Core executes SELECT * FROM Orders
IEnumerable<Order> allOrders = db.Orders;

// Step 2: All filtering in memory
var filtered = allOrders
    .Where(o => o.Total > 1000)
    .OrderByDescending(o => o.OrderDate)
    .Take(10);
```

### 3.2 When IEnumerable Is the Only Option

```csharp
var orders = await db.Orders.ToListAsync();

// 1. Custom C# method calls
var filtered = orders.Where(o => SomeComplexBusinessRule(o));

// 2. GroupBy with custom aggregation
var groups = orders.GroupBy(o => o.CustomerId).Select(g => new
{
    CustomerId = g.Key,
    Score = g.Sum(o => o.Total) * ComputeWeight(g.Key)
});

// 3. External services during projection
var results = orders.Select(o => new { o.Id, Status = _externalService.GetStatus(o.Id) }).ToList();
```

### 3.3 The Performance Trap

```csharp
// BAD: ToList before Where — loads ALL rows
var orders = (await db.Orders.ToListAsync()).Where(o => o.Total > 1000).ToList();
-- SQL: SELECT * FROM Orders (no WHERE!)

// GOOD: Where before ToList — filters in SQL
var orders = await db.Orders.Where(o => o.Total > 1000).ToListAsync();
-- SQL: SELECT * FROM Orders WHERE Total > 1000

// BAD: IEnumerable variable declaration
IEnumerable<Order> query = db.Orders;
query = query.Where(o => o.Total > 1000);

// GOOD: IQueryable variable declaration
IQueryable<Order> query = db.Orders;
query = query.Where(o => o.Total > 1000);
```

### 3.4 IEnumerable and Dapper

```csharp
// Dapper always returns in-memory collections
IEnumerable<Order> orders = await connection.QueryAsync<Order>(
    "SELECT * FROM Orders WHERE Total > @MinTotal",
    new { MinTotal = 1000 });

// Further filtering happens in memory
var filtered = orders.Where(o => o.OrderDate > DateTime.UtcNow.AddDays(-30));

// For SQL-level filtering, write a new query:
var recentOrders = await connection.QueryAsync<Order>(
    "SELECT * FROM Orders WHERE Total > @MinTotal AND OrderDate > @Since",
    new { MinTotal = 1000, Since = DateTime.UtcNow.AddDays(-30) });
```

## 5. Section 5 — Common Pitfall: Implicit ToList() and Client Evaluation

### 5.1 ToList() in Loops (N+1 Materialization)

```csharp
// BAD: Materializing data inside a loop
var customerIds = new[] { "CUST001", "CUST002", "CUST003" };

foreach (var customerId in customerIds)
{
    var orders = await db.Orders
        .Where(o => o.CustomerId == customerId)
        .ToListAsync();  // Separate DB round-trip per iteration!
    ProcessOrders(orders);
}
-- 3 SQL queries: SELECT ... FROM Orders WHERE CustomerId = 'CUST001'

// GOOD: Single round trip
var allOrders = await db.Orders
    .Where(o => customerIds.Contains(o.CustomerId))
    .ToListAsync();  // Single DB round trip
-- SQL: SELECT ... FROM Orders WHERE CustomerId IN ('CUST001', 'CUST002', 'CUST003')
```

### 5.2 Implicit Client Evaluation in Select

```csharp
// GOOD: All operations translatable to SQL
var result = await db.Orders
    .Select(o => new OrderDto
    {
        Id = o.Id,
        CustomerId = o.CustomerId,
        Total = o.Total,
        ItemCount = o.OrderItems.Count()
    })
    .ToListAsync();

-- SQL:
-- SELECT [o].[Id], [o].[CustomerId], [o].[Total],
--        (SELECT COUNT(*) FROM [OrderItems] AS [oi]
--         WHERE [o].[Id] = [oi].[OrderId]) AS [ItemCount]
-- FROM [Orders] AS [o]

// BAD: Non-translatable operations force client evaluation
var result = (await db.Orders.ToListAsync())  // Full table load!
    .Select(o => new OrderDto { Total = o.Total, ItemCount = o.OrderItems.Count })
    .ToList();
-- SQL: SELECT * FROM Orders (no subquery!)
```

### 5.3 Mixing IQueryable and IEnumerable in the Same Chain

```csharp
// Subtle bug: mixing interfaces
IQueryable<Order> query = db.Orders.Where(o => o.Total > 1000);

var result = query
    .AsEnumerable()                   // Switch to IEnumerable
    .Where(o => o.CustomerId != null) // Client-side!
    .OrderBy(o => o.OrderDate)        // Client-side!
    .Take(10)                         // Client-side!
    .ToList();

-- SQL: SELECT ... FROM Orders WHERE Total > 1000 (no ORDER BY, no TOP!)
-- The ORDER BY and TOP happen in memory after all matching rows are loaded.

// CORRECT: Keep everything on the server
var result = await db.Orders
    .Where(o => o.Total > 1000)
    .Where(o => o.CustomerId != null)
    .OrderBy(o => o.OrderDate)
    .Take(10)
    .ToListAsync();
-- SQL: SELECT TOP(10) ... WHERE Total > 1000 AND CustomerId IS NOT NULL ORDER BY OrderDate
```

### 5.4 The AsEnumerable / ToList Trap

```csharp
// THE TRAP: AsEnumerable used to "fix" a non-translatable query
var query = db.Orders
    .Where(o => o.Total > 1000)
    .Select(o => new { o.Id, o.Total, DiscountedTotal = ApplyDiscount(o.Total, o.CustomerId) });

// Developer "fixes" with AsEnumerable:
var result = db.Orders
    .Where(o => o.Total > 1000)
    .AsEnumerable()
    .Select(o => new { o.Id, o.Total, DiscountedTotal = ApplyDiscount(o.Total, o.CustomerId) })
    .ToList();

-- SQL: SELECT ... FROM Orders WHERE Total > 1000 (ApplyDiscount runs in C#)

// BETTER: Filter on server, project on client
var query = await db.Orders
    .Where(o => o.Total > 1000)
    .Select(o => new { o.Id, o.Total, o.CustomerId })
    .ToListAsync();

var result = query.Select(o => new
{
    o.Id, o.Total,
    DiscountedTotal = ApplyDiscount(o.Total, o.CustomerId)
}).ToList();
```

## 6. Section 6 — EF Core IQueryable Pipeline in Detail

### 6.1 Query Lifecycle

```csharp
// Full lifecycle of an EF Core IQueryable query:

// 1. Query Definition (expression tree construction)
IQueryable<Order> query = db.Orders
    .Where(o => o.Total > 1000 && o.IsDeleted == false)
    .OrderBy(o => o.OrderDate)
    .Skip(20)
    .Take(10);

// 2. Query Compilation
// EF Core validates the expression tree
// Identifies translatable vs non-translatable nodes
// EF Core 3+ throws for non-translatable nodes

// 3. SQL Generation
var sql = query.ToQueryString();
-- SELECT [o].[Id], [o].[CustomerId], [o].[OrderDate], [o].[Total], [o].[IsDeleted]
-- FROM [Orders] AS [o]
-- WHERE [o].[Total] > 1000 AND [o].[IsDeleted] = 0
-- ORDER BY [o].[OrderDate]
-- OFFSET 20 ROWS FETCH NEXT 10 ROWS ONLY

// 4. Database Execution
var orders = await query.ToListAsync();

// 5. Change Tracking (if not AsNoTracking)
// Each materialized entity added to ChangeTracker with snapshot
```

### 6.2 Compiled Queries (EF.CompileQuery)

Caches expression tree to SQL translation for frequently-used queries:

```csharp
private static readonly Func<SalesDbContext, decimal, int, Task<List<Order>>>
    GetHighValueOrders = EF.CompileAsyncQuery(
        (SalesDbContext db, decimal minTotal, int take) =>
            db.Orders
                .Where(o => o.Total > minTotal)
                .OrderByDescending(o => o.Total)
                .Take(take)
                .ToList());

// Translation cached — only parameters change
var orders1 = await GetHighValueOrders(db, 1000m, 10);
var orders2 = await GetHighValueOrders(db, 5000m, 5);

-- All calls generate same SQL with different param values:
-- SELECT TOP(@take) [o].[Id], [o].[CustomerId], [o].[OrderDate], [o].[Total]
-- FROM [Orders] AS [o]
-- WHERE [o].[Total] > @minTotal
-- ORDER BY [o].[Total] DESC
```

### 6.3 IQueryable and AsNoTracking

```csharp
// With tracking (default):
var orders = await db.Orders.Where(o => o.Total > 1000).ToListAsync();
// Each entity added to ChangeTracker — more memory, slower

// Without tracking:
var orders = await db.Orders
    .AsNoTracking()
    .Where(o => o.Total > 1000)
    .ToListAsync();
// Entities materialized but NOT tracked — faster, less memory

// Identity resolution (EF Core 6+):
var orders = await db.Orders
    .AsNoTrackingWithIdentityResolution()
    .Include(o => o.OrderItems)
    .Where(o => o.Total > 1000)
    .ToListAsync();
```

### 6.4 IQueryable and AsSplitQuery

```csharp
// Single query — Cartesian product (dangerous with large collections)
var orders = await db.Orders
    .Include(o => o.OrderItems)
    .Include(o => o.Shipments)
    .ToListAsync();

// Split query — one SQL per Include
var orders = await db.Orders
    .Include(o => o.OrderItems)
    .Include(o => o.Shipments)
    .AsSplitQuery()
    .ToListAsync();

-- Query 1: SELECT ... FROM Orders
-- Query 2: SELECT ... FROM OrderItems WHERE OrderId IN (...)
-- Query 3: SELECT ... FROM Shipments WHERE OrderId IN (...)
```

## 7. Section 7 — Dapper: Always In-Memory Collections

### 7.1 Dapper Query Methods Return IEnumerable

```csharp
// All Dapper methods execute immediately and return materialized collections:
Task<IEnumerable<T>> QueryAsync<T>(...)
Task<T> QueryFirstOrDefaultAsync<T>(...)
Task<T> QuerySingleAsync<T>(...)
Task<IEnumerable<TReturn>> QueryAsync<T1,T2,TReturn>(...)

var orders = await connection.QueryAsync<Order>(
    "SELECT * FROM Orders WHERE Total > @Min",
    new { Min = 1000 });
// orders is IEnumerable<Order> — already in memory, no deferred execution
```

### 7.2 LINQ on Dapper Results = Client-Side

```csharp
var orders = await connection.QueryAsync<Order>(sql);

// These all happen in memory:
var highValue = orders.Where(o => o.Total > 5000);       // Client-side
var sorted = highValue.OrderByDescending(o => o.Total);  // Client-side sort
var top10 = sorted.Take(10);                              // Client-side limit
var summary = orders.GroupBy(o => o.CustomerId)           // Client-side grouping
    .Select(g => new { CustomerId = g.Key, Sum = g.Sum(o => o.Total) });

// The SQL loaded ALL matching rows — no TOP, no GROUP BY in SQL!
```

### 7.3 Performance Impact: Dapper vs EF Core IQueryable

```csharp
// Dapper with optimal SQL (one round trip, filtered):
var sql = @"SELECT TOP(10) Id, CustomerId, Total, OrderDate
             FROM Orders
             WHERE CustomerId = @Cust AND IsDeleted = 0
             ORDER BY Total DESC";
var top10 = await connection.QueryAsync<Order>(sql, new { Cust = "CUST001" });
-- Optimal: only 10 rows transferred

// Dapper with suboptimal SQL (client-side processing):
var sql = @"SELECT Id, CustomerId, Total, OrderDate FROM Orders
             WHERE CustomerId = @Cust AND IsDeleted = 0";
var all = await connection.QueryAsync<Order>(sql, new { Cust = "CUST001" });
var top10 = all.OrderByDescending(o => o.Total).Take(10).ToList();
-- BAD: all matching rows (maybe 500+) transferred, then sorted in memory
```

### 7.4 Dapper With QueryMultiple for Related Data

```csharp
using var multi = await connection.QueryMultipleAsync(@"
    SELECT * FROM Orders WHERE CustomerId = @Cust;
    SELECT * FROM OrderItems WHERE OrderId IN (
        SELECT Id FROM Orders WHERE CustomerId = @Cust);
", new { Cust = "CUST001" });

var orders = (await multi.ReadAsync<Order>()).ToList();
var items = (await multi.ReadAsync<OrderItem>()).ToList();
// Both orders and items are in-memory lists — no IQueryable composition
```

### 7.5 When Dapper Wins on Performance

```csharp
// EF Core: expression tree -> translation -> SQL generation -> execution
var efResult = await db.Orders
    .Where(o => o.Id == id)
    .Select(o => new { o.Id, o.Total })
    .FirstOrDefaultAsync();

// Dapper: raw SQL -> direct execution (no query pipeline overhead)
var dapperResult = await connection.QueryFirstOrDefaultAsync(
    "SELECT Id, Total FROM Orders WHERE Id = @Id",
    new { Id = id });
```

## 8. Section 8 — Debugging IQueryable vs IEnumerable Bugs

### 8.1 Using ToQueryString() to Detect Client Evaluation

```csharp
var query = db.Orders.Where(o => o.Total > 1000).OrderBy(o => o.OrderDate);
var sql = query.ToQueryString();
Console.WriteLine(sql);
-- SELECT [o].[Id], [o].[CustomerId], [o].[OrderDate], [o].[Total]
-- FROM [Orders] AS [o]
-- WHERE [o].[Total] > 1000
-- ORDER BY [o].[OrderDate]

// If you see no WHERE clause when you expected one -> client evaluation!
var badQuery = (IEnumerable<Order>)db.Orders;
var filtered = badQuery.Where(o => o.Total > 1000);
// Can't call ToQueryString() on IEnumerable!
```

### 8.2 EF Core 3+ Throws for Non-Translatable Queries

```csharp
// EF Core 3.0+ throws InvalidOperationException:
var orders = await db.Orders
    .Where(o => SomeLegacyFunction(o.CustomerId))
    .ToListAsync();
-- System.InvalidOperationException:
-- The LINQ expression could not be translated.
-- Either rewrite the query or insert a call to AsEnumerable/ToList.
```

## 9. Section 9 — Gotchas, Pitfalls, and Best Practices

### 9.1 List.Where() Is Always IEnumerable (Client Evaluation)

```csharp
// COMMON BUG: ToList() breaks IQueryable chain
List<Order> list = await db.Orders.ToListAsync();
var filtered = list.Where(o => o.Total > 1000);    // WHERE in memory!
var sorted = filtered.OrderBy(o => o.OrderDate);   // SORT in memory!

// FIX: Keep IQueryable as long as possible
IQueryable<Order> query = db.Orders;
query = query.Where(o => o.Total > 1000);          // WHERE in SQL
query = query.OrderBy(o => o.OrderDate);           // ORDER BY in SQL
var result = await query.ToListAsync();
```

### 9.2 Mixing IQueryable and IEnumerable Causes Table Scans

```csharp
// Once IEnumerable, always IEnumerable
// Once you cross the boundary, all subsequent operations are in-memory

// INSIDIOUS BUG: Variable type determines execution
IQueryable<Order> query = db.Orders.Where(o => o.Total > 1000);

if (someCondition)
{
    query = query.Where(o => o.CustomerId != null); // Still IQueryable
}

// But if someone writes:
IEnumerable<Order> query = db.Orders.Where(o => o.Total > 1000);
query = query.Where(o => o.CustomerId != null);    // IEnumerable.Where!
// Full table scan!

// Always declare as IQueryable<T> explicitly:
IQueryable<Order> query = db.Orders;  // BE EXPLICIT!
```

### 9.3 Debugging Triggers Client Evaluation

```csharp
// BUG: Inspecting IQueryable in debugger materializes it!
var query = db.Orders.Where(o => o.Total > 1000);

// If you hover over 'query' in debugger, it executes the query!
// This causes unexpected database round-trips.

// FIX: Use ToQueryString() to inspect SQL without executing
var sql = query.ToQueryString();  // No DB round-trip
```

### 9.4 Implicit ToList() in Loops (Repeated Execution)

```csharp
// BAD: Repeated materialization in loop
foreach (var customerId in customerIds)
{
    var orders = await db.Orders
        .Where(o => o.CustomerId == customerId)
        .ToListAsync();  // Executes every iteration!
}

// GOOD: Batch once
var allOrders = await db.Orders
    .Where(o => customerIds.Contains(o.CustomerId))
    .ToListAsync();

// WORSE: IEnumerable deferred execution re-executes the query
IEnumerable<Order> query = db.Orders.Where(o => o.Total > 1000);
var count = query.Count();   // First execution
var list = query.ToList();   // Second execution!
// BOTH queries are sent to the database!
```

### 9.5 Dapper — No Query Composition

```csharp
// BAD: Loading data and filtering in memory
var allOrders = await connection.QueryAsync<Order>("SELECT * FROM Orders");
var recent = allOrders.Where(o => o.OrderDate > cutoff);

// GOOD: Filter in SQL
var recentOrders = await connection.QueryAsync<Order>(
    "SELECT * FROM Orders WHERE OrderDate > @Cutoff",
    new { Cutoff = cutoff });

// BAD: Loading all columns then projecting
var heavy = await connection.QueryAsync<Order>("SELECT * FROM Orders WHERE Total > 1000");
var light = heavy.Select(o => new { o.Id, o.Total });

// GOOD: Select only needed columns
var light = await connection.QueryAsync(
    "SELECT Id, Total FROM Orders WHERE Total > 1000");
```

### 9.6 EF Core 3+ Breaking Change (No Implicit Client Evaluation)

```csharp
// EF Core 2.x: silently client-evaluated
var orders = await db.Orders.Where(o => MyFunc(o.Total)).ToListAsync();

// EF Core 3.0+: throws InvalidOperationException
var orders = await db.Orders.Where(o => MyFunc(o.Total)).ToListAsync(); // THROWS!

// Fix: Materialize first, then apply in-memory function
var orders = await db.Orders.ToListAsync();
var filtered = orders.Where(o => MyFunc(o.Total)).ToList();

// Or use AsEnumerable to explicitly split:
var orders = db.Orders.Where(o => o.Total > 1000)
    .AsEnumerable()  // Explicit split: server up to here, client after
    .Where(o => MyFunc(o.Total))
    .ToList();
```

### 9.7 IQueryable and Async Methods

```csharp
// WRONG: Mixing async and sync
var query = db.Orders.Where(o => o.Total > 1000);
var list = query.ToList();  // Sync — blocks thread!

// CORRECT: Use async materialization
var list = await query.ToListAsync();
var array = await query.ToArrayAsync();
var first = await query.FirstOrDefaultAsync();
var count = await query.CountAsync();

// CORRECT: await only at materialization
var filtered = await db.Orders
    .Where(o => o.Total > 1000)
    .ToListAsync();
```

### 9.9 Compiled Queries vs IQueryable Flexibility

| Aspect | Compiled Query | IQueryable |
|--------|---------------|------------|
| Translation caching | Cached after first execution | Re-translated every time |
| Dynamic composition | Fixed structure, only params vary | Full dynamic composition |
| Code complexity | More boilerplate | Standard LINQ |
| Use case | Hot path, same shape | Ad-hoc, dynamic queries |

```csharp
// Compiled query: fast but rigid
var orders = await GetOrdersByCustomer(db, "CUST001", 10);

// IQueryable: flexible but re-translates
var orders = await db.Orders
    .Where(o => o.CustomerId == "CUST001")
    .Take(10)
    .ToListAsync();
```

### 9.10 Summary of Best Practices

| Rule | Best Practice |
|------|---------------|
| Variable type | Always declare as `IQueryable<T>` (not `var` when the type matters) |
| Materialization | Call `ToListAsync()` only at the very end of the query chain |
| Filter placement | Put `Where` filters BEFORE `ToListAsync()` |
| Loop queries | Never call `ToListAsync()` inside a loop |
| Debug inspection | Use `ToQueryString()` instead of hovering in debugger |
| IEnumerable boundary | Once `AsEnumerable()` or `ToList()` is called, all ops are client-side |
| EF Core 3+ | Non-translatable queries throw — use `AsEnumerable` deliberately |
| Dapper | Filter in SQL, not in C# — every Dapper result is in-memory |
| Mixed approaches | Keep IQueryable for EF, raw SQL for Dapper |
| Compiled queries | Use `EF.CompileQuery` for hot-path queries with fixed shape |
| AsSplitQuery | Use when loading multiple collections to avoid Cartesian explosion |
| AsNoTracking | Use for read-only queries to avoid change tracking overhead |
| Projection | Use `Select` to return only needed columns |
| Indexing | Ensure filtered columns used in Where have covering indexes |
| Monitoring | Log slow queries; watch for SELECT * with no WHERE on large tables |

### 9.11 Decision Matrix: IQueryable vs IEnumerable vs Dapper

| Scenario | IQueryable (EF Core) | IEnumerable (EF Core) | Dapper |
|----------|---------------------|----------------------|--------|
| Simple WHERE filter | Best (SQL WHERE) | Loads all rows | Best (write SQL directly) |
| Complex joins | Translates to JOIN | N/A (already loaded) | Manual JOIN SQL |
| Client-side logic | Must use AsEnumerable | Works naturally | Works naturally |
| Bulk updates | ExecuteUpdateAsync | Load then update | ExecuteAsync with UPDATE |
| Dynamic query building | Composability | In-memory only | String concatenation |
| Performance (raw speed) | Medium | Slow (memory + bandwidth) | Fast |
| Development speed | Fast | Slow (bugs) | Medium |
| Safety (filter correctness) | High (compile-time) | Low | Low (runtime) |
| Compiled/cached queries | EF.CompileQuery | N/A | N/A |
| Stored procedures | FromSql | N/A | Best |
