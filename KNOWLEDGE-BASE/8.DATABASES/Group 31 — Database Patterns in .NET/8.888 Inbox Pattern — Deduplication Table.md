# 8.888 — Inbox Pattern — Deduplication Table

## 1. Overview — What and Why




## 1. Overview — What and Why

The Inbox pattern provides idempotent message processing on the consumer side. When a message broker delivers a message (possibly multiple times due to at-least-once delivery guarantees, producer retries, or network issues), the consumer uses an inbox table to track which messages have already been processed. Before processing a message, the consumer checks the inbox. If the message ID exists, the message is skipped (or a cached result is returned). If not, the message is processed and the ID is recorded in the same transaction.

The inbox table is the consumer-side counterpart to the outbox table (see [[8.886 — Outbox Pattern — Database Implementation]]). Together, they provide end-to-end at-least-once delivery with idempotent processing, which is the closest practical approximation to exactly-once processing in distributed systems.

## 2. Problem Statement — Duplicate Message Delivery

Message brokers guarantee at-least-once delivery by default. This means:

- The **Outbox publisher** might publish the same message twice (crash after publish, before marking as processed).
- The **Broker** might redeliver a message if the consumer does not acknowledge it in time.
- The **Consumer** might crash after processing but before acknowledging the message to the broker.

Without deduplication, these scenarios cause:
- **Duplicate orders**: A payment event processed twice creates two charges.
- **Inconsistent state**: The same event applied twice to an aggregate produces incorrect state.
- **Idempotency violations**: Non-idempotent side effects (emails, API calls, file writes) executed multiple times.

The inbox pattern solves this by recording processed message IDs in the consumer's database — durable, transactional, and queryable.

## 3. Solution Architecture — Inbox with Deduplication

The inbox pattern at the database level uses an InboxMessages table. The consumer:

1. **Receives** a message from the broker.
2. **Checks** the inbox table for the message ID.
3. If **found**: skips processing (already processed).
4. If **not found**:
   a. **Processes** the message (updates business data).
   b. **Records** the message ID in the inbox table (same transaction).
   c. **Acknowledges** the message to the broker.
5. If the consumer crashes between (a) and (c), the broker redelivers, and the inbox prevents reprocessing.

### 4.2 Inbox with Deduplication Check Stored Procedure

`sql
-- ============================================================
-- Inbox Pattern — TryConsume (Atomic Check + Insert)
-- ============================================================

CREATE OR ALTER PROCEDURE [Messaging].[TryConsumeMessage]
    @MessageId       NVARCHAR(256),
    @MessageType     NVARCHAR(500),
    @Content         NVARCHAR(MAX),
    @ConsumerName    NVARCHAR(200),
    @CorrelationId   NVARCHAR(100) = NULL
AS
BEGIN
    SET NOCOUNT ON;

    -- Check if already processed
    IF EXISTS (SELECT 1 FROM [Messaging].[InboxMessages]
               WHERE [MessageId] = @MessageId)
    BEGIN
        SELECT CAST(0 AS BIT) AS [Consumed];  -- Already processed
        RETURN;
    END;

    -- Insert and indicate first-time consumption
    INSERT INTO [Messaging].[InboxMessages]
        ([MessageId], [MessageType], [Content], [ConsumerName],
         [CorrelationId], [ReceivedAt], [ProcessedAt])
    VALUES
        (@MessageId, @MessageType, @Content, @ConsumerName,
         @CorrelationId, SYSUTCDATETIME(), SYSUTCDATETIME());

    SELECT CAST(1 AS BIT) AS [Consumed];  -- Newly consumed
END;
GO
`

### 4.3 Inbox Cleanup Stored Procedure

`sql
-- ============================================================
-- Inbox Pattern — Cleanup Processed Messages
-- ============================================================

CREATE OR ALTER PROCEDURE [Messaging].[CleanupInboxMessages]
    @RetentionHours   INT = 168,  -- 7 days
    @BatchSize        INT = 1000
AS
BEGIN
    SET NOCOUNT ON;

    DECLARE @CutoffDate DATETIME2(7) = DATEADD(HOUR, -@RetentionHours, SYSUTCDATETIME());
    DECLARE @Deleted INT = 1;
    DECLARE @TotalDeleted INT = 0;

    WHILE @Deleted > 0
    BEGIN
        DELETE TOP (@BatchSize) FROM [Messaging].[InboxMessages]
        WHERE [ProcessedAt] < @CutoffDate;

        SET @Deleted = @@ROWCOUNT;
        SET @TotalDeleted = @TotalDeleted + @Deleted;
    END;

    SELECT @TotalDeleted AS DeletedCount;
END;
GO
`

CREATE OR ALTER VIEW [Messaging].[vw_InboxStatistics]
AS
SELECT
    [ConsumerName],
    [MessageType],
    COUNT_BIG(*)                                    AS TotalProcessed,
    MIN([ProcessedAt])                              AS FirstProcessed,
    MAX([ProcessedAt])                              AS LastProcessed,
    AVG(CAST([ProcessingTimeMs] AS BIGINT))         AS AvgProcessingTimeMs,
    MAX(CAST([ProcessingTimeMs] AS BIGINT))         AS MaxProcessingTimeMs,
    COUNT_BIG(CASE WHEN [Error] IS NOT NULL THEN 1 END) AS ErrorCount
FROM [Messaging].[InboxMessages]
GROUP BY [ConsumerName], [MessageType];
GO

### 5.3 Inbox Repository — EF Core

`csharp
// ============================================================
// Inbox — EF Core Repository with Deduplication
// ============================================================

namespace Infrastructure.Persistence.InboxRepositories;

using Domain.Messaging;
using Infrastructure.Data;
using Microsoft.EntityFrameworkCore;

public sealed class EfCoreInboxRepository
{
    private readonly SalesDbContext _context;

    public EfCoreInboxRepository(SalesDbContext context)
    {
        _context = context ?? throw new ArgumentNullException(nameof(context));
    }

    public async Task<bool> TryConsumeAsync(
        string messageId,
        string messageType,
        string content,
        string consumerName,
        string? correlationId = null,
        CancellationToken ct = default)
    {
        // Check if already processed
        var exists = await _context.Set<InboxMessage>()
            .AnyAsync(i => i.MessageId == messageId, ct);

        if (exists)
            return false;  // Already consumed

        // Insert new inbox record
        var inboxMessage = new InboxMessage(
            messageId, messageType, content, consumerName, correlationId);

        _context.Set<InboxMessage>().Add(inboxMessage);

        try
        {
            await _context.SaveChangesAsync(ct);
            return true;  // Newly consumed
        }
        catch (DbUpdateException ex)
            when (ex.InnerException?.Message.Contains("UQ_InboxMessages_MessageId") == true)
        {
            // Race condition — another consumer inserted the same message
            return false;
        }
    }

    public async Task<bool> TryConsumeWithBusinessTransactionAsync(
        string messageId,
        string messageType,
        string content,
        string consumerName,
        Func<SalesDbContext, Task> businessAction,
        string? correlationId = null,
        CancellationToken ct = default)
    {
        // Strategy: Use a single transaction for inbox insert + business logic
        await using var transaction = await _context.Database
            .BeginTransactionAsync(ct);

        try
        {
            // Check first (within the transaction)
            var exists = await _context.Set<InboxMessage>()
                .AnyAsync(i => i.MessageId == messageId, ct);

            if (exists)
            {
                await transaction.RollbackAsync(ct);
                return false;
            }

            // Insert inbox record
            var inboxMessage = new InboxMessage(
                messageId, messageType, content, consumerName, correlationId);
            _context.Set<InboxMessage>().Add(inboxMessage);

            // Execute business action (updates orders, etc.)
            await businessAction(_context);

            // Save both in one transaction
            await _context.SaveChangesAsync(ct);
            await transaction.CommitAsync(ct);

            return true;
        }
        catch
        {
            await transaction.RollbackAsync(ct);
            throw;
        }
    }
}
`

    private readonly EfCoreInboxRepository _inboxRepo;
    private readonly ILogger<OrderConfirmedEventConsumer> _logger;

    public OrderConfirmedEventConsumer(
        SalesDbContext context,

        _context = context;
        _inboxRepo = inboxRepo;
        _logger = logger;
    }

    public async Task HandleAsync(
        string messageId,
        string content,
        CancellationToken ct = default)
    {
        var consumed = await _inboxRepo.TryConsumeWithBusinessTransactionAsync(
            messageId,
            typeof(OrderConfirmedEvent).FullName!,
            content,
            "shipping-service",
            async (ctx) =>
            {
                var evt = JsonSerializer.Deserialize<OrderConfirmedEvent>(content);
                if (evt is null)
                {
                    _logger.LogWarning("Failed to deserialize OrderConfirmedEvent from message {Id}", messageId);
                    return;

                // Business logic: create a shipment record
                var shipment = new Shipment
                {
                    OrderId = evt.OrderId,
                    Status = ShipmentStatus.Pending,
                    CreatedAt = DateTime.UtcNow

                ctx.Set<Shipment>().Add(shipment);

                _logger.LogInformation("Created shipment for order {OrderId} (message {Id})",
                    evt.OrderId, messageId);
            },
            ct: ct);

        if (consumed)
        {
            _logger.LogDebug("Message {Id} consumed successfully.", messageId);
        }
        else
        {
            _logger.LogDebug("Message {Id} was already consumed (duplicate).", messageId);
        }
    }
}

### 6.1 Dapper Inbox Repository

`csharp
// ============================================================
// Inbox — Dapper Repository with Deduplication
// ============================================================

namespace Infrastructure.Persistence.DapperInbox;

using System.Data;
using System.Text.Json;
using Dapper;
using Microsoft.Data.SqlClient;

public sealed class DapperInboxRepository
{
    private readonly string _connectionString;

    public DapperInboxRepository(string connectionString)
    {
        _connectionString = connectionString ?? throw new ArgumentNullException(nameof(connectionString));
    }

    private IDbConnection CreateConnection() => new SqlConnection(_connectionString);

    /// <summary>
    /// Atomically checks if a message was already processed and inserts it if not.
    /// Uses a unique constraint and handles the duplicate key error for race conditions.
    /// </summary>
    public async Task<bool> TryConsumeAsync(
        string messageId,
        string messageType,
        string content,
        string consumerName,
        string? correlationId = null,
        CancellationToken ct = default)
    {
        const string sql = @"
            IF NOT EXISTS (SELECT 1 FROM [Messaging].[InboxMessages]
                           WHERE [MessageId] = @MessageId)
            BEGIN
                INSERT INTO [Messaging].[InboxMessages]
                    ([MessageId], [MessageType], [Content], [ConsumerName],
                     [CorrelationId], [ReceivedAt], [ProcessedAt])
                VALUES
                    (@MessageId, @MessageType, @Content, @ConsumerName,
                     @CorrelationId, SYSUTCDATETIME(), SYSUTCDATETIME());

                SELECT CAST(1 AS BIT) AS [Consumed];
            END
            ELSE
            BEGIN
                SELECT CAST(0 AS BIT) AS [Consumed];
            END;";

        using var db = CreateConnection();

        var consumed = await db.ExecuteScalarAsync<bool>(
            new CommandDefinition(sql, new
            {
                MessageId = messageId,
                MessageType = messageType,
                Content = content,
                ConsumerName = consumerName,
                CorrelationId = correlationId
            }, cancellationToken: ct));

        return consumed;
    }

    /// <summary>
    /// Consumes a message with business logic in the same transaction.
    /// The inbox insert and business operations are atomic.
    /// </summary>
    public async Task<bool> TryConsumeWithBusinessTransactionAsync(
        string messageId,
        string messageType,
        string content,
        string consumerName,
        Func<IDbConnection, IDbTransaction, Task> businessAction,
        string? correlationId = null,
        CancellationToken ct = default)
    {
        using var db = CreateConnection();
        await db.OpenAsync(ct);
        await using var transaction = db.BeginTransaction(IsolationLevel.ReadCommitted);

        try
        {
            // Check if already processed (within the transaction)
            var exists = await db.ExecuteScalarAsync<bool>(
                "SELECT CAST(COUNT_BIG(1) AS BIT) FROM [Messaging].[InboxMessages] WHERE [MessageId] = @MessageId",
                new { MessageId = messageId },
                transaction,
                commandTimeout: 10);

            if (exists)
            {
                await transaction.RollbackAsync(ct);
                return false;
            }

            // Insert inbox record
            const string insertSql = @"
                INSERT INTO [Messaging].[InboxMessages]
                    ([MessageId], [MessageType], [Content], [ConsumerName],
                     [CorrelationId], [ReceivedAt], [ProcessedAt])
                VALUES
                    (@MessageId, @MessageType, @Content, @ConsumerName,
                     @CorrelationId, SYSUTCDATETIME(), SYSUTCDATETIME());";

            await db.ExecuteAsync(new CommandDefinition(insertSql, new
            {
                MessageId = messageId,
                MessageType = messageType,
                Content = content,
                ConsumerName = consumerName,
                CorrelationId = correlationId
            }, transaction: transaction, cancellationToken: ct));

            // Execute business action within the same transaction
            await businessAction(db, transaction);

            transaction.Commit();
            return true;
        }
        catch
        {
            transaction.Rollback();
            throw;
        }
    }

    /// <summary>
    /// Bulk check: given a set of message IDs, returns which ones have NOT been processed.
    /// Useful for batch processing scenarios.
    /// </summary>
    public async Task<HashSet<string>> GetUnprocessedIdsAsync(
        IEnumerable<string> messageIds,
        CancellationToken ct = default)
    {
        const string sql = @"
            SELECT [MessageId]
            FROM [Messaging].[InboxMessages]
            WHERE [MessageId] IN @MessageIds;";

        using var db = CreateConnection();

        var processedIds = await db.QueryAsync<string>(
            new CommandDefinition(sql,
                new { MessageIds = messageIds.ToList() },
                cancellationToken: ct));

        var processedSet = new HashSet<string>(processedIds, StringComparer.OrdinalIgnoreCase);
        return processedSet;
    }
}

### 6.2 Dapper Consumer — Kafka Consumer with Inbox

`csharp
// ============================================================
// Inbox — Dapper Kafka Consumer with Idempotency
// ============================================================

namespace Infrastructure.Consumers.DapperInbox;

using System.Text.Json;
using Confluent.Kafka;
using Infrastructure.Persistence.DapperInbox;
using Microsoft.Extensions.Hosting;
using Microsoft.Extensions.Logging;
using Microsoft.Extensions.Options;

public sealed class KafkaInboxConsumer : BackgroundService
{
    private readonly IConsumer<string, string> _consumer;
    private readonly DapperInboxRepository _inboxRepo;
    private readonly ILogger<KafkaInboxConsumer> _logger;
    private readonly string _consumerName;

    public KafkaInboxConsumer(
        IOptions<KafkaConsumerOptions> options,
        DapperInboxRepository inboxRepo,
        ILogger<KafkaInboxConsumer> logger)
    {
        _inboxRepo = inboxRepo;
        _logger = logger;
        _consumerName = $"KafkaInbox-{Environment.MachineName}";

        var config = new ConsumerConfig
        {
            BootstrapServers = options.Value.BootstrapServers,
            GroupId = options.Value.GroupId,
            AutoOffsetReset = AutoOffsetReset.Earliest,
            EnableAutoCommit = false,  // Manual commit for transactional safety
            EnablePartitionEof = false
        };

        _consumer = new ConsumerBuilder<string, string>(config)
            .Build();
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        _consumer.Subscribe("order-events");

        _logger.LogInformation(
            "Kafka inbox consumer '{Consumer}' subscribed to 'order-events'.", _consumerName);

        while (!stoppingToken.IsCancellationRequested)
        {
            try
            {
                var consumeResult = _consumer.Consume(stoppingToken);

                if (consumeResult is null || consumeResult.IsPartitionEOF)
                    continue;

                // The Kafka message key serves as the deduplication ID
                var messageId = consumeResult.Message.Key;
                var messageType = consumeResult.Message.Headers
                    .FirstOrDefault(h => h.Key == "event-type")?.GetValue() is byte[] bytes
                    ? System.Text.Encoding.UTF8.GetString(bytes)
                    : "Unknown";

                var content = consumeResult.Message.Value;

                _logger.LogDebug("Received message {Id} of type {Type}", messageId, messageType);

                // Try to consume with deduplication
                var consumed = await _inboxRepo.TryConsumeWithBusinessTransactionAsync(
                    messageId,
                    messageType,
                    content,
                    _consumerName,
                    async (db, tx) =>
                    {
                        // Business logic: deserialize and process
                        var evt = JsonSerializer.Deserialize<OrderConfirmedEvent>(content);
                        if (evt is null) return;

                        const string insertShipmentSql = @"
                            INSERT INTO [Shipping].[Shipments]
                                ([OrderId], [Status], [CreatedAt])
                            VALUES
                                (@OrderId, 0, SYSUTCDATETIME());";

                        await db.ExecuteAsync(insertShipmentSql,
                            new { evt.OrderId }, tx);

                        _logger.LogInformation("Created shipment for order {OrderId}", evt.OrderId);
                    },
                    ct: stoppingToken);

                if (consumed)
                {
                    // Commit the Kafka offset (mark as delivered)
                    _consumer.Commit(consumeResult);
                    _logger.LogDebug("Message {Id} consumed and committed.", messageId);
                }
                else
                {
                    // Already processed — still commit the offset to skip it
                    _consumer.Commit(consumeResult);
                    _logger.LogDebug("Message {Id} was duplicate — committed offset.", messageId);
                }
            }
            catch (OperationCanceledException) when (stoppingToken.IsCancellationRequested)
            {
                break;
            }
            catch (ConsumeException ex)
            {
                _logger.LogError(ex, "Kafka consume error");
                await Task.Delay(1000, stoppingToken);
            }
            catch (Exception ex)
            {
                _logger.LogError(ex, "Error processing Kafka message");
                // Do NOT commit — message will be redelivered
            }
        }

        _consumer.Close();
    }

    public override void Dispose()
    {
        _consumer?.Dispose();
        base.Dispose();
    }
}

public sealed class KafkaConsumerOptions
{
    public string BootstrapServers { get; set; } = "localhost:9092";
    public string GroupId { get; set; } = "inbox-consumer-group";
}

`csharp
// ============================================================
// Inbox — Dapper RabbitMQ Consumer with Idempotency
// ============================================================

namespace Infrastructure.Consumers.DapperInbox;

using System.Text;
using System.Text.Json;
using Infrastructure.Persistence.DapperInbox;
using Microsoft.Extensions.Hosting;
using Microsoft.Extensions.Logging;
using RabbitMQ.Client;
using RabbitMQ.Client.Events;

public sealed class RabbitMqInboxConsumer : BackgroundService
{
    private readonly IConnection _connection;
    private readonly IModel _channel;
    private readonly DapperInboxRepository _inboxRepo;
    private readonly ILogger<RabbitMqInboxConsumer> _logger;
    private readonly string _consumerName;

    public RabbitMqInboxConsumer(
        DapperInboxRepository inboxRepo,
        ILogger<RabbitMqInboxConsumer> logger)
    {
        _inboxRepo = inboxRepo;
        _logger = logger;
        _consumerName = $"RabbitInbox-{Environment.MachineName}";

        var factory = new ConnectionFactory { HostName = "localhost" };
        _connection = factory.CreateConnection();
        _channel = _connection.CreateModel();

        _channel.QueueDeclare("order-events", durable: true,
            exclusive: false, autoDelete: false);
    }

    protected override Task ExecuteAsync(CancellationToken stoppingToken)
    {
        var consumer = new EventingBasicConsumer(_channel);

        consumer.Received += async (model, ea) =>
        {
            var body = ea.Body.ToArray();
            var messageId = ea.BasicProperties?.MessageId ?? Guid.NewGuid().ToString();
            var messageType = ea.BasicProperties?.Type ?? "Unknown";
            var content = Encoding.UTF8.GetString(body);

            try
            {
                var consumed = await _inboxRepo.TryConsumeWithBusinessTransactionAsync(
                    messageId,
                    messageType,
                    content,
                    _consumerName,
                    async (db, tx) =>
                    {
                        var evt = JsonSerializer.Deserialize<OrderConfirmedEvent>(content);
                        if (evt is null) return;

                        const string sql = @"
                            INSERT INTO [Shipping].[Shipments]
                                ([OrderId], [Status], [CreatedAt])
                            VALUES
                                (@OrderId, 0, SYSUTCDATETIME());";

                        await db.ExecuteAsync(sql, new { evt.OrderId }, tx);
                    },
                    ct: stoppingToken);

                if (consumed)
                {
                    _channel.BasicAck(ea.DeliveryTag, multiple: false);
                    _logger.LogDebug("RabbitMQ message {Id} consumed and acked.", messageId);
                }
                else
                {
                    _channel.BasicAck(ea.DeliveryTag, multiple: false);
                    _logger.LogDebug("RabbitMQ message {Id} was duplicate — acked.", messageId);
                }
            }
            catch (Exception ex)
            {
                _logger.LogError(ex, "Error processing RabbitMQ message {Id}. Sending NACK.", messageId);
                _channel.BasicNack(ea.DeliveryTag, multiple: false, requeue: true);
            }
        };

        _channel.BasicConsume(queue: "order-events",
            autoAck: false,
            consumer: consumer);

        _logger.LogInformation("RabbitMQ inbox consumer '{Consumer}' started.", _consumerName);

        return Task.CompletedTask;
    }

    public override void Dispose()
    {
        _channel?.Close();
        _connection?.Close();
        base.Dispose();
    }
}

namespace Infrastructure.Persistence.DapperInbox;

    {
        _connectionString = connectionString;
    }

    /// <summary>
    /// Retry all failed inbox messages (those with errors).
    /// Resets the error and processing time so the consumer will reprocess.
    /// </summary>
    public async Task<int> RetryFailedMessagesAsync(
        string? consumerName = null,
        CancellationToken ct = default)
    {
        const string sql = @"
            UPDATE [Messaging].[InboxMessages]
            SET
                [Error] = NULL,
                [ProcessedAt] = SYSUTCDATETIME(),
                [ProcessingTimeMs] = NULL
            WHERE [Error] IS NOT NULL
              AND (@ConsumerName IS NULL OR [ConsumerName] = @ConsumerName);";

        using var db = new SqlConnection(_connectionString);

        var affected = await db.ExecuteAsync(new CommandDefinition(sql,
            new { ConsumerName = consumerName }, cancellationToken: ct));

        return affected;
    }

    /// <summary>
    /// Get all dead-letter messages for monitoring.
    /// </summary>
    public async Task<IReadOnlyList<DeadLetterItem>> GetDeadLettersAsync(
        int take = 100, CancellationToken ct = default)
    {
        const string sql = @"
            SELECT TOP (@Take)
                [MessageId],
                [MessageType],
                [ConsumerName],
                [ReceivedAt],
                [ProcessedAt],
                [Error]
            FROM [Messaging].[InboxMessages]
            WHERE [Error] IS NOT NULL
            ORDER BY [ProcessedAt] DESC;";

        using var db = new SqlConnection(_connectionString);

        var results = await db.QueryAsync<DeadLetterItem>(
            new CommandDefinition(sql, new { Take = take }, cancellationToken: ct));

        return results.AsList();
    }
}

## 7. Comparison — EF Core Inbox vs Dapper Inbox

| Aspect | EF Core Inbox | Dapper Inbox |
|--------|---------------|--------------|
| Deduplication mechanism | FindAsync + conditional insert with unique constraint exception handling | IF NOT EXISTS SQL with atomic insert |
| Transaction management | Implicit via SaveChangesAsync; explicit via BeginTransactionAsync | Explicit SqlTransaction in code |
| Race condition handling | Catches DbUpdateException with unique constraint violation | SQL-level IF NOT EXISTS is atomic; no exception needed |
| Business logic integration | Func delegate within same DbContext | Func delegate with raw SQL |
| Performance | Slower — change tracking overhead, entity materialization | Faster — minimal overhead, direct SQL |
| Readability | Higher — LINQ, strongly typed entities | Lower — raw SQL strings |
| Bulk operations | Requires AddRange + SaveChanges | SqlBulkCopy or TVP-based batch inserts |
| Best for | Teams standardized on EF Core, complex domain logic | High-throughput scenarios, maximum control over SQL |

## 8. Gotchas — Pitfalls and Mitigations

### 8.1 Inbox Cleanup

The inbox table grows monotonically. Each processed message adds a row. Without cleanup, the table grows indefinitely, degrading lookup performance and consuming storage.

**Mitigation**:
- Schedule a cleanup job (SQL Agent, Azure Scheduler, or hangfire) that deletes processed messages older than a retention period (e.g., 7-30 days).
- Use table partitioning with a sliding window — drop old partitions instead of deleting rows.
- Archive inbox records to a data warehouse before deletion for auditability.

### 8.2 Message Ordering Across Partitions

When consuming from a partitioned topic (e.g., Kafka with multiple partitions), messages from the same aggregate may arrive out of order.

**Mitigation**:
- Ensure the producer partitions by aggregate ID.
- Use a single consumer per partition or a single queue for ordered processing.
- Design idempotent handlers that can handle out-of-order delivery gracefully.

### 8.3 At-Least-Once Guarantees

The inbox pattern provides at-least-once processing semantics.

**Mitigation**:
- Always use a single transaction for inbox insert + business logic.
- Never insert the inbox record in a separate transaction from the business update.
- Use try/commit/rollback carefully.

### 8.4 Large Content in Inbox

Storing full message payloads in the inbox table is useful but makes the table large.

**Mitigation**:
- Store only essential fields (MessageId, MessageType) and keep the full payload elsewhere.
- Use compressed storage or columnstore indexes.
- Archive message payloads to blob storage.

### 8.5 Monitoring and Alerting

Failed messages in the inbox table are invisible unless actively monitored.

**Mitigation**:
- Create a monitoring view (vw_InboxStatistics) that tracks error counts and processing times.
- Set up alerts when Error IS NOT NULL rows exceed a threshold.
- Expose inbox metrics via Application Insights or Prometheus.

### 8.6 Dual Inbox/Outbox Usage

Teams sometimes confuse the inbox and outbox patterns or use both for the same message.

**Mitigation**:
- Use the outbox on the publishing side.
- Use the inbox on the consuming side.
- Never use both for the same message in the same service.

### 8.7 Testing Idempotency

Idempotent processing is hard to test because duplicates are rare in development.

**Mitigation**:
- Write unit tests that call the consumer twice with the same message.
- Verify the business logic executes only once.
- Write integration tests with a real broker that simulate redelivery.

### 8.8 Key Selection for Deduplication

Choosing the wrong deduplication key can cause missed duplicates or false positives.

**Mitigation**:
- Use the message ID assigned by the broker or producer.
- For idempotent commands, use the idempotency key from the request header.
- Do NOT use business data as the deduplication key.
- For at-most-once scenarios, use the offset/sequence number from the broker.
