### Where This Fits

You've just typed `git log --oneline --graph --decorate --all` for the fortieth time this week, or you've fat-fingered `git commit -m` into `git commit -m` with a typo and want a one-keystroke amend. You're setting up a new laptop and want your muscle-memory shortcuts to come with you. This is the moment you stop tolerating repetitive typing and start encoding your own Git vocabulary directly into `.gitconfig`.

## What You're Doing and Why

A Git alias is a named shortcut — stored in `.gitconfig` under an `[alias]` section — that expands to a longer Git command (or a shell command, via the `!` prefix) when you type it; you're doing this to collapse multi-flag commands you run dozens of times a day into something you can type without thinking, and because the shortcut travels with your config file across every machine and every repo you clone.

### Mental Model

Think of `.gitconfig` aliases as a personal phrasebook layered on top of Git's vocabulary — Git doesn't get new functionality, you just get to rename `git log --oneline --graph --decorate --all` to `git lg`. Two flavors exist: a **plain alias** (no `!`) is purely a Git-command substitution and can only call other Git subcommands; a **shell alias** (prefixed with `!`) hands the whole expansion to your shell, so it can pipe, chain with `&&`, reference `$1`/`$@` for arguments, or call any external program — `jq`, `grep`, `xargs`, whatever's on your `PATH`.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git config --global alias.<name> '<git-subcommand>'`|Creates a plain Git alias|Wrapping a Git subcommand + flags only|
|`git config --global alias.<name> '!<shell-command>'`|Creates a shell alias|Need piping, chaining, args, or non-Git tools|
|`git <alias-name>`|Runs the alias|Daily use, once defined|
|`git config --get-regexp alias`|Lists every alias currently defined|Auditing what's already set up on a machine|
|`git config --global --unset alias.<name>`|Removes a single alias|Retiring a shortcut you no longer use|
|`git <alias> -- --help`|Shows underlying command's help (plain aliases only)|Forgot what flags the wrapped command takes|

## Step-by-Step Walkthrough

### The Scenario

You're three weeks into working on `OrderService`, and you've noticed you type the same handful of commands constantly: a decorated one-line log to see branch topology, a quick status check, and an "oops" amend when you forget a file. You decide to formalize these into `.gitconfig` so they're available the instant you sit down at any machine with this config synced.

### Full Walkthrough

```bash
$ cd ~/repos/OrderService
$ git config --global alias.lg "log --oneline --graph --decorate --all"
# Plain alias: pure Git subcommand + flags, no shell needed

$ git lg
* 4a3f9c2 (HEAD -> feature/order-refunds, origin/feature/order-refunds) fix: prevent double refund
* 9d81b3a (origin/main, main) chore: bump EF Core to 9.0.3
* 71ecf40 feat: add idempotency key to RefundHandler
| * c2f0a91 (origin/feature/payment-retry) feat: add retry policy for PaymentGateway calls
|/
* 5b6a2d1 initial scaffolding for OrderService

$ git config --global alias.st "status -sb"
# Short-form status: branch line plus compact file states, not the verbose default

$ git st
## feature/order-refunds...origin/feature/order-refunds
 M src/OrderService/RefundHandler.cs

$ git config --global alias.amend "commit --amend --no-edit"
# Shell-free plain alias — reuses the last commit message, just restages

$ git add src/OrderService/RefundHandler.cs
$ git amend
[feature/order-refunds 7b2e114] fix: prevent double refund on concurrent requests
 Date: Fri Jun 26 14:02:11 2026 -0400
 1 file changed, 14 insertions(+), 3 deletions(-)

$ git config --get-regexp alias
alias.lg log --oneline --graph --decorate --all
alias.st status -sb
alias.amend commit --amend --no-edit
```

Each alias gets defined once via `git config --global`, which writes directly into `~/.gitconfig` — there's no separate "alias file," it's the same config file you already worked with in [[9.019 — Git Config — Global, Local, and System Scope]].

### What Could Go Wrong Mid-Walkthrough

If you try to alias something that needs a pipe or an external tool, the plain-alias syntax silently fails to do what you want — it just passes the pipe character as a literal argument to Git, which doesn't understand it:

```bash
$ git config --global alias.last "log -1 | cat"
$ git last
fatal: ambiguous argument '|': unknown revision or path not in the working tree.
```

The fix is the `!` shell prefix, covered in the next section.

## Production Scenario

### Realistic Context

The `OrderService` team has standardized on a small set of team-wide aliases that every new hire installs on day one, because three of the most common on-call diagnostic steps — "what's deployed where," "what changed since the last tag," and "clean up my local stale branches" — were each a five-flag command nobody could remember under pressure. You're writing the onboarding script that installs these.

### Full Implementation / Resolution

```bash
#!/usr/bin/env bash
# scripts/setup-git-aliases.sh
# Installs the OrderService team's standard Git aliases into the
# current user's global .gitconfig. Idempotent — safe to re-run.

set -euo pipefail

declare -A ALIASES=(
  [lg]="log --oneline --graph --decorate --all"
  [st]="status -sb"
  [amend]="commit --amend --no-edit"
  [unstage]="restore --staged ."
  [branches]="!git for-each-ref --sort=-committerdate refs/heads/ --format='%(refname:short)|%(committerdate:relative)|%(authorname)' | column -t -s '|'"
  [sincetag]="!git log $(git describe --tags --abbrev=0)..HEAD --oneline"
  [cleanup]="!git fetch --prune && git branch --merged main | grep -v '\\* \\|main' | xargs -r git branch -d"
  [whoami]="!git config user.name && git config user.email"
)

for name in "${!ALIASES[@]}"; do
  git config --global "alias.$name" "${ALIASES[$name]}"
  echo "Installed: git $name"
done
```

```bash
$ chmod +x scripts/setup-git-aliases.sh

Installed: git sincetag

Installed: git whoami
```

### Verifying It Worked

Run `git config --get-regexp alias` and confirm all eight names appear with the exact expansion strings shown above; then run each alias once (`git lg`, `git st`, `git branches`, etc.) and confirm it produces output rather than an error. For the shell-prefixed ones specifically, a clean exit with no `fatal:` or `command not found` line confirms the underlying shell syntax was accepted.

## Gotchas and Troubleshooting

### "fatal: ambiguous argument '|': unknown revision or path not in the working tree."

**Root cause:** You defined an alias as a plain Git alias (no `!` prefix) but the expansion contains a pipe, `&&`, or reference to a shell variable. Git tries to parse the entire string as arguments to itself rather than handing it to a shell.

```bash
git config --global alias.last "log -1 | cat"   # missing !
```

**Fix:**

```bash
git config --global --unset alias.last
git config --global alias.last "!git log -1 | cat"
```

**Prevention:** Any alias expansion containing `|`, `&&`, `;`, `$()`, or a call to a non-Git binary needs the `!` prefix — make this a reflexive check before saving.

### "git: 'lg' is not a git command. See 'git --help'."

**Root cause:** The alias was set in a different config scope than the one Git is reading — most commonly, it was set with `--local` in one repo and you're now in a different repo, or it was set without `--global` and defaulted to local in a repo with no `.git` directory at the working path.

```bash
$ cd ~/repos/OtherService
$ git lg
git: 'lg' is not a git command. See 'git --help'.
```

**Fix:**

```bash
git config --global alias.lg "log --oneline --graph --decorate --all"
```

**Prevention:** Default to `--global` for personal productivity aliases; reserve `--local` only for repo-specific aliases the whole team agrees to via a committed `.gitconfig` snippet (see the `includeIf` pattern below).

### "fatal: bad config line N in file /home/spider/.gitconfig"

**Root cause:** An unescaped quote or stray character inside an alias string corrupted the INI syntax of `.gitconfig`, usually from editing the file by hand instead of going through `git config`.

```bash
$ cat ~/.gitconfig
[alias]
    lg = log --oneline --graph --decorate --all
    bad = log "broken
```

**Fix:**

```bash
git config --global --edit
# delete or correct the malformed line, save, exit
```

**Prevention:** Always set aliases through `git config --global alias.<name> "<value>"` rather than hand-editing `.gitconfig` directly — the CLI handles quoting correctly.

### Alias silently does nothing different from running the raw command

**Root cause:** You ran `git config alias.<name> ...` without `--global` or `--local`/`--system`, inside a directory that isn't a Git repo, so Git wrote to a config file path that nothing reads.

```bash
$ cd ~/Downloads
$ git config alias.st "status -sb"
$ git st
fatal: not a git repository (or any of the parent directories): .git
```

**Fix:**

```bash
git config --global alias.st "status -sb"
```

**Prevention:** Get in the habit of always typing `--global` explicitly for personal aliases rather than relying on Git's ambient scope resolution.

### Shell alias works in your terminal but fails for a teammate who runs the same `.gitconfig`

**Root cause:** The shell alias calls an external tool (`column`, `jq`, `fzf`) that exists on your `PATH` but isn't installed on theirs.

```bash
$ git branches
zsh:1: command not found: column
```

**Fix:** Document the dependency, or rewrite the alias to avoid the external tool:

```bash
git config --global alias.branches "!git for-each-ref --sort=-committerdate refs/heads/ --format='%(refname:short) — %(committerdate:relative)'"
```

**Prevention:** When sharing team aliases via a script (as in the Production Scenario above), stick to tools guaranteed present in your team's baseline dev environment, or check for the tool's existence in the setup script before installing the alias that depends on it.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# A practical daily-driver set beyond the ones already shown
git config --global alias.co "checkout"
git config --global alias.br "branch"
git config --global alias.ci "commit"
git config --global alias.df "diff"
git config --global alias.last "log -1 HEAD --stat"
git config --global alias.undo "reset --soft HEAD~1"
git config --global alias.aliases "config --get-regexp alias"
```

### Automation Opportunity

Keeping aliases in a versioned dotfiles repo means every new machine gets the same shortcuts in one command, instead of re-typing `git config` invocations from memory.

```bash
#!/usr/bin/env bash
# scripts/sync-gitconfig.sh — symlinks a tracked .gitconfig.aliases file
# into the include path of the user's main .gitconfig, so the alias
# set is version-controlled and shared across machines via dotfiles repo.

DOTFILES_DIR="$HOME/dotfiles"
TARGET="$HOME/.gitconfig.aliases"

ln -sf "$DOTFILES_DIR/git/.gitconfig.aliases" "$TARGET"

if ! git config --global --get-all include.path | grep -q "gitconfig.aliases"; then
  git config --global --add include.path "$TARGET"
  echo "Linked and included $TARGET"
else
  echo "Already included"
fi
```

### Keyboard / CLI Efficiency Notes

- Use `git config --global --edit` to open `.gitconfig` in your configured `core.editor` for bulk alias edits instead of issuing one `git config` call per alias.
- Tab-completion for custom aliases doesn't come for free in most shells — if you live in Bash/Zsh, pull in `git-completion.bash` / `git-completion.zsh` and your aliases will tab-complete alongside built-in subcommands once they're under `[alias]`.
- For aliases you use dozens of times a day (`st`, `co`, `ci`), keep them to two characters — the savings compound across hundreds of invocations a week.

### Spoken Answers

**Q: Walk me through setting up a Git alias that greps commit messages for a ticket number and shows the matching commits with full diffs.**

> **Average answer:** "You'd do `git config --global alias.findticket 'log --grep'` and then pass the ticket number." — gets the general shape right but doesn't notice this needs the `!` shell prefix to accept a positional argument cleanly, and doesn't mention `--patch` for full diffs.

> **Great answer:** "Since I need to inject an argument into the middle of the command, I'd make it a shell alias: `git config --global alias.findticket '!f() { git log --grep="$1" --patch; }; f'` — wrapping it in a shell function lets `$1` bind to whatever I pass after the alias name, like `git findticket ORD-4521`. I'd test it against a known ticket commit first to confirm the grep pattern matches commit message conventions on this repo before trusting it during an actual search."

**Q: You add an alias and it fails with `'<name>' is not a git command` — what do you check, in order?**

> **Average answer:** "I'd just re-run the `git config` command." — doesn't diagnose why it failed in the first place.

> **Great answer:** "First I check scope — am I in the same directory/repo where I originally set it, since a `--local` alias only exists inside that one repo's `.git/config`. I run `git config --get-regexp alias` from my current location to see what's actually visible from here. If it's missing entirely, I check whether the original `git config` call had a typo in `alias.<name>` or was missing `--global`. If it shows up but still errors, the expansion itself is malformed — usually a missing `!` for a shell-dependent alias."

**Q: Tell me about a time a team-wide Git alias prevented a mistake or saved meaningful time during an incident.**

> **Average answer:** "We had some aliases that made things faster, like for status and log." — vague, no specific incident.

> **Great answer:** "During an OrderService incident we needed to know exactly what had shipped to production since the last release tag, fast, to scope the blast radius of a bad deploy. We had a `git sincetag` alias already wired up — `log $(git describe --tags --abbrev=0)..HEAD --oneline` — so instead of someone fumbling to remember the tag syntax under pressure, we had the commit list in two seconds and could immediately tell the incident channel which three PRs were in scope."

### War Story Prompt

A teammate's shell alias for cleaning up merged branches used `git branch -D` instead of `-d` — force-delete instead of safe-delete — and on a slow Friday it ran against a branch that had unpushed work nobody had merged yet, because the merge-detection grep pattern had a typo that matched too broadly. The fix was rewriting the alias to use `-d` (which refuses to delete unmerged branches) and adding a `--dry-run` echo step before the actual delete, plus a comment in the team dotfiles repo explaining exactly why `-D` is banned in that alias.

## Decision Framework

### When to Reach for This

- [ ] You type the same multi-flag Git command more than a few times a week
- [ ] You're standardizing a small set of diagnostic or workflow commands across a team
- [ ] You want a shortcut that survives across every repo and every machine your `.gitconfig` syncs to

### When NOT To

- [ ] The command is genuinely one-off — aliasing it adds maintenance overhead with no payoff
- [ ] The "shortcut" is complex enough to need real argument parsing, branching logic, or error handling — that belongs in a proper script (see [[9.436 — Writing a Bash Deployment Script]]), not a one-line `.gitconfig` entry
- [ ] You need the shortcut to be discoverable/self-documenting for a large team with varying shell familiarity — a documented Makefile target or `dotnet`/`just` task may communicate intent better than a personal alias buried in a dotfiles repo

### Tool Comparison

||Git Alias|Shell Function/Script|
|---|---|---|
|Speed|Instant to define, instant to run|Slightly more setup (file, executable bit, PATH)|
|Safety/reversibility|Easy to `--unset`; low blast radius if scoped to Git commands|Can do more damage if it touches files/processes beyond Git|
|When it's the right call|Wrapping Git subcommands and flags you run constantly|Multi-step logic, argument validation, calling multiple unrelated tools|

## Self-Check

### Conceptual Questions

1. What's the difference between how a plain alias and a `!`-prefixed shell alias are executed?
2. Predict the output of `git config --get-regexp alias` after running `git config --global alias.st "status -sb"` and nothing else.
3. What flag do you add to `git config` to make an alias visible only inside the current repo instead of globally?
4. You see `fatal: bad config line N in file ~/.gitconfig` — what's the most likely root cause and the safest way to fix it?
5. What's the safer alternative to writing `git branch -D` inside a "cleanup merged branches" alias?
6. What should you avoid doing when an alias needs to reference an argument like a ticket number or branch name?
7. How does this topic connect to [[9.021 — Git Hooks — pre-commit and pre-push Basics]]?

<details> <summary>Answers</summary>

1. A plain alias is expanded and handed straight to Git's own argument parser — it can only invoke Git subcommands and their flags. A `!`-prefixed alias hands the entire expansion to the system shell, so it can use pipes, `&&`, external binaries, and shell variables.
2. `alias.st status -sb`
3. `--local` (or simply omit `--global`/`--system` while inside the target repo, since `--local` is Git's implicit default).
4. Most likely cause: a malformed line from hand-editing `.gitconfig` directly (unescaped quotes, broken bracket syntax). Fix with `git config --global --edit` to open it in your editor and correct or remove the bad line, rather than continuing to hand-edit blind.
5. Use `-d` (lowercase) instead of `-D` — it refuses to delete a branch that hasn't been merged, preventing accidental loss of unpushed work.
6. Avoid defining it as a plain alias and trying to splice in positional shell-style arguments directly — plain aliases just append whatever you type after the alias name as trailing arguments to the wrapped Git command, which doesn't let you inject the argument mid-command. For mid-command injection, use a `!`-prefixed shell function alias.
7. Both live in personal/team workflow customization stored in or near `.gitconfig` — aliases shortcut commands you run yourself, while hooks (`pre-commit`, `pre-push`) automatically run scripts at specific Git lifecycle points; together they form the two main levers for shaping your local Git ergonomics beyond Git's defaults.

**Drill 2 — Fix the broken thing**

```bash
$ git config --global alias.recent "log --since='2 weeks ago' | grep Spider"
$ git recent
fatal: ambiguous argument '|': unknown revision or path not in the working tree. Use '--' to separate paths from revisions, like this:
'git <command> [<revision>...] -- [<file>...]'
```

<details> <summary>Solution</summary>

```bash
$ git config --global --unset alias.recent
$ git config --global alias.recent "!git log --since='2 weeks ago' | grep Spider"
$ git recent
4a3f9c2 fix: prevent double refund on concurrent requests (Spider)
71ecf40 feat: add idempotency key to RefundHandler (Spider)
```

**Drill 3 — Under time pressure**

An incident is live on `OrderService`. You need to know, in the next 30 seconds, exactly which commits have shipped to `main` since the last production tag, without remembering the exact `git describe` syntax from memory. Walk through exactly what you type, assuming no alias exists yet for this.

<details> <summary>Solution</summary>

```bash
$ git describe --tags --abbrev=0
# Confirms the latest tag name first — don't guess it, read it.
v2.4.1

$ git log v2.4.1..main --oneline
# Lists every commit reachable from main but not from the tag —
# exactly the shipped-but-not-yet-tagged-as-reviewed change set.
4a3f9c2 fix: prevent double refund on concurrent requests
9d81b3a chore: bump EF Core to 9.0.3
71ecf40 feat: add idempotency key to RefundHandler

# Once the incident is over, THIS is the moment to alias it for next time:
$ git config --global alias.sincetag '!git log $(git describe --tags --abbrev=0)..HEAD --oneline'
```

Each command is safe under pressure because neither mutates any state — both are pure read operations, so there's zero risk of making the incident worse while gathering information.

</details>
