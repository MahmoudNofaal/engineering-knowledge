### Where This Fits

You've been working in `OrderService` for an hour. Before you commit anything, or before you open a PR, you need to know exactly what changed: which files, which lines, staged vs. unstaged. This is the read-before-you-write step ‚Äî the moment you run `git status` to orient yourself and `git diff` to see the actual delta before it becomes permanent.

---

## What You're Doing and Why

`git status` tells you where each modified file sits across Git's three zones (working tree, index, neither). `git diff` shows the exact line-level changes in those files. Together they answer the two questions you always need before committing: _what did I change?_ and _what am I about to commit?_ The two commands serve different purposes ‚Äî `status` gives you the map, `diff` gives you the territory.

### Mental Model

Think of the working tree as your desk, the index (staging area) as your outbox tray, and the repository as the filing cabinet. `git status` tells you which documents are on the desk, which are in the outbox, and which are already filed. `git diff` lets you read the red-pen edits on each document before it moves anywhere.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git status`|Shows all modified, staged, and untracked files|Before every commit; after every pull or merge|
|`git status -s`|Same output, compact one-line-per-file format|When you want a quick scan without the prose|
|`git diff`|Line-level diff of unstaged changes (working tree vs index)|To review edits you haven't staged yet|
|`git diff --staged`|Line-level diff of staged changes (index vs HEAD)|To verify what will actually be in the next commit|
|`git diff HEAD`|All changes since last commit, staged + unstaged combined|When you want the total delta without caring about staging state|
|`git diff <branch>`|Changes between current branch and another branch|Before merging; in PR prep|
|`git diff --stat`|File-level summary: insertions and deletions counts|Quick size-of-change overview without full patch|
|`git diff --name-only`|Just the filenames of changed files|Feeding file lists to scripts or `xargs`|
|`git diff <commit>..<commit>`|Changes between two specific commits|Investigating a regression between known-good and known-bad|

`git diff HEAD` shows everything ‚Äî staged and unstaged ‚Äî as one combined delta against the last commit. Use this when you want the full picture without caring about staging state.

```bash
$ git status -s

 M src/OrderService/OrderValidator.cs
?? docs/refund-edge-cases.md
```

The short format: the first column is the index state, the second is the working-tree state. `M` (first column) = staged modification. `M` (second column) = unstaged modification. `??` = untracked. Once you internalize this, `-s` replaces the verbose output for quick checks.

### What Could Go Wrong Mid-Walkthrough

**If `git diff` produces no output even though `git status` shows changes:**

The file is already fully staged. `git diff` (no args) only shows unstaged changes. Switch to `git diff --staged` to see it.

```bash
$ git diff              # empty ‚Äî file is staged
$ git diff --staged     # shows the change
```

**If `git diff` opens in a pager and you can't exit:**

You're in `less`. Press `q` to quit. To disable paging permanently for diffs: `git config --global core.pager cat`. To disable it just for one command: `git --no-pager diff`.

## Gotchas and Troubleshooting

### "`git diff` shows nothing, but I definitely changed something"

**Root cause:** The file is fully staged. `git diff` (no arguments) compares working tree to index ‚Äî if a file is staged, there's nothing left to compare.

```bash
$ git status
Changes to be committed:
        modified:   src/OrderService/RefundHandler.cs
$ git diff
# (empty ‚Äî RefundHandler.cs is staged, not in working tree diff)
```

**Fix:**

```bash
$ git diff --staged    # see what's staged
$ git diff HEAD        # see everything, staged + unstaged
```

**Prevention:** Make `git diff --staged` a pre-commit habit. Alias it to `git ds` (see Section 6).

---

### "`git diff` opens an endless stream of whitespace-only changes"

**Root cause:** A team member or tool (Prettier, `.editorconfig` change, `dos2unix`) reformatted files without changing logic. The diff is technically correct but visually useless.

```bash
$ git diff
-    public async Task<ChargeResult> ProcessAsync(ChargeRequest request, CancellationToken ct)
+   public async Task<ChargeResult> ProcessAsync(ChargeRequest request, CancellationToken ct)
# ... hundreds of lines of indent changes
```

**Fix:**

```bash
$ git diff -w                     # ignore all whitespace changes
$ git diff --ignore-blank-lines   # ignore blank-line-only hunks
$ git diff --word-diff            # inline word-level diff ‚Äî reveals real changes inside whitespace noise
```

**Prevention:** Commit formatting changes separately, in their own commit with message `style: apply dotnet format`. This keeps logic diffs readable.

---

### "`warning: LF will be replaced by CRLF`" floods `git status` output

**Root cause:** `core.autocrlf` is set to `true` on Windows, converting line endings on every file every time it's touched, making every file appear modified.

```bash
$ git status
modified:   src/OrderService/RefundHandler.cs
modified:   src/OrderService/OrderValidator.cs
modified:   src/PaymentGateway/ChargeProcessor.cs
# ... (entire repo appears modified)
```

**Fix:**

```bash
$ git config --global core.autocrlf input   # on Mac/Linux
$ git config --global core.autocrlf true    # on Windows (if team uses mixed OS)
$ git rm --cached -r .                      # clear index
$ git reset --hard                          # re-checkout with correct line endings
```

**Prevention:** Add a `.gitattributes` file to the repo root enforcing line endings per file type ‚Äî see [[9.018 ‚Äî .gitattributes]]. This removes the dependency on per-machine config.

---

### "`git diff <branch>` shows far more than expected ‚Äî includes other people's commits"

**Root cause:** Two-dot `git diff branch1..branch2` shows all changes between the tips, including commits that diverged on both sides. You wanted only what _your_ branch added.

```bash
$ git diff main..feature/order-refunds   # includes main commits your branch doesn't have
```

**Fix:**

```bash
$ git diff main...feature/order-refunds  # three dots: diff from common ancestor
```

The three-dot form is almost always what you want when reviewing a feature branch ‚Äî it shows only what the branch contributed, not the divergence noise.

**Prevention:** Default to `...` for branch comparisons. Use `..` only when you explicitly need the tip-to-tip delta.

---

### "Untracked files I don't want keep appearing in `git status`"

**Root cause:** Build artifacts, `.DS_Store`, `bin/`, `obj/`, or IDE metadata folders are not in `.gitignore`.

```bash
$ git status
Untracked files:
        bin/Debug/net8.0/OrderService.dll
        obj/Debug/net8.0/
        .DS_Store
        .idea/
```

**Fix:**

```bash
$ cat >> .gitignore << 'EOF'
bin/
obj/
.DS_Store
.idea/
*.user
EOF
$ git status   # those paths are now silent
```

**Prevention:** Start every .NET repo from the GitHub-provided `.gitignore` template for Visual Studio. See [[9.010 ‚Äî .gitignore ‚Äî Patterns and Common .NET Templates]].

---

### "`git diff` shows a binary file changed but no content"

**Root cause:** Git can't diff binary files (images, compiled assets, PDFs) as text, so it only reports that they changed.

```bash
$ git diff
diff --git a/assets/logo.png b/assets/logo.png
index 3a8f1c2..9b2de01 100644
Binary files a/assets/logo.png and b/assets/logo.png differ
```

**Fix ‚Äî verify the change is intentional:**

```bash
$ git show HEAD:assets/logo.png > /tmp/logo-before.png
$ diff /tmp/logo-before.png assets/logo.png   # binary-level compare
```

**Fix ‚Äî if the binary shouldn't be tracked:**

```bash
$ echo "assets/*.png" >> .gitignore
$ git rm --cached assets/logo.png
```

**Prevention:** Track large binaries with Git LFS (see [[9.023 ‚Äî Git LFS]]). Avoid committing generated binary assets to the main repo.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# ~/.gitconfig ‚Äî add under [alias]
[alias]
  s     = status -s                        # compact status ‚Äî the one you'll use 50x a day
  d     = diff                             # unstaged diff
  ds    = diff --staged                    # staged diff ‚Äî run this before every commit
  dh    = diff HEAD                        # full delta since last commit
  dstat = diff --stat                      # file summary without full patch
  dw    = diff --word-diff                 # word-level diff ‚Äî great for prose/config files
  dn    = diff --name-only                 # just filenames ‚Äî useful in scripts
```

Add to shell (`~/.zshrc` or `~/.bashrc`):

```bash
# Show compact status inline anywhere
alias gs='git status -s'
alias gd='git diff'
alias gds='git diff --staged'
```

### Automation Opportunity

Pre-commit hook that shows a `--stat` summary before you type the commit message, so you can abort if something unexpected appears:

```bash
#!/usr/bin/env bash
# .git/hooks/prepare-commit-msg
# Shows staged diff stat before the commit message editor opens
echo ""
echo "=== Staged changes ==="
git diff --staged --stat
echo "======================"
echo ""
```

Install it:

```bash
$ cp scripts/prepare-commit-msg .git/hooks/prepare-commit-msg
$ chmod +x .git/hooks/prepare-commit-msg
```

### Keyboard / CLI Efficiency Notes

- **In `less` (the default pager):** `j`/`k` scroll line by line, `Space`/`b` scroll by page, `/` searches forward, `q` quits. Learning these five means never reaching for the mouse inside a diff.
- **`git diff --color-words`** ‚Äî highlights only the changed words within a line instead of the full line. Invaluable for config files and long strings where a single word changed.
- **`git diff --compact-summary`** ‚Äî shows a per-file summary inline with the diff header. Less verbose than `--stat`, more informative than `--name-only`.
- **Pipe to `grep`** to scan for a specific method name across a large diff: `git diff HEAD | grep -A5 "ProcessRefundAsync"`.

### Spoken Answers

**Q: What's the difference between `git diff`, `git diff --staged`, and `git diff HEAD`?**

> **Average answer:** "`git diff` shows what's changed. `--staged` shows what's staged. `HEAD` shows everything since the last commit."

> **Great answer:** "They're answering three different questions. `git diff` with no arguments compares the working tree to the index ‚Äî so it only shows me edits I _haven't_ staged yet. `git diff --staged` compares the index to HEAD ‚Äî it shows me exactly what the next commit will contain, which is the one I run before every single commit because it's the last safety check before code becomes history. `git diff HEAD` is the union of both ‚Äî working tree vs. HEAD, staged and unstaged together ‚Äî useful when I want the total delta since my last commit without caring about the staging boundary. A concrete example: if I've staged `RefundHandler.cs` and modified but not staged `OrderValidator.cs`, then `git diff` shows only `OrderValidator.cs`, `git diff --staged` shows only `RefundHandler.cs`, and `git diff HEAD` shows both."

---

**Q: You run `git diff` and see no output, but `git status` shows a file as modified ‚Äî what's happening?**

> **Average answer:** "Maybe there's a caching issue or the file wasn't saved."

> **Great answer:** "Almost certainly the file is fully staged. `git diff` with no arguments compares the working tree to the index. If I've done `git add src/OrderService/RefundHandler.cs`, it's moved into the index ‚Äî there's no longer a working-tree delta for it, so `git diff` comes up empty. The fix is `git diff --staged`, which shows the index-to-HEAD delta ‚Äî that's where staged files live. One way to catch this quickly is to look at `git status` output first: if the file appears under 'Changes to be committed' rather than 'Changes not staged for commit,' that's your answer."

---

**Q: Tell me about a time a diff review before committing caught a mistake you were about to ship.**

> **Average answer:** "Yeah, I've caught things in diff review before. It's a good habit."

> **Great answer:** "We were shipping a config change to the `NotificationWorker` service. I ran `git diff --staged` before committing ‚Äî standard habit ‚Äî and saw that I'd accidentally staged a change to `appsettings.Production.json` that had a connection string I'd temporarily modified locally for debugging. It was completely wrong for production, and it had a real password in it that would have hit our Git history permanently. The fix was `git restore --staged appsettings.Production.json`, then verifying the remaining staged files were clean before committing. That one `git diff --staged` probably saved us a secrets rotation incident and a history rewrite."

### War Story Prompt

During a late-night release, an engineer on the `InventoryApi` team committed what they thought was a one-line config change. Because they hadn't run `git diff --staged`, they didn't see that their IDE had also auto-formatted the entire `InventoryService.cs` file ‚Äî 800 lines of whitespace changes wrapped around one real change. The PR reviewer approved it as "just a config tweak." Three days later, a merge conflict in that file took two hours to resolve because the history was polluted with noise. Adding `git diff --staged` and `git diff --stat` to the pre-commit checklist eliminated this class of incident entirely.

## Decision Framework

### When to Reach for This

- [ ] You're about to run `git add` or `git commit` ‚Äî always check `git status` first
- [ ] You're reviewing a colleague's branch before merging ‚Äî use `git diff main...their-branch --stat` first, then full diff per file
- [ ] Your PR description needs to summarize what changed ‚Äî `git diff --stat main...HEAD` gives you the file list
- [ ] Something is behaving unexpectedly and you need to confirm your working tree is clean ‚Äî `git status -s` is the fastest check
- [ ] You're debugging a regression and need to know what changed between two commits ‚Äî `git diff <good-sha>..<bad-sha>`

### When NOT To

- [ ] Don't use `git diff` to review the full history of a feature ‚Äî use `git log -p` or the GitHub PR diff instead; the context is clearer
- [ ] Don't use `git diff HEAD` when you specifically need to validate staged-only changes before committing ‚Äî `--staged` is the right scope
- [ ] Don't rely on `git diff` alone for binary files ‚Äî it can only tell you they changed, not what changed; use dedicated tools for images and documents

### Tool Comparison

||`git diff` (no args)|`git diff --staged`|`git diff HEAD`|
|---|---|---|---|
|Scope|Unstaged changes only|Staged changes only|All changes since last commit|
|Use case|Reviewing edits in progress|Pre-commit verification|Full working tree audit|
|Shows staged files?|No|Yes|Yes|
|Shows unstaged files?|Yes|No|Yes|

## Self-Check

### Conceptual Questions

1. What exactly does `git diff` (no arguments) compare ‚Äî which two zones?
2. You have two files: `A.cs` is staged, `B.cs` is modified but not staged. Which appears in `git diff`? Which in `git diff --staged`?
3. What flag makes `git diff` show only the filenames of changed files, without the patch?
4. You see this in `git status -s`: `M src/OrderService/RefundHandler.cs`. What does the space in the first column mean?
5. What's the difference between `git diff branch1..branch2` and `git diff branch1...branch2`?
6. What should you NOT skip right before running `git commit` if you want to validate exactly what's going into the commit?
7. How does this connect to [[9.007 ‚Äî Resolving Merge Conflicts]]? What role does `git diff` play during conflict resolution?

<details> <summary>Answers</summary>

1. It compares the **working tree** (your edited files on disk) to the **index** (the staging area). It does NOT compare against HEAD.
    
2. Only **`B.cs`** appears in `git diff` (unstaged). Only **`A.cs`** appears in `git diff --staged`. `git diff HEAD` shows both.
    
3. `--name-only`: `git diff --name-only`
    
4. The space in the **first column** means no staged change. The `M` in the **second column** means a working-tree modification. So `RefundHandler.cs` has been modified but **not staged**. Compare to `M` (first-column M) which means staged and not further modified since staging.
    
5. Two-dot (`..`) compares the **tips** of both branches directly. Three-dot (`...`) compares from the **merge base** (common ancestor) to the tip of the second branch ‚Äî showing only what the second branch added. Three-dot is almost always what you want when reviewing a feature branch.
    
6. You should NOT skip running `git diff --staged`. Committing without checking `--staged` means you might commit files you didn't intend, or miss that you haven't staged the actual change you meant to commit.
    
7. During conflict resolution, `git diff` shows you the conflict markers and the competing hunks in conflicted files. After resolving and before staging, `git diff` lets you verify your resolution looks correct. After staging the resolution, `git diff --staged` confirms the final merged content matches your intent before you complete the merge commit. See [[9.007 ‚Äî Resolving Merge Conflicts ‚Äî Manual Workflow]].
    

</details>

### Hands-On Drills

**Drill 1 ‚Äî Do it from a clean state**

Starting from a fresh clone of any multi-file .NET repo, make a change to two files ‚Äî stage one, leave one unstaged. Then use three different `git diff` variants to show: (a) only the unstaged change, (b) only the staged change, (c) both changes combined. Also produce a compact status line that shows both files in a single-line-per-file format.

<details> <summary>Solution</summary>

```bash
$ git clone https://github.com/your-org/OrderService.git
$ cd OrderService

# Make two changes
$ echo "// unstaged edit" >> src/OrderService/OrderValidator.cs
$ echo "// staged edit" >> src/OrderService/RefundHandler.cs

# Stage only one
$ git add src/OrderService/RefundHandler.cs

# (a) Unstaged only
$ git diff
# Shows OrderValidator.cs only

# (b) Staged only
$ git diff --staged
# Shows RefundHandler.cs only

# (c) Both combined
$ git diff HEAD
# Shows both files
```
