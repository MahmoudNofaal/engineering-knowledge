### Where This Fits

Two situations force you to understand Git config scope: you sit down on a new machine and your first commit shows up as `Unknown User <unknown@unknown>` in the PR, or you're working across multiple GitHub identities — personal account for open source, work account for `InterVision` — and the wrong email keeps appearing on commits. Both come from the same root: Git resolves settings by reading three config files in a defined priority order, and if you don't know that order, debugging misconfigured commits is guesswork.

---

## What You're Doing and Why

`git config` reads and writes settings across three scopes: **system** (affects every user on the machine), **global** (affects every repo for the current OS user), and **local** (affects only the current repo). When Git needs a setting — your name, your email, your preferred merge strategy, your diff tool — it walks from local to global to system and uses the first value it finds. Local always wins. Understanding this cascade is how you set a work identity in `~/dev/intervision` repos without contaminating your personal open-source contributions.

The settings live in three plain-text files in INI format. You can `cat` them directly, but `git config` is the safe interface — it validates syntax and handles quoting, so use it instead of hand-editing except when debugging.

### Mental Model

Think of it as CSS specificity for settings. System scope is the browser default stylesheet — it applies everywhere unless overridden. Global scope is your personal stylesheet — it applies to all your sites unless a page overrides it. Local scope is the page's own inline style — most specific, always wins. Like CSS, you almost never touch the system file; you set your defaults in global and override per-repo in local when needed.

```
Resolution order (highest priority wins):
┌─────────────────────────────────────────────────┐
│  1. LOCAL   — .git/config           (repo only) │  ← always checked first
│  2. GLOBAL  — ~/.gitconfig          (this user) │
│  3. SYSTEM  — /etc/gitconfig        (all users) │  ← checked last
└─────────────────────────────────────────────────┘
```

One additional scope — **worktree** (`--worktree`) — exists for git worktree setups and sits between local and global in priority. It's rarely needed and not covered in this note.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git config --list --show-origin`|Show every active setting and which file it came from|Diagnosing "why is this value wrong"|
|`git config --global user.name "Name"`|Set your display name for all repos|New machine setup|
|`git config --global user.email "email"`|Set your commit email for all repos|New machine setup|
|`git config --local user.email "work@co.com"`|Override email for one repo only|Multi-identity workflow|
|`git config --global --edit`|Open the global config file in your editor|Batch-editing multiple settings|
|`git config --local --edit`|Open the local `.git/config` in your editor|Inspecting or batch-editing repo settings|
|`git config --global --unset <key>`|Remove a key from global config|Cleaning up a mistaken global setting|
|`git config --get <key>`|Print the resolved (winning) value of one key|Checking what Git will actually use|
|`git config --get-all <key>`|Print all values for a key across all scopes|Finding duplicate/conflicting values|
|`git config --global core.editor "code --wait"`|Set VS Code as the commit message editor|Replacing the default `vi`|
|`git config --global init.defaultBranch main`|Name new repos `main` instead of `master`|One-time global setup|
|`git config --global merge.ff false`|Always create merge commits, never fast-forward|Enforcing team merge discipline globally|
|`git config --global fetch.prune true`|Auto-prune deleted remote branches on every fetch|Keeping remote-tracking refs clean|

## Step-by-Step Walkthrough

### The Scenario

You've just been issued a new MacBook for your role at InterVision. You need to set up Git from a blank slate: establish your work identity globally, verify it's correct, configure quality-of-life defaults, and then override your email for a personal open-source repo that lives alongside your work repos — so commits to each go out under the right identity.

### Full Walkthrough

**Part A — Inspect the starting state on a new machine**

```bash
$ git config --list --show-origin
# (no output, or only system-level entries)

$ git config --get user.name
# (no output — key not set)

$ git config --get user.email
# (no output — key not set)
```

On a fresh install, nothing is set at global scope. If you commit now, Git may fall back to the OS username and hostname for the email, which produces `firstlast@MacBook-Pro.local` in your commits.

---

**Part B — Set global identity**

```bash
$ git config --global user.name "Alex Mercer"
$ git config --global user.email "alex.mercer@intervision.com"
```

These two settings are the minimum required before any commit.

```bash
$ git config --get user.name
Alex Mercer

$ git config --get user.email
alex.mercer@intervision.com
```

---

**Part C — Set quality-of-life global defaults**

```bash
# Use VS Code as the editor for commit messages, rebase instructions, etc.
$ git config --global core.editor "code --wait"

# Default branch name for new repos
$ git config --global init.defaultBranch main

# Always create merge commits — no accidental fast-forwards on team repos
$ git config --global merge.ff false

# Auto-prune deleted remote-tracking branches on every fetch
$ git config --global fetch.prune true

# Colour output in the terminal
$ git config --global color.ui auto

# Show a cleaner, more informative diff algorithm
$ git config --global diff.algorithm histogram

# Pull uses rebase by default instead of creating a merge commit
$ git config --global pull.rebase true

# Push only the current branch, not all tracking branches
$ git config --global push.default current
```

---

**Part D — Verify the full global config**

```bash
$ git config --global --list
user.name=Alex Mercer
user.email=alex.mercer@intervision.com
core.editor=code --wait
init.defaultbranch=main
merge.ff=false
fetch.prune=true
color.ui=auto
diff.algorithm=histogram
pull.rebase=true
push.default=current
```

Or open the file directly to see the INI structure:

```bash
$ git config --global --edit
```

This opens `~/.gitconfig` in VS Code (since that's your configured editor):

```ini
[user]
	name = Alex Mercer
	email = alex.mercer@intervision.com
[core]
	editor = code --wait
[init]
	defaultBranch = main
[merge]
	ff = false
[fetch]
	prune = true
[color]
	ui = auto
[diff]
	algorithm = histogram
[pull]
	rebase = true
[push]
	default = current
```

---

**Part E — Override identity for a personal open-source repo**

You have a personal project cloned at `~/dev/oss/dotnet-extensions`. Commits there should go out under your personal GitHub identity, not your InterVision work email.

```bash
$ cd ~/dev/oss/dotnet-extensions

$ git config --local user.name "amercer"
$ git config --local user.email "amercer@gmail.com"
```

Verify the resolution: local wins over global.

```bash
$ git config --get user.email
amercer@gmail.com        # local value — overrides global

$ git config --list --show-origin | grep user
file:/Users/amercer/.gitconfig          user.name=Alex Mercer
file:/Users/amercer/.gitconfig          user.email=alex.mercer@intervision.com
file:.git/config                        user.name=amercer
file:.git/config                        user.email=amercer@gmail.com
```

The `--show-origin` flag reveals exactly which file each value came from — and you can see both values exist, with the local file's value being the one Git will use.

---

**Part F — Confirm the correct identity before committing**

```bash
$ git commit --allow-empty -m "test: verify commit identity"
[main 7a3c91d] test: verify commit identity
 Author: amercer <amercer@gmail.com>

$ git log --format="%an <%ae>" -1
amercer <amercer@gmail.com>
```

The personal identity is on the commit. The InterVision email never touched this repo.

### What Could Go Wrong Mid-Walkthrough

**If `code --wait` doesn't open VS Code when Git tries to launch an editor:**

```bash
$ git commit
hint: Waiting for your editor to close the file...
# hangs or opens wrong editor
```

VS Code's `code` CLI may not be on your `$PATH`. Fix:

```bash
# In VS Code: Cmd+Shift+P → "Shell Command: Install 'code' command in PATH"
# Then verify:
$ which code
/usr/local/bin/code

# Re-set the editor to confirm the path works
$ git config --global core.editor "code --wait"
```

## Production Scenario

### Realistic Context

The InterVision platform team onboards three new engineers simultaneously. Each needs an identical Git baseline: work identity set correctly, sensible defaults active, and a per-repo override ready for the one engineer (`patel.dev`) who also maintains a personal open-source library and can't mix identities. The senior engineer writes a one-time setup script rather than walking each person through the steps manually. The script is idempotent — safe to re-run if something is wrong — and prints a summary at the end so each engineer can verify their own state.

### Full Implementation / Resolution

```bash
#!/usr/bin/env bash
# scripts/git-setup.sh
# Configures Git global settings for InterVision engineers.
# Usage: bash git-setup.sh "Full Name" "firstname.lastname@intervision.com"
# Idempotent — safe to re-run.

set -euo pipefail

NAME="${1:?First argument must be your full name, e.g. \"Alex Mercer\"}"
EMAIL="${2:?Second argument must be your work email, e.g. \"alex.mercer@intervision.com\"}"

echo "==> Setting Git identity..."
git config --global user.name  "$NAME"
git config --global user.email "$EMAIL"

echo "==> Setting editor and branch defaults..."
git config --global core.editor          "code --wait"
git config --global init.defaultBranch   main
git config --global color.ui             auto

echo "==> Setting fetch/push/merge behaviour..."
git config --global fetch.prune   true
git config --global pull.rebase   true
git config --global push.default  current
git config --global merge.ff      false

echo "==> Setting diff and log quality-of-life..."
git config --global diff.algorithm          histogram
git config --global log.abbrevCommit        true
git config --global log.date                iso

echo "==> Setting rebase safety options..."
git config --global rebase.autoStash        true
git config --global rebase.autoSquash       true

echo "==> Setting credential helper..."
if [[ "$OSTYPE" == "darwin"* ]]; then
  git config --global credential.helper osxkeychain
elif [[ "$OSTYPE" == "linux-gnu"* ]]; then
  git config --global credential.helper "store --file ~/.git-credentials"
fi

echo ""
echo "==> Verification — active global config:"
git config --global --list

echo ""
echo "==> Identity that will appear on commits:"
echo "    Name:  $(git config --get user.name)"
echo "    Email: $(git config --get user.email)"
echo ""
echo "Done. To override identity for a specific repo:"
echo "  cd <repo> && git config --local user.email <personal-email>"
```

Run it:

```bash

==> Setting editor and branch defaults...
==> Setting fetch/push/merge behaviour...
==> Setting diff and log quality-of-life...
==> Setting rebase safety options...
==> Setting credential helper...

==> Verification — active global config:
user.name=Alex Mercer
user.email=alex.mercer@intervision.com
core.editor=code --wait
init.defaultbranch=main
color.ui=auto
fetch.prune=true
pull.rebase=true
push.default=current
merge.ff=false
diff.algorithm=histogram
log.abbrevcommit=true
log.date=iso
rebase.autostash=true
rebase.autosquash=true
credential.helper=osxkeychain

==> Identity that will appear on commits:
    Name:  Alex Mercer
    Email: alex.mercer@intervision.com

Done. To override identity for a specific repo:
  cd <repo> && git config --local user.email <personal-email>
```

For `patel.dev` who maintains `~/dev/oss/async-helpers`:

```bash
$ cd ~/dev/oss/async-helpers
$ git config --local user.name  "patel-dev"
$ git config --local user.email "patel@gmail.com"

$ git config --list --show-origin | grep user
file:/Users/patel/.gitconfig    user.name=Dev Patel
file:/Users/patel/.gitconfig    user.email=dev.patel@intervision.com
file:.git/config                user.name=patel-dev
file:.git/config                user.email=patel@gmail.com
```

### Verifying It Worked

```bash
# Make a test commit in each repo and check the author field
$ git commit --allow-empty -m "chore: verify git identity"
$ git log --format="%an <%ae>" -1

```

Both identities show up correctly. Delete the test commits before pushing:

```bash
$ git reset --soft HEAD~1
```

## Gotchas and Troubleshooting

### Commits showing wrong author after config change

**Root cause:** `git config` changes only affect _future_ commits. Any commits already in the repo were written with the old identity at commit time and are immutable objects — changing the config does not retroactively update them.

```bash
$ git log --format="%an <%ae>" -3
Alex Mercer <alex.mercer@intervision.com>    # recent — correct
Unknown User <user@MacBook-Pro.local>        # before config was set — wrong
Unknown User <user@MacBook-Pro.local>        # before config was set — wrong
```

**Fix — rewrite history for local-only commits (before pushing):**

```bash
$ git rebase -i HEAD~2    # open the last 2 commits for interactive rebase
# Mark both as 'edit' in the editor

# For each commit, amend the author:
$ git commit --amend --author="Alex Mercer <alex.mercer@intervision.com>" --no-edit
$ git rebase --continue
```

For a larger range or after pushing (with team agreement), use `git filter-repo`:

```bash
$ pip install git-filter-repo --break-system-packages
$ git filter-repo --email-callback 'return b"alex.mercer@intervision.com"'
```

**Prevention:** Run `git config --get user.email` as the first step on any new machine or new clone before making a single commit. The `git-setup.sh` script above handles this automatically.

---

### "Please tell me who you are" error on first commit

**Root cause:** Neither `user.name` nor `user.email` is set at any scope. Git refuses to commit without both.

```bash
$ git commit -m "initial commit"
Author identity unknown

*** Please tell me who you are.

Run

  git config --global user.email "you@example.com"
  git config --global user.name "Your Name"

to set your account's default identity.
Omit --global to set the identity only in this repository.

fatal: empty ident name (for <>) not allowed
```

**Fix:**

```bash
$ git config --global user.name "Alex Mercer"
$ git config --global user.email "alex.mercer@intervision.com"
```

**Prevention:** The `git-setup.sh` script eliminates this entirely for new team members. Add it to the developer onboarding checklist.

---

### Local config not taking effect — global value keeps winning

**Root cause:** The local config was written to the wrong file — the most common cause is running `git config --local` outside a Git repo, which fails silently or writes to an unexpected location, or running it in the wrong directory.

```bash
$ cd ~/dev/oss/async-helpers
$ git config --local user.email "patel@gmail.com"

$ git config --get user.email
dev.patel@intervision.com    # still showing global value — something is wrong
```

**Diagnose:**

```bash
$ git config --list --show-origin | grep user.email
file:/Users/patel/.gitconfig        user.email=dev.patel@intervision.com
# No local entry — the local write didn't stick
```

**Fix — verify you are inside the repo and try again:**

```bash
$ pwd
/Users/patel/dev/oss/async-helpers
$ ls .git
config  description  HEAD  hooks  info  objects  refs

$ git config --local user.email "patel@gmail.com"
$ git config --get user.email
patel@gmail.com    # now correct
```

**Prevention:** After any `git config --local` command, immediately run `git config --get <key>` to confirm the resolved value. Never assume the write succeeded.

---

### `git config --global --unset` leaves duplicate or orphaned section headers

**Root cause:** After unsetting the last key in an INI section, `git config` removes the key but may leave the empty section header in `~/.gitconfig`. This is cosmetically harmless but can confuse manual inspection.

```bash
$ git config --global --unset alias.rbup
$ cat ~/.gitconfig
...
[alias]
	# (empty — only entry was rbup)
...
```

**Fix — open and clean the file directly:**

```bash
$ git config --global --edit
# Remove the empty [alias] section manually, save and close
```

**Prevention:** Use `--remove-section` when you want to delete an entire section at once:

```bash
$ git config --global --remove-section alias
```

---

### `git config --get` returns nothing but the setting appears in `--list`

**Root cause:** A key appears in `--list` when duplicated across scopes, but `--get` returns only the winning (highest-priority) value. If you expect a global value but a local value with the same key exists, `--get` returns the local one — which may look like "nothing" if you're checking the wrong scope.

```bash
$ git config --get user.email
patel@gmail.com     # local value — you expected the global one

$ git config --global --get user.email
dev.patel@intervision.com    # specify the scope to see that scope's value
```

**Fix:** Use `--show-origin` to see all values and which file owns each:

```bash
$ git config --list --show-origin | grep user.email
file:/Users/patel/.gitconfig        user.email=dev.patel@intervision.com
file:.git/config                    user.email=patel@gmail.com
```

**Prevention:** Default to `git config --list --show-origin` when diagnosing config issues — it shows the full picture in one pass.

---

### Wrong editor opens for commit messages despite `core.editor` being set

**Root cause:** An environment variable — `GIT_EDITOR`, `VISUAL`, or `EDITOR` — is set in your shell profile and overrides `core.editor`. Git checks environment variables before reading config files for editor selection.

```bash
$ git config --get core.editor
code --wait

$ git commit   # opens vim instead of VS Code
```

**Diagnose:**

```bash
$ echo $GIT_EDITOR
vim       # overriding core.editor
$ echo $VISUAL
          # not set
$ echo $EDITOR
vim       # fallback — also overriding
```

**Fix — unset or update the environment variable in your shell profile:**

```bash
# In ~/.zshrc or ~/.bashrc — remove or change:
export EDITOR="code --wait"    # or remove the line entirely if you only want git config to control it
# Then reload:
$ source ~/.zshrc
```

**Prevention:** Keep `EDITOR` and `GIT_EDITOR` consistent with `core.editor`, or leave environment variables unset and let `core.editor` be the single source of truth.

---

### includeIf — per-directory identity not resolving

**Root cause:** The `includeIf` directive in `~/.gitconfig` (used to automatically apply a different config file for all repos under a given directory path) silently fails when the directory path doesn't end with `/`.

```ini
# ~/.gitconfig — broken
[includeIf "gitdir:~/dev/oss"]
    path = ~/.gitconfig-oss

# ~/.gitconfig — correct
[includeIf "gitdir:~/dev/oss/"]
    path = ~/.gitconfig-oss
```

```bash
$ cd ~/dev/oss/async-helpers
$ git config --get user.email
dev.patel@intervision.com    # includeIf not firing — missing trailing slash
```

**Fix:**

```bash
$ git config --global --edit
# Change:  gitdir:~/dev/oss
# To:      gitdir:~/dev/oss/
```

**Prevention:** Always include the trailing slash in `includeIf "gitdir:..."` paths. Test immediately after adding an `includeIf` block with `git config --list --show-origin` from inside a repo in that directory.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# In ~/.gitconfig [alias] section

[alias]
  # Show all config with file origins — the single most useful config diagnostic command
  cfg     = config --list --show-origin

  # Show only the resolved (winning) value for a given key
  cfgget  = config --get

  # Open global config in editor
  cfgedit = config --global --edit

  # Show current commit identity — run this before committing in any new repo
  whoami  = "!echo \"Name:  $(git config user.name)\nEmail: $(git config user.email)\""
```

Usage:

```bash
$ git cfg                          # full config diagnostic
$ git cfgget user.email            # what email will commits use?
$ git cfgedit                      # open ~/.gitconfig in VS Code
$ git whoami                       # quick identity check
Name:  Alex Mercer
Email: alex.mercer@intervision.com
```

### Automation Opportunity

For teams managing multiple developer identities across work and personal repos, `includeIf` in `~/.gitconfig` is the cleanest solution — no per-repo `git config --local` needed, just keep all work repos under one directory and all personal repos under another:

```ini
# ~/.gitconfig

[user]
    name  = Alex Mercer
    email = alex.mercer@intervision.com    # default (work)

# Override for anything under ~/dev/oss/
[includeIf "gitdir:~/dev/oss/"]
    path = ~/.gitconfig-oss

# Override for anything under ~/dev/personal/
[includeIf "gitdir:~/dev/personal/"]
    path = ~/.gitconfig-personal
```

```ini
# ~/.gitconfig-oss
[user]
    name  = amercer-oss
    email = amercer@gmail.com
```

```ini
# ~/.gitconfig-personal
[user]
    name  = amercer
    email = amercer@personal-domain.com
```

Now identity switches automatically based on which directory a repo lives in — no manual `git config --local` ever needed.

### Keyboard / CLI Efficiency Notes

- `git config --list --show-origin` is the one command to know cold — memorise it, alias it to `git cfg`, run it first in any config debugging session
- `git config --global --edit` is faster than typing individual `git config --global key value` commands when setting more than two keys at once — the INI format is trivial to edit in bulk
- `git config --get-all <key>` reveals every value for a key across all scopes simultaneously — use it when `--list --show-origin` output is large and you only care about one key
- On Windows (PowerShell), the global config lives at `C:\Users\<username>\.gitconfig` — same INI format, same commands, just a different path. `$env:USERPROFILE\.gitconfig` is the reliable env-var reference.

### Spoken Answers

**Q: Walk me through the three Git config scopes, their file locations, and the precedence order.**

> **Average answer:** "There's global, local, and system. Local overrides global overrides system."

> **Great answer:** "System scope lives at `/etc/gitconfig` on Linux/Mac — it applies to every user on the machine and is usually set by a system administrator or a package manager. Global scope lives at `~/.gitconfig` — one per OS user, applies to every repo that user works in. Local scope lives at `.git/config` inside each repo — applies only to that repo. Git resolves any setting by checking local first, then global, then system — so local always wins. In practice I almost never touch system config. I set my identity, editor, and workflow defaults in global once per machine, and use local only when I need to override something for a specific repo — like a different email for a personal project. The command `git config --list --show-origin` shows all three scopes simultaneously with the file path next to each value, which is the fastest way to see which file is actually winning for any given key."

---

**Q: A new engineer's commits are showing up as `Unknown User <user@MacBook-Pro.local>`. What's the root cause and what's the fix?**

> **Average answer:** "They haven't set `user.name` and `user.email`. Run the two `git config --global` commands."

> **Great answer:** "Root cause is that `user.name` and `user.email` are not set at any scope, so Git falls back to constructing an identity from the OS username and hostname — which gives you something like `user@MacBook-Pro.local`. Fix is `git config --global user.name` and `git config --global user.email`. But there's a subtlety: that fix only affects _future_ commits. Any commits already made with the wrong identity are immutable objects — the config change doesn't rewrite them. If those bad commits haven't been pushed, they can be amended with `git rebase -i` and `git commit --amend --author`. If they have been pushed and are already in a PR, the team needs to decide whether to rewrite the branch history — which requires a force push — or just leave the author field wrong on those early commits and move on. The real prevention is an onboarding script that sets these values as step one, before cloning any repo."

---

**Q: You work on both InterVision repos and personal open-source projects on the same machine. How do you ensure the right email appears on each set of commits?**

> **Average answer:** "Use `git config --local user.email` in each personal repo."

> **Great answer:** "The `--local` approach works but doesn't scale — you have to remember to run it in every new personal repo you clone, and if you forget once you get the wrong email. The better solution is `includeIf` in `~/.gitconfig`. You structure your filesystem so all work repos live under `~/dev/intervision/` and all personal repos under `~/dev/oss/`, then add two `includeIf "gitdir:..."` blocks to your global config, each pointing to a separate config file with the appropriate `[user]` identity. The trailing slash on the directory path is critical — without it, the match silently fails. Once set up, the identity switches automatically based on where the repo lives on disk. I test it immediately after setup with `git config --list --show-origin` inside a repo in each directory to confirm the right file is being included."

### War Story Prompt

A developer at InterVision pushed six months of commits to a public GitHub repo with their corporate email — `firstname.lastname@intervision.com` — exposed in every commit's author field. The company had a policy against publishing internal email addresses in public repos. The commits had already been merged to `main` and tagged in releases, so rewriting history would break downstream forks. The incident traced back to a single missed step in the onboarding doc: "set up Git config" was item 14 on a 20-item list, easy to skip. After the incident the team replaced the doc step with a mandatory `git-setup.sh` script that the onboarding automation ran on every new machine provisioning — and added an `includeIf` block that automatically applied a personal identity to anything under `~/dev/personal/`, so the developer never needed to remember the override.

## Decision Framework

### When to Reach for This

- [ ] Setting up Git on a new machine — global config first, before any commits
- [ ] The wrong author name or email appears on a commit — config scope is the first thing to check
- [ ] You need different identities for different sets of repos on the same machine
- [ ] A team member asks why their editor / merge behaviour / pull behaviour is wrong — config scope explains most of these
- [ ] Onboarding new engineers — a config setup script eliminates the most common day-one mistakes
- [ ] You're debugging a Git behaviour that "should" be set but isn't taking effect — `--show-origin` reveals the conflict

### When NOT To

- [ ] Do not touch system scope (`/etc/gitconfig`) for personal preferences — it affects every user on the machine; reserve it for machine-wide policy set by ops/infrastructure teams
- [ ] Do not commit `.git/config` to version control — it contains per-developer paths, remote URLs, and potentially credentials; it's listed in `.gitignore` by default and should stay there
- [ ] Do not use `git config` to set secrets or tokens — use a credential helper or environment variables instead; values in `.gitconfig` are plain text

### Tool Comparison

||`--global` (per user)|`--local` (per repo)|`includeIf` (per directory)|
|---|---|---|---|
|Scope|All repos for this OS user|This one repo only|All repos under a given filesystem path|
|Setup effort|One command|One command per repo|One-time setup in `~/.gitconfig`|
|Scales to many repos|Yes — set once|No — must repeat per repo|Yes — automatic once directory structure is established|
|Best for|Identity, editor, workflow defaults|One-off overrides (unusual remote, specific merge strategy)|Multi-identity workflows across structured directory trees|
|Risk|Setting a bad value here affects every repo|Risk is contained to one repo|Silently fails if directory path is missing trailing slash|

## Self-Check

### Conceptual Questions

1. List the three config scopes in order from lowest to highest priority. Where is each file located on Linux/Mac?
2. What command shows you every active Git config setting alongside the file it came from?
3. You run `git config --global user.email "new@email.com"` but `git config --get user.email` still returns the old value. What's the most likely explanation?
4. What is the exact error message Git produces when `user.name` and `user.email` are not set and you attempt a commit?
5. How do you make Git automatically use a different identity for all repos under `~/dev/oss/` without setting `git config --local` in each one?
6. What environment variables does Git check for editor selection, and in what order do they take precedence relative to `core.editor`?
7. How does `git config --get-all <key>` differ from `git config --get <key>`, and when is the former more useful? (Connect to the multi-remote scenario in [[9.012 — Git Remote Management — Multiple Remotes]])

<details> <summary>Answers</summary>

1. Lowest to highest: **system** (`/etc/gitconfig`) → **global** (`~/.gitconfig`) → **local** (`.git/config` inside the repo). Local always wins.
    
2. `git config --list --show-origin`. Each line is prefixed with the file path of the config file that provided that setting.
    
3. The most likely explanation is that a **local** config in the current repo's `.git/config` has `user.email` set to the old value — it's overriding the global config you just changed. `git config --list --show-origin | grep user.email` will reveal both values and confirm which file is winning.
    

```
Author identity unknown

*** Please tell me who you are.

Run

  git config --global user.email "you@example.com"
  git config --global user.name "Your Name"

to set your account's default identity.
Omit --global to set the identity only in this repository.

fatal: empty ident name (for <>) not allowed
```

5. Use `includeIf` in `~/.gitconfig`:
    
    ```ini
    [includeIf "gitdir:~/dev/oss/"]
        path = ~/.gitconfig-oss
    ```
    
    Where `~/.gitconfig-oss` contains the `[user]` block with the alternate identity. The trailing slash on the directory path is required for the match to fire.
    
6. Git checks in this order for editor selection: `GIT_EDITOR` environment variable → `core.editor` config setting → `VISUAL` environment variable → `EDITOR` environment variable. `GIT_EDITOR` has the highest priority and overrides `core.editor` entirely.
    
7. `git config --get <key>` returns only the single winning value — the one Git will actually use. `git config --get-all <key>` returns every value for that key across all scopes, one per line, without indicating which one wins. `--get-all` is most useful when you suspect a key is set in multiple places and `--get` is returning an unexpected value — it reveals all the competing entries at once. This parallels the multi-remote situation in [[9.012]]: `git remote -v` shows you all remotes (analogous to `--get-all`), while `git config --get remote.origin.url` shows you the single winning URL for `origin`.
    

</details>

**Drill 3 — Under time pressure**

It's 16:55 on a Friday. You've just discovered that the last 4 commits to your local `feature/payment-retry` branch (not yet pushed) were made with the author `CI Bot <ci@build-server.local>` — a ghost identity left over from a build server config that leaked into your shell environment via `GIT_AUTHOR_EMAIL`. Production isn't affected (the branch isn't deployed), but the PR opens Monday morning and the wrong author will be visible. You have 5 minutes before you need to leave. Fix the author on all 4 commits without losing any of the code changes.

<details> <summary>Solution</summary>

```bash
# 1 — Identify the bad commits
$ git log --format="%h %an <%ae> %s" -5
3c8f21a CI Bot <ci@build-server.local> fix: handle 503 in retry loop
b7d19e2 CI Bot <ci@build-server.local> feat: configurable retry count
a4f93c1 CI Bot <ci@build-server.local> feat: add RetryHandler skeleton
9e2b87d CI Bot <ci@build-server.local> chore: add RetryHandler test project
1a3f09c Alex Mercer <alex.mercer@intervision.com> feat: base PaymentGateway client  # good

# 2 — Fix the env variable causing the problem so it doesn't recur this session
$ unset GIT_AUTHOR_EMAIL
$ unset GIT_AUTHOR_NAME
$ unset GIT_COMMITTER_EMAIL
$ unset GIT_COMMITTER_NAME
# Also remove from ~/.zshrc or ~/.bashrc if it's set there

# 3 — Rewrite the last 4 commits with the correct author
# Using git rebase with exec to amend each commit's author:
$ git rebase -i HEAD~4
# In the editor, change all 4 lines from 'pick' to 'exec git commit --amend --author="Alex Mercer <alex.mercer@intervision.com>" --no-edit --reset-author'
# ... OR use the faster environment-variable rebase approach:

$ git rebase HEAD~4 --exec 'git commit --amend --author="Alex Mercer <alex.mercer@intervision.com>" --no-edit'
[detached HEAD 7f3a21b] fix: handle 503 in retry loop
[detached HEAD 2c8e19d] feat: configurable retry count
[detached HEAD 5b9f4a1] feat: add RetryHandler skeleton
[detached HEAD 8d2c73e] chore: add RetryHandler test project
Successfully rebased and updated refs/heads/feature/payment-retry.

# 4 — Verify all 4 commits now have the correct author
$ git log --format="%h %an <%ae> %s" -5
8d2c73e Alex Mercer <alex.mercer@intervision.com> chore: add RetryHandler test project
5b9f4a1 Alex Mercer <alex.mercer@intervision.com> feat: add RetryHandler skeleton
2c8e19d Alex Mercer <alex.mercer@intervision.com> feat: configurable retry count
7f3a21b Alex Mercer <alex.mercer@intervision.com> fix: handle 503 in retry loop
1a3f09c Alex Mercer <alex.mercer@intervision.com> feat: base PaymentGateway client   # unchanged

# 5 — Confirm code changes are intact
$ git diff HEAD~4 --stat
# Should show the same files changed as before — no code lost

# Branch not yet pushed, so no force-push needed. Done.
```

**Why `--exec` on rebase:** It's faster than marking each commit `edit` individually — the `--exec` flag runs the given shell command after every commit in the rebase sequence. Combined with `--amend --no-edit`, it rewrites only the author metadata, leaving the commit message and content untouched.

</details>
