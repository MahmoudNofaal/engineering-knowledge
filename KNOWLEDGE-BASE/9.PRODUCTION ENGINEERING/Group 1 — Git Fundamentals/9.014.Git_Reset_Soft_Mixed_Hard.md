### Where This Fits

You committed something you shouldn't have — wrong files, wrong message, debug code left in, or three commits that should have been one. Or you staged a file by mistake and need it back in the working tree. Or a feature branch went sideways and you need to blow it back to a known-good commit and start the last hour of work over. All three scenarios are `git reset` territory — the command that repositions HEAD and optionally resets the index and working tree to match.

---

## What You're Doing and Why

`git reset` moves the current branch's HEAD pointer to a target commit and then — depending on the mode — leaves your index and working tree in varying states. The three modes are a spectrum from "undo the commit, keep everything staged" (`--soft`) to "undo the commit, unstage everything, keep files" (`--mixed`, the default) to "undo the commit and discard all changes" (`--hard`). The mode controls how much of the undo reaches down into your working state.

Reset only rewrites _local, unpushed_ history safely. Once commits are on a shared remote, reset + force-push is a team hazard — see [[9.036 — Force Push — Risks and Safer Alternatives]] before going there. For already-pushed commits, `git revert` (see [[9.013 — Undoing Changes — Checkout, Restore, Revert]]) is the safe alternative.

### Mental Model

Think of the three zones as three stacked layers — working tree on top, index in the middle, HEAD (the last commit) on the bottom:

```
  Working tree  ← files you can see and edit
  Index         ← staged snapshot (what the next commit will contain)
  HEAD          ← the last committed snapshot
```

`git reset` always moves HEAD to the target commit. The mode decides how many layers below HEAD it also resets to match:

```
  --soft:   move HEAD only        → index and working tree unchanged
  --mixed:  move HEAD + reset index → working tree unchanged (files intact, unstaged)
  --hard:   move HEAD + reset index + reset working tree → everything matches target commit
```

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git reset --soft HEAD~1`|Undo last commit; keep changes staged|Committed too early; want to re-commit with a better message or more files|
|`git reset HEAD~1`|Undo last commit; unstage changes, keep files|Committed wrong files; need to re-stage selectively|
|`git reset --mixed HEAD~1`|Same as above (default mode)|Identical to above — `--mixed` is the default|
|`git reset --hard HEAD~1`|Undo last commit and discard all its changes|Committed something you want completely gone locally|
|`git reset --hard HEAD`|Discard all staged and unstaged changes; return to last commit|Nuclear working-tree cleanup; no commits lost|
|`git reset --hard origin/main`|Reset local branch to exactly match remote|"Throw away everything local and start fresh from remote"|
|`git reset HEAD <file>`|Unstage a specific file (keep working tree edit)|Accidentally staged the wrong file; equivalent to `git restore --staged <file>` in modern Git|
|`git reset <commit>`|Move HEAD to a specific commit (mixed mode default)|Collapsing a range of commits back to working tree for re-staging|
|`git reset --soft <commit>`|Move HEAD back N commits; keep all their changes staged|Squashing N commits into one without interactive rebase|

## Step-by-Step Walkthrough

### The Scenario

You're on `feature/order-refunds` in the `OrderService` repo. You made three commits in the last hour, but they should have been one — they're all part of the same logical change. Plus, your last commit accidentally included a debug file. You need to undo, clean up, and re-commit properly.

### Full Walkthrough

```bash
$ git log --oneline
9f8e7d6 (HEAD -> feature/order-refunds) debug: temp logging for refund test
7b2c1d3 feat: add refund validation logic
4a3f9c2 feat: scaffold RefundHandler
d7e8b31 (origin/main, main) fix: prevent null ref in RefundHandler
```

Four local commits — three belong to this feature. The top commit (`9f8e7d6`) includes debug logging that shouldn't ship. Below it are two feature commits that should be one. Plan: hard-reset the debug commit away, then soft-reset the two feature commits into the index to squash them.

```bash
$ git reset --hard HEAD~1
HEAD is now at 7b2c1d3 feat: add refund validation logic
```

`--hard HEAD~1` removes the debug commit entirely — its changes are gone from both the index and the working tree. No trace remains. Use this only when you're certain you want the changes deleted, not just uncommitted.

```bash
$ git log --oneline
7b2c1d3 (HEAD -> feature/order-refunds) feat: add refund validation logic
4a3f9c2 feat: scaffold RefundHandler
d7e8b31 (origin/main, main) fix: prevent null ref in RefundHandler
```

The debug commit is gone. Now squash the two remaining feature commits into one.

```bash
$ git reset --soft HEAD~2
```

`--soft HEAD~2` moves HEAD back two commits (to `d7e8b31`) without touching the index or working tree. All the changes from those two commits are now staged — exactly as if you had written everything in one session and staged it all at once.

```bash
$ git status
On branch feature/order-refunds
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        new file:   src/OrderService/RefundHandler.cs
        modified:   src/OrderService/OrderValidator.cs
        modified:   src/OrderService/OrderService.csproj
```

Everything is staged. No working-tree modifications. This is the clean slate for one well-formed commit.

```bash
$ git diff --staged --stat
 src/OrderService/RefundHandler.cs   | 48 ++++++++++++++++++++++++++++++++++++
 src/OrderService/OrderValidator.cs  | 12 +++++++++++-
 src/OrderService/OrderService.csproj |  2 +-
 3 files changed, 61 insertions(+), 1 deletion(-)
```

Verify what's staged before committing — `git diff --staged` is mandatory after any reset that leaves things in the index. This is what [[9.003 — Git Status and Diff — Reading the Working Tree]] is for.

```bash
$ git commit -m "feat: implement RefundHandler with validation

- Add RefundHandler.cs with idempotency check
- Add validation in OrderValidator for refund eligibility
- Add RefundResult enum to OrderService.csproj"
[feature/order-refunds 2c4e6a8] feat: implement RefundHandler with validation
 3 files changed, 61 insertions(+), 1 deletion(-)
 create mode 100644 src/OrderService/RefundHandler.cs
```

One clean commit replacing two messy ones and one debug commit. History looks like it was planned from the start.

```bash
$ git log --oneline
2c4e6a8 (HEAD -> feature/order-refunds) feat: implement RefundHandler with validation
d7e8b31 (origin/main, main) fix: prevent null ref in RefundHandler
```

### What Could Go Wrong Mid-Walkthrough

**If you did `--hard` and immediately regret it:**

The commit is gone from the branch but not yet from Git's object store. `git reflog` has it:

```bash
$ git reflog
7b2c1d3 HEAD@{0}: reset: moving to HEAD~1
9f8e7d6 HEAD@{1}: commit: debug: temp logging for refund test
```

```bash
$ git reset --hard 9f8e7d6   # or: git cherry-pick 9f8e7d6
HEAD is now at 9f8e7d6 debug: temp logging for refund test
```

Recovery window is roughly 30 days before Git's garbage collection removes unreferenced objects. See [[9.031 — Git Reflog — Recovering Lost Commits]] for the full recovery procedure.

**If you used `--mixed` expecting `--soft` (files are unstaged, not staged):**

```bash
$ git status
Changes not staged for commit:   # ← not what you wanted
        modified:   src/OrderService/RefundHandler.cs
```

Re-stage everything and continue — no data was lost, just staging state:

```bash
$ git add -p    # stage selectively, or
$ git add .     # stage everything
```

## Production Scenario

### Realistic Context

You're the on-call engineer for `PaymentGateway`. Thirty minutes ago you committed a config change directly to your local `main` (it was an emergency) that included a hardcoded connection string pointing to your local SQL Server — not the production one. Before you pushed it, you caught the error. Now you need to cleanly undo that commit, recover the legitimate config change without the hardcoded string, and re-commit correctly. The branch has not been pushed.

### Full Implementation / Resolution

```bash
$ git log --oneline
f3c2b1a (HEAD -> main) fix: update retry policy and connection timeout
d7e8b31 (origin/main) fix: prevent null ref in ChargeProcessor
a1b2c3d chore: bump Stripe.net to 43.4.0
```

The bad commit is `f3c2b1a`. It has two legitimate changes (retry policy, connection timeout) and one poisonous one (hardcoded connection string). A `--hard` reset would lose the legitimate changes. A `--soft` reset keeps everything staged — right tool here.

```bash
$ git reset --soft HEAD~1
```

```bash
$ git status
On branch main
Changes to be committed:
        modified:   src/PaymentGateway/appsettings.json
        modified:   src/PaymentGateway/ChargeProcessor.cs
        modified:   src/PaymentGateway/PaymentGateway.csproj
```

All three files are staged. Now inspect exactly what's in `appsettings.json` to find the bad line:

```bash
$ git diff --staged -- src/PaymentGateway/appsettings.json
diff --git a/src/PaymentGateway/appsettings.json b/src/PaymentGateway/appsettings.json
index c4d5e6f..7a8b9c0 100644
--- a/src/PaymentGateway/appsettings.json
+++ b/src/PaymentGateway/appsettings.json
@@ -3,7 +3,8 @@
   "ConnectionStrings": {
-    "PaymentDb": "#{PaymentDb_ConnectionString}#"
+    "PaymentDb": "Server=localhost;Database=PaymentDev;User Id=sa;Password=devpassword123;"
   },
   "RetryPolicy": {
-    "MaxRetries": 3
+    "MaxRetries": 5
   },
-  "ConnectionTimeout": 30
+  "ConnectionTimeout": 60
```

The hardcoded connection string is on the `appsettings.json` change. The retry and timeout changes (`ChargeProcessor.cs`, `PaymentGateway.csproj`) are fine. Unstage only the bad file, fix it, re-stage:

```bash
$ git restore --staged src/PaymentGateway/appsettings.json
```

```bash
$ git status
Changes to be committed:
        modified:   src/PaymentGateway/ChargeProcessor.cs
        modified:   src/PaymentGateway/PaymentGateway.csproj

Changes not staged for commit:
        modified:   src/PaymentGateway/appsettings.json
```

Fix the working-tree file to restore the placeholder token instead of the hardcoded string:

```bash
$ git diff -- src/PaymentGateway/appsettings.json
-    "PaymentDb": "Server=localhost;Database=PaymentDev;User Id=sa;Password=devpassword123;"
+    "PaymentDb": "#{PaymentDb_ConnectionString}#"
```

```bash
$ git add src/PaymentGateway/appsettings.json
$ git commit -m "fix: increase retry policy to 5 and connection timeout to 60s

Connection string token left as pipeline placeholder #{PaymentDb_ConnectionString}#"
[main 8b3d1e2] fix: increase retry policy to 5 and connection timeout to 60s
 3 files changed, 4 insertions(+), 2 deletions(-)
```

```bash

@@ -3,4 +3,4 @@
   "ConnectionStrings": {
-    "PaymentDb": "#{PaymentDb_ConnectionString}#"
+    "PaymentDb": "#{PaymentDb_ConnectionString}#"   ← token, not a real string

d7e8b31 (origin/main) fix: prevent null ref in ChargeProcessor
```

No hardcoded credentials in the diff. No extra commits. Safe to push.

## Gotchas and Troubleshooting

### "`git reset --hard HEAD~1` — I lost work I needed"

**Root cause:** `--hard` discards working tree changes without confirmation. If the commit contained the only copy of some work, it's gone from the branch — but not from Git's object store yet.

```bash
$ git reset --hard HEAD~1
HEAD is now at 7b2c1d3 feat: add refund validation logic
# Oh no — that commit had 3 hours of work
```

**Fix:**

```bash
$ git reflog
9f8e7d6 HEAD@{1}: commit: feat: implement payment retry
7b2c1d3 HEAD@{0}: reset: moving to HEAD~1

$ git reset --hard 9f8e7d6   # restore the "lost" commit
HEAD is now at 9f8e7d6 feat: implement payment retry
```

**Prevention:** Before any `--hard` reset, run `git stash` to preserve the working tree, or at minimum read `git diff HEAD` to confirm you know what you're about to discard. Reserve `--hard` for changes you have consciously decided to delete.

---

### "`git reset --hard origin/main` wiped my feature branch work"

**Root cause:** You were on a feature branch and reset it to `origin/main` — this forcibly moves the branch pointer back to `origin/main`'s tip, discarding all local commits and working tree changes on that branch.

```bash
# On feature/order-refunds with 6 local commits
$ git reset --hard origin/main   # ← intended for main, typed on wrong branch
HEAD is now at d7e8b31 fix: prevent null ref in RefundHandler
```

**Fix:**

```bash
$ git reflog
d7e8b31 HEAD@{0}: reset: moving to origin/main
9f8e7d6 HEAD@{1}: commit: feat: add refund status endpoint
# ... all 6 commits are still in reflog

$ git reset --hard 9f8e7d6   # jump back to where the branch was
```

**Prevention:** Always run `git branch` to confirm which branch you're on before any `--hard` reset. The alias `git branch --show-current` (or `git rev-parse --abbrev-ref HEAD`) takes two seconds and can save hours.

---

### "`git reset` says 'fatal: ambiguous argument HEAD~3': unknown revision"

**Root cause:** You're trying to reset further back than the branch's history — the branch has fewer commits than the `~N` offset you specified.

```bash
$ git log --oneline
4a3f9c2 feat: scaffold RefundHandler   ← only 1 commit on this branch
d7e8b31 (origin/main) fix: null ref

$ git reset --soft HEAD~3
fatal: ambiguous argument 'HEAD~3': unknown revision or path not in the working tree.
```

**Fix:**

```bash
$ git log --oneline | wc -l   # count commits available
# Use a smaller offset, or use the exact commit hash of the target
$ git reset --soft d7e8b31    # reset back to the specific commit you want
```

**Prevention:** Use `git log --oneline` to count commits before `HEAD~N` resets on young branches. Prefer commit hashes over `~N` offsets when the target is more than 3 commits back.

---

### "After `git reset --soft`, my files are staged but `git diff` shows nothing"

**Root cause:** This is correct behavior — not a bug. After `--soft`, changes are in the index. `git diff` (no args) compares working tree to index and shows nothing because they match. The changes are in `git diff --staged`.

```bash
$ git reset --soft HEAD~2
$ git diff
# (empty — working tree matches index)
$ git diff --staged
# (shows all the changes from the two reset commits — this is what you want)
```

**Fix:** Run `git diff --staged` instead of `git diff`. See [[9.003 — Git Status and Diff — Reading the Working Tree]] for the full `diff` zone map.

**Prevention:** After any reset, always run both `git status` and `git diff --staged` to see the full picture. `git status` alone tells you files exist in the index; `git diff --staged` shows you the actual content.

---

### "I reset a commit that was already pushed — now my push is rejected"

**Root cause:** `git reset` rewrites local history. If the commit you reset was already on `origin`, your local branch is now behind _and_ diverged — `git push` sees a non-fast-forward and refuses.

```bash
$ git push origin feature/order-refunds
To github.com:intervision/OrderService.git
 ! [rejected]        feature/order-refunds -> feature/order-refunds (non-fast-forward)
error: failed to push some refs to 'github.com:intervision/OrderService.git'
hint: Updates were rejected because the tip of your current branch is behind
hint: its remote counterpart.
```

**Fix — if you are the only person on this branch:**

```bash
$ git push --force-with-lease origin feature/order-refunds
```

**Fix — if others have pulled from this branch:** Do not force-push. Use `git revert` to create a new commit that undoes the bad commit instead. See [[9.013 — Undoing Changes — Checkout, Restore, Revert]] and [[9.036 — Force Push — Risks and Safer Alternatives]].

**Prevention:** Before resetting, run `git log origin/feature/order-refunds..HEAD` — if it shows the commit, it's already pushed and reset + force-push will be required. That's the signal to reach for `revert` instead.

---

### "`git reset HEAD <file>` doesn't work in Git 2.23+"

**Root cause:** In Git 2.23+, `git restore --staged <file>` replaced `git reset HEAD <file>` as the canonical way to unstage a single file. The old form still works but triggers a deprecation hint in some versions, and the intent is clearer with `restore`.

```bash
$ git reset HEAD src/OrderService/RefundHandler.cs
Unstaged changes after reset:
M       src/OrderService/RefundHandler.cs
# Works, but:

$ git restore --staged src/OrderService/RefundHandler.cs
# Same result, clearer intent, no hint
```

**Prevention:** Use `git restore --staged <file>` for per-file unstaging. Reserve `git reset` for moving HEAD.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# ~/.gitconfig — add under [alias]
[alias]
  undo     = reset --soft HEAD~1       # undo last commit, keep staged — most common reset
  unstage  = restore --staged          # unstage a file: git unstage src/Foo.cs
  wipe     = reset --hard HEAD         # discard all working-tree changes, keep commits
  nuke     = reset --hard              # full reset to a target: git nuke HEAD~3
  rb       = reset --hard origin/main  # sync local main to remote — use carefully
```

The `undo` alias is the one you'll reach for multiple times a week: commit too early, realize message is wrong or files are incomplete, `git undo`, fix, recommit.

### Automation Opportunity

Pre-push hook that blocks `--force` pushes to `main` or `release/*` branches — catches the most dangerous consequence of reset misuse before it reaches the remote:

```bash
#!/usr/bin/env bash
# .git/hooks/pre-push
# Prevents force-push to protected branches

remote="$1"
branch=$(git rev-parse --abbrev-ref HEAD)

protected_pattern="^(main|release/.*)$"

if [[ "$branch" =~ $protected_pattern ]]; then
  # Check if this is a force push by comparing with remote tip
  remote_sha=$(git ls-remote "$remote" "refs/heads/$branch" | cut -f1)
  local_sha=$(git rev-parse HEAD)

  if [ -n "$remote_sha" ]; then
    merge_base=$(git merge-base "$local_sha" "$remote_sha")
    if [ "$merge_base" != "$remote_sha" ]; then
      echo "ERROR: Force push to $branch is blocked by pre-push hook."
      echo "Use 'git revert' for already-pushed commits on protected branches."
      exit 1
    fi
  fi
fi

exit 0
```

```bash
$ chmod +x .git/hooks/pre-push
```

### Keyboard / CLI Efficiency Notes

- **`git reset --soft HEAD~1` is the single most useful reset** — memorize it as the "uncommit" command. Everything else follows from understanding that one.
- **Always pair reset with `git reflog`** in your mental model. Nothing reset does is truly permanent for ~30 days. Knowing this makes you bolder and faster with cleanup operations.
- **`git log --oneline -5`** before any `HEAD~N` reset — five keystrokes of confirmation that prevent the "wrong branch" or "too far back" class of mistake.
- **After `--mixed` or `--soft`, immediately run `git diff --staged`** — muscle memory that takes two seconds and validates the reset did exactly what you intended.

### Spoken Answers

**Q: Explain the three modes of `git reset`.**

> **Average answer:** "`--soft` keeps your changes staged, `--mixed` unstages them, `--hard` deletes them."

> **Great answer:** "All three modes move HEAD to the target commit — that part is always the same. What changes is how many of Git's three zones get reset along with HEAD. `--soft` moves HEAD and nothing else — your index (staging area) and working tree are untouched, so all the changes from the undone commits are still staged and ready to be re-committed. `--mixed` is the default — it moves HEAD and resets the index to match the target commit, but leaves your working tree files intact; the changes are now unstaged, sitting in the working tree waiting to be `git add`-ed again. `--hard` moves HEAD and resets both the index and the working tree — the changes from the undone commits are completely gone from your machine. A concrete way I use all three: I use `--soft HEAD~N` to squash commits before a PR — all the work stays staged and I write one clean commit message. I use `--mixed` when I staged a wrong file and want to un-stage it selectively. I use `--hard` when I want to abort an entire line of experimentation and return to a known-good state."

---

**Q: When should you use `git revert` instead of `git reset`?**

> **Average answer:** "Use `revert` when commits are already pushed."

> **Great answer:** "The decision is whether the commits you're undoing have already been shared with other people. `git reset` rewrites history — it moves the branch pointer backwards, which means anyone who has pulled those commits will have a diverged history when you force-push. `git revert` creates a _new_ commit that inverts the changes, leaving history intact — safe to push without force, safe for the team. My rule: if the commit is on a remote branch that anyone else has or might have pulled, I always `revert`. If it's purely local — hasn't left my machine — I can freely `reset`. There's a middle case on solo feature branches where I'm confident no one else has pulled it — there I'll `reset` and `--force-with-lease` push, but I double-check with `git log origin/feature-branch..HEAD` first to confirm it was actually pushed before making that call."

---

**Q: You ran `git reset --hard` and lost work. What do you do in the next 60 seconds?**

> **Average answer:** "Check `git reflog` to find the commit."

> **Great answer:** "First: don't panic and don't run any more Git commands that could move HEAD further — every command from here is read-only until I've recovered. I immediately run `git reflog` — this shows every position HEAD has been at for the last 30 days, including commits that are no longer reachable from any branch. I find the entry just before the reset — it'll show the commit hash with a label like `HEAD@{1}: commit: feat: add refund validation`. I then run `git reset --hard <that-hash>` to restore the branch to that commit, or `git cherry-pick <that-hash>` if I only want the content without moving HEAD. The commit is still in Git's object store — `git reset` never actually deletes objects, it just removes branch references to them. GC won't clean them up for roughly 30 days. The reflog is the safety net that makes `--hard` recoverable."

### War Story Prompt

A senior engineer on `InventoryApi` ran `git reset --hard origin/main` intending to sync a stale local `main`. They were on `feature/inventory-bulk-import` — a branch they'd been working on for two days with eleven local commits, none pushed. The reset moved the feature branch pointer back to `origin/main`, dropping all eleven commits from the branch. They didn't notice for twenty minutes. The recovery: `git reflog` surfaced the most recent feature commit hash; `git reset --hard <hash>` restored the branch in under thirty seconds. Total loss: zero lines of code. The new habit: `git branch --show-current` before any destructive operation, and a shell alias that confirms the branch name before executing any `--hard` reset.

## Decision Framework

### When to Reach for This

- [ ] You committed too early and want to re-stage differently — `--soft HEAD~1`
- [ ] You staged the wrong file and want to unstage it without losing the edit — `git restore --staged <file>` (single file) or `--mixed HEAD` (everything)
- [ ] You want to squash multiple local commits into one — `--soft HEAD~N` then re-commit
- [ ] A line of work is completely wrong and you want to wipe it and start over — `--hard <target-commit>`
- [ ] Your working tree is a mess and you want to return to the last clean commit — `--hard HEAD`

### When NOT To

- [ ] Never use `--hard` to undo commits that are already on a remote branch others may have pulled — use `git revert` instead
- [ ] Don't use `git reset HEAD <file>` to unstage in modern Git — use `git restore --staged <file>`; cleaner intent
- [ ] Don't use `--hard` when `--soft` or `--mixed` would do — `--hard` is the most destructive option; don't reach for it when you're uncertain about what you want to keep
- [ ] Don't use reset to clean up pushed history on `main` or `release/*` — the pre-push hook (Section 6) should prevent this; use `revert` instead

### Tool Comparison

||`git reset --soft`|`git reset --mixed`|`git reset --hard`|`git revert`|
|---|---|---|---|---|
|Moves HEAD|Yes|Yes|Yes|No (adds new commit)|
|Resets index|No|Yes|Yes|N/A|
|Resets working tree|No|No|Yes|N/A|
|Data loss risk|None|None|High|None|
|Safe after push|No|No|No|Yes|
|Creates new commit|No|No|No|Yes|
|When to use|Re-commit; squash|Re-stage; selective add|Abort; wipe experiment|Already pushed commits|

6. Why is `git reset --hard origin/main` dangerous when you're on a feature branch, not `main`?
7. How does this note connect to [[9.026 — Interactive Rebase — Squash, Reword, Drop]]? When would you reach for reset instead of interactive rebase to squash commits?

<details> <summary>Answers</summary>

1. All three modes always **move the current branch's HEAD pointer** to the target commit. That part is unconditional — the mode only controls what happens to the index and working tree after HEAD moves.
    
2. Three commits are removed from branch history. Their changes go into the **index (staging area)** — they appear under "Changes to be committed" in `git status`, as if you had staged all of them yourself in one session.
    
3. `git reset --hard HEAD` resets the index and working tree to match the **current** last commit — it discards unstaged and staged changes but keeps HEAD where it is. `git reset --hard HEAD~1` moves HEAD **back one commit** and resets index and working tree to match that prior commit — it discards the last commit's changes entirely.
    
4. They appear under **"Changes not staged for commit"** — the working tree zone. `--mixed` resets the index to match the target commit, so the changes are no longer staged; they remain as working-tree modifications.
    
5. `git reflog` to find the commit hash, then `git reset --hard <hash>` or `git cherry-pick <hash>` to restore it.
    
6. When you're on a feature branch, `git reset --hard origin/main` moves the **feature branch pointer** to `origin/main`'s tip — it does not switch you to `main`. All local commits on the feature branch are dropped from the branch history (though recoverable via reflog). You've effectively destroyed your feature branch by pointing it at main.
    
7. Both achieve squashing, but with different tradeoffs. `git reset --soft HEAD~N` squashes N commits into the index in one step — fast, simple, no interactive editor, but you get one commit from all N. Interactive rebase (`git rebase -i HEAD~N`) lets you squash selectively, reword individual messages, reorder, and drop — more powerful but more steps. Use `--soft` reset for "I want these N commits to be one commit, period." Use interactive rebase when the commits need different treatment — some squashed, some kept, some reworded.
    

</details>

**Drill 3 — Under time pressure**

You're in a production incident. Fifteen minutes ago you committed a hardcoded API key to `feature/notification-webhook` while testing locally:

```bash
$ git log --oneline
3a4b5c6 (HEAD -> feature/notification-webhook) test: hardcode SendGrid key for local test
2b3c4d5 feat: implement webhook dispatch for NotificationWorker
1a2b3c4 feat: scaffold NotificationWebhookService
d7e8b31 (origin/main, main) fix: null ref fix
```

The branch was pushed to origin two minutes ago — a teammate already has it checked out. You need to remove the API key from history, rotate the key (that's the security team's job, not yours right now), and push a clean version of the branch. You have 3 minutes before the next standup. Walk through exactly what you type.

<details> <summary>Solution</summary>

```bash
# 1. Confirm the bad commit is the top one and the key is only there
$ git show HEAD --stat
commit 3a4b5c6
    test: hardcode SendGrid key for local test
 src/NotificationWorker/WebhookDispatcher.cs | 3 ++-
 1 file changed, 2 insertions(+), 1 deletion(-)

# 2. Soft-reset: keep changes staged so we can fix only the key line
$ git reset --soft HEAD~1

# 3. Unstage the file, fix the key line in working tree
$ git restore --staged src/NotificationWorker/WebhookDispatcher.cs

# 4. Edit the file: replace hardcoded key with config reference
#    e.g. change: _client = new SendGridClient("SG.abc123...");
#    to:          _client = new SendGridClient(_config["SendGrid:ApiKey"]);
$ code src/NotificationWorker/WebhookDispatcher.cs   # fix inline

# 5. Re-stage the clean file and commit
$ git add src/NotificationWorker/WebhookDispatcher.cs
$ git diff --staged   # verify key is gone
$ git commit -m "test: verify webhook dispatch (config key, not hardcoded)"
[feature/notification-webhook 9f0e1d2] test: verify webhook dispatch (config key, not hardcoded)

# 6. Force-push with lease — warns if teammate pushed anything since you last fetched
$ git push --force-with-lease origin feature/notification-webhook
Enumerating objects: 7, done.
...
To github.com:intervision/NotificationWorker.git
 + 3a4b5c6...9f0e1d2 feature/notification-webhook -> feature/notification-webhook (forced update)

# 7. Tell teammate immediately — they need to reset their local copy
# git fetch origin && git reset --hard origin/feature/notification-webhook
```

The `--soft` + `restore --staged` pattern is the right choice here over `--hard`: you want to keep the legitimate code changes in `WebhookDispatcher.cs` (the webhook dispatch logic) and only remove the one bad line. `--hard` would erase all the changes from that commit, including the code you spent time writing.

</details>
