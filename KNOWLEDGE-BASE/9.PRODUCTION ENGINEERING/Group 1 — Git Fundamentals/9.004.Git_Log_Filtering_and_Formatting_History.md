### Where This Fits

You're on a call at 11 PM. A bug just appeared in `OrderService`'s refund calculation. You need to know: which commit changed `RefundCalculator.cs` in the last two weeks, who wrote it, and whether it touched the same lines that are now broken. Or it's Monday morning and you're reviewing what the team merged over the weekend before you start your own feature. In both cases you reach for `git log` — not the GitHub UI, which is slower and unavailable when the VPN is down. `git log` is the primary interface to history, and knowing how to filter and format it is what separates fast debugging from random guessing.

---

## What You're Doing and Why

`git log` walks the commit graph backward from a starting ref (default: `HEAD`) and prints metadata for each commit it visits. Without flags, the output is verbose and unsorted for human scanning; with the right flags it becomes a surgical tool. You filter by author, date, file path, or commit message keyword to narrow thousands of commits down to the handful relevant to your investigation. You format the output — one line per commit, graph view, custom fields — to make the result scannable at a glance.

### Mental Model

Think of the commit graph as a river flowing backward in time. `git log` is a net you drag upstream. The filter flags (`--author`, `--since`, `--grep`, `-- <path>`) change the mesh size of the net. The format flags (`--oneline`, `--graph`, `--format`) change what the net surfaces — the whole fish, or just the label on the side.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git log`|Full verbose log, most recent first|Rarely — too noisy without flags|
|`git log --oneline`|One SHA + subject line per commit|Daily — the default starting point|
|`git log --oneline --graph --all`|ASCII branch/merge graph across all refs|Understanding branch topology|
|`git log --oneline -n 10`|Last 10 commits only|Quick "what just happened" check|
|`git log --author="Fatima"`|Commits by one author|Reviewing a teammate's recent work|
|`git log --since="2 weeks ago"`|Commits newer than a relative date|Scoping an incident to a time window|
|`git log --until="2024-12-01"`|Commits older than an absolute date|Historical audit at a specific release|
|`git log --grep="refund"`|Commits whose message contains a string|Finding where a feature was introduced|
|`git log -S "RefundCalculator"`|Commits that added/removed this string in a file|**Pickaxe search** — the most powerful single filter|
|`git log -- src/OrderService/`|Commits touching a specific path|Narrowing to one service in a monorepo|
|`git log --follow -- OldName.cs`|Track a file through renames|When a file was moved but history is needed|
|`git log --merges`|Merge commits only|Auditing when branches landed|
|`git log --no-merges`|Exclude merge commits|Cleaner feature-commit history|
|`git log --stat`|Show files changed and line counts per commit|Mid-detail view without full diff|
|`git log -p`|Show full patch (diff) for each commit|Deepest inspection — slow on large repos|
|`git log --format="%h %an %ar %s"`|Custom format: short SHA, author, relative date, subject|Scripting, copy-pasteable for reports|
|`git log branchA..branchB`|Commits in B not in A|What's in a PR / what's ahead of main|
|`git log origin/main..HEAD`|Commits not yet pushed|Pre-push sanity check|
|`git log --all --oneline --decorate`|All refs with branch/tag labels|Seeing where all heads are|
|`git shortlog -sn`|Commit count per author, sorted|Team activity summary|

## Step-by-Step Walkthrough

### The Scenario

You're investigating a bug in `PaymentGateway`'s charge-back handler. A duplicate charge appeared on 14 June 2026. You need to find every commit that touched `src/PaymentGateway/ChargebackHandler.cs` between 1 June and 16 June, see who wrote each one, and then inspect the diff for the most suspicious commit.

### Full Walkthrough

```bash
$ cd ~/repos/PaymentGateway

# Step 1: Get a quick orientation — what does recent history look like overall?
$ git log --oneline -n 15
a7c3f12 fix: retry logic for Stripe webhook timeout
b2d81e0 feat: add chargeback reason code mapping
c5a9314 chore: update NuGet packages
d4b7f2a fix: prevent duplicate chargeback on 402 response
e1c0983 feat: chargeback handler — initial implementation
f8d3a01 refactor: extract IChargebackRepository interface
g9b2c44 fix: null check on PaymentIntent before state transition
...
```

> Orientation pass: get your bearings before narrowing. The `--oneline` flag makes 15 commits fit on one screen.

```bash
# Step 2: Narrow to the incident window and the specific file
$ git log --oneline --since="2026-06-01" --until="2026-06-16" -- src/PaymentGateway/ChargebackHandler.cs
d4b7f2a fix: prevent duplicate chargeback on 402 response
b2d81e0 feat: add chargeback reason code mapping
e1c0983 feat: chargeback handler — initial implementation
```

> Three commits in the window. The `-- src/PaymentGateway/ChargebackHandler.cs` path filter tells git to only walk commits that touched this exact file — not other files in the same commit.

```bash
# Step 3: Add author and relative date to see who and when
$ git log --format="%h  %an  %ar  %s" --since="2026-06-01" --until="2026-06-16" -- src/PaymentGateway/ChargebackHandler.cs
d4b7f2a  Yasmine Morsi  5 days ago  fix: prevent duplicate chargeback on 402 response
b2d81e0  Omar Khalil    9 days ago  feat: add chargeback reason code mapping
e1c0983  Omar Khalil   12 days ago  feat: chargeback handler — initial implementation
```

> Custom format `%h %an %ar %s`: short hash, author name, relative age, subject. Immediately shows Yasmine's fix landed 5 days before the report — investigate that one first.

```bash
# Step 4: Inspect the suspicious commit's full diff
$ git show d4b7f2a
commit d4b7f2a31c84a9f2e7d501bc3812fd4a8e291c09
Author: Yasmine Morsi <yasmine@intervision.io>
Date:   Mon Jun 9 14:22:07 2026 +0200

    fix: prevent duplicate chargeback on 402 response

    Stripe returns 402 for declined cards. Previously the handler
    retried the chargeback creation, resulting in two chargeback
    records for the same PaymentIntentId.

diff --git a/src/PaymentGateway/ChargebackHandler.cs b/src/PaymentGateway/ChargebackHandler.cs
index 3c84a12..8f2d91a 100644
--- a/src/PaymentGateway/ChargebackHandler.cs
+++ b/src/PaymentGateway/ChargebackHandler.cs
@@ -47,6 +47,11 @@ public async Task HandleAsync(ChargebackEvent evt, CancellationToken ct)
+        var existing = await _repo.FindByPaymentIntentAsync(evt.PaymentIntentId, ct);
+        if (existing is not null)
+        {
+            _logger.LogWarning("Duplicate chargeback suppressed for {PaymentIntentId}", evt.PaymentIntentId);
+            return;
+        }
+
         var chargeback = Chargeback.Create(evt);
```

> `git show <hash>` gives the full commit metadata + diff. This is where you confirm whether the logic change matches the bug report.

```bash
# Step 5: Use pickaxe to find every commit that ever added or removed the dedup check
$ git log -S "FindByPaymentIntentAsync" --oneline -- src/PaymentGateway/ChargebackHandler.cs
d4b7f2a fix: prevent duplicate chargeback on 402 response
```

> `-S` is the pickaxe: finds commits where the string count changed (i.e., it was added or removed). Only one commit touched this call — confirming d4b7f2a introduced the dedup logic, and nothing removed it afterward.

```bash
# Step 6: Compare what's in the release branch vs main (did this fix make it to prod?)
$ git log --oneline main..release/v2.4
(no output)
```

> No output means `release/v2.4` has no commits that `main` doesn't — the branch is fully merged. If there had been output, those commits would be "in main but not in release" — a deployment gap.

### What Could Go Wrong Mid-Walkthrough

**If the file was renamed mid-history, the path filter stops at the rename:**

```bash
# Wrong — stops at the rename boundary
$ git log --oneline -- src/PaymentGateway/ChargebackHandler.cs

# Correct — follows the file through renames
$ git log --follow --oneline -- src/PaymentGateway/ChargebackHandler.cs
a1b2c3d refactor: move ChargebackHandler to PaymentGateway subfolder
e4f5g6h feat: chargeback handler first pass
```

> `--follow` only works for a single file path (not a directory), but it's essential when files have been moved.

## Gotchas and Troubleshooting

### `git log -- <path>` returns nothing but the file exists on disk

**Root cause:** The path doesn't match Git's internal path (case-sensitive on Linux hosts, case-insensitive on macOS). Or the file was added in the current working tree but hasn't been committed yet. Or you forgot the `--` separator and Git is treating the path as a ref name.

```bash
# Wrong — Git may interpret "src/OrderService" as a branch name
$ git log src/OrderService/RefundHandler.cs

# Correct — the double-dash separates refs from paths
$ git log -- src/OrderService/RefundHandler.cs
```

**Fix:** Always use `--` before paths. Verify the exact case with `git ls-files | grep -i refund`.

**Prevention:** Add to your `.gitconfig` aliases so the correct form is the default — see Section 6.

---

### `git log -S` returns nothing even though the string is in the file

**Root cause:** `-S` finds commits where the _count_ of that string changed. If the string appears twice and a commit added one and removed one (net zero change), `-S` won't find it. Alternatively, `-S` is case-sensitive by default.

```bash
# If case sensitivity is the issue, use -i with -G (regex variant)
$ git log -G "maxretryattempts" -i --oneline
9a4e812 feat: add exponential backoff to charge retry handler
```

**Fix:** Switch from `-S` (literal, count-based) to `-G` (regex, line-based) when the pattern might span case variations or when net-zero count changes are possible.

**Prevention:** Document in your team runbook which pickaxe mode to reach for and why — this is a common source of "git can't find it" confusion at 2 AM.

---

### `fatal: bad default revision 'HEAD'`

**Root cause:** You ran `git log` in a directory that has been `git init`ed but has no commits yet — `HEAD` doesn't point to anything.

```bash
$ git log
fatal: your current branch 'main' does not have any commits yet
```

**Fix:** Make at least one commit first, or confirm you're in the right repository directory.

**Prevention:** `git log` in a brand-new empty repo is never useful — use `git status` first to confirm the repo has history.

---

### `git log --oneline --graph` shows a tangled mess impossible to read

**Root cause:** A repo with many long-lived feature branches and non-linear merge history produces an illegible ASCII graph. Also triggered when `--all` adds remote-tracking refs from dozens of stale branches.

```bash
# Limit graph to the branches you actually care about
$ git log --oneline --graph --decorate main origin/main feature/refund-rework HEAD
* a7c3f12 (HEAD -> main, origin/main) fix: retry logic for Stripe webhook timeout
*   f9c1a34 Merge pull request #1847 from feature/exponential-backoff-retry
|\
| * 9a4e812 (origin/feature/exponential-backoff-retry) feat: add exponential backoff
| * 3c1d7b2 fix: increase retry delay
|/
* c5a9314 chore: update NuGet packages
```

**Fix:** Name only the refs you want instead of `--all`. Or use `git log --first-parent main` to see only the main-line commits without merge branches.

**Prevention:** `--first-parent` is the cleanest view of a trunk-based development history (see [[7.917 — Trunk-Based Development]]). Add it to your log aliases for daily use.

---

### `git log --since` and `--until` use author date, not commit date

**Root cause:** Git has two timestamps per commit: author date (when the commit was originally created) and commit date (when it was last applied, e.g., after a rebase). `--since`/`--until` filter on **author date** by default. A commit rebased into `main` today might have an author date from last month and appear to be missing from a recent-date filter.

```bash
# Symptom: commit from today doesn't appear in --since="1 day ago"
# because it was authored 3 weeks ago and just rebased now

# Fix: add --format to check both dates
$ git log --format="%h  authored: %ad  committed: %cd  %s" \
    --date=short -n 5 -- src/OrderService/
```

**Fix:** Use `--format="%cd"` to inspect commit dates directly. For CI/incident filtering where you want "when did this land," add `--date-order` or switch to searching by commit hash range rather than date.

**Prevention:** After a large rebase, always use commit hash ranges (`<base>..<head>`) rather than date ranges to scope history accurately.

---

### `git log -p` buries the commit you need under megabytes of diff output

**Root cause:** `-p` shows the full patch for every matching commit. On a large file with many historical changes, this is gigabytes of output piped to your terminal.

```bash
# Wrong for investigation — too much output
$ git log -p -- src/OrderService/RefundHandler.cs

# Correct — inspect one commit at a time
$ git log --oneline -- src/OrderService/RefundHandler.cs
# Then for the specific commit:
$ git show a7c3f12 -- src/OrderService/RefundHandler.cs
```

**Fix:** Use `git log --oneline` first to identify the hash, then `git show <hash>` to inspect one commit. Never use `git log -p` without also adding `-n 1` or a tight hash range.

**Prevention:** `-p` belongs in a pipeline to a pager (`| less`) or when scripting — not in interactive investigation.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# Add to ~/.gitconfig under [alias]

# The daily driver — one line per commit, last 20
lg = log --oneline --decorate -n 20

# Graph view across current branch and main
lgg = log --oneline --graph --decorate main HEAD

# Who changed what file, formatted for scanning
lf = log --format="%h  %an  %ar  %s" --

# Pickaxe search shorthand — usage: git ls "SearchString"
ls = log -S --oneline --

# Last N commits with stat (file-change counts)
lst = log --oneline --stat -n 10

# What have I done today?
today = log --oneline --author="$(git config user.email)" --since="midnight"

# What merged into main this week?
week = log --oneline --merges --since="1 week ago" main
```

Apply immediately in any shell session:

```bash
$ git config --global alias.lg "log --oneline --decorate -n 20"
$ git config --global alias.lgg "log --oneline --graph --decorate main HEAD"
$ git config --global alias.lf "log --format='%h  %an  %ar  %s' --"
```

### Automation Opportunity

A daily standup digest script — what landed in `main` since your last standup:

```bash
#!/usr/bin/env bash
# scripts/standup-digest.sh
# Usage: ./standup-digest.sh [hours-back]
# Example: ./standup-digest.sh 24

set -euo pipefail

HOURS="${1:-24}"
BRANCH="${2:-main}"

echo "=== Changes in ${BRANCH} in the last ${HOURS}h ==="
git log \
  --oneline \
  --no-merges \
  --since="${HOURS} hours ago" \
  --format="%C(yellow)%h%Creset  %C(cyan)%an%Creset  %ar  %s" \
  "${BRANCH}"

echo ""
echo "=== Files touched ==="
git log \
  --name-only \
  --no-merges \
  --since="${HOURS} hours ago" \
  --format="" \
  "${BRANCH}" \
  | sort -u | grep -v '^$'
```

```bash
$ chmod +x scripts/standup-digest.sh
$ ./standup-digest.sh 24 main

a7c3f12  Tarek Saleh    3 hours ago    fix: retry logic for Stripe webhook timeout
3c1d7b2  Yasmine Morsi  8 hours ago    fix: increase retry delay for payment provider timeouts

```

### Keyboard / CLI Efficiency Notes

Five habits that separate fast `git log` users from slow ones:

1. **Always start with `--oneline`** — verbose output hides the signal. Expand with `git show <hash>` only on the specific commit you identified.
2. **Use `-- <path>` before `--author` or `--grep`** — path filtering is cheapest (index-assisted); it narrows the set that the more expensive filters operate on.
3. **`git log origin/main..HEAD` before every push** — confirms exactly what you're about to push. Make it a `git config alias.unpushed "log --oneline origin/main..HEAD"`.
4. **`q` to exit the pager** — `git log` pipes through `less` by default. If your terminal freezes, you're in the pager. Press `q`.
5. **`--format` over `--pretty=format`** — they're identical but `--format` is shorter to type.

### Spoken Answers

**Q: Walk me through how you'd find the commit that introduced a specific bug using only `git log`.**

> **Average answer:** "I'd use `git log` with `--grep` to search commit messages for something related to the bug, then look at the diff."

> **Great answer:** "My first filter is always the file path — `git log --oneline -- src/OrderService/RefundHandler.cs` — because that's cheapest and immediately eliminates noise. From there I add a date window if I know roughly when the bug appeared: `--since='2 weeks ago'`. I scan the resulting list and pick the most suspicious commit by message. If nothing obvious stands out, I switch to pickaxe search: `git log -S 'RefundCalculator' --oneline` — that finds every commit where that string was added or removed, regardless of message wording. Then `git show <hash>` on the candidate gives me the full diff. If I still can't tell, I hand it off to `git bisect` which turns it into a binary search. What I don't do is scroll through hundreds of commits manually — that's what filters are for."

---

**Q: `git log -S "SomeMethodName"` returns nothing even though that method exists in the file. What would you check?**

> **Average answer:** "Maybe the string doesn't match exactly — I'd double check the spelling."

> **Great answer:** "Three things in order. First, am I looking at the right file scope? By default `-S` searches the entire repo. If I forgot `-- src/` I might be getting false negatives due to the result set being too large or the method not being 'new' in that context. Second, `-S` counts occurrences — if a commit added and removed the same method name in the same change, the net count is zero and `-S` won't catch it. In that case I'd switch to `-G 'SomeMethodName'` which does a regex line-by-line match instead of a count. Third, case sensitivity — `-S` is case-sensitive by default, so `someMethodName` won't match `SomeMethodName`. I'd check with `git log -G 'somemethodname' -i --oneline` to rule that out."

---

**Q: Tell me about a time you used `git log` to diagnose a production incident.**

> **Average answer:** "We had a bug and I used git log to find when it was introduced by looking at recent commits."

> **Great answer (war story to adapt):** "We had a payment double-charge incident on a Friday evening. The first thing I did was run `git log --oneline --since='3 days ago' --no-merges -- src/PaymentGateway/ChargebackHandler.cs`. Three commits came back. One of them — from Wednesday — was titled 'fix: prevent duplicate chargeback on 402 response' which sounded exactly like the area. I ran `git show d4b7f2a` and saw it had added a dedup check using `FindByPaymentIntentAsync`. I then ran `git log -S 'FindByPaymentIntentAsync' --oneline` to confirm no subsequent commit had removed that check — it hadn't. So the logic was still there. That told me the bug wasn't a missing check, it was a check that wasn't being reached. The commit had also added a `LogWarning` — I went to Application Insights and searched for that exact log message. It wasn't appearing. That told me the chargeback events weren't even hitting that code path. Forty minutes later we found the real cause: a message bus subscription had been accidentally removed. But `git log` ruled out the code change angle in about five minutes, which saved us from going down the wrong rabbit hole."

### War Story Prompt

During a post-deploy incident, `OrderService`'s payment processing started rejecting valid cards with a cryptic `NullReferenceException` in `PaymentContextBuilder`. The on-call engineer assumed it was an infrastructure issue and spent 90 minutes restarting pods. A second engineer ran `git log --oneline --since="6 hours ago" -- src/OrderService/Payment/` — three commits appeared, the most recent of which added a required `PaymentContext.Currency` field that the caller wasn't passing. `git show` confirmed it in 30 seconds. Total time from "run the git log command" to "found the cause" was under four minutes. The first engineer hadn't been taught to filter by path and time window — they were scrolling the full log manually.

## Decision Framework

### When to Reach for This

- [ ] You need to find what changed in a specific service or file in the last sprint / last incident window
- [ ] You need to know who authored a set of commits (for a PR review, blame triage, or standup)
- [ ] You need to confirm whether a fix is present in the deployed ref
- [ ] You need to produce a change summary for an incident timeline
- [ ] You know a string that should or shouldn't be in the codebase and want to find when it appeared
- [ ] You want to compare what's in one branch versus another before a merge or deploy

### When NOT To

- [ ] You want to find _which line_ introduced a bug and bisect to the exact commit — reach for `git bisect` ([[9.030 — Git Bisect — Finding the Commit That Broke Things]])
- [ ] You want to see who last changed a _specific line_ — reach for `git blame` ([[9.017 — Git Show and Blame — Inspecting History]])
- [ ] You need to _recover_ a commit that's no longer reachable from any ref — reach for `git reflog` ([[9.031 — Git Reflog — Recovering Lost Commits]])
- [ ] You want to inspect a _single_ known commit in detail — `git show <hash>` is simpler than `git log -n 1 <hash> -p`

||`git log` with filters|GitHub PR / commit UI|

|Speed|Instant, works offline|Requires network + VPN + browser|

## Self-Check

### Conceptual Questions

1. What does the `--` separator do in `git log -- src/OrderService/`?
2. What is the difference between `git log -S "Foo"` and `git log -G "Foo"`?
3. Which flag shows the files changed and line counts per commit, without showing the full diff?
4. You see `git log A..B`. What does this return — commits in A not in B, or commits in B not in A?
5. What flag would you add to `git log --oneline --graph` to limit the ASCII graph to the main-line commits only, without showing the branches that merged in?
6. `git log --since="1 week ago"` is filtering on which timestamp — author date or commit date?
7. How does `git log --follow -- OldName.cs` differ from `git log -- NewName.cs`, and when does the difference matter?

<details> <summary>Answers</summary>

1. `--` separates ref names from file paths. Without it, Git may try to interpret `src/OrderService/` as a branch or tag name and fail or return wrong results. With `--`, everything after is treated unambiguously as a path.
    
2. `-S "Foo"` finds commits where the _count_ of occurrences of `"Foo"` changed (pickaxe: added or removed). `-G "Foo"` finds commits where _any line_ matching the regex `"Foo"` was added or removed. `-G` catches cases where a refactor swapped one occurrence for another in the same commit (net-zero count change), while `-S` would miss it.
    
3. `--stat`. Output looks like:
    
    ```
    a7c3f12 fix: retry logic
     src/OrderService/Retry/ChargeRetryHandler.cs | 4 ++--
     1 file changed, 2 insertions(+), 2 deletions(-)
    ```
    
4. `git log A..B` returns commits that are in B but **not** in A. Read it as "B minus A" — what B has that A doesn't. Most common use: `git log main..feature/my-branch` to see what your feature branch adds.
    
5. `--first-parent`. This follows only the first parent of each commit, which in a merge-based workflow is always the main-line commit. It hides the merged-in branch history and shows a clean linear view of what landed on `main`.
    
6. Author date. `--since` and `--until` filter on the **author date** — when the commit was originally written. If a commit was rebased recently, its commit date changed but its author date didn't. This is a gotcha in repos that use heavy rebasing; use hash ranges instead of date ranges for accuracy.
    
7. `git log -- NewName.cs` only shows commits where the file was committed under its _current_ name. If it was previously named `OldName.cs` and was renamed with `git mv`, history from before the rename is invisible. `git log --follow -- NewName.cs` (or `--follow -- OldName.cs`) crosses rename boundaries and shows the full history under both names. The difference matters whenever a file has been moved or renamed — common in refactors.
    

</details>

**Drill 1 — Do it from a clean state**

Starting from any non-empty git repository (your own or `git clone https://github.com/dotnet/aspnetcore` for a realistic large repo), do the following without opening GitHub or any browser:

1. Find the 5 most recent commits that touched any file containing "Middleware" in its path.
2. Find which author has the most commits in the last 30 days.
3. Find every commit where the string `"UseRouting"` was added or removed.
4. Show only commits that were merge commits in the last month.

<details> <summary>Solution</summary>

```bash
# 1. Five most recent commits touching any Middleware path
$ git log --oneline -n 5 -- '*Middleware*'
# Note: glob patterns need quoting to prevent shell expansion

# 2. Most commits in the last 30 days — use shortlog
$ git shortlog -sn --since="30 days ago" --no-merges | head -5
    42  Yasmine Morsi

9b3c1a2 feat: add routing middleware to pipeline
```

**Drill 2 — Fix the broken thing**

An engineer tried to find all commits touching the payment service in the last week and got this:

```bash
$ git log --oneline src/PaymentGateway --since="1 week ago"
fatal: ambiguous argument 'src/PaymentGateway': both revision and filename
Use '--' to separate paths from revisions, like this:
'git <command> [<revision>...] -- [<file>...]'
```

<details> <summary>Solution</summary>

**Diagnosis:** Missing `--` separator. Git is trying to interpret `src/PaymentGateway` as both a possible ref name and a path simultaneously, and fails because it's ambiguous.

```bash
# Fix: add -- before the path
$ git log --oneline --since="1 week ago" -- src/PaymentGateway/
a7c3f12 fix: retry logic for Stripe webhook timeout
3c1d7b2 fix: increase retry delay for payment provider timeouts
9a4e812 feat: add exponential backoff to charge retry handler
```

Note the flag order: `--since` comes before `--`, and the path comes after `--`. Both orderings work as long as `--` is present, but putting path filters at the end after `--` is the convention.

</details>

**Drill 3 — Under time pressure**

Production is alerting. `InventoryApi`'s stock reservation endpoint started returning `500` at 14:32 UTC today. You need to find every commit to the reservation logic since the last known-good deploy tag `v3.1.0`, sorted by recency, with author and date visible — in under 60 seconds.

<details> <summary>Solution</summary>

```bash
# The fastest correct sequence

# Step 1: What's changed in the reservation path since v3.1.0?
$ git log --format="%h  %an  %ad  %s" --date=short \
    v3.1.0..HEAD \
    -- src/InventoryApi/Reservation/
# ~5 seconds — no network required, runs against local clone

# Step 2: If the service path is uncertain, widen to the whole service
$ git log --oneline --no-merges v3.1.0..HEAD -- src/InventoryApi/
# Same pattern, broader path

# Step 3: Inspect the most recent commit immediately
$ git show HEAD -- src/InventoryApi/Reservation/
# Or by hash if HEAD is a merge commit:
$ git log --oneline --no-merges v3.1.0..HEAD -- src/InventoryApi/Reservation/ | head -1
# Copy the hash, then:
$ git show <hash>
```

Why each step is safe under pressure:

- `v3.1.0..HEAD` is a definitive range — not a date filter, so rebased commits don't escape it
- `--no-merges` removes noise from merge commits that didn't change code
- `--format` with `%ad --date=short` gives you exact dates without relative time ambiguity
- `git show <hash>` at the end is read-only — zero risk of accidental state change

</details>
