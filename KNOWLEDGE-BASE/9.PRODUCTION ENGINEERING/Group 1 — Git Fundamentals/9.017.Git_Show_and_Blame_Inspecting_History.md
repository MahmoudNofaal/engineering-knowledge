### Where This Fits

You're reviewing a bug report: `OrderService` is calculating refund amounts incorrectly for orders over £500. A colleague pastes a commit hash from the GitHub PR feed. You need to see exactly what changed in that commit — the full diff, the author, the message, and the context — without opening a browser. Or you're on a call and someone asks "who last changed line 47 of `RefundCalculator.cs` and why?" — the GitHub Blame UI is three clicks away and requires VPN; `git blame` answers in under a second from your terminal. Both tools exist for the same purpose: understand the history of a specific thing, not the broad history of everything.

---

## What You're Doing and Why

`git show` inspects a single Git object — most commonly a commit, but also a tag, a tree, or a blob — and prints its full contents: metadata (author, date, message) plus the complete diff of every file it touched. It is the "zoom in" complement to `git log`'s "zoom out." `git blame` annotates every line of a file with the commit hash, author, and date that last touched it, letting you answer "who wrote this and when" without leaving the file context.

Together they form a two-step pattern that covers the majority of "why does this code look like this" investigations: `git blame` to identify the line and commit, then `git show <hash>` to understand what that commit was doing and whether the change was deliberate.

### Mental Model

`git log` is the table of contents. `git show` is opening to a specific page and reading it fully. `git blame` is reading the current page and seeing who wrote each paragraph — it tells you the author but not the full context of what they were thinking. You always follow `git blame` with `git show` to get that context.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git show <hash>`|Full commit: message + diff of all changed files|Inspecting a specific commit found via `git log`|
|`git show <hash> -- <path>`|Commit diff scoped to one file|Large commits — see only the file you care about|
|`git show <hash>:<path>`|Print the file contents at that commit (no diff)|Recovering or comparing a file at a specific point in time|
|`git show HEAD`|The most recent commit on current branch|Confirming what just got committed|
|`git show HEAD~1`|The commit before the most recent|Quick "what was the last thing before this"|
|`git show HEAD~3..HEAD`|Range: last 3 commits as a combined diff|Summarising a short run of commits|
|`git show v2.3.1`|Annotated tag contents + tagged commit diff|What changed at a release tag|
|`git show --stat <hash>`|Files changed + line counts, no line-level diff|Quick "what did this commit touch" overview|
|`git show --name-only <hash>`|Only filenames touched, no diff at all|Fastest scan of a commit's scope|
|`git show --word-diff <hash>`|Inline word-level diff instead of line-level|Prose changes, config value edits|
|`git blame <file>`|Every line annotated with last-touching commit|Finding who wrote a specific line|
|`git blame -L 40,60 <file>`|Blame for lines 40–60 only|Scoping to the region that's broken|
|`git blame -w <file>`|Ignore whitespace-only changes in blame|Prevents reformatting commits from hiding real authors|
|`git blame -M <file>`|Detect lines moved within the file|Reveals true origin when code was rearranged|
|`git blame -C <file>`|Detect lines copied from other files|Finds origin when code was copied from elsewhere|
|`git blame <hash> -- <file>`|Blame at a specific commit (not HEAD)|Inspecting blame at the time a bug was introduced|
|`git log -L <start>,<end>:<file>`|Full history of a line range, not just last touch|Deep dive: see every change to a region over time|

### What Could Go Wrong Mid-Walkthrough

**If `git blame` points to a reformatting commit, not the real author:**

```bash
# All lines blame to a single "style: apply dotnet format" commit
$ git blame src/PaymentGateway/Fees/FeeCalculator.cs | head -10
f9a3c812 (GitHub Actions  2026-06-10 03:14:22 +0000  1) using YourCompany...
f9a3c812 (GitHub Actions  2026-06-10 03:14:22 +0000  2) using YourCompany...
# Every line blames to the same formatter commit — useless

# Fix: ignore whitespace changes with -w, and detect moves with -M
$ git blame -w -M src/PaymentGateway/Fees/FeeCalculator.cs | head -10
c8f3a12b (Omar Khalil    2026-06-19 10:14:33 +0300  1) using YourCompany...
e4b8f031 (Layla Hassan   2026-04-02 14:55:07 +0300  2) using YourCompany...
# Real authors restored
```

> `-w` ignores whitespace-only changes (indentation shifts, line ending normalisation) and `-M` detects lines that were moved within the file. Always use both in repos with automated formatting.

## Gotchas and Troubleshooting

### `git blame` shows only one author across the entire file — a bot or CI account

**Root cause:** An automated commit (formatter, import sorter, copyright header inserter) touched every line of the file. `git blame` shows the most recent commit per line, so a mass-reformatting commit blankets the whole file.

```bash
# Symptom
$ git blame src/OrderService/RefundHandler.cs | awk '{print $2}' | sort | uniq -c | sort -rn | head -5
    312 (github-actions[bot]
# One author owns 312 of the file's lines — the formatter

# Fix: ignore whitespace changes and detect moved lines
$ git blame -w -M -C src/OrderService/RefundHandler.cs | awk '{print $2}' | sort | uniq -c | sort -rn | head -5
     87 (Yasmine
     63 (Omar
     41 (Tarek
# Real distribution restored
```

**Prevention:** Configure `.gitattributes` to mark `*.cs` as `linguist-generated=false` when using automated formatters. More importantly, establish the team convention that `git blame -w -M` is the default for .NET files. See [[9.018 — .gitattributes — Line Endings and Diff Behavior]].

---

### `git show <hash>` exits immediately with `fatal: bad object <hash>`

**Root cause:** Three common causes: (1) the hash was truncated and isn't unique — Git needs at least 7 characters and sometimes more in large repos; (2) the object exists on the remote but hasn't been fetched; (3) the hash was copied with a leading/trailing space or newline.

```bash
# Symptom
$ git show c8f3a
fatal: bad object c8f3a

# Fix 1: lengthen the hash
$ git show c8f3a12b
# Works — needed 8 chars to be unique in this repo

# Fix 2: fetch first if the hash came from a remote PR
$ git fetch origin
$ git show c8f3a12b
# Now works — object was on remote, not local

# Fix 3: verify there's no invisible whitespace
$ echo "c8f3a12b" | cat -A
c8f3a12b$
# Clean — $ shows the line end, no trailing space
```

**Prevention:** Copy commit hashes from `git log --oneline` output (already short and clean) rather than from web UIs that may include surrounding characters. In scripts, always `git rev-parse` to validate and normalise a hash before using it.

---

### `git blame` points to a merge commit, not the real author

**Root cause:** In repos using squash merges or merge commits, `git blame` can surface the merge commit (or the person who pressed "Merge" on GitHub) rather than the author of the original line.

```bash
# Symptom: blame shows "Merge pull request #1234" as the last touch
$ git blame src/OrderService/RefundHandler.cs -L 47,47
9c1f3d22 (Omar Khalil 2026-06-21 12:00:00 +0000 47)     var refund = amount * _config.RefundRate;
# Commit message: "Merge pull request #1821 from feature/refund-rates"
# Omar didn't write this — he just merged it

# Fix: inspect the merge commit's parents to find the real author
$ git show 9c1f3d22
# Look at the Parents: line in the output header
commit 9c1f3d22...
Merge: 4a1b2c3 7d8e9f0        ← two parent hashes

# The real author is in the second parent (the feature branch)
$ git show 7d8e9f0 --stat
Author: Yasmine Morsi <yasmine@intervision.io>
# Yasmine wrote it
```

**Prevention:** In trunk-based development (see [[7.917 — Trunk-Based Development]]), prefer squash-and-merge — this makes the PR author the blame target for all lines they changed, which is usually the correct attribution.

---

### `git log -L` hangs or takes minutes on a large file

**Root cause:** `git log -L` follows a line range through every rename and rewrite in history, which on a file with thousands of commits (e.g., a large generated file or a long-lived domain model) can walk the entire DAG.

```bash
# Symptom: command appears to hang
$ git log -L 100,120:src/OrderService/Order.cs
# (no output for 30+ seconds)
```

**Fix:**

```bash
# Add --no-patch to see just the commit messages first (much faster)
$ git log -L 100,120:src/OrderService/Order.cs --oneline --no-patch
c8f3a12 feat: add PREMIUM tier fee multiplier
b3c1d9f2 refactor: simplify reservation flow
# Now inspect individual commits with git show

# Or limit to recent history
$ git log -L 100,120:src/OrderService/Order.cs --since="6 months ago" --oneline
```

**Prevention:** Use `git blame -L` for a quick last-touch answer. Reserve `git log -L` for when you specifically need the full evolutionary history of a region and are prepared to wait.

---

### `git show HEAD~3` gives the wrong commit — unexpected output

**Root cause:** `~N` follows the **first parent** chain. In a repo with merge commits, `HEAD~3` may skip over branch commits entirely and land on a commit from `main` that looks unrelated to what you expected.

```bash
# In a repo with merges, HEAD~3 might be:
$ git log --oneline -5
a7c3f12 (HEAD -> main) fix: retry logic for Stripe webhook timeout  ← HEAD
9c1f3d22 Merge pull request #1821 from feature/refund-rates         ← HEAD~1
4a1b2c3d chore: update packages                                      ← HEAD~2 (first parent of merge)
b3c1d9f2 refactor: simplify reservation flow                         ← HEAD~3
# feature/refund-rates' commits are not in the HEAD~N chain

# Fix: use the actual commit hash when precision matters
$ git show b3c1d9f2

# Or use ^2 to access the second parent of a merge
$ git show 9c1f3d22^2
# Shows the tip of feature/refund-rates at merge time
```

**Prevention:** Use `git log --oneline --graph -10` to visualise the parent structure before using `HEAD~N` notation. For precision work, always copy the explicit hash.

---

### `git show <hash>:<path>` outputs binary content to the terminal

**Root cause:** The file at that commit is binary (an image, a compiled asset, a certificate). Git shows raw binary bytes, which corrupts terminal output.

```bash
# Symptom: terminal fills with garbage characters and beeps
$ git show c8f3a12:wwwroot/images/logo.png
# Garbage output, terminal may need reset

# Fix: redirect to a file
$ git show c8f3a12:wwwroot/images/logo.png > /tmp/logo-old.png
# Now compare with current version
$ diff /tmp/logo-old.png wwwroot/images/logo.png

# Reset a corrupted terminal
$ reset
```

**Prevention:** Before running `git show <hash>:<path>`, check whether the file is binary: `git show --stat <hash>` will show `Bin <N> -> <M> bytes` for binary files, letting you redirect before the raw output hits the terminal.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# Add to ~/.gitconfig under [alias]

# Show a commit with stats first, then diff — easier to orient
show-stat = show --stat

# Show only the files changed in a commit (no diff content)
show-files = show --name-only --format=""

# Blame with whitespace and move detection — should be your default
bl = blame -w -M

# Blame a specific line range: usage: git bll 40 60 src/File.cs
bll = "!f() { git blame -w -M -L \"$1,$2\" \"$3\"; }; f"

# Line history — usage: git lh 40 60 src/File.cs
lh = "!f() { git log -L \"$1,$2:$3\" --oneline; }; f"

# Show what HEAD just committed (the most common git show use case)
last = show HEAD --stat

# Print a file as it was at a given commit: usage: git cat-at abc1234 src/File.cs
cat-at = "!f() { git show \"$1:$2\"; }; f"
```

Apply the most-used ones immediately:

```bash
$ git config --global alias.bl "blame -w -M"
$ git config --global alias.last "show HEAD --stat"
$ git config --global alias.show-files "show --name-only --format=''"
```

### Automation Opportunity

A blame-to-PR lookup script — takes a file and line number, runs blame, extracts the commit hash, then uses `gh` CLI to find the PR it came from:

```bash
#!/usr/bin/env bash
# scripts/blame-pr.sh
# Usage: ./blame-pr.sh src/OrderService/RefundHandler.cs 47
# Prints: the commit hash, author, date, and PR link for line N

set -euo pipefail

FILE="${1:?Usage: blame-pr.sh <file> <line>}"
LINE="${2:?Usage: blame-pr.sh <file> <line>}"

HASH=$(git blame -w -M -L "${LINE},${LINE}" "${FILE}" | awk '{print $1}')
FULL_HASH=$(git rev-parse "${HASH}")

echo "=== Line ${LINE} of ${FILE} ==="
git show "${FULL_HASH}" --format="%h  %an  %ai%n%s%n%b" --no-patch

echo ""
```

### Keyboard / CLI Efficiency Notes

Five habits that make `git show` and `git blame` fast in practice:

1. **`git show HEAD --stat` before any other show** — the stat view loads instantly and tells you whether the full diff is worth reading (1 file changed vs 47 files changed).
2. **`git blame -L <start>,<end>`** — never blame a 2,000-line file and scroll. Know the approximate line number first (from the stack trace, from the IDE, from the error message) and scope immediately.
3. **`-w -M` on every blame** — formatter commits are everywhere in CI-heavy repos. These two flags are almost always the right default. Add the `bl` alias above so you never forget.
4. **`git show <hash>:<path> | code -`** — pipe the file content into VS Code (or Rider) for syntax-highlighted reading: `git show c8f3a12:src/PaymentGateway/Fees/FeeCalculator.cs | code -`. Works for any editor that accepts stdin.
5. **`q` to exit** — `git show` and `git blame` pipe through `less`. If output stops unexpectedly, press `q`. If the terminal looks corrupted after a binary show, run `reset`.

### Spoken Answers

**Q: Walk me through how you'd use `git blame` and `git show` together to investigate a bug in production.**

> **Average answer:** "I'd run `git blame` on the file and find who last changed the broken line, then look at their commit."

> **Great answer:** "It's a two-step pattern. First, I need to know which lines are relevant — either from the stack trace, the test output, or knowing the function name. I don't blame the whole file; I scope to the region with `git blame -L <start>,<end> <file>` and add `-w -M` to see through formatter commits and code moves. That gives me the commit hash for each line in the region. I then take the most suspicious hash and run `git show <hash>` — not just to see the diff, but to read the commit message, which often tells me whether the change was intentional and what it was supposed to accomplish. If the message says 'moved validation to domain model' but I can see the validation isn't in the domain model either, that's my root cause. The blame told me where; the show told me what and why. If I still need the full evolutionary history of that region — not just the last touch — I use `git log -L <start>,<end>:<file>` which shows every commit that ever modified those lines."

**Q: `git blame` is pointing to a formatter bot as the author of every line in a file. How do you find the real authors?**

> **Average answer:** "I'd look at older commits to find the real author."

> **Great answer:** "Two flags solve this immediately: `-w` ignores whitespace-only changes — which is what formatters almost exclusively produce — and `-M` detects lines that were moved within the file. So `git blame -w -M <file>` skips straight through the formatter commit to the commit that last changed the actual content. If the code was copied from another file rather than just moved within the same file, I'd add `-C` which detects cross-file copies. The combination `git blame -w -M -C` is the most thorough option. For repos with heavy automated formatting, I add a `bl` alias for `blame -w -M` so that's always my default. It's also worth running `git log --oneline <file>` and scrolling past the formatter commits to find the substantive changes — that gives you the candidate hashes to inspect with `git show`."

### War Story Prompt

During a post-release review of `PaymentGateway`, a QA engineer flagged that refund amounts for multi-currency orders were off by exactly 2%. The team had no idea when this started. ## Decision Framework

### When to Reach for This

- [ ] You have a commit hash (from Slack, from `git log`, from a PR link) and need to understand what it changed and why
- [ ] You need to know who last modified a specific line or region of a file
- [ ] You want to see what a file looked like at a specific point in time without checking out the whole repo
- [ ] You need to recover the exact content of a deleted or overwritten function
- [ ] You're reviewing a PR and want the full context of a dependency commit the PR builds on
- [ ] You want to understand the full evolutionary history of a function or region (reach for `git log -L`)

### When NOT To

- [ ] You need to find the commit that introduced a bug across many commits — reach for `git bisect` ([[9.030 — Git Bisect — Finding the Commit That Broke Things]]), which automates the search
- [ ] You need to find every commit that ever touched a string or symbol — reach for `git log -S` / `git log -G` ([[9.004 — Git Log — Filtering and Formatting History]]), which searches content rather than annotating lines
- [ ] You need to see who last touched a line AND the full commit context in one step — reach for `git log -L`, which is slower but gives both together
- [ ] You want to undo what `git show` revealed — reach for `git revert` or `git restore` ([[9.013 — Undoing Changes — Checkout, Restore, Revert]])

### Tool Comparison

||`git blame`|`git log -L`|`git log -S`|
|---|---|---|---|
|What it answers|Who last touched each line|Every commit that ever touched a line range|Every commit that added/removed a string|
|Speed|Fast|Slow on deep history|Medium|
|Output|Line-by-line annotation|Full commit list with diffs|Commit list|
|Best for|"Who wrote this?"|"How did this evolve?"|"When did this string appear?"|
|Follow-up tool|`git show <hash>`|Already includes diffs|`git show <hash>`|

## Self-Check

### Conceptual Questions

1. What is the difference between `git show abc1234 -- src/File.cs` and `git show abc1234:src/File.cs`?
2. `git blame` shows that line 47 was last touched by a merge commit. Who actually wrote that line, and how do you find out?
3. What do the flags `-w`, `-M`, and `-C` do in `git blame`, and why would you use all three together?
4. How do you use `git show` to recover the content of a file as it existed two commits ago, without checking out that commit?
5. What is `git log -L` and how does it differ from `git blame -L`?
6. What does `HEAD~3` mean and what's the risk of using it in a repo with merge commits?
7. How would you find out which PR a specific commit was part of, using only CLI tools?

<details> <summary>Answers</summary>

1. `git show abc1234 -- src/File.cs` shows the **diff** for `src/File.cs` as it was changed in commit `abc1234` — what was added and removed, in unified diff format, plus the commit metadata. `git show abc1234:src/File.cs` (colon, not `--`) prints the **full file contents** as they existed at that commit — no diff, just the raw file. The `--` form is for reviewing what changed; the `:` form is for reading or recovering the file at a point in time.
    
2. The merge commit is just the merge action — the real author is in one of the parent commits. Run `git show <merge-hash>` and look for the `Merge: <hash1> <hash2>` line. The second parent (`<hash2>`) is typically the branch being merged in. Run `git show <hash2>` to find the actual author and the real diff. Alternatively, `git log <merge-hash>^2 -1` shows the tip of the merged branch.
    
3. `-w` ignores whitespace-only changes — lines that differ only in spaces, tabs, or indentation are treated as the same line, so formatter commits don't claim authorship. `-M` detects lines that were moved within the same file — if a function was cut and pasted to a different location in the same file, `-M` credits the original author rather than the person who moved it. `-C` detects lines copied from other files — if a block was copy-pasted from another `.cs` file, `-C` finds the original source. Together they produce the most accurate attribution in codebases with heavy automated formatting and refactoring.
    
4. `git show HEAD~2:<path>` — substituting the actual relative ref or commit hash. Example: `git show HEAD~2:src/OrderService/RefundHandler.cs`. This prints the file as it existed two commits before the current HEAD, directly to stdout. Redirect to a file (`> /tmp/RefundHandler-old.cs`) to save it. No checkout needed, no branch switching.
    
5. `git log -L <start>,<end>:<file>` shows every commit in history that ever touched the specified line range, with the full diff for each — the complete evolutionary history of a region. `git blame -L <start>,<end> <file>` shows only the **most recent** commit that touched each line in that range — a snapshot of the current state of authorship. `git log -L` is slower (walks all history) but reveals how the code reached its current state step by step. `git blame -L` is fast and answers "who wrote this" but not "how did it get here."
    
6. `HEAD~3` means the commit that is three hops up the **first-parent chain** from HEAD. In a repo with merge commits, the first-parent chain follows only the mainline (the commit that `main` was on before each merge) — it does not traverse into merged feature branches. So `HEAD~3` might skip over several feature branch commits and land on a commit from `main` that looks unrelated to the feature you're investigating. In repos with merge commits, always check `git log --oneline --graph -10` first to understand the parent structure, then use explicit commit hashes for precision.
    
7. Two approaches. With the `gh` CLI: `gh pr list --search "<hash>" --state merged --json number,title,url`. Without `gh`: `git log --merges --ancestry-path <hash>..HEAD --oneline | tail -1` finds the merge commit that brought this hash into the mainline; from the merge commit message you can extract the PR number (GitHub formats them as "Merge pull request #N from ..."). The `blame-pr.sh` script in Section 6 automates both steps.
    

</details>

### Hands-On Drills

**Drill 1 — Do it from a clean state**

In any git repo with at least 10 commits, do the following without using a browser:

1. Find the commit that last modified line 1 of any `.cs` file in the repo.
2. Use `git show` to read that commit's full message and diff.
3. Print the contents of that file as it existed at that commit.
4. Find how many total files that commit touched.

<details> <summary>Solution</summary>

```bash
# Pick any .cs file — use git ls-files to find one
$ CS_FILE=$(git ls-files "*.cs" | head -1)
$ echo "Using: ${CS_FILE}"

# Step 1: Find the commit that last touched line 1
$ HASH=$(git blame -L 1,1 "${CS_FILE}" | awk '{print $1}')
$ echo "Commit hash: ${HASH}"

# Step 2: Full commit — message and diff
$ git show "${HASH}"
# Read the full output: Author, Date, message, diff of all changed files

# Step 3: Print the file as it was at that commit
$ git show "${HASH}:${CS_FILE}"
# Full file contents — redirect to /tmp/old-file.cs if you want to save it

# Step 4: How many files did that commit touch?
$ git show "${HASH}" --stat | tail -1
# Output: "N files changed, M insertions(+), K deletions(-)"
# The N is your answer
```

</details>

**Drill 3 — Under time pressure**

Production alert: `OrderService`'s `ProcessRefund` method is returning wrong amounts. A stack trace in Sentry shows the error originates from `src/OrderService/Refunds/RefundProcessor.cs`, line 73. You have 3 minutes to identify the commit responsible, the author, the full diff, and what the line looked like before the change.

<details> <summary>Solution</summary>

```bash
# 60 seconds: blame the specific line
$ git blame -w -M -L 70,76 src/OrderService/Refunds/RefundProcessor.cs
# Read the hash for line 73 — e.g., c8f3a12b

HASH="c8f3a12b"   # substitute real hash from blame output

# 30 seconds: commit summary (stat first — orient before reading the full diff)
$ git show "${HASH}" --stat
# See: N files changed — if only 1-2 files, the full diff will be fast to read

# 60 seconds: full commit — read the message and the diff
$ git show "${HASH}"
# Read: author, date, message (was this intentional?), and the +/- lines for line 73

# 30 seconds: what did line 73 look like BEFORE this commit?
$ git show "${HASH}~1:src/OrderService/Refunds/RefundProcessor.cs" | sed -n '70,76p'
# Prints lines 70-76 as they were before the change — direct comparison

# Total elapsed: ~3 minutes
# Output: hash, author, date, commit intent (from message), before/after for line 73
# Decision: deliberate change gone wrong → fix forward; accidental deletion → restore from git show output
```

Why each step is safe under pressure:

- `git blame -w -M` is read-only and returns in milliseconds
- `git show --stat` orients you before committing to reading a potentially huge diff
- `git show <hash>~1:<path> | sed` is read-only and gives you the restoration text directly
- None of these commands modify state — zero risk of making the incident worse

</details>
