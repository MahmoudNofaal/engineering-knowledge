### Where This Fits

You've finished work on a feature branch — `feature/payment-retry` — and you're ready to bring it back into `main`. You type `git merge` and either Git silently advances a pointer or it creates a new merge commit with two parents. These two outcomes look identical in the file system but have very different histories. Knowing which one happened — and which one you _wanted_ — is the skill this note covers.

---

## What You're Doing and Why

`git merge` integrates one branch into another. The command is simple; what matters is the _strategy_ Git chooses, because that determines the shape of the commit history your teammates will read, your CI pipeline will trace, and your future self will bisect through at 2 AM.

The two strategies are **fast-forward** (Git just slides the target branch pointer forward — no merge commit created) and **three-way merge** (Git creates a new commit with two parents, permanently recording that two lines of development converged). Fast-forward gives you a clean, linear history; three-way gives you an accurate, branched history. Which one Git picks depends entirely on whether the target branch has moved since you branched off it.

> For the policy question of _when_ to prefer each — see [[7.917 — Trunk-Based Development]] and [[9.027 — Rebase vs Merge — Choosing the Workflow]]. This note covers the mechanics: what happens on disk, how to force or prevent each mode, and how to read the output to know which one fired.

### Mental Model

Think of your branches as two hikers on a trail. **Fast-forward:** hiker A stopped to rest while hiker B kept walking. When A is ready to "merge," you just move A's marker to where B already is — no fork in the trail ever existed. **Three-way merge:** both hikers kept walking in different directions from a junction. To merge them, you need to build a new meeting point that has both trails leading into it. That junction is the merge commit.

Git's "three-way" name comes from the algorithm: it looks at three snapshots — the common ancestor (the junction), the tip of your branch, and the tip of the target branch — and combines the differences.

```
Fast-Forward (main hasn't moved since branch point):

Before:            After:
main → A           main (and feature) → A → B → C
         \
          B → C  ← feature

Three-Way (main moved after branch point):

Before:                      After:
main → A → D → E             main → A → D → E → M (merge commit)
         \                                  /
          B → C  ← feature        B → C --
```

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git merge <branch>`|Merge `<branch>` into current branch; auto-picks strategy|Standard daily merge|
|`git merge --no-ff <branch>`|Force a merge commit even if fast-forward is possible|When you want branch history preserved regardless|
|`git merge --ff-only <branch>`|Abort if a fast-forward is not possible|When you only want clean linear integration|
|`git merge --squash <branch>`|Collapse all branch commits into staged changes (no merge commit yet)|Before a single-commit integration onto `main`|
|`git merge --abort`|Cancel an in-progress merge with conflicts|When you want to bail out and start over|
|`git log --oneline --graph`|Visualize whether a merge commit was created|After every merge to confirm the shape of history|
|`git merge --no-commit --no-ff`|Stage the merge but don't commit it yet|When you want to inspect or modify the result before committing|

## Step-by-Step Walkthrough

### The Scenario

You're working on the `InterVision` repo. You cut `feature/payment-retry` from `main` at commit `a1b2c3d`, added two commits, and `main` has not moved since. A straightforward merge should fast-forward. Then you'll also see what happens when `main` _has_ moved — and how to force the merge commit form when your team requires it for traceability.

### Full Walkthrough

**Part A — Fast-Forward Merge**

```bash
$ git log --oneline --graph main feature/payment-retry
* 9f3e21c (feature/payment-retry) fix: retry on transient 503 from payment provider
* 4c8a17b feat: add configurable retry count to PaymentGateway client
* a1b2c3d (HEAD -> main, origin/main) chore: bump PaymentGateway SDK to 3.2.1

# main and feature/payment-retry share a1b2c3d as their common point
# main has not moved — this is the fast-forward precondition
```

```bash
$ git checkout main

Your branch is up to date with 'origin/main'.

$ git merge feature/payment-retry
Updating a1b2c3d..9f3e21c
Fast-forward
 src/PaymentGateway/RetryHandler.cs | 41 +++++++++++++++++++++++++++++++++++++
 src/PaymentGateway/Options.cs      |  6 ++++++
 2 files changed, 47 insertions(+)
 create mode 100644 src/PaymentGateway/RetryHandler.cs
```

Note `Fast-forward` in the output — Git confirmed the strategy used. No merge commit was created.

$ git log --oneline --graph
* 9f3e21c (HEAD -> main, feature/payment-retry) fix: retry on transient 503 from payment provider
* 4c8a17b feat: add configurable retry count to PaymentGateway client
* a1b2c3d (origin/main) chore: bump PaymentGateway SDK to 3.2.1

**Part B — Three-Way Merge (main has moved)**

```bash
$ git log --oneline --graph main feature/order-validation
* d7f91aa (main) fix: correct timezone handling in OrderService scheduler
* e2c5088 (feature/order-validation) feat: validate order total before submission
* b3a9f12 refactor: extract OrderValidator into separate class
* a1b2c3d chore: bump PaymentGateway SDK to 3.2.1
```

`main` moved to `d7f91aa` after `feature/order-validation` was cut from `a1b2c3d`. Fast-forward is impossible — a three-way merge will fire.

```bash
$ git checkout main
Already on 'main'

$ git merge feature/order-validation
Merge made by the 'ort' strategy.
 src/OrderService/OrderValidator.cs         | 58 ++++++++++++++++++++++++++++++
 src/OrderService/OrderSubmissionHandler.cs |  9 ++++-
 2 files changed, 67 insertions(+), 0 deletions(-)
 create mode 100644 src/OrderService/OrderValidator.cs
```

Note `Merge made by the 'ort' strategy` — that's the three-way merge. An editor may open for the merge commit message; the default is `Merge branch 'feature/order-validation'`.

```bash
$ git log --oneline --graph
*   c4d8e3f (HEAD -> main) Merge branch 'feature/order-validation'
|\
| * e2c5088 (feature/order-validation) feat: validate order total before submission
| * b3a9f12 refactor: extract OrderValidator into separate class
* | d7f91aa fix: correct timezone handling in OrderService scheduler
|/
* a1b2c3d chore: bump PaymentGateway SDK to 3.2.1
```

`c4d8e3f` is the merge commit — it has two parents, permanently recording the convergence.

**Part C — Forcing --no-ff When fast-forward Would Fire**

Some teams require merge commits for every feature branch — even when fast-forward is possible — for traceability.

```bash
$ git checkout main
$ git merge --no-ff feature/payment-retry
```

An editor opens for the commit message (or accept the default):

```
Merge branch 'feature/payment-retry'
# Add any notes about what this branch contained
```

```bash
[main 7a2b9c1] Merge branch 'feature/payment-retry'
```

```bash
$ git log --oneline --graph
*   7a2b9c1 (HEAD -> main) Merge branch 'feature/payment-retry'
|\
| * 9f3e21c fix: retry on transient 503 from payment provider
| * 4c8a17b feat: add configurable retry count to PaymentGateway client
|/
* a1b2c3d chore: bump PaymentGateway SDK to 3.2.1
```

Now the branch history is preserved in the graph even though a fast-forward was technically possible.

### What Could Go Wrong Mid-Walkthrough

**If you see a conflict during a three-way merge:**

```bash
$ git merge feature/order-validation
Auto-merging src/OrderService/OrderSubmissionHandler.cs
CONFLICT (content): Merge conflict in src/OrderService/OrderSubmissionHandler.cs
Automatic merge failed; fix conflicts and then commit the result.
```

At this point you're in a mid-merge state. Do not run any new `git merge` commands. Resolve the conflict, then:

```bash
$ git add src/OrderService/OrderSubmissionHandler.cs
$ git commit
# editor opens with pre-filled merge commit message — save and close
```

See [[9.007 — Resolving Merge Conflicts — Manual Workflow]] for the full step-by-step conflict resolution procedure.

To abort and return to the pre-merge state instead:

```bash
$ git merge --abort
```

## Production Scenario

### Realistic Context

The `InterVision` platform runs an `OrderService` and a `NotificationWorker`. Two engineers worked in parallel over a sprint: one on `feature/order-validation` (three commits, touches `OrderService` only) and one on `hotfix/notification-retry-deadlock` (one commit, touches `NotificationWorker` only). Both branches need to land on `main` before the 18:00 release window. The release engineer needs to merge both, confirm history looks correct for the post-deploy audit trail, and push — without creating a tangled graph that makes `git bisect` useless if something goes wrong in production.

### Full Implementation / Resolution

$ git fetch origin
$ git log --oneline --graph origin/main feature/order-validation hotfix/notification-retry-deadlock

## Gotchas and Troubleshooting

### "Already up to date."

**Root cause:** The branch you're merging is already fully contained in the current branch — every commit on the source branch is an ancestor of HEAD. Nothing was merged because nothing needed to be.

```bash
$ git merge feature/payment-retry
Already up to date.
```

**Fix:** Confirm you're merging the right branch and that you haven't already merged it previously:

```bash
$ git branch --merged main   # lists branches already fully merged into main
  feature/payment-retry      # ← it's here, meaning it was already merged
```

**Prevention:** Delete merged branches immediately after merging — `git branch -d feature/payment-retry` — so they don't appear as merge candidates later.

---

### "fatal: refusing to merge unrelated histories"

**Root cause:** The two branches have no common ancestor commit at all. This happens when you `git init` a new repo locally, make a commit, then try to merge from a remote that was initialised separately (e.g., GitHub created it with a README).

```bash
$ git merge origin/main
fatal: refusing to merge unrelated histories
```

**Fix:** Only use this flag when you're certain the two histories should be combined (e.g., initial project setup):

```bash
$ git merge origin/main --allow-unrelated-histories
```

**Prevention:** When creating a new repo, either initialize it empty on GitHub (no README/LICENSE/gitignore) and push your first commit, or clone it before adding any local commits.

---

### Merge created a commit you didn't want — fast-forward was expected

**Root cause:** `main` had commits you weren't aware of (a colleague pushed while you were working), triggering a three-way merge instead of the expected fast-forward.

```bash
$ git merge feature/payment-retry
Merge made by the 'ort' strategy.
# You expected "Fast-forward" but got a merge commit
```

**Fix — undo the merge commit (before pushing):**

```bash
$ git reset --hard HEAD~1   # removes the merge commit, returns both branches to pre-merge state
```

Then rebase your feature branch onto the updated `main` before merging:

```bash
$ git checkout feature/payment-retry
$ git rebase main
$ git checkout main
$ git merge feature/payment-retry   # now fast-forward fires
```

**Prevention:** Always `git pull --ff-only origin main` immediately before merging a feature branch, so you know `main`'s exact state.

---

### "--ff-only" merge fails: "fatal: Not possible to fast-forward, aborting."

**Root cause:** You used `--ff-only` to guard against accidental merge commits, but the target branch has diverged — fast-forward genuinely is not possible.

```bash
$ git merge --ff-only feature/order-validation
fatal: Not possible to fast-forward, aborting.
```

**Fix — Option A: Rebase the feature branch to make fast-forward possible:**

```bash
$ git checkout feature/order-validation
$ git rebase main
$ git checkout main
$ git merge --ff-only feature/order-validation   # now succeeds
```

**Fix — Option B: Accept the three-way merge:**

```bash
$ git merge feature/order-validation   # drop --ff-only
```

**Prevention:** `--ff-only` is a deliberate guard, not a bug. If your team requires linear history, Option A (rebase then merge) is the correct workflow. See [[9.027 — Rebase vs Merge]] for the full decision tree.

---

### Merge commit has the wrong message or wrong branch listed

**Root cause:** You accepted Git's default merge commit message (`Merge branch 'feature/x'`) without editing it, and the branch name isn't meaningful to reviewers or the audit log.

**Fix — amend the most recent commit (before pushing):**

```bash
$ git commit --amend -m "merge: feature/order-validation — order total validation gate before PaymentGateway submission"
```

**Prevention:** Always use `-m` on merge commands in production release workflows, or configure a merge commit message template via `.gitmessage`.

---

### Merge succeeds locally but CI fails with "merge conflict" on the PR

**Root cause:** You merged `main` _into_ your feature branch to resolve conflicts, but the PR base branch has received additional commits since then. The merge state on GitHub differs from your local state.

**Fix:**

```bash
$ git fetch origin
$ git rebase origin/main   # bring feature branch fully up to date
$ git push --force-with-lease origin feature/order-validation
```

**Prevention:** Prefer `rebase` over `merge` when updating a feature branch against `main` mid-PR. Merging `main` into a feature branch creates nested merge commits that make the PR diff noisy and can confuse GitHub's conflict detection.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# In ~/.gitconfig or via: git config --global alias.<name> '<command>'

# Merge with no-ff and open editor for a clean message every time
[alias]
  mgnff = merge --no-ff

# Merge and immediately confirm what the graph looks like
  mgcheck = "!f() { git merge --no-ff \"$1\" && git log --oneline --graph -8; }; f"

# Safe pull: abort instead of creating a merge commit if origin has diverged
  pullff = pull --ff-only

# List all branches already merged into current branch (safe to delete)
  merged = branch --merged HEAD
```

Usage:

```bash
$ git mgnff feature/payment-retry
$ git mgcheck feature/order-validation
$ git pullff origin main
$ git merged
```

### Automation Opportunity

Add a pre-merge check to your release script so every merge to `main` is explicitly `--no-ff` with a meaningful message, and the graph is printed for visual confirmation before push:

```bash
#!/usr/bin/env bash
# scripts/merge-to-main.sh
# Usage: ./scripts/merge-to-main.sh feature/order-validation "order total validation gate"

set -euo pipefail

BRANCH="${1:?Usage: merge-to-main.sh <branch> <description>}"
DESC="${2:?Usage: merge-to-main.sh <branch> <description>}"

git checkout main
git pull --ff-only origin main

git merge --no-ff "$BRANCH" -m "merge: ${BRANCH} — ${DESC}"

echo ""
echo "=== History after merge (last 10 commits) ==="
git log --oneline --graph -10

echo ""
read -rp "Push to origin/main? [y/N] " confirm
if [[ "$confirm" =~ ^[Yy]$ ]]; then
  git push origin main
  echo "Pushed. CI pipeline: https://github.com/intervision/order-platform/actions"
else
  echo "Push skipped. Run: git push origin main"
fi
```

### Keyboard / CLI Efficiency Notes

- `git log --oneline --graph --all` — the fastest way to see the _entire_ repo branch topology; add `-20` to limit output
- After `git merge --abort`, run `git status` immediately to confirm you're fully back to a clean state — don't assume the abort worked without checking
- `git diff main...feature/order-validation` (three dots) shows only the commits that are on the feature branch but not on `main` — the exact diff that the merge will introduce; useful for a quick sanity check before merging
- `git log main..feature/order-validation` (two dots) lists commits reachable from the feature branch but not from `main` — the commits that will be merged in

### Spoken Answers

**Q: Walk me through what actually happens on disk when Git performs a fast-forward merge.**

> **Average answer:** "Git moves the branch pointer forward. No new commit is created."

> **Great answer:** "Git looks up the common ancestor of the two branches. If the target branch (say `main`) is _itself_ that ancestor — meaning it hasn't moved since the feature branch was created — then a fast-forward is valid. All Git does is update the ref file for `main` (in `.git/refs/heads/main`) to point to the tip commit of the feature branch. No new commit object is written to the object store. The working tree updates to match, but the history is now linear — there's no record in the graph that a branch ever existed. That's the tradeoff: you get a clean history, but you lose the explicit record of when and where work was integrated."

---

**Q: Your `git merge` produced a merge commit but your team expects linear history — how do you fix that before pushing?**

> **Average answer:** "I'd reset and rebase."

> **Great answer:** "First, I confirm I haven't pushed yet — `git log origin/main..HEAD` to see what's local-only. If the merge commit is local, `git reset --hard HEAD~1` removes it and puts both branches back to their pre-merge state. Then I check out the feature branch and `git rebase main` to replay its commits on top of the current `main` tip. Now the feature branch is a straight-line extension of `main`, so the subsequent `git merge --ff-only` will fast-forward cleanly. I'd also add `--ff-only` to our pull/merge scripts so this class of mistake gets caught automatically in future."

---

**Q: A junior engineer ran `git merge main` inside a feature branch to pick up upstream changes. What's the problem?**

> **Average answer:** "It creates an extra merge commit in the feature branch that'll show up in the PR."

> **Great answer:** "The immediate problem is a merge commit in the feature branch history — when that branch is eventually merged into `main`, `git log --graph` shows a backwards-merge braid that's confusing to read and nearly impossible to bisect through cleanly. The deeper problem is that subsequent `--no-ff` merges of that feature branch into `main` will pull those upstream `main` commits in _again_ via the merge parent, creating duplicate-looking history. The fix is to undo the merge commit locally (`git reset --hard HEAD~1`), then `git rebase origin/main` to replay the feature branch commits on top — which achieves the same goal (picking up upstream changes) without the merge commit. The PR diff also becomes clean again."

### War Story Prompt

A release engineer merged twelve feature branches directly into `main` over a Friday afternoon, all with default `git merge` (allowing fast-forwards). On Monday, a 502 spike appeared in production. Running `git log --oneline --graph` showed a perfectly linear history — every branch had fast-forwarded — so there was no way to tell _when_ any particular feature had landed, or whether the spike correlated with a specific branch's code. `git bisect` took 40 minutes and 15 build cycles to isolate the commit because the "merge boundary" information was gone. After that incident, the team added `--no-ff` as a required flag in the release merge script, so every branch integration is permanently marked in the graph — even if the history is technically linear enough to fast-forward.

## Decision Framework

### When to Reach for This

- [ ] You've finished work on a feature/fix/hotfix branch and need to bring it into `main` or a release branch
- [ ] You want to integrate the work of a completed PR locally before the GitHub merge button is available
- [ ] You're doing a local release preparation — merging multiple branches into a staging branch to test together
- [ ] You're the release engineer and need to explicitly record _what_ was merged and _when_ in the commit graph

### When NOT To

- [ ] You want to pick up upstream `main` changes _into_ your still-in-progress feature branch — use `git rebase origin/main` instead (keeps the PR diff clean)
- [ ] You want to bring in a single commit from another branch — use `git cherry-pick` ([[9.029 — Git Cherry-Pick]])
- [ ] You're updating your local `main` from `origin/main` — use `git pull --ff-only`, not `git merge`
- [ ] The team's branch protection policy requires PRs — don't merge locally and push to `main` directly; use the GitHub merge button so required checks are enforced

### Tool Comparison

||Fast-Forward (`git merge`)|Three-Way (`git merge --no-ff`)|Rebase + Fast-Forward|
|---|---|---|---|
|History shape|Linear — no branch ever visible|Branched — merge commit permanently present|Linear — but without a merge commit|
|Traceability|Low — no record of when branch landed|High — merge commit marks the integration point|Medium — commits are present, boundary isn't explicit|
|Bisect-ability|Excellent (linear)|Good (bisect skips merge commits)|Excellent (linear)|
|When it's the right call|Solo work, trivial fixes, no audit requirement|Team workflows, release branches, compliance audit trails|Teams that want linear history _and_ want to preserve rebased feature work|
|Reversibility|`git reset --hard HEAD~N` (count commits manually)|`git revert -m 1 <merge-commit>` or `git reset --hard HEAD~1`|Same as fast-forward|

## Self-Check

### Conceptual Questions

1. What condition must be true for Git to perform a fast-forward merge?
2. What does `git log --oneline --graph` look like after a fast-forward merge vs after a `--no-ff` merge?
3. What flag prevents a merge from proceeding if a three-way merge would be required?
4. You see `Merge made by the 'ort' strategy.` in the output — what does this tell you?
5. What is the safer alternative to `git merge main` when you want to update a feature branch with upstream changes?
6. What does `git merge --abort` do, and what state does it leave the repo in?
7. How does choosing fast-forward vs three-way merge affect the usefulness of `git bisect`? (Connect to [[9.030 — Git Bisect]])

<details> <summary>Answers</summary>

1. The current branch (the merge target) must be a direct ancestor of the branch being merged in — i.e., the target has not received any new commits since the feature branch was created from it.
    
2. **Fast-forward:** a straight vertical line — `* commit` entries with no branching characters. **`--no-ff`:** a merge commit node with `|\` and `|/` showing two parents converging.
    
3. `--ff-only`. The command is `git merge --ff-only <branch>`. If fast-forward is impossible, Git exits with `fatal: Not possible to fast-forward, aborting.` rather than silently creating a merge commit.
    
4. Git used the "ort" strategy (the default three-way merge strategy since Git 2.33) to combine changes from both branches by identifying the common ancestor, computing diffs from each side, and applying both. A merge commit was created.
    
5. `git rebase origin/main` from the feature branch. This replays the feature branch commits on top of the current `main` tip, keeping the branch history linear and the PR diff clean — as opposed to `git merge main` which creates a merge commit in the feature branch.
    
6. `git merge --abort` terminates an in-progress merge that has paused due to conflicts, and restores the working tree and index to exactly the state they were in before the merge command was run. The repo is left on the same branch, with no staged changes and no MERGE_HEAD reference.
    
7. **Fast-forward** produces linear history which `git bisect` traverses most efficiently — every commit is testable in sequence and the range halves cleanly. **Three-way merges** create merge commits, which `git bisect` skips by default (they aren't meaningful test points). Both are bisect-able; fast-forward is marginally faster for bisect but loses the "what landed when" context. Missing merge boundaries (from accidental fast-forwards of feature branches) makes it harder to identify _which feature_ introduced a bug — you have the commit, but not the branch grouping. See [[9.030 — Git Bisect]] for the full bisect workflow.
    

</details>

**Drill 2 — Fix the broken thing**

```bash
$ git checkout main
$ git merge feature/inventory-sync
Merge made by the 'ort' strategy.
 src/InventoryApi/SyncJob.cs | 22 ++++++++++++++++++++++
 1 file changed, 22 insertions(+)
```

Your team uses `--no-ff` on all merges. This merge silently fast-forward merged (the output says `ort` strategy but a merge commit _was_ created — wait, actually here is the problem: you expected `Fast-forward` to be blocked but it wasn't). On closer inspection, you ran plain `git merge` without `--no-ff`. The merge commit happened to be three-way (ort) here, but in another case it would have fast-forwarded and you'd have lost branch history. Fix the team's workflow to enforce `--no-ff` automatically.

<details> <summary>Solution</summary>

**Diagnosis:** The team relies on engineers remembering `--no-ff`. That's not reliable at scale. The fix is to set it as the default in the repo's `.gitconfig`:

```bash
# Option A — enforce no-ff for this repo only
$ git config merge.ff false

# Verify
$ cat .git/config | grep -A2 '\[merge\]'
[merge]
	ff = false

# Now git merge always behaves as --no-ff
$ git merge feature/inventory-sync
# Will always create a merge commit, even if fast-forward is possible
```

```bash
# Option B — enforce globally for this engineer's machine
$ git config --global merge.ff false
```

```bash
# Option C — script-level enforcement (most reliable for a team)
# In scripts/merge-to-main.sh, always pass --no-ff explicitly
git merge --no-ff "$BRANCH" -m "merge: ${BRANCH} — ${DESC}"
# This beats any local git config that might have been changed
```

If the unwanted merge commit is already local (not pushed), undo it and redo with the correct flag:

```bash
$ git reset --hard HEAD~1
$ git merge --no-ff feature/inventory-sync -m "merge: feature/inventory-sync — nightly inventory sync job"
```

**Drill 3 — Under time pressure**

Production is showing a spike in `OrderService` 500 errors. You've identified that `feature/discount-engine` landed in the last release and may be the cause. You need to revert the merge and redeploy in under 3 minutes. Walk through exactly what you type.

<details> <summary>Solution</summary>

```bash
# 1 — Find the merge commit SHA (fastest method)
$ git log --oneline --graph -15
# Scan for the merge commit for feature/discount-engine
*   c7d3e2a (HEAD -> main, origin/main) merge: feature/discount-engine — promotional discount calculation
|\
| * f1a9b3c feat: apply discount tiers to OrderService pricing
| * 2d7c41e feat: add DiscountEngine service registration
|/
* b8e0f12 chore: update appsettings schema

# 2 — Revert the merge commit (-m 1 means "keep main's side as the baseline")
$ git revert -m 1 c7d3e2a
# Editor opens with pre-filled message: "Revert "merge: feature/discount-engine ...""
# Save and close immediately

[main 9f2a8b1] Revert "merge: feature/discount-engine — promotional discount calculation"
 2 files changed, 0 insertions(+), 30 deletions(-)

# 3 — Push immediately
$ git push origin main
# CI triggers automatically; deployment follows per pipeline

# 4 — Confirm on origin
$ git log --oneline -3 origin/main
9f2a8b1 (HEAD -> main, origin/main) Revert "merge: feature/discount-engine..."
c7d3e2a merge: feature/discount-engine — promotional discount calculation
b8e0f12 chore: update appsettings schema
```

**Why `revert -m 1` not `reset --hard`:** `reset --hard` rewrites history, which requires a force-push to `origin/main`. On a protected branch that's often blocked, and it destroys the audit trail. `revert` creates a new commit that undoes the changes, is safe to push normally, and keeps the full history intact for the post-mortem. `-m 1` tells Git which parent to treat as the "mainline" for the revert — parent 1 is always the branch you merged _into_ (main), which is what you want to restore.

See [[9.392 — Rolling Back a Bad Deploy Under Time Pressure]] for the full incident rollback procedure.

</details>
