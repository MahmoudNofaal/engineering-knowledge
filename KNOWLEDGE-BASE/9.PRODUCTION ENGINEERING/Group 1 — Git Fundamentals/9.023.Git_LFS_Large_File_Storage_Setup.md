## What You're Doing and Why

Git LFS (Large File Storage) replaces large binary files in your Git history with small pointer files — plain text stubs that record the file's hash and size — while storing the actual bytes on a separate LFS server. Every `git clone`, `git fetch`, and `git checkout` transfers only the pointers that are checked out, not the full binary history. This means a repo that contains 2 GB of historical test fixtures clones in seconds because developers only pull the version their branch currently needs. LFS is the right tool when binaries must live alongside code (signed certs, ML model weights, realistic database seed files, design exports) but must not metastasize the repo's pack files. For artifacts that belong in a registry rather than source control — Docker images, NuGet packages, npm bundles — see [[9.124 — Tagging and Pushing Images to a Registry]] instead; LFS is for files that are genuinely source-controlled inputs, not build outputs.

### Mental Model

Think of LFS as a coat-check counter bolted onto Git. When you commit a tracked binary, Git hands the file to the coat-check and keeps only the numbered ticket (the pointer). When you need the file — on checkout or pull — Git presents the ticket and the coat-check hands back the coat. Every other branch that doesn't check out that commit never fetches the coat at all. The ticket is tiny; the coats stay off the floor.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git lfs install`|Installs LFS hooks into `.git/hooks` for the current repo|First step when enabling LFS on any repo|
|`git lfs track "*.psd"`|Adds a pattern to `.gitattributes` telling LFS to manage matching files|Every time you add a new binary file type|
|`git lfs ls-files`|Lists every file currently tracked by LFS in the working tree|Verifying which files are under LFS management|
|`git lfs status`|Shows LFS file state (modified, staged, committed) like `git status` but for large files|Checking LFS staging before committing|
|`git lfs pull`|Downloads LFS objects that weren't fetched on clone/fetch|After a shallow clone or when LFS files show as pointers on disk|
|`git lfs fetch --all`|Downloads all LFS objects for all refs, not just HEAD|Before migrating a repo or creating a full backup|
|`git lfs migrate import --include="*.zip"`|Rewrites history to move matching files from Git history into LFS|Retroactively fixing a repo that already has committed binaries|
|`git lfs migrate export --include="*.zip"`|Rewrites history to pull files out of LFS back into Git|Rarely — reverting an LFS migration|
|`git lfs prune`|Deletes LFS objects that are no longer referenced by any local ref|Reclaiming local disk space|
|`git lfs env`|Prints LFS configuration: endpoint URL, transfer settings|Diagnosing connection issues with the LFS server|

## Step-by-Step Walkthrough

### The Scenario

The NotificationWorker repo has accumulated 340 MB of `.zip` email template archives in `assets/templates/` committed directly into Git. Every new developer who onboards spends 20+ minutes on the initial clone. You've been asked to enable LFS for all future `.zip` files in the repo. A full history rewrite is out of scope for now — that's the production scenario below.

### Full Walkthrough

```bash
# Confirm LFS is installed on your machine (comes with Git for Windows; install separately on macOS/Linux)
$ git lfs version
git-lfs/3.5.1 (GitHub; linux amd64; go 1.21.1)

# Clone the repo if you haven't already
$ git clone git@github.com:intervision/notification-worker.git

remote: Counting objects: 100% (4821/4821), done.
remote: Compressing objects: 100% (1203/1203), done.

Resolving deltas: 100% (2018/2018), done.

$ cd notification-worker

# Install LFS hooks into THIS repo's .git directory — one-time per clone
$ git lfs install

# Track all future .zip files with LFS — this writes to .gitattributes
$ git lfs track "*.zip"

# .gitattributes itself must be committed — it's what tells every clone to use LFS
$ git add .gitattributes
$ git commit -m "chore: enable Git LFS tracking for zip and bin assets"
[main 7c3a1f8] chore: enable Git LFS tracking for zip and bin assets

# Now add a new binary file — LFS intercepts it automatically

$ git add assets/templates/q3-email-templates-v2.zip
$ git lfs status

[main 2d8f4c9] feat: add Q3 email template archive

$ git push origin main

# Another developer clones — they get a fast clone; LFS binary downloads on checkout
$ git clone git@github.com:intervision/notification-worker.git nw-dev2

```

Every step here:

1. `git lfs install` — without this the hooks aren't present and LFS tracking is silently skipped
2. `git lfs track "*.zip"` — writes the filter rule; not tracking a pattern first is the #1 LFS mistake
3. Committing `.gitattributes` — the pattern lives here; if this isn't committed, no other clone will honour it
4. `git lfs status` before committing — confirms the file is staged as an LFS object, not as a plain blob
5. Checking the pointer with `git show` — proves the committed content is a small text pointer, not 49 MB of binary


## Production Scenario

### Realistic Context

The InterVision monorepo has grown to 4.1 GB. A `git log --all --stat` investigation reveals that `tests/fixtures/` contains 2.8 GB of `.zip` database seed archives committed directly across 18 months of history. Every CI runner does a full clone on each build, consuming 40+ minutes before a single `dotnet restore` runs. The decision: migrate all existing `.zip` and `.bak` files in `tests/fixtures/` out of Git history and into LFS, then update the CI pipeline to clone with `--filter=blob:none` so runners only fetch what they need. This is a destructive history rewrite — coordinate with the team, merge or close all open PRs first, and everyone must re-clone after.

## Gotchas and Troubleshooting

### "This repository is over its data quota. Account responsible for LFS bandwidth should purchase more data packs to restore access."

**Root cause:** GitHub Free/Team accounts have 1 GB LFS storage and 1 GB monthly bandwidth included. Heavy CI usage — especially if every runner fetches the full LFS object set — burns bandwidth rapidly. Each `git lfs pull` on a CI runner counts against the monthly quota.

```bash
# Typical state: CI runner tries to fetch LFS objects and gets HTTP 422
$ git lfs pull
Error downloading object: tests/fixtures/seed-2024-q1.zip (298 MB):
  [422] This repository is over its data quota.
```

**Fix:**

```bash
# Short term — purchase additional LFS data packs in GitHub billing settings
# Or: use a self-hosted LFS server (Gitea, AWS S3 via git-lfs-s3-proxy) instead

# Immediate mitigation: only fetch the LFS objects you actually need
$ GIT_LFS_SKIP_SMUDGE=1 git clone git@github.com:intervision/intervision.git
$ cd intervision
$ git lfs pull --include="tests/fixtures/seed-2024-q1.zip"
```

**Prevention:** Set `GIT_LFS_SKIP_SMUDGE=1` in CI environments that don't need the large files, and use `git lfs pull --include` to fetch only what a given job needs. Add an LFS bandwidth alert in GitHub billing settings at 80% of your monthly quota.

### "Encountered X file(s) that should have been pointers, but weren't"

**Root cause:** A file matched by a `.gitattributes` LFS pattern was committed as a plain Git blob — usually because the developer didn't have `git lfs install` run in that clone before staging.

```bash
$ git push origin main
Remote "origin" does not support the Git LFS locking API.
batch response: Encountered 2 file(s) that should have been pointers, but weren't:
        tests/fixtures/seed-new.zip
        assets/certs/test-cert.pfx
error: failed to push some refs to 'git@github.com:intervision/intervision.git'
```

**Fix:**

```bash
# Confirm the files are blobs, not pointers
$ git cat-file -t HEAD:tests/fixtures/seed-new.zip
blob
$ git show HEAD:tests/fixtures/seed-new.zip | head -c 4
PK    # binary, not a pointer

# Convert them: remove from index, re-add so LFS filter intercepts
$ git rm --cached tests/fixtures/seed-new.zip assets/certs/test-cert.pfx
$ git add tests/fixtures/seed-new.zip assets/certs/test-cert.pfx
$ git lfs status
Objects to be committed:
        tests/fixtures/seed-new.zip (LFS: e9f1a3b5)
        assets/certs/test-cert.pfx (LFS: c7d9e1f3)

$ git commit --amend --no-edit
$ git push origin main
```

**Prevention:** Add `git lfs install` to your repo's setup script (`scripts/setup-dev.sh`) and enforce it in a `pre-push` hook that checks `git lfs status` for non-pointer blobs before pushing.

### "smudge filter lfs failed" on `git checkout` or `git clone`

**Root cause:** `git-lfs` is not installed on the machine, or the LFS server is unreachable, and Git's smudge filter (which replaces pointer files with real content on checkout) is failing. The file on disk is left as a raw pointer text file.

```bash
$ git checkout main
Downloading tests/fixtures/seed-2024-q1.zip (298 MB)
Error downloading object: tests/fixtures/seed-2024-q1.zip:
  [35] SSL connect error

# The file on disk is just the pointer text:
$ cat tests/fixtures/seed-2024-q1.zip
version https://git-lfs.github.com/spec/v1
oid sha256:e8a3f29b17c4d12...
size 298412032
```

**Fix:**

```bash
# If git-lfs is missing: install it, then re-pull
$ sudo apt-get install git-lfs  # or: brew install git-lfs
$ git lfs install
$ git lfs pull

# If it's an SSL/network error, check the LFS endpoint and proxy settings
$ git lfs env | grep -i endpoint
Endpoint=https://github.com/intervision/intervision.git/info/lfs (auth=none)

# Test connectivity directly
$ curl -I https://github.com/intervision/intervision.git/info/lfs
HTTP/2 200
```

**Prevention:** Include `git lfs version` in your environment setup checks. On CI, run `git lfs install` explicitly as an early step before checkout.

### "`git lfs migrate import` rewrote history but old blobs are still in `.git/objects`"

**Root cause:** The migration rewrites commits and moves objects into LFS, but `git gc` has not yet run to expire the old unreachable blobs. The `.git` directory still contains the original large files until they're pruned.

```bash
# Immediately after migration — .git is still large
$ du -sh .git
4.3G    .git

# Unreachable objects are still present
$ git count-objects -v
count: 0
size: 0
in-pack: 0
packs: 0
size-pack: 0
prune-packable: 0
garbage: 0
size-garbage: 0
# But old pack files from before the migration:
$ ls -lh .git/refs/
# The refs are cleaned, but stale packs may remain
```

**Fix:**

```bash
# Expire reflog entries and run aggressive GC to remove unreachable objects
$ git reflog expire --expire=now --all
$ git gc --aggressive --prune=now
Counting objects: 18204, done.
Delta compression using up to 8 threads.
Compressing objects: 100% (10823/10823), done.
Writing objects: 100% (18204/18204), done.
Total 18204 (delta 7391), reused 18204 (delta 7391)

$ du -sh .git
312M    .git
```

**Prevention:** Always run `git reflog expire --expire=now --all && git gc --prune=now` as the final step of any LFS history migration before pushing.

### `git lfs push --all origin` stalls or times out on large binary sets

**Root cause:** Pushing thousands of large LFS objects in a single batch overwhelms the transfer queue, especially over high-latency connections. The default concurrent transfer count (8) combined with very large files can exhaust the HTTP keep-alive window.

```bash
$ git lfs push --all origin
Uploading LFS objects:   3% (12/340), 847 MB | 0 B/s, done.
error: insufficient progress
batch response: [0] <nil>
```

**Fix:**

```bash
# Reduce concurrent transfers to avoid overwhelming the connection
$ git config lfs.concurrenttransfers 2

# Split the push by pushing LFS objects for recent refs only, not --all at once
$ git lfs push origin refs/heads/main
$ git lfs push origin refs/heads/develop
# Then push remaining refs once those succeed

# Or: use the standalone file transfer mode which is more resilient
$ GIT_LFS_STANDALONE_FILE=1 git lfs push --all origin
```

**Prevention:** For initial LFS migrations with large object counts, run the push from a machine on the same network as the LFS server, or use `lfs.concurrenttransfers 2` permanently in the repo's local config.

---

### "Repository rule violations found for refs/heads/main — push ruleset: max file size exceeded"

**Root cause:** GitHub's push ruleset (or branch protection rule) rejects files above a configured size threshold. If you're pushing a file that should be in LFS but was committed as a plain blob, the push is rejected at the ruleset level before LFS even enters the picture.

```bash
$ git push origin main
remote: error: GH013: Repository rule violations found for refs/heads/main.
remote: - Repository push ruleset: (rule: max_file_size)
remote:   - tests/fixtures/seed-new.zip: 298.4 MB exceeds the maximum file size of 100 MB.
To git@github.com:intervision/intervision.git
 ! [remote rejected] main -> main (push declined due to repository rule violations)
error: failed to push some refs to 'git@github.com:intervision/intervision.git'
```

**Fix:**

```bash
# The file must be re-committed through LFS — see "Encountered X files that should be pointers" fix above
$ git rm --cached tests/fixtures/seed-new.zip
$ git add tests/fixtures/seed-new.zip  # LFS must be installed and pattern must be tracked
$ git commit --amend --no-edit
$ git push origin main
Uploading LFS objects: 100% (1/1), 298 MB | 9.2 MB/s, done.
```

**Prevention:** Enable GitHub's "max file size" push ruleset at 100 MB on protected branches. This turns accidental blob commits into a push failure rather than a silent repo bloat problem.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# Add to ~/.gitconfig [alias] section or run as `git config --global alias.<name> <cmd>`

# List all LFS-tracked files in the current checkout with sizes
[alias]
    lfs-ls = lfs ls-files --long
    lfs-size = "!git lfs ls-files --long | awk '{sum += $3} END {print sum/1024/1024 \" MB total LFS content\"}'"
    lfs-verify = lfs fsck

# Shell function: fetch only the LFS files you actually need by glob
# Add to ~/.bashrc or ~/.zshrc
lfs-pull-match() {
    local pattern="${1:?Usage: lfs-pull-match <glob>}"
    git lfs pull --include="$pattern"
}
# Usage: lfs-pull-match "tests/fixtures/seed-2024-*"
```

### Automation Opportunity

Drop this into a new-developer onboarding script (`scripts/setup-dev.sh`). Every developer who sets up the repo from scratch gets LFS configured correctly in one shot:

```bash
#!/usr/bin/env bash
# scripts/setup-dev.sh — run once after cloning InterVision
set -euo pipefail

echo "==> Checking git-lfs installation..."
if ! command -v git-lfs &>/dev/null; then
    echo "ERROR: git-lfs is not installed."
    echo "  macOS:  brew install git-lfs"
    echo "  Ubuntu: sudo apt-get install git-lfs"
    echo "  Windows: winget install GitHub.GitLFS"
    exit 1
fi

echo "==> Installing LFS hooks for this repo..."
git lfs install

echo "==> Pulling LFS objects for current branch..."
# Only pull what HEAD needs — skip the full history
git lfs pull

echo "==> LFS setup complete."
git lfs ls-files | wc -l | xargs -I{} echo "  {} LFS file(s) checked out"
```

### Keyboard / CLI Efficiency Notes

- `GIT_LFS_SKIP_SMUDGE=1 git clone <url>` — fastest way to clone a repo with LFS without downloading any LFS objects; hydrate selectively with `git lfs pull --include` afterward. Essential for CI jobs that don't need the binary files at all.
- `git lfs fetch --recent` — fetches LFS objects for refs touched in the last 7 days (configurable with `lfs.fetchrecentrefsdays`). A good middle ground between fetching everything and fetching nothing.
- `git lfs status` before every commit when working in a repo with LFS — takes under a second and confirms your binary is staged as an LFS object, not a blob.
- `git lfs env` is the first command to run when diagnosing any LFS connectivity issue — it shows the endpoint, auth method, and transfer settings in one shot.

### Spoken Answers

**Q: Walk me through the process of adding Git LFS support to an existing repo and migrating files that are already in history.**

> **Average answer:** "You run `git lfs install`, then `git lfs track` for the file patterns you want, commit the `.gitattributes`, and push. For existing files, you use `git lfs migrate import`."

> **Great answer:** "The setup itself is quick — `git lfs install` hooks LFS into the repo, `git lfs track` writes the pattern to `.gitattributes`, commit that file, and from that point new commits of matching file types are automatically stored via LFS. The harder part is migrating existing history, because `git lfs migrate import --everything` rewrites every commit that touched those files — it changes commit SHAs, so every developer must re-clone, every open PR becomes invalid, and branch protection rules on the remote need to allow a force push temporarily. I'd coordinate by merging or closing all open PRs first, notifying the team, running the migration on a fresh full clone, then running `git reflog expire --expire=now --all && git gc --prune=now` before force-pushing all branches. The test is confirming the `.git` folder is dramatically smaller and `git lfs ls-files` shows the expected objects locally."

**Q: A developer clones the repo and their LFS files show up as plain text pointer files instead of the actual content. What do you check?**

> **Average answer:** "They probably don't have git-lfs installed. Tell them to run `git lfs install` and then `git lfs pull`."

> **Great answer:** "First I confirm whether it's an installation issue or a connectivity issue. `git lfs version` tells you immediately if the binary is missing — if it is, that's the answer: install it and re-run `git lfs pull`. If LFS is installed, the next question is whether the smudge filter ran but failed, or whether it was skipped. Check if `GIT_LFS_SKIP_SMUDGE=1` is set in their environment or CI config — that explicitly skips hydration on clone. If neither, run `git lfs pull` and watch for the error — it'll be either an auth failure (HTTP 401/403, common with SSH-authenticated remotes that need a different LFS auth token) or a network error. `git lfs env` shows the endpoint and auth mode, and `curl -I <endpoint>` tests connectivity directly. The smudge filter failing silently is the most dangerous scenario — the developer gets pointer text in a binary slot and their tests fail with 'not a zip file' errors rather than an obvious LFS message."

**Q: Tell me about a time when a large binary in a Git repo caused a real operational problem and how you resolved it.**

> **War Story Prompt (adapt as your own if you've experienced something similar):** Three weeks into a new project, a developer committed a 450 MB recorded HTTP traffic capture file (`paymentgateway-traffic-capture.pcap`) directly to the PaymentGateway repo's `tests/` directory — a one-time thing to share with the team, they said. A week later, every CI build was taking 55 minutes just to clone the repo, and two new contractors couldn't even finish onboarding because the clone was timing out over their VPN. I traced it with `git rev-list --objects --all | git cat-file --batch-check` and found the culprit immediately. The fix was `git lfs migrate import --include="*.pcap,*.pcapng" --everything`, followed by a forced re-clone notice to the team and a force push. We also added a GitHub push ruleset capping individual files at 50 MB so this class of mistake becomes a push error rather than a silent repo bloat. CI clone time dropped from 55 minutes to under 2 minutes.

### Decision Framework

## Decision Framework

### When to Reach for This

- [ ] You have binary files that are genuine source-controlled inputs — test fixtures, signed certificates, design exports, ML model checkpoints — that must live in the repo but are too large for normal Git objects
- [ ] Individual files are regularly > 10 MB, or the repo's total binary content is > 100 MB and growing
- [ ] Your remote host (GitHub, GitLab, Azure DevOps) has a documented file size limit and you're approaching it
- [ ] CI clone time is visibly increasing sprint-over-sprint and `git rev-list --objects --all | git cat-file --batch-check` reveals large blobs as the cause
- [ ] Multiple developers need to work with different versions of a large binary (the asset has a meaningful version history, not just a "current" state)

### When NOT To

- [ ] The large file is a **build output** (Docker image, NuGet package, compiled binary) — these belong in registries (see [[9.124 — Tagging and Pushing Images to a Registry]]), not in LFS
- [ ] The file is only needed by one CI job and could be fetched from S3/Azure Blob at build time — adding it to LFS creates unnecessary coupling between the Git remote and binary storage
- [ ] The file never changes — a truly static asset that never has meaningful version history is better served by a CDN or artifact store with a stable URL
- [ ] Your team can't coordinate the force-push required after a history migration right now — defer the migration, use `.gitignore` to stop the bleeding, and schedule the rewrite for a quiet sprint

### Tool Comparison

||Git LFS|Artifact Registry (ACR, Azure Artifacts, S3)|Plain Git Blob|
|---|---|---|---|
|Version history|Full, tied to commit history|Manual, via tags or path conventions|Full, but bloats pack files|
|CI integration|`git lfs pull --include` in clone step|Explicit download step (`az storage blob download`, etc.)|Automatic on clone|
|Storage cost|LFS quota on host + optional self-hosted|Pay per GB/request but scales indefinitely|Free but repo size explodes|
|When it's the right call|Source-controlled binary inputs with real version history|Build artifacts, images, large datasets with no code coupling|Files under ~5 MB with text-diffable content|

## Self-Check

### Conceptual Questions

1. What does `git lfs install` actually do to a local repo — where does it write, and what happens if you skip it?
2. You run `git add assets/model-weights.onnx` in a repo where `*.onnx` is tracked in `.gitattributes` via LFS. What does `git status` show differently from a non-LFS file add?
3. What flag do you pass to `git clone` to skip downloading LFS objects entirely at clone time?
4. A CI job logs `Error downloading object: ... [401] Unauthorized`. What's the first thing you check and what command gives you that information?
5. What is the difference between `git lfs fetch` and `git lfs pull`?
6. Why should you never run `git lfs migrate import` without first merging or closing all open PRs?
7. How does Git LFS interact with `git log --oneline --graph` and `git diff` — what do you see for LFS-tracked files?

<details> <summary>Answers</summary>

1. `git lfs install` writes three hooks into `.git/hooks/`: `pre-push` (uploads LFS objects before refs are pushed), `post-checkout` (fetches LFS objects on checkout), and `post-merge` (fetches LFS objects after merge). It also sets the `filter.lfs.*` config entries in `.git/config`. If you skip it, the smudge/clean filters are never invoked — files are committed as plain blobs regardless of `.gitattributes` patterns, and you get the "should have been pointers, but weren't" push error.
    
2. `git status` shows the file as a staged new file exactly like any other — but `git lfs status` shows it as `(LFS: <sha>)` rather than `(Git: <sha>)`. Running `git show :<path>` reveals the LFS pointer text rather than binary content. The filesize in the staging area is 127–130 bytes (the pointer), not the actual file size.
    
3. `GIT_LFS_SKIP_SMUDGE=1 git clone <url>` — or equivalently `git -c filter.lfs.smudge= -c filter.lfs.required=false clone <url>`. In GitHub Actions: `lfs: false` in the `actions/checkout` step parameters.
    
4. First check: `git lfs env` — it shows the LFS endpoint and the `AccessDownload`/`AccessUpload` fields that indicate the auth method in use. A `[401]` typically means the LFS server is using HTTP Basic auth but the credentials helper isn't configured, or the token has expired. On GitHub, this usually means the GITHUB_TOKEN in CI doesn't have `contents: read` permission, or the repo is private and the runner's credential helper isn't passing the PAT to LFS requests.
    
5. `git lfs fetch` downloads LFS objects into the local LFS cache (`.git/lfs/objects`) but does **not** update the files in the working tree — pointers on disk stay as pointers. `git lfs pull` does both: it fetches the objects AND runs the smudge filter to replace pointers in the working tree with actual file content. Use `fetch` when you want objects ready for future checkouts; use `pull` when you need the files usable right now.
    
6. `git lfs migrate import --everything` rewrites every commit SHA in the affected history. Any open PR that was opened against the old SHAs becomes impossible to merge cleanly — GitHub can't reconcile the rewritten base branch against the PR's original base commit. All open work must land on the main branch first, then the migration runs, then everyone re-clones from the rewritten history.
    
7. In `git log --oneline --graph`, LFS-tracked files appear in commits exactly like any other file — the log output is identical. In `git diff`, LFS-tracked files show a diff of the pointer file text rather than a binary diff, which means you see the `oid sha256:...` line change if the file changes. This is actually more useful than `Binary files a/... and b/... differ`. However, `git diff --stat` shows the pointer size (127 bytes) as changed, not the actual file size, which can be misleading.
    

</details>

**Drill 3 — Under time pressure**

You're on call. A CI pipeline for the PaymentGateway service is failing because a contractor force-pushed directly to `main` without LFS installed, converting 3 LFS pointer files into plain text files in the working tree on all new clones. The pointer text is being passed to the integration test suite, which is now failing with `System.IO.InvalidDataException: End of Central Directory record could not be found`. Every new clone is broken. You need to fix the remote state within 5 minutes.

<details> <summary>Solution</summary>

```bash
# 1. Get a clean clone where LFS is properly installed
$ GIT_LFS_SKIP_SMUDGE=1 git clone git@github.com:intervision/payment-gateway.git pg-fix
$ cd pg-fix
$ git lfs install
# — 45 seconds

# 2. Find which files are currently pointers-as-blobs (stored as plain text in Git)
$ git lfs status
On branch main
Objects not in LFS:
        tests/fixtures/payment-flow-capture.pcap (Git: corrupted)
        tests/fixtures/card-tokenization-testdata.zip (Git: corrupted)
        tests/fixtures/fraud-model-v3.onnx (Git: corrupted)
# — 10 seconds

# 3. Remove all three from the index and re-add through LFS
$ git rm --cached tests/fixtures/payment-flow-capture.pcap \
                  tests/fixtures/card-tokenization-testdata.zip \
                  tests/fixtures/fraud-model-v3.onnx
$ git add tests/fixtures/payment-flow-capture.pcap \
          tests/fixtures/card-tokenization-testdata.zip \
          tests/fixtures/fraud-model-v3.onnx
# LFS pull to populate local cache if needed
$ git lfs pull --include="tests/fixtures/*"
# — 60-90 seconds depending on file sizes

# 4. Confirm LFS objects now staged
$ git lfs status
Objects to be committed:
        tests/fixtures/payment-flow-capture.pcap (LFS: b2c3d4e5)
        tests/fixtures/card-tokenization-testdata.zip (LFS: f6a7b8c9)
        tests/fixtures/fraud-model-v3.onnx (LFS: 1d2e3f4a)

# 5. Amend the bad commit — keep the message, fix the content
$ git commit --amend --no-edit
# — 5 seconds

# 6. Force push (branch protection allows force push for repo admins)
$ git push --force-with-lease origin main
Uploading LFS objects: 100% (3/3), 842 MB | 9.1 MB/s, done.
Writing objects: 100% (3/3), 514 bytes | 514.00 KiB/s, done.
To git@github.com:intervision/payment-gateway.git
 + abc1234...9f8e7d6 main -> main (forced update)
# — 2-3 minutes for upload depending on connection

# 7. Notify CI to re-run — new clone will be clean
# Post to incident channel: "main force-fixed — re-trigger your builds"
```

**Why each step is safe under pressure:**

- `GIT_LFS_SKIP_SMUDGE=1` on clone: fastest clone possible; you don't need the old corrupt state
- `git rm --cached` not `git rm`: removes from index only, never touches disk — no risk of data loss
- `--force-with-lease` not `--force`: fails if someone else pushed between your fetch and your push — safer than bare `--force`
- `git commit --amend`: replaces only the last commit, which is the bad one the contractor pushed — no deeper history rewrite
