### Prerequisites

- None. This is the first hands-on Git note in the domain — it assumes only that Git is installed (`git --version` returns something).

### Where This Fits

You're starting a new service — `OrderService` needs its own repo — or you've just been handed a link to an existing one and need to get it onto your machine before you can touch a single line of code. Every single day-to-day Git action in this domain (committing, branching, pushing) only matters once you've got a repository initialized or cloned and a remote correctly wired up. Get the remote wrong here and every `git push` for the next three weeks fights you.

## What You're Doing and Why

`git init` creates a new, empty repository by writing a `.git` directory that tracks history as a sequence of snapshots; `git clone` does the same thing but also downloads an existing repository's full history and automatically wires up the remote connection back to where it came from. A "remote" is just a named URL Git remembers so you don't have to type `https://github.com/org/repo.git` every time you want to push or pull — see [[7.917 — Trunk-Based Development]] for how branching policy interacts with where you push.

### Mental Model

Think of `.git` as a database of snapshots, not diffs — every commit is a complete picture of the project at that moment, and branches are just movable pointers into that history. `git clone` is "copy the whole database and point my pointer at the same address the original uses." `git remote` is your address book: a short name (`origin`) mapped to a URL, nothing more.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git init`|Creates a new `.git` directory in the current folder|Starting a brand-new project with no existing repo|
|`git clone <url>`|Downloads a repo's full history and sets up `origin`|Joining a project that already exists on GitHub/Azure DevOps|
|`git clone <url> <dir>`|Clones into a specific folder name instead of the repo's default name|Avoiding name collisions, or cloning multiple copies locally|
|`git clone --branch <name> <url>`|Clones and checks out a specific branch immediately|You only need a feature branch, not `main`|
|`git remote -v`|Lists configured remotes and their URLs (fetch + push)|Confirming you're pointed at the right repo before pushing|
|`git remote add <name> <url>`|Adds a new named remote|Wiring up `origin` after `git init`, or adding `upstream` for a fork|
|`git remote set-url <name> <url>`|Changes an existing remote's URL|Repo moved orgs, or you switched from HTTPS to SSH|
|`git remote remove <name>`|Deletes a remote|Cleaning up a stale fork reference|
|`git remote rename <old> <new>`|Renames a remote|Standardizing naming after adding a remote with the wrong name|

## Step-by-Step Walkthrough

### The Scenario

You've been asked to spin up a brand-new microservice, `InventoryApi`, that doesn't exist anywhere yet. Your team lead has already created an empty repo on GitHub at `github.com/contoso/InventoryApi` and given you push access. You need to get a local working copy connected to it correctly — the team uses SSH remotes, not HTTPS, because everyone has SSH keys already set up (see [[9.020 — SSH Keys for Git — Setup and Multiple Identities]]).

### Full Walkthrough

```bash
$ mkdir InventoryApi
$ cd InventoryApi
$ git init
Initialized empty Git repository in /home/dev/InventoryApi/.git/
```

_Creates the local repo. Nothing is tracked yet — `git status` would show no commits._

```bash
$ git remote add origin git@github.com:contoso/InventoryApi.git
$ git remote -v
origin  git@github.com:contoso/InventoryApi.git (fetch)
origin  git@github.com:contoso/InventoryApi.git (push)
```

_Names the remote `origin` by convention — first commit pushed here will need `-u` to set the tracking branch (see [[9.008 — Git Push and Pull — Tracking Branches]])._

```bash
$ echo "# InventoryApi" > README.md
$ git add README.md
$ git commit -m "chore: initial commit"
[main (root-commit) 7c1a9e2] chore: initial commit
 1 file changed, 1 insertion(+)
 create mode 100644 README.md

$ git push -u origin main
Enumerating objects: 3, done.
Counting objects: 100% (3/3), done.
Writing objects: 100% (3/3), 230 bytes | 230.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0), pack-reused 0
To github.com:contoso/InventoryApi.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.
```

_`-u` links your local `main` to `origin/main` so future `git push`/`git pull` need no arguments._

### What Could Go Wrong Mid-Walkthrough

If the repo on GitHub was created _with_ a README or `.gitignore` (some teams enable "Initialize this repository" by default), your local history and the remote's history have unrelated commits. In that case, skip the manual `git init` + `git remote add` flow entirely and clone instead:

```bash
$ git clone git@github.com:contoso/InventoryApi.git
Cloning into 'InventoryApi'...
remote: Enumerating objects: 5, done.
remote: Counting objects: 100% (5/5), done.
remote: Compressing objects: 100% (3/3), done.
remote: Total 5 (delta 0), reused 0 (delta 0), pack-reused 0
Receiving objects: 100% (5/5), done.
$ cd InventoryApi
$ git remote -v
origin  git@github.com:contoso/InventoryApi.git (fetch)
origin  git@github.com:contoso/InventoryApi.git (push)
```

Cloning always wins over manual setup when the remote already has content — it avoids the "unrelated histories" problem entirely (see [[9.046 — Resolving "Diverged Branches" Cleanly]] if you hit it anyway).

## Production Scenario

### Realistic Context

`PaymentGateway` was originally cloned by a contractor over HTTPS using a personal access token that has since expired. Every `git push` now fails for the whole team, and the org has since migrated to SSH-only access for this repo. You need to fix the remote URL on every developer's existing clone without anyone having to re-clone the 4-year-old repo (it's 1.2GB — nobody wants to re-download it).

### Full Implementation / Resolution

```bash
$ git remote -v
origin  https://github.com/contoso/PaymentGateway.git (fetch)
origin  https://github.com/contoso/PaymentGateway.git (push)

$ git remote set-url origin git@github.com:contoso/PaymentGateway.git

$ git remote -v
origin  git@github.com:contoso/PaymentGateway.git (fetch)
origin  git@github.com:contoso/PaymentGateway.git (push)
```

Roll this out to the whole team as a one-liner they can paste without thinking about it:

```bash
# fix-remote.sh — run from inside any existing PaymentGateway clone
#!/usr/bin/env bash
set -euo pipefail
git remote set-url origin git@github.com:contoso/PaymentGateway.git
echo "origin now points to: $(git remote get-url origin)"
```

### Verifying It Worked

```bash
$ git fetch origin
remote: Enumerating objects: 12, done.
remote: Counting objects: 100% (12/12), done.
remote: Total 8 (delta 3), reused 8 (delta 3), pack-reused 0
Unpacking objects: 100% (8/8), done.
From github.com:contoso/PaymentGateway
   a1b2c3d..e4f5g6h  main       -> origin/main
```

A successful `git fetch` with no auth prompt and no `fatal:` line confirms the SSH remote is correctly wired and the key is accepted.

## Gotchas and Troubleshooting

### "fatal: remote origin already exists."

**Root cause:** You ran `git remote add origin <url>` on a repo that already has an `origin` configured — often because it was cloned, not init'd from scratch.

```bash
$ git remote add origin git@github.com:contoso/InventoryApi.git
fatal: remote origin already exists.
```

**Fix:**

```bash
$ git remote set-url origin git@github.com:contoso/InventoryApi.git
```

**Prevention:** Run `git remote -v` before adding a remote — if anything's already listed, use `set-url` instead of `add`.

### "fatal: repository 'https://github.com/contoso/InventoryApi.git/' not found"

**Root cause:** Either the repo name is wrong, it's private and you're not authenticated, or you typo'd the org name.

```bash
$ git clone https://github.com/contoso/InventroyApi.git
Cloning into 'InventroyApi'...
fatal: repository 'https://github.com/contoso/InventroyApi.git/' not found
```

**Fix:**

```bash
$ git clone https://github.com/contoso/InventoryApi.git
```

**Prevention:** Copy the clone URL directly from the GitHub "Code" button instead of typing it from memory.

### "Permission denied (publickey)."

**Root cause:** You're cloning over SSH but your local machine has no SSH key registered with GitHub, or the wrong key is being offered.

```bash
$ git clone git@github.com:contoso/InventoryApi.git
Cloning into 'InventoryApi'...
git@github.com: Permission denied (publickey).
fatal: Could not read from remote repository.
```

**Fix:**

```bash
$ ssh -T git@github.com
$ # If that fails too, generate and register a key — see 9.020
$ ssh-add ~/.ssh/id_ed25519
```

**Prevention:** Run `ssh -T git@github.com` once after machine setup, before you ever need to clone under time pressure.

### "fatal: refusing to merge unrelated histories"

**Root cause:** You ran `git init` + `git remote add` + a local commit, then tried to pull from a remote that already had its own initial commit (e.g., GitHub's auto-generated README) — the two histories share no common ancestor.

```bash
$ git pull origin main
fatal: refusing to merge unrelated histories
```

**Fix:**

```bash
$ git pull origin main --allow-unrelated-histories
```

**Prevention:** When the remote repo was created with "Initialize this repository with a README" checked, `git clone` it instead of `git init`-ing locally — see the "What Could Go Wrong" branch above.

### "fatal: 'origin' does not appear to be a git repository"

**Root cause:** The remote name exists in config but the URL is malformed or was never actually set (common after a botched `git remote add` with a typo'd URL).

```bash
$ git push origin main
fatal: 'origin' does not appear to be a git repository
fatal: Could not read from remote repository.
```

**Fix:**

```bash
$ git remote -v
$ git remote set-url origin git@github.com:contoso/InventoryApi.git
```

**Prevention:** Always run `git remote -v` immediately after `git remote add` to confirm it took.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# ~/.gitconfig — shortcuts that pay off on day one of every new repo
[alias]
    rv = remote -v
    rao = "!f() { git remote add origin \"$1\"; }; f"
```

### Automation Opportunity

```bash
#!/usr/bin/env bash
# scripts/new-repo.sh — initializes a repo, wires origin, and pushes an initial commit in one step
set -euo pipefail
REPO_NAME="$1"
REMOTE_URL="$2"

mkdir "$REPO_NAME" && cd "$REPO_NAME"
git init -b main
echo "# $REPO_NAME" > README.md
git add README.md
git commit -m "chore: initial commit"
git remote add origin "$REMOTE_URL"
git push -u origin main
echo "Repo $REPO_NAME initialized and pushed to $REMOTE_URL"
```

### Keyboard / CLI Efficiency Notes

- `git clone <url> .` clones directly into the _current_ (already-created and empty) directory instead of making a new subfolder.
- `git init -b main` sets the initial branch name to `main` in one step, avoiding a separate `git branch -M main` if your global default is still `master`.
- Tab-completion on remote names (`git push or<TAB>` → `origin`) works out of the box with the standard Git Bash completion script — worth confirming it's sourced in your `.bashrc`/`.zshrc`.

### Spoken Answers

**Q: Walk me through setting up a brand-new repo locally and connecting it to an empty GitHub repo.**

> **Average answer:** "You run `git init`, then `git remote add origin` with the URL, then commit and push." Correct syntax, but no mention of checking whether the remote already has content, or what `-u` does on the first push.

> **Great answer:** Narrates checking first whether the GitHub repo was created with a README or `.gitignore` — because if it was, cloning is correct and `git init` will cause unrelated-histories pain later. Mentions running `git remote -v` immediately after adding the remote to confirm it took, and explains that `-u` on the first push sets up the tracking branch so nobody has to type `origin main` on every future push.

**Q: You run `git remote add origin <url>` and get `fatal: remote origin already exists` — what do you check, and what do you run instead?**

> **Average answer:** "I'd run `git remote remove origin` and add it again." Works, but skips a step and risks losing the fetch/push distinction if they're different.

> **Great answer:** Runs `git remote -v` first to see what's actually there and whether it's simply wrong or already correct. If it's wrong, uses `git remote set-url origin <url>` directly instead of remove-then-add, since `set-url` is the safer, more direct fix and preserves any other remote-specific config.

**Q: Tell me about a time a misconfigured remote caused a real problem for your team.**

> **Average answer:** "We had the wrong URL once and had to fix it." No specifics, no detail on diagnosis or fix.

> **Great answer:** Describes a specific symptom (e.g., pushes silently going to a personal fork instead of the org repo), how it was noticed (a PR never showed up on the expected repo), the diagnostic command run (`git remote -v`), and the fix (`set-url`) plus a prevention step like a repo-setup script that standardizes the remote name and URL for everyone.

### War Story Prompt

A new hire spent half a day debugging why their pushes to `OrderService` "weren't showing up in PRs" — turned out `git clone` had been run against their personal fork URL copied from an old Slack message, so every push was silently succeeding against the fork instead of the org repo. `git remote -v` caught it in ten seconds once someone thought to check.

## Decision Framework

### When to Reach for This

- [ ] You're starting a project that has zero history anywhere yet
- [ ] An empty remote repo already exists and you need a local copy wired to it
- [ ] A remote URL has changed (org rename, protocol switch, repo move) and existing clones need fixing

### When NOT To

- [ ] The remote already has commits you need locally — clone, don't init
- [ ] You just need a temporary local-only experiment with no intent to ever push — `git init` is fine, but don't bother wiring a remote at all

### Tool Comparison

||`git init` + `git remote add`|`git clone`|
|---|---|---|
|Speed|Slightly slower (multiple commands)|One command|
|Safety/reversibility|Risk of unrelated-histories conflict if remote isn't actually empty|Always history-consistent with the remote|
|When it's the right call|Remote repo is genuinely empty (no README/commits)|Remote already has any content at all|

2. What does `git remote -v` print for a repo with a single `origin` remote over SSH?
3. Which flag on `git push` sets up branch tracking so future pushes need no arguments?
4. You see `fatal: remote origin already exists` — what's the actual fix?

<details> <summary>Answers</summary>

1. `git init` creates a brand-new empty `.git` directory with no history and no remote. `git clone` creates the same `.git` structure but populates it with the full history downloaded from an existing remote, and automatically configures that remote as `origin`.
2. Two lines: `origin <url> (fetch)` and `origin <url> (push)` — typically the same URL for both unless push/fetch are deliberately split.
3. `-u` (or `--set-upstream`), e.g. `git push -u origin main`.
4. Run `git remote set-url origin <url>` instead of trying to add it again — `set-url` overwrites the existing URL directly.
5. `git remote set-url origin <new-url>` — one command, no risk of losing the remote's other config.
6. Don't `git init` locally and manually wire the remote — clone it instead, or you'll hit `fatal: refusing to merge unrelated histories` on the first pull.
7. [[9.002 — Git Add, Commit, and the Staging Area]].
