## What You're Doing and Why

Renaming or moving a file on disk and then running `git add` is not the same as telling Git you renamed a file — it looks like a delete plus an add, which breaks `git blame` and `git log --follow` unless Git can detect the similarity automatically. Using `git mv` (or understanding when Git auto-detects the rename) ensures the operation is staged correctly in one step and that similarity detection kicks in at commit time. The distinction matters in production code reviews: a reviewer looking at `git log Services/Refunds/RefundHandler.cs` needs to see the full history back to when the file was called `Services/RefundHandler.cs`, not just today's "add."

### Mental Model

Think of a Git rename as a forwarding address at the post office. If you just move your office without telling the post office, your old mail piles up at the old address and your new address starts fresh. `git mv` is filing the forwarding card — Git records "the content that used to live at path A now lives at path B," and history-aware commands like `git log --follow` and `git blame` know to look at the old address for older commits.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git mv OldName.cs NewName.cs`|Renames and stages in one step|Renaming a file you want history to follow|
|`git mv src/Services/File.cs src/Services/Refunds/File.cs`|Moves and stages in one step|Moving a file to a new directory|
|`git status`|Confirms the rename shows as `renamed:` not `deleted` + `new file`|Always — verify before committing|
|`git log --follow -- path/to/File.cs`|Shows history across renames|Auditing file history after a rename|
|`git log --diff-filter=R --summary`|Lists all renames in a commit range|Finding when a rename happened|
|`git diff --staged`|Shows the rename diff before commit|Verifying similarity score is captured|
|`git add -A`|Stages all changes including renames (via similarity detection)|When you already moved with `mv` on the filesystem|

## Step-by-Step Walkthrough

### The Scenario

You're working on the `InterVision` repo. A PR review comment asks you to rename `Services/PaymentValidator.cs` to `Services/PaymentValidationService.cs` (to match the naming convention) and move it from the flat `Services/` folder into `Services/Payments/`. The file has 18 months of commit history you cannot afford to lose.

### Full Walkthrough

```bash
$ cd ~/repos/InterVision

# Confirm current location and state
$ git status
On branch feature/rename-payment-validator
nothing to commit, working tree clean

# Step 1: Create the destination directory (git mv won't create it for you)
$ mkdir -p src/InterVision.Api/Services/Payments

# Step 2: Move AND rename in one git mv command — stages immediately
$ git mv \
  src/InterVision.Api/Services/PaymentValidator.cs \
  src/InterVision.Api/Services/Payments/PaymentValidationService.cs

# Step 3: Verify Git staged it as a rename, not a delete+add
$ git status
On branch feature/rename-payment-validator
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        renamed:    src/InterVision.Api/Services/PaymentValidator.cs -> src/InterVision.Api/Services/Payments/PaymentValidationService.cs
```

The `renamed:` label confirms Git captured the operation as a rename — similarity detection will preserve history.

```bash
# Step 4: Update any internal namespace or using references (now a code change, not a Git operation)
$ sed -i 's/namespace InterVision.Api.Services;/namespace InterVision.Api.Services.Payments;/' \
  src/InterVision.Api/Services/Payments/PaymentValidationService.cs

# Step 5: Stage the namespace fix
$ git add src/InterVision.Api/Services/Payments/PaymentValidationService.cs

# Step 6: Check status again — should show renamed + modified
$ git status
On branch feature/rename-payment-validator
Changes to be committed:
        renamed:    src/InterVision.Api/Services/PaymentValidator.cs -> src/InterVision.Api/Services/Payments/PaymentValidationService.cs
```

```bash
# Step 7: Commit — message convention: refactor for renames that don't change behaviour
$ git commit -m "refactor: rename PaymentValidator to PaymentValidationService and move to Payments folder"
[feature/rename-payment-validator 7d3a1b4] refactor: rename PaymentValidator to PaymentValidationService and move to Payments folder
 1 file changed, 1 insertion(+), 1 deletion(-)
 rename src/InterVision.Api/Services/{PaymentValidator.cs => Payments/PaymentValidationService.cs} (96%)
```

The `(96%)` is the similarity score — any score above 50% and Git records it as a rename.

```bash
# Step 8: Verify history is preserved across the rename
$ git log --follow --oneline -- src/InterVision.Api/Services/Payments/PaymentValidationService.cs
7d3a1b4 refactor: rename PaymentValidator to PaymentValidationService and move to Payments folder
3c89f10 fix: handle duplicate payment token edge case
a1d4788 feat: add Luhn validation to PaymentValidator
e22b401 chore: initial PaymentValidator scaffold
```

All 18 months of commits are visible — history is intact.

### What Could Go Wrong Mid-Walkthrough

**Scenario: You already moved the file with `mv` (not `git mv`) before reading this note**

```bash
$ mv src/InterVision.Api/Services/PaymentValidator.cs \
     src/InterVision.Api/Services/Payments/PaymentValidationService.cs

$ git status
On branch feature/rename-payment-validator
Changes not staged for commit:
        deleted:    src/InterVision.Api/Services/PaymentValidator.cs

Untracked files:
        src/InterVision.Api/Services/Payments/PaymentValidationService.cs
```

Git sees it as delete + new untracked. Fix: stage everything with `-A` and let Git's rename detection figure it out.

```bash
# git add -A stages deletions AND new files, then Git detects the rename automatically
$ git add -A

$ git status
On branch feature/rename-payment-validator
Changes to be committed:
        renamed:    src/InterVision.Api/Services/PaymentValidator.cs -> src/InterVision.Api/Services/Payments/PaymentValidationService.cs
```

Git detected the rename because the content similarity exceeded 50%. This works as long as you haven't heavily edited the file at the same time as moving it — if you refactored more than ~50% of the content, Git may not auto-detect (see Section 5).

## Gotchas and Troubleshooting

### "git log path/to/NewFile.cs only shows commits after the rename"

**Root cause:** `git log` without `--follow` stops at path boundaries. The moment the filename changed, Git's default log considers it a different file with no earlier history.

```bash
# Wrong — truncates history at the rename boundary
$ git log --oneline -- src/Services/Payments/PaymentValidationService.cs
7d3a1b4 refactor: rename PaymentValidator to PaymentValidationService
```

**Fix:**

```bash
# Correct — --follow traces renames backward through the log
$ git log --follow --oneline -- src/Services/Payments/PaymentValidationService.cs
7d3a1b4 refactor: rename PaymentValidator to PaymentValidationService
3c89f10 fix: handle duplicate payment token edge case
a1d4788 feat: add Luhn validation to PaymentValidator
e22b401 chore: initial PaymentValidator scaffold
```

**Prevention:** Make `--follow` a habit for any file you know has been renamed. Alias `git lf` to `git log --follow --oneline --` (see Section 6).

---

### "renamed: ... (50%)" or Git not detecting the rename at all — shows as deleted + new file

**Root cause:** When you move a file AND make substantial edits in the same staging operation, the content similarity score drops below Git's rename detection threshold (default 50%). Git treats it as unrelated delete + add.

```bash
# This scenario — moved the file AND rewrote most of it at the same time
$ git status
        deleted:    src/Services/PaymentValidator.cs
        new file:   src/Services/Payments/PaymentValidationService.cs
```

**Fix — two options:**

```bash
# Option A: Commit the move first (before edits), then commit the edits
$ git mv src/Services/PaymentValidator.cs src/Services/Payments/PaymentValidationService.cs
$ git commit -m "refactor: move PaymentValidator to Payments folder (rename only)"
# Now make your content edits and commit them separately

# Option B: Lower the rename detection threshold for this operation
$ git diff --staged -M30  # detect renames at 30% similarity instead of 50%
# Or configure globally (use with caution — produces false positives)
$ git config diff.renameLimit 999
```

**Prevention:** Always separate structural moves from content edits into two commits. This also makes PR reviews cleaner — reviewers can skip the rename commit and focus on the content changes.

---

### `fatal: not under version control, source=...`

**Root cause:** You tried to `git mv` a file that Git isn't tracking — either a new untracked file or a path with a typo.

```bash
$ git mv src/Services/PaymntValidator.cs src/Services/Payments/PaymentValidationService.cs
fatal: not under version control, source=src/Services/PaymntValidator.cs
```

**Fix:**

```bash
# Verify the exact path Git knows about
$ git ls-files src/Services/ | grep -i payment
src/Services/PaymentValidator.cs

# Re-run with the correct path
$ git mv src/Services/PaymentValidator.cs src/Services/Payments/PaymentValidationService.cs
```

**Prevention:** Tab-complete the source path or run `git ls-files | grep <pattern>` before typing the `git mv` command.

---

### `git mv` succeeded but `git blame` in GitHub/Azure DevOps still shows a fresh history

**Root cause:** The web UI's blame view may not follow renames by default, or the similarity score was logged correctly in Git but the hosting platform's blame renderer doesn't honour it. This is a UI limitation, not a Git problem.

```bash
# Verify locally — if --follow works here, Git has the data; it's a platform rendering issue
$ git log --follow --oneline -- src/Services/Payments/PaymentValidationService.cs
# If this shows full history, Git is fine
```

**Fix:** On GitHub, use the "Blame" button on the file page and click "View blame prior to this change" at the rename commit. On Azure DevOps, use the Annotate view and step back through changesets manually. Alternatively, use `git blame` locally — it always follows with `git log --follow`.

**Prevention:** Document in your team's contribution guide that `git blame` history requires `--follow` in CLI and a manual step in the web UI for renamed files.

---

### Rename committed without `--follow` awareness, now `git bisect` skips file history

**Root cause:** `git bisect` operates on the full commit graph, not file paths — but if you're debugging a regression in a renamed file and not using `--follow`, your test script may reference the wrong path for older commits, causing bisect to misidentify the bad commit.

```bash
# Bisect test script referencing the new path — fails on old commits where path didn't exist yet
$ cat bisect-test.sh
#!/usr/bin/env bash
dotnet test --filter "PaymentValidationServiceTests" && exit 0 || exit 1

# Git reports the wrong commit as "bad" because the test fails to compile on pre-rename commits
```

**Fix:**

```bash
# Make the bisect script path-aware
#!/usr/bin/env bash
# Try new path, fall back to old path
if [ -f "src/Services/Payments/PaymentValidationService.cs" ]; then
  dotnet test --filter "PaymentValidationServiceTests" && exit 0 || exit 1
elif [ -f "src/Services/PaymentValidator.cs" ]; then
  dotnet test --filter "PaymentValidatorTests" && exit 0 || exit 1
else
  exit 125  # tells bisect to skip this commit
fi
```

**Prevention:** When running `git bisect` on a repo with known renames, keep a note of the rename commit hash and use `git bisect skip <rename-commit>` to avoid the path-change commit itself.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# In ~/.gitconfig

[alias]
  # log with follow — essential for renamed files
  lf  = log --follow --oneline --
  # log with follow and full graph
  lfg = log --follow --oneline --graph --decorate --
  # show renames in last N commits
  renames = log --diff-filter=R --summary --oneline
  # move file and show status immediately
  gmv = "!f() { git mv \"$1\" \"$2\" && git status; }; f"

# Shell alias for the most common pattern
alias gmv='git mv'
```

### Automation Opportunity

For large-scale restructuring (moving dozens of files at once), a script avoids the tedium of individual `git mv` calls:

```bash
#!/usr/bin/env bash
# scripts/git-bulk-move.sh
# Usage: ./scripts/git-bulk-move.sh src/OldDir/ src/NewDir/
# Moves all .cs files from source dir to dest dir, preserving git history.
# Idempotent: skips files already at destination.

set -euo pipefail

SRC_DIR="${1:?Usage: $0 <source-dir> <dest-dir>}"
DST_DIR="${2:?Usage: $0 <source-dir> <dest-dir>}"

if [ ! -d "$DST_DIR" ]; then
  mkdir -p "$DST_DIR"
  echo "Created destination: $DST_DIR"
fi

find "$SRC_DIR" -maxdepth 1 -name "*.cs" | while read -r file; do
  filename=$(basename "$file")
  dest="$DST_DIR/$filename"
  if [ -f "$dest" ]; then
    echo "SKIP (already exists): $dest"
  else
    git mv "$file" "$dest"
    echo "Moved: $file -> $dest"
  fi
done

echo ""
echo "Staged renames:"
git status --short | grep "^R"
```

```bash
# Run it
$ chmod +x scripts/git-bulk-move.sh
$ ./scripts/git-bulk-move.sh \
    src/NotificationWorker/Handlers/ \
    src/NotificationWorker/Handlers/Email/
```

### Keyboard / CLI Efficiency Notes

- **Tab completion on `git mv`**: type `git mv src/Services/Pay<TAB>` — bash/zsh completes Git-tracked paths, not just filesystem paths. This catches typos before Git does.
- **`git status -s`** (short format): shows `R OldPath -> NewPath` in one line instead of the multi-line verbose output — faster to scan when moving many files.
- **`git diff --staged --stat`**: shows the rename summary with similarity scores before you commit — the single fastest way to confirm all renames were detected correctly.
- **`git log --follow --oneline -- <path>`**: add this as a post-commit sanity check step for any rename PR — run it once, confirm history, then push.

### Spoken Answers

**Q: What's the difference between `mv` and `git mv` when renaming a file?**

> **Average answer:** `git mv` stages the rename automatically so you don't have to `git add` it separately. `mv` just changes the file on disk and you have to add it yourself.

> **Great answer:** The staging difference is the mechanical part, but the important part is what Git records. Both paths _can_ end up with history preserved — Git uses content similarity detection at commit time, not at staging time — but `git mv` guarantees the rename is staged correctly as a single operation. With `mv` plus `git add -A`, you're relying on Git's auto-detection threshold (50% content similarity by default). If you also edited the file heavily at the same time, the similarity drops and Git records it as delete + add, which breaks `git log` without `--follow` and confuses `git blame`. The safe pattern is: `git mv` first, commit the move cleanly, then make content edits in a second commit. That way the rename commit has 100% similarity and history is unambiguous.

---

**Q: `git log OrderService.cs` only shows one week of history after a rename. What happened?**

> **Average answer:** You need to use `--follow`. `git log --follow -- OrderService.cs` will show the full history including before the rename.

> **Great answer:** By default `git log` follows file paths, not content identity. The moment the path changed from `OrderProcessor.cs` to `OrderService.cs`, `git log` on the new path starts a fresh history. `--follow` tells Git to detect the rename boundary and cross it — it finds the commit where similarity was recorded and continues following the old path backward from there. I'd run `git log --follow --oneline -- src/Services/OrderService.cs` to verify full history is there. If `--follow` _still_ doesn't show old commits, it means the rename was committed with similarity below 50% — most likely because code changes were bundled into the same commit as the rename. At that point the history really is broken in Git's model; the only fix is a history rewrite with `git filter-repo`, which requires force-pushing and coordinating with the team.

---

**Q: Walk me through renaming and moving 30 files across 5 directories in a single PR without losing history.**

> **Average answer:** Use `git mv` for each file instead of `mv`, then commit.

> **Great answer:** I'd do it in two commits, not one. First commit: all the moves and renames, no content changes — `git mv` each file to its new path, then commit with a message like `refactor: restructure X into domain subdirectories`. I specifically do not touch namespaces or usings yet, because mixing content edits with renames risks dropping similarity below the detection threshold. With zero content changes, every rename shows 100% similarity and history is guaranteed. Second commit: fix all the namespaces and using statements — `find . -name "*.cs" | xargs sed -i ...` or let the IDE do it, then stage and commit. This two-commit structure also makes the PR review far cleaner: reviewers can skip the first commit entirely and focus their attention on the namespace fix commit, where the actual logic could have gone wrong.

### War Story Prompt

Six months into a project, we restructured the `InventoryApi` service from a flat handler folder into a domain-grouped structure — about 40 files. The engineer doing the move ran `mv` and `git add .` rather than `git mv`, but also happened to update several namespaces in the same staging operation. ## Decision Framework

### When to Reach for `git mv`

- [ ] You are renaming a file and want `git blame` / `git log` history to follow it
- [ ] You are moving a file to a new directory (even without renaming)
- [ ] You are doing both (move + rename) in the same operation
- [ ] You want to keep the staging clean without a separate `git add`

### When NOT To

- [ ] **If you already ran `mv` and nothing else** — just run `git add -A` and let similarity detection work; `git mv` is no longer available for that file
- [ ] **If the file is untracked** — `git mv` fails on untracked files; just move it, then `git add`
- [ ] **If you're inside a pre-commit hook or script that already handles staging** — don't add a second `git mv` call that double-stages
- [ ] **If you're in detached HEAD state** — the rename will succeed but committing it is non-trivial; get back on a branch first (see [[9.044 — Detached HEAD State]])

### Tool Comparison

||`git mv`|`mv` + `git add -A`|
|---|---|---|
|History preservation|Guaranteed (staged as rename immediately)|Probable (depends on similarity ≥ 50% and no heavy content edits)|
|Safety when editing content simultaneously|High — forces you to do move and edit separately|Low — content edits mixed with move can drop similarity below threshold|
|Convenience|One command, done|Two commands, extra mental overhead|
|When it's the right call|Any intentional rename or move|When you've already moved with `mv` and can't undo it|

1. `git mv OldName.cs NewName.cs` — renames on disk and stages the rename as a single operation.
    
2. Without `--follow`: shows only commits where the file existed at the path `src/Services/NewName.cs` — history stops at the rename boundary. With `--follow`: crosses the rename boundary and shows commits from both the old and new paths, giving the complete history.
    
3. `--follow` — e.g. `git log --follow --oneline -- path/to/file.cs`.
    
4. 48% is the content similarity score Git calculated between the deleted file and the new file. 5. 50% — configurable via `git config diff.renameLimit` (controls the number of candidates checked, not the threshold itself) and `-M<n>%` flag on diff commands (e.g. `-M30` for 30% threshold).
    
6. Two commits: first commit moves/renames the file with zero content changes (`git mv`, then commit), ensuring 100% similarity; second commit makes the content edits. This guarantees history is preserved regardless of how much content changes.
    
7. When `git bisect` runs a test script across a commit range that spans a rename, the test script must reference the correct file path for each commit. On pre-rename commits the new path doesn't exist, so the test fails for the wrong reason and bisect misidentifies the bad commit. The fix is a path-aware bisect script that checks which path exists, or using `git bisect skip` on the rename commit itself. See [[9.030 — Git Bisect]] for the full bisect workflow.
    

</details>

**Drill 2 — Fix the broken thing**

```bash
$ mv src/PaymentGateway/Validators/CardValidator.cs \
     src/PaymentGateway/Validators/CreditCardValidator.cs

$ git add src/PaymentGateway/Validators/CreditCardValidator.cs

$ git status
Changes to be committed:
        new file:   src/PaymentGateway/Validators/CreditCardValidator.cs

Changes not staged for commit:
        deleted:    src/PaymentGateway/Validators/CardValidator.cs
```

Git is tracking this as a new file + a pending deletion. History will be broken if committed as-is.

<details> <summary>Solution</summary>

**Diagnosis:** The file was moved with `mv` (not `git mv`), and only the new file was staged — the deletion of the old path was not staged. Git therefore sees a new unrelated file and a pending unstaged deletion.

```bash
# Stage the deletion so Git can see both sides and run similarity detection
$ git add src/PaymentGateway/Validators/CardValidator.cs

$ git status
Changes to be committed:
        renamed:    src/PaymentGateway/Validators/CardValidator.cs -> src/PaymentGateway/Validators/CreditCardValidator.cs

# Git detected the rename (100% similarity — no content changes)
# Now commit cleanly
$ git commit -m "refactor: rename CardValidator to CreditCardValidator"
[main b3f2a11] refactor: rename CardValidator to CreditCardValidator
 1 file changed, 0 insertions(+), 0 deletions(-)
 rename src/PaymentGateway/Validators/{CardValidator.cs => CreditCardValidator.cs} (100%)
```

Key insight: `git add -A` stages both new files and deletions in one command — it's the safest catch-all when you've already used `mv` on the filesystem.

</details>

**Drill 3 — Under time pressure**

Production CI is failing because a PR renamed `InventoryApi/Services/StockChecker.cs` to `InventoryApi/Services/InventoryCheckService.cs` using `mv` + `git add .`, but also rewrote 80% of the file content in the same commit. Git recorded it as delete + add. The team lead wants the history fixed before the PR merges in 10 minutes. Walk through exactly what you type.

<details> <summary>Solution</summary>

```bash
# The history is broken in the current commit tree.
# Two options depending on whether the PR branch has been pushed (and whether force-push is allowed).

# OPTION A — branch not yet pushed (or force-push allowed):
# Amend or rebase-fixup to split the commit into two

# Step 1: Undo the last commit but keep the working tree changes
$ git reset HEAD~1
# Working tree now has: deleted StockChecker.cs + modified InventoryCheckService.cs

# Step 2: Save content edits to a temp file
$ cp src/InventoryApi/Services/InventoryCheckService.cs /tmp/InventoryCheckService.cs.bak

# Step 3: Restore the original file from the last good commit
$ git checkout HEAD -- src/InventoryApi/Services/StockChecker.cs

# Step 4: Commit the rename alone, zero content changes
$ git mv src/InventoryApi/Services/StockChecker.cs \
         src/InventoryApi/Services/InventoryCheckService.cs
$ git commit -m "refactor: rename StockChecker to InventoryCheckService (rename only)"
# Output: rename ... (100%) — history preserved

# Step 5: Apply the content edits
$ cp /tmp/InventoryCheckService.cs.bak \
     src/InventoryApi/Services/InventoryCheckService.cs
$ git add src/InventoryApi/Services/InventoryCheckService.cs
$ git commit -m "refactor: update InventoryCheckService implementation"

# Step 6: Force-push the rewritten branch
$ git push --force-with-lease origin feature/rename-stock-checker
# --force-with-lease is safer than --force — it refuses if someone else pushed since your last fetch

# OPTION B — already merged, force-push not possible:
# Document the break, add a note in the next commit referencing the rename commit hash,
# and use git log -S "method name" to manually trace history when needed.
# A full history rewrite (git filter-repo) is possible but requires team coordination.
```

</details>
