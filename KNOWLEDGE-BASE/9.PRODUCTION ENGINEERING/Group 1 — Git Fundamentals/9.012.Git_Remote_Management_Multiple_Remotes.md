### Where This Fits

Three situations force you into multi-remote territory: you forked an open-source repo and need to pull upstream changes while pushing to your own fork; your team mirrors the main repo to an Azure DevOps backup and CI needs to push to both; or you're migrating a repo from one hosting platform to another and need both remotes live simultaneously during cutover. In each case, the default `origin` isn't enough, and knowing exactly how to add, rename, inspect, and remove remotes — and how to fetch, push, and set tracking branches across them — is what keeps you unblocked.

---

## What You're Doing and Why

A Git remote is just a named URL stored in `.git/config`. The name `origin` is a convention, not a requirement — you can have as many named remotes as you want, each pointing to a different hosting location. `git remote` is the command that manages those entries: adding new ones, renaming them, changing their URLs, and removing stale ones.

The most common multi-remote patterns are: **fork + upstream** (your fork is `origin`, the canonical repo is `upstream`), **mirror push** (push to two registries simultaneously), and **platform migration** (old host is `old-origin`, new host is `origin`, you drain traffic from one to the other). Each pattern uses the same underlying `git remote` primitives — the difference is in which remote you fetch from and which you push to.

> For the policy context of when a fork workflow is the right team choice, see [[7.917 — Trunk-Based Development]]. This note covers the mechanics of configuring and operating multiple remotes.

### Mental Model

Think of remotes as entries in a contacts list. `git remote add upstream <url>` is adding a new contact. `git fetch upstream` is calling that contact to get their latest news. `git push origin main` is sending your work to a specific contact. The contacts list lives in `.git/config` — it's a plain text file you can read and edit directly, though the `git remote` commands are the safe interface for doing so.

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`git remote -v`|List all remotes with their fetch and push URLs|First thing to run when diagnosing any remote issue|
|`git remote add <name> <url>`|Register a new remote|Adding `upstream` after forking, or adding a mirror|
|`git remote rename <old> <new>`|Rename a remote|Renaming `origin` to `old-origin` during a migration|
|`git remote set-url <name> <url>`|Change the URL of an existing remote|After a repo is moved to a new host or renamed|
|`git remote remove <name>`|Delete a remote entry|Cleaning up after a migration is complete|
|`git remote show <name>`|Full detail on a remote — tracked branches, HEAD, stale refs|Auditing what a remote knows about|
|`git fetch <name>`|Download all refs from a named remote without merging|Checking upstream for new commits|
|`git fetch --all`|Fetch from every configured remote at once|Syncing all remotes in one pass|
|`git push <name> <branch>`|Push a branch to a specific named remote|Pushing to `upstream` vs `origin` explicitly|
|`git branch -u <name>/<branch>`|Set the upstream tracking branch for the current local branch|Fixing "no tracking information" warnings|
|`git remote prune <name>`|Remove local refs to branches deleted on the remote|Cleaning up stale `origin/feature/X` refs|

## Step-by-Step Walkthrough

### The Scenario

You've forked the `InterVision/order-platform` repo on GitHub to work on a vendor integration feature. Your fork lives at `your-org/order-platform`. You need to: add the canonical repo as `upstream`, fetch its latest `main`, rebase your feature branch onto it, push your work to your fork (`origin`), and keep the two remotes clearly separated so you never accidentally push to the wrong one.

### Full Walkthrough

**Part A — Inspect the starting state after cloning your fork**

```bash
$ git clone git@github.com:your-org/order-platform.git
Cloning into 'order-platform'...
remote: Enumerating objects: 1847, done.
remote: Counting objects: 100% (1847/1847), done.
remote: Compressing objects: 100% (612/612), done.
Receiving objects: 100% (1847/1847), 4.23 MiB | 8.11 MiB/s, done.
Resolving deltas: 100% (1204/1204), done.

$ cd order-platform

$ git remote -v
origin  git@github.com:your-org/order-platform.git (fetch)
origin  git@github.com:your-org/order-platform.git (push)
```

Only `origin` (your fork) — the canonical repo is not registered yet.

---

**Part B — Add the canonical repo as `upstream`**

```bash
$ git remote add upstream git@github.com:InterVision/order-platform.git

$ git remote -v
origin    git@github.com:your-org/order-platform.git (fetch)
origin    git@github.com:your-org/order-platform.git (push)
upstream  git@github.com:InterVision/order-platform.git (fetch)
upstream  git@github.com:InterVision/order-platform.git (push)
```

You now have two named remotes. The name `upstream` is conventional for the source-of-truth repo in a fork workflow — any name works, but deviating from this convention costs you readability.

---

**Part C — Fetch from upstream without affecting your local branches**

```bash
$ git fetch upstream
remote: Enumerating objects: 23, done.
remote: Counting objects: 100% (23/23), done.
remote: Compressing objects: 100% (11/11), done.
Unpacking objects: 100% (14/14), done.
From github.com:InterVision/order-platform
 * [new branch]      main                    -> upstream/main
 * [new branch]      feature/payment-gateway -> upstream/feature/payment-gateway
 * [new tag]         v2.4.1                  -> v2.4.1
```

`git fetch upstream` downloads all refs from the canonical repo into the `upstream/*` remote-tracking namespace. Your local `main` branch is untouched.

```bash
$ git branch -a
* feature/vendor-integration
  main
  remotes/origin/HEAD -> origin/main
  remotes/origin/main
  remotes/origin/feature/vendor-integration
  remotes/upstream/main
  remotes/upstream/feature/payment-gateway
```

`remotes/upstream/main` is now a local reference to the canonical repo's `main` — you can diff against it, rebase onto it, or check it out.

---

**Part D — Rebase your feature branch onto upstream/main**

```bash
$ git checkout feature/vendor-integration
Already on 'feature/vendor-integration'

$ git rebase upstream/main
Successfully rebased and updated refs/heads/feature/vendor-integration.
```

Your feature branch commits are now replayed on top of the canonical repo's latest `main`. No merge commit, no noise in your fork's PR.

---

**Part E — Push only to your fork (origin), never to upstream**

```bash
$ git push origin feature/vendor-integration
Enumerating objects: 9, done.
Counting objects: 100% (9/9), done.
Delta compression using up to 10 threads
Compressing objects: 100% (6/6), done.
Writing objects: 100% (6/6), 1.87 KiB | 1.87 MiB/s, done.
Total 6 (delta 3), reused 0 (delta 0), pack-reused 0
To github.com:your-org/order-platform.git
 * [new branch]      feature/vendor-integration -> feature/vendor-integration
```

Explicit remote name (`origin`) prevents any ambiguity. If you had push access to `upstream` (you likely don't in an open-source fork, but you might in an internal fork workflow), naming the remote explicitly is what keeps you from accidentally pushing to the wrong place.

---

**Part F — Prevent accidental pushes to upstream by making it fetch-only**

```bash
$ git remote set-url --push upstream DISABLE

$ git remote -v
origin    git@github.com:your-org/order-platform.git (fetch)
origin    git@github.com:your-org/order-platform.git (push)
upstream  git@github.com:InterVision/order-platform.git (fetch)
upstream  DISABLE (push)
```

Now `git push upstream` will fail with a "repository not found" error — an intentional hard stop that prevents accidents.

```bash
$ git push upstream main
fatal: 'DISABLE' does not appear to be a git repository
fatal: Could not read from remote repository.
```

### What Could Go Wrong Mid-Walkthrough

**If `git rebase upstream/main` hits conflicts:**

```bash
$ git rebase upstream/main
Auto-merging src/OrderService/VendorClient.cs
CONFLICT (content): Merge conflict in src/OrderService/VendorClient.cs
error: could not apply 3f8a21c... feat: add VendorClient retry policy
hint: Resolve all conflicts manually, mark them as fixed with
hint: "git add/rm <conflicted_files>", then run "git rebase --continue".
```

Resolve the conflict in the file, then:

```bash
$ git add src/OrderService/VendorClient.cs
$ git rebase --continue
# Editor may open for the commit message — save and close
```

To abort the rebase entirely and return to the pre-rebase state:

```bash
$ git rebase --abort
```

See [[9.028 — Resolving Rebase Conflicts Step-by-Step]] for the full procedure.

## Production Scenario

### Realistic Context

The `InterVision` engineering team is migrating the `PaymentGateway` repo from Azure DevOps (`dev.azure.com/intervision/PaymentGateway`) to GitHub (`github.com/intervision/payment-gateway`). The migration must be zero-downtime: the Azure DevOps repo stays live as the source of truth until the GitHub repo is confirmed healthy and all pipelines have been redirected. During the cutover window, engineers need both remotes active simultaneously, with a mirror-push step in CI that writes every commit to both hosts. After sign-off, `azure-devops` gets removed and `origin` is re-pointed to GitHub permanently.

### Full Implementation / Resolution

```bash
# Step 1 — Clone from the current source-of-truth (Azure DevOps)
$ git clone git@ssh.dev.azure.com:v3/intervision/PaymentGateway/PaymentGateway payment-gateway
Cloning into 'payment-gateway'...
remote: Azure Repos
remote: Found 2341 objects to send. (145 ms)
Receiving objects: 100% (2341/2341), 11.4 MiB | 12.3 MiB/s, done.

$ cd payment-gateway

$ git remote -v
origin  git@ssh.dev.azure.com:v3/intervision/PaymentGateway/PaymentGateway (fetch)
origin  git@ssh.dev.azure.com:v3/intervision/PaymentGateway/PaymentGateway (push)
```

```bash
# Step 2 — Rename origin to reflect the source clearly during migration
$ git remote rename origin azure-devops

$ git remote -v
azure-devops  git@ssh.dev.azure.com:v3/intervision/PaymentGateway/PaymentGateway (fetch)
azure-devops  git@ssh.dev.azure.com:v3/intervision/PaymentGateway/PaymentGateway (push)
```

Renaming makes intent explicit — everyone working in this repo during cutover knows which remote is which.

```bash
# Step 3 — Add the new GitHub remote as origin
$ git remote add origin git@github.com:intervision/payment-gateway.git

$ git remote -v
azure-devops  git@ssh.dev.azure.com:v3/intervision/PaymentGateway/PaymentGateway (fetch)
azure-devops  git@ssh.dev.azure.com:v3/intervision/PaymentGateway/PaymentGateway (push)
origin        git@github.com:intervision/payment-gateway.git (fetch)
origin        git@github.com:intervision/payment-gateway.git (push)
```

```bash
# Step 4 — Push all branches and tags to GitHub
$ git push origin --all
Enumerating objects: 2341, done.
Counting objects: 100% (2341/2341), done.
Delta compression using up to 10 threads
Compressing objects: 100% (1834/1834), done.
Writing objects: 100% (2341/2341), 11.4 MiB | 9.2 MiB/s, done.
To github.com:intervision/payment-gateway.git
 * [new branch]      main -> main
 * [new branch]      release/2.4 -> release/2.4
 * [new branch]      release/2.5 -> release/2.5

$ git push origin --tags
To github.com:intervision/payment-gateway.git
 * [new tag]         v2.4.0 -> v2.4.0
 * [new tag]         v2.4.1 -> v2.4.1
 * [new tag]         v2.5.0-rc1 -> v2.5.0-rc1
```

```bash
# Step 5 — Set up a push URL on azure-devops to also mirror to GitHub
# (used in CI to dual-write during the cutover window)
# In CI pipeline — push explicitly to both after every merge to main:

$ git push azure-devops main
$ git push origin main
```

For a one-command dual-push, configure a push URL list on a dedicated `mirrors` remote:

```bash
$ git remote add mirrors git@ssh.dev.azure.com:v3/intervision/PaymentGateway/PaymentGateway
$ git remote set-url --add --push mirrors git@github.com:intervision/payment-gateway.git

$ git remote -v
mirrors  git@ssh.dev.azure.com:v3/intervision/PaymentGateway/PaymentGateway (fetch)
mirrors  git@ssh.dev.azure.com:v3/intervision/PaymentGateway/PaymentGateway (push)
mirrors  git@github.com:intervision/payment-gateway.git (push)
```

Now `git push mirrors main` pushes to both URLs in one command.

```bash
# Step 6 — After cutover is confirmed: remove azure-devops, leave only origin
$ git remote remove azure-devops
$ git remote remove mirrors

$ git remote -v
origin  git@github.com:intervision/payment-gateway.git (fetch)
origin  git@github.com:intervision/payment-gateway.git (push)
```

Migration complete. All engineers on the team need to update their local clone's remotes — provide them with the one-liner:

```bash
$ git remote set-url origin git@github.com:intervision/payment-gateway.git
```

## Gotchas and Troubleshooting

### "fatal: 'upstream' does not appear to be a git repository"

**Root cause:** The remote name `upstream` was typed in a command but was never registered with `git remote add`. This often happens when someone copies a command from documentation without checking whether their repo has that remote configured.

```bash
$ git fetch upstream
fatal: 'upstream' does not appear to be a git repository
fatal: Could not read from remote repository.
```

**Fix:**

```bash
# First, check what remotes actually exist
$ git remote -v
origin  git@github.com:your-org/order-platform.git (fetch)
origin  git@github.com:your-org/order-platform.git (push)
# upstream is absent — add it
$ git remote add upstream git@github.com:InterVision/order-platform.git
$ git fetch upstream
```

**Prevention:** Always run `git remote -v` at the start of any forked-repo session to confirm the expected remotes are present. Add this to your onboarding checklist for new contributors.

---

### "error: remote origin already exists"

**Root cause:** You ran `git remote add origin <url>` on a repo that was cloned (which automatically creates `origin`), or you ran `add` twice with the same name.

```bash
$ git remote add origin git@github.com:intervision/payment-gateway.git
error: remote origin already exists.
```

**Fix — update the URL of the existing remote instead:**

```bash
$ git remote set-url origin git@github.com:intervision/payment-gateway.git
# Confirm
$ git remote -v
origin  git@github.com:intervision/payment-gateway.git (fetch)
origin  git@github.com:intervision/payment-gateway.git (push)
```

**Prevention:** Use `git remote set-url` when changing a remote URL, and `git remote add` only when adding a _new_ name. They are not interchangeable.

---

### "Your branch has no tracking information"

**Root cause:** You created a local branch manually (`git checkout -b feature/x`) rather than checking it out from a remote, and no tracking upstream was ever set. When multiple remotes exist, Git can't infer where to push or pull.

```bash
$ git pull
There is no tracking information for the current branch.
Please specify which branch you want to merge with.
See git-pull(1) for details.

    git pull <remote> <branch>
```

**Fix — set the tracking branch explicitly:**

```bash
$ git branch -u origin/feature/vendor-integration
Branch 'feature/vendor-integration' set up to track remote branch 'feature/vendor-integration' from 'origin'.
```

Or set it during a push:

```bash
$ git push -u origin feature/vendor-integration
```

**Prevention:** Always use `git push -u origin <branch>` on the first push of a new branch. The `-u` flag sets the tracking relationship in one step and prevents this error on all future `git pull` calls on that branch.

---

### Push goes to the wrong remote

**Root cause:** When `git push` is run without a remote name and the tracking branch is configured against a remote you didn't intend (e.g., `upstream` instead of `origin`), Git silently pushes to the wrong place.

```bash
$ git push
# Appears to succeed, but goes to upstream instead of origin
Enumerating objects: 5, done.
To github.com:InterVision/order-platform.git
   a1b2c3d..9f3e21c  feature/vendor-integration -> feature/vendor-integration
```

**Fix — push explicitly to the correct remote:**

```bash
$ git push origin feature/vendor-integration
```

**And remove the wrong tracking relationship:**

```bash
$ git branch -u origin/feature/vendor-integration
```

**Prevention:** In multi-remote repos, always name the remote explicitly in push commands. Set `upstream` to push-disabled (`git remote set-url --push upstream DISABLE`) so accidental pushes to the canonical repo fail loudly. Never rely on the implicit tracking remote in a multi-remote context.

---

### Stale remote-tracking branches after upstream deletes a branch

**Root cause:** A branch was deleted on the remote, but your local `remotes/upstream/feature/old-branch` ref still exists. `git branch -a` shows branches that no longer exist on the server.

```bash
$ git branch -a
  remotes/upstream/feature/old-feature-x    # deleted on upstream weeks ago
  remotes/upstream/feature/old-feature-y    # deleted on upstream months ago
  remotes/upstream/main
```

**Fix:**

```bash
$ git remote prune upstream
Pruning origin
URL: git@github.com:InterVision/order-platform.git
 * [pruned] upstream/feature/old-feature-x
 * [pruned] upstream/feature/old-feature-y
```

**Prevention:** Run `git fetch --prune upstream` (or `git fetch -p upstream`) instead of bare `git fetch upstream` — `--prune` automatically removes stale remote-tracking refs as part of the fetch. Set it globally:

```bash
$ git config --global fetch.prune true
```

---

### `.git/config` shows duplicate or conflicting remote entries after a bad `set-url`

**Root cause:** Manual editing of `.git/config`, or a failed `git remote` command, left duplicate `[remote "origin"]` sections.

```bash
$ cat .git/config
[remote "origin"]
    url = git@github.com:your-org/order-platform.git
    fetch = +refs/heads/*:refs/remotes/origin/*
[remote "origin"]
    url = git@github.com:InterVision/order-platform.git
    fetch = +refs/heads/*:refs/remotes/origin/*
```

**Fix — remove the duplicate manually:**

```bash
$ git remote remove origin
$ git remote add origin git@github.com:your-org/order-platform.git
```

Or edit `.git/config` directly — remove the duplicate `[remote "origin"]` block entirely. Git reads the first matching section; duplicates cause unpredictable behaviour.

**Prevention:** Never edit `.git/config` manually for remote management. Use `git remote add`, `set-url`, `rename`, and `remove` — they validate and write the config atomically.

---

### "Updates were rejected because the remote contains work that you do not have locally"

**Root cause:** In a multi-remote workflow, you pushed to `origin` from machine A, then switched to machine B which still has the old tracking state. When machine B tries to push, the remote has commits it doesn't know about.

```bash
$ git push origin main
To github.com:your-org/order-platform.git
 ! [rejected]        main -> main (fetch first)
error: failed to push some refs to 'github.com:your-org/order-platform.git'
hint: Updates were rejected because the remote contains work that you do not have locally.
```

**Fix:**

```bash
$ git fetch origin
$ git rebase origin/main    # or git merge origin/main if your team prefers merge
$ git push origin main
```

**Prevention:** Always `git fetch origin` before starting work in any session. In multi-remote setups, this is even more important because changes can arrive from multiple directions.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# In ~/.gitconfig

[alias]
  # Show all remotes with URLs — first thing to run when something seems wrong
  rv     = remote -v

  # Fetch all remotes and prune stale branches in one pass
  fall   = fetch --all --prune

  # Fetch upstream specifically (fork workflow)
  fu     = fetch upstream

  # Rebase current branch onto upstream/main (fork workflow daily loop)
  rbup   = rebase upstream/main

  # Push to origin with tracking set (safe first-push of any new branch)
  puo    = push -u origin HEAD

  # Show detailed remote info including tracked branches and stale refs
  rshow  = "!f() { git remote show \"${1:-origin}\"; }; f"
```

Usage:

```bash
$ git rv              # quick sanity check
$ git fall            # sync everything + clean stale refs
$ git fu && git rbup  # morning sync in a fork workflow
$ git puo             # push current branch to origin with tracking
```

### Automation Opportunity

A daily fork-sync script that keeps your local `main` and feature branches current with `upstream/main` without manual steps:

```bash
#!/usr/bin/env bash
# scripts/sync-fork.sh
# Syncs local fork with upstream/main and optionally rebases a feature branch.
# Usage: ./scripts/sync-fork.sh [feature-branch-name]

set -euo pipefail

FEATURE_BRANCH="${1:-}"

echo "==> Fetching upstream..."
git fetch upstream --prune

echo "==> Updating local main from upstream/main..."
git checkout main
git reset --hard upstream/main
git push origin main

echo "==> Local main is now at: $(git log --oneline -1)"

if [[ -n "$FEATURE_BRANCH" ]]; then
  echo "==> Rebasing $FEATURE_BRANCH onto upstream/main..."
  git checkout "$FEATURE_BRANCH"
  git rebase upstream/main
  echo "==> Pushing rebased $FEATURE_BRANCH to origin..."
  git push --force-with-lease origin "$FEATURE_BRANCH"
  echo "==> Done. $FEATURE_BRANCH is current."
fi
```

```bash
# Morning workflow for fork contributors:
$ ./scripts/sync-fork.sh feature/vendor-integration
```

### Keyboard / CLI Efficiency Notes

- `git remote -v` is three keystrokes with the `rv` alias — run it every time you sit down in an unfamiliar repo before touching anything
- `git fetch --all --prune` is the one-command equivalent of running fetch + prune across every registered remote — worth the `fall` alias
- `git remote show origin` (or `upstream`) gives you the full picture: tracked branches, push/pull URLs, and which local branches are stale relative to the remote — far more informative than `-v` alone, and underused
- When configuring a push-disabled upstream, `DISABLE` is idiomatic but any non-existent URL works — some teams use `no-push` for clarity in `git remote -v` output

### Spoken Answers

**Q: Walk me through the exact commands to set up a fork workflow from scratch.**

> **Average answer:** "Clone your fork, then `git remote add upstream <url>`, then `git fetch upstream`, then rebase."

> **Great answer:** "After cloning my fork, I immediately run `git remote -v` to confirm `origin` points to my fork — not the canonical repo. Then `git remote add upstream <canonical-url>`. At this point I also lock down the push URL on upstream: `git remote set-url --push upstream DISABLE`. That's a hard stop that prevents me from ever accidentally pushing to the source of truth. Then `git fetch upstream` to pull down all the remote-tracking refs. From there, my morning workflow is `git fetch upstream --prune` to stay current and prune deleted branches, then `git rebase upstream/main` on whatever feature branch I'm working on. I push only to `origin`, always with the remote named explicitly — `git push origin <branch>` — never a bare `git push` in a multi-remote repo."

---

**Q: You're migrating a repo from Azure DevOps to GitHub with zero downtime. Walk me through the multi-remote strategy.**

> **Average answer:** "Add GitHub as a new remote, push everything there, then switch over."

> **Great answer:** "I'd rename `origin` to `azure-devops` first so the intent is visible in every engineer's terminal. Then `git remote add origin <github-url>`. Push all branches with `git push origin --all` and all tags with `git push origin --tags`. During the cutover window, I'd set up a `mirrors` remote with two push URLs — one pointing to Azure DevOps, one to GitHub — so CI can dual-write with a single push command. I'd verify the branch and tag counts match between remotes before proceeding. Once GitHub pipelines are confirmed green and all teams have updated their configs, I remove `azure-devops` and `mirrors`, and the remaining engineers update their local clone with `git remote set-url origin <github-url>`. The key is that nothing is destroyed until the new host is verified — both remotes coexist through the entire transition."

---

**Q: After a month in a fork, `git branch -a` shows dozens of stale `remotes/upstream/*` branches. How do you clean them up and prevent recurrence?**

> **Average answer:** "Run `git remote prune upstream`."

> **Great answer:** "`git remote prune upstream` removes all local refs to branches that no longer exist on the upstream remote — I'd run that first to clear the backlog. To prevent recurrence, I'd replace bare `git fetch upstream` with `git fetch upstream --prune` in my daily workflow and aliases, and set `git config --global fetch.prune true` so pruning is automatic on every fetch across all remotes. I'd also add this to the team's onboarding docs — stale remote-tracking refs are harmless but they make `git branch -a` output unreadable at scale, which slows down branch discovery."

### War Story Prompt

A team migrating their `NotificationWorker` service from Bitbucket to Azure DevOps kept both remotes live for a two-week cutover. Midway through, a junior engineer ran `git push` without specifying a remote — their local branch was still tracking `origin` (Bitbucket), so the push went to the old host instead of the new one. A hotfix landed in Bitbucket but not Azure DevOps, and the Azure DevOps pipeline never triggered. The fix was live in production (deployed from Bitbucket) but invisible to the team's new CI system. The incident was discovered three days later during a code review when a reviewer noticed the fix wasn't in the Azure DevOps history. After that, the team made `DISABLE` the push URL on the old remote as soon as the migration push was complete — so any accidental push to the old host would fail loudly rather than silently succeed.

## Decision Framework

### When to Reach for This

- [ ] You've forked a repo and need to pull changes from the canonical source while pushing work to your fork
- [ ] You're migrating a repo between hosting platforms and need both live simultaneously during cutover
- [ ] CI needs to mirror-push to two registries (e.g., Azure Container Registry + GitHub Packages) from a single pipeline step
- [ ] A team member needs to contribute to a repo they don't have push access to — fork + upstream is the correct model
- [ ] You need to fetch from one org's repo (read access) while pushing to another (write access)

### When NOT To

- [ ] You just want to push to a different URL than what `origin` is currently set to — use `git remote set-url origin <new-url>` instead of adding a second remote
- [ ] You're working in a single-remote team repo with standard branch protection — multiple remotes adds complexity for no benefit
- [ ] You want to test a commit in a throwaway location — use a local branch and a temporary `git stash` rather than a second remote

### Tool Comparison

||Single `origin`|Fork + `upstream`|Mirror remote (dual-push URLs)|
|---|---|---|---|
|Setup complexity|Minimal — clone and go|Low — one `git remote add`|Medium — requires `set-url --add --push`|
|Push safety|Relies on branch protection|Push only to fork; upstream is fetch-only|Push to all URLs simultaneously — risky if misconfigured|
|When it's the right call|Team repo with direct push access|Open-source contribution or internal fork workflow|Platform migration or mandatory backup writes|
|Risk|Accidental force-push to shared branch|Accidentally pushing to upstream if not DISABLE-d|Partial push failure (one URL succeeds, one fails) leaves remotes diverged|

## Self-Check

### Conceptual Questions

1. Where does Git physically store remote definitions, and what does an entry look like?
2. What is the difference between `git remote set-url origin <url>` and `git remote add origin <url>`?
3. How do you configure `upstream` so that `git fetch upstream` works but `git push upstream` always fails?
4. After running `git fetch upstream`, what is the exact ref path where upstream's `main` is stored locally?
5. What does `git remote prune origin` do, and how is it different from `git fetch --prune origin`?
6. You have two remotes and run a bare `git push`. Which remote does Git push to, and how does it decide?
7. How do you add a second push URL to an existing remote so that one `git push` command writes to two locations? (Connect to the mirror pattern in [[9.333 — Scripting a Full Environment Setup]])

<details> <summary>Answers</summary>

1. In `.git/config`. Each remote is a `[remote "name"]` section with `url` and `fetch` keys. Example:
    
    ```
    [remote "upstream"]
        url = git@github.com:InterVision/order-platform.git
        fetch = +refs/heads/*:refs/remotes/upstream/*
    ```
    
2. `git remote add` creates a new entry — it fails if the name already exists. `git remote set-url` updates the URL of an _existing_ entry — it fails if the name does _not_ exist. Using `add` when you should use `set-url` gives `error: remote origin already exists`; using `set-url` when you should use `add` gives `fatal: No such remote 'upstream'`.
    
3. `git remote set-url --push upstream DISABLE`. This sets a separate push URL (the `pushurl` key in `.git/config`) to a non-existent value. `git fetch upstream` reads the `url` key and succeeds; `git push upstream` reads `pushurl` and fails with `fatal: 'DISABLE' does not appear to be a git repository`.
    
4. `refs/remotes/upstream/main` — or equivalently referred to as `upstream/main` in most Git commands. It's a file at `.git/refs/remotes/upstream/main` containing the SHA of the last-fetched tip of `upstream`'s `main` branch.
    
5. `git remote prune origin` removes stale local remote-tracking refs (branches that were deleted on the remote) but does _not_ fetch new commits. `git fetch --prune origin` does both in one pass: fetches new commits and objects, _and_ prunes stale refs. Prefer `fetch --prune` in practice — `remote prune` is mainly useful when you want to clean up without downloading anything.
    
6. Git pushes to the tracking remote of the current branch — whatever is stored in the `[branch "name"]` section's `remote` key in `.git/config`. If no tracking remote is set, Git errors with "no tracking information." In a multi-remote repo, always name the remote explicitly in push commands rather than relying on this implicit lookup.
    
7. `git remote set-url --add --push <name> <second-url>`. This appends a second `pushurl` entry to the remote section in `.git/config`. When you run `git push <name>`, Git pushes to all listed `pushurl` values in sequence. Fetch still reads the single `url`. This is the mechanism behind the `mirrors` remote pattern in the production scenario above.
    

</details>

### Hands-On Drills

**Drill 1 — Do it from a clean state**

Starting from a fresh clone of any GitHub repo you own (or a local bare repo created with `git init --bare`), configure it to have two remotes: `origin` pointing to its original location and `backup` pointing to a second location (can be another local bare repo at `/tmp/backup-repo.git`). Push all branches and tags to `backup`, then set `backup`'s push URL to `DISABLE` and verify that `git push backup main` fails with the expected error while `git fetch backup` still succeeds.

<details> <summary>Solution</summary>

```bash
# Create a backup bare repo to simulate a second remote
$ git init --bare /tmp/backup-repo.git
Initialized empty Git repository in /tmp/backup-repo.git/

# Clone the source repo
$ git clone git@github.com:your-org/order-platform.git drill-remote && cd drill-remote

# Confirm starting state
$ git remote -v
origin  git@github.com:your-org/order-platform.git (fetch)
origin  git@github.com:your-org/order-platform.git (push)

# Add the backup remote
$ git remote add backup /tmp/backup-repo.git

$ git remote -v
backup  /tmp/backup-repo.git (fetch)
backup  /tmp/backup-repo.git (push)
origin  git@github.com:your-org/order-platform.git (fetch)
origin  git@github.com:your-org/order-platform.git (push)

# Push all branches and tags to backup
$ git push backup --all
Enumerating objects: ...
To /tmp/backup-repo.git
 * [new branch]      main -> main

$ git push backup --tags
# (tags pushed if any exist)

# Disable push to backup
$ git remote set-url --push backup DISABLE

$ git remote -v
backup  /tmp/backup-repo.git (fetch)
backup  DISABLE (push)
origin  git@github.com:your-org/order-platform.git (fetch)
origin  git@github.com:your-org/order-platform.git (push)

# Verify push fails
$ git push backup main
fatal: 'DISABLE' does not appear to be a git repository
fatal: Could not read from remote repository.

# Verify fetch still works
$ git fetch backup
From /tmp/backup-repo.git
 * branch            main     -> FETCH_HEAD
# Success — fetch reads the url key, not pushurl
```

</details>

**Drill 3 — Under time pressure**

A colleague just told you that they accidentally pushed a breaking change directly to `upstream/main` (your team has direct push access to the canonical repo). You need to identify which commit it was, revert it on `upstream/main`, and push the revert — all within 3 minutes before the next CI deploy trigger fires. You're in a repo that has both `origin` (your fork) and `upstream` (canonical) configured.

<details> <summary>Solution</summary>

```bash
# 1 — Fetch the latest state of upstream immediately
$ git fetch upstream
From github.com:InterVision/order-platform
   a1b2c3d..f9e3b21  main -> upstream/main

# 2 — Find the breaking commit
$ git log --oneline upstream/main -5
f9e3b21 (upstream/main) feat: experimental payment routing (DO NOT MERGE — WIP)
a1b2c3d fix: correct currency rounding in PaymentGateway
...
# f9e3b21 is the accidental commit

# 3 — Check it out to a local tracking branch so you can push
$ git checkout -b hotfix/revert-accidental main
$ git reset --hard upstream/main   # bring local up to date with upstream first

# 4 — Revert the bad commit
$ git revert f9e3b21 --no-edit
[hotfix/revert-accidental 2c8a91b] Revert "feat: experimental payment routing (DO NOT MERGE — WIP)"
 2 files changed, 8 deletions(-)

# 5 — Push the revert directly to upstream/main
$ git push upstream hotfix/revert-accidental:main
Total 3 (delta 1), reused 0 (delta 0), pack-reused 0
To github.com:InterVision/order-platform.git
   f9e3b21..2c8a91b  hotfix/revert-accidental -> main

# 6 — Verify
$ git fetch upstream
$ git log --oneline upstream/main -3
2c8a91b (upstream/main) Revert "feat: experimental payment routing (DO NOT MERGE — WIP)"
f9e3b21 feat: experimental payment routing (DO NOT MERGE — WIP)
a1b2c3d fix: correct currency rounding in PaymentGateway
```

**Why `push upstream hotfix/revert-accidental:main`:** The `<local-branch>:<remote-branch>` refspec lets you push a local branch to a differently-named remote branch without checking out the remote branch locally. This is faster than checking out `main` locally and pulling — you stay on your hotfix branch and push the revert directly. The `upstream` push URL must not be set to `DISABLE` for this to work — temporarily restore it if needed, then re-disable after the emergency is resolved.

</details>
