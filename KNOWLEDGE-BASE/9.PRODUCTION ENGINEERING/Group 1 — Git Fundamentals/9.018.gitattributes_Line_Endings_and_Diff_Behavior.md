### Where This Fits

A Windows developer clones `InventoryApi`, opens a `.cs` file in VS Code, saves it without changing anything, and `git diff` explodes with hundreds of lines changed — every single line. Or: a teammate's PR shows a two-line logic fix but the diff is 400 lines long because their editor silently converted the entire file from LF to CRLF. Or: `git blame` on a shell script used in CI is useless because a line-ending normalisation commit touched every line six months ago. All three of these are `.gitattributes` problems — and all three are preventable with a correctly authored `.gitattributes` in the repo root that you write once and never touch again.

## What You're Doing and Why

`.gitattributes` is a per-repo configuration file that tells Git how to handle files on checkout (what line endings to write to disk) and on add/commit (what to store in the object store). It overrides every developer's personal `core.autocrlf` setting, making line-ending behaviour consistent regardless of OS or editor. Beyond line endings, `.gitattributes` also controls which diff driver Git uses per file type — marking binary files as binary so Git stops trying to text-diff them, assigning language-aware diff drivers to `.cs` or `.csproj` files so `git diff` shows method names in hunk headers instead of line numbers, and controlling merge strategy per path. The key insight is that `.gitattributes` is committed into the repo itself, so it applies identically to every contributor and every CI agent — no per-machine setup required.

### Mental Model

Think of `.gitattributes` as a customs manifest at the border between your working tree and the Git object store. Every file crossing the border inbound (checkout → disk) and outbound (disk → staging → object store) passes through customs. The manifest says: "text files get LF stored in the object store, and get converted to the local OS convention on checkout; shell scripts always stay LF even on Windows; PNG files get stamped BINARY so customs stops trying to read them as text." Without the manifest, each developer's machine applies its own rules — and the resulting chaos shows up in your diffs.

```
Working Tree (disk)          Git Object Store
───────────────────          ────────────────
CRLF on Windows  ──[checkout]──▶  LF (normalized)
LF on macOS/Linux             │
                   ◀──[add]───┘
```

### Command / Tool Quick Reference

|Command / Action|What It Does|When You Reach for It|
|---|---|---|
|`cat .gitattributes`|Reads current attribute rules|First step when diagnosing line-ending diffs|
|`git check-attr -a <file>`|Shows all attributes applied to a specific file|Verifying a rule actually matches the file|
|`git add --renormalize .`|Re-stages all tracked files applying current `.gitattributes` rules|After adding or changing `.gitattributes` to fix existing files|
|`git ls-files --eol`|Lists every tracked file with its index/working-tree EOL status|Auditing what's actually stored in the object store|
|`git diff --ignore-space-at-eol`|Diffs ignoring trailing EOL differences|One-shot view to confirm only real changes exist|
|`git config core.autocrlf`|Shows the per-machine CRLF setting|Diagnosing why a developer's checkout looks different|
|`git config --global core.autocrlf input`|Sets the recommended global for macOS/Linux|Per-machine fallback when no `.gitattributes` exists|
|`git config --global core.autocrlf false`|Disables CRLF conversion globally|Appropriate when `.gitattributes` handles everything|
|`git rm --cached -r .` then `git reset --hard`|Nuclear re-checkout: forces Git to re-apply `.gitattributes` to the whole tree|Last resort when `--renormalize` alone isn't enough|

## Step-by-Step Walkthrough

### The Scenario

The `InterVision` monorepo was started on macOS by two engineers and is now being cloned by a third developer on Windows. Before the Windows developer commits anything, you're setting up `.gitattributes` to ensure line endings are normalised in the object store (LF only) and shell scripts used in the CI pipeline are never converted to CRLF — which would break them. The repo contains C# source, `.csproj` files, YAML pipelines, Bash scripts, and PNG assets.

### Full Walkthrough

```bash
$ git status
On branch main
Your branch is up to date with 'origin/main'.
nothing to commit, working tree clean
```

Start from a clean working tree — you don't want uncommitted changes mixed in with the normalisation commit.

```bash
$ ls .gitattributes
ls: cannot access '.gitattributes': No such file or directory
```

Confirm there's no existing `.gitattributes` — this repo has been running on luck and `core.autocrlf` defaults.

```bash
$ cat > .gitattributes << 'EOF'
# Normalize all text files to LF in the object store.
# Git converts to OS-native on checkout (LF on macOS/Linux, CRLF on Windows)
# unless overridden below.
* text=auto eol=lf

# Explicit text files — always LF in the object store
*.cs        text eol=lf
*.csproj    text eol=lf
*.sln       text eol=lf
*.props     text eol=lf
*.targets   text eol=lf
*.json      text eol=lf
*.yaml      text eol=lf
*.yml       text eol=lf
*.md        text eol=lf
*.xml       text eol=lf
*.config    text eol=lf
*.toml      text eol=lf
*.tf        text eol=lf
*.bicep     text eol=lf
*.sql       text eol=lf
*.http      text eol=lf

# Shell scripts — MUST stay LF even when checked out on Windows
# (CRLF in a .sh file causes "bad interpreter: No such file or directory" in CI)
*.sh        text eol=lf
*.bash      text eol=lf
Makefile    text eol=lf

# PowerShell scripts — CRLF on Windows, LF in store
*.ps1       text eol=lf
*.psm1      text eol=lf
*.psd1      text eol=lf

# Dockerfile — always LF
Dockerfile  text eol=lf
*.dockerfile text eol=lf

# Binary files — tell Git not to attempt text conversion or text diffing
*.png       binary
*.jpg       binary
*.jpeg      binary
*.gif       binary
*.ico       binary
*.webp      binary
*.pdf       binary
*.zip       binary
*.nupkg     binary
*.dll       binary
*.exe       binary
*.pdb       binary
*.snupkg    binary
*.woff      binary
*.woff2     binary
*.ttf       binary
*.eot       binary

# Diff drivers — show method/class context in hunk headers
*.cs        diff=csharp
*.md        diff=markdown
EOF
```

Write the complete `.gitattributes`. The `* text=auto eol=lf` catch-all handles any file type not listed explicitly — Git auto-detects whether it's text and normalises to LF.

```bash
$ git add .gitattributes
$ git status
On branch main
Changes to be committed:
  new file:   .gitattributes
```

Stage the file before running `--renormalize` — the new rules must be in the index before Git can apply them.

```bash
$ git add --renormalize .
```

This re-stages every tracked file through the new `.gitattributes` rules. Files whose stored line endings didn't match the rules get updated in the index.

```bash
$ git status
On branch main
Changes to be committed:
  new file:   .gitattributes
  modified:   scripts/build.sh
  modified:   scripts/deploy.sh
```

Two shell scripts had CRLF stored in the object store — now their index versions have been normalised to LF. The `.cs` files show no change because they were already LF (macOS origin).

```bash
$ git diff --cached scripts/build.sh | head -20
diff --git a/scripts/build.sh b/scripts/build.sh
index 4a3f9c2..7e2b1a0 100755
--- a/scripts/build.sh
+++ b/scripts/build.sh
@@ -1,6 +1,6 @@
-#!/bin/bash^M
-set -euo pipefail^M
-^M
-echo "Building InterVision..."^M
-dotnet build InterVision.sln --configuration Release^M
+#!/bin/bash
+set -euo pipefail
+
+echo "Building InterVision..."
+dotnet build InterVision.sln --configuration Release
```

The `^M` characters (CRLF) are being removed. This confirms the problem existed and is being fixed.

```bash
$ git commit -m "chore: add .gitattributes with LF normalisation for all text files

Normalises all text files to LF in the Git object store.
Fixes CRLF contamination in scripts/build.sh and scripts/deploy.sh.
Binary assets marked explicitly to prevent text diffing.
C# diff driver configured for method-context hunk headers."
[main 9a1f0e2] chore: add .gitattributes with LF normalisation for all text files
 3 files changed, 48 insertions(+), 2 deletions(-)
 create mode 100644 .gitattributes
```

Commit the normalisation in a dedicated commit — makes it easy to audit and easy to skip in `git blame` with `--ignore-rev`.

```bash
$ git push origin main
```

Push immediately so the next `git clone` automatically picks up the `.gitattributes`.

### What Could Go Wrong Mid-Walkthrough

**`git add --renormalize .` shows no changes but you expected some:**

```bash
$ git add --renormalize .
$ git status
On branch main
Changes to be committed:
  new file:   .gitattributes
# No modified files listed
```

This means all tracked files already matched the rules before `.gitattributes` was introduced — either because the repo was entirely macOS/Linux-origin (already LF), or because a previous `core.autocrlf=input` setting was doing the right thing. This is fine — commit `.gitattributes` as-is so future contributors' machines are governed by the file rather than their personal config.

**A file you expected to be normalised isn't showing as modified:**

```bash
$ git ls-files --eol | grep scripts/deploy.sh
i/crlf  w/crlf  attr/text=auto,eol=lf  scripts/deploy.sh
```

The `i/crlf` (index) and `w/crlf` (working tree) both show CRLF, but `--renormalize` didn't update it. This means `.gitattributes` wasn't staged yet when you ran `--renormalize`. Fix: stage `.gitattributes` first, then re-run:

```bash
$ git add .gitattributes
$ git add --renormalize .
$ git ls-files --eol | grep scripts/deploy.sh
i/lf    w/lf    attr/text=auto,eol=lf  scripts/deploy.sh
```

## Gotchas and Troubleshooting

### `/bin/bash^M: bad interpreter: No such file or directory`

**Root cause:** A shell script was stored in the Git object store with CRLF line endings and then checked out on Linux/macOS. The shebang line (`#!/bin/bash`) ends with `\r`, making the interpreter path `/bin/bash\r` — which doesn't exist. This is the most damaging manifestation of missing `.gitattributes`.

```bash
$ bash scripts/run-tests.sh
bash: scripts/run-tests.sh: /bin/bash^M: bad interpreter: No such file or directory

# Confirm CRLF in the file:
$ file scripts/run-tests.sh
scripts/run-tests.sh: Bourne-Again shell script, ASCII text executable, with CRLF line terminators
```

**Fix:**

```bash
# Immediate one-file fix (don't commit this — fix .gitattributes instead):
$ sed -i 's/\r//' scripts/run-tests.sh

# Permanent fix: ensure .gitattributes pins *.sh to eol=lf,
# then renormalise and commit as shown in the walkthrough.
$ git check-attr eol scripts/run-tests.sh
scripts/run-tests.sh: eol: lf
# ← confirms the rule is now in effect
```

**Prevention:** The `*.sh text eol=lf` rule in `.gitattributes` is the only prevention. `core.autocrlf=input` on the committer's machine helps but isn't sufficient — if one Windows developer has `autocrlf=true`, they'll re-contaminate the file on their next commit touching it. Only `.gitattributes` enforces the rule repo-wide.

### `git diff` shows every line changed after a teammate commits from Windows

**Root cause:** A developer with `core.autocrlf=true` (Windows default) committed files with CRLF line endings stored in the object store. Your local checkout on macOS/Linux shows the same file as LF on disk — so `git diff` sees every single line as changed (CRLF → LF on each line).

```bash
$ git diff src/OrderService/OrderHandler.cs | head -10
diff --git a/src/OrderService/OrderHandler.cs b/src/OrderService/OrderHandler.cs
index 4a3f9c2..7e2b1a0 100644
--- a/src/OrderService/OrderHandler.cs
+++ b/src/OrderService/OrderHandler.cs
@@ -1,5 +1,5 @@
-using System;^M
-using Microsoft.AspNetCore.Mvc;^M
-^M
-namespace OrderService;^M
+using System;
+using Microsoft.AspNetCore.Mvc;
+
+namespace OrderService;
```

**Fix:**

```bash
# Immediate: view the diff ignoring EOL to confirm only EOL differs
$ git diff --ignore-space-at-eol src/OrderService/OrderHandler.cs
# (no output = confirmed, only line endings differ)

# Permanent: add .gitattributes, renormalise, commit (see walkthrough)
# Ask the Windows developer to set:
PS> git config --global core.autocrlf false
# Then re-clone or run: git rm --cached -r . && git reset --hard
```

**Prevention:** `.gitattributes` with `* text=auto eol=lf` catches this at the source — the Windows developer's commit is normalised to LF before it enters the object store, regardless of their `core.autocrlf` setting.

---

### `git blame` shows a normalisation commit touching every line

**Root cause:** A bulk line-ending normalisation commit (like the one this note produces) rewrites every line of every affected file. Subsequent `git blame` runs show the normalisation commit as the last-touched for every line, obscuring the actual author of the logic.

```bash
$ git blame src/NotificationWorker/NotificationDispatcher.cs | head -5
9a1f0e2 (Alice Chen 2026-01-15 14:32:07 +0000  1) using Microsoft.Extensions.Logging;
9a1f0e2 (Alice Chen 2026-01-15 14:32:07 +0000  2) using NotificationWorker.Models;
9a1f0e2 (Alice Chen 2026-01-15 14:32:07 +0000  3)
9a1f0e2 (Alice Chen 2026-01-15 14:32:07 +0000  4) namespace NotificationWorker;
# Every line blames the normalisation commit
```

**Fix:**

```bash
# Option A: --ignore-rev flag (one-off)
$ git blame --ignore-rev 9a1f0e2 src/NotificationWorker/NotificationDispatcher.cs

# Option B: .git-blame-ignore-revs file (permanent, team-wide)
$ cat .git-blame-ignore-revs
9a1f0e2  # chore: normalise all line endings to LF via .gitattributes

# Git automatically picks up this file if configured:
$ git config --global blame.ignoreRevsFile .git-blame-ignore-revs

# GitHub's blame UI also respects .git-blame-ignore-revs automatically
```

**Prevention:** Always create `.git-blame-ignore-revs` as a second commit immediately after a normalisation commit, and wire it in the team's global config. Document this in the repo's `CONTRIBUTING.md`.

### `.gitattributes` rule exists but file still has wrong line endings

**Root cause:** The `.gitattributes` rule was added after the file was already committed with the wrong line endings. Rules in `.gitattributes` apply at add/checkout time — they don't retroactively fix what's already in the object store.

```bash
$ git check-attr eol scripts/deploy.sh
scripts/deploy.sh: eol: lf       # ← rule IS in effect

$ git ls-files --eol | grep deploy.sh
i/crlf  w/crlf  attr/text=auto,eol=lf  scripts/deploy.sh
# ← but the INDEX still has CRLF
```

**Fix:**

```bash
# Force re-application of .gitattributes to the existing index
$ git add --renormalize scripts/deploy.sh

$ git ls-files --eol | grep deploy.sh
i/lf    w/lf    attr/text=auto,eol=lf  scripts/deploy.sh
# ← now LF

$ git commit -m "chore: renormalise scripts/deploy.sh to LF"
```

**Prevention:** Always run `git add --renormalize .` immediately after committing a new or modified `.gitattributes`. The two steps should be treated as a single atomic operation.

---

### `git diff` shows binary garbage for a text file

**Root cause:** A file extension was accidentally matched by an overly broad `binary` rule in `.gitattributes`, or a file was committed with a BOM (byte-order mark) that caused Git's auto-detection to classify it as binary.

```bash
$ git diff src/OrderService/appsettings.json
Binary files a/src/OrderService/appsettings.json and b/src/OrderService/appsettings.json differ
```

**Fix:**

```bash
# Check what attributes are applied
$ git check-attr -a src/OrderService/appsettings.json
src/OrderService/appsettings.json: binary: set
# ← found it — a wildcard binary rule is matching

# Review .gitattributes for an overly broad pattern
# e.g., *.json binary  ← wrong; fix to *.json text eol=lf

# If the file has a UTF-8 BOM, remove it:
$ file src/OrderService/appsettings.json
src/OrderService/appsettings.json: UTF-8 Unicode (with BOM) text
$ sed -i '1s/^\xef\xbb\xbf//' src/OrderService/appsettings.json

# After fixing .gitattributes, renormalise
$ git add --renormalize src/OrderService/appsettings.json
$ git add .gitattributes
$ git commit -m "fix: correct .gitattributes — appsettings.json is text, not binary"
```

**Prevention:** Be specific with binary rules — list exact extensions (`*.png`, `*.dll`) rather than broad patterns. After writing or modifying `.gitattributes`, run `git check-attr -a` on a sample of files from each type to confirm attributes are being applied as intended.

---

### `warning: LF will be replaced by CRLF` on commit

**Root cause:** A developer's `core.autocrlf=true` (Windows default) is fighting the `.gitattributes` `eol=lf` rule. When `.gitattributes` specifies `eol=lf` and `core.autocrlf=true`, Git emits this warning but still stores LF in the object store — the warning is alarming but harmless in this case. If `.gitattributes` is absent and `autocrlf=true`, Git silently stores CRLF.

```bash
$ git add src/PaymentGateway/Program.cs
warning: LF will be replaced by CRLF in src/PaymentGateway/Program.cs.
The file will have its original line endings in your working-directory.
```

**Fix:**

```bash
# With .gitattributes eol=lf in place, this warning can be silenced:
PS> git config --global core.autocrlf false
# Prevents the conflict between autocrlf=true and eol=lf rules

# Verify the file is still stored correctly:
$ git ls-files --eol | grep Program.cs
i/lf    w/lf    attr/text=auto,eol=lf  src/PaymentGateway/Program.cs
# ← LF in index despite the warning — .gitattributes won
```

**Prevention:** When `.gitattributes` governs EOL, set `core.autocrlf=false` globally for all team members. Document this in the repo's `README.md` or `CONTRIBUTING.md` setup section. The two systems are redundant and their interaction causes confusing warnings.

## Speed and Efficiency

### Aliases and Shortcuts

```bash
# ~/.gitconfig — add under [alias]

# Audit all tracked files for their EOL status in one shot
eol-audit = ls-files --eol

# Check all attributes on a specific file
attrs = check-attr -a

# Renormalise entire tree after a .gitattributes change
renorm = add --renormalize .

# Show diff ignoring EOL noise (useful during migration)
diffeol = diff --ignore-space-at-eol
```

```bash
# Shell function — audit and report CRLF files in the object store
git-crlf-report() {
  git ls-files --eol | grep 'i/crlf' | awk '{print $NF}' | sort
}
# Usage: git-crlf-report
# Output: list of file paths stored with CRLF in the object store
```

### Automation Opportunity

A pre-commit hook that rejects commits containing files with wrong line endings — catches the problem at the developer's machine before it reaches the shared repo:

```bash
#!/usr/bin/env bash
# .git/hooks/pre-commit  (or install via pre-commit framework)
# Rejects staged files that violate .gitattributes EOL rules.

set -euo pipefail

BAD_FILES=$(git ls-files --eol | grep '^i/crlf' | awk '{print $NF}')

if [ -n "$BAD_FILES" ]; then
  echo "ERROR: The following files have CRLF stored in the index."
  echo "Run: git add --renormalize . && git commit"
  echo ""
  echo "$BAD_FILES"
  exit 1
fi
```

For teams using the `pre-commit` framework (see [[9.021 — Git Hooks — pre-commit and pre-push Basics]]):

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: mixed-line-ending
        args: ['--fix=lf']        # auto-fix to LF on commit
      - id: end-of-file-fixer     # ensures files end with exactly one newline
```

### Keyboard / CLI Efficiency Notes

- `git ls-files --eol` is the single most useful command for this topic and most engineers don't know it exists. Run it as the first diagnostic step whenever someone reports line-ending weirdness — it tells you exactly what's stored vs what's on disk vs what the rules say, all in one output.
- `git check-attr -a <file>` tells you every attribute currently in effect for a file. Faster than reading `.gitattributes` manually when you want to confirm a rule is matching.
- `git diff --ignore-space-at-eol` (or `-w` to ignore all whitespace) is the fastest way to verify that a suspicious diff is "only line endings, no real changes" — one command to separate noise from signal.
- Adding `blame.ignoreRevsFile = .git-blame-ignore-revs` to the global config once means `git blame` automatically skips normalisation commits on every repo that has the file, with no per-blame flags needed.

### Spoken Answers

**Q: Walk me through how you'd diagnose and fix a PR that shows 400 lines changed when only 3 were edited.**

> **Average answer:** "It's a line-ending issue. I'd add `.gitattributes` and set `core.autocrlf`."

> **Great answer:** "First I'd confirm the hypothesis with `git diff --ignore-space-at-eol` on the changed file — if that produces no output, it's purely a line-ending problem, zero content change. Then I'd run `git ls-files --eol | grep <filename>` to see what's actually stored in the index vs the working tree. If the index shows `i/crlf` and the working tree shows `w/lf`, a Windows developer committed CRLF into the object store and my macOS machine is showing it as LF on disk. The fix is: add `.gitattributes` with `* text=auto eol=lf` and the specific extensions for the repo, stage it, run `git add --renormalize .` to fix the existing index, commit as a labelled normalisation commit, and immediately add that commit to `.git-blame-ignore-revs` so blame doesn't become useless. I'd also tell the Windows developer to set `core.autocrlf=false` so their editor stops fighting the `.gitattributes` rules, and document it in CONTRIBUTING.md."

---

**Q: Shell script fails in CI with `/bin/bash^M: bad interpreter` — root cause and fix?**

> **Average answer:** "The script has Windows line endings. Convert it with `dos2unix`."

> **Great answer:** "`dos2unix` fixes it on disk right now but doesn't fix the object store — the next time someone with `core.autocrlf=true` touches that file, the CRLF comes back. The root cause is that the script was committed with CRLF line endings stored in Git, and the Linux CI agent checked it out as-is. The shebang line ends with `\r`, making the interpreter path `/bin/bash\r` — which doesn't exist. The permanent fix is a `.gitattributes` rule: `*.sh text eol=lf` — this tells Git to always store shell scripts as LF and always check them out as LF, even on Windows, regardless of the developer's `core.autocrlf` setting. After adding the rule, run `git add --renormalize .` to re-normalise any already-contaminated scripts in the index, commit, and push. The CI agent then checks out clean LF files every time. I'd also add a pre-commit hook that scans for CRLF in staged `.sh` files so the problem can't recur."

---

**Q: You added `.gitattributes` to a repo that's been running without one. What else do you need to do?**

> **Average answer:** "Commit the `.gitattributes` file."

> **Great answer:** "Adding `.gitattributes` and committing it only governs future file operations — it doesn't retroactively fix files already in the object store with the wrong line endings. You need three more steps. First, stage `.gitattributes`. Second, run `git add --renormalize .` which re-stages every tracked file through the new rules — any file whose stored line endings don't match the rules gets updated in the index. Third, commit the result as a clearly labelled normalisation commit. After that, every team member needs to pull and re-checkout: `git rm --cached -r . && git reset --hard` — otherwise their local working trees still reflect the old object store. I'd also add the normalisation commit SHA to `.git-blame-ignore-revs` immediately, because that commit will touch every affected file and make `git blame` appear to blame me for every line in the repo."

### War Story Prompt

A `PaymentGateway` CI pipeline was intermittently failing on the Linux build agent with `/bin/bash^M: bad interpreter: No such file or directory` in `scripts/integration-test.sh` — but only on some runs, not all. The investigation took two hours because the script ran fine locally (macOS) and the error pointed to the script's interpreter, not to the actual bug. The real cause: the script had stable LF line endings in the object store, but one Windows developer had `core.autocrlf=true` and periodically touched `integration-test.sh` as part of larger commits — each time they saved and staged it, their editor silently converted it to CRLF before Git's autocrlf kicked in, meaning the stored version alternated between LF (everyone else's commits) and CRLF (their commits) depending on who last touched the file. The CI failure was perfectly correlated with that developer's deployment commits, which nobody noticed until someone ran `git log --oneline scripts/integration-test.sh` and saw the pattern. A single `*.sh text eol=lf` line in `.gitattributes` would have made the Windows developer's `autocrlf` irrelevant and ended the flakiness permanently.

## Decision Framework

### When to Reach for This

- [ ] Setting up any new repo that will have contributors on both Windows and macOS/Linux
- [ ] A PR shows far more lines changed than the author claims to have edited
- [ ] A shell script or Makefile works locally but fails on a Linux CI agent with a `^M` error
- [ ] `git blame` is returning a single bulk commit as the author of every line
- [ ] Binary assets (PNGs, DLLs, NuGet packages) are showing up in text diffs
- [ ] A diff driver isn't showing method context in hunk headers for `.cs` files
- [ ] Onboarding the first Windows developer to a previously macOS-only repo
- [ ] CI or a new team member reports that `git diff` is always noisy with CRLF changes

### When NOT To

- [ ] The repo is exclusively Linux/macOS and there is no possibility of Windows contributors — the default `core.autocrlf=input` behaviour is sufficient and `.gitattributes` adds maintenance overhead for no benefit
- [ ] You want to silently fix one developer's `core.autocrlf` setting — that's a per-machine config issue (see [[9.019 — Git Config — Global, Local, and System Scope]]); `.gitattributes` is a repo-wide policy

### Tool Comparison

||`.gitattributes`|`core.autocrlf` per machine|
|---|---|---|
|Scope|Repo-wide, committed, applies to everyone|Per-machine, per-clone, overridden by `.gitattributes`|
|Requires team coordination|No — committed into the repo|Yes — every developer and CI agent must be configured|
|Handles binary files|Yes — `binary` attribute|No — only handles text files|
|Handles per-extension rules|Yes — wildcard patterns|No — global setting only|
|Survives new team members|Yes — applied on first clone|No — new developer must be told to configure it|
|When it's the right call|Any shared repo, always|Fallback for repos without `.gitattributes`; set `input` on macOS/Linux, `false` on Windows when `.gitattributes` governs EOL|

## Self-Check

### Conceptual Questions

1. What is the difference between what `eol=lf` and `text=auto` do in a `.gitattributes` rule, and what does combining them as `text=auto eol=lf` achieve?
2. You run `git ls-files --eol` and see `i/crlf w/lf attr/text=auto,eol=lf` for a file. What does each column mean, and is this a problem?
3. What exact command re-applies `.gitattributes` rules to files already in the index without creating a new checkout?
4. A developer's commit triggers `warning: LF will be replaced by CRLF in src/PaymentGateway/Program.cs`. Does this mean CRLF was stored in the object store?
5. What does marking a file as `binary` in `.gitattributes` actually change about Git's behaviour — beyond diff output?
6. What is the single most dangerous omission in a `.gitattributes` file for a repo that runs CI on Linux?
7. How does `.git-blame-ignore-revs` connect to `.gitattributes` in a real team workflow?

<details> <summary>Answers</summary>

1. `text=auto` tells Git to auto-detect whether a file is text or binary; for files it classifies as text, it applies normalisation. `eol=lf` specifies which line ending to store in the object store (LF) and to use on checkout (LF on all platforms, overriding the OS-default behaviour of converting to CRLF on Windows). Combined as `* text=auto eol=lf`: auto-detect text files and, for those detected as text, always store LF and always check out LF — the safest catch-all rule.
    
2. `i/crlf` = the object store (index) contains CRLF. `w/lf` = the working tree file has LF. `attr/text=auto,eol=lf` = the `.gitattributes` rule currently in effect. This IS a problem: the rule says `eol=lf` but the index still has CRLF — meaning `.gitattributes` was added after the file was committed with CRLF, and `git add --renormalize` has not been run yet. Run `git add --renormalize <file>` to fix the index.
    
3. `git add --renormalize .` — re-stages every tracked file through the current `.gitattributes` rules without touching the working tree. Files whose index representation doesn't match the rules get updated. This command must be run after `.gitattributes` is staged (i.e., `git add .gitattributes` must come first).
    
4. Not necessarily. When `.gitattributes` specifies `eol=lf`, the LF-in-object-store rule takes precedence over `core.autocrlf=true`. The warning means Git detected the conflict between the local `autocrlf=true` setting and the `.gitattributes` `eol=lf` rule — it emits the warning but still stores LF. Run `git ls-files --eol` to confirm `i/lf` for the file. The warning is eliminated by setting `core.autocrlf=false` on the Windows machine.
    
5. Marking a file `binary` in `.gitattributes` does four things: (1) disables text diffing — `git diff` shows "Binary files differ" instead of a text diff; (2) disables line-ending conversion — no CRLF/LF translation ever; (3) disables merge — Git will not attempt a three-way text merge and will instead force a conflict requiring manual resolution (keeping one side's file entirely); (4) disables keyword expansion if any filter drivers are configured.
    
6. Missing `*.sh text eol=lf` (and similar for `Makefile`, `.bash`). Without it, a Windows developer's commit can store CRLF in shell scripts. When the Linux CI agent checks them out, shebangs become `/bin/bash\r` and the scripts fail with `bad interpreter`. This causes intermittent, OS-specific CI failures that can be extremely confusing to diagnose.
    
7. A line-ending normalisation commit produced by `.gitattributes` + `git add --renormalize .` touches every line of every affected file, making `git blame` show that commit as the last author for every line. `.git-blame-ignore-revs` is a file listing commit SHAs that `git blame` should skip, restoring blame output to show the real authors of the code logic. The connection: every time `.gitattributes` changes produce a normalisation commit, that commit SHA should be added to `.git-blame-ignore-revs` as an immediate follow-up. See [[9.017 — Git Show and Blame — Inspecting History]] for the full `git blame` workflow.
    

</details>

**Drill 2 — Fix the broken thing**

```bash
# A CI pipeline on a Linux agent fails with:
$ bash scripts/migrate-db.sh
bash: scripts/migrate-db.sh: /bin/bash^M: bad interpreter: No such file or directory

# The .gitattributes in the repo currently contains only:
$ cat .gitattributes
*.png    binary
*.jpg    binary

# The file runs fine on the developer's Windows machine.
# git ls-files --eol shows:
$ git ls-files --eol | grep migrate-db
i/crlf  w/crlf  attr/               scripts/migrate-db.sh
```

Diagnose the problem and apply the permanent fix — not a one-file `dos2unix` patch.

<details> <summary>Solution</summary>

**Diagnosis:** `scripts/migrate-db.sh` is stored with CRLF line endings in the Git object store (`i/crlf`). The Linux CI agent checks it out as-is, giving the shebang a `\r` suffix, making the interpreter `/bin/bash\r` — a path that doesn't exist. The Windows developer's machine runs it fine because Windows-native tools ignore `\r`. The current `.gitattributes` has no rule for `.sh` files, so nothing is preventing CRLF from being committed.

```bash
# Step 1: Add the missing rule to .gitattributes
$ cat >> .gitattributes << 'EOF'

# Shell scripts — must always be LF, even on Windows checkout
*.sh     text eol=lf
*.bash   text eol=lf
Makefile text eol=lf
EOF

# Step 2: Stage .gitattributes first
$ git add .gitattributes

# Step 3: Renormalise — this updates the index for migrate-db.sh
$ git add --renormalize .

# Step 4: Verify the index is now LF
$ git ls-files --eol | grep migrate-db
i/lf    w/lf    attr/text=auto,eol=lf  scripts/migrate-db.sh
# ← LF in index

# Step 5: Verify only EOL changed, no content
$ git diff --cached --ignore-space-at-eol scripts/migrate-db.sh
# (no output — confirmed, only line endings changed)

# Step 6: Commit
$ git commit -m "fix: pin shell scripts to eol=lf in .gitattributes

Fixes /bin/bash^M bad interpreter error on Linux CI agent.
scripts/migrate-db.sh was stored with CRLF; renormalised to LF."
[main 7b2c4d1] fix: pin shell scripts to eol=lf in .gitattributes

# Step 7: Verify CI fix — the next pipeline run will check out LF
$ git check-attr eol scripts/migrate-db.sh
scripts/migrate-db.sh: eol: lf
```

</details>
