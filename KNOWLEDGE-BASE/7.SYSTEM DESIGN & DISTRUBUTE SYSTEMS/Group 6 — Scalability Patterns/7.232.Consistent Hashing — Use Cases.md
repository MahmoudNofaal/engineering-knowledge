## Core Mental Model

Consistent hashing is a **routing primitive** — it maps keys to nodes deterministically while minimizing redistribution when the node set changes. Its three properties (monotonicity, spread, load) make it suitable for any distributed system that must:

1. Route requests to the correct node without a central lookup table.
2. Survive node failures without reshuffling all data.
3. Add capacity without rehashing every key.

Every real-world use of consistent hashing is an optimization of these three properties for a specific workload characteristic (read-heavy, write-heavy, ordered scans, low-latency). The implementation differences between systems reflect different tradeoffs in the hash function choice, virtual node count, replication strategy, and failure detection.

### Three Fundamental Use-Case Categories

```
Use Case              Primary Property Exploited      Key Design Choice
──────────────────    ─────────────────────────────    ──────────────────────
Cache/CDN             Monotonicity (minimal           No replication needed
                      redistribution on add/remove)   (data loss = cache miss)
                      
KV Store / Database   Monotonicity + Replication      Replication factor (RF)
                      (survive failures without       determines durability
                      data loss)                      vs write latency
                      
Coordination /        Spread (even distribution       VN count determines
Messaging             across heterogeneous nodes)     partition granularity
```

### Consistency Model Impact

| Use Case | Typical Consistency | Implication for Consistent Hashing |
|---|---|---|
| CDN / HTTP cache | Eventual | Ring update BEFORE data copy is safe (cache miss = fetch from origin) |
| KV store (Cassandra) | Tunable (eventual to quorum) | VNs + replication factor × quorum settings determine when ring update is safe |
| KV store (DynamoDB) | Strong (via quorum) | Token range pre-splitting — no dynamic ring update during normal operation |
| Distributed cache (Memcached) | None | Ring update immediate — no durability guarantee |
| Message queue (Kafka) | Durable + ordered | Partition reassignment requires data copy even with consistent hashing |
| Real-time messaging (Discord) | At-least-once + ordering per guild | Consistent hashing per shard — ring update triggers per-guild rebalancing |

## How It Works — Use Cases by Category

### Caching and CDN (Akamai, Memcached)

**Architectural role:** Consistent hashing is used as the **routing layer** that maps URL paths (or cache keys) to cache server nodes.

**Akamai's "SureRoute" / content-routing system:** Akamai was one of the earliest production users of consistent hashing (Karger et al., 1997). An Akamai edge server maps a URL to a hash ring position. The successor edge server on the ring caches the content. When a user requests a URL, the DNS resolution and HTTP request routing use consistent hashing to find the closest edge server that should have the content.

```
// Port: Cache routing — simple, no replication needed
// Consistent hashing maps: URL → Hash → Ring Position → Cache Node

class CacheRouter
{
    const int VIRTUAL_NODES = 160;
    
    public CacheNode GetCacheNode(string url)
    {
        var hash = HashUrl(url);
        var position = _ring.FindWrapping(hash);  // O(log N)
        return _nodesByPosition[position];
    }
}

// Invalidation: Content is time-to-live (TTL) based, not explicit
// Node failure: Ring update in microseconds, cache miss fetches from origin
// Key property: No data loss on failure — origin is source of truth
```

**Memcached with consistent hashing (libketama):** Before consistent hashing, Memcached clients used simple `hash(key) % N` modulo routing. Adding a node invalidated ~1/N of ALL cached entries — a cache-stampede event. libketama (used by Twitter, Reddit, and early Facebook) uses consistent hashing with VNs to minimize cache invalidation on topology change.

```
// Implementation pattern: libketama
// Each physical node maps to 40 virtual nodes
// Ring is precomputed and distributed as a configuration file

class KetamaRing
{
    // Pre-computed sorted list of (hash, node) pairs
    // 40 VNs × N nodes = 40N entries
    
    public KetamaNode GetNode(string key)
    {
        var hash = Hash(key);
        // Binary search on the sorted ring
        var index = _sortedRing.BinarySearch(hash);
        if (index < 0)
            index = ~index;           // First greater
        if (index >= _sortedRing.Count)
            index = 0;                // Wrap around
        return _sortedRing[index].Node;
    }
}
```

**When to use:** Any distributed cache where (a) cache misses are acceptable (origin data store exists), (b) nodes are added/removed dynamically (auto-scaling), and (c) the data size per node exceeds the per-node memory limit.

**Cassandra (Dynamo-style key-value store):** Cassandra uses consistent hashing with virtual nodes (default VN=128) for partition-level routing. Each node owns a set of token ranges. Writes are routed to the coordinator node (the first replica), which writes to all RF replicas. Reads are routed to the coordinator, which requests from all RF replicas and reconciles based on the consistency level.

```
// Adapter: Cassandra partitioner (Murmur3Partitioner, default)
// Hashes the partition key using Murmur3 (64-bit)
// Output range: [-2^63, 2^63) — a full 64-bit signed integer range

class CassandraPartitioner
{
    const int VIRTUAL_NODES = 128;
    
    public long HashPartitionKey(byte[] partitionKey)
    {
        return Murmur3.Hash64(partitionKey); // Deterministic, fast, uniform
    }
    
    public List<CassandraNode> GetReplicas(byte[] partitionKey, int rf, ITokenRing ring)
    {
        long token = HashPartitionKey(partitionKey);
        // Walk the ring clockwise, collecting RF unique physical nodes
        var replicas = new List<CassandraNode>();
        foreach (var entry in ring.GetClockwiseEntries(token))
        {
            var physicalNode = entry.PhysicalNode;
            if (!replicas.Contains(physicalNode))
                replicas.Add(physicalNode);
            if (replicas.Count >= rf)
                break;
        }
        return replicas; // Unique physical nodes for the key
    }
}
```

Cassandra's key consistent hashing decisions:

| Decision | Cassandra Choice | Rationale |
|---|---|---|
| Hash function | Murmur3 (64-bit) | Fast, uniform, avalanche property — avoids hotspotting on sequential keys |
| Virtual nodes | 128 (default) | CV ≈ 1/√128 ≈ 0.09 — near-perfect load distribution |
| Token assignment | Random (with VNs) or ordered (without VNs) | Random for VNs avoids hot spots from sequential token assignment |
| Replication | Token-range based (walk ring for RF unique nodes) | Each key replicated to RF consecutive distinct physical nodes |
| Ring update on failure | Gossip-based failure detection → hinted handoff | Nodes are NOT immediately removed — hints accumulate for up to 3 hours |
| Node addition | Bootstrap — streams data from predecessors in parallel | New node's VN ranges are distributed across ~N predecessors |

**Amazon Dynamo (the original paper, not DynamoDB):** The Dynamo paper (DeCandia et al., 2007) introduced consistent hashing to the NoSQL world. Dynamo uses a 128-bit MD5 hash (later SHA-1 in implementations) and assigns each node a random position on the ring. The key insight: Dynamo treats each node as a "virtual node" that manages a contiguous token range, not multiple interleaved ranges. A Dynamo node owns the arc from its token to the next node's token.

```
// Port: Dynamo's consistent hashing model
// Nodes own CONTIGUOUS token ranges (not interleaved VNs)
// Each physical node manages a single arc [token_i, token_{i+1})

// When a node joins:
// 1. Coordinator assigns the new node a token (or range)
// 2. The new node splits an existing arc
// 3. Data for the split arc is streamed from the predecessor
// 4. Membership change is gossiped to all nodes

// Compare with Cassandra VNs:
// Dynamo:    1 physical node = 1 contiguous arc (no VNs originally)
// Cassandra: 1 physical node = ~128 interleaved arcs (VNs)
```

**DynamoDB (AWS DynamoDB, not the paper):** DynamoDB uses a variant of consistent hashing where the token ring is **pre-split** into partitions. Each partition handles 10 GB of data and 3,000 read / 1,000 write capacity units. The partition key is hashed (using a 128-bit hash function internal to DynamoDB) and mapped to a partition. DynamoDB handles the ring internally — users only specify the partition key.

```
// User perspective: DynamoDB consistent hashing
// Users do NOT manage the ring directly
// They choose a partition key, and DynamoDB handles the rest

// Table creation: Partition key = UserId (string)
// DynamoDB hashes UserId → internal partition
// DynamoDB manages ring expansion via "split" operations
// Users see no ring changes — only eventual throttling if a partition is hot

// Why not user-managed consistent hashing?
// 1. Complexity — managing token ranges is error-prone
// 2. Auto-scaling — DynamoDB splits partitions automatically
// 3. Provisioned throughput — DynamoDB handles capacity allocation per partition

// Tradeoff: Less control, more abstraction
// When to choose DynamoDB over Cassandra:
// "Use DynamoDB when you want managed partitioning and auto-scaling,
//  at the cost of higher per-request cost and limited custom consistency,
//  acceptable when operational overhead reduction outweighs the cost premium."
// "Use Cassandra when you want tunable consistency, VN-based load distribution,
//  and lower per-request cost, at the cost of operational complexity,
//  acceptable when you have the ops team to manage a self-hosted cluster."
```

**Voldemort (LinkedIn's key-value store):** Voldemort uses consistent hashing with a **fixed number of partitions** (P) per node. The number of partitions is determined at cluster creation time and does not change. When nodes are added, partitions are REASSIGNED (moved to new nodes) but not RE-SPLIT. This avoids the data-splitting overhead of Dynamo-style token rebalancing.

```
// Voldemort's partition-based consistent hashing:
// Cluster has: N nodes, P total partitions
// Each node initially owns P/N partitions
// When a node is added:
//   1. Partitions are REASSIGNED from existing nodes to the new node
//   2. Each existing node gives up ~1/(N+1) of its partitions
//   3. No partition splitting — partitions are indivisible units
//   4. Data movement: reassigned partitions are streamed to the new node

// Key difference from Cassandra VNs:
// Voldemort: Fixed partition count, partitions move between nodes
// Cassandra: Virtual nodes are created/destroyed as nodes join/leave
```

**Azure Cosmos DB:** Cosmos DB uses consistent hashing with a two-level partitioning model. At the physical level, partitions ("physical partitions") are mapped to the hash ring. Each physical partition handles up to 10,000 RU/s and 50 GB of data. At the logical level, partition keys hash to logical partitions, which are grouped into physical partitions. The ring is managed by Cosmos DB's internal resource governor.

```
// Port: Azure Cosmos DB partitioning — abstracted consistent hashing
// Users specify a partition key path (e.g., "/UserId")
// Cosmos DB hashes the partition key value using a 256-bit hash
// The hash is mapped to a physical partition using consistent hashing

// Key constraints:
// - 10,000 RU/s max per logical partition key value
// - 50 GB max per physical partition
// - No user control over ring management
// - Hot partition detection: monitor PartitionKeyRUConsumption metric

// Cosmos DB vs Cassandra:
// "Use Cosmos DB for globally distributed multi-model workloads
//  with SLA-backed latency and throughput guarantees,
//  at the cost of RU-based pricing and 10K RU/s logical partition limit,
//  acceptable when you need turnkey global distribution."
// "Use Cassandra for self-managed clusters with tunable consistency
//  and no logical partition throughput ceiling,
//  at the cost of operational complexity and no global distribution SLA,
//  acceptable when you have an ops team and need lower per-request cost."
```

### Distributed Coordination and Messaging (Discord, Apache Kafka)

**Discord's consistent hashing for server/guild sharding:** Discord uses consistent hashing to distribute guilds (servers) across application instances (shards). Each guild ID is hashed onto the ring and assigned to the nearest shard. When a new shard is added (to handle growth), only ~1/(N+1) of guilds need to be re-assigned.

```
// Port: Discord's guild → shard routing (simplified)

class DiscordShardRouter
{
    // Hash guild ID to a ring position
    public ShardId GetShardForGuild(GuildId guildId)
    {
        var hash = Hash(guildId.Value);
        var position = _ring.FindWrapping(hash);
        return _positionToShard[position];
    }
    
    // Guild reassignment: When a new shard is added,
    // the successor shard gives up ~1/(N+1) of its guilds
    // Each guild's session state is migrated to the new shard
    // During migration: guild is temporarily read-only
    
    // Key constraint: Guild state is session-based (in-memory)
    // Data migration = transferring WebSocket connections and session state
    // Not a database operation — must be fast (< 5 seconds)
}

// Why consistent hashing for Discord?
// 1. Guild count grows unpredictably — need dynamic scaling
// 2. Guilds are independent — no cross-shard coordination needed
// 3. Session state is ephemeral — data loss = guild reconnects, not data loss
// 4. Shard count changes frequently — Discord was at hundreds of shards
```

**Apache Kafka's partition assignment:** Kafka does NOT use consistent hashing for its core partition-to-broker assignment. Instead, Kafka uses a **controller** that assigns partitions to brokers using a configurable partition assignment strategy (range assignor, round robin, sticky, cooperative sticky). However, Kafka does use consistent hashing for **consumer group rebalancing**: the `PartitionAssignor` implementations (like `StickyAssignor`) use consistent-hashing-like logic to minimize partition movement when consumers join or leave a group.

```
// Kafka consumer rebalancing — consistent-hashing-like behavior
// When a consumer joins or leaves a consumer group:
//   1. Group coordinator triggers a rebalance
//   2. Each consumer performs partition assignment locally
//   3. StickyAssignor: minimizes partition movement (consistent hashing property)
//      — keeps as many existing assignments as possible
//      — only reassigns the partitions that MUST move

// Why not full consistent hashing for Kafka partition-to-broker?
// Kafka partitions are ordered and append-only — hashing breaks ordering
// Kafka needs range-based assignment for log compaction and offset management
// Consistent hashing is used at the CONSUMER level, not the STORAGE level
```

### Load Balancing

**HTTP load balancers (HAProxy, NGINX, Envoy) with consistent hashing:** NGINX upstream consistent hashing (`hash $request_uri consistent;`) uses consistent hashing to map request attributes (URI, IP, cookie) to backend servers. This ensures that requests from the same client or for the same resource always hit the same backend — critical for session affinity without sticky cookies.

```
# NGINX configuration: Consistent hashing for session affinity
# Maps URI to backend server — same URI always hits same server
# No sticky cookies needed — affinity is inherent in the routing

upstream backend {
    consistent;                # Enable consistent hashing ring
    hash $request_uri;         # Hash the request URI
    server 10.0.0.1:8080;     # Backend nodes
    server 10.0.0.2:8080;
    server 10.0.0.3:8080;
}

# When a backend is added/removed:
# Only ~1/(N+1) of URIs are remapped to a different server
# Cache locality is preserved for ~N/(N+1) of URIs
```

**Redis Cluster's hash slots:** Redis Cluster uses a variant of consistent hashing called **hash slot partitioning**. The hash space is divided into 16,384 fixed slots (2^14). Each slot is assigned to a Redis node. A key is mapped to a slot via `CRC16(key) % 16384`. When a node is added, slots are reassigned from existing nodes to the new node using `CLUSTER ADDSLOTS` or the resharding tool.

```
// Port: Redis Cluster hash slots — fixed-slot variant of consistent hashing

// Slot assignment: Each key is mapped to one of 16,384 slots
// Slot → node mapping is maintained by every cluster node
// Mapping changes require explicit slot migration

class RedisClusterRouter
{
    public int GetSlot(string key)
    {
        // Hash tag support: keys with {hash tag} use the hash tag
        var effectiveKey = ExtractHashTag(key);
        return (int)(CRC16(effectiveKey) % 16384);
    }
    
    public RedisNode GetNode(string key)
    {
        var slot = GetSlot(key);
        return _slotToNode[slot]; // O(1) lookup
    }
    
    // On MOVED redirect (slot ownership changed):
    // Client updates its slot → node mapping
    // No lock needed — slot migration is atomic at the Redis level
}

// Redis Cluster vs pure consistent hashing:
// "Use Redis Cluster hash slots when you need O(1) key lookup
//  and non-disruptive slot migration (MOVED/ASK redirects),
//  at the cost of fixed 16,384 slot limit and explicit slot management,
//  acceptable when the cluster size is < 1,000 nodes and
//  operational tooling (redis-cli --cluster reshard) is acceptable."
```

**Why 16,384 slots?** The Redis authors chose 2^14 = 16,384 as a balance: (a) large enough to distribute load evenly across up to 1,000 nodes (~16 slots per node), (b) small enough that the slot-to-node mapping can be stored in a small fixed-size array (16,384 × 6 bytes ≈ 96 KB) and included in every gossip message (cluster bus heartbeat), and (c) CRC16 produces 16-bit output, so using 14 bits truncates to a power-of-two range that enables efficient bitwise modulo.

## Failure Modes

### Category-Specific Failure Modes

**Cache/CDN failure mode: Cache stampede on ring update.** When a cache node is removed from the ring, all keys that previously mapped to that node are now cache misses. If the origin database cannot handle the sudden load spike (up to 1/N of all requests going directly to the origin), the origin may fall over. Mitigation: (a) gradual VN drain, (b) pre-warming new nodes before routing traffic, (c) origin database read replicas with standby capacity.

**KV store failure mode: Hinted handoff accumulation.** Cassandra's hinted handoff accumulates up to 3 hours of writes for a failed node. When the node returns, a burst of hints is replayed, spiking CPU and potentially causing a secondary failure. Mitigation: (a) monitor hinted handoff queue depth, (b) reduce `max_hint_window_in_ms`, (c) use incremental repair instead of bulk hint replay.

**Redis Cluster failure mode: Inconsistent slot assignment during partial cluster split.** If a Redis Cluster node is partitioned, some nodes may believe slot ownership has migrated while others disagree. Clients receive MOVED redirects to nodes that may be unreachable. Mitigation: (a) replica nodes for failover, (b) cluster-require-full-coverage no (availability over consistency).

**Discord failure mode: Guild state loss during shard rebalancing.** When a guild is reassigned to a new shard, all WebSocket connections and in-memory session state for that guild must be transferred. If the transfer fails (network timeout, serialization error), the guild's users are disconnected and must reconnect. Mitigation: (a) stateless guild state where possible, (b) graceful disconnect with reconnect instructions, (c) rollback if transfer fails within timeout.

| Library / System | Hash Function | Virtual Nodes | Ring Data Structure | Language |
|---|---|---|---|---|
| libketama | MD5 | 40 VN per node | Pre-sorted array | C (C# bindings) |

### Production Tuning Guide

**Tuning virtual node count for different workloads:**

| Workload | Recommended VN | Rationale |
|---|---|---|
| Cache (Memcached, Redis OSS Cluster) | 160 (or 16,384 slots) | Low overhead, VNs only for load balancing |
| KV store (Cassandra) | 64–256 | Default 128; higher VN = better distribution but more gossip overhead |
| Database sharding (Azure SQL, sharded SQL Server) | 160 | Matches Cassandra convention; any SQL sharding framework uses this |
| Real-time messaging (Discord-style shards) | 1 VN per shard | Shards own contiguous guild ranges; guild reassignment is manual or session-transfer-based |
| HTTP load balancer (NGINX, Envoy) | 100–200 | Must be high enough to avoid hot spots, but low enough for fast binary search on each request |

**Tuning for heterogenous hardware:**

- "Use more VNs for a stronger node: in Cassandra, set `num_tokens` per node so that stronger hardware owns more VNs. A node with 2× the CPU/RAM should get 2× the VNs."
- "In Envoy, use weight-scaled VNs: `consistentHash.lbConfig.hashBalanceFactor = 0.1`. Higher-weight upstream hosts get proportionally more VNs and proportional request share."
- "For custom implementations, assign VN count proportional to node capacity: `VN_i = Round(VN_base × capacity_i / avg_capacity)`. Recalculate on each topology change."

### .NET Implementation: UseCase-Driven Router

```csharp
// Port: Consistent hashing router with use-case-specific configuration
// Supports cache, database, session, and load-balancer patterns

public enum UseCase
{
    Cache,
    DatabaseUnreplicated,
    DatabaseReplicated,
    SessionShard,
    LoadBalancerAffinity
}

public class ConsistentHashRouter<TNode>
{
    private readonly SortedList<ulong, VirtualNode<TNode>> _ring = new();
    private readonly int _baseVnCount;
    private readonly UseCase _useCase;
    private readonly IHashFunction _hash;
    
    public ConsistentHashRouter(
        IEnumerable<TNode> nodes,
        UseCase useCase,
        IHashFunction hash = null,
        int baseVnCount = 160)
    {
        _useCase = useCase;
        _baseVnCount = baseVnCount;
        _hash = hash ?? new Murmur3Hash(); // Default: Murmur3 64-bit
        _baseVnCount = useCase switch
        {
            UseCase.Cache => Math.Min(baseVnCount, 160), // Cache: fewer VNs is fine
            UseCase.DatabaseUnreplicated => Math.Max(baseVnCount, 160), // DB: need even distribution
            UseCase.DatabaseReplicated => baseVnCount,
            UseCase.SessionShard => 1, // Session shards: contiguous ranges
            UseCase.LoadBalancerAffinity => baseVnCount,
            _ => baseVnCount
        };
        
        foreach (var node in nodes)
            AddPhysicalNode(node, GetVnCountForNode(node));
    }
    
    private int GetVnCountForNode(TNode node)
    {
        // Heterogeneous nodes: override VN count per node
        if (node is IWeightedNode weighted)
            return (int)(_baseVnCount * weighted.Weight);
        return _baseVnCount;
    }
    
    public TNode GetNode(string key)
    {
        var hash = _hash.ComputeHash(Encoding.UTF8.GetBytes(key));
        var position = BitConverter.ToUInt64(hash, 0);
        
        // Find the first VN with hash > key hash
        int index = _ring.Keys.BinarySearch(position);
        if (index < 0)
            index = ~index;
        if (index >= _ring.Count)
            index = 0; // Wrap around
        
        return _ring.Values[index].PhysicalNode;
    }
    
    public void AddPhysicalNode(TNode node, int vnCount)
    {
        for (int i = 0; i < vnCount; i++)
        {
            var vnHash = ComputeVnHash(node, i);
            _ring[vnHash] = new VirtualNode<TNode>(node, vnHash);
        }
    }
    
    public void RemovePhysicalNode(TNode node)
    {
        // Get all VN entries for this node
        var vnKeys = _ring
            .Where(kvp => kvp.Value.PhysicalNode.Equals(node))
            .Select(kvp => kvp.Key)
            .ToList();
            
        foreach (var key in vnKeys)
            _ring.Remove(key);
    }
    
    // Use-case specific behavior
    public async Task HandleTopologyChangeAsync(
        TNode node, ChangeType change, CancellationToken ct)
    {
        switch (_useCase)
        {
            case UseCase.Cache:
                // Immediate ring update — cache misses are OK
                if (change == ChangeType.Added)
                    AddPhysicalNode(node, GetVnCountForNode(node));
                else
                    RemovePhysicalNode(node);
                break;
                
            case UseCase.DatabaseUnreplicated:
                // Data-first ring update — must copy before routing
                await CopyDataToBeforeRoutingAsync(node, ct);
                if (change == ChangeType.Added)
                    AddPhysicalNode(node, GetVnCountForNode(node));
                else
                    await DrainDataBeforeRemovalAsync(node, ct);
                    RemovePhysicalNode(node);
                break;
                
            case UseCase.DatabaseReplicated:
                // Immediate ring update if RF >= 3 — replica is hot standby
                if (change == ChangeType.Added)
                    AddPhysicalNode(node, GetVnCountForNode(node));
                else
                    RemovePhysicalNode(node);
                await RebuildReplicasAsync(node, ct);
                break;
        }
    }
}

// Interface for weighted nodes (heterogeneous hardware)
public interface IWeightedNode
{
    double Weight { get; } // 1.0 = baseline, 2.0 = double capacity
}
```

### .NET Integration: Azure Cosmos DB Sharding with Consistent Hashing

For scenarios where Cosmos DB's built-in partitioning is insufficient (a hot partition exceeds 10,000 RU/s), you can use consistent hashing as a "super-partition key" — a composite synthetic key that distributes requests across multiple logical partition key values.

```csharp
// Adapter: Consistent hashing + synthetic partition key for Cosmos DB hot partition mitigation

public class CosmosDbConsistentHashRouter
{
    private readonly ConsistentHashRouter<string> _hashRouter;
    private readonly CosmosClient _client;
    
    public CosmosDbConsistentHashRouter(
        CosmosClient client, 
        int numSyntheticPartitions = 100)
    {
        _client = client;
        
        // Create synthetic partition key suffixes: [0..numSyntheticPartitions)
        var partitionSuffixes = Enumerable
            .Range(0, numSyntheticPartitions)
            .Select(i => i.ToString())
            .ToList();
            
        _hashRouter = new ConsistentHashRouter<string>(
            partitionSuffixes, 
            UseCase.Cache, // Cosmos DB handles storage; we just need routing
            baseVnCount: 1); // One synthetic partition per node
        
        // Each "node" is a synthetic partition suffix
        // Keys are distributed across N suffixes using consistent hashing
        // Each suffix corresponds to a Cosmos DB logical partition key value
    }
    
    public string GetSyntheticPartitionKey(string userId)
    {
        // Consistent hashing: same userId always gets same suffix
        var suffix = _hashRouter.GetNode(userId);
        return $"{userId}_{suffix}";
    }
    
    public async Task WriteUserEventAsync(string userId, object eventData)
    {
        var partitionKey = GetSyntheticPartitionKey(userId);
        var container = _client.GetContainer("EventsDb", "UserEvents");
        
        await container.CreateItemAsync(
            eventData,
            partitionKey: new PartitionKey(partitionKey));
    }
    
    public async Task<IEnumerable<object>> ReadUserEventsAsync(string userId)
    {
        // Must query ALL synthetic partitions for this user
        var results = new List<object>();
        for (int i = 0; i < 100; i++)
        {
            var pkey = $"{userId}_{i}";
            var query = new QueryDefinition(
                "SELECT * FROM c WHERE c.userId = @userId")
                .WithParameter("@userId", userId);
                
            var iterator = _container.GetItemQueryIterator<object>(
                query,
                requestOptions: new QueryRequestOptions
                {
                    PartitionKey = new PartitionKey(pkey)
                });
                
            while (iterator.HasMoreResults)
            {
                results.AddRange(await iterator.ReadNextAsync());
            }
        }
        return results;
    }
}

// "Use synthetic partition keys with consistent hashing when
//  a single logical partition key exceeds 10,000 RU/s in Cosmos DB,

//  acceptable when individual user traffic spikes are rare
//  and the read fan-out is bounded (< 100 partitions per query)."
```

## Gotchas and Production Pitfalls

### [Pitfall Name] Using Consistent Hashing for Ordered Range Queries

**Pitfall:** Consistent hashing destroys key ordering. If your workload requires range scans (`SELECT * FROM orders WHERE order_date >= '2024-01-01' AND order_date < '2024-02-01'`), consistent hashing scatters the ordered keys across ALL nodes. Every range query becomes a scatter-gather across the entire cluster.

```csharp
// ❌ Wrong — consistent hashing for ordered range queries
// Orders are hashed by order_id → scattered across 100 nodes
// Range query by date must query ALL 100 nodes
public class OrderService
{
    private readonly ConsistentHashRouter<IDatabase> _router;
    
    public async Task<List<Order>> GetOrdersByDateRange(
        DateTime start, DateTime end)
    {
        var results = new List<Order>();
        // Scatter-gather across ALL 100 nodes — slow, expensive
        foreach (var db in _router.AllNodes)
        {
            var range = await db.QueryAsync<Order>(
                "SELECT * FROM Orders WHERE order_date >= @start " +
                "AND order_date < @end",
                new { start, end });
            results.AddRange(range);
        }
        return results;
    }
}

// ✅ Right — use range-based sharding for ordered queries
// or add a secondary index (Elasticsearch, Azure Cosmos DB composite index)
public class OrderService
{
    // Option A: Range-based sharding by date
    // Shard 1: orders 2023-Q1, Shard 2: 2023-Q2, ...
    // Range query hits 1–2 shards (bounded)
    
    // Option B: Consistent hashing for WRITES + secondary index for READS
    // Write to consistent hash shard by order_id (for point lookups)
    // Write to Elasticsearch for date-range queries
    private readonly ElasticClient _elastic;
    
    public async Task<List<Order>> GetOrdersByDateRange(
        DateTime start, DateTime end)
    {
        // Use secondary index — O(log N), not O(N) scatter-gather
        var response = await _elastic.SearchAsync<Order>(s => s
            .Query(q => q
                .DateRange(r => r
                    .Field(f => f.OrderDate)
                    .GreaterThanOrEquals(start)
                    .LessThan(end))));
        return response.Documents.ToList();
    }
}
```

**Symptom:** Range queries are 10–100× slower than expected. Explain plans show queries hitting ALL shards. The database CPU spikes during every range-based report.

**Fix:** (a) Use range-based sharding (`7.224`) instead of consistent hashing for workloads with ordered range queries. (b) Or use a hybrid approach: consistent hashing for point lookups (by ID), secondary index (Elasticsearch, Azure Search, or Cosmos DB composite index) for range queries. (c) For Cassandra, use clustering columns for ordering WITHIN a partition — Cassandra supports ORDER BY within a partition key (consistent hashing maps the partition key, clustering columns handle ordering).

**Cost of not fixing:** Every range query is a full cluster scan. As the cluster grows from 10 to 1,000 nodes, range query latency grows linearly — from 10× to 1,000× slower than a single-shard query.

### [Pitfall Name] Choosing Consistent Hashing When Rebalancing Is Rare

**Pitfall:** Consistent hashing adds complexity (hash ring, VN management, gossip protocol) that is wasted if the cluster topology changes infrequently (once per year or less). For a cluster with stable membership, a simple modulo-based or lookup-table approach is simpler and has no data-distribution penalty.

```csharp
// ❌ Wrong — using consistent hashing for a static 3-node cluster
// This cluster has not changed in 2 years
var router = new ConsistentHashRouter<string>(
    new[] { "db1", "db2", "db3" },
    UseCase.DatabaseReplicated,
    baseVnCount: 160);
// Complexity: Ring maintenance, binary search on each lookup
// Benefit: None — topology never changes

// ✅ Right — simple modulo for static clusters
// Complexity: O(1) modulo operation
// No ring, no VNs, no gossip
public string GetShardForTenant(string tenantId)
{
    var shardIndex = Math.Abs(tenantId.GetHashCode()) % 3;
    return _shards[shardIndex]; // ["db1", "db2", "db3"]
}

// If membership changes once per year:
// Schedule a maintenance window for the rehashing
// Data migration is acceptable at that frequency
```

**Symptom:** The system uses consistent hashing with VN=160 and a gossip-based failure detector, but the cluster is 3 nodes behind a load balancer and has never been resized. The operational overhead (ring maintenance, VN health checks) provides zero benefit.

**Fix:** Use consistent hashing ONLY when (a) nodes are added/removed frequently (auto-scaling, elastic clusters), or (b) node failure must not cause data redistribution (cache clusters), or (c) heterogeneous hardware requires weighted load distribution. For small, static clusters, use modulo routing or a config file mapping.

**Cost of not fixing:** Unnecessary operational complexity. The ring maintenance adds code paths that can fail (gossip delays, split brain) with no benefit over a config file.

### [Pitfall Name] Misunderstanding When Consistency Guarantees Apply

**Pitfall:** Consistent hashing does NOT provide consistency guarantees — it provides ROUTING guarantees. A common interview mistake: "Consistent hashing ensures that a write and subsequent read see the same data." It does not. Consistent hashing ensures that the write and read go to the SAME NODE (deterministic routing). Whether the read sees the write depends on the node's local consistency model (durable write, async replication, sync replication).

```csharp
// ❌ Wrong — assuming consistent hashing provides consistency
var node = router.GetNode("user_123");
await node.WriteAsync("key", "value");
var result = await node.ReadAsync("key"); // Might return OLD value
// The write and read go to the SAME node
// But the node might have returned "write OK" before persisting
// Or replication lag may cause read-your-writes violation

// ✅ Right — consistent hashing provides routing, NOT consistency
// You still need:
// 1. Node-level consistency: Write-Ahead Log (WAL), fsync
// 2. Read-your-writes: Read-after-write consistency via timestamp or version
// 3. Replication consistency: Quorum reads/writes (Cassandra), 
//    multi-region writes (Cosmos DB)
// 4. Failure consistency: Hinted handoff, read repair (Cassandra)

// Consistent hashing = "This key goes to Node X"
// Consistency = "Node X returns the latest value for this key"
// These are ORTHOGONAL concerns
```

**Symptom:** After migrating from a single-node database to a consistent-hashing-sharded database, users report seeing stale data after writes. Investigation shows the writes went to the correct shard, but the shard's replication was asynchronous.

**Fix:** Consistent hashing is not a consistency mechanism. Add node-level (WAL, fsync) and cluster-level (quorum, version vectors) consistency on top of the routing layer.

**Cost of not fixing:** Data integrity violations. Users see stale data. The system violates its consistency SLAs.

### [Pitfall Name] Hash Collisions in Weak Hash Functions

**Pitfall:** Using a weak hash function (e.g., `string.GetHashCode()` in .NET, which varies by process and is not guaranteed stable) for consistent hashing. The `GetHashCode()` result differs between .NET Framework and .NET Core, between 32-bit and 64-bit processes, and even between app domain restarts. If the hash function is not deterministic across restarts, the ring is different every time the process starts — keys route to different nodes on each restart.

```csharp
// ❌ Wrong — using non-deterministic hash
var hash = key.GetHashCode(); // Varies by .NET version, bitness, process
var position = hash % ringSize; // Invalid — not consistent

// ✅ Right — using deterministic, cryptographic-quality hash
var hashBytes = SHA256.HashData(Encoding.UTF8.GetBytes(key));
var position = BitConverter.ToUInt64(hashBytes, 0); // Deterministic

// Murmur3 and xxHash are preferred over SHA for performance
// The hash function must be:
// 1. Deterministic: Same input → same output forever
// 2. Uniform: Outputs are evenly distributed across the range
// 3. Fast: Sub-microsecond per key
// 4. Avalanche: Small input change → large output change

// Acceptable hash functions for production consistent hashing:
// - Murmur3 (64-bit or 128-bit): Cassandra, DynamoDB-inspired systems
// - xxHash (64-bit): High throughput for cache routing
// - SHA-1 (160-bit): Overkill but acceptable if already used
// - CRC32 / CRC16: Redis Cluster (CRC16), acceptable for small clusters
// 
// UNACCEPTABLE:
// - string.GetHashCode(): Non-deterministic
// - MD5: Deprecated, though still used in libketama
// - Simple XOR / adler32: Poor avalanche → hotspotting
```

**Symptom:** After a process restart, the ring is different. Keys route to different nodes. For a cache, this means a full cache flush — every request is a cache miss. For a database, this means requests route to the wrong shard — data integrity violations.

**Fix:** Use a well-known, deterministic, uniform hash function (Murmur3, xxHash, SHA-256 truncated). Store the hash function selection in configuration. Write a unit test that verifies hash function determinism: `Assert.Equal(hash("key"), hash("key"))` and `Assert.NotEqual(hash("key"), hash("key2"))`.

**Cost of not fixing:** Full cache flush on every restart. Wrong-shard routing for databases. A P0 incident every time the application deploys.

## Tradeoffs and Decision Framework

### When to Use Consistent Hashing vs Alternatives

| Factor | Consistent Hashing | Range-Based Sharding | Hash-Based (mod) | Directory-Based |
|---|---|---|---|---|
| Node add/remove impact | 1/N of keys move | All keys in range may need rebalancing | ~N/(N+1) of ALL keys move | Zero (metadata update only) |
| Ordered range queries | Not supported (scatter-gather) | Natively supported | Not supported | Supported if directory maps ranges |
| Hotspot handling | VNs distribute load, but hot keys still hotspot | Sequential keys on same node = hotspots | Modulo distributes evenly IF data is random | Directory can rebalance manually |
| Operational complexity | Medium (ring management, VNs) | Low (contiguous ranges) | Low (modulo routing) | High (directory server, metadata management) |
| Lookup cost | O(log N) binary search | O(1) with range index | O(1) | O(1) in-memory + 1 network hop |
| .NET tooling | Custom implementation or Cassandra driver | SQL Server partition functions | Simple modulo | Azure SQL Elastic Scale, custom shard map |
| Best for | Elastic caching, KV stores, session sharding | Time-series, ordered IDs, reporting | Small static clusters | Multi-tenant SaaS, tenant-per-shard |

### Decision Tree

```
Topology change frequency?
├── Frequent (auto-scaling, elastic) ──→ Consistent hashing or Directory-based
│   ├── Ordered range queries needed? ──→ Directory-based
│   └── Point lookups only? ──→ Consistent hashing (simpler ring, no directory server)
│
├── Rare (once per year maintenance) ──→ Range-based or Hash-based (mod)
│   ├── Ordered range queries needed? ──→ Range-based
│   └── Point lookups only? ──→ Hash-based (mod) — simplest
│
└── Manual (migrate tenants one at a time) ──→ Directory-based
    └── Each tenant is large (100 GB+) and independent
        → Directory-based allows per-tenant migration without cluster-wide rebalancing
```

### Scale Thresholds

- **Consistent hashing for caching is the default up to ~100 nodes** — beyond that, the gossip overhead of maintaining the ring becomes significant, and most systems switch to a configuration-service-based ring (ZooKeeper, etcd).
- **Consistent hashing for KV stores (Cassandra) is viable up to 1,000+ nodes** — Cassandra's gossip protocol scales with O(log N) messages per node and O(N log N) worst-case convergence.
- **Redis Cluster (16,384 slots) is practical up to ~1,000 nodes** — below 1,000 nodes, each node has 16+ slots; above 1,000, slot granularity becomes too coarse.
- **DynamoDB's pre-split partitions are manageable up to ~1 million per table** — beyond that, the metadata layer (partition metadata) becomes a bottleneck.
- **Discord-style per-guild sharding with consistent hashing is practical up to ~250 shards** — beyond that, the shard management overhead (rebalancing guilds, session migration) becomes the bottleneck.
- **Directory-based sharding becomes impractical above 10,000 shards** — the shard map directory becomes a bottleneck for read/write metadata operations.

### Real-World Case Studies

**Case Study 1: Amazon Dynamo (2007).** The Dynamo paper showed consistent hashing on a 128-bit MD5-based ring. Each node owns a contiguous range of tokens. Key insight: Dynamo allowed each node to have multiple tokens (not just one), which is the precursor to virtual nodes. Dynamo's approach was optimized for the high-availability, eventually-consistent (AP from CAP) workload of Amazon's shopping cart — where data loss is unacceptable but momentary inconsistency is acceptable.

**Case Study 2: Cassandra VNode adoption (2014).** Cassandra originally used single-token nodes (one contiguous range per node). Adding a node required manual token calculation and caused uneven load distribution. With CASSANDRA-7032 (v2.0, 2014), Cassandra introduced virtual nodes (num_tokens). The default was set to 256 (later reduced to 128). Result: load distribution improved from CV ~0.3–0.5 to CV ~0.09, and node addition became automatic (no token calculation).

**Case Study 3: Discord's guild sharding (2017–present).** Discord grew from a few shards to 250+. Each shard handles a set of guilds. Discord uses consistent hashing to map guild IDs to shards. When a shard reaches capacity (CPU or memory), a new shard is added, and guild IDs rebalance across shards. Key constraint: each guild's state (voice connections, message buffers) is in-memory. Rebalancing requires migrating WebSocket connections and session state — not data from disk.

**Case Study 4: Redis Cluster (2015–present).** Redis Cluster's 16,384-slot approach differs from pure consistent hashing in that slot ownership is explicit, not implicit (no hash ring). The key advantage: slot migration is O(keys_in_slot) with MOVED/ASK redirects, and the slot table fits in a 16 KB array (16,384 × 1 byte). The key limitation: the fixed slot count limits cluster scalability beyond ~1,000 nodes.

### Spoken Answers

**Q1: Name four production systems that use consistent hashing and how they differ.**

> **Average answer:** Cassandra, Dynamo, Redis Cluster, and Akamai CDN all use consistent hashing.

> **Great answer:** Four major production systems, each with a distinct implementation. First, **Akamai CDN** — the original production deployment from 1997. It uses consistent hashing to map URL paths to edge cache servers. No replication: cache miss = fetch from origin. The point is minimizing cache invalidation on edge server churn. Second, **Cassandra** — uses Murmur3-based consistent hashing with 128 virtual nodes per physical node. VNs distribute load evenly (CV ≈ 0.09) and enable automatic bootstrap. Replication factor determines durability; hinted handoff handles temporary failures. Third, **Redis Cluster** — uses 16,384 fixed hash slots instead of a dynamic hash ring. Keys map to slots via CRC16; slots map to nodes via a distributed array. Slot migration uses MOVED and ASK redirects for transparent resharding. Fourth, **DynamoDB** — uses a pre-split token ring managed internally. Users never see the ring; they specify a partition key and DynamoDB handles the partitioning. The key differentiators are replication strategy (DynamoDB RF=3 across AZs by default, Cassandra tunable), VN model (Cassandra uses interleaved VNs, DynamoDB uses contiguous pre-split ranges), and user visibility (DynamoDB is fully managed, Cassandra requires operator token management).

**Q2: Compare Cassandra's VNs with Redis Cluster's slots.**

> **Average answer:** Cassandra uses virtual nodes and Redis uses slots. They are similar.

> **Great answer:** Both distribute keys across nodes, but they differ in flexibility, granularity, and management. Cassandra's **virtual nodes** are dynamically created per node (default 128 per physical node). VNs are NOT fixed — when a node joins, it gets 128 new VNs interleaved with existing VNs. The hash ring grows as nodes are added. The advantage is elastic scalability: nodes can be added with any capacity weight (heterogeneous hardware). Redis Cluster's **16,384 slots** are fixed at cluster creation. Each slot is assigned to a node; slot ownership changes via explicit migration. The advantage is O(1) routing (array lookup vs binary search), small metadata (16 KB per node), and transparent resharding via MOVED/ASK redirects. The tradeoff: Cassandra's VNs are more flexible for heterogeneous clusters (weight each node by VN count) and elastic scaling (no slot count limit), while Redis Cluster's slots give simpler routing and easier client implementation. In practice, Cassandra scales to 1,000+ nodes, while Redis Cluster is practical to ~1,000 nodes due to the slot granularity limit.

**Q4: When would you choose DynamoDB over Cassandra?**

> **Great answer:** "Choose DynamoDB when you want turnkey global distribution with SLA-backed latency and throughput, at the cost of RU-based pricing and a 10,000 RU/s logical partition key limit, acceptable when your workload fits within a single partition key's throughput ceiling and you prefer managed operations. Choose Cassandra when you need tunable consistency, no logical partition throughput limit, and lower per-request cost at scale, at the cost of operational complexity and self-managed cluster maintenance, acceptable when you have an ops team and your workload requires custom consistency levels per operation."

**Q7: An IoT platform ingests 1 million events/second from 100,000 devices. Each device produces ordered events. Partition using consistent hashing — but how do you support per-device ordered event retrieval?**

> **Great answer:** We need to solve the conflict between consistent hashing (which scatters keys for load distribution) and ordered retrieval (which requires keys to be co-located). The solution is a two-level partitioning scheme. **Level 1 — Partition key:** Use `device_id` as the consistent hashing key. All events for a device go to the same physical partition. This preserves ordering WITHIN a device because all device events are co-located on the same partition. **Level 2 — Clustering/ordering column:** Within each partition, use a composite primary key `(device_id, timestamp)`. The partition key `device_id` determines the node (via consistent hashing). The clustering column `timestamp` provides sorted order within the partition for `SELECT * WHERE device_id = X ORDER BY timestamp`. This is exactly Cassandra's data model — partition key for distribution, clustering columns for ordering. The tradeoff is for distribution: if some devices produce 1,000× more events than others, those devices' partitions become hot. Mitigate with synthetic partition keys for heavy devices: `device_id_heavy` maps to a synthetic suffix, splitting the hot device across multiple partitions, at the cost of scatter-gather for the device's ordered read.

## ADR and Self-Check

### ADR Template: Partitioning Strategy Selection

**Status:** Proposed | Accepted | Deprecated | Superseded by [ADR-NNN]

**Context:** Our system needs to distribute data across [N] nodes. The workload is [read-heavy / write-heavy / mixed]. Key access patterns: [point lookups by ID / range queries by date / ordered scans / multi-key aggregations]. Topology changes occur [frequently / rarely / on-demand].

**Decision:** We will use [consistent hashing / range-based sharding / hash-based (mod) / directory-based sharding] for data distribution.

**Consequences:**
- Positive: [e.g., minimal data movement on topology change; natural support for ordered range queries; simple O(1) routing]
- Negative: [e.g., no native range query support; hot partition management required; directory server becomes SPOF]
- Risk: [e.g., cascading successor overload on node failure; hash function non-determinism; directory update contention]
- Mitigation: [e.g., VN=160 distributes load; Murmur3 hash with unit test verification; etcd-based directory with lease-based caching]

**Rationale:**
- "Use consistent hashing because it gives minimal data redistribution on topology changes (1/N of keys move), at the cost of no native range query support, acceptable when the workload is point-lookup-only and nodes are added/removed dynamically."
- "Use range-based sharding because it gives native range query support (single-shard scan), at the cost of manual range rebalancing when nodes are added, acceptable when topology changes are rare (once per year) and range queries are the primary access pattern."
- "Use directory-based sharding because it gives zero data movement on topology change (metadata update only), at the cost of a directory server becoming a single point of failure and a metadata bottleneck, acceptable when each data unit (tenant) is large and independent (100 GB+ per tenant) and cross-tenant queries are not required."

**Reversal:** If we see [hot partition imbalance exceeding 2× average load / range query latency exceeding 100 ms for 95th percentile / directory server latency exceeding 5 ms per lookup], we will reconsider the partitioning strategy and evaluate alternatives.

### Self-Check Quiz

**Q1:** Why does Akamai CDN use consistent hashing without replication?

**A1:** Because a CDN cache is a cache, not a durable store. If a cache node fails, the content is fetched from the origin server (or a parent cache). The cost is a cache miss, not data loss. Removing replication eliminates the write amplification of maintaining RF=3 copies per cache server, reducing hardware costs by ~3×.

---

**Q2:** Cassandra has VN=128 and N=10. A new node joins. How many VNs are added to the ring, and how many existing nodes see data movement?

**A2:** 128 VNs are added (one per existing node, on average). Each existing node loses ~12.8 VNs (128/10) to the new node. All 10 existing nodes see data movement of ~12.8/128 = 10% of their arcs. Total data movement: the new node receives ~10% of the total data, but that 10% comes from ALL 10 existing nodes in parallel, so data movement completes in ~1/10th the time it would take with single-token approach.

---

**Q3:** A Redis Cluster has 10 nodes. How many hash slots does each node own? If a node fails, what happens to its slots, and how does a client know about the change?

**A3:** Each node initially owns 16,384 / 10 ≈ 1,638 slots (with 4 leftover). If a node fails, its slots are inaccessible until a replica is promoted (if replicas are configured). Clients discover the change via MOVED redirects (client sends a request to the old owner, the node replies with MOVED slot_id new_owner_ip:port). The client updates its local slot table. Without replicas, the failed node's data is lost.

**Q4:** A social media app has 50M users and needs to partition user profiles. Each profile is a single JSON document. Profile reads are by user_id. Profile writes are by user_id. There is no range query across users. Choose a partitioning strategy.

**A4:** Consistent hashing by `user_id` is ideal. (1) Point lookups by user_id — consistent hashing maps user_id to one node deterministically. (2) No range queries — the lack of order support is irrelevant. (3) Profile growth means node count will change — consistent hashing minimizes data movement on node add/remove. (4) Each profile is atomic and small (KB-scale) — no need for intra-profile range queries. The implementation: use Murmur3 hash of user_id, VN=160 with 10 initial nodes. Monitor partition sizes — if a single user's profile grows large (e.g., profile with millions of posts), use a synthetic partition key: `user_id + "_profile"` for profile data, `user_id + "_post_{post_id}"` for individual posts within the profile.

**Q5:** For the same social media app, a new requirement: "Show me all users who joined in the last 24 hours." How do you support this with your consistent-hashing-based partition?

**A5:** Consistent hashing does not support range queries. You cannot query all users by `joined_date` without scanning ALL nodes. Solution: add a secondary index. (a) Write user profiles to the consistent-hashing partition (for point lookups by user_id). (b) Write user join events to a separate time-series store or Elasticsearch index (for the "new users" query). (c) Use Elasticsearch's range query for the "joined in last 24 hours" requirement. The list of user_ids from Elasticsearch is then used for point lookups against the consistent-hashing shard. This is the read-scatter pattern: first hit the secondary index (which supports the range query), then fan-out to consistent-hashing shards for the detail records.

**Q6:** An IoT platform partitions device events by device_id using consistent hashing. Some devices produce 100,000 events/day while most produce 10 events/day. What goes wrong?

**A6:** Hot partition problem. The high-volume devices' partitions become hotspots — the node owning those partitions sees 10,000× more write traffic than average. Consistent hashing distributes keys evenly, but if a few keys (hot devices) have much more traffic, the node hosting those keys is overloaded. Mitigation: (a) Use hashing per event, not per device: hash on `(device_id, event_id)` to distribute each event independently. But this breaks per-device ordered queries. (b) Use synthetic partition keys: for high-volume devices, split the device into N virtual devices: `device_id_0`, `device_id_1`, ..., `device_id_{N-1}`. Each virtual device maps to a different partition. Reads for the device fan-out across N partitions. (c) Use Cassandra's model: partition key = device_id, clustering column = timestamp. This co-locates all events for a device on one partition. For hot devices, Cassandra handles this with row-caching and compaction — up to ~10 GB per partition is practical.

**Q7:** A team uses `string.GetHashCode()` for their consistent hashing ring. During a deployment, the ring is rebuilt and all keys map to different nodes. What happened?

**A7:** `string.GetHashCode()` in .NET is not deterministic across application restarts, .NET versions, or build configurations. Each application startup produces different hash values for the same strings. When the ring was rebuilt during deployment, the hash values changed, so every key mapped to a different node than it did before. For a cache, this means a full cache flush — 100% cache miss rate until the caches warm up. For a database, this means requests route to the wrong shard — data integrity violations. Fix: Use a deterministic hash function (Murmur3, xxHash, SHA-256) that produces the same output for the same input across all environments and versions.

**Q8:** Compare the data-movement overhead of adding a node to Cassandra (VN=128, N=10, RF=3) vs adding a node to Redis Cluster (3 master nodes, 3 replica nodes).

**Q8:** In Cassandra with N=10, VN=128, RF=3: each key is stored on 3 nodes (primary + 2 replicas). When a new node joins, it acquires VNs and must stream data from its predecessors for each VN. Since RF=3, the new node must receive (a) primary data for VNs it owns and (b) replica data for VNs where it is the replica of another node. Total data moved: approximately 1/(N+1) of total cluster data × RF ≈ 10% × 3 = 30% of cluster data. But this is streamed in parallel from ~10 predecessors, and the streams are distributed across the VN assignments. In Redis Cluster with 3 masters, 3 replicas: each master owns ~5,461 slots. Adding a master requires migrating slots from existing masters. Redis slot migration copies keys from the source slot to the target slot, then updates the slot ownership. Total data moved: the number of keys that fit in the migrated slots. With 3 nodes, adding a 4th means migrating ~1/4 of slots (~4,096 slots) from existing nodes. Each slot migration uses the MIGRATE command (single-key or multi-key pipeline). Data copy rate is limited by network and source node CPU.

### Pitfall Summary

| Pitfall | Severity | Detection | Prevention |
|---|---|---|---|
| Using consistent hashing for range queries | High | Range queries hit all shards | Use range-based sharding or secondary index |
| Choosing consistent hashing for static clusters | Low | No topology changes for months | Use modulo routing for small, static clusters |
| Confusing routing with consistency | Critical | Stale reads after write | Add node-level (WAL) and cluster-level (quorum) consistency |
| Non-deterministic hash function | Critical | Full cache flush on restart | Use Murmur3, xxHash, or SHA-256 |
| Hot partition from uneven key distribution | High | Node CPU > 80% while others < 20% | Synthetic partition keys, scatter-gather reads |
| Ignoring replication in ring update | High | Data loss on node failure | Ensure RF ≥ 2 or data-first ring update |
| Misunderstanding slot vs VN granularity | Medium | Uneven load after migration | Match slot/VN count to cluster size and node capacity |
