# 7.021 — Strangler Fig Pattern — Migrating Legacy Systems

## Section 0 — Quick Reference Card

> [!ABSTRACT] Quick Reference Card
>
> **Intent:** Incrementally replace a monolithic legacy system by gradually intercepting and redirecting specific functionality to a new system through a proxy layer, until the legacy system can be safely decommissioned.
>
> **Also known as:** Incremental Migration, Gradual Replacement, Proxy-Based Migration, Strangulation Pattern
>
> **Category:** Migration & Modernization Pattern
>
> **Key Principle:** A proxy/gateway intercepts all requests and routes them to either the legacy system or the new system based on routing rules. As capabilities are migrated, the proxy shifts more traffic to the new system until the legacy system is "strangled" and removed.
>
> **Typical Flow:**
> 1. Identify a bounded capability (e.g., Order Submission, Inventory Lookup)
> 2. Build the new implementation in parallel — **do not modify the legacy system**
> 3. Configure the proxy intercepting layer to redirect the chosen capability
> 4. Monitor, test, validate — shift a percentage of traffic
> 5. Reach 100% on that capability, remove the legacy code path
> 6. Repeat for next capability until the entire legacy system is decommissioned
>
> **When to Use:**
> - Legacy system is stable but cannot evolve to meet new requirements
> - Rewriting the entire system at once is too risky or expensive
> - Business requires continuous delivery during the migration
> - Regulatory or compliance constraints demand incremental validation
>
> **When NOT to Use:**
> - Legacy system is trivial (< 5,000 lines of code) — a full rewrite is cheaper
> - Legacy system is already scheduled for decommissioning in < 3 months
> - The legacy data model cannot be mapped to the new system incrementally (e.g., tightly coupled monolithic schema with no bounded contexts)
> - No proxy or API gateway exists and adding one adds unacceptable latency (+5ms per hop) for real-time trading systems
>
> **Key Metrics:**
> - Migration velocity: capabilities migrated per sprint (target: >= 2)
> - Coexistence latency overhead: < 15ms p95 added by proxy layer
> - Dual-write inconsistency rate: < 0.01% of records out of sync
> - Rollback time: < 30 minutes for any single capability
> - Legacy decommission timeline: <= 18 months from first capability cutover
>
> **Common Tools (Azure/.NET):**
> - YARP (proxy routing), Azure API Management (policy-based routing), Feature Management (Microsoft.FeatureManagement), Azure App Configuration (dynamic routing rules), Azure SQL (dual-write coordination), Azure Service Bus (integration events)
>
> **Risk Profile:** Medium — lower than big-bang rewrite, but requires operational maturity in monitoring, feature flags, and rollback automation.

## Section 1 — Navigation & Context

> [!INFO] Production Encounter Map
>
> You will encounter the Strangler Fig pattern when:
>
> 1. **Your team is asked to modernize a monolith that processes $50M+ in annual transaction value** — the cost of failure in a full rewrite is too high, so incremental migration is the only viable path.
>
> 2. **A legacy system built on .NET Framework 4.x must move to .NET 8** — the framework version gap is too large for in-place upgrades; capabilities are migrated one at a time through a YARP proxy.
>
> 3. **A third-party vendor platform is being replaced by an in-house system** — the proxy layer lets you test the new system with real traffic while keeping the vendor system as a fallback during the 6-12 month coexistence period.
>
> 4. **Your architecture review board rejected a big-bang rewrite after a risk assessment** — the decision record explicitly cites the Strangler Fig pattern as the approved approach.
>
> 5. **An audit requires that you demonstrate migration safety to regulators** — each capability has a documented cut-over plan, rollback script, and validation dashboard.
>
> 6. **A startup acquired by an enterprise must integrate their platform while keeping both systems running** — the strangler proxy allows gradual integration without disrupting existing customers.
>
> **Place in the Clean Architecture Landscape:**
>
> The Strangler Fig pattern operates at the **infrastructure and deployment boundary** of Clean Architecture. It does not change the inner layers (entities, use cases) of either the legacy or new system. Instead, it introduces a **routing and interception layer** at the outermost boundary — the interface adapters and framework layers — that controls which system processes each request.
>
> This pattern works alongside [[7.020 — Anti-Corruption Layer Pattern]] to ensure the new system's domain model remains pure. The ACL translates legacy data contracts into the new system's types at the proxy boundary, while the Strangler Fig decides *which* system receives the request.
>

>
> This note is part of **Group 1 — Clean Architecture and Layering**. The Strangler Fig complements other layering patterns:
>
> - [[1.002 — Clean Architecture]] provides the conceptual layering that makes strangulation feasible — without clean boundaries, you cannot isolate a capability for migration.
> - [[7.020 — Anti-Corruption Layer Pattern]] provides the translation boundary at the strangler proxy.
> - [[2.013 — Repository Pattern]] is typically the first layer migrated in a strangler — data access is extracted from the legacy monolith into the new system.
>
> **Domain 7 Context:**
>
> Within [[7 — System Design & Distributed Systems]], the Strangler Fig is a **migration and evolution pattern**. It sits alongside other system design patterns for managing change in distributed environments:
>
> - [[7.008 — Circuit Breaker Pattern]] prevents cascading failures during migration
> - [[7.015 — Saga Pattern]] handles distributed transactions across legacy and new systems
> - [[7.019 — CQRS Pattern]] enables read-first migration strategies

## Section 2 — Core Mental Model

> [!TIP] Non-Obvious Insight
>
> The Strangler Fig is **not** primarily a software pattern — it is a **deployment and operations pattern** that imposes a software architecture requirement. The actual software change is the proxy layer and the ACL. The "strangling" happens in the deployment topology, not in the code. If you cannot independently deploy the proxy, the legacy system, and the new system as separate services, you cannot execute the Strangler Fig pattern.
>
> **Most teams get this wrong by:** building the proxy into the legacy application (tight coupling defeats the purpose) or building the proxy into the new application (you cannot decommission the legacy system cleanly). The proxy **must** be an independent deployable unit.

### Key Properties

| Property | Description |
|----------|-------------|
| **Incremental** | Capabilities are migrated one at a time; each is complete before the next begins |
| **Reversible** | Every capability migration can be rolled back by changing a feature flag; no data loss |
| **Coexistent** | Legacy and new systems run in parallel for months or years; data synchronization is required |
| **Transparent** | Clients do not know which system processes their request — the proxy abstracts the topology |

| **Non-Invasive** | The legacy system is not modified; all migration logic lives in the proxy and the new system |

## Section 3 — Deep Mechanics

### How It Works

The Strangler Fig pattern achieves incremental migration through three interdependent mechanisms:

**Mechanism 1 — Request Interception (The Proxy Layer)**

Every incoming request is routed through a reverse proxy that sits in front of both the legacy and new systems. The proxy evaluates each request against a route table and a feature flag state to determine the target system. The route table maps URL patterns, HTTP methods, and request attributes (e.g., tenant ID, customer tier, geographic region) to migration status.

\\\
Source: Client -> YARP Proxy -> [Route Table + Feature Flags] -> Legacy | New
\\\

The proxy MUST be:
- **Stateless**: Routing decisions are based solely on the request and feature flag state; no session affinity to either system
- **Fast**: Route evaluation must complete in < 5ms p99 to avoid compounding latency
- **Observable**: Every routing decision is logged with a correlation ID, the target system, and the feature flag version used
- **Configurable without deployment**: Route table changes and traffic percentage shifts should not require restarting the proxy

**Mechanism 2 — Routing Decision Evaluation (The Control Plane)**

The proxy evaluates a multi-step decision chain for each request:

1. **Extract route key**: Parse the URL, method, headers, and body to identify the capability
2. **Check capability status**: Query the feature flag system for the migration status. Valid statuses: legacy-only, partial-migrate, fully-migrated
3. **Evaluate traffic percentage** (if partial-migrate): Apply a consistent hash on a stable request attribute to determine which system processes this request
4. **Check circuit breaker**: If the new system's circuit breaker is open, fall back to legacy regardless of routing rules
5. **Transform request** (if routing to new): Apply ACL transformations to convert legacy contracts to new system contracts
6. **Route and track**: Send the request, record the decision in distributed tracing, monitor the response

**Mechanism 3 — Data Synchronization (The Coexistence Bridge)**

During the coexistence period, both systems process data. The synchronization strategy depends on which system is the source of truth:

- **Legacy as source of truth** (early migration): The new system reads from legacy databases (read-only) and writes to its own store. A background job or CDC pipeline synchronizes legacy changes to the new system.
- **New as source of truth** (late migration): The new system is the primary writer. Dual-writes publish integration events to Azure Service Bus, which are consumed by the legacy system to stay in sync.
- **Bidirectional sync** (complex scenarios): Both systems can write. Conflict resolution uses last-writer-wins with a timestamp authority or manual reconciliation queue. High-risk; avoid if possible.

|-------|---------------|-----------|-------------------|---------------|

### Failure Modes

> [!DANGER] 3AM Production Signal: Silent Data Divergence
>
> **Observable signal:** Reconciliation report shows 342 records out of sync between legacy and new systems for Order Management, but both systems return HTTP 200 for all API calls. Error rates are normal. P95 latency is unchanged. The inconsistency is discovered by a weekly batch job.
>
> **Root cause:** The dual-write mechanism uses Azure Service Bus with AtLeastOnce delivery, but the legacy consumer processes events out of order. When two updates arrive for the same order within 100ms, the legacy system applies them in reverse order, causing the final state to differ.
>
> **Detection:** Add a version field (Order.Version as monotonically increasing integer) to all integration events. The legacy consumer rejects out-of-order events and writes them to a dead-letter queue for manual reconciliation.
>
> **Recovery:**
> 1. Identify the 342 divergent records using the reconciliation query
> 2. Determine source of truth (new system, since migration is at 100%)
> 3. Replay the correct state from new -> legacy via a repair job
> 4. Add event sequencing validation to the legacy consumer
> 5. Increase reconciliation frequency from weekly to hourly during active migration
>
> **Prevention:**
> \\\csharp
> // Add idempotency key and sequence number to integration events
> public sealed record OrderUpdatedIntegrationEvent(
>     Guid EventId,
>     Guid OrderId,
>     int Version,
>     OrderState State,
>     DateTime OccurredAtUtc,
>     Guid IdempotencyKey)
> {
>     public bool IsSequential(int lastProcessedVersion) => Version == lastProcessedVersion + 1;
> }
> \\\

> [!DANGER] 3AM Production Signal: Cascade Failure During Partial Migration
>
> **Observable signal:** At 03:14 UTC, the YARP proxy's P99 latency jumps from 12ms to 4,200ms. All routes to the new system return 503. The legacy system's P95 latency also rises to 2,000ms. The on-call engineer sees "ThreadPool starvation" in the proxy's .NET runtime counters.
>
> **Root cause:** The new system's circuit breaker opened due to a transient Azure SQL throttling event. The proxy's fallback logic is implemented synchronously — every request that times out on the new system blocks a thread waiting for the fallback to complete on the legacy system. Under 2,000 RPS, thread pool exhaustion causes all proxy operations (including feature flag queries) to queue.
>
> **Detection:** Thread pool starvation is visible in dotnet-counters as 	hreadpool-queue-length > 1000 and 	hreadpool-thread-count at maximum. The proxy's CPU is at 40% but throughput drops to near zero.
>
> **Recovery:**
> 1. Scale the proxy horizontally from 3 to 10 instances (takes 3 min with AKS HPA)
> 2. Disable the circuit breaker fallback temporarily (route all traffic to legacy)
> 3. After the new system recovers, re-enable circuit breaker with a semaphore-slim limit of max 50 concurrent fallback requests
> 4. Implement async fallback using Task.WhenAny with per-request timeout
>
> **Prevention:**
> \\\csharp
> // Use semaphore-slim to limit concurrent fallback requests
> public sealed class CircuitBreakerFallbackHandler
> {
>     private readonly SemaphoreSlim _fallbackSemaphore = new(initialCount: 50, maxCount: 50);
>
>     public async Task<HttpResponseMessage> ExecuteWithFallbackAsync(
>         HttpRequestMessage request,
>         CancellationToken cancellationToken)
>     {
>         var timeoutPolicy = HttpPolicyExtensions
>             .HandleTransientHttpError()
>             .Or<TimeoutRejectedException>()
>             .CircuitBreakerAsync(
>                 handledEventsAllowedBeforeBreaking: 5,
>                 durationOfBreak: TimeSpan.FromSeconds(30));
>
>         try
>         {
>             await _fallbackSemaphore.WaitAsync(cancellationToken);
>             return await timeoutPolicy.ExecuteAsync(
>                 () => _httpClient.SendAsync(request, cancellationToken));
>         }
>         finally
>         {
>             _fallbackSemaphore.Release();
>         }
>     }
> }
> \\\

> [!DANGER] 3AM Production Signal: Feature Flag Drift
>
> **Observable signal:** Ten percent of requests for order-creation are routed to the legacy system even though the feature flag shows status: fully-migrated and percentage: 100. The discrepancy is not consistent — the same customer ID is sometimes routed to legacy, sometimes to new.
>
> **Root cause:** The YARP proxy caches feature flags with a 60-second TTL, but the real issue is that two of the five proxy instances have stale cache because their AppConfigFeatureFlagProvider is configured with CacheExpirationInterval = TimeSpan.FromMinutes(5) while the others use TimeSpan.FromSeconds(60). The proxy instances are not configured identically.
>
> **Detection:** Checking the X-Route-Decision header (added by the proxy for debugging) shows different values for the same request hitting different proxy instances. The dashboard shows a "route split" — a non-zero percentage going to legacy even at 100% migration.
>
> **Recovery:**
> 1. Unify the proxy configuration by deploying from a single configuration package
> 2. Force cache invalidation by calling ppConfigProvider.ClearCache() on all instances
> 3. Verify all 5 instances return the same routing decision for a test request
> 4. Add a health check endpoint that reports the current feature flag version
>
> **Prevention:**
> - Use Azure App Configuration's snapshot feature for atomic flag updates
> - Add a deployment-time validation test that verifies identical configuration across instances
> - Monitor X-Route-Decision header distribution as a dashboard metric

/// <summary>
/// Evaluates routing rules and feature flags to determine the target system for each request.
/// </summary>
public sealed class StranglerRoutingEngine
{
    private readonly IFeatureFlagManager _featureFlags;
    private readonly IMemoryCache _routeCache;
    private readonly ILogger<StranglerRoutingEngine> _logger;
    private readonly StranglerProxyOptions _options;
    private static readonly ConcurrentDictionary<string, int> _routeFrequency
        = new();

    public StranglerRoutingEngine(
        IFeatureFlagManager featureFlags,
        IMemoryCache routeCache,
        IOptionsSnapshot<StranglerProxyOptions> options,
        ILogger<StranglerRoutingEngine> logger)
    {
        _featureFlags = featureFlags;
        _routeCache = routeCache;
        _logger = logger;
        _options = options.Value;
    }

    /// <summary>
    /// Determines the routing target for the given HTTP context.
    /// </summary>
    public async ValueTask<RoutingDecision> EvaluateAsync(
        HttpContext httpContext,
        CancellationToken cancellationToken)
    {
        var sw = ValueStopwatch.StartNew();
        var routeKey = ExtractRouteKey(httpContext);

        try
        {
            var state = await GetMigrationStateAsync(routeKey, cancellationToken);
            var correlationId = httpContext.TraceIdentifier;

            httpContext.Items["RouteKey"] = routeKey;
            httpContext.Items["CorrelationId"] = correlationId;
            _routeFrequency.AddOrUpdate(routeKey, 1, (_, count) => count + 1);

            return state.Status switch
            {
                MigrationStatus.LegacyOnly => new RoutingDecision
                {
                    RouteKey = routeKey,
                    TargetUri = BuildLegacyUri(httpContext),
                    TargetSystem = "legacy",
                    RequiresTransformation = false,
                    CorrelationId = correlationId,
                    EvaluationDuration = sw.GetElapsedTime()
                },

                MigrationStatus.FullyMigrated => new RoutingDecision
                {
                    RouteKey = routeKey,
                    TargetUri = BuildNewSystemUri(httpContext),
                    TargetSystem = "new",
                    RequiresTransformation = true,
                    CorrelationId = correlationId,
                    EvaluationDuration = sw.GetElapsedTime()
                },

                MigrationStatus.PartialMigrate => await EvaluatePartialMigrationAsync(
                    httpContext, routeKey, state, correlationId, sw, cancellationToken),

                MigrationStatus.RolledBack => new RoutingDecision
                {
                    RouteKey = routeKey,
                    TargetUri = BuildLegacyUri(httpContext),
                    TargetSystem = "legacy",
                    RequiresTransformation = false,
                    CorrelationId = correlationId,
                    FallbackReason = "RolledBack",
                    EvaluationDuration = sw.GetElapsedTime()
                },

                _ => throw new InvalidOperationException($"Unknown migration status: {state.Status}")
            };
        }
        catch (Exception ex) when (ex is not OperationCanceledException)
        {
            _logger.LogError(ex, "Routing evaluation failed for route key {RouteKey}", routeKey);
            return CreateFallbackDecision(httpContext, routeKey, "EvaluationFailure", sw);
        }
    }

    private async ValueTask<RoutingDecision> EvaluatePartialMigrationAsync(
        HttpContext httpContext,
        string routeKey,
        CapabilityMigrationState state,
        Guid correlationId,
        ValueStopwatch sw,
        CancellationToken cancellationToken)
    {
        // Use consistent hashing on a stable request attribute
        var consistencyKey = ExtractConsistencyKey(httpContext);
        var hash = Math.Abs(consistencyKey.GetDeterministicHashCode()) % 100;
        var routeToNew = hash < state.TrafficPercentage;

        if (routeToNew)
        {
            httpContext.Items["CircuitBreakerCheck"] = true;
            var circuitBreakerState = await CheckCircuitBreakerAsync(
                routeKey, cancellationToken);

            if (!circuitBreakerState.IsOpen)
            {
                return new RoutingDecision
                {
                    RouteKey = routeKey,
                    TargetUri = BuildNewSystemUri(httpContext),
                    TargetSystem = "new",
                    RequiresTransformation = true,
                    CorrelationId = correlationId,
                    EvaluationDuration = sw.GetElapsedTime()
                };
            }

            _logger.LogWarning(
                "Circuit breaker OPEN for {RouteKey}, falling back to legacy. " +
                "Open since: {OpenedAt}, failures: {FailureCount}",
                routeKey, circuitBreakerState.OpenedAt, circuitBreakerState.FailureCount);

            httpContext.Items["CircuitBreakerFallback"] = true;
        }

        return new RoutingDecision
        {
            RouteKey = routeKey,
            TargetUri = BuildLegacyUri(httpContext),
            TargetSystem = "legacy",
            RequiresTransformation = false,
            CorrelationId = correlationId,
            FallbackReason = routeToNew ? "CircuitBreakerOpen" : "ConsistentHashLegacy",
            EvaluationDuration = sw.GetElapsedTime()
        };
    }

    private async ValueTask<CapabilityMigrationState> GetMigrationStateAsync(
        string routeKey, CancellationToken cancellationToken)
    {
        var cacheKey = $"migration-state:{routeKey}";

        return await _routeCache.GetOrCreateAsync(
            cacheKey,
            async entry =>
            {
                entry.AbsoluteExpirationRelativeToNow =
                    TimeSpan.FromSeconds(_options.CacheDurationSeconds);

                var featureFlagName = $"strangler/{routeKey}/status";
                var flagState = await _featureFlags
                    .GetFlagStateAsync(featureFlagName, cancellationToken);

                var percentageFlagName = $"strangler/{routeKey}/percentage";
                var percentage = await _featureFlags
                    .GetFlagPercentageAsync(percentageFlagName, cancellationToken);

                return MapToMigrationState(routeKey, flagState, percentage);
            })!;
    }

    private static string ExtractRouteKey(HttpContext httpContext)
    {
        var path = httpContext.Request.Path.Value?.Trim('/').ToLowerInvariant() ?? string.Empty;
        var method = httpContext.Request.Method.ToUpperInvariant();

        return (method, path) switch
        {
            ("POST", "/api/orders" or var p when p.StartsWith("api/orders")) => "order-creation",
            ("GET", var p) when p.StartsWith("api/orders/") => "order-retrieval",
            ("PUT", var p) when p.StartsWith("api/orders/") => "order-update",
            ("GET", var p) when p.StartsWith("api/inventory/") => "inventory-lookup",
            ("POST", var p) when p.StartsWith("api/shipping/") => "shipping-creation",
            _ => "unknown"
        };
    }

    private static string ExtractConsistencyKey(HttpContext httpContext)
    {
        return httpContext.Request.Headers["X-Customer-Id"].FirstOrDefault()
               ?? httpContext.Request.Headers["X-Tenant-Id"].FirstOrDefault()
               ?? httpContext.Connection.RemoteIpAddress?.ToString()
               ?? Guid.NewGuid().ToString();
    }

    private static Uri BuildLegacyUri(HttpContext httpContext)
    {
        return new Uri($"http://legacy-orders.internal:8080{httpContext.Request.Path}");
    }

    private static Uri BuildNewSystemUri(HttpContext httpContext)
    {
        return new Uri($"http://orders-service.prod:8080{httpContext.Request.Path}");
    }

    private static CapabilityMigrationState MapToMigrationState(
        string routeKey, string flagState, int percentage)
    {
        return flagState.ToLowerInvariant() switch
        {
            "legacy-only" => new CapabilityMigrationState
            {
                CapabilityName = routeKey,
                RouteKey = routeKey,
                Status = MigrationStatus.LegacyOnly,
                TrafficPercentage = 0,
                LastModifiedUtc = DateTime.UtcNow
            },
            "partial-migrate" => new CapabilityMigrationState
            {
                CapabilityName = routeKey,
                RouteKey = routeKey,
                Status = MigrationStatus.PartialMigrate,
                TrafficPercentage = Math.Clamp(percentage, 0, 99),
                LastModifiedUtc = DateTime.UtcNow
            },
            "fully-migrated" => new CapabilityMigrationState
            {
                CapabilityName = routeKey,
                RouteKey = routeKey,
                Status = MigrationStatus.FullyMigrated,
                TrafficPercentage = 100,
                LastModifiedUtc = DateTime.UtcNow
            },
            _ => new CapabilityMigrationState
            {
                CapabilityName = routeKey,
                RouteKey = routeKey,
                Status = MigrationStatus.LegacyOnly,
                TrafficPercentage = 0,
                LastModifiedUtc = DateTime.UtcNow
            }
        };
    }

    private static RoutingDecision CreateFallbackDecision(
        HttpContext httpContext, string routeKey, string reason, ValueStopwatch sw)
    {
        return new RoutingDecision
        {
            RouteKey = routeKey,
            TargetUri = BuildLegacyUri(httpContext),
            TargetSystem = "legacy",
            RequiresTransformation = false,
            CorrelationId = Guid.NewGuid(),
            FallbackReason = reason,
            EvaluationDuration = sw.GetElapsedTime()
        };
    }

    private async ValueTask<CircuitBreakerState> CheckCircuitBreakerAsync(
        string routeKey, CancellationToken cancellationToken)
    {
        await Task.CompletedTask;
        return new CircuitBreakerState { IsOpen = false };
    }
}

{
    public bool IsOpen { get; init; }
    public DateTime? OpenedAt { get; init; }
    public int FailureCount { get; init; }
}

/// <summary>
/// Transforms incoming legacy-format requests to new-format requests before forwarding.
/// </summary>
public sealed class RequestTransformationMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<RequestTransformationMiddleware> _logger;
    private readonly IEnumerable<IRequestTransformer> _transformers;

    public RequestTransformationMiddleware(
        RequestDelegate next,
        IEnumerable<IRequestTransformer> transformers,
        ILogger<RequestTransformationMiddleware> logger)
    {
        _next = next;
        _transformers = transformers;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext httpContext)
    {
        if (httpContext.Items["TargetSystem"] is not "new")
        {
            await _next(httpContext);
            return;
        }

        var routeKey = httpContext.Items["RouteKey"] as string;
        if (string.IsNullOrEmpty(routeKey))
        {
            await _next(httpContext);
            return;
        }

        try
        {
            httpContext.Request.EnableBuffering();

            foreach (var transformer in _transformers)
            {
                if (transformer.CanTransform(routeKey))
                {
                    await transformer.TransformAsync(httpContext.Request);
                }
            }

            await _next(httpContext);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Request transformation failed for route {RouteKey}", routeKey);
            httpContext.Items["TransformationFailed"] = true;
            httpContext.Response.StatusCode = 502;
            await httpContext.Response.WriteAsJsonAsync(new
            {
                error = "request_transformation_failed",
                routeKey,
                message = "The proxy was unable to transform this request for the new system."
            });
        }
    }
}

/// <summary>
/// Transforms legacy order creation requests to the new system's schema.
/// </summary>
public sealed class OrderCreationTransformer : IRequestTransformer
{
    private static readonly FrozenSet<string> SupportedRoutes =
        new[] { "order-creation" }.ToFrozenSet();

    public bool CanTransform(string routeKey) => SupportedRoutes.Contains(routeKey);

    public async Task TransformAsync(HttpRequest request)
    {
        var body = await request.Body.ReadAsStringAsync();
        var legacyOrder = JsonSerializer.Deserialize<LegacyOrderRequest>(body)
            ?? throw new InvalidOperationException("Failed to deserialize legacy order request");

        var newOrder = new NewOrderRequest
        {
            CustomerId = legacyOrder.CustomerId,
            LineItems = legacyOrder.Items
                .Select(i => new LineItem
                {
                    ProductId = i.Sku,
                    Quantity = i.Qty,
                    UnitPrice = i.Price
                })
                .ToList(),
            RequestedAt = DateTime.UtcNow,
            Source = "legacy-migration"
        };

        var newBody = JsonSerializer.SerializeToUtf8Bytes(newOrder);
        request.Body = new MemoryStream(newBody);
        request.ContentType = "application/json";
        request.ContentLength = newBody.Length;

        await Task.CompletedTask;
    }
}

/// <summary>
/// Publishes integration events when the new system processes a request,
/// allowing the legacy system to stay synchronized during coexistence.
/// </summary>
public sealed class MigrationEventPublisher : IAsyncDisposable
{
    private readonly ServiceBusSender _sender;
    private readonly ILogger<MigrationEventPublisher> _logger;

    public MigrationEventPublisher(
        ServiceBusClient serviceBusClient,
        IOptionsSnapshot<StranglerProxyOptions> options,
        ILogger<MigrationEventPublisher> logger)
    {
        var topicName = options.Value.ReconciliationTopicName;
        _sender = serviceBusClient.CreateSender(topicName);
        _logger = logger;
    }

    public async Task PublishMigrationEventAsync(
        string routeKey,
        string correlationId,
        string legacyEntityId,
        string newEntityId,
        string action,
        object? payload = null,
        CancellationToken cancellationToken = default)
    {
        var integrationEvent = new MigrationIntegrationEvent
        {
            EventId = Guid.NewGuid(),
            RouteKey = routeKey,
            CorrelationId = correlationId,
            LegacyEntityId = legacyEntityId,
            NewEntityId = newEntityId,
            Action = action,
            Payload = payload,
            OccurredAtUtc = DateTime.UtcNow,
            EventVersion = 2
        };

        var message = new ServiceBusMessage(BinaryData.FromObjectAsJson(integrationEvent))
        {
            MessageId = integrationEvent.EventId.ToString(),
            CorrelationId = correlationId,
            ApplicationProperties =
            {
                ["routeKey"] = routeKey,
                ["action"] = action,
                ["eventVersion"] = 2
            }
        };

        try
        {
            await _sender.SendMessageAsync(message, cancellationToken);
            _logger.LogDebug(
                "Published migration event {EventId} for {RouteKey} ({Action}) " +
                "legacy={LegacyId} new={NewId}",
                integrationEvent.EventId, routeKey, action, legacyEntityId, newEntityId);
        }
        catch (ServiceBusException ex) when (ex.Reason == ServiceBusFailureReason.MessagingEntityNotFound)
        {
            _logger.LogCritical(ex,
                "Service Bus topic {TopicName} not found. Dual-write events will be lost!",
                _sender.EntityPath);
            throw;
        }
    }

    public sealed record MigrationIntegrationEvent
    {
        public Guid EventId { get; init; }
        public string RouteKey { get; init; } = string.Empty;
        public string CorrelationId { get; init; } = string.Empty;
        public string LegacyEntityId { get; init; } = string.Empty;
        public string NewEntityId { get; init; } = string.Empty;
        public string Action { get; init; } = string.Empty;
        public object? Payload { get; init; }
        public DateTime OccurredAtUtc { get; init; }
        public int EventVersion { get; init; }
    }

    public async ValueTask DisposeAsync()
    {
        await _sender.DisposeAsync();
        GC.SuppressFinalize(this);
    }
}

### Common Variants

| Variant | Description | When to Use | Tradeoff |
|---------|-------------|-------------|----------|
| **Proxy-Based Strangler** (Primary) | Dedicated reverse proxy handles all routing decisions | Most scenarios; clear separation of concerns | Added network hop (+3-8ms latency), proxy becomes SPOF |
| **API Gateway Strangler** | Routing rules embedded in API Management / Azure Front Door | Organization already uses API gateway; wants policy-based routing | Less control over transformation logic; gateway policy languages are limited |
| **Client-Side Strangler** | Client apps updated to call new system directly for migrated capabilities | Mobile apps with feature flag SDKs; when proxy cannot be inserted | Requires client updates; harder to roll back; inconsistent UX |
| **Database-First Strangler** | Migrate data access first, then business logic; new system reads/writes shared DB | Database is the primary bottleneck; new system uses same schema initially | Data coupling prevents clean separation; hard to decommission old schema |
| **Branch-by-Abstraction** | Add abstraction layer in the legacy codebase itself; swap implementations | In-process migration within a single deployable unit; when network boundary isn't feasible | Legacy codebase must be modified; partial migration is harder |
| **Cell-Based Strangler** | Tenant or customer cohorts are migrated in groups | Multi-tenant SaaS with tenant isolation; regulated industries needing per-customer validation | Complex routing rules; per-cell state management; slower overall |

## Section 5 — Gotchas and Production Pitfalls

> [!DANGER] Pitfall 1: Route Table Drift (Architecture-Level)
>
> **The Problem:** The proxy's route table is maintained separately from the actual deployment topology. When a new microservice is added to the new system, the route table is not updated. Requests that SHOULD be routed to the new system are sent to the legacy system, creating a "dark" migration that operators aren't tracking.
>
> **Production Signal:** Migration dashboard shows a capability stuck at "Partially Migrated — 95%" for 3 weeks. Reconciliation reports show no new records in the new system for that capability despite the feature flag showing 100%. No errors are logged — the legacy system silently handles the traffic.
>
> **Root Cause:** The route table in Azure App Configuration was never updated when the shipping microservice was deployed. The proxy evaluates strangler/shipping-creation/status = fully-migrated but matches the incoming URL pattern /api/shipping/create against legacy patterns only, defaulting to legacy.
>
> **Prevention:**
> - Automate route table generation from the actual service discovery registry (e.g., Consul, AKS Service discovery)
> - Add a "coverage check" health endpoint that verifies every known capability has a matching route
> - Use NetArchTest to enforce that route table entries exist for all new service endpoints
> - Deploy a canary proxy instance that validates 100% of the route table against actual traffic before promoting to production

> [!DANGER] Pitfall 2: Dual-Write Ordering Violations (.NET-Specific)
>
> **The Problem:** The legacy system receives events from Azure Service Bus but processes them in the wrong order because the consumer is parallel. Two updates to the same order arrive within 50ms; the later update is processed first because of thread scheduling.
>
> **Production Signal:** Reconciliation job finds 23 orders where the legacy system has a different lastModifiedDate than the new system. The discrepancy pattern shows a specific time window around 14:32:17 where 12 orders have out-of-order applied updates.
>
> **Root Cause:** The legacy event consumer uses ServiceBusProcessor with MaxConcurrentCalls = 10 and no session-based ordering. When the same OrderId has two events in the queue, they can be dispatched to different processor threads and applied in any order.
>
> **Prevention:**
> \\\csharp
> // Use Service Bus Sessions for per-order ordering
> var processor = serviceBusClient.CreateSessionProcessor("topic", "subscription",
>     new ServiceBusSessionProcessorOptions
>     {
>         MaxConcurrentSessions = 100,
>         MaxConcurrentCallsPerSession = 1,
>         SessionIdleTimeout = TimeSpan.FromSeconds(30)
>     });
>
> // Enable sessions when publishing:
> var message = new ServiceBusMessage(body)
> {
>     SessionId = orderId.ToString(),
>     MessageId = $"order:{orderId}:{version}"
> };
> \\\
>
> **Detection:**
> - Add a Version field to all events; reject out-of-order versions
> - Monitor the dead-letter queue for version-rejected events
> - Set up an alert when DLQ depth > 0 for migration events

> [!DANGER] Pitfall 3: Session Affinity Breakage (Architecture-Level)
>
> **The Problem:** The legacy system uses in-process session state (ASP.NET Session State). When the proxy routes a request to the new system but subsequent requests in the same session go to the legacy system (or vice versa), the user's session state is lost.
>
> **Production Signal:** Users report being "logged out" intermittently during multi-step order workflows. The error rate spikes during the checkout wizard but not on single-step pages. Affected users have X-Customer-Id that falls in the 25-35% hash range (the partially migrated bracket).
>
> **Root Cause:** A 3-step checkout wizard routes Step 1 (cart review) to the legacy system, Step 2 (shipping) to the new system (25% migration bracket hit the new system), and Step 3 (payment) back to legacy. The new system does not share session state with the legacy system.
>
> **Prevention:**
> - Use consistent hashing on the session ID (not customer ID) for routing during partial migration
> - Extract session state from the legacy system and store it in a shared Redis cache during transition
> - Migrate entire user workflows, not individual endpoints
> - Add middleware that detects session flow violations and redirects all session requests to the same system

> [!DANGER] Pitfall 4: Timeout Mismatch Between Systems (.NET-Specific)
>
> **The Problem:** The legacy system has a 60-second request timeout; the new system has a 5-second timeout. The proxy uses a blanket 30-second timeout. When the new system returns a 503 after 5 seconds, the proxy's fallback retries on legacy, but the client has already timed out (client timeout = 10 seconds).
>
> **Production Signal:** Error rate shows 0.5% "timeout" errors on POST /api/orders that do NOT correlate with any backend error — both legacy and new systems show 100% success. The proxy logs show fallback completions, but the client logs show HTTP 408/504.
>
> **Prevention:**
> \\\csharp
> // Use a per-request timeout that accounts for fallback budget
> public sealed class TimeboxedRequestHandler
> {
>     private readonly TimeSpan _clientTimeout;
>     private readonly TimeSpan _primaryTimeout;
>     private readonly TimeSpan _fallbackTimeout;
>
>     public TimeboxedRequestHandler(IOptions<TimeoutOptions> options)
>     {
>         _clientTimeout = options.Value.ClientTimeout;
>         _primaryTimeout = TimeSpan.FromMilliseconds(_clientTimeout.TotalMilliseconds * 0.6);
>         _fallbackTimeout = TimeSpan.FromMilliseconds(_clientTimeout.TotalMilliseconds * 0.4);
>     }
>
>     public async Task<HttpResponseMessage> ExecuteAsync(
>         HttpRequestMessage request, CancellationToken ct)
>     {
>         using var clientTimeoutCts = CancellationTokenSource.CreateLinkedTokenSource(ct);
>         clientTimeoutCts.CancelAfter(_clientTimeout);
>
>         try
>         {
>             var primaryCts = CancellationTokenSource.CreateLinkedTokenSource(
>                 clientTimeoutCts.Token);
>             primaryCts.CancelAfter(_primaryTimeout);
>
>             var response = await _httpClient.SendAsync(request, primaryCts.Token);
>             if (response.IsSuccessStatusCode) return response;
>
>             var elapsed = _clientTimeout - _primaryTimeout;
>             var remaining = _clientTimeout - elapsed;
>             if (remaining <= TimeSpan.Zero) throw new TimeoutException("No time for fallback");
>
>             var fallbackCts = CancellationTokenSource.CreateLinkedTokenSource(
>                 clientTimeoutCts.Token);
>             fallbackCts.CancelAfter(remaining);
>
>             return await _fallbackClient.SendAsync(request, fallbackCts.Token);
>         }
>         catch (OperationCanceledException)
>         {
>             throw new TimeoutException(
>                 $"Request could not be completed within {_clientTimeout.TotalMilliseconds}ms");
>         }
>     }
> }
> \\\

> [!DANGER] Pitfall 5: Rollback Race Condition (Architecture-Level)
>
> **The Problem:** When rolling back a capability from "Fully Migrated" to "Legacy Only", the feature flag change propagates asynchronously across proxy instances. During propagation (10-60 seconds), some instances route to new while others route to legacy. If the new system is being de-provisioned as part of the rollback, requests hitting the new system fail with 503.
>
> **Production Signal:** During a rollback of "inventory-lookup", 3% of requests fail with 503 for 45 seconds even though the feature flag shows "legacy-only" and the legacy system is healthy.
>
> **Root Cause:** The rollback procedure scales new system to 0 before updating the feature flag. When the new system is scaled down first, the 10-60 second stale cache window causes proxy instances to route to a non-existent system.
>
> **Prevention (correct order):**
> \\\
> 1. Update feature flag: strangler/inventory-lookup/status = "legacy-only"
> 2. Wait 2x cache duration (configurable; default = 60 seconds)
> 3. Check all proxy instances: GET /ops/migration-status -> verify status = "legacy-only"
> 4. Wait for in-flight requests to drain (30 seconds or monitor active request count)
> 5. ONLY THEN: Scale new system resources down
> \\\

> [!DANGER] Pitfall 6: Azure API Management Policy Limits (Azure-Specific)
>
> **The Problem:** The strangler routing logic is implemented as Azure API Management (APIM) policies using XML policy expressions. As the migration progresses from 5 to 50 capabilities, the policy document grows beyond APIM's 64 KB policy size limit.
>
> **Production Signal:** A PUT to APIM policy endpoint returns 400 Bad Request with error code PolicyContentSizeExceeded. The deployment pipeline fails. No migration can proceed until the policy is refactored.
>
> **Prevention:**
> - Split routing across multiple APIM APIs (one per domain) rather than one monolithic policy
> - Move routing logic to a backend YARP proxy and use APIM only for authentication and rate limiting
> - Use APIM's \set-backend-service\ policy with a routing service URL — routing logic lives in the backend
>
> \\\xml
> <!-- APIM policy (thin): delegate routing to backend -->
> <inbound>
>     <base />
>     <authentication-managed-identity resource="https://proxy.internal" />
>     <set-backend-service base-url="https://strangler-proxy.azurewebsites.net" />
> </inbound>
> \\\

> [!DANGER] Pitfall 7: Monitoring Blind Spot During Migration (Observability)
>
> **The Problem:** Application Insights is configured with server-side sampling at 5% for the legacy system but 100% for the new system. During partial migration, operators compare error rates and conclude incorrectly that the new system is worse.
>
> **Production Signal:** The migration dashboard shows new system error rate at 0.5% while legacy is at 0.02%. The team halts migration. After 2 weeks of investigation, they discover the legacy error rate is actually 0.6% — the legacy sampling rate of 5% gave false confidence.
>
> **Prevention:**
> - Set identical sampling rates for both systems during migration comparison periods
> - Add sample size and confidence intervals to the migration dashboard
> - Create a dedicated "migration telemetry" processor that never samples migration-related events

> [!DANGER] Pitfall 8: Stale Data Visibility After Migration (Data Consistency)
>
> **The Problem:** After an order capability is 100% migrated, the new system becomes the source of truth. But the legacy system's reporting tools and data warehouse ETL processes still read from the legacy database, which no longer receives updates for migrated orders.
>
> **Production Signal:** The finance team reports monthly revenue showing a 12% drop compared to the previous month. The drop coincides with "order-creation" reaching 100% migration. The new system has all orders correctly, but the data warehouse shows zero orders from the legacy system.
>
> **Prevention:**
> - For each capability migration, create a "Data Consumer Migration" work item covering all downstream consumers
> - During "Full Migration", run dual-writes AND dual-reads with a comparison check
> - Use a data virtualization layer (e.g., Azure Data Lake with PolyBase) that queries both systems
> - Maintain a "data source map" tracking which system is the source of truth per domain

> [!DANGER] Pitfall 9: Certificate and Authentication Split (Security)
>
> **The Problem:** The legacy system uses Windows Authentication (Kerberos) integrated with AD. The new system uses Azure AD (Microsoft Entra ID) with OAuth 2.0 and managed identities. When requests are routed to the new system, the Kerberos token is rejected.
>
> **Production Signal:** Authentication failure rate jumps from 0.1% to 15% on the new system's routes. Error is consistently 401 Unauthorized with WWW-Authenticate: Bearer. New system logs show "IDX10503: Signature validation failed".
>
> **Prevention:**
> - Have the ACL perform token translation: validate incoming Kerberos token, generate new JWT Bearer token
> - Configure the proxy to strip the Authorization header; let the ACL handle auth entirely
> - Use Azure Front Door with Azure AD authentication at the edge
>
> \\\csharp
> public sealed class TokenTranslationDelegatingHandler : DelegatingHandler
> {
>     private readonly DefaultAzureCredential _credential;
>
>     public TokenTranslationDelegatingHandler(DefaultAzureCredential credential)
>     {
>         _credential = credential;
>     }
>
>     protected override async Task<HttpResponseMessage> SendAsync(
>         HttpRequestMessage request, CancellationToken cancellationToken)
>     {
>         request.Headers.Remove("Authorization");
>         var tokenRequest = new TokenRequestContext(
>             ["https://orders-service.azurewebsites.net/.default"]);
>         var token = await _credential.GetTokenAsync(tokenRequest, cancellationToken);
>         request.Headers.Authorization =
>             new AuthenticationHeaderValue("Bearer", token.Token);
>         return await base.SendAsync(request, cancellationToken);
>     }
> }
> \\\

> [!DANGER] Pitfall 10: Cost Explosion During Coexistence (Azure-Specific)
>
> **The Problem:** Running both the legacy system and the new system in parallel doubles infrastructure costs. A 6-month coexistence period estimate extends to 14 months due to operational complexity.
>
> **Production Signal:** Azure Cost Management shows a month-over-month increase of ,000 in the "Migration" cost center. AKS cluster: ,000/month. Legacy VMs: ,000/month. Azure SQL DTU + Cosmos DB RU/s: ,000/month. Service Bus Premium: ,000/month.

> **Root Cause:** Migration plan waits until ALL capabilities are migrated before decommissioning anything. Both systems run at full capacity for 14 months.

> **Prevention:**
> - Include a "Decommission Legacy Capability" work item for each capability
> - Scale down legacy system VMs as capabilities are migrated
> - Use Azure reservations/RI for the known coexistence period
> - Automate decommission: after a capability reaches 100% and passes the 14-day observation window, an Azure Automation runbook removes the legacy route and reduces capacity

> [!WARNING] When NOT to Apply
>
> 1. **Tiny legacy system:** < 5,000 lines of code. A Strangler Fig adds more complexity (proxy, feature flags, dual-writes, reconciliation) than it saves. Do a full rewrite in 4-6 weeks.
>
> 2. **End-of-life in < 6 months:** If the legacy system is already scheduled for decommissioning in Q3, do not start a Strangler in Q1. The proxy investment won't pay back.
>
> 3. **Hard real-time requirements:** If P99 latency must be < 50ms (e.g., high-frequency trading, real-time ad bidding), the proxy's 5-15ms overhead and fallback latency variance may violate the SLA.
>
> 4. **No bounded contexts:** If the legacy system is a "ball of mud" where every feature touches every table, you cannot isolate capabilities. Decompose first using [[7.020 — Anti-Corruption Layer Pattern]] and [[2.013 — Repository Pattern]] before attempting a Strangler Fig.
>
> 5. **Single development team:** A Strangler Fig requires two parallel workstreams — maintaining legacy and building the new system. With one team, you can't do both. Grow the team or choose a faster approach.
>
> 6. **Database is the bottleneck:** If the legacy database schema is the primary constraint, an API-level strangler won't help because both systems compete for the same database. Use database-first migration or data federation.
>
> 7. **Missing executive sponsorship:** A Strangler Fig takes 6-18 months. Without active executive support, the project gets defunded at month 4 when costs are visible but benefits haven't materialized. Secure full-project budget and quarterly reviews before starting.

### Spoken Answers

#### Q1 — Average Answer

> "The Strangler Fig pattern is where you build a new version of a legacy system piece by piece, and a proxy routes traffic to either the old or new system depending on what's been migrated. You use it when you don't want to do a big-bang rewrite because it's too risky. You add a proxy in front of both systems, and as you migrate features, you change the proxy to send more traffic to the new system until the old one isn't used anymore and can be shut down."

#### Q1 — Great Answer

> "The Strangler Fig pattern is an incremental system migration strategy named after the strangler fig vine that slowly grows around a host tree until the host is replaced. The key insight is that you never modify the legacy system — you build the new system independently and use a routing proxy to intercept requests and redirect specific capabilities. The proxy sits in front of both systems and evaluates each request against a route table combined with feature flags to decide the target. The migration proceeds one bounded capability at a time through stages: legacy-only -> shadow-mode -> partial-migration -> full-migration -> decommissioned.
>
> You'd apply this when the legacy system is business-critical, cannot be taken offline for a rewrite, and has clean enough boundaries that you can extract individual capabilities. I've used it to migrate a .NET Framework 4.8 e-commerce platform serving  annual GMV to .NET 8 microservices — we used YARP as the proxy, Azure App Configuration for dynamic feature flags, and Azure Service Bus for dual-write synchronization. The alternative — a big-bang rewrite — was rejected because a single week of downtime would cost  in lost revenue."

#### Q5 — Average Answer

> "For rollback, you just change the feature flag back to send traffic to the legacy system instead of the new one. It should be quick if you've designed it right."

#### Q5 — Great Answer

> "Rollback in the Strangler Fig pattern must be designed as a first-class operation, not an afterthought. Every capability in partial- or full-migration state must have a documented, tested, and automated rollback runbook with a target of under 30 minutes for full recovery.
>
> The rollback has four phases:
> 1. **Cut Traffic:** Change the feature flag from 'fully-migrated' or any partial percentage to 'legacy-only' — this is instantaneous in a centralized system, but must account for the 10-60 second cache propagation window across all proxy instances.
> 2. **Drain In-Flight:** Wait for any requests already routed to the new system to complete (monitor active request count; typically 30 seconds).
> 3. **Data Reconciliation:** Run a reconciliation job to identify records that were written by the new system during the migration window but not synchronized to the legacy system — replay those events.
> 4. **Scale Down:** Only after steps 1-3 are complete, scale down the new system resources.
>
> The most common rollback failure I've seen is reversing steps 1 and 2 — people de-provision the new system first, which causes 503 errors while the proxy's cached routing still points to a now-dead system. We automated this with an Azure DevOps pipeline that reads the proxy instances' /ops/migration-status endpoint to confirm 100% flag propagation before proceeding."

#### Q8 — Average Answer

> "I'd set up a YARP proxy in front of the legacy system, then pick one feature like order creation, build it in .NET 8, and start routing some traffic to it. If it works, route more. Eventually route all traffic and shut down the legacy part. Keep doing that for each feature."

#### Q8 — Great Answer

> "For a 50,000 orders/day retail platform on .NET Framework 4.8, my migration strategy has seven phases:
>
> **Phase 1 — Assessment (2 weeks):** Map bounded contexts. For a retail platform, the domains are: Order Management, Inventory, Catalog, Pricing, Shipping, Payments, User Management. Measure each domain's complexity and traffic share. Order Management is ~30% of traffic (15K req/day).
>
> **Phase 2 — Infrastructure Foundation (3 weeks):** Deploy YARP reverse proxy on AKS with 3 replicas (autoscaling up to 10 at 1,000 RPS per pod). Configure Azure App Configuration with feature flags per domain. Deploy Azure Service Bus Premium for dual-write events. Set up OpenTelemetry with 100% sampling for migration routes.
>
> **Phase 3 — ACL Implementation (2 weeks):** Build the Anti-Corruption Layer as a separate ASP.NET Core service. Map legacy contracts to new domain models using FluentValidation for both schemas.
>
> **Phase 4 — Pioneer Migration: Inventory Lookup (4 weeks):** Start with Inventory because it's read-only (no dual-write complexity), low traffic (~5K req/day), immediate value. Run shadow mode for 7 days comparing responses byte-for-byte. Shift 1% -> 10% -> 50% -> 100% over 2 weeks.
>
> **Phase 5 — Order Management Migration (8 weeks):** Involves dual-writes for every order created. Event-driven sync uses Azure Service Bus with sessions (per-order ordering) and idempotency keys. Reconciliation job runs every hour comparing legacy and new databases. Stay at 25% for 5 business days, then proceed to 50% -> 100%.
>
> **Phase 6 — Remaining Domains (6 months):** Each domain takes 4-8 weeks. Catalog (read-heavy, 20K req/day), Shipping (write-heavy), User Management (auth-critical), Pricing (computation-heavy, Cosmos DB). Parallelize two domains per sprint.
>
> **Phase 7 — Decommission (2 weeks):** After all domains pass the 14-day observation window: remove proxy entries for legacy routes, archive legacy database, deallocate legacy VMs — saving ~/month in compute costs."

### Follow-Up Chain

**Follow-Up 1: What happens when the feature flag is wrong — you route a request to the new system but it hasn't fully implemented that capability yet?**

> **Model Answer:** "This is why we use consistent hashing on a stable identifier. If the proxy routes customer A to the new system at 25% migration, customer A should ALWAYS route to the new system for that capability — even if the percentage changes slightly — because the hash of their customer ID is deterministic. This prevents the 'wizard split' problem where a user's multi-step operation jumps between systems.
>
> That said, we also implement a 'routing guard' in the ACL. If the new system receives a request for a capability it doesn't support, it returns a 421 Misdirected Request with a header X-Correct-System: legacy. The proxy intercepts this and re-routes to the legacy system transparently, logging the misconfiguration as a high-severity alert."

**Follow-Up 2: How do you test the migration without affecting production users?**

> **Model Answer:** "We use three testing layers:
>
> **Layer 1 — Shadow Mode:** Before routing any real traffic, the new system runs in shadow mode. The proxy sends a copy of every request to both systems, but only returns the legacy response to the client. A background job compares the responses byte-for-byte and reports differences.
>
> **Layer 2 — Synthetic Traffic:** We use Azure Load Testing to replay 10,000 recorded production requests against both systems. The test validates that response structures match and latency is within 1.5x.
>
> **Layer 3 — Testcontainers Integration Tests:** Each capability migration includes a full integration test suite using Testcontainers (Azure SQL, Cosmos DB emulator, Service Bus emulator) and Respawn to reset state. NetArchTest enforces that the proxy project doesn't reference legacy or new system projects directly."

**Follow-Up 3: At what point do you decommission the legacy system entirely?**

> **Model Answer:** "We decommission AFTER every capability has passed a 14-day observation window at 100% traffic with:
> - Error rate delta < 2% compared to legacy baseline
> - Dual-write inconsistency rate < 0.01% for write capabilities
> - No rollback necessary during the observation window
> - All downstream consumers updated to read from the new system
>
> The decommission runbook:
> 1. Disable legacy write endpoints (read-only mode for 7 days)
> 2. Remove proxy entries mapping to legacy routes
> 3. After 7 days with no issues, archive legacy database (final snapshot)
> 4. After 30 days, deallocate legacy compute resources
> 5. After 90 days (or as required by compliance), delete legacy data"

## Section 8 — Architecture Decision Record

### ADR-021: Strangler Fig Migration Strategy for Order Management System

**Status:** Approved

**Context:**
The current Order Management System (OMS) is a .NET Framework 4.8 monolith deployed on 12 on-premise VMs running Windows Server 2016. It processes 50,000 orders/day with a P95 response time of 250ms. Key challenges:

- The monolith cannot be containerized without a full rewrite (.NET Framework 4.8 incompatible with Linux containers)
- Feature velocity dropped from 8 features/sprint to 2 features/sprint over 18 months
- Legacy SQL Server 2016 database has 15 stored procedures > 500 lines each
- Azure SQL Elastic Pool migration is blocked by .NET Framework library incompatibilities
- Infrastructure requires all workloads on AKS by Q3 2027
- Total annual technical debt carrying cost: .2M

**Options Considered:**

| Option | Description | Pros | Cons |
|--------|-------------|------|------|
| **A — Strangler Fig via YARP** | Incremental proxy-based migration over 14 months | Lowest risk, incremental value, per-capability rollback | 5-15ms proxy latency, dual infrastructure (~/month), complex dual-write logic |
| **B — Big-Bang Rewrite** | 6-month full rewrite in .NET 8, then cut over | Single event, no proxy, no dual-writes | /week downtime risk; 70% of big-bang rewrites fail or exceed budget by 200% |
| **C — Lift-and-Shift to .NET 8** | Port entire codebase with minimal changes | No proxy, no architectural change | Does not solve monolith problems; legacy patterns carried forward |
| **D — Do Nothing** | Continue on .NET Framework 4.8 | No migration cost | End of support; velocity drops 15%/yr; AKS deadline unmet |

**Decision:**
Adopt **Option A — Strangler Fig via YARP Proxy**.

**Rationale:**
- 14-month timeline aligns with AKS Q3 2027 deadline with 6 months buffer
- Per-capability migration demonstrates value within 6 weeks (inventory lookup)
- Rollback limits blast radius to one capability at a time
- Proxy infrastructure (YARP on AKS, 3 replicas) costs /month but eliminates /month legacy VM cost after decommissioning
- Dual-write complexity justified by zero data loss requirement
- 8 independently migratable capabilities with clear interface boundaries

**Consequences:**

| Consequence | Type | Mitigation |
|-------------|------|------------|
| 5-15ms added P95 latency through proxy hop | Negative | Target < 15ms; benchmark YARP at 1.2us per evaluation; CDN caching for read-heavy capabilities |
| /month additional infrastructure during coexistence | Negative | Per-capability decommission reduces cost incrementally; budget approved with quarterly review |
| Team must learn YARP, Feature Management, dual-write patterns | Negative | 2-week spike with PoC; pair programming for first 2 capabilities |
| Legacy team must maintain feature parity | Negative | Dedicated legacy maintenance rotation; automated regression tests per capability |
| ACL becomes a critical dependency | Negative | ACL deployed as separate service with 3 replicas + circuit breaker; SLA = 99.95% |
| Downstream systems must be updated per capability | Negative | Data consumer mapping completed before each migration; dual-reads during transition |
| First-migrated capability may need refactoring | Negative | Expectation set with stakeholders: first 2 are "pioneers" |
| Successful migration reduces annual technical debt by .2M | Positive | Tracked quarterly; savings reinvested in platform improvements |
| Team builds modern .NET 8 microservices expertise | Positive | Training byproduct; improved retention |

**Review Trigger:**
This ADR should be reviewed when ANY of the following occur:

1. **Capability migration velocity drops below 1 per 2 sprints** for 2 consecutive months
2. **Dual-write inconsistency rate exceeds 0.1%** for any capability for more than 7 days
3. **Proxy P95 latency overhead exceeds 50ms** for any capability for more than 2 consecutive hours
4. **Infrastructure cost during coexistence exceeds /month** (budgeted buffer limit)
5. **A critical security vulnerability is discovered in .NET Framework 4.8** that cannot be patched in the legacy system
6. **Business requirements change** such that the legacy system must be maintained beyond the 18-month timeline
7. **A rollback that takes > 60 minutes** due to runbook gaps or automation failures

## Section 9 — Self-Check

### Conceptual Questions (12)

<details>
<summary>Q1: What is the primary architectural requirement for applying the Strangler Fig pattern?</summary>

**Answer:** The legacy system must be decomposable into bounded contexts with clear interface boundaries. If the system is a "ball of mud" with no isolatable capabilities, you cannot extract individual capabilities for migration. You must first introduce an [[7.020 — Anti-Corruption Layer Pattern]] or facade to create the necessary boundaries before applying the Strangler Fig.
</details>

<details>
<summary>Q2: Why is the proxy layer critical to the Strangler Fig pattern?</summary>

**Answer:** The proxy is the sole traffic arbitrator — it intercepts every request, evaluates routing rules against dynamic feature flags, and forwards to either the legacy or new system. It enables zero-downtime migration, per-request routing decisions, transparent rollback (via flag change), and independent deployability of both systems. Without an independent proxy, you cannot perform per-capability, hot-switch routing.
</details>

<details>
<summary>Q3: What is the difference between "shadow mode" and "partial migration"?</summary>

**Answer:** In **shadow mode**, the proxy sends traffic to both systems but only returns the legacy system's response to the client. The new system's responses are compared off-line to validate correctness. Traffic percentage = 0%. In **partial migration**, the proxy routes a real percentage of traffic to the new system (e.g., 25%) and returns the new system's response to the client. The percentage is controlled by feature flag and uses consistent hashing for session stability.
</details>

<details>
<summary>Q4: How do you ensure a user doesn't switch between systems mid-session during partial migration?</summary>

**Answer:** Use **consistent hashing** on a stable request attribute (e.g., customer ID, tenant ID, session ID). The hash determines which system the request routes to. Because the hash is deterministic for a given customer ID, the same user always routes to the same system regardless of which proxy instance handles the request. Consistent hashing ensures the split is stable even as the percentage changes slightly (within a single percentage bracket).
</details>

<details>
<summary>Q5: What is the rollback procedure for a capability that has reached "fully migrated" state?</summary>

**Answer:**
1. Change feature flag from ully-migrated to legacy-only
2. Wait for cache propagation across all proxy instances (2x cache TTL, typically 60 seconds)
3. Verify via /ops/migration-status that all proxies show legacy-only
4. Run reconciliation job to identify records written by new system but not replicated to legacy — replay events
5. After verification, scale down new system resources (not before!)
6. Document the rollback reason and any data inconsistencies found
7. Total target: < 30 minutes from decision to full recovery
</details>

<details>
<summary>Q6: How does the Strangler Fig pattern handle authentication and authorization during migration?</summary>

**Answer:** Three approaches:
1. **Authenticate at proxy level** — the proxy validates the token and passes it to either system; both systems trust the proxy's validation
2. **Token translation in ACL** — the proxy strips the incoming token; the ACL validates against the legacy IdP then generates a new token for the new system
3. **Edge authentication** — Azure Front Door or APIM authenticates at the edge; the proxy only handles routing with a pre-authenticated principal

Approach 3 is preferred because it separates authentication from migration concerns.
</details>

<details>
<summary>Q7: What happens to data created by the new system after a rollback?</summary>

**Answer:** During rollback, a reconciliation job identifies records that exist in the new system but not in the legacy system, or records that differ. These records are replayed from the new system to the legacy system via integration events or batch sync. The legacy system becomes the source of truth again after the rollback, so all new-system-only records must be replicated to legacy before the new system is scaled down. A "reconciliation window" of 15-30 minutes after flag change accounts for this.
</details>

<details>
<summary>Q8: Can the Strangler Fig pattern be used for database-level migration?</summary>

**Answer:** Yes — the **database-first strangler** variant. Instead of a proxy intercepting HTTP requests, a data access layer intercepts database queries and routes them to either the legacy or new database based on entity type. However, this is harder because: (a) queries often span entity types (JOINs across migrated and non-migrated tables), (b) transactions may need XA/2PC coordination, and (c) the data access layer must be inserted into the legacy codebase (violating "don't modify legacy"). The API-level strangler is preferred.
</details>

<details>
<summary>Q9: How do you handle caching during the coexistence period?</summary>

**Answer:**
- **Read-only data** (catalog): Cache in both systems independently; short TTL (60 seconds)
- **User session data**: Shared Redis cache accessible by both systems; proxy consistent hashing ensures session stickiness
- **Aggregated data** (dashboards): Data virtualization layer queries appropriate source based on migration state
- **Distributed cache** (Azure Redis): Separate instances per system during migration, merge after full migration

During partial migration, invalidate both caches when data changes.
</details>

<details>
<summary>Q10: What is the relationship between Strangler Fig and Strangler Fig Application in DDD?</summary>

**Answer:** The Strangler Fig **Application** pattern is the same concept within a single codebase — it uses branch-by-abstraction to gradually replace a module in-process. The classical Strangler Fig (this note) operates at the deployment topology level using a proxy. The Application variant is appropriate when: (a) you cannot insert a network proxy, (b) legacy is a single process with no network boundary, or (c) migration is at the library/component level. The Application variant modifies the legacy codebase, which the proxy-based strangler explicitly avoids.
</details>

<details>
<summary>Q11: What metrics indicate that a migration is failing and should be paused?</summary>

**Answer:** Red flags requiring immediate pause:
1. **Error rate delta > 5%** between new and legacy for any 5-minute window
2. **Dual-write inconsistency rate > 0.1%** for 2 consecutive reconciliation runs
3. **Proxy P95 latency overhead > 50ms** for 2 hours
4. **Rollback time > 60 minutes** in a drill (runbook deficiency)
5. **Migration velocity < 1 capability per 2 sprints** for 2 consecutive months
6. **3AM production signal** from any failure mode (see Section 3)
7. **Team attrition** — if key engineers leave mid-migration, pause to document and stabilize
</details>

<details>
<summary>Q12: How does the Strangler Fig pattern support zero-downtime deployments?</summary>

**Answer:** Zero-downtime is inherent to the pattern because:
- The legacy system never goes down; it continues serving non-migrated capabilities
- The new system is deployed independently; updates don't affect legacy or proxy
- Feature flag changes are configuration-only — no deployment needed to shift traffic
- Each capability migration impacts only that capability's traffic
- The proxy itself uses rolling updates on AKS with health checks

However, zero-downtime requires: >= 2 proxy replicas, circuit breakers for backend failures, proper health check configuration, and Kubernetes maxSurge: 1 / maxUnavailable: 0.
</details>

### Scenario Challenges (6)

<details>
<summary>Scenario 1: Migrating a legacy API serving 1,000 RPS. At 25% partial migration, new system error rate is 5% while legacy is 0.5%. What do you do?</summary>

**Answer:**

Step 1 — **Stop migration immediately.** Change flag to legacy-only.

Step 2 — **Analyze the 5% errors:** Are they business logic (validation) or infrastructure (timeouts)? Correlated with specific input patterns (e.g., orders over ,000)?

Step 3 — **Hypothesis testing:**
- Business logic: Legacy likely accepts edge cases (quantity = 0) that new rejects. Add compatibility mode.
- Infrastructure: At 250 RPS (25% of 1,000), new system may need more replicas (scale 2->5).
- Correlation: Add transformation exception for those inputs; route specific requests to legacy.

Step 4 — **Fix and re-test:** Run synthetic traffic replay (10,000 recorded requests) and verify error rate < 2%.

Step 5 — **Re-migrate at 1%** (not 25%). Start at 1% for 2 days, escalate to 5%, 10%, 25% over 2 weeks.

**Key lesson:** Always start partial migration at 1% — never 25%.
</details>

<details>
<summary>Scenario 2: Stakeholder demands all 8 capabilities migrated in 4 months, not 14. How do you respond?</summary>

**Answer:**

**First, understand their motivation** — cost, acquisition deadline, strategic initiative? Address the underlying need.

Step 1 — **Explain the risk curve:** "Strangler Fig at 14 months has ~12% incident probability. Compressing to 4 months requires 2 migrations in parallel per month, increasing risk of dual-write conflicts, team context-switching, and lost observation windows. Post-migration incident probability rises to ~38%."

Step 2 — **Offer compromise:** "What if we deliver a high-value capability (Inventory Lookup: 4 weeks, read-only) in the first 6 weeks, then maintain 2-capabilities-per-sprint from month 3? If behind at month 6, we add a third workstream."

Step 3 — **Quantify speed cost:** "Compressing to 4 months adds ~ for overtime/temp staff and /month peak infrastructure cost."

Step 4 — **If they insist:** Document the decision in an ADR with acknowledged risks, add daily reconciliation, and include a "pause and assess" checkpoint at month 2.
</details>

<details>
<summary>Scenario 3: During 50% partial migration, you discover a legacy system bug that loses 0.1% of orders. New system doesn't have this bug. Accelerate or pause?</summary>

**Answer:**

**Pause migration — but pivot to accelerate the affected capability while pausing cut-over of new ones.**

Step 1 — **Assess impact:** 0.1% of 50,000 orders/day = 50 orders/day lost. At /order = ,000/day (.46M/year).

Step 2 — **Modified approach:**
- Accelerate order-creation by shortening observation windows at 50% -> 75% -> 100% from 2 weeks to 3 days if thresholds stay within limits
- Do NOT skip brackets — still pass through each, but with shorter validation
- Add hourly reconciliation (not daily) during acceleration
- Communicate to stakeholders: "Legacy bug loses ,000/day. We can fix it 4 weeks early with increased operational risk."

Step 3 — **Post-migration:** After order-creation reaches 100%, verify the bug is fixed. Reset velocity to normal for remaining capabilities.
</details>

<details>
<summary>Scenario 4: Your team has 5 engineers. Legacy has 150K LOC and 8 capabilities. How do you staff it?</summary>

**Answer:**

Serialized migration with single focus per sprint.

**Staffing:**
- Engineer A: Proxy & Infrastructure (YARP, Azure, CI/CD, monitoring)
- Engineer B: New System Capability Lead (domain model, business logic)
- Engineer C: New System Data (database, dual-write events, reconciliation)
- Engineer D: Anti-Corruption Layer (contract mapping, transformation, testing)
- Engineer E: Legacy Maintenance + QA (keep legacy running, test harnesses, validate)

**Schedule (serialized, 2 capabilities per pair):**
- Sprints 1-2: Foundation (proxy, ACL, CI/CD, environments)
- Sprints 3-5: Inventory Lookup (read-only, shadow -> partial)
- Sprint 6: Decommission Inventory; prepare Order Creation
- Sprints 7-10: Order Creation (all engineers focused; dual-writes)
- Sprint 11: Decommission Order; prepare Catalog
- Sprints 12-14: Catalog (read-heavy)
- Sprints 15-20: Remaining 4 capabilities (2 sprints each)

**Key constraint:** With 5 engineers, no parallel capabilities. Each takes 3-5 sprints (6-10 weeks). Total: ~24 sprints = 12 months.

**Risk mitigation:** Cross-train, limit WIP (1 capability at a time), rotate legacy maintenance (1 day/week/engineer).
</details>

<details>
<summary>Scenario 5 — Azure Production: Healthcare patient portal migration uses Strangler Fig with YARP on AKS, Azure API Management, Azure App Configuration, Azure Service Bus. At 30% migration for Appointment Booking, reconciliation finds 1,247 out-of-sync records. Legacy shows confirmed; new shows cancelled. What happened?</summary>

**Answer:**

**Immediate response (within 5 minutes):**
1. **Pause migration:** Change strangler/appointment-booking/status from partial-migrate (30%) to legacy-only.
2. **Acknowledge the page.**

**Investigation (15 minutes):**

Step 1 — **Identify direction:** Legacy = "confirmed" (source of truth pre-full-migration). New = "cancelled". The new system received events and applied them incorrectly.

Step 2 — **Trace the event flow:**
- New system publishes AppointmentCreatedIntegrationEvent when appointment created
- Legacy consumes and creates the appointment
- Legacy sends email confirmation -> email service calls back with communication.sent
- This event is replicated to new system as generic AppointmentUpdated
- New system's duplicate detection uses PatientEmail + AppointmentDate as idempotency key
- If same patient books two appointments on different dates, first appointment's confirmation email triggers update to the SECOND appointment, setting its status to "cancelled" (misconfigured event type mapping)

Step 3 — **Root cause:** Faulty event mapping in dual-write consumer. CommunicationSent event from legacy email service is interpreted as a cancellation signal.

**Remediation (30 minutes):**

1. **Fix bug:** Update event type translation in ACL's EventTypeMapper to ignore CommunicationSent events for appointments.
2. **Data repair:** For the 1,247 records where new = "cancelled" but legacy = "confirmed": replay the original creation event from legacy to new, overwriting incorrect status.
3. **Prevent recurrence:**
   - Add EventTypeAllowlist to dual-write consumer
   - Add integration test: "Given appointment created by new system, when legacy publishes CommunicationSent, appointment status MUST NOT change"
   - Add validation rule: "Appointment status mismatches where new = cancelled and legacy = confirmed in absence of cancellation event flagged HIGH priority"
4. **Post-mortem:** ADR entry: "Add event type validation to all dual-write consumers before additional capabilities."
5. **Resume:** Run at 1% for 24 hours, verify zero mismatches, escalate to 30% over 5 days.
</details>

<details>
<summary>Scenario 6: After 10 months of migration, proxy P95 latency increased from 12ms to 87ms. Route table grew from 5 to 48 routes. What happened?</summary>

**Answer:**

**Diagnosis:** 12ms -> 87ms P95 (7.25x degradation) is not a hardware/scaling issue. It indicates an algorithmic O(n) or worse problem in routing evaluation.

**Investigation:**
1. Check if feature flag cache hit rate dropped (cache misses per second)
2. Check if route table lookup uses linear search (O(n)) instead of dictionary/trie lookup
3. Check if transformation middleware iterates all 48 transformers for every request
4. Check if circuit breaker state queries are causing lock contention

**Likely root cause:** The StranglerRoutingEngine.ExtractRouteKey uses a linear list of pattern matches. With 5 routes, the linear scan takes ~0.5us. With 48 routes, the same scan takes ~5us. But the 87ms suggests something worse — possibly the feature flag cache is not scaling. If each of 48 routes issues a separate cache GET and the cache has no batching, each request triggers 48 Redis/App Configuration calls.

**Remediation:**

1. **Profile the routing engine:**
\\\csharp
// Add tracking to identify which component is slow
public class InstrumentedRoutingEngine
{
    private readonly Histogram<double> _routeKeyExtractionDuration;
    private readonly Histogram<double> _featureFlagFetchDuration;

    public async ValueTask<RoutingDecision> EvaluateAsync(HttpContext context, CancellationToken ct)
    {
        using var sw = new DiagnosticListener("StranglerProxy");

        var routeKeyDuration = Measure(() => ExtractRouteKey(context));
        _routeKeyExtractionDuration.Record(routeKeyDuration.TotalMilliseconds);

        var featureFlagDuration = Measure(() => GetMigrationStateAsync(routeKey, ct));
        _featureFlagFetchDuration.Record(featureFlagDuration.TotalMilliseconds);

        // ...
    }
}
\\\

2. **Batch feature flag queries:**
\\\csharp
// Instead of one call per route, batch all calls
public async Task<IReadOnlyDictionary<string, MigrationState>> GetAllStatesAsync()
{
    // Single Azure App Configuration snapshot call
    var snapshot = await _appConfigClient.GetSnapshotAsync("migration-snapshot");
    return snapshot.ToDictionary(kv => kv.Key, kv => ParseState(kv.Value));
}
\\\

3. **Use a trie or compiled regex for route matching instead of linear string scanning:**
\\\csharp
// Compile route patterns into a decision tree at startup
public sealed class RouteTrie
{
    private readonly FrozenDictionary<string, RouteMatcher> _matchers;

    public string Match(string path, string method) =>
        _matchers.TryGetValue(method, out var matcher)
            ? matcher.Match(path) ?? "unknown"
            : "unknown";
}
\\\

4. **Set up a performance regression test:**
\\\csharp
[Test]
public void RouteEvaluation_With48Routes_ShouldStayUnder5us()
{
    var engine = BuildEngineWithNRoutes(48);
    var context = CreateHttpContext("/api/orders", "POST");
    var sw = Stopwatch.StartNew();
    for (int i = 0; i < 1000; i++) engine.Evaluate(context, CancellationToken.None);
    sw.Stop();
    Assert.That(sw.Elapsed.TotalMicroseconds / 1000, Is.LessThan(5));
}
\\\

5. **Set up an alert:** When proxy P95 latency exceeds 15ms for any 5-minute window, page the on-call engineer.
</details>
