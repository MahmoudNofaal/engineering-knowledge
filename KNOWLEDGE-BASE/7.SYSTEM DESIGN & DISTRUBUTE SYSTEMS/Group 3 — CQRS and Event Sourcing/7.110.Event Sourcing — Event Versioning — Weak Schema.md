# 7.110 — Event Sourcing — Event Versioning — Weak Schema

## Overview

**Weak schema evolution** is a family of strategies that accept **loose coupling between stored event data and the consuming code's type system**. Instead of transforming old events to a canonical latest schema ([[7.109 — Event Sourcing — Event Versioning — Upcasting]]), weak-schema systems tolerate missing, extra, or differently-shaped fields at deserialization time. The core idea: **make consumers resilient to schema drift rather than forcing every event into a single rigid type**.

| Concept | Description |
|---|---|
| **Tolerant Reader** | Consumer ignores unknown fields and fills missing optional fields with default values |
| **Schema-on-Read** | The interpretation of event data is deferred to read time; the store has no schema enforcement |
| **Schema-on-Write** | Events are validated against a schema at write time; all consumers see a consistent shape |
| **Weak Schema** | No strict type enforcement; JSON objects with optional, nullable, and discardable fields |
| **Strong Schema** | Contract-first (Protobuf, Avro, Thrift); schema evolution is governed by explicit rules |
| **Schema Registry** | Centralized repository of schemas that maps schema IDs to schema definitions |
| **Forward Compatibility** | New consumer can read events written by an older producer |
| **Backward Compatibility** | Old consumer can read events written by a newer producer |

```
┌─────────────────────────────────────────────────────────────────────┐
│                  Schema Evolution Spectrum                          │
│                                                                     │
│  Weak Schema                          Strong Schema                 │
│  ──────────────────────────────────────────────────────────────►   │
│                                                                     │
│  ┌──────────┐   ┌───────────┐   ┌──────────┐   ┌───────────┐       │
│  │ Raw JSON │   │ Tolerant  │   │ Registry │   │ Contract  │       │
│  │ (no      │   │ Reader    │   │ + JSON   │   │ First     │       │
│  │ schema)  │   │ (skip     │   │ Schema   │   │ (Protobuf,│       │
│  │          │   │ unknown)  │   │ (draft)  │   │ Avro)     │       │
│  └──────────┘   └───────────┘   └──────────┘   └───────────┘       │
│                                                                     │
│  Max flexibility                        Max safety                 │
│  Minimal tooling                        Max tooling investment      │
│  Runtime errors possible                Compile-time type checks    │
└─────────────────────────────────────────────────────────────────────┘
```

## 1. Core Concepts — Weak vs Strong Schema

### 1.1 The Two Philosophies

Every event-sourced system must answer: **how strictly do we enforce the shape of stored events?**

| Dimension | Weak Schema | Strong Schema |
|---|---|---|
| **Storage format** | JSON with optional fields | Protobuf, Avro, Thrift binary |
| **Write-time validation** | Minimal (shape is advisory) | Required (must match schema ID) |
| **Schema registry** | Optional (documentation-driven) | Required (contract enforcement) |
| **Evolution cost** | Low — add field, consumer ignores if unneeded | Medium — must register new schema version, verify compat |
| **Deserialization** | `JsonDocument` / `JsonNode` with fallback | Generated code from `.proto` / `.avsc` |
| **Upcasting needed?** | Rarely — consumer tolerates both shapes | Often — old binary cannot be read by new codecs |
| **Type safety** | Loose — `string?` / `JsonElement` | Strong — compiler rejects missing required fields |
| **Operational visibility** | Low — silent data degradation possible | High — schema registry shows exact picture |
| **Polyglot friendliness** | Excellent — every language has a JSON parser | Good — Protobuf/Avro have broad but not universal support |

### 1.2 When to Choose Weak Schema

| Scenario | Recommendation |
|---|---|
| Rapid prototyping / startup phase | Weak schema — evolve fast without schema governance |
| Event payloads are predominantly optional | Weak schema — no required fields to break consumers |
| Small team, single bounded context | Weak schema — team can coordinate changes informally |
| High consumer diversity (mixed languages) | Weak schema — JSON is the universal interchange |
| Regulatory/compliance requires audit trails | Weak schema + envelope pattern — payload is self-describing |
| High-throughput, latency-sensitive | Strong schema — binary serialization is smaller and faster |
| Multi-team, multi-service event contracts | Strong schema — registry provides explicit compatibility guarantees |
| Financial transactions requiring precise schema | Strong schema — no silent field drops allowed |

### 1.3 Relationship to Upcasting ([[7.109]])

Weak schema and [[7.109 — Event Sourcing — Event Versioning — Upcasting]] are complementary, not mutually exclusive:

```
                         ┌────────────────────────────┐
                         │    Event Store Stream       │
                         │  v1 events, v2 events, ...  │
                         └────────────┬───────────────┘
                                      │
                    ┌─────────────────┼─────────────────┐
                    │                 │                 │
                    ▼                 ▼                 ▼
          ┌─────────────────┐ ┌─────────────────┐ ┌───────────────┐
          │  Strong Schema  │ │  Upcaster +     │ │  Weak Schema  │
          │  Only (needs    │ │  Weak Schema    │ │  Only (direct  │
          │  upcast to v3)  │ │  (hybrid)       │ │  consumer)     │
          └─────────────────┘ └─────────────────┘ └───────────────┘
```

- **Pure weak schema** — the consumer reads `JsonDocument` directly and extracts fields it cares about. Missing fields yield `null` or defaults. No transformation needed.
- **Hybrid** — upcasters normalize the most disruptive changes (e.g., type changes), but the consumer still tolerates optional missing fields.
- **Pure strong schema** — every event must be upcast to the latest type before consumption.

Most mature systems adopt the **hybrid** approach: upcasters handle breaking structural changes, while tolerant readers handle additive/optional changes.

## 2. Tolerant Reader Pattern

The **Tolerant Reader** pattern (derived from Martin Fowler's "Tolerant Reader" and the Robustness Principle — "be conservative in what you send, be liberal in what you accept") dictates that an event consumer should:

1. **Ignore unknown fields** — extra properties in the JSON do not cause deserialization failures.
2. **Provide sensible defaults for missing fields** — optional fields that are absent receive a safe default.
3. **Gracefully handle type coercion** — a string `"42"` is accepted where an integer is expected.
4. **Never crash on unexpected shapes** — malformed fields are logged and skipped, not thrown.

### 2.1 Tolerant Deserialization with `System.Text.Json`

```csharp
using System.Text.Json;
using System.Text.Json.Nodes;

namespace Career.Events.WeakSchema;

// Tolerant event type — all properties are optional/nullable

{

    public string? CustomerId { get; init; }

    public string? Currency { get; init; }

    public string? ProductId { get; init; }   // legacy field
    public List<LineItem>? Items { get; init; }
    public DateTime? OccurredAt { get; init; }

public sealed record LineItem(
    string? Sku,

    decimal? UnitPrice
```

### 2.2 Custom Tolerant Converter

```csharp
using System.Text.Json;
using System.Text.Json.Serialization;

namespace Career.Events.WeakSchema.Serialization;

public sealed class TolerantEventConverter<T> : JsonConverter<T> where T : class
{
    private static readonly JsonSerializerOptions FallbackOptions = new()
    {
        PropertyNameCaseInsensitive = true,
        PropertyNamingPolicy = JsonNamingPolicy.CamelCase,
        DefaultIgnoreCondition = JsonIgnoreCondition.WhenWritingNull,
    };

    public override T? Read(ref Utf8JsonReader reader, Type typeToConvert, JsonSerializerOptions options)
    {
        using JsonDocument doc = JsonDocument.ParseValue(ref reader);
        JsonElement root = doc.RootElement;

        // Phase 1: attempt standard deserialization
        string raw = root.GetRawText();
        T? result = JsonSerializer.Deserialize<T>(raw, FallbackOptions);

        // Phase 2: gather unknown extensions
        if (result is not null && HasExtensionProperty(result))
        {
            var knownProperties = new HashSet<string>(
                typeof(T).GetProperties()
                    .Select(p => JsonNamingPolicy.CamelCase.ConvertName(p.Name)),
                StringComparer.OrdinalIgnoreCase);

            var extensions = new Dictionary<string, JsonElement>(StringComparer.OrdinalIgnoreCase);
            foreach (JsonProperty property in root.EnumerateObject())
            {
                if (!knownProperties.Contains(property.Name) && property.Name != "$type" && property.Name != "$version")
                {
                    extensions[property.Name] = property.Value.Clone();
                }
            }

            if (extensions.Count > 0)
            {
                SetExtensions(result, extensions);
            }
        }

        return result;
    }

    public override void Write(Utf8JsonWriter writer, T value, JsonSerializerOptions options)
    {
        JsonSerializer.Serialize(writer, value, typeof(T), FallbackOptions);
    }

    private static bool HasExtensionProperty(T value)
    {
        return typeof(T).GetProperty("Extensions",
            BindingFlags.Instance | BindingFlags.Public) is not null;
    }

    private static void SetExtensions(T value, Dictionary<string, JsonElement> extensions)
    {
        var prop = typeof(T).GetProperty("Extensions",
            BindingFlags.Instance | BindingFlags.Public);
        prop?.SetValue(value, extensions);
    }
}
```

### 2.4 Tolerant Reader — Manual Fallback with JsonDocument

When automatic converters are insufficient (complex polymorphic scenarios), use manual `JsonDocument` inspection:

```csharp
using System.Text.Json;

namespace Career.Events.WeakSchema.Readers;

public sealed class TolerantOrderPlacedReader
{
    private static readonly JsonSerializerOptions Options = new()
    {
        PropertyNameCaseInsensitive = true,
        PropertyNamingPolicy = JsonNamingPolicy.CamelCase,
    };

    public OrderPlacedEvent? Read(ReadOnlySpan<byte> rawJson)
    {
        using JsonDocument doc = JsonDocument.Parse(rawJson);
        JsonElement root = doc.RootElement;

        // Extract known fields — tolerate any missing
        string? orderId = GetStringOrNull(root, "orderId");
        string? customerId = GetStringOrNull(root, "customerId");
        string? customerEmail = GetStringOrNull(root, "customerEmail");
        decimal? total = GetDecimalOrNull(root, "total");
        string? currency = GetStringOrNull(root, "currency");
        string? sku = GetStringOrNull(root, "sku")
                    ?? GetStringOrNull(root, "productId"); // fallback to legacy
        DateTime? occurredAt = GetDateTimeOrNull(root, "occurredAt");

        // Build nested MoneyTotal if absent, derive from flat total+currency
        MoneyValue? totalMoney = null;
        if (root.TryGetProperty("totalMoney", out JsonElement tm))
        {
            totalMoney = JsonSerializer.Deserialize<MoneyValue>(tm.GetRawText(), Options);
        }
        else if (total.HasValue)
        {
            totalMoney = new MoneyValue(total, currency ?? "USD");
        }

        // Extract extensions (unknown fields)
        var extensions = new Dictionary<string, JsonElement>(StringComparer.OrdinalIgnoreCase);
        var known = new HashSet<string>(StringComparer.OrdinalIgnoreCase)
        {
            "orderId", "customerId", "customerEmail", "total",
            "currency", "sku", "productId", "occurredAt", "totalMoney",
            "items", "$type", "$version"
        };

        foreach (JsonProperty prop in root.EnumerateObject())
        {
            if (!known.Contains(prop.Name))
            {
                extensions[prop.Name] = prop.Value.Clone();
            }
        }

        List<LineItem>? items = null;
        if (root.TryGetProperty("items", out JsonElement itemsElement)
            && itemsElement.ValueKind == JsonValueKind.Array)
        {
            items = JsonSerializer.Deserialize<List<LineItem>>(
                itemsElement.GetRawText(), Options);
        }

        return new OrderPlacedEvent
        {
            OrderId = orderId,
            CustomerId = customerId,
            CustomerEmail = customerEmail,
            Total = total,
            Currency = currency,
            Sku = sku,
            Items = items ?? [],
            OccurredAt = occurredAt,
            TotalMoney = totalMoney,
            Extensions = extensions.Count > 0 ? extensions : null,
        };
    }

    private static string? GetStringOrNull(JsonElement element, string propertyName)
    {
        if (element.TryGetProperty(propertyName, out JsonElement prop)
            && prop.ValueKind == JsonValueKind.String)
        {
            return prop.GetString();
        }
        return null;
    }

    private static decimal? GetDecimalOrNull(JsonElement element, string propertyName)
    {
        if (element.TryGetProperty(propertyName, out JsonElement prop))
        {
            switch (prop.ValueKind)
            {
                case JsonValueKind.Number:
                    return prop.GetDecimal();
                case JsonValueKind.String:
                    return decimal.TryParse(prop.GetString(),
                        System.Globalization.NumberStyles.Any,
                        System.Globalization.CultureInfo.InvariantCulture,
                        out decimal val) ? val : null;
            }
        }
        return null;
    }

    private static DateTime? GetDateTimeOrNull(JsonElement element, string propertyName)
    {
        if (element.TryGetProperty(propertyName, out JsonElement prop))
        {
            switch (prop.ValueKind)
            {
                case JsonValueKind.String:
                    return DateTime.TryParse(prop.GetString(),
                        System.Globalization.CultureInfo.InvariantCulture,
                        System.Globalization.DateTimeStyles.AdjustToUniversal,
                        out DateTime dt) ? dt : null;
            }
        }
        return null;
    }
}
```

### 2.5 Tolerant Reader — Functional Composition

Chain tolerant readers into a pipeline:

```csharp
namespace Career.Events.WeakSchema.Pipeline;

public interface ITolerantReader<T>
{
    T? Read(ReadOnlySpan<byte> rawJson, T? fallback = default);
}

public sealed class ChainedTolerantReader<T> : ITolerantReader<T>
{
    private readonly IReadOnlyList<ITolerantReader<T>> _readers;

    public ChainedTolerantReader(IReadOnlyList<ITolerantReader<T>> readers)
    {
        _readers = readers;
    }

    public T? Read(ReadOnlySpan<byte> rawJson, T? fallback = default)
    {
        foreach (ITolerantReader<T> reader in _readers)
        {
            T? result = reader.Read(rawJson);
            if (result is not null)
                return result;
        }
        return fallback;
    }
}

// Specific readers for different schema versions
public sealed class OrderPlacedV1Reader : ITolerantReader<OrderPlacedEvent>
{
    public OrderPlacedEvent? Read(ReadOnlySpan<byte> rawJson, OrderPlacedEvent? fallback = null)
    {
        using JsonDocument doc = JsonDocument.Parse(rawJson);
        JsonElement root = doc.RootElement;

        // v1 shape: flat orderId, customerId, total (decimal), productId
        if (!root.TryGetProperty("productId", out _))
            return null; // not v1

        return new OrderPlacedEvent
        {
            OrderId = root.TryGetProperty("orderId", out var oid) ? oid.GetString() : null,
            CustomerId = root.TryGetProperty("customerId", out var cid) ? cid.GetString() : null,
            Total = root.TryGetProperty("total", out var tot) ? tot.GetDecimal() : null,
            ProductId = root.TryGetProperty("productId", out var pid) ? pid.GetString() : null,
            Currency = "USD", // v1 default
        };
    }
}

public sealed class OrderPlacedV2Reader : ITolerantReader<OrderPlacedEvent>
{
    public OrderPlacedEvent? Read(ReadOnlySpan<byte> rawJson, OrderPlacedEvent? fallback = null)
    {
        using JsonDocument doc = JsonDocument.Parse(rawJson);
        JsonElement root = doc.RootElement;

        // v2 shape: has sku (not productId), may have customerEmail
        if (!root.TryGetProperty("sku", out _))
            return null;

        return new OrderPlacedEvent
        {
            OrderId = GetString(root, "orderId"),
            CustomerId = GetString(root, "customerId"),
            CustomerEmail = GetString(root, "customerEmail"),
            Sku = GetString(root, "sku"),
            Total = GetDecimalOrNull(root, "total"),
            Currency = "USD",
        };
    }

    private static string? GetString(JsonElement el, string prop)
        => el.TryGetProperty(prop, out var v) ? v.GetString() : null;
}
```

### 2.6 Tolerant Reader with JsonNode Mutations

```csharp
using System.Text.Json.Nodes;

namespace Career.Events.WeakSchema.Readers;

public sealed class JsonNodeTolerantReader
{
    public OrderPlacedEvent ReadFromNode(JsonNode node)
    {
        JsonObject obj = node.AsObject();

        // Normalize field: accept sku OR productId
        string? sku = GetString(obj, "sku")
                   ?? GetString(obj, "productId");

        // Normalize total: could be decimal directly or Money object
        decimal? total = null;
        string? currency = "USD";

        if (obj.TryGetPropertyValue("totalMoney", out JsonNode? tmNode) && tmNode is JsonObject tmObj)
        {
            total = tmObj["amount"]?.GetValue<decimal>();
            currency = tmObj["currency"]?.GetValue<string>() ?? currency;
        }
        else if (obj.TryGetPropertyValue("total", out JsonNode? tNode))
        {
            total = tNode.GetValue<decimal>();
        }

        // Coerce: if a numeric string arrives, parse it
        if (total is null && obj.TryGetPropertyValue("total", out JsonNode? tStr)
            && tStr is JsonValue jv && jv.TryGetValue(out string? strVal))
        {
            total = decimal.TryParse(strVal,
                System.Globalization.NumberStyles.Any,
                System.Globalization.CultureInfo.InvariantCulture,
                out decimal d) ? d : null;
        }

        return new OrderPlacedEvent
        {
            OrderId = GetString(obj, "orderId"),
            CustomerId = GetString(obj, "customerId"),
            CustomerEmail = GetString(obj, "customerEmail"),
            Sku = sku,
            Total = total,
            Currency = currency,
            TotalMoney = total.HasValue ? new MoneyValue(total, currency) : null,
            OccurredAt = GetDateTime(obj, "occurredAt"),
        };
    }

    private static string? GetString(JsonObject obj, string key)
        => obj.TryGetPropertyValue(key, out JsonNode? n) && n is JsonValue v
            ? v.TryGetValue(out string? s) ? s : null
            : null;

    private static DateTime? GetDateTime(JsonObject obj, string key)
    {
        if (obj.TryGetPropertyValue(key, out JsonNode? n) && n is JsonValue v)
        {
            if (v.TryGetValue(out DateTime dt)) return dt;
            if (v.TryGetValue(out string? s) && DateTime.TryParse(s, out dt))
                return dt;
        }
        return null;
    }
}
```

## 3. Schema-on-Read vs Schema-on-Write

### 3.1 Definitions

**Schema-on-Write:** The producer validates the event payload against a known schema **before storing it**. The event store rejects events that do not conform. Consumers can assume the shape is well-known.

**Schema-on-Read:** The store accepts any JSON object. Interpretation is deferred to consumers, which each apply their own schema locally. The store is a "dumb" blob repository.

```
Schema-on-Write                    Schema-on-Read
┌─────────┐                        ┌─────────┐
│Producer │                        │Producer │
└────┬────┘                        └────┬────┘
     │ Validate against schema          │ Write any JSON
     ▼                                  ▼
┌─────────────┐                   ┌─────────────┐
│Event Store  │                   │Event Store   │
│(enforced    │                   │(no schema)   │
│ schema)     │                   │              │
└──────┬──────┘                   └──────┬───────┘
       │                                │
       ▼                                ▼
┌─────────────┐                   ┌─────────────┐
│Consumer A   │                   │Consumer A    │
│(trusts      │                   │(reads what   │
│ schema)     │                   │ it needs)    │
└─────────────┘                   └─────────────┘
                                  ┌─────────────┐
                                  │Consumer B    │
                                  │(different    │
                                  │ projection)  │
                                  └─────────────┘
```

### 3.2 Comparison

| Aspect | Schema-on-Write | Schema-on-Read |
|---|---|---|
| **Write throughput** | Lower (validation cost) | Higher (no validation) |
| **Read flexibility** | Lower (fixed schema per consumer) | Higher (each consumer interprets freely) |
| **Data quality** | High (rejects bad data at ingest) | Low (defects are detected late) |
| **Consumer coupling** | Tight (all consumers share schema) | Loose (each consumer evolves independently) |
| **Evolution mechanism** | Schema registry + versioned contracts | Tolerant readers + documentation |
| **Typical use case** | Multi-service event bus | Single-service, team-coordinated |
| **Tooling investment** | High (schema registry, CI checks) | Low (JSON, tests) |
| **Polyglot support** | Needs per-language codegen | Universal (raw JSON) |

### 3.3 Schema-on-Read in Practice

```csharp
using System.Text.Json;

namespace Career.Events.WeakSchema.SchemaOnRead;

// The event store stores an opaque JSON blob.
// Each consumer reads what it needs, ignoring the rest.

public sealed record StoredEvent(
    string EventType,
    int Version,
    ReadOnlyMemory<byte> Data
);

public sealed class ConsumerAOnlyNeedsOrderId
{
    public string? ExtractOrderId(StoredEvent stored)
    {
        using JsonDocument doc = JsonDocument.Parse(stored.Data.Span);
        return doc.RootElement.TryGetProperty("orderId", out JsonElement prop)
            ? prop.GetString()
            : null;
    }
}

public sealed class ConsumerBNeedsTotalAndCurrency
{
    public (decimal? Total, string? Currency) ExtractTotals(StoredEvent stored)
    {
        using JsonDocument doc = JsonDocument.Parse(stored.Data.Span);
        JsonElement root = doc.RootElement;

        decimal? total = null;
        string? currency = null;

        // Try structured money first
        if (root.TryGetProperty("totalMoney", out JsonElement tm))
        {
            if (tm.TryGetProperty("amount", out JsonElement amt))
                total = amt.GetDecimal();
            if (tm.TryGetProperty("currency", out JsonElement cur))
                currency = cur.GetString();
        }
        // Fallback to flat total
        else if (root.TryGetProperty("total", out JsonElement flat))
        {
            total = flat.ValueKind switch
            {
                JsonValueKind.Number => flat.GetDecimal(),
                JsonValueKind.String => decimal.TryParse(flat.GetString(), out decimal d) ? d : null,
                _ => null
            };
        }

        currency ??= root.TryGetProperty("currency", out JsonElement c)
            ? c.GetString()
            : "USD";

        return (total, currency);
    }
}
```

### 3.4 Schema-on-Write with Lightweight Validation

```csharp
using System.Text.Json;
using System.Text.Json.Nodes;

namespace Career.Events.WeakSchema.SchemaOnWrite;

public sealed class LightweightSchemaValidator
{
    private static readonly JsonSerializerOptions Options = new()
    {
        PropertyNamingPolicy = JsonNamingPolicy.CamelCase,
    };

    /// <summary>
    /// Validates that the event JSON contains at least the required fields
    /// and that field types are compatible. Does NOT reject unknown fields.
    /// </summary>
    public (bool IsValid, IReadOnlyList<string> Errors) Validate(
        string eventTypeName,
        int version,
        ReadOnlySpan<byte> json)
    {
        var errors = new List<string>();

        try
        {
            using JsonDocument doc = JsonDocument.Parse(json);
            JsonElement root = doc.RootElement;

            // Required fields depend on event type + version
            var required = GetRequiredFields(eventTypeName, version);

            foreach ((string field, JsonValueKind expectedKind) in required)
            {
                if (!root.TryGetProperty(field, out JsonElement prop))
                {
                    errors.Add($"Missing required field '{field}'");
                    continue;
                }

                if (prop.ValueKind != expectedKind && expectedKind != JsonValueKind.Undefined)
                {
                    // Allow numeric coercion: number can be read as string and vice versa
                    if (!IsCoercible(prop.ValueKind, expectedKind))
                    {
                        errors.Add(
                            $"Field '{field}': expected {expectedKind}, got {prop.ValueKind}");
                    }
                }
            }
        }
        catch (JsonException ex)
        {
            errors.Add($"Invalid JSON: {ex.Message}");
        }

        return (errors.Count == 0, errors.AsReadOnly());
    }

    private static Dictionary<string, JsonValueKind> GetRequiredFields(
        string eventType, int version)
    {
        // In a real system, these would come from a schema registry
        return (eventType, version) switch
        {
            ("orderPlaced", 1) => new()
            {
                ["orderId"] = JsonValueKind.String,
                ["customerId"] = JsonValueKind.String,
                ["total"] = JsonValueKind.Undefined, // any kind
            },
            ("orderPlaced", 2) => new()
            {
                ["orderId"] = JsonValueKind.String,
                ["customerId"] = JsonValueKind.String,
                ["customerEmail"] = JsonValueKind.String,
                ["sku"] = JsonValueKind.String,
                ["total"] = JsonValueKind.Undefined,
            },
            _ => new Dictionary<string, JsonValueKind>(),
        };
    }

    private static bool IsCoercible(JsonValueKind actual, JsonValueKind expected)
    {
        return (actual == JsonValueKind.Number && expected == JsonValueKind.String)
            || (actual == JsonValueKind.String && expected == JsonValueKind.Number);
    }
}
```

## 4. Nullable and Discardable Unknown Fields

### 4.1 The Unknown Field Problem

When an event consumer is written against schema v2 but encounters a v3 event with a new field `discountCode`, there are three options:

1. **Crash** — deserialization throws because `discountCode` is unmapped.
2. **Ignore** — the field is silently discarded.
3. **Capture** — the field is stored in an `Extensions` dictionary for future inspection.

Strategy 2 (ignore) is the **weak schema default**. Strategy 3 (capture) is useful when consumers want to forward unknown fields to downstream systems.

### 4.2 JsonSerializerOptions — Discard or Capture Unknown

```csharp

        // Explicit: no UnmappedMemberHandling error

    {
```

### 4.5 Nullable Wrapper Types

For fields that change type across versions, wrap in a nullable-aware union:

```csharp
using System.Text.Json;
using System.Text.Json.Serialization;

namespace Career.Events.WeakSchema;

[JsonConverter(typeof(MaybeValueConverter))]
public readonly struct Maybe<T>
{
    public T? Value { get; }
    public bool HasValue { get; }
    public bool IsAbsent { get; }  // field was not present at all
    public bool IsNull { get; }    // field was present but set to null

    private Maybe(T? value, bool hasValue, bool isAbsent, bool isNull)
    {
        Value = value;
        HasValue = hasValue;
        IsAbsent = isAbsent;
        IsNull = isNull;
    }

    public static Maybe<T> Present(T value) => new(value, true, false, false);
    public static Maybe<T> Null() => new(default, false, false, true);
    public static Maybe<T> Absent() => new(default, false, true, false);

    public TResult Match<TResult>(
        Func<T, TResult> onPresent,
        Func<TResult> onNull,
        Func<TResult> onAbsent)
        => HasValue ? onPresent(Value!)
           : IsNull ? onNull()
           : onAbsent();
}

public sealed class MaybeValueConverter : JsonConverterFactory
{
    public override bool CanConvert(Type typeToConvert)
    {
        return typeToConvert.IsGenericType
            && typeToConvert.GetGenericTypeDefinition() == typeof(Maybe<>);
    }

    public override JsonConverter? CreateConverter(Type typeToConvert, JsonSerializerOptions options)
    {
        Type elementType = typeToConvert.GetGenericArguments()[0];
        Type converterType = typeof(MaybeConverterInner<>).MakeGenericType(elementType);
        return (JsonConverter)Activator.CreateInstance(converterType)!;
    }

    private sealed class MaybeConverterInner<T> : JsonConverter<Maybe<T>>
    {
        public override Maybe<T> Read(ref Utf8JsonReader reader, Type typeToConvert, JsonSerializerOptions options)
        {
            // If this converter is invoked, the field exists in JSON
            if (reader.TokenType == JsonTokenType.Null)
                return Maybe<T>.Null();

            T? value = JsonSerializer.Deserialize<T>(ref reader, options);
            return Maybe<T>.Present(value!);
        }

        public override void Write(Utf8JsonWriter writer, Maybe<T> value, JsonSerializerOptions options)
        {
            value.Match(
                onPresent: v => JsonSerializer.Serialize(writer, v, options),
                onNull: () => writer.WriteNullValue(),
                onAbsent: () => { /* write nothing */ }
            );
        }
    }
}
```

## 5. Backward and Forward Compatibility

### 5.1 Definitions

| Term | Meaning | Weak Schema Approach |
|---|---|---|
| **Backward compatibility** | New consumer reads old events | Consumer uses `Maybe<T>` / `Extensions` / defaults for fields that old events lack |
| **Forward compatibility** | Old consumer reads new events | Old consumer ignores unknown fields via tolerant reader |
| **Full compatibility** | Both directions work | Both strategies combined |
| **No compatibility** | Any schema change breaks consumers | Rare in weak schema; happens with required fields or type changes |

### 5.2 Rules for Maintaining Compatibility in Weak Schema

| Rule | Description | Example |
|---|---|---|
| **Never remove a field** | Old consumers may still read it | Keep `ProductId` even after renaming to `Sku` |
| **Only add optional fields** | New fields must have a safe default | `CustomerEmail` defaults to `null` or sentinel |
| **Never change field type** | Tolerant readers can coerce limited cases | Avoid `decimal` → `MoneyValue`; add new field instead |
| **Never make a field required** | No `required` keyword in weak schema events | Always use nullable/reference types |
| **Document deprecated fields** | Mark them `[Obsolete]` in code but keep in JSON | `[Obsolete("Use Sku instead")]` |
| **Use `JsonExtensionData` liberally** | Forward compat requires ignoring unknowns | Add `Dictionary<string, JsonElement>? Extensions` |
| **Version the envelope, not the payload** | Envelope version tells readers what to expect | [[7.103 — Event Sourcing — Event Envelope Pattern]] |

### 5.3 Compatibility Matrix

```
Producer writes →
┌─────────────────────────────────────────────────────────────────────────────┐
│ Consumer reads │   v1 Event       │   v2 Event       │   v3 Event          │
├────────────────┼──────────────────┼──────────────────┼─────────────────────┤
│  v1 Consumer   │  ✅ Full         │  ✅ Forward      │  ✅ Forward          │
│                │  compatibility   │  (ignores new    │  (ignores new        │
│                │                  │   fields)        │   fields)            │
├────────────────┼──────────────────┼──────────────────┼─────────────────────┤
│  v2 Consumer   │  ✅ Backward     │  ✅ Full         │  ✅ Forward          │
│                │  (defaults for   │                  │  (ignores v3 fields) │
│                │   missing fields)│                  │                     │
├────────────────┼──────────────────┼──────────────────┼─────────────────────┤
│  v3 Consumer   │  ✅ Backward     │  ✅ Backward     │  ✅ Full             │
│                │  (more defaults) │  (may coerce)    │                     │
└────────────────┴──────────────────┴──────────────────┴─────────────────────┘
```

All entries are green because weak schema **defaults to compatibility**. The cost is that consumers must handle nulls, missing fields, and unexpected shapes at every access point.

### 5.4 Testing Compatibility — Automated Check

```csharp
using System.Text.Json;

namespace Career.Events.WeakSchema.Tests;

public static class CompatibilityVerifier
{
    /// <summary>
    /// Verifies that events written in one version can be read by consumers
    /// at another version without exceptions.
    /// </summary>
    public static CompatibilityResult VerifyBackward<T>(
        string oldEventJson,
        JsonSerializerOptions options) where T : class
    {
        try
        {
            T? result = JsonSerializer.Deserialize<T>(oldEventJson, options);
            if (result is null)
                return CompatibilityResult.Failure("Deserialization returned null");

            return CompatibilityResult.Success();
        }
        catch (JsonException ex)
        {
            return CompatibilityResult.Failure($"JsonException: {ex.Message}");
        }
        catch (Exception ex)
        {
            return CompatibilityResult.Failure($"Unexpected: {ex.GetType().Name}: {ex.Message}");
        }
    }

    /// <summary>
    /// Verifies that an old consumer can read a new event (forward compatibility).
    /// </summary>
    public static CompatibilityResult VerifyForward<T>(
        string newEventJson,
        JsonSerializerOptions options) where T : class
    {
        return VerifyBackward<T>(newEventJson, options);
    }
}
```

## 6. Event Schema Registry — Design and Client

### 6.1 Purpose

A schema registry centralizes schema definitions, assigns numeric IDs to schema versions, and provides compatibility checks. In a weak-schema context, the registry is used for:

1. **Documentation** — consumers discover what fields exist.
2. **Validation** — producers optionally validate against registered schemas.
3. **Migration assistance** — the registry computes diffs between versions.
4. **Schema ID in envelope** — consumers fetch the schema definition by ID for dynamic deserialization.

```
┌──────────────┐     Register schema v1     ┌──────────────────┐
│  Producer A  │ ─────────────────────────►  │                  │
└──────────────┘                             │  Schema Registry │
                                             │                  │
┌──────────────┐     Register schema v2      │  Subjects:       │
│  Producer B  │ ─────────────────────────►  │  - order-placed │
└──────────────┘                             │    v1, v2, v3   │
                                             │  - item-added   │
┌──────────────┐     Fetch schema by ID      │    v1, v2       │
│  Consumer C  │ ◄─────────────────────────  │                  │
└──────────────┘                             └──────────────────┘
```

### 6.2 Schema Registry API

### 6.3 Schema Registry Client

```csharp
using System.Net.Http.Json;
using System.Text.Json;

namespace Career.Events.SchemaRegistry.Client;

public interface ISchemaRegistryClient
{
    Task<SchemaDefinition> RegisterSchemaAsync(
        string subject, string schema, SchemaType schemaType,
        CancellationToken ct = default);

    Task<SchemaDefinition> GetLatestSchemaAsync(
        string subject, CancellationToken ct = default);

    Task<SchemaDefinition> GetSchemaByIdAsync(
        int schemaId, CancellationToken ct = default);

    Task<SchemaDefinition> GetSchemaByVersionAsync(
        string subject, int version, CancellationToken ct = default);

    Task<SchemaCompatibilityCheck> CheckCompatibilityAsync(
        string subject, string schema,
        CancellationToken ct = default);

    Task<IReadOnlyList<int>> GetSchemaVersionsAsync(
        string subject, CancellationToken ct = default);
}

public sealed class SchemaRegistryClient : ISchemaRegistryClient
{
    private readonly HttpClient _httpClient;
    private readonly MemoryCache _cache;

    public SchemaRegistryClient(HttpClient httpClient)
    {
        _httpClient = httpClient;
        _cache = new MemoryCache(new MemoryCacheOptions
        {
            SizeLimit = 512, // max 512 cached schemas
        });
    }

    public async Task<SchemaDefinition> RegisterSchemaAsync(
        string subject, string schema, SchemaType schemaType,
        CancellationToken ct = default)
    {
        var request = new RegisterSchemaRequest
        {
            Subject = subject,
            Schema = schema,
            SchemaType = schemaType.ToString().ToLowerInvariant(),
        };

        HttpResponseMessage response = await _httpClient.PostAsJsonAsync(
            $"schemas/{subject}/versions", request, ct);
        response.EnsureSuccessStatusCode();

        SchemaDefinition created = (await response.Content
            .ReadFromJsonAsync<SchemaDefinition>(JsonContext.Default, ct))!;

        CacheSchema(created);
        return created;
    }

    public async Task<SchemaDefinition> GetLatestSchemaAsync(
        string subject, CancellationToken ct = default)
    {
        string cacheKey = $"latest:{subject}";

        if (_cache.TryGetValue(cacheKey, out SchemaDefinition? cached) && cached is not null)
            return cached;

        HttpResponseMessage response = await _httpClient.GetAsync(
            $"schemas/{subject}/versions/latest", ct);
        response.EnsureSuccessStatusCode();

        SchemaDefinition schema = (await response.Content
            .ReadFromJsonAsync<SchemaDefinition>(JsonContext.Default, ct))!;

        CacheSchema(schema);
        return CacheSchema(cacheKey, schema);
    }

    public async Task<SchemaDefinition> GetSchemaByIdAsync(
        int schemaId, CancellationToken ct = default)
    {
        string cacheKey = $"id:{schemaId}";

        if (_cache.TryGetValue(cacheKey, out SchemaDefinition? cached) && cached is not null)
            return cached;

        HttpResponseMessage response = await _httpClient.GetAsync(
            $"schemas/ids/{schemaId}", ct);
        response.EnsureSuccessStatusCode();

        SchemaDefinition schema = (await response.Content
            .ReadFromJsonAsync<SchemaDefinition>(JsonContext.Default, ct))!;

        return CacheSchema(cacheKey, schema);
    }

    public async Task<SchemaDefinition> GetSchemaByVersionAsync(
        string subject, int version, CancellationToken ct = default)
    {
        string cacheKey = $"{subject}:v{version}";

        if (_cache.TryGetValue(cacheKey, out SchemaDefinition? cached) && cached is not null)
            return cached;

        HttpResponseMessage response = await _httpClient.GetAsync(
            $"schemas/{subject}/versions/{version}", ct);
        response.EnsureSuccessStatusCode();

        SchemaDefinition schema = (await response.Content
            .ReadFromJsonAsync<SchemaDefinition>(JsonContext.Default, ct))!;

        return CacheSchema(cacheKey, schema);
    }

    public async Task<SchemaCompatibilityCheck> CheckCompatibilityAsync(
        string subject, string schema,
        CancellationToken ct = default)
    {
        var request = new CompatibilityCheckRequest
        {
            Subject = subject,
            Schema = schema,
        };

        HttpResponseMessage response = await _httpClient.PostAsJsonAsync(
            $"compatibility/{subject}/versions/latest", request, ct);
        response.EnsureSuccessStatusCode();

        return (await response.Content
            .ReadFromJsonAsync<SchemaCompatibilityCheck>(JsonContext.Default, ct))!;
    }

    public async Task<IReadOnlyList<int>> GetSchemaVersionsAsync(
        string subject, CancellationToken ct = default)
    {
        HttpResponseMessage response = await _httpClient.GetAsync(
            $"schemas/{subject}/versions", ct);
        response.EnsureSuccessStatusCode();

        return (await response.Content
            .ReadFromJsonAsync<List<int>>(JsonContext.Default, ct))!
            .AsReadOnly();
    }

    private SchemaDefinition CacheSchema(SchemaDefinition schema)
    {
        string idKey = $"id:{schema.SchemaId}";
        _cache.Set(idKey, schema, new MemoryCacheEntryOptions
        {
            Size = 1,
            AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(15),
        });
        return schema;
    }

    private SchemaDefinition CacheSchema(string key, SchemaDefinition schema)
    {
        _cache.Set(key, schema, new MemoryCacheEntryOptions
        {
            Size = 1,
            AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(15),
        });
        return schema;
    }

    // Request/response DTOs
    private sealed record RegisterSchemaRequest
    {
        public string Subject { get; init; } = string.Empty;
        public string Schema { get; init; } = string.Empty;
        public string SchemaType { get; init; } = "jsonschema";
    }

    private sealed record CompatibilityCheckRequest
    {
        public string Subject { get; init; } = string.Empty;
        public string Schema { get; init; } = string.Empty;
    }
}

// Source-generated JSON context for AOT compatibility
```

### 6.4 Dynamic Deserialization Using Schema Registry

```csharp
using System.Text.Json;
using System.Text.Json.Nodes;

namespace Career.Events.WeakSchema.Dynamic;

/// <summary>
/// Deserializes event payloads dynamically using schema definitions
/// fetched from the registry. The consumer is not compiled against
/// a known CLR type — it navigates the JSON using the schema as a map.
/// </summary>
public sealed class DynamicSchemaDeserializer
{
    private readonly ISchemaRegistryClient _registry;
    private readonly ILogger<DynamicSchemaDeserializer> _logger;

    public DynamicSchemaDeserializer(
        ISchemaRegistryClient registry,
        ILogger<DynamicSchemaDeserializer> logger)
    {
        _registry = registry;
        _logger = logger;
    }

    public async Task<DynamicEvent> DeserializeAsync(
        EventEnvelope envelope,
        CancellationToken ct = default)
    {
        // Resolve schema from envelope metadata
        string subject = envelope.EventTypeName;
        SchemaDefinition schema;

        if (envelope.DataSchema is not null && int.TryParse(envelope.DataSchema, out int schemaId))
        {
            schema = await _registry.GetSchemaByIdAsync(schemaId, ct);
        }
        else
        {
            schema = await _registry.GetLatestSchemaAsync(subject, ct);
        }

        // Parse payload
        using JsonDocument doc = JsonDocument.Parse(envelope.Data.Span);
        JsonElement root = doc.RootElement.Clone();

        // Build a navigable dynamic event
        return new DynamicEvent
        {
            EventType = envelope.EventTypeName,
            SchemaId = schema.SchemaId,
            SchemaVersion = schema.Version,
            Root = root,
            Index = BuildFieldIndex(root, schema),
        };
    }

    private Dictionary<string, JsonElement> BuildFieldIndex(
        JsonElement root, SchemaDefinition schema)
    {
        var index = new Dictionary<string, JsonElement>(StringComparer.OrdinalIgnoreCase);

        foreach (JsonProperty prop in root.EnumerateObject())
        {
            index[prop.Name] = prop.Value.Clone();
        }

        // Add defaults for schema-declared fields that are missing
        // In a real implementation, parse the schema JSON to discover declared fields
        // For now, just return what's present
        return index;
    }
}

public sealed record DynamicEvent
{
    public string EventType { get; init; } = string.Empty;
    public int SchemaId { get; init; }
    public int SchemaVersion { get; init; }
    public JsonElement Root { get; init; }
    public IReadOnlyDictionary<string, JsonElement> Index { get; init; }
        = new Dictionary<string, JsonElement>();

    public bool TryGetValue<T>(string fieldName, out T? value)
    {
        value = default;
        if (Index.TryGetValue(fieldName, out JsonElement element))
        {
            try
            {
                value = JsonSerializer.Deserialize<T>(element.GetRawText());
                return true;
            }
            catch (JsonException)
            {
                return false;
            }
        }
        return false;
    }

    public T? GetValueOrDefault<T>(string fieldName, T? defaultValue = default)
    {
        return TryGetValue(fieldName, out T? value) ? value : defaultValue;
    }
}
```

## 7. Protobuf and Avro Schema Evolution Comparison

### 7.1 Why Compare?

Protobuf (Protocol Buffers) and Avro are the two most popular **strong schema** serialization formats with explicit evolution rules. Understanding their rules helps a weak-schema practitioner appreciate:

- What weak schema gives up (compile-time safety, compact binary).
- What weak schema retains (maximum flexibility, no code generation).
- How hybrid systems can combine both (e.g., store as JSON, use registry for documentation).

### 7.2 Evolution Rules at a Glance

| Change Type | Protobuf (proto3) | Avro | Weak Schema (JSON) |
|---|---|---|---|
| **Add optional field** | Safe — old readers ignore, new readers see default | Safe — default value from schema | Safe — consumer uses `Maybe<T>` |
| **Add required field** | Not supported (proto3 has no required) | Breaking — must update all readers | Not applicable (no required) |
| **Remove field** | Breaking unless field is reserved | Breaking unless field has default | Breaking — consumer expects it |
| **Rename field** | Safe — tag number stays | Safe — schema resolution by name | Breaking — consumer uses name |
| **Change field type** | Breaking (with few exceptions) | Breaking (except widening) | Breaking — tolerant reader may coerce |
| **Add enum value** | Safe — unknown values become field default | Safe — unknown values become default | Safe — consumer tolerates unknown |
| **Remove enum value** | Breaking unless value is unused | Breaking | Breaking if consumer checks |
| **Change default** | Not applicable (proto3 has no custom defaults) | Safe — readers use new default | Safe — consumer defines own default |
| **Add new message type** | Safe — not referenced by old code | Safe — optional in array | Safe — new field not present |

### 7.3 Protobuf Evolution — Tag-Based Identification

Protobuf identifies fields by **tag number** (1–536,870,911), not by name. This makes renaming safe but collaboration requires careful tag management.

```protobuf
// v1: order-placed.proto
syntax = "proto3";

message OrderPlaced {
  string order_id = 1;
  string customer_id = 2;
  double total = 3;
  string product_id = 4;
}

// v2: Added customer_email, renamed product_id → sku (tag 4 still used)
message OrderPlaced {
  string order_id = 1;
  string customer_id = 2;
  double total = 3;
  string product_id = 4;            // keep old tag for backward compat
  string customer_email = 5;        // new tag
  string sku = 6;                   // new tag — duplicate info with 4
}

// v3: Deprecate field 4 (product_id), use field 6 exclusively
message OrderPlaced {
  string order_id = 1;
  string customer_id = 2;
  double total = 3;
  reserved 4;                        // tag 4 is retired
  reserved "product_id";             // name is retired
  string customer_email = 5;
  string sku = 6;
  MoneyValue total_money = 7;        // new field, old 'total' kept for compat
}

message MoneyValue {
  double amount = 1;
  string currency = 2;
}
```

**Key Protobuf evolution rules for weak schema comparison:**

- **Never change the tag number** of an existing field — this is equivalent to "never rename a JSON property that consumers depend on."
- **Never reuse a retired tag** — always `reserved` it.
- **Remove fields only if you reserve both tag and name** — otherwise old binaries may silently map the wrong type.
- **New fields are optional by default** in proto3 — no `required` keyword exists.
- **Unknown fields are preserved** during deserialization and re-serialization (proto3).

### 7.4 Avro Evolution — Schema Resolution

Avro uses **full schema JSON** stored with data (or referenced by ID from a registry). The reader's schema and writer's schema are resolved at read time.

```json
// v1: order-placed.avsc
{
  "type": "record",
  "name": "OrderPlaced",
  "namespace": "com.career.events",
  "fields": [
    { "name": "orderId", "type": "string" },
    { "name": "customerId", "type": "string" },
    { "name": "total", "type": "double" },
    { "name": "productId", "type": "string" }
  ]
}

// v2: Added customerEmail (default null), added sku, kept productId
{
  "type": "record",
  "name": "OrderPlaced",
  "namespace": "com.career.events",
  "fields": [
    { "name": "orderId", "type": "string" },
    { "name": "customerId", "type": "string" },
    { "name": "total", "type": "double" },
    { "name": "productId", "type": "string" },
    { "name": "customerEmail", "type": ["null", "string"], "default": null },
    { "name": "sku", "type": ["null", "string"], "default": null }
  ]
}

// v3: Use union for total (MoneyValue or double), deprecate productId
{
  "type": "record",
  "name": "OrderPlaced",
  "namespace": "com.career.events",
  "fields": [
    { "name": "orderId", "type": "string" },
    { "name": "customerId", "type": "string" },
    { "name": "total", "type": ["double", "MoneyValue"], "default": 0.0 },
    { "name": "productId", "type": ["null", "string"], "default": null },
    { "name": "customerEmail", "type": ["null", "string"], "default": null },
    { "name": "sku", "type": ["null", "string"], "default": null }
  ]
}
```

**Key Avro evolution rules:**

- **Adding a field is safe if a default is provided** — readers use the default when the field is absent from the writer's schema.
- **Removing a field is safe if the reader's schema has a default** — the reader substitutes the default.
- **Renaming is NOT safe by default** — Avro resolves fields by name. Use aliases in the reader schema to map old names.
- **Changing type is safe only if widening** — `int` → `long`, `string` → `bytes`, or using a union with default.
- **The reader's schema and writer's schema are resolved at read time** — Avro generates no exceptions for unknown fields; it ignores them.

### 7.6 When to Choose Protobuf over Weak Schema

| Condition | Choose Protobuf | Reason |
|---|---|---|
| Inter-service communication across network | Yes | 10–20× smaller than JSON JSON |
| Strict latency requirements (< 1 ms) | Yes | No JSON parsing; direct struct mapping |
| Many consumers in different languages | Yes | Codegen ensures consistent types |
| Complex nested types with deep structures | Yes | More efficient encoding than JSON |
| Need to evolve tag-based (rename without breaking) | Yes | Tags decouple name from identity |
| Small team, single service, rapid iteration | No | Protobuf adds codegen friction |
| Ad-hoc queries against event store (SQL) | No | Binary is not human-readable |
| Event payloads are predominantly string key-value | No | JSON is simpler and self-describing |

### 7.7 When to Choose Avro over Weak Schema

| Condition | Choose Avro | Reason |
|---|---|---|
| Big-data / Hadoop / Kafka ecosystem | Yes | Avro is native in these ecosystems |
| Schema registry is already mandated | Yes | Avro + registry = first-class compat checks |
| Need row-oriented storage with splittable format | Yes | Avro is splittable; JSON is not |
| Strong schema governance required | Yes | Avro inter-schema resolution is precise |
| Ad-hoc scripting and analysis (Python, Spark) | Mixed | Avro has good library support |
| Event store is a relational database | No | JSON is easier to query with SQL/JSONB |
| Need human-readable audit logs | No | JSON is directly readable; Avro needs schema |

## 8. Protobuf Any Pattern in C#

### 8.1 The Problem

When using Protobuf for event sourcing, you need to store and retrieve **polymorphic events** of different types. The `google.protobuf.Any` type wraps any protobuf message with its type URL. In combination with weak schema, the `Any` pattern enables:

- **Schema-on-read** with protobuf binary efficiency.
- **Forward/backward compatibility** at the wrapper level.
- **Dynamic deserialization** based on the type URL.

### 8.2 Protobuf Definitions

### 8.3 C# — Dynamic Unpacking with Weak Schema Fallback

```csharp
using Google.Protobuf;
using Google.Protobuf.WellKnownTypes;
using System.Text.Json;

namespace Career.Events.Protobuf;

// Maps type URLs to protobuf message parsers
public sealed class AnyTypeRegistry
{
    private readonly Dictionary<string, MessageParser> _parsers = new();
    private readonly Dictionary<string, Type> _clrTypes = new();

    public void Register<T>(string? typeUrl = null) where T : IMessage<T>, new()
    {
        MessageParser parser = new MessageParser<T>(() => new T());
        string url = typeUrl ?? $"type.googleapis.com/career.events.v1.{typeof(T).Name}";

        _parsers[url] = parser;
        _clrTypes[url] = typeof(T);
    }

    public bool TryUnpack(Any any, out IMessage? message)
    {
        message = null;

        if (any.TypeUrl is null || !_parsers.TryGetValue(any.TypeUrl, out MessageParser? parser))
            return false;

        message = parser.ParseFrom(any.Value);
        return true;
    }

    public Type? GetClrType(string typeUrl)
    {
        return _clrTypes.GetValueOrDefault(typeUrl);
    }
}
```

### 8.4 Weak Schema Fallback for Unknown Any Types

When a consumer encounters an `Any` with an unregistered type URL, the **weak schema fallback** extracts the raw JSON equivalent:

```csharp
using Google.Protobuf;
using Google.Protobuf.WellKnownTypes;
using System.Text.Json;

namespace Career.Events.Protobuf.Fallback;

public sealed class AnyWeakSchemaReader
{
    private readonly AnyTypeRegistry _registry;

    public AnyWeakSchemaReader(AnyTypeRegistry registry)
    {
        _registry = registry;
    }

    /// <summary>
    /// Attempts to unpack the Any into a typed message.
    /// If the type is unknown, falls back to JSON-like navigation using
    /// the protobuf FieldMask or raw descriptor inspection.
    /// </summary>
    public ReadResult Read(Any any)
    {
        // Attempt typed unpack
        if (_registry.TryUnpack(any, out IMessage? message))
        {
            // Convert to JSON for uniform consumption
            string json = JsonFormatter.Default.Format(message);
            return ReadResult.Typed(message, json);
        }

        // Fallback: parse raw bytes as a protobuf field dictionary
        // This gives us key-value access without the type definition.
        Dictionary<int, object?> rawFields = ExtractRawFields(any);
        return ReadResult.Untyped(any.TypeUrl, rawFields);
    }

    private static Dictionary<int, object?> ExtractRawFields(Any any)
    {
        // Use protobuf reflection on the unknown type
        // Since we don't have the .proto, we use CodedInputStream
        var fields = new Dictionary<int, object?>();

        using var stream = new CodedInputStream(any.Value);
        while (!stream.IsAtEnd)
        {
            uint tag = stream.ReadTag();
            int fieldNumber = WireFormat.GetTagFieldNumber(tag);
            WireFormat.WireType wireType = WireFormat.GetTagWireType(tag);

            object? value = wireType switch
            {
                WireFormat.WireType.Varint => (object?)stream.ReadUInt64(),
                WireFormat.WireType.Fixed64 => (object?)stream.ReadFixed64(),
                WireFormat.WireType.LengthDelimited => (object?)stream.ReadBytes().ToByteArray(),
                WireFormat.WireType.Fixed32 => (object?)stream.ReadFixed32(),
                _ => null,
            };

            fields[fieldNumber] = value;
        }

        return fields;
    }
}

public sealed record ReadResult
{
    public bool IsTyped { get; init; }
    public string TypeUrl { get; init; } = string.Empty;
    public IMessage? TypedMessage { get; init; }
    public string? Json { get; init; }
    public IReadOnlyDictionary<int, object?>? RawFields { get; init; }

    public static ReadResult Typed(IMessage message, string json) => new()
    {
        IsTyped = true,
        TypeUrl = message.Descriptor.FullName,
        TypedMessage = message,
        Json = json,
    };

    public static ReadResult Untyped(string typeUrl, Dictionary<int, object?> rawFields) => new()
    {
        IsTyped = false,
        TypeUrl = typeUrl,
        RawFields = rawFields.AsReadOnly(),
    };
}
```

### 8.5 Protobuf Any with JSON Bridge

For systems that use JSON for storage and protobuf for transport:

```csharp
using Google.Protobuf;
using Google.Protobuf.WellKnownTypes;
using System.Text.Json;

namespace Career.Events.Protobuf.Bridge;

/// <summary>
/// Converts between JSON event representations and Protobuf Any.
/// Allows a hybrid approach: store as JSON (weak schema), transport as protobuf.
/// </summary>
public sealed class JsonToAnyConverter
{
    private readonly AnyTypeRegistry _registry;

    public JsonToAnyConverter(AnyTypeRegistry registry)
    {
        _registry = registry;
    }

    /// <summary>
    /// Converts a JSON string + type URL to a Protobuf Any.
    /// </summary>
    public Any? FromJson(string typeUrl, string json)
    {
        Type? clrType = _registry.GetClrType(typeUrl);
        if (clrType is null)
            return null;

        // Use protobuf JSON parser
        var parser = new JsonParser(new JsonParser.Settings(100));
        IMessage message = (IMessage)parser.Parse(json, clrType);

        return Any.Pack(message);
    }

    /// <summary>
    /// Converts a Protobuf Any to JSON. If the type is unknown,
    /// uses a generic field-name approach.
    /// </summary>
    public string ToJson(Any any)
    {
        if (_registry.TryUnpack(any, out IMessage? message))
        {
            return JsonFormatter.Default.Format(message);
        }

        // Fallback: unknown type — use field numbers as keys
        var fields = ExtractFields(any);
        return JsonSerializer.Serialize(fields);
    }

    private static Dictionary<string, object?> ExtractFields(Any any)
    {
        var fields = new Dictionary<string, object?>();
        using var stream = new CodedInputStream(any.Value);
        while (!stream.IsAtEnd)
        {
            uint tag = stream.ReadTag();
            int fieldNumber = WireFormat.GetTagFieldNumber(tag);
            WireFormat.WireType wireType = WireFormat.GetTagWireType(tag);

            fields[$"field_{fieldNumber}"] = wireType switch
            {
                WireFormat.WireType.Varint => stream.ReadUInt64(),
                WireFormat.WireType.Fixed64 => stream.ReadFixed64(),
                WireFormat.WireType.LengthDelimited => stream.ReadBytes().ToByteArray(),
                WireFormat.WireType.Fixed32 => stream.ReadFixed32(),
                _ => null,
            };
        }
        return fields;
    }
}
```

## 9. Weak Schema Deserializer Pipeline — Reference Implementation

### 9.1 Goals

The pipeline provides a production-grade deserializer that:

1. Accepts JSON events of any schema version.
2. Tolerates missing, extra, and type-coerced fields.
3. Captures unknown fields as extensions.
4. Applies optional upcasters for breaking changes (hybrid mode).
5. Logs schema drift for observability.
6. Caches deserialization plans for performance.

### 9.2 Pipeline Architecture

```
EventEnvelope (raw JSON, version, type)
         │
         ▼
┌──────────────────────────┐
│ 1. Version Router        │  Reads EventTypeVersion from envelope
│    (select strategy)     │  Routes to upcaster chain or direct tolerant read
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ 2. Upcaster Chain        │  Optional: runs registered upcasters
│    (optional)             │  (e.g., type change coercion)
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ 3. Tolerant Deserializer │  JsonDocument parse, extract known fields,
│                          │  capture extensions, apply defaults
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ 4. Post-Processor        │  Validate invariants, enrich, log drift
│                          │  Return typed event or DynamicEvent
└──────────────────────────┘
```

### 9.4 Tolerant Deserializer Implementation

```csharp
using Microsoft.Extensions.Logging;
using System.Text.Json;
using System.Text.Json.Nodes;

namespace Career.Events.WeakSchema.Pipeline;

public sealed class TolerantWeakSchemaDeserializer<T> : IWeakSchemaDeserializer<T>
    where T : class
{
    private readonly ILogger<TolerantWeakSchemaDeserializer<T>> _logger;
    private readonly IUpcasterRegistry? _upcasterRegistry;
    private readonly JsonSerializerOptions _options;

    private static readonly HashSet<string> MetaFields = new(StringComparer.OrdinalIgnoreCase)
    {
        "$type", "$version", "$schemaId", "eventType", "eventTypeVersion",
        "eventId", "streamId", "timestamp",
    };

    public TolerantWeakSchemaDeserializer(
        ILogger<TolerantWeakSchemaDeserializer<T>> logger,
        IUpcasterRegistry? upcasterRegistry = null,
        JsonSerializerOptions? options = null)
    {
        _logger = logger;
        _upcasterRegistry = upcasterRegistry;
        _options = options ?? new JsonSerializerOptions
        {
            PropertyNameCaseInsensitive = true,
            PropertyNamingPolicy = JsonNamingPolicy.CamelCase,
            DefaultIgnoreCondition = JsonIgnoreCondition.WhenWritingNull,
        };
    }

    public DeserializationResult<T> Deserialize(
        EventEnvelope envelope,
        CancellationToken ct = default)
    {
        var warnings = new List<DeserializationWarning>();
        ReadOnlySpan<byte> data = envelope.Data.Span;

        // Phase 1: Optional upcasting
        data = TryUpcast(envelope, data, warnings);

        // Phase 2: Parse JSON
        using JsonDocument doc;
        try
        {
            doc = JsonDocument.Parse(data);
        }
        catch (JsonException ex)
        {
            warnings.Add(new DeserializationWarning(
                "PARSE_ERROR",
                $"Failed to parse JSON: {ex.Message}"));
            return new DeserializationResult<T>(null, warnings.AsReadOnly(), null);
        }

        JsonElement root = doc.RootElement;

        // Phase 3: Standard deserialization
        T? evt;
        Dictionary<string, JsonElement>? extensions = null;

        try
        {
            evt = JsonSerializer.Deserialize<T>(root.GetRawText(), _options);
        }
        catch (JsonException ex)
        {
            warnings.Add(new DeserializationWarning(
                "DESERIALIZATION_ERROR",
                $"Standard deserialization failed: {ex.Message}"));

            // Phase 3a: Fallback — manual tolerant read using JsonElement
            evt = ManualTolerantDeserialize(root, warnings);
        }

        // Phase 4: Capture extensions (unknown fields)
        extensions = CaptureExtensions(root, warnings);

        // Phase 5: Log schema drift
        if (warnings.Count > 0)
        {
            _logger.LogWarning(
                "Deserialized {EventType} v{Version} with {WarningCount} warnings: {Warnings}",
                envelope.EventTypeName,
                envelope.Version,
                warnings.Count,
                string.Join("; ", warnings.Select(w => $"[{w.Code}] {w.Message}")));
        }

        return new DeserializationResult<T>(evt, warnings.AsReadOnly(), extensions);
    }

    private ReadOnlySpan<byte> TryUpcast(
        EventEnvelope envelope,
        ReadOnlySpan<byte> data,
        List<DeserializationWarning> warnings)
    {
        if (_upcasterRegistry is null)
            return data;

        try
        {
            IReadOnlyList<IUpcaster> pipeline =
                _upcasterRegistry.GetPipeline(envelope.EventTypeName, envelope.Version);

            if (pipeline.Count == 0)
                return data;

            ReadOnlySpan<byte> current = data;
            foreach (IUpcaster upcaster in pipeline)
            {
                current = upcaster.Upcast(current);
                warnings.Add(new DeserializationWarning(
                    "UPCAST",
                    $"Applied upcaster {upcaster.GetType().Name} v{upcaster.FromVersion}→v{upcaster.ToVersion}"));
            }
            return current;
        }
        catch (Exception ex)
        {
            warnings.Add(new DeserializationWarning(
                "UPCAST_FAILED",
                $"Upcasting failed, using raw data: {ex.Message}"));
            return data;
        }
    }

    private T? ManualTolerantDeserialize(
        JsonElement root,
        List<DeserializationWarning> warnings)
    {
        // Fallback: create instance and populate what we can
        // This uses reflection to set properties from JSON fields
        try
        {
            T? instance = Activator.CreateInstance<T>();

            if (instance is null)
                return null;

            foreach (JsonProperty prop in root.EnumerateObject())
            {
                if (MetaFields.Contains(prop.Name))
                    continue;

                var clrProp = typeof(T).GetProperty(prop.Name,
                    BindingFlags.IgnoreCase | BindingFlags.Public | BindingFlags.Instance);

                if (clrProp is null || !clrProp.CanWrite)
                {
                    warnings.Add(new DeserializationWarning(
                        "UNKNOWN_FIELD",
                        $"Field '{prop.Name}' has no matching property on {typeof(T).Name}",
                        prop.Name));
                    continue;
                }

                try
                {
                    object? value = JsonSerializer.Deserialize(
                        prop.Value.GetRawText(),
                        clrProp.PropertyType,
                        _options);
                    clrProp.SetValue(instance, value);
                }
                catch (Exception ex)
                {
                    warnings.Add(new DeserializationWarning(
                        "FIELD_COERCION_FAILED",
                        $"Could not coerce field '{prop.Name}' to {clrProp.PropertyType.Name}: {ex.Message}",
                        prop.Name));
                }
            }

            return instance;
        }
        catch (Exception ex)
        {
            warnings.Add(new DeserializationWarning(
                "MANUAL_DESERIALIZATION_FAILED",
                $"Manual tolerant deserialization failed: {ex.Message}"));
            return null;
        }
    }

    private Dictionary<string, JsonElement>? CaptureExtensions(
        JsonElement root,
        List<DeserializationWarning> warnings)
    {
        var known = new HashSet<string>(StringComparer.OrdinalIgnoreCase);
        foreach (var prop in typeof(T).GetProperties())
        {
            known.Add(JsonNamingPolicy.CamelCase.ConvertName(prop.Name));
        }
        known.UnionWith(MetaFields);

        Dictionary<string, JsonElement>? extensions = null;

        foreach (JsonProperty prop in root.EnumerateObject())
        {
            if (!known.Contains(prop.Name))
            {
                extensions ??= new Dictionary<string, JsonElement>(StringComparer.OrdinalIgnoreCase);
                extensions[prop.Name] = prop.Value.Clone();

                warnings.Add(new DeserializationWarning(
                    "EXTENSION",
                    $"Captured unknown field '{prop.Name}' in extensions",
                    prop.Name));
            }
        }

        return extensions;
    }
}
```

```csharp
    {
        _deserializer = deserializer;
        _logger = logger;
    }

    {
        DeserializationResult<OrderPlacedEvent> result =
            _deserializer.Deserialize(envelope, ct);

        if (result.Event is null)

                "Failed to deserialize {EventType} v{Version} — {Warnings}",
                envelope.EventTypeName, envelope.Version,

            return;

        // Store unknown fields in a schema_drift table for later analysis
```

## 10. ADR — Weak Schema Evolution Strategy Decision

### ADR-007: Event Versioning — Weak Schema as Primary Strategy

| Field | Value |
|---|---|
| **Title** | Adopt weak schema evolution (tolerant reader + JSON) as primary versioning strategy |
| **Status** | Accepted |
| **Date** | 2026-06-14 |
| **Drivers** | Small team, single bounded context, rapid feature development, JSON-native event store |
| **Supersedes** | ADR-005 (pure upcasting — found too costly for additive changes) |

### Context

The team recently experienced friction with [[7.109 — Event Sourcing — Event Versioning — Upcasting]] when adding an optional `CustomerEmail` field to `OrderPlaced`. Every consumer required upcaster updates, even though most consumers did not use the email field. The team wants a strategy that allows **additive changes without modifying existing consumers**.

### Decision

We will adopt **weak schema evolution** as our primary versioning strategy with the following rules:

1. **All event properties are nullable/optional** — no field is ever marked `required`.
2. **Every event type carries a `Dictionary<string, JsonElement>? Extensions`** property annotated with `[JsonExtensionData]` for forward compatibility.
3. **New fields are additive only** — never rename or remove a field. Deprecated fields are marked `[Obsolete]` but kept in the type.
4. **Type changes are handled by adding a new field** — e.g., `TotalMoney` alongside `Total`. Old consumers continue reading `Total`.
5. **Breaking changes are handled via new event type** — e.g., `OrderPlacedV2` instead of upcasting.
6. **A schema registry is used for documentation and optional validation** — but not for write-time enforcement.
7. **Consumers log schema drift** — unknown fields are captured and logged for observability.

### Consequences

#### Positive

- Zero upcaster maintenance for additive changes.
- Consumers evolve independently; adding a field does not require a coordinated deploy.
- JSON is human-readable in the event store — debugging is straightforward.
- New event types can be introduced without modifying existing projections (they ignore them).
- The `Extensions` dictionary provides a safety net for unexpected data.

#### Negative

- Every field access in a consumer must handle `null` — defensive coding is mandatory.
- No compile-time guarantee that a field exists — runtime errors from `null` dereference are possible.
- The event store grows faster because deprecated fields are never removed.
- Schema drift is silent unless actively monitored.
- Polyglot consumers must each implement their own tolerant reader logic.

#### Mitigations

| Risk | Mitigation |
|---|---|
| Null reference exceptions | C# 12 required members analysis; null-forgiving operator only in test code |
| Silent data degradation | Schema drift logged at Warning level; dashboards alert on drift rate |
| Accidental breaking change | CI pipeline runs compatibility tests (v1 consumer reads v2 event) |
| Missing field documentation | Schema registry serves as the source of truth; auto-generated from event types |
| Extensions dictionary bloat | Producers must not add fields that are never consumed; periodic drift review |

### Alternatives Considered

| Alternative | Reason for Rejection |
|---|---|
| Pure upcasting ([[7.109]]) | High maintenance cost for additive changes; every field addition touches all consumers |
| Pure protobuf with schema registry | Too much overhead for current team size; codegen friction during rapid prototyping |
| Strict schema-on-write with validation | Kills development velocity; team rightsizing for current stage |
| Event-store-native versioning (Marten) | Vendor lock-in; want storage-agnostic strategy |

## 11. Pitfalls and Anti-Patterns

### 11.1 Pitfall 1 — Silent Data Loss from Unknown Fields

**Problem:** A consumer that uses `[JsonExtensionData]` silently captures unknown fields, but other consumers may discard them. If a producer adds a field that Consumer A stores in Extensions but Consumer B ignores, Consumer B will never see that field when it is later promoted to a known property.

**Solution:** Log unknown fields at Warning level. Periodically review extensions to promote them to known properties. Never rely on extensions for data that must be preserved — use a schema registry.

```csharp
// Bad: silent discard
var evt = JsonSerializer.Deserialize<OrderPlacedEvent>(json);
// evt.Extensions is null; unknown fields are GONE

// Better: capture and log
var result = tolerantDeserializer.Deserialize(envelope);
if (result.Extensions?.Count > 0)
{
    logger.LogWarning("Unknown fields: {Fields}",
        string.Join(", ", result.Extensions.Keys));
}
```

### 11.2 Pitfall 2 — NullReferenceException on Required-Optional Confusion

**Problem:** A field that was originally always present becomes optional in a later schema version. Existing consumers that dereference it without null check crash.

**Solution:** Always use nullable reference types for event properties. Use the `Maybe<T>` pattern for fields that may be absent vs. null. Run compatibility tests in CI.

```csharp
// Vulnerable: will crash if CustomerId is missing
string customerId = evt.CustomerId!; // null-forgiving is a lie
ProcessCustomer(customerId);

// Safe: handle null
if (evt.CustomerId is { } cid)
    ProcessCustomer(cid);
else
    logger.LogWarning("Order {OrderId} has no CustomerId", evt.OrderId);
```

### 11.3 Pitfall 3 — Accidental Breaking Change via Type Change

**Problem:** Changing `Total` from `decimal` to `MoneyValue` (object) causes `JsonException` in consumers that still expect `decimal`.

```csharp
// Producer v3 changes total from 99.99 to {"amount": 99.99, "currency": "USD"}

// Consumer crash:
decimal total = evt.Total!.Value; // InvalidOperationException: object cannot be cast to decimal

// Solution — keep both fields:
public decimal? Total { get; init; }         // legacy — keep forever
public MoneyValue? TotalMoney { get; init; } // new — null in old events
```

### 11.4 Pitfall 4 — Too Much Tolerance Hides Bugs

**Problem:** A producer accidentally emits `OrderPlaced` with a misspelled field `custmerEmail`. The tolerant consumer silently accepts it (logs it as unknown), the email is not sent to the customer, and nobody notices.

**Solution:** Monitor extension field usage. Set up alerts when the count of unknown fields exceeds a threshold. Use schema-on-write validation for critical fields.

```csharp
// Monitoring: Prometheus counter for unknown fields
public class UnknownFieldMetrics
{
    private readonly Counter _unknownFieldCounter = Metrics
        .CreateCounter("events_unknown_fields_total",
            "Total number of unknown fields encountered",
            new CounterConfiguration
            {
                LabelNames = ["event_type", "field_name"]
            });

    public void Record(string eventType, string fieldName)
    {
        _unknownFieldCounter.WithLabels(eventType, fieldName).Inc();
    }
}

// Alert if a single field appears unknown more than N times
```

### 11.5 Pitfall 5 — Extensions Dictionary Becomes a Dumping Ground

**Problem:** Over time, the `Extensions` dictionary accumulates dozens of abandoned fields. Memory footprint grows. No one knows which fields are still relevant.

**Solution:** Produce a weekly report of extension field usage. Remove fields from producers that have been present in Extensions for > 90 days with zero consumption.

```csharp
// Drift report query (PostgreSQL):
// SELECT
//     jsonb_object_keys(extensions) AS unknown_field,
//     COUNT(*) AS occurrence_count,
//     MIN(recorded_at) AS first_seen,
//     MAX(recorded_at) AS last_seen
// FROM events
// WHERE extensions IS NOT NULL
// GROUP BY unknown_field
// ORDER BY occurrence_count DESC;
```

### 11.6 Pitfall 6 — Assuming All Consumers Are Tolerant

**Problem:** A third-party consumer (e.g., analytics pipeline) may not implement tolerant reading. Adding a field causes their deserialization to crash.

**Solution:** Document the weak schema contract explicitly. Provide a client library with the tolerant reader built in. Use the schema registry to generate compatibility reports.

### 11.7 Pitfall 7 — Protobuf Any Without Type Registry

**Problem:** Using `google.protobuf.Any` without a registered type mapping for each event type leads to all consumers becoming untyped (working with raw bytes). This defeats the purpose of using protobuf.

**Solution:** Always pair `Any` with a registry. For deserialization, use the `AnyWeakSchemaReader` that falls back to raw field extraction when the type is unknown.

### 11.8 Pitfall 8 — Schema-on-Read with No Documentation

**Problem:** New team members cannot determine what fields exist on an event because there is no authoritative schema. They must examine producer code or inference from the event store.

**Solution:** Publish schemas to a schema registry even when using schema-on-read. The registry serves as the documentation layer.

### 11.9 Pitfall 9 — Mixing Strong and Weak Schema Without Clear Boundaries

**Problem:** A service boundary uses protobuf for inter-service communication and JSON for storage. Mismatches between the protobuf schema and the stored JSON schema cause data loss during translation.

**Solution:** Define a canonical schema in the registry. Both the protobuf definition and the JSON event type derive from the same canonical definition. Validate equivalence in CI.

### 11.10 Pitfall 10 — Optimistic Deserialization Masks Corruption

**Problem:** Bit-level corruption in the event store produces malformed JSON. The tolerant reader catches the `JsonException` and returns `null`, which the projection interprets as "no event to process" rather than "data corruption".

**Solution:** Separate "parse failure" from "field not found". If JSON parsing fails, store the raw bytes in a DLQ and alert.

```csharp
try
{
    using JsonDocument doc = JsonDocument.Parse(data);
    // ...
}
catch (JsonException ex)
{
    // DO NOT silently swallow — route to dead letter queue
    await deadLetterHandler.MoveToDeadLetterAsync(envelope, ex, ct);
    throw new EventCorruptionException(envelope, ex);
}
```

## 12. Interview Questions

### Q1: What is the Tolerant Reader pattern and how does it apply to event sourcing?

**A:** The Tolerant Reader pattern (derived from the Robustness Principle) dictates that a consumer should accept messages that contain data it does not expect and provide sensible defaults for missing data. In event sourcing, this means:

- Unknown JSON fields are ignored, not rejected.
- Missing optional fields receive default (null/sentinel) values.
- Type coercion is attempted when safe (e.g., string "42" → int 42).

This allows producers to add fields without breaking existing consumers, enabling schema evolution without coordinated deployments.

### Q2: Compare weak schema (JSON-tolerant) with strong schema (Protobuf/Avro) for event versioning. When would you choose each?

**A:** Weak schema (JSON tolerant reader) is best for:
- Small teams, rapid prototyping, single bounded context.
- Events with mostly optional fields.
- Systems where human-readable audit logs are important.
- Polyglot environments with diverse consumer languages.

Strong schema (Protobuf/Avro + schema registry) is best for:
- Multi-service, multi-team event contracts.
- High-throughput, low-latency systems (binary is smaller/faster).
- Environments requiring strict compatibility gates (finance, compliance).
- Big-data ecosystems (Kafka, Hadoop, Spark).

Hybrid is common: store as JSON (weak), transport as protobuf (strong) via the `Any` pattern.

### Q3: How do you handle a field rename in a weak schema system without breaking consumers?

**A:** In weak schema, **you never rename a field** — you add a new field with the new name and keep the old field. Consumers are updated to prefer the new field but fall back to the old one:

```csharp
string sku = evt.Sku ?? evt.ProductId ?? "UNKNOWN";
```

The old field is marked `[Obsolete]` but never removed. After all consumers have migrated (which may be never), the field remains in the stored events but is ignored by new consumers.

### Q4: What is `JsonExtensionData` and how does it support forward compatibility?

**A:** `[JsonExtensionData]` is a `System.Text.Json` attribute that captures all JSON properties that don't match declared properties on the target type into a `Dictionary<string, JsonElement>`. This supports forward compatibility because when a newer event has fields the consumer doesn't know about, those fields are captured instead of causing a deserialization error. The consumer can inspect, log, or forward these extensions.

### Q5: How does schema-on-read differ from schema-on-write, and what are the trade-offs?

**A:** Schema-on-write validates event payloads against a known schema at storage time. Schema-on-read defers interpretation to each consumer.

Schema-on-write:
- Pro: High data quality, consistent shape, early error detection.
- Con: Higher write latency, requires coordinated schema changes, tighter coupling.

Schema-on-read:
- Pro: Maximum flexibility, consumers evolve independently, lower write latency.
- Con: Silent data degradation, each consumer must handle missing fields, schema knowledge is distributed.

### Q6: Explain the role of a schema registry in a weak-schema system.

**A:** Even in a weak-schema system (where the store doesn't enforce schemas), a schema registry provides:
1. **Documentation** — a centralized source of truth for event field definitions.
2. **Compatibility checks** — CI gate that warns before a producer adds a potentially breaking field.
3. **Dynamic deserialization** — consumers fetch the schema definition to navigate unknown event shapes.
4. **Governance** — track which versions exist, what changed, and when.

The registry is advisory, not mandatory. Producers can still write events not in the registry, but those events won't be discoverable.

### Q7: What are the risks of being "too tolerant" in event deserialization?

**A:** Risks include:
1. **Silent data loss** — misspelled fields are accepted and logged but never processed.
2. **Masked corruption** — bit-level corruption produces valid JSON but nonsensical data.
3. **Missing field defaults** — a default (null, 0, empty string) may be indistinguishable from an actual value, leading to incorrect business logic.
4. **Extensions bloat** — abandoned fields accumulate in the Extensions dictionary, consuming storage and memory.
5. **False sense of compatibility** — the system appears to handle evolution but silently drops or mangles semantic content.

Mitigations: log warnings, alert on drift, separate parse failure from semantic failure, and use schema-on-write for critical fields.

### Q8: How would you implement a hybrid system that stores events as JSON but transports them as Protobuf?

**A:** Using the Protobuf `google.protobuf.Any` pattern:

1. In the event store, events are stored as JSON (weak schema, human-readable).
2. When publishing to a message bus, the event is converted to Protobuf using `Any.Pack()`.
3. The consumer deserializes the `Any` by looking up the type URL in a registry.
4. If the type is known, it unpacks to the typed Protobuf message.
5. If the type is unknown, it falls back to raw field extraction (weak schema fallback).

The `JsonToAnyConverter` class handles the JSON→Protobuf direction. The `AnyWeakSchemaReader` handles the fallback for unknown types. This gives the benefits of compact binary transport with JSON-native storage.
