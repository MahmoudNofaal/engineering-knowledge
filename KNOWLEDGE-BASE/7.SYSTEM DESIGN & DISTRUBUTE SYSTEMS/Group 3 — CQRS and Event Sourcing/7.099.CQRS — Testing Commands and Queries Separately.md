# 7.099 — CQRS — Testing Commands and Queries Separately

## Overview

Testing a CQRS system requires a fundamental shift in mindset: **commands** (writes) and **queries** (reads) have distinctly different failure modes, dependencies, and verification strategies. Commands mutate state through side effects (events, DB writes, message bus publishes) and their correctness is measured by what *changed*. Queries return data without side effects and their correctness is measured by what *was returned*. Testing them together conflates unrelated concerns, masks regression sources, and produces brittle suites that fail for ambiguous reasons. Testing them **separately** yields faster feedback, clearer ownership of failures, and higher coverage per test — each test answers exactly one question.

This note provides a complete, production-grade testing strategy for CQRS architectures: unit tests for handlers in isolation, integration tests with real infrastructure via Testcontainers + Respawn, projection tests with idempotency verification, pipeline behavior tests with stub handlers, and end-to-end command→event→read model tests for critical business flows. Every technique is illustrated with executable C# 12 / .NET 8 xUnit code using NSubstitute for mocking, FluentAssertions for readable assertions, and Respawn for database reset.

## 1. Why Test Commands and Queries Separately

### 1.1 Fundamental Differences

| Concern | Command | Query |
|---|---|---|
| Side effects | Yes (events, DB writes, bus publishes) | None |
| Failure modes | Concurrency conflicts, invariant violations, validation errors, optimistic lock exceptions | Stale data, missing projections, cache misses, authorization denials |
| Dependency heaviness | Repositories, Unit of Work, event bus, message broker, domain services | Read model abstractions, cache providers, lightweight DTO factories |
| Test shape | Arrange (seed) → Act (handler) → Assert (state + side effects) | Arrange (seed read model) → Act (handler) → Assert (return value) |
| Idempotency concern | Critical — commands must be safe to retry | Rarely needed — queries are inherently idempotent |
| Transaction scope | Yes — commands wrap a DB transaction | No — queries read committed data |
| Performance sensitivity | Moderate — typically sub-100ms | High — queries are user-facing, often <10ms SLA |

### 1.2 The Cost of Mixing

When a test mixes command and query concerns, a failure leaves the developer guessing: Did the command handler fail to persist the data? Did the event bus fail to deliver the event? Did the projection fail to update the read model? Did the query handler fail to map the read model? Each question corresponds to a different layer. A separated test pinpoints the exact layer instantly. A mixed test requires manual debugging to disambiguate.

### 1.3 When to Break the Rule

There are two cases where a combined test is justified: (1) **Critical-path smoke tests** — one or two per bounded context that validate the entire round trip (command → event → projection → query). These are not diagnostic; they are confidence checks. (2) **Regression verification** — after a deep refactor of the event pipeline, run a combined test to catch integration regressions before they reach staging.

## 2. Unit Testing Command Handlers

A command handler should be testable with only a **repository abstraction** mocked. No database, no message bus, no read model — just the handler's business logic. The handler loads an aggregate, calls a domain method, and persists the result.

### 2.1 Core Abstractions

### 2.2 Aggregate Under Test

`csharp
// Domain / Orders / Order

public sealed class Order : AggregateRoot<Guid>
{
    public string CustomerId { get; private set; }
    public string ShippingAddress { get; private set; }
    public OrderStatus Status { get; private set; }
    public string? CancellationReason { get; private set; }
    public Currency Currency { get; private set; }
    private readonly List<OrderLineItem> _items = [];
    public IReadOnlyList<OrderLineItem> Items => _items.AsReadOnly();
    public DateTime CreatedAt { get; private set; }
    public DateTime? ModifiedAt { get; private set; }
    public long Version { get; private set; }

    private Order() { }

    public Order(Guid id, string customerId, string shippingAddress, Currency currency)
    {
        Id = id; CustomerId = customerId; ShippingAddress = shippingAddress;
        Status = OrderStatus.Pending; Currency = currency;
        CreatedAt = DateTime.UtcNow; Version = 1;
        RaiseDomainEvent(new OrderPlaced(id, customerId, shippingAddress, 0m, currency, CreatedAt));
    }

    public void AddItem(string sku, int quantity, decimal unitPrice)
    {
        if (Status != OrderStatus.Pending)
            throw new DomainException(\"Can only add items to a pending order.\");
        var existing = _items.FirstOrDefault(i => i.Sku == sku);
        if (existing is not null)
            existing.UpdateQuantity(existing.Quantity + quantity);
        else
            _items.Add(new OrderLineItem(Guid.NewGuid(), Id, sku, quantity, unitPrice));
        ModifiedAt = DateTime.UtcNow; Version++;
        RaiseDomainEvent(new OrderItemAdded(Id, sku, quantity, unitPrice));
    }

    public decimal CalculateTotal() => _items.Sum(i => i.Quantity * i.UnitPrice);

    public void Cancel(string reason)
    {
        if (Status == OrderStatus.Cancelled) return;
        if (Status == OrderStatus.Shipped)
            throw new DomainException(\"Cannot cancel a shipped order.\");
        Status = OrderStatus.Cancelled; CancellationReason = reason;
        ModifiedAt = DateTime.UtcNow; Version++;
        RaiseDomainEvent(new OrderCancelled(Id, reason, ModifiedAt.Value));
    }

    public void Ship()
    {
        if (Status != OrderStatus.Pending)
            throw new DomainException(\"Can only ship a pending order.\");
        if (_items.Count == 0)
            throw new DomainException(\"Cannot ship an order with no items.\");
        Status = OrderStatus.Shipped; ModifiedAt = DateTime.UtcNow; Version++;
        RaiseDomainEvent(new OrderShipped(Id, ModifiedAt.Value));
    }
}

public enum OrderStatus { Pending, Shipped, Cancelled }

public sealed class OrderLineItem
{
    public Guid Id { get; private set; } public Guid OrderId { get; private set; }
    public string Sku { get; private set; } public int Quantity { get; private set; }
    public decimal UnitPrice { get; private set; }
    internal OrderLineItem(Guid id, Guid orderId, string sku, int quantity, decimal unitPrice)
    { Id = id; OrderId = orderId; Sku = sku; Quantity = quantity; UnitPrice = unitPrice; }
    internal void UpdateQuantity(int newQuantity) => Quantity = newQuantity;
}

public enum Currency { USD, EUR, GBP, JPY }
`



### 2.3 Command and Handler

`csharp
// Application / Orders / Commands / CancelOrder

public record CancelOrderCommand(Guid OrderId, string Reason) : ICommand<Unit>;

internal sealed class CancelOrderCommandHandler(
    IOrderRepository repo,
    ILogger<CancelOrderCommandHandler> logger)
    : ICommandHandler<CancelOrderCommand, Unit>
{
    public async Task<Unit> Handle(CancelOrderCommand command, CancellationToken ct)
    {
        logger.LogInformation("Cancelling order {OrderId} for reason: {Reason}",
            command.OrderId, command.Reason);
        Order? order = await repo.GetByIdAsync(command.OrderId, ct)
            ?? throw new NotFoundException($\"Order {command.OrderId} was not found.\");
        order.Cancel(command.Reason);
        await repo.SaveChangesAsync(ct);
        logger.LogInformation(\"Order {OrderId} successfully cancelled.\", command.OrderId);
        return Unit.Value;
    }
}

public class NotFoundException(string message) : Exception(message);
public class DomainException(string message) : Exception(message);
`

    public async Task Handle_WhenOrderMissing_ThrowsNotFoundException()

    public async Task Handle_WhenOrderMissing_DoesNotCallSave()

        _repo.GetByIdAsync(Arg.Any<Guid>(), Arg.Any<CancellationToken>())

        await act.Should().ThrowAsync<NotFoundException>();
        await _repo.DidNotReceive().SaveChangesAsync(Arg.Any<CancellationToken>());

        await _handler.Handle(
            new CancelOrderCommand(order.Id, \"new reason\"), CancellationToken.None);
        order.CancellationReason.Should().Be(\"original reason\");

    [Fact]
    public async Task Handle_WhenOrderShipped_ThrowsDomainException()

        var order = ShippedOrder();
        _repo.GetByIdAsync(order.Id, Arg.Any<CancellationToken>())
            .Returns(Task.FromResult<Order?>(order));
        Func<Task> act = () => _handler.Handle(

    public async Task Handle_WithEmptyReason_StillSucceeds()

        var order = PendingOrder();

            .Returns(Task.FromResult<Order?>(order));
        await _handler.Handle(
            new CancelOrderCommand(order.Id, string.Empty), CancellationToken.None);
        order.Status.Should().Be(OrderStatus.Cancelled);

## 3. Unit Testing Query Handlers

Query handlers have no side effects. They fetch data from a **read model interface** and project it into a response DTO. Tests are pure Arrange → Act → Assert with no mutation verification.

### 3.1 Query Handler

`csharp
// Application / Orders / Queries / GetOrderSummary

public record GetOrderSummaryQuery(Guid OrderId, string RequestedByUserId) : IQuery<OrderSummaryDto?>;

internal sealed class GetOrderSummaryQueryHandler(
    IOrderSummaryReadModel readModel,
    IAuthorizationService auth,
    ILogger<GetOrderSummaryQueryHandler> logger)
    : IQueryHandler<GetOrderSummaryQuery, OrderSummaryDto?>
{
    public async Task<OrderSummaryDto?> Handle(GetOrderSummaryQuery query, CancellationToken ct)
    {
        logger.LogInformation(\"Fetching order summary {OrderId} for user {UserId}\",
            query.OrderId, query.RequestedByUserId);
        var summary = await readModel.GetOrderSummaryAsync(query.OrderId, ct);
        if (summary is null) return null;
        if (!await auth.CanViewOrderAsync(query.RequestedByUserId, summary.CustomerId))
            throw new ForbiddenException(
                $\"User {query.RequestedByUserId} cannot view order {query.OrderId}.\");
        return summary;
    }
}

public interface IOrderSummaryReadModel
{
    Task<OrderSummaryDto?> GetOrderSummaryAsync(Guid orderId, CancellationToken ct);
    Task<IReadOnlyList<OrderSummaryDto>> GetOrdersByCustomerAsync(
        string customerId, int page, int size, CancellationToken ct);
    Task InsertAsync(OrderSummaryDto dto, CancellationToken ct);
    Task UpdateStatusAsync(Guid orderId, string status, DateTime modifiedAt, CancellationToken ct);
    Task UpdateStatusAndTotalAsync(Guid orderId, string status, decimal total,
        DateTime modifiedAt, CancellationToken ct);
}

public record OrderSummaryDto(
    Guid OrderId, string CustomerId, decimal Total, string Status, DateTime LastModified);
`

## 4. Testing Pipeline Behaviors in Isolation

MediatR pipeline behaviors cross-cut validation, logging, auth, retry, and transaction wrapping. Test each behavior with a stub inner handler — no real command or query handler involved.

### 4.1 Logging Behavior

`csharp
internal sealed class LoggingBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse> where TRequest : IBaseRequest
{
    private readonly ILogger<LoggingBehavior<TRequest, TResponse>> _logger;
    public LoggingBehavior(ILogger<LoggingBehavior<TRequest, TResponse>> logger) => _logger = logger;

    public async Task<TResponse> Handle(TRequest request,
        RequestHandlerDelegate<TResponse> next, CancellationToken ct)
    {
        _logger.LogInformation(\"Handling {Request} at {Time}\", typeof(TRequest).Name, DateTime.UtcNow);
        Stopwatch sw = Stopwatch.StartNew();
        try
        {
            TResponse response = await next();
            sw.Stop();
            _logger.LogInformation(\"Handled {Request} in {ElapsedMs}ms — success\",
                typeof(TRequest).Name, sw.ElapsedMilliseconds);
            return response;
        }
        catch (Exception ex)
        {
            sw.Stop();
            _logger.LogError(ex, \"Handled {Request} in {ElapsedMs}ms — failed with {ExceptionType}\",
                typeof(TRequest).Name, sw.ElapsedMilliseconds, ex.GetType().Name);
            throw;
        }
    }
}
`

#### Unit Test

### 4.2 Validation Behavior

`csharp
internal sealed class ValidationBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse> where TRequest : IBaseRequest
{
    private readonly IEnumerable<IValidator<TRequest>> _validators;
    public ValidationBehavior(IEnumerable<IValidator<TRequest>> validators) => _validators = validators;

    public async Task<TResponse> Handle(TRequest request,
        RequestHandlerDelegate<TResponse> next, CancellationToken ct)
    {
        if (!_validators.Any()) return await next();
        var context = new ValidationContext<TRequest>(request);
        var failures = _validators
            .Select(v => v.Validate(context))
            .SelectMany(r => r.Errors)
            .Where(f => f != null).ToList();
        if (failures.Count != 0) throw new ValidationException(failures);
        return await next();
    }
}

### 4.3 Retry Behavior

`csharp
internal sealed class RetryBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse> where TRequest : IBaseRequest
{
    private readonly int _maxRetries;
    private readonly TimeSpan _baseDelay;
    private readonly IEnumerable<Type> _transientExceptionTypes;

    public RetryBehavior(int maxRetries, TimeSpan baseDelay, IEnumerable<Type> transientExceptionTypes)
    { _maxRetries = maxRetries; _baseDelay = baseDelay; _transientExceptionTypes = transientExceptionTypes; }

    public async Task<TResponse> Handle(TRequest request,
        RequestHandlerDelegate<TResponse> next, CancellationToken ct)
    {
        int attempt = 0;
        while (true)
        {
            try { return await next(); }
            catch (Exception ex) when (attempt < _maxRetries
                && _transientExceptionTypes.Any(t => t.IsInstanceOfType(ex)))
            {
                attempt++;
                await Task.Delay(_baseDelay * Math.Pow(2, attempt - 1), ct);
            }
        }
    }
}
`

### 4.5 Transaction Behavior

`csharp
internal sealed class TransactionBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse> where TRequest : IBaseRequest
{
    private readonly IUnitOfWork _uow;
    public TransactionBehavior(IUnitOfWork uow) => _uow = uow;

    public async Task<TResponse> Handle(TRequest request,
        RequestHandlerDelegate<TResponse> next, CancellationToken ct)
    {
        if (_uow.HasActiveTransaction) return await next();
        await using var tx = await _uow.BeginTransactionAsync(ct);
        try
        {
            var response = await next();
            await tx.CommitAsync(ct);
            return response;
        }
        catch { await tx.RollbackAsync(CancellationToken.None); throw; }
    }
}

### 5.6 Why Testcontainers + Respawn?

| Concern | In-Memory DB | Testcontainers + Respawn |
|---|---|---|
| Engine fidelity | Low — different engine, no constraints | Exact SQL Server/Postgres |
| Schema enforcement | None | Full FK, index, constraint enforcement |
| Concurrency behavior | No locking simulated | Real locking, deadlocks, row versioning |
| Transaction semantics | No real transactions | Full commit/rollback |
| CI portability | High — no deps | Requires Docker |
| Per-test isolation | Per-test context (fast) | Respawn truncates (fast, ~50ms) |
| Setup time | None | First test: ~10s container start. Subsequent: ~50ms |

public class OrderFulfillmentSaga(
    ICommandBus commandBus, ISagaRepository sagaRepository, ILogger<OrderFulfillmentSaga> logger)
    : INotificationHandler<OrderPlaced>, INotificationHandler<InventoryReserved>,
      INotificationHandler<PaymentSucceeded>, INotificationHandler<OrderShipped>

    public async Task Handle(OrderPlaced @event, CancellationToken ct)
    {
        var saga = new FulfillmentSagaData(@event.OrderId, @event.CustomerId, @event.Total, DateTime.UtcNow);
        await sagaRepository.SaveAsync(saga, ct);
        await commandBus.SendAsync(new ReserveInventoryCommand(@event.OrderId, @event.Items), ct);
    }

    public async Task Handle(InventoryReserved @event, CancellationToken ct)
    {
        var saga = await sagaRepository.LoadAsync<FulfillmentSagaData>(@event.OrderId, ct);
        if (saga is null) return;
        saga.InventoryReserved = true;
        await sagaRepository.SaveAsync(saga, ct);
        await commandBus.SendAsync(new ChargePaymentCommand(@event.OrderId, saga.CustomerId, saga.Total), ct);
    }

    public async Task Handle(PaymentSucceeded @event, CancellationToken ct)
    {
        var saga = await sagaRepository.LoadAsync<FulfillmentSagaData>(@event.OrderId, ct);

        await sagaRepository.SaveAsync(saga, ct);

    public async Task Handle(OrderShipped @event, CancellationToken ct)
    {
        var saga = await sagaRepository.LoadAsync<FulfillmentSagaData>(@event.OrderId, ct);
        if (saga is null) return;
        saga.CompletedAt = DateTime.UtcNow;
        await sagaRepository.SaveAsync(saga, ct);

}

## 10. Testing Query Caching Strategies

### 10.1 Cached Query Decorator

`csharp
public sealed class CachedQueryHandlerDecorator<TQuery, TResponse>(
    IQueryHandler<TQuery, TResponse> inner,
    ICacheProvider cache,
    ILogger<CachedQueryHandlerDecorator<TQuery, TResponse>> logger)
    : IQueryHandler<TQuery, TResponse> where TQuery : IQuery<TResponse>
{
    private static readonly string Prefix = typeof(TQuery).FullName!;

    public async Task<TResponse?> Handle(TQuery query, CancellationToken ct)
    {
        var key = $\"{Prefix}:{JsonSerializer.Serialize(query)}\";
        var cached = await cache.GetAsync<TResponse>(key, ct);
        if (cached is not null) { logger.LogDebug(\"Cache hit {Key}\", key); return cached; }
        logger.LogDebug(\"Cache miss {Key}\", key);
        var result = await inner.Handle(query, ct);
        if (result is not null) await cache.SetAsync(key, result, TimeSpan.FromMinutes(5), ct);
        return result;
    }
}
`

## 11. Mocking Strategies for CQRS Tests

### 11.1 What to Mock

| Dependency | Mock in Command Test | Mock in Query Test | Real in Integration | Notes |
|---|---|---|---|---|
| Repository (write side) | Yes | N/A | Yes | IOrderRepository |
| Read model | N/A | Yes | Yes | IOrderSummaryReadModel |
| Unit of Work | Implicit via repo | N/A | Yes | Real transaction |
| Event bus / message broker | Yes | N/A | In-memory spy | Never mock in E2E |
| Pipeline behaviors | No (tested separately) | No (tested separately) | Yes (registered) | Verify ordering |
| Time provider | Yes | Yes | Real clock | Use TimeProvider |
| Current user / auth | Yes | Yes | Stub | ICurrentUser |
| Logger | Yes | Yes | Substitute | ILogger<T> |
| Cache provider | N/A | Yes (decorator) | If needed | ICacheProvider |

### 11.2 Strict vs. Loose

- **Command tests** — loose mocks (default). Verify aggregate state + key side effects like SaveChangesAsync call count.
- **Query tests** — loose. Verify only the read model call and optionally logger.
- **Pipeline behaviors** — stricter. Verify 
ext() was/was-not called, exceptions caught/rethrown.
- **Saga tests** — strict on command bus (exact commands in order), loose on saga repository.

## 14. ADR: Testing Strategy for CQRS

`yaml
ADR-0010: Testing Strategy for CQRS
Status: Accepted
Date: 2026-06-13
Deciders: Platform Architecture Team

Context:
  The team adopted CQRS with MediatR pipeline behaviors and Event Sourcing.
  The testing strategy must: (1) give fast feedback during development,
  (2) catch regressions at the correct layer, (3) run reliably in CI,
  (4) support parallel execution, (5) be reproducible across machines.

Decision:
  1. Command handlers: unit test with mocked repository interfaces.
     Assert aggregate state AND SaveChangesAsync call count.
  2. Query handlers: unit test with mocked read model interfaces.
     Assert returned DTOs, verify authorization and caching decorators.
  3. Pipeline behaviors: unit test with stub RequestHandlerDelegate.
     No real handler or DB involved.
  4. Projections: unit test with mocked read models, including idempotency
     tests (replay same event twice, assert no corruption).
  5. Integration tests: Testcontainers (SQL Server 2022 / Postgres 16)
     with Respawn for per-test cleanup. Never use in-memory databases.
  6. E2E tests: 1–3 per bounded context for critical flows.
     Command → event → projection → query via in-memory bus spy.
  7. Unit tests run in parallel; integration/E2E sequential per collection.
  8. Test data created via factory methods and builders (not shared fixtures).

Consequences:
  + Each test answers exactly one question.
  + Real DB integration catches engine-specific constraints and locking.
  + Respawn enables fast per-test cleanup (~50ms).
  + Pipeline isolation tests prevent cross-cutting from masking failures.
  – E2E tests are slow (~1-3s each) — must be kept minimal.
  – Docker required for integration tests in CI.

Rejected Alternatives:
  - In-memory databases (EF Core InMemory, SQLite): miss engine-specific
    constraints, different locking semantics, no FK enforcement.
  - Shared database per test class: order-dependent tests.
  - Full E2E for every handler: too slow and fragile.

Implementation Notes:
  - Integration tests tagged [Trait(\"Requires\", \"Docker\")] for CI stage separation.
  - CqrsTestcontainersFixture is a singleton per collection.
  - InMemoryEventBus spy avoids needing a real message broker in tests.
`

## 15. Interview Questions

### Q1: Why should commands and queries be tested separately in CQRS?

Commands mutate state and are verified through side effects (events, DB writes). Queries are pure reads verified through returned data. Testing together conflates concerns, doubling failure reasons without improving diagnostics. Separate tests give faster feedback and clearer failure localization.

### Q2: What is the ideal isolation boundary for a command handler unit test?

Mock the repository interface (e.g., IOrderRepository), return a known aggregate from GetByIdAsync, call the handler, assert aggregate state, verify SaveChangesAsync call count. Never mock the aggregate itself. Never involve a real database.

### Q3: How do you test that a command handler publishes the right events?

Use an in-memory event bus spy or mock. After calling Handle, assert PublishAsync was called with the expected event type and properties. For stronger coverage, combine with a projection unit test that verifies the event is consumed correctly.

### Q4: How do you handle integration tests without Docker?

Options ranked: (1) Local full DB instance + Respawn, (2) GitHub Actions service containers (CI-only), (3) LocalDB (partial engine). Avoid in-memory databases — they miss engine-specific constraints, locking, and transaction semantics, and have caused production bugs.

### Q5: What's the difference between testing a projection and a query handler?

**Projection** = event consumer that *writes* to the read model. Test by feeding events, assert read model changes. **Query handler** = read model *reader* that returns a DTO. Test by seeding read model state, assert returned DTO. Projection is the writer; query handler is the reader.

### Q6: How do you test idempotency of a projection?

Replay the same event twice and assert the read model state is identical. For upsert-based projections: verify no duplicates and correct state. For deduplication-based: verify the second replay was skipped and state is correct.

### Q7: Should you write E2E command→event→projection→query tests?

Only for critical business flows (1–3 per bounded context). E2E tests are slow (1-3s each), fragile (any layer change breaks them), and have poor diagnostic value. Cover the same logic with faster unit + integration tests.

### Q8: How do you manage test data cleanup in integration tests?

Use **Respawn** — truncates all tables in FK-safe order before each test. Combined with Testcontainers: InitializeAsync starts the container and creates the Respawner. Each test seeds data → acts → asserts → calls ResetDatabaseAsync(). No container restart needed.

### Q9: How do you test a saga that coordinates multiple steps?

Unit test: mock the command bus and saga repository. Feed events in sequence and verify the exact commands sent at each step. Integration test: use the real saga repository, in-memory command bus, and Testcontainers DB — verify the saga completes end-to-end.

### Q10: How do you test optimistic concurrency?

Unit test: mock the repository to throw DbUpdateConcurrencyException on save. Integration test: launch two concurrent tasks operating on the same aggregate; assert exactly one succeeds and one gets a concurrency exception.

### Q11: How do you test a cached query handler?

Use the decorator pattern. Test: (1) cache hit — mock cache returns value, verify inner handler NOT called. (2) cache miss — mock cache returns null, verify inner handler called and cache.SetAsync called. (3) null result — verify SetAsync not called. (4) cache failure — verify graceful fallthrough to inner handler.

### Q12: What's the role of ICollectionFixture in xUnit for CQRS tests?

It shares a heavyweight fixture (e.g., Testcontainers container) across test classes without recreating it per test. Combined with DisableParallelization = true, it enforces sequential execution for integration tests that share a database.

## 16. Self-Check Questions

### Primary (12)

1. **What three assertions should every command handler unit test include?**  \
   *Aggregate state mutation, SaveChangesAsync call count (Received(1)), domain event emission.*

2. **How does a query handler unit test differ from a command handler in side-effect verification?**  \
   *Query handlers have zero side effects. Assert only the return value; verify no write operations were called.*

3. **What is the purpose of the RequestHandlerDelegate<TResponse> stub in a behavior test?**  \
   *It simulates the next delegate in the MediatR pipeline, allowing the test to control whether next() succeeds or throws.*

4. **Why verify SaveChangesAsync call count, not just existence?**  \
   *A handler must call it exactly once. Received(1) catches zero-call bugs (missing persist) and multi-call bugs (unnecessary round trips).*

5. **Minimum mocks for an OrderSummaryProjection unit test?**  \
   *IOrderSummaryReadModel and ILogger<OrderSummaryProjection>.*

6. **How does Respawn achieve fast DB cleanup?**  \
   *Discovers all tables, generates TRUNCATE TABLE in FK-safe order, executes them. ~50ms vs. ~10s container restart.*

7. **What columns should a deduplication table include?**  \
   *EventId (unique event identifier) and HandlerId (projection handler name) — composite PK prevents reprocessing.*

8. **Why verify 
ext() was NOT called on validation failure?**  \
   *To prove the behavior short-circuits the pipeline. If next() runs after validation failure, the handler executes with invalid data.*

9. **Why are in-memory databases insufficient for CQRS integration tests?**  \
   *Miss engine-specific constraints, different locking/transaction semantics, no production query plans. Bugs caught only in production cost more.*

10. **Parallelization strategy for unit vs. integration tests?**  \
    *Unit: parallel (default). Integration: sequential per collection (DisableParallelization=true). Different DB schemas = separate collections = parallel.*

11. **How to test a command's event is consumed by the correct projection?**  \
    *E2E pattern: execute command with real repo + in-memory bus spy, dispatch captured events to the projection, query read model, assert state.*

12. **Role of ICollectionFixture in xUnit?**  \
    *Creates a single fixture instance shared across test classes in the collection; disposes once after all tests. Combined with CollectionDefinition, enforces sequential execution for integration tests.*

### Advanced (6)

13. **Design a test for RetryBehavior that retries 3 times and then throws.**  \
    *Mock next() to throw TimeoutException each time. Counter tracks invocations. Assert 3 calls to next() then exception propagates. Also test non-transient exceptions do NOT retry (1 call only).*

14. **How to test transaction rollback on domain invariant violation?**  \
    *Integration: seed valid aggregate, send command that triggers domain exception (e.g., cancel shipped order), assert no state changed via direct SQL.* \
    *Unit: mock repo to throw on SaveChangesAsync, verify handler throws without side effects.*

15. **Strategy for testing eventual consistency — asserting projection caught up?**  \
    *Insert a marker event after the command event. Projection processes events in order; when it processes the marker, signal completion. Test polls with timeout. For synchronous testing, invoke projection inline (E2E pattern).*

16. **Test a query handler with authorization checks?**  \
    *Seed read model with order belonging to customer C. Mock auth to return true for user A, false for user B. Assert A gets DTO, B gets ForbiddenException. Integration: real DB + stubbed auth.*

17. **Test a saga orchestrating multi-step workflow?**  \
    *Unit: Feed OrderPlaced → InventoryReserved → PaymentSucceeded in sequence, verify correct command sent at each step via mocked command bus.* \
    *Integration: real saga repo, in-memory command bus, Testcontainers DB — feed first event, simulate command execution, verify saga completes.*

18. **Test read model versioning ignoring stale out-of-order events?**  \
    *Read model stores lastEventTimestamp. Projection skips events older than stored version. Test: send T2, then T1 (T1 < T2), assert T1 ignored. Send T3 > T2, assert T3 applied. Verify via both mock (call count) and integration (final state).*
