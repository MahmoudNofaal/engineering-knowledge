Event-sourced aggregates are tested differently than state-based aggregates because there is no mutable state to assert against. Instead, you test **command → event** mappings: given a sequence of past events (the aggregate history), when a command is executed, assert the expected new events are produced (or an error is thrown).

### Test Structure

Every test follows the "Given-When-Then" pattern:

- **Given** — a list of past events that form the aggregate's current state
- **When** — a command is executed against the aggregate
- **Then** — assert the exact events emitted, or the exact rejection/error

```csharp
// Example: testing a bank account aggregate
[Fact]
public void Withdraw_reduces_balance()
{
    var past = new Event[] {
        new AccountOpened("ACC-001", 1000m),
    };

    var command = new Withdraw("ACC-001", 300m);

    var expected = new Event[] {
        new Withdrawn("ACC-001", 300m, remaining: 700m),
    };

    var result = Aggregate.Execute<Account>(past, command);

    Assert.Equal(expected, result.Events);
}
```

### What to Test

- **Happy path** — Command produces the expected event(s)
- **Business rule violations** — Command is rejected with the correct error (overdraft, duplicate, stale version)
- **Idempotency** — Replaying the same event twice does not double-apply
- **Snapshot loading** — Loading from a snapshot + remaining events produces the same state as full replay
- **Concurrency** — Optimistic concurrency check rejects stale commands (wrong expected version)
- **Edge cases** — Empty stream, single event stream, boundary values (zero, negative, max)

### Anti-Patterns

- **Testing internal state** — Don't test private fields or `When`/`Apply` methods directly. Only test via the public command API.
- **Testing event order arbitrarily** — Events have a defined order; assert the exact sequence.
- **Over-mocking** — Aggregate tests should not mock event store, bus, or projections. Those are integration tests.

### Given-When-Then Libraries

Dedicated frameworks (like `AggregateTesting` or `EventStore.Client` test fixtures) provide fluent APIs:

```csharp
Given(pastEvents)
.When(new MyCommand())
.Then(expectedEvents)
.Assert();
```

For .NET, a simple test base class handles the plumbing: replay events, execute command, collect new events, and compare.
