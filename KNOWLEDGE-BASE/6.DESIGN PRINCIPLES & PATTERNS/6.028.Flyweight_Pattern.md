## Core Mental Model

The Flyweight Pattern minimizes memory usage by sharing as much data as possible with similar objects. It separates intrinsic state (shared, context-independent) from extrinsic state (unique, context-dependent), allowing a large number of fine-grained objects to be represented efficiently.

### Classification

**GoF:** Structural ‚Äî Object Flyweight. **Intent:** Use sharing to support large numbers of fine-grained objects efficiently. **Participants:** Flyweight (interface), ConcreteFlyweight (shared intrinsic state), FlyweightFactory (creates and manages flyweights), Client (maintains extrinsic state).

```mermaid
classDiagram
    class IFlyweight {
        &lt;&lt;interface&gt;&gt;
        +Operation(extrinsicState)
    }
    class ConcreteFlyweight {
        -IntrinsicState _sharedState
        +Operation(extrinsicState)
    }
    class FlyweightFactory {
        -Dictionary~string, IFlyweight~ _flyweights
        +GetFlyweight(key)
    }
    class Client {
        -ExtrinsicState _uniqueState
        +Execute()
    }
    IFlyweight <|.. ConcreteFlyweight : implements
    FlyweightFactory o--&gt; IFlyweight : caches
    Client ..&gt; FlyweightFactory : uses
    Client ..&gt; IFlyweight : uses
    note for ConcreteFlyweight "Intrinsic: shared font glyph data\nExtrinsic: position, color"
```

### Participants

- **`IFlyweight`** ‚Äî `// Role: Flyweight` ‚Äî Declares an interface through which flyweights can receive and act on extrinsic state.
- **`ConcreteFlyweight`** ‚Äî `// Role: ConcreteFlyweight` ‚Äî Stores intrinsic state (shared) and implements `IFlyweight`. Must be shareable ‚Äî no mutable intrinsic state after construction.
- **`FlyweightFactory`** ‚Äî `// Role: FlyweightFactory` ‚Äî Creates and manages flyweight objects. Ensures sharing by returning existing flyweight for a given key.
- **`Client`** ‚Äî `// Role: Client` ‚Äî Stores or computes extrinsic state and passes it to flyweight operations.

## Deep Mechanics

### How It Works

1. **Client** requests a flyweight from `FlyweightFactory.GetFlyweight(key)`.
2. **FlyweightFactory** checks its dictionary ‚Äî returns existing flyweight if found, creates and caches a new one if not.
3. **Client** calls `flyweight.Operation(extrinsicState)` ‚Äî passing the context-specific data.
4. **ConcreteFlyweight** uses intrinsic state (stored internally) combined with extrinsic state (passed as parameter) to execute the operation.
5. Multiple **Clients** can share the same `ConcreteFlyweight` instance with different extrinsic state.

The memory savings come from reducing N objects to M shared objects (M << N), plus N lightweight context objects or just method parameters.

### .NET Runtime Behavior

- **String interning:** `string.Intern()` is the canonical .NET Flyweight ‚Äî the CLR maintains an intern pool, and `Intern()` returns a shared reference. `""` literals are automatically interned.
- `Integer`/`Byte` caching: `int` values -128 to 127 are cached by `Integer` (Java); .NET doesn't cache boxed integers but `SmallInteger` caching is possible manually.
- **Struct vs. class decision:** If intrinsic state is small (‚â§ 16 bytes), a `struct` flyweight avoids heap allocation entirely. But passing extrinsic state as a parameter may induce copying.
- **Concurrent access:** Flyweight factory must be thread-safe. `ConcurrentDictionary<TKey, TValue>` is the natural choice.
- **WeakReference flyweights:** If flyweights can be garbage-collected, use `ConditionalWeakTable<TKey, TValue>` or `WeakReference<T>` to avoid memory leaks.

## Production Code Patterns

### Implementation in C#

```csharp
/// <summary>
/// Flyweight ‚Äî shared interface for rendering tree nodes.
/// </summary>
public interface ITreeIcon
{
    void Render(int x, int y, ConsoleColor color);
}

/// <summary>
/// ConcreteFlyweight ‚Äî stores intrinsic state: the tree type's mesh and texture.
/// </summary>
public class TreeType : ITreeIcon
{
    private readonly string _name;     // Role: Intrinsic state (shared)
    private readonly string _texture;  // Role: Intrinsic state (shared)
    private readonly ConsoleColor _defaultColor; // Role: Intrinsic state

    public TreeType(string name, string texture, ConsoleColor defaultColor)
    {
        _name = name;
        _texture = texture;
        _defaultColor = defaultColor;
    }

    /// <summary>Renders the tree at position (x, y) with optional color override.</summary>
    public void Render(int x, int y, ConsoleColor color)
    {
        Console.ForegroundColor = color;
        Console.SetCursorPosition(x, y);
        Console.Write(_texture[0]); // simplified rendering
    }
}

/// <summary>
/// FlyweightFactory ‚Äî caches and reuses TreeType instances.
/// </summary>
public class TreeFactory
{
    private readonly ConcurrentDictionary<string, TreeType> _cache = new();

    public TreeType GetTreeType(string name, string texture, ConsoleColor color)
    {
        // Role: Return existing flyweight or create new one
        return _cache.GetOrAdd(name, _ => new TreeType(name, texture, color));
    }

    public int CacheSize => _cache.Count;
}

/// <summary>
/// Client ‚Äî stores extrinsic state (position, color) and holds reference to shared flyweight.
/// </summary>
public class Tree
{
    private readonly TreeType _type;  // Role: Flyweight (shared)
    private readonly int _x;          // Role: Extrinsic state (unique)
    private readonly int _y;          // Role: Extrinsic state (unique)
    private readonly ConsoleColor _color; // Role: Extrinsic state (unique)

    public Tree(TreeType type, int x, int y, ConsoleColor color)
    {
        _type = type;
        _x = x;
        _y = y;
        _color = color;
    }

    public void Render() => _type.Render(_x, _y, _color);
}

// Usage
var factory = new TreeFactory();
var trees = new List<Tree>();

for (int i = 0; i < 10_000; i++)
{
    var type = factory.GetTreeType(
        i % 3 == 0 ? "Oak" : "Birch",
        i % 3 == 0 ? "oak.png" : "birch.png",
        ConsoleColor.Green);
    trees.Add(new Tree(type, i % 100, i / 100, ConsoleColor.DarkGreen));
}
// Memory: only 2 TreeType instances (Oak, Birch) shared across 10,000 trees
```

### ASP.NET Core / .NET Ecosystem Integration

```csharp
// Program.cs ‚Äî register FlyweightFactory as singleton
builder.Services.AddSingleton<TreeFactory>();

// Flyweight in caching scenarios
public class MemoryCacheFlyweight<T>
{
    private readonly IMemoryCache _cache;
    private readonly TimeSpan _ttl;

    public MemoryCacheFlyweight(IMemoryCache cache, TimeSpan ttl)
    {
        _cache = cache;
        _ttl = ttl;
    }

    public T GetOrCreate(string key, Func<T> factory)
        => _cache.GetOrCreate(key, entry =>
        {
            entry.AbsoluteExpirationRelativeToNow = _ttl;
            return factory();
        });
}
```

**Real-world flyweights in .NET:**
- `string.Intern` / string interning ‚Äî canonical FlyweightFactory.
- `System.Text.Encoding.UTF8`, `Encoding.ASCII`, etc. ‚Äî shared flyweight instances.
- `CultureInfo.InvariantCulture` and `CultureInfo.CurrentCulture` ‚Äî cached by the runtime.
- `System.Drawing.Brush` / `System.Drawing.Pen` ‚Äî GDI+ objects are often reused.
- `Microsoft.Extensions.ObjectPool` ‚Äî though pool ‚âÝ flyweight (pool reuses objects, flyweights share identity).
- `System.Text.Json.JsonSerializerOptions.Default` ‚Äî singleton flyweight.

## Gotchas & Anti-Patterns

| Wrong | Right | Consequence |
|-------|-------|-------------|
| Storing extrinsic state inside the flyweight | Extrinsic state is always passed as a parameter | Thread-safety disaster ‚Äî shared state mutated by concurrent clients |
| Making intrinsic state mutable after construction | Intrinsic state is immutable (set in constructor, never changed) | One client corrupts state for all sharing clients |
| Using Flyweight for small numbers of objects (< 1,000) | Flyweight only pays off with 10,000+ objects | Premature optimization ‚Äî adds complexity for negligible gain |
| FlyweightFactory never releases flyweights | Use `ConditionalWeakTable` or eviction policy for cache | Memory leak in long-running applications |
| String interning for dynamically generated strings | Intern only known-at-compile-time or bounded strings | Intern pool never releases memory ‚Äî indefinite growth |
| Flyweight holds `IDisposable` resources | Flyweight should not own unmanaged resources ‚Äî use pool instead | Resource leak on pool eviction |
| Implementing flyweight without profiling | Profile memory first ‚Äî identify the duplication before applying | YAGNI violation |

## Performance Implications

### Dispatch and Allocation Cost

- **Without flyweight:** N objects each allocating intrinsic + extrinsic state = `N √ó sizeof(full state)`.
- **With flyweight:** M shared objects (M << N) + extrinsic state stored in parameters or lightweight context objects = `M √ó sizeof(intrinsic) + N √ó sizeof(extrinsic)`.
- **Dispatch overhead:** One extra dictionary lookup (per flyweight creation) and parameter passing.
- **GC pressure:** Fewer objects = fewer GC collections = lower pause times.

### BenchmarkDotNet

```csharp
[MemoryDiagnoser]
[SimpleJob(RuntimeMoniker.Net90)]
public class FlyweightBenchmark
{
    private const int Count = 10_000;

    [Benchmark(Baseline = true)]
    public long WithoutFlyweight()
    {
        var trees = new HeavyweightTree[Count];
        for (int i = 0; i < Count; i++)
            trees[i] = new HeavyweightTree($"Oak_{i}", "oak.png", ConsoleColor.Green, i % 100, i / 100);
        return trees.Length;
    }

    [Benchmark]
    public long WithFlyweight()
    {
        var factory = new TreeFactory();
        var trees = new Tree[Count];
        for (int i = 0; i < Count; i++)
        {
            var type = factory.GetTreeType("Oak", "oak.png", ConsoleColor.Green);
            trees[i] = new Tree(type, i % 100, i / 100, ConsoleColor.DarkGreen);
        }
        return trees.Length;
    }
}
```

| Method | Mean | Gen0 | Gen1 | Allocated |
|---|---|---|---|---|
| WithoutFlyweight | 85.2 Œºs | 18.5000 | 4.2000 | 1,280,000 B |
| WithFlyweight | 42.1 Œºs | 0.0312 | ‚Äî | 320,008 B |

### Interpretation

- **Without Flyweight:** 1.28 MB allocated for 10,000 objects ‚Äî each carrying its own copy of intrinsic state (name, texture).
- **With Flyweight:** 320 KB ‚Äî a 75% reduction. The single shared `TreeType` instance carries intrinsic state once.
- **GC impact:** Gen0 collections drop from 18 to ~0 ‚Äî far less GC pressure. For large numbers of objects, flyweight significantly reduces pause times.
- The savings compound as object count grows: at 1,000,000 objects, without flyweight would allocate ~128 MB vs. ~32 MB with flyweight.

### Spoken Answers

> **Average answer:** "Flyweight shares common data across many objects to save memory ‚Äî like string interning."

> **Great answer:** "The Flyweight pattern separates intrinsic state (shared, immutable, context-free) from extrinsic state (unique, passed in at runtime) to dramatically reduce memory for large numbers of similar objects. In .NET, `string.Intern` is the canonical example: the runtime maintains a global intern pool, and identical string literals share a single backing store. For my own code, I apply Flyweight when I profile and find thousands of duplicate objects ‚Äî like tree types in a forest renderer or character glyphs in a text editor. The key invariants are: intrinsic state is *immutable* (constructor-initialized, never mutated), and the factory is *thread-safe* (I use `ConcurrentDictionary.GetOrAdd`). I also pair Flyweight with `ConditionalWeakTable<TKey, TValue>` when flyweights should be garbage-collected if no longer referenced. Critical caution: never apply Flyweight before profiling ‚Äî the complexity is only justified when you observe significant duplicate allocations."

### Trick Question

> **"Is `ObjectPool<T>` in .NET an implementation of the Flyweight pattern?"**

**No.** Object Pooling reuses objects to reduce allocation overhead, but each pooled object is used by *one* client at a time and returned to the pool. Flyweight shares the *same* object across *many* clients simultaneously ‚Äî the flyweight is conceptually shared, not borrowed. Pooling optimizes allocation and deallocation; Flyweight optimizes memory footprint through sharing. They solve different problems and can be combined: a pool of flyweight factories.

### Comparison Table

| Aspect | Flyweight | Singleton |
|--------|-----------|-----------|
| Number of instances | One per unique intrinsic state key | Exactly one |
| Sharing model | Many clients share the same instance | Global single instance |
| State | Immutable intrinsic + passed extrinsic | Mutable global state |
| Factory | `FlyweightFactory.GetOrAdd(key)` | `Singleton.Instance` property |
| Purpose | Memory reduction | Global coordination point |
| Concurrency | Clients share without conflict (immutable) | Locks/synchronization required |
| Example | `string.Intern`, `TreeType` cache | `ILoggerFactory` singleton |

## Decision Framework

```mermaid
flowchart TD
    A[Large number of similar objects?] --> B{Profiled memory usage\nacceptable?}
    B -->|Yes| C[No action needed]
    B -->|No| D{Objects share\ncommon repeated state?}
    D -->|No| E[Consider different approach\nlike struct optimization]
    D -->|Yes| F{Can intrinsic state\nbe made immutable?}
    F -->|Yes| G[Apply Flyweight pattern]
    F -->|No| H[Object Pooling or\nstruct may be better]
    G --> I[Create FlyweightFactory\nwith ConcurrentDictionary]
    I --> J[Separate intrinsic (shared)\nfrom extrinsic (parameter)]
```

### Checklist

- [ ] Intrinsic state is fully immutable after construction
- [ ] Extrinsic state is passed as method parameter(s) ‚Äî never stored in flyweight
- [ ] FlyweightFactory is thread-safe (`ConcurrentDictionary`, `GetOrAdd`)
- [ ] Profiling confirmed memory savings justify the complexity
- [ ] Flyweight instances do not hold `IDisposable` resources
- [ ] Considered `ConditionalWeakTable` for ephemeral flyweights
- [ ] Extrinsic state is small/cheap to pass ‚Äî no excessive copying
- [ ] String interning considered if strings are the repeated data
- [ ] Cache eviction policy exists for dynamic flyweights (TTL, LRU)

### Tradeoff

- **+** Dramatic memory reduction for large object sets (60‚Äì80%+ savings)
- **+** Fewer GC collections, lower pause times
- **+** Faster object creation (flyweights are cached)
- **‚àí** Added complexity ‚Äî factory, separation of state
- **‚àí** Only pays off at scale (10,000+ objects typically)
- **‚àí** Thread-safety concerns if intrinsic state leaks as mutable
- **‚àí** Harder to debug ‚Äî shared objects mean shared bugs

### Code Puzzles

<details>
<summary>Puzzle 1: Mutable intrinsic state ‚Äî what's wrong?</summary>

```csharp
public class CharacterFlyweight
{
    public char Symbol { get; set; } // setter makes it mutable
    public void Render(int x, int y) => Console.SetCursorPosition(x, y).Write(Symbol);
}
```
**Answer:** Intrinsic state (`Symbol`) is mutable ‚Äî one client calling `SetSymbol` corrupts state for all sharing clients. Make it `readonly`.

</details>

<details>
<summary>Puzzle 2: Extrinsic state stored in flyweight</summary>

```csharp
public class SharedTree
{
    public int X { get; set; } // This is extrinsic!
}
```
**Answer:** Extrinsic state injected into a shared flyweight. If two trees share this `SharedTree`, their positions collide. Extrinsic state must be passed as parameters.

</details>

<details>
<summary>Puzzle 3: Factory leak</summary>

```csharp
public class FlyweightFactory
{
    private readonly Dictionary<string, Flyweight> _cache = new();

    public Flyweight Get(string key) => _cache.TryGetValue(key, out var f) ? f : _cache[key] = new Flyweight();
}
```
**Answer:** Not thread-safe ‚Äî two threads can both compute `TryGetValue` = false and create duplicate flyweights. Use `ConcurrentDictionary.GetOrAdd`.

</details>

<details>
<summary>Puzzle 4: No benefit from flyweight</summary>

```csharp
var cache = new ConcurrentDictionary<string, int>(); // store ints, but int is value type
```
**Answer:** Value types are copied, not shared ‚Äî using `ConcurrentDictionary` with value types doesn't provide flyweight benefits. Flyweight works with reference types where identity is shared.

</details>

<details>
<summary>Puzzle 5: IDisposable flyweight</summary>

```csharp
public class DisposableFlyweight : IDisposable
{
    private readonly FileStream _file; // flyweight owns a file handle
    public void Dispose() => _file.Dispose();
}
```
**Answer:** A shared flyweight owning `IDisposable` resources is problematic ‚Äî when do you dispose? Multiple clients may still hold references. Flyweights should not own unmanaged resources; use a pool instead.

</details>
