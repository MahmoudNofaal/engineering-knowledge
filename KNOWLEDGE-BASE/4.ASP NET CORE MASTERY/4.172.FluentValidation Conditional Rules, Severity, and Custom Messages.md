**What this unlocks after:**
- Building highly complex, multi-step data entry APIs (like tax forms or onboarding wizards) where required fields change dynamically based on previous inputs.
- Delivering extremely clean API error contracts to mobile and frontend developers, eliminating noise and false-positive validation failures.

**Why this matters to a production engineer at scale:**
In enterprise systems, DTOs (Data Transfer Objects) are rarely flat, unconditional data structures. Consider a Payment API: if a customer pays by Credit Card, the `CardToken` is required. If they pay by Wire Transfer, the `CardToken` must be empty, but the `IbanNumber` is required.
If you use simplistic validation, you end up writing manual `if` statements scattered throughout your Controller Actions, bypassing the validation pipeline entirely. This destroys the consistency of your HTTP 400 responses.
FluentValidation's `.When()` and `.Unless()` modifiers allow you to encode complex, branched business logic directly into the declarative validation layer. This ensures that the validation pipeline accurately reflects the real-world domain, prevents unnecessary database calls (skipping async rules when the condition isn't met), and keeps your `errors` JSON dictionary completely free of irrelevant noise, providing a stellar Developer Experience (DX) for API consumers.

## PART 1 — The Core Mental Model

> **The Fundamental Rule**
> **FluentValidation's `.When()` and `.Unless()` methods evaluate predicates against the model *before* running specific rules, allowing dynamic validation paths. `.WithSeverity()` categorizes failures (Error vs Warning), though ASP.NET Core treats all severities as HTTP 400s by default unless customized. `.WithMessage()` and `.WithErrorCode()` dictate the exact string payloads that are pushed into the `ModelState` dictionary, directly shaping the final JSON response sent to the client.**

**The Plain-Language Analogy**
Imagine filling out a government tax form.
The form has a question: "Are you a foreign citizen?"
Below it is a box: "If yes, please enter your Visa Number."
`.When()` is the instruction that tells the reviewer: **"Only check the Visa Number box if they answered Yes to the first question."** If they answered No, the reviewer completely ignores the Visa Number box, even if the person scribbled gibberish inside it.
`Severity` is the color of the pen the reviewer uses. A Red pen (Error) means the form is rejected immediately. A Yellow pen (Warning) means the reviewer notes it, but might let it pass depending on the supervisor's policy.
`.WithMessage()` is the exact text the reviewer stamps on the rejected form so the citizen knows precisely how to fix it without guessing.

## PART 2 — Deep Mechanics

### 2.1 — `.When()` and `.Unless()` Mechanics
These extension methods accept a lambda expression representing a condition. By default, when chained onto a rule, they apply *only* to the specific rule they are attached to.

```csharp
RuleFor(x => x.Iban)
    .NotEmpty() // This rule is conditional
    .When(x => x.PaymentMethod == "WireTransfer");

RuleFor(x => x.CardToken)
    .NotEmpty() // This rule is conditional
    .Unless(x => x.PaymentMethod == "WireTransfer");
```

**Pipeline Execution:**
During `ValidateAsync`, the FluentValidation engine evaluates `x.PaymentMethod`. If it is `"CreditCard"`, the `Iban` rule is completely bypassed. It doesn't matter if `Iban` is null, empty, or a malicious SQL injection string—the rule simply does not execute. This prevents false-positive errors from polluting the HTTP 400 response.

### 2.2 — `DependentRules`
Sometimes you have a cascade of 5 or 6 rules that should only run if a condition is met. Repeating `.When()` on every single one is tedious and error-prone. The framework provides the top-level `When` block.

```csharp
When(x => x.IsInternationalShipping, () =>
{
    // ALL rules inside this block only execute if IsInternationalShipping is true
    RuleFor(x => x.CustomsDeclaration).NotEmpty();
    RuleFor(x => x.TariffCode).Length(6, 10);
    RuleFor(x => x.PassportNumber).NotEmpty();
});
```

### 2.3 — `ApplyConditionTo.CurrentValidator` vs `AllValidators`
A critical mechanic to understand when chaining rules:

```csharp
RuleFor(x => x.TaxId)
    .NotEmpty()
    .Length(9)
    .When(x => x.IsCorporateClient); // ONLY applies to .Length(9) by default!
```

In the default configuration, `.When()` only applies to the validator immediately preceding it. In the code above, `.NotEmpty()` will run for EVERYONE, but `.Length(9)` will only run for corporate clients.
To apply the condition to the entire chain for that property, you must specify the enum:

```csharp
RuleFor(x => x.TaxId)
    .NotEmpty()
    .Length(9)
    .When(x => x.IsCorporateClient, ApplyConditionTo.AllValidators);
```
*(Note: As of newer FluentValidation versions, `ApplyConditionTo` dictates this explicitly, but reading the docs for your specific major version is critical as defaults have shifted historically).*

### 2.4 — `.WithMessage()` and Placeholders
FluentValidation allows you to inject property values dynamically into the error message without relying on string interpolation, ensuring messages are clean and performant.

```csharp
RuleFor(x => x.Username)
    .MinimumLength(5)
    .WithMessage("The username '{PropertyValue}' is too short. It must be at least {MinLength} characters.");
```
If the user inputs "Bob", the HTTP 400 response will read: `"The username 'Bob' is too short. It must be at least 5 characters."`

### 2.5 — `.WithErrorCode()`
Enterprise APIs often require more than just human-readable strings. Machines need to parse errors. By using `.WithErrorCode()`, you attach a constant identifier to the failure.

```csharp
RuleFor(x => x.Amount)
    .GreaterThan(0)
    .WithErrorCode("ERR_PAYMENT_NEGATIVE_AMOUNT");
```
By default, ASP.NET Core `ModelState` discards Error Codes. However, you can intercept the `ValidationResult` in an `IAsyncActionFilter` or via `InvalidModelStateResponseFactory` (covered in 4.174) to project these strongly-typed codes directly into the JSON response for frontend consumption.

### 2.6 — `.WithSeverity()`
FluentValidation supports three severity levels: `Error`, `Warning`, and `Info`.

```csharp
RuleFor(x => x.Password)
    .Matches("[A-Z]")
    .WithSeverity(Severity.Warning)
    .WithMessage("Password should contain a capital letter.");
```

**ASP.NET Core Integration Mechanics:**
When ASP.NET Core's auto-validation runs, it iterates through all FluentValidation failures. **By default, ASP.NET Core treats ALL severities as fatal errors and returns an HTTP 400.** If you want the pipeline to ignore Warnings and let the request reach the Controller, you must explicitly configure the validation interceptor to filter out Warnings before they are dumped into `ModelState`.

## PART 3 — Production Code Patterns

### Pattern 1: Fintech — The Payment Method Polymorphism
APIs that handle multiple payment instruments are the classic use case for conditional validation.

```csharp
public class PaymentRequestValidator : AbstractValidator<PaymentRequestDto>
{
    public PaymentRequestValidator()
    {
        RuleFor(x => x.Method).IsInEnum();

        // Wire Transfer Rules
        When(x => x.Method == PaymentMethod.Wire, () =>
        {
            RuleFor(x => x.Iban)
                .NotEmpty().WithMessage("IBAN is required for wire transfers.")
                .Matches("^[A-Z]{2}[0-9]{2}[A-Z0-9]{11,30}$").WithMessage("Invalid IBAN format.");
                
            RuleFor(x => x.SwiftCode).NotEmpty();
        });

        // Credit Card Rules
        When(x => x.Method == PaymentMethod.CreditCard, () =>
        {
            RuleFor(x => x.CardToken).NotEmpty().WithMessage("Stripe Card Token missing.");
            RuleFor(x => x.Iban).Empty().WithMessage("IBAN must not be provided for card payments.");
        });
    }
}
```

### Pattern 2: Logistics — Hazardous Material Chain
Conditions can check complex domain states, not just simple enums.

```csharp
public class ShipmentValidator : AbstractValidator<ShipmentDto>
{
    public ShipmentValidator()
    {
        RuleFor(x => x.WeightKg).GreaterThan(0);

        When(x => x.Contents.Any(item => item.IsHazmat), () =>
        {
            RuleFor(x => x.HazmatClassId)
                .NotEmpty()
                .WithErrorCode("HAZMAT_CLASS_REQUIRED")
                .WithMessage("Shipment contains hazardous materials. HazmatClassId is required.");

            RuleFor(x => x.EmergencyContactPhone)
                .NotEmpty()
                .Matches(@"^\+[1-9]\d{1,14}$") // E.164 format
                .WithMessage("International emergency contact phone required for hazmat.");
        });
    }
}
```

### Pattern 3: E-Commerce — Dependent State Validations (Promos)
Sometimes a field is optional, but IF it is provided, it must undergo heavy validation (like a database check).

```csharp
public class CheckoutValidator : AbstractValidator<CheckoutDto>
{
    public CheckoutValidator(IPromoRepository promoRepo)
    {
        // Only run these rules if the user actually typed something into the PromoCode box
        When(x => !string.IsNullOrWhiteSpace(x.PromoCode), () =>
        {
            RuleFor(x => x.PromoCode)
                .MinimumLength(5)
                .MustAsync(async (code, ct) => await promoRepo.IsValidAsync(code, ct))
                .WithMessage("The promo code '{PropertyValue}' is invalid or expired.");
        });
    }
}
```

### Pattern 4: RuleSets for Create vs Update Operations
Often, you use the exact same DTO class for a `POST` (Create) and a `PUT` (Update) request. On Create, the `Id` must be empty. On Update, the `Id` must be provided. Instead of creating two separate DTOs and two separate Validators, you can use `RuleSets`.

```csharp
public class UserDtoValidator : AbstractValidator<UserDto>
{
    public UserDtoValidator()
    {
        // Rules that apply to EVERYTHING
        RuleFor(x => x.Email).NotEmpty().EmailAddress();

        // Rules that ONLY apply during CREATE
        RuleSet("Create", () => 
        {
            RuleFor(x => x.Id).Empty().WithMessage("ID must be empty when creating a user.");
        });

        // Rules that ONLY apply during UPDATE
        RuleSet("Update", () => 
        {
            RuleFor(x => x.Id).NotEmpty().WithMessage("ID is required for updates.");
        });
    }
}
```
*Usage in Minimal APIs:*
```csharp
app.MapPost("/users", async (UserDto dto, IValidator<UserDto> validator) => {
    // Manually specify which RuleSet to run!
    var result = await validator.ValidateAsync(dto, options => options.IncludeRuleSets("Create", "default"));
    // ...
});
```
*(Note: ASP.NET Core Auto-Validation evaluates the "default" ruleset unless customized via interceptors).*

### Pattern 5: Localization via `IStringLocalizer`
If your API serves multiple languages, hardcoding English strings in `.WithMessage()` is an anti-pattern. FluentValidation integrates cleanly with ASP.NET Core's localization.

```csharp
public class RegisterValidator : AbstractValidator<RegisterDto>
{
    // Inject the standard Microsoft IStringLocalizer
    public RegisterValidator(IStringLocalizer<SharedResource> localizer)
    {
        RuleFor(x => x.Password)
            .MinimumLength(8)
            // Use the localizer dynamically. FV will resolve it on every request.
            .WithMessage(x => localizer["PasswordTooShort"]); 
    }
}
```

## PART 4 — Gotchas & Anti-Patterns

### Gotcha 1: `.When()` Placement Scope Trap
Developers constantly attach `.When()` to the end of a long chain and assume it applies to the entire chain.

// ⚠️ WRONG CODE
```csharp
RuleFor(x => x.Age)
    .GreaterThan(18).WithMessage("Must be adult.")
    .LessThan(65).WithMessage("Must be working age.")
    .When(x => x.Employed); 
```

// HTTP consequence (wrong path):
// An unemployed 12-year-old submits the form. The API returns HTTP 400 "Must be adult."
// The developer is confused because `Employed` is false.
// Why? By default, `.When()` only applies to the `.LessThan(65)` rule immediately preceding it. The `.GreaterThan(18)` rule ran unconditionally.

// ✅ CORRECT CODE
```csharp
RuleFor(x => x.Age)
    .GreaterThan(18).WithMessage("Must be adult.")
    .LessThan(65).WithMessage("Must be working age.")
    .When(x => x.Employed, ApplyConditionTo.AllValidators); // Explicitly apply to the whole chain
```
// Or use the block syntax: `When(x => x.Employed, () => RuleFor(x => x.Age)...)`

### Gotcha 2: Null Reference Exceptions in `.When()` Predicates
The lambda expression inside `.When()` executes against the incoming DTO. If the DTO contains nested objects, checking them can throw a NRE.

// ⚠️ WRONG CODE
```csharp
RuleFor(x => x.ShippingAddress.ZipCode)
    .NotEmpty()
    // 💥 If ShippingAddress is null, x.ShippingAddress.Country throws a NullReferenceException!
    .When(x => x.ShippingAddress.Country == "US"); 
```

// HTTP consequence (wrong path):
// Unhandled `NullReferenceException` results in a raw HTTP 500 Internal Server Error instead of a clean HTTP 400.

// ✅ CORRECT CODE
```csharp
// Use the null-conditional operator
.When(x => x.ShippingAddress?.Country == "US");
```

### Gotcha 3: The Async Condition Trap
Sometimes a developer wants to run a rule only if a database condition is met.

// ⚠️ WRONG CODE
```csharp
RuleFor(x => x.DiscountCode)
    .NotEmpty()
    // 💥 COMPILER ERROR: .When() does not support async/await!
    .When(async x => await _db.IsPromoActiveAsync(x.PromoId)); 
```

// ✅ CORRECT CODE
// FluentValidation provides `WhenAsync()` specifically for this scenario.
```csharp
RuleFor(x => x.DiscountCode)
    .NotEmpty()
    .WhenAsync(async (x, ct) => await _db.IsPromoActiveAsync(x.PromoId, ct));
```
*(Warning: `WhenAsync` forces the entire validation pipeline for that property to become asynchronous. Use it sparingly).*

### Gotcha 4: Imperative Validation in Controllers
The biggest anti-pattern is abandoning FluentValidation the moment a condition becomes slightly complex, and resorting to `if` statements in the Controller.

// ⚠️ FATAL ANTI-PATTERN
```csharp
[HttpPost]
public IActionResult Submit(OrderDto dto)
{
    // ❌ Mixing validation concerns into HTTP handler
    if (dto.Type == "Digital" && dto.ShippingAddress != null) {
        return BadRequest("Digital orders cannot have shipping addresses.");
    }
    // ...
}
```

// Why this is fatal:
// The response shape for the manual `BadRequest` will not match the automated RFC 7807 `ValidationProblemDetails` shape generated by `ModelState`. Mobile clients parsing the JSON will crash because the `errors` dictionary is suddenly missing or formatted completely differently.
// Keep ALL conditional validation logic inside FluentValidation using `.When()` to guarantee a unified HTTP 400 contract.

## PART 5 — Performance Implications

### Request Pipeline Characteristics

| Validation Strategy | Execution Impact | Recommendation |
|---|---|---|
| Unconditional Rules | Runs Always | Baseline. |
| `.When()` (Sync) | Near Zero | Excellent. Evaluating a boolean property takes sub-nanoseconds. |
| `.When()` skipping `.MustAsync` | **Massive Win** | Saves 10ms+ per skipped rule by completely avoiding network/DB calls. |
| Deeply Nested `DependentRules` | O(Rules) | Negligible overhead. |
| `.WhenAsync()` | +Async State Machine | Adds ~0.05ms overhead + the cost of the I/O. Use only when the condition itself requires external data. |

**Performance Verdict:**
Using `.When()` extensively is actually a performance *optimization*, especially when guarding expensive async operations. By aggressively short-circuiting validation branches that do not apply to the current payload, you save significant CPU cycles and database round-trips.

## PART 6 — Interview Arsenal

### A. The Question Bank

**Question 1:** "If a user selects 'PayPal' as their payment method, we want to ensure the 'CreditCardToken' field is null or empty. We have a robust FluentValidation setup. How would you implement this?"
- **Average Answer:** "I would write an if statement in the controller and return a 400 if it's not empty."
- **Why That's Insufficient:** Violates the Single Responsibility Principle and corrupts the API error payload consistency.
- **Great Answer:** "I would keep the logic entirely within the FluentValidation layer to ensure the API returns a consistent `ValidationProblemDetails` response. I would use the `.When()` method on the validator. Specifically: `RuleFor(x => x.CardToken).Empty().WithMessage("Card tokens are not allowed for PayPal.").When(x => x.PaymentMethod == "PayPal");`. This declaratively enforces the domain rule during model binding."

**Question 2:** "You've chained `.NotEmpty().MinimumLength(5).When(x => x.IsActive)` on a property. A user submits an inactive account with a null property. The validation fails with 'Cannot be empty'. Why?"
- **Average Answer:** "Because FluentValidation evaluates rules in order."
- **Why That's Insufficient:** Doesn't address the specific scoping mechanics of the `When` operator.
- **Great Answer:** "Because by default, the `.When()` modifier only applies to the specific rule immediately preceding it—in this case, `.MinimumLength(5)`. The `.NotEmpty()` rule executes unconditionally for everyone. To fix this, you must either explicitly pass `ApplyConditionTo.AllValidators` to the `When` clause, or wrap the entire chain inside a top-level `When(..., () => { })` block."

**Question 3:** "I want to attach a specific internal error code to my validation failures, like 'ERR_1042', so the frontend can display specific translations without parsing the English message. How do you do this in ASP.NET Core?"
- **Average Answer:** "You put the code in the `.WithMessage()` string."
- **Why That's Insufficient:** Forces the frontend to use regex to parse the string.
- **Great Answer:** "You chain `.WithErrorCode("ERR_1042")` to the rule in FluentValidation. However, by default, ASP.NET Core's `ModelState` dictionary discards FluentValidation error codes, keeping only the error messages. To expose the error codes to the client, you must intercept the `ValidationResult`—either by writing a custom action filter, configuring the `InvalidModelStateResponseFactory`, or mapping the validation result manually in a Minimal API to inject the error codes into the RFC 7807 Problem Details `extensions` dictionary."

### B. The Trick Questions

**Trick Question:** "I set `.WithSeverity(Severity.Warning)` on a rule. When I submit invalid data, my ASP.NET Core API still returns an HTTP 400 Bad Request and blocks the request. Why?"
- **The Trap:** Assuming FluentValidation controls the HTTP pipeline directly.
- **The Correct Answer:** "FluentValidation is just a rules engine; it doesn't return HTTP responses. When integrated with ASP.NET Core auto-validation, the framework iterates over the `ValidationResult.Errors` collection. By default, the ASP.NET Core adapter treats *all* severities (Error, Warning, Info) as fatal invalid states and dumps them into `ModelState`, triggering the HTTP 400. To let Warnings pass through, you must configure a custom interceptor or action filter to strip out non-Error severities before they hit `ModelState`."

### C. Red Flags to Avoid
- 🚩 **"I don't use `When`, I just check conditions inside a `Custom` rule."** (Technically works, but destroys readability and makes the validator incredibly difficult to unit test).
- 🚩 **"I use `RuleSets` to validate different properties based on JSON payload values."** (`RuleSets` are chosen by the *caller*—the Controller/API—not dynamically evaluated based on the data. Use `When` for data-driven conditionals).

### B. Code Puzzles

**Puzzle 1: The Stubborn Condition**
```csharp
RuleFor(x => x.DriversLicense)
    .NotEmpty()
    .Matches("^[A-Z0-9]+$")
    .When(x => x.Age > 18);
```
*Scenario:* A 16-year-old submits the form with an empty `DriversLicense`. The API rejects the request with "DriversLicense must not be empty." Why?
<details>
<summary>Answer</summary>
The `.When(Age > 18)` modifier only applies to the `.Matches()` rule immediately preceding it. The `.NotEmpty()` rule executed unconditionally for the 16-year-old.
*Fix:* Change it to `.When(x => x.Age > 18, ApplyConditionTo.AllValidators);`
</details>

**Puzzle 2: The Async Crash**
```csharp
RuleFor(x => x.PromoCode)
    .NotEmpty()
    .When(async x => await _db.IsCartEligibleAsync(x.CartId));
```
*Scenario:* The compiler throws an error stating that a lambda expression cannot be converted to `Func<T, bool>`.
<details>
<summary>Answer</summary>
The standard `.When()` method only accepts synchronous predicates. You cannot use `async/await` inside it.
*Fix:* Change `.When` to `.WhenAsync`. E.g., `.WhenAsync(async (x, ct) => await _db.IsCartEligibleAsync(x.CartId, ct))`
</details>

**Puzzle 3: The Missing Error Code**
```csharp
RuleFor(x => x.Age)
    .GreaterThan(0)
    .WithErrorCode("INVALID_AGE");
```
*Scenario:* The validation fails. The client inspects the HTTP 400 Problem Details JSON. They see the error message, but "INVALID_AGE" is nowhere to be found.
<details>
<summary>Answer</summary>
ASP.NET Core's default `ModelState` dictionary structure maps Property Names to an array of string Messages. It fundamentally has no concept of FluentValidation's "Error Codes".
*Fix:* You must configure a custom `InvalidModelStateResponseFactory` in `Program.cs` that extracts the original `ValidationResult` and maps the `ErrorCode` property into the JSON output explicitly.
</details>
