# C# Native AOT, Trimming, and Publish-Time Constraints

> Native AOT compiles your entire .NET application to a self-contained native binary at publish time, eliminating the JIT and runtime dependency — but imposing hard constraints on reflection, dynamic code generation, and late-bound loading that force you to move runtime discovery to compile time.

## When To Use It

Reach for Native AOT when cold-start time, memory footprint, or binary size are primary constraints:

- **Serverless / FaaS** — Lambda, Azure Functions, Cloud Run: cold start is a cost and latency driver. Native AOT can cut it from seconds to milliseconds.
- **CLI tools** — Tight startup loop; users run them many times per session.
- **Containers** — Smaller images, lower RSS, faster scale-up.
- **Embedded / real-time** — No JIT pauses; deterministic execution time.
- **Security-sensitive deployments** — No JIT means a reduced attack surface; no IL in the binary to decompile.

Enable trimming without full Native AOT when you want a smaller binary but need the JIT (e.g., still use `Task.Run` or dynamic dispatch). The warnings are the same, so trimming compatibility is a prerequisite to Native AOT compatibility.

**Don't use Native AOT when:**

- The application uses unanalysable reflection (loading arbitrary plugins, scripting hosts)
- WPF or WinForms (unsupported in .NET 8/9)
- `Assembly.LoadFrom` or `Assembly.LoadFile` for runtime plugin loading
- Roslyn scripting, LINQ Expression compilation at runtime, or `Emit`-based code generation
- Third-party libraries that aren't trim-annotated (they will silently break)

## Core Concept

The .NET runtime normally JIT-compiles IL to native code at first use. This gives flexibility — types can be created from strings, generic types closed over unknown type parameters at runtime, and IL emitted on the fly. The price is startup time (JIT overhead), memory (JIT compiler resident), and binary size (IL must ship).

**Native AOT** inverts this: at publish time the entire call graph is traced, everything reachable gets compiled to machine code, and the result is a standalone native binary with zero runtime dependency. Nothing is JIT-compiled at startup because there's nothing to JIT.

**The fundamental constraint**: the publish-time compiler can only include what it can see. Reflection, `Activator.CreateInstance(Type)`, `Assembly.GetTypes()`, and `MethodInfo.Invoke()` operate on types and members discovered at runtime — the compiler has no way to know which ones, so it can't guarantee to keep them. The trimmer (which runs before AOT compilation) removes any code that isn't reachable from the known entry points, and anything accessed only via reflection is silently deleted.

**The solution is compile-time code generation.** Source generators run during `dotnet build` or `dotnet publish` with full access to the compilation's type model. They emit the specific code that would otherwise require reflection, making it a direct call the AOT compiler can see and preserve.

Three annotation attributes bridge the gap when you can't eliminate reflection entirely:

- `[DynamicallyAccessedMembers]` — annotates a `Type` parameter to say "the code using this type needs these specific member kinds". The trimmer uses this to preserve the right members.
- `[RequiresDynamicCode]` — marks an API as needing runtime code generation (JIT). Calling it in Native AOT produces a warning/error.
- `[RequiresUnreferencedCode]` — marks an API as using reflection in a way the trimmer can't analyse. Calling it when trimming produces a warning.

## Version History

|.NET Version|What changed|
|---|---|
|.NET 5|ReadyToRun (R2R) — pre-JIT to native, JIT still present as fallback|
|.NET 6|`PublishTrimmed` improvements; `ILLink` warnings for reflection; experimental Native AOT console apps|
|.NET 7|Native AOT reaches production quality for console apps and gRPC services; `[DynamicallyAccessedMembers]` fully enforced|
|.NET 8|ASP.NET Core minimal APIs support Native AOT; `[JsonSerializable]` source gen mature; `UnsafeAccessor` as reflection escape hatch|
|.NET 9|Expanded AOT compatibility across more BCL types; `[GeneratedComInterface]` for COM interop; improved diagnostics|

_ReadyToRun (R2R) is not Native AOT — R2R ships pre-compiled native code alongside IL and the JIT. Native AOT ships only native code; the JIT, CLR runtime, and IL are absent from the output._

## Performance

|Metric|JIT-compiled app|Native AOT app|
|---|---|---|
|Cold start (simple API)|200–800 ms|5–30 ms|
|Memory (RSS at idle)|40–80 MB|5–20 MB|
|Publish binary size|Depends on trim|Much smaller (typical 5–15 MB for an API)|
|Steady-state throughput|JIT optimises hot paths|Slightly lower — no profile-guided re-optimisation|
|Publish time|Seconds|30–120 s (full AOT compilation)|
|Max throughput|Higher (PGO)|Slightly lower without PGO|

**Allocation behaviour:** No change to runtime allocation patterns — GC still runs. The difference is startup: no JIT means no warm-up allocations for code compilation. Steady-state GC behaviour is identical.

**Benchmark notes:** Native AOT wins dramatically on cold start and idle memory — those are its design goals. For long-running, high-throughput services that stay warm, JIT with Profile-Guided Optimisation (PGO, enabled by default in .NET 8+) can outperform Native AOT because the JIT re-optimises based on actual runtime behaviour. Measure your specific workload rather than assuming AOT is universally faster.

## The Code

**Enabling Native AOT and trimming in the project file**

```xml
<!-- .csproj -->
<PropertyGroup>
  <!-- Native AOT: publish as native binary -->
  <PublishAot>true</PublishAot>

  <!-- Trimming only (smaller binary, JIT still present) -->
  <!-- <PublishTrimmed>true</PublishTrimmed> -->

  <!-- Trim warnings treated as errors during development -->
  <TreatTrimmerWarningsAsErrors>true</TreatTrimmerWarningsAsErrors>

  <!-- Invariant globalization: smaller binary, no ICU data -->
  <InvariantGlobalization>true</InvariantGlobalization>
</PropertyGroup>
```

```bash
# Publish a self-contained Native AOT binary for Linux x64
dotnet publish -r linux-x64 -c Release

# Output: bin/Release/net8.0/linux-x64/publish/MyApp (single native binary)
```

**`[DynamicallyAccessedMembers]` — tell the trimmer what to preserve**

```csharp
using System.Diagnostics.CodeAnalysis;

// PROBLEM: trimmer can't see which members 'type' needs — may delete them
public object CreateInstance(Type type)
    => Activator.CreateInstance(type)!; // trim warning ILLink2067

// SOLUTION: annotate the parameter — trimmer preserves the specified member kinds
public object CreateInstance(
    [DynamicallyAccessedMembers(DynamicallyAccessedMemberTypes.PublicParameterlessConstructor)]
    Type type)
    => Activator.CreateInstance(type)!; // no warning — trimmer knows to keep the ctor

// Common DynamicallyAccessedMemberTypes values:
//   PublicParameterlessConstructor — for Activator.CreateInstance(Type)
//   PublicConstructors             — for DI containers resolving by type
//   PublicMethods                  — for method discovery
//   PublicProperties               — for serialisers, mappers
//   All                            — keep everything (last resort)

// Propagate the annotation through a generic factory
public T Create<
    [DynamicallyAccessedMembers(DynamicallyAccessedMemberTypes.PublicParameterlessConstructor)]
    T>()
    where T : new()
    => new T(); // trimmer follows the annotation through the generic parameter
```

**`[RequiresDynamicCode]` and `[RequiresUnreferencedCode]` — marking unsafe APIs**

```csharp
// Mark APIs that can't work in Native AOT
[RequiresDynamicCode("This method uses runtime code generation via Emit. Not compatible with Native AOT.")]
public static Delegate BuildLambda(Type delegateType, Expression body)
{
    // Internally calls Expression.Compile() which emits IL at runtime
    return Expression.Lambda(delegateType, body).Compile();
}

// Mark APIs that use reflection in ways the trimmer can't analyse
[RequiresUnreferencedCode("This method reflects over all types in the assembly. Mark types with [PreserveAttribute] or use a source generator.")]
public static IEnumerable<Type> FindImplementors<T>()
{
    return Assembly.GetExecutingAssembly()
        .GetTypes()
        .Where(t => typeof(T).IsAssignableFrom(t) && !t.IsAbstract);
}

// The compiler will warn any caller of these methods when trimming/AOT is enabled,
// allowing the warning to propagate up the call chain to the real decision point.
```

**System.Text.Json source generation — the canonical AOT serialisation pattern**

```csharp
using System.Text.Json.Serialization;

// Step 1: Define your model types
public record OrderDto(int Id, string CustomerName, decimal Total, OrderStatus Status);
public enum OrderStatus { Pending, Processing, Shipped, Delivered }

// Step 2: Declare a JsonSerializerContext — source generator fills in the rest
[JsonSerializable(typeof(OrderDto))]
[JsonSerializable(typeof(List<OrderDto>))]
[JsonSerializable(typeof(OrderStatus))]
[JsonSourceGenerationOptions(
    PropertyNamingPolicy = JsonKnownNamingPolicy.CamelCase,
    WriteIndented = false)]
public partial class AppJsonContext : JsonSerializerContext { }

// Step 3: Use the context everywhere instead of the reflection-based overloads
// BAD — uses reflection, breaks under Native AOT:
// string json = JsonSerializer.Serialize(order);
// OrderDto? dto = JsonSerializer.Deserialize<OrderDto>(json);

// GOOD — uses generated code, fully AOT-compatible:
string json   = JsonSerializer.Serialize(order, AppJsonContext.Default.OrderDto);
OrderDto? dto = JsonSerializer.Deserialize(json,  AppJsonContext.Default.OrderDto);

// In ASP.NET Core minimal APIs (.NET 8+):
app.MapGet("/orders/{id}", (int id) =>
    Results.Ok(new OrderDto(id, "Alice", 99.99m, OrderStatus.Shipped)));

// Register the context so the minimal API pipeline uses it
builder.Services.ConfigureHttpJsonOptions(opts =>
    opts.SerializerOptions.TypeInfoResolverChain.Insert(0, AppJsonContext.Default));
```

**`[GeneratedRegex]` — compile-time regex (C# 7 / .NET 7+)**

```csharp
using System.Text.RegularExpressions;

public partial class OrderValidator
{
    // BAD — compiled at runtime, doesn't work in Native AOT without annotation:
    private static readonly Regex EmailRegex = new(@"^[^@\s]+@[^@\s]+\.[^@\s]+$",
        RegexOptions.Compiled | RegexOptions.IgnoreCase);

    // GOOD — compiled at build time, fully AOT-compatible, faster startup:
    [GeneratedRegex(@"^[^@\s]+@[^@\s]+\.[^@\s]+$", RegexOptions.IgnoreCase)]
    private static partial Regex EmailRegexGenerated();

    [GeneratedRegex(@"^\d{4}-\d{2}-\d{2}$")]
    private static partial Regex DateFormatRegex();

    public bool IsValidEmail(string email) => EmailRegexGenerated().IsMatch(email);
    public bool IsValidDate(string date)   => DateFormatRegex().IsMatch(date);
}
```

**Diagnosing trim warnings — what to do with each category**

```csharp
// CATEGORY 1: You own the code — annotate it
// Warning: Calling 'Activator.CreateInstance(Type)' on a type from a method
//          annotated with 'RequiresUnreferencedCode'
// Fix: Add [DynamicallyAccessedMembers] to the Type parameter (shown above)

// CATEGORY 2: Your code calls a library API you don't own
// Warning: 'MyThirdPartyLib.Configure(Type)' is annotated with RequiresUnreferencedCode
// Options:
//   a) Find if the library has an AOT-safe overload (many libs added them in .NET 8)
//   b) Switch to a source-generator-based alternative
//   c) Suppress with explanation if the usage is safe (last resort)
#pragma warning disable IL2026 // RequiresUnreferencedCode
    MyThirdPartyLib.Configure(typeof(MyConfig)); // we verified this only reads public props
#pragma warning restore IL2026

// CATEGORY 3: Dynamic plugin loading — fundamentally incompatible with Native AOT
// Assembly.LoadFrom("plugin.dll");  // compile error in Native AOT
// Solution: build plugins into the binary, use DI + compile-time registration

// CATEGORY 4: Generic type closed over unknown T at runtime
// typeof(List<>).MakeGenericType(runtimeType);  // works trimmed, fails Native AOT
// Fix: enumerate all expected type arguments and generate specific closed types via source gen
```

## Real World Example

An ASP.NET Core minimal API migrated to Native AOT. The before/after shows the three changes needed: project file, JSON serialisation, and eliminating reflection-based service discovery.

```csharp
// Program.cs — Native AOT compatible minimal API

// 1. The project file has <PublishAot>true</PublishAot>
// 2. All models registered with source-generated JsonSerializerContext
// 3. Services registered explicitly — no assembly scanning

var builder = WebApplication.CreateSlimBuilder(args); // Slim = AOT-optimised host

// Register JSON source context — replaces reflection-based serialisation
builder.Services.ConfigureHttpJsonOptions(opts =>
    opts.SerializerOptions.TypeInfoResolverChain.Insert(0, AppJsonContext.Default));

// Explicit DI registration — NO assembly scanning (FindImplementors pattern removed)
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();
builder.Services.AddScoped<IOrderService, OrderService>();
builder.Services.AddScoped<OrderValidator>();
builder.Services.AddSingleton<ILogger>(NullLogger.Instance);

var app = builder.Build();

app.MapGet("/orders", async (IOrderService svc, CancellationToken ct) =>
    Results.Ok(await svc.GetAllAsync(ct)));

app.MapGet("/orders/{id:int}", async (int id, IOrderService svc, CancellationToken ct) =>
{
    var order = await svc.GetAsync(id, ct);
    return order is null ? Results.NotFound() : Results.Ok(order);
});

app.MapPost("/orders", async (OrderRequest request, IOrderService svc, CancellationToken ct) =>
{
    var order = await svc.CreateAsync(request, ct);
    return Results.Created($"/orders/{order.Id}", order);
});

app.Run();

// Models
public record OrderDto(int Id, string CustomerName, decimal Total, string Status);
public record OrderRequest(string CustomerName, decimal Total);
public record ApiError(string Code, string Message);

// Source-generated JSON context — ALL types used in HTTP handlers must be registered here
[JsonSerializable(typeof(OrderDto))]
[JsonSerializable(typeof(List<OrderDto>))]
[JsonSerializable(typeof(OrderRequest))]
[JsonSerializable(typeof(ApiError))]
[JsonSourceGenerationOptions(PropertyNamingPolicy = JsonKnownNamingPolicy.CamelCase)]
public partial class AppJsonContext : JsonSerializerContext { }
```

```csharp
// OrderService.cs — the service itself needs no AOT-specific changes
// because it doesn't use reflection. The constraint is at the boundaries
// (serialisation, DI registration, HTTP request/response handling).

public class OrderService : IOrderService
{
    private readonly IOrderRepository _repo;

    public OrderService(IOrderRepository repo) => _repo = repo;

    public async Task<List<OrderDto>> GetAllAsync(CancellationToken ct)
    {
        var orders = await _repo.GetAllAsync(ct);
        return orders.Select(o => new OrderDto(o.Id, o.CustomerName, o.Total, o.Status.ToString())).ToList();
    }

    public async Task<OrderDto?> GetAsync(int id, CancellationToken ct)
    {
        var order = await _repo.FindAsync(id, ct);
        return order is null ? null : new OrderDto(order.Id, order.CustomerName, order.Total, order.Status.ToString());
    }

    public async Task<OrderDto> CreateAsync(OrderRequest req, CancellationToken ct)
    {
        var order = await _repo.CreateAsync(req.CustomerName, req.Total, ct);
        return new OrderDto(order.Id, order.CustomerName, order.Total, order.Status.ToString());
    }
}
```

**What the migration actually required:**

```
BEFORE (reflection-based, not AOT-compatible)             AFTER (AOT-compatible)
─────────────────────────────────────────────────         ─────────────────────────────────────
WebApplication.CreateBuilder(args)              →         WebApplication.CreateSlimBuilder(args)
JsonSerializer.Serialize(obj)                   →         JsonSerializer.Serialize(obj, AppJsonContext.Default.OrderDto)
JsonSerializer.Deserialize<T>(json)             →         JsonSerializer.Deserialize(json, AppJsonContext.Default.OrderDto)
services.AddAssemblyScanning<IService>()        →         Explicit services.AddScoped<IService, Impl>()
new Regex(@"pattern", Compiled)                 →         [GeneratedRegex(@"pattern")] partial Regex
Assembly.GetTypes().Where(t => ...)             →         Compile-time list or source-generated registry
```

_The key insight: the application logic (`OrderService`, repositories, validators) needs zero changes. AOT constraints only appear at the application's integration points — HTTP serialisation, DI container registration, and any framework that uses reflection to discover types. Migrating to AOT is mostly about moving from runtime discovery to compile-time registration._

## Common Misconceptions

**"Native AOT means no garbage collector"** Native AOT still uses the .NET GC. The heap, GC generations, finalisation, `IDisposable`, and `Span<T>` all work identically. What's removed is the JIT compiler, the IL code, and the CLR runtime host — not memory management. The GC is compiled into the native binary directly.

**"Trim warnings are just warnings — the app will still work"** Trim warnings are the trimmer telling you it deleted code your application will try to access at runtime. The publish succeeds and the binary runs fine in your local test environment because the debug build doesn't trim. In the published Native AOT binary, that code is gone. Ignoring trim warnings and testing locally produces a false sense of correctness; the failure happens in production.

**"Adding `[DynamicallyAccessedMembers]` to everything fixes AOT compatibility"** `[DynamicallyAccessedMembers]` works for patterns where you receive a `Type` and access specific, predictable member kinds. It doesn't help for patterns that scan all types in an assembly, close generic types over arbitrary runtime types, or emit new IL. For those patterns, the correct answer is a source generator. `[DynamicallyAccessedMembers]` is a scalpel for preserving what reflection legitimately needs — it's not a general suppress-all-warnings attribute.

## Gotchas

- **`TreatTrimmerWarningsAsErrors` is your best friend during migration.** Set it in the project file during development to see all warnings at build time rather than discovering failures in a trimmed publish. The default is warnings-only, which is easy to miss in a busy build output.
    
- **`WebApplication.CreateBuilder` vs `CreateSlimBuilder` for ASP.NET Core.** `CreateBuilder` includes a large set of default services (Kestrel, authentication, routing, JSON, etc.) that come with reflection-based initialisation. `CreateSlimBuilder` provides the minimal set compatible with Native AOT. Switching is usually non-breaking but you may need to add back services you relied on implicitly.
    
- **Third-party libraries may not be trim-annotated.** A library that internally calls `Activator.CreateInstance(typeof(T))` without `[DynamicallyAccessedMembers]` will produce warnings (or silently break) when you publish with trimming. Check the library's compatibility notes. Many popular libraries added AOT support in .NET 8 cycles — Dapper, EF Core (limited), MassTransit, etc. Some require you to opt into source-generated paths explicitly.
    
- **`Expression.Compile()` does not work in Native AOT.** Compiling expression trees at runtime requires JIT code generation. This breaks AutoMapper (version < 13), some LINQ providers, and any custom mapper that uses `Expression.Lambda(...).Compile()`. Use pre-compiled delegates or source generators instead. AutoMapper 13+ and Mapster both have Native AOT paths.
    
- **`Assembly.GetTypes()` and `Assembly.GetExecutingAssembly()` break under trimming.** The trimmer removes unreachable types, so the set of types returned is unpredictable. Patterns like scanning for all `IValidator<T>` implementations at startup fail. Replace with explicit registration or a source generator that emits the registration code.
    
- **`PublishSingleFile` is not the same as Native AOT.** `PublishSingleFile=true` bundles the .NET runtime and all assemblies into a single file executable, but the JIT is still present and IL is still inside. It's a deployment convenience feature, not a performance or security change. Native AOT produces a true native binary with no IL.

## Interview Angle

**What they're really testing:** Whether you understand the fundamental constraint — static analysis can't see through reflection — and can explain how source generators solve it by moving runtime discovery to compile time.

**Common question forms:**

- "What is Native AOT and what are the trade-offs vs JIT?"
- "Why does reflection break in trimmed / Native AOT apps?"
- "How would you migrate a reflection-heavy service to be AOT-compatible?"
- "What is `[DynamicallyAccessedMembers]` and when do you use it?"
- "What is the difference between `PublishTrimmed`, `PublishSingleFile`, and `PublishAot`?"

**The depth signal:** A junior says "Native AOT compiles to native code and starts faster." A senior explains the static analysis problem: the AOT compiler and trimmer can only preserve code they can prove is reachable from the entry point. Reflection breaks this by accessing code whose identity is only known at runtime. The solution is source generators — they run during `dotnet build` with access to the full type model, emit the specific typed code that would have been discovered reflectively, and produce direct calls the AOT compiler can see. A senior also knows the performance nuance: steady-state throughput can be slightly lower than JIT with PGO, so AOT is the right choice for cold-start-sensitive workloads, not universally for all .NET apps.

**Follow-up questions to expect:**

- "Can you use Entity Framework Core with Native AOT?"
- "What does `WebApplication.CreateSlimBuilder` do differently from `CreateBuilder`?"
- "If a third-party library isn't AOT-compatible, what are your options?"
