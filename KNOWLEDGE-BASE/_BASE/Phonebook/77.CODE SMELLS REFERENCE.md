# CATEGORY 77 — CODE SMELLS REFERENCE

---

## 77.01 — Complete Code Smell Catalog

**Long Method** — > 20 lines, does too many things
Fix: Extract methods, use helper classes

**Large Class** — too many fields and methods
Fix: Split by responsibility (Single Responsibility Principle)

**Long Parameter List** — > 3-4 parameters
Fix: Introduce parameter object / request DTO

**Divergent Change** — one class changes for many different reasons
Fix: Split into focused classes (each changes for one reason)

**Shotgun Surgery** — one change requires edits in many places
Fix: Move related data/behavior to one place

**Feature Envy** — method accesses other object's data more than its own
Fix: Move method to the class whose data it uses

**Data Clumps** — groups of data that always appear together
Fix: Introduce a class or record to hold them
```csharp
// Smell: street, city, country, zip always appear together
void CreateUser(string name, string street, string city, string country, string zip) { }

// Fix:
void CreateUser(string name, Address address) { }
public record Address(string Street, string City, string Country, string Zip);
```

**Primitive Obsession** — using primitives where objects would be clearer
Fix: Value objects (covered in 51.01)

**Switch Statements / Long if-else** — type-based behavior
Fix: Polymorphism, Strategy pattern
```csharp
// Smell:
decimal CalculateDiscount(string customerType, decimal price)
{
    if (customerType == "VIP") return price * 0.2m;
    if (customerType == "Regular") return price * 0.05m;
    if (customerType == "New") return price * 0.1m;
    return 0;
}

// Fix: Strategy + polymorphism
public interface IDiscountStrategy { decimal Calculate(decimal price); }
public class VIPDiscount : IDiscountStrategy { public decimal Calculate(decimal p) => p * 0.2m; }
public class RegularDiscount : IDiscountStrategy { public decimal Calculate(decimal p) => p * 0.05m; }
```

**Parallel Inheritance Hierarchies** — every time you add a subclass, you add another in a parallel hierarchy
Fix: Remove inheritance, use composition

**Lazy Class** — class that doesn't do enough to justify its existence
Fix: Inline the class or merge it

**Speculative Generality** — code written "just in case" for features that don't exist yet
Fix: Delete it. YAGNI (You Aren't Gonna Need It)

**Temporary Field** — instance field only used in some situations
Fix: Extract a method that takes the fields as parameters

**Message Chains** — `a.b().c().d().e()` — Law of Demeter violation
Fix: Intermediate method, or restructure

**Middle Man** — class that only delegates
Fix: Remove it, call the delegated class directly

**Inappropriate Intimacy** — two classes that depend too heavily on each other's internals
Fix: Move methods, use interfaces, reduce coupling

**Alternative Classes with Different Interfaces** — two classes that do the same thing with different names
Fix: Extract common interface, possibly merge

**Incomplete Library Class** — library doesn't have what you need
Fix: Extension methods (in C# — cleanest solution)

**Data Class** — class with only fields and getters/setters, no behavior
Fix: Move behavior that operates on this data into the class

**Refused Bequest** — subclass doesn't use all inherited methods
Fix: Replace inheritance with composition

**Comments** — as a code smell (when comments explain WHAT instead of WHY)
Fix: Rename the thing so the comment becomes unnecessary
```csharp
// SMELL: Comment explains what code does
// Check if user is older than 18
if (user.Age >= 18) { ... }

// FIX: Code explains itself
if (user.IsAdult()) { ... }

// GOOD comment: explains WHY (non-obvious reason)
// Must process in reverse order because the SDK modifies the collection during iteration
for (int i = items.Count - 1; i >= 0; i--) { ... }
```
