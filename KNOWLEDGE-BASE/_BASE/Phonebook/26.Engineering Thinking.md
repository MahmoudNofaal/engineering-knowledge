## 26.01 — Code Review Excellence

**As a reviewer**
```markdown
# Good review comments

## Specific and actionable
❌ "This is wrong."
✓ "This method calls GetUser() inside the loop. For 100 orders, that's 100 database 
   queries. Consider loading all users upfront or using Include() in the initial query."

## Distinguish blocking from non-blocking
[Required] The null check is missing — this will throw NullReferenceException when 
           user is not found.
[Suggestion] Consider using TryGetValue here instead of ContainsKey + indexer access 
             to avoid the double lookup.
[Nitpick] Variable name 'x' could be more descriptive — 'order' perhaps?

## Ask before assuming
"Is there a reason this is sorted descending? I expected ascending by date."
Rather than:
"This sort is wrong."
```

**As an author**
- The PR description explains WHAT and WHY (the code says HOW)
- Tests are included and passing
- Screenshots for UI changes
- Linked to relevant issue/ticket
- No unrelated changes mixed in

## 26.02 — Technical Writing

**Commit messages (Conventional Commits)**
```
feat(orders): add bulk order cancellation endpoint

Adds POST /api/orders/bulk-cancel that accepts a list of order IDs
and cancels all eligible (Pending/Confirmed) orders atomically.

Returns a summary with cancelled count and any orders that could
not be cancelled with the reason.

Closes #234

BREAKING CHANGE: Response format changed from {success: bool} 
to {cancelled: number, failed: [{orderId, reason}]}
```

Format: `type(scope): description`
Types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `build`, `ci`

**Pull request descriptions**
```markdown
## What
Implement caching layer for product catalog using Redis.

## Why
Product catalog is read 200x per second but changes < 5x per day.
Without caching, every request hits SQL Server unnecessarily.
Expected: 90% reduction in product catalog database load.

## How
- Added `CachingProductRepository` decorator wrapping `SqlProductRepository`
- Cache-aside pattern with 10-minute TTL
- Cache invalidated on product update/create via domain event

## Testing
- Unit tests for the caching decorator (mocked Redis)
- Integration test verifying cache is populated on first read
- Integration test verifying stale data is served during DB downtime

## Checklist
- [x] Tests added
- [x] No secrets committed
- [x] README updated if needed
```
