# CATEGORY 68 — DATABASE MIGRATIONS STRATEGY

---

## 68.01 — Zero-Downtime Deployment Strategy

**The constraint**
Multiple instances run during deployment. Old code and new code run simultaneously against the same database. Database migrations must work with both versions.

**Rules for safe migrations**

**ADDING:**
- ✅ Add nullable column → safe (old code ignores it, new code uses it)
- ✅ Add new table → safe
- ✅ Add non-clustered index → safe (online indexing)
- ✅ Add new stored procedure → safe

**CHANGING:**
- ⚠️ Make column NOT NULL → must backfill first (3-step migration)
- ⚠️ Rename column → 3-step (add new, migrate data, remove old)
- ⚠️ Change column type → may require temporary parallel columns
- ⚠️ Add foreign key → may fail if referential integrity violated

**REMOVING:**
- ❌ Drop column referenced by code → code crashes
- ❌ Drop table referenced by code → code crashes
- ✅ Drop column in step 2 after code no longer uses it

**Three-step rename pattern**
```
Version 1.0 → 1.1 (Migration A):
  ADD COLUMN NewName VARCHAR(100);
  
Deploy Version 1.1 (code writes to both columns, reads from OldName):
  UPDATE SET NewName = OldName WHERE NewName IS NULL;  -- Backfill
  
Version 1.1 → 1.2 (Migration B):
  Make NewName NOT NULL;
  
Deploy Version 1.2 (code writes to both columns, reads from NewName):

Version 1.2 → 1.3 (Migration C):
  DROP COLUMN OldName;
  
Deploy Version 1.3 (code writes/reads only NewName):
```

**Why three steps:**
- Between step 1 and step 3, old and new code can coexist
- Each step is small and individually safe
- Rollback at any step is safe

## 68.02 — EF Core Migration Best Practices

```csharp
// Check migration status in CI/CD — fail build if pending migrations
public class MigrationCheckTests : IClassFixture<ApiTestFixture>
{
    private readonly ApiTestFixture _fixture;
    
    [Fact]
    public async Task Database_ShouldHaveNoMigrationGap()
    {
        using var scope = _fixture.Services.CreateScope();
        var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
        
        var pending = await db.Database.GetPendingMigrationsAsync();
        pending.Should().BeEmpty("All migrations should be applied");
    }
}

// Generate SQL script for review (DBA-friendly)
// dotnet ef migrations script --idempotent --output migration.sql
// Idempotent: safe to run multiple times, checks if migration already applied

// Production strategy: apply migrations via init container in K8s
// api-init-container:
//   command: ["dotnet", "MyApp.dll", "--migrate-only"]

// Program.cs
if (args.Contains("--migrate-only"))
{
    using var scope = app.Services.CreateScope();
    var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
    await db.Database.MigrateAsync();
    Console.WriteLine("Migrations applied successfully");
    return;  // Exit
}
```
