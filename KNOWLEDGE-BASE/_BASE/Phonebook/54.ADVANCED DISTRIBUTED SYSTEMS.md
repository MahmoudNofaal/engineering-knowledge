## 54.01 — Consensus Algorithms

**The problem**
In a distributed system with multiple nodes, how do nodes agree on a single value when nodes can fail and messages can be delayed?

**Raft — the approachable consensus algorithm**
Used by: etcd (Kubernetes configuration store), CockroachDB, Consul

**Key concepts**
- **Leader election**: Nodes elect a leader via randomized timeouts
- **Log replication**: Leader receives client writes, replicates to followers
- **Commit**: Entry committed after majority acknowledgment
- **Terms**: Monotonically increasing epochs; each has at most one leader

**States each node can be in**
```
Follower (default): Receives log entries from leader
Candidate:          Requests votes to become leader
Leader:             Handles all client writes, sends heartbeats
```

**Leader election**
```
1. Follower receives no heartbeat within election timeout (150-300ms)
2. Follower becomes Candidate, increments term
3. Candidate votes for itself, requests votes from others
4. If majority responds: becomes Leader, sends immediate heartbeat
5. If another leader found (higher term): reverts to Follower
6. If no majority (split vote): wait for new election timeout
```

**Log replication**
```
Client → Leader: "SET x = 5"
Leader appends to own log (uncommitted)
Leader sends AppendEntries to all followers
Followers append to own logs, respond OK
Leader receives majority OK → commits
Leader responds to client: success
Leader sends committed status to followers on next heartbeat
```

**Practical implications**
- At most 1 leader per term — no split-brain
- All reads from leader guarantee linearizability
- Leader failure: election within ~300-600ms
- With 5 nodes, can tolerate 2 failures (floor(5/2) = 2)

## 54.02 — Distributed Tracing Theory

**Trace model**
```
Trace ID: abc123 (spans entire request across services)

Span 1: API → received request              [   0ms -  5ms]
  Span 2: API → called OrderService         [   2ms -  80ms]
    Span 3: OrderService → DB query          [   5ms -  45ms]
    Span 4: OrderService → Redis cache       [   46ms - 52ms]
    Span 5: OrderService → PaymentService    [   53ms -  78ms]
      Span 6: PaymentService → Stripe API    [   55ms -  76ms]
Span 7: API → sent response               [  80ms -  82ms]

Total latency: 82ms
Bottleneck: PaymentService → Stripe (21ms)
```

**Context propagation**
Trace context is propagated through HTTP headers:
- W3C TraceContext: `traceparent: 00-abc123-def456-01`
- B3 (Zipkin): `X-B3-TraceId`, `X-B3-SpanId`, `X-B3-Sampled`

```csharp
// OpenTelemetry propagates automatically through HttpClient
// For message queues, propagate manually:

// On publish
var activity = Activity.Current;
var properties = new Dictionary<string, object>
{
    ["traceparent"] = activity?.Id ?? string.Empty,
    ["tracestate"] = activity?.TraceStateString ?? string.Empty
};
await _bus.PublishAsync(message, properties);

// On consume
var parentId = headers["traceparent"];
var activityContext = ActivityContext.TryParse(parentId, null, out var ctx) ? ctx : default;
using var activity = Source.StartActivity("ProcessMessage", ActivityKind.Consumer, activityContext);
```

**Sampling strategies**
```
Head-based sampling: Decision made at trace start, propagated to all spans
  - Simple, consistent
  - May miss rare errors (they're sampled out)

Tail-based sampling: Decision made after trace completes
  - Keeps all error traces regardless of rate
  - Requires a collector to buffer traces before sampling
  - More complex but better coverage of errors

Adaptive sampling:
  - Low rate for fast/common paths (1%)
  - High rate for slow/error paths (100%)
```

## 54.03 — Event-Driven Architecture Patterns (Advanced)

**Event-Carried State Transfer**
Include enough data in the event that consumers don't need to call back.

```json
// Thin event (NOT recommended for most cases):
{
  "type": "OrderCreated",
  "orderId": 42
}
// Consumer must call back to get order details → coupling, latency, availability dependency

// Fat event (Event-Carried State Transfer):
{
  "type": "OrderCreated",
  "orderId": 42,
  "customerId": 7,
  "customerEmail": "alice@example.com",
  "total": 299.99,
  "items": [{"productId": 1, "name": "Widget", "qty": 2, "price": 149.99}],
  "createdAt": "2026-06-01T23:00:00Z"
}
// Consumer has everything it needs — no callback required
// Tradeoff: larger messages, duplicating some data
```

**Event versioning**
Events evolve. Old consumers must still process new events.

```csharp
// Additive changes (backward compatible):
// Add optional fields — old consumers ignore unknown fields
// Never remove required fields
// Never change field semantics

// Breaking changes → new event type or version
public record OrderCreatedV1(int OrderId, int CustomerId, decimal Total);
public record OrderCreatedV2(int OrderId, int CustomerId, decimal Total, string Currency);

// Consumer handles both:
switch (eventType)
{
    case "OrderCreated.V1":
        var v1 = Deserialize<OrderCreatedV1>(data);
        Handle(v1);
        break;
    case "OrderCreated.V2":
        var v2 = Deserialize<OrderCreatedV2>(data);
        Handle(v2);
        break;
}
```

**Choreography vs. Orchestration — detailed comparison**

| Aspect | Choreography | Orchestration |
|---|---|---|
| Control | Distributed — each service decides independently | Centralized — orchestrator directs |
| Coupling | Lower — services don't know each other | Higher — orchestrator knows all |
| Visibility | Hard to see the full workflow | Easy — workflow is explicit in orchestrator |
| Error handling | Complex — each service handles own compensation | Easier — orchestrator coordinates rollback |
| Testing | Hard — must simulate all services | Easier — can unit test orchestrator |
| Failure | Service down → workflow stalls silently | Orchestrator down → all workflows stop |
| Best for | Simple workflows, decoupled services | Complex multi-step transactions |

## 54.04 — Two-Phase Commit (2PC) and Why It's Avoided

**What it is**
A distributed transaction protocol ensuring atomicity across multiple databases/services.

**Phase 1 — Prepare**
```
Coordinator: "Prepare to commit transaction X"
Participant A: Locks resources, writes undo log → "Ready"
Participant B: Locks resources, writes undo log → "Ready"
```

**Phase 2 — Commit (or Abort)**
```
All ready → Coordinator: "Commit"
Any not ready → Coordinator: "Abort"
```

**Why 2PC is avoided in microservices**
1. **Coordinator failure during Phase 2**: Participants are in prepared state, resources locked. System hangs until coordinator recovers. No progress possible.
2. **Blocking**: All participants lock resources during the protocol. High latency.
3. **Availability**: A single participant failure aborts the entire transaction.
4. **CAP theorem**: 2PC maximizes consistency but sacrifices availability.

**Alternatives**
- **Saga pattern**: Choreography or orchestration with compensating transactions
- **Outbox pattern**: Local transaction + eventual consistency via messaging
- **Exactly-once processing**: Idempotent consumers + at-least-once delivery
