## 93.01 — Bounded Contexts in Practice

**What is a Bounded Context?**
A portion of your domain where a particular model applies. The same word may mean different things in different contexts.

```
"Customer" in different bounded contexts:
- Sales context:      Customer = person with billing info, credit limit, discount tier
- Support context:    Customer = person with tickets, satisfaction score, contact history
- Marketing context:  Customer = person with campaign response history, preferences
- Shipping context:   Customer = person with delivery addresses, signature requirements

Each context has its own "Customer" model — different fields, different logic.
Sharing one giant "Customer" table across all contexts creates coupling and confusion.
```

**Context mapping patterns**
```
Shared Kernel:
  Two contexts share a small subset of the domain model (e.g., UserId type)
  Requires coordination between teams to change shared code
  
Customer-Supplier:
  One context (supplier) produces output consumed by another (customer)
  Supplier: Orders context produces OrderShipped event
  Customer: Shipping context consumes it
  
Anti-Corruption Layer (ACL):
  A translation layer that prevents an external model from "corrupting" your domain model
  When integrating with legacy systems or external APIs
  
  External API returns:    { "cust_id": 42, "purch_amt": "99.99", "stat_cd": "P" }
  ACL translates to:       { CustomerId = 42, Total = 99.99m, Status = OrderStatus.Pending }
  
Open Host Service:
  Provide a published language that others can integrate with
  Your public REST API is an Open Host Service
  
Published Language:
  Shared domain language (think: FHIR for healthcare, EDI for retail)
```

**Strategic design vs. tactical design**
```
Strategic (big picture):
  Bounded contexts, context maps, team topology
  Answers: "What are the domains?" "Who owns what?"
  
Tactical (implementation):
  Aggregates, entities, value objects, repositories, domain events
  Answers: "How do we model this domain?"
  
Common mistake: Apply tactical DDD patterns without strategic thinking
→ You get a complex, over-engineered CRUD system
```

## 93.02 — Aggregate Design Rules

**Rule 1: Design aggregates around business invariants**
```csharp
// WRONG: Separate aggregates for Order and OrderItem
// Problem: Can't enforce "order total < $10,000" as an invariant
// because it requires reading all items

// RIGHT: Order is the aggregate root, Items are part of it
// Invariant enforced within one aggregate boundary
public class Order  // Aggregate Root
{
    private readonly List<OrderItem> _items = new();
    public IReadOnlyList<OrderItem> Items => _items.AsReadOnly();
    
    public void AddItem(Product product, int quantity)
    {
        var newTotal = _items.Sum(i => i.Subtotal) + product.Price * quantity;
        if (newTotal > 10_000)
            throw new DomainException("Order total cannot exceed $10,000");
        
        _items.Add(new OrderItem(product.Id, quantity, product.Price));
    }
}
```

**Rule 2: Reference other aggregates by ID only**
```csharp
// WRONG: Order holds reference to Customer aggregate
public class Order
{
    public Customer Customer { get; private set; } = null!;  // Direct reference
    // Problem: Loading Order loads Customer loads Customer.Orders → recursive!
}

// RIGHT: Reference by ID
public class Order
{
    public CustomerId CustomerId { get; private set; }  // Just the ID
    // Load Customer separately if needed
}
```

**Rule 3: Keep aggregates small**
```csharp
// WRONG: Giant aggregate that contains everything
public class Order
{
    public Customer Customer { get; } = null!;
    public Address ShippingAddress { get; } = null!;
    public List<OrderItem> Items { get; } = new();
    public Payment Payment { get; } = null!;
    public List<OrderStatusHistory> StatusHistory { get; } = new();
    public List<Note> Notes { get; } = new();
    public Shipment Shipment { get; } = null!;
    // 200+ lines of invariants...
}
// Problem: Loading the aggregate loads everything → slow, memory-intensive
// Concurrent modification: any change to ANY part → write conflict

// RIGHT: Multiple smaller aggregates
public class Order { /* Items, core order data */ }
public class Payment { /* Payment-specific data, state machine */ }
public class Shipment { /* Shipment tracking, addresses */ }
```

**Rule 4: Modify only one aggregate per transaction**
```csharp
// WRONG: Single transaction modifying two aggregates
public async Task ProcessPaymentAsync(int orderId, decimal amount, CancellationToken ct)
{
    var order = await _orders.GetAsync(orderId, ct);
    var account = await _accounts.GetAsync(order.CustomerId, ct);
    
    order.MarkPaymentReceived();     // Modify aggregate 1
    account.DeductFunds(amount);      // Modify aggregate 2
    
    await _db.SaveChangesAsync(ct);  // Saves both — tight coupling
}

// RIGHT: Single transaction, one aggregate. Other aggregate updated via event.
public async Task ProcessPaymentAsync(int orderId, decimal amount, CancellationToken ct)
{
    var order = await _orders.GetAsync(orderId, ct);
    order.MarkPaymentReceived();  // Raises PaymentReceivedEvent
    await _db.SaveChangesAsync(ct);  // Saves order AND publishes event
}

// Event handler (different transaction, eventually consistent)
public async Task Handle(PaymentReceivedEvent evt, CancellationToken ct)
{
    var account = await _accounts.GetAsync(evt.CustomerId, ct);
    account.DeductFunds(evt.Amount);
    await _db.SaveChangesAsync(ct);
}
```
