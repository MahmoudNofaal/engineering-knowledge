## 1. THE REQUEST PIPELINE & MIDDLEWARE

### What It Actually Is
The middleware pipeline is a linked chain of `RequestDelegate` functions — each one is `Func<HttpContext, Task>`. When a request arrives, it enters the chain head-first; each middleware can do work, call the *next* delegate, then do more work on the way back out. Think of it as an onion: you peel in on request, peel out on response. The key insight is that **order of registration is execution order**, which makes bugs in ordering silent and catastrophic.

### How It Works — Mechanically

```
Incoming HTTP Request
        │
        ▼
┌───────────────────┐
│  ExceptionHandler │◄── wraps everything, catches all thrown exceptions
└────────┬──────────┘
         │  next()
         ▼
┌───────────────────┐
│  HTTPS Redirect   │◄── 301/302 if HTTP, terminates if redirected
└────────┬──────────┘
         │  next()
         ▼
┌───────────────────┐
│   Static Files    │◄── short-circuits if file found (no next() call)
└────────┬──────────┘
         │  next()
         ▼
┌───────────────────┐
│     Routing       │◄── matches URL to endpoint, sets RouteData
└────────┬──────────┘
         │  next()
         ▼
┌───────────────────┐
│  Authentication   │◄── populates HttpContext.User (ClaimsPrincipal)
└────────┬──────────┘
         │  next()
         ▼
┌───────────────────┐
│  Authorization    │◄── checks policy against HttpContext.User
└────────┬──────────┘
         │  next()
         ▼
┌───────────────────┐
│    Endpoint       │◄── controller action or minimal API handler executes
└────────┬──────────┘
         │  returns
         ▼
   (unwind back up through each middleware's post-next code)
```

**Step-by-step execution for a single middleware:**
1. ASP.NET Core calls `middleware.InvokeAsync(context)` 
2. Middleware executes its *pre-next* logic (request phase)
3. Middleware calls `await _next(context)` — execution transfers down the chain
4. Eventually the terminal middleware (endpoint) executes and writes the response
5. Control returns to this middleware *after* `await _next(context)` returns
6. Middleware executes its *post-next* logic (response phase)
7. Control returns to the *previous* middleware in the chain

**How `Use` vs `Run` vs `Map` differ:**
- `Use` — adds middleware that calls next (bidirectional)
- `Run` — adds terminal middleware (no next call, short-circuits)
- `Map` — branches the pipeline based on path prefix (does NOT rejoin)
- `UseWhen` — branches based on predicate and **rejoins** the main pipeline
- `MapWhen` — branches based on predicate and **does NOT rejoin**

### The Real-World Code

```csharp
// ── Correct middleware order — this order is not arbitrary ──────────────────
public void Configure(IApplicationBuilder app, IWebHostEnvironment env)
{
    // 1. Exception handler MUST be first — catches exceptions from everything below
    app.UseExceptionHandler();

    // 2. HSTS before redirect so the header is set even on first HTTPS request
    if (!env.IsDevelopment())
        app.UseHsts();

    // 3. HTTPS redirect before routing — terminates before route matching
    app.UseHttpsRedirection();

    // 4. Static files before authentication — images/CSS don't need auth
    app.UseStaticFiles();

    // 5. Routing must run before authorization (auth needs the route endpoint's metadata)
    app.UseRouting();

    // 6. CORS must be after routing (needs route match) but before auth
    app.UseCors();

    // 7. Authentication populates User; must precede authorization
    app.UseAuthentication();

    // 8. Authorization checks User against endpoint policy
    app.UseAuthorization();

    // 9. Output cache respects auth — only after auth so we don't serve
    //    a cached authenticated response to an anonymous user
    app.UseOutputCache();

    // 10. Rate limiter after auth so we can partition by user identity
    app.UseRateLimiter();

    // 11. Endpoints run last
    app.MapControllers();
}


// ── Production-grade middleware class ────────────────────────────────────────
public sealed class CorrelationIdMiddleware
{
    private const string HeaderName = "X-Correlation-Id";
    private readonly RequestDelegate _next;
    private readonly ILogger<CorrelationIdMiddleware> _logger;

    public CorrelationIdMiddleware(RequestDelegate next, ILogger<CorrelationIdMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        // Try to use the caller's correlation ID so it propagates across services
        var correlationId = context.Request.Headers[HeaderName].FirstOrDefault()
                            ?? Guid.NewGuid().ToString("N");

        // Store on context.Items — survives for the entire request lifetime
        context.Items[HeaderName] = correlationId;

        // Add to response regardless of outcome so caller can trace it
        context.Response.OnStarting(() =>
        {
            // OnStarting fires just before headers are sent — safe mutation point
            context.Response.Headers.TryAdd(HeaderName, correlationId);
            return Task.CompletedTask;
        });

        // Push into logging scope so every log line in this request carries the ID
        using (_logger.BeginScope(new Dictionary<string, object>
               { ["CorrelationId"] = correlationId }))
        {
            await _next(context);
        }
    }
}

public static class CorrelationIdMiddlewareExtensions
{
    public static IApplicationBuilder UseCorrelationId(this IApplicationBuilder app)
        => app.UseMiddleware<CorrelationIdMiddleware>();
}


// ── UseWhen vs MapWhen ────────────────────────────────────────────────────────
// UseWhen: API requests get extra logging, then REJOIN the main pipeline
app.UseWhen(
    ctx => ctx.Request.Path.StartsWithSegments("/api"),
    apiApp => apiApp.UseMiddleware<ApiRequestLoggingMiddleware>()
);

// MapWhen: webhook requests get completely separate handling, NO rejoin
app.MapWhen(
    ctx => ctx.Request.Path.StartsWithSegments("/webhooks"),
    webhookApp =>
    {
        webhookApp.UseMiddleware<WebhookSignatureValidationMiddleware>();
        webhookApp.UseRouting();
        webhookApp.UseEndpoints(e => e.MapControllers());
    }
);
```

## 2. KESTREL WEB SERVER

### What It Actually Is
Kestrel is ASP.NET Core's built-in, cross-platform HTTP server. It is not IIS — it is a raw socket-level server built on `System.IO.Pipelines` and `libuv`/socket abstraction. In production you typically put a reverse proxy (Nginx, IIS, Azure Application Gateway) in front of it, but Kestrel can also face the internet directly. Its pipeline-based I/O model avoids copying bytes between buffers, which is why it benchmarks faster than most alternatives.

### How It Works — Mechanically

1. **Listen** — Kestrel opens a TCP socket on the configured port. For HTTPS it performs the TLS handshake using the certificate you configure.
2. **Accept connections** — Each accepted connection gets a `ConnectionContext` wrapping `System.IO.Pipelines.Pipe` for reads and writes.
3. **HTTP parsing** — For HTTP/1.1, Kestrel parses the raw bytes into an `HttpRequestFeature`. For HTTP/2, it handles framing, stream multiplexing, and HPACK header decompression internally.
4. **Feature collection** — The parsed data is exposed via `IFeatureCollection` on `HttpContext`. Middleware reads/writes via these feature interfaces, not raw bytes.
5. **Dispatch** — After parsing headers, Kestrel calls the `RequestDelegate` chain (your middleware pipeline).
6. **Response write** — When middleware writes to `HttpContext.Response`, it writes to the output `Pipe`; Kestrel flushes it to the socket.

```
TCP Socket → TLS → HTTP Parser → IFeatureCollection → RequestDelegate chain → Response Pipe → Socket
```

### The Real-World Code

```csharp
// ── Complete Kestrel configuration for a production service ──────────────────
builder.WebHost.ConfigureKestrel((context, options) =>
{
    // Read limits from config so they can be changed without redeployment
    var kestrelConfig = context.Configuration.GetSection("Kestrel");
    options.Configure(kestrelConfig, reloadOnChange: true);

    // Explicit tuning — these numbers matter for high-throughput services
    options.Limits.MaxConcurrentConnections = 5_000;
    options.Limits.MaxConcurrentUpgradedConnections = 1_000; // WebSocket, HTTP/2 upgrades
    options.Limits.MaxRequestBodySize = 50 * 1024 * 1024;   // 50 MB — tune per your API contract
    options.Limits.MaxRequestHeadersTotalSize = 32_768;      // 32 KB header limit

    // Timeouts — critical for preventing slow-loris attacks
    options.Limits.RequestHeadersTimeout = TimeSpan.FromSeconds(15);
    options.Limits.KeepAliveTimeout = TimeSpan.FromMinutes(2);

    // HTTP/2 stream limits
    options.Limits.Http2.MaxStreamsPerConnection = 100;
    options.Limits.Http2.InitialConnectionWindowSize = 131_072; // 128 KB
    options.Limits.Http2.InitialStreamWindowSize = 98_304;      // 96 KB

    // Listen configuration
    options.ListenAnyIP(8080, listenOptions =>
    {
        // Prefer HTTP/3 (QUIC) → HTTP/2 → HTTP/1.1 in that order
        listenOptions.Protocols = HttpProtocols.Http1AndHttp2AndHttp3;
    });

    options.ListenAnyIP(8443, listenOptions =>
    {
        listenOptions.Protocols = HttpProtocols.Http1AndHttp2;
        listenOptions.UseHttps(httpsOptions =>
        {
            // Rotate certs without restart by reloading from store
            httpsOptions.ServerCertificateSelector = (_, serverName) =>
                CertificateHelper.LoadFromStore(serverName);
        });
    });
});


// appsettings.json for the options.Configure(kestrelConfig) call above:
// {
//   "Kestrel": {
//     "Limits": {
//       "MaxConcurrentConnections": 5000,
//       "MaxRequestBodySize": 52428800
//     }
//   }
// }
```

## 3. HOST LIFECYCLE & APPLICATION LIFETIME

### What It Actually Is
The `WebApplication` (or `IHost`) is the container that owns the DI container, the server, and the application's lifetime. It manages the startup and shutdown sequence. `IHostApplicationLifetime` exposes `CancellationToken`s you can hook into for graceful shutdown — stopping background work, draining connections, or running cleanup. If you ignore this, you get data corruption on pod restarts.

### How It Works — Mechanically

1. `WebApplication.CreateBuilder(args)` — creates the builder, registers default services (logging, config, DI).
2. `builder.Build()` — constructs the DI container, validates service registrations (in dev).
3. `app.Run()` — starts Kestrel, calls `IHostedService.StartAsync` on all registered hosted services, begins accepting requests.
4. SIGTERM / Ctrl+C received — triggers `IHostApplicationLifetime.ApplicationStopping` token.
5. **Graceful period** — configured via `hostOptions.ShutdownTimeout` (default 5s). Kestrel drains in-flight requests.
6. `IHostedService.StopAsync` called on all hosted services (in reverse order of registration).
7. DI container disposed — all `IDisposable`/`IAsyncDisposable` singletons disposed.
8. Process exits.

```
CreateBuilder → Build (DI container) → Run
                                         ├── StartAsync (all IHostedServices)
                                         ├── Begin accepting requests
                                         │
                                    [SIGTERM]
                                         │
                                         ├── ApplicationStopping token fires
                                         ├── Stop accepting new requests
                                         ├── Drain in-flight requests (ShutdownTimeout)
                                         ├── StopAsync (all IHostedServices, reverse order)
                                         └── Dispose DI container
```

### The Real-World Code

```csharp
// ── Program.cs with full lifecycle hooks ─────────────────────────────────────
var builder = WebApplication.CreateBuilder(args);

builder.Services.Configure<HostOptions>(options =>
{
    // Give background services 30 seconds to finish gracefully
    // Default is 5 seconds — almost always too short for real workloads
    options.ShutdownTimeout = TimeSpan.FromSeconds(30);
});

var app = builder.Build();

var lifetime = app.Services.GetRequiredService<IHostApplicationLifetime>();

lifetime.ApplicationStarted.Register(() =>
    app.Logger.LogInformation("Application started. PID: {Pid}", Environment.ProcessId));

lifetime.ApplicationStopping.Register(() =>
    app.Logger.LogInformation("Graceful shutdown initiated. Draining..."));

lifetime.ApplicationStopped.Register(() =>
    app.Logger.LogInformation("Application fully stopped."));

await app.RunAsync();


// ── Using IHostApplicationLifetime inside a hosted service ───────────────────
public class DatabaseMigrationService : IHostedService
{
    private readonly IServiceScopeFactory _scopeFactory;
    private readonly IHostApplicationLifetime _lifetime;
    private readonly ILogger<DatabaseMigrationService> _logger;

    public DatabaseMigrationService(
        IServiceScopeFactory scopeFactory,
        IHostApplicationLifetime lifetime,
        ILogger<DatabaseMigrationService> logger)
    {
        _scopeFactory = scopeFactory;
        _lifetime = lifetime;
        _logger = logger;
    }

    public async Task StartAsync(CancellationToken cancellationToken)
    {
        try
        {
            await using var scope = _scopeFactory.CreateAsyncScope();
            var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();

            _logger.LogInformation("Applying database migrations...");
            await db.Database.MigrateAsync(cancellationToken);
            _logger.LogInformation("Database migrations applied.");
        }
        catch (Exception ex)
        {
            _logger.LogCritical(ex, "Database migration failed. Stopping application.");
            // Stop the entire app if migrations fail — don't serve traffic with a broken DB
            _lifetime.StopApplication();
        }
    }

    public Task StopAsync(CancellationToken cancellationToken) => Task.CompletedTask;
}
```

## 4. DEPENDENCY INJECTION — INTERNALS

### What It Actually Is
ASP.NET Core's built-in DI container (`ServiceCollection`/`ServiceProvider`) is an IoC container that builds and resolves object graphs. When you call `GetRequiredService<T>()`, it traverses the registered descriptor graph, recursively creates all constructor dependencies, and returns the constructed object — respecting lifetime scoping. Think of it as a factory that memoizes based on lifetime: Singleton = memoize forever, Scoped = memoize per scope, Transient = never memoize.

### How It Works — Mechanically

1. **Registration** (`IServiceCollection`) — you add `ServiceDescriptor` entries: `{ServiceType, ImplementationType, Lifetime}`.
2. **Build** (`BuildServiceProvider()`) — the container compiles resolution trees at build time (not per-resolve). In .NET 6+ it validates scopes in Development mode.
3. **Root scope** — the `ServiceProvider` itself is the root scope. Singletons live here.
4. **Request scope** — ASP.NET Core creates a new `IServiceScope` per HTTP request. Scoped services are resolved from and stored on this scope. Scope is disposed at request end.
5. **Resolution** — `GetRequiredService<T>()` looks up the descriptor, checks its lifetime bucket (singleton = root cache, scoped = current scope cache, transient = always new), constructs if not cached.
6. **Disposal** — when a scope disposes, all `IDisposable` scoped and transient instances that were resolved through that scope get `Dispose()` called.

**The captive dependency bug:**
```
Singleton (lives forever)
  └── Scoped (should live per-request)
        ← BUG: Scoped is captured by Singleton's constructor
           It now lives for the app's entire lifetime.
           Every request shares the same "scoped" instance.
           If it holds a DbContext, you have a thread-safety disaster.
```

### The Real-World Code

```csharp
// ── Full service registration patterns ──────────────────────────────────────
var builder = WebApplication.CreateBuilder(args);

// Transient: stateless, cheap to create (validators, mappers)
builder.Services.AddTransient<IOrderValidator, OrderValidator>();

// Scoped: one per request (DbContext, CurrentUserService, UnitOfWork)
builder.Services.AddScoped<AppDbContext>();
builder.Services.AddScoped<ICurrentUserService, CurrentUserService>();
builder.Services.AddScoped<IUnitOfWork, UnitOfWork>();

// Singleton: expensive to create, stateless or thread-safe (caches, HTTP clients)
builder.Services.AddSingleton<IEmailTemplateCache, EmailTemplateCache>();

// Factory registration — conditional construction
builder.Services.AddScoped<IPaymentGateway>(sp =>
{
    var config = sp.GetRequiredService<IConfiguration>();
    var httpFactory = sp.GetRequiredService<IHttpClientFactory>();
    return config["Payment:Provider"] switch
    {
        "Stripe" => new StripeGateway(httpFactory),
        "PayPal" => new PayPalGateway(config["Payment:PayPal:ClientId"]!),
        _ => throw new InvalidOperationException("Unknown payment provider")
    };
});

// Open generic registration — one line covers all IRepository<T>
builder.Services.AddScoped(typeof(IRepository<>), typeof(EfRepository<>));

// Keyed services (.NET 8+) — multiple implementations under same interface
builder.Services.AddKeyedSingleton<IStorageProvider, AzureBlobStorageProvider>("azure");
builder.Services.AddKeyedSingleton<IStorageProvider, S3StorageProvider>("s3");
builder.Services.AddKeyedSingleton<IStorageProvider, LocalDiskStorageProvider>("local");

// Validate on startup — catches misconfiguration before the first request
builder.Services.AddOptions<DatabaseOptions>()
    .BindConfiguration("Database")
    .ValidateDataAnnotations()
    .ValidateOnStart();


// ── Correct pattern: singleton using scoped services ─────────────────────────
public sealed class OutboxPublisher : IOutboxPublisher
{
    // IServiceScopeFactory is itself a singleton — safe to inject into singletons
    private readonly IServiceScopeFactory _scopeFactory;
    private readonly ILogger<OutboxPublisher> _logger;

    public OutboxPublisher(IServiceScopeFactory scopeFactory, ILogger<OutboxPublisher> logger)
    {
        _scopeFactory = scopeFactory;
        _logger = logger;
    }

    public async Task PublishPendingAsync(CancellationToken ct)
    {
        // Create a new scope per invocation — each gets a fresh DbContext, fresh UoW
        await using var scope = _scopeFactory.CreateAsyncScope();
        var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
        var publisher = scope.ServiceProvider.GetRequiredService<IMessageBus>();

        var pending = await db.OutboxMessages
            .Where(m => m.ProcessedAt == null)
            .Take(50)
            .ToListAsync(ct);

        foreach (var msg in pending)
        {
            await publisher.PublishAsync(msg.EventType, msg.Payload, ct);
            msg.ProcessedAt = DateTime.UtcNow;
        }
        await db.SaveChangesAsync(ct);
    }
}


// ── Keyed service resolution ──────────────────────────────────────────────────
public class FileUploadService
{
    private readonly IStorageProvider _storage;

    // Constructor injection with key attribute
    public FileUploadService(
        [FromKeyedServices("azure")] IStorageProvider storage)
    {
        _storage = storage;
    }
}

// Or dynamically — useful when the key is runtime data
public class StorageRouter
{
    private readonly IServiceProvider _sp;
    public StorageRouter(IServiceProvider sp) => _sp = sp;

    public IStorageProvider GetProvider(string tenantPreference)
        => _sp.GetRequiredKeyedService<IStorageProvider>(tenantPreference);
}


// ── Decorator pattern without Scrutor ────────────────────────────────────────
// Adds caching to IProductRepository without modifying it
builder.Services.AddScoped<IProductRepository>(sp =>
{
    var inner = new EfProductRepository(sp.GetRequiredService<AppDbContext>());
    var cache = sp.GetRequiredService<IMemoryCache>();
    var logger = sp.GetRequiredService<ILogger<CachingProductRepository>>();
    return new CachingProductRepository(inner, cache, logger);
});
```

## 5. CONFIGURATION SYSTEM

### What It Actually Is
ASP.NET Core's configuration system is a layered key-value store that merges multiple providers (JSON files, environment variables, Azure Key Vault, command-line args) into a unified `IConfiguration`. Later providers override earlier ones. The hierarchy is expressed with colons: `"ConnectionStrings:Default"` maps to `appsettings.json → ConnectionStrings → Default`. Environment variables use double underscore as separator (`ConnectionStrings__Default`) because colons are invalid in many shells.

### How It Works — Mechanically

1. `WebApplication.CreateBuilder()` registers default providers **in this order** (each overrides previous):
   - `appsettings.json`
   - `appsettings.{Environment}.json` (e.g., `appsettings.Production.json`)
   - User Secrets (Development only)
   - Environment Variables
   - Command-line arguments
2. Each provider implements `IConfigurationProvider` which exposes a flat `IDictionary<string, string>`.
3. `ConfigurationBuilder.Build()` merges all providers (last-wins) into `IConfigurationRoot`.
4. `IConfiguration["Key:SubKey"]` traverses the merged dictionary.
5. `GetSection("Key")` returns a sub-tree view; `Bind<T>()` maps it to a POCO.

### The Real-World Code

```csharp
// ── Program.cs: adding providers beyond defaults ──────────────────────────────
builder.Configuration
    .AddJsonFile("appsettings.json", optional: false, reloadOnChange: true)
    .AddJsonFile($"appsettings.{builder.Environment.EnvironmentName}.json",
        optional: true, reloadOnChange: true)
    .AddEnvironmentVariables(prefix: "MYAPP_")  // Only MYAPP_-prefixed env vars
    .AddCommandLine(args);

// In production, add Key Vault — secrets never live in appsettings
if (builder.Environment.IsProduction())
{
    var keyVaultUri = new Uri(builder.Configuration["KeyVault:Uri"]!);
    builder.Configuration.AddAzureKeyVault(keyVaultUri, new DefaultAzureCredential());
}


// ── appsettings.json structure ───────────────────────────────────────────────
// {
//   "ConnectionStrings": {
//     "Default": "Server=...;Database=...;",
//     "Redis": "localhost:6379"
//   },
//   "Email": {
//     "SmtpHost": "smtp.example.com",
//     "SmtpPort": 587,
//     "From": "no-reply@example.com"
//   },
//   "Features": {
//     "NewCheckout": false,
//     "BetaDashboard": true
//   }
// }


// ── Reading configuration correctly ──────────────────────────────────────────
// Never inject IConfiguration into application services — too broad, no typing
// Use the Options pattern instead (see §6)

// OK in Program.cs/infrastructure only:
var connectionString = builder.Configuration.GetConnectionString("Default");
var smtpPort = builder.Configuration.GetValue<int>("Email:SmtpPort", defaultValue: 587);

// In services, always use IOptions<T> (see §6)


// ── Custom configuration provider (e.g., database-backed feature flags) ──────
public class DatabaseConfigurationProvider : ConfigurationProvider
{
    private readonly string _connectionString;

    public DatabaseConfigurationProvider(string connectionString)
        => _connectionString = connectionString;

    public override void Load()
    {
        using var conn = new SqlConnection(_connectionString);
        conn.Open();
        using var cmd = new SqlCommand(
            "SELECT [Key], [Value] FROM AppSettings WHERE IsActive = 1", conn);
        using var reader = cmd.ExecuteReader();
        var data = new Dictionary<string, string?>(StringComparer.OrdinalIgnoreCase);
        while (reader.Read())
            data[reader.GetString(0)] = reader.GetString(1);
        Data = data;
    }
}

public class DatabaseConfigurationSource : IConfigurationSource
{
    private readonly string _connectionString;
    public DatabaseConfigurationSource(string connectionString)
        => _connectionString = connectionString;

    public IConfigurationProvider Build(IConfigurationBuilder builder)
        => new DatabaseConfigurationProvider(_connectionString);
}

// Register it
builder.Configuration.Add(
    new DatabaseConfigurationSource(
        builder.Configuration.GetConnectionString("Default")!));
```

## 6. THE OPTIONS PATTERN

### What It Actually Is
The Options pattern is the correct way to deliver strongly-typed, validated configuration to your services. Instead of injecting `IConfiguration` (which is stringly-typed and gives you zero validation), you define a POCO, bind config to it, validate it at startup, and inject `IOptions<T>`, `IOptionsSnapshot<T>`, or `IOptionsMonitor<T>` depending on whether you need live-reload. The three interfaces differ in lifetime and refresh behavior — choosing the wrong one causes stale config bugs.

### How It Works — Mechanically

- **`IOptions<T>`** — Singleton. Reads config once at first access. Never updates during app lifetime. Use for config that never changes (database connection strings, feature flags at boot).
- **`IOptionsSnapshot<T>`** — Scoped. Reads config once per request. Reflects JSON file changes (because `reloadOnChange: true`). Use in request-handling services where you want hot reload without restarting.
- **`IOptionsMonitor<T>`** — Singleton. Actively watches for changes and fires `OnChange` callback. Use in singletons that need live config updates (e.g., rate limiter thresholds).

```
appsettings.json (reloadOnChange: true)
       │
       ▼
IConfigurationRoot (live reference)
       │
       ├── IOptions<T>         → reads once at startup, frozen
       ├── IOptionsSnapshot<T> → reads per-request (scoped)
       └── IOptionsMonitor<T>  → reads live, OnChange callback
```

### The Real-World Code

```csharp
// ── Options class — the model ─────────────────────────────────────────────────
public sealed class EmailOptions
{
    public const string SectionName = "Email";

    [Required]
    public string SmtpHost { get; init; } = null!;

    [Range(1, 65535)]
    public int SmtpPort { get; init; } = 587;

    [Required, EmailAddress]
    public string FromAddress { get; init; } = null!;

    public bool EnableSsl { get; init; } = true;

    [Range(1, 10)]
    public int MaxRetries { get; init; } = 3;
}

// ── Registration with validation ──────────────────────────────────────────────
builder.Services.AddOptions<EmailOptions>()
    .BindConfiguration(EmailOptions.SectionName)
    .ValidateDataAnnotations()
    .Validate(options =>
        // Custom cross-field validation DataAnnotations can't express
        !(options.EnableSsl && options.SmtpPort == 25),
        "Port 25 cannot be used with SSL")
    .ValidateOnStart(); // Fail immediately on bad config, not on first use


// ── Consuming options — choosing the right interface ─────────────────────────

// IOptions<T>: use when config is static, service is singleton/scoped
public sealed class SmtpEmailService : IEmailService
{
    private readonly EmailOptions _options;

    public SmtpEmailService(IOptions<EmailOptions> options)
    {
        // .Value throws OptionsValidationException if validation failed
        _options = options.Value;
    }

    public async Task SendAsync(string to, string subject, string body, CancellationToken ct)
    {
        using var client = new SmtpClient(_options.SmtpHost, _options.SmtpPort);
        client.EnableSsl = _options.EnableSsl;
        // ...
    }
}


// IOptionsMonitor<T>: use when service is singleton but config can change
public sealed class RateLimitingService
{
    private volatile RateLimitOptions _current;

    public RateLimitingService(IOptionsMonitor<RateLimitOptions> monitor)
    {
        _current = monitor.CurrentValue;

        // OnChange fires when appsettings.json changes on disk
        // volatile ensures reads see the latest write across threads
        monitor.OnChange(updated => _current = updated);
    }

    public bool IsAllowed(string clientId)
    {
```

## 7. ROUTING — DEEP DIVE

### What It Actually Is
Routing is the process of matching an incoming HTTP request (method + path) to an endpoint and extracting route parameters. ASP.NET Core uses **endpoint routing** (introduced in 3.0): routing is split into two middleware — `UseRouting()` (matches the request to an endpoint and stores the result) and `UseEndpoints()`/`MapControllers()` (executes the matched endpoint). This split allows authorization middleware to run *between* route matching and execution, knowing which endpoint is about to execute.

### How It Works — Mechanically

1. `UseRouting()` runs the **route matching engine** — it evaluates all registered endpoints' route templates against the current URL and HTTP method.
2. If a match is found, the matched `Endpoint` is stored on `HttpContext.Features.Get<IEndpointFeature>()`.
3. **Constraint evaluation** — route constraints (`:int`, `:guid`, `:minlength(3)`, etc.) are checked. Mismatches fall through to the next candidate.
4. Authorization middleware reads the matched endpoint's metadata (e.g., `[Authorize]` attribute or `RequireAuthorization()`) to make access decisions *before* execution.
5. The endpoint executes — for controllers, this means model binding, action filter pipeline, action method, result execution.

**Route template precedence (when multiple templates match):**
```
1. Literal segments    /api/orders/new         (most specific)
2. Constrained params  /api/orders/{id:int}
3. Unconstrained params /api/orders/{id}
4. Catch-all           /api/{**slug}           (least specific)
```

### The Real-World Code

## 8. CONTROLLERS, ACTIONS, MODEL BINDING

### What It Actually Is
`ControllerBase` (for APIs) and `Controller` (for MVC with views) are the classes from which your API controllers inherit. `[ApiController]` attribute changes default behaviors: automatic model validation with 400 response, binding source inference (route params come from route, complex types from body), and problem details formatting. Model binding is the process by which ASP.NET Core populates action method parameters from the HTTP request — from route, query string, headers, body, or form data.

### How It Works — Mechanically

**Model binding order (without explicit `[From*]` attributes):**
1. Route values (`[FromRoute]`)
2. Query string (`[FromQuery]`)
3. Request body (`[FromBody]`) — only for complex types when `[ApiController]` is present

**`[ApiController]` implicit behaviors:**
- Auto-400 on `ModelState.IsValid == false` (before your action runs)
- `[FromBody]` inferred for complex types
- `[FromRoute]` inferred for action params matching route template tokens
- `[FromQuery]` inferred for everything else

**Action execution pipeline:**
```
Request → Route Matched → Model Binding → ModelState Validation
       → Authorization Filters → Resource Filters → Action Filters
       → Action Method → Result Filters → Action Result Execution
       → Response written
```

### The Real-World Code

## 9. MINIMAL APIS

### What It Actually Is
Minimal APIs are a lightweight alternative to controllers, introduced in .NET 6. Instead of a class hierarchy, you register handlers as lambdas or static methods directly on `WebApplication`. They use the same endpoint routing and middleware pipeline as controllers — the difference is purely in the registration API. In .NET 7+ with route groups and endpoint filters, they're feature-complete with controllers for most APIs and have less overhead per request (no filter pipeline boxing, no controller activation).

### How It Works — Mechanically

1. `app.MapGet("/path", handler)` registers an endpoint descriptor with the route template and HTTP method.
2. The `handler` delegate's parameters are analyzed at startup (via source generation in .NET 7+ or reflection in earlier versions) to determine binding sources.
3. At request time, the binder populates each parameter and invokes the handler.
4. The handler's return type determines the response: `IResult` → written directly, `T` (POCO) → serialized as JSON, `Task<T>` → awaited then serialized.

**Parameter binding precedence (Minimal APIs):**
- Known service types → injected from DI
- `CancellationToken`, `HttpContext`, `HttpRequest`, `HttpResponse` → framework special cases
- Complex types (class/record) → body (JSON)
- Simple types (int, string, Guid) → route first, then query string
- `[FromServices]`, `[FromRoute]`, `[FromQuery]`, `[FromBody]`, `[FromHeader]` → explicit

### The Real-World Code

```csharp
public sealed class ValidationFilter<TRequest> : IEndpointFilter
{
    public async ValueTask<object?> InvokeAsync(
        EndpointFilterInvocationContext context,
        EndpointFilterDelegate next)
    {
        // Find the request object among the handler's arguments
        var argument = context.Arguments.OfType<TRequest>().FirstOrDefault();

        if (argument is not null)
        {
            var validator = context.HttpContext.RequestServices
                .GetService<IValidator<TRequest>>();

            if (validator is not null)
            {
                var result = await validator.ValidateAsync(
                    argument,
                    context.HttpContext.RequestAborted);

                if (!result.IsValid)
                    // Returns 400 with RFC 7807 problem details format
                    return TypedResults.ValidationProblem(result.ToDictionary());
            }
        }

        return await next(context);
    }
}
```

## 10. FILTERS (MVC)

### What It Actually Is
MVC filters run within the MVC action pipeline — after routing and model binding but before/after the action method. They are not middleware (they run later, after middleware has resolved the endpoint). The five filter types each intercept a different phase: Authorization → Resource → Action → Exception → Result. They let you add cross-cutting concerns (logging, caching, validation) to specific controllers or actions without modifying the action code.

### How It Works — Mechanically

```
[Authorization Filter]  ← runs first, short-circuits if unauthorized
       │
[Resource Filter]       ← can short-circuit before model binding (good for caching)
       │
  [Model Binding]       ← populates action parameters
       │
[Action Filter (before)]← runs before action method
       │
  [Action Method]       ← your code
       │
[Action Filter (after)] ← runs after action method, can modify result
       │
[Exception Filter]      ← catches exceptions from action and result execution
       │
[Result Filter (before)]← runs before result execution
       │
  [Result Execution]    ← writes response to wire
       │
[Result Filter (after)] ← runs after result execution
```

### The Real-World Code

```csharp
// ── Action filter: transaction per request ────────────────────────────────────
public class TransactionFilter : IAsyncActionFilter
{
    private readonly AppDbContext _db;

    // Filters can receive scoped services via constructor (they're resolved per request)
    public TransactionFilter(AppDbContext db) => _db = db;

    public async Task OnActionExecutionAsync(
        ActionExecutingContext context,
        ActionExecutionDelegate next)
    {
        // Skip for read-only HTTP methods — no transaction needed
        if (HttpMethods.IsGet(context.HttpContext.Request.Method) ||
            HttpMethods.IsHead(context.HttpContext.Request.Method))
        {
            await next();
            return;
        }

        await using var transaction = await _db.Database.BeginTransactionAsync(
            context.HttpContext.RequestAborted);
        try
        {
            var result = await next();

            if (result.Exception is null || result.ExceptionHandled)
            {
                await transaction.CommitAsync(context.HttpContext.RequestAborted);
            }
            else
            {
                await transaction.RollbackAsync();
            }
        }
        catch
        {
            await transaction.RollbackAsync();
            throw;
        }
    }
}

// Register globally so every controller gets transactions
builder.Services.AddControllers(options =>
    options.Filters.Add<TransactionFilter>());

// Or per-controller or per-action with [ServiceFilter(typeof(TransactionFilter))]
```

## 11. AUTHENTICATION — ALL SCHEMES

### What It Actually Is
Authentication in ASP.NET Core is the process of determining *who* is making a request — establishing identity. The `UseAuthentication()` middleware calls the registered authentication handler(s), which examine the request (check the Authorization header, cookies, API key header) and produce a `ClaimsPrincipal` stored on `HttpContext.User`. This happens every request, regardless of whether the endpoint requires auth. Authorization (§12) then decides what that identity is *allowed* to do.

### How It Works — Mechanically

1. `UseAuthentication()` middleware runs.
2. It calls `IAuthenticationService.AuthenticateAsync(context, scheme)` for the default scheme.
3. The scheme's `AuthenticationHandler` reads the request (e.g., reads `Authorization: Bearer <token>`).
4. Handler validates the credential (verifies JWT signature, decrypts cookie, validates API key).
5. On success, handler creates a `ClaimsPrincipal` with `ClaimsIdentity` and stores it on `HttpContext.User`.
6. On failure, `HttpContext.User` remains anonymous (not null — just an unauthenticated principal).
7. `UseAuthorization()` then checks `HttpContext.User` against the endpoint's policy.

### The Real-World Code

```csharp

public class ApiKeyAuthenticationOptions : AuthenticationSchemeOptions { }

public sealed class ApiKeyAuthenticationHandler
    : AuthenticationHandler<ApiKeyAuthenticationOptions>
{
    private const string ApiKeyHeaderName = "X-Api-Key";
    private readonly IApiKeyService _apiKeyService;

    public ApiKeyAuthenticationHandler(
        IOptionsMonitor<ApiKeyAuthenticationOptions> options,
        ILoggerFactory logger,
        UrlEncoder encoder,
        IApiKeyService apiKeyService)
        : base(options, logger, encoder)
    {
        _apiKeyService = apiKeyService;
    }

    protected override async Task<AuthenticateResult> HandleAuthenticateAsync()
    {
        if (!Request.Headers.TryGetValue(ApiKeyHeaderName, out var apiKeyValues))
            return AuthenticateResult.NoResult(); // No API key — let next scheme try

        var apiKey = apiKeyValues.FirstOrDefault();
        if (string.IsNullOrWhiteSpace(apiKey))
            return AuthenticateResult.NoResult();

        var keyRecord = await _apiKeyService.ValidateAsync(apiKey, Context.RequestAborted);
        if (keyRecord is null)
            return AuthenticateResult.Fail("Invalid API key");

        var claims = new[]
        {
            new Claim(ClaimTypes.NameIdentifier, keyRecord.ServiceId),
            new Claim(ClaimTypes.Name, keyRecord.ServiceName),
            new Claim("scope", keyRecord.Scope),
            new Claim("auth_method", "api_key")
        };

        var identity = new ClaimsIdentity(claims, Scheme.Name);
        var principal = new ClaimsPrincipal(identity);
        var ticket = new AuthenticationTicket(principal, Scheme.Name);

        return AuthenticateResult.Success(ticket);
    }

    protected override Task HandleChallengeAsync(AuthenticationProperties properties)
    {
        Response.StatusCode = StatusCodes.Status401Unauthorized;
        Response.Headers.WWWAuthenticate = $"ApiKey realm=\"{Request.Host}\"";
        return Task.CompletedTask;
    }
}
```

## 12. JWT BEARER AUTHENTICATION

### What It Actually Is
JWT (JSON Web Token) is a compact, self-contained token that encodes identity claims and is signed (or encrypted) by the issuer. The receiver validates the signature without calling the issuer — making JWT **stateless**. The tradeoff: tokens cannot be revoked before expiry without a token blacklist. Structure is `Base64(header).Base64(payload).Base64(signature)`. The `Authorization: Bearer <token>` header carries it on every request.

### How It Works — Mechanically

**Token generation (issuer side):**
1. Create `ClaimsIdentity` with the user's claims.
2. Create `SecurityTokenDescriptor` with issuer, audience, expiry, signing key.
3. `JwtSecurityTokenHandler.CreateToken(descriptor)` builds the three base64 parts.
4. Return token string to client.

**Token validation (receiver side):**
1. `JwtBearerHandler` extracts the token from `Authorization: Bearer` header.
2. Calls `JwtSecurityTokenHandler.ValidateToken(token, parameters, out SecurityToken)`.
3. Verifies signature using the configured key.
4. Checks `exp` (expiry), `iss` (issuer), `aud` (audience) claims.
5. If valid, creates `ClaimsPrincipal` from token claims.

```
Client                          API Server                    Auth Server
  │                                │                               │
  │─── POST /auth/login ──────────►│                               │
  │                                │──── ValidateCredentials ─────►│
  │                                │◄──── User + Roles ────────────│
  │                                │  CreateJwt(claims, secret)    │

  │─── GET /api/orders ───────────►│                               │
  │  Authorization: Bearer <jwt>   │  ValidateJwt(token, secret)   │
  │                                │  (no network call needed)     │
  │◄─── 200 OK orders ─────────────│                               │
```

```csharp
// ── Token generation service ──────────────────────────────────────────────────
public sealed class JwtTokenService : IJwtTokenService
{
    private readonly JwtOptions _options;
    private readonly SymmetricSecurityKey _signingKey;
    private readonly SigningCredentials _credentials;

    public JwtTokenService(IOptions<JwtOptions> options)
    {
        _options = options.Value;
        _signingKey = new SymmetricSecurityKey(
            Encoding.UTF8.GetBytes(_options.Secret));
        // HS256 is fine for single-service; RS256 needed for distributed issuers
        _credentials = new SigningCredentials(_signingKey, SecurityAlgorithms.HmacSha256);
    }

    public TokenPair GenerateTokenPair(User user)
    {
        var accessToken = CreateAccessToken(user);
        var refreshToken = CreateRefreshToken();

        return new TokenPair(
            AccessToken: accessToken,
            RefreshToken: refreshToken,
            AccessTokenExpiresAt: DateTime.UtcNow.Add(_options.AccessTokenExpiry),
            RefreshTokenExpiresAt: DateTime.UtcNow.Add(_options.RefreshTokenExpiry));
    }

    private string CreateAccessToken(User user)
    {
        var claims = new List<Claim>
        {
            new(JwtRegisteredClaimNames.Sub, user.Id.ToString()),
            new(JwtRegisteredClaimNames.Email, user.Email),
            new(JwtRegisteredClaimNames.Name, user.FullName),
            new(JwtRegisteredClaimNames.Jti, Guid.NewGuid().ToString("N")), // Unique token ID
            // Roles as individual claims — standard approach for policy checks
            ..user.Roles.Select(r => new Claim(ClaimTypes.Role, r)),
            // Custom claims for your domain
            new("tid", user.TenantId.ToString()),
            new("plan", user.SubscriptionPlan)
        };

        var descriptor = new SecurityTokenDescriptor
        {
            Subject = new ClaimsIdentity(claims),
            Expires = DateTime.UtcNow.Add(_options.AccessTokenExpiry),
            NotBefore = DateTime.UtcNow,
            IssuedAt = DateTime.UtcNow,
            Issuer = _options.Issuer,
            Audience = _options.Audience,
            SigningCredentials = _credentials
        };

        var handler = new JwtSecurityTokenHandler();
        var token = handler.CreateToken(descriptor);
        return handler.WriteToken(token);
    }

    private static string CreateRefreshToken()
    {
        // Refresh tokens are opaque random strings — not JWTs
        // They're stored in the database and exchanged for new access tokens
        var bytes = RandomNumberGenerator.GetBytes(64);
        return Convert.ToBase64String(bytes);
    }
}


// ── Refresh token endpoint ────────────────────────────────────────────────────
app.MapPost("/api/auth/refresh", async (
    RefreshTokenRequest request,
    IJwtTokenService tokenService,
    IRefreshTokenRepository refreshTokenRepo,
    IUserRepository userRepo,
    CancellationToken ct) =>
{
    var stored = await refreshTokenRepo.GetAsync(request.RefreshToken, ct);

    if (stored is null || stored.IsRevoked || stored.ExpiresAt < DateTime.UtcNow)
        return Results.Problem(
            statusCode: StatusCodes.Status401Unauthorized,
            title: "Invalid or expired refresh token");

    var user = await userRepo.GetByIdAsync(stored.UserId, ct);
    if (user is null || !user.IsActive)
        return Results.Problem(statusCode: 401, title: "User not found or deactivated");

    // Revoke old token and issue new pair (refresh token rotation)
    await refreshTokenRepo.RevokeAsync(stored.Id, ct);
    var newTokenPair = tokenService.GenerateTokenPair(user);
    await refreshTokenRepo.SaveAsync(newTokenPair, user.Id, ct);

    return Results.Ok(newTokenPair);
}).AllowAnonymous();
```

## 13. AUTHORIZATION — POLICY, RESOURCE, CLAIMS

### What It Actually Is
Authorization determines what an authenticated user is *allowed* to do. ASP.NET Core's policy-based authorization is more powerful than role-based: a policy is a set of requirements, each evaluated by a handler. Requirements can check roles, claims, custom data, or even make database calls. Resource-based authorization evaluates permissions in the context of a specific resource (e.g., "can this user edit *this* order?").

### How It Works — Mechanically

1. `UseAuthorization()` middleware reads the matched endpoint's authorization metadata.
2. Calls `IAuthorizationService.AuthorizeAsync(user, resource, policyName)`.
3. `AuthorizationService` resolves the policy, extracts its requirements.
4. For each requirement, it finds registered `IAuthorizationHandler<TRequirement>` instances.
5. Each handler calls `context.Succeed(requirement)` or does nothing (fail is the default).
6. If all requirements succeed → 200. If any fail → 403 (authenticated) or 401 (anonymous).

### The Real-World Code

```csharp
// ── Policy definition ─────────────────────────────────────────────────────────
builder.Services.AddAuthorization(options =>
{
    // Simple role-based
    options.AddPolicy("AdminOnly", policy =>
        policy.RequireRole("Admin"));

    // Claims-based
    options.AddPolicy("VerifiedUser", policy =>
        policy.RequireClaim("email_verified", "true"));

    // Combined
    options.AddPolicy("CanPublishContent", policy =>
        policy
            .RequireAuthenticatedUser()
            .RequireRole("Editor", "Admin")
            .RequireClaim("subscription", "pro", "enterprise")
            .AddRequirements(new MinimumAccountAgeRequirement(TimeSpan.FromDays(30))));

    // Default policy — applied when [Authorize] has no policy name
    options.DefaultPolicy = new AuthorizationPolicyBuilder()
        .RequireAuthenticatedUser()
        .Build();

    // Fallback policy — applied to endpoints that have no [Authorize] or [AllowAnonymous]
    // WARNING: Setting this secures ALL endpoints by default — a secure-by-default posture
    options.FallbackPolicy = new AuthorizationPolicyBuilder()
        .RequireAuthenticatedUser()
        .Build();
});


// ── Custom requirement and handler ────────────────────────────────────────────
public record MinimumAccountAgeRequirement(TimeSpan MinimumAge) : IAuthorizationRequirement;

public sealed class MinimumAccountAgeHandler
    : AuthorizationHandler<MinimumAccountAgeRequirement>
{
    protected override Task HandleRequirementAsync(
        AuthorizationHandlerContext context,
        MinimumAccountAgeRequirement requirement)
    {
        var registeredAtClaim = context.User.FindFirstValue("registered_at");

        if (registeredAtClaim is null ||
            !DateTimeOffset.TryParse(registeredAtClaim, out var registeredAt))
        {
            // Don't call Fail() — just don't succeed, allowing other handlers to decide
            return Task.CompletedTask;
        }

        if (DateTimeOffset.UtcNow - registeredAt >= requirement.MinimumAge)
            context.Succeed(requirement);

        return Task.CompletedTask;
    }
}

builder.Services.AddSingleton<IAuthorizationHandler, MinimumAccountAgeHandler>();


// ── Resource-based authorization ──────────────────────────────────────────────
// "Can this user edit THIS order?" — authorization depends on the specific order

public record EditOrderRequirement : IAuthorizationRequirement;

public sealed class EditOrderHandler : AuthorizationHandler<EditOrderRequirement, Order>
{
    protected override Task HandleRequirementAsync(
        AuthorizationHandlerContext context,
        EditOrderRequirement requirement,
        Order order) // The specific resource being accessed
    {
        var userId = context.User.FindFirstValue(ClaimTypes.NameIdentifier);

        // Admin can edit any order
        if (context.User.IsInRole("Admin"))
        {
            context.Succeed(requirement);
            return Task.CompletedTask;
        }

        // Users can only edit their own orders, and only if not yet shipped
        if (order.CustomerId.ToString() == userId && order.Status < OrderStatus.Shipped)
            context.Succeed(requirement);

        return Task.CompletedTask;
    }
}

// Use in endpoint
app.MapPut("/api/orders/{id:int}", async (
    int id,
    UpdateOrderRequest request,
    IOrderRepository repo,
    IAuthorizationService authz,
    HttpContext http,
    CancellationToken ct) =>
{
    var order = await repo.GetByIdAsync(id, ct);
    if (order is null) return Results.NotFound();

    // Resource-based check: pass the actual resource
    var result = await authz.AuthorizeAsync(http.User, order, new EditOrderRequirement());
    if (!result.Succeeded)
        return Results.Forbid();

    await repo.UpdateAsync(order.ApplyUpdate(request), ct);
    return Results.NoContent();
});
```

## 14. EXCEPTION HANDLING & PROBLEM DETAILS

### What It Actually Is
Exception handling in ASP.NET Core is the mechanism for converting unhandled exceptions into HTTP responses. In production you never want stack traces leaking to clients. ASP.NET Core 8 introduced `IExceptionHandler` — a chain of handlers tried in registration order, each getting the chance to handle specific exception types. Problem Details (RFC 7807) is the standard JSON format for API error responses: `{ type, title, status, detail, instance }`.

### How It Works — Mechanically

1. `UseExceptionHandler()` middleware wraps the entire pipeline in a try/catch.
2. When an unhandled exception propagates out of any downstream middleware, `UseExceptionHandler` catches it.
3. It iterates through registered `IExceptionHandler` implementations in registration order.
4. Each handler's `TryHandleAsync` returns `true` if it handled the exception, `false` to pass to next.
5. If no handler claims it, the default problem details response writer kicks in (if `AddProblemDetails()` is registered).

### The Real-World Code

```csharp
// ── Registration ──────────────────────────────────────────────────────────────
builder.Services.AddProblemDetails(options =>
{
    // Customize the default problem details for unhandled exceptions
    options.CustomizeProblemDetails = ctx =>
    {
        ctx.ProblemDetails.Extensions["traceId"] =
            Activity.Current?.Id ?? ctx.HttpContext.TraceIdentifier;
        ctx.ProblemDetails.Extensions["timestamp"] =
            DateTimeOffset.UtcNow;

        // Don't include exception details in production
        if (ctx.HttpContext.RequestServices
            .GetRequiredService<IHostEnvironment>()
            .IsProduction())
        {
            ctx.ProblemDetails.Detail = null;
        }
    };
});

builder.Services.AddExceptionHandler<ValidationExceptionHandler>();
builder.Services.AddExceptionHandler<DomainExceptionHandler>();
builder.Services.AddExceptionHandler<NotFoundExceptionHandler>();
builder.Services.AddExceptionHandler<GlobalExceptionHandler>(); // Always last — catches all

app.UseExceptionHandler(); // Activates the IExceptionHandler chain


// ── Exception handlers — specific to general ─────────────────────────────────
public sealed class ValidationExceptionHandler : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext context,
        Exception exception,
        CancellationToken ct)
    {
        // FluentValidation ValidationException
        if (exception is not FluentValidation.ValidationException validationEx)
            return false;

        var errors = validationEx.Errors
            .GroupBy(e => e.PropertyName)
            .ToDictionary(
                g => g.Key,
                g => g.Select(e => e.ErrorMessage).ToArray());

        context.Response.StatusCode = StatusCodes.Status400BadRequest;
        await context.Response.WriteAsJsonAsync(
            new HttpValidationProblemDetails(errors)
            {
                Type = "https://tools.ietf.org/html/rfc7807",
                Title = "Validation failed",
                Status = StatusCodes.Status400BadRequest
            }, ct);

        return true;
    }
}

public sealed class DomainExceptionHandler : IExceptionHandler
{
    private readonly ILogger<DomainExceptionHandler> _logger;
    public DomainExceptionHandler(ILogger<DomainExceptionHandler> logger)
        => _logger = logger;

    public async ValueTask<bool> TryHandleAsync(
        HttpContext context, Exception exception, CancellationToken ct)
    {
        if (exception is not DomainException domainEx) return false;

        // Domain exceptions are business rule violations — warning level, not error
        _logger.LogWarning(domainEx, "Domain rule violation: {Message}", domainEx.Message);

        context.Response.StatusCode = StatusCodes.Status422UnprocessableEntity;
        await context.Response.WriteAsJsonAsync(new ProblemDetails
        {
            Status = 422,
            Title = "Business rule violation",
            Detail = domainEx.Message,
            Type = $"https://api.example.com/errors/{domainEx.ErrorCode}",
            Extensions = { ["errorCode"] = domainEx.ErrorCode }
        }, ct);

        return true;
    }
}

public sealed class NotFoundExceptionHandler : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext context, Exception exception, CancellationToken ct)
    {
        if (exception is not NotFoundException notFoundEx) return false;

        context.Response.StatusCode = StatusCodes.Status404NotFound;
        await context.Response.WriteAsJsonAsync(new ProblemDetails
        {
            Status = 404,
            Title = "Resource not found",
            Detail = notFoundEx.Message
        }, ct);

        return true;
    }
}

public sealed class GlobalExceptionHandler : IExceptionHandler
{
    private readonly ILogger<GlobalExceptionHandler> _logger;
    public GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger)
        => _logger = logger;

    public async ValueTask<bool> TryHandleAsync(
        HttpContext context, Exception exception, CancellationToken ct)
    {
        // Log at Error level — this is unexpected
        _logger.LogError(exception,
            "Unhandled exception for {Method} {Path}",
            context.Request.Method,
            context.Request.Path);

        context.Response.StatusCode = StatusCodes.Status500InternalServerError;
        await context.Response.WriteAsJsonAsync(new ProblemDetails
        {
            Status = 500,
            Title = "An unexpected error occurred",
            Type = "https://tools.ietf.org/html/rfc7231#section-6.6.1",
            Extensions =
            {
                ["traceId"] = context.TraceIdentifier
            }
        }, ct);

        return true; // Always claim this exception — it's the catch-all
    }
}
```

## 15. VALIDATION

### What It Actually Is
Validation in ASP.NET Core has two layers: Data Annotations (declarative, attribute-based, works for simple rules) and FluentValidation (imperative, class-based, handles complex cross-property and async rules). With `[ApiController]`, Data Annotations validation runs automatically before the action method — returning 400 if `ModelState` is invalid. FluentValidation integrates the same way via its ASP.NET Core adapter.

## 16. IHTTPCLIENTFACTORY & HTTPCLIENT PATTERNS

### What It Actually Is
`IHttpClientFactory` manages the lifetime of `HttpMessageHandler` instances — the actual TCP socket pool under `HttpClient`. If you `new HttpClient()` yourself, either you leak sockets (disposable not disposed) or you exhaust socket handles (disposing too often, which doesn't release the underlying socket immediately due to `TIME_WAIT`). The factory pools handlers, respects DNS TTL (handlers cycle after 2 minutes by default), and integrates with Polly for resilience.

### How It Works — Mechanically

1. You register named or typed clients via `AddHttpClient()`.
2. Each client name has a pool of `HttpMessageHandler` instances.
3. When `CreateClient(name)` is called, the factory returns an `HttpClient` sharing a pooled handler.
4. `HttpClient` is a lightweight wrapper — cheap to create and throw away. The *handler* is expensive.
5. After 2 minutes (configurable), the handler is expired from the pool — a new one is created, which re-resolves DNS. Old ones are kept alive until all in-flight requests complete.

### The Real-World Code

```csharp
// ── Registration patterns ─────────────────────────────────────────────────────

// Named client
builder.Services.AddHttpClient("payments-api", (sp, client) =>
{
    var config = sp.GetRequiredService<IConfiguration>();
    client.BaseAddress = new Uri(config["PaymentsApi:BaseUrl"]!);
    client.DefaultRequestHeaders.Add("Accept", "application/json");
    client.Timeout = TimeSpan.FromSeconds(30);
})
.AddStandardResilienceHandler(); // .NET 8: built-in retry + circuit breaker


// Typed client — the preferred pattern for complex clients
builder.Services.AddHttpClient<IPaymentsApiClient, PaymentsApiClient>((sp, client) =>
{
    client.BaseAddress = new Uri(sp.GetRequiredService<IOptions<PaymentsOptions>>().Value.BaseUrl);
    client.Timeout = TimeSpan.FromSeconds(30);
})
.AddResilienceHandler("payments", (pipeline, ctx) =>
{
    pipeline
        .AddRetry(new HttpRetryStrategyOptions
        {
            MaxRetryAttempts = 3,
            BackoffType = DelayBackoffType.Exponential,
            UseJitter = true,
            // Retry on transient errors and 429/503
            ShouldHandle = args => ValueTask.FromResult(
                args.Outcome.Exception is HttpRequestException ||
                args.Outcome.Result?.StatusCode
                    is HttpStatusCode.TooManyRequests
                    or HttpStatusCode.ServiceUnavailable)
        })
        .AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions
        {
            FailureRatio = 0.5,           // Open when 50% of calls fail
            SamplingDuration = TimeSpan.FromSeconds(10),
            MinimumThroughput = 10,        // Need 10 calls before evaluating
            BreakDuration = TimeSpan.FromSeconds(30)
        })
        .AddTimeout(TimeSpan.FromSeconds(10));
});
```

## 17. BACKGROUND SERVICES

### What It Actually Is
Background services are `IHostedService` implementations that run concurrently with the HTTP server for the lifetime of the application. They're registered with the DI container and started/stopped by the host. `BackgroundService` is the abstract base class that simplifies the implementation to a single `ExecuteAsync` method. Common uses: outbox processing, cache warm-up, scheduled jobs, message queue consumers.

### How It Works — Mechanically

1. All `IHostedService` implementations are registered via `AddHostedService<T>()`.
2. On `app.Run()`, the host calls `StartAsync()` on each service **in registration order** and awaits each call before starting the next.
3. Each service should return from `StartAsync` quickly — start background work via `Task.Run` or by returning to the `ExecuteAsync` loop without blocking.
4. On shutdown, the host calls `StopAsync(CancellationToken)` on each service in **reverse registration order**, passing a token that fires after `ShutdownTimeout`.

### The Real-World Code

```csharp
// ── Outbox processor — the canonical background service pattern ───────────────
public sealed class OutboxProcessorService : BackgroundService
{
    private readonly IServiceScopeFactory _scopeFactory;
    private readonly ILogger<OutboxProcessorService> _logger;

    // BackgroundService is a singleton — NEVER inject scoped services directly.
    // Use IServiceScopeFactory to create scopes per execution.
    public OutboxProcessorService(
        IServiceScopeFactory scopeFactory,
        ILogger<OutboxProcessorService> logger)
    {
        _scopeFactory = scopeFactory;
        _logger = logger;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        _logger.LogInformation("Outbox processor started");

        // PeriodicTimer is cleaner than Task.Delay in a loop — handles drift better
        using var timer = new PeriodicTimer(TimeSpan.FromSeconds(5));

        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            try
            {
                await ProcessBatchAsync(stoppingToken);
            }
            catch (OperationCanceledException) when (stoppingToken.IsCancellationRequested)
            {
                // Graceful shutdown — don't log as error
                break;
            }
            catch (Exception ex)
            {
                // Don't crash the service — log and continue. Next tick will retry.
                _logger.LogError(ex, "Error processing outbox batch");
            }
        }

        _logger.LogInformation("Outbox processor stopped");
    }

    private async Task ProcessBatchAsync(CancellationToken ct)
    {
        // New scope per batch = fresh DbContext, clean transaction boundary
        await using var scope = _scopeFactory.CreateAsyncScope();
        var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
        var publisher = scope.ServiceProvider.GetRequiredService<IMessageBus>();

        // Pessimistic lock: SKIP LOCKED prevents competing instances from picking same rows
        var messages = await db.OutboxMessages
            .FromSqlRaw("""
                SELECT TOP 50 * FROM OutboxMessages
                WHERE ProcessedAt IS NULL
                ORDER BY CreatedAt
            """)
            .ToListAsync(ct);

        if (messages.Count == 0) return;

        _logger.LogDebug("Processing {Count} outbox messages", messages.Count);

        foreach (var message in messages)
        {
            try
            {
                await publisher.PublishAsync(message.EventType, message.Payload, ct);
                message.ProcessedAt = DateTime.UtcNow;
            }
            catch (Exception ex)
            {
                message.RetryCount++;
                message.LastError = ex.Message;
                _logger.LogWarning(ex, "Failed to publish {EventType}, retry {Count}",
                    message.EventType, message.RetryCount);
            }
        }

        await db.SaveChangesAsync(ct);
    }
}
```

## 18. HEALTH CHECKS

### What It Actually Is
Health checks are HTTP endpoints that report the status of your application and its dependencies. Kubernetes, load balancers, and monitoring systems hit these endpoints to decide whether to route traffic to your instance. **Liveness** (is the process alive?) and **Readiness** (is it ready to handle traffic?) are two distinct concerns and should be separate endpoints. Confusing them causes rolling restarts to kill pods during startup.

### The Real-World Code

```csharp
// ── Registration ──────────────────────────────────────────────────────────────
builder.Services.AddHealthChecks()
    .AddSqlServer(
        connectionString: builder.Configuration.GetConnectionString("Default")!,
        name: "sql-server",
        failureStatus: HealthStatus.Unhealthy,
        tags: new[] { "ready", "db" })
    .AddRedis(
        connectionMultiplexerFactory: sp =>
            sp.GetRequiredService<IConnectionMultiplexer>(),
        name: "redis",
        tags: new[] { "ready", "cache" })
    .AddCheck<ExternalPaymentApiHealthCheck>(
        name: "payments-api",
        tags: new[] { "ready", "external" })
    .AddCheck<DiskSpaceHealthCheck>(
        name: "disk-space",
        tags: new[] { "live" });

// Liveness: is the process alive? Check only things that can't recover (disk space)
app.MapHealthChecks("/health/live", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("live"),
    ResponseWriter = WriteJsonResponse
}).AllowAnonymous().RequireHost("*:8080"); // Only accessible on internal port

// Readiness: can we handle traffic? Check all dependencies
app.MapHealthChecks("/health/ready", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("ready"),
    ResponseWriter = WriteJsonResponse
}).AllowAnonymous().RequireHost("*:8080");


// ── Custom health check ───────────────────────────────────────────────────────
public sealed class ExternalPaymentApiHealthCheck : IHealthCheck
{
    private readonly IPaymentsApiClient _client;
    private readonly ILogger<ExternalPaymentApiHealthCheck> _logger;

    public ExternalPaymentApiHealthCheck(
        IPaymentsApiClient client,
        ILogger<ExternalPaymentApiHealthCheck> logger)
    {
        _client = client;
        _logger = logger;
    }

    public async Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext context,
        CancellationToken cancellationToken = default)
    {
        try
        {
            // Use a short timeout — health check shouldn't block for long
            using var cts = CancellationTokenSource.CreateLinkedTokenSource(cancellationToken);
            cts.CancelAfter(TimeSpan.FromSeconds(3));

            var isReachable = await _client.PingAsync(cts.Token);

            return isReachable
                ? HealthCheckResult.Healthy("Payments API is reachable")
                : HealthCheckResult.Degraded("Payments API returned unhealthy status");
        }
        catch (OperationCanceledException)
        {
            return HealthCheckResult.Degraded("Payments API health check timed out");
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Payments API health check threw exception");
            return HealthCheckResult.Unhealthy("Payments API is unreachable", ex);
        }
    }
}


// ── JSON response writer ──────────────────────────────────────────────────────
static Task WriteJsonResponse(HttpContext ctx, HealthReport report)
{
    ctx.Response.ContentType = "application/json";

    var response = new
    {
        status = report.Status.ToString(),
        duration = report.TotalDuration.TotalMilliseconds,
        entries = report.Entries.ToDictionary(
            kvp => kvp.Key,
            kvp => new
            {
                status = kvp.Value.Status.ToString(),
                duration = kvp.Value.Duration.TotalMilliseconds,
                description = kvp.Value.Description,
                error = kvp.Value.Exception?.Message,
                tags = kvp.Value.Tags
            })
    };

    return ctx.Response.WriteAsJsonAsync(response);
}
```

## 19. RATE LIMITING

### What It Actually Is
Rate limiting controls how many requests a client can make in a time window. ASP.NET Core 7+ has built-in rate limiting middleware supporting four algorithms: Fixed Window (simple but allows burst at window boundary), Sliding Window (smooths the boundary burst), Token Bucket (allows bursts up to bucket capacity), and Concurrency (limits parallel requests, not rate). The right algorithm depends on your traffic pattern.

### How It Works — Mechanically

**Algorithm comparison:**
```
Fixed Window (100 req/min):
  :00 - :59  → 100 requests allowed
  :59        → 90 requests used
  :00        → counter resets — 10 unused requests evaporate, new 100 available
               Problem: 90 req at :59 + 100 req at :00 = 190 req in 2 seconds

Sliding Window (100 req/min, 6 segments):
  Each 10-second segment tracks its own count
  New segment opens → oldest closes, its count is released
  Smooths the boundary burst — max 100 req in any rolling 60-second window

Token Bucket (100 tokens, refill 20/10s):
  Bucket starts with 100 tokens
  Each request costs 1 token
  Allows burst of 100 immediately, then sustained 2 req/sec
  Good for bursty-then-sustained traffic patterns
```

### The Real-World Code

```csharp
// ── Complete rate limiter setup ───────────────────────────────────────────────
builder.Services.AddRateLimiter(options =>
{
    // Global fallback for unannotated endpoints
    options.GlobalLimiter = PartitionedRateLimiter.Create<HttpContext, string>(ctx =>
        RateLimitPartition.GetSlidingWindowLimiter(
            partitionKey: ctx.User.Identity?.IsAuthenticated == true
                ? $"user:{ctx.User.FindFirstValue(ClaimTypes.NameIdentifier)}"
                : $"ip:{ctx.Connection.RemoteIpAddress}",
            factory: _ => new SlidingWindowRateLimiterOptions
            {
                PermitLimit = 100,
                Window = TimeSpan.FromMinutes(1),
                SegmentsPerWindow = 6
            }));

    // Strict limit for expensive endpoints
    options.AddPolicy("strict", ctx =>
        RateLimitPartition.GetFixedWindowLimiter(
            partitionKey: ctx.User.FindFirstValue(ClaimTypes.NameIdentifier) ?? "anonymous",
            factory: _ => new FixedWindowRateLimiterOptions
            {
                PermitLimit = 5,
                Window = TimeSpan.FromMinutes(1),
                QueueProcessingOrder = QueueProcessingOrder.OldestFirst,
                QueueLimit = 2
            }));

    // Auth endpoints — tighter, IP-based (prevent credential stuffing)
    options.AddPolicy("auth", ctx =>
        RateLimitPartition.GetFixedWindowLimiter(
            partitionKey: ctx.Connection.RemoteIpAddress?.ToString() ?? "unknown",
            factory: _ => new FixedWindowRateLimiterOptions
            {
                PermitLimit = 10,
                Window = TimeSpan.FromMinutes(15),
                QueueLimit = 0 // No queueing — fail fast
            }));

    // Customize the 429 response
    options.OnRejected = async (ctx, ct) =>
    {
        ctx.HttpContext.Response.StatusCode = StatusCodes.Status429TooManyRequests;

        if (ctx.Lease.TryGetMetadata(MetadataName.RetryAfter, out var retryAfter))
            ctx.HttpContext.Response.Headers.RetryAfter =
                ((int)retryAfter.TotalSeconds).ToString();

        await ctx.HttpContext.Response.WriteAsJsonAsync(new ProblemDetails
        {
            Status = 429,
            Title = "Too Many Requests",
            Detail = "You have exceeded the rate limit. Please slow down.",
            Extensions = { ["retryAfter"] = retryAfter.TotalSeconds }
        }, ct);
    };
});

app.UseRateLimiter();

// Apply to endpoints
app.MapPost("/api/auth/login", LoginHandler)
    .RequireRateLimiting("auth")
    .AllowAnonymous();

app.MapPost("/api/reports/generate", GenerateReportHandler)
    .RequireRateLimiting("strict");
```

## 20. OUTPUT CACHING

### What It Actually Is
Output caching stores the HTTP response on the server and serves it for subsequent requests without executing the endpoint. Unlike `ResponseCache` (which relies on HTTP headers and client cooperation), output caching is entirely server-side, supports tag-based invalidation, and works even for authenticated responses (with careful policy design). Introduced in .NET 7.

### The Real-World Code

```csharp
// ── Setup with policies ───────────────────────────────────────────────────────
builder.Services.AddOutputCache(options =>
{
    // Default: cache all GET/HEAD responses for 60 seconds
    options.AddBasePolicy(policy => policy.Expire(TimeSpan.FromSeconds(60)));

    // Product listing: longer cache, vary by query, tag for invalidation
    options.AddPolicy("product-list", policy =>
        policy
            .Expire(TimeSpan.FromMinutes(10))
            .SetVaryByQuery("category", "page", "pageSize", "sortBy")
            .Tag("products"));

    // User-specific: vary by auth header so each user gets their own cached response
    options.AddPolicy("user-dashboard", policy =>
        policy
            .Expire(TimeSpan.FromMinutes(2))
            .SetVaryByHeader("Authorization")
            .SetVaryByQuery("*"));

    // No cache: for endpoints that must always be fresh
    options.AddPolicy("no-cache", policy => policy.NoStore());
});

app.UseOutputCache();


// ── Applying to endpoints ─────────────────────────────────────────────────────
app.MapGet("/api/products", GetProducts)
    .CacheOutput("product-list");

app.MapGet("/api/dashboard", GetDashboard)
    .CacheOutput("user-dashboard")
    .RequireAuthorization();

app.MapGet("/api/orders/{id}", GetOrder)
    .CacheOutput("no-cache"); // Orders change frequently


// ── Tag-based invalidation ────────────────────────────────────────────────────
public class ProductService : IProductService
{
    private readonly IProductRepository _repo;
    private readonly IOutputCacheStore _cacheStore;

    public ProductService(IProductRepository repo, IOutputCacheStore cacheStore)
    {
        _repo = repo;
        _cacheStore = cacheStore;
    }

    public async Task UpdateProductAsync(UpdateProductCommand cmd, CancellationToken ct)
    {
        await _repo.UpdateAsync(cmd, ct);

        // Evict all responses tagged with "products" — immediate consistency
        await _cacheStore.EvictByTagAsync("products", ct);
    }
}
```

## 21. RESPONSE COMPRESSION

### What It Actually Is
Response compression reduces the byte size of HTTP responses by compressing them (Brotli, Gzip) before sending to clients that advertise support via `Accept-Encoding`. The server-side middleware in ASP.NET Core handles this transparently. Enable Brotli first (better compression ratio), Gzip as fallback. The BREACH attack warning: enabling compression for HTTPS with secrets in the body can leak them — mitigate with request forgery tokens and avoiding secrets in repeatedly-compressed responses.

### The Real-World Code

```csharp
// ── Registration ──────────────────────────────────────────────────────────────
builder.Services.AddResponseCompression(options =>
{
    // Only enable for HTTPS after understanding BREACH — most APIs are safe
    options.EnableForHttps = true;
    options.Providers.Add<BrotliCompressionProvider>();  // Preferred — better ratio
    options.Providers.Add<GzipCompressionProvider>();    // Fallback
    options.MimeTypes = ResponseCompressionDefaults.MimeTypes.Concat(new[]
    {
        "application/json",
        "application/xml",
        "image/svg+xml",
        "application/javascript",
        "text/plain"
    });
});

builder.Services.Configure<BrotliCompressionProviderOptions>(options =>
    // Optimal is slow, Fastest is ~30% worse compression but microseconds faster
    // For APIs: Fastest is usually the right call (CPU is cheaper than bandwidth)
    options.Level = CompressionLevel.Fastest);

builder.Services.Configure<GzipCompressionProviderOptions>(options =>
    options.Level = CompressionLevel.Fastest);

// Must be registered before static files and routing
app.UseResponseCompression();
```

## 22. CORS

### What It Actually Is
CORS (Cross-Origin Resource Sharing) is the browser security mechanism that blocks JavaScript in a browser from making requests to a different origin (protocol + domain + port) than the page was served from. The server tells the browser which origins are allowed via response headers (`Access-Control-Allow-Origin`, etc.). Your API doesn't enforce CORS — browsers do. CORS doesn't protect APIs from non-browser clients. Misconfiguring CORS (allowing `*` with credentials) is a security vulnerability.

### How It Works — Mechanically

**Preflight (OPTIONS request):**
1. Browser sends `OPTIONS /api/resource` with `Origin: https://app.example.com` and `Access-Control-Request-Method: POST`.
2. Server's CORS middleware responds with `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`, `Access-Control-Max-Age`.
3. If allowed, browser sends the actual POST request.

**Simple request (no preflight):**
- GET/POST with only `application/x-www-form-urlencoded` or `text/plain` body.
- Browser sends `Origin` header directly.
- Server responds with `Access-Control-Allow-Origin`.

### The Real-World Code

```csharp
// ── CORS configuration ────────────────────────────────────────────────────────
builder.Services.AddCors(options =>
{
    // Default policy for most API endpoints
    options.AddDefaultPolicy(policy =>
        policy
            .WithOrigins(
                "https://app.example.com",
                "https://admin.example.com")
            .WithMethods("GET", "POST", "PUT", "DELETE", "PATCH")
            .WithHeaders(
                HeaderNames.Authorization,
                HeaderNames.ContentType,
                "X-Correlation-Id")
            .WithExposedHeaders(
                "X-Total-Count",
                "X-Correlation-Id",
                HeaderNames.Location)
            .AllowCredentials()
            .SetPreflightMaxAge(TimeSpan.FromHours(1)));

    // Development policy — permissive for local development
    options.AddPolicy("Development", policy =>
        policy
            .WithOrigins("http://localhost:3000", "http://localhost:5173")
            .AllowAnyMethod()
            .AllowAnyHeader()
            .AllowCredentials());

    // Webhook policy — allow any origin but no credentials (3rd party webhooks)
    options.AddPolicy("Webhooks", policy =>
        policy
            .AllowAnyOrigin()
            .WithMethods("POST")
            .WithHeaders(HeaderNames.ContentType));
});

// Apply after UseRouting, before UseAuthentication
app.UseCors();

// Endpoint-specific CORS
app.MapPost("/webhooks/stripe", HandleStripeWebhook)
    .RequireCors("Webhooks")
    .AllowAnonymous();

// In development, apply the permissive policy
if (app.Environment.IsDevelopment())
{
    app.UseCors("Development");
}
```

## 23. SIGNALR

### What It Actually Is
SignalR is ASP.NET Core's real-time communication library. It abstracts WebSocket, Server-Sent Events, and Long Polling into a unified API, automatically selecting the best transport for the client. The Hub is the server-side class that clients connect to and call methods on — and the server can call methods on connected clients. The scale-out problem: multiple server instances can't share connection state, so you need a backplane (Redis, Azure SignalR Service).

### The Real-World Code

```csharp
// ── Hub definition ────────────────────────────────────────────────────────────
[Authorize]
public class NotificationHub : Hub<INotificationClient>
{
    private readonly IConnectionManager _connections;
    private readonly ILogger<NotificationHub> _logger;

    public NotificationHub(IConnectionManager connections, ILogger<NotificationHub> logger)
    {
        _connections = connections;
        _logger = logger;
    }

    public override async Task OnConnectedAsync()
    {
        var userId = Context.UserIdentifier!; // ClaimTypes.NameIdentifier claim

        // Add to user-specific group for targeted notifications
        await Groups.AddToGroupAsync(Context.ConnectionId, $"user:{userId}");

        // Add to tenant group for broadcast notifications
        var tenantId = Context.User!.FindFirstValue("tid")!;
        await Groups.AddToGroupAsync(Context.ConnectionId, $"tenant:{tenantId}");

        _logger.LogInformation("User {UserId} connected via {ConnectionId}",
            userId, Context.ConnectionId);

        await base.OnConnectedAsync();
    }

    public override async Task OnDisconnectedAsync(Exception? exception)
    {
        var userId = Context.UserIdentifier!;
        _logger.LogInformation("User {UserId} disconnected", userId);
        await base.OnDisconnectedAsync(exception);
    }

    // Client calls this to join a specific channel
    public async Task JoinOrderUpdates(int orderId)
    {
        // Verify the user owns or can view this order
        await Groups.AddToGroupAsync(Context.ConnectionId, $"order:{orderId}");
    }

    public async Task LeaveOrderUpdates(int orderId)
    {
        await Groups.RemoveFromGroupAsync(Context.ConnectionId, $"order:{orderId}");
    }
}

// Strongly-typed client interface
public interface INotificationClient
{
    Task ReceiveNotification(NotificationDto notification);
    Task OrderStatusChanged(int orderId, string newStatus);
    Task ReceiveError(string message);
}


// ── Registration ──────────────────────────────────────────────────────────────
builder.Services.AddSignalR(options =>
{
    options.ClientTimeoutInterval = TimeSpan.FromSeconds(60);
    options.KeepAliveInterval = TimeSpan.FromSeconds(15);
    options.HandshakeTimeout = TimeSpan.FromSeconds(15);
    options.MaximumReceiveMessageSize = 32 * 1024; // 32 KB
})
.AddStackExchangeRedis(builder.Configuration["Redis:ConnectionString"]!,
    options => options.Configuration.ChannelPrefix = RedisChannel.Literal("signalr"));

app.MapHub<NotificationHub>("/hubs/notifications");


// ── Sending from a service (outside the hub) ──────────────────────────────────
public class OrderStatusService : IOrderStatusService
{
    private readonly IHubContext<NotificationHub, INotificationClient> _hubContext;

    public OrderStatusService(IHubContext<NotificationHub, INotificationClient> hubContext)
        => _hubContext = hubContext;

    public async Task UpdateOrderStatusAsync(int orderId, OrderStatus newStatus, int customerId)
    {
        // Notify everyone watching this specific order
        await _hubContext.Clients
            .Group($"order:{orderId}")
            .OrderStatusChanged(orderId, newStatus.ToString());

        // Also notify the customer in their personal group
        await _hubContext.Clients
            .User(customerId.ToString())
            .ReceiveNotification(new NotificationDto
            {
                Title = "Order Update",
                Message = $"Your order #{orderId} is now {newStatus}",
                Type = NotificationType.OrderStatus
            });
    }
}
```

## 24. LOGGING & STRUCTURED LOGS

### What It Actually Is
ASP.NET Core uses `ILogger<T>` as its logging abstraction — the actual sink (where logs go) is determined by the provider registered at startup. `Serilog` is the de-facto production logging library: it writes structured JSON logs, supports sinks (Seq, Elasticsearch, Application Insights, files), and enriches logs automatically. Structured logging means log arguments are stored as properties, not just string-interpolated — enabling powerful querying.

### The Real-World Code

```csharp
// ── NuGet packages ────────────────────────────────────────────────────────────
// Serilog.AspNetCore
// Serilog.Sinks.Console
// Serilog.Sinks.File
// Serilog.Sinks.Seq
// Serilog.Enrichers.Environment
// Serilog.Enrichers.Process
// Serilog.Enrichers.Thread


// ── Program.cs ────────────────────────────────────────────────────────────────

    .WriteTo.Console()
    .CreateBootstrapLogger();

try
{
    var builder = WebApplication.CreateBuilder(args);

    builder.Host.UseSerilog((context, services, config) =>
        config
            .ReadFrom.Configuration(context.Configuration) // appsettings.json
            .ReadFrom.Services(services)                    // DI-registered sinks
            .Enrich.FromLogContext()                         // ILogger.BeginScope data
            .Enrich.WithMachineName()
            .Enrich.WithProcessId()
            .Enrich.WithThreadId()
            .Enrich.WithProperty("Application", builder.Environment.ApplicationName)
            .Enrich.WithProperty("Environment", builder.Environment.EnvironmentName)
            .WriteTo.Console(new RenderedCompactJsonFormatter()) // JSON to console
```

## 25. API VERSIONING

### What It Actually Is
API versioning lets you evolve your API without breaking existing clients. Clients specify the version they want (URL segment, query string, or header), and ASP.NET Core routes the request to the matching controller or endpoint group. The `Asp.Versioning.Http` package provides the plumbing.

### The Real-World Code

```csharp
// ── NuGet: Asp.Versioning.Mvc, Asp.Versioning.Mvc.ApiExplorer ────────────────

builder.Services.AddApiVersioning(options =>
{

    options.AssumeDefaultVersionWhenUnspecified = true;

    options.ApiVersionReader = ApiVersionReader.Combine(
        new UrlSegmentApiVersionReader(),    // /api/v1/orders
        new HeaderApiVersionReader("X-Api-Version"), // X-Api-Version: 1.0
        new QueryStringApiVersionReader("api-version")); // ?api-version=1.0
})
.AddMvc()
.AddApiExplorer(options =>
{
    options.GroupNameFormat = "'v'VVV"; // v1, v2
    options.SubstituteApiVersionInUrl = true;
});
```

## 27. TESTING ASP.NET CORE

### What It Actually Is
`WebApplicationFactory<TProgram>` creates an in-memory test server running your actual ASP.NET Core application — complete with the real middleware pipeline, DI container, and routing — but without a network connection. You call it with `HttpClient` and it processes requests in-process. This means your integration tests actually test the full stack (except the real database, which you substitute with SQLite/TestContainers).

### The Real-World Code

```csharp
// ── Base integration test fixture ─────────────────────────────────────────────
public class ApiFactory : WebApplicationFactory<Program>, IAsyncLifetime
{
    // Use TestContainers for a real PostgreSQL instead of SQLite
    // (SQLite has different behavior — FK enforcement, JSON functions, etc.)
    private readonly PostgreSqlContainer _postgres = new PostgreSqlBuilder()
        .WithImage("postgres:16-alpine")
        .WithDatabase("testdb")
        .Build();

    protected override void ConfigureWebHost(IWebHostBuilder builder)
    {
        builder.ConfigureAppConfiguration(config =>
        {
            // Override config for tests
            config.AddInMemoryCollection(new Dictionary<string, string?>
            {
                ["ConnectionStrings:Default"] = _postgres.GetConnectionString(),
                ["Jwt:Secret"] = "test-secret-key-minimum-32-characters!!",
                ["Features:EmailEnabled"] = "false"
            });
        });

        builder.ConfigureTestServices(services =>
        {
            // Replace external HTTP clients with mocks
            services.RemoveAll<IPaymentsApiClient>();
            services.AddSingleton<IPaymentsApiClient, FakePaymentsApiClient>();

            // Replace email service — don't actually send emails in tests
            services.RemoveAll<IEmailService>();
            services.AddSingleton<IEmailService, FakeEmailService>();
        });
    }

    public async Task InitializeAsync()
    {
        await _postgres.StartAsync();

        // Apply migrations to the test database
        using var scope = Services.CreateScope();
        var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
        await db.Database.MigrateAsync();
    }

    public new async Task DisposeAsync()
    {
        await _postgres.StopAsync();
    }

    // Helper to create an authenticated client
    public HttpClient CreateAuthenticatedClient(string role = "User", string? userId = null)
    {
        var client = CreateClient();
        var token = GenerateTestToken(userId ?? Guid.NewGuid().ToString(), role);
        client.DefaultRequestHeaders.Authorization =
            new AuthenticationHeaderValue("Bearer", token);
        return client;
    }

    private static string GenerateTestToken(string userId, string role)
    {
        var claims = new[]
        {
            new Claim(ClaimTypes.NameIdentifier, userId),
            new Claim(ClaimTypes.Role, role),
            new Claim("tid", "test-tenant")
        };

        var key = new SymmetricSecurityKey(
            Encoding.UTF8.GetBytes("test-secret-key-minimum-32-characters!!"));
        var token = new JwtSecurityToken(
            issuer: "test-issuer",
            audience: "test-audience",
            claims: claims,
            expires: DateTime.UtcNow.AddHours(1),
            signingCredentials: new SigningCredentials(key, SecurityAlgorithms.HmacSha256));

        return new JwtSecurityTokenHandler().WriteToken(token);
    }
}
```

## 28. PROGRAM.CS — COMPLETE BOOTSTRAPPING

### What It Actually Is
`Program.cs` is the application's entry point and composition root. It wires together all infrastructure: server, DI container, configuration, middleware. In .NET 6+, there's no `Startup.cs` — everything is in `Program.cs`. Good architecture keeps `Program.cs` thin by using extension methods to group related registrations. This file is also the seam for test factories.

### The Real-World Code

```csharp
// ── Complete production Program.cs ────────────────────────────────────────────

using Serilog;

// Bootstrap logger before anything else — captures startup errors
Log.Logger = new LoggerConfiguration()
    .WriteTo.Console()
    .CreateBootstrapLogger();

try
{
    var builder = WebApplication.CreateBuilder(args);

    // ── Configuration ─────────────────────────────────────────────────────────
    builder.Configuration
        .AddJsonFile("appsettings.json", optional: false)
        .AddJsonFile($"appsettings.{builder.Environment.EnvironmentName}.json", optional: true)
        .AddEnvironmentVariables("MYAPP_");

    if (builder.Environment.IsProduction())
        builder.Configuration.AddAzureKeyVault(
            new Uri(builder.Configuration["KeyVault:Uri"]!),
            new DefaultAzureCredential());

    // ── Logging ───────────────────────────────────────────────────────────────
    builder.Host.UseSerilog((ctx, services, config) =>
        config.ReadFrom.Configuration(ctx.Configuration)
              .ReadFrom.Services(services)
              .Enrich.FromLogContext()
              .Enrich.WithMachineName()
              .Enrich.WithProperty("Version", Assembly.GetExecutingAssembly().GetName().Version?.ToString()));

    // ── Services ──────────────────────────────────────────────────────────────
    builder.Services
        .AddInfrastructure(builder.Configuration)    // DB, Redis, external services
        .AddApplication()                            // MediatR, validators, domain services
        .AddApiServices(builder.Configuration)       // Auth, CORS, rate limiting, versioning
        .AddHealthChecks(builder.Configuration)
        .AddHostedService<OutboxProcessorService>();

    builder.Services.Configure<HostOptions>(o =>
        o.ShutdownTimeout = TimeSpan.FromSeconds(30));

    // ── Build ─────────────────────────────────────────────────────────────────
    var app = builder.Build();

    // ── Middleware pipeline ───────────────────────────────────────────────────
    app.UseExceptionHandler();
    app.UseCorrelationId();
    app.UseSerilogRequestLogging();

    if (!app.Environment.IsDevelopment())
        app.UseHsts();

    app.UseHttpsRedirection();
    app.UseResponseCompression();
    app.UseStaticFiles();
    app.UseRouting();
    app.UseCors();
    app.UseAuthentication();
    app.UseAuthorization();
    app.UseOutputCache();
    app.UseRateLimiter();

    // ── Endpoints ─────────────────────────────────────────────────────────────
    app.MapControllers();
    app.MapOrderEndpoints();
    app.MapProductEndpoints();
    app.MapHealthEndpoints();

    if (app.Environment.IsDevelopment())
    {
        app.MapOpenApi();
        app.UseSwaggerUI(c => c.SwaggerEndpoint("/openapi/v1.json", "API v1"));
    }

    // ── Run ───────────────────────────────────────────────────────────────────
    await app.RunAsync();
    return 0;
}
catch (Exception ex) when (ex is not HostAbortedException)
{
    Log.Fatal(ex, "Application failed to start");
    return 1;
}
finally
{
    await Log.CloseAndFlushAsync();
}
```

## QUICK REFERENCE — GOTCHAS & RULES

| Rule | Why |
|---|---|
| `UseAuthentication` before `UseAuthorization` | Auth populates `User`; Authz reads `User` |
| `UseRouting` before `UseAuthorization` | Authz needs matched endpoint's policy metadata |
| Never inject `IConfiguration` into domain services | Use `IOptions<T>` — it's typed, validated, testable |
| Never `new HttpClient()` in loops | Socket exhaustion. Use `IHttpClientFactory` always |
| `BackgroundService` is singleton | Never inject scoped services. Use `IServiceScopeFactory` |
| `IOptions<T>` vs `IOptionsSnapshot<T>` | `IOptions` = frozen at startup. `IOptionsSnapshot` = per-request (hot reload) |
| `[ApiController]` changes defaults | Auto-400 on invalid ModelState, binding source inference |
| `app.Map` vs `app.UseWhen` | `Map` branches and DOES NOT rejoin. `UseWhen` branches and DOES rejoin |
| `CancellationToken` everywhere | Enables graceful shutdown. Not passing it = requests that won't stop on pod kill |
| `ClockSkew` default is 5 minutes | JWT tokens valid 5 min after expiry. Set to 30s for security |
| Health: liveness ≠ readiness | Liveness = process alive. Readiness = dependencies ready. Never combine |
| CORS is browser enforcement | CORS headers don't protect APIs from non-browser clients (Postman, curl) |
