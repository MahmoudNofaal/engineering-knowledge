# CATEGORY 7 — DESIGN PRINCIPLES

---

## 7.01 — SOLID

**S — Single Responsibility**: One reason to change = one responsibility.

**O — Open/Closed**: Open for extension (new implementations), closed for modification (existing code).
```csharp
// Add new discount types without changing OrderService
public interface IDiscountStrategy { decimal Calculate(Order order); }
public class VipDiscount : IDiscountStrategy { ... }
public class MemberDiscount : IDiscountStrategy { ... }
```

**L — Liskov Substitution**: Subtypes must be substitutable without breaking behavior. Square/Rectangle is the classic violation.

**I — Interface Segregation**: Prefer many small interfaces over one large one. Don't force clients to implement methods they don't need.

**D — Dependency Inversion**: Depend on abstractions (interfaces), not concretions. Inject dependencies.
```csharp
public class OrderService
{
    private readonly IOrderRepository _repository; // abstraction
    public OrderService(IOrderRepository repository) => _repository = repository;
}
```

## 7.02 — DRY, KISS, YAGNI

**DRY**: Every piece of knowledge has one authoritative representation. About knowledge duplication, not code duplication.

**KISS**: Prefer the simplest solution. Complexity has a cost paid by everyone reading the code forever.

**YAGNI**: Don't build features until needed. Speculative generality creates unused complexity.

---

## 7.03 — Clean Code

**Naming**: Intention-revealing, pronounceable, searchable. Methods describe what they do (verb phrases). Variables describe what they hold (noun phrases).

**Functions**: Do one thing. One level of abstraction per function. Command-query separation: a function either does something OR returns something, not both.

**Comments**: Explain WHY, not WHAT. Code says what. Comments explain reasoning, constraints, intent.

**Error handling**: Prefer exceptions for exceptional conditions. Use Result<T> pattern for expected failures. Don't use exceptions for control flow.
