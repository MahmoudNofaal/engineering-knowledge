## 79.01 — Twelve-Factor App

**Factor 1: Codebase** — One codebase, tracked in version control, many deploys
**Factor 2: Dependencies** — Explicitly declare and isolate dependencies (`.csproj`, `packages.lock.json`)
**Factor 3: Config** — Store config in environment (not code) — never hardcode; use env vars or secrets manager
**Factor 4: Backing Services** — Treat backing services (database, Redis, SMTP) as attached resources (via config URL)
**Factor 5: Build/Release/Run** — Strictly separate build and run stages; Docker image = the immutable build artifact
**Factor 6: Processes** — Execute as stateless processes; session state in Redis, not in-process memory
**Factor 7: Port binding** — Export services via port; ASP.NET Core binds to PORT env var
**Factor 8: Concurrency** — Scale out via process model (multiple instances), not threads
**Factor 9: Disposability** — Fast startup (< 5 seconds), graceful shutdown (drain requests, cleanup)
**Factor 10: Dev/Prod Parity** — Keep dev, staging, prod as similar as possible (Docker solves most of this)
**Factor 11: Logs** — Treat logs as event streams; write to stdout; aggregator collects them
**Factor 12: Admin Processes** — Run admin tasks (migrations, seed) as one-off processes, same codebase

**Factor 9 in ASP.NET Core — Graceful shutdown**
```csharp
// ASP.NET Core handles SIGTERM automatically when using IHostedService
// The WebApplication.RunAsync() loop exits when SIGTERM received

// For custom shutdown logic:
var lifetime = app.Services.GetRequiredService<IHostApplicationLifetime>();

lifetime.ApplicationStopping.Register(() =>
{
    // Called when SIGTERM received — before process exits
    // Drain queues, complete in-flight requests, close connections
    _logger.LogInformation("Application stopping — draining queues...");
    _backgroundWorker.GracefulStop();
});

// Kubernetes: sends SIGTERM → waits terminationGracePeriodSeconds (default 30s) → SIGKILL
// Configure in k8s:
// terminationGracePeriodSeconds: 60

// Configure in .NET:
builder.WebHost.UseShutdownTimeout(TimeSpan.FromSeconds(30));
```

## 79.02 — Service Mesh Concepts

**What is a service mesh?**
An infrastructure layer for service-to-service communication. Runs as sidecar proxies alongside each service.

Popular: Istio (Envoy proxy), Linkerd, Consul Connect

**What it provides**
```
Mutual TLS (mTLS): All service-to-service traffic encrypted + authenticated
Traffic management: Weighted routing, canary deploys, circuit breaking at proxy level
Observability: Automatic distributed traces, metrics, logs without code changes
Policy enforcement: Authorization, rate limiting at network level
```

**Why it matters**
```
Without service mesh:
  Service A → HTTP → Service B
  Trust: "If it reaches port 8080, it's authorized"
  
With service mesh (mTLS):
  Service A + Envoy → mTLS → Envoy + Service B
  Trust: Certificate-based identity, not network location
  "Service A's certificate proves it is Service A"
  Compromised pod without valid cert = rejected
```

## 79.03 — Kubernetes Patterns

**ConfigMap and Secret usage**
```yaml
# ConfigMap — non-sensitive config
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  ASPNETCORE_ENVIRONMENT: Production
  Logging__LogLevel__Default: Information
  ConnectionStrings__Redis: "redis-service:6379"

---
# Secret — sensitive config (base64 encoded, should be encrypted at rest)
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets
type: Opaque
data:
  ConnectionStrings__Default: <base64 of connection string>
  Jwt__Key: <base64 of JWT signing key>
  
---
# Deployment using both
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      containers:
      - name: api
        envFrom:
        - configMapRef:
            name: app-config
        - secretRef:
            name: app-secrets
```

**Resource limits — essential for stability**
```yaml
resources:
  requests:                # Minimum needed — used for scheduling
    cpu: "250m"            # 250 millicores = 0.25 CPU
    memory: "256Mi"
  limits:                  # Maximum allowed — enforced
    cpu: "1000m"           # 1 CPU
    memory: "512Mi"
    # OOM Killed if memory exceeds limit
    # CPU throttled if CPU exceeds limit
```

**Horizontal Pod Autoscaler**
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70       # Scale up when avg CPU > 70%
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
```
