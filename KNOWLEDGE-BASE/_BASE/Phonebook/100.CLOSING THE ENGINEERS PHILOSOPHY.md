# CATEGORY 100 — CLOSING: THE ENGINEER'S PHILOSOPHY

---

## 100.01 — What Makes a Great Software Engineer

**It is NOT:**
- Knowing every framework
- Memorizing every algorithm
- Having the most lines of code committed

**It IS:**

**Problem solving**
The engineer's core skill: given an ambiguous problem, find a clear, pragmatic solution. Break it down. Measure. Iterate.

**Communication**
Code is communication — with the computer, with your team, with your future self. Every naming decision, every comment, every API design is a communication act.

**Judgment**
Knowing when to add abstraction and when not to. When to optimize and when not to. When to refactor and when to ship. Technical skill without judgment produces over-engineered systems.

**Curiosity**
The best engineers are endlessly curious. They ask "why does this work this way?" They read source code of the tools they use. They follow the rabbit holes.

**Ownership**
Production is real. Users are real. Own the outcomes of your software. When it breaks at 3am, you care. This attitude produces more careful, observable, testable software than any process.

**Continuous learning**
Technology changes. The fundamentals do not change as quickly (algorithms, architecture, communication, judgment). Master the fundamentals. Stay current on the technology.

## 100.02 — The Pragmatic Engineer's Rules

**Rule 1:** Make it work. Make it right. Make it fast. In that order.

**Rule 2:** You can't improve what you don't measure. Measure before optimizing.

**Rule 3:** The best code is the code that doesn't need to exist.

**Rule 4:** Complexity is the enemy. Every abstraction must earn its place.

**Rule 5:** Code is read far more than it is written. Optimize for the reader.

**Rule 6:** Tests are not overhead. They are the safety net that lets you move fast.

**Rule 7:** Distributed is not better. It is just different — and significantly harder. Start simple.

**Rule 8:** Rollback should be faster than rollforward. Design for it.

**Rule 9:** Everything fails. Design for failure before designing for the happy path.

**Rule 10:** The best architecture is the one your team can build, understand, and maintain.

## 100.03 — Daily Habits of Senior Engineers

**Code:**
- Write one test that would have caught the last bug you saw
- Name one thing better than you found it
- Leave one comment that explains WHY, not WHAT

**System:**
- Check the dashboards before someone complains about slowness
- Review one slow query per week
- Read one error log per week end-to-end

**Learning:**
- Read one commit from a codebase you admire per week
- Follow the upstream release notes for your core dependencies
- Teach one concept to a colleague per week (teaching cements understanding)

**Writing:**
- Document decisions when you make them, not after
- Update the README when you change setup steps
- Write the postmortem within 48 hours of an incident

## 100.04 — Resources for Continued Growth

**Books (must-read)**
- Clean Code — Robert C. Martin (code quality fundamentals)
- The Pragmatic Programmer — Hunt & Thomas (career and craft)
- Designing Data-Intensive Applications — Martin Kleppmann (distributed systems)
- Domain-Driven Design — Eric Evans (DDD blue book, dense but foundational)
- Building Microservices — Sam Newman (practical microservices)
- Software Engineering at Google — Winters, Manshreck, Wright (engineering at scale)
- A Philosophy of Software Design — John Ousterhout (complexity management)
- Working Effectively with Legacy Code — Michael Feathers (real-world rescue)
- The Staff Engineer's Path — Tanya Reilly (technical leadership)
- System Design Interview — Alex Xu (Vol 1 & 2, design exercises)

**Online resources**
- Martin Fowler's blog (martinfowler.com) — architecture, patterns, DDD
- .NET Blog (devblogs.microsoft.com/dotnet) — official C# and .NET updates
- High Scalability (highscalability.com) — real-world system architecture case studies
- Architecture Notes (newsletter) — distributed systems patterns
- Increment magazine — engineering culture and practice

**Practice**
- LeetCode / HackerRank — algorithmic thinking
- NeetCode.io — curated 150 problems covering all patterns
- System design exercises: design Twitter, YouTube, Uber, Airbnb
- Contribute to open source: read good code, write good code with feedback
