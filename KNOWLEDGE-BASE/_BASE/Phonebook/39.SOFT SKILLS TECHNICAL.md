## 39.01 — Estimation

**Why estimation is hard**
- Unknown unknowns (things you don't know you don't know)
- Optimism bias (underestimate bad case)
- Scope creep (requirements grow during implementation)
- Interruptions (estimates assume 100% focus)
- Integration surprises (your code + other code = more work)

**Reference class forecasting**
Instead of estimating from scratch, ask "how long did similar tasks take?"

**The Planning Poker method**
Team independently estimate, reveal simultaneously, discuss disagreements.

**Story points vs. time**
Story points measure relative complexity. They absorb team velocity variability. Time estimates require knowing how many hours are in a "day".

**Adding buffer (cynically: "padding")**
```
Optimistic estimate: 3 days (everything goes perfectly)
Realistic estimate:  5 days (normal bumps)
Pessimistic estimate: 10 days (something is unknown or breaks)

Three-point estimate: (O + 4×R + P) / 6 = (3 + 20 + 10) / 6 = 5.5 days

Add explicit buffer for:
- Code review and feedback cycles: +20%
- Testing and bug fixing: +30%
- Integration and deployment: +20%
```

**Communicating estimates**
```
WRONG: "It will take 3 days."
RIGHT: "Based on similar features we've built, I estimate 3–5 days. 
        The uncertainty is around whether the payment gateway has a good 
        test environment — if not, add 2 more days."
```

## 39.02 — Technical Debt

**What it is**
The implied cost of additional rework caused by choosing an easy solution now instead of a better approach that would take longer.

**Types of technical debt**
- **Deliberate, prudent**: "We know we're cutting corners and we'll fix it after launch"
- **Deliberate, reckless**: "We don't have time for proper design"
- **Inadvertent, prudent**: "Now we know what we should have done"
- **Inadvertent, reckless**: "What's layering?" — the team doesn't know better

**Managing debt**
```markdown
# Technical Debt Register

| ID | Area | Description | Impact | Effort | Priority |
|----|------|-------------|--------|--------|----------|
| TD-001 | Auth | JWT tokens not rotated on sensitive operations | Security risk | 2 days | High |
| TD-002 | DB | Missing index on Orders.CustomerId + Status | Performance: 500ms queries | 2 hours | Medium |
| TD-003 | API | Error responses not following RFC 7807 | Inconsistent client experience | 3 days | Low |
| TD-004 | Tests | Integration tests use SQLite instead of SQL Server | Missed SQL-specific bugs | 1 week | Medium |
```

**The "boy scout rule"**
Leave the code slightly better than you found it. Don't do a full refactor on every PR, but fix the small obvious thing while you're in the file.
