## 35.01 — Incident Response

**The runbook**

**Detect** (ideally automated)
- Alert fires: "API error rate > 5% for 3 minutes"
- Or: a user reports an issue

**Acknowledge**
- Someone takes ownership immediately
- Post in #incidents channel: "I'm looking at the elevated error rate alert"
- Set status page to "Investigating"

**Assess impact**
- How many users affected? (% of traffic)
- Which features/endpoints?
- Is it getting worse or stable?
- Can users work around it?

**Investigate**
1. Look at error logs: what errors are occurring?
2. Look at metrics: CPU, memory, latency, error rate — what changed?
3. Look at deployments: did anything deploy recently?
4. Look at dependencies: are external services healthy?

**Mitigate** (stop the bleeding, even if not a full fix)
- Roll back the last deployment
- Increase capacity
- Enable feature flag to disable the broken feature
- Route traffic to a healthy region
- Clear a corrupted cache

**Fix**
- Root cause analysis while mitigation is in place
- Fix the actual bug
- Deploy the fix
- Verify the fix works

**Communicate**
```
[Investigating] 2026-06-01 23:15 UTC — We are investigating elevated error rates 
on the checkout flow. ~15% of checkout attempts are failing. We will update in 15 minutes.

[Identified] 23:32 UTC — We have identified the issue: a database connection pool 
exhaustion caused by a recent deployment increasing connection hold time. Rolling back.

[Resolved] 23:41 UTC — The rollback is complete. Error rates are back to normal. 
All affected users should retry their checkouts.
```

**Post-mortem (blameless)**
```markdown
# Post-Mortem: Checkout Errors — 2026-06-01

## Summary
For 26 minutes (23:15 – 23:41 UTC), ~15% of checkout attempts failed.
Root cause: connection pool exhaustion due to an increase in connection hold 
time introduced in deployment v1.4.2.

## Impact
- 3,400 failed checkout attempts
- ~$12,000 estimated lost revenue
- 0 data loss, 0 security impact

## Timeline
23:10 - v1.4.2 deployed
23:15 - Alert fires: error rate > 5%
23:17 - On-call engineer acknowledges
23:22 - Database team joined the bridge
23:32 - Root cause identified: connection hold time increase
23:35 - Rollback initiated
23:41 - Error rates normalize, all-clear declared

## Root Cause
v1.4.2 introduced a new query that held database connections open during a 
slow external API call. Under load, this exhausted the connection pool.

## Why Wasn't It Caught?
- Load testing didn't include the new code path
- Integration tests used SQLite (which has no connection pool)
- The external API call was mocked in tests (response was instant)

## Action Items
| Action | Owner | Due |
|---|---|---|
| Add connection hold time monitoring alert | @infra | 2026-06-08 |
| Add load test for checkout flow | @qa | 2026-06-15 |
| Use real database in integration tests | @platform | 2026-06-30 |
| Implement connection pool metrics dashboard | @infra | 2026-06-15 |

## What Went Well
- Alert fired quickly (5 minutes into incident)
- Clear communication with users on status page
- Rollback process was fast and effective

## Lessons Learned
External API calls must never be made while holding a database connection.
Introduce a review checklist item for this pattern.
```

## 35.02 — Common Production Issues in .NET

**Memory leaks**
```csharp
// LEAK: Event subscription not removed
public class DashboardService
{
    private readonly IMetricsCollector _collector;
    
    public DashboardService(IMetricsCollector collector)
    {
        _collector = collector;
        _collector.MetricsUpdated += OnMetricsUpdated;  // Subscription
        // DashboardService never unsubscribes — _collector holds reference
        // DashboardService can never be GC'd
    }
}

// FIX: Implement IDisposable
public class DashboardService : IDisposable
{
    public DashboardService(IMetricsCollector collector)
    {
        _collector = collector;
        _collector.MetricsUpdated += OnMetricsUpdated;
    }
    
    public void Dispose()
    {
        _collector.MetricsUpdated -= OnMetricsUpdated;  // Unsubscribe!
    }
}

// LEAK: Static dictionary growing unbounded
public static class Cache
{
    private static readonly Dictionary<string, object> _store = new();  // Never cleared!
    
    public static void Store(string key, object value)
        => _store[key] = value;  // Grows forever
}

// FIX: Use IMemoryCache with eviction policies
```

**Thread pool starvation**
```csharp
// PROBLEM: Blocking async code on the thread pool
public class BadService
{
    public string GetData()
    {
        // Blocks a thread pool thread
        return GetDataAsync().Result;  // DEADLOCK or thread starvation risk
    }
    
    private async Task<string> GetDataAsync() => await FetchFromApiAsync();
}

// Symptoms: Response times increase. Thread pool threads maxed out.
// ThreadPool.GetAvailableThreads() returns near 0.

// FIX: Async all the way
public async Task<string> GetDataAsync()
    => await GetDataInternalAsync();
```

**Connection pool exhaustion**
```csharp
// PROBLEM: Connections not properly returned to pool
using var connection = new SqlConnection(connString);
connection.Open();
var command = new SqlCommand("SELECT * FROM Orders", connection);
var reader = await command.ExecuteReaderAsync();
// Exception thrown here — connection never disposed — stays in "busy" state
// Eventually pool exhausted
// System.InvalidOperationException: Timeout expired. Unable to obtain a connection.

// FIX: Always use using statements
await using var connection = new SqlConnection(connString);
await connection.OpenAsync();
await using var command = new SqlCommand("SELECT * FROM Orders", connection);
await using var reader = await command.ExecuteReaderAsync();
```

**GC pressure from large temporary objects**
```csharp
// PROBLEM: Large temporary strings/arrays allocated in LOH repeatedly
public string BuildReport(IEnumerable<Order> orders)
{
    var report = "";
    foreach (var order in orders)
        report += $"Order {order.Id}: {order.Total}\n";  // N² allocations
    return report;
}

// FIX: StringBuilder for string building, ArrayPool for temporary arrays
public string BuildReport(IEnumerable<Order> orders)
{
    var sb = new StringBuilder();
    foreach (var order in orders)
        sb.AppendLine($"Order {order.Id}: {order.Total}");
    return sb.ToString();
}
```
