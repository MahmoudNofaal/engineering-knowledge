## 42.01 — Owned Entities and Value Objects

**What it is**
Map DDD value objects as owned entities — stored in the parent table without a separate table or identity.

```csharp
// Domain value object
public record Address(string Street, string City, string Country, string PostalCode);

// Entity using value object
public class Customer
{
    public int Id { get; private set; }
    public string Name { get; private set; } = null!;
    public Address BillingAddress { get; private set; } = null!;
    public Address? ShippingAddress { get; private set; }
}

// EF Core configuration
public class CustomerConfiguration : IEntityTypeConfiguration<Customer>
{
    public void Configure(EntityTypeBuilder<Customer> builder)
    {
        builder.OwnsOne(c => c.BillingAddress, addr =>
        {
            addr.Property(a => a.Street).HasColumnName("BillingStreet").HasMaxLength(200);
            addr.Property(a => a.City).HasColumnName("BillingCity").HasMaxLength(100);
            addr.Property(a => a.Country).HasColumnName("BillingCountry").HasMaxLength(100);
            addr.Property(a => a.PostalCode).HasColumnName("BillingPostalCode").HasMaxLength(20);
        });
        
        builder.OwnsOne(c => c.ShippingAddress, addr =>
        {
            addr.Property(a => a.Street).HasColumnName("ShippingStreet").HasMaxLength(200);
            // ...
        });
    }
}

// Stored in Customers table: Id, Name, BillingStreet, BillingCity, ...
// No separate Address table
```

**Strongly typed IDs with value conversions**
```csharp
public record OrderId(int Value);

public class OrderConfiguration : IEntityTypeConfiguration<Order>
{
    public void Configure(EntityTypeBuilder<Order> builder)
    {
        builder.Property(o => o.Id)
            .HasConversion(
                id => id.Value,        // To database: extract int
                value => new OrderId(value))  // From database: wrap in OrderId
            .ValueGeneratedOnAdd();
        
        builder.HasKey(o => o.Id);
    }
}
```

## 42.02 — EF Core Migrations in Production

**Migrations as code**
```bash
dotnet ef migrations add AddOrderShippedDate --project MyApp.Infrastructure --startup-project MyApp.Api
dotnet ef migrations remove                 # Remove last migration (if not applied)
dotnet ef database update                   # Apply all pending migrations
dotnet ef database update 20260101_Init     # Roll back to specific migration
dotnet ef migrations script                 # Generate SQL script for all migrations
dotnet ef migrations script 20260101_Init 20260601_AddColumns  # Script between versions
```

**Applying migrations programmatically (on startup)**
```csharp
// Safe for development and single-instance deployments
// NOT safe for multi-instance deployments without distributed locking
if (app.Environment.IsDevelopment())
{
    using var scope = app.Services.CreateScope();
    var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
    await db.Database.MigrateAsync();
}

// Better: run migrations as a separate job before app starts
// In Kubernetes: InitContainer runs migration job before app pods start
```

**Safe migration practices**
```csharp
// DANGEROUS: Dropping a column that code still references
// migrationBuilder.DropColumn("Orders", "OldStatus");
// If old code is still running → exception on every query

// SAFE: Multi-step migration
// Step 1: Deploy new code that ignores the old column
// Step 2: Wait until all instances run new code
// Step 3: Deploy migration that drops the column

// Adding NOT NULL column to existing table
// DANGEROUS: 
migrationBuilder.AddColumn<string>("Status", "Orders", nullable: false);
// Existing rows have NULL → constraint violation

// SAFE:
// Step 1: Add nullable column
migrationBuilder.AddColumn<string>("Status", "Orders", nullable: true);
// Step 2: Backfill existing rows
migrationBuilder.Sql("UPDATE Orders SET Status = 'Pending' WHERE Status IS NULL");
// Step 3: Add NOT NULL constraint
migrationBuilder.AlterColumn<string>("Status", "Orders", nullable: false);
```

## 42.03 — Complex Mapping Scenarios

**Table splitting (two entities, one table)**
```csharp
// Order summary (often accessed) and Order detail (rarely accessed)
// Split into same table for performance — avoid joins for common case

public class OrderSummary  { public int Id { get; set; }; public string Status { get; set; } = ""; }
public class OrderDetails  { public int Id { get; set; }; public string Notes { get; set; } = ""; public List<OrderItem> Items { get; set; } = new(); }

builder.Entity<OrderSummary>().ToTable("Orders");
builder.Entity<OrderDetails>().ToTable("Orders");  // Same table!
builder.Entity<OrderSummary>().HasOne<OrderDetails>().WithOne().HasForeignKey<OrderDetails>(d => d.Id);
```

**TPH, TPT, TPC (inheritance mapping)**
```csharp
// Table Per Hierarchy (default, single table with discriminator)
public abstract class Payment { public int Id { get; set; } public decimal Amount { get; set; } }
public class CreditCardPayment : Payment { public string CardLastFour { get; set; } = ""; }
public class BankTransferPayment : Payment { public string AccountNumber { get; set; } = ""; }

builder.Entity<Payment>()
    .HasDiscriminator<string>("PaymentType")
    .HasValue<CreditCardPayment>("CreditCard")
    .HasValue<BankTransferPayment>("BankTransfer");

// Payments table: Id, Amount, PaymentType, CardLastFour (nullable), AccountNumber (nullable)

// Table Per Type (TPC) — .NET 7+ — each concrete type has its own table, no nulls
builder.Entity<CreditCardPayment>().UseTpcMappingStrategy();
// CreditCardPayments table: Id, Amount, CardLastFour
// BankTransferPayments table: Id, Amount, AccountNumber
```
