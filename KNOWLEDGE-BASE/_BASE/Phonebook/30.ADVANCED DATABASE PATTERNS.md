## 30.01 — Database Sharding

**What it is**
Horizontal partitioning: splitting a database table across multiple databases (shards) based on a shard key. Each shard holds a subset of the data.

**Sharding strategies**

**Range-based sharding**
```
Shard 1: CustomerId 1 – 1,000,000
Shard 2: CustomerId 1,000,001 – 2,000,000
Shard 3: CustomerId 2,000,001 – 3,000,000
```
Pros: Simple, range queries efficient within a shard.
Cons: Hot spots (new users all go to last shard).

**Hash-based sharding**
```
ShardIndex = Hash(CustomerId) % NumShards
```
Pros: Even distribution, no hot spots.
Cons: Range queries must hit all shards. Resharding is expensive.

**Directory-based sharding**
A lookup service maps shard keys to shard locations.
Pros: Flexible, can move data between shards.
Cons: Lookup service is a single point of failure.

**Consistent hashing (used by Cassandra, DynamoDB)**
Keys mapped to a ring. Shards own ranges of the ring. Adding/removing shards only affects neighboring shards.

**Problems sharding introduces**
- **Cross-shard queries**: JOIN across shards requires application-level aggregation
- **Cross-shard transactions**: Require distributed transactions (2PC) or sagas
- **Schema changes**: Must be applied to all shards
- **Resharding**: Moving data when shard boundaries change is expensive

**In practice**
Shard only when you have proven need (hundreds of millions of rows, single DB can't keep up). Start with read replicas and vertical scaling first.

## 30.02 — Read Replicas

**What it is**
Secondary database instances that receive a copy of all writes from the primary and serve read queries.

**Setup with SQL Server Always On / Azure SQL**
```
Primary (Read/Write)
    ├── Replica 1 (Read Only) — Same datacenter, sync replication
    ├── Replica 2 (Read Only) — Same datacenter, sync replication
    └── Replica 3 (Read Only) — Different region, async replication
```

**In EF Core — routing reads to replica**
```csharp
// Multiple DbContext registrations
builder.Services.AddDbContext<WriteDbContext>(options =>
    options.UseSqlServer(primaryConnectionString));

builder.Services.AddDbContext<ReadDbContext>(options =>
    options.UseSqlServer(replicaConnectionString)
           .UseQueryTrackingBehavior(QueryTrackingBehavior.NoTracking));

// Query handler uses read context
public class GetOrdersHandler : IRequestHandler<GetOrdersQuery, PagedResult<OrderDto>>
{
    private readonly ReadDbContext _readDb;
    
    public async Task<PagedResult<OrderDto>> Handle(GetOrdersQuery query, CancellationToken ct)
    {
        // Reads go to replica
        return await _readDb.Orders
            .Where(o => o.CustomerId == query.CustomerId)
            .Select(o => o.ToDto())
            .ToPagedResultAsync(query.Page, query.PageSize, ct);
    }
}

// Command handler uses write context
public class CreateOrderHandler : IRequestHandler<CreateOrderCommand, int>
{
    private readonly WriteDbContext _writeDb;
    
    public async Task<int> Handle(CreateOrderCommand cmd, CancellationToken ct)
    {
        var order = Order.Create(cmd.CustomerId);
        _writeDb.Orders.Add(order);
        await _writeDb.SaveChangesAsync(ct);
        return order.Id;
    }
}
```

**Replication lag**
Async replication introduces lag (milliseconds to seconds). A user who creates an order and immediately reads it may get the old state. Solutions:
- **Read-after-write consistency**: After a write, read from primary for a short window
- **Sticky sessions**: Route the writing user to the primary for a TTL
- **Accept eventual consistency**: Most reads don't need immediate consistency

## 30.03 — Change Data Capture (CDC)

**What it is**
A pattern for capturing every database row-level change (insert, update, delete) and making those changes available as a stream.

**SQL Server CDC**
```sql
-- Enable CDC on database
EXEC sys.sp_cdc_enable_db;

-- Enable CDC on table
EXEC sys.sp_cdc_enable_table
    @source_schema = N'dbo',
    @source_name = N'Orders',
    @role_name = NULL;

-- Query captured changes
SELECT
    __$operation,  -- 1=delete, 2=insert, 3=before-update, 4=after-update
    __$start_lsn,  -- Log Sequence Number
    Id,
    Status,
    Total
FROM cdc.dbo_Orders_CT
WHERE __$start_lsn > @lastProcessedLsn
ORDER BY __$start_lsn;
```

**Debezium (open source CDC tool)**
Debezium reads database transaction logs and publishes changes to Kafka:
```
SQL Server Transaction Log → Debezium Connector → Kafka Topic
```

Each change becomes a Kafka message:
```json
{
  "op": "u",
  "before": {"id": 42, "status": "Pending"},
  "after": {"id": 42, "status": "Shipped"},
  "source": {"table": "Orders", "ts_ms": 1700000000}
}
```

**Use cases**
- Event-driven architectures (react to DB changes without polling)
- Search index synchronization (keep Elasticsearch in sync with SQL Server)
- Cache invalidation (invalidate cache when data changes)
- Audit logging
- Replication to analytics databases

## 30.04 — Full-Text Search

**SQL Server Full-Text Search**
```sql
-- Create full-text catalog and index
CREATE FULLTEXT CATALOG ProductCatalog AS DEFAULT;
CREATE FULLTEXT INDEX ON Products(Name, Description)
KEY INDEX PK_Products
ON ProductCatalog;

-- Search
SELECT ProductId, Name, Description,
       KEY_TBL.RANK
FROM Products
INNER JOIN CONTAINSTABLE(Products, *, '"keyboard" OR "mechanical"') AS KEY_TBL
ON Products.ProductId = KEY_TBL.[KEY]
ORDER BY KEY_TBL.RANK DESC;

-- Phrase search
SELECT * FROM Products WHERE CONTAINS(Name, '"gaming keyboard"');

-- Proximity search
SELECT * FROM Products WHERE CONTAINS(*, 'NEAR((gaming, keyboard), 3)');

-- Thesaurus expansion
SELECT * FROM Products WHERE FREETEXT(Description, 'buy');
-- Also finds: purchase, acquire, get
```

**Elasticsearch for production search**
SQL Server FTS is limited for complex scenarios. Elasticsearch offers:
- Relevance scoring (BM25)
- Fuzzy matching (handles typos)
- Faceted search (filter by category, price range)
- Autocomplete
- Multi-language support

```csharp
// NEST / Elastic.Clients.Elasticsearch
var response = await _elasticClient.SearchAsync<ProductDocument>(s => s
    .Index("products")
    .Query(q => q
        .Bool(b => b
            .Must(m => m
                .MultiMatch(mm => mm
                    .Fields(f => f.Field(p => p.Name, boost: 2).Field(p => p.Description))
                    .Query(searchTerm)
                    .Fuzziness(new Fuzziness("AUTO"))))
            .Filter(f => f
                .Range(r => r
                    .Field(p => p.Price)
                    .GreaterThanOrEquals(minPrice)
                    .LessThanOrEquals(maxPrice)))))
    .Aggregations(a => a
        .Terms("categories", t => t.Field(p => p.Category)))
    .From((page - 1) * pageSize)
    .Size(pageSize));
```
