## 3.01 — The Garbage Collector: Generations and Architecture

### What It Actually Is
The .NET Garbage Collector (GC) is a tracing, compacting, generational memory manager. It automatically tracks object reachability from "roots" (stack variables, static fields, CPU registers) and reclaims memory for unreachable objects. The generational hypothesis states that newly created objects die quickly, while older objects tend to stay alive. The GC optimizes by dividing the heap into generations (Gen 0, Gen 1, Gen 2) and collecting the youngest generation most frequently to minimize pause times.

### How It Works — Mechanically

**The Generational Heap:**
```
Gen 0 (Youngest): ~4-8 MB per CPU core. New allocations go here.
                  Collected frequently (hundreds of times per second).
                  Pause time: < 1 ms.
Gen 1 (Buffer):   ~16 MB. Objects that survive one Gen 0 collection.
                  Buffer to prevent short-lived objects from reaching Gen 2.
Gen 2 (Oldest):   Unlimited size. Long-lived objects (caches, singletons).
                  Collected rarely. A Gen 2 collection is a "Full GC" and
                  causes the longest pauses (10-100+ ms).
```

**The Special Heaps:**
```
Large Object Heap (LOH): Objects ≥ 85,000 bytes (e.g., large arrays).
                         Always collected as part of Gen 2.
                         Historically not compacted (leading to fragmentation),
                         but can be explicitly compacted in modern .NET.
Pinned Object Heap (POH): (.NET 5+) Objects explicitly allocated as pinned.
                          Never compacted. Solves fragmentation caused by
                          pinning objects for native interop.
```

**The Mark and Sweep Process:**
1. **Mark Phase:** The GC suspends execution (Stop-The-World) or runs concurrently. It traces from roots, marking all reachable objects.
2. **Plan Phase:** Determines if compaction is necessary based on fragmentation.
3. **Relocate/Sweep Phase:** Updates references to point to new memory locations.
4. **Compact Phase:** Moves surviving objects contiguously in memory to eliminate gaps, updating pointers.

**Write Barriers and Card Tables:**
How does a Gen 0 collection know if a Gen 2 object references a Gen 0 object? Scanning all of Gen 2 during a Gen 0 collection would be too slow. The JIT injects a "write barrier" into every reference assignment (`a.b = c`). If a Gen 2 object is modified to point to a Gen 0 object, a bit is set in a "Card Table". The GC only scans the modified "cards" in Gen 2.

### The Real-World Code

```csharp
using System;
using System.Runtime.InteropServices;
using System.Threading;

public static class GcMechanics
{
    // ──────────────────────────────────────────────────────────────
    // Observing GC Generations and Memory Info
    // Use this to monitor GC health, not to drive application logic.
    // ──────────────────────────────────────────────────────────────
    public static void PrintGcStats()
    {
        var info = GC.GetGCMemoryInfo();
        Console.WriteLine($"Heap Size:      {info.HeapSizeBytes / 1024 / 1024} MB");
        Console.WriteLine($"Fragmented:     {info.FragmentedBytes / 1024 / 1024} MB");
        Console.WriteLine($"Pause Time:     {info.PauseTimePercentage}%");
        
        Console.WriteLine($"Gen 0 Collects: {GC.CollectionCount(0)}");
        Console.WriteLine($"Gen 1 Collects: {GC.CollectionCount(1)}");
        Console.WriteLine($"Gen 2 Collects: {GC.CollectionCount(2)}");
    }

    // ──────────────────────────────────────────────────────────────
    // Forcing a GC (Testing / Benchmarking ONLY)
    // NEVER do this in production unless you are releasing a massive
    // amount of memory (GBs) and want to force compaction immediately.
    // ──────────────────────────────────────────────────────────────
    public static void ForceFullGc()
    {
        // 1. Collect all generations (0, 1, 2)
        // 2. Forced mode (do it now)
        // 3. Blocking (wait for it to finish)
        // 4. Compacting (force LOH compaction if possible)
        GC.Collect(
            generation: 2,
            mode: GCCollectionMode.Forced,
            blocking: true,
            compacting: true);
            
        // Wait for objects in the finalization queue to be processed
        GC.WaitForPendingFinalizers();
        
        // Collect again to clean up objects that were just finalized
        GC.Collect(2, GCCollectionMode.Forced, true, true);
    }

    // ──────────────────────────────────────────────────────────────
    // GC.KeepAlive: Preventing premature collection
    // When dealing with unmanaged resources, the GC might collect an
    // object while a native method is still executing if there are no
    // managed references to it after the native call begins.
    // ──────────────────────────────────────────────────────────────
    [DllImport("kernel32.dll")]
    private static extern void SomeNativeMethod(IntPtr handle);

    public static void CallNative(SafeHandle wrapper)
    {
        IntPtr rawHandle = wrapper.DangerousGetHandle();
        
        // Call into native code
        SomeNativeMethod(rawHandle);
        
        // Ensures 'wrapper' is not collected until this point.
        // Without this, if SomeNativeMethod blocks, the GC could run,
        // collect 'wrapper', run its finalizer, and close the handle
        // while SomeNativeMethod is still using it!
        GC.KeepAlive(wrapper);
    }
}
```

## 3.02 — GC Modes and Latency Tuning

### What It Actually Is
The .NET GC is not one-size-fits-all. It has different operating modes optimized for different workloads: Workstation (UI apps, low memory footprint), Server (high throughput, multiprocessor), and Background (concurrent marking to reduce pause times). .NET 8 introduced DATAS (Dynamic Adaptation To Application Sizes), which automatically switches between Server and Workstation modes based on application load and memory limits.

### How It Works — Mechanically

**Workstation GC:**
- Single GC thread.
- Heap is shared across all cores.
- Fast allocations, but GC phase blocks threads and scales poorly on many-core machines.
- Default for console apps and desktop apps.

**Server GC:**
- One dedicated GC thread and one dedicated heap per logical CPU core.
- Threads allocate lock-free in their core's local heap.
- When a GC occurs, all GC threads work in parallel to collect their respective heaps.
- Extremely high throughput, but high baseline memory usage (creating 32 heaps on a 32-core machine takes memory).
- Default for ASP.NET Core.

**Background GC (Concurrent):**
- Gen 0/1 are always Stop-The-World (they are so fast it doesn't matter).
- Gen 2 collections run concurrently on a background thread.
- Threads can continue allocating in Gen 0 while Gen 2 is being collected.
- If Gen 0 fills up while Gen 2 is being collected, the application pauses until the background GC finishes.

**DATAS (Dynamic Adaptation - .NET 8+):**
- `ServerGarbageCollection` can lead to OOM in constrained container environments (e.g., 512MB limit but 16 cores available, Server GC creates 16 heaps and blows past the limit).
- DATAS adjusts the number of heaps dynamically based on actual throughput and memory limits, decoupling heap count from core count.

### The Real-World Code

```csharp
using System;
using System.Runtime;

// ──────────────────────────────────────────────────────────────────
// Configuration: Typically done in .csproj or runtimeconfig.json
// ──────────────────────────────────────────────────────────────────
/*
<PropertyGroup>
  <!-- Force Server GC -->
  <ServerGarbageCollection>true</ServerGarbageCollection>
  
  <!-- Enable Background GC (enabled by default) -->
  <ConcurrentGarbageCollection>true</ConcurrentGarbageCollection>
  
  <!-- Enable DATAS (.NET 8+) -->
  <GarbageCollectionAdaptationMode>1</GarbageCollectionAdaptationMode>
</PropertyGroup>
*/

public static class GcTuning
{
    // ──────────────────────────────────────────────────────────────
    // Runtime detection of GC mode
    // ──────────────────────────────────────────────────────────────
    public static void PrintMode()
    {
        Console.WriteLine($"Is Server GC: {GCSettings.IsServerGC}");
        Console.WriteLine($"Latency Mode: {GCSettings.LatencyMode}");
    }

    // ──────────────────────────────────────────────────────────────
    // SustainedLowLatency: The "Do Not Disturb" sign for the GC
    // Suspends full blocking Gen 2 collections.
    // Use ONLY for short, critical operations (e.g., HFT trading window,
    // realtime animation frame).
    // WARNING: If you allocate too much in this mode, you will OOM.
    // ──────────────────────────────────────────────────────────────
    public static void RunCriticalPath(Action criticalWork)
    {
        GCLatencyMode oldMode = GCSettings.LatencyMode;
        
        try
        {
            // Suppress expensive Gen 2 blocking GCs
            GCSettings.LatencyMode = GCLatencyMode.SustainedLowLatency;
            
            criticalWork();
        }
        finally
        {
            // MUST restore, even if exception occurs
            GCSettings.LatencyMode = oldMode;
        }
    }
}
```

## 3.03 — JIT Compilation and Tiered Compilation

### What It Actually Is
C# compiles to IL (Intermediate Language). The runtime executes native machine code. The bridge is RyuJIT, the Just-In-Time compiler. Compiling highly optimized native code takes time, which delays app startup. Tiered Compilation solves this by compiling methods twice: first quickly with zero optimizations (Tier 0) to get the app running instantly, and later with aggressive optimizations (Tier 1) for methods that are called frequently.

### How It Works — Mechanically

**Tier 0 (Quick JIT):**
- Triggered on the first call to a method.
- No inlining, no loop unrolling, no hardware intrinsics optimization.
- Generates unoptimized machine code very fast.
- App startup is snappy.
- Inserts a call-counting stub.

**Tier 1 (Optimized JIT):**
- When the call counter reaches a threshold (e.g., 30 calls), the method is queued for recompilation on a background thread.
- RyuJIT applies aggressive optimizations (inlining, devirtualization, CPU-specific AVX instructions).
- When compilation finishes, the method pointer is swapped. Subsequent calls execute the fast Tier 1 code.

**On-Stack Replacement (OSR - .NET 7+):**
What if a method is called once but contains a `while (true)` loop (like a web server listener)? It would never reach 30 calls, so it would run unoptimized forever. OSR allows the JIT to recompile the method and swap the machine code *while the method is currently executing on the stack*.

### The Real-World Code

```csharp
using System;
using System.Runtime.CompilerServices;

// ──────────────────────────────────────────────────────────────────
// MethodImplOptions: Telling the JIT what to do
// ──────────────────────────────────────────────────────────────────
public static class JitControl
{
    // ──────────────────────────────────────────────────────────────
    // AggressiveInlining: 
    // Force the JIT to copy the method body into the caller.
    // Eliminates method call overhead (stack frame setup, jumps).
    // Use for: tiny, frequently called methods (e.g., math, property wrappers).
    // Do NOT use for large methods (causes instruction cache bloat).
    // ──────────────────────────────────────────────────────────────
    [MethodImpl(MethodImplOptions.AggressiveInlining)]
    public static int FastMultiply(int a, int b) => a * b;

    // ──────────────────────────────────────────────────────────────
    // NoInlining:
    // Prevent the JIT from inlining this method.
    // Use for: cold paths (error handling, throwing exceptions).
    // Keeps the hot path small and tightly packed in the CPU instruction cache.
    // ──────────────────────────────────────────────────────────────
    public static void Process(int value)
    {
        if (value < 0)
        {
            // Call is not inlined, keeping Process() body small
            ThrowArgumentOutOfRange(); 
        }
        // ... hot path work ...
    }

    [MethodImpl(MethodImplOptions.NoInlining)]
    private static void ThrowArgumentOutOfRange() 
        => throw new ArgumentOutOfRangeException();

    // ──────────────────────────────────────────────────────────────
    // AggressiveOptimization:
    // Bypass Tier 0. JIT this method immediately at Tier 1.
    // Use for: math-heavy algorithms, physics engines, startup-critical hot loops.
    // Downside: caller blocks longer on first execution.
    // ──────────────────────────────────────────────────────────────
    [MethodImpl(MethodImplOptions.AggressiveOptimization)]
    public static double CalculateFractal(int iterations)
    {
        double result = 0;
        for (int i = 0; i < iterations; i++) { result += Math.Sin(i); }
        return result;
    }
}
```

## 3.04 — Ahead-of-Time Compilation (NativeAOT)

### What It Actually Is
NativeAOT compiles your C# code directly to a self-contained native executable (like C or Go) at publish time. It completely bypasses IL and the JIT compiler at runtime. The resulting binary has no dependency on the .NET runtime, starts instantly, and uses vastly less memory. The tradeoff is severe: you cannot use dynamic code generation (`Reflection.Emit`), unlimited reflection requires explicit hints, and the compile time is significantly longer.

### How It Works — Mechanically

**The AOT Compilation Pipeline:**
```
1. C# Compiler (Roslyn) → IL
2. IL Compiler (ILC)    → Analyzes entire program to find all reachable code.
3. Trimmer              → Deletes any class, method, or property that isn't statically called.
4. RyuJIT (AOT mode)    → Compiles the remaining IL to native object files (.obj / .o).
5. Native Linker        → Links object files with a minimal static GC into a single executable.
```

**The Reflection Problem:**
If you do `Type.GetType("MyNamespace.MyClass")` dynamically based on user input, the Trimmer doesn't know `MyClass` is needed at compile time. It will delete it. At runtime, the reflection call fails.
To fix this, AOT uses **Trimming Attributes** to trace reflection statically.

### The Real-World Code

```csharp
using System;
using System.Diagnostics.CodeAnalysis;
using System.Reflection;

// ──────────────────────────────────────────────────────────────────
// Writing AOT-safe reflection code
// ──────────────────────────────────────────────────────────────────
public class PluginSystem
{
    // ──────────────────────────────────────────────────────────────
    // [DynamicallyAccessedMembers]:
    // Tells the compiler/trimmer: "I am going to use reflection to look for
    // public constructors on the type passed into this parameter. Do NOT delete them."
    // ──────────────────────────────────────────────────────────────
    public static object CreateInstance(
        [DynamicallyAccessedMembers(DynamicallyAccessedMemberTypes.PublicConstructors)] Type type)
    {
        // Safe: Trimmer preserved the constructor because of the attribute
        return Activator.CreateInstance(type)!; 
    }

    // ──────────────────────────────────────────────────────────────
    // [RequiresUnreferencedCode]:
    // Tells the compiler: "This method uses unsafe dynamic reflection. 
    // Emit a compile-time warning if someone calls this in an AOT build."
    // It acts as a contagion marker — callers must either suppress it or mark themselves.
    // ──────────────────────────────────────────────────────────────
    [RequiresUnreferencedCode("Instantiates a type by string name, which breaks AOT.")]
    public static object CreateByName(string typeName)
    {
        Type? type = Type.GetType(typeName); // Trimmer cannot see this statically!
        return Activator.CreateInstance(type!)!;
    }
}

// ──────────────────────────────────────────────────────────────────
// JSON Serialization in AOT
// Standard reflection-based JsonSerializer fails in AOT.
// You MUST use Source Generators, which write the serialization C# code
// at compile time, completely eliminating reflection.
// ──────────────────────────────────────────────────────────────────
using System.Text.Json;
using System.Text.Json.Serialization;

public record UserDto(int Id, string Name);

// The source generator inspects this at compile time and writes the parsing logic
[JsonSerializable(typeof(UserDto))]
internal partial class AppJsonSerializerContext : JsonSerializerContext { }

public static class AotJson
{
    public static string Serialize(UserDto user)
    {
        // Pass the generated context. No reflection used. AOT safe.
        return JsonSerializer.Serialize(user, AppJsonSerializerContext.Default.UserDto);
    }
}
```

## 3.05 — Assembly Loading and AssemblyLoadContext (ALC)

### What It Actually Is
Historically (.NET Framework), DLLs were isolated using `AppDomain`s. In .NET Core/.NET 5+, AppDomains are deprecated. Assembly isolation and dynamic loading/unloading is handled by `AssemblyLoadContext` (ALC). An ALC is a scope for loaded assemblies. You can load a plugin and its dependencies into a custom ALC, execute it, and then unload the entire ALC from memory, allowing hot-swapping of code in a running server.

### How It Works — Mechanically

**Default ALC vs Custom ALC:**
- **Default ALC:** Contains the runtime itself and your main application DLLs. Cannot be unloaded.
- **Custom ALC:** Created manually. When it resolves a dependency (e.g., `Newtonsoft.Json`), it checks if it's already in the Default ALC. If so, it shares it. If not, it loads its own copy.

**Unloading Mechanics:**
You cannot unload a single DLL. You unload the entire ALC.
The ALC will only unload if there are **zero strong references** to any type, object, or assembly loaded inside it. If the main app holds a reference to a plugin instance, the ALC cannot be GC'd, creating a massive memory leak.

### The Real-World Code

```csharp
using System;
using System.Reflection;
using System.Runtime.Loader;

// ──────────────────────────────────────────────────────────────────
// Creating an unloadable ALC for Plugins
// ──────────────────────────────────────────────────────────────────
public class PluginLoadContext : AssemblyLoadContext
{
    private readonly AssemblyDependencyResolver _resolver;

    public PluginLoadContext(string pluginPath) 
        // isCollectible: true is REQUIRED for unloading
        : base(isCollectible: true) 
    {
        _resolver = new AssemblyDependencyResolver(pluginPath);
    }

    // Resolves dependencies that the plugin DLL references
    protected override Assembly? Load(AssemblyName assemblyName)
    {
        string? assemblyPath = _resolver.ResolveAssemblyToPath(assemblyName);
        if (assemblyPath != null)
        {
            return LoadFromAssemblyPath(assemblyPath);
        }
        return null; // Fall back to Default ALC
    }
}

public static class PluginHost
{
    // Crucial: This method is marked NoInlining.
    // If it were inlined, the compiler might keep hidden stack references 
    // to the ALC or plugin types, preventing the GC from unloading the ALC.
    [System.Runtime.CompilerServices.MethodImpl(System.Runtime.CompilerServices.MethodImplOptions.NoInlining)]
    public static WeakReference ExecuteAndUnload(string pluginPath)
    {
        var alc = new PluginLoadContext(pluginPath);
        Assembly assembly = alc.LoadFromAssemblyPath(pluginPath);
        
        // Find and execute plugin
        Type pluginType = assembly.GetType("MyPlugin.EntryPoint")!;
        var executeMethod = pluginType.GetMethod("Execute")!;
        executeMethod.Invoke(null, null);

        // Initiate unload
        alc.Unload();

        // Return a weak reference to the ALC to track if it actually unloads
        return new WeakReference(alc);
    }

    public static void Run()
    {
        WeakReference alcRef = ExecuteAndUnload("/path/to/plugin.dll");

        // Force GC to sweep the uncollected ALC
        for (int i = 0; i < 3 && alcRef.IsAlive; i++)
        {
            GC.Collect();
            GC.WaitForPendingFinalizers();
        }

        Console.WriteLine($"Plugin Unloaded: {!alcRef.IsAlive}");
    }
}
```

## 3.06 — Object Layout and Type System Internals

### What It Actually Is
In C#, you deal with objects. In the CLR, an object is a contiguous block of memory on the heap. Every reference type instance has hidden overhead injected by the runtime (the object header and MethodTable pointer). Understanding this layout is essential for high-performance computing, predicting memory consumption, and understanding how virtual dispatch and locking actually work under the hood.

### How It Works — Mechanically

**The Reference Type Heap Layout:**
For a class `class Player { int Health; byte Level; }` on a 64-bit system:

```
[ Heap Address ]
-08: Sync Block Index (8 bytes)  ← Used for lock(), hash codes, GC flags
+00: MethodTable Ptr  (8 bytes)  ← Points to type info (virtual tables, RTTI)
+08: Health (int)     (4 bytes)  ← First field
+12: Level (byte)     (1 byte)   ← Second field
+13: Padding          (3 bytes)  ← Aligns object to 8-byte boundary
     ------------------------
     Total: 24 bytes for 5 bytes of actual data.
```
*Minimum object size in .NET 64-bit is 24 bytes.*

**The MethodTable (EEClass):**
The MethodTable pointer defines the type of the object. It points to a structure containing:
- Type metadata (name, base class, interfaces).
- The **Virtual Method Table (vtable)**: An array of function pointers. When you call `obj.ToString()`, the CLR reads the MethodTable, finds the vtable index for `ToString`, and jumps to that memory address.

### The Real-World Code

```csharp
using System;
using System.Runtime.InteropServices;
using System.Runtime.CompilerServices;

public static class ObjectLayout
{
    // ──────────────────────────────────────────────────────────────
    // Value Types (Structs) have NO header. 
    // They are raw data.
    // ──────────────────────────────────────────────────────────────
    [StructLayout(LayoutKind.Sequential)]
    public struct PointStruct 
    { 
        public int X; // 4 bytes
        public int Y; // 4 bytes 
    } // Total size: 8 bytes (0 overhead)

    public class PointClass 
    { 
        public int X; 
        public int Y; 
    } // Total size: 24 bytes (16 bytes overhead)

    // ──────────────────────────────────────────────────────────────
    // Controlling struct layout for Interop and CPU cache efficiency
    // StructLayout allows you to pack data tightly.
    // ──────────────────────────────────────────────────────────────
    
    // Auto: Let the CLR reorder fields to minimize padding.
    [StructLayout(LayoutKind.Auto)] 
    public struct AutoStruct
    {
        public byte B1;
        public int I1;
        public byte B2;
        // CLR reorders to: I1 (4), B1 (1), B2 (1), Padding (2) = 8 bytes
    }

    // Explicit: You control the exact byte offsets (like a C union).
    [StructLayout(LayoutKind.Explicit)]
    public struct IntFloatUnion
    {
        [FieldOffset(0)] public int IntValue;
        [FieldOffset(0)] public float FloatValue; // Overlaps IntValue!
    }

    public static unsafe void Explore()
    {
        // Unsafe.SizeOf measures the payload size
        Console.WriteLine($"Struct payload: {Unsafe.SizeOf<PointStruct>()}"); // 8
        
        // Classes are references, SizeOf returns pointer size
        Console.WriteLine($"Class ref size: {Unsafe.SizeOf<PointClass>()}");  // 8

        // Fast float-to-int bitwise conversion without math
        var union = new IntFloatUnion { FloatValue = 3.14f };
        Console.WriteLine($"Bits: {union.IntValue:X}");
    }
}
```

## 3.07 — Boxing, Unboxing, and the Heap Interface

### What It Actually Is
Value types (structs, ints, enums) live on the stack or inline within objects. Reference types live on the heap. **Boxing** is the act of converting a value type to a reference type (e.g., casting `int` to `object` or `IComparable`). The runtime must allocate a brand new object on the heap, copy the value type's fields into it, and return a reference. **Unboxing** is extracting the value back out. Boxing is the #1 cause of hidden GC pressure in legacy or poorly written generic code.

### How It Works — Mechanically

**The Box Operation (IL instruction `box`):**
```csharp
int x = 42;
object o = x; // BOXING
```
1. CLR calculates size of `int` (4 bytes).
2. Allocates a new object on the heap (16 bytes header + 4 bytes payload = 24 bytes).
3. Sets the MethodTable pointer to `System.Int32`.
4. Copies the value `42` into the payload.
5. `o` now points to this heap object.

**The Unbox Operation (IL instruction `unbox.any`):**
```csharp
int y = (int)o; // UNBOXING
```
1. CLR checks if `o` is non-null.
2. CLR checks if the MethodTable of `o` is exactly `System.Int32` (type check).
3. If true, it returns a pointer to the payload inside the box.
4. The value is copied from the box into the stack variable `y`.

### The Real-World Code

```csharp
using System;
using System.Diagnostics;

public static class BoxingTraps
{
    // ──────────────────────────────────────────────────────────────
    // Trap 1: Interface calls on structs
    // Casting a struct to an interface BOXES the struct.
    // ──────────────────────────────────────────────────────────────
    public interface ICounter { void Increment(); }
    
    public struct Counter : ICounter 
    { 
        public int Value; 
        public void Increment() => Value++; 
    }

    public static void InterfaceBoxing()
    {
        Counter c = new Counter();
        
        // BOXING! A new heap object is created.
        ICounter i = c; 
        
        // Mutates the heap object, NOT the local stack struct!
        i.Increment(); 
        
        Console.WriteLine(c.Value); // Outputs 0! The original is untouched.
    }

    // ──────────────────────────────────────────────────────────────
    // Solution 1: Generic constraints prevent boxing
    // ──────────────────────────────────────────────────────────────
    public static void IncrementGeneric<T>(ref T counter) where T : ICounter
    {
        // NO BOXING. The JIT generates a specific method for Counter 
        // and calls Increment() directly on the struct.
        counter.Increment(); 
    }

    // ──────────────────────────────────────────────────────────────
    // Trap 2: Enum.HasFlag in older .NET
    // Prior to .NET Core 2.1, Enum.HasFlag took an `Enum` (which is a class).
    // Calling it boxed the arguments.
    // .NET Core 2.1+ intrinsicified it to zero-allocation.
    // Standard bitwise math is still universally allocation-free.
    // ──────────────────────────────────────────────────────────────
    [Flags] public enum Roles { None = 0, User = 1, Admin = 2 }

    public static bool IsAdminFast(Roles role)
    {
        // Universal, zero-allocation, fastest bitwise check
        return (role & Roles.Admin) == Roles.Admin;
    }
}
```

## 3.08 — Thread Pool Internals and Starvation

### What It Actually Is
The .NET ThreadPool manages a pool of worker threads to execute async continuations, TPL tasks, and timers. Creating a new OS thread is brutally expensive (~1MB stack allocation, kernel context switches). The ThreadPool reuses threads. However, if your code blocks a ThreadPool thread (e.g., `Task.Wait()`, Thread.Sleep, sync I/O), the pool runs out of available threads. This is **Thread Pool Starvation**.

### How It Works — Mechanically

**Queues and Work Stealing:**
- **Global Queue:** Used for top-level work items (`Task.Run`). All threads lock to dequeue from here.
- **Local Queues:** Each ThreadPool thread has its own local queue. When a thread spawns child tasks (e.g., `await` continuations), they go to the local queue. Dequeuing locally is lock-free and LIFO (Locality of reference).
- **Work Stealing:** If a thread empties its local queue, it looks at other threads' local queues and steals work from the tail (FIFO) to balance the load.

**Hill Climbing Algorithm:**
If all threads are busy, should the pool create a new thread?
- Creating too many threads causes CPU thrashing (context switching overhead).
- Creating too few causes bottlenecks.
The ThreadPool uses a "Hill Climbing" heuristic. It slowly injects a new thread (about 1-2 per second) and measures if overall throughput improves. If yes, it keeps adding. If throughput drops, it kills threads.

**Starvation:**
Because Hill Climbing injects threads slowly (1 per second), if a sudden burst of 50 requests all do `Thread.Sleep(5000)`, the pool is instantly exhausted. It will take 50 seconds to spawn enough threads to handle them. The application hangs.

### The Real-World Code

```csharp
using System;
using System.Threading;
using System.Threading.Tasks;

public static class ThreadPoolMechanics
{
    // ──────────────────────────────────────────────────────────────
    // Monitoring the Thread Pool
    // ──────────────────────────────────────────────────────────────
    public static void PrintPoolState()
    {
        ThreadPool.GetAvailableThreads(out int worker, out int io);
        ThreadPool.GetMaxThreads(out int maxWorker, out int maxIo);
        ThreadPool.GetMinThreads(out int minWorker, out int minIo);

        Console.WriteLine($"Workers: {maxWorker - worker} active / {maxWorker} max");
        Console.WriteLine($"Min Workers: {minWorker} (Threads instantly available)");
    }

    // ──────────────────────────────────────────────────────────────
    // Mitigating Burst Starvation (The Band-Aid)
    // If you have blocking code you cannot fix, increase the MinThreads.
    // This pre-allocates threads so the Hill Climbing delay is avoided.
    // (Common in legacy Redis/Db connections migrating to cloud).
    // ──────────────────────────────────────────────────────────────
    public static void ConfigureForBlockingWorkloads()
    {
        // 100 threads instantly available before Hill Climbing kicks in
        ThreadPool.SetMinThreads(100, 100); 
    }

    // ──────────────────────────────────────────────────────────────
    // Sync-Over-Async (The ultimate starvation trigger)
    // NEVER do this in ASP.NET Core. 
    // ──────────────────────────────────────────────────────────────
    public static void DeadlockOrStarve()
    {
        // Thread A from pool starts this.
        Task<string> task = HttpClient.GetStringAsync("http://slow.com");
        
        // Thread A BLOCKS. It is dead. It cannot process other work.
        // When HTTP call finishes, it needs a ThreadPool thread to run the continuation.
        // If all threads are blocked like Thread A, the continuation cannot run.
        // Deadlock.
        string result = task.Result; 
    }
}
```

## 3.09 — Pinned Objects and the POH (Pinned Object Heap)

### What It Actually Is
When passing a managed object (like a `byte[]` buffer) to an unmanaged OS API (like a socket read), the OS needs the raw memory pointer. But the .NET GC is a compacting GC—it moves objects around to defragment the heap. If the GC moves the array while the OS is writing to it, memory corruption occurs. **Pinning** locks the object in place. Historically, pinning fragmented the Gen 0/LOH heaps. .NET 5 introduced the **Pinned Object Heap (POH)**, a dedicated heap for objects that will never move.

### How It Works — Mechanically

**The `fixed` block:**
A lightweight, stack-based pin. Instructs the GC: "Do not move this object as long as this stack frame is executing." Very cheap, but cannot span across `async/await` boundaries.

**`GCHandle.Alloc(obj, GCHandleType.Pinned)`:**
A heavyweight pin. Registers the object in a global handle table. Survives across async boundaries. Fragments the heap if the object was in Gen 0/1, as the GC must compact *around* it, leaving holes.

**The POH (.NET 5+):**
Instead of allocating normally and then pinning (causing fragmentation), you tell the runtime *at allocation time* that the object will be pinned. It is allocated directly on the POH. The GC knows to simply ignore the POH during compaction phases.

### The Real-World Code

```csharp
using System;
using System.Runtime.InteropServices;
using System.Threading.Tasks;

public static class PinningPatterns
{
    // ──────────────────────────────────────────────────────────────
    // 1. The fixed statement (Stack-scoped, zero-overhead pinning)
    // Use for fast, synchronous interop.
    // ──────────────────────────────────────────────────────────────
    [DllImport("kernel32.dll")]
    private static extern unsafe void ProcessBuffer(byte* buffer, int length);

    public static unsafe void SyncNativeCall(byte[] data)
    {
        // The GC will not move 'data' while inside this block
        fixed (byte* ptr = data)
        {
            ProcessBuffer(ptr, data.Length);
        } // Pin released automatically
    }

    // ──────────────────────────────────────────────────────────────
    // 2. The Pinned Object Heap (.NET 5+)
    // Use for long-lived buffers passed to async I/O or persistent native APIs.
    // ──────────────────────────────────────────────────────────────
    public static void AllocateOnPoh()
    {
        // Allocate directly on the POH. 
        // This array will never move for its entire lifetime.
        byte[] buffer = GC.AllocateUninitializedArray<byte>(
            length: 1024 * 64, 
            pinned: true);

        // We can safely get a pointer without a 'fixed' block,
        // because we know it's on the POH.
        IntPtr ptr = Marshal.UnsafeAddrOfPinnedArrayElement(buffer, 0);

        // Pass 'ptr' to overlapping/async native I/O safely.
    }
}
```

## 3.10 — IL (Intermediate Language) and Execution

### What It Actually Is
C# does not compile to machine code. It compiles to CIL (Common Intermediate Language). IL is an object-oriented, stack-based assembly language. Understanding IL is crucial because it removes the magic of C# syntax sugar (like `yield`, `await`, and `lock`). Furthermore, libraries like Dapper, AutoMapper, and Entity Framework use IL generation (Reflection.Emit) at runtime to dynamically create incredibly fast mapping methods.

### How It Works — Mechanically

**The Evaluation Stack:**
IL doesn't use CPU registers (like EAX, RBX). It uses a conceptual "Evaluation Stack". Operations push values onto the stack, and operators pop them, compute, and push the result.

```csharp
int a = 5;
int b = 10;
int c = a + b;
```
Compiles to IL:
```text
ldc.i4.5    // Push integer constant 5 onto stack
stloc.0     // Pop stack and store in local variable 0 (a)
ldc.i4.s 10 // Push integer constant 10 onto stack
stloc.1     // Pop stack and store in local variable 1 (b)

ldloc.0     // Push local 0 onto stack
ldloc.1     // Push local 1 onto stack
add         // Pop two values, add them, push result
stloc.2     // Pop stack and store in local variable 2 (c)
```

**Virtual vs Non-Virtual Calls:**
- `call`: Calls a method statically. Fast. Used for static methods, structs, and `base.Method()`.
- `callvirt`: Checks for null (throws NullReferenceException if so), then looks up the MethodTable to find the most derived implementation. Slower. Used for class instance methods.

### The Real-World Code

```csharp
using System;
using System.Reflection.Emit;

public static class EmitPatterns
{
    // ──────────────────────────────────────────────────────────────
    // DynamicMethod (Reflection.Emit)
    // Generating IL at runtime. This is how Dapper maps SQL results
    // to objects 100x faster than standard reflection.
    // ──────────────────────────────────────────────────────────────
    public static Func<int, int, int> CreateDynamicAdder()
    {
        // 1. Create a dynamic method (Name, ReturnType, ParameterTypes)
        var method = new DynamicMethod(
            name: "DynamicAdd",
            returnType: typeof(int),
            parameterTypes: new[] { typeof(int), typeof(int) });

        // 2. Get an IL Generator
        ILGenerator il = method.GetILGenerator();

        // 3. Emit IL instructions
        il.Emit(OpCodes.Ldarg_0); // Push 1st argument (int a) to stack
        il.Emit(OpCodes.Ldarg_1); // Push 2nd argument (int b) to stack
        il.Emit(OpCodes.Add);     // Pop a & b, add them, push result
        il.Emit(OpCodes.Ret);     // Return the value on top of the stack

        // 4. Compile the IL to a callable Delegate (triggers the JIT)
        return (Func<int, int, int>)method.CreateDelegate(typeof(Func<int, int, int>));
    }
    
    // Usage:
    // var adder = CreateDynamicAdder();
    // int result = adder(5, 10); // Runs as fast as native C# code!
}
```

## 3.11 — Diagnostics, EventPipe, and Profiling

### What It Actually Is
The .NET runtime is heavily instrumented. It emits thousands of diagnostic events (GC pauses, JIT compilation times, exceptions, thread pool expansion) via a high-performance cross-platform tracing subsystem called **EventPipe**. Tools like `dotnet-trace`, `dotnet-counters`, and Visual Studio Profiler subscribe to EventPipe. You can also subscribe to these events *in-process* to build self-diagnosing microservices.

### How It Works — Mechanically

- **EventSource:** The base class for emitting events. The runtime has built-in sources like `Microsoft-Windows-DotNETRuntime`.
- **EventListener:** An in-process subscriber that can listen to any `EventSource`.
- **EventPipe:** The multiplexer that routes events from Sources to Listeners, or writes them to IPC channels (named pipes/Unix domain sockets) for external tools to read.
- **dotnet-counters:** Uses EventPipe to request aggregated metric payloads every 1 second without high tracing overhead.

### The Real-World Code

```csharp
using System;
using System.Diagnostics.Tracing;

// ──────────────────────────────────────────────────────────────────
// In-Process EventListener
// Capturing deep runtime metrics (like GC pauses or Exception rates)
// programmatically to push to Prometheus/Datadog.
// ──────────────────────────────────────────────────────────────────
public sealed class RuntimeEventListener : EventListener
{
    // The specific GUID/Name for the CoreCLR runtime event source
    private const string RuntimeEventSourceName = "Microsoft-Windows-DotNETRuntime";

    protected override void OnEventSourceCreated(EventSource eventSource)
    {
        if (eventSource.Name == RuntimeEventSourceName)
        {
            // Subscribe to GC events. 
            // Keywords filter the noise (we only want GC, not JIT/Exceptions).
            // Level is Informational.
            EnableEvents(eventSource, EventLevel.Informational, (EventKeywords)0x1); 
            Console.WriteLine("Subscribed to GC Events");
        }
    }

    protected override void OnEventWritten(EventWrittenEventArgs eventData)
    {
        // EventId 1 = GCStart, EventId 2 = GCEnd
        if (eventData.EventName == "GCStart_V2")
        {
            // The payload contains data like the Reason and Generation
            int generation = (int)eventData.Payload![1]!;
            int reason = (int)eventData.Payload![2]!;
            
            Console.WriteLine($"[GC START] Generation: {generation}, Reason: {reason}");
        }
        else if (eventData.EventName == "GCEnd_V1")
        {
            Console.WriteLine($"[GC END]");
        }
    }
}

// Usage:
// using var listener = new RuntimeEventListener();
// GC.Collect(); // Will trigger the Console.WriteLines in the listener
```
