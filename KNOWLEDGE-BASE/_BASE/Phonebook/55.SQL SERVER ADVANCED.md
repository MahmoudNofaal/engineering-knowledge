## 55.01 — Locking and Concurrency

**Lock types**
```
Shared (S):     Multiple readers can hold simultaneously. Blocks exclusive.
Exclusive (X):  Only one writer. Blocks all other locks.
Update (U):     Used during UPDATE scanning phase. Blocks other U and X.
Intent:         Indicates intent to lock at lower granularity (IS, IX, IU).
Schema:         Blocks schema-altering DDL during query execution.
```

**Lock granularity**
```
Row → Page (8KB) → Partition → Table

Lock escalation: SQL Server promotes many row locks to a table lock to save memory.
Default threshold: 5,000 row locks → table lock.
Problem: Table lock blocks all other queries.
Fix: Disable escalation for specific tables or use RCSI.
```

**Detecting and resolving deadlocks**
```sql
-- Enable deadlock trace flag (session-level, for debugging)
DBCC TRACEON(1222, -1);

-- Or configure trace in SQL Server Management:
-- Server Properties → Advanced → Deadlock Priority/Timeout

-- Deadlock victim: SQL Server picks the least expensive transaction to kill
-- That transaction receives error 1205: "Transaction was deadlocked"

-- Preventing deadlocks:
-- 1. Consistent lock order (always lock Table A before Table B)
-- 2. Keep transactions short
-- 3. Use RCSI (readers don't block writers)
-- 4. Use covering indexes (fewer locks per query)

-- Read committed snapshot isolation — zero lock contention for reads
ALTER DATABASE MyDatabase SET READ_COMMITTED_SNAPSHOT ON;
-- Now reads use row versioning instead of locking — no reader/writer contention
```

**Optimistic concurrency with rowversion**
```sql
-- ETag-style concurrency with rowversion
CREATE TABLE Orders (
    Id INT PRIMARY KEY,
    Status VARCHAR(50),
    RowVersion ROWVERSION NOT NULL  -- Auto-updated on every modification
);

-- Read:
SELECT Id, Status, RowVersion FROM Orders WHERE Id = 42;
-- Returns: Id=42, Status='Pending', RowVersion=0x0000000000000ABC

-- Update — only succeeds if RowVersion hasn't changed:
UPDATE Orders
SET Status = 'Shipped'
WHERE Id = 42
AND RowVersion = 0x0000000000000ABC;  -- Pass back the value you read

-- If 0 rows affected → concurrent modification detected → conflict!

-- In EF Core with rowversion:
modelBuilder.Entity<Order>()
    .Property(o => o.RowVersion)
    .IsRowVersion();  // EF Core automatically uses this for concurrency checks
```

## 55.02 — Advanced Query Optimization

**Statistics and cardinality estimation**
```sql
-- Statistics contain histograms of column value distribution
-- Optimizer uses them to estimate rows returned (cardinality)

-- Check statistics
DBCC SHOW_STATISTICS('Orders', 'IX_Orders_CustomerId');

-- Update stale statistics
UPDATE STATISTICS Orders;
UPDATE STATISTICS Orders IX_Orders_CustomerId WITH FULLSCAN;

-- Auto-update statistics (on by default, but threshold matters)
-- SQL Server updates stats when 20% of rows change (2008+) or with trace flag 2371

-- Cardinality estimation bug: two correlated columns
-- Optimizer may wildly underestimate rows if two filter columns are correlated
-- Example: State='TX' AND City='Houston' — optimizer assumes independence
-- Fix: Create statistics on multiple columns
CREATE STATISTICS stat_State_City ON Customers(State, City);
```

**Query hints (use sparingly)**
```sql
-- Force index use
SELECT * FROM Orders WITH (INDEX(IX_Orders_CustomerId))
WHERE CustomerId = 42;

-- Force join type
SELECT * FROM Orders o
INNER MERGE JOIN Customers c ON o.CustomerId = c.Id;
-- LOOP, HASH, MERGE — override optimizer's choice

-- NOLOCK (dirty reads) — use only for dashboards/reports with tolerance for stale data
SELECT COUNT(*) FROM Orders WITH (NOLOCK);

-- RECOMPILE — regenerate plan for each execution
SELECT * FROM Orders WHERE Status = @status OPTION (RECOMPILE);

-- MAXDOP — limit parallelism
SELECT * FROM LargeTable OPTION (MAXDOP 4);  -- Use at most 4 CPU cores
OPTION (MAXDOP 1)  -- Disable parallelism entirely
```

**Columnstore indexes**
```sql
-- For analytics/reporting queries on large tables
-- Stores data column-by-column (great for aggregations, bad for row lookups)
-- Very high compression, batch processing (up to 10x faster analytics)

CREATE COLUMNSTORE INDEX IX_Orders_Columnstore
ON Orders (OrderDate, Status, Total, CustomerId);

-- Query uses batch mode (processes 900 rows at a time, vectorized)
SELECT
    YEAR(OrderDate) AS Year,
    MONTH(OrderDate) AS Month,
    Status,
    SUM(Total) AS Revenue,
    COUNT(*) AS OrderCount
FROM Orders
WHERE OrderDate >= '2025-01-01'
GROUP BY YEAR(OrderDate), MONTH(OrderDate), Status;
-- 100x faster than row store for analytics queries on large data
```

## 55.03 — Partitioning

**Table partitioning**
```sql
-- Partition function: maps values to partition numbers
CREATE PARTITION FUNCTION pfOrderDate (DATE)
AS RANGE RIGHT FOR VALUES (
    '2024-01-01', '2025-01-01', '2026-01-01'
);
-- Partition 1: < 2024-01-01
-- Partition 2: 2024-01-01 – 2024-12-31
-- Partition 3: 2025-01-01 – 2025-12-31
-- Partition 4: >= 2026-01-01

-- Partition scheme: maps partitions to filegroups
CREATE PARTITION SCHEME psOrderDate
AS PARTITION pfOrderDate
ALL TO ([PRIMARY]);  -- Or distribute across multiple filegroups for I/O

-- Apply to table
CREATE TABLE Orders (
    Id INT NOT NULL,
    OrderDate DATE NOT NULL,
    Status VARCHAR(50),
    Total DECIMAL(18, 2),
    PRIMARY KEY (Id, OrderDate)  -- Partition column must be in PK
) ON psOrderDate(OrderDate);

-- Benefits:
-- 1. Partition elimination: query with WHERE OrderDate = '2026-01-15' only scans 1 partition
-- 2. Partition switching: archive old data instantly (no row-by-row delete)
-- 3. Maintenance: rebuild index for one partition at a time

-- Archive old partition (instant, zero copy)
ALTER TABLE Orders SWITCH PARTITION 1 TO OrdersArchive PARTITION 1;
```
